# cookimport/llm

Optional LLM integrations live here.

Recipe correction now enters through the thin `codex_farm_orchestrator.py` facade and the `recipe_stage/` owner package (`planning.py`, `runtime.py`, `validation.py`, `promotion.py`, `recovery.py`, `reporting.py`), with the private backing implementation currently living in `recipe_stage_shared.py`. Strict compact contracts still live in `codex_farm_contracts.py`, direct subprocess execution in `codex_exec_runner.py`, and fake/legacy helpers in `fake_codex_farm_runner.py` plus `codex_farm_runner.py`.
The shared shard-runtime foundation now lives in `phase_worker_runtime.py`: it writes phase/shard manifests, round-robin worker assignments, per-worker sandboxes, per-shard proposed outputs, promotion summaries, and telemetry without promoting anything itself.
Naming contract for this folder: a `shard` is the planning/scheduling bundle, a `task` is the smallest file-backed work item that writes one result, a `packet` is the model-facing content slice owned by one task, and a `worker` is the long-lived Codex session that executes assigned shards/tasks. Line-role, recipe, and knowledge all follow that contract even when older filenames still say `SHARD_PACKET.md`.
`codex_exec_runner.py` now has two direct-exec modes: a one-shot structured prompt mode and a long-lived `workspace_worker` mode that lets one Codex session read local task files and write local `out/*.json` results inside the sterile workspace before the runner syncs them back to the repo-owned worker folder.
Workspace-worker roots now also get a repo-written `worker_manifest.json`, and the shared worker contract explicitly says to open the named local files directly instead of running `pwd`/`ls`/`find`/`tree` just to orient. Non-leased task queues can now expose either a repo-written current-batch bundle (`current_batch.json`, `CURRENT_BATCH.md`, `CURRENT_BATCH_FEEDBACK.md`) or a narrower current-task bundle (`current_task.json`, `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`) depending on the stage. The knowledge stage is now batch-first, and the live planner treats `knowledge_prompt_target_count` as a target bounded by the packet floor: repo code sizes ordered packets first, then keeps one task per packet so no review-eligible packet is dropped. Repo code still batches a few consecutive task rows at a time for one worker session, prewrites `scratch/current_batch/<task_id>.json` drafts, and mirrors the advanced batch sidecars source -> execution again after each accepted install so the same long-lived worker session sees the next repo-owned batch without inventing its own shell scheduler. Those knowledge batch sidecars are intentionally compact now: `CURRENT_BATCH.md` is the cheap first read, `current_batch.json` keeps only batch-local task facts, `assigned_tasks.json` is fallback/debug inventory only, and the repo-written helper surfaces now talk in packet ids, block decisions, and idea groups rather than chunk-result wrappers or cue-taxonomy scaffolds. The default batch caps are now `8` tasks or `48_000` input bytes so the worker loop amortizes more task rows per helper cycle. After each successful install, the happy path is now to re-open `CURRENT_BATCH.md` and `CURRENT_BATCH_FEEDBACK.md`; `current_batch.json` is only needed when the next batch's machine-readable task paths or task rows are actually needed. When a stage uses packet leasing, that root instead carries `current_packet.json`, `current_hint.md`, `current_result_path.txt`, `packet_lease_status.json`, and `scratch/`, so the worker sees one authoritative current packet at a time while the repo advances the lease.
The live knowledge planner surface is intentionally small: `build_knowledge_jobs(...)` takes full blocks, candidate spans, recipe spans, workbook identity, output dir, `context_blocks`, and `prompt_target_count`. There are no extra planner override knobs hiding under that public seam.
Knowledge shard metadata is shard-native too: manifests keep `owned_packet_ids`, `packet_count`, `owned_block_indices`, and total `char_count`, so recovery/reporting logic sizes the whole shard bundle instead of pretending a multi-packet shard is one packet.
The knowledge stage now optimizes for durable cooking utility rather than broad cooking-related factuality. The billed `knowledge/in/*.json` payload stays raw and mechanically true, and the model owns the final `knowledge|other` call plus the related-idea grouping inside each packet.
When a workspace-worker run finishes without a JSON `turn.completed` usage payload, `codex_exec_runner.py` now also falls back to the plain-text `Token usage: ...` footer the current Codex CLI prints, so normal worker `usage.json` artifacts can stay populated without weakening the existing fail-closed reporting path.
When repo supervision decides a workspace-worker session is `completed` or `completed_with_failures`, the runner now gives Codex a longer grace window before sending termination so late `turn.completed` usage or plain-text usage footer output can still land in `events.jsonl` / `usage.json` instead of being cut off by the repo-owned stop decision. Line-role and knowledge success watchdogs now also wait for a post-finalize / post-install completion signal, or a short quiet period, before asking the runner to terminate at all.
Line-role and knowledge worker roots now also persist raw `stdout.txt` / `stderr.txt` sidecars when the direct-exec subprocess emitted text, so missing-usage runs are diagnosable from the saved artifacts instead of only from `events.jsonl`.
Direct workspace-worker telemetry now also fails closed on spend: if a worker row shows real activity but no usable Codex usage payload, the runtime summary marks `token_usage_status` as `partial` or `unavailable` and leaves billed-token totals blank instead of reporting literal zero spend.
Knowledge now also enforces that batch contract at watchdog time, but more narrowly than before: only ad hoc queue-control rewrites are treated as off-contract main-worker behavior and killed for retry. Bounded local automation over only the active batch drafts is tolerated, helper-source reads of `tools/knowledge_worker.py` are discouraged telemetry rather than an immediate kill by themselves, and `assigned_tasks.json` dumps, single-task helper detours, plus broad shell scheduler loops are now discouraged fallback moves rather than fatal watchdog events.
Watchdog policy is now split by transport: structured retry / repair calls still kill on the first shell command, while workspace-worker main attempts can use ordinary local shell syntax throughout the session, including pipes, redirections, heredocs, absolute paths, parent-traversal-looking strings, root-relative scratch files, and glob reads.
The main-session watchdog is now executable-based instead of path-policed: it blocks clearly egregious network/package-manager/remote/container commands plus mutating `git` verbs, and otherwise relies on the command-loop budget to catch shell thrash without outputs. Read-only `git` inspection no longer counts as fatal. That loop budget is now slightly progress-aware too: recent valid `out/*.json` writes buy a worker extra tolerance before the session is classified as pure shell churn.
The shared worker contract now also pushes workers away from broad manifest dumps and ad hoc `jq` schedulers: open the repo-written current batch/task surface or the other prompt-named files directly, and if a local helper is needed, prefer a short targeted `python3` or one-off query against that named file. Worker roots can now also mirror a repo-written `tools/` directory plus `OUTPUT_CONTRACT.md` and `examples/`, and the knowledge stage uses that paved road directly via `tools/knowledge_worker.py`.
Direct telemetry now records command-policy classifications per observed command so the next bad run can show the difference between helpful local file reads and true workspace drift. Recipe-specific rereads now surface distinctly too: contract/example/helper-source lookups classify as `recipe_contract_lookup_command`, and hint+input+draft reread bundles classify as `recipe_task_bundle_read_command`.
Shard-v1 runtime is now mixed by attempt type rather than by stage: recipe, canonical line-role, and knowledge all use the workspace-worker mode for their main attempt on every worker assignment, while stage-specific retry / repair follow-up attempts can still use structured calls.
When shard-v1 worker counts are left unset, recipe, knowledge, and line-role now default to the planned shard/job count for that book+phase, capped at `20`; explicit `*_worker_count` overrides still win.
The shared phase-worker runtime now also launches assigned workers concurrently instead of iterating worker assignments in one serial loop. Recipe, knowledge, and line-role direct runtimes all execute their assigned workers concurrently, so `active workers: N` corresponds to real in-flight worker assignments rather than preplanned buckets waiting their turn.
Shard-v1 recipe, line-role, and knowledge planning all use `*_prompt_target_count` as operator-facing sizing controls. For knowledge, hard packet caps define a minimum packet floor, and the live planner now keeps one task per packet rather than collapsing multiple packets into one lossy shard when the requested target is lower.
Recipe no longer uses path-mode handoff under CodexFarm process. `codex_farm_orchestrator.py` now gives every recipe worker assignment a shared `prompt.txt`, `assigned_shards.json`, worker-local `assigned_tasks.json`, and authoritative `in/*.json`, then harvests `out/*.json` per task from one long-lived Codex worker session before deterministically joining those task outputs back into shard proposals. Recipe recovery is now same-session-first: repo code rewrites `CURRENT_TASK_FEEDBACK.md` for the same active task, the worker is expected to loop on `check-current` and `install-current`, and `finalize-all scratch/` is only a happy-path shortcut when the whole prepared batch is already clean. Task roots now write `same_session_fix_status.json`, and structured repair is only the bounded fallback after same-session recovery exhausts its local budget or the worker session stops in a way that makes continuation impossible. Canonical line-role now uses the same main-attempt transport shape, but its worker-control files are slimmer than before: `assigned_tasks.json` and `current_task.json` are metadata-first, repo code also writes a cheap `CURRENT_TASK.md`, and the normal loop is now current-task-brief -> scratch draft -> hint, with `in/*.json` as fallback authority instead of the ordinary first read. Knowledge now matches that main-attempt transport too, but the queue contract is batch-shaped: every assignment writes a skinny `assigned_tasks.json`, repo-owned `current_batch.json`, `CURRENT_BATCH.md`, `CURRENT_BATCH_FEEDBACK.md`, compatibility `CURRENT_TASK*` fallback files for the first active task, `OUTPUT_CONTRACT.md`, `examples/`, `tools/knowledge_worker.py`, authoritative `in/*.json`, and task-local `out/*.json` under one long-lived worker session. Each planned knowledge task now owns one packet id, and the helper CLI talks about packet ids, block decisions, and idea groups rather than chunk-result wrappers or cue-taxonomy hints. Repo code live-validates task-local outputs, advances by contiguous valid prefix inside each batch, mirrors the advanced batch bundle back into the sterile execution workspace, and tells the worker to reopen the batch brief/feedback after `install-batch` instead of treating one installed task as the end of the session. Queue completion is now the success authority for this stage: if stable files exist but the repo-owned queue controller cannot prove completion, the run lands as `workspace_validated_task_queue_incomplete` instead of a successful `workspace_outputs_stabilized` finish.
`cf-debug preview-prompts` is predictive-only now: it prefers the paired `vanilla` benchmark artifact when one exists, rebuilds planned shard payloads locally, tokenizes reconstructable prompt/output structure, reports stages as unavailable instead of faking prompt-length guesses when structure is missing, does not depend on saved live payload copies under `recipe_phase_runtime/inputs` or `knowledge/in`, and hard-refuses Codex-backed or ambiguous processed runs. Retrospective costs now live under the post-run artifact surface instead of preview.

Run settings now include shard-v1 pipeline ids plus optional shard worker/size/turn knobs, and they reject removed recipe/knowledge/line-role ids instead of normalizing them.
Run settings also include the optional workspace override (`codex_farm_workspace_root`) so recipeimport can target external codex-farm pipeline packs without code edits.
RecipeImport now also forces its CodexFarm subprocesses onto the dedicated `~/.codex-recipe` home by default at the runner layer, so recipe/knowledge/line-role/prelabel/model-discovery calls do not silently fall back to the main `~/.codex` profile. Explicit subprocess env overrides can still replace that home when needed.
Direct `codex exec` shard calls now add a second isolation layer too: the live subprocess `cwd` / `--cd` is no longer the repo-owned worker artifact folder under `data/output/...`. Instead, `codex_exec_runner.py` mirrors only the local worker inputs into a sterile workspace under `~/.codex-recipe/recipeimport-direct-exec-workspaces/`, writes a task-local `AGENTS.md` and `worker_manifest.json` there, rewrites any worker-local prompt paths to that mirrored location, and launches Codex from the sterile workspace. Because that workspace lives outside the git repo, the runner also passes `--skip-git-repo-check`. Repo worker folders remain the durable artifact surface; the sterile workspace exists only to control instruction discovery and local file visibility during the live shard call.
Strict-JSON watchdog kills now also preserve the offending command text in `reason_detail`, which makes it obvious when a worker was stopped for a trivial `ls` / `pwd` orientation reflex versus a more substantive off-contract command.
The fake subprocess seam in `scripts/fake-codex-farm.py` now mirrors that worker contract closely enough for zero-token tests: normal structured exec still returns one final JSON message, while workspace-worker exec reads local `assigned_shards.json` + `in/*.json` files and writes `out/*.json` files before emitting its completion event stream.
See `RECIPE_CODEX_HOME.md` in this folder for the AI-coder explainer on how RecipeImport forces CodexFarm onto the dedicated recipe Codex home.

Codex Farm model discovery for interactive run settings now comes from `codex-farm models list --json`, and subprocess orchestration validates configured pipeline ids via `codex-farm pipelines list --root ... --json` before recipe-correction execution.

When `codex-farm process --json` fails and returns a `run_id`, runner errors now include a follow-up summary from `codex-farm run errors --run-id ... --json`.
If that summary reports `no last agent message` (typically `nonzero_exit_no_payload` chunk failures), the runner now treats it as recoverable partial-output mode so orchestrators can continue and mark only affected bundles as missing.

Runner process metadata now surfaces CodexFarm `process --json.telemetry_report` as `telemetry_report`, best-effort `run autotune --json` output as `autotune_report`, and keeps `codex_exec_activity.csv` slices as `telemetry`; recipe/knowledge/tags manifests persist all three for prompt-tuning analysis.

For zero-token runtime rehearsal, point `--codex-farm-cmd` at `scripts/fake-codex-farm.py` and still run execute mode with `--allow-codex`. That path exercises the real shard-runtime input/output folder choreography through the subprocess runner without calling a live model.

When a caller provides a progress callback, `SubprocessCodexFarmRunner` requires `codex-farm process --progress-events --json`, parses `__codex_farm_progress__` stderr JSON events, and forwards spinner-friendly `task X/Y` status messages through the existing callback channel used by stage + benchmark flows.
The runner also suppresses legacy `run=... queued=... running=...` stderr snapshots when they are just redundant progress noise beside the structured progress channel.
`phase_worker_runtime.py` now emits repo-owned structured outer progress snapshots too, so recipe shard work reports configured workers, queued shards, and active worker buckets through the same callback channel the CLI spinner already understands.

Canonical line-role fallback support now lives in `canonical_line_role_prompt.py` plus direct-exec orchestration in `parsing/canonical_line_roles.py` and Codex-backed prelabel adapters in `labelstudio/prelabel.py` (`prelabel.freeform.v1`).

Prediction runs now also write `prediction-run/prompt_budget_summary.json`, which is the post-run actual-costs artifact: it merges recipe, knowledge, and line-role telemetry into one repo-owned whole-run budget summary, including nested line-role batch/attempt telemetry when the top-level benchmark copy only has metadata. Run manifests now also expose that file under `actual_costs_json`.

Prompt artifact export now lives in `prompt_artifacts.py`. It has a descriptor boundary:
- `discover_codexfarm_prompt_run_descriptors(...)` adapts current raw CodexFarm layout.
- the discoverer also follows benchmark `run_manifest.json` pointers like `stage_run_dir` / `processed_output_run_dir` when the prompt request starts from an eval root instead of the real stage run root.
- prompt export also supplements recipe/knowledge rows with copied live `line-role-pipeline/prompts/*` artifacts plus `line_role` rows in `prompts/full_prompt_log.jsonl` when a processed run has canonical line-role Codex interactions.
- prompt export also writes `prompts/thinking_trace_summary.jsonl` and `prompts/thinking_trace_summary.md` from the final merged prompt log so reviewers can inspect trace coverage without chasing raw `.trace.json` paths under `data/output`.
- `discover_prompt_run_descriptors(...)` is a pluggable dispatcher so future stage/cookbook layouts can provide new discovery adapters without changing rendering.
- `render_prompt_artifacts_from_descriptors(...)` writes `prompts/` artifacts from normalized stage descriptors.
- `build_prompt_response_log(...)` is the topology-neutral builder that accepts either explicit descriptors or injected discoverers.
- `build_codex_farm_prompt_response_log(...)` is the convenience wrapper used by CLI call sites.
- `prompt_preview.py` is the zero-token existing-output helper used by `cf-debug preview-prompts`; it rebuilds prompt previews from saved stage artifacts without invoking Codex.

Compact recipe prompt variants now live behind explicit pipeline ids (`recipe.correction.compact.v1` and `recipe.knowledge.packet.v1`), and those ids are now the default recipe-correction / knowledge-stage selections when CodexFarm recipe parsing or optional knowledge extraction is enabled. Line-role prompt compaction is controlled locally by `COOKIMPORT_LINE_ROLE_PROMPT_FORMAT=compact_v1`, which now also becomes the default when unset.

The non-recipe knowledge review stage now reviews the review-eligible Stage 7 non-recipe spans from `cookimport/staging/nonrecipe_stage.py`, plans explicit packet-owned shard entries in `codex_farm_knowledge_jobs.py`, executes each shard through `codex_exec_runner.py`, validates exact owned block coverage plus in-surface block evidence in `codex_farm_knowledge_ingest.py`, and then merges surviving `block_decisions` back into final `knowledge` versus `other` authority. Prompt preview now rebuilds knowledge payloads from that same review-eligible span set so preview shard counts match the live runtime's actual work set. Validation is now semantic as well as structural: useful rows must carry both `knowledge` decisions and snippets, non-useful rows must still cover every owned block with `other`, and packet-local idea groups must cover every kept block exactly once. Missing or semantically rejected shards remain seed-kept, but manifests now report them as partial/unreviewed review instead of successful reviewed-empty output. The live model-facing bundle is intentionally compact: top-level `v` / `bid` / `b`, optional `x` packet context, and optional `g` recipe guardrails. Reviewer artifacts still land under `knowledge/<workbook_slug>/snippets.jsonl` and `knowledge.md`, while runtime artifacts now also live under `raw/llm/<workbook_slug>/knowledge/` as `phase_manifest.json`, `shard_manifest.jsonl`, `worker_assignments.json`, `promotion_report.json`, `telemetry.json`, `failures.json`, per-shard proposals, and per-shard `prompt.txt` / `events.jsonl` / `usage.json` files under `workers/<worker>/shards/<shard>/`.
Knowledge workspace-worker prompts now restate the semantic packet contract directly, show one short good-snippet pattern, and show one explicit bad pattern for packet/prose echoing. Knowledge workspace sessions also stop themselves once every assigned `out/<task_id>.json` file exists and stabilizes, so a finished worker does not keep spending commands on post-write hygiene checks. Strict JSON knowledge retries now classify the latest agent message as absent, malformed, or JSON-shaped; malformed pseudo-finals are killed promptly instead of suppressing the other watchdog guards. The deterministic salvage seam is intentionally narrow: it trims literal trailing `EOF` and obvious shell-wrapper suffix noise after a valid JSON object, but otherwise fails closed. Low-trust semantic packet candidates now also fail closed when snippets are non-grounded, effectively dump the full owned packet text, or substantially copy one evidence quote verbatim. Snippet-copy-only failures no longer count as an immediate poisoned-worker skip class: the stage first tries one structured snippet-body-only repair that preserves packet ids, block decisions, and evidence pointers. Repeated uniform malformed / low-trust / watchdog-boundary failures can still classify a worker as poisoned so later retry or repair attempts record explicit skip reasons instead of spawning another repair storm. Monolithic watchdog retries are still refused for oversized single-task shard follow-ups with reason code `watchdog_retry_oversized_skipped`, but the common packet-owned parent-shard case now recovers at task scope instead of restating the whole shard inline. Strict JSON knowledge watchdog retries are still bounded twice: they carry a hard subprocess timeout and a shorter no-activity timeout so silent retries cannot hold the benchmark open indefinitely.

Knowledge planning is now packet-based. Deterministic code still supplies bounded ordered packet inputs, but the worker owns the final `knowledge|other` decision and related-idea grouping inside each packet.

The live knowledge phase now runs on `recipe.knowledge.packet.v1`. The important cutover is that shard ownership, worker reuse, proposal validation, and fallback behavior are now repo-owned instead of being implicit in one direct bundle pass. `raw/llm/<workbook_slug>/knowledge/in/` is the immutable shard-input seam, and `raw/llm/<workbook_slug>/knowledge/proposals/` is the validated proposal seam.
`knowledge_runtime_state.py` now defines the repo-owned packet-ledger vocabulary for this stage, and `knowledge_runtime_replay.py` is the read-only replay helper that rebuilds packet/follow-up/worker/missing-artifact summaries from a saved knowledge root plus its paired benchmark root without spending tokens.

The canonical recipe setting id is now `llm_recipe_pipeline=codex-recipe-shard-v1`. The live recipe phase now groups nearby recipes into explicit shard payloads, splits those shards into recipe-local task files, executes those tasks through direct `codex exec` using the compact pack id `recipe.correction.compact.v1`, validates exact owned `recipe_id` coverage plus exact compact output keys, and then promotes validated per-task outputs back into deterministic final assembly. Recipe worker roots now carry `SHARD_PACKET.md`, `OUTPUT_CONTRACT.md`, `examples/valid_{repaired,fragmentary,not_a_recipe}_task_output.json`, `tools/recipe_worker.py`, repo-written `CURRENT_TASK.md`, and `CURRENT_TASK_FEEDBACK.md`; the repo also prewrites `scratch/<task_id>.json` drafts plus `scratch/_prepared_drafts.json` before the worker starts. The default recipe loop is now same-session and active-task-first: open `SHARD_PACKET.md`, then the current-task sidecars, edit the active `scratch_draft_path`, run `python3 tools/recipe_worker.py check-current`, and keep fixing that same draft until validation is clean before `install-current` advances the queue. `finalize-all scratch/` still exists, but only as a happy-path shortcut when the whole prepared batch is already clean. `assigned_tasks.json` remains present as fallback/debug inventory only; it is not a normal startup dump surface. `hint_path` and `input_path` remain fallback surfaces when the shard packet, current-task sidecars, and active draft are insufficient, while `OUTPUT_CONTRACT.md`, `examples/*.json`, and `tools/recipe_worker.py` remain fallback contract/debug surfaces rather than the normal loop. `current`, `next`, `overview`, `show`, `prepare-all`, and `finalize` stay available as fallback/debug verbs, and `stamp-status` can bulk-mark prepared drafts as `fragmentary` or `not_a_recipe`. Recipe task rows now also expose `metadata.input_path`, `metadata.hint_path`, `metadata.scratch_draft_path`, and `metadata.result_path` so the helper can install only to the declared `out/<task_id>.json` target. The prepared-drafts manifest now stays compact by keeping only draft/output paths, owned ids, short title hints, and the current task id, and `finalize-all` now rewrites the queue sidecars immediately to the completed state. Prepared recipe drafts now prewrite deterministic empty-mapping reasons when possible and fail closed to `fragmentary` when the source hints are too weak for a schema-valid repaired object. Retryable recipe watchdog kills are now split by failure shape: mixed-output misses can still recover per task, while early whole-shard deaths now use one shard-packed structured retry and then split the validated shard payload back into task-local rows. If a task fails validation and the worker session stops before rewriting it, repo code records the loss of same-session continuation explicitly and only then falls back to bounded structured repair. The recipe watchdog also now reasons over both the mirrored worker root and the sterile direct-exec root, so startup shell scripts under `~/.codex-recipe/...` classify the same way in live kills and in telemetry. Legacy worker keys such as `results`, `recipe_id`, `repair_status`, `canonical_recipe`, `not_a_recipe`, `fragmentary`, and `notes` are intentionally rejected on the runtime seam even though the internal Pydantic models still support verbose field names for repo-side normalization helpers. `raw/llm/<workbook_slug>/recipe_phase_runtime/inputs/` is the immutable shard-input seam, worker-local `workers/*/out/*.json` validity is the authoritative main-attempt success signal, `raw/llm/<workbook_slug>/recipe_phase_runtime/proposals/` is the validated proposal seam, `task_manifest.jsonl` plus worker-local `assigned_tasks.json` are the task-loop seam, and `recipe_correction_audit/` remains the per-recipe human/debug summary surface. Prose or markdown closing summaries from a workspace worker are telemetry only now; they no longer poison an otherwise valid file-backed task result.

Current recipe-object transport note: the correction loader uses native nested objects for `canonical_recipe`. Fake-runner outputs follow the same contract.

Current recipe-correction hardening note: manifest rows expose `final_mapping_status` / `final_mapping_reason` for the deterministic final assembly that consumes `ingredient_step_mapping`. Repaired outputs with 2+ ingredients or 2+ steps now fail closed when `ingredient_step_mapping` is empty and `ingredient_step_mapping_reason` is blank; they must provide a reason token such as `unclear_alignment` or a real mapping.

Default behavior remains deterministic unless a non-`off` recipe Codex pipeline is explicitly enabled.
