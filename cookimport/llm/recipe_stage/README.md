Recipe-stage owner map.

Open `planning.py`, `runtime.py`, `validation.py`, `promotion.py`, `recovery.py`, `reporting.py`, `task_file_contract.py`, or `worker_io.py` for the specific question you are changing; `cookimport/llm/recipe_stage_shared.py` is now the shrinking runtime coordinator instead of the only owner. `task_file_contract.py` owns the worker-visible payload shape and answer-schema descriptor. `worker_io.py` owns prompt/jsonl/input writing and the local path helpers used by recipe runtime. Recipe `task.json` is now slimmer: it keeps the stage identity, units, and `answer_schema`, but no longer serializes `helper_commands`, `workflow`, or `next_action`. The live helper surface is still wrapper-first in the prompt and workspace root: `task-summary`, `task-show-unit`, optional `task-template` plus `task-apply`, then `task-handoff`, with `task-status` and `task-doctor` available from the workspace root. The runner also injects guarded `cat`/`ls`/`python3`/`python` shims that redirect mild drift back to those helpers. Helper modes auto-discover hidden state from `./_repo_control/`, and repair rewrites preserve the same compact task-file shape. Recipe runtime may also spend one bounded fresh-session recovery attempt when a clean first session preserved useful state without completing; that recovery is separate from watchdog policy. Catastrophic first-session failures such as watchdog kills or retryable worker-process timeouts can now also spend one bounded fresh-worker replacement attempt that resets the assignment back to the original repo-authored task/state before trying again. Shared retry/replacement budgets now live in `cookimport/llm/repair_recovery_policy.py`.
