# Bench Module

Offline benchmark orchestration code lives in this folder.
Durable benchmark scoring/scheduler/output contracts live in `cookimport/bench/CONVENTIONS.md`.
Agent quick-start for QualitySuite in this folder lives in `cookimport/bench/AGENTS.md`.

Current scoring contract:
- Predictions come from stage evidence manifests (`stage_block_predictions.json`), not pipeline chunk tasks.
- Gold can include multiple labels per block; eval accepts any matching gold label and logs multi-label diagnostics to `gold_conflicts.jsonl`.
- Missing gold rows for predicted blocks are defaulted to `OTHER` and logged in `gold_conflicts.jsonl`.
- Canonical-text eval supports optional disk-backed alignment reuse (`cookimport/bench/canonical_alignment_cache.py`); all-method runs share a per-source cache at `.cache/canonical_alignment`.
- Canonical-text eval uses `COOKIMPORT_BENCHMARK_SEQUENCE_MATCHER=dmp` only; any non-`dmp` value is rejected and missing `fast-diff-match-patch` fails runtime selection.
- Run-level prediction-stage replay uses `cookimport/bench/prediction_records.py` (`PredictionRecord` JSONL schema v1) for `labelstudio-benchmark --predictions-out/--predictions-in`.
- Public `labelstudio-benchmark` runs always use pipelined execution; all-method orchestration keeps a private skip-evaluation prediction pass for prediction-record generation/reuse.
- `bench eval-stage` evaluates existing stage outputs against freeform gold and writes stage-block diagnostics (`eval_report.*`, `missed_gold_blocks.jsonl`, `wrong_label_blocks.jsonl`, boundary mismatch JSONL artifacts).
- canonical-text eval outputs include `aligned_prediction_blocks.jsonl` so DMP alignment mappings can be audited directly.
- line-role Milestone-5 diagnostics now live in `cutdown_export.py`, `pairwise_flips.py`, and `slice_metrics.py`; canonical line-role runs emit joined-line tables, flips, slice/routing metrics, and stable sampled cutdown JSONLs keyed by one `sample_id` source table.
- `line-role-pipeline/joined_line_table.jsonl` is intentionally conservative: it only attaches candidate-label / `decided_by` / recipe-span metadata when a line-role prediction can be matched to the canonical line by exact text (same index+text first, then exact-text sequence alignment). Split/merged or ambiguous duplicate lines stay unmatched instead of borrowing another line's telemetry.
- `followup_bundle.py` now powers the standalone `cf-debug` CLI for deterministic follow-up work on `upload_bundle_v1/`. The intended top-level loop is `upload_bundle_v1 -> web AI follow-up request manifest -> followup_dataN/` packet; lower-level selector/export/audit commands remain available under the same module. Request templates now pick a real bundle-local example case when possible and fall back to empty selectors for sparse bundles instead of hardcoding a case ID that may not exist.
- Multi-book single-profile `upload_bundle_v1` group bundles are now curated first-pass packets: target is ~30MB, raw prompt logs/full-context trace dumps stay local for follow-up, and `upload_bundle_index.json` records the actual final bundle bytes plus any trimmed paths.
- Single-book benchmark `upload_bundle_v1` now defaults to that same curated first-pass shape for upload use: target is ~30MB instead of a full 100MB+ lossless dump, and the full evidence path is expected to flow through follow-up packets rather than turn-1 bundle bulk.
- `upload_bundle_v1` generation now has explicit flexibility seams: `upload_bundle_v1_model.py` defines the normalized source model, `upload_bundle_v1_existing_output.py` adapts current benchmark roots into that model, and `upload_bundle_v1_render.py` hosts topology-tolerant rendering helpers used by the script wrapper.
- Recipe-stage topology now flows through that same model seam: `analysis.recipe_pipeline_context` and `analysis.stage_separated_comparison` render one semantic recipe-stage contract from `UploadBundleSourceModel.topology` (`recipe_topology_key` plus ordered `recipe_stages`) instead of old pass-slot fields.
- Knowledge follow-up selectors accept either the historical book slug or the bundle's exact `source_key`; sparse single-profile bundles do not need a codex-enabled run just to resolve `--include-knowledge-source-key`.
- Deterministic speed regression tooling lives in:
  - `speed_suite.py` (`bench speed-discover` target discovery from pulled gold exports)
  - `speed_runner.py` (`bench speed-run` repeated stage/benchmark timing samples, including optional `benchmark_all_method_multi_source`)
  - `speed_compare.py` (`bench speed-compare` baseline-vs-candidate regression gating)
- Deterministic quality regression tooling now mirrors the speed loop:
  - `quality_suite.py` (`bench quality-discover` defaults to curated CUTDOWN target IDs: `saltfatacidheatcutdown`, `thefoodlabcutdown`, `seaandsmokecutdown`, `dinnerfor2cutdown`, `roastchickenandotherstoriescutdown`; falls back to representative stratified selection when unavailable, and retries filename matching when importer-scored discovery is empty)
  - `quality_runner.py` (`bench quality-run` supports adaptive parallel experiment execution with persistent canonical/eval + prediction-reuse caches under `data/golden/bench/quality/.cache`; race mode auto-falls back to exhaustive when finalists cannot prune the current variant set; gentle output-write pacing is enabled by default to reduce WSL disk I/O spikes and can be disabled via `--io-pace-every-writes 0` or `--io-pace-sleep-ms 0`)
  - `quality_compare.py` (`bench quality-compare` baseline-vs-candidate quality gating)
  - `quality_lightweight_series.py` (`bench quality-lightweight-series` main-effects-first orchestration: category screening, combined winner check, and interaction smoke variants with resume-compatible fold artifacts)
- Benchmark retention/GC tooling lives in `artifact_gc.py` and is surfaced via `cookimport bench gc` (dry-run by default, `--apply` for destructive pruning; quality/speed roots under `data/golden/bench/*` still require durable CSV confirmation; label-studio benchmark roots under `data/golden/benchmark-vs-golden/*` now keep the newest five by default; timestamped run roots directly under `data/output/*` are wiped by default while non-run folders such as `history/dashboard` are preserved; any run root containing `.gc_keep*`/`.keep`/`.pinned` is never pruned; GC never rewrites/prunes `performance_history.csv`).
- `labelstudio-benchmark` auto-prunes only transient benchmark slop (excluded gate/gated/smoke/test/debug/quick/probe/sample/trial/regression suffix runs and `/bench/`-scoped benchmark artifacts), and keeps normal interactive run outputs; matching processed-output roots are pruned only for excluded runs.
- Interactive `C3imp` menu sessions force prune suppression for benchmark evals; interactive outputs are retained unless you run an explicit cleanup command later.
- Interactive single-book and single-profile benchmark runs still honor the selected top-tier profile family, but Codex-backed benchmark selections expand into a paired `vanilla` then `codexfarm` benchmark sequence so same-run comparisons are available.
- Interactive multi-book single-profile progress now renders as a shared book grid: one column per selected book, per-book state/progress/ETA rows, and one worker row per active slot.
- Unified operator decision path is documented in `docs/07-bench/qualitysuite-product-suite.md`.
- Oracle upload is a post-bundle flow in `oracle_upload.py`: interactive benchmark runs now auto-start two detached Oracle review lanes by default after writing `upload_bundle_v1`: `quality` (push benchmark quality toward `>95%`) and `token` (cut spend without giving back current quality). Each lane gets its own launch dir under `upload_bundle_v1/.oracle_upload_runs/<timestamp>-<profile>/`, its own `oracle_upload.log`, `oracle_upload.json`, `oracle_upload_status.json`, and its own staged lane brief plus path-selected `upload_bundle_payload.jsonl` subset. The wrapper records Oracle version, status, session id, `oracle session <id>` reattach command, review profile, and conversation URL either from the log or from Oracle's own session store under `~/.local/share/oracle/sessions/`, and it classifies saved logs as `running`, `succeeded`, `invalid_grounding`, `reattachable`, or `failed` instead of trusting exit code alone. Detached benchmark runs also start a local `oracle_auto_followup` worker beside each turn-1 lane run; it finalizes the saved turn-1 status from the completed Oracle log, writes `oracle_auto_followup.json` plus `oracle_auto_followup.log`, and if Oracle requested follow-up data it builds `followup_data1/` and sends turn 2 into the same chat automatically. `cookimport bench oracle-upload <existing root>` remains the manual retry path, now with `--profile quality|token|all` (`all` default), and foreground browser runs persist that same per-lane `.oracle_upload_runs/<timestamp>-<profile>/` artifact set instead of only printing terminal output. `--mode dry-run` stays zero-cost: it uses Oracle's real dry-run for small bundles and falls back to a local preview when the payload file exceeds Oracle's inline size cap. Browser-mode uploads auto-shard oversized bundle files into Oracle-safe staged attachments before launching, pass hidden Oracle recovery flags (`--browser-reuse-wait`, `--browser-profile-lock-timeout`, `--browser-auto-reattach-*`), optionally honor `COOKIMPORT_ORACLE_CHATGPT_URL` for a dedicated ChatGPT project URL, call the canonical local Oracle wrapper at `/home/mcnal/.local/bin/oracle`, and therefore pick up the editable `oracle-dev/package` mirror plus the machine-wide Chromium launcher (`/home/mcnal/.local/bin/chromium-oracle-auto`) before falling back to the global install. Browser uploads still use the canonical Oracle browser profile at `~/.local/share/oracle/browser-profile` with `~/.oracle/browser-profile` kept only as a compatibility symlink, pass `--browser-model-strategy select` so the requested benchmark review model is explicitly selected instead of inheriting the current browser mode, and default to `ORACLE_GENUINE_MODEL` with a `gpt-5-pro` fallback unless `--model` overrides it. `ORACLE_TEST_MODEL` is the separate Instant-backed smoke/test lane; `ORACLE_PRO_MODEL`, `ORACLE_REVIEW_MODEL`, and `ORACLE_DEEP_REVIEW_MODEL` remain accepted as compatibility aliases for the genuine-run lane. Thinking is intentionally not part of the local default/operator policy. Benchmark-owned Oracle browser turn 1 uploads and turn 2 `continue-session` calls now rely on Oracle's default browser-lifecycle behavior instead of passing a negated keep-browser flag, because the installed Oracle CLI only accepts the positive `--browser-keep-browser` form. The local Oracle `0.8.6` install also needs the repo-verified composer patch in `/home/mcnal/.nvm/versions/node/v20.19.6/lib/node_modules/@steipete/oracle/dist/src/browser/actions/promptComposer.js` so browser uploads clear the composer before inserting the benchmark prompt.
- Turn-1 timeout recovery now writes any reattached answer back into the source `oracle_upload.log` before the follow-up worker re-audits it, and automatic turn 2 can continue from an `invalid_grounding` first answer when Oracle asked for more data to resolve the mismatch.
- Oversized browser uploads now try a compact first-pass starter packet before full sharding: if `upload_bundle_index.json` exposes `root_files`, `starter_pack`, and `per_run_summaries` row locators, `oracle_upload.py` trims `upload_bundle_payload.jsonl` down to just those rows for turn 1 and tells Oracle that attached `path` values are real while `payload_row` numbers still refer to the full local bundle. Full payload sharding is now only the fallback when that compact packet cannot be built.
- With `--browser-bundle-files`, Oracle may show ChatGPT a single synthetic attachment such as `attachments-bundle.txt` even though the logical bundle still consists of `upload_bundle_overview.md`, `upload_bundle_index.json`, and payload shard files. Benchmark prompts now call that out explicitly so the instructions match the actual UI attachment shape.
- The post-benchmark Oracle review prompts are stored in `llm_pipelines/prompts/benchmark.oracle-upload.prompt.md` (`quality`) and `llm_pipelines/prompts/benchmark.oracle-upload.token.prompt.md` (`token`). Edit those files for text-only prompt changes and keep `{{BUNDLE_SCOPE}}`, `{{BENCHMARK_ROOT}}`, and `{{LANE_BRIEF_FILE}}` so runtime can fill in the current target and staged lane brief.
- The quality prompt explicitly frames Oracle as an accuracy-improvement aid, while the token prompt frames Oracle as a spend-reduction reviewer with quality guardrails. Both require a parse-friendly `Requested follow-up data` section so the existing local follow-up bridge can keep working.
- `cookimport bench oracle-followup <existing root>` is still the turn-2 bridge and repair seam: it reads a saved Oracle answer from `.oracle_upload_runs/<timestamp>/oracle_upload.log`, normalizes the `Requested follow-up data` block into `cf.followup_request.v1`, writes `codex_followup_handoff.md`, builds `followup_data1/`, renders the turn-2 prompt from `llm_pipelines/prompts/benchmark.oracle-followup.prompt.md`, and then calls Oracle's same-chat `continue-session` command unless `--dry-run` is set. Detached benchmark runs now invoke that same bridge automatically after a grounded turn-1 answer.
