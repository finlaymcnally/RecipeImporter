Private canonical line-role package. The live worker contract is now single-file and assignment-first: `planning.py` builds contiguous shard owners, `runtime.py` writes one editable `task.json` plus repo-owned debug artifacts, and `validation.py` expands validated answers back into shard `out/<shard>.json` outputs. Worker-visible line evidence is raw-text-first and stable-provenance-only; deterministic label hints and deterministic reason hints stay repo-side. The isolated single-file execution workspace still starts from just `task.json` plus `AGENTS.md`; repo-side `debug/`, `in/`, `out/`, and related worker-root artifacts remain observability surfaces, not the startup contract. `task.json` now also carries repo-authored `helper_commands`, `next_action`, and `answer_schema` metadata, workers should start with `python3 -m cookimport.llm.editable_task_file --summary` plus narrow `--show-unit` or `--show-unanswered` reads instead of dumping the full file, and that summary now samples large unanswered-unit lists instead of echoing every unit id. The same-session helper exposes read-only `--status` and `--doctor`, and the runtime may spend one bounded fresh-session recovery attempt when the first clean session preserved useful state without completing. Those helper modes auto-discover hidden state from `./_repo_control/`, and repair rewrites keep the same visible guidance metadata in the rewritten `task.json`. That restart is separate from watchdog policy. Workspace-worker supervision stays warning-first for ordinary drift, but line-role success is now authoritative when the repo-owned same-session state says `completed` and the expected `out/<shard>.json` files are present/stable. The watchdog keys that helper visibility off the runner’s most recent completed stage-helper command, not the last arbitrary command overall, so a later `editable_task_file --summary` read cannot resurrect `workspace_final_message_missing_output`. Final assistant prose is only telemetry on that happy path. The explicit fail-fast seam after a final agent message remains as fallback-only behavior: if the worker says it is done and the repo still cannot prove helper-complete success after the configured grace window, runtime records a deterministic recovery assessment, may spend the one shared fresh-session retry budget on the preserved workspace, and otherwise fails closed with explicit missing-output metadata instead of grinding until context exhaustion. Same-session worker repair is still the primary recovery path, retryable watchdog kills may still get one bounded structured retry, and otherwise the shard fails closed. Deterministic fallback is not final authority on the live worker path.
