Private canonical line-role package. The live worker contract is single-file, assignment-first, and now direct-batch: `planning.py` builds contiguous shard owners, `runtime_taskfile.py` owns task-file payload construction plus output expansion/incomplete-run surfacing, `runtime_watchdog.py` owns the direct-exec preflight/watchdog/retry policy helpers, `runtime_workers.py` owns the remaining taskfile-assignment, structured-assignment, direct-worker orchestration, worker-runner payload aggregation, inline prompt/repair packet shaping, and telemetry-summary bands, `runtime.py` keeps the coordinator/root wrapper seam plus final projection, `runtime_recovery.py` owns same-session recovery and workspace-status assessment, and `validation.py` expands validated answers back into shard `out/<shard>.json` outputs. Inline JSON now uses the shared line-role contract plus shard-profile evidence, asks for one ordered `labels` array instead of echoed row ids, and no longer semantically vetoes structurally valid model output just because repo heuristics think a shard looks like front matter or contents. Deterministic code owns the row bookkeeping on that inline path: it zips returned labels back onto shard order, accepts unambiguous packet-local ordinal row replies as a first-class inline shape, trims the one obvious trailing context-row spill, and narrows repair packets to only unresolved owned rows. If rows still remain unresolved after repair/retry, the shard fails closed instead of being padded out from the deterministic baseline. Active pre-grouping line-role candidates are now sanitized before the runtime does anything interesting with them: stale `recipe_id`, `within_recipe_span`, and span-derived rule tags are stripped so cache/debug/preview surfaces stay raw-text-first until recipe grouping happens later. Workers should open `task.json` directly, read the full shard in order, fill every `answer.label` in place, run `task-handoff`, and reopen the rewritten file only if repair feedback appears. `task-status` and `task-doctor` remain optional troubleshooting helpers, while `task-show-unit` and `task-show-unanswered` are fallback-only helpers for local ambiguity, not the default path. The visible execution workspace is still just `task.json` plus `AGENTS.md`; repo-side `debug/`, `in/`, `out/`, and related worker-root artifacts remain observability surfaces, not the startup contract. The task file now advertises editability through `answer_schema.editable_pointer_pattern` instead of `editable_json_pointers`, and line-role no longer uses `answers.json`, `task-template`, or `task-apply` on the happy path. Ordinary local reads of `task.json` and `AGENTS.md` are allowed, while real boundary escapes still fail closed. One bounded fresh-session recovery attempt and one bounded fresh-worker replacement attempt remain available when repo-owned state shows recoverable progress. The runtime root now also writes `phase_plan.json` and `phase_plan_summary.json`, and worker `status.json`, stage `telemetry.json`, and `line_role_stage_summary.json` all expose the active shared `repair_recovery_policy` rows with allowed-versus-spent counts. For inline line-role specifically, that shared policy row is now one structured repair follow-up plus one watchdog retry budget, with shard status metadata also carrying the actual transport so summaries can tell inline from taskfile runs. `planning.py`, `policy.py`, and `validation.py` now import their dependencies explicitly instead of cloning the package runtime namespace. Runtime-level patchability for tests still goes through explicit package/root hooks rather than copied package globals. Start with `contracts.py` for the prediction model/normalization helpers, `prompt_inputs.py` for row serialization, `artifacts.py` for runtime/prompt artifact helpers, `planning.py`, `policy.py`, `validation.py`, `runtime_taskfile.py`, `runtime_watchdog.py`, `runtime_workers.py`, and `runtime.py`. `__init__.py` is the small package seam.

Change map:
- For shard planning, taskfile shape, or runtime recovery changes, start in the matching `runtime_*` owner or `planning.py`, read `docs/04-parsing/04-parsing_readme.md`, and run `pytest tests/parsing/test_canonical_line_roles_runtime.py tests/parsing/test_canonical_line_roles_taskfile.py -q`.
- For inline front-matter / contents regressions, start with `validation.py` and `runtime_workers.py`, then run `pytest tests/parsing/test_canonical_line_roles_front_matter_pathology.py -q`. Shard-profile evidence here is reviewer context only; it should not invalidate otherwise legal model labels.
- For row-level policy veto tuning, start with `policy.py` and run `pytest tests/parsing/test_canonical_line_roles_policy_tuning.py -q`.
- If the change touches direct worker behavior, also check `cookimport/llm/README.md` because line-role now shares the same direct-exec runtime family and watchdog policy shape.
