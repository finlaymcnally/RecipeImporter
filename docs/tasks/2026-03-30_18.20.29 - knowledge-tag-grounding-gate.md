Problem statement

The split knowledge classifier still treats cooking-adjacent prose as enough for `knowledge`. It needs a stricter retrieval-grade gate: every kept block must name a portable concept and ground it to the Cookbook tag ontology or a proposed new tag under an existing category.

Acceptance criteria

- The repo has a checked-in machine-readable Cookbook tag catalog plus a refresh script.
- Classification task files expose the ontology at task scope and per-unit `candidate_tag_keys` hints.
- `knowledge` answers require `retrieval_concept` plus valid grounding to existing `tag_keys` or `proposed_tags`.
- Final shard outputs, promotion/reporting artifacts, and reviewer-facing knowledge outputs preserve grounding metadata and proposal counts.
- The new grounding contract is locked by targeted loader, contract, prompt, promotion, and observability tests.

- `source .venv/bin/activate && pytest tests/llm/test_knowledge_grounding_contract.py -q`
- `source .venv/bin/activate && pytest tests/llm/test_knowledge_block_classification_contract.py -q`

Evidence

- Loader/script tests prove the checked-in ontology matches the SQL seed shape.
- Contract/promotion tests prove `knowledge` rows now carry retrieval concepts and validated grounding.
- Runtime/observability tests prove grounding metadata and proposal counts survive the real assignment-first worker path.

Constraints/gotchas + decision

- Deterministic code may package ontology evidence and validate catalog membership, but it must not semantically override the model’s keep/drop call.
- Proposed tags are sidecar review artifacts only; they do not mutate Cookbook data or become authority outside the validated stage outputs.

Rollback / compensation plan

- Revert the ontology catalog, knowledge contract/runtime, reporting, and docs together so the prior split-classification contract remains the last coherent state.
