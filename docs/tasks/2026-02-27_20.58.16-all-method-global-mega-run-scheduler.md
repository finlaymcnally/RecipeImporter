## Purpose / Big Picture

After this change, an all-matched all-method benchmark run is scheduled as one global work queue across all matched books and all generated configs, instead of one independent config scheduler per source. The practical user-visible result is that bulk runs stay interleaved across sources for longer, heavy slots are shared across the full run, and the endgame tail is less likely to collapse to one isolated source scheduler.

You can see this working by running an all-method benchmark over multiple matched gold sets and observing that active config workers include multiple sources in one scheduler stream, plus one global scheduler snapshot for the whole run. The run still writes per-source and combined reports, and scoring behavior remains canonical-text with no Codex Farm parsing enabled.

## Surprises & Discoveries

- Observation: Current all-method bulk runs launch multiple sources in parallel, but each source spins up an independent config scheduler and independent split-phase gate directory.
  Evidence: `_run_all_method_benchmark_multi_source(...)` calls `_run_all_method_benchmark(...)` per source job, and each source run creates `split_phase_gate_dir = root_output_dir / ".split_phase_slots"`.

- Observation: Current eval-signature dedupe is per-source invocation, not global across the whole bulk run.
  Evidence: `_run_all_method_benchmark(...)` computes signatures from its local `prediction_success_rows` and writes one per-source `all_method_benchmark_report.json`.

- Observation: Source scheduling strategy (`tail_pair`) and sharding improve source-level throughput, but they do not create one global queue.
  Evidence: `_plan_all_method_source_jobs(...)` plans source jobs, then `_run_all_method_benchmark_multi_source(...)` dispatches those jobs to a source thread pool.

- Observation: A user-configured `all_method_max_parallel_sources` cap can hide CPU capacity and make runs look less interleaved than expected.
  Evidence: `_resolve_all_method_source_parallelism(...)` enforces configured cap; discovery note `docs/understandings/2026-02-27_20.56.32-all-method-source-cap-via-cookimport-setting.md`.

- Observation: Legacy multi-source combined report payload could crash after adding additive global queue counters because `total_failed_config_runs` was referenced before assignment.
  Evidence: `NameError` surfaced in `tests/labelstudio/test_labelstudio_benchmark_helpers.py` legacy-path tests; fixed by computing `total_failed_config_runs` in `_run_all_method_benchmark_multi_source_legacy(...)`.

- Observation: Existing multi-source tests that monkeypatch `_run_all_method_benchmark(...)` implicitly depended on legacy dispatch and began exercising the new global path once default scope switched to `global`.
  Evidence: tests started executing real EPUB extraction instead of test doubles until calls explicitly passed `scheduler_scope=legacy`.

## Context and Orientation

Today, all-method bulk orchestration has two scheduler layers:

- Source layer: `_run_all_method_benchmark_multi_source(...)` in `cookimport/cli.py` dispatches source jobs.
- Per-source config layer: each source job calls `_run_all_method_benchmark(...)`, which runs configs and then eval-signature dedupe for that source.

“Global mega-run scheduler” in this plan means a single config scheduler loop that owns all configs from all matched sources in one queue. It does not mean changing parser behavior, evaluation math, labels, or all-method variant generation.

Key files for implementation:

- `cookimport/cli.py`
  - source planning helpers (`_plan_all_method_source_jobs`, sharding helpers)
  - current multi-source orchestrator (`_run_all_method_benchmark_multi_source`)
  - current per-source scheduler (`_run_all_method_benchmark`)
  - prediction/eval helpers (`_run_all_method_prediction_once`, `_run_all_method_evaluate_prediction_record_once`, signature helpers)
  - interactive settings + scheduler summary output
- `tests/labelstudio/test_labelstudio_benchmark_helpers.py`
  - scheduler, retries/timeouts, multi-source ordering, report-contract tests
- `docs/02-cli/02-cli_README.md`
- `docs/07-bench/07-bench_README.md`

## Plan of Work

Milestone 1 introduces run-wide planning objects. Add a global work-item dataclass in `cookimport/cli.py` that carries source identity, source output roots, source dashboard index, local config index, variant, and deterministic global dispatch index. Build this worklist by reusing `_plan_all_method_source_jobs(...)` and then flattening each source plan’s `variants` into individual global items.

Milestone 2 implements one global prediction scheduler loop. Add a new helper such as `_run_all_method_benchmark_global_queue(...)` that replaces per-source `_run_all_method_benchmark(...)` calls in bulk mode. This loop should use one `ProcessPoolExecutor`, one global split gate directory under the run root, and one global scheduler telemetry stream. Admission control should reuse the same heavy/wing/eval-headroom logic but compute counts across all active configs in the global queue.

Milestone 3 moves eval-signature dedupe scope from per-source to run-wide. After prediction phase completes (including retries/timeouts), group all successful prediction rows by eval signature across the entire run. Execute evaluation once per signature, then hydrate all grouped rows with reused metrics/provenance. Keep existing cache safety constraints and sequence-matcher semantics.

Milestone 4 restores per-source report outputs from global rows. Because downstream tooling expects per-source `all_method_benchmark_report.json`, generate source reports by grouping global rows by source group key and writing source report files under each source folder. Keep multi-source report shape stable and include additive fields that describe scheduler scope (`global` vs `legacy`) and global queue counters.

Milestone 5 wires rollout and rollback controls. Add a new settings key and menu item to choose scheduler implementation mode. Default should be global mode for all-method bulk runs after tests pass. Legacy mode must remain available for quick rollback and for A/B performance comparison.

Milestone 6 validates behavior and updates docs. Add tests that prove cross-source config interleaving, global cap enforcement, and report compatibility. Update bench and CLI docs with exact scheduler semantics and troubleshooting notes.

## Validation and Acceptance

Acceptance is behavior-based and must be demonstrated by tests plus one manual run.

1. Global queue behavior

- In all-matched mode, all planned configs are admitted from one global pending queue, not isolated per-source queues.
- Scheduler snapshots reflect one run-wide heavy/wing/eval count set.

2. Cross-source interleaving

- With at least three matched sources and source parallelism > 1, observed submission/completion order includes configs from multiple sources before any one source finishes all its configs.

3. Split-slot sharing

- A single split-phase slot gate is shared across the run so heavy-phase occupancy is bounded globally, not independently per source.

4. Run-wide eval dedupe

- Eval signatures are grouped run-wide and executed once per unique signature.
- Duplicate rows retain correct `evaluation_result_source` and representative references.

5. Contract compatibility

- Combined report keys required by dashboard/analytics remain present.
- Per-source report files still exist at expected paths and contain prior core metrics/counters.

6. Rollback safety

- Legacy scheduler mode can be enabled by settings and reproduces legacy behavior path without code changes.

## Idempotence and Recovery

All benchmark outputs remain timestamped and additive. Re-running after partial failure creates a new run root and does not destroy previous runs.

If global scheduler implementation fails in practice:

- switch scheduler mode to legacy in `cookimport.json` (new setting from this plan),
- rerun benchmark to recover production workflow,
- debug global mode using timestamped artifacts and scheduler timeseries.

Timeout and retry behavior should remain idempotent: failed config retries rerun failed items only, and successful config artifacts remain unchanged.
