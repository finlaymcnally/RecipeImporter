## Problem

Paired upload bundles were reconstructing summary runtime/cost totals correctly, but the exported starter-pack call inventory was thinner than the real codex prompt log.
`line_role` calls were being dropped from the row export, and row-level runtime/token/cost fields from `request_telemetry` were not being carried into the exported JSONL.

## Acceptance Criteria

- Starter-pack / upload-bundle call inventory rows retain `line_role` prompt rows when they exist in `full_prompt_log.jsonl`.
- Exported call inventory rows include row-level runtime/token/cost fields when `request_telemetry` provides them.
- Focused starter-pack and upload-bundle runtime regressions pass in `.venv`.

## Verification

- `. .venv/bin/activate && pytest tests/bench/test_benchmark_cutdown_for_external_ai.py -q -k "starter_pack_writes_recipe_triage_and_call_inventory or includes_line_role_rows_with_runtime_fields or realistic_codex_call_telemetry or call_rows_lack_runtime_signal"`
- `. .venv/bin/activate && pytest tests/bench/test_benchmark_cutdown_for_external_ai.py -q -k "call_runtime"`
- `. .venv/bin/activate && pytest tests/bench/test_benchmark_cutdown_for_external_ai.py -q -k upload_bundle`

## Evidence

- Added a starter-pack regression that writes a codex run with one `line_role` prompt row carrying telemetry and verifies the exported `02_call_inventory.jsonl` row keeps runtime/token fields.
- Extended the realistic upload-bundle runtime merge regression to assert the derived `02_call_inventory.jsonl` payload keeps the `line_role` row and its runtime fields.

## Constraints / Decision

- Keep the fix local to upload-bundle / starter-pack call inventory generation; do not change unrelated prompt-sampling or recipe-triage behavior.
- Treat `line_role` as a call-inventory/runtime-export stage without promoting it into recipe-stage-only diagnostics.
