# Minimize Workspace-Worker Watchdog Policy

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The user-visible goal is to stop fighting Codex’s normal local-work behavior during long-lived worker sessions. After this change, the main workspace-worker session for line-role, recipe, and knowledge should behave like a permissive local file worker: it can inspect local files, run normal local shell commands, and write local outputs without being killed just because the exact shell syntax was not anticipated by the watchdog.

The intended result is not “a slightly larger allowlist.” The intended result is a hard policy simplification. Structured retry and repair calls should remain strict one-shot JSON calls that die on any tool use, but the main workspace-worker sessions should only be killed for obvious repo escape, obvious off-contract tools, or truly pathological command loops. A benchmark rerun should show that watchdog kills are no longer the dominant reason recipe and knowledge stay dead.

## Surprises & Discoveries

- Observation: the last two benchmark reruns showed that loosening the watchdog by enumerating more command shapes helped, but it did not remove the core failure mode.
  Evidence: the March 20, 2026 00:15:56 run still died on `jq '.[0] | keys'`, bare `rg`, and `jq ... > out/...json`; the March 20, 2026 00:50:25 run improved again but still died on heredoc writes such as `cat <<'EOF' > out/...json`.

- Observation: line-role is now less blocked by the watchdog than recipe and knowledge, but it still triggers the same policy seam.
  Evidence: the March 20, 2026 00:50:25 run completed line-role with no failed shards, yet worker live-status artifacts still recorded `watchdog_command_execution_forbidden` on a `jq` local transform. The stage survived only because the retry path recovered, not because the main-session policy was truly correct.

- Observation: the current classifier is too syntactic for the problem it is trying to solve.
  Evidence: the shared classifier in `cookimport/llm/codex_exec_runner.py` still tries to reason about shell tokens, metacharacters, orientation commands, output-path allowances, and helper executables. The benchmark evidence shows Codex keeps discovering a new harmless local shell shape faster than the allowlist grows.

- Observation: the user’s actual preference is clear and materially changes the design bar.
  Evidence: the user explicitly rejected “10 passes slowly stripping it back” and asked for a design closer to “do nothing and add rules only as needed.” This means the default policy should be permissive, not enumerative.

- Observation: even after the first relaxation pass, the classifier still silently preserved two old syntactic restrictions that violated the intended minimal policy.
  Evidence: before this change, direct `classify_workspace_worker_command(...)` probes returned `forbidden_outside_workspace_path` for `/bin/bash -lc 'cat temp.json'` and `forbidden_glob_path` for `/bin/bash -lc 'cat out/*.json'`, even though both commands stay inside the worker workspace.

## Context and Orientation

This repository runs three Codex-backed correction stages on cookbook data: line-role labeling, recipe correction, and non-recipe knowledge review. All three now use a long-lived worker model for their main attempt. A long-lived worker model means the repo creates a bounded local workspace, writes task files into it, launches one Codex session against that workspace, and expects the worker to read local files and write `out/<task_id>.json` files over time.

The main files relevant to this change are:

- `cookimport/llm/codex_exec_runner.py`: the shared direct-exec runner and the shared workspace-worker watchdog classifier. This is the primary implementation site.
- `cookimport/parsing/canonical_line_roles.py`: the line-role stage-specific worker prompt builder and its stage watchdog wrapper.
- `cookimport/llm/codex_farm_orchestrator.py`: the recipe stage worker prompt builder and stage watchdog wrapper.
- `cookimport/llm/codex_farm_knowledge_orchestrator.py`: the knowledge stage worker prompt builder and stage watchdog wrapper.
- `tests/llm/test_codex_exec_runner.py`: the primary shared-policy test file.
- `tests/parsing/test_canonical_line_roles.py`, `tests/llm/test_recipe_phase_workers.py`, and `tests/llm/test_codex_farm_knowledge_orchestrator.py`: stage-level tests that assert prompt wording and tolerated watchdog behavior.

The current problem is that the shared classifier still tries to decide whether a shell command is acceptable by understanding the exact syntax of that command. That was somewhat workable for a tiny set of `cat` or `sed` reads, but it breaks down immediately when Codex uses other normal local-shell patterns such as `jq` filters with pipes, bare `rg`, heredoc file writes, or redirected output creation. The March 20 benchmark runs show that this is still the dominant operational failure for recipe and knowledge, and still a lingering failure for line-role.

For this plan, “minimal policy” means the main workspace-worker session is allowed to use ordinary local shell syntax by default. The watchdog still kills the session if the command appears to invoke an obviously off-contract executable such as `git`, `python`, `node`, `curl`, `wget`, or a package manager, if a visible path escapes the local workspace via absolute paths or parent traversal, or if the worker burns too many shell commands without producing outputs. Everything else should be tolerated.

## Plan of Work

Start in `cookimport/llm/codex_exec_runner.py`. Remove the parts of `classify_workspace_worker_command(...)` that try to parse shell structure in detail. The function should stop making semantic decisions about pipelines, redirections, heredocs, `jq` filters, `find` arguments, or whether a command contains a named local file path. Instead, parse enough to identify the top-level executable, inspect visible path-like tokens for absolute or parent-traversal escapes, and then allow the command unless it falls into the minimal denylist or violates the workspace path boundary.

The resulting shared policy should be straightforward. If the command cannot be parsed at all, keep the existing fail-closed behavior. If the top-level executable is an obvious off-contract tool such as `git`, `python`, `python3`, `node`, `npm`, `npx`, `curl`, `wget`, `ssh`, `scp`, `docker`, or package-manager style tools, reject it. If any visible path token is absolute or uses `..`, reject it. Otherwise, allow it. Preserve the loop watchdog separately so a worker can still be stopped for long shell-only thrashing without outputs.

Then simplify the prompt contract in the three stage modules. In `cookimport/parsing/canonical_line_roles.py`, `cookimport/llm/codex_farm_orchestrator.py`, and `cookimport/llm/codex_farm_knowledge_orchestrator.py`, remove wording that still implies a narrow helper-command allowlist. Replace it with plain language: broad local shell use is acceptable, stay inside the workspace, prefer the named task files first, and avoid repo/network/interpreter tools. The worker manifest written by `cookimport/llm/codex_exec_runner.py` should carry the same simpler description so the local workspace tells the same story as the prompt.

Next, update the test suite to match the new policy. Shared tests in `tests/llm/test_codex_exec_runner.py` should stop asserting that specific shell constructs are forbidden just because of their syntax. Add or update tests so the shared classifier explicitly allows representative commands that used to fail in the benchmark, including `jq` filters with pipes, output redirections to `out/*.json`, heredoc writes to `out/*.json`, and broad local searches such as bare `rg`. Keep negative tests only for obvious off-contract tools and path escapes. Stage-level tests should assert the revised prompt wording and should continue to prove that the shared classifier state is surfaced into live status artifacts.

Finally, rerun the focused tests and the `llm` and `parsing` domain suites. Only after those pass should a human-approved benchmark rerun be used to judge whether the operational collapse has truly moved from “watchdog blocks normal shell use” to some deeper semantic or output-shape issue.

## Validation and Acceptance

This change is acceptable only if all of the following are true.

First, the shared classifier allows ordinary local shell forms by default. The focused test suite must show that commands structurally similar to the benchmark failures are tolerated when they stay inside the workspace.

Second, stage prompts and worker manifests must tell the truth about the runtime contract. A reviewer opening one worker prompt for line-role, recipe, or knowledge should read that broad local shell use is allowed, not a narrow helper-only fiction.

Third, structured retry and repair calls must remain strict. The tests and code path for one-shot structured prompts should still fail closed on shell use.

Fourth, path escape and obvious off-contract tool protection must remain active. Tests must still reject visible parent traversal and tools such as `git` or `python`.

Fifth, after a user-approved benchmark rerun, the operational failure surface should change materially. Recipe and knowledge should no longer predominantly fail with `watchdog_command_execution_forbidden` on normal local shell writes or queries. If quality is still poor after that, the next problem is semantic prompting or output validation, not shell-shape policing.

## Idempotence and Recovery

This change is safe to retry because it is a simplification of policy, not a data migration. If the shared classifier becomes too permissive in a way that breaks tests, the test failures should surface that quickly before any benchmark rerun.

If implementation lands partially, the safest checkpoint is after the shared classifier and shared tests agree but before stage prompt wording is updated. At that point, the runtime behavior may already be correct, and the remaining work is documentation and prompt alignment rather than a half-migrated data contract.

If a benchmark rerun after implementation still shows watchdog kills, inspect the live worker `live_status.json` files first. The remaining forbidden commands should now be only obvious off-contract tools or true path escapes. If the new kills are still ordinary local shell writes, the classifier simplification was incomplete.

## Interfaces and Dependencies

Keep the implementation local to the existing runtime. Do not introduce a new process supervisor or a new worker-orchestration subsystem.

At the end of this plan, the following broad interface expectations must hold:

In `cookimport/llm/codex_exec_runner.py`, `classify_workspace_worker_command(...)` still returns `WorkspaceCommandClassification`, but its policy is boundary-based rather than shell-shape-based. The function must be simple enough that a future contributor can understand why a command was rejected by reading only that function and its tests.

In `cookimport/parsing/canonical_line_roles.py`, `cookimport/llm/codex_farm_orchestrator.py`, and `cookimport/llm/codex_farm_knowledge_orchestrator.py`, the worker prompt builders must describe the same boundary-based policy the runner actually enforces.

In the stage watchdog wrappers, workspace-worker main sessions must continue to call the shared classifier while structured retry and repair calls remain on their strict zero-tool path.

At the benchmark level, the main expected dependency is not a new library but the existing worker artifact tree under `data/output/.../workers/...`. That tree remains the primary place to prove whether the simplified watchdog succeeded.
