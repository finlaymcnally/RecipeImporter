# Enable Pass4 Knowledge Harvest In Benchmark Runs

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds. This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a benchmark run can execute the same optional pass4 knowledge harvest that `cookimport stage` already supports. That means `cookimport labelstudio-benchmark` and benchmark-driven speed runs can write pass4 artifacts such as `pass4_knowledge_manifest.json` and `knowledge/<workbook_slug>/snippets.jsonl`, and processed benchmark outputs can use those snippets when building stage evidence labels.

The visible proof is straightforward. Running `cookimport labelstudio-benchmark ... --llm-knowledge-pipeline codex-farm-knowledge-v1 --allow-codex --no-upload` should leave pass4 knowledge artifacts under the prediction-run root, and benchmark Codex variants created from benchmark run settings should preserve knowledge enablement instead of silently forcing it off.

## Surprises & Discoveries

- Observation: benchmark documentation already says `allow_codex` covers recipe, line-role, knowledge, and tags, but the actual `labelstudio-benchmark` signature only exposes recipe and line-role switches.
  Evidence: `cookimport/cli.py` benchmark help text mentions knowledge/tags in the `--allow-codex` description while the signature lacks `--llm-knowledge-pipeline`.

- Observation: the benchmark pred-run builder already knows how to consume knowledge snippets later in the flow, but it never generates them because the knowledge pipeline never reaches `RunSettings`.
  Evidence: `cookimport/labelstudio/ingest.py` resolves `knowledge_snippets_path` from `llm_report` before writing processed outputs, yet `generate_pred_run_artifacts(...)` only passes `llm_recipe_pipeline` into `build_run_settings(...)`.

## Decision Log

- Decision: implement pass4 knowledge for benchmark runs by aligning benchmark inputs with existing stage-style `RunSettings` fields instead of introducing benchmark-only flags or a separate orchestration path.
  Rationale: stage and benchmark already share the same knowledge orchestrator and processed-output writer behavior once `RunSettings.llm_knowledge_pipeline` is populated.
  Date/Author: 2026-03-06 / Codex

- Decision: keep benchmark baselines deterministic with `llm_knowledge_pipeline=off`, but make benchmark Codex variants opt into knowledge alongside the recipe Codex variant.
  Rationale: baseline-vs-Codex comparisons should remain clean, and forcing knowledge off in Codex variants would continue to hide the feature the user explicitly wants benchmarked.
  Date/Author: 2026-03-06 / Codex

- Decision: extend the existing benchmark plan-mode forwarding test instead of adding a new end-to-end benchmark fixture.
  Rationale: the plan-mode test already intercepts `generate_pred_run_artifacts(...)`, which is the exact seam where the benchmark command had been dropping knowledge settings.
  Date/Author: 2026-03-06 / Codex

## Outcomes & Retrospective

Benchmark runs now have parity with stage runs for pass4 knowledge configuration. `labelstudio-benchmark` exposes the knowledge flags directly, benchmark-driven prediction generation passes those settings into `RunSettings`, benchmark plan mode records the knowledge surface in its saved config, and benchmark Codex variants created from all-method settings now enable the knowledge pipeline instead of silently forcing it off.

The focused regression suite passed after the change. I did not run a live Codex benchmark invocation in this turn, so the remaining validation gap is observational rather than structural: a real offline benchmark run with `--llm-knowledge-pipeline codex-farm-knowledge-v1 --allow-codex` would be the next step if we want a fresh artifact sample on disk.

## Context and Orientation

`cookimport stage` and benchmark prediction generation are similar but not identical. The stage command is implemented in `cookimport/cli.py` and the worker path in `cookimport/cli_worker.py`; both already accept `llm_knowledge_pipeline`, `codex_farm_pipeline_pass4_knowledge`, and `codex_farm_knowledge_context_blocks`, then call `run_codex_farm_knowledge_harvest(...)` when knowledge is enabled.

Benchmark prediction generation lives in `cookimport/cli.py::labelstudio_benchmark(...)` and `cookimport/labelstudio/ingest.py::generate_pred_run_artifacts(...)`. The benchmark CLI normalizes many run settings and passes them into `generate_pred_run_artifacts(...)` when running offline (`--no-upload`). The shared adapter `cookimport/config/run_settings_adapters.py::build_benchmark_call_kwargs_from_run_settings(...)` is used by speed-suite and scheduler helpers to turn a `RunSettings` object into benchmark CLI kwargs.

Benchmark all-method variants are produced in `cookimport/cli.py::_build_all_method_variants(...)`. The helper applies a baseline benchmark contract from `cookimport/config/codex_decision.py::apply_benchmark_baseline_contract(...)` and, when Codex permutations are requested, builds a Codex variant through `apply_benchmark_codex_contract_from_baseline(...)`. Today that Codex variant turns on recipe and line-role surfaces but leaves knowledge forced off because the Codex patch does not set `llm_knowledge_pipeline`.

The tests that protect this area already exist and should be extended rather than replaced. `tests/cli/test_run_settings_adapters.py` checks what benchmark kwargs are emitted from `RunSettings`. `tests/labelstudio/test_labelstudio_benchmark_helpers_scheduler.py` checks benchmark all-method variant normalization. `tests/cli/test_cli_llm_flags.py` checks CLI help output for LLM-related stage flags and can be extended to cover benchmark help.

## Plan of Work

First, extend `cookimport/cli.py::labelstudio_benchmark(...)` so the benchmark command accepts the same pass4 knowledge knobs the stage command already exposes: `--llm-knowledge-pipeline`, `--codex-farm-pipeline-pass4-knowledge`, and `--codex-farm-knowledge-context-blocks`. Normalize those values with the same helper functions used by `stage`, include them in the Codex decision payload, and forward them to every call into `generate_pred_run_artifacts(...)`. Update any benchmark plan-only metadata payloads so the saved run config accurately records knowledge enablement.

Second, update `cookimport/labelstudio/ingest.py::generate_pred_run_artifacts(...)` so its signature accepts the pass4 knowledge fields, validates them, and passes them through `build_run_settings(...)`. Also update the plan-only `planned_work` structure to include a knowledge-harvest plan whenever `llm_knowledge_pipeline != off`, so plan mode surfaces the added work instead of hiding it. The later post-processing path should not need behavioral changes because it already respects `llm_report["knowledge"]` when present.

Third, update `cookimport/config/run_settings_adapters.py::build_benchmark_call_kwargs_from_run_settings(...)` so benchmark-driving callers such as `bench speed-run` and interactive benchmark schedulers forward the pass4 knowledge settings into `labelstudio_benchmark(...)`. This is the change that lets benchmark runs driven from stored `RunSettings` actually use knowledge harvesting instead of silently dropping it.

Fourth, update `cookimport/config/codex_decision.py` so the benchmark Codex contract enables `llm_knowledge_pipeline=codex-farm-knowledge-v1` while the baseline contract continues to pin it to `off`. Then adjust `_build_all_method_variants(...)` expectations indirectly through tests so Codex benchmark variants include the knowledge surface and baseline variants remain deterministic.

Finally, update the focused tests and a short benchmark/LLM doc note so the new benchmark behavior is discoverable and verified.

## Validation and Acceptance

Acceptance is behavioral, not just structural.

`labelstudio-benchmark --help` must list the new knowledge flags so a user can opt in directly.

Building benchmark kwargs from a `RunSettings` object that enables `llm_knowledge_pipeline="codex-farm-knowledge-v1"` must include that setting plus the pass4 pipeline id and context-block count.

Benchmark all-method baseline variants must still resolve `llm_knowledge_pipeline="off"`, while Codex variants must resolve `llm_knowledge_pipeline="codex-farm-knowledge-v1"`.

An offline benchmark run with knowledge enabled should now be capable of writing pass4 knowledge artifacts beneath its prediction-run root, reusing the same orchestrator already exercised by stage runs.

## Idempotence and Recovery

These edits are additive and safe to re-run. If a test fails, fix the missing call path and re-run only the failing targeted tests first. No destructive migration is involved.
