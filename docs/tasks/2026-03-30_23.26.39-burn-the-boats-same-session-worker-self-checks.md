# Burn The Boats Same-Session Worker Self-Checks

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Exact User Desire

> "i want all of the LLM pass workers to be able to ensure their outputs are valid and correct if wrong (they should just run at end of editing once done , check once at end)"

## Structured Restatement Of User Desire

Every live LLM worker that edits a repo-owned `task.json` should finish its normal edit, run one repo-owned validation command exactly once at the end, and then only continue if that command rewrites the task file into a repair form for the same worker session. The worker should not blindly stop and leave deterministic repo code to discover simple structural mistakes later. Instead, the worker should get one immediate same-session chance to fix its own bad edit while deterministic code still remains the authority on validation, repair rewrites, and final artifact expansion. This is burn-the-boats work for the live editable-task-file worker contract. The happy path becomes: edit `task.json`, run the repo helper, obey the helper result, stop when it says complete.

## Purpose / Big Picture

After this change, recipe, knowledge, and canonical line-role workspace workers all follow the same honest local story. A worker opens one repo-written `task.json`, edits only the answer fields, then runs one repo-owned helper command. If the answers are structurally valid, the helper either finishes the worker’s outputs directly or advances the same session to the next step. If the answers are invalid but repairable, the helper rewrites `task.json` into repair mode, preserving immutable evidence and attaching deterministic validation feedback so the same worker can fix the mistake without losing context. The user-visible proof is simple: focused tests show recipe and line-role workers recovering from malformed first-pass edits in the same session, and the fake worker runtime simulates that same helper loop instead of depending on a later repo-side repair launch.

## Context and Orientation

The live editable-task-file worker contract in this repository means the repo writes one `task.json` file into a sterile worker-visible workspace. That file contains immutable evidence under each unit plus editable answer fields under `/units/*/answer`. The worker is supposed to read the file, fill in answers, save the same file, and stop. Deterministic repo code then validates the edited file, expands accepted answers into stage outputs, and may fail the shard or task if the edit is structurally wrong.

Three stage families matter here.

`cookimport/llm/knowledge_same_session_handoff.py` is the working model. It owns a hidden repo-side state file, validates the current `task.json`, rewrites `task.json` into repair mode when needed, advances the same worker session from classification to grouping when appropriate, and writes final outputs once validation passes.

`cookimport/llm/recipe_stage_shared.py` is the current recipe owner. It already knows how to build the recipe task file, evaluate edited answers with `_evaluate_recipe_task_file_answers(...)`, and build a repair `task.json`. The current gap is that recipe runs those steps only after the worker exits, so repair currently requires a second worker launch.

`cookimport/parsing/canonical_line_roles/runtime.py` plus `cookimport/parsing/canonical_line_roles/validation.py` are the current canonical line-role owners. They already know how to convert edited task-file answers back into shard `rows`, validate those rows, and resolve full shard coverage. The current gap is that the worker never runs a repo-owned validation helper itself, so the runtime only discovers bad exclusion reasons or similar mistakes after the worker has already stopped.

`cookimport/llm/codex_exec_runner.py` contains the fake direct-exec runner used by focused tests. That fake runner already simulates the knowledge same-session helper loop when the knowledge helper state env var is present. This file must grow the same simulation for recipe and line-role so tests reflect the real contract.

The core helper utility is `cookimport/llm/editable_task_file.py`. It owns `TASK_FILE_NAME`, task-file load/save, contract validation, and `build_repair_task_file(...)`. The repo already treats repair as “rewrite the same `task.json` with deterministic validation feedback and previous answers.” This plan extends that same shape to all live task-file worker surfaces.

## Plan of Work

First, add one narrow repo-owned helper module for recipe and one for canonical line-role. Each helper should mirror the knowledge helper shape: load hidden repo state, load the visible `task.json`, validate the current edit, update the hidden state counters/history, and then either rewrite `task.json` into repair mode or finish by writing the accepted final output artifacts into the existing worker-owned `out/` folder. Recipe does not need multi-phase advancement like knowledge, but it does need to validate per-unit answers and produce final per-task payloads. Line-role does not need a second semantic phase either, but it does need to validate full shard coverage and write shard `out/<shard>.json` payloads from the accepted answer edits.

Second, rewire the live worker prompts. `cookimport/llm/recipe_stage_shared.py` and `cookimport/parsing/canonical_line_roles/planning.py` should stop saying “deterministic validation happens after you save” as the terminal contract. Instead, they should instruct the worker to edit `task.json`, run the new helper command, and then obey the helper result. If the helper rewrote `task.json` into repair mode, the worker should reopen the same file, fix only the named issues, and run the helper again. If the helper reports completion, the worker should stop. The prompt should remain calm and narrow: read one file, edit allowed answer fields, run one repo helper, stop when complete.

Third, rewire the live runtimes so the helper state exists before the worker launches and the post-session validation path consumes that helper state instead of treating repair as an external follow-up. For recipe, `cookimport/llm/recipe_stage_shared.py` should initialize hidden same-session state, pass a state-path env var into `runner.run_workspace_worker(...)`, and then read final outputs plus counters from that state when the worker returns. Recipe should no longer launch a second repair worker session on the normal contract path. For line-role, `cookimport/parsing/canonical_line_roles/runtime.py` should initialize hidden same-session state and let the helper write shard outputs into `workers/<id>/out/*.json` during the same worker session. The existing watchdog retry for genuine missing-output sessions can remain separate, but ordinary invalid task-file answers should be handled by the same-session helper path rather than falling through as unrecoverable invalid output.

Fourth, update `cookimport/llm/codex_exec_runner.py` so the fake runner simulates the new helper loops. The existing knowledge branch is the pattern: fake worker edits the current task file, runs the helper, and repeats only when the helper advances or rewrites the task file for repair. Add equivalent recipe and line-role state env hooks so focused tests can verify the real worker contract without launching live Codex sessions.

Fifth, add or update focused tests. Recipe needs direct helper tests plus runtime tests proving a malformed first pass gets repaired in-session and does not consume a second happy-path worker launch. Canonical line-role needs helper tests that cover an invalid exclusion reason rewriting the file into repair mode and then succeeding after the fix. Fake runner tests should prove the new env var hooks drive the expected helper loop. Existing line-role tests about immutable evidence drift must keep passing.

Sixth, update the short docs that describe the live worker story. The active LLM docs and the line-role package README should describe the end-of-edit helper truthfully: edit the file, run the helper once, fix and rerun only if the helper rewrote repair instructions, otherwise stop.

## Validation and Acceptance

This work is accepted only when the repository proves that the worker contract changed in behavior, not just in prompt wording.

The first acceptance criterion is recipe same-session repair. A recipe worker that writes an invalid first-pass answer into `task.json` must receive a repair rewrite in the same visible file, fix the answer, rerun the helper, and finish without a second normal worker launch. Focused tests should prove the helper counters and runtime telemetry reflect one happy-path worker session.

The second acceptance criterion is canonical line-role same-session repair. A line-role worker that uses an invalid exclusion reason such as `nonrecipe` must have that mistake detected by the helper, rewritten into repair mode, corrected in the same session, and then promoted into a valid shard output. The run must no longer fail closed for that specific repairable worker mistake if the worker fixes it before stopping.

The third acceptance criterion is preserved authority boundaries. Deterministic repo code must still own validation, repair rewrites, and final artifact expansion. The worker may only edit `/units/*/answer`. Immutable evidence drift handling already present in line-role must remain intact.

The fourth acceptance criterion is honest fake-runner behavior. The fake direct-exec runner must simulate the same helper loop so repository tests exercise the same end-of-edit contract the real worker prompts describe.

The fifth acceptance criterion is documentation clarity. Active docs must describe the worker story as “edit `task.json`, run the repo helper, continue only if the helper rewrites repair instructions, otherwise stop.”

## Idempotence and Recovery

The helper commands introduced by this plan must be deterministic for a given workspace state. Running a helper twice without any intervening task-file edits should not silently mutate more state than the first run already produced. The safest behavior is to return a stable status such as “completed” or “no new edits detected” while leaving the current `task.json` and final outputs unchanged.

If implementation is interrupted midway, do not ship a mixed state where prompts instruct workers to run a helper that the runtime did not wire up. The safe checkpoint is always: helper entry point exists, env/state wiring exists, fake-runner tests prove the contract, then prompt/docs updates land.

Nearby local edits in line-role runtime and editable-task-file validation should be preserved. If a conflict appears in those files, re-read the current diff and merge with it instead of resetting the file to an older version.

## Artifacts and Notes

The expected happy-path worker story after this plan should look like this in spirit:

    worker opens task.json
    worker edits only /units/*/answer
    worker runs one repo helper command
    helper returns completed
    out/*.json already exists
    no second happy-path worker session launched

The expected repair story should look like this in spirit:

    worker opens task.json
    worker writes one structurally bad answer
    worker runs one repo helper command
    helper rewrites task.json into mode=repair with validation_feedback
    worker fixes only the named answer fields
    worker reruns the helper
    helper returns completed and writes final outputs

The line-role-specific proof case should include a bad exclusion reason like `nonrecipe` or `nonrecipe_excluded`, because that is the real failure mode this plan is meant to prevent from escaping the worker session.
