## Problem Statement

`recipeimport` Codex Farm integration still had caller-contract drift versus `shared/CodexFarm/CONNECTION_INSTRUCTIONS.md`: model discovery came from local cache, pipeline ids were not prevalidated against pack listings, and process failures did not follow up with `run errors` diagnostics.

## Acceptance Criteria

- Interactive run-settings model picker discovers models via `codex-farm models list --json`.
- Runtime pass orchestration validates configured pipeline ids using `codex-farm pipelines list --root <pack> --json` before execution.
- Subprocess runner parses `process --json` payload and appends `run errors --run-id ... --json` context when available.
- Existing codex-farm orchestrator/model-picker tests continue passing.

## Constraints / Gotchas + Decision

- Model discovery is best-effort in interactive UI: failures in `models list --json` should not block run setup, so picker keeps fallback options.
- Pipeline validation is strict only for subprocess-backed execution (`runner is None`), preserving fake/injected runners used by tests.
- `process --json` parsing is enforced for failure diagnostics, while successful empty/non-JSON stdout remains tolerated for compatibility with test doubles.
