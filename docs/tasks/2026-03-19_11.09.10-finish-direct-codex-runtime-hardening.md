## Purpose / Big Picture

RecipeImport has mostly moved past “CodexFarm as the runtime brain” and now owns its direct `codex exec` execution model inside the repo. That is good, but the runtime is still uneven. A human can now inspect strong post-run artifacts for recipe, knowledge, and canonical line-role shards, but there are still three practical pain points. First, a bad shard can still waste time and tokens before the repo notices. Second, recipe correction does not yet have the same near-miss repair maturity that knowledge and line-role already have. Third, the new sterile direct-exec workspace exists operationally but is not yet fully promoted into the repo’s normal logging and debugging surfaces.

After this plan is complete, a human should be able to run focused local tests and zero-token rehearsals and observe a direct runtime that is consistent across all three Codex-backed stages: the live worker starts in a sterile workspace with narrow instructions, the repo can supervise the shard while it runs, unsafe or pathological shards are terminated or quarantined early, each stage has a conservative repair path for near-miss outputs, and the resulting artifacts make it obvious what happened without spelunking through ad hoc local folders.

The most important user-visible outcome is that direct Codex work becomes cheap to understand and cheap to fail. For the strict JSON-returning stages, “the model is doing something weird” should no longer mean “wait for an opaque subprocess to burn more time and tokens.” It should mean “the repo rejected the shard before launch, killed it quickly for a clear reason, retried it once with a tiny stricter packet, or left a durable failure artifact that explains why it stopped.” When a shard goes bad, the operator should not have to guess whether the problem was prompt wording, schema drift, invalid ownership, or an uncontrolled coding-agent detour. The runtime should say so, and the artifacts should make that explanation durable.

## Context and Orientation

The direct runtime has three active Codex-backed phases in this repo.

Recipe correction lives in `cookimport/llm/codex_farm_orchestrator.py`. It builds recipe shard plans from deterministic intermediate recipe state, executes one direct `codex exec` call per shard, validates owned `recipe_id` coverage, and then promotes successful corrected structures into deterministic final assembly.

Knowledge review lives in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. It builds non-recipe knowledge shards from deterministic stage output, executes direct `codex exec` calls, validates owned `chunk_id` coverage, and already supports one-shot repair for near-miss invalid outputs.

Canonical line-role labeling lives in `cookimport/parsing/canonical_line_roles.py`. It builds contiguous row-review shards, executes direct `codex exec` calls against worker-local task files, validates exact row coverage, and already supports one-shot repair for near-miss invalid outputs.

The shared direct subprocess seam is `cookimport/llm/codex_exec_runner.py`. That file is where the repo builds the `codex exec --json --output-schema ...` command, creates the sterile execution workspace, writes the generated `AGENTS.md`, launches the subprocess, parses the JSON event stream, and computes post-run telemetry fields such as command counts, reasoning counts, and token accounting.

In this repository, a “shard” means one immutable owned task payload. A “worker” means one local execution bucket that may own one or more shard ids. The durable worker folder remains in the repo output tree. The sterile execution workspace is a mirrored external folder under `~/.codex-recipe/recipeimport-direct-exec-workspaces/` used only for the live subprocess cwd.

The current runtime is already strong on post-run logging. Per-shard folders contain files such as `prompt.txt`, `events.jsonl`, `usage.json`, `status.json`, proposals, and repair artifacts. The remaining work is about making the runtime equally strong before and during process execution, and about making those runtime details easier to understand from normal stage and run summaries.

This plan uses four plain-English terms consistently. A “preflight quarantine” is a deterministic repo-owned check that rejects a shard before any live model call because the input is already malformed, absurdly large, or outside the safe contract for that stage. A “watchdog” is repo-owned logic that receives live shard progress snapshots and decides whether to let the run continue, terminate it, or mark it as retryable. A “cohort” is the sibling shard set inside one phase run, and a “cohort-relative outlier” means a shard is killed because it is wildly slower than its siblings, not merely because some absolute timeout expired. A “retry packet” is one fresh repo-owned second attempt that includes the original owned task, the kill or validation reason, and possibly tiny curated sibling success examples; it is not a message injected into the already-running subprocess.

## Plan of Work

Start in the shared runner, because that is where every direct shard run converges. Add a streaming supervision mode to `cookimport/llm/codex_exec_runner.py` that can read Codex JSON event lines incrementally instead of only after `subprocess.run(...)` finishes. Preserve the current final `CodexExecRunResult` contract so downstream code keeps receiving the same basic result object, but add additive supervision metadata that can record whether the shard was quarantined, killed, retried, repaired, or completed normally. The first milestone is intentionally a proof step: prove live event visibility exists in this environment before letting the rest of the plan depend on it.

Once the runner can stream events, add repo-owned watchdog policies for the strict JSON stages first. Knowledge and canonical line-role should be able to reject obviously unsafe shard inputs before any live call and terminate runs for forbidden tool use, pathological command loops, repeated reasoning-without-output patterns, or clear cohort-relative outlier runtime. The policy layer belongs in the stage orchestrators because those files understand ownership counts, sibling shard cohorts, and stage-specific tolerance for retries. The initial policy for those strict JSON stages should be intentionally unforgiving: tool use is a smell, not a helpful detour.

After live supervision exists for the strict stages, bring recipe correction up to maturity parity. Recipe uses a different payload contract and more nuanced outputs, so its repair seam should not be copy-pasted blindly from knowledge or line-role. Instead, add a conservative recipe repair path for structurally near-correct outputs such as missing one owned `recipe_id`, malformed top-level wrapper objects, or compact tagging/linkage errors where the model clearly attempted the right task. Keep it one-shot, additive, and artifact-heavy. Recipe should also get the same basic supervision vocabulary and status surfaces as the strict stages, even if its kill predicates stay narrower.

In parallel with the supervision and repair work, promote sterile workspace provenance into repo-owned artifacts. Every direct worker should have an explicit pointer to its sterile execution root, the generated `AGENTS.md`, and any mirrored task files that differed from the repo artifact path. Add a tiny live-status artifact so the operator can see elapsed seconds, last event time, command and reasoning counters, and current watchdog policy state before final completion. The point is not to keep giant ephemeral folders forever. The point is to make the existence and identity of the sterile workspace and the current watchdog state visible in normal manifests and debug packets.

Finish by tightening the reporting surfaces. Stage summaries, worker payloads, and `prompt_budget_summary.json` should clearly separate normal completions, preflight rejections, watchdog kills, repair attempts, retry attempts, successful recoveries, and unrecovered failures. The operator should not need to infer these states indirectly from raw `events.jsonl`, and they should not need to inspect the external sterile workspace to answer “what happened to this shard?”

## Milestones

### Milestone 1: Add live event streaming and supervision hooks to the direct runner

The first milestone is to prove that `codex exec --json` can be supervised while it runs. Implement this in `cookimport/llm/codex_exec_runner.py` by replacing the all-at-once `subprocess.run(..., capture_output=True)` path with a process-management seam built on a live child process handle. That seam must consume stdout and stderr incrementally, parse JSON event lines as they arrive, and still produce the same final `CodexExecRunResult` shape on normal completion.

This milestone must stay additive. The existing fake runner and the existing final telemetry helpers should continue to work. The new behavior should appear as optional callbacks, watchdog state, and additive metadata rather than a destructive rewrite of every caller. A good live snapshot for this repo includes shard id, worker id, elapsed seconds, parsed event count so far, command-execution count so far, reasoning-item count so far, the most recent command text, whether a final agent message has appeared, and whether the process has crossed a coarse timeout floor. The watcher interface only needs a few explicit answers such as continue, terminate with reason, or mark retryable.

Acceptance for this milestone is local and deterministic: a focused runner test proves that the repo can observe command-execution and reasoning events before process exit, can make an early supervision decision from that stream, and still preserves the partial event history plus termination metadata in the final result.

### Milestone 2: Add preflight quarantine and live watchdog policies for strict JSON stages

Once the runner can stream events, implement the supervision rules that make cheap failure possible. Knowledge and canonical line-role are the first targets because tool use is always suspect for their strict JSON-only contracts. Add deterministic preflight checks in `cookimport/llm/codex_farm_knowledge_orchestrator.py` and `cookimport/parsing/canonical_line_roles.py` that can reject obviously unsafe or malformed shard inputs before launch. Then add live watchdog rules that terminate a shard for forbidden `command_execution`, repeated identical command loops, multiple reasoning items before any valid final output, or clear no-progress conditions after watchdog activity begins.

These watchdogs must be stage-aware. They should write explicit artifact records, not merely raise exceptions. Each killed shard should produce a stable reason code, the observed event counters that triggered the kill, and a status artifact that explains whether the shard was preflight-rejected or live-killed. Use stable codes such as `preflight_rejected`, `watchdog_command_execution_forbidden`, `watchdog_repeated_command_loop`, and `watchdog_reasoning_without_output`. Add a small live-status artifact such as `live_status.json` or `watchdog_status.json` that updates while the worker is running so a human does not have to wait for the final `status.json` to see whether the worker is still healthy.

Acceptance for this milestone is a local fake-run rehearsal where one synthetic knowledge shard and one synthetic line-role shard trigger the watchdog and are terminated with stable reason codes while sibling shards finish normally.

### Milestone 3: Add cohort-relative outlier detection and one-shot retries

Absolute timeouts are too blunt on their own. After basic watchdog support exists, use existing per-stage worker coordination to let the stage orchestrators compare sibling shard runtimes. If one shard is many times slower than the sibling median and is still emitting suspicious behavior or no meaningful progress, terminate it as a cohort-relative outlier. Keep the thresholds explicit and artifacted. A good first rule for this repo is “wait until at least three sibling shards have completed normally, then consider kill only if a minimum absolute floor has passed and the running shard has exceeded a configurable multiple of the sibling median.”

After a shard is killed, add one-shot retry support for retryable watchdog reasons. This retry must be a fresh shard attempt, not a mid-run coaching message. For knowledge and line-role, the retry packet can include a very small summary of why the first attempt was killed and, when available, one or two repo-prepared examples of what a successful sibling output looked like. The retried shard should never be told to wander sibling directories on its own; the repo should materialize the tiny example packet directly. Preserve both attempt histories in repo artifacts. Do not retry more than once, do not retry preflight-rejected shards, and do not retry obviously nonsensical first attempts.

Acceptance for this milestone is a stage-level local test where one synthetic shard is killed as a cohort-relative outlier, retried once with a stricter packet, and the artifacts show both the original kill reason and the retry result.

### Milestone 4: Bring recipe correction up to repair and supervision parity

Recipe correction should stop being the least mature direct stage. Add preflight validation that catches obviously malformed recipe shard inputs before launch. Add supervision hooks that can record or terminate blatantly pathological live runs when the runner surfaces those events. Then add a one-shot repair seam for structurally near-correct recipe outputs. Recipe does not need to copy the exact knowledge and line-role rules, but it should share the same additive artifact model, stable status vocabulary, and explicit retry-versus-repair boundaries.

The recipe repair logic belongs in `cookimport/llm/codex_farm_orchestrator.py`. It must be conservative and grounded in recipe-owned ids. Good candidates include missing exactly one owned `recipe_id`, malformed wrapper objects around otherwise usable corrected recipes, or compact linkage/tagging fields that can be fixed from validator errors. Bad candidates, such as wildly wrong recipe sets or invented recipe ids, should remain unrepaired and explicit failures.

Acceptance for this milestone is a local recipe orchestrator test where a near-miss invalid recipe response is repaired once and promoted successfully, while an obviously bad response remains failed with a clear artifact trail.

### Milestone 5: Make sterile workspace provenance first-class in repo artifacts

The sterile execution workspace already exists and matters, but it is not yet a first-class citizen in the repo artifact model. Add explicit fields to worker or runner manifests that point to the sterile execution root, the generated `AGENTS.md`, and the mirrored worker-local task file paths actually referenced by the live prompt. Decide whether those fields belong on worker assignments, runner payloads, or both, then keep the answer consistent across recipe, knowledge, and line-role.

Also add a small repo-owned workspace manifest artifact that captures the minimum useful information from the sterile directory without forcing the operator to inspect the external filesystem directly. This can be a JSON file containing the source worker path, the execution workspace path, the generated `AGENTS.md` path, and the mirrored `in/` and `debug/` file names. If a live-status artifact exists from earlier milestones, it should point back to this manifest so a novice can correlate “where the worker is running” with “what the watchdog is currently seeing.”

Acceptance for this milestone is a local run or test artifact inspection that shows a novice where the live subprocess actually ran and which local files were mirrored there.

### Milestone 6: Tighten reporting, docs, and zero-token acceptance

The last milestone is to make the whole runtime understandable from normal repo surfaces. Update `cookimport/llm/prompt_budget.py`, any stage summary writers, and the relevant docs so that the operator can see preflight rejections, watchdog kills, retries, repairs, recoveries, and sterile-workspace provenance in the same place they already inspect prompt budgets and stage telemetry. Add or update understanding notes when new seams emerge during implementation.

Keep acceptance local and zero-token by default. The main validation loop should be focused pytest commands and, when useful, fake direct-exec rehearsals that exercise the real worker/orchestrator choreography without spending tokens. An optional live benchmark smoke can remain explicitly out of scope unless the user later approves it.

Acceptance for this milestone is that the focused test commands below pass and the resulting local artifacts clearly expose the new runtime states without requiring manual inspection of raw process logs.

## Validation and Acceptance

This plan is complete when a novice can verify all of the following locally.

First, the shared direct runner can observe Codex JSON events before process exit and can record or act on those events without losing the final result shape.

Second, knowledge and canonical line-role can reject obviously bad shards before launch, kill pathological live runs with stable reason codes, and retry once when the kill reason is retryable.

Third, recipe correction has matching maturity: at minimum a conservative repair seam for near-miss structural failures plus the same basic execution-status and provenance visibility as the other direct stages.

Fourth, every direct stage records the sterile execution workspace as a first-class part of the run, including where the subprocess actually ran and which generated `AGENTS.md` governed it.

Fifth, the standard summaries and prompt-budget artifacts clearly report preflight rejections, live kills, retries, repairs, recoveries, unrecovered failures, and workspace provenance.

The default acceptance bar is local and zero-token: the focused pytest commands above exit `0`, and the resulting artifacts demonstrate the behavior directly. A live benchmark rerun is optional and explicitly separate. If such a rerun is later approved, success means pathological shards stop early, obvious bad inputs get no model spend, and extreme outliers do not hold the whole stage open while burning hidden direct-exec overhead.

## Idempotence and Recovery

All planned changes should be additive and safe to apply incrementally. The runner should preserve the existing final `CodexExecRunResult` contract while adding streaming supervision and new metadata, so downstream code can migrate in small steps. Killed, quarantined, retried, or repaired shards must never erase the primary evidence. Every state transition should write its own status artifacts and preserve the original run artifacts. Retry must remain capped at one new attempt per shard. If a threshold proves too aggressive in tests, raise or disable that specific rule while keeping the surrounding artifact and supervision scaffolding intact.

Because live token validation is not part of acceptance, recovery should favor narrow local test loops. After each milestone, rerun only the suites needed to prove that milestone and update this plan immediately when a design decision changes.
