## Problem statement

Compare & Control still shared data/state with the `History Table & Trend` subsection (table filters, quick filters, and subset writes). The request was to sever these interactions so both subsections are independent.

## Acceptance criteria

- Compare & Control does not consume `currentPreviousRunsFilterResult()` rows.
- Compare & Control does not write selected groups into Previous Runs table filters.
- Compare & Control dynamic chart is fed from compare/control scope only.
- Previous Runs view presets do not overwrite compare/control state.
- Compare/control analytics tests still pass.

## Verification steps

1. `source .venv/bin/activate && pytest tests/analytics/test_stats_dashboard.py -k "js_supports_previous_runs_column_header_filters or compare_control"`
2. `source .venv/bin/activate && pytest tests/analytics/test_dashboard_state_server.py`

## Evidence

- `7 passed, 70 deselected` for compare-control + previous-runs-header-filter JS contract slice.
- `2 passed` for dashboard UI-state server behavior.

## Constraints / gotchas

- Dashboard JS is emitted from `cookimport/analytics/dashboard_render.py`; small string changes can break JS contract assertions.
- Node-based compare/control harness runs with bootstrap disabled, so helper functions used in harness must not require runtime-loaded `DATA`.

## Decision

Introduced a dedicated compare/control render path and local subset filter semantics. Removed old table-filter mutation path and removed compare/control from Previous Runs preset capture/apply to avoid hidden cross-state coupling.

## Rollback / compensation plan

Revert this task diff to restore the old shared-data wiring (`matchedRecords` input + table-filter subset writes).
