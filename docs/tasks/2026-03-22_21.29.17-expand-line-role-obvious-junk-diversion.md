# Expand line-role obvious-junk diversion without turning it back into a semantic knowledge classifier

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

After this change, the first LLM line-role pass should divert more obviously useless outside-recipe material away from the knowledge stage while still leaving genuinely ambiguous cookbook prose for semantic review. The user-visible outcome is simple: `data/output/<ts>/08_nonrecipe_review_exclusions.jsonl` should contain more low-risk publishing junk than it does today, `raw/llm/<workbook>/knowledge/in/*.json` should carry fewer obvious blurbs and marketing fragments, and the knowledge stage should spend more of its prompts on material that has a real chance of becoming durable cooking knowledge.

The change must stay inside the current ownership boundary. Line-role is still only a routing and exclusion stage for outside-recipe text. It must not start deciding that borderline prose is final `knowledge` or final `other`. If there is any realistic chance that the text is useful cooking explanation, lesson framing, or technique guidance, it should still stay review-eligible and proceed to the knowledge stage.

## Context and Orientation

The line-role stage lives in [cookimport/parsing/canonical_line_roles.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles.py). In this repository, “line-role” means the first LLM correction pass over ordered atomic lines of extracted cookbook text. It can assign recipe-structure labels such as `RECIPE_TITLE`, `INGREDIENT_LINE`, or `INSTRUCTION_LINE`, and for outside-recipe `OTHER` rows it can optionally set `review_exclusion_reason` to signal that a row is too obviously useless to send to knowledge review.

The prompt for that stage is built in [cookimport/llm/canonical_line_role_prompt.py](/home/mcnal/projects/recipeimport/cookimport/llm/canonical_line_role_prompt.py). That file contains both the inline prompt fallback and the file-backed worker prompt instructions. Any change to what line-role is allowed to exclude must be reflected there, because the worker needs the same contract the parser enforces.

Stage 7 routing lives in [cookimport/staging/nonrecipe_stage.py](/home/mcnal/projects/recipeimport/cookimport/staging/nonrecipe_stage.py). In this repository, “Stage 7” is the deterministic step that groups final nonrecipe block categories and records which blocks are review-eligible versus excluded before the optional knowledge stage runs. The stage writes `08_nonrecipe_spans.json` and `08_nonrecipe_review_exclusions.jsonl`.

Knowledge job construction lives in [cookimport/llm/codex_farm_knowledge_jobs.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_jobs.py). That module writes `raw/llm/<workbook>/knowledge/in/*.json` from the review-eligible Stage 7 spans only. If upstream diversion improves, this module should see fewer clearly useless blocks, but its semantic authority should not otherwise change.

The current benchmark evidence for this plan comes from the Salt Fat cutdown run at `data/golden/benchmark-vs-golden/2026-03-22_20.16.29/` and the paired stage outputs at `data/output/2026-03-22_20.16.29/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-22_20.17.03/`. That run is only the last verified baseline. The user has already changed the code since then, so the plan must treat the counts as historical evidence and require fresh measurement before promotion.

“Divert more” means one concrete thing in this plan: increase `review_excluded_block_count` and shrink `review_eligible_block_count` by removing additional overwhelmingly obvious nonrecipe junk, while leaving the final semantic `knowledge` versus `other` decision to the knowledge stage for everything else.

## Plan of Work

Start in [cookimport/parsing/canonical_line_roles.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles.py). Expand `_outside_recipe_review_exclusion_reason(...)` and its helper functions so the first pass can exclude more low-risk junk families than it does today. The key gaps shown by the March 22 baseline are endorsement quote paragraphs, ebook-download and mailing-list prompts, generic publisher registration prompts, and short runs of publisher promo or purchase encouragement that currently remain review-eligible `OTHER`. The safest implementation path is to add narrow helper predicates for these families and then add cluster-aware promotion rules that only exclude them when either the line itself is unmistakable or nearby outside-recipe rows show the same family.

Do not solve this by teaching line-role “what useful knowledge is.” Solve it by teaching line-role more precise “what obvious junk looks like” patterns. Good candidates are:

- endorsement paragraph detection for quoted blurbs that praise the book rather than teach cooking, especially when they are followed or preceded by endorsement bylines;
- publisher/download/registration prompts containing strong product or signup language such as “downloaded this ebook”, “join our mailing list”, “click here to sign up”, “register this ebook”, “recommended reads”, or similar clearly promotional cues;
- strongly patterned copyright, edition, and publisher metadata lines that are already supposed to map to existing `publishing_metadata` or `copyright_legal`;
- packet-level promotion for dense junk clusters, such as a short run of endorsement quote paragraphs plus endorsement bylines, or a short run of signup boilerplate plus one imperative call to action.

Keep the reason vocabulary honest. If existing codes can still describe the new exclusions truthfully, reuse them. If not, introduce one or two new reason codes with plain names and wire them through normalization, prompts, Stage 7 summaries, and output docs. Do not add a large taxonomy. The artifact should remain easy for a reviewer to scan.

After the parser seam, update [cookimport/llm/canonical_line_role_prompt.py](/home/mcnal/projects/recipeimport/cookimport/llm/canonical_line_role_prompt.py) so workers are explicitly told to exclude these stronger junk families. The prompt should include examples of what now gets excluded and what must still stay review-eligible. For example, a quoted endorsement paragraph that merely praises the book should now be excludable when the packet context makes that obvious, while `How Salt Works` and explanatory cooking prose must remain review-eligible `OTHER`.

Then update Stage 7 surfaces in [cookimport/staging/nonrecipe_stage.py](/home/mcnal/projects/recipeimport/cookimport/staging/nonrecipe_stage.py) and the writer path that emits the staged outputs so the larger diversion remains inspectable. The exclusion ledger must clearly show which newly diverted families were filtered, and the summary counts in `08_nonrecipe_spans.json`, `knowledge_manifest.json`, and any benchmark-facing summaries must stay aligned.

Finally, clean up benchmark diagnostics that still tell the older semantic story. The historical `knowledge_budget.json` artifact is now misleading for this line of work because the metric a human really needs is “how much obvious junk was filtered before knowledge review.” Replace or supplement that surface with a routing-focused artifact that reports excluded block count, excluded reason counts, excluded examples, and the review-eligible remainder.

## Idempotence and Recovery

This plan is safe to implement incrementally. The safest order is to add new helper predicates and tests first, then widen the prompt text, then align Stage 7 and benchmark diagnostics. If one exclusion family proves too aggressive, remove or narrow only that family and keep the rest of the routing cleanup intact.

The work should remain idempotent because it changes only deterministic routing code, prompt text, tests, and docs. Re-running the same tests after a partial rollback should return the repo to a coherent state without needing to wipe caches or outputs.

Benchmark reruns are the only expensive and approval-sensitive step. If a rerun reveals quality loss, revert the offending exclusion family, keep the prompt and parser in sync, and re-validate locally before asking for another approved spend.
