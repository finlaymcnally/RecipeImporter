# Broadly Relax Main Workspace-Worker Watchdog

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The user asked for the broadest possible relaxation of the main workspace-worker watchdog because command-policy kills are ruining benchmark throughput and confidence in the runtime. After this change, a long-lived `workspace_worker` session in recipe, knowledge, or line-role should only be killed by command policy when it visibly invokes an explicitly off-contract executable such as `git`, `curl`, `wget`, `ssh`, `scp`, `pip`, `npm`, `docker`, `kubectl`, or similar package-manager / network / repo tools. Path-style policing, heredoc parseability, absolute-path scanning, parent-traversal string detection, and output-path restrictions on the main worker path all become telemetry-only concerns or disappear entirely from kill behavior.

The practical result should be that the models are allowed to do normal local shell work without being second-guessed by brittle regexes. If a worker still fails after this change, the dominant causes should be real logic errors, model quality issues, or the separate liveness guards, not watchdog command parsing.

## Context and Orientation

The enforcement seam is `detect_workspace_worker_boundary_violation(...)` in `cookimport/llm/codex_exec_runner.py`. Stage runtimes use that seam through their watchdog callbacks. `classify_workspace_worker_command(...)` is the telemetry-facing companion. Right now both functions still inherit path-centric assumptions from earlier tightening work, which means commands that merely look scary to the detector can kill a worker even when the executable is ordinary local shell tooling.

The relevant evidence chain is:

- `tests/llm/test_codex_exec_runner.py`, which still expects path-based command kills.
- `tests/llm/test_codex_exec_runner_workspace.py`, which already contains healthy local command shapes the runtime should continue to tolerate.
- `tests/llm/test_codex_farm_knowledge_orchestrator.py`, which still assumes the knowledge workspace-worker callback must kill absolute-path shell commands outside the execution root.
- `cookimport/llm/codex_exec_runner.py`, where executable checks are already clearly separated from path / heredoc policing.

## Plan of Work

First, lock in the target contract with regressions. Add or update tests so main workspace-worker detection allows the exact false-kill shapes seen in the March 22 benchmark family:

- multiple inline Python heredocs in one shell body
- heredoc shell writes that contain slash-heavy JSON or task payload text
- absolute-path shell commands on the main workspace-worker path
- parent-traversal shell paths on the main workspace-worker path

At the same time, keep proving that explicitly off-contract executables such as `pip`, `git`, or `curl` still trigger `forbidden_non_helper_executable`.

Second, simplify `detect_workspace_worker_boundary_violation(...)` so it only looks for forbidden executables in the visible command shape and returns `None` otherwise. The detector no longer needs to parse heredoc Python, classify path arguments, or reject absolute paths on the main workspace-worker path. Those helper functions can stay in the file if other code still uses them, but they stop driving kill behavior here.

Third, keep the classification surface honest enough for day-to-day debugging without reintroducing kills. Orientation-command classification should stay as-is because it is local telemetry, but path-based boundary violations should no longer be surfaced as kills for main workspace workers. The main requirement is behavioral: the supervision callback must stop terminating workers for shell/path style.

Fourth, update the short docs so they say what the code now does: main workspace-worker command enforcement is executable-only plus liveness guards; everything else is tolerated for completion flow.

## Validation and Acceptance

The change is accepted when:

- `detect_workspace_worker_boundary_violation(...)` returns `None` for the March 22 false-kill command shapes, including multi-heredoc Python shell bodies and heredoc writes with slash-heavy payloads.
- `detect_workspace_worker_boundary_violation(...)` still returns `forbidden_non_helper_executable` for clearly off-contract tools like `pip`, `git`, and `curl`.
- the knowledge workspace-worker watchdog callback no longer kills the absolute-path shell test that previously asserted `forbidden_absolute_path`.
- focused `.venv` tests covering the shared runtime and knowledge callback pass.

## Outcomes & Retrospective

Implemented. The main `workspace_worker` command watchdog is now intentionally blunt: if the visible command uses an explicitly off-contract executable, it still dies; otherwise command-policy enforcement returns `None` and lets the session continue. The shared runtime no longer kills main workers for absolute paths, parent-traversal-looking paths, output-path restrictions, heredoc parseability, or slash-heavy shell payload text.

The test suite now matches that contract. `tests/llm/test_codex_exec_runner.py` and `tests/llm/test_codex_exec_runner_workspace.py` cover both legacy tolerated local helpers and the exact March 22 false-kill shapes, while `tests/llm/test_codex_farm_knowledge_orchestrator.py` no longer expects the knowledge main-worker watchdog to kill an absolute-path shell command under `allow_workspace_commands=True`.
