## What was implemented

Recipeimport was updated to consume CodexFarm execution telemetry (including retry context, Heads Up details, normalized failure diagnostics, output fingerprints/previews, and codex event summaries) from `codex_exec_activity.csv` and persist compact telemetry payloads into recipeimport LLM artifacts.

## Code changes

- `cookimport/llm/codex_farm_runner.py`
  - Added `CodexFarmPipelineRunResult` return type for `run_pipeline(...)`.
  - Added helper `as_pipeline_run_result_payload(...)` for manifest/report serialization.
  - Added best-effort CSV telemetry ingestion keyed by `run_id` + `pipeline_id`.
  - Added command parsing for `--data-dir` resolution of CodexFarm telemetry CSV.
  - Included telemetry context in failure diagnostics without making telemetry mandatory.

- `cookimport/llm/fake_codex_farm_runner.py`
  - Updated fake runner to return `CodexFarmPipelineRunResult` for protocol parity.

- `cookimport/llm/codex_farm_orchestrator.py`
  - Captured pass-level runner metadata for pass1/pass2/pass3.
  - Persisted these under `process_runs` in recipe `llm_manifest` and `llm_report`.

- `cookimport/llm/codex_farm_knowledge_orchestrator.py`
  - Captured pass4 runner metadata.
  - Persisted under `process_run` in pass4 `llm_report`/manifest.

- `cookimport/tagging/codex_farm_tags_provider.py`
  - Captured pass5 runner metadata.
  - Persisted under `process_run` in pass5 `llm_report`/manifest.
