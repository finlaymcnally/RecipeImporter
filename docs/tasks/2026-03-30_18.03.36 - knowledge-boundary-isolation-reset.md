Problem statement

The live knowledge worker contract was still making `knowledge|other` and grouping decisions in one editable task-file pass. That let grouping pressure bleed into keep/drop classification for nonrecipe blocks.

Acceptance criteria

- Knowledge task files are split into a classification step (`nonrecipe_classify`) and a grouping-only step (`knowledge_group`).
- Classification answers use only `category` and `reviewer_category`.
- Grouping answers use only `group_key` and `topic_label`, and only accepted knowledge blocks enter that step.
- Final promoted shard outputs still validate against the existing packet output contract.
- Telemetry/docs describe the split semantic story and the new happy-path worker-session cap.

Evidence

- The new task-contract tests lock the split schemas and repair scoping.
- The runtime leasing/progress tests now show two happy-path knowledge sessions when grouping is needed.
- Prompt budget and stage observability tests still pass with the split telemetry story.

Constraints/gotchas + decision

- Keep the final promoted packet/proposal shape unchanged so downstream ingest/promotion code does not need a second migration.
- Reuse the generic editable task-file machinery, but allow per-step schema versions so the worker-facing contract is explicit.

Rollback / compensation plan

- Revert the knowledge task-contract/runtime/doc changes together. The old one-pass `nonrecipe_finalize` task-file contract is the prior coherent state.
