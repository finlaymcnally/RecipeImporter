## Purpose / Big Picture

The previous recipe recovery pass fixed correctness but not efficiency. The saved Salt Fat benchmark rerun at `data/golden/benchmark-vs-golden/2026-03-22_12.08.53/single-book-benchmark/saltfatacidheatcutdown/codexfarm/` shows recipe is back to `validated_shard_count: 5` and `watchdog_killed_shard_count: 0`, but it still spent `93` command executions and `473128` command-execution tokens. That is worse than both the failed churn-reduction run at `2026-03-22_04.26.00` (`91` commands) and the earlier good recipe baseline at `2026-03-22_03.33.50` (`64` commands, `453733` tokens).

The remaining waste is concentrated in three places. First, one recipe worker still dies on a false-positive startup command under the direct-exec workspace root in `~/.codex-recipe/...`, then recovers through five per-task structured retries. Second, the surviving recipe workers still spend too many turns manually `cat`-ing the manifest, contract, hints, and inputs before using `prepare-all`. Third, whole-shard early deaths still recover expensively through one follow-up call per task instead of a single packed retry.

After this change, the same recipe benchmark shape should keep the recovery win and also reclaim the command-count win. The target outcome is recipe `validated_shard_count: 5`, recipe `watchdog_killed_shard_count: 0`, recipe `watchdog_recovered_shard_count: 0` for this failure shape, recipe `command_execution_count_total` below `91` and then below `64`, and recipe `command_execution_tokens_total` below `473128` and ideally below `453733`, while line-role remains non-regressed.

## Surprises & Discoveries

- Observation: the recipe reliability fix worked exactly where it was intended.
  Evidence: `data/golden/benchmark-vs-golden/2026-03-22_12.08.53/.../prompt_budget_summary.json` reports recipe `validated_shard_count: 5`, `watchdog_killed_shard_count: 0`, and `watchdog_recovered_shard_count: 1`, while `.../worker-005/shards/recipe-shard-0004-r0020-r0024/status.json` records `status: "validated"` and `reason_code: "watchdog_retry_recovered"`.

- Observation: the remaining recipe recovery cost is dominated by one false-positive startup kill.
  Evidence: `.../worker-005/live_status.json` still ends in raw `state: "watchdog_killed"` and `reason_code: "watchdog_command_execution_forbidden"`, while `.../worker-005/status.json` shows `command_execution_count_total: 1`, `structured_followup_call_count: 5`, and `structured_followup_tokens_total: 90885`.

- Observation: the startup command that triggers the kill is not an off-road recipe edit; it is the runner bootstrapping inside the sterile execution workspace root.
  Evidence: `.../worker-005/live_status.json` records a forbidden command that starts with `/bin/bash -lc "cd /home/mcnal/.codex-recipe/... && cat worker_manifest.json ..."`; that absolute root is the execution workspace, not the mirrored worker root under `data/output/.../recipe_phase_runtime/workers/worker-005`.

- Observation: the current telemetry already believes that command should be workspace-local.
  Evidence: `.../worker-005/status.json` includes a first command-policy entry with `policy: "shell_script_workspace_local"` and `allowed: true`, which means the later kill path and the persisted policy summary disagree about the same startup command.

- Observation: the largest remaining command churn is plain exploration, not recovery.
  Evidence: `worker-001` through `worker-004` report command counts of `18`, `29`, `20`, and `25` with zero structured follow-ups, while `worker-002/events.jsonl` shows a long startup sequence of `cat OUTPUT_CONTRACT.md`, `cat worker_manifest.json`, `cat current_task.json`, `cat hints/*.md`, and `cat in/*.json` before the first `python3 tools/recipe_worker.py prepare-all --dest-dir scratch/`.

- Observation: the recipe helper surface is better than before but still does not make the low-turn path unavoidable enough.
  Evidence: the prior plan added `scratch/_prepared_drafts.json` and `stamp-status`, yet the new benchmark still shows workers manually reading per-task files before batch prep instead of starting from a single packed batch context.

- Observation: the current retry granularity is wrong for the recovered failure shape.
  Evidence: `worker-005` produced no validated main-session outputs before dying, but recovery still happened as five separate `inline_watchdog_retry` follow-ups, one for each task in the shard.

- Observation: recipe startup churn dropped cleanly into the existing direct-exec current-task bundle instead of needing a brand-new mirrored file family.
  Evidence: writing recipe-specific `CURRENT_TASK.md` and `CURRENT_TASK_FEEDBACK.md` before `run_workspace_worker(...)` was enough for `codex_exec_runner.py` to mirror them automatically, and the focused worker tests now assert those files appear in both the manifest contract and the workspace root.

- Observation: the shard-packed retry shape composes cleanly with the existing per-task aggregator as long as the recovered shard payload is split back into one-task payloads before final aggregation.
  Evidence: the new multi-task regression in `tests/llm/test_codex_farm_orchestrator.py` proves one retry call is enough when all task outputs are missing, while the final shard status still records task-level recovered entries under `task_watchdog_retry_status_by_task_id`.

## Context and Orientation

The current direct-exec flow has two relevant roots:

- the source worker root under `data/output/.../recipe_phase_runtime/workers/worker-XXX`
- the sterile execution workspace root under `~/.codex-recipe/...`

`cookimport/llm/codex_exec_runner.py` already prepares both and wraps supervision callbacks around the live session. The likely bug is that recipe-stage watchdog logic still receives or persists enough source-root information to classify steady-state commands correctly, but not enough execution-root context to avoid false positives during startup shell scripts. The first workstream is to make the recipe watchdog reason over the actual execution root used by the running command, not just the mirrored output root.

The recipe worker UX is split across the orchestrator prompt text, the worker manifest/current-task/control files, and `cookimport/llm/recipe_workspace_tools.py`. The current surface still encourages workers to inspect many individual files before starting batch prep. The next helper iteration should provide one packed startup artifact or command that exposes everything the worker needs for the whole shard: manifest summary, contract summary, task IDs, hints, and input paths or contents. The goal is not to hide the underlying files, but to make separate `cat` loops unnecessary in the common case.

The recovery loop today is task-scoped. That was good enough to normalize recovered recipe shards, but it is too expensive when the worker dies before producing any usable outputs at all. The orchestrator should distinguish between "some outputs exist, retry only missing tasks" and "the whole shard died early, use one packed shard retry." That distinction matters for cost and should be visible in worker status artifacts and benchmark summaries.

## Milestones

### Milestone 1: Remove the false-positive startup kill

The runner and recipe watchdog agree on the same allowed-root interpretation for startup shell scripts inside the direct-exec workspace. The exact `worker-005` startup command no longer dies as `watchdog_command_execution_forbidden`, and the runner test suite covers that regression directly.

### Milestone 2: Make recipe startup cheaper by default

Recipe workers get a single cheap way to see shard context and draft locations, then go straight to `prepare-all` and edit/finalize. The common worker path stops requiring repeated `cat` calls for manifest, contract, hints, and input files.

### Milestone 3: Compress recovery cost for early shard deaths

When a recipe worker dies before producing any usable outputs, recovery uses one shard-scoped or packed follow-up path instead of one structured retry per task. Recovered shard accounting remains correct and explicit.

### Milestone 4: Verify locally and leave benchmark rerun gated

Focused `.venv` tests pass. The plan records the exact benchmark rerun command to use later, but no new benchmark spend happens unless the user explicitly approves it.

## Plan of Work

### Workstream A: Execution-root-aware watchdog classification

Thread the real execution workspace root through the supervision snapshot or callback plumbing so recipe-stage policy decisions can distinguish between the mirrored worker root and the actual `~/.codex-recipe/...` root. Add a regression test that replays the saved startup command shape from `worker-005` and proves it is treated as workspace-local rather than forbidden. Keep outside-root and clearly forbidden commands rejected.

### Workstream B: Lower-command recipe startup surface

Add one packed startup artifact or command for recipe workers, likely generated alongside `prepare-all`, that summarizes the shard contract, task roster, hints, inputs, and prepared-draft locations. Update the recipe prompt and helper docs so the intended flow is:

1. read the one packed shard context
2. run `prepare-all`
3. edit drafts or use `stamp-status`
4. run `finalize-all`

Per-task `cat` exploration should remain available for debugging but stop being the cheapest default path.

### Workstream C: Shard-scoped retry packing

Detect the case where a recipe worker dies before producing any usable outputs and recover with one packed shard follow-up rather than one follow-up per task. Preserve the current task-level retry path for mixed-output failures where only some task outputs are missing or invalid. Update persisted shard and worker status files so recovered-via-packed-shard and recovered-via-per-task remain distinguishable.

### Workstream D: Regression tests and docs

Expand the focused runner, orchestrator, helper, and prompt tests to cover the new execution-root startup rule, packed startup context, and shard-scoped retry path. Update the short LLM docs once behavior changes land. Keep the benchmark rerun documented but still user-approval-gated.

- mixed-output failures still recover per task when only some outputs are missing or invalid
