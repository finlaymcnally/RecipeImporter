# Bundle Knowledge Prompts And Tighten Noise Routing

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a codex-enabled knowledge run should review larger, local sections of non-recipe text at once instead of sending one tiny chunk per prompt. The user-visible effect is that `prompt_preview` for a codex-shaped run should show far fewer knowledge prompts than today while preserving the ability for the LLM to refine final `knowledge` versus `other` authority from Stage 7 seed non-recipe spans.

Today the knowledge stage is too fragmented. On `data/golden/benchmark-vs-golden/2026-03-16_20.23.25/single-offline-benchmark/saltfatacidheatcutdown/line_role_only/`, the hypothetical codex-shaped preview wrote `324` knowledge prompts because almost every non-noise chunk became its own job. That is expensive, repeats prompt boilerplate hundreds of times, and deprives the model of neighboring context that would help it decide whether adjacent prose is all one coherent introduction, sidebar, or reference section.

## Surprises & Discoveries

- Observation: the knowledge-stage preview currently fans out from all seed Stage 7 non-recipe spans, not only deterministic `knowledge_spans`.
  Evidence: `cookimport/llm/prompt_preview.py` and `cookimport/llm/codex_farm_knowledge_orchestrator.py` both call `build_knowledge_jobs(...)` with `nonrecipe_stage_result.seed_nonrecipe_spans or nonrecipe_stage_result.nonrecipe_spans`.

- Observation: the job builder currently writes one Codex input file per surviving chunk.
  Evidence: `cookimport/llm/codex_farm_knowledge_jobs.py` iterates `for chunk in chunks:` and writes one `KnowledgeCompactJobInputV1` file for each chunk.

- Observation: the current skip gate only removes `noise`, so every chunk scored as `knowledge` survives into a Codex prompt.
  Evidence: `build_knowledge_jobs(...)` defaults `skip_suggested_lanes=("noise",)` and the `saltfatacidheatcutdown` preview produced `324` written jobs and only `85` skipped `noise` chunks.

- Observation: the compact knowledge contract and output schema are single-chunk contracts.
  Evidence: `cookimport/llm/codex_farm_knowledge_contracts.py` defines `KnowledgeCompactJobInputV1` with one `chunk`, and `cookimport/llm/codex_farm_knowledge_models.py` defines `KnowledgeOutputV1` with one `chunk_id`.

- Observation: bundling only within each individual Stage 7 seed span barely reduced fan-out on the motivating run.
  Evidence: the first bundled preview on `2026-03-16_20.23.25` still wrote `313` knowledge prompts for `320` surviving chunks, with only `7` multi-chunk prompts.

- Observation: packing contiguous chunks across neighboring seed spans is the lever that actually changes prompt count.
  Evidence: after switching the planner to global contiguous packing, the same codex-shaped preview dropped from `324` knowledge prompts to `91`, with `62` multi-chunk prompts and `320` surviving chunks packed into those `91` jobs.

## Decision Log

- Decision: keep the public pipeline id `recipe.knowledge.compact.v1` and evolve the contract underneath it rather than creating a new user-facing pipeline toggle.
  Rationale: this is a solo project, the CLI and defaults already point to `recipe.knowledge.compact.v1`, and the user wants the improved behavior rather than another menu/config branch.
  Date/Author: 2026-03-16 / Codex

- Decision: bundle local, adjacent chunks globally in block order, even when they come from neighboring seed non-recipe spans.
  Rationale: span-local bundling barely changed fan-out because Stage 7 seed spans are already fragmented. The right boundary is “contiguous local outside-recipe region,” not “single seed span.”
  Date/Author: 2026-03-16 / Codex

- Decision: tighten deterministic `noise` only for obvious junk such as blurbs, navigation, marketing copy, and attribution-only fragments, while leaving explanatory cooking prose eligible for review.
  Rationale: the user wants to trade tokens for quality, not to hide useful material behind aggressive heuristics. Deterministic suppression should only cover clearly low-value text.
  Date/Author: 2026-03-16 / Codex

- Decision: the new knowledge prompt should return per-chunk results inside one bundled output, and the ingest path should flatten those results back into the existing block-level refinement flow.
  Rationale: the refinement seam already operates on block indices. Bundled prompts should reduce overhead without changing the authoritative merge contract.
  Date/Author: 2026-03-16 / Codex

## Context and Orientation

The knowledge stage sits after deterministic non-recipe detection. Stage 7 produces a `NonRecipeStageResult` in `cookimport/staging/nonrecipe_stage.py`. That result contains seed non-recipe spans and a block-category map telling the rest of the pipeline which blocks are currently considered `knowledge` or `other`. The optional Codex knowledge stage in `cookimport/llm/codex_farm_knowledge_orchestrator.py` reviews those seed spans, can change the final `knowledge` versus `other` boundary, and writes reviewer-facing snippets to `09_knowledge_outputs.json` plus `knowledge/<slug>/`.

The important modules are:

`cookimport/parsing/chunks.py`: deterministic structure-first chunking for non-recipe text. This file decides chunk boundaries and assigns each chunk a suggested lane such as `knowledge` or `noise`.

`cookimport/llm/codex_farm_knowledge_jobs.py`: turns Stage 7 seed spans and chunking output into JSON input bundles for Codex. Today it writes one file per chunk.

`cookimport/llm/codex_farm_knowledge_contracts.py`: Pydantic input contracts for knowledge-stage bundles.

`cookimport/llm/codex_farm_knowledge_models.py`: Pydantic output contracts for Codex responses. Today this is one response per chunk.

`cookimport/llm/codex_farm_knowledge_ingest.py`: reads output JSON files back from Codex and validates them.

`cookimport/llm/codex_farm_knowledge_orchestrator.py`: end-to-end live knowledge-stage runner. It builds jobs, runs the pipeline, ingests outputs, and merges block-level decisions back into final non-recipe authority.

`cookimport/llm/prompt_preview.py`: zero-token reconstruction tool that must stay aligned with the live knowledge-stage contract so prompt counts answer “what would a codex-enabled run look like?”

`llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md` and `llm_pipelines/schemas/recipe.knowledge.v1.output.schema.json`: the prompt template and JSON schema enforced by codex-farm for the knowledge stage.

Tests already cover the current seams:

`tests/parsing/test_chunks.py` covers chunk lane heuristics.

`tests/llm/test_knowledge_job_bundles.py` covers job bundle shape and skip behavior.

`tests/llm/test_knowledge_output_ingest.py` covers output validation.

`tests/llm/test_codex_farm_knowledge_orchestrator.py` covers end-to-end orchestration with the fake runner.

`tests/llm/test_prompt_preview.py` covers prompt reconstruction and budget summaries.

## Plan of Work

Start with deterministic noise routing in `cookimport/parsing/chunks.py`. The current lane scorer is already able to label strong blurbs as `noise`, but it is too permissive for several obvious junk categories that should never consume Codex budget. Add narrowly-targeted helpers for cases that are clearly not cooking knowledge: endorsement or praise blurbs, navigation and contents fragments, signup or marketing copy, byline-only or attribution-only lines, and dedication or epigraph fragments. Keep the rules intentionally conservative. If the text contains real cooking guidance, technique language, storage guidance, substitutions, or tip-like phrasing, it should remain eligible for `knowledge`.

Then change the knowledge job builder from single-chunk jobs to local bundled jobs in `cookimport/llm/codex_farm_knowledge_jobs.py`. The new builder should still begin from `chunks_from_non_recipe_blocks(...)`, but instead of writing one file per chunk, it should group adjacent surviving chunks into local bundles. The final implementation should pack globally in block order, not only within one seed span, because neighboring seed spans are already fragmented. Bundling should be governed by explicit limits so prompts stay local and recoverable. The implementation should cap by both chunk count and approximate rendered character count, and it should force table-heavy chunks to stay isolated because table structure is brittle and already has special handling.

After that, replace the single-chunk input and output contracts. In `cookimport/llm/codex_farm_knowledge_contracts.py`, define bundled compact input models that carry a list of chunk payloads, per-chunk heuristics, and shared context/guardrails. The context should remain local to the bundle and should continue to identify nearby recipe blocks so the model can avoid leaking recipe content into outside-recipe authority. In `cookimport/llm/codex_farm_knowledge_models.py`, define a bundled output contract that returns an array of per-chunk results. Each per-chunk result should preserve the current information shape: whether the chunk is useful, the block-level `knowledge` versus `other` decisions, and any extracted snippets with evidence.

Update `cookimport/llm/codex_farm_knowledge_ingest.py` to parse the new bundled outputs and normalize them into a chunk-keyed map, because the rest of the refinement path already understands chunk-keyed block decisions.

Update `cookimport/llm/codex_farm_knowledge_orchestrator.py` to surface the new bundling metrics in the manifest. The report should include both job count and chunk count so prompt-preview and live runs remain easy to compare. The orchestrator’s merge logic should not change conceptually: it should still flatten all per-chunk block decisions and apply them through `refine_nonrecipe_stage_result(...)`.

Update the pipeline assets in `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md` and `llm_pipelines/schemas/recipe.knowledge.v1.output.schema.json` to describe bundled local review instead of single-chunk review. The prompt should explicitly tell Codex that it is looking at multiple neighboring chunks from the same outside-recipe region and must emit one result object per `chunk_id`.

Finally, update `cookimport/llm/prompt_preview.py` so hypothetical codex-shaped previews use the new job builder unchanged. That keeps preview aligned with the live orchestrator. Add tests that prove prompt preview now writes bundled knowledge prompts and that the `saltfatacidheatcutdown`-style fan-out meaningfully drops while line-role and recipe preview behavior remain unchanged.

## Validation and Acceptance

Acceptance is behavioral.

Deterministic noise routing is correct when new tests prove obvious junk such as blurbs, attribution-only lines, or navigation fragments become `ChunkLane.NOISE`, while real cooking-principles prose still remains `ChunkLane.KNOWLEDGE`.

Bundled knowledge jobs are correct when `tests/llm/test_knowledge_job_bundles.py` proves that one written input file can contain multiple chunk payloads, that bundling stays local and contiguous in block order, and that table-heavy chunks remain isolated.

Ingest is correct when `tests/llm/test_knowledge_output_ingest.py` proves the new bundled output shape parses successfully and normalizes into a consistent chunk-keyed result map.

Orchestration is correct when `tests/llm/test_codex_farm_knowledge_orchestrator.py` still shows that block-level authority refinement works, snippets are written, and the manifest reports bundle-aware counts.

Prompt preview is correct when the codex-shaped preview for `data/golden/benchmark-vs-golden/2026-03-16_20.23.25/.../line_role_only` no longer shows the current `324` knowledge prompts and the preview artifacts clearly show multi-chunk bundled knowledge inputs.

## Idempotence and Recovery

The deterministic tests and prompt-preview rebuilds are safe to rerun. `build_knowledge_jobs(...)` must remain idempotent: rerunning it on the same inputs should rewrite identical JSON payloads. If the bundled schema lands but the prompt or output parser fails, back out only the new pipeline asset change, keep the bundled builder/tests in place, and continue validating the builder and orchestrator against synthetic bundled outputs until the prompt/schema pair are aligned.
