# Task: CodexFarm spinner progress bridge

## Problem statement

RecipeImport only showed coarse codex-farm phase labels because `codex-farm process` was executed with blocking output capture, so stage and benchmark spinners could not display live codex-farm task progress.

## Acceptance criteria

- Subprocess codex-farm runner can consume `process --progress-events` stderr lines live and emit callback-safe status messages.
- Stage import and benchmark prediction-generation flows receive codex-farm progress via existing callback/spinner plumbing.
- Progress text includes `task X/Y` when totals are known.
- Runner falls back safely when installed codex-farm does not support `--progress-events`.
- Tests cover progress-event parsing path and unsupported-flag fallback path.

## Verification steps

Run from repo root with venv active:

```bash
. .venv/bin/activate
pytest tests/llm/test_codex_farm_orchestrator.py -q
```

## Evidence

- `cookimport/llm/codex_farm_runner.py` now supports streaming stderr progress event parsing and unsupported-flag retry behavior.
- `cookimport/llm/codex_farm_orchestrator.py` and `cookimport/llm/codex_farm_knowledge_orchestrator.py` now accept/pass progress callbacks.
- Stage/benchmark call sites now pass existing callbacks into codex-farm orchestrators (`cookimport/cli.py`, `cookimport/cli_worker.py`, `cookimport/labelstudio/ingest.py`).
- New tests in `tests/llm/test_codex_farm_orchestrator.py` assert event forwarding and fallback behavior.

## Constraints / gotchas and decision

- Constraint: keep compatibility with injected/fake runners used by tests and non-subprocess flows.
- Decision: only apply streaming progress-event logic in `SubprocessCodexFarmRunner`; injected custom runners continue unchanged.

## Rollback / compensation plan

- Revert this task diff to restore prior blocking `process --json` runner behavior and coarse phase-only spinner messages.
