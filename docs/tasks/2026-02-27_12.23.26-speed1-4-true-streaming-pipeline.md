This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This repository includes `docs/PLANS.md` at the repository root for ExecPlan rules. This plan must be maintained in accordance with that file. This plan also builds directly on `docs/plans/OGplan/speed1-4.md`, and resolves the remaining implementation gap that was identified in `docs/understandings/2026-02-27_10.51.52-speed1-4-implementation-gap-audit.md`.

## Purpose / Big Picture

After this change, `cookimport labelstudio-benchmark --execution-mode pipelined` will do real stage overlap instead of phase overlap. Today, prediction runs first and evaluation starts after prediction has fully finished. After this change, prediction records are consumed by an evaluation worker while prediction is still running, so pipeline behavior matches the original speed1-4 conception.

A user should be able to run the same benchmark in `legacy` and `pipelined` modes, see matching metrics, and verify from run telemetry/log ordering that evaluation consumption began before prediction completion.

## Surprises & Discoveries

- Observation: Current `run_pipelined(...)` in `cookimport/cli.py` already uses a bounded queue and producer thread, but it only collects records and returns a list; evaluation starts later in `labelstudio_benchmark(...)`.
  Evidence: `run_pipelined(...)` returns `(prediction_bundle, prediction_records, prewarmed_canonical_paths)` and evaluation is invoked only afterward.

- Observation: `evaluate_stage(...)` currently accepts file paths, not a `PredictionRecord` iterator.
  Evidence: `evaluate_stage(...)` signature is path-based (`stage_predictions_path`, `extracted_archive_path`), so streaming must either add a new iterator-based adapter or an incremental artifact builder boundary.

- Observation: Some canonical benchmark fixtures legitimately produce zero prediction records (`block_count=0`), so streaming replay reconstruction cannot assume non-empty input.
  Evidence: Existing canonical helper coverage includes zero-block fixtures; the pipelined consumer now falls back to original stage paths when no records are emitted.

## Context and Orientation

`labelstudio-benchmark` lives in `cookimport/cli.py`. It currently routes execution through:

- `predict_stage(...)` to emit per-block `PredictionRecord`s (`schema_kind=stage-block.v1`).
- `run_legacy(...)` for sequential behavior.
- `run_pipelined(...)` for bounded-queue prediction record production and canonical prewarm overlap.
- `evaluate_stage(...)` for mode-specific scoring.

Important current behavior:

- In `pipelined` mode, prediction, canonical prewarm, and record-consumption replay assembly overlap; final scoring still runs after EOS using replay artifact paths.
- `--predictions-out` writes all records after pipeline return.
- `--predictions-in` supports both per-block replay and legacy run-pointer replay.

Terms used in this plan:

- Producer means the prediction-stage worker that emits `PredictionRecord`s.
- Consumer means an evaluation-stage worker that reads those records as they are produced.
- End-of-stream means a sentinel message that explicitly signals no more records will arrive.
- Backpressure means the producer blocks when queue capacity is full, so memory does not grow unbounded.

## Plan of Work

Milestone 1 introduces a new streaming runtime boundary in `cookimport/cli.py` (or a small helper module under `cookimport/bench/` if extraction improves readability). The new runtime must run producer and consumer concurrently, with one bounded queue for records and one control channel for exceptions. It must keep today’s deterministic ordering guarantees and must propagate any producer or consumer failure as a run failure.

Milestone 2 implements stage-blocks streaming consumption. Add a consumer path that loads gold labels once, ingests records incrementally, and updates the same counts needed for the final report shape. Finalization writes the same report/artifact structure expected by existing callers. The goal is no metric drift and observable overlap, not a scoring algorithm rewrite.

Milestone 3 implements canonical-text streaming consumption in a safe way: while records stream in, build replay artifacts incrementally and complete canonical prewarm. At end-of-stream, call `evaluate_canonical_text(...)` exactly once using finalized artifacts. This preserves current canonical scoring semantics while still establishing a true consumer stage that runs during prediction.

Milestone 4 wires the new runtime into `labelstudio_benchmark(...)` for `execution_mode=pipelined` and preserves existing `legacy` and `predict-only` behavior. `--predictions-out` must still write complete, deterministic per-block records. `--predictions-in` replay behavior must remain unchanged, including legacy pointer compatibility.

Milestone 5 adds focused tests and docs updates. Tests must prove overlap ordering (consumer activity before producer completion), parity vs `legacy`, and clean error propagation. Update `docs/plans/OGplan/speed1-4.md` progress/outcomes and add a short understanding note if implementation reveals non-obvious constraints.

- `predict-only` still skips evaluation and writes prediction records.
- `--predictions-in` still supports both per-block records and legacy pointer records.
- Canonical-text scoring outputs remain unchanged relative to pre-change behavior for deterministic fixtures.
