# Restore Knowledge Shard Count Semantics

User-stated desires for this plan, preserved verbatim:

    "this is wrong. knowledge should be the same as the others. a shard is a slice of the work (1/N) that is given to an agent/worker. we need to fix this... make an execplan."

    "my understanding was that # shards = # workers and each shard was 1/N of the total work to be done, where N is the number of shards I pick in the CLI interactive menu"

Restated product intent:

The knowledge stage must obey the same operator-facing contract as recipe correction and block labelling. When the interactive CLI asks for a shard count, the chosen value `N` must mean "divide this stage into approximately `N` planned slices of work." Those planned shards are the units handed to workers. The runtime may still choose to run fewer workers than shards when concurrency is capped, but it must never silently reinterpret "shard count" as some other concept for knowledge while preserving the original meaning for the other stages.

In short, knowledge should stop being the exception. The CLI, settings, planner, preview tooling, manifests, and runtime summaries should all teach one consistent story: shard count controls how the total stage workload is partitioned, and worker count controls how many of those shards execute at the same time.

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

Today the interactive CLI teaches one meaning for "shard count" on the recipe and block-labelling stages, but a different meaning on the knowledge stage. For recipe and block labelling, the operator chooses how many shards to split the work into. For knowledge, the operator can only toggle the stage on or off, and the runtime silently uses one deterministic chunk per shard instead. That makes the menu misleading and breaks a simple mental model that should hold everywhere: a shard is one of `N` slices of the stage's total work, and workers are the separate question of how many of those shards run at the same time.

After this work lands, a human should be able to enable knowledge in the interactive CLI, enter a knowledge shard count just like the other Codex-backed stages, and trust that the runtime will partition the knowledge review workload into approximately that many shards. The visible proof is that interactive benchmark and import flows ask for a knowledge shard count, prompt preview and run manifests report that count, and the knowledge runtime artifacts for a stable fixture show a shard count driven by the chosen target rather than by the raw deterministic chunk count.

## Decision Log

- Decision: Restore a public `knowledge_prompt_target_count` field instead of inventing a new vocabulary term.
  Rationale: The existing recipe and line-role surfaces already teach `*_prompt_target_count` as the operator-facing shard-count knob. Reusing the same name pattern keeps settings, adapters, preview tooling, prompt budget summaries, and CLI help aligned. This also restores the mental model the user expects without creating a third name for the same concept.
  Date/Author: 2026-03-25 / Codex

- Decision: Keep `knowledge_worker_count` as a separate optional concurrency override.
  Rationale: The runtime already separates planning from execution for other stages. The shard count should answer "into how many pieces should the total work be divided," while the worker count should answer "how many of those pieces may run at once." Removing that distinction would keep the current confusion alive even if the menu started asking one more question.
  Date/Author: 2026-03-25 / Codex

- Decision: Do not change the meanings of `block`, `chunk`, `task`, or `worker`.
  Rationale: The terminology problem is specifically at the shard-planning layer. A `block` is still one normalized document unit, a `chunk` is still a coherent group of non-recipe blocks, a `task` is still the worker-runtime unit written into `assigned_tasks.json`, and a `worker` is still one concurrent execution slot. The fix is to let a shard own multiple chunks again when the chosen shard target calls for that.
  Date/Author: 2026-03-25 / Codex

- Decision: This plan intentionally supersedes the knowledge-specific shard-removal portion of the March 23 compatibility purge.
  Rationale: The earlier removal was internally consistent with a cost-driven "one chunk per shard" design, but it produced a product contract that is now explicitly rejected by the user. The repo should optimize cost without corrupting the operator-facing meaning of "shard."
  Date/Author: 2026-03-25 / Codex

- Decision: Keep knowledge task rows one-per-shard instead of splitting a multi-chunk shard into multiple worker task ids.
  Rationale: The existing knowledge runtime, proposal aggregation, and worker-side batch contract already accept one task payload that owns multiple chunk ids cleanly. Preserving that shape restored the operator-facing shard semantics with a smaller, safer diff than redesigning task ids and recovery ladders in the same change.
  Date/Author: 2026-03-25 / Codex

## Outcomes & Retrospective

Knowledge now behaves like the other shard-backed Codex stages again. `knowledge_prompt_target_count` is part of the canonical run-settings model, the interactive chooser prompts for it, `cf-debug preview-prompts` accepts it, and both preview and live planning partition the same review-eligible deterministic knowledge chunks into approximately that many shards before worker count is resolved.

The shipped runtime keeps worker count separate from shard count. Knowledge phase metadata now records both the configured prompt target and the configured worker override, and the focused planner/runtime tests show the intended split: for example, a ten-chunk fixture can plan four knowledge shards while still executing with two workers.

## Context and Orientation

This repository has three important layers for this change.

The first layer is the operator-facing run-settings surface. `cookimport/config/run_settings.py` defines the persistent settings model. `cookimport/config/run_settings_builders.py`, `cookimport/config/run_settings_adapters.py`, and `cookimport/config/run_settings_contracts.py` decide which settings move between interactive flows, saved config, CLI commands, and runtime adapters. `cookimport/cli_ui/run_settings_flow.py` owns the interactive "Workflow for this run" and per-surface shard-count prompts.

The second layer is knowledge planning. `cookimport/llm/codex_farm_knowledge_jobs.py` converts review-eligible non-recipe material into knowledge job bundles. In this repository, a `block` is one normalized document unit such as a paragraph, heading, or table row. A `chunk` is a coherent section of non-recipe blocks grouped by the deterministic chunker in `cookimport/parsing/chunks.py`. A `shard` is the planning-level LLM work unit represented by `ShardManifestEntryV1` in `cookimport/llm/phase_worker_runtime.py`. The current runtime writes one chunk per shard for knowledge, but that is a policy choice, not a fundamental type constraint.

The third layer is knowledge execution. `cookimport/llm/knowledge_stage/runtime.py` builds the knowledge jobs, resolves worker count, distributes shards to workers, and launches direct Codex worker assignments. In this repository, a `worker` is one concurrent execution slot. An `assignment` is the set of shard ids given to one worker. A `task` is the runtime row written into `assigned_tasks.json` for that worker. Knowledge tasks currently mirror shards one-to-one, which is why the current "one chunk per shard" design leaks into the queue shape.

There are two important adjacent surfaces. `cookimport/llm/prompt_preview.py` reconstructs predicted stage plans without running Codex and must show the same shard semantics as the live runtime. `cookimport/llm/prompt_budget.py` and benchmark manifests summarize how many shards and workers a run used; those summaries must not keep teaching "knowledge uses worker count while the other stages use shard count."

Next, change knowledge planning to honor the restored shard target. The correct home is `cookimport/llm/codex_farm_knowledge_jobs.py`. Keep deterministic chunking exactly as it is, because chunks are still the semantic building blocks of the stage. Change only the bundling policy so `build_knowledge_jobs(...)` can partition the ordered chunk list into approximately `knowledge_prompt_target_count` shards. The behavior should match the user expectation from recipe and line-role: choosing `N` means "split the total work into roughly `N` slices," not "launch `N` workers." Use the existing `target_chunks_per_shard` seam if helpful, but the end state should be driven by an explicit prompt target count rather than a private helper override. The plan must remain deterministic, preserve ordered chunk ownership, keep exact `owned_ids` and `owned_block_indices`, and continue isolating pathological cases such as very large table-heavy chunks when a single chunk cannot be sensibly merged.

Once the planner can build multi-chunk knowledge shards again, update the runtime in `cookimport/llm/knowledge_stage/runtime.py`. The runtime currently calls `build_knowledge_jobs(...)` with no shard target and resolves worker count from the final shard count. Change it so the planner receives `run_settings.knowledge_prompt_target_count`. Preserve the worker-count rule from the other stages: when `knowledge_worker_count` is unset, the runtime may default to `min(shard_count, cap)`, but that is an execution detail after planning, not the definition of the shards themselves. Review any code paths that currently assume one chunk per shard, especially task-manifest creation, progress accounting, deterministic bypass bookkeeping, and worker-side batch/task metadata. The implementation may keep "one task per shard" if that remains simplest, but if knowledge tasks continue to be the normal answer unit, then each task must now be able to own multiple chunk ids cleanly instead of assuming a single chunk everywhere.

After live planning is fixed, align the preview and summary surfaces. In `cookimport/llm/prompt_preview.py`, restore knowledge to the same planning contract as the other stages so previewed knowledge rows and stage plans are generated from `knowledge_prompt_target_count`, not from raw chunk count plus optional worker override. In `cookimport/llm/prompt_budget.py` and any stage-summary writers touched by the runtime, ensure the operator-facing metadata reports both the configured shard target and the resolved worker count. The UI and artifacts should make it impossible to confuse "how many slices was the work planned into" with "how many workers ran at once."

Then update the docs and short notes. `docs/02-cli/02-cli_README.md` must say that knowledge now prompts for a shard count in the interactive Codex menus. `docs/10-llm/10-llm_README.md` must stop claiming that knowledge always writes one shard per deterministic chunk and instead explain that chunks are still deterministic semantic groupings while shards are planner-controlled slices that may contain one or more chunks. Add a short note to the most relevant code-owner folder if a local README already exists there and its current wording would mislead a reviewer.

Finally, update focused tests around the old assumptions. The tests that matter most are the run-settings model tests, the interactive CLI prompt tests, the knowledge job-bundle tests, the knowledge runtime tests, the prompt-preview tests, and the benchmark-helper propagation tests. The critical acceptance is behavioral: a stable fixture with many knowledge chunks must be able to produce fewer shards than chunks when the chosen knowledge shard target is lower than the chunk count, and the resulting worker count must remain independently configurable.

## Validation and Acceptance

The change is correct only if all of the following are true.

Interactive acceptance means that, in an interactive benchmark or import run with CodexFarm enabled and knowledge turned on, the CLI asks for `Knowledge shard count for this run:` after the existing block-labelling and recipe prompts. Pressing `Esc` must still return to the previous step cleanly.

Planning acceptance means that for a deterministic fixture with more knowledge chunks than the chosen target, the knowledge planner produces approximately the requested number of shards rather than one shard per chunk. "Approximately" is important because the planner may still need to keep a huge chunk alone, but the operator-visible contract must remain "target shard count," not "always one chunk per shard."

Runtime acceptance means that the final worker count can differ from the chosen shard count. For example, a run may plan `10` knowledge shards and execute them with `4` workers when `knowledge_worker_count=4`, or with `10` workers when the worker override is unset and the phase cap permits that. Artifacts and summaries must show both numbers distinctly.

Preview acceptance means that `cookimport cf-debug preview-prompts` and any internal phase-plan readers no longer infer knowledge concurrency from chunk count alone. The preview output must reflect the configured knowledge shard target just as it already reflects recipe and line-role shard targets.

Documentation acceptance means that [docs/02-cli/02-cli_README.md](/home/mcnal/projects/recipeimport/docs/02-cli/02-cli_README.md) and [docs/10-llm/10-llm_README.md](/home/mcnal/projects/recipeimport/docs/10-llm/10-llm_README.md) no longer teach the removed one-chunk-per-shard knowledge contract.

## Idempotence and Recovery

This work should be implemented additively and is safe to retry. Adding the run-setting field, threading it through adapters, and restoring the CLI prompt are idempotent edits. The risk is semantic drift across the planner, runtime, preview, and docs, not data loss.

If the implementation reaches a halfway state where the setting exists but the runtime still ignores it, stop and fix that before merging anything else. A hidden or ignored shard-count knob is worse than no knob because it lies to the operator. If the planner change causes task-level validation assumptions to fail, keep the new run-setting field and CLI prompt on the branch but gate the runtime path until the multi-chunk shard assumptions are repaired in tests.
