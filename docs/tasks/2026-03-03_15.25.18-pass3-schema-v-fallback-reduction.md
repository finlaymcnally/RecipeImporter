# Pass3 Schema_v Fallback Reduction

## Problem statement

Recent SeaAndSmoke codex runs showed pass3 fallback dominated by `draft_v1` validation errors (`schema_v` missing). Several pass3 outputs contained legacy shapes (`name`/`instructions`, schema.org-only, or pass2-like payloads) instead of strict `RecipeDraftV1`.

## Acceptance criteria

- Legacy pass3 `draft_v1` object shapes are coerced into valid `RecipeDraftV1` payloads before validation.
- Missing `schema_v` no longer forces fallback when usable title/instruction evidence exists.
- Regression tests cover the observed legacy output families.

## Verification steps

- `source .venv/bin/activate && pytest tests/llm/test_codex_farm_orchestrator.py -k "legacy_pass3_draft_shape_without_schema_v or pass2_like_pass3_draft_shape_to_draft_v1"`
- `source .venv/bin/activate && pytest tests/llm/test_codex_farm_orchestrator.py`

## Evidence

- `source .venv/bin/activate && pytest tests/llm/test_codex_farm_orchestrator.py -k "legacy_pass3_draft_shape_without_schema_v or pass2_like_pass3_draft_shape_to_draft_v1"` -> `2 passed, 27 deselected, 1 warning in 3.31s`
- `source .venv/bin/activate && pytest tests/llm/test_codex_farm_orchestrator.py` -> `29 passed, 1 warning in 3.14s`
- `source .venv/bin/activate && pytest tests/llm/test_codex_farm_contracts.py` -> `10 passed in 0.09s`
- Replay check on previous failing Sea pass3 bundles (`c0,c2,c4,c5,c12,c13,c17`) now validates as `RecipeDraftV1` after normalization.

## Constraints/gotchas + decision

- Keep strict pass3 quality gates and deterministic fallback behavior unchanged.
- Decision: normalize/coerce pass3 `draft_v1` payloads in orchestrator before `RecipeDraftV1` validation, rather than loosening validation contracts.

## Rollback / compensation

- Revert normalization helpers/tests if this coercion masks quality regressions; pass3 will return to strict fallback-on-shape-mismatch behavior.
