# Inline Recipe Tags And Delete Standalone Tagging

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, recipe tags are produced as part of the same recipe-processing LLM call that already corrects the recipe. A person running `cookimport stage ...` will no longer have a second tags-specific pipeline, tags-specific CLI flags, or a separate `tags/` output tree. Instead, the recipe-correction pipeline will return tags directly, recipeimport will normalize those tags deterministically before outputs are written, and the final staged recipe JSON plus intermediate JSON-LD will already contain the normalized tags.

The user-visible effect is simple. New stage runs will contain tags in `final drafts/<workbook_slug>/r{index}.json` and matching `keywords` in `intermediate drafts/<workbook_slug>/r{index}.jsonld`, but there will be no standalone tags pass to enable, no `--llm-tags-pipeline`, no `tag-catalog` or `tag-recipes` commands, and no `raw/llm/<workbook_slug>/tags/` or `tags/<workbook_slug>/` artifacts. Prompt preview will still show one recipe surface, but the recipe prompts will now include the tagging instructions because tagging is part of recipe correction itself.

This plan intentionally burns the boats. It does not preserve compatibility shims for the old standalone tagging path. The old tags pipeline, tags catalog wiring, tags prompt/schema pack, tags stage artifacts, tags CLI, and their run settings are deleted in the same branch that inlines tagging into recipe correction.

## Surprises & Discoveries

- Observation: the best integration point for deterministic tag cleanup is before outputs are written, not after.
  Evidence: `cookimport/staging/import_session.py` already runs recipe correction, then knowledge work, then writes intermediate and final outputs from the in-memory `ConversionResult`. A normalization step can be inserted after `run_codex_farm_recipe_pipeline(...)` and before `write_intermediate_outputs(...)` / `write_draft_outputs(...)`.

- Observation: the current standalone tags pass duplicates recipe text in a second LLM request and then mutates outputs after the fact.
  Evidence: `cookimport/tagging/codex_farm_tags_provider.py::_build_job_payload(...)` sends title, description, notes, ingredients, and instructions again, and `cookimport/tagging/orchestrator.py` projects results back into already-written drafts and JSON-LD files.

- Observation: the current recipe-correction contract is already the authoritative place where the corrected recipe object is returned and validated.
  Evidence: `cookimport/llm/codex_farm_contracts.py` defines `MergedRecipeRepairInput` and `MergedRecipeRepairOutput`, and `cookimport/llm/codex_farm_orchestrator.py::_corrected_candidate_from_output(...)` copies the parsed output directly into the `RecipeCandidate`.

- Observation: if the old tags system is truly deleted, the removal surface is larger than `cookimport/tagging/`.
  Evidence: references also exist in `cookimport/cli.py`, `cookimport/config/run_settings.py`, `cookimport/config/codex_decision.py`, `cookimport/config/prediction_identity.py`, `cookimport/config/run_settings_adapters.py`, `cookimport/analytics/*`, `cookimport/llm/prompt_artifacts.py`, `llm_pipelines/README.md`, `llm_pipelines/pipelines/recipe.tags.v1.json`, `llm_pipelines/prompts/recipe.tags.v1.prompt.md`, `llm_pipelines/schemas/recipe.tags.v1.output.schema.json`, and tests/docs across `cli`, `llm`, `staging`, `tagging`, `analytics`, and `labelstudio`.

- Observation: cookbook-level semantic tag merging should not be solved inside recipeimport right now.
  Evidence: the immediate pain called out by the user is orthographic duplication such as capitalization and spelling variants. That can be handled by deterministic normalization at import time without recreating the old catalog-driven standalone subsystem.

  Rationale: the recipe-correction call already has the recipe text in context, so this avoids a second read of the same recipe and removes an entire runtime subsystem.

## Outcomes & Retrospective

The completed state for this plan is stricter than the prior embedded-tags work:

- one recipe LLM contract returns recipe corrections and raw tag selections together;
- one deterministic normalization step finalizes tags before any recipe outputs are written;
- final drafts and JSON-LD contain the normalized tags directly;
- there is no separate tags pipeline, tags CLI, tags run setting, tags raw IO tree, or tags sidecar report tree left in active runtime code.

Implemented results:

- recipe correction now receives a shared tagging guide and emits `selected_tags`;
- prompt preview reconstructs that larger recipe input without adding a fourth prompt surface;
- `cookimport/staging/import_session.py` normalizes tags before writer code serializes `recipe.tags` and JSON-LD `keywords`;
- `cookimport/tagging/`, `recipe.tags.v1`, tags CLI commands, tags-stage artifact handling, and tags run settings are deleted;
- tags-only docs were removed or rewritten instead of being preserved as compatibility guidance.

## Context and Orientation

There are three current code paths that matter.

The first is recipe correction. `cookimport/staging/import_session.py` calls `cookimport/llm/codex_farm_orchestrator.py::run_codex_farm_recipe_pipeline(...)` when `RunSettings.llm_recipe_pipeline` is enabled. That orchestrator builds one `MergedRecipeRepairInput` per recipe, sends it through CodexFarm using the `recipe.correction.compact.v1` pipeline pack, parses `MergedRecipeRepairOutput`, and updates the in-memory `ConversionResult`.

The second is output writing. After recipe correction and knowledge work complete, `cookimport/staging/import_session.py` calls `write_intermediate_outputs(...)` and `write_draft_outputs(...)` from `cookimport/staging/writer.py`. Those serializers read `RecipeCandidate` objects from the in-memory `ConversionResult` and write `intermediate drafts/<workbook_slug>/*.jsonld` and `final drafts/<workbook_slug>/*.json`.

The third is the old standalone tags subsystem. Today it lives under `cookimport/tagging/`, is wired into `cookimport/cli.py` via `run_stage_tagging_pass(...)`, uses its own CodexFarm pipeline pack `recipe.tags.v1`, and is configured through `llm_tags_pipeline`, `codex_farm_pipeline_tags`, and `tag_catalog_json`. It also exposes `tag-catalog` and `tag-recipes` CLI commands and writes `tags/` plus `raw/llm/.../tags/` artifacts. This entire subsystem is the deletion target for this plan.

For this plan, “inline tags” means the recipe-correction LLM response itself contains a list of selected tags. “Normalization” means a deterministic cleanup step that turns raw selected tags into stable output tags by trimming whitespace, lowercasing, collapsing punctuation, and deduplicating equivalent spellings. “Standalone tagging” means the old second-pass system under `cookimport/tagging/` plus all of its settings, prompt packs, docs, tests, and artifacts.

## Plan of Work

### Milestone 1: Extend The Recipe-Correction Contract To Return Tags

At the end of this milestone, recipe correction is the only LLM surface responsible for recipe tags. The contract in `cookimport/llm/codex_farm_contracts.py` will define a new structured tag payload on both input and output.

Add a small fixed tagging-guide structure to `MergedRecipeRepairInput`. This should not be the old catalog, shortlist, or category-to-candidate map. It should be a static guide that the model can understand cheaply. The guide should contain:

- a stable set of category keys such as `dish_type`, `protein`, `method`, `diet`, `occasion`, and other categories you want to preserve now;
- a one-sentence description per category;
- a few representative examples per category;
- a short instruction that zero tags in a category is valid and that the model must prefer grounded, obvious tags only.

Keep the guide compact and checked into code so prompt preview and live recipe runs share the same logic. A good home is a new helper module under `cookimport/llm/`, for example `cookimport/llm/recipe_tagging_guide.py`. Do not build the guide from the old tag catalog or from `cookimport/tagging/`.

Extend `MergedRecipeRepairOutput` with a tags field such as `selected_tags`. The output should use a strict array of objects rather than a freeform map so validation is simple and order is stable. A good shape is:

    {
      "category": "method",
      "label": "pressure cooker",
      "confidence": 0.92
    }

Do not ask the model for canonical cookbook tag IDs or catalog keys. It only needs to emit category plus human-readable label. `confidence` is optional but useful for audit; if it adds too much fragility, keep it nullable rather than removing the field later.

Update the recipe correction prompt pack in `llm_pipelines/prompts/recipe.correction.compact.v1.prompt.md` so the model is explicitly told to return both the corrected canonical recipe and grounded tags. Update `llm_pipelines/schemas/recipe.correction.v1.output.schema.json` to match the new structured output. Update `llm_pipelines/pipelines/recipe.correction.compact.v1.json` only if the description text needs to mention tags; the pipeline id stays the same.

In `cookimport/llm/codex_farm_orchestrator.py`, update `_build_recipe_correction_input(...)` to include the tagging guide and update `_corrected_candidate_from_output(...)` so the parsed tags are copied onto the `RecipeCandidate`. If you keep both raw and normalized tag forms, the `RecipeCandidate` should carry the raw LLM-selected tag objects temporarily and the final flat normalized `tags` field only after normalization. If `RecipeCandidate` should stay simpler, store only `tags` on the candidate and write the richer raw tag detail into the recipe-correction audit artifact instead.

Acceptance for this milestone is that a single recipe-correction response can produce both corrected recipe content and a raw tag selection without using `cookimport/tagging/` at all.

### Milestone 2: Add Deterministic Tag Normalization Before Output Writing

At the end of this milestone, recipeimport will normalize tag labels before final outputs are written, so simple variants such as capitalization or punctuation differences collapse into one stable output form.

Create a new deterministic normalization module outside the old `cookimport/tagging/` package. A good location is `cookimport/staging/recipe_tag_normalization.py` or `cookimport/llm/recipe_tag_normalization.py`. The module should define two responsibilities.

The first responsibility is per-tag normalization. Given one raw LLM tag label, normalize it by trimming whitespace, lowercasing, converting repeated whitespace to single spaces, replacing obvious separators like `_` and `-` with spaces when that keeps the same meaning, and deduplicating identical results. Keep this conservative. Do not attempt broad stemming, fuzzy matching, or semantic ontology inference in this repository.

The second responsibility is per-workbook collapse. Given all recipes in one `ConversionResult`, normalize every recipe’s tags through the same rules and build a workbook-level alias map so exact orthographic variants collapse consistently across the run. For example, if one recipe emits `Gluten Free` and another emits `gluten-free`, both should become `gluten free`. The module may emit a lightweight normalization report for debugging if that is useful, but it must not recreate the old `tags/` report tree.

Wire this step into `cookimport/staging/import_session.py` after recipe correction has updated `result` and before `write_intermediate_outputs(...)` and `write_draft_outputs(...)` run. This placement matters. It keeps tags inside normal recipe processing, applies the same normalized tags to both final and intermediate outputs, and avoids post-write mutation.

Update `cookimport/staging/draft_v1.py` and `cookimport/staging/jsonld.py` so they read the normalized recipe tags from the `RecipeCandidate` and serialize them directly. The final draft should continue to use `recipe.tags`. Intermediate JSON-LD should continue to use `keywords`, derived from the same normalized ordered list.

Decide whether to expose richer tag detail in final draft output. If you keep it, name it clearly, for example `recipe.tag_details`, and make it derived from the normalized pass rather than the removed standalone tags report format. If in doubt, ship only `recipe.tags` and keep raw detail in the recipe-correction audit artifact.

Acceptance for this milestone is a stage run where tags appear in final drafts and JSON-LD with no separate tags stage and where obvious case and punctuation variants are normalized consistently across the workbook.

## Idempotence and Recovery

The code changes in this plan are safe to apply iteratively. Contract and prompt changes can be made first, then normalization can be wired in, then the old standalone tags surface can be deleted. Tests should be run after each milestone so regressions are caught before broad deletions land.

The risky part is the deletion milestone because many tests and docs currently mention the old settings and commands. The safe recovery path is not to reintroduce compatibility shims. Instead, if a deletion breaks an unrelated surface, fix that surface to understand the new architecture or remove its tags-specific assumptions entirely.

If a broad deletion leaves the tree temporarily red, the next contributor should continue until all references are either removed or rewritten. Do not stop in an in-between state where both architectures exist.
