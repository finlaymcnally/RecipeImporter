# Burn The Boats Calm Worker Contract Reset

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan is intentionally sequenced after the fixed-assignment reset, the single editable task-file reset, and the non-lethal single-file runtime reset. Those earlier plans changed the runtime shape, visible-root shape, and supervision policy. This plan finishes the psychological contract. The worker should no longer feel like it is entering a trap-filled protocol maze. It should feel like a normal coding-model edit session: one file, one bounded job, deterministic validation after save, and no hidden state to discover on the happy path.

## Purpose / Big Picture

After this change, the live recipe, knowledge, and line-role worker prompts all teach the same calm story. `task.json` is the whole job. The worker should read it, edit only the answer fields, save, and stop. If a same-session helper exists, it should be explained as one narrow validate-and-advance seam, not as a second hidden control protocol. The prompt wording should stop sounding adversarial or booby-trapped while still preserving the hard boundary against real repo, network, and package-manager escapes.

The user-visible proof is straightforward. The shared worker `AGENTS.md` and stage-local prompts should state positive guarantees instead of only prohibitions: there is no extra hidden context to discover on the happy path, brief local rereads or false starts are not fatal, and deterministic validation happens after save. Focused tests should assert that this wording exists in the live prompt builders.

## Outcomes & Retrospective

This plan was implemented in the same turn it was authored. The runtime shape did not change again; the live contract wording did. The shared worker `AGENTS.md`, the recipe task-file prompt, the knowledge same-session task-file prompt, and the line-role worker prompt now all explicitly teach a calmer story: `task.json` is the whole job, there is no extra hidden control state to discover on the happy path, and brief local rereads or false starts should be corrected rather than treated as fatal.

The main lesson is that a one-file runtime can still produce fear-shaped behavior if the wording implies hidden protocol state or punishment-first supervision. The fix was not another architectural rewrite. It was making the existing runtime sound like the simple edit session it already is.

## Context And Orientation

The current live Codex Exec runtime for recipe, knowledge, and line-role already uses a single worker-visible `task.json` file. The earlier burn-the-boats plans removed queue leasing from the happy path, moved repair into deterministic validation, and demoted heuristic kills to warnings. Even so, the worker could still infer that the repo might be hiding important state or waiting to punish a small local mistake, because much of the prompt wording was inherited from the earlier queue-shaped era.

In this repository, “calm contract” means the live prompt teaches three things in plain language. First, `task.json` is the whole job on the happy path. Second, deterministic validation happens after the worker saves, so the worker does not need to preemptively browse for hidden repo state. Third, brief local rereads or false starts are acceptable and should be corrected rather than escalated into a defensive ritual.
