This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan is intentionally later and narrower: once the direct-batch contract exists and the tag gate is already demoting under-grounded rows, remove the remaining semantic overhead fields `reviewer_category` and `retrieval_concept` from the knowledge system entirely.

## Operator Intent (Structured) - DO NOT DELETE

The operator wants a hard simplification after the direct-batch contract and tag-gate demotion are already in place. The knowledge stage should stop asking workers to supply extra commentary metadata about why they chose a label or how they would name the concept. The only semantic classification asks that remain should be the ones that matter to final behavior. The operator explicitly does not want these fields preserved as compatibility metadata, debug ballast, or half-dead artifact leftovers. The program should judge the model by the actual keep/drop outcomes, the surviving grounding, the final grouped knowledge output, and whatever limited observable evidence already exists in worker files and group snippets.

## Purpose / Big Picture

After this change, the knowledge system will no longer carry `reviewer_category` or `retrieval_concept` anywhere in its live contract. Knowledge classification answers will become simpler: a unit is either `knowledge` or `other`, and a kept `knowledge` unit must still have valid grounding. Grouping will still produce `topic_label`, `group_id`, and evidence snippets, so the resulting knowledge output remains understandable to a human, but the stage will stop forcing the worker to create extra explanatory metadata for every kept or rejected block.

This matters because the two removed fields are currently adding semantic work without adding commensurate authority. `reviewer_category` is mostly redundant for kept rows because knowledge rows already must use `reviewer_category=knowledge`. `retrieval_concept` is more expensive: it pushes the worker to manufacture a short concept phrase while it is still deciding whether the block should be kept at all. In the sampled `saltfatacidheatcutdown` run, false-positive keeps reused vague phrases like `Fundamental cooking flavor techniques` across many unrelated blocks, which strongly suggests the field is acting as rationalization pressure rather than as useful evidence.

The user-visible proof is deterministic and local. After implementation, no live code, prompt, test fixture, or runtime artifact in the active program will mention `reviewer_category` or `retrieval_concept`. The knowledge task file will only ask for `category` and `grounding`. Promoted knowledge artifacts will still show grounded kept blocks and grouped outputs, but they will no longer show synthetic retrieval-concept strings or reviewer-category counts.

## Context and Orientation

The knowledge stage is the part of the pipeline that reviews outside-recipe candidate blocks and decides whether they survive as final `knowledge` or end up as final `other`. In this repository, “grounding” means linking a kept `knowledge` block to one or more known Cookbook tag keys, or to a valid proposed tag under an existing category. Grounding still matters after this plan. The two removed fields do not.

`reviewer_category` is currently a second label attached to each classification answer. For knowledge rows it must be exactly `knowledge`, which means it is redundant. For other rows it can say things like `front_matter` or `memoir_or_scene_setting`, but that classification is used only for reporting and test expectations, not for final authority.

`retrieval_concept` is currently a short phrase attached to each kept knowledge row. It shows up in previews and proposal rollups, but it is not required for final authority or grouping. It is a summary field, not a control field.

The main code paths that currently depend on these fields are:

`llm_pipelines/prompts/recipe.knowledge.packet.v1.prompt.md` is the worker-facing packet prompt. It currently teaches the worker to emit `rc` and `rt` in addition to final category and grounding.

`llm_pipelines/schemas/recipe.knowledge.packet.v1.output.schema.json` and `llm_pipelines/schemas/recipe.knowledge.v1.output.schema.json` describe the packet output contract. They must be updated so the simplified contract is explicit and grep-clean.

`cookimport/llm/codex_farm_knowledge_models.py` defines the canonical packet output types and the semantic packet serializer. This is where the removed fields are currently validated and serialized as compact keys.

`cookimport/llm/knowledge_stage/task_file_contracts.py` defines the direct worker task file for classification and grouping. It currently builds blank answers with `reviewer_category` and `retrieval_concept`, validates those keys, and threads them into validated answers.

`cookimport/llm/knowledge_stage/recovery.py` writes the worker instructions that mention those answer keys explicitly.

`cookimport/llm/knowledge_stage/promotion.py` collects decision rollups, reviewer-category counts, retrieval concepts, and grounding details for later stage artifacts.

`cookimport/llm/codex_farm_knowledge_writer.py` writes `knowledge_groups.json` and `knowledge.md`. It currently preserves `retrieval_concept` in the grounded-block preview.

`cookimport/staging/nonrecipe_stage.py` stores `reviewer_category_counts` and records `reviewer_category` in changed-block reports.

`cookimport/staging/writer.py` writes final nonrecipe authority artifacts, including `knowledge_grounding_by_block`, which currently includes retrieval concepts through the promotion layer.

`cookimport/llm/codex_exec_runner.py` and `cookimport/llm/fake_codex_farm_runner.py` synthesize knowledge answers for fallback and zero-token testing. They currently inject both removed fields.

The test surface is broad. The main affected tests are in `tests/llm/`, plus staging tests in `tests/staging/test_nonrecipe_stage.py`, and a couple of Label Studio fixture helpers in `tests/labelstudio/`.

## Milestones

### Milestone 1: Simplify the knowledge classification answer contract

At the end of this milestone, classification answers will only contain `category` and `grounding`. A kept row still needs valid grounding. An `other` row must still leave grounding empty. The worker will no longer be asked for `reviewer_category` or `retrieval_concept`, and the packet prompt/schema will no longer mention them.

The proof for this milestone is a deterministic classification task fixture whose `answer_schema.required_keys` only lists `category` and `grounding`, plus task-file validator tests that no longer accept or require the removed keys.

### Milestone 2: Remove the fields from runtime models and output normalization

At the end of this milestone, the canonical knowledge models, fake output builders, fallback direct-exec synthesis, packet ingest normalization, and handoff validation will all agree on the new slimmer shape. There will be no hidden place where the old fields are recreated after the worker stops emitting them.

The proof for this milestone is that knowledge worker outputs, fake runner outputs, and fallback direct-exec outputs all serialize without `reviewer_category` or `retrieval_concept`, and the ingest path still validates and promotes them.

### Milestone 3: Remove the fields from reporting and staged artifacts

At the end of this milestone, no live artifact writer or stage summary will preserve reviewer-category counts or retrieval-concept text. The remaining human-readable surfaces will be grouped knowledge output, group labels, snippets, and grounding.

The proof for this milestone is a deterministic nonrecipe stage fixture that writes `09_nonrecipe_authority.json`, `knowledge_groups.json`, and `knowledge.md` without the removed fields while preserving kept/dropped block behavior.

### Milestone 4: Update tests and docs until the active program is grep-clean

At the end of this milestone, the active code, tests, and current docs will no longer mention `reviewer_category` or `retrieval_concept`. Historical run artifacts and historical plans may still contain those strings, but the live program surface will not.

The proof for this milestone is a grep-based acceptance check over `cookimport`, `llm_pipelines`, `tests`, and current docs, plus passing targeted test suites.

## Plan of Work

Start at the contract edge, not at reporting. In `cookimport/llm/knowledge_stage/task_file_contracts.py`, change `_blank_classification_answer()` so it returns only `category` and `grounding`. Rewrite `_knowledge_classification_answer_schema()` so `required_keys` contains only those two keys and the examples no longer show `reviewer_category` or `retrieval_concept`. Remove the enum dependency on `ALLOWED_KNOWLEDGE_REVIEWER_CATEGORIES` entirely.

Still in `task_file_contracts.py`, rewrite the validator so it no longer reads, validates, or returns `reviewer_category` or `retrieval_concept`. The only semantic checks should be:

1. `category` must be `knowledge` or `other`.
2. `knowledge` rows must have surviving valid grounding after deterministic normalization.
3. `other` rows must have empty grounding.

All validation error names, repair feedback payloads, and invariance helpers that mention `reviewer_category` or `retrieval_concept` must be removed or renamed. The validated answer payload that flows into transition code should be the new minimal shape, not a superset with dead keys set to `None`.

Then update the prompt-facing contract. In `llm_pipelines/prompts/recipe.knowledge.packet.v1.prompt.md`, remove the internal reviewer-category section, remove all `rt` instructions, and rewrite the compact output-key description so each decision row carries only `i`, `c`, and `gr`. A kept row still requires grounding. The prompt should describe the simplified contract plainly: classify the block and, if and only if it is kept, ground it. Update both schema files under `llm_pipelines/schemas/` so they match the live prompt and no longer mention the removed keys.

After the task contract changes, update the model layer in `cookimport/llm/codex_farm_knowledge_models.py`. Remove `reviewer_category` and `retrieval_concept` from both `KnowledgeBlockDecisionV1` and `KnowledgeSemanticBlockDecisionV1`, delete `ALLOWED_KNOWLEDGE_REVIEWER_CATEGORIES`, and simplify validators accordingly. Update `serialize_canonical_knowledge_packet(...)` so decision rows only serialize `i`, `c`, and `gr`. Because this plan is burn-the-boats work, do not keep optional support for `rc` or `rt` on the new live models.

Mirror that simplification through the normalization/ingest seam in `cookimport/llm/codex_farm_knowledge_ingest.py`. The packet normalizer should accept only the new slimmer decision rows and should no longer produce canonical outputs that contain removed keys. If helper metadata or error classification logic still uses those names, replace it with category/grounding-based logic.

Update the runner seams next. In `cookimport/llm/codex_exec_runner.py`, remove both fields from the synthesized nonrecipe-classification fallback answer. ## Idempotence and Recovery

This refactor must land as one coherent contract cutover. Do not merge an intermediate state where prompts stop mentioning the fields but validators still require them, or where models drop them but reporting still expects them. Make the contract edits, runner edits, and reporting edits in one pass, then update tests immediately.

Because this is burn-the-boats work, do not add compatibility parsers, old-field aliases, or transitional serializers. If an old fixture breaks, rewrite the fixture to the new shape. If an old historical artifact becomes unreadable by active test helpers, update the helper or fixture so the tests only exercise the new live contract.

If the packet schema and model layers become temporarily inconsistent during the refactor, fix the schema and the canonical serializer before touching downstream promotion logic. The schema is the narrowest stable source of truth for the new contract.
