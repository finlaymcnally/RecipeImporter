Problem statement

The first rename pass fixed the public worker-contract surface, but active code, test files, helper modules, watchdog labels, and current docs still expose `workspace_worker` terminology. This change aligns the remaining implementation-facing active surface with the chosen `taskfile` / `packet` vocabulary so the repo uses one naming scheme consistently.

Acceptance criteria

- Active helper/module/function names that describe the `taskfile` worker contract use `taskfile` terminology instead of `workspace_worker`.
- Active watchdog labels and test expectations use `taskfile_v1` instead of `workspace_worker_v1`.
- Current docs and active test filenames use the aligned vocabulary where they describe the same contract.
- Historical logs and design-history docs may keep old names when they are recording historical rollout details rather than current behavior.

Verification steps

- Run focused pytest coverage for the direct-exec runner, taskfile progress helpers, recipe/knowledge/line-role runtimes, and prompt-budget/reporting seams touched by the rename.
- Run a targeted `rg` over active code/tests/current docs to confirm the retired `workspace_worker` naming no longer appears there except in preserved history/log contexts.

Evidence

- Pending implementation.

Constraints/gotchas + brief decision

- Do not overwrite unrelated in-flight edits already present in the worktree.
- Keep the rename on active code/tests/current docs first; historical logs can stay literal when they describe past states.
- If test file renames happen, update `tests/conftest.py` marker and slow-file routing in the same change.

Rollback / compensation plan

- Revert the rename atomically if any remaining import path, marker routing, or test discovery seam still depends on the old names.
