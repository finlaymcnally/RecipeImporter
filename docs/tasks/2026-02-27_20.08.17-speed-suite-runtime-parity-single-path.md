# Make SpeedSuite Use A Single Runtime Path Shared With Import And Benchmark

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This repository includes `docs/PLANS.md` at the repository root for ExecPlan rules. This plan must be maintained in accordance with that file.

This plan builds on `docs/plans/2026-02-27_18.51.16-speed-regression-benchmark-suite-from-pulled-goldens.md` and narrows scope to representativeness and drift prevention.

## Purpose / Big Picture

After this change, SpeedSuite remains easy to run but stops being a second implementation of import/benchmark runtime knobs. A speed run will execute the same stage/benchmark/all-method call-construction path used by real CLI workflows, with SpeedSuite adding only orchestration concerns (target discovery, repeats, timing capture, and comparison). This means one place to maintain runtime settings and a much lower chance of benchmark and import behavior drifting apart.

You will be able to prove this by running one speed suite and seeing a recorded run-settings signature in artifacts, then comparing two runs with a strict check that warns or fails when settings differ.

## Surprises & Discoveries

- Observation: SpeedSuite already executes production entrypoints (`cli.stage`, `cli.labelstudio_benchmark`, and all-method runner internals), so the main remaining risk is not separate execution code, but duplicated argument construction.
  Evidence: `cookimport/bench/speed_runner.py` delegates execution to CLI functions but still assembles scenario kwargs locally.

- Observation: Interactive import and interactive benchmark currently build large run-settings-derived kwargs independently in `cookimport/cli.py`.
  Evidence: `common_args` in interactive import and `benchmark_kwargs` in interactive benchmark each map many `RunSettings` fields manually.

- Observation: SpeedSuite run summaries currently do not include a canonical run-settings hash, which makes baseline/candidate comparisons vulnerable to accidental settings drift.
  Evidence: `summary.json` includes scenarios/repeats/sequence matcher but not a full `RunSettings` identity payload.

- Observation: A concrete drift path already existed in matched-target discovery; this was an early indicator that shared contracts should be explicit and test-protected.
  Evidence: Before the current refactor, `_resolve_all_method_targets` had duplicate source-hint fallback logic.

- Observation: Including default-valued kwargs in adapter output can still break compatibility expectations in tests/callers that assert call shape.
  Evidence: `test_interactive_benchmark_single_offline_mode_skips_credentials` failed when adapter started passing `allow_labelstudio_write=False`; removing that explicit key restored prior call contract.

- Observation: Settings-parity enforcement requires legacy-awareness; runs created before this change may lack `run_settings_hash`.
  Evidence: `compare_speed_runs` now treats missing/different hashes as mismatch by default, so old summaries without hash metadata are intentionally flagged unless override is used.

## Outcomes & Retrospective

Implemented.

Delivered outcomes:

- SpeedSuite, interactive import/benchmark, and import entrypoint wiring now share one run-settings mapping layer (`cookimport/config/run_settings_adapters.py`) for stage/benchmark kwargs.
- Speed-suite artifacts now persist effective settings identity (`run_settings`, `run_settings_summary`, `run_settings_hash`) in both `summary.json` and speed run manifest `run_config`.
- `bench speed-run` now resolves effective settings from `cookimport.json` or `--run-settings-file`, validates via `RunSettings.from_dict(...)`, and can optionally override matcher with `--sequence-matcher`.
- `bench speed-compare` now emits settings parity fields, reports mismatch explicitly in markdown/json, and defaults verdict to FAIL on mismatch unless `--allow-settings-mismatch` is passed.
- Focused regression tests covering speed runner/compare/CLI wiring and benchmark-helper compatibility passed after refactor.

Lessons learned:

- Adapter consolidation is high-leverage but caller contract shape still matters; direct-call tests quickly surface accidental kwargs drift.
- Persisting settings hash in artifacts is a low-cost guardrail that prevents misleading speed verdicts.

## Context and Orientation

The relevant modules are split across CLI command wiring, run-settings schema, and bench orchestration.

`cookimport/config/run_settings.py` defines `RunSettings`, including defaults, normalization, summary rendering, and stable hashing. This is the canonical schema for run knobs such as workers, EPUB extractor options, OCR options, and sequence matcher mode.

`cookimport/cli.py` now routes interactive import and interactive single-offline benchmark through `build_stage_call_kwargs_from_run_settings(...)` and `build_benchmark_call_kwargs_from_run_settings(...)` from `cookimport/config/run_settings_adapters.py`.

`cookimport/bench/speed_runner.py` now accepts one effective `RunSettings` object and uses the same adapter helpers for stage and benchmark scenarios; all-method scenario setup also consumes that same `RunSettings` object.

`cookimport/bench/speed_suite.py` handles speed target discovery and validation. It now also powers all-method target discovery via shared matching logic (already refactored).

`cookimport/bench/speed_compare.py` now compares scenario metrics and enforces run-settings parity by default using `run_settings_hash` fields from speed summaries.

Important term definitions used in this plan:

- Run-settings adapter: a helper that converts a `RunSettings` object into kwargs for one concrete runtime call (`stage` or `labelstudio_benchmark`).
- Settings parity: baseline and candidate runs were executed with equivalent effective run settings (same normalized settings hash).
- Orchestrator-only harness: a tool that schedules and measures calls but does not reimplement business behavior already owned by runtime entrypoints.

## Plan of Work

Milestone 1 creates shared run-settings adapters and removes duplicated kwargs assembly in interactive CLI flow. This milestone introduces a small shared adapter module (or shared helper section in `cookimport/cli.py`, whichever is simpler and clearer) that builds call kwargs for stage and benchmark entrypoints from `RunSettings`. Interactive import and interactive single-offline benchmark are migrated to call these helpers.

Milestone 2 migrates SpeedSuite scenario execution to use the same adapters. Stage and benchmark scenarios in `cookimport/bench/speed_runner.py` stop assembling their own runtime kwargs and instead call adapter helpers. All-method scenario setup also reads from the same effective run settings payload for benchmark-related knobs.

Milestone 3 adds explicit settings sourcing and artifact identity for SpeedSuite. `bench speed-run` will resolve effective settings from `cookimport.json` by default, optionally from a user-provided JSON settings file, then include `run_settings`, `run_settings_summary`, and `run_settings_hash` in speed-run summary/manifests.

Milestone 4 upgrades compare semantics for parity safety. `bench speed-compare` checks baseline/candidate settings hash and emits a clear mismatch section. The default behavior should fail comparison verdict when settings differ, with an escape hatch flag to allow mismatch when intentional.

Milestone 5 adds regression tests and docs. Tests verify that adapter helpers are used by interactive and speed paths, and that settings mismatch behavior in compare is explicit. Benchmark docs and runbook are updated with the new parity contract and commands.

## Validation and Acceptance

Acceptance is behavior-first, not code-structure-first.

SpeedSuite must be demonstrably representative of runtime behavior:

- Stage and benchmark scenario calls are assembled from shared run-settings adapters, not hand-built speed-only kwargs.
- Interactive import and interactive single-offline benchmark use the same adapter helpers.
- Adding or changing a `RunSettings` field requires updating one adapter layer and corresponding tests; there should not be separate mapping edits in interactive and speed paths.

Speed comparisons must be safe to interpret:

- `summary.json` includes `run_settings` payload, human summary string, and stable hash.
- `speed-compare` reports settings hash for baseline/candidate and flags mismatch explicitly.
- Default compare verdict fails on settings mismatch (unless override flag is used).

Policy acceptance:

- No implementation step enables recipe codex-farm parsing or any AI parsing/cleaning import path.
- Existing policy lock behavior remains enforced by `RunSettings` normalization and runtime checks.

Test acceptance:

- Focused bench and CLI tests pass for speed discover/run/compare plus interactive benchmark helper coverage.
- No new test file is added without marker mapping updates in `tests/conftest.py`.

## Idempotence and Recovery

All new speed artifacts remain timestamped and additive under `data/golden/bench/speed/`; rerunning does not overwrite prior runs.

If a refactor step breaks behavior, recovery is straightforward:

- Re-run focused tests after each milestone.
- Keep adapter introduction and caller migration in small commits so rollback can target one milestone without reverting unrelated work.

If settings-parity compare causes temporary workflow friction, use the explicit mismatch-override CLI flag for transitional runs, then remove mismatch dependency once baseline and candidate share settings again.
