## Purpose / Big Picture

After this change, `cf-debug` can answer pass4 knowledge-harvest follow-up questions deterministically instead of only line-role and pass1-pass3 recipe questions. A reviewer who already has `upload_bundle_v1` will be able to ask for pass4-specific evidence, and the follow-up packet can include a pass4 selector, a pass4 audit/export JSONL, and pass4 artifacts plus row locators back into the base bundle.

The concrete proof is that `cf-debug` should accept a pass4-aware selector flow and emit a follow-up pack containing both the existing files and a new pass4 audit file. The request-template/build-followup flow should also allow pass4 asks without hand-editing internals.

## Surprises & Discoveries

- Observation: `upload_bundle_index.json.navigation.row_locators.pass4_by_run` is not a safe primary join key for follow-up logic because the displayed `run_id` values repeat (`vanilla`, `codexfarm`) across books.
  Evidence: inspection of the March 6, 2026 group bundle showed repeated `run_id` values and mismatched-looking payload references when sampling rows directly.

- Observation: `run_diagnostics[*].source_key` is a hashed bundle-internal key in the March 6 group bundle, so a human-facing pass4 selector cannot rely on that field alone.
  Evidence: the same bundle exposes book-identifying text through `output_subdir` (`02_saltfatacidheatcutdown/codexfarm`) while `source_key` is an opaque hash.

## Decision Log

- Decision: Build pass4 follow-up selection around per-run `output_subdir` plus `source_key`, not around `run_id` alone.
  Rationale: `run_id` is not globally unique in multi-book bundles, but `output_subdir` and `source_key` are stable and already present in `analysis.pass4_knowledge`.
  Date/Author: 2026-03-06 / Codex

- Decision: Add a dedicated pass4 audit/export output instead of teaching `prompt_link_audit` about pass4.
  Rationale: the current prompt-link audit is atomic-line based and verifies line-role prompt linkage; pass4 is a run-level knowledge-harvest surface with different artifacts and questions.
  Date/Author: 2026-03-06 / Codex

- Decision: Keep internal `source_key` intact for compatibility, but make pass4 selectors human-addressable via book slug and `output_subdir`.
  Rationale: the existing bundle internals still need the hashed source key, but reviewers and operators need to target pass4 by recognizable book/run labels.
  Date/Author: 2026-03-06 / Codex

## Outcomes & Retrospective

Implemented. `cf-debug` now supports pass4-aware follow-up packets without disturbing the older line-role flow. The result is a separate run-level pass4 seam: selectors can target pass4 runs, `audit-pass4-knowledge` emits pass4 evidence rows, and `pack` / `build-followup` include that output when requested. The main lesson was that human-facing follow-up tools cannot rely on bundle-internal hashed keys; they need stable book/output identifiers.

## Context and Orientation

The existing follow-up tool lives in [cookimport/bench/followup_bundle.py](/home/mcnal/projects/recipeimport/cookimport/bench/followup_bundle.py). It reads `upload_bundle_v1/upload_bundle_index.json` and `upload_bundle_payload.jsonl`, then builds deterministic selectors, audits, and compact follow-up packets. Today the selector model is recipe or line-range based, and the supported audits are line-role oriented.

The CLI surface lives in [cookimport/cf_debug_cli.py](/home/mcnal/projects/recipeimport/cookimport/cf_debug_cli.py). It exposes `select-cases`, `build-followup`, `pack`, and related commands. Tests live in [tests/bench/test_cf_debug_cli.py](/home/mcnal/projects/recipeimport/tests/bench/test_cf_debug_cli.py).

For pass4, the base upload bundle now surfaces run-level knowledge information in `analysis.pass4_knowledge` and `navigation.row_locators.pass4_by_run`. The local benchmark run tree can also contain pass4 artifacts such as:

    <run>/prompts/prompt_type_samples_from_full_prompt_log.md
    <run>/prompts/prompt_task4_pass4_knowledge.txt
    <run>/prediction-run/prompt_budget_summary.json
    <run>/prediction-run/raw/llm/<workbook_slug>/pass4_knowledge_manifest.json

In this plan, a `pass4 selector` means a selector row that targets one benchmark run's knowledge-harvest artifacts rather than a recipe or line range. A `pass4 audit` means a JSONL export that summarizes pass4 status, row locators, and local file presence for the selected run.

## Plan of Work

First, extend `FollowupBundleContext` and `RunContext` so pass4 data can be resolved from `analysis.pass4_knowledge` plus local run files. That includes a helper that maps one bundle run to its pass4 artifact paths and payload locators without depending on non-unique `run_id` values.

Next, update selector construction so `select-cases` can emit pass4 selectors. The CLI should accept explicit pass4 selection inputs rather than trying to infer them from recipe deltas. The selector manifest should preserve the existing recipe and line-range behavior unchanged.

Then add a dedicated pass4 output path. In `cookimport/bench/followup_bundle.py`, define a `write_pass4_knowledge_audit(...)` style function that emits per-selector rows describing the run, enabled status, available artifacts, payload row locators, and local file manifests. Update `write_case_export(...)` so pass4 selectors also export the most relevant pass4 artifacts and references, and update `write_followup_pack(...)` plus `write_followup_request_packet(...)` so pass4 output participates in generated follow-up packets.

Finally, expose the new command in `cookimport/cf_debug_cli.py`, refresh benchmark/docs references, and add focused tests. The tests should prove that a pass4-aware selector round-trips through `select-cases`, that the dedicated pass4 audit file is written, and that `pack`/`build-followup` include pass4 outputs when requested.

## Validation and Acceptance

Acceptance is behavioral:

1. `cf-debug select-cases` can create selectors that target pass4 run-level evidence.
2. `cf-debug audit-pass4-knowledge` writes a JSONL file with one or more rows describing pass4 status, artifacts, and row locators for the selected run(s).
3. `cf-debug pack` writes the new pass4 audit file when selectors include pass4 rows, and the pack index reports it.
4. `cf-debug build-followup` accepts a request whose outputs include the new pass4 audit/export and writes the expected ask files.
5. Focused tests in `tests/bench/test_cf_debug_cli.py` pass inside `.venv`.

## Idempotence and Recovery

The changes are additive. Re-running the CLI commands should overwrite generated JSON/JSONL outputs deterministically. If a test fixture based on an existing bundle becomes stale, the safe recovery path is to regenerate that bundle with the existing upload-bundle builder, not to rerun a full benchmark.

## Artifacts and Notes
