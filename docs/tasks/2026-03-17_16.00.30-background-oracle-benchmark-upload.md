# Background Oracle Benchmark Upload

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document follows [docs/PLANS.md](docs/PLANS.md).

## Purpose / Big Picture

After this change, interactive benchmarks still trigger Oracle automatically, but the benchmark itself finishes immediately instead of hanging on browser automation. A user can run single-offline or multi-book single-profile benchmarking, see the normal wrap-up, and still get an Oracle PID plus on-disk log/metadata paths showing that the review started in the background.

## Surprises & Discoveries

- Observation: the previous “manual-only” fix had removed the blocking hang, but it also removed the whole point of benchmark auto-review for this workflow.
  Evidence: `docs/tasks/2026-03-17_15.39.42 - stop benchmark wrapup from blocking on oracle.md`.
- Observation: detached Oracle uploads cannot reuse the synchronous `TemporaryDirectory` staging path because payload shards would be deleted as soon as benchmark wrap-up returned.
  Evidence: `cookimport/bench/oracle_upload.py` previously staged browser shards inside `TemporaryDirectory(prefix="oracle-benchmark-upload-")`.

## Decision Log

- Decision: keep the detached-launch logic in `cookimport/bench/oracle_upload.py` and let `cookimport/cli.py` remain a thin best-effort wrapper.
  Rationale: the durable staging, command assembly, and log/metadata writing all belong with the Oracle transport code rather than being duplicated in interactive CLI helpers.
  Date/Author: 2026-03-17 / Codex
- Decision: store detached launch state under each bundle in `.oracle_upload_runs/<timestamp>/`.
  Rationale: the launch should stay colocated with the exact bundle it came from, and the hidden directory does not interfere with the three-file `upload_bundle_v1` contract.
  Date/Author: 2026-03-17 / Codex
- Decision: keep `cookimport bench oracle-upload <bundle>` as the explicit retry path even after re-enabling automatic background starts.
  Rationale: background start solves wrap-up latency, but manual replay is still needed when Oracle fails to launch or the browser session needs operator intervention.
  Date/Author: 2026-03-17 / Codex

## Outcomes & Retrospective

Interactive benchmark wrap-up now matches the intended operator flow again: Oracle starts automatically, but the benchmark no longer waits on it. The main extra complexity was not process detachment itself; it was making sharded browser inputs durable after the parent exits. The resulting behavior is narrowly scoped, tested, and leaves obvious log/metadata breadcrumbs next to each bundle.

## Context and Orientation

The wrap-up entrypoints live in `cookimport/cli.py`. Single-offline writes one top-level bundle under `single-offline-benchmark/<source_slug>/upload_bundle_v1/`. Multi-book single-profile writes many per-book bundles plus one top-level group bundle under `single-profile-benchmark/upload_bundle_v1/`; only that group bundle is auto-started in the background.

The Oracle transport lives in `cookimport/bench/oracle_upload.py`. That module already knew how to assemble browser commands, work around Oracle's 1 MB per-file limit by sharding oversized payloads, and run the synchronous CLI path. This plan extends that same module with a detached launch function that writes:

- `upload_bundle_v1/.oracle_upload_runs/<timestamp>/oracle_upload.log`
- `upload_bundle_v1/.oracle_upload_runs/<timestamp>/oracle_upload.json`
- `upload_bundle_v1/.oracle_upload_runs/<timestamp>/staging/*` when oversized files need shard staging

The interactive CLI only needs to call that helper best-effort and print the resulting PID/log paths.

## Implementation Notes

Add a detached-launch data model plus `start_oracle_benchmark_upload_background(...)` in `cookimport/bench/oracle_upload.py`. Reuse the existing prompt/build/sharding logic, but do not use `TemporaryDirectory` for the detached path. Instead, create a timestamped launch directory under the bundle, write the staged files there when sharding is required, write a short preamble into `oracle_upload.log`, then spawn Oracle with `subprocess.Popen(..., stdin=DEVNULL, stdout=log_handle, stderr=STDOUT, start_new_session=True)`.

In `cookimport/cli.py`, replace the manual-only helper with a best-effort `_start_benchmark_bundle_oracle_upload_background(...)` wrapper that resolves the bundle, calls the detached launcher, and prints the command, PID, log path, metadata path, and manual retry command. Keep the synchronous `bench oracle-upload` command unchanged.

Update the interactive helper tests so they assert the new background-launch wrapper instead of the removed manual-only hint. Add a direct transport test showing that oversized background uploads write persistent staged shard files and log/metadata outputs.
