# Split LLM runtime tests and tighten assertions

Problem statement:
The current LLM test suite still has oversized runtime-heavy modules, especially in the knowledge orchestrator and direct Codex exec areas. Some tests are structurally in the right domain but are crowded into hot files, and a few assertions are broad enough that they prove "something happened" rather than the specific runtime contract the test name claims.

Acceptance criteria:
- Knowledge orchestrator runtime/progress tests move into a dedicated focused module instead of staying inside the mega orchestrator file.
- Direct Codex exec workspace/runtime tests move into a dedicated focused module instead of staying inside the mixed helper file.
- The moved tests assert the concrete contract they claim to cover, not only broad existence or completion signals.
- Centralized test marker routing and short testing docs are updated for any new files.

Verification steps:
- `. .venv/bin/activate && pytest tests/llm/test_codex_farm_knowledge_orchestrator_runtime.py`
- `. .venv/bin/activate && pytest tests/llm/test_codex_exec_runner_workspace.py`
- `. .venv/bin/activate && pytest tests/llm/test_codex_farm_knowledge_orchestrator.py -k "not runtime and not progress and not concurrent and not packet"`
- `. .venv/bin/activate && pytest tests/llm/test_codex_exec_runner.py -k "not workspace and not subprocess and not prepare_direct_exec"`

Evidence:
- Before the split, the runtime/progress tests lived inside much larger files: `tests/llm/test_codex_farm_knowledge_orchestrator.py` and `tests/llm/test_codex_exec_runner.py`.
- Before the new files exist, direct targeted runs against their new nodeids fail because pytest cannot find those dedicated modules.

Constraints / gotchas + brief decision:
- Keep this pass test-only. Do not rewrite runtime code just to make the split look cleaner.
- Respect existing in-flight edits in the large LLM files; move only cohesive test seams and leave unrelated neighboring tests in place.
- Prefer the same reuse pattern already used by split orchestrator transport coverage: import the base module for shared imports/helpers instead of rebuilding support code.

Rollback / compensation plan:
- Revert the new focused test modules plus the matching deletions from the old files if the split makes targeted reruns or review worse rather than better.
