# Split Massive Test Files

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

`docs/PLANS.md` is checked into this repository and this document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the repo should no longer hide major test surfaces inside a handful of 3000-8000 line files. A contributor should be able to open a focused module for canonical line-role runtime recovery, stats dashboard rendering, benchmark cutdown upload bundles, or Label Studio ingest prediction-run behavior without wading through several unrelated subsystems first.

Success is visible in two ways. First, the outlier files are replaced by smaller seam-focused modules with shared helper support where needed. Second, pytest still collects those new modules under the correct domain markers and slow/smoke slices, and targeted runs for the moved tests still pass unchanged.

## Outcomes & Retrospective

The repo no longer has any collected test files above 3000 lines in the targeted areas. The worst outliers from parsing, analytics, bench, and Label Studio were split into focused files with local `*_support.py` helpers, and the marker/docs surfaces now describe those new seams.

The one real trap was seam integrity: a mechanical split dropped a `@pytest.mark.parametrize(...)` decorator in the canonical line-role runtime file. That was easy to repair, but it reinforced the need to run real targeted pytest commands after any bulk file move.

The second pass reduced the remaining >2000-line files to smaller modules as well, including QualitySuite, upload-bundle runtime, benchmark progress dashboard rendering, and Codex orchestrator support files. The only non-green verification surface left in this task is the orchestrator suite, and that failure reproduces with the original unsplit file, so it is not caused by the split.

## Plan of Work

Start by extracting support modules where the helpers are the real reason a test file ballooned. For canonical line-role, move the shared settings helpers, runtime fixture, progress-message helpers, and synthetic runner classes into a parsing-local support module, then create new collected files for workspace-watchdog and runtime-recovery behavior. Leave the remaining prompt, deterministic-label, compact-prompt, cache, and progress tests in a smaller `tests/parsing/test_canonical_line_roles.py`.

For stats dashboard, move the file-scoped sample CSV/report fixtures and browser harness helpers into `tests/analytics/stats_dashboard_support.py`. Split the class groups into separate collected files with names that state the seam directly, keeping each file focused on one class family and importing the shared helpers.

For benchmark cutdown, create a bench-local support module that owns the fixture builders and script loader. Keep a smaller `tests/bench/test_benchmark_cutdown_for_external_ai.py` as the smoke-oriented base suite, then move starter-pack, upload-bundle, and high-level multi-book assertions into neighboring collected files.

For Label Studio ingest, create a support module for the conversion/prediction-run fixture builders and split import-planning versus prediction-run/authority artifact seams into separate collected files. For benchmark helper artifacts, reuse the existing `benchmark_helper_support.py` import pattern and split prompt-log and prompt-budget tests into separate files with the same shared support import block.

For the second pass, keep using the same strategy. Add a bench-local support module for QualitySuite runner fixtures, an LLM-local support module for Codex orchestrator fixtures and fake workspace runners, extend the existing benchmark cutdown support module with the upload-bundle fixture builder, and split the benchmark progress suite into helper/ETA versus dashboard-rendering files without introducing a new support module.

After the file moves, update `tests/conftest.py` marker maps and slow/smoke sets, then update `docs/12-testing/12-testing_README.md` so the documented split seams match reality. Finish with targeted `pytest --collect-only` and focused test runs inside `.venv`.

## Validation and Acceptance

Acceptance is behavioral:

The new files must collect successfully under pytest, and their basenames must be wired into the expected markers. A domain run such as `pytest --collect-only -q -m parsing` should still discover the canonical line-role suites, while the new analytics, bench, and labelstudio files must appear in their own domain slices once `_FILE_MARKERS` is updated.

The moved tests must keep their previous behavior. Run focused pytest commands over the split files in each domain. The exact file list can change as seams settle, but the final verification must include the new parsing watchdog/runtime files, the new analytics split files, the bench cutdown split files, and the Label Studio split files.

When a focused verification failure appears in a behavior-heavy suite, prove whether it is split-induced or pre-existing by rerunning the original unsplit file against the same worktree. This plan already hit that case in the orchestrator suite, and the original file reproduced the same failure.

The docs must stop lying. `docs/12-testing/12-testing_README.md` should name the new split seams for canonical line-role, stats dashboard, benchmark cutdown helpers, and any Label Studio suite split by this task.

## Idempotence and Recovery

This refactor is safe to apply incrementally. Each split should leave the suite collecting before moving on to the next area. If a helper move breaks imports, restore the previous import in the affected test file and keep the support-module extraction smaller.

Because the work is test-only, rollback is straightforward: revert the moved files together with `tests/conftest.py` and the testing docs. No production migration or data repair is required.
