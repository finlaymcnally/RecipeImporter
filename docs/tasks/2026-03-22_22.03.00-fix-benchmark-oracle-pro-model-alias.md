# Task

## Problem

Benchmark Oracle turn-1 review lanes were launched with `--model gpt-5-pro`, but the local Oracle browser wrapper stack was remapping that alias through machine defaults to `GPT-5.4`.
On the March 22, 2026 `21:47:46` benchmark run, that meant both new review lanes landed on the wrong ChatGPT model even though RecipeImport's own launch logs still said `gpt-5-pro`.

## Acceptance criteria

- Benchmark Oracle browser uploads normalize ambiguous GPT-5 aliases to explicit browser-safe targets before invoking Oracle.
- Same-chat Oracle follow-up continues to avoid raw `gpt-5-pro` alias selection for explicit turn-2 overrides.
- Targeted bench tests cover the browser upload command and the follow-up continue-session command using the explicit normalized model target.

## Verification

- `./.venv/bin/pytest tests/bench/test_benchmark_oracle_upload.py tests/bench/test_oracle_followup.py -q`

## Evidence

- `data/golden/benchmark-vs-golden/2026-03-22_21.47.46/.../.oracle_upload_runs/*/oracle_upload.json` recorded `model: "gpt-5-pro"` while the corresponding Oracle session metadata under `~/.local/share/oracle/sessions/you-are-the-{quality,token}-lane/meta.json` recorded `browser.config.desiredModel: "GPT-5.4"`.
- The local Oracle wrapper stack sources `/home/mcnal/projects/shared/oracle-dev/oracle-model-defaults.sh`, which currently defaults `ORACLE_GENUINE_MODEL` / `ORACLE_PRO_MODEL` to `gpt-5.4`.
- After the fix, the focused bench/follow-up pytest suite passed locally.

## Constraints / decision

- Keep the fix local to RecipeImport instead of patching the shared Oracle wrapper stack.
- Normalize only the ambiguous browser aliases (`gpt-5-pro`, `gpt-5.1-pro`, `gpt-5`, `gpt-5.1`) to explicit browser-safe targets so the command metadata and actual picker target agree.
- Preserve non-GPT model ids and already-explicit GPT variants unchanged.

## Rollback / compensation

- Revert the benchmark/follow-up browser-model normalization in `cookimport/bench/oracle_upload.py` and `cookimport/bench/oracle_followup.py` if the local Oracle wrapper stack is later fixed to treat `gpt-5-pro` as a stable explicit Pro target again.
