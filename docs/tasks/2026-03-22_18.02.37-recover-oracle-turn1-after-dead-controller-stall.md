# Task

## Problem

Some benchmark Oracle uploads do not end as `assistant-timeout`, but still lose the local controller before the benchmark run captures the visible ChatGPT answer.
In that state the follow-up worker stays at `waiting_for_turn_1` forever because it only retries session capture for explicit `assistant-timeout`.
After recovering the answer, Oracle may also request follow-up data using exact recipe-stage keys from the benchmark packet such as `recipe_llm_correct_and_link` and `build_final_recipe`; the follow-up bridge must not reject those stage filters.

## Acceptance criteria

- Automatic benchmark follow-up retries turn-1 capture when Oracle session metadata is still `running` but the saved browser controller PID is already dead.
- A recovered answer from that dead-controller state updates the source Oracle status and allows automatic turn 2 to launch.
- Follow-up packet building accepts recipe-stage filters that Oracle can legitimately cite from the benchmark packet.

## Verification

- `./.venv/bin/pytest tests/bench/test_oracle_followup.py -q`

## Evidence

- `data/golden/benchmark-vs-golden/2026-03-22_16.52.22/.../.oracle_upload_runs/2026-03-22_16.59.06/oracle_auto_followup.json` remained at `waiting_for_turn_1`.
- The matching Oracle session store entry `~/.local/share/oracle/sessions/you-are-reviewing-a-benchmark-323/meta.json` still said `status: running` while its recorded `browser.runtime.controllerPid` no longer existed, and the benchmark-side `oracle_upload.log` still had no captured `Answer:` block even though the user could see the answer in ChatGPT.
- Once recovered, that same run's Oracle answer asked for `stage_filters: recipe_llm_correct_and_link, build_final_recipe`, and the bridge failed before turn 2 with `Unsupported stage filter(s): build_final_recipe, recipe_llm_correct_and_link`.

## Constraints / decision

- Keep the change narrow: reuse the existing turn-1 reattach path rather than inventing a new Oracle transport.
- Trigger that same recovery when the saved Oracle session is obviously stale (`status=running` plus dead controller PID), not only for `assistant-timeout`.
- Accept exact recipe-stage keys in follow-up selector manifests instead of forcing Oracle to translate them into a narrower custom vocabulary.
