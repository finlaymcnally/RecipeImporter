# Task

## Problem

Automatic Oracle turn 2 for benchmark follow-up can fail before sending because the follow-up bridge passes a raw model alias like `gpt-5-pro` into `oracle continue-session`.
Oracle's continuation browser path does not normalize that alias the same way the initial upload path does, so the ChatGPT model picker cannot match it inside the existing conversation.

## Acceptance criteria

- Automatic same-chat Oracle follow-up does not force the default turn-1 alias back into `continue-session` when the operator did not explicitly choose a turn-2 model.
- Manual or explicit turn-2 model overrides translate GPT browser aliases into browser-visible labels before calling `oracle continue-session`.
- Regression tests cover both the default no-override path and an explicit `gpt-5-pro` follow-up override.

## Verification

- `./.venv/bin/pytest tests/bench/test_oracle_followup.py -q`

## Evidence

- `data/golden/benchmark-vs-golden/2026-03-22_19.16.34/.../2026-03-22_20.03.30/oracle_upload.log` failed with `Unable to find model option matching "gpt-5-pro" in the model switcher`.
- The source run recovered turn 1 successfully, so the root cause was the turn-2 continuation launch, not turn-1 recovery.

## Constraints / decision

- Keep the fix local to RecipeImport's Oracle follow-up bridge; do not patch the shared Oracle CLI here.
- Preserve explicit operator model overrides, but treat the default same-chat follow-up case as "reuse the existing conversation model" instead of "force-select a raw alias again."
