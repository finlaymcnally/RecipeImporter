# Cut knowledge current-task turn overhead with a repo-owned micro-batch worker contract

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The knowledge stage now skips more obviously unhelpful source material, but the saved benchmark evidence still shows that too much spend is tied up in the worker-control loop itself. After this change, a knowledge worker should stop paying one full repo-owned handoff cycle for every single task packet. The worker-visible unit of work should become a small repo-owned batch of consecutive packets, so one startup read, one draft scaffold, one validator cycle, and one install cycle can cover several tasks at once while the repo still keeps task-level authority under the hood.

The user-visible result is that a future approved benchmark rerun should keep knowledge quality and full queue completion while reducing the number of worker turns, reducing wrapper/cached-context overhead, and eliminating the need for workers to invent their own bounded shell schedulers just to keep moving through a long queue. This plan assumes the separate telemetry-accounting fix has already landed, so post-change runtime acceptance should be judged from corrected knowledge spend counters rather than from the older duplicated row totals.

  Evidence: `data/output/2026-03-22_14.44.46/.../knowledge/workers/worker-001/status.json` shows `34` knowledge rows for one worker and a first row that still spends `63` commands on one worker session built around repeated `complete-current`, `check-current`, `install-current`, and sidecar rereads.

- Observation: the model is compensating for the packet-granular contract by inventing its own local batch scheduler.
  Evidence: the same worker trace includes a bounded shell loop beginning `for n in $(seq 1 40)` that repeatedly reads `current_task.json`, opens the hint/input files for the current task, runs `complete-current`, `check-current`, and `install-current`, and keeps going until the repo-owned current task disappears.

- Observation: the current sidecars are excellent for correctness but expensive as the unit of motion.
  Evidence: `cookimport/llm/knowledge_workspace_tools.py` renders `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`, and `current_task.json` around exactly one task, and the prompt in `cookimport/llm/codex_farm_knowledge_orchestrator.py` explicitly instructs the worker to reopen those files after every successful install.

- Observation: the current recipe traces provide a warning for knowledge too.
  Evidence: the newer recipe current-task loop improved correctness and control flow, but the corresponding benchmark regressed from `47` to `113` commands and from `536003` to `1329282` `command_execution_tokens_total`, showing that a per-task helper loop can easily increase spend even when it makes the worker safer.

- Observation: the downstream knowledge authority seam is already task-local and should not be replaced.
  Evidence: `docs/10-llm/nonrecipe_knowledge_review.md` states that the authoritative seam remains immutable `knowledge/in/*.json`, worker-local semantic packet results, repo-side validation, and promotion into the final staging outputs. The expensive part is the worker-facing control loop, not the downstream contract.

- Observation: one last runtime-control sync is required after the worker exits, not only during live supervision callbacks.
  Evidence: the first batch cut left source-root batch/task sidecars stale on the relaunch-cap regression until `cookimport/llm/codex_exec_runner.py` synced `_DIRECT_EXEC_RUNTIME_CONTROL_PATHS` once more after the workspace-worker subprocess finished.

## Context and Orientation

The live knowledge runtime still lives mainly in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, `cookimport/llm/knowledge_workspace_tools.py`, and `cookimport/llm/codex_exec_runner.py`. The orchestrator builds worker assignments, writes the worker prompt, validates worker outputs, decides when the repo advances the worker-visible state, and records why a worker session ended. The workspace-tools module now renders both the batch-first helper CLI and the worker-visible sidecars `CURRENT_BATCH.md`, `CURRENT_BATCH_FEEDBACK.md`, `current_batch.json`, plus compatibility `CURRENT_TASK*` fallback files for the first active task. The direct-exec runner mirrors the durable worker root into the sterile execution root and now also pushes the advanced batch/task sidecars back into the live workspace after each accepted install or final worker exit.

After this implementation, each worker assignment still owns many task packets, but the worker-visible unit is now one small repo-owned “micro-batch” of consecutive tasks at a time. A task packet is still one row in `assigned_tasks.json` plus its authoritative `in/<task_id>.json`, helper hint `hints/<task_id>.md`, and task-local `out/<task_id>.json`. The normal helper loop is now: read the current batch sidecars, run `python3 tools/knowledge_worker.py complete-batch`, edit the prewritten drafts under `scratch/current_batch/`, run `check-batch`, then run `install-batch`. `install-batch` accepts the longest valid prefix in order, rewrites the current batch around the first unaccepted task, and leaves the downstream authority seam unchanged at task-local `out/<task_id>.json`.

That design is correct but expensive when the queue is long. The saved benchmark at `data/output/2026-03-22_14.44.46/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-22_14.45.22/raw/llm/saltfatacidheatcutdown/knowledge/` shows the exact failure shape. The upstream refactor reduced knowledge tasks from `420` to `167`, but workers still pay one full sidecar/helper/validator loop per remaining packet. The worker traces also show the model inventing local bounded loops over `current_task.json`, which is a strong signal that the repo should own a batch-shaped loop instead of asking the worker to simulate one on top of the single-task contract.

This plan therefore changes the worker-visible control surface, not the downstream authority seam. It also does not change parser-owned review gating or line-role labeling rules; those belong to the separate line-role benchmark-recovery plan. A “micro-batch” in this plan means a small ordered run of consecutive task packets that the repo leases to the worker as one unit of visible work. A “batch sidecar” means the repo-written files that explain that current batch to the worker. A “batch scaffold” means the prewritten `scratch/` draft payload for the current batch. A “contiguous-prefix install” means the repo can accept the first N valid tasks from the batch in order even if task N+1 still fails, instead of forcing the worker to reinstall the whole batch from scratch.

## Milestones

### Milestone 1: Choose the batch contract and prove it fits the existing validator

The first milestone is a design-and-prototype milestone, not a code dump. The goal is to prove that the knowledge worker can move through several tasks at once without changing the downstream authority seam. At the end of this milestone, the repo will have an explicit micro-batch contract written down in code-adjacent terms: the exact batch files in the worker root, the exact helper verbs, the exact draft shape, and the exact install semantics.

This milestone must compare two batch acceptance shapes. The first is an atomic batch install: `check-batch` passes only when every task in the batch is valid, and `install-batch` writes all batch tasks at once. The second is a contiguous-prefix install: `check-batch` reports validity per task in order, and `install-batch` accepts the longest valid prefix before stopping at the first invalid task. The preferred result is contiguous-prefix install, but only if restart and status semantics stay local to the existing knowledge runtime.

Acceptance for this milestone is a concrete prototype note inside this plan that names the batch files, helper verbs, and install semantics well enough that a stateless agent can implement them without inventing details.

### Milestone 2: Add repo-owned batch sidecars and batch scaffolds

The second milestone changes what the worker sees. Instead of leading with one current task, the repo should write one current batch surface. The worker root should gain `current_batch.json`, `CURRENT_BATCH.md`, `CURRENT_BATCH_FEEDBACK.md`, and a batch scaffold such as `scratch/current_batch.json`. The existing single-task sidecars can remain temporarily as compatibility/debug surfaces, but the prompt should point to the batch surface as the normal path.

The batch sidecars must already include the information that workers keep reopening packet-local files to recover: task ids, result paths, owned chunk ids, block spans, strong-knowledge-cue flags, and short per-task summaries derived from the current hint/input files. Raw `hints/*.md` and `in/*.json` should stay available as fallbacks, not the expected first read for every task inside the batch.

Acceptance for this milestone is a generated worker root that exposes the batch files and makes the batch surface visibly sufficient for normal work.

### Milestone 3: Replace the task loop with batch helper verbs

The third milestone changes the paved road. The helper CLI in `tools/knowledge_worker.py` should gain batch verbs such as `current-batch`, `complete-batch`, `check-batch`, `install-batch`, `next-batch`, and `show-batch`. The old single-task verbs can remain as fallbacks during rollout, but the worker prompt and sidecars should steer the worker to the batch loop by default.

The important behavior change is that one scaffold/check/install cycle should now cover several consecutive tasks. The batch scaffold should be prewritten by the repo, not synthesized ad hoc by the worker. `check-batch` should validate each embedded task result with the existing task-level validator. `install-batch` should write accepted task-local outputs back to the same `out/<task_id>.json` paths the runtime already trusts.

Acceptance for this milestone is a local helper CLI that can scaffold, check, and install a batch without any benchmark spend and without changing the downstream output contract.

### Milestone 4: Teach the orchestrator and runner to advance by batch, not by task

The fourth milestone changes the runtime control flow. `cookimport/llm/codex_farm_knowledge_orchestrator.py` should plan micro-batches from the ordered task rows in each worker assignment, write the initial batch sidecars into the worker root, and advance from batch N to batch N+1 after each accepted install. `cookimport/llm/codex_exec_runner.py` should mirror the new batch files into the sterile execution root and push the advanced batch state back after install, just as it already does for the current-task bundle.

This milestone must also preserve restart safety. If the worker exits mid-batch, the repo should resume from the first unaccepted task inside that batch rather than replaying earlier accepted tasks. If contiguous-prefix install is chosen, the runtime must update both the batch state and the task ledger to make that restart position explicit.

Acceptance for this milestone is a deterministic local runtime test that shows the batch sidecars advancing, accepted tasks not being replayed unnecessarily, and the session finishing with the same task-level outputs as before.

### Milestone 5: Lock the lower-turn path into docs and tests, then compare on an approved rerun

The fifth milestone is the guardrail phase. Update the short LLM docs so reviewers can see that knowledge workers now drain a repo-owned batch rather than one task at a time. Add focused tests that make it hard to regress back to the packet-by-packet loop. Only after that local proof exists should an approved narrow benchmark rerun be considered.

The runtime acceptance bar for the approved rerun is simple: corrected knowledge spend should go down because corrected worker-turn counts go down, while validated knowledge outputs and batch/task-level correctness remain stable.

## Idempotence and Recovery

This work must be safe to stage additively. The first safe shape is parallel implementation: add batch sidecars and batch helper verbs while keeping the single-task current-task path available as a compatibility fallback. Do not delete the existing task-local validator or task-local output contract while the batch path is still being proven.

The batch planner must be deterministic. Given the same ordered `assigned_tasks.json` rows, it should produce the same micro-batches every time unless a user-facing setting explicitly changes the caps. That determinism matters for restart safety, saved-artifact debugging, and benchmark comparisons.

If contiguous-prefix install is adopted, the runtime must write enough repo-owned state that rerunning `install-batch` after an interruption does not duplicate already-accepted earlier tasks or lose track of the first unaccepted task. If atomic batch install is adopted instead, the plan must keep the batch size small enough that full-batch revalidation remains cheap and understandable.

If the batch path proves too invasive or fails local correctness tests, the safe fallback is not “do nothing.” The minimum fallback is still to keep the packed batch sidecars and batch scaffold so the worker sees several tasks at once, even if installation temporarily remains task-local under the hood. What must not return is the old model-facing contract where one repo-owned handoff cycle is the only official unit of work.

As with the rest of this repository’s benchmark work, no live rerun is allowed without explicit user approval.
