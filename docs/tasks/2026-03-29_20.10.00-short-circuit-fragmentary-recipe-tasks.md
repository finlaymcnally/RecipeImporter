## Problem

Recipe correction is sending obviously fragmentary recipe tasks into the workspace-worker LLM loop.
In the 2026-03-29 benchmark run, those tasks were already scaffolded as terminal fragmentary outputs but still consumed recipe-stage tokens.

## Acceptance Criteria

- Recipe runtime finalizes terminal fragmentary recipe tasks locally before launching workspace workers.
- Those terminal tasks do not appear in the worker queue or assigned task list sent to the workspace worker.
- Aggregated shard outputs still include both locally finalized fragmentary tasks and worker-produced repaired tasks.

## Verification

- `source .venv/bin/activate && pytest tests/llm/test_recipe_phase_workers.py -k fragmentary`

## Evidence

- Added regression coverage for a mixed shard containing one runnable recipe task and one deterministically fragmentary task.
- `source .venv/bin/activate && pytest tests/llm/test_recipe_phase_workers.py -k fragmentary -q`
- `source .venv/bin/activate && pytest tests/llm/test_recipe_phase_workers.py -q`

## Constraints / Gotchas

- Keep the existing recipe promotion/aggregation contract intact.
- Do not widen the change into broader recipe-span heuristics in this patch.

## Decision

- Use the repo-authored scaffold as the authority boundary: if the scaffold is already terminal (`fragmentary` or `not_a_recipe`), validate and install it locally instead of sending it to the LLM worker queue.
