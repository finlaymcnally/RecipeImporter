Problem statement

Recipe-phase direct Codex workers are overriding `CODEX_HOME` to a run-local `.codex-recipe` folder under `recipe_phase_runtime/`. That folder has no auth state, so recipe workers can hit OpenAI with no bearer token and fail with 401 even though the configured recipe Codex profile is authenticated.

Acceptance criteria

- Recipe-phase direct workers keep using the configured/authenticated recipe Codex home.
- The sterile direct-exec workspace still lives under that Codex home's `recipeimport-direct-exec-workspaces/`.
- Recipe runs no longer create or depend on a run-local `.codex-recipe` auth home.

Verification steps

- Run `source .venv/bin/activate && pytest -q tests/llm/test_recipe_phase_workers.py::test_recipe_phase_runtime_uses_configured_codex_home_for_sterile_exec_workspace`.
- Run `source .venv/bin/activate && pytest -q tests/llm/test_codex_exec_runner_workspace.py::test_workspace_worker_uses_configured_codex_home_even_under_output_run_root`.

Evidence

- Before the fix, the new auth-home regression test fails because recipe workers place `execution_working_dir` under `run_root/.codex-recipe/...`.
- After the fix, the recipe-path test passes and the execution workspace resolves under the configured recipe Codex home.
- The shared direct-exec runner now also has a transport-level regression test that proves a worker launched from a `data/output/...` path still isolates under the configured Codex home instead of under the run root.

Constraints/gotchas + brief decision

- Keep the sterile worker workspace behavior.
- Only remove the bad recipe-stage `CODEX_HOME` override; the direct-exec runner already owns workspace isolation.

Rollback / compensation plan

- Revert the recipe-stage env change and regression test together if this affects another caller unexpectedly.
