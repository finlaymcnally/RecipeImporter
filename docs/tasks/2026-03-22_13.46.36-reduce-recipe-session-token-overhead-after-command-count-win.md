# Reduce recipe session token overhead after the command-count win

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The previous recipe passes fixed the dangerous failures and then cut recipe command count from `93` to `47` on the saved Salt Fat benchmark slice at `data/golden/benchmark-vs-golden/2026-03-22_13.22.18/single-book-benchmark/saltfatacidheatcutdown/codexfarm/`. That is a real win, but the user still cannot point to a recipe run that is both reliable and cheap, because recipe `command_execution_tokens_total` stayed very high at `536003` and `tokens_total` stayed at `2421041`.

After this change, the recipe phase should still validate all `5` shards with `0` watchdog kills and `0` structured follow-up retries, but it should stop spending most of its session budget on repeated local rereads of contract files, examples, helper source, and per-task input bundles. A human should be able to inspect the saved worker status files and see a much shorter, cleaner path: open the repo-written current-task surface, edit the prewritten draft, and finalize, with raw input and helper-source inspection left as fallback behavior rather than the normal path.

## Surprises & Discoveries

- Observation: recipe command count is no longer the main problem.
  Evidence: `data/golden/benchmark-vs-golden/2026-03-22_13.22.18/.../prompt_budget_summary.json` reports recipe `command_execution_count_total: 47`, which beats the earlier `64`-command recipe baseline, but `command_execution_tokens_total` is still `536003`.

- Observation: recipe token spend is dominated by long-lived session reuse, not by large shell output.
  Evidence: the same summary reports recipe `command_execution_tokens_total: 536003` but overall recipe `tokens_total: 2421041`, `tokens_cached_input: 1138944`, and `wrapper_overhead_tokens: 1277608`. Those values show the cost is mostly session context carry-forward.

- Observation: the current prompt still teaches workers to reread too much before editing drafts.
  Evidence: `data/output/2026-03-22_13.22.18/.../recipe_phase_runtime/workers/worker-004/status.json` shows the prompt explicitly says “open `worker_manifest.json`, then `CURRENT_TASK.md`, then `current_task.json`, then `CURRENT_TASK_FEEDBACK.md`, then `OUTPUT_CONTRACT.md`,” then “open `hints/<task_id>.md` first, then `in/<task_id>.json`, then edit the `metadata.scratch_draft_path`.”

- Observation: one worker already shows the desired direction, but the plan still tolerates too much per-task rereading.
  Evidence: `worker-001/status.json` only uses `9` commands and does finish with one `finalize-all`, but even that “good” worker still rereads `hint + input + draft` for every task and spends `tokens_total: 146538` in its primary row, with `wrapper_overhead_tokens: 76599`.

- Observation: contract lookup is still leaking into examples and helper-source inspection.
  Evidence: `worker-004/status.json` and `worker-005/status.json` both read `examples/valid_*.json`, search `tools/recipe_worker.py`, and inspect slices of that file before `finalize-all`, even though the repo already writes `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`, `current_task.json`, and prewritten drafts.

- Observation: `in/<task_id>.json` is still functioning as a first-class orientation surface even though the draft already contains the output shape and identifiers.
  Evidence: `worker-001/status.json`, `worker-004/status.json`, and `worker-005/status.json` all show default commands that read `hints/<task_id>.md`, then `in/<task_id>.json`, then `scratch/<task_id>.json` for each task or for the whole shard.

- Observation: recipe reliability is currently healthy, so this next pass can target efficiency directly.
  Evidence: the same `2026-03-22_13.22.18` summary reports recipe `validated_shard_count: 5`, `watchdog_killed_shard_count: 0`, `watchdog_recovered_shard_count: 0`, and `structured_followup_call_count: 0`.

- Observation: once the repo-owned current-task controller advances through all written outputs, the correct final sidecar state is “queue complete,” not a stale copy of task 1.
  Evidence: the new recipe queue-controller regression now ends with no `current_task.json`, `CURRENT_TASK.md` saying “No current task is active,” and `CURRENT_TASK_FEEDBACK.md` preserving the queue overview while marking the assignment complete.

## Context and Orientation

Recipe correction in this repository lives under `cookimport/llm/`. The main orchestrator is `cookimport/llm/codex_farm_orchestrator.py`. It builds per-shard worker roots under `data/output/.../recipe_phase_runtime/workers/`, writes repo-owned task surfaces such as `worker_manifest.json`, `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`, `current_task.json`, `assigned_tasks.json`, and prewritten `scratch/*.json` drafts, then launches one long-lived workspace-worker session through `cookimport/llm/codex_exec_runner.py`.

A “workspace worker session” means one Codex session that can run local shell commands inside a bounded temporary working directory and can keep context across multiple turns. That persistence is why cheap local reads can still become expensive overall: every extra reread becomes another reasoning turn inside the same session, and the saved telemetry records that growth as cached-input and wrapper-overhead tokens.

Recipe helper commands live in `cookimport/llm/recipe_workspace_tools.py`. After this implementation the worker can still call the older fallback verbs such as `prepare-all`, `show`, `check`, `finalize`, `finalize-all`, and `stamp-status`, but the paved road is now the repo-owned current-task family: `current`, `next`, `check-current`, and `install-current`. The worker-visible docs are summarized in `cookimport/llm/README.md` and `docs/10-llm/10-llm_README.md`.

The saved evidence for this plan is in two places. First, the aggregate benchmark summary is `data/golden/benchmark-vs-golden/2026-03-22_13.22.18/single-book-benchmark/saltfatacidheatcutdown/codexfarm/prompt_budget_summary.json`. Second, the per-worker telemetry is in `data/output/2026-03-22_13.22.18/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-22_13.22.51/raw/llm/saltfatacidheatcutdown/recipe_phase_runtime/workers/worker-00{1,4,5}/status.json`.

This plan assumes the prior recipe passes remain in place: recipe already has prewritten drafts, startup false-positive kills are fixed, and early whole-shard watchdog recovery no longer burns five structured follow-up calls. Those are now the baseline to preserve while reducing session token overhead.

## Plan of Work

The first implementation milestone was to change the recipe worker surface from a shard-oriented “go inspect files until you feel oriented” loop into a repo-owned current-task loop. That landed by extending the recipe helper and orchestrator so the repo now exposes one authoritative current task at a time, validates accepted `out/<task_id>.json` files as they appear, and advances the current-task sidecars without requiring the model to reread `assigned_tasks.json`, hints, or input files to reconstruct queue state. The helper now exposes `current`, `next`, `check-current`, and `install-current` directly.

The second milestone is to compress the worker-visible contract into the repo-written sidecars. `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`, or a new recipe-specific sidecar should state the exact allowed compact keys, the allowed `st` and `mr` values the worker needs, and one tiny example each for repaired, fragmentary, and not-a-recipe outputs. Those examples should be small enough that the worker no longer needs to read `examples/*.json` or search `tools/recipe_worker.py` during normal operation. `OUTPUT_CONTRACT.md` can remain for debugging, but it should stop being required first-pass reading.

The third milestone was to demote raw task input to fallback-only. The worker now treats `scratch/<task_id>.json` plus the repo-written current-task brief as the default editing surface. `in/<task_id>.json` still exists for ambiguous cases, but the prompt, helper docs, and sidecar language now say plainly that workers should not open raw input unless the draft and brief leave something unresolved.

The fourth milestone was to add soft reread pressure in the recipe workspace loop. The runner already classified observed commands, and now recipe-specific rereads show up distinctly too: `OUTPUT_CONTRACT.md` / `examples/*.json` / `tools/recipe_worker.py` lookups classify as `recipe_contract_lookup_command`, while `hint + input + draft` bundle rereads classify as `recipe_task_bundle_read_command`. This landed as observability and steering, not as a new hard kill.

The fifth milestone is to update the short docs and prompt wording so the paved road matches the implementation. `cookimport/llm/README.md`, `docs/10-llm/10-llm_README.md`, and the new or updated ExecPlan should all describe the same happy path: open the repo-written current-task surface, edit the prewritten draft directly, install/finalize, and advance. The old `hint -> input -> draft` sequence must become explicit fallback behavior.

## Validation and Acceptance

Acceptance is about proving that recipe became cheaper without undoing the reliability work.

The first acceptance layer is local behavior. The recipe worker tests must show that the happy path no longer requires reading `OUTPUT_CONTRACT.md`, `examples/*.json`, or `tools/recipe_worker.py` before producing valid outputs. The prompt and helper tests should show that the default path is repo-owned current task surface -> prewritten scratch draft -> install/finalize -> next task, with `in/<task_id>.json` and debug helper verbs still available only as fallbacks.

The second acceptance layer is runner and orchestration behavior. The runner tests should show that repeated contract/example/helper-source rereads are classified distinctly from normal productive local commands, and the orchestrator tests should show that recipe still completes all owned tasks correctly while advancing through the queue with the new current-task flow.

The third acceptance layer is the benchmark, but only if the user later approves it. The accepted benchmark outcome is:

- recipe `validated_shard_count` remains `5`
- recipe `watchdog_killed_shard_count` remains `0`
- recipe `watchdog_recovered_shard_count` remains `0`
- recipe `structured_followup_call_count` remains `0`
- recipe `command_execution_count_total` does not regress above `47`
- recipe `command_execution_tokens_total` falls below `536003`, then below `473128`, and should target the older `453733` bar
- the saved worker status files stop showing normal-flow reads of `examples/*.json` and `tools/recipe_worker.py`
- the normal worker path stops showing `hint -> input -> draft` rereads for every task unless the worker hit a documented fallback case

## Idempotence and Recovery

This work is safe to stage incrementally. It is safe to land the prompt and current-task-surface changes first, then the helper verbs, then the runner reread classification, and finally the short docs. Each focused `.venv` test command above is safe to rerun as many times as needed.

If the repo-owned current-task loop proves too invasive for one pass, the safe fallback is to keep the existing shard worker shape but make the current-task sidecars rich enough that workers can finish from `CURRENT_TASK.md` plus `scratch/<task_id>.json` alone. That fallback should still remove example/helper-source inspection and should still reclassify contract rereads distinctly in telemetry.

If a hard anti-reread watchdog starts causing recipe regressions, back it down to soft classification plus prompt steering and keep the plan moving. The saved benchmark artifacts under `data/golden/benchmark-vs-golden/2026-03-22_13.22.18/` and `data/output/2026-03-22_13.22.18/` are evidence only and must remain unchanged.
