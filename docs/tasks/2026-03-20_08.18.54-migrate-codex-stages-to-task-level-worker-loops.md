# Migrate Codex Stages from Shard Answers to Task-Level Worker Loops

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The user-visible goal is to make RecipeImport behave like a bounded file-processing pipeline instead of a fragile series of giant follow-up prompts. After this change, the Codex-backed stages should no longer depend on “repair the whole shard” or “retry the whole shard” when a shard is too large or partly complete. Instead, each worker should receive many small explicit task files, process them in a loop, write one result file per task, and let the repository validate and join those results deterministically.

This matters because some cookbooks are large enough that a shard-level follow-up prompt can exceed safe prompt size or spend heavily while trying to restate a huge payload inline. The new behavior should let a run survive worker kills, resume from partial completion, and requeue only unfinished or invalid tasks. A human operator should be able to inspect a worker folder and see completed task outputs accumulating during the run instead of waiting for one monolithic final answer.

This plan is correct when all three active Codex-backed stages (`line_role`, `recipe`, and `knowledge`) use long-lived worker loops over task files as the primary execution model, when recovery is task-scoped by default, when deterministic joiners rebuild the existing stage outputs from validated task files, and when no normal recovery path requires restating an entire oversized shard inline.

- Observation: the current runtime already has most of the physical worker workspace shape needed for this redesign.
  Evidence: `cookimport/llm/codex_exec_runner.py` already supports `workspace_worker` mode, writes `worker_manifest.json`, mirrors local `in/`, `out/`, `debug/`, and `hints/` folders, and syncs worker-written `out/` files back into the repo-owned workspace.

- Observation: the main remaining mismatch is not “no worker loop exists.” The mismatch is that stage-owned recovery still assumes one full shard answer.
  Evidence: `cookimport/parsing/canonical_line_roles.py` still launches `_run_line_role_repair_attempt(...)` and `_run_line_role_watchdog_retry_attempt(...)` per shard, both using `runner.run_structured_prompt(...)` and both restating the whole shard’s owned rows inline.

- Observation: recipe, line-role, and knowledge already differ in details, but they all share the same bad assumption at the follow-up boundary: the answer unit is the shard.
  Evidence: `cookimport/llm/codex_farm_orchestrator.py` runs recipe repair per shard, `cookimport/parsing/canonical_line_roles.py` runs line-role retry/repair per shard, and `cookimport/llm/codex_farm_knowledge_orchestrator.py` runs knowledge retry/repair per shard.

- Observation: the shared runtime vocabulary in `cookimport/llm/phase_worker_runtime.py` is still shard-first, not task-first.
  Evidence: the current shared dataclasses are `PhaseManifestV1`, `ShardManifestEntryV1`, `WorkerAssignmentV1`, `ShardProposalV1`, and `WorkerExecutionReportV1`; there is no task manifest or task result abstraction yet.

- Observation: knowledge already contains a partial hint toward the desired future model.
  Evidence: knowledge has split-retry support that can derive smaller retry shards from a failed shard, which proves the codebase already accepts the idea that a failed shard does not have to be recovered as one giant unit.

- Observation: the user’s ingestion concern about preserving existing source structure is real, but it is orthogonal to this worker-runtime migration.
  Evidence: the current runtime problem is downstream of ingestion. Oversized shard repair can fail even if the source was parsed correctly. ## Outcomes & Retrospective

The plan is now implemented. The shared runtime and worker contract understand ordered task manifests, and all three active stages have crossed the architectural boundary from “shard = answer” to “shard = container, task = answer.” Line-role workers process bounded line packets, recipe workers process recipe-local tasks inside multi-recipe shards, and knowledge workers process bounded chunk packets inside multi-chunk shards. Deterministic code now rejoins validated task outputs into the existing shard proposal, telemetry, and promotion surfaces, while structured follow-up on the normal path is bounded to one affected task rather than one whole oversized shard.

## Context and Orientation

RecipeImport already has a mixed deterministic and LLM-backed pipeline. A deterministic stage extracts text and intermediate structures from the source cookbook. The active Codex-backed stages are then:

1. `line_role`, which reviews atomic lines and adjusts labels such as `RECIPE_TITLE`, `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `KNOWLEDGE`, and `OTHER`.
2. `recipe`, which takes grouped recipe candidates, corrects them into the intermediate recipe schema, emits ingredient-step linking, and returns tags.
3. `knowledge`, which reviews non-recipe spans and keeps useful cooking knowledge while rejecting noise or memoir-like filler.

The important current files are:

- `cookimport/llm/codex_exec_runner.py`, which owns the direct Codex subprocess boundary, the sterile worker workspace, and the split between `structured_json` and `workspace_worker` modes.
- `cookimport/llm/phase_worker_runtime.py`, which owns the shared shard-oriented manifest and worker-assignment dataclasses.
- `cookimport/parsing/canonical_line_roles.py`, which owns line-role planning, worker launch, validation, retry, repair, and promotion.
- `cookimport/llm/codex_farm_orchestrator.py`, which owns recipe correction planning, validation, repair, and deterministic final recipe assembly.
- `cookimport/llm/codex_farm_knowledge_orchestrator.py`, which owns knowledge job construction, validation, retry/split-retry/repair, and deterministic promotion into knowledge outputs.
- `docs/10-llm/10-llm_README.md`, which is the current code-verified LLM runtime reference and must be updated when behavior changes.

Define these terms plainly and use them consistently throughout the implementation:

- A “shard” is the larger ownership bucket already used by the runtime today. It groups related owned ids and is the unit assigned to a worker for planning, telemetry, and reviewer-facing reporting.
- A “task” is the new smaller work packet inside a shard. A task is what the worker actually completes one at a time.
- A “worker assignment” is the ordered set of shards and tasks owned by one long-lived worker session.
- A “deterministic joiner” is the ordinary repository code that validates many task outputs and merges them into the existing stage-wide outputs. The model is not the authority that assembles final stage outputs across many tasks.
- “Task-scoped recovery” means rerunning or repairing only one failed or unfinished task, not reconstructing a whole shard in a follow-up prompt.

The design target is simple. A worker should start from named local manifests, read small task files, write `out/<task_id>.json` as it goes, and continue until it has exhausted its assigned tasks or truly cannot proceed. The repository should then validate those per-task files, mark tasks complete or invalid, and either join them into shard-level artifacts or requeue only the remaining tasks. The main worker session should be the common path. Structured follow-up should be rare, bounded, and task-sized.

This plan intentionally preserves the existing stage semantics. Recipes should still flow through the intermediate schema and deterministic final assembly. Knowledge should still be promoted conservatively. Line-role should still produce one label per atomic line. The change is operational: smaller answer units, better persistence, and deterministic joins.

## Milestones

### Milestone 1: Shared task-runtime foundation

Add a task-oriented layer to the shared worker runtime without deleting the current shard-oriented dataclasses on day one. At the end of this milestone, the repository should be able to write an ordered task manifest for one worker, persist immutable task input files, harvest per-task output files, and report which tasks are complete, pending, invalid, or failed. This is the lowest-risk slice because it does not yet change stage semantics; it only changes how work is represented on disk and collected after a worker session.

The acceptance for this milestone is local and file-based. A fake or fixture-driven worker session should be able to complete tasks `t001`, `t002`, and `t003` while leaving `t004` absent; the repository should then mark only `t004` pending and preserve the validated outputs for the first three tasks.

### Milestone 2: Worker contract and workspace upgrade

Revise the workspace-worker contract so it explicitly tells the model to process many local task files rather than treating the assignment as one implicit prompt blob. At the end of this milestone, `worker_manifest.json` must describe tasks as first-class runtime objects, `assigned_tasks.json` must exist beside `assigned_shards.json`, and the worker prompt must instruct the agent to iterate over tasks, write one output file per task, and stop once all assigned tasks are complete.

The acceptance for this milestone is observable in the worker artifact tree. A human should be able to open a worker folder and understand, without reading source code, which task files were assigned, which outputs were produced, and which tasks remain pending.

### Milestone 3: Line-role task packets

Convert line-role from “one shard answer” to “many line packets inside a shard.” At the end of this milestone, a line-role worker assignment should contain several small task files, each owning a bounded run of atomic lines. The worker should write one output file per task, and the stage should join those validated packet outputs back into the stage-wide line-role predictions. Whole-shard repair should disappear as the ordinary fallback path. If one packet is invalid or missing, only that packet should be retried or repaired.

The acceptance for this milestone is that a simulated partial worker run still promotes all valid completed line packets and requeues only the unfinished or invalid packets. No recovery prompt should inline the text of an entire large line-role shard.

### Milestone 4: Recipe task packets

Convert recipe correction to task files that usually own one recipe candidate each. At the end of this milestone, recipe workers should loop over recipe task files, write one corrected output per task, and leave deterministic code to assemble final recipe JSON and JSON-LD from those validated task outputs. The ingredient-step linking and tags should remain part of the task output contract, but the worker should never need to emit one giant shard blob just because several recipes share a worker assignment.

The acceptance for this milestone is that a worker can finish some recipe tasks, be killed, and on resume the repository only requeues the remaining recipe tasks. Final recipe outputs should still match the current staged contracts once all owned recipe tasks are validated.

### Milestone 5: Knowledge task packets

Convert knowledge review to task files that own small groups of nearby chunk ids. At the end of this milestone, oversized knowledge recovery should become ordinary task requeue rather than “split a failed shard into smaller retry shards and restate them inline.” The validator should still be conservative, but the worker should operate on small task packets with compact local evidence, not giant follow-up prompts.

The acceptance for this milestone is that a large failed knowledge shard no longer triggers one monolithic strict retry. Instead, completed knowledge tasks remain complete, unfinished tasks remain pending, and any structured follow-up that still exists operates on one bounded task packet only.

### Milestone 6: Docs, telemetry, and compatibility joiners

Update the runtime docs and stage summaries so the repo tells the truth. At the end of this milestone, `docs/10-llm/10-llm_README.md` and any nearby README or benchmark helper docs should describe tasks, worker sessions, pending/resumed work, and task-scoped recovery. Existing external readers that expect shard-level proposals should continue to work via deterministic compatibility joiners that aggregate task outputs into the current proposal/report surfaces.

The acceptance for this milestone is that an operator can inspect a finished run and see both the worker-session level view and the task-level view without losing the existing stage outputs or reviewer artifacts.

Do not delete the old shard-level artifact writers yet. Instead, add compatibility joiners that can derive shard-level proposals from validated task outputs until all readers have moved.

Next, update `cookimport/llm/codex_exec_runner.py`. The current worker-mode workspace already mirrors files and syncs outputs back, so do not replace it. Evolve it. `worker_manifest.json` must now include a task-oriented section that names:

- worker id
- `assigned_tasks_path`
- allowed input and output folders
- the required output filename pattern

Update the worker-facing prompt text for `workspace_worker` mode so it says, in plain language, to:

- open `worker_manifest.json`
- open `assigned_tasks.json`
- iterate tasks in order unless a task file says otherwise
- read `in/<task_id>.json` directly
- write exactly one `out/<task_id>.json` per finished task
- continue until all tasks are complete or truly blocked

Keep the repository and network isolation rules. The worker should still stay inside the bounded workspace. The difference is that reading local task files is now the intended workflow, not something to avoid.

Then migrate line-role in `cookimport/parsing/canonical_line_roles.py`. Preserve the existing deterministic baseline and final prediction surfaces, but introduce task packets inside each shard. A line-role task should own a contiguous packet of atomic lines. Use a conservative target size such as 40 to 120 owned lines per task, with local neighboring text included in the task payload itself so each task is semantically legible without restating a huge shard. The task output should contain one final label row per owned `atomic_index` in that packet. The validator should continue to enforce exact owned-id coverage and allowed labels. The deterministic joiner should merge validated line-role task outputs into the current shard proposal and final prediction surfaces.

Remove shard-sized repair as the default path. If a line-role task output is missing or invalid, first requeue just that task for another worker assignment. If one strict structured follow-up is still needed for a malformed task output, the prompt must include only that task packet, not the whole parent shard.

Then migrate recipe correction in `cookimport/llm/codex_farm_orchestrator.py`. A recipe task should usually own one `recipe_id`. Only use multi-recipe tasks for tiny nearby recipes where the total source payload is still comfortably bounded. Each task input must include the deterministic recipe candidate, the source evidence rows, and any nearby recipe-local context required for correction. Each task output must include the corrected intermediate recipe object, ingredient-step linking, selected tags, and any existing recipe-stage status fields such as `repair_status` or `status_reason` that still make sense. Deterministic code should then join validated per-task outputs into the current staged recipe artifacts and final assembly.

Then migrate knowledge in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. Build task files over small nearby chunk groups using conservative block-count and char-count bounds. Each task output should preserve the compact knowledge contract over only its owned chunk ids. The validator should remain strict about owned chunk coverage and evidence. Replace oversized shard retry with ordinary task requeue. If structured follow-up remains for knowledge, it must operate on one bounded task packet and should not recreate the old “restated shard” failure mode.

Throughout the migration, keep deterministic compatibility surfaces alive. That means stage-level `proposals/*.json`, `promotion_report.json`, `telemetry.json`, and final outputs should still exist in recognizable form even if they are now derived from `task_results.jsonl` plus per-task outputs. External readers should not have to understand the new task layer on day one in order to keep functioning.

Finally, update docs and tests. `docs/10-llm/10-llm_README.md` must say that worker sessions process task files, that shards remain ownership/reporting containers, and that retry or repair is task-scoped. If benchmark or prompt-budget readers surface counts that change materially, update `docs/07-bench/07-bench_README.md` and any affected helper README to explain the new counters. Only create a new note under `docs/understandings/` if the implementation reveals a durable, non-obvious constraint that is not already captured by this plan or the existing runtime docs.

## Validation and Acceptance

This plan is implemented successfully only if all of the following are true.

First, each active Codex-backed stage must write and consume task files as first-class runtime artifacts. A finished run should expose a stage-level task manifest, worker-local assigned task lists, and one output file per completed task.

Second, a worker interruption must no longer force the stage to redo all work in its parent shard. Tests must prove that already written valid task outputs are harvested and promoted, while only pending or invalid tasks are requeued.

Third, the normal recovery path must stop depending on inline restatement of an entire large shard. For line-role, recipe, and knowledge, the default follow-up scope must be one task packet. If any structured repair remains, it must be bounded to a single task packet and must not serialize the whole parent shard.

Fourth, the existing deterministic stage outputs must still be produced. The migration is not allowed to break the current final recipe artifacts, line-role predictions, or knowledge outputs. The new task layer is an execution and recovery upgrade, not a user-facing output redesign.

Fifth, telemetry and status artifacts must become more legible, not less. A human should be able to tell which worker session owned which tasks, which tasks completed before a watchdog kill, which tasks were retried, and whether a final shard-level proposal was assembled from fully validated task results.

Sixth, the new tests must demonstrate both the success path and the partial-completion path. At least one focused test per active stage must prove: completed tasks survive, unfinished tasks remain pending, and only those pending tasks are retried.

Seventh, the docs must match the code. `docs/10-llm/10-llm_README.md` must no longer describe whole-shard structured retry or repair as the ordinary recovery shape once the migration lands.

## Idempotence and Recovery

This migration should be landed additively and safely. Keep the existing shard-level artifact surfaces alive while introducing the task layer so that tests and reader code can continue to pass during the transition. If a stage migration proves too large for one pass, leave the task-runtime foundation in place, keep the partially migrated stage on the old shard behavior, and update this plan’s `Progress` and `Decision Log` to say exactly where the transition stopped.

Do not delete the old shard dataclasses or proposal readers before a compatibility joiner exists. A worker kill, partial implementation, or failed test run should never leave the runtime unable to read earlier shard-based artifacts. When retrying implementation work, prefer adding task-level writers and readers first, then flipping one stage at a time.

If a stage-specific task shape turns out to be too large, shrink the task packet size rather than relaxing the bounded-task rule. This plan exists specifically to avoid letting prompt size drift back toward whole-shard recovery.
