## Problem

Interactive CodexFarm single-book benchmark runs could crash deep in the knowledge stage with `name '_KnowledgeWorkspaceStageCommandViolation' is not defined`.
`cookimport.llm.knowledge_stage.recovery` used planning-owned helper types `_KnowledgeWorkspaceStageCommandViolation` and `_KnowledgeFollowupDecision` without importing them, and no focused test or unresolved-name audit covered that split package.

## Acceptance Criteria

- `cookimport.llm.knowledge_stage.recovery` binds the planning-owned helper types it uses.
- A focused fast test fails on the broken baseline and passes after the fix.
- The knowledge-stage split package has a lightweight unresolved-global audit in routine pytest coverage.

## Verification

- `. .venv/bin/activate && pytest tests/llm/test_knowledge_stage_bindings.py -q`
- `. .venv/bin/activate && pytest tests/llm/test_knowledge_orchestrator_contracts.py -q`

## Evidence

- Before the fix, the new `test_knowledge_stage_bindings.py` file was red and the live CLI could crash once the run reached the knowledge-stage watchdog/recovery seam.
- After the fix, the focused binding tests and the existing knowledge orchestrator contract file pass offline in `.venv`.

## Constraints / Decision

- Keep the fix at the module boundary by importing the planning-owned types explicitly.
- Keep the coverage offline and cheap; do not require a real CodexFarm cookbook run just to catch split-module binding regressions.
