## Problem

Offline single-book benchmark wrap-up can skip upload-bundle generation when the direct `scripts.benchmark_cutdown_for_external_ai` import is unavailable.
The fallback loader in `cookimport/cli_support/bench_oracle.py` was resolving `cookimport/scripts/...` instead of repo-root `scripts/...`, so the recovery path could never find the helper.

## Acceptance Criteria

- The benchmark helper fallback in `cookimport/cli_support/bench_oracle.py` resolves the helper from repo-root `scripts/benchmark_cutdown_for_external_ai.py`.
- A focused regression test forces the direct import to fail, fails on the broken baseline, and passes after the fallback-path fix.
- The test covers both starter-pack and upload-bundle helper loaders so parallel path regressions are caught together.

## Verification

- `. .venv/bin/activate && pytest tests/labelstudio/test_labelstudio_benchmark_helpers_single_book_artifacts.py -q -k repo_root_script_path`
- `. .venv/bin/activate && pytest tests/labelstudio/test_labelstudio_benchmark_helpers_single_book_artifacts.py -q -k fallback`

## Evidence

- Before the fix, the new fallback-path regression test failed because the loader asked `importlib` for `/home/mcnal/projects/recipeimport/cookimport/scripts/benchmark_cutdown_for_external_ai.py`.
- After the fix, the same focused test passes and both helper loaders resolve the repo-root helper path.

## Constraints / Decision

- Keep scope to the helper fallback path regression only; do not change the normal direct import path.
- Use one shared repo-root constant for both fallback loaders so starter-pack and upload-bundle behavior stay aligned.
