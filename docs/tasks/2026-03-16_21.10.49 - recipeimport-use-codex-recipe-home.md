# Problem

RecipeImport launches CodexFarm for recipe correction, knowledge, line-role, prelabel, and CodexFarm discovery calls, but it currently does not force those subprocesses onto the dedicated `~/.codex-recipe` home. That lets runs silently fall back to the default Codex home.

# Acceptance Criteria

- All RecipeImport CodexFarm subprocesses launched through `cookimport/llm/codex_farm_runner.py` use `~/.codex-recipe` as `CODEX_HOME` by default.
- An explicit per-call env override can still change the Codex home when needed.
- The default path is also exposed as `CODEX_FARM_CODEX_HOME_RECIPE` for compatibility with CodexFarm profile-aware paths.
- Focused tests cover the injected environment for both `process` execution and `models list`.
- Docs mention the new default behavior.

# Verification

- `. .venv/bin/activate && pytest tests/llm/test_codex_farm_orchestrator_runner_transport.py -q`

# Evidence

- 2026-03-16: the focused runner transport suite passed after adding default Codex-home injection plus override coverage for `process` and `models list`.

# Constraints / Decision

- Keep the change centralized at the RecipeImport -> CodexFarm transport layer instead of adding more CLI surface.
- Prefer env injection over shell aliases so non-interactive subprocesses behave the same everywhere.

# Rollback

- Revert the runner helper and the related tests/docs to restore previous inherited-environment behavior.
