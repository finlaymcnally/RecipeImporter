## Problem statement

`pytest -m slow` included many files that now run quickly, and `test_stats_dashboard.py` mixed one expensive browser pixel harness with otherwise fast tests.

## Acceptance criteria

- Dashboard pixel-overflow rerender test is isolated in a dedicated slow file.
- Main dashboard test file remains focused on fast renderer/schema/collector checks.
- `_SLOW_FILES` includes only genuinely expensive suites.
- Marker mapping remains complete for all `test_*.py` files.

## Verification steps

- `. .venv/bin/activate && pytest tests/analytics/test_stats_dashboard.py`
- `. .venv/bin/activate && pytest tests/analytics/test_stats_dashboard_slow.py`
- `. .venv/bin/activate && pytest -m slow --collect-only -q`
- `. .venv/bin/activate && python - <<'PY' ... marker coverage check ... PY`

## Evidence

- `tests/analytics/test_stats_dashboard.py`: `70 passed in 9.65s`
- `tests/analytics/test_stats_dashboard_slow.py`: `1 passed in 5.39s`
- `pytest -m slow --collect-only -q` now collects:
  - `tests/analytics/test_stats_dashboard_slow.py`
  - `tests/cli/test_cli_output_structure_slow.py`
  - `tests/labelstudio/test_labelstudio_benchmark_helpers.py`
  - `tests/llm/test_codex_farm_orchestrator.py`
- Marker coverage check: `missing 0`, `extra 0`.

## Constraints/gotchas

- Keep browser pixel harness in slow slice; it depends on Playwright/Chromium availability.
- Do not re-expand slow slice with sub-3s suites unless they regress materially.

## Decision

Adopt explicit fast/slow modularization by extracting the dashboard browser harness into `*_slow.py` and trimming stale slow markers.

## Rollback / compensation

Move the pixel test back into `test_stats_dashboard.py` and restore removed entries in `_SLOW_FILES` if broad slow categorization is preferred.
