Problem statement

The LLM stages currently mix `shard`, `task`, and `packet` in user-facing docs and progress labels, which makes it hard to tell whether a term means a scheduling bundle, a file-backed work item, or the actual content slice the model sees.

Acceptance criteria

- The LLM docs define one consistent naming contract for `shard`, `task`, `packet`, and `worker`.
- Line-role progress and worker wording use `task` consistently for counted work items instead of mixing `task packet` with `task`.
- Knowledge examples/tests stop teaching stale `queued shards` / `task packet` wording where the live runtime already counts tasks.

Verification steps

- From `/home/mcnal/projects/recipeimport`, run:
  - `source .venv/bin/activate && pytest tests/parsing/test_canonical_line_roles.py tests/llm/test_label_phase_workers.py tests/labelstudio/test_labelstudio_benchmark_helpers_progress.py -q`
- Confirm the targeted suites pass and the updated docs describe:
  - `shard` as the planning bundle
  - `task` as the smallest validated work item
  - `packet` as the model-facing content slice owned by one task
  - `worker` as the executor

Evidence

- Before the change, line-role progress/worker wording used `task packet`, while knowledge examples still mixed `queued shards` with task counts.
- After the change, counted work units are `tasks`, and the docs explicitly define how `packet` differs from `task`.

Constraints / gotchas + brief decision

- Keep behavior unchanged. This is a naming cleanup, not a planner/runtime redesign.
- Do not rename stable file paths or broad internal identifiers unless they are directly user-facing.
- Decision: standardize the public/runtime vocabulary first and leave deeper internal renames for a separate refactor if still desired.

Rollback / compensation plan

- Revert the doc/string/test changes together if the new wording proves misleading. No data or artifact migration is involved.
