# Build Benchmark Selective Retry/Resume For Codex-Farm Recipe Passes

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds. This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a benchmark run that loses only a small number of Codex-backed recipe rows during pass2 or pass3 will try to heal only the missing rows instead of accepting the first partial result and moving on. The benchmark should keep every successful original output file, rerun only the missing bundle files for the failed pass, merge any recovered outputs back into the same run root, and then continue the remaining downstream work for recipes that now have valid inputs.

The visible proof is an offline benchmark run that would previously have finished with permanently missing pass bundles now showing a retry attempt in its artifacts. A reviewer should be able to open the benchmark root and tell, with no source-code knowledge, which rows were missing after the first pass, which were retried, which recovered, which remained skipped, and that no already-successful rows were rerun.

## Context and Orientation

This repository has one shared benchmark prediction-generation path. `cookimport/cli.py::labelstudio_benchmark(...)` is the CLI entrypoint for offline benchmark runs. When `--no-upload` is used, it calls `cookimport/labelstudio/ingest.py::generate_pred_run_artifacts(...)`. That function builds a canonical `RunSettings` object through `cookimport/config/run_settings.py::build_run_settings(...)`, then the recipe Codex pipeline is eventually executed by `cookimport/llm/codex_farm_orchestrator.py::run_codex_farm_recipe_pipeline(...)`.

The recipe pipeline is a three-pass file-based workflow. A “pass” means one `codex-farm process` subprocess call over a directory of JSON bundle files. A “bundle file” means one recipe-row JSON file such as `raw/llm/<workbook_slug>/pass2_schemaorg/in/r0007_foo.json`. The pass output for that same row should appear at `raw/llm/<workbook_slug>/pass2_schemaorg/out/r0007_foo.json`. A “selective retry” therefore means copying only the missing input bundle files into a retry input directory, running the same pass again for just those files, and copying any newly produced outputs back into the original `out/` directory.

`run_codex_farm_recipe_pipeline(...)` already does most of the heavy lifting needed here. It creates the `pass1_chunking`, `pass2_schemaorg`, and `pass3_final` `in/` and `out/` directories. It writes pass2 input bundles only for recipes that survived pass1. It writes pass3 input bundles only for recipes that have acceptable pass2 output. After each pass, it validates outputs per recipe by checking whether the expected bundle file exists and then loading the output contract. Missing files already become recipe-scoped errors; they do not crash the whole orchestrator by themselves.

`cookimport/llm/codex_farm_runner.py::SubprocessCodexFarmRunner.run_pipeline(...)` is the boundary that talks to the external `codex-farm` binary. That runner can already treat some non-zero exits as recoverable partial-output runs in benchmark mode. The important current behavior is that benchmark mode may continue when the tool wrote most output bundles and the failure family is limited to the existing “no last agent message” and timeout-style cases. That means the new retry feature does not need to decide whether the whole book survives. The runner already answers that. The new feature only decides whether it should attempt to fill the remaining holes before the orchestrator marks those rows as missing.

The relevant manifests already exist. The recipe pipeline writes `raw/llm/<workbook_slug>/llm_manifest.json`, which already contains `process_runs` and per-recipe status data. The prediction-run and benchmark roots also write `run_manifest.json` and already link the LLM manifest through artifacts. This feature should extend those artifacts, not replace them.

The existing tests also give concrete seams. `tests/llm/test_codex_farm_orchestrator.py` is the main orchestrator behavior suite. `tests/llm/test_codex_farm_orchestrator_runner_transport.py` covers the subprocess runner and benchmark/extract transport differences. `tests/labelstudio/test_labelstudio_benchmark_helpers_eval_payload.py`, `tests/labelstudio/test_labelstudio_benchmark_helpers.py`, and `tests/labelstudio/test_labelstudio_benchmark_helpers_single_profile.py` protect benchmark manifest and helper behavior. Extend those tests instead of inventing a new end-to-end harness.

## Plan of Work

First, add the benchmark selective retry settings to the canonical config layer in `cookimport/config/run_settings.py`. Keep the surface small: `codex_farm_benchmark_selective_retry_enabled: bool` and `codex_farm_benchmark_selective_retry_max_attempts: int`. Add those fields to `RunSettings`, accept them in `build_run_settings(...)`, and make sure they serialize into manifests exactly like the other benchmark-visible Codex settings. Do not add a third setting for configurable failure categories; keep the allowed failure family as a module constant near the retry helper.

Second, wire those settings through the benchmark call path. `cookimport/cli.py::labelstudio_benchmark(...)` should expose benchmark-only CLI options with clear names and help text:

    --codex-farm-benchmark-selective-retry/--no-codex-farm-benchmark-selective-retry
    --codex-farm-benchmark-selective-retry-max-attempts

Those values must flow into `cookimport/labelstudio/ingest.py::generate_pred_run_artifacts(...)`, then into `build_run_settings(...)`, then back out through `cookimport/config/run_settings_adapters.py::build_benchmark_call_kwargs_from_run_settings(...)`. The important rule is that interactive benchmark helpers and benchmark suites that start from `RunSettings` must preserve the same behavior; the CLI cannot be the only surface that knows about the feature.

Third, add a small retry subsystem inside `cookimport/llm/codex_farm_orchestrator.py`. The safest shape is one helper, called from `run_codex_farm_recipe_pipeline(...)`, that operates at a pass boundary and only for pass2 and pass3. The helper should receive the pass label, pipeline id, original input and output directories, the expected bundle filenames for that pass, the shared runner/env/root/workspace/model settings, and the retry settings from `RunSettings`. Its algorithm should be:

1. Compare the expected bundle filenames against the current output directory and build a sorted list of missing bundle filenames.
2. Stop immediately if the list is empty, the run is not benchmark mode, retry is disabled, or the original run metadata does not match the allowed recoverable failure family.
3. Create `retry_attempt_01/in` and `retry_attempt_01/out` under the original pass directory.
4. Copy only the missing input bundle files into that retry `in/` directory.
5. Invoke the same `codex_runner.run_pipeline(...)` with the same pipeline id and environment settings as the original pass.
6. Copy only newly produced output files back into the original `out/` directory. Never overwrite an output file that already existed from the original pass.
7. Recompute the missing-file set. If `max_attempts` allows another retry and the failure is still in the allowed family, repeat with `retry_attempt_02`, and so on.
8. Return a manifest-ready retry payload that names attempted, recovered, and unrecovered bundles plus the per-attempt process payloads.

Fourth, decide the allowed retry family once and keep it local. The retry helper should run only when the original pass was already considered recoverable enough for the book to continue and when the observed failure family stays within the current benchmark partial-output contract. In practical terms, the first implementation should allow only the same benchmark partial failure family that the runner already softens: `nonzero_exit_no_payload` and benchmark-mode timeout mixes associated with “no last agent message” partial-output runs. Invalid JSON outputs, schema-validation failures, transport mismatches, or low-quality pass3 outputs are not missing-file problems and must not trigger retry.

Fifth, keep the rest of the orchestrator unchanged. After retry, the existing per-recipe validation loops should still be the code that decides whether a recipe ends up `ok`, `degraded`, `fallback`, or `error`. If the retry produced a valid pass2 output, that recipe naturally becomes eligible for the current pass2-to-pass3 routing logic. If the retry produced a valid pass3 output, that recipe naturally becomes eligible for final overrides. This is why the retry hook must happen before the existing validation loops.

Sixth, extend manifest writing without bloating it. `llm_manifest.json` should gain a top-level `selective_retries` object with one optional entry for `pass2` and one optional entry for `pass3`. Each entry should include the resolved settings snapshot, the original missing count, a final recovered count, a final unrecovered count, the attempted bundle filenames, the corresponding recipe ids when they can be inferred from `states`, and an `attempts` array that stores one record per retry attempt. Each attempt record should include the retry directory, attempted bundle filenames, copied output filenames, the remaining missing bundle filenames after that attempt, and the serialized `process_run` payload.

Seventh, keep the benchmark root summary terse. The benchmark or prediction-run `run_manifest.json` should not repeat every bundle filename. It should expose only a few summary fields such as `selective_retry_attempted`, `selective_retry_pass2_attempts`, `selective_retry_pass2_recovered`, `selective_retry_pass3_attempts`, and `selective_retry_pass3_recovered`, plus the existing artifact link to `llm_manifest_json`. That makes the root manifest legible while leaving the detailed truth in the LLM manifest.

Finally, update the benchmark and LLM reference docs once code lands. The doc change should explain the precise boundary: benchmark-only, pass2/pass3 only, missing-bundle only, successful original outputs preserved, and audit details available in `raw/llm/<workbook_slug>/llm_manifest.json`.

## Validation and Acceptance

The feature is complete only when the repository demonstrates all of the following behaviors.

Benchmark config can express the feature. A benchmark run can enable or disable selective retry and choose the maximum retry count, and those values survive through `RunSettings`, benchmark kwargs, and manifest config snapshots.

Selective retry runs only in benchmark recipe mode. The same partial-output condition in extract mode must behave exactly as it does today.

Retry batches contain only missing bundle inputs. If pass2 originally wrote 27 out of 29 outputs, the retry input directory must contain exactly the 2 missing input filenames, not all 29 original inputs.

Successful original outputs are preserved. The implementation must not overwrite or regenerate an output file that already exists from the original pass.

Resume happens from the pass boundary. A recovered pass2 file must be processed by the existing pass2 validation and pass3-routing logic. A recovered pass3 file must flow into final overrides without rerunning pass1 or pass2.

Retry is limited to the recoverable partial-output family. Missing-file retries are allowed only for the existing benchmark partial-output cases. Invalid contract output, malformed JSON on disk, low-quality pass3 drafts, or transport mismatches remain ordinary recipe-level failures.

Artifacts tell the truth. A human should be able to inspect `raw/llm/<workbook_slug>/llm_manifest.json` and answer which pass retried, how many attempts ran, which bundles were attempted, which recipe ids were affected, which rows recovered, and which remained missing.

## Idempotence and Recovery

This implementation must be safe to run repeatedly while developing. The original `pass2_schemaorg/out` and `pass3_final/out` directories are the source of truth and must never be deleted as part of retry. Retry attempts should write into numbered `retry_attempt_XX` directories so earlier evidence remains readable.

If a retry attempt fails halfway, the operator should still be able to inspect the original pass output directory, the retry `in/` directory, the retry `out/` directory, and the serialized retry `process_run` payload. The safe rollback is to disable the new feature and rerun the benchmark. No migration, cleanup step, or destructive command should be required.

If a developer reruns the same benchmark root during implementation, it is acceptable to recreate only the retry attempt directory being generated for the current run, but it is not acceptable to delete prior original outputs in order to “start clean.”
