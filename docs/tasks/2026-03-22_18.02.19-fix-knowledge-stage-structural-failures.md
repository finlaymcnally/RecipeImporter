## Purpose / Big Picture

The knowledge stage already has the repo-owned micro-batch loop, the knowledge-specific watchdog, fail-closed queue-completion semantics, and stronger `semantic_all_false_empty_shard` feedback. Those fixes closed the most obvious correctness gaps from the March 22 `16:52:22` run, but the stage is still structurally too expensive. The remaining problem is not “workers can finish early without queue proof” anymore. The remaining problem is that the stage still creates too many tiny task packets and still exposes too much worker surface area for the happy path.

After this remaining work lands, a human should be able to rerun the same approved Salt Fat benchmark slice and see the stage spend materially less on queue motion and orientation chatter because it is processing fewer, larger task packets through a smaller happy-path interface. The visible proof is that the stage stops looking pathological for overhead reasons: ordinary worker traces should no longer be dominated by one-chunk-at-a-time queue motion, repeated helper-surface rediscovery, or broad prompt/manifest rereads.

This plan now follows the executed recovery work in [docs/plans/2026-03-22_17.45.01-recover-knowledge-after-failed-16.52.22-run.md](/home/mcnal/projects/recipeimport/docs/plans/2026-03-22_17.45.01-recover-knowledge-after-failed-16.52.22-run.md). That earlier plan fixed the correctness and repair gaps. The remaining diagnosis is narrower: the stage is still over-fragmented and still exposes too much worker surface area.

## Context and Orientation

The knowledge stage is the optional non-recipe review and snippet-extraction phase. It reviews deterministic non-recipe chunks, decides whether each owned block is `knowledge` or `other`, and emits grounded snippet evidence for useful cooking knowledge. The authoritative stage artifacts live under `data/output/<timestamp>/raw/llm/<workbook_slug>/knowledge/`, and the final promoted outputs live in `08_nonrecipe_spans.json`, `09_knowledge_outputs.json`, and reviewer-facing `knowledge/` artifacts.

The runtime currently has four important seams.

`cookimport/llm/codex_farm_knowledge_orchestrator.py` builds shard and task manifests, writes the main workspace-worker prompt, supervises the long-lived knowledge worker session, and decides whether a worker run completed, failed, or should be retried. Queue completion is now fail-closed there, but it still plans one task per chunk and still writes a broad main prompt.

`cookimport/llm/knowledge_workspace_tools.py` writes the worker-visible sidecars, renders `tools/knowledge_worker.py`, prepares draft scaffolds, checks batch or task drafts, installs validated outputs, and writes batch/task feedback. It now has better strong-cue feedback and smaller batch sidecars, but it still exposes many single-task and low-level helper verbs in the same parser.

`cookimport/llm/codex_farm_knowledge_ingest.py` is the authoritative validator for semantic packet results. It enforces chunk ownership, block coverage, snippet grounding, snippet anti-copy rules, and the strong-cue empty-shard rejection called `semantic_all_false_empty_shard`. That validator metadata is already richer now and should be preserved.

`cookimport/llm/codex_exec_runner.py` is the shared direct Codex execution seam used by recipe, line-role, and knowledge. It prepares the sterile execution root, mirrors repo-owned control files such as `current_batch.json`, and writes the common worker-orientation text. This file should stay stage-generic, but the remaining work must account for how much broad worker freedom and queue visibility it still advertises.

Two terms matter in this plan.

A “task packet” means one worker-owned result file under `workers/*/out/<task_id>.json`. For knowledge, that packet now usually owns a small consecutive chunk group rather than exactly one chunk. A “micro-batch” means the repo-owned consecutive group of task packets exposed together through `current_batch.json` and `CURRENT_BATCH.md`.

The central diagnosis is now narrower than when this file was first written: correctness and strong-cue repair got better, but the stage still feeds its worker loop too many tiny packets and too many visible fallback surfaces. The result is a long queue with a broad prompt, broad helper namespace, and repeated orientation work.

## Milestones

### Milestone 1: Replace one-chunk packets with small multi-chunk task packets

At the end of this milestone, the knowledge stage should no longer turn every chunk into its own task row. The new normal should be a small packet of consecutive chunks per task, still bounded enough for local validation and repair. The packet must remain owned and local: one `out/<task_id>.json` still maps to one task row, but that task row can now own a few consecutive chunks instead of exactly one.

This is the biggest structural spend reduction in the plan. It cuts queue length before prompt/runtime concerns even begin. The implementation should keep ownership easy to reason about by using consecutive chunks from the same parent shard, preserving existing order and provenance metadata, and keeping packets small enough that one failed packet is still easy to repair.

Acceptance for this milestone is local and visible. Task manifests should show task rows with `chunk_count > 1` for ordinary multi-chunk shards, helper scaffolds should emit the expected number of result rows for those packets, and task-local validation should still work without changing the downstream authoritative proposal/promotion seam.

### Milestone 2: Split the happy path from recovery/debug surfaces

At the end of this milestone, the knowledge worker should see one obvious paved road and a separate, intentionally less-prominent recovery surface. The happy-path helper should expose only the commands needed for normal batch work. Recovery commands such as single-task helpers, raw per-task scaffold/install verbs, and queue-inspection tools should either move into a separate helper entrypoint or become explicitly hidden unless the prompt or repo-owned feedback points to them.

The point is not to remove every escape hatch. The point is to stop presenting every escape hatch as co-equal to the main route. A worker following the main prompt should no longer need to decide among ten verbs and then remember which ones are “discouraged.” The normal path should have a smaller namespace and smaller prose surface than the fallback path.

Acceptance for this milestone is a generated worker root whose main helper surface is visibly smaller than today, plus focused tests proving that the prompt and worker manifest direct normal work through that smaller surface.

### Milestone 3: Shrink the prompt and direct-exec orientation surface

At the end of this milestone, the main knowledge worker prompt and the stage-generic worker-orientation text should reinforce the smaller happy path instead of reintroducing breadth through prose. The main prompt should stop advertising orientation helpers such as `overview` and `show-batch` as ordinary moves, should mention recovery/debug surfaces only as fallback, and should preserve the now-correct queue-completion and strong-cue rules without turning the happy path back into a long list of caveats.

This milestone matters because even a split helper surface will not reduce orientation chatter if the prompt and worker manifest still present too many startup options. The happy path should look small in both code and prose.

Acceptance for this milestone is a generated prompt/worker-manifest surface that is visibly narrower than today, plus focused tests that assert the main path no longer advertises the broader recovery verbs as normal first reads.

### Milestone 4: Revalidate the saved bad path with approval
After local proof is in place, and only with explicit user approval, rerun the same benchmark slice that exposed the problem. The runtime acceptance bar is now specifically about the remaining structural fixes: fewer task packets, less orientation chatter, and materially lower runtime overhead while preserving the queue-complete and strong-cue safeguards that already landed.

## Plan of Work

Start in `cookimport/llm/codex_farm_knowledge_orchestrator.py` at `_build_knowledge_task_plans(...)`. Replace the current “one chunk becomes one task plan” logic with a small packet planner that groups consecutive chunks from the same parent shard into bounded task packets. Keep the packet ordering stable and deterministic. The safest starting policy is a small cap by chunk count plus a byte or block-count cap so large or table-heavy chunks do not create unwieldy packets. Update the task metadata so each packet still advertises exact owned chunk ids, exact owned block indices, and the input/hint/result paths the worker root uses today.

Then update `cookimport/llm/knowledge_workspace_tools.py` to treat those multi-chunk packets as the new normal answer unit. The scaffold, checker, installer, batch sidecars, and feedback writers must all understand that a task packet may own more than one chunk. Keep the packet-local validator seam the same: one output file, validated locally, then installed to the same declared `result_path`.

In the same module, separate the worker helper surface. The main helper entrypoint should expose only the happy-path commands needed for batch work, compact status lookup, and failure explanation. Recovery/debug commands such as `current`, `next`, `show`, `scaffold`, `check`, `install`, `complete-current`, `check-current`, and `install-current` should move behind an explicitly separate debug helper or a clearly separate parser namespace that the main prompt does not advertise. The code change should be simple enough that the current tests can assert both surfaces without relying on prompt wording alone.

Still in `knowledge_workspace_tools.py`, preserve the smaller sidecars and strong-cue feedback that already landed, but adapt them to multi-chunk packets and the smaller happy-path helper surface. `CURRENT_BATCH.md` should remain the cheapest readable starting point. `current_batch.json` should stay machine-readable but avoid duplicating queue-wide material that already exists elsewhere. `CURRENT_BATCH_FEEDBACK.md` and `install-batch` stdout should both tell the worker exactly what to do next with minimal reread pressure.

Return to `cookimport/llm/codex_farm_knowledge_orchestrator.py` and tighten the main prompt. The main prompt should name only the happy-path helper surface, the minimal sidecar read order, and the already-correct completion rule. Recovery/debug tools can still exist, but they should no longer be listed as equal first-class options in the ordinary loop.

Touch `cookimport/llm/codex_farm_knowledge_ingest.py` only as needed to keep the existing strong-cue metadata precise once packets become multi-chunk. Do not weaken the validator or reopen the now-working repair story.

Update `cookimport/llm/codex_exec_runner.py` last and minimally. Keep it stage-generic, but align the worker-manifest notes and common orientation text with the smaller happy-path surface. The runner should mirror any new or renamed control files needed for the slimmer main helper surface, but it should not own knowledge-specific policy decisions.

Finally, update the short reviewer-facing docs in `cookimport/llm/README.md`, `docs/10-llm/10-llm_README.md`, and `docs/10-llm/nonrecipe_knowledge_review.md`. Keep these notes short. They should explain that knowledge now uses multi-chunk task packets and a smaller happy-path helper surface, while preserving the already-landed queue-complete success semantics and compact strong-cue empty-shard feedback.
