Problem statement

`cookimport bench oracle-upload` fails for normal single-offline bundles when `upload_bundle_payload.jsonl` exceeds Oracle CLI's per-file input cap, even in browser mode.

Acceptance criteria

- Browser-mode benchmark uploads no longer fail locally just because one bundle file is larger than Oracle's per-file limit.
- Oversized bundle files are transformed only for transport; the on-disk `upload_bundle_v1` contract stays unchanged.
- Dry-run behavior remains zero-cost and still reports a local preview fallback for oversized bundles.

Verification steps

- Activate `.venv`.
- Run `pytest tests/bench/test_benchmark_oracle_upload.py`.

Evidence

- Added a browser upload test that verifies oversized payloads are replaced with Oracle-safe shard attachments and that the prompt explains the shard layout.
- Existing browser and dry-run command assembly tests still pass against the updated wrapper.
- Verification result: `source .venv/bin/activate && pytest tests/bench/test_benchmark_oracle_upload.py` -> `8 passed, 1 warning in 2.25s`.

Constraints/gotchas + decision

- The 1 MB cap is enforced inside the installed Oracle CLI before browser attachment upload logic runs.
- Fix in this repo by sharding oversized text bundle files into temporary ordered parts for browser mode instead of changing the bundle format or patching upstream Oracle here.

Rollback / compensation plan

- Revert the `oracle_upload.py` transport shim and its tests if it causes regressions; on-disk benchmark artifacts are untouched by this change.
