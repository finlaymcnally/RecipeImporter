# Purge remaining historical compatibility rot

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the repository should describe and implement only the current runtime contract. A reader, test, or helper should no longer learn old benchmark topology, old Oracle model env names, old follow-up syntax, or old recipe-workspace payload shapes by reading current code. The visible proof is straightforward: the active code paths use only current manifests, current selectors, and current payload shapes; the docs stop teaching old alternatives; and a repo-wide grep for true historical terms across `cookimport`, `tests`, and `docs` comes back empty or nearly empty for intentionally unavoidable plain-English prose that is explicitly called out in this plan before implementation.

This is a destructive cleanup, not a compatibility migration. Older local artifacts, older benchmark chats, and older helper/test doubles are not goals to preserve. If something only exists to read or describe an old shape, the default action is delete it.

## Context and Orientation

The current architecture baseline is this:

- Stage and benchmark flows are supposed to use semantic-stage artifacts, current manifests, and explicit canonical pointers.
- New benchmark handoff work is supposed to use `upload_bundle_v1` plus additive follow-up packets.
- New LLM worker surfaces are supposed to use the current recipe task shape and current taskfile/session contracts.
- Analytics is supposed to describe current benchmark rows, not reconstruct older benchmark meaning from path heuristics.

The earlier cleanup already removed the worst guessed-path manifest readers for knowledge and recipe manifests. What remains is the smaller but still important second layer of compatibility rot: readers, selectors, env aliases, and test/doc stories that preserve old shapes.

From the audit, the remaining live families are:

1. Analytics historical-record interpretation.

   `cookimport/analytics/benchmark_semantics.py` still decides benchmark variant and AI profile from older paired-run path conventions such as `/codex-exec/` and `/vanilla/`. The dashboard support tests in `tests/analytics/stats_dashboard_support.py` and `tests/analytics/test_stats_dashboard_benchmark_semantics.py` explicitly preserve those cases.

2. Oracle and benchmark compatibility naming.

   `cookimport/bench/oracle_upload.py` still resolves Oracle browser model selection through current env names plus older aliases such as `ORACLE_GENUINE_MODEL`, `ORACLE_TEST_MODEL`, `ORACLE_REVIEW_MODEL`, `ORACLE_DEEP_REVIEW_MODEL`, `ORACLE_FAST_MODEL`, and `ORACLE_FAST_SMOKE_MODEL`. The bench docs still teach those aliases as part of the live contract.

3. Follow-up and cutdown old-path tolerance.

   `cookimport/bench/external_ai_cutdown/artifact_paths.py` still falls back to `run_dir / "prediction-run"` when the manifest does not point at a prediction run. `cookimport/bench/followup_bundle.py` still falls back to `prediction-run/prompt_budget_summary.json` and `run_dir/prompt_budget_summary.json` instead of treating the manifest/payload locator as authoritative. `cf-debug` still intentionally accepts old `source:start-end` line-range syntax, and the docs still advertise that.

4. Recipe workspace old-shape support.

   `cookimport/llm/codex_exec_runner.py` still builds a legacy recipe-task request and normalizes compact historical output rows after trying the current output builder. `cookimport/llm/recipe_workspace_tools.py` and `cookimport/llm/recipe_stage_shared.py` still include explicit “legacy key” suggestion diagnostics. The tests preserve those older shapes directly.

5. Historical prose and naming that keeps the old mental model alive.

   The bench and LLM docs still describe accepted old syntaxes, old env aliases, and historical phase layouts. Some tests and helpers still use names like `legacyCodexfarmRecord`, `legacyRunDirVanillaRecord`, `_dispatch_fake_prediction_stage_via_legacy_benchmark_double`, and similar historical labels even when the code under test is now current-only.

The depth is therefore moderate, not endless. The core stage pipeline no longer looks dual-backbone. The remaining work is still destructive and cross-cutting, but it is concentrated in a small set of modules that interpret artifacts or define helper/test semantics.

## Plan of Work

The work should proceed in four milestones. Each milestone must leave the repo in a passing state before moving on.

### Milestone 1: Delete active compatibility behavior from analytics

The analytics owner is `cookimport/analytics/benchmark_semantics.py`, with supporting readers in `cookimport/analytics/compare_control_fields.py` and dashboard regressions in `tests/analytics/`.

The target contract after this milestone is simple: benchmark variant and AI-assistance profile must come from explicit current metadata such as `benchmark_variant`, `ai_assistance_profile`, and current run-config surfaces. Analytics must stop inferring official paired-run meaning from old path segments like `/codex-exec/` or `/vanilla/`. If a record is missing current metadata, analytics should classify it as unknown or incomplete rather than reconstructing an old benchmark story from path layout.

Concretely:

- Remove path-based variant/profile inference in `cookimport/analytics/benchmark_semantics.py` that exists only to rescue older paired benchmark rows.
- Revisit any dashboard field helpers that still map `full_stack` to `codex-exec` or `deterministic` to `vanilla` because of older paired-run assumptions. Keep only the current explicit mapping that is still needed by current CSV rows.
- Rewrite `tests/analytics/stats_dashboard_support.py`, `tests/analytics/test_stats_dashboard_benchmark_semantics.py`, and any affected dashboard tests so they build current benchmark records instead of “legacy run-dir” records.
- Update analytics docs so they describe backfill only as a one-time repair tool, not a live historical-rows product surface.

Acceptance for this milestone is that analytics no longer recovers benchmark semantics from old path naming alone, and the analytics tests pass using current record shapes.

### Milestone 2: Delete benchmark Oracle, follow-up, and cutdown compatibility seams

The owners are `cookimport/bench/oracle_upload.py`, `cookimport/bench/followup_bundle.py`, `cookimport/bench/external_ai_cutdown/artifact_paths.py`, relevant CLI wrappers under `cookimport/cli_commands/bench.py`, and the bench/Label Studio tests and docs.

The target contract after this milestone is:

- Oracle browser model selection reads only current env names: `ORACLE_INSTANT_MODEL`, `ORACLE_PRO_MODEL`, and `ORACLE_THINKING_MODEL`.
- `upload_bundle_v1` and current lane packet files are the only accepted benchmark review handoff shape.
- Follow-up and cutdown readers trust manifest pointers, payload locators, or current packet-local metadata only. They do not guess `prediction-run` roots or prompt-budget files from historical layout.
- `cf-debug` accepts only the canonical line-range selector syntax `source:start:end`.

Concretely:

- Remove old Oracle env alias resolution and the related tests/support helpers that still seed those envs.
- Remove `LEGACY_BENCHMARK_UPLOAD_BUNDLE_FILE_NAMES` and any old-format rejection logic that exists only because the old names are still part of the mental model.
- Remove `run_dir / "prediction-run"` fallback in `cookimport/bench/external_ai_cutdown/artifact_paths.py` unless a current manifest explicitly points there.
- Remove `prediction-run/prompt_budget_summary.json` and run-root prompt-budget fallback in `cookimport/bench/followup_bundle.py`; require current manifest or payload-locator resolution instead.
- Remove old selector syntax acceptance in `cf-debug` and rewrite the related tests and docs.
- Rewrite bench docs so they describe only the current selectors, current Oracle env names, and current packet layout.

Acceptance for this milestone is that the benchmark review and follow-up stack uses current manifests and selectors only, and all related tests pass without old env names or old selector syntax.

### Milestone 3: Delete recipe workspace historical payload support

The owners are `cookimport/llm/codex_exec_runner.py`, `cookimport/llm/recipe_workspace_tools.py`, `cookimport/llm/recipe_stage_shared.py`, and the corresponding LLM tests/docs.

The target contract after this milestone is:

- direct-exec recipe workers build and validate only the current recipe answer shape,
- invalid keys are just invalid keys,
- there is no secondary legacy request or output normalization path.

Concretely:

- Remove `_build_legacy_recipe_task_output_request(...)` from `cookimport/llm/codex_exec_runner.py` and remove the second-chance builder call that tries the old compact shape.
- Tighten `_normalize_recipe_task_answer_payload(...)` so it accepts only the current output shape. If compact historical rows still matter for an offline test, rewrite the test instead of keeping the reader.
- Replace “legacy key” diagnostics in `cookimport/llm/recipe_workspace_tools.py` and `cookimport/llm/recipe_stage_shared.py` with neutral unexpected-key errors.
- Rewrite `tests/llm/test_recipe_workspace_tools.py`, `tests/llm/test_recipe_phase_workers.py`, and any related tests to assert current-only behavior and current wording.
- Remove historical wording from `docs/10-llm/10-llm_README.md` and `cookimport/llm/README.md`.

Acceptance for this milestone is that recipe workspace helpers no longer know how to ask for or normalize old recipe payload shapes.

### Milestone 4: Purge remaining historical wording, helper names, and test names

This milestone is the repo-wide scrub after the active behavior seams are gone.

Concretely:

- Rename helpers and tests whose names still encode the old story when they now cover current behavior.
- Delete or rewrite prose in `docs/07-bench/07-bench_README.md`, `docs/10-llm/10-llm_README.md`, `docs/08-analytics/08-analytics_readme.md`, and any owner README that still explains historical acceptance paths.
- Sweep for the true historical terms `legacy`, `historical`, `compatible`, `compatibility`, and `archived` across `cookimport`, `tests`, and `docs`, and either remove the hit or make a deliberate note in this plan why one unavoidable hit remains.
- Re-run the docs index so it still renders cleanly after doc rewrites.

Acceptance for this milestone is that current code, tests, and docs no longer teach or preserve the old compatibility story.

## Idempotence and Recovery

This purge should be implemented in small deletions so it is safe to repeat the grep, rerun the same tests, and continue. When a test fails, classify it before editing:

- If the code still preserves an old contract, remove the code and update the test.
- If the test is only preserving a historical fixture or helper name, rewrite or delete the test.
- If a doc describes an old path that no longer exists, rewrite the doc to current behavior rather than reintroducing the path.

Do not recover by adding new fallback readers. If a current artifact or manifest is missing in a fixture, fix the fixture to emit the current contract.

## Interfaces and Dependencies

- Analytics benchmark semantics accept explicit current record metadata and current run-config fields only. Path layout is not an interface.

No new dependencies are needed. This is a deletion and contract-tightening change across existing owners.
