## Purpose / Big Picture

After this change, `Isolate For X` supports numeric comparisons (`>`, `>=`, `<`, `<=`) for numeric fields, and those rules still flow through the same table-filter pipeline used by `Previous Runs`.

## Surprises & Discoveries

- Observation: Numeric operators cannot be reliable if clause activation only checks non-empty text.
  Evidence: threshold operators need numeric parsing; non-numeric text must be treated inactive.

- Observation: Isolate matching must call the same operator evaluator as table filters to avoid drift.
  Evidence: reusing `evaluatePreviousRunsFilterOperator(...)` keeps `eq/neq/gt/gte/lt/lte` behavior aligned.

## Decision Log

- Decision: Keep isolate UI operator options field-typed (`is`/`is not` for categorical, comparators for numeric).
  Rationale: avoids invalid operator/field combinations and keeps UX clear.
  Date/Author: 2026-03-03 / assistant

- Decision: Normalize numeric isolate values via `maybeNumber` before clause activation.
  Rationale: prevents broken active rules and invalid sync into table filters.
  Date/Author: 2026-03-03 / assistant
