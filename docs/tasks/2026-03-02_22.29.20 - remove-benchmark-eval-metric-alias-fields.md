## Problem

Stage/canonical benchmark evaluators currently emit alias keys (`precision/recall/f1`, `practical_*`) that mirror other metrics and create ambiguous contracts.

## Acceptance Criteria

- Stage/canonical eval report generation stops emitting alias metric fields.
- Explicit benchmark metrics remain available (`strict_accuracy`, `overall_block_accuracy`/`overall_line_accuracy`, `macro_f1_excluding_other`).
- CLI/analytics consumers that previously read alias keys continue working by deriving values from explicit metrics (with fallback for legacy artifacts).
- Tests cover the alias-removal contract.

## Constraints / Decision

- Preserve backward compatibility for historical artifacts by adding fallback readers where needed.
- New artifacts should use explicit metric naming as source of truth.
- Compatibility readers only collapse strict/practical aliases when explicit benchmark metrics are present; legacy split `precision/recall` and `practical_precision/practical_recall` remain preserved when explicit fields are absent.
- Dashboard `Previous Runs` + trend chart now render explicit metric names (`strict_accuracy`, `macro_f1_excluding_other`) while retaining legacy fallback ingestion for old benchmark artifacts.

## Rollback / Compensation

- Revert this task's code/docs changes to restore alias field emission in benchmark eval reports.
