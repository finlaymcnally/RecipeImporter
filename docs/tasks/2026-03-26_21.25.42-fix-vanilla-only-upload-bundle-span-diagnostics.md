## Problem

Vanilla-only single-book upload bundles were reporting `recipe_span_count: 0` and emitting a triage note that implied active recipe spans were not discovered.
The underlying vanilla run still had line-role projection telemetry with real recipe-span counts; the bundle helper was just pair-only.

## Acceptance Criteria

- Vanilla-only `upload_bundle_v1` preserves single-run recipe-span counts from benchmark-side line-role projection telemetry when no comparison pairs exist.
- Pairless triage messaging says the bundle is unpaired instead of claiming spans were not discovered.
- Pairless turn-1/overview guidance points operators at span/stage views before empty pair inventory.
- Existing pair-mode upload-bundle behavior stays green.

## Verification

- `. .venv/bin/activate && pytest tests/bench/test_benchmark_cutdown_for_external_ai.py -q -k "single_run_preserves_span_signal or includes_analysis_payloads or includes_navigation_and_locators"`
- `. .venv/bin/activate && pytest tests/bench/test_benchmark_cutdown_for_external_ai.py -q -k upload_bundle`

## Evidence

- Added a vanilla-only regression test that writes one benchmark run plus line-role projection telemetry and asserts the bundle keeps `recipe_span_count=29`, uses pairless notes, and reorders first-pass guidance.
- The upload-bundle-focused pytest slice passes in `.venv` after the change.

## Constraints / Decision

- Keep scope to upload-bundle diagnostics only; do not invent synthetic pair deltas for unpaired runs.
- Preserve pair-only sections as empty when no comparison exists, but fill the single-run span signal from existing benchmark artifacts and make the wording explicit.
