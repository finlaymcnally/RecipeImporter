## Problem

Several split CLI support modules resolve command callables through tiny `*_command()` wrappers.
Those wrappers depended on `cookimport.cli_commands.<family>` already having command exports populated by Typer registration side effects, so deleting or missing the module-level export made wrappers fail with `AttributeError` even though the command owner was otherwise valid.

## Acceptance Criteria

- Split CLI command wrappers recover the expected command callable even when the owner module export is missing.
- A focused offline test fails on the old wrapper behavior and passes after the fix.
- The broader unresolved-global audit still passes in `.venv`.

## Verification

- `. .venv/bin/activate && pytest tests/cli/test_cli_command_resolution.py -q`
- `. .venv/bin/activate && pytest tests/core/test_benchmark_undefined_names.py -q`

## Evidence

- Before the fix, deleting `cookimport.cli_commands.stage.stage` or `cookimport.cli_commands.labelstudio.labelstudio_benchmark` caused the corresponding split wrapper to fail immediately.
- After the fix, wrappers recover the expected command by name and the focused CLI wrapper test passes offline.

## Constraints / Decision

- Keep the fallback cheap and local: import `cookimport.cli` if needed, then fall back to owner-module `register(...)` if the export is still missing.
- Keep coverage generic by auto-discovering `*_command` wrappers under `cookimport.cli_support`.
