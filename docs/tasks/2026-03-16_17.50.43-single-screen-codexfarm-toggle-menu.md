Problem statement

The interactive CodexFarm submenu should let the operator see and adjust every Codex-backed step from one screen.
Checkbox highlighting was too ambiguous, and separate yes/no prompts still felt like multiple steps instead of one real submenu.

Acceptance criteria

- When interactive import or benchmark chooses CodexFarm, the follow-up UI is one shared submenu listing each available step with aligned `[Yes]` / `[No]` boxes.
- Up/down moves between rows, left/right sets the current row to `[Yes]` or `[No]` in place with a visible active marker, Enter can still flip the current step row, and `Continue` accepts the current state for all listed steps at once.
- The top workflow menu shows only the workflow family labels; the concrete pipeline ids live on the Codex submenu rows.
- Import still exposes recipe correction plus knowledge harvest only; benchmark still exposes recipe correction, block labelling, and knowledge harvest.
- Targeted chooser tests and interactive benchmark helper tests pass.

Verification steps

- `source .venv/bin/activate && pytest tests/cli/test_c3imp_interactive_menu.py -k 'choose_run_settings or prompt_codex_surface_menu or interactive_import_passes_knowledge_pipeline_settings' -q`
- `source .venv/bin/activate && pytest tests/labelstudio/test_labelstudio_benchmark_helpers_interactive.py -k 'interactive_benchmark' -q`

Evidence

- Before the change, the Codex submenu either used checkbox highlighting or walked through separate yes/no prompts for each step.
- After the change, the targeted chooser tests and interactive benchmark helper tests pass with the single-screen toggle list, left/right in-place state changes, and `Continue`.

Constraints/gotchas + decision

- Keep the behavior inside `cookimport/cli_ui/run_settings_flow.py` so import and benchmark continue to share one chooser implementation.
- Preserve the flow-specific option set: import does not expose block labelling because stage call adapters do not carry that setting into `stage(...)`.
