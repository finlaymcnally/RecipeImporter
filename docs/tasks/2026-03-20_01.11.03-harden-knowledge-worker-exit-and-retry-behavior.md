# Harden Knowledge Worker Exit, Retry, and Oversize-Shard Behavior

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The user-visible problem is that Codex-backed non-recipe knowledge review can still hold an interactive benchmark open long after the useful work is over, or spend a large amount of time and tokens on a shard that ends in an obviously bad result. In the March 20 Salt Fat Acid Heat runs, one knowledge worker died almost immediately on an exploratory `rg`, then a strict retry spent about seventeen minutes thrashing before returning a semantically empty all-`u:false` answer. A later run showed the complementary failure: the main worker successfully wrote a large `out/<shard>.json` file, then kept probing its own output with shell commands until the watchdog killed it, forcing an unnecessary retry.

After this change, the knowledge stage should behave like a bounded worker system rather than a hopeful open-ended conversation. Main workers should stop once they have produced an owned output file. Strict retries should stop quickly when they emit malformed pseudo-final output, go silent, or receive an oversized shard that is structurally likely to thrash. The benchmark should still preserve the current conservative validation behavior, but it should fail fast and report clearly instead of letting one bad shard dominate the whole run.

This plan is correct when a future operator can run the same single-offline benchmark flow and observe all of the following. A worker that writes a valid owned output file does not keep wandering into post-write shell checks. A strict retry that emits malformed JSON-ish chatter or stays silent is terminated promptly with an explicit reason. Very large retry payloads are either subdivided into smaller deterministic follow-up work units or rejected quickly without spending a full long-form retry turn. The resulting manifests and live status files make it obvious why the shard ended the way it did.

## Context and Orientation

In this repository, “knowledge stage” means the optional Codex-backed non-recipe review that runs on Stage 7 seed non-recipe spans. The runtime entrypoint is `cookimport/llm/codex_farm_knowledge_orchestrator.py`. That module builds shard bundles, writes worker folders under `data/output/<timestamp>/.../raw/llm/<source_slug>/knowledge/workers/`, launches Codex via `cookimport/llm/codex_exec_runner.py`, validates the returned payloads, and then promotes accepted decisions and snippets into the final knowledge authority.

There are two distinct attempt types in the current runtime. A “workspace worker” is the main long-lived Codex session that starts in a sterile workspace, reads local files such as `worker_manifest.json`, `assigned_shards.json`, `hints/<shard>.md`, and `in/<shard>.json`, and writes `out/<shard>.json`. A “strict retry” is a follow-up direct `codex exec` call that receives the shard input inline and is supposed to return one compact JSON answer with no tools. The workspace worker is good at bounded local file work. The strict retry is meant to rescue a failed shard without giving the model permission to explore the filesystem again.

The key files for this plan are:

- `cookimport/llm/codex_farm_knowledge_orchestrator.py`: knowledge-stage planning, worker prompt text, retry supervision callback, retry launch, promotion, and reporting.
- `cookimport/llm/codex_exec_runner.py`: shared direct-exec process wrapper, event parsing, live snapshots, timeout enforcement, and workspace-command classification.
- `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md`: the strict no-tools knowledge prompt template used by one-shot and retry calls.
- `llm_pipelines/schemas/recipe.knowledge.v1.output.schema.json`: the exact compact output schema for a knowledge shard.
- `cookimport/llm/codex_farm_knowledge_ingest.py`: semantic validation and conversion of knowledge outputs into repo-owned models.
- `cookimport/llm/README.md`: short runtime-facing documentation for the active LLM contracts.

The two saved regressions that motivate this plan are:

- The `2026-03-20_00.15.56` run under `data/output/2026-03-20_00.15.56/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-20_00.16.47/`, especially `raw/llm/saltfatacidheatcutdown/knowledge/workers/worker-005/`.
- The `2026-03-20_00.50.25` run under `data/output/2026-03-20_00.50.25/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-20_00.51.02/`, especially `raw/llm/saltfatacidheatcutdown/knowledge/workers/worker-008/`.

The current tree already contains two partial mitigations. First, the workspace worker prompt for knowledge now explicitly restates the compact output row keys `cid`, `u`, `d`, and `s`, and forbids drifting back to `block_decisions` / `snippets`. Second, strict retries now carry both a hard subprocess timeout and a silence timeout. This plan therefore focuses on the remaining gaps rather than re-implementing those landed changes.

## Plan of Work

The implementation should proceed in four narrative steps.

The first step is to harden the meaning of “final” inside the strict retry supervisor. Today, the live snapshot only knows whether some agent message exists. That is too weak, because the March 20 retry proved that a malformed JSON-ish answer can appear early, flip the “final” bit, and still lead to a long bad turn. The supervision logic in `cookimport/llm/codex_farm_knowledge_orchestrator.py` should start distinguishing between three states: no final answer yet, malformed provisional final output, and schema-shaped final output. The malformed state should not suppress the reasoning and outlier guards. It should either trigger immediate termination or at least keep the run eligible for the same silence and overrun protections as a non-final turn.

The second step is to make the workspace worker stop when the useful work is done. The later `ks0007.nr` run showed that the worker can finish writing `out/<shard>.json` and then waste the remainder of the session on self-checking. The worker prompt should state in plainer language that once the shard output file exists and contains one complete JSON object, the worker must stop and return. The runtime should also enforce this behavior from the outside where possible. If the workspace loop observes a completed `out/<shard>.json` file for every assigned shard and no newer useful file changes are happening, it should end the worker session instead of waiting for the model to voluntarily stop exploring.

The third step is to stop giving very large failed shards one giant strict no-tools retry. The code already has deterministic shard planning for the main pass, but the retry path still inlines one huge bundle when a main worker fails. For shards above a retry-safe size threshold, likely based on a mix of chunk count, prompt characters, and maybe estimated visible input tokens, the system should choose one of two deterministic alternatives. The preferred path is to split the failed shard into smaller retry children whose ownership is explicit and whose outputs can be rejoined by the existing knowledge ingest path. The fallback path, if split retry would be too invasive for the first pass, is to skip the monolithic retry entirely and mark the shard as unreviewed with a specific oversized-retry reason. The key requirement is that the runtime must no longer feed a very large inline shard into one fragile strict JSON turn and hope for the best.

The fourth step is to align reporting and tests with the new exit behavior. The worker and retry status files should make it obvious whether a shard ended because it completed valid work, stopped after writing output, emitted malformed pseudo-final output, hit a silence timeout, or was refused as oversized for monolithic retry. The tests should replay the two March 20 artifact patterns without live tokens and prove that the runtime exits decisively instead of reproducing the same long waits.

## Milestone 1: Final-message quality gating for strict retries

In `cookimport/llm/codex_exec_runner.py`, add a lightweight helper that inspects the last agent message text for strict JSON calls and classifies it into one of three buckets: absent, malformed, or plausibly JSON-like. “Malformed” here does not mean full schema validation in the runner. It means the message is obviously not a valid final JSON object for this transport, for example because it starts with prose, contains visible planning text before the first `{`, or cannot be parsed as a single JSON object at all. Keep the helper transport-agnostic enough that knowledge can use it now and other strict JSON callers could adopt it later.

In `cookimport/llm/codex_farm_knowledge_orchestrator.py`, update the strict retry supervision callback to use that richer classification. If the retry emits a malformed pseudo-final message, treat it as a retryable failure immediately or keep all existing reasoning, silence, and outlier guards active instead of considering the shard “final-like”. Do not let one bad malformed agent message effectively immunize the turn from later watchdog decisions.

Record the richer status in `live_status.json` and final `status.json`. Add fields that explain whether the latest agent message was absent, malformed, or parseable-enough to count as a candidate final answer. These fields are meant for debugging, not for becoming a second validator.

## Milestone 2: Stop workspace workers after owned output files exist

Update the knowledge workspace-worker prompt builder in `cookimport/llm/codex_farm_knowledge_orchestrator.py` so the stop rule is explicit and hard to misread. The current step “Continue until every assigned shard has an output file or you cannot proceed” is directionally right but too easy for the model to reinterpret as “write the file, then run extra checks.” Strengthen it so it says that once the required output file exists for the assigned shard, the worker should stop and not run further shell checks against the file unless the file is clearly incomplete.

Add a repo-owned supervisor path that notices when all assigned shard output files exist and have stabilized. The exact implementation can be conservative. For a single-shard worker, it is enough to notice that `out/<shard>.json` exists, is non-empty, and no new writes have occurred for a short grace period. For multi-shard workers, the same rule should apply when every assigned shard has a non-empty output file. Once that condition holds, terminate the worker session as completed instead of giving it more time to drift.

Keep this bounded to knowledge workers in the first pass. The implementation can live in shared runner utilities if that is simpler, but the behavior change should be enabled only for knowledge until the tests prove it is safe.

## Milestone 3: Deterministic oversized-retry handling

Define a retry-safe size policy in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. The policy should use deterministic bundle properties that already exist or can be derived locally, such as owned chunk count, prompt character count, or estimated visible input token count. The threshold should be chosen from the saved failure evidence, not by guesswork. The earlier `ks0004.nr` retry is the negative example: a large inline bundle plus a strict one-line JSON contract produced bookkeeping thrash and a useless all-empty answer.

When a failed shard exceeds the retry-safe threshold, do not launch the existing monolithic strict retry. Instead, prefer a split-retry path that partitions the owned chunks into smaller contiguous retry children and runs the current strict retry contract over each child. Rejoin those child outputs through the existing knowledge ingest path so exact owned coverage still holds.

This implementation landed the first-pass oversized refusal mode. Multi-chunk watchdog-killed shards now mark themselves unreviewed with `watchdog_retry_oversized_skipped`, preserve seed authority, and report the deterministic size reason in proposal/status metadata instead of launching one monolithic strict retry.

## Validation and Acceptance

The implementation is acceptable only if all of the following are true.

First, the earlier March 20 retry-thrash pattern is no longer possible. A replay-style test or a fake-runner test must show that a malformed pseudo-final agent message does not buy the retry an open-ended long turn. The retry must either terminate quickly with an explicit malformed-output reason or remain subject to the same silence and overrun guards until a truly parseable final answer appears.

Second, the later March 20 post-output-drift pattern is no longer possible for knowledge workers. A test must show that once `out/<shard>.json` exists for the assigned shard and has stabilized, the worker session is treated as complete. The system should not wait for the model to decide on its own that no more shell checks are needed.

Third, a very large failed shard is no longer sent through one monolithic uncontrolled strict retry. The acceptance proof can be either a split-retry test that yields smaller child retries or a refusal-path test that records a deterministic oversized-skip reason and leaves the shard unreviewed.

Fourth, the knowledge validator remains conservative. The saved all-`u:false`, `d=[]`, `s=[]` answer from the earlier bad retry must still be rejected as semantically empty. This plan is about runtime bounding, not about loosening validation to accept junk.

Fifth, the status artifacts are more legible than before. A human reading `live_status.json` and final `status.json` should be able to tell whether a shard ended due to malformed pseudo-final output, silence timeout, subprocess timeout, oversized retry refusal, post-output completion, or ordinary completion.

## Idempotence and Recovery

This work is safe to retry if implemented carefully. The runtime changes are additive and bounded to knowledge supervision. If a test or implementation attempt fails midway, the repository can be left in a consistent state by keeping existing status fields and only adding new ones. New reason codes should be additive; do not rename old ones unless every reader and test is updated in the same change.

If split retry is attempted and proves too invasive, it is acceptable to stop at the oversized refusal milestone as long as the plan document is updated to reflect that design choice and the remaining split-retry work is moved into `Progress` as incomplete follow-up. Do not silently leave half a split-retry path wired into production.

Because the saved regression artifacts live under `data/output/` and `data/golden/`, do not mutate those historical files during implementation. Tests should copy or synthesize inputs rather than editing the evidence packets in place.
