# Simplify knowledge stage to single-chunk review tasks

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

The current knowledge stage still has an extra layer of grouping that the rest of the pipeline no longer needs. Deterministic chunking first creates knowledge chunks from review-eligible non-recipe text, then the planner bundles several chunks into one shard, then the orchestrator can split that shard again into smaller task packets, and finally the worker must make block-level semantic decisions inside those packets. That shape saves some queue rows, but it creates the exact confusion the user called out: one useful sentence can make a mixed memoir, praise, or book-framing packet feel globally useful even when only one block deserves `knowledge`.

After this change, the knowledge stage should work more like line-role. Each review task should own exactly one deterministic knowledge chunk and a small amount of neighboring context that is clearly marked as non-authoritative context. The worker should decide block-by-block within that one chunk only. A human should be able to inspect `raw/llm/<workbook>/knowledge/in/*.json`, `task_manifest.jsonl`, and the worker prompts and see one clean story: one chunk to judge, nearby context for grounding, one result file for that one chunk.

The user-visible benefit is simpler behavior and simpler debugging. If a chunk is memoir mixed with one useful mechanism sentence, the worker no longer sees unrelated neighboring chunks in the same billed task. The visible proof is that input artifacts now show one chunk per task, the confusing packet vocabulary disappears from the happy path, and approved benchmark reruns should stop showing broad mixed-packet promotions where framing blocks ride along with a nearby useful block.

## Context and Orientation

The knowledge stage is the optional non-recipe semantic review phase. It runs after Stage 7 has already decided which outside-recipe blocks are immediately excluded junk and which blocks are review-eligible. The knowledge stage is the only component allowed to convert review-eligible outside-recipe text from final `other` to final `knowledge`.

There are four important terms in this repository for this plan.

A “block” is one extracted unit of book text. This is the final authority unit scored in `08_nonrecipe_spans.json`, `09_knowledge_outputs.json`, and benchmark stage-block artifacts.

A “chunk” is a deterministic grouping of one or more neighboring non-recipe blocks created by `cookimport/parsing/chunks.py`. Chunking exists so the knowledge reviewer sees a coherent local unit rather than one isolated line with no neighborhood at all.

A “shard” currently means one JSON input file under `raw/llm/<workbook>/knowledge/in/*.json`. Today a shard can contain several chunks because `cookimport/llm/codex_farm_knowledge_jobs.py` bundles nearby chunks together.

A “task packet” currently means the orchestrator’s second-stage split of that shard into worker-owned task rows. `cookimport/llm/codex_farm_knowledge_orchestrator.py` uses `_build_knowledge_task_plans(...)` to take one multi-chunk shard and break it into smaller worker tasks again.

That two-layer design is the core thing this plan removes.

The key code paths are:

- `cookimport/parsing/chunks.py`
  This file creates deterministic non-recipe `KnowledgeChunk` objects and summarizes per-chunk utility cues. This plan does not remove chunking itself.

- `cookimport/llm/codex_farm_knowledge_jobs.py`
  This file turns deterministic chunks into model-facing JSON inputs under `knowledge/in/*.json`. Right now it bundles multiple chunks together through `_bundle_prepared_chunks(...)`. This plan changes that so one chunk becomes one input file.

- `cookimport/llm/codex_farm_knowledge_orchestrator.py`
  This file creates runtime task manifests, worker prompts, progress labels, and supervision behavior. Right now `_build_knowledge_task_plans(...)` creates one or more task packets from each multi-chunk shard. This plan removes that second packetization layer.

- `cookimport/llm/knowledge_workspace_tools.py`
  This file renders `CURRENT_BATCH.md`, `current_batch.json`, `CURRENT_TASK.md`, `OUTPUT_CONTRACT.md`, and the helper tool under `tools/knowledge_worker.py`. This plan updates the worker-facing language so the happy path talks about one chunk-task at a time instead of generic “packet” review.

- `cookimport/llm/codex_farm_knowledge_ingest.py`
  This file validates worker outputs, enforces exact chunk/block ownership, and promotes accepted results into final authority. The first landing should disturb this as little as possible.

- `docs/10-llm/10-llm_README.md`
  This is the short user-facing description of how the knowledge stage works. After implementation, it must stop describing multi-chunk packet review as the normal model.

The intended end state is simple:

1. Deterministic chunking still happens.
2. Each chunk becomes exactly one knowledge review task.
3. That task includes only that chunk as authoritative owned input, plus small nearby context if configured.
4. The worker returns a result file for that single chunk.
5. The validator and promotion seams still operate on exact block ownership and final block-level `knowledge|other` decisions.

This plan does not change the final semantic authority rule. The knowledge stage remains the sole semantic judge for review-eligible outside-recipe text. It only changes the shape of the review task so that the worker is not asked to semantically judge several chunks together.

## Milestones

### Milestone 1: Replace multi-chunk shard bundling with one-chunk input files

At the end of this milestone, `cookimport/llm/codex_farm_knowledge_jobs.py` should write one model-facing JSON input file per deterministic knowledge chunk. The current `_bundle_prepared_chunks(...)` consolidation step should be bypassed or removed from the happy path. Each input file should still include local context blocks through the existing `x` payload so the reviewer has immediate neighbors, but the authoritative owned chunk list `c` should contain exactly one chunk.

This milestone is the main architectural simplification. It changes the semantic unit of review at the source instead of trying to keep rebundling and then teaching the prompt to resist rebundling side effects. After this milestone, `knowledge/in/*.json` should already read like line-role-style local review units.

Acceptance for this milestone is visible on disk. For a deterministic fixture that currently produces one multi-chunk input file, the stage should instead write one file per chunk, each file should have exactly one `c[*]` entry, and context should still appear only in `x`.

### Milestone 2: Remove orchestrator packet splitting and make shard equal task

At the end of this milestone, `cookimport/llm/codex_farm_knowledge_orchestrator.py` should stop constructing `_KnowledgeTaskPlan` groups from multi-chunk shards. Each planned shard should already be the final worker task. In practice that means `_build_knowledge_task_plans(...)` becomes trivial or disappears, task manifests directly mirror the one-chunk shard entries, and progress/telemetry stop talking about “task packets” as a separate abstraction.

This milestone matters because the planner simplification is incomplete if the orchestrator still preserves a second packet vocabulary. The runtime should have one queue concept, not two.

Acceptance for this milestone is a runtime artifact tree where `task_manifest.jsonl` and `knowledge/in/*.json` have the same review-unit granularity and where progress labels no longer need to distinguish “parent shard” from “packet”.

### Milestone 3: Simplify worker prompts and helper sidecars to single-chunk tasks

At the end of this milestone, the worker prompt, `CURRENT_BATCH.md`, `CURRENT_TASK.md`, and `OUTPUT_CONTRACT.md` should all describe a single-chunk review task. The happy path should say, in plain language, that each task owns one chunk, nearby context is non-authoritative, and final decisions remain block-local. The prompt should no longer need to warn about promoting the whole packet because a single useful sentence is present in a different chunk; that specific confusion source should have been deleted by Milestones 1 and 2.

This milestone is the user-visible cleanup. A novice opening a worker root should be able to understand the stage without mentally translating packet language into chunk language.

Acceptance for this milestone is focused prompt/helper tests that assert the new worker wording and artifact fields, plus a short manual inspection of generated sidecars for one local fixture.

### Milestone 4: Keep validator and promotion seams stable while shrinking the task shape

At the end of this milestone, `cookimport/llm/codex_farm_knowledge_ingest.py` and the final staging artifacts should still behave the same semantically even though tasks are smaller. Each accepted result should still yield exact block-level final `knowledge|other` decisions and grounded snippets. The first landing should deliberately keep the top-level worker result schema narrow and compatible: one task file still contains `packet_id` plus `chunk_results`, but `chunk_results` must now contain exactly one row.

This milestone keeps the change reviewable. The refactor’s purpose is to simplify the review unit, not to rewrite every downstream consumer at once.

Acceptance for this milestone is the existing ingest and promotion tests continuing to pass after task granularity changes, plus at least one new regression test proving that a one-chunk task still validates and promotes normally.

### Milestone 5: Update docs, compatibility notes, and approval-gated benchmark proof

At the end of this milestone, the short knowledge-stage docs should describe the new design accurately, compatibility settings that used to influence bundling should have an explicit no-op or warning story, and the plan should record the exact live benchmark command to run if and only if the user approves token spend.

The benchmark part remains approval-gated, but the plan must still define what proof matters. The question is no longer “did packet count go down?” The new proof is “did the simpler single-chunk review shape reduce mixed-packet semantic mistakes enough to justify the design.”

## Plan of Work

Start in `cookimport/llm/codex_farm_knowledge_jobs.py`. Remove the current consolidation step that groups several `_PreparedKnowledgeBundleChunk` rows into one semantic review input. The simplest safe landing is to keep `_PreparedKnowledgeBundleChunk` and `_build_bundle_job_payload(...)`, but call that builder with exactly one prepared chunk at a time. Each emitted `ShardManifestEntryV1` should own exactly one chunk id, one ordered block range, and one `input_payload["c"]` row. Continue populating context blocks in `x` and recipe guardrails in `g` exactly as today.

While doing that, keep compatibility surfaces explicit. If `knowledge_prompt_target_count` or `knowledge_shard_target_chunks` are set, the planner should not secretly reintroduce multi-chunk bundling. Instead, record a planning warning or manifest note that this single-chunk design ignores bundling knobs. This plan prefers a loud compatibility warning over hidden legacy behavior.

Then move to `cookimport/llm/codex_farm_knowledge_orchestrator.py`. Delete or collapse `_build_knowledge_task_plans(...)`, `_KnowledgeTaskPlan`, and the packet-specific metadata fanout so that each shard manifest entry is already the final worker task. `TaskManifestEntryV1` generation should become straightforward: one task id per shard id, one owned chunk id, one `result_path`. Progress reporting should stop saying “task packet” and instead say “task” or “chunk review task” consistently.

Keep the batch sidecar model. This plan does not remove `current_batch.json` or the worker queue. A worker should still process many tasks over one session. What changes is the size of each task, not the existence of the queue.

Next update `cookimport/llm/knowledge_workspace_tools.py`. The helper surface can remain batch-first, but the language should stop talking about packets that own many chunks. `CURRENT_BATCH.md` should explain that each task row owns one chunk plus nearby context. The scaffold and checker should still emit the current semantic result shape, but they should assert that exactly one `chunk_results[*]` row is present for each task. Update any examples or strong-cue repair text that currently assume a multi-chunk packet.

After that, update the worker prompt text in `cookimport/llm/codex_farm_knowledge_orchestrator.py` and the compact prompt asset `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md`. The prompt should explicitly say that only one chunk is owned at a time, that the local context is informational rather than authoritative, and that the worker must make block-local decisions within that one chunk. Remove wording that exists only to mitigate cross-chunk packet confusion.

Touch `cookimport/llm/codex_farm_knowledge_ingest.py` only where task/task-count assumptions changed. The validator should still enforce exact chunk id coverage, but in the first landing it should now expect one `chunk_results` row per task because one task owns one chunk. Preserve the current snippet-grounding, anti-echo, and fail-closed semantic checks.

Finally, update the short docs in `docs/10-llm/10-llm_README.md` and `cookimport/llm/README.md`. Keep the notes short and concrete: one deterministic chunk per task, small context kept, batch queue retained, packet layer removed.

## Idempotence and Recovery

The implementation steps in this plan should be idempotent. Re-running the deterministic planners or tests should overwrite generated local artifacts cleanly and not require manual cleanup beyond the repo’s normal temporary outputs.

The safest migration order is additive then subtractive:

1. First make `build_knowledge_jobs(...)` capable of writing one-chunk inputs while keeping downstream result handling compatible.
2. Then simplify the orchestrator task planning once the input granularity is already stable.
3. Then simplify the worker-facing prose and docs after the runtime shape is true in code.

If the refactor stalls halfway, the recovery rule is simple: do not partially flatten the worker result schema at the same time. Keep the current `chunk_results` compatibility shape until the one-chunk task design is stable and tested. That keeps rollback limited to task planning rather than promotion semantics.

If a benchmark rerun later shows the simpler task shape raises runtime overhead too far, the fallback is not to restore multi-chunk semantic packets immediately. The safer fallback is to keep one-chunk semantic ownership and then optimize worker assignment, queue batching, or local context size separately.

## Interfaces and Dependencies

At the end of this plan, the following repository interfaces must exist and behave as described.

In `cookimport/llm/codex_farm_knowledge_jobs.py`:

- `build_knowledge_jobs(...)` must emit one `ShardManifestEntryV1` per deterministic chunk.
- The emitted `ShardManifestEntryV1.input_payload["c"]` must contain exactly one chunk.
- Local context through `x` and recipe guardrails through `g` must still be supported.
- Compatibility settings that used to steer bundling must no longer silently rebundle chunks.

In `cookimport/llm/codex_farm_knowledge_orchestrator.py`:

- shard planning and task planning must collapse to one review unit.
- worker/task progress terminology must stop distinguishing parent shard from packet in the happy path.
- the worker prompt must describe one-chunk tasks with local context.

In `cookimport/llm/knowledge_workspace_tools.py`:

- scaffolded task outputs must still use the current semantic packet-result schema for the first landing, but each task must own exactly one chunk.
- helper sidecars must describe one-chunk tasks plainly.

In `cookimport/llm/codex_farm_knowledge_ingest.py`:

- validation must continue to enforce exact owned chunk coverage and exact owned block coverage.
- the first landing should remain compatible with the current top-level semantic result envelope.
