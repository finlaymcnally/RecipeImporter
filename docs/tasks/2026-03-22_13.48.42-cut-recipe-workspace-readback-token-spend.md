# Cut recipe workspace readback token spend after the command-count win

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The recipe benchmark is now reliable again, but it is still too expensive because recipe workers spend too much of each long-lived Codex workspace session rereading local files before they edit the prepared drafts. After this change, a recipe worker should be able to open one repo-written shard packet, trust the prewritten drafts, edit them, and run the repo-owned `check-current` / `install-current` loop without defaulting to repeated reads of manifests, contracts, examples, or helper source. The user-visible proof is a rerun of the same Salt Fat single-book codexfarm slice where recipe keeps `validated_shard_count: 5`, keeps `watchdog_killed_shard_count: 0`, keeps `watchdog_recovered_shard_count: 0`, and reduces recipe command-execution token spend below the current `536003` while holding command count at or below the current `47`.

## Surprises & Discoveries

- Observation: recipe correctness is no longer the limiting problem.
  Evidence: `data/golden/benchmark-vs-golden/2026-03-22_13.22.18/single-book-benchmark/saltfatacidheatcutdown/codexfarm/prompt_budget_summary.json` records recipe `validated_shard_count: 5`, `watchdog_killed_shard_count: 0`, `watchdog_recovered_shard_count: 0`, and `structured_followup_call_count: 0`.

- Observation: recipe command count fell sharply, but recipe command-execution tokens still went up.
  Evidence: the same summary shows recipe `command_execution_count_total: 47`, which beats both `93` at `2026-03-22_12.08.53` and `91` at `2026-03-22_04.26.00`, while `command_execution_tokens_total: 536003` is worse than `473128` and slightly worse than `531073`.

- Observation: the expensive part is the interactive workspace session, not the bytes printed by local shell commands.
  Evidence: `data/output/2026-03-22_13.22.18/.../recipe_phase_runtime/workers/worker-001/status.json` shows `command_execution_count: 9`, `visible_input_tokens: 154`, `tokens_cached_input: 69760`, and `wrapper_overhead_tokens: 76599` for the primary telemetry row.

- Observation: the prompt still explicitly told recipe workers to open too many files before they acted.
  Evidence: `.../worker-001/prompt.txt` said to open `worker_manifest.json`, `CURRENT_TASK.md`, `current_task.json`, `CURRENT_TASK_FEEDBACK.md`, and `OUTPUT_CONTRACT.md` up front, then to open `hints/<task_id>.md` and `in/<task_id>.json` before editing each draft.

- Observation: the repo-written surfaces were strongest for the first task, not for the entire shard.
  Evidence: `.../worker-001/CURRENT_TASK.md` gave a good starting brief for task 1, but the later worker status files still showed per-task `cat` loops for tasks 2 through 5 before `finalize-all`.

- Observation: the strict output contract made the model conservative enough to reread examples and helper source.
  Evidence: `.../worker-004/status.json` and `.../worker-005/status.json` both showed reads of `examples/*.json` and slices of `tools/recipe_worker.py` even after the startup churn fixes were landed.

- Observation: a better recipe helper path needed to change the worker trust model, not just cut more commands from the manifest.
  Evidence: previous changes already added `prepare-all`, `finalize-all`, `scratch/_prepared_drafts.json`, and `stamp-status`, but the model still voluntarily paid orientation cost before using them.

- Observation: the recipe helper already contained most of the right current-task vocabulary; the missing piece was that the standalone CLI did not actually advance the sidecars after `install-current`.
  Evidence: `cookimport/llm/recipe_workspace_tools.py` already rendered `check-current` / `install-current` guidance into `CURRENT_TASK.md` and `CURRENT_TASK_FEEDBACK.md`, but the embedded `tools/recipe_worker.py` script only rewrote feedback before this implementation pass.

## Decision Log

- Decision: keep this plan tightly scoped to recipe workspace readback and token spend.
  Rationale: the immediate user question was why recipe still spends too many tokens after command count fell; line-role and knowledge regressions are real but they are different workstreams and would blur the acceptance bar for this plan.
  Date/Author: 2026-03-22 / Codex

- Decision: make one repo-written shard packet the only required startup read in the normal recipe path.
  Rationale: the old prompt and manifest both encouraged workers to read several files before they could feel safe acting. A single authoritative packet is the clearest way to remove that ambiguity and make the cheapest path obvious.
  Date/Author: 2026-03-22 / Codex

- Decision: treat `OUTPUT_CONTRACT.md`, mirrored examples, and `tools/recipe_worker.py` as fallback debugging surfaces rather than normal worker inputs.
  Rationale: the worker should not need to inspect reference docs or tool source just to complete the common shard flow. If those files remain default reads, the token problem is unlikely to improve materially.
  Date/Author: 2026-03-22 / Codex

- Decision: push more task progression into repo-owned helper state instead of leaving later-task discovery entirely to the model.
  Rationale: the old `CURRENT_TASK.md` only helped for the first task. The worker still spent tokens deciding what to open next across the rest of the shard, which was avoidable once the helper/runtime precomputed more of that loop.
  Date/Author: 2026-03-22 / Codex

- Decision: use one repo-written `SHARD_PACKET.md` plus the existing current-task sidecars instead of switching recipe all the way over to the knowledge helper shape.
  Rationale: recipe already had useful current-task files and draft preparation. The smallest successful change was to make one packed shard summary authoritative at startup and then make `install-current` advance the existing sidecars.
  Date/Author: 2026-03-22 / Codex

- Decision: keep benchmark reruns approval-gated.
  Rationale: this repository explicitly forbids token spend without user approval, and the implemented local work was enough to close the code/documentation portion of the plan.
  Date/Author: 2026-03-22 / Codex

## Outcomes & Retrospective

This plan is implemented locally. Recipe workers now start from one packed `SHARD_PACKET.md`, rely on repo-written current-task sidecars for the active task, and use a repo-owned `check-current` / `install-current` loop that advances the queue after each accepted task. The prompt, helper surfaces, prepared-draft metadata, docs, and focused regressions now all point at the same cheaper path: edit the named draft, validate it, install it, reopen the advanced sidecars, and continue immediately if another task is active.

The original stale version of this plan stopped at design intent, but the implementation did land in code and tests. ## Context and Orientation

Recipe workspace workers live primarily in `cookimport/llm/codex_farm_orchestrator.py` and `cookimport/llm/recipe_workspace_tools.py`. The orchestrator writes the worker prompt, worker root, helper files, and task queue, then harvests validated task outputs back into final shard proposals. The workspace-tools module renders the helper CLI and the repo-written sidecars such as `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`, and now `SHARD_PACKET.md`. `cookimport/llm/codex_exec_runner.py` mirrors those files into the sterile direct-exec workspace and records command telemetry for the long-lived Codex session.

In this repository, a "recipe worker assignment" means one long-lived workspace session that owns several task packets from one shard. A "task packet" is one row in `assigned_tasks.json` plus its `in/<task_id>.json`, `hints/<task_id>.md`, `scratch/<task_id>.json`, and `out/<task_id>.json` paths. A "current-task surface" is the repo-written bundle `current_task.json`, `CURRENT_TASK.md`, and `CURRENT_TASK_FEEDBACK.md`. A "shard packet" is the new human-readable `SHARD_PACKET.md` file that summarizes the compact output rules, allowed statuses, queue roster, and short per-task summaries so the worker does not need to reopen raw hint/input files during the common path.

The benchmark evidence that motivated this plan is in `data/golden/benchmark-vs-golden/2026-03-22_13.22.18/single-book-benchmark/saltfatacidheatcutdown/codexfarm/prompt_budget_summary.json` plus the worker artifacts under `data/output/2026-03-22_13.22.18/.../recipe_phase_runtime/workers/`. The implemented code path is reflected in `cookimport/llm/README.md`, `docs/10-llm/10-llm_README.md`, and the focused regression coverage under `tests/llm/`.

## Milestones

### Milestone 1: Replace startup readback with one packed shard summary

This milestone changed the normal recipe worker startup flow so the worker no longer begins by opening a manifest, contract, examples, and helper source just to feel safe. The repo now writes `SHARD_PACKET.md` before the worker starts, and the prompt treats that packet as the authoritative packed shard summary. At the end of this milestone, the worker can read one startup packet, inspect the active current-task sidecars, edit the prepared draft, and stay on the paved road.

The observable proof is in the generated worker root and the prompt text. `SHARD_PACKET.md` exists beside `CURRENT_TASK.md`, and the prompt explicitly demotes `OUTPUT_CONTRACT.md`, `examples/*.json`, and `tools/recipe_worker.py` to fallback/debug reads.

### Milestone 2: Make later tasks as cheap as task 1

This milestone closed the main later-task gap by making `install-current` advance `current_task.json`, `CURRENT_TASK.md`, and `CURRENT_TASK_FEEDBACK.md` after each accepted install. At the end of this milestone, the worker no longer needs to rediscover task 2 through N by sweeping raw hints and inputs. The repo-owned sidecars advance to the next pending task and keep the same local loop intact for the whole assignment.

The observable proof is the helper behavior in `tools/recipe_worker.py` as rendered from `cookimport/llm/recipe_workspace_tools.py`, plus the regression tests that assert queue advancement after `install-current`.

### Milestone 3: Keep the contract small and local

This milestone made the compact contract cheap enough to trust during ordinary execution. The current-task brief and `SHARD_PACKET.md` now summarize the compact keys, allowed statuses, draft path, result path, and per-task summaries. `scratch/_prepared_drafts.json` also carries packed task summaries and current-task metadata so regenerated draft batches still point at the same queue state.

The observable proof is that the helper and prompt both describe the common path without requiring `OUTPUT_CONTRACT.md`, examples, or helper source as startup reads, while the final output contract itself remains unchanged.

### Milestone 4: Lock the path in tests and docs

This milestone updated the focused recipe regressions and the short LLM docs so the repo now enforces the intended worker path. The tests assert the shard packet, current-task loop, fallback-only contract/example/tool reads, and queue advancement semantics. The docs explain the preferred worker loop and name the old heavy surfaces as fallback/debug references only.

The observable proof is the focused `.venv` regression set that passed locally on 2026-03-22 14:48 EDT and the updated notes in `cookimport/llm/README.md` and `docs/10-llm/10-llm_README.md`.

## Plan of Work

The implemented change rewrote the normal recipe worker startup path in `cookimport/llm/codex_farm_orchestrator.py::_build_recipe_workspace_worker_prompt(...)`. The worker now starts from `SHARD_PACKET.md`, then uses `CURRENT_TASK.md`, `current_task.json`, and `CURRENT_TASK_FEEDBACK.md` as the authoritative active-task surface. The prompt now explicitly says that raw `hints/*.md`, `in/*.json`, `OUTPUT_CONTRACT.md`, `examples/*.json`, and `tools/recipe_worker.py` are fallback reads only when the packed packet and sidecars are still insufficient.

The main implementation work landed in `cookimport/llm/recipe_workspace_tools.py`. That module now renders `SHARD_PACKET.md`, enriches `scratch/_prepared_drafts.json` with packed task metadata, keeps the current-task brief aligned with the compact contract, and makes `check-current` and `install-current` update repo-readable validation feedback plus the next active task. The helper still preserves `prepare-all`, `finalize`, and `finalize-all` as fallback/debug verbs, but the ordinary queue loop now stays current-task-oriented for the entire assignment.

The worker wrapper and manifest path remained narrow. `cookimport/llm/codex_exec_runner.py` only needed to stay aligned with the new packet and current-task surfaces so the mirrored files, telemetry, and workspace guidance stayed consistent. Short operational docs in `cookimport/llm/README.md` and `docs/10-llm/10-llm_README.md` were updated to describe the new packet-plus-current-task loop and the fallback-only status of the old heavier references.
