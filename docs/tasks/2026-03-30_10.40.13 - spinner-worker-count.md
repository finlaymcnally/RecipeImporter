summary: "Task spec for making the CLI spinner worker panel respect the selected worker count."
read_when:
  - "When changing benchmark or import progress rendering for worker rows"
---

# Problem

Structured stage-progress snapshots can report `worker_total=5`, but the shared spinner still pads the worker panel to eight rows. That makes the Codex Exec status window look like it launched more workers than the user selected.

# Acceptance Criteria

- Structured worker panels render only the reported worker slots for a five-worker run.
- The spinner no longer shows synthetic `worker 06` to `worker 08` idle rows when `worker_total=5`.
- Existing larger worker-count behavior still works.

# Verification

- Repro script before fix: run the structured `worker_total=5` spinner scenario and confirm `worker 06: idle` appears.
- Automated proof after fix:
  - `source .venv/bin/activate && pytest tests/labelstudio/test_labelstudio_benchmark_helpers_progress.py -k "five_worker or ten_knowledge_workers"`

# Evidence

- Before fix, the structured repro printed `worker 06: idle => True` and `worker 08: idle => True`.
- After fix, the new regression test should prove the five-worker panel stops at `worker 05`.

# Constraints / Decision

- Keep the change inside the shared CLI progress renderer so stage runtimes do not need per-stage workarounds.
- Do not change unrelated progress wording or follow-up panels.

# Rollback

- Revert the renderer/test/doc changes together to restore the old eight-row floor.
