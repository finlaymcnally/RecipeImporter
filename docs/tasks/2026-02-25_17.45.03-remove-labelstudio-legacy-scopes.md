# Remove legacy Label Studio scopes from import/export flows

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan must be maintained in accordance with `docs/PLANS.md`.


## Purpose / Big Picture

Label Studio benchmark scoring now uses stage block evidence, so the old Label Studio scope variants `pipeline` and `canonical-blocks` are dead weight in import/export paths and still create confusing menus, flags, manifests, and tests. After this change, Label Studio workflows in this repo are single-mode: freeform spans only (`freeform-spans`) for gold creation and export.

A user should be able to run Label Studio import/export/eval without seeing or choosing legacy scopes, and should get a clear error if they point export at a legacy project/manifests from old runs. Benchmark behavior must stay unchanged and continue scoring `stage_block_predictions.json`.

## Surprises & Discoveries

- Observation: Stage-block prediction generation still needed extracted-archive/normalization helpers that used to live in legacy-named `chunking.py`.
  Evidence: `cookimport/staging/stage_block_predictions.py` imported `build_extracted_archive` and `normalize_display_text` from `cookimport/labelstudio/chunking.py` pre-refactor, so deleting legacy scope code required moving shared helpers to a neutral module.

- Observation: Interactive export still benefits from scope inference even after removing user-facing scope choices.
  Evidence: `_select_export_project(...)` in `cookimport/cli.py` uses `_discover_manifest_project_scopes(...)` / `_infer_scope_from_project_payload(...)` to display a detected type tag in the picker, while runtime export rejects non-freeform scopes.

- Observation: Run-manifest parity assertions tied to legacy task payload locations became brittle after freeform-only task generation.
  Evidence: `tests/staging/test_run_manifest_parity.py` needed to derive line bounds from `extracted_archive.json` when legacy `data.location` fields are unavailable.

## Context and Orientation

In this repository, a Label Studio "scope" means the task payload format and label config used during import/export. Legacy scopes are `pipeline` and `canonical-blocks`. The active workflow is `freeform-spans`, where annotators highlight spans and exports write `freeform_span_labels.jsonl`.

Current end-state after this plan:

- `cookimport/cli.py` Label Studio command contracts are freeform-only (no scope flags/prompts for import/export/eval).
- `cookimport/labelstudio/ingest.py` and `cookimport/labelstudio/export.py` are freeform-only runtime paths.
- `cookimport/labelstudio/export.py` rejects legacy scope manifests/projects/export payloads explicitly.
- Legacy helper modules were removed, with shared archive helpers relocated to `cookimport/labelstudio/archive.py`.

Current benchmark scoring is already stage-block based and should not regress:

- `cookimport/cli.py` (`labelstudio-benchmark`)
- `cookimport/bench/eval_stage_blocks.py`
- `cookimport/staging/stage_block_predictions.py`

Key tests currently covering legacy scope behavior:

- `tests/labelstudio/test_labelstudio_ingest_parallel.py`
- `tests/labelstudio/test_labelstudio_freeform.py`
- `tests/labelstudio/test_labelstudio_benchmark_helpers.py`
- `tests/labelstudio/test_labelstudio_export.py`
- `tests/staging/test_run_manifest_parity.py`


## Plan of Work

Milestone 1 removes legacy scope choices from CLI contracts so users cannot select or export pipeline/canonical modes anymore. `labelstudio-import` becomes explicitly freeform-focused, `labelstudio-export` exports freeform only, and `labelstudio-eval` no longer accepts canonical mode. Interactive menu flows are updated to remove dead prompts and dead option branches.

Milestone 2 simplifies `cookimport/labelstudio/ingest.py` and `cookimport/labelstudio/export.py` to single-mode behavior. This means deleting `task_scope` and `export_scope` branch blocks for legacy scopes, removing obsolete manifest fields that only existed for those branches, and preserving only freeform task IDs (`segment_id`) and artifacts. Legacy manifest/project detection remains only to produce explicit "legacy scope unsupported" messages.

Milestone 3 removes dead helpers and updates imports. Legacy label configs, canonical eval helpers, and pipeline chunk-task generators are removed or reduced to generic utilities only if still needed by freeform/stage-block code. If shared helpers in `chunking.py` are still required by stage-block evidence generation, they should be moved to a scope-neutral helper module so naming no longer implies pipeline-task generation.

Milestone 4 rewrites tests and docs to match the new contract and verifies behavior end-to-end. This includes removing tests that only exercise legacy scope outputs, adding rejection tests for legacy manifests/project types, and updating docs so freeform is the only Label Studio scope documented.

## Validation and Acceptance

Acceptance is complete only when all of the following are true:

1. `cookimport labelstudio-import --help` no longer advertises pipeline/canonical task-scope options or scope-specific options tied to those modes.
2. Interactive Label Studio import/export no longer prompts for scope choice and no longer shows legacy scope options.
3. `cookimport labelstudio-export` exports freeform artifacts only, and legacy-scope projects/manifests fail with an explicit unsupported-scope message.
4. `cookimport labelstudio-eval` runs freeform-only path and has no canonical evaluation branch.
5. `cookimport labelstudio-benchmark` and `cookimport bench run` still evaluate from `stage_block_predictions.json` and pass their relevant tests.
6. Updated test suites pass with no remaining assertions expecting legacy scope behavior.
7. Documentation matches freeform-only scope support and no longer instructs users to choose `pipeline` or `canonical-blocks`.


## Idempotence and Recovery

This migration is code and documentation only; it does not mutate production data roots. Commands and tests are safe to re-run. If a partial edit leaves imports broken, retry by running targeted pytest files after each milestone and cleaning up stale imports immediately.

Legacy run directories remain on disk and are not rewritten. Recovery for a bad refactor is standard git rollback of the code commit; no artifact migration rollback is required.
