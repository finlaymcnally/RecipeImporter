# Preserve EPUB Table Structure For Knowledge Extraction

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this work, table-heavy EPUB books should stop losing their table structure before the knowledge-table pipeline ever sees them. A user should be able to run:

    cookimport stage data/input/thefoodlabCUTDOWN.epub --table-extraction on --write-markdown

and then open `data/output/<timestamp>/tables/thefoodlabcutdown/tables.md` and see non-empty, readable table artifacts for at least the easier reference-table sections. If they also enable pass4 knowledge harvesting, relevant `raw/llm/.../pass4_knowledge/in/*.json` bundles should contain non-null `chunk.blocks[*].table_hint` entries for those table rows.

This plan exists because the older table plan already shipped the downstream half of the feature: `cookimport/parsing/tables.py`, table-aware chunking, `tables.jsonl`, `tables.md`, and pass4 `table_hint` support already exist. The missing piece is upstream preservation. In current EPUB extraction, table rows are often collapsed into plain space-joined text before `tables.py` sees them, so the detector has to guess. Food Lab-style conversion pages prove that this can erase the row and column signals entirely.

## Surprises & Discoveries

- Observation: the shipped table feature is already downstream-complete, so the highest-value fix is not a better writer or chunker.
  Evidence: `docs/04-parsing/04-parsing_readme.md` already documents `cookimport/parsing/tables.py`, table-aware chunking, and pass4 hints as active contracts.

- Observation: the Food Lab benchmark roots that prompted this investigation are not proof of detector failure by themselves because those runs used `table_extraction=off`.
  Evidence: `docs/understandings/2026-03-13_22.40.48-thefoodlab-benchmark-table-expectation.md`.

- Observation: current EPUB row builders join cell text with ordinary spaces, which destroys the most reliable delimiters before `tables.py` tries to infer rows and columns.
  Evidence: `cookimport/parsing/epub_extractors.py` and `cookimport/plugins/epub.py` both build table-row blocks with `text=" ".join(cell_text)`.

- Observation: unstructured already exposes enough HTML to recover deterministic row structure; the missing step was converting `metadata.text_as_html` into per-row blocks and keeping empty cells.
  Evidence: probing `unstructured.partition.html` on Food Lab spine 5 showed `Table` elements whose metadata included full `<table>...</table>` HTML for the conversion charts.

- Observation: row preservation alone did not put Food Lab conversion charts into `tables/` because those sections were still being admitted as recipe candidates.
  Evidence: `data/output/2026-03-13_22.59.32` wrote only 3 unrelated tables; after adding the narrow reference-title recipe penalty, rerun `data/output/2026-03-13_23.01.23` wrote 6 tables including `COMMON INGREDIENTS BY VOLUME AND MASS*`, `COMMON WEIGHT CONVERSIONS`, and `COMMON TEMPERATURE CONVERSIONS`.

## Decision Log

- Decision: treat upstream EPUB structure preservation as the primary fix and flattened-reference salvage as a secondary recovery path.
  Rationale: once table rows are reduced to one giant plain-text line, downstream detection becomes guesswork. Preserving row and cell information earlier is lower-risk and more general.
  Date/Author: 2026-03-13 / Codex

- Decision: keep the existing `ExtractedTable` output contract, chunk semantics, and pass4 evidence rules stable.
  Rationale: the old table rollout already established `tables.jsonl`, `tables.md`, `table_id`, `table_hint`, and verbatim evidence behavior. This plan should improve recall, not redesign those user-facing artifacts.
  Date/Author: 2026-03-13 / Codex

- Decision: support both “structured row metadata” and “deterministic delimiter in block text” for EPUB table rows.
  Rationale: metadata is the durable contract for downstream logic, while a visible delimiter in `block.text` preserves debuggability and gives the existing detector a sane fallback.
  Date/Author: 2026-03-13 / Codex

- Decision: scope flattened-table salvage narrowly to reference-table sections such as conversions, weights, temperatures, doneness, and equivalency charts.
  Rationale: broad free-text salvage would create false positives in narrative prose. Section-gated salvage is easier to audit and easier to disable if it misbehaves.
  Date/Author: 2026-03-13 / Codex

- Decision: validate the work first with `thefoodlabCUTDOWN.epub` rather than only synthetic fixtures.
  Rationale: Food Lab is the motivating failure case and already shows the exact hostile extracted-text shape this plan is meant to repair.
  Date/Author: 2026-03-13 / Codex

- Decision: parse unstructured `Table.metadata.text_as_html` into row blocks instead of trying to reverse-engineer columns from the flattened `Table.text`.
  Rationale: the HTML already contains `<tr>` and `<td>` boundaries plus empty cells, so using it is both more accurate and simpler than salvage-only parsing.
  Date/Author: 2026-03-13 / Codex

- Decision: add a narrow recipe-likeness penalty for obvious reference/conversion section titles.
  Rationale: once EPUB table rows were preserved, Food Lab still failed the user-visible `tables/` check because conversion chapters were being emitted as fake recipes; rejecting those titles keeps them available to the non-recipe table path without redesigning the whole scorer.
  Date/Author: 2026-03-13 / Codex

## Outcomes & Retrospective

Implementation shipped. The core win came from structured EPUB row preservation, not from the flattened-reference fallback: BeautifulSoup row builders now preserve empty cells and emit visible `|` delimiters, while the unstructured path expands `metadata.text_as_html` tables into per-row `epub_table_row` blocks. `cookimport/parsing/tables.py` now consumes that metadata directly, so Food Lab conversion charts no longer depend on guessing from flattened prose.

Real-book validation succeeded with `cookimport stage data/input/thefoodlabCUTDOWN.epub --table-extraction on --write-markdown`. The rerun at `data/output/2026-03-13_23.01.23` wrote 6 tables, including `COMMON INGREDIENTS BY VOLUME AND MASS*`, `COMMON WEIGHT CONVERSIONS`, and `COMMON TEMPERATURE CONVERSIONS`, and the corresponding chunk artifacts carried those `table_ids` into the knowledge lane. The additional recipe-likeness penalty for reference titles was necessary to make those sections land in `nonRecipeBlocks` and therefore under `tables/`.

Remaining gap: pass4 end-to-end bundles were not revalidated against the real-book run because Codex-backed knowledge surfaces were disabled in this environment. Unit coverage still passed for table-hint bundle shaping, but the live `raw/llm/.../pass4_knowledge/in/*.json` proof remains outstanding.

## Context and Orientation

This repository stages cookbook-import artifacts under `data/output/<timestamp>/` for normal runs and under timestamped benchmark roots for `labelstudio-benchmark` runs. The table feature relevant to this plan already exists in downstream parsing and staging. `cookimport/parsing/tables.py` detects row-like non-recipe blocks and emits `ExtractedTable` objects. `cookimport/staging/writer.py` writes `tables/<workbook_slug>/tables.jsonl` and `tables.md`. `cookimport/parsing/chunks.py` keeps table rows intact during chunk splitting and forces table-bearing chunks into the `knowledge` lane. `cookimport/llm/codex_farm_knowledge_jobs.py` carries `table_hint` into pass4 bundles.

The missing upstream layer lives in EPUB extraction. A “table row block” in this repository means one `Block` object whose text comes from one HTML `<tr>` row or an unstructured extractor table element. The current HTML-side EPUB row builders are in `cookimport/parsing/epub_extractors.py` and `cookimport/plugins/epub.py`. They already identify rows and mark them with `features.epub_table_row`, but they join cell text with ordinary spaces, so a row like `["INGREDIENT", "TYPE", "AMOUNT", "WEIGHT"]` becomes `INGREDIENT TYPE AMOUNT WEIGHT`. That is barely acceptable for simple tables and disastrous once a full section collapses into one long line.

`cookimport/parsing/epub_postprocess.py` is the shared cleanup layer for EPUB blocks after extraction. It normalizes text and may split multiline blocks into per-line blocks. This plan must respect that module because it is the shared boundary for EPUB cleanup across extractor backends.

Food Lab exposes two distinct failure modes that this plan must handle. The first mode is structured HTML table rows that survive extraction but lose their cell boundaries because the row builder joins with spaces. The second mode is already-flattened “whole reference section as one line” content, such as `COMMON INGREDIENTS BY VOLUME AND MASS*` followed by one giant block containing headers and every row. The first mode should be fixed by preserving structure in extraction. The second mode needs a narrower salvage path in `cookimport/parsing/tables.py`, guarded by heading context and confidence notes.

This plan builds on the older checked-in plan `docs/plans/old/data-table-knowledge.md`, but it does not assume the reader has memorized that file. The older plan delivered downstream table outputs and pass4 hints. This plan is specifically about improving upstream EPUB recall without breaking the already-shipped output contracts.

## Plan of Work

### Milestone 1: Preserve EPUB Row Structure At Extraction Time

At the end of this milestone, EPUB table rows should carry stable structural information into the normal block stream instead of discarding it immediately. A contributor should be able to inspect extracted blocks and see not only `features.epub_table_row`, but also the original cell texts and enough metadata to tell whether a row is a header row or a body row.

Edit the BeautifulSoup-side row builders in `cookimport/parsing/epub_extractors.py` and `cookimport/plugins/epub.py`. When `_build_table_row_block(...)` collects `cell_text`, preserve that array on the block as something explicit such as `features.epub_table_cells`, `features.epub_table_column_count`, and `features.epub_table_row_kind` when a row contains `<th>` cells or otherwise looks header-like. Preserve `spine_index` and source location features as they already exist. Also change the visible `block.text` to a deterministic delimiter form such as `cell1 | cell2 | cell3` or tab-delimited text. The key requirement is that the same input row always yields the same delimiter and the same cell array. Do not change non-table blocks.

Check the unstructured path too. If unstructured EPUB extraction yields `BlockType.TABLE` blocks that already represent one row, preserve cell splits there when possible. If unstructured does not expose true cells, still add a stable `table_row_source="unstructured"` marker and avoid throwing away any delimiter it preserves naturally. The plan does not require perfect cell recovery from unstructured output, but it does require stopping unnecessary loss where the repo already knows it is dealing with a table row.

Acceptance is concrete. A synthetic EPUB fixture with `<table><tr><td>1 cup</td><td>sugar</td></tr></table>` should still produce a row whose text is visibly delimited and whose features include the original cell array. Existing ingredient-like behavior from those rows must remain intact so this does not break recipe or knowledge heuristics that already benefit from EPUB table rows.

### Milestone 2: Add A Structured-Row Fast Path In `tables.py`

At the end of this milestone, `cookimport/parsing/tables.py` should reconstruct tables from preserved EPUB row metadata without needing to re-derive columns from ambiguous free text. A contributor should be able to hand `detect_tables_from_non_recipe_blocks(...)` a list of non-recipe blocks with `epub_table_cells` features and get back a stable `ExtractedTable` even if the human-readable `text` would have been hard to parse.

Extend `cookimport/parsing/tables.py` so `_find_candidate_rows(...)` or an adjacent helper recognizes structured EPUB row metadata first. When a block contains a usable cell array, turn that into a candidate row directly, carrying through block index, row kind, column count, and parse mode such as `epub_structured`. Keep the existing pipe/tab/multispace parsing path for PDF, flattened text, and any other source that does not provide structured features.

While doing this, keep the current output contract stable. `ExtractedTable` should still emit `table_id`, `caption`, `headers`, `rows`, `row_texts`, `markdown`, `confidence`, and `notes`. The only additive changes allowed are notes or metadata that help explain why a table was detected. If header rows are available from EPUB metadata, trust that first. If not, keep the current header heuristics.

Acceptance is that a test using EPUB-style structured blocks produces the expected headers, rows, and `row_texts` even if there are no pipes, tabs, or repeated spaces in `block.text`. Existing text-only tests must still pass.

### Milestone 3: Add Narrow Salvage For Already-Flattened Reference Sections

At the end of this milestone, `tables.py` should recover at least some reference-table content from sections that were already flattened before extraction reached the normal row path. This is explicitly a fallback path, not the main design. It exists because Food Lab and similar books may still contain one-block conversion or equivalency sections even after Milestone 1.

Add a helper in `cookimport/parsing/tables.py` that looks for heading-plus-body shapes characteristic of reference tables. The intended targets are sections with headings like `CONVERSIONS`, `VOLUME EQUIVALENCIES`, `WEIGHT CONVERSIONS`, `TEMPERATURE CONVERSIONS`, `COMMON ... CONVERSIONS`, or similar durable reference headings. The body blocks under those headings often contain repeated units, measures, temperatures, or equivalency markers such as `=`. That makes them suitable for deterministic but narrow salvage rules.

The salvage path should be conservative. Do not attempt arbitrary natural-language table reconstruction. Instead, split only when the text contains repeatable conversion-like patterns or row families. A section consisting of several consecutive short lines such as `3 teaspoons = 1 tablespoon` and `2 tablespoons = 1 fluid ounce` can be emitted as a two-column table or a one-row-per-fact reference table. A fully collapsed giant line like Food Lab’s common ingredient mass chart may need a separate pattern family that looks for repeated amount-weight groups after a heading. Any such result must carry a lower confidence and notes such as `salvaged_from_flattened_reference_section`.

Acceptance is that the salvage path recovers at least one Food Lab conversion/equivalency table into `ExtractedTable` output where the current detector would return nothing, while narrative prose sections with ordinary sentences do not start producing false tables.

### Milestone 4: Thread Validation Through Stage And Benchmark Flows

At the end of this milestone, there should be one deterministic stage workflow and one benchmark workflow that prove the improvement is real in a book-sized run, not only in unit tests.

The normal stage proof uses `data/input/thefoodlabCUTDOWN.epub`. Run stage with `--table-extraction on` and inspect `tables/thefoodlabcutdown/tables.jsonl` plus `tables.md`. The benchmark proof reruns Food Lab through `cookimport labelstudio-benchmark run` with `--table-extraction on` so the prediction root contains table artifacts and pass4 input bundles with non-null `table_hint` where applicable.

Because benchmark roots historically did not enable table extraction, update the relevant docs and any benchmark contract helpers or defaults that should preserve this setting. The plan does not require making `table_extraction=on` the universal default for every benchmark. It does require that when a user turns it on explicitly, benchmark manifests, prediction runs, and pass4 inputs carry it through consistently and visibly.

Acceptance is behavioral. The saved run manifests must show `table_extraction: on`, `tables.md` must exist, and at least one pass4 input bundle should contain a non-null `table_hint` for a recovered table block in the Food Lab rerun.

### Milestone 5: Finish Docs, Tests, And Explicit Limits

At the end of this milestone, the implementation should be understandable to the next contributor without redoing this investigation. Update `docs/03-ingestion/03-ingestion_readme.md` or `docs/04-parsing/04-parsing_readme.md` to explain that EPUB table rows now preserve cell structure and that `tables.py` has both structured-row and flattened-reference fallback paths. Update `docs/05-staging/05-staging_readme.md` only if output behavior changes in a user-visible way. Update `docs/10-llm/knowledge_harvest.md` if the better table recovery meaningfully changes pass4 expectations.

Write or update a short note in `docs/understandings` capturing what Food Lab proved after implementation: which tables now survive, which ones still collapse, and whether the salvage path is worth keeping. That note is part of the deliverable because this feature has already had one “it shipped, but did it really work on real books?” loop.

Acceptance is that a novice can read the updated docs, rerun the commands in this plan, and understand both the success case and the remaining limits.

## Validation and Acceptance

Validation must prove four different things.

First, EPUB table rows must retain structure at extraction time. A synthetic EPUB table fixture should produce row blocks whose features preserve the original cell array and whose visible text contains a deterministic delimiter. Existing quantity-like behavior for table rows must still work, because some downstream heuristics already benefit from that shape.

Second, `cookimport/parsing/tables.py` must reconstruct tables from structured row metadata even when `block.text` alone would be ambiguous. The proof is focused tests that fail before the change and pass after it, plus `tables.jsonl` entries whose headers and rows match the structured input.

Third, the fallback salvage path must recover at least one reference-table family from Food Lab or a targeted synthetic fixture without broadly hallucinating tables from prose. The proof is one recovered table artifact plus negative tests on ordinary narrative sections.

Fourth, end-to-end stage and benchmark runs must show the user-visible behavior. Acceptance means:

- `cookimport stage data/input/thefoodlabCUTDOWN.epub --table-extraction on --write-markdown` writes `tables/thefoodlabcutdown/tables.md` and `tables.jsonl`.
- `tables.md` is non-empty and contains at least one readable table or reference-row rendering that was previously absent.
- A stage or benchmark run with pass4 enabled writes at least one `pass4_knowledge/in/*.json` bundle containing non-null `table_hint`.
- Benchmark manifests for explicit reruns show `table_extraction: on` so the evidence root is auditable.

The plan does not require perfect recovery of every Food Lab table. It does require a measurable before-and-after improvement and written documentation of the remaining misses.

## Idempotence and Recovery

The implementation should be additive and safe to rerun. Structured EPUB metadata is additive to existing blocks, not a destructive schema rewrite. If the structured-row fast path in `tables.py` misbehaves, it can be disabled or ignored while leaving the older pipe/tab/multispace path intact. That is the intended recovery posture for Milestones 1 and 2.

The flattened-reference salvage path must be clearly marked by lower confidence and notes so it can be audited or rolled back without destabilizing the upstream-preservation fix. If salvage heuristics prove noisy, the recovery path is to keep the structured-row fast path and remove or narrow the salvage logic rather than backing out the upstream-preservation work.

Every stage and benchmark validation run must write a fresh timestamped root. Do not overwrite historical benchmark roots from March 6. If a rerun produces no tables, inspect the saved manifests and extracted text first before changing code again; “no tables” may still mean the source lost structure before the detector saw it.
