# Fix Rejected Pseudo-Recipe Label Leak

## Problem

Single-book benchmark runs can fail when label-first grouping rejects a pseudo-recipe span but leaves its block labels as recipe-local values such as `RECIPE_TITLE`.
Stage 7 then sees that block as non-recipe and aborts because non-recipe routing only accepts `knowledge` or `other`.

## Acceptance Criteria

- Rejected pseudo-recipe spans no longer leave recipe-local authoritative block labels on non-recipe blocks.
- A focused grouping regression test proves rejected title-only spans demote downstream authoritative block labels.
- A stage-session regression test exercises the live label-first to Stage 7 path and proves the run no longer crashes on a title-only pseudo-recipe heading.

## Verification

- `. .venv/bin/activate && pytest tests/parsing/test_recipe_span_grouping.py -q`
- `. .venv/bin/activate && pytest tests/staging/test_import_session.py -q`

## Evidence

- Added a grouping-level regression that fails when rejected pseudo-recipe blocks keep `RECIPE_TITLE`.
- Added a stage-session regression that reproduces the Stage 7 crash path with a synthetic heading sequence.
- Patched recipe-span grouping to demote rejected pseudo-recipe block labels to downstream-safe `OTHER` authority.

## Constraints / Decision

Keep Stage 7 strict.
The fix belongs at the recipe-span grouping authority seam because that is where recipe candidates are accepted or rejected, and downstream staging should only see post-grouping authoritative block labels.

## Rollback

Revert the grouping normalization helper, the two regression tests, and this task file together.
