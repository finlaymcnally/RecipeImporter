# Task: QualitySuite WSL Safety Guard

## Problem statement

`bench quality-run` could stack parallelism layers on WSL (`parallel experiments` x `all-method inflight` x per-config worker counts), leading to host instability and repeated remote disconnects.

## Acceptance criteria

- Parallel quality-run on WSL prefers subprocess experiment isolation in auto mode.
- WSL runs with parallel experiments apply a conservative nested-parallelism safety guard.
- Guard status and effective cap values are visible in `experiments_resolved.json`.
- Existing non-WSL behavior remains unchanged.
- New/updated tests cover executor selection and guard application on WSL.

## Verification steps

Run from repo root with venv active:

```bash
. .venv/bin/activate
pytest tests/bench/test_quality_suite_runner.py -q
pytest tests/bench/test_bench.py -k "quality_run" -q
```

## Evidence

- `tests/bench/test_quality_suite_runner.py` passed with new WSL-specific tests:
  - `test_run_quality_suite_prefers_subprocess_executor_for_parallel_wsl_runs`
  - `test_run_quality_suite_applies_wsl_safety_guard_to_nested_parallelism`
- `tests/bench/test_bench.py -k "quality_run"` passed.

## Constraints / gotchas and decision

- Constraint: keep non-WSL contract stable to avoid changing existing Linux/macOS parallel throughput defaults.
- Decision: apply the safety guard only when WSL is detected and experiment fanout is >1, with explicit env opt-out (`COOKIMPORT_QUALITY_WSL_DISABLE_SAFETY_GUARD=1`) for deliberate tuning.

## Rollback / compensation plan

- If this guard is too conservative for a specific run, set `COOKIMPORT_QUALITY_WSL_DISABLE_SAFETY_GUARD=1` and rerun.
- Full rollback is a clean revert of this task’s code/doc diff.
