## Purpose / Big Picture


After this change, the repo has a deterministic, fast-to-run quality lane that mirrors `bench speed-*` ergonomics but measures benchmark quality instead of runtime. A user (or external agent script) can discover a stable representative suite, run one or more all-method experiments over that same suite, and compare baseline vs candidate with explicit pass/fail regression gates.

You can see this working by running:

    cookimport bench quality-discover ...
    cookimport bench quality-run ...
    cookimport bench quality-compare ...

and observing reproducible target selection, per-experiment quality summaries with run-settings hashes, and a comparison verdict that fails on quality regressions or run-settings mismatch (unless explicitly overridden).

## Surprises & Discoveries


- Observation: multi-source all-method reports already provide enough counters for quality orchestration, but not one ready-made practical metric aggregate.
  Evidence: `_run_all_method_benchmark_multi_source(...)` writes `sources[*].winner_metrics` (strict precision/recall/f1) while practical metrics are available inside each per-source `all_method_benchmark_report.json` `winner_by_f1` payload.

- Observation: `RunSettings.from_dict(...)` ignores unknown keys by design, which is unsafe for experiment-patch payloads because typoed keys would silently no-op.
  Evidence: `cookimport/config/run_settings.py:RunSettings.from_dict` logs unknown keys and continues with `extra="ignore"` model behavior.

- Observation: source sharding can split one source into multiple source rows, which can distort naive per-row quality aggregation.
  Evidence: `_run_all_method_benchmark_multi_source(...)` emits `source_shard_index`/`source_shard_total`; one source can appear multiple times in `sources`.

- Observation: current multi-source payloads are already collapsed to one row per `source_group_key`, but still carry shard-level report paths (`report_json_paths`) needed to recover practical winner metrics consistently.
  Evidence: `_aggregate_source_rows(...)` in `cookimport/cli.py` returns one source row plus optional `source_shards`/`report_json_paths`; top-level `winner_metrics` remains strict-only.

- Observation: codex-farm import parsing remains explicitly policy-locked and must stay locked in quality flows too.
  Evidence: `_resolve_all_method_codex_choice(...)` forces effective false and project policy requires no AI parsing/cleaning import path.


## Decision Log


- Decision: QualitySuite will use a deterministic representative selector with an explicit algorithm version recorded in suite artifacts.
  Rationale: deterministic selection is the core trust contract for baseline/candidate quality comparisons.
  Date/Author: 2026-02-27 / Codex GPT-5

- Decision: Experiment patch payloads will be strict-validated (unknown keys cause command failure) before `RunSettings.from_dict(...)` normalization.
  Rationale: silent no-op patches produce misleading experiments and false confidence.
  Date/Author: 2026-02-27 / Codex GPT-5

- Decision: Quality aggregation will be per source-group key (not per row) and will use winner metrics from per-source reports to include strict and practical metrics.
  Rationale: row-level aggregation can double-count sharded sources and multi-source top-level rows do not include practical aggregates.
  Date/Author: 2026-02-27 / Codex GPT-5

- Decision: `quality-run` executes experiments sequentially and continues after per-experiment failure, marking failed experiments in summary output.
  Rationale: deterministic output order and resilient long-running loops are more important than parallel run throughput here.
  Date/Author: 2026-02-27 / Codex GPT-5

- Decision: `quality-compare` compares explicitly selected experiment IDs and fails by default when run-settings hashes differ.
  Rationale: avoid ambiguous “first row wins” behavior and prevent invalid comparisons under settings drift.
  Date/Author: 2026-02-27 / Codex GPT-5

- Decision: `quality-run` marks experiment status `incomplete` when source/config coverage is partial (for example failed source groups), while command-level exceptions remain `failed`.
  Rationale: compare gating needs a deterministic way to reject partial-quality runs without conflating orchestration failures and partial benchmark completion.
  Date/Author: 2026-02-27 / Codex GPT-5

- Decision: `quality-compare` treats missing gate metrics as regression-fail reasons instead of silently skipping that gate.
  Rationale: missing strict/practical/source coverage metrics should not allow accidental PASS verdicts.
  Date/Author: 2026-02-27 / Codex GPT-5

- Decision: Codex-farm permutations stay disabled in QualitySuite and any experiment request that attempts to enable recipe AI parsing must not take effect.
  Rationale: required by project policy.
  Date/Author: 2026-02-27 / Codex GPT-5

## Context and Orientation


QualitySuite is mostly orchestration over existing contracts.

Target discovery and source-hint fallback already exist in `cookimport/bench/speed_suite.py` (`match_gold_exports_to_inputs(...)`).

All-method runtime/scoring is in `cookimport/cli.py` via:
- `_resolve_all_method_codex_choice(...)`
- `_resolve_all_method_markdown_extractors_choice(...)`
- `_build_all_method_target_variants(...)`
- `_run_all_method_benchmark_multi_source(...)`

Multi-source reports are written to `all_method_benchmark_multi_source_report.json` and per-source reports under per-source run roots as `all_method_benchmark_report.json`.

Terms used in this plan:

- Representative suite: a deterministic subset of matched `(source_file, gold_spans_path)` targets intended to stay small but coverage-preserving.
- Experiment: one named `RunSettings` payload derived from a shared base settings payload plus a patch.
- Source-group aggregate: one quality row per source logical identity (`source_group_key`), even if scheduler sharding produces multiple source rows.
- Settings parity: baseline and candidate selected experiments have matching `run_settings_hash` values.


## Plan of Work


Milestone 1 adds deterministic quality-suite discovery models in `cookimport/bench/quality_suite.py`. The module reuses `match_gold_exports_to_inputs(...)`, computes lightweight descriptors for each matched target, and emits a manifest containing all matched targets, unmatched diagnostics, and selected representative IDs.

Representative-selection algorithm (must be encoded exactly and versioned in suite payload):

1. Compute descriptor per matched target:
   - `canonical_text_chars`: byte size of sibling `canonical_text.txt` when present, otherwise `0`.
   - `gold_span_rows`: JSONL line count of `freeform_span_labels.jsonl`.
   - `label_count`: unique label count from parsed gold rows (missing/blank labels ignored).
2. Build deterministic strata using two bucket dimensions:
   - size bucket from `canonical_text_chars` terciles (`small`, `medium`, `large`),
   - label-density bucket from `label_count` terciles (`sparse`, `medium`, `dense`).
3. Within each stratum, sort by `target_id` ascending.
4. Select targets with round-robin traversal over sorted stratum keys until `max_targets` reached.
5. If `max_targets` is null or `>= matched_count`, select all matched targets.

The suite JSON must include `selection.algorithm_version`, `selection.seed`, `selection.max_targets`, `selection.strata_counts`, and `selected_target_ids` so reruns are reproducible and explainable.

Milestone 2 adds experiment execution in `cookimport/bench/quality_runner.py`.

`quality-run` accepts a suite and an experiment-definition file. Experiment definitions are machine-editable JSON and must be strict validated:
- top-level unknown keys fail,
- experiment unknown keys fail,
- `id` must be unique and slug-safe,
- `run_settings_patch` must be object,
- patch keys must be known `RunSettings` fields.

Experiment settings resolution flow:

1. Resolve base settings payload from `--base-run-settings-file` (if provided) else `cookimport.json`.
2. For each experiment, merge `run_settings_patch` over base payload.
3. Validate merged payload with `RunSettings.from_dict(...)`.
4. Record `run_settings`, `run_settings_summary`, `run_settings_hash` per experiment.
5. Force codex choice through `_resolve_all_method_codex_choice(False)` contract (effective false).

Execution flow per experiment:

1. Load selected suite targets only.
2. Convert suite targets to `AllMethodTarget` rows.
3. Build variants via `_build_all_method_target_variants(...)` using experiment `RunSettings`.
4. Run `_run_all_method_benchmark_multi_source(...)` into `runs/<timestamp>/experiments/<experiment_id>/`.
5. Parse experiment report and per-source report JSON files.
6. Build one normalized experiment summary row with:
   - strict macro metrics (`strict_precision_macro`, `strict_recall_macro`, `strict_f1_macro`),
   - practical macro metrics (`practical_precision_macro`, `practical_recall_macro`, `practical_f1_macro`),
   - source success coverage (`sources_planned`, `sources_successful`, `source_success_rate`),
   - config counters (`configs_planned`, `configs_completed`, `configs_successful`),
   - eval dedupe counters (`evaluation_signatures_unique`, `evaluation_runs_executed`, reuse counters),
   - status and error fields.

Aggregation rule must collapse by `source_group_key` so sharded rows do not double count. When multiple shard rows exist for one source group, use the shard row with highest strict F1 winner as that source-group representative and include shard count diagnostics in summary.

Milestone 3 adds comparison in `cookimport/bench/quality_compare.py`.

`quality-compare` selects one baseline experiment and one candidate experiment from two run directories using explicit CLI options:
- `--baseline-experiment-id`
- `--candidate-experiment-id`

If omitted, resolver behavior is deterministic:
- use `baseline`/`candidate` IDs when present,
- otherwise require exactly one successful experiment in each run,
- otherwise fail with actionable error.

Comparison payload includes:
- settings parity fields (`baseline_run_settings_hash`, `candidate_run_settings_hash`, `settings_match`, `allow_settings_mismatch`),
- metric deltas for strict/practical macro precision/recall/f1,
- source coverage delta,
- verdict and regression reasons.

Default regression gates (configurable via CLI options and JSON payload):
- `strict_f1_drop_max = 0.005`
- `practical_f1_drop_max = 0.005`
- `source_success_rate_drop_max = 0.0`

Verdict logic:
- fail when selected experiment row is failed/incomplete,
- fail when settings mismatch and `allow_settings_mismatch` is false,
- fail when any gated metric drop exceeds threshold,
- otherwise pass.

## Validation and Acceptance


Acceptance is behavior-first.

- `quality-discover` is deterministic for identical inputs and writes a suite payload that fully explains selection (`algorithm_version`, strata counts, selected IDs).
- `quality-run` writes one timestamped run root with `summary.json`, `report.md`, and per-experiment subfolders.
- Each experiment summary row includes strict/practical macro metrics, source coverage, config counters, run-settings hash, and status.
- Failed experiments do not stop later experiments; failures are explicit in summary/report output.
- `quality-compare` supports explicit experiment-ID selection, enforces settings parity by default, and reports threshold deltas/reasons for FAIL.
- Codex-farm parsing remains locked off in all quality-suite paths.
- Focused tests for discovery/runner/compare/CLI wiring pass.


## Idempotence and Recovery


All outputs are additive and timestamped under `data/golden/bench/quality/`.

Suite discovery is safe to rerun with same destination path; rerun replaces only that suite file with deterministic content.

Runner recovery behavior:
- if one experiment fails, the run still completes and preserves successful experiment outputs,
- rerunning with same suite/experiments creates a new run timestamp and does not overwrite previous runs.

Compare is read-only over prior run artifacts. If compare fails due settings mismatch, rerun candidate with matching settings or pass explicit mismatch override.
