## Problem

Recipe runtime can now skip obviously terminal fragmentary / not-a-recipe tasks before the LLM queue.
That saves tokens, but the high-level run reporting does not clearly say how many tasks were handled locally or which recipes were skipped.

## Acceptance Criteria

- Recipe promotion reporting includes a clear `handled_locally_skip_llm` section with a count and per-recipe rows.
- `recipe_manifest.json` mirrors the topline local-skip count in existing recipe counts.
- `recipe_stage_summary.json` surfaces the local-skip count in its high-level rollups.

## Verification

- `source .venv/bin/activate && pytest tests/llm/test_recipe_phase_workers.py -k fragmentary -q`
- `source .venv/bin/activate && pytest tests/staging/test_stage_observability.py -k recipe_stage_summary_reports_same_session_fix_rollups -q`

## Evidence

- Promotion reports now carry `handled_locally_skip_llm.count`, status counts, and per-recipe rows keyed by recipe/task/shard.
- Recipe manifest counts now include `recipe_correction_handled_locally_skip_llm`.
- Recipe stage summary followups/context now expose `handled_locally_skip_llm_count`.

## Constraints / Gotchas

- Keep the reporting change additive; do not change promotion or staging authority semantics.
- Reuse the existing promotion report / manifest / stage summary surfaces instead of inventing a new top-level report file.

## Decision

- Derive the reporting rows from validated proposal metadata, specifically the existing `deterministic_terminal_scaffold` reason recorded per task, so the audit surface matches the real runtime decision point.
