Problem

Compare & Control treated `run_timestamp` as categorical text, so choosing time on the compare axis collapsed runs into grouped averages instead of showing one point per run.

Acceptance Criteria

- Compare & Control treats `run_timestamp` as a time axis for numeric charting.
- Raw Compare & Control charts can plot one point per run when compare=`run_timestamp` and outcome is numeric like `all_token_use`.
- The dashboard chart uses a datetime x-axis and keeps the human-readable timestamp in tooltips.

Verification

- `. .venv/bin/activate && pytest tests/analytics/test_compare_control_engine.py tests/analytics/test_stats_dashboard.py -q`

Evidence

- Added a focused engine test proving the field catalog marks `run_timestamp` as numeric/time-like.
- Added a dashboard harness test proving Compare & Control renders a datetime scatter with one point per run for timestamp-vs-token selections.

Constraints / Decision

- Keep the change local to Compare & Control field normalization and chart rendering.
- Use the existing cached-input-discounted `all_token_use` metric rather than inventing a new token metric.

Rollback / Compensation

- Revert this task's code changes to restore the old categorical timestamp behavior.
