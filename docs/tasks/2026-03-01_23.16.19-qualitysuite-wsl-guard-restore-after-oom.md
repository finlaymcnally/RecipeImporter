# Task: Restore WSL guard after OOM evidence

## Problem statement

WSL kept crashing during QualitySuite runs even after disk cleanup. Kernel OOM logs in the crash window showed runaway `cookimport` process fanout exhausting RAM and swap while `experiments_resolved.json` reported `wsl_safety_guard_reason=retired_unhobble`.

## Acceptance criteria

- WSL runs apply a real nested-parallelism guard by default.
- Guard caps run-settings worker fanout and all-method runtime fanout before experiment execution.
- `experiments_resolved.json` guard telemetry reflects applied/disabled status.
- `COOKIMPORT_QUALITY_WSL_DISABLE_SAFETY_GUARD=1` still bypasses the guard.
- WSL-focused runner tests cover guarded behavior and env opt-out.

## Verification steps

Run from repo root with venv active:

```bash
. .venv/bin/activate
pytest tests/bench/test_quality_suite_runner.py -k "wsl_safety_guard" -q
```

## Evidence

- `cookimport/bench/quality_runner.py` now rewrites WSL experiment payload/runtime fanout to conservative caps.
- `tests/bench/test_quality_suite_runner.py` now asserts guard application for parallel and single-slot WSL runs.
- Added explicit env-opt-out coverage (`disabled_by_env`).
- Updated `docs/07-bench/07-bench_README.md` and `docs/07-bench/07-bench_log.md` to match runtime behavior.

## Constraints / gotchas and decision

- Constraint: keep non-WSL behavior unchanged.
- Constraint: retain opt-out escape hatch for deliberate high-throughput runs.
- Decision: enforce conservative WSL caps in one place (`_apply_wsl_quality_safety_guard`) and keep telemetry explicit so operators can verify what happened.

## Rollback / compensation plan

- Temporary bypass: set `COOKIMPORT_QUALITY_WSL_DISABLE_SAFETY_GUARD=1`.
- Full rollback: revert this task’s diff in `quality_runner` + WSL test expectations/docs.
