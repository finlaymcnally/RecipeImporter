Problem statement

The first CLI refactor pass moved public command registration into `cookimport/cli_commands/`, but `cookimport/cli.py` still physically contains dead legacy command bodies and the command modules still import the old CLI root wholesale. That leaves the ownership story misleading even though the active Typer tree already delegates.

Acceptance criteria

- `cookimport/cli.py` is reduced to the public composition root plus compatibility exports.
- Active command-family modules no longer import `cookimport.cli` wholesale for implementation ownership.
- Direct `cookimport.cli.<command>` call sites used by tests/helpers still work.
- The affected ExecPlans and short ownership docs reflect the final state.

Verification steps

- `source .venv/bin/activate && pytest tests/bench/test_bench_quality_cli.py tests/analytics/test_benchmark_csv_backfill_cli.py tests/cli/test_stage_progress_dashboard.py`
- `source .venv/bin/activate && pytest tests/bench/test_bench_speed_cli.py tests/bench/test_bench.py`
- `npm run docs:list`

Evidence

- The targeted pytest slices should pass after the purge.
- `cookimport/cli.py` should no longer contain decorated command bodies such as `@app.command()` or `@bench_app.command(...)`.

Constraints/gotchas + brief decision

- Many tests still patch helper names on `cookimport.cli`, so the public root must keep compatibility exports even after ownership moves out.
- The chosen approach is to keep `cookimport/cli.py` as the composition root, move the shared helper surface behind `cookimport/cli_support.py`, and sync command-module globals from the public root so old monkeypatch seams still work.

Rollback / compensation plan

- If the purge breaks direct-call helper tests, restore the previous `cookimport/cli.py` and keep the compatibility wrapper approach while re-scoping helper extraction more narrowly.
