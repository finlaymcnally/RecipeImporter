# Remove runtime `epub_extractor=auto`

## Problem statement

Runtime extractor mode `auto` makes behavior less explicit and harder to reason about. The project is moving to explicit extractor choices only.

## Acceptance criteria

- Stage and prediction-generation validators reject `epub_extractor=auto`.
- Run settings expose explicit extractor choices only.
- All-method benchmark permutations no longer include `auto`.
- Legacy saved run settings containing `epub_extractor=auto` are migrated safely.

## Verification steps

- `source .venv/bin/activate && pytest tests/llm/test_run_settings.py tests/labelstudio/test_labelstudio_ingest_parallel.py tests/labelstudio/test_labelstudio_benchmark_helpers.py tests/cli/test_cli_output_structure.py`
- `source .venv/bin/activate && cookimport stage --help` (confirm extractor help shows explicit choices only).

## Evidence

- Focused suite result: `86 passed, 7 warnings`.
- `stage --help` shows extractor choices: `unstructured`, `legacy`, `markdown`, `markitdown` (no `auto`).

## Constraints / gotchas

- Keep historical analytics/report fields (`epub_extractor_requested`, `epub_extractor_effective`) for compatibility.
- Do not remove debug extractor race internals (`select_epub_extractor_auto`) from debug-only workflows.

## Decision

Remove `auto` from runtime wiring while keeping debug scorer utilities and adding run-settings migration (`auto` -> `unstructured`) for old stored payloads.

## Rollback / compensation plan

Revert this change set if needed; no schema migration is required because run-config keys are unchanged.
