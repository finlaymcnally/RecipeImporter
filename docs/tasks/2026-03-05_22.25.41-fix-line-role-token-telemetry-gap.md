# Fix Missing Token Telemetry for Line-Role and Benchmark History

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds. This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The goal of this change is to make token accounting trustworthy for runs that use the canonical line-role Codex pipeline. Right now, a benchmark run can clearly use `codex-line-role-v1` and still leave `tokens_input`, `tokens_cached_input`, `tokens_output`, `tokens_reasoning`, and `tokens_total` blank in `.history/performance_history.csv`. That is the exact failure mode that made the March 4 QualitySuite incident hard to quantify after the fact.

After this change, any run that uses line-role Codex must leave behind durable token telemetry even when recipe Codex is off. A user should be able to inspect one prediction run directory and see a dedicated line-role telemetry artifact, inspect the prediction or eval manifest and see that artifact referenced, and inspect the history CSV or dashboard collector output and see nonblank token totals for the same run. Failed and retried line-role attempts must count toward this telemetry so the stored numbers track real usage rather than only successful final responses.

## Surprises & Discoveries

- Observation: The low-level Codex wrapper already knows how to extract usage when JSON events are enabled.
  Evidence: `cookimport/llm/codex_exec.py` has `argv_with_json_events(...)`, `normalize_usage(...)`, and `extract_response_and_usage(...)`, and `run_codex_json_prompt()` can return `payload["usage"]` when `track_usage=True`.

- Observation: The line-role pipeline explicitly disables usage tracking today.
  Evidence: In `cookimport/parsing/canonical_line_roles.py`, `_run_codex_prompt_with_retry()` calls `run_codex_json_prompt(..., track_usage=False, ...)`.

- Observation: The benchmark and analytics layers already know how to carry token values once they exist in a manifest.
  Evidence: `cookimport/cli.py` passes `PredRunContext.tokens_*` into `append_benchmark_csv(...)`, and both `cookimport/analytics/perf_report.py` and `cookimport/analytics/dashboard_collect.py` already backfill token fields from prediction manifests when telemetry is present.

- Observation: The line-role artifact path is already stable enough to host a telemetry file without inventing a new directory layout.
  Evidence: Line-role runs already write `prediction-run/line-role-pipeline/line_role_predictions.jsonl`, `freeform_span_predictions.jsonl`, `stage_block_predictions.json`, and `extracted_archive.json`.

- Observation: Retry accounting must be included in the fix, otherwise the stored telemetry will still undercount real usage.
  Evidence: `_run_codex_prompt_with_retry()` can make up to three attempts, but current prompt artifacts only preserve final prompt/response files and a dedup list, not a durable per-attempt usage summary.

- Observation: The existing analytics and CLI token readers were already structured for additive merges, so the line-role fix only needed a sibling loader plus a shared “sum when present” rule.
  Evidence: After adding `line_role_pipeline_telemetry_path` readers in `cookimport/cli.py`, `cookimport/analytics/perf_report.py`, and `cookimport/analytics/dashboard_collect.py`, the downstream CSV/dashboard behavior changed without any history schema edits.

## Decision Log

- Decision: Solve this as a line-role telemetry pipeline problem, not as a generic billing estimator project.
  Rationale: The repository already has prompt files and analytics fallbacks. The missing piece is durable first-party telemetry at the point where line-role Codex calls are made.
  Date/Author: 2026-03-05 / Codex

- Decision: Add a dedicated line-role telemetry artifact instead of overloading `llm_codex_farm` or forging a fake CodexFarm manifest shape.
  Rationale: The line-role pipeline is a different execution model from CodexFarm. Reusing the CodexFarm schema would blur concepts and make future debugging harder. A dedicated artifact with an explicit manifest key is the simpler and more truthful fix.
  Date/Author: 2026-03-05 / Codex

- Decision: Count every line-role attempt, including failed or incomplete retries, in the telemetry summary.
  Rationale: The user-visible bug is incomplete cost visibility. A summary that only records successful final attempts would keep understating real usage.
  Date/Author: 2026-03-05 / Codex

- Decision: Keep the history CSV totals as combined run totals, but preserve component-level detail in the new line-role telemetry artifact.
  Rationale: The history CSV already acts as a coarse trend surface. The artifact can hold the richer line-role-only detail without forcing a schema explosion in the main CSV.
  Date/Author: 2026-03-05 / Codex

- Decision: Keep this change bounded and do not attempt a broad refactor to deduplicate all token-helper logic between `cli.py`, `perf_report.py`, and `dashboard_collect.py`.
  Rationale: The repository already has some duplicated CodexFarm token extraction helpers. For this fix, correctness and a safe landing are more important than a large cleanup.
  Date/Author: 2026-03-05 / Codex

- Decision: Store line-role telemetry under the existing `line-role-pipeline/` artifact directory and let projection/manifest writers publish the path opportunistically when the file exists.
  Rationale: This keeps the new artifact colocated with the rest of the line-role outputs and preserves backward compatibility for runs that predate the telemetry file.
  Date/Author: 2026-03-05 / Codex

## Context and Orientation

This repository has three distinct layers involved in benchmark token telemetry.

The first layer is the low-level Codex wrapper in `cookimport/llm/codex_exec.py`. This is the function that actually invokes the `codex` CLI. When called with `track_usage=True`, it adds `--json`, parses the emitted JSON event stream, and returns a `usage` dictionary containing input, cached input, output, and reasoning tokens. This part already exists.

The second layer is the caller-specific pipeline code. CodexFarm already persists rich telemetry in a manifest shape that the rest of the repository understands. The canonical line-role pipeline lives in `cookimport/parsing/canonical_line_roles.py`. It writes prompt files, parsed predictions, cache entries, and line-role projection artifacts, but it currently throws away usage totals because `_run_codex_prompt_with_retry()` disables tracking and `_CodexBatchResult` does not carry any usage metadata.

The third layer is the benchmark and analytics surface. `cookimport/labelstudio/ingest.py` writes prediction-run `manifest.json` and stage/eval `run_manifest.json`. `cookimport/cli.py:_load_pred_run_recipe_context()` reads the prediction run manifest and builds a `PredRunContext` that later feeds `append_benchmark_csv(...)`. `cookimport/analytics/perf_report.py` and `cookimport/analytics/dashboard_collect.py` can already backfill token fields when a manifest contains them, but the current logic is CodexFarm-specific.

For this plan, the term “prediction run manifest” means `prediction-run/manifest.json`, which is the local artifact that describes the generated prediction run and its paths. The term “run manifest” means `run_manifest.json`, which is the higher-level manifest written for stage runs and benchmark eval runs. The term “telemetry artifact” means a JSON file written under `prediction-run/line-role-pipeline/` that contains token totals and attempt details. The term “history CSV” means `.history/performance_history.csv`, which is the coarse trend log consumed by the dashboard.

The exact files that matter for this fix are:

- `cookimport/llm/codex_exec.py` for low-level usage extraction.
- `cookimport/parsing/canonical_line_roles.py` for line-role Codex execution, retries, prompt logging, and cache writes.
- `cookimport/labelstudio/canonical_line_projection.py` for standardized line-role artifact writing.
- `cookimport/labelstudio/ingest.py` for threading line-role artifact paths into prediction and run manifests.
- `cookimport/cli.py` for `PredRunContext` construction and benchmark CSV append.
- `cookimport/analytics/perf_report.py` for history CSV append and manifest-based token backfill.
- `cookimport/analytics/dashboard_collect.py` for dashboard ingestion and secondary backfill.
- `tests/parsing/test_canonical_line_roles.py`, `tests/labelstudio/test_labelstudio_benchmark_helpers.py`, and `tests/analytics/test_stats_dashboard.py` for regression coverage.

## Plan of Work

### Milestone 1: Capture line-role usage at the source and keep retry truth

Start in `cookimport/parsing/canonical_line_roles.py`. The current `_CodexBatchResult` only carries predictions and the parse-error flag. Replace that narrow return contract with a richer internal shape that can carry token usage and retry metadata without changing the public `label_atomic_lines(...)` signature. The public function should continue returning `list[CanonicalLineRolePrediction]` so callers do not need a mass rewrite.

The actual change happens inside `_run_codex_prompt_with_retry()` and `_run_codex_batch_task()`. `_run_codex_prompt_with_retry()` must stop calling `run_codex_json_prompt(..., track_usage=False)` and instead call it with `track_usage=True`. It must keep a list of attempt records, not just the last payload. Each attempt record should capture attempt index, return code, turn-failed message, whether usage was present, and the normalized token values when present. If an attempt returns no usage, record that explicitly rather than silently converting it to zero.

Add a small internal summary builder in `canonical_line_roles.py` that can sum attempts across one batch and then across the whole line-role run. The totals must include failed or incomplete attempts when they emitted usage. The summary must also count how many attempts had usage versus how many did not, so the artifact can truthfully say whether telemetry was complete or partial.

Persist the result to a new file at `prediction-run/line-role-pipeline/telemetry_summary.json` whenever `artifact_root` is available and the line-role pipeline is `codex-line-role-v1`. The file must be written even if some attempts lacked usage. The JSON should be explicit rather than clever. It should include a `schema_version`, the pipeline name, a `token_usage_enabled` boolean, a `summary` object with aggregated token totals and attempt counts, and a `batches` list with one row per prompt batch. Each batch row should include `prompt_index`, candidate count, parse-error state, and an `attempts` list.

The repository already writes prompt files under `line-role-pipeline/prompts/`. Leave those files intact. The new telemetry artifact is not a replacement for prompt logs; it is the durable numeric summary that the rest of the pipeline can trust.

### Milestone 2: Thread the new telemetry artifact into prediction and eval manifests

Once the line-role pipeline writes `telemetry_summary.json`, make the path visible everywhere that already publishes line-role projection artifacts.

Start in `cookimport/labelstudio/canonical_line_projection.py`. Extend `write_line_role_projection_artifacts(...)` so it notices `line-role-pipeline/telemetry_summary.json` and returns a new path key such as `telemetry_summary_path` when the file exists. Do not make this function responsible for creating the telemetry file; it should only publish the path if `canonical_line_roles.py` has already written it.

Then update `cookimport/labelstudio/ingest.py` in both places where it serializes line-role artifact paths into prediction results and manifests. Today it already writes keys such as `line_role_pipeline_stage_block_predictions_path` and `line_role_pipeline_line_role_predictions_path`. Add a sibling key named `line_role_pipeline_telemetry_path`. Also add a matching `run_manifest` artifact entry named `line_role_pipeline_telemetry_json` so higher-level consumers can resolve the path without guessing.

Finally, update the benchmark-side artifact publishing in `cookimport/cli.py` when eval artifacts are assembled. If the prediction run or eval output includes the line-role telemetry file, the eval `run_manifest.json` should publish that artifact too. This ensures upload bundles, dashboard collectors, and future audit tools can find the telemetry without parsing prompts.

### Milestone 3: Merge line-role telemetry into benchmark context and CSV history

The benchmark CSV append path is already wired to accept token numbers through `PredRunContext`. The gap is that `_load_pred_run_recipe_context()` only knows about `llm_codex_farm` telemetry. Fix this in `cookimport/cli.py`.

Add a loader for the new line-role telemetry artifact. The loader should accept the decoded prediction run manifest payload, resolve `line_role_pipeline_telemetry_path` if present, and parse the `summary` object from the JSON file. If the file is absent or malformed, return `None` values and let the caller continue safely.

Then change `_load_pred_run_recipe_context()` so it computes two component-level token payloads: one from `llm_codex_farm` and one from the line-role telemetry artifact. The `PredRunContext.tokens_*` values must become the sum of both components when both are present. If only one component exists, use that one. If neither exists, leave the fields `None` exactly as today.

Do not throw away the component distinction completely. Preserve the line-role path in the manifest and keep the artifact readable so an investigator can still tell whether a total came from recipe Codex, line-role Codex, or both.

Because `append_benchmark_csv(...)` already takes `tokens_input`, `tokens_cached_input`, `tokens_output`, `tokens_reasoning`, and `tokens_total`, no history schema change is required. The behavioral change is that line-role-only benchmark rows should stop arriving with blank token columns.

### Milestone 4: Teach backfill and dashboard ingestion to recover line-role telemetry from manifests

The dashboard and CSV collector code must be able to repair old rows once the new manifest shape exists. Make these changes in `cookimport/analytics/perf_report.py` and `cookimport/analytics/dashboard_collect.py`.

In `perf_report.py`, add a line-role-aware sibling to the existing CodexFarm token extraction helpers. The helper should read `line_role_pipeline_telemetry_path` or the `line_role_pipeline_telemetry_json` artifact, parse the new JSON, and return the five numeric token fields. Then update the backfill path so it merges CodexFarm totals and line-role totals before deciding whether to fill blank CSV fields.

In `dashboard_collect.py`, make the same enrichment change for benchmark records gathered from nested manifests. This collector already backfills token fields when manifest telemetry is present; the missing behavior is support for the dedicated line-role telemetry artifact. Keep the merge policy simple and consistent with `cli.py`: sum component values when both are present, otherwise use the available component.

The acceptance condition for this milestone is that a benchmark row with blank token columns but a populated line-role telemetry artifact is automatically enriched by the collector without needing prompt-byte estimation.

### Milestone 5: Add regression coverage that proves the gap is closed

The tests must prove four things.

First, `tests/parsing/test_canonical_line_roles.py` must prove that a line-role Codex run with mocked `run_codex_json_prompt()` usage writes `line-role-pipeline/telemetry_summary.json` and that the summary counts retries correctly. Include one test where the first attempt fails with usage present and the second attempt succeeds with usage present; the summary must include both attempts in the totals.

Second, `tests/labelstudio/test_labelstudio_benchmark_helpers.py` must prove that the prediction-run manifest now publishes `line_role_pipeline_telemetry_path` and that `_load_pred_run_recipe_context()` includes those tokens when `llm_recipe_pipeline=off` and line-role is on.

Third, `tests/analytics/test_stats_dashboard.py` must prove that a benchmark CSV row with blank token fields can be backfilled from the new line-role telemetry artifact and that the resulting `BenchmarkRecord.tokens_total` is populated.

Fourth, add one mixed-source test where both CodexFarm telemetry and line-role telemetry are present. The merged `tokens_total` in the benchmark context and dashboard record must equal the sum of both components. This prevents a future regression where one component silently overwrites the other.

## Validation and Acceptance

The implementation is complete only when all of the following are true.

A mocked unit test can run the line-role Codex path, emit usage for one or more attempts, and produce `line-role-pipeline/telemetry_summary.json` with a nonzero `summary.tokens_total`.

A prediction-run manifest produced by the benchmark/import helpers includes `line_role_pipeline_telemetry_path`, and the higher-level `run_manifest.json` includes `line_role_pipeline_telemetry_json` in its artifacts.

A benchmark row that previously would have landed with blank token columns now arrives with populated token fields when the line-role telemetry artifact exists.

The dashboard collector can read a manifest with line-role telemetry and backfill `tokens_input`, `tokens_cached_input`, `tokens_output`, `tokens_reasoning`, and `tokens_total` without relying on prompt-byte estimation.

A mixed run that uses both recipe Codex and line-role Codex reports the combined token totals in `PredRunContext` and in the history/dashboard surfaces.

If a live smoke run is performed, the human-visible proof should look like this.

- In `prediction-run/line-role-pipeline/telemetry_summary.json`, `summary.tokens_total` is a positive integer.
- In the corresponding `.history/performance_history.csv` row, `tokens_total` is not blank.
- If the dashboard collector is rerun, the same row remains populated rather than reverting to empty token columns.

## Idempotence and Recovery

This plan is safe to execute incrementally. The new line-role telemetry artifact is additive, and existing prompt files, manifests, and CSV columns remain valid.

If a partially implemented change writes the telemetry artifact but does not yet thread it into manifests, delete the generated test run directories and rerun the targeted pytest suite after finishing the propagation steps.

If a manifest reader lands before the writer, it must treat a missing `line_role_pipeline_telemetry_path` as “no data available” rather than as an error. Preserve that behavior so mixed-version artifacts remain readable.

If a retry-accounting test is flaky, freeze the mocked `run_codex_json_prompt()` sequence in the test instead of adding sleeps or timing assumptions.
