# Relax Workspace-Worker Watchdog To Stop False Kills

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The workspace-worker watchdog is supposed to stop genuine escapes and dead loops, but in practice it is killing normal local work across all three live LLM stages. After this change, a main `workspace_worker` session in recipe, knowledge, or line-role must be allowed to use the same ordinary local shell shapes that the current models already gravitate toward inside the bounded worker workspace. The user-visible result is that benchmark runs stop dying on harmless local helper behavior such as heredocs, `cp`, `mv`, local `python3`, and `cd` back into the session's own execution root. The proof is not merely "the code changed." It is that the same benchmark family that currently records false watchdog kills instead records either successful completion, a real boundary violation, or a real no-progress timeout.

This plan exists because the current watchdog has become a murder spree. The March 22 benchmark `data/output/2026-03-22_12.08.53/.../codexfarm/2026-03-22_12.09.24/` shows one killed worker in every live LLM stage even though the commands that triggered those kills were normal bounded local work. Knowledge was hit worst because one false kill wiped out a worker that owned seventy-nine packet tasks, which then turned into seventy-nine watchdog-attributed task failures. The point of this plan is to stop punishing the models for following their normal local-work instincts when those instincts stay inside the workspace boundary.

## Surprises & Discoveries

- Observation: all three live LLM stages suffered a watchdog kill in the same benchmark even though the killed commands were normal local helper actions.
  Evidence: in `data/output/2026-03-22_12.08.53/.../knowledge/workers/worker-003/live_status.json`, `recipe_phase_runtime/workers/worker-005/live_status.json`, and `line-role-pipeline/runtime/line_role/workers/worker-001/live_status.json`, the reasons are `watchdog_inline_python_heredoc_forbidden` or `watchdog_command_execution_forbidden`, not true workspace escape codes.

- Observation: knowledge is the stage most damaged by the watchdog not because it had more killed worker sessions, but because one killed worker owned far more tasks than the other stages.
  Evidence: `knowledge/task_status.jsonl` records `79` `watchdog_killed` task rows, all under `worker-003`, while that worker wrote zero `out/*.json` files and had `79` assigned tasks. Recipe's single killed worker owned `5` tasks, and line-role's single killed worker still ended with all `4` task outputs present.

- Observation: the killed command shapes are exactly the kinds of local shell behavior the current models naturally choose even when the prompt tells them about helpers.
  Evidence: the killed recipe worker tried a local `cd ... && cat worker_manifest.json ...` sequence inside its own execution root; the killed line-role worker used `cp scratch/... out/...`; the killed knowledge worker used an inline Python heredoc to build a local draft before `check-current`.

- Observation: stage-level telemetry already separates command classification from enforcement, so the repo has the seam needed to demote most command-shape kills into telemetry.
  Evidence: `docs/10-llm/10-llm_README.md` documents that `classify_workspace_worker_command(...)` is the reviewer-facing telemetry layer and `detect_workspace_worker_boundary_violation(...)` is the enforcement seam.

- Observation: the remaining March 22 knowledge false kill was not coming from the shared detector anymore; it was coming from a stage-local launch override that re-enabled the heredoc-specific kill on the main worker path.
  Evidence: `cookimport/llm/codex_farm_knowledge_orchestrator.py` passed `forbid_inline_python_heredocs=True` into `_build_strict_json_watchdog_callback(...)` at both main knowledge workspace-worker launch sites, and the new regression `test_knowledge_orchestrator_workspace_workers_do_not_force_heredoc_kills` failed before that override was removed.

## Context and Orientation

The main enforcement seam lives in `cookimport/llm/codex_exec_runner.py`. That file defines the direct `codex exec` worker runtime, the workspace mirroring logic, the command classifier, and the boundary detector. In this repository, a `workspace_worker` session is a long-lived Codex subprocess that runs inside a sterile execution workspace under `~/.codex-recipe/recipeimport-direct-exec-workspaces/...` while the durable artifact root stays under `data/output/.../workers/worker-XXX`.

The shared workspace watchdog already has two distinct concepts that must not be conflated. A command classifier is the reviewer-facing logic that labels what kind of shell behavior the worker attempted. A boundary detector is the kill/no-kill logic that decides whether the worker actually left the allowed workspace surface. The docs already say the runtime should be boundary-based, but the saved benchmark artifacts show that the live policy is still killing on harmless command styles in some cases.

The most relevant files for this work are:

- `cookimport/llm/codex_exec_runner.py`, especially `classify_workspace_worker_command(...)`, `detect_workspace_worker_boundary_violation(...)`, `should_terminate_workspace_command_loop(...)`, `_wrap_workspace_supervision_callback(...)`, and the worker-facing `AGENTS.md` text builder.
- `cookimport/llm/codex_farm_knowledge_orchestrator.py`, `cookimport/llm/codex_farm_orchestrator.py`, and `cookimport/parsing/canonical_line_roles.py` or their helper seams, because those stage runtimes consume the shared watchdog outputs and may carry stage-local guardrails or retry behavior that assumes the old kill policy.
- `tests/llm/test_codex_exec_runner_workspace.py`, plus the relevant recipe, knowledge, and line-role runtime tests that assert watchdog reasons or recovery behavior.

The benchmark evidence that motivates this plan is concrete:

- Knowledge false kill: `watchdog_inline_python_heredoc_forbidden` on `worker-003`, which then expands into `79` `watchdog_killed` task rows.
- Recipe false kill: `watchdog_command_execution_forbidden` on `worker-005` for a local `cd ... && cat ...` command, later recovered by watchdog retry.
- Line-role false kill: `watchdog_command_execution_forbidden` on `worker-001` for a local `cp scratch/... out/...` command, with stage recovery afterward.

Those are not repository escapes. They are normal local worker behavior. This plan assumes that the user wants the main worker path to tolerate such behavior and only kill on genuine safety or liveness problems.

## Plan of Work

The first milestone is proof, not editing. Add failing tests around `cookimport/llm/codex_exec_runner.py` that encode the exact command shapes the benchmark killed. At minimum, cover inline Python heredocs for knowledge-style local draft building, `cd` into the current execution root followed by local reads for recipe-style orientation, and local `cp` or `mv` from `scratch/` to `out/` for line-role or recipe finalization. Those tests should fail under the current policy if the command still triggers a kill and pass only once the command becomes telemetry-only.

The second milestone is the policy rewrite in the shared watchdog. Update `detect_workspace_worker_boundary_violation(...)` so it no longer treats command shape by itself as a kill reason on the main worker path. The detector should continue to reject only the narrow set of genuinely dangerous behaviors: leaving the workspace, parent traversal, non-temp absolute paths outside approved roots, or clearly forbidden executables such as `git`, `curl`, `wget`, `ssh`, package managers, and similar repo/network tools. If a command stays inside the execution root, `scratch/`, `out/`, `in/`, `hints/`, or approved temp roots, it should survive even if it uses heredocs, `cp`, `mv`, or multi-command shell scripts.

The third milestone is a blunt command-policy rewrite for the main worker path. `classify_workspace_worker_command(...)` should keep reporting what happened for telemetry, but the runtime should stop converting those telemetry labels into kills unless the boundary detector returns a real violation. This is the heart of the change. A classification like `shell_script_workspace_local`, `tolerated_workspace_shell_command`, or even a local `cd` back into the current execution workspace should become reviewable metadata, not a termination decision.

The fourth milestone is preserving the watchdog's real job. Keep or tighten only the protections that still matter:

- real repo or network tool use such as `git`, `curl`, `wget`, `ssh`, `scp`, package managers, container tools, or similar
- parent traversal and other obvious attempts to leave the worker workspace
- non-temp absolute paths outside the allowed execution root and approved temp roots
- pathological no-progress loops, silence timeouts, or other liveness failures

Everything else on the main worker path should become telemetry-only. That includes:

- inline Python heredocs
- local `python3` or `node` transforms that touch only workspace-local paths
- `cp`, `mv`, `cat`, `jq`, `sed`, pipes, redirections, and short shell scripts
- `cd` into the worker's own execution root
- reading `worker_manifest.json`, `current_task.json`, `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`, `assigned_tasks.json`, `in/*.json`, `hints/*.md`, `scratch/*`, and `out/*`

The fifth milestone is stage-level validation. Because recipe and line-role currently recover from their false kills while knowledge does not, the tests and optional benchmark proof must explicitly show both the shared policy change and the stage outcomes. Acceptance is not "the kill counter dropped in one stage." It is "the shared main worker policy stopped killing harmless local commands in all three stages, and the remaining failures are real logic issues rather than shell-style false positives."

## Very Blunt Policy List

This section is intentionally direct. It is the target policy, not a vague aspiration.

Keep as hard kill reasons on main `workspace_worker` sessions:

- any visible use of repo/network/package-manager/container/remote tools such as `git`, `curl`, `wget`, `ssh`, `scp`, `npm`, `pip`, `docker`, `kubectl`, and similar
- parent-directory traversal or any command that clearly leaves the bounded worker workspace
- any non-temp absolute path that is not the current execution root or a child of it
- genuine no-progress loops, silence timeouts, or cohort/runtime outlier logic if those guards are still needed

Demote to telemetry-only on main `workspace_worker` sessions:

- inline Python heredocs
- local `python3` or `node` scripts
- `cp`, `mv`, `cat`, `jq`, `sed`, `awk`, pipes, redirections, and shell scripts
- `cd` to the current execution root
- bulk local helper flows that touch only `in/`, `hints/`, `scratch/`, `out/`, and the repo-written sidecars
- manifest reads, local queue inspection, and file copying inside the worker root

Do not change in this plan:

- the strict one-shot structured retry / repair watchdog behavior
- the semantic validators
- the knowledge mid-queue early-exit problem
- any prompt-level or queue-handoff fixes unrelated to watchdog de-escalation

## Validation and Acceptance

This change is accepted only when the watchdog demonstrably stops killing harmless local worker behavior on the main worker path.

At the unit-test level, the new tests must prove that the following command families no longer trigger `detect_workspace_worker_boundary_violation(...)` on main `workspace_worker` sessions when all visible paths remain local:

- inline Python heredocs
- local `cd` back into the current execution workspace
- local `cp` or `mv` from `scratch/` to `out/`

At the shared-runtime level, telemetry should still classify those commands, but the supervision layer must not terminate the worker for them. The kill/no-kill seam should key off real boundary violations and true no-progress pathologies only.

At the stage level, the March 22 style false kills should disappear:

- knowledge should no longer produce a `watchdog_inline_python_heredoc_forbidden` kill for a worker-local draft-building command
- recipe should no longer kill a worker for `cd ... && cat ...` inside the execution workspace
- line-role should no longer kill a worker for local `cp scratch/... out/...`

At the optional benchmark level, acceptance means the watchdog no longer creates the dominant failure mode for these stages. If a benchmark still fails, the remaining failures should be attributable to queue continuity, semantic invalidity, or other logic bugs rather than false command-style kills.

## Idempotence and Recovery

The watchdog policy change must be safe to apply repeatedly. Re-running the same worker command should produce the same telemetry classification and the same non-kill outcome when the command stays inside the allowed workspace boundary.

If a relaxation accidentally opens a real escape path, the safe recovery is to restore only the specific boundary check that guards that path, not to reintroduce broad command-style kills. This plan is specifically trying to remove style policing while preserving real safety boundaries.

As with the user's broader benchmarking work, live benchmark reruns remain optional and must not happen without explicit approval.

## Interfaces and Dependencies

In `cookimport/llm/codex_exec_runner.py`, the following interfaces should remain stable but change behavior:

- `classify_workspace_worker_command(...)` should keep emitting rich telemetry labels and reasons.
- `detect_workspace_worker_boundary_violation(...)` should become stricter about real escape detection and looser about harmless local shell shape.
- `should_terminate_workspace_command_loop(...)` should remain the no-progress/liveness guardrail.

In the stage runtimes that consume watchdog decisions, the meaning of the shared reason codes should stay understandable after the change. If some stage-specific code assumes that any local shell script is inherently kill-worthy, update that assumption there rather than re-tightening the shared detector.

The most important external dependency for this plan is the benchmark artifact surface itself. The implementation should keep using the saved `live_status.json`, `telemetry.json`, and per-stage task ledgers as the source of truth for whether the watchdog is still the dominant failure mode.
