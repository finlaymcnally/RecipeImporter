# Bound Task Summaries And Clarify Line-Role Worker Surface

## Problem

The helper-first startup path still allows `editable_task_file --summary` to grow with every unanswered unit id, so the "compact summary" claim is not actually enforced on large task files.
The completed line-role reason-hint plan also still lacks normal docs front matter, and the current docs/tests should be explicit that repo-side debug artifacts are not the worker-visible single-file startup surface.

## Acceptance Criteria

- `python3 -m cookimport.llm.editable_task_file --summary` returns a bounded summary shape whose unit-id listing is sampled and explicitly marked when truncated.
- Focused tests cover both the small-summary and truncated-summary contracts.
- Current worker-surface docs stay accurate about helper-first startup and about line-role deterministic semantic hints staying out of the worker-visible `task.json`.
- The line-role reason-hint ExecPlan has normal front matter so it shows up cleanly in `npm run docs:list`.

## Evidence

- Added a regression that builds a task file with more than the summary sample limit and expects a sampled `unanswered_unit_ids` list plus `unanswered_unit_ids_truncated=True`.
- Preserved the small-task summary assertions so bounded summaries still stay straightforward on tiny task files.
- Refreshed docs/plan metadata so the active contract and plan index match the shipped behavior.

## Constraints / Decisions

- Keep the helper summary change local to `cookimport.llm.editable_task_file`; do not widen this into a larger worker-command policy refactor.
- Keep line-role repo-owned debug artifacts intact. The goal is to keep the worker-visible startup contract narrow and documented correctly, not to remove operator/debug evidence from repo artifacts.
- Prefer additive summary metadata over opaque shortening so workers and humans can tell when a summary is sampled.
