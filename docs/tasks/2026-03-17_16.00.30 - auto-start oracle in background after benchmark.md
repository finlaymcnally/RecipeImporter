Problem statement

Interactive benchmark wrap-up should still fire Oracle automatically, but it must not block the benchmark menu while Oracle browser automation runs.

Acceptance criteria

- Interactive single-offline benchmark wrap-up auto-starts Oracle in the background after writing `upload_bundle_v1`.
- Interactive multi-book single-profile wrap-up auto-starts Oracle in the background for the top-level group bundle only.
- Wrap-up returns immediately and prints the detached Oracle PID plus `oracle_upload.log` / `oracle_upload.json` paths.
- Oversized browser uploads keep working when detached.

Verification steps

- Activate `.venv`.
- Run `pytest tests/bench/test_benchmark_oracle_upload.py tests/labelstudio/test_labelstudio_benchmark_helpers_single_offline_run.py tests/labelstudio/test_labelstudio_benchmark_helpers_single_profile.py tests/labelstudio/test_labelstudio_benchmark_smoke.py`.

Evidence

- Detached Oracle launches now use `start_oracle_benchmark_upload_background(...)` and write durable launch state under `upload_bundle_v1/.oracle_upload_runs/<timestamp>/`.
- Interactive helper tests now assert background Oracle launch calls instead of the manual-only hint.
- Verification result: `source .venv/bin/activate && pytest tests/bench/test_benchmark_oracle_upload.py tests/labelstudio/test_labelstudio_benchmark_helpers_single_offline_run.py tests/labelstudio/test_labelstudio_benchmark_helpers_single_profile.py tests/labelstudio/test_labelstudio_benchmark_smoke.py` -> `32 passed, 1 warning in 2.38s`.

Constraints/gotchas + decision

- Sharded browser uploads cannot rely on `TemporaryDirectory` once Oracle is detached; staged files must live under the bundle until the run finishes.
- Keep `cookimport bench oracle-upload` as the manual retry path even though interactive runs now auto-start Oracle.

Rollback / compensation plan

- Revert the interactive helper call sites in `cookimport/cli.py` back to manual-only wrap-up if detached Oracle proves too flaky in practice.
