## Operator Intent (Structured) - DO NOT DELETE

The operator wants a real per-book cache for repeated benchmark work, not just the current narrow benchmark helper reuse seam. The same EPUB or PDF should be reusable across many runs, across interactive shard planning and real execution, and ideally across benchmark and stage flows too. The cache should be keyed by what the book is and what deterministic work was requested, not by which temporary run folder happened to be used that day. The operator runs the same benchmark on the same files hundreds of times, so the feature is only successful if warm reruns skip most of the deterministic book prep work.

## Purpose / Big Picture

After this change, rerunning the same benchmark or stage job on the same book is mostly a cache hit. The repo now stores conversion, deterministic prep, and preview artifacts under one stable shared root (`.cache/cookimport/book-cache/` by default), keys deterministic prep and conversion by source bytes instead of source path, and reports those cache layers back through benchmark manifests and CLI progress.

The implementation keeps the three cache layers separate on purpose. Conversion cache reuses importer work, deterministic prep cache reuses the label-first recipe-boundary bundle, and preview cache reuses shard-planning manifests. Stage and benchmark flows now meet at the deterministic prep layer: benchmark flows can build or reuse it directly, while stage can reuse an existing bundle on a warm rerun and persist a fresh one after a cold run finishes.

## Decision Log

- Decision: The fully fledged feature should be a layered per-book cache, not a single giant cache artifact.
  Rationale: Repeated reruns reuse three different kinds of work with different invalidation rules. Importer conversion, deterministic prep, and shard planning should each have their own cache identity and their own manifest. That keeps invalidation honest and avoids rebuilding deterministic prep just because a preview-only setting changed.
  Date/Author: 2026-04-04 / Codex

- Decision: Content hash, not source path, must be the authority for "same book."
  Rationale: The operator explicitly wants repeated reuse across the same EPUB files. If identical bytes copied to a different path miss the cache, the feature will still feel wrong in practice. Source path should remain metadata for operator visibility, not part of the identity contract.
  Date/Author: 2026-04-04 / Codex

- Decision: The shared book cache must live under one stable cache root outside timestamped run outputs.
  Rationale: Output-root-local caches are too tied to one run family. A real per-book cache must survive many benchmark and stage runs and should not disappear just because the output folder moved or was cleaned.
  Date/Author: 2026-04-04 / Codex

- Decision: `cookimport stage` and benchmark flows should use the same per-book cache service.
  Rationale: The expensive deterministic work is not benchmark-specific. Keeping stage as a separate non-adopter would leave the repo with two parallel stories and would miss a large part of the operator's actual rerun cost.
  Date/Author: 2026-04-04 / Codex

- Decision: Existing output-root-local caches do not need on-disk migration.
  Rationale: This is a solo project, the old caches are implementation details, and a cold rebuild into the new shared cache root is cheaper and safer than writing migration code. Cleanup support should remove the old cache folders later.
  Date/Author: 2026-04-04 / Codex

- Decision: All cache entries must be immutable once complete, with atomic manifest writes and explicit lock files during construction.
  Rationale: Repeated benchmark runs and future multi-book scheduling can hit the same entry concurrently. Immutable entries plus atomic completion are the cleanest way to avoid partial reuse and race corruption.
  Date/Author: 2026-04-04 / Codex

- Decision: Preview-cache identity should include only settings that can change shard survivability outputs.
  Rationale: The operator reruns the same benchmarks often. If changing a model label or reasoning effort invalidates preview cache even when shard counts would be identical, the repo will still redo cheap-but-annoying work for no good reason.
  Date/Author: 2026-04-04 / Codex

- Decision: Higher-level benchmark prediction-reuse caches should stay above the new book cache instead of being collapsed into it.
  Rationale: The repo already has broader benchmark reuse seams keyed by prediction identity. The new per-book cache should provide stable low-level book prep reuse under those layers, not replace every benchmark cache concept in one refactor.

## Outcomes & Retrospective

The full shared per-book cache landed. The repo now has a stable cache root in `.cache/cookimport/book-cache/`, a shared conversion-cache layer keyed by source bytes plus importer-sensitive settings, a deterministic prep layer keyed by source bytes plus deterministic stage settings, and a preview layer keyed by shard-planning inputs only. Benchmark manifests now surface `book_cache` telemetry, `cookimport stage` can reuse cached recipe-boundary state on warm reruns, and `bench gc` can preview or prune the shared cache root plus legacy local cache folders.

The main lesson from the implementation is that the repo did not need a new architecture so much as one honest place for the existing layers to meet. The conversion, deterministic prep, and preview seams were already present in some form; the real work was making their identity rules clean, giving them one stable home, and teaching stage to participate without distorting its cold-run behavior.

## Context and Orientation

This repository has two major runtime families that touch repeated per-book work.

The first family is stage execution. `cookimport stage ...` writes timestamped run outputs under `data/output/<YYYY-MM-DD_HH.MM.SS>/`, and the shared semantic session lives under `cookimport/staging/import_session.py` and `cookimport/staging/import_session_flows/output_stage.py`. Stage runs already have a deterministic boundary between importer conversion and later semantic stages. That boundary is what makes per-book caching possible.

The second family is benchmark and Label Studio execution. `cookimport labelstudio-benchmark ...` and the benchmark helpers under `cookimport/cli_support/` run the same books repeatedly while exploring variants, shard counts, and later scoring. Those flows already have multiple cache-like seams because repeated benchmark work is expensive.

In this plan, "per-book cache" means a stable, repo-owned cache rooted outside timestamped run outputs that stores immutable artifacts for the same source bytes. A cache "layer" means one reuse boundary with its own identity payload and manifest. This plan uses three layers.

The conversion layer stores importer outputs that depend on source bytes and importer-sensitive parsing settings. This is the earliest expensive step. Today the nearest equivalent is the single-book split cache in benchmark code.

The deterministic prep layer stores all deterministic stage-owned artifacts needed before optional LLM stages. That includes the canonical extracted source model, authoritative labels, recipe spans, recipe ownership, and nonrecipe routing context. Today the nearest equivalent is the deterministic prep bundle in `cookimport/staging/deterministic_prep.py`.

The preview layer stores prompt-preview or shard-planning outputs that depend on one deterministic prep bundle plus shard-planning settings. Today the nearest equivalent is the `prompt-preview-manifests/` folder nested under a deterministic prep bundle.

The current design weakness is not that the repo lacks these layers. The weakness is that they are not yet one content-addressed cross-workflow system. Identity is still partly path-based, roots still partly live under run outputs, and stage has not adopted the shared cache story end to end.

## Plan of Work

Start by creating one neutral cache owner module, preferably `cookimport/cache/book_cache.py`, with a small supporting package if the file gets too large. This owner must not live inside benchmark helpers. Its job is to resolve the shared cache root, build identity payloads, acquire entry locks, validate manifests, and write atomic completion markers. Keep the existing stage-specific logic in `cookimport/staging/deterministic_prep.py`, but make it build on the new neutral cache owner instead of inventing its own root and identity rules.

Add one canonical shared cache root resolver in `cookimport/paths.py`. The default should be repo-local and stable across workflows, for example `<repo>/.cache/cookimport/book-cache/`, with an explicit environment override and optional CLI override. Do not derive this root from the processed output root. The processed output root controls run artifacts, not shared cache identity.

Replace path-based identity with content-addressed identity. Introduce three explicit identity builders. `build_conversion_cache_key(...)` should hash source bytes plus importer-sensitive settings such as EPUB extractor, HTML parser version, split-sensitive parsing inputs, OCR knobs when applicable, and any schema version constant for that layer. `build_deterministic_prep_key(...)` should hash source bytes plus deterministic stage inputs while excluding LLM-only settings. `build_preview_cache_key(...)` should hash only the settings that can change shard planning results, such as pipeline enablement, prompt target counts, context/budget knobs, and line-role shard sizing. Source path should be recorded in manifests as the latest seen path, but not as part of the identity.

Generalize the old benchmark split cache into the new conversion cache layer. The implementation can start by reusing the existing `ConversionResult` serialization and lock semantics from `cookimport/cli_support/bench_cache.py`, but the naming and owner module should become neutral. Benchmark callers should stop thinking in terms of "single-book split cache" and instead ask the shared cache owner for a conversion entry. Stage callers should use the same entrypoint before importer work begins.

Refactor deterministic prep so it resolves through the shared cache owner and no longer chooses its own cache root under `processed_output_root`. Keep the manifest human-readable. Preserve the current ability to reconstruct a `RecipeBoundaryResult`, because that is the clean resume depth for later execution. The new deterministic prep entry should record which conversion cache entry it depended on so a future operator can inspect lineage.

Narrow preview cache identity and make it an explicit third cache layer. The preview manifest should no longer be keyed by the entire run config dict. It should be keyed by the deterministic prep key plus a smaller preview payload that includes only shard-planning inputs. This change matters because warm reruns often tweak model choice without changing shard survivability.

Adopt the new cache service across benchmark flows. `cookimport/cli_support/bench_single_book.py`, `cookimport/cli_support/bench_single_profile.py`, `cookimport/cli_support/bench_all_method*.py`, and `cookimport/labelstudio/ingest_flows/prediction_run.py` should all resolve conversion and deterministic prep through the shared cache service. Keep higher-level prediction reuse caches as they are. They can simply sit above the new book cache and benefit from cheaper misses.

Adopt the same cache service for `cookimport stage`. The simplest safe rollout is to let stage resolve the shared conversion cache and deterministic prep cache, then resume from cached recipe-boundary state when possible. Stage still writes fresh run-owned outputs under the timestamped output root. Cache entries remain immutable and must never become the live write target for a stage run.

Add operator-visible telemetry. Benchmark manifests, stage run manifests, and interactive progress text should report which cache layers were hit or rebuilt, which keys were used, how long each layer took, and where the shared cache root lives. The operator does not need enterprise dashboards, but they do need enough visibility to trust that the reuse is real.

Extend cleanup support. The existing `cookimport bench gc` command should grow shared book-cache awareness rather than leaving cache cleanup to manual shell work. Add options to preview or apply pruning for the shared cache root by age and total retained bytes, while keeping the dry-run-first behavior. The same cleanup path should also delete obsolete output-root-local cache folders such as `.deterministic-prep-cache` and old split-cache folders when they are discovered under benchmark roots.

Finally, update docs and tests. The owning docs are `docs/01-architecture/01-architecture_README.md`, `docs/05-staging/05-staging_readme.md`, `docs/07-bench/07-bench_README.md`, and `docs/12-testing/12-testing_README.md`. Keep documentation short and honest. The plan is only complete when the docs describe one shared layered per-book cache instead of three unrelated local caches.

## Milestone 1: Create The Neutral Shared Book Cache Owner

This milestone introduces the new shared cache root, immutable entry layout, content-addressed key builders, and lock/manifest helpers. At the end of the milestone, the repo will have one neutral cache owner that can create and validate cache entries for conversion, deterministic prep, and preview layers without changing runtime behavior yet.

The proof is a focused test suite that creates cache keys for the same source bytes at two different paths and gets the same key, then creates keys for the same path with different relevant settings and gets the expected misses. No benchmark or stage caller has to adopt the new owner yet, but the owner must be complete enough that later milestones can build on it without inventing new identity rules.

## Milestone 2: Migrate Conversion Cache Into The Shared Per-Book Layer

This milestone replaces the old single-book split cache naming and ownership with the neutral conversion cache layer. At the end of the milestone, benchmark callers resolve conversion results through the shared book cache, and the repo can show a warm conversion hit for the same source bytes even when the file path changed.

The proof is one benchmark-helper test that reruns the same source with the same conversion inputs and sees a cache hit, plus one path-copy test that points at a second copy of the same fixture and still sees the same conversion key. A changed importer-sensitive setting such as EPUB extractor should produce a cache miss and a new entry.

## Milestone 3: Migrate Deterministic Prep Cache Onto The Shared Root

This milestone moves deterministic prep onto the shared cache root and removes path-aware identity from that layer. At the end of the milestone, deterministic prep reuse no longer depends on `processed_output_root`, and the cached prep bundle records lineage back to the conversion cache entry it used.

The proof is that the same source bytes and deterministic settings produce a warm deterministic prep hit across repeated benchmark runs, even when the processed output root differs. Changing only LLM model or reasoning settings must still hit the deterministic prep cache. Changing a deterministic parsing or stage input must miss.

## Milestone 4: Narrow And Reuse Preview Cache For Shard Planning

This milestone turns prompt-preview manifest reuse into an explicit preview cache layer keyed only by shard-planning inputs. At the end of the milestone, interactive shard planning can reuse preview manifests whenever shard survivability math would be identical, even if unrelated run settings changed.

The proof is a test that changes only model or reasoning settings and still gets a preview cache hit, plus a test that changes a real planning input such as prompt target count and gets a miss. Interactive single-book shard planning should still return the same `minimum_safe_shard_count` values as before.

## Milestone 5: Adopt The Shared Cache In Benchmark Execution

This milestone wires the new shared cache through single-book, single-profile, and all-method benchmark execution. At the end of the milestone, benchmark flows reuse the shared conversion and deterministic prep layers before any higher-level prediction reuse logic is considered.

The proof is that a second benchmark run over the same fixture book reports cache hits at the shared book-cache layers and still writes the same benchmark outputs as before. Multi-book benchmark scheduling should also show one deterministic prep build per unique cache key rather than one per variant.

## Milestone 6: Adopt The Shared Cache In `cookimport stage`

This milestone makes stage a first-class consumer of the same shared cache service. At the end of the milestone, repeated stage runs on the same book can skip conversion and deterministic prep work when their inputs match, while still writing fresh timestamped run-owned outputs.

The proof is one stage-focused test or controlled fixture run that executes the same source twice and shows that the second run reuses shared cache entries rather than rebuilding the deterministic book state. The cache entries themselves must remain read-only during stage output writing.

## Milestone 7: Add Operator Controls, Telemetry, And Cleanup

This milestone finishes the feature from the operator's point of view. At the end of the milestone, the CLI exposes a small shared cache control surface, run manifests record cache behavior, and `bench gc` can preview and prune the shared cache root along with obsolete local cache directories.

The proof is one CLI test for dry-run cleanup reporting, one runtime test that verifies manifests include cache hit or miss data, and one manual operator story showing a warm rerun with visible cache-hit reporting.

## Validation and Acceptance

Acceptance is behavioral. The feature is done only when the operator can see repeated same-book reuse and the repo can prove the invalidation boundaries are correct.

For conversion cache correctness:

1. Run the same source twice with identical importer-sensitive settings and observe a conversion cache hit the second time.
2. Copy the same EPUB fixture to a second path, rerun, and observe the same conversion cache key and a cache hit.
3. Change one importer-sensitive setting such as EPUB extractor and observe a cache miss.

For deterministic prep correctness:

1. Run the same source twice with identical deterministic settings and observe a deterministic prep cache hit the second time.
2. Change only LLM model or reasoning settings and observe that deterministic prep still hits.
3. Change a deterministic stage input and observe a miss.

For preview cache correctness:

1. Run shard planning twice with the same deterministic prep and the same planning inputs and observe a preview cache hit.
2. Change only an unrelated run setting such as model label and observe that preview still hits.
3. Change a planning-relevant input such as prompt target count or context budget and observe a miss.

For benchmark and stage adoption:

1. Repeat the same benchmark workflow on the same fixture book and observe conversion hit, deterministic prep hit, and preview hit on the warm run.
2. Repeat the same stage workflow on the same fixture book and observe conversion hit and deterministic prep hit on the warm run.
3. Confirm that both workflows still write fresh run-owned outputs under timestamped run roots.

For cleanup:

1. Run `cookimport bench gc --dry-run ...` and observe shared book-cache reporting.
2. Confirm that the command reports reclaimable cache bytes separately from benchmark run roots.
3. Confirm that old output-root-local cache directories are discoverable and removable through the same cleanup path.

The feature is only complete when a same-book warm rerun can hit all applicable cache layers even if the input file was copied to a new path and even if the output root for the run changed.

## Idempotence and Recovery

This design must be safe to retry. Every cache entry should be immutable after completion, and incomplete entries should be ignored until their manifest is fully written and atomically published. Rebuilding the same entry after a crash should either wait on the live lock or replace the incomplete entry cleanly.

When cache schema or identity rules change, bump the relevant layer schema version and allow a cold rebuild. Do not try to mutate old cache entries in place. This project does not need migration code for private cache artifacts.

If a cache manifest points at missing files or fails validation, the runtime must treat it as a miss, delete or quarantine the broken entry, and rebuild from the previous layer. Never partially trust a broken entry.

Because stage and benchmark will both consume this cache, they must never write new outputs into cache-owned directories. Cached entries are read-only sources for later runs. Fresh run artifacts always belong under the normal timestamped run roots.

## Artifacts and Notes

The final feature should create one stable cache tree with clearly named layers. One acceptable shape is:

    .cache/cookimport/book-cache/
      conversion/<conversion_key>/conversion_manifest.json
      deterministic-prep/<prep_key>/deterministic_prep_bundle_manifest.json
      preview/<prep_key>/<preview_key>.json

Each manifest should be human-readable JSON and should include at minimum:

- a schema version
- the cache key
- the source hash
- the latest seen source path for operator reference
- the settings payload used to build the key
- lineage to the previous layer when applicable
- creation time
- cache-hit or build timing details
- the owned artifact paths for that layer

Expected warm rerun story:

    operator runs benchmark on book A
    -> conversion miss
    -> deterministic prep miss
    -> preview miss
    -> benchmark writes normal run outputs

    operator reruns benchmark on same book A
    -> conversion hit
    -> deterministic prep hit
    -> preview hit
    -> benchmark writes fresh run outputs without rebuilding shared prep

Expected copied-file story:

    book A at path one and copied book A at path two have identical bytes
    -> same source hash
    -> same conversion key
    -> same deterministic prep key
    -> same preview key for identical planning settings

Expected invalidation story:

    same source bytes + changed EPUB extractor
    -> conversion miss
    -> deterministic prep miss downstream

    same source bytes + same deterministic settings + changed model label
    -> conversion hit
    -> deterministic prep hit
    -> preview hit if shard-planning settings are unchanged

## Interfaces and Dependencies

The final implementation should expose neutral interfaces that reflect the three cache layers directly.

In `cookimport/cache/book_cache.py`, define strongly typed entry result objects similar to:

    @dataclass(frozen=True, slots=True)
    class ConversionCacheEntry:
        key: str
        source_hash: str
        artifact_root: Path
        manifest_path: Path
        conversion_result_path: Path
        cache_hit: bool
        timing: dict[str, float]

    @dataclass(frozen=True, slots=True)
    class DeterministicPrepCacheEntry:
        key: str
        source_hash: str
        artifact_root: Path
        manifest_path: Path
        processed_run_root: Path
        conversion_entry_key: str
        cache_hit: bool
        timing: dict[str, float]

    @dataclass(frozen=True, slots=True)
    class PreviewCacheEntry:
        key: str
        prep_key: str
        manifest_path: Path
        recommendations_by_step: dict[str, dict[str, Any]]
        cache_hit: bool

Also define neutral root and key helpers with stable naming:

    resolve_book_cache_root(
        *,
        override: Path | None = None,
    ) -> Path

    build_conversion_cache_key(
        *,
        source_hash: str,
        run_settings: RunSettings,
        source_file: Path,
    ) -> str

    build_deterministic_prep_key(
        *,
        source_hash: str,
        run_settings: RunSettings,
        source_file: Path,
    ) -> str

    build_preview_cache_key(
        *,
        prep_key: str,
        selected_settings: RunSettings,
    ) -> str

And define resolver entrypoints used by runtime code:

    resolve_or_build_conversion_cache_entry(...)
    resolve_or_build_deterministic_prep_entry(...)
    resolve_or_build_preview_cache_entry(...)

`cookimport/staging/deterministic_prep.py` should remain the stage-aware owner of reconstructing `RecipeBoundaryResult` and other semantic runtime bundles, but it should depend on the neutral cache owner for identity, root resolution, lock semantics, and manifest I/O.

`cookimport/labelstudio/ingest_flows/prediction_run.py` and `cookimport/staging/import_session_flows/output_stage.py` should consume the new cache entry objects instead of passing raw manifest paths around loosely. If a manifest path still needs to cross a CLI boundary, keep the typed object at the runtime boundary and reduce to a path only at the final CLI serialization layer.

`cookimport/paths.py` should own the default shared cache root resolution. `cookimport/cli_commands/bench.py` should own the cleanup command changes. `docs/01-architecture/`, `docs/05-staging/`, `docs/07-bench/`, and `docs/12-testing/` are the required documentation owners once implementation changes reality.
