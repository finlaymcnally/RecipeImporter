## Purpose / Big Picture

After this change, routine non-slow pytest runs should stop paying for files that behave like heavyweight integration suites. The user-visible effect is shorter agent feedback loops: `./scripts/test-suite.sh fast` and per-domain non-slow runs should spend their time on routing and contract checks, while truly expensive helper-internal coverage moves into the explicit `slow` slice.

## Context and Orientation

Routine suite entrypoints live in [scripts/test-suite.sh](/home/mcnal/projects/recipeimport/scripts/test-suite.sh), which calls one plain pytest process per requested slice. Marker routing and the fast-vs-slow classification live in [tests/conftest.py](/home/mcnal/projects/recipeimport/tests/conftest.py). The main policy doc for the test suite is [docs/12-testing/12-testing_README.md](/home/mcnal/projects/recipeimport/docs/12-testing/12-testing_README.md).

In this repository, a “routing test” is a test that verifies argument handoff or CLI/control-flow decisions without needing the full helper internals beneath it. A “helper-internal test” is a test that intentionally exercises artifact writing, comparison generation, dashboard refreshes, or other multi-step helper behavior. Routing tests belong in the routine non-slow slice; helper-internal tests may belong in `slow` when they are materially expensive.

The current cleanup follows the pattern already used in labelstudio: identify files where the fast slice is accidentally exercising the real helper internals, then either stub at the boundary or reclassify the dedicated heavy helper file as `slow`.

## Plan of Work

First, measure the remaining domains with non-slow pytest invocations and record which domains dominate runtime. Then, within the slowest domains, run each file individually under pytest to find the true hotspots. For each hotspot, inspect whether the file is doing routing assertions, helper-internal assertions, or a mix.

If a file is routing-only but currently calls into heavy helper internals, edit the test file to patch the helper boundary and assert the handoff payload instead. If a file is a legitimate heavy helper-internal suite with focused value, move it into `_SLOW_FILES` in [tests/conftest.py](/home/mcnal/projects/recipeimport/tests/conftest.py) and add a short policy note in the closest test docs. If a mixed file contains both fast routing checks and slow helper-internal checks, keep the fast checks in the non-slow file and move the heavy coverage into the slow slice only when the split is necessary.

After each optimization wave, rerun the affected files and then rerun the affected domain with `-m 'not slow'`. At the end, run a broad non-slow suite command to prove the aggregate improvement.

## Validation and Acceptance

Acceptance is met when the edited domains show clearly lower non-slow runtime, the moved slow files still pass in the `slow` slice, and broad routine non-slow runs retain the expected pass/fail surface without silently dropping needed coverage. The new fast/slow boundaries must also be documented in the nearest lightweight docs file.

## Idempotence and Recovery

These edits are safe to repeat because they only change test classification and mocking boundaries. If a reclassification weakens intended coverage, the rollback is to remove the file from `_SLOW_FILES` or restore the prior test seam in the specific test file.
