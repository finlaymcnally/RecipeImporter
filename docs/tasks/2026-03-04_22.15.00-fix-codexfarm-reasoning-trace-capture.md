# Fix CodexFarm Reasoning Trace Capture for Nested Events

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds. This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The user-visible goal was to make recipeimport reports show real Codex thinking traces again instead of the fallback `_No thinking trace captured for this sample._`. The failure turned out to be upstream in the shared `CodexFarm` repo: trace artifacts were being written, but nested reasoning events wrapped as `item.completed` were not being classified into `reasoning_events`.

After this change, recipeimport can once again surface reasoning excerpts from those traces without any exporter-side workaround. The observable proof is a CodexFarm trace artifact whose `reasoning_event_count` becomes non-zero for nested reasoning events, which recipeimport can already render because it reads `text` from captured reasoning events.

## Surprises & Discoveries

- Observation: recipeimport did not need a code change.
  Evidence: `cookimport/cli.py:_extract_reasoning_excerpt(...)` already accepts `text`, `summary_text`, `summary`, `delta`, and `content`, which covers the nested reasoning payload once CodexFarm captures it.

- Observation: the real blind spot was not just key-only matching; it was wrapper semantics.
  Evidence: CodexFarm classified trace events by top-level `event["type"]` and dictionary key names, so `{"type":"item.completed","item":{"type":"reasoning"}}` produced `reasoning_event_count=0` even though the reasoning hint was present as the nested value `item.type`.

- Observation: fixing the classifier in `run_codex_exec`-side trace persistence is the right boundary.
  Evidence: worker runs, CLI `one`, and other CodexFarm execution surfaces all flow through `src/codex_farm/codex_exec.py`, so a single classifier fix restores recipeimport and any other consumer that reads `.trace.json`.

## Decision Log

- Decision: Fix the bug in the shared `CodexFarm` repository instead of adding a recipeimport-side fallback parser.
  Rationale: recipeimport only sees persisted `.trace.json` artifacts. If the upstream artifact omits reasoning events, any repo-local fallback would need to reparse raw stdout that recipeimport does not own. The clean fix is to repair trace classification where the artifact is created.
  Date/Author: 2026-03-05 / Codex

- Decision: Make trace hint matching value-aware and keep an explicit `item.completed` special case.
  Rationale: value-aware matching catches current nested shapes and future wrappers that move reasoning/action labels into string values, while the explicit `item.completed` branch protects the known failing Codex event shape directly.
  Date/Author: 2026-03-05 / Codex

- Decision: Add the regression at `_persist_trace_artifact(...)` rather than only through a higher-level subprocess test.
  Rationale: this is the boundary that actually writes `reasoning_events`, so the regression stays focused on the classification logic and avoids unnecessary subprocess scaffolding.
  Date/Author: 2026-03-05 / Codex

## Context and Orientation

recipeimport reads CodexFarm telemetry and trace artifacts after benchmark or stage runs. The relevant consumer code is in `cookimport/cli.py`, where prompt-log post-processing resolves `trace_path`, loads the `.trace.json` artifact, and extracts a short excerpt from `thinking_trace.reasoning_events`.

The trace artifact itself is not produced in this repository. It is written by the sibling checkout at `/home/mcnal/projects/shared/CodexFarm`, specifically `src/codex_farm/codex_exec.py`. That file parses Codex JSONL stdout, classifies likely action and reasoning events, persists `.trace.json`, and records summary counts/types into `codex_exec_activity.csv`.

The failing event shape was:

    {"type":"item.completed","item":{"type":"reasoning","text":"Decision made."}}

Before the fix, CodexFarm only matched reasoning by top-level event type names and dictionary keys. Because `reasoning` appeared as a nested string value (`item.type`) instead of a key, the artifact recorded `reasoning_events=[]`.

## Plan of Work
