# Remove `prev_text` And `next_text` From `AtomicLineCandidate`

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

Today the repo is in a half-clean state: the expensive file-backed line-role payload no longer repeats `prev_text` and `next_text`, but the in-memory `AtomicLineCandidate` model still carries those neighbor fields. That means one line object still secretly depends on surrounding rows, which is convenient in the short term but makes the codebase harder for future AI agents to edit correctly because the dependency is implicit instead of explicit.

After this refactor, `AtomicLineCandidate` will mean “one line and its own metadata only.” Any code that needs neighboring lines will ask for an ordered context explicitly. The user-visible result is not a new feature; it is a cleaner program boundary that makes future prompt changes, cache changes, and line-role heuristics easier to maintain without hidden coupling. A contributor should be able to inspect the candidate model, see no neighbor fields on it, run the focused line-role and preview tests, and confirm that inline context rows, cache identity, and local heuristics still behave correctly through explicit ordered-context helpers.

## Decision Log

- Decision: remove neighbor fields from `AtomicLineCandidate` instead of merely de-emphasizing them in docs.
  Rationale: the user’s stated long-term goal is to optimize for AI-maintainable code, and hidden context on a supposedly single-line object is exactly the kind of implicit coupling that misleads future agents.
  Date/Author: 2026-03-18 / Codex

- Decision: replace embedded neighbor fields with one explicit ordered-context helper surface rather than ad hoc local recomputation at each call site.
  Rationale: deleting `prev_text` and `next_text` but then re-deriving them differently in six places would trade one hidden dependency for six slightly different ones. The repo should end with one obvious way to ask for local context.
  Date/Author: 2026-03-18 / Codex

- Decision: keep this refactor scoped to the candidate model and explicit context plumbing, not to another payload minimization or prompt-schema redesign.
  Rationale: the payload-trim work is already tracked elsewhere. This plan is about removing implicit adjacency from the core data model.
  Date/Author: 2026-03-18 / Codex

- Decision: preserve current cache semantics unless implementation proves they are actively harmful.
  Rationale: if a line’s neighbors change, it is reasonable for cache identity to change too. The important improvement is to make that adjacency dependency explicit in the fingerprint builder rather than hidden on the candidate object.
  Date/Author: 2026-03-18 / Codex

- Decision: use two small parser-owned helpers (`build_atomic_index_lookup(...)` and `get_atomic_line_neighbor_texts(...)`) rather than introducing a new context-window model type.
  Rationale: the repo already used `by_atomic_index` as the explicit ordered-context seam in line-role heuristics, so extending that same concept kept the change small and made the final signatures obvious without inventing another object family.
  Date/Author: 2026-03-18 / Codex

## Context and Orientation

An `AtomicLineCandidate` is the parser-owned record for one atomic line of cookbook text. It lives in `cookimport/parsing/recipe_block_atomizer.py` and now carries only line-local facts: line text, provenance-ish metadata such as `block_id` and `block_index`, and label hints such as `rule_tags`. Neighbor text is no longer embedded on the candidate. Explicit adjacency now comes from `build_atomic_index_lookup(...)` plus `get_atomic_line_neighbor_texts(...)` in the same module.

This matters most in canonical line-role. Canonical line-role is the Codex-backed stage that reviews ordered atomic lines and returns labels such as `RECIPE_TITLE`, `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `KNOWLEDGE`, or `OTHER`. The active single-pass implementation writes one model-facing shard file under `line-role-pipeline/runtime/line_role/workers/*/in/*.json`, sends a short wrapper prompt, validates exact `atomic_index` coverage, and then promotes validated labels. The recent payload trim already removed duplicated neighbor fields from those `in/*.json` files, but the in-memory objects feeding that system still carry adjacency.

The relevant files for this refactor are:

- `cookimport/parsing/recipe_block_atomizer.py`: defines `AtomicLineCandidate` and the main block-to-candidate builder.
- `cookimport/labelstudio/ingest.py`: rebuilds `AtomicLineCandidate` rows for prediction-run / Label Studio flows.
- `cookimport/parsing/label_source_of_truth.py`: rebuilds `AtomicLineCandidate` rows for label-source-of-truth paths.
- `cookimport/llm/prompt_preview.py`: rebuilds `AtomicLineCandidate` rows for prompt preview and related local-only artifacts.
- `cookimport/llm/canonical_line_role_prompt.py`: renders selective inline `ctx:<atomic_index>|prev=...|line=...|next=...` rows for ambiguous targets.
- `cookimport/parsing/canonical_line_roles.py`: owns line-role planning, cache identity, and some adjacency-sensitive heuristics.
- `tests/parsing/test_canonical_line_roles.py`, `tests/llm/test_prompt_preview.py`, and `tests/llm/test_label_phase_workers.py`: the narrowest useful proof surface.

The direct-neighbor consumers still break down into three categories, but they now all use explicit context. First, candidate builders emit only single-line candidates. Second, inline prompt rendering emits selective `ctx:` rows by deriving neighbors from ordered lookup. Third, line-role cache identity and the existing title/knowledge heuristics consult adjacency through ordered context rather than hidden candidate fields. The goal of the plan was not to remove local context from the program; it was to stop hiding that dependency on the candidate model itself.

## Plan of Work

Start in `cookimport/parsing/recipe_block_atomizer.py`. Remove `prev_text` and `next_text` from the `AtomicLineCandidate` model. Do not change the other fields. Introduce an explicit context type or helper near this model so the repo has one canonical way to talk about neighbor context. The final form may be a small dataclass such as `AtomicLineContextWindow`, a helper function such as `context_window_for_index(rows, atomic_index)`, or a small family of helpers, but the result must be clear from signatures alone. A novice reading the code should be able to tell when a function requires ordered context and when it only needs one line.

Then update the four candidate builders that currently populate neighbor fields: `atomize_blocks(...)` in `cookimport/parsing/recipe_block_atomizer.py`, the rebuilders in `cookimport/labelstudio/ingest.py` and `cookimport/parsing/label_source_of_truth.py`, and the two preview rebuilders in `cookimport/llm/prompt_preview.py`. These builders should stop manufacturing `prev_text` / `next_text`. If any of them still need adjacency for a downstream consumer in the same function, keep that ordered source list around separately and pass it to the next helper explicitly.

After the data model is cleaned up, change the inline prompt seam in `cookimport/llm/canonical_line_role_prompt.py`. Today `_render_local_context_rows(...)` reads `candidate.prev_text` and `candidate.next_text` directly. Replace that with an explicit context source. The most maintainable shape is to pass either `by_atomic_index: Mapping[int, AtomicLineCandidate]` or a dedicated context-window lookup callback into `_render_local_context_rows(...)` and any helper it calls. The prompt output should remain semantically identical: ambiguous or outside-recipe targets still get selective `ctx:` rows, but those rows must now be derived from ordered context instead of hidden candidate fields.

Next, update `cookimport/parsing/canonical_line_roles.py`. `_canonical_candidate_fingerprint(...)` must stop reading neighbor text off the candidate object. Instead, make the fingerprint builder derive adjacency from the ordered candidate sequence it already receives. Preserve the current semantic intent: if a line’s neighbors change, the fingerprint should still change. Also inspect any helper that currently relies on nearby text indirectly. Some heuristics such as `_looks_recipe_title_context(...)` already use `by_atomic_index`; those should stay explicit. If any helper still assumes adjacency is on the candidate model, rewrite its signature to accept ordered context or a lookup mapping.

Then update preview reconstruction in `cookimport/llm/prompt_preview.py`. The line-role preview rows must continue to match live behavior, and any inline prompt reconstruction path that needs context windows must derive them from the ordered reconstructed candidates rather than object fields. The important invariant is that preview stays behaviorally aligned with the live line-role runtime after the model refactor.

Finally, update the docs and tests. `docs/04-parsing/04-parsing_readme.md` and `docs/10-llm/10-llm_README.md` should say plainly that `AtomicLineCandidate` is now single-line only and that adjacency for inline context or heuristics is passed explicitly. The focused tests should be rewritten so they no longer instantiate `AtomicLineCandidate(prev_text=..., next_text=...)`. Instead, tests that need local context should build an ordered candidate set and prove that the explicit context helper produces the same `ctx:` output and heuristic decisions as before.

3. `cookimport/llm/canonical_line_role_prompt.py` still emits `ctx:<atomic_index>|prev=...|line=...|next=...` rows when required, but derives them from explicit ordered context rather than candidate attributes.
4. `_canonical_candidate_fingerprint(...)` in `cookimport/parsing/canonical_line_roles.py` still changes when neighbor lines change, but no longer reads adjacency from candidate fields.

Behavioral acceptance means:

2. The line-role runtime still validates exact owned `atomic_index` coverage and still promotes returned labels normally.
3. Prompt preview still reconstructs line-role artifacts successfully.
4. No model-facing line-role shard payload regresses back to embedding duplicated neighbor fields.

Maintainability acceptance means:

1. A reader can identify every adjacency-dependent seam from function signatures instead of hidden object fields.
2. The updated docs describe the boundary accurately: candidate objects are single-line facts, while adjacency is a separate input.
3. No tests or docs encourage future contributors to add `prev_text` / `next_text` back onto `AtomicLineCandidate`.

## Idempotence and Recovery

This refactor is safe to iterate on because it is mostly signature cleanup plus explicit context plumbing. If a first pass removes the fields but breaks a prompt or heuristic, do not reintroduce the fields on the candidate model as a quick fix. Instead, repair the explicit context helper or widen the ordered-context parameter passed to that call site.

The zero-token preview output is written to a fresh timestamped directory, so rerunning it is safe. If cache behavior becomes confusing during implementation, add or update a narrow test around `_canonical_candidate_fingerprint(...)` before changing semantics. That is the recovery path for cache-related regressions.
