# Decouple Prompt Artifact Export From Pipeline Topology

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan is self-contained. It assumes no prior knowledge of the current prompt exporter beyond the repository state at the time the plan was written.

## Purpose / Big Picture

After this change, the prompt artifact exporter should keep working even if cookbook bundling, staging, run layout, or stage sequencing changes substantially. In plain language, the exporter should stop “knowing” that recipe prompts live in `pass1_chunking`, `pass2_schemaorg`, and `pass3_final`, and instead consume a normalized description of the prompt stages that actually ran.

The user-visible proof is that a future run with a different cookbook processing layout can still produce the same benchmark-facing prompt artifacts under `prompts/`: `prompt_request_response_log.txt`, `full_prompt_log.jsonl`, `prompt_type_samples_from_full_prompt_log.md`, and category files. The exact filenames for category files may still vary by stage label, but the exported content and run-manifest artifact pointers remain stable and truthful.

The point of this plan is not to redesign recipe extraction. The recipe data types, processing outputs, and end products can stay almost unchanged. The goal is to isolate prompt artifact export behind a stable adapter contract so the upcoming cookbook refactor only needs to teach one discovery layer how to describe the new run shape.

## Surprises & Discoveries

- Observation: the main coupling problem is no longer just human-facing labels. Discovery, normalization, and rendering are all still fused inside `_build_codex_farm_prompt_response_log()` in `cookimport/cli.py`.
  Evidence: the function both scans raw `raw/llm/<book>/...` directories and writes the final artifacts, while also embedding Codex telemetry, prompt-template reconstruction, trace loading, and category-file naming.

- Observation: the repo already treats `full_prompt_log.jsonl` as the source of truth, but the code that generates it is still tied to today’s CodexFarm layout.
  Evidence: `docs/10-llm/10-llm_README.md` says literal prompt truth lives in `prompts/full_prompt_log.jsonl`, yet the current builder still starts from `pass_dir_map`, `pass_manifest_name_map`, and current manifest file names in `cookimport/cli.py`.

- Observation: downstream tooling depends more on artifact presence and row semantics than on the exporter’s internal implementation.
  Evidence: `scripts/benchmark_cutdown_for_external_ai.py` and `cookimport/bench/followup_bundle.py` mostly consume `full_prompt_log.jsonl`, run-manifest artifact pointers, and a few convenience filenames; they do not need to know how prompt stages were discovered as long as those contracts stay stable.

- Observation: there are two distinct seams and they should not be solved together by accident.
  Evidence: reviewer-facing prompt artifacts can already label merged repair dynamically, but raw storage still lives under legacy pass-slot paths like `raw/llm/<book>/pass2_schemaorg`. Renaming raw storage is a separate refactor from decoupling prompt artifact export.

- Observation: the most brittle downstream hardcoding today is not the sample markdown; it is the small set of “special-case file” assumptions in follow-up and cutdown tooling.
  Evidence: `cookimport/bench/followup_bundle.py` still resolves pass4 artifacts through fixed names like `prompt_task4_pass4_knowledge.txt`, and `scripts/benchmark_cutdown_for_external_ai.py` also names `prompt_task4_pass4_knowledge.txt` explicitly for bundle reporting.

- Observation: prompt category manifest entries may be absolute paths in current outputs; downstream loaders need to tolerate both absolute and relative manifest rows.
  Evidence: existing exporter writes absolute path rows to `prompt_category_logs_manifest.txt`, so pass4 compatibility lookups were implemented with absolute-aware manifest parsing in both follow-up and cutdown tooling.

## Context and Orientation

The current prompt artifact exporter lives in `cookimport/cli.py`. The main function is `_build_codex_farm_prompt_response_log()`. It reads raw LLM inputs and outputs from a prediction or stage run under `raw/llm/<workbook>/...`, reconstructs rendered prompts from telemetry and run assets, writes `prompts/full_prompt_log.jsonl`, and then writes convenience artifacts such as `prompt_request_response_log.txt`, `prompt_type_samples_from_full_prompt_log.md`, and category files like `prompt_task1_pass1_chunking.txt`.

In this repository, a prompt artifact means a file under a run’s `prompts/` directory that explains what the LLM saw or returned. The most important one is `full_prompt_log.jsonl`, which is a JSON Lines file where each non-empty line is one normalized prompt call row. A JSON Lines file is a text file containing one complete JSON object per line. It is easy for scripts to stream and parse without loading the whole file at once.

The current exporter has three jobs mixed together. Discovery means finding which prompt stages ran and where their input/output/manifests live. Normalization means converting those raw files and telemetry rows into one stable row shape for `full_prompt_log.jsonl`. Rendering means turning the normalized rows into human-facing helper files such as sample markdown and category dumps.

The main downstream consumers are:

`scripts/benchmark_cutdown_for_external_ai.py`. This script packages benchmark output for external AI review. It prefers `full_prompt_log.jsonl` as the source of truth and uses the convenience files only when useful for reviewers or when older runs are missing the JSONL.

`cookimport/bench/followup_bundle.py`. This module builds follow-up bundles and still has a few fixed assumptions about pass4 artifact filenames such as `prompt_task4_pass4_knowledge.txt`.

`cookimport/cli.py` itself. Stage runs, benchmark runs, and Label Studio prediction runs all write prompt artifact pointers into `run_manifest.json` via artifact keys such as `codexfarm_prompt_request_response_txt`, `codexfarm_full_prompt_log_jsonl`, and `codexfarm_prompt_type_samples_from_full_prompt_log_md`.

The key risk for the upcoming cookbook refactor is that discovery is currently tied to the present CodexFarm topology. The exporter still assumes:

- raw prompt stages live under specific directories such as `pass1_chunking`, `pass2_schemaorg`, `pass3_final`, `pass4_knowledge`, and `pass5_tags`,
- manifests use current names such as `llm_manifest.json`, `pass4_knowledge_manifest.json`, and `pass5_tags_manifest.json`,
- stage ordering can be inferred from legacy pass numbers,
- some downstream tools can safely special-case known files such as `prompt_task4_pass4_knowledge.txt`.

The plan below removes those assumptions from the exporter itself while leaving the external artifact contract stable.

## Plan of Work

Start by extracting the prompt artifact code from `cookimport/cli.py` into a dedicated module. A good target is `cookimport/llm/prompt_artifacts.py` or `cookimport/bench/prompt_artifacts.py`. The exact module path matters less than the boundary: prompt artifact export must become a reusable service, not an embedded helper inside the CLI giant module.

Define two versioned normalized data shapes in that module. The first is a prompt stage descriptor. A prompt stage descriptor is a small record that tells the exporter what stage exists, how to sort it, what human label to use, which files or directories contain its inputs and outputs, which pipeline id produced it, and which manifest or run-asset files belong to it. The second is a prompt call record. A prompt call record is one normalized `full_prompt_log.jsonl` row with stable keys such as `call_id`, `legacy_pass`, `stage_key`, `stage_label`, `pipeline_id`, `request_messages`, `parsed_response`, `request_telemetry`, `thinking_trace`, and source file paths.

Make the exporter consume only those normalized structures. Rendering code must stop inspecting raw directories directly. Instead, rendering should accept `list[PromptStageDescriptor]` and `iterator[PromptCallRecord]`, then write the same artifact set as today. This includes `prompt_request_response_log.txt`, `full_prompt_log.jsonl`, `prompt_type_samples_from_full_prompt_log.md`, and prompt category files. Category filenames should be derived from descriptor policy fields like `artifact_stem` rather than current pass names.

After the normalized contract exists, implement a CodexFarm adapter. This adapter is the only layer that should know current raw paths and manifest names. It should live alongside the new exporter module and expose one discovery function such as `discover_codexfarm_prompt_stages(run_root: Path) -> list[PromptStageDescriptor]`. That function should encapsulate everything that is currently hardcoded in `_build_codex_farm_prompt_response_log()`: raw LLM stage directory detection, manifest resolution, process-run telemetry linkage, run-assets loading, and any legacy pass numbering.

Once the CodexFarm adapter exists, move the current row-building logic into a normalization function that converts one descriptor into zero or more normalized prompt call records. This is where prompt-template reconstruction, telemetry enrichment, trace lookup, context-block extraction, and parsed response embedding should live. Keep the current row keys stable unless the plan explicitly says to add a new key. Do not remove the existing top-level fields used by cutdown tooling unless all consumers are updated first.

Next, update the current callers in `cookimport/cli.py` to call the new module. The CLI should no longer implement prompt export itself. It should only say “build prompt artifacts for this run root” and then store the returned file paths in the run manifest as it does today. The run-manifest keys should stay unchanged so external consumers remain stable.

Then audit and soften downstream “special file” assumptions. The immediate targets are `cookimport/bench/followup_bundle.py` and `scripts/benchmark_cutdown_for_external_ai.py`. Where they truly need one known file, keep compatibility fallback logic. Where they only need prompt-stage information in general, switch them to read normalized row metadata or the category manifest instead of naming `prompt_task4_pass4_knowledge.txt` directly. The design goal is that a future non-pass prompt stage can still be included in bundles without another script edit.

Finally, add synthetic-topology tests. The current suite already covers the standard pass layout and one merged-repair label variant. Add at least one test fixture that describes a non-pass topology through descriptors only, for example `chunking -> repair -> tags` or `book segmentation -> recipe repair`, with no raw `passN_*` naming. Prove that the renderer still writes truthful prompt artifacts and stable JSONL rows.

## Milestones

### Milestone 1: Define the normalized contract and pull rendering out of `cookimport/cli.py`

At the end of this milestone, the repo contains a dedicated prompt artifact module with explicit data shapes for prompt stages and prompt call rows. A novice contributor can open one module and understand the exporter’s public surface without reading the entire CLI file.

The proof for this milestone is local and structural. The new module exists, `cookimport/cli.py` calls into it, and the focused prompt artifact tests still pass for the current layout. No cookbook refactor is required yet. This milestone is about creating the boundary.

### Milestone 2: Implement the CodexFarm discovery adapter and move current normalization logic behind it

At the end of this milestone, the current exporter behavior is preserved, but all current raw-layout knowledge lives in one adapter function instead of being spread across rendering and row-writing code. A novice can read one adapter and understand how today’s CodexFarm layout maps into the normalized contract.

The proof for this milestone is behavior. Stage and benchmark prompt artifacts generated from the current layout remain unchanged in meaning, and the same targeted prompt artifact tests pass. If filenames differ only where the new stage-label policy already intended them to differ, record that explicitly.

### Milestone 3: Make downstream tooling consume the stable contract rather than fixed convenience filenames where possible

At the end of this milestone, cutdown packaging and follow-up bundles no longer assume that prompt-stage identity is encoded in one fixed category filename. They should prefer `full_prompt_log.jsonl`, run-manifest artifact pointers, and category manifests, with compatibility fallbacks kept only where necessary.

The proof for this milestone is that existing cutdown and follow-up tests still pass, and at least one new synthetic case proves the tools can tolerate a stage name that does not look like `pass4_knowledge`.

### Milestone 4: Prove the exporter survives a synthetic non-pass topology

At the end of this milestone, the prompt artifact system has one test path where no current raw `pass1/pass2/pass3` directory names are used. The exporter should still render prompt samples, category files, and JSONL rows from normalized descriptors.

This is the real de-risking milestone for the user’s upcoming cookbook refactor. If this milestone passes, a large staging or bundling rewrite can focus on writing one new discovery adapter instead of reopening exporter internals.

## Validation and Acceptance

Acceptance is behavior-first.

First, current behavior must remain intact for today’s CodexFarm layout. Running the focused prompt artifact tests should still produce the same core artifacts and the same stable JSONL row semantics. Existing run-manifest artifact keys must still point to `prompt_request_response_log.txt`, `full_prompt_log.jsonl`, and `prompt_type_samples_from_full_prompt_log.md`.

Second, the exporter must become topology-agnostic. This does not mean every downstream script must stop caring about known convenience files on day one. It means the exporter itself can render artifacts from normalized stage descriptors without reading current raw `passN_*` directory names directly.

Third, at least one synthetic topology must be proven. The synthetic test should show that prompt artifacts still render when stage identities are provided through descriptors such as `chunking`, `merged_repair`, and `tags` without assuming those are hardcoded pass slots. If the exporter still needs `pass_dir_map` to function, the plan has not succeeded.

Fourth, downstream tooling must remain operational. `scripts/benchmark_cutdown_for_external_ai.py` and `cookimport/bench/followup_bundle.py` may keep compatibility fallbacks, but they must not regress existing prompt artifact bundle generation.

## Idempotence and Recovery

This plan is additive and can be executed incrementally. The safest path is to keep the current CLI entry points and run-manifest artifact keys stable while moving internals behind the new boundary. That makes each milestone rerunnable: if a partial refactor fails, revert the new module wiring and keep the existing artifacts intact.

Do not remove compatibility fields like `pass`, `legacy_pass`, or current artifact keys during this plan. Those removals would widen scope and complicate rollback. The recovery path for each milestone is to restore the old caller wiring while leaving any new normalized data classes and tests in place if they are harmless and self-contained.

Avoid destructive changes to existing benchmark output roots. Validation should always run against fresh timestamped test output or temporary directories.

The normalized stage descriptor introduced by this plan should end in a shape close to the following. The exact class names may differ, but the meaning should not:

    PromptStageDescriptor(
        schema_version="prompt_stage_descriptor.v1",
        stage_id="merged_repair",
        legacy_pass="pass2",

        stage_label="Merged Repair",

        output_dir="/abs/path/to/out",

The normalized prompt call row should remain compatible with today’s `full_prompt_log.jsonl`, while making stage identity explicit:

    {
      "schema_version": "prompt_call_record.v1",
      "legacy_pass": "pass2",
      "stage_key": "merged_repair",
      "stage_label": "Merged Repair",
      "pipeline_id": "recipe.merged-repair.compact.v1",
      "call_id": "r0000",
      "request_messages": [{"role": "user", "content": "..."}],
      "parsed_response": {"result": "..."}
    }

The module boundary after refactor should read like this in prose:

`discover_*` functions know current run layout.

`collect_*` functions know how to normalize prompt rows.

`render_*` functions know how to write human-facing artifacts.

No one function should do all three jobs at once.
