# Split post-benchmark Oracle review into separate quality and token lanes

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

Right now a benchmark run writes one `upload_bundle_v1/` and launches one Oracle review chat with one broad prompt. For the March 22, 2026 `20:16:29` Salt Fat benchmark at `data/golden/benchmark-vs-golden/2026-03-22_20.16.29/single-book-benchmark/saltfatacidheatcutdown`, that single turn-1 Oracle upload was staged as four files and Oracle estimated about `443,667` input tokens before answering. That one chat has to reason about two different jobs at once: how to push quality much higher and how to reduce spend.

After this change, the same benchmark bundle should launch two detached Oracle reviews instead of one. The `quality` lane should get a quality-focused prompt plus a quality-focused packet and should concentrate on the shortest path toward `>95%` benchmark quality. The `token` lane should get a spend-focused prompt plus a lighter runtime/cost packet and should concentrate on the sharpest token reductions that preserve at least the current quality. A human should be able to run the existing benchmark flow or `cookimport bench oracle-upload <bundle>` and see two separate launch directories, two separate conversation URLs, and two separate logs under `upload_bundle_v1/.oracle_upload_runs/`.

The proof is local first. Dry-run and unit tests should show that each lane renders the correct prompt, the correct staged files, and the correct lane metadata. Live Oracle calls remain explicitly approval-gated because they spend real tokens.

## Surprises & Discoveries

- Observation: the current single-lane Oracle review is already expensive before it even answers.
  Evidence: `upload_bundle_v1/.oracle_upload_runs/2026-03-22_20.21.25/oracle_upload.log` says `Launching browser mode (gpt-5-pro) with ~443,667 tokens.`

- Observation: the current first-pass Oracle packet is a starter-pack subset, not the full payload, and it is still large.
  Evidence: the same log says the repo trimmed `upload_bundle_payload.jsonl` to `17` starter-pack rows and `1,091,494` bytes before browser upload.

- Observation: the current benchmark is nowhere near `95%` overall quality, so the quality lane must focus on the blockers to that target rather than pretending the run is close.
  Evidence: [single_book_summary.md](/home/mcnal/projects/recipeimport/data/golden/benchmark-vs-golden/2026-03-22_20.16.29/single-book-benchmark/saltfatacidheatcutdown/single_book_summary.md) reports `strict_accuracy=0.647340` and `macro_f1_excluding_other=0.777327` for `codexfarm`.

- Observation: recipe boundaries are not the dominant remaining problem in this run.
  Evidence: `upload_bundle_index.json.analysis.structure_label_report` reports `boundary_exact_ratio=1.000000` for both codex and baseline, while `nonrecipe_core` average F1 is only `0.578081`.

- Observation: most scored lines live outside active recipe spans, which is exactly where the quality lane should spend its time.
  Evidence: [upload_bundle_overview.md](/home/mcnal/projects/recipeimport/data/golden/benchmark-vs-golden/2026-03-22_20.16.29/single-book-benchmark/saltfatacidheatcutdown/upload_bundle_v1/upload_bundle_overview.md) reports `outside_share_of_scored_lines=0.839018` and outside-span delta `0.118699` versus inside-span delta `0.004237`.

- Observation: the benchmark’s token problem is overwhelmingly the knowledge stage.
  Evidence: `upload_bundle_index.json.analysis.call_inventory_runtime.summary` reports `33,699,633` total tokens with `nonrecipe_knowledge_review_token_share=0.8535`, and [prompt_budget_summary.json](/home/mcnal/projects/recipeimport/data/golden/benchmark-vs-golden/2026-03-22_20.16.29/single-book-benchmark/saltfatacidheatcutdown/codexfarm/prompt_budget_summary.json) reports `knowledge.tokens_total=28,762,251`.

  Evidence: the same prompt-budget summary reports `wrapper_overhead_tokens=14,637,956` for knowledge, `1,586,371` for line role, and `1,055,336` for recipe correction.

- Observation: the current code already has the right abstraction seam for lane-specific temporary staging, and the checked bundle itself does not need to mutate on disk.
  Evidence: [cookimport/bench/oracle_upload.py](/home/mcnal/projects/recipeimport/cookimport/bench/oracle_upload.py) already stages temporary browser-upload files under `.oracle_upload_runs/<timestamp>/staging/`, already supports payload subsetting, and [docs/07-bench/07-bench_README.md](/home/mcnal/projects/recipeimport/docs/07-bench/07-bench_README.md) explicitly says transport sharding is upload-time glue and `upload_bundle_v1` should stay unchanged on disk.

- Observation: the current single detached Oracle run is still the only launch Oracle knows about, and its follow-up worker is blocked on turn 1.
  Evidence: `oracle_upload_status.json` still says `status: running`, while `oracle_auto_followup.json` says `status: waiting_for_turn_1`.

- Observation: logical artifact paths were the stable seam for lane packet selection; payload row numbers were too bundle-specific.
  Evidence: the implemented selectors now build profile packets by `path` lookup inside `upload_bundle_payload.jsonl`, while tests synthesize arbitrary row ordering and still get the right staged subsets.

  Evidence: the implementation had to resolve the active template path at prompt-build time instead of capturing it only inside the profile registry, otherwise the editable-template prompt test stopped exercising the patched file.

## Decision Log

- Decision: implement two explicit review profiles named `quality` and `token`.
  Rationale: the user asked for one lane focused on pushing quality above `95%` and one lane focused on reducing token spend. Treating these as named review profiles makes prompt selection, staging, logging, and CLI replay concrete.
  Date/Author: 2026-03-22 / Codex

- Decision: keep `upload_bundle_v1/` immutable on disk and build lane-specific staged packets only inside `.oracle_upload_runs/<timestamp>-<profile>/staging/`.
  Rationale: the current bundle is the stable on-disk contract, and the repo already treats upload-time sharding as temporary transport glue. Lane-specific staging belongs in the detached launch dir, not in the checked bundle.
  Date/Author: 2026-03-22 / Codex

- Decision: both lanes will share the same three common logical files from the bundle root, but each lane will get its own tiny lane brief and its own lane-specific `upload_bundle_payload.jsonl` subset.
  Rationale: `upload_bundle_overview.md` and `upload_bundle_index.json` already provide the shared navigation contract. The expensive part is the payload. A small lane brief steers the model without forcing it to rediscover the benchmark’s main signal.
  Date/Author: 2026-03-22 / Codex

- Decision: the `quality` lane packet should be accuracy-first and case-oriented, but still follow the “starter pack plus follow-up” philosophy.
  Rationale: the current single-lane upload is already expensive. The quality lane should not get the entire payload by default. It should get the metrics, triage, per-recipe, explicit-escalation, and prompt-sample rows that make a first-pass quality diagnosis possible, then request narrow follow-up when necessary.
  Date/Author: 2026-03-22 / Codex

- Decision: the `token` lane packet should be materially smaller than the current starter pack and should be built around spend summaries, prompt-budget rollups, and runtime examples rather than wrong-line context dumps.
  Rationale: the token lane’s job is to explain where `33.7M` benchmark tokens are going and which cuts are safest. It does not need the full quality casebook to do that.
  Date/Author: 2026-03-22 / Codex

- Decision: keep the existing parse-friendly `Requested follow-up data` shape and shared turn-2 follow-up machinery.
  Rationale: the profile-specific intent already lives in the turn-1 prompt and the saved conversation. Reusing the existing request schema avoids growing `cf-debug` in the same change. The first version of this feature should split turn 1 cleanly before it invents new follow-up output types.
  Date/Author: 2026-03-22 / Codex

- Decision: auto-launch and manual `bench oracle-upload` should default to launching both lanes, but the CLI must also allow replaying only `quality` or only `token`.
  Rationale: the desired product behavior is “call two oracles, not just one,” but debugging and reruns need single-lane control.
  Date/Author: 2026-03-22 / Codex

- Decision: keep the current generic prompt file as a compatibility alias for the `quality` lane, and add a new token-lane prompt file rather than deleting the old one outright.
  Rationale: existing tests and operator expectations already point at `benchmark.oracle-upload.prompt.md`. Mapping that file to the quality lane preserves a sensible default while the new token-lane path lands beside it.
  Date/Author: 2026-03-22 / Codex

- Decision: live Oracle launches remain approval-gated.
  Rationale: the repo instructions explicitly forbid AI-agent token spend without explicit user approval. Dry-run and unit tests are the non-spend proof story for implementation.
  Date/Author: 2026-03-22 / Codex

## Outcomes & Retrospective

This plan is now implemented locally. Benchmark-side Oracle upload no longer treats the first review as one generic chat. The uploader now launches two explicit review lanes by default: `quality` for the path to `>95%`, and `token` for spend reduction with quality guardrails.

The most important implementation choice was to keep `upload_bundle_v1/` immutable and move all lane shaping into temporary staged packets under `.oracle_upload_runs/<timestamp>-<profile>/staging/`. That preserved the existing checked-bundle contract while still letting each Oracle lane see a different prompt, a different lane brief, and a different logical-path payload subset.

The proof story is local and approval-safe. Unit coverage now exercises prompt rendering, profile selection, staged payload subsets, background launch metadata, and CLI default behavior for both lanes. No live Oracle upload was run because the repo rules still require explicit approval before spending real Oracle tokens.

## Context and Orientation

The current benchmark-side Oracle flow has four moving parts:

- [cookimport/cli.py](/home/mcnal/projects/recipeimport/cookimport/cli.py) starts the detached Oracle upload after a benchmark writes `upload_bundle_v1/`, and it also exposes the manual `cookimport bench oracle-upload` and `cookimport bench oracle-followup` commands.
- [cookimport/bench/oracle_upload.py](/home/mcnal/projects/recipeimport/cookimport/bench/oracle_upload.py) resolves the bundle target, renders the prompt, stages any browser-safe payload subset, launches Oracle, and writes `oracle_upload.log`, `oracle_upload.json`, and `oracle_upload_status.json`.
- [llm_pipelines/prompts/benchmark.oracle-upload.prompt.md](/home/mcnal/projects/recipeimport/llm_pipelines/prompts/benchmark.oracle-upload.prompt.md) is the current editable turn-1 review prompt.
- `upload_bundle_v1/` is the stable three-file bundle contract:
  - `upload_bundle_overview.md`
  - `upload_bundle_index.json`
  - `upload_bundle_payload.jsonl`

The current uploader already knows how to build a smaller browser-safe payload subset from `upload_bundle_index.json.navigation.row_locators`. The relevant helpers are `_collect_browser_safe_payload_rows(...)`, `_prepare_browser_safe_starter_upload_inputs(...)`, and `_prepare_browser_upload_inputs(...)` in [cookimport/bench/oracle_upload.py](/home/mcnal/projects/recipeimport/cookimport/bench/oracle_upload.py). Today those helpers always build one generic packet and one generic prompt.

For the concrete March 22, 2026 `20:16:29` Salt Fat run, the important benchmark facts are:

- Current Codex quality is `strict_accuracy=0.647340` and `macro_f1_excluding_other=0.777327`.
- Boundary exactness is already `1.0`, so the remaining quality losses are mainly label-choice and routing mistakes, especially `KNOWLEDGE` versus `OTHER`.
- `83.9%` of scored lines live outside active recipe spans.
- The benchmark spent `33,699,633` tokens in total, with `28,762,251` of them in knowledge.
- The single current Oracle turn-1 upload was estimated at about `443,667` tokens.

That evidence dictates the split:

- The quality lane must be accuracy-first, nonrecipe-heavy, and case-oriented.
- The token lane must be spend-first, overhead-aware, and quality-guardrailed.

This plan intentionally does not require widening `upload_bundle_v1_render.py` or `upload_bundle_v1_existing_output.py` in the first pass. The on-disk bundle already contains enough payload rows for both lanes. If implementation uncovers a genuinely missing artifact path, record that in `Surprises & Discoveries` and only then widen the bundle model.

## Milestones

### Milestone 1: Define review profiles and prompt templates

At the end of this milestone, the repo should have explicit Oracle review-profile definitions for `quality` and `token`, each with a stable profile id, display label, prompt-template path, and lane-brief renderer. The current single generic prompt should stop being an implicit global constant. The proof is that unit tests can render both prompts from one bundle target and see profile-specific language, section names, and metrics.

### Milestone 2: Build lane-specific staged packets from one immutable bundle

At the end of this milestone, the uploader should be able to stage two different browser-upload packets from the same `upload_bundle_v1/` without editing that bundle on disk. The `quality` lane should stage a case-and-metrics packet. The `token` lane should stage a smaller runtime-and-cost packet. The proof is a dry-run or unit-test staging check that writes both lane briefs and both lane-specific payload subsets under separate launch dirs.

### Milestone 3: Launch, log, and summarize two detached Oracle runs

At the end of this milestone, benchmark auto-launch and manual CLI replay should be able to launch both profiles by default. Each lane should write its own `oracle_upload.log`, `oracle_upload.json`, `oracle_upload_status.json`, and its own follow-up worker log/status. The proof is local in mocked-launch tests and CLI summary output. Live browser runs remain approval-gated.

### Milestone 4: Document the dual-lane contract and keep compatibility sane

At the end of this milestone, the bench docs should explain that post-benchmark Oracle upload now launches `quality` and `token` review lanes, how to replay only one lane, and where to find the separate logs. The compatibility story for the existing prompt file and older single-run mental model should be written down so a future operator is not surprised.

## Validation and Acceptance

Local acceptance must prove four things.

First, prompt selection must be correct. A test should render the `quality` prompt and the `token` prompt from the same bundle target and confirm that they differ in objective, section headings, and lane-brief filename, while still preserving the parse-friendly `Requested follow-up data` contract.

Second, lane packet staging must be correct. A test should synthesize a small `upload_bundle_v1` fixture with enough payload rows to represent both lanes and then assert that the `quality` lane receives only the planned quality rows, the `token` lane receives only the planned token rows, and both staged payload files keep the original logical `path` values unchanged.

Third, detached launch plumbing must be correct. A mocked-launch test should verify that `--profile all` creates two launch dirs, records `review_profile` in metadata and status payloads, and starts one follow-up worker per launched profile. The CLI summary does not need to be pretty; it must be unambiguous.

Fourth, compatibility must be sane. Existing single-profile tests should still pass when replaying only `quality`, and the old prompt file should still behave as the quality-lane template rather than becoming dead configuration.

The minimum local acceptance bar is:

- `pytest -q tests/bench/test_benchmark_oracle_upload.py` passes.
- New or expanded tests cover:
  - profile-specific prompt rendering
  - path-based lane packet selection
  - multi-launch metadata and status writing
  - CLI `--profile quality|token|all` behavior
- No live Oracle browser call is required for merge.

An approved live acceptance run should additionally show:

- two launch dirs under one bundle’s `.oracle_upload_runs/`
- one log and one status file per lane
- distinct `review_profile` metadata
- two conversation URLs or session ids
- a materially smaller token-lane turn-1 estimate than the current single generic upload

## Idempotence and Recovery

This change should be safe to retry. The lane packet builder writes only into a fresh timestamped launch dir, so rerunning `cookimport bench oracle-upload ... --profile all` should never overwrite the checked `upload_bundle_v1/` or an older launch. If a previous launch exists, it should simply remain beside the new one.

If one lane fails to stage because a payload path is missing, that lane should fail clearly without corrupting the other lane. The other lane may still proceed. Record any missing-path surprise in the lane brief and in the raised error message so the operator knows whether the failure is a true repo bug or just sparse bundle evidence.

If live Oracle launch fails after staging succeeds, the failure should remain isolated to the lane’s own launch dir. The other lane should still preserve its prompt, staged files, and metadata for debugging.
