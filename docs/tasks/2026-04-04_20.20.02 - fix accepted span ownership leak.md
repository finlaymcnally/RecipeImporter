Problem statement

Accepted recipe spans can currently keep `NONRECIPE_CANDIDATE` labels on owned blocks.
That leaks nonrecipe routing authority past recipe ownership and can abort offline benchmark or stage runs before recipe correction/nonrecipe finalize.

Acceptance criteria

- Accepted recipe-span blocks no longer retain nonrecipe route labels.
- Rejected recipe-local spans still route back to nonrecipe as before.
- A focused regression test fails before the fix and passes after it.

Verification steps

- `. .venv/bin/activate && pytest tests/parsing/test_recipe_span_grouping.py tests/staging/test_nonrecipe_stage.py`

Evidence

- Before fix: `. .venv/bin/activate && pytest tests/parsing/test_recipe_span_grouping.py` -> `1 failed, 5 passed`
- After fix: `. .venv/bin/activate && pytest tests/parsing/test_recipe_span_grouping.py tests/staging/test_nonrecipe_stage.py` -> `20 passed`

Constraints / gotchas

- Recipe ownership must remain the authority boundary.
- Do not silently widen nonrecipe routing; only explicit recipe divestment may hand owned blocks back to nonrecipe.

Decision

- Normalize accepted-span nonrecipe-labelled blocks into `RECIPE_NOTES` at the recipe-boundary seam, symmetric with the existing rejected-span route normalization.

Rollback / compensation plan

- Revert this change if accepted-span normalization corrupts recipe-local scoring, then revisit with a separate owned-vs-route contract split.
