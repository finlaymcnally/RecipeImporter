# Task: benchmark resume scope mismatch

## Problem statement

Benchmark upload could fail with `Existing project uses task_scope=freeform-spans` when local stale manifests were checked during resume even though the run had created a new Label Studio project.

## Acceptance criteria

- Resume metadata is only loaded when the target Label Studio project already exists before this run.
- New-project uploads do not fail on stale local `manifest.json` `task_scope` values.
- Benchmark upload auto-recovers scope collisions for auto-named projects by creating a deduped project title (`-1`, `-2`, ...) instead of failing.
- Regression coverage exists for the new-project + stale-manifest case.

## Verification steps

Run from repo root:

    source .venv/bin/activate
    pytest -q tests/test_labelstudio_ingest_parallel.py tests/test_labelstudio_benchmark_helpers.py

## Evidence

- `pytest -q tests/test_labelstudio_ingest_parallel.py tests/test_labelstudio_benchmark_helpers.py`
- Result: `49 passed, 2 warnings in 3.30s`

## Constraints / gotchas and decision

- Constraint: preserving existing `--resume` behavior for genuinely pre-existing projects is required.
- Constraint: interactive benchmark upload does not prompt for project name, so "use a new project name" must be handled automatically in this path.
- Decision: gate resume metadata loading with `had_existing_project`, and add benchmark-only fallback (`auto_project_name_on_scope_mismatch`) that creates a deduped project title on scope mismatch.

## Rollback / compensation plan

- Revert the resume gate/fallback in `cookimport/labelstudio/ingest.py`, remove benchmark wiring in `cookimport/cli.py`, and remove regression tests in `tests/test_labelstudio_ingest_parallel.py` and `tests/test_labelstudio_benchmark_helpers.py`.
