## Structured Restatement Of User Desire

The repository needs a full worker-runtime reset, not another local optimization on top of a bad design. The current live Codex Exec stages burn tokens on repo-controlled leasing loops, repeated workspace rereads, and shell-mediated JSON movement that do not add semantic value. This plan must replace that design with one cleaner runtime shape and remove the old one completely. The new live path should give each worker a fixed immutable assignment up front, let the worker do the semantic work it is good at, let deterministic code validate outputs afterward, and use only one narrow repair path for failures. This is explicitly burn-the-boats work. There is no compatibility mode, no legacy fallback, no dual-path migration contract, and no obligation to preserve the old leased or phase-ledger stories for future readers. The user is the only operator and wants one better runtime, not two runtimes plus adapters.

## Purpose / Big Picture

After this change, a user can run the live recipe, knowledge, or line-role Codex Exec stages and get a runtime that behaves like a compact semantic batch worker rather than a tiny local queue operator. Each worker will read one assignment file, read the immutable shard or task inputs it owns, write one result file per owned unit, and then stop. Deterministic repo code will validate those outputs, promote valid results, and build a bounded repair assignment only for the small subset that actually failed validation. The happy path will no longer depend on repo-controlled lease advancement, repeated `current_packet.json` rereads, or `CURRENT_PHASE*` loops.

The user-visible proof is straightforward. Prompt-budget summaries should show far less wrapper overhead from queue choreography, even if the worker surface is still multi-file at this stage. Activity traces should stop showing dozens of repeated manifest and control-file rereads for one worker session. Worker roots should contain fixed assignment artifacts such as `assigned_shards.json` or `assigned_tasks.json`, immutable `in/*.json` files, and stable `out/*.json` targets rather than active lease state such as `current_packet.json`, `current_result_path.txt`, `packet_lease_status.json`, `CURRENT_PHASE.md`, or repo-authored queue choreography files. A novice should be able to inspect one worker root and immediately understand the whole job: these are the owned units, these are the outputs to write, these are the validation rules, and this optional repair batch exists only if some units failed. The later one-file follow-on plan simplifies that worker root further; this plan’s job is to delete queue choreography first.

  Evidence: `recipe_refine` used about `3.25M` tokens over only `5` calls with `75` command executions, and `line_role` used about `1.95M` tokens over `5` calls with `41` command executions.

  Evidence: The traces show repeated cycles of reading worker manifests, reading current packet or current phase files, reading result-path control files, writing JSON via shell here-docs, and then rereading queue state to ask the repo what comes next.

  Evidence: The current traces show large JSON payloads both when the worker reads input through `cat ...json` and when it writes output through shell here-docs, so the model repeatedly pays to see payload text that should have remained narrow file I/O.

  Evidence: `docs/10-llm/10-llm_README.md` describes recipe and knowledge as leased-packet workers and line-role as a phase-ledger workspace-worker stage.

- Observation: The recently added hard-cap plan helps but is not sufficient on its own.
  Evidence: Hard caps stop silent shard explosions, but they do not remove the repeated lease rereads, shell-churn loops, or command-heavy JSON transport inside each surviving worker session.

## Context And Orientation

In this repository, a “worker” is one long-lived `codex exec` session operating in a bounded local workspace. A “shard” or “task” is one owned semantic unit such as a recipe shard, a knowledge shard, or a line-role shard. A “lease” means the repo reveals only one active packet at a time and makes the worker reopen control files to discover what comes next. A “phase-ledger workspace” means the repo exposes a current phase file and the worker keeps re-opening phase state while applying changes to a work ledger. Both shapes are forms of queue-shaped control. Both are the problem this plan deletes.

The current live stage owners are spread across three main areas:

- Recipe runtime lives under `cookimport/llm/recipe_stage/` with important behavior still centralized in `cookimport/llm/recipe_stage_shared.py`.
- Knowledge runtime lives under `cookimport/llm/knowledge_stage/` with active workspace behavior in `runtime.py`, `workspace_run.py`, and `recovery.py`.
- Line-role runtime lives under `cookimport/parsing/canonical_line_roles/` with worker surfaces backed by `cookimport/parsing/line_role_workspace_tools.py`.
- Shared worker setup, telemetry, and workspace supervision live in `cookimport/llm/codex_exec_runner.py` and `cookimport/llm/phase_worker_runtime.py`.
- Prompt-budget and benchmark reporting live in `cookimport/llm/prompt_budget.py`, `cookimport/runs/stage_observability.py`, Label Studio benchmark helpers, and related analytics readers.

The current live runtime shapes differ in surface detail but share one bad property. Recipe and knowledge ask the worker to reopen repo-controlled lease files to discover the next packet or repair. Line-role asks the worker to operate a phase-ledger workspace with repo-controlled current-phase state. In all three cases, the model ends up paying for repeated control-file reads, command invocations, and shell-visible JSON movement that add little or no semantic value.

The new runtime shape in this plan is called a “fixed-assignment worker.” A fixed-assignment worker receives all of its owned units at the beginning of the session through one immutable assignment file. The assignment file names every owned input and every required output path. The worker reads that assignment once, reads the owned inputs directly, writes one output per owned unit, and exits. Deterministic repo code then validates every output. If any outputs fail validation, the repo builds one explicit repair assignment containing only the failed owned units plus their exact structural errors. That repair assignment is a second bounded runtime step, not a continuation of the original queue loop.

This plan assumes the semantic authority decisions from the existing burn-the-boats stage plans remain in force. Recipe still owns recipe plausibility, tags, and ingredient-to-step mapping. Knowledge still owns `knowledge` versus `other` and grouping semantics. Line-role still owns line labels. Deterministic code still owns evidence packaging, allowed-shape validation, ownership checks, promotion, and telemetry. Only the worker-control shape changes here.

## Plan Of Work

The first step is to define one shared fixed-assignment contract in the common LLM runtime layer and make every live Codex Exec stage use it. In practical terms, `cookimport/llm/codex_exec_runner.py` and the stage-specific runtime modules must stop advertising current-packet or current-phase control files as the happy path. The common happy-path worker root should contain a stable `worker_manifest.json`, one immutable assignment file such as `assigned_shards.json` or `assigned_tasks.json`, immutable `in/*.json` files, any narrow static hints, and stable `out/*.json` target paths. The shared prompt template must describe this contract directly: read the assignment, process your owned units in order, write the declared outputs, and stop.

The second step is to remove repo-controlled happy-path lease advancement from recipe and knowledge. These stages currently ask the worker what to do next by rewriting control files mid-session. That behavior must be deleted. Their planners should instead assign all owned shards or tasks up front. Recipe may still keep one semantic output per recipe id. Knowledge may still keep its two semantic passes if the semantic contract requires them, but those passes must become explicit assigned work units, not repo-leased transitions inside one queue loop. If knowledge still needs a pass-1 and pass-2 distinction, the planner should materialize them as explicit immutable units in the initial assignment or as an explicit post-validation repair-style follow-on batch produced by deterministic code, not as a lease continuation.

The third step is to replace the line-role phase-ledger happy path with the same fixed-assignment story. The worker should not need to inspect `CURRENT_PHASE*`, mutate a running work ledger, or ask the repo which phase is active. The line-role planner should produce immutable owned shard inputs and fixed outputs. The worker’s job becomes simple: label the owned rows in each assigned shard and write the finished outputs. Any missing-row or malformed-output repair should happen through the same narrow repair mechanism as the other stages.

The fourth step is to redesign repair around explicit small follow-up assignments. The happy path becomes “assignment in, outputs out.” After that, deterministic validation runs. The validator records exact structural errors for each failed owned unit. The runtime then creates one repair assignment containing only those failed units, their original semantic inputs, the rejected prior output, and a compact machine-readable error list. The repair worker writes corrected outputs to the same stable output paths or to a mirrored `repair_out/` namespace that the repo then installs. There is one repair pass only. If repair still fails, the unit fails closed. This preserves the useful “check and correct” loop while preventing the repair mechanism from becoming the main runtime shape.

The fifth step is to reset telemetry and benchmark reporting so the numbers describe the fixed-assignment runtime honestly. Prompt-budget summaries should count real worker sessions, real owned units, and real repair batches separately. Stage summaries and benchmark helpers must stop presenting lease advancement or phase advancement as if those are core semantic work units. The reports at the end of this plan should answer three simple questions: how many worker sessions ran, how many owned units were processed, and how many owned units required repair. The later one-file and non-lethal plans will further simplify the file-edit and supervision story from there.

The sixth step is deletion. Remove live code paths, prompts, tests, and docs that teach or preserve the old lease or phase stories. This includes removing live references to `current_packet.json`, `current_result_path.txt`, `packet_lease_status.json`, `CURRENT_PHASE.md`, `CURRENT_PHASE_FEEDBACK.md`, `current_phase.json`, and any repo-authored queue choreography files as the normal path. Historical artifacts in old benchmark directories may remain as historical evidence, but live code and live docs must teach only the new fixed-assignment runtime until the later one-file follow-on plans replace that worker surface entirely.

## Validation And Acceptance

This work is accepted only when the live Codex Exec stages all share the same observable runtime shape: fixed assignment first, deterministic validation second, and one bounded repair batch third. Acceptance is not “some tests passed.” Acceptance is that the repository no longer teaches or requires happy-path lease advancement or phase-ledger loops.

The first acceptance criterion is worker-root shape. Recipe, knowledge, and line-role workers must all expose immutable assignment artifacts and stable output targets. Focused tests must fail if the happy path writes or depends on live lease or phase control files.

The second acceptance criterion is bounded repair. A malformed output should trigger one explicit repair assignment and no more. Tests must prove that a valid repair installs successfully and that a second failure terminates cleanly without reopening a general queue loop.

The third acceptance criterion is token honesty at the fixed-assignment level. Prompt-budget and telemetry tests must prove that worker-session count, owned-unit count, and repair-batch count are reported separately. The queue/phase protocol overhead that dominated the old runtime should be gone even if the worker surface is still multi-file at this stage.

The fourth acceptance criterion is documentation clarity. The live docs must explain only the fixed-assignment runtime. A new contributor should not discover the old lease or phase stories in active docs and think they are current.

## Idempotence And Recovery

The implementation sequence is safe to repeat because each milestone is test-driven and additive until the old path is deleted. If a partial edit leaves one stage half-converted, revert to the last green state for that stage and rerun the focused tests before continuing. Do not add adapters that quietly accept both old and new control surfaces. If a migration step becomes unclear, prefer finishing one stage end-to-end on the new contract and then applying the same shape to the next stage rather than widening the temporary compatibility window.
