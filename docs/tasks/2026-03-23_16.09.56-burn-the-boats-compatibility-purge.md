# Burn the boats compatibility purge

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

This repository has already moved to a newer architecture in several places, but it still carries old readers, old aliases, old worker contracts, and old knobs that exist only so removed behavior does not fail immediately. That is the opposite of the intended operating model for this project. This is a solo codebase. The goal of this plan is to make the code tell the truth: current contracts only, current names only, and hard failures when older shapes appear.

After this work, a human should be able to read the live runtime, the CLI surface, the tests, and the main docs without learning two systems at once. The visible proof is that removed contracts no longer parse, removed knobs no longer appear or silently no-op, repo searches for the targeted compatibility seams go to zero in active code, and the surviving tests prove the current contract rather than legacy acceptance.

The tone of this plan is intentionally destructive. This is not a migration plan, not a deprecation plan, and not a dual-path rollout. The replacement already exists. The remaining work is to burn the boats behind it.

## Context and Orientation

The core of the current runtime is a stage pipeline that expects importers to produce canonical `SourceBlock` data, then runs deterministic label-first authority, optional recipe Codex refinement, deterministic non-recipe routing, and optional knowledge review. In this repository, a “compatibility seam” means code that still accepts an older contract, old artifact shape, old label name, old config key, or old filesystem layout instead of rejecting it.

The important files for this cleanup are:

- `cookimport/core/source_model.py`, which turns importer output into canonical source blocks and still contains a migration reader for legacy `extracted_rows`.
- `cookimport/staging/pipeline_runtime.py` and `cookimport/parsing/label_source_of_truth.py`, which call that source-model resolver on live stage paths.
- `cookimport/llm/codex_farm_knowledge_ingest.py` and `cookimport/llm/codex_farm_knowledge_models.py`, which validate knowledge-worker outputs and still normalize older worker payload shapes.
- `cookimport/llm/codex_farm_orchestrator.py`, which plans recipe shards and still supports multi-recipe bundling via `recipe_shard_target_recipes`.
- `cookimport/config/run_settings.py`, `cookimport/config/run_settings_adapters.py`, `cookimport/cli.py`, `cookimport/labelstudio/ingest.py`, and `cookimport/llm/prompt_preview.py`, which together define the live setting and CLI surface and still preserve removed or no-op knobs.
- `cookimport/labelstudio/label_config_freeform.py`, `cookimport/labelstudio/eval_freeform.py`, `cookimport/parsing/label_source_of_truth.py`, and several benchmark helpers, which still normalize old label vocabulary like `TIP`.
- `cookimport/bench/oracle_upload.py`, `cookimport/bench/artifact_gc.py`, `cookimport/llm/codex_farm_runner.py`, and related benchmark/tooling files, which still keep read-side support for older filesystem or subprocess-output conventions.
- The tests under `tests/core`, `tests/llm`, `tests/labelstudio`, `tests/bench`, and `tests/cli`, which must be rewritten so they protect the new destructive stance.

For this plan, “burn the boats” means four concrete rules. First, removed contracts must fail closed. Second, removed knobs must disappear instead of warning and continuing. Third, removed names and aliases must stop parsing. Fourth, docs and tests must describe only the surviving contract.

## Plan of Work

Begin with the highest-value runtime contract purge: remove legacy source-model reconstruction and remove legacy knowledge-worker payload acceptance. In `cookimport/core/source_model.py`, delete `_render_legacy_row_text(...)` and `build_source_model_from_legacy_result(...)`, then simplify `resolve_conversion_source_model(...)` so it accepts only present-day canonical `source_blocks` or the explicit `full_blocks` argument already supplied by current call sites. Keep the error message blunt and user-facing: stage input is missing canonical source blocks. Do not replace the deleted path with another normalizer. The point is to make any remaining old importer/result path break loudly. After deleting that code, update `cookimport/core/README.md` and remove or rewrite the tests that currently assert legacy reconstruction. Add a new test that proves `resolve_conversion_source_model(...)` raises when only `extracted_rows` are present.

In parallel with that, cut the knowledge-worker compatibility ladder. In `cookimport/llm/codex_farm_knowledge_ingest.py`, remove the fallback parse of `KnowledgeBundleOutputV2`, remove alias rewrites that only exist to translate old category names, and stop inventing missing `reason_code` values. The only accepted worker contract after this change should be the semantic packet contract that the repo currently owns. In `cookimport/llm/codex_farm_knowledge_models.py`, delete `default_legacy_knowledge_reason_code(...)` and any repo-side serializer paths that still emit legacy-default reason codes purely for compatibility. If a repo-owned internal helper still needs to build a semantic packet from a richer object, make it emit the current explicit fields directly instead of going through a compatibility default. Rewrite the tests so old bundle payloads now fail validation and the expected error messages mention the current semantic contract.

Once the repo fails closed on old data contracts, remove old runtime choices that still preserve older behavior. In `cookimport/config/run_settings.py`, delete `recipe_shard_target_recipes`, `knowledge_prompt_target_count`, and `knowledge_shard_target_chunks` from the model and the builder helpers. The current recipe story is one candidate-owned shard by default, and the current knowledge story is one deterministic chunk per shard. Keeping knobs that re-enable bundling or pretend to tune a planner that no longer honors them undermines that story. In `cookimport/llm/codex_farm_orchestrator.py`, simplify recipe shard planning so every recipe owns its own shard. Remove `_shard_target_recipe_count(...)` and stop threading `recipe_shard_target_recipes` through phase settings, preview output, and tests. In `cookimport/llm/codex_farm_knowledge_orchestrator.py`, stop passing the removed knowledge shard knobs to the planner and phase settings. In `cookimport/llm/codex_farm_knowledge_jobs.py`, delete the warning-producing dead-path code because the knobs themselves no longer exist.

After the runtime stops accepting old contracts and old knobs, clean the surfaces that still advertise or serialize them. In `cookimport/cli.py`, remove hidden options such as `--knowledge-prompt-target-count` and any matching hidden recipe sharding knobs that only exist to preserve a dead path. In `cookimport/config/run_settings_adapters.py`, remove internal field lists for deleted settings so stage and benchmark adapters only emit live keys. In `cookimport/labelstudio/ingest.py`, `cookimport/llm/prompt_preview.py`, and `cookimport/cf_debug_cli.py`, delete parameters, preview rows, settings payload fields, and experiment examples that mention the removed knobs. In `docs/examples/shard_sweep_examples.json`, delete old examples that still teach multi-recipe recipe shards or chunk-per-shard tuning knobs. If a debug-only CLI absolutely must keep one escape hatch for experimentation, that must be renamed to describe the current behavior honestly rather than preserving old production naming.

Then remove ontology and naming aliases that preserve removed language. In `cookimport/labelstudio/label_config_freeform.py`, shrink normalization to canonical label spelling only. If the surviving product labels are `RECIPE_TITLE`, `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `HOWTO_SECTION`, `YIELD_LINE`, `TIME_LINE`, `RECIPE_NOTES`, `RECIPE_VARIANT`, `KNOWLEDGE`, and `OTHER`, then any old label like `TIP`, `TIME`, `YIELD`, `NOTE`, or `VARIANT` should now fail or be treated as invalid input rather than quietly remapped. Update `cookimport/labelstudio/eval_freeform.py`, `cookimport/parsing/label_source_of_truth.py`, benchmark evaluators, and prelabel helpers to use the stricter normalization. Rewrite the tests that currently assert back-compat alias acceptance. The new tests should prove that invalid labels are rejected or classified as invalid in a way the caller can see.

Remove remaining read-side legacy filesystem and subprocess readers. In `cookimport/bench/oracle_upload.py`, stop scanning `ORACLE_LEGACY_BROWSER_PROFILE_DIR`; use only the current configured profile dir or an explicit environment override. In `cookimport/bench/artifact_gc.py`, remove `_LEGACY_TIMESTAMP_DIR_RE` and accept only the canonical timestamp format `YYYY-MM-DD_HH.MM.SS`, which is already the project-wide convention. In `cookimport/llm/codex_farm_runner.py`, remove `_parse_legacy_run_progress_line(...)` and the logic that silently suppresses old stderr progress formats. If current Codex Farm emits a structured progress event or current “run started” line, keep that. Otherwise let unknown stderr through instead of curating it with backward-compatibility rules.

Finish with smaller naming aliases and wording drift. In `cookimport/parsing/epub_html_normalize.py`, remove the `semantic_v1 -> br_split_v1` alias and keep one honest name. Update `cookimport/cli.py` interactive menu text so the user sees only live preprocess modes. In `cookimport/config/run_settings.py`, remove small spelling aliases such as `public -> product` if no current caller requires them. In any help text or readme that says “compatibility-only”, “legacy opt-out”, “legacy mode”, or “currently aliases”, replace the description by deleting the feature or renaming the remaining one. The aim is that new readers should never need to infer which name is the real one.

The final pass is tests, docs, and search guards. Rewrite every affected test module so it either disappears with the removed behavior or flips to assert failure on removed inputs. Update the main living docs that currently mention compatibility-only knobs or legacy readers: `docs/01-architecture/01-architecture_README.md`, `docs/03-ingestion/03-ingestion_readme.md`, `docs/05-staging/05-staging_readme.md`, `docs/06-label-studio/06-label-studio_README.md`, `docs/10-llm/10-llm_README.md`, `docs/12-testing/12-testing_README.md`, `cookimport/core/README.md`, and any nearby package README that still narrates the removed paths. The end state is not “documented compatibility”; it is “no compatibility remains to document.”

## Validation and Acceptance

This plan is complete only when all of the following are true.

The first acceptance rule is contract honesty. Stage-backed runtime code must reject legacy importer/result payloads that do not provide canonical `source_blocks`, and knowledge-worker validation must reject older bundle payloads instead of translating them.

The second acceptance rule is surface honesty. Removed knobs such as `knowledge_prompt_target_count`, `knowledge_shard_target_chunks`, and `recipe_shard_target_recipes` must be gone from live settings, live CLI wiring, preview output, helper adapters, docs, and tests. They must not remain as hidden options, serialized config fields, or warning-only placeholders.

The third acceptance rule is ontology honesty. Old freeform labels such as `TIP`, `TIME`, `YIELD`, `NOTE`, and `VARIANT` must stop being silently accepted as equivalent to current labels. The surviving code must either reject them clearly or treat them as invalid input in a visible, test-covered way.

The fourth acceptance rule is tooling honesty. Benchmark and worker tooling must no longer prefer or silently read older filesystem locations, timestamp formats, or old stderr progress formats when the current contract is absent.

The fifth acceptance rule is repository honesty. The major domain suites named above must pass, and the final search sweep must show that the targeted compatibility seams no longer appear in active runtime code or active docs/tests.

The sixth acceptance rule is documentation honesty. The main READMEs and package notes must describe only the surviving contract. A novice reading the repo after this work should not learn that there used to be another accepted path unless they intentionally open historical plans or logs.

## Idempotence and Recovery

This work is intentionally destructive, but it can still be performed safely in repeatable passes. Each pass deletes one compatibility family and immediately reruns the smallest domain tests that prove the replacement path is still correct. If a pass uncovers a hidden live dependency on a removed seam, do not restore the compatibility path broadly. Instead, decide which of two outcomes is true and record it here: either the dependency was itself legacy and should be deleted too, or the dependency reflects a real surviving contract that needs a narrowly named current-home implementation.

If deleting an old reader makes an archived output folder or one stale fixture unreadable, that is not automatically a bug. The default recovery path is to regenerate the fixture or accept that the archived artifact is no longer part of the supported live product surface. Only restore a read path if the artifact is still required by a current, intentional workflow, and if restored, rename and document it as a current feature rather than a legacy adapter.

If a test failure reveals that a removed knob is still threaded through a helper, remove the helper argument and update the caller in the same change. Do not keep a dead parameter just to preserve call-site shape.
