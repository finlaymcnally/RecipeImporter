# Stop Knowledge Workers From Conversationally Ending Mid-Queue

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a knowledge workspace worker that has already advanced from task 1 to task 2 must keep processing the same assignment instead of stopping to say “If you want, I can continue.” The worker should experience the queue as one continuing assignment with many repo-owned steps, not as a series of isolated mini-jobs that invite permission-seeking after each successful install. The user-visible proof is simple: in a regenerated worker root and in the next benchmark rerun, workers should validate multiple sequential knowledge tasks in one session, and clean mid-queue exits should no longer dominate `missing_output`.

This plan exists because the previous handoff fix worked technically but not behaviorally. In `data/output/2026-03-22_12.08.53/.../knowledge/workers/worker-001/`, the worker now correctly sees `current_task.json`, `CURRENT_TASK.md`, and `CURRENT_TASK_FEEDBACK.md` advance to task 2 after `install-current`, yet [worker-001 last_message.json] still ends with “If you want, I can continue immediately with task-002.” The old bug was stale handoff state. The new bug is that the prompt, sidecars, and runtime still leave too much room for the model’s conversational “checkpoint and ask permission” instinct.

- Observation: the worker prompt already tells the agent to continue, but the wording still leaves a conversational escape hatch.
  Evidence: `data/output/2026-03-22_12.08.53/.../knowledge/workers/worker-001/prompt.txt` says “If a new current task is active, continue with it,” yet `last_message.json` still ends with “If you want, I can continue immediately with task-002.”

- Observation: the repo-written current-task files still feel discrete and self-contained enough to encourage “task complete” closure.
  Evidence: `CURRENT_TASK.md` opens with one task header, one task’s files, and one task’s loop; `CURRENT_TASK_FEEDBACK.md` says there is “No repo-written validation feedback exists yet for this task”; neither file restates that the broader worker assignment is still active and incomplete.

- Observation: this is not just a one-task-only failure anymore.
  Evidence: `worker-004` in the same benchmark writes four outputs and advances to `task-005`, then still exits with `workspace_validated_task_queue_incomplete`. The failure mode is now “stop at an arbitrary conversational checkpoint,” not necessarily “stop after exactly one task.”

- Observation: a prompt-only fix is unlikely to be durable enough by itself.
  Evidence: the benchmark already contains explicit continuation wording, but the model still interprets a successful install plus advanced sidecars as a good moment to summarize and ask for permission. That implies the runtime needs a deterministic backstop in addition to stronger wording.

- Observation: a fake workspace-worker regression is enough to prove the runtime backstop without spending tokens on a live Codex run.
  Evidence: `tests/llm/test_codex_farm_knowledge_orchestrator.py::test_run_direct_knowledge_workers_relaunches_premature_clean_mid_queue_stop` now forces a clean informational stop after task 1, and the saved `live_status.json` finishes with `workspace_relaunch_count: 1`, `queue_validated_task_count: 2`, and `reason_code: workspace_validated_task_queue_completed`.

- Observation: the helper script had its own embedded success-copy strings separate from the pure Python sidecar renderers.
  Evidence: after the first prose patch, `install-current` stdout updated immediately but `CURRENT_TASK_FEEDBACK.md` generated by `tools/knowledge_worker.py` still said “continue with that task” until the script template itself was updated.

- Observation: exact parity tests are the cheapest way to stop the generated helper from drifting again.
  Evidence: `tests/llm/test_knowledge_workspace_tools.py` now compares `CURRENT_TASK_FEEDBACK.md` written by `tools/knowledge_worker.py` byte-for-byte against `render_current_task_feedback_text(...)` for both the pending and validator-accepted states.

## Outcomes & Retrospective

This plan is now implemented locally. The prompt, current-task sidecars, helper CLI, and worker-session finalization path all reinforce the same contract: `install-current` is a handoff, not a stopping point, and a clean exit after the queue has visibly advanced is not accepted as the end of the assignment. The original fake-runner regression proved the repo could rescue one such stop and reach `workspace_validated_task_queue_completed` without any live model call. A later structural follow-up then broadened that into the small capped auto-resume path used in the current runtime. The relaunch reason is also durable now: final `live_status.json`, worker reports, and both knowledge telemetry summaries carry `workspace_relaunch_history`, `workspace_relaunch_reason_codes`, and `workspace_premature_clean_exit_count` instead of losing that context once the relaunch succeeds.

The main remaining gap is live confirmation. The local regression proves the runtime semantics and worker-facing copy, but it does not prove what the live model will do on the next benchmark run. That optional benchmark rerun still requires explicit user approval.

## Context and Orientation

The knowledge stage lives primarily in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, `cookimport/llm/knowledge_workspace_tools.py`, and `cookimport/llm/codex_exec_runner.py`. The orchestrator plans the worker assignments, writes the worker prompt, validates task outputs, and records why a worker session ended. The workspace-tools module writes the human-readable sidecars such as `CURRENT_TASK.md` and `CURRENT_TASK_FEEDBACK.md` and renders `tools/knowledge_worker.py`, which is the repo-written helper CLI inside the worker root. The direct-exec runner launches the long-lived Codex session inside a sterile mirrored workspace and synchronizes selected files between the durable worker root and the live execution workspace.

In this repository, a “worker assignment” means one long-lived worker session that owns many task packets. A “task packet” is one repo-owned current task row from `assigned_tasks.json` plus its named `in/*.json`, `hints/*.md`, and `out/*.json` files. A “current-task bundle” is the trio `current_task.json`, `CURRENT_TASK.md`, and `CURRENT_TASK_FEEDBACK.md` that tells the worker what to do now. A “premature clean exit” means the worker session ends without a watchdog kill, while `queue_complete` is still false and the repo has already advanced the current-task bundle to a later task.

The current benchmark proof is in `data/output/2026-03-22_12.08.53/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-22_12.09.24/raw/llm/saltfatacidheatcutdown/knowledge/`. `stage_status.json` shows `validated: 7`, `missing_output: 308`, and `worker_reason_codes.workspace_validated_task_queue_incomplete: 8`. Worker 001 proves the new seam clearly. `live_status.json` says `current_task_id` is task 2, `CURRENT_TASK.md` also shows task 2, and `last_message.json` says the worker re-opened the new bundle and then stopped with “If you want, I can continue immediately with task-002.” Worker 004 proves the same failure after four tasks rather than one.

That means the new implementation must solve three related problems. First, the prompt still allows the worker to think of “install succeeded and I can see the next task” as a reasonable time to stop and narrate progress. Second, the current-task sidecars still feel too much like a sequence of disconnected mini-jobs instead of one continuous assignment with remaining work. Third, the runtime currently records `workspace_validated_task_queue_incomplete` but still lets the worker session end there, which means the system depends too heavily on the model never taking that conversational off-ramp.

The files that matter most for this plan are:

- `cookimport/llm/codex_farm_knowledge_orchestrator.py`, especially `_build_knowledge_workspace_worker_prompt(...)`, `_build_strict_json_watchdog_callback(...)`, and the worker-session end-state handling around `workspace_validated_task_queue_incomplete`.
- `cookimport/llm/knowledge_workspace_tools.py`, especially `render_current_task_brief_text(...)`, `render_current_task_feedback_text(...)`, `write_current_task_sidecars(...)`, and the rendered `tools/knowledge_worker.py` output for `install-current`.
- `cookimport/llm/codex_exec_runner.py`, especially the final-message and supervision seams used by workspace workers when a session ends without queue completion.
- `tests/llm/test_codex_farm_knowledge_orchestrator.py` and `tests/llm/test_codex_exec_runner_workspace.py`, which already cover current-task handoff and should grow to cover premature clean exits and anti-stop wording.

## Plan of Work

The first milestone is a local failing reproduction for the new failure mode. Add or extend a test that simulates a worker session which successfully installs task 1, correctly sees task 2, and then emits a clean informational final message that sounds like “I can continue if you want.” That test should currently end as `workspace_validated_task_queue_incomplete` with no further progress. A second test should pin the desired prompt and sidecar surface so the worker contract clearly says the assignment is not complete until the queue is complete.

The second milestone is worker-prompt hardening. Update `_build_knowledge_workspace_worker_prompt(...)` in `cookimport/llm/codex_farm_knowledge_orchestrator.py` so the valid stopping states are explicit. The prompt should say that the assignment is incomplete while `queue_complete` is false, that successful `install-current` is only a handoff to the next step, and that the worker must not ask for permission to continue, summarize progress mid-queue, or end the session after a successful install unless the queue is complete or the current task remains blocked on validator feedback. The wording should stay practical and imperative, not abstract.

The third milestone is assignment-flowing sidecars and helper output. Rewrite `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`, and the `install-current` stdout in `cookimport/llm/knowledge_workspace_tools.py` so they feel like a continuing assignment. The sidecars should restate queue progress in assignment language such as “assignment progress: 1 of 79 validated; 78 remain,” explicitly say that the broader assignment is still active, and make the next action an imperative “continue now” instruction instead of a polite suggestion. Consider adding one small persistent assignment-level sidecar such as `WORKER_ASSIGNMENT.md` if the current-task files alone cannot carry the continuity strongly enough, but only if the added file clearly improves the model-facing contract.

The fourth milestone is the runtime backstop. Adjust the worker-session completion logic so a clean workspace-worker exit with `queue_complete=false`, an already-advanced `current_task_id`, no blocking validation errors, and no watchdog kill is treated as a recoverable premature stop rather than an accepted terminal state. The simplest acceptable implementation is to relaunch the same worker assignment once on the remaining queue with an explicit “continue the existing assignment” prompt context. A stronger implementation is to distinguish this reason code clearly, such as `workspace_premature_clean_exit_mid_queue`, so telemetry and retries no longer blur it with genuine incomplete-state failures.

The fifth milestone is regression protection. Add tests that prove all of the following: the prompt explicitly forbids permission-seeking mid-queue exits; the sidecars and helper output present the assignment as continuous; a clean informational final message while `queue_complete=false` triggers the new recovery path; and a session that already advanced to task 2 cannot be treated as finished while the queue is still incomplete. These tests should remain narrow and deterministic; they should not require a live benchmark to prove the behavior.

The sixth milestone is documentation and optional benchmark proof. Update the short LLM docs so they describe the stronger knowledge-worker assignment contract and the runtime backstop for premature clean exits. Only with explicit user approval, rerun a narrow benchmark slice and compare it against `2026-03-22_12.08.53`. Acceptance on the benchmark is not “workers can see task 2.” It is “workers no longer stop to ask permission after reaching task 2, multi-task progress per worker materially increases, and `workspace_validated_task_queue_incomplete` stops being the dominant clean-exit reason.”

## Validation and Acceptance

This work is accepted only when both the worker-facing behavior and the repo-facing recovery behavior are observable.

At the prompt-contract level, the main knowledge worker prompt must explicitly say that the worker assignment is incomplete until `queue_complete` is true, that successful `install-current` is only a handoff to the next current task, and that the worker must not ask for permission to continue or end the session after a successful install while later tasks remain.

At the sidecar-contract level, `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`, and `install-current` stdout must make the assignment feel continuous rather than discrete. Acceptance is not “the next task id is visible.” It is “the worker-facing text explicitly says the assignment is still active, names the remaining queue progress, and directs the worker to continue immediately.”

At the runtime level, a clean informational worker exit while `queue_complete=false` must no longer be accepted as the practical end of the worker session. The repo must either keep the session going through the existing call or relaunch the same remaining assignment deterministically. A fake-runner or simulated-session regression should make this visible without requiring a live model.

At the optional benchmark level, acceptance means that the `2026-03-22_12.08.53` failure pattern disappears: workers no longer stop with messages such as “If you want, I can continue immediately with task-002,” and sequential validated-task counts per worker materially increase.

## Idempotence and Recovery

The sidecar writer must remain safe to call repeatedly. Rewriting `CURRENT_TASK.md` and `CURRENT_TASK_FEEDBACK.md` for the same active task should be a no-op in effect and should not duplicate or drift assignment progress text.

If the runtime backstop relaunches an assignment after a premature clean exit, that relaunch must be idempotent with respect to already-validated outputs. The follow-up worker must start from the advanced `current_task.json` and must not revalidate accepted earlier tasks unless the current repo contract already requires that.

If the first implementation cannot safely continue the same live session, prefer a deterministic repo-owned re-entry on the remaining queue over trying to infer user intent from the worker’s final prose. The system should fail closed toward continuing the queued assignment, not toward accepting a conversational pause as completion.

As with the earlier knowledge benchmark work, live reruns remain optional and must not happen without explicit user approval.
