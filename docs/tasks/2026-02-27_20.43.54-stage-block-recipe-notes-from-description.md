# Task: Stage block `RECIPE_NOTES` from description-derived notes

## Problem statement

`RECIPE_NOTES` block labeling in `stage_block_predictions` used only `recipe.comment` text.
Many parsed recipes carry recipe-specific notes in `description`, causing benchmark predictions to miss `RECIPE_NOTES` entirely.

## Acceptance criteria

- `stage_block_predictions` can label `RECIPE_NOTES` when recipe-specific note text exists only in `description`.
- Existing comment-based note labeling still works.
- Regression test covers description-only note path.

## Verification steps

1. Fail-before (new test, before code fix):
   - `source .venv/bin/activate && COOKIMPORT_PYTEST_VERBOSE_OUTPUT=1 pytest -o addopts='' tests/staging/test_stage_block_predictions.py::test_build_stage_block_predictions_marks_notes_from_description_only -vv --tb=short --show-capture=all --assert=rewrite`
2. Pass-after:
   - `source .venv/bin/activate && pytest tests/staging/test_stage_block_predictions.py tests/parsing/test_tip_recipe_notes.py -q`

## Evidence

- Before fix, new test failed with:
  - `assert 'OTHER' == 'RECIPE_NOTES'` for block `7`.
- After fix, the targeted staging/parsing tests above pass.

## Constraints / gotchas

- Keep note extraction deterministic and avoid introducing duplicate match rows.
- Do not enable any LLM/codex-farm parsing path.

## Decision

Reuse existing deterministic parser utility `extract_recipe_specific_notes(candidate)` (already used in draft-v1), and merge its results with `recipe.comments` for stage note matching.

## Rollback / compensation

- Revert `cookimport/staging/stage_block_predictions.py` and the new regression test if note sourcing needs to return to comment-only behavior.
