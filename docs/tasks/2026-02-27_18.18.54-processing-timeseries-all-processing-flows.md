## Problem

`scheduler_timeseries.jsonl` existed for all-method benchmark scheduling, but other processing flows had no persisted status+CPU time-series output.

## Acceptance Criteria

- Stage runs write run-local `processing_timeseries.jsonl`.
- Spinner-driven processing flows write JSONL status telemetry with CPU sampling:
  - Label Studio import
  - Label Studio benchmark prediction/evaluation phases
  - Bench run/sweep
  - Interactive all-method wrapper progress
- Stage run manifest and benchmark run manifests expose new telemetry artifacts where applicable.

## Verification

1. `source .venv/bin/activate && pytest tests/labelstudio/test_labelstudio_benchmark_helpers.py -k 'run_with_progress_status_writes_processing_timeseries or writes_scheduler_timeseries or smart_scheduler_improves_heavy_slot_utilization'`
2. `source .venv/bin/activate && pytest tests/cli/test_cli_output_structure.py -k stage_output_structure`

## Evidence

- Added `_ProcessingTimeseriesWriter` + shared CPU sampling helper in `cookimport/cli.py`.
- Added stage run telemetry output (`processing_timeseries.jsonl`) and manifest artifact registration.
- Added benchmark/import/bench telemetry plumbing via `_run_with_progress_status(... telemetry_path=...)`.
- Added tests:
  - `test_run_with_progress_status_writes_processing_timeseries`
  - `test_stage_output_structure` now asserts stage processing telemetry artifact.

## Constraints / Gotchas

- No new dependencies allowed for CPU sampling.
- Keep existing spinner text contracts unchanged while adding persistence.

## Decision

Use a shared JSONL writer that records status snapshots + CPU utilization from `/proc/stat`, with state-change writes and heartbeat sampling, then wire it through shared spinner plumbing plus stage’s dedicated progress loop.
