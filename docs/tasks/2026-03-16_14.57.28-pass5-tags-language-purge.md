Problem statement

Active tagging runtime and docs still expose the retired `pass5` name for the optional LLM tag-suggestion stage. That old terminology is still present in live CLI/config/test surfaces even though stage observability already uses semantic `tags` naming.

Acceptance criteria

- Live config and runtime code use `codex_farm_pipeline_tags` instead of `codex_farm_pipeline_pass5_tags`.
- Live CLI surfaces use `--codex-farm-pipeline-tags` instead of `--codex-farm-pipeline-pass5-tags`.
- Raw stage tagging artifacts use `raw/llm/<workbook>/tags/` rather than `pass5_tags`.
- Focused tests and current docs match the renamed contract.

Verification steps

- `source .venv/bin/activate && pytest tests/tagging/test_tagging.py tests/cli/test_cli_llm_flags.py tests/cli/test_c3imp_interactive_menu.py tests/cli/test_run_settings_adapters.py tests/llm/test_llm_pipeline_pack.py -q`
- `rg -n "pass5|codex_farm_pipeline_pass5_tags|pass5_tags" cookimport tests docs llm_pipelines -S`

Evidence

- Before: live grep showed runtime/test hits for `codex_farm_pipeline_pass5_tags`, `--codex-farm-pipeline-pass5-tags`, and `pass5_tags`.
- After: targeted tests pass and active-surface grep only shows historical docs/log context, if any.

Constraints and decision

- Keep scope tight to the tagging rename seam; do not widen into the broader recipe `pass1/pass2/pass3` legacy cleanup.
- Use a destructive rename instead of compatibility aliases because the user explicitly wants the old terminology removed.

Rollback / compensation plan

- If a rename breaks a historical or archived reader, restore only that narrow reader path or defer the historical artifact. Do not reintroduce `pass5` naming into the live runtime contract.
