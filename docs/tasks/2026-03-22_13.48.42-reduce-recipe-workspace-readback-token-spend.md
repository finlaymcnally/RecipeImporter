# Reduce recipe workspace readback token spend

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The recipe benchmark is now reliable again, but it is still too expensive because recipe workers spend too much of each long-lived Codex workspace session rereading local files before they edit the prepared drafts. After this change, a recipe worker should be able to open one repo-written shard packet, trust the prewritten drafts, edit them, and run `finalize-all` without defaulting to repeated `cat` loops over manifests, contracts, examples, or helper source. The user-visible proof is a rerun of the same Salt Fat single-book codexfarm slice where recipe keeps `validated_shard_count: 5`, keeps `watchdog_killed_shard_count: 0`, keeps `watchdog_recovered_shard_count: 0`, and reduces recipe command-execution token spend below the current `536003` while holding command count at or below the current `47`.

## Surprises & Discoveries

- Observation: recipe correctness is no longer the limiting problem.
  Evidence: `data/golden/benchmark-vs-golden/2026-03-22_13.22.18/single-book-benchmark/saltfatacidheatcutdown/codexfarm/prompt_budget_summary.json` records recipe `validated_shard_count: 5`, `watchdog_killed_shard_count: 0`, `watchdog_recovered_shard_count: 0`, and `structured_followup_call_count: 0`.

- Observation: recipe command count fell sharply, but recipe command-execution tokens still went up.
  Evidence: the same summary shows recipe `command_execution_count_total: 47`, which beats both `93` at `2026-03-22_12.08.53` and `91` at `2026-03-22_04.26.00`, while `command_execution_tokens_total: 536003` is worse than `473128` and slightly worse than `531073`.

- Observation: the expensive part is the interactive workspace session, not the bytes printed by local shell commands.
  Evidence: `data/output/2026-03-22_13.22.18/.../recipe_phase_runtime/workers/worker-001/status.json` shows `command_execution_count: 9`, `visible_input_tokens: 154`, `tokens_cached_input: 69760`, and `wrapper_overhead_tokens: 76599` for the primary telemetry row.

- Observation: the prompt still explicitly tells recipe workers to open too many files before they act.
  Evidence: `.../worker-001/prompt.txt` says to open `worker_manifest.json`, `CURRENT_TASK.md`, `current_task.json`, `CURRENT_TASK_FEEDBACK.md`, and `OUTPUT_CONTRACT.md` up front, then to open `hints/<task_id>.md` and `in/<task_id>.json` before editing each draft.

- Observation: the repo-written surfaces are strongest for the first task, not for the entire shard.
  Evidence: `.../worker-001/CURRENT_TASK.md` gives a good starting brief for task 1, but the later worker status files still show per-task `cat` loops for tasks 2 through 5 before `finalize-all`.

- Observation: the strict output contract makes the model conservative enough to reread examples and helper source.
  Evidence: `.../worker-004/status.json` and `.../worker-005/status.json` both show reads of `examples/*.json` and slices of `tools/recipe_worker.py` even after the startup churn fixes were landed.

- Observation: a better recipe helper path probably needs to change the worker trust model, not just cut more commands from the manifest.
  Evidence: previous changes already added `prepare-all`, `finalize-all`, `scratch/_prepared_drafts.json`, and `stamp-status`, but the model still voluntarily pays orientation cost before using them.

- Observation: the recipe helper already contained most of the right current-task vocabulary; the missing piece was that the standalone CLI did not actually advance the sidecars after `install-current`.
  Evidence: `cookimport/llm/recipe_workspace_tools.py` already rendered `check-current` / `install-current` guidance into `CURRENT_TASK.md` and `CURRENT_TASK_FEEDBACK.md`, but the embedded `tools/recipe_worker.py` script only rewrote feedback before this implementation pass.

- Decision: make one repo-written shard packet the only required startup read in the normal recipe path.
  Rationale: the current prompt and manifest both encourage workers to read several files before they can feel safe acting. A single authoritative packet is the clearest way to remove that ambiguity and make the cheapest path obvious.
  Date/Author: 2026-03-22 / Codex

  Rationale: the worker should not need to inspect reference docs or tool source just to complete the common shard flow. If those files remain default reads, the token problem is unlikely to improve materially.

  Rationale: the current `CURRENT_TASK.md` only helps for the first task. The worker still spends tokens deciding what to open next across the rest of the shard, which is avoidable if the helper/runtime precomputes more of that loop.
  Date/Author: 2026-03-22 / Codex

- Decision: keep benchmark reruns approval-gated.
  Rationale: this repository explicitly forbids token spend without user approval, and the current turn only asks for an ExecPlan.

## Context and Orientation

The relevant benchmark evidence lives under `data/golden/benchmark-vs-golden/2026-03-22_13.22.18/single-book-benchmark/saltfatacidheatcutdown/codexfarm/`. In this run, recipe reliability is good but cost is not. Recipe finished with `validated_shard_count: 5`, `watchdog_killed_shard_count: 0`, `watchdog_recovered_shard_count: 0`, `structured_followup_call_count: 0`, `command_execution_count_total: 47`, and `command_execution_tokens_total: 536003`. The lower command count proves the earlier churn work helped, but the high token count shows the remaining problem is the shape of the surviving workspace sessions.

Recipe workspace behavior is assembled in three places. `cookimport/llm/codex_farm_orchestrator.py` writes worker prompts, manifests, and repo-owned control files such as `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`, `current_task.json`, and `assigned_tasks.json`. `cookimport/llm/recipe_workspace_tools.py` creates the worker-side helper CLI that currently supports operations such as `prepare-all`, `finalize-all`, and `stamp-status`, and it also decides what helper-generated files such as `scratch/_prepared_drafts.json` look like. `cookimport/llm/codex_exec_runner.py` is the long-lived workspace session wrapper that tracks command telemetry and enforces workspace-shell boundaries, but this plan does not need new boundary semantics unless the helper loop changes how files are exposed.

The current common recipe path still makes the model feel responsible for too much orientation. The prompt tells the worker to open several startup files, then to open per-task hint and input files before editing the draft. The manifest still advertises many mirrored files as entry points. The contract is strict enough that the model sometimes opens `OUTPUT_CONTRACT.md`, mirrored examples, and even `tools/recipe_worker.py` to reassure itself that it is writing valid compact recipe JSON. In practice, that means the long-lived session burns cached-input and wrapper-overhead tokens on orientation rather than on the actual draft edits.

This plan introduces two core ideas. First, recipe workers should get one authoritative shard packet that already contains the exact contract summary, task roster, status rules, draft paths, and per-task hint/input summaries they need for the whole shard. Second, the helper/runtime should own more of the current-task progression so that task 2 through task 5 do not require the model to rediscover queue state by opening more files. The older files should still exist for debugging, but the prompt and tests should treat them as exceptional, not normal.

## Milestones

### Milestone 1: Replace the startup read bundle with one authoritative shard packet

At the end of this milestone, a novice should be able to inspect the recipe worker root and see one obvious file that explains the whole shard, the exact allowed workflow, and the location of every prepared draft. The prompt should direct the worker to read that one packet first, not the current five-file opening sequence. The acceptance proof is a worker-prompt regression that fails if the default startup instructions still require opening `worker_manifest.json`, `current_task.json`, `CURRENT_TASK_FEEDBACK.md`, and `OUTPUT_CONTRACT.md` before work begins.

### Milestone 2: Make later-task progression repo-owned instead of model-discovered

At the end of this milestone, the worker should not need to reread queue-level files or each task’s raw hint/input pair to understand what comes next. The helper should either precompute enough shard state in the packet and prepared drafts or expose a small current-task progression command that rewrites the repo-owned current-task surfaces as the worker advances. The acceptance proof is that the common worker path can complete a shard by following the packet and helper flow without needing reference-doc or tool-source reads.

### Milestone 3: Demote contract docs, examples, and tool source to fallback-only surfaces

At the end of this milestone, recipe workers still have access to `OUTPUT_CONTRACT.md`, mirrored examples, and `tools/recipe_worker.py`, but the normal prompt and packet do not require them. The acceptance proof is new regressions that inspect the generated prompt, manifest, and packet content and confirm that those files are described as fallback-only debugging surfaces.

### Milestone 4: Verify locally and keep benchmark spend gated

At the end of this milestone, the focused `.venv` tests pass for the worker prompt, helper packet generation, and any orchestrator or runner surfaces touched by the new flow. The benchmark rerun command is recorded here for later use, but it is not executed unless the user explicitly approves token spend.

## Plan of Work

Begin in `cookimport/llm/codex_farm_orchestrator.py`. Find the code that writes recipe worker prompts, entry files, and current-task sidecars. Replace the current startup wording with one normal path: open the shard packet, edit the prepared drafts, use `stamp-status` only for bulk terminal cases, and run `finalize-all`. The prompt should explicitly say that `OUTPUT_CONTRACT.md`, mirrored examples, and `tools/recipe_worker.py` are available only when the shard packet seems insufficient or inconsistent. Update the manifest so `entry_files` presents the shard packet as the primary entry surface and stops advertising the full bundle of startup files as mandatory reads.

Then extend `cookimport/llm/recipe_workspace_tools.py` to generate the shard packet and any supporting queue metadata it needs. The packet should be repo-written before the workspace session starts and must include, in plain language, the compact recipe output rules, allowed statuses, the task roster for the shard, each task’s scratch draft path, and short per-task summaries derived from the existing hints and inputs. Resolve the main ambiguity here in favor of precomputation: the packet should summarize the hint/input material itself rather than merely listing file paths, because the goal is to remove the worker’s need to open those files during the common path. If helper state needs to track which task is current, put that state in a helper-owned JSON file under `scratch/` and keep its schema documented in the plan and tests.

Still in `cookimport/llm/recipe_workspace_tools.py`, tighten the prepared-draft flow so the draft files themselves carry enough temporary helper metadata to make reference-doc and example reads unnecessary. That helper metadata must stay internal to the scratch phase and be stripped or ignored by `finalize-all` so the final output contract remains unchanged. If the cleanest implementation is to enrich `scratch/_prepared_drafts.json` instead of individual drafts, do that, but the packet must surface the same information in human-readable form.

Return to `cookimport/llm/codex_farm_orchestrator.py` and decide how the worker advances across later tasks. Prefer a repo-owned loop. The first acceptable design is a new helper command that updates `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`, and `current_task.json` to the next unfinished task from prepared helper state. The second acceptable design is a fully packed shard packet that is strong enough to make those current-task files unnecessary after startup. Pick one path and record the choice in the `Decision Log` when implementing. The non-negotiable requirement is that tasks 2 through N no longer depend on the model reopening raw hint/input files as a normal step.

If the chosen task-loop design changes the helper entry points or worker root contents, inspect `cookimport/llm/codex_exec_runner.py` only as needed to keep the workspace wrapper, current-task file mirroring, and command telemetry aligned. This plan does not seek new policy relaxation; any runner change should be a narrow adaptation to the new packet or helper state layout rather than a new shell boundary rule.

Update the short docs in `cookimport/llm/README.md` and `docs/10-llm/10-llm_README.md` after the code lands. Keep those notes short and operational. They should explain the new recipe shard packet, the intended worker flow, and which old files are now fallback-only.
