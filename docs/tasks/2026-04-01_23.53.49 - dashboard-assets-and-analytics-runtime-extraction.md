Problem statement

The dashboard renderer still stores large JavaScript assets inside Python modules, and analytics keeps benchmark manifest runtime/token enrichment logic duplicated across the dashboard collector and perf-report backfill path. That makes safe edits harder than they need to be.

Acceptance criteria

- `cookimport/analytics/dashboard_renderers/script_filters.py` and `script_tables.py` become thin Python seams over real `.js` asset files.
- Analytics manifest runtime/token extraction lives in one owner module reused by `dashboard_collect.py` and `perf_report.py`.
- Targeted analytics tests still pass without changing dashboard output contracts.

Verification steps

- Activate `.venv`, install dev deps if needed, and run:
  - `pytest tests/analytics/test_stats_dashboard.py tests/analytics/test_stats_dashboard_collectors.py tests/analytics/test_perf_report.py`
- Confirm the rendered dashboard still writes `assets/dashboard.js`.

Evidence

- `source .venv/bin/activate && pytest tests/analytics/test_stats_dashboard.py tests/analytics/test_stats_dashboard_benchmark_semantics.py tests/analytics/test_stats_dashboard_collectors.py tests/analytics/test_perf_report.py` -> `77 passed in 16.87s`
- `source .venv/bin/activate && pytest tests/architecture/test_ai_readiness_boundaries.py::test_second_wave_owner_roots_stay_small_and_explicit` still fails because the pre-existing unrelated guard for `cookimport/config/run_settings.py` line count is already over budget.

Constraints/gotchas + brief decision

- Keep runtime semantics stable: this is an ownership cleanup, not a product change.
- Preserve the small Python module names (`script_filters.py`, `script_tables.py`) so current import surfaces and architecture checks keep working while the JS source of truth moves to `dashboard_renderers/assets/`.

Rollback / compensation plan

- Revert the dashboard asset loader and analytics helper extraction together so the previous inline-Python asset/runtime ownership returns as one coherent state.
