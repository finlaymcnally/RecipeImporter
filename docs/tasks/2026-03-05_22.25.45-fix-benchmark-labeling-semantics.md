# Fix Misleading Benchmark Labeling for Hybrid And Deterministic Runs

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds. This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The user-visible goal is simple: when reading benchmark output in analytics, compare/control, and related summaries, a row should never look like `vanilla`, `AI off`, or a true deterministic baseline unless both the recipe LLM path and the line-role LLM path are actually off. After this change, retained hybrid rows such as `llm_recipe_pipeline=off` plus `line_role_pipeline=codex-line-role-v1` will surface as their own explicit benchmark semantics instead of being silently grouped into `vanilla` or `AI off` buckets.

A novice should be able to implement this plan and then verify the fix by rebuilding the dashboard, loading synthetic records that previously mislabeled hybrid rows, and observing that the UI now distinguishes at least these states: true deterministic baseline, recipe-only AI, line-role-only hybrid, full-stack Codex, and unknown/other. The same semantic split must also hold inside the compare/control backend so dashboard and CLI analysis agree.

## Surprises & Discoveries

- Observation: the current analytics stack treats `llm_recipe_pipeline=off` as effectively synonymous with `vanilla` when path hints are weak.
  Evidence: `cookimport/analytics/dashboard_render.py` and `cookimport/analytics/compare_control_engine.py` both use path-or-pipeline logic that returns `"vanilla"` whenever the recipe pipeline is `off`, without requiring `line_role_pipeline=off`.

- Observation: `AI off` in the dashboard is currently derived from the inferred benchmark variant, not from a full check of both AI-related pipelines.
  Evidence: `aiEffortLabelForRecord(...)` in both analytics implementations returns `"AI off"` when `benchmarkVariantForRecord(record) === "vanilla"`, which inherits the weak variant classifier.

- Observation: the dashboard already has one safe strict surface: the quick filter for “official benchmarks only” is path-based (`benchmark-vs-golden` + `single-offline-benchmark`) and is not the same as the contaminated QualitySuite population.
  Evidence: the current report verification found that official single-offline `vanilla` rows explicitly set `line_role_pipeline=off`, while the misleading semantics mainly affect hybrid rows outside that exact official pair contract.

- Observation: this is not only a frontend problem. The compare/control backend duplicates the same labeling rules, so fixing the browser only would leave CLI and dashboard disagreeing.
  Evidence: `cookimport/analytics/compare_control_engine.py` contains matching `benchmark_variant_for_record(...)`, `ai_model_label_for_record(...)`, and `ai_effort_label_for_record(...)` helpers.

- Observation: single-profile benchmark diagnostics still depend on paired `vanilla` / `codexfarm` variant identity even though the quick filter for "official benchmarks only" is intentionally narrower.
  Evidence: `tests/analytics/test_stats_dashboard.py::TestRenderer::test_runtime_summary_aggregates_tokens_across_latest_run_group` only recovered the expected `quality_delta_vs_vanilla` behavior after variant preservation was extended to `single-profile-benchmark` paths without widening the quick filter.

## Context and Orientation

This repository has two separate benchmark-labeling worlds that must not be conflated.

The first world is the official paired benchmark contract. In `cookimport/cli.py`, `_interactive_single_offline_variants(...)` builds the deliberate paired variants `vanilla` and `codexfarm`. In that builder, `vanilla` is explicitly forced to `llm_recipe_pipeline=off`, `line_role_pipeline=off`, and `atomic_block_splitter=off`. This is a true deterministic baseline and the `vanilla` name is correct there.

The second world is generic benchmark history and analytics records. Those records come from many places: QualitySuite, all-method runs, single-profile runs, and other benchmark families. In those records, the current analytics stack often infers `vanilla` from only two weak signals: the artifact path containing `/vanilla/`, or the recipe pipeline being `off`. That is unsafe because a row can have `llm_recipe_pipeline=off` while still using `line_role_pipeline=codex-line-role-v1`. The March 4 QualitySuite issue is exactly this shape.

The files that matter most are:

- `cookimport/analytics/dashboard_render.py`: emits the browser-side JavaScript that labels benchmark rows, assigns `AI off`, builds quick filters, and groups records by variant.
- `cookimport/analytics/compare_control_engine.py`: Python backend that mirrors dashboard semantics for compare/control CLI and agent mode.
- `cookimport/analytics/dashboard_collect.py` and `cookimport/analytics/dashboard_schema.py`: collector and schema layer for benchmark records; these files are where additive normalized semantics fields should be stored if needed.
- `cookimport/cli.py`: authoritative source of official paired `vanilla` / `codexfarm` behavior and the right place to preserve path-contract semantics for single-offline and single-profile sessions.
- `tests/analytics/test_stats_dashboard.py` and `tests/analytics/test_compare_control_engine.py`: dense regression suites that currently encode binary benchmark semantics assumptions and must be updated alongside the implementation.
- `docs/08-analytics/dashboard_readme.md` and `docs/08-analytics/08-analytics_readme.md`: analytics contract documentation that must explain the new benchmark semantics plainly.

Terms used in this plan:

- “Benchmark variant” means the authored label for an official paired run, such as `vanilla` or `codexfarm` under `single-offline-benchmark`.
- “AI assistance profile” means what AI paths actually ran in the row: deterministic-only, line-role-only, recipe-only, full-stack, or unknown.
- “Hybrid line-role-only” means `llm_recipe_pipeline=off` but `line_role_pipeline` is not off.
- “True deterministic” means both recipe and line-role AI paths are off.

The implementation must preserve the official paired benchmark contract while making the assistance profile explicit everywhere else.

## Plan of Work

### Milestone 1: Define a shared benchmark semantics contract

Start by writing down the semantic states in code before changing any UI labels. Add a small shared helper module under `cookimport/analytics/` such as `benchmark_semantics.py`. This module should expose a minimal set of functions that a novice can read and trust:

- a function that determines whether a record is part of the official paired benchmark contract (`official_single_offline` or `official_single_profile`) using path and run-family rules,
- a function that derives the true AI assistance profile from `run_config` and `run_config_summary`,
- a function that returns the display-safe benchmark variant label, which may be `vanilla`, `codexfarm`, or `other`, but only returns `vanilla` when the record is both path-matched and truly deterministic,
- a function that returns the new display-safe assistance label such as `AI off`, `Line-role only`, `Recipe only`, `Full stack`, or `Unknown`.

Do not use only `llm_recipe_pipeline` for this classification. The helper must inspect at least:

- `llm_recipe_pipeline`
- `line_role_pipeline`
- path segments such as `/single-offline-benchmark/`, `/single-profile-benchmark/`, `/vanilla/`, `/codexfarm/`

The helper should treat the path-derived `vanilla` label as authoritative only when `line_role_pipeline` is also off. If the path says `vanilla` but the run config shows line-role AI still on, the helper should downgrade the display variant to something non-misleading like `other` and emit the assistance profile `Line-role only`.

### Milestone 2: Wire the shared semantics into the compare/control backend

Update `cookimport/analytics/compare_control_engine.py` to stop owning its own private path-or-pipeline heuristic. Replace `_benchmark_variant_from_path_or_pipeline(...)`, `benchmark_variant_for_record(...)`, `ai_model_label_for_record(...)`, and `ai_effort_label_for_record(...)` with wrappers around the shared semantics helper.

The backend must expose enough fields that the compare/control CLI can reason over both concepts separately. Concretely:

- preserve `benchmark_variant` behavior for official paired rows so existing paired-run analytics still work,
- add a new derived field such as `ai_assistance_profile` or `benchmark_semantics_profile`,
- make `ai_effort` return `AI off` only for true deterministic rows or explicit runtime-failure rows, not for hybrid line-role-only rows,
- keep full-stack and recipe-only rows distinct from deterministic rows even when model or effort metadata is missing.

Update field catalogs and any derived-field lists so the new profile appears in compare/control discovery and filtering. This is important because the user needs to isolate hybrid rows directly without reverse-engineering them from raw config strings.

### Milestone 3: Wire the same semantics into dashboard rendering

Update `cookimport/analytics/dashboard_render.py` so the browser mirrors the exact same contract. The easiest safe path is to have Python serialize the semantic labels into each collected record and let the browser read them directly. If that is too invasive, the second-best approach is to port the exact helper rules into a single emitted JavaScript block and delete the weaker existing heuristic.

The dashboard-specific work must include:

- replacing `benchmarkVariantFromPathOrPipeline(...)` and `benchmarkVariantForRecord(...)` logic with semantics-aware classification,
- updating `aiModelLabelForRecord(...)` and `aiEffortLabelForRecord(...)` so hybrid rows no longer render `off` or `AI off`,
- making `isOfficialGoldenBenchmarkRecord(...)` stay strict to real official paired rows only,
- updating any chart series, run-group splits, per-label comparison headers, and quick-filter text that currently assume only `vanilla` and `codexfarm` exist,
- adding a visible field in the Previous Runs table for the new assistance profile if that is the simplest way to make the fix obvious to the user.

Do not destroy the paired-run UX. The dashboard should still present official `vanilla` / `codexfarm` pairs cleanly. The change is that hybrid rows outside that contract must stop impersonating `vanilla`.

### Milestone 4: Preserve official single-offline and single-profile comparison artifacts

Review `cookimport/cli.py` comparison and diagnostics helpers, especially the single-offline and single-profile comparison artifact writers, to make sure the official paired artifacts still read naturally as `codex_vs_vanilla_comparison` only when both sides are truly the authored pair.

This milestone is mostly defensive. The point is not to rename the official pair artifacts. The point is to ensure that any newly exposed semantic fields in analytics do not drift away from the official benchmark contract documented in the CLI and benchmark docs.

If useful, add additive metadata to comparison payloads or dashboard rows describing the assistance profile that produced each variant. Keep the existing schemas stable unless a schema bump is unavoidable; additive fields are preferred.

### Milestone 5: Tests and migration of assumptions

Update and expand tests so the new semantics are locked down. Focus on behavior, not implementation details.

In `tests/analytics/test_compare_control_engine.py`, add explicit fixture rows for:

- true deterministic official `vanilla`
- official `codexfarm`
- hybrid line-role-only row (`llm_recipe_pipeline=off`, `line_role_pipeline=codex-line-role-v1`)
- recipe-only AI row (`llm_recipe_pipeline=codex-farm-3pass-v1`, `line_role_pipeline=off`)
- ambiguous/metadata-poor row that should land in `other` / `Unknown`

Assert the new variant and assistance labels for each case.

In `tests/analytics/test_stats_dashboard.py`, replace brittle assertions such as “if variant is `vanilla`, effort label must be `AI off`” with stronger semantic assertions. Add a regression case that reproduces the March 4 hidden bug shape and proves it does not show up as `vanilla` or `AI off` anywhere in the generated JS-derived results.

If new serialized record fields are added through the collector layer, add coverage in `tests/analytics/test_perf_report.py` or any collector tests so the fields survive history, report enrichment, and render-time use.

### Milestone 6: Documentation and language cleanup

After the code and tests are stable, update the analytics and benchmark documentation so the contract is understandable without reading source.

At minimum update:

- `docs/08-analytics/dashboard_readme.md`
- `docs/08-analytics/08-analytics_readme.md`
- any benchmark README section that currently implies `llm_recipe_pipeline=off` is enough to count as `vanilla`

The documentation must plainly say:

- `vanilla` is reserved for the official deterministic paired benchmark variant,
- hybrid line-role-only rows are not vanilla,
- `AI off` means both recipe and line-role AI paths are off.

## Validation and Acceptance

Acceptance is behavioral.

### Scenario 1: True deterministic row stays vanilla

Given a benchmark record whose path is an official single-offline `.../vanilla` directory and whose run config has `llm_recipe_pipeline=off` and `line_role_pipeline=off`, the dashboard and compare/control backend should both report:

- benchmark variant: `vanilla`
- assistance profile: `AI off`

### Scenario 2: Hybrid line-role-only row is no longer vanilla

Given a benchmark record with `llm_recipe_pipeline=off` and `line_role_pipeline=codex-line-role-v1`, the dashboard and compare/control backend should both report:

- benchmark variant: not `vanilla`
- assistance profile: `Line-role only`
- AI effort label: not `AI off`

### Scenario 3: Recipe-only row is distinct from full-stack Codex

Given a row with `llm_recipe_pipeline=codex-farm-3pass-v1` and `line_role_pipeline=off`, the system should label it separately from both deterministic and full-stack rows.

### Scenario 4: Official quick filter remains strict

Given the `Official benchmarks only` dashboard quick filter, the filtered result set should continue to include only the true authored single-offline paired rows, not hybrid QualitySuite rows that merely happen to have recipe AI off.

### Scenario 5: Dashboard and compare/control agree

For the same synthetic fixtures, the Python compare/control engine and the emitted dashboard JS must return the same variant and assistance labels. Any mismatch is a failure.

## Idempotence and Recovery

This plan is designed to be implemented safely in small, repeatable steps.

The shared semantics helper should be additive at first. During the refactor, keep existing public names such as `benchmark_variant_for_record(...)` alive as wrappers so tests and callers continue to run while the behavior changes underneath. Only remove dead helper code after the new tests pass.

If a step goes sideways, recovery is straightforward:

- revert the most recent helper integration change,
- rerun the focused analytics tests,
- reintroduce the next change one surface at a time.

The dashboard rebuild is safe to repeat. The analytics tests are safe to rerun repeatedly.
