# Make KNOWLEDGE high-evidence and generalize the line-role prose boundary

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The most recent line-role pass fixed the narrow `HOWTO_SECTION` and outside-recipe `INSTRUCTION_LINE` bait in `saltfatacidheatcutdown`, but the benchmark still regressed because the deeper problem moved to the prose boundary: the system is unstable about when outside-recipe prose should be `KNOWLEDGE` and when it should stay `OTHER`. After this change, line-role should treat `KNOWLEDGE` the same way it now treats `HOWTO_SECTION`: as a high-evidence label, not a default sink for anything educational-sounding.

The second goal is to keep Salt Fat Acid Heat as a hard canary without turning it into the spec. After this change, validation should explicitly prove two things at once: the system does not over-upgrade memoir, blurbs, exhortation, and book-framing prose into `KNOWLEDGE`, and it still preserves real explanatory/reference prose in at least one contrasting knowledge-heavy source. The intended result is a general prose-boundary policy that behaves coherently across books, not a Salt Fat patch.

## Context and Orientation

The active line-role stage lives in `cookimport/parsing/canonical_line_roles.py`. It labels each atomic line with one global structural label such as `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `HOWTO_SECTION`, `KNOWLEDGE`, or `OTHER`. The line-role runtime is label-first: deterministic code proposes labels, Codex may correct them for task packets, and parser-owned sanitizers may still reject unsupported outputs before grouped spans are projected downstream.

The recent HOWTO hardening is already in place. `canonical_line_roles.py` now has a book-level `HOWTO_SECTION` availability seam, worker task files carry packet summaries and policy text, and prompts/hints already fence recipe-internal structure more conservatively. Current task packets may also carry reference-only neighbor overlap in `context_before_rows` and `context_after_rows`. That transport change should remain intact. This new plan is about the outside-recipe prose boundary, not about re-opening the structure work.

In this repository, `KNOWLEDGE` should mean reusable explanatory or reference prose about cooking, technique, science, or ingredients. It should not mean “anything food-related outside a recipe.” Memoir, blurbs, endorsement lines, book-framing prose, narrative anecdotes, exhortation such as “keep reading,” and many action-verb advice paragraphs may still belong in `OTHER` even if they mention cooking ideas. The current code seams that matter are the parser heuristics `_looks_knowledge_heading_with_context(...)`, `_looks_knowledge_prose_with_context(...)`, `_looks_pedagogical_knowledge_prose(...)`, `_looks_endorsement_credit(...)`, `_looks_narrative_prose(...)`, and the final `_sanitize_prediction(...)` step.

The benchmark evidence that motivates this plan is the Salt Fat run under `data/golden/benchmark-vs-golden/2026-03-21_16.10.40/single-book-benchmark/saltfatacidheatcutdown/`. In that run, the specific structure errors improved, but CodexFarm still lost to vanilla on strict accuracy because `OTHER -> KNOWLEDGE` and `KNOWLEDGE -> OTHER` overwhelmed the benefit. The most informative wrong-label samples are memoir-like or editorial paragraphs getting promoted to `KNOWLEDGE`, and genuine explanatory passages or headings being flattened to `OTHER`.

Two repository surfaces matter for proving that a future fix is general. First, the parsing and LLM docs in `docs/04-parsing/04-parsing_readme.md` and `docs/10-llm/10-llm_README.md` describe the current line-role packet/hint/prompt authority split. Second, the gold exports under `data/golden/pulled-from-labelstudio/*/exports/canonical_span_labels.jsonl` give deterministic contrast sources. This plan uses `saltfatacidheatcutdown` as the canary for memoir/editorial drift and `thefoodlabcutdown` as the knowledge-heavy contrast source that prevents global `KNOWLEDGE` suppression.

## Milestones

### Milestone 1: Audit what currently counts as `KNOWLEDGE`

At the end of this milestone, the repository will have one clear understanding of which current heuristics push rows into `KNOWLEDGE` and which current benchmark examples show those heuristics are too broad. The implementation should not start by editing prompt prose in the dark. It should start by explicitly mapping the current positive and negative evidence seams in `canonical_line_roles.py` and matching them to real benchmark rows.

This milestone matters because the current regression is bidirectional. If the next pass only blocks false-positive `KNOWLEDGE`, it risks flattening real explanatory prose. The acceptance for this milestone is a concrete list of over-broad cues and must-preserve cues, grounded in repo code and benchmark evidence.

### Milestone 2: Make outside-recipe `KNOWLEDGE` high-evidence and low-default

At the end of this milestone, parser-owned line-role policy will require stronger positive evidence before outside-recipe prose can become `KNOWLEDGE`. The implementation can use a new evidence seam, more explicit negative cues, a stricter sanitizer, or some combination, but it must make “outside recipe, cooking-themed, but still `OTHER`” a first-class supported outcome.

The acceptance for this milestone is one obvious code seam where `KNOWLEDGE` evidence is judged and one obvious parser-owned fallback path when the evidence is insufficient. The goal is not to delete `KNOWLEDGE`; it is to narrow the contract so the label means “reusable explanation/reference” rather than “food-related prose.”

### Milestone 3: Preserve true explanatory prose

At the end of this milestone, a novice reader should be able to see that the implementation preserves genuine `KNOWLEDGE` rows. The tests should include causal explanations, reference headings, and concise domain-knowledge prose that still classify as `KNOWLEDGE` even after the new narrowing.

This milestone matters because the current wrong-label sample already shows the opposite failure too: real knowledge headings and explanatory prose collapsing to `OTHER`. The acceptance for this milestone is at least one compact fixture where `KNOWLEDGE` still wins for the right reasons.

### Milestone 4: Reject memoir, blurbs, and exhortation in a general way

At the end of this milestone, the system should explicitly reject the prose families that currently dominate the Salt Fat errors: memoir/anecdote, endorsement and attribution blurbs, book-framing exhortation, and advice that reads like broad encouragement rather than reference or explanation. The important part is that the rule should be general and textual, not “special-case these Salt Fat lines.”

The acceptance for this milestone is a compact boundary suite in `tests/parsing/test_canonical_line_roles.py` showing that those prose families stay `OTHER` while real reference/explanation prose still stays `KNOWLEDGE`.

### Milestone 5: Validate without overfitting

At the end of this milestone, deterministic validation should pass and the proof story should explicitly cover both the Salt Fat canary and the knowledge-heavy contrast source. The default path is deterministic: parser tests plus any repo-local gold-count or fixture assertions. A live benchmark rerun can be added only after explicit user approval, and if run it must include at least Salt Fat and `thefoodlabcutdown`.

The acceptance for this milestone is not “best Salt Fat score.” It is “the prose boundary is expressed in general policy terms, covered by tests, and visibly not dependent on one cookbook title.”

## Validation and Acceptance

The implementation is acceptable only if all of the following are true.

The code now has one obvious parser-owned seam where outside-recipe `KNOWLEDGE` evidence is judged. A novice reader should be able to locate that seam and understand why a row becomes `KNOWLEDGE` instead of `OTHER`.

The new policy is not Salt Fat-specific. No logic should key off one book title or one benchmark path. The acceptance story must be phrased in general terms such as “memoir/editorial prose stays `OTHER` even when it mentions cooking” and “true explanatory/reference prose still becomes `KNOWLEDGE`.”

The tests cover both sides of the boundary. There must be compact fixtures where explanatory/reference prose stays `KNOWLEDGE`, and compact fixtures where memoir, blurbs, exhortation, or action-verb advice stay `OTHER`.

The HOWTO and outside-recipe instruction hardening must remain intact. This plan must not reopen the earlier structure regression by making `KNOWLEDGE` so weak that old `HOWTO_SECTION` or `INSTRUCTION_LINE` mistakes return through a different path.

Deterministic validation must explicitly mention both Salt Fat and the knowledge-heavy contrast source. If a live benchmark rerun is approved, the results must be interpreted in cross-book policy terms, not only in terms of Salt Fat.

No live-token benchmark or model-backed validation is required unless the user explicitly approves model spend for that rerun.

## Idempotence and Recovery

This plan should be implemented additively. The current packet/hint/prompt transport already exists and should remain stable. Prefer adding or narrowing parser-owned evidence seams, sanitizers, and tests over redesigning the runtime or adding new live stages.

If an attempted fix improves Salt Fat by flattening true explanatory prose back to `OTHER` everywhere, treat that as a regression, not a win. The safe rollback is to revert only the new `KNOWLEDGE` narrowing seam or sanitizer clause, then rerun the compact prose-boundary tests before touching the broader validation set.

If `thefoodlabcutdown` turns out not to be sufficient as the only knowledge-heavy contrast source, do not force the implementation around it. Update this plan, select an additional deterministic contrast source from repo-local gold, and continue. The governing principle is “prove real `KNOWLEDGE` survives,” not loyalty to one book.
