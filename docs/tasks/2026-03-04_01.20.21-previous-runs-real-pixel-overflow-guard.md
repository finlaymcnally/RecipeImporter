# Problem

Previous Runs still appeared to grow rightward in live usage. Prior string-level/CSS checks were insufficiently real.

# Acceptance Criteria

- A browser-level pixel test measures document/section overflow over repeated rerenders.
- Test goes red for real overflow and green after fix.
- Runtime includes containment for long rendered compare/control tokens and stale persisted width state.

# Verification

- `source .venv/bin/activate && pytest tests/analytics/test_stats_dashboard.py -k 'previous_runs_stays_within_viewport_pixels_after_rerenders or dashboard_js_clamps_persisted_table_column_widths or previous_runs_table_has_horizontal_scroll_css or benchmark_trend_host_rerender_starts_from_clean_host'`
- `source .venv/bin/activate && pytest tests/analytics/test_stats_dashboard.py`

# Evidence

- Red: pixel overflow assertion failed with `max_doc_overflow_px = 2195`.
- Green targeted: `4 passed, 68 deselected in 6.02s`.
- Green full dashboard suite: `72 passed in 14.40s`.

# Constraints / Gotchas

- Pixel harness must not depend on external Highcharts CDN stability; test stubs chart scripts.
- Local table horizontal scrolling must remain intact (do not shrink table contract itself).

# Decision

Added a Playwright pixel harness over generated dashboard HTML; fixed overflow by section containment + long-token wrapping in compare/control results/discovery cards; added persistent column-width normalization/clamp (`72..1200px`) across load/save/resize so stale browser UI-state cannot re-inflate layout width.
