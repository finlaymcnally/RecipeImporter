# Migrate Recipe Codex Outputs To Native Nested Objects

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan is self-contained. It assumes no prior knowledge of earlier March CodexFarm plans, although it intentionally builds on the existing merged repair prototype already present in the repository.

## Purpose / Big Picture

After this change, the recipe Codex paths should stop asking the model to return JSON strings that themselves contain nested JSON objects. Instead, Codex should return native nested objects directly in its structured output. In plain terms, the model should return a recipe object as an object, not as a quoted string that recipeimport later has to parse again.

The user-visible effect is that the recipe Codex pipeline becomes easier to reason about, less fragile, and closer to the real product goal: “show an LLM a recipe JSON and have it fix small errors.” The concrete proof is that a one-book benchmark run using the merged repair pipeline completes successfully, the raw LLM output bundles contain nested JSON objects instead of JSON-string wrapper fields, and the downstream recipeimport artifacts are unchanged in meaning.

  Rationale: keeping a mixed world where some recipe outputs are native objects and others are JSON-string wrappers would preserve the same mental overhead that this plan is trying to remove. The recipe family is small enough that a coordinated migration is reasonable.

- Decision: keep backward-compatible loaders during and after the migration.

- Decision: do not widen this plan to non-recipe pipelines such as pass4 knowledge or pass5 tags.
  Rationale: the user’s goal and the current pain both live in the recipe extraction and repair path. Pulling in unrelated pipelines would slow the migration and increase risk without improving the “LLM fixes recipe JSON” workflow.
  Date/Author: 2026-03-14 / Codex

- Decision: preserve downstream artifact meaning even if raw Codex payload shape changes.
  Rationale: this plan is about removing JSON-in-JSON encoding, not redesigning recipeimport’s internal recipe semantics. A successful migration changes the transport shape while leaving the resulting schema.org recipe, `RecipeDraftV1`, warnings, and mapping meaning intact.
  Date/Author: 2026-03-14 / Codex

- Decision: define the migrated nested objects with explicit required keys plus nullable fields instead of trying to preserve old “omit the key entirely” prompt behavior.

## Context and Orientation

In this repository, a “pipeline asset” is the small pack of JSON schema and prompt files under `llm_pipelines/` that tells Codex what to read and what shape to return. The pipeline JSON file in `llm_pipelines/pipelines/*.json` points at a prompt template in `llm_pipelines/prompts/*.prompt.md` and an output schema in `llm_pipelines/schemas/*.output.schema.json`.

The active recipe Codex logic lives in `cookimport/llm/`. The most important files for this plan are:

- `cookimport/llm/codex_farm_contracts.py`, which defines the Pydantic models that recipeimport uses after it reads a Codex output file.
- `cookimport/llm/codex_farm_orchestrator.py`, which runs pass1, pass2, pass3, or the merged repair stage, then converts Codex output into recipeimport’s downstream artifacts.
- `llm_pipelines/schemas/recipe.schemaorg.v1.output.schema.json`, which currently says `schemaorg_recipe` and `field_evidence` are strings even though recipeimport ultimately wants dictionaries.
- `llm_pipelines/schemas/recipe.final.v1.output.schema.json`, which currently says `draft_v1` and `ingredient_step_mapping` are strings even though recipeimport ultimately wants dictionaries.
- `llm_pipelines/schemas/recipe.merged-repair.v1.output.schema.json`, which currently says `canonical_recipe` and `ingredient_step_mapping` are strings even though recipeimport ultimately wants an object and a dictionary.
- `llm_pipelines/prompts/recipe.schemaorg.compact.v1.prompt.md`, `llm_pipelines/prompts/recipe.final.compact.v1.prompt.md`, and `llm_pipelines/prompts/recipe.merged-repair.compact.v1.prompt.md`, which currently instruct Codex to serialize some nested objects as JSON strings.
- `cookimport/llm/fake_codex_farm_runner.py`, which provides deterministic fake payloads for tests and must emit the new shape once the assets change.
- `tests/llm/test_codex_farm_contracts.py`, `tests/llm/test_codex_farm_orchestrator.py`, and `tests/llm/test_llm_pipeline_pack_assets.py`, which are the fastest tests that prove the migration works.

The phrase “JSON-string wrapper field” means a field whose schema type is `string`, but whose contents are expected to be a serialized JSON object such as `"{\"title\": \"Toast\"}"`. That design works, but it forces the model to escape JSON inside JSON and forces recipeimport to parse the nested object a second time. This plan removes that wrapper layer for recipe outputs.

For this plan, “native nested object” means that the raw Codex output JSON file already contains the object in normal JSON form. For example, instead of this:

    {
      "canonical_recipe": "{\"title\":\"Toast\",\"ingredients\":[\"1 slice bread\"],\"steps\":[\"Toast the bread.\"]}"
    }

the output should look like this:

    {
      "canonical_recipe": {
        "title": "Toast",
        "ingredients": ["1 slice bread"],
        "steps": ["Toast the bread."],
        "description": null,
        "recipeYield": null
      }
    }

The user-visible product goal does not change. The model is still assembling or repairing a recipe from grounded source rows. The migration only changes how that structured answer is encoded.

## Milestones

### Milestone 1: Define the native object contracts and keep the loaders compatible

At the end of this milestone, the recipe output schemas and internal contracts are fully specified for native nested objects, and recipeimport can read either the new native object format or the old JSON-string wrapper format. Nothing user-visible has to run yet, but the contract layer is stable enough that fixtures and prompts can migrate without ambiguity.

Work in `cookimport/llm/codex_farm_contracts.py` first. The end state must preserve the existing runtime types: `Pass2SchemaOrgOutput.schemaorg_recipe` remains a dictionary, `Pass2SchemaOrgOutput.field_evidence` remains a dictionary, `Pass3FinalDraftOutput.draft_v1` remains a dictionary, `Pass3FinalDraftOutput.ingredient_step_mapping` remains a dictionary, `MergedRecipeRepairOutput.canonical_recipe` remains a `MergedCanonicalRecipe`, and `MergedRecipeRepairOutput.ingredient_step_mapping` remains a dictionary. The crucial point is that the contract loader must accept those values directly as nested JSON objects instead of requiring them to be hidden inside strings.

Then define the exact output-schema shapes under `llm_pipelines/schemas/`. Be explicit. The migration should cover the full recipe output family:

- `recipe.schemaorg.v1.output.schema.json`
- `recipe.final.v1.output.schema.json`
- `recipe.merged-repair.v1.output.schema.json`

For each of those schemas, remove the string wrapper shape for fields that are semantically objects. Preserve nullable fields using `["string", "null"]` or the object equivalent where needed, and continue to list every top-level property in `required` because Codex structured outputs require that stricter subset. For nested objects, do not leave the shape vague. Define the keys that recipeimport actually depends on.

Acceptance for this milestone is local and narrow: the asset tests and contract tests pass, and it is clear from reading the schemas that recipe Codex outputs are now ordinary nested JSON objects.

### Milestone 2: Migrate prompt assets, fake outputs, and orchestration tests

At the end of this milestone, Codex would be asked for the new native-object shape, and all deterministic fake outputs and unit tests expect the same thing. A novice should be able to read the prompt templates and understand that the model now returns actual objects, not escaped JSON strings.

Update the prompt files in `llm_pipelines/prompts/` so they describe object returns plainly. Remove wording like “return as a JSON string” for the affected recipe fields. Keep the grounded-behavior instructions unchanged. The model should still be told to preserve source truth, avoid inventions, emit warning codes, and return `ingredient_step_mapping_reason` when the mapping is empty. Only the encoding strategy changes.

Update `cookimport/llm/fake_codex_farm_runner.py` so fake recipe outputs mirror the new shape. This matters because many tests never call live Codex and instead assert on the fake runner’s payloads. A partial migration where prompts and schemas change but the fake runner still emits stringified objects would create confusing failures and hide real regressions.

Next, update the orchestrator and contract tests that construct literal output payloads. Most runtime logic should not need deep changes because it already consumes real dictionaries after validation, but the tests must stop normalizing around the old wrapper format. When a test wants to exercise compatibility behavior, it should do so explicitly by passing the old stringified form and naming that as a compatibility case.

Acceptance for this milestone is that the fastest test slice passes and clearly proves both behaviors: the normal path uses native nested objects, and the compatibility path still accepts old JSON-string wrapper payloads.

### Milestone 3: Prove the merged repair benchmark works with native objects

At the end of this milestone, the active merged repair benchmark path has been exercised on one real book and the run completes past the previous failure point. The novice proof is not “the code compiles”; it is “the benchmark writes merged repair outputs, the raw Codex output bundles contain objects instead of stringified JSON, and downstream evaluation artifacts are still produced.”

Use `saltfatacidheatcutdown` as the first live book because it already produced the failure that motivated this plan. Run the same style of single-offline benchmark that previously died in the merged repair stage. Then inspect the raw LLM output bundle under the new run root and confirm that the relevant fields are nested objects. Finally, ensure the downstream derived artifacts still exist: the merged repair audit, the prediction-run output, and the evaluation report.

If the merged repair path passes, run at least one narrow comparison against the vanilla baseline or prior merged-repair benchmark root and record the outcome in this plan. The purpose is not to optimize quality yet; it is to prove that the new native-object transport shape behaves like the old semantics while being easier to understand and less brittle.

Acceptance for this milestone is that the benchmark runs end-to-end and the raw payload shape is visibly simpler without changing the meaning of the evaluated recipe output.

## Plan of Work

Start in `cookimport/llm/codex_farm_contracts.py`. Review each recipe output model and make the compatibility rule explicit: the runtime should accept either a real nested object or the older stringified JSON object. Keep `_coerce_json_object_field(...)` in place or split it into a clearer helper if that improves readability, but do not remove old-artifact compatibility during this plan.

Next, edit the schema assets in `llm_pipelines/schemas/`. The schema files must become the authoritative description of the native nested output shape. Do not leave them as a half-migration where the prompt says “object” but the schema still says “string.” Be deliberate about nested structure. `MergedCanonicalRecipe` should be represented directly as an object with `title`, `ingredients`, `steps`, `description`, and `recipeYield`. `ingredient_step_mapping` should be represented as an object whose keys are ingredient indexes encoded as strings and whose values are arrays of step indexes. `draft_v1` should be represented as the existing final-draft object shape that recipeimport already expects, not as a string wrapper around that object.

Then edit the prompt assets so they teach Codex the same object shape that the schemas enforce. The prompt wording must no longer instruct the model to serialize nested objects as strings. Instead, it should say plainly that `canonical_recipe`, `schemaorg_recipe`, `draft_v1`, `field_evidence`, and `ingredient_step_mapping` are JSON objects in the returned payload. Preserve all grounding, warning, and omission rules.

After the assets are updated, update `cookimport/llm/fake_codex_farm_runner.py` and the focused tests under `tests/llm/`. This migration is successful only if the test fixtures reflect the simpler transport. Add or update tests that explicitly prove both the new normal path and the old compatibility path. `tests/llm/test_llm_pipeline_pack_assets.py` should continue enforcing that every top-level schema property appears in `required`, and it should be extended if needed to assert that the migrated fields are no longer typed as strings. `tests/llm/test_codex_farm_contracts.py` should include cases where native nested objects are accepted directly.

Finally, rerun a single real merged repair benchmark book and inspect the raw output bundle. The benchmark command and the artifact inspection command must be recorded here after implementation. If the live run reveals a nested-schema limitation from Codex’s structured-output subset, document the exact limitation in `Surprises & Discoveries`, adjust the schema to stay within the supported subset, and keep the underlying “native nested object” goal intact.

## Validation and Acceptance

Validation is successful only when all of the following are true.

First, the narrow asset and contract tests pass, proving that the schema assets, prompts, and loaders all agree on the new native nested-object format.

Second, the orchestrator tests pass, proving that the recipe pipeline still derives schema.org, `RecipeDraftV1`, warnings, and mapping semantics correctly from the migrated Codex payloads.

Third, a live merged repair benchmark completes on `saltfatacidheatcutdown` without failing at the schema-validation step. The run root must contain merged repair outputs, not just pass1 inputs and a Codex error trace.

Fourth, inspection of a raw output bundle must show native nested objects instead of JSON-string wrapper fields. A novice should be able to open the file and recognize the recipe object directly without mentally decoding escaped JSON.

Fifth, the downstream meaning must remain intact: evaluation still runs, `merged_repair_audit` still exists, and the derived recipe outputs are materially the same kind of artifacts as before.

## Idempotence and Recovery

This migration should be implemented additively and is safe to retry. The schemas, prompts, and tests can be edited repeatedly without damaging benchmark data. The live benchmark step writes to a fresh timestamped output root, so rerunning it does not overwrite earlier evidence.

If implementation stalls halfway, keep the compatibility loaders in place and do not delete support for the old stringified-object shape. That allows older fixtures and benchmark artifacts to continue loading while the asset migration is finished. If a live run exposes a Codex structured-output limitation that makes one nested object shape invalid, revise only the schema and prompt shape needed to stay inside the supported subset. Do not fall back to JSON-string wrapper fields unless the plan records a concrete reason and updates the `Decision Log`.
