Problem statement

`cookimport bench oracle-upload` still fails after payload sharding because the local Oracle browser wrapper has drifted back to a visible-browser path that launches Chromium without xvfb and steals desktop focus.

Acceptance criteria

- Browser-mode benchmark uploads no longer depend on the drifted local wrapper.
- The benchmark uploader invokes Oracle with the known Linux off-screen Chromium path and local debug-host override.
- Oversized bundle sharding continues to work unchanged.

Verification steps

- Activate `.venv`.
- Run `pytest tests/bench/test_benchmark_oracle_upload.py`.

Evidence

- Browser command assembly now proves the uploader calls the real Oracle CLI with `--engine browser`, `--browser-manual-login`, `--browser-chrome-path /home/mcnal/.local/bin/chromium-nosandbox-xvfb`, forced browser attachments, and `ORACLE_BROWSER_REMOTE_DEBUG_HOST=127.0.0.1`.
- Existing dry-run and oversized-shard coverage still exercises the updated transport path.
- Verification result: `source .venv/bin/activate && pytest tests/bench/test_benchmark_oracle_upload.py` -> `8 passed, 1 warning in 2.25s`.

Constraints/gotchas + decision

- The current `/home/mcnal/.local/bin/oracle-browser-headless` script is not actually headless here; it forces a visible browser, pins port `9222`, and launches `/home/mcnal/.local/bin/chromium-nosandbox`, which fails without X and can steal focus.
- Fix in repo by bypassing that wrapper from `cookimport/bench/oracle_upload.py` and using the previously documented Linux xvfb launcher directly.

Rollback / compensation plan

- Revert the Oracle browser transport change if it breaks uploads; the benchmark bundle format and dry-run fallback remain untouched.
