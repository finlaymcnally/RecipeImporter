## Problem statement
Dashboard showed token totals but did not quantify quality gained per token, so token spend vs quality improvement was hard to judge.

## Acceptance criteria
- `Benchmark Runtime` shows quality/token efficiency for the latest run-group.
- Runtime card includes delta quality efficiency vs vanilla when both variants exist.
- Runtime card includes quality/token peer comparison across token-spending run-groups.
- `Previous Runs` exposes a numeric calculated field `quality_per_million_tokens` for sort/filter/compare.

## Verification steps
- `source .venv/bin/activate && pytest tests/analytics/test_stats_dashboard.py -k "test_js_renders_previous_runs_table_and_links_timestamp_to_artifact or test_js_supports_previous_runs_column_header_filters or test_runtime_summary_aggregates_tokens_across_latest_run_group or test_previous_runs_quality_per_million_tokens_calculation"`
- `source .venv/bin/activate && pytest tests/analytics/test_stats_dashboard.py`

## Evidence
- Added JS renderer/harness tests for runtime quality-token summaries and derived Previous Runs quality/token field behavior.

## Constraints / decision
- Quality score selection is deterministic: `strict_accuracy`, then `macro_f1_excluding_other`, then `f1`.
- Efficiency is normalized as quality per 1,000,000 discounted tokens to keep values human-readable.

## Rollback / compensation
- Remove `quality_per_million_tokens` derivation and runtime quality-efficiency rows, restoring token-only runtime diagnostics.
