# Task: Elapsed spinner ticker + post-candidate importer phases

## Problem statement

Long phases after `candidate X/Y` extraction left the spinner text unchanged, which looked frozen even though conversion/upload was still running.

## Acceptance criteria

- Callback-driven CLI status wrappers append elapsed seconds after prolonged unchanged phase text.
- Label Studio import/decorate, benchmark import, and bench run/sweep use the shared elapsed-status wrapper.
- EPUB and PDF converters emit explicit post-candidate phase callbacks before finalization.
- Tests cover elapsed-status formatting/ticker behavior and post-candidate progress messages.

## Verification steps

- `. .venv/bin/activate && pip install -e .[dev]`
- `. .venv/bin/activate && pytest -q tests/test_labelstudio_benchmark_helpers.py -k "status_progress_message or run_with_progress_status"`
- `. .venv/bin/activate && pytest -q tests/test_epub_importer.py tests/test_pdf_importer.py -k post_candidate_progress`
- `. .venv/bin/activate && pytest -q tests/test_bench_progress.py`

## Evidence

- CLI status helper now renders messages like `... Extracting candidate 46/46... (17s)` when a phase stays unchanged past threshold.
- EPUB/PDF progress callbacks now continue with `Analyzing standalone knowledge blocks...` and `Finalizing ... extraction results...` after candidate loops.
- Targeted tests pass for new helper behavior and importer progress messages.

## Constraints and decisions

- Reused existing `progress_callback` plumbing; no second indicator system was added.
- Elapsed ticker threshold defaults to 8s and updates once per second to stay readable without noisy churn.
- Importer phase updates are additive text only; extraction semantics and output artifacts are unchanged.

## Rollback plan

- Revert `_run_with_progress_status` usage in `cookimport/cli.py` and restore direct `console.status(...)/status.update(...)` wrappers.
- Revert EPUB/PDF post-candidate `_notify(...)` additions if message volume needs to be reduced.
