# Fix line-role false `workspace_final_message_missing_output` kills after helper completion

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a canonical line-role worker that already ran the repo-owned `python3 -m cookimport.parsing.canonical_line_roles.same_session_handoff` command successfully will finish cleanly instead of being watchdog-killed for `workspace_final_message_missing_output`. The user-visible result is that line-role runtime artifacts stop claiming "final message, missing output" for sessions that were actually already complete, and telemetry stops counting those cases as watchdog recoveries or `repair_recovered`.

You should be able to see this working without rerunning a CodexFarm benchmark. The proof should come from narrow tests in `.venv` plus one synthetic runtime test that reproduces the bad sequence: helper reports `completed`, the worker sends its final message, output visibility lags briefly, and the runtime still exits as clean success rather than watchdog kill plus recovery.

## Surprises & Discoveries

- Observation: The archived bad run is not merely "output appeared a moment after the kill." The worker event stream already contains the successful helper command before the final agent message.
  Evidence: `events.jsonl` for `worker-001` includes command `python3 -m cookimport.parsing.canonical_line_roles.same_session_handoff` with aggregated output `{"completed": true, "final_status": "completed", "status": "completed", ...}` and only later emits the final agent message.

- Observation: The current runtime postmortem contradicts the live watchdog decision.
  Evidence: `live_status.json` says `reason_code = "workspace_final_message_missing_output"` at `2026-03-31T23:53:18.487697Z`, while `final_message_recovery_assessment.json` created at `2026-03-31T23:53:19.158585Z` says `same_session_completed = true`, `diagnosis_code = "completed"`, and `workspace_output_status.complete = true`.

- Observation: The live runner metadata was already rich enough to preserve helper completion even after later commands. The fragile part was making every line-role path consistently prefer the stage-helper completion record over the last arbitrary command.
  Evidence: `tests/llm/test_codex_exec_runner.py::test_build_codex_exec_live_snapshot_surfaces_completed_handoff_command_result` shows `last_completed_command` can be a later `editable_task_file --summary` call while `last_completed_stage_helper_command` still correctly points at `same_session_handoff`.

- Observation: The runtime tests had drifted behind the actual quiescence/grace settings and were still assuming 2-3 second clean termination.
  Evidence: the synthetic watchdog/runtime fixtures only passed again after deriving their final callback timestamps from `_LINE_ROLE_WORKSPACE_COMPLETION_QUIESCENCE_SECONDS` and `_LINE_ROLE_FINAL_MESSAGE_MISSING_OUTPUT_GRACE_SECONDS`, which both default to 15 seconds in current settings.

- Observation: Existing tests already protect the two edge cases around this seam, but not the one that actually happened in production.
  Evidence: `tests/parsing/test_canonical_line_roles.py` covers `test_line_role_workspace_watchdog_kills_after_final_message_missing_output_grace_window`, `test_line_role_workspace_watchdog_completes_after_authoritative_same_session_success`, and `test_line_role_workspace_watchdog_waits_when_helper_completed_but_outputs_not_yet_present`, but none model a successful helper command in the event stream before file visibility catches up.

## Decision Log

- Decision: Fix this at the live supervision seam first, not by adding more deterministic recovery after the kill.
  Rationale: The archived worker was already done. Recovery currently papers over a false negative, but it still poisons telemetry and spends needless recovery paths. The first job is to stop misclassifying success as kill.
  Date/Author: 2026-03-31 / Codex

- Decision: Extend `CodexExecLiveSnapshot` with explicit recent-command completion metadata rather than making line-role parse raw event JSON itself.
  Rationale: The runner already owns event parsing. Exposing structured metadata there keeps the stage watchdog simple and makes the signal reusable by other workspace-worker stages if needed later.
  Date/Author: 2026-03-31 / Codex

- Decision: Treat `last_completed_stage_helper_command` as the durable live helper signal and use `last_completed_command` only as fallback compatibility.
  Rationale: Later narrow helper commands such as `editable_task_file --summary` are normal and should not erase proof that the repo-owned same-session helper already completed.
  Date/Author: 2026-03-31 / Codex

- Decision: Keep the post-run authoritative override even after fixing the live watchdog race.
  Rationale: The live path should usually avoid the false kill, but the deterministic post-run assessment is cheap and guarantees that a shard with `diagnosis_code=completed` plus durable outputs never persists as `repair_recovered` or `validated_after_watchdog_kill`.
  Date/Author: 2026-03-31 / Codex

- Decision: Keep failure-closed behavior when helper-complete evidence never arrives and output/state proof stays absent.
  Rationale: This plan is for false positives only. Truly missing output must still fail, and the existing retry/recovery paths stay valid for that case.
  Date/Author: 2026-03-31 / Codex

## Outcomes & Retrospective

The line-role runtime now resolves the false-kill sequence cleanly in both places that mattered. Live supervision waits through the configured completion/grace window and only emits `workspace_final_message_missing_output` after two consecutive post-deadline passes with no helper proof and no durable outputs. Post-run finalization now also reclassifies the old bad artifact pattern (`workspace_final_message_missing_output` followed immediately by deterministic `diagnosis_code=completed` plus outputs present) into ordinary `completed` instead of `validated_after_watchdog_kill` or telemetry recovery counts.

The targeted verification stayed local to `.venv` and did not require a CodexFarm benchmark rerun. The highest-value regression additions were the synthetic runtime case that includes a later `editable_task_file --summary` completion after the helper, and the timing fixes that made the tests follow the real 15-second workspace quiescence/grace settings rather than an older 2-second assumption.

## Context and Orientation

Canonical line-role is the stage that labels each atomic text line as recipe content or non-recipe content. In the current live path, one long-lived Codex worker edits a repo-written `task.json` in place and then runs the repo-owned helper command `python3 -m cookimport.parsing.canonical_line_roles.same_session_handoff`. That helper validates the edited answers, writes shard outputs under `out/<shard_id>.json`, updates `_repo_control/line_role_same_session_state.json`, and reports a status such as `completed` or `repair_required`.

There are two filesystem roots involved. The repo-owned worker root lives under `data/output/.../workers/worker-###/`. The actual Codex session runs inside a fenced execution workspace created by `cookimport/llm/codex_exec_runner.py`. During the session, the runner periodically syncs selected paths such as `task.json`, `_repo_control/`, and `out/` back from the execution workspace into the repo-owned worker root so the stage watchdog can inspect them.

The relevant code is split across these files:

- `cookimport/llm/codex_exec_runner.py` builds `CodexExecLiveSnapshot` from the Codex JSON event stream and wraps the stage supervision callback.
- `cookimport/parsing/canonical_line_roles/runtime.py` creates the line-role watchdog callback, reads same-session state plus output files, decides whether to terminate the worker, and finalizes shard status/telemetry after the run.
- `cookimport/parsing/canonical_line_roles/same_session_handoff.py` is the repo-owned helper that writes line-role outputs and updates `_repo_control/line_role_same_session_state.json`.
- `tests/parsing/test_canonical_line_roles.py` already contains most runtime and watchdog regression coverage for this stage.
- `docs/10-llm/10-llm_README.md` and `cookimport/parsing/canonical_line_roles/README.md` are the short docs that describe the line-role single-file worker contract.

The current bug is this: the worker runs `same_session_handoff`, that command succeeds, and the event stream records success; however, the line-role watchdog still eventually decides `workspace_final_message_missing_output` because it never sees the file-visible success proof in time. After the session is killed, the runtime re-reads the same worker root, now sees the completed state and output file, and records the shard as recovered. That is the behavior to remove.

## Plan of Work

Start in `cookimport/llm/codex_exec_runner.py`. Extend the live event summarization so `CodexExecLiveSnapshot` contains structured information about the most recent completed command, including at minimum the command text, exit code, and whether the command is a repo-owned same-session helper that reported `completed`. Keep the parsing generic inside the runner and stage-neutral in naming. The line-role stage should not need to scan raw event payloads.

The simplest shape is one new snapshot field that carries a small dictionary or dataclass-like payload for the last completed command, plus a helper boolean for "repo handoff reported completed." The parser should recognize the line-role helper command by command text and inspect the command's `aggregated_output` for the JSON object that the helper prints. It should treat `{"status":"completed"}` or `{"final_status":"completed"}` as positive completion evidence. Do not infer success from the final agent message; only trust the repo-owned helper command output.

Then edit `cookimport/parsing/canonical_line_roles/runtime.py`. In `_build_strict_json_watchdog_callback`, add a second success lane besides file-visible proof. When the callback sees positive helper-completion evidence from the live snapshot, it must stop treating the final agent message as a missing-output failure trigger. Instead, it should enter a bounded "completion visibility wait" state: keep syncing and checking for `_repo_control/line_role_same_session_state.json` plus the required `out/*.json` files to catch up, and surface that waiting state in `live_status.json`. If file/state proof appears, resolve as clean completion. If the process exits cleanly and the post-run sync makes outputs visible, still resolve as clean completion. Only fail with `workspace_final_message_missing_output` if there is neither file-visible proof nor helper-complete evidence by the end of the grace window.

Make the post-run finalization path agree with that live decision. In `cookimport/parsing/canonical_line_roles/runtime.py`, after `run_workspace_worker(...)` returns and the runner has performed its final sync, do one last authoritative reconciliation before assigning shard status. If the run was previously tagged `workspace_final_message_missing_output` but the worker session showed helper-complete evidence and the final sync now proves same-session completion plus output presence, normalize the outcome to the clean session path instead of `repair_recovered`. The goal is that these sessions stop incrementing `watchdog_recovered_shard_count` and stop showing `validated_after_watchdog_kill` / `repair_recovered` for what was really a clean success.

Add regression coverage in two layers. In `tests/llm/test_codex_exec_runner.py`, build a small event fixture containing a completed `same_session_handoff` command item and assert the live snapshot exposes the new completion metadata. In `tests/parsing/test_canonical_line_roles.py`, add a callback-level test where the first snapshot says the helper already completed but the output file is still absent, then a later snapshot makes the output visible; the watchdog must never emit `workspace_final_message_missing_output`. Add an end-to-end synthetic runtime test that reproduces the archived bad ordering and assert the final shard status uses `session_completed`, the worker `live_status.json` is clean success, and the stage telemetry reports zero watchdog kills or recoveries for that shard. Keep the existing negative tests for true missing output unchanged.

Finally, update the short docs. In `docs/10-llm/10-llm_README.md` and `cookimport/parsing/canonical_line_roles/README.md`, add one short paragraph saying line-role now trusts repo-owned helper completion evidence from the live worker stream and only falls back to `workspace_final_message_missing_output` when neither helper proof nor output proof materializes.

## Validation and Acceptance

Acceptance is behavioral, not structural.

The change is complete when all of the following are true:

1. A runner-level test proves that a completed `python3 -m cookimport.parsing.canonical_line_roles.same_session_handoff` command is surfaced in `CodexExecLiveSnapshot` as positive helper-completion evidence.

2. A line-role callback test proves this sequence no longer kills the worker:
   the helper has already reported `completed`, the final agent message arrives, file-visible output is still absent on that snapshot, and the output file appears on a later snapshot. The callback must return `None` while waiting and later resolve to `completed`, never `workspace_final_message_missing_output`.

3. A synthetic line-role runtime test that models the archived bad sequence ends with:
   `worker live_status.json.state == "completed"` or `"completed_with_warnings"`,
   shard `finalization_path == "session_completed"`,
   no `repair_recovered`,
   no `validated_after_watchdog_kill`,
   and stage telemetry counts `watchdog_killed_shard_count == 0` and `watchdog_recovered_shard_count == 0` for that run.

4. Existing true-failure coverage still passes. In particular, a worker that sends a final message without helper-complete evidence and without output/state proof must still fail closed with `workspace_final_message_missing_output`.

## Idempotence and Recovery

These edits are safe to apply incrementally. The runner snapshot changes are additive, so existing code can ignore the new fields until the line-role runtime is updated. The new callback behavior should be implemented so that absence of the new helper metadata preserves the old behavior. If a partial implementation breaks tests midway, keep the worktree and continue until the runner and runtime agree on the new signal; do not paper over failures by broadening deterministic recovery.

The validation commands above are idempotent and safe to rerun. They operate only on temp directories created by pytest. If a test fixture needs renaming during iteration, prefer adjusting the fixture or assertion rather than deleting unrelated artifacts.
