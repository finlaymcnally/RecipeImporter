# Stop long variant prose from becoming `HOWTO_SECTION`

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan builds directly on `docs/plans/2026-03-21_15.25.37-make-howto-section-book-optional-and-generalize-line-role-validation.md` and `docs/plans/2026-03-21_15.38.01-fix-line-role-packet-mode-misclassification.md`, but it repeats the implementation context needed to execute the fix from scratch.

## Purpose / Big Picture

The front-matter packet bug is fixed, but the next benchmark run at `data/golden/benchmark-vs-golden/2026-03-21_16.10.40` exposed a narrower, later-stage over-structuring problem. Long prose paragraphs that begin with `To make ...` or `To serve ...` are being treated as subsection headings and ending up as `HOWTO_SECTION`, even though the gold labels say they are recipe variants, ordinary instructions, `OTHER`, or `KNOWLEDGE`. After this change, those long variant paragraphs should stop being tagged or preserved as subsection headings. A real short heading such as `FOR THE SAUCE` or `TO SERVE` should still survive when it actually splits one recipe into components or step families.

The user-visible outcome is concrete. If an operator reruns the Salt Fat line-role benchmark after this change, the residual `HOWTO_SECTION` error family should collapse from “long `To make ...` prose paragraphs” to only genuine hard edge cases, and the line-role packet outputs should no longer show full multi-sentence variant paragraphs labeled as `HOWTO_SECTION`. Deterministically, the pytest suite should prove that long `To make ...` variant prose resolves to `RECIPE_VARIANT` or another conservative non-heading label while genuine short subsection headings still resolve to `HOWTO_SECTION`.

## Context and Orientation

The line-role stage assigns one structural label to each atomic text line in a cookbook. An atomic line is a small unit of text such as a heading, ingredient line, step sentence, note, or paragraph. The relevant code for this task is concentrated in two files:

- `cookimport/parsing/recipe_block_atomizer.py` creates the early `rule_tags` for each line. One of those tags is `howto_heading`.
- `cookimport/parsing/canonical_line_roles.py` converts atomic lines plus `rule_tags` into deterministic labels, packet guidance, Codex review inputs, and post-Codex sanitization.

The current bad path looks like this. In `recipe_block_atomizer.py`, `_is_howto_heading(text)` returns `True` for any text matching `_HOWTO_PREFIX_RE`, which includes lines beginning with `To make ...` or `To serve ...`. That means long instructional paragraphs and compact subsection headers receive the same `howto_heading` tag. In `canonical_line_roles.py`, `_deterministic_label(...)` treats `howto_heading` as enough to emit `HOWTO_SECTION` when `_howto_section_label_allowed(...)` passes. Later, if Codex keeps or re-emits `HOWTO_SECTION`, `_sanitize_prediction(...)` only demotes it when `_howto_section_label_allowed(...)` returns `False`. Because the current helper is designed around local recipe support, not heading shape, long prose can survive when nearby rows look recipe-local.

The concrete benchmark evidence is in `data/golden/benchmark-vs-golden/2026-03-21_16.10.40/single-book-benchmark/saltfatacidheatcutdown/`. The actionable source file is `codexfarm/line-role-pipeline/line_role_predictions.jsonl`, which contains 13 unique `HOWTO_SECTION` predictions. Those 13 lines are:

This plan must preserve those.

## Plan of Work

Start in `cookimport/parsing/recipe_block_atomizer.py`. Replace the current one-bit `_is_howto_heading(text)` behavior with a more discriminating heading-shape rule. The implementation should continue recognizing compact headings like `FOR THE SAUCE`, `TO SERVE`, and `FOR GARNISH`, but it should stop tagging long sentence-like or paragraph-like `To make ...` / `To serve ...` prose as `howto_heading`. The simplest durable shape is likely: how-to prefix plus heading-length limits plus no sentence punctuation plus no multi-clause prose shape. The plan should keep the logic local and explicit rather than hiding it in a giant regex.

Then move to `cookimport/parsing/canonical_line_roles.py`. Add one helper that answers a question the current code does not ask clearly enough: “does this line look like a subsection heading, or does it look like variant/instruction prose that happens to begin with a how-to phrase?” Use that helper in both deterministic labeling and sanitization. The end state should make long `To make ...` paragraphs fail closed away from `HOWTO_SECTION` even when local component evidence exists. When those paragraphs live inside or next to a recipe, the likely correct fallback is `RECIPE_VARIANT`; when they are outside accepted recipe span but still explanatory, the fallback may be `OTHER`, `KNOWLEDGE`, or `INSTRUCTION_LINE` depending on the exact line shape. Resolve that policy explicitly in code rather than leaving it to indirect heuristics.

While making that change, handle the second error family too: all-caps conceptual headings such as `HOW SALT WORKS` should not be allowed through the subsection path merely because they are short and heading-like. They should be pushed through the knowledge-heading path instead. The implementation should not hardcode this exact phrase, but it should ensure that concept/lesson headings with no local component structure do not survive as `HOWTO_SECTION`.

After the parser-owned logic is corrected, update the worker-facing contract in `cookimport/llm/canonical_line_role_prompt.py` and, if needed, the prompt assets under `llm_pipelines/prompts/`. Add one short negative rule that a full procedural sentence or paragraph beginning with `To make ...` or `To serve ...` is usually a variant or instruction, not a subsection heading. This is not the primary fix seam, but it keeps the file-backed prompt aligned with the parser contract and reduces the chance that Codex recreates the old mistake.

Then expand `tests/parsing/test_canonical_line_roles.py`. Add direct regressions for the benchmark family instead of abstract placeholders. At minimum, include one long outside-span `To make ...` variant paragraph, one long in-recipe `To make ...` variant paragraph, one long `To serve ...` sentence, and one all-caps concept heading like `HOW SALT WORKS`. For each, assert that the line does not survive as `HOWTO_SECTION`. Pair those with positive controls: short `FOR THE ...` or `TO SERVE` headings that still become `HOWTO_SECTION` in a real subsection context, ideally reusing the existing Sea-and-Smoke-style examples.

Finally, update the short parsing docs in `docs/04-parsing/04-parsing_readme.md` if the active contract changes. The new doc text should state plainly that `howto_heading` is now heading-shape-sensitive and that long `To make ...` prose is treated as variant/procedural content rather than subsection structure.

## Validation and Acceptance

The change is acceptable only if all of the following are true.

Long `To make ...` and `To serve ...` prose no longer survives as `HOWTO_SECTION`. In deterministic tests, both outside-span and inside-recipe examples modeled on the 2026-03-21_16.10.40 benchmark must resolve to non-`HOWTO_SECTION` labels.

Variant paragraphs recover to a sensible label. At least one benchmark-modeled `To make ...` example that describes an alternate finished dish must resolve to `RECIPE_VARIANT`, not merely to a vague fallback such as `OTHER`.

Real short subsection headings remain intact. A compact heading such as `FOR THE JUNIPER VINEGAR` or `TO SERVE` in a real component-level context must still resolve to `HOWTO_SECTION`.

Concept headings are not mistaken for subsections. A heading shaped like `HOW SALT WORKS` must resolve to `KNOWLEDGE` or another non-`HOWTO_SECTION` label when local component structure is absent.

Prompt and parser contracts stay aligned. The file-backed line-role prompt must include the updated negative rule about long `To make ...` / `To serve ...` prose so the worker is not told to preserve the old bad prior.

Deterministic proof must be recorded. Run the focused parsing tests, the full `tests/parsing/test_canonical_line_roles.py` file, and `./scripts/test-suite.sh domain parsing` in `.venv`, then paste the commands and short pass/fail evidence into this plan’s `Artifacts and Notes` section.

Benchmark proof is explicitly optional for default acceptance. A live Codex-backed rerun remains the best final proof, but it still requires explicit user approval because it spends model tokens.

## Idempotence and Recovery

These edits are additive and safe to repeat. The tests can be rerun without mutating historical benchmark roots. Existing benchmark artifacts under `data/golden/benchmark-vs-golden/2026-03-21_16.10.40` and `data/output/2026-03-21_16.10.40` should remain untouched and serve only as preserved evidence.

If the first implementation pass over-demotes genuine short subsection headings, do not revert the whole `HOWTO_SECTION` policy. Instead, keep the heading-shape split and relax the positive control path in `_howto_section_label_allowed(...)` until the real subsection cases pass again. The failure mode to avoid is restoring paragraph-shaped `HOWTO_SECTION` seeds just to recover short headings.

If the change fixes the variant paragraphs but leaves `HOW SALT WORKS` or similar concept headings mislabeled, add a separate knowledge-heading boundary rather than widening the prose ban indiscriminately. The benchmark evidence shows two subfamilies, and the implementation should be willing to address them separately.
