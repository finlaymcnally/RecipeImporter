# Burn The Boats: Final Repo-Wide Old-World Purge

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the repository should stop presenting itself as a codebase that still needs to carry old architectures, old artifact shapes, old benchmark history, or deleted tagging surfaces. A person opening the repo, grepping the tree, or running the current CLI should see only the current paradigm: the active runtime contract, current tests, and current docs. The proof is straightforward. A repo-wide seam search should stop finding live transition branches and stale older folders, the checked-in test tree should no longer contain compiled cache junk or dead tagging directories, and the docs tree should no longer keep old migration-era plans and reports around as first-class project context.

This plan is intentionally destructive. It does not preserve read-side transition support for older outputs, older docs, or older helper interfaces unless current runtime behavior still demonstrably depends on them. If a branch exists only to read, describe, or soften an old world, remove it and update the current code or docs to speak only about the new one.

## Context and Orientation

This repository is a Python project rooted at `/home/mcnal/projects/recipeimport`. The live product code is under `cookimport/`. The tests are under `tests/`. The docs live under `docs/`. In this plan, an “old-world seam” means any code path, test, or document that exists only to preserve or describe an older architecture, older output format, older helper signature, removed setting value, or deleted feature family. “Current docs” means docs that still function as operator or maintainer guidance for the current product, such as `docs/01-architecture/01-architecture_README.md`, `docs/02-cli/02-cli_README.md`, `docs/05-staging/05-staging_readme.md`, `docs/06-label-studio/06-label-studio_README.md`, `docs/07-bench/07-bench_README.md`, `docs/08-analytics/08-analytics_readme.md`, `docs/10-llm/10-llm_README.md`, plus module-local `README.md` and `CONVENTIONS.md` files under `cookimport/`. “Archival docs” means documents that exist mainly as history rather than current product guidance, including `docs/plans/old/`, `docs/plans/OGplan/`, `docs/reports/Refactor.md`, old purge task files, and migration-era understandings that only catalog removed seams.

The 2026-03-16 audit established the current baseline. The remaining live code seams were small and concentrated. The main ones were in `cookimport/cli.py`, where helper loaders still tolerated older signatures and report readers still accepted older metric shapes; in `cookimport/analytics/dashboard_schema.py`, where benchmark rows still exposed historical alias metrics; and in a handful of comments or docstrings across parsing, scoring, and staging that still framed current behavior as transition-friendly instead of current-only. The tests contained some honest coverage that removed values fail fast, but they also contained stale tagging-domain references in `tests/CONVENTIONS.md`, `tests/README.md`, `tests/AGENTS.md`, `tests/conftest.py`, and the dead fixture folders `tests/tagging/` and `tests/tagging_gold/`. The docs were the largest remaining cluster: current docs still mentioned fallback readers and older paths, while archival docs kept earlier migration phases visible in the active tree.

The important constraint is that this purge must leave the repo more coherent, not merely smaller. If an adapter branch is removed from code, the matching tests and docs must be updated in the same pass. If an archival doc is deleted, any current doc that points at it must be rewritten or the link removed. If a deleted surface still has a real current consumer, the right fix is to move that consumer to the current contract, not to resurrect the old-world seam.

## Plan of Work

Start with the active runtime code because that is the only part that can still change behavior at execution time. In `cookimport/cli.py`, remove the older eval-report fallback behavior in the benchmark report helpers so current CLI output relies only on the current metric keys. Remove the `TypeError` fallback calls around `build_starter_pack_for_existing_runs(...)` and `build_upload_bundle_for_existing_output(...)`; those fallbacks exist only for older helper signatures and should be replaced by a single current call shape. Remove the older freeform label-name inference in the Label Studio project-scope detection so only the current label taxonomy is recognized. Delete the transition wrapper around `_run_all_method_prediction_once(...)` and make callers use the canonical function directly. Remove the downstream alias fields currently written alongside scheduler metrics so the output contract publishes only the canonical names. In `cookimport/analytics/dashboard_schema.py`, delete the historical alias metric fields (`precision`, `recall`, `f1`, `practical_*`) from the schema and update all code that still reads or writes them to use `strict_accuracy` and `macro_f1_excluding_other` only. In nearby runtime files such as `cookimport/core/scoring.py`, `cookimport/parsing/instruction_parser.py`, `cookimport/parsing/text_section_extract.py`, `cookimport/staging/writer.py`, and any other remaining active source hits, rewrite comments and docstrings so they describe current behavior plainly instead of framing it as a older wrapper or transition default.

Once the runtime seams are gone, clean the test tree so it matches the new world. Delete every checked-in `__pycache__` directory under `tests/`. Delete `tests/tagging/` entirely if it still contains only cache artifacts. Delete `tests/tagging_gold/` unless a remaining live test still consumes one of those fixtures through the new inline tagging path; if any fixture is still genuinely needed, move it under the domain that actually owns it and rename it so it no longer carries the deleted standalone-tagging identity. Update `tests/CONVENTIONS.md`, `tests/README.md`, `tests/AGENTS.md`, `tests/conftest.py`, and `tests/paths.py` so the tagging domain is no longer presented as a first-class standalone test surface. Rewrite or delete any tests whose only purpose is to assert older wording, older filenames, or transition aliases. Keep tests that prove current behavior or that removed values fail fast, but rewrite them to avoid misleading naming where possible.

The third pass is the current docs pass. Update the current docs under `docs/01-architecture`, `docs/02-cli`, `docs/05-staging`, `docs/06-label-studio`, `docs/07-bench`, `docs/08-analytics`, `docs/10-llm`, and the module-local docs under `cookimport/` so they describe only the current product contract. Any sentence that says “older behavior remains,” “transition fallback,” “historical rows are still supported,” or similar needs one of two outcomes: either rewrite it into a current-contract statement, or remove it if the behavior is being deleted. Be especially aggressive in analytics and dashboard docs, because those currently teach fallback CSV roots, old timestamp patterns, and historical metric aliases as active knowledge. Update `cookimport/labelstudio/README.md`, `cookimport/labelstudio/CONVENTIONS.md`, `cookimport/bench/README.md`, `cookimport/bench/CONVENTIONS.md`, `cookimport/analytics/CONVENTIONS.md`, `cookimport/staging/CONVENTIONS.md`, and `cookimport/CONVENTIONS.md` so they stop referring to removed project scopes, removed metrics, removed helper tolerances, or deleted runtime families as supported context.

The fourth pass is the archival-doc deletion pass. Delete `docs/plans/old/` and `docs/plans/OGplan/` entirely. Delete `docs/reports/Refactor.md` if it is only a migration design record and no longer needed to operate the current product. Delete task files and understanding notes that only discuss already-completed older purges, stale migration seam clusters, or removed standalone tagging architecture. Keep only docs that still help operate, debug, or review the current product. If a current doc still points to one of these deleted artifacts, remove the reference in the same branch. This is also the moment to delete any stray files whose names alone keep older concepts visible in repo listings, as long as they are not still part of the active product.

Finish with a repo-wide language and shape sweep. Re-run the seam inventory search across `cookimport/`, `tests/`, and `docs/`. The final tree should not contain active-code or current-doc references to `older`, `transition`, `retired`, `removed`, or pass-slot names unless they refer to genuinely external facts that still matter, such as rejecting binary `.doc` files or explaining an unsupported older import format. If a remaining hit is merely a comment, fixture name, directory name, or archival breadcrumb, remove or rename it.

## Idempotence and Recovery

This purge is safe to execute in passes because each seam family is separable. Runtime-code cleanup, test-tree cleanup, current-doc rewrites, and archival-doc deletion should each land with their own local verification so failures are easy to isolate. Deleting `__pycache__` directories is naturally idempotent. Deleting stale test directories and archival docs is also idempotent once all references are removed.

The risky area is deleting docs or fixtures that still have a hidden consumer. The recovery rule is simple: do not restore a whole transition layer. Instead, identify the live consumer, update it to the current contract, or move the still-needed fixture/document into the current domain with a current name. If one of the broad test-suite domains fails due to an unrelated pre-existing issue, rerun the narrow test nodeids for the touched files and record the unrelated failure in the `Surprises & Discoveries` section before continuing.
