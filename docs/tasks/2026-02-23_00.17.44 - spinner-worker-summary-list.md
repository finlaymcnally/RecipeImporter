# Problem Statement

Spinner output showed only one aggregate status line, so users could not see what each active worker was processing during parallel worker phases.

# Acceptance Criteria

- Callback spinner path can accept worker-activity telemetry without replacing the main status line.
- Spinner renders one worker summary line per active worker beneath the main status line.
- Freeform prelabel parallel workers emit worker activity (`task X/Y + segment range`) and reset summary state when the worker phase ends.
- Split conversion worker loops in Label Studio import emit worker activity (`job X/Y`) and reset summary state when split work completes.
- Existing ETA/task-counter behavior remains intact.

# Evidence

- Added worker telemetry helpers in `cookimport/core/progress_messages.py` (`format_worker_activity`, `format_worker_activity_reset`, `parse_worker_activity`).
- `cookimport/cli.py:_run_with_progress_status(...)` now renders multiline worker summaries under the main spinner line.
- `cookimport/labelstudio/ingest.py` prelabel parallel and split conversion worker paths now emit worker activity and reset events.
- Added/updated tests:
  - `tests/labelstudio/test_labelstudio_benchmark_helpers.py` covers worker-summary rendering.
  - `tests/labelstudio/test_labelstudio_ingest_parallel.py` checks worker telemetry events are emitted for both prelabel and split worker phases.
- Verification command passed: `60 passed, 2 warnings in 3.46s`.

# Constraints / Gotchas

- Worker telemetry must not overwrite the main status message; ETA parsing still depends on the right-most `X/Y` counter in main status text.

# Decision

Use a serialized worker-activity side-channel in progress callbacks and keep all rendering logic centralized in the shared spinner helper.

# Rollback / Compensation

Remove worker-activity parsing/rendering from `_run_with_progress_status(...)` and stop emitting worker telemetry from prelabel/split worker loops.
