Those plans got the repository to the current live state: knowledge classification and grouping are semantically separated, but the runtime now pays for that separation by launching up to two happy-path workspace-worker sessions per worker assignment. This plan keeps the semantic split and fixes the worker-shape regression. The final live behavior after this plan is: one worker assignment, one happy-path Codex session, classification first, then repo validation/advance inside that same session, then grouping immediately in the same context if needed. This is burn-the-boats work. There is no goal of preserving the current two-session happy path as a fallback.

## Structured Restatement Of User Desire

The knowledge stage should keep the classification-first, grouping-second semantic contract, but it should stop paying for that contract by tearing down worker context between the two steps. One knowledge worker assignment should mean one happy-path Codex session. Inside that session, the worker edits the classification `task.json`, runs one repo-owned validate-and-advance command, and if that validation succeeds the same session immediately receives the grouping `task.json` and continues with the second step. If validation instead finds a small structural mistake, the same command should rewrite `task.json` into a repair form for that same step and let the same worker patch it without losing context. The user-visible result should be simple and honest: if the UI says five knowledge workers, the happy path should launch five knowledge worker sessions, not up to ten.

## Purpose / Big Picture

After this change, a user can choose `N` knowledge workers and expect the happy path to use at most `N` knowledge workspace-worker sessions. Each worker assignment will still preserve the current semantic discipline: classification first, grouping second, and grouping sees only accepted knowledge rows. What changes is the handoff shape. The same worker session will validate its classification output, receive the grouping input immediately if classification passed, and finish the grouping work without losing context or paying for a second session startup.

The proof is concrete and user-visible. In fake-exec and prompt-budget artifacts, a knowledge run with five worker assignments should report five happy-path workspace sessions rather than up to ten. The worker prompt and docs should describe one blessed local helper command for knowledge: edit `task.json`, run the validate-and-advance helper, then either continue on the rewritten `task.json` or stop. `task_status.jsonl`, `telemetry.json`, `knowledge_stage_summary.json`, `prompt_budget_summary.json`, and `stage_observability.json` should all reflect that the live happy path is one session per assignment with same-session step transitions instead of two separate session launches.

## Context and Orientation

In this repository, the knowledge stage is the optional Codex-backed part of `nonrecipe-finalize`. It decides whether outside-recipe text is final `knowledge` or final `other`, then groups accepted knowledge rows into related idea groups. The current live stage is implemented mainly in `cookimport/llm/knowledge_stage/workspace_run.py`, `cookimport/llm/knowledge_stage/task_file_contracts.py`, `cookimport/llm/knowledge_stage/recovery.py`, and `cookimport/llm/knowledge_stage/reporting.py`. The shared workspace runner lives in `cookimport/llm/codex_exec_runner.py`. The active operator-facing explanation of the stage lives in `docs/10-llm/10-llm_README.md`.

Three terms matter here and must stay concrete.

A “worker assignment” is the bundle of owned knowledge shards given to one worker id. In the current code, that is represented by `WorkerAssignmentV1`.

A “workspace-worker session” is one `codex exec` process launched through `run_workspace_worker(...)`. This is what the user experiences as “one agent run.”

A “same-session handoff” is the design this plan introduces. It means the worker stays inside one live workspace-worker session while repo-owned validation transforms `task.json` from one semantic step to the next. The worker does not lose context and the repo does not launch a second happy-path `codex exec` process just to continue the same assignment.

The live flow after this plan works like this. `build_knowledge_classification_task_file(...)` creates the initial classification `task.json`. `cookimport/llm/knowledge_stage/workspace_run.py` writes hidden helper state beside the worker root, launches one workspace session, and instructs the worker to edit `task.json` and run `python3 -m cookimport.llm.knowledge_same_session_handoff`. The helper validates the current file against the hidden original snapshot, rewrites `task.json` into repair or grouping form when needed, and writes final `out/*.json` shard payloads when the assignment is complete. Repo code reads the hidden helper state after the session exits to produce telemetry and final outputs.

The runner shape still matters. `cookimport/llm/codex_exec_runner.py` preserves the single-file visible root, but now passes a hidden helper-state path through `RECIPEIMPORT_KNOWLEDGE_SAME_SESSION_STATE_PATH`. The worker never needs to browse helper metadata manually, and the same visible `task.json` file remains the only edit surface throughout classification, repair, and grouping.

This implementation preserved the semantic boundary from the knowledge-boundary isolation plan. Classification remains grouping-free. Grouping still sees only accepted knowledge rows. The only thing that changed is the handoff seam between those steps.

## Plan of Work

The first block of work landed as `cookimport/llm/knowledge_same_session_handoff.py`. That helper is the repo-owned command the worker runs inside its session. It reads the current edited `task.json`, validates it against repo-owned hidden state, and returns one of four statuses: `repair_required`, `advance_to_grouping`, `completed_without_grouping`, or `completed_with_grouping`. It owns rewriting `task.json` for repair/grouping and owns writing final `out/*.json` shard payloads when the assignment is complete.

The second block of work extended the task-file contract in `cookimport/llm/knowledge_stage/task_file_contracts.py`. The classification and grouping schemas stayed separate, but new transition helpers now convert validated classification answers into repair files, grouping files, or final outputs, and do the same for grouping completion. This kept the semantic split intact while making the same-session handoff explicit in repo code.

The third block of work rewired `cookimport/llm/knowledge_stage/workspace_run.py` to treat one worker assignment as one happy-path workspace session. The knowledge worker prompt now tells the worker to edit `task.json`, run the helper, reopen the rewritten file when told, and stop only when the helper reports completion. The runtime initializes hidden helper state before launch and reads that state after exit to classify incomplete handoffs and to report accurate telemetry.

The fourth block of work kept the visible root minimal. `cookimport/llm/codex_exec_runner.py` now passes hidden same-session state through `RECIPEIMPORT_KNOWLEDGE_SAME_SESSION_STATE_PATH`, and the visible worker root is still just `task.json` plus the standard runtime files. The fake direct-exec runner and `scripts/fake-codex-farm.py` were updated to simulate the same helper loop so repository tests exercise the same contract the real prompt describes.

The fifth block of work updated reporting. Prompt-budget summaries, stage observability, worker status payloads, and knowledge-stage reporting now count real workspace sessions and separate same-session transitions. The new counters include `classification_validation_count`, `grouping_validation_count`, `same_session_transition_count`, `same_session_repair_rewrite_count`, and `grouping_transition_count`.

The sixth block of work was tests and docs. `tests/llm/test_knowledge_same_session_handoff.py` now proves direct helper transitions and same-session repair. The runtime, subprocess, observability, prompt-preview, and writer tests were updated to assert the new one-session model and the newer grounding contract. `docs/10-llm/10-llm_README.md`, `docs/plain-english-explanation.md`, and `cookimport/llm/knowledge_stage/README.md` now describe the same-session handoff rather than a two-session happy path.

## Validation and Acceptance

This work is accepted only when the repository proves the user’s requested worker shape in behavior, not just in code structure.

The first acceptance criterion is happy-path session count. A knowledge run with `N` worker assignments must launch at most `N` happy-path workspace-worker sessions. Internal same-session validation, repair, and grouping transitions do not count as extra happy-path sessions.

This criterion is now satisfied in the live runtime and tests. The focused concurrency test in `tests/llm/test_knowledge_orchestrator_runtime_progress.py` now proves that a two-shard, two-worker run reports `workspace_worker_session_count == 2`, not four, while still observing concurrent in-flight workers.

The second acceptance criterion is preserved semantic isolation. The classification task file must remain grouping-free. Grouping must still see only accepted knowledge rows. This plan must not regress the semantic protection introduced by the knowledge-boundary isolation plan.

The third acceptance criterion is same-session handoff. A focused test must prove that one worker can edit the classification file, run the helper, receive a grouping file in place, edit it, run the helper again, and finish without a second `run_workspace_worker(...)` call.

The fourth acceptance criterion is same-session repair. A focused test must prove that a malformed classification or grouping edit can be rewritten into repair mode inside the same session and completed without losing worker context. External repair sessions may still exist only as bounded fallback after worker death or equivalent interruption.

The fifth acceptance criterion is honest reporting. `prompt_budget_summary.json`, `telemetry.json`, `knowledge_stage_summary.json`, and stage observability artifacts must stop implying two happy-path sessions per assignment. They should instead report one session per assignment plus same-session transition counts.

The sixth acceptance criterion is documentation clarity. Active docs must stop telling users that five knowledge workers can mean up to ten happy-path worker launches. A novice reading the docs after this plan should come away with the correct mental model immediately.

## Idempotence and Recovery

The implementation steps are safe to repeat if they are done in the order above. The new helper command should be deterministic and idempotent for a given workspace state: running it twice without any intervening task-file edits should either return the same status again or fail with an explicit “no new worker edits detected” style message rather than silently mutating the task file further.

If implementation is interrupted partway through, the safest rollback point is the current live two-session runtime. Do not ship a hybrid state where reporting still claims one session per assignment but the runtime still launches two happy-path sessions. Keep tests authoritative: if the one-session handoff tests are red, the docs and telemetry changes are not done yet.

If a same-session worker dies after classification but before grouping completes, repo code may still use a bounded external recovery path. That recovery path must be reported as follow-up or recovery work, not folded into the normal happy-path session count.
