# Problem

After the long-lived worker migration, main worker sessions were still being killed for normal workspace-local shell activity such as opening named shard files with `cat` or `sed`. That zero-tolerance rule still makes sense for strict one-shot structured retry/repair calls, but it is too harsh for the workspace-worker model where Codex is explicitly acting like a local file worker processing many tasks over time.

# Acceptance Criteria

- Workspace-worker main attempts for recipe, knowledge, and line-role can keep using narrow workspace-local shell/helper commands on named files for the whole session.
- Workspace-worker main attempts only terminate for command loops without outputs or for commands that leave the allowed workspace-local surface.
- Structured retry / repair follow-up calls still kill immediately on any shell command.
- Live status artifacts record when command execution was tolerated instead of killed.
- Focused tests cover the shared command classifier plus stage-level workspace tolerance.

# Verification

- `./.venv/bin/pytest -q tests/llm/test_codex_exec_runner.py tests/parsing/test_canonical_line_roles.py tests/llm/test_codex_farm_knowledge_orchestrator.py tests/llm/test_recipe_phase_workers.py`
- `./scripts/test-suite.sh domain llm`

# Evidence

- The focused watchdog/worker suites passed after the change.
- `./scripts/test-suite.sh domain llm` passed with `131 passed, 30 deselected, 1 warning`.

# Constraints / Decision

- Keep the tolerance narrow in scope even though it is no longer capped to one or two commands: allow only workspace-local helper commands on named local files, not broad exploration or repo traversal.
- Do not relax the structured retry/repair watchdog policy.
- Prolonged command loops, broader exploration commands, and non-local shell use still terminate the worker.
