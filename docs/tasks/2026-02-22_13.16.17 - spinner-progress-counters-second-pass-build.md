# Task: Build spinner-progress-counters second pass

## Problem statement

Known-size loops outside Label Studio prelabel/decorate still emitted phase-only status text, so bench and split-merge workflows lacked concrete throughput visibility.

## Acceptance criteria

- Shared helper exists for safe counter formatting (`task`/`item`/`config`/`phase`).
- `bench run` emits `item X/Y` per-item progress messages.
- `bench sweep` emits `config X/Y` and nested forwarding as `config X/Y | item X/Y ...`.
- Split-merge status callback emits `merge phase X/Y: ...` messages with deterministic phase ordering.
- Tests cover helper formatting, bench counter emissions, and merge phase counter sequencing.
- CLI/bench/staging docs and conventions reflect the final counter contract.

## Verification steps

- `. .venv/bin/activate && python -m pip install -e .[dev]`
- `. .venv/bin/activate && pytest -q tests/test_progress_messages.py tests/test_bench_progress.py tests/test_split_merge_status.py tests/test_labelstudio_ingest_parallel.py tests/test_labelstudio_prelabel.py -k 'prelabel_task_progress or decorate_dry_run or test_'`
- `. .venv/bin/activate && npm run docs:list`

## Evidence

- Automated test run: `23 passed`.
- Emitted-message examples now asserted in tests:
  - `item 1/2 [alpha] Processing...`
  - `config 1/2 | item 1/1 [alpha] Evaluating...`
  - `merge phase 1/12: Merging job payloads...`
  - `merge phase 12/12: Merge done`

## Constraints and decisions

- Counter strings are generated in runtime loops where totals are known (runner/sweep/merge), then forwarded by CLI wrappers.
- Shared formatter is additive and dependency-free, so existing callback plumbing stays intact.
- Split-merge phase total accounts for optional chunk-write phase to keep `X/Y` monotonic.

## Rollback plan

- Revert helper adoption in `cookimport/bench/runner.py`, `cookimport/bench/sweep.py`, and `cookimport/cli.py`.
- Revert new helper/tests/docs files if counter contract needs to return to previous behavior.
