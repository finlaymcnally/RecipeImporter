This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Operator Intent (Structured) - DO NOT DELETE

The operator wants the analytics dashboard to stop behaving like a mini reporting product with many generated HTML files and an oversized implementation. The target is one main dashboard that still answers the useful questions, with all-method benchmark detail folded back into that main surface instead of living in a sidecar page family. The plan should prefer deletion and simplification of extra page surfaces and fake architectural layers, while preserving the main-page dashboard capabilities the operator actively uses. After implementation feedback, this preservation requirement is explicit: the main page feature surface should remain intact, and `compare/control` is a growth area rather than a pruning target.

## Purpose / Big Picture

After this change, running `cookimport stats-dashboard --serve` should produce one dashboard that feels small enough to understand and maintain. A user should be able to open `index.html`, inspect recent benchmark and import history, inspect the latest benchmark diagnostics, and, when useful, inspect all-method benchmark summaries from inside that same page instead of chasing dozens of generated HTML files.

This matters because the current analytics surface is doing more work than the operator actually uses. Today the repo generates one main page plus many standalone all-method pages, while the implementation underneath has grown into a large collector, a large static frontend, and a second all-method rendering stack. The visible proof after implementation is simple: `stats-dashboard` writes one HTML page plus its shared assets, the UI still answers the practical questions the operator uses, and the code footprint for rendering and frontend behavior is materially smaller.

## Context and Orientation

This repository is a local Python CLI named `cookimport`. The command `cookimport stats-dashboard` builds a static analytics site from CSV-backed run history plus selective JSON fallback data. The default output root for repo-local usage is `.history/dashboard`. With `--serve`, the repo starts a simple local HTTP server and serves `index.html`, typically at `http://127.0.0.1:8765/index.html`.

The current analytics implementation has four major pieces. First, `cookimport/analytics/dashboard_collect.py` reads analytics data from the canonical history CSV and optional report scans and produces one in-memory `DashboardData` payload defined in `cookimport/analytics/dashboard_schema.py`. Second, `cookimport/analytics/dashboard_render.py` writes the static site directly using the real shell/CSS/JS owners in `cookimport/analytics/dashboard_renderers/`. Third, `cookimport/analytics/dashboard_state_server.py` serves the static files and allows browser UI state to round-trip through `assets/dashboard_ui_state.json` while `--serve` is active. Fourth, `cookimport/analytics/compare_control_*.py` provides a benchmark comparison and filtering engine that the dashboard frontend can use.

The renderer shape is now honest. `cookimport/analytics/dashboard_render.py` writes `index.html`, `assets/dashboard.js`, `assets/style.css`, `assets/dashboard_data.json`, and `assets/dashboard_ui_state.json` directly. The renderer package keeps only the real shell/CSS/JS owners plus checked-in JS source files. There is no extra HTML page family and no facade stack whose only job is to hide string assembly or asset reads.

The current frontend is static HTML plus one large CSS asset and one large assembled JavaScript asset. The JavaScript is assembled from Python string modules and checked-in `.js` files. The result is a feature-rich page with diagnostics cards, a benchmark history table, filters, trend charts, view presets, and compare/control analysis. That functionality is intentionally preserved, but the implementation size still means even small changes are likely to feel risky and expensive. Any follow-up simplification should therefore focus on clearer ownership and easier extension of the main-page code rather than removing capabilities the operator actively uses.

For this plan, “single-surface dashboard” means this exact product contract:

1. `cookimport stats-dashboard` writes one HTML document at `<dashboard root>/index.html`.
2. Shared assets under `<dashboard root>/assets/` remain allowed.
3. No extra benchmark-summary HTML pages are generated under `<dashboard root>/all-method-benchmark/`.
4. All useful analytics views must either live in the main page or be dropped.

For this plan, “simple and sane” does not mean throwing away analytics. It means keeping the practical operator questions front and center:

1. What happened in the latest benchmark run?
2. How are recent runs trending?
3. Which run settings, models, or variants are doing well or badly?
4. If all-method runs matter, what were the best configs and broad outcomes, without requiring a separate HTML site per run and per book?

## Plan of Work

Start by defining the minimum surviving dashboard product in code and tests before deleting anything. Read the current analytics tests in `tests/analytics/` and write down which behaviors are product-critical versus historical accretion. The minimum product should be the main `index.html` with a latest benchmark diagnostics section, a previous-runs table, a benchmark trend chart, practical filters, and a compact all-method summary surface if all-method reporting still matters. Everything else must justify its existence against that product.

In the implemented pass, that keep/fold/delete inventory became explicit. Keep the latest diagnostics cards, the previous-runs table/trend workspace, per-label diagnostics, existing filters, and compare/control. Fold all-method sweep reporting into the main page as a compact summary plus collapsed previous-runs rows. Delete the generated sweep run/detail HTML pages entirely.

Once that minimum was explicit, the implementation removed standalone all-method page generation at the renderer boundary. The old seam in `cookimport/analytics/dashboard_renderers/assets.py` that wrote assets and then called `_render_all_method_pages(out_dir, data)` was deleted, along with the now-unneeded renderer code in `cookimport/analytics/dashboard_renderers/all_method_pages.py`. The useful all-method information stayed alive as compact summary data for the main page rather than as run-level and per-book standalone HTML generation under a different abstraction.

After the page-family deletion, simplify the dashboard data model around one page. `cookimport/analytics/dashboard_schema.py` should describe only the payload needed by the main dashboard surface. `cookimport/analytics/dashboard_collect.py` should stop shaping data solely to support deleted all-method pages. If there are synthetic all-method records or ranking helpers used only by standalone pages, delete them. If some useful all-method rollup survives, make it an explicit and compact field on the main `DashboardData` payload instead of a side-channel understood only by a special renderer.

Next, simplify the renderer package. The target shape is straightforward: one collector module, one schema module, one small render entrypoint, one HTML shell, one CSS asset, and one JavaScript asset, all clearly centered on one page. Facades that exist only to stitch together a sprawling implementation should be collapsed where doing so makes ownership clearer. Keep a small number of files when they represent real boundaries, but delete “tiny facade over another tiny facade” layers that merely hide size without reducing it.

Then reduce frontend complexity carefully. Start from the surviving product contract and simplify implementation that creates disproportionate code bulk without shrinking the main-page feature surface. In practice this likely means:

1. Reorganize view presets, table-column resizing persistence, and compare/control code so each feature has clearer ownership.
2. Keep filtering, sorting, trend views, and compare/control behavior available from the main page.
3. Prefer simpler internal state management over a highly tangled browser mini-application.
4. Move the frontend toward ordinary checked-in `.js` source files as the source of truth instead of splitting core logic between giant Python string modules and giant asset files.

The frontend refactor should be done in behavior-first slices. First make the single-page product work with a simplified static layout and reduced script surface, then delete the old logic. Do not attempt a heroic “rewrite everything at once and hope the tests still describe the right product.” Add or update tests to lock in the smaller product at each step.

Finally, update the docs and regression suite. `docs/08-analytics/08-analytics_readme.md` and any other relevant analytics notes must stop documenting standalone all-method pages as part of the live contract. The tests under `tests/analytics/` should prove the new contract: one generated HTML page, useful diagnostics, working trend/history views, and no `all-method-benchmark/*.html` output. The docs should describe the dashboard the operator actually opens, not a reporting surface that happens to exist because the code kept growing.

## Milestone 1: Define The Surviving Dashboard Product

This milestone establishes the post-purge product before code deletion begins. At the end of the milestone, the repo has an explicit “keep / fold in / delete” map for dashboard features and a first set of tests that describe the intended single-page contract.

## Milestone 2: Delete Standalone All-Method Pages

This milestone removes the largest product sprawl seam. At the end of the milestone, running `cookimport stats-dashboard` writes `index.html` and assets only. No run-level or per-book HTML pages are emitted under `all-method-benchmark/`.

If all-method rollups are still useful, the milestone should also add a compact in-page summary section to `index.html`. That summary can be as small as a ranked table of recent all-method runs and their best configs. It should not recreate the old multipage navigation model inside the main page.

Acceptance for this milestone is behavioral: generate the dashboard and verify that `index.html` still renders and that the output directory no longer contains `all-method-benchmark/*.html`.

## Milestone 3: Collapse The Renderer To One Honest Path

This milestone reduces renderer indirection and deletes renderer code that existed only for the removed page family. At the end of the milestone, a novice should be able to read the rendering code and understand, in one pass, how `DashboardData` becomes `index.html`, `dashboard.js`, and `style.css`.

The surviving owners are `cookimport/analytics/dashboard_render.py`, `cookimport/analytics/dashboard_renderers/html_shell.py`, `cookimport/analytics/dashboard_renderers/style_asset.py`, `cookimport/analytics/dashboard_renderers/script_bootstrap.py`, `cookimport/analytics/dashboard_renderers/script_compare_control.py`, and the checked-in JS source files under `cookimport/analytics/dashboard_renderers/assets/`. The deleted seams were `assets.py`, `index_page.py`, `templates.py`, `script_asset.py`, and other tiny wrappers whose only job was to disguise size rather than represent a real boundary.

Acceptance is code and behavior together: fewer files or clearly justified file boundaries, fewer lines of renderer logic, and a passing dashboard generation test that still proves the main page works.

## Milestone 4: Shrink The Dashboard Data Contract

This milestone removes data shaping that only existed for deleted views. At the end of the milestone, `cookimport/analytics/dashboard_schema.py` and `cookimport/analytics/dashboard_collect.py` describe a smaller payload that directly serves the surviving page.

This is where synthetic all-method-only records, page-specific ranking helpers, or historical fallback logic that survives solely to feed deleted UI should be removed. Keep the CSV-first contract intact. The collector still needs to be authoritative and deterministic; it just does not need to over-prepare data for a dashboard surface that no longer exists.

Acceptance is that collector tests pass with the smaller payload and that the generated `assets/dashboard_data.json` is visibly simpler when inspected in a focused fixture.

## Milestone 5: Simplify Frontend Implementation Without Shrinking The Main Page

This milestone turns the dashboard from a static site with tangled frontend machinery into a smaller-feeling codebase with focused ownership. At the end of the milestone, the frontend still supports the practical benchmark-history workflow and the existing main-page tools, but it no longer carries as much incidental complexity in the implementation.

Approach this by implementation clarity, not by cutting product scope. Keep what materially helps answer the operator’s questions and what the operator explicitly values. Simplify or reorganize what mostly exists because the implementation grew over time. In particular, treat compare/control as a supported extension surface, while still reconsidering how view preset persistence, column resize persistence, and other stateful UI behavior are structured internally.

Acceptance is a clearer JavaScript and CSS ownership model, passing frontend-oriented analytics tests, and a manual sanity run of `cookimport stats-dashboard --serve` that still feels useful from the main page alone.

## Milestone 6: Rewrite The Docs Around One Dashboard

This milestone updates the docs after the code is clean. At the end of the milestone, the analytics docs describe one dashboard product and one output topology. A future contributor should not learn that there are many generated page families because there no longer are.

Update `docs/08-analytics/08-analytics_readme.md` first, then any supporting analytics notes under `cookimport/analytics/` that describe the dashboard contract. Keep the docs short and direct. The dashboard should be easy to explain once the implementation is smaller.

Acceptance is that the docs match the generated output exactly and that a novice can read the docs and know what files `stats-dashboard` writes without caveats.

## Validation and Acceptance

Acceptance is behavioral and product-facing, not merely structural.

First, `cookimport stats-dashboard --serve` must still work as the operator uses it. Run the command, open `http://127.0.0.1:8765/index.html`, and confirm that the page still shows the latest benchmark diagnostics and a useful previous-runs workflow from that one main page.

Second, the generated-output contract must be smaller. After the refactor, the dashboard output should contain:

1. `index.html`
2. `assets/dashboard_data.json`
3. `assets/dashboard.js`
4. `assets/style.css`
5. `assets/dashboard_ui_state.json`

It must not contain a generated `all-method-benchmark/` page family. If some all-method summary remains, it must be reachable from the main page without generating separate HTML files.

Third, the implementation must be simpler in a way a human can verify. The renderer should have one obvious path from `DashboardData` to `index.html`, and the frontend should no longer depend on giant sidecar logic solely for deleted dashboard surfaces. The plan does not require the collector to become tiny, but it does require that page-specific and feature-specific cruft be deleted when their product surface is deleted.

Fourth, tests must prove the new contract. At minimum:

1. Dashboard generation tests should assert the single-page output topology.
2. Any tests that used to assert standalone all-method pages should be rewritten to reflect the intentional product change.
3. Frontend or state-server tests should continue proving the one-page served dashboard works.

The work is complete when the operator can keep using the same main dashboard URL, the page remains useful, and the code no longer feels like it is secretly maintaining a second dashboard product off to the side.

## Idempotence and Recovery

This refactor is deletion-heavy, so make each deletion in a test-backed slice. It is safe to regenerate the dashboard repeatedly because `stats-dashboard` is a pure write of derived artifacts. If a deletion breaks a useful operator workflow, restore only the minimum necessary behavior inside the main page rather than reintroducing standalone page generation.

If a test fails during the refactor, classify it honestly before changing code or tests. State whether the failure is a regression, an intentional contract change, or an outdated test that encoded the dashboard sprawl being removed. Only then update the implementation or the test. Do not preserve a deleted surface just because an old test still expects it.

If the compare/control UI turns out to be too intertwined to shrink cleanly in one pass, keep the backend analysis engine stable and temporarily reduce the frontend surface to a simpler view rather than blocking the entire dashboard simplification on one advanced widget.

## Interfaces and Dependencies

Keep the existing CLI command entrypoint `cookimport/cli_commands/analytics.py::stats_dashboard`. The user-facing command should not change. The simplification is about what the command writes and how the dashboard code is structured, not about inventing a new entrypoint.

Keep `cookimport/analytics/dashboard_state_server.py` unless the simplification reveals it to be unnecessary. It already serves the one page the operator uses. Its role is compatible with the target architecture.
