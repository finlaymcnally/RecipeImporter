# Separate Operator-Kill Fallout From Knowledge Runtime Failures

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md`.

This plan complements, rather than replaces, the broader knowledge-runtime repair plans already checked into `docs/plans/`. Those broader plans focus on preventing watchdog churn, worker drift, stale follow-up calls, and packet-level progress bugs. This plan is narrower: it fixes the attribution problem that appeared in the Salt Fat postmortem, where operator-triggered termination and pre-kill runtime failures were too easy to blur together.

## Purpose / Big Picture

After this change, a novice reading a saved knowledge-stage run should be able to answer two separate questions without guessing. First: what was already broken before the operator stopped the run? Second: which missing artifacts or incomplete summaries are only missing because the operator killed the benchmark before wrap-up finished? Today those answers are mixed together in ad hoc postmortem prose and by omission in the artifact tree. After this change, the stage root and benchmark root should explicitly record interruption cause, finalization completeness, and which expected artifacts were skipped because the run was aborted.

The user-visible proof is simple. Reproduce an interrupted knowledge-heavy benchmark in a deterministic test or fake-runner harness. The resulting benchmark and stage roots should include a machine-readable interruption record that says the run was interrupted by the operator, a summary that distinguishes pre-kill runtime failures from kill-induced artifact gaps, and artifact manifests that say whether `phase_manifest.json`, `telemetry.json`, `promotion_report.json`, `failures.json`, `knowledge_manifest.json`, and `proposals/*` are present, intentionally absent because of interruption, or unexpectedly missing.

## Surprises & Discoveries

- Observation: the saved Salt Fat run already had strong evidence of genuine pre-kill failures before the operator kill happened.
  Evidence: worker `live_status.json` files under `data/output/2026-03-20_21.44.20/.../knowledge/workers/` already showed watchdog-killed main workers, malformed completions, stale follow-up calls, and large retry/repair churn before the benchmark was terminated.

- Observation: the missing stage-level knowledge artifacts in that same run are ambiguous by themselves.
  Evidence: under `data/output/2026-03-20_21.44.20/.../knowledge/`, the run retained `task_manifest.jsonl`, `shard_manifest.jsonl`, and `worker_assignments.json`, but lacked `phase_manifest.json`, `promotion_report.json`, `telemetry.json`, `failures.json`, `knowledge_manifest.json`, and `proposals/*`. After the operator kill, the artifact tree alone does not say whether those files are missing because of interruption or because the runtime failed earlier.

- Observation: benchmark-level interruption handling already leaves useful top-level clues, but stage-level knowledge artifacts do not mirror that clarity yet.
  Evidence: existing tests around `benchmark_status.json` and `partial_benchmark_summary.json` in `tests/labelstudio/test_labelstudio_benchmark_helpers_eval_payload_execution.py` show benchmark-root partial-finalization behavior, while the saved knowledge stage root still lacks an authoritative interruption marker of its own.

- Observation: this attribution bug is partly a product bug, not just a documentation bug.
  Evidence: the postmortem wording needed manual correction once the user pointed out that the missing stage-finalization artifacts were a consequence of the kill. That means the runtime currently does not leave enough evidence to keep future reports honest by default.

- Observation: worker-local `live_status.json` files are enough to preserve truthful pre-kill failure counts even when rich stage wrap-up never lands.
  Evidence: the implemented stage status writer can still count worker `state` and `reason_code` values from `knowledge/workers/*/live_status.json` after an interrupted run, so the saved status distinguishes real pre-kill failures from missing wrap-up files.

- Observation: artifact attribution stays honest only if it is recomputed against the filesystem at read time.
  Evidence: the shared classifier turns a declared-`present` artifact into `unexpectedly_missing` when the file is actually absent, which prevents stale metadata from hiding partial copies or broken finalization.

## Context and Orientation

The affected system has three layers.

The first layer is the knowledge stage runtime in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. In this repository, the “knowledge stage root” is the directory under `data/output/<ts>/.../raw/llm/<workbook_slug>/knowledge/` that is supposed to contain stage-level runtime artifacts such as `phase_manifest.json`, `promotion_report.json`, `telemetry.json`, `failures.json`, shard proposals, and worker-local subdirectories. In the interrupted Salt Fat run, that root only retained the early-planned manifests plus worker-local artifact trees.

The second layer is benchmark interruption handling in `cookimport/cli.py` and the benchmark helper seams exercised by `tests/labelstudio/test_labelstudio_benchmark_helpers_eval_payload_execution.py`. This layer already writes benchmark-root artifacts such as `benchmark_status.json` and `partial_benchmark_summary.json` when a benchmark is interrupted. Those files help at the benchmark root, but they do not currently guarantee that the knowledge stage root carries its own interruption-aware explanation.

The third layer is postmortem and artifact reading. In practice, that includes human-written reports under `docs/reports/`, prompt-budget readers, upload-bundle readers, and any future operator summary helpers. These readers need a stable way to tell whether a missing artifact means “the stage never finalized because the user stopped it” versus “the stage claimed to finish but never wrote what it promised.” Without that distinction, honest reports require manual reconstruction from process history and worker subtrees.

The concrete saved run for this plan is:

`data/golden/benchmark-vs-golden/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/`

with the corresponding interrupted knowledge stage root at:

`data/output/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-20_21.44.56/raw/llm/saltfatacidheatcutdown/knowledge/`

That run is the regression fixture. It already contains real pre-kill failures. The plan is not to erase or reinterpret those failures. The plan is to leave a future saved run with enough structured evidence that a reader can say, “these failures were real before the kill, and these missing wrap-up files are missing because the operator stopped the parent.”

## Plan of Work

Start by defining one small, explicit attribution vocabulary shared by the knowledge stage and the benchmark root. The shipped fields are `stage_state`, `termination_cause`, and `finalization_completeness`. The implemented values stay plain: `completed`, `completed_with_failures`, `failed`, `interrupted`, `runtime_error`, `operator_interrupt`, `complete`, `stopped_before_finalization`, and `interrupted_before_finalization`.

Then add a lightweight stage-root status artifact in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. The implemented artifact is `stage_status.json` beside the other knowledge runtime files. Its job is not to replace `phase_manifest.json` or `telemetry.json`. Its job is to survive cases where those richer artifacts were never written. It records the stage termination cause, whether final stage summaries were completed, what pre-kill worker outcomes were already known, and an artifact-state map for the expected stage files. The shipped artifact-state map uses `present`, `skipped_due_to_interrupt`, and `unexpectedly_missing`.

After that, update the benchmark interruption path in `cookimport/cli.py` so the benchmark-root partial summary explicitly points at the stage-root attribution artifact when one exists. The benchmark root should also echo the distinction between “pre-kill runtime failures already observed” and “missing wrap-up artifacts caused by interruption.” Keep the benchmark summary short and machine-readable first, human-readable second.

Next, add a small summary helper or normalization seam for artifact readers. This can live wherever the current benchmark and prompt-budget readers already normalize stage artifact paths. The helper should accept a stage root and return a normalized object saying which expected artifacts are present and why any missing artifacts are missing according to the new attribution artifact. This prevents each future report from re-implementing the same reasoning.

Finally, update the short docs so they describe the new interruption semantics. The docs should be explicit that missing `phase_manifest.json` and friends are only evidence of a bug when the stage finalization status says they were expected but not written. If the stage says it was interrupted before finalization, then those files are a kill consequence, not proof of a deeper runtime-finalization bug.

## Validation and Acceptance

The implementation is complete when all of the following are true.

First, an interrupted knowledge-stage run leaves a stage-root status artifact that explicitly says the stage was interrupted and whether rich finalization artifacts were skipped because of that interruption. A reader should not need the process table or a human report to know this.

Second, the benchmark root leaves a partial summary that distinguishes two categories:

- failures already observed before interruption, such as watchdog-killed workers or invalid outputs;
- artifact gaps caused by interruption, such as missing `phase_manifest.json` or `promotion_report.json`.

Third, an uninterrupted successful run still classifies the same artifacts as `present`, and an intentionally injected broken-finalization test classifies truly missing expected artifacts as `unexpectedly_missing`, not `skipped_due_to_interrupt`.

Fourth, the docs clearly state the new rule: a missing stage-finalization file is not automatically evidence of a runtime bug if the saved stage status says the run was interrupted before finalization.

## Idempotence and Recovery

This work should be additive and safe to repeat. The new attribution artifact should be a small JSON file written beside the stage root, so retrying a local test run simply overwrites that run’s own status file. If it proves too invasive to guarantee rich artifact finalization under all interruption paths, keep the fallback artifact tiny and robust. It is better to always write one small truthful status file than to promise a larger finalization package and miss it again.

Do not reinterpret old saved runs retroactively unless the evidence is explicit. For historical runs that predate the new attribution artifact, readers should continue to say “interruption cause inferred from surrounding evidence” rather than pretending the distinction is authoritative. The whole point of this plan is to stop relying on inference for future runs.

    - stage key
    - stage state
    - termination cause
    - finalization completeness
    - known pre-kill worker outcome counts

Keep the interface plain and stable. Prefer explicit JSON field names like:

    termination_cause
    finalization_completeness
    pre_kill_failure_counts
    artifact_states

Do not hide this behind overly generic abstractions. The whole purpose is to make future human and automated readers less likely to confuse interruption fallout with actual runtime failures.
