Problem statement

Knowledge worker assignments currently spend up to two happy-path workspace sessions per assignment: one for classification and a second for grouping. The goal is to keep classification-first semantics while letting the same worker validate, advance, repair, and finish grouping inside one session.

Acceptance criteria

- A knowledge worker assignment uses one happy-path `run_workspace_worker(...)` call.
- The worker can edit classification answers, run one repo-owned helper, receive a grouping `task.json`, and finish grouping in the same session.
- Structural mistakes rewrite `task.json` into repair mode in the same session instead of launching a second happy-path worker session.
- Telemetry, prompt-budget, and stage observability report one happy-path session per assignment plus same-session transition counts.

Verification steps

- `source .venv/bin/activate && pytest tests/llm/test_knowledge_same_session_handoff.py`
- `source .venv/bin/activate && pytest tests/llm/test_knowledge_orchestrator_runtime_leasing.py`
- `source .venv/bin/activate && pytest tests/llm/test_codex_exec_runner_workspace.py`
- `source .venv/bin/activate && pytest tests/staging/test_stage_observability.py -k knowledge`

Evidence

- Before the change, the runtime test expects two happy-path workspace sessions for one worker assignment.
- After the change, the runtime test expects one happy-path workspace session plus same-session transition counters.

Constraints / gotchas + decision

- Keep the semantic split: classification stays grouping-free and grouping still sees only accepted knowledge rows.
- Do not reintroduce model-visible helper trees. The helper uses hidden repo-owned state under `_repo_control` plus one blessed command from the sterile workspace.

Rollback / compensation plan

- Revert the knowledge same-session helper, runtime handoff wiring, and reporting changes together.
- If one-session tests fail, do not keep the reporting/docs updates without the runtime cutover.
