Problem statement

Broad pytest runs can trigger heavyweight benchmark follow-up side effects such as dashboard refresh scans, upload-bundle generation, and Oracle background launch from tests that only meant to exercise routing or summary logic. In this repo that can spike CPU, RAM, and disk activity hard enough to make the workstation unusable.

Acceptance criteria

- Routine pytest runs default to skipping heavyweight benchmark follow-up side effects.
- The skip behavior applies broadly enough that child processes and helper tests inherit it by default.
- Tests that intentionally exercise the real heavy paths must opt in explicitly.
- Label Studio benchmark-helper tests use lightweight defaults instead of the real dashboard/upload-bundle paths unless they override them on purpose.

Verification steps

- Inspect `tests/conftest.py` and confirm pytest now exports a default heavy-side-effect disable env for the session.
- Inspect `cookimport/cli_support/dashboard.py` and `cookimport/cli_support/bench_oracle.py` and confirm heavy follow-up helpers return early when the guard is active.
- Inspect `tests/labelstudio/benchmark_helper_support.py` and confirm the shared helper suite now stubs dashboard refresh and upload-bundle generation by default.
- Inspect `tests/bench/test_benchmark_oracle_upload.py` and confirm the direct Oracle-upload test opts in explicitly.
- Run:
  `source .venv/bin/activate && python -m py_compile cookimport/cli_support/test_safety.py cookimport/cli_support/dashboard.py cookimport/cli_support/bench_oracle.py tests/conftest.py tests/labelstudio/benchmark_helper_support.py tests/bench/test_benchmark_oracle_upload.py tests/labelstudio/test_labelstudio_benchmark_helpers_single_book_artifacts.py`

Evidence

- Added a shared runtime guard module for heavyweight pytest side effects.
- Added pytest-session env defaults so the guard is fail-closed during test runs.
- Replaced routine Label Studio benchmark-helper heavy follow-up work with lightweight shared stubs.
- The touched Python files passed `python -m py_compile`.
- I did not rerun pytest because the failure mode being fixed was "broad pytest can destabilize the workstation"; the safe verification here was static inspection plus syntax compilation.

Constraints/gotchas + brief decision

- Do not disable the public `stats-dashboard` or upload-bundle commands outright; only the best-effort follow-up helpers should skip automatically during pytest.
- Use an explicit opt-in env escape hatch so dedicated tests can still cover the heavy paths.
- Keep the default helper-suite stubs lightweight but shape-compatible with existing single-book assertions.

Rollback / compensation plan

- Revert the new guard module and the pytest env defaults.
- Remove the new helper-suite stubs and return the benchmark-helper tests to their previous behavior.
