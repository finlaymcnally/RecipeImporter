# Fail closed on recipe correction outputs that omit mapping reasons

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The current recipe-correction runtime accepts a bad class of worker output: a recipe can be marked `repaired`, include multiple ingredients and multiple steps, return an empty ingredient-to-step mapping in `m`, and omit the machine-readable explanation in `mr`. The runtime only notices this later, after promotion, where the recipe ends up as `mapping_status="missing_reason"` and `structural_status="degraded"`. After this change, fresh recipe runs must fail closed on that invalid shape. A repaired recipe must either provide a real mapping, or provide an explicit reason token explaining why the empty mapping is acceptable. The user-visible proof is that targeted LLM tests reject the old bad payload, orchestrator tests stop promoting it as a successful repaired result, and an approved benchmark rerun no longer produces fresh `empty_mapping_without_reason` recipe rows.

## Surprises & Discoveries

- Observation: this is not primarily a model-generation failure; it is a contract-enforcement failure.
  Evidence: `llm_pipelines/prompts/recipe.correction.compact.v1.prompt.md` already says “When returning `[]`, set `mr`...”, but `cookimport/llm/codex_farm_contracts.py` still defines `ingredient_step_mapping_reason` as optional and does not reject blank `mr` when `m` is empty.

- Observation: the saved bad run returned structurally plausible recipes, not blank garbage.
  Evidence: `data/golden/benchmark-vs-golden/2026-03-22_14.44.46/.../prompts/prompt_recipe_llm_correct_and_link.txt` shows rows like “Bright Cabbage Slaw” with canonical ingredients and steps, `m: {}`, and `mr: null`.

- Observation: fresh runs can silently carry this invalid shape all the way into per-recipe audits.
  Evidence: `data/output/2026-03-22_14.44.46/.../recipe_correction_audit/urn_recipeimport_..._label_span_0.json` shows `repair_status: "repaired"`, `ingredient_step_mapping: {}`, `ingredient_step_mapping_reason: null`, while `deterministic_final_assembly.mapping_status` is already `missing_reason`.

- Observation: the repo already knows how to classify the failure; it just does so too late.
  Evidence: `cookimport/llm/codex_farm_orchestrator.py::_classify_recipe_correction_mapping_status(...)` returns `missing_reason`, and `_classify_recipe_correction_structural_audit(...)` emits `empty_mapping_without_reason` for exactly this shape.

- Observation: single-step recipes are already treated as a legitimate empty-mapping exception, so the fix must stay narrow.
  Evidence: the same bad run contains recipes with `mr="not_needed_single_step"` and `mapping_status="not_needed"`, and those rows are not degraded.

- Observation: the workspace worker’s shared `prompt.txt` is not the only helper-facing contract surface; the detailed compact rule actually lands in `SHARD_PACKET.md`.
  Evidence: the final patch added the fail-closed wording to `render_recipe_worker_shard_packet(...)`, and the targeted regression in `tests/llm/test_recipe_phase_workers.py` now asserts the emitted `SHARD_PACKET.md` contains `2+ non-empty ingredients` plus `unclear_alignment`.

- Observation: once the contract rejects the bad shape, the orchestrator no longer produces a degraded repaired row; it settles on a normal recipe-stage error path after the existing repair ladder exhausts itself.
  Evidence: the new regression in `tests/llm/test_codex_farm_orchestrator.py` records `recipe_correction_error=1`, `build_final_recipe=error`, empty final overrides, and `proposal["payload"] is None`.

- Observation: the first fail-closed rule was still too narrow because compact recipe steps are not the final authority on how many steps the user will see.
  Evidence: in the saved run rooted at `data/output/2026-03-22_16.52.22/.../2026-03-22_16.53.00/...`, `label_span_20` and `label_span_23` have `output.step_count=1`, `ingredient_step_mapping_reason=null`, and `structural_audit.reason_codes=["empty_mapping_without_reason"]`, while `deterministic_final_assembly.final_step_count=4`.

- Observation: ingredient count alone is enough to require an explicit empty-mapping reason for repaired outputs.
  Evidence: the surviving benchmark misses are dressings with five or six ingredients and one compact “do everything” step. Even before deterministic step splitting, returning `m={}` with blank `mr` is ambiguous and should fail closed.

## Outcomes & Retrospective

The recipe-correction stage now stops silently accepting “repaired but unexplained empty mapping” outputs under the broader rule that survived the benchmark follow-up. A repaired payload with 2+ non-empty ingredients or 2+ non-empty steps must now provide either a real mapping or a non-empty reason token. The exact formerly accepted payload from this plan now raises a `ValidationError` at `MergedRecipeRepairOutput.model_validate(...)`, the worker-local draft validator rejects the same shape, and the orchestrator regression proves that the bad output no longer promotes as a successful repaired recipe.

The main lesson held: prompt text and human-facing helper docs were not enough while the runtime contract stayed permissive. The durable fix was to make the acceptance seam fail closed, then mirror that same rule in the worker helper path so local draft validation matches orchestrator behavior. The benchmark follow-up added one more lesson: compact recipe step counts are not always the last word, so the acceptance rule cannot rely on “multiple steps” alone.

No new benchmark rerun was performed after the broadened rule landed. The only benchmark evidence recorded here is from the already-existing user-provided run rooted at `2026-03-22_16.52.22`, which exposed the remaining one-step compact-output gap. A fresh rerun remains optional and approval-gated.

## Context and Orientation

The recipe-correction stage lives in `cookimport/llm/codex_farm_orchestrator.py` and `cookimport/llm/codex_farm_contracts.py`. In this repository, “recipe correction” means the Codex-backed phase that takes deterministic candidate recipes, decides whether each one is a real recipe, fragmentary, or not a recipe, and then returns a compact JSON result per recipe. The compact keys are:

- `st`: repair status such as `repaired`
- `cr`: canonical recipe object with title, ingredients, steps, description, and yield
- `m`: ingredient-to-step mapping, meaning which ingredient indexes clearly belong to which step indexes
- `mr`: machine-readable reason for an empty or non-standard mapping

The live prompt asset for this stage is `llm_pipelines/prompts/recipe.correction.compact.v1.prompt.md`. Workspace-worker helper guidance is mirrored from `cookimport/llm/recipe_workspace_tools.py`. The contract parser that turns raw worker JSON into repo models is `cookimport/llm/codex_farm_contracts.py`, especially the `MergedRecipeRepairOutput` model. After a worker result validates, the orchestrator promotes it into deterministic final assembly in `cookimport/llm/codex_farm_orchestrator.py`, where helper functions such as `_classify_recipe_correction_mapping_status(...)` and `_classify_recipe_correction_structural_audit(...)` currently detect the missing-reason problem after acceptance.

The motivating evidence comes from the saved benchmark run rooted at `data/golden/benchmark-vs-golden/2026-03-22_14.44.46/single-book-benchmark/saltfatacidheatcutdown/` and its paired runtime outputs under `data/output/2026-03-22_14.44.46/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-22_14.45.22/`. That run is valuable because it shows the exact failure mode in a real artifact tree:

- `recipe_manifest.json` records many `correction_output_status="repaired"` rows that also have `mapping_status="missing_reason"`.
- `recipe_correction_audit/*.json` preserves the per-recipe canonical recipe, empty mapping, and degraded structural audit.
- `prompts/prompt_recipe_llm_correct_and_link.txt` preserves both the prompt input and the accepted raw output that triggered the issue.

The relevant tests already live in this area:

- `tests/llm/test_codex_farm_contracts.py` covers parsing and normalization for compact worker payloads.
- `tests/llm/test_recipe_phase_workers.py` covers workspace-worker output shapes and helper-oriented flows.
- `tests/llm/test_codex_farm_orchestrator.py` covers promotion, manifest writing, and fake-runner orchestration behavior.

## Milestones

### Milestone 1: Make the compact recipe contract fail closed

This milestone strengthens the acceptance seam in `cookimport/llm/codex_farm_contracts.py`. At the end of it, a repaired recipe that has at least two non-empty ingredients and at least two non-empty steps may not return an empty `m` with blank `mr`. The contract parser must reject that payload before the orchestrator treats it as a valid repaired result.

This is the key behavioral shift: the bad payload is no longer “accepted but later downgraded.” It becomes invalid input at the contract boundary. The expected proof is a focused contract test that fails before the change and passes after it.

### Milestone 2: Align prompt and helper guidance with the runtime rule

This milestone updates the human/model-facing guidance so the cheapest correct behavior is obvious. The prompt asset in `llm_pipelines/prompts/recipe.correction.compact.v1.prompt.md` and the helper text in `cookimport/llm/recipe_workspace_tools.py` should both say, in plain language, that repaired recipes with empty mappings must provide an explicit reason token, and that multi-step or multi-ingredient recipes without either a mapping or a reason are invalid.

The proof is textual and testable: the prompt asset and helper quick-reference examples now match the runtime rule, and prompt-oriented tests or fixture assertions can point at the new language directly.

### Milestone 3: Stop the orchestrator from promoting the old bad shape

This milestone ensures the orchestrator path in `cookimport/llm/codex_farm_orchestrator.py` treats the invalid payload as a failed proposal instead of a successful repaired one. Fresh fake-runner tests should show that the stage does not produce a normal promoted repaired result when `m` is empty and `mr` is missing for a multi-step repaired recipe.

The proof is an orchestrator regression that previously accepted the payload and now rejects it, plus manifest/runtime assertions showing the stage records failure or invalid output rather than a silently degraded repaired result.

### Milestone 4: Lock the behavior into docs and local acceptance

This milestone updates the short LLM docs in `cookimport/llm/README.md` and `docs/10-llm/10-llm_README.md` so future contributors know the active recipe-correction contract. It also finalizes the narrow test set that proves the change locally without spending benchmark tokens.

If the user later approves spend, this milestone also extends to one controlled benchmark rerun of the same Salt Fat slice and records the before/after artifact differences here.
