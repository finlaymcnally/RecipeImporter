# Burn the boats facade purge

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

RecipeImport has already done the hard part of several refactors: the new owning modules exist. What remains is the coward’s ending that AI coding agents love to leave behind: compatibility facades, sync wrappers, alias readers, and tests that keep monkeypatching old locations so the wrappers never die. That leftover machinery makes the code lie about where behavior lives and which contracts are still real.

After this change, a reader should be able to open the code and see one truth only. The CLI root should be a real composition root instead of a monkeypatch relay. `labelstudio.ingest` and `staging.import_session` should either be honest owners or disappear as facades. Knowledge worker output should accept only the current packet contract. Freeform labels and recipe-mode settings should accept only current names. The visible proof is that the targeted facades and alias readers are deleted, the tests patch the real owner modules instead of compatibility exports, and the repo’s remaining searches for `compat`, `legacy`, or removed names in active code go to zero or near-zero.

This plan is intentionally destructive. It is not a migration plan. It is not a “keep old imports working” plan. It is not a “compat for tests” plan. The replacement paths already exist. The goal is to delete the old seams and make wrong inputs fail.

## Context and Orientation

The three main facade problems are:

First, the CLI. In this repository, a “composition root” means the module that builds the Typer app tree and wires command groups together. A healthy composition root should do little more than instantiate apps, register command groups, and hand off to owning modules. The current [cookimport/cli.py](/home/mcnal/projects/recipeimport/cookimport/cli.py) is not only a composition root. It re-exports almost the entire helper surface from [cookimport/cli_support.py](/home/mcnal/projects/recipeimport/cookimport/cli_support.py), wraps those exports so patched globals are copied around at call time, and repopulates command-module globals so old tests that patch `cookimport.cli.*` still work. The active implementation ownership already lives in `cookimport/cli_commands/*.py`; the wrapper layer is what keeps the old fiction alive.

Second, Label Studio ingest. The real implementation now lives under [cookimport/labelstudio/ingest_flows/](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest_flows) plus [cookimport/labelstudio/ingest_support.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest_support.py). But [cookimport/labelstudio/ingest.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest.py) still mutates several submodules before dispatch so callers and tests can patch the old top-level names. That makes `ingest.py` look like an owner even though it mostly exists as a relay.

Third, staging import session. The real implementation path is [cookimport/staging/import_session_flows/output_stage.py](/home/mcnal/projects/recipeimport/cookimport/staging/import_session_flows/output_stage.py) plus a few helpers in `authority.py`, `reporting.py`, and `pipeline_runtime.py`. But [cookimport/staging/import_session.py](/home/mcnal/projects/recipeimport/cookimport/staging/import_session.py) still mutates those modules before delegating so old import and monkeypatch sites keep functioning.

The runtime contract leftovers are separate from those facades. They live in modules that still accept removed shapes or names:

- [cookimport/llm/codex_farm_knowledge_ingest.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_ingest.py) still rewrites old knowledge payload fields into the new packet schema and invents missing `reason_code` values.
- [cookimport/labelstudio/label_config_freeform.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/label_config_freeform.py) still remaps old freeform labels into the current label set.
- [cookimport/labelstudio/export.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/export.py) still recognizes legacy task scopes `pipeline` and `canonical-blocks`.
- [cookimport/config/run_settings.py](/home/mcnal/projects/recipeimport/cookimport/config/run_settings.py) and [cookimport/labelstudio/ingest_flows/normalize.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest_flows/normalize.py) still accept old recipe-mode names.
- [cookimport/llm/codex_farm_runner.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_runner.py) still carries an unused legacy stderr progress regex.

The implementation rule for this plan is simple: when you find a seam that exists only to keep the old name, old patch point, or old payload alive, delete it and move the tests to the new truth. Do not invent a new compatibility layer to soften the landing.

## Plan of Work

Start with the CLI because it is the largest ownership lie and the broadest blast radius. In [cookimport/cli.py](/home/mcnal/projects/recipeimport/cookimport/cli.py), delete `_compat_export`, `_publish_runtime_compat_exports`, and `_wrap_typer_callbacks`. Stop wildcard-importing the full helper surface from `cli_support`. Replace the current root with a plain composition root that imports the command modules, builds the Typer trees, registers command families, and exports only the public app objects plus any intentionally public non-command helpers that truly belong at the root. The goal is that `cookimport/cli.py` no longer mutates itself based on `cli_support`, and `cli_support.py` no longer imports `cookimport.cli` back to resynchronize globals.

To make that possible, rewrite the patch-heavy tests and helper callers first or alongside the cut. Anywhere a test currently patches `cookimport.cli.<something>`, move that patch to the actual owning module. For bench command helpers, that means `cookimport/cli_commands/bench.py`. For stage command helpers, that means `cookimport/cli_commands/stage.py`. For analytics, compare-control, interactive mode, and Label Studio command helpers, patch the corresponding module in `cookimport/cli_commands/`. If one small group of tests genuinely needs a stable import point for direct command invocation, create one explicit and narrow testing helper module under `tests/` instead of preserving a runtime compatibility surface in production code.

Once the CLI is honest, do the same surgery to Label Studio ingest. In [cookimport/labelstudio/ingest.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest.py), delete `_sync_ingest_compat_state`, the local `_compat_export`, and the wrapped re-exports. Either turn `ingest.py` into a thin honest module that directly re-exports selected functions without mutation, or delete it as a compatibility module and move imports to the true owners under `ingest_flows` and `ingest_support`. The default choice for this plan is more aggressive: keep `ingest.py` only if it becomes a zero-magic public API file. If removing it is practical, remove it from active imports and let callers import the real owners directly.

Then cut the staging facade. In [cookimport/staging/import_session.py](/home/mcnal/projects/recipeimport/cookimport/staging/import_session.py), delete `_sync_import_session_compat` and the delegation wrapper. Make callers import [cookimport/staging/import_session_flows/output_stage.py](/home/mcnal/projects/recipeimport/cookimport/staging/import_session_flows/output_stage.py) directly for `execute_stage_import_session_from_result`, and patch tests at the owning locations. If the `StageImportSessionResult` type is the only part worth keeping as a top-level import, re-export that one type honestly without any global mutation.

After the facade deletions, tighten runtime contracts so the repo stops silently accepting old inputs. In [cookimport/llm/codex_farm_knowledge_ingest.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_ingest.py), remove `_normalize_legacy_semantic_payload` and `_apply_legacy_reason_code_defaults`. `normalize_knowledge_worker_payload(...)` must validate only the current semantic packet result v1 contract and raise on missing `packet_id`, stale `bundle_id`, stale `bundle_version`, or missing `reason_code`. Rewrite the affected tests so they assert rejection instead of normalization.

Tighten the freeform label surface next. In [cookimport/labelstudio/label_config_freeform.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/label_config_freeform.py), delete `_FREEFORM_LABEL_ALIASES` and make `normalize_freeform_label(...)` normalize only formatting differences that preserve the same canonical spelling, such as case, spaces, or hyphens. Do not remap old ontology names. Then update [cookimport/labelstudio/eval_freeform.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/eval_freeform.py) and any callers so invalid labels are either rejected or explicitly ignored with a visible error path. The important part is that a removed label name can no longer masquerade as a current one.

Then remove the remaining old names from run settings and ingest normalizers. In [cookimport/config/run_settings.py](/home/mcnal/projects/recipeimport/cookimport/config/run_settings.py) and [cookimport/labelstudio/ingest_flows/normalize.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest_flows/normalize.py), accept only `extract` and `benchmark` for `codex_farm_recipe_mode`. Delete support for `default`, `line-label`, and `line-labels`, and rewrite tests to expect failure on those removed names. The point is that the UI and the runtime should speak the same vocabulary.

Finish by deleting smaller active leftovers that still teach removed paths. In [cookimport/labelstudio/export.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/export.py), stop recognizing `pipeline` and `canonical-blocks` as valid scopes for current export flows. The export path should support `freeform-spans` only and fail clearly when older project types are detected. In [cookimport/llm/codex_farm_runner.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_runner.py), remove the dead `_CODEX_FARM_LEGACY_RUN_PROGRESS_PATTERN`. If it is not used, it does not belong in the file. If implementation uncovers other dead compatibility names during this pass, delete them in the same spirit rather than opening a follow-up loophole.

As each area is cut over, update the nearby short docs so they describe the surviving ownership map and only the surviving contract names. At minimum update [docs/02-cli/02-cli_README.md](/home/mcnal/projects/recipeimport/docs/02-cli/02-cli_README.md), [docs/05-staging/05-staging_readme.md](/home/mcnal/projects/recipeimport/docs/05-staging/05-staging_readme.md), [docs/06-label-studio/06-label-studio_README.md](/home/mcnal/projects/recipeimport/docs/06-label-studio/06-label-studio_README.md), [cookimport/cli_commands/README.md](/home/mcnal/projects/recipeimport/cookimport/cli_commands/README.md), [cookimport/labelstudio/ingest_flows/README.md](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest_flows/README.md), and [cookimport/staging/import_session_flows/README.md](/home/mcnal/projects/recipeimport/cookimport/staging/import_session_flows/README.md). Keep the notes short and specific: who owns what now, and which removed names/contracts no longer work.

## Milestones

### Milestone 1: Delete facade ownership lies

At the end of this milestone, the big sync facades are gone or reduced to honest zero-magic public API files. The CLI root is a plain Typer composition root. Label Studio ingest and staging import session no longer mutate downstream modules before dispatch. A newcomer can follow the real owner modules directly.

The work in this milestone is primarily mechanical but not optional: update tests first, then cut the shims. When a test currently monkeypatches `cookimport.cli`, `cookimport.labelstudio.ingest`, or `cookimport.staging.import_session`, move that patch to the actual owner module and then delete the sync function that existed only to keep the old patch point alive. The acceptance proof is that the CLI and the targeted domain tests pass without any facade sync helpers left in active code.

### Milestone 2: Tighten runtime contracts to current names only

At the end of this milestone, the runtime no longer accepts old knowledge payload keys, old freeform label names, old task scopes, or old recipe-mode aliases. Removed inputs now fail loudly. This is the step that makes the refactor true rather than cosmetic.

The work here is to delete normalization helpers that translate removed inputs into current ones, then rewrite tests from “alias accepted” to “removed name rejected.” The acceptance proof is that the targeted validation tests pass and the old alias strings now appear only in intentional rejection tests or historical docs.

### Milestone 3: Update docs, notes, and seam searches

At the end of this milestone, the repository documentation and folder notes match the surviving code. A reviewer should not learn two ownership models or two contracts at once. The acceptance proof is a final search sweep that shows the targeted compatibility terms are gone from active code and active docs, with any remaining hits limited to historical plans/logs or explicit rejection tests.

## Validation and Acceptance

This plan is complete only when all of the following are true.

The public CLI still works, but [cookimport/cli.py](/home/mcnal/projects/recipeimport/cookimport/cli.py) is visibly just a composition root. Opening it should not show wildcard helper imports, runtime export publication, or callback wrapping used to preserve old patch points.

Either they are deleted from active imports or they are reduced to honest direct imports and re-exports without sync machinery.

Knowledge worker outputs that use stale top-level keys such as `bundle_id` or omit required `reason_code` fields now fail validation. The focused LLM tests should demonstrate that rejection.

Freeform label inputs that use removed names such as `TIP`, `NOTE`, `NOTES`, `VARIANT`, `YIELD`, or `TIME` no longer silently normalize into current labels. The freeform tests should now assert rejection or explicit invalid-label handling.

`codex_farm_recipe_mode` accepts only `extract` and `benchmark`. Removed names such as `default`, `line-label`, and `line-labels` fail with a clear error.

Label Studio export for the current flow supports only `freeform-spans`. Legacy scope detection for `pipeline` and `canonical-blocks` is no longer treated as a live supported surface.

The targeted test slices above pass inside the project `.venv`, and `npm run docs:list` still completes successfully after the new plan/doc updates.

## Idempotence and Recovery

The steps in this plan are safe to repeat because they are deletion-oriented and test-driven. Re-running the searches and focused tests should confirm the same surviving state without creating new artifacts beyond ordinary test outputs.

If a facade deletion breaks a broad set of tests, do not restore the sync shim by default. First move the failing tests to the owning module patch points and re-run the targeted slice. Only revert a deletion if you discover a real production caller that still depends on the old public surface and you cannot move that caller in the same change. Even then, prefer a temporary direct re-export over a mutable sync layer.

If a contract-tightening change breaks old fixture files, update the fixtures to the current contract unless the test is intentionally proving rejection behavior. Old-shape fixtures should survive only inside explicit negative tests.
