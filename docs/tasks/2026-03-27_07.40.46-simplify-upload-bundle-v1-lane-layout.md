# Simplify Upload Bundle V1 Lane Layout

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a human can open one benchmark `upload_bundle_v1/` directory and immediately see the two review packets that matter now: `quality/` and `token/`. Each lane folder will contain one coherent first-pass upload packet with exactly three main files: `overview.md`, `index.json`, and `payload.json`. The old top-level shared packet files and timestamped per-lane staging folders will no longer be the primary contract for current benchmark runs.

The user-visible goal is simple: `upload_bundle_v1/` should read like a handoff folder a human can upload manually without guessing which generated files are superseded. Oracle automation still needs its own runtime state, but that state should stop polluting the main handoff surface.

## Surprises & Discoveries

- Observation: the current bundle already has two different packet concepts layered on top of each other.
  Evidence: the example `upload_bundle_v1/` contains top-level `upload_bundle_overview.md`, `upload_bundle_index.json`, and `upload_bundle_payload.jsonl`, while `.oracle_upload_runs/<timestamp>-quality/staging/` and `.oracle_upload_runs/<timestamp>-token/staging/` separately contain the lane-specific packet a human actually wants.

- Observation: RecipeImport Oracle upload currently stages a lane-specific payload subset at send time instead of making that lane packet the durable bundle contract.
  Evidence: `cookimport/bench/oracle_upload.py` builds lane briefs and selected payload subsets into per-run staging under `.oracle_upload_runs/.../staging/`, while its source bundle reader still points at top-level `upload_bundle_index.json` and `upload_bundle_payload.jsonl`.

## Decision Log

- Decision: the durable handoff contract will become `upload_bundle_v1/{quality,token}/` with no timestamped lane folder names.
  Rationale: those are the only two review packets the user wants to inspect or upload manually, and timestamps inside the handoff tree add clutter without adding meaning.
  Date/Author: 2026-03-27 / Codex

- Decision: the lane packet filenames will be normalized to `overview.md`, `index.json`, and `payload.json`.
  Rationale: shorter names make the human upload surface easier to scan, and the lane folder already tells you this is part of `upload_bundle_v1`.
  Date/Author: 2026-03-27 / Codex

- Decision: Oracle runtime-only launch state should not remain part of the main `upload_bundle_v1/` browsing experience.
  Rationale: logs, follow-up state, and reattach metadata are operational artifacts, not first-pass handoff files. The user explicitly wants the bundle itself to be coherent for manual use.
  Date/Author: 2026-03-27 / Codex

## Outcomes & Retrospective

New benchmark bundles now write only two durable lane folders under `upload_bundle_v1/`: `quality/` and `token/`. Each lane contains `overview.md`, `index.json`, and `payload.json`. For normal single-book/single-profile bundles, each lane keeps the full artifact-row payload so follow-up/debug tooling can still walk the bundle from one lane packet. For `high_level_only` group bundles, each lane payload stays lane-curated so the total two-lane bundle can still respect the configured size cap.

Oracle upload now treats those lane folders as the source of truth. Browser uploads stage temporary lane-local copies and, when needed, a trimmed upload-time `payload.json` subset from the durable lane packet. Oracle runtime logs/status moved to a sibling `.oracle_upload_runs/` directory beside `upload_bundle_v1/`, which keeps the handoff folder itself clean for manual inspection and manual upload.

## Context and Orientation

`upload_bundle_v1` is RecipeImport’s benchmark handoff packet. Today it is assembled from three code areas:

- `cookimport/bench/upload_bundle_v1_model.py`, `cookimport/bench/upload_bundle_v1_existing_output.py`, and `cookimport/bench/upload_bundle_v1_render.py` build the durable bundle files written beside benchmark outputs.
- `cookimport/bench/oracle_upload.py` consumes that durable bundle, then creates lane-specific Oracle staging under `.oracle_upload_runs/<timestamp>-<profile>/staging/` before launching browser uploads.
- `tests/bench/test_benchmark_oracle_upload.py` and other upload-bundle tests lock in the current file names and path assumptions.

In this plan, “lane packet” means the three files a human or Oracle should read for one review profile: the summary markdown (`overview.md`), the locator/index JSON (`index.json`), and the lane-specific payload JSON (`payload.json`). “Runtime-only launch state” means logs and follow-up status such as `oracle_upload.log`, `oracle_upload_status.json`, and `oracle_auto_followup.json`. Those runtime artifacts may still exist somewhere, but they should stop shaping the main `upload_bundle_v1/` browsing contract.

## Plan of Work

Start from the upload-bundle renderer. Find where the current top-level `upload_bundle_overview.md`, `upload_bundle_index.json`, and `upload_bundle_payload.jsonl` are written, then change that write path so the durable bundle becomes two sibling lane directories: `upload_bundle_v1/quality/` and `upload_bundle_v1/token/`. Each lane directory should write its own `overview.md`, `index.json`, and `payload.json`. Remove or stop emitting the old top-level packet files for the active contract.

Then update `cookimport/bench/oracle_upload.py` so it treats those lane-local files as the source of truth. The Oracle upload path should stop needing an extra timestamped staging folder merely to get a lane-local overview/index/payload. If the upload path still needs temporary runtime files, move those out of the main handoff surface into a separate operational directory that does not make `upload_bundle_v1/` look noisy to a human.

After the code paths are updated, revise the tests to assert the new directory layout and file names, then update the benchmark docs so a reader opening `upload_bundle_v1/` knows exactly what belongs there and why.

## Validation and Acceptance

Acceptance requires all of the following:

1. A freshly rendered benchmark bundle writes `upload_bundle_v1/quality/overview.md`, `upload_bundle_v1/quality/index.json`, `upload_bundle_v1/quality/payload.json`, and the same three files under `upload_bundle_v1/token/`.
2. The old top-level `upload_bundle_overview.md`, `upload_bundle_index.json`, and `upload_bundle_payload.jsonl` are no longer part of the active bundle contract for new runs.
3. Oracle upload reads the lane-local files directly instead of reconstructing a lane packet under `.oracle_upload_runs/<timestamp>-<profile>/staging/`.
4. Runtime-only Oracle launch artifacts no longer clutter the main `upload_bundle_v1/` handoff surface.
5. Focused RecipeImport benchmark/Oracle tests pass in `.venv`.

## Idempotence and Recovery

This refactor changes a generated artifact contract. Old bundles on disk may still have the previous shape, so the implementation should either keep reads tolerant long enough for tests and diagnostics to remain useful or fail with a direct error that names the expected new lane-local file path. Regenerating a benchmark bundle should always rewrite the new `quality/` and `token/` directories cleanly. Runtime-only directories should be safe to recreate on every upload attempt.
