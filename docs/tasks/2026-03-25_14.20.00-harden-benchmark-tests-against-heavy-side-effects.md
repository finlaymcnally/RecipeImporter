# Harden Benchmark Tests Against Heavy Side Effects

User-stated desires for this plan, preserved verbatim:

    "no this isnt your fault. we need to make it much harder to do this. tests shouldn't be able to kneecap my computer. this feels like bad design"

    "is it possible to make the tests also just less structurally able to do these insane operations? i dont want to nerf their speed though. but like damn. also unfortunately AI agents like to ignore warnings. so thats why I ask"

Restated product intent:

Benchmark tests must become structurally unable to trigger repo-scale follow-up work by accident. A routine pytest run should be able to exercise benchmark planning, result shaping, and CLI wiring without rebuilding dashboards from real repo roots, without generating upload bundles from large artifact trees, and without launching background Oracle workflows. Those heavyweight actions must require explicit consent in code and explicit opt-in in tests. Warnings alone are not sufficient because future agents and future contributors will ignore or miss them.

The end state is a simple product rule that holds across the benchmark stack: computation is testable by default, while publication side effects require explicit consent. In production CLI flows, the real follow-up work still happens. In tests, heavy side effects fail closed unless a test explicitly declares that it wants them.

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

Today several benchmark helper tests only mean to verify routing or summary behavior, but they can still run heavyweight post-benchmark follow-up work. The concrete dangerous examples are dashboard refresh, upload-bundle generation, starter-pack generation, and background Oracle launch. Those side effects are expensive because they may traverse repo-scale `data/output` or `data/golden` trees, write many files, or spawn detached workers. The result is that a broad pytest run can saturate disk, CPU, and memory even when the tests themselves are conceptually small.

After this plan lands, a human should be able to run benchmark-related tests without fear that a helper path will silently wander into repo-wide scans or artifact generation. The visible proof is threefold. First, routine benchmark-helper tests run through lightweight injected fakes and no-op result publishers rather than real follow-up work. Second, calling a heavy follow-up helper under pytest without explicit opt-in fails immediately with a clear error naming the blocked side effect. Third, production CLI behavior remains intact: interactive single-book and single-profile benchmark flows still write their artifacts and launch their intended background work outside pytest.

## Decision Log

- Decision: keep a low-level benchmark-side-effect guard even after the larger refactor lands.
  Rationale: explicit dependency injection is the primary design fix, but it only protects code paths that already use the new interfaces. A low-level guard in the heavy helpers is still valuable as a final fail-safe against future ad hoc tests, future helper shortcuts, and future agent mistakes.
  Date/Author: 2026-03-25 / Codex

- Decision: under pytest, the final heavy-side-effect guard should raise a clear error by default rather than silently skipping work.
  Rationale: silent skipping is a useful emergency stopgap, but it can hide missing assertions and create false confidence. The desired long-term contract is that heavyweight side effects require explicit opt-in, so tests that forget that contract should fail loudly and immediately.
  Date/Author: 2026-03-25 / Codex

- Decision: separate computation from publication instead of only adding more monkeypatches.
  Rationale: monkeypatching is easy to miss and leaves ambient behavior in place. Splitting "compute benchmark result" from "publish follow-up artifacts" changes the shape of the code so lightweight tests naturally exercise the cheap path and heavy publication becomes an explicit second step.
  Date/Author: 2026-03-25 / Codex

- Decision: move repo-default roots to CLI-edge code only.
  Rationale: helper functions that silently reach for `DEFAULT_GOLDEN`, `DEFAULT_OUTPUT`, or nested history roots are structurally dangerous in tests. Internal helpers should require explicit roots or explicit pre-resolved targets so they cannot discover repo-scale state by accident.
  Date/Author: 2026-03-25 / Codex

- Decision: the benchmark test capability should use both a pytest marker and an environment-variable escape hatch.
  Rationale: the marker makes intent visible in the test source and integrates with `--strict-markers`. The environment variable allows subprocesses or helper code launched from that test to inherit the permission. Using both is more robust than either mechanism alone.
  Date/Author: 2026-03-25 / Codex

- Decision: this plan covers single-book, single-profile, and all-method benchmark helper flows together.
  Rationale: the same design mistake appears across these flows: mixed computation and publication concerns, plus ambient repo-root defaults. Fixing only single-book would reduce risk but leave the broader benchmark family teaching the wrong architecture.
  Date/Author: 2026-03-25 / Codex

- Decision: keep `golden_root` injectable on the interactive helper signatures but retain a compatibility fallback to `DEFAULT_GOLDEN` for now.
  Rationale: the production CLI edges now pass explicit roots and the dashboard refresh helper requires an explicit golden root, which closes the dangerous ambient-root path for the flows touched here. Keeping a compatibility fallback on the interactive helper signatures avoided a much larger same-turn migration across older direct tests while still making the safer path explicit and testable.
  Date/Author: 2026-03-25 / Codex

## Outcomes & Retrospective

This plan is no longer speculative. The benchmark heavy-helper guard is now fail-fast under pytest, the single-book and single-profile interactive helpers compute first and publish second, and routine benchmark-helper tests now have shared lightweight publishers instead of a growing pile of scattered heavy-helper monkeypatches.

The most important practical outcome is ergonomic: a normal pytest run now fails immediately when it reaches real benchmark publication work without explicit opt-in, instead of quietly scanning repo-scale roots or launching background Oracle work. The new `heavy_side_effects` marker plus `allow_heavy_test_side_effects` fixture make the rare real-heavy-path tests visible in source and inherited by subprocesses.

One compatibility seam remains intentionally small: the interactive helper signatures still accept an omitted `golden_root` and fall back to `DEFAULT_GOLDEN`, even though the production CLI edges now pass the root explicitly and the dashboard refresh helper itself requires an explicit `golden_root`. That tradeoff kept the change atomic enough to land with focused proof while still moving the dangerous ambient-root behavior out of the production benchmark entrypoints that triggered this plan.

## Context and Orientation

This section defines the terms and files needed to implement the plan from scratch.

A "heavy benchmark side effect" means any follow-up action that can traverse large repo-local artifact trees, write many files, or spawn background processes. In the current codebase the main examples are dashboard refresh, upload-bundle generation, starter-pack generation, and background Oracle launch. These are heavy because they can scan `data/output` or `data/golden`, walk many nested directories, or detach long-lived background work.

A "publication side effect" means the post-computation work that turns a benchmark result into review artifacts, dashboards, or external-review handoff. In this repo, publication side effects currently live in helpers such as `_write_benchmark_upload_bundle(...)`, `_write_single_book_starter_pack(...)`, `_refresh_dashboard_after_history_write(...)`, and `_start_benchmark_bundle_oracle_upload_background(...)`.

A "computation phase" means the part of a benchmark flow that resolves inputs, runs predictions or evaluations, computes metrics, and assembles a result summary. A computation phase may still write the minimal per-run benchmark artifacts that define the benchmark result itself, but it must not implicitly trigger heavy publication side effects.

A "publication phase" means the explicit second step that takes a completed benchmark result and decides which optional heavy side effects to perform. This includes writing `upload_bundle_v1`, starter-pack exports, dashboard refresh, and Oracle background launch.

A "capability marker" means a pytest marker that declares a test intends to exercise real heavy side effects. In this plan the concrete marker name is `heavy_side_effects`. Because `pytest.ini` uses `--strict-markers`, this marker becomes a checked contract rather than an informal convention.

The current single-book benchmark helper lives in [bench_single_book.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/bench_single_book.py). `_interactive_single_book_benchmark(...)` both computes benchmark results and publishes follow-up artifacts. `_write_single_book_comparison_artifacts(...)` also mixes payload generation with optional starter-pack follow-up work.

The current single-profile benchmark helper lives in [bench_single_profile.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/bench_single_profile.py). `_interactive_single_profile_all_matched_benchmark(...)` resolves targets from repo defaults, executes per-book variants, writes upload bundles, launches Oracle, and refreshes dashboards in one broad coordinator.

The current all-method interactive helper lives in [bench_all_method.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/bench_all_method.py). Even though its computation shape is different, it still mixes broad discovery, benchmark execution, and publication concerns in one public helper.

The current heavy publication helpers live in [bench_oracle.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/bench_oracle.py) and [dashboard.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/dashboard.py). They are intentionally production-oriented, and they currently rely on ambient defaults and large helper scripts such as [benchmark_cutdown_for_external_ai.py](/home/mcnal/projects/recipeimport/scripts/benchmark_cutdown_for_external_ai.py).

The current pytest control surfaces live in [pytest.ini](/home/mcnal/projects/recipeimport/pytest.ini), [tests/conftest.py](/home/mcnal/projects/recipeimport/tests/conftest.py), and the benchmark-helper support module in [benchmark_helper_support.py](/home/mcnal/projects/recipeimport/tests/labelstudio/benchmark_helper_support.py). Those files control markers, compact output, and test-wide stubs.

The temporary stopgap guard now lives in [test_safety.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/test_safety.py). It should be treated as a transitional safety layer, not the final architecture. The full refactor in this plan will build around it, tighten its behavior, and reduce how often it needs to save the test suite from ambient design mistakes.

## Milestones

### Milestone 1: Codify the heavy-side-effect capability boundary

At the end of this milestone, the repository has one authoritative definition of what counts as a heavy benchmark side effect, one pytest marker that grants permission to exercise those paths, and one fail-fast test-runtime guard that blocks those paths by default. This milestone is about policy and safety rails, not about redesigning the benchmark coordinators yet.

Implement this in the current low-level helpers first. Expand [test_safety.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/test_safety.py) from a silent-stopgap helper into an explicit policy module. Add functions that can answer three questions: are we under pytest, has the current run explicitly allowed heavy side effects, and what error message should be raised when a blocked helper is called. Replace the current silent returns in [dashboard.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/dashboard.py) and [bench_oracle.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/bench_oracle.py) with a fail-fast path that raises a clear exception naming the blocked helper and the required opt-in mechanism.

Then update [pytest.ini](/home/mcnal/projects/recipeimport/pytest.ini) and [tests/conftest.py](/home/mcnal/projects/recipeimport/tests/conftest.py). Declare a new `heavy_side_effects` marker. Add an autouse fixture or session-level hook that sets the default test env to "heavy side effects forbidden." Add a targeted fixture such as `allow_heavy_test_side_effects` that sets `COOKIMPORT_ALLOW_HEAVY_TEST_SIDE_EFFECTS=1` for the current test and any subprocesses it spawns. Add a guard that fails early if a test uses the marker but forgets the fixture, or uses the fixture without the marker. The exact mechanics can vary, but the end state must make capability intent explicit in both source and runtime.

Acceptance for this milestone is simple and observable. Calling `_refresh_dashboard_after_history_write(...)` or `_write_benchmark_upload_bundle(...)` from a normal pytest test should fail immediately with a capability error. A dedicated test marked `heavy_side_effects` and using the opt-in fixture should reach the real helper body.

### Milestone 2: Split single-book benchmark computation from publication

At the end of this milestone, single-book benchmark code computes a result object first and publishes optional follow-up work second. The goal is that most tests can stop at the computation result and never touch the heavy publication surface.

Create a small result model in a new module under `cookimport/cli_support/`, for example `bench_results.py` or `bench_followups.py`. The exact file name is less important than the responsibility: represent the computed outcome of a single-book benchmark session without performing publication side effects. That result should include the session roots, per-variant statuses, comparison-artifact eligibility, source path, split-cache summary, and the minimal metadata needed for later publication.

Refactor `_interactive_single_book_benchmark(...)` in [bench_single_book.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/bench_single_book.py) into at least two internal steps. The first step resolves inputs, runs the variant benchmarks, and returns a `SingleBookBenchmarkComputationResult` (or similarly named type). The second step consumes that result and performs optional publication through an explicit publisher object or callback bundle. `_write_single_book_comparison_artifacts(...)` should also stop triggering starter-pack work directly. It should write the comparison payload only. Starter-pack publishing becomes part of the explicit publication phase.

Add a production publisher implementation that preserves current behavior outside pytest: write comparison JSON/Markdown, write starter pack when requested, write upload bundle when there was at least one successful run, launch Oracle when a bundle exists, and refresh the dashboard once the batch is done. The important part is not the class structure; it is that the heavy follow-up work is now provided explicitly rather than being ambient in the computation function.

Acceptance for this milestone means a unit test can call the computation phase, inspect the returned result object, and prove comparison or summary eligibility without writing upload bundles or refreshing dashboards. Production CLI behavior for interactive single-book benchmark remains unchanged when the real publisher is injected.

### Milestone 3: Apply the same split to single-profile and all-method benchmark helpers

At the end of this milestone, single-profile and all-method interactive benchmark flows follow the same design rule as single-book: compute first, publish second, and wire heavy publication explicitly.

Refactor `_interactive_single_profile_all_matched_benchmark(...)` in [bench_single_profile.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/bench_single_profile.py) so that the loop over matched books and variants returns a structured batch result before any upload-bundle, Oracle, or dashboard work runs. The top-level group upload bundle and the optional per-book bundles become publication outputs driven by a publisher object rather than hard-coded helper calls.

Do the same for `_interactive_all_method_benchmark(...)` in [bench_all_method.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/bench_all_method.py) where practical. The all-method flow has more scheduler and planning logic, so the milestone does not require rewriting the scheduler. It does require that target discovery, variant computation, and heavyweight publication are separate enough that tests can stop after computation. If the existing coordinator is too large to split in one pass, introduce a small intermediate result layer first and migrate the publication calls behind that layer.

Acceptance for this milestone is behavioral. A routine test for single-profile or all-method benchmark planning should be able to execute the compute path without risking upload-bundle generation or dashboard refresh. A production run should still be able to call the real publisher and get the same external outputs as before.

### Milestone 4: Move repo-default roots and discovery behavior to CLI edges

At the end of this milestone, internal benchmark helpers no longer discover repo-scale roots by reaching for `DEFAULT_GOLDEN`, `DEFAULT_OUTPUT`, or other ambient defaults. Only CLI-edge code resolves those defaults, then passes explicit roots or explicit target lists into lower-level helpers.

Audit the current benchmark entrypoints and discovery helpers. The dangerous examples are calls such as `_resolve_all_method_targets(DEFAULT_GOLDEN)` and helper paths that infer dashboard roots or benchmark roots from runtime defaults. Change the public CLI commands in `cookimport/cli.py`, `cookimport/cli_commands/labelstudio.py`, and adjacent CLI-support entrypoints to resolve defaults once, at the edge, before handing off to the internal benchmark functions.

Then change the internal helper signatures. `_interactive_single_profile_all_matched_benchmark(...)` should accept explicit matched targets or an explicit discovery context rather than reaching straight into repo defaults. `_interactive_all_method_benchmark(...)` should accept explicit roots or explicit resolved targets. The dashboard refresh helper should require an explicit `golden_root` from the caller rather than silently defaulting to `runtime.DEFAULT_GOLDEN` when invoked from lower-level benchmark code. The goal is not to remove defaults from CLI commands. It is to remove defaults from testable internals.

Acceptance for this milestone means a test can fully exercise a benchmark helper by constructing all needed roots and targets under `tmp_path`, with no internal call path able to fall back to the real repo tree.

### Milestone 5: Replace monkeypatch-heavy benchmark-helper tests with explicit lightweight publishers

At the end of this milestone, the benchmark-helper tests stop relying on patching scattered low-level helpers and instead use explicit lightweight publishers or fake follow-up bundles that match the new production interfaces.

Update [tests/labelstudio/benchmark_helper_support.py](/home/mcnal/projects/recipeimport/tests/labelstudio/benchmark_helper_support.py) to construct a standard lightweight publisher/follow-up bundle for routine benchmark-helper tests. This fake publisher should be shape-compatible with production publication but should only write tiny deterministic files under `tmp_path` when a test needs a returned path. Then migrate the single-book, single-profile, and smoke benchmark-helper tests to use that shared fake publisher rather than patching `_write_benchmark_upload_bundle(...)`, `_refresh_dashboard_after_history_write(...)`, and `_start_benchmark_bundle_oracle_upload_background(...)` directly in each test. Some tests will still patch specific seams when they are asserting a narrow handoff contract, but that should become the exception rather than the default style.

Add one dedicated small test file for the heavy publication capability boundary itself. That file should contain the positive and negative contracts: blocked by default under pytest, allowed only with marker plus fixture, and child subprocesses inherit the permission. That keeps the safety policy visible and small.

Acceptance for this milestone means the benchmark-helper tests express their intent through fake publishers and explicit capability tests, not through a growing pile of scattered monkeypatches.

## Validation and Acceptance

The change is correct only if all of the following are true.

Pytest-capability acceptance means that a normal benchmark-related test cannot accidentally call a heavy publication helper. If a test tries, it fails immediately with a clear message naming the blocked side effect and the required opt-in. A test marked `heavy_side_effects` and using the matching fixture/env opt-in can reach the real helper body.

Single-book acceptance means that a test can exercise the main single-book benchmark computation path and inspect its result without generating `upload_bundle_v1`, starter-pack output, dashboard rebuilds, or Oracle background launch. Production CLI usage still performs those actions through the real publisher when appropriate.

Single-profile and all-method acceptance mean the same separation holds at batch scope. Tests can stop at computed batch results, while production flows can explicitly publish group upload bundles, dashboard updates, and Oracle launches afterward.

Root-resolution acceptance means that lower-level benchmark helpers no longer discover repo-scale roots or matched targets by reaching for ambient defaults. A test that passes explicit `tmp_path` roots and explicit targets cannot accidentally drift into the real repo tree.

Documentation acceptance means [docs/07-bench/07-bench_README.md](/home/mcnal/projects/recipeimport/docs/07-bench/07-bench_README.md), [docs/08-analytics/08-analytics_readme.md](/home/mcnal/projects/recipeimport/docs/08-analytics/08-analytics_readme.md), [docs/12-testing/12-testing_README.md](/home/mcnal/projects/recipeimport/docs/12-testing/12-testing_README.md), and [tests/labelstudio/README.md](/home/mcnal/projects/recipeimport/tests/labelstudio/README.md) all teach the same policy: computation is cheap-by-default in tests; heavy publication side effects are explicit and opt-in.

The most important real-world acceptance is ergonomic rather than cosmetic: an AI agent that ignores warnings and runs the wrong benchmark-helper test file should still hit a fast explicit failure instead of silently pegging the workstation.

## Idempotence and Recovery

This refactor should be implemented incrementally and is safe to retry if each milestone preserves one coherent boundary.

Milestone 1 is intentionally additive and should land before the broader benchmark-helper refactors. If later milestones stall, the fail-fast capability guard still provides protection. That is a safe stopping point.

The computation-versus-publication split should also be additive. Introduce new result objects and publisher interfaces first, wire production callers to them second, and only then remove old ambient publication calls. If a midway state leaves both the old and new paths alive, tests must prove which one is authoritative before any cleanup is merged.

If a refactor accidentally breaks production publication behavior, the quickest safe rollback is to keep the new computation result type but temporarily rewire the production entrypoint to the old publisher path while preserving the pytest capability guard. Do not remove the guard to "make tests pass." The guard is the safety property being introduced.

## Artifacts and Notes

The central artifact for this plan is not one benchmark payload. It is the new shape of the code and tests. The most useful proof snippets to record during implementation are:
