## Problem
CLI/report token usage for freeform prelabel only showed input/cached/output totals. Users asked to see reasoning-token counts in the same run summary.

## Acceptance Criteria
- `prelabel_report.json` token usage includes `reasoning_tokens` aggregate.
- CLI completion summary prints `reasoning=<count>` in `Prelabel token usage: ...`.
- Parser remains backward compatible when Codex usage payload omits reasoning fields (report `0`).
- Tests cover both no-reasoning and nested-reasoning usage payload shapes.

## Verification
- `source .venv/bin/activate && pytest tests/labelstudio/test_labelstudio_prelabel.py::test_codex_provider_tracks_usage_from_json_events tests/labelstudio/test_labelstudio_prelabel.py::test_codex_provider_tracks_reasoning_tokens_from_nested_usage tests/labelstudio/test_labelstudio_benchmark_helpers.py::test_labelstudio_import_prints_prelabel_token_usage_with_reasoning`

## Evidence
- Added assertions for `usage["reasoning_tokens"]` in provider tests.
- Added CLI summary assertion for `reasoning=...` in import completion output.

## Constraints / Gotchas
- Codex JSON events may not emit reasoning fields on all versions/models; parser must gracefully default to `0`.
- Keep usage aggregation thread-safe because one provider instance is shared by parallel prelabel workers.

## Decision
- Implemented best-effort reasoning extraction across likely usage payload shapes (`reasoning_tokens`, nested `output_tokens_details`, and aliases), while preserving existing usage keys and summary format.

## Rollback
- Revert edits in `cookimport/labelstudio/prelabel.py`, `cookimport/cli.py`, and corresponding tests/docs to restore prior 3-field token reporting.
