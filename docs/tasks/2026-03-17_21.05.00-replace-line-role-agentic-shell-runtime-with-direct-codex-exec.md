# Replace Line-Role Agentic Shell Runtime With Direct Codex Exec

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

After this change, the canonical line-role LLM pass still reads every target line, but it does so the simple way: one direct model request per shard, with the full shard prompt sent once. The current runtime inflates token use because the model is launched as an agent, then shells out to `cat` / `awk` / `sed` the shard file and replays those outputs inside the conversation. The clean target is to delete that transport for line-role and replace it with one `codex exec --output-schema` call per shard.

From an operator point of view, `line_role_pipeline=codex-line-role-shard-v1` should keep the same user-facing meaning, but the cost profile should change dramatically. A `5`-prompt line-role run on `saltfatacidheatcutdown` should no longer produce multi-million input-token telemetry driven by shell transcript replay. Instead, line-role cost should track the visible shard prompt size plus modest runtime overhead.

Stated in plain operator language: if the prompt target count is `5`, RecipeImport should split the book into `5` contiguous line-role shards, send each shard to Codex exactly once, and get back one structured set of improved labels for the owned `atomic_index` rows in that shard. The model should not have to browse files, inspect the workspace, or rediscover the shard contents through tool calls.

## Surprises & Discoveries

- Observation: the visible line-role prompt is not the main problem anymore.
  Evidence: on the `2026-03-17_16.37.00` run, the source canonical text was about `88,172` tokens and the full visible line-role prompt text across five shards was about `102,146` tokens, only about `+15.8%`.

- Observation: the active line-role pipeline wrapper is tiny; the real cost comes from file-reading subturns after launch.
  Evidence: the wrapper prompt in [worker-001 trace](/home/mcnal/projects/recipeimport/data/output/2026-03-17_16.37.00/line-role-pipeline/runtime/workers/worker-001/out/.codex-farm-traces/6760aff4a1424314b67fe22c4db5dd8f/eb2265cedeaa4bbc992bba6898480095.trace.json) is only about `720` chars, but the same trace shows a full `cat` of the shard file plus extra `awk` reads.

- Observation: path handoff is not opaque workspace context under the current transport.
  Evidence: the line-role trace contains `command_execution` items whose `aggregated_output` fields contain the full shard prompt body and later slices of the same body.

- Observation: retries and shell experimentation make the blow-up worse, not just one clean file read.
  Evidence: [worker-003 trace](/home/mcnal/projects/recipeimport/data/output/2026-03-17_16.37.00/line-role-pipeline/runtime/workers/worker-003/out/.codex-farm-traces/e91986e272d0413fab3d133354fe1bd5/b7864f804b3b4c6c92054f208fae8744.trace.json) shows `16` shell attempts, `5` failures, and over `1,079,725` input tokens for one shard.

- Observation: the repo already exposes the right low-level primitive for the replacement target.
  Evidence: `codex exec --help` supports stdin prompts, `--json`, and `--output-schema`, which is exactly the “one prompt in, one structured response out” contract line-role needs.

- Observation: line-role could reuse the same direct runner that knowledge already uses, but it still needed line-role-specific telemetry bridging.
  Evidence: `cookimport/llm/codex_exec_runner.py` already returned structured usage and event rows, but `line-role-pipeline/telemetry_summary.json` and prompt export readers still expected per-batch metadata such as `prompt_index`, `requested_atomic_indices`, and one attempt payload per shard.

  Rationale: the problem was traced concretely on line-role, and line-role already has the clearest prompt builder and validation seam. The replacement pattern can later be extended to other surfaces once one direct-call runtime is proven.
  Date/Author: 2026-03-17 / Codex

- Decision: delete line-role’s `codex-farm process` path-mode transport instead of adding a second runtime flag or compatibility mode.
  Rationale: the current path-mode runtime is the source of the token explosion. Keeping it as a side path would preserve dead code and confusion around which transport is “real.”
  Date/Author: 2026-03-17 / Codex

- Decision: the clean target is one direct `codex exec --json --output-schema <schema> -` invocation per shard, with the full shard prompt sent on stdin.
  Rationale: this preserves the user’s intended mental model: one shard equals one actual prompt. It also removes the need for Codex to read the shard file with shell tools.
  Date/Author: 2026-03-17 / Codex

- Decision: line-role should remain a pure “read this shard once and relabel it” surface, not an open-ended agent task.
  Rationale: the line-role job already has a complete, deterministic input bundle. Agentic browsing adds cost and variance without adding useful capability for this phase.
  Date/Author: 2026-03-17 / Codex

- Decision: keep the existing public line-role setting name `codex-line-role-shard-v1`.
  Rationale: operators should not have to learn a new setting for what is conceptually the same surface. The implementation changes underneath; the public feature name does not.
  Date/Author: 2026-03-17 / Codex

- Decision: keep worker folders and prompt artifacts for observability, but stop using worker input files as model evidence.
  Rationale: artifacts are still valuable for debugging and review. The problem is not the existence of files on disk; the problem is making the model read them through shell commands.
  Date/Author: 2026-03-17 / Codex

- Decision: keep the line-role pack schema and pack inventory files for compatibility, but stop treating the pack prompt wrapper as the live transport contract.
  Rationale: the live runtime no longer needs the path-mode wrapper, but pack inventory/tests and schema resolution still reference the line-role pack. Retaining the pack metadata avoids widening this transport refactor into unrelated preview/tooling churn.

## Context and Orientation

The current line-role LLM path starts in `cookimport/parsing/canonical_line_roles.py`. `_build_line_role_shard_plans(...)` builds contiguous shards and produces `prompt_text` with `build_canonical_line_role_prompt(...)` from `cookimport/llm/canonical_line_role_prompt.py`. `_run_line_role_shard_runtime(...)` then sends those shards into `run_phase_workers_v1(...)` in `cookimport/llm/phase_worker_runtime.py`.

`run_phase_workers_v1(...)` creates `workers/<id>/in/*.json`, `workers/<id>/out/*.json`, and `workers/<id>/logs/`, then calls `runner.run_pipeline(...)`. For live runs, that runner is `SubprocessCodexFarmRunner` from `cookimport/llm/codex_farm_runner.py`. That runner currently launches `codex-farm process`, resolves the line-role pipeline asset from `llm_pipelines/pipelines/line-role.canonical.v1.json`, and uses the prompt wrapper in `llm_pipelines/prompts/line-role.canonical.v1.prompt.md`.

That wrapper currently says “Read the task file already placed in the worker folder at `{{INPUT_PATH}}`.” In practice, Codex does not treat that as an invisible local file. It launches shell commands to read the file, and those shell outputs become transcript content. The transcript then grows across subturns, which is why cached-input tokens dominate line-role telemetry.

For this plan, “direct Codex exec” means a non-interactive Codex CLI call that reads one prompt from stdin and writes structured JSON output back once. `codex exec --help` shows the critical flags already exist: `--json`, `--output-schema`, stdin prompt input via `-`, and an explicit working directory with `--cd`.

For this plan, “one prompt per shard” is literal. If there are `5` line-role shards, there should be `5` real model requests. The model may spend tokens on the prompt text and its answer, but it must not spend extra tokens on shell-based file discovery inside the turn.

The public line-role surface must stay the same: `line_role_pipeline=codex-line-role-shard-v1`. What changes is the transport under that surface. After this refactor, line-role should no longer depend on the line-role CodexFarm pipeline wrapper asset or on `phase_worker_runtime.py` for live execution.

## Plan of Work

The refactor should proceed by replacing the live line-role executor, not by stacking a new option beside the old one.

First, add a repo-owned direct runner for one structured Codex call. The best location is a new module under `cookimport/llm/`, for example `cookimport/llm/codex_exec_runner.py`. This module should define a small result object that captures the command used, the final response text, parsed usage totals from `turn.completed`, any `turn.failed` detail, and a trace/event log path when available. The runner should build a `codex exec` command with `--json`, `--output-schema`, `--model` when configured, `-c model_reasoning_effort="..."` when configured, and `-` so the prompt is sent once on stdin. It should not rely on CodexFarm pipeline packs or `INPUT_PATH`.

Second, change `cookimport/parsing/canonical_line_roles.py` so `_run_line_role_shard_runtime(...)` executes shards through the new direct runner instead of `run_phase_workers_v1(...)`. The shard planning logic can stay. The prompt artifact state can stay. Validation can stay. The change is specifically that each `_LineRoleShardPlan.prompt_text` becomes the actual stdin prompt body for one direct model call. Worker folders should still exist under `line-role-pipeline/runtime/workers/worker-XXX/`, but `in/*.json` should become debug artifacts only, not model evidence. The line-role runtime should write the actual prompt text used, the raw JSONL event stream, the last agent message, parsed usage, and the validated proposal JSON under the worker folder so prompt/debug readers still have concrete artifacts.

Third, delete line-role-only `codex-farm process` compatibility code from the live runtime. This includes removing the live line-role path through `SubprocessCodexFarmRunner.run_pipeline(...)` and removing line-role’s dependency on `phase_worker_runtime.py`. Keep `llm_pipelines/schemas/line-role.canonical.v1.output.schema.json` as the structured-output contract. The pack inventory files may remain for compatibility/tooling as long as the live runtime no longer depends on them for prompt transport.

Fourth, update observability and prompt-cost readers to match the new transport. `cookimport/llm/prompt_artifacts.py`, `cookimport/llm/prompt_preview.py`, and any line-role budget readers must stop assuming line-role telemetry comes from CodexFarm `process` payloads. The new direct runner should emit one repo-owned manifest row per shard with `tokens_input`, `tokens_cached_input`, `tokens_output`, command metadata, and any local raw event log path. Prompt preview for line-role should remain easy because it already rebuilds the real shard prompt text from `build_canonical_line_role_prompt(...)`.

Fifth, remove dead settings or audit fields that only existed for the old transport. In particular, line-role `max_turns_per_shard` no longer makes semantic sense if one shard is always one direct `codex exec` call. Audit fields and docs that describe line-role as `classic_task_farm_v1` or path-mode `INPUT_PATH` should be updated or deleted.

## Validation and Acceptance

Acceptance is behavioral, not just structural.

On the real `saltfatacidheatcutdown` line-role benchmark with `5` prompts:

- line-role still processes all atomic lines and produces validated proposal rows for every owned `atomic_index`
- each shard corresponds to exactly one direct `codex exec` invocation
- the line-role raw event stream contains no `command_execution` items and no evidence of shell reads of shard files
- line-role actual cost reporting comes from the new runner’s usage telemetry, not from CodexFarm `process` payloads
- line-role input-token usage falls from the old `~1.8M` range to a small multiple of visible prompt size; specifically, total line-role `tokens_input` should be less than `3x` the visible prompt token count for that run

The regression suite must also prove:

- fake or mocked line-role runs still validate exact owned-row coverage
- prompt artifact export still captures line-role prompts and responses from the new artifact shape
- prompt preview still reconstructs the same shard prompt text locally
- line-role settings and interactive CLI flows still work without introducing a new public pipeline name

## Idempotence and Recovery

This refactor is safe to stage incrementally as long as the public line-role setting remains stable. The key recovery rule is to avoid shipping a mixed transport where some line-role calls still use `codex-farm process` and others use direct exec. Keep the old path in place only until the replacement path passes focused tests; then delete it in the same branch before final validation.

Worker artifact folders are safe to regenerate. The direct runner should overwrite `prompt.txt`, `events.jsonl`, `last_message.json`, `usage.json`, `proposal.json`, and `status.json` for each shard run. If a run fails partway through, rerunning the benchmark should rebuild those artifacts cleanly.

  - prompt text
  - output schema path
  - working directory
  - model override
  - reasoning effort override

  - final response text
  - parsed usage totals
