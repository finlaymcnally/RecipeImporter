# Make Stage 7 Review Routing Only And Knowledge The Sole Outside-Recipe Authority

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the outside-recipe pipeline will have one clear owner for semantic `KNOWLEDGE` versus `OTHER`: the optional knowledge LLM review stage. The earlier line-role pass will still do valuable work, but only to find recipe structure and to exclude overwhelming obvious-junk non-recipe material such as tables of contents, endorsements, publishing matter, and navigation. Stage 7 will stop acting like a second semantic judge and will instead become a deterministic review-routing builder that says which outside-recipe blocks are excluded immediately and which blocks must be sent to knowledge review.

The user-visible effect is that recipe detection stays early and cheap, obvious junk still avoids token spend, and review-eligible outside-recipe headings and prose are no longer vulnerable to being rewritten by a deterministic or semi-deterministic binary authority layer after line-role. A human will be able to see the new ownership split by opening `08_nonrecipe_spans.json`, `08_nonrecipe_review_exclusions.jsonl`, `09_knowledge_outputs.json`, and the final `line_role_predictions.jsonl` or `.bench/*/stage_block_predictions.json` artifacts and confirming that only the knowledge stage decides review-eligible `KNOWLEDGE` versus `OTHER`.

## Context and Orientation

The relevant pipeline starts in `cookimport/parsing/label_source_of_truth.py`. That module atomizes full blocks into atomic lines, asks `cookimport/parsing/canonical_line_roles.py` for one final line-role label per atomic line, promotes those labels back to authoritative block labels, and groups accepted recipe spans. Inside this stage, “line-role” means the label set used for freeform and benchmark-facing surfaces such as `RECIPE_TITLE`, `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `RECIPE_NOTES`, `RECIPE_VARIANT`, `HOWTO_SECTION`, `YIELD_LINE`, `TIME_LINE`, `KNOWLEDGE`, and `OTHER`.

The key change in this plan is that outside-recipe `KNOWLEDGE` is no longer supposed to be line-role-owned final authority for review-eligible text. Line-role should do two jobs only. First, it should find recipe-local structure. Second, for text outside accepted recipe spans, it should identify obviously useless material that should not consume knowledge-review tokens. In this repository those obvious-junk examples include front matter, table of contents entries, endorsements, marketing copy, copyright/legal boilerplate, publisher metadata, page-number-only rows, and similar navigation or publishing surfaces. Ambiguous prose, essays, lesson headings, definitions, and cookbook teaching text must not be finalized here. They should remain review-eligible.

Stage 7 lives in `cookimport/staging/nonrecipe_stage.py`. Today it consumes the authoritative block labels plus accepted recipe spans and assigns every outside-recipe block a binary category in `block_category_by_index`: `knowledge` or `other`. It also records review-eligible and review-excluded subsets for the optional knowledge stage. The problem is that this file is still doing more than routing. Its seed map is treated as an actual semantic answer, and later code assumes that answer can be projected back onto final labels.

The optional knowledge review stage lives in `cookimport/llm/codex_farm_knowledge_orchestrator.py` and related job/planning files. It receives Stage 7 review-eligible spans, asks the LLM to judge whether those blocks are useful general cooking knowledge, and can refine final outside-recipe ownership. This is already the correct place for the subtle semantic boundary. The repo already writes rich artifacts for this stage, including `knowledge/in/*.json`, `knowledge_manifest.json`, `stage_status.json`, `task_status.jsonl`, `promotion_report.json`, `08_nonrecipe_spans.json`, and `09_knowledge_outputs.json`.

Final non-recipe projection currently happens in `cookimport/labelstudio/ingest.py`, especially `_apply_nonrecipe_authority_to_predictions(...)`. That helper can overwrite outside-recipe `OTHER` and `KNOWLEDGE` predictions using the final `NonRecipeStageResult.block_category_by_index`. This was useful when Stage 7 was intended to be real final authority, but it is wrong for the new contract if the category came only from deterministic seed routing or from an incomplete knowledge-review run.

Benchmark scoring depends on `cookimport/staging/stage_block_predictions.py`. That module currently adds `KNOWLEDGE` labels directly from `nonrecipe_stage_result.block_category_by_index`. Under the new design, benchmark-scored `KNOWLEDGE` labels must come only from final reviewed authority, never from deterministic Stage 7 seed guesses for review-eligible blocks.

The term “review eligible” in this plan means: an outside-recipe block that is not obvious junk and therefore must be offered to the knowledge LLM before the repo claims final `KNOWLEDGE` or final `OTHER`. The term “review excluded” means: an outside-recipe block that is so obviously not reusable cooking knowledge that it can be kept as final `OTHER` without spending knowledge-review tokens. These terms are routing terms. They are not new public user-facing labels.

The term “seed” in this plan means the deterministic routing map produced before knowledge review. After this refactor, the seed should answer only routing questions for the review-eligible subset, not the final semantic question. The term “final authority” means the categories that downstream scoring and projection are allowed to trust as the true outside-recipe `KNOWLEDGE|OTHER` answer.

## Plan of Work

The first milestone is to redefine the contract in code before changing runtime behavior. Start in `cookimport/staging/nonrecipe_stage.py` and replace the assumption that every outside-recipe block has an immediate deterministic semantic answer. The new deterministic Stage 7 result should have enough information to express three states internally: excluded final `other`, review-eligible pending review, and reviewed final `knowledge|other`. The external surface can still collapse to `knowledge|other` once final authority exists, but the internal model must stop pretending that seed routing for a review-eligible block is the same thing as a semantic judgment.

A practical way to do this without a large taxonomy explosion is to preserve `block_category_by_index` as the final-authority map only, and introduce a separate explicit review-routing map for the seed phase. That routing map can record whether each outside-recipe block is `excluded_other` or `review_eligible`. The seed routing object should also keep the current review-exclusion reasons and the preview text already written to artifacts. For backward safety during migration, if the knowledge stage is disabled, Stage 7 still needs a conservative final-authority story: excluded rows become final `other`, and review-eligible rows should remain unresolved until a deterministic fallback policy is explicitly chosen. This plan chooses the safer fallback: unresolved review-eligible rows remain final `other` only in the final output artifact, but they must be marked in metadata as unreviewed seed-kept rather than reviewed semantic `other`.

The second milestone is to update the knowledge stage so it explicitly consumes Stage 7 review-routing state rather than a fake binary seed authority. `cookimport/llm/codex_farm_knowledge_jobs.py` and `cookimport/llm/codex_farm_knowledge_orchestrator.py` should read only the review-eligible span set when building `knowledge/in/*.json`. When the stage completes successfully, it should write the final semantic decisions back into the final-authority map. When the stage does not run, is interrupted, or semantically rejects a shard, the final-authority map should preserve excluded `other` blocks as authoritative and mark the review-eligible remainder as “seed kept / unreviewed” in the refinement report instead of pretending those rows were semantically decided.

The third milestone is to stop downstream projections from treating deterministic seed routing as final semantic truth. In `cookimport/labelstudio/ingest.py`, `_apply_nonrecipe_authority_to_predictions(...)` must stop rewriting review-eligible outside-recipe rows from a deterministic or unresolved seed category. It should project only true final-reviewed authority, plus excluded final `other` rows. Review-eligible unresolved rows should keep their line-role-side provisional label unless the new design explicitly says otherwise. This is the seam that will prevent correct raw line-role headings from being turned back into `OTHER` by a seed routing decision.

The fourth milestone is benchmark alignment. `cookimport/staging/stage_block_predictions.py` must stop deriving benchmark `KNOWLEDGE` labels from a seed map that still includes unresolved review-eligible rows. Benchmark-scored `KNOWLEDGE` should come only from final reviewed authority. Review-eligible unresolved rows should score as `OTHER` if the knowledge stage did not promote them, but the notes and artifact metadata must say they were unresolved or unreviewed, not semantically classified by deterministic Stage 7.

The fifth milestone is artifact clarity and docs. `08_nonrecipe_spans.json` and `09_knowledge_outputs.json` must become novice-readable explanations of the new ownership split. A human should be able to open those files and see: how many blocks were excluded upstream, how many blocks were sent to review, how many decisions the knowledge stage applied, how many review-eligible rows stayed unreviewed, and whether the current output reflects final authority or seed-kept fallback. Update `docs/05-staging/05-staging_readme.md`, `docs/10-llm/nonrecipe_knowledge_review.md`, and `docs/04-parsing/04-parsing_readme.md` after the code change so the current docs describe the new contract accurately.

The sixth milestone is regression coverage. The tests must cover positive and negative cases for all three boundaries: line-role exclusion routing, Stage 7 review routing, and final authority projection. Specifically, add tests proving that obvious junk rows are excluded from knowledge review, that review-eligible headings such as `Balancing Fat` are not deterministically finalized as `other`, that interrupted or missing knowledge-review outputs do not silently turn into “reviewed `other`”, and that final projection only mutates rows when real final authority exists.

## Validation and Acceptance

This plan is acceptable only if all of the following are true.

The line-role stage still performs recipe detection and obvious-junk outside-recipe exclusion, but review-eligible outside-recipe semantic `KNOWLEDGE|OTHER` is no longer finalized there.

Stage 7 no longer pretends to have deterministic semantic answers for review-eligible outside-recipe text. Its seed result must be auditable as routing state rather than misdescribed as final reviewed authority.

The optional knowledge stage becomes the sole semantic authority for review-eligible outside-recipe `KNOWLEDGE|OTHER`. When it runs successfully, its decisions appear as final authority in `09_knowledge_outputs.json` and any downstream projection that depends on `NonRecipeStageResult`.

If the knowledge stage does not run, is interrupted, or leaves a subset unresolved, the artifacts must show that explicitly. Those rows may remain final `OTHER` in practical output surfaces, but they must not be represented as deterministically reviewed semantic `other`.

`cookimport/labelstudio/ingest.py` must only overwrite outside-recipe line-role `OTHER|KNOWLEDGE` rows when true final non-recipe authority exists. Seed routing alone is insufficient to trigger an overwrite for review-eligible rows.

`cookimport/staging/stage_block_predictions.py` must only emit benchmark `KNOWLEDGE` from final reviewed authority. It must not score deterministic review-eligible seed routing as `KNOWLEDGE`.

`08_nonrecipe_spans.json` and `09_knowledge_outputs.json` must let a novice explain why a given outside-recipe block was excluded, reviewed, changed, or left unreviewed without reading code.

Targeted tests must prove all of the following:

- obvious junk outside recipe is excluded before knowledge review
- review-eligible headings remain review-eligible and are not deterministically finalized as `other`
- final projection ignores unresolved seed routing for review-eligible rows
- final projection still respects excluded final `other` rows
- final projection respects actual knowledge-stage final authority when it exists

Acceptance should be demonstrated by passing targeted tests plus a broad staging/LLM/parsing run. If the user later approves a benchmark rerun, acceptance also includes artifact-level proof that the old `nonrecipe_authority:other` false rewrites are gone for review-eligible rows.

## Idempotence and Recovery

This work should be implemented additively. During migration, prefer adding explicit routing and authority fields before deleting older assumptions. That keeps artifact diffs intelligible and lowers the risk of silently changing benchmark semantics.

If any downstream surface breaks because it still expects `block_category_by_index` to be a total deterministic semantic answer, the safe recovery path is to keep the new routing fields and make that surface consume reviewed final authority plus explicit seed-kept fallback metadata instead of reverting to implicit overloading.

If the new contract causes too much churn in benchmark helpers, keep the old fields temporarily but rename or document them clearly as seed routing rather than final authority. Do not reintroduce semantic overlap just to preserve a smaller diff.

If the knowledge stage runtime fails during development, the safe fallback is not to pretend success. Preserve excluded rows as final `OTHER`, leave review-eligible rows marked as seed-kept or unreviewed, and make projection and scoring honor that state explicitly.
