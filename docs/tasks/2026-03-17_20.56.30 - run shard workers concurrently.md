Problem statement

RecipeImport's shard-v1 benchmark UI could show `active workers: 5` even though the underlying runtime still launched worker assignments serially. The shared phase-worker runtime looped through assignments one by one, and the direct knowledge and direct canonical line-role runtimes had copied that same pattern. The result was misleading progress plus unnecessary wall-clock delay on multi-shard Codex runs.

Acceptance criteria

- Shared shard-v1 worker execution launches assigned worker buckets concurrently instead of iterating them in one serial loop.
- Direct knowledge and direct canonical line-role runtimes do the same, so their live `running N` status reflects real in-flight worker assignments.
- Progress callbacks and runtime artifacts still preserve stable ordering and existing validation/promotion behavior.

Verification steps

- `source .venv/bin/activate && pytest tests/llm/test_phase_worker_runtime.py tests/llm/test_recipe_phase_workers.py tests/llm/test_codex_farm_knowledge_orchestrator.py tests/llm/test_label_phase_workers.py tests/parsing/test_canonical_line_roles.py tests/llm/test_prompt_preview.py tests/parsing/test_canonical_line_role_env.py -q`
- `source .venv/bin/activate && python -m cookimport.cli bench speed-discover --out .tmp/speed_suite.json`
- `source .venv/bin/activate && python -m cookimport.cli bench speed-run --suite .tmp/speed_suite.json --out-dir .tmp/speed-runs --scenarios benchmark_canonical_pipelined --warmups 0 --repeats 1 --max-targets 1`
- `source .venv/bin/activate && python -m cookimport.cli bench speed-run --suite .tmp/speed_suite.json --out-dir .tmp/speed-runs --scenarios benchmark_canonical_pipelined --warmups 0 --repeats 1 --max-targets 1`
- `source .venv/bin/activate && python -m cookimport.cli bench speed-compare --baseline .tmp/speed-runs/2026-03-17_20.55.30 --candidate .tmp/speed-runs/2026-03-17_20.55.41 --out-dir .tmp/speed-compares`

Evidence

- Added concurrency regressions for the shared phase-worker runtime, direct knowledge runtime, and direct canonical line-role runtime.
- Deterministic speed-suite sanity check passed on the sampled target, with `benchmark_canonical_pipelined` improving from `1.3476s` to `1.3054s`.

Constraints / gotchas

- Keep per-worker output aggregation deterministic even though workers now finish out of order.
- Do not change the per-worker Codex call contract: recipe still runs `codex-farm process --workers 1` inside each worker sandbox, and knowledge/line-role still run one direct `codex exec` per owned shard.
- `bench speed-run --include-codex-farm` is blocked in agent-run shells, so the speed-suite verification here can only sanity-check deterministic paths; live Codex concurrency is covered by focused tests and the benchmark artifact diagnosis that motivated the change.

Decision

- Use `ThreadPoolExecutor` at the worker-assignment layer and then collect results back in planned assignment order so artifact summaries remain stable.
- Apply the same concurrency model to the direct knowledge and direct canonical line-role runtimes instead of leaving them as bespoke serial loops.

Rollback / compensation plan

- Revert the thread-pool worker-launch changes in the shared runtime and the two direct runtimes together if the environment shows runner thread-safety issues or operator machines become unstable under the added fan-out.
