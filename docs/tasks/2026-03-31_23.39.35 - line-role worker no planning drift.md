Problem statement

Canonical line-role workspace workers can end a session with a partial-progress status summary instead of finishing the bounded task-file workflow. That leaves no installed shard ledger and causes fail-closed benchmark/stage runs even though the worker had enough guidance to continue.

Acceptance criteria

- The line-role workspace prompt explicitly frames the job as execution, not planning/status reporting.
- Final worker messages that admit partial progress or defer `task-apply` / `task-handoff` are treated as off-contract by line-role supervision.
- Repo-owned `answers.json` batch work counts as preserved progress for the one bounded fresh-session retry path.
- Focused parsing tests cover the new prompt/watchdog/recovery behavior.

Verification steps

- `. .venv/bin/activate && ./scripts/test-suite.sh domain parsing tests/parsing/test_canonical_line_roles_workspace.py`
- `. .venv/bin/activate && ./scripts/test-suite.sh domain parsing tests/parsing/test_canonical_line_roles_runtime_recovery.py`

Evidence

- Before: worker sessions could complete with a progress-summary final message, no `task-handoff`, and `repair_status=not_attempted`.
- After: line-role prompt/watchdog/recovery prevent that path from being accepted as a normal completion, and the focused parsing tests passed:
  - `pytest tests/parsing/test_canonical_line_roles_workspace.py` -> `23 passed in 2.19s`
  - `pytest tests/parsing/test_canonical_line_roles_runtime_recovery.py` -> `16 passed in 2.35s`

Constraints / decisions

- Keep the change narrow to canonical line-role workspace workers.
- Prefer repo-owned workflow enforcement over semantic heuristics about cookbook content.
- Treat `answers.json` only as resumable worker progress, not final authority.

Rollback / compensation

- Revert the line-role prompt/watchdog/recovery patch and the paired parsing tests if the new guardrails over-trigger on valid worker sessions.
