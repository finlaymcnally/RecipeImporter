# Align RecipeImport CodexFarm Benchmark Flags

## Problem statement

Interactive Label Studio benchmark CodexFarm runs fail against the current local CodexFarm install because RecipeImport still shells out with the retired `--benchmark-mode` process flag.

## Acceptance criteria

- RecipeImport benchmark-mode subprocess calls use CodexFarm's current RecipeImport benchmark flag name.
- Extract-mode subprocess calls do not send any benchmark-only process flag.
- The fake CodexFarm shim and focused transport tests match the live CLI contract.

## Verification steps

- `. .venv/bin/activate && pytest tests/llm/test_codex_farm_orchestrator_runner_transport.py -q`

## Evidence

- Before fix: local `codex-farm process --help` exposed `--recipeimport-benchmark-mode`, while RecipeImport still built `--benchmark-mode`, reproducing the user's interactive benchmark failure.
- After fix: `. .venv/bin/activate && pytest tests/llm/test_codex_farm_orchestrator_runner_transport.py -q` exited `0`.
- After fix: the runner emits `--recipeimport-benchmark-mode line_label_v1` only for benchmark mode, and sends no benchmark-only flag for extract mode.

## Constraints / gotchas + brief decision

- `codex_farm_recipe_mode=benchmark` is still a RecipeImport-side semantic mode; only the subprocess transport changes.
- `extract` should not be translated into any CodexFarm benchmark CLI flag because benchmark-native dispatch is only needed for line-label benchmark mode.

## Rollback / compensation plan

- Revert the runner, fake shim, tests, and doc updates together if downstream CodexFarm contract expectations change again.
