# Task: Unhobble QualitySuite parallelism

## Problem statement

QualitySuite had accumulated WSL-specific guardrails that forced conservative inner worker/runtime caps and subprocess isolation, reducing throughput significantly even when hosts could tolerate higher concurrency.

## Acceptance criteria

- QualitySuite no longer applies WSL nested-parallelism safety cap rewrites to experiment runtime settings.
- Auto experiment cap defaults to detected CPU count (instead of a fixed low ceiling) unless overridden by `COOKIMPORT_QUALITY_AUTO_MAX_PARALLEL_EXPERIMENTS`.
- WSL runs follow the same executor decision path as non-WSL in auto mode.
- Tests cover new default ceiling and no-guard/no-forced-subprocess WSL behavior.
- Bench docs reflect the new runtime contract.

## Verification steps

Run from repo root with venv active:

```bash
. .venv/bin/activate
pytest tests/bench/test_quality_suite_runner.py -q
pytest tests/bench/test_bench.py -k "quality_run" -q
```

## Evidence

- Updated `cookimport/bench/quality_runner.py` auto ceiling and WSL guard behavior.
- Updated WSL-focused tests in `tests/bench/test_quality_suite_runner.py`.
- Updated benchmark docs in `docs/02-cli/02-cli_README.md` and `docs/07-bench/07-bench_README.md`.

## Constraints / gotchas and decision

- Constraint: keep existing env overrides (`COOKIMPORT_QUALITY_AUTO_MAX_PARALLEL_EXPERIMENTS`, `COOKIMPORT_QUALITY_EXPERIMENT_EXECUTOR_MODE`) working.
- Decision: retain telemetry fields for WSL guard metadata but mark guard path retired/no-op so payload compatibility remains stable.

## Rollback / compensation plan

- Revert this task diff to restore prior WSL safety guard and fixed auto ceiling behavior.
