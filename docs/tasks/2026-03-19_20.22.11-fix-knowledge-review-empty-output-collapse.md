# Fix knowledge review empty-output collapse

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The user-visible problem is that the optional non-recipe knowledge review can now run, spend real tokens, and still contribute nothing useful. In the March 19 Salt Fat Acid Heat benchmark, four knowledge shards returned structurally valid but semantically empty outputs, one shard failed outright, and the stage ended with zero promoted useful chunks, zero snippets, and zero authority changes even though the input included obviously reusable cooking guidance. After this fix, the knowledge stage must still be conservative, but it must no longer accept the degenerate “everything is useless” answer on clearly useful shards, and it must report partial or failed review honestly instead of making the stage look successfully reviewed-but-empty.

This plan intentionally excludes two neighboring problem families. It does not solve workspace-worker watchdog kills or Codex runtime drift, because that work is being handled separately. It also does not solve upstream line-role seed quality or downstream recipe/final-stage fallout, because those are separate tracks. The target here is the knowledge stage itself: prompt semantics, output validation, promotion/apply logic, and missing-shard handling/reporting.

The change is correct when a no-token fixture replay can demonstrate all of the following. A truly low-value front-matter shard still validates as all `other`. A shard containing obvious technique/reference prose can no longer validate as blanket `u=false` with empty decisions/snippets. A missing shard remains seed-kept but is reported as unreviewed or partially reviewed rather than silently treated like a meaningful review. A positive fake-runner case yields non-zero useful chunks, decisions applied, and snippets written.

## Surprises & Discoveries

- Observation: the knowledge stage was independently broken even when the input chunks were clearly useful.
  Evidence: surviving knowledge proposals for chunks covering topics like heating oil properly, smoke points, crispness, garnishing acids, umami, and salt effects on meat/seafood/eggs still returned `u=false`, `d=[]`, and `s=[]` for every chunk.

- Observation: the current ingest validator is almost entirely structural.
  Evidence: `cookimport/llm/codex_farm_knowledge_ingest.py` checks schema shape, owned `chunk_id` coverage, and out-of-surface evidence, but it does not check exact per-block decision coverage, `u`/snippet consistency, or semantically degenerate all-false shard outputs.

- Observation: the prompt currently teaches the model to fail safe by rejecting uncertain chunks, but it does not create enough pressure to positively harvest useful knowledge when present.
  Evidence: `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md` says “If uncertain, still emit conservative `other` decisions and `s: []` for every input chunk,” while the March 19 run shows the model collapsing to exactly that behavior even on useful chunks.

- Observation: the stage currently reports “validated shards” even when those shards provided no useful review.
  Evidence: the March 19 manifest reports `validated_shards = 4` together with `useful_chunks_promoted = 0`, `decisions_applied = 0`, and `snippets_written = 0`.

- Observation: the missing knowledge shard was not harmless filler.
  Evidence: the failed shard included “How Salt Affects …” sections for meat, seafood, fat, and eggs, with durable cooking guidance that should have been excellent knowledge-stage material.

- Observation: the Stage 7 seed path limits the stage’s ceiling, but it does not explain the zero-value collapse by itself.
  Evidence: the stage only sees `seed_nonrecipe_spans`, yet even inside that seed set the model failed to recognize obvious technique/reference chunks as useful.

- Observation: using the Stage 7 seed category by itself as a semantic-empty cue was too broad for short taxonomy headings.
  Evidence: a short `SALT` taxonomy fixture needs to remain valid all-`other`, while longer technique/reference chunks in the same phase should still be protected from reviewed-empty collapse.

## Outcomes & Retrospective

The implementation landed across the prompt, bundle metadata, ingest validator, orchestrator reporting, tests, and short docs. The knowledge prompt now requires positive extraction for useful chunks, shard metadata carries per-chunk ordered block indices plus conservative strong-cue flags, and ingest rejects both exact-coverage mistakes and the March 19 reviewed-empty collapse on clearly substantive knowledge shards. Reporting now distinguishes complete review from partial/unreviewed seed-kept runs through explicit reviewed-useful, reviewed-all-other, semantic-rejection, and unreviewed counts, plus `knowledge_partially_reviewed_seed_kept` / `knowledge_unreviewed_seed_kept` authority modes when nothing meaningfully changed. The main lesson was that “shape-valid” was far too weak a success condition for this stage; it needed one deterministic semantic guard, but that guard had to stay narrow enough not to reject short taxonomy/title rows.

## Context and Orientation

In this repository, the “knowledge stage” means the optional Codex-backed review that runs after deterministic Stage 7 non-recipe grouping. The deterministic Stage 7 output is represented by `cookimport/staging/nonrecipe_stage.py` as `NonRecipeStageResult`, which includes `seed_nonrecipe_spans`, `seed_knowledge_spans`, `seed_other_spans`, and `seed_block_category_by_index`. The knowledge stage does not read the whole book directly. It only reviews those seed non-recipe spans, so the stage’s maximum recall is bounded by earlier labeling, but its own correctness is still independently testable on the chunks it does receive.

The main orchestration entrypoint is `cookimport/llm/codex_farm_knowledge_orchestrator.py` in `run_codex_farm_nonrecipe_knowledge_review(...)`. That function builds shard inputs with `cookimport/llm/codex_farm_knowledge_jobs.py`, renders live prompt text through `cookimport/llm/knowledge_prompt_builder.py`, runs workers through the shared phase runtime and Codex exec runner, validates proposals through `cookimport/llm/codex_farm_knowledge_ingest.py`, merges accepted block decisions back into Stage 7 authority with `refine_nonrecipe_stage_result(...)`, and writes reviewer artifacts with `cookimport/llm/codex_farm_knowledge_writer.py`.

The compact model-facing schema lives in `cookimport/llm/codex_farm_knowledge_models.py` and `llm_pipelines/schemas/recipe.knowledge.v1.output.schema.json`. The prompt template is `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md`. The validator today understands “did the output use the right shape and stay on owned ids?” but it does not fully understand “did the output meaningfully review the shard?” That missing semantic layer is the center of this plan.

The existing tests already give this work a good home. `tests/llm/test_knowledge_prompt_builder.py` covers prompt text. `tests/llm/test_knowledge_output_ingest.py` covers schema and ingest rules. `tests/llm/test_codex_farm_knowledge_orchestrator.py` covers end-to-end behavior via the fake runner. `tests/llm/test_knowledge_job_bundles.py` covers how shard inputs are built and therefore what validator metadata can depend on. `tests/staging/test_nonrecipe_stage.py` covers the deterministic stage that the knowledge layer refines.

For this plan, “semantically empty output” means a shard payload that covers every owned chunk id but contributes no usable review. The concrete March 19 example is a payload where every row says `u=false`, every `d` array is empty, and every `s` array is empty, even though the shard input contains reusable cooking knowledge. “Partial review” means the stage kept seed authority because one or more shards were missing or semantically rejected; it does not mean the stage genuinely reviewed and affirmed that every surviving seed label was correct.

## Plan of Work

### Milestone 1: Make the prompt contract positively extractive instead of only fail-safe

Update `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md` so the knowledge reviewer is still skeptical but no longer primed to reject everything whenever it feels uncertainty. Keep the front-matter and navigation examples, but add explicit positive examples that match the actual useful prose seen in this repo’s cookbook books: technique explanations, “How X Works” sections, troubleshooting advice, temperature guidance, storage guidance, and short headed reference chunks.

The key wording change is to replace the current blanket instruction that uncertainty should become conservative `other` plus empty snippets. The replacement should say, in plain language, that uncertainty about a chunk’s overall value is not permission to ignore obviously useful block-local guidance. When a chunk contains durable cooking knowledge, the reviewer must mark the relevant blocks `knowledge`, set `u=true`, and provide at least one snippet backed by chunk-local evidence. Only chunks that genuinely contain no reusable cooking knowledge should end as `u=false` with empty snippets.

The prompt should also teach the output invariants that the validator will enforce after this change. Each chunk result must account for every input block in order. A `u=true` chunk must include at least one `knowledge` block decision and at least one snippet. A `u=false` chunk must still contain block decisions that classify the chunk’s blocks as `other`; it may not leave `d=[]` merely because the chunk was rejected. That requirement is important because the current prompt already says “`d` must include every block in order,” but the current validator does not enforce it and the current run shows the model ignoring it.

Keep the prompt template compact and inline-json-based. Do not reintroduce file-path handoff or repo inspection instructions. The worker transport is out of scope here.

Update `cookimport/llm/knowledge_prompt_builder.py` only as needed to keep the direct prompt builder in sync with the revised template and to preserve the existing inline JSON contract tests.

### Milestone 2: Strengthen the knowledge output model and ingest validator

Add semantic invariants in `cookimport/llm/codex_farm_knowledge_models.py` so obviously self-contradictory per-chunk results cannot pass model validation. `KnowledgeChunkResultV2` should enforce these rules:

`u=true` requires at least one block decision categorized as `knowledge` and at least one snippet.

`u=false` must not contain any `knowledge` block decisions and must not contain snippets.

Block decisions must cover each input block exactly once and in original order. The output model cannot check that by itself because it does not currently know the input block set, so the exact coverage check belongs in ingest. The model should still enforce uniqueness and internal chunk-level consistency.

Then update `cookimport/llm/codex_farm_knowledge_ingest.py` so validation is no longer purely structural. The validator must confirm exact per-chunk block decision coverage against the shard input: one decision per owned input block, no missing block indices, no duplicates, and no out-of-order block ids. This means `ShardManifestEntryV1.metadata` must contain enough deterministic input metadata for each chunk. If that metadata is not already present, add it in Milestone 3 rather than reconstructing it indirectly during validation.

Add a semantic rejection path for “empty collapse” shards. The validator or a small follow-on review helper should reject a shard when all of the following are true: the output marks every chunk `u=false`; the shard contains one or more chunks with strong deterministic knowledge cues; and the output supplies no snippets and no `knowledge` decisions anywhere. The point is not to build a second model. The point is to prevent the exact March 19 failure from passing as a successful review.

The deterministic knowledge cues should come from repo-owned shard metadata, not fresh heuristic guessing from raw output text during ingest. That keeps the rule stable and testable. Good cue sources are chunk lane, table-hint presence, compact heading levels, chunk title text such as “How X Works,” and whether the Stage 7 seed lane for the chunk was already `knowledge`.

Reject only clearly impossible all-false outcomes. Do not reject honest all-`other` shards that are truly blurbs, tables of contents, endorsements, or publisher matter.

### Milestone 3: Enrich knowledge shard metadata so semantic validation is deterministic

Update `cookimport/llm/codex_farm_knowledge_jobs.py` and any related contracts used to build `ShardManifestEntryV1.metadata` so the validator can reason about chunk coverage and semantic expectations without re-parsing the original bundle. The model-facing input payload should stay compact. The extra metadata belongs in the shard manifest, not the model prompt.

For each shard, record enough chunk-local metadata to support the validator and reporting layer:

The ordered chunk ids in the shard.

For each chunk id, the ordered input block indices.

The seed lane or seed Stage 7 category for the chunk, if known.

Whether the chunk includes heading blocks or table hints.

A coarse deterministic “knowledge cue” flag that becomes true for chunks that already look like technique/reference material before the model runs. This should be conservative and explainable. Seed `knowledge` chunks, table-heavy technical chunks, and headings like “How X Works,” “Storage,” “Troubleshooting,” “Smoke Points,” and similar technique/reference forms are strong cues. Front-matter and navigation headings are not.

Do not turn this metadata into a second classifier or a hidden ground truth. Its role is narrower: it gives ingest enough information to reject the semantically absurd case where a shard full of strong knowledge cues comes back as blanket `u=false` with zero decisions and zero snippets.

Update any prompt-preview or manifest-writing code that assumes knowledge shard metadata only contains owned block indices so the richer metadata remains visible in runtime artifacts and tests.

### Milestone 4: Make promotion/apply logic honest about missing and semantically rejected shards

Update `cookimport/llm/codex_farm_knowledge_orchestrator.py` so the knowledge stage distinguishes four outcomes instead of flattening them into “validated shard” versus “missing shard”:

Useful reviewed shard. The shard produced valid outputs with at least one useful chunk or meaningful block decisions and can affect final authority.

Reviewed-but-all-other shard. The shard produced valid outputs that genuinely classify the content as all `other`, and deterministic metadata does not contradict that outcome.

Semantically rejected shard. The output passed basic JSON shape checks but failed semantic validation, such as the March 19 all-false empty collapse.

Missing shard. The worker never produced a usable output wrapper or payload.

For semantically rejected and missing shards, keep seed authority, but mark those chunks and blocks as unreviewed in the manifest/reporting layer. Do not count them toward “reviewed block count” or make the final report look like the stage actively affirmed the seed state. The current `knowledge_reviewed_seed_kept` authority mode is too optimistic when a shard was never meaningfully reviewed.

Expand the knowledge-stage report and manifest counts to include at least:

`semantic_rejection_shard_count`

`reviewed_shards_with_useful_chunks`

`reviewed_shards_all_other`

`unreviewed_chunk_count`

`unreviewed_block_count`

`all_false_empty_shard_count`

Keep the existing counts such as `decisions_applied`, `useful_chunks_promoted`, `chunks_missing`, and `snippets_written`, but make their meaning clearer by separating “no useful chunks because the book content was noise” from “no useful chunks because the stage failed to review meaningfully.”

Missing-shard handling in this milestone is reporting and authority semantics only. Do not change watchdog policy, worker orchestration, or retry transport here.

### Milestone 5: Add focused regression tests and a saved-artifact replay path

Expand `tests/llm/test_knowledge_prompt_builder.py` so the prompt text asserts the new positive extraction contract. The tests should prove the prompt no longer tells the model to resolve uncertainty by blanket-empty output, and should explicitly assert wording that ties `u=true` to snippets and knowledge decisions and `u=false` to block-level `other` decisions.

Expand `tests/llm/test_knowledge_output_ingest.py` with three new groups of cases. First, validate exact block-decision coverage against owned input blocks. Second, validate chunk-level consistency between `u`, `d`, and `s`. Third, add semantic-empty rejection fixtures. One fixture should represent true front matter and be allowed to validate as all `other`. Another should represent a technique/reference shard and should be rejected if returned as blanket `u=false`, `d=[]`, `s=[]`.

Expand `tests/llm/test_codex_farm_knowledge_orchestrator.py` so fake-runner cases cover both sides of the new behavior. One fake-runner path should return a proper useful knowledge payload and prove that `promoted_snippet_count`, `useful_chunks_promoted`, and `decisions_applied` become non-zero. Another should return the March-19-style empty collapse on a shard with strong knowledge cues and prove that the shard is semantically rejected, does not count as meaningfully reviewed, and leaves a visible warning/report count. A third path should simulate a missing shard and prove that the manifest reports partial review rather than silently flattening it into reviewed seed-kept success.

If needed, add a small fixture helper under `tests/llm/fixtures/` or similar using compact bundle shapes derived from the existing knowledge job contract. The fixture should be hand-sized but representative of the actual failed content types: short front matter, technique explanations, reference headings, and headed knowledge prose.

### Milestone 6: Update short docs for the new knowledge-stage contract

Update `cookimport/llm/README.md` so the knowledge-stage paragraph explains the strengthened contract in operator language: exact chunk ownership still matters, but validated proposals must now also satisfy chunk-level usefulness semantics and exact block-decision coverage. Mention that missing or semantically rejected shards remain seed-kept but are recorded as partial review rather than successful empty review.

Update `llm_pipelines/README.md` and the knowledge prompt asset comments if they materially describe the old conservative-empty contract. Keep these edits short. The most important goal is that a future agent reading the repo can tell why the knowledge stage rejects semantically empty all-false outputs.

## Validation and Acceptance

Acceptance is behavior, not just code shape.

First, a front-matter-only fixture must still validate as unhelpful. A shard made only of endorsements, publisher text, contents pages, or decorative headings may legitimately produce `u=false`, block-level `other` decisions, and zero snippets. That result should remain valid and should not trigger semantic rejection.

Second, a technique/reference fixture must no longer validate as blanket empty output. A shard containing headed cooking guidance such as smoke points, rendering, storage, troubleshooting, “How X Works,” or salt effects on ingredients must be rejected if it comes back with every chunk `u=false`, no decisions, and no snippets. The validator should report a clear semantic reason so the orchestrator can surface it.

Third, a positive fake-runner orchestration test must show actual stage value. Given a useful knowledge shard, the final report should show non-zero useful chunks promoted, non-zero decisions applied, and non-zero snippets written. The refined Stage 7 result should reflect any applicable knowledge-vs-other updates from the accepted review.

Fourth, a missing or semantically rejected shard must leave seed authority unchanged but must not count as meaningfully reviewed. The manifest and review summary should show partial review or unreviewed counts. The important human-visible effect is that operators can tell the difference between “the model reviewed this and found nothing useful” and “the stage never really reviewed this shard.”

Fifth, all targeted tests listed in `Concrete Steps` must pass in the project virtual environment.

## Idempotence and Recovery

These changes are safe to implement incrementally because the stage already has a seed-kept fallback path. If a proposed semantic guard is too aggressive, the correct rollback is to narrow the guard while keeping the new reporting fields and exact block-coverage checks. Do not remove the new reporting semantics merely because an early heuristic needs tuning.

The prompt changes are idempotent: re-running tests after editing the template should not require cleaning artifacts. The validator and orchestrator changes are also idempotent because the tests use temporary directories and fake-runner seams. If a test fixture proves too synthetic to catch the failure mode, replace it with a compact saved-artifact-shaped fixture rather than loosening the validator.

Because this plan intentionally does not touch the watchdog/runtime track, any failing test that still centers on command execution kills should be treated as out of scope for this plan unless the failure comes from new knowledge-stage accounting behavior.
