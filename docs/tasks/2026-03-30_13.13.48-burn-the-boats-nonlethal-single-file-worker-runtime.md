## Structured Restatement Of User Desire

The live Codex Exec runtime should finish the burn-the-boats move to one compact editable task file and stop behaving like a fragile shell-driven mini-application. The worker should read one file, edit the allowed answer fields in that same file, save, and stop. Deterministic repo code should own the hidden snapshot bookkeeping, validation, artifact expansion, installation, and bounded repair. The model-visible workspace should expose only the one file it needs, not helper files or fallback surfaces that invite defensive browsing. Just as importantly, the repository must stop trusting stage-local watchdog heuristics to kill workers for imperfect local behavior. The user has already lost multiple days to false-positive watchdog terminations. This plan therefore treats warnings, logging, and observability as the default response to messy local behavior, and treats new automatic kill rules as forbidden. The final live path should be one new runtime only: single editable task file, hidden bookkeeping outside the visible root, and non-lethal supervision.

## Purpose / Big Picture

After this change, a user can run the live recipe, knowledge, or line-role Codex Exec stages and know exactly what the worker will see and do. The worker-visible root contains one compact `task.json`. The worker reads that file once, edits only the allowed `answer` fields, saves, and stops. The repo then validates the edits using the shared answer-extraction and expansion contract defined in the earlier single-editable-file plan, while keeping the original snapshot and bookkeeping outside the worker-visible root.

The other user-visible gain is safety. A worker that reopens the wrong local section, reads an immutable field again, uses extra local file operations, or generally behaves clumsily should no longer be killed by newly invented stage-local watchdog heuristics. Those behaviors should surface in telemetry and warnings, not as benchmark-ruining terminations. The only automatic termination behavior left in the live single-file path should be the narrow hard-boundary protection needed to stop real escapes from the workspace or truly forbidden commands. Everything else becomes observable drift, not slaughter. The proof is concrete: happy-path activity traces show one file read plus edits instead of shell heredocs; stage telemetry reports warnings rather than `watchdog_killed`; and the focused test suite proves that repeated harmless local missteps do not terminate the run.

## Context and Orientation

This repository currently has three live Codex-backed worker stages: recipe, knowledge, and line-role. Recipe is the stage that turns recipe-like spans into structured recipe outputs. Knowledge is the stage that decides whether non-recipe text is `knowledge` or `other` and, for `knowledge`, assigns grouping semantics. Line-role is the stage that labels canonical lines with semantic line types. All three stages use `codex exec` in a local workspace, and all three currently rely on repo-owned control surfaces that invite repeated file reads and shell-visible JSON movement.

A “worker-visible workspace” means the mirrored directory the model sees as its current working directory during a `codex exec` run. After this implementation, that visible root is just `task.json` plus `AGENTS.md`; the richer worker tree remains repo-owned in the source artifact root. A “hard-boundary violation” means the worker clearly attempted to leave the allowed workspace or invoke obviously forbidden repo, network, or package-manager commands. A “heuristic kill rule” means any automatic termination behavior that guesses the worker is going bad based on things such as command count, repeated reasoning, elapsed time, cohort median comparisons, or repeated local retries. This plan keeps the former and deletes the latter from the final live single-file path.

The relevant runtime owners are spread across a few files:

- `cookimport/llm/codex_exec_runner.py` builds the shared direct-exec workspace, worker prompt text, and runner payloads.
- `cookimport/llm/phase_worker_runtime.py` still carries shared worker-assignment structures that the fixed-assignment plan is already replacing.
- `cookimport/llm/recipe_stage_shared.py` still contains recipe workspace supervision callbacks and live worker integration.
- `cookimport/llm/knowledge_stage/workspace_run.py`, `cookimport/llm/knowledge_stage/recovery.py`, and `cookimport/llm/knowledge_phase_workspace_tools.py` still carry the live knowledge packet and repair stories.
- `cookimport/parsing/canonical_line_roles/runtime.py` and `cookimport/parsing/line_role_workspace_tools.py` still carry the line-role workspace and watchdog-retry behavior.
- `cookimport/llm/prompt_artifacts.py`, `cookimport/llm/prompt_budget.py`, and `cookimport/runs/stage_observability.py` expose the current telemetry and benchmark-facing story.

The two checked-in predecessor plans already define the broad direction. The fixed-assignment plan removes queue leasing and phase-ledger loops. The single-editable-file plan says the final model-visible surface should be one editable `task.json`, with deterministic validation and expansion afterward. This plan adds the missing operational contract around that file. The repo, not the worker, owns the original snapshot of `task.json`, the edit extraction, the artifact expansion, the repair rewrite, and the supervision status bookkeeping. The worker-visible root should contain only the one file that matters.

## Plan of Work

The first implementation step is to make `cookimport/llm/codex_exec_runner.py` capable of launching a live worker whose visible workspace root contains only `task.json`. The runner must stop writing model-visible `worker_manifest.json`, `OUTPUT_CONTRACT.md`, `examples/`, `tools/`, `assigned_shards.json`, `assigned_tasks.json`, `current_packet.json`, `current_result_path.txt`, and similar helper surfaces for the new live path. Instead, it should write `task.json` into the visible root and save all repo-owned bookkeeping outside that root. That hidden bookkeeping includes the original immutable snapshot of `task.json`, the runner metadata needed for activity traces, and any internal validation scaffolding. The shared answer extraction, deterministic expansion, and repair rewrite contract from the earlier single-editable-file plan should be reused here rather than redefined.

The second implementation step is to wire the runner and stage owners to the shared repo-side completion seam from the earlier single-editable-file plan. On launch, the runner writes an original copy of `task.json` into a hidden repo-owned location. On completion, the repo loads both the original and edited versions, computes the exact changed JSON paths, rejects any change outside the allowed `editable_json_pointers`, and extracts only the `answer` payloads that the worker owns. The stage-specific deterministic validators already know how to reason about the semantic output. They should consume those extracted `answer` payloads rather than expecting the worker to write heavyweight artifact JSON directly. This plan does not introduce a second output language. It finishes the runtime around the shared `task.json` contract that the earlier plan already defines.

The third implementation step is to reuse the bounded repair rewrite from the earlier single-editable-file plan and delete the old special retry stories around it. If validation fails, deterministic code rewrites `task.json` into repair mode using the shared repair shape. The worker then gets one bounded repair session editing the same `answer` fields in the same file shape. There is no separate watchdog-retry prompt, no separate helper-owned repair ledger, and no second live output language. If repair fails again, the unit fails closed and the repo records the failure in the normal stage summaries.

The fourth implementation step is to replace stage-local kill-first supervision with non-lethal supervision. The new live path must delete or demote automatic termination rules based on command count, repeated local reads, repeated reasoning items, elapsed-runtime outliers, or any similar heuristic. Instead, the runner should record soft supervision observations in a small repo-owned status artifact, such as whether the worker stayed on the happy path, used shell fallback, reread immutable sections repeatedly, or exited without valid edits. These observations should not terminate the run. They exist so the user can see cost and quality problems without losing the run. The only automatic termination logic left in the new live path should be the narrow hard-boundary seam that already protects against genuine escapes from the workspace or forbidden repo/network/package-manager commands.

The fifth implementation step is to make the supervision vocabulary honest. The live code should stop calling every off-happy-path observation a “watchdog” event. The shared telemetry should distinguish between normal happy-path completion, off-happy-path-but-accepted completion, hard-boundary termination, and deterministic repair requested. Existing counters and labels such as `watchdog_killed_shard_count`, `watchdog_retry`, and other kill-oriented names should be removed from the live single-file path or renamed to reflect the new behavior. Historical artifacts may still contain the old names, but the active runtime, active docs, and active tests should use the new supervision vocabulary.

The sixth implementation step is to make temptation removal structural. Once the new live path exists, the worker-visible root for the single-file runtime should contain only `task.json`. If the runner still needs a local prompt header, it should be injected through the direct prompt or embedded into immutable fields inside `task.json`, not written as separate visible files. If the repo still needs logs, examples, manifests, or output-contract text for debugging, those should live outside the worker-visible root. The model should not see them on the happy path.

The seventh implementation step is to update stage-specific owners. Recipe, knowledge, and line-role all need to stop treating their current watchdog callbacks and retry flows as live behavior. Recipe should remove the command-loop and reasoning-without-output kill path from the single-file worker mode. Knowledge should stop advertising packet-lease helpers and repair flows in the live path once the earlier plan replaces them. Line-role should remove the current strict-JSON watchdog callback, its cohort-runtime-outlier logic, and the line-role watchdog-retry branch from the new single-file path. In all three cases, repair should come only from deterministic validation rewriting `task.json`, not from a supervision callback killing the session and launching a special retry prompt.

The eighth implementation step is to reset telemetry and tests so the user can prove the new contract. Prompt-budget summaries should record whether a run stayed on the happy path or drifted off it, but should not present benign drift as termination. Activity traces should be able to show “the worker reread immutable fields and still finished successfully” without converting that into failure. The focused test suite must cover both clean and messy local behavior. A worker that reads the wrong local section a few times, or uses extra local file operations, must still finish successfully if it ends with a valid edited `task.json`. That is the key behavioral proof this plan adds beyond the earlier two plans.

At the end of this milestone, the worker no longer writes heavyweight artifact JSON directly for the new path. Instead, the repo compares original and edited `task.json`, extracts only the allowed `answer` edits, validates them, and expands them into the final stage artifacts. A failed validation produces one repair rewrite of `task.json` and no special watchdog-retry flow. The proof is a focused test that fails validation on the initial edit, rewrites the repair task, accepts the repaired edit, and shows that the final promoted artifacts came from deterministic expansion rather than model-written heavyweight JSON.

### Milestone 3: Non-lethal supervision replaces heuristic kill rules

At the end of this milestone, the live single-file path no longer terminates for command-count heuristics, reasoning-count heuristics, repeated local rereads, or cohort-runtime-outlier logic. The runner still records these behaviors in telemetry, but the worker is allowed to finish if it eventually produces a valid edit. The proof is a focused supervision test that simulates repeated harmless local missteps and proves the run completes with an `off_happy_path` warning rather than a termination. Another focused test should still prove that true hard-boundary escapes terminate immediately.

### Milestone 4: Stage adoption and deletion of the old live stories

At the end of this milestone, recipe, knowledge, and line-role all use the shared one-file runtime and no longer advertise model-visible helper files or stage-local watchdog-retry logic as live behavior. The proof is that the stage tests for recipe, knowledge, and line-role all pass under the new contract, and that searching the live runtime owners no longer finds active single-file-path branches that terminate for `watchdog_command_loop_without_output`, `watchdog_reasoning_without_output`, or `watchdog_cohort_runtime_outlier`.

### Milestone 5: Telemetry, benchmarks, and docs tell the new truth

At the end of this milestone, prompt-budget summaries, activity traces, stage observability, and live docs all describe the new runtime accurately: one editable file, repo-side extraction, bounded repair, warnings for drift, and hard-boundary termination only. The proof is a scripted fake-exec run whose activity trace shows direct file edits rather than heredoc JSON writes, whose prompt-budget output stops centering watchdog-kill vocabulary, and whose docs teach only the new contract.

## Validation and Acceptance

This plan is accepted only when the live single-file runtime is both simpler and safer. Simpler means the worker-visible root contains only `task.json`, the worker edits that file in place, and the repo owns the snapshot, diff, validation, expansion, and repair rewrite. Safer means that ordinary messy local behavior no longer terminates the worker under stage-local heuristics.

The first acceptance criterion is workspace shape. A focused runner test must prove that the visible root for the new live path contains only `task.json`. The original task snapshot, expansion bookkeeping, trace metadata, and any repair bookkeeping must live outside the worker-visible root. If the visible root still contains `worker_manifest.json`, `OUTPUT_CONTRACT.md`, `examples/`, `assigned_shards.json`, `current_packet.json`, or similar helper surfaces on the live path, this plan is not done.

The second acceptance criterion is write/install behavior. A focused test must prove that the worker edits `task.json` in place, the repo detects only the allowed changed `answer` paths, and stage-specific deterministic code expands those answers into the heavyweight artifacts. Another focused test must prove that a failed validation rewrites the same file into repair mode and that the repaired file can then promote successfully.

The third acceptance criterion is non-lethal supervision. A focused supervision test must prove that repeated harmless local retries do not terminate the run. This includes rereading immutable sections, opening the wrong local area and correcting course, or using extra local file operations before producing a valid edit. If those behaviors still terminate the live single-file path, the plan has failed its most important user requirement.

The fourth acceptance criterion is narrow hard-boundary enforcement. A focused test must still prove that a real escape attempt, such as a forbidden repo/network/package-manager command or an obvious path escape outside the allowed local workspace or approved temporary roots, terminates immediately. The acceptance point is not “no kills ever.” The acceptance point is “kills only for true hard-boundary violations.”

The fifth acceptance criterion is deletion of heuristic kill logic. The live single-file path must no longer terminate for reasons equivalent to command-loop count, repeated reasoning without output, runtime-outlier cohort comparisons, or similar heuristic guesses. Those reasons may survive only in historical artifacts or old archived benchmark trees, not in the live path. Searching the live runtime owners should show they are removed or demoted to non-lethal telemetry.

The sixth acceptance criterion is honest telemetry. Prompt-budget and activity-trace outputs must describe off-happy-path behavior as warnings or drift, not as `watchdog_killed`, unless a true hard-boundary termination occurred. Stage observability must report repair requests separately from supervision drift. The user should be able to tell “this run was messy but usable” apart from “this run was forcibly stopped.”

The seventh acceptance criterion is documentation clarity. `docs/10-llm/10-llm_README.md` and any touched stage-local README or note must teach only the new single-file, non-lethal runtime story. A novice should not discover active docs that still imply the live path is packet-leased, phase-ledger-driven, or kill-first.

## Idempotence and Recovery

The implementation should be safe to repeat because each milestone is anchored by focused tests and because the new runtime contract is additive until the old path is deleted. If the shared runner lands before all stages adopt it, keep the old and new code paths separate by explicit mode and delete the old mode only when the stage-specific tests are green. The final merged repository must still ship only the new live path, but temporary branch-local scaffolding is allowed while the migration is underway.

If a step fails halfway through, do not paper over the problem by reintroducing new kill heuristics. Instead, keep the failing worker result, inspect the edited `task.json`, inspect the repo-owned supervision summary, and tighten the deterministic validation or task-file shaping. The entire purpose of this plan is to make debugging possible without losing runs to false-positive termination.

If a temporary bridge helper is required during implementation, keep it branch-local if possible. If it must briefly exist in the tree, make it consume the same edited `task.json` that the final path will use and delete it before final acceptance. Do not let a temporary helper become a second live contract.
