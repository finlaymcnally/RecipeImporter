# Problem Statement

Trend hover cards were run-group oriented and did not clearly show the hovered dot's own score/source, which made multi-book run groups feel aggregated.

# Acceptance Criteria

- Hovering a trend dot shows that dot's exact score.
- Tooltip includes source/book context for the hovered dot.
- Tooltip includes dot-level run context (`variant`, eval-row timestamp).
- Existing run-group tooltip context remains available.

# Verification Steps

1. `source .venv/bin/activate && pytest tests/analytics/test_stats_dashboard.py -k points_include_source_metadata_for_tooltips`
2. `source .venv/bin/activate && pytest tests/analytics/test_stats_dashboard.py`

# Evidence

- Added `test_benchmark_trend_points_include_source_metadata_for_tooltips` (initially failed before renderer updates, now passes).
- Full dashboard analytics renderer suite passes in `tests/analytics/test_stats_dashboard.py`.

# Constraints / Gotchas

- Trend series points are grouped by run-group timestamp, so tooltip logic must preserve grouped context while surfacing hovered-point specifics first.

# Decision

Extended each trend point `custom` payload with source/variant/timestamp metadata and updated the tooltip formatter to render hovered-point details first, then optional run-group values.

# Rollback Plan

Revert the trend point `custom` metadata additions and tooltip formatter block in `cookimport/analytics/dashboard_render.py` if the new card content causes UX regressions.
