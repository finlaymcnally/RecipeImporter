# Parallelize interactive all-method benchmark with bounded resource use

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

`docs/PLANS.md` is checked into the repo and this document must be maintained in accordance with it.

## Purpose / Big Picture

Right now, interactive all-method benchmark executes each configuration one-by-one. On long EPUB runs this can take a very long time, and total CPU usage often stays lower than the machine can handle. After this change, all-method benchmark will run configs through a bounded queue with `4` in-flight pipelines, while hard-gating the heavy split-worker phase to `2` concurrent pipelines. This keeps the expensive phase saturated without letting all pipelines hit peak CPU/RAM at once.

User-visible behavior after implementation:

1. Interactive all-method flow shows planned run count plus scheduler limits (`in-flight pipelines` and `split-phase slots`).
2. Benchmark executes multiple configs concurrently (when parallelism > 1) and prints live completion progress.
3. Existing per-config and aggregate reports stay in the same locations and formats.
4. Benchmark CSV history rows remain valid (no corruption/race damage).

## Surprises & Discoveries

- Observation: `_run_all_method_benchmark(...)` currently executes a strict serial `for` loop and calls `labelstudio_benchmark(...)` once per config.
  Evidence: `cookimport/cli.py` in `_run_all_method_benchmark`.

- Observation: `labelstudio_benchmark(...)` temporarily mutates process-global `C3IMP_EPUB_*` environment variables during prediction generation.
  Evidence: `_temporary_epub_extractor(...)` and `_temporary_epub_unstructured_options(...)` in `cookimport/cli.py`.

- Observation: all-method config runs for a source append benchmark rows to the same per-source history CSV path.
  Evidence: `labelstudio_benchmark(...)` uses `append_benchmark_csv(..., history_path(processed_output_dir))`; in all-method flow `processed_output_dir` values share one parent source root.

- Observation: prediction generation already has its own split-worker multiprocessing; naive outer parallelism can over-subscribe CPU/RAM.
  Evidence: `_plan_parallel_convert_jobs(...)` and `ProcessPoolExecutor` usage in `cookimport/labelstudio/ingest.py`.

- Observation: this sandbox environment blocks multiprocessing semaphore creation (`PermissionError`), so process-pool tests are not reliable here.
  Evidence: `ProcessPoolExecutor` startup failed in test runs with `[Errno 13] Permission denied`; test coverage was adapted to deterministic thread-based queue checks and gate-loop unit tests.

## Decision Log

- Decision: Parallelize at the all-method config layer (outer loop), not by rewriting ingestion internals.
  Rationale: It directly targets the user-visible bottleneck with the smallest behavior surface change and keeps existing benchmark evaluation/report contracts.
  Date/Author: 2026-02-24 / Codex

- Decision: Use process-based outer parallel execution, not threads.
  Rationale: benchmark paths currently use process-global environment overrides (`C3IMP_EPUB_*`), which are unsafe for same-process threaded concurrency.
  Date/Author: 2026-02-24 / Codex

- Decision: Keep a queue of configs with `max_inflight_pipelines=4` and refill immediately when one completes.
  Rationale: this keeps the pipeline busy (prep + heavy split + post phases) and avoids idle gaps between configs.
  Date/Author: 2026-02-24 / Codex

- Decision: Add a hard phase gate `max_concurrent_split_phases=2` that is acquired only during the split-worker-heavy section and released immediately after.
  Rationale: this matches observed machine behavior: keep the intensive phase at two simultaneous books/configs while allowing additional pipelines to pre-stage and post-process.
  Date/Author: 2026-02-24 / Codex

- Decision: Preserve configured split worker counts per pipeline during the gated heavy phase (for example `~10` split jobs), rather than globally dividing workers across all 4 in-flight pipelines.
  Rationale: the phase gate already controls concurrent heavy-load count; preserving per-pipeline split width keeps throughput high.
  Date/Author: 2026-02-24 / Codex

- Decision: Keep all existing output/report file contracts and ranking semantics unchanged.
  Rationale: dashboard/report/docs already consume these artifacts; performance improvement should not require downstream rewrites.
  Date/Author: 2026-02-24 / Codex

- Decision: Propagate split-slot settings through internal contextvars in `cookimport/cli.py` rather than exposing new public `labelstudio-benchmark` CLI options.
  Rationale: keeps user-facing CLI stable while allowing all-method orchestration to inject scheduler constraints.
  Date/Author: 2026-02-24 / Codex

- Decision: Use dependency-free advisory file locks for both split-slot gating and CSV appends.
  Rationale: avoids new packages and gives inter-process coordination on Linux in both benchmark generation and history writes.
  Date/Author: 2026-02-24 / Codex

## Context and Orientation

The relevant runtime path is all in `cookimport/cli.py`:

- `_interactive_all_method_benchmark(...)` picks scope and launches all-method runner(s).
- `_build_all_method_variants(...)` builds config permutations.
- `_run_all_method_benchmark_multi_source(...)` orchestrates per-source sweeps.
- `_run_all_method_benchmark(...)` runs all configs for one source/gold pair and writes `all_method_benchmark_report.{json,md}`.
- `labelstudio_benchmark(...)` is the single-run benchmark primitive used per config.

The benchmark import path (called by `labelstudio_benchmark`) lives in `cookimport/labelstudio/ingest.py` and can already spawn split workers for PDF/EPUB jobs.

Benchmark history write logic is in `cookimport/analytics/perf_report.py` (`append_benchmark_csv` and shared CSV helpers).

Terms used in this plan:

- Outer parallelism: running multiple all-method configs at the same time.
- Inner parallelism: per-config worker fanout inside prediction generation (split jobs).
- Worker budget: deliberate throttling so outer parallel configs do not each run with full inner worker counts.

## Plan of Work

Milestone 1 introduces explicit scheduler controls (`in-flight` and `split-phase slots`). Milestone 2 extracts a single-config execution helper so serial and parallel paths share one implementation. Milestone 3 adds bounded process-based queue execution in `_run_all_method_benchmark(...)`. Milestone 4 adds split-phase gating in prediction generation so only two configs can run the heavy split section simultaneously. Milestone 5 hardens benchmark CSV appends for concurrency. Milestone 6 updates tests and docs.

### Milestone 1: Add scheduler controls for in-flight and split-phase limits

In `cookimport/cli.py`, add helper logic that decides:

- `max_inflight_pipelines` (at least 1, default 4, at most total variants),
- `max_concurrent_split_phases` (at least 1, default 2, at most `max_inflight_pipelines`).

Keep this simple and bounded. First implementation rules:

- serial mode (`max_inflight_pipelines == 1`): keep current behavior,
- queued mode (`max_inflight_pipelines > 1`): run up to 4 configs concurrently but enforce 2 split-phase slots.

Interactive all-method should surface these settings clearly before final confirmation.

Acceptance for Milestone 1:

- all-method interactive flow displays both scheduler limits before execution,
- computed values are deterministic for the same inputs.

### Milestone 2: Factor one-config execution into a reusable helper

In `cookimport/cli.py`, extract the current per-config body from `_run_all_method_benchmark(...)` into a dedicated helper that:

- runs one benchmark config,
- returns a normalized row payload (status, metrics, run-config context, artifact paths),
- never mutates caller-owned state.

Both serial and parallel modes must use this helper so behavior stays aligned.

Acceptance for Milestone 2:

- current serial behavior still produces identical report payloads (except for any new metadata fields intentionally added).

### Milestone 3: Add bounded queue execution for config runs

In `_run_all_method_benchmark(...)`, support two execution paths:

- serial path (existing behavior),
- queue path using a process executor with `max_workers=max_inflight_pipelines`.

Requirements:

1. Preserve existing config directory naming and artifact layout.
2. Keep progress readable (`Running config X/Y`, `Completed config X/Y`).
3. Continue after per-config failures and include failures in final report.
4. Keep ranking deterministic: sort successful rows by metrics as today, independent of completion order.
5. If process executor startup fails (for example `PermissionError` on restricted environments), print a warning and fall back to serial mode.
6. Queue semantics must be explicit: submit up to `max_inflight_pipelines` initial tasks, then submit one new config immediately when one completes.

Acceptance for Milestone 3:

- with `max_inflight_pipelines > 1`, multiple config runs are in-flight concurrently,
- report row content and winner logic remain equivalent to serial semantics.

### Milestone 4: Add split-phase gating (max two heavy split phases)

Add a cross-process split-phase limiter in prediction generation path (recommended in `cookimport/labelstudio/ingest.py` around the `len(job_specs) > 1` split conversion block).

Requirements:

1. Acquire one split slot immediately before running split-worker-heavy conversion.
2. Release the split slot immediately after split conversion merges (even on exception via `finally`).
3. Non-split runs (`len(job_specs) == 1`) should bypass split-slot gating.
4. Slot count comes from scheduler config (`max_concurrent_split_phases`), default 2.
5. Logging/progress should show slot waits/acquire/release events so queue behavior is observable.

Acceptance for Milestone 4:

- at most two pipelines are in split-worker-heavy phase simultaneously,
- other in-flight pipelines can continue non-split prep/post work while waiting for a slot.

### Milestone 5: Make benchmark CSV append concurrency-safe

Parallel configs can append benchmark rows at the same time. Harden CSV writes in `cookimport/analytics/perf_report.py` so concurrent appends do not corrupt files.

Scope:

- wrap append critical sections in an inter-process file lock,
- cover both `append_benchmark_csv(...)` and `append_history_csv(...)` because both use the same CSV contract.

Keep lock logic local and dependency-free (no new third-party package).

Acceptance for Milestone 5:

- concurrent benchmark append attempts produce valid CSV with one header and intact rows.

### Milestone 6: Tests and docs

Update tests in:

Add/adjust coverage for:

2. fallback-to-serial path when process executor cannot start,
3. split-phase gate enforces max two concurrent heavy split sections,
4. parallel benchmark CSV appends remain valid.

## Idempotence and Recovery

All-method runs already write to timestamped roots, so reruns are additive and safe. If a run fails mid-way:

- rerun the interactive flow to create a new timestamped root,
- compare old/new report payloads for failed configs.

If process-based parallelism fails at startup in the current environment:

- runner must auto-fallback to serial mode and continue producing artifacts.

No destructive operations (`git reset --hard`, deleting global history, etc.) are needed.
