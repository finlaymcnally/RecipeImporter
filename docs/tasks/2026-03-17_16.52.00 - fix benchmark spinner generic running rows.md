Problem statement

Benchmark import status panels were not reflecting recent worker-row updates for canonical line-role progress. The shared spinner only expanded `running N` into worker rows for `codex-farm ...` messages, and some codex-farm builds were also leaking repetitive legacy queue/running stderr snapshots.

Acceptance criteria

- Benchmark status panels show worker rows for generic benchmark messages shaped like `task X/Y | running N`.
- Canonical line-role benchmark progress no longer looks like a single stale line when workers are active.
- Legacy codex-farm `run=... queued=... running=...` stderr snapshots are treated as progress noise, not surfaced as ordinary stderr.

Verification steps

- `source .venv/bin/activate && pytest tests/labelstudio/test_labelstudio_benchmark_helpers_progress.py -k 'generic_running_task_progress or canonical_line_role_progress'`
- `source .venv/bin/activate && pytest tests/llm/test_codex_farm_orchestrator_runner_transport.py -k 'legacy_run_progress or progress_events'`

Evidence

- Added a benchmark progress regression test that asserts canonical line-role `task 1/4 | running 4` renders `active workers: 4` plus worker rows.
- Added a codex-farm runner regression test that drops legacy `run=... queued=... running=...` stderr snapshots from non-progress stderr extraction.

Constraints / gotchas

- Keep the change in the shared progress wrapper instead of adding a benchmark-only special case.
- Do not regress the existing codex-farm worker-panel behavior or ETA formatting.

Decision

- Parse generic `running N` hints whenever the callback message already includes a task counter.
- Suppress legacy codex-farm queue/running stderr snapshots at the subprocess runner boundary and in stderr summarization.

Rollback / compensation plan

- Revert this change if generic `running N` matching proves too broad; the regression tests isolate the intended message shapes.
