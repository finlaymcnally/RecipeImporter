# Complete the second AI-readiness refinement wave

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

After the first AI-readiness umbrella plan, RecipeImport’s public roots and obvious fake facades are in much better shape, but several private implementation owners are still so large that a new contributor must read thousands of lines before making a safe local change. This next wave finishes the same philosophy one layer deeper. When this plan is complete, a contributor should be able to work on benchmark helpers, canonical line-role runtime, and knowledge-stage runtime by opening one focused owner module instead of one giant package root.

The user-visible proof is not a new product feature. The proof is safer and faster changeability. The benchmark, parsing, knowledge, analytics, and settings domains should keep their current behavior, but the file system should reflect the real ownership map more honestly, and the existing domain suites should still pass. A newcomer should be able to discover the right starting file without reverse-engineering architecture from a 7k to 14k line module.

### Why these belong in one plan

These are not arbitrary large files. They are the remaining places where the file system still asks a newcomer to read around inside one broad implementation owner to find the real seam. Fixing them together is useful because the same design rule applies in all cases:

1. keep the public behavior stable,
2. move one responsibility family into one clearly named owner module,
3. keep the package root small and readable,
4. prove the change through the existing domain suites and a strengthened architecture guard.

## Idempotence and Recovery

This plan is intentionally structural and should be implemented incrementally. Each milestone should be safe to retry because the acceptance harness is the existing test suite plus the architecture guard. If one extraction creates regressions, do not keep pushing deeper into the next milestone. Fix the local seam, rerun the touched domain suite, and only then continue.

When moving helpers between modules, preserve the current import path and compatibility surface until tests are green. This repository is full of narrow monkeypatch-based tests. Do not delete a compatibility export just because a new owner file exists; delete it only after the behavior is proven and the architecture test has been updated deliberately.

If a proposed split turns out to create circular imports, stop and adjust the ownership cut rather than forcing more compatibility glue. The goal of this plan is to reduce coordination surfaces, not to create a second wave of loader tricks or fake facades.
