# Problem Statement

EPUB/PDF imports could sit for long stretches on `Analyzing standalone knowledge blocks...` because standalone topic containers were processed serially.

# Acceptance Criteria

- EPUB and PDF standalone knowledge-block analysis supports bounded parallel container processing.
- Progress for that phase reports `task X/Y` counters.
- Output ordering remains deterministic.
- Ingestion/parsing tests covering EPUB/PDF and tip extraction pass.

# Verification Steps

```bash
source .venv/bin/activate
pytest tests/parsing/test_tip_extraction.py tests/ingestion/test_epub_importer.py tests/ingestion/test_pdf_importer.py -q -o addopts=''
```

# Evidence

- `pytest tests/parsing/test_tip_extraction.py tests/ingestion/test_epub_importer.py tests/ingestion/test_pdf_importer.py -q -o addopts=''` -> `37 passed, 7 warnings in 10.97s`.

# Constraints / Gotchas

- Threaded worker logic must not append directly into shared lists; each container worker must use local buffers and merge in sorted container order.
- Existing callback contract must remain compatible with spinner ETA parsing.

# Decision

Implement container-level `ThreadPoolExecutor` parallelism in `_extract_standalone_tips(...)` for both importers, with deterministic merge-by-container-index and `task X/Y` progress updates.

# Rollback / Compensation

Set `C3IMP_STANDALONE_ANALYSIS_WORKERS=1` to force serial behavior without reverting code.
