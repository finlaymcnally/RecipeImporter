# Build long-lived manifest-driven Codex workers

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The user-visible goal is to make Codex-backed stages work the way a strong coding agent naturally works: give one worker a bounded local workspace, a manifest of many small owned tasks, and let it work through them over a longer horizon by reading files and writing output files incrementally. After this change, `line_role`, `knowledge`, and `recipe` should no longer depend on one monolithic final answer per shard. Instead, each worker session should process many task files in one local loop, leave durable per-task outputs behind, and allow deterministic validation and resume to happen task by task.

The change is correct when a stage run shows fewer fresh Codex sessions than task packets, when worker folders contain explicit task manifests plus per-task outputs, when partial progress survives a worker kill or restart, and when stage promotion is driven by validated task outputs rather than one giant end-of-session blob. This plan intentionally prefers long-lived worker sessions over many tiny fresh model calls. Live-token benchmark reruns remain optional and require explicit user approval.

## Surprises & Discoveries

- Observation: the current runtime already has the folder shape that looks like a worker loop, but not the execution model.
  Evidence: `cookimport/llm/README.md` and `docs/10-llm/10-llm_README.md` both describe shard-v1 as “one direct `codex exec` structured call per shard,” even though the runtime also writes worker folders, assigned-shard manifests, and sterile workspaces.

- Observation: the shard refactor reduced orchestration ambiguity but did not change the core transport contract enough to match the intended human model of “give the worker many tasks and let it work through them.”
  Evidence: `cookimport/llm/README.md` says recipe and knowledge still send one full shard prompt once on stdin, and the line-role March 19 failure showed the same mismatch in practice: first-pass tasks either tried tools or returned `{"rows":[]}`, while repairs tried to emit one huge final object.

- Observation: the current direct-exec runner is optimized for “one prompt in, one final payload out,” not for a long-lived workspace session that edits files incrementally.
  Evidence: `CodexExecRunner.run_structured_prompt(...)` in `cookimport/llm/codex_exec_runner.py` is schema-centered and returns one `response_text`; the current telemetry and supervision model also assume one final answer event.

- Observation: the sterile workspace mechanism is still useful for the long-lived design, but its instructions must invert.
  Evidence: `_build_direct_exec_agents_text(...)` in `cookimport/llm/codex_exec_runner.py` currently tells the worker not to inspect local files for orientation and to return only final JSON. That is correct for one-shot structured prompts and wrong for a local task-loop worker.

- Observation: the first useful landing did not need a brand-new cross-stage `assigned_tasks.json` abstraction.
  Evidence: the line-role pilot was able to reuse the existing worker-local `assigned_shards.json` plus `in/<shard_id>.json` files as the worker-visible task queue, which kept the transport change smaller while still proving the long-lived local loop.

- Observation: the fake subprocess seam needed to learn the new contract too.
  Evidence: `scripts/fake-codex-farm.py exec` originally only emitted one final JSON message on stdout, so subprocess tests fell back with `missing_output_file` until the fake exec path was taught to read worker-local `assigned_shards.json` + `in/*.json` and write `out/*.json`.

## Context and Orientation

This repository already has the beginnings of a worker-oriented runtime, but the live direct-exec transport still behaves like a one-shot request-response system.

A “stage” in this repository is one major Codex-backed reasoning seam: `line_role` in `cookimport/parsing/canonical_line_roles.py`, `knowledge` in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, and `recipe` in `cookimport/llm/codex_farm_orchestrator.py`. Each stage plans owned work units from deterministic upstream data, invokes Codex, validates the result, and promotes only validated outputs.

A “shard” today is an ownership unit. It has a `shard_id`, explicit owned IDs, input payload, and artifacts written under a worker folder. The current shard refactor already created this structure. The problem is that a shard is still also the live model-call boundary. For recipe and knowledge, the full shard prompt is sent inline once and the stage expects one final structured answer. For canonical line-role, the stage writes a worker-local file and sends a wrapper prompt, but it still expects one final answer object for the whole shard.

A “worker” in the new plan means something stronger than the current runtime. It is not merely a bucket that owns several shards while the code still calls the model separately for each shard. It is one long-lived Codex session with one bounded local workspace. That workspace contains a manifest plus many small task files. The worker is explicitly told to read local task files, process them sequentially, and write one output file per task. The repo then validates those outputs and promotes them.

The existing relevant files are:

- `cookimport/llm/codex_exec_runner.py`: the direct `codex exec` transport, live event streaming, supervision, sterile workspace preparation, and current structured-output assumptions.
- `cookimport/llm/phase_worker_runtime.py`: worker assignment, worker folder preparation, shard manifests, and phase-level worker bookkeeping. This is the natural home for generic task-manifest runtime helpers.
- `cookimport/parsing/canonical_line_roles.py`: the current line-role shard runtime and the simplest first migration target.
- `cookimport/llm/codex_farm_knowledge_orchestrator.py`: the current knowledge shard runtime.
- `cookimport/llm/codex_farm_orchestrator.py`: the current recipe shard runtime.
- `cookimport/llm/canonical_line_role_prompt.py`, `cookimport/llm/knowledge_prompt_builder.py`, and recipe prompt-building helpers in `cookimport/llm/codex_farm_orchestrator.py`: these currently build one-shot prompts and will need to be split into worker instructions plus per-task file shapes.
- `docs/10-llm/10-llm_README.md` and `cookimport/llm/README.md`: both currently describe the one-shot-per-shard model and must be updated once the migration lands.

The long-lived worker design in this plan is intentionally file-centric. The first shipped landing uses the existing shard vocabulary rather than introducing generic task names immediately, but the worker still succeeds by doing the following inside its own sterile workspace:

1. read `assigned_shards.json`
2. inspect immutable inputs under `in/`
3. write per-task or per-shard outputs under `out/`
4. optionally rely on repo-written `logs/`, `debug/`, or `shards/` artifacts that are mirrored into the sterile workspace when the stage/runtime needs them
5. leave a small final worker summary message when done

The worker should not need to inspect the repository, hunt for docs, or reconstruct its own task list from prose. The local files are the work contract.

## Plan of Work

The first milestone is to add a new generic direct-exec transport for a long-lived local worker session. Extend `cookimport/llm/codex_exec_runner.py` with a second execution mode beside `run_structured_prompt(...)`. The new mode should run one `codex exec` session per worker assignment without an output schema, pass a worker instruction prompt on stdin, allow local file reads and writes inside the sterile workspace, stream events as today, and return a session report that includes the same supervision metadata plus the final worker message if one exists. The important design choice is that the repo stops treating the final message as the stage output. The authoritative outputs become files written inside the workspace and copied back into the repo-owned worker artifact folder after normal exit or after a watchdog kill.

The second milestone is to turn the existing sterile workspace into a real task workspace. `prepare_direct_exec_workspace(...)` in `cookimport/llm/codex_exec_runner.py` should keep the external isolation root, but `_build_direct_exec_agents_text(...)` must be split by runtime mode. The new long-lived worker mode should tell Codex to use the local task files, not avoid them. The worker-local workspace must contain:

- `worker_manifest.json` with stage name, worker ID, output contract, and allowed folders
- `assigned_tasks.json` listing ordered task IDs
- `tasks/<task_id>.json` for immutable owned inputs
- `schemas/` or `contracts/` files when the stage needs structured output guidance
- `out/` for completed task outputs
- `failed/` or `quarantine/` for repo-side quarantine after validation failures
- `logs/` for copied event streams and any repo-written status snapshots

The third milestone is to add a generic task-loop runtime in `cookimport/llm/phase_worker_runtime.py`. The current phase runtime knows how to assign shards to workers, but it still assumes that the stage itself will call the model once per shard. Replace that assumption with explicit task manifests. Add a stage-agnostic task dataclass such as `WorkerTaskEntryV1` with fields for `task_id`, `task_kind`, `owned_ids`, `input_payload`, `validation_contract`, and stage-local metadata. Add a stage-agnostic result dataclass such as `WorkerTaskResultV1` with `task_id`, `status`, `output_path`, validation metadata, and failure metadata. The generic runtime should write immutable task files, launch one long-lived worker session per assignment, harvest any task outputs found under `out/`, validate them one by one, and leave unfinished tasks in a pending state for retry or resume.

The fourth milestone is to define the long-lived worker contract itself. The worker prompt should be short and durable. It should say, in plain English, that this is one bounded stage worker, that all needed work is listed in `assigned_tasks.json`, that the worker must process tasks in order unless a task file explicitly says otherwise, that each task file defines its owned IDs and output shape, that the worker should write one output file per completed task to `out/<task_id>.json`, and that it should continue until all tasks are complete or it cannot proceed. The prompt should forbid repository exploration, but it must explicitly permit reading all local task files and writing allowed local outputs. This is the key inversion from the current one-shot prompt model.

The fifth milestone is the line-role pilot in `cookimport/parsing/canonical_line_roles.py`. Replace “one shard = one Codex answer” with “one worker = many line-role tasks.” A line-role task should be a much smaller packet than a shard, such as one contiguous run of 50 to 120 atomic lines, with local context included in the task file itself. The deterministic planner should still build a larger shard for ownership and telemetry if that is useful, but the worker-visible work unit should be the task packet. Each task output file should contain only the rows for that packet. The stage should validate exact owned `atomic_index` coverage per task file, then join validated task outputs back into the stage-wide line-role predictions. Repair should become task-scoped retry by default, not a giant whole-shard repair prompt.

The sixth milestone is the knowledge migration in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. Knowledge already has chunk-level ownership, which fits the long-lived model well. Build task files that own a small number of nearby `chunk_id` values. The worker should refine or reject those chunks one task at a time and write normalized output files per task. The stage should validate per-`chunk_id` coverage and evidence references exactly as today, but now against file outputs rather than one returned blob. Keep the current knowledge artifact surface, but add task-level runtime artifacts so a reviewer can see which tasks a worker completed before a stop or retry.

The seventh milestone is the recipe migration in `cookimport/llm/codex_farm_orchestrator.py`. Recipe tasks should usually own one recipe or a very small nearby recipe group, never a huge bundle that requires one giant final answer. Each recipe task file should include the owned `recipe_id` values, the deterministic intermediate recipe object, and the allowed output shape for corrected structure and ingredient-step mapping. The worker writes one output file per task. The stage validates ownership per task and then promotes the corrected recipe outputs into the existing deterministic final assembly. This preserves the current recipe authority model while making Codex operate on a manifest of local files rather than one huge prompt blob.

The eighth milestone is retry, resume, and watchdog behavior. The current repair and watchdog-retry machinery is shaped around one-shot outputs. In the long-lived system, the default recovery path should be much simpler: if a worker is killed, copy back any already-written task outputs, validate them, mark those tasks complete, and requeue only the remaining pending tasks for another worker session. A task-level retry packet is still allowed for genuinely invalid outputs, but “repair the whole shard” should disappear as the normal recovery mechanism. This is one of the main practical wins of the long-lived design.

The ninth milestone is telemetry and docs. Update the phase summaries, prompt budget summaries, and progress messages so they speak in terms of worker sessions, assigned task count, completed task count, pending task count, resumed task count, and per-task validation failures. The docs in `docs/10-llm/10-llm_README.md` and `cookimport/llm/README.md` must stop claiming that each stage is one direct structured call per shard once the migration lands.

## Validation and Acceptance

Acceptance is behavior-first.

After implementation, a human inspecting a stage run should be able to see that one worker session processed multiple task files. The proof should be present on disk. Under a worker root there should be a manifest plus explicit task files and per-task outputs. A successful line-role run, for example, should no longer depend on one final `{"rows":[...]}` response for a whole shard. Instead, the stage should validate many smaller task outputs and join them.

If a worker is watchdog-killed mid-session, already-written local outputs must still be harvested, validated, and counted as completed work. Only the unfinished tasks should be retried. This is the most important operational acceptance criterion because it proves the system is no longer hostage to one giant final answer.

The docs must also stop telling a false story. `docs/10-llm/10-llm_README.md` and `cookimport/llm/README.md` should both describe the long-lived manifest-driven worker loop once all three stages are migrated.

The minimum expected command set for acceptance is:

    . .venv/bin/activate && pytest -q tests/llm/test_codex_exec_runner.py -k "direct_exec or workspace"
    . .venv/bin/activate && pytest -q tests/parsing/test_canonical_line_roles.py -k "worker or task or resume"
    . .venv/bin/activate && pytest -q tests/llm/test_prompt_preview.py

For broader confidence after stage migrations:

    ./scripts/test-suite.sh domain parsing
    ./scripts/test-suite.sh domain llm

Optional live-token proof, only with approval:

    cookimport labelstudio-benchmark ...  # exact command depends on the stage/benchmark under study

The acceptance observation from a live rerun is:

- worker-session count is smaller than task count
- completed-task artifacts appear under worker folders
- rerunning after a forced interruption resumes remaining tasks instead of repeating the entire stage

## Idempotence and Recovery

This migration should be implemented additively at first. Do not delete the old one-shot path until the long-lived path is proven in at least the line-role pilot and then migrated across knowledge and recipe. During the migration, stage selection or an internal runtime flag may temporarily choose the old or new path. Once all three stages are stable on the new path, remove the one-shot path instead of carrying two permanent systems.

Task files must be immutable and safe to regenerate from deterministic upstream artifacts. Output files must be written atomically, ideally by writing a temporary file and renaming it into place. Resume must be based on existing validated outputs, not on trusting an in-memory worker state. If a worker crashes, kill handling must copy back whatever was already written in the sterile workspace, then stage code should validate those outputs and requeue only the remaining tasks.

Do not delete benchmark or failure artifacts used as migration evidence. They are the proof that the old transport misfit was real.

## Interfaces and Dependencies

Preserve the top-level stage identities and deterministic promotion boundaries. Do not change the public pipeline IDs solely for this runtime migration.

Add a new direct-exec runner interface in `cookimport/llm/codex_exec_runner.py` alongside the existing structured prompt method. The concrete name may vary, but by the end of the migration there must be a worker-session method with semantics equivalent to:

    run_workspace_worker(
        *,
        prompt_text: str,
        working_dir: Path,
        env: Mapping[str, str],
        model: str | None = None,
        reasoning_effort: str | None = None,
        timeout_seconds: int | None = None,
        workspace_task_label: str | None = None,
        supervision_callback: Callable[[CodexExecLiveSnapshot], CodexExecSupervisionDecision | None] | None = None,
    ) -> CodexExecRunResult

Extend `cookimport/llm/phase_worker_runtime.py` with stable task-level data structures. By the end of the migration there must be stage-agnostic equivalents of:

    @dataclass(frozen=True)
    class WorkerTaskEntryV1:
        task_id: str
        task_kind: str
        owned_ids: tuple[str, ...]
        input_payload: Any
        metadata: dict[str, Any] = field(default_factory=dict)

    @dataclass(frozen=True)
    class WorkerTaskResultV1:
        task_id: str
        status: str
        output_path: str | None
        validation_errors: tuple[str, ...] = ()
        metadata: dict[str, Any] = field(default_factory=dict)

Each stage orchestrator must then implement three stage-local responsibilities on top of that generic runtime:

1. plan task packets from deterministic upstream artifacts
2. validate one task output at a time against exact owned IDs
3. join validated task outputs back into the existing stage authority model

Do not make the runner stage-aware. The runner owns process launch, workspace prep, event streaming, supervision, and workspace harvesting. Stage files own task planning, task validation, and promotion.
