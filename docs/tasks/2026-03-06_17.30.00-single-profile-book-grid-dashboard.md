# Single-profile book grid dashboard

## Problem statement

The shared spinner for interactive single-profile multi-book benchmarks rendered a queue-style list that cycled one task string at a time. In real CodexFarm runs that made it hard to see which worker belonged to which book, and there was no ETA per book.

## Acceptance criteria

- Multi-book single-profile benchmark status renders one column per selected book instead of a queue list.
- The shared dashboard shows per-book state, progress, and ETA.
- Worker activity is rendered as one row per worker slot across the book columns.
- Nested per-book benchmark calls still suppress their own spinners/summaries and drive only the shared dashboard.

## Verification steps

- Run:
  - `source .venv/bin/activate && pytest tests/labelstudio/test_labelstudio_benchmark_helpers_progress.py tests/labelstudio/test_labelstudio_benchmark_helpers_single_profile.py`
- Confirm the single-profile progress tests cover:
  - the book-column dashboard render,
  - worker-row rendering,
  - per-book ETA display,
  - shared-spinner integration for the interactive multi-book flow.

## Evidence

- Verification result:
  - `38 passed, 1 warning in 43.42s`

## Constraints and decision

- Constraint: the outer live panel already exists in `_run_with_progress_status(...)`, so the change should avoid inventing a second spinner layer.
- Decision: keep the outer panel unchanged and replace only the single-profile dashboard model. The new dashboard consumes raw nested benchmark callback messages and derives per-book columns, worker rows, and ETA locally.

## Rollback / compensation

- Safe rollback is to restore the previous `_AllMethodProgressDashboard` usage inside `_interactive_single_profile_all_matched_benchmark(...)` and revert the matching progress/single-profile tests.
