# Shrink `NonRecipeStageResult` To Explicit Owned Layers

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, `cookimport/staging/nonrecipe_stage.py` will stop pretending one mixed object can safely represent seed guesses, review routing, final authority, and review status all at once. A reader should be able to open the stage result type and see four explicit owned layers: seed, routing, authority, and review status. The live runtime already mostly behaves this way; this refactor makes the type match the behavior so future code cannot silently drift back into the old mixed contract.

You can see the change working by running the focused staging, LLM, and Label Studio tests that currently exercise the routing-only contract. The important observable proof is that `nonrecipe-route` still only builds a review queue and obvious-junk exclusions, while scoring and downstream projections still read only final authority.

## Outcomes & Retrospective

The implementation now matches the design the runtime had already been drifting toward. `cookimport/staging/nonrecipe_stage.py` exposes an explicit `NonRecipeSeedResult` and a thin `NonRecipeStageResult` container, and the live builders now construct `seed`, `routing`, `authority`, and `review_status` directly instead of synthesizing them from whichever flat fields happened to be present. That removes the main footgun: a caller can no longer omit fields and silently get a different authority story than they intended.

There is no generic compatibility shim left. `tests/nonrecipe_stage_helpers.py` now contains only explicit builders for the owned test-layer objects so test code can stay readable without recreating the old mixed-field semantics. The broader staging and LLM suites passed after this cutover, which is a useful signal that the live runtime was already consuming the owned layers correctly and mostly needed the type definition to catch up.

## Context and Orientation

The relevant runtime lives in `cookimport/staging/nonrecipe_stage.py`. That module now defines:

- `NonRecipeSpan`, a contiguous outside-recipe block range with a normalized category and stable block ids.
- `NonRecipeRoutingResult`, the routing-only output of `nonrecipe-route`: review queue, obvious-junk exclusions, preview text, and warnings.
- `NonRecipeAuthorityResult`, the final machine-readable truth for outside-recipe `knowledge` versus `other`.
- `NonRecipeReviewStatusResult`, the process-status view over reviewed and still-unreviewed review-eligible rows.
- `NonRecipeSeedResult`, the seed outside-recipe view immediately after label-first normalization.
- `NonRecipeStageResult`, a thin container that holds `seed`, `routing`, `authority`, `review_status`, and `refinement_report`.

The important downstream readers are:

- `cookimport/staging/pipeline_runtime.py`, which runs `nonrecipe-route` and `knowledge-final`.
- `cookimport/staging/writer.py`, which writes `08_nonrecipe_seed_routing.json`, `09_nonrecipe_authority.json`, and `09_nonrecipe_review_status.json`.
- `cookimport/staging/stage_block_predictions.py`, which must read only final authority for scored `KNOWLEDGE`.
- `cookimport/llm/knowledge_stage/runtime.py`, which must plan work from the routing queue, not from seed semantic maps.
- `cookimport/labelstudio/ingest_flows/artifacts.py`, which projects final authority and suppresses unreviewed seed guesses in scoring views.

The main hazard is reintroducing a broad “all categories at once” map as if it were authoritative. This refactor succeeds only if the type system now forces the reader to choose the owned layer they mean.

## Plan of Work

First, replace the broad `NonRecipeStageResult` definition with a thin container and a new `NonRecipeSeedResult`. Seed will hold the outside-recipe rows exactly as they came out of label-first normalization: seed spans and the seed `knowledge|other` map. Routing, authority, and review-status will remain the only owned downstream layers. The new container will expose only those four layer objects plus the refinement report. The old `__post_init__` synthesis must be deleted entirely so omitted fields cannot quietly mean something different depending on constructor shape.

Second, move the derivation work into named builders inside `cookimport/staging/nonrecipe_stage.py`. `build_nonrecipe_stage_result(...)` will build seed, routing, exclusion-only authority, and review status explicitly from label-first inputs. `refine_nonrecipe_stage_result(...)` will keep the original seed and routing, build final authority from reviewed blocks plus deterministic exclusions, and build review status from the remaining unreviewed review-eligible rows. Helper functions should build spans and category maps from one ownership layer at a time; they must not recreate the old broad mixed contract.

Third, update the live readers that still reach through old flat fields. `cookimport/llm/knowledge_stage/runtime.py` and `cookimport/staging/writer.py` still need seed facts for metrics or block-id lookup; those should read `stage_result.seed`. Any code that needs final truth should continue reading `stage_result.authority`. Any code that needs incompleteness should continue reading `stage_result.review_status`.

Fourth, update tests so they construct explicit stage results instead of relying on the old mixed constructor. The staging, knowledge, prompt-preview, fake-codex, and Label Studio tests currently create many tiny stage-result fixtures. Those fixtures should be switched to explicit owned-layer builders in `tests/nonrecipe_stage_helpers.py`, not to the broad historical constructor shape inside production code.

Finally, update the staging docs if the public contract wording needs to become more precise now that the in-memory type has caught up with the live runtime story, then run focused verification and record the exact commands and results here.

## Validation and Acceptance

The refactor is correct only if all of the following are true.

`NonRecipeStageResult` has no field-level synthesis. A reader should be able to instantiate it only with explicit `seed`, `routing`, `authority`, and `review_status` objects.

`build_nonrecipe_stage_result(...)` still routes obvious-junk exclusions directly to final `other`, keeps surviving rows in the review queue, and does not manufacture final semantic authority for the review queue.

`refine_nonrecipe_stage_result(...)` keeps the original seed/routing view intact while moving reviewed decisions into final authority and leaving still-unreviewed seed guesses out of the authority map.

`stage_block_predictions.py` still labels outside-recipe `KNOWLEDGE` only from final authority, and focused tests still prove unreviewed review-eligible rows stay `OTHER` for scoring.

The focused staging, LLM, and Label Studio tests listed above must pass inside `.venv`.

## Idempotence and Recovery

This refactor is safe to rerun because it is an in-repo contract cleanup, not a data migration. If a test fixture update goes wrong, the safe recovery path is to keep the new explicit production type and repair only the failing test helper or fixture. Do not reintroduce the old mixed `__post_init__` synthesis as a shortcut.
