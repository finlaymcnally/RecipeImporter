## Context and Orientation

The line-role pipeline starts in `cookimport/parsing/canonical_line_roles.py`. It labels each atomic text line with a canonical role such as `RECIPE_TITLE`, `INGREDIENT_LINE`, or `OTHER`. Today that model is `CanonicalLineRolePrediction`, which carries a label, decision provenance (`rule`, `codex`, or `fallback`), reason tags, explicit escalation reasons, and three scalar scores: `confidence`, `trust_score`, and `escalation_score`.

The label-first refactor converts those predictions into authoritative staged artifacts in `cookimport/parsing/label_source_of_truth.py`. That module defines `AuthoritativeLabeledLine`, `AuthoritativeBlockLabel`, and `RecipeSpan`, then writes them into stage output through `cookimport/staging/import_session.py`. Reviewer-oriented exports in `cookimport/bench/followup_bundle.py`, `cookimport/bench/cutdown_export.py`, and `scripts/benchmark_cutdown_for_external_ai.py` currently still copy or threshold the same score fields.

This plan deletes the scalar fields from those types and payloads. The replacement contract is simple: decisions are explained by `decided_by`, `reason_tags`, and `escalation_reasons` only. Any logic that previously compared numbers must instead look for explicit reasons or deterministic conditions.

## Plan of Work

Start in `cookimport/parsing/canonical_line_roles.py`. Remove the score fields from `CanonicalLineRolePrediction`, delete score normalization helpers and threshold-based escalation helpers, and refactor `_apply_prediction_decision_metadata(...)` so it only enriches `escalation_reasons`. The new `_should_escalate_candidate(...)` should be reason-based: unresolved, sanitized, fallback, disagreement, or veto conditions may escalate; ordinary structured labels should not escalate just because a numeric score was below a threshold because that score no longer exists.

Next update `cookimport/parsing/label_source_of_truth.py` and `cookimport/parsing/recipe_span_grouping.py` so authoritative line, block, and span models only carry labels and explicit reasons. Remove score aggregation from block/span construction. Then update `cookimport/staging/import_session.py` and any label projection surfaces to stop writing the deleted fields into JSON and JSONL artifacts.

After the runtime types are clean, update downstream reviewer/export code. `cookimport/bench/followup_bundle.py`, `cookimport/bench/cutdown_export.py`, and `scripts/benchmark_cutdown_for_external_ai.py` should stop reading or emitting the deleted score fields and should derive “uncertainty” or follow-up packets solely from explicit escalation reasons when that behavior still exists.

Finally update the narrow docs that currently describe the label-first seam as score-carrying, add one new understanding note under `docs/understandings/`, and rewrite the affected tests so they assert the new score-free contract. The test edits should prove both model shape changes and behavioral changes in the escalation gate.
