## Purpose / Big Picture

After this change, a benchmark review no longer stops at Oracle's first answer. A first-pass Oracle review will be able to ask for narrow follow-up evidence in its normal chat response, a local Codex-driven seam in this repository will turn that request into a deterministic `followup_dataN/` packet, and Oracle will send a second message into the same ChatGPT conversation carrying that follow-up packet so the same Pro model can refine its analysis.

The user-visible proof is simple. Starting from an existing `upload_bundle_v1/`, an operator will run one recipeimport command, see a bundle-local follow-up workspace created beside the existing Oracle launch artifacts, and then see a second Oracle run linked to the first one. The second Oracle run must preserve the existing ChatGPT conversation URL and must produce a new answer that explicitly analyzes the follow-up data instead of restarting from scratch.

## Surprises & Discoveries

- Observation: The follow-up packet generation seam is already strong. `cookimport/bench/followup_bundle.py` and `cookimport/cf_debug_cli.py` already implement `cf-debug request-template`, `select-cases`, `build-followup`, and `pack`, so the missing work is the bridge from Oracle prose to those existing packet builders.
  Evidence: `write_followup_request_packet(...)` already writes a full additive follow-up packet rooted at `cf.followup_packet.v1`.

- Observation: Oracle's current browser session support stops at reattach and read. It can reopen a stored conversation and capture the assistant answer, but it does not currently expose a CLI surface that posts a second prompt into that same chat after the first run has completed.
  Evidence: `oracle session --help` lists only inspection/attach flags, and `/home/mcnal/projects/shared/oracle-dev/package/dist/src/browser/reattach.js` waits for assistant output without any prompt submission step.

- Observation: Oracle already persists enough state to make same-chat turn 2 practical. It stores `conversationUrl`, `conversationId`, and browser runtime hints in the local session metadata.
  Evidence: `/home/mcnal/projects/shared/oracle-dev/package/dist/src/browser/sessionRunner.js` returns `conversationUrl` and `conversationId`, and `/home/mcnal/projects/shared/oracle-dev/package/dist/src/cli/sessionDisplay.js` surfaces the saved conversation URL.

- Observation: Same-chat continuation did not need a second browser executor. Reusing the normal browser run path with `browserConfig.url` pointed at the saved conversation URL was enough.
  Evidence: The live Instant smoke created `hello-world-url-test-turn-2`, linked it back to `hello-world-url-test`, kept the same `conversationUrl`, and returned `hello again`.

- Observation: The detached benchmark uploader had a second missing seam beyond turn-2 orchestration. It also never came back to finalize the saved turn-1 status from the completed Oracle log, so successful turn-1 runs could remain stuck at `running` forever in `oracle_upload_status.json`.
  Evidence: The `2026-03-19_17.13.29` run for session `you-are-reviewing-a-benchmark-290` had a complete `Answer:` block in `oracle_upload.log`, but its saved status file still said `running` because the original detached parent wrote status only once near launch time.

- Observation: Automatic turn 2 can only be as good as the answer Oracle actually persisted. Some completed benchmark sessions still save only a short summary answer with no `Requested follow-up data` block, so recipeimport must fail explicitly instead of silently pretending turn 2 was skipped.
  Evidence: After the worker repaired `you-are-reviewing-a-benchmark-290`, the saved turn-1 status became `succeeded`, but both `oracle_upload.log` and `oracle session you-are-reviewing-a-benchmark-290` still contained only a two-sentence 55-token answer rather than the structured four-section response needed for parsing.

- Observation: Oracle `assistant-timeout` recovery has two fragile assumptions that do not hold for large benchmark chats: it preferred sidebar re-discovery even when a full saved `/c/...` URL existed, and it treated the current whole-thread turn count as the lower bound for the answer it was trying to recapture.
  Evidence: Session `you-are-reviewing-a-benchmark-294` kept timing out even though the saved session metadata already contained `https://chatgpt.com/c/69bca0a6-7284-832f-b221-935958d3062b`, and manual reattach logs showed visible assistant text in the conversation snapshot while the reattach path still raised `Unable to capture assistant response`.

## Decision Log

- Decision: Keep Oracle's first answer as normal human-readable Markdown, but make the `Requested follow-up data` section follow a stricter plain-text subformat instead of asking Oracle to emit raw JSON.
  Rationale: The user explicitly wants the follow-up request to arrive as text in the chat. A stricter Markdown subformat remains readable in ChatGPT while still being deterministic enough for a local parser to turn into `cf.followup_request.v1`.
  Date/Author: 2026-03-19 / Codex

- Decision: Implement same-conversation turn 2 in Oracle itself, not as a recipeimport browser hack.
  Rationale: Recipeimport already owns benchmark packaging and follow-up packet generation, but the act of reopening an existing ChatGPT conversation and sending another prompt belongs to Oracle's browser automation layer. Keeping that logic in Oracle avoids duplicating fragile browser code in this repository.
  Date/Author: 2026-03-19 / Codex

- Decision: Record turn 2 as a new linked local Oracle session, even though the browser submission occurs in the same ChatGPT conversation.
  Rationale: Oracle's current session store is one-prompt/one-answer oriented. Mutating the original session into a multi-turn transcript would complicate existing status, rendering, and summary paths. A new local session entry with `continued_from_session_id`, `root_conversation_id`, and the same `conversation_url` preserves observability without breaking the current model.
  Date/Author: 2026-03-19 / Codex

- Decision: Keep the implementation solo-project simple. There will be no daemon, queue, service, or persistent scheduler. The whole loop remains file-backed and operator-driven from local CLI commands.
  Rationale: The project is explicitly solo-operated. A single command plus bundle-local artifacts is easier to understand, test, and repair than a background coordinator.
  Date/Author: 2026-03-19 / Codex

- Decision: For the detached interactive benchmark flow, use one extra short-lived local worker process rooted in the source run directory instead of trying to make the original turn-1 Oracle subprocess also own turn 2.
  Rationale: The original detached launcher already exists and already writes the turn-1 staging and metadata layout. A second local watcher process can finalize that saved turn-1 status and then reuse the same `cookimport bench oracle-followup` bridge without introducing a service, queue, or cross-run scheduler.
  Date/Author: 2026-03-19 / Codex

- Decision: Implement Oracle continuation by retargeting the normal browser runner to the saved conversation URL instead of inventing a second browser-only continuation engine.
  Rationale: The saved browser session metadata already contains enough information to reopen the same chat. Reusing the normal browser path kept the Oracle patch small and made `continue-session` a generic CLI feature rather than a recipeimport special case.
  Date/Author: 2026-03-19 / Codex

## Context and Orientation

The benchmark review flow currently ends at a first Oracle answer. In this repository, the relevant benchmark handoff lives under an existing `upload_bundle_v1/` directory. That directory is a compact benchmark evidence package containing `upload_bundle_overview.md`, `upload_bundle_index.json`, and `upload_bundle_payload.jsonl`. Recipeimport already knows how to send that package to Oracle through `cookimport bench oracle-upload`, which is wired in [cookimport/cli.py](/home/mcnal/projects/recipeimport/cookimport/cli.py) and implemented in [cookimport/bench/oracle_upload.py](/home/mcnal/projects/recipeimport/cookimport/bench/oracle_upload.py).

The current first-pass Oracle review prompt is stored as editable text in [llm_pipelines/prompts/benchmark.oracle-upload.prompt.md](/home/mcnal/projects/recipeimport/llm_pipelines/prompts/benchmark.oracle-upload.prompt.md). That prompt already tells Oracle to include a `Requested follow-up data` section when the initial bundle is insufficient. Today, however, that section is only advice for a human reader. There is no code in recipeimport that extracts it, turns it into a machine-readable request manifest, or packages the requested evidence automatically.

The deterministic follow-up packet surface already exists. In [cookimport/bench/followup_bundle.py](/home/mcnal/projects/recipeimport/cookimport/bench/followup_bundle.py), the schema version `cf.followup_request.v1` defines a follow-up request manifest, and `write_followup_request_packet(...)` turns that manifest into an additive `followup_dataN/` packet. The CLI wrapper for those helpers lives in [cookimport/cf_debug_cli.py](/home/mcnal/projects/recipeimport/cookimport/cf_debug_cli.py). The important command is `cf-debug build-followup --bundle <upload_bundle_v1> --request <cf.followup_request.v1 json> --out <dir>`, which already produces a deterministic packet rooted at `cf.followup_packet.v1`.

Oracle itself is no longer owned only by the installed npm folder. The editable local Oracle workspace is `/home/mcnal/projects/shared/oracle-dev/`, and the active package mirror is `/home/mcnal/projects/shared/oracle-dev/package`. The browser CLI entrypoint is `/home/mcnal/projects/shared/oracle-dev/package/dist/bin/oracle-cli.js`. Browser session execution lives in `/home/mcnal/projects/shared/oracle-dev/package/dist/src/cli/sessionRunner.js` and `/home/mcnal/projects/shared/oracle-dev/package/dist/src/browser/sessionRunner.js`. Browser reattach logic lives in `/home/mcnal/projects/shared/oracle-dev/package/dist/src/browser/reattach.js`. Session display and status rendering live in `/home/mcnal/projects/shared/oracle-dev/package/dist/src/cli/sessionDisplay.js`.

An "Oracle session" in this plan means Oracle's local saved metadata entry under `~/.local/share/oracle/sessions/<session_id>/`. That session metadata already records the ChatGPT conversation URL and conversation ID for browser runs. A "follow-up packet" means a new local directory such as `followup_data1/` containing only the additional evidence the reviewer asked for, not a replacement `upload_bundle_v1`. A "turn 2" means a second user message sent into the same ChatGPT conversation after Oracle's first answer, using the requested follow-up packet as attachments and asking for updated analysis.

The missing seam is now precise. Recipeimport can already generate follow-up packets, and Oracle can already open and inspect an existing conversation URL, but nothing currently joins those two capabilities into a turn-2 loop.

## Validation and Acceptance

Acceptance is behavioral, not structural.

First, when Oracle turn 1 answers with a non-`None` `Requested follow-up data` section, recipeimport must be able to parse that text and write a valid `cf.followup_request.v1` manifest without hand-editing. The deterministic parser tests must include at least one fully specified request, one partially specified request that falls back to template defaults, and one `None` case.

Second, `cookimport bench oracle-followup <upload_bundle_v1> --from-run latest --dry-run` must succeed without tokens. It must create a new bundle-local launch directory under `.oracle_upload_runs/<timestamp>/`, write the extracted turn-1 answer, the normalized follow-up manifest, the generated `followup_data1/` packet, and the turn-2 prompt preview, then print the exact Oracle continuation command it would run.

Third, the Oracle continuation command must prove same-chat behavior. A live Instant-model smoke must create a new Oracle local session whose metadata links back to the prior session and whose `conversation_url` matches the source session's URL. The new answer must appear as a later user/assistant exchange inside the same ChatGPT conversation, not as a fresh conversation.

Fourth, the real benchmark loop must remain observable from the bundle folder. After a non-dry-run `cookimport bench oracle-followup ...`, the new `.oracle_upload_runs/<timestamp>/` directory must contain the normal Oracle launch artifacts plus follow-up-specific metadata: source session ID, continued session ID, conversation URL, follow-up request manifest path, and follow-up packet path. `cookimport` should still print the usual reattach command and log path.

Fifth, the model policy must remain split exactly as requested. All automated tests and smokes must use the test lane through `ORACLE_TEST_MODEL`. All genuine Oracle benchmark analysis, including real turn-2 review of benchmark evidence, must default to the genuine Pro lane through `ORACLE_GENUINE_MODEL`.

## Idempotence and Recovery

All recipeimport follow-up preparation steps must be safe to rerun. Every invocation should create a fresh `.oracle_upload_runs/<timestamp>/` directory and must never overwrite a previous follow-up packet or request manifest. If the Oracle continuation command fails before sending, the generated follow-up workspace should remain on disk so the operator can inspect the prompt, the manifest, and the packet.

If Oracle continuation succeeds but the local wrapper loses the answer, the bundle-local metadata must still record the new Oracle session ID and conversation URL so `oracle session <id>` can recover the result. If the parsed follow-up request is incomplete, the bridge should stop before sending and write both a machine-readable validation report and a Codex-facing handoff file that says which fields could not be normalized. The operator or Codex can then supply `--request-file <path>` with an edited manifest and rerun the same command safely.

The new Oracle continuation command must refuse to run when the source session has no saved `conversationUrl` or `conversationId`, because silently starting a fresh chat would violate the purpose of the feature. That failure should be explicit and should point the operator at the saved session metadata path.
