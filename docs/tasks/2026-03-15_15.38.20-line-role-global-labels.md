# Remove Line-Role Candidate Labels Entirely

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

Canonical line-role originally had per-row candidate-label shortlists that could block Codex from returning the right label. The first pass removed that prompt/parser restriction. The second pass removes the remaining candidate-label machinery entirely from the active line-role path so parsing models, exports, upload bundles, docs, and tests all describe the same simpler contract: Codex gets the full label vocabulary, and current artifacts do not emit shortlist metadata.

## Context and Orientation

Line-role atomization starts in `cookimport/parsing/recipe_block_atomizer.py`. Deterministic and Codex-assisted labeling lives in `cookimport/parsing/canonical_line_roles.py`. Benchmark export surfaces live in `cookimport/bench/cutdown_export.py`, while upload-bundle rendering for existing outputs lives in `scripts/benchmark_cutdown_for_external_ai.py`.

The obsolete behavior used to span all of those layers:

- atomizer rows carried `candidate_labels`,
- Codex prompt rows and parser logic enforced them,
- predictions re-emitted them,
- joined-line exports and upload bundles analyzed them,
- tests/docs kept asserting on them long after the user-facing restriction was gone.

This plan removes the rest of that chain so the active system no longer mentions candidate labels outside historical notes.

## Plan of Work

Edit tests and active docs so they assert against `rule_tags`, confidence summaries, and actual emitted artifact fields instead of deleted candidate-label metadata.
