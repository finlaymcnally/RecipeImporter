# Make stage always run through source jobs

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

After this change, `cookimport` should have one runtime story for source processing:

1. plan one or more source jobs
2. execute those jobs
3. merge the job outputs into one whole-book bundle
4. run the shared semantic pipeline once on that merged book

That story should be true even when the source is small and only produces one job. “Unsplit” should stop being a separate architectural mode. A one-job run is just the smallest possible job plan.

The user-visible gain is not a new feature. The gain is that the runtime becomes teachable and debuggable. When a book behaves strangely, the human should no longer have to ask whether they are looking at the direct single-file path or the split-job path. There should only be one path.

This plan is the runtime counterpart to the adjacent simplification work:

- [2026-03-23_10.44.32-redesign-importers-as-source-normalizers.md](/home/mcnal/projects/recipeimport/docs/plans/2026-03-23_10.44.32-redesign-importers-as-source-normalizers.md)
- [2026-03-23_10.59.16-simplify-stage-authority-into-five-stage-pipeline.md](/home/mcnal/projects/recipeimport/docs/plans/2026-03-23_10.59.16-simplify-stage-authority-into-five-stage-pipeline.md)

Those plans simplify what jobs produce and what happens after merge. This plan simplifies the runtime spine in the middle.

## Non-Goals

This plan does not by itself redesign importer semantics. It does not decide whether importers emit recipes or only source blocks; that work belongs to the source-normalizer plan.

This plan does not by itself redesign the post-merge semantic stages; that work belongs to the five-stage pipeline plan.

This plan does not require model-spend proof or live benchmark runs without explicit user approval. Local deterministic tests and cheap fixture runs are the default acceptance path.

This plan does not require every source type to support ranged slicing. Text, spreadsheet, schema, and export inputs may still produce exactly one whole-source job. The point is one runtime shape, not fake page ranges where none exist.

## Outcomes & Retrospective

The runtime cut landed. Stage now has one source-processing story: `plan_source_jobs(...)`, `execute_source_job(...)`, `_merge_source_jobs(...)`, then one shared semantic session. The direct `stage_one_file()` path is gone, and one-job sources now write `.job_parts/.../job_0/raw/...` and merge through the same book-assembly seam as multi-job PDF/EPUB runs.

The most important outcome is that “whole source” versus “split source” is no longer an architectural fork. It is only a difference in planned job count. Label Studio prediction planning now shares the same planner module, and the run-settings worker-resolution helper moved to that same planning owner so future split-capability changes are single-sourced.

Verification stayed cheap and local as intended. `./scripts/test-suite.sh domain cli` passed, `./scripts/test-suite.sh domain labelstudio` passed after a small unrelated prompt-budget drift fix, and targeted raw pytest slices covering the new planner/merge/worker seams passed inside `.venv`.

## Context and Orientation

The stage runtime now has one story. [cookimport/staging/job_planning.py](/home/mcnal/projects/recipeimport/cookimport/staging/job_planning.py) owns `JobSpec`, page/spine inspection, and the shared planner. [cookimport/cli.py](/home/mcnal/projects/recipeimport/cookimport/cli.py) always plans jobs for every source, dispatches [cookimport/cli_worker.py](/home/mcnal/projects/recipeimport/cookimport/cli_worker.py)`:` `execute_source_job(...)` for each job, and then calls `_merge_source_jobs(...)` once the full job set for that file returns.

`execute_source_job(...)` is intentionally bounded work. It runs importer conversion for the planned range or whole source, writes raw artifacts into `.job_parts/<workbook_slug>/job_<index>/raw/...`, clears `result.raw_artifacts`, and returns a mergeable partial payload. It does not write final stage outputs or enter the semantic session.

`_merge_source_jobs(...)` is the first whole-book seam. It sorts job payloads, offsets block indices, rebuilds merged `full_text.json` when present, reassigns recipe IDs, applies `--limit`, and then calls `execute_stage_import_session_from_result(...)` exactly once on the merged bundle. That is true even when the planner emitted only one whole-source job.

The target runtime should be:

- planning always happens
- execution always happens through job records
- merge always happens, even if there is only one job
- semantic processing always happens once, after merge

That target should hold not just for `cookimport stage`, but for sibling flows that rely on the same source-processing reality, especially Label Studio ingestion and benchmark prediction planning. They do not all need to share every helper immediately, but they should converge on one planning and job contract rather than carrying parallel storylines.

The cleanest mental model is:

- a source job is one bounded conversion unit
- a bounded conversion unit may cover a whole source or a range inside it
- job execution produces mergeable extraction payloads, not final book truth
- merge produces the first whole-book bundle
- the shared semantic pipeline interprets the merged book exactly once

## Milestones

### Milestone 1: Create one shared source-job planner and make it universal

At the end of this milestone, the codebase will have one planner module that emits job records for all source-processing callers. PDF and EPUB files may produce many ranged jobs. Other importers may produce exactly one whole-source job. The planner should stop encoding “split” as a special mode and instead encode optional range metadata on an otherwise uniform job shape.

Acceptance is that stage, Label Studio, and benchmark-related callers can all ask one planner for jobs and receive a list whose length may be one or many without changing the caller’s orchestration model.

### Milestone 2: Replace direct single-file execution with one universal job executor

At the end of this milestone, worker code will have one generic source-job execution function that accepts a planned job and returns one mergeable payload. The executor may still branch internally for page-range, spine-range, or whole-source importer arguments, but those branches should live inside one worker path rather than as separate top-level runtime modes.

Acceptance is that `stage_one_file()` is gone, not hidden behind a compatibility wrapper, and caller code no longer decides between “single-file stage” and “split job.”

### Milestone 3: Make merge universal and turn `_merge_split_jobs()` into the normal book assembly path

At the end of this milestone, every stage run will feed job results into one merge function. That merge function should stop thinking of itself as a split-only special case. It should assemble one whole-book conversion bundle from one-or-many job payloads, write the merged raw artifacts and merged `full_text.json`, then enter the shared semantic session once.

Acceptance is that a one-job run and a many-job run both go through the same merge-time assembly logic before calling the semantic session.

### Milestone 4: Delete split-only names and mode-shaped assumptions from the runtime

At the end of this milestone, the runtime will stop using names and comments that imply a special split branch where none exists anymore. Temporary roots like `.job_parts` can stay if renaming them is noisy, but comments, docs, and helper names should describe source jobs or slices rather than “the split path.” If a rename is cheap and clarifying, take it. If it is noisy, leave a small note and move on.

Acceptance is that a new reader can open the stage runtime and see one source-job path instead of two competing stories.

### Milestone 5: Update docs and tests so the simplified story is the only documented story

At the end of this milestone, the architecture, ingestion, and any flow-specific docs will describe one source-job runtime. Focused tests will cover one-job whole-source planning, multi-job PDF planning, multi-job EPUB planning, markitdown whole-book EPUB planning, and merge-before-semantics invariants.

Acceptance is passing focused tests plus docs that no longer teach “split versus unsplit” as a user-facing architecture decision.

## Plan of Work

Start at the planning seam. The older planner-extraction work already identified the right boundary: planning should move into a shared module under `cookimport/staging/` and stop living separately in CLI and Label Studio code. The difference now is scope. The planner should not merely decide “split or not.” It should become the required first step for every source-processing run.

Once planning is shared, change worker execution. Replace the trio of `stage_one_file()`, `stage_pdf_job()`, and `stage_epub_job()` with one job executor that accepts a job record plus run settings and returns a mergeable partial result. The executor may still pass `start_page`, `end_page`, `start_spine`, and `end_spine` into `_run_import(...)` when present, but the caller should no longer branch on runtime mode.

After that, simplify merge. `_merge_split_jobs()` should be renamed or reshaped into a universal book-assembly seam that accepts ordered job results, writes merged raw artifacts, normalizes merged block order, rebases block indices, reassigns recipe IDs as needed, and then calls the shared semantic session once. This is also the right seam to align with the source-normalizer plan, because it is where merged canonical source blocks should eventually become primary.

Then clean up the stage command. `cookimport stage` should always:

1. resolve effective workers
2. plan jobs
3. execute planned jobs
4. merge job results
5. write run summaries

No direct single-file semantic path should remain.

With stage simplified, update sibling flows that still depend on parallel split-planning logic. Label Studio and benchmark code do not need a giant rewrite, but they do need to stop carrying private assumptions about split-only behavior if this repo is going to have one clean runtime story.

Finally, update docs and remove stale language. The architecture and ingestion docs should say plainly that the runtime always processes sources through planned jobs, and that a one-job run is the normal degenerate case of the same machinery.

This should be implemented as a hard cut, not as a long coexistence period. Do the narrowest temporary scaffolding needed to keep the repo working while a change set is in flight, but do not leave behind runtime flags, compatibility shims, dual callers, or “legacy path” escape hatches in the checked-in result.

## Validation and Acceptance

Acceptance is that the codebase now has one source-processing runtime, not two.

The first acceptance criterion is structural deletion. `stage_one_file()` must be gone, and caller code must no longer branch between direct single-file processing and split-job processing.

The second acceptance criterion is semantic consistency. A one-job run and a many-job run must both reach the semantic session only after job-result merge and whole-book assembly.

The third acceptance criterion is planner clarity. Planning must live in one shared owner that stage and sibling flows use, and “one job” must be expressed as planner output rather than as the absence of planning.

The fourth acceptance criterion is debuggability. A human inspecting artifacts should be able to understand every run with the same story: planned jobs produced partial extraction payloads, those payloads were merged, then one semantic pass happened.

The fifth acceptance criterion is documentation. The architecture and ingestion docs must teach source jobs as the only runtime story and must stop presenting split-versus-unsplit as a meaningful architecture distinction.

The sixth acceptance criterion is no escape hatch. There must be no legacy setting, compatibility toggle, or preserved old-path helper that lets the runtime fall back to the direct pre-merge path.

## Idempotence and Recovery

This refactor can land in slices, but the checked-in end state must not preserve both runtimes. Do not leave compatibility shims, runtime toggles, or legacy adapters in the repo after the work lands.

If one-job runs and many-job runs diverge during migration, write a narrow test that captures the divergence before deciding which behavior is correct. The goal is not merely to preserve old behavior. The goal is to make the remaining behavior understandable.

If sibling flows such as Label Studio lag behind the stage cleanup, keep the shared planner authoritative and finish their migration rather than preserving a second runtime story. Do not reintroduce duplicate planning logic and do not ship a permanent dual-path compromise.

## Interfaces and Dependencies

The target runtime interface is conceptually:

    jobs = plan_source_jobs(...)
    job_results = [execute_source_job(job, ...) for job in jobs]
    merged_book = merge_source_job_results(job_results, ...)
    session = run_stage_import_session(merged_book, ...)

The exact function names may differ, but the ownership may not:

- planning owns job creation
- job execution owns bounded source conversion only
- merge owns whole-book assembly
- the semantic session owns recipe/nonrecipe/knowledge interpretation after merge
