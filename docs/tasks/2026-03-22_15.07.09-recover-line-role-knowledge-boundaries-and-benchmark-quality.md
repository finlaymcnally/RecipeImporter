## Purpose / Big Picture

The current Salt Fat canonical-text benchmark is still too far from the practical target because most remaining misses are not recipe extraction failures. They are line-role boundary mistakes on outside-recipe text, especially the boundary between reusable cooking lesson material labeled `KNOWLEDGE` and surrounding text labeled `OTHER`. After this change, the line-role stage should keep genuine teaching prose and lesson headings as `KNOWLEDGE`, keep memoir, front matter, exhortation, and navigation as `OTHER`, and preserve obvious recipe-local instructions without overpromoting prose into structure. The visible proof is a materially better Salt Fat benchmark run, with the dominant `KNOWLEDGE -> OTHER` and `OTHER -> KNOWLEDGE` buckets shrinking while the already-recovered variant/title/instruction fixes stay intact.

This plan is intentionally broader than the earlier targeted Salt Fat line-role cleanup notes in `docs/tasks/2026-03-22_12.43.40-recover-line-role-quality-after-saltfat-12.08.53.md` and `docs/tasks/2026-03-22_11.32.34-stop-generic-to-make-steps-from-leaking-into-recipe-variant.md`. Those documents remain useful historical context, but this ExecPlan restates the current problem in full so a new contributor can execute it without reading anything else.

## Context and Orientation

The active line-role implementation lives in [cookimport/parsing/canonical_line_roles.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles.py). In this repository, “line role” means the label assigned to each canonical text line before those lines are grouped back into recipe spans and before the optional knowledge stage reviews the harder outside-recipe borderline rows that still remain as `OTHER`. The important labels for this plan are `KNOWLEDGE`, `OTHER`, `RECIPE_TITLE`, `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `RECIPE_NOTES`, `RECIPE_VARIANT`, and `HOWTO_SECTION`.

The line-role system has three practical layers. The first layer is packet planning and worker guidance, which decides whether a bundle of lines looks like `lesson_prose`, `front_matter_navigation`, `recipe_body`, `recipe_adjacent_notes`, `memoir_front_matter`, or `mixed_boundaries`. Those packet modes are built in `canonical_line_roles.py` and rendered into worker prompts through [cookimport/llm/canonical_line_role_prompt.py](/home/mcnal/projects/recipeimport/cookimport/llm/canonical_line_role_prompt.py) plus the template files [llm_pipelines/prompts/canonical-line-role-v1.prompt.md](/home/mcnal/projects/recipeimport/llm_pipelines/prompts/canonical-line-role-v1.prompt.md) and [llm_pipelines/prompts/line-role.canonical.v1.prompt.md](/home/mcnal/projects/recipeimport/llm_pipelines/prompts/line-role.canonical.v1.prompt.md). The second layer is deterministic postprocessing in `canonical_line_roles.py`, where helper functions allow or reject labels, rescue obvious misses, and demote unsafe promotions. The third layer is regression coverage in [tests/parsing/test_canonical_line_roles.py](/home/mcnal/projects/recipeimport/tests/parsing/test_canonical_line_roles.py), which already includes prompt rendering, packet-context, deterministic, and Codex-side tests for this stage.

The current benchmark evidence for this plan is under `data/golden/benchmark-vs-golden/2026-03-22_14.44.46/single-book-benchmark/saltfatacidheatcutdown/`. The most important files are:

    single_book_summary.md
    codexfarm/eval_report.json
    codexfarm/line-role-pipeline/wrong_label_lines.sample.jsonl
    codexfarm/line-role-pipeline/line_role_flips_vs_baseline.sample.jsonl
    upload_bundle_v1/upload_bundle_overview.md

The current failure pattern is visible directly in those files. Many lesson headings and explanatory prose rows are gold `KNOWLEDGE` but predicted `OTHER`. A smaller but real set of action-oriented or reflective prose rows are gold `OTHER` but predicted `KNOWLEDGE`. The plan therefore needs to sharpen the positive `KNOWLEDGE` cues and the negative memoir/front-matter/advice cues at the same time.

The most important active code seams are these functions in `canonical_line_roles.py`: `_line_role_default_posture(...)`, `_line_role_strong_signals(...)`, `_line_role_flip_policy(...)`, `_looks_knowledge_heading_with_context(...)`, `_looks_knowledge_prose_with_context(...)`, `_outside_recipe_knowledge_label_allowed(...)`, `_looks_recipe_title_with_context(...)`, `_instruction_line_label_allowed(...)`, `_instruction_line_fallback_label(...)`, `_variant_label_allowed(...)`, and `_variant_fallback_label(...)`. If the worker contract changes materially, the matching prompt expectations in `tests/parsing/test_canonical_line_roles.py` must also change.

## Milestones

### Milestone 1: Freeze the current benchmark miss families into exact regressions

At the end of this milestone, the exact Salt Fat rows that motivate this work are represented in deterministic and Codex-side tests, so the implementation can be changed safely without guessing. The work is entirely local: no benchmark rerun is needed yet. The result should be a small battery of tests that fail against the current behavior for the right reasons and then pass only when the intended boundaries are restored.

The minimum scope is to add exact fixtures for at least four families from `2026-03-22_14.44.46`: lesson headings that should be `KNOWLEDGE`, long explanatory lesson paragraphs that should be `KNOWLEDGE`, false-positive `OTHER -> KNOWLEDGE` advice or memoir rows that must stay `OTHER`, and outside-span imperative prose that should be rescued to `INSTRUCTION_LINE`. The tests should use the real line texts from the benchmark artifacts instead of synthetic paraphrases. That keeps the plan anchored to the actual failure mode.

Acceptance for this milestone is straightforward. Running the focused parsing slice should show the new tests exercising the current Salt Fat texts, and the prompt-rendering tests should still describe the same packet-mode story the code intends to tell.

### Milestone 2: Tighten the worker contract for `lesson_prose` and `front_matter_navigation`

At the end of this milestone, line-role workers will receive clearer packet guidance about when to keep a row as reusable cooking knowledge and when to leave it as non-knowledge prose. The worker contract should explicitly say that short lesson headings such as `Balancing Fat` or `Cooking Acids` can be `KNOWLEDGE` when surrounded by explanatory prose, but that memoir, front matter, contents lists, endorsements, and broad exhortation do not become `KNOWLEDGE` merely because they mention cooking.

This milestone touches the parser-owned packet context and the prompt templates together. The packet summaries, default posture text, strong signals, weak signals, and flip policy in `canonical_line_roles.py` should all align with the prompt template wording consumed by `cookimport/llm/canonical_line_role_prompt.py`. The change is successful when the visible prompt text reflects the sharpened boundary without widening recipe structure labels or inventing `HOWTO_SECTION`.

Acceptance is behavioral. The prompt-rendering tests should show the stronger `lesson_prose` and `front_matter_navigation` language, and the new benchmark-derived test rows should start moving in the intended direction under the Codex-side sanitization path.

### Milestone 3: Add narrow deterministic rescue and demotion seams

At the end of this milestone, the parser will have small, exact repair hooks that convert obvious model mistakes into the correct local label without reclassifying whole packet families blindly. The emphasis is on failing closed. A real lesson heading with neighboring knowledge prose should be able to survive as `KNOWLEDGE`; a vague or narrative row should still fall back to `OTHER`; an outside-span imperative row with recipe-local support should be able to become `INSTRUCTION_LINE`; and weak outside-span title or variant promotions should still be rejected.

The exact helper seams are already present. `_looks_knowledge_heading_with_context(...)`, `_looks_knowledge_prose_with_context(...)`, and `_outside_recipe_knowledge_label_allowed(...)` are the primary positive knowledge gates. `_looks_narrative_prose(...)` and `_looks_book_framing_or_exhortation_prose(...)` are the primary negative gates. `_instruction_line_label_allowed(...)` and `_instruction_line_fallback_label(...)` are the instruction rescue seams. `_looks_recipe_title_with_context(...)`, `_variant_label_allowed(...)`, and `_variant_fallback_label(...)` are guardrails that must be preserved while knowledge recall improves.

Acceptance for this milestone is passing deterministic and Codex-side regression tests that prove the exact benchmark families are now repaired without reopening the older title, variant, and `HOWTO_SECTION` regressions.

### Milestone 4: Revisit parser-owned outside-recipe review exclusion metadata only

At the end of this milestone, obvious lesson rows should not be silently excluded from downstream knowledge review, and obviously useless outside-recipe rows should still be pruned. The current parsing documentation already notes that outside-recipe line-role is fail-open `OTHER` plus optional review-exclusion metadata rather than final semantic authority. This milestone makes that contract practical for the current benchmark.

The main question is whether some current Salt Fat misses are being doomed by parser-written review exclusion metadata before the knowledge stage can help. The implementation should inspect the `knowledge_review_excluded` evidence in benchmark artifacts and the parser-owned exclusion logic in `canonical_line_roles.py`. The goal is not to offload line-role responsibility to the knowledge stage and not to redesign the knowledge runtime. The goal is only to stop prematurely discarding rows that are plausible lesson material while continuing to exclude navigation, page furniture, and obvious front matter.

Acceptance is that the exact benchmark-derived tests and any small artifact-level checks show lesson headings and lesson prose staying review-eligible when they should, while navigation-style junk still stays excluded.

### Milestone 5: Validate locally, then prove with benchmark reruns only after approval

At the end of this milestone, the implementation should have strong local evidence from parser tests and then, only if the user approves benchmark spend, one live Salt Fat rerun plus one contrast-book rerun. Salt Fat is the canary for memoir-versus-knowledge and over-structuring. `thefoodlabcutdown` is the best contrast when the fix is mostly about outside-recipe `KNOWLEDGE`. `seaandsmokecutdown` is the best contrast if the risk surface starts touching heading structure or `HOWTO_SECTION`.

Success on Salt Fat means more than a topline gain. The benchmark should materially reduce `KNOWLEDGE -> OTHER` misses, avoid a compensating surge in `OTHER -> KNOWLEDGE`, preserve the recent variant/title guardrails, and keep the benchmark diagnosis centered on better line-role labeling rather than simply moving errors around.

Acceptance is behavioral, not merely structural.

The run must materially reduce `KNOWLEDGE -> OTHER` misses, avoid a compensating surge in `OTHER -> KNOWLEDGE`, preserve the existing title/variant guardrails, and improve the single-book quality story beyond the current `strict_accuracy=0.712824` baseline. ## Idempotence and Recovery

This plan is safe to execute incrementally. The benchmark artifacts under `data/golden/benchmark-vs-golden/2026-03-22_14.44.46/` are read-only references and should not be modified. Parser and prompt tests can be rerun repeatedly without creating persistent state beyond normal pytest caches. The safest order is always tests first, then prompt and parser edits, then tests again, then benchmark reruns only with explicit approval.

If an implementation pass improves `KNOWLEDGE -> OTHER` but causes a noticeable rise in `OTHER -> KNOWLEDGE`, back out the most recent knowledge widening before touching title or variant logic. The likely failure surface will be `_looks_knowledge_prose_with_context(...)` or the packet guidance text, not recipe-title or variant helpers.

If a pass repairs lesson headings but reopens `OTHER -> RECIPE_TITLE` or `INSTRUCTION_LINE -> RECIPE_VARIANT`, stop and restore the current title or variant guardrails before continuing. Those seams were already repaired recently and should be treated as protected behavior, not collateral damage.

If the benchmark still underperforms after the parser and prompt fixes but the local tests are green, inspect review-exclusion and knowledge-stage eligibility before widening knowledge heuristics again. The safe recovery path is to determine whether the row is being excluded, not to keep loosening `KNOWLEDGE` rules blindly.
