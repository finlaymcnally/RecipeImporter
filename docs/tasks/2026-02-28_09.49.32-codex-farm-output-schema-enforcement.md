## Problem Statement

Recipeimport codex-farm subprocess calls did not pass explicit `--output-schema`, which made schema enforcement less explicit at the caller boundary and easier to drift from expected output contracts.

## Acceptance Criteria

- Subprocess codex-farm runs resolve per-pipeline `output_schema_path` from the selected pack.
- `codex-farm process` includes `--output-schema <resolved-path>` for recipe/pass4/pass5 subprocess flows.
- Runner fails before subprocess execution when pipeline definition or schema path is missing/invalid.
- Runner rejects `process --json` payloads that omit `output_schema_path` when JSON payload is present.
- `llm_report`/manifest payloads surface resolved schema paths for runtime auditability.
- Existing codex-farm runner/orchestrator tests continue passing.

## Verification Steps

1. Install/refresh dev deps in project venv:
   - `. .venv/bin/activate && pip install -e .[dev]`
2. Run targeted codex-farm suites:
   - `. .venv/bin/activate && pytest -o addopts='' tests/llm/test_codex_farm_orchestrator.py tests/llm/test_codex_farm_knowledge_orchestrator.py tests/tagging/test_tagging.py -k "codex or llm or pass5 or tags" -q`

## Evidence

- Targeted suite pass result:
  - `17 passed, 19 deselected, 2 warnings in 2.10s`
- New coverage:
  - `test_subprocess_runner_passes_root_and_workspace_flags` now asserts `--output-schema` is passed.
  - `test_subprocess_runner_fails_before_process_when_output_schema_missing` verifies fail-fast before subprocess invocation.
  - `test_subprocess_runner_rejects_process_payload_missing_output_schema_path` verifies process JSON contract enforcement.

## Constraints / Gotchas + Decision

- Pipeline IDs do not reliably map to filenames, so schema resolution scans pipeline JSON definitions by `pipeline_id`.
- Resolution is strict to avoid running codex-farm with unknown schema gates.
- Resolution is cached per `(root_dir, pipeline_id)` to avoid repeated filesystem scans in multi-pass runs.
