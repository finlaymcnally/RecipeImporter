# Problem

Dashboard trend hosts could appear to keep stretching rightward over time during repeated rerenders, but existing regression tests only checked page-level overflow and missed host-level width drift.

# Acceptance Criteria

- Add a browser-level drift harness that samples trend host widths 5 times, 5 seconds apart.
- Harness forces periodic rerenders and goes red when trend host `scrollWidth` drifts.
- Renderer pins Highcharts trend chart width to measured host width to stop drift.

# Verification

- `source .venv/bin/activate && pytest tests/analytics/test_stats_dashboard_slow.py -k benchmark_trend_hosts_do_not_gain_horizontal_drift_over_time`
- `source .venv/bin/activate && pytest tests/analytics/test_stats_dashboard_slow.py`
- `source .venv/bin/activate && pytest tests/analytics/test_stats_dashboard.py`

# Evidence

- Harness red metrics before fix: `max_trend_host_overflow_px=1396`, `max_compare_control_host_overflow_px=1516`.
- Green (after fix):
  - new slow test: `1 passed, 1 deselected` in `23.10s`
  - slow dashboard file: `2 passed` in `27.95s`
  - fast dashboard file: `72 passed` in `15.54s`
  - combined dashboard suites: `74 passed` in `42.77s`

# Constraints / Gotchas

- CDN-loaded Highcharts behavior is hard to reproduce deterministically in tests.
- Page-level overflow can stay flat while host-level chart width still drifts.

# Decision

- Added a served-mode Playwright drift harness with:
  - synthetic UI-state polling updates (`saved_at` churn) to force timed rerenders,
  - deterministic Highcharts stub that increases width when renderer does not provide explicit chart width.
- Updated trend host rendering to pass explicit `chart.width` from measured host width on every rerender.
