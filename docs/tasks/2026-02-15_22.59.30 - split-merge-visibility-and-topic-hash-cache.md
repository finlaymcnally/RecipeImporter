# Problem statement

Split runs could look hung after worker completion because `MainProcess` stayed on a generic merge label while doing long write work. Knowledge-heavy runs also spent excessive merge write time repeatedly hashing the same source file while assigning topic IDs.

# Acceptance criteria

- Main-process split merge emits phase-level status updates in the worker dashboard.
- Topic-candidate output writing hashes the source file once per file version (not once per candidate).
- Regression tests cover both behaviors.

# Verification steps

- Fail-before (on pre-change baseline):
  - `. .venv/bin/activate && pytest -q tests/test_tip_writer.py::test_write_topic_candidates_hashes_source_file_once tests/test_split_merge_status.py::test_merge_split_jobs_reports_main_process_phases`
- Pass-after:
  - `. .venv/bin/activate && pytest -q tests/test_tip_writer.py tests/test_split_merge_status.py`

# Evidence

- Pass-after command succeeds (`3 passed`).
- New merge status test verifies callback phases include topic-candidate writing/report/raw-merge steps.
- New writer test verifies only one `_hash_file` call while writing multiple topic candidates.

# Constraints/gotchas

- Keep split merge output contract unchanged (same files and IDs).
- Progress updates must be best-effort and never fail merge execution.

# Decision

Add a status callback path to split merge helpers and metadata-based hash caching in writer hash resolution.

# Rollback / compensation plan

- Remove status callback plumbing from split-merge helpers and revert to static merge labels.
- Remove cached hash helper and return to direct `_hash_file` calls in `_resolve_file_hash`.
