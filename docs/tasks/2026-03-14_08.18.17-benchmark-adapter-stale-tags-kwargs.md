# Problem

Interactive and helper-driven benchmark runs build `labelstudio_benchmark(...)` kwargs through `build_benchmark_call_kwargs_from_run_settings(...)`.
A recent benchmark surface cleanup removed tag-pipeline args from `labelstudio_benchmark`, but the adapter still forwards them, so single-offline and other adapter-backed benchmark paths fail before prediction generation starts.

# Acceptance Criteria

- Benchmark adapter output no longer includes kwargs unsupported by `cookimport.cli.labelstudio_benchmark`.
- A regression test fails on the old adapter and passes once the stale kwargs are removed.
- Internal config docs no longer claim tag-pipeline settings are wired through benchmark prediction generation.

# Verification

- `source .venv/bin/activate && pytest tests/cli/test_run_settings_adapters.py`

# Evidence

- Before fix: adapter output contains `llm_tags_pipeline`, `tag_catalog_json`, and `codex_farm_pipeline_pass5_tags`, which are absent from the `labelstudio_benchmark` signature and raise `unexpected keyword argument`.
- After fix: the signature check reports no extra kwargs, and `source .venv/bin/activate && pytest tests/cli/test_run_settings_adapters.py` passes (`6 passed`).

# Constraints / Decisions

- Keep stage tag-pipeline wiring intact; this fix is benchmark-only.
- Prefer a contract test against the real command signature so future stale kwargs fail fast.
