Problem statement

The interactive Codex-enabled flows should not split the workflow decision and the Codex-backed surface toggles into unrelated prompt styles.
The operator wants one high-level CodexFarm choice and then one reused consolidated submenu anywhere interactive CLI offers that Codex-vs-vanilla decision.

Acceptance criteria

- The interactive chooser asks for a high-level workflow first, rendering the existing `off` and `codex-farm-single-correction-v1` values as `Vanilla / deterministic only` and `CodexFarm`.
- When the operator chooses CodexFarm in interactive `Import` or any interactive benchmark mode, the follow-up UI is one reused Codex submenu expressed as explicit yes/no select prompts.
- The import Codex submenu exposes recipe correction plus knowledge harvest; the benchmark Codex submenu exposes recipe correction, block labelling, and knowledge harvest.
- Interactive run-settings still resolve the same underlying `RunSettings` fields and still prompt for Codex AI settings when at least one Codex surface remains enabled.
- Targeted CLI tests cover both the shared chooser behavior and the interactive import/benchmark call sites.

Verification steps

- `source .venv/bin/activate && pytest tests/cli/test_c3imp_interactive_menu.py -k 'choose_run_settings or interactive_import_passes_knowledge_pipeline_settings' -q`
- `source .venv/bin/activate && pytest tests/labelstudio/test_labelstudio_benchmark_helpers_interactive.py -k 'interactive_benchmark' -q`

Evidence

- Before the first part of the change, `choose_run_settings(...)` only knew how to prompt `Block labelling for this run?` and `Knowledge harvest for this run?` as separate benchmark menus.
- Before the follow-up extension, interactive import still stopped after the workflow choice and never reused the consolidated Codex submenu.
- After the change, the targeted chooser tests and interactive benchmark helper tests pass with the shared workflow menu plus the reused yes/no Codex submenu, and the interactive import test now proves that import opts into the same submenu machinery.

Constraints/gotchas + decision

- Keep the implementation localized to `cookimport/cli_ui/run_settings_flow.py` plus the interactive call sites in `cookimport/cli.py` so both import and benchmark continue to delegate the run-settings contract to one shared helper.
- Preserve the existing underlying pipeline ids and top-tier harmonization logic; only the interactive chooser wiring should change.
- Keep the submenu flow-specific: import does not expose block labelling because stage call adapters do not carry `line_role_pipeline`/`atomic_block_splitter`, while benchmark modes still do.
