# Reduce codex-farm fallback churn and runtime with pass2 soft gating, selective pass3 routing, and outside-span narrative precision

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md` from the repository root.

## Purpose / Big Picture

After this change, codex-farm should keep its quality edge over vanilla while spending less time in pass3 and avoiding fail-closed fallback for page/layout-only pass2 warnings. Outside active recipe spans, long narrative prose should default to `OTHER` unless explicit note/knowledge cues are present, reducing avoidable `OTHER -> KNOWLEDGE` and `OTHER -> RECIPE_NOTES` confusion.

A novice should be able to verify this with one paired SeaAndSmoke benchmark rerun plus speed-suite baseline/candidate comparison: codex still beats vanilla on strict/macro quality, pass3 call volume is reduced (or at minimum not expanded), and runtime regression gates do not fail.

## Context and Orientation

The active recipe codex-farm path lives in `cookimport/llm/codex_farm_orchestrator.py` and now behaves as follows:

- Pass2 degradation reasons are produced by `_pass2_degradation_reasons(...)`, then classified by `_pass2_degradation_severity(...)` (`soft` vs `hard`).
- Hard degradation still fail-closes to deterministic fallback (`pass3_status="fallback"`); soft degradation remains `pass2_status="degraded"` but is pass3-route eligible.
- Pass3 routing is explicit via `_should_run_pass3_llm(...)`, with additive per-recipe metadata (`pass2_promotion_policy`, `pass3_execution_mode`, `pass3_routing_reason`).
- `llm_manifest` now records severity/routing metadata and aggregate counters (`pass2_degraded_soft/hard`, pass3 execution-mode counts).

Canonical outside-span line labeling lives in `cookimport/parsing/canonical_line_roles.py`:

- `_deterministic_label(...)` now defaults outside-span prose to `OTHER` and upgrades to `KNOWLEDGE` only when `_looks_explicit_knowledge_cue(...)` is true.
- Recipe-note promotions for explicit note cues are unchanged.

Benchmark cutdown ingestion/reporting lives in `scripts/benchmark_cutdown_for_external_ai.py`:

- `_load_llm_manifest_recipe_diagnostics(...)` now reads additive pass2/pass3 routing fields from manifests.
- Starter-pack triage CSV/JSON summaries and selected packets now carry pass2 severity/promotion plus pass3 execution/routing fields.

## Milestones

### Milestone 1: Pass2 severity model and promotion policy metadata

Add explicit pass2 degradation classification (`hard` vs `soft`) and persist promotion policy metadata in recipe state and `llm_manifest`. Hard degradation must continue to force deterministic fallback. Soft degradation must remain `pass2_status="degraded"` for compatibility but be eligible for pass3.

### Milestone 2: Selective pass3 routing with deterministic skip path

Add a deterministic pass3 routing function for pass2-eligible recipes so low-risk rows can skip LLM pass3 and use deterministic draft promotion. Keep status compatibility; add additive routing/execution metadata so dashboards/cutdowns can distinguish true pass3 calls from deterministic promotion.

### Milestone 3: Outside-span narrative precision tightening

Refine deterministic outside-span prose classification so long narrative prose defaults to `OTHER` unless explicit note/knowledge cues are present. Preserve explicit note-prefix and heading/title protections.

### Milestone 4: Diagnostics/tests, benchmark proof, and speed regression check

Extend diagnostics/reporting and tests for new fields, rerun paired SeaAndSmoke benchmark with fixed model/effort, then run speed-suite baseline/candidate comparison to confirm runtime impact under repo-required speed tooling.

## Plan of Work

First, refactor pass2 degradation handling in `cookimport/llm/codex_farm_orchestrator.py` to classify reasons before setting pass3 status. Introduce explicit hard/soft reason classification, where `warning_bucket:page_or_layout_artifact` is soft and instruction-evidence failures remain hard. Keep existing `pass2_status` and `pass3_status` enums unchanged, but add additive recipe fields such as `pass2_degradation_severity`, `pass2_promotion_policy`, `pass3_execution_mode`, and `pass3_routing_reason`.

Next, implement selective pass3 routing in the orchestrator by adding a deterministic eligibility function for pass3 invocation. For pass2-eligible recipes, route to either LLM pass3 or deterministic promotion. Deterministic promotion must still validate against `RecipeDraftV1` and be recorded as `pass3_status="ok"` with `pass3_execution_mode="deterministic"`.

Then tighten outside-span line-role logic in `cookimport/parsing/canonical_line_roles.py` by centralizing outside-span prose routing into an explicit helper that prefers `OTHER` by default and upgrades only on explicit cues.

After behavior changes, update cutdown diagnostics in `scripts/benchmark_cutdown_for_external_ai.py` to read and export the new additive manifest fields without breaking old-manifest compatibility. Extend tests in llm/parsing/bench suites to lock the new contracts and compatibility behavior.

Finally, update docs (`10-llm`, `07-bench`, `04-parsing`), run targeted tests, run paired benchmark with fixed codex model/effort, and run speed-suite baseline/candidate comparison.

## Validation and Acceptance

Acceptance requires all of the following:

- Quality guardrails on paired rerun:
  - codex still beats vanilla on `strict_accuracy` and `macro_f1_excluding_other`.
  - codex `strict_accuracy` and `macro_f1_excluding_other` each do not drop by more than `0.01` absolute from frozen baseline.

- Pass2/pass3 policy behavior:
  - Unit test proves `warning_bucket:page_or_layout_artifact`-only degradation is classified `soft`, keeps `pass2_status="degraded"`, and remains pass3-eligible.
  - Unit test proves `missing_instructions` remains `hard` and still forces deterministic fallback without pass3 input.
  - `llm_manifest` contains additive severity/routing metadata fields and counts.

- Runtime behavior:
  - `pass3_inputs` does not increase versus frozen baseline and should decrease when selective routing identifies low-risk rows.
  - `bench speed-compare` does not fail regression gates for candidate vs baseline runs.
  - Paired rerun codex `eval_report.json.timing.prediction_seconds` is lower than frozen baseline, or if roughly flat/noisy, accompanying evidence shows reduced pass3 invocation with stable quality.

- Outside-span precision:
  - Confusion counts for `OTHER -> KNOWLEDGE` and `OTHER -> RECIPE_NOTES` are not both worse than baseline; at least one should improve.
  - Canonical line-role regression tests pass for explicit note/title/variant cases.
