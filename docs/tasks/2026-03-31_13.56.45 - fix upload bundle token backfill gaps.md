# Fix Upload Bundle Token Backfill Gaps

## Problem

The upload bundle can underreport total benchmark spend when call-inventory rows exist for a stage but their per-call telemetry is blank.
The Oracle token brief also still reads legacy stage keys, so current `prompt_budget_summary.json` files can show knowledge/recipe totals as unavailable even when the data exists.

## Acceptance Criteria

- Upload-bundle runtime summaries backfill stage and total token/runtime totals from `prompt_budget_summary.json` when those totals are more complete than the call-inventory rollup.
- The merge keeps better row-derived coverage where the prompt-budget summary lacks an equivalent metric, especially estimated-cost coverage and stage runtime stats for stages with usable per-call telemetry.
- Oracle token-lane briefs read both current normalized stage keys (`nonrecipe_finalize`, `recipe_refine`) and legacy aliases (`knowledge`, `recipe_correction`).
- Focused tests cover the same-call-count missing-telemetry case and the token brief stage-name lookup.

## Verification

- `source .venv/bin/activate && pytest tests/bench/test_benchmark_cutdown_for_external_ai.py -q -k "prompt_budget_summary or missing_stage_telemetry"`
- `source .venv/bin/activate && pytest tests/bench/test_benchmark_oracle_upload.py -q -k "token_lane"`

## Evidence

- Added a regression where `nonrecipe_finalize` call rows are present but have null telemetry, while `prompt_budget_summary.json` has complete stage totals; the bundle summary must still report the full spend.
- Added a regression for Oracle token-lane brief rendering with current stage keys so the brief no longer prints `unavailable` for knowledge/recipe totals when the prompt-budget file is complete.

## Constraints / Decisions

- Do not invent per-call token values when only stage-level totals exist; fix the summary/topline path instead.
- Keep the change local to upload-bundle reporting and Oracle brief rendering; do not widen it into a broader benchmark artifact refactor.
