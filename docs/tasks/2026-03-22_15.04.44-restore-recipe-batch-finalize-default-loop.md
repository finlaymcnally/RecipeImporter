# Restore the recipe batch-finalize default loop and remove per-task helper churn

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The recipe worker refactor that introduced the repo-owned `check-current` / `install-current` loop kept recipe reliable, but it made recipe much more expensive. After this change, recipe workers should keep the good parts of the refactor, specifically `SHARD_PACKET.md`, prewritten drafts, and fallback helper tools, while returning to the cheaper working style: read the packed shard summary once, edit the drafts directly, and run one `finalize-all` pass at the end. A human should be able to rerun the same Salt Fat benchmark slice after implementation and see recipe keep `validated_shard_count: 5`, keep all watchdog and follow-up counts at zero, and bring recipe command count and command-execution tokens back toward or below the earlier `2026-03-22_13.22.18` / `2026-03-22_03.33.50` recipe baselines.

## Surprises & Discoveries

- Observation: the recipe regression is specifically a control-flow regression, not a correctness regression.
  Evidence: `data/golden/benchmark-vs-golden/2026-03-22_14.44.46/.../prompt_budget_summary.json` still reports recipe `validated_shard_count: 5`, `watchdog_killed_shard_count: 0`, `watchdog_recovered_shard_count: 0`, and `structured_followup_call_count: 0`.

- Observation: the new recipe prompt replaced one end-of-shard `finalize-all` flow with a much chattier per-task validation/install loop.
  Evidence: `data/output/2026-03-22_13.22.18/.../recipe_phase_runtime/workers/worker-001/prompt.txt` told the worker to edit batch drafts and “Finish with `python3 tools/recipe_worker.py finalize-all scratch/` once after the drafts are ready,” while `data/output/2026-03-22_14.44.46/.../recipe_phase_runtime/workers/worker-001/prompt.txt` now requires `check-current`, then `install-current`, then reopening current-task sidecars after each task.

- Observation: the command regression is large enough that it cannot be explained by ordinary prompt variance.
  Evidence: recipe `command_execution_count_total` rose from `47` at `2026-03-22_13.22.18` to `113` at `2026-03-22_14.44.46`, and recipe `command_execution_tokens_total` rose from `536003` to `1329282`.

- Observation: the new recipe worker path causes repeated rereads of startup surfaces, current-task sidecars, and draft files.
  Evidence: `data/output/2026-03-22_14.44.46/.../recipe_phase_runtime/workers/worker-001/status.json` shows startup reads of `worker_manifest.json`, `SHARD_PACKET.md`, `CURRENT_TASK.md`, `current_task.json`, and `CURRENT_TASK_FEEDBACK.md`, then an all-drafts scratch read, then additional `in/*.json` and `hints/*.md` reads, then repeated `check-current` / `install-current` calls plus sidecar rereads.

- Observation: `SHARD_PACKET.md` itself is not the problem.
  Evidence: the old cheaper `2026-03-22_13.22.18` path was more expensive than desired because the worker had to rediscover too much contract context. The new packet is a useful improvement, but it got paired with a more expensive task loop.

- Observation: recipe stage `tokens_total` is not the right acceptance metric for this regression because the runtime summary duplicates worker-session totals per task row.
  Evidence: for both recipe runs, the sum of the first telemetry row `tokens_total` across workers equals stage `command_execution_tokens_total`, while stage `tokens_total` is much larger because each worker session total is repeated across every task row in the worker summary.

- Observation: the recipe fix should stay recipe-local and should not be bundled with knowledge changes.
  Evidence: the user asked specifically how to fix the statement “Recipe clearly regressed after the step refactor,” and the recipe traces show a self-contained control-flow regression that can be fixed without reopening knowledge design choices.

- Observation: restoring the cheap recipe path did not require changing queue-controller mechanics.
  Evidence: only prompt/sidecar/helper wording plus recipe docs/tests changed, and the focused worker/orchestrator coverage still passed after those wording-only edits.

## Context and Orientation

Recipe workspace workers live primarily in `cookimport/llm/codex_farm_orchestrator.py` and `cookimport/llm/recipe_workspace_tools.py`. The orchestrator writes the recipe worker prompt, worker root, helper files, and task queue, then gathers validated `out/*.json` task results back into final shard proposals. The workspace-tools module renders the helper CLI plus worker-facing files such as `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`, `SHARD_PACKET.md`, and `scratch/_prepared_drafts.json`. `cookimport/llm/codex_exec_runner.py` mirrors those files into the sterile direct-exec workspace and records worker command telemetry.

In this repository, a "recipe worker assignment" means one long-lived Codex workspace session that owns multiple task rows from one recipe shard. A "task row" is one recipe-owned JSON packet from `assigned_tasks.json`, plus its `in/<task_id>.json`, `hints/<task_id>.md`, `scratch/<task_id>.json`, and `out/<task_id>.json` files. A "batch finalize path" means the worker edits one or more prewritten draft files under `scratch/` and then runs `python3 tools/recipe_worker.py finalize-all scratch/` once after those edits are complete. A "per-task helper loop" means the worker repeatedly runs `check-current`, `install-current`, and sidecar rereads after each accepted task. The plan’s goal is to restore the first pattern as the default recipe behavior while retaining the second pattern only for fallback and recovery.

The most relevant benchmark evidence is:

The older cheaper recipe prompt told the worker to inspect the draft set and finish with one `finalize-all` pass. The newer prompt tells the worker to step through the shard one current task at a time, running `check-current`, then `install-current`, then reopening sidecars after each accepted task. That new default path is what this plan must reverse.

## Milestones

### Milestone 1: Restore the cheap recipe happy path in the prompt

This milestone rewrites the recipe worker prompt so the safe-looking default path is again: open `SHARD_PACKET.md`, inspect the prewritten drafts, edit the needed `scratch/<task_id>.json` files directly, and run one `finalize-all` pass at the end. At the end of this milestone, the prompt no longer tells the worker to treat `check-current` / `install-current` as the normal queue loop. Those verbs remain mentioned only as fallback/debug tools when a single draft needs targeted validation or recovery.

The proof is a generated recipe worker prompt that mentions `finalize-all` as the normal completion step, demotes `check-current` / `install-current`, and no longer tells the worker to reopen current-task sidecars after every accepted task.

### Milestone 2: Demote current-task loop surfaces without throwing them away

This milestone keeps the current-task files and helper verbs present, but changes their wording so they stop pulling the worker into a task-by-task validation/install rhythm. `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`, and related helper output should still make sense for targeted inspection and failure recovery, but they should no longer read like the main recipe assignment contract.

The proof is that the worker-facing sidecars describe the draft path and contract summary without making “validate, install, reread, repeat” sound mandatory for every task in a healthy shard.

### Milestone 3: Keep the packet, drafts, and compact contract while removing the churn

This milestone keeps the good parts of the recent refactor: `SHARD_PACKET.md`, enriched `scratch/_prepared_drafts.json`, and the packetized compact contract summary. The repo should not regress to requiring `OUTPUT_CONTRACT.md`, `examples/*.json`, or helper source as ordinary startup reads. The difference is only that the worker edits the prepared drafts in a batch and promotes them in one go.

The proof is a generated worker root where the packet remains authoritative, the batch draft list is still present, and `finalize-all` works against the same compact contract as before.

### Milestone 4: Lock the rollback in tests and short docs

This milestone updates the recipe-focused tests and docs so the repo will reject future drift back toward the per-task helper loop as the default path. The tests should check prompt wording, sidecar wording, helper affordance classification, and at least one command-trace expectation that distinguishes the batch finalize path from the current-task loop.

The proof is a focused recipe regression suite that passes locally and short docs that accurately describe the restored happy path.

## Idempotence and Recovery

This work is safe to stage incrementally. It is safe to first change the prompt text, then revise sidecar/helper wording, then update tests and docs. The helper verbs can remain implemented throughout, which keeps the worker root stable while the default path changes back.

If restoring `finalize-all` as the default path unexpectedly causes a quality regression in local tests, the fallback is not to restore the whole per-task loop. The first fallback is to keep `finalize-all` as the normal route while allowing one explicit `check-current` pass for the currently edited draft only when the batch validator reports a concrete problem. The second fallback is to add one batch-oriented helper such as `validate-all` that checks every prepared draft in one step before `finalize-all`, but only if the current tests prove that one end-of-shard validation seam is necessary. Do not fall back to reopening sidecars and installing one task at a time unless new evidence shows that recipe correctness cannot be preserved otherwise.
