# Compact Knowledge Review Model Payload

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

Today the non-recipe knowledge review stage sends a grounded subset of the book to Codex, but the model-facing payload spends more tokens on JSON scaffolding than on raw block text. After this change, the knowledge stage should still give Codex the same useful evidence and the same review task, but in a denser input contract that removes or compresses machine-readable ceremony the model does not need. The user-visible proof is a new zero-token Food Lab prompt preview showing a lower `nonrecipe_knowledge_review` token estimate while the stage still carries the same chunk evidence and passes focused knowledge tests.

## Surprises & Discoveries

- Observation: current knowledge cost is not driven by duplicate chunk bundling.
  Evidence: on `data/output/2026-03-18_17.15.00-foodlab-current-preview`, repeated chunk ids across bundles are `0`, repeated block indices across bundles are `0`, and `context` lists are empty because default knowledge context is `0`.

- Observation: current knowledge input spends more on wrapper structure than on raw block text.
  Evidence: tokenizing the current Food Lab knowledge payloads with `o200k_base` shows about `212k` task-body input tokens against about `86.6k` tokens of raw `blocks[*].text`.

- Observation: the biggest safe savings came from compact field aliases plus omitting empty optional structures, not from changing chunk coverage.
  Evidence: after the refactor, Food Lab still produced `5` knowledge calls and the same knowledge output estimate, but knowledge input dropped from `214,665` to `120,077` tokens.

## Decision Log

- Decision: target the model-facing knowledge input contract first, not the chunker or output schema.
  Rationale: the current measured waste is mostly transport overhead around chunk evidence, so a compact input contract should recover tokens without changing stage semantics or reviewer-facing outputs.

## Context and Orientation

The non-recipe knowledge review stage lives in `cookimport/llm/codex_farm_knowledge_jobs.py`, `cookimport/llm/knowledge_prompt_builder.py`, `cookimport/llm/codex_farm_knowledge_orchestrator.py`, and `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md`. The chunker builds eligible non-recipe chunks from parser output, then the knowledge worker sends one inline prompt per shard through direct `codex exec`. The live model-facing input is now a compact JSON object with top-level `v`, `bid`, `c`, optional `x`, and optional `g`; each chunk carries `cid`, `b`, and optional `h`; and each block carries `i`, `t`, and optional table or heading cues.

The ingest validator in `cookimport/llm/codex_farm_knowledge_ingest.py` validates only the output contract and the owned chunk/block surface. It does not require the current verbose input field names. That means we can compact the model-facing input contract as long as the prompt text teaches Codex the new shape and the live/preview builders stay aligned.

## Plan of Work

First, define a denser model-facing knowledge input contract in `cookimport/llm/codex_farm_knowledge_contracts.py`. Keep the same semantic evidence, but shorten top-level and nested field names and remove provenance-heavy fields the model does not need. Do not change the output schema.

Next, update `cookimport/llm/codex_farm_knowledge_jobs.py` so `build_knowledge_jobs(...)` writes the compact contract to `input_payload`. Keep shard metadata rich enough for local validation and reporting even if the model-facing payload gets smaller.

Then update `cookimport/llm/knowledge_prompt_builder.py` and `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md` so the inline prompt explains the compact shape clearly, no longer references removed verbose fields, and does not spend tokens on pretty-printed JSON. Preview must use the same builder, so `cookimport/llm/prompt_preview.py` should not need separate prompt logic beyond whatever payload shape changes are necessary.

Finally, update the focused tests in `tests/llm/test_knowledge_prompt_builder.py`, `tests/llm/test_knowledge_job_bundles.py`, `tests/llm/test_knowledge_phase_workers.py`, and any preview tests that inspect knowledge prompt rows. Then rerun a zero-token Food Lab full preview and record the new `nonrecipe_knowledge_review` budget in this plan.
