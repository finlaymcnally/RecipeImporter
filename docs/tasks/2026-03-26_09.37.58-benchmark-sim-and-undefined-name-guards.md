## Purpose / Big Picture

After this change, routine local test runs will catch the two failure modes that have recently escaped into manual CLI usage: missing helper/name bindings in the split benchmark modules, and crashes that only appear when the real interactive single-book benchmark path runs all the way through the benchmark helper stack. A user should be able to run focused pytest coverage and see one simulated offline vanilla benchmark run and one simulated offline codex-shaped benchmark run complete without spending real LLM tokens, while a separate static guard fails fast on undefined names in the benchmark CLI/helper surface.

## Surprises & Discoveries

- Observation: The repo already has several direct tests for `labelstudio_benchmark(...)`, `run_pipelined(...)`, and `_run_offline_benchmark_prediction_stage(...)`, but the interactive smoke boundary still stops above the real benchmark command.
  Evidence: `tests/labelstudio/test_labelstudio_benchmark_smoke.py` patches `labelstudio_benchmark`, while `tests/labelstudio/test_labelstudio_benchmark_helpers_eval_payload_*.py` exercises deeper non-interactive seams.

- Observation: The benchmark stack already exposes a clean fakeable leaf boundary in `generate_pred_run_artifacts(...)` plus `evaluate_canonical_text(...)`, so a real offline sim does not need to stub the whole command.
  Evidence: `cookimport/cli_commands/labelstudio.py` builds a `BenchmarkPredictionBundle` from `generate_pred_run_artifacts(...)` and then routes evaluation through `evaluate_stage(...)`.

- Observation: `ruff` is not installed in the current project `.venv`, so a static undefined-name guard will need an explicit dev dependency addition or another packaged checker.
  Evidence: `source .venv/bin/activate && python -m ruff --version` currently fails with `No module named ruff`.

- Observation: Ruff cannot be aimed directly at the split benchmark helper modules because they intentionally rely on dynamic `globals().update(...)` facade snapshots and produce thousands of false positives.
  Evidence: `python -m ruff check --select F821 cookimport/cli_support/bench*.py cookimport/cli_commands/labelstudio.py` initially reported 1788 errors, almost all from split helper modules that populate names dynamically at runtime.

- Observation: The new static guard exposed three real undefined names in `cookimport/cli_commands/labelstudio.py`, including a missing interrupt finalizer that existing tests were meant to cover but were not being run routinely.
  Evidence: Ruff flagged `_finalize_interrupted_benchmark_run` and `_coerce_int` in `labelstudio.py`; after adding those helpers, both Ruff and the existing interrupt tests pass.

- Observation: The split benchmark modules still carried many unresolved globals even after the smoke sims and Ruff guard passed; the dynamic facade wiring hid them until a manual run reached the right branch.
  Evidence: A bootstrapped `LOAD_GLOBAL` audit over `bench_artifacts`, `bench_all_method`, `bench_cache`, `bench_single_book`, `bench_single_profile`, `bench_oracle`, and `bench_compare` initially reported 24 unresolved names, including `_normalize_timing_payload`, `_path_for_manifest`, `_report_count`, `_resolve_artifact_path`, and `RECIPE_CODEX_FARM_PIPELINE_SHARD_V1`.

## Outcomes & Retrospective

The final result is a layered guardrail instead of another one-off regression test. The benchmark command surface now has a static undefined-name check via Ruff, the split benchmark modules now have a bootstrapped unresolved-global audit, and the interactive smoke slice now includes two offline whole-run single-book simulations that execute the real benchmark runtime for both vanilla and codex-shaped settings without live LLM calls. This directly closes the gap that allowed repeated `NameError` crashes to escape into manual CLI use.

The work also exposed and fixed three real undefined names in `cookimport/cli_commands/labelstudio.py`, then a further 24 unresolved globals across the split benchmark helper modules. Existing interrupt-path tests now pass once the command-local helper is restored, and the new bootstrapped audit stays green once the helper modules bind their own cross-module names explicitly instead of hoping the facade snapshot happened to contain them.

## Context and Orientation

The recent crashes occurred in the interactive single-book benchmark workflow. That workflow starts in `cookimport/cli.py` interactive mode, routes into `cookimport/cli_support/bench_single_book.py`, and then invokes the real benchmark command owned by `cookimport/cli_commands/labelstudio.py`. The benchmark command performs prediction generation, creates benchmark prediction records from authoritative staged artifacts, reconstructs replay artifacts, evaluates them, writes `eval_report.json` and `run_manifest.json`, and then returns to the single-book helper so it can write `single_book_summary.md`, optional comparison artifacts, and upload-bundle follow-up outputs.

The current smoke file is `tests/labelstudio/test_labelstudio_benchmark_smoke.py`. It does use the real interactive menu flow, but it patches `labelstudio_benchmark(...)` itself. That means it only proves menu routing and top-level argument shape. It does not execute the code in `cookimport/cli_support/bench_artifacts.py`, `cookimport/cli_support/bench_single_book.py`, or the benchmark command body where recent `NameError` crashes happened.

The repo already has useful lower-level benchmark tests. `tests/labelstudio/test_labelstudio_benchmark_helpers_eval_payload_execution.py` and `tests/labelstudio/test_labelstudio_benchmark_helpers_eval_payload_pipelined.py` cover command-level execution with synthetic prediction runs. `tests/labelstudio/benchmark_helper_support.py` contains shared helper builders such as `_write_benchmark_prediction_run_fixture(...)`, `_install_noop_benchmark_eval_mocks(...)`, and `_make_lightweight_single_book_publisher(...)`. This plan will reuse those helpers rather than building a second fixture framework.

The static undefined-name guard is needed because Python can hide missing names until runtime. A reference like `_report_optional_metric(...)` inside a function can import successfully and still fail only when a real benchmark run reaches that branch. A checker like Ruff’s `F821` rule catches undefined names without needing the exact runtime path.

## Plan of Work

First, add Ruff to the project’s dev dependency set in `pyproject.toml` so the local `.venv` can install and run it reproducibly. Then add a focused test under `tests/core/` that shells out through `sys.executable -m ruff check --select F821` against `cookimport/cli_commands/labelstudio.py`. The original design considered linting `cookimport/cli_support/bench*.py` as well, but the split benchmark helper modules intentionally depend on runtime facade-populated globals and generate unusable false positives under Ruff. The command module is still valuable because it uses explicit imports and recently carried real undefined names.

Next, strengthen the interactive smoke/sim coverage. The safest place is `tests/labelstudio/test_labelstudio_benchmark_smoke.py`, because that file is already smoke-marked and already exercises `cli._interactive_mode()`. Replace the current “fake the whole benchmark command” helper with an additive helper for real offline simulation, or add a second helper beside it. The new helper should still patch menu selection, settings loading, benchmark publication side effects, and dashboard refresh. It should stop patching `labelstudio_benchmark(...)`.

Instead, patch `_resolve_benchmark_gold_and_source(...)` to return synthetic `tmp_path` fixtures, patch `generate_pred_run_artifacts(...)` to write a minimal prediction-run directory using `_write_benchmark_prediction_run_fixture(...)`, and patch `evaluate_canonical_text(...)` plus `ensure_canonical_gold_artifacts(...)` to produce deterministic canonical evaluation results from synthetic local files. Leave the real `labelstudio_benchmark(...)`, `run_pipelined(...)`, `predict_stage(...)`, replay-artifact building, and single-book result publication flow intact.

The vanilla sim test should select benchmark settings with all Codex-backed surfaces off and assert that one benchmark run completes successfully, writes the expected benchmark session outputs, and does not request a Codex-enabled path. The codex-shaped sim test should select codex-enabled benchmark settings and assert that two real single-book variants run (`off` and `codex-recipe-shard-v1`), both succeed offline, and write the comparison/session artifacts without any live LLM usage. The fake prediction-generation function should capture `allow_codex`, `llm_recipe_pipeline`, and model/reasoning settings so the test can prove that the codex-shaped branch was exercised while still remaining synthetic.

Finally, update `docs/12-testing/12-testing_README.md` with the new coverage boundary: the smoke slice now includes offline simulated whole-run single-book coverage, while live LLM execution remains stubbed at the leaf generation/evaluation seam. If a new `tests/core/test_...` file is added for Ruff, update `tests/conftest.py` with the correct domain marker and smoke classification if appropriate.

## Interfaces and Dependencies

The new static guard test should use Python’s `subprocess.run(...)` with `sys.executable` to invoke Ruff from the current `.venv`, not a bare `ruff` shell command that might hit a global install.

The whole-run sim helper should continue using:

- `cli._interactive_mode()` as the top-level exercised entrypoint,
- `cli._interactive_single_book_benchmark(...)` and `labelstudio_benchmark(...)` as the real runtime under test,
- `_write_benchmark_prediction_run_fixture(...)` from `tests/labelstudio/benchmark_helper_support.py` to build synthetic prediction-run artifacts,
- `_make_lightweight_single_book_publisher(...)` to avoid heavyweight upload/dashboard side effects,
- patched `generate_pred_run_artifacts(...)`, `ensure_canonical_gold_artifacts(...)`, and `evaluate_canonical_text(...)` as the leaf stubs that guarantee zero live LLM/token use.
