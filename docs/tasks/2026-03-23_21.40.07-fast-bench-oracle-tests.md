This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

Routine `./scripts/test-suite.sh domain bench` runs are currently too slow for normal editing loops. The slowest part is not live LLM work; it is test architecture. After this change, the bench domain should keep the fast default slice focused on synthetic local fixtures and small contract checks, while any realism-heavy Oracle benchmark tests are either reduced to tiny fixtures or moved into explicit slow files. A contributor should be able to run the bench domain and get a materially shorter turnaround without losing the important contract coverage.

## Surprises & Discoveries

- Observation: The visually largest bench files were not the main runtime problem.
  Evidence: `tests/bench/test_benchmark_cutdown_for_external_ai.py` took about `4.9s`, while `tests/bench/test_oracle_followup.py` took about `52.5s`.

- Observation: `tests/bench/test_oracle_followup.py` copies a real benchmark root over and over.
  Evidence: the source root at `data/golden/benchmark-vs-golden/2026-03-21_11.17.08/single-book-benchmark/saltfatacidheatcutdown` is about `388M` and `_copy_sample_bundle_root(...)` is used twelve times in the file.

- Observation: `tests/bench/test_benchmark_oracle_upload.py` also pays real wall time in production polling code.
  Evidence: `cookimport/bench/oracle_upload.py` uses a `3.0s` background audit poll window in `_poll_oracle_background_audit(...)`, and the tests exercise that path.

- Observation: even after the Oracle fixes, repeated host process-worker probes were still a measurable drag in the quality-suite runner tests.
  Evidence: `tests/bench/test_quality_suite_runner.py` dropped from about `ELAPSED=4.25` to about `ELAPSED=3.16` after making `_probe_all_method_process_pool_executor()` a file-level default stub and leaving the explicit override tests intact.

## Outcomes & Retrospective

The refactor hit the user-visible goal: routine bench-domain runs are now fast enough for normal coding loops without weakening the no-live-token test contract. The biggest win came from deleting the repeated dependency on checked-in benchmark roots, not from micro-optimizing pytest itself. The remaining Oracle bench files are still broad enough that future maintainability work could split them further by behavior, but they are no longer blocking routine iteration.

One additional cleanup fell out of the follow-up pass: several bench CLI tests had started patching the wrong module after the `cookimport.cli` split. Fixing those stale seams and restoring the missing integer-coercion helper in `cli_support` were required to keep the faster bench slice reliably green.

The later profiling pass showed that the remaining Oracle-heavy runtime was no longer big fixtures or polling; it was repeated `oracle --version` shell-outs hidden inside fast tests. Stubbing that probe at the file level was the highest-leverage final cut.

One last small pass found another file-level cost: many quality-suite runtime tests were still paying for a host process-pool capability probe even when they only cared about metadata propagation or executor selection around a mocked runner. Making the default test seam explicit brought the bench domain down into the `~7s` wall-clock range.

## Context and Orientation

The fast bench-domain wrapper lives in [scripts/test-suite.sh](/home/mcnal/projects/recipeimport/scripts/test-suite.sh). The `domain bench` mode runs `pytest tests/bench -m "not slow"`. Bench file routing and slow-file membership are controlled centrally in [tests/conftest.py](/home/mcnal/projects/recipeimport/tests/conftest.py). The current testing contract is documented in [docs/12-testing/12-testing_README.md](/home/mcnal/projects/recipeimport/docs/12-testing/12-testing_README.md).

The three slowest bench files are:

- [tests/bench/test_oracle_followup.py](/home/mcnal/projects/recipeimport/tests/bench/test_oracle_followup.py), which validates the turn-2 Oracle follow-up packet flow and its turn-1 recovery logic.
- [tests/bench/test_benchmark_oracle_upload.py](/home/mcnal/projects/recipeimport/tests/bench/test_benchmark_oracle_upload.py), which validates Oracle upload prompt building, upload command assembly, background launch staging, and log auditing.
- [tests/bench/test_cf_debug_cli.py](/home/mcnal/projects/recipeimport/tests/bench/test_cf_debug_cli.py), which validates `cf-debug` selector, packet, and knowledge-audit CLI behavior.

An `upload_bundle_v1` is the local benchmark review packet written under a benchmark run root. For these tests, the important contract is not the full production benchmark tree; it is the presence of a valid `upload_bundle_index.json`, a valid `upload_bundle_payload.jsonl`, minimal `run_diagnostics`, and whichever tiny run artifacts a given assertion reads.

## Plan of Work

First, add a domain-local support module under `tests/bench/` that builds a tiny synthetic benchmark session root plus `upload_bundle_v1`. The builder will create only the files that Oracle follow-up and `cf-debug` selectors actually read: minimal `upload_bundle_index.json`, payload rows for changed lines, recipe triage, per-recipe breakdown, and one codex run plus one baseline run with tiny line-role prompt artifacts.

Second, replace the giant-copy helper in `tests/bench/test_oracle_followup.py` with the synthetic builder. Keep the file focused on Oracle follow-up behavior, but split it into narrower fast files if the resulting module is still too broad. If any realism-heavy coverage still needs a checked-in benchmark root, move only that slice into a dedicated slow file.

Third, reduce `tests/bench/test_benchmark_oracle_upload.py` runtime by patching the background-audit poll constants in tests that only care about command or metadata shape, then split that file into narrower prompt/command/audit/background files so future hotspot work stays localized.

Fourth, replace the copied bundle setup in `tests/bench/test_cf_debug_cli.py` with the same synthetic support fixture where possible, keeping knowledge-only additions local to the test helper instead of mutating a copied checked-in benchmark tree.

Finally, update [tests/conftest.py](/home/mcnal/projects/recipeimport/tests/conftest.py) and [docs/12-testing/12-testing_README.md](/home/mcnal/projects/recipeimport/docs/12-testing/12-testing_README.md) so the new file layout and fast/slow boundaries are explicit.
