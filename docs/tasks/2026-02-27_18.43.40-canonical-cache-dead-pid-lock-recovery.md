## Problem statement

Canonical alignment cache locking only treated locks as stale by age (`wait_seconds`, default 3600s).  
When a worker dies while holding a lock, other workers can wait far longer than config timeout, causing slow tails and failures.

## Acceptance criteria

- Dead-owner lock files are treated as stale immediately (without waiting for age threshold).
- Existing age-based stale-lock fallback remains for malformed/non-PID lock files.
- Canonical cache tests cover dead-PID lock recovery behavior.

## Verification steps

- `source .venv/bin/activate && pytest tests/bench/test_canonical_alignment_cache.py -k dead_owner -q`
- `source .venv/bin/activate && pytest tests/bench/test_canonical_alignment_cache.py -q`
- `source .venv/bin/activate && pytest tests/bench/test_eval_stage_blocks.py -k stale_lock -q`

## Evidence

  - `COOKIMPORT_PYTEST_VERBOSE_OUTPUT=1 pytest -o addopts='' -vv --tb=short --show-capture=all --assert=rewrite tests/bench/test_canonical_alignment_cache.py -k dead_owner`
  - failure: `assert 5.01962693198584 < 1.0`
- Pass-after:

  - `source .venv/bin/activate && pytest tests/bench/test_canonical_alignment_cache.py -q` (exit 0)
  - `source .venv/bin/activate && pytest tests/bench/test_eval_stage_blocks.py -k stale_lock -q` (exit 0)

## Constraints/gotchas

- Keep lock-file format backward-compatible (`pid=... started_at=...`).
- Avoid false-positive stale detection when lock owner process is alive.

## Decision

Use lock metadata PID liveness checks as the primary stale test, with existing age threshold as fallback.

## Rollback / compensation plan

Revert `canonical_alignment_cache` lock staleness logic and remove the dead-owner regression test.
