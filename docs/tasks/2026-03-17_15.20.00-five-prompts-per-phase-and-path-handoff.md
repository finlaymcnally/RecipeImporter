# Make Shard-V1 Default To Five Prompts Per Phase And Use File-Path Handoff

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, shard-v1 recipe, knowledge, and line-role planning should aim for about five prompts per phase by default instead of many smaller prompt-sized shards. The operator should also be able to edit that number directly per phase without mentally converting from “recipes per shard” or “lines per shard” into total prompt count.

At the same time, the Codex handoff should stop inlining the full shard payload into the prompt template. The active pipelines should hand Codex a short instruction prompt plus a file path inside the worker workspace so the data already lives on disk in the pre-made runtime folder structure.

## Context and Orientation

The live shard-v1 runtime lives in three phase-specific planners plus one shared worker loop.

`cookimport/llm/codex_farm_orchestrator.py` plans recipe shards from deterministic `RecipeCandidate` inputs and currently sizes shards with `recipe_shard_target_recipes`.

`cookimport/llm/codex_farm_knowledge_jobs.py` plans knowledge shards from deterministic non-recipe chunks and currently sizes bundles with `target_chunks_per_shard`.

`cookimport/parsing/canonical_line_roles.py` plans line-role shards from ordered atomic line candidates and currently sizes shards with `line_role_shard_target_lines` or the older codex batch size.

`cookimport/llm/prompt_preview.py` reconstructs the same shard plans for zero-token preview and must stay aligned with the live planners so preview prompt counts match runtime planning.

`cookimport/llm/phase_worker_runtime.py` materializes `workers/<id>/` folders, writes assigned shard inputs, calls the runner once per worker assignment, and validates outputs. Right now it points `--workspace-root` at `workers/<id>/sandbox`, while the actual input files live in sibling `workers/<id>/in`.

The active pipeline assets live under `llm_pipelines/`. Today they all use `prompt_input_mode: "inline"` and prompt templates that embed `{{INPUT_TEXT}}`, which is why prompt preview still shows huge inline payloads.

## Plan of Work

First, extend `cookimport/config/run_settings.py` with three new optional settings that describe desired prompt count directly: one each for recipe, knowledge, and line-role. Set their defaults so shard-v1 planning targets five prompts per phase unless a caller overrides them. Keep the current worker-count and shard-size settings intact for lower-level tuning.

Next, add small planning helpers in the recipe, knowledge, and line-role modules that convert “desired prompt count” into “items per shard” using ceiling division over the real item count. Use those helpers in both live runtime planning and `prompt_preview.py` so the same benchmark root produces the same planned prompt count in preview and execution.

Then, adjust `cookimport/llm/phase_worker_runtime.py` so the Codex-visible workspace is the full worker folder rather than the isolated `sandbox/` leaf. Keep the worker structure deterministic and continue writing inputs/outputs/logs there, but make the workspace root match the folder that already contains the shard files.

After that, change the active pipeline pack JSON files under `llm_pipelines/pipelines/` from `prompt_input_mode: "inline"` to `prompt_input_mode: "path"`, and rewrite the corresponding prompt templates under `llm_pipelines/prompts/` so they instruct Codex to read the shard file at `{{INPUT_PATH}}` instead of inlining the JSON blob into the prompt body.

Finally, update focused tests across run settings, pipeline-pack assets, phase runtime, and phase-specific shard planners. The validation should prove both the new default prompt counts and the new file-path transport shape.

## Validation and Acceptance

Acceptance is behavioral:

- when shard-v1 phases are enabled with no explicit overrides, the planners should target about five prompts per phase by default
- when the operator edits the new per-phase prompt-count setting, planned shard counts should follow that number in both live runtime metadata and prompt preview
- worker runtime folders should still be deterministic and replayable, but Codex prompts should now reference input files instead of inlining the entire shard payload
- `prompt_preview_budget_summary.*` should shrink sharply in prompt text size when switching from inline payload prompts to file-path prompts

## Idempotence and Recovery

These changes are safe to rerun. If the path-based prompt transport proves incompatible with a specific phase, revert only that phase’s pipeline asset and keep the prompt-count settings intact while the transport issue is debugged. The worker runtime artifacts remain additive and should not require destructive cleanup.
