Problem statement

Benchmark/import spinners were only as informative as the latest plain status string. When a stage omitted `running N`, worker rows disappeared, and knowledge harvest was barely reporting anything while it ran.

Acceptance criteria

- Shared CLI status rendering accepts richer structured stage-progress payloads in addition to plain strings.
- Canonical line-role and knowledge harvest emit enough stage metadata to keep benchmark/import status panels informative.
- `processing_timeseries*.jsonl` persists stage labels, task counts, active tasks, worker counts, and detail lines when available.

Verification steps

- `source .venv/bin/activate && pytest tests/labelstudio/test_labelstudio_benchmark_helpers_progress.py -k 'structured_stage or generic_running_task_progress or canonical_line_role_progress or processing_timeseries'`
- `source .venv/bin/activate && pytest tests/parsing/test_canonical_line_roles.py -k 'progress_callback_reports or codex_max_inflight_override_takes_precedence or defaults_workers_to_shard_count_when_unspecified'`
- `source .venv/bin/activate && pytest tests/llm/test_codex_farm_knowledge_orchestrator.py -k 'structured_progress_snapshots or writes_manifest_and_artifacts'`
- `source .venv/bin/activate && pytest tests/labelstudio/test_labelstudio_benchmark_helpers_single_profile.py -k 'shared_spinner_dashboard'`
- `source .venv/bin/activate && pytest tests/llm/test_codex_farm_orchestrator_runner_transport.py -k 'legacy_run_progress or progress_events'`

Evidence

- Added structured stage-progress coverage in the benchmark spinner tests.
- Added knowledge-orchestrator coverage proving structured progress payloads are emitted.
- Existing line-role progress tests still pass after moving that stage onto structured payloads.

Constraints / gotchas

- Keep backward compatibility for existing plain `task X/Y` progress strings and worker-activity payloads.
- Avoid turning the progress system into a separate service; this repo only needs a small serialized payload plus better shared rendering.

Decision

- Use one repo-owned serialized `stage_progress` callback payload in `cookimport/core/progress_messages.py`.
- Keep the human-visible message text inside the payload so old display semantics stay recognizable while richer telemetry travels alongside it.

Rollback / compensation plan

- Revert the stage-specific payload emitters first if a consumer outside the shared CLI paths cannot handle the new structured messages yet; the shared parser can remain in place without affecting plain-string emitters.
