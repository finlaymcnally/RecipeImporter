# Deduplicate All-Method Canonical Evaluation By Signature

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This repository includes `docs/PLANS.md` at the repository root for ExecPlan rules. This plan must be maintained in accordance with that file.

## Purpose / Big Picture

After this change, all-method benchmark runs should stop paying full canonical-text evaluation cost for every permutation when many permutations produce equivalent prediction content. The system will run prediction for each permutation, compute a deterministic evaluation signature, execute canonical evaluation once per unique signature, and reuse that result for all duplicate signatures.

A user can see this working by running all-method on a source with many variants and observing that `total_config_runs_completed` remains the same while `evaluation_runs_executed` is much lower than total configs, with identical ranking metrics for reused configs.

## Surprises & Discoveries

- Observation: all-method currently executes `labelstudio_benchmark(...)` once per config via `_run_all_method_config_once(...)`, so prediction and evaluation are tightly coupled at config granularity.
  Evidence: `cookimport/cli.py` in `_run_all_method_config_once` calls `labelstudio_benchmark(...)` and then requires per-config `eval_report.json`.

- Observation: evaluate-only mode already exists and is production-ready through prediction records.
  Evidence: `labelstudio_benchmark` supports `--predictions-in` (skip prediction generation) and `--execution-mode predict-only` (generate predictions without evaluation).

- Observation: canonical alignment cache exists but only avoids some inner alignment recompute; it does not avoid full per-config evaluation orchestration.
  Evidence: `evaluate_stage(...)` receives `alignment_cache_dir`; all-method still schedules full benchmark invocation for each config.

- Observation: all-method test doubles needed explicit predict-phase and evaluate-phase artifact writers because the runtime now invokes `labelstudio_benchmark(...)` in two distinct modes.
  Evidence: Existing tests that only wrote `eval_report.json` failed until fakes emitted `prediction-records.jsonl` for predict-only and handled `predictions_in` evaluate-only.

- Observation: deterministic signature dedupe is sensitive to all prediction payload fields, including `block_features`.
  Evidence: tests expecting two configs to dedupe produced three signatures until the fake prediction payload stabilized `block_features` across intended duplicates.

## Decision Log

- Decision: Reuse should be keyed by a deterministic evaluation signature built from prediction content + gold identity + evaluator settings, not by config slug.
  Rationale: Different permutations can produce identical predictions; config identity is too strict and misses reuse opportunities.

## Context and Orientation

Current all-method flow lives in `cookimport/cli.py` and has three important layers:

- `_run_all_method_benchmark_multi_source(...)`: schedules source jobs and aggregates one combined report.
- `_run_all_method_benchmark(...)`: schedules config variants for one source, currently each config runs prediction+evaluation.
- `_run_all_method_config_once(...)`: executes one config by calling `labelstudio_benchmark(...)` and reading `eval_report.json`.

`labelstudio_benchmark(...)` already supports separate execution phases:

- `execution_mode="predict-only"` writes prediction artifacts and skips evaluation.
- `predictions_in=<jsonl>` replays prediction records and runs evaluate-only.
- `evaluate_stage(...)` performs canonical evaluation and writes `eval_report.json`.

Key term definitions used in this plan:

- Evaluation signature: a stable hash representing all inputs that can change evaluation results. If two configs have the same signature, reusing one evaluation result is valid.
- Representative config: one config chosen to execute evaluation for a signature group.
- Signature group: all config rows sharing one evaluation signature.

## Plan of Work

Milestone 1 introduces a prediction-first data model for all-method configs. Refactor `_run_all_method_config_once(...)` into separate prediction and evaluation helpers so config runs can stop after prediction. The prediction helper returns a structured row containing paths to prediction artifacts, timing, run config hash, and enough metadata to compute an evaluation signature.

Milestone 2 introduces deterministic signature generation and grouping. Add a helper that builds a canonical payload from prediction records (or stage-block prediction artifact), selected eval mode, selected sequence matcher, and gold identity metadata, then hashes it (for example SHA-256). Group config rows by this signature inside `_run_all_method_benchmark(...)` after prediction phase completion.

Milestone 3 executes evaluation once per signature group. For each unique signature, run evaluate-only by calling existing `labelstudio_benchmark(...)` with `predictions_in` and reuse the generated metrics for every duplicate config in that signature group. Duplicate rows should still appear in ranked outputs, but their evaluation provenance should point to the representative config.

Milestone 4 adds optional persistent signature-result cache across runs (separate from canonical alignment cache). Write normalized eval summaries keyed by signature under a stable cache root (for example under `data/golden/benchmark-vs-golden/.cache/eval_signature_results/<source_group_key>/`). On cache hit, skip evaluate-only entirely and hydrate config rows from cache.

Milestone 5 updates report contracts and docs. Extend per-source and multi-source report payloads with dedupe counters and reuse provenance fields, then document behavior in benchmark docs.

## Validation and Acceptance

Acceptance criteria are behavior-based:

- For a source with duplicate-signature configs, all configs still appear in report rankings.
- `evaluation_runs_executed` equals the number of unique signatures, not total successful configs.
- Reused configs have metrics identical to their representative config.
- Existing non-all-method `labelstudio-benchmark` behavior remains unchanged.
- Existing canonical alignment cache behavior remains enabled and compatible.
- Codex-farm remains policy-locked off unless existing unlock path is used.

## Idempotence and Recovery

The new flow is additive and rerunnable. If a run is interrupted after prediction phase, rerun all-method normally; prediction artifacts can be regenerated safely. Signature-result cache entries are content-addressed and read-only after write, so repeated runs can reuse them safely. If a cache entry appears corrupt, ignore it and re-run evaluation for that signature, then overwrite with a fresh payload.
