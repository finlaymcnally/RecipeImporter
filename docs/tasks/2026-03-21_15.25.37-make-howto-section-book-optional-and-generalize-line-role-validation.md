# Make HOWTO_SECTION book-optional and generalize line-role validation

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The line-role worker-context hardening already fixed the main outside-recipe heading contamination in `saltfatacidheatcutdown`, but it also exposed a broader policy question: `HOWTO_SECTION` is genuinely useful in some cookbooks and effectively absent in others. After this change, line-role should treat `HOWTO_SECTION` as a high-evidence label that a book can legitimately use zero times. A cold-start worker should default away from subsection structure unless the local packet and the broader book context both support it.

The second goal is to keep Salt Fat Acid Heat as a stress test without turning it into the spec. After this change, the validation story should explicitly require both the hard Salt Fat cutdown and at least one contrasting cookbook that actually uses subsection-like structure. The intended result is a line-role policy that performs better on Salt Fat because it is more general and more conservative, not because it learned Salt Fat-specific quirks.

## Surprises & Discoveries

- Observation: the worker-context hardening achieved its narrow goal, but the benchmark still advertises a broader `HOWTO_SECTION` policy problem.
  Evidence: in `data/golden/benchmark-vs-golden/2026-03-21_14.53.27/single-book-benchmark/saltfatacidheatcutdown/single_book_summary.md`, CodexFarm improved to `strict_accuracy=0.723056` and `macro_f1_excluding_other=0.624992`, yet the per-label table still shows `HOWTO_SECTION` with `gold=0` and `pred=51`.

- Observation: Salt Fat is now mostly useful as a canary for over-structuring, not as a justification for removing the label globally.
  Evidence: the new run cut total line-role flips to `42`, reduced `KNOWLEDGE -> HOWTO_SECTION` to `1`, and corrected the specific headings `Balancing Fat`, `WHAT IS ACID?`, and `Cooking Acids`, but it still has one remaining `KNOWLEDGE -> HOWTO_SECTION` error on `DRESSINGS`.

- Observation: once heading contamination dropped, the next biggest residual problem became outside-recipe prose being promoted to procedure.
  Evidence: the latest Salt Fat flip distribution is dominated by `OTHER -> INSTRUCTION_LINE` (`19`) and `KNOWLEDGE -> INSTRUCTION_LINE` (`6`), which means a `HOWTO_SECTION`-only fix would be incomplete.

- Observation: “book-optional label” is a better abstraction than “rare label.”
  Evidence: the user’s qualitative note matches the benchmark facts: some books may have many recipe-local subsection headings, while Salt Fat’s current cutdown has none in gold. The correct policy is not “predict fewer HOWTO_SECTION everywhere” but “require enough evidence that zero is allowed.”

- Observation: repo-local gold already contains a clean contrast book, so this plan did not need a new helper script or live benchmark to establish the anti-overfit boundary.
  Evidence: `data/golden/pulled-from-labelstudio/saltfatacidheatcutdown/exports/canonical_span_labels.jsonl` contains `HOWTO_SECTION=0`, while `data/golden/pulled-from-labelstudio/seaandsmokecutdown/exports/canonical_span_labels.jsonl` contains `HOWTO_SECTION=111`.

- Observation: explicit `FOR THE ...` / `TO SERVE` headings cannot be gated through `_looks_recipe_title(...)` because that helper intentionally rejects how-to prefixes.
  Evidence: the first implementation pass incorrectly demoted the existing hollandaise test case until `_has_recipe_local_howto_support(...)` was split so explicit how-to prefixes can qualify with local component evidence even though they are not recipe titles.

## Context and Orientation

The active line-role stage lives in `cookimport/parsing/canonical_line_roles.py`. It labels each atomic line with one structural label such as `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `HOWTO_SECTION`, `KNOWLEDGE`, or `OTHER`. The current runtime already uses worker-local packet files under `line-role-pipeline/runtime/line_role/workers/*/in/*.json`, rich hint sidecars under `.../hints/*.md`, and worker-local contrast examples under `.../examples/*.md`.

The just-landed worker-context pass in `docs/plans/2026-03-21_13.24.39-line-role-worker-context-hardening.md` solved the narrow “cold-start worker lacks packet semantics” problem by adding packet summaries, conservative flip policy text, and contrast examples. That change should be treated as a foundation, not re-litigated here. This new plan builds on it and addresses the next layer: how a worker should think about a label that may be absent in some books altogether.

The benchmark evidence that motivates this plan is the Salt Fat single-book run under `data/golden/benchmark-vs-golden/2026-03-21_14.53.27/single-book-benchmark/saltfatacidheatcutdown/`. In that run, CodexFarm beat vanilla and fixed the main `KNOWLEDGE -> HOWTO_SECTION` mistakes, but the summary still reports `HOWTO_SECTION` as `gold=0` and `pred=51`. That means the current system has become much less reckless, but it still assumes subsection structure more often than this book warrants.

Two repository surfaces matter for proving that a future fix is general. First, the benchmark contract in `docs/07-bench/07-bench_README.md` describes how single-book canonical-text runs are stored, compared, and scored. Second, the parsing and LLM docs in `docs/04-parsing/04-parsing_readme.md` and `docs/10-llm/10-llm_README.md` describe the current packet/hint/prompt authority split. Any implementation under this plan should preserve that transport shape and modify policy, priors, and validation rather than redesigning the runtime again.

In this plan, “book-optional label” means a label that the system must be willing to predict zero times for an entire book or cutdown if the evidence says so. “High-evidence, low-default” means the model should only emit the label when multiple strong signals line up, while ambiguous cases should stay in safer labels such as `KNOWLEDGE` or `OTHER`.

## Milestones

### Milestone 1: Measure where `HOWTO_SECTION` actually exists

At the end of this milestone, the repository will have one clear audit surface that answers a novice question: “Which benchmark books or gold exports actually contain `HOWTO_SECTION`, and how often?” The implementation can use existing benchmark outputs, canonical gold labels, or a small repo-owned audit helper, but the result must be deterministic and cheap to re-run. The acceptance for this milestone is a concrete identified contrast book, not a guess.

This milestone matters because without it, every later decision is underdetermined. If the system never measures which books truly use subsection structure, it cannot distinguish “general conservative policy” from “Salt Fat suppression hack.”

### Milestone 2: Make `HOWTO_SECTION` explicitly book-optional in the line-role policy

At the end of this milestone, the line-role policy will allow a packet, shard, or whole book to use zero `HOWTO_SECTION` labels without treating that as suspicious. The implementation can use book-level priors, local packet priors, stronger gating rules, or some combination, but it must make the absence of the label a first-class supported outcome.

The acceptance for this milestone is that the code has one obvious seam where `HOWTO_SECTION` prior strength is determined and one obvious seam where the worker/prompt/hint contract states that subsection structure requires stronger evidence than prose labels. This must be written in repo code and covered by tests, not left as implicit benchmark folklore.

### Milestone 3: Preserve real subsection books while tightening Salt Fat

At the end of this milestone, a contrast validation surface will prove that the policy did not simply delete `HOWTO_SECTION` from the product. The implementation may use a benchmark helper, a deterministic fixture pack, or both. The important thing is that one no-subsection book and one real-subsection book are now part of the proof story.

The acceptance for this milestone is a pair of focused tests or benchmark checks with opposite expectations:
- Salt Fat style lesson headings should stay `KNOWLEDGE` or `OTHER`.
- A real recipe-internal subsection heading in the contrast book should still be able to become `HOWTO_SECTION`.

### Milestone 4: Rebalance the residual outside-recipe structure mistakes

At the end of this milestone, the remaining error family revealed by the latest Salt Fat run should also be addressed: outside-recipe prose should not be promoted into `INSTRUCTION_LINE` just because it uses verbs or imperative phrasing. This milestone is intentionally coupled to the `HOWTO_SECTION` work because both are manifestations of the same over-structuring tendency.

The acceptance for this milestone is new prompt/hint/test wording that makes both boundaries explicit: outside-recipe prose does not become `HOWTO_SECTION` without local recipe structure, and outside-recipe guidance does not become `INSTRUCTION_LINE` without local recipe ownership.

### Milestone 5: Validate without overfitting

At the end of this milestone, deterministic tests should pass and the benchmark story should be explicit about what was and was not run. The default validation path is deterministic: targeted tests, any new audit helper, and replay or compare steps that do not spend tokens. A live benchmark rerun can be added only after explicit user approval, and if run it must cover both Salt Fat and the contrast book.

The acceptance for this milestone is not “best-ever Salt Fat score.” It is “the policy is coherent across at least two meaningfully different books, and any live benchmark improvement is explained in those general terms.”

## Plan of Work

Start by adding a deterministic audit seam for `HOWTO_SECTION` demand. This can live in `cookimport/parsing/canonical_line_roles.py`, a small benchmark helper, or a dedicated parsing/bench utility, but it should read existing gold or benchmark artifacts and summarize whether a source meaningfully uses subsection structure. The output does not need to be complicated: counts, sources, and one or two sample lines are enough. What matters is that the repo can now distinguish “this book never uses the label” from “the current run just happened not to predict it.”

Once that audit exists, update the line-role policy in `cookimport/parsing/canonical_line_roles.py`. The exact mechanism is open, but the code should make `HOWTO_SECTION` prior strength a first-class concept. For example, packet summaries or shard-level metadata could carry a field such as `howto_section_prior=absent|weak|available`, and the prompt/hint writer could use that field to tell the worker how skeptical to be. The decision should not depend on memorized book titles; it must derive from text and gold/benchmark evidence available in the repo.

Then update the worker-facing contract in `cookimport/llm/canonical_line_role_prompt.py` and any prompt template files under `llm_pipelines/prompts/`. The wording should say plainly that `HOWTO_SECTION` is optional at the book level, that the absence of subsection headings is normal in some books, and that recipe-internal headings need stronger evidence than prose labels. In the same pass, strengthen the negative rules around outside-recipe `INSTRUCTION_LINE` so the plan addresses the current leading residual flip family rather than only the old heading issue.

After the policy and prompt updates, extend tests in `tests/parsing/test_canonical_line_roles.py`. Add one fixture family for “book with no real subsection headings” and one for “book with genuine recipe-internal subsection headings.” Keep those fixtures compact and local rather than reconstructing whole benchmark books in tests. The goal is to lock the decision boundary, not to duplicate benchmark corpora inside pytest.

If the benchmark helper surface is the easiest place to keep the anti-overfit contract visible, add a small deterministic benchmark assertion or report helper under the existing benchmark/test seams. This could be as simple as a helper that reads eval reports and asserts that one selected contrast book has nonzero gold `HOWTO_SECTION` while Salt Fat’s selected cutdown has zero. Keep it deterministic and cheap.

Finally, update the short docs in `docs/04-parsing/04-parsing_readme.md`, `docs/10-llm/10-llm_README.md`, and if useful `docs/07-bench/07-bench_README.md`. The documentation should say that `HOWTO_SECTION` is now treated as a book-optional, high-evidence label and that benchmark validation for line-role changes must include both a hard no-subsection case and a real-subsection contrast case.

## Validation and Acceptance

The implementation is acceptable only if all of the following are true.

The code now supports `HOWTO_SECTION` as a book-optional label. A novice reading the relevant parsing/prompt code should be able to identify where that prior is computed and how it affects worker guidance.

The new policy is not Salt Fat-specific. No mechanism should key off book title or one benchmark path by name. The acceptance story must be expressed in general terms such as “books with no subsection evidence may emit zero `HOWTO_SECTION`” and “books with strong recipe-internal subsection evidence may still emit it.”

The tests cover both sides of the boundary. There must be at least one compact fixture where a lesson or prose heading stays `KNOWLEDGE`/`OTHER`, and at least one compact fixture where a real recipe-internal heading can still become `HOWTO_SECTION`.

The residual outside-recipe procedure boundary is tighter. New tests or prompt assertions must show that generic or advisory prose outside recipe context does not become `INSTRUCTION_LINE` without stronger local evidence.

The benchmark story is anti-overfit by design. Deterministic validation must explicitly mention Salt Fat plus at least one contrast source. If a live benchmark rerun is approved, the plan must record both runs and explain the result in cross-book terms, not only in Salt Fat terms.

No live-token benchmark or production-quality run is required unless the user explicitly approves model spend for that rerun.

## Idempotence and Recovery

This plan should be implemented additively. The current packet/hint/prompt transport already works and should remain stable. Prefer adding explicit priors, audit helpers, and tests over refactoring the runtime again.

If an attempted fix improves Salt Fat by suppressing `HOWTO_SECTION` everywhere, treat that as a regression, not a win. The safe rollback is to revert only the policy seam or prompt wording that flattened the contrast-book subsection cases, then rerun the compact boundary tests before touching broader benchmark validation.

If the chosen contrast book turns out not to be a reliable subsection case, do not force the implementation around it. Update this plan, select a better contrast source deterministically, and continue. The plan’s principle is “at least one real-subsection contrast case,” not loyalty to one guessed book.
