# Recover line-role knowledge, title, instruction, and variant quality after the 2026-03-22_12.08.53 Salt Fat rerun

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan follows the already-implemented fix in `docs/plans/2026-03-22_11.32.34-stop-generic-to-make-steps-from-leaking-into-recipe-variant.md`, but it repeats the runtime context needed to execute the next pass from scratch.

## Purpose / Big Picture

After this change, rerunning the Salt Fat single-book canonical-text benchmark should still keep the fixed Caesar-dressing line family correct while recovering the much larger remaining error buckets: real lesson prose should stop collapsing into `OTHER`, front-matter and contents-style headings should stop leaking into `RECIPE_TITLE`, obvious imperative recipe steps should stop falling to `OTHER`, and short `Variation`/`Variations` runs should recover without reopening the generic `To make the ...` leak.

The visible proof is concrete. In `data/golden/benchmark-vs-golden/2026-03-22_12.08.53/single-book-benchmark/saltfatacidheatcutdown/`, CodexFarm reached `strict_accuracy = 0.693042` and `macro_f1_excluding_other = 0.797862`, but `wrong_label_lines.jsonl` still shows `450` wrong rows. The dominant bucket is `KNOWLEDGE -> OTHER` with `332` rows. The next important families are `OTHER -> KNOWLEDGE` (`39`), `OTHER -> RECIPE_TITLE` (`19`), `INSTRUCTION_LINE -> OTHER` (`13`), and `RECIPE_VARIANT -> OTHER` (`6`). After implementation, those families should shrink materially while the already-fixed Caesar rows remain `INSTRUCTION_LINE`, `RECIPE_NOTES`, and `RECIPE_NOTES`.

This plan does not attempt to redesign the whole line-role stage or change benchmark scoring. It is scoped to the line-role worker contract, prompt/examples, and the narrow deterministic sanitizer/rescue seams in `cookimport/parsing/canonical_line_roles.py` that can correct obvious Codex misses without reintroducing earlier regressions.

## Surprises & Discoveries

- Observation: the benchmark is not primarily blocked by deterministic parser mistakes anymore.
  Evidence: joining `wrong_label_lines.jsonl` with `line-role-pipeline/line_role_predictions.jsonl` shows `329 / 332` `KNOWLEDGE -> OTHER` rows, all `39` `OTHER -> KNOWLEDGE` rows, all `13` `INSTRUCTION_LINE -> OTHER` rows, and `17 / 19` `OTHER -> RECIPE_TITLE` rows were emitted directly by Codex with `reason_tags=["codex_line_role"]`.

- Observation: the current line-role worker posture is intentionally conservative in exactly the packet modes that dominate the remaining misses.
  Evidence: `_line_role_default_posture(...)` in `cookimport/parsing/canonical_line_roles.py` tells `lesson_prose` packets to default to `OTHER` unless reusable cooking explanation is clear, and tells `front_matter_navigation` packets to preserve structure only with strong local recipe evidence.

- Observation: real reusable lesson prose is falling on the wrong side of that conservative posture.
  Evidence: current wrong rows include declarative teaching lines such as `Salt, Fat, Acid, and Heat were the four elements that guided basic decision making ...` and many surrounding salt-science paragraphs, all predicted as `OTHER` despite gold `KNOWLEDGE`.

- Observation: front matter and contents-style recipe-name lists are still being overpromoted to recipe titles by Codex, not by deterministic title rescue.
  Evidence: rows such as `The Four Elements of Good Cooking`, `Winter: Roasted Radicchio and Roquefort`, and nearby menu/list entries are wrong as `OTHER -> RECIPE_TITLE`, and the joined line table marks them `decided_by="codex"`.

- Observation: one smaller regression is parser-owned, not prompt-owned.
  Evidence: in the current run, in-recipe `Variation` plus `To add a little heat, add 1 teaspoon minced jalapeño.` was emitted as `RECIPE_VARIANT` by Codex but sanitized to `OTHER` / `INSTRUCTION_LINE` with `reason_tags=["codex_line_role", "sanitized_variant_without_local_support"]`.

- Observation: benchmark `line_index` and line-role `atomic_index` are not always the same row id.
  Evidence: `line_index=18` in `wrong_label_lines.jsonl` maps to `line_role_prediction_atomic_index=23` in `line-role-pipeline/joined_line_table.jsonl`, so future debugging should use `joined_line_table.jsonl` instead of assuming those ids are interchangeable.

- Observation: the decisive knowledge misses were mostly blocked by the first-person narrative veto, not by missing domain vocabulary alone.
  Evidence: after expanding the explanation-cue detector (`distinguish`, `guided`, `patterns`) but before changing `_looks_narrative_prose(...)`, Salt Fat lesson rows like `As I improved, I began to detect the nuances ...` still failed because `_FIRST_PERSON_SINGULAR_RE` short-circuited them to memoir.

- Observation: rescuing Codex `OTHER` to `INSTRUCTION_LINE` must stay stricter than the general outside-span instruction heuristic.
  Evidence: an initial `OTHER -> INSTRUCTION_LINE` rescue based on `_looks_instructional_neighbor(...)` alone incorrectly upgraded prose-only controls; tightening the rescue to direct imperative starts / instruction tags preserved the cabbage-step recovery without turning inside-recipe prose paragraphs into instructions.

## Context and Orientation

The relevant code is concentrated in `cookimport/parsing/canonical_line_roles.py`. That file does three different jobs that matter here. First, it builds the line-role worker contract: packet summaries, default posture, strong/weak signals, flip policy, and example-file selection. Second, it holds deterministic label helpers and sanitizers such as `_looks_recipe_title_with_context(...)`, `_outside_recipe_knowledge_label_allowed(...)`, `_instruction_line_label_allowed(...)`, `_variant_heading_label_allowed(...)`, and `_variant_label_allowed(...)`. Third, it embeds worker-facing example text used to keep the line-role prompt aligned with parser policy.

There are also two prompt asset files under `llm_pipelines/prompts/`: `canonical-line-role-v1.prompt.md` and `line-role.canonical.v1.prompt.md`. These should stay aligned with the parser-owned contract when worker posture changes. The prompt assets are not the only authority, but they are still the visible worker-facing explanation surface for benchmark runs and prompt previews.

The benchmark evidence for this plan comes from `data/golden/benchmark-vs-golden/2026-03-22_12.08.53/single-book-benchmark/saltfatacidheatcutdown/`. The most useful files are:

- `single_book_summary.md` for the top-line metrics.
- `codexfarm/wrong_label_lines.jsonl` for the error family counts and example texts.
- `codexfarm/line-role-pipeline/line_role_predictions.jsonl` for the actual emitted labels and reason tags.
- `codexfarm/line-role-pipeline/joined_line_table.jsonl` for the authoritative mapping between benchmark `line_index`, line-role `atomic_index`, predicted label, gold label, and `decided_by`.

The current major failure families are:

- real knowledge prose predicted as `OTHER`,
- front-matter or contents-like headings predicted as `RECIPE_TITLE`,
- imperative recipe steps predicted as `OTHER`,
- short variant headings or variant follow-up lines predicted or sanitized as `OTHER` / `INSTRUCTION_LINE`.

Representative examples from the current run include:

- `KNOWLEDGE -> OTHER`: `Salt, Fat, Acid, and Heat were the four elements that guided basic decision making ...`
- `OTHER -> RECIPE_TITLE`: `The Four Elements of Good Cooking`
- `INSTRUCTION_LINE -> OTHER`: `Quarter the cabbage through the core. Use a sharp knife to cut the core out at an angle. ...`
- `RECIPE_VARIANT -> OTHER`: `Variations`
- `RECIPE_VARIANT -> INSTRUCTION_LINE`: `To add a little heat, add 1 teaspoon minced jalapeño.`

The important distinction for implementers is that most of these are not deterministic misclassifications from the current parser seeds. They are Codex worker outputs that the current prompt/hint posture is failing to steer, with only the short in-recipe `Variation` case currently being over-sanitized by the parser.

## Plan of Work

Begin by tightening the worker-facing contract in `cookimport/parsing/canonical_line_roles.py` and the two line-role prompt assets. The goal is not to make the worker less conservative everywhere. The goal is to make the current packet modes more explicit about what counts as reusable cooking knowledge, what counts as front matter or contents navigation, and what counts as a real recipe step even when recipe-span ownership is uncertain. Add or revise example snippets so the worker sees direct contrasts for the exact families that are failing now: declarative teaching prose that should be `KNOWLEDGE`, front-matter recipe-name lists that should stay `OTHER`, imperative prep prose that should stay `INSTRUCTION_LINE`, and short `Variation` runs that should stay `RECIPE_VARIANT`.

Then update the parser-owned rescues in `cookimport/parsing/canonical_line_roles.py`. The knowledge seam should become slightly more tolerant for outside-recipe explanatory prose clusters, especially when a packet is already in lesson-prose mode or has neighboring knowledge evidence. The title seam should become stricter for outside-span recipe-title promotion so isolated title-like strings in contents/front matter do not survive without strong recipe-local support. The instruction seam should rescue obvious imperative recipe steps that Codex incorrectly flattens to `OTHER` when there is nearby recipe-local structure. The variant seam should let `Variation` / `Variations` headings and their immediate short follow-up modification lines survive when they are clearly recipe-local adjustments, without weakening the recent generic `To make the ...` fix.

After the code and prompt surfaces line up, extend `tests/parsing/test_canonical_line_roles.py` with exact benchmark-shaped regressions for the current four families. Add both deterministic and Codex-side tests where appropriate. The variant tests should explicitly cover the in-recipe `Variation` sanitizer failure and the outside-span `Variations` block that currently falls to `OTHER`. The knowledge, title, and instruction tests should use the exact Salt Fat texts from `joined_line_table.jsonl` so the prompt-facing seam is grounded in the real benchmark.

If behavior changes materially, update the short line-role notes in `docs/04-parsing/04-parsing_readme.md` so the parser reference stays aligned with the worker contract. The doc should mention the positive `KNOWLEDGE`/`front_matter_navigation`/`RECIPE_TITLE` boundaries only where the active contract actually changed.
