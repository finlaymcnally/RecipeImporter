# Remove Recipe Legacy Transport And Compatibility Artifacts

## Goal

Recipe already used direct exec for live work, but the code still carried one recipe-only adapter back into legacy CodexFarm process mode and still wrote compatibility `recipe_correction/{in,out}` mirrors. The goal was to delete both and move the remaining readers onto current runtime artifacts.

## Checklist

- [x] Remove the recipe-only legacy runner adapter from `cookimport/llm/codex_farm_orchestrator.py`
- [x] Stop validating recipe direct exec through `codex-farm pipelines list`; load schema/prompt assets directly from repo-owned pipeline files
- [x] Stop writing `raw/llm/<workbook_slug>/recipe_correction/{in,out}`
- [x] Write shard request payloads under `raw/llm/<workbook_slug>/recipe_phase_runtime/inputs/`
- [x] Keep `recipe_correction_audit/` as the only per-recipe debug summary surface and embed serialized input/output payloads there
- [x] Move follow-up, prompt-artifact, and Label Studio readers to `recipe_phase_runtime` plus audit
- [x] Update recipe tests to use direct-exec test doubles instead of the legacy farm runner
- [x] Update short docs to describe the new runtime/debug seam

## Final State

Recipe runtime now has one owned story:

- live execution: direct `codex exec`
- shard debug inputs: `recipe_phase_runtime/inputs/*.json`
- validated shard outputs: `recipe_phase_runtime/proposals/*.json`
- per-shard prompt/event/cost files: `recipe_phase_runtime/workers/.../shards/...`
- per-recipe summary/debug file: `recipe_correction_audit/*.json`

The deleted seam is the old compatibility pair `recipe_correction/{in,out}` plus the adapter that translated recipe direct-exec calls back into legacy farm process runs.
