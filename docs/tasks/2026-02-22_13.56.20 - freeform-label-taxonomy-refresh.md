# Task: Freeform label taxonomy refresh

## Problem statement

The freeform Label Studio taxonomy used `TIP` / `NOTES` / `VARIANT`, which is ambiguous for annotation and mismatched with the desired golden-set naming.

## Acceptance criteria

- Freeform canonical labels are exactly:
  - `RECIPE_TITLE`, `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `YIELD_LINE`, `TIME_LINE`, `RECIPE_NOTES`, `RECIPE_VARIANT`, `KNOWLEDGE`, `OTHER`
- Label config order matches the canonical order above.
- Legacy labels normalize correctly (`TIP`, `NOTES`/`NOTE`, `VARIANT`, `YIELD`, `TIME`, `NARRATIVE`).
- Freeform eval chunk mapping emits the renamed labels (`RECIPE_NOTES`, `RECIPE_VARIANT`, `KNOWLEDGE`).
- CLI project-scope inference still recognizes both new and legacy freeform label configs.
- Label Studio docs/conventions reflect the new taxonomy.

## Verification steps

- `. .venv/bin/activate && pip install -e .[dev]`
- `. .venv/bin/activate && pytest -q tests/test_labelstudio_freeform.py tests/test_labelstudio_prelabel.py tests/test_labelstudio_benchmark_helpers.py`

## Evidence

## Constraints and decisions

- Preserve backward compatibility by alias normalization instead of dropping old labels.
- Keep app-aligned freeform metrics contracts unchanged (supported-label subset is still structural + `OTHER`).

## Rollback plan

- Revert the taxonomy mapping changes in:
  - `cookimport/labelstudio/label_config_freeform.py`
  - `cookimport/labelstudio/eval_freeform.py`
  - `cookimport/cli.py`
- Revert associated test and docs updates to restore prior naming behavior.
