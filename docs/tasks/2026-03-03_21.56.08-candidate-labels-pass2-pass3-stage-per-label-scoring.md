# Build candidate-label diagnostics plus pass2/pass3 per-label stage scoring for upload bundles

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md` from the repository root.

## Purpose / Big Picture

After this change, upload bundles will stop saying that candidate labels and pass2/pass3 per-label stage scores are unavailable when the run data is present. A reviewer will be able to open `upload_bundle_index.json` and see: (1) candidate-label availability and top candidates for line-role decisions, and (2) real per-label precision/recall/F1 for pass2 and pass3 stages instead of hardcoded `label_scored=false`.

This is benchmark/evaluation-only work. It must not turn on codex-farm or LLM parsing for normal ingestion defaults.

## Surprises & Discoveries

- Observation: The line-role prediction model already has a `candidate_labels` field, but current SeaAndSmoke artifact rows do not contain it.
  Evidence: `cookimport/parsing/canonical_line_roles.py` defines `candidate_labels`, while `.../line_role_predictions.jsonl` rows in `2026-03-03_20.49.14` contain only `label/confidence/decided_by`.

- Observation: Upload-bundle candidate-label ingestion already supports multiple field shapes.
  Evidence: `scripts/benchmark_cutdown_for_external_ai.py` `_upload_bundle_extract_candidate_labels(...)` checks `candidate_labels`, `label_candidates`, `candidates`, `top_candidates`, `candidate_label_scores`, `label_scores`, and `candidate_distribution`.

- Observation: Pass2 and pass3 contract outputs are recipe-structure payloads, not stage label metrics.
  Evidence: `Pass2SchemaOrgOutput` exposes `schemaorg_recipe`, `extracted_ingredients`, `extracted_instructions`, `field_evidence`; `Pass3FinalDraftOutput` exposes `draft_v1`, `ingredient_step_mapping`, `warnings`.

- Observation: The repository already has a block-label evaluation engine with per-label metrics and confusion.
  Evidence: `cookimport/bench/eval_stage_blocks.py::compute_block_metrics(...)` and `evaluate_stage_blocks(...)`.

## Context and Orientation

This work spans five areas:

1. `cookimport/parsing/canonical_line_roles.py` and `cookimport/labelstudio/canonical_line_projection.py`
   These produce `line-role-pipeline/line_role_predictions.jsonl`, which is the source for upload-bundle candidate-label diagnostics.

2. `cookimport/llm/codex_farm_contracts.py` and `cookimport/llm/codex_farm_orchestrator.py`
   These define and run pass2 and pass3 pipelines and are the right place to emit pass-specific stage artifacts.

3. `cookimport/staging/stage_block_predictions.py`
   This defines `stage_block_predictions.v1` payload shape used by benchmark scoring.

4. `cookimport/bench/eval_stage_blocks.py`
   This computes per-label metrics and confusion from stage-block predictions against freeform gold.

5. `scripts/benchmark_cutdown_for_external_ai.py`
   This creates `upload_bundle_v1` and currently hardcodes pass2/pass3 per-label as unavailable.

Terms used below:

- Candidate labels: the allowed label set for a line-role decision (for example `["INGREDIENT_LINE", "OTHER"]`), optionally with ranked scores.
- Stage-block projection: converting a pass output into a per-block label manifest (`stage_block_predictions.v1`) so it can be evaluated by the block metrics engine.
- Pass2 stage scoring: per-label metrics computed from pass2 stage projection versus gold.
- Pass3 stage scoring: per-label metrics computed from pass3 stage projection versus gold.

## Milestones

### Milestone 1: Candidate-label data is always present in line-role artifacts

Ensure new runs emit candidate-label data in `line_role_predictions.jsonl` for all decision modes (`rule`, `codex`, `fallback`). Keep current fields unchanged and additive. If ranked candidates are feasible, include `label_scores`; otherwise at minimum include `candidate_labels`.

Acceptance: a fresh benchmark run has non-zero `rows_with_candidate_labels` in upload-bundle analysis.

### Milestone 2: Pass2 stage projection and per-label metrics

Add pass2 stage projection artifacts derived from pass2 outputs (including `field_evidence`) and evaluate them with the existing stage-block evaluator against the same gold labels used for run scoring.

Acceptance: run output includes a pass2 per-label metrics artifact (JSON) with real precision/recall/F1, and upload bundle consumes it.

### Milestone 3: Pass3 stage projection and per-label metrics

Add pass3 stage projection artifacts and evaluate them with the same evaluator. If pass3 output lacks enough evidence to map blocks reliably, extend pass3 contract/prompt output with additive evidence fields required for projection.

Acceptance: run output includes pass3 per-label metrics artifact (JSON) with real metrics for pass3 stage.

### Milestone 4: Upload-bundle stage-separated comparison consumes real pass2/pass3 metrics

Replace hardcoded unavailable pass2/pass3 per-label entries in `stage_separated_comparison` with loaded per-label metrics from Milestones 2 and 3 when present, while preserving explicit fallback messaging for older runs.

Acceptance: `analysis.stage_separated_comparison.per_label[*].pass2_stage.label_scored` and `.pass3_stage.label_scored` are `true` for a new run with these artifacts.

### Milestone 5: Validation and docs updates

Add tests for candidate-label propagation and stage metric ingestion, run targeted suites, then run one paired benchmark and verify upload-bundle output fields.

Acceptance: tests pass; a new upload bundle shows populated candidate labels and pass2/pass3 per-label metrics.

## Plan of Work

Start with candidate labels because the upload-bundle side already knows how to read them. Make artifact emission explicit and test that emission at the writer boundary (`line_role_predictions.jsonl`) and the bundle boundary (`candidate_label_signal`).

Next implement pass2 stage projection as the lower-risk proof point. Pass2 has explicit `field_evidence`, so mapping to block labels is practical and deterministic. Once pass2 scoring is stable, implement pass3 projection. If pass3 currently cannot support reliable block mapping from existing outputs, add an additive evidence payload in pass3 contract and update prompt/schema accordingly.

After stage artifacts exist, wire upload-bundle builder to read these pass2/pass3 metric artifacts and populate stage-separated per-label sections. Keep backward compatibility by falling back to existing `label_scored=false` messaging when artifacts are absent.
