## Problem statement

Compare & Control chart was a clone of the benchmark trend chart, so it did not reflect Compare/Control settings or currently visible rows in a meaningful way.

## Acceptance criteria

- Compare & Control chart is generated from current visible rows + compare/control state.
- Scatter chart is default implementation for Compare & Control.
- Numeric compare fields render row-level scatter; categorical compare fields render group-mean scatter.
- Split-by produces separate series.
- Chart generation path is extensible for additional chart types.
- Compare/control chart stays blank on load/state-restore until user interaction in Compare & Control controls.
- Existing Compare & Control behavior/tests (controlled weighting, subset filter, reset, secondary metrics) remain green.

## Verification steps

1. `source .venv/bin/activate && pytest tests/analytics/test_stats_dashboard.py -k "compare_control_dynamic_chart or dashboard_js_supports_compare_control_analysis or benchmark_trend_chart_uses_fixed_height"`
2. `source .venv/bin/activate && pytest tests/analytics/test_stats_dashboard.py -k "compare_control and not pixel and not drift"`

## Evidence

- Targeted chart + contract slice: `6 passed, 71 deselected`
- Broader compare/control slice: `6 passed, 71 deselected`
- Full dashboard analytics suite: `77 passed`

## Constraints / gotchas

- Dashboard JS is emitted from `cookimport/analytics/dashboard_render.py` string templates.
- Keep compare/control row selection aligned with existing Previous Runs filter result seam.

## Decision

Implemented a builder-driven compare/control chart-definition path (`buildCompareControlChartDefinition`) with `scatter` as initial chart type, plus a dedicated compare/control chart host renderer. This keeps today’s behavior dynamic while making future chart-type additions low-risk.

- Revert this task diff to restore prior clone-based compare/control chart behavior.
