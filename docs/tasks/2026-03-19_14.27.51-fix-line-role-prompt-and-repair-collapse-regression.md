# Harden line-role repair against degenerate shard output

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The user-visible problem is severe: a Codex-backed line-role run can now appear to succeed while silently validating a nonsensical shard, which then destroys benchmark quality. The specific March 19 failure was a repair-path collapse where every line in every repaired shard became `INGREDIENT_LINE`. After this fix, the canonical line-role stage should still use the current direct `codex exec` runtime and artifact layout, but it must no longer accept a repaired payload merely because it has the right JSON shape and covers every owned `atomic_index`.

The change is correct when an invalid or incomplete shard output is repaired from the authoritative shard rows, when obviously degenerate repaired outputs are rejected instead of promoted, and when the existing deterministic baseline becomes the fail-closed fallback for those rejected shards. Live-token benchmark reruns remain optional and require explicit user approval.

## Surprises & Discoveries

- Observation: the benchmark collapse is driven by line-role misclassification, not by benchmark scoring drift or a knowledge-only failure.
  Evidence: `data/golden/benchmark-vs-golden/2026-03-19_13.22.48/single-offline-benchmark/saltfatacidheatcutdown/single_offline_summary.md` reports CodexFarm `strict_accuracy` around `0.117` while vanilla remains around `0.705`.

- Observation: every line-role shard in the failed run was invalid at first and then accepted after repair.
  Evidence: `data/golden/benchmark-vs-golden/2026-03-19_13.22.48/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/prompt_budget_summary.json` reports `invalid_output_shard_count = 5` and `repaired_shard_count = 5` for line-role.

- Observation: the line-role prompt and both follow-up prompts currently teach the JSON shape with a literal one-row `INGREDIENT_LINE` example.
  Evidence: `llm_pipelines/prompts/line-role.canonical.v1.prompt.md`, `cookimport/llm/canonical_line_role_prompt.py`, `_build_line_role_repair_prompt(...)`, and `_build_line_role_watchdog_retry_prompt(...)` all contain `{"rows":[{"atomic_index":123,"label":"INGREDIENT_LINE"}]}`.

- Observation: line-role repair currently emphasizes validator errors and previous invalid output, but not the authoritative shard rows that actually need relabeling.
  Evidence: `_build_line_role_repair_prompt(...)` in `cookimport/parsing/canonical_line_roles.py` includes validator errors and previous invalid output, while the real shard rows are only passed in the runner payload.

- Observation: the runtime already has a deterministic baseline that safely fills any atomic indices left without validated LLM predictions.
  Evidence: `_label_atomic_lines_internal(...)` builds `deterministic_baseline`, then after shard runtime completion it assigns baseline predictions for any `candidate.atomic_index` missing from `predictions`.

- Observation: the current validator is structural only, so a full-coverage all-one-label payload can pass even when it is obviously wrong.
  Evidence: `_validate_line_role_shard_proposal(...)` checks JSON object shape, owned ids, duplicates, and allowed labels, but does not reject uniform or semantically pathological outputs.

- Observation: hand-written four-line candidate fixtures were too weak for a robust regression test because the deterministic baseline often collapses ambiguous isolated lines to `OTHER`.
  Evidence: the first draft of the regression test only produced two baseline labels, so the new guard did not trigger; switching to the `hollandaise_merged_block.json` fixture produced a realistic multi-label shard and exercised the failure mode correctly.

## Decision Log

- Decision: fix the regression inside `cookimport/parsing/canonical_line_roles.py` and the line-role prompt surfaces rather than in the shared `cookimport/llm/codex_exec_runner.py`.
  Rationale: the runner is only the transport and supervision seam. The failure came from stage-owned wording and acceptance policy, and `docs/10-llm/10-llm_README.md` already places strict JSON repair behavior in the line-role stage code.
  Date/Author: 2026-03-19 / Codex

- Decision: keep the current direct-exec shard runtime, worker artifact layout, and deterministic baseline mechanism.
  Rationale: the bug is not in shard planning or transport. The existing baseline already gives the stage a safe local recovery path; the missing piece is to reject repaired outputs that are structurally valid but clearly degenerate.
  Date/Author: 2026-03-19 / Codex

- Decision: add a narrow fail-closed semantic rejection step instead of trying to solve the entire problem by prompt wording alone.
  Rationale: prompt wording can reduce the chance of collapse, but it cannot guarantee correctness. The stage needs a final guard that prevents catastrophic promotion of a uniform repaired payload, even if the model ignores the prompt.
  Date/Author: 2026-03-19 / Codex

- Decision: acceptance will rely on deterministic tests and artifact-shaped simulations by default, with live benchmark reruns called out as optional follow-up only after explicit approval.
  Rationale: the repo instruction is to avoid token spend without approval. This bug can be reproduced and prevented locally with fake runner outputs and prompt-content assertions.
  Date/Author: 2026-03-19 / Codex

- Decision: keep the first semantic guard conservative and shard-local by comparing repaired label concentration against the same shard's deterministic baseline rather than adding a broad semantic classifier.
  Rationale: the goal is to block catastrophic collapse, not to adjudicate every subtle label disagreement. The deterministic baseline already exists, is local, and is good enough as a sanity rail for obvious all-one-label repairs.
  Date/Author: 2026-03-19 / Codex

## Context and Orientation

Canonical line-role labeling is the stage that assigns one label such as `RECIPE_TITLE`, `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `KNOWLEDGE`, or `OTHER` to each atomic line in the book. In this repository, an “atomic line” is the smallest line-like record emitted by `cookimport/parsing/recipe_block_atomizer.py` and consumed by `cookimport/parsing/canonical_line_roles.py`.

The active live line-role surface is the direct-exec shard runtime in `cookimport/parsing/canonical_line_roles.py`. That file plans contiguous shards, writes worker-local task files under `line-role-pipeline/runtime/line_role/workers/*/in/*.json`, invokes `codex exec` through `cookimport/llm/codex_exec_runner.py`, validates the returned JSON, and optionally runs one repo-owned repair attempt and one watchdog-triggered retry attempt. The prompt text for the first attempt is assembled by `cookimport/llm/canonical_line_role_prompt.py` and the file-backed asset `llm_pipelines/prompts/line-role.canonical.v1.prompt.md`.

The March 19 failure matters because structural validation alone is not enough. `_validate_line_role_shard_proposal(...)` currently proves only that the output is shaped correctly and covers the shard’s owned atomic indices. It does not prove that the labels are grounded in the shard’s text. In the failed benchmark, the initial prompt returned only one owned row, the repair prompt repeated that one-row `INGREDIENT_LINE` example, the repair output expanded that example across the full shard, and the stage accepted the result because it covered every owned id with an allowed label.

The stage already has a safe backstop. `_label_atomic_lines_internal(...)` computes a deterministic baseline before any live LLM work. If a shard never produces a validated proposal, the stage falls back to those baseline labels for every missing atomic index. The missing behavior is to keep semantically degenerate repaired outputs in the invalid state so that existing fallback path can do its job.

The relevant files are:

- `llm_pipelines/prompts/line-role.canonical.v1.prompt.md`: the file-backed wrapper prompt for live line-role shards.
- `cookimport/llm/canonical_line_role_prompt.py`: Python fallback and prompt-builder helpers, plus the file prompt wrapper used by prompt preview and tests.
- `cookimport/parsing/canonical_line_roles.py`: shard planning, deterministic baseline creation, direct-exec launch, output validation, repair prompt construction, watchdog retry prompt construction, repair-status writing, and proposal promotion.
- `tests/parsing/test_canonical_line_roles.py`: the main behavior and runtime regression suite for canonical line-role.
- `tests/llm/test_prompt_preview.py`: prompt-preview and reconstructed artifact coverage when line-role prompts change shape.
- `docs/10-llm/10-llm_README.md`: the current operator-facing line-role runtime contract that must stay accurate after the change.

## Plan of Work

The first edit is to neutralize the prompt anchoring in all line-role prompt surfaces. Update `llm_pipelines/prompts/line-role.canonical.v1.prompt.md` and `cookimport/llm/canonical_line_role_prompt.py` so the primary prompt still defines the exact JSON object shape, but stops using the one-row `INGREDIENT_LINE` exemplar. Use either schema-in-words wording or a neutral multi-row placeholder example that does not bias a shard toward one label. Apply the same neutralization to `_build_line_role_repair_prompt(...)` and `_build_line_role_watchdog_retry_prompt(...)` in `cookimport/parsing/canonical_line_roles.py`.

The second edit is to ground repair and watchdog retry on the actual shard rows. Add a small helper in `cookimport/parsing/canonical_line_roles.py` that renders a compact text block from `shard.input_payload["rows"]`. Use that helper inside both `_build_line_role_repair_prompt(...)` and `_build_line_role_watchdog_retry_prompt(...)` so the model sees the exact rows it must relabel. Keep validator errors and previous invalid output in the repair prompt, but demote them to warning context. The authoritative shard rows must be the main evidence, and the prompt must explicitly say “recompute the full shard from these rows; do not copy the previous output.”

The third edit is to add a narrow degenerate-output rejection step in the direct worker runtime, after JSON parsing and structural validation but before proposal promotion. This step should be stage-owned, local to `cookimport/parsing/canonical_line_roles.py`, and intentionally conservative. The concrete rule for this fix is: if a repair or watchdog-retry result produces a uniform or near-uniform label distribution that is grossly inconsistent with the shard’s deterministic baseline, reject it as invalid. The safest first version is to compare the repaired payload’s label diversity against the deterministic baseline for that same shard and only reject clear catastrophes, such as a large shard that becomes entirely `INGREDIENT_LINE` even though the deterministic baseline contains multiple distinct labels. Record a machine-readable validation error such as `pathological_uniform_label_output:INGREDIENT_LINE` in `repair_validation_errors`, `watchdog_retry_status.json`, `status.json`, and the proposal wrapper metadata.

The fourth edit is to preserve fail-closed behavior. When the new semantic rejection step fires, do not promote the repaired payload into `payload`, do not set `proposal_status = "validated"`, and do not record the run as a successful watchdog example. Leave the proposal invalid so the existing missing-prediction path in `_label_atomic_lines_internal(...)` fills the shard from `deterministic_baseline`. This keeps the recovery behavior additive and uses code that already exists and is already testable.

The fifth edit is to add regression coverage in `tests/parsing/test_canonical_line_roles.py`. One test should assert that `build_canonical_line_role_file_prompt(...)` no longer hardcodes the single-row `INGREDIENT_LINE` exemplar. Another should simulate the March 19 failure: an initial response with only one owned row labeled `INGREDIENT_LINE`, followed by a repair response that labels every row `INGREDIENT_LINE`. The expected result is that the repair is rejected, the shard remains invalid, and final predictions for that shard come from the deterministic baseline rather than the degenerate repair payload. Add one focused assertion that the new error code is written into the repair status or proposal metadata. If prompt-preview expectations change, update the smallest necessary assertions in `tests/llm/test_prompt_preview.py`.

The sixth edit is to update the short runtime docs after the code lands. `docs/10-llm/10-llm_README.md` should note two facts: repair and watchdog retry prompts now restate the authoritative shard rows, and structurally valid but pathological repaired outputs are rejected so the stage falls back to deterministic labels instead of silently promoting nonsense.

## Validation and Acceptance

Acceptance is behavior-first:

- A copied-example invalid first response such as `{"rows":[{"atomic_index":123,"label":"INGREDIENT_LINE"}]}` must no longer be the canonical example taught by the primary, repair, or watchdog-retry prompts.
- A repair attempt must be instructed from authoritative shard rows, not merely from owned ids and the bad output.
- A repaired or retried payload that collapses a large diverse shard into one label must be rejected as invalid instead of promoted.
- When that rejection happens, the final returned predictions for the shard must come from the deterministic baseline path that already exists in `_label_atomic_lines_internal(...)`.
- Focused canonical line-role tests must fail before the implementation and pass after it.
- The runtime path must continue writing the same shard-local artifacts, especially `repair_prompt.txt`, `repair_status.json`, `repair_last_message.json`, `watchdog_retry_status.json`, `proposal.json`, and `status.json`.
- No live benchmark rerun is required for completion of the code change, but if the user later approves token spend, rerunning the Salt Fat Acid Heat single-offline benchmark should show that the run no longer craters into almost-all-`INGREDIENT_LINE` output.

## Idempotence and Recovery

These edits are safe to repeat. The prompt-asset and Python prompt-builder changes are plain text updates, the runtime validation change is local to line-role, and the test additions are additive. If an intermediate attempt makes the repair prompt too verbose or breaks prompt-preview assertions, revert only the latest prompt wording change and rerun the focused tests. If the semantic rejection rule is too aggressive and blocks plausible repaired shards, narrow only that helper and rerun the targeted repair regression test plus the full `test_canonical_line_roles.py` module. There is no migration or destructive data rewrite involved.

Do not delete prior benchmark artifacts. They are the evidence for this regression and are useful for reviewing whether the new tests truly capture the failure mode.
