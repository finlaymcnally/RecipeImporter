# Split Non-Recipe Artifacts Into Seed Routing, Final Authority, and Review Status

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

After this change, a human should be able to answer three separate questions without reading code or reverse-engineering one overloaded JSON file.

The first question is, “What did deterministic Stage 7 decide before the knowledge model ran?” The second question is, “What outside-recipe blocks are actually final truth that downstream tools may trust?” The third question is, “What was reviewed, skipped, excluded, changed, or left unresolved during knowledge review?” Right now [`08_nonrecipe_spans.json`](/home/mcnal/projects/recipeimport/cookimport/staging/writer.py#L1176) tries to answer all three at once. The result is a misleading primary artifact whose name sounds canonical but whose contents mix seed routing, excluded junk, reviewed final authority, and unreviewed seed-kept fallback rows.

After this refactor, the non-recipe output contract will be split into separate canonical artifacts for seed routing, final authority, and review status. Downstream scoring, Label Studio projection, and reviewer tooling will consume only the final-authority artifact when they need truth. Humans debugging routing or runtime incompleteness will open the seed-routing or review-status artifacts instead. The visible proof is that opening one artifact tells one story, and no consumer needs to guess whether a category map is provisional or final.

## Context and Orientation

The “non-recipe” path begins after label-first regrouping has separated recipe spans from everything outside recipes. The deterministic Stage 7 builder in [`cookimport/staging/nonrecipe_stage.py`](/home/mcnal/projects/recipeimport/cookimport/staging/nonrecipe_stage.py) currently creates an object named `NonRecipeStageResult`. That object already contains several distinct layers of information:

- Seed routing. This means the deterministic pre-knowledge-review understanding of which outside-recipe blocks are obvious junk and which ones should be sent to the knowledge stage for semantic review.
- Review exclusion. This means outside-recipe blocks that can immediately be treated as final `other` because they are obvious junk such as navigation, endorsements, page furniture, or publishing matter.
- Final authority. This means the subset of outside-recipe blocks whose categories are final truth because they were either excluded upstream or actually reviewed and finalized by the knowledge stage.
- Unreviewed fallback. This means review-eligible blocks that still carry a practical fallback category, but that category is not reviewed semantic truth and must not be trusted by downstream scoring or projection.

The current writer flattens all of those into [`08_nonrecipe_spans.json`](/home/mcnal/projects/recipeimport/cookimport/staging/writer.py#L1176) and [`09_knowledge_outputs.json`](/home/mcnal/projects/recipeimport/cookimport/staging/writer.py#L1361). Those files therefore answer too many questions at once. They are currently consumed, documented, and mentally modeled as if they were one clean contract. That mismatch is the root problem.

Several runtime readers already know better. [`cookimport/staging/stage_block_predictions.py`](/home/mcnal/projects/recipeimport/cookimport/staging/stage_block_predictions.py#L87) and the Label Studio projection code in [`cookimport/labelstudio/ingest.py`](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest.py) already use the authoritative subset when they need final truth. The plan below turns that implicit discipline into an explicit contract.

The key files involved in this refactor are:

- [`cookimport/staging/nonrecipe_stage.py`](/home/mcnal/projects/recipeimport/cookimport/staging/nonrecipe_stage.py), which defines the current in-memory stage result.
- [`cookimport/staging/import_session.py`](/home/mcnal/projects/recipeimport/cookimport/staging/import_session.py), which builds Stage 7 state, runs optional knowledge review, and passes the final result to the writers.
- [`cookimport/llm/codex_farm_knowledge_orchestrator.py`](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_orchestrator.py), which refines review-eligible non-recipe blocks and writes knowledge-stage runtime artifacts.
- [`cookimport/staging/writer.py`](/home/mcnal/projects/recipeimport/cookimport/staging/writer.py), which serializes the current mixed artifacts.
- [`cookimport/staging/stage_block_predictions.py`](/home/mcnal/projects/recipeimport/cookimport/staging/stage_block_predictions.py), which must consume only final-authority truth.
- [`cookimport/labelstudio/ingest.py`](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest.py), which must also consume only final-authority truth.
- [`docs/05-staging/05-staging_readme.md`](/home/mcnal/projects/recipeimport/docs/05-staging/05-staging_readme.md) and [`docs/01-architecture/01-architecture_README.md`](/home/mcnal/projects/recipeimport/docs/01-architecture/01-architecture_README.md), which define the human-facing contract and must be updated when the implementation lands.

For this plan, the exact target artifact set is:

- `08_nonrecipe_seed_routing.json`
  This is the canonical deterministic Stage 7 output before optional knowledge review. It answers only routing questions.

- `08_nonrecipe_review_exclusions.jsonl`
  This remains the canonical row-level ledger for excluded obvious-junk outside-recipe blocks.

- `09_nonrecipe_authority.json`
  This is the canonical final truth artifact for downstream consumers. It contains only blocks and spans that are actually authoritative.

- `09_nonrecipe_review_status.json`
  This is the canonical runtime-status artifact for reviewed, skipped, changed, and unresolved review-eligible rows.

- `09_nonrecipe_debug_rollup.json`
  This is optional. If kept, it is debug-only and must never be the primary truth surface. It may contain a combined inspection view for humans.

The existing artifact names `08_nonrecipe_spans.json` and `09_knowledge_outputs.json` are removed from the live write path at the end of this plan. If any old fixtures or narrow tooling still need to read them during migration, that compatibility must be read-side only and explicitly temporary.

## Plan of Work

The first milestone is to split the in-memory model. Start in [`cookimport/staging/nonrecipe_stage.py`](/home/mcnal/projects/recipeimport/cookimport/staging/nonrecipe_stage.py). Replace the single overloaded `NonRecipeStageResult` payload model with three plain-language components that match the contract above. The recommended structure is:

- `SeedNonRecipeRouting`
  This object owns deterministic Stage 7 routing only. It contains seed spans, seed category maps if still useful for debugging, review eligibility, exclusions, exclusion reasons, and block previews.

- `FinalNonRecipeAuthority`
  This object owns only the categories and spans that downstream readers may trust. Excluded final `other` and reviewed final `knowledge|other` belong here. Unreviewed review-eligible rows do not.

- `NonRecipeReviewStatus`
  This object owns process-state facts. It contains counts, authority mode, reviewed and unreviewed block indices, changed blocks, reason-code rollups, and any distinction between “review not attempted,” “review incomplete,” and “review completed.”

Keep one wrapper object if it materially simplifies function signatures, but the wrapper must expose those three components explicitly and stop encouraging callers to grab a generic “block_category_by_index” without thinking about authority level.

The second milestone is to update stage construction and refinement. In [`cookimport/staging/import_session.py`](/home/mcnal/projects/recipeimport/cookimport/staging/import_session.py), Stage 7 should now return seed routing immediately. Optional knowledge refinement in [`cookimport/llm/codex_farm_knowledge_orchestrator.py`](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_orchestrator.py) should then produce a new `FinalNonRecipeAuthority` plus an updated `NonRecipeReviewStatus`. The knowledge stage must continue to refine only review-eligible rows; excluded obvious-junk rows remain final `other` from the start.

The third milestone is the writer split. In [`cookimport/staging/writer.py`](/home/mcnal/projects/recipeimport/cookimport/staging/writer.py), replace `write_nonrecipe_stage_outputs()` and `write_knowledge_outputs_artifact()` with a clearer writer surface. The recommended writer functions are:

- `write_nonrecipe_seed_routing_artifact(...)`
- `write_nonrecipe_review_exclusion_ledger(...)`
- `write_nonrecipe_authority_artifact(...)`
- `write_nonrecipe_review_status_artifact(...)`
- optional `write_nonrecipe_debug_rollup_artifact(...)`

The writer implementation must make one file answer one question. In particular:

- `08_nonrecipe_seed_routing.json` may contain seed spans, review-eligible spans, excluded spans, review-exclusion reasons, and block preview text, but it must not publish a broad final-authority category map.
- `09_nonrecipe_authority.json` may contain only authoritative spans, authoritative block categories, final-authority block indices, and the minimum notes needed for downstream trust.
- `09_nonrecipe_review_status.json` may contain reviewed counts, unreviewed counts, changed-block details, authority mode, and rollups of reviewer reason codes and failure or incompleteness states.

The fourth milestone is reader cutover. In [`cookimport/staging/stage_block_predictions.py`](/home/mcnal/projects/recipeimport/cookimport/staging/stage_block_predictions.py) and [`cookimport/labelstudio/ingest.py`](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest.py), remove any dependency on mixed category maps. These readers must consume only `FinalNonRecipeAuthority`. They should not need to know how seed routing or unresolved review status is represented internally.

The fifth milestone is manifest and summary alignment. Update [`cookimport/runs/manifest.py`](/home/mcnal/projects/recipeimport/cookimport/runs/manifest.py), any run-summary generation in [`cookimport/cli.py`](/home/mcnal/projects/recipeimport/cookimport/cli.py), and any stage-observability or benchmark metadata that indexes non-recipe artifacts. The new manifest must list the new canonical filenames. If any human-readable summary mentions the old files by name, change it.

The sixth milestone is docs and fixture cleanup. Update the staging and architecture docs so they teach the new artifact model directly. Any fixture-based tests or sample artifacts that still encode the old names must be migrated. Narrow read-side compatibility helpers are acceptable only while fixtures are being regenerated; they should not become permanent write-side aliases.

The seventh milestone is cleanup and deletion. Once all active readers and docs are migrated, remove the old write paths for `08_nonrecipe_spans.json` and `09_knowledge_outputs.json`. Do not leave stale write-side compatibility files behind “just in case.” If a debug rollup is kept, name it as debug and document that it is not a primary contract artifact.

## Idempotence and Recovery

This migration should be implemented additively first and destructively second. Start by defining the new in-memory types and writers while the old tests still guide the behavior. Then migrate readers. Only after every active reader has moved should the old write path be removed.

If the migration becomes confused midway, the safe recovery path is:

1. Keep the new in-memory split.
2. Restore temporary read-side adapters only where a consumer still expects the old shape.
3. Do not restore the old mixed write path as a long-term fallback.
4. Re-run the focused tests listed above before touching broader suites again.

The file-writing changes are idempotent. Re-running stage after the migration should simply overwrite the new artifact set in a new timestamped output directory. No destructive data migration of existing output folders is required.
