Problem statement

`pytest -W error` still fails in EPUB ingestion because BeautifulSoup emits `XMLParsedAsHTMLWarning` while nav detection parses XHTML spine content as HTML. The warning aborts conversion before EPUB health warnings are computed.

Acceptance criteria

- `tests/ingestion/test_epub_extraction_quickwins.py::test_epub_health_warnings_are_written_to_report_and_raw_artifacts` passes under `-W error`.
- EPUB conversion no longer aborts on this warning in the nav-detection helper path.

Verification steps

- `source .venv/bin/activate && COOKIMPORT_TEST_SUITE=1 COOKIMPORT_PYTEST_VERBOSE_OUTPUT=1 pytest -o addopts='' -W error --maxfail=1 --tb=short tests/ingestion/test_epub_extraction_quickwins.py::test_epub_health_warnings_are_written_to_report_and_raw_artifacts`
- `source .venv/bin/activate && COOKIMPORT_TEST_SUITE=1 pytest -W error --maxfail=1 --tb=short`

Evidence

- Before the fix, the targeted quickwins test failed because `result.report.warnings` was empty after EPUB conversion aborted on `XMLParsedAsHTMLWarning`.
- After the fix, the same test should reach the health-warning assertions and pass.

Constraints/gotchas + brief decision

- Keep the fix local to the actual BeautifulSoup constructors instead of relying on broad module-level warning filters, because order-dependent pytest warning handling has already bypassed those.
- Do not change EPUB extraction semantics beyond suppressing this known non-fatal parser warning.
- The remaining warning-gate blockers after the EPUB fix were stale tests plus one real fake-runtime gap: the fake CodexFarm subprocess had to mirror the current leased-packet knowledge loop instead of the removed phase-ledger helper flow.
