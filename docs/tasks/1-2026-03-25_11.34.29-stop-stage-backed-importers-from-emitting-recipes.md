## Structured Restatement

The intended architecture is blunt. Any importer that feeds the shared stage session should stop emitting recipe candidates as part of its live contract. It should emit source facts only: canonical source blocks, optional non-authoritative support, raw artifacts, and a report. The recipe-boundary stage should become the first place where recipes exist at runtime. Once that is true, the repo can delete the importer-vs-authority mismatch branch, delete `authority_mismatch.json`, and stop teaching readers that stage-backed flows still carry a shadow importer recipe truth to compare against.

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

This plan is intentionally the hard-cut continuation of [docs/tasks/2026-03-23_10.44.32-redesign-importers-as-source-normalizers.md](/home/mcnal/projects/recipeimport/docs/tasks/2026-03-23_10.44.32-redesign-importers-as-source-normalizers.md). That earlier plan pushed importers toward source normalization and moved recipe authority downstream, but the live runtime still preserves one transitional seam: `execute_stage_import_session_from_result(...)` can compare importer recipe count to authoritative recipe count and write `authority_mismatch.json`. This plan removes that remaining seam entirely.

## Purpose / Big Picture

After this change, a human should be able to say one simple sentence about stage-backed importers: they read source data and expose source facts; they do not emit recipes. That sentence should be true for `cookimport stage`, for the shared stage session used by processed Label Studio flows, and for any other runtime path that enters the label-first five-stage pipeline.

The user-visible gain is architectural honesty. A zero-recipe stage run should no longer come with wording like "the importer reported recipe candidates." Instead, the only explanation surface should be label-first diagnostics such as `recipe_spans.json` and `span_decisions.json`. A reader debugging a bad run should ask "what source blocks did the importer expose?" and "why did `group_recipe_spans` reject or accept spans?" They should not have to reason about a stale importer recipe count that the architecture no longer wants to treat as meaningful.

The proof is concrete. After implementation, a stage run should still produce the normal recipe, non-recipe, and knowledge artifacts, but it should not write `group_recipe_spans/<workbook_slug>/authority_mismatch.json`. The code should no longer branch on `original_result.recipes` to decide whether authoritative grouping disagreed with the importer, and the docs should stop teaching importer candidates as part of the live recipe-boundary story.

This plan is not merely a doc cleanup. It is a runtime contract cleanup. If the importer recipe count still exists in stage-backed flows after implementation, the architecture has not actually burned the boats.

## Context and Orientation

The key term in this plan is "stage-backed importer." In this repository that means an importer whose `ConversionResult` is fed into the shared five-stage stage session: `extract`, `recipe-boundary`, `recipe-refine`, `nonrecipe-route`, and `knowledge-final`. Those flows include `cookimport stage` and any sibling runtime that reuses the same shared stage import session for processed output generation.

The current live seam looks like this. Importers still return `ConversionResult`, and some of them still populate `recipes`. Then `cookimport/staging/import_session_flows/output_stage.py` runs the recipe-boundary stage, gets an authoritative `conversion_result` back, and compares `original_result.recipes` to `result.recipes`. If the importer had recipes but label-first grouping yields zero recipes, the code appends a warning and writes `group_recipe_spans/<workbook_slug>/authority_mismatch.json` through `cookimport/staging/import_session_flows/authority.py`.

Meanwhile, the parser-owned recipe-boundary path already has the data needed to explain itself without that comparison. `cookimport/parsing/recipe_span_grouping.py` groups authoritative block labels into span decisions. `cookimport/parsing/label_source_of_truth.py` records accepted spans as `RecipeSpan` and rejected ones as `RecipeSpanDecision` values such as `rejected_pseudo_recipe_span` or `rejected_missing_recipe_body`. `cookimport/staging/import_session_flows/authority.py` already writes `recipe_spans.json` and `span_decisions.json` from those structures.

The importers that matter live under `cookimport/plugins/`. The architecture docs already describe them as source normalizers, but the transition is incomplete because the stage-backed runtime can still observe importer recipe counts. The important files for this plan are:

- `cookimport/core/models.py`, where `ConversionResult` defines the shared importer output shape
- `cookimport/plugins/*.py`, where importers currently decide what fields to populate
- `cookimport/staging/import_session_flows/output_stage.py`, where the live mismatch branch exists
- `cookimport/staging/import_session_flows/authority.py`, where the mismatch artifact is written
- `docs/01-architecture/01-architecture_README.md`, which still teaches the transitional seam
- `docs/03-ingestion/03-ingestion_readme.md`, which describes importer responsibilities
- `docs/plain-english-explanation.md`, which should eventually stop talking about importer guesses entirely

The intended end state is simple: stage-backed importers expose source facts only, the recipe-boundary stage creates the first recipes, and zero-recipe diagnostics are explained only through label-first outputs.

## Plan of Work

Start by making the runtime contract explicit before deleting anything. In `cookimport/core/models.py`, document and, if needed, tighten the stage-backed expectation that `ConversionResult.recipes` is not a live field for stage-backed importers. If the repo already has non-stage consumers that still need recipe-populated importer outputs, do not break them blindly; instead, isolate the stage-backed import path so it normalizes or clears `recipes` before the shared stage session sees the result. The checked-in result must make it impossible for `execute_stage_import_session_from_result(...)` and its callers to treat importer recipes as live data.

Next, remove the importer-comparison logic from the stage session. In `cookimport/staging/import_session_flows/output_stage.py`, delete the branch that checks `original_result.recipes` against the authoritative `result.recipes`, appends the warning, and stores `authority_mismatch_path`. In `cookimport/staging/import_session_flows/authority.py`, delete `_write_label_first_authority_mismatch_artifact(...)`. If any helper or return type still exposes `authority_mismatch_path`, remove that field or make it impossible for live code to populate it. The stage session should still write `recipe_spans.json` and `span_decisions.json`; those become the only first-class recipe-boundary explanation surfaces.

Then cut the importers over completely. Audit every live stage-backed importer under `cookimport/plugins/` and remove recipe-population from the stage-backed path. The expected end state is that stage-backed importers populate canonical source blocks, optional source support, raw artifacts, and the report, but they do not populate `recipes`, importer-owned `non_recipe_blocks`, or similar semantic leftovers for the shared stage runtime. If a plugin has mixed responsibilities because it also supports a non-stage path, split the stage-backed behavior from the legacy/non-stage behavior explicitly rather than preserving an ambiguous "sometimes recipes, sometimes not" contract.

Once the runtime contract is clean, update the focused tests. `tests/staging/test_import_session.py` should stop asserting that `authority_mismatch.json` exists and should instead prove that zero-recipe runs keep `recipe_spans.json` empty while `span_decisions.json` records the rejection. Add or update importer-facing tests so at least one stage-backed importer family proves the new source-only contract directly. Any Label Studio or helper tests that still refer to `authority_mismatch.json` or importer candidates should be updated to the new explanation surface.

Finally, update the docs so they stop teaching the transitional model. In `docs/01-architecture/01-architecture_README.md`, remove wording about importer candidates existing, candidate-first ownership, or authoritative regrouping disagreeing with importer candidates. In `docs/03-ingestion/03-ingestion_readme.md`, say plainly that stage-backed importers are source readers, not recipe emitters. In `docs/plain-english-explanation.md`, rewrite the recipe-boundary section so it describes accepted spans, rejected pseudo-recipes, and `span_decisions.json` without mentioning importer guesses at all.

The implementation should be a hard cut, not a long coexistence period. A small internal adapter is acceptable while code is in flight, but the final checked-in result should no longer preserve importer-recipe comparisons in runtime behavior, artifacts, tests, or docs.

## Validation and Acceptance

The change is correct only if all of the following are true.

Runtime acceptance means no live stage-backed path can compare importer recipe count to authoritative recipe count. Searching for `authority_mismatch`, `importer_recipe_count`, `authoritative_recipe_count`, or the `original_result.recipes` mismatch branch should return no live stage-backed implementation references.

Importer-contract acceptance means stage-backed importers no longer emit recipes for the shared stage runtime. Focused tests or assertions should prove that a stage-backed importer returns canonical source facts without pre-populated recipe candidates on the live stage path.

Recipe-boundary acceptance means zero-recipe explanations come entirely from label-first artifacts. A focused test should show that `recipe_spans.json` is empty, `span_decisions.json` contains the rejected span decisions, and no mismatch artifact is written.

Documentation acceptance means `docs/01-architecture/01-architecture_README.md`, `docs/03-ingestion/03-ingestion_readme.md`, and `docs/plain-english-explanation.md` no longer describe stage-backed recipe-boundary behavior in importer-candidate terms.

Behavioral acceptance means a cheap deterministic stage run still succeeds and writes the normal stage outputs after the cut. The architecture cleanup must not break the actual recipe and non-recipe pipeline.

## Idempotence and Recovery

The safest implementation order is additive first, destructive second. If importer code still needs a temporary adapter while the stage session is being cut over, keep that adapter private and stage-only. Do not write new artifacts or public helper names that preserve the old importer-recipe wording.

If a checkpoint removes the mismatch artifact before zero-recipe diagnostics are clearly visible in `span_decisions.json`, stop and fix the diagnostics surface before continuing. If a checkpoint removes importer recipes from some plugins but not others, document exactly which importers still need the cut and do not update the docs to claim success early.

If a legacy non-stage surface still genuinely depends on importer-populated recipes, split that path explicitly instead of silently keeping the stage-backed contract ambiguous. The whole point of this plan is to make the live shared stage path unambiguous.
