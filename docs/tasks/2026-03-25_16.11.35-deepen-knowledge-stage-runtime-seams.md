This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

The knowledge stage already speaks the packet-based contract, but several internal seams are still transitional. After this change, the configured knowledge prompt target count will shape real planned shards instead of only emitting warnings, each knowledge shard will own one or more packet tasks instead of pretending `shard == task`, and the public package surface will stop hiding most of its behavior behind `cookimport.llm.knowledge_stage.__init__`. A human should be able to inspect a run and see meaningful multi-packet shard manifests, packet-native task manifests, clean task-to-shard aggregation, and a smaller explicit public seam.

## Surprises & Discoveries

- Observation: the current task/runtime surface was already more packet-aware than the shard planner. `knowledge_workspace_tools.py` already has `_packet_rows(...)` and batch/task rendering helpers that can naturally support more than one packet shape even though the current shard planner always emits one packet per shard.
  Evidence: `cookimport/llm/knowledge_workspace_tools.py` uses `_packet_rows(...)` for task summaries and validation help.

- Observation: the knowledge reporting layer already expects partial promotion metadata such as `owned_packet_count`, `promoted_packet_count`, and `missing_packet_ids`, which means multi-packet shard promotion can fit the existing rollup shape instead of requiring a brand new report family.
  Evidence: `cookimport/llm/knowledge_stage/reporting.py` computes unreviewed packet counts from those metadata fields.

- Observation: the package-surface problem was deeper than `__init__.py`. `planning.py`, `promotion.py`, `recovery.py`, and `runtime.py` were also reading the package object out of `sys.modules` and treating it as a shared internal namespace, which meant the package seam was still doing hidden dependency injection even after the public orchestrator alias was removed.
  Evidence: each of those modules began with `runtime = sys.modules["cookimport.llm.knowledge_stage"]` and a `globals().update(...)` over the package namespace.

## Outcomes & Retrospective

The transitional seams are now materially smaller.

`knowledge_prompt_target_count` is now honest: packet planning still writes one immutable packet payload per file under `knowledge/in/*.json`, but the planner then groups those packets into real shard entries whose `owned_ids` and input payloads can span several packet ids. That means the knob changes shard count instead of only emitting warnings.

The task layer is now real too. Multi-packet shards are decomposed into one task per owned packet, worker result files stay packet-native, shard aggregation preserves packet order, and shard proposals can now promote several packet outputs from a single proposal file while still recording partial promotion metadata such as `missing_packet_ids`.

The public seam is now explicit. `cookimport/llm/codex_farm_knowledge_orchestrator.py` re-exports the three public knowledge-stage symbols directly from their defining modules, `knowledge_stage.__init__` now exposes only that same small surface, and the stage submodules import shared state from `_shared.py` or sibling modules rather than from the package object.

The runtime seam is smaller too. `runtime.py` still owns the top-level stage flow and manifest/report coordination, but the broad workspace-worker execution path now lives in `workspace_run.py`. That module owns worker preflight, workspace-session lifecycle, relaunch bookkeeping, and taskized post-session packet/shard finalization. Retry and repair attempt primitives remain in `recovery.py`, which was already the natural follow-up owner.

## Context and Orientation

The knowledge stage lives under `cookimport/llm/knowledge_stage/`. In plain language, a packet is one ordered bounded slice of outside-recipe blocks that the model reviews semantically. A shard is the larger planned unit handed to a worker assignment. Today those two meanings are mostly collapsed together: `cookimport/llm/codex_farm_knowledge_jobs.py` writes one packet per shard, `cookimport/llm/knowledge_stage/planning.py` turns each shard back into one task, and `cookimport/llm/codex_farm_knowledge_ingest.py` validates exactly one packet result against each shard. The runtime still writes task manifests and queue sidecars, but the extra layer is mostly nominal.

The relevant public entrypoint is `cookimport/llm/codex_farm_knowledge_orchestrator.py`, which currently aliases the whole `cookimport.llm.knowledge_stage` package module. The package itself re-exports most of its runtime, recovery, planning, and reporting internals from `__init__.py`. That keeps old imports working but makes ownership blurry and makes the package feel broader than it really is.

The key files for this work are:

- `cookimport/llm/codex_farm_knowledge_jobs.py` for packet planning and shard manifest creation.
- `cookimport/llm/knowledge_stage/planning.py` for shard-to-task planning.
- `cookimport/llm/knowledge_stage/runtime.py` for worker assignment orchestration, aggregation, and proposal writing.
- `cookimport/llm/codex_farm_knowledge_ingest.py` for output normalization, validation, and proposal promotion.
- `cookimport/llm/knowledge_workspace_tools.py` for task sidecars and worker-helper behavior.
- `cookimport/llm/knowledge_stage/recovery.py` for preflight checks, worker prompt text, retry splitting, and follow-up bookkeeping.
- `cookimport/llm/knowledge_stage/__init__.py` and `cookimport/llm/codex_farm_knowledge_orchestrator.py` for the public seam.

## Plan of Work

First, change the planner in `cookimport/llm/codex_farm_knowledge_jobs.py` so it still computes deterministic ordered packets, but then bundles those packets into real planned shards according to `knowledge_prompt_target_count`. The planner should write shard payloads that explicitly own one or more packet payloads in order, preserve packet ids, preserve packet-level context and guardrails, and record shard metadata such as owned packet ids, packet count, owned block indices, and shard order. The stage should still fail closed when no packets exist, but the prompt-target knob must now affect the number of shard entries rather than only warning when it is lower than the packet floor.

Second, make the task layer meaningful. `cookimport/llm/knowledge_stage/planning.py` should derive one task per owned packet inside a shard and decorate that task with the existing workspace metadata. The shard payload is now the planning container; the task payload is the semantic work unit. The task aggregation function must stop returning the first accepted packet payload. Instead it must aggregate the validated task outputs for the parent shard into a shard-level payload that preserves every promoted packet result and records missing packet ids when the shard only partially validates. `cookimport/llm/codex_farm_knowledge_ingest.py` must then validate and promote that shard-level wrapper back into packet-native promoted results.

Third, adapt runtime and recovery to the new contract. `cookimport/llm/knowledge_stage/runtime.py` and `cookimport/llm/knowledge_stage/recovery.py` should preflight shard bundles, render worker prompts that describe packet tasks inside a shard, and split retries at the task-packet level without pretending the shard payload itself is one packet. Follow-up logic should continue to validate packet-sized outputs, but the final proposal file for a shard must be able to hold more than one promoted packet result.

Fourth, shrink the public seam. `cookimport/llm/knowledge_stage/__init__.py` should expose only the small public surface the rest of the repo needs, and `cookimport/llm/codex_farm_knowledge_orchestrator.py` should stop aliasing the entire package module through `sys.modules`. Replace that with explicit imports and re-exports so the public boundary is obvious.

Finally, update the short docs that teach the live contract. `cookimport/llm/knowledge_stage/README.md` should explain the new shard-versus-task meaning in one small paragraph, and `docs/10-llm/10-llm_README.md` should stop implying that packet count and shard count are effectively the same thing on the live path.

## Validation and Acceptance

Acceptance is reached when all of the following are true:

1. A knowledge job build with several packets and a lower `knowledge_prompt_target_count` produces fewer shard entries than packets while still preserving every packet id and block range in shard metadata.
2. The knowledge task manifest shows one task per packet inside those shards rather than one task per shard.
3. A runtime proposal for a multi-packet shard can promote multiple packet outputs from one shard proposal file without losing packet ids or silently discarding missing packets.
4. The public import seam is explicit: importing from `cookimport.llm.codex_farm_knowledge_orchestrator` no longer aliases the entire package module object.
5. The focused pytest commands above pass in `.venv`.

## Idempotence and Recovery

The planner and runtime artifact writers already overwrite their own generated manifests and proposal files, so re-running the tests and the stage-owned helper writers is safe. If a partial refactor leaves a mixed contract behind, the safe recovery path is to finish the contract migration in one wave rather than preserving a long-lived adapter. This plan intentionally prefers fail-closed runtime validation over broad back-compat shims.
