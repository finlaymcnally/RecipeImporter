# Problem

Some codex-farm builds still emit plain stderr progress lines like `run=...` and `queued=... running=... done=...`. Cookimport was treating those as real stderr, so interactive benchmark runs could print `codex-farm stderr (...)` blocks while the live status panel was active.

# Acceptance Criteria

- Legacy plain stderr progress lines are consumed as progress/control output.
- Those lines update the normal progress callback path instead of surfacing as stderr noise.
- Structured `__codex_farm_progress__` handling still works.

# Verification

- Fail before: `source .venv/bin/activate && pytest tests/llm/test_codex_farm_orchestrator_runner_transport.py -k legacy_stderr_progress_lines -q`
- Pass after: `source .venv/bin/activate && pytest tests/llm/test_codex_farm_orchestrator_runner_transport.py -k 'legacy_stderr_progress_lines or progress_events' -q`
- Docs check: `npm run docs:list`

# Evidence

- The new legacy-progress transport test failed on the baseline and passed after the runner change.
- Focused progress transport tests passed after the fix.

# Constraints / Decision

- Keep the fix narrow to the runner transport layer instead of rewriting interactive benchmark status handling.
- Continue preferring structured progress events; plain stderr parsing remains compatibility-only.

# Rollback

- Revert the runner parser/test/docs change set if the compatibility parser proves too permissive.
