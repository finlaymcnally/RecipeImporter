# Relax Workspace Temp-Root Watchdog and Refine Worker Shell Guidance

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The knowledge benchmark saved at `data/golden/benchmark-vs-golden/2026-03-22_01.34.12/single-book-benchmark/saltfatacidheatcutdown/codexfarm/` shows that the recent semantic-packet fix worked, but the main knowledge workers still die for watchdog boundary violations. The failures are no longer primarily schema drift. They are mostly `watchdog_command_execution_forbidden` kills triggered by local temp-path and shell-exploration behavior that is wasteful but not dangerous in this solo-project runtime.

After this change, long-lived workspace workers must be allowed to use local temp roots such as `/tmp` and `/var/tmp` for short-lived helper files without being killed, while still forbidding repo/network/package-manager tools and parent-directory escapes. The worker instructions must also be sharper: they should open the repo-named files directly, avoid broad inventory dumps or ad hoc shell schedulers, and prefer short targeted helpers over sprawling `jq` exploration. The observable result is that the direct-exec watchdog no longer treats temp-root use as a boundary escape, and the worker prompts/manifests stop nudging agents toward the exact exploratory shell behavior that burned the last benchmark.

## Context and Orientation

The shared workspace-worker enforcement lives in `cookimport/llm/codex_exec_runner.py`. It has two separate concepts:

- `detect_workspace_worker_boundary_violation(...)`: the hard kill/no-kill seam. This is the only place that should decide whether a workspace worker crossed a forbidden boundary.
- `classify_workspace_worker_command(...)`: telemetry-only classification. This can say a command was exploratory or unclassified without necessarily making it kill-worthy.

The knowledge main-attempt worker prompt lives in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. That prompt currently tells the worker to use `scratch/` and open repo-named files directly, but it still leaves enough ambiguity that workers dump `assigned_tasks.json`, build `jq` scripts, and in some cases write helper files under `/tmp`, which the shared watchdog still treats as forbidden.

The focused tests already exist:

- `tests/llm/test_codex_exec_runner.py` covers command classification and boundary detection.
- `tests/llm/test_codex_exec_runner_workspace.py` covers the worker-manifest contract written into sterile direct-exec workspaces.
- `tests/llm/test_codex_farm_knowledge_orchestrator.py` covers the knowledge worker prompt and taskized worker-manifest surface.

The relevant docs are:

- `docs/10-llm/10-llm_README.md`
- `docs/10-llm/nonrecipe_knowledge_review.md`
- `cookimport/llm/README.md`

## Plan of Work

Edit `cookimport/llm/codex_exec_runner.py` first. Introduce an explicit allowlist for local temp roots such as `/tmp`, `/var/tmp`, and `/private/tmp`. Use that allowlist in both path-argument classification and raw shell-text boundary scanning so the kill seam stops treating those temp roots as generic absolute-path escapes. Do not weaken the existing bans on parent traversal, repo/network/package-manager tools, `$HOME`, or other absolute paths.

In the same module, update the shared direct-exec worker text and worker-manifest notes/examples. Remove examples that encourage broad inventory probing such as `jq '.[0] | keys' assigned_tasks.json`. Replace them with examples that open the exact prompt-named file, use short targeted helpers, and keep final outputs on the approved `out/` path while allowing temp helper files under `scratch/` or local temp roots.

Then edit `_build_knowledge_workspace_worker_prompt(...)` in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. Keep the semantic-packet contract changes from the previous fix, but add sharper guidance that the worker should not dump the whole `assigned_tasks.json` back to itself, should not invent its own batch scheduler, and should prefer a short direct `python3` helper or one-off query against the current task row’s named files over sprawling exploratory shell scripts.

After the code changes, update the short docs in `docs/10-llm/10-llm_README.md`, `docs/10-llm/nonrecipe_knowledge_review.md`, and `cookimport/llm/README.md` so the documented boundary matches the real one: temp roots are allowed for short-lived local helper files, but workers should still prefer direct file opens and minimal targeted helpers.

## Idempotence and Recovery

The temp-root allowlist is safe to re-run and should be deterministic. The worker-instruction changes are also idempotent: reapplying them should only rewrite the same static prompt/manfiest text.

If a patch accidentally starts allowing non-temp absolute paths, stop immediately and restore the narrower boundary before rerunning tests. If prompt edits drift away from the worker-manifest docs, update the tests and docs in the same patch so the contract remains obvious and self-consistent.
