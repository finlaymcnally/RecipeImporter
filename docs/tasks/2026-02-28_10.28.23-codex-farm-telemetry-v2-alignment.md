## Problem statement

CodexFarm upgraded telemetry to schema-v2 caller automation (`telemetry_report` with `insights` and `tuning_playbook`). Recipeimport persisted compact CSV telemetry slices but did not expose embedded `process --json.telemetry_report` as a first-class `process_run` field.

## Acceptance criteria

- `CodexFarmPipelineRunResult` includes top-level `telemetry_report` when present in `process --json` payload.
- Recipe/pass4/pass5 manifest/report `process_run(s)` payloads carry `telemetry_report` via the shared runner serialization path.
- Existing CSV-slice telemetry (`telemetry`) remains available and non-fatal.
- Targeted tests cover the new field and continue passing.

## Evidence

- Added assertions that `run_pipeline(...)` surfaces `telemetry_report.schema_version == 2` when provided by process payload.
- Added assertions that pass4/pass5 `process_run` payloads include a `telemetry_report` key.
- Updated LLM docs to describe primary embedded report contract plus CSV-slice fallback.

## Constraints/gotchas and decision

- Keep compatibility with older CodexFarm builds or `--no-telemetry-report` runs: `telemetry_report` may be absent, and this must not fail conversion paths.
- Decision: preserve existing CSV-slice telemetry for row-level context while elevating embedded telemetry report to first-class metadata.
