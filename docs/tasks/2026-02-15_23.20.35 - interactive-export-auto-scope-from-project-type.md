# Problem statement

Interactive Label Studio export showed detected project type in the picker but still asked users to choose `Export scope` immediately after selection.

# Acceptance criteria

- Interactive export uses detected scope automatically when a selected project has a known type (`pipeline`, `canonical-blocks`, `freeform-spans`).
- Interactive export only prompts for `Export scope` when detected type is unknown or project name is entered manually.
- Existing project picker labels and manual-entry fallback behavior remain intact.

# Verification steps

- `. .venv/bin/activate && pytest -q tests/test_labelstudio_benchmark_helpers.py -k "interactive_labelstudio_export or select_export_project"`

# Evidence

- Targeted tests pass.
- `test_interactive_labelstudio_export_routes_to_export_command` now asserts no `Export scope` prompt when scope is detected.
- `test_interactive_labelstudio_export_selects_project_before_scope` still verifies prompt order when detected scope is unavailable.

# Constraints/gotchas

- Scope auto-selection relies on existing detection signals (local manifest scope first, then Label Studio payload heuristics).
- Unknown scope projects still require manual scope choice.

# Decision

Return both selected project title and detected scope from export project selection, then gate the scope prompt behind unknown/manual detection.

# Rollback / compensation plan

Restore unconditional interactive `Export scope` prompt after project selection and remove scope-carrying selection path/tests.
