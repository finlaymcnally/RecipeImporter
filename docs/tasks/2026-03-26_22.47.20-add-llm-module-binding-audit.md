## Problem

The repo had already hit several missing-name crashes in split modules, and the knowledge-stage recovery crash showed the LLM package was not covered by the earlier CLI-focused audits.
Without a package-level audit, the next split LLM binding mistake can still wait until a long run reaches the right branch.

## Acceptance Criteria

- A routine offline test imports every `cookimport.llm.*` module except `__main__`.
- That same test fails on unresolved `LOAD_GLOBAL` / `LOAD_NAME` references in module-owned functions.
- The new audit passes in `.venv` on the fixed tree.

## Verification

- `. .venv/bin/activate && pytest tests/llm/test_llm_module_bindings.py -q`
- `. .venv/bin/activate && pytest tests/llm/test_knowledge_stage_bindings.py -q`

## Evidence

- A manual preflight sweep over 43 `cookimport.llm.*` modules was clean after the knowledge-stage recovery fix.
- The same sweep now runs in pytest instead of relying on ad hoc one-off scripts.

## Constraints / Decision

- Keep the audit static and offline; it should not spin up live CodexFarm work.
- Fail on import errors too, not just unresolved names, because split-package drift often shows up as broken imports first.
