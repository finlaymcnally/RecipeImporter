# Minimize Single-Pass Line-Role Payloads To Need-To-Know Data

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

Canonical line-role no longer pays for giant inline prompts, and it no longer repeats `prev_text` and `next_text` in every model-facing row. The remaining waste is that the single-pass line-role shard file still carries a lot of operator-only metadata that Codex does not need to label lines. After this change, the runtime should keep the rich local/debug row data on our side while the model-facing `in/*.json` payload contains only the minimum fields Codex needs to do the job.

The user-visible result is that line-role preview and live execution still behave the same, but prompt budget drops because Codex stops reading fields such as `block_index`, `block_id`, `recipe_id`, `within_recipe_span`, `rule_tags`, and `escalation_reasons` on every row. A contributor should be able to run zero-token prompt preview on `thefoodlabcutdown`, inspect both the compact model payload and the richer local debug copy, and see a meaningful budget reduction relative to the current single-pass baseline.

## Outcomes & Retrospective

This plan is implemented. Live line-role workers now write compact tuple payloads to `workers/*/in/*.json`, write the richer object rows to `workers/*/debug/*.json`, and expose the same split in preview via `line-role-pipeline/in/*.json` plus `line-role-pipeline/debug_in/*.json`. The focused tests passed, and the saved Food Lab preview showed the expected budget win: estimated line-role input tokens dropped from `337,923` to `157,935`, with total tokens dropping from `361,968` to `181,980` across the same `5` calls. The main lesson is that the file-backed seam still had a large amount of recoverable waste even after neighbor-text trimming, and the recovery path stayed safe because the richer local copy never disappeared.

## Context and Orientation

Canonical line-role is the Codex-backed stage that reviews ordered atomic lines from the canonicalized cookbook text and emits labels such as `RECIPE_TITLE`, `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `KNOWLEDGE`, or `OTHER`. An atomic line is one small text row. A shard is one contiguous batch of atomic lines that one model call owns.

The current live line-role surface is a single pass with `phase_key: "line_role"`. The relevant files are:

- `cookimport/parsing/canonical_line_roles.py`: plans line-role shards, builds per-row payloads, writes worker runtime artifacts, launches direct Codex execution, validates returned rows, and writes prompt/debug artifacts.
- `cookimport/llm/canonical_line_role_prompt.py`: renders the file-backed wrapper prompt that tells Codex how to read the task file and what JSON to return.
- `cookimport/llm/prompt_preview.py`: reconstructs the same line-role input payload and wrapper prompt in zero-token preview mode.
- `cookimport/llm/prompt_artifacts.py`: exports reviewer-facing prompt/debug artifacts.
- `cookimport/llm/prompt_budget.py`: estimates prompt cost from preview artifacts and summarizes actual-cost telemetry after live runs.
- `tests/parsing/test_canonical_line_roles.py`, `tests/llm/test_prompt_preview.py`, and `tests/llm/test_label_phase_workers.py`: focused tests around runtime payloads, preview output, and worker artifacts.

Before implementation, the same row object served two jobs at once: it was both the local debug artifact and the model-facing transport. The pre-change model-facing preview payload under `data/output/2026-03-18_16.00.00-foodlab-line-role-no-neighbor-context-preview/line-role-pipeline/in/line_role_input_0001.json` looked like:

    {
      "atomic_index": 0,
      "block_index": 0,
      "block_id": "block:0",
      "recipe_id": null,
      "within_recipe_span": null,
      "deterministic_label": "TIME_LINE",
      "rule_tags": ["time_metadata"],
      "escalation_reasons": [],
      "current_line": "Eggs boiled for 30-second intervals from 0 to 12 minutes."
    }

The model likely needs only a fraction of that. The current wrapper prompt already supplies the rest of the task contract in words: the file is an ordered contiguous slice, `current_line` is the line to label, and neighboring rows in `rows[*]` provide local context. That means a much smaller model-facing row is now viable.

After implementation, “rich local artifact” means the full object rows written to `workers/*/debug/*.json` and preview `debug_in/*.json`. “Compact model payload” means the active `{"v":1,"shard_id":...,"rows":[[atomic_index,label_code,current_line], ...]}` transport written to `in/*.json`.

## Plan of Work

Start by isolating the payload-building seam in `cookimport/parsing/canonical_line_roles.py`. The current `_line_role_plan_row(...)` helper should stop being the one-size-fits-all row serializer. Replace it with two explicit builders:

- one builder for the rich local/debug row
- one builder for the compact model-facing row

The rich local artifact should preserve the current fields. Write it under a new stable sibling path such as:

    line-role-pipeline/runtime/line_role/workers/<worker-id>/debug/<shard>.json

and in preview:

    line-role-pipeline/debug_in/line_role_input_<n>.json

The compact model-facing payload should remain the active file written to:

    line-role-pipeline/runtime/line_role/workers/<worker-id>/in/<shard>.json

and in preview:

    line-role-pipeline/in/line_role_input_<n>.json

The compact schema should be prescriptive and stable. Use a single version marker plus a positional row array so repeated key names disappear from every row. The model-facing row should be:

    [atomic_index, deterministic_label_code, current_line]

with legends for `deterministic_label_code` rendered once in the wrapper prompt. Do not include `block_index`, `block_id`, `recipe_id`, `within_recipe_span`, `rule_tags`, or `escalation_reasons` in the compact payload.

The runtime must continue to validate exact `atomic_index` coverage exactly as it does today. This plan must not alter fallback behavior, shard planning, cache behavior, or final label promotion. Only the model-facing payload shape is changing.

After the runtime builders are split, update `cookimport/llm/canonical_line_role_prompt.py` so the wrapper prompt documents the new compact tuple schema and its legends. The prompt should continue to say that the task file is an ordered contiguous slice and that neighboring rows supply local context when needed. It must not claim that removed metadata fields exist in the task file.

Then update `cookimport/llm/prompt_preview.py`, `cookimport/llm/prompt_artifacts.py`, and `cookimport/llm/prompt_budget.py`. Preview should write both the compact model-facing payload and the rich debug copy. Prompt rows should continue to treat the compact `request_input_file` and `request_input_payload` as the estimated model input. Add an explicit debug-path field such as `debug_input_file` for the rich local copy so humans can still inspect it.

Finally, update the focused tests. The runtime tests must assert that `workers/*/in/*.json` now contains positional tuples or otherwise compact rows with only the required fields, while `workers/*/debug/*.json` keeps the current rich objects. The preview tests must assert the same split for `line-role-pipeline/in/*.json` and `line-role-pipeline/debug_in/*.json`. The budget tests must continue to estimate only the compact payload, not the debug copy.

## Validation and Acceptance

Validation must prove both correctness and savings.

First, run the focused tests and expect them to pass. The updated assertions must prove all of the following:

1. The compact model-facing payload omits `block_index`, `block_id`, `recipe_id`, `within_recipe_span`, `rule_tags`, and `escalation_reasons`.
2. The rich local artifact still preserves those fields.
3. The wrapper prompt references the compact `request_input_file` and correctly describes the compact row schema and label-code legends.
4. Proposal validation still enforces exact owned `atomic_index` coverage.

Second, run predictive prompt preview on `thefoodlabcutdown` and inspect `prompt_preview_budget_summary.json`. Acceptance is:

1. `line-role-pipeline/in/*.json` uses the compact row schema and is the file referenced by `request_input_file`.
2. `line-role-pipeline/debug_in/*.json` exists and preserves the richer local object rows.
3. The line-role input-token estimate drops materially relative to the current single-pass baseline of about `337,923` input tokens and `361,968` total tokens across `5` calls.
4. The reduction is explained by metadata removal only; the transport should still carry the same row order and the same `current_line` text as before.

For this plan, “materially” should mean at least a six-figure input-token reduction on Food Lab. The current measurement suggests about `144,840` task-body input tokens remain available beyond a minimal `atomic_index` + coded `deterministic_label` + `current_line` seam. If the measured reduction is much smaller than that, inspect which operator-only fields are still reaching the compact payload or whether repeated key names / pretty-printed JSON are still present in the model-facing file.

## Idempotence and Recovery

This change is safe to iterate on because it is additive around artifact families. The compact `in/*.json` payloads and the new `debug/*.json` or `debug_in/*.json` files can be regenerated repeatedly under a new timestamped preview directory. If the compact schema drops information that the runtime still needs for validation or promotion, the rich local artifact is the comparison surface that lets a contributor restore the missing field deliberately rather than blindly reverting the whole change.

Do not remove the rich artifact path until the compact transport has passed the focused tests and the Food Lab preview validation. The debugging surface is the recovery path.
