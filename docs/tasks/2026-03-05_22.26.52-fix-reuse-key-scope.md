# Narrow prediction and line-role reuse keys without changing prediction correctness

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md` from the repository root.

## Purpose / Big Picture

After this change, `recipeimport` will stop rerunning expensive prediction work just because runtime-only settings changed. The user-visible result is that two benchmark variants that produce the same prediction artifacts will now share prediction outputs and line-role cache entries, while genuinely output-changing settings will still force fresh work.

The specific win is that changing knobs such as worker counts, split parallelism, warm-up behavior, or other execution-shape settings will no longer burn extra line-role Codex or all-method prediction runs when the actual predicted content would be identical. The proof is behavioral: targeted tests will show cache hits across runtime-only setting changes, and all-method benchmark reports will show reuse instead of extra executed predictions for those variants.

## Scope Guardrails (Intent Lock)

This plan fixes discrete problem 4 only. It does not change default AI-on behavior, baseline labeling, or token telemetry. It also does not redefine `RunSettings.stable_hash()` globally.

The guardrails are:

- Keep `RunSettings.stable_hash()` semantics unchanged. It is already used across reports, tests, and analytics as a general run-settings fingerprint.
- Add stage-specific identity helpers for prediction reuse and line-role cache reuse instead of mutating repo-wide hash behavior.
- Be conservative: only exclude a setting from a reuse key if the plan also adds a test proving that prediction artifacts remain identical when that setting changes.
- Allow old caches to age out naturally. Do not add destructive cleanup or migration requirements.

## Surprises & Discoveries

- Observation: the all-method prediction reuse key is broader than the prediction call surface actually needs.
  Evidence: `cookimport/cli.py::_all_method_prediction_reuse_key_payload` hashes `run_settings.to_run_config_dict()` minus only `benchmark_sequence_matcher`.

- Observation: the benchmark adapter already exposes a near-complete inventory of prediction-shape settings.
  Evidence: `cookimport/config/run_settings_adapters.py::build_benchmark_call_kwargs_from_run_settings` explicitly enumerates the settings that become `labelstudio_benchmark(...)` prediction kwargs.

- Observation: the line-role cache key is doubly encoding upstream state.
  Evidence: `cookimport/parsing/canonical_line_roles.py::_resolve_line_role_cache_path` includes both `settings.stable_hash()` and a `candidate_fingerprint` built from atomic text, neighbor text, candidate labels, and rule tags.

- Observation: line-role candidate fingerprints already capture the prompt-relevant content that parser/preprocess settings ultimately influence.
  Evidence: `_canonical_candidate_fingerprint(...)` hashes `text`, `prev_text`, `next_text`, `candidate_labels`, and `rule_tags` for every atomic candidate.

- Observation: changing `RunSettings.stable_hash()` would make this fix much larger and riskier than necessary.
  Evidence: test files such as `tests/labelstudio/test_labelstudio_benchmark_helpers_scheduler.py` and `tests/bench/test_quality_suite_runner.py` assert or surface `stable_hash()` widely outside the prediction reuse problem.

- Observation: using `RunSettings.model_copy(update=...)` with raw strings for enum fields is unsafe in these scheduler tests because the copied model keeps the unvalidated raw value.
  Evidence: the first draft of the forced-miss scheduler test failed with `"'str' object has no attribute 'value'"`; switching those variants to `RunSettings.from_dict(...)` fixed the test and kept the payload types canonical.

## Outcomes & Retrospective

Implementation complete for the planned scope.

The main design choice held up: adding stage-specific identity helpers solved the reuse waste without changing `RunSettings.stable_hash()` semantics anywhere else in the repo. All-method prediction reuse now keys on an explicit prediction identity, split-convert grouping no longer treats worker/job-splitting knobs as input changes, and canonical line-role cache keys now rely on candidate content plus a narrow line-role identity instead of the full run-settings hash.

The main remaining risk is future drift between prediction behavior and the identity allow-lists. That risk is now bounded by regression tests that prove runtime-only changes reuse safely while output-shape changes still force misses.

## Context and Orientation

This section explains the moving parts as if the reader knows nothing about the repository.

A “reuse key” in this repository is a hash used to decide whether prediction work can be skipped and copied from a previous run. A “prediction artifact” is the generated output from the prediction phase, such as `prediction-records.jsonl` plus the copied scratch and processed directories. A “line-role cache entry” is the JSON file in `.cache/canonical_line_role` that stores the canonical line-role predictions for a specific set of atomic line candidates.

The two current over-broad identities are:

`cookimport/cli.py::_all_method_prediction_reuse_key_payload`

This function currently hashes almost the full `RunSettings.to_run_config_dict()` payload. It intentionally excludes only `benchmark_sequence_matcher`, because that matcher affects evaluation only. The problem is that many other settings in `run_config` also do not affect prediction artifacts, so they should not block reuse.

`cookimport/parsing/canonical_line_roles.py::_resolve_line_role_cache_path`

This function currently includes `settings.stable_hash()` in the line-role cache identity. That is much broader than the prompt semantics for the line-role classifier. The same function also hashes a `candidate_fingerprint`, which already captures the atomic text and prompt-relevant context that upstream extractor changes ultimately produce.

The most useful file for understanding what really affects prediction output is:

`cookimport/config/run_settings_adapters.py`

In particular, `build_benchmark_call_kwargs_from_run_settings(...)` is the place where `RunSettings` values are projected into actual benchmark prediction kwargs. It already distinguishes prediction settings from output paths and top-level benchmark runtime arguments.

The main tests that must be extended are:

- `tests/labelstudio/test_labelstudio_benchmark_helpers_scheduler.py`
- `tests/parsing/test_canonical_line_roles.py`
- `tests/cli/test_run_settings_adapters.py` if a new shared projection helper lives near the adapters

The short discovery note for this plan is in:

- `docs/understandings/2026-03-05_22.26.52-reuse-key-scope-findings.md`

## Plan of Work

### Milestone 1: Create explicit stage-specific identity helpers

The first milestone creates a single source of truth for “what can change prediction artifacts” and a second source of truth for “what can change line-role Codex prompt semantics beyond the candidate content.” The result of this milestone is that the repository no longer uses the whole run settings hash as a proxy for either stage.

Create a new helper module at `cookimport/config/prediction_identity.py`. Define two public helpers there.

The first helper must be named `build_all_method_prediction_identity_payload(settings: RunSettings) -> dict[str, Any]`.

This helper must return only output-affecting settings for the prediction phase. It must include the settings that change extracted content, deterministic normalization, recipe construction, line-role application, and recipe LLM behavior. It must not include output paths, evaluation knobs, scheduler knobs, or runtime-only execution knobs.

### Milestone 2: Replace the over-broad all-method prediction reuse key

Once the shared helper exists, wire it into `cookimport/cli.py`.

Update `_all_method_prediction_reuse_key_payload(...)` so it uses the new `build_all_method_prediction_identity_payload(...)` helper instead of hashing almost the full `run_config`. Keep `source_file` and the schema version in the payload.

Bump `ALL_METHOD_PREDICTION_REUSE_KEY_SCHEMA_VERSION` from `all_method_prediction_reuse.v1` to `all_method_prediction_reuse.v2`.

Keep the existing file layout and lock behavior for the prediction reuse cache. The point of this milestone is only to change the identity, not the cache transport.

Also update `_all_method_split_convert_input_key_payload(...)`. This helper is used for reporting safe reuse candidates versus blocked reuse candidates. If it remains broader than the real prediction identity, the summary will become misleading. Narrow it to a true “pre-prediction input identity” that excludes the same runtime-only knobs removed from the real reuse key. It should remain stricter than the line-role cache identity but no longer count worker-count changes as input changes.

Do not collapse the two keys into one. They answer different questions. The prediction reuse key decides whether full prediction artifacts may be reused. The split-convert input key is still useful for explaining which groups share the same upstream input shape.

### Milestone 3: Replace the over-broad line-role cache key

Update `cookimport/parsing/canonical_line_roles.py::_resolve_line_role_cache_path(...)` so it no longer hashes `settings.stable_hash()`.

Instead, call `build_line_role_cache_identity_payload(settings)` and include that smaller payload in the cache key under a descriptive field name such as `line_role_identity`.

Keep the following pieces in the line-role key exactly because they are still necessary:

- `source_hash`
- `candidate_fingerprint`
- `codex_timeout_seconds`
- `codex_batch_size`

Bump `_LINE_ROLE_CACHE_SCHEMA_VERSION` from `canonical_line_role_cache.v1` to `canonical_line_role_cache.v2`.

Do not attempt to migrate old cache files. The schema version bump will naturally create new cache filenames. Old cache entries can remain on disk until normal cleanup or manual pruning.

### Milestone 4: Add proof tests for safe hits and required misses

This milestone is mandatory because the new identities are correctness-sensitive.

In `tests/labelstudio/test_labelstudio_benchmark_helpers_scheduler.py`, add tests that prove all-method prediction reuse now works when only runtime-only settings differ.

## Validation and Acceptance

This plan is complete only when the following behavior is observable.

First, all-method prediction reuse must survive runtime-only changes. Two variants that differ only in worker counts, split worker counts, warm-model behavior, or command-path details must share the same prediction reuse key and must reuse prediction artifacts.

Second, all-method prediction reuse must still miss when the prediction output can actually change. Two variants that differ in `atomic_block_splitter`, `line_role_pipeline`, or a recipe-LLM output-shaping field must produce different prediction reuse keys and must execute separate prediction runs.

Third, line-role cache reuse must survive unrelated run-settings changes when the atomic candidates are identical. A second `label_atomic_lines(...)` invocation with the same candidates but different unrelated extractor or worker settings must hit the cache.

Fourth, line-role cache reuse must still miss when line-role prompt semantics or batching semantics change. Changing the line-role mode or the batching controls must lead to a different cache identity.

The acceptance standard is behavioral, not just structural. Passing tests are necessary but not sufficient; at least one small benchmark-style reuse scenario must also show the expected reuse counts in a report artifact.

## Idempotence and Recovery

This change is safe to apply incrementally because it is versioned by schema constants. If implementation stops halfway, the old cache files remain untouched and the new code will simply create a second generation of cache entries.

If a safe-hit test fails after removing a field from an identity, restore that field to the identity helper and rerun the tests. Do not “fix” a stale-cache bug by broadening unrelated identities elsewhere.

If a forced-miss test unexpectedly reuses artifacts, treat that as a correctness bug. Re-add the missing field to the relevant identity helper, bump the schema version again if necessary, and rerun the targeted tests before continuing.
