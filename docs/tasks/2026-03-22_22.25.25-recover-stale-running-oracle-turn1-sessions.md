# Task

## Problem

Benchmark Oracle auto-follow-up currently reattaches turn 1 only when Oracle marks the session `assistant-timeout` or when the saved controller PID is dead.
That misses a third real failure mode: the chat already has a visible assistant answer, but Oracle still leaves the session `running` with a live controller, so the worker stays at `waiting_for_turn_1` forever and turn 2 never launches.

## Acceptance criteria

- Benchmark Oracle auto-follow-up attempts one bounded turn-1 recovery for stale-running sessions even when the controller PID is still alive.
- The stale-running heuristic is based on persisted turn-1 launch age, so it does not fire immediately on fresh launches.
- Regression coverage proves that a stale-running visible-answer case recovers turn 1 and proceeds to turn 2.

## Verification

- `./.venv/bin/pytest tests/bench/test_oracle_followup.py -q`

## Evidence

- `data/golden/benchmark-vs-golden/2026-03-22_21.47.46/.../.oracle_upload_runs/2026-03-22_21.51.13-quality/oracle_auto_followup.json` and the matching `...21.51.17-token/oracle_auto_followup.json` remained at `waiting_for_turn_1`.
- The corresponding Oracle session metadata under `~/.local/share/oracle/sessions/you-are-the-{quality,token}-lane/meta.json` still said `status: running` with live `controllerPid`, even though the operator could already see turn-1 answers in ChatGPT.

## Constraints / decision

- Keep the recovery narrow: do one bounded `oracle session <id>` retry after a stale-running grace period rather than introducing a new transport.
- Preserve the existing faster recovery paths for explicit `assistant-timeout` and dead-controller sessions.
- Treat this as a benchmark follow-up robustness fix; do not redesign Oracle launch or follow-up packet logic here.
