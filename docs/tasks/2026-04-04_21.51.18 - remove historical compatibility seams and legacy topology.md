This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Operator Intent (Structured) - DO NOT DELETE

The operator wants the old architecture to stop existing as active repo behavior, not merely to be documented as obsolete. The desired result is that active benchmark, follow-up, staging, and web-AI handoff paths speak only the current semantic architecture: stage-owned truth, manifest-driven artifact discovery, semantic stage names, and the route-versus-finalize split for outside-recipe work. Historical compatibility code should be deleted when practical, and where a tiny archived-reader seam still must remain, it should be isolated, clearly labeled as historical input only, and prevented from teaching old mental models back into current outputs.

## Purpose / Big Picture

After this change, the repo should no longer need to warn external reviewers and web AIs so heavily about old topology because active code paths will stop carrying it around. A new benchmark bundle, follow-up packet, or architecture handoff should describe the current pipeline directly without also carrying fallback logic that reconstructs numbered stages, guesses artifact paths, or revives importer-first truth stories.

The visible proof is threefold. First, current benchmark and follow-up tooling should resolve artifacts from manifests and semantic stage indices, not from legacy path guesses. Second, new reviewer-facing outputs should expose only current semantic stage names and current authority boundaries. Third, `docs/AI_context.md` should be able to shrink its historical-noise warning from a large protective section into a short note about archived artifacts, because the active codebase no longer keeps reintroducing the old story.

## Outcomes & Retrospective

Implementation landed. Active benchmark and follow-up readers now resolve recipe and knowledge manifests from current canonical pointers: explicit run-manifest artifact keys, prediction-run manifest pointers, or `stage_observability.json`. The active path no longer scans raw `raw/llm/*` layouts to recover those manifests when the current pointer contract is missing.

The biggest practical lesson was that stale fixtures are part of the compatibility seam. Two benchmark tests were still constructing “current” runs without current pointers, which made the suite preserve old behavior by accident. Updating those fixtures was the correct fix. This was an outdated-test problem, not a regression in the desired runtime contract.

The docs are now simpler because the code no longer needs them to carry as much defensive framing. `docs/AI_context.md` now gives one direct current-architecture story with a short historical note instead of a larger “historical noise” block.

## Context and Orientation

This repository is a local Python CLI named `cookimport`. The current architecture is stage-owned. “Stage-owned truth” means importers normalize source material into canonical source blocks, then shared stage logic decides recipe ownership, recipe semantics, and outside-recipe classification. The current stage index is semantic, not numbered. The important current names are `extract`, `recipe-boundary`, `recipe-refine`, `nonrecipe-route`, and `nonrecipe-finalize`, plus the recipe-semantic sub-stages `recipe_build_intermediate`, `recipe_refine`, and `recipe_build_final` where applicable in recipe LLM reporting.

“Historical noise” in this repo means old mental models and old readers that survived after the refactor. The main examples are importer-first truth stories, numbered pass-slot topology, mixed nonrecipe classifier stories that collapse route and finalize together, guessed artifact-path discovery instead of manifest-based discovery, and older “LLM as advisory diff against deterministic baseline” language.

The most relevant current owner files are these:

- `docs/01-architecture/01-architecture_README.md`, which states the current architecture and already identifies the remaining read-side debt.
- `docs/07-bench/07-bench_README.md`, which defines the active benchmark and upload-bundle contract.
- `docs/AI_context.md`, which is the standalone architecture handoff for web AIs that cannot inspect the repo.
- `cookimport/bench/upload_bundle_v1_model.py`, `cookimport/bench/upload_bundle_v1_existing_output.py`, and `cookimport/bench/upload_bundle_v1_render.py`, which normalize existing outputs into reviewer-facing bundle topology.
- `cookimport/bench/followup_bundle.py`, which builds deterministic follow-up packets from an existing upload bundle.
- `cookimport/bench/external_ai_cutdown/artifact_paths.py` and `scripts/benchmark_cutdown_for_external_ai.py`, which still contain the heaviest compatibility logic for older benchmark and review artifacts.
- `cookimport/runs/stage_observability.py`, which is the semantic stage-index owner for current run-level stage naming and artifact pointers.

For this plan, “active path” means any code that reads or renders current runs, current benchmark bundles, current follow-up packets, or current architecture docs. “Archived-reader-only” means code that may read older artifacts locally but does not shape new outputs and is clearly marked as history-only input handling.

## Plan of Work

Start by making the repo’s remaining compatibility story explicit in code. Audit every still-live seam that does one of the following: guesses artifact paths instead of reading manifests, reconstructs semantic stage meaning from numbered-stage names or raw folder layout, republishes older stage names into new output fields, treats importer output as authority rather than source normalization, or teaches the old deterministic-baseline-versus-LLM override story in current docs. Record each seam in the plan as either delete now, quarantine as archived-reader-only, or keep because it is genuinely part of the current contract. The default should be delete now.

Then clean the artifact-discovery layer. In `cookimport/bench/external_ai_cutdown/artifact_paths.py`, `cookimport/bench/upload_bundle_v1_existing_output.py`, `cookimport/bench/followup_bundle.py`, and `scripts/benchmark_cutdown_for_external_ai.py`, remove fallback discovery that rummages through historical paths when current manifests or current stage-observability data should already tell the truth. The replacement rule is simple: for current runs, use the canonical manifest pointer or semantic stage index; if that pointer is missing, fail closed with a precise diagnostic rather than silently searching a wider historical tree. If archived-run support is still needed, move it behind a clearly named archived-reader helper that is not used when current manifest metadata exists.

Next, clean the topology and terminology layer. Review all current benchmark bundle renderers, summaries, and follow-up packets to ensure they emit only current semantic stage names and current authority boundaries. Delete any field generation, rendering helper, or explanatory prose that reintroduces pass-slot terminology, numbered nonrecipe stages, or importer-comparison diagnostics for current runs. If an archived reader must map old names into current semantic names for local analysis, do that mapping privately and emit only the normalized semantic result.

After the code paths are clean, collapse the docs to match reality. Update `docs/01-architecture/01-architecture_README.md` and `docs/07-bench/07-bench_README.md` so they no longer speak about the remaining compatibility seams as open-ended coexistence. Update `docs/AI_context.md` last. Its long “historical noise” guidance should become much shorter once the active code no longer re-teaches stale concepts. The goal is not to pretend history never existed; it is to stop making new readers carry a large anti-confusion payload just to survive current tooling.

Keep the implementation deletion-first and test-backed. When a helper is only serving stale fixtures or historical guesses, prefer removing it and updating those fixtures or archived-reader call sites instead of layering more indirection around it. This is a solo project, so there is no need to preserve broad transition shims for hypothetical external integrators.

## Idempotence and Recovery

This plan is deletion-heavy but should still be executed safely. Delete compatibility seams in small, test-backed slices. If removing a helper breaks an archived fixture that still matters, either update the fixture to current contracts or move the minimum necessary logic into an explicitly archived-reader helper. Do not restore the old fallback into the active path just to make a test green.

When a deletion changes current diagnostics, update the tests to match the new fail-closed behavior rather than reintroducing path guessing. If a change turns out to break a genuinely current workflow, record that in `Surprises & Discoveries`, revert only the specific deletion that was too aggressive, and document the current-contract reason for keeping that seam.

## Interfaces and Dependencies

The implementation should continue using the current repo owners rather than inventing a new abstraction layer for its own sake.

In `cookimport/runs/stage_observability.py`, the semantic stage index remains the current run-level authority for stage names and artifact pointers. Active benchmark and review readers should consume that model or the canonical run manifests instead of reconstructing topology from raw paths.

In `cookimport/bench/upload_bundle_v1_existing_output.py`, keep one normalized source model that converts current run artifacts into reviewer-facing bundle data. If older runs are still supported, normalize them privately here or in an explicitly archived helper, but emit only current semantic stage names and current artifact pointer semantics.

In `cookimport/bench/external_ai_cutdown/artifact_paths.py` and `scripts/benchmark_cutdown_for_external_ai.py`, keep path resolution current-contract-first and fail closed for missing current pointers. Historical recovery helpers, if any remain, should be clearly named as archived-only and should not sit inline on the active path.

In `cookimport/bench/followup_bundle.py`, current follow-up packet assembly should trust the canonical locators recorded by the bundle or run manifests. It should not infer a second truth source from directory guesses when the active contract is missing data.

At the end of the work, no current doc or current output should need to define or expose pass-slot or numbered-stage field names.
