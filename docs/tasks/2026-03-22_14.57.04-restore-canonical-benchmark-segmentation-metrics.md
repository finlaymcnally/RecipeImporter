# Restore canonical benchmark segmentation metrics

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

Canonical-text benchmark runs currently write `eval_report.json` files that include classification metrics and boundary-overlap counts, but they do not include the structural segmentation metrics that the benchmark comparison code expects. As a result, single-book comparisons can only say that a win was `classification_signal_only` even when the run may also have improved or regressed on recipe-boundary structure. After this change, canonical-text benchmark reports will carry structural segmentation metrics directly in `eval_report.json`, single-book codex-vs-vanilla comparisons will attribute deltas using those metrics when present, and downstream benchmark consumers will keep seeing a complete, stable contract.

The change is observable in three ways. First, a canonical-text eval report will now include a `segmentation` object with `boundaries.overall_micro.f1` and related per-boundary details. Second, a single-book paired benchmark comparison will stop claiming segmentation was unavailable when both runs were canonical-text runs. Third, focused benchmark tests that currently cover canonical evaluation and single-book comparison will assert the restored segmentation payload.

## Surprises & Discoveries

- Observation: the benchmark comparison code already knows how to read segmentation metrics from eval reports; the root bug is that canonical-text reports never serialize them.
  Evidence: `cookimport/cli.py` already uses `_single_book_eval_segmentation_summary(...)`, but that helper returns `segmentation_not_present_in_eval_report` for current canonical-text runs.

- Observation: canonical-text eval already has the exact projected gold/pred label sequences needed for structural boundary scoring, so this is a contract omission rather than a hard modeling gap.
  Evidence: `cookimport/bench/eval_canonical_text.py` builds `gold_for_metrics` / `pred_for_metrics` and line-aligned canonical labels before writing the report.

- Observation: the simplest reliable path was to reuse the same structural projection and error-taxonomy helpers already used by stage-block eval instead of inventing a second canonical-only segmentation contract.
  Evidence: importing the existing helpers let canonical-text reports emit `label_projection=core_structural_v1`, `boundaries.overall_micro`, and matching taxonomy fields with no new ontology drift.

## Context and Orientation

Canonical-text benchmark evaluation lives in [cookimport/bench/eval_canonical_text.py](/home/mcnal/projects/recipeimport/cookimport/bench/eval_canonical_text.py). That scorer aligns prediction blocks to canonical gold text, projects labels onto canonical lines, computes classification metrics, writes `eval_report.json`, and emits alignment diagnostics. Stage-block evaluation lives in [cookimport/bench/eval_stage_blocks.py](/home/mcnal/projects/recipeimport/cookimport/bench/eval_stage_blocks.py); it already computes structural segmentation metrics under `report["segmentation"]` using [cookimport/bench/segmentation_metrics.py](/home/mcnal/projects/recipeimport/cookimport/bench/segmentation_metrics.py). The single-book codex-vs-vanilla comparison builder lives in [cookimport/cli.py](/home/mcnal/projects/recipeimport/cookimport/cli.py) and reads variant eval reports to decide whether a result was classification-driven, segmentation-driven, mixed, or data-limited.

In this repository, “segmentation metrics” means structural boundary metrics derived from projected label sequences, not the looser canonical overlap counts stored in `report["boundary"]`. The structural metrics report true/false-positive and missed boundary counts across ingredient starts/ends, instruction starts/ends, and recipe splits, plus an `overall_micro` aggregate F1. Those are the metrics the comparison and dashboard logic use when they want to reason about whether a benchmark win came from structure or label semantics.

## Validation and Acceptance

Acceptance is behavioral.

Running canonical-text evaluation on a fixture with recipe-title, ingredient, and instruction lines must produce an `eval_report.json` whose top-level `segmentation` object contains `label_projection`, `boundary_tolerance_blocks`, and `boundaries.overall_micro`. That payload must be derived from canonical-line projections and must coexist with the older `boundary` overlap counts.

Running the single-book comparison artifact builder over paired eval reports that include canonical segmentation metrics must produce a comparison payload whose `metadata.variant_diagnostics.variants.*.segmentation.boundary_f1` fields are populated and whose likely-driver attribution no longer falls back to `classification_signal_only` solely because canonical eval omitted segmentation.

## Idempotence and Recovery

This change is additive and local to benchmark/eval contracts. Re-running the eval or comparison commands after the patch should simply overwrite the same `eval_report.json` and `codex_vs_vanilla_comparison.json` artifacts with the richer schema. If a partial edit leaves tests failing, the safe recovery path is to finish the scorer contract first, then update the comparison tests and docs so all benchmark consumers agree on the same report shape.
