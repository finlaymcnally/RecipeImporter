## Purpose / Big Picture

The benchmark run at `data/golden/benchmark-vs-golden/2026-03-21_14.53.27/single-book-benchmark/saltfatacidheatcutdown/codexfarm/` shows that the recent watchdog local-tool-surface change solved the original forbidden-command problem, but two follow-on failures still keep the Codex farm path from being trustworthy.

The first failure is in the line-role stage. A worker can write enough valid output for a shard to validate, then keep issuing helper commands until the workspace loop watchdog kills the session. The shard then ends up with contradictory state: `status == "validated"` but `state == "watchdog_killed"`. After this plan is implemented, line-role workspace workers must stop as `completed` once all expected packet outputs exist and stabilize, so a shard that already validated is no longer reported as a watchdog failure.

The second failure is in the knowledge stage. The current knowledge workspace contract relies on the repository mutating `current_packet.json`, `current_hint.md`, `current_result_path.txt`, and `packet_lease_status.json` while a long-lived worker session is already running. In the saved run, most workers completed or exited without watchdog intervention, yet 424 task packets never produced an output file. After this plan is implemented, knowledge workspace workers must receive an authoritative ordered task queue up front and must be able to complete every assigned packet without depending on timing-sensitive live lease-file mutation.

The observable result is simple. After implementation, targeted tests will show that line-role records validated shards as `completed` rather than `watchdog_killed`, and knowledge taskized workers will produce complete per-task packet coverage instead of `missing_owned_chunk_results` failures dominating the stage. A full live Codex rerun is useful, but it is explicitly out of scope for completing this plan unless the user later approves token spend.

## Context and Orientation

This repository has a “Codex farm” execution path that breaks stage work into shards and tasks, runs a workspace-oriented worker session, collects per-task JSON outputs, and then validates and aggregates those outputs back into stage results. A “watchdog” is the supervision callback that inspects the live worker snapshot and decides whether to let the session continue or terminate it with a reason code such as `watchdog_command_execution_forbidden` or `watchdog_command_loop_without_output`.

The line-role stage lives in `cookimport/parsing/canonical_line_roles.py`. The main workspace-worker path is `_run_line_role_workspace_worker_assignment_v1`. Its watchdog callback is built by `_build_strict_json_watchdog_callback`, and generic shell-loop termination lives in `cookimport/llm/codex_exec_runner.py::should_terminate_workspace_command_loop`. The line-role runtime writes one `out/<task_id>.json` packet per assigned atomic-line task, then validates and aggregates those packets into a shard-level `status.json` and proposal file.

The knowledge stage lives in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. The taskized workspace-worker path is `_run_knowledge_workspace_worker_assignment_taskized_v1`. Its current packet-flow helper is `_KnowledgePacketLeaseController`, which writes `current_packet.json`, `current_hint.md`, `current_result_path.txt`, and `packet_lease_status.json` inside each worker root and advances them from a background watcher thread after outputs appear. After the worker exits, the stage reads `out/<task_id>.json` files, validates each task packet with `_evaluate_knowledge_response`, and aggregates tasks back into shard proposals. Validation rejects shards with `missing_owned_chunk_results` whenever the returned task packets do not cover all chunk IDs owned by that shard.

The saved benchmark root that motivated this plan is `data/golden/benchmark-vs-golden/2026-03-21_14.53.27/single-book-benchmark/saltfatacidheatcutdown/codexfarm/`. The corresponding detailed runtime artifacts are under `data/output/2026-03-21_14.53.27/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-21_14.54.14/`. Keep both roots handy while implementing because the benchmark root has rolled-up counts, while the runtime root has the worker-level `status.json`, `live_status.json`, `events.jsonl`, and stage ledger files that explain why those counts happened.

## Milestone 1: Make line-role stop as completed once outputs stabilize

At the end of this milestone, a line-role workspace worker that already wrote all expected packet outputs will stop with `supervision_state == "completed"` and reason `workspace_outputs_stabilized` before the generic shell-loop watchdog can misclassify the shard as a failure. The proof is both behavioral and file-based: a focused line-role test must show `status.json` with `status == "validated"` and `state == "completed"`, and the summary counters must no longer increment `watchdog_killed_shard_count` for this scenario.

Implement this in `cookimport/parsing/canonical_line_roles.py`. Extend `_build_strict_json_watchdog_callback` so it can receive the list of expected workspace output paths for the worker, just as the knowledge stage already does. Track whether all expected files exist and whether the set of files is stable across consecutive supervision snapshots. If the outputs are complete and stable, return a `CodexExecSupervisionDecision.terminate(...)` with `supervision_state="completed"` and reason code `workspace_outputs_stabilized` before invoking `should_terminate_workspace_command_loop`.

Wire the expected output paths in from `_run_line_role_workspace_worker_assignment_v1`, which already knows every runnable task and their `out/<task_id>.json` destinations. After that, review the shard finalization path in the same function, especially the `status.json` writer and `_failure_reason_from_run_result`, so that a run result with `supervision_state == "completed"` is preserved as completed and does not get reported as a watchdog failure. The goal is not to paper over a kill after the fact; the watchdog should now exit cleanly, and the final status files should report that truthfully.

## Milestone 2: Replace the knowledge live-lease contract with an authoritative ordered queue

At the end of this milestone, a knowledge workspace worker will receive its full ordered packet queue before the session starts. The worker prompt will tell the model to iterate through that queue directly, using the provided `input_path`, `hint_path`, and `result_path` for each task. The proof is that a focused knowledge test with multiple tasks assigned to one worker produces one durable `out/<task_id>.json` file per assigned task and no longer relies on repository-side mid-session lease updates.

Implement this in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. Replace the current dynamic progression contract centered on `_KnowledgePacketLeaseController` with a simpler manifest-driven contract. The authoritative worker input should be `assigned_tasks.json`, not a mutable `current_packet.json`. Each task entry must include at least `task_id`, `parent_shard_id`, `owned_ids`, `input_path`, `hint_path`, `result_path`, `lease_sequence`, and `lease_total`. The worker prompt from `_build_knowledge_workspace_worker_prompt` must be rewritten to say exactly that: open `worker_manifest.json`, then iterate over `assigned_tasks.json` in order, write one result file per task to the declared `result_path`, and stop when every listed task has been handled.

Do not keep the old background mutation approach alive in parallel. It adds complexity without solving the failure we saw. Remove `_KnowledgePacketLeaseController` entirely or reduce it to a thin compatibility helper that only writes static manifests before the run starts. The worker root should still contain small, easy-to-navigate files, but none of them should require the model to notice asynchronous repository updates after the session begins.

When updating the runtime manifests, align `worker_manifest.json`, worker prompt assertions, and any examples or hints so they describe the new queue-driven contract. Keep the per-task `out/<task_id>.json` output format unchanged so `_evaluate_knowledge_response`, retry logic, repair logic, and shard aggregation stay compatible. The only thing changing is how the worker learns what to process next.
