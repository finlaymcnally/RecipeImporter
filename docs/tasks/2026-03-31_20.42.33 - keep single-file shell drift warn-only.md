# Keep single-file shell drift warn-only

Problem statement:
Single-file `task.json` workers currently get killed for repeated shell drift or ad hoc `task.json` rewrites even when they stay inside the workspace boundary. That wastes tokens and turns a steerable prompt-shape problem into a run failure.

Acceptance criteria:
- Recipe, knowledge, and canonical line-role workspace watchdogs record `single_file_shell_drift` warnings but do not terminate the session for that warning alone.
- Hard boundary violations still terminate the session.
- Tests cover repeated drift and ad hoc transform cases as warn-only behavior.

Verification steps:
- From `/home/mcnal/projects/recipeimport`, activate `.venv`.
- Run `pytest tests/llm/test_recipe_phase_workers.py tests/llm/test_knowledge_orchestrator_contracts.py tests/parsing/test_canonical_line_roles.py -q -k "single_file_shell_drift or egregious_single_file_shell_transform or line_role_workspace_watchdog or recipe_workspace_watchdog"`

Evidence:
- Pending test run after implementation.

Constraints/gotchas + brief decision:
- Keep the existing command classification and warning metadata because that is the steering signal and operator visibility surface.
- Change only the escalation behavior: boundary violations remain kill-worthy, shell drift becomes warn-only.

Rollback / compensation plan:
- Revert the watchdog callback changes in the three stage runtimes if warn-only behavior proves too permissive.
