# Prevent Knowledge Worker Drift, Watchdog Churn, and Stalled Benchmark Runs

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the non-recipe knowledge phase in a CodexFarm benchmark should behave like a bounded local worker system instead of a chaotic agent sandbox. A worker should know exactly what files to read, what commands are allowed, where scratch work belongs, and what will get it killed. The allowed path should be easier than the forbidden path. The watchdog should remain a backstop, but it should stop real boundary violations and pathological no-output loops, not punish normal bounded workspace automation after the fact. When a run still fails, retries and repairs should stay bounded, stale follow-up calls should not linger forever, the benchmark UI should report the true packet-level state, and an interrupted run should still leave a readable partial-result package.

The user-visible proof is straightforward. In a no-token replay or fake-runner harness built from the saved Salt Fat Acid Heat artifacts, the knowledge phase should report packet progress instead of `task 0/10`, worker rows should explain what each worker is doing, off-contract commands should be prevented early and clearly, local in-workspace shell helpers should not cause wasteful main-worker deaths, and once a shard is terminal the runtime should stop spawning or leaving behind stale follow-up calls. In a live rerun, which requires explicit user approval because it spends tokens, the same benchmark should either finish or fail closed without the previous retry storm.

## Surprises & Discoveries

- Observation: the knowledge stage was not one clean hang. It mixed productive packet writing, main-worker watchdog kills, malformed “completed” workers, stale repair calls, and a UI that stayed at shard-level `0/10`.
  Evidence: the saved run under `data/output/2026-03-20_21.44.20/.../knowledge/` contains `485` task packets, `339` top-level worker `out/*.json` files, `115` repair status files, `75` watchdog-retry status files, and multiple still-running follow-up statuses when the parent benchmark was killed.

- Observation: several worker deaths came from commands that look like bounded local automation, but some of those commands also used patterns the project should still treat as off-limits, especially absolute-path scratch files and inline interpreters.
  Evidence: saved `live_status.json` files show `/tmp/process_tasks.sh` heredoc scripts for worker `003`, `python - <<'PY'` for workers `005` and `008`, and a `bash` helper script around `jq` for worker `006`.

- Observation: relaxing watchdog kills alone would not have solved the run because some “successful” workers were already bulk-generating low-trust or malformed outputs.
  Evidence: worker `001` wrote `49/49` top-level output files but every file was invalid JSON with a literal trailing `EOF`, while worker `007` explicitly planned to mark every packet as useful knowledge and batch-generate all outputs with one `jq` loop.

- Observation: the current recovery path is too expensive for its yield.
  Evidence: in the saved run there were `199` structured follow-up calls started (`122` repair live statuses plus `77` watchdog-retry live statuses), but only `5` repairs ended as `repaired` and only `1` watchdog retry ended as `validated`.

- Observation: the current top-level stage never finalized enough metadata for a useful post-mortem without directly traversing worker trees.
  Evidence: the saved knowledge root contains `task_manifest.jsonl`, `shard_manifest.jsonl`, and `worker_assignments.json`, but no `phase_manifest.json`, `telemetry.json`, `promotion_report.json`, `failures.json`, or `knowledge_manifest.json`.

- Observation: packet-progress accounting must use the same fallback helper that the runtime uses for shard-owned tasks, not raw task-plan counts, or the UI can undercount unsplit fallback work.
  Evidence: the final implementation now computes displayed task totals from `_progress_task_ids_for_knowledge_shard(...)` inside `_run_direct_knowledge_workers_v1(...)` so both split packets and one-packet shard fallbacks report truthfully.

## Context and Orientation

The affected system spans six connected areas.

The first area is the long-lived workspace worker transport. In this repository, a “workspace worker” is a Codex session started in an isolated worker directory that already contains repo-written files such as `worker_manifest.json`, `assigned_shards.json`, `assigned_tasks.json`, `hints/*.md`, `in/*.json`, and `out/*.json`. The worker is supposed to read those files, decide one task packet at a time, and write one output file per task packet. The shared implementation lives in `cookimport/llm/codex_exec_runner.py` and `cookimport/llm/phase_worker_runtime.py`.

The second area is the knowledge stage itself in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. In this repository, a “shard” is a larger ownership bundle assigned to one worker, while a “task packet” is a smaller unit inside that shard that usually corresponds to one `out/<task_id>.json` file. The knowledge stage writes `task_manifest.jsonl` and `worker_assignments.json`, supervises the workspace worker, validates packet outputs, and may spawn task-scoped recovery work.

The third area is structured recovery. In this repository, “watchdog retry” means a strict one-shot follow-up call after a main worker died or drifted, and “repair” means another strict one-shot call that tries to salvage a near-miss output that was close to the schema but not acceptable. These calls are intentionally stricter than the main workspace worker and should stay that way. Their artifact trees currently live under `workers/*/shards/*.task-*/watchdog_retry/` and `workers/*/shards/*.task-*/repair_*`.

The fourth area is watchdog policy. The current runtime distinguishes between user-facing telemetry classification and kill/no-kill enforcement, but the saved run shows that the system still needs a stronger upstream contract. The important policy terms in this plan are:

- “Boundary violation”: a command or behavior that visibly leaves the worker’s local world or invokes a forbidden tool family. Examples are parent traversal, absolute-path writes outside the worker root, network tools, and inline interpreters.
- “Allowed local automation”: shell loops, redirects, pipes, `jq`, heredocs, and similar bounded file work that stay inside the worker root.
- “Prevention”: prompt, manifest, helper-surface, and early-supervision work that steers the model away from forbidden behavior before the watchdog has to kill anything.

The fifth area is the benchmark UI and timeseries in `cookimport/cli.py` and the benchmark helper code. The current UI consumes structured stage-progress payloads. For knowledge, the visible counter still drifts into shard semantics, which is why a `485`-packet run can look like `task 0/10 | running 10`.

The sixth area is partial finalization. When the user interrupts a broken benchmark, the current run may preserve raw worker trees but still fail to write enough top-level summary artifacts for easy diagnosis. That makes a long token-spending run much harder to inspect after the fact.

This plan applies one additional design rule everywhere: treat the worker as a zero-context copy of us. Assume it appears in the workspace cold, with no repo memory and no hidden understanding of the runtime. If that cold agent would still need to guess what file to open first, what tools are allowed, or what “done” looks like, the worker setup is incomplete.

The saved run this plan is built around lives at:

- benchmark root: `data/golden/benchmark-vs-golden/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/`
- knowledge root: `data/output/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-20_21.44.56/raw/llm/saltfatacidheatcutdown/knowledge/`

That run is the primary acceptance fixture for this plan. The implementation should make the runtime behave well on those same shapes without requiring live token spend to verify every change.

## Plan of Work

### Milestone 1: Rewrite the worker contract so the allowed path is obvious and shorter than the forbidden path

Start in the shared workspace-worker setup code, primarily `cookimport/llm/phase_worker_runtime.py` and the worker-manifest writing logic in `cookimport/llm/codex_exec_runner.py`. The worker root must become a clearer local environment. Add an explicit workspace-local `scratch/` directory, and make the worker manifest say in plain language that temporary helper scripts belong there if the worker truly needs them. Explicitly state that `/tmp` is forbidden, that parent traversal is forbidden, and that `python`, `node`, and other interpreters are forbidden even for “small helpers.”

Revise the worker-facing instructions so the first-read files answer five questions without ambiguity: what to read first, what files are authoritative, where outputs go, where scratch work goes, and what command families are forbidden. Give concrete good and bad examples. The good examples should include local `jq`, loops over `assigned_tasks.json`, redirects to `out/*.json`, and writes to `scratch/*.sh`. The bad examples should include `/tmp`, `python - <<'PY'`, `git`, `curl`, and anything that reaches back into the repo outside the worker root.

The worker prompt and manifest should also stop encouraging vague exploration. The current system leaves too much room for the model to infer its own tool policy from the environment. Replace that with a short “do this, not that” contract and one or two repo-owned example transforms that show how to read one task packet and write one output packet. The goal is not to forbid intelligence; the goal is to remove needless ambiguity so a bounded agent does not reach for a general interpreter or an off-root scratch file.

Design this milestone as onboarding for another copy of ourselves. The very first worker-facing file should orient a cold agent in one pass: what to read first, what files are authoritative, what command families are forbidden, where temporary work belongs, what success looks like, and what happens if a packet cannot be salvaged. The worker should not need repo source spelunking to become productive.

This milestone should also add a narrow helper surface for the most common temptations. If the worker repeatedly wants to loop over task files or create a tiny script, the runtime should pre-create a place and a pattern for that. If the worker keeps needing to inspect one example output shape, include one explicit example file or manifest note rather than forcing the model to reverse-engineer the contract from unrelated artifacts. The allowed path must feel pre-built.

Acceptance for this milestone is deterministic and local. The new worker manifest and prompt fixtures should make it possible to write tests that assert the instructions explicitly mention `scratch/`, explicitly forbid `/tmp` and interpreters, and include at least one workspace-local good example plus one forbidden example.

### Milestone 2: Change watchdog philosophy from shell-shape punishment to boundary enforcement plus early prevention

Continue in `cookimport/llm/codex_exec_runner.py` and the stage-specific watchdog call sites. Keep the existing idea that the watchdog may kill a worker, but separate three concepts cleanly.

The first concept is telemetry. The runtime should still classify command shapes for debugging and post-mortem reporting. It is useful to know whether a worker used loops, heredocs, `jq`, or something stranger.

The second concept is prevention. Add an early intervention seam for main workspace workers so the first visible forbidden-like move can trigger a corrective response before the runtime spends minutes or kills a productive worker. The exact implementation can be a one-time worker-local correction note, a repo-owned supervisor message, or another bounded mechanism, but the product goal is the same: if the model starts down a predictable forbidden path, the system should redirect it immediately instead of silently waiting to kill later.

The third concept is enforcement. Main workspace workers should die only for real boundary violations or clearly pathological no-output behavior. Real boundary violations include forbidden interpreters, repo/network tools, off-root absolute paths, parent traversal, or another explicit escape rule added by this plan. Shell complexity alone must not be a kill reason. A shell loop that reads `assigned_tasks.json`, writes `out/*.json`, and stores temporary text under `scratch/` should remain alive. Structured retry and repair calls should stay strict: command execution there remains off-contract unless the stage explicitly documents a narrow exception.

This milestone should also improve the kill reasons themselves. The live-status payload should clearly distinguish “used forbidden interpreter,” “used absolute off-root path,” “used parent traversal,” and “looped without output.” That distinction matters for future tuning. A single umbrella `watchdog_command_execution_forbidden` reason is not enough once the product wants to teach and prevent specific failures.

Acceptance for this milestone requires focused tests around the shared helper and at least the knowledge stage. The tests must prove all of the following: a bounded shell script under `scratch/` is allowed, `/tmp` is still forbidden, `python - <<'PY'` is still forbidden, and the first forbidden-like main-worker behavior triggers the new early-prevention path before the runtime escalates to kill.

### Milestone 3: Bound recovery churn and make terminal states truly terminal

The saved run shows that follow-up churn is now one of the main product failures. The runtime started almost two hundred structured recovery calls for a single knowledge phase, and most of them produced no useful result. This milestone fixes that in `cookimport/llm/codex_farm_knowledge_orchestrator.py` and any shared retry helpers it uses.

First, give every task packet a simple explicit terminal-state machine. A packet should move through named states such as “not started,” “main output accepted,” “main output rejected,” “watchdog retry accepted,” “watchdog retry rejected,” “repair accepted,” and “terminal failed.” Once a packet is terminal, the orchestrator must not spawn another follow-up for it. The current worker trees suggest the runtime can leave stale or duplicate follow-up work alive even after enough information already exists.

Second, bound retries and repairs per packet. The exact cap should be chosen during implementation and recorded in this plan’s `Decision Log`, but the system should never be allowed to create the kind of `115` repair status files and `75` watchdog-retry status files seen in the saved run without a hard stop. If a packet still fails after the bounded recovery budget is spent, fail it closed, record why, and move on.

Third, reap stale follow-up calls aggressively. The saved run contained nine still-running follow-up live statuses at shutdown, including calls that had been silent for several minutes after the parent shard was already effectively done. When the parent packet or shard becomes terminal, any still-running child follow-up should be cancelled or at least marked superseded so the runtime no longer waits on it or reports it as active work.

Fourth, distinguish malformed-but-productive workers from empty workers. Workers `001` and `002` “completed” in a way that produced many outputs but still left malformed or invalid artifacts. That should not be treated the same way as a dead worker with no outputs. The runtime should make an explicit decision about whether to salvage, repair, or fail the packet set, and the stage summary should say which happened.

Acceptance for this milestone is a new deterministic replay-oriented test harness. The tests should replay the saved run shapes and assert that the orchestrator stops after the bounded recovery budget, does not leave stale follow-up statuses once the packet is terminal, and records a machine-readable summary of accepted, failed, retried, repaired, and superseded packets.

### Milestone 4: Make the benchmark UI and stage artifacts report packet truth, follow-up truth, and partial truth

Now that the runtime has a cleaner contract and bounded recovery behavior, make the operator surface honest. Continue in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, `cookimport/cli.py`, and the relevant benchmark helper seams.

The visible task counter for knowledge should report task packets, not top-level shards. The worker rows should show shard ownership plus packet progress and current follow-up state when relevant. If a worker is currently on a retry or repair path, that should be visible in the row label or detail text. If a packet budget is exhausted and the packet is terminally failed, the progress display should still move forward rather than implying a stuck worker.

At the end of the phase, whether it succeeds or fails, the knowledge root must contain enough summary files that a novice can diagnose the result without drilling into every worker tree. At minimum this means a phase-level manifest, a telemetry summary, a failure summary, and a promotion summary or explicit no-promotion summary. The current artifact gap is too large.

Interrupted benchmark handling belongs here too. If the user stops a long knowledge-heavy benchmark, the benchmark root should still contain a partial summary that points at the preserved worker trees, records packet counts and terminal states known so far, and says the run was interrupted rather than completed. The point is not to pretend success; the point is to leave a coherent evidence package.

Acceptance for this milestone is twofold. The benchmark progress tests must assert packet-based progress and worker-row clarity, and a targeted interruption test must assert that a partial but readable benchmark package is written even when the user aborts mid-phase.

## Validation and Acceptance

This plan is complete when all of the following are true.

First, the worker contract is explicit and testable. The worker manifest and prompt clearly tell the model where scratch work belongs, what is forbidden, and what the good local patterns look like. A novice reading the manifest files in the repository can tell that `scratch/` is allowed, `/tmp` is forbidden, and inline interpreters are forbidden.

Second, the worker root is genuinely zero-context friendly. A fresh coding agent reading the first worker-facing file can determine the ordered task loop, the exact files to trust, the output target, the command policy, and the completion condition without relying on hidden repo lore.

Third, the main workspace-worker watchdog only kills on real violations or explicit liveness failures. A local shell loop under the worker root is allowed. `/tmp` writes, parent traversal, network tools, and inline interpreters are still killed. The first predictable violation path receives an early corrective intervention before the runtime escalates to kill.

Fourth, packet recovery is bounded and terminal. The orchestrator does not keep creating follow-up calls after a packet is terminal, does not leave stale child statuses alive after parent completion, and records packet-level accepted, failed, retried, repaired, and superseded counts in stage summary artifacts.

Fifth, the benchmark UI reports packet truth. In the knowledge phase, the visible task counter uses packet counts, worker rows explain shard ownership plus packet or follow-up state, and terminal failures still advance the visible progress rather than leaving the stage apparently stuck.

Sixth, interrupted runs are inspectable. If the user aborts a benchmark mid-knowledge, the benchmark root still contains a partial summary that names the interruption, points to preserved worker trees, and records known packet/shard counts and failure categories.

Seventh, the saved March 20 Salt Fat replay shapes are covered by deterministic tests. The acceptance path must not rely on “hopefully the live run is better” as the primary proof.

## Idempotence and Recovery

This plan is intentionally additive. Start with tests and saved-artifact fixtures so the current failure modes stay pinned while the runtime changes land.
