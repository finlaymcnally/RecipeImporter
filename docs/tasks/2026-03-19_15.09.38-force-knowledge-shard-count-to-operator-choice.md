# Problem

`knowledge_prompt_target_count` still behaves as a soft target. Live knowledge runs can emit more shards than the operator selected when char, locality, or table heuristics trigger extra splits.

# Acceptance Criteria

- Knowledge planning uses the selected `knowledge_prompt_target_count` as the literal shard count when it is set.
- Forced knowledge shards stay contiguous and deterministic.
- Oversized forced shards emit warnings instead of silently increasing shard count.
- Focused knowledge planner/orchestrator tests prove the old overflow behavior is gone.

# Verification

- `source .venv/bin/activate && pytest tests/llm/test_knowledge_job_bundles.py tests/llm/test_codex_farm_knowledge_orchestrator.py -q`

# Evidence

- Before code change, the updated focused knowledge tests exited non-zero because the planner still expanded `5 -> 10` and `2 -> 3`.
- After code change, `source .venv/bin/activate && pytest tests/llm/test_knowledge_job_bundles.py tests/llm/test_codex_farm_knowledge_orchestrator.py tests/llm/test_knowledge_phase_workers.py -q` exits `0`.

# Constraints / Decision

- Keep explicit `target_chunks_per_shard` behavior unchanged.
- Only the knowledge `knowledge_prompt_target_count` path changes from soft target to literal shard-count override.

# Rollback

- Revert this task's code/doc changes to restore the prior soft-target knowledge planner.
