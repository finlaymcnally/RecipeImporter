# Rebuild Knowledge Workers Around Bounded Task Results, Cheap Recovery, and Truthful Progress

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md`.

This plan supersedes the narrower March 20 knowledge-runtime plans when the work overlaps. Those earlier plans remain useful as artifact-level history, but this document is the one to execute for the full fix.

## Purpose / Big Picture

After this change, the non-recipe knowledge phase should behave like a bounded task processor instead of a free-form shell session that sometimes happens to produce useful outputs. A successful run should review task packets, produce canonical outputs without schema drift, stop when its owned work is done, surface packet-scale progress in the UI, and fail closed cheaply when a worker goes off task. A broken run should leave explicit terminal artifacts instead of a mystery folder full of half-finished worker state.

The user-visible proof is straightforward. Re-run the Salt Fat Acid Heat single-offline CodexFarm benchmark in a no-token fake-runner harness or, with explicit approval, in a live run. The knowledge stage should no longer sit at `task 0/10 | running 10` while hundreds of packets churn underneath. Successful workers should not emit thousands of shell commands. Invalid top-level outputs should not explode into a giant repair storm. When the stage ends, every task packet should have a terminal state, every worker should have a non-ambiguous reason code, and the benchmark root should contain a clear final or partial summary package.

## Outcomes & Retrospective

This plan is not implemented yet. The main outcome so far is architectural clarity: the problem is no longer described as “knowledge workers sometimes get watchdog killed.” The real problem is that the knowledge stage currently has no crisp notion of what the worker is allowed to decide, what the repo is responsible for assembling, when a task packet becomes terminal, or when follow-up work should stop. This plan resolves those ambiguities in favor of a bounded, observable task system.

The main open risk is the amount of plumbing needed in `cookimport/llm/codex_exec_runner.py` and `cookimport/llm/codex_farm_knowledge_orchestrator.py` to support a more repo-owned task loop without breaking the current long-lived workspace session transport. That is why the first milestone below includes a narrow transport prototype rather than treating the execution mechanism as already solved.

## Context and Orientation

The current knowledge phase lives primarily in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. In this repository, a “shard” is a top-level bundle assigned to one worker, and a “task packet” is a smaller unit inside that shard, usually one chunk bundle that should eventually become one reviewed task result. The current runtime starts one long-lived Codex workspace worker per shard assignment. That worker receives files such as `worker_manifest.json`, `assigned_tasks.json`, `hints/*.md`, and `in/*.json`, then writes `out/*.json` files directly into its workspace.

The direct-exec transport lives in `cookimport/llm/codex_exec_runner.py`. It supervises long-lived workspace sessions and one-shot structured follow-up calls. The workspace path is relatively permissive because the worker is supposed to read and write bounded local files. The structured follow-up path is much stricter because it expects a direct JSON answer and not shell use.

The knowledge stage also depends on `cookimport/llm/knowledge_prompt_builder.py`, the prompt file `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md`, the schema file `llm_pipelines/schemas/recipe.knowledge.v1.output.schema.json`, and the ingest/promotion helpers in `cookimport/llm/codex_farm_knowledge_ingest.py` and nearby knowledge modules. The benchmark progress UI flows through `cookimport/core/progress_messages.py`, `cookimport/cli.py`, and the benchmark orchestration around `cookimport/labelstudio/ingest.py`.

The saved failure root for this plan is `data/output/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-20_21.44.56/raw/llm/saltfatacidheatcutdown/knowledge/`. That root is the local truth bundle for the command-explosion and recovery-churn failures. The paired benchmark truth for operator-visible progress is `data/golden/benchmark-vs-golden/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/processing_timeseries_prediction.jsonl`.

This plan uses four plain-language terms that matter:

“Main worker” means the long-lived workspace session that owns a whole shard assignment.

“Follow-up call” means a later strict retry or repair subprocess created after the main worker failed, drifted, or produced invalid output.

“Repo-owned serialization” means the repository converts reviewed task judgments into the final canonical JSON output file. The model should not need to hand-type the final file shape.

“Terminal task state” means a task packet is done forever for this run. Examples are `validated_from_main_worker`, `validated_from_retry`, `repaired`, `failed_schema`, `failed_watchdog`, or `skipped_oversized_retry`. A terminal task must not keep spawning new work.

## Plan of Work

The work starts by changing what a successful knowledge worker is allowed to do. Today the worker is effectively told to inspect its local files and somehow leave valid `out/*.json` files behind. That blurs semantic review with file-generation mechanics and invites shell scripting. The new contract must separate those roles. The worker should decide whether a packet is useful, which blocks remain knowledge versus other, and which grounded snippets support that conclusion. The repository should then turn that decision into canonical output files and terminal task records.

That separation should be implemented with a new knowledge task result schema and a repo-owned task ledger. Each task packet should receive exactly one task result object from the main worker or a follow-up call. The ledger should record who produced that result, whether it parsed, whether it validated semantically, and whether the task is still eligible for one more recovery action. The existing shard-level status files should become summaries derived from this ledger instead of loosely coupled ad hoc files.

The worker transport itself then needs to become bounded. The main worker should still be allowed to read local files, but it should no longer need to invent a shell pipeline to batch-write the final outputs. The prompt should explicitly forbid bulk shell output generation and example hunting after the initial workspace orientation. The watchdog should gain phase-specific budgets: a small orientation budget before the first reviewed task result, a small per-task command budget between results, and an immediate stop condition once all owned packets have terminal reviewed results. Repeated schema-hunting `rg` loops, post-output self-audits, and long no-result command streaks should end the worker with explicit reasons such as `watchdog_exploration_without_result` or `watchdog_post_completion_drift`.

Recovery must also become task-ledger-driven. A main worker kill should not cause a broad shard retry storm. Instead, only the packets still lacking terminal reviewed results should be eligible for follow-up. Each task should have a small, explicit allowance such as “one watchdog retry” or “one repair if the output is near-valid.” If the task result is malformed in a way the repo can classify cheaply, the follow-up should be skipped and the terminal state should explain why. Once a parent shard is terminal because all packets are terminal, any in-flight follow-up calls tied to that shard must be canceled or ignored deterministically.

The user-facing progress and artifacts then need to derive from the same truth. Packet counts, worker rows, follow-up counts, and finalization should all read from the task ledger. That means the benchmark UI should report packet progress, the worker list should describe main and follow-up state honestly, and the knowledge root should always emit `phase_manifest.json`, `telemetry.json`, `failures.json`, and a promotion summary even for interrupted or failed runs.

## Milestone 1: Prototype and land the knowledge worker v2 transport contract

Before rewriting the whole phase, add a narrow prototype in `cookimport/llm/codex_exec_runner.py` and `cookimport/llm/codex_farm_knowledge_orchestrator.py` that proves the repository can drive a long-lived knowledge worker through explicit per-task review turns without asking the model to batch-write final outputs. The prototype does not need to cover every production path. It only needs to show that one long-lived workspace session can be asked to review one named task packet, return one structured task result, and then move on to the next packet while the repository writes the canonical stored result.

The target production contract after this milestone is that a main worker may still inspect `worker_manifest.json`, `assigned_tasks.json`, `hints/*.md`, and `in/*.json`, but it should never need to create `out/*.json` directly. The repository should create a new worker-local input or control surface that says which task is currently active and where the worker should look for the task packet. The final reviewed result should come back through a strict structured answer path or another equally bounded repo-owned channel. If the prototype proves that a reused conversation with one task per turn is workable, use that as the target. If the prototype reveals a hard transport limitation, keep the session model but still shift final-file serialization out of the worker.

This milestone is independently successful when a fake-runner or saved-artifact replay can process a small multi-task knowledge shard without any `cat <<EOF > out/...`, `python - <<'PY'`, or `jq > out/...` writes from the worker, and the repo still stores canonical task results.

## Milestone 2: Replace free-form final JSON writing with repo-owned task result assembly

Once the transport can carry bounded task reviews, introduce a new internal result type under the knowledge runtime modules. The result type should contain only semantic fields: task id, chunk ids, useful-versus-not-useful decision, per-block keep-or-drop authority, reviewer category, and grounded snippet evidence. The repo should then map that semantic object into the canonical stored output shape.

The implementation should touch `cookimport/llm/codex_farm_knowledge_models.py`, `cookimport/llm/codex_farm_knowledge_ingest.py`, and the orchestrator. The main worker and all follow-up calls should now be judged on semantic validity, not on whether they reproduced the exact canonical JSON shape. This is the milestone that eliminates worker `001`-style failures where an otherwise interpretable answer became unusable only because the worker literally wrote `EOF` into the file.

This milestone is complete when malformed final file syntax from the worker is no longer a failure class, because the worker no longer owns the final file syntax.

## Milestone 3: Add command budgets and off-task detection that match the new worker contract

After the worker contract is narrowed, update the workspace watchdog logic in `cookimport/llm/codex_exec_runner.py` and `cookimport/llm/codex_farm_knowledge_orchestrator.py` so it enforces useful boundaries instead of generic shell permissiveness. The main worker should be allowed a short orientation period to inspect the manifest and the first task packet. After that, the watchdog should measure progress in reviewed task results, not just output files or raw command counts.

Add three cheap liveness rules. First, if a worker crosses a small pre-first-result command budget while mostly searching for schema examples, kill it as exploration drift. Second, once the worker has started producing reviewed task results, cap the command budget between results so repeated `rg`, `sed`, or `cat` loops without a new reviewed result are treated as off-task churn. Third, once all owned task packets are terminal, end the session immediately instead of allowing post-completion self-audits.

The goal is not zero shell. The goal is that a `48`-packet worker no longer emits `500` to `2000` commands. The acceptance target for the v2 contract is that a healthy successful worker should stay well below `150` completed shell commands for a `48` to `49` packet shard, and most of those commands should be direct local file reads rather than search loops or output scripting.

## Milestone 4: Replace shard-wide recovery storms with ledger-driven, task-bounded recovery

Introduce a repo-owned knowledge task ledger in `cookimport/llm/codex_farm_knowledge_orchestrator.py` or a nearby knowledge runtime module. The ledger must record one row per task packet and hold the current terminal or non-terminal state, the latest producer, validation outcome, and remaining recovery allowance.

Main worker failure should now leave some packets already terminal and others unresolved. Only unresolved packets should be eligible for follow-up. A packet that failed schema validation but contains a near-complete semantic object may get one repair. A packet whose main worker never returned a result may get one watchdog retry. A packet whose retry is oversized, malformed, semantically empty, or reasoning-only should become terminal failed with an explicit reason and no more follow-up work. This milestone should delete the current behavior where a handful of bad main-worker outputs can explode into more than a hundred recovery subprocesses.

This milestone is complete when the March 20 artifact replay would have produced a small bounded number of follow-up calls, with a target of no more than one strict retry plus one repair per unresolved task packet and zero running follow-up calls after the parent shard became terminal.

## Milestone 5: Make progress, telemetry, and summaries derive from the task ledger

Update `cookimport/core/progress_messages.py`, `cookimport/llm/codex_farm_knowledge_orchestrator.py`, and `cookimport/cli.py` so packet truth and worker truth come from the same state model. The stage counter should use packet counts, not shard counts. Worker rows should show the shard id plus packet progress and follow-up state. Detail lines should include terminal packet counts by outcome, active follow-up counts, and remaining unresolved packets.

Also update benchmark finalization in the benchmark orchestration path so the knowledge root always writes a small, explicit artifact set even when the stage fails or the benchmark is interrupted. At minimum, the knowledge root should contain `phase_manifest.json`, `telemetry.json`, `failures.json`, and a promotion or summary artifact that points to the task ledger and worker roots. The benchmark root should then reference that partial or final stage package.

This milestone is complete when a future operator can open the benchmark root and tell, without reading raw worker event logs, how many packets succeeded from the main worker, how many needed retry, how many were repaired, how many failed closed, and why the stage stopped.

## Milestone 6: Tighten prompts and hints so workers stop hunting for schema examples

Update `cookimport/llm/knowledge_prompt_builder.py`, `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md`, and the worker-facing hint generation so the worker no longer needs to infer the schema by searching the workspace. The prompt should embed one exact positive example and one exact negative example of the v2 task result object in the compact style the repo now expects. It should also say, plainly, that schema hunting, bulk shell scripting, and post-output verification loops are off task for the main worker.

The hints should reinforce grounding instead of lane metadata leakage. Workers should not infer “all keep” or “all drop” from deterministic lane tags. They should review the packet text, produce the bounded semantic result, and stop.

This milestone is complete when the no-token worker traces stop showing repeated `rg` and `cat` searches for schema terms such as `block_decision`, `cid`, `u`, or `rc` after the initial orientation phase.

## Validation and Acceptance

The implementation is complete when the following behavior is true.

First, the main worker no longer owns final file syntax. A worker may still inspect its task packet locally, but the repo writes the canonical stored task result. The March 20 `EOF` residue failure class must disappear entirely because there is no path where a worker’s raw heredoc text is treated as the authoritative final file.

Second, a successful `48` or `49` packet knowledge worker must not look like a shell loop author. In the saved-artifact harness or fake-runner tests, the main worker command count must remain bounded and materially smaller than the March 20 numbers. The target is under `150` completed commands for a successful `48` to `49` packet shard, with no repeated schema-search or bulk output-generation patterns after the initial orientation.

Third, every task packet must reach exactly one terminal state in the task ledger, and terminal packets must not keep creating work. When the stage finishes or is interrupted, there must be zero live follow-up calls that still matter to a terminal shard.

Fourth, the recovery layer must be sparse. A saved-artifact replay of the March 20 failure class must not explode into the old `115` repairs and `75` watchdog retries. The acceptance target is that unresolved packets get at most one strict retry and one repair, with many malformed or oversized cases becoming terminal failures without spending both.

Fifth, the UI and summaries must tell the truth. The stage must report packet counts, worker ownership, and follow-up counts from the same ledger, and the knowledge root must always contain explicit summary artifacts even on failure or interruption.

Sixth, worker outcome labels must be unambiguous. Every completed or failed main worker and follow-up call must carry a non-null reason code that distinguishes success, boundary violation, off-task exploration, no-result churn, malformed result, semantic failure, retry skip, repair failure, and operator interruption where applicable.

## Idempotence and Recovery

The implementation should be additive and safe to repeat. The first changes should be tests and a side-by-side prototype path rather than an immediate destructive rewrite. If the v2 worker transport proves harder than expected, keep the old knowledge path behind a temporary feature switch or stage-local branch while landing the task ledger and repo-owned serialization first. Do not leave the repo with half a contract. At every stopping point, the plan and the code should make clear which path is authoritative.

If a milestone reveals that the transport prototype cannot support one-task-per-turn long-lived sessions cleanly, document that in `Surprises & Discoveries` and switch to the narrower fallback already named in Milestone 1: keep the current session model but still move final-file ownership and terminal accounting into repo code. That fallback still addresses most March 20 failures and is preferable to postponing the whole rebuild.

Do not delete the March 20 artifact roots. They are the regression fixture set for this plan and the best proof that the new system is genuinely better.
