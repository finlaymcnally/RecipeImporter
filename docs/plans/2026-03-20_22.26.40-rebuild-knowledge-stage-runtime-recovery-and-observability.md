## Purpose / Big Picture

After this change, the non-recipe knowledge phase should stop behaving like an expensive, ambiguous agent sandbox and start behaving like a bounded production stage with honest progress, explicit packet outcomes, bounded recovery, and reliable postmortem artifacts. A run that reaches knowledge review should do six user-visible things correctly: main workers should understand a narrow local contract, off-task drift should be prevented earlier, malformed or low-trust task outputs should be rejected immediately, recovery should be bounded and worth the cost, the UI should report real packet-level state rather than fake shard-level calm, and an interrupted or failed run should still leave a coherent stage-level summary package.

The concrete proof is straightforward. First, run a deterministic no-token replay or fake-runner harness built from the saved Salt Fat Acid Heat benchmark roots. The knowledge phase should report packet progress rather than `task 0/10`, follow-up calls should stay bounded, stale repair or retry calls should not remain live after the parent task or shard is terminal, and the stage root should always contain its own manifest, telemetry, failures, and promotion summary files even on interruption. Then, only with explicit user approval for model spend, rerun the Salt Fat single-offline CodexFarm benchmark and verify that the stage either completes or fails closed without the previous retry storm, stale follow-up calls, and empty top-level artifact package.

## Surprises & Discoveries

- Observation: the knowledge phase was not a simple hang. It mixed productive packet writing, main-worker watchdog kills, malformed “completed” workers, stale follow-up calls, and a UI that never left shard-level `0/10`.
  Evidence: the saved run under `data/output/2026-03-20_21.44.20/.../knowledge/` contains `485` task packets, `339` top-level worker `out/*.json` files, `115` repair status files, `75` watchdog-retry status files, and multiple still-running follow-up statuses at shutdown.

- Observation: the runtime contract is internally contradictory. Repo docs say bounded local shell work is allowed for main workspace workers, yet several workers were killed for shell automation patterns that stayed within the workspace.
  Evidence: workers `003`, `005`, `006`, and `008` ended `watchdog_command_execution_forbidden` on local heredoc or helper-script patterns recorded in their `live_status.json` files.

- Observation: relaxing the watchdog alone would not have repaired this run.
  Evidence: worker `001` “completed” with `49/49` output files but every file was invalid JSON with a literal trailing `EOF`, while worker `007` explicitly planned to mass-generate blanket `keep/knowledge` judgments for every task packet.

- Observation: “completed” worker state is not currently a reliable success signal.
  Evidence: workers `001` and `002` ended with `reason_code=process_exited_without_watchdog_intervention` and `final_agent_message_state=malformed`, while workers `004`, `007`, `009`, and `010` ended with `workspace_outputs_stabilized`.

- Observation: the recovery lane is currently low-yield and can itself spiral or stall.
  Evidence: the saved run started `122` repair calls and `77` watchdog-retry calls, but only `5` repairs ended `repaired`, only `1` watchdog retry ended `validated`, `9` follow-up calls were still `running` at shutdown, and one repair call died with `watchdog_reasoning_without_output`.

- Observation: the benchmark UI and timeseries are only as honest as the structured payload emitted by the orchestrator.
  Evidence: the knowledge stage had `485` task packets in `task_manifest.jsonl`, but the benchmark tick log remained `task 0/10 | running 10` for about `653` seconds because the orchestrator still reported shard totals.

- Observation: the stage did not finalize its own contract after interruption.
  Evidence: the knowledge root contains `task_manifest.jsonl`, `shard_manifest.jsonl`, and `worker_assignments.json`, but no `phase_manifest.json`, `telemetry.json`, `promotion_report.json`, `failures.json`, `knowledge_manifest.json`, or proposal summary files.

- Observation: “workers off task” includes more than shell-policy violations.
  Evidence: worker event logs show metadata-driven or blanket output strategies such as lane-driven bulk generation, per-task templating with generic reasons, and all-knowledge `jq` loops over every assigned task.

## Outcomes & Retrospective

This plan is not implemented yet. The current outcome is a single canonical execution plan that replaces the narrower March 20 drafts with one end-to-end knowledge-stage program. The main lesson from the postmortem is that the failures came from one broken contract spread across prompt text, worker manifests, watchdog policy, output validation, retry and repair policy, progress reporting, and finalization. The implementation therefore needs milestones, but the final result must be evaluated as one coherent system.

## Context and Orientation

The non-recipe knowledge phase is the optional Codex-backed stage that reviews outside-recipe spans and decides whether each chunk is reusable cooking knowledge or merely other non-recipe text. The stage lives primarily in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. Validation and ingest logic live in `cookimport/llm/codex_farm_knowledge_ingest.py`. Prompt assembly lives in `cookimport/llm/knowledge_prompt_builder.py`. The model-facing prompt and schema live in `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md` and `llm_pipelines/schemas/recipe.knowledge.v1.output.schema.json`.

The runtime uses two execution modes. A “workspace worker” is a long-lived Codex session inside an isolated worker directory. That directory contains repo-written files such as `worker_manifest.json`, `assigned_shards.json`, `assigned_tasks.json`, `hints/*.md`, `in/*.json`, `out/*.json`, and per-task `shards/...` subdirectories. A “structured follow-up” is a one-shot strict-JSON Codex call used for watchdog retry or repair after the main worker produced no accepted result. The shared subprocess and live-supervision seam for both modes lives in `cookimport/llm/codex_exec_runner.py`.

In this repository, a “shard” is one larger ownership bundle assigned to a worker. A “task packet” is the smaller unit inside that shard that should yield one `out/<task_id>.json` file and one packet-level terminal outcome. The failing Salt Fat benchmark used `10` shard workers but `485` task packets in the knowledge phase. That is why shard-only progress reporting, shard-only completion logic, and shard-only recovery accounting were misleading.

The benchmark UI and progress timeseries live in `cookimport/cli.py` and related benchmark helper code. They do not invent stage truth on their own. They display structured progress payloads emitted by the orchestrator through `cookimport/core/progress_messages.py`. If the orchestrator lies in shard units or leaves active follow-up work invisible, the UI will faithfully display that lie.

The stage is expected to write a phase package under `raw/llm/<slug>/knowledge/`, including at least `phase_manifest.json`, `shard_manifest.jsonl`, `task_manifest.jsonl`, `worker_assignments.json`, `promotion_report.json`, `telemetry.json`, `failures.json`, and proposal or explicit failure artifacts. The postmortem run did not finish this package, which means the phase failed its own runtime contract, not merely its cosmetic UI.

The primary saved evidence bundle for this plan is:

- Benchmark root: `data/golden/benchmark-vs-golden/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/`
- Processed run root: `data/output/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-20_21.44.56/`
- Knowledge root: `data/output/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-20_21.44.56/raw/llm/saltfatacidheatcutdown/knowledge/`

Anyone implementing this plan should preserve those roots and use them as the primary regression fixture while building deterministic tests and replay helpers.

## Plan of Work

### Milestone 1: Rewrite the main worker contract so the allowed path is explicit, narrow, and local

Start by aligning the worker prompt, worker manifest, and workspace layout so the worker does not have to infer its own policy from the environment. This work belongs in `cookimport/llm/phase_worker_runtime.py`, the workspace-worker setup code in `cookimport/llm/codex_exec_runner.py`, the knowledge-specific prompt assembly in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, `cookimport/llm/knowledge_prompt_builder.py`, and the prompt asset `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md`.

Add one explicit workspace-local scratch area, for example `scratch/`, and make the worker manifest explain in plain language that temporary helper scripts belong there if they are needed at all. The manifest and prompt must answer five questions without ambiguity: what file to open first, which files are authoritative, where outputs belong, where scratch work belongs, and which command families are forbidden. The good examples must include local `jq`, loops over `assigned_tasks.json`, redirects to `out/*.json`, and writes to `scratch/*.sh`. The bad examples must include `/tmp`, parent traversal, `python - <<'PY'`, `node`, `git`, and network tools.

At the same time, remove vague invitation to explore. The worker should be told to decide each task from chunk-local evidence in `in/*.json`, use `hints/*.md` only as guidance, and stop once every assigned task has a terminal local result. The prompt and manifest should also say that bulk templating from metadata, lane hints, or one blanket transform over every packet is off-contract even if it stays inside the workspace.

The acceptance for this milestone is local and deterministic. A test or fixture that inspects the written worker manifest and prompt must make it obvious to a novice that `scratch/` is allowed, `/tmp` is forbidden, bounded local shell automation is allowed, inline interpreters are forbidden by default, and task-local grounded review is the intended path.

### Milestone 2: Add early prevention so predictable forbidden or off-task behavior is stopped before expensive worker death

Once the contract is clearer, teach the runtime to intervene earlier. This work belongs mainly in `cookimport/llm/codex_exec_runner.py` and the knowledge-stage supervision path in `cookimport/llm/codex_farm_knowledge_orchestrator.py`.

The current system largely notices bad behavior only when it is ready to kill. That is too late. Add a small prevention layer that recognizes obvious bad paths before they turn into long doomed runs. For example, if a main worker reaches for `/tmp`, an inline interpreter, or a large schema-searching exploration loop, the runtime should record a corrective event and either reject the command immediately or terminate the worker promptly with a precise reason. The important product goal is not “be nicer before killing.” It is “stop predictable waste early.”

This milestone must also formalize the difference between bounded local automation and true boundary or drift violations. Main workers should be allowed to use local shell constructs, `jq`, redirects, pipes, and loops that stay inside the worker root. They should still be denied parent traversal, absolute-path scratch files, network access, package managers, and general interpreters. The key is that the allow-list and deny-list must match the manifest and prompt from Milestone 1.

The acceptance is one pair of deterministic tests for each side of the policy. A local shell helper under the worker root must be tolerated. A `/tmp` helper or `python - <<'PY'` must be rejected. A metadata-driven schema-search loop must be classified as off-task or unproductive rather than allowed to burn minutes and then die ambiguously.

### Milestone 3: Rebuild packet state so every task reaches exactly one explicit terminal outcome

The current runtime loses control because task state is implicit and spread across local files, main-worker status, retry status, and repair status. Rebuild packet truth in `cookimport/llm/codex_farm_knowledge_orchestrator.py` so each task packet has one explicit lifecycle and one explicit terminal outcome. Keep the representation private if possible, but make the resulting behavior obvious in code and tests.

Each task packet should move through named states such as `not_started`, `main_output_seen`, `main_output_accepted`, `main_output_rejected`, `retry_started`, `retry_accepted`, `retry_rejected`, `repair_started`, `repair_accepted`, `repair_rejected`, `cancelled`, or `terminal_failed`. The exact names can change during implementation, but the core property must not: once a packet is terminal, no further follow-up work may be started for it, and any stale child work must stop affecting stage truth.

Shard completion must be derived from packet truth rather than from loose worker-local heuristics. A shard is done when all of its packets are terminal, whether they ended accepted or failed. That is the only honest way to derive stage progress, follow-up counts, and stage finalization.

The acceptance is a deterministic replay-oriented orchestrator test that reproduces the saved March 20 mixture of main worker death, malformed output, partial packet coverage, retry attempts, repair attempts, and stale follow-up calls. The test must prove that the new packet state machine converges to terminal states rather than leaking ambiguity.

### Milestone 4: Reject malformed, schema-invalid, semantically empty, and low-trust outputs immediately

The postmortem showed that “output file exists” is too weak. Worker `001` wrote every top-level output file and still contributed almost entirely bad work. This milestone tightens the first acceptance seam. The main changes belong in `cookimport/llm/codex_farm_knowledge_orchestrator.py` and `cookimport/llm/codex_farm_knowledge_ingest.py`.

As soon as a new `out/<task_id>.json` file appears, parse it. If it is not valid JSON, mark the packet as malformed output immediately and record the reason. If it is valid JSON but fails the schema, record the schema failure immediately. If it passes the schema but fails semantic grounding checks such as chunk coverage, useful-versus-non-useful invariants, or semantically empty collapse, record that too. Do not allow dozens of bad files to pile up before the stage notices.

This is also where the runtime must become more skeptical of low-trust bulk outputs that are technically parseable. Deterministic structural checks should catch patterns such as identical all-knowledge decisions across many packets, whole-chunk echo snippets that show no selection, reason strings copied from metadata instead of grounded in text, wrong bundle versions, or other template-like shapes seen in the postmortem. This is not a speculative AI detector. It is a deterministic contract check for the exact failure modes already observed.

The acceptance is a focused test set that includes the literal trailing-`EOF` malformed JSON shape, the saved worker-002 wrong-version shape, semantically empty or coverage-mismatched retry outputs, and one blanket templated knowledge output. Each should be rejected immediately with a specific recorded reason.

### Milestone 5: Replace open-ended retry and repair churn with bounded, explicit recovery

Now that packet truth and first-line validation are explicit, rebuild recovery in `cookimport/llm/codex_farm_knowledge_orchestrator.py` and any tiny shared helper seams in `cookimport/llm/codex_exec_runner.py`.

Each packet must get a small bounded recovery budget. The exact budget should be chosen during implementation and recorded in the `Decision Log`, but the core policy is fixed: at most one watchdog retry and one repair attempt unless a specific test-backed reason justifies one extra path. After the budget is exhausted, the packet is terminally failed. Do not let the runtime create the current fan-out of `122` repair calls and `77` watchdog-retry calls without a hard stop.

Then add aggressive stale-follow-up cancellation. If a shard is terminal, any still-running task-level retry or repair under that shard must be cancelled or marked superseded immediately. If a packet has already been accepted or terminally failed, older child follow-ups must be ignored and, where practical, terminated. If the benchmark or stage is interrupted, still-running follow-up calls must be marked cancelled and finalized rather than left forever `running`.

The saved worker-007 stale repair and the worker-008 repair that died with `watchdog_reasoning_without_output` are regression fixtures for this work. The acceptance bar is that no child follow-up remains authoritative or alive after the parent packet or shard is terminal, and that a bad packet stops after its bounded recovery budget instead of creating churn.

### Milestone 6: Make progress, worker rows, and benchmark timeseries packet-based and follow-up-aware

Once packet state is explicit and bounded, rebuild the progress surface in `cookimport/llm/codex_farm_knowledge_orchestrator.py` and keep the renderer changes in `cookimport/cli.py` as small as possible.

For the knowledge phase, `task_current` and `task_total` must mean task packets, not shards. Detail lines should preserve shard counts, follow-up counts, and configured worker counts. Worker labels should show shard ownership and packet progress while the main worker is alive, and if a worker is currently on retry or repair work that should be visible too. The progress surface should be able to tell the truth about a stage like the saved run, where some main workers are done, some are dead, and some follow-up calls are still active.

The rendered user-visible goal is something like:

    Running codex-farm non-recipe knowledge review... task 233/485 | running 6
    configured workers: 10
    completed shards: 3/10
    retry calls running: 2
    repair calls running: 4
    worker 10: saltfatacidheatcutdown.ks0009.nr (47/48 task packets)

The acceptance is that the saved-run harness can no longer produce a flat `task 0/10 | running 10` display for the whole knowledge phase, and that worker rows expose the parent shard plus current packet or follow-up truth.

### Milestone 7: Finalize stage-level artifacts on success, failure, and interruption

The knowledge stage must always write its own summary package. This work belongs mainly in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, with any necessary integration touches in `cookimport/labelstudio/ingest.py`, `cookimport/cli.py`, and `cookimport/runs/stage_observability.py`.

Move finalization earlier and make it unconditional on “happy completion.” On success, partial failure, watchdog collapse, or interruption, the stage root must contain a coherent `phase_manifest.json`, `telemetry.json`, `failures.json`, `promotion_report.json`, and any proposal or explicit failed-packet summaries. A user who stops a long run must still get a readable stage-level outcome, not only raw worker debris.

Then connect this stage truth to the benchmark interruption path. The benchmark root should continue to record its own partial summary files, but those files must point at a valid stage-level knowledge package rather than a half-empty worker tree. If fully graceful signal handling proves invasive, the minimum acceptable behavior is still: stop spawning new follow-ups, mark the stage interrupted, finalize the stage package, finalize the benchmark partial summary, and exit cleanly.

The acceptance is a deterministic interruption test that stops a knowledge-heavy benchmark or stage harness mid-phase and verifies both benchmark-level and stage-level artifacts exist and tell a coherent interrupted story.

### Milestone 8: Extend observability so future postmortems are one file read, not a manual census

After the runtime is correct, make it inspectable. This work likely touches `cookimport/llm/prompt_artifacts.py`, prompt-budget reporting helpers, and `cookimport/runs/stage_observability.py`.

Add compact knowledge-stage counters such as total packets, accepted packets, malformed packets, schema-invalid packets, semantic-invalid packets, packets repaired, packets retry-recovered, packets terminally failed, main workers completed, main workers watchdog-killed, malformed completions, follow-up calls started, follow-up calls completed, follow-up calls cancelled, stale follow-ups reaped, and off-task watchdog counts. The user should be able to open one stage summary and understand why the phase failed without counting `repair_status.json` files by hand.

This milestone should also improve prompt-budget and stage summary output so the knowledge phase can clearly distinguish work done by main workspace workers from work done by structured retries and repairs. The saved run made that distinction invisible at the high level, which obscured where time and token risk were really going.

The acceptance is a new or extended summary artifact that surfaces these counts for a saved-artifact or fake-runner regression test, plus short docs that describe the new meaning.

### Milestone 9: Lock the whole program down with deterministic replay tests and one optional live rerun

The last milestone is verification, not feature creep. Add or extend deterministic tests in the nearest relevant files, likely including:

- `tests/llm/test_codex_farm_knowledge_orchestrator.py`

## Validation and Acceptance

First, the worker contract is explicit and testable. A novice reading the written worker manifest and prompt can tell that `scratch/` is the only allowed scratch area, bounded local shell automation is allowed inside the worker root, task decisions must come from chunk-local evidence, and inline interpreters or `/tmp` helpers are forbidden by default.

Second, the main workspace-worker watchdog kills only on real boundary or liveness violations. A local shell loop under the worker root is tolerated. `/tmp`, parent traversal, network tools, package managers, and general interpreters are rejected. Predictable violations are prevented or cut off earlier than the current kill-after-minutes behavior.

Third, every task packet reaches exactly one explicit terminal outcome. There is no packet left implicitly unresolved because a worker died, a malformed file appeared, or a follow-up subprocess lingered.

Fourth, malformed, schema-invalid, semantically empty, and low-trust outputs are detected at first appearance rather than being counted as productive work simply because a file exists.

Fifth, recovery is bounded. A known-bad packet stops after the configured retry and repair budget, stale follow-up calls are cancelled or marked superseded once the parent packet or shard is terminal, and the stage cannot create another March 20-style retry storm.

Sixth, the benchmark UI and timeseries report packet truth. The equivalent of the saved Salt Fat phase displays packet counters such as `task 233/485`, not `task 0/10`, and worker rows explain shard ownership plus packet or follow-up state.

Seventh, the stage always finalizes its own artifact package on success, failure, and interruption. At minimum, the knowledge root contains `phase_manifest.json`, `telemetry.json`, `failures.json`, and `promotion_report.json` or an explicit no-promotion summary in all terminal conditions.

Eighth, future postmortems can be done from the top-level stage summary rather than a manual file census. The summary surfaces malformed-output counts, semantic failure counts, watchdog-kill counts, retry and repair counts, and cancellation counts.

## Idempotence and Recovery

This program is intentionally additive and should be implemented in small, coherent steps. Start by pinning the current failure shapes in tests and replay fixtures before changing runtime behavior. If the worker-contract rewrite becomes too broad, keep the explicit examples, `scratch/` support, and prompt/manifest clarifications, then defer any speculative prompt polish to a follow-up task.

If the prevention layer proves noisy, reduce it to one corrective intervention per worker session rather than removing it entirely. The product requirement remains to stop predictable waste before watchdog kill. If the packet-state machine lands before the UI polish, keep the new packet truth and stage summaries even if the renderer wording still needs adjustment; stage truth is the safer source of authority.

If live follow-up cancellation proves hard to implement everywhere, the fallback is still to mark child work superseded and stop waiting on it. The stage must never remain blocked on a child call once the parent packet or shard is terminal. Do not delete or rewrite the March 20 saved artifacts; they are the canonical evidence bundle for replay verification.
