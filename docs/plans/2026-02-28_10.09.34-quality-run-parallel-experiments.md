This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

After this change, `cookimport bench quality-run` can execute multiple experiments concurrently (bounded), instead of always running experiment rows one-at-a-time. It now defaults to CPU-aware auto experiment parallelism and adapts the active worker target based on host load, while preserving per-experiment reports and summary output contracts.

## Surprises & Discoveries

- Observation: Existing docs and logs explicitly describe experiment-level sequential execution as a deliberate contract.
  Evidence: `docs/02-cli/02-cli_README.md` quality-run section; `docs/07-bench/07-bench_log.md` entries around quality-suite decisions.

- Observation: Running experiments concurrently requires a thread-safe progress callback wrapper to avoid interleaved spinner/status writes from concurrent experiment workers.
  Evidence: parallel experiment execution now routes callback messages through a lock-protected wrapper in `run_quality_suite(...)`.

- Observation: A fully eager fanout is too aggressive for nested all-method workloads; adaptive admission should ramp up and clamp down quickly under load.
  Evidence: runner now computes desired worker target from load-per-CPU and ramps up gradually while allowing immediate downshift when load is hot.

- Observation: The active tournament run was still underutilizing host CPU because auto mode hard-capped experiment workers at 8 despite low load (`~0.12 load/cpu`) and 9 runnable experiments.
  Evidence: `data/golden/bench/quality/tournaments/2026-02-28_11.23.00_top_tier_tournament.log` (`workers_cap=8`, `worker target` stepping to `8/8` while load stayed low).

- Observation: Even after removing the experiment-cap bottleneck, host-wide CPU remained low in environments where process workers cannot initialize due `/dev/shm` permission failures; thread-backed experiment fanout remained GIL-limited.
  Evidence: repeated `PermissionError: [Errno 13]` process-pool fallback logs plus direct `/dev/shm` write probe failure in this runtime.

## Decision Log

- Decision: Use a thread pool for experiment-level parallelism rather than process-based fanout.
  Rationale: each experiment already invokes all-method internals that may use process/thread workers; adding another process layer increases nesting risk. A thread orchestrator gives bounded concurrency with lower integration risk.
  Date/Author: 2026-02-28 / Codex

- Decision: Preserve deterministic artifact row ordering by writing summary results in resolved experiment order, even if completion order differs.
  Rationale: avoids noisy diffs and keeps compare/report behavior stable.
  Date/Author: 2026-02-28 / Codex

- Decision: Default `--max-parallel-experiments` to auto (`None`) and interpret explicit values as fixed caps only.
  Rationale: users get faster throughput without manual tuning while still keeping an override for reproducibility and constrained hosts.
  Date/Author: 2026-02-28 / Codex

- Decision: Use `os.getloadavg()/cpu_count` as a lightweight host-pressure proxy for adaptive experiment admission.
  Rationale: avoids extra dependencies while giving enough signal to avoid over-driving hosts with background contention.
  Date/Author: 2026-02-28 / Codex

- Decision: Increase the default auto experiment ceiling to 16 and add `COOKIMPORT_QUALITY_AUTO_MAX_PARALLEL_EXPERIMENTS` for explicit tuning.
  Rationale: the prior fixed cap of 8 was leaving CPU headroom on high-core hosts; new ceiling plus load-aware admission keeps safety while allowing materially higher throughput.
  Date/Author: 2026-02-28 / Codex

- Decision: Auto-switch experiment fanout to subprocess workers when process-pool probing fails (with override via `COOKIMPORT_QUALITY_EXPERIMENT_EXECUTOR_MODE`).
  Rationale: subprocess fanout bypasses the in-process thread/GIL bottleneck in `/dev/shm`-restricted runtimes without changing per-experiment scoring logic.
  Date/Author: 2026-02-28 / Codex

## Outcomes & Retrospective

Implemented bounded parallel experiment execution for `bench quality-run` with no regression in summary/report contracts, then extended it with CPU-aware auto defaults and adaptive worker targeting.

What shipped:

- New CLI option `--max-parallel-experiments` (optional fixed cap; omitted means CPU-aware auto).
- Runner-level thread pool dispatch for experiments when `max_parallel_experiments > 1`.
- Deterministic summary ordering preserved by collecting results by original experiment index.
- Existing continue-on-failure semantics preserved under parallel execution.
- Auto mode records requested/effective/mode/cpu metadata in `experiments_resolved.json`.

- Run metadata now records experiment executor selection (`thread` vs `subprocess`) and reason in `experiments_resolved.json`.
- Auto mode adapts active worker target during runs based on load-per-CPU pressure.
- Auto mode now seeds its initial worker target from current load and ramps up in larger bounded steps to reach available capacity faster.
- New tests cover parallel execution boundary and CLI forwarding.
- Additional tests cover default auto ceiling and env-override ceiling behavior.
- Added test coverage for process-pool-unavailable -> subprocess executor fallback.
- Bench/CLI docs updated to describe the option and behavior.

## Context and Orientation

`quality-run` entrypoint is in `/home/mcnal/projects/recipeimport/cookimport/cli.py` (`bench_quality_run`). It delegates to `/home/mcnal/projects/recipeimport/cookimport/bench/quality_runner.py:run_quality_suite`.

`run_quality_suite` currently:

- resolves experiments from schema v1/v2 definitions,
- runs experiments sequentially by default but can dispatch them in a bounded thread pool,
- runs each experiment through `_run_single_experiment(...)`,
- writes `summary.json` + `report.md`.

Existing per-experiment all-method parallelism remains inside `_run_all_method_for_round(...)` and downstream `_run_all_method_benchmark_multi_source(...)`; this plan only changes scheduling across experiments.

## Plan of Work

Add a new quality-run parameter `max_parallel_experiments` and thread it from CLI to `run_quality_suite(...)`. Implement bounded parallel dispatch with `ThreadPoolExecutor`, making sure:

- each experiment keeps isolated output root (`experiments/<experiment_id>`),
- errors remain per-experiment and continue-on-failure behavior remains intact,
- results are reassembled in original experiment order before summary/report generation.

Then update tests in `tests/bench/test_quality_suite_runner.py` and `tests/bench/test_bench.py` for forwarding and parallel behavior. Finally update docs in `docs/02-cli/02-cli_README.md` and `docs/07-bench/07-bench_README.md`.
