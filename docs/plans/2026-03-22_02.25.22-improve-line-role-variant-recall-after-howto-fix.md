# Improve line-role `RECIPE_VARIANT` recall after the `HOWTO_SECTION` fix

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan builds directly on the completed HOWTO fix documented in `docs/tasks/2026-03-21_18.06.42-stop-line-role-variant-prose-from-becoming-howto-section.md`. The earlier change succeeded at removing false `HOWTO_SECTION` predictions, and this plan addresses the next concrete quality gap it exposed: too many true variant lines now fall back to non-variant labels.

## Purpose / Big Picture

After this change, a user rerunning the Salt Fat line-role benchmark should still see zero false `HOWTO_SECTION` predictions, but more of the real variant material should recover to `RECIPE_VARIANT` instead of collapsing to `OTHER`, `INGREDIENT_LINE`, `INSTRUCTION_LINE`, or `KNOWLEDGE`. The point is not merely to tag the lead sentence `To make Caprese Salad ...` correctly. The system should also understand that nearby ingredient, instruction, and storage lines can belong to one variant subsection when the cookbook is presenting an alternate version of the recipe rather than the main recipe body.

The visible proof is specific. In `data/golden/benchmark-vs-golden/2026-03-22_01.34.12`, `HOWTO_SECTION` is already at zero predictions, but `RECIPE_VARIANT` recall is only `0.3429` with `Gold 70 / Pred 38`. After the implementation described here, the next Salt Fat rerun should increase `RECIPE_VARIANT` recall materially, recover more of the variant-local runs around the slaw, tomato salad, and crouton sections, and keep `HOWTO_SECTION pred=0`.

## Surprises & Discoveries

- Observation: the HOWTO fix worked exactly on its target surface.
  Evidence: `data/golden/benchmark-vs-golden/2026-03-22_01.34.12/single-book-benchmark/saltfatacidheatcutdown/single_book_summary.md` shows `HOWTO_SECTION | Gold 0 | Pred 0`, and the codex line-role predictions now contain zero `HOWTO_SECTION` rows.

- Observation: variant recall is now the obvious remaining line-role weakness in this benchmark.
  Evidence: the same summary shows `RECIPE_VARIANT` precision `0.6316`, recall `0.3429`, `Gold 70`, `Pred 38`. That is far below the surrounding label quality and far below what the HOWTO fix made possible.

- Observation: the main misses are not limited to the lead `To make ...` line.
  Evidence: in the new codex predictions, many variant-local runs still break apart into mixed labels. Around atomic rows `1128` through `1135`, the slaw variation section is currently `OTHER`, `OTHER`, `OTHER`, `KNOWLEDGE`, `RECIPE_NOTES`, `OTHER`, `OTHER`, `OTHER`. Around rows `1188` through `1193`, the tomato-salad variation subsection is mostly `INGREDIENT_LINE`, `INSTRUCTION_LINE`, and `OTHER`. Around rows `1282` through `1285`, the crouton variation subsection is mostly `INSTRUCTION_LINE` and `OTHER`.

- Observation: the current positive variant seam is still too line-local.
  Evidence: `_classify_non_heading_howto_prose(...)` in `cookimport/parsing/canonical_line_roles.py` promotes only the individual long `To make ...` line to `RECIPE_VARIANT`. Adjacent ingredient and instruction lines are still free to fall back to their normal structural labels because there is no durable “this row lives inside a variant run” concept.

- Observation: the benchmark still contains some variant false positives, so recall cannot be improved by a blanket promotion rule.
  Evidence: `wrong_label_lines.jsonl` contains variant-related wrong rows where `pred_label=RECIPE_VARIANT` but `gold_label` is not variant. The plan must therefore grow a more specific variant-context signal rather than treating every recipe-adjacent alternate phrase as a variant.

- Observation: some benchmark variant sections are deliberately outside accepted recipe span.
  Evidence: the recovered variant lead lines such as rows `1194`, `1217`, `1307`, `1319`, and `1320` are currently labeled `RECIPE_VARIANT` with `within_recipe_span=False`. This means the fix cannot depend only on `within_recipe_span=True`; it must work for recipe-adjacent variant sections too.

- Observation: not every variant-looking heading should propagate variant context to the rows that follow it.
  Evidence: after the first implementation pass, `tests/parsing/test_canonical_line_roles.py::test_label_atomic_lines_omelet_variant_and_ingredient_rules` failed because `DINER-STYLE MUSHROOM, PEPPER, AND ONION OMELET` incorrectly pulled its ingredient and instruction rows into `RECIPE_VARIANT`. Narrowing continuation anchors to generic `Variations` headings and long `To make ...` leads fixed that regression.

- Observation: the generic heading `Variations` is too ambiguous to trust without local body evidence.
  Evidence: the new false-positive control in `tests/parsing/test_canonical_line_roles.py::test_label_atomic_lines_bare_variations_heading_without_variant_body_stays_other` only passes when the heading stays `OTHER` unless nearby rows actually look like alternate-version body content.

- Observation: note-style rows should not ride a variant run just because they immediately follow a real variant paragraph.
  Evidence: the 2026-03-22_03.33.50 benchmark recovered variant recall strongly, but rows such as `Refrigerate leftover dressing...`, `Ideal for romaine...`, and `Ideal for drizzling...` were newly overpromoted to `RECIPE_VARIANT`. Removing note-like rows from the continuation rule fixed that family without weakening the recovered `Variations` subsections.

## Context and Orientation

The line-role stage labels each atomic cookbook line with one structural class such as `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `RECIPE_TITLE`, or `RECIPE_VARIANT`. In this repository, `RECIPE_VARIANT` does not only mean a short heading like `Variation`. The benchmark also uses `RECIPE_VARIANT` for lines that belong to an alternate-version subsection: the lead “make this variation” sentence, the variation-specific ingredient lines, the variation-specific steps, and sometimes the variation-specific storage or service lines that stay inside that alternate-version run.

The active code is concentrated in `cookimport/parsing/canonical_line_roles.py`. The earlier HOWTO fix added `_classify_non_heading_howto_prose(...)`, which detects long `To make ...` prose and can promote that one line to `RECIPE_VARIANT`. That helper sits at the top of `_deterministic_label(...)`, so it now wins before the generic instruction and prose fallbacks. However, nothing in current code records that nearby rows belong to the same variant subsection. As soon as the parser moves to the next ingredient or instruction line, the row is judged on its own and often becomes `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `RECIPE_NOTES`, `KNOWLEDGE`, or `OTHER`.

The benchmark evidence comes from `data/golden/benchmark-vs-golden/2026-03-22_01.34.12/single-book-benchmark/saltfatacidheatcutdown/`. The headline metrics are in `single_book_summary.md`. The line-level misses are in `codexfarm/wrong_label_lines.jsonl`. The underlying codex predictions are in `codexfarm/line-role-pipeline/line_role_predictions.jsonl`.

The concrete miss families that this plan must address are visible in those line-role predictions:

- Slaw variation run around rows `1128` through `1135`: variant-local preparation, seasoning, storage, and alternate-version lines currently collapse mostly to `OTHER`, `KNOWLEDGE`, and `RECIPE_NOTES`.
- Tomato-salad variation run around rows `1188` through `1193`: variant-local ingredient lines and instructions currently stay `INGREDIENT_LINE` and `INSTRUCTION_LINE` instead of promoting to `RECIPE_VARIANT`.
- Crouton variation run around rows `1282` through `1285`: variant-local steps and notes currently stay `INSTRUCTION_LINE` and `OTHER`.

These runs demonstrate the real requirement: once the parser recognizes that one alternate-version subsection has started, it must be able to keep subsequent rows in that subsection inside variant context until a clear boundary ends the run.

The existing positive seams relevant to this work are:

- `_deterministic_label(...)` in `cookimport/parsing/canonical_line_roles.py` for first-pass label choice.
- `_classify_non_heading_howto_prose(...)` in the same file for current long-`To make ...` handling.
- `_looks_variant_heading_text(...)` in the same file for short header-like variant cues.
- `_outside_span_has_neighboring_component_structure(...)` and neighboring-context helpers in the same file, which already model some local structural evidence.

The active tests already cover the HOWTO fix, but they do not yet prove that a multi-line variant run can survive as `RECIPE_VARIANT`.

## Plan of Work

Start in `cookimport/parsing/canonical_line_roles.py`. Keep the existing HOWTO rejection logic intact, but add a parser-owned notion of variant context. The implementation should answer a new question for each row: “is this line part of an alternate-version run that has already been established nearby?” The answer should not depend only on the row’s own surface text. It should also depend on whether an immediately neighboring row strongly establishes a variant subsection, such as a `Variation` heading, a long `To make ...` variant lead line, or another row already judged variant by deterministic logic.

Use that new context in `_deterministic_label(...)`. The first promotion seam should still recognize direct variant cues on the current row, but it should then allow adjacent ingredient, instruction, note, and storage lines to inherit `RECIPE_VARIANT` when they are clearly still inside the same alternate-version run. The implementation should fail closed when the row instead looks like ordinary main-recipe structure, general lesson prose, or a new recipe title.

The context model should also have a clear stop condition. A variant run should end when the parser sees strong evidence that the subsection is over, such as a new recipe title, a lesson heading, an unrelated knowledge run, a yield/title boundary for a different recipe, or a sufficiently long break in local variant evidence. The goal is not fuzzy global propagation. The goal is a short, local alternate-version run detector.

After that deterministic seam is working, review sanitization. If Codex emits `RECIPE_VARIANT` for rows that are not inside a variant run and do not look variant-like themselves, `_sanitize_prediction(...)` should be able to demote those rows back to safer labels. That gives the system a way to improve recall without accepting every model-side variant upgrade blindly.

Then update the worker-facing contract only if the parser-owned policy has changed materially. The prompt does not need a broad rewrite. A short clarification is enough if the implementation introduces a concrete new idea such as “rows may stay `RECIPE_VARIANT` across a local alternate-version run after a variant lead-in.”

Expand `tests/parsing/test_canonical_line_roles.py` with benchmark-shaped regressions. Do not stop at one `To make ...` line. Add at least one multi-line variant block that includes a lead line plus following ingredient and instruction rows. Add another regression for a `Variations` heading followed by variant-local prose. Pair those with false-positive controls that must not become `RECIPE_VARIANT`, such as lesson headings, general how-to headings, or ordinary recipe-body steps that only happen to contain substitution language.

If the active contract changes, update the short parsing docs in `docs/04-parsing/04-parsing_readme.md` to say plainly that `RECIPE_VARIANT` can now cover local alternate-version runs rather than only isolated heading-like cues.

## Validation and Acceptance

The change is acceptable only if all of the following are true.

The next benchmark still shows zero false `HOWTO_SECTION` predictions. Improving variant recall must not restore the just-fixed HOWTO leak.

`RECIPE_VARIANT` recall improves measurably from the 2026-03-22 baseline of `0.3429`. The benchmark does not need perfection, but it should clearly recover more than the current `38` predicted variant rows against `70` gold rows.

At least one benchmark-shaped variant run is recovered across multiple lines. A lead `To make ...` line plus its following ingredient and instruction rows should stay `RECIPE_VARIANT` together when they clearly belong to one alternate-version subsection.

The parser does not simply relabel every alternate-sounding line as variant. A false-positive control such as a lesson heading, a general technique heading, or an ordinary recipe-body step must remain non-variant.

## Idempotence and Recovery

These edits are additive and safe to repeat. The deterministic tests can be rerun without mutating historical benchmark roots. The benchmark evidence under `data/golden/benchmark-vs-golden/2026-03-22_01.34.12` should remain read-only and serve as the fixed baseline for this plan.

If the first implementation increases `RECIPE_VARIANT` recall by relabeling too much ordinary recipe body, keep the variant-run idea but narrow the continuation rules. The safest recovery path is to require a stronger anchor row or a shorter propagation radius, not to delete the concept entirely.

If the first implementation recovers only the lead `To make ...` row and not the following variant-local lines, do not stop there. The benchmark evidence shows that the real gap is multi-line variant runs, so continue until at least one such run is recovered end to end.

If the variant recall fix unexpectedly reintroduces `HOWTO_SECTION` predictions, treat that as a hard regression. Restore the strict HOWTO gate first, then re-approach variant recall through its own dedicated positive path.
