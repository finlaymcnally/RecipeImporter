This file is plan 5 of 6 for the March 20 knowledge-stage runtime rebuild. It should follow the packet-ledger and recovery work in the earlier companion plans, because truthful progress and reliable finalization both depend on packet truth already being authoritative.

## Purpose / Big Picture

The operator-facing runtime surface must tell the truth while the stage is running and must leave a coherent explanation after the stage stops. After this change, the knowledge phase should report packet totals rather than shard totals, reduce main-worker blast radius through smaller bundles or worker rotation, stop spawning work promptly on interruption, and always write a coherent stage package plus benchmark partial summary under every terminal condition.

This plan is where the system stops being locally correct but operationally confusing. The packet ledger and bounded recovery logic from earlier plans will not help much if the terminal UI still says `task 0/10` during packet work or if an interrupted run still leaves the operator with raw worker debris instead of a stage-level explanation.

The visible proof is that the March 20 shape `task 0/10 | running 10` becomes impossible once packet work is happening, and an interrupted run still leaves a readable stage-level explanation instead of only raw worker debris.

## Outcomes & Retrospective

This plan is not implemented yet. Its intended outcome is that the stage becomes operator-truthful during runtime and readable after runtime. If it succeeds, a solo operator should be able to look at the progress surface and the final stage package and answer the basic questions immediately: what was happening, what finished, what failed, and what remains worth investigating.

The main open risk is split responsibility across multiple surfaces. Progress payloads come from the orchestrator, rendering comes from CLI helpers, and finalization touches benchmark code and observability helpers. This plan therefore needs to be explicit about which surface is the truth source. The orchestrator should emit truthful packet-based state, and everything else should render or summarize that truth.

## Context and Orientation

Progress rendering currently flows through `cookimport/llm/codex_farm_knowledge_orchestrator.py`, `cookimport/core/progress_messages.py`, and `cookimport/cli.py`. Finalization and interrupted benchmark summaries touch `cookimport/labelstudio/ingest.py` and `cookimport/runs/stage_observability.py`.

The March 20 run showed two coupled failures:

- shard-oriented progress lied about the real work unit because the stage had split into packets but the UI still reported shard totals;
- interrupted finalization left the operator with evidence but not a usable explanation because stage-level summaries were missing or incomplete.

This plan also owns `blast radius` reduction. In this context, blast radius means how many packets one long-lived main worker can contaminate before the runtime rotates or otherwise limits that worker’s ownership. Packet leasing narrows the current task, but session rotation or smaller worker bundles can still add useful safety.

## Plan of Work

### Milestone 1: Reduce main-worker blast radius

Add either smaller packet bundles per main worker or worker rotation after a configured number of accepted packet results. Rotation is preferred if it can be done without losing the benefits of packet leasing, because it limits drift accumulation inside a long session while keeping the worker contract consistent.

The point is not throughput micro-optimization. The point is to ensure that one bad session cannot quietly contaminate a large portion of the stage before the runtime reacts. This milestone should also make the effective worker bundle size visible in runtime metadata so tests and operators can confirm the policy is real.

### Milestone 2: Rebuild progress payloads around packet truth

Once packet truth is authoritative, rebuild the structured progress payload for the knowledge stage. `task_current` and `task_total` must mean packet totals, not shard totals. Worker rows should show shard ownership plus current packet or follow-up truth. Detail fields should include:

- completed shards;
- active main workers;
- retry counts;
- repair counts;
- canceled follow-ups;
- terminal packet totals by category.

Terminal failed packets must still advance visible progress. A packet that is done failing is still done.

### Milestone 3: Keep renderer changes small but honest

The main fix should be in the emitted knowledge payload, not a CLI-only special case. Update CLI rendering only as needed to make packet-based payloads legible. The benchmark timeseries and the terminal UI should tell the same story. A saved run like March 20 should no longer be renderable as `task 0/10 | running 10` once packet work is happening.

### Milestone 4: Rebuild interruption and finalization

On interruption, stop spawning new work, cancel active children, finalize the stage package, then finalize the benchmark partial summary. Always write `phase_manifest.json`, `promotion_report.json`, `telemetry.json`, `failures.json`, and `knowledge_manifest.json` or an explicit interrupted equivalent.

Stage-level finalization is the primary truth surface here. The benchmark partial summary should point at that stage package rather than acting like an independent parallel explanation.

### Milestone 5: Preserve validated packet truth alongside transport anomalies

If a worker exits oddly after all or most accepted packet work is already settled, keep the accepted packet truth and record the transport anomaly as telemetry or as a distinct worker outcome category. Do not erase validated work merely because the subprocess ended in an unexpected way after useful work was already completed.

## Validation and Acceptance

This plan is complete when:

- the knowledge phase reports packet totals rather than shard totals at the top level;
- worker rows and detail fields expose follow-up truth instead of hiding it;
- interrupted runs stop spawning new work promptly;
- stage-level and benchmark-level summary artifacts are always written under terminal conditions;
- accepted packet truth survives late transport anomalies as a reported category, not as lost work.

The best acceptance shape is a deterministic test that simulates mixed main-worker and follow-up activity and still renders a human-readable packet-based progress surface, plus an interruption test that asserts the presence of a coherent stage package and benchmark summary.

## Idempotence and Recovery

If worker rotation lands after packet-based progress, keep the progress path stable and extend it rather than rewriting it again. Finalization paths should always stay additive; do not replace one partial-artifact path with another ad hoc one. Re-running finalization after an interruption should update or preserve the same stage package rather than scattering multiple competing partial summaries.
