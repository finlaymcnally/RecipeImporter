# Add Line-Role Boundary Neighbor Context

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the live file-backed line-role worker payload will finally include the reference-only neighbor rows that the prompt contract and parsing docs already describe. A human inspecting `data/output/.../line-role-pipeline/runtime/line_role/workers/*/in/*.json` after a future run should see `context_before_rows` and `context_after_rows` when a shard has real adjacent rows in the book, instead of seeing only shard guidance plus owned `rows`.

The user-visible outcome is not a new label policy. The line-role stage should still be the pre-grouping recipe-boundary stage, so it will still often lack authoritative recipe-span truth. The point of this plan is narrower and cleaner: when the model is asked to review a shard, it should at least be able to see the immediate line just before the shard and the immediate line just after the shard when those rows exist. That gives the worker the missing local boundary evidence that the current live payload omits.

You can see the result working without running live Codex-backed processing. Focused unit tests should build shard plans from synthetic `AtomicLineCandidate` rows and prove that interior shards carry one reference-only row before and after, while the first shard omits `context_before_rows` and the last shard omits `context_after_rows`. A saved artifact or synthetic payload inspection should show the new keys in `manifest_entry.input_payload` and the debug payload.

## Outcomes & Retrospective

This plan is not implemented yet. The intended outcome is that the live file-backed line-role worker contract finally matches the contract already described in current docs and prompt text. After implementation, the stage will still be conservative and pre-grouping, but it will no longer force the worker to infer boundary evidence from a large shard with no explicit neighbor rows.

The pre-plan lesson is simple: the current problem is not that the repo forgot to invent a context mechanism. The mechanism already exists in docs, prompt text, and helper imports. The bug is that the live payload builder never actually populates it.

## Context and Orientation

The line-role stage is the LLM-backed correction pass that reviews deterministic labels for one ordered slice of atomic cookbook lines before those labels are grouped into recipe spans. In this repository, an “atomic line” is one small text unit represented by `AtomicLineCandidate` in `cookimport/parsing/recipe_block_atomizer.py`. Each candidate has an `atomic_index`, a `block_index`, text, and a `within_recipe_span` field. At this stage, `within_recipe_span` is often intentionally `None` because recipe ownership has not been finalized yet.

The live worker payload is assembled in two steps. `cookimport/parsing/canonical_line_roles/planning.py` partitions the ordered candidate list into contiguous shards in `_build_line_role_canonical_plans(...)`. For each shard, it calls two builder functions in `cookimport/parsing/canonical_line_roles/policy.py`:

- `build_line_role_debug_input_payload(...)` builds the richer local debug copy used for artifact inspection.
- `build_line_role_model_input_payload(...)` builds the authoritative worker input stored under `line-role-pipeline/runtime/line_role/workers/*/in/*.json`.

Those payloads currently carry shard guidance such as `shard_mode`, `context_confidence`, `default_posture`, and `flip_policy`, followed by compact owned `rows`. The docs and prompt contract already say that the file-backed transport can also include `context_before_rows` and `context_after_rows`, which are reference-only rows immediately outside the shard. The current implementation does not populate those keys, so the worker sees a large shard and a conservative prompt, but not the immediate predecessor or successor line that could disambiguate shard boundaries.

The helper functions that should power this fix already exist in `cookimport/parsing/recipe_block_atomizer.py` and are already imported into the canonical line-role package through `cookimport/parsing/canonical_line_roles/__init__.py`:

- `build_atomic_index_lookup(candidates)` builds a lookup from `atomic_index` to candidate.
- `get_atomic_line_neighbor_texts(candidate, by_atomic_index=...)` returns the immediate previous and next line texts for a given candidate.

The prompt-input serializers also already exist in `cookimport/parsing/canonical_line_roles/prompt_inputs.py`:

- `serialize_line_role_model_context_row(...)` produces the compact `[atomic_index, current_line]` form used by the file-backed prompt contract.
- `serialize_line_role_debug_context_row(...)` produces the object form used by the richer debug payload.

There is one important constraint. `docs/04-parsing/04-parsing_readme.md` says pre-grouping line-role candidates no longer inherit importer recipe provenance. That means this plan must not try to populate or infer stronger `within_recipe_span` truth. The worker will still often see `within_recipe_span=None` in debug surfaces and “unknown” span counts in shard hints. This plan only adds immediate neighbor rows so the worker can judge shard boundaries with the best local evidence that is safely available at this stage.

Finally, tests already cover this area. `tests/parsing/test_canonical_line_roles.py` contains a focused shard-plan test that currently asserts `context_before_rows` and `context_after_rows` are absent. That test must be updated to assert the opposite behavior after the fix. Cache behavior also matters because line-role results are memoized under a versioned cache path. The implementation must bump `_LINE_ROLE_CACHE_SCHEMA_VERSION` in `cookimport/parsing/canonical_line_roles/__init__.py` or provide an equivalent invalidation mechanism.

## Plan of Work

Start in `cookimport/parsing/canonical_line_roles/policy.py`. Introduce one small helper whose only job is to compute boundary neighbor rows for a shard. It should accept the current shard’s `candidates`, plus a whole-book `by_atomic_index` lookup. The helper should look at the first owned candidate and, if there is an immediately preceding candidate with `atomic_index - 1`, serialize it as a single entry in `context_before_rows`. It should then look at the last owned candidate and, if there is an immediately following candidate with `atomic_index + 1`, serialize it as a single entry in `context_after_rows`. Use the existing serializer helpers instead of inventing new ad hoc shapes.

Then change both `build_line_role_debug_input_payload(...)` and `build_line_role_model_input_payload(...)` so they accept the whole-book lookup and include these context arrays when present. Keep the existing payload shapes otherwise unchanged. Do not add per-row operator metadata back into the model payload, and do not change `rows` ordering or label-code serialization. If a shard sits at the very start or end of the book, omit the missing side instead of inserting empty placeholder rows.

Next, edit `cookimport/parsing/canonical_line_roles/planning.py`. Build the whole-book lookup once from `ordered_candidates` before partitioning. Pass that lookup into both payload builders for each shard. Do not change shard partitioning, prompt-target logic, or worker-count logic in this plan. The point is to enrich each existing shard payload, not to repartition the stage.

After the payload builders are wired, update tests in `tests/parsing/test_canonical_line_roles.py`. Replace the current assertions that context rows are absent with precise assertions about boundary behavior. One focused test should build a single shard at the start of a short candidate list and assert there is no `context_before_rows` but there is a `context_after_rows` entry when a next line exists. Another focused test should build a middle shard from a longer candidate list and assert both context arrays are present and contain the expected immediate neighbor `atomic_index` values and texts. Keep one prompt-rendering test that shows the file prompt still describes these fields correctly.

Then handle cache invalidation. Update `_LINE_ROLE_CACHE_SCHEMA_VERSION` in `cookimport/parsing/canonical_line_roles/__init__.py` so new runs cannot silently reuse no-context line-role cache entries. If any focused cache test relies on the old value indirectly, update the test so it verifies the intended behavior rather than the exact previous version string.

Finish by updating `docs/04-parsing/04-parsing_readme.md`. The current docs already say the transport includes `context_before_rows` and `context_after_rows`; after implementation, add one short clarifying sentence that the live file-backed worker path now writes one immediate out-of-shard neighbor row on each side when available. Keep the note short and specific so reviewers can understand the behavioral change from the docs alone.

## Validation and Acceptance

Acceptance requires all of the following:

1. `build_line_role_model_input_payload(...)` writes `context_before_rows` and `context_after_rows` when the shard has immediate out-of-shard neighbors, using the compact `[atomic_index, current_line]` transport shape already documented.
2. `build_line_role_debug_input_payload(...)` mirrors the same boundary context so the debug artifact and worker payload tell the same story.
3. The first shard of a synthetic ordered candidate list omits `context_before_rows` and keeps `context_after_rows` when a next row exists. The last shard does the opposite. An interior shard carries both.
4. Existing focused tests in `tests/parsing/test_canonical_line_roles.py` no longer assert the old broken absence behavior and instead lock in the new boundary-context behavior.
5. `_LINE_ROLE_CACHE_SCHEMA_VERSION` changes so cached no-context results are not reused after the fix.
6. `docs/04-parsing/04-parsing_readme.md` matches the now-true runtime behavior and explains the boundary-neighbor emission briefly and accurately.

The human-level behavior to verify is simple: if you inspect a future worker input JSON under `line-role-pipeline/runtime/line_role/workers/*/in/*.json`, you should see explicit reference-only neighbor rows around the shard boundary when they exist. A novice should not have to infer that the prompt supports context from docs alone; the payload should actually contain it.

## Idempotence and Recovery

This plan is a prompt-payload and test-contract fix. It does not migrate user data and does not require destructive operations. Re-running the focused tests is safe. Rebuilding payloads is safe because the files are derived artifacts.

The only risky seam is cache reuse. If the implementation is applied without cache invalidation, stale no-context results may still be replayed. The recovery path is explicit: bump `_LINE_ROLE_CACHE_SCHEMA_VERSION`, delete any local line-role cache only if needed for manual inspection, and re-run the focused tests. No live Codex-backed processing is required to prove correctness.

If a first attempt produces the wrong context shape, revert only the touched payload-builder and test edits, keep the new failing test in place, and retry with the existing serializer helpers rather than inventing a new transport shape.
