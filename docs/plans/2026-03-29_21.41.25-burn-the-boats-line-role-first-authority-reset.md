## Exact User Desire

the original problem we are solving is that the current line-role codex path is not actually improving classification. it is behaving like a deterministic-answer review loop, leaking deterministic labels into the worker surface, and then effectively rubber-stamping those labels instead of doing real semantic work. in the observed bad run, codex line-role changed zero seed labels across the entire book, so we paid for the llm but did not get materially better line-role labeling.

the execplan must make line-role a clean first-authority semantic classifier for line labels rather than a deterministic-answer review loop. The line-role worker should receive ordered line text, nearby context, and the label contract, and then make the label decision itself. It should not be judged against deterministic labels during runtime, it should not use the golden set as part of runtime acceptance, and it should not be nudged back toward weak deterministic semantics. 

this plan starts from the simpler and cleaner model: raw line text plus local context. This is burn-the-boats work. There is no compatibility layer, no legacy path preserved in the live code, and no half-measure where the old deterministic-semantic loop still controls the new line-role runtime.

## Purpose / Big Picture

After this change, a user can point RecipeImport at an arbitrary cookbook, including one with no gold set, and trust that the line-role stage is doing real first-pass semantic labeling rather than rubber-stamping deterministic hints. The worker will read the lines of the book, apply the route-first label family that exists after the prerequisite plan, and produce the authoritative line-role result directly. Deterministic code will still do important work, but only in the places it is good at: packaging raw evidence, validating ownership and output shape, enforcing allowed-label rules, and surfacing telemetry. It will stop acting like a semantic co-pilot for line-role.

The user-visible proof is that line-role prompt artifacts and runtime workspaces no longer expose deterministic labels, no longer pre-seed the work ledger with final labels, and no longer describe the job as comparing Codex output to baseline answers. The final proof is that the repository can run the line-role stage on books with no gold set and no baseline-comparison loop, and still produce valid, testable, route-first outputs that downstream recipe-span grouping and knowledge routing can consume normally.

## Context and Orientation

This repository has a label-first parsing path. The relevant entry point is [cookimport/parsing/label_source_of_truth.py](/home/mcnal/projects/recipeimport/cookimport/parsing/label_source_of_truth.py). It takes ordered atomic lines from a source book, runs line-role labeling, groups accepted recipe spans, and then lets downstream staging and optional knowledge review project those results into outputs and benchmark artifacts. In plain English, line-role happens early and its results strongly influence the rest of the book understanding pipeline.

The line-role worker implementation lives under [cookimport/parsing/canonical_line_roles/](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles). In this repository, a “shard” means one contiguous slice of owned input rows that a worker must label, validate, and install. The important modules are:

- [cookimport/parsing/canonical_line_roles/__init__.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles/__init__.py), which currently wires together deterministic baseline construction and live line-role execution.
- [cookimport/parsing/canonical_line_roles/planning.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles/planning.py), which partitions line-role rows into worker-owned shards.
- [cookimport/parsing/canonical_line_roles/prompt_inputs.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles/prompt_inputs.py), which defines the row payloads sent into the line-role worker surface.
- [cookimport/parsing/line_role_workspace_tools.py](/home/mcnal/projects/recipeimport/cookimport/parsing/line_role_workspace_tools.py), which currently writes seeded work ledgers, validates output payloads, and manages the worker helper script.
- [cookimport/llm/canonical_line_role_prompt.py](/home/mcnal/projects/recipeimport/cookimport/llm/canonical_line_role_prompt.py), which defines what the worker is told the task means.

After the prerequisite routing-reset plan lands, the line-role label family should already contain the recipe-local labels plus the route-first non-recipe labels. This plan assumes that is true. The remaining problem is authority. Today the Codex path still feels like “deterministic labels first, LLM review second.” The goal here is to make it “LLM semantics first, deterministic validation second.”

In this plan, “first-authority” means the worker’s accepted label is the semantic truth for line-role. Deterministic code may reject malformed output, missing rows, illegal labels, or impossible field combinations, but it may not silently rewrite a valid worker label to match a heuristic baseline. “Gold” means human-labeled benchmark data under `data/golden/`; it is useful only for offline evaluation and must not be part of runtime acceptance. “Deterministic baseline” means the current rule/fallback label generator that previously leaked label answers into the worker surface and was later compared against the worker’s result.

The current prerequisite plan already handles the meaning of non-recipe routing labels. This plan deliberately does not reopen that design. Instead, it changes how line-role reaches those labels and how its outputs are accepted, validated, and described.

## Plan of Work

This work starts by deleting the notion that the live line-role worker needs to see or compare itself against deterministic semantic answers. In [cookimport/parsing/canonical_line_roles/prompt_inputs.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles/prompt_inputs.py), remove deterministic labels, rule-tag-derived semantic cues, and escalation metadata from the first-read model-facing transport. The model-facing `in/*.json` contract should become a raw row-and-context transport owned by line-role itself. Each owned row should carry only the stable row identifier and raw text. The neighboring context rows should remain available as read-only context, but they should also be raw text only. If the worker needs a label glossary, that glossary belongs in the prompt and helper contract, not as per-row answer hints.

At the same time, rewrite the workspace contract in [cookimport/parsing/line_role_workspace_tools.py](/home/mcnal/projects/recipeimport/cookimport/parsing/line_role_workspace_tools.py). Delete the seed-ledger builders from the live path. A worker should not open a `work/<shard_id>.json` file that already contains the semantic answer. The workspace should instead contain either no work file until the worker creates it, or a scaffold that contains only owned `atomic_index` rows with blank or absent labels. The validator should accept an incomplete scaffold in the editable `work/` file as an intermediate state but must still require a fully populated valid label ledger before `install-phase` succeeds. The installed `out/<shard_id>.json` file must remain strict and complete.

Next, simplify the prompt and helper story. In [cookimport/llm/canonical_line_role_prompt.py](/home/mcnal/projects/recipeimport/cookimport/llm/canonical_line_role_prompt.py) and the line-role helper files that write `CURRENT_PHASE.md`, remove language that frames deterministic labels as hints, seed codes as weak guides, or baseline agreement as desirable behavior. Rewrite the prompt so it plainly says: here are the owned rows, here is the label family, here is the nearby context, make the best label decision you can for each owned row. Because the prerequisite routing-reset plan will already have established the live non-recipe route labels, this prompt rewrite should explain those route labels in plain English and stop talking about final `KNOWLEDGE` versus `OTHER` as if line-role owns them.

Then change the runtime authority plumbing in [cookimport/parsing/canonical_line_roles/__init__.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles/__init__.py) and [cookimport/parsing/label_source_of_truth.py](/home/mcnal/projects/recipeimport/cookimport/parsing/label_source_of_truth.py). Today the live path still constructs both baseline and final predictions, then folds both into authoritative labeled lines. Replace that story with one live-authority output. On the Codex line-role path, the authoritative prediction set should come from accepted worker output only. Deterministic labeling may still exist as an offline benchmark/control seam or as a separate explicit pipeline mode if the broader product still wants it, but it must not shape the live Codex path. In `AuthoritativeLabeledLine` and related block-label assembly, keep fields that are useful for debugging and later stages, but remove any implication that runtime acceptance requires semantic comparison against a deterministic answer.

After that, remove semantic baseline comparison from the live acceptance and repair logic. In this repository, “repair” should mean “fill in missing rows, resolve malformed output shape, or satisfy strict field constraints,” not “justify why you disagree with the heuristic baseline.” Update validation and same-session repair so the worker is judged on output completeness, allowed labels, route-label invariants from the prerequisite plan, and field-specific rules such as non-recipe exclusion reason validity. It should not be judged on closeness to deterministic or gold. If the worker emits an allowed label that later looks wrong in benchmarks, that is a benchmark issue or prompt-quality issue, not a runtime rejection reason.

The ingredient-parser boundary should be made explicit while this rewrite is in progress. In docs and code comments where needed, state that line-role v2 is text-and-context labeling only. Ingredient parsing remains in the recipe-stage conversion path in [cookimport/staging/draft_v1.py](/home/mcnal/projects/recipeimport/cookimport/staging/draft_v1.py). Do not add ingredient parser outputs to line-role shard inputs in this plan. If later work proves that line-role needs evidence-only sidecars such as parsed quantity/unit structure, that should be a later plan with its own benchmark evidence and should still avoid answer leakage. This plan intentionally keeps line-role simple.

Once the runtime contract is rewritten, rework the artifacts and telemetry so they tell the new story. Prompt artifacts should show raw input rows and accepted line-role outputs, not seed-vs-final comparisons. Runtime telemetry should describe worker coverage, validation status, repair state, and output completeness, not semantic agreement with deterministic labels. Benchmark-side comparisons may still compare one run to another or a run to gold, but that logic must sit entirely outside the line-role runtime authority path. The main repository rule after this plan lands is simple: benchmark code may compare outputs; live line-role runtime may not.

Finally, update tests, docs, and naming to remove the old worldview completely. Because this is burn-the-boats work, delete or rewrite tests that enshrine deterministic label exposure, seed-ledger behavior, or runtime baseline comparison. Update the parsing, LLM, and Label Studio docs to say that line-role is now first-authority semantics over raw line/context shards. Rename the live line-role pipeline id and any contract-version labels so that historical artifacts remain distinguishable from the new world, but do not keep compatibility normalization branches for the older names in live code.

## Milestones

### Milestone 1: Make the worker input and workspace truly blind to deterministic semantics

At the end of this milestone, the live line-role worker sees only raw rows, neighboring context, stable row ids, and the label contract. The worker does not see deterministic labels in `in/*.json`, in first-read hints, or in prefilled `work/*.json` files. A parsing test should prove that model-facing shard input omits deterministic labels and that a workspace scaffold can exist without semantic labels already filled in.

### Milestone 2: Rewrite the prompt and helper contract for first-authority labeling

At the end of this milestone, line-role prompt text and helper surfaces describe one job only: label each owned row directly using text and context. The prompt explains the route-first label family that now exists after the prerequisite plan, explains recipe-local labels in plain language, and does not mention deterministic-baseline agreement. A prompt-artifact test should prove that no worker-first file mentions deterministic label hints or seed output.

### Milestone 3: Remove deterministic semantic comparison from runtime authority

At the end of this milestone, the live Codex line-role path no longer constructs or consumes a deterministic semantic baseline as part of acceptance. Accepted worker output becomes the authoritative line-role result directly. Deterministic code may still enforce mechanical invariants such as ownership, output completeness, allowed labels, and field validity. A parsing/runtime test should prove that a valid but baseline-disagreeing worker output is accepted on the live line-role path.

### Milestone 4: Keep line-role simple and text-first by making the ingredient-parser boundary explicit

At the end of this milestone, line-role still does not consume ingredient parser outputs. The docs, prompt artifacts, and tests all reflect the simpler model: line-role uses raw row text and local context only. Recipe drafting continues to use ingredient parsing downstream in [cookimport/staging/draft_v1.py](/home/mcnal/projects/recipeimport/cookimport/staging/draft_v1.py). A regression test should prove that line-role shard inputs do not depend on ingredient-parser settings or parse outputs.

### Milestone 5: Rewire telemetry, artifacts, and docs to the new runtime story

At the end of this milestone, prompt artifacts, runtime artifacts, benchmark helpers, and docs no longer present line-role as a deterministic-audit path. They show a direct first-authority worker result, validation outcomes, and downstream projection results. Offline benchmark code may still compare run outputs to gold or to another run, but the live runtime must not. A Label Studio artifact test should prove that prompt artifacts and runtime telemetry no longer refer to seeded answers or runtime baseline agreement.

### Milestone 6: Delete the old path completely and prove the new one on real tests

At the end of this milestone, the old line-role seed path and its compatibility vocabulary are gone from live code. The targeted parsing and Label Studio suites are green, the project fast suite is green, and the docs are updated. If the user later explicitly approves a CodexFarm benchmark run, that run should show a model-facing line-role surface with no answer leakage. Such a benchmark is not part of this plan’s required implementation loop because `AGENTS.md` forbids CodexFarm-enabled benchmark processing without explicit approval.

## Validation and Acceptance

Acceptance is behavioral.

The first acceptance criterion is worker blindness. Add or update tests that open the live model-facing line-role shard payload and confirm it contains only raw rows, neighboring context, and stable ids. It must not contain deterministic label codes, `deterministic_label`, precomputed semantic rule outcomes, or seeded final-label ledgers. This criterion is the main guardrail against repeating the zero-delta rubber-stamp failure.

The second acceptance criterion is first-authority runtime acceptance. Create or update a line-role runtime test where accepted worker output differs from what the old deterministic baseline would have said. The worker output should still install successfully as long as it satisfies ownership, allowed-label, and field-validity constraints. This proves the new runtime no longer treats deterministic semantics as judge and jury.

The third acceptance criterion is no gold-aware runtime logic. The line-role runtime path should be runnable on arbitrary synthetic input with no gold files anywhere nearby. Tests should prove that the runtime authority path never reaches into benchmark or gold-set helpers when accepting worker output.

The fourth acceptance criterion is ingredient-parser separation. Add or update a regression test that toggles ingredient-parser settings while keeping the same line-role shard input, and confirm that the live line-role shard payload does not change because of ingredient-parser configuration. This proves that line-role remains text-and-context first in this plan.

The fifth acceptance criterion is artifact clarity. Prompt artifacts, runtime artifacts, and Label Studio projection artifacts must describe the line-role result as a direct first-authority worker output, not as a seeded correction pass. A test should fail if prompt-artifact files or helper-generated summaries mention seeded label codes, runtime baseline agreement, or similar old-story wording.

The sixth acceptance criterion is documentation cleanup. The parsing and LLM docs must explain the new line-role contract in plain language. A novice reading the updated docs should understand that line-role labels rows directly, deterministic code validates structure and authority boundaries, and gold remains an offline benchmark concern only.

## Idempotence and Recovery

This work should be implemented as one coherent contract rewrite after the prerequisite routing-reset plan is complete. The safe order is the milestone order given above: blind the worker surface first, then rewrite prompt/helper semantics, then remove runtime baseline comparison, then clean up artifacts and docs. Each step is safe to rerun because it is additive until the old path is deleted.

If the codebase enters a mixed state where some producer still writes deterministic seed fields and some consumer expects the new blind contract, do not add a compatibility bridge. Continue the rewrite until every producer and consumer speaks the new first-authority contract. Because this is burn-the-boats work, a partial mixed state is a signal to finish the cutover, not to support both worlds.

If targeted tests fail during the transition, rerun them after fixing the missing producer/consumer pair. The commands in `Concrete Steps` are safe to rerun. If the local `.venv` gets stale, reactivate it and rerun `pip install -e .[dev]`.

Offline historical artifacts under `data/golden/` do not need runtime compatibility. They may continue to exist for inspection, but the live line-role runtime should not attempt to interpret both the old answer-leaking contract and the new first-authority contract.
