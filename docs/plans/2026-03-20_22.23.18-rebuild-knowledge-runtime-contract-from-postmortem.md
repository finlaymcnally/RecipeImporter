## Purpose / Big Picture

After this change, the non-recipe knowledge phase should behave like a trustworthy production stage instead of a pile of loosely coupled heuristics. A benchmark or stage run that reaches knowledge review should do four user-visible things reliably: workers should stay on-contract and produce grounded task results, the runner should fail closed instead of silently churning through retries, the UI should show true task-level progress and follow-up state, and a stopped or failed run should still leave a complete stage-level postmortem package rather than only raw worker debris.

The concrete proof is straightforward. Run a no-token rehearsal of the Salt Fat Acid Heat knowledge phase using the existing fake-runner and saved-artifact harnesses. The stage should report packet progress instead of `task 0/10`, workers should either finish with validated or explicitly failed task outcomes, stale follow-up calls should not survive past shard completion, and the stage root should always contain `phase_manifest.json`, `promotion_report.json`, `failures.json`, `telemetry.json`, and proposal or failure artifacts even when the run is interrupted. With explicit user approval for token spend, a live Salt Fat single-offline CodexFarm rerun should then either complete or fail closed without the earlier retry storm.

## Surprises & Discoveries

- Observation: the knowledge phase did substantial work before appearing "stuck," but the stage never converged because raw worker-local files are not the same thing as accepted stage outputs.
  Evidence: the saved Salt Fat run wrote `339` top-level worker `out/*.json` files across `485` task packets, yet the stage root lacked `phase_manifest.json`, `promotion_report.json`, `telemetry.json`, `failures.json`, and `proposals/*`.

- Observation: the main workspace-worker contract is internally contradictory. Repo docs and manifests say workspace-local shell use is allowed, but several workers were killed for exactly that style of local automation.
  Evidence: the live status files for workers `003`, `005`, `006`, and `008` show `watchdog_command_execution_forbidden` on heredoc shell scripts, `jq` loops, and `python - <<'PY'` snippets in the bounded workspace.

- Observation: completed worker status is not a reliable success signal.
  Evidence: workers `001` and `002` finished with `reason_code=process_exited_without_watchdog_intervention` and `final_agent_message_state=malformed`, while workers `004`, `007`, `009`, and `010` finished with `workspace_outputs_stabilized`.

- Observation: a worker can appear "complete" while still leaving unusable artifacts that trigger massive repair churn.
  Evidence: worker `001` wrote `49/49` task files, but each file was invalid JSON because the heredoc terminator `EOF` was written into the file body, forcing `38` repair attempts.

- Observation: the recovery layer is currently low-yield and self-defeating.
  Evidence: in the saved run, `115` repair attempts produced only `5` repaired outcomes, while `75` watchdog-retry attempts produced only `1` validated outcome.

- Observation: follow-up processes can outlive the useful work they were meant to recover.
  Evidence: worker `007` had already completed with `workspace_outputs_stabilized`, but a task-level repair follow-up still sat in `running` state for roughly eight minutes with no new events before the benchmark was killed.

- Observation: the CLI and benchmark timeseries are only as truthful as the structured progress payload emitted by the orchestrator.
  Evidence: the knowledge phase wrote `task_manifest.jsonl` with `485` packets, but the UI remained at `task 0/10 | running 10` because the orchestrator still reported shard counts, not packet counts.

- Observation: the benchmark interruption path preserves raw artifacts but still leaves the stage without an authoritative stage-level failure record.
  Evidence: the earlier March 20 partial-finalization work covers benchmark roots in `cookimport/cli.py`, but the saved knowledge stage root itself still lacked its own manifest and failure summaries after interruption.

- Observation: "workers off task" is not only a watchdog problem. Some workers stayed within the workspace boundary but still bulk-generated templated results from metadata rather than doing grounded review.
  Evidence: worker `007` planned to mark every chunk useful and every block `keep/knowledge` using one `jq` loop over all tasks, while worker `004` explicitly reasoned from lane metadata instead of chunk-local evidence in `events.jsonl`.

  Rationale: the main workers do need local workspace access, while follow-up calls are supposed to emit direct strict JSON; collapsing them into one policy created the current confusion.

## Outcomes & Retrospective

This plan is not implemented yet. The current outcome is a consolidated program of work that replaces three narrower plans with one end-to-end knowledge-runtime plan. The main new insight is that the failures are not isolated bugs. They come from one broken contract spread across worker prompts, workspace policy, task validation, retry escalation, progress reporting, and finalization. The implementation should therefore be staged, but the final behavior must be evaluated as one coherent system.

## Context and Orientation

The non-recipe knowledge phase is the optional Codex-backed stage that reviews outside-recipe spans and decides whether each chunk is general cooking knowledge or just other non-recipe text. In this repository, the phase lives primarily in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, with validation and ingest logic in `cookimport/llm/codex_farm_knowledge_ingest.py`, prompt assembly in `cookimport/llm/knowledge_prompt_builder.py`, and prompt/schema assets in `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md` and `llm_pipelines/schemas/recipe.knowledge.v1.output.schema.json`.

The runtime uses two different execution modes. A "workspace worker" is a long-lived Codex session running inside a sterile local worker root that contains `worker_manifest.json`, `assigned_shards.json`, `assigned_tasks.json`, `hints/*.md`, `in/*.json`, `out/*.json`, and `shards/<task-or-shard>/...`. A "structured follow-up" is a one-shot strict-JSON Codex call used for watchdog retry or repair after the main worker produced no usable result. The shared subprocess and live-supervision seam for both modes lives in `cookimport/llm/codex_exec_runner.py`.

The knowledge stage currently fans `10` shard workers into many smaller "task packets." A task packet is the actual unit of work that should yield one `out/<task_id>.json` file. The saved failing benchmark at `data/golden/benchmark-vs-golden/2026-03-20_21.44.20/.../codexfarm/` used `10` shard workers but `485` task packets in the knowledge stage. That is why shard-only progress reporting and shard-only completion logic were misleading.

The benchmark progress UI lives in `cookimport/cli.py` and depends on structured progress payloads produced by `cookimport/core/progress_messages.py`. The benchmark interruption and partial-summary path also currently lives in `cookimport/cli.py`. Stage observability and run-level artifact indexing live in `cookimport/runs/stage_observability.py` and `cookimport/llm/prompt_artifacts.py`.

The stage runtime is expected to write a proper phase package under `raw/llm/<slug>/knowledge/` including `phase_manifest.json`, `shard_manifest.jsonl`, `task_manifest.jsonl`, `worker_assignments.json`, `promotion_report.json`, `telemetry.json`, `failures.json`, and `proposals/*.json`, plus the separate `knowledge_manifest.json` index at the workbook root. The postmortem run did not finish this package, which means the stage did not merely "look bad"; it failed to complete its own contract.

The key saved run roots for this plan are:

- Benchmark root: `data/golden/benchmark-vs-golden/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/`
- Processed run root: `data/output/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-20_21.44.56/`
- Knowledge stage root: `data/output/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-20_21.44.56/raw/llm/saltfatacidheatcutdown/knowledge/`

Anyone implementing this plan should keep those artifact roots intact and use them as the primary evidence bundle while adding deterministic tests and local replay helpers.

## Plan of Work

### Milestone 1: Define and enforce one coherent knowledge worker contract

Start by aligning the worker prompt, worker manifest notes, workspace policy, and follow-up policy so they describe the same behavior. The current mismatch is the root of several later failures. In `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md`, `cookimport/llm/knowledge_prompt_builder.py`, and the workspace-worker prompt assembly in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, rewrite the worker instructions so they stop inviting broad speculative automation and instead state the exact contract:

- main workspace workers may perform bounded local file work inside the worker root;
- they must decide each task from chunk-local evidence, not from metadata shortcuts such as lane hints or blanket template logic;
- they must produce valid JSON files atomically or not at all;
- they must stop once every assigned task has reached a terminal local result;
- they must not use repo-level or interpreter tools unless the stage explicitly permits a tiny named helper surface.

At the same time, make the shared watchdog semantics in `cookimport/llm/codex_exec_runner.py` and the knowledge-stage live callback in `cookimport/llm/codex_farm_knowledge_orchestrator.py` match that contract. Main workspace-worker enforcement should remain boundary-based and liveness-based. Structured follow-ups should remain strict. The missing piece is an explicit classification of off-task behavior that stays inside the workspace boundary but clearly drifts from grounded task work. Add a narrow, testable concept of "productive workspace work" versus "exploratory or repetitive drift" based on signals already visible in the live snapshot and event stream: command count, repeated schema searching, repeated AGENTS rereads, repeated zero-output passes, and large batch-script generation before any valid outputs land. The goal is not to punish normal shell use. The goal is to stop a worker that is still bounded but clearly not doing the assigned review work.

This milestone is complete when the prompt text, manifest notes, watchdog callback comments, and tests all describe the same main-worker and follow-up behavior in plain language.

### Milestone 2: Make task packets first-class runtime entities with explicit terminal states

The next change is structural. In `cookimport/llm/codex_farm_knowledge_orchestrator.py`, stop treating task packets as transient local details and instead give them explicit runtime records with terminal states. A task packet should end in one of a small number of named outcomes such as `validated`, `invalid_output`, `missing_output`, `repair_recovered`, `repair_failed`, `retry_recovered`, `retry_failed`, `watchdog_killed`, or `cancelled_due_to_parent_completion`. Each task must reach exactly one terminal state. A shard should then derive its own state from the set of its task states instead of from loosely coupled local file checks.

This work should introduce a task-level runtime artifact tree and summary files that the orchestrator can write incrementally. The existing `task_manifest.jsonl` is a good seed, but it is not enough. Add a task-status artifact that records current state, last attempt type, validation result, and owning worker. Then teach the shard aggregation path to read that task-level status as the source of truth. This immediately solves several postmortem problems at once: progress can count packets honestly, retries can be bounded per packet, stale follow-ups can be cancelled when the task already has a terminal state, and finalization can still write a useful stage package even when many packets failed.

Keep the data model simple and repository-local. This is a solo-project runtime, not a distributed scheduler. A small set of JSON status files and in-memory helpers is enough.

### Milestone 3: Reject malformed and low-trust outputs at the moment they appear

The postmortem showed that "output present" is too weak. Worker `001` wrote all `49` files, yet every file was invalid JSON. This milestone adds immediate validation at the first useful seam. In the workspace supervision path and the task aggregation path in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, inspect newly appeared `out/<task_id>.json` files as soon as they appear. If a file is not parseable JSON, mark the task `invalid_output` immediately and record a concise failure reason. If the JSON is parseable but violates the schema or the semantic grounding rules in `cookimport/llm/codex_farm_knowledge_ingest.py`, record the exact validation error and move the task to bounded recovery. Do not let large numbers of bad files accumulate before the stage notices.

This is also the right place to tighten semantic validation against templated or metadata-driven outputs. The stage already has useful grounding rules such as exact chunk coverage and useful-versus-non-useful semantics. Extend those checks so the stage is more skeptical of suspicious bulk patterns:

- all blocks marked identically across many tasks with no chunk-specific reasoning;
- snippets that simply echo every block wholesale without real selection;
- reason strings that are generic or copied from metadata rather than grounded in text;
- outputs using wrong bundle version or wrong decision vocabulary;
- files whose structure implies a bulk template rather than a task-local judgment.

The implementation does not need a speculative AI detector. It only needs deterministic structural checks that catch the specific failure modes seen in the saved run.

### Milestone 4: Bound recovery and cancel stale follow-ups aggressively

The current retry and repair layer creates too much churn for too little gain. Rebuild that layer around strict per-task budgets. A task should get at most one watchdog retry and one repair attempt unless a test-backed reason justifies a second path. Once the budget is exhausted, the task is terminally failed and the shard must carry that failure honestly. This change belongs in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, with any small shared helper changes in `cookimport/llm/codex_exec_runner.py`.

Then add active cancellation and cleanup rules. If a parent shard has already reached a terminal state, cancel any still-running task-level follow-up calls for that shard. If a task has already been repaired or otherwise reached a terminal good state, cancel any older retry or repair subprocess still marked running for that task. If the benchmark or stage is interrupted, mark any still-running follow-ups as cancelled and finalize their status files instead of leaving them as eternal `running`.

The saved worker `007` stale repair is the regression fixture for this work. The acceptance bar is that no follow-up call remains live after its parent shard has already completed or after the stage is told to stop.

### Milestone 5: Rebuild knowledge progress around task packets and follow-up state

Once task packets have explicit terminal states, rebuild progress reporting around them. This milestone absorbs the current focused packet-progress plan. In `cookimport/llm/codex_farm_knowledge_orchestrator.py`, create a small private progress-state helper that tracks packet totals, packet terminal counts, shard totals, shard terminal counts, and per-worker current labels. Use packet counts as `task_current/task_total` for the knowledge phase. Put shard counts and follow-up summaries into detail lines. Worker labels should include shard ownership and local packet progress, and if a worker is in a retry or repair follow-up, the label should say so explicitly.

In `cookimport/cli.py`, keep renderer changes minimal and backward-compatible. The preferred fix is to improve the knowledge stage payloads rather than to create knowledge-specific UI logic. The rendered view should make the saved run legible, for example:

    task 233/485 | running 6
    configured workers: 10
    completed shards: 3/10
    repair calls running: 4
    retry calls running: 2
    worker 10: saltfatacidheatcutdown.ks0009.nr (47/48 task packets)

This milestone is complete when the benchmark timeseries and the terminal UI both tell the truth about packet progress and follow-up churn.

### Milestone 6: Finalize stage-level artifacts on success, failure, and interruption

The knowledge stage must always write its own phase package. Move the finalization logic in `cookimport/llm/codex_farm_knowledge_orchestrator.py` so the stage writes `phase_manifest.json`, `promotion_report.json`, `telemetry.json`, `failures.json`, and any available proposals or explicit failed-task summaries under all terminal conditions: success, partial failure, watchdog collapse, and interruption. A run that dies mid-stage should still leave a finished description of how far it got and why it stopped.

Then wire that stage-level truth into the benchmark interruption path in `cookimport/cli.py`, `cookimport/labelstudio/ingest.py`, and, if needed, `cookimport/runs/stage_observability.py`. The benchmark root should continue to write `benchmark_status.json` and `partial_benchmark_summary.json`, but those summary files should now point at a valid stage-level knowledge manifest rather than a half-empty raw worker tree.

This is also the milestone that should normalize cancellation semantics. Ctrl+C or `KeyboardInterrupt` should not leave the orchestrator respawning workers after the user asked it to stop. If fully graceful signal propagation proves invasive, the minimum acceptable behavior is that the benchmark stops spawning new knowledge follow-ups, marks the stage interrupted, writes stage and benchmark summaries, and exits cleanly.

### Milestone 7: Improve observability, prompt-budget reporting, and reviewer artifacts

After the runtime is correct, make it inspectable. Extend the existing runtime summaries and prompt-budget surfaces so a finished or interrupted run makes the pathologies visible without rereading every worker `events.jsonl`. This likely touches `cookimport/llm/prompt_budget.py`, `cookimport/llm/prompt_artifacts.py`, and `cookimport/runs/stage_observability.py`.

Add explicit knowledge-stage metrics such as:

- task packets total, validated, invalid, missing, repaired, retry-recovered, retry-failed, cancelled;
- main workers completed, watchdog-killed, malformed-completed;
- follow-up calls started, completed, failed, cancelled, stale-killed;
- invalid JSON file count;
- schema-invalid count;
- semantic-invalid count;
- off-task watchdog count.

Keep the metrics compact, but make them sufficient for future postmortems. The user should not need to manually count `repair_status.json` files again.

## Validation and Acceptance

This plan is complete only when all of the following are true.

For worker behavior, bounded local workspace automation remains allowed for main workers, but workers that drift into repetitive schema-searching, metadata-only bulk templating, or other clearly unproductive behavior are terminated early with explicit reasons. The tests must prove both sides: allowed bounded work and denied off-task behavior.

For task outcomes, every packet reaches exactly one explicit terminal state. There is no packet that remains implicitly unresolved just because a worker died or a follow-up subprocess kept running.

For malformed outputs, invalid JSON and schema-invalid files are detected as soon as they appear, and the stage records those failures explicitly instead of treating file presence as progress.

For recovery, each task has a bounded retry and repair budget. The stage cannot create unbounded retry storms. The tests must prove that a known-bad task stops after the configured budget and that stale follow-ups are cancelled once the parent task or shard has already reached a terminal state.

For progress reporting, the knowledge phase reports packet-based progress. The equivalent of the saved Salt Fat run must display packet counters such as `task 233/485`, not `task 0/10`, and worker rows must show packet progress or follow-up state.

For finalization, the knowledge stage always writes its own phase package on success, failure, and interruption. At minimum, the stage root must contain `phase_manifest.json`, `promotion_report.json`, `telemetry.json`, and `failures.json`, even when many tasks failed or the benchmark was interrupted. The benchmark root must point at those artifacts in its own partial-summary files.

For observability, a finished or interrupted run must surface enough metrics to explain whether the stage failed because of malformed outputs, semantic validation, watchdog kills, retry exhaustion, or cancellation.

## Idempotence and Recovery

This work should be done additively and is safe to repeat. The first step of each milestone is test-first, so current bad behavior is pinned before runtime code changes. If a milestone proves too wide, split it further in `Progress` rather than broadening a partial fix without updating the plan.

If the worker-contract changes become too permissive, narrow the off-task detection rules or the prompt contract rather than reintroducing shell-shape policing. If progress reporting becomes too noisy, emit only on meaningful task-state changes. If stage finalization under interruption proves difficult to wire through a fully graceful shutdown path, prefer writing an explicit interrupted stage package first and improving signal propagation later.

Do not delete the saved March 20 artifact roots while this work is in progress. They are the canonical evidence bundle for this plan and the easiest way to verify that deterministic tests still model the original failures.

## Interfaces and Dependencies

In `cookimport/llm/codex_farm_knowledge_orchestrator.py`, define or extend private helpers that make task packets first-class runtime records. By the end of this plan, that module must own:

- a task-level status model with one explicit terminal outcome per task;
- a progress-state helper that emits packet-based structured progress;
- bounded retry and repair orchestration with cancellation hooks;
- stage-level finalization that writes manifests and failure summaries under all terminal conditions.

In `cookimport/llm/codex_exec_runner.py`, preserve the split between telemetry classification and real boundary enforcement, and add any small shared support needed for off-task detection, malformed-output supervision, and cancellation-aware finalization. Do not turn this into a generic scheduling framework.

In `cookimport/llm/codex_farm_knowledge_ingest.py`, make the ingest and validation layer prescriptive about what counts as grounded, useful, semantically valid knowledge output. This module is the authority for rejecting schema-valid but semantically bogus outputs.

In `cookimport/core/progress_messages.py`, keep the existing structured transport as the default. Prefer to reuse:

- `task_current/task_total` for packet counts;
- `running_workers/worker_total` for worker slots;
- `active_tasks` for descriptive worker labels;
- `detail_lines` for shard counts, retry counts, and follow-up summaries.

In `cookimport/cli.py`, keep rendering backward-compatible for older stages and older progress payloads. Any wording change should remain generic enough to help the knowledge stage without breaking other stages.
