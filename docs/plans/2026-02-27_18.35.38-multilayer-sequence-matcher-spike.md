# MultiLayer SequenceMatcher Spike Against Current `cydifflib` Baseline

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This repository includes `docs/PLANS.md` at the repository root for ExecPlan rules. This plan must be maintained in accordance with that file.

## Purpose / Big Picture

After this change, the benchmark stack can run an additional opt-in matcher backend (`multilayer`) using the 2026 hierarchical-LCSTR approach, without changing existing defaults. This allows direct apples-to-apples parity and timing comparisons against the current selector path (`auto`, typically `cydifflib`) on the same code path used by canonical evaluation.

You should be able to run one benchmark script and see whether `multilayer` is faster or slower than `cydifflib` on representative inputs, while parity tests verify scoring behavior remains identical.

## Context and Orientation

Sequence matcher selection for canonical evaluation is centralized in `cookimport/bench/sequence_matcher_select.py` and consumed by `cookimport/bench/eval_canonical_text.py` via `SelectedSequenceMatcher`. Current modes are `auto|stdlib|cydifflib|cdifflib`; `auto` prefers accelerated drop-ins.

Parity protections already exist in `tests/bench/test_sequence_matcher_dropin_parity.py` and include opcode/matching-block checks plus end-to-end canonical artifact byte equality checks between `stdlib` and `auto`.

This spike adds a new local Python backend implementation and uses the same selector/testing surfaces so results are directly comparable.

## Plan of Work

Milestone 1 adds a MultiLayer matcher module under `cookimport/bench/` implementing a SequenceMatcher-like API (`__init__`, `set_seq*`, `get_matching_blocks`, `get_opcodes`). The implementation should preserve difflib-compatible outputs (including sentinel and adjacent-match merge semantics) and expose an optional memory multiplier control through an env var for tuning.

Milestone 2 wires selector support for a new explicit mode (`multilayer`) while preserving current `auto` ordering. Telemetry should naturally reflect `alignment_sequence_matcher_impl=multilayer` through existing plumbing.

Milestone 3 extends parity tests with dedicated `multilayer` assertions and keeps existing accelerated-dropin tests unchanged.

Milestone 4 extends `scripts/bench_sequence_matcher_impl.py` so one command compares `stdlib`, `auto`, and `multilayer` timing and parity signatures.

Milestone 5 executes focused tests and benchmark commands, then records findings in docs with a keep/tune/drop recommendation.

## Validation and Acceptance

Acceptance for this spike is evidence-based:

- `multilayer` mode can be selected without breaking existing modes.
- `stdlib` vs `multilayer` parity tests pass for opcodes and matching blocks.
- End-to-end canonical artifact parity test passes for `stdlib` vs `multilayer` on fixture input.
- Timing command reports `multilayer` vs `auto` results; recommendation states clearly whether `multilayer` is promising, neutral, or worse on tested inputs.

## Idempotence and Recovery

This spike is additive and reversible. If any issue appears, force `COOKIMPORT_BENCHMARK_SEQUENCE_MATCHER=stdlib` or leave unset (`auto`) to restore current behavior. Because `auto` behavior is unchanged, rollback is simply avoiding `multilayer` mode.
