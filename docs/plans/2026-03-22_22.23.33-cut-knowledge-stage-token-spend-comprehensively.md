# Cut knowledge stage token spend comprehensively

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

The knowledge stage is now semantically useful. In the March 22, 2026 `21:47:46` Salt Fat benchmark, CodexFarm materially beat vanilla on quality, validated every shard, and promoted useful cooking knowledge into final authority. The remaining problem is cost. The same run spent `26,413,491` knowledge tokens across only five workers because the runtime still pays too many full workspace turns for too many small packet units, and it still asks the model to reconsider a large amount of material that deterministic cues already describe as low-value.

After this work lands, a human should be able to rerun the same approved benchmark slice and see the knowledge stage keep its current quality wins while spending materially fewer tokens. The visible proof is not just “some lower number.” The proof is that the stage produces fewer task packets, fewer expensive worker turns, fewer calls per worker session, less cached replay, less wrapper overhead, and less batch-helper churn while preserving full shard validation, clean queue completion, and the current quality advantage over vanilla.

This plan intentionally treats the cost problem from several angles at once. The root cause is not one bug. The current cost comes from a stack: parent shards are fragmented into tiny task packets, the worker loop is still verbose around those packets, the prompt and sidecar surfaces are still larger than they need to be, and the stage still sends many obvious-`other` packets to the model instead of resolving them deterministically.

## Outcomes & Retrospective

At authoring time, this plan has not been implemented yet. Its main contribution is to unify the latest benchmark evidence into one coherent cost program instead of one more narrow patch. Earlier plans already repaired correctness, queue completion, and the basic utility-first semantic contract. This plan assumes those improvements are real and focuses on what still makes the stage expensive after quality became good.

The expected outcome is not one giant rewrite. The expected outcome is a sequence of measurable wins. First, fewer task packets. Second, fewer expensive worker turns per packet. Third, less worker-side reread churn. Fourth, fewer packets sent to the model at all when deterministic evidence is already strong enough to keep them `other`. If implementation reveals that any later layer endangers quality, the earlier layers should still stand on their own as useful cost reductions.

## Context and Orientation

The knowledge stage takes non-recipe text that survived deterministic staging and asks a model to decide whether that text contains durable cooking leverage worth preserving. In this repository, “durable cooking leverage” means information that helps future cooking decisions, diagnosis, substitution, storage, safety, or technique. It does not mean “any sentence about cooking that happens to be true.”

The relevant files divide into four roles.

The first role is parent-shard planning in `cookimport/llm/codex_farm_knowledge_jobs.py`. This file builds larger parent shard payloads from non-recipe chunks. Its current default bundle caps are already relatively generous: `_DEFAULT_KNOWLEDGE_BUNDLE_MAX_CHUNKS = 12`, `_DEFAULT_KNOWLEDGE_BUNDLE_MAX_CHARS = 14000`, and `_DEFAULT_KNOWLEDGE_BUNDLE_MAX_GAP_BLOCKS = 10`.

The second role is task-packet planning and workspace runtime orchestration in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. This file currently takes each parent shard and splits it into task packets before the live worker sees them. This is where the latest cost problem becomes concrete. `_build_knowledge_task_plans(...)` currently groups chunks into packets with a hard `2`-chunk cap and a `12_000` byte cap. The same file also builds the long worker prompt and owns the main workspace-worker session lifecycle.

The third role is worker-local batch control in `cookimport/llm/knowledge_workspace_tools.py`. This file writes `current_batch.json`, `CURRENT_BATCH.md`, `CURRENT_BATCH_FEEDBACK.md`, `assigned_tasks.json`, `scratch/current_batch/*.json`, and the `tools/knowledge_worker.py` helper surface. It currently limits each micro-batch to four tasks or `24_000` input bytes, writes and checks batch drafts, and installs a valid prefix into `out/*.json`.

The fourth role is validation and promotion in `cookimport/llm/codex_farm_knowledge_ingest.py` plus the benchmark and prompt-budget summaries under `data/output/.../knowledge/` and `data/golden/.../prompt_budget_summary.json`. The ingest code enforces exact chunk ownership, exact block coverage, snippet grounding, and the strong-cue empty-shard guard. Those correctness rules are not the main target of this plan, but their metadata and observability are part of the cost program because any deterministic fast path must still be explainable and auditable.

The latest motivating evidence is the benchmark run rooted at `data/golden/benchmark-vs-golden/2026-03-22_21.47.46/` with runtime artifacts under `data/output/2026-03-22_21.47.46/.../knowledge/`. In that run, the stage validated all five shards and produced good quality. The cost problem is therefore not “the stage is broken.” The cost problem is “the stage is healthy enough that its remaining structural inefficiencies are now visible.”

## Milestones

### Milestone 1: Make the cost shape explicit and hard to misread

Before changing behavior, add or tighten summary code so a reader can see where knowledge cost comes from without manually spelunking `telemetry.json`, `task_manifest.jsonl`, and the benchmark artifacts. After this milestone, the repo should expose packet-count distributions, packet-size distributions, worker-session totals, call-churn ratios, and deterministic-triage opportunity counts in one place, and tests should pin those summaries.

The acceptance for this milestone is simple. A novice should be able to point at one repo-owned summary or one focused test fixture and answer these questions: how many parent shards existed, how many task packets they became, how large those packets were, how many packets were bypassed deterministically, how much cost came from worker sessions versus follow-up calls, and whether the stage is still burning many calls per worker session or many calls per packet.

### Milestone 2: Cut task-packet count aggressively without weakening correctness

The highest-leverage runtime change is to stop creating seventy-nine tiny packets from five already-reasonable parent shards. After this milestone, ordinary knowledge packets should contain more than two chunks when safe, and very small low-risk packets should stop paying a full workspace-turn share by themselves.

The acceptance for this milestone is behavioral, not stylistic. On the same source fixture, `task_manifest.jsonl` should shrink materially, packet ownership should remain exact and deterministic, retry safety should remain intact, and the validator should still receive one packet-local semantic result per declared `result_path`.

### Milestone 3: Shrink the worker’s happy-path surface and batch churn

Once packet count falls, the next cost lever is the worker loop itself. The worker should read less, rerun fewer helper steps, and need fewer raw hint/input rereads per installed batch. After this milestone, the worker prompt, batch sidecars, helper output, and workspace-manifest notes should all steer the agent through a smaller, cheaper happy path, and a successful `install-batch` should normally hand off enough next-step context that the worker does not need a mandatory sidecar reread tour.

The acceptance for this milestone is visible in telemetry and in the generated worker root. A novice should see smaller startup surfaces, fewer helper commands required to continue after install, fewer instructions that force reopening `CURRENT_BATCH*` after success, and a clearer distinction between ordinary batch work and narrow debug fallback.

### Milestone 4: Add deterministic no-LLM triage for obvious low-value packets

After the runtime is cheaper, add a controlled bypass for packets that are already strongly negative and low-risk. This is the most semantically sensitive milestone, so it must be implemented conservatively and with explicit observability. After this milestone, some packets should never reach the model because the repo can already justify a deterministic `other` outcome from strong negative utility evidence and the absence of strong positive cues.

The acceptance for this milestone is not just fewer packets. The acceptance is that the repo records exactly which packets were bypassed, why they were bypassed, and how many chunks/blocks that affected, and that focused tests prove useful positive packets are not incorrectly absorbed by the bypass.

### Milestone 5: Prove the final shape locally, then benchmark only with approval

The final milestone is proof. Tighten tests, update docs, and only then, with explicit user approval, rerun a narrow benchmark slice. The benchmark must show that cost fell for the reasons this plan claims, not because a hidden accounting change masked the spend.

The acceptance for the approval-gated rerun is that the quality wins from the latest run remain intact while `task_packet_total`, `call_count`, `tokens_input`, `tokens_cached_input`, `wrapper_overhead_tokens`, and command churn all fall materially from the `2026-03-22_21.47.46` baseline.

## Plan of Work

Start in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. Replace the fixed `_KNOWLEDGE_TASK_PACKET_MAX_CHUNKS = 2` assumption with a dynamic packet planner. The default should become materially larger than two chunks, and the grouping logic should prefer keeping neighboring low-risk chunks together instead of splitting them into many tiny packets. Preserve deterministic order, exact `owned_ids`, exact `owned_block_indices`, and the current retry/relaunch bookkeeping. Do not change the validator contract. The worker still writes one semantic packet result per packet. The cost win comes from having fewer packets, not from redefining what a packet means.

When redesigning packet planning, use the metadata already attached to chunks. In practice, this means the planner should look at `char_count`, table hints, heading-heavy fragments, strong positive utility cues, strong negative utility cues, and likely mixed-risk seams. Large table-heavy or very long prose chunks can remain isolated when needed. The win should come from merging the easy low-risk neighbors, not from forcing every chunk into a giant packet.

Then update `cookimport/llm/knowledge_workspace_tools.py`. Increase or redesign the micro-batch limits so the worker can amortize more work per helper cycle once task packets are larger. The current `4`-task and `24_000`-byte caps were chosen for a much more fragmented queue. Re-evaluate them against the new packet planner. At the same time, shrink the batch surfaces. `CURRENT_BATCH.md` should remain the cheap first read, but it should not repeat material that the worker will immediately reread from `OUTPUT_CONTRACT.md` or per-task hints. `install-batch` should print enough next-step context that the worker does not need a status tour after every successful install, and the prompt/helper surface should stop treating post-install sidecar reopening as the default success path.

Still in `knowledge_workspace_tools.py`, cut the happy-path helper noise. The current prompt and telemetry show repeated `complete-batch`, repeated draft dumps, repeated contract rereads, and repeated raw hint/input reads. The helper should make the obvious path cheaper by prewriting stronger default drafts, minimizing the need to dump them back to the model, and keeping failure explanations packet-local. Preserve the `debug` namespace for narrow recovery, but keep the happy path focused on `current-batch`, `complete-batch`, `check-batch`, `install-batch`, and `explain-failure`.

Return to `cookimport/llm/codex_farm_knowledge_orchestrator.py` and shrink the worker prompt. The prompt already encodes the right semantic standard, but it is longer than it needs to be for every worker session. Keep the utility-first knowledge definition, the repo-owned queue rule, and the packet result contract, but remove repeated wording and lower-value prose. The prompt should rely on the batch sidecars and contract files for specifics instead of restating everything in multiple ways. The worker should still know what to do if a batch fails, but the normal path should dominate the visible text.

Then inspect `cookimport/llm/codex_exec_runner.py`. Keep it stage-generic, but reduce knowledge-stage orientation duplication where possible. The runner should still mirror the necessary control files and preserve `usage.json`, `workspace_manifest.json`, and other session artifacts, but it should not expand the broad workspace orientation if the knowledge stage now provides a tighter worker manifest and prompt.

After the packet and worker-surface wins are in place, add the deterministic triage layer. The correct place to start is before the model sees a packet, in the knowledge task-planning or staging boundary, not after paying for the workspace session. Define a conservative deterministic `other` bypass for packets where every chunk is already seeded `other`, negative utility cues dominate, no strong positive utility cues are present, and no chunk is marked as a strong knowledge cue or utility-borderline candidate. The bypass must write repo-owned artifacts that look like a justified deterministic resolution, not like a silent disappearance. It must also preserve final-authority accounting so downstream code can distinguish “LLM reviewed and stayed other” from “repo bypassed LLM as confidently other.”

Update `cookimport/llm/codex_farm_knowledge_ingest.py`, `cookimport/llm/prompt_budget.py`, and any stage-summary helpers only as needed to make the new shape observable. The benchmark and prompt-budget summaries should report how many packets were LLM-reviewed, how many were deterministically bypassed, how much cost belonged to main worker sessions, and the churn ratios that explain wrapper-heavy runs. At minimum that means exposing enough information to derive or read directly: calls per worker session, calls per task packet, and token totals per reviewed chunk or packet. If a packet bypasses the LLM, the summary should say so explicitly instead of letting the packet vanish from cost accounting.

Do not use this plan to slim the external-review upload bundle. Heavy artifacts such as `prompt_request_response_log.txt`, `prompt_nonrecipe_knowledge_review.txt`, or `knowledge_manifest.json` can be addressed later in a separate benchmark-upload cleanup task. This plan is about the live knowledge-runtime spend, not attachment transport size.

Finally, update short docs. `docs/10-llm/10-llm_README.md` and `docs/10-llm/nonrecipe_knowledge_review.md` should say that knowledge now uses larger dynamic packets, a cheaper batch-first happy path, and an explicit deterministic low-value bypass when the repo can already justify `other` confidently. Keep those notes short and operator-oriented.

## Validation and Acceptance

Local acceptance comes first.

The planning tests must prove that ordinary knowledge work now produces materially fewer task packets from the same parent shards. A novice should be able to compare a fixture before and after the code change and see that the stage no longer defaults to `2`-chunk packets when the chunks are safe to keep together.

The helper and prompt tests must prove that the worker’s happy path is smaller than it is today. The generated worker prompt, `CURRENT_BATCH.md`, `CURRENT_BATCH_FEEDBACK.md`, helper stdout, and worker manifest should all tell the same short story: read the current batch, complete/check/install it, and continue. They should not require the worker to rediscover the queue or reread large contract files after every success, and a successful install should usually hand off directly into the next active batch without a mandatory orientation lap.

The ingest and runtime tests must prove that larger packets did not weaken the correctness rules. Exact owned chunk coverage, exact owned block coverage, strong-cue empty-shard rejection, snippet grounding, queue-completion semantics, and retry/relaunch safety must all remain intact.

## Idempotence and Recovery

This work is safe to stage incrementally. The recommended order is: first improve observability, then enlarge packet planning, then retune the worker batch loop and prompt surface, then add deterministic triage, then update docs and benchmark summaries, then run local tests, then perform any approval-gated benchmark proof.

Do not modify the saved benchmark evidence under `data/golden/benchmark-vs-golden/2026-03-22_21.47.46/` or the paired runtime artifacts under `data/output/2026-03-22_21.47.46/`. They are read-only evidence for this plan.

If larger packets cause one packet to become too hard to validate or repair, lower the dynamic packet caps or split only the high-risk packet classes. Do not revert the entire system to a hard `2`-chunk default. If deterministic triage shows any sign of eating useful knowledge, keep the packet and worker-surface wins and gate the triage layer back behind a narrower rule or a temporary setting while preserving the new observability.
