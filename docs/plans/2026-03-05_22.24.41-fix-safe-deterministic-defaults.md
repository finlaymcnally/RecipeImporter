# Fix Safe Deterministic Defaults for Shared Run Settings

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds. This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The goal of this change is to make the repository safe by default. After this work, a developer who runs the ordinary shared CLI flows without explicitly opting into Codex should get deterministic behavior, not silent live LLM usage. The user-visible result is simple: when someone creates `RunSettings()` or calls the generic settings loader with no config file, the resolved settings should be safe and deterministic, and any Codex usage should require an explicit choice.

This plan covers discrete problem 1 only: wrong default behavior. In this repository, “wrong default behavior” means the shared defaults currently enable live recipe Codex and live line-role Codex even when the caller does not explicitly ask for them. This plan intentionally does not solve the separate QualitySuite baseline-construction bug, analytics labeling bug, or reuse-key problem. Those are distinct follow-up changes.

## Context and Orientation

The main settings object in this repository is `RunSettings`, defined in `cookimport/config/run_settings.py`. It is the canonical source of default behavior for many code paths. When code calls `RunSettings()` with no explicit overrides, or when CLI helpers build settings from an empty or missing config file, these defaults flow outward into processing, benchmarking, and reporting.

Three fields matter for this plan.

`llm_recipe_pipeline` controls whether the recipe-extraction CodexFarm flow runs. The current live-Codex value is `codex-farm-3pass-v1`; the safe deterministic value is `off`.

`line_role_pipeline` controls whether the canonical line-role pipeline uses Codex. The current live-Codex value is `codex-line-role-v1`; the safe deterministic value is `off`. There is also `deterministic-v1`, which is deterministic but still turns on the canonical line-role machinery. For the shared defaults in generic flows, this plan chooses plain `off` because the current bug is about doing extra work without explicit intent.

`atomic_block_splitter` controls whether benchmark-native atomic line blocks are generated. It is not an LLM by itself, but in this repository it is tightly coupled with the line-role benchmark path. The current default is `atomic-v1`; the deterministic baseline value is `off`.

The low-level call site for live Codex usage is `cookimport/llm/codex_exec.py`. It already requires `COOKIMPORT_ALLOW_LLM=1`. That gate is a last-ditch safety barrier, not the primary user-facing policy. The policy we want after this plan is stronger: ordinary defaults should not even resolve to live-Codex settings.

The highest-risk test files for this change are `tests/llm/test_run_settings.py` and `tests/cli/test_cli_output_structure_fast.py`. They currently assert that generic defaults are AI-on. Several docs also describe the current posture, especially `docs/02-cli/02-cli_README.md`, `docs/07-bench/07-bench_README.md`, and `docs/AI_Context.md`.

A novice implementing this plan should keep one distinction in mind the whole time. “Shared defaults” means what happens when the caller does not ask for anything special. “Explicit opt-in variants” means named or deliberate paths such as `codexfarm`, benchmark pairs, or prompts that intentionally enable Codex. This plan changes the first category and preserves the second.

## Plan of Work

The implementation should proceed in four broad phases.

First, change the canonical defaults in `cookimport/config/run_settings.py`. The default constructor for `RunSettings` must resolve to deterministic behavior. Set `llm_recipe_pipeline` to `off`, `line_role_pipeline` to `off`, and `atomic_block_splitter` to `off`. Keep the enum choices and validation intact; only the default values should change. The UI metadata descriptions should also be adjusted if they imply live behavior is the standard posture.

Second, update the generic CLI behavior that mirrors these defaults. The central target is `cookimport/cli.py`, especially `_load_settings()` and any command signature defaults that currently assume the “top tier profile.” This includes `stage` and `labelstudio_import`, which are pinned by `tests/cli/test_cli_output_structure_fast.py`. The implementation should make those generic defaults match the safe `RunSettings` posture. The plan should not weaken already-safe commands like `labelstudio_benchmark` that already default recipe and line-role pipelines to `off`.

Third, preserve and clarify explicit opt-in flows. The single-offline and single-profile paired benchmark builders already have an explicit concept of deterministic `vanilla` versus opt-in `codexfarm`. Keep those builders explicit. If any helper currently relied on shared AI-on defaults to produce the codex variant, replace that reliance with deliberate setting of `llm_recipe_pipeline=codex-farm-3pass-v1`, `line_role_pipeline=codex-line-role-v1`, and `atomic_block_splitter=atomic-v1`. The implementation should prefer explicitness over inheritance here.

Fourth, update tests and docs. Rewrite tests that assert AI-on defaults so they instead assert deterministic shared defaults plus explicit opt-in paths where relevant. Then update the docs named above so they teach the new mental model: Codex is opt-in, deterministic is the repository default, and named codex-enabled modes remain available when asked for directly.

## Validation and Acceptance

Acceptance is behavioral, not just structural.

A successful implementation must satisfy all of the following.

When a developer creates `RunSettings()` with no overrides, the resolved run config must show `off/off/off` for recipe LLM, atomic block splitter, and line-role pipeline.

When `_load_settings()` runs with no config file present, the returned settings dictionary must also show `off/off/off` for those same fields.

When the user runs the CLI paths that rely on shared defaults, they must no longer silently select Codex-enabled settings. The easiest proof is the updated `tests/cli/test_cli_output_structure_fast.py`, which should explicitly assert the new deterministic defaults.

When the user chooses an explicit opt-in flow such as a `codexfarm` benchmark variant, the code must still construct the Codex-enabled settings deliberately rather than inheriting them from the global defaults.

The following test commands should pass in the venv after implementation:

    source .venv/bin/activate
    pytest tests/llm/test_run_settings.py tests/cli/test_cli_output_structure_fast.py -q

A stronger acceptance pass is to run one explicit opt-in benchmark-planning test that checks `codexfarm` still resolves to `codex-farm-3pass-v1` and `codex-line-role-v1`, while default settings remain off.

## Idempotence and Recovery

This plan is safe to implement incrementally. The settings-default changes are pure source edits with no database or artifact migration.

If a test fails after flipping defaults, do not immediately revert the default change. First determine whether the test is asserting shared defaults, explicit opt-in behavior, or unrelated downstream semantics. Update only the tests that were intentionally encoding the old default posture.

If documentation drifts during implementation, treat the docs as part of the change and update them in the same branch. The repository guidance here is “no ship without docs,” so leaving docs stale is a failed implementation even if tests pass.

Rollback is straightforward. Revert the default field values and any directly related tests/docs. No destructive cleanup is required.

## Interfaces and Dependencies

The relevant interfaces at the end of this work are existing ones with corrected defaults rather than new public APIs.

In `cookimport/config/run_settings.py`, `RunSettings` must continue to expose the same fields and enum choices, but its zero-argument constructor must resolve to deterministic settings.

In `cookimport/cli.py`, `_load_settings()` must continue returning the same settings dictionary shape, but the default contents for the Codex-related fields must match the deterministic `RunSettings` values.

Named opt-in flows must continue producing the existing explicit Codex-enabled values:

- `llm_recipe_pipeline = codex-farm-3pass-v1`
- `line_role_pipeline = codex-line-role-v1`
- `atomic_block_splitter = atomic-v1`

The implementation should not add a new environment variable, new config file, or new command-line flag for this problem. The needed interfaces already exist; the bug is that the defaults are wrong.
