# Replace Recipe Refine Taskfile Sessions With Thin Transport And Codex Policy

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, recipe refinement will stop spending most of its tokens on shell choreography and taskfile ceremony. A user will still be able to run the same benchmark or prompt-preview flow and get the same kind of recipe decisions, the same deterministic validation, and the same promotion behavior, but the default recipe Codex path will become a thin structured answer exchange instead of a mini interactive shell session. The repo will also gain explicit Codex policy generation so stages that do not need shell can disable it entirely, while any remaining taskfile stage runs under an allowlisted command policy rather than relying on prompt text alone.

This matters because the current `recipe_refine` stage is paying taskfile-worker costs that are wildly out of proportion to the actual semantic payload. In the `2026-04-06_22.46.09` benchmark root for `saltfatacidheatcutdown`, `recipe_refine` spent `1.718M` billed tokens across only `22` recipes and `5` worker calls, roughly `78k` billed tokens per recipe. The visible prompt body per worker was only about `2.6k` characters, and the repo-written JSON inputs per worker were only tens of kilobytes. The activity traces show the real cause: the model spent a large share of its budget acting like a shell operator, running helper commands and rewriting `answers.json` via heredoc instead of simply returning recipe decisions in a structured payload.

The user-visible proof after implementation is:

1. The default recipe Codex transport can run in a thin structured mode that writes the same durable recipe outputs and still fails closed on malformed answers.
2. `prompt_budget_summary.json` and related prompt-budget artifacts make the savings visible by separating semantic payload from protocol overhead for recipe stages and by exposing which transport was used.
3. Codex worker policy is generated by the repo, so packet-style stages can run with shell disabled and any remaining taskfile stage can reject non-allowlisted shell usage before it turns into token-heavy churn.

## Surprises & Discoveries

- Observation: `recipe_refine` is the least defensible token sink in the current Codex pipeline.
  Evidence: In `data/golden/benchmark-vs-golden/2026-04-06_22.46.09/single-book-benchmark/saltfatacidheatcutdown/codex-exec/prompt_budget_summary.json`, recipe refine consumed `1.718M` billed tokens across `5` calls for `22` recipe units, about `78k` billed tokens per recipe.

- Observation: The semantic payload is modest; the task is not intrinsically large enough to justify the spend.
  Evidence: Worker-local recipe packets in that run were only about `11 KB` to `28 KB`, `task.json` and original task snapshots were about `15 KB` to `46 KB`, and the worker-visible prompt body was about `2.6k` characters.

- Observation: The dominant waste is shell-oriented execution churn rather than model reasoning.
  Evidence: The worker activity traces show `task-summary`, `task-show-unanswered`, `task-template`, `cat answers.json`, a large heredoc answer rewrite, `task-apply`, and `task-handoff`. The run did not report meaningful reasoning-token usage; it mainly paid for wrapper and session overhead.

- Observation: The current runner already contains the essential seam needed for a thin recipe path.
  Evidence: `cookimport/llm/codex_exec_runner.py` already has recipe-specific helpers for building a direct answer request and normalizing a structured answer payload. The missing piece is not a brand-new concept; it is wiring that seam into recipe runtime and config.

- Observation: The current command builder does not yet inject repo-generated Codex policy/config.
  Evidence: `cookimport/llm/codex_exec_command_builder.py` currently assembles the `codex exec` command with model, schema, sandbox, and reasoning settings, but not a stage-specific config file that can disable shell or install stricter rules/hooks.

- Observation: Prompt instructions alone are not enough to prevent shell-shaped waste.
  Evidence: `cookimport/llm/recipe_stage/task_file_contract.py` already tells the worker to use the helper-guided workflow and avoid arbitrary shell behavior, but the recorded sessions still devolved into file-inspection and heredoc-writing patterns.

- Observation: Repo policy can be stricter than prompt language because Codex supports tool restriction at runtime.
  Evidence: Official Codex configuration supports disabling the shell tool entirely for a run and supports rule- or hook-based restrictions for Bash usage. The repo is not using those capabilities yet.

## Context and Orientation

This repository imports cookbook data, stages deterministic evidence, and then asks Codex-backed workers to make fuzzy semantic judgments in a few places. A “transport” in this repo means the mechanical shape of that worker exchange: either a thin packet-style request where the model returns structured JSON in one response, or a taskfile session where the repo creates a workspace file such as `task.json` and the worker edits or fills answers through a more session-like interaction.

Recipe refinement is the Codex-backed stage that turns packaged recipe evidence into repaired or normalized recipe answers that can later be validated and promoted. The existing recipe taskfile contract lives in `cookimport/llm/recipe_stage/task_file_contract.py`. It builds the worker-visible prompt and taskfile payload and currently recommends a helper-driven workflow. The recipe runtime and promotion surfaces live across `cookimport/llm/recipe_stage/runtime.py`, `cookimport/llm/recipe_stage_shared.py`, `cookimport/llm/recipe_stage/validation.py`, `cookimport/llm/recipe_stage/reporting.py`, and `cookimport/llm/recipe_stage/worker_io.py`.

The generic Codex execution runner lives in `cookimport/llm/codex_exec_runner.py`. This module already handles worker invocation, response normalization, usage accounting, and stage-specific plumbing for both taskfile and structured response modes. It already contains a recipe-specific direct-answer seam through `_build_recipe_task_answer_request(...)`, `_normalize_recipe_task_answer_payload(...)`, and `_build_workspace_task_unit_answer(...)`. That seam is the intended foundation for the thin recipe path.

The code that actually constructs the `codex exec` command line lives in `cookimport/llm/codex_exec_command_builder.py`. This is where repo-generated policy/config must be injected. The existing taskfile command classifier and shell-boundary logic live in `cookimport/llm/codex_exec_taskfile_policy.py`. This module already knows how to recognize helper commands, egregious commands, and workspace-policy violations. It should remain the repo’s deterministic interpreter of shell activity even after thin mode becomes the default.

Run settings are owned under `cookimport/config/`. `cookimport/config/run_settings.py`, `cookimport/config/run_settings_types.py`, `cookimport/config/run_settings_builders.py`, and related adapter modules already expose `line_role_codex_exec_style` and `knowledge_codex_exec_style`, both defaulting to `inline-json-v1`. Recipe does not yet have a comparable `recipe_codex_exec_style`, which is why recipe still behaves as if taskfile mode were mandatory. The interactive CLI owns user-facing configuration prompts in `cookimport/cli_ui/run_settings_flow.py`.

Prompt-budget reporting and benchmark summaries live in `cookimport/llm/prompt_budget_runtime.py` and the benchmark helper code under the Label Studio and bench domains. These modules already emit `prompt_budget_summary.json` and related artifacts, but they need a clearer recipe transport story so a human can immediately tell whether a cost came from semantic payload, thin structured transport, or taskfile/session overhead.

This plan builds on the earlier de-bloating work captured in `docs/tasks/2026-04-04_23.33.14 - taskfile worker runtime simplification and de-bloating.md`. The key context from that earlier plan is repeated here so this document stands alone: non-recipe stages already moved toward `inline-json-v1`, shared prompt-budget fields now separate protocol overhead from semantic payload, and recipe taskfile payloads have already been slimmed somewhat. What remains is the stage with the clearest remaining waste: recipe_refine.

The implementation must not run live Codex benchmarks on its own. This repo’s `AGENTS.md` explicitly forbids agent-run LLM-enabled processing and benchmarks without explicit user approval. All end-to-end validation in this plan therefore uses deterministic prompt preview, fake Codex runners, or targeted tests unless the user later requests and approves a real benchmark run.

## Plan of Work

Start by formalizing recipe transport as a first-class run setting. Add a recipe transport selector to the run-setting types and defaults, mirroring the existing line-role and knowledge selectors. The allowed values should be `taskfile-v1` and `inline-json-v1`. Set the product default to `inline-json-v1` only after the new path is implemented and validated; until then, the migration can temporarily preserve `taskfile-v1` while the new code lands behind a switch. Update the CLI flow so recipe transport can be shown and persisted in the same surface that already handles stage-specific Codex settings.

Next, add a repo-generated Codex policy/config layer. Create one small module under `cookimport/llm/` that produces stage-scoped Codex config artifacts for a worker run. The exact filename can be chosen during implementation, but it should live near `codex_exec_command_builder.py` and own only policy generation, not stage semantics. The generated policy must support three modes:

1. `shell disabled`, used by thin structured stages such as recipe `inline-json-v1`, where the worker should return structured JSON and should never need arbitrary shell access.
2. `taskfile allowlist`, used by remaining taskfile stages, where a very small set of repo-owned helper commands is allowed and common high-churn patterns such as arbitrary `cat`, heredoc answer rewrites, or generic `bash -lc` write flows are rejected.
3. `debug permissive`, an explicit opt-in escape hatch for local troubleshooting if the repo still needs it for development, but never the default benchmark path.

The generated policy should be passed into `codex exec` from `cookimport/llm/codex_exec_command_builder.py`. The repo should not assume that prompt instructions will keep the worker honest. If the Codex runtime supports disabling shell entirely, the thin recipe path should use that mode. If the runtime supports Bash rules or pre-tool hooks, the taskfile fallback should use a repo-generated allowlist rather than a prose request in the prompt. `cookimport/llm/codex_exec_taskfile_policy.py` should remain the local classifier for recorded command behavior and for deterministic reporting, but the actual prevention should happen before the model can waste tokens.

Then implement the thin recipe transport. The goal is not to redesign recipe semantics. The goal is to swap the mechanical conversation shape from “interactive taskfile session” to “one structured answer request and one structured answer response.” Use the existing recipe helpers in `cookimport/llm/codex_exec_runner.py` as the starting point. The runner should be able to package each recipe unit into the structured request it already knows how to build, invoke the packet-style worker path, parse the structured answer payload, normalize it into the same internal unit-answer objects the rest of the recipe pipeline expects, and then feed it through the existing deterministic validation and promotion steps.

The recipe stage code should continue to own recipe semantics. `cookimport/llm/recipe_stage_shared.py`, `cookimport/llm/recipe_stage/runtime.py`, `cookimport/llm/recipe_stage/validation.py`, and the reporting modules should not start duplicating generic transport logic. Instead, they should select the transport, ask the runner for the appropriate execution mode, and receive back the same internal answer shape regardless of whether the source was a thin structured response or the taskfile fallback. That compatibility layer is important because it allows the benchmark, validation, and promotion logic to stay stable while the transport changes underneath.

Once the thin path exists, harden taskfile fallback rather than deleting it immediately. Rewrite the recipe taskfile prompt in `cookimport/llm/recipe_stage/task_file_contract.py` so the fallback contract is smaller, more explicit, and aligned with the new policy layer. The worker-visible taskfile should remain an owned answer ledger plus answer schema and minimal stage hints. Remove or sharply reduce prose that suggests a multi-command helper workflow. The fallback should still support durable answer editing where genuinely needed, but it should no longer silently bless the shell-heavy dance that caused the observed benchmark overhead.

After the execution paths are in place, update observability so the transport story is visible in artifacts. `cookimport/llm/prompt_budget_runtime.py` and any stage-level reporting helpers should record the selected recipe transport, the Codex policy mode, and the semantic-versus-protocol split for recipe calls. The benchmark artifacts should make it easy to answer all of the following without reading raw worker logs: which recipe transport was used, whether shell was disabled, how many recipe units were processed, how many calls were made, and how much of the total spend came from visible payload versus wrapper overhead.

Finally, update docs and tests. The relevant docs are the LLM runtime README, the bench README, the CLI README if it mentions transport selection, and a short note in the recipe-stage folder if that folder already contains implementation-oriented documentation. The tests must cover both the new thin default and the remaining taskfile fallback. Because live benchmark runs require explicit approval, the default proof for this plan is fake-runner coverage plus prompt-preview or prompt-budget artifact generation that exercises the new code without spending live model tokens.

## Milestone 2: Generate Codex Policy/Config Per Stage

This milestone gives the repo enforceable control over Codex tool usage. At the end of the milestone, the command builder can attach a repo-generated policy/config artifact to a worker run, and tests can prove that thin recipe runs disable shell while taskfile fallback runs install a narrow allowlist.

Create a small stage-policy builder module near `cookimport/llm/codex_exec_command_builder.py`. This module should accept the stage key, transport style, and any explicit debug override flags, then write the minimal Codex config or rule artifacts needed for that run. The config must be ephemeral and safe to regenerate. The module should also expose enough metadata for reporting and tests so the rest of the repo can say which policy mode was used without parsing the raw config text later.

Integrate that module into the command builder. The command builder should continue owning the mechanical `codex exec` argv assembly, but it should now also ask the new policy builder for any config files, hook files, or command-line settings it must include. Keep the implementation additive and easy to back out. The policy builder should not mutate recipe prompts or runner behavior directly; it should only control Codex execution policy.

Acceptance for this milestone is:

1. A focused command-builder test proves that thin recipe runs request shell-disabled policy.
2. A focused command-builder or policy test proves that taskfile recipe runs request the allowlisted taskfile policy instead of the shell-disabled policy.
3. The fake runner or subprocess harness can still execute through the new command builder without needing live Codex access.

## Milestone 3: Implement Thin Recipe Refine Transport Behind A Switch

This milestone delivers the main product change. At the end of the milestone, recipe refinement can run through a packet-style structured exchange that returns normalized recipe answers without entering a taskfile session.

Implement the recipe thin path by extending the current recipe runtime to route `recipe_codex_exec_style=inline-json-v1` through the structured-answer seam in `cookimport/llm/codex_exec_runner.py`. Make the runner package recipe evidence into the existing request builder, invoke the packet worker path, normalize the structured answer payload, and hand the result back to the recipe stage in the same internal answer shape that taskfile mode already produces. The deterministic validation and promotion layers must remain the authority on structure, completeness, and whether the answers can be promoted.

Add tests that prove the thin path handles the happy path and the key fail-closed cases. A malformed or incomplete structured response must still be rejected deterministically. If the current runner already has a same-session repair or structured-retry concept for packet responses, recipe should reuse it where appropriate instead of inventing one more retry loop.

Acceptance for this milestone is:

1. Focused recipe worker tests prove that `inline-json-v1` can produce valid recipe unit answers without taskfile artifacts.
2. Focused runner tests prove that malformed recipe structured responses are rejected and surfaced cleanly.
3. The recipe stage still writes the same kind of validated internal outputs and promotion artifacts once answers are accepted.

## Milestone 4: Shrink And Harden Taskfile Fallback

Remove helper-language that suggests an exploratory shell session. The prompt should explain the owned file surface and the final handoff expectation in plain language, but it should not teach the worker to operate like a general-purpose shell user.

Update `cookimport/llm/codex_exec_taskfile_policy.py` so the deterministic command classifier matches the new intended fallback surface. Add tests that prove disallowed command patterns are recognized and reported. If the generated Codex policy layer already blocks these commands before execution, the local classifier still matters for artifact interpretation and for any debug runs that expose recorded command traces.

Acceptance for this milestone is:

1. Focused fallback tests prove recipe taskfile mode still writes valid answers and promotions when used explicitly.
2. Focused policy tests prove high-churn shell patterns that caused the earlier benchmark waste are now rejected or classified as boundary violations.
3. The recipe taskfile prompt and taskfile artifacts are materially shorter and calmer than the baseline shape.

## Idempotence and Recovery

All planned code changes are additive or local refactors and should be safe to apply incrementally. The repo-generated Codex policy/config artifacts must be ephemeral and safe to regenerate; they must not become manually edited durable source files. The thin recipe path should be introduced behind an explicit transport selector first so the fallback path remains available if a regression appears.

If a milestone fails halfway, restore progress by keeping recipe on `taskfile-v1` in defaults while leaving the new thin path behind an explicit test-only or developer-only switch until the validation gap is closed. Do not delete the taskfile fallback until the thin path has passed the focused tests and the fake-runner or prompt-preview proof. If the command-builder policy integration causes runner breakage, keep the policy builder module in place but temporarily route taskfile mode through the old command builder while the argv integration is fixed. The important safety rule is that semantic validation and promotion must remain deterministic and unchanged in authority even if transport work is reverted temporarily.

## Interfaces and Dependencies

At the end of this plan, the following interfaces or equivalent seams must exist.

In `cookimport/config/run_settings_types.py` and the related settings modules, define a recipe transport field equivalent to the existing non-recipe transport selectors. The final shape can follow existing conventions, but it must allow at least `taskfile-v1` and `inline-json-v1`.

In `cookimport/llm/codex_exec_command_builder.py`, add a stage-policy input so command construction can include repo-generated Codex config or rule artifacts. The command builder should remain the only place that knows how those policy artifacts become `codex exec` arguments.

In a new module near the command builder, define a policy builder that accepts stage key, transport style, and any explicit debug override flags and returns:

    a policy mode label such as shell-disabled or taskfile-allowlist
    any ephemeral files or config text that must be materialized for codex exec
    the command-builder metadata needed to attach those artifacts to a worker invocation

In `cookimport/llm/codex_exec_runner.py`, keep or strengthen the existing recipe structured-answer helpers so recipe can route through the packet-style runner and still normalize into the same internal answer objects used by validation and promotion.

In `cookimport/llm/prompt_budget_runtime.py` and any related artifact helpers, add stable transport and policy labels to the recipe stage summaries so cost reviews remain easy after this migration.
