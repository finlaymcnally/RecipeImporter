# Raise All-Method Tail Throughput With Source Scheduling, Heavy-Source Sharding, And Tail Pairing

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This repository includes `docs/PLANS.md` at the repository root for ExecPlan rules. This plan must be maintained in accordance with that file.

## Purpose / Big Picture

After this change, all-method benchmark runs should spend less time in low-utilization endgame tails where only one heavy source remains active. The user-visible result is that benchmark runs finish sooner on multi-core machines, and the spinner/dashboard more often shows multiple active sources deeper into the run instead of dropping to one giant book at the end.

This plan implements three concrete capabilities together: keep more sources alive deeper into the run, split heavy source workloads into smaller schedulable units, and dispatch sources with a tail-pair strategy so giant sources are less likely to be stranded as the final source.

## Surprises & Discoveries

- Observation: Source dispatch in multi-source all-method is currently FIFO by discovery order, not cost-aware.
  Evidence: `_run_all_method_benchmark_multi_source(...)` builds `pending_jobs = list(source_jobs)` and pops from the front (`pop(0)`) in `cookimport/cli.py`.

- Observation: Discovery order is based on freeform export mtime, not expected runtime.
  Evidence: `_discover_freeform_gold_exports(...)` sorts by `(mtime, path)` descending in `cookimport/cli.py`.

- Observation: Runtime skew between sources is large enough that a bad source order creates long one-source tails.
  Evidence: `data/golden/benchmark-vs-golden/2026-02-26_18.07.45/all-method-benchmark/all_method_benchmark_multi_source_report.json` includes source wall times ranging from ~29s (`hix_written`) to ~5109s (`amatteroftastecutdown`) and ~4501s (`saltfatacidheatcutdown`).

- Observation: Late-run CPU underutilization can be structurally normal when only one source remains and only a few unique canonical alignment keys are active.
  Evidence: `docs/understandings/2026-02-27_18.49.32-all-method-final-source-eval-tail-cpu-cap.md`.

## Context and Orientation

All-method has two scheduler layers:

1. Source layer (`_run_all_method_benchmark_multi_source` in `cookimport/cli.py`): dispatches independent sources (books) with a thread pool and `max_parallel_sources`.
2. Per-source config layer (`_run_all_method_benchmark` in `cookimport/cli.py`): runs each source’s variant configs with split/eval phase-aware scheduling.

This plan changes only the source layer and source planning input to it. It does not change canonical-text scoring semantics.

Relevant current files:

- `cookimport/cli.py`
  - source discovery and matching (`_discover_freeform_gold_exports`, `_resolve_all_method_targets`)
  - all-matched interactive orchestration (`_interactive_all_method_benchmark`)
  - source-level dispatcher (`_run_all_method_benchmark_multi_source`)
  - per-source config scheduler (`_run_all_method_benchmark`)
  - settings defaults/menu wiring (`_load_settings`, `_settings_menu`)
- `tests/labelstudio/test_labelstudio_benchmark_helpers.py`
  - most all-method scheduler and multi-source behavior tests
- `docs/02-cli/02-cli_README.md`
- `docs/07-bench/07-bench_README.md`

Terms used in this plan:

- Source job: one schedulable unit passed to `_run_all_method_benchmark(...)`.
- Heavy source: a source with high estimated runtime relative to peers.
- Workload shard: one source job containing a subset of a source’s variants.
- Tail pairing: source dispatch order that intentionally pairs heavy and light jobs to reduce makespan tails.

## Plan of Work

Milestone 1 introduces a source planning step before dispatch. Add helper functions in `cookimport/cli.py` that compute a rough runtime estimate per source, then produce a planned source job list with metadata. Use cheap deterministic signals: variant count, canonical text size (from `exports/canonical_text.txt` when available), and optional prior timing summary if discoverable.

Milestone 2 adds heavy-source workload sharding at the source layer. For sources above a configured threshold, split variants into multiple shards (disjoint subsets), producing multiple source jobs for the same source file. Each shard runs via existing `_run_all_method_benchmark(...)`, so evaluator semantics stay unchanged. Add shared canonical cache override support so shards for the same source can reuse alignment artifacts.

Milestone 3 adds tail-pair dispatch. Replace pure FIFO source submission with a planned order generated by strategy:

- `discovery` (legacy behavior)
- `tail_pair` (new behavior): interleave heavy and light planned jobs so long jobs start earlier and are less likely to be last.

Milestone 4 tightens defaults and runtime caps for deeper source parallelism on capable machines. Keep existing `all_method_max_parallel_sources` as the user-facing cap, but update default/effective resolution logic so modern multi-core hosts can keep more sources active without destabilizing per-source scheduler budgets.

Milestone 5 updates tests and docs, and verifies on a real mixed-size all-matched run that source-level tail behavior improves.

## Validation and Acceptance

Acceptance is behavior-based.

1. Source planning

- A new source-plan metadata structure is generated for all-matched runs and persisted in multi-source report JSON.
- Legacy strategy (`discovery`) reproduces previous ordering exactly.

2. Heavy-source sharding

- When sharding is enabled and thresholds are met, at least one source is split into multiple workload shards.
- Combined report still presents coherent per-source outcomes (success/fail counts, timing, winner metrics), with shard metadata available for audit.
- No evaluator contract changes: all runs remain canonical-text benchmark runs with the same scoring surface.

3. Tail pairing

- `tail_pair` strategy dispatch order interleaves heavy and light source jobs deterministically.
- Tests prove bounded source concurrency is still respected.

4. Parallel depth

- Source-level effective parallelism can exceed the old default of 2 on capable hosts when configured/allowed.
- Existing knobs still work: setting cap low forces low parallelism.

5. Safety and compatibility

- Existing tests for all-method dashboard rendering, report generation, timeout/retry behavior, and source ordering assumptions are updated and passing.
- Existing report keys remain present; new keys are additive.

6. Manual run check

- On a mixed-size all-matched benchmark (including at least one very large source), run wall-clock tail is reduced relative to prior baseline under comparable settings.

## Idempotence and Recovery

All output remains timestamped and additive. If a run fails, rerun with a fresh timestamp as today.

Rollback toggles must be available via settings:

- force legacy dispatch: `all_method_source_scheduling = "discovery"`
- disable sharding: `all_method_source_shard_max_parts = 1` or threshold set high
- cap source parallelism: lower `all_method_max_parallel_sources`

If regressions appear, keep code but run in legacy mode first, then revert code as a second step if needed.
