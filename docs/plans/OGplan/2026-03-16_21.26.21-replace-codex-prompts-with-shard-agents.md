# Replace Codex Prompt Bundles With Bounded Phase Workers

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, RecipeImport will keep its current label-first staged architecture, but it will stop paying for one fresh Codex execution on every small label bundle, recipe, or knowledge chunk. The Codex-backed parts of the pipeline will instead run as bounded phase workers that each process multiple explicit shards under one phase-scoped instruction context. The authoritative pipeline ids will still be `codex-line-role-shard-v1`, `codex-knowledge-shard-v1`, and `codex-recipe-shard-v1`, but the runtime beneath those ids will be a manifest-driven worker harness rather than “one shard equals one agent.”

This is a burn-the-boats refactor. The old prompt-per-bundle Codex surfaces, `codex-line-role-v1`, `codex-farm-knowledge-v1`, and `codex-farm-single-correction-v1`, exist only as migration context while implementation is in flight. They are not alternate long-term paths, and the finished system must remove them from active code paths, defaults, docs, and operator-facing choices. The final system should preserve the existing semantic stage model, staged artifacts, deterministic promotion rules, Label Studio flows, benchmark scoring, and downstream review contracts. What changes is how Codex-backed work is executed, validated, and observed.

The user-visible outcome is that a human can run a shard-aware preview on an existing processed or benchmark root, then run a live benchmark outside the agent shell and compare it against the current baseline cut. The system should now report worker counts, shard counts, shards per worker, promotion success, retries, wall time, and live token totals instead of pretending prompt count is the whole story. The shard runtime must also preserve the existing RecipeImport session-isolation seam: every CodexFarm subprocess should still launch `codex exec` under the dedicated RecipeImport Codex home, and each phase worker should execute inside a clean phase-local sandbox that exposes only the files needed for that phase. The cutover must explicitly preserve and update the three human-facing Codex evidence seams that exist today: zero-token `prompt_preview`, benchmark-level `prompts/` exports, and `upload_bundle_v1`.

## Outcomes & Retrospective

This section is intentionally incomplete because implementation has not started yet.

The expected end state is a repo where the only active Codex-backed runtime for `label_llm_correct`, `recipe_llm_correct_and_link`, and `extract_knowledge_optional` is the bounded phase-worker harness under the shard-v1 pipeline ids. Deterministic stages remain deterministic, stage-backed artifacts remain authoritative, and downstream tools continue to read staged outputs rather than worker sandboxes. The final retrospective must explain whether the chosen worker-count and shard-size defaults converged across all three phases or whether each phase needed a different operating shape.

The main question to answer at completion is not “did prompt count go down?” It is whether the replacement achieved the desired runtime contract: fewer fresh Codex workers, full coverage, deterministic promotion, clean replay by shard or worker, and better real-book cost behavior without breaking the existing stage-backed product surface.

## Context and Orientation

The semantic stage model stays exactly the same:

1. `label_det`
2. `label_llm_correct`
3. `group_recipe_spans`
4. `build_intermediate_det`
5. `recipe_llm_correct_and_link`
6. `build_final_recipe`
7. `classify_nonrecipe`
8. `extract_knowledge_optional`
9. `write_outputs`

Within that stage model, the deterministic stages remain deterministic. `label_det`, `group_recipe_spans`, `build_intermediate_det`, `build_final_recipe`, `classify_nonrecipe`, and `write_outputs` stay machine-owned. The Codex-backed phases remain optional refinement/correction stages from the product’s perspective, but when enabled they will now run through one shared phase-worker harness instead of through many one-shot bundle prompts.

The current Codex-backed execution seams live in three places. `cookimport/parsing/canonical_line_roles.py` owns the current line-role correction path that feeds `label_llm_correct`. `cookimport/llm/codex_farm_orchestrator.py` owns recipe correction for `recipe_llm_correct_and_link`. `cookimport/llm/codex_farm_knowledge_orchestrator.py` plus related knowledge job/ingest files own `extract_knowledge_optional`. Preview reconstruction for these three surfaces currently lives in `cookimport/llm/prompt_preview.py`.

The transport seam is `cookimport/llm/codex_farm_runner.py`. Right now that runner assumes one structured-output task per input file and audits the runtime as non-agentic. That same runner is also the Codex-home isolation seam. `cookimport/llm/RECIPE_CODEX_HOME.md` documents that RecipeImport injects `CODEX_HOME` and `CODEX_FARM_CODEX_HOME_RECIPE` there so CodexFarm uses the dedicated RecipeImport profile by default instead of inheriting the operator’s main Codex home. The phase-worker refactor must preserve that behavior.

Definitions for this plan:

- A phase is one Codex-backed stage family for one run or workbook, such as `label_llm_correct`, `recipe_llm_correct_and_link`, or `extract_knowledge_optional`.
- A shard is a bounded logical ownership and validation unit. A shard has a stable `shard_id`, explicit owned IDs, explicit evidence references, a strict output contract, and exact coverage rules.
- A worker is a bounded Codex execution context dedicated to one phase. One worker may process multiple assigned shards in sequence during one phase session.

The runtime must preserve stage-backed downstream compatibility. Label Studio, benchmark, analytics, review bundles, and write-output flows must continue to consume staged artifacts, not a second agent-only output model. Workers write proposals; deterministic code validates and promotes them into the existing stage artifacts.

Three downstream evidence seams must be treated as first-class cutover work rather than as incidental cleanup:

- `cookimport/llm/prompt_preview.py` plus `cf-debug preview-prompts` and `cf-debug preview-shard-sweep`
- benchmark-level `prompts/` exports such as `prompts/full_prompt_log.jsonl`, `prompts/prompt_request_response_log.txt`, `prompts/prompt_type_samples_from_full_prompt_log.md`, and `prompts/thinking_trace_summary.*`
- `upload_bundle_v1`, especially the adapter/render seams in `cookimport/bench/upload_bundle_v1_existing_output.py` and `cookimport/bench/upload_bundle_v1_render.py`

## Milestones

### Milestone 1: Shared phase-worker harness and manifests

At the end of this milestone, the repo will have one reusable phase-worker runtime that can build a phase manifest, partition work into shards, assign shards to a bounded worker pool, materialize clean per-worker sandboxes, execute workers, collect structured proposals, validate them deterministically, and promote valid outputs into existing stage artifacts. This is the heart of the refactor.

Start by adding a shared runtime module, preferably `cookimport/llm/phase_worker_runtime.py`. This module should define the stable types used across phases: a `PhaseManifestV1` for phase-level planning, a `ShardManifestEntryV1` for explicit shard ownership, a `WorkerAssignmentV1` for worker-to-shard mapping, a `ShardProposalV1` for the worker’s proposed output bundle, and a `WorkerExecutionReportV1` for runtime telemetry and failures. The runtime should own shard partitioning helpers, worker assignment, workspace materialization, result collection, validation hooks, promotion hooks, and replay helpers.

The required runtime artifacts for every Codex-backed phase are:

- `phase_manifest.json`
- `shard_manifest.jsonl`
- `worker_assignments.json`
- `promotion_report.json`
- `telemetry.json`
- `failures.json`
- per-worker logs and status files
- per-shard proposed outputs

Keep these runtime artifacts separate from promoted stage artifacts. Do not create a second downstream product model, but do make the runtime inspectable.

Each worker must run inside a clean phase-local sandbox. The sandbox should contain only the worker instructions for that phase, the worker’s shard assignment manifest, the evidence files for those shards, schemas and output contracts, minimal helper adapters if needed, and a local `.codex` directory for that worker. Do not run workers from the repo root. Do not expose unrelated repo files or broad project instructions. Do not let one phase workspace persist into the next phase. Persist only logs, manifests, telemetry, and proposed outputs outside the sandbox.

At the transport layer, extend `cookimport/llm/codex_farm_runner.py` so it can intentionally run a bounded worker session rather than a single structured-output request. Add a distinct runtime audit result such as `structured_loop_agentic_v1` so logs and manifests prove the refactor is real. Preserve the existing runner-owned RecipeImport Codex home injection described in `cookimport/llm/RECIPE_CODEX_HOME.md`; the phase-worker harness must not invent a second Codex-home selection mechanism.

Acceptance for this milestone is a fake or synthetic test proving that one phase builds manifests, assigns multiple shards to at least one worker, runs through the new runtime, writes proposals instead of mutating authoritative outputs in place, and records worker/shard telemetry under the new artifact set.

### Milestone 2: Refactor `label_llm_correct` into a phase-worker stage

At the end of this milestone, the stage name remains `label_llm_correct`, but its active Codex-backed execution path is `codex-line-role-shard-v1` running through the shared phase-worker harness.

`label_det` stays deterministic and continues to produce deterministic labels, uncertainty markers, reason tags, structural hints, and row ownership metadata. The refactor here is only in how the optional correction stage executes. In `cookimport/parsing/canonical_line_roles.py`, replace the current prompt-batch execution unit with label shards that are contiguous local windows, non-overlapping in owned rows or blocks, sized for local review rather than whole-book reinterpretation, and explicit about owned IDs. A label worker may process multiple label shards in one session.

For each label shard, provide the worker with the owned rows, a bounded neighborhood view, deterministic labels, uncertainty markers, structural hints, parser or rule warnings, and a strict output schema. For each owned row, the worker returns one final label plus optional ambiguity or change flags. Validation must enforce exact-once coverage of owned rows, no edits outside owned IDs, and inspectable diffs between deterministic and corrected labels. Promotion should write back into the existing `label_llm_correct` artifact structure so `group_recipe_spans` and the rest of the staged pipeline stay unchanged.

Preview and telemetry for this stage must now report `worker_count`, `fresh_agent_count`, `shard_count`, `shards_per_worker`, shard-size distribution, first-turn payload distribution, promotion success/failure counts, retry counts, and observed turns/tokens when available. Do not describe the runtime as “15 prompts” or similar once this cutover lands.

Acceptance is that canonical-text scoring still works against the promoted line-role stage artifacts, all owned rows are covered exactly once, and the new runtime clearly shows workers processing multiple label shards rather than one fresh Codex execution per batch.

### Milestone 3: Preserve deterministic recipe staging, refactor `recipe_llm_correct_and_link` into a phase-worker stage

At the end of this milestone, `group_recipe_spans` and `build_intermediate_det` remain deterministic and authoritative, while `recipe_llm_correct_and_link` becomes a phase-worker stage executed through `codex-recipe-shard-v1`.

`group_recipe_spans` must continue to group from final corrected labels and preserve the current authority behavior when regrouping disagrees with importer-era recipe candidates. `build_intermediate_det` must continue to build deterministic intermediate recipe objects, preserve provenance back to source block IDs, and attach parser warnings or structural issues. The refactor here is only the recipe correction execution unit.

In `cookimport/llm/codex_farm_orchestrator.py`, keep building one deterministic intermediate correction input per `recipe_id`, but replace one-fresh-agent-per-recipe execution with recipe shards that are local in recipe order, bounded in recipe count and payload size, and stable enough for replay. A recipe shard is the ownership unit; a recipe worker may process multiple recipe shards in sequence.

For each assigned recipe shard, provide the worker with the original labeled blocks for each owned recipe, grouped-span metadata, deterministic intermediate recipe objects, parser warnings or ambiguity signals, provenance references, and the strict output contract. For every owned `recipe_id`, the worker writes a corrected intermediate recipe object, a linkage payload for deterministic final assembly, issue flags, and completion status. Validation must enforce exact-once ownership, schema-valid corrected structures, linkage integrity, provenance preservation, and no edits outside owned `recipe_id`s.

Promotion must feed the existing `recipe_llm_correct_and_link` artifacts so `build_final_recipe` remains deterministic and continues to consume promoted corrected structure plus linkage payloads. Inline recipe tagging continues to ride on the recipe-correction contract; do not reintroduce a separate tagging subsystem. Use this milestone to move repeated recipe guidance such as the tagging guide and global instructions into shard-level worker context wherever possible, while keeping per-recipe payloads focused on evidence and deterministic hints.

Acceptance is that the staged recipe artifacts and final recipe writer still see one corrected result per `recipe_id`, but the runtime telemetry now shows bounded recipe workers processing multiple recipe shards in one phase session.

### Milestone 4: Preserve deterministic non-recipe ownership, refactor `extract_knowledge_optional` into a phase-worker stage

At the end of this milestone, `classify_nonrecipe` remains deterministic and authoritative, while `extract_knowledge_optional` becomes a bounded phase-worker stage executed through `codex-knowledge-shard-v1`.

Eligibility stays label-driven. Only spans or chunks already classified as `knowledge` are eligible for this phase. Do not turn this back into broad whole-book mining. `classify_nonrecipe` remains stage-backed and machine-readable; reviewer-facing snippets remain downstream evidence, not the ownership source of truth.

In `cookimport/llm/codex_farm_knowledge_jobs.py`, preserve the current deterministic pruning and local grouping logic, but reinterpret it as shard planning input rather than as one-shot Codex job execution. Knowledge shards should be contiguous outside-recipe regions or bounded region groups, explicit in owned IDs, replayable, and bounded in size. A knowledge worker may process multiple knowledge shards in one session.

For each knowledge shard, provide the worker with the eligible knowledge spans or chunks, evidence references, deterministic metadata or warnings, and the strict output contract. For every owned chunk or span, the worker writes proposed knowledge output, evidence snippets or references compatible with current review artifacts, issue flags, and completion status. Validation must reject ownership violations, schema-invalid outputs, and spans outside the eligible knowledge surface. Hard deterministic exclusion of obvious noise such as table of contents, navigation, legal boilerplate, endorsements, marketing copy, and signup prompts remains in place.

Promotion must feed the existing `extract_knowledge_optional` stage artifacts and keep deterministic classification artifacts intact if the phase fails. The live and preview reporting surface must now describe worker count, fresh-agent count, shard count, promotion results, retries, wall time, and observed turns/tokens rather than prompt counts.

Acceptance is that knowledge remains optional, label-driven, and stage-backed, but the runtime no longer executes as one fresh Codex call per compact bundle.

### Milestone 5: Preview cutover, benchmark `prompts/` export cutover, upload-bundle cutover, and old-path removal

At the end of this milestone, all three human-facing Codex evidence surfaces describe the new phase-worker runtime correctly: zero-token preview, benchmark-level `prompts/` exports, and `upload_bundle_v1`. The old one-shot Codex pipeline ids are removed from active choices and defaults.

Replace prompt-centric preview with worker-centric preview. `cookimport/llm/prompt_preview.py` should continue to be the local planning entrypoint, but for the shard-v1 surfaces it must now build phase manifests, shard manifests, worker assignment plans, and first-turn worker task estimates. `cookimport/llm/prompt_budget.py` should publish worker-and-shard-centric summaries: `phase_name`, `worker_count`, `fresh_agent_count`, `shard_count`, `shards_per_worker`, shard-size distribution, first-turn payload distribution, promotion success/failure counts, retry counts, phase wall time, and observed turns/tokens when live telemetry exists. Add `cf-debug preview-shard-sweep` as a local tuning helper that compares several candidate shard settings against an existing run root.

Update benchmark-level `prompts/` exports so they remain the stable reviewer/debug contract. The merged prompt log under `prompts/full_prompt_log.jsonl`, the human-readable `prompt_request_response_log.txt`, prompt samples, stage-named prompt files, and thinking-trace summaries should all continue to exist, but their semantics must now reflect phase workers and shard proposals rather than one-shot prompt bundles. When one worker processes multiple shards, the merged prompt/debug outputs must still let a reviewer trace which phase, worker, shard, and owned IDs a given interaction belonged to.

Update `upload_bundle_v1` explicitly. The adapter in `cookimport/bench/upload_bundle_v1_existing_output.py` and the renderer in `cookimport/bench/upload_bundle_v1_render.py` must continue to expose benchmark prompts and Codex evidence coherently under the new runtime. The bundle should surface worker/shard telemetry through normalized analysis/index fields without inventing a second artifact model, and bundle-local prompt/debug references should still resolve from the benchmark root’s `prompts/` folder and related stage-local artifacts.

For live validation, keep using `cookimport labelstudio-benchmark run` and `cookimport labelstudio-benchmark compare`. This preserves the existing benchmark/eval/reporting contract while allowing a human to run the phase-worker cutover candidate outside the agent shell. Do not route live validation through `bench quality-run`; that command is deterministic-only and remains out of scope.

Finally, remove the old one-shot Codex pipeline ids from active CLI choices, defaults, preview semantics, and docs. If temporary compatibility aliases remain for old saved settings, they must normalize immediately to the new shard-v1 ids and log that normalization in manifests.

Acceptance is that a human can run worker-centric preview, run one live benchmark with the three shard-v1 pipeline ids, compare it against the current baseline, and see no active product surface that still presents the old one-shot Codex runtime as a first-class option.

## Plan of Work

Begin by changing the shared runtime and config surface rather than hacking the three phases independently. In `cookimport/config/run_settings.py`, update the active pipeline-id enums so the target values are `codex-line-role-shard-v1`, `codex-knowledge-shard-v1`, and `codex-recipe-shard-v1`, and add active tuning knobs for worker count, shard sizing, and turn caps. Mirror those settings through `cookimport/config/run_settings_adapters.py`, the interactive settings flow if needed, and the command boundaries in `cookimport/cli.py` and `cookimport/cf_debug_cli.py`.

Then add `cookimport/llm/phase_worker_runtime.py` as the shared engine rather than scattering execution logic across three orchestrators. This module should own phase manifests, shard manifests, worker assignment manifests, workspace materialization, worker lifecycle control, proposed-output collection, deterministic validation hooks, promotion hooks, telemetry emission, and replay helpers. Each worker sandbox must be deliberately clean and phase-local: only the files needed for that worker’s assigned shards should be materialized so `codex exec` does not see unrelated repo clutter. Preserve the existing runner-owned `CODEX_HOME` and `CODEX_FARM_CODEX_HOME_RECIPE` injection described in `cookimport/llm/RECIPE_CODEX_HOME.md`.

Once the shared runtime exists, refactor the three Codex-backed stages in stage order. Replace the line-role execution unit inside `label_llm_correct` with label shards and label workers. Leave `group_recipe_spans` and `build_intermediate_det` deterministic, then replace the execution unit inside `recipe_llm_correct_and_link` with recipe shards and recipe workers. Leave `classify_nonrecipe` deterministic and label-driven, then replace the execution unit inside `extract_knowledge_optional` with knowledge shards and knowledge workers over eligible knowledge spans only.

Rewrite preview and telemetry in parallel with the runtime changes. `cookimport/llm/prompt_preview.py` should plan phase manifests, shards, and worker assignments for shard-v1 surfaces. `cookimport/llm/prompt_budget.py` should stop implying that prompt count equals runtime shape. Add `cf-debug preview-shard-sweep` with a small repo-local candidate file, preferably `docs/examples/shard_sweep_examples.json`, so the operator can compare phase-worker configurations locally.

Update benchmark prompt/debug export code in the same pass. ## Validation and Acceptance

Acceptance is behavioral and must be proven at four levels.

The first level is deterministic-boundary preservation. `label_det`, `group_recipe_spans`, `build_intermediate_det`, `build_final_recipe`, `classify_nonrecipe`, and `write_outputs` must remain deterministic, stage-backed, and authoritative. New tests or assertions must prove that the refactor does not reintroduce recipe-candidate-first fallback logic or a second downstream artifact model.

The second level is worker and shard correctness. New tests must prove that each shard planner covers the full intended worklist exactly once, that worker assignments can span multiple shards, that ownership is enforced, and that proposals are validated before promotion. A label phase run must still yield one final label per owned row. A recipe phase run must still yield one corrected result per `recipe_id`. A knowledge phase run must still yield one result per owned chunk or span.

The runner and workspace seam also needs explicit proof. Focused transport/runtime tests must show that worker execution still launches CodexFarm with the dedicated RecipeImport Codex home by default, that an explicit override still wins when provided, and that phase-worker sandboxes contain only phase-local files plus the worker-local `.codex` state.

The third level is preview and observability behavior. `cf-debug preview-prompts` and `cf-debug preview-shard-sweep` must both describe worker-and-shard plans rather than old prompt-per-bundle counts. Preview summaries must clearly label cost numbers as conservative bounds or first-turn estimates whenever the runtime may loop internally.

The benchmark prompt/debug export contract also needs explicit proof. Benchmark-level `prompts/` folders must still contain the merged prompt log, human-readable request/response log, prompt samples, and thinking-trace summaries, and those artifacts must let a reviewer trace phase, worker, shard, and owned IDs under the new runtime.

The upload-bundle contract needs explicit proof as well. `upload_bundle_v1` must continue to include coherent prompt/debug and stage-analysis evidence derived from the normalized model, and bundle-local references to prompt/debug material must still resolve correctly after the phase-worker cutover.

The fourth level is live benchmark behavior. A human must be able to run at least one shard-v1 candidate outside the agent shell and compare it against the current baseline with `cookimport labelstudio-benchmark compare`. The replacement is accepted only if the compare report, live manifests, and runtime telemetry together make the new behavior legible: phase name, worker count, fresh-agent count, shard count, promotion results, retries, wall time, observed turns, and token totals.

The fifth level is burn-the-boats completion. By the end of the refactor, no active product surface should still present the old one-shot Codex runtimes as first-class options. Legacy ids may remain readable only as immediate normalization shims for old saved settings; they must not preserve the old runtime as a maintained parallel branch.

## Idempotence and Recovery

All preview sweeps and focused tests must be safe to rerun. Phase and shard planners should derive deterministic ids from stable input order so rerunning preview or a fake runner yields identical shard ownership unless the configured worker-count or shard-sizing knobs change.

Workers must write proposed outputs rather than mutating authoritative artifacts in place. That makes replay and recovery explicit. If a worker or shard fails, keep completed proposals, record the failure in `failures.json`, and allow retry by shard or by worker assignment rather than rerunning the whole book.

Temporary coexistence with old pipeline ids is permitted only during implementation. The final code should normalize or remove those ids, and no user-facing or operator-facing surface should still present the old one-shot runtimes as supported choices once the refactor is complete. If an old saved setting must remain readable for a while, normalize it immediately to the corresponding shard-v1 id and emit a manifest note that the setting was upgraded.
