# Blend External-AI Starter Pack Requests into One Export Contract

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md` from the repository root.

## Purpose / Big Picture

After this change, one run of `scripts/benchmark_cutdown_for_external_ai.py` will produce a deterministic `starter_pack_v1` surface that combines both requested views: benchmark diagnosis (labels, changed lines, per-recipe deltas) and bridge/handoff diagnosis (pass1->pass2 loss, warning buckets, pass3 mapping health, outside-span trace). A reviewer should be able to answer, in minutes, whether regression is mostly label behavior or pipeline handoff loss, and then request only targeted deep artifacts.

User-visible result: each cutdown output root gains a compact, standardized starter-pack directory with fixed filenames, fixed schemas, deterministic case selection, explicit legacy-to-starter mapping, and manifest pointers to heavy follow-up artifacts.

## Surprises & Discoveries

- Observation: The current exporter already covers several requested roots (`changed_lines`, per-recipe breakdown, targeted prompt cases, label policy, process manifest), so this is mostly a schema/packaging merge rather than a greenfield build.
  Evidence: `scripts/benchmark_cutdown_for_external_ai.py` currently writes `changed_lines.codex_vs_vanilla.jsonl`, `per_recipe_or_per_span_breakdown.json`, `targeted_prompt_cases.md`, `label_policy_adjudication_notes.md`, and `process_manifest.json`.

- Observation: Bridge-critical signals exist today but are fragmented across per-run artifacts and ad-hoc subpackets instead of one deterministic starter-pack table.
  Evidence: `single-offline-benchmark_cutdown_md` root has benchmark files, while bridge views are in separate folders (`*_bridge_debug_packet*`) and not integrated into one all-recipes triage artifact.

- Observation: Current targeted-case ranking favors changed-line volume, which can miss low-change but severe handoff-collapse recipes.
  Evidence: `per_recipe_or_per_span_breakdown.json` shows recipes like `c0` can have small changed-line counts yet still be bridge-relevant per user-provided packet feedback.

- Observation: The first draft listed target files but did not fully lock down required keys and deterministic conditional thresholds for several new artifacts.
  Evidence: `README.md`, `02_call_inventory.jsonl`, `06_selected_recipe_packets.jsonl`, and `08_outside_span_trace.sample.jsonl` had intent-level descriptions but no mandatory key list; outside-span inclusion used subjective terms like “material” and “significant”.

## Context and Orientation

The exporter lives in `scripts/benchmark_cutdown_for_external_ai.py`. It currently builds per-run summaries plus root comparison artifacts, then optionally flattens output to `*_md`. Pair-level diagnostics are generated in `_build_pair_diagnostics(...)` and `_build_comparison_summary(...)`, while root files are written in `main()`.

Current root outputs already include:

- `comparison_summary.json`
- `changed_lines.codex_vs_vanilla.jsonl`
- `per_recipe_or_per_span_breakdown.json`
- `targeted_prompt_cases.md`
- `label_policy_adjudication_notes.md`
- `process_manifest.json`

The requested blend from `docs/plans/OGplan/ProPayload.md` adds a stronger “starter-pack first look” contract: compact run overview, full recipe triage with bridge counters, call inventory, selected case packets, outside-span trace when warranted, and explicit label-policy + manifest surfaces.

Legacy-to-starter mapping that must remain explicit in docs and manifest:

- root `comparison_summary.json` corresponds to `starter_pack_v1/11_comparison_summary.json` and is summarized in `starter_pack_v1/00_run_overview.md`.
- root `per_recipe_or_per_span_breakdown.json` corresponds to `starter_pack_v1/12_per_recipe_or_per_span_breakdown.json` and is bridge-augmented in `starter_pack_v1/01_recipe_triage.csv`.
- root `changed_lines.codex_vs_vanilla.jsonl` remains unchanged for compatibility, while starter pack emits `starter_pack_v1/03_changed_lines.codex_vs_baseline.jsonl`.
- root `label_policy_adjudication_notes.md` remains unchanged for compatibility, while starter pack emits `starter_pack_v1/09_label_policy.md`.
- root `process_manifest.json` remains unchanged for compatibility, while starter pack emits `starter_pack_v1/10_process_manifest.json`.

Key test surface for this script is `tests/bench/test_benchmark_cutdown_for_external_ai.py`.

- `pass3_step_count`
- `pass3_mapping_count`
- `pass3_empty_mapping`

For missing/extra block calculations, prefer set comparison when stable block identifiers are present in payloads; fall back to deterministic count deltas when only counts are available.

Feed this data into:

- `01_recipe_triage.csv` (all recipes, one row each)
- `05_bridge_summary.jsonl` (all recipes, one row each, bridge-focused projection of triage fields)
- `04_warning_and_trace_summary.json` (aggregate counts by pass, warning buckets, empty mapping, outside-span trace status)

Acceptance for Milestone 2: recipe triage clearly exposes high-loss handoff recipes even when changed-line counts are modest.

### Milestone 3: Add selected packet and casebook logic with blended selection policy

Implement deterministic case selection that blends both responses:

- top 3 by `changed_lines_codex_vs_baseline`
- top 2 by `pass1_vs_pass2_missing_block_count`
- top 2 by `pass3_empty_mapping` with upstream evidence
- top 1 outside-span contamination case
- top 1 healthy control

Dedupe and cap to 6-10 total cases in stable sort order.

Generate:

- `06_selected_recipe_packets.jsonl` with per-case end-to-end summaries (recipe metadata, changed lines, pass1/pass2/pass3 summaries, local block window excerpt)
- `07_casebook.md` as human-readable explanation of why each case was selected and what it suggests

Outside-span file rule:

- Define `outside_span_accuracy_gap = inside_span_accuracy - outside_span_accuracy`.
- Write `08_outside_span_trace.sample.jsonl` when either `outside_span_wrong_line_count >= 10` or `outside_span_accuracy_gap >= 0.05`.
- If neither threshold is met, omit the file and write `omitted_reason` in `10_process_manifest.json`.

Deterministic ranking/tie-break rules:

- Sort each metric-specific candidate pool by metric descending, then by `abs(delta_codex_minus_baseline)` descending, then by `recipe_id` ascending.
- “Upstream evidence” for `pass3_empty_mapping` means at least one of: `pass1_selected_block_count >= 8`, `pass2_warning_count >= 2`, or `pass2_extracted_instruction_count == 0`.
- Outside-span contamination case comes from highest `outside_span_wrong_line_count` using the same tie-break chain.
- Healthy control comes from recipes with `pass1_vs_pass2_missing_block_count == 0` and `pass3_empty_mapping == false`, sorted by `codex_accuracy` descending then `recipe_id` ascending.

Acceptance for Milestone 3: starter pack alone supports first-pass diagnosis and produces a precise follow-up request list.
