# Close Nonrecipe Authority Gap

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

After this change, the repository will do three user-visible things it does not reliably do today. First, any outside-recipe row labeled `NONRECIPE_EXCLUDE` by line-role will be guaranteed to land as final `other`, never final `knowledge`. Second, the outside-recipe routing rubric used by the line-role prompt will better match the benchmark policy for this cookbook, so contents-style lists, front matter, endorsements, memoir framing, and isolated topic headings stop getting promoted into knowledge review by default. Third, the saved benchmark follow-up packet and its deterministic tests will point at the real failure seam more honestly: when a line-role exclusion leaked through to final authority, the artifact story will say so directly instead of collapsing the whole problem into a vague “line-role regression.”

The proof is local and deterministic. Focused tests will show that the known leaked rows from the `2026-03-31_13.18.44` packet can no longer become final `KNOWLEDGE`, that the line-role prompt/examples now treat benchmark-style front matter and catalog lines more strictly, and that benchmark follow-up outputs expose this failure class explicitly without rerunning a live CodexFarm benchmark.

## Context and Orientation

The relevant pipeline here has three distinct stages that matter for outside-recipe rows. The first is canonical line-role routing, which labels each outside-recipe line as either `NONRECIPE_CANDIDATE` or `NONRECIPE_EXCLUDE`. In this repository, that worker-visible prompt comes from `cookimport/llm/canonical_line_role_prompt.py`, and the workspace-worker task builder and prompt assembly live under `cookimport/parsing/canonical_line_roles/`.

The second stage is the nonrecipe staging seam. `cookimport/staging/nonrecipe_stage.py` constructs the deterministic route seed and the final nonrecipe authority object. The important current contract is already spelled out in `docs/05-staging/05-staging_readme.md`: excluded rows are immediate final `other`, candidate rows may later be reviewed into final `knowledge` or `other`, and only the final authority artifact `09_nonrecipe_authority.json` is allowed to publish semantic outside-recipe truth.

The third stage is benchmark and follow-up reporting. The canonical benchmark view ultimately scores dense `line_index` labels derived from final authority. The upload-bundle and follow-up packet builder in `cookimport/bench/followup_bundle.py` assembles the human-facing “what changed and why” story from saved artifacts such as changed-line ledgers, line-role audit rows, and explicit-escalation packets.

The two earlier checked-in ExecPlans are prerequisites but not a substitute for this work. [2026-03-31_14.00.47-knowledge-worker-semantic-review-vibe.md](/home/mcnal/projects/recipeimport/docs/plans/2026-03-31_14.00.47-knowledge-worker-semantic-review-vibe.md) reshapes the worker mindset so the knowledge worker behaves like a reviewer instead of a heuristic script author. [2026-03-31_14.02.46-knowledge-worker-surface-hardening.md](/home/mcnal/projects/recipeimport/docs/plans/2026-03-31_14.02.46-knowledge-worker-surface-hardening.md) shrinks repair pressure and grouping-task size so helper-first same-session work remains realistic. This new plan assumes those improvements remain desirable, but it addresses the remaining system-level gaps they do not cover.

The concrete benchmark failure that motivates this plan is preserved locally under `data/golden/benchmark-vs-golden/2026-03-31_13.18.44/single-book-benchmark/saltfatacidheatcutdown/`. The upload bundle at `upload_bundle_v1/quality/overview.md` says strict accuracy is `0.560027`, `OTHER -> KNOWLEDGE` is the dominant confusion family, and all scored comparison mass in that bundle lies outside active recipe spans. The same bundle’s explicit-escalation packet shows several rows where line-role explicitly excluded the line, yet the final codex prediction still became `KNOWLEDGE`. Those rows are the clearest proof that prompt policy is not the only problem.

In this plan, “exclusion leak” means a row for which line-role’s authoritative routing result was `NONRECIPE_EXCLUDE`, but some later stage or projection still surfaces the row as final `KNOWLEDGE`. “Routing-policy mismatch” means the line-role prompt/examples invite `NONRECIPE_CANDIDATE` too broadly for this benchmark’s gold semantics, especially for contents/catalog lines, front matter, endorsements, memoir framing, and isolated topic headings. “Attribution gap” means benchmark/follow-up artifacts tell a vague or stage-misaligned story that points operators toward the wrong seam.

## Plan of Work

The first milestone is downstream invariant enforcement. Start in `cookimport/staging/nonrecipe_stage.py`, `cookimport/staging/nonrecipe_seed.py`, and `cookimport/staging/nonrecipe_authority.py`. Read the stage build and refine paths closely and identify every place where candidate-block updates can expand final authority beyond the original candidate set or where excluded rows could be reintroduced into the final authority map. The implementation must make one behavior impossible: if a block was routed as `exclude`, no later refinement or projection path may allow that block to end up in `authoritative_block_category_by_index` as `knowledge`. If the easiest robust fix is to hard-filter refinement inputs against `routing.candidate_block_indices`, do that there and cover it with tests. If another projection path is leaking excluded rows after authority construction, fix that seam instead or as well, but keep the invariant centrally testable.

The second milestone is routing-policy tightening for benchmark semantics. The main prompt surface is `cookimport/llm/canonical_line_role_prompt.py`, and any helper builders that mirror the same language under `cookimport/parsing/canonical_line_roles/planning.py` must stay aligned. Tighten the negative rules and few-shot examples so benchmark-style front matter, acknowledgments, foreword/introduction memoir, title-only catalog entries, seasonal contents lists, isolated topic headings, and short section-name lines default more conservatively. The repo should still treat true lesson prose as a possible knowledge candidate, but headings, catalogs, and framing prose should no longer drift toward knowledge review by default just because they are outside a recipe. Phrase this as routing policy, not final semantic truth: the aim is to reduce obviously bad candidates while leaving ambiguous expository prose to the knowledge stage.

The third milestone is saved-artifact regression coverage. Extend focused tests so the repository can prove, from local fixtures or embedded saved-packet snippets, that the known leaked rows cannot regress silently. The most direct proof is to encode the preserved cases from the explicit-escalation packet for lines 87, 88, 104, 119, 189, 192, and 196. The tests do not need the whole live benchmark run; they need enough representative inputs to prove two things. First, excluded rows remain final `other`. Second, the benchmark packet builder still exposes these rows correctly when they appear, with the right leak classification or blame wording. Keep the fixtures narrow and deterministic so routine local test loops stay fast.

The fourth milestone is attribution and follow-up reporting. The current upload bundle heavily emphasizes line-role blame, which is directionally useful but not specific enough when an exclusion leak is really a downstream authority failure. Update `cookimport/bench/followup_bundle.py` and its tests so explicit-escalation rows can distinguish at least three situations: prompt-level route broadness, post-route label change, and exclusion leak into final authority. The output should make it obvious when line-role said “never send this to knowledge” but a later stage still did.

The fifth milestone is documentation and plan completion. Update the current docs that define the routing/authority contract so they mention the new explicit leak-prevention and the narrower benchmark-policy examples. The main docs are `docs/05-staging/05-staging_readme.md`, `docs/04-parsing/04-parsing_readme.md`, `docs/06-label-studio/06-label-studio_README.md`, and the LLM doc only if prompt contract text changed enough to warrant a note. Then update this plan’s `Progress`, `Decision Log`, and `Outcomes & Retrospective` sections with the shipped behavior and the exact focused tests that passed.

## Validation and Acceptance

This plan is complete only when all of the following are true.

The first acceptance criterion is invariant enforcement. A row that line-role routed as `NONRECIPE_EXCLUDE` must never become final `knowledge` in the nonrecipe authority map. Focused staging tests must prove this directly, including a case shaped like the preserved benchmark leak.

The second acceptance criterion is routing-policy tightening. The line-role prompt and prompt-builder tests must show that benchmark-style front matter, catalog/title-list rows, memoir framing, and isolated topic headings are no longer broadly encouraged into `NONRECIPE_CANDIDATE` by default. The prompt should still leave real expository lesson prose available for later semantic review.

The third acceptance criterion is saved-artifact regression proof. Focused tests must use preserved packet-style evidence or narrow synthetic equivalents to pin the seven leaked rows or their exact failure shape, so a future change cannot silently reintroduce the exclusion leak.

The fourth acceptance criterion is attribution clarity. Benchmark/follow-up outputs must make it visible when a row was excluded at line-role time but still surfaced as final knowledge, instead of reporting the whole problem only as generic line-role churn.

The fifth acceptance criterion is authority-boundary safety. Deterministic code may enforce excluded rows as final `other`, may narrow routing prompts, and may improve reporting, but it must not infer ambiguous `knowledge|other` semantics for candidate rows or overwrite valid LLM semantic decisions.

The sixth acceptance criterion is local proof. The focused pytest commands above must pass in `.venv` without running live CodexFarm book processing, benchmark reruns, or Oracle review flows.

## Idempotence and Recovery

The staging invariant should be deterministic and safe to rerun. A repeated refinement pass with the same seed routing and the same candidate updates must not produce different final authority, and excluded rows must remain excluded each time. If a first implementation blocks too much, relax the candidate filter only with a new regression test that proves excluded rows still cannot leak through.

The routing-prompt changes are also safe to iterate on, but they should be changed together with their prompt tests so the benchmark-policy examples and the generated prompt text do not drift apart. If prompt tightening unexpectedly harms unrelated prompt tests, keep the new benchmark-specific examples and relax only the wording that caused the mismatch; do not reopen the old broad “memoir and generic advice are usually candidates” default without an explicit decision logged here.

The saved-artifact regression tests must stay lightweight. Do not wire them to the whole benchmark session tree. If a test starts depending on large packet dumps or on regenerated bundle roots, replace that dependency with a narrower fixture excerpt checked into the test or assembled inline from the preserved row fields already visible in the packet.
