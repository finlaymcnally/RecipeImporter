# Remove Stale Block-Era Fallbacks from Row-Native Loaders and Previews

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

The active row-only migration waves already made the live recipe, nonrecipe, and freeform-eval contracts row-native. What remains is older compatibility code at a few narrow ingestion and preview seams: loaders still quietly accept semantic block-era keys like `owned_block_indices`, `divested_block_indices`, `start_block_index`, `end_block_index`, `block_indices`, and `focus_block_indices` even when those surfaces are already row-native. This cleanup removes those stale fallbacks while explicitly keeping legitimate `source_block_*` provenance and source-layout metadata.

  Evidence: `npm run docs:list` fails with `ENOTSUP` from the local `tsx` IPC socket path under `/mnt/c/.../tsx-1000/*.pipe`, so this plan uses the owning domain docs directly instead of the generated list.

- Observation: the clearest remaining debt is narrow compatibility translation, not another large semantic contract rewrite.
  Evidence: the touched row-native readers already prefer row keys first; the lingering debt is fallback acceptance for block-era keys in `recipe_row_evidence.py`, `recipe_ownership.py`, `prompt_preview.py`, import-session authority normalization, prelabel focus parsing, and freeform eval helper ingestion.

- Observation: not every `block_*` field in the repo is stale debt.
  Evidence: fields such as `source_block_index`, `source_block_ids`, packet/block offset metadata, and similar deterministic source-layout identifiers still describe provenance rather than semantic labeling authority.

  Rationale: rows still come from source layout, so `source_block_*` identifiers and packet/block offset metadata remain legitimate deterministic evidence rather than semantic contract debt.
