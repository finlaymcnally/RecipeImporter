# Test The Top Five CodexFarm Follow-Up Fixes

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the Codex-backed full-stack benchmark should keep the useful line-role lift while cutting the three concrete failure families identified in the March 6 follow-up investigation: false title promotion, pass1/pass2 boundary loss that turns into `missing_instructions`, and pass3 LLM calls that spend tokens only to return an empty ingredient-to-step mapping.

The user-visible proof is not “the code looks cleaner.” The proof is that rerunning the same weak three-book slice on `saltfatacidheatcutdown`, `seaandsmokecutdown`, and `thefoodlabcutdown` produces fewer title/header regressions, fewer hard fallbacks on prose-heavy recipes, materially fewer pass3 LLM calls for pass2-ok rows, and a new follow-up packet whose targeted asks are less bad for the exact recipes called out in `data/golden/benchmark-vs-golden/2026-03-06_15.22.11/single-profile-benchmark/followup_data1/investigation_summary.md`.

This plan deliberately treats the five recommended changes as one coordinated quality pass. Two are deterministic staging guardrails, two are routing and fallback changes inside the Codex recipe orchestrator, and one is a pass3 prompt/schema change so empty mappings become observable and cheap to reason about instead of silent waste.

## Surprises & Discoveries

- Observation: `cf-debug ablate` does not compute causal attribution. It writes the comparable variant/source matrix, but the March 6 attribution readout had to be derived from scorecards, triage rows, runtime, and blame buckets.
  Evidence: `cookimport/bench/followup_bundle.py` writes `ablation_matrix.json` directly from bundle metadata; the measured attribution summary lives separately in `followup_data1/attribution_summary.json`.

- Observation: the current false-title family is not only a raw line-role problem. Two deterministic staging paths can promote the wrong block into the recipe name even when the line-role surface is noisy rather than catastrophic.
  Evidence: `cookimport/staging/stage_block_predictions.py::_find_title_block_index(...)` falls back to `candidate_blocks[0]`, and `cookimport/staging/draft_v1.py::apply_line_role_spans_to_recipes(...)` overwrites `recipe.name` from the first `RECIPE_TITLE` span with no recipe-local heading guard.

- Observation: the March 6 follow-up line-role audit has a canonical-versus-atomic join caveat, so it is good for locating case families but not good enough to serve as the only source of truth when deciding whether a mistake started in raw line-role output or later deterministic promotion.
  Evidence: `docs/understandings/2026-03-06_21.20.00-followup-attribution-and-line-role-join-caveats.md`.

- Observation: the current pass3 compact prompt explicitly encourages `{}` when links are unclear, which makes “empty mapping” cheap for the model to choose but expensive for the system to interpret after the fact.
  Evidence: `llm_pipelines/prompts/recipe.final.compact.v1.prompt.md` says “If unclear, return `{}` as a JSON string” and the March 6 packet found that every pass3 LLM call in the investigated slice returned `empty_mapping=true`.

## Decision Log

- Decision: implement the five changes in three code milestones rather than as five unrelated edits.
  Rationale: the title fixes share the same staging surface, the pass1 fixes share the same recipe-boundary surface, and the pass3 fixes share one routing/prompt/contract surface. Grouping them this way keeps the tests and rerun matrix readable.
  Date/Author: 2026-03-13 / Codex

- Decision: keep the first pass narrow and targeted to the March 6 failure slice instead of broad prompt churn across unrelated books.
  Rationale: the investigation already identified concrete selectors, code seams, and books. The right next move is to falsify those hypotheses with small, high-signal changes, not to refactor the whole pipeline.
  Date/Author: 2026-03-13 / Codex

- Decision: make the pass3 schema extension backward-compatible during rollout.
  Rationale: existing fixtures and cached outputs already load `Pass3FinalDraftOutput` with only `draft_v1`, `ingredient_step_mapping`, and `warnings`. Add the new mapping-status field in a way that accepts missing legacy payloads as `unknown` during migration.
  Date/Author: 2026-03-13 / Codex

- Decision: do not start with “retry pass3 on empty mapping.” Start by narrowing routing and making empty mappings explicit.
  Rationale: the March 6 evidence says the biggest problem is over-routing low-value rows to pass3 at all. Retrying the same low-value route before narrowing it is likely to multiply cost faster than it improves quality.
  Date/Author: 2026-03-13 / Codex

- Decision: freeze the new pass1 diagnostics and pass3 mapping-status field names in the plan instead of leaving them as implementation details.
  Rationale: `docs/PLANS.md` requires enough specificity that a novice can implement the plan without guessing at interface shape. These fields are part of the user-visible benchmark evidence, not incidental internals.
  Date/Author: 2026-03-13 / Codex

- Decision: rebuild the post-rerun follow-up packet by generating a fresh request template from the new bundle, then transplanting the March 6 ask definitions into it.
  Rationale: `cf-debug build-followup` validates `bundle_sha256`, so copying the old request file verbatim is brittle. Using a fresh template preserves the exact new bundle path/hash while keeping the asks apples-to-apples with March 6.

The relevant code paths are split across three repository surfaces.

The first surface is deterministic staging. `cookimport/staging/stage_block_predictions.py` turns a recipe candidate plus block evidence into stage labels such as `RECIPE_TITLE`, `INGREDIENT_LINE`, and `INSTRUCTION_LINE`. Its `apply_line_role_spans_to_recipes(...)` function currently sets `recipe.name` from the first projected `RECIPE_TITLE` span with very little filtering.

The second surface is the Codex recipe orchestrator in `cookimport/llm/codex_farm_orchestrator.py`. In this file, pass1 means “recipe chunking,” pass2 means “schema.org extraction,” and pass3 means “final draft generation.” The follow-up packet showed that some prose-heavy or multi-part recipes arrive at pass2 with the wrong block window, often because pass1 returned a span that still contained previous-recipe tail prose or because later clamps cut away the real instructions. The third surface is the pass3 compact contract. `llm_pipelines/prompts/recipe.final.compact.v1.prompt.md` defines what the model is told to do, `llm_pipelines/schemas/recipe.final.v1.output.schema.json` defines the JSON shape the pipeline promises to return, and `cookimport/llm/codex_farm_contracts.py::Pass3FinalDraftOutput` is the Python model that loads it. Right now that contract can only say “here is a mapping object,” not “there was no useful mapping work to do,” which hides an important routing signal.

## Plan of Work

Start with the two deterministic title guardrails because they are cheap, local, and strongly supported by the follow-up packet. In `cookimport/staging/stage_block_predictions.py`, rewrite `_find_title_block_index(...)` so it stops returning `candidate_blocks[0]` when no exact or high-confidence near-exact title match exists. The fallback rule should become conservative: prefer an exact normalized text match, allow a narrow near-exact match only when the candidate block is heading-like and does not look like yield, time, instruction lead-in, or a generic section header, and otherwise return `None`. The rest of the caller should tolerate “no title block found” instead of manufacturing one.

Then harden `cookimport/staging/draft_v1.py::apply_line_role_spans_to_recipes(...)`. Add a recipe-local heading guard before overwriting `recipe.name` from projected `RECIPE_TITLE` spans. A span should only win when its text is non-empty, is not a generic section header such as “Notes” or “Preparation,” and is plausible as a recipe heading rather than prose. If no such span exists, preserve the existing `recipe.name` instead of replacing it with the first projected title candidate. This change is the second title-related fix from the top-five list, and it must be tested against the exact false-title families from the March 6 ask-1 packet.

Next, repair pass1 boundary selection rather than trying to rescue everything in pass2. In `cookimport/llm/codex_farm_orchestrator.py`, keep the current eligibility gate but add a focused re-anchor step after `_apply_pass1_recipe_eligibility_gate(...)` and before `_apply_pass1_midpoint_clamps(...)`. The new step should look inside the current accepted window for a strong local title anchor and compare evidence on each side of that anchor. If the window is dominated by prose before the title and has better ingredient or instruction evidence after the title, reset the start boundary to the title block or to the first block immediately after it when the title block is clearly layout-only. If the window remains prose-dominant with no useful ingredient or instruction evidence after re-anchoring, drop the recipe before pass2 rather than sending a doomed window into extraction. Record the repair or drop reason in the manifest diagnostics so the follow-up packet can explain it without prompt logs.

Keep `_apply_pass1_midpoint_clamps(...)`, but make the clamp logic preserve title-forward evidence instead of blindly splitting every overlap at the midpoint. When two recipe windows overlap and one of them contains a stronger local title anchor near the overlap boundary, bias the split away from chopping off the title-following ingredient or instruction blocks. The goal is not globally optimal segmentation. The goal is to stop cases like `ALL-AMERICAN POT ROAST` and `Rice Wine Vinaigrette` from losing the part of the window that actually contains instructions.

After pass1 is less brittle, narrow pass3 aggressively. In `_build_pass3_utility_signal_for_pass2_ok(...)` and `_should_run_pass3_llm(...)`, revise the policy so pass2-ok rows only go to LLM pass3 when the deterministic writer is genuinely at risk of producing a worse draft. A pass2-ok row with non-placeholder instructions, non-empty ingredient evidence, and a usable recipe name should prefer deterministic promotion even if the canonical text is short or the schema name is missing, because the March 6 session showed that these “maybe risky” rows mostly paid for empty mappings. Keep soft-degradation rows eligible for pass3 when deterministic promotion would likely lose real content, but make the routing reason explicit and measurable in the manifest.

Finally, change the pass3 compact contract so empty mappings are explicit. Update `llm_pipelines/prompts/recipe.final.compact.v1.prompt.md`, `llm_pipelines/schemas/recipe.final.v1.output.schema.json`, and `cookimport/llm/codex_farm_contracts.py::Pass3FinalDraftOutput` to add a new `mapping_status` field. The allowed values for newly generated outputs are `linked`, `unclear`, and `not_needed`; the Python loader must also accept a missing legacy field as `unknown` during migration. The invariants are part of this plan: `linked` means `ingredient_step_mapping` contains at least one non-empty link; `not_needed` means an empty mapping is acceptable because the draft was already good enough without link work; `unclear` means the mapping may be empty or partial, but the output or manifest diagnostics must also carry a named uncertainty reason; `unknown` is load-only compatibility for old artifacts and must not be emitted by fresh prompt outputs. The prompt must tell the model to use `not_needed` when the draft is already sound and there is no useful mapping work to do, and to use `unclear` when the evidence is insufficient for trustworthy links. Then update the orchestrator so it records this status in warnings or manifest rows. For the first implementation, do not loop or retry automatically when the status is `unclear`; only record it and narrow future routing. If the benchmark rerun still shows high-value cases landing in pass3 and returning `unclear`, that is when a retry or deterministic replacement experiment becomes justified.

Documentation changes should stay short but must exist. Update the relevant benchmark or LLM README only if a public behavior or contract changes in a way a later reviewer must know. If the new mapping-status field lands, document it briefly near the pass3 compact contract notes. If the pass1 repair step introduces a new manifest diagnostic field, document that where the benchmark artifact contract is described.

## Milestones

### Milestone 1: Stop false title promotion at the deterministic staging layer

At the end of this milestone, title/header confusion should no longer be explainable by “the deterministic layer picked the first candidate block and called it a recipe title.” The code should either find a real title match or decline to promote one. The draft projection layer should also stop overwriting a good recipe name with section headers or off-span prose.

Edit `cookimport/staging/stage_block_predictions.py` and `cookimport/staging/draft_v1.py`, then add or update tests in `tests/staging/test_stage_block_predictions.py`, `tests/staging/test_draft_v1_priority6.py`, and `tests/staging/test_draft_v1_staging_alignment.py`.

Acceptance is that the focused staging tests pass and the new tests explicitly cover at least one false-title regression from The Food Lab, one from Sea and Smoke, and one positive control where a real recipe title still promotes correctly.

### Milestone 2: Repair pass1 window selection before pass2 extraction

At the end of this milestone, the system should stop sending obvious previous-recipe tails, chapter prose, or title-at-window-end spans into pass2 as if they were valid recipe evidence. The manifest diagnostics should make it clear whether a recipe proceeded, was re-anchored, was clamped, or was dropped before pass2.

Edit `cookimport/llm/codex_farm_orchestrator.py` and add or update tests in `tests/llm/test_codex_farm_orchestrator.py` and `tests/llm/test_codex_farm_orchestrator_stage_integration.py`.

Acceptance is that targeted tests cover the known `missing_instructions -> hard_fallback` family and show the repaired path producing either a better pass2 input window or an earlier, explicit drop with diagnostics.

### Milestone 3: Make pass3 selective and make empty mappings explicit

At the end of this milestone, pass2-ok rows should hit LLM pass3 much less often, and when pass3 still runs it should tell the rest of the system whether an empty mapping was a healthy “not needed” outcome or a low-confidence “unclear” outcome.

Edit `cookimport/llm/codex_farm_orchestrator.py`, `cookimport/llm/codex_farm_contracts.py`, `llm_pipelines/prompts/recipe.final.compact.v1.prompt.md`, and `llm_pipelines/schemas/recipe.final.v1.output.schema.json`. Extend tests in `tests/llm/test_codex_farm_orchestrator.py` and `tests/llm/test_codex_farm_contracts.py`.

Acceptance is that focused tests prove backward-compatible loading of older pass3 outputs, correct parsing of the new mapping-status field, and reduced pass3 routing for pass2-ok rows that already have enough evidence for deterministic promotion.

### Milestone 4: Rerun the three-book slice and prove the changes on the same case families

At the end of this milestone, the code changes must be reflected in real benchmark artifacts, not only unit tests. The same weak books from the March 6 investigation should show less structural damage and less pass3 waste, and a fresh follow-up packet should confirm or falsify the same hypotheses on the same selectors.

Run three new canonical-text full-stack benchmark evaluations for `saltfatacidheatcutdown`, `seaandsmokecutdown`, and `thefoodlabcutdown`, compare each against the March 6 vanilla baseline, then build a compact follow-up packet from the fresh outputs using the same ask structure as the March 6 packet.

Acceptance is that the new benchmark artifacts and follow-up packet give a straight answer to these questions: did false title promotion drop, did `missing_instructions` hard fallback drop on the selected recipes, and did pass3 LLM empty mappings shrink enough to justify keeping pass3 in the loop for this family at all.

## Validation and Acceptance

Acceptance is behavior-first and has five required checks.

First, false title promotion must become harder to trigger. In a fresh follow-up packet, the title/header regressions from `EXTRA-CRISPY SUNNY-SIDE-UP EGGS`, `CREAMY Vegetable Soups WITHOUT A RECIPE`, `Extra-Cheesy Grilled Cheese Sandwiches`, `BARELY WARMED PRAWNS WITH BARELY POACHED ROE`, and `BRAISED SEAWEED WITH DUNGENESS CRAB AND BROWN BUTTER` should either disappear or clearly move from deterministic promotion into raw model ambiguity. If they remain, the packet must show a new, narrower failure mode rather than the old `candidate_blocks[0]` style promotion.

Second, pass1/pass2 boundary loss must decrease on the selected missing-instructions cases. The targeted cases from the March 6 ask-2 packet should now show either a repaired title-forward window with real instructions reaching pass2 or an earlier drop with explicit pass1 diagnostics. They should not continue reaching pass2 with obvious previous-recipe tails or prose-only windows and then hard-fallbacking because instructions were never in scope.

Third, pass3 LLM usage must drop for pass2-ok rows. The manifest counts in `raw/llm/<workbook_slug>/llm_manifest.json` should show fewer `pass3_execution_mode_llm` rows and fewer `pass3_pass2_ok_llm_calls` than the March 6 candidate runs for the same books. This is the concrete test of whether the current uplift problem is mostly leverage and routing rather than total model collapse.

Fourth, pass3 empty mapping must become interpretable. Fresh pass3 outputs should either avoid the LLM call through deterministic routing or return the new mapping-status field so the system can distinguish `not_needed` from `unclear`. A run that still produces many `unknown` or empty mappings without routing reduction is not accepted.

Fifth, the overall quality tradeoff must remain favorable. The per-book compare outputs must not show a new major regression in Sea and Smoke while fixing Food Lab and Salt, Fat, Acid, Heat. If a deterministic guardrail saves the weak books but clearly crushes line-role or recipe quality in the cleaner book, that guardrail is too blunt and must be revised before this plan can be called complete.

## Idempotence and Recovery

Do not modify or delete the March 6 benchmark root. It is the baseline reference for this plan. Every new benchmark run should land under a fresh timestamped root, and every new compare or follow-up packet should live beside it rather than overwriting the original `followup_data1`.

The code changes are additive and retry-safe if implemented carefully. Focused tests can be rerun repeatedly. Benchmark reruns can be repeated because each run writes a new timestamped root. If a pass3 contract change breaks old fixtures or cached outputs, recover by making the new field backward-compatible in `Pass3FinalDraftOutput` instead of deleting old artifacts.

If the benchmark rerun shows that one milestone helped only one book while regressing another, stop after that milestone, record the evidence in this plan, and compare before stacking the next change. The plan is intentionally ordered so each stage can be validated independently before the next one lands.
