# Implement File-Backed Codex Transport For Canonical Line-Role

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

Today the canonical line-role Codex path does not "give Codex a file." It serializes an entire shard into one giant prompt string and sends that string to `codex exec` on standard input. On large books such as `thefoodlabcutdown`, that means one line-role call can contain hundreds of labeled lines plus grounding windows in the initial request body, which is how the run reached prompt files between roughly 352 thousand and 590 thousand characters and hit a context-window failure before the model had even done useful work.

After this change, canonical line-role should use a file-backed transport. The runtime should write one authoritative shard input file inside the worker workspace, send Codex a short prompt that tells it where that file lives and what output schema to produce, and let Codex inspect the file from disk. The user-visible result is not "free tokens"; the model may still need to read a lot of text. The intended win is that transport overhead stops forcing the entire shard through the initial prompt, context-window failures caused purely by giant inline prompt bodies disappear, and the runtime finally behaves like the operator mental model of "give Codex the shard file and let it work through it."

The change is complete when a future contributor can run prompt preview on a large deterministic benchmark root and see a short line-role wrapper prompt plus an explicit input-file artifact, then run the live line-role phase with explicit approval and observe that `prompt.txt` is small, `workers/*/in/*.json` is the authoritative model input, and the first-line failure mode is no longer "the prompt body itself is too large."

## Surprises & Discoveries

- Observation: the current line-role runtime already writes `workers/*/in/<shard>.json`, but those files are debug copies, not the authoritative model input.
  Evidence: `cookimport/parsing/canonical_line_roles.py` writes `input_payload` into `ShardManifestEntryV1`, but it also renders the full prompt by calling `build_canonical_line_role_prompt(...)`, and `cookimport/llm/codex_exec_runner.py` sends that prompt text to `codex exec` on stdin.

- Observation: the shared recipe runtime already contains a partial file-path rendering seam that line-role can copy.
  Evidence: `cookimport/llm/codex_farm_orchestrator.py` writes `workers/*/in/<shard>.json` and has `render_recipe_direct_prompt(...)`, which replaces both `{{INPUT_TEXT}}` and `{{INPUT_PATH}}` in the recipe prompt template.

- Observation: a file-backed design is not automatically a cost-reduction design.
  Evidence: once Codex reads a file, the file contents may still be loaded into model context and billed. The guaranteed improvement is smaller initial prompt bodies and lower transport overhead, not zero-cost file reading.

- Observation: preview and actual-cost reporting will become harder, not easier, if the runtime moves to file-backed input.
  Evidence: current preview tokenizes the rendered prompt text directly. A path-backed design needs a different accounting vocabulary because the visible wrapper prompt is small while the real model work may depend on file reads whose token footprint is only fully known after execution.

- Observation: the current line-role implementation had already split into two internal phases, so file-backed transport had to land for both `recipe_region_gate` and `recipe_structure_label`, not just one shared line-role prompt.
  Evidence: `cookimport/parsing/canonical_line_roles.py` now plans separate phase shard sets and writes separate runtime/prompt artifact families under `line-role-pipeline/runtime/<phase>/` and `line-role-pipeline/prompts/<phase>/`.

- Observation: a zero-token direct dry run can still fail even when the live runtime is correct if the fake `codex-farm` exec shim answers from the example JSON embedded in the wrapper prompt instead of opening the deposited task file.
  Evidence: the first post-restore rehearsal returned placeholder `atomic_index:123` rows for both line-role phases until `scripts/fake-codex-farm.py` was changed to resolve `{{INPUT_PATH}}` and load the real worker `in/*.json` payload.

## Decision Log

- Decision: scope this plan to canonical line-role first rather than converting recipe, knowledge, and line-role together.
  Rationale: the motivating production failure is line-role on `thefoodlabcutdown`, and line-role is the phase that currently lacks any model-facing file-path contract. A narrow first implementation reduces risk and gives a concrete benchmark target.
  Date/Author: 2026-03-17 / Codex

- Decision: treat file-backed transport as a runtime replacement for line-role, not a long-term parallel product surface.
  Rationale: the repo already has too many transport concepts. Keeping both inline and file-backed line-role modes indefinitely would complicate preview, docs, and debugging. A short-lived prototype gate is acceptable, but the durable goal is one active line-role transport.
  Date/Author: 2026-03-17 / Codex

- Decision: make the worker input file the authoritative model input artifact for line-role.
  Rationale: the current `workers/*/in/*.json` copies are the natural place to anchor the new design. Reusing that path preserves the shard-runtime artifact layout and keeps the change additive to the existing worker sandbox model.
  Date/Author: 2026-03-17 / Codex

- Decision: do not promise that file-backed transport lowers total billed tokens; promise only that it removes the forced giant inline prompt body and exposes the true read cost separately.
  Rationale: billing behavior for file reads is model/runtime dependent. The plan must stay honest and define success as eliminating transport-caused context blowups plus improving observability.
  Date/Author: 2026-03-17 / Codex

## Context and Orientation

Canonical line-role is the Codex-backed stage that reviews one ordered slice of cookbook atomic lines and decides labels such as `RECIPE_TITLE`, `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `KNOWLEDGE`, or `OTHER`. An atomic line is one small text row from the canonicalized cookbook text. A shard is one contiguous batch of atomic lines owned by a single model call. A worker is one bounded execution context under `line-role-pipeline/runtime/workers/<worker-id>/` that owns one or more shards.

The current live line-role transport lives primarily in three files. `cookimport/parsing/canonical_line_roles.py` plans shards, renders the line-role prompt text, launches the direct Codex runtime, validates exact `atomic_index` coverage on the way back, and promotes accepted predictions into the existing `label_llm_correct` outputs. `cookimport/llm/canonical_line_role_prompt.py` defines the current inline prompt format; it serializes every target line and many grounding windows directly into the prompt body. `cookimport/llm/codex_exec_runner.py` is the repo-owned direct Codex subprocess seam. It currently takes a single `prompt_text` string and sends that prompt to `codex exec` on stdin.

The shared shard artifact contract comes from `cookimport/llm/phase_worker_runtime.py`. Even though line-role does not reuse every helper there directly, it mirrors the same concepts: `phase_manifest.json`, `shard_manifest.jsonl`, `worker_assignments.json`, per-worker sandboxes, per-shard proposals, `telemetry.json`, and `failures.json`.

The current preview and reporting surfaces are also important. `cookimport/llm/prompt_preview.py` rebuilds likely prompt shapes from deterministic or `vanilla` runs without spending tokens. `cookimport/llm/prompt_artifacts.py` exports reviewer-facing prompt artifacts from completed runs. `prediction-run/prompt_budget_summary.json` and `cookimport/llm/prompt_budget.py` are the post-run actual-cost readers. All three assume, either directly or indirectly, that the prompt text itself is the main model-facing artifact. A file-backed design must update those assumptions.

There is already one nearby pattern in `cookimport/llm/codex_farm_orchestrator.py`, the direct recipe runtime. That module writes one input file per shard and has a renderer that can mention both `INPUT_TEXT` and `INPUT_PATH`. The current recipe README text still describes recipe as inline, but the presence of that renderer means the repo already has a natural place to copy a file-path prompt template pattern from.

The motivating evidence is the failed benchmark root `data/golden/benchmark-vs-golden/2026-03-17_22.10.53/single-offline-benchmark/thefoodlabcutdown`. Its line-role phase planned `2981` atomic lines into `5` shards, which means about `597` lines per shard. The saved prompt files under the failed run root were roughly `590007`, `476803`, `439908`, `352493`, and `395182` bytes. One shard failed with `Codex ran out of room in the model's context window`, and two others hit the model usage limit after the run had already spent substantial tokens. That is the benchmark this plan should use as the reality check.

## Plan of Work

Start with a prototype milestone that proves the core assumption: `codex exec` can succeed on a one-turn structured call when the prompt says "read this shard input file from the current worker workspace and return strict JSON." Do not start by rewriting preview or docs. Instead, add the smallest possible file-backed line-role path behind a temporary development gate so a contributor can run it against a tiny synthetic shard with the existing fake runner and, if token approval exists, one live shard. The goal of this first step is binary: either Codex can reliably operate from the file path in a single turn, or it cannot. If it cannot, stop and record the failure here rather than letting a speculative transport refactor spread through the repo.

Once the prototype passes, replace the current line-role inline prompt construction with a file-backed prompt contract. The new authoritative shard input should be the worker-local JSON file already represented by `input_payload` in the shard manifest. Keep the data compact and stable. The model should not be asked to reconstruct line order from scattered files or external provenance. One shard still equals one model call; the difference is that the prompt becomes a short instruction wrapper that names the input file path, defines the JSON shape it must read, and states the exact output schema. The worker runtime should keep writing `prompt.txt`, `events.jsonl`, `usage.json`, and `cost_breakdown.json`, but `prompt.txt` should now be short enough that a human can actually read it and see the transport contract clearly.

In `cookimport/parsing/canonical_line_roles.py`, stop treating the rendered inline prompt as the canonical shard artifact. The canonical artifact should instead be the structured shard input file plus a short rendered wrapper prompt. The shard manifest should continue to carry `input_payload`, `owned_ids`, `evidence_refs`, and metadata, because proposal validation and fallback behavior still depend on them. The line-role worker execution path should write the shard JSON file before the model call, render the file-backed prompt using a dedicated helper, and then call `CodexExecRunner.run_structured_prompt(...)` with that wrapper prompt while keeping `working_dir` inside the worker sandbox so Codex can reach the file path it was given.

In `cookimport/llm/canonical_line_role_prompt.py`, introduce a separate renderer for the file-backed prompt contract instead of overloading the current inline serializer. Keep the current label legend, precedence rules, and negative rules, because those are the behavioral instructions that mattered in the inline design too. What must change is the transport format. The new renderer should describe the file contents in plain language, refer to `INPUT_PATH`, define the fields Codex must read from the JSON, and preserve the strict output contract. The old inline serializer can remain as a test helper during the transition, but by the end of the work the live path should not call it for real line-role execution.

In `cookimport/llm/codex_exec_runner.py`, do not redesign the subprocess boundary. The runner should keep accepting a prompt string and a working directory. The important change is observability: teach the telemetry row and any helper payloads to record `prompt_input_mode`, `request_input_file`, and, when available, the shard input file size. That makes the post-run artifacts honest. The visible prompt is no longer the whole task. The task now has two parts: the wrapper prompt and the file Codex was instructed to read.

In `cookimport/llm/prompt_preview.py`, stop pretending that a file-backed line-role run can be previewed with the same "tokenize the whole prompt body" logic used for inline transport. Add a file-backed preview path that writes the would-be line-role input file beside the preview artifacts, renders the short wrapper prompt, and reports at least two separate quantities: wrapper prompt tokens and shard input file token estimate or byte size. Treat the file estimate as an upper bound or structural reference, not a guaranteed billed-token prediction, and say that explicitly in the preview output. The preview surface must remain useful even if it cannot predict exact live billed tokens for agentic file reads.

In `cookimport/llm/prompt_artifacts.py` and `cookimport/llm/prompt_budget.py`, preserve the same honesty. Reviewer artifacts should expose that line-role ran in file-backed mode and should point at the authoritative input file. Actual-cost readers should continue trusting finished-run telemetry for billed tokens, but they should also surface the split between wrapper prompt size and input file size so a human can understand whether the cost came from real content or from transport overhead. If the runtime does not naturally expose file-read token counts, say so directly instead of inventing a fake exact number.

Finish by updating the current docs, not just the plan. `docs/10-llm/10-llm_README.md`, `docs/04-parsing/04-parsing_readme.md`, and `docs/07-bench/07-bench_README.md` should describe that line-role is now file-backed, explain what the worker input file contains, and clarify how preview versus actual-cost reporting should be interpreted. Keep the explanation grounded in this repository's artifacts and commands, not in external Codex marketing language.

## Validation and Acceptance

Validation starts with focused tests. `tests/parsing/test_canonical_line_roles.py` must prove that the live line-role prompt no longer inlines the whole shard body and that proposal validation still enforces exact owned-row coverage. `tests/llm/test_prompt_preview.py` must prove that preview now exports file-backed line-role inputs and reports them honestly. Add or update any worker-runtime tests needed so that a contributor can inspect artifact paths without running a live model.

Behavioral acceptance for local deterministic preview is:

1. Running `cf-debug preview-prompts` on the existing Food Lab benchmark root writes a line-role preview directory that contains both a short wrapper prompt and a separate authoritative shard input file.
2. The preview output makes it obvious that line-role is file-backed. It must not silently pretend the whole shard still lives in `prompt.txt`.
3. The largest line-role preview `prompt.txt` file is small enough to read comfortably as instructions. The exact threshold can be recorded during implementation, but it should be orders of magnitude smaller than the old `352k` to `590k` character inline prompts.

Behavioral acceptance for live execution, only when explicitly approved, is:

1. A large-book line-role run writes `workers/*/in/<shard>.json` and those files are the model-facing shard inputs described by the prompt.
2. `prompt.txt` is short and references the shard input path.
3. The first line-role failure mode on Food Lab is no longer a context-window error caused by a giant inline prompt body.
4. Finished-run telemetry and cost artifacts show file-backed mode and expose enough metadata to tell wrapper prompt size apart from shard input file size.

If the live run still spends many tokens, that is acceptable as long as the artifact story is honest and the transport-caused context blowup is gone. If the live run still hits context limits after the transport change, record whether that happened after Codex read the shard file or before it could do so; that distinction determines the next plan.

## Idempotence and Recovery

This work should be additive until the prototype passes. The temporary prototype gate, if added, should default to the current stable behavior until tests cover the new path. Once the file-backed transport is proven and the live path flips over, remove the temporary gate rather than leaving a permanent transport fork.

The worker input files are safe to regenerate. The preview outputs are safe to overwrite as long as they live under a new timestamped output directory. If a live benchmark confirmation fails midway, keep the failed artifacts, update this plan with the failure details, and use those artifacts as the starting point for the next debugging turn instead of deleting them.
