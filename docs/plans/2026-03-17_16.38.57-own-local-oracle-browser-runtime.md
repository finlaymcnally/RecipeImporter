# Own Local Oracle Browser Runtime

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document follows [docs/PLANS.md](docs/PLANS.md).

## Purpose / Big Picture

After this change, local Oracle browser automation uses one canonical Chromium profile everywhere and one machine-wide auto browser launcher: visible Chromium when a usable display exists, xvfb otherwise. Oracle also no longer depends on brittle ChatGPT model-picker automation just to submit a run. A human can log into Oracle once and expect the same profile and browser behavior rule whether the entrypoint is `cookimport`, `oracle`, `oracle-browser-auto`, or the login helper.

## Decision Log

- Decision: treat `/home/mcnal/.local/share/oracle` as the canonical Oracle home and `/home/mcnal/.local/share/oracle/browser-profile` as the canonical browser profile, with `~/.oracle/browser-profile` retained only as a compatibility symlink.
  Rationale: that path already matches the newer wrapper behavior, follows the XDG-style local share convention better than `~/.oracle`, and avoids a split between two real Chromium user-data roots.
  Date/Author: 2026-03-17 / Codex
- Decision: patch the installed Oracle package directly in addition to local wrappers.
  Rationale: wrapper-only fixes kept getting bypassed by direct Oracle invocations and upstream defaults; owning the local install is the smallest reliable way to make the behavior consistent.
  Date/Author: 2026-03-17 / Codex
- Decision: stop forcing DevTools port `9222` in the local wrappers unless the operator explicitly exports `ORACLE_BROWSER_PORT`.
  Rationale: pinned port behavior was a recurring source of stray-browser attachment and connection confusion, while Oracle itself can choose a free port safely when none is requested.
  Date/Author: 2026-03-17 / Codex
- Decision: make the machine-wide default browser path an auto launcher rather than either the fixed visible launcher or the fixed xvfb launcher.
  Rationale: the user wants Oracle to work anywhere on this machine, and the two successful/failed smoke tests proved that display availability differs by environment.
  Date/Author: 2026-03-17 / Codex
- Decision: keep the repo-side benchmark uploader's canonical-profile helper logic for now even though the filesystem and local Oracle install are now consolidated.
  Rationale: it is now mostly belt-and-suspenders, but it does no harm and keeps benchmark logs explicit about which browser profile was used during detached launches.
  Date/Author: 2026-03-17 / Codex

## Outcomes & Retrospective

The local Oracle stack is now substantially more coherent. The installed Oracle package defaults, the user-facing `oracle` help text, the bridge snippet output, and the wrapper/login helper scripts now all agree that the canonical Oracle home is `/home/mcnal/.local/share/oracle` and the canonical browser profile is `/home/mcnal/.local/share/oracle/browser-profile`. The legacy `~/.oracle/browser-profile` path remains only as a compatibility symlink.

The biggest reliability win beyond profile consolidation was eliminating two stale defaults from the owned local Oracle runtime: browser model-picker `select` is no longer the baseline, and local wrappers no longer pin DevTools port `9222` unless asked. The final machine-wide default is now an auto launcher instead of a fixed visible or fixed xvfb choice. That matches the actual environments on this machine better than either extreme.

Validation now includes real browser smoke tests, not just help/config inspection. The visible-browser smoke test failed for a real environmental reason (`$DISPLAY` missing in the agent shell), the xvfb-backed smoke test then completed successfully with `gpt-5.2-instant`, and a fresh generic `oracle` smoke test completed successfully through the new auto launcher with launcher log `MODE=xvfb`. That proves the owned Oracle runtime can execute a real browser run end to end with the canonical profile and the machine-wide auto-launch rule.

## Context and Orientation

There are two layers in play.

The repository-owned layer is `cookimport/bench/oracle_upload.py` plus `cookimport/cli.py`, which launches Oracle after benchmarks and prints user-facing status. That layer already moved to visible Chromium, detached launches, and short wrap-up output.

The machine-owned layer is the actual Oracle command stack under `/home/mcnal/.local/bin` and `/home/mcnal/.nvm/versions/node/v20.19.6/lib/node_modules/@steipete/oracle`. The wrapper scripts decide how `oracle` and helper commands invoke the installed CLI. The installed package decides browser defaults such as Oracle home, manual-login profile directory, and model-picker strategy.

The current bug class exists because those machine-owned pieces still carry conflicting defaults. This plan resolves that by making the machine-owned layer agree on one Oracle home/profile and one browser interaction strategy.

## Plan of Work

First, patch the installed Oracle package in place. `dist/src/oracleHome.js` should default to `/home/mcnal/.local/share/oracle` instead of `~/.oracle`. `dist/src/browser/config.js` should derive the manual-login profile path from the Oracle home helper rather than hardcoding `~/.oracle/browser-profile`. `dist/src/browser/constants.js` should stop defaulting browser runs to brittle picker automation; the browser default should prefer the current visible/manual model instead of trying to select `GPT-5.2 Pro`. `dist/src/cli/bridge/claudeConfig.js` should emit the same canonical Oracle home/profile in its snippet defaults.

Second, reduce wrapper drift. Add one small shared shell helper under `/home/mcnal/.local/bin` that resolves and exports the canonical Oracle home/profile paths, remapping any legacy `.oracle` inputs back to the canonical tree. Then source that helper from `oracle-browser-safe`, `oracle-browser-headless`, and `chromium-chatgpt-login` so they stop re-implementing the same path logic differently.

Third, document the owned runtime. Update the benchmark docs to mention the canonical Oracle browser profile and the machine-wide auto browser launcher, and add a short task note summarizing that the local Oracle install is now intentionally patched.
