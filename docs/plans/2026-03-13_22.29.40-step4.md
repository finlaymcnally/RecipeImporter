This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the repository should contain a benchmark-runnable alternative to the current three-pass path: a new pipeline id that keeps pass1-style boundary selection but replaces the current pass2 -> pass3 seam with one source-grounded assembly-and-repair stage. The user-visible proof is that the same benchmark book can be run through both the repaired `codex-farm-3pass-v1` control path and the new merged-stage prototype, and the outputs can be compared directly on quality, cost, and failure transparency.

This plan is a prototype plan, not a promotion plan. It must keep the current path alive as the control and prove the merged design on the same benchmark slice before any default-switch discussion.

## Surprises & Discoveries

- Observation: the merged-stage argument is not “LLM everywhere.” It is “stop paying seam tax where the seam is now causing more loss than savings.”
  Evidence: the March 13 merged-stage report repeatedly emphasizes that pass2 and pass3 preserve the right philosophy but may over-decompose the task for the current model and transport stack.

- Observation: the prototype still depends on clean source authority and explicit invariants.
  Evidence: the merged-stage report insists on transport verification, authority ordering, and fail-fast guards before expensive repair calls. The prototype is not a replacement for observability.

- Observation: parallel final artifacts are a drift hazard.
  Evidence: the March 13 merged-stage report argues for one canonical internal recipe object with deterministic derivation of schema.org and any other export shapes afterward.

- Observation: the real blocker for a second recipe pipeline was not the orchestrator alone; several normalization and approval layers still assumed `codex-farm-3pass-v1` was the only non-`off` recipe Codex pipeline.
  Evidence: implementation needed coordinated edits in `cookimport/config/run_settings.py`, `cookimport/config/codex_decision.py`, `cookimport/cli.py`, and `cookimport/labelstudio/ingest.py` before the new prototype could even reach the orchestrator.

- Observation: the cheapest additive prototype was to reuse the existing pass2 artifact slot while changing its payload contract, then skip pass3 entirely for prototype rows.
  Evidence: `codex-farm-2stage-repair-v1` now writes merged-stage inputs and outputs under the existing `pass2_schemaorg/` directories but records `recipe.merged-repair.compact.v1` in manifests and writes separate `merged_repair_audit/` artifacts for review.

## Decision Log

- Decision: keep pass1 or equivalent boundary selection separate in the first prototype.
  Rationale: boundary selection is still needed for provenance and cost control. The seam under evaluation is the current pass2 -> pass3 handoff, not the entire pipeline.
  Date/Author: 2026-03-13 / Codex

- Decision: require one canonical internal recipe object in the prototype.
  Rationale: maintaining multiple “final” truths is part of what makes the current design hard to reason about. The prototype must remove that ambiguity to test the architecture honestly.
  Date/Author: 2026-03-13 / Codex

- Decision: keep the prototype behind a new pipeline id and forbid default promotion inside this plan.
  Rationale: this is an A/B architecture evaluation. Replacing the control path too early would destroy the baseline needed to judge the prototype.
  Date/Author: 2026-03-13 / Codex

- Decision: use one narrow canonical recipe object (`title`, `ingredients`, `steps`, plus optional `description` and `recipeYield`) for the first merged-stage contract.
  Rationale: that was enough to derive the repo’s current downstream shapes deterministically without expanding the prompt contract into another ambiguous “almost schema.org, almost final draft” hybrid.
  Date/Author: 2026-03-13 / Codex

## Context and Orientation

The key files for this plan are the active CodexFarm LLM pipeline surfaces in `cookimport/llm/`, the run-settings and pipeline-selection surfaces described in `docs/10-llm/10-llm_README.md`, and the benchmark commands that already exercise `llm_recipe_pipeline=codex-farm-3pass-v1`.

In this repository, a merged stage means one LLM-backed assembly-and-repair call that receives the authoritative source span plus any current draft hints and emits one canonical recipe object, optional ingredient-to-step mapping, and machine-readable warnings. Source-grounded means the authoritative source span outranks any earlier draft or mapping artifact. Existing drafts are hints, not law.

Canonical recipe object means one internal representation that the rest of the system treats as the truth. Schema.org and any other export shape must be derived deterministically from that object rather than being maintained as separate competing final outputs.

This prototype should be judged against the same March 6 benchmark root at `data/golden/benchmark-vs-golden/2026-03-06_15.22.11`, using the same three weak books: `saltfatacidheatcutdown`, `seaandsmokecutdown`, and `thefoodlabcutdown`.

## Plan of Work

Begin by adding a new pipeline id, for example `codex-farm-2stage-repair-v1`, in the run-settings and CLI normalization surfaces documented in `docs/10-llm/10-llm_README.md`. Keep the current `codex-farm-3pass-v1` path unchanged so benchmark tooling can run both candidates side by side.

Next, define the canonical merged-stage contract in `cookimport/llm/codex_farm_contracts.py` or a neighboring module. The input should include the authoritative source span or normalized source rows, current extracted ingredients and instructions if they exist, current schema.org draft if it exists, current final recipe object if one exists, and any useful warnings or provenance hints. The output should include one canonical recipe object, optional ingredient-to-step mapping, warnings, and enough provenance metadata to explain omissions or degraded recovery.

After the contract exists, implement the merged assembly-and-repair stage in the orchestrator or a new helper module. The stage should obey explicit authority ordering: source span first, then explicit section structure inferred from source, then existing drafts as hints, and existing mappings only as hints. Do not ask the stage to maintain multiple final artifacts. Emit one canonical object and derive schema.org or any other export views afterward.

Add preflight guards so the merged stage does not run on obviously nonsensical input, such as zero source rows, zero ingredients and zero instructions, or heavily corrupted text. Those rows should degrade to a named recovery path, not run through the expensive merged call under false pretenses.

Finally, add auditable artifacts for the prototype. Each merged-stage run should write enough data to explain the source span used, the authority-ranked input summary, the canonical output summary, mapping presence or omission, and warning codes. Keep the artifacts concise but sufficient for benchmark investigation.

Implementation note: this now lives in `raw/llm/<workbook>/merged_repair_audit/*.json`, and recipe-level manifest rows embed the same audit payload for quick review.

## Validation and Acceptance

Acceptance is behavior-first.

First, tests must prove that the prototype emits one canonical internal recipe object, derives downstream export views deterministically, preserves explicit warning semantics when mapping is partial or omitted, and can recover from degraded source payloads without inventing unsupported data.

Second, the prototype must be benchmark-runnable under a new pipeline id. It is not enough to have scaffolding in code; the CLI and benchmark surface must actually execute it.

Third, the benchmark comparison must answer a real question. On the shared three-book slice, the merged-stage candidate must either outperform the repaired current path on the same failure families or fail clearly enough that the project can keep the current architecture without ambiguity. If the prototype is merely different, this plan has not yet delivered its purpose.

## Idempotence and Recovery

This plan is additive. The new pipeline id must coexist with `codex-farm-3pass-v1`, and benchmark reruns must use fresh timestamped output roots. If the prototype underperforms, the recovery path is simple: keep it behind its own pipeline id, leave it experimental, and backport only the clearly winning ideas. Do not overwrite the March 6 baseline root or remove the repaired current-path control.
