# Knowledge Grouping Batching And Partial Repair

## Purpose / Big Picture

After this change, the nonrecipe knowledge grouping step should behave more like the current line-role step. A grouping response must cover the full set of rows it was asked to review, but a mostly-correct response should not throw away every accepted row in the shard. Grouping should split oversized work into smaller batches, keep validated rows, and send repair followups containing only the unresolved rows. A human should be able to rerun a benchmark like `2026-04-10_18.22.20` and see that oversized grouping packets are no longer sent as one giant call, while partial grouping answers recover through narrow repair instead of invalidating hundreds of rows.

## Surprises & Discoveries

- Observation: The configured grouping limits are currently observability-only metadata, not real planning constraints.
  Evidence: `cookimport/llm/knowledge_stage/task_file_contracts.py` records `max_units_per_batch` and `max_evidence_chars_per_batch` in `grouping_batch`, but `build_knowledge_grouping_task_files(...)` still returns a single task file containing all units.
- Observation: The failed `2026-04-10_18.22.20` grouping shards were only partially wrong, not structurally unreadable.
  Evidence: `task_status.jsonl` shows `ks0000`, `ks0001`, and `ks0007` had 60/61, 61/63, and 65/68 changed units accepted respectively, with only a few proposal-candidate rows failing `group_key`, `topic_label`, and `proposal_decision`.
- Observation: The shard-wide fallout was much larger than the direct model mistake.
  Evidence: those three invalid shards owned 101, 101, and 100 candidate rows, which exactly explains the 302 unresolved candidate lines seen in benchmark telemetry.
- Observation: The line-role runtime already demonstrates the desired authority pattern.
  Evidence: line-role keeps accepted rows, narrows repair packets to unresolved rows only, and only fails the still-unresolved subset after bounded repair exhaustion.
- Observation: the same-session taskfile path had the batching hooks in state already, but the grouping transition code never consumed them.
  Evidence: `pending_grouping_unit_batches` and grouping batch counters were already present in `knowledge_same_session_handoff.py`, while `transition_knowledge_grouping_task_file(...)` still always finalized after one successful grouping pass.
- Observation: grouping quality was also being hurt by prompt/catalog vocabulary drift, not just runtime batching.
  Evidence: the prompt nudged the model toward labels like `diagnostics`, while deterministic validation only accepts real catalog category keys such as `techniques`, `ingredients`, `equipment`, and `storage`.

## Decision Log

- Decision: Fix this in the knowledge grouping planner and runtime rather than by loosening validation.
  Rationale: the repo is correct to require one answer per requested grouping row, especially for proposal-candidate rows that must be explicitly approved or rejected. The bug is that batching and repair semantics are too weak, not that the contract is too strict.
  Date/Author: 2026-04-10 / Codex
- Decision: Make grouping adopt line-role-style partial authority semantics.
  Rationale: accepted grouping answers are already represented as per-unit validated answers in metadata. Preserving them and repairing only unresolved units reduces wasted work and avoids shard-wide dropouts from 1-3 bad rows.
  Date/Author: 2026-04-10 / Codex
- Decision: Keep grouping repair in the existing prepared workspace and narrow each repair packet to current unresolved units only.
  Rationale: this matches the current knowledge structured-session repair shape and minimizes new transport risk while still making the repair surface small and deterministic.
  Date/Author: 2026-04-10 / Codex
- Decision: allow kept `knowledge` rows to propose one or more new tags during grouping, but keep category keys constrained to the checked-in ontology.
  Rationale: this gives the model a fuzzy escape hatch when the existing tag set is semantically wrong without weakening deterministic validation or downstream grouping analytics. Allowing multiple tags also avoids unnecessary run failures when the model wants to emit more than one plausible retrieval handle for the same row.
  Date/Author: 2026-04-10 / Codex

## Outcomes & Retrospective

Knowledge grouping is now honest about both planning and authority. Large grouping queues are split before launch, structured classification/grouping packets carry exact-length `rows` schemas, accepted grouping answers survive across attempts, and grouping repair follows up only on unresolved rows. The same-session taskfile path now advances across multiple grouping batches instead of assuming a single shard-sized grouping turn. Grouping also now sees the real ontology and may approve one or more new tags for an already-kept `knowledge` row when the existing catalog fit is poor.

The main remaining risk to watch is quality rather than contract correctness: smaller grouping batches reduce failure blast radius, but they could still slightly change topic coherence versus one huge shard-wide grouping pass. The next benchmark rerun should confirm both recovery rate and final semantic quality.

## Context and Orientation

The relevant runtime lives in `cookimport/llm/knowledge_stage/`. In this repository, “classification” means deciding whether a nonrecipe row is `knowledge`, `proposal_candidate`, or `other`. “Grouping” means taking the kept `knowledge` and `proposal_candidate` rows and assigning each one a `group_key`, a human-readable `topic_label`, and a proposal decision. The knowledge runtime is split into a contract layer and a worker orchestration layer.

The contract layer is `cookimport/llm/knowledge_stage/task_file_contracts.py`. It builds classification task files, builds grouping task files, validates edited grouping answers, and constructs repair task files when a worker answer is incomplete or invalid. The live worker orchestration is `cookimport/llm/knowledge_stage/workspace_run.py`. That file builds grouping packets from task files, sends them through the direct Codex runner, validates the result, records artifacts, and currently issues structured repair followups.

The benchmark evidence for the failing run lives under `data/output/2026-04-10_18.22.20/.../nonrecipe_finalize/`. The important files are `task_manifest.jsonl`, which records packet sizes and shard membership; `task_status.jsonl`, which records validated answers and failed units; `knowledge_stage_summary.json`, which summarizes the stage-level fallout; and `failures.json`, which records the invalid shards. In that run, three grouping shards (`ks0000`, `ks0001`, and `ks0007`) came back almost complete but left a few proposal-candidate rows blank. The runtime then failed those entire shards closed.

The reference behavior to copy lives in `cookimport/parsing/canonical_line_roles/runtime_workers.py` and the focused tests in `tests/parsing/test_canonical_line_roles_runtime.py`. That runtime enforces exact response coverage, keeps accepted rows, and narrows repair packets to the unresolved subset only.

After each repair response, revalidate against the full original batch task file so the runtime can tell whether the batch is now complete. The final `proposal_payload` and `proposal_metadata` should come from the merged accepted answers across all grouping batches plus any successful repairs. ## Validation and Acceptance

Acceptance means all of the following are true:

1. Grouping task construction honors the configured `knowledge_group_task_max_units` and `knowledge_group_task_max_evidence_chars` limits instead of always emitting one large batch.
2. A partial grouping response that leaves only a few rows unresolved produces a repair packet containing only those unresolved rows.
3. Accepted grouping answers from the initial turn remain authoritative after repair; the runtime no longer throws away every accepted row in the batch or shard just because a few rows were bad.
4. If repair resolves the missing rows, the final shard output is validated and promoted.
5. The focused knowledge grouping and runtime tests pass in the project `.venv`.

## Idempotence and Recovery

This change is safe to iterate on. Grouping task files and structured-session artifacts are runtime outputs, so regenerating them is harmless. If a patch breaks grouping repair, the quickest recovery path is to compare `task_status.jsonl` metadata before and after the change: the fixed runtime should show `validated_answers_by_unit_id` surviving across grouping repairs and `failed_unit_ids` shrinking to only the unresolved subset. If batching logic regresses unexpectedly, the fallback diagnostic is to inspect `grouping_batch` metadata in generated task files and confirm `total_batches > 1` when limits are exceeded.

    tuple[list[dict[str, Any]], dict[str, str], list[list[str]]]

where the first element contains one task file per grouping batch, the second maps `unit_id` to parent shard id, and the third contains the batch-local ordered `unit_id` lists. Each task file must include accurate `grouping_batch` metadata with batch index, total batch count, current unit count, and configured limits.

In `cookimport/llm/knowledge_stage/workspace_run.py`, the grouping loop must preserve:

    grouping_answers_by_unit_id
    grouping_batch_answers_by_unit_id

as merged accepted answers, and `build_repair_task_file(...)` must receive only the current failed `unit_id` values for the current grouping batch. The runtime must revalidate the merged answers against the full original batch task file before deciding whether the batch still blocks progress.
