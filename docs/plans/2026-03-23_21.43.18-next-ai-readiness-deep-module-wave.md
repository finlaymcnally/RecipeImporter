# Complete the next AI-readiness deep-module wave

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

RecipeImport is now in a better place than it was a day ago: the public roots are thin, the old `*_runtime.py` facades are gone, and the active implementation lives in more honest packages. That was necessary, but it did not finish the job described in [docs/reports/AI-codebase.md](/home/mcnal/projects/recipeimport/docs/reports/AI-codebase.md). The next barrier is not public entrypoints; it is the remaining giant private coordinators and over-broad contracts that still force a newcomer to open thousands of lines before making one safe change.

After this plan is complete, a contributor should be able to approach the main remaining problem domains through small, explicit public seams instead of giant read-around files. The visible proof is that `cookimport/cli_support.py`, `cookimport/parsing/canonical_line_roles.py`, `cookimport/llm/codex_farm_knowledge_orchestrator.py`, `cookimport/analytics/dashboard_renderers/templates.py`, and `cookimport/config/run_settings.py` are no longer the first and only places to understand those domains; focused modules, package-local notes, and boundary tests exist; domain test suites still pass; and the repo contains at least one enforced guard against collapsing back into cross-domain import sprawl.

This plan is intentionally broad. The user has already chosen the project philosophy: coding is cheap, structural clarity is valuable, and this repo should be optimized for safe local change by fresh humans and fresh AI agents. The work below therefore treats architecture as product, not cleanup.

## Context and Orientation

This plan assumes no prior memory and defines the main terms it uses.

A **deep module** in this repo means one capability package or module that exposes a small, explicit interface but hides substantial internal machinery behind it. A **public seam** means the stable function, type, command, or package entrypoint that another domain should use instead of reaching into internals. A **boundary test** means a test that locks down behavior or import direction at that seam so later edits cannot casually bypass it.

The current repo already has thin public roots in several places:

- [cookimport/cli.py](/home/mcnal/projects/recipeimport/cookimport/cli.py) is now the CLI composition root.
- [cookimport/labelstudio/ingest.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest.py) is now a compatibility facade.

- [cookimport/analytics/dashboard_render.py](/home/mcnal/projects/recipeimport/cookimport/analytics/dashboard_render.py) is now a public facade.

## Milestones

### Milestone 1: Split `cookimport/cli_support.py` into capability-owned support modules

At the end of this milestone, the private CLI support surface will still exist, but it will no longer be one giant file. Instead, the CLI package will have one small root support module and a set of capability-owned support modules such as `cli_support/stage.py`, `cli_support/labelstudio.py`, `cli_support/bench.py`, `cli_support/analytics.py`, `cli_support/interactive.py`, and one narrower cross-cutting module for shared progress/rendering helpers. The visible outcome is that a contributor changing one command family’s support logic can open one module instead of `23,000` lines of unrelated helpers.

Acceptance is that `cookimport` command behavior is unchanged, command-domain tests still pass, and the remaining root `cli_support` surface is clearly described as a package-level seam rather than the implementation owner of every command family.

### Milestone 2: Split canonical line-role parsing into a true package with small seams

At the end of this milestone, canonical line-role work will be discoverable through a package-oriented layout rather than one giant parser file. The package should expose a small top-level API for “plan line-role runtime,” “run or resume line-role stage,” and “validate/promote packet outputs,” while moving policy families, packet building, hint assembly, validation, and artifact writing into owned modules.

Acceptance is that line-role benchmark and parsing tests still pass, prompt-preview or runtime artifacts still land in the same observable places unless explicitly renamed in docs/tests, and a new contributor can start from package-local notes and the top-level line-role entry module before diving into internals.

### Milestone 3: Split knowledge orchestration into bounded runtime, planning, recovery, and reporting modules

At the end of this milestone, the optional knowledge stage will still work the same from the operator’s perspective, but the orchestration layer will no longer be one broad coordinator file. Planning, worker-runtime control, task recovery/repair logic, validation/promotion, and summary/reporting should each have a clear home.

Acceptance is that knowledge-stage tests still pass, run artifacts remain stable or are explicitly migrated with docs/tests updated, and the orchestrator root becomes a small seam that names the real owned submodules.

### Milestone 4: Split dashboard renderer internals so analytics has real page/component ownership

At the end of this milestone, analytics rendering should be navigable by page and component family instead of one huge templates file. The dashboard renderer package should make it obvious where run pages, all-method pages, diagnostics cards, filters, and shared layout fragments live.

Acceptance is that analytics/dashboard tests still pass, generated dashboard output remains equivalent or intentionally improved, and the renderer package contains package-local notes describing the ownership map.

### Milestone 5: Shrink `RunSettings` into a smaller product surface with explicit internal layers

At the end of this milestone, the repo should no longer teach one giant settings object as if every field were the product. Product-facing choices, internal persisted knobs, benchmark/debug overrides, and derived/effective fields should have explicit homes. The public contract module already exists; this milestone uses it to make the live surface honest.

Acceptance is that stage/benchmark/config tests still pass, manifests and adapters remain stable where intended, and docs can explain the settings surface without teaching a 70-plus-field object as the main mental model.

### Milestone 6: Add boundary enforcement and package-local notes

At the end of this milestone, the new seams should be protected. This can be import-lint tests, package-boundary assertions, narrow “allowed imports” checks, or a small repo-owned guard tool that fails in CI/test runs when one capability reaches into another capability’s internals incorrectly. Package-local README notes should exist wherever a newcomer now has a new package to open first.

Acceptance is that at least one automated boundary check exists for the new deep-module packages, it fails if a forbidden import pattern is introduced, and the docs point readers toward the new public seams instead of the internal files.

## Target Architecture

This section is prescriptive. The exact internal file names may change during implementation if a better shape emerges, but the ownership model must remain.

The target package layout is:

- `cookimport/cli.py` as the public Typer composition root only.
- `cookimport/cli_support/__init__.py` as the private package facade.
- `cookimport/cli_support/stage.py`
- `cookimport/cli_support/labelstudio.py`

- `cookimport/cli_support/analytics.py`

### Analytics renderer target shape

The renderer package should make page ownership obvious. A plausible target is:

- `cookimport/analytics/dashboard_renderers/public.py` or a small package facade
- `cookimport/analytics/dashboard_renderers/layout.py`
- `cookimport/analytics/dashboard_renderers/diagnostics.py`

- `cookimport/analytics/dashboard_renderers/all_method_pages.py`
- `cookimport/analytics/dashboard_renderers/components/*.py` if needed
- `cookimport/analytics/dashboard_renderers/templates/` only if there are truly reusable template fragments that merit a subpackage

The explicit design rule is that “templates” should not become a new junk drawer. Page/component ownership comes first.

### Run-settings target shape

The target settings architecture should distinguish four things clearly:

1. Product-facing run choices.
2. Internal persisted knobs that remain useful for benchmarking or archaeology.
3. Derived/effective values that are not user input.
4. Contract/adapter metadata used to serialize and normalize the surface.

The likely file shape is:

- `cookimport/config/run_settings.py` as the smallest honest user-facing model or facade that still needs that historic import path.
- `cookimport/config/run_settings_contracts.py` as the contract owner.
- `cookimport/config/run_settings_product.py` for the product-facing choices, if the split is large enough to merit it.
- `cookimport/config/run_settings_internal.py` for internal-only persisted knobs, if they remain necessary.
- `cookimport/config/run_settings_builders.py` or equivalent for construction/normalization helpers that do not belong on the model itself.

The final result must make it harder to confuse “stored internal detail” with “real user choice.”

### Boundary enforcement target shape

By the end of the campaign, the repo should contain at least one automated guard for these new packages. The mechanism can be simple. Examples that fit this repo are:

- a small test file that defines allowed import prefixes for certain packages and fails when a module reaches into forbidden internals,
- a repo-owned script run by tests that scans imports with `ast` and asserts package boundary rules,
- or package-specific tests that assert only public modules are imported by neighboring domains.

What matters is not elegance. What matters is that the boundaries become enforceable.

## Plan of Work

Start with the CLI support split because it has the highest current supervision cost and the clearest domain map already exposed by the command package. The command-family modules already tell us what the private support groupings should be. The work here is to cut the private helper surface along the same product seams as the public commands: stage, Label Studio, benchmark, analytics/compare-control, and interactive/settings. Keep shared progress or formatting helpers in one small cross-cutting home only if they are genuinely shared. The key anti-goal is replacing `cli_support.py` with a new `cli_support/helpers.py` blob.

Once the CLI support split lands, move to canonical line-role. That file needs a real package because it has grown into a subsystem, not a helper. Start by extracting contracts and artifact/path ownership, because those are the safest, most legible seams. Then split packet planning and policy, then runtime/recovery, and only after that move prompt assembly or fallback logic. Keep one small top-level entry module that callers can import from, and update parsing docs plus any line-role-specific package note so a newcomer knows where to start.

After line-role, split the knowledge orchestrator using the same staged pattern. First extract contracts and summary/reporting so the current orchestrator loses passive data-shape clutter. Then extract planning and worker assignment. Then extract recovery/repair logic. Then extract promotion/summary rollups. The orchestrator root should end as a short narrative of the stage rather than a place where all details live. The reason to place this after line-role is that the knowledge stage shares the same general runtime style, but its reporting and recovery surface is more complex; the line-role split will teach what package patterns actually work in this repo.

Once the two runtime-heavy domains are split, tackle analytics renderer internals. The recent public split already established that [dashboard_render.py](/home/mcnal/projects/recipeimport/cookimport/analytics/dashboard_render.py) is only a facade. Now finish the job inside the renderer package by splitting the giant templates file along page/component ownership. This is lower risk than line-role or knowledge because the main observable output is static HTML and tests already cover much of it, but it is still worth doing because analytics is one of the main fresh-agent entrypoints for benchmark history work.

Then do the run-settings decomposition. This milestone should use the run-settings audit as a source of truth and avoid accidental churn. Do not try to delete every internal setting immediately if it is still needed for manifests, benchmarks, or historical replay. Instead, separate the product-facing surface from the internal persisted surface and make the current public seam honest. This is the place to keep engineering taste high: fewer, clearer models are better than one giant settings object and better than ten tiny objects with no ownership story.

Finish by adding boundary enforcement and package-local notes. The implementation order matters here. If import rules land too early, they will mostly be waived while the split is in flight. Once the new packages exist, add the enforcement rules for them and write the small README notes that tell future contributors where to start.

Throughout the campaign, use the same burn-the-boats rule as the recent purge: when a new owner is proven and tests pass, remove the old implementation body rather than keeping a broad mirror layer around.

## Validation and Acceptance

This plan is complete only when all of the following are true.

First, the biggest remaining domains are navigable through real package seams. A newcomer should be able to start with one small entry module or package note for CLI support, canonical line-role, knowledge-stage orchestration, analytics rendering, and run-settings ownership. “Start by opening the giant file” is no longer acceptable as the normal path.

Second, the public or package-root interfaces are smaller than the internals they hide. If a split produces five new modules but still requires most callers to import ten internal helpers directly, the split failed. The whole point is progressive disclosure.

Third, the observable runtime behavior still works. CLI commands, stage flows, Label Studio prediction/import flows, benchmark flows, analytics dashboard generation, line-role and knowledge stages, and run-settings persistence/tests must still pass their domain validations.

Fourth, the docs teach the new ownership model honestly. The relevant subsystem docs and package-local notes must point readers at the real public seams and the new owning packages instead of stale giant files.

Fifth, at least one automated boundary guard exists and protects the new packages from obvious cross-domain drift.

Sixth, the old bottleneck files are either gone, drastically smaller, or demoted to tiny facades/package roots. The campaign does not count as complete if the code simply moved from `foo.py` to `foo_helpers.py` without changing how a reader discovers the architecture.

## Idempotence and Recovery

This work should be done incrementally and safely. Each milestone is intentionally independent enough to validate before moving on to the next. If a split becomes too broad or starts breaking unrelated domains, stop the split at the nearest honest boundary and record that decision here rather than shoving more code across the line. The right fallback is a smaller but truthful package seam, not a restored giant file.

All moves should be additive first and subtractive second. Create the new modules, route one caller or one domain family through them, run the narrowest useful tests, then remove the old implementation body. This keeps the work repeatable and makes it easy to retry a failed pass without resetting the whole campaign.

If one proposed target turns out not to deserve a package of its own, record that in `Decision Log` and leave it as a deep module with a small facade rather than forcing a package split for aesthetics. The standard is better architecture, not more files.

## Interfaces and Dependencies

At the end of this plan, the following kinds of interfaces must exist.

For CLI support, each command family should have one private support module with a narrow exported surface used by [cookimport/cli_commands/](/home/mcnal/projects/recipeimport/cookimport/cli_commands/). Shared progress and settings helpers may remain shared, but they must live in modules whose names describe that shared responsibility honestly.

For canonical line-role, there must be a small package-level entry surface that callers can use without importing policy internals directly. That surface should describe planning, runtime execution or resume, and validation/promotion in stable names.

For knowledge-stage orchestration, there must be a small orchestrator seam backed by explicit planning, runtime, recovery, promotion, and reporting modules. The reporting/summary model should not be buried inside the runtime loop.

For analytics rendering, there must be a clear division between public page assembly and lower-level HTML/component helpers. The word “templates” should only mean reusable HTML/template fragments, not “all rendering logic that did not fit elsewhere.”

For run settings, the stable public contract module and the user-facing model/facade must remain separated from internal builder or benchmark-only machinery. The public model should be small enough that docs can teach it without apologizing for the rest of the persistence surface.

For boundary enforcement, there must be at least one repo-owned automated rule that can fail when a forbidden internal import pattern appears. This rule must live in the repo, be runnable locally, and be documented in the relevant package note or subsystem doc.
