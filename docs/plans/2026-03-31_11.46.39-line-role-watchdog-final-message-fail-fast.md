## Purpose / Big Picture

After this change, canonical line-role workers will stop for the real reason and only after one real recovery chance. Ordinary workspace drift such as using repo-owned helpers, taking longer than sibling workers, or producing warning-only telemetry will remain warning-only. The only new watchdog-triggered hard stop is the narrow case where a workspace worker has already emitted a final agent message but still has not produced the required `out/<shard_id>.json` file after a short grace window.

The grace window is not passive. If the output file appears during that window, the shard succeeds normally. If the file is still missing when the grace window expires, deterministic repo code will terminate the current worker session with an explicit reason, inspect the preserved workspace using the line-role doctor/status tools, and decide whether one fresh worker session against the same workspace root is justified. If that rescue session succeeds, the shard completes normally. If the workspace is not recoverable, the budget is already spent, or the rescue session still does not produce valid output, the runtime fails closed with explicit durable metadata instead of letting the original worker grind until context exhaustion.

The user-visible proof is concrete. Focused tests must show the watchdog entering the grace-window state only after a final agent message plus missing output, the runtime collecting a deterministic recovery assessment, one bounded fresh-session rescue being launched only when allowed, and the shard either succeeding from that rescue or failing closed with a precise missing-output reason after the rescue budget is exhausted.

## Outcomes & Retrospective

This plan is now implemented. Canonical line-role workspace workers remain warning-first for ordinary drift, but once a worker emits a final message without producing the required `out/<shard_id>.json`, runtime now opens a 2-second grace window, kills the session with `workspace_final_message_missing_output` if the file is still absent, writes a deterministic recovery assessment from `--status` / `--doctor`, and may spend the one shared fresh-session retry budget on a preserved-workspace rescue.

The focused tests and the broader parsing slice both passed after implementation. The main design lesson held: a grace window alone only solves delayed file writes. Real recovery still needed a smaller follow-up prompt plus a deterministic gate that can say “no” when answers are missing, repair answers are still blank, or the shared retry budget is already spent.

## Context And Orientation

Canonical line-role is the LLM-assisted parsing stage that labels canonical book lines with values such as `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `RECIPE_NOTES`, `NONRECIPE_CANDIDATE`, and `NONRECIPE_EXCLUDE`. The live stage runtime is implemented in [cookimport/parsing/canonical_line_roles/runtime.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles/runtime.py). The worker-facing same-session helper lives in [cookimport/parsing/canonical_line_roles/same_session_handoff.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles/same_session_handoff.py).

The relevant execution mode is the workspace-worker path. In plain language, deterministic repo code writes one visible `task.json` into a sterile worker workspace. The worker is supposed to edit only the allowed answer fields in that file and then run a repo-owned same-session handoff command. Deterministic code validates the edit, rewrites `task.json` into repair mode if needed, writes final `out/<shard_id>.json` files when the answers are valid, and supervises the worker through snapshot callbacks while the worker session is still alive.

The current line-role runtime already has three recovery-related mechanisms that matter here:

1. a main workspace-worker session for the happy path;
2. a generic fresh-session retry gate for clean-but-incomplete worker exits;
3. a shard-level inline watchdog retry path for older strict-JSON watchdog reasons.

This plan changes how the first and second mechanisms interact for one narrow case. It does not change the direct-exec transport into a multi-turn supervisor and it does not broaden the shard-level inline retry path.

The production flow required by this plan is:

1. let the live worker continue briefly if it has emitted a final message but the expected output file is still missing;
2. if the file appears during that grace window, accept the normal success path;
3. if the file is still missing when the grace window expires, terminate the worker with `workspace_final_message_missing_output`;
4. run a deterministic recovery assessment against the preserved workspace using the line-role `--status` and `--doctor` logic;
5. if that assessment says one fresh session is worthwhile and the shared stage retry budget is still unused, launch exactly one fresh worker session against the same workspace root with a short rescue prompt;
6. if the rescue session produces valid outputs, accept success; otherwise fail closed with durable rescue metadata.

The failed benchmark artifact under `data/output/2026-03-31_02.57.01/.../worker-005/live_status.json` is the motivating example. That worker had emitted a final message, still had no output file, and then burned more context until failure. This plan exists to replace that shape with a bounded and explicit sequence.

## Plan Of Work

First, treat the prerequisite tools plan as required repo context, not optional inspiration. Before implementing this plan, confirm that canonical line-role workspaces already expose repo-owned `--status` and `--doctor` entrypoints from the workspace root, that `task.json` includes stage-correct helper commands and answer-shape guidance, and that the stage runtime can relaunch one worker session against an existing workspace root. If any of those pieces landed with slightly different field names, add a line-role-local adapter in the runtime; do not make watchdog or recovery code parse human-oriented prose.

Second, add a small deterministic recovery-assessment helper in the line-role runtime. This helper must read the preserved workspace state after the main session has ended and normalize the line-role doctor/status payloads into a stable internal structure. The structure must include whether outputs are already present, whether same-session completion already happened, the diagnosis code, the recommended command, a short resume summary, whether the fresh-session retry budget is still available, and the final boolean `recoverable_by_fresh_session`.

Third, define the rescueability mapping explicitly in code and mirror it in this plan. For line-role, use the following deterministic mapping after reading the doctor/status payload plus current state:

- `completed`: not recoverable, because the same-session helper already completed and the runtime should trust outputs or fail based on validation rather than relaunching a worker.
- `answers_present_helper_not_run`: recoverable, because saved answers exist and the likely missing step is running the same-session helper.
- `ready_for_validation`: recoverable, for the same reason.
- `repair_ready_helper_not_run`: recoverable, because the workspace is already in repair mode with saved answers and the helper appears to be the missing step.
- `awaiting_answers`: not recoverable, because the worker declared completion without actually saving answers worth resuming.
- `repair_answers_missing`: not recoverable, because repair mode is active but the required corrected answers are still absent.
- any unknown diagnosis code: not recoverable unless the adapter can prove outputs are already present and valid without a new model session.

The adapter must also force `recoverable_by_fresh_session=False` whenever any of these hard guards are true: the shared stage retry budget is already spent, the same-session state file is missing or unreadable, `task.json` is missing or unreadable, a hard boundary failure occurred, or valid output files are already present and should instead flow through normal validation.

Fourth, update `_build_strict_json_watchdog_callback(...)` so it explicitly tracks the new grace-window state. Add a new constant named `_LINE_ROLE_FINAL_MESSAGE_MISSING_OUTPUT_GRACE_SECONDS` with default `2.0`. Once `snapshot.has_final_agent_message` becomes true while required workspace outputs are still missing, record the grace-window start and deadline in live status, keep the worker alive until the deadline, and then terminate with reason code `workspace_final_message_missing_output` if the output file is still absent. If the file appears before the deadline, clear the grace-window state and let the existing output-complete path proceed.

Do not reuse the existing completion-wait flags for this state. The current completion-wait path is about outputs that already exist and need a short quiescence period. The new state is the opposite: the worker has announced completion but the required outputs are still absent. The live status must expose separate fields such as:

- `final_message_missing_output_grace_active`
- `final_message_missing_output_started_at_elapsed_seconds`
- `final_message_missing_output_grace_seconds`
- `final_message_missing_output_deadline_elapsed_seconds`
- `final_message_missing_output_deadline_reached`

Fifth, route the new reason through the stage runtime with one owner for retry accounting. The existing line-role fresh-session retry budget in same-session state remains the only budget for extra worker sessions. When the main workspace worker ends with `workspace_final_message_missing_output`, the runtime must bypass `_should_attempt_line_role_fresh_session_retry(...)` and instead call a new recovery-specific gate, for example `_should_attempt_line_role_final_message_recovery(...)`, that consumes the same persisted `fresh_session_retry_limit` and `fresh_session_retry_count` fields. There must never be more than one extra workspace-worker session total for the assignment, regardless of whether that extra session was spent by the older generic preserved-progress path or by this new final-message rescue path.

Sixth, persist a durable recovery-assessment artifact before deciding whether to relaunch. Write a repo-owned JSON artifact under the worker root or shard root that captures the prior session supervision reason, the doctor payload, the status payload, the normalized recovery assessment, and the current retry-budget state. This artifact is the authoritative explanation for why rescue was attempted, skipped, or denied.

Seventh, launch one fresh rescue session only when that deterministic assessment says it is warranted. Reuse the existing worker root, existing `task.json`, existing hidden same-session state, and existing output directories. The rescue prompt must be shorter than the original workspace prompt and must assume the worker is resuming an existing workspace. It should instruct the worker to run the repo-owned `--status` command first, run `--doctor` if the next step is still unclear, follow `recommended_command`, prefer repo-owned helpers over shell scripting, and stop once the same-session helper reports completion.

Eighth, record rescue outcome metadata in the same stage-owned surfaces that humans already inspect. At minimum, write whether recovery was attempted, whether it was skipped, succeeded, or exhausted, which retry budget entry was consumed, and what the final supervision reason was. This data must appear in shard `status.json`, stage telemetry summary rows, and failure rows so later debugging does not collapse the story into a generic subprocess failure.

Ninth, narrow the older inline watchdog retry path. Keep the shard-level strict-JSON inline retry only for the older watchdog reason codes that still justify a structured recompute. For `workspace_final_message_missing_output`, never invoke `_run_line_role_watchdog_retry_attempt(...)`. That case must stay on the workspace-preserving rescue path defined here.

Finally, update the short docs after code and tests are real. The parsing and LLM docs must describe the production contract exactly: ordinary drift remains warning-only, final-message-plus-missing-output gets a short grace window, the runtime may spend one shared fresh-session retry budget on a deterministic workspace rescue, and unresolved cases fail closed with explicit metadata.

## Validation And Acceptance

This plan is accepted only when the repository proves the full rescue-plus-fail-fast behavior, not merely the kill rule.

The first acceptance criterion is narrow watchdog behavior. A canonical line-role workspace worker must not be killed for ordinary drift such as tolerated helper-command use, slow progress, or runtime outlier shape alone. The new missing-output termination must activate only after `has_final_agent_message=True`, the expected output file is still missing, and the grace window has elapsed.

The second acceptance criterion is grace-window success. If a final agent message appears but the expected output file lands during the grace window, the watchdog must not terminate for the new reason and the normal output-authority success path must remain authoritative.

The third acceptance criterion is deterministic rescueability. When the main session ends for `workspace_final_message_missing_output`, the runtime must write a machine-readable recovery-assessment artifact that includes doctor payload, status payload, normalized recovery fields, and current retry-budget state.

The fourth acceptance criterion is shared budget ownership. The new final-message rescue path and the existing generic line-role fresh-session retry path must consume the same single persisted retry budget. Focused tests must prove there is no way for one assignment to silently receive two extra workspace-worker sessions by taking both paths.

The fifth acceptance criterion is bounded fresh-session rescue. If the deterministic assessment says the workspace is recoverable and the shared budget is still available, the runtime must launch exactly one fresh line-role workspace-worker session against the same preserved workspace root. If that rescue session succeeds, the shard completes successfully.

The sixth acceptance criterion is bounded failure. If the deterministic assessment says “not recoverable,” if the shared budget is already spent, or if the rescue session still does not produce valid output, the runtime must fail closed with durable rescue metadata. There must be no unbounded loop of extra sessions.

The seventh acceptance criterion is honest artifacts. `live_status.json`, shard `status.json`, stage telemetry, and failure rows must preserve the new missing-output reason and the rescue outcome so future debugging does not flatten everything into a generic subprocess failure.

The eighth acceptance criterion is preserved authority boundaries. Deterministic repo code still owns validation, repair rewrites, rescue eligibility, retry accounting, and final artifact expansion. The repo must not infer semantic line-role labels from this state or overwrite a model answer with deterministic guesses.

The ninth acceptance criterion is documentation honesty. The active parsing and LLM docs must explain the real production contract: ordinary drift is warning-only, final-message-without-output gets a short grace window, one shared fresh-session retry budget may be spent on deterministic workspace rescue, and unresolved cases fail closed explicitly.

## Idempotence And Recovery

The callback-level tests in this plan operate in temporary directories and are safe to rerun repeatedly. The recovery-assessment logic must also be idempotent for a stable workspace. Running the same doctor/status-based assessment twice on unchanged workspace files must produce the same normalized result and the same rescueability decision.

The shared fresh-session retry budget must be spent durably in repo-owned state. Once the line-role runtime records that the one extra workspace-worker session has been used for an assignment, rerunning the same stage must not silently create another extra rescue session unless the operator intentionally recreates the assignment from scratch.

If implementation stops midway, do not ship a mixed state where the callback emits `workspace_final_message_missing_output` but the post-session runtime has no deterministic interpretation for it. The safe checkpoint order is:

1. recovery-assessment adapter and tests;
2. grace-window callback state and tests;
3. post-session rescue orchestration and tests;
4. doc updates last.
