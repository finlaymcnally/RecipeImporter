# Line-Role Watchdog Final-Message Fail-Fast

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Exact User Desire

> "i like this. please make an execplanm to replace the current implementation with this one"

The “this one” in the quoted request is the simpler fail-fast behavior discussed immediately beforehand: once a canonical line-role workspace worker has produced a final agent message, deterministic repo code should stop treating the session as an open-ended conversation. If the required shard output file still has not appeared after a short grace window, the watchdog should terminate the worker promptly with a clear missing-output reason instead of letting the same session keep issuing helper commands until the model eventually runs out of context.

## Purpose / Big Picture

After this change, a failed canonical line-role worker session will fail for the right reason at the right time. If a worker says it is done but does not actually hand back the required `out/<shard_id>.json` file, the repo will stop that session quickly and record a specific reason such as “final message without expected output” rather than letting the session keep drifting inside the same context. The user-visible proof is straightforward: focused tests show the line-role watchdog waiting briefly after a final message, then terminating with the new reason code when the output file is still missing, while still allowing successful sessions to finish normally when the output file appears.

## Surprises & Discoveries

- Observation: The current canonical line-role watchdog already tracks both “expected workspace outputs are present” and “the worker has emitted a final agent message,” but those two signals are not combined into a fail-fast rule when outputs are still missing.
  Evidence: `cookimport/parsing/canonical_line_roles/runtime.py` currently records `workspace_output_complete`, `workspace_output_missing_files`, `workspace_output_stable_passes`, and `has_final_agent_message`, yet the callback only uses output stability for status reporting and only auto-terminates for hard boundary violations.

- Observation: The knowledge watchdog already has the more honest output-based completion pattern for the success case: once expected outputs are present and stable, deterministic code ends the session even without a queue controller.
  Evidence: `tests/llm/test_knowledge_orchestrator_contracts.py::test_knowledge_workspace_watchdog_completes_after_stable_outputs_without_queue_controller` asserts `reason_code == "workspace_expected_outputs_completed"`.

- Observation: The concrete production failure that motivated this plan was not “the worker never spoke again.” The worker did speak, and the repo observed a final agent message, but no shard output file appeared.
  Evidence: `data/output/2026-03-31_02.57.01/.../worker-005/live_status.json` shows `has_final_agent_message: true` together with `workspace_output_complete: false` and missing `line-role-canonical-0005-a001177-a001470.json`.

- Observation: The current relaxed workspace-worker policy is intentionally warning-first for helper-command drift, so a session can keep burning context after a final message if deterministic code does not enforce a file-level stopping rule.
  Evidence: `cookimport/parsing/canonical_line_roles/runtime.py` records `command_loop_without_output` as a warning rather than a kill path when `allow_workspace_commands=True`.

## Decision Log

- Decision: Scope this plan to canonical line-role only, not recipe or knowledge.
  Rationale: The user request came from a concrete canonical line-role benchmark failure. Knowledge already has output-based completion logic, and broadening the change across all worker stages would turn one targeted runtime fix into a larger refactor.
  Date/Author: 2026-03-31 / Codex

- Decision: Treat “final agent message but missing expected output” as a deterministic contract failure, not as a semantic validation failure.
  Rationale: The repo should not guess whether the worker “meant” to finish. The only contract that matters here is whether the required output file exists after the worker has effectively declared completion.
  Date/Author: 2026-03-31 / Codex

- Decision: Keep a short grace window after the final message before terminating.
  Rationale: A worker can legitimately finish speaking a moment before the filesystem write is visible or stable. The grace window prevents false positives while still stopping genuinely stuck sessions quickly.
  Date/Author: 2026-03-31 / Codex

- Decision: Mark the new termination as retryable.
  Rationale: The session itself failed its file-output contract, but that does not prove the shard is unrecoverable. Existing outer retry or recovery seams should remain free to decide whether to attempt one more bounded worker run.
  Date/Author: 2026-03-31 / Codex

- Decision: Preserve the current success path where stable expected outputs end the worker cleanly, even if no final agent message exists.
  Rationale: The repo already treats output files as the authority on the happy path. This plan adds one new early-failure path; it does not roll back the successful file-authority behavior that already works.
  Date/Author: 2026-03-31 / Codex

## Outcomes & Retrospective

This plan is not implemented yet. At the time of writing, the repository still allows canonical line-role workspace sessions to continue after a final agent message if the required output file is missing, which can waste context and surface a late generic model failure instead of a precise local one. The intended outcome of this plan is to replace that drift with a brief, deterministic grace window and a clear termination reason that future debugging can trust.

## Context and Orientation

Canonical line-role is the LLM-assisted stage that assigns labels such as `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `RECIPE_NOTES`, and `NONRECIPE_CANDIDATE` to the book’s canonical atomic lines. In this repository, that runtime lives under `cookimport/parsing/canonical_line_roles/`. The live worker orchestration owner is `cookimport/parsing/canonical_line_roles/runtime.py`.

The relevant transport is the workspace-worker path. A workspace worker is a long-lived `codex exec` session that runs in a sterile temporary workspace and edits a repo-written `task.json`. Deterministic repo code supervises that worker by receiving live snapshots about what the worker has done so far, such as whether the worker issued shell commands, whether a final agent message has been seen, and whether the expected output file exists.

The exact callback that makes supervision decisions is `_build_strict_json_watchdog_callback(...)` in `cookimport/parsing/canonical_line_roles/runtime.py`. Despite the old name, this callback is used for the relaxed `workspace_worker_v1` policy too. It already writes `live_status.json` with fields such as:

- `has_final_agent_message`
- `workspace_output_complete`
- `workspace_output_missing_files`
- `workspace_output_stable_passes`
- `workspace_completion_waiting_for_exit`
- `reason_code`

The current problem is not that these fields are unavailable. The problem is that the callback does not currently enforce the obvious contract:

1. if outputs are present and stable, finish successfully;
2. if there is no final message yet, keep waiting;
3. if there is a final message but outputs are still missing after a short grace window, terminate promptly with a clear reason.

The failed benchmark from `2026-03-31_02.57.01` demonstrated the gap. Worker `worker-005` under `data/output/2026-03-31_02.57.01/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-31_02.57.26/line-role-pipeline/runtime/line_role/workers/worker-005/` reported `has_final_agent_message=true` yet never produced `out/line-role-canonical-0005-a001177-a001470.json`. The session then kept issuing shell-helper rewrites until the model later failed with a context-window error. This plan exists to stop that exact shape much earlier and with a more truthful local error.

The future implementer must also be aware that the worktree is already dirty in nearby files, including `cookimport/parsing/canonical_line_roles/runtime.py`, `cookimport/llm/codex_exec_runner.py`, `docs/04-parsing/04-parsing_readme.md`, `docs/10-llm/10-llm_README.md`, and `tests/parsing/test_canonical_line_roles.py`. Do not reset those files. Re-read the current contents immediately before editing and merge carefully with existing local changes.

## Plan of Work

Start in `cookimport/parsing/canonical_line_roles/runtime.py` inside `_build_strict_json_watchdog_callback(...)`. Introduce one new branch that only applies when `allow_workspace_commands=True` and expected workspace output paths were supplied. The branch should activate when `snapshot.has_final_agent_message` is true but `workspace_output_status["complete"]` is still false. On the first snapshot that matches that condition, record the start of a short “final message grace window” in callback-local state. Keep writing that status into `live_status.json` so the repo can see that the watchdog is now waiting specifically for a missing output after the worker has effectively finished.

On later snapshots, if the required output file appears and becomes stable before the grace window expires, clear the waiting state and allow the normal success path to proceed. If the grace window expires and the expected output file is still missing, terminate the session with a new explicit reason code. Use a descriptive code such as `final_message_without_expected_output` or `workspace_final_message_missing_output`. The reason detail should plainly say that the worker emitted a final agent message but did not create the expected output file(s) within the grace window. Mark the decision retryable so existing outer retry seams can still decide what to do next.

Do not fold this new rule into generic command-loop warnings. The point of the change is to replace “drifts until context exhaustion” with “fails immediately for the real local reason.” The new rule should therefore outrank the existing warning-only `command_loop_without_output` path once a final message has already been observed. The output file remains the authority.

Keep the existing successful output-stabilization behavior intact. The callback already tracks output stability and writes `workspace_output_stable_passes`. That logic should remain the basis for success. This plan only adds an early-failure mirror image: final message seen, no output appears, grace window expires, terminate.

The callback currently tracks completion-wait state for output stabilization. Reuse that general shape rather than inventing an unrelated second state machine. However, keep the meanings separate in code and status. One state means “waiting for stable success after outputs appeared.” The new state means “waiting briefly for the output file to appear after the worker already gave a final message.” Those are opposite situations and should not share a confusing status name.

After the runtime change, update `tests/parsing/test_canonical_line_roles.py`. Add a focused callback-level test that creates a watchdog with one expected output path, sends one snapshot with `has_final_agent_message=True` while the output path is still absent, then sends a later snapshot after the chosen grace interval and asserts that the callback now returns a termination decision with the new reason code. Also assert that `live_status.json` records the missing output and the new state cleanly.

Add a second focused test for the non-failure case. That test should send a snapshot with `has_final_agent_message=True`, then create the expected output file before the grace window expires, then send a later snapshot and assert that the watchdog does not terminate with the new reason. This prevents the plan from shipping an over-eager fail-fast rule that kills legitimate successful sessions simply because the file write lands slightly after the final message.

If the higher-level runtime already has a good place to preserve the new reason code into shard status artifacts, update that path as well so the clearer reason survives beyond `live_status.json`. The relevant status writing already happens in `cookimport/parsing/canonical_line_roles/runtime.py` where shard `status.json` payloads are assembled from the normalized outcome. Keep that flow honest so a future failure report shows the new reason instead of a generic exit code.

Finally, update the short docs. In `docs/04-parsing/04-parsing_readme.md`, adjust the canonical line-role runtime note so it explains that a workspace worker can finish in two honest ways: either expected outputs become stable and the repo completes the session, or the worker emits a final message and the expected output file must appear quickly or the watchdog aborts the session with a missing-output reason. In `docs/10-llm/10-llm_README.md`, update the shared worker-contract explanation to reflect the same boundary: output files are authoritative, and a final message without output is now a prompt failure signal, not an invitation for the same session to keep consuming context.

## Validation and Acceptance

This plan is accepted only when the repository proves the new behavior directly.

The first acceptance criterion is the new early-failure rule. A canonical line-role workspace watchdog given `has_final_agent_message=True`, a missing expected output path, and enough elapsed time to exceed the grace window must terminate the session with the new explicit missing-output reason instead of returning `None` and continuing indefinitely.

The second acceptance criterion is the no-false-positive case. If `has_final_agent_message=True` but the expected output file appears before the grace window expires, the watchdog must not terminate with the new reason. The output-authority success path should remain available.

The third acceptance criterion is honest status reporting. `live_status.json` must clearly record that the output is missing, that the worker has already emitted a final agent message, and, once the grace window expires, that the new reason code is the cause of termination.

The fourth acceptance criterion is preserved authority boundaries. Deterministic repo code must not attempt to infer or repair semantic line-role labels from this state. The only new deterministic judgment here is whether the required file appeared after the worker effectively announced completion.

The fifth acceptance criterion is documentation clarity. The active parsing and LLM docs must describe the new rule plainly so future debugging starts from the real contract rather than from the old warning-only behavior.

## Idempotence and Recovery

The code changes in this plan are safe to rerun and safe to test repeatedly. The callback-level tests operate in temporary directories and should not depend on any persistent artifact state. If a test fails partway through implementation, rerun just the focused `pytest` command after each edit until the new reason code and grace-window behavior are stable.

Because the worktree is already dirty in the exact files this plan touches, the safest recovery path is to inspect the current file contents again before each patch and merge with local changes rather than trying to restore an older copy. Do not use destructive git resets or checkouts.

If the first implementation proves too eager and starts killing valid sessions before their output files appear, keep the new reason code but lengthen or clarify the grace-window logic rather than reverting to the old “keep drifting after final message” behavior. The product goal is still fail-fast with a small buffer, not no buffer at all.

## Artifacts and Notes

The new desired timeline for the failure case is:

    worker edits task.json
    worker emits a final agent message
    expected out/<shard_id>.json is still missing
    watchdog waits a short grace interval
    output file is still missing
    watchdog terminates with reason_code=<new explicit missing-output code>

The new desired timeline for the success case is:

    worker edits task.json
    worker emits a final agent message
    expected out/<shard_id>.json appears shortly afterward
    watchdog observes the file and lets the normal output-stability success path finish
