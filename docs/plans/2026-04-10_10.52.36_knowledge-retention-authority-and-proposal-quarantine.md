# Rebuild Knowledge Retention Authority So New Tags Expose Real Search-Worthy Gaps Instead Of Justifying Weak Rows

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the non-recipe knowledge path will stop asking one overloaded question. Today the model is effectively deciding “is this cooking-ish, and can I ground it somehow?” That makes it too easy to keep broad, obvious, rhetorical, or motivational prose as `knowledge` merely because the model can invent a plausible proposed tag. It also makes it too easy to invent broad chapter-theme labels that may sound intelligent but would be weak frontend search handles. After this change, the repository will ask two separate questions in sequence: first, is this exact row worth retrieving later as standalone cooking knowledge; second, in the same second-pass resolution step that already groups kept knowledge, does this row fit existing tags or does it reveal a real ontology gap that deserves a new tag proposal.

The user-visible result is that rows such as “Cooking is part artistry, part chemistry...” or other broad true-but-generic coaching prose will stop surviving as final retained knowledge just because the model can name a new tag. At the same time, the system will still be able to discover real ontology gaps. The difference is that proposed tags will become evidence of a missing catalog concept and a strong candidate frontend search term, not an escape hatch that turns weak rows into retained knowledge.

The product lens for this plan is explicit: tags are not just ontology bookkeeping. They power frontend search. That means a proposed tag should clear a higher bar than “semantically related.” It should feel like a concrete search handle a cook might plausibly type later. Specific technique, ingredient-behavior, storage, diagnostic, or equipment terms such as `rendering`, `bain-marie`, `smoke-point`, or `protein-coagulation` are good examples of the kind of proposal this system should preserve room for. Broad editorial or pedagogy-summary abstractions such as `heat-science`, `fat-management`, `fat-role`, or `ingredient-quality` are the kind of proposal this system should make rarer or reject.

This plan is comprehensive. It covers prompt contracts, task-file and inline-json validation, promotion logic, staging authority, observability, tests, and docs. It also replaces the obsolete ingest cleanup layer with an LLM-owned retention decision so the repository stops overriding fuzzy semantics after the fact. The plan defines one single operating mode for all runs; it does not add a profile matrix.

## Outcomes & Retrospective

This plan has not been implemented yet. The useful outcome today is clarity: the repository now has a concrete migration path away from “invent a tag to justify keeping the row” and toward a staged retention authority model where ontology discovery is observable and auditable without introducing settings slop or a detached proposal-review pass.

The biggest remaining implementation risk is increasing the complexity of the second pass now that it must own both grouping and proposal resolution. That risk is acceptable because the current system already pays the complexity cost indirectly through weak-grounding exceptions, the now-removed ingest cleanup layer, and benchmark confusion. The proposed design makes that complexity explicit and observable instead of hiding it across prompt wording and ingest heuristics.

## Context and Orientation

The relevant part of the repository is the optional LLM-backed nonrecipe finalize stage, which decides whether outside-recipe rows become final `knowledge` or final `other`. In this repository, “knowledge” means a piece of reusable cooking guidance that is worth retrieving later on its own. “Other” means everything else: memoir, framing, rhetoric, decorative headings, broad philosophy, obvious encouragement, navigation, or any text that may be true but is not worth retaining as a standalone retrieval target.

The current knowledge flow has two live steps. First, classification decides `knowledge` vs `other`. Second, grouping optionally clusters kept knowledge into idea groups when `knowledge_grouping_enabled` is on, although that second step is off by default today. The active owners are:

- `cookimport/llm/knowledge_stage/task_file_contracts.py`, which builds the worker-visible classification contract and validates the edited answers.
- `cookimport/llm/knowledge_stage/structured_session_contract.py`, which builds the inline-json prompt and packet shape for the same classification contract.
- `cookimport/llm/codex_farm_knowledge_ingest.py`, which normalizes worker payloads and now stays on structural normalization/validation rather than deterministic semantic demotion after the model responds.
- `cookimport/llm/knowledge_stage/promotion.py`, which aggregates kept rows, grounding metadata, proposed-tag rollups, and weak-grounding rollups.
- `cookimport/llm/knowledge_stage/workspace_run.py` and `cookimport/llm/knowledge_stage/runtime.py`, which execute the stage, drive repairs, and write telemetry.
- `cookimport/staging/nonrecipe_authority.py` and `cookimport/staging/pipeline_runtime.py`, which translate promoted knowledge decisions into final staged nonrecipe authority.
- `cookimport/config/run_settings.py` and related config owners, which define what knobs benchmark and product runs can actually set.

Right now, the first-pass worker can answer `knowledge` with either existing `tag_keys` or invented `proposed_tags`. That is the core design problem. The validator will reject malformed tags, exact duplicates of existing tags, and bad category keys, but it will still keep a row as `knowledge` even if grounding becomes weak after normalization. The code then tries to clean up some obvious bad promotions with deterministic low-utility heuristics in ingest. That is why the current system feels both too permissive and too “repo-opinionated” at the same time.

It also misses the actual product question for tags. In this project, tags are used as frontend search handles. A proposed tag therefore has to answer a harder question than “is this cooking adjacent?” It should be something a user might plausibly search for later. That is why this plan wants to preserve concrete missing concepts such as `rendering` while getting much stricter about broad, fuzzy labels such as `heat-science` that summarize a lesson without helping retrieval.

This plan replaces that structure with one that is easier to reason about. The first pass will decide retention only. Existing-tag knowledge can stay on the fast path. Rows that seem useful but do not fit the current catalog will be routed into proposal-candidate handling that lives inside the same second pass that already groups knowledge. Final public staged authority will still expose only `knowledge` and `other`, but proposal-related rows will travel through explicit internal states and explicit artifacts first.

## Milestones

### Milestone 1: Split retention from tag invention

At the end of this milestone, the first-pass knowledge classifier will no longer be able to invent tags. Its job will be to read a row and decide one of three internal outcomes: `knowledge` when an existing tag fit is real, `other` when the row is not worth retaining, or `proposal_candidate` when the row appears retrieval-worthy but does not cleanly fit the current catalog. The worker-visible classification packet will still include the full existing tag catalog, because the model should still match real tags directly when it can.

This milestone matters because it forces the retention question to be answered before ontology invention happens. The concrete proof is that classification task files, structured packets, and fake-runner fixtures no longer show `proposed_tags` in the first pass, and targeted contract tests can demonstrate that a row must route to `proposal_candidate` rather than immediately becoming `knowledge` via an invented tag.

### Milestone 2: Merge proposal resolution into grouping

At the end of this milestone, proposal candidates will be resolved in the same second LLM step that already groups kept knowledge rows. That combined step will either reject the proposal candidate back to `other` or approve one concrete proposed tag plus short justification metadata while also assigning or confirming the row's knowledge group. Approved proposal-resolved rows will become final `knowledge`; there is no alternate authority mode or optional bypass.

This milestone matters because it turns proposed tags into observable, reviewable ontology-gap evidence instead of a silent rescue path, and it does so without introducing yet another semantic pass. It also gives the model the right moment to decide whether a proposal is actually search-worthy or merely a broad summary label. The concrete proof is a fake-runner end-to-end test where a `proposal_candidate` row flows through the combined grouping/proposal-resolution step and becomes final `knowledge` only after that step approves it.

### Milestone 3: Remove deterministic semantic cleanup and publish the new behavior

At the end of this milestone, the old ingest cleanup layer will be gone or reduced to non-semantic validation only, and the new staged retention behavior will be visible in reports, telemetry, and staging outputs. Operators will be able to see how many rows were kept by existing tags, how many became proposal candidates, and how many proposal candidates were approved or rejected inside the combined second pass.

This milestone matters because the repository should no longer be making fuzzy semantic corrections after the LLM has answered. The concrete proof is that the old deterministic bypass counters disappear or shrink to compatibility shims, while new proposal-resolution counts appear in stage outputs and tests and optional grouping settings no longer control authority behavior.

## Plan of Work

Start by extending the knowledge classification contract in `cookimport/llm/knowledge_stage/task_file_contracts.py`. The current `_knowledge_classification_review_contract()` already asks a retrieval-worthiness question, but the answer schema still allows `proposed_tags` in the same step. Change the first-pass answer schema so `knowledge` means “existing-tag-grounded knowledge only,” `other` means drop, and `proposal_candidate` means “this row might be worth keeping, but not via an existing tag.” The first-pass grounding payload should keep `tag_keys` and `category_keys`, but it must reject `proposed_tags` entirely for both inline and taskfile flows.

Make the same contract change in `cookimport/llm/knowledge_stage/structured_session_contract.py`. The structured prompt must no longer tell the worker to propose a new tag during classification. Instead it must say: use existing tags first; if no existing tag fits but the row still seems like a standalone retrieval target, return `proposal_candidate` with empty grounding. The response shape examples, repair text, and packet-local row-id mapping must all reflect that change so repairs do not try to add proposals back into the first step.

Then extend the existing second pass in the knowledge runtime package so it handles both grouping and proposal resolution together. Do not create a standalone third semantic stage. Instead, evolve the current grouping contract so it receives all direct-knowledge rows plus all `proposal_candidate` rows, groups the kept knowledge rows, and resolves proposal candidates in the same structured output. That combined answer contract should be intentionally expensive for proposal rows: approve or reject the candidate, allow at most one proposed tag, require an existing `category_key`, and require short strings for `why_no_existing_tag` and `retrieval_query`. Those fields are not there for deterministic semantic grading; they are there to discourage broad fluff and to make proposal artifacts auditable by humans.

When updating the worker-facing wording for that second pass, phrase the proposal bar in search terms, not taxonomy terms. The prompt should say, in effect: only approve a proposal if it is a strong standalone search handle a cook might actually use later; prefer concrete kitchen vocabulary; avoid broad chapter-theme, editorial, or pedagogy-summary labels; if the row is useful but the only new label that comes to mind is broad or bookish, reject the proposal rather than inventing a weak search tag.

Wire that combined second-pass contract through both execution styles. In `cookimport/llm/knowledge_stage/workspace_run.py`, change the current grouping step so it always runs after classification and so it owns proposal resolution as part of the same step. Inline-json and taskfile flows should share the same semantics even if their transport details differ. Update same-session handoff state, repair/recovery policy usage, activity-trace summaries, and fake-runner support so this unified second pass appears as a normal stage-owned step rather than an ad hoc patch.

Do not introduce a new authority profile matrix. The repository should have one behavior only. Remove or retire `knowledge_grouping_enabled` and make the combined grouping/proposal-resolution step part of the normal knowledge pipeline whenever the knowledge LLM path is on. This keeps the runtime simple and matches the user's explicit request to avoid settings slop.

Update promotion and staging authority next. In `cookimport/llm/knowledge_stage/promotion.py`, split grounding counts into at least four clear families: existing-tag-kept knowledge, proposal candidates, approved proposal-resolved knowledge, and rejected proposal candidates. Keep the final public `knowledge` vs `other` block-category map binary, but write additive proposal-resolution artifacts alongside it so a human can see which rows depended on ontology gaps. In `cookimport/staging/nonrecipe_authority.py` and `cookimport/staging/pipeline_runtime.py`, make the final authority path consume those promotion results without introducing a third public nonrecipe label.

Once the LLM-owned retention path exists, delete the old semantic cleanup logic from `cookimport/llm/codex_farm_knowledge_ingest.py`. Keep syntactic normalization, exact existing-tag conflict checks, and structural coverage validation, but remove any helper that infers low-value prose buckets from repo heuristics and rewrites the category. If compatibility reporting needs a temporary bridge, keep only an observability note that a row traveled through `proposal_candidate` or `proposal_candidate_rejected`; do not re-run heuristic semantic cleanup after the LLM has answered.

Finally, update reporting and docs. `cookimport/staging/writer.py` and any runtime/stage-summary owners that surface grounding counts should show the new proposal-resolution counts. `docs/10-llm/10-llm_README.md` must explain the new staged retention contract and explicitly note that proposed tags are no longer part of first-pass classification. `docs/05-staging/05-staging_readme.md` and `docs/07-bench/07-bench_README.md` should explain that the knowledge pipeline now always runs its combined second pass when enabled, because grouping and ontology-gap resolution are now the same authority step.

## Validation and Acceptance

This work is complete when all of the following are true and observable:

First-pass classification no longer allows proposal invention. In task files and inline prompts, `proposed_tags` are absent from the first-pass answer schema. A row can be `knowledge` only when it grounds to an existing tag, `proposal_candidate` when it seems retrieval-worthy but lacks an existing fit, or `other` when it should be dropped.

Proposal resolution is a real part of the second step, not a prompt note. Runtime artifacts and fake-runner traces show that the grouping pass also resolves `proposal_candidate` rows. That combined step can approve or reject the proposal, and it writes stable artifacts that include the proposed tag and short structured justification fields.

Approved proposals are search-grade, not just semantically plausible. Focused prompt and runtime tests show that concrete search-friendly cooking terms can survive, while broad umbrella lesson-summary labels are pushed down or rejected. A row should not become final retained knowledge merely because the model coined a smart-sounding abstraction.

There is one single operating mode. The same approved proposal-resolved row becomes final `knowledge` in both product and benchmark runs, because benchmarks are supposed to reflect the real product behavior rather than a stricter alternate mode.

The deterministic low-utility bypass no longer owns the semantic cleanup. After implementation, rows are not rewritten from `knowledge` to `other` by `_low_utility_reason_for_block(...)` or equivalent heuristic demotion logic. Any remaining ingest logic must be structural validation only.

The motivating benchmark-style examples behave correctly in focused tests. At minimum, tests should cover:

- a broad true-but-generic paragraph such as “Cooking is part artistry, part chemistry...” becoming `other` or `proposal_candidate -> rejected`, but never final retained `knowledge` merely because of a proposed tag;
- a concrete existing-tag row staying direct `knowledge` with existing grounding and never entering proposal review;
- a genuinely specific missing-ontology row such as a `rendering`-like technique concept becoming `proposal_candidate -> approved` inside the combined grouping step and then final retained `knowledge`;
- a broad umbrella-tag attempt such as `heat-science`, `fat-role`, or `ingredient-quality` being rejected inside the combined grouping step.

## Idempotence and Recovery

The plan is intentionally additive before subtractive. Implement the internal `proposal_candidate` path and combined grouping/proposal-resolution artifacts first, then switch promotion and staging authority to consume them, and only then delete the deterministic low-utility bypass. That order keeps the repo debuggable and prevents a half-migrated state where weak rows can neither be retained nor clearly reported.

All migration steps are safe to rerun. During the transition, existing grouping-related settings can temporarily remain wired while the new always-on second pass is being implemented, but the final result should delete or retire those knobs rather than preserve them as compatibility clutter.

If implementation is interrupted midstream, the safe restart point is Milestone 1. The repository can temporarily tolerate an internal `proposal_candidate` value in contracts and tests even before the combined second pass is fully wired, as long as final promotion still rejects or reports it loudly rather than silently treating it as `knowledge`.
