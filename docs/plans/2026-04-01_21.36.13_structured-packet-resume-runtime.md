# Structured Packet Resume Runtime For Line Role And Knowledge

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

This plan now lands as an additive `codex_exec_style` option rather than a hard replacement. It keeps the semantic simplifications and authority boundaries from [docs/plans/2026-04-01_13.39.31_direct-batch-worker-contract.md](/home/mcnal/projects/recipeimport/docs/plans/2026-04-01_13.39.31_direct-batch-worker-contract.md), [docs/plans/2026-04-01_14.22.32_knowledge-tag-gate-demotion.md](/home/mcnal/projects/recipeimport/docs/plans/2026-04-01_14.22.32_knowledge-tag-gate-demotion.md), and [docs/plans/2026-04-01_15.00.11_remove-knowledge-review-metadata.md](/home/mcnal/projects/recipeimport/docs/plans/2026-04-01_15.00.11_remove-knowledge-review-metadata.md). It must also align with the ownership boundary described in [docs/plans/2026-04-01_15.16.22_recipe-ownership-invariant.md](/home/mcnal/projects/recipeimport/docs/plans/2026-04-01_15.16.22_recipe-ownership-invariant.md). The point of this plan is not to change those semantics. The point is to add a second live Codex transport for the targeted stages: keep `taskfile-v1` as the default contract, and add `structured-resume-v1` as an opt-in `packet` contract that keeps repair and grouping in the same resumed Codex session.

## Purpose / Big Picture

After this change, `codex-exec` runs can choose between two transports for `line_role` and nonrecipe finalize. The default remains the current `taskfile-v1` contract. The new `structured-resume-v1` contract starts one non-interactive structured Codex session per owned shard, sends one immutable assignment packet, requires one tiny JSON response, validates it deterministically, and, if needed, resumes the same session with a smaller repair packet. For knowledge shards, successful classification that keeps any rows immediately continues into the grouping prompt by resuming that same session rather than spawning a second worker. The final human-visible behavior is the same as today: line-role outputs still cover every owned atomic line, nonrecipe finalize still emits `knowledge` versus `other` plus grouped knowledge outputs, and deterministic code still owns every structural and authority boundary. What changes is the available transport choice and the worker ergonomics.

This matters because the current `taskfile` contract fixed the worst queue problems but it still spends a lot of complexity on agent-facing file editing, same-session helper commands, mirrored workspaces, shell drift policy, and task-file rewrite plumbing. That is useful when the model truly needs tools. For `line_role`, `nonrecipe_classify`, and `knowledge_group`, it is extra machinery around a job that is naturally “read all owned input once, return JSON, repair a few rows if needed.” The `packet` contract matches that cognition more directly.

The user-visible proof is deterministic and local. No Codex-backed book processing or benchmark run is required. The proof is that prompt preview, fake-runner execution, stage runtime tests, and domain suites all pass while the targeted stages stop using the `taskfile` happy path and instead write structured packet artifacts plus structured prompt telemetry. A future operator should be able to inspect one shard folder and see `initial_packet.json`, optional `repair_packet_01.json`, optional `grouping_packet.json`, the rendered prompt text for each turn, the validated outputs, and the final promoted stage artifacts.

## Context and Orientation

The live repo currently has two different worker contracts for these stages.

The first worker contract is `taskfile`. In that mode, the runtime creates a sterile mirrored working directory, writes repo-owned files such as `task.json`, starts `codex exec` in that directory, and lets the model interact with those local files. That is the live happy path for `recipe_refine`, and until this plan lands it is also the live happy path for `line_role`, `nonrecipe_classify`, and `knowledge_group`.

The second worker contract is `packet`. In that mode, the runtime calls `codex exec` once, gives it a prompt and optional schema, and expects a single JSON-like final response. There is no worker-visible editable file and no helper command loop. Today that mode is already used for narrower retry and repair follow-up paths.

This plan moves `line_role`, `nonrecipe_classify`, and `knowledge_group` from the `taskfile` contract to the `packet` contract, while adding one critical missing ability: resume the same non-interactive session for repair and grouping. In this plan, a “structured session” means one non-interactive Codex conversation started by `codex exec` and optionally continued by `codex exec resume --last` from the same isolated execution workspace. A “packet” means the immutable JSON payload the repo writes for one turn. A “repair packet” means a smaller immutable JSON payload containing only the invalid rows or invalid groups plus deterministic validator feedback. A “grouping packet” means the second-pass immutable packet that contains only the knowledge rows kept by a successful classification pass.

For resumed turns, the critical safety invariant is simple: one assignment workspace may have only one live packet-session lineage. Repo code may use `codex exec resume --last` as the live resume primitive only because the workspace itself is supposed to identify exactly one lineage. The runtime should therefore persist repo-owned lineage metadata in that workspace, record any explicit session id when the CLI exposes it, and fail closed if later evidence suggests multiple candidate sessions were started from the same workspace.

If the current sizes are too large, shrink shard targets as part of this plan before enabling the new happy path.

## Milestones

### Milestone 0: Freeze the packet semantic contract

At the end of this milestone, the target packet contract for nonrecipe classification will already be the final simplified semantic surface: `category` plus `grounding`, with grounding-gate demotion and no `reviewer_category` or `retrieval_concept`. This milestone exists so the transport migration does not quietly carry dead semantic baggage into a fresh runtime surface.

The proof for this milestone is deterministic. The packet-facing validators, prompt preview outputs, and fixture builders for knowledge classification must all agree on the simplified contract before resumed-session transport is considered complete.

### Milestone 1: Prove the resumed structured-session transport

At the end of this milestone, the repo will have one clear runner API for starting and resuming structured non-interactive Codex sessions. The fake runner and real runner will both know how to execute an initial structured prompt and then resume the most recent session from the same isolated execution workspace. The implementation must explicitly encode the limitation that resumed turns do not get CLI output-schema enforcement.

The proof for this milestone is deterministic. Add runner tests that inspect the built command lines and call records. The initial structured turn should invoke `codex exec` with `--output-schema` when requested. The resumed turn should invoke `codex exec resume --last` from the same execution workspace, should not try to pass `--output-schema`, and should still return normal streamed events, usage, and final-message extraction. Fake-runner tests must also prove that one assignment can have an initial turn followed by one or more resumed turns without losing prior structured state. Add one explicit lineage-safety test as well: if repo-owned workspace metadata indicates that more than one packet session lineage was started from the same assignment workspace, the runtime must fail the assignment instead of guessing which session `--last` will pick.

### Milestone 2: Build a common structured packet and repair framework

At the end of this milestone, the targeted stages will no longer need the `taskfile` happy path in order to describe owned work. Instead, the repo will have one common packet envelope and one common repair-packet envelope for structured sessions. The runtime will write packet artifacts to disk for observability, render them into prompt text, parse JSON responses, merge valid subsets across turns, and produce deterministic validator feedback for resumed repair prompts.

The proof for this milestone is deterministic contract tests over synthetic packet payloads. A novice should be able to inspect one packet artifact and one rendered prompt and understand the ownership, allowed outputs, and validator feedback without seeing any task-file helper metadata or mirrored workspace logic. Packet-budget tests or preview assertions must also prove that the rendered prompt shape includes explicit budget headroom for the expected resumed follow-up path rather than only for the first turn.

### Milestone 3: Migrate `line_role` to structured sessions

At the end of this milestone, `line_role` will no longer use the `taskfile` contract or same-session handoff helpers on the happy path. It will build one immutable shard packet, render one inline structured prompt, parse one JSON response covering every owned atomic line exactly once, and, if needed, resume the same structured session with a smaller repair packet for the failing rows. Once validation succeeds, deterministic code will still write the final shard outputs and promote them exactly as today.

The proof for this milestone is a deterministic worker-fixture test that runs line-role with the fake structured-session runner, shows one initial structured turn and, when needed, one resumed repair turn, and still writes the same runtime artifacts and validated shard outputs expected by downstream code. Prompt preview for line-role must also render the new packet-based prompt rather than the old `taskfile` worker prompt.

### Milestone 4: Migrate nonrecipe classification to structured sessions

At the end of this milestone, nonrecipe classification will stop creating `taskfile` classification assignments. It will instead build one immutable classification packet per owned shard, ask for one JSON response that covers every owned row exactly once, apply the grounding gate deterministically, and, if needed, resume the same structured session with a smaller repair packet. Successful all-`other` shards should complete immediately without any grouping turn.

The proof for this milestone is a deterministic classification runtime test that shows three cases: a fully valid shard that completes with no grouping, an under-grounded `knowledge` answer that is deterministically demoted to `other` without repair, and an invalid-structure case that triggers a resumed repair packet in the same session. Promotion and manifest artifacts must keep the same semantic meaning as before.

### Milestone 5: Migrate knowledge grouping into the same resumed session

At the end of this milestone, successful classification that keeps any rows as `knowledge` will no longer write a grouping `task.json` and wait for a file-edit helper loop. Instead, the runtime will build a grouping packet from only those kept rows and resume the same structured session with the grouping prompt. Deterministic code will validate exact kept-row coverage, exact grouping coverage, and the existing grouping invariants before writing final shard outputs.

The proof for this milestone is a deterministic knowledge runtime test that shows one initial classification turn followed by one resumed grouping turn in the same assignment session, with final `idea_groups` and `block_decisions` matching the current semantic contract.

### Milestone 6: Remove the targeted `taskfile` happy path and update docs

At the end of this milestone, the active program surface will describe the targeted stages as `packet` stages rather than `taskfile` stages. The old `taskfile` happy path and helper-command language will remain only where still intentionally used by recipe refine. Line-role and knowledge docs, prompt preview, fake-runner flows, and tests will all match the new transport. Any stage-specific task-file helpers or same-session handoff modules that are now dead for the targeted stages must be removed or reduced to recipe-only responsibilities.

The proof for this milestone is passing domain suites plus grep-based cleanup over the targeted stage docs and code. A reader inspecting `docs/10-llm/10-llm_README.md` and the stage package notes should see the new transport described plainly and should not be told that `line_role` or nonrecipe finalize still use the `taskfile` contract on the happy path.

## Plan of Work

Start by freezing the packet semantic contract at the new boundary. If any packet-facing knowledge code still expects `reviewer_category` or `retrieval_concept`, remove those fields first so the transport rewrite lands on the simplified `category` plus `grounding` surface from day one. Do not let the new packet runtime become a staging area for dead semantics.

Then move to `cookimport/llm/codex_exec_runner.py`. Add an explicit resumed packet-session seam rather than trying to smuggle resume behavior through the current structured prompt shape invisibly. The cleanest repo-facing naming is to expose `run_taskfile_worker(...)` for the existing `taskfile` contract, `run_packet_worker(...)` for initial `packet` turns, and `resume_packet_worker(...)` for resumed `packet` turns. The resumed method should not accept `output_schema_path` because the CLI does not support it. Both packet methods should return `CodexExecRunResult`, but that result type must grow enough metadata for downstream telemetry to tell which assignment turn it represents, whether it was resumed, and what execution workspace it used. If an explicit session id is available from the underlying CLI response, record it as telemetry. Do not make correctness depend on it. The correctness contract should be “resume the last session in this isolated execution workspace.”

In the same module, keep using the prepared execution workspace machinery rather than inventing a second temp-root strategy. For the targeted structured-session stages, the execution workspace should still be created under the normal direct-exec root so that streamed events, stdout/stderr sidecars, packet artifacts, usage, and prompt/debug files stay in the same operator-visible tree. The difference is that the structured-session path should no longer sync a worker-facing editable `task.json`; it should sync only prompt artifacts, packet artifacts, response artifacts, telemetry, and other repo-owned debug surfaces. This lets the runtime keep one uniform artifact tree across transports.

Use `codex exec resume --last` from the prepared execution workspace as the live resume command. Because `codex exec resume` is cwd-filtered by default, the execution workspace itself becomes the stable session locator. This is more robust than parsing and replaying a session id, but only if the repo actively preserves the one-workspace-one-lineage invariant. Add a small repo-owned lineage sidecar such as `packet_session.json` or `session_lineage.json` under the assignment artifact root. It should record the assignment id, execution workspace path, packet turn count, and any explicit session id observed from the CLI. Before every resumed turn, validate that repo-owned lineage metadata still points to exactly one prior packet session for that workspace. If the workspace lineage is ambiguous, fail the assignment with a deterministic error instead of resuming whatever `--last` happens to choose.

Next, introduce one stage-neutral structured packet helper layer under `cookimport/llm/`, for example `structured_session_packets.py` plus `structured_session_runtime.py`. Keep the names boring and explicit. This layer should define the common envelope shape used by all targeted stages, the helper that writes packet JSON artifacts under the assignment artifact root, the helper that renders inline prompt text from a packet plus a stage prompt template, the helper that parses JSON-only final responses, and the helper that constructs repair packets from validator failures plus prior valid rows. The common envelope should include `schema_version`, `stage_key`, `assignment_id`, `run_mode`, stable stage context, and `units`. Keep it immutable. The worker should never be asked to edit or echo packet metadata.

As part of that shared layer, make budget assumptions explicit. Add a prompt-budget helper or preview assertion that computes the expected worst live path for each targeted stage. For `line_role`, reserve headroom for one initial packet plus at least one resumed repair turn. For knowledge classification, reserve headroom for one initial classification turn plus either one repair turn or one grouping turn in the same session, whichever is larger. If prompt preview shows the current shard sizes leaving inadequate headroom for those paths, shrink shard targets in planning rather than hoping the live model will squeeze through.

Do not carry the `pro-idea.md` sample `keep` plus `tag_key` nonrecipe output shape into the live design. For knowledge classification, preserve the post-simplification semantic authority boundary. That means the structured classification output should be one answer row per owned unit with `category` plus `grounding`, and the grounding gate must still demote under-grounded `knowledge` to canonical `other`. If [docs/plans/2026-04-01_15.00.11_remove-knowledge-review-metadata.md](/home/mcnal/projects/recipeimport/docs/plans/2026-04-01_15.00.11_remove-knowledge-review-metadata.md) is not yet implemented when this plan starts, implement that semantic cutover first inside the new structured packet contract. Do not migrate a soon-to-be-deleted `reviewer_category` or `retrieval_concept` field into the new transport and then remove it a day later.

Once the packet helper layer exists, migrate `line_role` first. The line-role modules already know how to build immutable shard input payloads and validate exact owned-line coverage. Reuse that knowledge, but replace the task-file scaffolding with a new line-role packet builder and a new line-role structured-response parser. The initial line-role turn should send the whole owned shard at once. Validation should still enforce exact `atomic_index` coverage and allowed labels. Repair should operate on the failing rows only and resume the same structured session. When validation succeeds, keep writing the same authoritative shard output JSON and the same downstream proposals and summaries. The change is transport, not final line-role semantics.

After line-role is stable, migrate knowledge classification. Replace the current classification task-file builder in `cookimport/llm/knowledge_stage/task_file_contracts.py` with a classification packet builder that emits the immutable owned-block rows, ontology context, and any lexical hints the model legitimately needs. Preserve the grounding gate and the retrieval-first meaning of `knowledge`. Keep deterministic validation semantic-light but strict on ownership, coverage, enum values, tag normalization, and grounding membership. If the initial classification response is structurally invalid, build a repair packet containing only the failing rows plus validator feedback and resume the same structured session. If the response is structurally valid and all rows end as `other`, finalize immediately with no grouping prompt.

Then migrate knowledge grouping. Grouping must remain a second semantic phase, but it should no longer be a second worker. Build a grouping packet only from the rows that survived classification as grounded `knowledge`, write it to the assignment artifact tree, and resume the same structured session with the grouping prompt. Deterministic code must still validate exact kept-row coverage, exact one-group-per-kept-row membership, and any group-key compatibility invariants. When grouping validates, write the same final shard output shape used by downstream promotion. If grouping fails validation, build a grouping repair packet and resume the same session again. Keep the repair count small and fixed.

During the knowledge migration, integrate with the ownership plan rather than bypassing it. The new classification packet builder should consume the same ownership-derived eligible-block set that nonrecipe finalize is supposed to use. ## Validation and Acceptance

Acceptance is behavioral and deterministic.

For the runner transport:

1. The real runner can start one structured non-interactive turn and resume the same session from the same isolated execution workspace with `codex exec resume --last`.
2. The fake runner can simulate the same lifecycle deterministically so the repo can test initial, repair, and grouping turns without live spend.
3. Initial turns may use `--output-schema`; resumed turns do not, and deterministic validation remains authoritative in both cases.
4. Repo-owned lineage metadata proves that each assignment workspace has exactly one packet-session lineage, and the runtime fails closed if resume would be ambiguous.

For line-role:

1. The live line-role happy path no longer uses the `taskfile` contract.
2. A successful line-role shard is answered in one structured JSON response covering every owned atomic line exactly once.
3. An invalid line-role response triggers a smaller repair packet and a resumed structured turn in the same session rather than a fresh worker session.
4. The final validated shard outputs, proposals, and downstream stage behavior remain the same as before.

For nonrecipe classification:

1. The live nonrecipe classification happy path no longer uses the `taskfile` contract.
2. The classification response uses the post-simplification semantic contract, which is `category` plus `grounding`.
3. The grounding gate still demotes under-grounded `knowledge` to canonical `other` without forcing a repair turn for that reason alone.
4. Structurally invalid answers still fail closed into a resumed repair packet.

For knowledge grouping:

1. Grouping remains a separate semantic pass, but it runs by resuming the same structured session rather than by starting a second worker or editing a grouping `task.json`.
2. Only rows that survived classification as valid grounded `knowledge` appear in the grouping packet.
3. Final shard outputs still contain validated `block_decisions` plus validated `idea_groups`.

For docs and artifacts:

1. The targeted shard artifact tree contains repo-owned packet and prompt artifacts such as `initial_packet.json`, optional `repair_packet_01.json`, optional `grouping_packet.json`, rendered prompt text, and validated outputs.
2. `docs/10-llm/10-llm_README.md` describes `line_role` and nonrecipe finalize as `packet` stages, not as `taskfile` stages.
3. Prompt preview and fake-runner outputs show the new transport accurately without requiring live model spend.

For test coverage:

1. New runner tests prove the exact initial and resume command construction.
2. New or rewritten stage tests prove line-role repair, knowledge classification demotion, and knowledge grouping all work through resumed structured sessions.
3. The repo-preferred `llm`, `parsing`, `staging`, and `labelstudio` domain suites pass in the project-local virtual environment.

## Idempotence and Recovery

Implement this plan in two safe phases. First, add the resumed `packet` runner seam and the new packet helpers while the old `taskfile` path still exists. Use those additive seams to migrate line-role and knowledge tests one stage at a time. Second, once the new transport passes the focused suites and the domain suites, remove the targeted `taskfile` happy path. Do not stop in a mixed permanent state where both worker contracts are documented and both remain equally live.

It is acceptable for implementation to carry a short-lived internal fallback while the first migrated stage is under test. It is not acceptable to merge a final state where `line_role` or nonrecipe finalize can silently choose either transport at runtime. The burn-the-boats final state is one live happy-path transport for the targeted stages.

If the resume transport proves unexpectedly unstable in one narrow place, do not paper over it by falling back to fresh-session repair on the happy path without documenting that change in the `Decision Log` and updating the plan. The whole point of this design is to keep repair and grouping in the same resumed session whenever the transport allows it.

If the workspace-lineage invariant proves harder to maintain than expected, do not weaken it into “best effort resume.” Tighten the workspace/session bookkeeping or fail the affected assignment deterministically. The final transport must never silently resume an uncertain session just because `--last` returned something.

If the ownership contract has not yet landed when the knowledge migration starts, isolate the temporary adapter at the packet-builder boundary and record that choice in the plan. Do not let raw `recipe_spans` leak back into multiple new structured-session modules.
