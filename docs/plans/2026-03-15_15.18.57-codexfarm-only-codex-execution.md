# Make CodexFarm The Only Codex Backend

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan is self-contained. It assumes no prior knowledge beyond the current repository state.

## Purpose / Big Picture

After this change, every feature in this repository that uses Codex must route through CodexFarm. The repo must stop making any direct local `codex exec` calls at runtime. A user should be able to choose whether Codex-backed behavior is enabled, but once enabled the execution backend must always be CodexFarm for every surface.

The user-visible proof is simple. Running a Codex-backed stage import, a Codex-backed benchmark, or a Label Studio import with prelabeling should produce CodexFarm process metadata and CodexFarm-owned raw artifacts only. There must be no code path that constructs `codex exec -`, no runtime shellout to local Codex CLI, and no mixed-backend run where some Codex work comes from CodexFarm while other Codex work comes from direct CLI calls.

Today the repo has five Codex-backed surfaces: `recipe`, `line_role`, `knowledge`, `tags`, and `prelabel`. Three already use CodexFarm. Two do not. This plan closes that split so all five surfaces use one backend.

## Surprises & Discoveries

- Observation: the repo already has a canonical surface classifier, so the migration target is well-bounded.
  Evidence: `cookimport/config/codex_decision.py::classify_codex_surfaces(...)` explicitly enumerates the five surfaces and already treats `prelabel` as a first-class Codex surface instead of a one-off helper.

- Observation: the mixed backend is not accidental; it is encoded in two separate runtime stacks.
  Evidence: `cookimport/llm/codex_farm_orchestrator.py`, `cookimport/llm/codex_farm_knowledge_orchestrator.py`, and `cookimport/tagging/codex_farm_tags_provider.py` instantiate `SubprocessCodexFarmRunner`, while `cookimport/parsing/canonical_line_roles.py` and `cookimport/labelstudio/prelabel.py` call `run_codex_json_prompt(...)`.

- Observation: `line_role` and `prelabel` do not merely differ by command string; they differ by execution model.
  Evidence: direct CLI paths build one prompt string and expect one response string immediately, while CodexFarm paths are pipeline-oriented and write input/output bundle directories plus manifests and process metadata.

- Observation: preserving behavior while changing backend will require introducing new CodexFarm pipeline definitions, not only swapping subprocess commands.
  Evidence: the current direct callers rely on prompt templates, strict JSON parsing, retries, token-usage capture, and per-task caching semantics that today live above `codex exec`. CodexFarm can replace the transport, but the repo must define equivalent pipeline contracts for it to run.

- Observation: the cheapest complete solution is not “keep both and prefer CodexFarm.” The request is stricter than that.
  Evidence: the user requirement is explicit: nothing should ever call local Codex directly. That means a fallback to `codex exec` is still a failure, even if CodexFarm is preferred most of the time.

## Outcomes & Retrospective

At plan creation time, no code has changed yet. The repository still has a mixed model: CodexFarm for recipe-family work and direct local Codex CLI for line-role and Label Studio prelabel. The expected outcome of this plan is stricter and easier to reason about: one backend, one telemetry style, one place to configure Codex execution, and one raw-artifact family to inspect when debugging.

The main implementation risk is shape mismatch. Direct `codex exec` helpers are single-call request/response tools, while CodexFarm expects pipeline definitions, schemas, directory I/O, and process manifests. The plan therefore focuses first on introducing CodexFarm-native adapters for the two remaining surfaces before removing the direct helper.

## Context and Orientation

This repository already has a stable concept called a Codex surface. A Codex surface is one feature area that may perform live Codex work. The source of truth for those surfaces is `cookimport/config/codex_decision.py`. Today it recognizes five surfaces:

`recipe`. This means recipe extraction or repair through `llm_recipe_pipeline`. It already uses CodexFarm through `cookimport/llm/codex_farm_orchestrator.py`.

`knowledge`. This means pass4 knowledge harvest through `llm_knowledge_pipeline`. It already uses CodexFarm through `cookimport/llm/codex_farm_knowledge_orchestrator.py`.

`tags`. This means pass5 tag generation through `llm_tags_pipeline`. It already uses CodexFarm through `cookimport/tagging/codex_farm_tags_provider.py`.

`line_role`. This means canonical line-role fallback labeling through `line_role_pipeline=codex-line-role-v1`. It does not use CodexFarm today. It uses `cookimport/llm/codex_exec.py::run_codex_json_prompt(...)` from `cookimport/parsing/canonical_line_roles.py`.

`prelabel`. This means Label Studio prelabel generation. It does not use CodexFarm today. It uses `run_codex_json_prompt(...)` through `cookimport/labelstudio/prelabel.py`.

CodexFarm in this repository means the external `codex-farm` command driven through `SubprocessCodexFarmRunner`. A pipeline in this repository means a named pack definition under `llm_pipelines/` with a prompt template, an output schema, and CodexFarm process metadata. Existing recipe, knowledge, and tag passes already follow that model.

The two direct-CLI paths share `cookimport/llm/codex_exec.py`, which builds a `codex exec -` command, feeds a prompt on stdin, optionally asks for JSON event output, and returns the raw response text plus token usage. The migration target is to remove this runtime shape entirely.

The most important files for this plan are:

`cookimport/config/codex_decision.py`. This is the canonical Codex surface classifier and the right place to make backend invariants explicit.

`cookimport/parsing/canonical_line_roles.py`. This owns Codex-backed canonical line-role fallback and currently uses direct Codex CLI.

`cookimport/labelstudio/prelabel.py`. This owns Label Studio prelabel generation and currently uses direct Codex CLI.

`cookimport/llm/codex_farm_runner.py`. This is the shared subprocess runner for CodexFarm and the right place to keep transport behavior centralized.

`cookimport/llm/codex_farm_orchestrator.py`, `cookimport/llm/codex_farm_knowledge_orchestrator.py`, and `cookimport/tagging/codex_farm_tags_provider.py`. These are the existing examples of surface-specific adapters sitting above CodexFarm.

`llm_pipelines/`. This is where new CodexFarm pipeline definitions for line-role and prelabel should live.

`tests/parsing/`, `tests/labelstudio/`, `tests/llm/`, and `tests/cli/`. These will need to prove both surface behavior and architectural invariants.

## Plan of Work

Start by introducing two new CodexFarm-backed adapters, one for canonical line-role and one for prelabel. The exact module names may vary, but good targets are `cookimport/llm/codex_farm_line_role_orchestrator.py` and `cookimport/labelstudio/prelabel_codex_farm.py` or one shared line-role/prelabel CodexFarm module under `cookimport/llm/`. Each adapter should preserve the existing repo-owned behavior around batching, retries, parsing, and fallback decisions while replacing the transport.

For canonical line-role, define a new CodexFarm pipeline pack in `llm_pipelines/` that accepts the same semantic inputs the direct path already constructs: a batch of candidate lines, each line’s allowed labels, and the prompt format version. The output schema should model the current parsed response shape exactly enough that `cookimport/parsing/canonical_line_roles.py` can keep its deterministic validation and fallback logic. The current line-role code is a good guide for the desired normalized output: atomic line index plus chosen label from an allowlist.

For prelabel, define a separate CodexFarm pipeline pack that accepts the same prompt context currently sent by `cookimport/labelstudio/prelabel.py`. The output schema should match the existing prelabel annotation suggestion shape so the downstream span-repair and task post-processing logic can remain repo-owned. Keep the prompt templates file-based and versioned under `llm_pipelines/prompts/`, just as the current prelabel prompt templates are.

Once those pipelines exist, refactor `cookimport/parsing/canonical_line_roles.py` so its live Codex path calls the new CodexFarm adapter rather than `run_codex_json_prompt(...)`. Keep the same deterministic-first policy, error handling, and fallback semantics. The important change is transport. The line-role module should no longer know what `codex exec` is.

Then refactor `cookimport/labelstudio/prelabel.py` so `CodexCliProvider` is replaced by a CodexFarm-backed provider or by direct calls into the new prelabel adapter. Preserve current caching, timeout, prompt logging, response normalization, and error reporting semantics where those are still useful. The command configuration exposed to the user should shift from direct Codex CLI wording to CodexFarm wording. A user may still pick model and reasoning effort overrides, but the executable and backend terminology must become CodexFarm-specific.

After those two surfaces are migrated, remove the direct CLI transport module from runtime use. `cookimport/llm/codex_exec.py` should either be deleted or reduced to a compatibility stub that raises a clear error if any caller still tries to use it. There should be no live code path that imports it for normal execution. Similarly, update CLI help text, interactive prompts, and config defaults so there is no remaining suggestion that local `codex exec` is a supported execution path.

Next, tighten the policy layer. `cookimport/config/codex_decision.py` should still classify surfaces, but any surface marked as Codex-enabled must now imply CodexFarm transport. Add one explicit helper or invariant check that can be used by tests and by command-boundary manifests to assert `codex_backend="codexfarm"` whenever live Codex work is enabled. The point is not just to move subprocess commands around; it is to make the architectural rule visible and testable.

Then update artifact and telemetry handling. Line-role and prelabel currently capture different metadata than CodexFarm-backed surfaces. New CodexFarm adapters should write process metadata, pipeline ids, and raw input/output artifacts in the same style as the existing recipe-family flows. Benchmark and Label Studio manifests should point to those artifacts so debugging one run no longer requires inspecting two unrelated backend formats.

Finally, remove or rewrite tests that assert `codex exec` behavior, replace them with CodexFarm-backed tests, and add one global invariant test that fails if runtime code still references `codex exec` in executable paths. Documentation in `docs/10-llm/10-llm_README.md` and `docs/02-cli/02-cli_README.md` must also be updated so the stated backend model matches reality.

## Milestones

### Milestone 1: Add CodexFarm-native pipeline contracts for line-role and prelabel

At the end of this milestone, the repository contains CodexFarm pipeline definitions and repo-owned adapters for the two remaining direct-CLI surfaces. A novice reader can inspect `llm_pipelines/` and the new adapter modules and understand how line-role and prelabel are meant to run without reading the old direct CLI helper.

The proof is that targeted unit tests can execute those adapters with fake or injected CodexFarm runners and receive the same normalized result shapes the current callers expect.

### Milestone 2: Switch runtime callers to the new adapters

At the end of this milestone, `cookimport/parsing/canonical_line_roles.py` and `cookimport/labelstudio/prelabel.py` no longer call `run_codex_json_prompt(...)`. Line-role and prelabel continue to work from the user’s point of view, but they now create CodexFarm-shaped process artifacts and use CodexFarm model/reasoning override flow.

The proof is behavior. A Codex-backed stage or benchmark run that enables line-role, and a Label Studio import that enables prelabel, both complete without invoking local `codex exec` and both emit CodexFarm process metadata.

### Milestone 3: Remove direct local Codex runtime support

At the end of this milestone, there is no supported runtime path that shells out to `codex exec`. Any remaining compatibility wrapper must fail closed with an explicit error instead of silently running a direct local Codex call.

The proof is structural and behavioral. Source audit tests pass, targeted runtime tests pass, and repository docs no longer advertise direct Codex CLI as a supported backend.

### Milestone 4: Normalize manifests, telemetry, and docs around one backend

At the end of this milestone, all five Codex surfaces produce one family of process metadata and one family of backend terminology. CLI summaries, run manifests, and docs should all make it obvious that Codex-backed work means CodexFarm-backed work.

The proof is that stage, benchmark, and Label Studio artifacts all point to CodexFarm process runs, and documentation matches observed behavior.

## Validation and Acceptance

Acceptance is behavior-first and invariant-driven.

First, all five Codex surfaces must be able to run without direct local Codex CLI. That means recipe, knowledge, tags, line-role, and prelabel all either execute via CodexFarm or remain off. No enabled surface may execute through `codex exec`.

Second, enabling line-role or prelabel must produce CodexFarm-shaped evidence. A successful run should include pipeline ids, process metadata, and raw input/output directories or manifest entries consistent with the existing recipe-family CodexFarm flows.

Third, configuration and UI text must stop advertising direct local Codex CLI. The user should configure CodexFarm command, pipeline root, workspace root, model override, and reasoning effort override, but not a separate local Codex CLI command for prelabel or line-role.

Fourth, the codebase must enforce the architectural rule in tests. There should be at least one test that asserts the runtime backend for every Codex surface is CodexFarm, and one source-audit style test or command that fails if executable runtime code still references direct `codex exec`.

Fifth, legacy historical data remains readable. This plan changes runtime generation, not archival interpretation. Old benchmark or Label Studio artifacts that mention direct Codex CLI may continue to exist, but new runs must not create them.

## Idempotence and Recovery

This migration should be done additively first and subtractively second. Add the new CodexFarm adapters and tests before removing the direct CLI helper. That gives a safe rollback path: if a migrated caller fails, temporarily restore its old wiring while keeping the new pipeline definitions and adapter tests in place.

Do not delete direct-path tests until the CodexFarm replacement tests cover the same behavioral surface. The recovery path for any milestone is to keep the new adapter module and pipeline definitions, revert the caller wiring for the broken surface, and continue from there.

Keep historical-manifest readers backward compatible. New runs should emit only CodexFarm backend metadata, but analytics or follow-up tooling should tolerate older runs that were created before the migration.
