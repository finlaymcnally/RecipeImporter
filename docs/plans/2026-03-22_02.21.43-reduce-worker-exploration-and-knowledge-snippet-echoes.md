# Reduce Worker Exploration and Knowledge Snippet Echoes

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The most recent knowledge-stage work solved the semantic category drift and the companion watchdog work is expected to allow local temp-root helper files such as `/tmp` without turning them into boundary kills. That still leaves two expensive behaviors in the mainline knowledge path. First, workers spend time exploring task manifests and inventing shell schedulers instead of just processing the next declared task row. Second, when they do write outputs, too many “useful” packets still fail because snippets echo most or all of the owned chunk text rather than extracting a small grounded claim, and the current validator still misses some shorter verbatim block-copy snippets.

After this change, a workspace worker should have a much simpler operating model: read one repo-declared task row, open the exact named files for that row, write the result, and move on. The shared worker contract should make that path cheaper than shell exploration. The knowledge prompt/examples should also make it easy to produce short extracted snippets instead of full-chunk echoes. The observable outcome is that a future benchmark should show less exploratory shell churn, fewer knowledge `semantic_snippet_echoes_full_chunk` failures, and fewer wasted worker turns before useful packets validate.

## Surprises & Discoveries

- Observation: once semantic category drift was fixed, the next dominant invalid-output bucket became snippet echo, not schema drift.
  Evidence: in `data/output/2026-03-22_01.34.12/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-22_01.34.51/raw/llm/saltfatacidheatcutdown/knowledge/task_status.jsonl`, the March 22 run shows `semantic_snippet_echoes_full_chunk` far more often than `schema_invalid`.

- Observation: the workers are still burning time on inventory exploration because the declared task seam is technically present but ergonomically weak.
  Evidence: the saved worker traces for the same run show repeated attempts to transform `assigned_tasks.json` with large `jq` commands before or alongside actual packet work, even though each task row already names `metadata.input_path`, `metadata.hint_path`, and `metadata.result_path`.

- Observation: in the March 22 run, four knowledge workers produced zero outputs while one worker completed all assigned result paths, so exploration ergonomics dominate throughput before semantic quality does.
  Evidence: `data/output/2026-03-22_01.34.12/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-22_01.34.51/raw/llm/saltfatacidheatcutdown/knowledge/task_status.jsonl` shows `worker-001`, `worker-002`, `worker-003`, and `worker-005` ending almost entirely in `watchdog_killed`, while `worker-004` produced `54` validated packets and `24` invalid outputs.

- Observation: the current watchdog already distinguishes boundary violations from telemetry, but it does not yet distinguish “recently made useful output progress” from “thrashing without outputs.”
  Evidence: the current workspace-worker path in `cookimport/llm/codex_exec_runner.py` classifies commands and tracks command-loop counts, but the broad loop budget still treats post-progress exploration and pre-progress exploration similarly once a worker stays alive.

- Observation: the knowledge prompt already names the right output schema, but that alone is not enough to bias the worker toward concise snippets.
  Evidence: the current knowledge prompt says snippet bodies must be “a short grounded extraction,” yet the saved March 22 run still produced many near-full-chunk snippets that the validator correctly rejected.

- Observation: the surviving productive worker is generating a broader low-trust snippet shape than the current validator fully captures: verbatim block-copy snippets where `body == evidence[0].quote`.
  Evidence: in `data/output/2026-03-22_01.34.12/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-22_01.34.51/raw/llm/saltfatacidheatcutdown/knowledge/workers/worker-004/out/*.json`, all `179` snippet bodies exactly match their sole evidence quote, while only the longest `24` became `semantic_snippet_echoes_full_chunk`.

- Observation: the shared direct telemetry summary already treats recovered outcomes as final truth, so repaired knowledge shards should not keep inflating `invalid_output_shard_count`.
  Evidence: during implementation, `tests/llm/test_codex_farm_knowledge_orchestrator.py::test_knowledge_orchestrator_repairs_near_miss_invalid_shards_once` had to be updated from `invalid_output_shard_count == 1` to `0` because `summarize_direct_telemetry_rows(...)` already prefers final proposal status over stale first-attempt invalid rows.

## Decision Log

- Decision: solve exploration primarily by making the “good path” cheaper, not by adding another pile of prohibitions.
  Rationale: the user explicitly said they do not want workers killed for stupid local exploration. The right fix is to reduce the need for exploration and leave the remaining bad behavior to prompt pressure and loop budgets.
  Date/Author: 2026-03-22 / Codex

- Decision: keep the cheaper per-task seam repo-owned and explicit.
  Rationale: workers should not need to infer the current task row from a broad inventory file. A tiny repo-written sidecar such as `current_task.json`, or a similarly direct single-row task file, gives the worker one obvious next read target.
  Date/Author: 2026-03-22 / Codex

- Decision: keep snippet-echo rejection strict while making the prompt/examples better.
  Rationale: the validator is rejecting genuinely low-value behavior. The fix should make good outputs easier to produce, not silently normalize chunk dumps into “acceptable” snippets.
  Date/Author: 2026-03-22 / Codex

- Decision: extend validation to reject substantial verbatim block-copy snippets, not just longest whole-chunk echoes.
  Rationale: the March 22 run shows a worker can mass-produce low-value “snippets” by copying one cited block verbatim. That is still not a grounded extraction even when the current full-chunk threshold does not fire.
  Date/Author: 2026-03-22 / Codex

- Decision: make watchdog tolerance depend partly on recent output progress, not just raw command counts.
  Rationale: a worker that has already written valid packet files is less risky than one that has only explored. The loop guard should reflect that difference.
  Date/Author: 2026-03-22 / Codex

## Outcomes & Retrospective

This plan is now implemented in code and tests. Non-leased workspace workers now get a repo-written `current_task.json` seam, the shared worker contract now points at that seam instead of rewarding manifest spelunking, the knowledge prompt shows both good and bad snippet patterns explicitly, and the validator now rejects both near-full chunk echoes and substantial verbatim evidence-quote copies. The narrow `.venv` suite passed, including the extra recipe and line-role regressions touched by the shared manifest seam.

The main residual risk is model quality: even with a cheaper task loop and better examples, some workers will still write semantically weak snippets. That is acceptable. The point of this change was to eliminate avoidable ergonomics waste and close the known validator blind spot, not to pretend knowledge extraction is deterministic. Benchmark revalidation is still deferred until the user explicitly approves token spend.

## Context and Orientation

Three areas matter for this work.

First, `cookimport/llm/codex_exec_runner.py` is the shared direct-exec seam. It writes the sterile workspace, the shared worker instructions, and the worker manifest. It also houses the workspace-worker loop watchdog and boundary classifier. This is where a cheaper shared per-task seam and a progress-aware loop guard would live.

Second, `cookimport/llm/codex_farm_knowledge_orchestrator.py` is the knowledge stage’s repo-owned runtime. It writes worker prompts, prepares worker-local task files, validates main worker outputs, and drives retry/repair behavior. This is where the knowledge-specific prompt and examples need to become more operationally explicit.

Third, `cookimport/llm/codex_farm_knowledge_ingest.py` and `tests/llm/test_knowledge_output_ingest.py` define and test what counts as a valid semantic packet. This validator already rejects `semantic_snippet_echoes_full_chunk`. That rejection should remain in place. The work here is to make the prompt/examples lead the worker toward valid snippets more reliably and to catch the broader low-trust case where a snippet simply repeats one cited prose block verbatim.

In this repository, a “task row” means one repo-written unit inside `assigned_tasks.json` that already names the packet id, the exact input file path, the hint file path, and the required output file path. A “cheaper per-task seam” means a worker-visible file or sidecar that exposes the next task row directly, so the worker can process one task without first scraping or reformatting the entire assigned queue.

## Plan of Work

Start in `cookimport/llm/codex_exec_runner.py`. Add a small shared helper for “current task row” delivery in workspace-worker mode. The exact artifact name can be `current_task.json` or another clear single-row filename, but it must be repo-written and mirrored into the sterile workspace before the worker turn starts. Keep `assigned_tasks.json` as the authoritative full ordered queue for auditing and progress, but make the per-task sidecar the default worker read target.

In the same module, refine the shared worker contract so it encodes one tiny loop in prose: open the current task row, open the named input and hint files, write the named result path, then move to the next current task row when the repo advances it. The contract should explicitly say not to dump or reformat `assigned_tasks.json` unless the prompt says that is the job. If examples remain in the manifest, they should demonstrate direct row access and a tiny helper, not manifest spelunking.

Then update the workspace-worker loop watchdog only if the shared watchdog still needs loop-pressure tuning after the temp-root relaxation lands. Use the existing worker-local `out/*.json` surface or whatever the loop supervision already tracks. If the worker has recently written new output files or advanced task completion, the loop watchdog should be slower to conclude that the session is pathological. Do not relax true boundary checks. This is about “productive but slightly messy” workers, not about allowing repo drift.

Next, edit the knowledge-specific prompt builder in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. Make the operational loop even more explicit than the shared contract. Add one compact “good snippet” example showing a short claim with a narrow evidence pointer, and one compact negative example showing why a whole-chunk paraphrase or direct block copy is invalid. If the stage already has worker-visible examples or hint sidecars, extend them rather than creating a parallel source of truth.

Finally, update the tests and validator. The shared direct-exec tests should prove that the new per-task sidecar exists and that prompt/manfiest text points to it. The watchdog tests should prove that recent output progress affects loop supervision behavior without weakening the hard boundary checks if that shared tuning is still needed. The knowledge tests should prove the prompt/example surface now explicitly discourages chunk-dump snippets, while the ingest layer rejects both long full-chunk echoes and substantial verbatim block-copy snippets.

## Validation and Acceptance

This change is accepted when all of the following are true:

1. A workspace-manifest regression proves that the worker sees a repo-written per-task seam that is cheaper than scraping `assigned_tasks.json`.
2. A shared worker-contract regression proves the direct-exec instructions tell the worker to process the declared task row directly rather than dump or reformat the whole queue.
3. If shared loop tuning is still needed, a loop-watchdog regression proves that recent output progress materially affects loop supervision without weakening hard boundary enforcement.
4. A knowledge prompt regression proves the stage now shows a positive “good snippet” pattern and a negative “chunk echo or verbatim block copy is invalid” pattern.
5. A knowledge ingest regression proves full-chunk or near-full-chunk snippet bodies are still rejected.
6. A knowledge ingest regression proves a substantial snippet body that simply repeats its sole evidence quote is also rejected as low trust.

## Idempotence and Recovery

The per-task seam should be safe to regenerate on every worker refresh; it is repo-owned state derived from the existing task manifest and should overwrite cleanly. Prompt/example updates are also idempotent and should only rewrite static text.

If the progress-aware watchdog starts letting truly idle workers run much longer than before, back it out and reintroduce the change with a narrower heuristic tied to actual new `out/*.json` writes rather than broad “recent activity.” If the per-task seam and `assigned_tasks.json` ever disagree, stop and fix the repo-owned writer first; the worker should never be left guessing which source of truth wins. If the broader snippet-copy validator starts rejecting obviously good short extracted labels or headings, narrow it to substantial prose copies rather than backing out the whole low-trust check.
