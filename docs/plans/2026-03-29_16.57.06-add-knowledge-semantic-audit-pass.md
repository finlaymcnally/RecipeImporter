## Purpose / Big Picture

After this change, the knowledge stage will no longer move directly from a structurally valid Pass 1 keep/drop result into Pass 2 grouping. Instead, repo code will run one narrow semantic suspicion audit over the accepted Pass 1 rows, write explicit worker-facing repair evidence for only the suspicious rows, and require the same workspace session to patch those rows before grouping can begin. The prompt cleanup will support that loop by making the keep/drop boundary and the "patch only flagged rows" behavior more explicit, without turning the prompt into a new schema redesign.

The user-visible proof is simple. A knowledge shard with mixed memoir/framing plus useful cooking guidance should now show a durable repo-written semantic-audit artifact, a worker-facing repair file when suspicious Pass 1 decisions are detected, and a same-session patch loop that clears those flags before Pass 2 runs. Prompt-preview and workspace sidecars should tell the same story. The change should tighten the worst current failure mode without adding a new micro-segmentation subsystem or a new per-row anchor-claim output contract.

The live architecture already matches the most important high-level recommendation from the external memo: final `knowledge|other` authority belongs in the knowledge stage, not upstream. ## Context and Orientation

The relevant system is the knowledge stage, which is the final semantic owner of reviewable outside-recipe prose. In plain terms: recipe-boundary decides what is inside recipes, `nonrecipe-route` excludes only obvious junk and builds the review queue, and `knowledge-final` is the stage that makes the final call on whether the surviving outside-recipe prose is durable cooking knowledge or just `other`.

The key files are:

- `docs/plain-english-explanation.md`, which explains the five-stage runtime and the current role of `nonrecipe-route` and `knowledge-final`.
- `docs/10-llm/10-llm_README.md`, which is the code-verified contract for the live knowledge runtime, worker files, repair surfaces, and prompt behavior.
- `cookimport/llm/codex_farm_knowledge_jobs.py`, which plans the knowledge shard input files under `knowledge/in/*.json`.
- `cookimport/llm/codex_farm_knowledge_models.py`, which defines the live worker output schema: top-level `packet_id`, ordered `block_decisions`, and `idea_groups` with grounded snippets.
- `cookimport/llm/codex_farm_knowledge_ingest.py`, which validates worker outputs and classifies failures for repair.
- `cookimport/llm/knowledge_workspace_tools.py`, which owns the repo-written worker helper surfaces, contract markdown, examples, validation feedback, and worker tool script.
- `cookimport/llm/knowledge_stage/planning.py`, `runtime.py`, `recovery.py`, and related stage modules, which control the same-session worker loop and the transition from Pass 1 to Pass 2.
- `llm_pipelines/prompts/recipe.knowledge.packet.v1.prompt.md`, which is the live knowledge prompt template.

Terms used in this plan:

- A "shard" is one ordered owned slice of the review queue. One worker session owns a shard through Pass 1 and Pass 2.
- "Pass 1" means the phase where the worker labels each owned block as `knowledge` or `other`.
- "Pass 2" means the phase where the worker groups only the kept `knowledge` blocks into one or more idea groups with grounded snippets.
- A "semantic suspicion audit" in this plan means a repo-owned checker that flags rows whose Pass 1 decisions look risky or overly smeared, then writes evidence for the worker to reconsider them. It is not allowed to silently change labels.

The live contract already has the right broad shape. `knowledge-final` is already the final semantic owner. The billed input payload is already compact and mechanical. The worker already has a same-session phase loop, repair files, and feedback files. What is missing is a first-class semantic suspicion pass between accepted Pass 1 output and Pass 2 grouping.

## Milestones

## Milestone 1: Define the Pass 1 semantic suspicion surface

At the end of this milestone, the knowledge runtime will have one explicit place where repo code can say, "this Pass 1 ledger is structurally valid, but these specific rows look semantically suspicious and should be rechecked before grouping starts." That surface must be durable enough for debugging and narrow enough that it does not become a second semantic owner.

Implement one repo-owned audit payload for each shard after Pass 1 validation succeeds but before Pass 2 begins. The audit payload should identify only suspicious rows and the evidence for suspicion. It should not rewrite labels. The worker-facing repair view should be a bounded local file, ideally reusing the existing repair-file convention rather than inventing a whole new control plane. The durable run-level artifact can live beside the normal shard artifacts in the knowledge runtime tree so later debugging and stage observability can explain why a shard paused before Pass 2.

Acceptance for this milestone is a focused test that starts from a valid Pass 1 ledger, runs the audit, and proves that suspicious keep/drop rows produce a durable audit artifact plus a worker-facing repair file instead of an immediate Pass 2 advance.

## Milestone 2: Gate Pass 2 on clearing semantic suspicion flags

At the end of this milestone, a shard cannot move from Pass 1 to Pass 2 merely because the Pass 1 ledger is structurally valid. If the semantic suspicion audit flags rows, the shard must remain in the same-session Pass 1 repair loop until those flags clear or the worker fails closed.

The important part is that this still uses the current worker philosophy. The repo writes bounded evidence, the worker edits only the named local ledger rows, the worker runs the same check/install commands, and Pass 2 starts only after the repo accepts the repaired Pass 1 state. There should be no detached semantic judge and no silent deterministic relabeling.

Acceptance for this milestone is a focused runtime test where a structurally valid but semantically suspicious Pass 1 ledger triggers repair feedback, the worker patches only the flagged rows, and only then does the runtime advance to Pass 2.

## Milestone 3: Tighten prompt and worker guidance for the new audit loop

At the end of this milestone, the prompt and worker-side contract text will support the new audit loop cleanly. The prompt should more explicitly discourage blanket "this whole region feels useful" or "this whole region feels memoir" judgments in mixed packets. The worker-facing contract should also explain how to respond when repo feedback flags a narrow subset of rows.

This is a cleanup milestone, not a redesign. Keep the current utility-first framing, weak-adjacency warning, and grounded-snippet requirements. Tighten wording around mixed packets, keep/drop discipline, and "patch only flagged rows". Do not add the heavier `reason_code + transferable_claim + anchor_line_ids` schema in this milestone.

Acceptance for this milestone is prompt-preview or prompt-builder coverage proving that the revised wording is present, plus worker-side sidecar coverage showing that semantic-audit repair instructions are readable from the normal local files.

## Milestone 4: Extend tests, status artifacts, and docs

At the end of this milestone, the new audit pass will be visible in tests and docs, and a novice will be able to understand the live contract without reading old plans or external notes.

Update the docs so they say, in current tense, that Pass 1 structural acceptance alone is not enough: repo code also runs a semantic suspicion audit before Pass 2 grouping. Update tests so future contributors cannot silently remove that gate. Update runtime-status and observability surfaces only as much as needed to expose when a shard paused for semantic suspicion repair versus ordinary structural validation repair.

Acceptance for this milestone is focused green tests plus updated docs that describe the new flow truthfully.

## Plan of Work

Start in the knowledge worker helper and stage runtime, because those are the authoritative seams for the current same-session loop. Add one repo-owned semantic suspicion checker that consumes the authoritative Pass 1 ledger and the immutable shard input payload. The checker should produce a narrow result: either no flags, or a list of flagged row ids with evidence strings and suspicion codes. The evidence can include reviewer-facing snippets such as "first-person narrative marked knowledge", "decorative heading marked knowledge", "isolated knowledge row surrounded by memoir-like rows", or "long uninterrupted run of identical keep/drop decisions across mixed-signal blocks". The checker may use deterministic surface cues and existing reviewer categories, but it must not silently rewrite categories.

Once that checker exists, wire it into the handoff between Pass 1 and Pass 2. The runtime should only advance to Pass 2 when Pass 1 is structurally valid and the semantic suspicion checker returns no open flags. When flags exist, rewrite the normal feedback surfaces so the worker sees a repair request for only the flagged rows. Reuse the current same-session repair posture: freeze accepted rows, expose only the unresolved or suspicious rows for editing, and require the worker to rerun the normal phase check/install flow.

After the runtime loop is sound, tighten the prompt and worker text. Edit `llm_pipelines/prompts/recipe.knowledge.packet.v1.prompt.md` so it more explicitly warns against packet-wide threshold smearing in mixed packets and tells the model to classify blocks on their own merits before grouping. Edit `cookimport/llm/knowledge_workspace_tools.py` so `OUTPUT_CONTRACT.md`, examples, and feedback text teach the worker how to respond to semantic suspicion repair files without implying that repo code already knows the correct semantic answer.

Finish with focused tests and doc updates. The tests must prove that semantic suspicion can block Pass 2, that repairing flagged rows clears the gate, and that grouped-snippet validation still works after the new pass is inserted. The docs must describe the live knowledge loop as: Pass 1 classify, repo structural validation, repo semantic suspicion audit, same-session patch if needed, then Pass 2 grouping.

## Validation and Acceptance

The acceptance story is behavioral.

First, construct or reuse a focused fixture where a Pass 1 knowledge ledger is structurally valid but semantically suspicious. A good example is a mixed shard where a decorative or memoir-like heading is marked `knowledge`, or where one isolated `knowledge` row sits inside an otherwise memoir/framing cluster. Run the focused worker/runtime tests and expect the shard to remain in Pass 1 with repo-written repair evidence instead of advancing immediately to Pass 2.

Second, prove the happy path after repair. The same fixture should be able to patch only the flagged rows, rerun the normal phase checks, clear the semantic suspicion gate, and then advance into Pass 2 grouping without reopening already accepted unflagged rows.

Third, prove that prompt and sidecar wording match the live behavior. Prompt-preview and worker-sidecar tests should show the tightened language around mixed packets, weak adjacency, and patch-only-flagged-row behavior.

Fourth, prove that observability remains truthful. Stage or runtime summary tests should show when a knowledge shard paused for semantic suspicion repair, when it recovered, and when it failed closed without flattening that situation into a generic structural validation error.

This plan is complete when a human can point to one test or fixture showing "valid Pass 1, semantic suspicion detected, same-session patch, then Pass 2" and another showing "valid Pass 1, no semantic suspicion, immediate Pass 2", while the docs and prompt surfaces tell the same story.

## Idempotence and Recovery

This plan is safe to implement incrementally. It changes worker feedback, runtime gating, prompt wording, and tests, but it does not require regenerating historical benchmark data or mutating checked-in benchmark artifacts. If a partial implementation leaves the knowledge stage unable to advance, recover by first restoring the old Pass 1 to Pass 2 gate while keeping any new audit helper behind an unused call site, then re-introduce the semantic gate once the worker feedback files and tests agree on the intended loop.

Do not "recover" by teaching deterministic code to rewrite semantically suspicious rows automatically. The whole point of this plan is to keep the final semantic choice with the LLM worker while letting repo code package suspicion evidence and enforce the phase boundary.

If the semantic suspicion checker becomes noisy during development, narrow it rather than weakening the overall gate. The right failure mode is "too few high-signal flags that the worker can patch" rather than "many low-confidence flags that make the same-session loop unusable."
