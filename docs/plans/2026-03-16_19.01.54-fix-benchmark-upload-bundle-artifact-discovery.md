# Fix Benchmark Upload Bundle Artifact Discovery

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, rebuilding an `upload_bundle_v1` from an existing benchmark session should faithfully describe the artifacts that the run actually produced. A person reviewing a fresh single-offline benchmark run should see Codex prompt logs, prompt samples, knowledge-manifest evidence, prompt-budget totals, and per-run knowledge locators reflected correctly in the bundle instead of seeing false `missing` statuses. The proof is concrete: regenerate the bundle for the existing session at `data/golden/benchmark-vs-golden/2026-03-16_18.10.25/single-offline-benchmark/saltfatacidheatcutdown/` and confirm that the index and overview now report the real codex prompt and knowledge artifacts.

This plan intentionally does not change benchmark scoring or model behavior. The run itself already works. The task is to repair the existing-output bundle builder so reviewer-facing reporting matches the post-refactor artifact layout.

- Observation: the benchmark run itself is healthy; the defect is in reviewer-facing bundle discovery.
  Evidence: both processed import reports have empty `errors` arrays, both eval reports exist, and the codex processed output contains `08_nonrecipe_spans.json`, `09_knowledge_outputs.json`, `raw/llm/.../recipe_manifest.json`, and `raw/llm/.../knowledge_manifest.json`.

- Observation: the codex run wrote `full_prompt_log.jsonl` under `codexfarm/prompts/`, but `_resolve_full_prompt_log_path(...)` does not search that location.
  Evidence: the function currently searches manifest artifact keys plus `run_dir/full_prompt_log.jsonl` and `run_dir/codexfarm/full_prompt_log.jsonl`, while the observed file is `codexfarm/prompts/full_prompt_log.jsonl`.

- Observation: the current run shape leaves `run_manifest.artifacts.pred_run_dir` unset, so helpers that require `_resolve_prediction_run_dir(...)` behave as if the prediction run is absent.
  Evidence: `codexfarm/run_manifest.json` has no `pred_run_dir` key, and `run_diagnostics` reports `preprocess_trace_failures_status = missing_prediction_run`.

- Observation: knowledge work is real but is invisible to `analysis.knowledge`.

- Observation: by the time implementation started, the live code already handled `prompts/full_prompt_log.jsonl`; the stale on-disk bundle was older than the current resolver.
  Evidence: a pre-fix rebuild into `/tmp/pre_fix_upload_bundle_v1` reported `codexfarm.full_prompt_log_status = complete` and `prompt_warning_aggregate_status = written`, while knowledge counts still stayed at zero.

- Observation: processed-output knowledge manifests cannot become bundle-local just by returning an absolute path.
  Evidence: non-high-level bundle payload collection only sweeps files under the session root, so external manifests need a derived payload row to make `knowledge_manifest_in_bundle` true and keep `knowledge_by_run` navigable.

## Plan of Work

Start by making artifact discovery explicit and deterministic for the current post-refactor run layout. The bundle builder must stop assuming that all interesting artifacts live under `prediction-run/`. In this repositoryãs current benchmark output, some reviewer-facing artifacts are at the benchmark run root, some are under `run_dir/prompts/`, and some are only present under the processed output root referenced by `run_manifest.artifacts.processed_output_run_dir` or `run_manifest.artifacts.stage_run_dir`.

The first code change is to introduce small helper functions that expose the artifact roots available for one benchmark run. Add a helper in `scripts/benchmark_cutdown_for_external_ai.py` that inspects `run_manifest.json` and returns the concrete search roots that exist for that run. At minimum include the benchmark run root itself, `run_dir/prompts`, the resolved prediction-run directory if one exists, and the processed-output directory from `artifacts.processed_output_run_dir` or `artifacts.stage_run_dir` when those paths exist. Keep the return value simple: a dictionary or named tuple of concrete `Path` objects is enough. Do not build a speculative generic registry; the helper should only represent current concrete roots.

Then update `_resolve_full_prompt_log_path(...)` so it checks `run_dir/prompts/full_prompt_log.jsonl` before falling back to older locations. This change should be enough for `build_run_record_from_existing_run(...)` to mark the codex runãs prompt log as complete, which in turn should repair `run_diagnostics.full_prompt_log_status` and the bundle topline fields that summarize prompt-log availability.

Next update `_resolve_prompt_budget_summary_path(...)`. Change its signature so it can see the benchmark run root in addition to the prediction-run directory, and make it search `run_dir/prompt_budget_summary.json` before or alongside `prediction-run/prompt_budget_summary.json`. The real observed run stores the summary at the run root, so the helper must accept that as canonical current output rather than a fallback accident. Update all call sites accordingly.

Then update `_resolve_knowledge_manifest_path(...)` so it does not depend exclusively on `_resolve_prediction_run_dir(...)`. When `pred_run_dir` is unavailable, inspect the processed-output root from the run manifest and search `raw/llm/*/knowledge_manifest.json` there. The bundle should report `knowledge_manifest_status = written` whenever the processed output already contains the knowledge manifest for that run.

After that, revisit the derived diagnostics that currently emit `missing_prediction_run`. In the inspected session, `preprocess_trace_failures_status` reports `missing_prediction_run`, but the deeper truth is that the run wrote enough downstream output for the bundle to reason about it. Update `_build_preprocess_trace_failure_rows(...)` and any helper it depends on so that it can use the best available extracted-archive source for the current run layout. Prefer an explicit prediction-run directory when present, but when it is absent fall back to current benchmark replay artifacts or processed-output artifacts already referenced by the run manifest. If the full prompt log is still genuinely missing after the resolver fix, the correct final status is `missing_full_prompt_log`, not `missing_prediction_run`.

Once discovery is fixed, tighten per-run locator generation. In the `knowledge_by_run` row-locator construction, remove basename-only fallback for per-run knowledge artifacts such as `prompt_budget_summary.json`, prompt samples, and knowledge manifests. Per-run locators should resolve from explicit per-run relative paths only. If a specific run has no discovered path, leave that locator null. This is important because the current basename fallback lets the `vanilla` row pick up the `codexfarm` payload row, which is a correctness bug.

Finally, extend the regression tests in `tests/bench/test_benchmark_cutdown_for_external_ai.py`. Add a test that constructs the exact problematic layout: a codex run with `prompts/full_prompt_log.jsonl`, `prompts/prompt_type_samples_from_full_prompt_log.md`, `prompts/prompt_extract_knowledge_optional.txt`, `prompt_budget_summary.json` at the run root, a processed-output root containing `raw/llm/<slug>/knowledge_manifest.json`, and no `artifacts.pred_run_dir`. Build the bundle and assert that the codex run is reported as complete for prompt-log status, prompt-sample status, prompt-budget status, and knowledge-manifest status, with non-zero knowledge call counts. Add a paired-run assertion that the `vanilla` row in `knowledge_by_run` does not point at the codex prompt-budget row.

If implementing the processed-output fallbacks needs a small amount of plumbing from run manifest artifacts into helper calls, do that as a narrow additive change. The plan is complete only when the tests capture the real run shape and the regenerated bundle for the inspected session shows the corrected reviewer-facing fields.

## Idempotence and Recovery

The helper and test edits are safe to apply incrementally. Re-running the targeted pytest command is idempotent. Rebuilding the bundle with `overwrite=True` and `prune_output_dir=False` is also safe for this task because it rewrites the three bundle files in place without deleting the larger benchmark session.

If a helper change goes too broad and begins binding the wrong runãs artifacts, back out only the overly permissive search roots or basename fallback logic. Do not ãsolveã missing discovery by reintroducing cross-run basename matching. Returning `None` for one run is safer than binding another runãs row.

If the processed-output fallback proves ambiguous, prefer a narrower first pass: resolve only the run-root prompt files and run-root prompt-budget summary, add tests for those, and then extend the knowledge-manifest fallback in a second commit. Keep the plan updated if you split the work this way.
