# Knowledge Grouping Repair Structural Salvage

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the knowledge grouping step should stop dropping otherwise-valid knowledge packets just because the model made a small structural mistake in the proposed-tag shape or because the repair loop forgot the original root cause. A human should be able to inspect a case like `saltfatacidheatcutdown.ks0000.nr`, see a grouping answer that is semantically fine but slightly malformed, and then watch repo-owned code salvage the structure, send one precise repair when salvage is impossible, and keep the repair loop focused on patching the broken groups instead of rewriting the whole packet.

The intended user-visible outcome is fewer `knowledge_block_missing_group` shard failures, fewer reviewed rows falling back to `OTHER` after valid review work, and clearer artifacts that explain the real reason a repair was requested.

## Surprises & Discoveries

- Observation: the model often understands the content well enough before the packet dies.
  Evidence: in `saltfatacidheatcutdown.ks0000.nr`, the first grouping answer split the kept rows into sensible topics like Salt fundamentals and Salt types, but the proposal payload failed validation.
- Observation: the validator seam is stricter than the model’s natural proposed-tag output shape.
  Evidence: the current contract requires `display_name` on each `grounding.proposed_tags[]` entry and also requires `why_no_existing_tag` plus `retrieval_query` at the group-answer level rather than nested under `grounding` or a proposed tag.
- Observation: later repair attempts lose the original problem statement.
  Evidence: once the first structurally close answer fails, later repair packets surface generic fallout such as missing group coverage and missing `group_id`, even though the real root cause was proposal-shape invalidity.
- Observation: the current repair loop invites semantic drift.
  Evidence: in the saved repairs for `ks0000`, the model moved from “fix the proposal object” to collapsing the whole packet into one giant Salt group and then malformed the answer further.
- Observation: this failure family is worth fixing even if the broader `KNOWLEDGE` versus `OTHER` boundary remains noisy.
  Evidence: these rows were already kept for review and then lost during grouping/finalization, so this is a direct recoverable runtime loss rather than an ontology disagreement.
- Observation: one invalid proposed tag often fans out into secondary validator complaints.
  Evidence: the new preserved-root-cause runtime test shows an invalid proposed `category_key` also triggers `approved_proposal_tag_required` and `group_grounding_required` once deterministic validation strips the bad proposal row.

## Context and Orientation

The relevant code lives under `cookimport/llm/knowledge_stage/`. In this repository, the knowledge stage has two semantic steps. The first step is classification: decide whether each non-recipe row is worth keeping for review. The second step is grouping: take the kept rows, split them into contiguous idea groups, and assign one shared grounding story to each group. The final row-level grounding is projected from those group answers.

`cookimport/llm/knowledge_stage/structured_session_contract.py` owns the inline grouping packet shape, the inline grouping prompt text, and the parser that translates a model `groups` answer back into row-keyed answers. This is where the model currently learns how proposed tags must be returned.

`cookimport/llm/knowledge_stage/task_file_contracts.py` owns structural validation for grouping answers. It enforces exact row coverage, exact group coverage, valid grouping continuity, valid existing tag keys, and the current proposed-tag contract. Today that contract is stricter than the model’s most natural output shape: each proposed tag needs a `display_name`, and each group that proposes new tags must also provide `why_no_existing_tag` and `retrieval_query` at the group answer level.

`cookimport/llm/knowledge_stage/workspace_run.py` owns the live inline worker loop. It builds grouping packets, records parse/validation artifacts, issues repair followups, merges accepted answers, and performs full-batch revalidation before promotion. This is where the repair loop currently loses the original root cause and starts emitting generic missing-group fallout.

The saved evidence for the main failure is under `data/output/2026-04-12_22.46.00/.../nonrecipe_finalize/` and the related report is `docs/reports/2026-04-12_21.55.25_saltfatacidheatcutdown-knowledge-failure-report.md`. The key packet is `saltfatacidheatcutdown.ks0000.nr`: classification kept 16 rows for review, grouping found sensible topics, but the final packet failed with `knowledge_block_missing_group` after multiple repairs because the contract and repair feedback kept drifting away from the true structural issue.

The main test surfaces are `tests/llm/test_knowledge_grouping_contract.py`, `tests/llm/test_knowledge_structured_session_contract.py`, `tests/llm/test_knowledge_orchestrator_runtime_leasing.py`, and `tests/llm/test_knowledge_orchestrator_post_merge_repair.py`. These tests are the preferred way to validate this work because they can reproduce the contract and repair behavior without running a new benchmark.

This plan is tracked by parent bead `recipeimport-hrl`. The current child issues are `recipeimport-hrl.1` for structural proposal-shape salvage, `recipeimport-hrl.2` for prompt and repair contract clarity, `recipeimport-hrl.3` for root-cause-preserving repair feedback, and `recipeimport-hrl.4` for eventual plan deletion after user approval.

## Plan of Work

Start by adding regression coverage for the exact failure family. Build one focused test that passes a semantically coherent grouping answer with proposal metadata in the “wrong but obvious” place, such as group-level justification nested under a proposed tag or a missing `display_name` that can be safely derived from the topic label. The test should demonstrate current failure and the intended structural salvage behavior. Add a second test that starts from a structurally close first answer, then runs a repair followup and proves the repair packet still carries the original root cause instead of degenerating into only missing-group symptoms.

Next, implement a small normalization seam in `task_file_contracts.py` or the closest lower-level helper that keeps the fix structural. The normalization should accept a narrow list of obvious proposal-shape mistakes and rewrite them into the validator’s canonical shape before final validation. Examples include moving `why_no_existing_tag` and `retrieval_query` from an obvious nested location up to the group-answer level, accepting the legacy singular `proposed_tag` alias when present, and filling a missing `display_name` only when there is exactly one clear source such as the already-provided topic label or proposed key. This normalization must not invent new semantic tags, rewrite topic labels, or merge/split groups.

Then tighten the grouping prompt and repair prompt in `structured_session_contract.py` so the proposed-tag contract is explicit. The prompt should include one short valid example for existing tags and one short valid example for proposed tags. It should say plainly that `why_no_existing_tag` and `retrieval_query` belong on the group object, not inside `grounding` or inside a proposed tag. The repair prompt should stop implying that the whole packet must be reconsidered from scratch. Instead it should pass the previously accepted groups back to the model, mark only the invalid groups as needing repair, and instruct the model to preserve existing group boundaries and topic labels unless the validator feedback explicitly says the grouping itself is wrong.

After that, change the repair loop in `workspace_run.py` so later retries preserve the first real contract failure. The repair packet should carry a compact packet-level root-cause summary and a group-scoped list of invalid paths or invalid groups, not just the row-level fallout caused by failed validation. If the first error was “proposed tags missing `display_name`,” that message should still be visible on later retries even if unresolved rows also now show up as missing group coverage. The repair packet should also include the previous `groups` answer so repair can behave as a patch rather than a semantic rewrite.

Finally, update the short docs. `docs/10-llm/10-llm_README.md` and `cookimport/llm/knowledge_stage/README.md` should explain that grouping repair now does limited structural proposal-shape normalization, keeps repair feedback rooted in the first contract miss, and treats later repair attempts as edits to the previous groups rather than fresh regrouping unless told otherwise.

## Idempotence and Recovery

This work is safe to iterate on because it changes only repo-owned prompt text, validation, repair packet construction, tests, and docs. Re-running the focused tests is safe. If the normalization seam proves too permissive, the fallback is to narrow it to only metadata relocation and exact-field synthesis while leaving every semantic decision untouched. If repair feedback becomes harder to read, compare the first saved repair packet and the second saved repair packet for the same shard: the second packet should still surface the original contract miss rather than only secondary missing-group fallout.

Because no new benchmark runs are authorized by default, the main recovery tool is saved-artifact replay plus focused tests. If a patch regresses promotion or final validation, inspect the saved grouping packet, the normalized edited answer, and the validator metadata together to confirm that only structure changed and that group semantics stayed intact.
