## Context and Orientation

The relevant runtime and test surfaces span multiple owners.

`tests/conftest.py`, `pytest.ini`, and `scripts/test-suite.sh` define how broad and narrow pytest runs are supposed to execute in this repository. Broad runs should go through the wrapper script, which exports `COOKIMPORT_TEST_SUITE=1` and preserves the repo’s compact-output and `--capture=sys` defaults.

`tests/architecture/test_ai_readiness_boundaries.py` contains architecture boundary assertions that intentionally pin some modules to maximum sizes or public-facade shapes. Those tests are useful only when they still reflect current architecture truth. If a file grew because the intended public seam drifted, that is a code/architecture regression. If the test still points at an old seam or stale size cap, that is a stale or overspecified test and the test should be updated with docs.

`cookimport/staging/nonrecipe_stage.py` is the first concrete file already implicated by the current failures. The architecture docs say it is supposed to be a thin public seam whose owned logic lives in smaller helper modules. That means the line-count failure must be classified carefully: either the code regressed by absorbing too much logic back into the seam, or the test’s exact cap no longer matches the intended public shape.

The LLM, parsing, Label Studio, and benchmark packages also appear in the cached failures. Because many of those areas use subprocesses, watchdogs, or multiprocessing helpers, some failures may be environment-specific in this WSL-like runtime. Those should be grouped explicitly so they do not get mistaken for semantic regressions.

## Plan of Work

First, gather the current failure set in a form that can be clustered without rerunning the entire suite after every small change. Because the old cache proved stale, the next fresh sources are a cache-cleared wrapper run plus scoped reruns of any new representative nodeids. The goal is to name failure families such as “stale architecture size caps,” “multiprocessing listener/socket failures,” “broken imports/unresolved names,” or “changed semantic contracts in knowledge/line-role tests.”

Second, work the highest-leverage cluster first. A high-leverage cluster is one where many failures share one cause, or where the failures block confident interpretation of the rest of the suite. For each cluster, explicitly record whether it is a regression, an intentional contract change, an outdated test, or an environment-specific test/runtime issue before editing code or tests.

Third, when a contract really changed, update the owning docs in the smallest correct doc slice instead of leaving the test and code to disagree silently. Architecture-facing changes belong in `docs/01-architecture/01-architecture_README.md`; test-runtime contract changes belong in `docs/12-testing/12-testing_README.md`; other changes should go to the owning domain doc.

Finally, rerun the narrow tests that prove the touched cluster is fixed, then periodically rerun the full wrapper-driven suite to measure whether the cluster work actually reduced the global failure count. Keep this plan current after each substantive discovery or repair.

## Validation and Acceptance

Acceptance is behavioral, not cosmetic.

The suite recovery effort is successful when the repository can produce a trustworthy full-suite result again. The strongest form is `./scripts/test-suite.sh full` passing. A weaker but still acceptable interim state is a much smaller remaining failure set where each cluster is documented and honest: environment/socket issues are separated from stale tests, stale tests have been rewritten or removed, and real regressions have been repaired with narrow reruns proving the fix.

Every repaired cluster must have:

- a recorded classification,
- narrow validation runs in `.venv`,
- updated docs when a user-visible or architecture-visible contract changed,
- and an updated note in this plan describing the evidence.

## Idempotence and Recovery

This plan is safe to execute incrementally. Cached failure data may change as tests are rerun, so keep snapshots or summaries in this document when they matter. If a broad run becomes too slow or too noisy to finish, stop it honestly, preserve the partial evidence, and continue with narrow clustered reruns; do not invent a final suite state from a partial run.

When a raw pytest rerun is necessary, preserve `--capture=sys` unless there is a very deliberate reason not to. Otherwise pytest can fall back to temp-file behavior that adds runner failures unrelated to the product code.
