# ExecPlan: Freeform Label Studio focus-only task text

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, Label Studio freeform tasks show only focus (labelable) rows, while AI prelabel prompts still include surrounding context rows. This makes Label Studio task text closer to the source rows that are actually being labeled and simplifies downstream dedupe, without regressing AI boundary context.

## Surprises & Discoveries

- Observation: prelabel prompt construction had a hard dependency on `source_map.blocks` offset-mapped into `segment_text`, so changing `segment_text` scope required explicit context metadata fields for prompts.
  Evidence: `_build_prompt(...)` previously read prompt block text only by slicing `segment_text` with `source_map.blocks[*].segment_start/end`.

- Observation: freeform coverage warnings were computed only from `source_map.blocks`, which would under-report coverage once `blocks` became focus-only.
  Evidence: `compute_freeform_task_coverage(...)` originally counted only `source_map.blocks` indices.

## Decision Log

- Decision: Keep annotation offsets anchored to focus-only `segment_text` and keep `source_map.blocks` as the single offset-authoritative map.
  Rationale: Export/eval contracts already depend on this deterministic offset path; minimizing offset semantics changes lowers regression risk.
  Date/Author: 2026-02-23 / Codex

- Decision: Introduce separate prompt-context arrays (`context_before_blocks`, `context_after_blocks`) rather than overloading `source_map.blocks` with non-offset context entries.
  Rationale: Mixed offset/non-offset entries in one list are brittle and complicate parser assumptions.
  Date/Author: 2026-02-23 / Codex

- Decision: Keep legacy payload compatibility in `prelabel._build_prompt(...)` by falling back to old behavior when new context arrays are absent.
  Rationale: Existing artifacts/tests/tasks should continue to prelabel correctly.
  Date/Author: 2026-02-23 / Codex

## Plan of Work

First split freeform payload semantics in task generation: `segment_text` and `source_map.blocks` should represent only focus rows, while context rows move into dedicated source_map arrays for prompt use.

Then update prelabel prompt construction to merge context-before + focus + context-after lines when new arrays are present, while preserving legacy prompt behavior when arrays are missing.

Then update coverage accounting and tests to reflect the new data contract and verify both new and backward-compatible paths.

Finally, update docs/conventions with the new contract and run focused verification tests.

## Validation and Acceptance

Acceptance criteria:

- Freeform tasks with `segment_focus_blocks < segment_blocks` store focus-only rows in `data.segment_text`.
- `data.source_map.blocks` offsets map only those focus rows.
- AI prelabel prompts still include context rows before/after focus when available.
- Legacy task payloads (without context arrays) still prompt/label correctly.
- Focused labelstudio tests pass.

## Idempotence and Recovery

All edits are additive and deterministic. If compatibility issues arise in old artifacts, prompt generation falls back to legacy `source_map.blocks` behavior when new context arrays are absent.

## Interfaces and Dependencies

Updated freeform task source_map contract:

- `source_map.blocks`: focus-only blocks with `segment_start`/`segment_end` offsets into `segment_text`.
- `source_map.context_before_blocks`: optional prompt-context rows before focus (block-index + text metadata).
- `source_map.context_after_blocks`: optional prompt-context rows after focus (block-index + text metadata).

`prelabel._build_prompt(...)` must continue to support both:

- new payloads with split focus/context fields, and
- legacy payloads where `source_map.blocks` includes full context.
