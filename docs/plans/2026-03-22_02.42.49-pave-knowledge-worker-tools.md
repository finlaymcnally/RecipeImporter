# Pave the Knowledge Worker Workspace With Repo-Provided Tools

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a knowledge worker should start in its bounded workspace, open one obvious repo-written task row, use one obvious repo-written helper CLI, and finish a packet without inventing queue digests, packet scaffolds, or ad hoc validators. The worker should spend its effort deciding what is real reusable knowledge and which short grounded snippets to keep. The repository should own the repetitive mechanics.

The user-visible success condition is that a generated knowledge worker root is self-explanatory by inspection. A novice should be able to open the workspace and immediately see the approved loop: open `worker_manifest.json`, open `current_task.json`, use `tools/knowledge_worker.py` to orient or scaffold when needed, validate locally, and install to the declared `out/<task_id>.json` path. A later benchmark rerun should show less shell churn and fewer invalid packet outputs, but this plan does not require a live rerun until the user explicitly approves token spend.

## Surprises & Discoveries

- Observation: the shared workspace runner already moved the repository closer to the right shape.
  Evidence: `cookimport/llm/codex_exec_runner.py` now gives non-leased worker sessions a repo-written `current_task.json` and explicitly tells workers to open it first, which removes much of the need for stage-specific task-index artifacts.

- Observation: knowledge is the one stage that most needs validator reuse instead of a bespoke local checker.
  Evidence: `cookimport/llm/codex_farm_knowledge_ingest.py` now rejects low-trust semantic packet shapes such as non-grounded snippet bodies, full-chunk echoes, and evidence-surface copying. A helper that reimplements those rules by hand would drift quickly.

- Observation: the original plan overfit to the absence of `current_task.json`.
  Evidence: the first draft proposed a separate `task_overview.jsonl` because the worker only had `assigned_tasks.json`. The shared runner now provides a cheaper orientation seam that all three stages can share.

- Observation: the right cross-stage unification point is workspace shape and helper UX, not validator semantics.
  Evidence: recipe, line-role, and knowledge all need the same discoverability and helper verbs, but only knowledge has snippet-quality checks tied directly to ingest-time semantic rules.

- Observation: the local invalid-example packet has to be built from the real task-local ids and a long enough copied surface, or the helper can no longer prove echo rejection inside a generated worker root.
  Evidence: a generic short `Preface` echo packet was structurally invalid as an example but too short to trip the local snippet-echo guard; building the example from the actual task-local chunk text fixed the helper `check` regression test.

## Context and Orientation

In this repository, the “knowledge worker” is the long-lived direct-exec LLM session that reviews non-recipe task packets after deterministic chunking and seed non-recipe ownership are already known. Each task packet corresponds to one local `in/<task_id>.json` file and one local `hints/<task_id>.md` file. The worker must write exactly one semantic packet-result JSON file to `out/<task_id>.json`.

The current shared direct-exec workspace contract lives in `cookimport/llm/codex_exec_runner.py`. For non-leased worker sessions, the runner already mirrors these files into the sterile execution workspace:

- `worker_manifest.json`, the authoritative workspace index
- `current_task.json`, the cheapest repo-written next task row when the stage does not use packet leasing
- `assigned_tasks.json`, the ordered queue and progress view
- `assigned_shards.json`, the parent-shard inventory
- `hints/`, `in/`, `out/`, and `scratch/`
- optional `OUTPUT_CONTRACT.md` and `examples/`

This plan standardizes knowledge on that same contract and adds a stage-local helper CLI under `tools/knowledge_worker.py`.

The semantic packet contract is knowledge-specific. The worker does not write the compact canonical `v` / `bid` / `r` bundle on the main path. Instead it writes a semantic packet result with top-level `packet_id` and `chunk_results`. Each `chunk_results` row names `chunk_id`, `is_useful`, `block_decisions`, `snippets`, and optional `reason_code`. Each `block_decisions` row names `block_index`, `category`, and optional `reviewer_category`. Each snippet names `body` and `evidence`, and each evidence row names `block_index` and `quote`. Repo code later normalizes and serializes the compact canonical bundle if the semantic result passes validation.

The key files are:

- `cookimport/llm/codex_farm_knowledge_orchestrator.py`, which builds worker prompts, worker roots, and the task metadata the helper should rely on
- `cookimport/llm/codex_farm_knowledge_ingest.py`, which already contains the authoritative packet validation rules
- `cookimport/llm/codex_farm_knowledge_models.py`, which defines the semantic packet schema
- `cookimport/llm/codex_exec_runner.py`, which owns shared workspace mirroring, the worker manifest, and the sandbox/path policy
- `tests/llm/test_codex_farm_knowledge_orchestrator.py`, which should prove the worker root is discoverable and prompt text points to the helper path
- `tests/llm/test_knowledge_output_ingest.py`, which should remain the validator-truth seam

“Scaffold” in this plan means “write the correct packet envelope with correct task-local ids and block ordering already filled in.” It does not mean “make semantic decisions.” “Install” means “validate a draft against the same rules repo ingest will use, then write it to the declared result path.”

## Plan of Work

Implement the knowledge paved road in one narrow stage-local module plus one shared runner adjustment.

First, create a new helper module under `cookimport/llm/` that owns deterministic knowledge workspace tooling. A good name is `cookimport/llm/knowledge_workspace_tools.py`. It should expose pure functions for:

- loading and summarizing task rows from `assigned_tasks.json` and `current_task.json`
- scaffolding a semantic packet draft for one task with exact `packet_id`, owned `chunk_id` rows, and block ordering
- validating a saved draft by routing through the same normalization and validation path used by `cookimport/llm/codex_farm_knowledge_ingest.py`
- installing a validated draft to the task row’s declared `metadata.result_path`
- rendering a self-contained workspace script body that can be mirrored to `tools/knowledge_worker.py`

Second, extend `cookimport/llm/codex_farm_knowledge_orchestrator.py` so each worker root writes:

- `OUTPUT_CONTRACT.md` describing the semantic packet-result contract in worker-facing language
- `examples/valid_semantic_packet.json`
- `examples/invalid_echo_packet.json`
- `tools/knowledge_worker.py`

The helper script must be standard-library-only when mirrored into the workspace. It must be runnable as `python3 tools/knowledge_worker.py ...` from the workspace root and must not depend on repository-relative imports at runtime.

Third, use the same helper verbs as the other two stages:

- `overview` prints the ordered queue concisely from `assigned_tasks.json`
- `show <task_id>` prints the current task’s paths, owned ids, and compact summary
- `scaffold <task_id> [--dest <path>]` emits a correct semantic packet envelope for that task
- `check <json_path>` validates a saved draft and prints a short pass/fail summary
- `install <json_path>` reruns validation and writes the file to the declared `metadata.result_path`

Fourth, update the knowledge worker prompt so the local loop is:

1. open `worker_manifest.json`
2. open `current_task.json`
3. if more context is needed, run `python3 tools/knowledge_worker.py overview` or `show <task_id>`
4. open the named `metadata.hint_path` and `metadata.input_path`
5. if useful, run `python3 tools/knowledge_worker.py scaffold <task_id> --dest scratch/<task_id>.json`
6. edit the draft
7. run `python3 tools/knowledge_worker.py check scratch/<task_id>.json`
8. run `python3 tools/knowledge_worker.py install scratch/<task_id>.json`

The prompt must explicitly say that broad queue dumps and ad hoc shell schedulers are unnecessary because the repo already provides `current_task.json` and the helper CLI.

Fifth, extend `cookimport/llm/codex_exec_runner.py` so `tools/` is a first-class mirrored workspace surface. The shared runner should:

- mirror `tools/` into the sterile execution workspace
- allow `tools/` in the workspace path policy
- advertise `tools_dir` and `mirrored_tool_files` in `worker_manifest.json`

This is the shared unification step. Knowledge should not implement its own separate mirroring logic.

## Validation and Acceptance

This change is accepted when all of the following are true.

The generated knowledge worker root visibly contains `worker_manifest.json`, `current_task.json`, `OUTPUT_CONTRACT.md`, `examples/`, and `tools/knowledge_worker.py`, and the worker manifest explicitly advertises the tools directory and mirrored tool files.

The worker prompt tells a novice to start from `current_task.json`, not from a broad queue dump, and gives the helper workflow in the shared verb language `overview`, `show`, `scaffold`, `check`, `install`.

The scaffold helper emits the correct semantic packet envelope for a representative task, including exact `packet_id`, exact owned `chunk_id` rows, and exact block ordering.

The check helper rejects a bad echo packet that would fail repo ingest and accepts a valid short grounded packet that repo ingest would later accept.

The install helper writes only to the declared `metadata.result_path` and refuses to install a failing draft.

The narrow `.venv` tests listed above pass.

If the user later approves a live benchmark rerun, acceptance also includes a behavioral check: compared with the saved `2026-03-22_01.34.12` knowledge run, workers should spend materially fewer commands on queue spelunking and improvised helper generation, and the dominant invalid-output class should shift away from preventable mechanical failures.

## Idempotence and Recovery

This plan is intentionally additive. Worker roots are regenerated from scratch, so rewriting `OUTPUT_CONTRACT.md`, `examples/`, and `tools/knowledge_worker.py` is safe to repeat. The helper’s `scaffold` command should write to stdout by default or to an explicit destination only. The helper’s `check` command must be read-only. The helper’s `install` command must be the only step that writes to the final declared output path.

If a partial implementation leaves the prompt referring to helper commands that are not yet mirrored into the workspace, stop and make the workspace surface consistent before running tests. If `check` starts disagreeing with `validate_knowledge_shard_output(...)`, stop and route it back through shared validator functions instead of patching special cases into the helper.
