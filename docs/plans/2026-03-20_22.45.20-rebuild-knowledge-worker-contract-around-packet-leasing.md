## Purpose / Big Picture

The current main-worker contract gives a workspace worker too much room to improvise across many packets at once. Even when the worker stays inside the workspace boundary, it can still choose batch automation, blanket metadata heuristics, schema-hunting loops, or other correlated bad behavior because the repo hands it too much authority up front. After this change, the main worker should see one authoritative current packet at a time, one explicit `scratch/` directory for bounded local helper work, and one clear manifest that states what is allowed, what is forbidden, how the worker returns a result, and how the repo advances to the next packet.

The intent is not to make workers weaker in an abstract sense. The intent is to make the good path so obvious and narrow that it is easier than the dangerous path. The worker should not need to infer the contract by exploring the workspace. The contract should be written into the workspace clearly enough that a human reviewer can open the generated manifest and understand what is supposed to happen.

The visible proof is that fake-runner or deterministic tests can drive one long-lived knowledge worker across multiple packets without handing it the whole free-form task queue, and the written manifest plus prompt make the contract obvious to a novice reading the worker root.

## Outcomes & Retrospective

This plan is not implemented yet. Its intended outcome is a main-worker contract that is explicit enough that later validation and recovery logic are cleaning up small misses, not battling a wide-open automation surface. If this plan succeeds, later repair policy should operate on packet-quality problems rather than on “worker invented a whole batch process” problems.

The main open risk is transport complexity. Reusing one long-lived session across multiple leased packets is desirable, but the current runner may not support that shape cleanly on the first attempt. If the first production step needs a temporary compatibility layer, that layer is acceptable only if repo code still controls packet acceptance and packet advancement.

## Context and Orientation

The current knowledge main-worker path is built in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, `cookimport/llm/phase_worker_runtime.py`, and `cookimport/llm/codex_exec_runner.py`. Today, the worker is effectively handed a multi-packet assignment and a workspace full of authoritative-looking files. That invites batch loops, metadata shortcuts, and large correlated errors because the worker can see the entire problem at once and improvise its own execution strategy.

`Packet leasing` in this plan means the repo exposes only one current packet as authoritative work. The worker may know coarse progress like “N packets remain,” but it should not have a free-form authoritative queue that encourages mass generation. The repo should explicitly decide when a packet becomes leased, when it becomes accepted, and when it becomes terminally failed.

The worker root should also stop being ambiguous. It should contain:

- one explicit manifest that says what to open first;
- one obvious current packet file or symlink target;
- one obvious current hint file;
- one obvious result-return target or mechanism;
- one explicit `scratch/` directory for allowed temporary helper work.

The prompt and manifest must answer the same policy questions. If the prompt says `scratch/` is allowed but the runner kills scratch-based shell helpers, the contract is broken. If the runner allows workspace-local loops but the prompt tells the worker never to use them, the contract is also broken. This plan’s job is to align the prompt, the manifest, the workspace layout, and the runner enforcement.

## Plan of Work

### Milestone 1: Make the worker root speak the contract clearly

Start by rewriting the knowledge worker manifest and prompt setup. The worker should not need to infer policy from examples scattered across the repo. The manifest should answer five questions explicitly:

- what file to open first;
- which packet is currently authoritative;
- where temporary files belong;
- how to return the current packet result;
- which command families are forbidden.

The good examples in the manifest should show bounded local file reads, `jq`, local loops, redirects, and writes under `scratch/`. The bad examples should show `/tmp`, parent traversal, network tools, repo-global tools, and inline interpreters such as `python - <<'PY'`.

### Milestone 2: Add packet-leasing progression to the orchestrator

Once the worker root speaks clearly, change the orchestrator to expose one current packet at a time. The repo should decide when a packet moves from pending to leased, when a leased packet is settled, and when the next packet is made current. The worker may keep one long-lived session, but it should not be able to treat the entire packet set as a free-form batch job.

This milestone should make the packet lease visible in runtime state. A developer reading the orchestrator should be able to answer:

- which packet is currently leased to worker X;
- whether that packet is still awaiting a result;
- what event will release the next packet.

### Milestone 3: Add pre-watchdog prevention

The runner should recognize obviously forbidden or off-task command families early. If the worker reaches for `/tmp`, inline interpreters, repo-global tools, or repeated no-result schema-hunting loops, the runtime should reject or redirect that attempt before letting the worker spend minutes on a doomed path and then die ambiguously. Prevention is not an elaborate coaching loop. It is a cheap early correction or stop.

This milestone should keep prevention narrow. A good rule of thumb is: intervene early for bad path families the runtime can classify deterministically; do not invent a second conversational supervisory system.

### Milestone 4: Add deterministic tests for the contract

Add tests that prove:

- packet leasing exposes one authoritative packet at a time;
- the manifest and prompt mention `scratch/` explicitly;
- workspace-local shell helpers remain allowed when they stay on contract;
- `/tmp` and inline interpreters remain forbidden;
- obvious off-task exploration can be cut off before expensive watchdog fallout.

These tests should assert both behavior and the generated worker-facing text. The text matters because the contract is supposed to be legible to the model and to human reviewers.

## Validation and Acceptance

This plan is complete when:

- a deterministic test proves one long-lived worker can process packets sequentially through repo-driven leasing;
- the worker manifest makes `scratch/` allowed and `/tmp` plus inline interpreters forbidden;
- obvious forbidden or off-task command families are rejected or redirected before expensive watchdog fallout;
- a human reviewer can inspect the worker root and tell what path is intended without reading the implementation.

The ideal acceptance transcript is one where the worker root clearly contains a current packet, a current hint, a scratch directory, and a manifest that names those paths directly.

## Idempotence and Recovery

If same-session packet leasing cannot land in one step, a temporary compatibility layer is acceptable, but only if repo code still controls packet acceptance and advancement. Do not fall back to “worker writes final authoritative files for all packets and the repo hopes for the best.” That would preserve the very contract this plan is supposed to remove.

The new manifest and packet layout should be safe to regenerate repeatedly. Re-running the workspace setup should not create contradictory packet pointers or stale current-packet files.

## Artifacts and Notes

The most important artifact is the new worker manifest contract written into the workspace root. It should be obvious enough that a human reviewer can tell what path is intended without reading code.

Other useful artifacts are:

- generated current-packet file layout;
- generated `scratch/` directory;
- deterministic tests that assert contract text as well as runner behavior.
