# Make CodexFarm Prompts Self-Contained With Inline Input Text

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

`docs/PLANS.md` is checked into this repository and this document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

User goal (verbatim intent): CodexFarm agents should receive 100% of required input in the prompt so they only think and output; no searching, no file lookup, no “read JSON from path” dependency.

After this change, recipe pipelines will render prompts that include the full input JSON payload text directly in the prompt body. The observable success signal is prompt logs that show payload text blocks and no instruction to read from a filesystem path.

## Decision Log

- Decision: implement inline placeholder support in CodexFarm first, then migrate recipeimport templates.
  Rationale: recipeimport alone cannot solve this because CodexFarm currently cannot inject input file contents into template text.
  Date/Author: 2026-03-02 / Codex

- Decision: preserve backward compatibility with existing `{{INPUT_PATH}}` templates.
  Rationale: avoids breaking existing packs and allows incremental migration.
  Date/Author: 2026-03-02 / Codex

- Decision: migrate pass4 knowledge and pass5 tags alongside pass1/2/3 instead of only the minimum three recipe passes.
  Rationale: the user goal is a blanket “no file lookup” prompt surface; all active recipe-related pack prompts should follow the same inline contract.
  Date/Author: 2026-03-02 / Codex

- Decision: use plain sentinel markers (`BEGIN_INPUT_JSON` / `END_INPUT_JSON`) around `{{INPUT_TEXT}}` instead of markdown code fences.
  Rationale: avoids malformed prompt structure if payload text ever contains fence-like sequences.
  Date/Author: 2026-03-02 / Codex

## Context and Orientation

There are two repositories involved:

1. CodexFarm runtime repo at `/home/mcnal/projects/shared/CodexFarm`.
2. recipeimport pack repo at `/home/mcnal/projects/recipeimport`.

CodexFarm is the execution engine that renders prompt templates and launches model calls. recipeimport supplies pipeline JSON definitions, prompt templates, and output schemas. Because prompt text is rendered inside CodexFarm, inline payload support must exist there before recipeimport templates can embed full input JSON.

“Inline payload prompt” means the full input file contents are present directly in the prompt text under a dedicated section. The model should not need to access disk or read by path.

## Plan of Work

Milestone 1 (CodexFarm rendering support): Update `src/codex_farm/pipeline_spec.py` so `render_prompt_template(...)` also replaces `{{INPUT_TEXT}}` with full input file contents (`utf-8`). Keep `{{INPUT_PATH}}` behavior unchanged. If needed, add a small helper that reads input text once and applies both substitutions deterministically.

Milestone 2 (CodexFarm contract + lint + tests): Add an optional pipeline config field such as `prompt_input_mode` with values `path` or `inline` (default `path`). In inline mode, pack lint should fail if prompt template omits `{{INPUT_TEXT}}`. Add/adjust tests in `tests/test_pipeline_spec.py`, `tests/test_pack_lint.py`, and worker-facing tests to prove inline rendering works and path mode remains stable.

Milestone 3 (recipeimport migration): Update recipeimport templates in `llm_pipelines/prompts/recipe.chunking.v1.prompt.md`, `recipe.schemaorg.v1.prompt.md`, and `recipe.final.v1.prompt.md` (plus pass4/pass5 if desired) to consume inline payload blocks. Replace path-read instructions with “use the provided payload below.” Update recipeimport tests that currently assert presence of `{{INPUT_PATH}}` so they assert the intended placeholder contract (or accept both during migration).

Milestone 4 (acceptance run): Execute one small CodexFarm run through recipeimport and verify logged prompts include embedded JSON payload text and no file-read dependency language.
