## Structured Restatement Of User Desire

The repository should stop teaching the model to operate a little shell-driven file choreography and instead let it behave like a normal coding model using one compact editable file. After the fixed-assignment reset lands, each live Codex Exec worker should receive one cutdown task file that already contains the full semantic input it needs, read that file once, edit only the answer fields inside that same file, optionally patch those same answer fields once if validation feedback comes back, and then stop. Deterministic code should keep all authority over validation, expansion, installation, and final heavyweight artifacts. This is explicit burn-the-boats work. There is no compatibility mode, no legacy path, no dual runtime for old workspaces, and no obligation to preserve the old shell-text or multi-file worker story. The final live system should expose only the new one-file way. This plan defines that one-file contract. The later non-lethal follow-on plan is responsible for hiding all remaining bookkeeping outside the visible root and for finalizing warning-first supervision.

## Purpose / Big Picture

After this change, a user can run the live recipe, knowledge, and line-role Codex Exec stages and know that each worker session is doing one simple thing: open `task.json`, fill the answer fields in place, save, and stop. The model-facing contract is no longer a pile of manifests, hints, output contracts, queue files, scratch files, or helper scripts. The model-facing semantic surface is one compact task file with immutable evidence and mutable answer slots. Deterministic repo code then validates the edited file, extracts only the model-owned semantic edits, and expands those edits into the full authoritative runtime artifacts.

The user-visible proof is concrete. Activity traces for the happy path should show direct task-file work rather than repeated `cat` commands, helper-script loops, or here-doc JSON writes. Prompt-budget summaries should stop attributing most spend to command execution. The stage contracts, validators, tests, and docs should all center on one editable `task.json` instead of stage-specific output ledgers or packet files. A novice should be able to inspect the task-file schema and immediately understand what is happening: this is the evidence, this is the editable answer surface, and deterministic code handles the final expansion. The later non-lethal follow-on plan will finish the stricter “only `task.json` is visible in the worker root” cleanup.

## Context And Orientation

In this repository, a “worker” is one `codex exec` session running in a bounded local workspace. The current live code spreads the worker story across shared setup in `cookimport/llm/codex_exec_runner.py`, recipe behavior in `cookimport/llm/recipe_stage/`, knowledge behavior in `cookimport/llm/knowledge_stage/`, and line-role behavior in `cookimport/parsing/canonical_line_roles/` plus `cookimport/parsing/line_role_workspace_tools.py`.

The earlier fixed-assignment reset plan replaces leased packets and phase-ledger loops with immutable worker ownership. That still leaves a multi-file workspace model: assignment files, hints, output contracts, output directories, helper tools, and other artifacts that the model can inspect or manipulate. This follow-on plan removes that model-visible sprawl.

The core term in this plan is “editable task file.” An editable task file is one compact JSON file, named `task.json`, that contains four kinds of data:

1. Immutable assignment metadata such as stage name, worker id, assignment id, and repair mode.
2. Immutable evidence rows or units such as recipe candidates, nonrecipe rows, or canonical line rows.
3. Optional immutable hint fields that deterministic code has already prepared.
4. Mutable answer fields that the model is allowed to fill or patch.

The validator will compare the edited `task.json` against the original repo-owned snapshot. Only explicitly allowed answer fields may change. Every other field is locked. This is how the same file can safely act as both input and output.

The model-visible happy path in the final design is intentionally tiny:

1. Open `task.json`.
2. Read the whole file once.
3. Fill or patch only the answer fields.
4. Save the same file.
5. Stop.

Everything else moves out of the model-visible workspace. Deterministic code reads the edited file, validates the answers, expands them into the full stage artifact shape, records telemetry, and either promotes the result or rewrites `task.json` into a repair version with explicit validation feedback for one bounded retry.

This plan assumes the fixed-assignment reset has already landed. In that state, recipe already has planner, runtime, validation, and promotion seams under `cookimport/llm/recipe_stage/`; knowledge already has planner, runtime, recovery, promotion, and reporting seams under `cookimport/llm/knowledge_stage/`; and line-role already has planning, runtime, and validation seams under `cookimport/parsing/canonical_line_roles/`. Those seams are what this plan will reuse. The plan does not preserve the old worker-file vocabulary. It rewrites those stage owners so they all consume the same one-file contract.

## Canonical `task.json` Schema

The implementation should not leave the exact `task.json` shape open to interpretation. The live contract after this plan must use one shared top-level envelope with one shared editing rule: only `/units/<index>/answer` may change. Nothing else is writable by the model.

The required top-level shape is:

    {
      "schema_version": "editable_task_file.v1",
      "stage_key": "recipe_refine" | "nonrecipe_finalize" | "line_role",
      "assignment_id": "<stable assignment id>",
      "worker_id": "<stable worker id>",
      "mode": "initial" | "repair",
      "editable_json_pointers": [
        "/units/0/answer",
        "/units/1/answer"
      ],
      "units": [
        {
          "unit_id": "<stable unit id>",
          "owned_id": "<recipe id | shard id | atomic index group id>",
          "evidence": { ... immutable ... },
          "answer": { ... mutable ... },
          "previous_answer": { ... immutable, repair mode only ... },
          "validation_feedback": { ... immutable, repair mode only ... }
        }
      ]
    }

The required validator behavior is:

1. `schema_version`, `stage_key`, `assignment_id`, `worker_id`, `mode`, and `editable_json_pointers` are immutable.
2. `units` is ordered and immutable in length.
3. Every `unit_id` and `owned_id` is immutable.
4. Every `evidence` field is immutable.
5. In `initial` mode, `previous_answer` and `validation_feedback` must be absent.
6. In `repair` mode, `previous_answer` and `validation_feedback` are immutable repo-authored fields.
7. The only writable subtree is `answer`.
8. Every changed JSON path must be covered by `editable_json_pointers`.
9. A unit that is not present in a repair file cannot be silently repaired later. The repair file contains only failed units.
10. Deterministic code compares the original repo-authored `task.json` to the edited file and extracts only changed `answer` payloads before stage-specific validation begins.

The repository should implement this as a strict edit-form contract, not a best-effort parser. If the model changes any immutable field, the validator should fail with a precise error that names the changed path.

## Stage-Specific Unit Schemas

The stages should share one top-level envelope but use stage-specific `evidence` and `answer` payloads inside each unit. These shapes should be implemented exactly as written below unless a later revision to this plan changes them and records the change in the `Decision Log`.

### Recipe Unit Shape

Each recipe unit represents one recipe candidate. The evidence should already be cut down to the candidate text and the deterministic hints that promotion needs. The model should not rewrite candidate ids, source rows, or weak parse hints.

The required recipe unit shape is:

    {
      "unit_id": "recipe::<recipe_id>",
      "owned_id": "<recipe_id>",
      "evidence": {
        "recipe_id": "<recipe_id>",
        "source_text": "<full cutdown candidate text>",
        "source_rows": [
          ["<source line id>", "<line text>"]
        ],
        "hint": {
          "title": "<weak title hint or null>",
          "ingredients": ["<weak ingredient hint>", "..."],
          "steps": ["<weak step hint>", "..."],
          "quality_flags": ["<deterministic hint flag>", "..."],
          "candidate_tags": ["<optional tag label>", "..."]
        }
      },
      "answer": {
        "status": "repaired" | "fragmentary" | "not_a_recipe",
        "status_reason": "<required when status is not repaired; optional otherwise>",
        "canonical_recipe": {
          "title": "<required when repaired>",
          "ingredients": ["<ingredient string>", "..."],
          "steps": ["<step string>", "..."],
          "description": "<string or null>",
          "recipe_yield": "<string or null>"
        } | null,
        "ingredient_step_mapping": [
          {
            "ingredient_indexes": [0, 1],
            "step_indexes": [0]
          }
        ],
        "ingredient_step_mapping_reason": "<required when mapping is empty for a repaired multi-ingredient or multi-step recipe; otherwise optional>",
        "selected_tags": ["<tag label>", "..."],
        "warnings": ["<optional short warning>", "..."]
      }
    }

`ingredient_step_mapping` is intentionally a readable edit surface, not the repo’s final internal shape. The indexes are zero-based and refer to the edited `canonical_recipe.ingredients` and `canonical_recipe.steps` arrays inside the same unit’s `answer`.

The recipe validator must enforce these exact semantic rules:

1. `status` must be one of `repaired`, `fragmentary`, or `not_a_recipe`.
2. When `status` is `repaired`, `canonical_recipe` must be present and complete.
3. When `status` is `fragmentary` or `not_a_recipe`, `canonical_recipe` must be `null`.
4. Every `ingredient_indexes` and `step_indexes` entry must be an integer within bounds of the edited canonical recipe arrays.
5. Duplicate mapping rows are invalid if they repeat the same ingredient-index set and step-index set.
6. `ingredient_step_mapping` may be empty, but the existing mapping-reason rule still applies.
7. `selected_tags` is just the raw semantic tag choice surface. Deterministic normalization still owns final tag installation.
8. Deterministic code converts this readable `answer` shape into the final recipe authority payloads and any compact audit or benchmark rows the repo still wants to emit.

### Knowledge Unit Shape

Each knowledge unit represents exactly one owned nonrecipe block. It is not a contiguous multi-block packet. The planner should already have cut the block down to the minimum evidence the model needs. The live contract after this plan removes model-visible pass 1 and pass 2. The model makes both decisions in one `answer`.

The required knowledge unit shape is:

    {
      "unit_id": "knowledge::<block_index>",
      "owned_id": "<stable block id>",
      "evidence": {
        "block_index": 123,
        "block_id": "<stable block id>",
        "text": "<owned nonrecipe text>",
        "context_before": "<optional short context or null>",
        "context_after": "<optional short context or null>",
        "routing_hints": ["<deterministic routing hint>", "..."]
      },
      "answer": {
        "category": "knowledge" | "other",
        "group_key": "<required when category=knowledge; null otherwise>",
        "topic_label": "<required when category=knowledge; null otherwise>"
      }
    }

Per-block is part of the contract here. Deterministic code may later merge adjacent accepted `knowledge` blocks into larger reviewer-facing artifacts, but the worker-visible edit unit stays one block per unit.

The knowledge validator must enforce these exact semantic rules:

1. Every unit gets exactly one `category`.
2. If `category` is `other`, `group_key` and `topic_label` must both be `null`.
3. If `category` is `knowledge`, `group_key` and `topic_label` must both be non-empty strings.
4. `block_index` and `block_id` are immutable and must still map 1:1 to the final normalized answer rows.
5. Deterministic code, not the model, owns final group-id canonicalization and final nonrecipe authority artifacts.
6. Repair mode includes only units that failed shape, coverage, or grouping validation.

This is a deliberate simplification from the current model-visible knowledge pass split. The model no longer advances from one pass contract into another. It edits one answer object once.

### Line-Role Unit Shape

Each line-role unit represents one canonical atomic line that the planner has already cut down and ordered. The model should only choose the final route label and, when needed, one exclusion reason. It should not rewrite row ids, text, deterministic hint labels, or ownership metadata.

The required line-role unit shape is:

    {
      "unit_id": "line::<atomic_index>",
      "owned_id": "<atomic_index>",
      "evidence": {
        "atomic_index": 123,
        "block_id": "<block id>",
        "text": "<canonical line text>",
        "within_recipe_span": true | false | null,
        "deterministic_hint_label": "<optional weak baseline label or null>",
        "reason_hints": ["<deterministic hint>", "..."]
      },
      "answer": {
        "label": "RECIPE_TITLE" | "INGREDIENT_LINE" | "INSTRUCTION_LINE" | "HOWTO_SECTION" | "YIELD_LINE" | "TIME_LINE" | "RECIPE_NOTES" | "RECIPE_VARIANT" | "NONRECIPE_CANDIDATE" | "NONRECIPE_EXCLUDE",
        "exclusion_reason": "navigation" | "front_matter" | "publishing_metadata" | "copyright_legal" | "endorsement" | "publisher_promo" | "page_furniture" | null
      }
    }

The line-role validator must enforce these exact semantic rules:

1. `label` must be one of the ten allowed labels above.
2. `exclusion_reason` is required only when `label` is `NONRECIPE_EXCLUDE`.
3. `exclusion_reason` must be `null` for every other label.
4. The validator must still enforce owned-row coverage, stable row order, and the existing line-role policy restrictions.
5. Deterministic code still owns downstream conversion from `NONRECIPE_CANDIDATE` rows into later knowledge-stage inputs.

## Exact Repair File Rules

Repair mode should not be vague. The same `task.json` file is rewritten by deterministic code into a smaller file containing only failed units. The model patches the same `answer` objects again.

The required repair-mode rules are:

1. `mode` becomes `repair`.
2. `units` contains only failed units from the immediately previous attempt.
3. Each repair unit keeps the same `unit_id`, `owned_id`, and immutable `evidence`.
4. Each repair unit adds immutable `previous_answer`.
5. Each repair unit adds immutable `validation_feedback`.
6. `validation_feedback` must be machine-readable first and may also contain one short human-oriented explanation string.
7. The model still edits only `answer`.
8. There is one repair pass only.

The required `validation_feedback` shape is:

    {
      "error_codes": ["<stable error code>", "..."],
      "error_details": [
        {
          "path": "/answer/...",
          "code": "<stable error code>",
          "message": "<short message>"
        }
      ],
      "repair_instruction": "<short repo-authored instruction>"
    }

## Exact Artifact Expansion Rules

The plan should also be explicit about what deterministic code reconstructs after the model edits `task.json`.

For recipe, deterministic code must read the edited `answer`, normalize it, and rebuild the authoritative recipe semantics payload, final draft-v1 payload, audit rows, promotion rows, and benchmark-facing summaries. The model is not allowed to write the final compact repo contract directly.

For knowledge, deterministic code must read the edited `answer`, separate `knowledge` from `other` on a per-block basis, canonicalize group ids from the model-authored `group_key` and `topic_label`, and rebuild the final machine-readable nonrecipe authority outputs and knowledge group outputs.

For line-role, deterministic code must read the edited `answer`, validate the label and exclusion reason, rebuild the final canonical prediction rows, and then continue the existing deterministic downstream routing.

The shared expansion helper should be able to take:

1. the original repo-authored `task.json`,
2. the edited `task.json`, and
3. the stage key,

and return one normalized stage-specific answer payload that the stage owners can consume without reparsing the whole workspace story.

No further schema-choice work should be considered open after this revision unless implementation reveals a hard blocker. The plan is now specific enough to build.

The third block of work is stage-specific shaping. Recipe should stop asking the model to write the whole compact output contract as a fresh object. Instead, recipe planners should materialize recipe units directly inside `task.json`, with recipe evidence and the exact readable answer schema defined above. Recipe validation in `cookimport/llm/recipe_stage/validation.py` should read the edited file, validate only the answer payloads, and feed deterministic expansion in `promotion.py` so the final authoritative recipe artifacts are assembled without the model serializing repeated wrappers. Any repo-written titles, ids, or evidence maps remain immutable input.

Knowledge should be simplified more aggressively. The current split between pass 1 classification and pass 2 grouping should disappear from the model-visible contract. The knowledge planner should build one cutdown task file where each owned block becomes one unit with immutable evidence and the exact mutable answer object defined above. Deterministic validation in the knowledge stage should enforce per-block coverage, answer shape, and grouping consistency. Deterministic promotion should then split the accepted answers into final knowledge authority outputs and group artifacts. Repair mode should rewrite only the failed units with clear error feedback.

Line-role should move from editable ledgers plus helper scripts to the same form-style contract. The line-role planner should emit one `task.json` containing the ordered canonical rows the worker owns, their immutable text and weak hints, and the exact mutable answer slots for label and exclusion-reason fields defined above. The line-role validator should reject edits outside those answer slots and should preserve existing row order and ownership. The line-role finalizer should expand accepted answer fields into the existing final prediction payloads and stage artifacts without the model ever writing `work/<shard>.json` or `out/<shard>.json`.

The fourth block of work is deterministic expansion. Shared helper code should take an original `task.json` plus an edited `task.json`, extract only allowed answer edits, and produce a normalized stage-specific answer payload. Stage owners should consume those normalized answers and build the heavyweight artifacts they already know how to produce. This is where repeated ids, metadata, schema wrappers, and human-readable audit payloads should be reconstructed. The model no longer writes those heavyweight objects directly.

The fifth block of work is bounded repair. After the first validation pass, deterministic code should rewrite the same `task.json` into repair mode when needed. Failed units keep their immutable evidence, carry their rejected `previous_answer`, and add explicit machine-readable `validation_feedback`. The model patches the same answer fields in the same file. There is one repair pass only. If repair still fails, the unit fails closed and the stage summary records that outcome.

The sixth block of work is telemetry and benchmark truthfulness. Prompt-budget accounting should stop centering shell command counts for the happy path and should instead record task-file edits, edited unit counts, repaired unit counts, and per-stage answer completeness. Activity-trace summaries should surface task-file work when the transport exposes it. Benchmark artifact writers and analytics readers should stop expecting model-written heavyweight `out/` payloads or lease/phase vocabulary in worker roots. The later non-lethal follow-on plan will finish the warning-first supervision vocabulary.

The seventh block of work is deletion. Remove the old live workspace tooling, tests, docs, and vocabulary from active code. Historical benchmark artifacts may remain on disk as history, but live code and live docs should teach only the one-file task-file runtime.

## Idempotence And Recovery

The implementation can be done incrementally in a branch, but the merged result must not keep a shipped dual-path contract. If a stage conversion is half-finished, restore that stage to the last passing branch state before continuing. Do not paper over ambiguity with compatibility readers. If shared runner support for the editable task-file contract is incomplete, stop and finish that shared seam first. Repair is idempotent by design because the repo always rewrites the full authoritative `task.json` from the original snapshot plus deterministic validation feedback. The worker never owns lease advancement, helper installs, or artifact promotion. If a repair attempt fails, the repo can safely rerender the same repair file again because the immutable evidence and prior answer snapshot remain repo-authored.
