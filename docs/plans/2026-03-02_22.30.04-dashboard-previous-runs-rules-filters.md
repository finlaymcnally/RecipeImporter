# ExecPlan: Dashboard Previous Runs Rules Filters

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document follows [`docs/PLANS.md`](docs/PLANS.md) and is self-contained for this specific dashboard feature.

## Purpose / Big Picture

After this change, dashboard users can filter `Previous Runs` with explicit rule rows and arbitrary boolean expressions, so they can isolate effects like "same source book, varying model/effort" or "fixed model across many books" without exporting data elsewhere.

This is visible directly in `data/.history/dashboard/index.html`: a new filter panel appears in `Previous Runs`, and both the table and benchmark trend chart react to the same filtered dataset.

## Surprises & Discoveries

- Observation: Replacing `previous-runs-section` via `section.innerHTML` on empty data would remove the new filter UI after one empty-filter state.
  Evidence: Existing empty-state branch in `renderPreviousRuns()` replaced full section markup.

- Observation: All-method rows are generated from grouped raw benchmark records, so filtering must happen before grouping to keep all-method summaries meaningful.
  Evidence: `renderPreviousRuns()` grouping path (`allMethodRunInfo` + run/group aggregation) operates on per-record input.

## Decision Log

- Decision: Implement a rule-builder + expression parser instead of a fixed set of dropdown filters.
  Rationale: User requested arbitrary boolean/rules combinations across fields.
  Date/Author: 2026-03-02 / Assistant.

- Decision: Discover filter fields dynamically from benchmark record keys (including nested objects) and include synthetic fields (`source_file_basename`, `all_method_record`, etc.).
  Rationale: Keeps filter coverage aligned with evolving schema without hardcoding every knob.
  Date/Author: 2026-03-02 / Assistant.

- Decision: On invalid expression, show status error and fall back to unfiltered rows.
  Rationale: Avoids blank dashboard states and keeps UI usable while expression is being edited.
  Date/Author: 2026-03-02 / Assistant.

## Context and Orientation

The dashboard is generated by `cookimport/analytics/dashboard_render.py` as static HTML/CSS/JS written to `data/.history/dashboard/`. Relevant sections:

- `_HTML`: page structure for `index.html`.
- `_CSS`: dashboard styling.
- `_JS`: client logic for rendering, table/chart data selection, and diagnostics.
- `tests/analytics/test_stats_dashboard.py`: renderer and generated-JS contract checks.

`Previous Runs` includes special all-method row bundling logic in `renderPreviousRuns()`, where multiple benchmark records are merged into one display row per all-method run.
