## Purpose / Big Picture

After this change, the knowledge stage will have one shared shard-planning builder that both prompt preview and live runtime use. A user running `C3imp` or reading prompt-preview artifacts should see the same requested shard count, budget-native shard count, survivability recommendation, launch shard count, and warnings that the live runtime writes beside the real run. The point is not to collapse the difference between preferred/native sizing and minimum-safe sizing; the point is to centralize that story so the repository stops recomputing it in several places.

## Decision Log

- Decision: Keep packet-budget sizing and survivability sizing as separate calculations inside one shared planner rather than forcing them into one number.
  Rationale: They answer different questions. Packet-budget sizing is a preferred/native split driven by prompt/output shape. Survivability sizing is the minimum-safe split driven by token/session caps. Flattening them into one number would throw away useful operator information.
  Date/Author: 2026-04-15 / Codex

- Decision: Refactor the shared knowledge stage plan assembly first, instead of immediately retuning survivability budgets.
  Rationale: The concrete bug the user pointed out was confusion and duplication in the planning surface, and the sampled run showed survivability was not causing the inflated shard count anyway.
  Date/Author: 2026-04-15 / Codex

- Decision: Put the new shared builder in the knowledge-stage package rather than extending the generic `phase_plan.py`.
  Rationale: The generic phase-plan module should stay stage-agnostic. The knowledge-specific shard spec calculation depends on knowledge prompt rendering and knowledge shard metadata, so it belongs in the owning knowledge package.
  Date/Author: 2026-04-15 / Codex

- Decision: Let the shared builder accept an explicit operator-requested shard count instead of always trusting `KnowledgeJobBuildReport.requested_shard_count`.
  Rationale: `KnowledgeJobBuildReport.requested_shard_count` is the effective capped count after packet planning. The phase-plan artifacts need to preserve the user-configured requested count separately from the actual launch count, or the preview/runtime story regresses.
  Date/Author: 2026-04-15 / Codex

## Context and Orientation

The knowledge stage is the Codex-backed non-recipe finalize step. In this repository, that means the stage takes the deterministic `nonrecipe-route` candidate queue, packages it into knowledge packets, runs a first pass that decides whether each row is `keep_for_review` or `other`, and then optionally runs grouping on the surviving rows. The important files for this refactor are:

`cookimport/llm/codex_farm_knowledge_jobs.py` is the deterministic packet builder. It creates the knowledge packet JSON files and returns `KnowledgeJobBuildReport`, which currently contains the requested shard count, the budget-native shard count, the warning list, and the planned shard entries.

`cookimport/llm/prompt_preview.py` rebuilds prompt-preview rows from an existing processed run. For knowledge, `_build_knowledge_preview_rows(...)` currently returns both prompt rows and a preview-only mini planning dictionary, and the outer preview path turns that into a generic phase plan with `_build_direct_shard_phase_plan(...)`.

`cookimport/llm/knowledge_stage/runtime.py` is the live knowledge runtime. After `build_knowledge_jobs(...)` returns, the runtime computes survivability, launches workers, and also rebuilds the knowledge phase plan locally inside `_run_direct_knowledge_workers_v1(...)`.

`cookimport/llm/phase_plan.py` is the generic artifact writer for `phase_plan.json` and `phase_plan_summary.json`. It should remain generic. The knowledge refactor should feed it cleaner inputs rather than teaching it knowledge-stage details.

`cookimport/llm/shard_survivability.py` owns the minimum-safe sizing logic. It should remain a separate calculation, but the new shared builder must attach its results in the same place for both preview and runtime.

`tests/llm/test_prompt_preview.py`, `tests/llm/test_knowledge_packet_planning.py`, `tests/llm/test_knowledge_orchestrator_runtime_progress.py`, and `tests/cli/test_c3imp_interactive_menu.py` cover the current preview/runtime planning story and will need to keep passing after the consolidation.

## Plan of Work

Create a knowledge-owned phase-plan helper in the knowledge-stage package. That helper accepts the planned knowledge shard entries plus the counts and warnings that already come from `KnowledgeJobBuildReport`, and it emits the final knowledge phase plan by calling the generic `build_phase_plan(...)` and optionally attaching survivability. It also owns the knowledge-specific shard spec calculation, including prompt rendering and work-unit metrics, so preview and runtime stop duplicating those details.

Update `cookimport/llm/prompt_preview.py` so `_build_knowledge_preview_rows(...)` returns the full `KnowledgeJobBuildReport` instead of a preview-only dict. The outer preview flow will call the new shared knowledge phase-plan helper with that report, the preview worker-count decision, and the user-configured requested shard count. The preview path then annotates the prompt rows from that shared phase plan exactly as before.

Update `cookimport/llm/knowledge_stage/runtime.py` so the live runtime passes the same `KnowledgeJobBuildReport` into `_run_direct_knowledge_workers_v1(...)`. Inside that function, replace the inlined `build_phase_plan(...)` call with the new shared helper, passing the live worker assignment map, the user-configured requested shard count, and the live survivability report so runtime still writes the same phase-plan artifacts but now through the shared builder.

Keep the distinction between packet-budget sizing and survivability sizing explicit in names, docs, and tests. The refactor should not reintroduce the earlier warning-only shortcut. It should instead make it obvious where the preferred/native count comes from and where the minimum-safe count comes from.

## Interfaces and Dependencies

Define one knowledge-owned helper that hides the knowledge-specific `phase_plan` setup. At the end of the refactor, preview and runtime should both call a stable function shaped like:

    build_knowledge_stage_phase_plan(
        *,
        build_report: KnowledgeJobBuildReport,
        pipeline_id: str,
        surface_pipeline: str,
        worker_count: int | None,
        requested_shard_count: int | None = None,
        worker_id_by_shard_id: Mapping[str, str] | None = None,
        survivability_report: Mapping[str, Any] | None = None,
    ) -> dict[str, Any]

The helper must:

- derive shard specs from `build_report.shard_entries`
- preserve the operator-requested shard count separately from the budget-native and launch counts
- preserve `budget_native_shard_count` and `planning_warnings`
- attach worker ownership when runtime provides a `worker_id_by_shard_id` map
- attach survivability when runtime or preview provides a survivability report
