# Widen Knowledge Workspace-Worker Watchdog To Allow Normal In-Root Local Work

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The March 22 benchmark at `data/golden/benchmark-vs-golden/2026-03-22_03.33.50/` shows that the knowledge stage is no longer collapsing mainly because of schema drift or total worker paralysis. The remaining runtime loss is a watchdog product mismatch. Two parent knowledge shards still suffer watchdog deaths because the command stream contains an absolute `cd /home/.../.codex-recipe/...` prefix that points at the already-assigned sterile worker root, and the shared workspace-worker watchdog still treats that as a generic forbidden absolute path. More importantly, that failure reveals a broader problem: the main `workspace_worker` watchdog is still too strict about command shape and visible absolute paths inside the already-assigned worker root, and that strictness is ruining otherwise productive runs.

After this change, a knowledge workspace worker should be allowed to run ordinary bounded local commands whenever every surfaced path resolves under the exact worker execution root that the repo already assigned for that session. The immediate visible effect is simple. A focused watchdog test that currently kills the exact `cd <execution_root> && printf ... && sed ...` startup/orientation command should pass after the change. The broader intended effect is that normal in-root shell work such as `cat`, `sed`, `python3`, `jq`, helper scripts, and root-local `cd ... && ...` command chains stop dying just because they expose an absolute path under the assigned worker root. The same commands aimed at an unrelated absolute path such as `/etc`, `/home/.../projects/recipeimport`, or a sibling workspace should still fail.

## Surprises & Discoveries

- Observation: the knowledge stage did not fail because every packet was bad. It failed because every parent shard ended up incomplete.
  Evidence: `data/output/2026-03-22_03.33.50/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-22_03.34.33/raw/llm/saltfatacidheatcutdown/knowledge/promotion_report.json` reports `validated_shards: 0`, while `task_status.jsonl` in the same directory still records `254` validated task packets.

- Observation: the remaining runtime loss is concentrated in two shards, not across the whole stage.
  Evidence: the same run's `stage_status.json` reports `watchdog_killed: 104`, and shard-level counts reconstructed from `task_status.jsonl` show `ks0002` with `23` watchdog-killed task packets and `ks0003` with `81`.

- Observation: the concrete watchdog kill is not a repo escape or network command. It is a wrapper-style startup command that begins with an absolute `cd` into the current worker root.
  Evidence: `telemetry.json` for `worker-004` records `supervision_reason_code = "watchdog_command_execution_forbidden"` for `/bin/bash -lc "cd /home/mcnal/.codex-recipe/recipeimport-direct-exec-workspaces/... && printf ... && sed ..."` even though the repo had already launched that worker in the same sterile root.

- Observation: the temp-root relaxation plan already landed, so this is not another `/tmp` problem.
  Evidence: `cookimport/llm/codex_exec_runner.py` now has `_WORKSPACE_ALLOWED_TEMP_ROOTS = ("/private/tmp", "/var/tmp", "/tmp")`, but `_classify_workspace_path_argument(...)` and `_detect_workspace_worker_boundary_violation_in_text(...)` still reject any other visible absolute path.

- Observation: the knowledge watchdog callback currently has no way to tell the shared detector which absolute path is the worker's own execution root.
  Evidence: `_build_strict_json_watchdog_callback(...)` in `cookimport/llm/codex_farm_knowledge_orchestrator.py` currently calls `classify_workspace_worker_command(snapshot.last_command)` and `detect_workspace_worker_boundary_violation(snapshot.last_command)` with no execution-root context, even though the same file passes `working_dir=worker_root` to `runner.run_workspace_worker(...)`.

- Observation: the product problem is broader than one `cd` false positive. The main `workspace_worker` path is still behaving like a shell-shape checker instead of a containment checker.
  Evidence: in the same run, `task_status.jsonl` records `254` validated task packets, but `promotion_report.json` still has `validated_shards: 0` because runtime kills and packet incompleteness poisoned whole parent shards. The actual damage is not one rejected command string; it is lost useful work.

- Observation: `tmp_path`-backed absolute paths are the wrong regression fixture for this policy because `/tmp` is already intentionally tolerated.
  Evidence: the first callback regression draft falsely passed an outside-root command only because the path lived under `/tmp`; switching the fixture to `/home/mcnal/...` produced the expected `forbidden_absolute_path` kill.

## Outcomes & Retrospective

This plan is implemented. `cookimport/llm/codex_exec_runner.py` now accepts explicit allowed execution roots at the shared classifier/detector seam, uses that same root-aware rule for event-telemetry classification, and still fails closed when no root context is supplied. `cookimport/llm/codex_farm_knowledge_orchestrator.py` now passes the assigned `worker_root` only for the long-lived knowledge main-worker callback, so structured retry / repair behavior did not widen.

The user-visible result is the intended one: the exact saved `cd <execution_root> && printf ... && sed ...` startup/orientation shape now survives when the path is the session's own execution root, and representative in-root absolute helper commands now survive too. Commands aimed at repo paths or other outside roots still die with `forbidden_absolute_path`, and absolute `out/...` helper reads remain blockable under the stricter helper mode.

The main lesson is that this policy must stay containment-based, not syntax-based. The fix was not “teach the watchdog one startup command”; it was “teach the watchdog which absolute root the repo itself assigned, then apply the same containment rule everywhere that path surfaces.” The remaining live proof gap is still the optional benchmark rerun, which this plan intentionally left for a later user-approved token-spend pass.

## Context and Orientation

The knowledge stage uses long-lived `workspace_worker` sessions for its main attempt. In this repository, a `workspace_worker` is a Codex session launched inside a sterile directory outside the repo. The repo copies task files into that directory, tells the worker to stay there, and expects it to read local files and write local results under repo-declared paths such as `out/<task_id>.json`.

Three code surfaces matter for this fix.

The shared command classifier and boundary detector live in `cookimport/llm/codex_exec_runner.py`. `classify_workspace_worker_command(...)` produces the telemetry label shown in live status files. `detect_workspace_worker_boundary_violation(...)` is the hard kill/no-kill seam. Both now accept optional `allowed_absolute_roots` context, and both eventually use `_detect_workspace_worker_boundary_violation_in_text(...)` and `_classify_workspace_path_argument(...)` so direct path-token checks and raw shell-text scans share the same containment rule. Without that context, the old strict behavior still applies: visible absolute paths are rejected except `/dev/null` and the explicit temp roots.

The knowledge stage watchdog callback lives in `cookimport/llm/codex_farm_knowledge_orchestrator.py` as `_build_strict_json_watchdog_callback(...)`. That callback is used for the long-lived main worker path launched by `runner.run_workspace_worker(...)`. At that launch site, the orchestrator already knows the exact worker root because it passes `working_dir=worker_root` into the runner. This plan's implementation adds `execution_workspace_root=` at that seam and passes it onward only for the long-lived knowledge main-worker path.

The saved evidence for this plan is in the paired runtime tree under `data/output/2026-03-22_03.33.50/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-22_03.34.33/raw/llm/saltfatacidheatcutdown/knowledge/`. `failures.json` shows which parent shards died from watchdog kills. `telemetry.json` preserves the actual command text that triggered the kill. `task_status.jsonl` shows that the stage still produced many validated packets, which is why this fix should stay narrow instead of reopening the whole knowledge stage design.

This plan deliberately does not change:

- semantic packet validation in `cookimport/llm/codex_farm_knowledge_ingest.py`
- poisoned-worker retry / repair rules
- temp-root policy
- recipe or line-role callback wiring

Those are adjacent concerns, not this fix.

## Plan of Work

First, change the shared watchdog API in `cookimport/llm/codex_exec_runner.py`. Extend `detect_workspace_worker_boundary_violation(...)` and `classify_workspace_worker_command(...)` so they can accept optional execution-root context for the current session. Use a name that says exactly what it is, such as `allowed_absolute_roots` or `execution_working_dir`. Keep the default behavior unchanged when that new context is absent.

Inside the shared detector, normalize the allowed execution root or roots with `Path(...).resolve()`. Teach `_classify_workspace_path_argument(...)` and `_detect_workspace_worker_boundary_violation_in_text(...)` to tolerate an absolute path only when it resolves inside one of those allowed roots. This must cover both the direct path token case and the raw shell-text scan case, because the failing command is shell text of the form `cd /home/...workspace && printf ... && sed ...`. Do not stop at one startup pattern. The implementation should make the main `workspace_worker` path tolerant of ordinary in-root local work, which means representative in-root `cat`, `sed`, `python3`, `jq`, helper-script, and `cd ... && ...` command chains should all live or die based on where they operate, not on whether they surface the assigned root as an absolute path. Preserve the current bans for `/etc`, `$HOME`, sibling workspaces, repo paths outside the assigned root, parent traversal, and forbidden executables such as `git`, `curl`, `wget`, `pip`, and `npm`.

Then wire the knowledge main-worker call site in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. The main worker launch already knows `worker_root` when it calls `runner.run_workspace_worker(...)`. Pass that same root into `_build_strict_json_watchdog_callback(...)`, and from there into the shared classifier and boundary detector. Keep the structured retry and repair call sites unchanged in this plan unless they already have the same safe root context available and can be updated without widening scope.

After the shared helper and knowledge callback are in place, add focused tests. In `tests/llm/test_codex_exec_runner.py` or `tests/llm/test_codex_exec_runner_workspace.py`, add a regression for the exact command shape seen in `worker-004`: a shell command that starts with `cd <execution_root> && printf ... && sed ...` and should now be allowed when `<execution_root>` is the explicitly provided worker root. Add at least one second regression for another representative in-root local command shape that currently ought to be tolerated under the same rule, such as an absolute-path `python3` or `jq` command rooted under that execution directory. Add companion regressions proving that the same commands aimed at `/home/mcnal/projects/recipeimport`, `/etc`, or another unrelated absolute root still fail with `forbidden_absolute_path`.

Add one knowledge-specific watchdog regression in `tests/llm/test_codex_farm_knowledge_orchestrator.py`. The callback should tolerate the exact startup/orientation command when the callback is built with the matching `worker_root`, and it should still kill the same command when the absolute `cd` points outside that root. That test is the user-visible proof that the saved benchmark failure path is actually fixed.

If the implementation changes the documented boundary rule, add one short note to `docs/10-llm/nonrecipe_knowledge_review.md` or `cookimport/llm/README.md` explaining that main knowledge workspace workers may surface their own assigned execution-root absolute path, while other absolute paths remain forbidden. Keep the doc update brief.

## Idempotence and Recovery

This change is safe to iterate on because it is confined to the shared watchdog helper, one knowledge callback seam, and focused tests. Re-running the narrow watchdog tests is safe and should be deterministic.

The main recovery rule is to fail closed on scope. If the first implementation attempt starts allowing arbitrary absolute paths, stop and tighten the rule to “inside the explicitly allowed execution root only” before changing any more code. If fully general in-root path resolution proves tricky, land the exact saved `cd <execution_root> && ...` fix first only if the same patch also leaves a clear follow-up seam for the broader “normal in-root local work” rule; do not silently pretend the broader product issue is solved if only one string shape was patched.

No existing artifacts need migration. This only changes how future watchdog decisions classify commands.
