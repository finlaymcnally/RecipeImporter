# Fail Closed Nonrecipe Routing On Invalid Labels

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the nonrecipe router will only operate on valid authoritative block labels. Outside-recipe rows with missing block labels or unexpected final labels will stop the run immediately instead of quietly entering the review queue and turning metadata corruption into knowledge-review work. You will be able to see the result by running the focused staging tests: valid `OTHER` and `KNOWLEDGE` rows keep the current routing behavior, while malformed block-label metadata now raises a clear error.

## Surprises & Discoveries

- Observation: the current implementation turns both missing labels and unknown labels into `other` with a warning, then derives review eligibility by subtracting explicit exclusions.
  Evidence: `build_nonrecipe_stage_result(...)` in `cookimport/staging/nonrecipe_stage.py` calls `_normalize_stage7_category(...)` on `getattr(block_label, "final_label", None)` and then populates `review_eligible_block_category_by_index` from every block not present in `review_exclusion_reason_by_block`.

- Observation: the existing test suite already exercises the nonrecipe routing split between review-eligible rows and explicit exclusion rows, so the new fail-closed behavior can be added as narrow regression tests in the same file.
  Evidence: `tests/staging/test_nonrecipe_stage.py` already covers review eligibility, explicit exclusions, and refinement behavior.

- Observation: the live bug was narrower than “unknown labels.” Outside-recipe blocks labeled with recipe-only final labels such as `RECIPE_TITLE` were still being normalized to `other`, so a typo-only unknown-label test would not have locked the real regression down.
  Evidence: baseline repro before the patch returned `{0: 'other'}` for a single outside-recipe block labeled `RECIPE_TITLE`.

## Decision Log

- Decision: fail closed on invalid nonrecipe block-label metadata instead of inventing a quarantine category or silently forcing invalid rows to `other`.
  Rationale: the router is a consumer of authoritative labels, not a repair layer. If authoritative metadata is broken, the run should crash loudly so the upstream bug gets fixed at the source.

- Decision: keep the routing contract for valid inputs unchanged.
  Rationale: only malformed metadata should change behavior. Valid `KNOWLEDGE`, valid `OTHER`, and explicit review exclusions already describe the intended business rule cleanly.

## Context and Orientation

`cookimport/staging/nonrecipe_stage.py` owns the deterministic outside-recipe routing result. In this repository, “routing” means deciding which outside-recipe blocks are obvious junk that can be excluded immediately and which rows survive for later knowledge review. The function `build_nonrecipe_stage_result(...)` walks the full extracted block list, removes recipe-owned blocks, reads the authoritative block labels produced upstream, and builds four derived views: the seed category map, the routing result, the final-authority result, and the review-status result.

Today that function mixes two concerns. It maps valid upstream labels into the outside-recipe categories used by this stage, and it also quietly repairs malformed metadata by mapping missing or unknown labels to `other`. That repair is the bug. A malformed authoritative block label should be treated as a broken invariant, because every downstream consumer assumes the routing stage is reading valid label-first authority rather than guessing.

The focused tests live in `tests/staging/test_nonrecipe_stage.py`. That file already has small helper constructors and direct assertions on the `NonRecipeStageResult` dataclasses, which makes it the right place to add fail-closed coverage without introducing a larger integration test.

The owning docs live in `docs/05-staging/05-staging_readme.md`. That doc should say plainly that the nonrecipe router accepts only valid authoritative outside-recipe labels and raises on malformed block-label metadata.

## Plan of Work

Start in `cookimport/staging/nonrecipe_stage.py`. Change `build_nonrecipe_stage_result(...)` so it requires an authoritative block label for every outside-recipe block it processes. If a block has no entry in `labels_by_index`, raise a `ValueError` that includes the outside-recipe block index. Replace `_normalize_stage7_category(...)` with a strict helper that accepts only the current valid final-label vocabulary for this stage: `knowledge` maps to `knowledge`, the current `_OTHER_LABELS` set maps to `other`, and any missing or unknown label raises immediately with the block index in the error.

While making that change, stop deriving review eligibility by subtracting `review_exclusion_reason_by_block` from all nonrecipe blocks. Instead, build `review_eligible_block_category_by_index` explicitly while iterating through the blocks. The logic should be: valid `knowledge` rows are review-eligible, valid `other` rows with a `review_exclusion_reason` are excluded final `other`, and valid `other` rows without an exclusion reason are review-eligible. This keeps the routing logic honest and makes malformed metadata impossible to smuggle into the review queue.

Then update `tests/staging/test_nonrecipe_stage.py`. Add one test that omits the authoritative block label for an outside-recipe block and asserts that `build_nonrecipe_stage_result(...)` raises a clear `ValueError`. Add one test that supplies an unknown `final_label` and asserts the same. Keep the existing valid-input tests, and add or adjust assertions so valid `OTHER` without an exclusion reason remains review-eligible while valid `OTHER` with an exclusion reason still becomes excluded final `other`.

Finally, update `docs/05-staging/05-staging_readme.md` so the nonrecipe-stage section teaches the new fail-closed rule. Keep the note short and specific: the router consumes authoritative labels, explicit exclusions remain the only immediate final-`other` path, and malformed authoritative block-label metadata is a hard error rather than a fallback path.
