The live contract should say exactly what the planner uses now: full blocks, candidate spans, recipe spans, workbook/source identity, output directory, context blocks, and prompt target count. Nothing else.

The user-visible gain is honesty. In a solo repo, fake knobs create stale docs and wasted investigation time. The production cleanup had already removed the dead parameters before this pass; the remaining work in this plan is to make the tests and docs teach that same simple story: if a planner input exists, it has a concrete effect.

You can see the result by running the focused knowledge-planning tests listed below and by grepping for the removed parameter names. The observable proof is that `build_knowledge_jobs(...)` has the smaller signature, the runtime no longer passes unused values, the tests assert the real public input surface, and repo docs/help stop implying hidden planner controls.

## Context and Orientation

The live knowledge planner entrypoint is `build_knowledge_jobs(...)` in `cookimport/llm/codex_farm_knowledge_jobs.py`. In this repository, a “packet” is the model-facing content slice and a “shard” is the scheduler bundle that can own one or more packets. The planner writes one immutable input file per packet under `knowledge/in/*.json` and returns `ShardManifestEntryV1` rows that the knowledge runtime later turns into work.

The current planner contract is already simple and live: the full block archive, the non-recipe candidate spans that survived routing, the current recipe spans, workbook identity, output location, knowledge context block count, and requested prompt target count. The runtime path in `cookimport/llm/knowledge_stage/runtime.py` and the preview path in `cookimport/llm/prompt_preview.py` both already call the planner through that smaller surface.

The remaining goal of this plan is to keep the planner signature, the tests, and the docs describing the same honest reality so the repo does not drift back into speculative controls.

## Plan of Work

Production code no longer needs surgery here. Keep `cookimport/llm/codex_farm_knowledge_jobs.py`, `cookimport/llm/knowledge_stage/runtime.py`, and `cookimport/llm/prompt_preview.py` aligned by confirming the smaller planner contract is what the live tree already exposes.

Tighten the tests in `tests/llm/`. Add or update one focused planner test that freezes the intended public input surface and proves the live behavior still works with that smaller signature. The test should exercise a normal `build_knowledge_jobs(...)` call and assert on meaningful outputs such as packet files or shard entries. The point is not to test Python introspection for its own sake; the point is to make the planner’s intended live contract small and explicit.

Then update the owning docs. Revisit `docs/10-llm/10-llm_README.md` and `cookimport/llm/README.md` so they state plainly that the operator-facing planning controls are the packet context count and requested prompt target count, not hidden lower-level shard-size overrides.
