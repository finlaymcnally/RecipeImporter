# Prune Dead Knowledge Planner Controls

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan is intentionally narrow. It does not overlap the active non-recipe authority and benchmark-fix work in `docs/plans/2026-03-25_17.43.36-fix-nonrecipe-authority-and-benchmark-incompleteness.md`. Its scope is only the live knowledge planner input contract: remove dead parameters, remove dead call-site plumbing, tighten the tests, and update the owning docs/help text so the repo stops teaching controls that do nothing.

## Purpose / Big Picture

After this change, a reader opening `cookimport/llm/codex_farm_knowledge_jobs.py` should be able to trust that every argument in `build_knowledge_jobs(...)` participates in real current behavior. The live contract should say exactly what the planner uses now: full blocks, candidate spans, recipe spans, workbook/source identity, output directory, context blocks, and prompt target count. Nothing else.

The user-visible gain is honesty. Today the knowledge planner signature still advertises speculative or retired controls that are immediately discarded. In a solo repo, those dead knobs create fake flexibility, stale docs, and wasted investigation time. After this plan lands, the code, tests, and docs will all teach the same simple story: if a planner input exists, it has a concrete effect.

You can see the result by running the focused knowledge-planning tests listed below and by grepping for the removed parameter names. The observable proof is that `build_knowledge_jobs(...)` has a smaller signature, the runtime no longer passes unused values, the tests assert the real public input surface, and repo docs/help stop mentioning deleted controls.

## Context and Orientation

The live knowledge planner entrypoint is `build_knowledge_jobs(...)` in `cookimport/llm/codex_farm_knowledge_jobs.py`. In this repository, a “packet” is the model-facing content slice and a “shard” is the scheduler bundle that can own one or more packets. The planner writes one immutable input file per packet under `knowledge/in/*.json` and returns `ShardManifestEntryV1` rows that the knowledge runtime later turns into work.

Today the planner’s real inputs are already simple: the full block archive, the non-recipe candidate spans that survived routing, the current recipe spans, workbook/source identity, output location, knowledge context block count, and requested prompt target count. The function also still accepts three extra parameters that have no current behavior:

- `overrides`
- `skip_suggested_lanes`
- `target_chunks_per_shard`

Those values are thrown away immediately inside the function. One of them, `overrides`, is still passed from `cookimport/llm/knowledge_stage/runtime.py`, which makes the runtime look more configurable than it really is. `cookimport/llm/prompt_preview.py` already calls the planner without those dead arguments, so the preview path is closer to the honest contract than the runtime path is.

The goal of this plan is to make the planner signature, the runtime call sites, the test surface, and the docs all describe the same reality.

## Plan of Work

Start in `cookimport/llm/codex_farm_knowledge_jobs.py`. Remove `overrides`, `skip_suggested_lanes`, and `target_chunks_per_shard` from the `build_knowledge_jobs(...)` signature, and delete the matching `del` lines near the top of the function. Leave the real active inputs in place. Do not replace the deleted parameters with placeholder comments or compatibility aliases. The end state should read like a function that was designed around the current product, not one that survived a partial rollback.

Then update the direct callers. In `cookimport/llm/knowledge_stage/runtime.py`, remove `overrides=overrides` from the `build_knowledge_jobs(...)` call. In `cookimport/llm/prompt_preview.py`, confirm that no deleted arguments are being passed and keep that call aligned with the new signature. Search for the deleted parameter names across the repository and remove any remaining help text, comments, or docs that still imply those are live controls.

After the production code is honest, tighten the tests. Add or update one focused planner test under `tests/llm/` that freezes the intended public input surface and proves the live behavior still works with the smaller signature. The test should exercise a normal `build_knowledge_jobs(...)` call and assert on meaningful outputs such as packet files or shard entries. The point is not to test Python introspection for its own sake; the point is to make the planner’s intended live contract small and explicit.

Finally, update the owning docs. At minimum, revisit `docs/10-llm/10-llm_README.md` and any nearby documentation or inline comments that teach knowledge planning inputs. The docs should only mention controls that genuinely affect planning now. If no docs mention the removed parameters, record that in the plan progress and leave the docs untouched apart from any small note needed to clarify the live contract.
