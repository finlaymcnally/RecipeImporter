# Rename Benchmark Prompt Spans to Row Ranges

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

The active row-native migration is mostly done, but benchmark and external-AI cutdown helpers still describe prompt-derived recipe spans with block-era names like `start_block_index` and `end_block_index`. In this slice those fields do not represent source-block provenance; they represent the row-range hinted by prompt evidence and later projected into canonical line diagnostics. After this change, the touched benchmark helpers, docs, and tests should describe those prompt spans with row-native names while preserving real source-block provenance elsewhere.

## Surprises & Discoveries

- Observation: the remaining old names are concentrated in prompt-span helpers, not in the core scorer anymore.
  Evidence: `cookimport/bench/noise.py`, `cookimport/bench/cost.py`, `cookimport/bench/external_ai_cutdown/line_projection.py`, and `prompt_case_views.py` still use `start_block_index` / `end_block_index`, while the current scorer docs already describe `source-rows` as the active benchmark contract.

- Observation: some nearby block-named fields in the same area are legitimate provenance and should not be renamed in this slice.
  Evidence: `projected_spans.jsonl` still carries `block_index`, and recipe-boundary / deterministic-prep artifacts in adjacent tests use `source_block_index`, `title_block_index`, or block lists as true source-layout metadata rather than prompt-span semantics.

- Observation: the prompt-span rename was mechanically local, but the benchmark docs had also drifted semantically by still describing these ranges as source-block coordinates.
  Evidence: `docs/07-bench/07-bench_README.md` originally described prompt `start_block_index` / `end_block_index` values as source-block coordinates, which needed to become source-row coordinates once the helper contract was renamed.

## Decision Log

- Decision: rename only prompt-span coordinates that mean row ranges, leaving actual source-block provenance untouched.
  Rationale: the user wants old semantic block code gone, not a destructive wipe of valid layout/provenance fields. Renaming provenance fields here would blur that line and likely break unrelated deterministic joins.

- Decision: do not leave compatibility aliases on the touched benchmark helpers.

## Outcomes & Retrospective

This slice is now implemented. The touched benchmark helpers no longer describe prompt-derived recipe spans with block-era start/end names; they now use row-native range keys while continuing to rely on block-indexed projection artifacts only as provenance for the later line mapping.

The main implementation lesson was that this old naming island was smaller than it looked. The real semantic debt was the prompt-span range contract and a few operator-facing explanations around it, not the surrounding projection artifact itself. That meant the destructive cleanup could stay narrow and avoid destabilizing the provenance-bearing block fields nearby.

- Prompt-span fields that describe row coverage should become `start_row_index` / `end_row_index`.
- True source-block provenance fields such as `source_block_index`, `block_index` inside projection rows, and block-layout coordinates in deterministic recipe-boundary artifacts stay as-is.
