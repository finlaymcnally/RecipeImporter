# Replace Source Rows With A Prose-First Splitter

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, `source_rows` will remain the authoritative row units for Label Studio, row-gold export, and benchmark scoring, but the rows will be smaller and more logical for prose-heavy books. Recipe titles, ingredient lines, yield lines, how-to headings, and instruction-like rows that already split well should keep behaving the same, while large knowledge paragraphs should break into cleaner sentence-sized or near-sentence-sized rows. The user-visible proof will be that a prose-heavy source file produces `source_rows.jsonl` and Label Studio tasks with fewer giant paragraph rows, no obvious `yield` or `for the` mid-sentence fractures, and still-good small structural rows for recipes.

## Surprises & Discoveries

- Observation: the current splitter is not a prose splitter. It is a whitespace flattener plus a few regex boundary passes.
  Evidence: `cookimport/parsing/source_rows.py` normalizes each block with `_normalize_text(...)`, then `_split_block_text(...)` applies `_split_before_pattern(...)` for note, yield, how-to, and numbered-step cues before any prose-vs-structure mode decision exists.

- Observation: current source rows lose original block-internal line structure before the splitter decides what kind of content it is looking at.
  Evidence: `_normalize_text(...)` replaces `\r` and `\n` with spaces and collapses all repeated whitespace before `_split_block_text(...)` does any splitting.

- Observation: prose paragraphs stay huge because there is no sentence-based split pass at all.
  Evidence: the sample paragraph beginning `The key to cooking starches properly...` returns one single row from `_split_block_text(...)` even though it contains many full sentences.

- Observation: prose can still split in the middle of sentences because recipe-ish regex boundaries are applied globally.
  Evidence: the sample sentence `At about 290°F, a melted sugar syrup will yield a firm nougat...` currently splits into `...will`, `yield a firm nougat...will`, and `yield toffee.` because `_BOUNDARY_YIELD_RE` fires inside plain prose.

- Observation: current yield handling can still glue part of the first ingredient row onto a yield row after whitespace flattening.
  Evidence: the sample `Serves 6-10 1 kg pork rind ... 2-3 tbsp cornish sea salt` returns a first row of `Serves 6-10 1 kg pork rind, with a 5mm layer of fat left on` instead of a clean yield line followed by ingredient rows.

- Observation: direct test coverage for this splitter is very light.
  Evidence: repository search shows `cookimport/parsing/source_rows.py` is used widely, but there are almost no focused tests for `_split_block_text(...)` or `build_source_rows(...)` failure modes.

## Decision Log

- Decision: treat this redesign as a prose-first replacement rather than a small regex tweak.
  Rationale: the current failure is architectural. The bad behavior comes from the order of operations: flatten everything, apply global regex boundaries, never run a real prose split pass. Tweaking one or two patterns would not solve the simultaneous under-splitting and over-splitting problem.
  Date/Author: 2026-04-14 / Codex

- Decision: keep `source_rows` as the authority surface instead of introducing a second “better rows” artifact.
  Rationale: Label Studio import/export, row-gold generation, and source-row benchmarks already depend on `source_rows` as the active end-to-end baseline. Replacing the splitter is lower-risk than introducing a parallel row authority.
  Date/Author: 2026-04-14 / Codex

- Decision: explicitly preserve the current good small recipe-structure rows as a design goal.
  Rationale: the user is happy with the present behavior for recipe titles, ingredient lines, instruction lines, yield lines, and similar short structural rows. The redesign should target large prose blocks first, not regress recipe rows that already label well.
  Date/Author: 2026-04-14 / Codex

- Decision: do not make “one sentence per row” the hard rule for version one of the redesign.
  Rationale: sentence-per-row is a strong baseline for prose, but some cookbook prose benefits from keeping very short sentence pairs or short setup-plus-example pairs together. The design should aim for sentence-first splitting with controlled merging and hard maximum row-size guardrails.
  Date/Author: 2026-04-14 / Codex

## Outcomes & Retrospective

This design work explains why the current splitter feels wrong to both humans and AI labelers: it is blind to prose mode, yet still applies recipe-ish boundaries inside prose. The replacement design below keeps the authority model unchanged while changing the row-construction strategy so prose can become smaller and more logical without throwing away good recipe rows. Implementation has not started yet, so there is no runtime outcome to compare yet.

## Context and Orientation

`source_rows` live in `cookimport/parsing/source_rows.py`. A source row is the repository’s current authoritative “smallest labelable text unit” below raw importer `source_blocks`. Each row carries a stable `row_id`, its text, and its start and end offsets inside the original block. These rows feed three important downstream systems.

First, `cookimport/labelstudio/freeform_tasks.py` turns ordered source rows into Label Studio `segment_text` windows and stores the task-local mapping in `data.source_map.rows`. That means the rows directly shape what the human annotator and any AI prelabel flow see in the Label Studio text pane.

Second, `cookimport/labelstudio/export.py` projects freeform span annotations back onto source rows to produce `row_gold_labels.jsonl`. That means row shape directly affects the benchmark gold surface.

Third, `cookimport/bench/eval_source_rows.py` scores predictions against row gold, so the benchmark authority follows these rows exactly.

The current splitter begins in `build_source_rows(...)`. For each raw source block, it normalizes the block text, splits it into row strings, resolves offsets for each row string back into the original normalized block text, and emits `SourceRow` objects. The important helper functions are:

- `_normalize_text(...)`, which currently replaces newlines with spaces and collapses repeated whitespace.
- `_split_block_text(...)`, which currently performs most splitting logic.
- `_split_before_pattern(...)`, `_split_yield_segment(...)`, and `_split_quantity_runs(...)`, which currently do the actual cuts.
- `_infer_rule_tags(...)`, which labels the finished row text with coarse hints such as `ingredient_like`, `yield_prefix`, `instruction_like`, or `explicit_prose`.

The important current limitation is that splitting happens before meaningful classification. The new design below changes that order.

## Plan of Work

Replace the current “flatten first, regex split globally” design with a staged splitter that decides the block mode before it starts cutting text aggressively.

The first stage should preserve more structure from the original block. Instead of immediately replacing newlines with spaces, introduce a block parser that keeps line boundaries available long enough to make decisions. This parser should produce a small internal representation for each block: original text, normalized text, original line spans, and a compact classification summary. The implementation can stay inside `cookimport/parsing/source_rows.py`; it does not need a new public module unless the file becomes unreadable.

The second stage should classify each block into one of two broad modes: `structured_recipe_like` or `prose_like`. The classification should be conservative and text-driven. A block should count as `structured_recipe_like` when it already looks like the cases that split well today: short title-like lines, how-to headings, note/yield/time lines, numbered step starts, quantity-led ingredient runs, or compact imperative instruction lines. A block should count as `prose_like` when it is dominated by ordinary sentence punctuation, paragraph-style length, first-person or explanatory wording, or other plain running prose.

The third stage should run different splitting logic by mode.

For `structured_recipe_like` blocks, keep most of today’s successful heuristics, but apply them only inside this mode. In other words, ingredient splitting, yield extraction, note prefixes, how-to headings, numbered instructions, and similar recipe-structure rules should survive, but they should stop firing inside plain prose blocks.

For `prose_like` blocks, introduce a sentence-first splitter. Start by finding sentence boundaries using punctuation that behaves like sentence endings in cookbook prose: period, question mark, exclamation point, and, where needed, colon-plus-heading transitions that clearly introduce a new sentence-like unit. Avoid splitting on every comma or on keywords such as `yield`, `serves`, or `for the` in prose mode. Once sentence candidates exist, merge very short fragments when they would be awkward alone. A good version-one target is “smaller than current paragraphs, not necessarily exactly one sentence every time.” For example, a single very short sentence followed by a longer explanatory sentence may stay paired if the combined row remains modest.

The fourth stage should apply hard row-shape guardrails after the mode-specific splitting. Prose rows should never stay as giant multi-sentence blobs simply because sentence detection was imperfect. Add a maximum prose row size expressed in both characters and sentence count. When a prose row exceeds that budget, force another pass that splits it at the best available sentence boundary. At the same time, prevent obvious bad micro-rows by merging fragments that are too short to label sensibly on their own unless they are real structural units such as titles or yield lines.

The fifth stage should keep offset resolution honest. Each final row still needs stable `start_char_in_block` and `end_char_in_block` values. The safest implementation path is to keep the final row text as contiguous substrings of a normalized block text that still maps back to the original block representation. Avoid any design that rewrites row text after offsets are chosen.

The sixth stage should add explicit regression tests in the parsing test suite. The tests must cover at least four categories: giant prose paragraphs now split into smaller logical units; prose containing words like `yield` no longer splits mid-sentence; recipe structure such as ingredient runs and short headings still split cleanly; and mixed cases such as a yield line followed by ingredients on separate lines remain clean after the redesign.

The seventh stage should validate downstream readability, not just unit tests. Regenerate `source_rows` for at least one prose-heavy book and inspect the resulting Label Studio task text manually. The acceptance bar is not merely “tests pass”; it is also that the text shown to annotators is easier to label.

## Validation and Acceptance

Acceptance is behavioral.

For prose-heavy paragraphs, the new splitter should produce smaller rows than the current one. A representative paragraph with six to eight ordinary sentences should not remain a single source row unless there is a very strong reason to keep it whole.

For prose containing recipe-ish words in ordinary sentences, those words should not create fractures. In particular, prose uses of `yield`, `serves`, `for the`, or similar phrases should not cause obvious mid-sentence rows.

For existing good recipe-structure cases, the redesign should preserve current small units. Ingredient lists should still split into ingredient-like rows. Short headings and titles should stay short headings and titles. Yield lines should remain yield rows rather than being absorbed into neighboring prose.

Downstream proof should include one manual Label Studio inspection. After regenerating tasks for a prose-heavy source, open a task and confirm that the text pane shows smaller logical rows rather than giant paragraphs or mid-sentence fragments.

Run the parsing-focused tests from the project venv and expect them to pass. If the implementation touches Label Studio task generation or export assumptions, run the Label Studio domain suite as well.

## Idempotence and Recovery

The splitter redesign should be an in-place deterministic replacement. Re-running source-row generation for the same input should always produce the same rows. No destructive migration step is needed beyond regenerating derived artifacts such as `source_rows.jsonl`, Label Studio task bundles, or row-gold outputs that depend on row boundaries.

If a candidate redesign regresses recipe structure, the safe fallback is to keep the current structured splitting branch and continue iterating only on the prose branch. That is why this plan separates mode detection from mode-specific splitting: the recipe path can remain stable while the prose path improves.

## Interfaces and Dependencies

Keep `cookimport/parsing/source_rows.py` as the public owner of source-row creation. `build_source_rows(...)`, `load_source_rows(...)`, `write_source_rows(...)`, and `source_rows_to_payload(...)` must remain as the stable outward-facing functions after the redesign.

It is acceptable to add private helper structures and functions inside `cookimport/parsing/source_rows.py`, for example a small internal block-analysis object and separate private splitters for structured and prose modes. Avoid introducing a new public dependency unless the implementation proves it is necessary. Version one should prefer deterministic repository code over adding a sentence-splitting library, because the main issue here is split order and mode selection, not a lack of external NLP.

The downstream interfaces in `cookimport/labelstudio/freeform_tasks.py` and `cookimport/labelstudio/export.py` should not need signature changes. They should simply receive better rows.
