# Replace Source Rows With A Prose-First Splitter

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, `source_rows` will remain the authoritative row units for Label Studio, row-gold export, and benchmark scoring, but the rows will be smaller and more logical for prose-heavy books. Recipe titles, ingredient lines, yield lines, how-to headings, and instruction-like rows that already split well should keep behaving the same because the existing splitter will remain the first pass. The new behavior will happen only after that first pass, when oversized rows get broken into smaller prose-like units. The user-visible proof will be that a prose-heavy source file produces `source_rows.jsonl` and Label Studio tasks with fewer giant paragraph rows, no obvious `yield` or `for the` mid-sentence fractures, and still-good small structural rows for recipes.

## Surprises & Discoveries

- Observation: the current splitter is not a prose splitter. It is a whitespace flattener plus a few regex boundary passes.
  Evidence: `cookimport/parsing/source_rows.py` normalizes each block with `_normalize_text(...)`, then `_split_block_text(...)` applies `_split_before_pattern(...)` for note, yield, how-to, and numbered-step cues before any prose-vs-structure mode decision exists.

- Observation: current source rows lose original block-internal line structure before the splitter decides what kind of content it is looking at.
  Evidence: `_normalize_text(...)` replaces `\r` and `\n` with spaces and collapses all repeated whitespace before `_split_block_text(...)` does any splitting.

- Observation: prose paragraphs stay huge because there is no sentence-based split pass at all.
  Evidence: the sample paragraph beginning `The key to cooking starches properly...` returns one single row from `_split_block_text(...)` even though it contains many full sentences.

- Observation: prose can still split in the middle of sentences because recipe-ish regex boundaries are applied globally.
  Evidence: the sample sentence `At about 290°F, a melted sugar syrup will yield a firm nougat...` currently splits into `...will`, `yield a firm nougat...will`, and `yield toffee.` because `_BOUNDARY_YIELD_RE` fires inside plain prose.

- Observation: current yield handling can still glue part of the first ingredient row onto a yield row after whitespace flattening.
  Evidence: the sample `Serves 6-10 1 kg pork rind ... 2-3 tbsp cornish sea salt` returns a first row of `Serves 6-10 1 kg pork rind, with a 5mm layer of fat left on` instead of a clean yield line followed by ingredient rows.

- Observation: direct test coverage for this splitter is very light.
  Evidence: repository search shows `cookimport/parsing/source_rows.py` is used widely, but there are almost no focused tests for `_split_block_text(...)` or `build_source_rows(...)` failure modes.

## Decision Log

- Decision: treat this redesign as a prose-first replacement rather than a small regex tweak.
  Rationale: the current failure is architectural. The bad behavior comes from the order of operations: flatten everything, apply global regex boundaries, never run a real prose split pass. Tweaking one or two patterns would not solve the simultaneous under-splitting and over-splitting problem.
  Date/Author: 2026-04-14 / Codex

- Decision: keep `source_rows` as the authority surface instead of introducing a second “better rows” artifact.
  Rationale: Label Studio import/export, row-gold generation, and source-row benchmarks already depend on `source_rows` as the active end-to-end baseline. Replacing the splitter is lower-risk than introducing a parallel row authority.
  Date/Author: 2026-04-14 / Codex

- Decision: explicitly preserve the current good small recipe-structure rows as a design goal.
  Rationale: the user is happy with the present behavior for recipe titles, ingredient lines, instruction lines, yield lines, and similar short structural rows. The redesign should target large prose blocks first, not regress recipe rows that already label well.
  Date/Author: 2026-04-14 / Codex

- Decision: do not make “one sentence per row” the hard rule for version one of the redesign.
  Rationale: sentence-per-row is a strong baseline for prose, but version one should still allow a narrow amount of deterministic fragment repair. The design should aim for sentence-first splitting with fixed threshold-based merging and hard maximum row-size guardrails rather than semantic judgments about whether two sentences “belong together.”
  Date/Author: 2026-04-14 / Codex

- Decision: remove the proposed block-level `structured_recipe_like` versus `prose_like` classifier from the design.
  Rationale: the user does not trust a new deterministic content classifier, and that concern is well founded in this repository. The safer design is to preserve the existing good splitter as the first pass and only add a second pass for rows that are still too large afterward. That narrows new deterministic judgment to a simpler question: “is this row oversized enough to need another split pass?” rather than “what kind of block is this?”
  Date/Author: 2026-04-14 / Codex

- Decision: version one of the oversized-row cleanup pass should use fixed numeric thresholds, not semantic judgments such as “reads naturally together” or “this split would be worse.”
  Rationale: deterministic semantic grouping is exactly the kind of logic that has already caused trouble in this repository. A safer first implementation is to use measurable signals only: character count, sentence count, and a small-fragment threshold for repair merges.
  Date/Author: 2026-04-14 / Codex

## Outcomes & Retrospective

This design work explains why the current splitter feels wrong to both humans and AI labelers: it never performs a real prose split pass, yet still applies recipe-ish boundaries broadly enough to fracture running prose. The revised design keeps the authority model unchanged and now chooses the safer migration path: preserve the current good recipe-row behavior, then only split rows that are still too large. Implementation has not started yet, so there is no runtime outcome to compare yet.

## Context and Orientation

`source_rows` live in `cookimport/parsing/source_rows.py`. A source row is the repository’s current authoritative “smallest labelable text unit” below raw importer `source_blocks`. Each row carries a stable `row_id`, its text, and its start and end offsets inside the original block. These rows feed three important downstream systems.

First, `cookimport/labelstudio/freeform_tasks.py` turns ordered source rows into Label Studio `segment_text` windows and stores the task-local mapping in `data.source_map.rows`. That means the rows directly shape what the human annotator and any AI prelabel flow see in the Label Studio text pane.

Second, `cookimport/labelstudio/export.py` projects freeform span annotations back onto source rows to produce `row_gold_labels.jsonl`. That means row shape directly affects the benchmark gold surface.

Third, `cookimport/bench/eval_source_rows.py` scores predictions against row gold, so the benchmark authority follows these rows exactly.

The current splitter begins in `build_source_rows(...)`. For each raw source block, it normalizes the block text, splits it into row strings, resolves offsets for each row string back into the original normalized block text, and emits `SourceRow` objects. The important helper functions are:

- `_normalize_text(...)`, which currently replaces newlines with spaces and collapses repeated whitespace.
- `_split_block_text(...)`, which currently performs most splitting logic.
- `_split_before_pattern(...)`, `_split_yield_segment(...)`, and `_split_quantity_runs(...)`, which currently do the actual cuts.
- `_infer_rule_tags(...)`, which labels the finished row text with coarse hints such as `ingredient_like`, `yield_prefix`, `instruction_like`, or `explicit_prose`.

The important current limitation is that the current splitter only has one pass and that pass must serve both recipe structure and prose. The revised design below keeps that pass for the cases it already handles well, then adds a second pass only for oversized rows.

## Plan of Work

Replace the current one-pass splitter with a two-stage splitter.

The first stage should keep the current splitter behavior as the baseline. In practical terms, `build_source_rows(...)` should still use the existing `_split_block_text(...)` heuristics first so recipe titles, ingredient lines, yield lines, how-to headings, numbered steps, and other already-good small units keep coming out the same way they do now.

The second stage should look only at the rows produced by that baseline pass and decide whether each row is small enough to keep or large enough to split again. This is the key simplification in the revised design. The new deterministic judgment is not “what kind of text is this block?” It is only “is this row oversized enough that we should try a prose-oriented split pass?” That is a much narrower and safer rule.

Oversized rows should be identified by simple row-shape signals rather than semantic classification. Version one should use explicit candidate thresholds so the behavior is reviewable. A good first target is: keep rows under about 220 characters when they contain at most two sentence endings; split any row over about 320 characters; split any row with three or more sentence endings; and leave already-good structural rows alone even if they contain punctuation. In other words, if the baseline pass already produced a short structural row, keep it. If it produced a long paragraph-like row, split it again.

The oversized-row split pass should be sentence-first. Start by finding sentence boundaries using punctuation that behaves like sentence endings in cookbook prose: period, question mark, exclamation point, and, where clearly justified, colon-plus-heading transitions. Do not split this prose pass on keywords such as `yield`, `serves`, or `for the`. Those words are exactly what caused the current weird fractures, so they must not act as secondary boundaries in the large-row cleanup pass.

Once sentence candidates exist, merge only by fixed fragment rules. Version one should not try to decide whether two sentences “read naturally together.” Instead, allow a repair merge only when one side is clearly too small to stand alone, for example roughly 70 characters or less, the merged result remains under the keep threshold such as 220 characters, and the merged result contains no more than two sentences. Do not merge when either side is already comfortably labelable, for example around 120 characters or more.

Apply hard row-shape guardrails after that cleanup pass. A prose cleanup row should not remain an enormous multi-sentence blob simply because sentence detection was imperfect. Version one should enforce a simple ceiling such as “never keep three or more prose sentences together” and “never keep a cleanup row above about 320 characters if a sentence boundary exists.” At the same time, protect against bad micro-rows only through the narrow repair-merge rule above. If there is no sentence boundary and the row is still absurdly long, splitting inside the sentence can remain an explicit last-resort fallback rather than a normal path.

Keep offset resolution honest. Each final row still needs stable `start_char_in_block` and `end_char_in_block` values. The safest implementation path is to preserve the current contiguous-substring model: every final row must still be a direct substring of the normalized block text. Avoid any design that rewrites text after offsets are chosen.

Add explicit regression tests in the parsing test suite. The tests must cover at least four categories: giant prose paragraphs now split into smaller logical units; prose containing words like `yield` no longer splits mid-sentence; recipe structure such as ingredient runs and short headings still split cleanly because the first pass stays intact; and mixed cases such as a yield line followed by ingredients on separate lines remain clean after the redesign.

Validate downstream readability, not just unit tests. Regenerate `source_rows` for at least one prose-heavy book and inspect the resulting Label Studio task text manually. The acceptance bar is not merely “tests pass”; it is also that the text shown to annotators is easier to label.

## Validation and Acceptance

Acceptance is behavioral.

For prose-heavy paragraphs, the new splitter should produce smaller rows than the current one. A representative paragraph with six to eight ordinary sentences should not remain a single source row unless there is a very strong reason to keep it whole.

For prose containing recipe-ish words in ordinary sentences, those words should not create fractures. In particular, prose uses of `yield`, `serves`, `for the`, or similar phrases should not cause obvious mid-sentence rows.

For existing good recipe-structure cases, the redesign should preserve current small units because those rows come from the unchanged first pass. Ingredient lists should still split into ingredient-like rows. Short headings and titles should stay short headings and titles. Yield lines should remain yield rows rather than being absorbed into neighboring prose.

Downstream proof should include one manual Label Studio inspection. After regenerating tasks for a prose-heavy source, open a task and confirm that the text pane shows smaller logical rows rather than giant paragraphs or mid-sentence fragments.

Run the parsing-focused tests from the project venv and expect them to pass. If the implementation touches Label Studio task generation or export assumptions, run the Label Studio domain suite as well.

## Idempotence and Recovery

The splitter redesign should be an in-place deterministic replacement. Re-running source-row generation for the same input should always produce the same rows. No destructive migration step is needed beyond regenerating derived artifacts such as `source_rows.jsonl`, Label Studio task bundles, or row-gold outputs that depend on row boundaries.

If a candidate redesign regresses recipe structure, the safe fallback is to keep the current first pass exactly as it is and continue iterating only on the oversized-row cleanup pass. That is why this revised plan avoids a top-level content classifier: the recipe path remains the current known-good baseline while the prose cleanup pass improves separately.

## Interfaces and Dependencies

Keep `cookimport/parsing/source_rows.py` as the public owner of source-row creation. `build_source_rows(...)`, `load_source_rows(...)`, `write_source_rows(...)`, and `source_rows_to_payload(...)` must remain as the stable outward-facing functions after the redesign.

It is acceptable to add private helper structures and functions inside `cookimport/parsing/source_rows.py`, for example one helper that decides whether a baseline row is oversized and another helper that splits oversized rows on sentence boundaries. Avoid introducing a new public dependency unless the implementation proves it is necessary. Version one should prefer deterministic repository code over adding a sentence-splitting library, because the main issue here is oversized-row cleanup and split order, not a lack of external NLP.

The downstream interfaces in `cookimport/labelstudio/freeform_tasks.py` and `cookimport/labelstudio/export.py` should not need signature changes. They should simply receive better rows.
