# I3.1 — Purpose / Big Picture: Unstructured-first EPUB block extraction (blocks + roles + traceability)

This ExecPlan is a living document and must be maintained as work progresses, per PLANS.md (repo root).
It is written to be executable by a stateless agent (or a novice human) with only the working tree and this file.

## Purpose / Big Picture

Today, EPUB ingestion produces a linear stream of Blocks that downstream systems treat as the “truth” for:
(1) recipe candidate segmentation (finding recipe boundaries, ingredients, and instructions), and
(2) knowledge extraction (chunking + tip/topic mining from non-recipe content).

When the EPUB “HTML → Blocks” step is noisy or poorly segmented, everything downstream becomes brittle:
recipes get chopped in the wrong places, ingredient/instruction grouping becomes unreliable, and technique chapters become an undifferentiated wall of prose that chunking struggles to lane as “knowledge” vs “noise”.

After this change, EPUB ingestion will produce cleaner, more semantically meaningful Blocks by using Unstructured’s HTML partitioner on each EPUB spine document. Those Blocks will carry explicit traceability metadata (including Unstructured element_id) and a lightweight deterministic “block_role” label. This should improve segmentation stability, improve knowledge chunk lane assignment, and make regressions far easier to diagnose by inspecting a persisted, element-level raw artifact.

What you should be able to do after this is implemented (observable outcomes):

1. Run:

    cookimport stage data/input/<book>.epub --workers 1

   with the Unstructured extractor enabled, and see:
   - the run completes successfully,
   - final drafts show fewer obvious boundary errors on known-problem recipes,
   - technique/intro material yields more “knowledge” chunks (not all “noise”).

2. Inspect the run artifacts and directly answer:
   “What did we extract?” and “Where did this block come from?” without re-running:
   - a new raw artifact file exists containing one JSON object per Unstructured element,
   - each Block retains the originating Unstructured element_id (and stable element_index) so diffs are actionable.

### Scope (explicit)

This ExecPlan standardizes the “all the way through unstructured-source” path for EPUB only:
- We partition EPUB spine HTML/XHTML using Unstructured.
- We DO NOT standardize PDFs (or other formats) in this plan.
  PDF standardization is a follow-up because PDF extraction uses different upstream primitives (page/bbox/OCR lines) and will require a separate adapter and validation strategy.

## Context and Orientation

If you are new to this repo, read this section carefully before editing code.

### Glossary (repo-local meanings)

Block:
A low-level ordered text unit used during extraction and segmentation. Blocks are the primary coordinate system for segmentation.
Blocks include text plus a features dictionary that stores extractor metadata (for EPUB, this includes spine_index and will now include Unstructured fields).

RecipeCandidate:
A segmented recipe record (schema-like) produced after segmentation finds a block-range that looks like a recipe and field extraction populates name/ingredients/instructions/etc.

nonRecipeBlocks:
Blocks not covered by any recipe candidate ranges. This is the main input to knowledge chunking later.

KnowledgeChunk:
A coherent non-recipe region built from nonRecipeBlocks; lane-assigned as “knowledge” or “noise” and later mined for tips/topics.

Raw artifact:
A persisted debug output written under the run root’s raw/ directory. For split jobs, workers write raw artifacts under .job_parts/... and the main process merges them into raw/ on success (collisions are resolved by prefixing filenames with the job index).

### Current end-to-end EPUB flow (high level)

When you run:

    cookimport stage data/input/<book>.epub

The concrete flow is:
- registry selects EpubImporter
- EpubImporter.convert() extracts a linear block stream (today via HTML structure)
- blocks are normalized and enriched with signals
- recipe candidate segmentation runs over block indices
- blocks not in recipes are collected as nonRecipeBlocks
- stage pipeline uses nonRecipeBlocks to generate chunks; writers emit drafts/tips/topics/chunks/raw/report

### Output structure you will use during validation

Stage writes a timestamped run root under the output directory (commonly data/output/<timestamp>/...).

Key outputs to inspect:
- final drafts/<workbook_slug>/r{index}.json
- chunks/<workbook_slug>/chunks.md (if chunks exist)
- raw/<importer>/<source_hash>/<location_id>.<ext>
- <workbook_slug>.excel_import_report.json (written at run root)

### Split-job behavior (EPUB)

EPUB stage can split by spine ranges when --workers > 1:
- each worker converts only [start_spine, end_spine)
- Blocks must carry absolute spine_index so merge ordering is deterministic across jobs
- raw artifacts are written to .job_parts/<workbook_slug>/job_<index>/raw/... and merged into raw/ on success

Known limitation to keep in mind:
- EPUB split jobs do not detect recipes that span spine boundaries; a recipe can be split at a spine boundary today.
  This plan does not fix that, but you must ensure the Unstructured integration does not make it worse (and does not break merge).

### Where you will likely edit code

You will almost certainly touch:
- cookimport/plugins/epub.py
  - the spine iteration + HTML-to-Block extraction path
  - the handling of start_spine/end_spine
- cookimport/parsing/
  - new: unstructured_adapter.py (Element→Block mapping)
  - new: block_roles.py (deterministic block_role labeling)
  - existing: signals.py and chunks.py (integration points for roles and heading levels)
- cookimport/staging/ (only if raw artifact writing needs a new helper or naming convention)

### Milestone 2 — Implement the Unstructured adapter: HTML → Blocks (with traceability)

Goal:
Build a small, testable mapping layer that converts Unstructured elements into the repo’s Block model while preserving ordering, split-job invariants, and traceability.

Key requirement added in this revision:
Capture Unstructured element_id explicitly and propagate it:
- into Block.features (per Block), and
- into the JSONL diagnostics artifact, so regressions are easy to diff.

Work:
Create a new module:

- cookimport/parsing/unstructured_adapter.py

Define a single, testable entrypoint:

    partition_html_to_blocks(
        html: str,
        *,
        spine_index: int,
        source_location_id: str,
    ) -> tuple[list[Block], list[dict]]

Returns:
- blocks: list of Block objects in stable extracted order (element order within a spine doc)
- diagnostics_rows: JSON-serializable dicts, one per element, suitable for JSONL persistence

Mapping rules (first pass; keep deterministic and easy to reason about):
1) Element filtering:
- Skip elements whose text is empty or whitespace-only after normalization.

2) Type mapping:
Map Unstructured element category to Block.type using a simple table:
- Title -> heading
- NarrativeText -> paragraph
- ListItem -> list_item
- Table -> table
- other/unknown -> paragraph (or other if the codebase has an “other” type)

3) Features and traceability (IMPORTANT):
In Block.features, include (at minimum):
- spine_index (absolute)
- source_location_id (so provenance can correlate)
- unstructured_category (string category name)
- unstructured_category_depth (int, default 0)
- unstructured_parent_id (string or null, if available)
- unstructured_element_id (string or null, if available)   <-- REQUIRED BY USER
- unstructured_element_index (int)                          <-- REQUIRED FOR STABLE DIFFS
- unstructured_stable_key (string)                          <-- RECOMMENDED

Rationale for stable_key:
Unstructured element_id may not be stable across versions/runs; a stable_key derived from coordinates is diff-friendly.
Recommended definition:

- unstructured_stable_key = f"{source_location_id}:spine{spine_index}:e{element_index}"

4) Heading levels:
If the element is a Title, derive a heading level using category depth as a proxy:

- heading_level = clamp(unstructured_category_depth + 1, 1, 6)

Store it where the repo expects (often Block.features["heading_level"] or a dedicated field if Block has one).

5) Text normalization:
Be careful not to “double-normalize” in a way that changes semantics.
If the repo already runs a shared cleaning pipeline (NFKC, whitespace standardization, etc.), align with that:
- either apply the same cleaning to element text here, or
- keep raw text here and rely on downstream normalization (preferred if that is already guaranteed).
Whichever you choose, document it in Decision Log.

6) Ordering invariants:
- Preserve Unstructured’s returned order within each spine document.
- Preserve spine document order by iterating spine_index ascending.
- Do not sort elements by text or category; ordering is part of the coordinate system used by segmentation.

Diagnostics rows (for JSONL):
For each element, emit a dict with:
- source_location_id
- spine_index
- element_index
- element_id (unstructured_element_id)
- stable_key (unstructured_stable_key)
- category, category_depth, parent_id
- text (full text; if size becomes an issue, add text_hash and optionally truncate text but keep hash)
- any other cheap metadata that helps debugging (e.g., html_tag if available)

Acceptance:
- The adapter can convert a small HTML snippet into Blocks and diagnostics deterministically.
- Each Block produced from Unstructured carries unstructured_element_id (when available) and always carries unstructured_element_index + stable_key.

### Milestone 3 — Wire the adapter into the EPUB importer (preserve split behavior)

Goal:
Enable Unstructured-based extraction for EPUBs without breaking:
- start_spine/end_spine slicing
- global ordering across slices
- merge correctness and downstream block index coordinate space

Work:
In cookimport/plugins/epub.py:
1) Identify the existing spine iteration loop and the function(s) that currently convert spine HTML/XHTML into Blocks.

2) Add an alternate extraction path selected by C3IMP_EPUB_EXTRACTOR:
- For each spine doc in the worker’s assigned [start_spine, end_spine):
  - obtain html string using existing EPUB extraction logic
  - call partition_html_to_blocks(html, spine_index=..., source_location_id=...)
  - accumulate returned blocks in a list (job-local)
  - accumulate diagnostics rows (job-local)

3) Ensure each Block has absolute spine_index in its features.
Downstream merge and ordering depend on it.

4) Ensure existing raw artifact “full_text” behavior is preserved.
If the EPUB importer currently writes extracted blocks JSON as full_text, keep doing that (legacy and unstructured should both produce comparable artifacts).

Acceptance:
- Running cookimport stage on a small EPUB with the switch enabled produces a coherent Block stream and does not crash.
- Split job conversion still runs for a spine range and emits consistent outputs.

### Milestone 4 — Persist Unstructured diagnostics as a raw artifact (jsonl)

Goal:
Make extraction debuggable without re-running with ad-hoc logging, and make regressions easy to diff.

Work:
1) Write a new raw artifact file containing diagnostics rows:

- name: unstructured_elements.jsonl
- format: one JSON object per line (no surrounding array)

2) Use the existing raw artifact mechanism so it lands under the standard raw path:
- raw/epub/<source_hash>/<location_id>.<ext>
(Exact naming conventions may vary; keep consistent with existing artifact writer behavior.)

3) Split-job handling:
- Each worker writes its own diagnostics file under .job_parts/<workbook_slug>/job_<index>/raw/...
- The main process merges/moves raw artifacts into raw/ on success.
- Name collisions are resolved by prefixing filenames with job index; do not rely on collisions as a strategy.

4) Include minimal mapping metadata somewhere:
Either:
- include a small header row at the top of the JSONL (not ideal for JSONL tooling), or
- write a companion sidecar file (recommended), e.g.:
  - unstructured_elements.meta.json
    containing unstructured version, extractor switch value, and mapping table.

Acceptance:
- After a successful run, unstructured_elements.jsonl exists under raw/ for EPUB runs when Unstructured is enabled.
- For --workers > 1, the artifact exists after merge and .job_parts is cleaned on success (and preserved on failure for debugging).

### Milestone 5 — Add deterministic block_role enrichment and integrate into chunk lane assignment

Goal:
Improve knowledge extraction by avoiding the “everything narrative becomes noise” failure mode, without introducing non-deterministic ML.

Work:
Create a module:

- cookimport/parsing/block_roles.py

Define:

    assign_block_roles(blocks: list[Block]) -> None

Rules should be:
- deterministic
- cheap
- explainable (easy to debug)
- aligned with Label Studio canonical/freeform vocabulary where possible

Recommended first-pass roles (stored as Block.features["block_role"]):
- recipe_title
- ingredient_line
- instruction_line
- tip_like
- narrative_science
- narrative_fluff
- other

Heuristic suggestions (keep conservative to avoid false positives):
- If block is heading and matches recipe title cues -> recipe_title
- If signals indicate ingredient-like (starts_with_quantity, has_unit, ingredient_
