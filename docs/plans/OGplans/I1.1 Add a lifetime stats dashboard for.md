# Add a lifetime stats dashboard for cookimport (imports + golden-set benchmarks)

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan must be maintained in accordance with `docs/PLANS.md` (ExecPlan requirements and workflow rules).

## Purpose / Big Picture

After this change, a user can generate a single, static “lifetime stats” dashboard (offline, local-only) summarizing:

* **Stage/import runs**: throughput and quality signals across all historical staged files (recipes/tips counts, runtime, derived rates, warnings/errors).
* **Golden-set benchmark runs**: how parsing quality metrics (precision/recall and other eval signals) trend over time across Label Studio evaluations.

The primary user-visible behavior is:

* Running a single CLI command generates a dashboard directory containing `index.html` plus local assets, and the dashboard renders correctly when opened in a browser via `file://...`.
* The dashboard clearly answers: “How fast are imports trending?”, “What are the slowest runs and why?”, “How are benchmark scores moving over time?”, and “Are there obvious regressions in warnings/errors/coverage?”

### Non-goals (v1)

* No web server, no network calls, no external CDNs.
* No attempt to re-run imports or benchmarks; the dashboard is a read-only view over artifacts already on disk.
* No requirement to open or parse full recipe outputs; prefer summarized metric surfaces.

## Context and Orientation

This repository produces multiple “metric surfaces” on disk. The dashboard should read those surfaces (preferably the already-normalized ones), aggregate them, and render a static HTML view.

### Key artifact surfaces (current, code-verified)

1. **Per-file conversion report** (stage/import health + timing)

Each staged source file writes a report JSON at:

* `data/output/<YYYY-MM-DD_HH.MM.SS>/<workbook_slug>.excel_import_report.json`

These reports include (at least) counts, warnings/errors, confidence aggregates, timing, and output footprint. Key fields called out in docs include:

* volume counts (`totalRecipes`, `totalTips`, `totalTipCandidates`, `totalTopicCandidates`)
* data health (`missingFieldCounts`, `skippedRows`, `lowConfidenceSheets`, `warnings`, `errors`)
* runtime timing (`timing.total_seconds`, `timing.parsing_seconds`, `timing.writing_seconds`, `timing.ocr_seconds`, plus checkpoints)
* output footprint (`outputStats.files`, `outputStats.largestFiles`)

2. **Cross-run performance history CSV** (small, lifetime-trendable)

* `data/output/.history/performance_history.csv`

This is generated from the per-file `*.excel_import_report.json` reports. It is already normalized to “one row per imported file,” and includes derived metrics like per-recipe/per-tip seconds and outlier flags. This should be the primary source for stage/import trend charts, because it is compact and intentionally designed for “lifetime history”.

3. **Label Studio benchmark/eval artifacts** (golden-set quality metrics)

Evaluation commands emit artifacts including:

* `eval_report.json`
* `eval_report.md`
* `missed_gold_spans.jsonl`
* `false_positive_preds.jsonl`

Label Studio ingestion can also emit:

* `coverage.json` (e.g., `extracted_chars`, `chunked_chars`, `warnings`)
* `manifest.json` (includes task upload counts and metadata)

Benchmark runs are commonly stored under:

* `data/golden/eval-vs-pipeline/<timestamp>/`

…and benchmark prediction artifacts may be co-located under:

* `<eval_output_dir>/prediction-run/`

### Output layout facts that matter for the dashboard (to avoid footguns)

* Run timestamp folders are dot-separated in the time portion: `%Y-%m-%d_%H.%M.%S`. Do not assume `:` is used.
* Stage outputs include intermediate/final drafts, tips, chunks, raw artifacts, and a report JSON at the run root. Do not assume a `reports/` subfolder.
* Split-job staging (PDF/EPUB) may create temporary artifacts under:

  * `data/output/.job_parts/<workbook_slug>/job_<index>/raw/...`
    These should be ignored by the dashboard unless explicitly debugging a failed merge; they are not “final runs”.

### Dashboard terms (define once and use consistently)

* **Stage file record**: one imported source file as represented by one row in `performance_history.csv` (or one `*.excel_import_report.json` when falling back).
* **Benchmark eval record**: one Label Studio evaluation run represented by one `eval_report.json` (optionally enriched by `coverage.json` / `manifest.json` in the same folder).
* **Run category**: a label used only by the dashboard to prevent mixing apples and oranges (e.g., `stage_import`, `labelstudio_import`, `benchmark_eval`, `benchmark_prediction`).

## Plan of Work

### Milestone 1: Inventory + stable data model (schema first)

At the end of this milestone, there is a clear, code-backed schema for what the dashboard consumes, and the schema is grounded in the existing on-disk metric surfaces. A novice can read the schema and understand what data is expected and where it comes from.

Work:

1. Create a small “dashboard schema” module using Pydantic v2 models.

   Suggested location (align with existing perf analytics code paths):

   * `cookimport/analytics/dashboard_schema.py`

   Define a top-level model like:

   * `DashboardData`

     * `generated_at` (ISO string)
     * `output_root` (string, default `data/output`)
     * `golden_root` (string, default `data/golden`)
     * `stage_records: list[StageRecord]`
     * `benchmark_records: list[BenchmarkRecord]`
     * `summary: DashboardSummary` (pre-aggregated totals, optional)

   Where:

   * `StageRecord` is designed to be directly populated from one CSV row (preferred) or from one conversion report (fallback).
   * `BenchmarkRecord` is designed to be populated from `eval_report.json` (and optionally coverage/manifest).

2. Be explicit about which fields are optional vs required.

   The dashboard must not “invent zeros” for missing metrics. Use `None` for missing numeric fields and omit those points from charts.

3. Record “source pointers” on each record so the UI can link/drill down without needing to load huge data:

   * `artifact_dir` (relative path from repo root if possible)
   * `report_path` (path to the CSV row source or JSON report)
   * `run_timestamp` (parsed from folder name or report field; store both raw + parsed if needed)

4. Add a small `schema_version` string to `DashboardData` so future iterations can change fields safely.

Acceptance:

* A developer can read `dashboard_schema.py` and know exactly what dashboard data looks like and which disk artifacts populate it.
* There is a tiny unit test that can instantiate `DashboardData` from minimal synthetic dicts.

### Milestone 2: Collectors (read-only, robust, skip footguns)

At the end of this milestone, running a Python function can scan local outputs and produce a populated `DashboardData` instance without errors, even when some runs are partial/missing.

Work:

1. Implement stage/import collector:

   Suggested module:

   * `cookimport/analytics/dashboard_collect.py`

   Behavior:

   * Prefer reading `data/output/.history/performance_history.csv` if present.
   * If it is missing, empty, or the user passes a flag like `--scan-reports`, fall back to scanning:

     * `data/output/<timestamp>/*.excel_import_report.json`
   * Never read:

     * `data/output/.job_parts/**`
     * any `prediction-run/**` directories (if they are under a searched root)
   * For each stage record, attempt to classify `run_category`:

     * If the artifact path indicates Label Studio ingestion (`.../labelstudio/...`), mark as `labelstudio_import` (and default it to hidden in the “Stage imports” view unless the user enables it).
     * Otherwise mark as `stage_import`.

   Implementation notes:

   * Do not assume CSV columns beyond what exists; load header dynamically and map known column names.
   * If derived metrics are missing (e.g., seconds-per-recipe), compute them only when numerator and denominator are present and denominator > 0.
   * Guard division-by-zero and “infinite” rates; store `None` instead.

2. Implement benchmark collector:

   * Scan for `eval_report.json` under `golden_root`, with a priority on `data/golden/eval-vs-pipeline/*/eval_report.json` but also allow nested patterns for other eval outputs.
   * For each eval directory:

     * Parse `eval_report.json` into a normalized `BenchmarkRecord`.
     * If present, read `coverage.json` and compute coverage ratio `chunked_chars / extracted_chars` (store as optional float).
     * If present, read `manifest.json` and capture task counts (optional fields).

   Classification:

   * Records from directories containing `eval_report.json` are `benchmark_eval`.
   * Do not treat `prediction-run` conversion outputs as benchmark eval; if you choose to surface them at all, treat them as `benchmark_prediction` and keep them separate from “stage imports”.

3. Add “collector resilience” rules:

   * If a JSON file is malformed, skip that record, add a warning to a `collector_warnings` list in `DashboardData` (optional), and continue.
   * If a directory name looks like a timestamp but fails parse, keep the raw string and sort such runs last (stable ordering).

Acceptance:

* A unit test can run collectors against a temp directory containing:

  * a tiny `performance_history.csv`
  * one fake `*.excel_import_report.json`
  * one fake `eval_report.json` (+ optional coverage/manifest)
    and returns `DashboardData` with correctly classified records.

### Milestone 3: Static HTML dashboard (one command)

At the end of this milestone, running a `cookimport` CLI command generates a static dashboard directory containing `index.html` plus local assets, and the dashboard renders correctly when opened in a browser via `file://...`.

Work:

1. Add a new CLI command (Typer) to generate the dashboard:

   * Command name: `cookimport stats-dashboard`
   * Flags (keep minimal; add more only if truly needed):

     * `--output-root` (default: `data/output`)
     * `--golden-root` (default: `data/golden`)
     * `--out-dir` (default: `data/output/.history/dashboard`)
     * `--open` (boolean; if true, open the generated HTML in the default browser)
     * `--since-days` (optional; limit to recent runs for faster generation)
     * `--scan-reports` (boolean; force scanning `*.excel_import_report.json` in addition to/without relying on performance_history.csv)

   The command should:

   * collect data
   * optionally aggregate summary
   * write dashboard files
   * print the path to `index.html`

2. Implement HTML generation with a separation of concerns:

   Output layout:

   * `<out-dir>/index.html`
   * `<out-dir>/assets/style.css`
   * `<out-dir>/assets/dashboard.js`
   * `<out-dir>/assets/dashboard_data.json`

   Requirements:

   * No external CDN dependencies.
   * `dashboard.js` loads `dashboard_data.json` and renders charts/tables client-side.
   * Keep JS minimal. Prefer simple SVG-based sparklines or very small custom charting over introducing a heavy library.

3. UI structure goals for v1 (readable, boring, useful)

   Header:

   * Generated timestamp
   * Data roots used
   * Totals (number of stage records, benchmark records, total recipes, total runtime seconds where available)

   Throughput section:

   * Trend of seconds-per-recipe over time (only for runs where recipe count > 0 and time is present)
   * A table of slowest runs (by total seconds) with artifact paths displayed and clickable local links when possible

   Benchmarks section:

   * Trend of recall/precision (and F1 if available) over time, grouped by eval scope if present
   * A table of recent benchmarks (timestamp, source identifier if present, key metrics, link to eval folder, link to eval_report.md if present)

   Filters:

   * Run category toggles (stage_import vs benchmark_eval; allow enabling labelstudio_import / benchmark_prediction as opt-in)
   * Importer/source filters (dropdown populated from data)
   * Date range presets (7/30/90 days)

4. Rendering robustness

   * Missing metrics: omit from charts rather than showing zero.
   * No benchmark data: hide the benchmark section with a friendly note.
   * Path links: if you can safely create relative links from `out-dir` to artifact dirs, do so; otherwise display absolute paths and add a “copy” button in JS.

Acceptance:

* Running the command produces the output files and opening `index.html` shows tables and at least one trend visualization when data exists.
* Links do not 404 when using default directory layouts.

### Milestone 4: Tests, docs, and hardening

At the end of this milestone, dashboard generation is covered by tests that run in CI without requiring real cookbooks or Label Studio. Documentation tells a novice how to generate the dashboard and what questions it answers.

Work:

1. Add fixtures for testing collectors:

   * A tiny `performance_history.csv` fixture with a handful of representative rows/columns.
   * A minimal `*.excel_import_report.json` fixture capturing the key fields used by the fallback scanner.
   * A minimal benchmark fixture directory containing:

     * `eval_report.json`
     * optionally `coverage.json` and `manifest.json`

   Keep these fixtures tiny and synthetic; they should not contain large recipe bodies.

2. Add unit tests:

   * Stage collector: parses CSV, computes derived fields safely (no division by zero), classifies run categories.
   * Benchmark collector: parses eval report, reads optional coverage/manifest, classifies.
   * Generator: writes expected files to a temp directory; `dashboard_data.json` validates against `DashboardData` schema.

3. Add a short doc:

   * Where to place: `docs/08-analytics/` or `docs/02-cli/` (choose the existing convention)
   * Include:

     * command usage
     * where it reads from (`data/output/.history/performance_history.csv`, `data/golden/**/eval_report.json`)
     * how to interpret the key charts/tables
     * common pitfalls (see below)

4. Hardening checklist (non-Progress narrative, but ensure it is done)

   * Ensure dashboard generation is idempotent: re-running overwrites only files under `out-dir`.
   * Ensure collectors never write into `data/output` or `data/golden` (read-only).
   * Ensure timestamp parsing matches dot-separated time format.
   * Ensure `.job_parts` is ignored.
   * Ensure benchmark prediction artifacts don’t get counted as normal stage imports.

Acceptance:

* CI can run tests without external services.
* A novice can run the command and understand what they are looking at.

## Validation and Acceptance

Acceptance is met when all of the following are true:

* Running `cookimport stats-dashboard` on a machine with existing `data/output` and/or `data/golden` artifacts produces:

  * `data/output/.history/dashboard/index.html`
  * `data/output/.history/dashboard/assets/dashboard_data.json`
  * `data/output/.history/dashboard/assets/dashboard.js`
  * `data/output/.history/dashboard/assets/style.css`

* Opening `index.html` via `file://...` shows:

  * stage/import totals and at least one trend plot when stage history exists,
  * benchmark section when `eval_report.json` exists under `golden_root`,
  * a slowest-runs table with correct artifact paths,
  * filters that hide/show categories without errors.

* Tests pass, and the collectors handle missing/malformed records by skipping them (with a warning) rather than failing the whole run.

## Idempotence and Recovery

* The dashboard command must be safe to run repeatedly.
* It must only write inside `--out-dir` (default `data/output/.history/dashboard`).
* If some artifacts are malformed (partial run, truncated JSON), the generator must:

  * skip that record,
  * record/log a warning,
  * still produce a dashboard from the remaining valid data.

If the user reports “dashboard is empty”, the recovery path should be:

* Confirm `data/output/.history/performance_history.csv` exists; if not, run:

  cookimport perf-report

  then rerun:

  cookimport stats-dashboard

* Confirm there are `eval_report.json` files under `data/golden/`; if not, run an eval workflow first.

## Artifacts and Notes

Common pitfalls to explicitly guard against:

* Timestamp parsing mismatch: folders use `%Y-%m-%d_%H.%M.%S` (dots in time). If you parse with colons, sorting/grouping will break.
* Double-counting benchmark predictions:

  * benchmark eval directories may contain `prediction-run/`
  * labelstudio ingestion outputs may live under `data/output/<timestamp>/labelstudio/...`
    These should not inflate stage/import lifetime stats unless explicitly enabled as separate categories.
* Split-job temporary artifacts:

  * ignore `data/output/.job_parts/**` unless intentionally debugging merges
  * do not treat job parts as completed runs

When adding new derived metrics, always do “safe math”:

* If denominator is 0 or missing, record `None`, not `0` or `Infinity`.
