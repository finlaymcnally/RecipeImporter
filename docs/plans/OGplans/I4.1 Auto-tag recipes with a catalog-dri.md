# Auto-tag recipes with a catalog-driven rule engine (LLM-ready)

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan must be maintained in accordance with `PLANS.md` at the repository root.


## Purpose / Big Picture

After this change, a user can run a single CLI command to automatically suggest (and optionally apply) recipe tags from the existing tag catalog, without hard-coding tag IDs. The system will:

1) Load the current tag catalog at runtime (so adding tags later is naturally supported).

2) Run a deterministic, rule-based tagger first (regex + lightweight parsing over recipe title/notes/ingredients/steps + a few numeric fields like total time).

3) Optionally call an LLM as a “second pass” only for the categories the deterministic tagger cannot confidently fill, using a short-listed candidate set so the prompt stays small.

The observable outcome is that you can point the tool at staged recipe drafts (`data/output/.../final drafts/.../*.json`) or at recipe rows in Postgres/Supabase, and get a clear list of tags with confidence and “why” evidence. If `--apply` is used, the tool will insert rows into `public.recipe_tag_assignments` safely and idempotently (re-running does not create duplicates).

How to see it working (end state):

    # 1) Export catalog once (or load directly from DB every time)
    cookimport tag-catalog export --db-url "$COOKIMPORT_DATABASE_URL" --out data/tagging/tag_catalog.json

    # 2) Suggest tags for staged drafts (dry run)
    cookimport tag-recipes suggest --draft-dir "data/output/<run>/final drafts/<workbook>/" \
      --catalog-json data/tagging/tag_catalog.json \
      --out-dir "data/output/<run>/tags/<workbook>/" \
      --explain

    # 3) Suggest + apply tags for one DB recipe (optional)
    cookimport tag-recipes apply --db-url "$COOKIMPORT_DATABASE_URL" --recipe-id "<uuid>" \
      --catalog-json data/tagging/tag_catalog.json \
      --explain


## Scope / Guardrails (avoid accidental damage)

- This plan is about **auto-suggesting** tags from the existing catalog and optionally writing **assignments**. It does **not** create new tags/categories or change the catalog schema.
- DB writes are **insert-only** (`ON CONFLICT DO NOTHING`). We will not delete/replace tags in this plan because the schema does not track “who assigned this tag,” and deletion risks removing user curation.
- Precision-first: prefer missing a tag over adding a wrong tag. Start conservative and expand coverage iteratively.
- LLM support is optional, disabled by default, and must fail safe (no provider configured ⇒ deterministic only).

## Quality Bar (explicit)

Before enabling DB apply on any meaningful batch:

- Determinism: same inputs + same catalog ⇒ identical outputs (ordering, confidence, evidence).
- Evidence: every suggested tag includes at least one human-readable reason (field + matched phrase where possible).
- Conservative defaults: especially for “single” categories (main protein/carb, course, meal type, etc.).
- A small gold set exists (even 20–50 recipes) with an eval harness to prevent regressions while rules evolve.

## Context and Orientation

This repository is a Python 3.12 ingestion and normalization pipeline with a Typer CLI (`cookimport`). A common workflow is `cookimport stage <path>`, which produces intermediate drafts and final cookbook3 drafts on disk under `data/output/<timestamp>/...`, including:

- `intermediate drafts/<workbook>/r{index}.jsonld`
- `final drafts/<workbook>/r{index}.json`

The transformation to final cookbook3 drafts happens in `write_draft_outputs(...)` via `recipe_candidate_to_draft_v1(...)`, where ingredient parsing and step linking run uniformly across sources.

Separately, the product database schema stores recipes and tagging using these tables:

- `public.recipes` for recipe-level fields (title, description, times, etc.)
- `public.recipe_steps` for ordered instructions
- `public.step_ingredient_lines` for ingredient lines (raw text + parsed numeric/unit fields)
- `public.recipe_tags` is the tag catalog (`id`, `key_norm`, `display_name`, `category_id`, ...)
- `public.recipe_tag_assignments` stores assignments (PK is `(recipe_id, tag_id)`)

The tag catalog is seeded (and should remain the source of truth for initial tags) via `supabase/seeds/catalog.sql`. The current catalog is “20 categories” and “203 recipe tags.”

This ExecPlan adds a new “tagging” subsystem to `cookimport` that can operate on staged drafts (files) and optionally on database rows (Postgres/Supabase).


## Definitions (avoid jargon)

- Tag catalog: The set of allowed tags and their metadata. In this repo, it’s the rows of `public.recipe_tags` plus the category table those tags reference.
- Tag key_norm: A stable, normalized string key for a tag (example style: `instant-pot`, `weeknight`). We key rules and I/O off this value.
- Category: A grouping of tags (“cooking-style”, “main-protein”, etc.) that may have constraints such as “pick at most one tag in this category.”
- Deterministic tagger: A rule engine that assigns tags based on explicit string/regex matches and simple numeric heuristics, producing repeatable outputs.
- LLM second pass: An optional step that chooses tags when deterministic rules are uncertain, using the existing catalog as the allowed label set.
- Evidence: A short explanation of why a tag was suggested (for example: “matched ‘instant pot’ in title”).


## Architecture Overview

We will build a small, testable tagging library plus CLI plumbing:

1) Catalog loading:
   - Load categories + tags from the database (preferred) or from an exported JSON snapshot file.
   - Build lookups: by `key_norm`, by `category_key_norm`, by `category_id`.

2) Signal extraction:
   - Convert a recipe (draft file or DB rows) into a `RecipeSignalPack`:
     - normalized text fields: title, description, notes
     - lists of ingredient strings and instruction strings
     - numeric metadata: total_time_minutes, prep_time_minutes, cook_time_minutes, yield phrase, etc.

3) Deterministic scoring:
   - For a curated subset of tags (starting with high-signal categories like cooking style, equipment, effort, main protein/carb, cooking method), compute a confidence score from 0..1 and attach evidence.
   - Apply category policies:
     - “single”: pick only the top tag if above threshold
     - “multi”: pick all tags above threshold, up to a max

4) Optional LLM second pass:
   - Only for categories that are required or important but missing/low-confidence after deterministic scoring.
   - Provide a short list of candidate tags per category (generated via fuzzy matching over tag display names) instead of dumping all 203 tags into the prompt.

## Milestone 3: Deterministic tagging engine (rules + category policies)

Scope:

At the end of this milestone, we can run deterministic tagging for a recipe and get a list of suggested tag `key_norm` values with confidence and evidence, and we can enforce simple “single vs multi” category rules.

Work:

1) Create `cookimport/tagging/rules.py` containing a pure-Python ruleset keyed by tag `key_norm`.

Rule format (keep it intentionally simple and editable):

- `TAG_RULES: dict[str, dict]` where each entry can contain:

  - `patterns`: list of pattern rules:
    - `field`: one of `title`, `description`, `notes`, `ingredients`, `instructions`
    - `regex`: a python regex string (assume normalized text)
    - `weight`: float (typical 0.2–1.0)
    - `evidence`: short text template for “why”
  - `exclude_patterns` (optional): list of regex strings; if any match, suppress this tag (prevents common false positives).
  - `min_score`: float threshold for this tag (default 0.6)

Example (do not use code fences inside this ExecPlan; this is shown as an indented shape only):

    TAG_RULES = {
      "instant-pot": {
        "min_score": 0.7,
        "patterns": [
          {"field": "title", "regex": r"\binstant pot\b|\bip\b", "weight": 0.8, "evidence": "matched Instant Pot in title"},
          {"field": "instructions", "regex": r"\bpressure cook\b|\bnatural release\b|\bquick release\b", "weight": 0.4, "evidence": "matched pressure-cooker phrasing in instructions"},
        ],
      },
      "slow-cooker": {
        "min_score": 0.7,
        "patterns": [
          {"field": "title", "regex": r"\bslow cooker\b|\bcrock[- ]pot\b", "weight": 0.8, "evidence": "matched slow cooker in title"},
        ],
      },
      ...
    }

Rule authoring footguns (read before adding patterns):

- Prefer ingredient-driven signals for main protein/carb; title-only matches are a common source of false positives.
- Beware short ambiguous tokens (e.g., `ip`). If you include them, require a second supporting match (instructions phrasing) or add exclude patterns for common collisions.
- Always use word boundaries in regex where appropriate; normalization can change punctuation/spacing and create accidental matches.
- Add at least one negative test when you add a new high-impact rule.

Start with a curated subset that delivers value quickly and is relatively low-risk:

- Cooking Style / Equipment / Cooking Method:
  - instant-pot, slow-cooker, air-fryer, grill, sheet-pan, one-pot, no-cook, stovetop-only, oven-only
  - blender, food-processor, dutch-oven, wok, cast-iron (equipment category)
  - bake/roast/grill/steam/simmer/saute (cooking-method category)

- Effort:
  - quick (total_time_minutes <= 25 or title contains “quick”)
  - hands-off-friendly (attention_level == set_and_forget OR “simmer for 2 hours” signals; keep conservative)
  - minimal-prep / minimal-chopping (only if clear phrases like “no chopping” / “pre-chopped” appear; otherwise skip)

- Main protein / main carb (ingredient-driven):
  - chicken/beef/pork/fish/shrimp/tofu/chickpeas/lentils/eggs
  - rice/pasta/potatoes/tortillas/bread
  - Keep these conservative: require ingredient matches, not just title mentions.

2) Category policies:

Create `cookimport/tagging/policies.py`:

- `CATEGORY_POLICIES: dict[str, dict]` keyed by category `key_norm`:

  - `mode`: `"single"` or `"multi"`
  - `min_confidence`: default threshold if tag-specific min not provided
  - `max_tags`: for multi categories

Suggested defaults:

- single:
  - `main-protein`, `main-carb`, `meal-type`, `course`
- multi:
  - `equipment`, `cooking-method`, `storage`, `occasion` (but deterministic may not cover all yet; policy still applies when it does)
- leave many categories to LLM later (`cuisine`, `dietary`, `flavor-profile`) by not writing deterministic rules for them yet.

3) Scoring engine:

Create `cookimport/tagging/engine.py`:

- `TagSuggestion`
  - `tag_key: str`
  - `category_key: str`
  - `confidence: float`
  - `evidence: list[str]`

- `suggest_tags_deterministic(catalog: TagCatalog, signals: RecipeSignalPack) -> list[TagSuggestion]`

Algorithm:

- Build normalized searchable strings:
  - title/description/notes: normalized text
  - ingredients/instructions: normalized text joined with newlines (and also keep per-line list for better evidence)
- For each tag in `TAG_RULES`:
  - If tag key is not present in `catalog.tags_by_key`, skip it (this is what makes the ruleset forward-compatible with catalog changes).
  - If any `exclude_patterns` match (in any relevant field), suppress this tag entirely.
  - Compute `raw_score = sum(weight for each matched pattern)`.
  - Convert to `confidence` by clamping: `confidence = min(1.0, max(0.0, raw_score))`.
  - Keep evidence for each matched pattern.
  - Keep the suggestion only if `confidence >= max(rule.min_score, category_policy.min_confidence)`.

- Apply category policies:
  - Group by `category_key`.
  - If mode=single: keep only the highest-confidence tag; if tie, prefer the tag whose `display_name` appears in the title; else stable sort by tag sort_order.
  - If mode=multi: keep up to `max_tags` sorted by confidence descending.

4) Output formatting:

Create `cookimport/tagging/render.py` with:

- `render_suggestions_text(...)` for console (Rich-friendly)
- `serialize_suggestions_json(...)` for writing `*.tags.json` per recipe

Acceptance:

Add a minimal test fixture recipe signal pack and prove deterministic tags work:

    pytest -q

And prove CLI can run deterministic tagger (draft file mode):

    cookimport tag-recipes suggest --draft "data/output/<run>/final drafts/<workbook>/r0.json" \
      --catalog-json data/tagging/tag_catalog.json \
      --explain

Expected output shape (example):

    Recipe: Instant Pot Chicken Chili
    Suggested tags:
      - cooking-style/instant-pot (0.80)  matched Instant Pot in title
      - main-protein/chicken (0.70)       matched 'chicken' in ingredient lines
      - cooking-method/pressure-cook (0.60) matched pressure cook phrasing in instructions

Exact tags will vary based on the fixture content, but the key acceptance criterion is: deterministic + evidence + category policy filtering works.

## Milestone 6: LLM second pass scaffolding (disabled by default)

Scope:

Work:

1) Add `cookimport/tagging/llm_second_pass.py` that exposes one function:

- `suggest_tags_with_llm(signals: RecipeSignalPack, catalog: TagCatalog, missing_categories: list[str], deterministic: list[TagSuggestion]) -> list[TagSuggestion]`

2) Prompt-injection and safety note:

- Treat recipe text as untrusted input. Instruct the model to output only from provided candidates, and still validate outputs strictly.

3) Candidate shortlisting (important for prompt size):

  - Build a shortlist of up to 10 candidate tags by fuzzy matching tag display_name against the recipe’s combined text.
  - You can reuse RapidFuzz (already used in repo) if available; otherwise implement a simple token overlap scorer.

4) LLM interface:

- Use the existing `cookimport/llm/` integration layer. If it is still mocked, implement a mock provider path that returns an empty list, but preserves the end-to-end call shape.
- Define a strict JSON schema for the LLM response, for example:

    {
      "suggestions": [
        {"category": "cuisine", "tag_key": "mexican", "confidence": 0.72, "why": "mentions tortillas + salsa + cumin"},
        ...
      ]
    }

5) Parse and validate (required):

- `tag_key` must exist in catalog.
- `category` must match the tag’s category.
- `confidence` must be 0..1.
- For now, require that `tag_key` is within the candidate shortlist for that category (this keeps prompts small and prevents the model from “inventing” tags).
- Reject (or ignore) any extra keys, free-form text, or schema violations.

6) Fail-safe behavior (required):

- If no provider is configured, warn once and return an empty list (deterministic only).
- If the provider errors or times out, warn and return an empty list.

7) CLI wiring:

- Add `--llm` flag to `cookimport tag-recipes suggest` and `apply`.
- Only call LLM if:
  - flag enabled AND provider configured AND there are missing categories
- Merge deterministic + LLM suggestions and re-apply category policies.

Acceptance:

- With `--llm` on but no provider configured, the CLI should print a single warning and continue with deterministic results.
- With mocked provider, a unit test should prove the code path executes and merges results without crashing.


## Testing Strategy (do not skip)

Add tests under `tests/`:

0) Gold set regression tests:
- The tiny gold set from Milestone 0 must run in CI and protect precision in high-impact categories.

1) Unit tests (fast, deterministic):
- Scoring: given a `RecipeSignalPack`, verify tag confidence and evidence for a small ruleset.
- Category policy: verify “single” keeps only the top tag and “multi” keeps multiple.
- Catalog resolution: verify missing tags in rules are skipped cleanly.

2) End-to-end test (file mode):
- Add a tiny fixture cookbook3 draft JSON under `tests/fixtures/`.
- Add a tiny fixture `tag_catalog.json` subset (just enough tags/categories for the fixture).
- Run `cookimport tag-recipes suggest` in a test and assert output JSON contains expected tag keys.

3) DB integration tests:
- Optional in this plan (depends on CI environment). If CI can run Postgres, add one integration test; otherwise keep DB code covered by unit tests + manual acceptance checks.


## Operational Notes (idempotence, safety, debugging)

- Idempotence:
  - File outputs should be overwritten deterministically.
  - DB inserts must use `ON CONFLICT DO NOTHING`.

- Debugging:
  - Always support `--explain` to show evidence.
  - Add `cookimport tag-recipes debug-signals` to inspect what the tagger “sees.”

- Catalog drift:
  - If applying tags to DB using a JSON snapshot, verify IDs exist in the target DB before writing.
  - Record `catalog_fingerprint` in every run report.

- Over-tagging:
  - Keep “single” category thresholds conservative; missing is better than wrong.
  - Prefer ingredient-driven evidence for main protein/carb.

- LLM safety:
  - Validate model outputs strictly; ignore anything outside candidates or wrong-category.

- Reporting:
  - Write a run report JSON that records:
    - catalog version info (counts, maybe a hash of `tag_catalog.json`)
    - inputs processed
    - per-recipe selected tags and confidences
    - failures with stack traces (shortened)

- Performance:
  - Keep per-recipe tagging O(number_of_rules) with precompiled regex where possible.
  - For bulk runs, load catalog once and reuse.


## You can do this later: “making new tags” (do not implement now)

This plan intentionally does not implement migrations/UX for creating new tags. The system is designed so new tags fit naturally:

- Add tags/categories to seed + DB.
- Re-export catalog JSON (or load from DB).
- Optionally add deterministic rules keyed by new tag `key_norm`.
- If no rules, LLM second pass (when enabled) can start using the new tag immediately.
- Evaluate and tune thresholds/rules based on false positives/negatives.
