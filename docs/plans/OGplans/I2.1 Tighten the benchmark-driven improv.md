# Tighten the benchmark-driven improvement loop for cookimport parsing (freeform golden sets)

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md`, which defines the required ExecPlan format and workflow rules.

## Purpose / Big Picture

After this change, a developer (or AI coding agent) can improve the EPUB/PDF parsing pipeline in fast, repeatable iterations using **existing freeform golden sets**, without needing Label Studio running and without needing to understand Label Studio internals.

The user-visible proof is a tight loop that works end-to-end with one command:

* Run an **offline** benchmark suite against freeform gold exports.
* Receive a machine-readable report plus an **iteration packet** (ranked examples + context + “why” signals + failure mode summaries + tunable knobs involved).
* Make a code or config change, rerun, and immediately see what improved/regressed and why.

A secondary proof is spend-awareness: each run quantifies “how many LLM repair calls would happen” under current thresholds (no paid calls), with estimated tokens/cost, so the fastest wins come from reducing noise and duplication before turning on any real LLM repair.

## Decision Log

* Decision: The benchmark loop will be built on top of the **existing freeform export + eval contracts** (freeform spans + segment manifest; eval outputs including missed/false-positive JSONL), rather than inventing a parallel scoring format.
  Rationale: This avoids schema drift and preserves continuity with existing golden sets and evaluation logic. It also makes offline benchmarking a “thin wrapper” around proven components.
  Date/Author: 2026-02-12 / ExecPlan author

* Decision: Add an explicit **non-interactive offline** benchmark path (suite-capable) that does not upload and does not require Label Studio credentials.
  Rationale: Current CLI benchmark behavior is upload-first; many users expect “benchmark” to be scoring-only. Offline iteration is the fastest path for heuristic improvement and is safer by default.
  Date/Author: 2026-02-12 / ExecPlan author

* Decision: Matching/scoring will remain **chunking-invariant** by default for freeform gold: use touched block index sets (or their overlap) as the stable coordinate system.
  Rationale: Freeform gold intentionally does not mirror internal chunk boundaries. Scoring should reward “picked it up in the right region,” while still optionally reporting boundary quality as a secondary metric.
  Date/Author: 2026-02-12 / ExecPlan author

* Decision: Precision will be reported with explicit caveats and additional “prediction density/noise” metrics.
  Rationale: Unlabeled regions in freeform gold are not guaranteed negatives. Treating all unlabeled text as negative makes precision misleading; density and “suspected FP” diagnostics are more actionable for cost control.
  Date/Author: 2026-02-12 / ExecPlan author

* Decision: Suite runs will default to writing under `data/golden/bench/…` (not a new top-level `data/bench/…`) and will reuse the project’s timestamp folder format.
  Rationale: Golden/eval artifacts already live under `data/golden`; keeping benchmark artifacts adjacent reduces path confusion and keeps “labeling/eval/bench” in one place.
  Date/Author: 2026-02-12 / ExecPlan author

## Context and Orientation

### What this repo does (relevant slice)

`cookimport` ingests recipes from EPUB/PDF (and other formats) by extracting a linear stream of text **blocks**, enriching them with **signals** (heading-ness, ingredient-likeness, etc.), segmenting them into recipe/tip candidates, and then writing structured outputs.

### Key terms (use these consistently)

* Block: The smallest stable text unit used for extraction and evaluation (roughly a paragraph/list/header line). Blocks are ordered and have indices `0..N-1` within a source. Block indices are the stable coordinate system for freeform evaluation.
* Pipeline predictions: The spans/ranges/chunks the pipeline believes correspond to labeled structures (recipe title, ingredient lines, instruction lines, tips, etc.). Internally this may be represented as candidates or tasks; for benchmarking we treat them as labeled block ranges/sets.
* Freeform gold span: A human-highlighted span (in Label Studio) with a label. In export, each gold span is mapped to the set of block indices it touches (directly or via a segment manifest mapping offsets back to blocks).
* Segment manifest: A JSONL mapping that ties uploaded “segment text” offsets back to original block indices so span highlights can be evaluated in block space.
* Suite item: One benchmark case mapping (source file + freeform gold export + per-item options like force-source-match).

### Where the relevant code lives today

Label Studio workflows are implemented under `cookimport/labelstudio/`:

* `ingest.py`: extraction + task generation + (optionally) upload + run artifacts
* `export.py`: export shaping to JSONL golden sets (including freeform spans + segment manifest)
* `eval_freeform.py`: freeform evaluation and report artifacts
* tests under `tests/test_labelstudio_*.py`

The offline benchmark loop in this plan must reuse the **existing export/eval contracts** and should reuse ingestion/task generation where possible, but must not require the Label Studio server when running offline.

### Current behavior that motivates this plan

* The non-interactive CLI benchmark path is currently **upload-first**: it runs a prediction import/upload before scoring. This is surprising to many users and blocks offline iteration.
* Freeform eval can collapse to near-zero matches if source identity mismatches or if split-job block-index rebasing is broken.
* Freeform labels and ID prefixes have had historical drift; eval normalizes some legacy labels, but any new tooling must not reintroduce drift.

### Known footguns to avoid (and how this plan prevents them)

1. “Benchmark” accidentally uploads or needs LS credentials.

   * Fix: offline mode must never initialize the Label Studio client and must not require `LABEL_STUDIO_URL/API_KEY`. Upload remains gated behind explicit consent and a separate flag/path.

2. Near-zero overlap due to source mismatch.

   * Fix: suite items must carry a `force_source_match` boolean, default false. When false, runs should fail fast with a clear error when the source identity/hash doesn’t match the gold export’s recorded identity. When true, record that override prominently in reports.

3. Near-zero overlap due to split-job merge index rebasing regression.

   * Fix: offline pred-run generation must reuse the same merge/rebase logic as stage/import paths. Add a regression test that a split-run produces globally increasing block indices and that eval overlap is non-trivially > 0 on a known fixture.

4. Misleading precision due to unlabeled ≠ negative in freeform gold.

   * Fix: emphasize recall/coverage and add “prediction density” and “suspected FP” diagnostics instead of pretending unlabeled blocks are ground-truth negatives.

5. Timestamp/path drift causing “where did my outputs go?” confusion.

   * Fix: write everything under a single predictable root (`data/golden/bench/...`) and include `suite_used.json` and absolute/relative paths in manifests.

6. ID/URN prefix drift (`urn:cookimport:*` vs `urn:recipeimport:*` historical confusion).

   * Fix: do not hardcode ID prefixes in new code. Parse and propagate whatever IDs the current ingestion code emits; only key off stable fields (source hash, block index, deterministic item IDs).

## Plan of Work

This plan is organized as milestones that each produce a verifiable, working outcome. The story is:

* Make offline benchmarking possible without Label Studio.
* Make it suite-capable and deterministic.
* Make outputs actionable for autonomous iteration (iteration packet).
* Then add tuning tooling (knobs + sweep) and cost-aware noise reduction.

### Milestone 0: Baseline reality check (preserve what already works)

Work:

* Identify at least one existing pred-run folder that freeform eval can consume (a prior labelstudio import/benchmark run).

Proof:

### Milestone 1: Define a bench suite manifest (dev/test) and validate it

At the end of this milestone, there is a suite manifest format checked into the repo (or stored in a deterministic local path) that enumerates benchmark items and validates inputs.

Suite item fields (minimum):

* `item_id`: stable string ID
* `source_path`: repo-relative path to the input file (EPUB/PDF, etc.)
* `gold_dir`: path to the gold export directory (the folder that contains `exports/…`)
* `force_source_match`: bool (default false)
* optional notes

Suite structure:

* Two manifests: `dev` (iteration) and `test` (held out).

Proof:

* A `cookimport bench validate --suite <path>` command (or equivalent) checks that:

  * source files exist,
  * gold export files exist,
  * required export filenames exist,
  * and the gold export’s recorded source identity matches the provided source (unless forced).

### Milestone 2: Add an explicit offline pred-run builder (no upload, no LS dependency)

At the end of this milestone, the system can generate a “pred-run” folder locally for a given source file without talking to Label Studio, and it is compatible with the existing freeform eval.

Design constraints:

* Must not require Label Studio server or env vars.
* Must not upload anything.
* Must write the same core artifacts that eval expects from a pred-run (for example tasks JSONL and any extracted archive/text it needs for context).

Implementation approach:

* Refactor the ingestion path so “generate tasks + write run artifacts” is separable from “upload tasks”.
* Add a function (or option) like `upload=False` / `offline=True` to the internal runner so CLI can request artifact generation only.
* Ensure this path still honors split-job planning/merge behavior so block indices are in the same global coordinate space as gold.

Proof:

* Running the offline pred-run builder on one suite item produces a pred-run directory.
* Running freeform eval using that pred-run directory and the gold export produces non-empty, sensible metrics and the usual eval artifacts.

### Milestone 3: Offline suite runner with deterministic, reproducible run artifacts

At the end of this milestone, a single command runs the whole dev suite offline and produces a stable run directory containing:

* per-item pred-run artifacts (or references to cached pred-runs),
* per-item eval outputs,
* an aggregate suite report (MD) and metrics (JSON),
* and a copy of the suite manifest used.

Determinism requirements:

* Any randomness (sampling, ordering, sweep seeds) must be removed or seeded and recorded.
* If caching is introduced (recommended), it must be keyed by source content hash + config hash and be safe to delete.

Proof:

* Running the suite twice with the same code/inputs/config produces identical aggregate metrics JSON (ignoring timestamps/paths).
* The run directory contains enough information to reproduce the run later (suite copy + effective config + version info).

### Milestone 4: Iteration packet (make the benchmark “explain itself”)

At the end of this milestone, each suite run produces an `iteration_packet/` designed to be directly consumable by an AI coding agent (or a human) to fix the next most impactful issues.

The packet must include:

* A short summary that highlights:

  * metric deltas vs an optional baseline run,
  * prediction volume/density deltas,
  * top regressions.
* Ranked failure cases with enough context to fix them:

  * missed gold spans,
  * suspected false positives,
  * fragmentation/over-splitting cases (one gold span covered by many tiny preds),
  * “confusion pairs” (pred label A overlaps gold label B).
* For each case, include:

  * the relevant block text context (a small window),
  * the signals/features for those blocks (heading/ingredient-ish/etc.),
  * and where possible the rule/threshold “reason codes” that led to the decision (see next milestone).
* A clear separation between:

  * chunking-invariant coverage metrics (primary for freeform),
  * strict overlap metrics (secondary),
  * and density/noise metrics (primary for cost control).

Proof:

* `iteration_packet/README.md` explains how to use the packet.
* `iteration_packet/cases.jsonl` contains self-contained case records.
* `iteration_packet/top_failures.md` renders the top N cases in a readable way with minimal context needed for a fix.

### Milestone 5: Optional decision tracing for segmentation/chunking (high leverage diagnostics)

At the end of this milestone, the parsing pipeline can run in a “trace” mode that records structured events explaining why spans started/ended, why blocks were classified, and which thresholds were applied.

Requirements:

* Off by default.
* Enabled by a CLI flag (and threaded through worker/split-job runs).
* Written as JSONL (one event per line), diffable and machine-readable.
* Trace events must include a stable correlation key that ties them to predicted spans and block indices.

Proof:

* With trace enabled, a pred-run contains a `traces/<item_id>.trace.jsonl`.
* The iteration packet links trace excerpts to top failure cases.

### Milestone 6: Knob registry + config loader (safe tuning without code hunting)

At the end of this milestone, key thresholds/weights that drive segmentation and classification can be overridden via a config file, with:

* a registry describing each knob (name, type, default, bounds, description),
* a loader that applies overrides,
* and run artifacts that record the effective knob values.

Constraints:

* Prefer to reuse any existing config patterns in the repo (do not introduce a second incompatible config system).
* The knob system should be explicit and safe: bounds checked, unknown keys rejected (or warned loudly).

Proof:

* `cookimport bench run --suite <dev> --config knobs.dev.json` changes outcomes and records effective knobs in the run directory.

### Milestone 7: Deterministic parameter sweep tooling (cheap local autotuning)

At the end of this milestone, there is a tool to try a small grid (or seeded search) of knob values on the dev suite and produce:

* a leaderboard of configs and scores,
* a “best config” file,
* and a confirmation run on the test suite to detect overfitting.

Objective definition (must be explicit in code and in output):

* maximize chunking-invariant coverage on core labels,
* minimize prediction density/noise,
* optionally improve strict overlap as a tie-breaker.

Proof:

* Re-running the sweep with the same seed produces the same best config and leaderboard ordering.

### Milestone 8: Noise reduction primitives + LLM cost estimator (counting only)

At the end of this milestone:

* prediction volumes drop substantially on dev without harming chunking-invariant coverage beyond an explicit tolerance,
* and the run metrics include an LLM escalation/cost section that changes as knobs change.

Implement/validate these primitives if they do not already exist:

* Deduplication: collapse identical/equivalent predicted spans.
* Consolidation: prefer a small set of high-confidence, minimally overlapping spans per label.
* Noise gating: reduce emission of narrative prose as actionable content.

LLM estimator requirements:

* Emit an `llm_escalation_queue.jsonl` (counting only, no calls).
* Estimate tokens using the intended prompt template and context window policy.
* Summarize cost using configurable pricing assumptions.

Proof:

* The suite report shows before/after prediction counts and the estimated cost impact.

### Milestone 9: Tests + docs (novice-proof, regression-proof)

At the end of this milestone, a novice can add a new gold export to the suite and run the loop offline.

Add tests for:

* freeform span → touched blocks mapping invariants (and legacy label normalization),
* offline pred-run generation does not touch Label Studio client code paths,
* split-job merge index rebasing remains correct,
* dedupe/consolidation invariants,
* knob loading and bounds checks.

Docs:

* Add a short runbook in `docs/` that shows:

  * how to create/update a suite,
  * how to run offline bench,
  * how to interpret iteration packet outputs,
  * and how to safely try knob edits/sweeps.

Proof:

* Running the repo test command passes, and the doc includes a minimal “quickstart transcript” with expected output filenames.

## Validation and Acceptance

Acceptance is behavior, not internal structure.

The work is accepted when:

* Offline benchmark runs end-to-end on the dev suite with one command and produces:

  * an aggregate markdown report,
  * a machine-readable metrics JSON,
  * per-item eval outputs compatible with current freeform eval,
  * and an iteration packet with ranked failure cases including block context and signals.

* Offline truly means offline:

  * No upload occurs.
  * No Label Studio server/env vars are required.
  * The tool fails fast if an upload-required path is attempted without explicit consent.

* Determinism holds:

  * Re-running with identical inputs/config yields identical metrics (ignoring timestamps/paths).

* The iteration packet is sufficient for autonomous iteration:

  * A reasonable engineer can fix at least one top failure by reading only the packet and the codebase (no Label Studio replay required).

* The knob/config system allows safe tuning and is recorded in run artifacts.

* Sweep results are deterministic and include a held-out test confirmation run.

* Tests cover the core matching/mapping logic and the offline pred-run generation path.

## Idempotence and Recovery

This plan must be safe to run repeatedly.

* All outputs are written under `data/golden/bench/…` with timestamped subfolders.
* Deleting a run folder must be sufficient cleanup; caches (if any) must be optional and safe to delete.
* Offline pred-run generation must not mutate gold exports.
* If a run is interrupted, rerunning should either resume safely or start fresh in a new timestamped directory without corrupting prior runs.
