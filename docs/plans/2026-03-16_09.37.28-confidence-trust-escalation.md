# Split Mixed Confidence Into Trust and Escalation

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan starts from the current post-Phase-4 runtime that already has label-first authority, deterministic recipe-span grouping, deterministic Stage 7 non-recipe ownership, one recipe-correction stage, and canonical run observability. The problem is narrower and more specific: the main label-first seam still stores a single scalar `confidence` that mixes trust and escalation meanings, even though the runtime now has enough stage structure to separate them.

## Purpose / Big Picture

After this change, a human inspecting a stage run should be able to answer two different questions without reading code. First: how trustworthy is this label, block, or recipe span? Second: why was this item sent to Codex, or why was it not? Those answers must come from explicit fields such as `trust_score`, `escalation_score`, `decided_by`, and `escalation_reasons`, not from guesswork around one overloaded `confidence` float.

The user-visible result should be a normal `cookimport stage` run whose `label_det`, `label_llm_correct`, and `group_recipe_spans` artifacts contain explicit trust and escalation metadata. Recipe grouping and Stage 7 knowledge eligibility should still be driven by final labels, not by scores. Benchmark and follow-up tooling should switch from “low confidence” language to “low trust” plus explicit escalation reasons. A contributor should be able to inspect one run root and tell whether a label was trusted, why it was or was not escalated, and whether a span’s uncertainty came from label disagreement, fallback routing, or structural inconsistency.

This plan is intentionally pragmatic. It does not try to invent a second classifier or a probabilistically calibrated model from thin air. It formalizes the signals the code already has, renames their responsibilities clearly, and removes the remaining places where the mixed scalar `confidence` still acts like architecture instead of supporting metadata.

## Surprises & Discoveries

- Observation: the major refactor already removed scalar confidence from the most important authority boundaries.
  Evidence: `cookimport/parsing/recipe_span_grouping.py` groups recipe spans from `final_label` sequences only, and `cookimport/staging/nonrecipe_stage.py` derives Stage 7 `knowledge` versus `other` from final labels only.

- Observation: the remaining live escalation path is much smaller than the user-facing policy section implies.
  Evidence: `cookimport/parsing/canonical_line_roles.py::_should_escalate_low_confidence_candidate(...)` is the one active runtime place where the mixed scalar confidence still directly controls Codex escalation.

- Observation: the repo already has enough structured metadata to support a pragmatic split without inventing a new model.
  Evidence: authoritative line and block rows already persist `deterministic_label`, `final_label`, `decided_by`, and `reason_tags`, and recipe spans already persist structural warnings.

- Observation: several distinct scoring systems share the word “confidence” but are not the same thing.
  Evidence: the label-first seam has line/block `confidence`; importer and web schema flows have schema confidence thresholds; recipe gating has `RecipeLikenessResult.score`; step-ingredient linking has line-level confidence penalties; benchmark follow-up tooling uses “confidence threshold” for triage exports.

- Observation: a full repo-wide rename would be too broad for one safe cutover.
  Evidence: `RecipeCandidate.confidence`, `provenance.confidence_score`, importer debug reports, and benchmark review packets all still refer to existing fields that many unaffected readers expect.

- Observation: the line-role cache seam was still implicitly single-list even though the runtime returns `(predictions, baseline_predictions)`.
  Evidence: cache-hit parsing tests failed with `ValueError: not enough values to unpack (expected 2, got 1)` until the cache payload started persisting `baseline_predictions` alongside `predictions`.

## Context and Orientation

The current post-refactor runtime has three different kinds of scoring or confidence-like signals.

The first kind is authoritative Stage 2 line and block labeling. `cookimport/parsing/canonical_line_roles.py` labels atomized lines deterministically, optionally escalates some of them to Codex, arbitrates ownership between deterministic syntax rules and Codex overrides, and returns `CanonicalLineRolePrediction` rows. `cookimport/parsing/label_source_of_truth.py` turns those rows into `AuthoritativeLabeledLine` and `AuthoritativeBlockLabel` objects, then groups recipe spans and writes stage artifacts. These models already store `decided_by` and `reason_tags`, but they still store only one scalar `confidence`.

The second kind is label-first downstream ownership. `cookimport/parsing/recipe_span_grouping.py` groups contiguous recipe spans from final block labels. `cookimport/staging/nonrecipe_stage.py` builds deterministic Stage 7 `knowledge` and `other` spans from final labels and recipe-span exclusions. `cookimport/staging/stage_block_predictions.py` then prefers Stage 7 ownership for `KNOWLEDGE`. None of these modules should be taught to make primary decisions from scalar scores.

The third kind is legacy or domain-specific scoring outside the label-first authority seam. `cookimport/core/scoring.py` computes recipe-likeness scores and tiers for importer-side recipe gating. `cookimport/config/run_settings.py` exposes thresholds such as `web_schema_min_confidence` and `recipe_score_*`. These are still legitimate domain-specific scores, but they are not the same thing as Stage 2 label trust or Stage 2 escalation.

In this plan, “trust score” means a scalar in `[0, 1]` that estimates how likely the chosen label or derived object is correct. It is a reviewer-facing and QA-facing signal. It can remain approximate and uncalibrated in the first cut, but it must no longer decide whether an item is grouped into a recipe or classified as knowledge. “Escalation score” means a scalar in `[0, 1]` that estimates how strongly an item should be sent to Codex or highlighted for expensive review. It is a control signal, not a correctness claim. “Decision metadata” means the structured evidence attached to a line, block, or span that explains how the system reached a decision. In this repository that should include fields derived from existing code paths such as `deterministic_label`, `final_label`, `decided_by`, `reason_tags`, structural warnings, and explicit escalation reasons.

The main file surfaces for this work are:

- `cookimport/parsing/canonical_line_roles.py`: deterministic labeling, Codex escalation, fallback normalization, ownership arbitration.
- `cookimport/parsing/label_source_of_truth.py`: authoritative line and block models, recipe-candidate bridge, stage-backed label artifact content.
- `cookimport/parsing/recipe_span_grouping.py`: deterministic grouping from final labels.
- `cookimport/staging/import_session.py`: writes label artifacts and builds Stage 7 from authoritative outputs.
- `cookimport/staging/nonrecipe_stage.py`: deterministic non-recipe ownership.
- `cookimport/labelstudio/ingest.py`: prediction-run reuse of authoritative label artifacts.
- `cookimport/bench/cutdown_export.py`, `cookimport/bench/followup_bundle.py`, and `cookimport/cf_debug_cli.py`: reviewer packets and uncertainty exports that still speak in low-confidence terms.

## Plan of Work

### Milestone 1: Define explicit trust and escalation contracts on the authoritative label seam

At the end of this milestone, the label-first runtime will have clear data models for trust and escalation, even if downstream readers still consume compatibility `confidence`. Start by extending the authoritative line and block models in `cookimport/parsing/label_source_of_truth.py` and the underlying line-role prediction model in `cookimport/parsing/canonical_line_roles.py`.

Add these fields to the line-level contract:

    trust_score: float
    escalation_score: float
    escalation_reasons: list[str]
    candidate_labels: list[str]
    rule_hits: list[str]
    parser_support: dict[str, Any]
    neighbor_agreement: dict[str, Any]
    source_structure_support: dict[str, Any]

Do not over-engineer these substructures. The goal is to capture the evidence the current code already has without inventing a brand-new classifier. A pragmatic first implementation is:

- `candidate_labels`: unique ordered labels seen across deterministic label, Codex candidate label if any, and final accepted label.
- `rule_hits`: deterministic rule tags and any normalized syntax-ownership rule markers.
- `parser_support`: small booleans or scores for line shapes the current code already recognizes, such as strict yield header, primary time line, note prefix, ingredient-like syntax, or subsection-heading context.
- `neighbor_agreement`: booleans or counts that summarize whether nearby lines support recipe-local structure.
- `source_structure_support`: booleans derived from block or location hints already carried into the atomized candidates.

Then add helper functions in `cookimport/parsing/canonical_line_roles.py` that compute:

- `trust_score`: derived from the accepted final decision. Rule-decided labels with strong syntax evidence should tend high. Fallback recoveries should be lower. Accepted Codex overrides can be medium-to-high depending on whether ownership arbitration accepted them cleanly. Sanitized or vetoed labels should have lower trust than direct strong rule hits.
- `escalation_score`: derived from the need for expensive review. Deterministic unresolved cases, low-trust structured labels, syntax-versus-Codex disagreement, fallback decisions, and structural oddities should push it upward. High-trust obvious syntax hits should keep it low.
- `escalation_reasons`: explicit machine-readable reasons such as `deterministic_unresolved`, `low_trust_structured_label`, `fallback_decision`, `codex_disagreed_with_rule`, `ownership_veto`, `outside_span_structured_label`, `recipe_span_warning`, and `missing_required_recipe_fields` where applicable.

In this milestone, keep the old `confidence` field alive but redefine it as a compatibility alias of `trust_score` whenever new label-first artifacts are written. Do not leave two independent scalar notions on the same row.

Acceptance for this milestone is that one authoritative labeled-line row can be inspected in memory or in a fixture and shows `trust_score`, `escalation_score`, `escalation_reasons`, `decided_by`, and compatibility `confidence`, with `confidence == trust_score`.

### Milestone 2: Replace live line-role escalation with explicit escalation metadata

At the end of this milestone, the live Codex escalation path in `cookimport/parsing/canonical_line_roles.py` will no longer read the mixed scalar confidence threshold directly. Replace `_should_escalate_low_confidence_candidate(...)` with a new helper such as:

    def _should_escalate_candidate(
        *,
        candidate: AtomicLineCandidate,
        deterministic_label: str | None,
        trust_score: float,
        escalation_score: float,
        escalation_reasons: Sequence[str],
        by_atomic_index: dict[int, AtomicLineCandidate],
    ) -> bool:

This helper must still preserve the existing core behavior that the repo relies on:

- obvious recipe titles are not escalated merely because their score is low;
- outside-recipe escalation remains disabled by default unless the existing environment gate is explicitly on;
- deterministic unresolved and structurally suspicious recipe-local lines should still be eligible for Codex review.

Add explicit run settings for the new control surfaces in `cookimport/config/run_settings.py`. Keep them internal-only for now:

    line_role_trust_low_threshold
    line_role_escalation_threshold

Do not proliferate knobs. The plan should keep the old `_CODEX_LOW_CONFIDENCE_THRESHOLD` logic as a derived default during migration, then move callers to the new setting names. The acceptance rule is not “the threshold exists”; it is “a future contributor can see exactly why a candidate escalated.” That means the plan-mode or debug artifact path must also record the reasons that caused escalation.

Update tests in `tests/parsing/test_canonical_line_roles.py` to cover:

- strong deterministic title stays non-escalated even if trust is modest;
- fallback or unresolved structured recipe lines escalate with explicit reasons;
- outside-span low-trust candidates still do not escalate by default;
- Codex disagreement plus ownership veto produces high escalation and explicit reasons without silently mutating trust semantics.

Acceptance for this milestone is that the line-role runtime escalates based on explicit escalation metadata and tests prove that score meaning is now separated from control meaning.

### Milestone 3: Roll trust and escalation up to block and recipe-span diagnostics without changing ownership rules

At the end of this milestone, recipe spans and normalized block labels will expose trust and escalation summaries, but grouping will still be based on final labels only. Extend `cookimport/parsing/label_source_of_truth.py` so block labels roll up:

- `trust_score`: use the representative or conservative aggregate for the chosen final block label. A pragmatic default is the minimum trust among supporting atomic rows or a weighted aggregate that penalizes disagreement.
- `escalation_score`: use the maximum escalation among contributing rows.
- `escalation_reasons`: union of the line-level reasons that actually contributed to the block-level decision.

Then update `cookimport/parsing/recipe_span_grouping.py` so `RecipeSpan` grows explicit summary metadata. Add fields such as:

    trust_score: float | None
    escalation_score: float | None
    escalation_reasons: list[str]
    decision_notes: list[str]

Compute these after the deterministic span boundaries are formed. Keep the grouping algorithm purely label-driven. This milestone must not introduce score thresholds into grouping. Instead, use the new metadata to explain odd spans:

- spans that start without a title should have elevated escalation and a clear reason;
- spans with low-trust title blocks should have elevated escalation;
- spans with mixed structured labels but weak local evidence should have explicit decision notes.

Add a small companion artifact under `group_recipe_spans/<workbook_slug>/`, for example `span_decisions.json`, that lists the per-span rollup and the concrete reasons. This file should be the reviewer/debug artifact for recipe-level uncertainty rather than forcing a reviewer to reverse-engineer span risk from line rows.

Update the temporary recipe bridge in `cookimport/parsing/label_source_of_truth.py::_build_recipe_candidate_from_span(...)` so `RecipeCandidate.confidence` and `provenance.confidence_score` become compatibility aliases for span trust, not arithmetic means of the old mixed line confidence. Preserve the old fields until downstream readers no longer need them, but stop teaching the code that recipe-level confidence comes from “average line confidence.”

Acceptance for this milestone is that a stage run writes span-level decision metadata, yet `group_recipe_spans` still behaves identically for the same final labels regardless of trust values.

### Milestone 4: Update stage and prediction-run artifact writers to publish the new metadata as first-class outputs

At the end of this milestone, all stage-backed label artifacts and authoritative prediction-run line-role artifacts will expose trust and escalation fields directly. Update `cookimport/staging/import_session.py::_write_label_first_artifacts(...)` so the following files carry explicit metadata:

- `label_det/<workbook_slug>/labeled_lines.jsonl`
- `label_det/<workbook_slug>/block_labels.json`
- `label_llm_correct/<workbook_slug>/labeled_lines.jsonl`
- `label_llm_correct/<workbook_slug>/block_labels.json`
- `group_recipe_spans/<workbook_slug>/authoritative_block_labels.json`
- the new span decision artifact from Milestone 3

Update `cookimport/labelstudio/ingest.py::_write_authoritative_line_role_artifacts(...)` so `line_role_predictions.jsonl` includes the new fields when authoritative labels are reused. The prediction-run path should not recompute them independently; it should reuse the stage-produced values or the same in-memory authoritative bundle.

Add stage observability notes or artifact registration only where it clarifies the new contract. Do not create a new stage just to hold metadata. The semantic stages remain `label_det`, `label_llm_correct`, and `group_recipe_spans`; the difference is that those stages now publish their decision contract honestly.

Update short docs in:

- `docs/04-parsing/04-parsing_readme.md`
- `docs/05-staging/05-staging_readme.md`
- `docs/06-label-studio/06-label-studio_README.md`

Each doc update should explain the same truth in the context of that subsystem: trust and escalation are separate metadata, grouping and Stage 7 ownership are label-driven, and compatibility `confidence` remains only as a derived legacy alias where still emitted.

Acceptance for this milestone is that a normal stage run and an offline prediction run both expose the same trust and escalation metadata on authoritative label artifacts.

### Milestone 5: Convert benchmark and reviewer tooling from “low confidence” to “low trust plus escalation reason”

At the end of this milestone, reviewer-facing exports stop teaching the old mental model. Update these surfaces:

- `cookimport/bench/followup_bundle.py`
- `cookimport/cf_debug_cli.py`
- `cookimport/bench/cutdown_export.py`

The new reviewer contract should be:

- “uncertainty” means low trust and/or explicit escalation reasons, not only numeric thresholding;
- reviewer packets should carry `trust_score`, `escalation_score`, `escalation_reasons`, and `decided_by`;
- packet naming and index entries should prefer `low_trust_changed_lines` or `escalated_changed_lines` over `low_confidence_changed_lines`;
- CLI options should migrate from `--confidence-threshold` to `--trust-threshold`, keeping the old flag only as a deprecated alias during migration.

Do not delete useful reviewer slices. Rename and recenter them. A packet that shows “all changed lines that were escalated” is still valuable; it just should not pretend that confidence alone is the reason.

Update the relevant tests under `tests/bench/` and any `tests/labelstudio/` fixtures that assert old field names. If a legacy compatibility view is retained for one release of the repo, make it explicit and write the new field names alongside the legacy ones instead of silently translating them.

Acceptance for this milestone is that a reviewer bundle or follow-up export can explain uncertainty using trust and escalation metadata alone, and no new artifact teaches that scalar confidence is the primary source of truth.

### Milestone 6: Validate calibration scaffolding without blocking the migration on a full calibration project

At the end of this milestone, the repo will not yet have a polished calibration dashboard, but it will have enough scaffolding to prove the new model is usable. Add a narrow reporting helper under a benchmark or analytics-owned module that bins authoritative labeled lines by `trust_score` and emits per-label counts plus correctness summaries when gold labels are available. Keep it simple:

- per-label trust bands such as `0.0-0.2`, `0.2-0.4`, `0.4-0.6`, `0.6-0.8`, `0.8-1.0`;
- counts of correct versus incorrect predictions in each band;
- counts by `decided_by` and top `escalation_reasons`.

This is not the milestone for designing a full product dashboard. It is the milestone for proving that the new trust field is at least measurable and that the repo can later calibrate it honestly. Add a short doc note in the nearest benchmark or analytics readme explaining where this report lives and what it means.

Acceptance for this milestone is that a benchmark or eval run can produce one calibration-oriented summary derived from `trust_score`, which is enough to show the field is not just decorative.

## Validation and Acceptance

Acceptance is behavioral and observable.

The line-role contract is correct when an authoritative labeled-line row exposes separate `trust_score` and `escalation_score`, when `confidence` is only a derived legacy alias of trust on that seam, and when the code can explain why a candidate escalated through explicit `escalation_reasons`.

The runtime contract is correct when recipe grouping and Stage 7 non-recipe ownership still depend on `final_label` and structural validation only. To prove this, tests must show that changing trust or escalation values without changing final labels does not change grouped recipe spans or Stage 7 knowledge eligibility.

The reviewer contract is correct when benchmark or follow-up exports can explain uncertainty using low trust and escalation reasons instead of treating scalar confidence as the one true explanation.

The migration is complete when:

- `cookimport/parsing/canonical_line_roles.py` no longer uses a function named or conceptualized as low-confidence escalation in the main path;
- authoritative label artifacts written by `cookimport/staging/import_session.py` and `cookimport/labelstudio/ingest.py` publish trust and escalation fields directly;
- `cookimport/parsing/recipe_span_grouping.py` and `cookimport/staging/nonrecipe_stage.py` still do not use scalar score thresholds as authority;
- benchmark/follow-up tooling stops teaching “low confidence” as the primary concept for new artifacts;
- one real stage run can be inspected manually and shows separate trust and escalation metadata.

## Idempotence and Recovery

This migration should be done additively first and subtractively second. Add explicit trust and escalation metadata while keeping compatibility `confidence` outputs alive. Move readers one by one to the new fields. Only after the focused tests and a real smoke run pass should you remove or rename reviewer artifacts that still use old terminology.

The work is safe to retry because all commands above either run tests or write to fresh timestamped output roots. If a partial implementation leaves readers split between old and new fields, keep compatibility writes in place and continue migrating readers. Do not leave the repo in a state where the runtime writes only new fields but reviewer tooling still assumes old ones with no fallback.

If a real Codex plan-mode run does not emit the expected metadata, inspect the stage root directly before changing more code. The likely failure modes are incomplete writer propagation or a prediction-run reuse helper that rebuilt rows from stale field names.
