## Operator Intent (Structured) - DO NOT DELETE

The operator wants one simple worker model for both `line_role` and nonrecipe classification: the worker should see an almost empty workspace with a readable `task.json`, read the full assignment once, make all semantic decisions in one pass, run one repo-owned handoff command, and then fix any repair feedback in the same session while the full context is still fresh. Any helper, file, prompt text, or workflow concept that does not directly support that loop should be removed rather than preserved. This refactor is explicitly in burn-the-boats mode: do not carry forward queue workflows, `answers.json`, compatibility shims, dual-mode behavior, or legacy surfaces just to ease migration. The goal is not to preserve the old design. The goal is to replace it cleanly with the new one.

## Purpose / Big Picture

After this change, the line-role workers and nonrecipe-finalize workers will use the same simple working style: open one readable `task.json`, read the assignment directly, fill every `answer` in place, run one repo-owned handoff command, fix any repair feedback in the same session, and stop when the helper reports completion. The worker workspace will stop nudging agents toward `task-summary`, queue stepping, `answers.json`, or shell-generated bulk rewrite scripts. The visible workspace should effectively feel like two files: `AGENTS.md` and a readable `task.json`.

This matters because the current worker contract fights the cognition we want. The agents are being told that `task.json` is the whole job, while the prompt and helper surface simultaneously steer them away from directly reading it. That contradiction is burning wrapper overhead, encouraging homemade tooling, and causing avoidable watchdog kills. After this refactor, the paved road will match the job: read once, label in one pass, validate, repair if needed, done.

The user-visible proof is not a live Codex benchmark run, because this repository forbids Codex-backed book processing without explicit approval. The proof is deterministic and local: task files become readable, worker prompts stop advertising the bad paths, the queue-specific knowledge classification helpers disappear, local `task.json` reads are no longer treated as drift, and the targeted `llm` plus `parsing` test suites pass.

## Outcomes & Retrospective

This plan captures the intended end state and the concrete repository seams to edit, but no implementation work has started yet. The most important lesson from the investigation is that the current failure mode is not “too many files” and not “too much agent prose.” The real problems are contradictory prompts, a minified task file, too many alternate helper paths, and stage-local enforcement that punishes the model for trying to batch a naturally batch-shaped job.

## Context and Orientation

The current worker system for these stages is spread across a few shared modules and a few stage-local modules.

`cookimport/llm/editable_task_file.py` builds and validates the one-file worker contract. Today it always adds `editable_json_pointers`, and its validator treats every non-`answer` field as immutable.

`cookimport/llm/task_file_guardrails.py` renders `task.json`. Right now `render_task_file_text(...)` uses `json.dumps(..., sort_keys=True, separators=(",", ":"))`, which creates a compact one-line file. That is a major reason the direct-file path is ergonomically bad.

`cookimport/llm/single_file_worker_commands.py` defines the helper command surface. It currently exposes a generic batch-edit toolset for most stages and a queue-style toolset for `nonrecipe_classify`. The helpers themselves are not the main cost; the problem is that the surface teaches the wrong workflow and invites repeated command traffic.

`cookimport/parsing/canonical_line_roles/planning.py` builds the line-role workspace worker prompt. It currently leads with `task-summary`, suggests `task-show-unit` and `task-show-unanswered`, advertises `task-template answers.json`, and tells the worker not to dump `task.json` directly.

`cookimport/llm/knowledge_stage/task_file_contracts.py` builds and transitions knowledge-stage task files. It currently builds classification as a queue-style task and grouping as a batch-style task. The classification task already stores all units in one file, but the prompt and helper commands force the worker through them one by one.

`cookimport/llm/knowledge_stage/recovery.py` builds the nonrecipe workspace prompt and applies stage-specific workspace command policy. It currently contains the queue-first instructions and explicit kill reasons for batch synthesis during classification.

`cookimport/llm/knowledge_same_session_handoff.py` is the repo-owned same-session handoff entrypoint for nonrecipe work. It validates the edited file, rewrites repair tasks, advances classification into grouping, and writes final outputs. This is the helper seam that should remain.

`cookimport/llm/codex_exec_runner.py` contains the shared workspace command classification and boundary detection logic. It also contains `_looks_like_editable_task_file_payload(...)`, which will need to tolerate the slimmer task-file contract.

The existing tests already cover most of these surfaces. Relevant suites live in `tests/llm/` and `tests/parsing/`, especially `tests/llm/test_editable_task_file.py`, `tests/llm/test_single_file_worker_commands.py`, `tests/llm/test_knowledge_same_session_handoff.py`, `tests/llm/test_codex_exec_runner_workspace.py`, `tests/parsing/test_line_role_workspace_tools.py`, `tests/parsing/test_line_role_same_session_handoff.py`, and `tests/parsing/test_canonical_line_roles_workspace.py`.

In this plan, “direct-batch contract” means a worker contract where the entire assignment is visible in one readable file, the worker edits answers in place across the whole assignment, and deterministic repo code performs validation and repair without queue stepping or scratch answer files.

## Milestones

### Milestone 1: Establish the shared direct-batch contract

At the end of this milestone, the repository will have one explicit definition of the worker behavior we want for these stages. The task file will be readable, slimmed down, and validated without relying on giant pointer lists. The shared helper surface will keep only the repo-owned commands that still add real value after direct file reading is restored.

The proof for this milestone is local and deterministic: build or snapshot a task file for one line-role shard and one nonrecipe classification shard, inspect the written `task.json`, and confirm that it is multiline, human-readable, and free of `editable_json_pointers`, `workflow`, `next_action`, and `answers.json` guidance. Run the focused `editable_task_file`, `single_file_worker_commands`, and workspace-runner tests and confirm they pass.

### Milestone 2: Migrate line_role to the direct-batch contract

At the end of this milestone, line-role workers will be told to open `task.json` directly, read the full shard, edit all `/units/*/answer` payloads, run `task-handoff`, repair in place if needed, and stop at completion. The line-role prompt will no longer lead with summary helpers or advertise `answers.json`. The line-role runtime will continue to validate and expand authoritative outputs exactly as before.

The proof for this milestone is a deterministic worker-fixture test that writes a line-role workspace, inspects the generated `AGENTS.md` and `task.json`, and verifies the new contract. Existing line-role runtime and same-session handoff tests must still pass.

### Milestone 3: Migrate nonrecipe classification to the same contract

At the end of this milestone, nonrecipe classification will stop being queue-style. The classification task file will expose all units at once, the prompt will match the direct-batch contract, the worker will fill all answer objects in one pass, and `task-handoff` will validate or rewrite repair mode. If knowledge survives, the same handoff path will still advance into a grouping task, but classification itself will no longer depend on `task-show-current`, `task-answer-current`, or `task-next`.

The proof for this milestone is a deterministic classification handoff test that edits a full task file in place, confirms repair-mode rewrites still work, and confirms valid classification can still advance into grouping.

### Milestone 4: Make grouping use the same editing model and remove kill-style workflow enforcement

At the end of this milestone, the follow-on grouping task will also use the direct-batch contract, and the stage-local kill logic for queue or batch synthesis will be gone. Real workspace-boundary enforcement remains, but local `cat`, `sed`, `rg`, and similar reads against `task.json` inside the workspace are no longer treated as a reason to terminate the session. Any remaining workflow telemetry can stay as warnings, not kills.

The proof for this milestone is that no targeted tests reference the old knowledge classification watchdog reason codes, and the workspace-policy tests continue to fail only on actual boundary escapes.

### Milestone 5: Update docs and leave the system easier to reason about

At the end of this milestone, the short package READMEs and the main LLM doc will describe the new happy path accurately. A future contributor should be able to read the docs and understand that line-role and nonrecipe workers are file-first, direct-read, warning-first systems.

The proof for this milestone is updated docs under `docs/10-llm/10-llm_README.md` plus short notes in the stage packages that match the code and tests.

## Plan of Work

Start in the shared task-file layer. In `cookimport/llm/task_file_guardrails.py`, replace the compact one-line renderer with a readable renderer for these worker task files. Use stable key ordering that favors comprehension over alphabetical sorting. At the top level, write the metadata in semantic order, then `units`. Within each unit, order keys as `unit_id`, `owned_id`, `evidence`, `answer`, then repair-only fields such as `previous_answer` and `validation_feedback`. The goal is to let a worker read the file straight through without wrapper commands.

In `cookimport/llm/editable_task_file.py`, remove the synthetic `editable_json_pointers` list from the task-file contract for the targeted stages. Replace its role with the already-present `answer_schema.editable_pointer_pattern`, which is enough to describe the editable surface. Update `summarize_task_file(...)`, `build_worker_task_brief(...)`, and `validate_edited_task_file(...)` so they no longer depend on explicit pointer enumeration. Validation should infer that only `units[*].answer` is mutable and should keep treating all other unit data and top-level metadata as immutable. Preserve repair-mode support.

In the same file, slim the task payload shape. For the targeted stages, `task.json` should keep only metadata that either affects semantic decisions or deterministic validation: `schema_version`, `stage_key`, `assignment_id`, `worker_id`, `mode`, `answer_schema`, `review_contract` when present, any stage-specific metadata that the worker genuinely needs, and `units`. Remove `helper_commands`, `workflow`, and `next_action` from the happy-path file content. Do not add compatibility adapters for the removed fields. Update all callers in the same refactor so the new contract is the only contract.

In `cookimport/llm/single_file_worker_commands.py`, reduce the exposed helper surface for line-role and nonrecipe tasks. Keep `task-handoff`, `task-status`, and `task-doctor` as the repo-owned control seam. Remove or stop exposing `task-template`, `task-apply`, `task-show-unanswered`, queue commands, and any content helpers that duplicate direct file reading. The worker should no longer need a command to see the data it already has in `task.json`. If a narrow read helper survives for repair-only debugging, it must be explicitly fallback-only and omitted from the default prompt text.

In `cookimport/parsing/canonical_line_roles/planning.py`, rewrite the line-role workspace prompt to match the desired mental model. The prompt should say, in plain language, that the worker should open `task.json` directly, read the full shard in order, fill every `answer.label`, run `task-handoff`, reopen `task.json` only if repair feedback appears, and stop when handoff reports completion. Remove all language that leads with `task-summary`, advertises `task-template answers.json`, or forbids direct local reads of `task.json`.

In the line-role runtime and tests, keep the deterministic authority boundary intact. The worker still edits only answers; repo code still validates coverage and writes shard outputs. The refactor is about worker ergonomics and overhead, not about moving authority into deterministic code.

In `cookimport/llm/knowledge_stage/task_file_contracts.py`, rebuild classification task files to use the same direct-batch contract as line-role. Keep the classification answer schema exactly as strict as today: `category`, `reviewer_category`, `retrieval_concept`, and `grounding` remain required and deterministically validated. What changes is the editing model, not the semantic contract. The classification task should be “all units visible, answer all units, hand off once,” not “show current, answer current, next.”

In the same module, make grouping tasks use the same direct-batch contract rather than the old helper-first batch contract. Grouping remains a separate semantic phase that the handoff helper can rewrite into after successful classification, but the grouping task file should be read and edited the same way as line-role: direct read, edit answers in place, hand off, repair if needed.

In `cookimport/llm/knowledge_stage/recovery.py`, rewrite the workspace prompt so classification and grouping both describe the direct-batch contract. Remove the queue-first instructions and the language that forbids normal local reads of `task.json`. Keep `task-status`, `task-doctor`, and `task-handoff` as the only repo-owned workflow helpers advertised in the prompt.

Also in `cookimport/llm/knowledge_stage/recovery.py`, delete `_detect_knowledge_workspace_stage_violation(...)` enforcement for batch classification behavior and any other stage-local kill logic that exists only to force queue usage. If you want to preserve operator telemetry, downgrade it to warnings that never terminate the worker. The only hard kills that should remain are true workspace-boundary or forbidden-command escapes handled by the shared boundary detector.

In `cookimport/llm/codex_exec_runner.py`, update `_looks_like_editable_task_file_payload(...)` so it recognizes the slimmer task-file contract without `editable_json_pointers`. While there, ensure the shared command-classification path does not treat ordinary local reads of `task.json` or `AGENTS.md` as drift severe enough to matter for these stages. A coding agent reading a local file is not an error.

Do not broaden the scope into benchmark harness changes or quality retuning. This refactor is only about the worker contract, workspace surface, validation/repair flow, and the stage-local prompts and enforcement surrounding those things.

## Validation and Acceptance

Acceptance is behavioral and deterministic.

For the task-file layer:

1. A freshly written line-role or nonrecipe task file is multiline, indented, and easy to read in a normal text viewer.
2. The task file no longer contains `editable_json_pointers`, `workflow`, `next_action`, or `helper_commands` as worker-facing scaffolding.
3. Validation still rejects edits outside `units[*].answer` and still supports repair-mode rewrites.

For the workspace surface:

1. The mirrored workspace for the targeted stages contains only `AGENTS.md`, `task.json`, `.codex`, and repo-owned hidden control state as needed. `answers.json` is not created on the normal path.
2. `AGENTS.md` tells the worker to read `task.json` directly and does not lead with `task-summary`, queue stepping, or `answers.json`.
3. The only required repo-owned helper on the happy path is `task-handoff`; `task-status` and `task-doctor` remain optional troubleshooting tools.

For line-role:

1. Existing same-session repair behavior still works.
2. Deterministic validation still expands accepted answers into the final line-role shard outputs.
3. The line-role workspace and runtime tests pass.

For nonrecipe finalize:

1. Classification no longer depends on `task-show-current`, `task-answer-current`, or `task-next`.
2. Valid classification still transitions into a grouping task when knowledge rows survive.
3. Grouping uses the same direct read/edit/handoff pattern.
4. Repair-mode rewrites still work for both classification and grouping.
5. No test or runtime path emits `watchdog_packet_contract_bypass_bulk_classification` for ordinary local bulk editing behavior, because that policy no longer exists.

For enforcement:

1. Local reads such as `cat task.json`, `sed -n '1,120p' task.json`, or `rg` inside the workspace are not treated as kill-worthy drift.
2. Real boundary violations such as parent-directory inspection, repo access, or network/package-manager commands remain blocked.

## Idempotence and Recovery

This refactor is safe to implement incrementally if done in the order described above, but the repository should never end a work session in a dual-contract state. Start by changing the shared validator and payload detector, then update the stage writers and prompts in the same implementation pass so the new contract becomes the only contract. Delete queue-only and `answers.json`-specific code as part of the migration, not as a later cleanup task.

If recipe or another stage currently depends on shared helper metadata, update that code in the same refactor so the shared layer has one clean contract. Do not add a temporary compatibility shim.

If a same-session handoff or repair test breaks during migration, do not paper over it by relaxing deterministic validation. The authority boundary must remain: deterministic code validates structure and writes outputs; the LLM makes the semantic choices inside the allowed answer payload.

## Interfaces and Dependencies

The final implementation should preserve these stable responsibilities.

In `cookimport/llm/editable_task_file.py`, `build_task_file(...)` must still produce the canonical worker-edited payload, and `validate_edited_task_file(...)` must still return:

    tuple[dict[str, dict[str, Any]] | None, tuple[str, ...], dict[str, Any]]

The difference is that validation infers the mutable answer path from the shared task-file contract instead of relying on a giant explicit pointer list.

In `cookimport/llm/single_file_worker_commands.py`, `build_single_file_worker_surface(...)` must still return a `SingleFileWorkerSurface`, but for `line_role`, `nonrecipe_classify`, and `knowledge_group` its `helper_commands` and `workflow` should represent the slim direct-batch contract rather than content-dump or queue workflows.

In `cookimport/llm/knowledge_stage/task_file_contracts.py`, `build_knowledge_classification_task_file(...)` and `build_knowledge_grouping_task_files(...)` must continue to produce deterministic task files, and `transition_knowledge_classification_task_file(...)` plus `transition_knowledge_grouping_task_file(...)` must continue to drive repair and stage advancement. The transition logic should not depend on queue-helper state once this refactor lands.

In `cookimport/llm/knowledge_same_session_handoff.py`, the same-session helper remains the authoritative repo-owned transition seam. That module should not be simplified away.

In `cookimport/llm/codex_exec_runner.py`, the shared workspace boundary detector remains the kill/no-kill enforcement seam, but it should only enforce true workspace boundaries for these stages, not workflow preferences that belong in prompt design.
