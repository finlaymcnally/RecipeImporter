# Add Task-Level Parallelism And Crash-Safe Resume To SpeedSuite

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

PLANS.md is checked into the repo root as `docs/PLANS.md`. This ExecPlan must be maintained in accordance with `docs/PLANS.md` and kept fully self-contained.

## Purpose / Big Picture

`cookimport bench speed-run` currently executes every scenario/target/phase sample in a serial loop and has no official resume contract after interruption. Long runs are slow, and an interrupt/crash means rerunning from scratch.

After this change, SpeedSuite will support bounded task-level parallel execution and explicit resume from an existing run directory. Users will be able to stop a run and continue from saved checkpoints instead of losing completed samples.

## Surprises & Discoveries

- Observation: SpeedSuite currently builds deterministic task lists then executes samples in a strict nested loop (`scenario_work_items` then warmup/repeat), with no executor path.
  Evidence: `cookimport/bench/speed_runner.py` loop structure at `run_speed_suite(...)`.
- Observation: QualitySuite already has robust checkpoint+resume patterns that can be adapted.
  Evidence: `cookimport/bench/quality_runner.py` includes `checkpoint.json`, partial artifacts, and `--resume-run-dir` CLI support.
- Observation: Per-sample folder structure already uniquely identified scenario/target/phase, so resume snapshots fit cleanly as `speed_sample_result.json` without introducing a second task-indexed storage tree.
  Evidence: existing speed sample output path shape under `scenario_runs/<target>/<scenario>/<phase_index>/...` and successful resume test reuse.
- Observation: A conservative default auto parallel cap avoids over-dispatch in mixed I/O+CPU scenarios and keeps behavior predictable across hosts.
  Evidence: `_resolve_speed_parallel_tasks_cap(...)` resolves auto as `min(total_tasks, cpu_count, 4)` and tests still prove deterministic ordering and summary aggregation.

## Decision Log

- Decision: Reuse QualitySuite-style checkpoint/snapshot concepts for SpeedSuite rather than inventing a separate recovery pattern.
  Rationale: Keeps operational behavior familiar and minimizes new maintenance surface.
  Date/Author: 2026-02-28 / ChatGPT
- Decision: Use a task-level `ThreadPoolExecutor` for SpeedSuite fanout rather than process-based orchestration.
  Rationale: SpeedSuite tasks already invoke subprocess-heavy/runtime-managed work; thread-level orchestration is sufficient for bounded dispatch and simpler to recover/flush checkpoints per completed task.
  Date/Author: 2026-02-28 / ChatGPT
- Decision: Validate resume compatibility against a normalized run-config payload (`suite`, targets, scenarios, warmups/repeats, run-settings hash, Codex confirmation flags) before reusing snapshots.
  Rationale: Prevents accidental resume into an incompatible configuration and avoids silently mixing samples from different settings.
  Date/Author: 2026-02-28 / ChatGPT

## Outcomes & Retrospective

SpeedSuite now supports both requested outcomes: bounded task-level parallel fanout and crash-safe resume from an existing run directory.

Delivered behavior:
- `bench speed-run --max-parallel-tasks N` now controls task-level fanout; omitting it enables bounded auto mode.
- `bench speed-run --resume-run-dir <run_dir>` now reuses completed per-task snapshots and executes only pending tasks.

## Plan of Work

Implement the feature in two layers.

Runner layer: introduce deterministic task objects, optional task-level thread dispatch, per-task result snapshots, and run-level checkpoint/partial artifacts updated after each completion. Resume mode will read existing snapshots/checkpoint and skip completed tasks.

CLI layer: expose `--max-parallel-tasks` and `--resume-run-dir` for `bench speed-run`, validate inputs, and pass through to the runner.

Testing layer: add focused tests to prove parallel dispatch executes concurrently when requested, resume reuses completed snapshots without rerunning completed tasks, and CLI wiring/validation forwards new options correctly.

Docs layer: update bench docs/runbook to document new options and recovery flow.
