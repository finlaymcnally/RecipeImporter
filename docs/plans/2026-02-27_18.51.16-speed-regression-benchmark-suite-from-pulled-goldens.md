# Build A Deterministic Speed Regression Suite From `pulled-from-labelstudio`

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This repository includes `docs/PLANS.md` at the repository root for ExecPlan rules. This plan must be maintained in accordance with that file.

## Purpose / Big Picture

After this change, an agent should be able to run one deterministic speed suite before and after a code change and get a clear answer to: did import and benchmark runtime improve, stay flat, or regress? The suite will source targets from `data/golden/pulled-from-labelstudio`, map each gold export to its matching file in `data/input`, run timed scenarios, and emit machine-readable comparison artifacts plus a concise markdown verdict.

This plan is intentionally speed-only. It designs output schemas so accuracy checks can be added later without replacing the speed harness.

## Surprises & Discoveries

- Observation: `cookimport bench run` is currently quality-focused and aggregates stage-block metrics; it is not a baseline/candidate speed regression harness.
  Evidence: `cookimport/bench/runner.py` + `cookimport/bench/report.py` aggregate recall/precision/F1 and do not compare runs for speed verdicts.

- Observation: Run-local timing data is already rich enough to avoid introducing a new telemetry source.
  Evidence: benchmark `run_manifest.json` and `eval_report.json` include `timing` plus `evaluation_telemetry`; stage run reports and manifests include timing fields and processing telemetry artifacts.

- Observation: Matching pulled gold exports to input files must use fallback hints, not manifest-only source fields.
  Evidence: several directories under `data/golden/pulled-from-labelstudio` have empty `run_manifest.json` source payloads, while `exports/freeform_span_labels.jsonl` still includes `source_file`.

- Observation: Existing all-method target matching logic is private to CLI internals and should be shared, otherwise drift risk is high.
  Evidence: `cookimport/cli.py:_resolve_all_method_targets(...)` already contains practical fallback behavior and unmatched diagnostics.

- Observation: Stage import report timing can be absent in some existing artifacts, so speed samples need a wall-clock fallback instead of assuming report `timing` always exists.
  Evidence: sampled `*.excel_import_report.json` artifacts under `data/output` with no `timing` payload while benchmark manifests still contained usable timing fields.

## Decision Log

- Decision: The suite will default to `data/golden/pulled-from-labelstudio` as the gold source of truth.
  Rationale: User requirement is explicit, and this folder is stable and annotation-backed.
  Date/Author: 2026-02-27 / Codex GPT-5

- Decision: The suite will benchmark two scenario families by default: import runtime and benchmark runtime.
  Rationale: The stated goal is to validate both “straight import speed” and “benchmarking speed.”
  Date/Author: 2026-02-27 / Codex GPT-5

- Decision: Comparison will use repeat samples and median-based scoring with configurable regression thresholds.
  Rationale: Median over repeats is simple, robust to local jitter, and appropriate for this solo setup.
  Date/Author: 2026-02-27 / Codex GPT-5

- Decision: The suite will be self-contained under dedicated speed-run artifact roots, not dependent on global dashboard/history rows.
  Rationale: Shared history is useful but can be noisy/incomplete; speed gating should not depend on unrelated runs.
  Date/Author: 2026-02-27 / Codex GPT-5

- Decision: LLM recipe correction remains explicitly off for these runs.
  Rationale: Project policy requires codex-farm recipe parsing to stay off, and speed suite must remain deterministic.
  Date/Author: 2026-02-27 / Codex GPT-5

- Decision: Shared matcher logic was moved into `cookimport/bench/speed_suite.py` and reused by all-method target resolution.
  Rationale: one fallback chain for source hint extraction prevents drift between all-method and speed-suite discovery.
  Date/Author: 2026-02-27 / Codex GPT-5

- Decision: Regression gating requires both percent and absolute-second thresholds.
  Rationale: this avoids false regressions from tiny absolute deltas on short runs while still catching large slowdowns.
  Date/Author: 2026-02-27 / Codex GPT-5

## Outcomes & Retrospective

Implemented.

The repo now has a deterministic speed-regression workflow under `cookimport bench speed-*`:
- `speed-discover` builds target manifests from pulled Label Studio gold exports with matched/unmatched diagnostics.
- `speed-run` executes repeat samples for import and benchmark scenarios, writes raw sample rows (`samples.jsonl`), run summaries (`summary.json`), and a concise report (`report.md`).
- `speed-compare` aligns baseline/candidate median timings by `target_id + scenario`, writes JSON/markdown outputs, and supports fail-on-regression automation.

The workflow is speed-only and keeps codex-farm recipe parsing off. A future follow-up can add quality gating on top of the same baseline/candidate artifact contract.

## Context and Orientation

Today the repository already has several benchmark and timing paths, but they are split by use case:

- `cookimport stage` writes import reports (`*.excel_import_report.json`) with runtime timing and writes `run_manifest.json`.
- `cookimport labelstudio-benchmark --no-upload --eval-mode canonical-text` writes benchmark eval artifacts and timing.
- interactive all-method benchmark writes scheduler-rich per-source/per-run reports but is not exposed as a dedicated non-interactive speed regression command.
- `cookimport bench run` is an offline quality loop with suite manifests, but it is not currently designed to answer “did runtime improve vs baseline?”

Terms used in this plan:

- Target: one `(source_file, gold_spans_path)` pair discovered from pulled exports.
- Scenario: one timed workflow run against a target (for example stage import or canonical benchmark).
- Sample: one repeated execution of a scenario-target pair.
- Baseline run: earlier speed-suite output used as the comparison reference.
- Candidate run: new speed-suite output generated after a code change.

Relevant existing paths and contracts:

- Gold root: `data/golden/pulled-from-labelstudio`
- Input root: `data/input`
- Existing bench defaults: `data/golden/bench/suites` and `data/golden/bench/runs`
- Benchmark timing payloads: `run_manifest.json` and `eval_report.json`
- Stage timing payloads: `*.excel_import_report.json` and stage `run_manifest.json`

## Plan of Work

Milestone 1 introduces a shared discovery layer for speed targets. Implement a new bench module that resolves pulled gold exports into deterministic target rows with explicit unmatched diagnostics. Move current source-hint matching behavior out of private CLI-only flow into a reusable function so all-method logic and the new speed suite share one matching contract.

Milestone 2 introduces speed scenario execution. Build a new speed runner that executes repeatable scenario samples for each target. Default scenarios are import (`stage`) and canonical benchmark (`labelstudio-benchmark --no-upload --eval-mode canonical-text`). Each sample captures wall clock plus normalized timing payloads already emitted by current run artifacts.

Milestone 3 adds baseline-vs-candidate comparison. Build a comparator that loads two speed runs, aligns scenario-target metrics, computes medians, percent deltas, and regression verdicts using configurable thresholds (percent and absolute-seconds floor). Emit machine and markdown reports and support strict exit codes for agent automation.

Milestone 4 wires CLI command surfaces under `cookimport bench`:

- `cookimport bench speed-discover` to generate/reserialize target manifests.
- `cookimport bench speed-run` to execute samples and write summary artifacts.
- `cookimport bench speed-compare` to compare two run directories and optionally fail on regression.

Milestone 5 adds focused tests and docs. Keep tests mostly unit and monkeypatched orchestration (no heavy real-book runs in CI tests). Update benchmark docs/runbook with one clear baseline/candidate workflow for future AI agents.

## Validation and Acceptance

Acceptance is behavior-based:

- `speed-discover` finds targets from `data/golden/pulled-from-labelstudio`, maps them to `data/input`, and reports unmatched cases with clear reasons.
- `speed-run` produces a timestamped run root with:
  - resolved suite manifest copy,
  - per-sample raw timing rows,
  - normalized summary JSON and markdown report.
- Import scenario metrics include at least `total_seconds`, `parsing_seconds`, `writing_seconds` when available.
- Benchmark scenario metrics include at least `prediction_seconds`, `evaluation_seconds`, `total_seconds`, and core checkpoint fields when available.
- `speed-compare` aligns baseline and candidate runs by scenario + target, computes deltas, writes markdown + JSON, and returns non-zero exit only when configured regression gates fail.
- No command in this flow enables recipe codex-farm parsing.

## Idempotence and Recovery

All suite outputs are timestamped and additive. Re-running discovery/run/compare should never mutate source inputs or pulled gold exports. If a run is interrupted, keep the partial timestamp directory for diagnosis and start a new run; no rollback is required. Compare commands are read-only over prior run artifacts.
