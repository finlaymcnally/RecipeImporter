# Prepare Structural Knowledge Anti-Stop Follow-Up

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the knowledge stage should stop depending on prompt wording to keep a worker moving through a long queue. If the current one-time relaunch backstop still leaves workers stopping at neat conversational checkpoints, the next lever must be structural. In this repository, “structural” means the repo, helper contract, and worker-session boundaries make continued work feel like the default path even when the model behaves like a polite chat assistant.

This plan prepares that next step now because the likely failure is already visible in the saved benchmark artifacts: knowledge workers can see the next current task but still stop after a clean subtask boundary. The goal is to make the next implementation path ready before the next rerun proves it is needed. The user-visible proof, once this plan is implemented, will be that a rerun no longer depends on “do not ask permission” wording to drain the queue. Either the repo will keep auto-resuming the remaining assignment deterministically, or the worker contract will shift to a small repo-owned batch that naturally flows across several packets before any “done” moment exists.

## Surprises & Discoveries

- Observation: the previous fix solved the stale handoff seam but not the “natural stopping point” seam.
  Evidence: in the `2026-03-22_12.08.53` benchmark artifacts, workers advanced `current_task.json` correctly to task 2 or task 5, then still emitted assistant-style progress summaries and ended the turn.

- Observation: recipe and line-role behave more like “drain a batch” than knowledge does.
  Evidence: those stages already push workers through broader flows such as `prepare-all` plus `finalize-all`, while knowledge still centers the worker on one current packet, one check, and one install at a time.

- Observation: a one-time relaunch backstop is a good safety net, but it still leaves the system depending on the model to stop misbehaving after one rescue.
  Evidence: the current implementation records and retries one premature clean exit, but a second identical clean stop would still be a live risk if the model keeps treating each validated step as a natural point to summarize.

- Observation: the next meaningful design choice is not “more wording.” It is where the repo draws the real unit of work.
  Evidence: current artifacts already include explicit anti-stop wording. The remaining failure mode is not ignorance of the instruction; it is that the current-task contract still gives the model a neat success boundary every time one packet installs cleanly.

- Observation: the current tree already had the beginnings of stronger auto-resume before this plan was implemented, but the failure still collapsed back into a generic incomplete state once the rescue cap was reached.
  Evidence: the runtime already tracked `workspace_relaunch_history` and `workspace_premature_clean_exit_count`, but there was no distinct final reason code for “we kept rescuing conversational stops until the cap.”

## Outcomes & Retrospective

The first half of this plan is now implemented. Knowledge workers no longer fall back to the vague `workspace_validated_task_queue_incomplete` state when the runtime had to rescue repeated conversational clean exits and then hit the rescue cap. The runtime now preserves durable relaunch history, keeps the small relaunch cap explicit in live status and telemetry, and terminates with `workspace_validated_task_queue_premature_clean_exit_cap_reached` when the repo has already resumed the queue as many times as it is willing to do automatically.

The remaining open question is whether this stronger auto-resume path is enough by itself on live reruns. If it is, the knowledge stage can keep its current task-packet contract. If it is not, this plan already defines the next structural move: make the worker-visible unit of work feel more batch-shaped instead of continuing to polish wording around one-packet checkpoints.

## Context and Orientation

The knowledge stage lives in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, `cookimport/llm/knowledge_workspace_tools.py`, and `cookimport/llm/codex_exec_runner.py`. The orchestrator owns worker assignments, prompt text, task status, validation, repair/retry ladders, and worker-session completion logic. The workspace-tools module writes the worker-visible sidecars and renders `tools/knowledge_worker.py`, the helper CLI inside each worker root. The direct-exec runner mirrors a durable worker root into a sterile execution workspace and synchronizes selected files between them while a long-lived Codex session runs.

In the current design, a knowledge worker assignment owns many task packets but processes them one current task at a time. A task packet is one row in `assigned_tasks.json` plus its repo-written `in/*.json`, `hints/*.md`, `out/*.json`, and current-task sidecars. A “current-task bundle” means `current_task.json`, `CURRENT_TASK.md`, and `CURRENT_TASK_FEEDBACK.md`. The worker uses `tools/knowledge_worker.py complete-current`, edits a draft, runs `check-current`, and then `install-current`. When the repo accepts that packet, it advances the current-task bundle to the next row.

The completed anti-stop plan immediately before this one already added a one-time relaunch backstop for a “premature clean exit,” meaning a worker session ended cleanly after visibly advancing the queue but before the queue was complete. That work lives in `cookimport/llm/codex_farm_knowledge_orchestrator.py` and is described in `docs/plans/2026-03-22_12.38.04-stop-knowledge-workers-from-conversationally-ending-mid-queue.md`. The important fact for this new plan is that the repo can now rescue one such conversational stop, but the broader work unit still feels like “complete this current packet,” not “drain this owned run of work.”

This plan uses two precise phrases:

“Stronger auto-resume” means the repo treats a clean mid-queue exit as an ordinary worker-session boundary rather than a meaningful stopping signal. The repo relaunches the remaining assignment deterministically from the already-advanced current-task state until one of three things happens: the queue completes, a real blocking failure appears, or a strict relaunch cap is reached.

“Micro-batch seam” means the repo changes the worker-visible unit of work from a single current packet to a small repo-owned batch of consecutive packets, such as three to five task rows at a time. The worker still writes task-local outputs and still uses repo-owned validators, but the natural user-facing checkpoint becomes “finish this small batch” rather than “finish this one packet.”

The reason this matters is visible in the benchmark artifacts under `data/output/2026-03-22_12.08.53/.../knowledge/`. Those workers already saw the advanced next task, but they still ended with messages like “If you want, I can continue immediately with task-002.” That means the repo has already fixed state handoff and still needs to fix work-shape.

## Plan of Work

The first milestone is to make the current auto-resume path robust enough that a conversational stop is treated as a routine runtime interruption rather than a special one-off exception. In `cookimport/llm/codex_farm_knowledge_orchestrator.py`, extend the worker-session completion logic so repeated clean exits can be resumed deterministically up to a deliberately small cap, and make the cap, stop reason, and resume history visible in `live_status.json`, task status metadata, and stage telemetry. This work should also correct the current mismatch between “relaunch once” documentation and the actual cap in code. The implementation should preserve the current packet validator seam and should not require any new prompt trickery.

The second milestone is regression coverage for the stronger auto-resume semantics. Add deterministic fake-runner tests in `tests/llm/test_codex_farm_knowledge_orchestrator.py` that prove all of the following: one clean mid-queue stop resumes and completes; repeated clean mid-queue stops stop at the configured cap with a clear final reason; and final telemetry still records that the worker needed repo-owned conversational-stop rescue even if the final reason ends as completed. If any direct-exec snapshot fields or callback payloads need to carry more runtime context to support those tests, update `cookimport/llm/codex_exec_runner.py` and pin that behavior in `tests/llm/test_codex_exec_runner_workspace.py`.

The third milestone is a paper prototype for a knowledge micro-batch seam. This is not yet a full implementation. Draft the exact repo-owned surfaces needed if stronger auto-resume is still not enough. That prototype should answer four concrete questions in the code comments or plan updates before implementation begins. First, where the batch lives: likely `current_batch.json`, `CURRENT_BATCH.md`, and `CURRENT_BATCH_FEEDBACK.md` next to the existing current-task files, or a replacement surface that still keeps repo authority. Second, how the helper works: likely `prepare-batch`, `check-batch`, and `install-batch`, or a slimmer hybrid where `complete-current` still exists but `install-batch` becomes the main “checkpoint.” Third, how finalization works: whether the batch helper installs task outputs one by one after local validation or writes a batch manifest that the repo atomically accepts. Fourth, how validation and resume interact: the repo must still be able to restart from the first unvalidated task in the batch without replaying accepted earlier tasks.

The fourth milestone is the actual decision gate. After the next approved narrow benchmark rerun, decide whether to stop at stronger auto-resume or to implement the micro-batch seam. If the rerun shows that workers now drain their queues materially better and mid-queue clean exits no longer dominate missing outputs, close this plan after the auto-resume hardening and doc updates. If the rerun still shows workers stopping at natural one-packet checkpoints even after resume hardening, proceed into micro-batch implementation.

The fifth milestone, only if needed, is the micro-batch implementation itself. In `cookimport/llm/knowledge_workspace_tools.py`, add repo-written batch sidecars and helper commands that make the worker’s paved road feel like “prepare and finish this small owned batch.” In `cookimport/llm/codex_farm_knowledge_orchestrator.py`, generate batch groups from ordered task rows, write the batch control files into the worker root, validate batch-installed outputs deterministically, and preserve the same task-level proposal/promotion logic after batch acceptance. The worker must still end with accepted `out/<task_id>.json` files for each task, not a brand-new opaque transport.

## Validation and Acceptance

This work is accepted in two stages.

The first acceptance stage is repo-owned auto-resume hardening. Acceptance means a fake-runner regression can force repeated clean mid-queue exits and the runtime behaves predictably: it resumes the remaining assignment, preserves a visible rescue count/history, and stops only at queue completion, a real blocking failure, or the configured cap. A novice should be able to run the narrow tests and see that conversational stop text is no longer enough to end the assignment on its own.

The second acceptance stage is conditional. If the next approved live rerun still shows knowledge workers stopping at one-packet conversational checkpoints, the micro-batch prototype becomes mandatory. Acceptance for the prototype means the plan names the exact repo-written batch files, helper commands, validation path, and resume semantics well enough that another stateless agent could implement them without guessing.

If the micro-batch implementation is needed, final acceptance means the knowledge worker root now feels structurally batch-shaped. A worker should be able to validate several consecutive task packets under one repo-owned batch checkpoint, and a clean conversational stop inside that batch should resume from the first unaccepted packet rather than drop the remaining work.

For any live benchmark confirmation, acceptance is behavioral, not textual. Workers must materially increase sequential validated-task counts per session, and clean mid-queue exits must stop dominating the knowledge stage’s missing-output population.

## Idempotence and Recovery

The stronger auto-resume path must remain idempotent. Relaunching from an already-advanced current-task state must not reprocess accepted earlier tasks unless the existing repo contract already requires revalidation. Resume history fields must tolerate repeated writes without duplicating or corrupting prior state.

If the relaunch cap is reached, the final worker state must remain explicit and debuggable. The repo should write a clear terminal reason such as “premature clean exit relaunch cap reached” instead of falling back to a generic incomplete reason that hides why the queue stopped draining.

If a micro-batch seam is introduced later, it must also be restart-safe. A partially completed batch must be restartable from the first repo-unaccepted task within that batch. The worker helper must not require a human to remember which task rows inside the batch were already accepted.

As with the surrounding benchmark work, live reruns remain optional and must not happen without explicit user approval.
