# Reduce recipe and line-role helper command churn without losing the paved road

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the recipe and canonical line-role workspace helpers should keep the reliability win from the new paved road without making workers spend more shell turns and wrapper tokens than the older ad hoc path. A fresh worker in either stage should still have one obvious safe workflow inside the bounded workspace, but that workflow should collapse repeated `scaffold`, `check`, and `install` subprocess calls into fewer, quieter local operations.

The user-visible success condition is simple. A rerun of the recipe benchmark slice should still show `validated_shard_count: 5` and `watchdog_killed_shard_count: 0`, and the line-role stage should keep its current no-watchdog-kill state, but both stages should use materially fewer helper-driven command turns than the current `2026-03-22_03.33.50` run. A novice opening either worker root should still see the stage-local helper under `tools/`, `current_task.json`, `OUTPUT_CONTRACT.md`, and the mirrored examples, but the prompt and helper commands should no longer push the worker through unnecessary per-task subprocess churn.

## Surprises & Discoveries

- Observation: the recipe helper solved the original reliability problem completely.
  Evidence: `data/golden/benchmark-vs-golden/2026-03-22_03.33.50/single-book-benchmark/saltfatacidheatcutdown/codexfarm/prompt_budget_summary.json` reports `validated_shard_count: 5`, `invalid_shard_count: 0`, `missing_output_shard_count: 0`, and `watchdog_killed_shard_count: 0` for `recipe_correction`.

- Observation: helper adoption was real, not accidental.
  Evidence: recipe worker prompts in `data/output/2026-03-22_03.33.50/.../recipe_phase_runtime/workers/worker-*/prompt.txt` include `tools/recipe_worker.py`, worker manifests list `mirrored_tool_files: ["recipe_worker.py"]`, and worker final messages explicitly mention `check` and `install` in files such as `.../worker-003/last_message.json`.

- Observation: the helper path is more expensive because it decomposes one dangerous big write into several safe subprocess rounds.
  Evidence: `worker-003/events.jsonl` from `2026-03-22_01.34.12` shows one giant `cat > out/... <<'EOF'` command, while the same worker in `2026-03-22_03.33.50` shows separate loops for `python3 tools/recipe_worker.py scaffold ...`, then `check ...`, then `install ...`.

- Observation: most of the cost increase is wrapper and command traffic, not a loss of recipe quality.
  Evidence: comparing `prompt_budget_summary.json` across the two runs gives:
    `call_count 21 -> 26`
    `command_execution_count_total 58 -> 64`
    `command_execution_tokens_total 371350 -> 453733`
    `wrapper_overhead_tokens 890020 -> 1221995`
    `tokens_total 1586642 -> 2318751`
    `validated_shard_count 5 -> 5`
    `watchdog_killed_shard_count 1 -> 0`

- Observation: the current prompt over-prescribes the safe workflow.
  Evidence: the recipe worker prompt currently teaches `scaffold`, then `check`, then `install` as the default path for every task, which nudges the model into extra round trips even when one batch prepare and one batch finalize would suffice.

- Observation: line-role shows the same over-ceremonial helper pattern, just with a different helper binary and no token-cost telemetry attached.
  Evidence: comparing `prompt_budget_summary.json` across `2026-03-22_01.34.12` and `2026-03-22_03.33.50` gives `command_execution_count_total 98 -> 106` for `line_role` with `watchdog_killed_shard_count 0 -> 0`, and `worker-003/events.jsonl` from `2026-03-22_03.33.50` contains many `python3 tools/line_role_worker.py ...` calls plus chained `scaffold -> check -> install` command lines.

- Observation: line-role currently pays this extra command count without billed-token visibility in the stage summary.
  Evidence: `prompt_budget_summary.json` reports `line_role command_execution_count_total: 106` but `command_execution_tokens_total: 0`, so command-count reduction is the more stable acceptance signal for that stage.

- Observation: the self-rendered line-role helper script is brittle whenever new f-strings are added because `render_line_role_worker_script()` still uses outer `.format(...)` substitution.
  Evidence: the first implementation pass failed `tests/parsing/test_line_role_workspace_tools.py` and a direct `.venv` render check with `KeyError: 'row_task_id'` until the new inner braces were escaped.

## Context and Orientation

The recipe workspace worker lives under `raw/llm/<workbook_slug>/recipe_phase_runtime/workers/worker-*/`. Each worker root includes `worker_manifest.json`, `current_task.json`, `assigned_tasks.json`, `assigned_shards.json`, `hints/<task_id>.md`, `in/<task_id>.json`, `out/`, `scratch/`, `OUTPUT_CONTRACT.md`, `examples/`, and `tools/recipe_worker.py`. The helper was added in `cookimport/llm/recipe_workspace_tools.py`, rendered into the worker root by `cookimport/llm/codex_farm_orchestrator.py`, and advertised by the shared direct-exec runner in `cookimport/llm/codex_exec_runner.py`.

The canonical line-role workspace worker lives under `line-role-pipeline/runtime/line_role/workers/worker-*/`. Its helper lives under `tools/line_role_worker.py`, and its prompt is built in `cookimport/parsing/canonical_line_roles.py`. The line-role worker uses the same top-level ergonomics as recipe: `worker_manifest.json`, `current_task.json`, `assigned_tasks.json`, `assigned_shards.json`, `hints/`, `in/`, `out/`, `scratch/`, `OUTPUT_CONTRACT.md`, mirrored examples, and the stage-local helper CLI under `tools/`.

Both helpers now expose the old `overview`, `show`, `scaffold`, `check`, and `install` verbs plus the new lower-churn `prepare`, `prepare-all`, `finalize`, and `finalize-all` verbs. The important contract change is that the happy path no longer requires one subprocess to scaffold, another to validate, and another to install for every task. The worker can now prepare all assigned drafts in one interpreter session, edit locally under `scratch/`, then finalize them in one interpreter session while preserving the existing validation rules.

In this plan, “batch verb” means one helper subcommand that processes more than one task in a single interpreter run. “Quiet mode” means helper success output is intentionally minimal so the event log and prompt-budget accounting do not fill with low-signal `OK` or full output paths. “Finalize” means “validate and install in one command,” so the worker does not need a separate `check` subprocess unless it explicitly wants diagnostic output before writing.

## Plan of Work

The first milestone is to redesign the recipe and line-role helper interfaces around fewer subprocess turns. In `cookimport/llm/recipe_workspace_tools.py` and the line-role helper seam in `cookimport/parsing/canonical_line_roles.py`, keep the current single-task pure functions, but add higher-level batch operations that the mirrored CLIs can call. A good minimal surface is:

- `prepare-all [--dest-dir scratch/]` to scaffold every assigned task in one interpreter session
- `prepare <task_id> [--dest <path>]` as the renamed single-task scaffold path
- `finalize <json_path>` to validate and install one draft in one step
- `finalize-all <glob-or-dir>` to validate and install multiple drafts in one interpreter session

The important change is not the exact names. It is that the normal worker loop should no longer require one subprocess for `check` and another for `install` for every task. `check` can remain for debugging, but it should stop being the prompt’s default happy path.

The second milestone is to make helper success output cheap. In both helper implementations, change the CLI so success emits the minimum useful text. `prepare-all` should print either one concise summary line or nothing in quiet mode. `finalize` and `finalize-all` should not print long absolute output paths on success. They should print a short relative summary such as `installed 5 task outputs` or, if needed, one relative `out/<task_id>.json` line per task. Save detailed error output for failures only.

The third milestone is to adjust the stage prompts to fit the cheaper helper. In `cookimport/llm/codex_farm_orchestrator.py`, rewrite `_build_recipe_workspace_worker_prompt(...)` so it still starts from `worker_manifest.json`, `current_task.json`, `OUTPUT_CONTRACT.md`, and the named hint/input files, but the default helper loop becomes lower-churn. In `cookimport/parsing/canonical_line_roles.py`, rewrite the line-role worker prompt in the same spirit so it points to the batch helper flow instead of `scaffold/check/install` per task. A concrete target is:

1. use `python3 tools/recipe_worker.py prepare-all --dest-dir scratch/` once per worker when that is helpful;
2. edit the generated drafts directly under `scratch/`;
3. run `python3 tools/recipe_worker.py finalize-all scratch/` once;
4. use `show` or `check` only when the worker needs diagnostics.

The prompt should explicitly say that `check` is optional debugging and that the repo-provided finalize command already validates before writing.

The fourth milestone is to keep the shared manifest examples aligned with this cheaper path. In `cookimport/llm/codex_exec_runner.py`, update the worker-manifest shell examples so recipe and line-role workers advertise the batch helper flow rather than the current `scaffold` and `install` per-task examples. Keep the generic shared structure, but make each stage-specific example list demonstrate one low-turn path.

The fifth milestone is tests. Extend `tests/llm/test_recipe_workspace_tools.py` so it covers `prepare-all`, `finalize`, `finalize-all`, and the quieter success output. Extend the line-role helper tests under `tests/parsing/test_canonical_line_roles.py` so the line-role prompt and mirrored helper examples move to the batch path too. Extend `tests/llm/test_recipe_phase_workers.py` and `tests/llm/test_codex_exec_runner_workspace.py` as needed. The tests must prove that the cheaper helper path preserves exact compact keys, exact `sid`, and exact owned-id coverage for recipe and the exact line-row coverage contract for line-role.

The sixth milestone is the benchmark proof. Do not run it without explicit user approval. If the user later approves a live rerun, compare the recipe and line-role sections of `prompt_budget_summary.json` against both the pre-helper baseline (`2026-03-22_01.34.12`) and the current helper run (`2026-03-22_03.33.50`). The intended result is: keep the reliability win from the latter while moving command count and command-execution tokens back down toward or below the former where those metrics exist.

## Idempotence and Recovery

This plan is additive and retry-safe. Recipe and line-role worker roots are regenerated for each run, so changing the mirrored helper scripts and prompts is safe to repeat. Batch prepare should only write into explicit destinations such as `scratch/`. Batch finalize should be the only helper path that writes to `out/`.

If a partial implementation leaves both old and new helper verbs in place, keep the old single-task verbs working until the prompt and tests fully move to the lower-turn interface. If quiet output makes debugging too hard, keep a verbose flag for developer use but make quiet behavior the default for worker runs.

Do not try to recover command efficiency by removing validation. The invariant checks on compact keys, exact `sid`, and exact owned `rid` coverage must remain intact.
