# Task-File Helper-First Worker Start

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

After this change, single-file Codex workers will no longer be taught to start by dumping the entire `task.json` into model context. Instead, the repo will provide a small helper-first read path: get a compact summary, inspect only the needed unit payloads, edit answers, then run the existing same-session handoff. This should reduce avoidable prompt bloat on large knowledge and line-role assignments while keeping deterministic code firmly in charge of validation and artifact expansion.

The user-visible proof is deterministic. Focused tests will show that `cookimport.llm.editable_task_file` can print a compact summary and targeted unit views, and the live worker prompts and workspace examples will explicitly prefer those helpers over `cat task.json`. No live Codex benchmark is required for acceptance.

## Surprises & Discoveries

- Observation: The largest clear performance spike is not broad orchestration overhead; it is whole-file worker startup reads on very large single-file tasks.
  Evidence: `wc -c` on `data/output/2026-03-31_13.18.44/.../nonrecipe_finalize/workers/worker-001/task.json` returned `10486541`.

- Observation: The current direct-exec runtime already contains the right conceptual seam (`editable_task_file --summary`), but it is too weak to displace `cat task.json`.
  Evidence: `cookimport/llm/editable_task_file.py` can summarize and apply answers, but it cannot print targeted unit payloads, and the live prompts still say “read the whole file once.”

- Observation: Benchmark traces show the worker actually follows the bad default.
  Evidence: `activity_trace_summary.md` includes startup commands such as `cat task.json` and `sed -n '1,260p' task.json` for recipe and line-role workers.

## Outcomes & Retrospective

This change stayed deliberately narrow and that was the right call. The runtime contract did not need another authority reset; it needed a cheaper first interaction. The shared task-file helper can now provide a compact summary plus targeted unit reads, and the live worker prompts now teach that path instead of normalizing whole-file dumps. Focused tests passed across the helper, shared workspace manifest, recipe prompt, knowledge prompt, line-role prompt, and knowledge task-file helper metadata.

The key lesson is that single-file worker runtimes can still be slow even after structural simplification if the first prompt tells the model to echo the entire file back into context. The benchmark evidence pointed straight at that behavior, and the fix was to make the smaller repo-owned read path explicit and easy.

## Context and Orientation

Single-file workspace workers live behind `cookimport/llm/codex_exec_runner.py`. Stage-local prompt builders live in `cookimport/llm/recipe_stage_shared.py`, `cookimport/llm/knowledge_stage/recovery.py`, and `cookimport/parsing/canonical_line_roles/planning.py`. The shared deterministic helper for `task.json` lives in `cookimport/llm/editable_task_file.py`.

In the current runtime, the happy path is: inspect `task.json`, edit only `/units/*/answer`, then run the stage-owned same-session handoff helper. The problem is that the live prompts and workspace examples still explicitly tell the worker to read the whole file, and the helper surface does not yet offer a repo-owned way to inspect only one unit or a small slice. On large knowledge-group or line-role tasks, that startup read can dominate the billed context.

The acceptance boundary is strict. Deterministic code may summarize state and expose narrow unit reads, but it must not infer or overwrite semantic answers. Validation, repair rewrites, and final artifact expansion remain stage-owned deterministic seams.

## Plan of Work

First, extend `cookimport/llm/editable_task_file.py` with a compact reader surface that is genuinely useful to a worker. The default summary should stay small enough to be a startup command, and the helper should support targeted reads by `unit_id` so the worker can inspect the exact evidence it needs without dumping the full file. Keep the JSON output deterministic and stage-agnostic.

Second, update the live prompts and worker-manifest examples so the first instruction is helper-first rather than whole-file-first. The shared single-file worker `AGENTS.md` text in `cookimport/llm/codex_exec_runner.py` and the stage-local prompts in recipe, knowledge, and line-role should tell the worker to start with the summary helper, then inspect only the needed unit payloads. Remove the shell examples that normalize `sed` or `cat task.json` as the expected startup move.

Third, add focused tests. The helper tests must prove that the new reader surface returns compact deterministic JSON and that answer-apply behavior is unchanged. Prompt tests must prove the live wording changed in the intended places. These tests are the before/after proof for the perfect-commit requirement because they pin the contract directly.

Fourth, update the active LLM docs with one short note describing the helper-first startup path and complete this ExecPlan with the final verification results.

## Validation and Acceptance

This plan is accepted only when all of the following are true.

The first acceptance criterion is helper usefulness. `python3 -m cookimport.llm.editable_task_file --summary` must stay compact, and the same CLI must be able to print a targeted unit payload by `unit_id` without mutating the file.

The second acceptance criterion is prompt behavior. The shared worker instructions and the recipe, knowledge, and line-role task prompts must prefer the repo-owned summary or unit helper instead of telling the worker to read the whole `task.json` at startup.

The third acceptance criterion is preserved authority boundaries. The new helper surfaces may only read or explicitly apply answers; they must not infer missing semantic values.

The fourth acceptance criterion is deterministic proof. The focused pytest slices listed above must pass in `.venv`.

## Idempotence and Recovery

The new helper read commands must be safe to run repeatedly because they are read-only. The existing answer-apply commands must remain idempotent. If any prompt test fails while the helper is being updated, fix the prompt and the test in the same turn so the contract never points at a missing or misleading helper.
