# Clean Up QualitySuite Winner Settings Boundary

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, choosing a recipe pipeline in the interactive CLI should stay clean. The menu should not dump a long compatibility warning block just because an old saved QualitySuite winner profile exists. The underlying behavior must remain compatibility-safe: older winner payloads still load, the current operator/product contract split still works, and new winner payloads should stop reintroducing deprecated or metadata-only seams into ordinary interactive flows.

## Surprises & Discoveries

- Observation: the interactive menu text itself is not the problem.
  Evidence: `cookimport/cli_ui/run_settings_flow.py` only defines short menu help; the dump appears after `choose_run_settings(...)` loads and re-harmonizes settings.

- Observation: there are two independent warning sources.
  Evidence: old winner payloads still carry retired keys, while `RunSettings.to_run_config_dict()` adds `bucket1_fixed_behavior_version`, which later gets fed back into `RunSettings.from_dict(...)` and logged as unknown.

- Observation: the repo already stores multiple projections in winner payloads, but loading still prefers the raw `run_settings` block.
  Evidence: `cookimport/config/last_run_store.py` writes `run_settings`, `operator_run_settings`, and summaries, yet `_load_run_settings_file(...)` currently reads `run_settings` first.

- Observation: the interactive warning was broader than the winner-store path.
  Evidence: `run_settings_flow.py` also re-parsed `to_run_config_dict()` in the default-profile path, the recipe-pipeline override path, and the Codex model/reasoning override path.

## Decision Log

- Decision: keep compatibility for legacy winner payloads, but normalize them once at load time.
  Rationale: the user-facing problem is repeated warning noise, not the existence of historical files. Loading should migrate forward and then return a clean `RunSettings` object.
  Date/Author: 2026-03-14 / Codex

- Decision: do not teach `RunSettings.from_dict(...)` that metadata like `bucket1_fixed_behavior_version` is a real run setting.
  Rationale: that would blur the persistence-schema boundary again. Metadata should be stripped or handled before `RunSettings` validation.
  Date/Author: 2026-03-14 / Codex

- Decision: new winner payloads should continue to include raw reproducibility data, but interactive selection should prefer a normalized operator-safe payload when available.
  Rationale: this preserves archaeology and reproducibility without forcing every ordinary caller to ingest the raw schema first. The landed implementation keeps raw payloads, adds an explicit `product_run_settings` slice, and sanitizes any loaded payload before interactive reuse.
  Date/Author: 2026-03-14 / Codex

- Decision: reuse the existing `project_run_config_payload(..., contract=\"full\")` helper instead of inventing a second “strip metadata” helper.
  Rationale: the helper already means “only keys that are real `RunSettings` fields.” Using it at the persistence boundary and the interactive revalidation boundary keeps the model/persistence split explicit and avoids another near-duplicate normalization API.
  Date/Author: 2026-03-14 / Codex

## Outcomes & Retrospective

This cleanup is implemented. The interactive chooser no longer needs to warn about persistence-only `bucket1_fixed_behavior_version` metadata because the flow now revalidates only model-field payloads, not raw persistence payloads. Legacy winner JSON files can still contain retired Bucket 1 keys, but `last_run_store.py` now strips those compatibility leftovers before constructing `RunSettings`, so the interactive menu does not dump the long retired-key warning block just for loading history.

The save path remains archaeology-friendly. Winner files now persist raw, product, and operator views together. That keeps reproducibility detail available while giving future callers a cleaner contract-specific payload to read if they do not need the full raw schema.

Update on 2026-03-14 later the same day: this winner-store compatibility stance was intentionally simplified further by `docs/plans/2026-03-14_14.08.00-run-settings-compat-shim-removal.md`. The repo no longer treats stale winner payloads as migration targets. The active behavior is now stricter: model-field-only revalidation remains, but stale winner caches are ignored instead of sanitized forward.

## Context and Orientation

`cookimport/config/run_settings.py` defines the canonical `RunSettings` model. It also defines compatibility loading in `RunSettings.from_dict(...)`, projection helpers like `to_operator_run_config_dict()`, and raw persistence via `to_run_config_dict()`. Raw persistence intentionally includes `bucket1_fixed_behavior_version`, which is metadata used for manifests and reproduction but is not a model field.

`cookimport/config/last_run_store.py` persists the QualitySuite winner profile at `history_root_for_output(output_dir)/qualitysuite_winner_run_settings.json`. The current writer stores several views: the raw `run_settings` payload, the smaller `operator_run_settings` projection, and human-facing summaries. The current loader still prefers `run_settings`, which means interactive flows ingest the noisiest possible shape even when a cleaner projection is available.

`cookimport/cli_ui/run_settings_flow.py` powers the shared interactive top-tier chooser used by import and interactive benchmark flows. When the user picks a codex profile, `choose_run_settings(...)` loads the saved winner settings, then calls `_harmonize_top_tier_pipeline_settings(...)`, which currently round-trips the winner through `settings.to_run_config_dict()` and `RunSettings.from_dict(...)`. That round-trip is what replays the metadata warning.

The desired boundary is straightforward. `RunSettings` should validate actual run settings. Metadata and retired-key compatibility should be handled at the persistence boundary, not splashed through interactive UX.

## Plan of Work

First, update `cookimport/config/last_run_store.py` so loading a winner profile intentionally prefers the cleanest compatible payload. If the file contains `operator_run_settings`, use that as the primary load source because it already reflects the modern operator contract. If the file only contains the older `run_settings` block or raw top-level fields, migrate those through compatibility loading once and return a clean `RunSettings` object. The load path should also support rewriting or normalizing the file only when a dedicated save path later chooses to do so; loading itself should remain read-safe.

Second, update the interactive harmonization path in `cookimport/cli_ui/run_settings_flow.py` so it does not feed raw persisted metadata back into `RunSettings.from_dict(...)` when it already has a `RunSettings` instance. The harmonizer should project to model fields only before validating again, or use a helper that strips persistence metadata before revalidation. The goal is to keep `bucket1_fixed_behavior_version` available for manifests and raw storage without making it look like a stray unknown key during interactive work.

Third, add focused tests. One set should prove that a legacy winner payload containing retired keys still loads into a clean `RunSettings` object. Another should prove that harmonizing a current winner setting does not emit unknown-key warnings for `bucket1_fixed_behavior_version`. A round-trip test should confirm that new winner payloads still store the raw and operator views needed by the rest of the system.

Finally, update this plan and the short understanding note if implementation reveals sharper boundaries worth recording.
