## Purpose / Big Picture

After this work lands, the single-profile benchmark pipeline will produce trustworthy processed-output reports, deterministic compatibility checks against gold blockization settings, consistent final draft schema shape, and tighter tip/topic extraction on narrative-heavy EPUBs.

A user should be able to run the same workflow that produced `data/output/2026-03-04_06.58.16/single-profile-benchmark`, then verify that report totals match on-disk artifacts, stage-block evaluation no longer fails unexpectedly from hidden extractor mismatch, final draft JSON shape is consistent across variants, and tip/topic outputs are less noisy while preserving useful knowledge snippets.

## Surprises & Discoveries

- Observation: report totals can be structurally valid JSON while still semantically wrong, because `enrich_report_with_stats(...)` does not populate count fields and assumes importers/merge paths already did that.
  Evidence: `cookimport/core/reporting.py::enrich_report_with_stats` only computes confidence/category stats and source metadata, not `totalRecipes` or tip/topic totals.

- Observation: parallel merge in Label Studio ingest currently builds a new `ConversionReport` with warnings only, which drops totals unless another layer rebuilds them.
  Evidence: `cookimport/labelstudio/ingest.py::_merge_parallel_results` creates `report = ConversionReport(warnings=warnings)` and does not set `total_recipes`, `total_tip_candidates`, or `total_topic_candidates`.

- Observation: run-setting mismatch is objective and visible in artifacts, not anecdotal.
  Evidence: processed reports in run `2026-03-04_06.58.16` use `epub_unstructured_html_parser_version=v2` and `epub_unstructured_preprocess_mode=semantic_v1`, while gold freeform span metadata carries `unstructured_html_parser_version=v1` and `unstructured_preprocess_mode=br_split_v1` across CUTDOWN books.

- Observation: final draft payload shape is mixed by variant, creating downstream contract ambiguity.
  Evidence: in this run, vanilla final drafts are `174/174` present for `recipe.title` and `steps`, but `0/174` for top-level `name`, `ingredients`, `instructions`; codexfarm is partial (`name=48`, `ingredients=64`, `instructions=43`).

- Observation: tip/topic volume is disproportionately high for long narrative books and correlates with long-block warnings.
  Evidence: vanilla `saltfatacidheatcutdown` has `168` tips and `1048` topic candidates; vanilla `thefoodlabcutdown` has `137` tips and `1230` topic candidates, with repeated `epub_too_many_super_long_blocks` warnings in report files.

## Decision Log

- Decision: fix report truth at the point report objects are assembled, not only in downstream analytics readers.
  Rationale: analytics and dashboards should not need to infer correctness around malformed producer payloads.

- Decision: add explicit gold/prediction blockization compatibility checks before stage-block scoring and keep override behavior explicit.
  Rationale: hidden extractor/preprocess drift causes false-negative benchmark quality conclusions and expensive reruns.
  Date/Author: 2026-03-04 / assistant

- Decision: enforce one canonical final-draft compatibility shape regardless of vanilla/codex origin.
  Rationale: mixed payload shapes increase downstream parser complexity and hide regressions.

## Plan of Work

### Phase 1: Report truth hardening (no silent zero-count drift)

Implement a shared report finalization helper that computes count fields from `ConversionResult` collections immediately before writing the report in benchmark ingest paths. The helper must set, at minimum, `totalRecipes`, `totalTips`, `totalTipCandidates`, `totalTopicCandidates`, `totalStandaloneBlocks`, `totalStandaloneTopicBlocks`, and related derived fields that are currently left implicit.

Apply this helper in `cookimport/labelstudio/ingest.py` after merge and before `write_report(...)`, and remove assumptions that upstream merge constructors always populate totals. Keep warning/error aggregation behavior unchanged.

Add an invariant check path in ingest writing that compares report counts to immediately available in-memory object counts. On mismatch, append explicit report warnings and include both values in structured report diagnostics rather than silently writing stale counts.

Extend `tests/labelstudio/test_labelstudio_ingest_parallel.py` so merged-result tests assert report totals and not just ordering/ID reassignment. Add one test that reproduces the prior failure mode (merged data present, report totals default zero) and verifies the fixed behavior.

### Phase 2: Gold/prediction blockization compatibility contract

Promote blockization compatibility to a first-class precondition for stage-block evaluation. Reuse and extend existing diagnostics in `cookimport/bench/eval_stage_blocks.py` so mismatch reasons are always emitted in a machine-readable artifact (for example, a concise compatibility summary JSON under eval output).

In benchmark orchestration (`cookimport/cli.py` labelstudio-benchmark path), add an explicit compatibility policy:

- default behavior: fail fast when severe mismatch is detected before scoring;
- optional explicit override flag for deliberate experiments, with report warnings that the run is not gold-compatible.

Add a small utility to extract gold-side blockization fingerprints from freeform labels (backend + relevant unstructured settings) and compare against prediction run settings. Keep this deterministic and local; do not call external services.

Extend benchmark tests to cover three cases: exact match (pass), benign drift (warning only), and severe mismatch (fail-fast with actionable message). The fail message must include concrete expected vs actual settings and a rerun hint.

### Phase 3: Final draft schema normalization invariants

Define one canonical compatibility shape for final drafts and enforce it in `write_draft_outputs(...)` regardless of source path (deterministic draft generation or codex override payload). The invariants for this plan are:

- `recipe.title` and `steps` remain authoritative;
- top-level compatibility aliases `name`, `ingredients`, and `instructions` are always present and deterministically derived;
- derived alias fields are never partially missing across variants in the same run.

Implement normalization in one location (writer boundary) to avoid divergence between producers. Reuse existing draft normalization utilities where possible (for example, step extraction/coercion semantics already implemented in codex-farm orchestrator helpers), but keep staging ownership in staging modules.

Add staging tests that assert both vanilla and override payloads emit the same alias-field presence semantics. Include edge cases with missing instructions and fallback step injection.

### Phase 4: Tip/topic precision hardening for narrative-heavy EPUBs

Tighten deterministic filtering in `cookimport/parsing/tips.py` and/or `cookimport/plugins/epub.py` for standalone narrative spans that currently leak into `general` tips and oversized topic candidates. Focus on precision, not recall expansion.

Implementation requirements:

- preserve clearly actionable cooking guidance;
- down-rank or reject long narrative prose with weak procedural signals;
- keep existing extraction deterministic and explainable via provenance fields and warnings.

Add or update parser tests with representative narrative-heavy snippets (from existing CUTDOWN books) to prove that non-actionable prose is filtered while true tips remain.

Add lightweight report diagnostics indicating how many candidates were filtered by new precision gates, so behavior changes are auditable.

### Phase 5: Integrated validation and documentation closure

Run targeted unit/integration tests first, then one reproducible benchmark smoke run with gold-compatible settings and no upload. Compare outputs against baseline artifacts and confirm acceptance gates.

Update docs minimally but explicitly:

## Validation and Acceptance

Acceptance is met only when all items below are true.

- Report truth gate:
  For processed outputs, report totals match artifact counts for recipes, tips, and topic candidates. No run should show `totalRecipes=0` when final draft files exist.

- Compatibility gate:
  Stage-block benchmark run with gold-compatible extractor settings succeeds through scoring for the smoke target. Mismatched settings produce deterministic, actionable fail-fast diagnostics.

- Draft schema gate:
  All final draft JSON outputs in validation runs include consistent top-level compatibility fields (`name`, `ingredients`, `instructions`) with deterministic derivation from canonical payload fields.

- Precision gate:
  Tip/topic extraction shows reduced narrative noise on targeted heavy books while preserving core actionable tips. This must be demonstrated by tests plus before/after counts for at least one heavy target.

- Regression gate:
  Relevant test slices pass with no new failures in touched modules.

## Idempotence and Recovery

All changes are additive and safe to rerun.

If a phase fails:

- Keep failing artifacts under their timestamped output roots for inspection.
- Re-run only the failed phase tests first.
- For benchmark smoke reruns, use a new timestamped root and do not overwrite prior evidence.

If blockization enforcement causes unexpected hard failures in existing workflows, temporarily use the explicit override flag introduced in Phase 2 while preserving warning annotations, then fix settings parity rather than disabling diagnostics globally.

If draft schema alias normalization causes downstream reader regressions, keep canonical `recipe` + `steps` unchanged and only adjust alias derivation logic; do not roll back core draft structure.

## Interfaces and Dependencies

Implement and enforce these interfaces by end of plan:

- Report finalization interface:
  A staging/ingest helper that accepts `ConversionResult` + `ConversionReport` and populates all count totals deterministically before `write_report(...)`.

- Blockization compatibility interface:
  A benchmark-eval helper that produces a stable compatibility object containing expected-gold settings, observed-prediction settings, mismatch severity, and operator guidance.

- Draft normalization interface:
  A writer-level normalization function that guarantees presence and consistency of compatibility alias fields without mutating canonical meaning of `recipe` and `steps`.

- Tip/topic filtering interface:
  Deterministic parser heuristics with explicit provenance/debug signals for filtered vs retained candidates.

External dependencies should remain unchanged for this effort. Do not enable codex-farm parsing/cleaning paths as part of this plan.
