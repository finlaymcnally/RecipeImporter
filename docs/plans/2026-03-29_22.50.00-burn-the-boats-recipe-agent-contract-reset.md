This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan is intentionally independent of the line-role and knowledge reset plans. ## Structured Restatement

The repository must keep using `codex exec` because the subscription economics are the point, but the recipe stage must stop behaving like a miniature coding workflow that burns tokens on queue management, scratch drafts, validator ping-pong, and shell churn. The new design must preserve the parts that genuinely require model judgment: deciding whether the candidate recipe is real and coherent, correcting the canonical recipe when it is real, rejecting fragmentary or non-recipe spans honestly, assigning grounded tags, and producing an ingredient-to-step mapping that makes semantic sense. Deterministic code must package evidence, include enough source truth for future raw-source cross-checking, validate outputs, and enforce authority boundaries, but it must not try to outsmart cookbook semantics. This is burn-the-boats work. The old scratch-draft recipe worker loop, its legacy artifact vocabulary, and its compatibility shims are to be removed rather than preserved.

## Purpose / Big Picture

After this change, a user can run recipe correction with the same `codex exec` subscription strategy but without paying millions of tokens for workflow ceremony. The recipe stage will still use a long-lived coding agent where that is economically useful, but the agent will be given one immutable packet at a time and asked to do semantic work, not clerical workspace maintenance. The visible proof will be that recipe workers stop reading and rewriting `CURRENT_TASK*`, `SHARD_PACKET.md`, `scratch/*.json`, and helper-command loops on the happy path; recipe telemetry will distinguish real model sessions from per-recipe outcomes; and prompt-budget artifacts will stop implying that a few thousand visible recipe tokens somehow required a huge number of pseudo-calls to accomplish almost nothing.

The user-visible behavior after implementation is simple. Running the recipe stage on a book should still produce corrected recipes, grounded tags, and ingredient-to-step mappings, but the live runtime should now look like a packet-driven semantic worker: one current recipe packet, one result path, one optional narrow repair packet, and then promotion. The stage should fail less often, spend less on wrapper overhead, and tell a much more honest story about where the remaining spend actually comes from.

A worker session is one long-lived `codex exec` process that may handle many leased packets in sequence. The worker session remains agentic and local, but the task surface becomes small, explicit, and packet-shaped.

## Plan of Work

This work is intentionally a full architectural reset rather than a local patch. The first step is to replace the recipe task contract. The live recipe stage will stop treating one worker assignment as a miniature editing workspace with drafts, install commands, feedback sidecars, and helper tools. Instead, planning will emit one immutable packet per recipe task. Each packet will carry the exact recipe evidence rows, a compact attempted deterministic hint, the recipe-local tag guide, the stable recipe id, enough provenance to support future raw-source cross-checking, and a compact statement of what the model is being asked to decide. There will be no prewritten draft file because the final output is the task result, not an intermediate scratch artifact.

The second step is to move the runtime onto the existing leased-packet workspace surface. Recipe workers will use `current_packet.json`, `current_hint.md`, `current_result_path.txt`, and `packet_lease_status.json` as the only active-task control seam. The packet will be authoritative. The hint file will be short, additive guidance only. The result-path file will tell the worker where to write its final answer. The repo will advance the lease after successful validation or after a terminal failure. The happy path will not involve `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`, `SHARD_PACKET.md`, `assigned_tasks.json`, `scratch/`, `tools/recipe_worker.py`, or `OUTPUT_CONTRACT.md`.

The third step is to simplify the prompt and recovery model around the actual semantic task. The live recipe prompt will be rewritten to describe one current packet at a time rather than a queue of drafts. It will still require the model to classify the task as `repaired`, `fragmentary`, or `not_a_recipe`, to produce grounded tags, and to emit a justified ingredient-to-step mapping. It will also explicitly say that the model is allowed to reject a recipe honestly instead of repairing junk. The first attempt must be command-free in the normal case. Deterministic validation will check the result shape and basic semantic invariants. If the result is invalid, the repo will write one repair packet that includes the original packet id, a concise list of validation failures, and the rejected prior output. The worker gets one repair attempt. If that repair still fails, the task fails. There will be no layered recipe-specific same-session fix budget, per-task watchdog retry path, or separate fallback repair pipeline after that.

The fourth step is to separate semantic outputs from cost accounting. The runtime will produce one set of task-result rows for per-recipe observability and one set of worker-session rows for real model spend. Prompt-budget summaries and analytics will use session rows for token totals, session counts, and pathological flags. Stage observability will use task-result rows for per-recipe statuses, tags, mapping availability, and promotion outcomes. If per-task token apportionment cannot be derived honestly from the runner, those fields should remain blank rather than being synthetically spread across task rows.

The fifth step is to remove old code rather than leave dead architecture standing. The recipe-specific scratch-draft tools and live helper markdown will be deleted or replaced with packet utilities. The old prompt asset and schema ids will be replaced with new packet-first names. Recipe config and CLI wiring will reject the old live pipeline id instead of normalizing it. Docs, tests, benchmark helpers, and analytics will be updated to describe only the new story.

## Milestones

### Milestone 1: Define the new packet-first recipe contract

At the end of this milestone, the repository has a new live recipe input and output contract that is shaped around one recipe task packet at a time, not around a batch of editable drafts. A novice should be able to open one packet file, understand the source evidence and required semantic decisions, and know exactly what the worker must return. The observable proof is that the repo contains new recipe packet models, new schema and prompt assets, and tests that validate one-packet happy-path behavior and one-packet repair behavior without any scratch drafts.

Implementation in this milestone should create explicit packet models inside the recipe-stage package rather than adding more shape logic to `recipe_stage_shared.py`. The preferred split is to introduce a contract module under `cookimport/llm/recipe_stage/` that defines the task packet, repair packet, output payload, and task-result summary records. The live packet should own exactly one recipe id. The output should also own exactly one recipe id. This is a deliberate simplification over the old shard output array shape. The compact key strategy may stay compact for token discipline, but the runtime must no longer pretend that one model output file contains a mini-batch of many recipe results.

The pipeline asset should also be renamed to reflect the new story. Create a new live prompt asset and schema under `llm_pipelines/` using a packet-first name such as `recipe.correction.packet.v2`. Remove the old live recipe correction asset from runtime wiring instead of teaching the code to accept both names. The task packet should include raw evidence rows, stable block or line indices, deterministic hints as hints only, and the recipe-local tagging guide. It should also carry an explicit source-truth field for future raw-source cross-checking so this repository does not need another contract rewrite later just to add raw-text verification.

The validation proof for this milestone is a set of narrow contract tests in the `tests/llm/` area that fail if the live packet contains batch-oriented fields, fails if a packet owns more than one recipe id, fails if the output omits tags or mapping-reason fields required by the schema, and fails if the repair packet omits the original validation errors or rejected prior output.

### Milestone 2: Replace the recipe runtime with leased packets

At the end of this milestone, the recipe stage runs on the existing leased-packet workspace surface instead of the scratch-draft helper loop. The worker session stays long-lived, but the repo now drives it one current packet at a time. The user-visible change is that live worker roots contain the packet-leasing files and packet artifacts, not the recipe-specific draft sidecars. The observable proof is that the recipe worker prompt and workspace files no longer mention `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`, `SHARD_PACKET.md`, `scratch/`, or `python3 tools/recipe_worker.py check-current`.

Implementation should reuse the packet-leasing support already present in `cookimport/llm/codex_exec_runner.py`. Add a dedicated recipe workspace runner module under `cookimport/llm/recipe_stage/` that claims the next task packet, writes the current packet files, waits for the worker to write the result path named in `current_result_path.txt`, validates the result, and advances or repairs the lease. The existing recipe-specific task queue controller should be removed from the live path. The old recipe helper tools file should either be deleted entirely or reduced to pure packet-rendering helpers that do not express a draft-edit loop.

This milestone must also tighten the recipe worker prompt. The prompt should tell the worker to start from `worker_manifest.json`, `current_packet.json`, `current_hint.md`, and `current_result_path.txt`, then read the packet, reason about the recipe semantically, and write the final output file. It should say that shell commands are not needed in the normal case. It should not instruct the worker to run helper scripts or repair the same JSON draft repeatedly. If a repair packet appears later, the prompt should tell the worker to re-open the current packet files and fix only the named issues.

The verification for this milestone is behavior, not just file creation. Add tests that run the recipe workspace runtime with a fake or scripted runner and assert that the live recipe worker root includes the packet-leasing files, excludes the old task-sidecar files, advances through multiple recipe packets in one session, and completes clean happy-path packets without command-execution telemetry. Add a regression that fails if the recipe runtime writes `assigned_tasks.json`, `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`, `SHARD_PACKET.md`, or `scratch/_prepared_drafts.json` in the live path.

### Milestone 3: Collapse retries to one narrow repair path

At the end of this milestone, recipe correction has one simple recovery story. The first output either validates or it does not. If it does not, the repo writes one repair packet naming the exact failures and giving the worker one follow-up chance. There is no recipe-specific same-session fix budget, no draft-level install command, no per-task watchdog retry directory, and no second repair channel layered on top of the first. The observable proof is that recipe status artifacts and tests no longer mention `same_session_fix`, `watchdog_retry`, or recipe draft-install flows.

Implementation here should remove the recipe-only recovery scaffolding from `cookimport/llm/recipe_stage_shared.py` and replace it with a smaller recovery module in the recipe-stage package. That module should know how to build one repair packet, store one rejected output copy, and mark the final task state as either `validated`, `repaired_after_followup`, or `failed_after_followup`. The shared Codex watchdog should remain for gross session liveness and boundary enforcement, but recipe runtime should stop treating that as a normal semantic recovery channel.

The important acceptance test is that a malformed first output can be recovered by one repair packet without reopening the whole session contract, while a repeatedly invalid output terminates cleanly with a clear failure artifact. Another required regression is that the repo no longer writes `same_session_fix_status.json`, `watchdog_retry/`, or any recipe-specific retry prompt files for the live happy path.

### Milestone 4: Reset telemetry, analytics, and benchmark artifacts

At the end of this milestone, recipe cost reporting tells the truth. Real model sessions are reported as sessions. Leased packets are reported as task throughput. Per-recipe outcomes are reported as semantic results. No artifact should imply that one worker session magically became dozens of independent model calls just because the repo later attached per-task rows to it. The user-visible proof is that `prompt_budget_summary.json`, stage observability, and upload-bundle diagnostics all separate session count, task-packet count, and recipe-result count.

Implementation should add or revise telemetry models so the recipe runtime emits at least three clearly named surfaces: worker-session telemetry, task-packet telemetry, and per-recipe semantic outcome rows. Prompt-budget code in `cookimport/llm/prompt_budget.py` should aggregate cost only from worker-session telemetry and expose leased-packet counts separately. Stage observability should keep its per-recipe summaries, but it must no longer borrow fake call counts from task rows. Benchmark helper code and analytics readers must be updated to use the new fields and to reject old assumptions rather than silently adapting them.

The tests for this milestone must prove that a run with five worker sessions and twenty-nine recipe packets reports five sessions, twenty-nine packets, and twenty-nine per-recipe semantic outcomes as three different numbers. The tests should also prove that missing per-packet token usage stays blank rather than being fabricated by spreading session totals across task rows. This milestone is where the old reporting confusion is intentionally broken and replaced.

### Milestone 5: Preserve and improve semantic recipe authority

At the end of this milestone, the refactor has not hollowed out the semantics. The model still decides whether the recipe is real, still corrects the recipe when it is real, still rejects fragmentary or non-recipe spans honestly, still assigns grounded tags, and still produces ingredient-to-step mappings only when justified. The packet contract should make those semantics clearer and cheaper, not weaker. The observable proof is that recipe-stage tests still cover repaired, fragmentary, and not-a-recipe outcomes, tag selection, and mapping reasoning under the new packet runtime.

Implementation in this milestone should tighten the packet contents rather than bloat them. The packet should contain the authoritative source rows, compact deterministic hints, and recipe-local tag guidance. It should not contain prewritten canonical recipe drafts that tempt the worker to rubber-stamp them. It should include enough contextual evidence for the worker to say "this is not really a recipe" or "this recipe is too fragmentary to normalize safely" without guessing. It should also include the source detail needed for a future raw-source cross-check seam without forcing that future work into this change.

Validation here should emphasize semantic coverage. Existing recipe tests in `tests/llm/` and staging observability tests should be updated to assert that the new runtime still preserves final promotion behavior, still emits mapping reasons, still leaves `fragmentary` and `not_a_recipe` visible as valid task outcomes, and still feeds deterministic final assembly only when the LLM says `repaired`. If the refactor saves tokens but weakens recipe correctness, it is a failure.

### Milestone 6: Delete the old live recipe path and update docs

At the end of this milestone, the repository no longer presents the scratch-draft recipe loop as a live option. The old pipeline asset id, old helper surfaces, old recipe workspace instructions, and old tests that encode the draft-edit story are gone or rewritten. The user-visible proof is that the docs explain only the packet-first recipe runtime and that new contributors no longer discover the old path and think it is current.

Implementation should remove or radically shrink `cookimport/llm/recipe_workspace_tools.py`, delete dead recipe-specific helper-command flows, update `cookimport/llm/README.md`, `cookimport/llm/recipe_stage/README.md`, `docs/10-llm/10-llm_README.md`, and any staging docs that describe the old runtime. The run-setting and CLI docs should be updated to point at the new live pipeline id and to describe recipe worker sessions in packet-first language. Historical benchmark directories under `data/golden/` do not need runtime compatibility. Tests and docs must be updated immediately rather than bridged by compatibility readers.

The final required validation is the project-preferred test flow from the local `.venv`, plus targeted recipe, staging, labelstudio, and benchmark helper suites that exercise the new contract. A live CodexFarm or Codex Exec benchmark run against a real book is explicitly optional until the user approves it. `AGENTS.md` forbids CodexFarm-enabled benchmark processing without explicit user approval, so this plan must not treat a live benchmark as required implementation work.

## Validation and Acceptance

Acceptance is behavioral and architectural.

The first acceptance criterion is contract clarity. One recipe task packet must own exactly one recipe id. The worker must be able to classify it as `repaired`, `fragmentary`, or `not_a_recipe`, assign grounded tags, and provide an ingredient-to-step mapping or a machine-readable reason for leaving the mapping empty. Tests should fail if the live contract reintroduces batch output arrays, prewritten scratch drafts, or helper-command guidance as the normal path.

The second acceptance criterion is packet-leased runtime behavior. A live recipe worker assignment must use the current-packet files from the shared Codex runner. The happy path must not depend on repo-written `CURRENT_TASK*` sidecars, `SHARD_PACKET.md`, `assigned_tasks.json`, `scratch/` drafts, helper tool invocations, or repeated validator loops. Tests should fail if those files appear in the live recipe path.

The third acceptance criterion is honest recovery. A malformed first output should trigger exactly one repair packet with explicit validation errors. A second invalid output should terminate cleanly with a clear failure artifact. Tests should fail if the live recipe path still emits `same_session_fix` counters, recipe-specific watchdog retry artifacts, or draft-install status files.

The fourth acceptance criterion is semantic preservation. The new runtime must still support grounded recipe correction, honest fragmentary and non-recipe rejection, tag output, and ingredient-to-step mapping. A refactor that saves tokens by weakening recipe semantics does not meet the goal. Tests should cover repaired recipes, fragmentary candidates, clear non-recipes, obvious tags, and mapping-required versus mapping-unnecessary cases.

The fifth acceptance criterion is truthful cost reporting. Prompt-budget summaries must report real worker session counts and task-packet counts separately. Per-recipe semantic outcome rows must not be misreported as model calls. If token data is unavailable at the task level, it must remain blank instead of being evenly spread across tasks. Tests should prove that this distinction survives through prompt-budget summaries, benchmark cutdown helpers, and stage observability.

The sixth acceptance criterion is burn-the-boats cleanup. Live docs, runtime code, tests, and pipeline assets must no longer advertise the old recipe draft loop, the old pipeline id, or the old helper-command story. Historical files under `data/golden/` may still exist on disk, but the live runtime must not support both contracts.

## Idempotence and Recovery

This plan is safe to implement incrementally as long as the milestones are followed in order and the repository is kept compiling between steps. The contract work is additive first: define the new packet models and prompt assets, add tests, and wire the new runtime beside the old code only long enough to finish the migration locally. Once the new runtime is passing its targeted tests, delete the old live path in the same branch. Do not leave both runtimes supported in final code.

If a step fails midway, the safe recovery path is to rerun the targeted tests for that milestone after finishing the missing producer or consumer. If a telemetry reader breaks because the new fields are not wired yet, fix the writer and the reader in the same milestone rather than fabricating adapter code. If `.venv` drifts, reactivate it and rerun `pip install -e .[dev]`. If a benchmark or live-book run would help answer a question, stop and wait for explicit user approval before running it.

## Interfaces and Dependencies

The implementation should end with a smaller and more explicit recipe-stage surface under `cookimport/llm/recipe_stage/`. At minimum, add or reshape modules so the live code has clearly named packet-contract, workspace-runtime, validation, reporting, and promotion seams. The exact filenames may vary, but the repository should no longer depend on `cookimport/llm/recipe_stage_shared.py` as the effective source of truth for the whole recipe runtime.

The live recipe packet contract should define stable models equivalent to the following concepts:

    RecipeTaskPacketV2
      One authoritative current-packet payload for exactly one recipe candidate.
      Includes:
        - task id
        - recipe id
        - authoritative source evidence rows
        - deterministic hints as hints only
        - recipe-local tag guide
        - source provenance sufficient for future raw-source cross-checks

    RecipeTaskRepairPacketV2
      One follow-up packet for the same task id.
      Includes:
        - original packet id
        - prior rejected output
        - explicit validation errors
        - instruction to fix only the named issues

    RecipeTaskOutputV2
      One final recipe-task output payload.
      Includes:
        - task id
        - recipe id
        - status
        - status reason
        - canonical recipe when repaired
        - ingredient-step mapping
        - mapping reason
        - tags
        - warnings

The runtime should expose a leased-packet recipe worker loop conceptually equivalent to:

    run_recipe_packet_worker_assignments(...)
      Build worker assignments.
      Lease one task packet at a time into each worker session.
      Validate each result.
      Emit at most one repair packet when needed.
      Promote validated task outputs.
      Write separate session telemetry and task-result summaries.

The reporting layer should expose separate aggregation functions conceptually equivalent to:

    summarize_recipe_worker_sessions(...)
      Returns real model-session counts and token totals.

    summarize_recipe_task_packets(...)
      Returns leased-packet throughput and repair counts.

    summarize_recipe_task_outcomes(...)
      Returns per-recipe semantic statuses, mapping statuses, tag counts, and promotion outcomes.

The pipeline and config layer must switch to the new live asset ids. Update the recipe pipeline id in `cookimport/config/run_settings.py`, `cookimport/config/run_settings_types.py`, `cookimport/config/codex_decision.py`, CLI or interactive settings code under `cookimport/cli_ui/`, and any docs or tests that assert the old live id. Reject the old live id in validation rather than silently mapping it to the new one.

The tests that matter most are the recipe worker tests, workspace-runner tests, stage observability tests, prompt-budget summary tests, benchmark artifact tests, and any tests that assert recipe pipeline ids or recipe worker artifact shapes. Those tests are part of the implementation surface, not optional cleanup.
