## Purpose / Big Picture

After this change, the pipeline should ask its LLM stages smaller, better-grounded questions and stop asking deterministic code to make fuzzy cookbook judgments. The goal is not merely to make prompt text prettier. The goal is to make the live line-role and knowledge stages behave more like the strong “handful of lines in a chat window” experience the user already trusts, while still preserving the pipeline’s reproducibility, validation, and stage authority boundaries.

The user-visible effect is a pipeline that remains deterministic-first for mechanics but becomes more LLM-centered for ambiguity. Line-role should receive better local boundary evidence without being handed fake recipe truth. Knowledge should receive smaller, more coherent shards and should stop starting from deterministic semantic scaffolds that teach the wrong lesson. A reviewer inspecting future worker roots should be able to say: the repository packaged the evidence carefully, the model made the fuzzy call, and the runtime validated exact coverage and normalized output shape without silently deciding semantics itself.

The target outcome is not “one benchmark got a little better.” The target outcome is a repo architecture that can plausibly climb toward `>90%` on arbitrary cookbooks by exploiting model capability instead of constraining it with oversized mixed-material packets and weak local context.

## Outcomes & Retrospective

This plan is not implemented yet. The intended outcome is a pipeline whose LLM stages are better fed and less constrained, without giving up the repository’s reproducibility and authority boundaries. The line-role stage should stop suffering from missing neighbor evidence. The knowledge stage should stop suffering from deterministic semantic scaffolds and mixed-material megashards. The repo should become a better witness, not a more opinionated fuzzy classifier.

If this plan is implemented correctly, the resulting system should look and feel more like a careful model-driven review pipeline and less like a deterministic pipeline with LLMs bolted onto the side. The most important success condition is not any one code diff. It is that a future worker root and its tests tell a coherent story: local evidence is visible, fuzzy calls belong to the model, and deterministic code keeps everything exact and auditable.

## Context and Orientation

The current product pipeline is explained in `docs/AI_context.md` and `docs/plain-english-explanation.md`. The short version is:

1. importer code creates normalized source blocks
2. line-role labeling happens in the `recipe-boundary` stage
3. grouped recipe spans become authoritative after line-role
4. recipe correction works only inside accepted recipe spans
5. non-recipe routing and knowledge review happen after recipe ownership is settled

There are three LLM seams in the current live architecture: `codex-line-role-shard-v1`, `codex-recipe-shard-v1`, and `codex-knowledge-shard-v1`. This plan focuses on the first and third seams because they are the places where poor task packaging most directly limits whole-book accuracy.

In this repository, “mechanical truth” means facts the repo can know without ambiguity. Examples include block order, exact shard ownership, the names of the active phase files, whether every owned row has been covered once, and how installed output ids should be normalized. “Semantic truth” means the fuzzy judgment about what a line or block means in cookbook context. Examples include whether a heading is a real recipe title or chapter taxonomy, whether prose is durable cooking knowledge or mere framing, and whether several kept knowledge blocks belong in one idea group.

The current line-role stage lives under `cookimport/parsing/canonical_line_roles/`. The key files are:

- `cookimport/parsing/canonical_line_roles/planning.py`, which partitions ordered atomic lines into contiguous shards and writes worker-facing payloads
- `cookimport/parsing/canonical_line_roles/policy.py`, which builds the debug and model payloads and computes shard-level posture
- `cookimport/parsing/canonical_line_roles/prompt_inputs.py`, which owns the row serializers for the worker transport
- `cookimport/parsing/canonical_line_roles/runtime.py`, which owns live worker orchestration and promotion
- `tests/parsing/test_canonical_line_roles.py`, which contains the focused contract tests for this surface

The important current line-role constraint is that pre-grouping candidates intentionally keep `within_recipe_span=None` until later grouping. That is a good authority decision, but it means the worker must be given other evidence if it is going to make strong fuzzy calls at shard boundaries. Right now the live payload often lacks the `context_before_rows` and `context_after_rows` fields that the docs already describe, which leaves the worker with less local evidence than intended.

The current knowledge stage lives under `cookimport/llm/knowledge_stage/` and `cookimport/llm/knowledge_phase_workspace_tools.py`. The key files are:

- `cookimport/llm/codex_farm_knowledge_jobs.py`, which builds the shard input packets under `knowledge/in/*.json`
- `cookimport/llm/knowledge_stage/workspace_run.py`, which writes the worker root and runs one worker through Pass 1 and Pass 2
- `cookimport/llm/knowledge_stage/recovery.py`, which builds the worker prompt and hint files
- `cookimport/llm/knowledge_phase_workspace_tools.py`, which defines the pass ledgers, helper CLI, validation, and final installed output shape
- `tests/llm/`, which contains the knowledge prompt and workspace contract tests

The important current knowledge constraint is different. The phase controller is already good. The problem is that the repo currently teaches semantics it does not know. Pass 1 starts from all `other`, and Pass 2 starts from blank group shells with synthetic ids. On top of that, packet planning currently produces large mixed-material shards, so a single worker packet can mix decorative headings, taxonomy, memoir, praise, and real cooking lessons.

The user’s explicit design requirement is that deterministic code must not take ownership of fuzzy cookbook decisions. Preserving that requirement means:

- do not add new heuristic semantic “features” whose real purpose is to decide cookbook ambiguity in code
- do add stronger local evidence and better packet planning so the model can decide with better context
- do keep deterministic validation, exact coverage checks, phase control, and output normalization

This plan therefore treats the repo as a packager, validator, and authority-boundary keeper. It does not treat the repo as a substitute semantic classifier.

## Milestones

## Milestone 1: Fix the line-role evidence hole without changing stage authority

At the end of this milestone, the live file-backed line-role worker payload will finally include immediate reference-only neighboring rows around each shard boundary when those rows exist. The worker will still not receive authoritative recipe truth before grouping, but it will stop losing the simplest and safest local evidence the repo already knows.

The work belongs in `cookimport/parsing/canonical_line_roles/policy.py`, `cookimport/parsing/canonical_line_roles/planning.py`, `cookimport/parsing/canonical_line_roles/prompt_inputs.py`, and the focused tests in `tests/parsing/test_canonical_line_roles.py`. Build one whole-book atomic-index lookup in planning, thread that lookup into the payload builders, and emit exactly one immediate out-of-shard row before and after the owned rows when available. Mirror the same context into the debug payload so artifact inspection and worker truth remain aligned.

The implementation must also invalidate stale line-role cache entries because the model-facing payload changes. Bump the cache schema version in `cookimport/parsing/canonical_line_roles/__init__.py` or use an equivalent invalidation mechanism. Update `docs/04-parsing/04-parsing_readme.md` to state plainly that the live worker transport now writes immediate boundary neighbors when present.

Acceptance for this milestone is a focused test that proves first, middle, and last shards serialize the correct boundary behavior, plus a payload inspection that shows `context_before_rows` and `context_after_rows` in the live worker input shape.

## Milestone 2: Make line-role shards smaller and more locally coherent

At the end of this milestone, line-role should stop asking one worker to carry about 295 rows of mixed material at once when a smaller local review surface would better match model strengths. This milestone does not change label policy and does not create stronger deterministic semantics. It changes only the planning and packaging of model work.

Start in `cookimport/parsing/canonical_line_roles/planning.py` and the run-setting surface that controls shard count. The current default target count of roughly five shards for a 1466-line book creates very large slices. Rework the default planning so the live and preview paths aim for materially smaller shards, especially for outside-recipe or front-matter-like material. Keep shard planning contiguous in source order, but reduce the amount of unrelated book material the model must carry in one ledger.

Use deterministic logic only for planning and packaging, not for semantic decisions. Examples of acceptable planning logic are splitting by size, source contiguity, and obvious transport volume thresholds. Examples of unacceptable logic are “deterministically decide this is cookbook taxonomy” or “deterministically rescue this as a recipe title.” The repo may decide where one worker chunk ends and another begins; it must not decide the ambiguous label itself.

Update the line-role docs and tests to reflect the smaller target shape. The acceptance test for this milestone is not “the benchmark score went up” because live runs require approval. The acceptance is that synthetic or preview shard plans show materially smaller owned-row counts by default, and that the worker payload remains valid and contiguous.

## Milestone 3: Redesign knowledge pass ownership so semantics belong to the model

At the end of this milestone, the knowledge stage will stop pretending that the repo has a useful semantic first draft. Pass 1 will become the model’s first-authority `knowledge|other` judgment over owned blocks. Pass 2 will become continuation over the accepted Pass 1 knowledge subset, not a second semantic re-litigation or a blank shell task.

This milestone is already substantially scoped by `docs/plans/2026-03-29_15.24.22-knowledge-pass-authority-and-semantic-ownership.md`. Implement that plan inside this umbrella milestone. The work belongs primarily in `cookimport/llm/knowledge_phase_workspace_tools.py`, `cookimport/llm/knowledge_stage/recovery.py`, and `cookimport/llm/knowledge_stage/workspace_run.py`.

The essential changes are:

- Pass 1 must stop starting from a universal repo-authored `other` semantic scaffold.
- Pass 2 must be built only from accepted Pass 1 knowledge rows.
- The repo should own stable structural identifier normalization such as final `group_id` spelling whenever that does not change semantics.
- Validation must continue to fail closed on exact coverage and exact kept-row grouping coverage.

Acceptance for this milestone is a focused test suite showing the new pass ledgers, phase handoff, and final installed output shape, plus worker prompt text and sidecars that clearly describe the new semantic ownership boundary.

## Milestone 4: Rebuild knowledge packet planning around coherent review units

At the end of this milestone, the knowledge stage will stop feeding huge mixed-material packets to a single worker. A packet should represent one reasonably coherent non-recipe review unit, not a giant bag of chapter taxonomy, memoir, praise, and useful teaching. This is the milestone that most directly addresses the “chatbot does well on a handful of lines” gap at the planning layer.

The work starts in `cookimport/llm/codex_farm_knowledge_jobs.py`. The current planner groups review-eligible non-recipe rows into a small number of contiguous packets, but the saved run shows that the resulting packets can still cover hundreds of blocks and very large index jumps. Rework packet planning so it preserves source contiguity while still splitting aggressively at obvious planning boundaries such as large block-index gaps, shifts between heading-dense taxonomy runs and prose runs, and similar mechanically observable transitions.

This milestone must stay honest about the repo’s role. The planner may use mechanical signals like block order, heading density, table hints, and gap size to avoid mixing unrelated regions. It may not deterministically decide which blocks are real knowledge. The result should be smaller, cleaner packets that are easier for the model to judge.

This milestone should also revisit the `knowledge_prompt_target_count` defaults and planning heuristics so the live default more closely matches model-friendly packet sizes. Add focused planner tests under `tests/llm/` that prove packet splitting behavior on synthetic mixed-material sequences. Update `docs/10-llm/10-llm_README.md` and `cookimport/llm/README.md` so they describe the new planning philosophy in plain language.

Acceptance for this milestone is planner tests that prove obvious mixed-material seams split into separate packets and that packet sizes shrink materially under the default settings without losing exact owned-block coverage.

## Milestone 5: Tighten knowledge hint utility after packet and pass redesign

At the end of this milestone, the knowledge hint file will feel like a compact local debugger sidebar instead of a second prose prompt. This milestone is already scoped by `docs/plans/2026-03-29_15.24.47-tighten-knowledge-hints-for-worker-utility.md` and should be executed after Milestones 3 and 4 so the hint reflects the redesigned packet and pass surfaces rather than the current ones.

The work belongs in `cookimport/llm/knowledge_stage/recovery.py` and the focused tests in `tests/llm/`. Preserve the sections that add real local signal, such as shard interpretation, decision policy, example pointers, and attention rows. Trim any repeated prompt-like semantic reminders that no longer help after the pass and packet redesign.

Acceptance for this milestone is a rendered hint that a human can scan in seconds to understand packet shape, packet risk, and likely failure mode, plus tests that keep the hint from drifting back into verbose prompt duplication.

## Milestone 6: Lock the new contracts in docs and offline proof surfaces

At the end of this milestone, the repo’s current docs and tests will tell one coherent story about the new LLM scaffolding strategy. A future contributor should not need chat history to understand why deterministic code is deliberately weak on semantics and strong on packaging and validation.

Update the current docs in:

- `docs/04-parsing/04-parsing_readme.md`
- `docs/10-llm/10-llm_README.md`
- `cookimport/llm/README.md`

Each doc should describe the live contract in current tense and explain the same ownership boundary: the repo packages evidence, controls phases, and validates coverage; the model owns the fuzzy semantic judgment. Keep the notes short and behavioral. Do not widen docs into design essays.

The offline proof surface for this milestone is the combined focused test suite. Because live Codex-backed runs need approval, the practical “green bar” for this plan is a set of tests that prove payload shape, phase handoff, packet planning behavior, and output normalization. When implementation reaches a natural stopping point, this ExecPlan should also be updated with an honest summary of what still requires live evaluation after user approval.

## Plan of Work

Begin with the narrowest high-signal fix, the missing line-role neighbor context, because it is a clear runtime bug against the current transport contract and does not depend on any larger architectural change. Keep that milestone atomic: payload builders, planning, tests, cache version, and one short parsing-doc note.

After that, change line-role shard planning defaults so workers receive smaller local review units. Keep this planning work mechanical and test-driven. Do not add new deterministic cookbook semantics while doing it. If a proposed change feels like it is deciding what a line means rather than how to package a line for review, it does not belong.

Then move to the knowledge stage. First implement the pass-ownership redesign so Pass 1 and Pass 2 stop teaching the model fake repo-authored semantics. Only after that redesign is in place should you revisit packet planning, because the packetizer should be optimized for the new, honest worker contract rather than the old scaffold-heavy one.

Once the pass contract is fixed, rework knowledge packet planning to shrink and de-mix shards using only mechanically observable boundaries. Follow that by tightening hint utility so the hint reflects the smaller, cleaner packet shape. Keep the hint as a local calibration sidecar, not a second prompt.

Throughout implementation, add or update focused tests before relying on artifact inspection. The tests are the safe proof surface that does not require token spend. They should prove each behavior change directly: neighbor context emission, shard-size planning, pass-ledger shape, pass handoff, packet splitting, and final installed output normalization.

Finally, bring the docs current. Update only the live docs that would otherwise lie after the change. This plan is explicitly not asking for a new benchmark framework or a new grand theory document. The repo already has enough documentation surfaces; they need to reflect the new reality, not multiply.

## Validation and Acceptance

1. The line-role worker payload includes immediate boundary neighbor rows when available and the focused parsing tests prove first, middle, and last shard behavior.
2. Line-role shards are materially smaller or more local under the default planning path, and focused tests prove the new planning behavior without relying on live LLM runs.
3. The knowledge Pass 1 ledger no longer teaches a repo-authored semantic default of universal `other`.
4. The knowledge Pass 2 surface is built only from accepted Pass 1 knowledge rows and is framed as continuation over accepted prior work.
5. The knowledge packet planner produces smaller, more coherent packets by using only mechanical planning signals, and focused tests prove the new split behavior.
6. Knowledge hints remain present but become shorter and more local, with tests guarding against a drift back into second-prompt verbosity.
7. Final installed outputs for knowledge still validate exact coverage and exact kept-block grouping coverage.
8. Current docs describe the same runtime ownership boundary the code now implements.

Behavioral acceptance for a human reviewer is this:

- A future line-role worker input JSON should visibly contain boundary context and should not force a worker to classify a shard edge in a vacuum.
- A future knowledge worker root should visibly show that the model owns the semantic `knowledge|other` call and that Pass 2 builds on accepted Pass 1 work instead of restarting the task.
- A future knowledge packet should look more like one coherent review unit and less like a giant mixed-material dump.

This plan does not require live benchmark numbers for acceptance because the project instructions prohibit live Codex-backed runs without explicit user approval. If those approvals arrive later, the first post-implementation live validation step should be a single-book paired benchmark with careful artifact review rather than an unbounded multi-run tuning spree.

## Idempotence and Recovery

Each milestone in this plan should be landable and reversible independently. Payload-shape fixes, planning changes, pass-ledger redesigns, and hint cleanups should be implemented as discrete atomic states, each with focused proof. If one milestone goes wrong, revert that milestone’s bounded diff and return to the previous coherent, tested state.

Do not recover from a failing milestone by slipping in stronger deterministic semantic guesses. That would violate the user’s stated design goal and would make the repo harder to reason about. The correct recovery path is either:

- shrink the mechanical change and add a narrower focused test, or
- split the work into a follow-up milestone while keeping the current milestone green

Cache-version bumps should accompany any model-facing payload change that could otherwise cause stale results to masquerade as the new behavior. Regenerating derived worker roots in tests is safe. No destructive data migration is required by this plan.
