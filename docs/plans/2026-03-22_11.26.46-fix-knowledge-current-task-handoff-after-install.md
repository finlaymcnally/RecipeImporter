# Fix Knowledge Current-Task Handoff After Install

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a knowledge workspace worker that successfully runs `complete-current`, `check-current`, and `install-current` for task 1 should immediately see task 2 as the new authoritative current task inside the same long-lived session. The worker-visible files `current_task.json`, `CURRENT_TASK.md`, and `CURRENT_TASK_FEEDBACK.md` must all advance together, and the worker must stay alive long enough to reopen them and continue. The user-visible proof is simple: in a regenerated knowledge worker root and in a later user-approved benchmark rerun, the first successful install should no longer be the end of the session. The same worker should validate multiple sequential packets before exiting.

This plan exists because the larger single-task-authority redesign landed only halfway. The runtime now enforces task-local validation and no longer lets workers declare success just by spraying files, but the first benchmark after that change showed a new failure mode: each worker completed exactly one task, the controller internally counted that validation, yet the worker-visible current-task sidecars stayed stuck on task 1 and the session exited. That means the sidecar handoff seam, not the helper semantics, is now the main blocker.

## Outcomes & Retrospective

This plan is not implemented yet. Its goal is to turn the current partial success into a durable one. Right now the runtime can get workers onto the repo-owned helper path, but it cannot yet keep them moving after the first successful install. If this plan succeeds, the next benchmark should show a worker validating multiple sequential packets in one session, and the worker-visible sidecars should always agree with the controller’s internal queue state.

The main risk is fixing only the visible files while missing the deeper exit trigger. The current evidence suggests two coupled problems: stale sidecars and a worker that believes it is done after task 1. The implementation must therefore prove both state continuity and session continuity.

## Context and Orientation

The knowledge stage lives primarily in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, `cookimport/llm/codex_exec_runner.py`, and `cookimport/llm/knowledge_workspace_tools.py`. The orchestrator builds worker roots, writes task files, validates outputs, and decides whether a worker session succeeded. The direct-exec runner mirrors a repo-owned worker folder into a sterile execution workspace, runs the long-lived model session there, and synchronizes selected files between the two roots. The helper module owns the repo-written task-sidecar text and the `tools/knowledge_worker.py` script rendered into the worker root.

In this repository, a “worker root” is the durable artifact folder under `data/output/.../knowledge/workers/worker-XXX`. The “execution workspace” is the temporary mirrored folder under `~/.codex-recipe/recipeimport-direct-exec-workspaces/...` that the live model actually sees. A “runtime control path” is a file like `current_task.json` or `CURRENT_TASK.md` that the repo may rewrite while the worker session is still running. A “handoff” is the moment right after a successful `install-current` when the repo should advance from task N to task N+1 and the worker should reopen the new current-task sidecars.

The current redesign already added the important ingredients: skinny `assigned_tasks.json`, `current_task.json`, `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`, helper verbs `current`, `complete-current`, `check-current`, `install-current`, `next`, and `explain-failure`, plus a queue controller that tracks validated tasks. The benchmark at `data/output/2026-03-22_04.26.00/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-22_04.26.33/raw/llm/saltfatacidheatcutdown/knowledge/` shows that the worker now uses the helper and validates task 1. The failure is that the worker-visible sidecars do not move to task 2, even though `live_status.json` says the controller did.

That means the likely bug is at one of three seams. First, the orchestrator may be updating source-side control files but then letting execution-side stale copies sync back over them. Second, the queue controller may be advancing internal counters without rewriting the persisted sidecars correctly. Third, even if the sidecars advance correctly, the worker may still exit because its final-message posture treats “one install succeeded” as terminal rather than as “reopen current-task files now.” The implementation must inspect and prove which seam is actually wrong.

The files that matter most for this fix are:

- `cookimport/llm/codex_farm_knowledge_orchestrator.py`, especially `_KnowledgeWorkspaceTaskQueueController`, `_run_knowledge_workspace_worker_assignment_taskized_v1`, and `_build_strict_json_watchdog_callback`.
- `cookimport/llm/codex_exec_runner.py`, especially `_DIRECT_EXEC_RUNTIME_CONTROL_PATHS`, `_populate_direct_exec_workspace`, `_sync_direct_exec_workspace_paths`, and `_wrap_workspace_supervision_callback`.
- `cookimport/llm/knowledge_workspace_tools.py`, especially `write_current_task_sidecars`, the rendered `tools/knowledge_worker.py`, and any helper text that should explicitly tell the worker to continue after install.
- `tests/llm/test_codex_farm_knowledge_orchestrator.py` and `tests/llm/test_codex_exec_runner_workspace.py`, which already cover the single-task contract and should grow to cover the broken handoff.

## Plan of Work

The first milestone is a local reproduction that does not require a benchmark rerun. Build or extend a narrow orchestrator test that simulates a multi-task knowledge worker session where the first task validates and installs successfully. The test must prove the current bug before any code changes: the queue controller reports task 2 internally, but the persisted worker-root files still show task 1 or the worker exits before observing task 2. This matters because the fix must be pinned to a deterministic, cheap test rather than to a saved benchmark artifact only.

The second milestone is source-of-truth analysis. Read the current synchronization flow in `cookimport/llm/codex_exec_runner.py` carefully and document exactly when control files are copied source-to-execution and when output or scratch files are copied execution-to-source. Determine whether `current_task.json`, `CURRENT_TASK.md`, and `CURRENT_TASK_FEEDBACK.md` are being rewritten only in the source worker root, only in the execution workspace, or in both places at different times. If the stale sidecars are being synced back from the execution workspace after the orchestrator advances them in the source root, the fix should make those files one-way authoritative during the session. If the controller is not actually rewriting the sidecars, the fix belongs in the orchestrator instead.

The third milestone is an atomic handoff implementation. The repository must rewrite all current-task sidecars together when task N becomes validated. That means `current_task.json`, `CURRENT_TASK.md`, and `CURRENT_TASK_FEEDBACK.md` for task N+1 must describe the same task and must stay in sync in both the durable worker root and the sterile execution workspace. The implementation should treat them as one logical bundle. If necessary, add a helper function that writes all three files plus any task-sequence metadata in one place, and make the queue controller call only that helper. Do not leave the handoff split across several call sites.

The fourth milestone is worker-session continuity. A successful `install-current` must no longer read like the end of the job. Update the rendered helper and/or the knowledge workspace prompt so that the natural next step after install is explicit: reopen `CURRENT_TASK.md`, `current_task.json`, and `CURRENT_TASK_FEEDBACK.md`, then continue if the queue is not complete. The worker should not need to infer whether the repo advanced. If helpful, make `install-current` print a short next-step hint such as “re-open current-task files; the repo may have advanced the queue.” If the queue is complete, it may still say so directly.

The fifth milestone is regression protection. Add tests that prove a worker can complete at least two sequential tasks in one session, that the persisted worker-root current-task bundle advances from task 1 to task 2 after a successful install, and that stale task-1 sidecars cannot overwrite task-2 state later in the same run. Also add a test that the final worker message or helper output after install does not sound terminal when the queue still has remaining tasks.

The sixth milestone is documentation and optional benchmark proof. Update the short LLM docs so they describe the stronger handoff contract, then, only if the user explicitly approves it, rerun a narrow benchmark slice. Acceptance on the benchmark is not merely “the tools appear.” It is “the same worker validates multiple sequential tasks before exit, and `missing_output` no longer dominates because the session stopped after the first install.”

## Validation and Acceptance

This work is accepted only when the handoff is observable, not merely when more code exists.

At the unit-test level, a simulated multi-task worker session must prove that after task 1 validates and installs, the persisted `current_task.json`, `CURRENT_TASK.md`, and `CURRENT_TASK_FEEDBACK.md` all describe task 2 and remain that way. A stale execution-workspace copy of task-1 sidecars must not be able to overwrite task-2 state in the durable worker root.

At the helper-contract level, a worker that runs `install-current` for a non-final task must be told to reopen the current-task bundle and continue. The helper and prompt must no longer make a successful single install sound like the end of the assignment unless the queue is complete.

At the phase-runtime level, a worker that succeeds on task 1 must be able to validate task 2 in the same session without spawning a new worker assignment. If the queue is still incomplete, the final reason must reflect the real stop cause, not a stale sidecar mismatch or a misleading “task completed” close-out.

At the optional benchmark level, acceptance means more than one validated task per worker. The `2026-03-22_04.26.00` pattern of exactly one validated task per worker and 394 missing outputs must be materially improved.

## Idempotence and Recovery

The sidecar writer must be safe to call repeatedly. Rewriting the current-task bundle for task 2 more than once should produce the same files. Re-running a successful install for task 1 should either be a no-op or deterministically leave the queue in the same advanced state, not oscillate between task 1 and task 2.

If the bug turns out to be caused by source/execution sync ordering, prefer changing the sync contract so current-task sidecars are one-way authoritative from source to execution during live runs. That is safer than trying to “repair” stale files after they overwrite good state. If the bug turns out to be primarily prompt/helper-driven, keep the state fix anyway; the runtime should not depend on ideal worker behavior to preserve correct current-task files.

As with the larger redesign, benchmark reruns remain optional and must not happen without explicit user approval.

## Interfaces and Dependencies

In `cookimport/llm/knowledge_workspace_tools.py`, the following interfaces must remain stable and may need to grow:

- `write_current_task_sidecars(...)` should remain the single authoritative writer for `current_task.json`, `CURRENT_TASK.md`, and `CURRENT_TASK_FEEDBACK.md`.
- The rendered `tools/knowledge_worker.py` should keep `complete-current`, `check-current`, and `install-current`, but `install-current` may need to print or expose a clearer “reopen current-task files now” continuation hint.

In `cookimport/llm/codex_farm_knowledge_orchestrator.py`, `_KnowledgeWorkspaceTaskQueueController` should become the only place that advances the current task after validation. If multiple sites can rewrite the same current-task files, reduce that to one site.

In `cookimport/llm/codex_exec_runner.py`, `_DIRECT_EXEC_RUNTIME_CONTROL_PATHS` and the workspace sync wrapper must preserve the source-of-truth direction for current-task sidecars. If a file can be rewritten by the repo during a live session, the execution workspace should consume that state, not overwrite it.
