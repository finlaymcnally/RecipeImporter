# Fix Knowledge Current-Task Handoff After Install

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a knowledge workspace worker that successfully runs `complete-current`, `check-current`, and `install-current` for task 1 must continue to task 2 inside the same long-lived session. The worker-visible files `current_task.json`, `CURRENT_TASK.md`, and `CURRENT_TASK_FEEDBACK.md` must all advance together, and the worker-facing prompt and helper text must clearly say that a successful install is a handoff point, not the end of the assignment. The user-visible proof is simple: in a regenerated worker root, the first successful install should be followed by a second successful task in the same session, and the later benchmark rerun should no longer show the one-task-per-worker collapse.

This plan exists because the larger single-task-authority redesign landed only halfway. The runtime now writes a full ordered queue to `assigned_tasks.json` and tracks live validation per task, but the first benchmark after that change showed a split contract. The queue controller internally advanced to task 2, while the durable current-task sidecars stayed on task 1, and the prompt still told the worker that `assigned_tasks.json` was only a progress view and that it should write one result for the current task only. The fix therefore has to close both seams: state handoff and worker contract.

  Evidence: in `data/output/2026-03-22_04.26.00/.../knowledge/workers/worker-001/events.jsonl` and sibling worker event logs, the agents read `CURRENT_TASK.md`, ran `python3 tools/knowledge_worker.py complete-current`, then `check-current`, then `install-current`, and the task ledger shows exactly one validated task per worker.

- Observation: the queue controller advanced internally, but the worker-visible sidecars did not advance with it.
  Evidence: `worker-001/live_status.json` reports `current_task_id: saltfatacidheatcutdown.ks0000.nr.task-002` and `queue_validated_task_count: 1`, while `worker-001/current_task.json`, `worker-001/CURRENT_TASK.md`, and `worker-001/CURRENT_TASK_FEEDBACK.md` still describe task 1.

- Observation: the ordered queue file itself was already good enough to support sequential work as a reference surface.
  Evidence: `worker-001/assigned_tasks.json` contains the expected `metadata.input_path`, `metadata.hint_path`, `metadata.result_path`, `lease_sequence`, and `lease_total` for each task row. The queue did not fail because the worker lacked per-task file paths.

- Observation: the benchmark now fails for a different reason than the earlier batch-scripting collapse.
  Evidence: `knowledge/stage_status.json` for `2026-03-22_04.26.00` reports `validated: 5` and `missing_output: 394`, not the older mass semantic invalidity pattern. Each worker wrote one output and then exited with `workspace_validated_task_queue_incomplete`.

- Observation: the main worker prompt still encodes the older current-task-only worldview, even though the runtime now expects a full queue to be processed in one session.
  Evidence: `_build_knowledge_workspace_worker_prompt(...)` in `cookimport/llm/codex_farm_knowledge_orchestrator.py` still says `assigned_tasks.json` is “only the ordered queue/progress view”, says the repo will mutate the current-task files after acceptance, and says to “Write one completed semantic packet result file for the current task only.”

- Observation: the helper feedback and worker close-out language still read as terminal after a successful first install.
  Evidence: `worker-001/CURRENT_TASK_FEEDBACK.md` says “You may run `python3 tools/knowledge_worker.py install-current` to write the final result path,” and `worker-001/last_message.json` ends with “Task completed and installed successfully,” with no explicit “re-open the new current-task bundle and continue” instruction.

- Observation: the actual state-loss bug was directional, not an absent write in the queue controller.
  Evidence: the new regression in `tests/llm/test_codex_exec_runner_workspace.py::test_workspace_supervision_pushes_advanced_current_task_bundle_back_to_execution_root` failed before the fix because `_wrap_workspace_supervision_callback(...)` copied runtime control paths execution-to-source before the callback, but never mirrored the advanced source-side bundle back into the execution workspace afterward.

## Outcomes & Retrospective

The runtime now keeps the repo-owned current-task bundle authoritative through the live handoff: `out/` and `scratch/` still sync execution-to-source before validation, the queue controller still advances the current task in the durable worker root, and the direct-exec runner now mirrors the updated runtime control bundle back into the sterile execution workspace after that validation callback. ## Context and Orientation

The knowledge stage lives primarily in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, `cookimport/llm/codex_exec_runner.py`, and `cookimport/llm/knowledge_workspace_tools.py`. The orchestrator builds worker roots, writes task files, validates outputs, and decides whether a worker session succeeded. The direct-exec runner mirrors a repo-owned worker folder into a sterile execution workspace, runs the long-lived model session there, and synchronizes selected files between the two roots. The helper module owns the repo-written task-sidecar text and the `tools/knowledge_worker.py` script rendered into the worker root.

In this repository, a “worker root” is the durable artifact folder under `data/output/.../knowledge/workers/worker-XXX`. The “execution workspace” is the temporary mirrored folder under `~/.codex-recipe/recipeimport-direct-exec-workspaces/...` that the live model actually sees. A “runtime control path” is a file like `current_task.json` or `CURRENT_TASK.md` that the repo may rewrite while the worker session is still running. A “handoff” is the moment right after a successful `install-current` when the repo should advance from task N to task N+1 and the worker should reopen the new current-task sidecars.

The current redesign already added the important ingredients: skinny `assigned_tasks.json`, `current_task.json`, `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`, helper verbs `current`, `complete-current`, `check-current`, `install-current`, `next`, and `explain-failure`, plus a queue controller that tracks validated tasks. The benchmark at `data/output/2026-03-22_04.26.00/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-22_04.26.33/raw/llm/saltfatacidheatcutdown/knowledge/` shows that the worker now uses the helper and validates task 1. The failure is that the durable current-task sidecars do not move to task 2, even though `live_status.json` says the controller did, and the prompt still teaches the older “current task only” contract.

That means the bug is no longer a broad mystery. There are two concrete seams to inspect and fix. First, the orchestrator or direct-exec sync layer may be letting stale task-1 sidecars remain durable after the controller advances to task 2. Second, the worker-facing contract in `_build_knowledge_workspace_worker_prompt(...)`, `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`, and the helper CLI output still tells the model to think in one-current-task terms and gives it no affirmative continuation instruction after install. The implementation must fix both so the runtime state and the worker’s mental model agree while preserving the repo-owned current-task bundle as the authoritative current step.

The files that matter most for this fix are:

- `cookimport/llm/codex_farm_knowledge_orchestrator.py`, especially `_KnowledgeWorkspaceTaskQueueController`, `_run_knowledge_workspace_worker_assignment_taskized_v1`, and `_build_strict_json_watchdog_callback`.
- `cookimport/llm/codex_exec_runner.py`, especially `_DIRECT_EXEC_RUNTIME_CONTROL_PATHS`, `_populate_direct_exec_workspace`, `_sync_direct_exec_workspace_paths`, and `_wrap_workspace_supervision_callback`.
- `cookimport/llm/knowledge_workspace_tools.py`, especially `write_current_task_sidecars`, `render_current_task_brief_text(...)`, `render_current_task_feedback_text(...)`, the rendered `tools/knowledge_worker.py`, and any helper text that should explicitly tell the worker to continue after install.
- `tests/llm/test_codex_farm_knowledge_orchestrator.py` and `tests/llm/test_codex_exec_runner_workspace.py`, which already cover the single-task contract and should grow to cover the broken handoff.

## Plan of Work

The first milestone is a local reproduction that does not require a benchmark rerun. Build or extend a narrow orchestrator or helper test that simulates a multi-task knowledge worker session where the first task validates and installs successfully. The test must prove the current bug before any code changes: the queue controller reports task 2 internally, but the persisted worker-root files still show task 1, or the worker contract still tells the model to stop after task 1. This matters because the fix must be pinned to a deterministic, cheap test rather than to a saved benchmark artifact only.

The second milestone is source-of-truth analysis. Read the current synchronization flow in `cookimport/llm/codex_exec_runner.py` carefully and document exactly when control files are copied source-to-execution and when output or scratch files are copied execution-to-source. Determine whether `current_task.json`, `CURRENT_TASK.md`, and `CURRENT_TASK_FEEDBACK.md` are being rewritten only in the source worker root, only in the execution workspace, or in both places at different times. If the stale sidecars are being synced back from the execution workspace after the orchestrator advances them in the source root, the fix should make those files one-way authoritative during the session. If the controller is not actually rewriting the sidecars, the fix belongs in the orchestrator instead.

The third milestone is an atomic handoff implementation. The repository must rewrite all current-task sidecars together when task N becomes validated. That means `current_task.json`, `CURRENT_TASK.md`, and `CURRENT_TASK_FEEDBACK.md` for task N+1 must describe the same task and must stay in sync in both the durable worker root and the sterile execution workspace. The implementation should treat them as one logical bundle. `write_current_task_sidecars(...)` in `cookimport/llm/knowledge_workspace_tools.py` already exists for this role; the implementation should tighten the write ownership around that function rather than creating another parallel writer. Do not leave the handoff split across several call sites, and do not substitute queue-first worker behavior for a real sidecar handoff fix.

The fourth milestone is worker-session continuity. A successful `install-current` must no longer read like the end of the job. Update `_build_knowledge_workspace_worker_prompt(...)`, `render_current_task_brief_text(...)`, `render_current_task_feedback_text(...)`, and the rendered `tools/knowledge_worker.py` output so that the natural next step after install is explicit: reopen `CURRENT_TASK.md`, `current_task.json`, and `CURRENT_TASK_FEEDBACK.md`, then continue if the queue is not complete. The prompt should describe `assigned_tasks.json` as the ordered queue and progress reference, not as the thing the worker should spelunk instead of the current-task bundle. The worker should not need to infer whether the repo advanced.

The fifth milestone is regression protection. Add tests that prove a worker can complete at least two sequential tasks in one session, that the persisted worker-root current-task bundle advances from task 1 to task 2 after a successful install, and that stale task-1 sidecars cannot overwrite task-2 state later in the same run. Add a prompt or helper-surface regression that fails if the main worker contract again says `assigned_tasks.json` is inventory-only or that one installed task should end the session.

The sixth milestone is documentation and optional benchmark proof. Update the short LLM docs so they describe the stronger handoff contract, then, only if the user explicitly approves it, rerun a narrow benchmark slice. Acceptance on the benchmark is not merely “the tools appear.” It is “the same worker validates multiple sequential tasks before exit, and `missing_output` no longer dominates because the session stopped after the first install.”

## Validation and Acceptance

This work is accepted only when the handoff is observable, not merely when more code exists.

At the unit-test level, a simulated multi-task worker session must prove that after task 1 validates and installs, the persisted `current_task.json`, `CURRENT_TASK.md`, and `CURRENT_TASK_FEEDBACK.md` all describe task 2 and remain that way. A stale execution-workspace copy of task-1 sidecars must not be able to overwrite task-2 state in the durable worker root.

At the sync-contract level, the same post-install handoff must be visible inside the live execution workspace before the worker continues. Acceptance is not just “the durable root eventually says task 2.” It is “the same running session sees task 2 as the new current task, and later execution-to-source sync cannot drag task 1 back.”

At the helper-contract level, a worker that runs `install-current` for a non-final task must be told to reopen the current-task bundle and continue. The helper and prompt must no longer make a successful single install sound like the end of the assignment unless the queue is complete.

At the phase-runtime level, a worker that succeeds on task 1 must be able to validate task 2 in the same session without spawning a new worker assignment. If the queue is still incomplete, the final reason must reflect the real stop cause, not a stale sidecar mismatch or a misleading “task completed” close-out. A fake-runner regression should make this observable without paying for a live benchmark.

At the optional benchmark level, acceptance means more than one validated task per worker. The `2026-03-22_04.26.00` pattern of exactly one validated task per worker and 394 missing outputs must be materially improved.

## Idempotence and Recovery

The sidecar writer must be safe to call repeatedly. Rewriting the current-task bundle for task 2 more than once should produce the same files. Re-running a successful install for task 1 should either be a no-op or deterministically leave the queue in the same advanced state, not oscillate between task 1 and task 2.

If the bug turns out to be caused by source/execution sync ordering, prefer changing the sync contract so current-task sidecars are one-way authoritative from source to execution during live runs. That is safer than trying to “repair” stale files after they overwrite good state. If the bug turns out to be primarily prompt/helper-driven, keep the state fix anyway; the runtime should not depend on ideal worker behavior to preserve correct current-task files.

As with the larger redesign, benchmark reruns remain optional and must not happen without explicit user approval.
