## Structured Restatement Of User Desire

The knowledge stage must stop using deterministic code to make semantic accept-or-reject decisions about `knowledge` versus `other`. Deterministic code is allowed to validate only mechanical facts: ownership, exact row coverage, allowed enum values, required grouping structure, and complete output shape. It must not emit heuristic warnings, must not run semantic suspicion audits, and must not block install because prose heuristics disagree with the LLM. This is burn-the-boats work. The old semantic gate must be deleted rather than preserved behind compatibility branches. The repository should end this plan with one clean semantic boundary: the LLM makes the fuzzy call, deterministic code checks structure, and runtime/docs/artifacts all describe that truth clearly.

## Purpose / Big Picture

After this change, a user can run RecipeImport on an arbitrary cookbook and trust that the knowledge stage is doing real semantic work instead of being second-guessed by deterministic heuristics. The LLM will decide which review-eligible outside-recipe rows are durable cooking knowledge and how the kept rows group together. Deterministic code will still matter, but only where it is actually trustworthy: building inputs, validating coverage and shape, recording transport/schema failures, and promoting accepted outputs.

The visible proof for this plan is semantic-authority purity. The live knowledge path no longer emits semantic suspicion audits, no longer tells the worker to repair heuristic disagreements, and no longer rejects a structurally valid output just because repo heuristics would have preferred a different semantic label. Transport simplification, packet-leasing, and cost reduction are intentionally deferred to the later Codex-exec runtime plan. This plan is about authority, not economics.

## Context and Orientation

The knowledge stage is the last semantic owner of outside-recipe prose. Upstream line-role and routing logic decide which text is recipe-local, which text is obvious junk that can be excluded from review, and which text remains review-eligible outside-recipe material. The knowledge stage then decides whether those remaining rows are durable cooking knowledge or just `other`. In this repository, “durable cooking knowledge” means reusable cooking leverage such as technique, cause-and-effect explanation, troubleshooting, substitution, storage or safety guidance, sensory explanation, or similar prose that materially helps future cooking decisions.

The current live knowledge pipeline is spread across several files:

- [cookimport/llm/codex_farm_knowledge_orchestrator.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_orchestrator.py), which owns the public runtime entry point for knowledge review.
- [cookimport/llm/codex_farm_knowledge_jobs.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_jobs.py), which builds review shard inputs.
- [cookimport/llm/knowledge_prompt_builder.py](/home/mcnal/projects/recipeimport/cookimport/llm/knowledge_prompt_builder.py), which builds the knowledge prompt body and follow-up payloads.
- [cookimport/llm/codex_farm_knowledge_ingest.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_ingest.py), which validates and normalizes model outputs.
- [cookimport/llm/knowledge_stage/](/home/mcnal/projects/recipeimport/cookimport/llm/knowledge_stage), which contains runtime, recovery, promotion, and reporting code.
- [cookimport/llm/knowledge_phase_workspace_tools.py](/home/mcnal/projects/recipeimport/cookimport/llm/knowledge_phase_workspace_tools.py), which currently contains the semantic suspicion machinery and repo-written helper surfaces.
- [cookimport/runs/stage_observability.py](/home/mcnal/projects/recipeimport/cookimport/runs/stage_observability.py), which builds stage summaries and attention counters from runtime artifacts.

Two terms matter here.

“Semantic authority” means the accepted LLM semantic decision is the truth for `knowledge` versus `other` and idea grouping. Deterministic code may reject malformed or incomplete output, but it may not silently overrule a structurally valid semantic answer.

“Structural validation” means deterministic checks that do not require semantic judgment: exact owned-row coverage, allowed enum values, valid grouping references, complete output shape, and similar mechanical facts.

This plan assumes the upstream route-first non-recipe contract already exists when implementation begins. This plan does not redefine routing labels. It defines only how the knowledge stage should respect the semantic boundary once it receives review-eligible rows.

## Plan Of Work

The first part of the work is to delete the live semantic suspicion machinery. Remove the pass-1 semantic audit builder, semantic-repair payloads, heuristic warning sidecars, and any acceptance logic that reclassifies structurally valid outputs as unresolved because of heuristic prose rules. The key invariant is simple: no live code is allowed to ask “does this prose look memoir-like enough to reject the model’s label?”

The second part is to make the validation boundary explicit in the ingest and recovery layers. The authoritative validation logic in [cookimport/llm/codex_farm_knowledge_ingest.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_ingest.py) and the knowledge recovery code must accept every output that satisfies the structural contract, even when repo heuristics would have preferred a different semantic answer. Retry or repair feedback may mention missing rows, missing groups, malformed JSON, or invalid enum values. It may not contain prose heuristics or semantic nudges.

The third part is prompt and artifact cleanup. The prompt builders, helper guidance, and follow-up payloads must stop teaching the worker that repo heuristics are semantic co-authors. If a live retry exists after this plan, it must describe only the structural contract violation. Prompt artifacts and worker-facing docs must not mention semantic suspicion, memoir-like versus guidance-like repair categories, or similar repo semantic advice.

The fourth part is observability cleanup. Stage summaries, benchmark helper aggregation, and docs must stop presenting the knowledge stage as a deterministic-audit workflow. They should describe the real semantic boundary plainly: deterministic code packages evidence and validates shape; the LLM decides semantics.

The fifth part is explicit scoping discipline. Do not use this plan to redesign the live Codex-exec transport, packet budgeting, shell policy, or final terminal-reason taxonomy. Those changes belong to the later knowledge-runtime plans. If implementation needs a small supporting transport edit to remove semantic hints from the live path, that is acceptable. A full transport reset is not.

## Milestones

### Milestone 1: Delete deterministic semantic gatekeeping from the live knowledge path

At the end of this milestone, no live knowledge runtime code can emit semantic-suspicion outcomes, semantic-audit files, or heuristic warning artifacts as part of normal execution. A repository search should show the old semantic suspicion vocabulary only in historical docs or deleted code, not in the live runtime.

### Milestone 2: Make every live retry or follow-up mechanically phrased

At the end of this milestone, any live follow-up path that survives the rewrite talks only about shape failures such as missing rows, missing groups, malformed JSON, or invalid enum values. It never says that a row “looks memoir-like” or “should really be other.” The proof is that retry fixtures and prompt artifacts contain mechanical feedback only.

### Milestone 3: Rebuild summaries and docs around the semantic-authority contract

At the end of this milestone, stage summaries, benchmark helper outputs, and docs can explain the knowledge stage without mentioning heuristic semantic gatekeeping. A novice should be able to read the updated docs and understand that semantic authority belongs to the LLM while deterministic code owns structure only.

### Milestone 4: Leave a clean handoff to the later runtime plans

At the end of this milestone, the live semantic contract is clean enough that the later packet-lease/cost plan can swap transports without reopening semantic-authority decisions. The proof is that the remaining open questions are about runtime shape and reporting details, not about whether deterministic heuristics may judge the model’s semantic labels.

## Validation And Acceptance

Acceptance is behavioral.

The first acceptance criterion is semantic-authority purity. A live knowledge output that covers every owned row, uses only allowed labels, and groups every kept knowledge row correctly must install successfully even if a deterministic prose heuristic would have preferred a different label. Tests must prove this with a structurally valid but heuristic-unfriendly fixture.

The second acceptance criterion is the disappearance of semantic suspicion. Repository search and targeted tests must show that the live runtime no longer emits semantic-suspicion failures, no longer writes semantic-audit payloads, and no longer asks a worker to repair labels because prose “looks memoir-like” or “looks guidance-like.”

The third acceptance criterion is mechanically phrased follow-up behavior. If the live runtime performs a retry or repair after this change, the feedback must describe only shape failures such as missing rows, invalid enum values, malformed JSON, or missing groups. It must never contain semantic advice.

The fourth acceptance criterion is truthful docs and summaries. Stage summaries and docs must explain the knowledge stage without teaching heuristic semantic gatekeeping as part of the happy path.

The fifth acceptance criterion is clean scoping. The resulting code and docs should still leave room for the later packet-lease/cost plan and terminal-reason plan to land without reopening semantic-authority arguments.

## Idempotence And Recovery

This plan should be implemented as one coherent semantic cutover rather than a compatibility migration. The safe order is the milestone order above: delete semantic gatekeeping first, tighten validation and follow-up prompts second, then rewire docs and reporting around the new semantic boundary. Do not stop in a mixed state and add adapters to support both stories.

The commands in `Concrete Steps` are safe to rerun. If narrow pytest runs fail midway through the rewrite, fix the missing producer/consumer pair and rerun the same command. If `.venv` goes stale, reactivate it and rerun `pip install -e .[dev]`. Because this plan deletes the old semantic gate, rollback means reverting the change set, not preserving two live semantic-authority models.
