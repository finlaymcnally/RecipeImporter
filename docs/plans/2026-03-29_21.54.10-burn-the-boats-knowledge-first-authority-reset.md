## Structured Restatement Of User Desire

The knowledge stage must stop using deterministic code to make semantic accept-or-reject decisions about `knowledge` versus `other`. Deterministic code is allowed to validate only mechanical facts: ownership, exact row coverage, allowed enum values, required grouping structure, and complete output shape. It must not emit heuristic warnings, must not run semantic suspicion audits, and must not block install because heuristic prose rules disagree with the LLM about ambiguous cookbook text. This is burn-the-boats work. The bad live path should be deleted rather than preserved behind compatibility or legacy branches. The repository should end with one clean knowledge-stage authority model: the LLM makes semantic judgments, deterministic code validates structure and installs valid outputs, and the runtime plus artifacts tell only that new story.

## Purpose / Big Picture

After this change, a user can run RecipeImport on an arbitrary cookbook and trust that the knowledge stage is doing real semantic work instead of being second-guessed by deterministic heuristics. The LLM will read the review-eligible outside-recipe shards, decide which owned blocks are durable cooking knowledge, group those kept blocks into related ideas, and install that output directly when it satisfies the structural contract. Deterministic code will still matter, but only in the places it is good at: building shard inputs, validating exact coverage and group completeness, recording retries for malformed or missing outputs, and writing benchmark/debug artifacts.

The user-visible proof is threefold. First, the runtime no longer contains a semantic suspicion audit or a same-session semantic repair loop. Second, the knowledge worker surface no longer burns tokens reopening `CURRENT_PHASE*`, `work/*.pass1.json`, and `repair/*.json` files across dozens of local shell commands. Third, benchmark and runtime summaries no longer collapse this class of failure into a vague `missing_output` story when the real cause was repo-owned semantic gatekeeping. A novice should be able to inspect the final docs, worker inputs, output validators, and stage summaries and see one simple rule: semantic labels come from the LLM; deterministic code validates shape only.

## Outcomes & Retrospective

This plan is authored but not yet implemented. The current outcome is clarity about the real bug class and the necessary scope of the fix. The bad run was not simply “the model chose poor labels.” The repository itself imposed deterministic semantic gatekeeping, turned that gatekeeping into an impossible repair loop on real prose, and then hid the reason inside sterile workspaces while charging workspace-session token overhead for the privilege. The intended retrospective after implementation is much simpler: the knowledge stage should either produce one structurally valid semantic shard output or fail for an explicit transport/schema reason, with no heuristic middle layer pretending to know better than the model.

## Context and Orientation

The knowledge stage is the last semantic owner of outside-recipe prose. Upstream line-role and routing logic decide which text is recipe-local, which text is obvious junk that can be excluded from review, and which text remains review-eligible outside-recipe prose. The knowledge stage then decides whether those remaining blocks are durable cooking knowledge or just `other`. In this repository, “durable cooking knowledge” means reusable cooking leverage: technique, cause-and-effect explanation, troubleshooting, substitution, storage or safety guidance, sensory explanation, or similar prose that materially helps future cooking decisions.

The current live knowledge pipeline is spread across several files:

- [cookimport/llm/codex_farm_knowledge_orchestrator.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_orchestrator.py), which owns the public runtime entry point for knowledge review.
- [cookimport/llm/codex_farm_knowledge_jobs.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_jobs.py), which builds review shards under `knowledge/in/*.json`.
- [cookimport/llm/knowledge_prompt_builder.py](/home/mcnal/projects/recipeimport/cookimport/llm/knowledge_prompt_builder.py), which builds the live prompt body.
- [cookimport/llm/codex_farm_knowledge_ingest.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_ingest.py), which validates and normalizes final model outputs.
- [cookimport/llm/codex_farm_knowledge_writer.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_writer.py), which promotes validated shard outputs into final stage artifacts.
- [cookimport/llm/knowledge_stage/](/home/mcnal/projects/recipeimport/cookimport/llm/knowledge_stage), which currently contains planning, workspace runtime, recovery, promotion, and reporting code for the phase-first worker contract.
- [cookimport/llm/knowledge_phase_workspace_tools.py](/home/mcnal/projects/recipeimport/cookimport/llm/knowledge_phase_workspace_tools.py), which currently generates the `CURRENT_PHASE*` files, `work/*.pass1.json`, `repair/*.json`, `tools/knowledge_worker.py`, and the semantic suspicion gate.
- [cookimport/runs/stage_observability.py](/home/mcnal/projects/recipeimport/cookimport/runs/stage_observability.py), which builds runtime summaries and attention counters from stage artifacts.

In the current design, a “shard” is one ordered knowledge packet with owned block ids under `knowledge/in/<shard_id>.json`. A “workspace worker” is a long-lived Codex session that receives a local working directory full of helper files such as `CURRENT_PHASE.md`, `CURRENT_PHASE_FEEDBACK.md`, `work/*.json`, and `repair/*.json`, then edits local ledgers and calls helper commands like `python3 tools/knowledge_worker.py check-phase`. The important problem is that this is not just a transport choice. The workspace contract currently makes repo code an active semantic participant because it performs the semantic suspicion audit after structural validation and blocks install when the heuristics disagree.

The architecture after this plan should be much simpler. The repo still builds deterministic shard inputs and still validates deterministic shape. But the LLM receives one shard payload, returns one structured semantic result, and the repo either accepts it because it satisfies the mechanical contract or rejects it because it is malformed or incomplete. There is no semantic middle layer. There is no second answer source. There is no `pass1` semantic ledger that a helper is allowed to overrule.

This plan assumes the upstream route-first non-recipe contract already exists when implementation begins. That upstream contract determines which lines are review-eligible and which are excluded before the knowledge stage. This plan does not redefine the routing labels. It only defines how the knowledge stage itself should work once it receives review-eligible outside-recipe shards.

## Plan Of Work

The first part of the implementation is a contract reset. Delete the live semantic suspicion machinery from [cookimport/llm/knowledge_phase_workspace_tools.py](/home/mcnal/projects/recipeimport/cookimport/llm/knowledge_phase_workspace_tools.py) and any callers that depend on it. This includes the pass-1 semantic audit builder, the `semantic_suspicion_requires_repair` error path, semantic repair payloads, semantic-audit feedback sections, and any acceptance logic that reclassifies structurally valid outputs as unresolved because of heuristic prose rules. The helper file should either disappear entirely from the happy path or shrink down to shape-only concerns if some tiny subset remains useful. The key invariant is that no live code is allowed to ask “does this prose look memoir-like enough to reject the model’s label?”

The second part is a transport simplification. Replace the phase-first workspace-worker knowledge happy path with a direct structured shard execution path driven by the existing packet schema. The current input files under `knowledge/in/*.json` are already the right level of deterministic packaging: they contain owned block ids, raw text, headings, and other mechanical truth. The direct call should hand that packet to the model together with the live prompt text and require one JSON output object containing `packet_id`, `block_decisions`, and `idea_groups`. This should reuse the authoritative validation logic in [cookimport/llm/codex_farm_knowledge_ingest.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_ingest.py) rather than asking the model to manage a local repair/install protocol.

The third part is a recovery rewrite. Today, recovery means “reopen the local ledger and try to reconcile with the semantic suspicion checker.” That must be deleted. The new recovery ladder should be narrow and mechanical. If the output is missing, malformed JSON, or structurally invalid, the repo may perform one or more bounded structured retries that tell the model exactly which mechanical contract was violated. Those retries must not contain any deterministic semantic hints. A retry may say “you omitted block 42” or “knowledge block 17 is missing a group,” but it may not say “block 42 looks memoir-like and should be `other`.” If repeated structured retries still fail, the shard should finish with an explicit malformed/missing-output reason, not with a fake semantic authority story.

The fourth part is observability and artifact cleanup. Runtime summaries, benchmark prompt budgets, worker telemetry, and stage attention counters must all be updated to reflect the new execution mode. The summary vocabulary should distinguish direct structured calls, structured retries, malformed outputs, and missing outputs. It should stop talking about semantic suspicion, pass-1 repair, or same-session semantic recovery because those concepts will no longer exist. The benchmark-side `prompt_budget_summary.json` should report the direct knowledge path accurately, and the stage summary should no longer depend on artifacts that only exist in sterile worker workspaces.

The fifth part is test and documentation cleanup. Remove tests that encode semantic suspicion behavior as desirable. Rewrite tests so they prove the new rule: semantic outputs are accepted when structurally valid, even if deterministic heuristics would have disliked them. Update [docs/10-llm/10-llm_README.md](/home/mcnal/projects/recipeimport/docs/10-llm/10-llm_README.md), [docs/04-parsing/04-parsing_readme.md](/home/mcnal/projects/recipeimport/docs/04-parsing/04-parsing_readme.md), and [docs/07-bench/07-bench_README.md](/home/mcnal/projects/recipeimport/docs/07-bench/07-bench_README.md) so a novice can understand the new knowledge-stage story without reading historical plans or logs. The docs must make the authority boundary explicit: deterministic code packages evidence and validates shape; the LLM decides `knowledge` versus `other`.

## Validation And Acceptance

Acceptance is behavioral, not just structural.

The first acceptance criterion is semantic-authority purity. A live knowledge shard output that covers every owned block, uses only allowed labels, and groups every kept knowledge block exactly once must install successfully even if a deterministic prose heuristic would have preferred a different label. Tests must prove this by feeding the validator a semantically valid but heuristic-unfriendly output and showing that it passes because the structure is correct.

The second acceptance criterion is the disappearance of semantic suspicion. Repository search and targeted tests must show that the live runtime no longer emits `semantic_suspicion_requires_repair`, no longer writes semantic-audit payloads, and no longer asks a worker to repair labels because prose “looks memoir-like” or “looks guidance-like.”

The third acceptance criterion is transport simplification. Prompt logs and runtime telemetry for the knowledge happy path must show direct structured shard calls, not workspace-worker sessions with dozens of shell commands. Tests should prove that the prompt builder and orchestrator can execute knowledge review without scaffolding `CURRENT_PHASE*`, `repair/*.json`, or `work/*.pass1.json`.

The fourth acceptance criterion is truthful observability. A stage summary built from the repo-stage artifacts alone must correctly report why a bad knowledge run failed. If the model returns malformed JSON, the summary must report malformed output. If no payload arrives, the summary must report missing output. It must not need hidden sterile workspace files to explain the failure.

The fifth acceptance criterion is benchmark-budget clarity. Prompt-budget summaries must reflect the new direct execution mode and should no longer report knowledge as one giant workspace-worker shell-churn stage. Existing fixture-based artifact tests should prove that the summary vocabulary and counters match the new contract.
