# Minimize Line-Role Codex Payloads To Need-To-Know Data

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

Canonical line-role already uses the healthier file-backed transport, but the Food Lab preview shows that the model-facing task files are still much larger than the underlying book text because they carry rich local metadata that Codex does not need in order to answer the task. After this change, the runtime should keep the rich local artifacts for debugging and reproducibility, but the Codex-facing `workers/*/in/*.json` payloads should contain only the minimum information needed to classify lines.

The user-visible result is that line-role preview and live execution keep the same observable behavior and the same local debugging surface, while prompt budget drops because Codex stops reading `block_id`, `recipe_id`, `rule_tags`, `escalation_reasons`, `within_recipe_span`, and other operator-only fields on every row. A contributor should be able to run prompt preview on `thefoodlabcutdown`, inspect both the rich local debug payloads and the small model-facing payloads, and see that the line-role input estimate materially dropped without spending live tokens.

## Outcomes & Retrospective

## Context and Orientation

Canonical line-role is the Codex-backed stage that reviews ordered atomic lines from the canonicalized cookbook text and emits labels such as `RECIPE_TITLE`, `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `KNOWLEDGE`, or `OTHER`. An atomic line is one small text row. A shard is one contiguous batch of atomic lines that one model call owns. Line-role currently has two internal phases:

- `recipe_region_gate`: decide whether a row is clearly inside a recipe, clearly outside a recipe, or ambiguous at the recipe boundary
- `recipe_structure_label`: assign the actual line label using the gate result as one input

The relevant files are:

- `cookimport/parsing/canonical_line_roles.py`: plans line-role shards, builds per-phase payloads, writes runtime artifacts, launches direct Codex execution, validates returned rows, and writes prompt/debug artifacts.
- `cookimport/llm/canonical_line_role_prompt.py`: renders the wrapper prompts that tell Codex how to read the task file and what JSON to return.
- `cookimport/llm/prompt_preview.py`: reconstructs predictive prompt artifacts and writes preview task files under `line-role-pipeline/in/`.
- `cookimport/llm/prompt_artifacts.py`: exports reviewer-facing prompt/debug artifacts.
- `cookimport/llm/prompt_budget.py`: estimates prompt cost from preview artifacts and summarizes actual-cost telemetry after live runs.
- `tests/parsing/test_canonical_line_roles.py`, `tests/llm/test_prompt_preview.py`, and `tests/llm/test_label_phase_workers.py`: existing focused tests for the line-role runtime, preview, and worker artifacts.

Today the same per-row object serves two jobs at once: it is the local debug artifact and the model-facing transport. On the Food Lab preview, those row objects carry fields such as `block_id`, `recipe_id`, `rule_tags`, `escalation_reasons`, `within_recipe_span`, and repeated JSON key names that humans may want to inspect later but Codex does not need to read for the task. This plan separates those two audiences.

In this plan, “rich local artifact” means the full row object that preserves human debugging context. “Compact model payload” means the smaller JSON file that Codex actually reads through the file-backed transport.

## Plan of Work

Start by isolating the payload-building seam in `cookimport/parsing/canonical_line_roles.py`. The current helpers that assemble `input_payload` for `recipe_region_gate` and `recipe_structure_label` should be split into two explicit builders:

- one builder that produces the rich local artifact
- one builder that produces the compact model-facing payload

The rich local artifact should remain object-shaped and readable. It should preserve the current fields so existing debugging workflows do not lose information. Write those artifacts under a new stable sibling path, for example:

    line-role-pipeline/runtime/<phase>/workers/<worker-id>/debug/<shard>.json

and in preview:

    line-role-pipeline/debug_in/<phase>_input_<n>.json

The compact model-facing payload should be the file written to the active transport path:

    line-role-pipeline/runtime/<phase>/workers/<worker-id>/in/<shard>.json

and in preview:

    line-role-pipeline/in/<phase>_input_<n>.json

The compact schema must be prescriptive and stable. Use one top-level version marker and a `rows` array of positional tuples to avoid repeated key names on every row. The plan deliberately keeps context as-is for now so the savings are attributable to metadata removal rather than to a mixed optimization.

For `recipe_region_gate`, the compact payload should be:

    {
      "v": "line_role_recipe_region_gate_input.v1",
      "rows": [
        [atomic_index, deterministic_label_code, prev_text_or_null, current_line, next_text_or_null]
      ]
    }

For `recipe_structure_label`, the compact payload should be:

    {
      "v": "line_role_recipe_structure_label_input.v1",
      "rows": [
        [atomic_index, deterministic_label_code, region_status_code, prev_text_or_null, current_line, next_text_or_null]
      ]
    }

Define the code legends once in `cookimport/llm/canonical_line_role_prompt.py` and render them in the wrapper prompt, not per row. This is where label codes such as `L0`, `L1`, and region codes such as `R`, `O`, `B` belong. The compact payload must not include `block_id`, `recipe_id`, `rule_tags`, `escalation_reasons`, or `within_recipe_span`.

The runtime should continue to validate exact `atomic_index` coverage exactly as it does today. The change is not allowed to alter fallback behavior, proposal validation, or final label promotion. Only the transport payload shape is changing.

After the runtime builders are split, update `cookimport/llm/prompt_preview.py` so preview writes both the compact transport file and the rich debug file. The preview manifest and exported prompt rows should make the distinction obvious. The compact transport file should remain the one referenced by `request_input_file`. Add a second metadata field such as `debug_input_file` for the rich local copy. Update `cookimport/llm/prompt_artifacts.py` and `cookimport/llm/prompt_budget.py` so they continue to treat only the compact transport file as the model-facing request body when computing billed-like estimates, while still surfacing the richer local copy for humans.

Finally, update the focused tests so they prove the split. The runtime tests should assert that `workers/*/in/*.json` omits operator-only fields, while `workers/*/debug/*.json` retains them. The preview tests should assert that `line-role-pipeline/in/*.json` uses the compact schema and that `line-role-pipeline/debug_in/*.json` still contains the rich object rows. The budget tests should continue to show `prompt_input_mode=path` and should measure the compact payload, not the debug copy.

## Validation and Acceptance

Validation must prove both correctness and savings.

First, run the focused tests and expect them to pass. The new or updated assertions must prove all of the following:

1. The compact model-facing payload omits `block_id`, `recipe_id`, `rule_tags`, `escalation_reasons`, and `within_recipe_span`.
2. The rich local artifact still preserves those fields.
3. The wrapper prompt still references the compact `request_input_file` and still describes the correct tuple schema and legends.
4. Proposal validation still enforces exact owned `atomic_index` coverage.

Second, run predictive prompt preview on `thefoodlabcutdown` and inspect `prompt_preview_budget_summary.json`. Acceptance is:

1. `line-role-pipeline/in/*.json` uses the compact tuple schema and is the file referenced by `request_input_file`.
2. `line-role-pipeline/debug_in/*.json` exists and preserves the richer local object rows.
3. The input-token estimate for line-role drops materially relative to the current baseline of about `1,248,207` input tokens.
4. The reduction can be explained by metadata removal alone; the transport should still carry the same rows and the same `prev_text`/`next_text` context as before.

For this plan, “materially” should mean at least a quarter-million input-token reduction on Food Lab. The current metadata-only breakdown suggests roughly `370k` input tokens are available to reclaim. If the measured reduction is much smaller than that, inspect which operator-only fields are still reaching the compact payload or whether pretty-printed JSON is still being used on the model-facing file.

## Idempotence and Recovery

This change is safe to iterate on because it is additive around artifact families. The compact `in/*.json` payloads and the new `debug/*.json` or `debug_in/*.json` files can be regenerated repeatedly under a new timestamped preview output directory. If a midway implementation breaks line-role validation, keep the rich local artifacts and use them to compare what the compact transport omitted incorrectly.

Do not remove the rich artifact path until the compact transport has passed focused tests and Food Lab preview validation. The debugging surface is the recovery path if a compact schema drops information that the runtime still needs for promotion or diagnosis.
