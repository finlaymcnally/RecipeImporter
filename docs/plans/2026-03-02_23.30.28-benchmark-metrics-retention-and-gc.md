# ExecPlan: Benchmark Metrics Retention + Artifact GC

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, benchmark runs can stay small on disk without sacrificing long-term trend visibility in the dashboard. The operator will be able to run one command to prune heavy benchmark artifacts safely, while keeping durable benchmark metrics history available for `stats-dashboard` and comparisons.

Today, benchmark metrics come from a mix of CSV history and recursive JSON scan under `data/golden`. That scan behavior makes old artifacts “load-bearing” for history and encourages unbounded disk growth. The target behavior is: metrics persist in lightweight history storage, while large raw benchmark payloads can be removed by policy.

## Surprises & Discoveries

- Observation: `stats-dashboard` currently always scans benchmark `eval_report.json` recursively in `cookimport/analytics/dashboard_collect.py`, even when CSV rows exist.
  Evidence: `_collect_benchmarks(...)` is called and scans `golden_root.rglob("eval_report.json")`; benchmark records are merged from JSON + CSV.

- Observation: Bench runs can be very large due to run payload folders under `data/golden/bench/quality/runs/*` and `data/golden/bench/speed/runs/*`.
  Evidence: local cleanup showed multi-GB single run folders and >60 GB reclaimed by deleting older run roots.

- Observation: CSV benchmark rows alone are not yet the full story for all diagnostics because some rich fields are currently sourced from report/manifests.
  Evidence: collector enriches with report/manifests (`per_label`, boundary, runtime enrichment) during benchmark scan.

- Observation: The minimal missing durable field for dashboard parity was per-label metrics; strict/macro/boundary/runtime fields were already mostly present in benchmark CSV rows.
  Evidence: `append_benchmark_csv` already persisted strict/macro and boundary columns; `per_label` only existed in eval-report JSON scans.

- Observation: A safe GC path can hydrate missing benchmark CSV fields from `run_dir/eval_report.json` before deleting old run roots, avoiding history regressions.
  Evidence: New `run_benchmark_gc(...)` test covers `per_label_json` + strict/macro/boundary hydration followed by run-root deletion with backup creation.

## Decision Log

- Decision: Introduce an explicit benchmark garbage-collection command (`cookimport bench gc`) instead of ad-hoc manual deletion.
  Rationale: Users need safe, repeatable, inspectable retention behavior with dry-run output and backups.
  Date/Author: 2026-03-02 / Assistant.

- Decision: Move benchmark dashboard history toward durable lightweight storage (CSV and/or compact history archive) as the primary source; treat JSON scan as optional backfill.
  Rationale: This is the only path to preserving trend history while allowing old artifact deletion.
  Date/Author: 2026-03-02 / Assistant.

- Decision: Keep two retention tiers: full recent runs and metrics-only long tail.
  Rationale: Recent deep debugging still needs richer artifacts, but old runs should not retain heavyweight payloads.
  Date/Author: 2026-03-02 / Assistant.

- Decision: Persist per-label benchmark metrics in CSV (`per_label_json`) instead of relying on recursive eval-report scans for dashboard diagnostics.
  Rationale: Per-label diagnostics are dashboard-visible and must survive artifact pruning.
  Date/Author: 2026-03-02 / Assistant.

- Decision: Keep benchmark recursive scanning available but opt-in (`--scan-benchmark-reports`), with automatic fallback only when benchmark CSV rows are unavailable.
  Rationale: Preserves low-cost default behavior while retaining backfill/debug capability.
  Date/Author: 2026-03-02 / Assistant.

  - consumes benchmark fields used in Previous Runs, trend, runtime/per-label/boundary diagnostics.
  - acceptance requires unchanged behavior for retained metrics after GC.

- `cookimport/bench/`
  - run-producing modules (`quality_runner.py`, `speed_runner.py`, canonical/stage eval modules) write benchmark artifacts.
  - we will add a new GC module here (for example `artifact_gc.py`) to centralize retention logic.

- `data/.history/performance_history.csv`
  - current durable history file.
  - must remain valid and not point at deleted benchmark paths unless that row is intentionally retained as metrics-only and dashboard can handle missing links gracefully.

Terms used in this plan:

- Full run: a benchmark run directory that retains all artifacts, including heavy prediction/intermediate payloads.
- Metrics-only run: a run state where only compact benchmark metric artifacts are retained (for example eval report plus compact metadata), while heavy payloads are removed.
- GC: deterministic garbage collection that applies retention policy and cleans old artifacts safely.

## Plan of Work

### Milestone 1: Define durable benchmark history contract for dashboard use

Create a compact benchmark history payload that is sufficient for dashboard sections currently shown on the main page (`Previous Runs`, trend, latest runtime, per-label, boundary). The implementation can use CSV augmentation plus a compact JSONL sidecar if some fields are too nested for CSV.

Implementation targets:

- `cookimport/analytics/perf_report.py`
- `cookimport/analytics/dashboard_schema.py`
- optionally new `cookimport/analytics/benchmark_history.py`

At end of this milestone, every new benchmark run writes all dashboard-critical benchmark fields into durable history storage independent of heavy run directories.

### Milestone 2: Make dashboard benchmark collection history-first

Adjust `collect_dashboard_data(...)` so benchmark history is read primarily from durable history storage, with recursive `eval_report.json` scanning moved behind an explicit opt-in path (for example `--scan-benchmark-reports`) or fallback-only behavior.

Implementation targets:

- `cookimport/analytics/dashboard_collect.py`
- `cookimport/cli.py` (`stats-dashboard` options/wiring)
- docs updates in `docs/08-analytics/*.md`

At end of this milestone, deleting old run artifact folders does not remove benchmark rows from dashboard history.

### Milestone 3: Implement `cookimport bench gc`

Add a new bench subcommand for retention policy enforcement.

Command shape (final names can vary, but behavior must match):

- `cookimport bench gc --dry-run`
- `cookimport bench gc --apply`
- policy flags such as:
  - `--keep-full-runs <N>`
  - `--keep-full-days <D>`
  - `--drop-speed-artifacts` (optional convenience switch)
  - `--golden-root <path>`
  - `--output-root <path>` (for deriving history path)

Behavior:

- Enumerate benchmark run roots by timestamped folder pattern.
- Keep recent runs in full form by count/date policy.
- For older runs, either convert to metrics-only (preferred) or remove run roots once durable history is confirmed.
- Prune history rows referencing deleted paths only when necessary; otherwise preserve metric records.
- Always write timestamped backups before mutating history files.
- Print a compact report: folders removed, bytes reclaimed, rows updated, warnings.

Implementation targets:

- new `cookimport/bench/artifact_gc.py`
- `cookimport/cli.py` (`bench_app` command wiring)
- tests under `tests/bench/` and `tests/analytics/`.

## Validation and Acceptance

Acceptance is met when all statements below are true:

- Running `cookimport bench gc --dry-run` reports intended deletions and row updates with no filesystem mutations.
- Running `cookimport bench gc --apply` reduces `data/golden` benchmark artifact disk usage according to policy and writes timestamped backup files.
- `cookimport stats-dashboard` after GC still shows historical benchmark rows for retained metric history (not just newest surviving artifact directory).
- Dashboard renders without JS/runtime errors and `Previous Runs` + trend remain coherent.
- New tests fail before implementation and pass after implementation.

## Idempotence and Recovery

GC must be safe to re-run. A second `--apply` run with unchanged policy should report minimal or zero additional work.

Before any mutation, create timestamped backups for history files. Recovery is: restore the latest backup CSV/archive and regenerate dashboard.

For artifact recovery, if a deleted run must be restored, recover from external backup or avoid deleting full runs by setting a more conservative policy (`--keep-full-runs` and/or `--keep-full-days`).
