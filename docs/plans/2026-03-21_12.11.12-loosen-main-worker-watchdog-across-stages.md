# Loosen The Main-Worker Watchdog Across Recipe, Line-Role, And Knowledge

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the main Codex workspace workers in recipe, line-role, and knowledge should stop dying for behaving like agents. A worker that opens leased files, orients locally, runs many local shell commands, writes helper scripts, reads and rewrites workspace-local files, iterates on outputs, and generally acts like a bounded local automation agent should be allowed to continue. The watchdog should become a rare backstop, not an active steering mechanism.

The user-visible proof is that the saved Salt Fat Acid Heat benchmark replay no longer shows main workers ending in `watchdog_off_task_command_prevented` for commands like `cat worker_manifest.json`, `cat current_packet.json`, `ls`, `pwd`, local `jq`/`sed`/`cat` pipelines, or helper-script creation, while still killing only truly rare cases where the worker is obviously outside the bounded local task surface or is clearly looping/circling in a materially token-wasting way.

## Context and Orientation

This repository has two different direct-execution modes that matter for this plan.

The first mode is a long-lived workspace worker. In this repository that means a Codex session launched inside an isolated worker root containing repo-written control files such as `worker_manifest.json`, `assigned_shards.json`, `assigned_tasks.json`, `current_packet.json`, `current_hint.md`, `current_result_path.txt`, `packet_lease_status.json`, `in/*.json`, `out/*.json`, and `scratch/`. The worker is supposed to open those local files, reason about the current leased task packet, write its result file inside the worker root, and keep looping until all leased packets are settled.

The second mode is a structured follow-up call. In this repository that means a one-shot retry or repair attempt that is supposed to emit one strict JSON answer directly. Those calls do not need the same freedom as a long-lived workspace worker and can keep stricter no-shell rules.

The watchdog is the repo-owned supervision logic that receives snapshots of what the worker just did. The current implementation lives primarily in `cookimport/llm/codex_exec_runner.py`, with stage-specific call sites in:

- `cookimport/llm/codex_farm_orchestrator.py` for recipe workers
- `cookimport/parsing/canonical_line_roles.py` for line-role workers
- `cookimport/llm/codex_farm_knowledge_orchestrator.py` for knowledge workers

## Current State And Intended Design

### Milestone 1: Reproduced the startup-kill regression with deterministic tests

The repo already contained useful watchdog-focused tests, and this implementation tightened them into an explicit anti-regression suite for the actual complaint: main processing workers must not die for acting like workspace-local agents. Knowledge, recipe, and line-role now all have focused workspace-worker tolerance coverage in addition to the shared direct-exec tests.

### Milestone 2: Made the shared main-worker watchdog much looser, with startup grace

The intended product change is stronger than the earlier draft: the shared transport should not kill a main workspace worker for shell shape, startup behavior, local iteration, helper-script creation, or ordinary file-heavy agent work. Startup access to leased packet files, bounded local orientation inside the worker root, repeated reads and writes, and many local commands should all be tolerated.

The target contract is:

- startup file opens inside the worker root are allowed
- bounded local shell work is allowed
- writing and running bounded workspace-local helper scripts is allowed
- high command counts by themselves are not suspicious
- command shape is telemetry, not a kill reason
- reviewer-facing command classification is telemetry
- boundary-violation detection is enforcement
- the watchdog kills only for rare, clearly bad situations:
  real workspace-boundary escape
  clearly off-contract external tool use if it leaves the bounded local processing surface
  gross no-progress looping/circling that is materially wasting tokens
- ordinary “agent behavior” is not watchdog-worthy

### Milestone 3: Applied the looser main-worker policy across recipe, line-role, and knowledge

The stage-specific orientation-policing branches are gone from the main workspace-worker paths. Recipe was already aligned. Knowledge no longer computes a second stricter prevention verdict for orientation commands, and line-role no longer routes workspace commands through a stricter classifier than the other main stages.

Structured retry and repair calls remain distinct from long-lived workspace workers and can keep their stricter handling. That separation was part of the implementation, not an abandoned design note.

### Milestone 4: Updated observability and docs so they describe the final contract

`docs/10-llm/10-llm_README.md` and the worker prompts now tell one consistent story: the main processing worker watchdog is intentionally rare and permissive across line-role, recipe, and knowledge, while structured follow-up attempts remain the stricter surface.
