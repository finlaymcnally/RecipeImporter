# Loosen The Main-Worker Watchdog Across Recipe, Line-Role, And Knowledge

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the main Codex workspace workers in recipe, line-role, and knowledge should stop dying for behaving like bounded local agents. A worker that opens its leased files, does small local orientation inside its own worker root, uses normal shell search and filter commands, and writes local result files should be allowed to continue. The watchdog should remain only as a backstop for real sandbox escape, clearly forbidden tools, or long no-output stalls.

The user-visible proof is that the saved Salt Fat Acid Heat benchmark replay no longer shows main workers ending in `watchdog_off_task_command_prevented` for commands like `cat worker_manifest.json`, `cat current_packet.json`, `ls`, or `pwd`, while still killing workers that invoke forbidden interpreters, package managers, network tools, or workspace escapes.

## Context and Orientation

This repository has two different direct-execution modes that matter for this plan.

The first mode is a long-lived workspace worker. In this repository that means a Codex session launched inside an isolated worker root containing repo-written control files such as `worker_manifest.json`, `assigned_shards.json`, `assigned_tasks.json`, `current_packet.json`, `current_hint.md`, `current_result_path.txt`, `packet_lease_status.json`, `in/*.json`, `out/*.json`, and `scratch/`. The worker is supposed to open those local files, reason about the current leased task packet, write its result file inside the worker root, and keep looping until all leased packets are settled.

The second mode is a structured follow-up call. In this repository that means a one-shot retry or repair attempt that is supposed to emit one strict JSON answer directly. Those calls do not need the same freedom as a long-lived workspace worker and can keep stricter no-shell rules.

The watchdog is the repo-owned supervision logic that receives snapshots of what the worker just did. Today that logic lives primarily in `cookimport/llm/codex_exec_runner.py`, with stage-specific watchdog call sites in:

- `cookimport/llm/codex_farm_orchestrator.py` for recipe workers
- `cookimport/parsing/canonical_line_roles.py` for line-role workers
- `cookimport/llm/codex_farm_knowledge_orchestrator.py` for knowledge workers

The saved regression fixture for this plan is:

- `data/golden/benchmark-vs-golden/2026-03-21_11.17.08/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/manifest.json`

## Plan of Work

### Milestone 1: Reproduce the startup-kill regression with deterministic tests

Start by writing fail-before tests that capture the exact user complaint instead of generic watchdog assertions. Model the startup behavior from the saved knowledge workers: reading `worker_manifest.json`, reading the current leased packet files, and doing a small amount of local orientation such as `ls` or `pwd` inside the worker root before the first output file exists. Those tests should assert that in main workspace-worker mode this behavior must not trigger a fatal watchdog decision.

Also add the matching negative tests. Main workspace workers must still die for real forbidden executables such as `python`, `node`, `git`, package managers, and network tools, and for visible workspace escape such as parent traversal or off-root absolute paths. These tests belong first in `tests/llm/test_codex_exec_runner.py`, then in stage-specific files so recipe, line-role, and knowledge all inherit the same contract.

This milestone is complete when the new tests fail for the right reason on current code: startup/orientation commands still trip a fatal watchdog prevention reason, while forbidden-tool tests continue to enforce the current strict behavior.

### Milestone 2: Make the shared main-worker watchdog much looser, with startup grace

Implement the real product change in `cookimport/llm/codex_exec_runner.py`. The main workspace-worker policy should become much looser than it is today in three concrete ways.

First, add a startup grace path. The first commands a worker uses to bootstrap itself inside the worker root must be allowed as long as they stay local. Opening `worker_manifest.json`, `current_packet.json`, `current_hint.md`, `current_result_path.txt`, `assigned_tasks.json`, and doing minimal local orientation like `pwd` or `ls` inside the worker root should not be fatal.

Second, change the kill policy for main workspace workers so shell complexity is never the reason for termination. Loops, pipes, redirects, `jq`, `rg`, `cat`, and small workspace-local helper scripts under `scratch/` should all be allowed when they stay inside the worker root. The watchdog should kill only for a short list of real violations: forbidden executables, visible parent traversal, off-root absolute paths except explicitly tolerated null sinks, or a clearly pathological no-output loop after generous time and command budgets.

Third, preserve telemetry while changing enforcement. The runtime should still record command classifications for post-mortem reporting, but those classifications must no longer be used as an automatic kill switch for main workspace workers.

This milestone is complete when the shared helper can distinguish startup-or-local-work from real boundary violations, and when shell shape alone is no longer fatal in main workspace-worker mode.

### Milestone 3: Apply the looser main-worker policy consistently across recipe, line-role, and knowledge

After the shared helper is fixed, thread the looser main-worker policy through `cookimport/llm/codex_farm_orchestrator.py`, `cookimport/parsing/canonical_line_roles.py`, and `cookimport/llm/codex_farm_knowledge_orchestrator.py`. The user explicitly asked for this to be fixed “in every step,” so this milestone is not optional.

Each stage may keep its own timeout and recovery rules, but they must all agree on the core main-worker contract:

- startup file opening inside the worker root is allowed
- bounded local shell work is allowed
- real boundary violations are killed
- generous no-output stall conditions are killed
- structured retry and repair calls remain strict

This milestone is complete when all three stages share the same loose main-worker behavior and their tests prove it.

### Milestone 4: Update observability and docs so future debugging cannot lie about the contract

Finish by updating the observability and the short current-state docs to match the new contract. The runtime should keep enough telemetry to explain why a worker was allowed, prevented, or killed, but those reason codes must now reflect the new design. Distinguish clearly between allowed startup/local orientation, allowed workspace-local shell work, real boundary violations, and no-output stall kills.

Update `docs/10-llm/10-llm_README.md` and, if needed, `docs/10-llm/nonrecipe_knowledge_review.md` so a novice reader can tell that main workspace workers are intentionally loose, structured follow-ups remain strict, and the 2026-03-21 Salt Fat failure was used as the regression fixture.

This milestone is complete when the docs describe the new contract in plain language and the zero-token tests prove it.

## Validation and Acceptance

In recipe, line-role, and knowledge tests, main workspace workers can use bounded local agent behavior without being killed for shell shape alone. They still die for real forbidden tools, workspace escape, or generous no-output stall conditions.

## Idempotence and Recovery

This plan is designed to be implemented test-first and rerun safely. The saved benchmark artifact under `data/golden/benchmark-vs-golden/2026-03-21_11.17.08/...` is read-only diagnostic input and can be consulted repeatedly without mutating it.

The recommended test commands are idempotent. If one stage-specific test file still encodes the old watchdog assumption after the shared helper is changed, update that test only after confirming the new behavior matches this plan. Do not “fix” failures by re-tightening the main-worker watchdog.

If an implementation attempt accidentally makes structured retry or repair calls loose as well, treat that as a regression and back it out. This plan loosens only main workspace workers.
