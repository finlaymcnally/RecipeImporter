# Fix line-role packet-mode misclassification before workers read the wrong story

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The line-role worker packets now have the right outer packaging: workers get authoritative packet JSON, hint sidecars, and small contrast examples in their local workspace. The remaining issue is that the packet classifier can still tell a worker the wrong story before the worker reads a single row. In the preserved 2026-03-21 benchmark run, the first worker packet is blurbs, front matter, and table-of-contents material, yet the hint says `recipe_body` with `high` confidence and tells the worker to preserve recipe structure. Packet 2 does the same for a contents-style list of recipe names such as `Blue Cheese Dressing`, `Green Goddess Dressing`, and `Six Ways to Cook Vegetables`. After this change, those packets should instead be framed conservatively as navigation/front-matter or at worst mixed-boundary packets, so the worker defaults to `OTHER` or `KNOWLEDGE` unless immediate local evidence proves real recipe structure.

The user-visible outcome is concrete. If a future operator opens `data/output/.../line-role-pipeline/runtime/line_role/workers/*/hints/*.md`, early front-matter and contents packets should no longer claim that recipe-local structure dominates. A worker should see a packet interpretation that matches the text in the packet itself. That should reduce the exact harmful early-book flips we observed in `data/golden/benchmark-vs-golden/2026-03-21_14.53.27`, where table-of-contents lines were upgraded into `RECIPE_TITLE`, `INGREDIENT_LINE`, `INSTRUCTION_LINE`, and `RECIPE_VARIANT`.

- Observation: the second preserved packet is a contents-style list, not a live recipe body, but it is also classified as `recipe_body`.

The authoritative rows include `Blue Cheese Dressing`, `Green Goddess Dressing`, `Vegetables`, `Six Ways to Cook Vegetables`, `Steamy Sauté: Garlicky Green Beans`, and similar title-list entries.

- Observation: later packets show that the packet-mode mechanism itself is useful when it lands correctly.
  Evidence: `.../worker-005/hints/line-role-canonical-0005-a000589-a000735.task-001.md` correctly classifies a pedagogical packet as `lesson_prose`, cites `Balancing Fat`, and tells the worker to default to `KNOWLEDGE` or `OTHER`.

- Observation: the current classifier is too willing to infer `recipe_body` from a few seeded structure labels and title-like rows, even when every span status is unknown.
  Evidence: `_classify_line_role_packet_mode(...)` in `cookimport/parsing/canonical_line_roles.py` promotes to `recipe_body` once `recipe_structure_count >= 2` and either `span_inside > 0` or `ingredient_like_count + instruction_like_count >= 3`. A TOC-like list can satisfy that by accident through deterministic `RECIPE_TITLE`, `INGREDIENT_LINE`, or `INSTRUCTION_LINE` seeds plus `instruction_like` tags, even though the packet has no trustworthy in-recipe evidence.

- Observation: the current `recipe_body` trigger effectively double-counts weak positive evidence.
  Evidence: in preserved worker-001 packet 1, the debug rows yield `recipe_structure_count=7` and `ingredient_like_count + instruction_like_count=14` even though the packet begins with `Praise for SALT FAT ACID HEAT`, endorsement blurbs, `CONTENTS`, `Foreword`, and `How to Use This Book`. Those totals come from both seeded structural labels and heuristic tags contributing to the same early `recipe_body` branch.

- Observation: once the packet summary is wrong, the worker prompt amplifies the mistake.
  Evidence: `cookimport/llm/canonical_line_role_prompt.py` and the recorded `prompt_line_role.txt` both tell the worker to treat the deterministic label as a strong prior and to respect packet-local guidance. A bad packet mode is therefore not harmless metadata; it materially shifts the worker’s default posture.

- Observation: the packet-context payload was already the only prompt seam that mattered for this fix; no prompt-template branch was required once the new packet mode text lived in parser-owned guidance.
  Evidence: the new prompt regression in `tests/parsing/test_canonical_line_roles.py` passes by feeding `front_matter_navigation` packet context through the existing `build_canonical_line_role_file_prompt(...)` renderer and asserting that the prompt includes `Packet mode: front_matter_navigation` plus `Do not over-structure recipe-name lists`.

- Observation: deduplicating recipe-body evidence down to row-level signals was enough to keep the preserved lesson packet healthy while stopping the early bad packets.
  Evidence: a direct `.venv` classifier check now returns `('front_matter_navigation', 'high')` for preserved worker-001 packets 1 and 2, and still returns `('lesson_prose', 'high')` for preserved worker-005 packet 1.

## Outcomes & Retrospective

Implementation is complete for the deterministic portion of this plan. A novice opening the preserved-style worker hints generated by the current classifier would now see early front matter and contents-style packets framed as `front_matter_navigation` rather than `recipe_body`, while the preserved `Balancing Fat` packet still resolves to `lesson_prose`.

The main tradeoff remains the one identified during design: avoid overfitting to Salt Fat Acid Heat while still failing closed on all-unknown-span title lists. The implemented classifier handles that by combining explicit front-matter/navigation detectors with stricter unknown-span `recipe_body` requirements and row-level evidence deduplication instead of book-specific title exceptions.

## Context and Orientation

This repository uses a “line-role pipeline” to assign one cookbook label to each atomic text line. An “atomic line” in this codebase is one small unit of cookbook text such as a heading, ingredient line, instruction sentence, note, or paragraph. The relevant code lives in `cookimport/parsing/canonical_line_roles.py` and `cookimport/llm/canonical_line_role_prompt.py`.

The current line-role stage works like this. Deterministic code creates a first-pass label for each atomic line. The stage then groups those lines into ordered “packets”, where each packet is one bounded slice of the book. A Codex worker does not read the whole book; it reads one or more local task packets. Each worker gets a workspace root under `data/output/.../line-role-pipeline/runtime/line_role/workers/<worker-id>/`. Inside that root:

- `in/<task-id>.json` is the authoritative packet input. It contains compact row tuples in the form `[atomic_index, label_code, current_line]`, along with top-level packet metadata such as `packet_mode`, `packet_summary`, `default_posture`, `flip_policy`, and `example_files`.
- `hints/<task-id>.md` is a human-readable explanation layer. It summarizes label counts, span-status counts, packet interpretation, decision policy, and selected “attention rows”.
- `examples/*.md` are short repo-written contrast examples used to calibrate the worker.

The packet-mode classifier is `_classify_line_role_packet_mode(...)` in `cookimport/parsing/canonical_line_roles.py`. That helper feeds `_build_line_role_packet_context(...)`, which in turn populates `packet_summary`, `default_posture`, `strong_signals`, `weak_signals`, `flip_policy`, and `example_files`. Those fields are then used in three places: the authoritative `in/*.json` payload, the worker hint markdown, and the file-backed line-role prompt built by `build_canonical_line_role_file_prompt(...)` in `cookimport/llm/canonical_line_role_prompt.py`.

The current active packet modes are `front_matter_navigation`, `recipe_body`, `recipe_adjacent_notes`, `lesson_prose`, `memoir_front_matter`, and `mixed_boundaries`. The preserved 2026-03-21 benchmark run exposed why the new mode is necessary: front matter and contents/navigation packets can look title-heavy enough to fool structure heuristics, but they are not live recipe bodies and should default to `OTHER` or `KNOWLEDGE`.

The concrete root-cause bug in the current classifier is simple. `recipe_body` is checked before `lesson_prose` or `memoir_front_matter`, and the positive evidence threshold is too weak when span status is entirely unknown. A packet can reach `recipe_body` from a handful of deterministic structural seeds plus `ingredient_like` or `instruction_like` tags, even if the packet is really front matter or a contents list. In preserved worker-001 packet 1, that branch fires with `recipe_structure_count=7` and `ingredient_like_count + instruction_like_count=14` despite zero trustworthy span evidence and obviously non-recipe rows at the top of the packet.

## Plan of Work

Start in `cookimport/parsing/canonical_line_roles.py` at `_classify_line_role_packet_mode(...)`. Fix the positive-evidence bug before doing anything else. Tighten the promotion rules for `recipe_body` so the classifier no longer accepts “a few structural seeds plus title-like rows” as sufficient evidence when all packet rows have `within_recipe_span=None`. In practice, that means de-risking the current branch that uses `recipe_structure_count` plus `ingredient_like_count + instruction_like_count`: do not let weak heuristic tags and deterministic seed labels compound into high-confidence `recipe_body` without stronger local proof that one contiguous recipe body is actually present.

After that threshold is repaired, add explicit negative signals for contents/navigation packets. The new logic should look for patterns that are common in preserved false positives: many short title-like rows in sequence, clusters of contents-style labels such as `CONTENTS`, `Foreword`, `Introduction`, `PART ONE`, chapter-title ladders, and dense lists of recipe names or section names without recipe-local yield/ingredient/instruction neighborhoods. If those signals dominate, either classify the packet as a new mode such as `contents_navigation` or route it into a stricter conservative fallback such as `mixed_boundaries` if that produces the same worker posture with less propagation churn. The key requirement is honest worker guidance, not a specific enum addition.

After the classifier change, extend the packet-context helpers in the same file. Update `_line_role_packet_summary(...)`, `_line_role_default_posture(...)`, `_line_role_strong_signals(...)`, `_line_role_flip_policy(...)`, and `_line_role_example_files(...)` so the worker sees the right explanation for whichever conservative interpretation wins. The new posture should default to `OTHER` for navigation/front matter and allow `KNOWLEDGE` for explanatory topic headings or reference prose, while requiring immediate packet-local evidence before upgrading any line to recipe-local structure. Also lower packet confidence whenever every row has unknown span status and the only positive evidence comes from weak structure heuristics or preexisting seed labels.

Then update `cookimport/llm/canonical_line_role_prompt.py` so file-backed prompt rendering reflects the repaired packet interpretation cleanly. The prompt text already has a packet-local guidance block; it just needs to render the conservative packet posture without ambiguity. Add one or two short negative rules for contents/front-matter packets so a worker sees that a list of recipe names inside a contents-style packet is not automatically a live recipe region.

Next, expand `tests/parsing/test_canonical_line_roles.py`. Keep the existing small packet-context tests for `lesson_prose` and `memoir_front_matter`, but add two preserved-packet reproductions:

1. A front-matter packet modeled on worker-001 task 001, containing blurbs, `CONTENTS`, `Foreword`, `Introduction`, and chapter-heading rows. Assert that `_build_line_role_packet_context(...)` does not return `recipe_body` and that its `default_posture` is conservative.

2. A contents-style packet modeled on worker-001 task 002, containing short title-list rows such as `Blue Cheese Dressing`, `Green Goddess Dressing`, `Vegetables`, `Steamy Sauté: Garlicky Green Beans`, `Stock and Soups`, and `Eggs`. Assert that the packet is not `recipe_body`, that confidence is not `high` unless there is real span evidence, and that the flip policy warns against promoting title-list packets into recipe structure.

Add one more branch-isolation regression assertion that does not depend on the preserved benchmark files: construct a compact synthetic packet where all rows have `within_recipe_span=None`, a few rows carry deterministic `RECIPE_TITLE` / `INGREDIENT_LINE` / `INSTRUCTION_LINE` seeds, and heuristic tags alone would satisfy the current `recipe_body` threshold. Assert that this packet does not resolve to `recipe_body`. This test is the tightest fail-before/pass-after proof for the actual heuristic bug.

Add one final regression assertion that a preserved pedagogical packet like `Balancing Fat` still resolves to `lesson_prose`. This keeps the fix honest: it must stop the bad early packets without breaking the already-correct later packets.

If the new packet mode changes active behavior enough to matter for future reviewers, update the short line-role section in `docs/04-parsing/04-parsing_readme.md`. Keep it brief. The doc should say that packet classification now distinguishes contents/front-matter packets from true recipe bodies and that all-unknown span status is treated conservatively.

## Validation and Acceptance

The change is acceptable only if all of the following are true.

Opening the preserved-style worker hints for an early-book packet no longer shows `recipe_body` for obvious front matter or contents-style packets. A novice should be able to read `hints/<task-id>.md` and agree that the packet interpretation matches the actual text shown in `in/<task-id>.json`.

The packet classifier is stricter under all-unknown span status. In tests, a packet made entirely of `within_recipe_span=None` rows must not reach high-confidence `recipe_body` unless it also carries clear local recipe-body evidence such as real ingredient clusters plus imperative steps that form one contiguous recipe.

The heuristic root cause is directly covered. A synthetic packet that previously satisfied the old `recipe_body` branch through deterministic structural seeds plus `ingredient_like` / `instruction_like` tags must now resolve to a conservative non-`recipe_body` mode. This proves the implementation fixed the branch logic rather than merely special-casing one preserved book packet.

The preserved good path stays intact. A packet modeled on `Balancing Fat`-style pedagogical prose must still resolve to `lesson_prose` and must still tell the worker to default to `KNOWLEDGE` or `OTHER`.

Prompt and hint surfaces stay aligned. The packet-local guidance rendered into the file-backed prompt must include the new packet mode and its conservative posture, and the corresponding hint markdown must expose the same meaning in plain English.

Deterministic proof must be recorded. Run the focused parsing tests and the full `tests/parsing/test_canonical_line_roles.py` file in `.venv`, and paste the commands plus short pass/fail evidence into this plan’s `Artifacts and Notes` section.

Benchmark proof is explicitly not required for default acceptance. A live benchmark rerun remains optional and requires explicit user approval because it spends model tokens.

## Idempotence and Recovery

These edits are additive and safe to repeat. The classifier and tests can be rerun without mutating preserved benchmark roots. Existing historical runtime artifacts should remain untouched; new behavior should only affect newly generated packet hints and packet JSON.

If the new packet mode proves too specific or harms later packets, back out only the classifier branch and helper text for that mode, then rerun the focused packet-context tests. Do not remove the worker hint scaffolding or revert the earlier context-hardening work; this plan is a narrow correction to packet interpretation, not a rollback of the broader worker-packet design.

If the fix makes too many packets fall into a low-confidence generic bucket, prefer relaxing the new negative signals slightly rather than widening `recipe_body` again. The preserved failure shows that false confident `recipe_body` framing is worse than a conservative mixed packet.
