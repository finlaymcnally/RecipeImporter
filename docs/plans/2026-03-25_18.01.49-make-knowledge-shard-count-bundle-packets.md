# Make Knowledge Shard Count Bundle Packets

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, `knowledge_prompt_target_count` will do what the repo says it does: it will change the number of knowledge shard bundles while preserving one immutable packet file per packet and one task per packet inside the worker runtime. A user will be able to lower the requested shard count, see fewer shard entries in `shard_manifest.jsonl`, and still see every packet preserved and executed through the existing task runtime.

You can see the result by running the focused knowledge-planning tests listed below. The observable outcome is that packet files under `knowledge/in/*.json` remain one-per-packet, while `KnowledgeJobBuildReport.shard_entries` and the shard payloads group those packets into real shard bundles when the requested shard count is lower than the packet count.

## Surprises & Discoveries

- Observation: the runtime already knows how to split a multi-packet shard back into packet-local tasks, so the planner does not need a new execution model.
  Evidence: `cookimport/llm/knowledge_stage/planning.py` accepts shard payloads with `p[]` and `_build_knowledge_task_plans(...)` emits one task per packet from that list.

- Observation: the tests already expect bundled shard behavior in several places, which means the intended contract is clearer than the live planner implementation.
  Evidence: `tests/llm/test_knowledge_packet_planning.py` and `tests/llm/test_knowledge_job_bundles.py` both expect one shard to own multiple packet ids when `prompt_target_count` is lower than packet count.

- Observation: bundled shard metadata was still lying about total shard size even after bundling shipped, because the planner wrote `packet_char_count` while recovery sizes only read `char_count`.
  Evidence: `cookimport/llm/codex_farm_knowledge_jobs.py` emitted `packet_char_count`, but `cookimport/llm/knowledge_stage/recovery.py::_classify_knowledge_watchdog_retry_size(...)` only consults `metadata["char_count"]`, so bundled shards reported `0` chars to retry sizing.

- Observation: `source_hash` was still threaded through the planner surface but had no effect anywhere in the planner.
  Evidence: `build_knowledge_jobs(...)` accepted `source_hash`, but the function body never read it and the packet payload contract has no `source_hash` field.

## Outcomes & Retrospective

The user-visible packet-versus-shard behavior was already in place by the time this plan was executed: packet files stayed one-per-packet, bundled shard payloads used `sid + p[]`, and the task runtime already split those bundles back into one packet-owned task per output file. The remaining gap was honesty. Shard metadata now reports bundle-wide `packet_count`, `owned_block_count`, and `char_count`, so recovery/reporting code sizes the real shard bundle instead of silently seeing `0` chars or a misleading packet-only name.

The planner surface is also smaller now. `build_knowledge_jobs(...)` no longer accepts unused `source_hash`, and the focused tests now prove both the shard-bundling contract and the 5-packets-to-2-shards partition case.

## Context and Orientation

The code path starts in `cookimport/llm/codex_farm_knowledge_jobs.py`. In this repository, a “packet” is one model-facing knowledge review input and a “shard” is the scheduling bundle that can own one or more packets. `build_knowledge_jobs(...)` now creates packets correctly and bundles them into contiguous shard entries when the requested shard count is lower than packet count. The remaining issue this plan closed was metadata honesty: bundled shards needed shard-native size fields and the planner entrypoint still exposed one dead argument.

The key contract types live in `cookimport/llm/codex_farm_knowledge_contracts.py`. `KnowledgePacketJobInputV1` is the one-packet payload. `KnowledgeShardJobInputV1` is the multi-packet shard payload with `sid` and `p`. The worker-side splitter lives in `cookimport/llm/knowledge_stage/planning.py`. That code reads `p[]` from the shard payload and emits one task per packet, so the runtime does not need a new execution model.

The operator-facing wording lives in `cookimport/config/run_settings.py`, and the owning human-facing explanation lives in `docs/10-llm/10-llm_README.md`. Those docs should explain that `knowledge_prompt_target_count` controls shard bundles, while `knowledge_worker_count` controls concurrency.
