## Purpose / Big Picture

After this change, `knowledge_prompt_target_count` will do what the repo says it does: it will change the number of knowledge shard bundles while preserving one immutable packet file per packet and one task per packet inside the worker runtime. A user will be able to lower the requested shard count, see fewer shard entries in `shard_manifest.jsonl`, and still see every packet preserved and executed through the existing task runtime.

You can see the result by running the focused knowledge-planning tests listed below. The observable outcome is that packet files under `knowledge/in/*.json` remain one-per-packet, while `KnowledgeJobBuildReport.shard_entries` and the shard payloads group those packets into real shard bundles when the requested shard count is lower than the packet count.

## Context and Orientation

The code path starts in `cookimport/llm/codex_farm_knowledge_jobs.py`. In this repository, a “packet” is one model-facing knowledge review input and a “shard” is the scheduling bundle that can own one or more packets. `build_knowledge_jobs(...)` currently creates packets correctly, but its live write path still behaves too much like `packet == shard` even though the contracts and tests already support `shard -> many packets`.

The key contract types live in `cookimport/llm/codex_farm_knowledge_contracts.py`. `KnowledgePacketJobInputV1` is the one-packet payload. `KnowledgeShardJobInputV1` is the multi-packet shard payload with `sid` and `p`. The worker-side splitter lives in `cookimport/llm/knowledge_stage/planning.py`. That code reads `p[]` from the shard payload and emits one task per packet, so the runtime does not need a new execution model.

The operator-facing wording lives in `cookimport/config/run_settings.py`, and the owning human-facing explanation lives in `docs/10-llm/10-llm_README.md`. Those docs should explain that `knowledge_prompt_target_count` controls shard bundles, while `knowledge_worker_count` controls concurrency.

## Plan of Work

First, edit `cookimport/llm/codex_farm_knowledge_jobs.py` so the planner still writes one JSON file per packet, but computes shard bundles after packet preparation. The planner should clamp the requested shard count to the packet count, preserve source order, partition packets into contiguous bundles, and build one `ShardManifestEntryV1` per bundle. A one-packet bundle should keep the current single-packet payload shape. A multi-packet bundle should write a `KnowledgeShardJobInputV1` payload with `sid` and `p`.

Second, make the shard metadata honest. Each shard manifest entry should expose all packet ids it owns, the union of block evidence refs, and metadata such as `packet_count`, `owned_packet_ids`, `owned_block_indices`, and `task_count`. Remove or stop threading planner parameters that are immediately discarded in the live code path.

Third, update the settings and docs so they describe the live behavior accurately. `knowledge_prompt_target_count` should be described as a shard-bundle target, not a chunk or packet count and not a concurrency knob.

Finally, run focused knowledge-planning tests from the activated project venv and record the result in this file.
