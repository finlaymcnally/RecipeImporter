## Context and Orientation

Two files are the current focus. `tests/llm/test_codex_farm_knowledge_orchestrator.py` is the large knowledge-stage integration suite. It contains several cohesive runtime/progress tests in the middle and later portions of the file, including progress snapshot reporting, live task-packet progress, worker concurrency, and packet leasing. Those tests are related to worker-runtime behavior, not to the full set of knowledge-stage behaviors in the file.

`tests/llm/test_codex_exec_runner.py` mixes pure helper tests with workspace preparation and subprocess supervision tests. The helper seam covers command classification and telemetry summarization. The workspace/runtime seam covers sterile execution workspaces, mirrored worker files, workspace-worker output syncing, and watchdog-driven subprocess termination.

Centralized test routing lives in `tests/conftest.py`. Any new test module must be added there or it will silently fall back to marker `core` instead of `llm`. The short current layout documentation lives in `docs/12-testing/12-testing_README.md` and should reflect any new intentional split seam.

## Plan of Work

First, create a dedicated knowledge-runtime test file under `tests/llm/` and move the selected knowledge orchestrator runtime/progress tests into it. Use the existing split-file pattern from the recipe orchestrator transport tests: import the base module, reuse its non-test imports and helpers, and keep the new file narrow. As part of the move, strengthen the assertions so the progress tests prove packet totals, active-task text, worker/session counts, and packet-lease finalization rather than only generic completion.

Second, create a dedicated direct-exec workspace test file under `tests/llm/` and move the workspace preparation and subprocess supervision tests there. Keep the pure helper/classification tests in the base `test_codex_exec_runner.py` file. Tighten the moved assertions so they check that the sterile execution workspace is the actual command cwd, that worker-manifest entry files match the intended mode, and that workspace-worker supervision only completes after synced outputs appear in the source root.

Third, trim the moved tests from the original files, update `tests/conftest.py` marker routing, and update the short testing documentation so the new split seams are explicit.

## Validation and Acceptance

The work is complete when both new focused modules exist, run directly, and contain the runtime tests that previously lived only inside the larger files. The remaining base files should still pass their targeted non-runtime slices. The strengthened assertions should prove exact runtime contracts such as packet totals, worker-session counts, sterile execution cwd selection, worker-manifest entry files, and packet-lease finalization.

## Idempotence and Recovery

This plan changes only tests and nearby docs. Re-running the test commands is safe. If the split introduces confusion, revert the new focused modules and restore the moved tests to their original files; no production data or generated artifacts are at risk.
