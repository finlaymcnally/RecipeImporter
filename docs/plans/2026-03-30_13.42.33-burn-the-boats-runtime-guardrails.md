# Burn The Boats Runtime Guardrails

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan is intentionally sequenced after the current burn-the-boats runtime trilogy:

- `docs/plans/2026-03-30_12.50.13-burn-the-boats-fixed-assignment-worker-reset.md`
- `docs/plans/2026-03-30_12.59.42-burn-the-boats-single-editable-task-file-reset.md`
- `docs/plans/2026-03-30_13.13.48-burn-the-boats-nonlethal-single-file-worker-runtime.md`

Those three plans define the main runtime rewrite: fixed assignment instead of lease loops, one editable `task.json` instead of multi-file shell choreography, and non-lethal supervision instead of heuristic watchdog slaughter. This follow-on plan adds the operational guardrails that keep that new design honest in production. It makes three things explicit and testable: actual worker sessions must stay bounded by the planned per-stage cap except for explicit repair, `task.json` size pressure must surface as warnings in interactive setup and run artifacts instead of silently becoming a new token bomb, and observability must live in repo-owned artifacts unless the worker truly needs it on the happy path.

## Structured Restatement Of User Desire

The new runtime shape is already chosen, but it still needs hard guardrails so it cannot drift back into expensive ceremony. The repository should enforce a simple truth: if a stage was planned to use at most N worker sessions, the live happy path must not quietly create more than N sessions unless deterministic repair explicitly asks for more work afterward. The repository should also measure the size of each worker-visible `task.json` and warn the operator when an assignment is getting large enough to threaten token efficiency, but it should do that through interactive setup text and durable run artifacts rather than by silently exploding the shard count again. Finally, the repository should make a structural distinction between information that the worker needs to do semantic work and information that the repo wants for debugging or analytics. The worker should see only the former. The latter should stay repo-side by default.

## Purpose / Big Picture

After this change, a user can trust the new single-file runtime to stay within the cost shape it advertises. If interactive setup says a stage will run with five workers, the run should not quietly fan out into twenty worker sessions unless explicit repair later adds a bounded follow-up worker. If a planned `task.json` is large enough to be risky, the operator should see that before or during the run through a warning in the interactive CLI and through durable run artifacts, rather than discovering it later from a giant token bill. If the repo needs rich traces, snapshots, or bookkeeping, those artifacts should live outside the worker-visible workspace unless they are truly needed for the worker to complete its job.

The proof is concrete and observable. Focused tests should show that worker-session counts are bounded by the planned cap for the happy path. Interactive setup and run summaries should surface task-size pressure as warnings without silently changing planned shard or worker counts. The worker-visible root for the single-file runtime should still contain only the minimum information needed for the worker to do the job, while repo-owned telemetry and analytics artifacts remain available outside that root. A novice should be able to inspect a finished run and answer three questions without guessing: how many worker sessions were planned, how many actually ran, and whether any assignment file was large enough to deserve attention.

## Outcomes & Retrospective

This plan is newly authored and not yet implemented. Its intended outcome is a runtime that stays simple in the places that matter operationally, not just architecturally. The expected end state is that the repo can say, with evidence, that the single-file runtime is bounded, warn-first, and lean: no surprise worker fan-out on the happy path, no invisible growth of `task.json` into a new token sink, and no model-visible observability clutter that exists only for repo debugging.

The main lesson behind this plan is that simplification is not finished when the file layout gets cleaner. It is finished when the clean layout stays bounded under real runs and the repo stops leaking its own diagnostic concerns into the worker-visible happy path.

## Context And Orientation

This repository has three live Codex-backed worker stages that matter here: recipe, knowledge, and canonical line-role. The predecessor plans already define the target runtime shape for those stages: fixed assignment instead of repo-advanced leasing, one editable `task.json` instead of multi-file shell choreography, and non-lethal supervision instead of kill-first heuristics. This plan does not change that target. It adds guardrails around it.

A “planned per-stage cap” in this plan means the operator-visible upper bound on worker sessions for one stage before repair begins. In the new runtime shape, planning still creates shards or tasks, workers still process those owned units, and repair may still create a small follow-up assignment after deterministic validation. The guardrail here is that the happy path must not silently exceed that cap once the stage starts.

A “task-size warning” in this plan means a repo-authored warning emitted when a worker-visible `task.json` grows large enough to threaten token efficiency or operator expectations. The warning is informational. It does not silently rewrite the plan, silently add workers, or silently split work further. The warning belongs in two places: the interactive operator surface before launch when possible, and durable run artifacts after planning or execution so the user can see what happened later.

A “worker-visible happy path” in this plan means only the files and fields the worker genuinely needs to do semantic work. The predecessor plans already push toward a single-file runtime. This plan makes the default boundary explicit: traces, snapshots, analytics helpers, detailed manifests, and other repo-owned observability artifacts should stay outside the worker-visible root unless they are genuinely required for the worker to produce valid semantic answers.

The main implementation owners are spread across a few areas. `cookimport/llm/codex_exec_runner.py` and `cookimport/llm/phase_worker_runtime.py` own the shared direct-exec workspace story and worker lifecycle. `cookimport/llm/recipe_stage/`, `cookimport/llm/knowledge_stage/`, and `cookimport/parsing/canonical_line_roles/` own stage planning, validation, reporting, and promotion. `cookimport/cli_ui/run_settings_flow.py` and related CLI helpers own interactive setup and warnings. `cookimport/llm/prompt_budget.py`, `cookimport/runs/stage_observability.py`, analytics collectors, and benchmark helper readers own durable reporting after the run. The tests that already exercise these areas live under `tests/llm/`, `tests/parsing/`, `tests/cli/`, `tests/staging/`, `tests/labelstudio/`, and `tests/analytics/`.

This plan assumes one important precondition from the user: the earlier shard-count explosion bug is already fixed separately. The work here should therefore encode that assumption as a lasting contract rather than trying to re-debug the old failure.

## Plan Of Work

The first block of work is worker-cap enforcement. Update the shared planning and runtime seams so each stage records two separate values in a stable way: the planned happy-path worker-session cap and the actual worker sessions launched before repair. The implementation should treat the planned cap as authoritative. Recipe, knowledge, and line-role happy paths must never create more worker sessions than that cap. If deterministic validation later creates an explicit repair assignment, the repair session count must be tracked separately rather than folded into the happy-path count. This work belongs in the planning and runtime owners, not just in analytics. The code must prevent the bad state, not merely describe it after the fact.

The second block of work is task-size measurement. The planner or runner that materializes worker-visible `task.json` files must compute deterministic size metadata for each file before launch. This metadata should include at least the serialized byte size of the final `task.json`, the unit count inside that file, and a rough token-pressure estimate derived from the actual serialized text the worker will read. The exact estimation method may be approximate, but it must be stable, deterministic, and tied to the real file content. This metadata belongs in repo-owned planning or run artifacts so later analytics and benchmarks can read it without guessing from folder layout.

The third block of work is warning surfaces. Interactive setup should inspect planned task-size metadata after shard planning but before execution starts and show concise warnings when one or more planned assignments cross a configured size threshold. The wording should explain that the system is not silently changing shard or worker counts and that the warning exists because the planned `task.json` size may hurt token efficiency. The same warning data should also land in durable run artifacts so the operator can inspect it later from run summaries, stage observability, or prompt-budget style artifacts even when the run was not launched interactively.

The fourth block of work is observability placement. Review the worker-visible root for the new single-file runtime and move any file that exists only for repo debugging, reporting, or analytics outside that root by default. Keep only the files and fields that the worker genuinely needs to complete semantic work. In the final single-file shape, the expected default is that the worker-visible root contains only `task.json`. If an exception exists, the plan implementation must justify it in code comments, tests, and docs as a true worker need rather than a convenience for the repo.

The fifth block of work is honest reporting. Update prompt-budget, stage observability, and any benchmark-facing postmortem summaries so they report planned happy-path cap, actual happy-path sessions launched, explicit repair sessions launched, task-size warning status, and any max observed `task.json` size metrics. The key behavioral requirement is that a reader can tell the difference between “the stage planned for five workers and used five workers” and “the stage planned for five workers, used five workers, and later launched one repair session.” The reporting should not bury repair inside the happy-path totals.

The sixth block of work is documentation and tests. Update the active LLM and CLI docs so they teach these guardrails as part of the live runtime contract. Add or update focused tests that fail when happy-path worker-session counts exceed the planned cap, fail when warning metadata is missing from interactive and durable run surfaces, and fail when worker-visible roots expose repo-only observability files on the single-file happy path.

## Milestones

### Milestone 1: Worker-session caps become a real runtime contract

At the end of this milestone, the shared runtime and stage owners treat planned happy-path worker count as a hard cap rather than a soft aspiration. The user-visible result is that runs stop surprising the operator with silent extra worker sessions before repair begins. The proof is a focused test that plans a stage with a fixed cap, runs the planner/runtime seam under fake execution, and shows that the recorded happy-path worker-session count never exceeds that cap.

### Milestone 2: `task.json` size pressure becomes visible without becoming automatic planner behavior

At the end of this milestone, planned assignments record size metadata for the actual `task.json` files the worker will read, and large assignments surface as warnings in both interactive setup and durable run artifacts. The user-visible result is that the operator gets advance notice when a one-file assignment is getting big enough to threaten token efficiency, but the system does not secretly change the plan behind the operator’s back. The proof is one focused CLI test that renders the warning and one focused artifact test that shows the same warning data survives into run summaries.

### Milestone 3: Repo-side observability stays out of the worker-visible happy path

At the end of this milestone, the new single-file runtime exposes only worker-needed information in the visible root by default. Repo-owned traces, snapshots, and analytics helpers live outside that root. The user-visible result is that the worker-facing workspace remains minimal even as post-run observability stays rich. The proof is a focused runner test that inspects the visible root and a second test that inspects repo-owned artifacts and shows that the hidden observability still exists where the repo needs it.

### Milestone 4: Reporting and docs tell the bounded-runtime story clearly

At the end of this milestone, prompt-budget, stage-observability, and active docs all describe the same contract: planned happy-path cap, actual happy-path sessions, explicit repair sessions, task-size warning state, and repo-side observability by default. The proof is that the focused tests pass and the active docs teach these guardrails as part of the live runtime story rather than as optional conventions.

## Validation And Acceptance

This work is accepted only when the new runtime is operationally bounded as well as architecturally simpler. The first acceptance criterion is worker-session enforcement. For each live stage, the implementation must record the planned happy-path cap and the actual happy-path sessions launched, and the latter must never exceed the former. Any repair sessions must be counted separately and must arise only from explicit deterministic repair flow.

The second acceptance criterion is warning visibility. Planned `task.json` size pressure must surface as operator-visible warnings in interactive setup when that surface is in use and must also survive into durable run artifacts after planning or execution. The warning must be tied to the real worker-visible file and must not silently rewrite planned worker or shard counts.

The third acceptance criterion is worker-visible minimalism. The worker-visible happy path for the new single-file runtime must expose only information that the worker genuinely needs. Repo-owned observability artifacts must remain available outside that root. If a repo-only helper file is still visible on the happy path, the implementation must either remove it or justify it as a true worker dependency in docs and tests.

The fourth acceptance criterion is honest reporting. Prompt-budget summaries, stage observability, and related run summaries must distinguish between planned happy-path cap, actual happy-path sessions, explicit repair sessions, and task-size warning state. A user should be able to inspect the run data and tell whether a stage stayed inside plan without reading raw logs.

The fifth acceptance criterion is documentation clarity. The active docs must explain that the new runtime is bounded by planned worker-session caps, warns on large `task.json` assignments, and keeps observability repo-side by default. A novice should not have to infer these rules from code or from historical plans.

## Idempotence And Recovery

This plan is safe to implement incrementally because each guardrail is testable on its own and can be added without running live Codex workloads. If one stage has not yet adopted the guardrail, keep the reporting for that stage explicit rather than pretending the contract already exists. Do not satisfy the plan by adding only post-hoc analytics; the runtime and planner seams must enforce the cap behavior directly.

If the size-warning thresholds need tuning during implementation, keep the thresholds configurable in code but the presence of warning metadata non-optional. The operator may later tune the warning line, but the repository should not regress into “no warning at all.” If a worker-visible file appears to need repo-only observability data, prefer moving the repo-owned artifact elsewhere and teaching the runner or reporting layer how to find it there before making it visible again.
