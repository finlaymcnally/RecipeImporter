# Heading OTHER With Short OTHER Grouping Context

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the gold set and live knowledge prompts will stop treating section headings as `KNOWLEDGE`. Headings will be labeled `OTHER`, which should reduce the current benchmark tendency to overcall short heading rows as knowledge and bring the gold policy in line with the desired editorial rule.

At the same time, knowledge pass 2 will not lose the useful structure those headings provide. Grouping will still operate only on rows that survived pass 1 as `keep_for_review`, but it will also receive short `OTHER` rows as context-only rows when those rows are `<=6` words and `<=45` characters. Those context-only rows must appear in the same reading order as the kept knowledge rows so the model can still see heading guidance, but the context-only rows must never become standalone grouped outputs. A human should be able to open a grouping packet or same-session `task.json`, see the kept rows interleaved with short `OTHER` context rows, and verify that only the kept rows are groupable.

## Surprises & Discoveries

- Observation: the current classification prompt actively encourages some short headings to survive as knowledge.
  Evidence: `cookimport/llm/knowledge_stage/structured_session_contract.py`, `cookimport/llm/knowledge_stage/task_file_contracts.py`, and `cookimport/llm/knowledge_stage/recovery.py` all currently include language such as “Short conceptual headings can still be knowledge” and “Keep a short action-key or strategy heading...”.

- Observation: the current grouping packet only has one authoritative row list, and validation assumes that every row in that list must be covered by exactly one group.
  Evidence: `build_knowledge_structured_prompt(...)` in `cookimport/llm/knowledge_stage/structured_session_contract.py` says “Every owned row in `rows` already survived the binary review and must belong to exactly one contiguous group”, and `validate_knowledge_grouping_task_file(...)` rejects missing, duplicate, overlapping, and non-contiguous coverage over those kept rows.

- Observation: the current packet already has separate “context” surfaces for classification, but grouping does not yet have an interleaved reading-order view that can show short `OTHER` rows between kept rows.
  Evidence: `structured_session_contract.py` already emits `context_before_rows`, `context_after_rows`, and `row_facts`, but those are edge-only or metadata-only surfaces and do not preserve the mixed reading order the user wants for grouping.

- Observation: the benchmark gap that motivated this plan is overwhelmingly a `KNOWLEDGE` versus `OTHER` disagreement, not a recipe-structure problem.
  Evidence: benchmark run `data/golden/benchmark-vs-golden/2026-04-13_14.28.08/.../codex-exec/wrong_label_lines.jsonl` contains 256 misses, of which 214 are `OTHER -> KNOWLEDGE` and 26 are `KNOWLEDGE -> OTHER`.

## Decision Log

- Decision: headings should be labeled `OTHER` in the gold set and live knowledge-classification guidance.
  Rationale: the user wants to stop fighting the model’s tendency to view short heading rows as non-knowledge while still preserving the headings as structural guidance for later grouping.
  Date/Author: 2026-04-13 / Codex

- Decision: the first implementation will use only the two explicit inclusion thresholds for grouping context rows: `<=6` words and `<=45` characters.
  Rationale: the user explicitly rejected extra heading-shape heuristics for the first implementation. If the thresholds are later tuned, that should happen from benchmark evidence, not from a more ornate first-pass rule.
  Date/Author: 2026-04-13 / Codex

- Decision: short qualifying `OTHER` rows will be context-only rows, not normal grouping rows.
  Rationale: grouping outputs and validation must remain authoritative only for `keep_for_review` rows. If context-only rows were allowed to behave like normal owned rows, the validator and final promotion logic would accidentally treat them as final knowledge.
  Date/Author: 2026-04-13 / Codex

- Decision: grouping packets will keep the current authoritative `rows` list for groupable kept rows and add a separate read-only mixed reading-order surface that interleaves kept rows and context-only `OTHER` rows.
  Rationale: this preserves the existing coverage and repair invariants for grouping while giving the model the row-order view the user wants. Overloading the existing `rows` field would break current validation semantics.
  Date/Author: 2026-04-13 / Codex

- Decision: the locality guard for short `OTHER` rows will be batch-local and conservative.
  Rationale: the packet should include only qualifying `OTHER` rows that are relevant to the kept rows in the current grouping batch. The implementation should not pull short `OTHER` rows from unrelated parts of the shard just because they satisfy the length thresholds.
  Date/Author: 2026-04-13 / Codex

3. final grouped outputs still contain only true `KNOWLEDGE` rows; and

## Context and Orientation

The knowledge stage in this repository is a two-pass non-recipe review system.

Pass 1 is the binary classification pass. It decides, row by row, whether a non-recipe row is worth carrying forward for knowledge review. The worker-visible category is `keep_for_review` or `other`. The main contract files for this pass are `cookimport/llm/knowledge_stage/structured_session_contract.py`, `cookimport/llm/knowledge_stage/task_file_contracts.py`, `cookimport/llm/knowledge_stage/recovery.py`, and `cookimport/llm/knowledge_prompt_builder.py`. Downstream code later converts `keep_for_review` into final `knowledge` and leaves `other` as final `other`.

Pass 2 is the grouping-and-grounding pass. It takes only the rows that survived pass 1, asks the model to partition them into contiguous groups, and attaches tag grounding to each group. The important invariant is that pass 2 groups only the kept rows. The kept rows are currently exposed as authoritative `rows`, and the validator in `cookimport/llm/knowledge_stage/task_file_contracts.py` assumes those rows must be covered exactly once by the returned groups.

The inline structured-session path lives in `cookimport/llm/knowledge_stage/structured_session_contract.py` and `cookimport/llm/knowledge_stage/workspace_run.py`. The same-session taskfile path lives in `cookimport/llm/knowledge_stage/task_file_contracts.py` and `cookimport/llm/knowledge_same_session_handoff.py`. Both paths need the same semantic behavior after this change: headings become `OTHER` in pass 1, and pass 2 sees qualifying short `OTHER` rows as context-only rows in mixed reading order while still grouping only the kept rows.

The benchmark evidence that motivated this plan is under `data/golden/benchmark-vs-golden/2026-04-13_14.28.08/single-book-benchmark/saltfatacidheatcutdown/codex-exec/`. The important point is that the current misses are dominated by `OTHER -> KNOWLEDGE` disagreements on explanatory prose and heading rows. The gold-policy part of this plan is meant to reduce those disagreements. The grouping-context part of this plan is meant to keep useful structural guidance that would otherwise be lost when headings move to `OTHER`.

## Plan of Work

Start by changing the semantic policy, not the batching code. The current classification prompts and tests explicitly teach the model that some short conceptual headings can be `knowledge`. Those instructions must be rewritten so heading rows are `other` by default. The new wording should preserve the binary first-pass shape, but the examples and rules must now say that a heading row can clarify nearby knowledge without itself being kept. The implementation should touch the shared prompt text and the prompt-builder tests together so the policy is visible in one place and verified in another.

Next, add one repo-owned helper that selects qualifying short `OTHER` rows for grouping context. This helper should operate on the original classification task rows plus the validated pass-1 answers. It should identify rows whose validated pass-1 category is `other` and whose raw text is both `<=6` words and `<=45` characters. No extra heading-shape, casing, punctuation, or title-style heuristic belongs in this first implementation. The helper must also apply the batch-local locality guard. The intended behavior is conservative: include qualifying `OTHER` rows only when they belong to the local reading run around the kept rows in the current grouping batch, not from unrelated parts of the shard.

Then split “groupable rows” from “display rows”. Keep the existing authoritative `rows` surface as the list of kept `keep_for_review` rows, because grouping validation and repair depend on that invariant. Add a new read-only mixed reading-order surface, named consistently across inline packets and same-session task files, that interleaves kept rows with context-only short `OTHER` rows. The kept rows in this mixed surface must still show their stable `rXX` ids. The context-only rows should use a different visible identifier such as `ctx01`, `ctx02`, and so on, so the model cannot confuse them with groupable rows. Each mixed row should include the original classification category, block index, and text in a compact stable format.

Update the grouping prompts after that data surface exists. The prompt must say clearly that only the `rXX` rows listed in the authoritative `rows` field are eligible for grouping. The mixed reading-order field exists only to help the model understand the local run. The prompt must explicitly say that `ctxXX` or equivalent context-only rows are structural context only and must not appear in group spans, must not be treated as standalone knowledge, and must not change the required exact coverage of the authoritative kept rows.

Once the prompt and packet surfaces are in place, thread the new field through both execution paths. The inline path will need packet-construction changes in `cookimport/llm/knowledge_stage/structured_session_contract.py` and likely small orchestration adjustments in `cookimport/llm/knowledge_stage/workspace_run.py`. The same-session path will need taskfile-construction updates in `cookimport/llm/knowledge_stage/task_file_contracts.py` and status/handoff wording updates in `cookimport/llm/knowledge_same_session_handoff.py`. Repair prompts must inherit the same context-only surface so repaired grouping batches still see the short `OTHER` rows around their kept rows.

Finally, update the long-lived docs and add focused tests. The durable docs should record the new contract: headings are `OTHER`; pass 1 remains binary; pass 2 may see qualifying short `OTHER` rows as context-only mixed reading-order rows; and final groups still contain only knowledge rows. The tests must prove both sides of the tradeoff: headings stop surviving classification just because they are headings, and grouping still sees short `OTHER` rows without promoting them into final groups.

## Validation and Acceptance

The implementation is acceptable only when all of the following are true:

1. knowledge classification guidance no longer teaches the model that short headings can survive as knowledge simply because they name a useful concept;
2. a heading row that is labeled `other` can still appear in pass-2 grouping context when it is `<=6` words and `<=45` characters and is local to the kept rows in the current batch;
3. grouping packets and same-session task files show the qualifying short `OTHER` rows in the same reading order as the kept rows;
4. returned group spans still cover only the authoritative kept `rXX` rows, not the context-only rows;
5. final `idea_groups` and final per-block `knowledge` decisions still come only from pass-1 kept rows and never from context-only `OTHER` rows; and

## Idempotence and Recovery

This change is safe to iterate on because it is contract- and prompt-focused. The main risk is accidentally widening the definition of groupable rows. To recover from that failure, inspect the grouping packet or same-session `task.json` and confirm that:

1. `rows` still lists only the kept `rXX` rows;
2. the mixed reading-order field includes both kept rows and qualifying short `OTHER` rows;
3. context-only rows use a different visible identifier from `rXX`; and
4. the validator still rejects any attempt to reference a context-only row inside a returned group span.

If prompt tests start failing unexpectedly, prefer updating the wording in one shared surface and then aligning the focused prompt-builder and contract tests to that wording, rather than hand-editing scattered expectations without confirming the user-visible contract.
