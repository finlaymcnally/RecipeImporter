# Speed up canonical line-role codex labeling with bounded parallel batches and cache reuse

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md` from the repository root.

## Purpose / Big Picture

After this change, canonical line-role codex labeling for benchmark/import runs no longer sends unresolved batches one-by-one. It sends up to four codex batches in flight per book, retries transient codex failures with bounded backoff, and reuses cached line-role outputs keyed by source hash and run settings so reruns can skip codex calls.

Operators will see shorter wall-clock time in the `Running canonical line-role pipeline...` phase and repeated runs on the same source/settings should hit cache and return quickly.

## Surprises & Discoveries

- Observation: the status panel can show `active workers: 0` during canonical line-role phase even while work is active because this phase waits on codex subprocess/network latency, not split-worker CPU tasks.
  Evidence: live run prompt artifacts advanced (`prompt_0009` -> `response_0009` -> `prompt_0010`) while CPU stayed low.

- Observation: current canonical line-role codex path is strictly serial over `_batch(...)` and uses a single `_PromptLogState` with non-thread-safe counter/dedup writes.
  Evidence: `cookimport/parsing/canonical_line_roles.py` loop over `for batch in _batch(...)` and mutable `_PromptLogState` internals.

## Decision Log

- Decision: implement bounded concurrency with a fixed default of four in-flight codex batches per book.
  Rationale: user explicitly requested “4 agents per book” for this stage, and observed workload is network-bound with low CPU.
  Date/Author: 2026-03-04 / assistant

- Decision: keep deterministic output assembly by collecting concurrent batch results then applying them in stable prompt-index order.
  Rationale: prevents output nondeterminism and keeps artifacts reproducible regardless of future completion order.
  Date/Author: 2026-03-04 / assistant

- Decision: cache final canonical line-role predictions on disk keyed by source hash plus run-settings hash and candidate fingerprint.
  Rationale: satisfies rerun reuse requirement while preventing stale-hit mismatches when parser output or settings change.
  Date/Author: 2026-03-04 / assistant

## Outcomes & Retrospective

Canonical line-role codex escalation is now latency-hiding instead of strictly serial. Per-book execution runs up to four in-flight codex batches, reassembles results deterministically, and keeps prompt artifact writing safe under concurrency. Retries absorb transient codex failures before fallback paths trigger, and disk cache reuse can skip codex entirely on repeat runs with the same source hash + settings + candidate fingerprint.

Scope stayed constrained to parsing/ingest seam and parsing docs/tests, matching the original user request without widening into unrelated benchmark scheduler layers.

## Context and Orientation

The canonical line-role engine lives in `cookimport/parsing/canonical_line_roles.py`. Function `label_atomic_lines(...)` computes deterministic rule labels first, then for `line_role_pipeline=codex-line-role-v1` escalates unresolved/low-confidence candidates to codex prompts. The current codex section creates prompt artifacts under `<artifact_root>/line-role-pipeline/prompts` and runs one batch at a time with `codex_batch_size=40`.

The main production caller is `cookimport/labelstudio/ingest.py` in the benchmark/import prediction path (`_notify("Running canonical line-role pipeline...")` then `label_atomic_lines(...)`). `file_hash` is already computed there and can be used as the source-hash cache key component.

Relevant test module is `tests/parsing/test_canonical_line_roles.py`.

## Plan of Work

Implement in `cookimport/parsing/canonical_line_roles.py`:

1. Add bounded thread-pool execution for codex batch tasks with default max in-flight = 4 per book.
2. Refactor single-batch codex logic into a worker helper that:
   - writes prompt artifact,
   - runs codex call with retry/backoff for transient failures,
   - writes response/parsed artifacts,
   - returns structured per-batch results.
3. Make `_PromptLogState` thread-safe:
   - lock counter increment,
   - lock dedup-file read/write append path.
4. Ensure deterministic merge:
   - assign stable prompt index before submit,
   - collect futures in any order,
   - apply batch outputs in ascending prompt index.
5. Add disk cache helpers:
   - resolve cache root,
   - compute cache key from source hash + settings + candidate fingerprint,
   - load/validate cached predictions,
   - atomically write cache after successful run.
6. Extend `label_atomic_lines(...)` signature with optional `source_hash` and `cache_root` to support cache keying and routing.

Implement in `cookimport/labelstudio/ingest.py`:

1. Pass `source_hash=file_hash` into `label_atomic_lines(...)` so cache key has source identity.
2. Keep existing behavior for runs without cache root/context.

Implement tests in `tests/parsing/test_canonical_line_roles.py`:

1. Verify deterministic output order and label mapping with out-of-order future completions.
2. Verify retry/backoff path retries transient codex failures and eventually succeeds.
3. Verify cache hit skips codex runner calls on second invocation with same source/settings/candidates.
4. Verify prompt logging artifacts remain coherent under concurrent execution.

## Validation and Acceptance

Acceptance criteria:

- Canonical line-role codex phase can run multiple batches concurrently (bounded at 4 in-flight per book by default).
- Prompt artifacts (`prompt_*.txt`, `response_*.txt`, `parsed_*.json`, dedup log, parse summary) remain valid and non-corrupt under concurrency.
- Final prediction output remains deterministic and stable by atomic index regardless of future completion order.
- Transient codex failures are retried with backoff before falling back.
- Second run with same source/settings/candidates can reuse cached predictions and skip codex calls.
- Targeted parsing tests pass.

## Idempotence and Recovery

The code changes are additive and safe to rerun. Cache writes use deterministic keys and should be safe to regenerate. If caching behavior is suspect, remove cache files under the resolved line-role cache root and rerun; functional output should still be correct without cache.
