This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

The knowledge-stage ordered-queue rollout fixed the original packet-progression bug: workers now receive `assigned_tasks.json` up front and can finish many more packets before stopping. The next failure is different. In the saved benchmark run at `data/golden/benchmark-vs-golden/2026-03-21_16.10.40/single-book-benchmark/saltfatacidheatcutdown/codexfarm/`, knowledge workers often produce output files, but too many of those files are rejected as invalid.

After this change, the knowledge worker contract must be hard to misunderstand. A worker prompt must name the exact allowed semantic categories, the ingest layer must recover a narrow set of known legacy aliases instead of failing the whole packet, and snippet guidance must keep the model from pasting entire chunks into `snippets[].body`. The observable outcome is that targeted local tests and saved-run replay-style checks show fewer `schema_invalid` packet failures and more validated packets, without spending tokens on a new live Codex run.

- Observation: the schema is much stricter than the prompt text implies.
  Evidence: `cookimport/llm/codex_farm_knowledge_models.py` defines `KnowledgeSemanticBlockDecisionV1.category` as `Literal["knowledge", "other"]`, with `reviewer_category` constrained to a fixed enum such as `knowledge`, `decorative_heading`, `endorsement_or_marketing`, and `other`.

## Outcomes & Retrospective

This plan is not implemented yet. The main outcome of the investigation is that the previous knowledge fix was successful in its own scope: workers no longer depend on mid-session lease mutation. The remaining work is now a cleaner problem. The stage needs a sharper semantic output contract and a small amount of deterministic compatibility normalization so that valid intent is not discarded as `schema_invalid`.

The likely residual risk, even after this plan lands, is that some outputs will still be rejected for semantic quality reasons such as full-chunk snippet echo or blanket `all_other` judgments on strong-cue shards. That is acceptable. Those are model-behavior failures worth surfacing, whereas the current category drift is a preventable contract failure.

## Context and Orientation

The knowledge stage is the optional LLM review pass that refines seed non-recipe spans and extracts reusable cooking knowledge. In this codebase, a "task packet" is the smallest workspace-worker job for this stage. Each task packet owns one or more non-recipe chunks and produces one semantic packet result JSON file.

The key runtime modules are:

- `cookimport/llm/codex_farm_knowledge_orchestrator.py`: builds worker prompts, prepares `assigned_tasks.json`, and runs the knowledge workspace workers.
- `cookimport/llm/codex_farm_knowledge_ingest.py`: normalizes worker payloads and validates whether a task packet result is acceptable.
- `cookimport/llm/codex_farm_knowledge_models.py`: defines the strict semantic packet schema and the canonical compact bundle schema.
- `cookimport/llm/knowledge_runtime_replay.py`: reconstructs packet outcomes from saved runtime artifacts without rerunning Codex.
- `tests/llm/test_knowledge_output_ingest.py`: focused unit tests for normalization and validation behavior.
- `tests/llm/test_codex_farm_knowledge_orchestrator.py`: prompt and worker-manifest regressions for the knowledge stage.
- `tests/llm/test_knowledge_runtime_replay.py`: replay-style tests that prove the runtime artifact interpretation matches real saved runs.
- `docs/10-llm/nonrecipe_knowledge_review.md`: short operator-facing note describing the knowledge stage contract.

Two schema terms matter here:

- Final `category`: the coarse final ownership written into the semantic packet. It is intentionally only `knowledge` or `other`.
- `reviewer_category`: the richer internal reason label attached to an `other` decision, such as `decorative_heading` or `endorsement_or_marketing`.

Today the prompt does not spell that distinction out clearly enough, so the model invents intermediate labels like `content`, `noise`, and `heading`. The ingest layer then rejects those outputs before the stage can benefit from the worker’s otherwise usable work.

## Plan of Work

### 1. Make the worker contract explicit in the prompt

Edit `_build_knowledge_workspace_worker_prompt(...)` in `cookimport/llm/codex_farm_knowledge_orchestrator.py`.

Replace the current vague field description with explicit allowed values:

- state that `category` must be exactly `knowledge` or `other`
- enumerate the allowed `reviewer_category` values
- explain that `reviewer_category` must be `knowledge` whenever `category == "knowledge"`
- add a negative instruction that values such as `content`, `noise`, and `heading` are invalid and must never be emitted
- state that `snippets[].body` must be a short grounded extraction, not a pasted block dump or a whole-chunk quote collection

Keep this concrete. The worker should not have to infer the contract from surrounding prose.

### 2. Add a narrow semantic-category compatibility normalizer

Edit `normalize_knowledge_worker_payload(...)` in `cookimport/llm/codex_farm_knowledge_ingest.py`.

Before validating the semantic packet against `KnowledgePacketSemanticResultV1`, add a helper that walks `chunk_results[*].block_decisions[*]` and rewrites only these known aliases:

- `content` -> `category = "knowledge"`, default `reviewer_category = "knowledge"`
- `noise` -> `category = "other"`, default `reviewer_category = "endorsement_or_marketing"`
- `heading` -> `category = "other"`, default `reviewer_category = "decorative_heading"`

This helper must:

- leave already-valid values unchanged
- attach metadata describing that compatibility alias mapping was used and how many decisions were rewritten
- preserve explicit valid `reviewer_category` values when present
- still raise a schema error for any unknown category string outside the known alias set

Do not broaden this into fuzzy text matching. The goal is deterministic recovery of the exact saved-run regression, not a soft parser.

### 3. Keep semantic-quality validation strict, but make it easier for the worker to satisfy

Do not relax `semantic_snippet_echoes_full_chunk` or `semantic_all_false_empty_shard` in `cookimport/llm/codex_farm_knowledge_ingest.py`.

Instead:

- strengthen prompt wording so snippets are framed as extracted claims, not transcript copies
- add one or two concise prompt bullets about acceptable snippet size and grounding
- ensure the tests cover the saved malformed endorsement example shape so the prompt contract and validator work together instead of fighting each other

If implementation reveals that a tiny deterministic cleanup is safe, limit it to structurally obvious noise only. Do not summarize or rewrite snippet bodies in the ingest layer.

### 4. Expand the tests around the saved failure shapes

In `tests/llm/test_knowledge_output_ingest.py`, add regressions for:

- semantic alias mapping for `content`, `noise`, and `heading`
- metadata recording that compatibility mapping was used
- unknown category values still failing with `schema_invalid`
- full-chunk or near-full-chunk snippet bodies still failing with `semantic_snippet_echoes_full_chunk`

In `tests/llm/test_codex_farm_knowledge_orchestrator.py`, update the prompt assertions so they require:

- explicit `category` values `knowledge|other`
- explicit allowed `reviewer_category` inventory
- a clear warning not to emit `content`, `noise`, or `heading`
- a clear warning not to paste entire chunks into snippet bodies

In `tests/llm/test_knowledge_runtime_replay.py`, add or extend a replay-style regression that uses the saved March 21 runtime shape to prove the stage is now counted as "worker wrote output but output was semantically rejected" only when that rejection is genuine, not because of category alias drift.

Prefer small fixture payloads copied from the saved run over a huge golden directory dependency. The plan goal is a fast, local proof.

### 5. Update the short docs note

Once the implementation is settled, update `docs/10-llm/nonrecipe_knowledge_review.md` with a short note that the main workspace-worker contract is:

- semantic packet output with final `category` values limited to `knowledge|other`
- richer reviewer-only reasons carried in `reviewer_category`
- snippet bodies must be grounded extractions rather than whole-chunk echoes

Keep the doc update short. This is a contract clarification, not a redesign.

## Idempotence and Recovery

The prompt edits and test additions are safe to repeat. The compatibility normalizer should also be idempotent: running it twice must leave the payload unchanged after the first rewrite.

If a partial implementation causes the prompt and tests to disagree, fix the prompt tests in the same patch that changes the prompt wording so the contract stays explicit. If the compatibility normalizer starts accepting unknown category values, stop and narrow it immediately; silent over-acceptance would hide future worker regressions.

No rollback beyond normal git hygiene should be needed. The safe recovery path is to revert only the prompt/ingest patch under development and re-run the narrow ingest tests.
