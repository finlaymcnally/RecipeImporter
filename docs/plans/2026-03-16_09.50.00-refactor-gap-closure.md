# Close The Remaining Refactor Gaps

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan builds on the already-checked-in phase plans at `docs/plans/phase1-observability-first.md`, `docs/plans/phase2-label-source-of-truth.md`, `docs/plans/phase3-collapse-recipe-reasoning.md`, and `docs/plans/phase4-shrink-knowledge-pipeline.md`. ## Purpose / Big Picture

After this change, the code, docs, tests, and run outputs should all tell the same story as `docs/reports/Refactor.md` and the OG phase plans. Stage 2 labels stay authoritative even when they disagree with importer heuristics. The canonical recipe Codex path is visibly `codex-farm-single-correction-v1` everywhere a new user looks. Stage 7 owns non-recipe classification without live runtime dependence on `ConversionResult.non_recipe_blocks`. New benchmark, review, and follow-up surfaces stop teaching pass-slot or pass4-first worldview as current truth.

The user-visible proof is concrete. A new `cookimport stage` run on a small book should write `label_det`, optional `label_llm_correct`, `group_recipe_spans`, `08_nonrecipe_spans.json`, `09_knowledge_outputs.json`, and semantic recipe/knowledge observability rows with no candidate-first fallback and no old recipe pipeline names in help text, docs, manifests, overviews, or starter-pack rendering. Old saved recipe pipeline values may still load as aliases where the phase plans explicitly allowed that, but new writes and user-facing descriptions must show only the current architecture.

- Observation: the largest remaining Phase 2 mismatch is not in labeling code; it is a defensive fallback in the stage session.
  Evidence: `cookimport/staging/import_session.py` currently discards the label-first bundle and restores `original_result` when the authoritative regrouping produces zero recipes even though the importer had found some.

- Observation: the canonical single-correction runtime is already implemented, but several user-facing surfaces still advertise the legacy names as if they were current options.
  Evidence: `cookimport/config/run_settings.py` only allows `off` and `codex-farm-single-correction-v1`, while `cookimport/cli.py`, `docs/02-cli/02-cli_README.md`, `docs/03-ingestion/03-ingestion_readme.md`, and stale CLI tests still describe `codex-farm-3pass-v1` and `codex-farm-2stage-repair-v1` as primary values.

- Observation: some compatibility readers are now harmless load-path aliases, but others still leak into user-facing benchmark/report text.
  Evidence: `cookimport/bench/upload_bundle_v1_existing_output.py`, `tests/bench/test_upload_bundle_v1_existing_output.py`, and `tests/bench/test_benchmark_cutdown_for_external_ai.py` still surface `codex-farm-2stage-repair-v1` / `Merged Repair` as live product topology rather than as historical normalization.

## Context and Orientation

This repository stages cookbook imports under `data/output/<timestamp>/`. A stage run starts from importer output, writes intermediate and final recipe artifacts, and also writes machine-readable debug and benchmark artifacts that later review tooling consumes.

For this plan, “authoritative label-first path” means the runtime path where Stage 2 labeled lines and normalized block labels decide recipe spans and non-recipe spans before any recipe-drafting or knowledge-extraction work. In this repository, that path is created by `cookimport/parsing/label_source_of_truth.py` and consumed by `cookimport/staging/import_session.py` and `cookimport/labelstudio/ingest.py`.

“Single-correction naming” means the canonical Phase 3 recipe Codex contract: `llm_recipe_pipeline=codex-farm-single-correction-v1` at the run-settings/CLI layer and `recipe.correction.compact.v1` at the compact pack layer. Old values such as `codex-farm-2stage-repair-v1` and `codex-farm-3pass-v1` may still appear only as load-time aliases for older saved payloads. They must not remain the preferred names in help text, docs, or new rendered outputs.

“Stage 7” means the deterministic non-recipe classifier in `cookimport/staging/nonrecipe_stage.py`. It consumes the authoritative block stream, authoritative final labels, and recipe spans, then classifies all outside-recipe blocks as `knowledge` or `other`. The canonical machine-readable outputs are `08_nonrecipe_spans.json` and `09_knowledge_outputs.json` at the run root.

The main files that need work are:

- `cookimport/staging/import_session.py`: main stage-backed runtime orchestration. This is where Phase 2 authority fallback and Phase 4 old-field reuse still live.
- `cookimport/labelstudio/ingest.py`: prediction and benchmark orchestration. This still carries some `non_recipe_blocks` and historical merge plumbing because it bridges old and new run shapes.
- `cookimport/parsing/label_source_of_truth.py`: authoritative label-first models and adapter. This file defines the Phase 2 handoff types and is where diagnostics should be added if label-first grouping produces surprising results.
- `cookimport/staging/nonrecipe_stage.py`: Stage 7 classification. This should remain the only ownership source for outside-recipe text.
- `cookimport/llm/codex_farm_orchestrator.py` and `cookimport/llm/codex_farm_knowledge_orchestrator.py`: recipe and knowledge Codex execution. These already mostly follow the new phase plans, but their reports and downstream renderers still need the final naming cleanup.
- `cookimport/runs/stage_observability.py`: canonical stage index. New runs should expose only the semantic stage vocabulary from the phase plans.
- `cookimport/cli.py`, `cookimport/cli_ui/run_settings_flow.py`, `cookimport/cli_ui/README.md`, `docs/02-cli/02-cli_README.md`, `docs/03-ingestion/03-ingestion_readme.md`, `docs/07-bench/07-bench_README.md`: these are the main user-facing surfaces that still advertise stale pre-cutover names or pass4-first behavior.
- `cookimport/bench/upload_bundle_v1_existing_output.py`, `cookimport/bench/followup_bundle.py`, and the related tests: benchmark/reviewer surfaces that still need to stop teaching old product truth for new-format runs.

## Milestone 1: Make Phase 2 truly authoritative

At the end of this milestone, stage-backed runs no longer have a supported candidate-first escape hatch. If the authoritative label-first path finds zero recipe spans in a source where the importer had candidate recipes, the run remains truthful about that fact instead of silently restoring the old ownership path.

Edit `cookimport/staging/import_session.py` first. Remove the branch that sets `label_first_result = None` and restores `original_result`. Replace it with a diagnostic path that stays on the label-first bundle. The diagnostic should be easy to inspect in the run root and in tests. Acceptable implementations are a warning in the report plus a JSON payload written beside `recipe_spans.json`, or a warning-only implementation if that proves sufficient in tests. The important rule is that `result` must continue to come from the label-first bundle rather than from importer candidates.

Then inspect `cookimport/labelstudio/ingest.py` for any remaining candidate-first restoration paths or plan-mode logic that assumes `result.non_recipe_blocks` or importer recipes are authoritative. The current plan-mode and processed-output flows already build `authoritative_label_result`; this milestone should make that bundle the stable source of truth even when counts are surprising.

Add or update targeted tests. Extend staging coverage so there is an explicit regression test for the “importer had recipes, label-first regrouping found zero” case and assert that the runtime stays on the authoritative bundle and surfaces a diagnostic instead of reverting. Keep the test local to staging or parsing; do not hide this behavior behind broad integration-only coverage.

Acceptance for this milestone is behavioral: there is no branch in the main stage runtime that silently restores `original_result` when the label-first bundle disagrees, and the new regression test proves that fact.

## Milestone 2: Make Stage 7 the only live non-recipe authority

At the end of this milestone, Stage 7 still may populate `ConversionResult.non_recipe_blocks` as a compatibility cache, but no Stage 4+ runtime decision uses that field as authority. Tables, chunks, plan-mode knowledge counts, optional knowledge extraction, and stage-block predictions all derive from Stage 7 inputs directly.

Update `cookimport/staging/import_session.py` so `extract_and_annotate_tables(...)` and chunk generation operate on Stage 7-derived block rows built from `archive_blocks` plus `NonRecipeStageResult`. If the existing helper signatures make this awkward, add narrow helper functions under `cookimport/staging/` that convert a `NonRecipeStageResult` or one `NonRecipeSpan` into the row format those helpers expect. Do not route through `result.non_recipe_blocks` first.

Update the plan-mode path in `cookimport/labelstudio/ingest.py` the same way. The current code already computes `nonrecipe_stage_result` before knowledge plan rendering. Make the plan counts and chunk counts come from Stage 7-derived rows or Stage 7 span math directly rather than from `result.non_recipe_blocks`.

Audit any remaining runtime reads of `result.non_recipe_blocks` in stage-backed code and either move them to Stage 7-derived inputs or explicitly leave them as non-authoritative compatibility-only behavior with a comment that says so. The goal is not to delete the field from every importer in this milestone. The goal is to burn the authority bridge in every live stage/benchmark/prediction codepath.

Acceptance for this milestone is that Stage 7 remains the only ownership source for outside-recipe text in stage-backed paths, while `result.non_recipe_blocks` is at most an output/cache shape and no longer drives runtime decisions.
