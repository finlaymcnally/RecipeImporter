# Clarify Stage 7 As Non-Recipe Knowledge Review

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a human reading the runtime docs, prompt previews, progress output, and stage artifacts should be able to answer a simple question without guessing: “What does the third Codex-backed phase do?” The answer should be: it reviews non-recipe chunks, keeps or extracts useful cooking knowledge, and filters out low-value non-recipe noise. The user-visible win is not a brand-new model capability. The win is that the stage stops feeling like a vague “do something with leftovers” pass and becomes an explicit, inspectable contract with clear inputs, outputs, and success criteria.

The observable outcome is that a contributor can run a zero-token preview or fake-run rehearsal, open the knowledge-stage artifacts, and see four things immediately: what non-recipe spans seeded the stage, what chunks were skipped before the model as noise, what chunk bundles were reviewed by the model, and what final knowledge snippets survived into `knowledge.md` and `snippets.jsonl`. The same wording should appear in docs, progress labels, manifests, and reviewer-facing exports.

## Context and Orientation

In this repository, “Stage 7” refers to the Codex-backed pass over the non-recipe side of a processed cookbook. A “non-recipe span” is a contiguous run of source blocks that were not grouped into recipes. A “chunk” is a smaller review unit built from those blocks by deterministic chunking logic in `cookimport/parsing/chunks.py`. A “bundle” or “shard” is one owned model task containing one or more chunks. A “worker” is one bounded execution context that owns one or more shards. A “proposal” is the model’s structured output for a shard, which is later validated and promoted by deterministic code.

The key files are:

- `cookimport/llm/codex_farm_knowledge_orchestrator.py` orchestrates the whole knowledge stage. It chooses whether the stage runs, writes the shard input files, launches direct Codex workers, reads validated outputs, and writes final reports.
- `cookimport/llm/codex_farm_knowledge_jobs.py` decides what reaches the model. It turns seed non-recipe spans into deterministic chunks, skips obvious noise, bundles the surviving chunks into shard payloads, and writes immutable JSON task files.
- `cookimport/llm/codex_farm_knowledge_models.py` defines the model-facing output contract. The important fields are per-chunk usefulness, per-block final category decisions, and evidence-backed snippets.
- `cookimport/llm/codex_farm_knowledge_ingest.py` validates that each shard result covers exactly the owned `chunk_id`s and does not point evidence outside the shard’s allowed block surface.
- `cookimport/llm/codex_farm_knowledge_writer.py` writes reviewer-facing `snippets.jsonl` and `knowledge.md`.
- `docs/10-llm/10-llm_README.md` and `docs/04-parsing/04-parsing_readme.md` are the human-facing source of truth for how this stage is supposed to work.

Today the mechanical contract is already close to the desired story, and the docs have started to catch up. The confusing part is that a user can still hit mixed language depending on where they look: the README may say “review non-recipe chunks and skip noise,” while CLI labels or prompt-sample outputs still say “knowledge harvest.” This plan exists to finish that last-mile alignment and to make the runtime contract explicit everywhere a human looks.

## Plan of Work

Start with naming and explanation, but assume the README work is partly done. Audit the remaining human-facing labels in the knowledge runtime and adjacent debug surfaces. That includes CLI progress messages, stage observability labels, prompt-sample headings, prompt artifact summaries, and any tests that still encode the old “knowledge harvest” wording. Change those labels and descriptions so they consistently say some version of “non-recipe knowledge review” or equally plain language. Preserve stable file paths such as `knowledge_manifest.json` and `extract_knowledge_optional`; the goal is better explanation, not broad path churn.

Next, strengthen the stage manifest and reporting seam. In `cookimport/llm/codex_farm_knowledge_orchestrator.py`, expand the knowledge-stage report so it explicitly records the count of seed non-recipe spans, the count of chunks built before pruning, the count of skipped `noise` or low-signal chunks, the count of shard bundles reviewed by Codex, and the count of promoted useful chunks or snippets. The existing `llm_report`, `knowledge_manifest.json`, and runtime artifact pointers should become sufficient for a novice to answer what the stage actually did. If any of this data is currently available only indirectly, compute it once in the orchestrator from `KnowledgeJobBuildReport`, validation summaries, and `KnowledgeWriteReport`.

Then tighten the prompt/debug and observability surfaces. The preview, prompt-artifact, and stage-observability layers should make it obvious that the model receives immutable chunk-bundle inputs and returns validated structured outputs. If the current knowledge prompt preview or reviewer-facing prompt exports obscure the fact that the stage works on chunk bundles, update those descriptions. The goal is not to redesign the prompt itself first. The goal is to make the prompt/export surface teach the right mental model and make the same artifact seams discoverable from tooling.

After that, update the docs only where the current worktree still lags. `docs/10-llm/10-llm_README.md` and `docs/10-llm/knowledge_harvest.md` already carry much of the intended story, so the remaining edits should be narrow consistency fixes rather than a second full rewrite. In `docs/04-parsing/04-parsing_readme.md`, keep the parser-owned responsibility explicit: `chunks.py` decides what non-recipe material becomes eligible for review and what gets suppressed as noise before the model is called.

Finally, add tests that prove the clarified contract. The right tests are not only low-level parsing tests. Add or update assertions in the knowledge-stage orchestration and preview tests so they fail if the manifest/report no longer exposes skipped-chunk counts, shard ownership, promoted snippet counts, or the clarified stage labels. The plan is complete when a zero-token dry or preview run makes Stage 7 legible without reading code.

## Validation and Acceptance

Acceptance is behavioral and documentary.

First, the focused knowledge tests must pass. They should prove that deterministic chunk building still skips noise, shard outputs still require exact owned `chunk_id` coverage, reviewer-facing snippet writing still works, the orchestrator still promotes only validated outputs, and the stage-observability / prompt-artifact surfaces still resolve the correct knowledge input and proposal directories.

Second, a zero-token preview must produce artifacts that teach the new mental model. A human opening the preview output should be able to say:

1. “These non-recipe spans seeded the stage.”
2. “These chunks were skipped before the model because they were noise or low-signal.”
3. “These shard bundles were actually reviewed by the model.”
4. “These final snippets are the promoted cooking knowledge outputs.”

Third, the docs must match the runtime. `docs/10-llm/10-llm_README.md`, `docs/10-llm/knowledge_harvest.md`, and `docs/04-parsing/04-parsing_readme.md` should describe the stage in the same terms the runtime manifests, CLI labels, and previews use. A human should not have to choose between contradictory descriptions like “knowledge harvest,” “leftover lines,” and “non-recipe chunk review.”

Fourth, the implementation must not spend live tokens unless explicit approval is granted. The default validation loop for this plan is tests plus zero-token preview and, if needed, zero-token fake-run rehearsal.

## Idempotence and Recovery

This plan should be implemented additively and safely. Most steps are doc, reporting, and validation changes around an existing runtime. Re-running the focused tests is safe. Re-running prompt preview is safe as long as a fresh timestamped output directory is used under `data/output/`.

If a label or artifact rename becomes too disruptive partway through, prefer restoring the stable machine-facing path and narrowing the change to human-facing labels plus manifest/report fields. Do not strand the repo in a half-renamed state where docs and tests disagree about the active artifact names.

If a rehearsal or preview produces confusing results, keep the output directory and write a short `docs/understandings/...` note rather than deleting the artifacts. This stage is exactly the kind of place where preserved evidence is more useful than a clean tree.
