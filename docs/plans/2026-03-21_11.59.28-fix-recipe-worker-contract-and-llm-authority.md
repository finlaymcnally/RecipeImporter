# Fix recipe worker contract and trust the LLM on recipe semantics

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the recipe correction stage should stop wasting nearly all of its spend on avoidable repair calls. A fresh zero-context Codex worker should be able to open its local workspace, understand the exact output contract from the files in front of it, write valid `out/<task_id>.json` files on the first pass, and finish without the repo treating its harmless closing summary as a malformed shard result.

The user-visible success condition is simple. When a Codex-backed recipe run is executed again, the recipe stage should spend most of its tokens on the main worker pass rather than on repair, main worker task outputs should validate directly without a mandatory second call for every task, and deterministic code should only enforce hard invariants such as exact owned ids and exact schema rather than overriding the model's recipe judgments, ingredient-step links, or tags.

## Surprises & Discoveries

- Observation: the main recipe workers were not "thinking badly" so much as speaking the wrong contract.
  Evidence: the preserved worker commands show writes such as `out/recipe-shard-0000-r0000-r0006.task-001.json` with payload keys `results`, `recipe_id`, `not_a_recipe`, and `notes`, while the repair prompt later quotes the validator error `results Extra inputs are not permitted`.

- Observation: the current worker prompt is not how a zero-context Codex worker would want to be briefed.
  Evidence: `cookimport/llm/codex_farm_orchestrator.py::_build_recipe_workspace_worker_prompt(...)` tells the worker to preserve `not_a_recipe` and `fragmentary`, but never shows the active compact schema or a valid output example. A fresh worker naturally emits the old human-readable keys.

- Observation: the prompt and the runtime disagree about the meaning of the final worker message.
  Evidence: the prompt says "Do not return the shard outputs in your final message. The authoritative result is the set of files written under `out/`.", but `cookimport/llm/codex_exec_runner.py::assess_final_agent_message(...)` still labels any non-JSON closing message as `malformed`, and recipe telemetry records that malformed state on all main worker tasks.

- Observation: recipe repair currently salvages many semantically acceptable answers, which means the main path is already close to useful.
  Evidence: the March 21 promotion report kept `17 repaired`, `6 not_a_recipe`, and `4 fragmentary` results after repair. The issue is first-pass contract alignment, not a fundamental inability to do the recipe task.

- Observation: most of the recipe-stage spend came from invalid main worker output, not from the cheap structured repairs themselves.
  Evidence: `prompt_budget_summary.json` for the run reports `19,627,345` recipe tokens total, `18,864,812` tokens in `main_workspace_workers`, `762,533` tokens in structured followups, and `18,864,812` invalid output tokens. The main path is where the waste lives.

- Observation: the live recipe validator was more permissive than the schema the workers were supposed to follow.
  Evidence: `RecipeCorrectionShardOutput` uses `populate_by_name=True`, so repo tests could still pass verbose keys like `recipe_id` and `canonical_recipe` even though the workspace contract and JSON Schema required `rid` and `cr`. The fix had to add an explicit compact-key gate in `cookimport/llm/codex_farm_orchestrator.py`.

- Observation: the workspace manifest already had the right seam for worker-visible contract files.
  Evidence: adding `OUTPUT_CONTRACT.md` and `examples/` to the mirrored workspace only required extending `cookimport/llm/codex_exec_runner.py` manifest/copy plumbing; the worker prompt and sterile workspace were already built around repo-written root files.

  Rationale: the workers here are the same family of model as the implementation agent. If the workspace would confuse a zero-context Codex worker, it is a bad workspace.

  Rationale: the whole point of the workspace-worker transport is to move authority into repo-owned files under `out/`. Penalizing a harmless prose signoff contradicts that model and forces unnecessary repairs.
  Date/Author: 2026-03-21 / Codex

- Decision: keep deterministic code strong on invariants and weak on semantics.
  Rationale: deterministic recipe-side heuristics in this repository are already known to be materially worse than the intended LLM correction behavior. The repo should enforce exact ownership, exact schema, required fields, and safe normalization, but it should not second-guess valid model judgments about `repaired` versus `fragmentary`, ingredient-step alignment, or tags.
  Date/Author: 2026-03-21 / Codex

- Decision: do not remove repair entirely in this pass.
  Rationale: repair is still the right escape hatch for genuine near-miss invalid outputs. The goal of this plan is to stop needing repair for every task, not to delete the recovery seam before the main contract is trustworthy.
  Date/Author: 2026-03-21 / Codex

- Decision: add compact contract examples directly into the worker-visible workspace, not only into Python source or hidden schema files.
  Rationale: a zero-context worker should not have to reverse-engineer abbreviations such as `rid`, `st`, `sr`, `cr`, `m`, `mr`, `g`, and `w` from a JSON schema or guess whether legacy names are accepted.

## Context and Orientation

The recipe correction stage lives primarily in `cookimport/llm/codex_farm_orchestrator.py`. That file builds the worker prompt, writes `assigned_tasks.json`, `in/*.json`, and `hints/*.md`, launches one long-lived workspace worker per worker assignment, reads `out/*.json`, evaluates each task response, optionally runs one structured repair attempt, and then promotes validated outputs into staged recipe artifacts.

The direct Codex execution transport lives in `cookimport/llm/codex_exec_runner.py`. This module defines the two workspace modes, writes the worker manifest into the sterile workspace, captures the worker's final agent message, and summarizes telemetry rows such as `final_agent_message_state`, `output_payload_present`, `command_execution_count`, and token usage.

The active recipe output contract lives in two places:

- `cookimport/llm/codex_farm_contracts.py`, which defines `RecipeCorrectionShardOutput` and `MergedRecipeRepairOutput` using compact aliases such as `sid`, `r`, `rid`, `st`, `sr`, and `cr`.
- `llm_pipelines/schemas/recipe.correction.v1.output.schema.json`, which mirrors the same compact schema as JSON Schema.

The important current mismatch is this:

- The worker prompt uses semantic prose such as "preserve `not_a_recipe` and `fragmentary`", but does not show the compact keys required by the live contract.
- The workers naturally write legacy human-readable JSON using keys like `results`, `recipe_id`, `not_a_recipe`, `fragmentary`, and `notes`.
- The repo evaluates those files against the compact contract, marks them invalid, and then runs a structured repair call that explicitly teaches the compact shape.

Definitions used in this plan:

- A "workspace worker" is one long-lived Codex session inside a sterile local folder that reads repo-written files and writes repo-consumed outputs under `out/`.
- A "task" is one recipe-local input file under `in/<task_id>.json`, usually owning one recipe candidate.
- A "compact recipe output" is the accepted live output object with top-level keys `v`, `sid`, `r` and per-recipe keys such as `rid`, `st`, `sr`, `cr`, `m`, `mr`, `g`, and `w`.
- A "legacy recipe output" is the more human-readable shape the workers emitted in the March 21 run, with keys like `results`, `recipe_id`, `not_a_recipe`, `fragmentary`, and `notes`.
- An "invariant" is something deterministic code is objectively good at checking: exact owned ids, exact schema, exact required fields, array lengths, integer ranges, and safe lossless normalization.
- A "semantic judgment" is something the model should own once it produces a valid contract-conforming answer: whether a candidate is `repaired`, `fragmentary`, or `not_a_recipe`, what the canonical recipe title is, how ingredients map to steps, and which grounded tags to keep.

The observed bad run to keep in mind is `data/golden/benchmark-vs-golden/2026-03-21_11.17.08/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/`. Its saved prompt and manifest artifacts are the evidence base for this plan.

## Plan of Work

The first milestone is to make the recipe workspace self-explanatory to a zero-context Codex worker. Update `cookimport/llm/codex_farm_orchestrator.py::_build_recipe_workspace_worker_prompt(...)` so it no longer relies on a worker to infer the compact alias contract from prose. The prompt must say, plainly and early, that the accepted file format is the compact schema, that legacy keys are invalid, and that the worker should copy the shape from the provided workspace examples rather than inventing field names. Keep the prompt short enough to stay readable, but stop assuming that semantic prose alone is enough.

In the same milestone, add worker-visible sidecar files in the recipe workspace. The worker should receive a tiny `OUTPUT_CONTRACT.md` or similarly named file at the worker root that explains each compact key in plain English, shows one fully valid `repaired` object, one valid `fragmentary` object, and one valid `not_a_recipe` object, and explicitly lists forbidden legacy keys. The worker should also receive one machine-readable example file, such as `examples/valid_recipe_task_output.json`, with the exact shape to imitate. The prompt should tell the worker to open `worker_manifest.json`, then `OUTPUT_CONTRACT.md`, then the named task files. This is the setup that a fresh Codex worker would actually want.

The second milestone is to align the worker hints with that contract. `cookimport/llm/codex_farm_orchestrator.py::_write_recipe_worker_hint(...)` should stop assuming the worker already knows how compact recipe outputs map back to semantic concepts. The hint file should keep the useful local recipe summaries, but it should add a short reminder of the accepted status vocabulary and should explain when to choose `repaired`, `fragmentary`, or `not_a_recipe` in terms of evidence sufficiency. It should not instruct the worker to "be deterministic" or to defer to heuristic recipe hints over the raw source rows.

The third milestone is to change workspace-worker acceptance so file outputs are authoritative and the final agent message is optional. `cookimport/llm/codex_exec_runner.py::assess_final_agent_message(...)` is currently generic and marks any non-JSON signoff as malformed. That is correct for structured one-shot runs and wrong for workspace workers that were explicitly told not to return outputs in the final message. Introduce a workspace-worker-aware acceptance path so recipe worker telemetry can record the closing message as absent or informational without poisoning the task result when `out/*.json` files validate successfully. The worker may still end with JSON, but it must no longer be required.

The fourth milestone is to narrow deterministic post-LLM intervention in the recipe phase. In `cookimport/llm/codex_farm_orchestrator.py`, review the recipe response evaluation and promotion path around `_evaluate_recipe_response(...)`, repair decisions, proposal assembly, and `_recipe_result_rows_from_proposals(...)`. Preserve hard validation and normalization. Do not let deterministic code rewrite a valid `st`, `sr`, `cr`, `m`, `mr`, `g`, or `w` payload just because a heuristic disagrees. Deterministic code may reject outputs that violate ownership or schema, may normalize accepted compact fields into the staged recipe model, and may add trace metadata, but it should not demote or rewrite semantically valid LLM outputs.

The fifth milestone is to strengthen the fake and test transport so the repository continuously exercises the same contract the real workers must satisfy. `cookimport/llm/fake_codex_farm_runner.py` already knows the compact recipe shape; use it as the model for new fixture examples and regression tests. The tests should make it obvious when the worker prompt or workspace files drift back toward the legacy contract.

The sixth milestone is documentation. Update `docs/10-llm/10-llm_README.md` to explain that recipe workspace workers now receive explicit compact-contract examples and that deterministic recipe promotion enforces invariants only. Add one very short note near the recipe runtime documentation or artifact folder description explaining that `out/*.json` file validity, not the worker's closing prose, is the authoritative success signal.

## Idempotence and Recovery

This plan is intentionally additive and retry-safe.

Rewriting the worker prompt, sidecar files, and validator behavior is idempotent. The tests can be run repeatedly. If a partial implementation breaks the recipe worker path, the recovery strategy is to keep the old repair path intact until the new prompt, examples, and workspace-worker acceptance tests are all passing.

The most sensitive change is workspace-worker final-message handling in `cookimport/llm/codex_exec_runner.py`. Keep that change scoped so it only relaxes workspace-worker acceptance. Do not weaken the strict JSON requirement for structured repair calls.
