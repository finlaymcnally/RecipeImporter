# Improve Worker-Facing Prompt, Input, and Hint Contracts For Codex Stages

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The user-visible goal is to make the long-lived Codex worker loop actually useful for recipeimport quality work. After this change, a worker that opens its assigned local files should be able to answer the real stage question from the files alone instead of spending most of its effort reconstructing context, guessing what the deterministic hints meant, or being forced into tiny ambiguous spans that cannot be repaired honestly.

The intended result is not “more deterministic cleanup after the model.” It is the opposite. The model-owned `out/*.json` file for each task should become the authoritative semantic answer once it passes shape and ownership validation. Deterministic code should prepare better task packets, validate coverage, and join completed outputs, but it should stop second-guessing valid model judgments after the fact.

This plan is correct when a future run shows three concrete improvements at once: the worker prompt tells the model exactly which local files to open and why; each stage exposes richer worker-facing hint files that are actually useful to a Codex-style agent; and the stage joiners accept validated model outputs as semantic authority instead of re-imposing deterministic relabeling or alignment decisions.

## Surprises & Discoveries

- Observation: the current line-role worker prompt is file-contract heavy but semantically underpowered.
  Evidence: `cookimport/parsing/canonical_line_roles.py` builds a worker prompt that explains `out/<shard_id>.json`, but the actual worker-facing `in/*.json` file only contains `[atomic_index, label_code, current_line]` tuples. In the sampled live worker input, the prompt never tells the worker what codes such as `L8` and `L9` mean, and it does not expose the richer neighboring context that exists elsewhere in the repo.

- Observation: the existing line-role `debug/*.json` file contains much better hint material than the billed `in/*.json`, but the worker contract does not treat it as first-class task input.
  Evidence: the sampled `debug/line-role-canonical-0001-a000000-a000353.json` includes `deterministic_label`, `rule_tags`, `escalation_reasons`, and `block_index`, while the billed `in/*.json` omits all of that. The worker prompt names only `in/<shard_id>.json` as required input.

- Observation: as a Codex-style worker, the current line-role hints do not feel very useful for hard cases.
  Evidence: the sampled shard starts with blurbs, table-of-contents entries, section headings, and lesson headings from *Salt Fat Acid Heat*. Those are exactly the rows that need neighborhood and section context, but the worker mostly sees bare lines plus opaque prior label codes. This matches the benchmark reports that still show `HOWTO_SECTION` confusion and weak `INSTRUCTION_LINE` recovery.

- Observation: the current recipe packet is often too thin to rescue honestly.
  Evidence: the sampled `recipe-shard-0000-r0000-r0005.json` asks the worker to correct six candidate recipes such as `The Four Elements of Good Cooking SALT What is Salt?` and `Poach It in Wine Roast It on a Bed of Fig Leaves`, but it provides almost no surrounding source context before or after the candidate span. A worker can see that these are suspicious, but it cannot confidently recover the true boundary or structure from the packet alone.

- Observation: the recipe tag guide is far smaller than the user’s stated web-app tag surface and is only mildly useful to a worker.
  Evidence: `cookimport/llm/recipe_tagging_guide.py` exposes nine broad categories with five example labels each. That is enough to hint at taxonomy shape, but not enough to support grounded selection from a larger known catalog when the real downstream goal is a much richer tag set.

- Observation: single-candidate recipe shards make the richer tag contract tractable.
  Evidence: after recipe correction defaulted to one candidate-owned shard per recipe, `cookimport/llm/recipe_tagging_guide.py` could safely emit a shard-local `recipe_tagging_guide.v3` with filtered `tg.s[*]` suggestions derived from that candidate’s text/hints instead of pretending one global tiny guide served a whole multi-recipe bundle.

- Observation: the knowledge packet is the most semantically explicit of the three, but it is still more “schema reminder” than “decision aid.”
  Evidence: `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md` defines `knowledge` versus `other` well, yet the sampled worker packet still provides little section-level orientation beyond chunk-local blocks. That is acceptable for obvious front matter and obvious technique prose, but weak for border cases such as headings, tables, and short reference fragments.

- Observation: the March 19 reports point to a semantics problem, not just a watchdog problem.
  Evidence: the workspace-worker restart report shows that workers now survive some local reads, yet benchmark quality is still held back by line-role misclassification and dead downstream stages. Better command tolerance alone will not fix packets that do not contain enough useful judgment context.

- Observation: the hint-sidecar design fit the current runtime with much less disruption than a full shard/task rewrite.
  Evidence: `cookimport/llm/codex_exec_runner.py` only needed a new `hints/` directory contract and worker-manifest fields, while the stage validators and `out/*.json` promotion paths stayed intact.

## Decision Log

- Decision: Keep the long-lived worker loop as the main operating model for all three stages.
  Rationale: This matches the user’s explicit decision and fits Codex’s strength at bounded file work over longer horizons better than one tiny fresh call per task.
  Date/Author: 2026-03-19 / Codex

- Decision: The authoritative semantic output for a completed task will be the model-written `out/<task_id>.json`, not a deterministic merge of model deltas back into a deterministic seed.
  Rationale: The user explicitly wants to avoid deterministic post-LLM decisions that overwrite better model labeling or alignment. Validation should check shape, ownership, and required fields, then accept the model’s answer.
  Date/Author: 2026-03-19 / Codex

- Decision: Prefer full owned output files over delta patches.
  Rationale: A full owned file is easier for a long-lived worker to reason about, easier to resume after interruption, and safer than applying deterministic diffs that could partially preserve bad seed semantics.
  Date/Author: 2026-03-19 / Codex

- Decision: Introduce a first-class worker hint sidecar for every stage instead of asking the worker to infer meaning from debug-only files.
  Rationale: The current debug artifacts contain useful metadata, but they are not shaped or explained as worker-facing inputs. A small explicit sidecar is a cleaner design than hoping the worker explores the right debug file.
  Date/Author: 2026-03-19 / Codex

- Decision: Keep authoritative machine inputs compact, but stop optimizing so hard for compactness that the worker loses the context it needs to make good decisions.
  Rationale: The current line-role packet is compact in transport terms, but it removes exactly the metadata that would help a worker discriminate cookbook lesson headings from recipe-internal headings.
  Date/Author: 2026-03-19 / Codex

- Decision: Recipe correction tasks should usually own one candidate recipe at a time once packet redesign begins.
  Rationale: The current multi-candidate recipe shard shape hides boundary mistakes inside a bundle. A worker can answer “is this candidate a recipe, and if so repair it” much more reliably when each task owns one candidate and its immediate boundary context.
  Date/Author: 2026-03-19 / Codex

- Decision: Land the worker hint sidecars before attempting the more disruptive recipe shard split.
  Rationale: The sidecars improve worker usability immediately, preserve current validators and worker loops, and create a better baseline for deciding whether one-candidate recipe tasks are still necessary afterward.
  Date/Author: 2026-03-19 / Codex

- Decision: Keep recipe one-candidate ownership as the default runtime contract, and reinterpret `recipe_prompt_target_count` as a worker-session target when `recipe_worker_count` is unset.
  Rationale: the long-lived worker loop already gives one worker many files to process. That means recipe quality no longer needs multi-candidate bundling just to limit fresh sessions, and explicit multi-recipe bundling can remain available only through `recipe_shard_target_recipes` for deliberate opt-out cases.
  Date/Author: 2026-03-19 / Codex

## Outcomes & Retrospective

The three intended implementation slices are now landed. The runtime still uses the same long-lived worker loop, but workers now get a real `hints/*.md` decision layer across line-role, knowledge, and recipe; recipe correction defaults to one candidate-owned shard per recipe; the recipe/knowledge prompt text now matches the richer packet shape better; and recipe promotion no longer re-inherits deterministic seed tags after a valid model output.

The most important result is practical rather than architectural: a worker folder is now easier to read and the recipe stage now asks a more honest question. A reviewer can open `worker_manifest.json`, `assigned_shards.json`, `hints/<shard_id>.md`, and `in/<shard_id>.json` and usually judge one candidate at a time instead of decoding a multi-recipe bundle. The main remaining gap is benchmark proof, which the user explicitly deferred to a separate run.

## Context and Orientation

RecipeImport currently runs three Codex-backed review stages relevant to this plan.

`cookimport/parsing/canonical_line_roles.py` owns the canonical line-role stage. It writes worker-local `assigned_shards.json`, compact `in/*.json`, richer `debug/*.json`, and harvests `out/*.json` after a long-lived workspace-worker session. The live worker prompt is built in `_build_line_role_workspace_worker_prompt(...)`. The compact file-prompt formatter lives in `cookimport/llm/canonical_line_role_prompt.py`.

`cookimport/llm/codex_farm_knowledge_orchestrator.py` owns non-recipe knowledge review. It writes worker-local `in/*.json` bundle files and harvests one `out/*.json` knowledge result per shard. The one-shot prompt template lives in `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md`, and `cookimport/llm/knowledge_prompt_builder.py` renders it.

`cookimport/llm/codex_farm_orchestrator.py` owns recipe correction. It writes worker-local `in/*.json` shard files containing candidate recipe evidence, deterministic hints, and a compact tag guide. The one-shot prompt template lives in `llm_pipelines/prompts/recipe.correction.compact.v1.prompt.md`. The shared broad tag guide lives in `cookimport/llm/recipe_tagging_guide.py`.

In this plan, a “task packet” means the full set of local files a worker needs to complete one owned unit of work. An “authoritative input” file is the strict machine-readable file the validator depends on. A “worker hint sidecar” is a second local file, written by the repo, whose only purpose is to help the worker make better semantic decisions. A “validator” means code that checks coverage, IDs, schema, and other deterministic contract rules. The validator must not silently rewrite a valid model answer into a different semantic answer.

## Plan of Work

The first change is to stop treating worker-local helper context as an accidental debug artifact. Each stage should write two explicit files per owned task. The first remains the authoritative machine input at `in/<task_id>.json`. The second is a worker-facing hint sidecar, preferably at `hints/<task_id>.md` plus an optional structured `hints/<task_id>.json` when the stage has dense metadata. The worker prompt must tell the model to open both files for every task. The hint sidecar must be short, task-specific, and written in plain language. It should answer: what kind of thing this task probably is, what the ambiguous edges are, what nearby evidence matters, and which deterministic hints are weak versus strong.

For line-role, edit `cookimport/parsing/canonical_line_roles.py` and `cookimport/llm/canonical_line_role_prompt.py` together. The current `in/*.json` tuple rows are too skinny for hard disambiguation. Keep them for validation-friendly ownership if desired, but add a worker-facing hint file that includes the label-code legend, the deterministic label in plain text, a short local neighborhood window for each ambiguous row, and a compact summary of row-level hazards such as “outside recipe,” “heading-like,” “ingredient-like,” or “generic teaching prose candidate.” Do not ask the worker to infer these from opaque codes. The line-role sidecar should also summarize section rhythm for the packet, for example “front matter blurbs,” “table of contents slice,” “lesson headings with explanatory prose,” or “recipe body.” This is the stage where the current hints are least personally useful to me as a tagging worker, so it should be the first redesign target.

For knowledge review, edit `cookimport/llm/codex_farm_knowledge_orchestrator.py` and `cookimport/llm/knowledge_prompt_builder.py`. The current prompt definitions are decent, but the task packet still needs better local orientation. Add a hint sidecar that tells the worker why the chunk bundle exists, whether nearby recipe blocks were excluded, what larger book section the chunk appears to belong to, and which chunks are likely decorative headings, navigation, or real reference material. Preserve the current grounded evidence rules and snippet requirements, but add more positive borderline examples to the prompt template so short tables, charts, and headings with technical meaning are not discarded just because they are concise.

For recipe correction, edit `cookimport/llm/codex_farm_orchestrator.py`, `llm_pipelines/prompts/recipe.correction.compact.v1.prompt.md`, and `cookimport/llm/recipe_tagging_guide.py`. Change the owned work unit so one task usually owns one candidate recipe, not a six-candidate bundle. Extend each task packet with immediate pre-span and post-span source rows, section-title context, and the line-role evidence that caused the candidate span to exist. The worker needs to answer three separate questions in order: is this actually a recipe, where are its true boundaries, and if it is a recipe what are the corrected ingredients, steps, tags, and ingredient-step links? Right now the packet mostly asks the third question without enough evidence for the first two. The tag system also needs a better worker contract: either expose the real supported tag surface in a compact searchable form or provide a stage-local filtered candidate tag list derived from source text, but stop pretending nine broad categories are enough when the downstream app wants something much richer.

After the task packets are richer, simplify the worker prompt builders in `cookimport/parsing/canonical_line_roles.py`, `cookimport/llm/codex_farm_knowledge_orchestrator.py`, and `cookimport/llm/codex_farm_orchestrator.py`. The worker prompt should focus on loop discipline, file ownership, and output rules. The semantic distinctions should mostly live in the stage prompt template and the hint sidecar. The current prompts are too dominated by transport instructions because they are compensating for thin task data. Once the packets improve, shorten the contract text and point the worker to the stage hint files first.

The deterministic joiners must then be audited. For line-role, recipe, and knowledge, locate every place where a valid model output is later transformed into a different semantic answer. Remove those rewrites unless they are pure normalization that preserves meaning, such as sorting, key renaming, or derived artifact generation. In particular, do not let deterministic post-processing reassign labels, reject ingredient-step links solely because they do not match a deterministic guess, or flatten recipe outputs back into deterministic boundaries when the model produced a valid grounded correction. The only allowed deterministic override after validation should be fail-closed rejection of invalid ownership, invalid schema, or impossible references.

Finally, document the new packet contract in `docs/10-llm/10-llm_README.md` and leave a short note near the relevant runtime folder if implementation adds new worker-facing folders such as `hints/`. The docs should explain that the system now separates strict input ownership from worker-facing semantic guidance, and that the authoritative answer is the validated model-owned output file.

## Validation and Acceptance

The implementation is acceptable only if all of the following are true.

First, a novice can inspect one worker folder from each stage and immediately understand how a worker was supposed to make the decision. Concretely, each task should expose an authoritative input file plus a readable hint sidecar. The worker prompt should name those files directly.

Second, the line-role packet should be personally useful to a Codex-style worker. A reviewer should be able to open one sampled `hints/<task_id>.md` file and see why a row is a likely lesson heading, likely front matter, likely recipe structure, or likely generic teaching prose without needing to cross-reference hidden repo state.

Third, the recipe packet should make honest rejection possible. A reviewer should be able to open one task for a suspicious candidate and see enough source context to say “this is not a recipe,” “this is fragmentary,” or “this is a real recipe but the deterministic span cut it wrong.”

Fourth, deterministic code must not semantically override a valid model answer after validation. This should be proven with focused tests that feed a valid model-style output through the joiner and assert that the same semantic answer survives into the promoted artifact.

Fifth, if a live benchmark is later approved, success should show up in the same failure regions named by the March 19 reports: fewer outside-recipe `HOWTO_SECTION` promotions, better `INSTRUCTION_LINE` recovery, nonzero validated downstream knowledge or recipe shards, and less reliance on seed-only fallback.

## Idempotence and Recovery

This work is additive and safe to retry if implemented carefully. Writing new `hints/` sidecars is non-destructive. Changing worker prompts is safe as long as the authoritative `in/*.json` and `out/*.json` contract remains stable during migration.

If a stage migration is only partially complete, keep the old packet reader and the new packet reader both working until the narrow tests pass. Do not remove old fields from authoritative `in/*.json` until every validator and prompt-preview path is updated. If a worker session is interrupted, any already-written `out/*.json` files should still be harvestable exactly as today; the new hint sidecars are read-only and do not change resume semantics.

## Artifacts and Notes

These short examples capture why the current packets need redesign.

Line-role live worker prompt currently says:

    - For each assigned shard, open `in/<shard_id>.json` directly.
    - Convert `label_code` into the correct full label string.

But the sampled worker file contains rows like:

    [0, "L8", "Praise for SALT FAT ACID HEAT"]

That is not enough by itself to make hard heading decisions confidently, and the prompt does not include the code legend.

Recipe live worker input currently includes suspicious candidates like:

    "txt":"The Four Elements of Good Cooking SALT What is Salt?"
    "f":["source_no_ingredient_lines","source_no_instruction_lines","hint_no_steps"]

That is a useful warning that the candidate is bad, but not enough context to repair or reject it confidently if the true answer depends on nearby omitted rows.

The current tag guide is intentionally tiny:

    dish_type: soup, stew, salad, sandwich, cake
    protein: chicken, beef, tofu, shrimp, beans

That is fine as a fallback taxonomy hint, but not enough if the real downstream requirement is selection from a much larger supported tag surface.

## Interfaces and Dependencies

Keep the design local to the existing stage modules. Do not introduce a new orchestration framework.

The implementation should leave these broad interfaces true at the end:

In `cookimport/parsing/canonical_line_roles.py`, the worker assignment writer should emit:

    in/<task_id>.json
    hints/<task_id>.md
    out/<task_id>.json

and the worker prompt should explicitly instruct the model to open both `in/` and `hints/` files.

In `cookimport/llm/codex_farm_knowledge_orchestrator.py`, the knowledge worker assignment writer should emit the same pattern and keep current result validation semantics for `bid`, `cid`, snippet evidence, and chunk coverage.

In `cookimport/llm/codex_farm_orchestrator.py`, the recipe worker assignment writer should move toward one candidate recipe per task, preserve current compact output keys, and keep current ownership validation for `sid`, `rid`, and compact mapping/tag structures.

In all three joiners, valid model-owned outputs must pass through as semantic authority after validation. Deterministic code may reject invalid ownership or impossible references, but it must not relabel or reinterpret a valid model answer merely because a deterministic hint disagrees.
