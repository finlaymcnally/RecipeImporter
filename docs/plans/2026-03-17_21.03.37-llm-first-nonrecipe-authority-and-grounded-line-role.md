This plan supersedes [docs/plans/2026-03-17_20.58.22-reground-line-role-without-reintroducing-rollbacks.md](/home/mcnal/projects/recipeimport/docs/plans/2026-03-17_20.58.22-reground-line-role-without-reintroducing-rollbacks.md) and absorbs the narrower tactical plan [docs/plans/2026-03-17_21.01.17-improve-knowledge-heading-and-prose-disambiguation.md](/home/mcnal/projects/recipeimport/docs/plans/2026-03-17_21.01.17-improve-knowledge-heading-and-prose-disambiguation.md). The older regrounding plan correctly identified that the current line-role prompt is under-grounded, but it still framed the solution as mostly repairing a deterministic seed plus one grounded correction prompt. The narrower heading/prose plan captured real benchmark failure buckets, but it still treated them mostly as a line-role heuristic problem. The user has now made the intended product constraint explicit: deterministic accuracy is not the product, the LLM-corrected path is. This replacement plan keeps the useful grounding work and the concrete benchmark buckets, but reorganizes the fix around LLM authority instead of deterministic recovery.

## Purpose / Big Picture

After this change, RecipeImport should stop asking one compressed line-role pass to solve the hardest whole-book semantic problem from starved evidence. The line-role LLM should still fix recipe structure, but outside-recipe meaning should be decided by an LLM-first non-recipe authority pass that sees bounded non-recipe spans, local neighbors, and coarse semantic hints instead of isolated line text. The user-visible outcome is that a cookbook benchmark such as `saltfatacidheatcutdown` can improve because the model is finally making `KNOWLEDGE` versus `OTHER` decisions in the seam that actually has enough context, rather than because deterministic rules got a little less bad.

Someone validating this change should be able to rerun the single-offline benchmark and see three concrete things. First, recipe-local labels such as `INGREDIENT_LINE` and `INSTRUCTION_LINE` remain strong. Second, obvious front matter, table of contents, chapter taxonomy, endorsements, and memoir/storytelling prose stop competing with cooking-science explanation inside the same starved line-role prompt. Third, final scored outside-recipe labels come from a richer Stage 7 LLM review path that explains why each span became `knowledge` or `other`.

## Outcomes & Retrospective

The implementation pass completed the architectural split but did not yet run the real benchmark acceptance step. Line-role is now narrower and more trustworthy: the prompt stays compact, carries span/hint metadata, and exposes explicit local `ctx:` windows only for the rows that need grounding. Stage 7 remains the durable non-recipe authority seam: knowledge bundles now carry richer internal reviewer context, but staging/export artifacts still collapse back to final `knowledge` versus `other`.

The important result is that the code now matches the intended product shape. The line-role LLM is no longer the only place asked to infer hard whole-book outside-recipe meaning from bare line text, and the knowledge stage now has a place to think in cookbook-native reviewer categories such as chapter taxonomy or front matter before deterministic code maps those decisions back to the stable scored contract. ## Context and Orientation

The current line-role implementation lives in [cookimport/parsing/canonical_line_roles.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles.py). That module builds one deterministic baseline `CanonicalLineRolePrediction` per `AtomicLineCandidate`, then, when `line_role_pipeline=codex-line-role-shard-v1`, sends the entire ordered set of atomic lines to Codex in contiguous shards. An atomic line is one normalized line of cookbook text with a stable `atomic_index`, the raw text, some heuristic `rule_tags`, and an upstream `within_recipe_span` hint.

The line-role prompt builder lives in [cookimport/llm/canonical_line_role_prompt.py](/home/mcnal/projects/recipeimport/cookimport/llm/canonical_line_role_prompt.py). It now serializes the shard as `atomic_index|label_code|span_code|hint_codes|current_line` and adds selective `ctx:` windows for ambiguous rows. That keeps the live prompt compact while giving the structure-correction pass enough local evidence to avoid treating every bare heading as a whole-book semantic question.

The repository already has a better architectural seam for outside-recipe ownership. Stage 7, implemented around [cookimport/staging/nonrecipe_stage.py](/home/mcnal/projects/recipeimport/cookimport/staging/nonrecipe_stage.py), produces `08_nonrecipe_spans.json` and `09_knowledge_outputs.json`. Those files are the machine-readable authority for whether outside-recipe material is `knowledge` or `other`. The optional knowledge stage, documented in [docs/10-llm/knowledge_harvest.md](/home/mcnal/projects/recipeimport/docs/10-llm/knowledge_harvest.md), already reviews bounded non-recipe spans and can refine the final outside-recipe authority.

The gap is that the current benchmark behavior still makes line-role carry too much semantic responsibility too early. The model sees isolated lines such as `SALT`, `What is Salt?`, a praise blurb, or memoir prose and must choose from the full global label set with weak context. That encourages plausible but benchmark-wrong labels. Meanwhile the Stage 7 seam, which already owns the final `knowledge` versus `other` contract, has not been elevated into the main place where the hardest non-recipe reasoning happens.

The relevant LLM/Stage 7 files for this plan are:

- [cookimport/parsing/canonical_line_roles.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles.py), which builds deterministic line labels, plans Codex shards, validates responses, and writes `line-role-pipeline/` artifacts.
- [cookimport/llm/canonical_line_role_prompt.py](/home/mcnal/projects/recipeimport/cookimport/llm/canonical_line_role_prompt.py), which defines the live line-role prompt contract.
- [cookimport/staging/nonrecipe_stage.py](/home/mcnal/projects/recipeimport/cookimport/staging/nonrecipe_stage.py), which defines seed and refined Stage 7 non-recipe spans and the refinement report.
- [cookimport/llm/codex_farm_knowledge_jobs.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_jobs.py), which decides which non-recipe chunks survive into bounded knowledge jobs.

- [cookimport/labelstudio/ingest.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest.py), which applies final non-recipe authority back onto scored predictions for benchmark/export surfaces.

3. Preserving the strong recipe-local slices while improving the two outside-recipe buckets above.

## Plan of Work

The plan has four connected edits.

First, narrow the role of line-role without turning deterministic back into the authority. The line-role LLM should stay responsible for recipe-local structure labels such as `RECIPE_TITLE`, `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `HOWTO_SECTION`, `YIELD_LINE`, `TIME_LINE`, `RECIPE_NOTES`, and `RECIPE_VARIANT`. It should still be allowed to label non-recipe rows when that is useful for grouping, but its prompt should stop being the only place where the cookbook’s hardest `KNOWLEDGE` versus `OTHER` reasoning happens. In practice this means rewriting [cookimport/llm/canonical_line_role_prompt.py](/home/mcnal/projects/recipeimport/cookimport/llm/canonical_line_role_prompt.py) so the task is framed as structure correction with grounded context, not as a free-form whole-book semantic parser.

Second, make Stage 7 non-recipe review the place where difficult outside-recipe semantics are resolved. In [cookimport/staging/nonrecipe_stage.py](/home/mcnal/projects/recipeimport/cookimport/staging/nonrecipe_stage.py) and the knowledge-stage input builders, preserve the existing external contract of final `knowledge` versus `other`, but enrich the LLM-facing review payload with coarse semantic hints and local evidence. The reviewer vocabulary should include categories that help the model reason: front matter, table of contents or navigation, endorsement or marketing, chapter taxonomy, memoir or narrative prose, cooking explanation, and practical cooking advice. The model can return one of those richer categories plus extraction evidence, and deterministic code then maps those categories to final `knowledge` or `other`. This gives the model a more natural task while keeping the scored artifact schema stable.

Third, rework the knowledge job builder so the LLM sees the right evidence units. [cookimport/llm/codex_farm_knowledge_jobs.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_jobs.py) currently prunes and bundles chunks for cost control. That remains good, but the builder must stop over-fragmenting semantically related non-recipe spans into tiny isolated questions when those spans are exactly what the model needs to distinguish book taxonomy from actual cooking knowledge. The new bounded unit should usually be a coherent non-recipe span with a short local neighborhood and span-level metadata, not just whichever tiny chunks survived the noise filter.

Fourth, keep deterministic improvements small and local. Deterministic should still tag obvious no-brainers and keep prompt volume under control, but this plan explicitly rejects a deterministic-first success criterion. Any deterministic edits in [cookimport/parsing/canonical_line_roles.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles.py) should be limited to better scaffolding for routing and obvious cases, not to heroically solving the semantic boundary that the LLM stage is supposed to own. The merged heading/prose work belongs here: tighten obvious chapter/contents/title promotion paths so book organization headings do not seed recipe structure too aggressively, and improve cheap recognition of compact pedagogical headings and explanatory prose where that materially helps Stage 7 see the right bounded units.

## Milestones

### Milestone 1: Reframe line-role as grounded structure correction

At the end of this milestone, the line-role LLM still processes the book in shards, but the prompt makes clear that its primary job is correcting structure with enough grounding to do that well. The prompt should once again include deterministic labels, compact escalation reasons, and short local context where ambiguity is real. It should also stop pretending that a bare line is enough evidence for difficult whole-book semantics.

The acceptance for this milestone is prompt-level, not benchmark-level. A reader should be able to inspect a regenerated line-role prompt and see that rows like `SALT`, `What is Salt?`, or a praise blurb now carry enough context to be reviewed intelligently. The prompt should read like a correction pass over grounded structure, not a full semantic classifier over anonymous lines.

### Milestone 2: Promote Stage 7 to the true LLM-first non-recipe authority seam

At the end of this milestone, difficult outside-recipe spans are reviewed by the knowledge/non-recipe LLM path using richer, span-shaped evidence. The model should be allowed to think in terms of coarse reviewer categories that better match cookbook reality, while deterministic code maps those richer decisions back to final `knowledge` or `other` in `08_nonrecipe_spans.json` and `09_knowledge_outputs.json`.

The acceptance for this milestone is artifact-level. A reader should be able to inspect a knowledge-stage payload and see a coherent non-recipe review unit with span metadata, local evidence, and reviewer categories that distinguish cookbook explanation from all the other outside-recipe junk. The refinement report should still end in final `knowledge` versus `other`, but it should become easier to explain why a given span landed there.

### Milestone 3: Rebalance the knowledge job builder around semantic coherence

At the end of this milestone, the cost-control logic still prunes obvious junk, but it no longer starves the hardest non-recipe questions of context. The bounded job unit should prefer semantically coherent non-recipe spans over tiny isolated fragments when those fragments were previously causing the model to confuse chapter headings, endorsements, and practical cooking teaching.

The acceptance for this milestone is planner-level and prompt-preview-level. A reader should be able to regenerate prompt previews and see that the non-recipe stage writes a manageable number of jobs whose contents look like meaningful review units rather than a spray of tiny ambiguous snippets.

### Milestone 4: Validate that `codexfarm` moves for the right reason

At the end of this milestone, a rerun of the single-offline benchmark on `saltfatacidheatcutdown` should show improvement driven by the LLM-corrected path. The desired shape is not “deterministic got less awful.” The desired shape is that the `codexfarm` run improves because the LLM is operating in the right seam with better evidence. Recipe-local structure should remain strong, while outside-recipe `knowledge` versus `other` errors should become more interpretable and materially lower.

The acceptance for this milestone is benchmark-level. The new run must show that `codexfarm` still clearly beats `vanilla`, and that the remaining errors concentrate in a smaller and more understandable set of semantic disagreements rather than the current broad confusion between cookbook taxonomy, memoir prose, and cooking explanation.

## Validation and Acceptance

This change is accepted only if all of the following are true.

The repository-level acceptance conditions are:

- the new tests for line-role prompt grounding and Stage 7 richer category mapping pass
- existing benchmark projection and Label Studio tests still pass
- no rollback or do-no-harm post-processing layer is reintroduced

The behavior-level acceptance conditions are:

- the `codexfarm` benchmark path still materially outperforms `vanilla`
- outside-recipe errors become less dominated by arbitrary line-role confusion and more by explicit Stage 7 semantic decisions
- the final scored outside-recipe labels still come from `08_nonrecipe_spans.json` and `09_knowledge_outputs.json`
- prompt artifacts show that line-role is now a grounded structure pass and the knowledge stage is now a richer semantic non-recipe pass

The benchmark-level acceptance conditions on `saltfatacidheatcutdown` are:

- the new `codexfarm` run is at least competitive with, and ideally better than, `2026-03-17_20.27.19`
- recipe-local structure recall does not materially regress
- `KNOWLEDGE -> OTHER` shrinks materially from the current dominant bucket
- obvious chapter/section false positives such as `SALT`, `What is Salt?`, `How Salt Works`, `Kitchen Basics`, and `The Four Elements of Good Cooking` shrink in the `OTHER -> HOWTO_SECTION` and `OTHER -> RECIPE_TITLE` buckets
- `KNOWLEDGE` versus `OTHER` mistakes become more legible when inspecting wrong-label artifacts, with fewer obvious cases where a bare isolated line was asked to carry whole-book semantics by itself

This plan does not require deterministic-only benchmark success as a gating condition. Deterministic may improve as scaffolding, but the success criterion is that the LLM-corrected path gets stronger because the architecture finally matches the product goal.

## Idempotence and Recovery

All steps in this plan are safe to repeat. Prompt previews, focused tests, and benchmark runs write timestamped artifacts or overwrite only the files being actively edited. The main recovery rule is architectural: if a change makes line-role prompts cleaner but weakens final outside-recipe authority, revert the seam split and keep Stage 7 as the single source of truth. Do not respond by reintroducing a hidden rollback layer.

If a richer knowledge-stage contract increases prompt size too much, retry by reducing redundant per-row text or shortening reviewer category explanations before changing the architecture. The intended retry path is to compress the richer non-recipe payload carefully, not to move the semantic burden back into starved line-role rows.

## Interfaces and Dependencies

At the end of this work, the following interfaces must exist in substance.

- `cookimport.llm.canonical_line_role_prompt.build_canonical_line_role_prompt(...)`
  This function must build a grounded structure-correction prompt. It must no longer behave like the sole semantic classifier for the cookbook’s outside-recipe meaning.

- `cookimport.parsing.canonical_line_roles._run_line_role_shard_runtime(...)`
  This runtime must continue to validate exact owned `atomic_index` coverage, but it must call the prompt builder with the richer grounding contract and stop assuming that a compressed bare-line prompt is sufficient.

- `cookimport.staging.nonrecipe_stage.refine_nonrecipe_stage_result(...)`
  This function must remain the place where final outside-recipe authority is recorded. It must accept richer internal reviewer decisions while still serializing final `knowledge` or `other` to `08_nonrecipe_spans.json`.

- `cookimport.llm.codex_farm_knowledge_jobs.build_knowledge_jobs(...)`
  This builder must emit semantically coherent non-recipe review units, not only tiny fragments optimized for minimum prompt bytes.

- `cookimport.labelstudio.ingest._apply_nonrecipe_authority_to_predictions(...)`
  This projection must continue to treat Stage 7 final authority as the scored outside-recipe truth, regardless of any richer internal review categories.

- The knowledge-stage prompt/schema assets under `llm_pipelines/`
  These assets must support richer reviewer categories and evidence fields while keeping the final external contract stable.
