# Make QualitySuite Baselines Truly Deterministic

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds. This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The goal of this change is to fix one specific bug: QualitySuite baseline construction currently allows a supposed baseline run to keep `line_role_pipeline=codex-line-role-v1` even when `llm_recipe_pipeline=off`. After this change, any QualitySuite baseline-style run will be a true deterministic baseline, not a hybrid run with hidden line-role Codex assistance.

A user should be able to run `cookimport bench quality-run --include-codex-farm ...` and inspect the generated `experiments_resolved.json` files. For every `baseline` experiment, they should see `llm_recipe_pipeline=off`, `line_role_pipeline=off`, and `atomic_block_splitter=off`. The corresponding Codex variants should still exist, but they should be explicit candidate variants rather than hidden behavior inside the baseline.

This plan is intentionally scoped to discrete problem 2 only. It does not change global defaults in `cookimport/config/run_settings.py`, it does not redesign dashboard labeling, and it does not solve the broader telemetry gap. Those are separate problems and should remain separate.

## Context and Orientation

QualitySuite is the repo’s benchmark runner for comparing many processing configurations across a set of “golden” reference sources. In this repository, QualitySuite uses the “all-method” benchmark path. “All-method” means it creates many `RunSettings` variants for one source file, runs prediction and evaluation for each variant, and then writes reports such as `all_method_benchmark_report.json` and `experiments_resolved.json`.

The current bug lives in how these all-method variants are constructed.

The key files are:

- `cookimport/cli.py`: contains the all-method variant construction logic and the single-offline benchmark builder that already implements correct baseline semantics.
- `cookimport/config/run_settings.py`: defines global defaults. This plan reads this file for context, but does not change it.
- `tests/labelstudio/test_labelstudio_benchmark_helpers_scheduler.py`: contains the direct unit tests for `_build_all_method_variants` and related all-method scheduling helpers.
- `tests/labelstudio/test_labelstudio_benchmark_helpers_single_profile.py`: contains tests for benchmark pairing patterns that are useful as a semantic reference for baseline-vs-codex behavior.
- `tests/labelstudio/test_labelstudio_benchmark_helpers_eval_payload.py`: contains proof that benchmark-specific payload shaping already exists elsewhere in the repo.
- `data/golden/bench/quality/runs/2026-03-04_21.16.39_deterministic-ultrafast-singleton-exhaustive-batch/.../experiments_resolved.json`: retained artifact proving the bug.

Terms used in this plan:

A “baseline contract” means the exact `RunSettings` values that define a trustworthy deterministic benchmark baseline. In this plan, that means recipe Codex off, line-role Codex off, and atomic splitting off.

A “candidate variant” means a non-baseline benchmark configuration that is compared against the baseline. In this plan, the important candidate is the Codex-enabled one.

A “hybrid baseline” means a run that looks like a baseline because recipe Codex is off, but still keeps some other AI-driven benchmark component on. That is the bug we are fixing.

## Plan of Work

### Milestone 1: Introduce explicit QualitySuite benchmark contracts

Start in `cookimport/cli.py`, close to `_build_all_method_variants` and `_interactive_single_offline_variants`.

Add two small helper functions that operate on plain run-config dictionaries.

The first helper should take an existing run-config payload and return a new payload that satisfies the deterministic baseline contract. Use a stable name such as `_all_method_apply_baseline_contract` or `_qualitysuite_apply_baseline_contract`. The function must copy the input payload and then explicitly set:

- `llm_recipe_pipeline = "off"`
- `llm_knowledge_pipeline = "off"`
- `llm_tags_pipeline = "off"`
- `line_role_pipeline = "off"`
- `atomic_block_splitter = "off"`

The second helper should take a payload that already satisfies the deterministic baseline contract and return the explicit Codex candidate payload. Use a stable name such as `_all_method_apply_codex_contract_from_baseline`. It must start from a copied deterministic baseline payload and then explicitly set:

- `llm_recipe_pipeline = "codex-farm-3pass-v1"`
- `line_role_pipeline = "codex-line-role-v1"`
- `atomic_block_splitter = "atomic-v1"`

Leave all other settings untouched. The point is to create a clean, intentional baseline/candidate pair while preserving the non-AI dimensions under study such as extractor choice, parser version, skip-headers choice, preprocess mode, or section detector backend.

This milestone is complete when the code contains named helpers that express the benchmark contracts in one place, rather than encoding them ad hoc.

### Milestone 2: Rebuild all-method baselines from the deterministic contract

Modify `cookimport/cli.py:_build_all_method_variants`.

Today, this function takes `base_settings`, creates a `base_payload`, expands sweep payloads, and emits variants directly from those payloads. The bug happens because these payloads inherit the incoming AI-on baseline values.

The fix is to normalize each emitted baseline-style payload through the deterministic baseline helper before the function calls `RunSettings.from_dict`, computes the stable hash, and appends an `AllMethodVariant`.

Concretely, change the control flow so that:

- every baseline-style payload is first passed through the deterministic baseline contract helper,
- `add_variant` uses that normalized baseline payload for the baseline variant,
- and, when `include_codex_farm` is true, the Codex variant is built from the normalized baseline payload via the Codex contract helper instead of by mutating the raw payload.

Do not change the meaning of the non-AI sweep dimensions. For example, if the sweep is comparing `epub_unstructured_html_parser_version` or `epub_unstructured_preprocess_mode`, those comparisons should remain intact. The only behavior change should be that baseline variants no longer inherit line-role Codex or atomic splitting from the incoming base settings.

Be careful with dedupe. The stable hash set should continue to operate on the final `RunSettings` produced after normalization. This matters because many incoming raw payloads may now collapse to the same true deterministic baseline payload. That dedupe is desirable, not a bug.

This milestone is complete when all baseline variants emitted by `_build_all_method_variants` satisfy the deterministic baseline contract, regardless of the starting `base_settings`.

### Milestone 3: Add targeted regression tests

Most of the needed tests belong in `tests/labelstudio/test_labelstudio_benchmark_helpers_scheduler.py` because that file already tests `_build_all_method_variants` directly.

Add a new regression test that starts from AI-on base settings. Use a `RunSettings.from_dict` payload that mirrors the current problematic state, for example:

- `llm_recipe_pipeline = "codex-farm-3pass-v1"`
- `line_role_pipeline = "codex-line-role-v1"`
- `atomic_block_splitter = "atomic-v1"`

Call `_build_all_method_variants(..., source_file=Path("book.epub"), include_codex_farm=True)`.

Assert that:

- all baseline-style variants (the ones without the codex suffix in `slug`) have `llm_recipe_pipeline == "off"`, `line_role_pipeline == "off"`, and `atomic_block_splitter == "off"`
- all Codex variants have `llm_recipe_pipeline == "codex-farm-3pass-v1"`, `line_role_pipeline == "codex-line-role-v1"`, and `atomic_block_splitter == "atomic-v1"`
- the non-AI sweep dimensions still vary as expected

Add a second test for `include_codex_farm=False` using AI-on incoming settings. That test should prove that the all-method baseline set still normalizes to deterministic baseline semantics even when no Codex candidate variants are requested.

Keep the existing count-oriented tests unless the normalization legitimately changes dedupe counts. If a count changes, update the assertion and document why in the test body with a short comment. The reason should be that formerly distinct hybrid payloads are now intentionally collapsing to the same deterministic baseline payload.

If an integration-oriented check is needed, add a lightweight test in `tests/labelstudio/test_labelstudio_benchmark_helpers_scheduler.py` or `tests/bench/test_quality_suite_runner.py` that inspects the serialized run-config payload for a baseline experiment and asserts `line_role_pipeline == "off"`.

This milestone is complete when the new tests fail against the old behavior and pass after the change.

## Validation and Acceptance

The change is accepted only if all of the following are true.

First, a direct variant-builder regression test starting from AI-on `base_settings` proves that all baseline variants now normalize to deterministic settings.

Second, the Codex variants still exist and still explicitly re-enable the intended Codex surfaces. The fix must not silently remove Codex candidates.

Third, a real generated `experiments_resolved.json` from `cookimport bench quality-run --include-codex-farm ...` shows truthful baseline settings. A human should be able to inspect the artifact and see that the baseline is genuinely no-AI.

Fourth, existing single-offline benchmark semantics remain conceptually aligned: `vanilla` continues to mean `off/off/off`, and the all-method QualitySuite baseline now matches that meaning.

A concise human-readable acceptance transcript should look like this.

    baseline ... llm_recipe_pipeline off
    baseline ... line_role_pipeline off
    baseline ... atomic_block_splitter off
    codex ... llm_recipe_pipeline codex-farm-3pass-v1
    codex ... line_role_pipeline codex-line-role-v1
    codex ... atomic_block_splitter atomic-v1

## Idempotence and Recovery

This work is safe to repeat.

The helper functions in `cookimport/cli.py` should be pure dictionary-to-dictionary transformers. Re-running the tests or the lightweight validation snippet should not mutate repo state.

The end-to-end `quality-run` validation writes only to `/tmp/qualitysuite-baseline-fix-check`, so it can be deleted and recreated freely.

If the implementation causes unexpected test fallout, recover in this order:

1. revert only the new all-method baseline contract helper calls,
2. rerun the focused scheduler tests,
3. compare the failing assertions to see whether the breakage is semantic or just an outdated expected count,
4. only after that consider broader changes.

Do not "fix" this bug by changing global defaults. That would mix this scoped plan into problem 1 and make validation ambiguous.
