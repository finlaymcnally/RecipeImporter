# Retire Compatibility Surfaces And Standardize Owned LLM Proposals

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the active Codex-backed parts of the repo should all follow one obvious contract that a human can verify from the artifact tree alone. The program writes immutable input payloads, the model returns structured proposals for only the ids it owns, deterministic code validates those proposals, and deterministic promotion writes the durable outputs. A contributor should be able to inspect any recipe, knowledge, or line-role run and immediately tell which files are authoritative, which files are only debugging convenience, and which old compatibility surfaces were intentionally removed.

The user-visible outcome is simpler runtime behavior and simpler debugging. A reviewer should no longer have to guess whether a folder is a live model seam, a compatibility mirror left behind for older readers, or a prompt-preview convenience export. The dormant prompt-builder code may remain in the repo for future prompt-mode experiments, but it must not be part of the active runtime or artifact contract.

## Context and Orientation

This repository has three active Codex-backed surfaces relevant to this plan.

Recipe correction lives primarily in `cookimport/llm/codex_farm_orchestrator.py`. The program groups nearby recipes into shards. A shard is one owned batch of items for one model call. A worker is one execution context that receives one or more shards. The model returns structured recipe correction outputs that include corrected recipe content, ingredient-to-step linking, and raw selected tags. Deterministic code validates exact owned `recipe_id` coverage, then promotes the accepted results into the final staged recipe outputs.

Knowledge review lives primarily in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, `cookimport/llm/codex_farm_knowledge_jobs.py`, and `cookimport/llm/codex_farm_knowledge_ingest.py`. The program first builds eligible non-recipe chunks from deterministic parsing. It prunes obvious low-value noise before the model sees anything. The model returns structured knowledge proposals for owned `chunk_id` values. Deterministic ingest validates owned coverage and rejects out-of-shard evidence before durable knowledge artifacts are written.

Canonical line-role lives primarily in `cookimport/parsing/canonical_line_roles.py`. The active worktree shape is now one single-pass file-backed `line_role` surface, not the older internal `recipe_region_gate` plus `recipe_structure_label` split. The program plans owned line shards across the ordered atomic lines of the book, writes immutable task payload files, receives structured labels for owned `atomic_index` values, validates exact coverage, and then promotes the accepted labels into the rest of the pipeline.

The repo also has prompt-preview and prompt-artifact readers under `cookimport/llm/prompt_preview.py` and `cookimport/llm/prompt_artifacts.py`. These readers are important because old compatibility mirrors often survive only because preview/debug code still depends on them. This plan must either move those readers onto authoritative runtime/proposal artifacts or remove the stale readers outright.

The key architectural distinction for this work is:

- An authoritative input payload is the immutable file or in-memory structured body the program prepares for a shard before model execution.
- An authoritative validated proposal is the structured model result after exact owned-id validation has passed.
- A promoted durable output is the final deterministic artifact that downstream pipeline stages and users should rely on.
- A compatibility surface is any extra file, folder, or export retained only so older readers or habits still work, even though it is not the authoritative runtime seam.

The user has explicitly chosen the authoritative contract:

1. program writes input payload
2. model returns structured owned output
3. deterministic code validates ownership and coverage
4. deterministic promotion writes durable outputs

This plan must make that the only active mental model in code, docs, and artifacts.

## Plan of Work

Begin with an explicit inventory. Inspect the writers and readers for recipe, knowledge, and line-role. For each artifact family, record whether it is an authoritative input payload, an authoritative validated proposal, a promoted durable output, a debug export that is still genuinely useful, or a compatibility mirror that exists only because older code expected it. The inventory work should update this plan with the exact files and functions involved so a future contributor can continue from the plan alone. For line-role, the inventory must also mark all remaining references to `recipe_region_gate` and `recipe_structure_label` as historical-only or delete them if they still act like active surfaces.

Next, clean up recipe, because it appears to have the highest compatibility burden. In `cookimport/llm/codex_farm_orchestrator.py`, identify every remaining write to `recipe_correction/{in,out}/` and determine whether prompt-preview and audit readers can consume `recipe_phase_runtime/inputs`, `recipe_phase_runtime/proposals`, and promoted staged outputs directly instead. Update any preview or artifact code that still reads the compatibility mirrors. Once no active readers depend on those mirrors, delete the writes and remove the stale folders from docs and tests.

Then clean up knowledge. Knowledge is already closer to the target contract, so the main work is making sure no stale documentation or reader code still treats `knowledge/{in,out}` as if it were an edit-in-place surface. If `knowledge/out/` is truly dead, remove the writer and any tests that still mention it. If `knowledge/in/` remains the active immutable input payload, keep it but document it as authoritative rather than as a legacy-looking compatibility folder.

Then clean up line-role. The active line-role runtime is already single-pass and file-backed. The work here is to finish deleting the abandoned gate split from pipeline packs, fake-runner logic, stale plans, understandings that no longer describe current runtime truth, and any preview or prompt-artifact code that still assumes multiple line-role phases. Docs, preview exports, and prompt artifacts must describe the worker-local or preview-local `line_role` input files as authoritative input payloads and the validated proposals as authoritative model results.

Once the runtime surfaces are cleaned up, align the docs. Update `docs/10-llm/10-llm_README.md`, `docs/04-parsing/04-parsing_readme.md`, and any narrow README files near the touched modules so they all say the same thing: immutable payload, owned proposal, validation, promotion. Do not preserve mixed narratives that say one thing for runtime and another for prompt preview.

Finally, harden with tests. Recipe, knowledge, and line-role should each have at least one focused test proving that the active runtime accepts immutable input payloads and only promotes validated owned proposals. Prompt-preview tests should prove that previews rebuild from authoritative runtime seams rather than stale compatibility mirrors. If an old compatibility folder disappears, add one regression test that would fail if someone tried to quietly reintroduce it as an active dependency.

## Idempotence and Recovery

This work should be done as a sequence of reader migrations followed by writer deletions. That makes it safe to rerun because removing a compatibility write should happen only after all active readers are already off that surface.

If a deletion breaks prompt preview or a benchmark reader, restore only the minimum reader path needed to inspect the failure, then continue the migration by moving that reader onto authoritative artifacts instead of restoring the compatibility surface wholesale.

Use new timestamped output folders for any preview validation so repeated dry runs do not overwrite evidence from prior attempts.

## Interfaces and Dependencies

The implementation should preserve the existing shared runtime vocabulary built around `PhaseManifestV1`, `ShardManifestEntryV1`, `ShardProposalV1`, `WorkerAssignmentV1`, and promotion reports. Those types already express the proposal-and-promotion paradigm well enough; the cleanup should use them more consistently rather than inventing a new runtime type system.

At the end of the work, each active surface should map onto the following concrete interfaces:

- Recipe:
  - authoritative input payloads written by the recipe shard planner in `cookimport/llm/codex_farm_orchestrator.py`
  - validated proposals under `recipe_phase_runtime/proposals/*.json`
  - deterministic durable outputs written through the normal staged recipe writers

- Knowledge:
  - authoritative input payloads written by `cookimport/llm/codex_farm_knowledge_jobs.py`
  - validated proposals read through `read_validated_knowledge_outputs_from_proposals(...)` in `cookimport/llm/codex_farm_knowledge_ingest.py`
  - deterministic durable outputs written by `write_knowledge_artifacts(...)`

- Line-role:
  - authoritative input payloads written by the active line-role shard runtime in `cookimport/parsing/canonical_line_roles.py`
  - validated proposals under `line-role-pipeline/runtime/line_role/proposals/*.json`
  - deterministic promoted line labels consumed by later grouping and staging code

Prompt preview and prompt artifact export must depend only on those authoritative seams plus the promoted durable outputs. They must not require compatibility mirrors or “edited output file” assumptions.
