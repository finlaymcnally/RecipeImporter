# Compact Pass4 Knowledge Pack

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

After this change, the shipped `llm_pipelines` pack will treat pass4 knowledge harvest the same way pass2 and pass3 are treated now: the default pass4 pipeline id will be a compact one, the shipped pack will only contain the compact pass4 prompt/pipeline asset, and the compact pass4 input bundle will carry less redundant context. A user can verify the change by checking that `RunSettings()` defaults to `recipe.knowledge.compact.v1`, the local pack contains only the compact pass4 pipeline asset, and the pass4 job-bundle tests show the compact payload is smaller than the legacy payload while preserving the required output behavior.

## Surprises & Discoveries

- Observation: Pass4 bundle size is driven mostly by input JSON shape, not by prompt prose. The current pass4 prompt is already short, but `build_pass4_knowledge_jobs(...)` writes full `chunk.blocks`, full before/after context block objects, and a full-stream `recipe_spans` guardrail list into every job bundle.
  Evidence: `cookimport/llm/codex_farm_knowledge_jobs.py` writes `KnowledgeBlockV1` rows for both chunk and context blocks, and `llm_pipelines/prompts/recipe.knowledge.v1.prompt.md` is only a short instruction wrapper around `{{INPUT_TEXT}}`.

- Observation: Once compact pass4 started omitting default-valued fields, existing tests that asserted the presence of empty `suggested_highlights`, `suggested_skip_reason`, and `blocks_before` needed to be relaxed to match the compact serialization contract.
  Evidence: the first compact test failures were `AssertionError`/`KeyError` on omitted empty heuristic/context fields rather than on semantic payload mismatches.

## Decision Log

- Decision: Add a new compact pass4 pipeline id (`recipe.knowledge.compact.v1`) and make it the shipped default, instead of mutating `recipe.knowledge.v1` in place.
  Rationale: This matches how pass2/pass3 were compacted, keeps an escape hatch for explicit legacy/custom pipeline ids, and lets tests compare compact vs. legacy payload shapes directly.
  Date/Author: 2026-03-06 / Codex

- Decision: Keep legacy pass4 payload generation code in the runtime for explicit non-compact pipeline ids even though the shipped pack will no longer include the legacy asset.
  Rationale: The repository already has compatibility seams for custom pipeline ids, and removing runtime support would be a broader behavior break than the user asked for.
  Date/Author: 2026-03-06 / Codex

- Decision: Make the low-level `build_pass4_knowledge_jobs(...)` helper default to compact format as well.
  Rationale: The helper is an internal repository seam, and its default should match the now-shipped/runtime default pass4 behavior; legacy coverage remains available through an explicit `job_format=legacy`.
  Date/Author: 2026-03-06 / Codex

## Outcomes & Retrospective

The repository now ships only the compact pass4 asset (`recipe.knowledge.compact.v1`) and defaults pass4 knowledge harvest to that id across `RunSettings`, CLI, Label Studio ingest, and benchmark prompt reconstruction. Compact pass4 bundles now serialize leaner chunk/context payloads and a local context-recipe guardrail instead of the old full-book recipe-span list. The compatibility gap that remains is intentional: explicit legacy/custom non-compact pass4 ids still use the legacy job writer, but the local shipped pack no longer includes the legacy prompt/pipeline asset.

## Context and Orientation

Pass4 is the optional knowledge-harvest stage that runs after the recipe extraction passes. The runtime entrypoint is `cookimport/llm/codex_farm_knowledge_orchestrator.py`. That file resolves the selected pass4 pipeline id, calls `build_pass4_knowledge_jobs(...)` in `cookimport/llm/codex_farm_knowledge_jobs.py` to write one JSON bundle per non-recipe chunk under `raw/llm/<workbook>/pass4_knowledge/in/`, runs `codex-farm process` through the shared runner, then validates outputs through `cookimport/llm/codex_farm_knowledge_ingest.py` and writes snippets/markdown via `cookimport/llm/codex_farm_knowledge_writer.py`.

The local pack lives in `llm_pipelines/`. A “pipeline asset” means a JSON file under `llm_pipelines/pipelines/` that names a pipeline id, a prompt template, and an output schema. The shipped compact pass4 asset is `llm_pipelines/pipelines/recipe.knowledge.compact.v1.json`, which points at `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md` and reuses `llm_pipelines/schemas/recipe.knowledge.v1.output.schema.json`.

Run-setting defaults for pass4 live in `cookimport/config/run_settings.py`, with CLI and Label Studio normalization surfaces in `cookimport/cli.py` and `cookimport/labelstudio/ingest.py`. The benchmark prompt reconstruction helper that maps pass names to pipeline ids lives in `scripts/benchmark_cutdown_for_external_ai.py`.

## Plan of Work

Add compact pass4 input contracts alongside the existing pass4 contracts in `cookimport/llm/codex_farm_knowledge_contracts.py`. The compact contract will keep the same bundle version and the same output schema on the response side, but it will reduce redundant input fields. The main reduction will come from shrinking context blocks to only the fields that help interpretation, trimming table hints to the essentials for the chunk itself, and replacing the full-book recipe-span guardrail list with a smaller guardrail summary scoped to nearby context.

Update `cookimport/llm/codex_farm_knowledge_jobs.py` so job writing branches on the selected pass4 pipeline id. When the pipeline id is `recipe.knowledge.compact.v1`, write the compact input shape. When a caller explicitly chooses a different pass4 pipeline id, keep writing the legacy/full shape so custom or historical pipelines still work. Add a focused test that compares compact and legacy bundle sizes from the same input blocks and asserts that the compact bundle keeps the needed chunk/context semantics while serializing fewer bytes.

Add the new shipped pass4 pack asset in `llm_pipelines/pipelines/recipe.knowledge.compact.v1.json` and `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md`. Then update default selections in `cookimport/config/run_settings.py`, `cookimport/llm/codex_farm_knowledge_orchestrator.py`, `cookimport/cli.py`, `cookimport/labelstudio/ingest.py`, and `scripts/benchmark_cutdown_for_external_ai.py` to use `recipe.knowledge.compact.v1`. After references are updated, remove the legacy shipped files `llm_pipelines/pipelines/recipe.knowledge.v1.json` and `llm_pipelines/prompts/recipe.knowledge.v1.prompt.md`.

Finally, update the pass4-related pack tests, run-settings tests, knowledge orchestrator tests, and benchmark-helper fixtures that assert the shipped/default pass4 pipeline id. Leave tests that intentionally cover custom pass4 ids unchanged.

## Idempotence and Recovery

The edits are idempotent. Re-running the same focused tests is safe. If the compact pass4 payload breaks a custom pipeline, the compatibility path is to set `codex_farm_pipeline_pass4_knowledge` to an explicit non-compact/custom id, which will continue to use the legacy payload writer retained in the runtime.
