# Knowledge Contiguous Tagged Groups

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the knowledge stage should stop asking the model to assign a group id row by row and then hoping the shared tag story lines up afterward. Instead, pass 1 should stay a simple binary `keep_for_review` versus `other`, and pass 2 should partition the kept rows into contiguous reading-order groups. Each group should carry one topic label and one shared grounding story, and deterministic code should project that accepted group grounding back onto the member rows. A human should be able to inspect a prompt, a repair packet, or a final `knowledge_groups.json` artifact and see that the model was explicitly asked to choose the split with the tag in mind.

## Surprises & Discoveries

- Observation: the current pass-2 contract is row-first in every owner layer, not just the prompt text.
  Evidence: `cookimport/llm/knowledge_stage/task_file_contracts.py` currently builds grouping task units with blank per-row answers containing `group_id`, `topic_label`, and `grounding`; `validate_knowledge_grouping_task_file(...)` validates those fields unit by unit.
- Observation: final group outputs are already reconstructed from row answers rather than stored as a first-class model result.
  Evidence: `_canonicalize_knowledge_groups_by_shard(...)` in `task_file_contracts.py` derives final `idea_groups` by walking the row answers, canonicalizing shared stories, and synthesizing final `g01`, `g02`, ... ids afterward.
- Observation: this makes the current validator good at detecting mixed stories, but bad at steering the model away from them.
  Evidence: the prompt says “rows about the same idea must share the same `group_id`,” but the response shape still asks the model to think row-by-row and only later maintain global consistency.
- Observation: the inline-json path was the real product contract to change first; the same-session taskfile fallback can keep its internal row-answer map as long as validation/finalization preserve contiguous runs and the public artifact shape stays stable.
  Evidence: `structured_session_contract.py` owns the live split-and-tag packet/prompt/parser, while downstream runtime and promotion already consume an internal row-keyed `grouping_answers_by_unit_id` map and can keep doing so after accepted groups are expanded back onto rows.

## Outcomes & Retrospective

The main product goal landed on the live inline path: pass 1 stays binary `keep_for_review` versus `other`, and pass 2 is now explicitly a split-and-tag step where the model partitions kept rows into contiguous groups and picks the shared tag grounding while making each split. The fake runner, prompt/packet artifacts, repair packets, and parser all speak that group-first language now.

Downstream public outputs stayed stable on purpose. Final shard payloads still expose row-level `block_decisions` plus grouped `idea_groups`, but repo-owned finalization now preserves contiguous runs instead of re-merging matching stories across an entire shard. That means repeated topic stories later in a shard stay separate groups if they were separated in reading order.

The main scope cut is the same-session taskfile fallback: it still stores grouping answers internally as row-local answers, then projects them into contiguous groups during validation/finalization. That is an internal compatibility choice, not a user-facing prompt contract. If fallback parity ever matters enough, the next step would be to collapse grouping task files into a one-answer batch surface that edits a top-level `groups` list directly.

## Context and Orientation

The relevant runtime is in `cookimport/llm/knowledge_stage/`. In this repository, “pass 1” means the first knowledge classification call that labels each candidate block as `keep_for_review` or `other`. “Pass 2” means the follow-up call that currently asks the model to answer every kept row with `group_id`, `topic_label`, `grounding`, `why_no_existing_tag`, and `retrieval_query`.

The current owners are:

- `cookimport/llm/knowledge_stage/task_file_contracts.py`
  This file builds classification and grouping task files, validates worker/task answers, and combines classification plus grouping results into the final shard payload.
- `cookimport/llm/knowledge_stage/structured_session_contract.py`
  This file builds the inline-json packet and prompt, parses model JSON back into edited task files, and shapes repair packets.
- `cookimport/llm/knowledge_stage/workspace_run.py`
  This file runs the inline knowledge flow and currently validates grouping batch answers row by row before combining them.
- `cookimport/llm/codex_farm_knowledge_ingest.py`
  This file validates promoted shard outputs, including whether knowledge groups align with the row-level grounding now projected into final outputs.
- `cookimport/llm/codex_farm_knowledge_writer.py`
  This file writes `knowledge_groups.json` and `knowledge.md` previews from the promoted `idea_groups`.

The important current behavior is:

- kept rows become grouping units
- grouping asks for one answer per row
- deterministic code then canonicalizes rows sharing the same topic/grounding story into final groups
- validator failures such as `knowledge_group_mixed_tag_story` happen after the model has already emitted row answers

The change in this plan is to flip that around. The model should emit the groups directly, as contiguous spans over the kept rows, and deterministic code should only project those accepted groups back to row answers afterward.

## Plan of Work

First, refit the grouping task builder in `cookimport/llm/knowledge_stage/task_file_contracts.py`. The live requirement is that final validation/finalization respect contiguous runs; the current implementation keeps taskfile fallback answers row-keyed internally, while the inline path is where the user-facing group-list contract now lives. Each accepted group still needs one `topic_label`, `grounding`, `why_no_existing_tag`, and `retrieval_query`.

Second, change the inline packet/prompt/parser in `cookimport/llm/knowledge_stage/structured_session_contract.py`. The grouping prompt should explicitly say that the model is partitioning the kept rows into contiguous groups and choosing the tag story while deciding each split. The parser should read a top-level group list, validate that every referenced row id belongs to the packet, and expand accepted groups back into the edited task-file shape expected by the rest of the runtime. Repair packets should preserve enough feedback for the model to fix gaps, overlaps, and invalid group grounding, but the repair surface should still talk in terms of groups rather than “answer every row again with `group_id`.”

Third, adjust validation in `validate_knowledge_grouping_task_file(...)` so it enforces the new invariants. Deterministic code must reject missing kept rows, duplicated kept rows, non-contiguous row membership inside a single group, out-of-order group spans, and invalid grounding stories. Once a group is accepted, deterministic code should expand it into the internal row-answer map so the existing downstream output combiner can keep projecting shared group grounding back to individual knowledge rows.

Fourth, update `combine_knowledge_task_file_outputs(...)`, resolution metadata, and any helper that still assumes group ids originated row-by-row. Final public outputs should stay the same: `block_decisions` at row level and `idea_groups` at group level. The difference is that accepted groups now come directly from the model and row grounding is a deterministic projection from those group spans.

Fifth, update tests and docs. The strongest coverage should live in `tests/llm/test_knowledge_grouping_contract.py`, `tests/llm/test_knowledge_structured_session_contract.py`, `tests/llm/test_knowledge_stage_promotion.py`, `tests/llm/test_knowledge_orchestrator_runtime_leasing.py`, and the adjacent fake-farm / output-ingest slices that still exercise fallback wiring. The docs to update are `docs/10-llm/10-llm_README.md`, `docs/07-bench/07-bench_README.md`, and the short owner note in `cookimport/llm/knowledge_stage/README.md`.

## Idempotence and Recovery

This change is safe to iterate on. The changed files are task-file contracts, inline prompt/packet helpers, runtime validation, and tests. If a patch breaks the new grouping parser, the quickest recovery check is to inspect the saved `grouping_packet_*.json`, `grouping_prompt_*.txt`, and validation metadata for whether the model output was parsed into groups at all. If promotion regresses, compare the final `idea_groups` and `block_decisions` against the prior row-projection shape; the public artifact shape should remain stable even though the pass-2 contract changed.
