## Purpose / Big Picture

After this change, shard-v1 Codex phases should actually use the worker fan-out they advertise. A benchmark run that says `active workers: 5` should mean five worker assignments are genuinely in flight, not one serial loop slowly working through five preplanned buckets. The visible success case is a multi-shard Codex phase finishing materially faster and emitting progress that matches what the runtime is really doing.

## Surprises & Discoveries

- Observation: the recent worker-count-default fix was real, but it only changed how many worker buckets were planned.
  Evidence: the diagnosed benchmark artifacts under `data/golden/benchmark-vs-golden/2026-03-17_20.27.19/.../codexfarm/processing_timeseries_prediction.jsonl` still showed recipe, knowledge, and line-role starts spaced apart like a serial loop even after `worker_count` resolved to `5`.

  Evidence: both `_run_direct_knowledge_workers_v1(...)` and `_run_line_role_direct_workers_v1(...)` created worker assignments up front, then iterated `for assignment in assignments:` without any concurrent launch.

  Evidence: `bench speed-run --include-codex-farm` is explicitly blocked in agent-run environments, so deterministic speed-suite coverage is the only available baseline-vs-candidate check here.

## Context and Orientation

The shared shard-v1 worker runtime lives in `cookimport/llm/phase_worker_runtime.py`. Recipe correction uses that module to create worker sandboxes, place one or more shard inputs into each worker bucket, run CodexFarm, validate outputs, and aggregate promotion telemetry.

Knowledge harvest does not call that shared execution helper. Its direct `codex exec` runtime lives in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, where `_run_direct_knowledge_workers_v1(...)` owns assignment, prompt emission, result validation, and progress callbacks for knowledge shards.

Canonical line-role also owns a direct runtime in `cookimport/parsing/canonical_line_roles.py`, where `_run_line_role_direct_workers_v1(...)` performs the same job for line-role shard prompts.

For this plan, a “worker assignment” means one runtime-owned bucket such as `worker-001` that may own one or more shard ids. The bug was not in assignment planning. The bug was that assigned workers were still executed one after another, so progress looked parallel while execution was serialized.

## Plan of Work

First, update `cookimport/llm/phase_worker_runtime.py` so `run_phase_workers_v1(...)` launches `_run_worker_assignment_v1(...)` calls concurrently. The shared runtime must still write the same manifests and aggregate proposals/failures in a stable order after the worker futures complete.

Second, apply the same pattern to `_run_direct_knowledge_workers_v1(...)` in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. Preserve the existing progress callback contract by protecting completion counters with a lock and recomputing active shard labels when each worker finishes a shard.

Third, apply the same pattern to `_run_line_role_direct_workers_v1(...)` in `cookimport/parsing/canonical_line_roles.py`, again preserving progress updates and stable artifact aggregation.

Fourth, add focused tests that prove more than one worker assignment can be in flight at the same time, and then run the required deterministic speed-suite sanity flow plus the focused pytest slice.
