# Reduce Line-Role Payload By Dropping Redundant Neighbor Context

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

Today the live file-backed canonical line-role transport no longer wastes tokens on giant inline wrapper prompts, but it still wastes a large amount of input budget inside the shard files themselves. Each model-facing row in the worker `in/*.json` file repeats the previous line and next line even though the shard file already contains one ordered contiguous slice of the book. After this change, line-role should keep the same file-backed transport and the same behavior, but the model-facing shard payload should stop copying `prev_text` and `next_text` into every row. The user-visible result is that prompt preview and live runtime artifacts remain easy to follow while the line-role task files become materially smaller and cheaper to read.

The change is complete when a contributor can run the existing zero-token line-role prompt preview on `thefoodlabcutdown`, inspect `line-role-pipeline/in/line_role_input_0001.json`, and see that each row carries the current line plus the fields the model truly needs, not duplicated neighbor text. The same contributor should then compare the new preview budget against the current baseline and observe a meaningful line-role input-token drop without changing the file-backed transport model.

## Surprises & Discoveries

- Observation: the redundancy is in the live model-facing JSON seam, not only in a legacy inline prompt helper.
  Evidence: `_line_role_plan_row(...)` in `cookimport/parsing/canonical_line_roles.py` currently returns `prev_text`, `current_line`, and `next_text`, and the preview writer in `cookimport/llm/prompt_preview.py` writes the same fields into `line-role-pipeline/in/line_role_input_*.json`.

- Observation: the current single-pass line-role wrapper already tells the model it is reading “one ordered slice of the book,” which means the file itself provides neighbor order.
  Evidence: the exported wrapper prompt in `data/output/2026-03-18_15.15.00-foodlab-file-preview/prompts/prompt_line_role_recipe_structure_label.txt` describes “one ordered slice of the book” and names the deposited task file path, so the shard ordering already exists independently of `prev_text` and `next_text`.

- Observation: the repo already contains budget evidence that neighbor duplication is large enough to matter on Food Lab.
  Evidence: `docs/understandings/2026-03-18_14.54.15-foodlab-line-role-preview-cost-is-mostly-duplicate-phase-reads-plus-local-context.md` reports that local context duplication contributed about `527.3k` task-body input tokens on the earlier two-phase Food Lab preview; even after the later single-pass cut, the same structural waste still exists row-by-row.

- Observation: removing only `prev_text` and `next_text` from the file-backed shard JSON cut the single-pass Food Lab preview by about forty-three percent without changing shard count.
  Evidence: `data/output/2026-03-18_15.36.00-foodlab-single-phase-preview/prompt_preview_budget_summary.json` reports `~636,852` estimated line-role tokens across `5` calls, while `data/output/2026-03-18_16.00.00-foodlab-line-role-no-neighbor-context-preview/prompt_preview_budget_summary.json` reports `~361,968` across the same `5` calls.

- Observation: after the neighbor-field cut, payload size is now dominated by the line text itself and secondarily by rule metadata rather than by row-order context.
  Evidence: summing serialized key contributions across the new `line-role-pipeline/in/line_role_input_*.json` files shows approximate character weight ordered as `current_line` (`567,725`), `rule_tags` (`140,756`), `deterministic_label` (`84,382`), and `escalation_reasons` (`83,083`).

## Decision Log

- Decision: scope this plan narrowly to removing redundant `prev_text` and `next_text` from the live model-facing single-pass line-role payload, not to a full prompt-schema redesign.
  Rationale: the user called out one specific waste source, the current live seam is easy to identify, and a narrow cut gives a clear before/after budget check without mixing in larger prompt-contract changes.
  Date/Author: 2026-03-18 / Codex

- Decision: do not remove neighbor fields from `AtomicLineCandidate` or deterministic parsing heuristics as part of this plan.
  Rationale: parsing and deterministic rule code may still use local adjacency while deciding baseline labels and escalation. The waste we want to eliminate is only the later duplication into the Codex-facing shard file.
  Date/Author: 2026-03-18 / Codex

- Decision: keep the shard as one ordered contiguous row list and teach the wrapper prompt to rely on row order for local context.
  Rationale: the model already receives the full ordered slice in one file. Paying again for explicit `prev_text` and `next_text` on every row is unnecessary transport overhead.
  Date/Author: 2026-03-18 / Codex

- Decision: preserve enough row metadata to avoid coupling this plan to a broader “minimal payload” redesign.
  Rationale: there is likely more token waste in `rule_tags`, `escalation_reasons`, `block_id`, and related keys, but that is a second optimization pass. This plan should first remove the most obviously redundant neighbor duplication and prove the savings cleanly.
  Date/Author: 2026-03-18 / Codex

- Decision: keep the inline compact prompt helpers and selective `ctx:` rows unchanged while shrinking only the file-backed shard JSON.
  Rationale: the active waste source was the shared runtime/preview task-file serializer. Inline compact prompt helpers are a separate seam and still rely on selective context windows for ambiguous rows.
  Date/Author: 2026-03-18 / Codex

## Outcomes & Retrospective

Implementation landed as planned. The shared line-role shard-row serializer used by live runtime and prompt preview now omits `prev_text` and `next_text`, and the file-backed wrapper prompt explicitly tells the model to use neighboring rows in the ordered task file for local context. Focused verification passed with:

    pytest tests/parsing/test_canonical_line_roles.py tests/llm/test_prompt_preview.py tests/llm/test_label_phase_workers.py -q

The zero-token Food Lab comparison stayed apples-to-apples at `5` shard calls and moved from `~636,852` estimated line-role tokens / `2,512,327` task-file chars in `data/output/2026-03-18_15.36.00-foodlab-single-phase-preview/` to `~361,968` estimated tokens / `1,352,128` task-file chars in `data/output/2026-03-18_16.00.00-foodlab-line-role-no-neighbor-context-preview/`. That is a reduction of `274,884` estimated tokens (`43.2%`) and `1,160,199` task-file chars (`46.2%`).

The reduced schema was enough to produce a clear budget win without touching runtime behavior, but the post-cut payload is still mostly `current_line`, `rule_tags`, `deterministic_label`, and `escalation_reasons`. A follow-up compact-payload plan could target those fields if more budget reduction is needed.

## Context and Orientation

Canonical line-role is the Codex-backed labeling pass that reviews ordered cookbook text rows and returns labels such as `RECIPE_TITLE`, `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `KNOWLEDGE`, or `OTHER`. A “shard” is one contiguous batch of atomic lines handled by one model call. In the current live design, the model-facing task file is written to `line-role-pipeline/runtime/line_role/workers/<worker-id>/in/<shard>.json` during execution and to `line-role-pipeline/in/line_role_input_*.json` during predictive preview.

The relevant files are:

- `cookimport/parsing/canonical_line_roles.py`: plans line-role shards, builds the model-facing `input_payload`, writes worker runtime artifacts, runs direct `codex exec`, and validates returned labels.
- `cookimport/llm/canonical_line_role_prompt.py`: renders the line-role wrapper prompt that tells Codex what file to read and what JSON shape to return.
- `cookimport/llm/prompt_preview.py`: reconstructs the same line-role input payload and wrapper prompt in zero-token preview mode.
- `cookimport/llm/prompt_budget.py`: estimates preview token cost from the locally reconstructed wrapper prompt plus deposited task-file body.
- `tests/parsing/test_canonical_line_roles.py`: focused runtime/payload/prompt tests for line-role.
- `tests/llm/test_prompt_preview.py`: preview reconstruction and prompt-budget tests.

Before implementation, `_line_role_plan_row(...)` in `cookimport/parsing/canonical_line_roles.py` wrote:

    {
      "atomic_index": ...,
      "block_index": ...,
      "block_id": ...,
      "recipe_id": ...,
      "within_recipe_span": ...,
      "deterministic_label": ...,
      "rule_tags": ...,
      "escalation_reasons": ...,
      "prev_text": ...,
      "current_line": ...,
      "next_text": ...
    }

That row shape was useful when prompts reviewed isolated or sparse target rows. It was wasteful in the file-backed shard design because the whole file already contains the ordered row sequence. After implementation, the live model-facing payload is still one ordered list of rows, but each row no longer copies its neighbors:

    {
      "atomic_index": ...,
      "block_index": ...,
      "block_id": ...,
      "recipe_id": ...,
      "within_recipe_span": ...,
      "deterministic_label": ...,
      "rule_tags": ...,
      "escalation_reasons": ...,
      "current_line": ...
    }

The current zero-token Food Lab preview baseline to compare against is the single-pass file-backed line-role run captured in `data/output/2026-03-18_15.36.00-foodlab-single-phase-preview/prompt_preview_budget_summary.json`. That baseline is `5` line-role calls and about `636.9k` predictive line-role tokens. This plan should not promise an exact new total in advance, but the after-state must be lower than that baseline and must explain the measured delta.

## Plan of Work

Start in `cookimport/parsing/canonical_line_roles.py`. Change `_line_role_plan_row(...)` so the model-facing row object no longer includes `prev_text` or `next_text`. Do not remove `current_line`. Do not change deterministic baseline generation, escalation logic, shard planning, validation, or fallback behavior. The only live runtime contract change should be the contents of the deposited shard file.

Then update the file-backed wrapper prompt in `cookimport/llm/canonical_line_role_prompt.py`. The current file-backed prompt text still says that each row includes `prev_text`, `current_line`, and `next_text`, and tells the model to use those fields as local context. Rewrite that contract so it instead says the task file is one ordered contiguous slice of the book, `current_line` is the line to label, and neighboring rows in the ordered `rows[*]` list provide local context. Keep the strict JSON output contract, allowed labels, precedence rules, and negative rules exactly as they are unless implementation proves one of those instructions depends on explicit neighbor fields.

Mirror the same payload change in `cookimport/llm/prompt_preview.py`. The preview-generated `line-role-pipeline/in/line_role_input_*.json` files must match the live model-facing payload shape. Update any preview row metadata or request-input snapshots so they show the compact row schema rather than the old neighbor-copying schema. The preview code should continue to write a reproducible `request_input_payload`, but that payload should now be the smaller live contract.

Update the targeted tests. In `tests/parsing/test_canonical_line_roles.py`, replace assertions that currently expect `prev_text` and `next_text` inside the deposited worker input file. Add explicit assertions that those keys are absent and that `current_line` remains present. In `tests/llm/test_prompt_preview.py`, update the preview payload assertions the same way. If any prompt text assertions mention `prev_text`/`next_text`, rewrite them to assert the new “ordered contiguous slice” contract instead.

Once the code and tests pass, run the saved Food Lab prompt preview again with the same command family used in the earlier zero-token checks. Compare the resulting `prompt_preview_budget_summary.json` against the current single-pass baseline. Record the exact before and after line-role totals in this plan’s `Progress`, `Surprises & Discoveries`, and `Outcomes & Retrospective` sections. If the reduction is smaller than expected, document which remaining metadata fields dominate the payload so the next plan can target them directly.

Finally, update the living docs. `docs/10-llm/10-llm_README.md` and `docs/04-parsing/04-parsing_readme.md` should explain that single-pass line-role task files are ordered row lists and no longer duplicate per-row neighbor context. Keep the note short and grounded in the real runtime artifact paths.

## Validation and Acceptance

Validation is both structural and budget-oriented.

Structural acceptance means:

1. A live-deposited worker task file under `line-role-pipeline/runtime/line_role/workers/<worker-id>/in/*.json` contains `current_line` but no `prev_text` or `next_text`.
2. A preview-deposited task file under `line-role-pipeline/in/line_role_input_*.json` matches that same row schema.
3. The wrapper prompt text explains that the file is an ordered contiguous slice and no longer claims each row contains explicit neighbor fields.
4. The targeted line-role and prompt-preview tests pass.

Budget acceptance means:

1. Re-running the Food Lab zero-token line-role preview yields a lower `line_role` estimate than the current single-pass baseline of about `636.9k` predictive tokens.
2. The reduced preview budget is explained honestly as a payload-shape win, not as a live billing guarantee.
3. If the drop is smaller than expected, the plan records which remaining row fields now dominate the input budget.

Behavioral acceptance means:

1. The line-role runtime still validates exact owned `atomic_index` coverage and still promotes returned labels normally.
2. No live token spend is required for this plan.

## Idempotence and Recovery

This plan is safe to execute repeatedly. The code edits are additive schema reductions on the live payload seam, and the preview outputs are written to a fresh timestamped directory. If the changed payload shape breaks a test or preview reader, restore the previous row keys with `git diff` as a guide, rerun the focused tests, and then re-apply the reduction more narrowly. Do not delete old preview directories; they are the baseline evidence for the before/after comparison.

If the runtime still needs human-friendly neighbor visibility after the compact payload lands, solve that in a separate follow-up by writing a debug-only artifact. Do not re-inflate the model-facing shard schema just to preserve human inspection convenience.
