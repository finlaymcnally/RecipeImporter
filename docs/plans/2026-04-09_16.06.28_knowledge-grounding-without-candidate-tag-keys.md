# Knowledge Grounding Without `candidate_tag_keys`

Superseded note: this plan removed `candidate_tag_keys`, but it also introduced a temporary ingest cleanup layer that was removed on 2026-04-10. Read it as historical context, not current behavior.

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the knowledge-review stage should still be able to keep genuinely useful cooking knowledge and still be able to invent new tags when the existing catalog is genuinely missing a concept, but it should stop keeping memoir, book-thesis, motivational, and scene-setting prose just because the model can dream up a plausible tag. The immediate user-visible behavior is that benchmark runs should stop promoting rows such as front matter, manifesto prose, and mixed memoir-plus-advice paragraphs into `knowledge` merely because they mention cooking.

The second user-visible behavior is that `candidate_tag_keys` disappear completely from the program. Knowledge workers should no longer see fuzzy deterministic tag guesses for each row. Instead, they should ground decisions against the real catalog and a stricter retrieval question: is this exact row worth retrieving later as standalone cooking knowledge?

## Context and Orientation

This repository has a stage called `nonrecipe_finalize`. That stage reviews blocks that are outside any accepted recipe and decides whether each block should end up as final `knowledge` or final `other`. In this repository, “knowledge” means a block that would be worth retrieving later as a standalone reusable cooking concept. A block can be true, interesting, or related to cooking and still fail that test if it is mostly memoir, promotional framing, broad motivation, or setup for the book rather than a reusable concept.

The main owners for this work are:

- `cookimport/llm/knowledge_stage/task_file_contracts.py`
  This file builds the worker-facing classification task file and validates the worker’s answers.

- `cookimport/llm/knowledge_tag_catalog.py`
  This file loads the existing knowledge tag catalog and currently generates fuzzy hint lists through `candidate_tag_keys_for_text(...)`.

- `cookimport/llm/codex_farm_knowledge_ingest.py`
  This file validates promoted knowledge outputs against the catalog and records grounding metadata.

- `cookimport/llm/codex_farm_knowledge_models.py`
  This file defines the final knowledge output schema and reason-code vocabulary.

- `cookimport/staging/pipeline_runtime.py` and the knowledge-stage runtime package under `cookimport/llm/knowledge_stage/`
  These files own execution and promotion flow for the knowledge stage.

- `cookimport/llm/fake_codex_farm_runner.py`
  This file mirrors the knowledge worker contract for no-token tests and must stay in sync with the real runtime contract.

The important current behavior is:

- workers classify rows as `knowledge` or `other`;
- a `knowledge` decision must include grounding;
- grounding may use either an existing tag key or a proposed new tag;
- proposed new tags already have to use an existing category key;
- the worker packet currently includes category names and `candidate_tag_keys`, but not the real full existing tag catalog.

The benchmark failure that motivated this plan was not a simple syntax failure. The model often attached plausible cooking-style grounding to bad rows. That means a purely syntactic validator is not enough. The repository needs a stricter review contract and a clearer deterministic authority boundary for “retrievable standalone knowledge” versus “book prose that happens to mention cooking.”

The user has made one hard product decision that this plan must honor: `candidate_tag_keys` are being removed completely.

## Plan of Work

The implementation should begin by changing the worker-facing contract in `cookimport/llm/knowledge_stage/task_file_contracts.py`. The current classification contract asks, in effect, whether a block is `knowledge` or `other`, and then permits grounding by either an existing tag or a proposed tag. Rewrite that contract so it becomes explicitly existing-tag-first. The worker instructions should say, in plain language, that it must first try to ground the row to one or more existing tags from the provided catalog. A proposed tag is allowed only when no existing tag is a clean fit and the row is still clearly worth retrieving as a standalone cooking concept.

At the same time, remove `candidate_tag_keys` from the task-file payload. Delete the field from the unit evidence payload in `cookimport/llm/knowledge_stage/task_file_contracts.py`, remove any code that expects or forwards it, and update the fake runner and any artifact readers accordingly. This is a contract change, so every packet builder, validator, fixture, and fake response path that currently references `candidate_tag_keys` must be updated in the same change.

After removing the fuzzy hints, expose the real tag catalog to the worker. The task file already includes `ontology`, but the live saved packet inspected during research showed only categories in the inline packet. Make the worker packet include the real existing tag list in a compact form. The worker should be able to see, for every tag, its key, display name, and category key. If token pressure becomes a concern, the compact form should still preserve full existing-tag visibility while minimizing wrapper overhead. The goal is that the worker can pick a real tag directly instead of inventing a new one from memory.

Then tighten proposed-tag rules in the validation and ingest path. Keep the existing constraints that proposed tags must use an existing category key and must not conflict with existing tags, but add a stronger product rule to the worker contract and the post-review sanitizer: a proposed tag cannot be the only reason a row survives if the row reads like memoir, front matter, navigation, book framing, praise, manifesto prose, or scene setting. In other words, plausible ontology shape is necessary but not sufficient. The row must still pass the retrieval-worthiness test.

Next, add deterministic demotion rules after knowledge review. The right owner is likely `cookimport/llm/codex_farm_knowledge_ingest.py` or a new small helper adjacent to it. The demotion logic should inspect the accepted row text and drop `knowledge` to `other` when the text matches already-observed low-utility buckets such as memoir, front matter, navigation, decorative headings, and generic filler.

The key principle is that a block should not survive as `knowledge` just because the model can attach a plausible technique tag. If the text is mostly framing, memoir, or motivation, the repository should keep authority and demote it.

The next part of the work is to align the reason-code surface with the binary classification surface. Today the output schema already knows about low-utility failure categories, but the classification path is still too binary and too grounding-focused. Extend the worker instructions and any intermediate metadata so the reviewer is explicitly asked to think in those failure categories even when the final category remains only `knowledge` or `other`. The worker does not need to emit a public multi-class answer if that complicates rollout, but the contract should explicitly say that memoir, framing, navigation, and low-utility truths must land in `other`.

Then update tests to cover the new contract. The key tests should prove all of the following:

- `candidate_tag_keys` no longer appear in knowledge task files or packets.
- workers can ground to existing tags when the correct tag exists.
- workers may still propose a new tag, but only with an existing category key.
- memoir, framing, navigation, and scene-setting rows are demoted to `other` even if the answer includes plausible proposed tags.
- genuinely reusable concept rows with no clean existing tag can still survive as `knowledge` with a new proposed tag.

Finally, update the owning docs. `docs/10-llm/10-llm_README.md` should explain the revised knowledge grounding contract, and the relevant benchmark or staging docs should note that `candidate_tag_keys` were intentionally removed because the deterministic fuzzy hinting produced misleading ontology framing.

## Validation and Acceptance

Acceptance is behavioral, not just structural.

This change is complete when a human can verify all of the following:

- knowledge worker packets no longer contain `candidate_tag_keys`;
- the worker can see the real existing tag catalog in the packet;
- when an existing tag is a reasonable fit, final grounding uses that existing tag instead of inventing a new one;
- when no existing tag fits and the row is genuinely standalone reusable cooking knowledge, a new proposed tag can still be accepted;
- when a row is memoir, front matter, book framing, praise, motivation, navigation, or otherwise low-utility, it lands in `other` even if the model suggests a plausible proposed tag;
- benchmark canaries like the “Fifteen years after arriving at the idea for this book...”, Cal anecdote, and similar front-matter rows no longer survive as final `knowledge`.

The test suite should include at least these classes of cases:

- a positive existing-tag case;
- a positive new-tag case;
- a negative memoir/framing case with plausible cooking words;
- a negative navigation/front-matter case;
- a negative “true but low utility” case.

The implementation should also prove that proposed tags remain category-bounded by existing category keys and that malformed proposals are dropped or rejected as before.

## Idempotence and Recovery

This plan is safe to implement incrementally because the work is additive around one contract boundary. Removing `candidate_tag_keys` is a packet-schema change, so the safest path is to update all producers and consumers in the same patch series before relying on old saved fixtures. If a test or fixture fails midway because it still expects `candidate_tag_keys`, update the fixture to the new packet shape instead of reintroducing the field temporarily.

If the existing-tag catalog inclusion creates token-pressure problems, do not fall back to `candidate_tag_keys`. Instead, compress the existing-tag view into a smaller but still authoritative list, such as compact dictionaries or category-grouped arrays. The recovery path must preserve the product decision that fuzzy hint keys are gone.

If that temporary cleanup layer proves too aggressive on the first attempt, narrow the conditions by targeting only the already-observed failure buckets. Do not weaken the contract by allowing memoir or framing prose back into `knowledge` merely because it mentions cooking techniques.

## Artifacts and Notes

The benchmark evidence that motivated this plan included rows like:

    "Fifteen years after arriving at the idea for this book, I began to write in earnest..."
    category: knowledge
    grounding: proposed tag "salt-fat-acid-heat"

and:

    "After that experience, tasting became reflexive in a whole new way..."
    category: knowledge
    grounding: proposed tag "taste-reflex"

These are the exact anti-patterns this plan is meant to stop. They may contain true cooking claims, but they are not good standalone retrieval units.

The key packet-shape change after implementation should look like this in spirit:

    before:
      unit.evidence.candidate_tag_keys = ["pressure-cook", "no-cook", ...]

    after:
      unit.evidence has no candidate_tag_keys field
      task_file.ontology includes real categories and tags

The key policy change after implementation should read, in plain English, like this:

    Use an existing tag if one fits.
    Propose a new tag only when no existing tag fits and the row is still worth retrieving as standalone cooking knowledge.
    If the row is memoir, framing, praise, navigation, or motivation, mark it as other.

## Interfaces and Dependencies

In `cookimport/llm/knowledge_stage/task_file_contracts.py`, keep the current binary final category interface:

    category: "knowledge" | "other"

Keep the current grounding shape, but enforce stronger semantics around it:

    grounding = {
        "tag_keys": [existing_tag_key, ...],
        "category_keys": [existing_category_key, ...],
        "proposed_tags": [
            {
                "key": normalized_slug,
                "display_name": "Short Name",
                "category_key": existing_category_key
            }
        ]
    }

In `cookimport/llm/knowledge_tag_catalog.py`, preserve the catalog loader and `task_scope_payload()` as the authority for existing categories and tags. Remove `candidate_tag_keys_for_text(...)` if nothing else uses it after this change; if some non-knowledge surface still uses it, split that helper away so the knowledge stage no longer depends on or exposes it.

In `cookimport/llm/codex_farm_knowledge_ingest.py`, add a small, explicit demotion seam that receives the row text plus normalized grounding and can return final `other` when the row matches low-utility prose buckets. Keep this deterministic and explainable.

In the fake runner and tests, preserve the ability to simulate both:

- a grounded existing-tag acceptance;
- a grounded proposed-tag acceptance;
- a grounded-but-demoted low-utility row.
