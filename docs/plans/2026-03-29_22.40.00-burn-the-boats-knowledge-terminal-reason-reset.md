## Structured Restatement Of User Desire

The knowledge stage must stop collapsing every no-result failure into the same vague `missing_output` bucket. After the semantic-authority reset and the leased-packet runtime reset land, the repository still needs truthful machine-readable terminal reasons that say what actually happened: whether the worker exited with the current packet still leased, whether validation blocked promotion, whether a repair packet was exhausted, whether the process died for a known watchdog reason, or whether no better evidence exists. This is burn-the-boats work. The old generic story should not remain as a compatibility path, and the new explicit terminal-reason model should become the only live knowledge runtime behavior, tests, docs, and summaries.

## Purpose / Big Picture

After this change, a user who inspects a failed knowledge run can immediately tell why a knowledge packet failed without opening opaque worker prose or reverse-engineering the workspace. If a leased packet was still active when the worker exited, the artifacts should say that. If a packet result existed but promotion was blocked by structural validation, the artifacts should say that. If the worker died for a watchdog reason, the artifacts should preserve that reason instead of flattening it to `missing_output`. A failed run should no longer lie by implying that all no-result cases are the same.

The visible proof is straightforward. Knowledge task records, packet status artifacts, failure summaries, stage summaries, and benchmark-facing reports should all agree on one specific terminal reason for each failed packet. The worker root should contain enough machine-readable state to derive that reason without relying on natural-language last messages. A novice should be able to inspect the relevant files and see one coherent story: what packet the worker owned, what state that packet reached, why promotion did not happen, and how the runtime classified the outcome.

## Context and Orientation

A “leased packet” in this repository means the one current semantic unit the worker is allowed to see and answer before the repo advances the lease. The worker root should expose files such as `current_packet.json`, `current_hint.md`, `current_result_path.txt`, and `packet_lease_status.json`, while the repo decides packet advancement, structural validation, repair-packet creation, and promotion.

Two terms need plain-English definitions here.

A “terminal reason” is the final machine-readable answer to “why did this packet end the way it did?” It is different from a generic state such as `completed` or `no_final_output` because it explains the actual failure mode, for example “worker exited with current packet still leased,” “repair packet exhausted,” or “watchdog-killed after boundary violation.”

A “lease surface” is the repo-owned set of files that tells the worker what it owns right now and what state that ownership is in. In this repository, that means `current_packet.json`, `current_result_path.txt`, `current_hint.md`, `packet_lease_status.json`, and the corresponding worker status artifacts. This plan makes those lease surfaces authoritative for no-result failure classification.

## Plan Of Work

The first part of the work is to define one explicit terminal-reason taxonomy for the leased-packet knowledge runtime and apply it everywhere. The exact names can change during implementation, but the taxonomy must distinguish at least these situations:

1. The worker exited while a current packet was still leased and no result had been finalized.
2. The worker produced a result candidate, but promotion was blocked by structural validation and the packet remained unresolved.
3. A repair packet or bounded follow-up path was exhausted without producing a promotable final result.
4. The worker was terminated for an existing watchdog or supervision reason that already explains the failure.
5. No better evidence exists, in which case a final generic no-result fallback may remain as the least-preferred branch.

The second part is to make the lease surface machine-readable enough to support those reasons. The runtime and shared runner must update `packet_lease_status.json` or an equivalent repo-synced status file with explicit fields such as current packet id, pass kind, lease state, validation state, repair state, whether a result file was observed, whether promotion was attempted, whether promotion succeeded, and which packet was active when the worker stopped. The runner should not need to scrape prose to know why promotion did not happen.

The third part is to replace the current “absent final output means `missing_output`” shortcut in the knowledge runtime. When the expected result does not exist or did not promote, the runtime should inspect repo-side worker state and classify the failure from strongest evidence to weakest evidence. It should honor a specific supervision reason first when the worker was watchdog-killed. Otherwise it should read the lease status and determine whether the packet was still leased, blocked in structural validation, exhausted after repair, or incorrectly completed without finalizing output. That classification should flow into proposal rows, worker failure rows, task-status terminal reasons, and stage summaries.

The fourth part is to remove or rewrite every downstream consumer that flattens explicit reasons back into generic `missing_output`. This includes knowledge recovery helpers, stage observability, prompt-budget summaries, benchmark helper reporting, and analytics surfaces that still count only `missing_output` packets. It is acceptable for some top-level summaries to keep a derived “no final output” rollup, but that must be a rollup over the new explicit reasons rather than the only surviving truth.

The fifth part is test and documentation cleanup. Tests must prove that the runtime distinguishes these specific no-result cases. Docs must explain the new truth source and reason-code story without referencing compatibility. A novice reading the LLM and analytics docs should be able to see which files are authoritative, which reason codes are possible, and how to interpret a failed knowledge worker run.

## Milestones

At the end of this milestone, the knowledge leased-packet runtime owns a clear machine-readable status contract in the repo-synced worker root. A focused unit or integration test can inspect `packet_lease_status.json` and see whether the current packet was leased, validated, blocked, repaired, promoted, or left unresolved without reading markdown or prose artifacts.

### Milestone 2: Replace generic `missing_output` with explicit terminal reasons

At the end of this milestone, the knowledge runtime no longer treats “no promotable final result exists” as a complete explanation. It classifies the outcome using lease-status fields plus live supervision facts. Targeted runtime tests can force each no-result branch and observe distinct terminal reason codes rather than one shared `missing_output`.

### Milestone 3: Propagate explicit reasons through every repo-side summary surface

At the end of this milestone, task-status artifacts, failure summaries, worker status, stage summaries, and benchmark-oriented summary layers all preserve the new explicit reasons. Summary builders and benchmark helper tests see the correct reason counts and do not silently flatten them back into `missing_output`.

### Milestone 4: Delete the old story from tests and docs

At the end of this milestone, active tests and docs teach only the new terminal-reason model. Repository search should no longer show active knowledge-runtime assertions that generic `missing_output` is the canonical explanation for every no-result failure.

## Validation And Acceptance

Acceptance is behavioral and must be demonstrated with tests plus artifact inspection.

The first acceptance criterion is explicit leased-packet classification. A test must create a knowledge worker root where the current packet remains leased and unresolved when the worker exits. The runtime must classify that packet with a specific leased-or-incomplete terminal reason, not `missing_output`.

The second acceptance criterion is explicit blocked-promotion classification. A test must simulate a packet result that exists but cannot be promoted because structural validation or repair exhaustion left the packet unresolved. The runtime must report that exact kind of terminal reason rather than generic no-output wording.

The third acceptance criterion is watchdog precedence. A test must show that when `live_status.json` records a real watchdog reason, the final knowledge task reason preserves that watchdog explanation and does not downgrade it to a generic no-result bucket.

The fourth acceptance criterion is summary propagation. Task-status artifacts, failure summaries, worker status, and the knowledge stage summary must all preserve the same terminal reason for the same packet. A stage summary may still expose an aggregate “no final output” count, but it must derive that from the explicit reason codes and keep the specific reason counts available.

The fifth acceptance criterion is documentation clarity. A novice must be able to read the updated docs and understand which files tell the truth when a knowledge worker fails under the leased-packet runtime.

## Idempotence And Recovery

The steps in this plan are safe to repeat because they are ordinary source edits plus tests. The safest implementation order is the milestone order above: define the new machine-readable lease status first, then switch runtime classification, then update downstream summaries and docs. If the work stops halfway, resume by finishing the end-to-end cutover rather than adding temporary adapters to support both the old generic story and the new explicit one. This plan is intentionally burn-the-boats: do not preserve compatibility branches that keep the old generic knowledge `missing_output` story alive.

If a test fails midway because a consumer still expects `missing_output`, update that consumer immediately to the new reason vocabulary rather than adding translation shims. The goal is one live truth model, not a bilingual transition layer.
