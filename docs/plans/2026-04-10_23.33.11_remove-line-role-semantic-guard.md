# Remove Line-Role Semantic Guard Entirely

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, canonical line-role runs should stop letting deterministic code veto structurally valid model labels for semantic reasons. A human should be able to rerun the same style of benchmark that failed on `2026-04-10_23.20.52` and see that line-role only fails on structural contract problems such as malformed JSON, wrong label count, missing rows, or illegal labels, not because repo heuristics decided that a row “looked like” an ingredient, instruction, note, or variant.

The user-visible reason is authority hygiene. This repository’s rules say deterministic code should package evidence and validate shape, while LLMs should make fuzzy semantic calls. The current semantic guard breaks that boundary by second-guessing the model after a valid answer already exists. Removing it restores the intended line-role contract.

## Surprises & Discoveries

- Observation: the benchmark failure was dominated by semantic-guard vetoes rather than malformed worker output.
  Evidence: the failed run wrote `semantic_invariant_violation:*` entries across seven line-role shards while the recorded repair responses were still valid JSON `labels` arrays.

- Observation: the current repair path compounds the problem by resuming prior transcript history.
  Evidence: one 8-row repair packet had roughly 4.9k visible input tokens but more than 208k billed total tokens in runtime telemetry.

- Observation: the current docs already contradict the broader project authority rule.
  Evidence: `docs/10-llm/10-llm_README.md` and `docs/04-parsing/04-parsing_readme.md` describe a live semantic-guard rejection layer even though the repo guidance says deterministic code should not make fuzzy cookbook-semantic calls.

## Outcomes & Retrospective

Canonical line-role no longer has an active deterministic semantic veto seam. Structurally valid inline `labels` arrays now stay accepted after the normal shape, ownership, and label-contract checks. Repair remains available only for structural failures. The focused runtime and pathology suites passed after the change, so the live contract is now aligned with the repo rule that deterministic code validates structure while the model owns fuzzy semantic calls.

The narrow cleanup also removed stale telemetry assumptions around semantic suspicion. Shard-status suspicion counts now reflect unresolved rows only, which better matches the live repair/failure behavior. What remains outside this plan is empirical rerun evidence on the same benchmark target, but the code path that produced `semantic_invariant_violation:*` has been deleted.

## Context and Orientation

Canonical line-role validation lives in `cookimport/parsing/canonical_line_roles/validation.py`. That module evaluates the returned `labels` array, maps it back onto the shard’s owned `atomic_index` order, records accepted rows, and currently applies an extra post-validation semantic guard. The semantic guard builds diagnostics such as `recipe_ingredient_anchor`, `recipe_instruction_anchor`, `recipe_note_anchor`, and `recipe_variant_anchor`, then converts those into `semantic_invariant_violation:*` errors and forces the shard back into repair.

Repair triggering lives in `cookimport/parsing/canonical_line_roles/runtime_watchdog.py`. That module treats `semantic_invariant_violation:*` like any other repair-triggering validation failure. Once the validation path stops producing those errors, the repair loop should naturally return to structural-only triggers.

The focused runtime behavior is covered by `tests/parsing/test_canonical_line_roles_runtime.py`. That file currently contains tests that explicitly assert semantic-guard rejection and repair metadata. `tests/parsing/test_canonical_line_roles_front_matter_pathology.py` also contains a small assertion about semantic diagnostics being absent on a safe front-matter case. The docs that describe current behavior are `docs/04-parsing/04-parsing_readme.md` and `docs/10-llm/10-llm_README.md`.

## Plan of Work

First, edit `cookimport/parsing/canonical_line_roles/validation.py` so the response-evaluation path no longer calls or relies on `_apply_line_role_semantic_guard(...)`. The simplest safe outcome is that structurally valid payloads return the structural validation errors and metadata only, with no semantic veto pass layered on top. If the semantic helper functions become dead code after that change, delete them so the module no longer advertises a latent semantic-rejection seam.

Next, update the focused tests. The runtime tests that currently expect `semantic_invariant_violation:*`, `semantic_diagnostics`, and `semantic_rejected_atomic_indices` should be rewritten to assert that the same structurally valid payload is accepted without repair. Keep structural-malformation tests intact. The front-matter pathology test should remain, but only as confirmation that the payload validates cleanly without semantic metadata rather than as evidence for a guard that no longer exists.

Then update the short docs. The parsing and LLM READMEs should say that line-role deterministic code validates structure, label legality, shard ownership, and exact-count rules, but it does not reject accepted labels for semantic disagreement after structural validation.

Finally, run the focused parsing tests inside the project `.venv` and record the observed result here.
