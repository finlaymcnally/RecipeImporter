## Purpose / Big Picture

After this change, the label-first benchmark should stop losing large amounts of obvious cookbook knowledge prose and stop promoting book/chapter headings into recipe structure. A human should be able to run the `saltfatacidheatcutdown` benchmark again and see better outside-recipe accuracy, better `KNOWLEDGE` recall, and fewer wrong labels like `SALT -> HOWTO_SECTION` or `HOW SALT WORKS -> OTHER`.

The user-visible behavior to enable is simple: when Codex or deterministic line-role reads a pedagogical cookbook, outside-recipe explanatory material such as concept headings, section labels like `Meat` or `Seafood`, and short cooking-science prose should be recognized as `KNOWLEDGE` much more reliably, while non-recipe table-of-contents and chapter headings should stop getting absorbed into recipe structure labels such as `HOWTO_SECTION` or `RECIPE_TITLE`.

## Surprises & Discoveries

- Observation: After span removal, the main benchmark gap is no longer recipe-span gating. The strongest recipe-local slices are already good, while outside-recipe remains weak.
  Evidence: In `data/golden/benchmark-vs-golden/2026-03-17_20.27.19/.../line-role-pipeline/slice_metrics.json`, `recipe_instructions` is `1.0000` accuracy, `recipe_ingredients` is `0.9818`, but `outside_recipe` is only `0.5830`.

- Observation: The biggest remaining miss path is `KNOWLEDGE -> OTHER`, not `KNOWLEDGE -> recipe structure`.
  Evidence: `wrong_label_lines.jsonl` + `missed_gold_lines.jsonl` for the same run show `356` gold `KNOWLEDGE` rows missed as `OTHER`, including `HOW SALT WORKS`, `How Salt Affects . . .`, `Meat`, and long explanatory salting/brining paragraphs.

- Observation: There is a second distinct failure mode where book/chapter headings are promoted into recipe structure labels.
  Evidence: The same run labels `SALT`, `What is Salt?`, `How Salt Works`, `Kitchen Basics`, and `About the Author` as `HOWTO_SECTION`, and `The Four Elements of Good Cooking` as `RECIPE_TITLE`.

- Observation: Codex is already rescuing many knowledge lines; the problem is the remaining tail, not complete inability.
  Evidence: `joined_line_table.jsonl` shows `332` gold `KNOWLEDGE` rows correctly labeled as `KNOWLEDGE` by Codex outside recipe, but `165` more still land in `OTHER`.

## Context and Orientation

The relevant runtime seam is the canonical line-role classifier in [cookimport/parsing/canonical_line_roles.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles.py). That module performs deterministic labeling first, then optional Codex shard review, then light sanitization. A pre-grouping `AtomicLineCandidate` is one atomic line from the book with local text context and heuristic `rule_tags`, defined in [cookimport/parsing/recipe_block_atomizer.py](/home/mcnal/projects/recipeimport/cookimport/parsing/recipe_block_atomizer.py). Since the recent architecture change, pre-grouping candidates default to `within_recipe_span=None`, which means â€śrecipe membership is still unknown.â€ť The classifier must therefore infer recipe versus non-recipe structure from the content itself rather than from inherited old recipe ranges.

The compact prompt seen by Codex is built in [cookimport/llm/canonical_line_role_prompt.py](/home/mcnal/projects/recipeimport/cookimport/llm/canonical_line_role_prompt.py) and the matching prompt template at [llm_pipelines/prompts/canonical-line-role-v1.prompt.md](/home/mcnal/projects/recipeimport/llm_pipelines/prompts/canonical-line-role-v1.prompt.md). That prompt now explicitly says there is no prior recipe-span authority. Any changes in prompt guidance must reinforce better knowledge/non-recipe classification without sneaking recipe membership hints back in.

The benchmark evidence for this plan comes from `data/golden/benchmark-vs-golden/2026-03-17_20.27.19/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/`. The key artifacts are:

- `wrong_label_lines.jsonl`, which records wrong predicted labels against the gold set
- `missed_gold_lines.jsonl`, which records missed gold labels
- `line-role-pipeline/joined_line_table.jsonl`, which is the easiest line-by-line alignment table for diagnosing how a given gold label flowed through line-role
- `line-role-pipeline/slice_metrics.json`, which summarizes recipe-local versus outside-recipe behavior

The current findings are already summarized in [docs/understandings/2026-03-17_21.42.00-knowledge-failures-after-span-removal-are-mostly-heading-and-prose-disambiguation.md](/home/mcnal/projects/recipeimport/docs/understandings/2026-03-17_21.42.00-knowledge-failures-after-span-removal-are-mostly-heading-and-prose-disambiguation.md). This ExecPlan includes the important parts here so it remains self-contained.

The main failure buckets to fix are:

1. Gold `KNOWLEDGE` collapsing to `OTHER`, especially for compact concept headings and explanatory prose that does not happen to contain the current strongest explicit knowledge cue phrases.
2. Non-recipe book/chapter headings being promoted to recipe-structure labels such as `HOWTO_SECTION` or `RECIPE_TITLE`.
3. Keeping the existing strong recipe-local slices intact while improving (1) and (2).

## Plan of Work

Start in [cookimport/parsing/canonical_line_roles.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles.py). Audit the knowledge-related helpers first: `_looks_knowledge_heading_with_context(...)`, `_looks_knowledge_prose_with_context(...)`, `_looks_domain_knowledge_prose(...)`, and the deterministic branches that currently send unknown-span prose to `OTHER` when no stronger cue is found. The benchmark examples show that short headings like `HOW SALT WORKS`, `How Salt Affects . . .`, `Meat`, `Seafood`, and `Fat` need a better non-recipe pedagogical heading path. The implementation should introduce a clearer notion of cookbook concept headings and section labels that can resolve to `KNOWLEDGE` when they are surrounded by explanatory non-recipe prose, even when they are short and lack explicit scientific cue words.

In the same file, tighten the recipe-structure promotion paths. The benchmark shows chapter-like or contents-like headings turning into `HOWTO_SECTION` or `RECIPE_TITLE`. The likely targets are `_looks_subsection_heading_context(...)`, `_looks_recipe_title_with_context(...)`, `_supports_recipe_title_context(...)`, and any deterministic branch that maps title-like or heading-like tags to recipe structure too aggressively. The new behavior should remain permissive for real recipe-local structure, but it should reject general book organization headings when the surrounding context is clearly outside-recipe pedagogical prose or table-of-contents style material.

After the deterministic heuristics are adjusted, inspect [cookimport/llm/canonical_line_role_prompt.py](/home/mcnal/projects/recipeimport/cookimport/llm/canonical_line_role_prompt.py) and [llm_pipelines/prompts/canonical-line-role-v1.prompt.md](/home/mcnal/projects/recipeimport/llm_pipelines/prompts/canonical-line-role-v1.prompt.md). Only change prompt guidance if the deterministic fix needs clearer Codex steering. If prompt edits are needed, they must remain aligned with the span-free contract. Good prompt additions would be statements such as: general chapter/part headings are not recipe structure by default, and compact pedagogical headings followed by explanatory prose should usually be `KNOWLEDGE` rather than `HOWTO_SECTION`.

Then add and update tests. The main test file is [tests/parsing/test_canonical_line_roles.py](/home/mcnal/projects/recipeimport/tests/parsing/test_canonical_line_roles.py). Add cases for:

- chapter/part/contents headings staying `OTHER` instead of promoting to `HOWTO_SECTION` or `RECIPE_TITLE`
- unknown-span explanatory prose continuing to resolve to `KNOWLEDGE`
- recipe-local how-to and title flows still behaving correctly so the new guards do not erase real recipe structure

If atomizer rule tags need to change to support the new distinction, update [cookimport/parsing/recipe_block_atomizer.py](/home/mcnal/projects/recipeimport/cookimport/parsing/recipe_block_atomizer.py) and [tests/parsing/test_recipe_block_atomizer.py](/home/mcnal/projects/recipeimport/tests/parsing/test_recipe_block_atomizer.py) accordingly. Keep that change minimal: the main intent is still classification in `canonical_line_roles.py`, not reintroducing a heavier pre-classification system in the atomizer.

Finally, validate on the benchmark artifacts. A full new single-offline rerun is ideal, but if runtime or token budget is a concern, first run the focused tests and then either run a narrow benchmark or at minimum regenerate the relevant label/eval artifacts for `saltfatacidheatcutdown`. Success means seeing fewer `KNOWLEDGE -> OTHER` misses and fewer `OTHER -> HOWTO_SECTION/RECIPE_TITLE` promotions on obvious non-recipe chapter material.

## Validation and Acceptance

Acceptance is behavioral and benchmark-visible.

At the unit-test level:

- `tests/parsing/test_canonical_line_roles.py` must include new cases proving that pedagogical knowledge headings and short concept labels can resolve to `KNOWLEDGE` in the span-free pre-grouping state.
- The same file must prove that chapter/contents/book headings do not become `HOWTO_SECTION` or `RECIPE_TITLE` just because they are heading-shaped.
- Existing recipe-local tests must continue to pass, proving that recipe structure did not regress while knowledge behavior improved.

At the benchmark level:

- The rerun should improve outside-recipe metrics in `line-role-pipeline/slice_metrics.json`, especially the `outside_recipe` slice.
- `wrong_label_lines.jsonl` and `missed_gold_lines.jsonl` should show fewer rows in the `KNOWLEDGE -> OTHER` bucket.
- The characteristic wrong rows `SALT`, `What is Salt?`, `How Salt Works`, `Kitchen Basics`, and similar chapter/section lines should shrink as `HOWTO_SECTION` or `RECIPE_TITLE` errors.
- The strong recipe-local slices from the span-free architecture should remain strong: ingredient and instruction accuracy should not materially collapse.

The strongest sign of success is a rerun where the benchmark still uses the intended span-free prompt contract, but outside-recipe accuracy and `KNOWLEDGE` recall move upward without reintroducing any old recipe-range hint path.

That file must continue to expose the same public output type, `CanonicalLineRolePrediction`, and the same main entrypoint, `label_atomic_lines(...)`. If the implementation needs new helper functions, they should live in the same module near the current knowledge/title/howto helpers so the knowledge-vs-structure reasoning remains in one place. Any helper must preserve the tri-state span model:

- `within_recipe_span is None` means pre-grouping unknown and is the normal state for the first line-role pass
- `within_recipe_span is True` means later projection has already established accepted recipe membership
- `within_recipe_span is False` means later projection has already established outside-recipe membership

Prompt edits, if any, belong in both `cookimport/llm/canonical_line_role_prompt.py` and `llm_pipelines/prompts/canonical-line-role-v1.prompt.md` so live prompt generation and prompt-preview stay aligned.
