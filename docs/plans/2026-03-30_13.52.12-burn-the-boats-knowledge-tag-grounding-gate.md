This plan assumes the live knowledge stage has already been simplified to a classification-first, grouping-second flow. The work here is the next semantic tightening step: once grouping pressure is gone, the classifier must still stop treating “culinary meaning” as enough. The new bar is retrieval-grade, ontology-groundable knowledge.

## Exact User Desire

> "The nice part is that this gives you a practical test for KNOWLEDGE:
>
> - portable across recipes
> - concept-shaped, not just inspirational or stylistic
> - attachable to at least one tag/category
> - useful later in retrieval without needing the original paragraph around it
>
> So the failure in the current run becomes very concrete: the model is treating 'culinary meaning' as enough, when the real bar should be 'retrieval-grade, ontology-groundable knowledge.'"

## Structured Restatement Of User Desire

After the current burn-the-boats knowledge reset lands, the first-pass knowledge classifier must stop asking a vague question such as “is this cooking-adjacent and useful-sounding?” Instead it must ask a storage and retrieval question: does this block express a stable concept that can stand on its own later, can be attached to the Cookbook ontology, and would be worth surfacing while someone cooks? If the answer is no, the block stays `other` even if the prose is true, elegant, or broadly wise.

The user already has a real ontology to support this: the Cookbook tag catalog currently contains 20 categories and 203 tags, copied into this repository for reference in [docs/11-reference/2026-03-30_13.47.43_cookbook-tag-catalog-snapshot.md](/home/mcnal/projects/recipeimport/docs/11-reference/2026-03-30_13.47.43_cookbook-tag-catalog-snapshot.md). The knowledge step must start using that ontology as a semantic anchor. Future tag growth is expected, so the design must also allow the model to propose a new tag candidate when a block is genuinely retrieval-grade but the current catalog is missing the concept.

## Purpose / Big Picture

After this change, the live knowledge stage will keep only outside-recipe text that is both semantically useful and grounding-ready. A kept block will have to produce a short retrieval concept plus at least one ontology anchor. In practice that means an existing Cookbook tag key when possible, or a crisp proposed tag under an existing category when the catalog is missing a stable concept. Text that is true but not storable, such as motivational advice, scene-setting, memoir, chapter navigation, or title lists, will stay `other`.

The user-visible proof is concrete. The current bad examples from the March 30, 2026 benchmark must stop being promotable just because they “sound like cooking.” Title-only lists such as vinaigrette or dressing names must stay `other` unless they encode a real durable concept. Advice like “Be present. Stir, taste, adjust.” must stay `other` because it is not retrieval-grade knowledge, even though it is good advice. By contrast, blocks that express durable cooking concepts must still remain `knowledge`, and they must now carry visible grounding metadata that explains what concept would later be searchable.

## Surprises & Discoveries

- Observation: The repository already has a real tag universe available, not just a tiny prompt helper list.
  Evidence: The copied Cookbook snapshot contains 20 categories and 203 tags covering cuisine, flavor profile, dietary, techniques, equipment, main protein, main carb, cooking method, and other retrieval-relevant axes.

- Observation: “Attachable to a category” by itself is too weak.
  Evidence: Generic advice like “Be present. Stir, taste, adjust.” can be hand-waved into a broad category like `techniques`, but it still fails the real storage test because it is not a stable standalone concept someone would intentionally retrieve later.

- Observation: Exact existing-tag matching by itself is also too weak as a long-term design.
  Evidence: The user explicitly wants more tags over time, and durable concepts can exist before the catalog grows to name them exactly. The design therefore needs a proposal seam for genuinely new concepts.

- Observation: The right semantic question is narrower than “is this culinary?”
  Evidence: The motivating false positives are not non-cooking. They are cooking-adjacent but not database-worthy. The classifier must learn “worth storing and grounding” rather than “sounds like cooking knowledge.”

- Observation: The copied Cookbook catalog still has at least one legacy wrinkle.
  Evidence: The snapshot contains both `meal` and `meal-type`, but only `meal-type` is populated. That does not break the design, but it means grounding code must treat the ontology as real-but-imperfect and validate against the checked-in runtime copy rather than assumptions.

## Outcomes & Retrospective

This plan is newly authored and not yet implemented. Its intended outcome is a knowledge boundary that is strict for the right reason. The stage should no longer decide “this feels like culinary wisdom, so keep it.” Instead it should decide “this is a portable concept a cook could later retrieve, and here is the ontology anchor that makes that retrieval possible.”

The main lesson from the recent benchmark analysis is that prompt strictness alone is not the right abstraction. The stage needs a better definition of what stored knowledge is for. The Cookbook ontology provides that missing definition, but only if the contract uses it as an actual gate rather than decorative context.

## Context and Orientation

In this repository, the “knowledge stage” is the optional LLM-backed nonrecipe-finalize stage that decides whether outside-recipe blocks become final `knowledge` or final `other`, then groups accepted knowledge blocks for reviewer artifacts and final staging outputs. The active public entrypoint is `cookimport/llm/codex_farm_knowledge_orchestrator.py`, and the active stage package is `cookimport/llm/knowledge_stage/`. Prompt assembly currently flows through `cookimport/llm/knowledge_prompt_builder.py` and the prompt templates under `llm_pipelines/prompts/`.

The prior burn-the-boats knowledge-boundary plan assumes the live contract becomes classification-first and grouping-second. That means there is already a first-pass classifier that decides `knowledge` versus `other` for one block at a time, plus a second grouping pass that only organizes accepted knowledge blocks. This plan changes the first pass only. Grouping still happens later, but the keep/drop decision becomes harder to earn.

The Cookbook ontology is currently available in this repo as a human-readable reference snapshot at `docs/11-reference/2026-03-30_13.47.43_cookbook-tag-catalog-snapshot.md`. That file is not a runtime artifact. The implementation in this plan must add a machine-readable checked-in runtime copy because the knowledge stage and its tests cannot depend on reading another repository or parsing Markdown.

The key terms in this plan are “retrieval-grade concept” and “ontology grounding.” A retrieval-grade concept is a short, stable idea that can stand alone without the rest of the original paragraph and would actually help later during search, filtering, or in-cooking retrieval. Ontology grounding means the concept can point to at least one current tag key, or can justify one proposed new tag under an existing category. Generic exhortation, posture, philosophy, memoir, scene-setting, and title lists do not qualify just because they are relevant to cooking.

## Plan of Work

First, add a machine-readable ontology artifact and one tiny sync tool. Create `cookimport/llm/knowledge_tag_catalog.json` as the checked-in runtime copy of the Cookbook ontology. Create `cookimport/llm/knowledge_tag_catalog.py` or `cookimport/llm/knowledge_stage/ontology.py` as the loader/helper module that reads the JSON, resolves category/tag lookups, and exposes compact task-ready structures. Add `scripts/sync_cookbook_tag_catalog.py` so a future contributor can refresh the checked-in JSON from a seed SQL file such as `~/projects/cookbook/supabase/seeds/catalog.sql` when the catalog changes. The sync script must be offline, explicit, and safe to rerun. Runtime code must never depend on the external Cookbook repo being present.

Second, extend the first-pass classification contract so `knowledge` means “kept and grounded,” not merely “kept.” The new classification `answer` payload must keep `category` and `reviewer_category`, but must also add `retrieval_concept` and a `grounding` object. `retrieval_concept` is the short standalone concept phrase that would later be worth surfacing to a cook. The `grounding` object must contain existing `tag_keys`, optional supporting `category_keys`, and optional `proposed_tags`. A block may only be labeled `knowledge` when it has a non-empty `retrieval_concept` and at least one anchor in `tag_keys` or `proposed_tags`. Category keys by themselves are not sufficient because they are too broad.

Third, change the classifier prompt so it teaches the right semantic bar. Update the classifier prompt template in `llm_pipelines/prompts/` and the assembly logic in `cookimport/llm/knowledge_prompt_builder.py` so the task explicitly asks: “Would this block be worth storing and later retrieving on its own, and if so, what ontology concept does it attach to?” The prompt must include both positive and negative examples. It must explicitly say that “true but not storable” advice remains `other`. It must explain that title lists, taxonomy, motivational prose, and broad posture advice do not become `knowledge` unless they resolve into a stable retrieval concept.

Fourth, give the model ontology evidence without turning deterministic code into a semantic classifier. Every classification task file should include the full category list and the compact tag catalog once at task scope, not repeated inside each unit. Deterministic code may also attach a lightweight `candidate_tag_keys` shortlist per unit using plain normalized lexical overlap against tag display names and category names. That shortlist is a hint only. The model may choose a different existing tag from the task-scope ontology or propose a new tag if the concept is real and the current catalog is missing it.

Fifth, add proposal capture without automatic writes. When a block is kept as `knowledge` using a proposed tag instead of an existing tag, record that proposal in a deduplicated stage sidecar such as `knowledge_tag_proposals.jsonl`. Promotion into final `09_nonrecipe_authority.json` still depends only on the validated `knowledge|other` decision. The proposed-tag sidecar is for future catalog growth and review, not a new source of authority.

Sixth, update validation and promotion. The classifier validator in the knowledge-stage ingest/validation seam must reject malformed grounding shapes, unknown tag keys, unknown category keys, empty retrieval concepts on kept blocks, and malformed proposed tags. It must not semantically judge whether a chosen tag is the best tag. Promotion code must carry grounding metadata into reviewer-facing artifacts and summaries so a contributor can answer three concrete questions from the run artifacts alone: how many kept blocks grounded to existing tags, how many needed proposed tags, and how many blocks were rejected because they were useful-sounding but not retrieval-grade.

Seventh, lock the behavior in tests and docs. Add real regression fixtures from the March 30, 2026 failures and make them permanent. Update `docs/10-llm/10-llm_README.md` so the active knowledge-stage contract teaches the ontology-grounded definition instead of the older “durable cooking leverage” phrasing alone. Update the new plan that introduced classification isolation if implementation details shift, so the two plans remain consistent.

## Canonical Classification Grounding Schema

After the prior classification-isolation plan lands, the first-pass knowledge classifier must extend its editable `answer` subtree to the following shape:

    {
      "category": "knowledge" | "other",
      "reviewer_category": "knowledge" | "chapter_taxonomy" | "decorative_heading" | "front_matter" | "toc_navigation" | "endorsement_or_marketing" | "memoir_or_scene_setting" | "reference_back_matter" | "other",
      "retrieval_concept": "<non-empty short phrase>" | null,
      "grounding": {
        "tag_keys": ["<existing tag key>", "..."],
        "category_keys": ["<existing category key>", "..."],
        "proposed_tags": [
          {
            "key": "<normalized proposed key>",
            "display_name": "<short display name>",
            "category_key": "<existing category key>"
          }
        ]
      }
    }

The required validator behavior is:

1. `category = knowledge` requires `reviewer_category = knowledge`.
2. `category = knowledge` requires a non-empty `retrieval_concept`.
3. `category = knowledge` requires at least one existing `tag_key` or at least one `proposed_tag`.
4. `category = other` requires `retrieval_concept = null`, no `tag_keys`, and no `proposed_tags`.
5. Every `tag_key` must exist in `cookimport/llm/knowledge_tag_catalog.json`.
6. Every `category_key` and every proposed-tag `category_key` must exist in the same checked-in catalog.
7. Proposed tags must have normalized unique keys, short display names, and must not duplicate an existing tag key exactly.
8. Deterministic validation must stop at structure, ownership, and catalog membership. It must not reject a structurally valid answer because repo heuristics disagree with the chosen grounding.

The important semantic point is that `category_keys` are explanatory support, not the acceptance gate. The acceptance gate is: does this block yield a crisp concept that grounds to an existing tag or a plausible new tag?

## Canonical Task-Scope Ontology Payload

Each classification assignment file must expose the ontology once at task scope in an immutable section such as:

    {
      "ontology": {
        "catalog_version": "cookbook-tag-catalog-2026-03-30",
        "categories": [
          {"key": "techniques", "display_name": "Techniques"},
          {"key": "flavor-profile", "display_name": "Flavor Profile"}
        ],
        "tags": [
          {"key": "emulsify", "display_name": "Emulsify", "category_key": "techniques"},
          {"key": "beans", "display_name": "Beans", "category_key": "main-protein"}
        ]
      },
      "units": [
        {
          "unit_id": "knowledge::3803",
          "evidence": {
            "text": "...",
            "candidate_tag_keys": ["emulsify", "beans"]
          },
          "answer": { ... editable ... }
        }
      ]
    }

The ontology must be immutable. The model uses it as a grounding surface, not as editable state. `candidate_tag_keys` are optional hints built by deterministic lexical matching and must never be treated as acceptance decisions.

## Regression Fixture Set

The fixture family for this plan must keep the motivating failure shapes alive in the repository.

The first group is “good advice but not storable.” This includes the exact motivational advisory example already identified in the benchmark analysis: “Be present. Stir, taste, adjust.” The expected answer is `other`, with no retrieval concept and no grounding.

The second group is “title-only or menu-like lists.” This includes short rows such as vinaigrette and dressing names that feel topical but do not express a durable concept by themselves. The expected answer is `other`.

The third group is “real retrieval-grade concepts.” These are positive examples from the same or similar cookbook material that express technique, cause and effect, troubleshooting, sensory interpretation, or ingredient behavior. The expected answer is `knowledge` with a retrieval concept and real grounding.

The fourth group is “novel but real concept not yet in the catalog.” Use at least one positive example that deserves `knowledge` but has no satisfying current tag key. The expected answer is `knowledge` with a proposed tag under an existing category, and the promotion/reporting path must surface that proposal without mutating the catalog.

The fifth group is “near-miss broadness.” These are blocks that can be loosely connected to a category like `techniques` or `flavor-profile` but still fail because they are too vague, too philosophical, or too tied to surrounding prose to be retrieved alone. These cases are the critical proof that category-only grounding is not enough.
