# Fix Recipe Prompt Target Shard Contract

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## User Requirement, Verbatim

"IF I SAY I WANT 5 SHARDS. I WANT ALL N (in this case, 29) RECIPES DIVIDED INTO 5 DIFFERENT SHARDS TO BE PROCESSED BY 5 UNIQUE AGENTS. this was not intentional design, that was drift."

## User Requirement, Restated

The required behavior is literal and non-negotiable. When the operator requests 5 recipe shards, the runtime must partition the full recipe workload across 5 recipe shards rather than creating one shard per recipe. If there are 29 recipes, those 29 recipes must be distributed across 5 shards as evenly as practical, and those 5 shards must be assigned to 5 distinct worker sessions when 5 workers are available. The current `29 recipes -> 29 shards -> 5 reused workers` behavior is a bug caused by drift and must be removed.

## Purpose / Big Picture

After this change, when an operator sets `recipe_prompt_target_count=5`, the recipe-correction stage must divide the full recipe work set across 5 recipe shards and run those 5 shards on 5 distinct worker sessions when 5 workers are available. The current runtime does not do that. In the benchmark run at `data/golden/benchmark-vs-golden/2026-03-27_07.07.27`, the user asked for 5 recipe shards but the runtime created 29 one-recipe shards and reused 5 workers to chew through them. After this fix, the same shape of run should produce roughly balanced shard ownership such as `6/6/6/6/5`, the recipe phase manifest should say `shard_count: 5`, the benchmark prompt-budget summary should report `requested_run_count: 5` and `actual_run_count: 5`, and the upload-bundle review should no longer describe recipe correction as oversharded by design.

The user-visible proof is simple. Run a recipe-correction job with more recipes than the prompt target, inspect `raw/llm/<workbook>/recipe_phase_runtime/phase_manifest.json`, `shard_manifest.jsonl`, and `worker_assignments.json`, and confirm that the recipe count is larger than the shard count while the shard count equals the requested target. Then run the focused pytest coverage described below and confirm the benchmark reporting surfaces preserve that same truth.

## Surprises & Discoveries

- Observation: the recipe runtime drift is not in the generic shard-target helper. The helper already does the right thing if `items_per_shard` is unset; the drift comes from the recipe planner passing `items_per_shard=1`.
  Evidence: `cookimport/llm/shard_prompt_targets.py` gives `items_per_shard` priority over `prompt_target_count`, and `cookimport/llm/recipe_stage_shared.py` passes `items_per_shard=1` in both live planning seams.

- Observation: the benchmark artifacts already distinguish worker sessions from shard count, so the reporting layer is mostly telling the truth. The main bug is that the recipe planner creates the wrong number of shards in the first place.
  Evidence: the investigated run reports `workspace_worker_session_count: 5`, `worker_count: 5`, and `actual_run_count: 29` in `codexfarm/prompt_budget_summary.json`.

- Observation: the current recipe worker runtime already supports multi-recipe shards. It writes `task_manifest.jsonl`, worker-local current-task sidecars, and per-task outputs under a parent shard, so this fix should restore an older intended capability rather than invent a new execution model.
  Evidence: `tests/llm/test_recipe_workspace_tools.py` and `docs/10-llm/10-llm_README.md` both describe task files living under a parent shard such as `recipe-shard-0000-r0000-r0001.task-001`.

- Observation: progress reporting already treated shard finalization separately from task progress, so moving to multi-recipe shards only required updating the planned inputs and expectations, not rewriting the progress payload shape.
  Evidence: after the fix, the recipe progress payload reports `task_total: 3`, `followup_total: 2`, and detail lines such as `completed shards: 0/2`.

## Context and Orientation

The recipe-correction runtime lives in `cookimport/llm/recipe_stage_shared.py`. In this repository, a "shard" is one model-owned packet of work and one runtime scheduling unit. A "worker" is one long-lived Codex workspace session that may process one or more shards. A "task" is one recipe-owned answer unit inside a shard. Recipe correction uses a compact single-correction topology: deterministic code builds one `RecipeState` per candidate recipe span, the runtime groups those states into shards, workers emit one validated output per task, and deterministic promotion merges those outputs back into final authoritative recipe payloads.

The relevant live seams are:

- `cookimport/llm/recipe_stage_shared.py`
  This file owns recipe shard planning, runtime manifests, worker assignment, task-manifest generation, and promotion entrypoints. `_build_recipe_shard_plans(...)` creates the live shard plans used by `run_codex_farm_recipe_pipeline(...)`. `_build_single_correction_execution_plan(...)` creates the preview/inspection plan used by reporting and prompt-preview surfaces.
- `cookimport/llm/shard_prompt_targets.py`
  This file contains `resolve_shard_count(...)` and `partition_contiguous_items(...)`. It already knows how to divide `N` items into a requested shard count, but only when the caller does not force `items_per_shard=1`.
- `tests/llm/test_recipe_phase_workers.py`
  This file is the primary behavior regression anchor for recipe shard planning and runtime contracts. It currently contains assertions that encode the drifted one-recipe-per-shard behavior and must be updated.
- `tests/llm/test_recipe_workspace_tools.py`
  This file proves that worker-local task surfaces already support several recipe tasks under one shard. It is the safety net that tells us the runtime can still function once recipe shards become multi-recipe again.
- `tests/labelstudio/test_labelstudio_benchmark_helpers_artifacts.py`
  This file covers `prompt_budget_summary.json` and the benchmark artifact summaries that surfaced the original problem. It should keep proving that requested-vs-actual run counts use real shard truth.
- `docs/02-cli/02-cli_README.md`
  This file already says `recipe_prompt_target_count` means requested shard count on the live runtime path.
- `docs/10-llm/10-llm_README.md`
  This file currently mixes correct prompt-target language with stale wording that says the live recipe path defaults to one candidate-owned shard per recipe. That wording must be repaired so docs match runtime.

The bug is specific and concrete. In `cookimport/llm/recipe_stage_shared.py`, both `_build_recipe_shard_plans(...)` and `_build_single_correction_execution_plan(...)` call `resolve_shard_count(...)` with `items_per_shard=1`. Because `resolve_shard_count(...)` treats `items_per_shard` as a stronger input than `prompt_target_count`, the requested target is effectively ignored whenever recipe count is larger than the target. The result is `N recipes -> N shards`, with workers merely reusing sessions across those shards.

The intended contract after this fix is:

1. `recipe_prompt_target_count` means requested recipe shard count.
2. When recipe count is greater than the target, contiguous recipe states are partitioned across that many shards as evenly as possible.
3. Each shard may own several recipe tasks.
4. `worker_count` controls concurrency and session fan-out only. It does not increase shard count.
5. Reporting must continue to distinguish shard count from task count and worker count.

## Plan of Work

First, repair the recipe shard planner in `cookimport/llm/recipe_stage_shared.py`. In `_build_recipe_shard_plans(...)`, stop forcing `items_per_shard=1` when calling `resolve_shard_count(...)`. The implementation should instead let `recipe_prompt_target_count` drive requested shard count directly, then use `partition_contiguous_items(...)` to split the prepared recipe inputs into that many contiguous groups. Preserve the existing shard-id format, but make the shard range cover the first and last recipe indices in that shard so names such as `recipe-shard-0000-r0000-r0005` remain stable and informative.

Next, make `_build_single_correction_execution_plan(...)` use the same planning logic as the live runtime rather than re-implementing a slightly different shard-count calculation. The simplest safe path is to centralize the shared logic so both the live planner and the execution-plan preview consume the same shard grouping result. The preview output must show `planned_shards` count equal to the requested target and must still emit one `planned_task` per recipe with the correct parent `shard_id`.

Then, audit runtime assumptions that may still quietly expect one task per shard. The main files to inspect while implementing are `cookimport/llm/recipe_stage_shared.py` around `_build_recipe_task_plans(...)`, `_run_direct_recipe_workers_v1(...)`, and any code that derives `task_id`, `parent_shard_id`, current-task sidecars, or proposal filenames. The invariant to preserve is that each task still owns one recipe id even when its parent shard owns several tasks. Proposal assembly must continue to aggregate all tasks under the parent shard into one validated shard proposal.

After the live planner is corrected, update the tests in `tests/llm/test_recipe_phase_workers.py`. Replace the current tests that assert `3 recipes -> 3 shards` when `recipe_prompt_target_count=2` with assertions that expect `3 recipes -> 2 shards`, balanced owned-id lists such as `[2, 1]`, and task manifests that still contain three recipe tasks attached to two parent shard ids. Add or expand a test that also checks `worker_assignments.json` for the case where requested shard count and worker count are both 2, proving that each worker owns one shard. Keep at least one test that exercises the single-recipe case so the trivial path stays covered.

Add or update benchmark/reporting regressions in `tests/labelstudio/test_labelstudio_benchmark_helpers_artifacts.py` so prompt-budget summaries continue to describe actual shard count rather than task rows. The important acceptance here is not a new formatter change; it is a guard that future recipe runtime changes cannot silently regress back to `actual_run_count = recipe count` while `worker_count` remains small. If an existing synthetic manifest fixture is enough, update it. If not, add a focused regression that builds a recipe stage summary with multi-recipe shards and confirms `requested_run_count == actual_run_count == 5` while task count remains larger.

Finally, update docs. In `docs/10-llm/10-llm_README.md`, remove or rewrite the sentence that says the live recipe implementation defaults to one candidate-owned shard per recipe. Replace it with explicit wording that recipe correction now groups contiguous recipe candidates into the requested shard count and keeps one task file per recipe inside each shard. In `docs/02-cli/02-cli_README.md`, keep the existing prompt-target contract wording but add one short clarifying sentence that the recipe count may exceed shard count because multiple recipes can share a shard. If benchmark docs mention oversharding as a normal possibility for recipe correction, update those notes only if they currently describe the drift as intended behavior.

## Idempotence and Recovery

The code changes in this plan are safe to apply incrementally. The recipe planner and its tests can be edited and rerun repeatedly without mutating persistent benchmark data. If a partial implementation leaves tests failing, recover by first restoring consistency between `_build_recipe_shard_plans(...)` and `_build_single_correction_execution_plan(...)`; those two seams must not disagree about shard ownership. If worker-runtime tests fail after the planner is fixed, inspect `task_manifest.jsonl` and `worker_assignments.json` before changing reporting code, because the most likely failure mode is a stale one-task-per-shard assumption in the recipe runtime tests or helper code.

Do not regenerate or rewrite the historical benchmark run under `data/golden/benchmark-vs-golden/2026-03-27_07.07.27` as part of the initial fix. That run is evidence of the bug. Use synthetic test fixtures first. If a follow-up validation run is needed later, create a new timestamped run root rather than mutating the old artifact tree.
