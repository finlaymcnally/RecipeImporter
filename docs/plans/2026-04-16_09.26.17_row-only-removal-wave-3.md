# Continue the Row-Only Removal in Recipe, Nonrecipe, and Freeform Eval Contracts

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

The previous row-only sweep removed the public scorer shims, recipe-boundary aliases, and prelabel block-era internals. The next remaining debt is narrower but still active: recipe worker contracts still talk about divesting and retaining blocks even when the owned evidence is row-native, nonrecipe authority dataclasses still expose block alias constructors and block-named convenience views, and freeform eval still models its canonical ranges as `start_block_index` / `end_block_index`. This wave keeps real source-block provenance, but removes semantic block naming from these active row-native contracts.

## Surprises & Discoveries

- Observation: the recipe surface is already row-owned in practice, but the worker-facing contract still advertises block divestment.
  Evidence: `cookimport/llm/codex_farm_contracts.py` still exposes `divested_block_indices`, `cookimport/llm/recipe_stage/task_file_contract.py` still requires `divested_block_indices`, and `cookimport/llm/recipe_stage_shared.py` still names the prepared owned evidence set `block_indices` even though it comes from the row-native recipe ownership result.

- Observation: nonrecipe authority is already row-native in its true fields, but its dataclasses still keep first-class block alias constructors and block-named convenience properties.
  Evidence: `cookimport/staging/nonrecipe_authority_contract.py` still accepts `block_start_index`, `block_end_index`, `block_indices`, and `block_ids` in `NonRecipeSpan`, and surrounding result types still expose properties like `candidate_block_indices`, `authoritative_block_indices`, and `final_blocks`.

- Observation: freeform eval is using row-native inputs while still grading them through block-era range names.
  Evidence: `cookimport/labelstudio/eval_freeform.py` already prefers `touched_row_indices` when reading gold, but the main dataclasses and summary keys still use `start_block_index`, `end_block_index`, and `_extract_block_indices(...)`.

- Observation: there is a second-tier doc debt around Label Studio helper instructions that still describes block markers and block indices for AI labeling guidance.
  Evidence: `docs/06-label-studio/AI-labelling-instructions.md` still refers to `{{FOCUS_BLOCK_INDICES}}` and block-oriented span marker lines even after the active prelabel prompt surface moved to rows.

- Observation: the nonrecipe writer and knowledge-evidence paths still had real runtime references to deleted block-era properties even after the contract/dataclass rename.
  Evidence: `cookimport/staging/writer.py` still iterated `span.block_indices`/`span.block_ids`, and `cookimport/staging/knowledge_row_evidence.py` still read `candidate_block_indices` / `unresolved_candidate_block_indices`.

- Observation: some taskfile/fake-runner tests were asserting obsolete intermediate artifacts rather than the real post-handoff contract.
  Evidence: `tests/llm/test_fake_codex_farm_cli.py` expected taskfile-mode per-shard outputs under `workers/*/out/*.json`, but the actual durable contract is the validated `proposals/*.json` set after `task-handoff`.

## Decision Log

- Decision: keep transport-level compact aliases only where they are truly needed for stable worker payload compatibility, but rename the Python model surface to rows.
  Rationale: the repo can preserve compact wire aliases like `db` without keeping `divested_block_indices` as the public semantic name in Python.
  Date/Author: 2026-04-16_09.26.17 / Codex

- Decision: treat nonrecipe contract block aliases as semantic debt, not provenance.
  Rationale: `NonRecipeSpan.row_indices` and related fields already hold the true active meaning. The `block_*` constructor args and convenience properties only keep the wrong semantic unit alive for callers.
  Date/Author: 2026-04-16_09.26.17 / Codex

- Decision: freeform eval ranges should be row-native even if task ids and some archive surfaces still encode source-block provenance.
  Rationale: the eval contract is about what semantic rows were labeled and scored. Provenance-bearing ids can remain separate without forcing the eval math itself to speak in block names.
  Date/Author: 2026-04-16_09.26.17 / Codex

## Context and Orientation

Three owner clusters define this wave.

The recipe contract cluster lives in `cookimport/llm/codex_farm_contracts.py`, `cookimport/llm/recipe_stage_shared.py`, `cookimport/llm/recipe_stage/task_file_contract.py`, and nearby recipe workspace helpers. It still speaks about `divested_block_indices`, owned block sets, and recipe authority decisions in block terms even though the actual active ownership path is row-native.

The nonrecipe cluster lives in `cookimport/staging/nonrecipe_authority_contract.py` plus its normal staging consumers in `nonrecipe_authority.py`, `nonrecipe_stage.py`, `pipeline_runtime.py`, prompt preview helpers, and staging tests. It already has row-native data, but the live dataclass/API surface still teaches callers to think in blocks.

The freeform eval cluster lives in `cookimport/labelstudio/eval_freeform.py` plus `docs/06-label-studio/06-label-studio_README.md`, `docs/06-label-studio/AI-labelling-instructions.md`, and the freeform-eval tests. Gold ingestion already prefers row-native touched payloads, so the remaining debt is mainly the main range model, helper naming, and summary wording.

## Validation and Acceptance

Acceptance is behavioral and visible:

Recipe worker/taskfile Python surfaces should read as row-native when opened directly, and worker-facing recipe docs should stop telling the model to divest or retain “blocks” as the semantic unit.

Nonrecipe authority types should expose rows as the normal API. A developer opening the contract layer should not see `block_*` alias constructor args or block-named convenience properties presented as equal first-class contract names.

Freeform eval should compute and report row ranges as rows. The active Label Studio docs/tests touched by this slice should reflect row-native range terminology, with any old block fallback pushed to ingestion only.

Exact passing pytest commands should be recorded back into this plan as milestones complete.

## Idempotence and Recovery

This cleanup is safe in small batches. If a persisted older artifact still arrives with block-era keys, keep the compatibility translation local to the exact loader or answer-ingest seam that needs it. Do not preserve block-named aliases on the live model/dataclass surface just to avoid touching one reader.

If implementation reveals that one cluster is materially larger than expected, split the corresponding bead rather than silently broadening this wave into the knowledge-stage packet/proposal contracts.
