# Replace Codex Prompts With Shard Agents

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, RecipeImport will no longer run the current Codex-backed surfaces as many one-shot structured-output prompts. Canonical line-role labeling, non-recipe knowledge refinement, and recipe correction will all move to shard-agent pipelines that cover the full book with a smaller number of bounded local Codex workers. The new authoritative pipeline ids will be `codex-line-role-shard-v1`, `codex-knowledge-shard-v1`, and `codex-recipe-shard-v1`.

This is a total refactor and a final replacement, not an optional side path. The old Codex prompt-per-bundle surfaces, `codex-line-role-v1`, `codex-farm-knowledge-v1`, and `codex-farm-single-correction-v1`, exist only as migration context while this plan is being implemented. By the end of the work, the repo should default to shard agents for all three tasks, preview should describe shard-agent work rather than old prompt counts, and benchmark artifacts should measure the new runtime directly.

The user-visible outcome is that a human can run a shard-aware preview on an existing processed or benchmark root, then run a live benchmark outside the agent shell and compare it against the current baseline cut. The important proof is not just fewer files under `raw/llm`. It is that all lines, all knowledge regions, and all recipe candidates are still covered, quality remains benchmarkable, and the repo now measures the true frontier in terms of fresh Codex workers, internal turns, tokens, wall time, and benchmark quality.

The shard runtime must also preserve the existing RecipeImport session-isolation seam: every CodexFarm subprocess should still launch `codex exec` under the dedicated RecipeImport Codex home and against a clean shard workspace that contains only the files for that shard. The plan must not introduce a second Codex-home selection mechanism in pipeline JSON or ask operators to manage shell aliases by hand.

## Surprises & Discoveries

- Observation: the current prompt budget is no longer dominated by transport waste. On `prompt_preview_2026-03-16_21.14.22`, line-role overhead is only `4,440` chars total while the run still estimates `~578,563` total tokens.
  Evidence: `data/golden/benchmark-vs-golden/2026-03-16_20.23.25/.../prompt_preview_2026-03-16_21.14.22/prompt_preview_budget_summary.md`.

- Observation: the current CodexFarm runner explicitly audits the active mode as `structured_output_non_agentic`, and flags requested tool affordances as invalid in that mode.
  Evidence: `cookimport/llm/codex_farm_runner.py:_build_runtime_mode_audit(...)` returns `"mode": "structured_output_non_agentic"` and marks `runtime_agentic_flag_present` as a reason code.

- Observation: recipe correction still pays a large repeated fixed payload. The inline `tagging_guide` is about `2,414` chars and appears in every recipe correction input.
  Evidence: `cookimport/llm/recipe_tagging_guide.py`, `cookimport/llm/codex_farm_orchestrator.py`, and the prompt-log inspection for `prompt_preview_2026-03-16_21.14.22`.

- Observation: some recipe prompts are weak targets even before sharding, including several `Variation` and two-row candidates. The shard replacement therefore needs both grouping and better local ownership rules, not just mechanical packing.
  Evidence: the current preview includes `11` recipe prompts with only `2` evidence rows and `15` with `<=3` evidence rows.

- Observation: live Codex-backed benchmark runs are blocked in agent-run shells, and `bench quality-run` is deterministic-only. Final cutover validation therefore requires both offline preview and human-run live benchmark execution outside the agent shell.
  Evidence: `docs/07-bench/07-bench_README.md` and `cookimport/cli.py` reject live benchmark Codex execution in agent environments.

- Observation: RecipeImport already forces CodexFarm subprocesses onto a dedicated RecipeImport Codex home at the runner layer because the local pipeline pack does not self-describe `codex_home_profile`.
  Evidence: `cookimport/llm/RECIPE_CODEX_HOME.md`, `docs/tasks/2026-03-16_21.10.49 - recipeimport-use-codex-recipe-home.md`, and `docs/understandings/2026-03-16_21.10.49-recipeimport-must-force-codex-home-at-runner-layer.md`.

## Outcomes & Retrospective

This section is intentionally incomplete because implementation has not started yet.

The expected end state is a repo where shard agents are the only Codex-backed path for line-role, knowledge, and recipe correction. The old one-shot Codex surfaces should be gone from defaults, active CLI help text, prompt preview expectations, and live stage orchestration. The final retrospective must explain whether one shard strategy worked across all three tasks or whether each stage converged on a different local-ownership shape.

The most important lesson to capture at completion is whether the replacement achieved what the user actually wants: fewer fresh Codex workers with full-book coverage, not merely fewer files or fewer static prompts. If one stage turns out to be a bad fit for long-lived shard agents, that fact needs to be stated clearly here because this plan’s intent is final replacement, not indefinite ambiguity.

## Context and Orientation

The current Codex-backed runtime has three relevant surfaces.

Canonical line-role labeling lives in `cookimport/parsing/canonical_line_roles.py`. Today it deterministically labels atomic lines first, then optionally escalates bounded batches to Codex through `_run_line_role_prompt_via_codex_farm(...)`. The current work unit is a prompt batch, not a long-lived workspace-owning agent. Preview reconstruction for this stage lives in `cookimport/llm/prompt_preview.py:_build_line_role_preview_rows(...)`.

Knowledge refinement lives in `cookimport/llm/codex_farm_knowledge_orchestrator.py` and `cookimport/llm/codex_farm_knowledge_jobs.py`. Today the system deterministically builds compact bundled knowledge jobs, writes bundle JSON inputs, calls Codex once per bundle, then merges per-chunk results back into final block authority. Preview reconstruction for this stage rebuilds those jobs in `cookimport/llm/prompt_preview.py:_build_knowledge_preview_rows(...)`.

Recipe correction lives in `cookimport/llm/codex_farm_orchestrator.py`. Today the deterministic pipeline builds one `MergedRecipeRepairInput` per authoritative recipe span, calls Codex once per recipe span, and deterministically rebuilds the final recipe draft from the corrected output. Preview reconstruction lives in `cookimport/llm/prompt_preview.py:_build_recipe_preview_rows(...)`.

The common transport is `cookimport/llm/codex_farm_runner.py`. Right now that runner assumes one structured-output task per input file and audits the runtime as non-agentic. This is the core seam that must change for the refactor to be real.

That same runner is also the Codex-home isolation seam. `cookimport/llm/RECIPE_CODEX_HOME.md` documents that RecipeImport injects `CODEX_HOME` and `CODEX_FARM_CODEX_HOME_RECIPE` there so CodexFarm uses the dedicated RecipeImport profile by default instead of inheriting the operator's main Codex home. The shard runtime must preserve that behavior. The clean workspace for a shard and the chosen Codex home are separate concerns: shard files live under a shard-local workspace root, while Codex home/profile selection stays centralized in the runner.

Benchmark and preview surfaces are split. `cf-debug preview-prompts` is the current zero-token planner for already-processed runs or benchmark roots. `cookimport labelstudio-benchmark run` is the current single-run benchmark surface that can execute live Codex-backed runs with explicit human approval. `cookimport labelstudio-benchmark compare` compares candidate and baseline benchmark roots. `cookimport bench quality-run` is not the live validation harness because it intentionally refuses Codex-backed quality runs.

A shard agent, in this plan, means one Codex worker that owns one bounded local work region and may take several internal steps before emitting one final strict result file. For line-role, the region is a contiguous atomic-line window. For knowledge, it is a contiguous outside-recipe region composed of surviving chunk groups. For recipe correction, it is a bounded group of neighboring authoritative recipe spans. A shard agent is not a whole-book free-roaming process; it must be local, restartable, and mergeable.

## Milestones

### Milestone 1: Shared shard-agent runtime and cutover config

At the end of this milestone, the repo will have one reusable shard-agent runtime that can describe, execute, and report one bounded Codex worker owning one local shard. The runner must visibly support an agentic shard mode rather than smuggling the same old one-shot JSON tasks through a new name.

Start in `cookimport/config/run_settings.py`, `cookimport/config/run_settings_adapters.py`, `cookimport/cli.py`, and any interactive settings flow that mirrors those values. Add stable shard knobs for all three replacement surfaces: `line_role_shard_target_lines`, `line_role_shard_max_turns`, `knowledge_shard_target_chunks`, `knowledge_shard_max_turns`, `recipe_shard_target_recipes`, and `recipe_shard_max_turns`. These are no longer “experimental” knobs in the final state; they are part of the active Codex-backed runtime surface.

Then add a reusable runtime module, preferably `cookimport/llm/shard_agent_runtime.py`. This module should own the stable types used across stages. It needs a `ShardPlanV1` describing one shard’s id, stage key, local ownership range, item ids, workspace inputs, and turn cap; a `ShardExecutionReportV1` describing completion, fresh-agent count, observed turn count, and any missing item ids; and the helper that executes one shard through CodexFarm. The runner changes belong in `cookimport/llm/codex_farm_runner.py`. Add a distinct runtime audit result such as `structured_loop_agentic_v1` so manifests and telemetry prove the refactor is real. The shard runtime should materialize a clean workspace containing only shard-local task files, but it must not choose Codex home/profile itself; it should rely on the existing runner-owned RecipeImport Codex home injection.

This milestone must also update common reporting. `cookimport/llm/prompt_preview.py`, `cookimport/llm/prompt_budget.py`, and the live stage manifests must all expose `fresh_agent_count`, `shard_count`, planned turn caps, observed turn counts when live telemetry exists, and shard-size distribution.

Acceptance for this milestone is a fake or synthetic test proving that one shard runs through the new runtime, the runner audits it as agentic shard mode rather than non-agentic structured-output mode, and the manifest records shard-specific telemetry.

### Milestone 2: Replace line-role Codex execution with `codex-line-role-shard-v1`

At the end of this milestone, `codex-line-role-shard-v1` is the authoritative Codex-backed line-role surface. The old `codex-line-role-v1` path may exist temporarily behind the scenes for cutover validation, but it must no longer be the active target pipeline id.

The primary work is in `cookimport/parsing/canonical_line_roles.py`. Today the Codex branch batches many ordered line candidates and asks for one structured response per batch. Replace that execution unit with a shard planner keyed by contiguous atomic-line ownership. Deterministic labels and escalation reasons should still be computed first; they become shard-local hints that the shard agent can refine. The safest first ownership model is one non-overlapping contiguous window per shard and deterministic ownership by `atomic_index`.

Add a line-role shard prompt/task asset under `llm_pipelines/` that tells the agent it owns one local line window and must emit one normalized result row per covered `atomic_index`. The output schema must stay strict and local. The canonical projection writer and evaluator must not need a new scoring contract just because the work unit changed.

Update `cookimport/llm/prompt_preview.py` so preview for line-role now plans shards rather than old prompt batches whenever the active pipeline id is `codex-line-role-shard-v1`. The preview summary should report shard counts, shard sizes, first-turn task chars, and cost bounds derived from turn caps.

Acceptance is that canonical-text scoring can run against line-role outputs produced by `codex-line-role-shard-v1`, and `tests/parsing/test_canonical_line_roles.py` plus `tests/llm/test_prompt_preview.py` prove that all `atomic_index` values are covered exactly once with fewer fresh workers than the old batch model on the motivating run.

### Milestone 3: Replace knowledge Codex execution with `codex-knowledge-shard-v1`

At the end of this milestone, `codex-knowledge-shard-v1` is the authoritative Codex-backed knowledge surface. The old `codex-farm-knowledge-v1` job-per-bundle path should no longer be the active Codex knowledge runtime after cutover.

Most deterministic preparation can remain. `cookimport/llm/codex_farm_knowledge_jobs.py` already decides which chunks survive pruning and how local chunk groups are formed. Reuse that logic as the seed for shard ownership instead of discarding it. The replacement happens at the execution unit: instead of one Codex response per compact bundle, the stage plans a smaller number of local shard agents, each owning a contiguous outside-recipe region and emitting per-`chunk_id` final decisions that still flatten back into the current merge contract in `cookimport/llm/codex_farm_knowledge_ingest.py` and `cookimport/llm/codex_farm_knowledge_orchestrator.py`.

The new shard prompt and schema should preserve the current evidence discipline. The agent must return one result per `chunk_id`, must keep block-level ownership grounded in the shard’s covered blocks, and must continue to produce reviewer-facing snippets compatible with `knowledge.md` and `snippets.jsonl`.

Preview and live manifests for this stage must distinguish surviving chunks, planned shard agents, observed turns, and final token totals. The old `jobs_written` field alone is no longer enough once one shard can loop internally. Keep backward-compatible metrics only as migration breadcrumbs; the final active reporting surface should speak in shard terms.

Acceptance is that the knowledge stage still writes `08_nonrecipe_spans.json`, `09_knowledge_outputs.json`, and snippets, but the live and preview manifests now describe shard agents as the only Codex knowledge work unit. `tests/llm/test_knowledge_job_bundles.py`, `tests/llm/test_knowledge_output_ingest.py`, and `tests/llm/test_codex_farm_knowledge_orchestrator.py` must prove that all planned `chunk_id`s are covered exactly once after the replacement.

### Milestone 4: Replace recipe Codex execution with `codex-recipe-shard-v1`

At the end of this milestone, `codex-recipe-shard-v1` is the authoritative Codex-backed recipe correction surface. The old `codex-farm-single-correction-v1` one-recipe-per-call path should no longer be the active recipe Codex runtime.

This stage needs the most care because the current contract is “one authoritative recipe span in, one corrected recipe out.” The replacement must preserve that output shape while changing only the execution unit. The deterministic builder in `cookimport/llm/codex_farm_orchestrator.py` should continue to create one span-local correction input per `recipe_id`, but a new grouping layer must pack nearby inputs into shard plans keyed by local recipe order and bounded by recipe-count and char-count caps. The shard prompt must require one final result object per owned `recipe_id`.

Use the replacement to amortize fixed overhead. The repeated tagging guide and global correction instructions should move out of each per-recipe payload and into the shard-level task framing wherever possible. Span-local payloads should only carry the evidence and deterministic hints needed for that recipe. This is the cleanest place to convert repeated per-call overhead into shared shard context.

Because recipe quality is sensitive, the initial grouping rule should be conservative: group only neighboring authoritative recipe spans that are already adjacent in recipe order and stay under explicit caps. Do not let one giant shard span a whole chapter in the first cutover.

Acceptance is that staged outputs and linkage artifacts stay identical in schema to the current recipe correction outputs, `tests/llm/test_prompt_preview.py` shows shard-agent recipe planning instead of `36` one-shot recipe prompts on the motivating run, and recipe regression tests still prove one result per `recipe_id` with no drops or duplicates.

### Milestone 5: Preview cutover, benchmark validation, and old-path removal

At the end of this milestone, preview and live validation both speak the new shard language, and the old one-shot Codex pipeline ids are removed from active choices and defaults.

First, replace the preview model. `cookimport/cf_debug_cli.py` should gain a shard-aware sweep command such as `cf-debug preview-shard-sweep`. It should take an existing processed run or benchmark root plus a small candidate-settings file and write one summary JSON and Markdown report under a timestamped output folder. That report must include, per candidate, `fresh_agent_count`, `shard_count`, shard-size distribution, first-turn prompt/task chars, and conservative cost bounds. `cf-debug preview-prompts` should also be updated so that its Codex-backed planning model now reflects shard agents rather than old prompt-per-bundle counts.

Second, wire `cookimport labelstudio-benchmark run` and `cookimport labelstudio-benchmark compare` to the new pipeline ids and shard knobs so a human can run the live cutover candidate outside the agent shell and compare it against the current baseline run. `bench quality-run` remains out of scope because it is deterministic-only.

Third, remove old active pipeline choices. Update `cookimport/config/run_settings.py`, CLI option validation, prompt preview, docs, and any stage-specific selector logic so the old prompt-per-bundle pipeline ids are no longer advertised as active Codex-backed choices. If temporary compatibility aliases remain for old saved settings, they must normalize directly to the new shard-v1 ids and log that normalization explicitly.

Acceptance is that a human can run shard-aware preview, run one live benchmark with the three shard-v1 pipelines, compare it against the current baseline, and see no active CLI path that still frames the old one-shot Codex surfaces as first-class options.

## Validation and Acceptance

Acceptance is behavioral and must be proven at three levels.

The first level is unit and orchestration behavior. New tests must prove that each shard planner covers the full intended worklist exactly once, that shard ids and shard ownership are stable across reruns, and that each stage’s post-merge artifacts preserve the same schema and ownership semantics as the current outputs. A line-role shard run must still yield one decision per `atomic_index`. A knowledge shard run must still yield one decision set per `chunk_id` and one final authority decision per covered block. A recipe shard run must still yield one corrected result per authoritative `recipe_id`.

The runner and workspace seam also needs explicit proof. Focused transport/runtime tests must show that shard execution still launches CodexFarm with the dedicated RecipeImport Codex home by default and that the shard workspace contains only shard-local files unless an explicit override is provided.

The second level is preview behavior. `cf-debug preview-prompts` and `cf-debug preview-shard-sweep` must both describe the new shard runtime rather than old prompt-per-bundle counts. The preview summaries must clearly label cost numbers as conservative bounds or first-turn estimates whenever the runtime may loop internally.

The third level is live benchmark behavior. A human must be able to run at least one shard-v1 candidate outside the agent shell and compare it against the current baseline with `cookimport labelstudio-benchmark compare`. The replacement is accepted only if the compare report, live manifests, and prompt/telemetry artifacts together make the new runtime legible: what changed in quality, what changed in total tokens, what changed in wall time, how many fresh agents were summoned, and how many internal turns they took.

## Idempotence and Recovery

All preview sweeps and focused tests must be safe to rerun. Shard planners should derive deterministic shard ids from stable input order so rerunning preview or a fake runner yields identical shard ownership unless the configured shard knobs change.

Temporary coexistence with old pipeline ids is permitted only during implementation. The final code should normalize or remove those ids. If an old saved setting must remain readable for a while, normalize it immediately to the corresponding shard-v1 id and emit a manifest note that the setting was upgraded.

If a live shard agent partially fails, recovery should mirror current recoverable CodexFarm behavior: keep completed shard outputs, mark only missing shard ids as failed, and allow retry of the missing shard subset. The manifest must record which shard ids were missing so a human can retry or inspect them without rerunning the whole book.
