# Repair Pass2/Pass3 Benchmark Reliability

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the compact CodexFarm recipe pipeline should stop failing for the wrong reasons on the exact benchmark family that triggered the March 13 investigation. In practical terms, the benchmark rooted at `data/golden/benchmark-vs-golden/2026-03-06_15.22.11` should no longer:

- split one recipe into an ingredient-only pass2 row and an instruction-only pass2 row because two pass1 candidates overlapped,
- reject an otherwise useful pass2 extraction only because auxiliary `field_evidence` text is malformed,
- pass corrupted `extracted_ingredients` or `extracted_instructions` text into pass3 unchanged, or
- waste live pass3 calls on rows that are already structurally broken before pass3 starts.

The user-visible proof is an exact rerun on `saltfatacidheatcutdown` that fixes the known bad families from the March 6 session: `c10` should not die on `field_evidence`, the `Rice Wine Vinaigrette` split should not recur as `c15`/`c16`, `c1` should not carry `jalape\u0014no` forward, and pass3 should stop seeing partial recipe bundles that it cannot plausibly repair.

## Surprises & Discoveries

- Observation: the compact pass2 and pass3 prompt templates are not the main problem. They are short and direct.
  Evidence: `llm_pipelines/prompts/recipe.schemaorg.compact.v1.prompt.md` and `llm_pipelines/prompts/recipe.final.compact.v1.prompt.md` are both simple single-message prompts with no hidden system or developer prompt layers in the March 6 prompt logs.

- Observation: the March 6 `saltfatacidheatcutdown` run made only two live pass3 calls; most rows never exercised the pass3 prompt.
  Evidence: `prediction-run/manifest.json` records `pass3_inputs = 2` and `pass3_pass2_ok_deterministic_skips = 19`.

- Observation: the ugliest pass2/pass3 artifacts came from pass1 overlap handling, not from the models misunderstanding a good bundle.
  Evidence: `llm_manifest.json` shows `c15` and `c16` both had raw pass1 span `1374..1385`, then `_apply_pass1_midpoint_clamps(...)` split them into `1374..1379` and `1380..1385`, producing one ingredients-only bundle and one instructions-only bundle.

- Observation: `field_evidence` is auxiliary metadata, but the current contract lets it fail the whole pass2 row.
  Evidence: `c10` failed pass2 validation because `field_evidence` contained `13010?, WAIT` even though the core recipe payload in `schemaorg_recipe`, `extracted_ingredients`, and `extracted_instructions` was otherwise plausible.

- Observation: pass2 can preserve clean text in `schemaorg_recipe` while corrupting the mirrored extractive arrays.
  Evidence: `c1` keeps `jalape\u00f1o` inside `schemaorg_recipe` but writes `jalape\u0014no` into `extracted_ingredients` and `extracted_instructions`; the evidence normalization artifact for the same recipe is clean.

- Observation: compact pass3 input assembly is intentionally dumb. It copies pass2 `extracted_*` lists and strips `recipeIngredient` and `recipeInstructions` from `schemaorg_recipe`; it does not attempt any repair or re-grounding.
  Evidence: `cookimport/llm/codex_farm_orchestrator.py::_build_pass3_input_compact(...)`.

## Decision Log

- Decision: treat this as a pipeline reliability repair, not a wholesale prompt rewrite.
  Rationale: the prompt text is mostly fine. The benchmark failures came from bad span windows, fragile auxiliary validation, and un-sanitized pass2 output fields. Fixing only the wording would leave the benchmark broken for the same reasons.
  Date/Author: 2026-03-13 / Codex

- Decision: fix the pass1 overlap split before tuning pass3 routing.
  Rationale: once the orchestrator midpoint-splits one true recipe into two half-recipes, no prompt-level pass2 or pass3 improvement can make both halves fully correct. The first repair must stop manufacturing those broken inputs.
  Date/Author: 2026-03-13 / Codex

- Decision: downgrade malformed `field_evidence` from a hard-fail trigger to a recoverable auxiliary issue.
  Rationale: `field_evidence` is helpful provenance but not the recipe itself. A bad `field_evidence` string should not invalidate an otherwise good pass2 extraction in benchmark or stage flows.
  Date/Author: 2026-03-13 / Codex

- Decision: keep the first validation loop anchored on the exact March 6 Salt/Fat/Acid/Heat benchmark root before broadening to other books.
  Rationale: the benchmark artifact already contains known bad recipe ids and exact expectations. That gives a much sharper acceptance bar than a generic “quality feels better” rerun.
  Date/Author: 2026-03-13 / Codex

## Context and Orientation

The failing benchmark evidence lives under `data/golden/benchmark-vs-golden/2026-03-06_15.22.11/single-profile-benchmark/02_saltfatacidheatcutdown/codexfarm/`.

The most important artifact files are:

- `prediction-run/manifest.json`, which records the high-level counts and the single hard pass2 failure for `c10`,
- `prediction-run/raw/llm/saltfatacidheatcutdown/llm_manifest.json`, which contains per-recipe pass1, pass2, pass3, transport, and degradation details,
- `prediction-run/raw/llm/saltfatacidheatcutdown/pass1_chunking/in|out/*.json`, which show the raw pass1 candidates and their chosen boundaries,
- `prediction-run/raw/llm/saltfatacidheatcutdown/pass2_schemaorg/in|out/*.json`, which show exactly what pass2 received and wrote,
- `prediction-run/raw/llm/saltfatacidheatcutdown/pass3_final/in|out/*.json`, which show the rare live pass3 inputs and outputs,
- `prompts/full_prompt_log.jsonl`, which shows the literal prompt text and raw responses.

The repair touches three code surfaces.

The first surface is pass1-to-pass2 window selection inside `cookimport/llm/codex_farm_orchestrator.py`. The key functions are `_apply_pass1_midpoint_clamps(...)`, which currently splits overlapping pass1 windows at the numeric midpoint, and the manifest/report-building code that records span loss metrics and routing diagnostics. This is where the March 6 `Rice Wine Vinaigrette` duplicate span turned into two half-recipes.

The second surface is pass2 contract loading in `cookimport/llm/codex_farm_contracts.py`. `Pass2SchemaOrgOutput` currently requires `field_evidence` to parse as a JSON object and fails the whole row if it cannot. The same module already has bounded JSON cleanup helpers such as `_normalize_json_text(...)`, `_repair_json_structure(...)`, and `_extract_first_json_object(...)`, but those helpers only repair object fields, not the string lists in `extracted_ingredients` and `extracted_instructions`.

The third surface is compact pass2/pass3 prompt and routing logic. The compact pass2 prompt in `llm_pipelines/prompts/recipe.schemaorg.compact.v1.prompt.md` tells the model to emit `field_evidence` as a JSON-escaped string and does not clearly distinguish actual instructions from section headers and usage notes. The compact pass3 prompt in `llm_pipelines/prompts/recipe.final.compact.v1.prompt.md` assumes pass2 handed it a coherent recipe bundle. The orchestrator routing helpers `_build_pass3_utility_signal_for_pass2_ok(...)` and `_should_run_pass3_llm(...)` decide whether pass3 should run live or deterministic; today they can still send a structurally incomplete pass2 row to live pass3.

The exact benchmark examples that must be fixed are:

- `c10`: valid-looking recipe, invalid `field_evidence`, hard pass2 failure.
- `c15`: title/yield/ingredients only after clamp, hard pass2 degradation `missing_instructions`.
- `c16`: instructions/notes/variation only after clamp, pass2 “ok” but no title or ingredients, then live pass3 with near-empty metadata.
- `c1`: control-character corruption in extractive arrays while the embedded `schemaorg_recipe` string remains clean.

## Plan of Work

Start by repairing overlap handling in `cookimport/llm/codex_farm_orchestrator.py`. The current midpoint clamp is too blunt. It treats all overlaps as generic geometry, but the March 6 failure was not “two different recipes overlap.” It was “two recipe ids claimed the same recipe span.” Add a new arbitration step before `_apply_pass1_midpoint_clamps(...)` that compares adjacent `pass1_status == "ok"` states using the raw pass1 span, the pass1 title, and the local heuristic span. When two adjacent states have the same or nearly the same raw span and the same title anchor, keep the better-aligned candidate and mark the duplicate as dropped or merged instead of splitting the span at the midpoint. For true partial overlaps that are not duplicates, keep clamping, but bias the split around title/yield/ingredient/instruction anchors rather than a pure numeric midpoint. Record the chosen action in the per-recipe manifest row so a future follow-up packet can explain why a state was preserved, merged, or clamped.

After overlap arbitration exists, add explicit pass1 completeness diagnostics. Extend the manifest payload so rows can say whether the final pass2 window still contains title/yield context, ingredient evidence, and instruction evidence after arbitration or clamp. This is not just for debugging. The same completeness signal will be reused in later routing decisions. The goal is to distinguish “complete enough recipe bundle” from “truncated head,” “truncated tail,” and “middle-only fragment.”

Next, harden pass2 contract loading in `cookimport/llm/codex_farm_contracts.py`. Keep `schemaorg_recipe` strict because it is the main structured output. Change `field_evidence` handling so malformed strings degrade to `{}` with a warning or degradation reason instead of failing the whole pass2 row. The contract loader should still attempt existing cleanup and first-object extraction first. Only when those repairs fail should it replace the field with `{}`. Add a post-parse sanitation helper for `extracted_ingredients` and `extracted_instructions` that strips or normalizes control bytes using the same translation policy already used by `_normalize_json_text(...)`. The sanitation must preserve ordinary Unicode like `ñ`; the acceptance case is that `jalapeño` remains intact and `jalape\u0014no` disappears.

Then tighten compact pass2 semantics in both prompt and orchestrator. Update `llm_pipelines/prompts/recipe.schemaorg.compact.v1.prompt.md` so `field_evidence` may be emitted as either a JSON object or an object serialized as a string, with the object form preferred. Also clarify that `extracted_instructions` should contain actual recipe-making steps, not bare section headers like `Variations` or pure usage notes like `Ideal for ...`. Do not attempt a full schema redesign here. The purpose of the prompt edit is to remove the brittle double-encoding instruction and reduce obedient copying of obvious non-step lines.

Back in `cookimport/llm/codex_farm_orchestrator.py`, add a pass2 completeness/degradation check after successful `Pass2SchemaOrgOutput` loading. A row should no longer count as clean `pass2 ok` when it has instructions but no title and no ingredient evidence, or when it has title/yield/ingredients but no instructions and the pass1 completeness signal already says the window lost the tail. Use a new degradation reason family such as `truncated_recipe_head`, `truncated_recipe_tail`, or `partial_recipe_window` rather than overloading everything into `missing_instructions`. These reasons should keep the per-recipe history honest and prevent live pass3 from seeing a bundle that only looks “ok” because the schema parser accepted it.

Finally, tighten pass3 routing and its prompt contract. Update `_build_pass3_utility_signal_for_pass2_ok(...)` and `_should_run_pass3_llm(...)` so live pass3 does not run when the pass2 completeness signal says the recipe window is partial. Those rows should route to deterministic fallback or remain degraded, not escalate to a pass3 LLM call that has no title and no ingredient evidence to work from. Update `llm_pipelines/prompts/recipe.final.compact.v1.prompt.md` so it explicitly refuses to synthesize a title from `workbook_slug` or other fallback text when `recipe_metadata.name` is missing. Compact pass3 should either preserve a real title from metadata or omit it with a warning. The benchmark acceptance case is that the March 6 `c16` family never reaches pass3 in its broken form; if a similar row does reach pass3 later, it must not fabricate the workbook slug as the recipe title.

Documentation changes must stay short but are required. Update `docs/10-llm/10-llm_README.md` when the pass2/pass3 contract or degradation behavior changes, and update `docs/07-bench/07-bench_README.md` if `llm_manifest.json` gains new per-recipe completeness or overlap-arbitration fields. Keep those notes short and behavior-focused.

## Milestones

### Milestone 1: Stop midpoint-splitting duplicate pass1 recipe spans

At the end of this milestone, the orchestrator should not turn one true recipe into two half-recipes just because two adjacent pass1 candidates produced the same raw span. Duplicate or near-duplicate pass1 windows should be merged or one should be dropped before pass2 bundle writing.

Edit `cookimport/llm/codex_farm_orchestrator.py` in the pass1 post-processing section around overlap clamp and manifest row generation. Add focused tests in `tests/llm/test_codex_farm_orchestrator.py` for:

- identical adjacent raw spans with the same title,
- nested or near-identical spans with one better title alignment,
- true overlaps that still need clamp but should preserve title-forward evidence.

Acceptance is that a synthetic `c15`/`c16`-style case no longer yields an ingredients-only bundle plus an instructions-only bundle after pass1 post-processing.

### Milestone 2: Make pass2 resilient to auxiliary-formatting failure and corrupted extractive text

At the end of this milestone, malformed `field_evidence` should no longer invalidate an otherwise good pass2 row, and control characters in extractive arrays should be repaired before pass3 sees them.

Edit `cookimport/llm/codex_farm_contracts.py` and add focused tests in `tests/llm/test_codex_farm_contracts.py` for:

- `field_evidence` object form,
- `field_evidence` string form,
- malformed `field_evidence` that degrades to `{}` instead of raising,
- control-byte cleanup in `extracted_ingredients` and `extracted_instructions`,
- preservation of ordinary non-ASCII characters like `ñ`.

Acceptance is that a `c10`-style row loads successfully with degraded `field_evidence`, and a `c1`-style row produces clean extractive text.

### Milestone 3: Tighten pass2/pass3 semantics for partial bundles

At the end of this milestone, a pass2 row that obviously lost the recipe head or tail should be marked degraded and should not route to live pass3 as if it were a normal pass2 success. The compact prompts should also stop encouraging the specific failure patterns seen in March 6 artifacts.

Edit:

- `cookimport/llm/codex_farm_orchestrator.py`
- `llm_pipelines/prompts/recipe.schemaorg.compact.v1.prompt.md`
- `llm_pipelines/prompts/recipe.final.compact.v1.prompt.md`

Add or extend tests in `tests/llm/test_codex_farm_orchestrator.py` for:

- partial-window degradation reasons,
- pass3 skip behavior for partial pass2 rows,
- preservation of deterministic promotion for healthy pass2 rows,
- no workbook-slug title synthesis in compact pass3 handling.

Acceptance is that a `c16`-style row is degraded before live pass3 or deterministically handled without a fake slug title.

### Milestone 4: Prove the repair on the original benchmark slice

At the end of this milestone, the exact `saltfatacidheatcutdown` benchmark rerun should show the four known failure families fixed or materially reduced.

Acceptance is not “tests passed.” It is that the rerun artifacts show:

- no hard pass2 failure on the `c10`-style auxiliary `field_evidence` issue,
- no duplicated split of `Rice Wine Vinaigrette` into head-only and tail-only pass2 bundles,
- no `jalape\u0014no`-style corruption in pass2 output arrays,
- no live pass3 call on a row that has no title and no ingredient evidence.

If this one-book rerun is clean, repeat the same checks on `seaandsmokecutdown` and `thefoodlabcutdown` to make sure the arbitration and degradation rules are not overfit to one local pattern.

## Validation and Acceptance

Acceptance is behavior-first and tied to the exact failure families already observed.

First, a `c10`-style malformed `field_evidence` row must not appear in the rerun as a hard pass2 validation failure. If `field_evidence` is still bad, the row should survive with degraded provenance, not die.

Second, the rerun must not contain the `Rice Wine Vinaigrette` split into one title/yield/ingredients-only pass2 bundle and one instructions-only pass2 bundle. The per-recipe manifest rows should show either one kept recipe or an explicit overlap-merge/drop decision, not a midpoint split of identical raw spans.

Third, the rerun must not contain control-byte corruption in pass2 extractive arrays. Searching the pass2 output directory for `\u0014` or raw control-byte matches should return zero results.

Fourth, live pass3 should not run on a pass2 row that lacks both title and ingredient evidence. If a partial row survives to pass2, it must be degraded and routed away from live pass3.

Fifth, the repair must preserve healthy pass2/pass3 behavior. The rerun should still produce pass2 outputs and deterministic pass3 promotions for normal recipes; the fix is not accepted if it merely suppresses CodexFarm recipe rows wholesale.

## Idempotence and Recovery

Do not modify or delete the March 6 benchmark root. It is the fixed baseline for this plan. Every rerun must land under a fresh timestamped output root.

The code changes are retry-safe if kept additive. Tests can be rerun repeatedly. Benchmark reruns can also be repeated safely because they write new roots.

If a prompt-contract change causes older cached outputs or fixtures to stop loading, recover by making the Python contract accept both the old and new forms during the migration window. Do not “fix” that by deleting old artifacts or weakening the core recipe payload validation.
