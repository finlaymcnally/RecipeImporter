# Pave a Local Tool Path for Recipe Workspace Workers

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a recipe correction worker should not need to rediscover the same local mechanics every benchmark run. A fresh worker should enter its bounded workspace, open one obvious repo-written task row, use one obvious repo-written helper CLI, and produce valid compact task outputs without improvising brittle heredocs, queue digests, or one-off schema scaffolds.

The user-visible success condition is that a generated recipe worker root is obviously usable by inspection. A novice should be able to open the workspace and see the approved loop immediately: open `worker_manifest.json`, open `current_task.json`, consult `OUTPUT_CONTRACT.md`, use `tools/recipe_worker.py` to orient or scaffold, validate locally, and install to `out/<task_id>.json`. A later benchmark rerun should show fewer local commands and no watchdog kills caused by output-writing improvisation, but this plan does not require a live rerun until the user explicitly approves token spend.

## Surprises & Discoveries

- Observation: the shared workspace runner already solved part of the original orientation problem.
  Evidence: `cookimport/llm/codex_exec_runner.py` now writes `current_task.json` for non-leased worker sessions, which makes the older idea of a recipe-only `TASK_INDEX.md` less important than it first appeared.

- Observation: recipe still benefits from a helper CLI even though its output contract is already documented.
  Evidence: the saved run shows repeated reads of `worker_manifest.json`, `OUTPUT_CONTRACT.md`, `assigned_tasks.json`, `assigned_shards.json`, `hints/*.md`, `in/*.json`, and every example file before one worker died on a brittle multi-output heredoc.

- Observation: recipe is the best candidate for an `install` step as a first-class command.
  Evidence: the compact recipe contract is strict about exact keys, exact `sid`, and exact owned `rid` coverage, so a helper that validates and writes the final file in one approved step is materially safer than freehand `cat > out/...`.

- Observation: the original plan overfit to custom task-index artifacts.
  Evidence: `current_task.json` plus `assigned_tasks.json` plus a helper `overview` command gives the worker the same information without a second inventory format that only recipe understands.

- Observation: the shared direct-exec runner was already generic enough to advertise stage-local helper verbs once a tool file existed under `tools/`.
  Evidence: `cookimport/llm/codex_exec_runner.py` now switches its worker-manifest shell examples to `tools/recipe_worker.py`, `tools/line_role_worker.py`, or `tools/knowledge_worker.py` based on the mirrored tool file set, so recipe could plug into an existing shared seam instead of needing recipe-only manifest plumbing.

## Context and Orientation

The recipe correction worker is the long-lived direct-exec LLM session that reads one or more task-local recipe packets and writes one compact result JSON file per task. The recipe worker lives under `raw/llm/<workbook_slug>/recipe_phase_runtime/workers/worker-*/`.

The compact recipe worker contract is already strict and already documented in worker-local files. Each output file is one JSON object with top-level compact keys `v`, `sid`, and `r`. Each recipe row under `r` uses compact keys such as `rid`, `st`, `sr`, `cr`, `m`, `mr`, `g`, and `w`. The helper must preserve that exact contract.

The shared direct-exec workspace contract lives in `cookimport/llm/codex_exec_runner.py`. For non-leased sessions the worker already gets:

- `worker_manifest.json`
- `current_task.json`
- `assigned_tasks.json`
- `assigned_shards.json`
- `OUTPUT_CONTRACT.md`
- `examples/`
- `hints/`, `in/`, `out/`, `scratch/`

This plan adds `tools/recipe_worker.py` and makes that helper the preferred way to work inside the recipe root.

The key files are:

- `cookimport/llm/codex_farm_orchestrator.py`, which builds recipe worker prompts, worker roots, hints, and contract sidecars
- `cookimport/llm/codex_farm_contracts.py`, which defines the compact recipe contract and normalization logic
- `cookimport/llm/codex_exec_runner.py`, which owns the shared workspace manifest, mirroring, and path policy
- `tests/llm/test_recipe_phase_workers.py`, which should prove the worker root is discoverable and the prompt points at the helper path
- `tests/llm/test_codex_exec_runner_workspace.py`, which should prove the shared runner mirrors and advertises the helper surface correctly

“Scaffold” in this plan means “emit a schema-correct draft with exact task-local `sid` and owned `rid` values already filled in.” “Install” means “validate a saved draft against the compact recipe invariants and then write it to `out/<sid>.json`.”

## Plan of Work

Implement the recipe paved road as one deterministic helper module plus one shared runner improvement.

First, add a new module under `cookimport/llm/` for deterministic recipe workspace tooling. A good name is `cookimport/llm/recipe_workspace_tools.py`. It should expose pure functions for:

- loading and summarizing task rows from `assigned_tasks.json` and `current_task.json`
- scaffolding a compact recipe output draft with exact `sid` and exact owned `rid` values
- validating a saved draft against compact recipe invariants
- installing a validated draft to the task row’s declared `metadata.result_path`
- rendering a self-contained `tools/recipe_worker.py` script body

Second, extend `cookimport/llm/codex_farm_orchestrator.py` so each recipe worker root writes:

- `tools/recipe_worker.py`
- the existing `OUTPUT_CONTRACT.md`
- the existing `examples/*.json`

The helper script must be runnable as `python3 tools/recipe_worker.py ...` from the worker root and must not depend on repository-relative imports at runtime.

Third, use the same helper verbs as the other two stages:

- `overview` prints the ordered queue concisely
- `show <task_id>` prints task-local paths, owned recipe ids, and compact task summary
- `scaffold <task_id> [--dest <path>]` emits a compact draft with exact ids and correct keys
- `check <json_path>` validates a saved draft without writing to `out/`
- `install <json_path>` reruns validation and writes the file to the declared `metadata.result_path`

Fourth, update the recipe worker prompt so the local loop is:

1. open `worker_manifest.json`
2. open `current_task.json`
3. if more context is needed, run `python3 tools/recipe_worker.py overview` or `show <task_id>`
4. open the named `hints/<task_id>.md` and `in/<task_id>.json`
5. if useful, run `python3 tools/recipe_worker.py scaffold <task_id> --dest scratch/<task_id>.json`
6. edit the draft
7. run `python3 tools/recipe_worker.py check scratch/<task_id>.json`
8. run `python3 tools/recipe_worker.py install scratch/<task_id>.json`

The prompt must explicitly say that large batch heredocs are unnecessary because the repo already provides a safe install path.

Fifth, extend `cookimport/llm/codex_exec_runner.py` so `tools/` is mirrored and advertised in `worker_manifest.json` through `tools_dir` and `mirrored_tool_files`. That shared runner change is part of the cross-stage unification and should be done once for all three stages.

## Validation and Acceptance

This change is accepted when all of the following are true.

The generated recipe worker root visibly contains `worker_manifest.json`, `current_task.json`, `OUTPUT_CONTRACT.md`, `examples/`, and `tools/recipe_worker.py`, and the worker manifest explicitly advertises the tools directory and mirrored tool files.

The recipe worker prompt tells a novice to start from `current_task.json`, not from a broad task inventory dump, and gives the helper workflow in the shared verb language `overview`, `show`, `scaffold`, `check`, `install`.

The scaffold helper emits a compact draft with the exact task-local `sid`, exact owned `rid` values, and compact keys only.

The check helper rejects legacy keys, wrong owned-id coverage, or a mismatched `sid`, and accepts a structurally valid compact draft.

The install helper writes exactly one output file to the declared `metadata.result_path` and refuses to install a failing draft.

The narrow `.venv` tests listed above pass.

If the user later approves a live benchmark rerun, acceptance also includes a behavioral check: compared with the saved `2026-03-22_01.34.12` recipe run, recipe validation quality must remain at least as good, `watchdog_killed_shard_count` should reach `0`, and command count and command-execution tokens should fall materially below the baseline.

## Idempotence and Recovery

This plan is intentionally additive. Recipe worker roots are rebuilt from scratch, so regenerating `OUTPUT_CONTRACT.md`, `examples/`, and `tools/recipe_worker.py` is safe to repeat. The helper’s `scaffold` command should write to stdout by default or to an explicit destination only. The helper’s `check` command must be read-only. The helper’s `install` command must be the only step that writes to the final declared output path.

If a partial implementation leaves the prompt referring to helper commands that are not yet mirrored into the workspace, stop and make the workspace surface consistent before running tests. If `install` proves too strict at first, loosen only helper ergonomics while preserving the hard invariants: exact compact keys, exact `sid`, and exact owned `rid` coverage.
