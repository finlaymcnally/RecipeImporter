# Make Knowledge Phase Runtime The Only Runtime

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## User Intent

The user wants the long-term design, not a stopgap. The repository should not keep two competing knowledge runtimes alive. The final system should fully embrace the shard-owned, phase-first worker model that was introduced for knowledge, finish the missing safety work, and then delete the older packet/task runtime instead of carrying it forward as protection for the previous design.

Restated plainly: one knowledge shard should be the worker’s real ownership unit, one long-lived worker session should stay on that shard through Pass 1 and Pass 2, repo code should validate and orchestrate the handoff tightly inside that same session, and any retry or recovery behavior should still speak the shard-owned phase contract instead of reviving packet-era prompts. The product should teach one story only.

Burn-the-boats rule for this plan: this repository has one human owner and one real product path. Old code exists because it was worse, not because it deserves preservation. This plan must therefore prefer deletion over preservation, replacement over adaptation, and one honest runtime over transitional dual support. If an old helper, field name, artifact, or code path only exists to preserve the previous design, the default action is to remove it.

## Purpose / Big Picture

After this change, the knowledge stage will have one honest runtime. A user who sets `knowledge_prompt_target_count` to `N` will get roughly `N` shard-owned knowledge ledgers, each worker will operate through `CURRENT_PHASE.md` plus shard-owned `in/work/repair/out` files, and repo code will provide the same resilience the old runtime had without falling back to the old packet/task architecture.

The visible proof is straightforward. New worker roots will expose only shard-owned phase artifacts. New runtime summaries and stage outputs will stop teaching packet/task/snippet vocabulary. When a shard fails, the recovery path will be either a same-contract shard rerun or an explicit shard split retry, never a hidden return to packet-first repair prompts. At the end of the implementation there will be one runtime path, one artifact vocabulary, and one mental model for a newcomer to follow.

Just as important, the repository will stop carrying old design baggage “just in case.” The desired end state is not “new runtime plus old fallback.” It is “new runtime only.”

## Surprises & Discoveries

- Observation: the new knowledge runtime is conceptually correct but currently incomplete.
  Evidence: `_run_direct_knowledge_workers_v1(...)` dispatches `_run_phase_knowledge_worker_assignment_v1(...)`, but that function immediately discards `output_schema_path`, `cohort_watchdog_state`, `recovery_governor`, and `interruption_requested` instead of re-expressing those controls in the new runtime.

- Observation: the old runtime below the new one still contains most of the hard-earned resilience logic.
  Evidence: `_run_knowledge_workspace_worker_assignment_v1(...)` still owns preflight rejection, interrupt checks, watchdog retry, split retry, repair gating, and terminal status reasoning. The repo therefore has two knowledge runtimes in one file, but only the simpler one is live.

- Observation: the new phase-worker path already proved that the worker-control model is the right one.
  Evidence: the worker helper and phase files in `cookimport/llm/knowledge_phase_workspace_tools.py` provide the right paved road: `CURRENT_PHASE*`, `check-phase`, `install-phase`, Pass 1 to Pass 2 handoff, and shard-owned `in/work/repair/out` files. The missing work is resilience and cleanup, not another redesign.

- Observation: the current artifact vocabulary is still lying about the ownership model.
  Evidence: live helper code, writer code, ingest code, and some runtime metadata still use `packet_id`, `task_status`, and packet-centric naming even though the happy path is now shard-owned and phase-first.

- Observation: the most important final cut is deleting the duplicate runtime, not preserving it as insurance.
  Evidence: if both the phase-owned runtime and the packet/task runtime remain in-tree, future fixes will drift again because every semantic change to recovery, telemetry, or docs will need to be applied twice.

- Observation: in this repository, “keeping the old shape around” is usually just a way to postpone deletion rather than a real product requirement.
  Evidence: the user is the only operator, and the design goal is to replace bad code with better code, not to keep both systems alive forever.

- Observation: the fake in-process worker seam can accidentally fight the repo-owned phase controller if it rewrites Pass 1 after a local `check-phase` already froze accepted rows.
  Evidence: `tests/llm/test_knowledge_orchestrator_runtime_leasing.py::test_knowledge_orchestrator_seeds_valid_pass1_ledgers_before_worker_edits` failed with `frozen_row_modified:0` until the fake runner stopped overwriting an already-valid Pass 1 ledger and stopped syncing mutated `CURRENT_PHASE*` files back onto the source workspace.

## Outcomes & Retrospective

This plan starts from a partially migrated tree, so the current outcome is mixed rather than finished. The repository has already chosen the right worker-control model, but it has not yet finished the burn-the-boats cutover. The implementation work for this plan is complete only when the knowledge stage has one runtime, one recovery model, one reporting vocabulary, and no packet/task fallback architecture left on the happy path.

The main lesson captured here is that “new path plus old path” is not a neutral midpoint. Once the live dispatch moved to the phase-owned runtime, the older runtime stopped being an implementation detail and became a source of ongoing drift. The long-term fix therefore has to be a real consolidation, not another layer.

An additional practical lesson from this implementation slice is that the fake runner is part of the product contract for this repo. When the phase controller owns durable `frozen_rows`, even test-only workspace runners must respect that ownership or they will teach the wrong mental model and hide regressions behind fake-only behavior.

## Context and Orientation

The knowledge stage is the optional LLM-backed review of non-recipe cookbook prose after deterministic routing. In current code, the stage lives under `cookimport/llm/knowledge_stage/` and writes Stage 9 authority artifacts for non-recipe material. A `shard` in this context is one contiguous slice of the review-eligible non-recipe blocks. A `phase` is the repo-owned current step for a shard inside one worker session. Today the intended phases are:

- Pass 1: classify each owned block as `knowledge` or `other`.
- Pass 2: for the blocks kept as `knowledge`, assign `group_id` and `topic_label`.

The key files and their current roles are:

- `cookimport/llm/knowledge_stage/runtime.py`
  This is the composition root for the knowledge worker runtime. It currently contains both the newer phase-owned worker function and the older packet/task runtime lower in the file.

- `cookimport/llm/knowledge_phase_workspace_tools.py`
  This module defines the worker-side paved road: phase brief/feedback rendering, Pass 1 and Pass 2 ledger builders, repair payload builders, validators, and the helper script that exposes `check-phase` and `install-phase`.

- `cookimport/llm/codex_farm_knowledge_jobs.py`
  This is the shard planner. It now emits shard-owned knowledge inputs, but parts of the downstream runtime still speak packet/task language.

- `cookimport/llm/codex_farm_knowledge_ingest.py`
  This validates and normalizes model outputs back into promotable knowledge-stage results. It still carries packet-era naming and some old-design preservation baggage even though the happy path is now shard-ledger-first.

- `cookimport/llm/codex_farm_knowledge_writer.py`
  This writes reviewer-facing knowledge artifacts. Snippets are already gone from the happy path, but record fields still teach packet terminology.

- `cookimport/llm/prompt_preview.py`
  This reconstructs the human-facing preview of LLM work. It must stop teaching packet-era defaults and knowledge payload names where the runtime no longer uses them.

- `docs/10-llm/10-llm_README.md`, `docs/05-staging/05-staging_readme.md`, and `docs/plain-english-explanation.md`
  These are the durable human-facing docs that must match the final runtime, not the migration leftovers.

The current defect structure is simple. The new phase-owned path is the right design, but it is missing parts of the old safety envelope. The old runtime still exists in the same file and still implements important behaviors, including preflight rejection, interruption checks, watchdog retry, shard split retry, repair gating, and richer terminal status logic. The final design is to move those behaviors into the phase-owned runtime in phase-owned form, then delete the old runtime and its vocabulary.

This plan is intentionally destructive toward the old design. It is not a migration guide for outside users. It is a repository cleanup and architecture completion plan for a solo project. When the new path is complete, the old one should disappear.

## Milestones

### Milestone 1: Make the phase-owned worker path internally correct

This milestone fixes the new phase runtime so the paved road actually works. At the end of the milestone, Pass 1 work ledgers are seeded in the correct ledger shape, `check-phase` and `install-phase` work on a fresh worker root without manual repair, and the harvest logic can produce a final installed shard result either from `out/<shard>.json` or from validated phase ledgers when the worker used the paved road but did not emit the final file directly.

The proof is one fake worker root. A novice should be able to open `CURRENT_PHASE.md`, edit the named work file, run `python3 tools/knowledge_worker.py check-phase`, run `python3 tools/knowledge_worker.py install-phase`, and watch the repo advance from Pass 1 to Pass 2 and then to the next shard without any packet-era files or hidden alternate control surfaces.

### Milestone 2: Port the safety envelope into the phase-owned runtime

This milestone moves the useful resilience behavior from the older runtime into the phase-owned one. At the end of the milestone, `_run_phase_knowledge_worker_assignment_v1(...)` owns preflight rejection, interrupt checks, watchdog callback/cohort state, bounded rerun policy, shard split retry policy, and terminal status bookkeeping. The old runtime is no longer needed to explain how the knowledge stage survives malformed sessions.

The proof is behavioral rather than structural. A preflight-rejected shard should be reported through the live phase runtime. A killed worker session should either recover through same-contract rerun logic or land with explicit fail-closed status. A forced interrupt should stop cleanly and leave honest summary files. All of that should happen without routing the work back through the old packet/task worker path.

### Milestone 3: Burn down packet/task-era naming and stale write-side baggage

This milestone makes new artifacts teach the truth. At the end of the milestone, shard-owned runtime artifacts and compact payloads no longer use `packet_id` or `task_status` on the write path. Replace those names with `shard_id`, `shard_status`, and other phase-owned terms across runtime summaries, promotion inputs, preview rows, reviewer-facing knowledge artifacts, and focused tests.

The proof is a grep over the live write path plus the generated artifacts from a fake or real run. New worker roots and new run outputs should not teach packet/task vocabulary as if it were current behavior. Old fixtures should be regenerated or deleted rather than forcing the live write path to keep old names alive.

### Milestone 4: Delete the old runtime and document the final design

This milestone removes the donor path once parity is proven. Delete `_run_knowledge_workspace_worker_assignment_v1(...)`, `_run_knowledge_workspace_worker_assignment_taskized_v1(...)`, packet-era repair prompt flows, batch-first helper assumptions, and any tests/docs that still describe those paths as live. The result is one knowledge runtime with one worker-control model.

The proof is that a newcomer reading the docs and the runtime file can no longer discover two competing implementations. They should see one clear story: shard-owned planning, phase-owned worker session, deterministic validation, bounded shard-level recovery, final promotion, and no packet/task happy path.

## Plan of Work

Start in `cookimport/llm/knowledge_stage/runtime.py`. Treat the current `_run_phase_knowledge_worker_assignment_v1(...)` as the canonical owner and the older `_run_knowledge_workspace_worker_assignment_v1(...)` as donor code. Do not attempt to keep both coherent. The first edit wave should make the new function fully correct in isolation: seed Pass 1 work ledgers with the Pass 1 ledger builder, not the final-output builder; make the post-session harvest logic understand the phase-owned control plane; and teach the runtime to synthesize a final shard payload from validated Pass 1 and Pass 2 work ledgers when appropriate.

Then port the safety envelope into the same phase-owned function. Move preflight rejection up before the workspace session starts so unrunnable shards are marked terminal without pretending the worker processed them. Reuse the same watchdog callback style already used for other direct-exec runtimes, but record results against shard-owned phase artifacts. Reintroduce interrupt checks around the shard finalization loop so `KeyboardInterrupt` can stop the phase runtime cleanly. Reattach the cohort watchdog state and recovery governor, but use them to gate only shard-owned recovery actions.

The permitted recovery actions in the final design are narrow. First, allow a fresh whole-shard rerun in a new workspace session when the original session appears poisoned or malformed. That rerun must still expose the same `CURRENT_PHASE*` surface and the same `in/work/repair/out` files. Second, allow explicit shard split retry for oversized or repeatedly invalid shards when deterministic sizing rules say a smaller shard is justified. Those split children must also be phase-owned shards. Do not preserve the old snippet-repair or general repair prompt flows as permanent mechanisms. Ordinary repair belongs inside `check-phase` and `install-phase`. If a recovery mechanism cannot be expressed in shard-owned phase terms, delete it instead of carrying it forward.

After runtime parity exists, clean up the vocabulary across the knowledge stage. In `cookimport/llm/knowledge_phase_workspace_tools.py`, `cookimport/llm/codex_farm_knowledge_ingest.py`, `cookimport/llm/codex_farm_knowledge_writer.py`, and `cookimport/llm/prompt_preview.py`, rename compact payload fields and metadata to `shard_id` and shard-owned terms where new writes occur. Update any manifest or status file names that still claim to be task-based even though the knowledge stage no longer has live packet tasks. Do not add old-name readers unless a test or fixture absolutely forces one temporarily, and if that happens the follow-up action is to regenerate or delete the stale fixture rather than normalize forever.

Finally, delete the old runtime path and all supporting packet-era helpers that only existed for that path. This includes the older batch-first worker control surfaces, packet-task manifest builders, packet-era repair prompt flows, and tests or docs that still treat the packet runtime as the normal route. The final repository should have one runtime and one story.

## Validation and Acceptance

The implementation is complete only when all of the following statements are true. “The old path still works” is not an acceptance criterion; deletion is.

Runtime acceptance:

- `_run_direct_knowledge_workers_v1(...)` still dispatches the phase-owned runtime, but that phase-owned runtime now owns preflight, interrupt handling, watchdog gating, bounded rerun, split retry, and terminal status reasoning itself.
- `runtime.py` no longer contains a second live packet/task runtime below the phase-owned one.

Worker-contract acceptance:

- A fresh worker root can complete Pass 1 and Pass 2 through `check-phase` and `install-phase` without manual file surgery.
- If the worker used the phase ledgers correctly but did not write `out/<shard>.json` directly, repo code can still harvest and validate the final shard result from the ledgers.
- Ordinary repair happens only through phase-local `repair/*.json` plus the same session’s work ledger, not through a separate packet-era repair prompt.

Recovery acceptance:

- Preflight rejection, watchdog retry, interrupt handling, and any shard rerun or split retry behavior all land in phase-owned shard artifacts and phase-owned status fields.
- There is no route where a failed shard silently falls back into a packet/task happy path.
- A worker session counts as successful only when repo code can prove all assigned shards reached valid installed outputs.

Vocabulary acceptance:

- New write-path artifacts and docs use shard/phase terminology rather than packet/task terminology.
- Reviewer-facing knowledge outputs and prompt preview stop teaching packet identity where the runtime no longer has packets.
- Old fixtures or stale docs no longer force the live runtime to keep packet/task naming alive. Regenerate, update, or delete them.

Deletion acceptance:

- `_run_knowledge_workspace_worker_assignment_v1(...)`, `_run_knowledge_workspace_worker_assignment_taskized_v1(...)`, packet-era repair flows, and batch-first knowledge helper assumptions are gone.
- Focused tests and docs pass while teaching only the final phase-owned runtime.

## Idempotence and Recovery

This refactor should be implemented additively and then subtractively. First make the phase-owned runtime capable of handling the full safety envelope. Only after the focused tests pass and one worker-root rehearsal looks correct should the old runtime and packet-era helpers be deleted.

The work is safe to repeat because the validation commands are read-mostly and because the final runtime contract is artifact-based. If a change leaves the tree half-migrated, the correct recovery move is not to preserve both runtimes longer. Instead, fix the remaining parity gap in the phase-owned path and keep the tests focused on that path until deletion is safe.

Do not introduce new write-side preservation files just to smooth the cutover. The repository already has enough historical teaching. Prefer deleting or regenerating stale fixtures over teaching the codebase to speak two architectures.
