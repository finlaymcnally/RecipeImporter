# Cut recipe workspace readback token spend after the command-count win

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The previous recipe workspace passes fixed reliability and finally cut recipe command count in the Salt Fat benchmark rerun at `data/golden/benchmark-vs-golden/2026-03-22_13.22.18/single-book-benchmark/saltfatacidheatcutdown/codexfarm/`. Recipe now validates all five shards with zero watchdog kills, zero watchdog recoveries, zero structured follow-up calls, and only `47` workspace commands. That part worked.

But the same run proves the remaining cost moved rather than disappeared. Recipe `command_execution_tokens_total` rose to `536003`, which is worse than both the earlier recovered run at `2026-03-22_12.08.53` (`473128`) and the older good recipe baseline at `2026-03-22_03.33.50` (`453733`). The workers are now staying on the paved road, but they are still spending too much of each long-lived workspace session rereading repo-written files before they trust themselves to finalize drafts.

After this change, recipe workers should be able to complete the same benchmark shape while reading less, not just retrying less. The target outcome is recipe `validated_shard_count: 5`, recipe `watchdog_killed_shard_count: 0`, recipe `watchdog_recovered_shard_count: 0`, recipe `structured_followup_call_count: 0`, recipe `command_execution_count_total` no worse than `47` and ideally lower, and recipe `command_execution_tokens_total` below `536003` and then below `473128`, without asking the worker to rediscover the output contract, example files, or helper source on every shard.

## Outcomes & Retrospective

Implementation has not started yet for this plan. The previous recipe plan got the runtime into a healthy state on correctness and command count, but the benchmark still shows a large token bill caused by long-lived session readback and contract verification behavior.

The key lesson from the `2026-03-22_13.22.18` run is that a paved road is not enough if the worker still feels obligated to audit the pavement before using it. The next pass must make the cheapest path also feel like the safest path.

## Context and Orientation

Recipe workspace workers live under `data/output/<timestamp>/.../recipe_phase_runtime/workers/worker-XXX` in repo-owned artifacts and under a mirrored sterile execution workspace under `~/.codex-recipe/...` during the live `codex exec` session. The runner is `cookimport/llm/codex_exec_runner.py`. The recipe workspace orchestrator is `cookimport/llm/codex_farm_orchestrator.py`. The recipe helper that builds and validates local drafts is `cookimport/llm/recipe_workspace_tools.py`.

The current recipe worker contract already mirrors these repo-written files into the workspace:

- `worker_manifest.json`, which describes the workspace layout
- `current_task.json`, which holds machine-readable state for the current task
- `CURRENT_TASK.md`, which is a human-readable brief for the current task
- `CURRENT_TASK_FEEDBACK.md`, which currently acts as a queue/feedback note
- `OUTPUT_CONTRACT.md`, which explains the compact JSON contract
- `examples/*.json`, which show valid repaired/fragmentary/not-a-recipe outputs
- `tools/recipe_worker.py`, which provides `prepare-all`, `stamp-status`, `check`, `finalize`, and `finalize-all`
- `scratch/*.json`, which are prewritten draft files for each task

The current problem is not that these files are missing. The problem is that the worker still feels compelled to open too many of them. The prompt explicitly tells the worker to open several files up front. The current-task surface is rich for task 1 but not for later tasks. The contract still feels “remote,” so workers reread examples and helper source to reassure themselves before writing outputs. Because this all happens inside one long-lived workspace session, each defensive read contributes to cached context and wrapper overhead, which is what drives the large token bill.

## Milestones

### Milestone 1: Make task 1 cheap and trustworthy

The recipe worker prompt and current-task sidecars stop instructing broad upfront readback. A worker can start from one small repo-written brief, trust the prewritten draft, and finalize without opening contract files or helper source in the normal case.

### Milestone 2: Make task N+1 as cheap as task 1

Recipe stops relying on a worker-managed queue walk for later tasks. The repo-owned current-task bundle advances after each accepted task or provides an equivalent repo-owned “next task” surface, so later tasks do not require manual per-task rediscovery.

### Milestone 3: Hide contract complexity behind one small local surface

The details the worker keeps rereading from `OUTPUT_CONTRACT.md`, `examples/*.json`, and `tools/recipe_worker.py` are distilled into one cheap authoritative artifact or helper verb that is safe to trust during normal execution.

### Milestone 4: Verify locally and keep benchmark rerun approval-gated

Focused `.venv` tests pass for the cheaper recipe current-task/readback path. Benchmark rerun remains blocked on explicit user approval.

## Plan of Work

### Workstream A: Rewrite the recipe default path to minimize mandatory reads

Change the recipe worker prompt in `cookimport/llm/codex_farm_orchestrator.py::_build_recipe_workspace_worker_prompt(...)` so the normal path no longer starts with “open five files.” The worker should start from a single repo-written current-task brief and the named draft path. `worker_manifest.json`, `OUTPUT_CONTRACT.md`, `examples/*.json`, and helper source should become explicit fallback/debug references instead of required first reads.

The intent is not to hide those files or to remove them. It is to stop telling the worker to reread them every time. The new prompt should only require the minimum artifact set needed for normal work: the current-task brief, the machine-readable task row, the hint, the input, and the draft.

### Workstream B: Promote recipe to a repo-owned current-task loop

Give recipe the same practical advantage knowledge already has: after a task is accepted, the repo should advance the current-task surface so the worker can continue with task N+1 without rediscovering the queue. This does not require copying knowledge’s exact helper verbs, but it does require recipe to stop behaving like “task 1 has a brief, tasks 2+ are manual exploration.”

There are two acceptable designs:

1. add recipe-specific current-task helper verbs such as `check-current`, `finalize-current`, and `next`, with the repo mutating `current_task.json`, `CURRENT_TASK.md`, and `CURRENT_TASK_FEEDBACK.md` after each accepted finalize
2. keep `finalize-all` but add a repo-owned queue packet that exposes the next task as one cheap deterministic surface after each completed task

The plan chooses the first option unless implementation evidence shows it is much more invasive than expected. The reason is simple: recipe currently pays token cost because the worker is self-scheduling later tasks. A repo-owned current-task loop is the most direct fix.

### Workstream C: Collapse contract/example/tool rereads into one cheap artifact

Move the small pieces of information that workers keep verifying into one small local source. That source could be a richer `CURRENT_TASK.md`, a machine-readable `current_contract.json`, or a recipe-helper command such as `python3 tools/recipe_worker.py current` that prints the minimal contract summary for the current task. The exact shape is less important than the constraint: the worker should not need to open `OUTPUT_CONTRACT.md`, `examples/*.json`, or `tools/recipe_worker.py` during the ordinary path.

The current evidence suggests the missing information is:

- the exact compact-key contract
- the valid status choices and when to use them
- the acceptable `mr` values for non-repaired cases
- confirmation that the prewritten draft already matches the required shape

Whatever surface is chosen must answer those questions cheaply.

### Workstream D: Reduce mirrored artifact temptation

Review `worker_manifest.json` notes, the recipe prompt, and the mirrored example/tool surfaces together. The worker should still have access to those files, but the instructions and manifest notes should stop advertising them as routine orientation material. If a file is only for fallback/debug, say so explicitly in both the prompt and the manifest notes.

Consider whether recipe should mirror fewer example files or whether the current examples can be replaced with one smaller “shape summary” artifact. The guiding principle is that the worker should not feel obligated to read multiple large files to avoid violating the contract.

### Workstream E: Add regression tests that assert the cheaper path

Add or update tests so the repo enforces the intended recipe behavior rather than drifting back toward “read everything first.” The tests should check prompt text, current-task sidecar contents, helper behavior, and at least one representative telemetry pattern. The regression should make it hard to accidentally reintroduce instructions that send the worker back to `OUTPUT_CONTRACT.md`, `examples/*.json`, or `tools/recipe_worker.py` in the normal path.

## Idempotence and Recovery

It is safe to first update prompt text and current-task sidecar content, then add or revise recipe helper verbs for a repo-owned current-task loop, then adjust manifest notes and mirrored contract surfaces, and finally update regression tests and short docs.
