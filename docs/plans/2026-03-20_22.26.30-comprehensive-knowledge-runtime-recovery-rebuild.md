## Purpose / Big Picture

After this change, the non-recipe knowledge phase should behave like a bounded, inspectable stage rather than an expensive improvisation loop. A user who runs a Codex-backed benchmark or stage import should see real packet progress instead of `task 0/10`, workers should either produce grounded task outputs or fail closed quickly, retries and repairs should stay rare and high-yield, and an interrupted run should still leave a complete stage-level package that explains what happened.

The proof is concrete. In deterministic no-token tests and replay-style harnesses built from the saved Salt Fat Acid Heat artifacts, the knowledge stage should count task packets honestly, validate each task output as soon as it appears, cancel stale follow-up calls, cap low-yield recovery churn, and always write `phase_manifest.json`, `promotion_report.json`, `telemetry.json`, `failures.json`, and `knowledge_manifest.json` or an explicit interrupted-stage equivalent. With explicit user approval for token spend, a live Salt Fat rerun should then either complete or fail closed without the previous storm of watchdog kills, stale repair calls, and misleading progress output.

## Surprises & Discoveries

- Observation: the knowledge phase did substantial work before appearing stuck, but raw worker-local output files did not translate into stage completion.
  Evidence: the saved Salt Fat run under `data/output/2026-03-20_21.44.20/.../knowledge/` contains `485` task packets and `339` top-level `out/*.json` files, yet the stage root never finished `phase_manifest.json`, `telemetry.json`, `promotion_report.json`, `failures.json`, or `knowledge_manifest.json`.

- Observation: completed worker status is not a reliable success signal.
  Evidence: workers `001` and `002` ended with `reason_code=process_exited_without_watchdog_intervention` and `final_agent_message_state=malformed`, while workers `004`, `007`, `009`, and `010` ended with `workspace_outputs_stabilized`.

- Observation: worker-local shell use was not the only problem. Several workers stayed inside the workspace boundary but still produced low-trust or clearly templated outputs.
  Evidence: worker `007` explicitly planned to mark every packet useful and every block as kept knowledge using one `jq` loop, worker `004` reasoned from metadata lanes rather than chunk-local evidence, and worker `001` wrote `49` malformed files with a literal trailing `EOF`.

- Observation: the current recovery layer is overwhelmingly low-yield.
  Evidence: in the saved run, `115` repair attempts produced only `5` `repaired` outcomes, while `75` watchdog retries produced only `1` `validated` outcome.

- Observation: the runtime currently treats “output exists” as more meaningful than it really is.
  Evidence: worker `001` wrote `49/49` files but every one was invalid JSON; worker `008` wrote `48/48` files but its main shard still died by watchdog.

- Observation: follow-up calls can outlive the useful work they were meant to recover.
  Evidence: at shutdown, there were still `9` running repair or retry live-status files, including a worker `007` repair with roughly eight minutes of silence after the parent shard had already completed.

- Observation: the visible progress surface is still shard-biased even though the real answer unit is the task packet.
  Evidence: the benchmark timeseries under `data/golden/benchmark-vs-golden/2026-03-20_21.44.20/.../processing_timeseries_prediction.jsonl` stayed on `task 0/10 | running 10` during a `485`-packet knowledge stage.

- Observation: the current stage writes enough low-level artifacts to debug manually, but not enough summarized artifacts to debug quickly.
  Evidence: the knowledge root has `task_manifest.jsonl` and `worker_assignments.json`, but no finished phase package or compact metrics describing malformed outputs, retry yield, stale follow-ups, or off-task drift.

- Decision: treat task packets, not shards, as the authoritative answer unit for progress, validation, recovery, and final summaries.
  Rationale: the knowledge stage may use shards as worker ownership bundles, but the postmortem failures all happened at the packet level and the user-visible work unit is the packet output file.
  Date/Author: 2026-03-20 / Codex

- Decision: fail closed early instead of trying to salvage every bad output.
  Rationale: the postmortem shows that broad recovery is mostly wasted time and tokens. The stage should recover only narrow near-miss cases and quickly record explicit failure for the rest.
  Date/Author: 2026-03-20 / Codex

## Outcomes & Retrospective

This plan is not implemented yet. The current outcome is a single canonical implementation map that goes beyond the earlier March 20 plans by adding the missing systemic fixes: deterministic salvage, strict repair eligibility, stage-level recovery caps, stale follow-up reaping, smaller worker sessions, and operator-facing metrics. ## Context and Orientation

The non-recipe knowledge phase is the optional Codex-backed stage that reviews Stage 7 outside-recipe spans and decides whether each chunk is reusable cooking knowledge or just other non-recipe material. In this repository, the main runtime lives in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. Validation and normalization live in `cookimport/llm/codex_farm_knowledge_ingest.py`. Prompt building lives in `cookimport/llm/knowledge_prompt_builder.py`. Final writing of knowledge-stage outputs lives in `cookimport/llm/codex_farm_knowledge_writer.py`. The prompt and schema assets are `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md` and `llm_pipelines/schemas/recipe.knowledge.v1.output.schema.json`.

The runtime has two execution modes. A “workspace worker” is a long-lived Codex session started inside an isolated worker root that already contains repo-written files such as `worker_manifest.json`, `assigned_tasks.json`, `hints/*.md`, `in/*.json`, and `out/*.json`. A “structured follow-up” is a one-shot strict-JSON direct-exec call used for watchdog retry or repair after the main worker produced no usable answer. The shared subprocess and supervision logic for both modes lives in `cookimport/llm/codex_exec_runner.py` and `cookimport/llm/phase_worker_runtime.py`.

The knowledge stage currently plans a small number of top-level shards but fans each shard into many smaller “task packets.” A task packet is the real answer unit and normally corresponds to one `out/<task_id>.json` file inside a worker root. The saved failing benchmark at `data/golden/benchmark-vs-golden/2026-03-20_21.44.20/.../codexfarm/` used `10` top-level workers but `485` task packets in the knowledge phase. That is why shard-based progress and shard-only completion semantics were misleading.

The stage is expected to write a full phase package under `data/output/<ts>/raw/llm/<slug>/knowledge/`, including `phase_manifest.json`, `shard_manifest.jsonl`, `task_manifest.jsonl`, `worker_assignments.json`, `promotion_report.json`, `telemetry.json`, `failures.json`, and proposal or failure artifacts. The stage should also write `knowledge_manifest.json` at the workbook root and keep run-level observability coherent through `cookimport/runs/stage_observability.py` and `cookimport/llm/prompt_budget.py`.

The benchmark and terminal UI depend on structured progress payloads produced from the stage runtime and rendered in `cookimport/cli.py` via `cookimport/core/progress_messages.py`. The current knowledge phase still reports shard progress at the top level, which is why the saved run looked frozen at `task 0/10`. Partial benchmark finalization currently lives in `cookimport/cli.py` and `cookimport/labelstudio/ingest.py`, but the saved run shows that stage-level finalization is still too weak when the user aborts mid-knowledge.

The canonical evidence bundle for this plan is the March 20 saved run:

- benchmark root: `data/golden/benchmark-vs-golden/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/`
- processed run root: `data/output/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-20_21.44.56/`
- knowledge root: `data/output/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-20_21.44.56/raw/llm/saltfatacidheatcutdown/knowledge/`

This plan assumes those historical artifacts remain read-only evidence. Tests and local replay helpers may copy or synthesize from them, but must not mutate them.

## Plan of Work

### Milestone 1: Rebuild the main worker contract so bounded packet work is the default path

Start in `cookimport/llm/phase_worker_runtime.py`, `cookimport/llm/codex_exec_runner.py`, `cookimport/llm/knowledge_prompt_builder.py`, and the prompt asset `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md`. The current worker contract is too permissive in tone and too ambiguous in practice. Rewrite it so a novice can tell exactly what the worker may read, where it may write outputs, where scratch work belongs, which tools are forbidden, and what counts as grounded packet review. Add an explicit workspace-local `scratch/` directory to the worker root and make the manifest say plainly that temporary helper scripts belong there if the worker truly needs them. State directly that `/tmp`, parent traversal, repo-global tools, network tools, and inline interpreters such as `python - <<'PY'` are forbidden.

Make the prompt hostile to blanket automation and metadata shortcuts. Tell the worker not to solve many packets with one template, not to classify from lane metadata or hints alone, not to mark every packet `keep` or every packet `drop`, and not to emit snippets that simply echo every block wholesale. The prompt should require chunk-local evidence and task-local judgment. Include one short good example that reads one packet and writes one output, plus one short bad example that shows the kind of templated or metadata-driven shortcut the runtime will reject.

This milestone should also add a narrow early-prevention seam in `cookimport/llm/codex_exec_runner.py` for main workspace workers. The purpose is not to let the watchdog “teach” forever. The purpose is to intercept the first predictable violation path and redirect once before the worker wastes time or dies. If the worker tries to read or write off-root, invoke an interpreter, or use another obviously forbidden family, the runtime should write a concise corrective note or structured intervention and only escalate to kill if the worker persists.

The acceptance for this milestone is textual and deterministic. New tests should assert that the worker manifest and prompt explicitly mention `scratch/`, explicitly forbid `/tmp` and interpreters, explicitly forbid cross-task templating, and include at least one bounded local automation example that the runtime considers allowed.

### Milestone 2: Introduce packet-level terminal states and immediate output validation

The current runtime waits too long to decide whether a worker output is usable. In `cookimport/llm/codex_farm_knowledge_orchestrator.py`, create an explicit packet-state model and make each packet end in exactly one terminal state. Reasonable states include `validated`, `invalid_json`, `schema_invalid`, `semantic_invalid`, `missing_output`, `repair_recovered`, `retry_recovered`, `terminal_failed`, `cancelled`, and `superseded`. The exact names can change during implementation, but the invariant must remain: one packet, one terminal state.

Then teach the runtime to inspect `out/<task_id>.json` as soon as it appears, not only after the whole worker finishes. Newly written files should immediately go through parse validation, schema validation, and the existing semantic checks in `cookimport/llm/codex_farm_knowledge_ingest.py`. If a file fails parse, mark it `invalid_json` immediately and record the exact failure. If it parses but violates the schema or semantic contract, mark that precise failure and move the packet into bounded salvage or recovery. Do not allow dozens of bad files to accumulate before the stage notices.

This milestone must also add deterministic checks for templated low-trust outputs that stay within the formal schema but are clearly not grounded packet review. Examples from the saved run include blanket all-knowledge outputs, hints-only reasoning, and reason strings copied from metadata. The runtime does not need a speculative AI detector. It only needs deterministic rules for the concrete bad shapes already seen: uniform all-keep or all-drop patterns across many packets, snippets that blindly echo whole packets, wrong version or vocabulary, and reason fields that are obviously generic or metadata-derived.

Acceptance for this milestone is a set of fail-before tests that reproduce the saved malformed-output class, including the literal trailing `EOF` case, plus new tests showing that the orchestrator marks the packet invalid at the moment the file is detected rather than after the worker has written many more files.

### Milestone 3: Add deterministic salvage before any model repair

Once immediate validation exists, implement a local salvage layer in `cookimport/llm/codex_farm_knowledge_orchestrator.py` or a small helper module near the ingest seam. The saved run shows at least one narrow malformed-output shape that should never spend another model call: files that are otherwise valid JSON but contain an obvious trailing heredoc terminator like `EOF`. The salvage layer should fix only narrow, mechanically safe cases. If a transformation is not clearly safe and deterministic, do not guess.

Examples of acceptable salvage work are trimming a literal trailing heredoc terminator, stripping obvious wrapper noise that the repo already knows how to identify, normalizing unambiguous version or field aliases if the structure is otherwise correct, and removing impossible snippet content from `u=false` rows if the rest of the packet is intact. Salvage should then rerun the same parse, schema, and semantic validation and record whether the packet became acceptable due to local repair.

This milestone matters because the current repair path is wasting model calls on cases the local runtime should solve itself. Acceptance is a focused test suite that proves salvage recovers the narrow malformed-output shapes from the postmortem and leaves ambiguous or broad junk untouched.

### Milestone 4: Narrow repair eligibility so only near-miss packets can spend a repair call

After deterministic salvage lands, tighten the repair gate in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. A packet should enter model repair only if it is a genuine near miss. Near miss means parseable JSON, clear packet identity, complete owned coverage, and a small number of validator failures that a strict repair prompt can realistically correct. Broad junk, zero-output workers, blatantly templated outputs, wrong-version payloads, off-task packet sets, and malformed outputs that deterministic salvage could not fix should never enter repair.

This is where the stage should stop pretending every bad output is worth another shot. The acceptance bar for repair should be high. If the packet does not meet it, mark the packet terminally failed and preserve seed authority or explicit unreviewed status as appropriate. The saved `115` repair attempts with only `5` wins are the negative regression fixture for this milestone.

Acceptance for this milestone is a small matrix of tests that shows repair is attempted for near-miss packets only, and never attempted for broad malformed-output classes, no-output classes, or obvious templated-output classes.

### Milestone 5: Replace open-ended retry and repair churn with strict budgets, a stage circuit breaker, and stale-follow-up reaping

Now rebuild recovery as a bounded system. In `cookimport/llm/codex_farm_knowledge_orchestrator.py`, set strict per-packet budgets. Each packet should get at most one watchdog retry and one repair path unless the test-backed implementation later justifies a different shape. Once the budget is exhausted, the packet is terminally failed. A terminal packet must never spawn another follow-up call.

Add a stage-level circuit breaker on top of the per-packet caps. The exact threshold can be refined during implementation and recorded in the `Decision Log`, but the stage must stop launching more recovery work once follow-up yield collapses. A simple and acceptable first version is a sliding threshold based on completed follow-ups and accepted recoveries. For example, after the first 20 or 25 follow-up calls, if the acceptance rate remains below a low threshold, stop creating more follow-ups and fail remaining bad packets closed. The saved run’s `5/115` and `1/75` outcomes are the reason this is mandatory.

Then add active stale-follow-up reaping. If a parent packet becomes terminal, any older running retry or repair for that packet must be cancelled or marked superseded. If a shard becomes terminal, child follow-ups that no longer matter must be cancelled. If a retry or repair goes silent or shows no progress beyond a bounded interval, mark it stale and stop waiting on it. The worker `007` stale repair and the nine running follow-up statuses at shutdown are the regression fixtures here.

Acceptance for this milestone is a replay-style fake-runner test harness showing that packets do not accumulate endless follow-up calls, stale follow-ups are reaped when their parent becomes terminal, and the stage-level circuit breaker can stop a low-yield storm without leaving ambiguous state behind.

### Milestone 6: Resize the main worker blast radius by shrinking or rotating workspace sessions

The current main worker assignment size lets one bad strategy contaminate too much of the stage. In `cookimport/llm/codex_farm_knowledge_jobs.py` and `cookimport/llm/codex_farm_knowledge_orchestrator.py`, change the runtime so a single long-lived workspace session does not normally own `48-49` packets without interruption. There are two acceptable shapes for the first implementation. The first is smaller packet bundles per worker, reducing the number of packets assigned to any one session. The second is worker rotation after N completed packets, where the orchestrator starts a fresh bounded workspace session once a worker has successfully completed a configured batch size.

The goal is not merely throughput tuning. The goal is to limit how much damage one bad automation strategy can do before the runtime reacts. This milestone should also make packet counts per worker visible in runtime metadata so tests and operators can confirm that a worker no longer owns an outsized batch by default.

Acceptance is a deterministic planning test showing smaller packet ownership bundles or explicit session rotation, plus one fake-runner regression showing that a worker failure cannot spray the same malformed behavior across dozens of packets before the runtime notices.

### Milestone 7: Rebuild progress, worker labels, and stage summaries around packet truth and follow-up truth

Once packet states are authoritative, update the visible progress model in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, `cookimport/core/progress_messages.py`, and `cookimport/cli.py`. The visible task counter for the knowledge phase must mean task packets, not top-level shards. Use `task_current/task_total` for packet counts. Move shard totals, follow-up counts, and worker counts into detail lines. Worker labels should show shard ownership plus local packet progress or explicit repair/retry state, not only the shard id.

The benchmark and terminal UI should make a saved run like this legible, for example:

    Running codex-farm non-recipe knowledge review... task 233/485 | running 6
    configured workers: 10
    completed shards: 3/10
    repair calls running: 2
    retry calls running: 1
    worker 10: saltfatacidheatcutdown.ks0009.nr (47/48 task packets)

This milestone must also ensure that terminal failed packets still advance the visible progress. If a packet is done failing, the counter must move. Acceptance is a set of progress-helper and CLI-rendering tests that prove the knowledge stage no longer looks frozen at `0/10` while packet work is happening.

### Milestone 8: Guarantee stage-level and benchmark-level finalization on success, failure, and interruption

Move finalization logic in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, `cookimport/runs/stage_observability.py`, `cookimport/labelstudio/ingest.py`, and `cookimport/cli.py` so the knowledge stage always writes a coherent phase package. On success, it should write the normal `phase_manifest.json`, `promotion_report.json`, `telemetry.json`, `failures.json`, proposals, and `knowledge_manifest.json`. On partial failure or interruption, it should still write a finished phase package that explains packet totals, terminal packet states, accepted and failed recoveries, and why the stage stopped.

The benchmark interruption path must point to that stage package. If the user aborts mid-knowledge, the benchmark root should still contain `benchmark_status.json` and `partial_benchmark_summary.json`, and those files should reference a complete stage-level interrupted package rather than a half-empty raw worker tree.

This milestone must also stop new follow-up spawning once the user interrupts the benchmark. If fully graceful signal handling is too invasive, the minimum acceptable behavior is that the orchestrator stops launching new work, marks active children interrupted or cancelled, writes the stage and benchmark summaries, and exits cleanly.

Acceptance is a targeted interruption test showing a partial but coherent stage and benchmark package after a synthetic abort.

### Milestone 9: Make observability first-class so the next postmortem is one file, not a manual artifact crawl

After the runtime is correct, make the metrics readable. Extend `cookimport/llm/prompt_budget.py`, `cookimport/llm/prompt_artifacts.py`, and the stage summary code so a finished or interrupted run reports compact but useful knowledge-stage metrics. At minimum, record:

- packet totals and terminal packet-state counts;
- main worker counts by outcome category;
- invalid JSON file count;
- deterministic salvage successes and failures;
- repair attempts, repairs accepted, repairs failed, and repair-skipped counts by reason;
- watchdog retries attempted, retries accepted, retries failed, and retries skipped by reason;
- stale or superseded follow-up count;
- off-task or prevention-intervention count;
- malformed-completion count.

These metrics should appear in stage summaries and, where appropriate, in the existing prompt-budget summary path so the operator can see the phase health without rereading every worker `events.jsonl`. Acceptance is a deterministic test that a partial or failed knowledge run writes these counts in a machine-readable summary.

Only after those deterministic tests pass and only with explicit user approval, run a live Salt Fat single-offline CodexFarm benchmark as the final confidence pass.

## Validation and Acceptance

This plan is complete only when all of the following are true.

First, the worker contract is explicit and testable. The manifest and prompt clearly state where scratch work belongs, what is forbidden, why cross-task templating is not allowed, and what grounded packet review looks like. Tests prove that bounded local shell work under the worker root is allowed, while `/tmp`, parent traversal, network tools, and inline interpreters remain forbidden.

Second, packet outputs are validated immediately. A malformed file such as a JSON payload with trailing `EOF` is caught at the moment it appears and does not trigger a long tail of downstream repairs. Deterministic salvage recovers only narrow, mechanically safe cases and records that it did so.

Third, repair eligibility is narrow and bounded. Near-miss packets can be repaired. Broad junk, no-output packets, and templated outputs cannot. Per-packet recovery budgets are strict, terminal packets remain terminal, and stale follow-up calls are reaped when they no longer matter.

Fourth, stage-level churn is bounded. The runtime no longer produces the postmortem-style `115` repairs and `75` retries for one knowledge phase without tripping a hard stop. The stage-level circuit breaker can fail closed and leave a readable summary.

Fifth, worker blast radius is reduced. One bad workspace session can no longer own an oversized `48-49` packet batch by default without rotation or smaller ownership bundles.

Sixth, the benchmark UI reports packet truth. A `485`-packet knowledge phase must report packet counts at the top level, worker labels must include shard ownership plus packet or follow-up state, and terminal failed packets must still move visible progress forward.

Seventh, success, failure, and interruption all write coherent stage-level artifacts. A knowledge-stage root must always contain the phase package or an explicit interrupted-stage equivalent, and benchmark-level partial summaries must point at that package.

Eighth, compact health metrics make future postmortems cheap. A reader should be able to open one stage summary and see malformed-output counts, salvage counts, repair and retry yield, stale-follow-up counts, and packet terminal-state counts.

## Idempotence and Recovery

This work should be implemented additively and is safe to retry when done milestone by milestone. Tests come first so the current bad behavior is pinned before runtime changes land. The saved March 20 artifact roots are read-only evidence and must remain untouched. Any local replay helper must copy or synthesize from them instead of mutating them.

If deterministic salvage grows too broad or ambiguous, stop and narrow it. Salvage must remain a small set of mechanically safe transformations, not a shadow repair engine. If the stage-level circuit breaker proves too aggressive in early tuning, lower its authority to warning-only in tests first, then re-enable the hard stop once acceptance thresholds are backed by evidence. If smaller worker sessions or rotation cause unacceptable throughput regressions, capture the measurements in `Surprises & Discoveries` and adjust the packet cap rather than abandoning the blast-radius reduction entirely.

If a milestone turns out to be too invasive to finish safely in one pass, land the bounded subset that preserves the new invariants and update this plan explicitly before moving on. Do not leave half-wired retry logic or ambiguous terminal-state semantics in production.
