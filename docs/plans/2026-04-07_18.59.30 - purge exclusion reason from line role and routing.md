# Purge Exclusion Reason From Line Role And Routing

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the program no longer asks line-role workers for `exclusion_reason`, does not validate or normalize that field, and does not write exclusion-reason analytics into staging or benchmark artifacts. A user should be able to inspect prompt previews, runtime shard packets, stage summaries, and nonrecipe routing output and see only the coarse authority boundary that still matters here: `NONRECIPE_CANDIDATE` versus `NONRECIPE_EXCLUDE`.

The user-visible result is a simpler contract and less churn. The line-role runtime stops spending effort on low-value subtype detail, and downstream artifacts stop pretending that exclusion-family counts are part of the product. The proof is focused: targeted parsing, staging, and benchmark tests pass with a contract that contains no `exclusion_reason` field anywhere in the active path.

## Surprises & Discoveries

- Observation: `exclusion_reason` is not isolated to line-role prompting. It also appears in nonrecipe routing artifacts, stage observability summaries, prompt previews, cutdown export, and benchmark leaderboard reporting.
  Evidence: `rg -n "exclusion_reason" .` returned matches across `cookimport/parsing`, `cookimport/staging`, `cookimport/bench`, `cookimport/llm`, and multiple test suites.

- Observation: The active codebase still contains deterministic helper logic for deriving outside-recipe exclusion reasons.
  Evidence: `cookimport/parsing/canonical_line_roles/policy.py` defines `_outside_recipe_exclusion_reason(...)` and uses it during normalization and policy rejection, so this purge must remove real behavior rather than just prompt text.

- Observation: Keeping the coarse exclude route without subtype baggage required a smaller policy seam, not a total deletion of outside-recipe exclusion support.
  Evidence: after the refactor, `cookimport/parsing/canonical_line_roles/policy.py` still needs one boolean gate, `_outside_recipe_exclude_allowed(...)`, so deterministic baseline and codex-policy vetoes can keep the existing `NONRECIPE_EXCLUDE` authority boundary without carrying reason codes through the runtime.

- Observation: Most breakage after the code purge was in frozen expectations, not in production logic.
  Evidence: the first test failures were collection/import issues plus outdated assertions about prompt text, routing ledgers, and same-session repair metadata. Once those fixtures were updated, the focused suites passed without further production rewrites.

## Context and Orientation

The canonical line-role path lives under `cookimport/parsing/canonical_line_roles/`. That package defines the prediction contract, prompt construction, validation, worker transports, and the runtime acceptance path that turns LLM output into block-level routing state. Today that contract still allows a second field named `exclusion_reason` on `NONRECIPE_EXCLUDE` rows.

The staging side lives under `cookimport/staging/`. `cookimport/staging/nonrecipe_stage.py`, `cookimport/staging/nonrecipe_routing.py`, `cookimport/staging/nonrecipe_authority_contract.py`, and `cookimport/staging/writer.py` currently preserve `exclusion_reason_by_block` maps and counts in persisted artifacts. Benchmark and observability code under `cookimport/bench/` and `cookimport/runs/` then surface those counts to leaderboard and summary views.

The prompt and packet surface spans both runtime prompt builders in `cookimport/llm/canonical_line_role_prompt.py` and the checked-in prompt references under `llm_pipelines/prompts/`. Tests that freeze this behavior live mostly under `tests/parsing/`, `tests/staging/`, `tests/llm/`, and `tests/bench/`.

## Plan of Work

Start at the contract owner. Remove the `exclusion_reason` field and related normalization helper from `cookimport/parsing/canonical_line_roles/contracts.py`, then simplify every consumer in validation, runtime, worker transports, and prompt builders so accepted row payloads only carry label authority. Delete deterministic policy logic whose only purpose was computing or checking exclusion reasons; keep the existing coarse labels and outside-recipe containment behavior, but without a subtype code.

Next, remove downstream artifact plumbing. Update nonrecipe routing and staging so they no longer collect `exclusion_reason_by_block` or emit reason counts. Remove the same field from prompt preview and benchmark/reporting surfaces so the entire active output graph matches the simplified contract.

Then update checked-in prompt references, docs, fixtures, and focused tests. Tests should now prove that valid row payloads contain only `label`, that routing artifacts no longer expose exclusion-reason detail, and that benchmark summaries no longer report exclusion-reason counts.
