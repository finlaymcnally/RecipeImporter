# Make Canonical Benchmark Gold Block-Authoritative For Freeform Mass-Labeling

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a human annotator can keep using broad freeform highlights in Label Studio to label many adjacent `KNOWLEDGE` or `OTHER` paragraphs at once, while the benchmark gold remains stable and block-authoritative. One long drag across five adjacent blocks must mean the same thing as five separate per-block highlights. The export should still preserve the raw freeform spans as an archive of what the annotator drew, but benchmark authority must come from derived per-block labels, not from the geometry of the freeform drag.

The user-visible proof is simple. After exporting a Label Studio project and running `labelstudio-benchmark --eval-mode canonical-text`, the canonical gold count should no longer change merely because one highlight crossed block boundaries differently. The benchmark output and docs should also stop implying that `freeform_span_labels.jsonl` is the scored artifact in canonical mode; canonical mode should clearly describe the derived block-authoritative canonical artifacts it uses.

## Surprises & Discoveries

- Observation: The current stage-block evaluator already does the desired semantic move: it reads `freeform_span_labels.jsonl`, extracts touched block indices, and treats labels as block assignments rather than preserving the exact span geometry.
  Evidence: `cookimport/bench/eval_stage_blocks.py:load_gold_block_labels(...)` builds `assignments[block_index] -> set[label]` from `touched_block_indices`.

- Observation: The current canonical export path does something different. It walks each freeform span's touched blocks and emits one canonical span row per touched block overlap, preserving cross-block freeform geometry.
  Evidence: `cookimport/labelstudio/canonical_gold.py:_build_canonical_spans(...)` loops over `touched_blocks`, computes overlap per block, and appends one canonical span row per overlapping block part.

- Observation: The recent benchmark confusion came from a real export rewrite, not a benchmark counter bug.
  Evidence: `data/golden/pulled-from-labelstudio/saltfatacidheatcutdown/exports/*` was rewritten at `2026-04-13 15:00:14 -0400`, and `summary.json` now reports `canonical_span_count: 1888` while the earlier benchmark artifact recorded `1881`.

- Observation: In the current repo, canonical benchmark mode scores `canonical_span_labels.jsonl`, but benchmark UX still tends to frame `freeform_span_labels.jsonl` as the selected gold input because that file is the entry point to the gold export bundle.
  Evidence: `docs/07-bench/07-bench_README.md` says canonical mode expects `canonical_text.txt`, `canonical_span_labels.jsonl`, and `canonical_manifest.json`, while CLI help and benchmark summaries still point operators at the freeform file path as the chosen gold export.

- Observation: Making the canonical artifact truly stable across equivalent labeling styles required dropping raw span-shape provenance from the benchmark-authoritative exports. Keeping fields like freeform `span_id`, `result_id`, or overlap-derived selected text inside canonical rows would still make two semantically identical exports differ on disk.
  Evidence: an interactive repro showed the mass-label and per-block exports produced the same labels and block coverage but different `span_id`/`result_id`/selected-text provenance until those fields were removed from `canonical_span_labels.jsonl` and `block_gold_labels.jsonl`.

- Observation: The existing tests were enough to protect the stage-block semantics once the loader was refactored, but canonical export needed a new dedicated regression proving equivalence between one broad drag and several per-block clicks.
  Evidence: `tests/bench/test_eval_stage_blocks.py` passed unchanged after the refactor, while `tests/labelstudio/test_labelstudio_export.py::test_labelstudio_export_mass_labeling_is_block_authoritative` was added to cover the new contract directly.

## Context and Orientation

This repository has two different gold representations today, and they are easy to confuse.

`freeform_span_labels.jsonl` is the raw Label Studio export. Each row describes what the annotator highlighted inside one Label Studio task segment, including the exact local offsets and the list of touched source blocks. This is a good record of human input, but it is not stable when the human labels broad runs of adjacent blocks with one drag instead of several smaller drags.

`canonical_text.txt` is the merged source text used by canonical benchmarking. It is built by ordering source blocks by `block_index` and joining them with a blank-line separator. `canonical_span_labels.jsonl` is the canonical-mode gold file that points into that merged text with `start_char` and `end_char`.

Today, `cookimport/labelstudio/export.py` writes both artifacts. It writes raw freeform spans first, then calls `cookimport/labelstudio/canonical_gold.py` to build canonical gold directly from those raw spans. That builder preserves span geometry across touched blocks. If one freeform highlight overlaps three blocks, the canonical export writes three canonical span rows for the overlapping fragments.

The stage-block benchmark path already has the behavior the user wants. `cookimport/bench/eval_stage_blocks.py` reads the same raw freeform span rows, extracts the touched block indices, and treats those rows as per-block labels. This means the same source data is already interpreted two different ways depending on benchmark mode.

The goal of this plan is to unify those interpretations around block-authoritative gold while preserving the raw freeform archive.

The key files and modules are:

- `cookimport/labelstudio/export.py`: owns Label Studio export artifact writing and `summary.json`.
- `cookimport/labelstudio/canonical_gold.py`: owns canonical gold construction and writing.
- `cookimport/bench/eval_stage_blocks.py`: already contains the repo's existing freeform-to-block gold semantics.
- `cookimport/bench/eval_canonical_text.py`: reads canonical gold artifacts during canonical benchmark scoring.
- `tests/labelstudio/test_labelstudio_export.py`: the main export artifact regression file.
- `tests/bench/test_eval_stage_blocks.py`: the main gold-loading and canonical-eval seam coverage.
- `docs/06-label-studio/06-label-studio_README.md` and `docs/07-bench/07-bench_README.md`: long-lived docs that define the export and scoring contracts.

In this document, "block-authoritative" means the following concrete rule: if a freeform annotation row touches block indices `101`, `102`, and `103`, then each of those source blocks receives that label for benchmark authority, regardless of whether the drag covered the whole block or only part of it. "Canonical block span" means the full char range in `canonical_text.txt` occupied by one source block in `canonical_block_map.jsonl`.

## Plan of Work

The first change is to factor the existing freeform-to-block logic into a shared helper that lives under `cookimport/labelstudio/` instead of only inside the stage-block evaluator. Create a small module such as `cookimport/labelstudio/gold_block_labels.py` that accepts freeform span rows and returns two things: a per-block label map and a richer row-oriented export payload. The row-oriented payload should contain at least `block_index`, the sorted set of allowed labels for that block, and compact provenance about which raw span ids contributed those labels. Keep this helper deterministic and geometry-agnostic beyond "which blocks were touched". Do not make it smarter than the current stage-block evaluator; reuse its semantics, including `HOWTO_SECTION` label-set handling, so canonical and stage-block gold stay aligned.

The second change is to make `cookimport/labelstudio/export.py` write that derived block-gold artifact into the export bundle. Use a new explicit artifact name such as `exports/block_gold_labels.jsonl`. Update `summary.json` and the export run manifest so the new file is visible beside the raw freeform spans and canonical artifacts. The summary should clearly distinguish archive input counts from benchmark-authoritative counts. For example, keep `counts.labeled` for raw freeform rows, but add a `block_gold` summary section with block count and conflict diagnostics if any exist.

The third change is to rebuild canonical gold from the derived block-gold artifact, not from raw freeform span fragments. In `cookimport/labelstudio/canonical_gold.py`, change the main bundle-builder interface so it can consume the canonical block map and the block-gold rows. For each exported source block and each label assigned to that block, emit one canonical span row that covers the full canonical char range of that block. This means a source block with two allowed labels becomes two full-block canonical spans. Preserve provenance fields like contributing span ids when possible, but stop encoding the exact freeform overlap geometry into canonical gold. The output file can stay named `canonical_span_labels.jsonl` because canonical benchmark mode still expects a span-shaped canonical artifact; only the source of truth changes.

The fourth change is to update the compatibility seam that currently lazily rebuilds canonical artifacts. `ensure_canonical_gold_artifacts(...)` should continue to work if an old export bundle only has `freeform_span_labels.jsonl`, but when rebuilding it should first derive `block_gold_labels.jsonl` and then derive canonical artifacts from the block-gold file. This keeps old exports usable while making new exports explicit.

The fifth change is to update operator-facing benchmark wording. Any benchmark or eval summary that says "Gold spans" or otherwise points to `freeform_span_labels.jsonl` in canonical mode should be rewritten to distinguish the gold export bundle entry file from the scored canonical artifacts. In canonical mode, the operator should see both the export bundle root and the canonical scored files. The wording must make it clear that the freeform file is the source export artifact, while `canonical_text.txt` plus `canonical_span_labels.jsonl` are the scored files.

The sixth change is docs. Update `docs/06-label-studio/06-label-studio_README.md` to explain the new contract: freeform spans are archive input, block gold is benchmark authority, canonical gold is derived from block gold. Update `docs/07-bench/07-bench_README.md` to explain that canonical mode still consumes canonical artifacts, but those artifacts now represent full-block authority rather than raw span fragments. Keep the explanation plain and operator-focused so future confusion does not recur.

The seventh change is tests. Add or update focused tests to prove the following behavior:

One, exporting two freeform examples that differ only in how a broad cross-block sweep was drawn produces the same `block_gold_labels.jsonl` and the same `canonical_span_labels.jsonl`.

Two, the canonical export now emits full-block canonical spans based on block labels rather than smaller overlap fragments.

Three, the stage-block evaluator and canonical export use the same underlying blockization semantics so they cannot drift apart silently.

Four, benchmark or CLI summaries in canonical mode mention the actual scored canonical files instead of only the freeform file path.

## Validation and Acceptance

The change is accepted when all of the following are true.

First, if two exports differ only in whether the annotator dragged one long `KNOWLEDGE` highlight across several adjacent blocks or clicked those blocks separately, the exported `block_gold_labels.jsonl` files are identical.

Second, canonical export count stability improves. Changing where inside a block the freeform drag begins or ends must not change `summary.json.canonical.canonical_span_count` unless the set of labeled blocks changes.

Third, canonical benchmark mode still works with the existing artifact names. Running a canonical benchmark from an exported gold bundle succeeds without requiring a new CLI flag or a different gold-selection flow.

Fourth, canonical benchmark UX is clearer. In canonical mode, operator-facing output identifies the scored canonical files and no longer implies that the raw freeform span file is itself the scored artifact.

Fifth, focused tests pass, and at least one export regression test proves the mass-labeling equivalence directly.

## Idempotence and Recovery

This change should remain safe to rerun. `labelstudio-export` already overwrites the target export folder in place, so the new derived `block_gold_labels.jsonl` artifact must also be rewritten deterministically on every export. Compatibility rebuild helpers must be able to recreate missing canonical artifacts from old export bundles that only contain the raw freeform spans. Do not make the new path depend on manual migration steps.

If implementation reveals that some old tests implicitly depend on raw overlap-fragment canonical spans, update those tests only after classifying them as intentional contract changes. Do not preserve the old geometry-sensitive behavior through hidden compatibility branches; that would defeat the purpose of the change.

## Interfaces and Dependencies

Define one shared helper module under `cookimport/labelstudio/` with stable entry points. The names may vary slightly during implementation, but the end state must include one obvious place where freeform rows become block-authoritative gold. A good target shape is:

    load_or_build_block_gold_rows(span_rows: list[dict[str, Any]]) -> list[dict[str, Any]]
    block_gold_map(block_gold_rows: list[dict[str, Any]]) -> dict[int, set[str]]

The exported block-gold row shape should include:

- `block_index`: source block index
- `labels`: sorted list of allowed labels for that block
- `source_hash`
- `source_file`
- compact provenance such as contributing `span_ids`

`cookimport/labelstudio/export.py` must write this artifact to `exports/block_gold_labels.jsonl` and publish its path in `summary.json` and the export run manifest.

`cookimport/labelstudio/canonical_gold.py` must accept block-gold rows as authoritative input and emit canonical rows that cover each block's full canonical char range.

`cookimport/bench/eval_stage_blocks.py` must use the shared block-gold semantics instead of its current private duplication so stage-block and canonical authority cannot drift.

`docs/06-label-studio/06-label-studio_README.md` and `docs/07-bench/07-bench_README.md` must be updated to reflect the new artifact contract and the canonical-mode scoring explanation.
