# Cut the remaining recipe workspace token spend after the batch-finalize rollback

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

Recipe reliability is back, and the worker now uses the intended batch-finalize path, but the latest approved benchmark rerun still shows recipe spending far more tokens than the good pre-regression baseline. After this change, a recipe worker should still keep the good parts of the refactor, specifically `SHARD_PACKET.md`, prewritten drafts, `scratch/_prepared_drafts.json`, and fallback helper verbs, while cutting the remaining waste from four specific sources: oversized startup surfaces, broad `assigned_tasks.json` dumps, stale post-finalize queue state, and schema-fix churn in the prewritten drafts themselves. A human should be able to rerun the same Salt Fat single-book codexfarm slice after implementation and see recipe keep `validated_shard_count: 5`, keep all watchdog and follow-up counts at zero, and move recipe command count and command-execution tokens back toward or below the `2026-03-22_13.22.18` baseline.

  Evidence: `data/golden/benchmark-vs-golden/2026-03-22_16.52.22/.../prompt_budget_summary.json` records recipe `command_execution_count_total: 64` and `command_execution_tokens_total: 847663`, which is far better than `113` / `1329282` at `2026-03-22_14.44.46` but still worse than `47` / `536003` at `2026-03-22_13.22.18`.

- Observation: the remaining regression is no longer dominated by `check-current` / `install-current`.
  Evidence: the `2026-03-22_16.52.22` worker `events.jsonl` files show one `python3 tools/recipe_worker.py finalize-all scratch/` happy-path finalize per worker and no stage-wide per-task helper loop resembling the `2026-03-22_14.44.46` regression.

- Observation: two workers still paid heavily for broad `assigned_tasks.json` reads.
  Evidence: stage-wide command-output attribution for `2026-03-22_16.52.22` shows `assigned_tasks_dump` at roughly `32397` estimated output tokens, driven mostly by one `cat assigned_tasks.json` in `worker-002` and one `CURRENT_TASK* + assigned_tasks.json` dump in `worker-003`.

- Observation: the static startup surfaces are now significantly larger than the good baseline.
  Evidence: the approximate startup surface for `worker-001` rose from about `3191` estimated tokens at `2026-03-22_13.22.18` to about `7312` at `2026-03-22_16.52.22`. The largest deltas are `SHARD_PACKET.md` at about `2387` tokens and `scratch/_prepared_drafts.json` at about `2143` tokens.

## Context and Orientation

Recipe workspace workers live primarily in `cookimport/llm/codex_farm_orchestrator.py` and `cookimport/llm/recipe_workspace_tools.py`. The orchestrator writes the recipe worker prompt, worker root, helper files, and task queue, then gathers validated `out/*.json` task results back into final shard proposals. The workspace-tools module renders helper CLI text and all recipe worker-facing files such as `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`, `SHARD_PACKET.md`, and `scratch/_prepared_drafts.json`. `cookimport/llm/codex_exec_runner.py` mirrors those files into the sterile direct-exec workspace and records worker command telemetry in `events.jsonl`, `live_status.json`, and `usage.json`.

In this repository, a "recipe worker assignment" means one long-lived Codex workspace session that owns multiple recipe task rows from one shard. A "task row" is one recipe-owned JSON packet from `assigned_tasks.json`, plus its `in/<task_id>.json`, `hints/<task_id>.md`, `scratch/<task_id>.json`, and `out/<task_id>.json` files. A "startup surface" means the small set of repo-written files the worker reads before editing drafts. Right now that startup surface effectively includes `worker_manifest.json`, `SHARD_PACKET.md`, `scratch/_prepared_drafts.json`, and often `CURRENT_TASK.md` / `current_task.json` / `CURRENT_TASK_FEEDBACK.md`. A "post-finalize completion surface" means the current-task files after `python3 tools/recipe_worker.py finalize-all scratch/` succeeds; those files should immediately and unambiguously say that the queue is complete.

## Milestones

### Milestone 1: Make the startup surfaces small enough to trust

This milestone shrinks `SHARD_PACKET.md`, `scratch/_prepared_drafts.json`, and any startup-facing manifest notes so the normal startup read is materially smaller than today. At the end of this milestone, the worker should not need `assigned_tasks.json` to see task order, draft paths, or the draft inventory, and the packet/manifest pair should no longer carry large repeated candidate summaries.

The proof is a generated worker root where the startup surface is still sufficient for the common path but is measurably smaller. For a five-task worker like the saved `2026-03-22_16.52.22` recipe `worker-001`, the static startup surface should fall well below the current `7312` estimated tokens and should target `4500` or less.

### Milestone 2: Make `assigned_tasks.json` genuinely fallback-only

This milestone changes prompt wording, worker manifest notes, and possibly entry-file ordering so `assigned_tasks.json` stops looking like a normal startup read. The worker should have enough information in `SHARD_PACKET.md`, `scratch/_prepared_drafts.json`, and the current-task sidecars to finish the happy path without dumping the full task inventory.

The proof is twofold: prompt/tests now explicitly frame `assigned_tasks.json` as a queue/debug inventory only, and focused regressions fail if recipe startup guidance drifts back toward broad inventory reads. The approved benchmark rerun should stop showing the large `assigned_tasks.json` dumps seen in `worker-002` and `worker-003` from `2026-03-22_16.52.22`.

### Milestone 3: Make `finalize-all` close the queue

This milestone extends the repo-owned finalize path so a successful `python3 tools/recipe_worker.py finalize-all scratch/` not only writes `out/*.json` files but also rewrites `current_task.json`, `CURRENT_TASK.md`, and `CURRENT_TASK_FEEDBACK.md` into a completed state immediately. At the end of this milestone, the worker should not have to probe whether finalize also completed the queue; the workspace should say so plainly.

The proof is a local helper test where `finalize-all` causes `current_task.json` to disappear and both current-task sidecars to report completion without any later command. In the approved benchmark rerun, post-finalize commands should drop sharply from the `14` extra commands seen in the current `16:52` run.

### Milestone 4: Prewrite drafts that are already schema-valid and conservatively trustworthy

This milestone improves `build_recipe_worker_scaffold(...)`, `prepare_recipe_worker_drafts(...)`, and any upstream recipe-task metadata used by those functions so the repo stops prewriting obviously invalid repaired drafts. Repaired drafts should get deterministic non-empty `mr` values when the empty mapping is legitimate, and low-information recipe rows should fail closed into a deterministic terminal status instead of repaired placeholders when the source evidence is too weak to support a valid repaired object.

The proof is local regression coverage showing that prepared drafts for multi-step or multi-ingredient repaired recipes no longer fail validation solely because `mr` is missing, and that clearly weak inputs no longer prewrite repaired placeholder objects with `__EDIT_ME__` debt unless the worker truly must supply semantic content.

### Milestone 5: Lock the smaller contract in docs and tests

This milestone updates prompt/sidecar/helper tests plus the short LLM docs so the repo preserves the smaller startup contract, the completed-after-finalize behavior, and the more trustworthy prewritten drafts. The tests should make it hard to accidentally re-expand the startup packet or let `finalize-all` leave the queue in an ambiguous state.

The proof is a focused recipe regression suite that passes locally and docs that describe the smaller paved road accurately.

## Plan of Work

Start in `cookimport/llm/recipe_workspace_tools.py`, because most of the remaining spend is caused by how the repo writes the startup and helper surfaces. First, shrink `_write_prepared_recipe_worker_manifest(...)` so `scratch/_prepared_drafts.json` keeps only the fields needed for the normal path: draft directory, draft paths, current task id, task ids, output paths, and perhaps one very short title/owned-id summary per task. Remove the large `candidate_summaries` block from the normal manifest; if those summaries remain useful for debugging, move them to a separate fallback file under `scratch/` or teach the worker to use `hints/<task_id>.md` for that deeper context instead of bloating the startup manifest.

In the same module, trim `render_recipe_worker_shard_packet(...)` so it stops repeating too much per-task detail that already exists elsewhere. Keep the compact contract quick reference, queue order, active draft path, and fallback paths, but replace repeated candidate-summary prose with a much shorter per-task line. The packet should tell the worker where to go next, not restate each recipe in detail.

Still in `cookimport/llm/recipe_workspace_tools.py`, make `build_recipe_worker_scaffold(...)` and any nearby helpers prewrite drafts that already satisfy the compact validation rules whenever the source hints are sufficient. The current benchmark evidence shows workers repeatedly burning turns to add `mr` for repaired drafts that legitimately keep `m=[]`. That deterministic field should be written by the repo, not discovered by the worker. If hint/input evidence is too weak for a valid repaired object, the repo should fail closed and prewrite a conservative terminal draft rather than an invalid repaired placeholder. The plan should keep this conservative: do not invent semantic recovery here, only remove obviously deterministic schema debt.

Next, change `finalize_recipe_worker_drafts(...)` and the embedded `render_recipe_worker_cli_script()` branch for `finalize-all`. After the validated outputs are installed, the helper should call the same sidecar rewrite path used by `install-current`, but with the completed state. A successful batch finalize must remove `current_task.json` and rewrite `CURRENT_TASK.md` plus `CURRENT_TASK_FEEDBACK.md` to the “queue complete” text immediately. If the easiest way to keep the library and the embedded script in sync is to expose one small helper that finalizes outputs and then refreshes queue sidecars, do that rather than duplicating the logic in two places.

Then edit `cookimport/llm/codex_farm_orchestrator.py` and any worker-manifest-writing path it uses. The prompt already says `assigned_tasks.json` is fallback-only, but the benchmark evidence shows that this is not strong enough. Tighten the recipe wording so broad dumps of `assigned_tasks.json` look clearly off the happy path. If it helps, reorder `worker_manifest["entry_files"]` or add a short explicit note that `assigned_tasks.json` is a full inventory for debugging, not a startup read, while `SHARD_PACKET.md` and `scratch/_prepared_drafts.json` already contain the required working set.

Finally, update the short operational notes in `cookimport/llm/README.md` and `docs/10-llm/10-llm_README.md`. Keep them short. They should say that recipe’s happy path uses a small shard packet and a small prepared-drafts manifest, that `assigned_tasks.json` is fallback/debug inventory only, that `finalize-all` now closes the queue state immediately, and that prewritten drafts aim to arrive schema-valid so the worker can spend turns on semantic corrections rather than deterministic contract cleanup.
