# Line-Role Inline Labels-Only Contract Migration

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

After this change, the inline-JSON canonical line-role worker will have one unambiguous contract: the model sees compact ordered row text and returns one top-level `labels` array in the same order. The runtime, schema, prompt text, repair flow, and tests will all agree on that contract. The user-visible outcome is simpler line-role prompting, fewer bookkeeping failure modes, and lower token overhead for the active line-role LLM surface.

## Decision Log

- Decision: Treat this as a full inline-contract migration rather than a prompt-only cleanup.
  Rationale: The prompt, schema, validator, repair transport, and telemetry all participate in shaping the model's behavior. Leaving old compatibility seams in the live path would preserve the same split-brain incentives.
  Date/Author: 2026-04-10 / Codex
- Decision: Keep repo-owned final shard outputs row-shaped internally where needed, but require labels-only on live inline model turns.
  Rationale: Repo-owned outputs may still need atomic-index-bearing rows for deterministic promotion and resume logic, while the model should only own semantic labels in order.

## Outcomes & Retrospective

The migration landed as intended for the live inline line-role path. The model-facing packet is now smaller and cleaner, the schema-enforced response shape is truly `{"labels":[...]}`, repair turns use the same schema discipline as initial turns, and row-shaped live inline replies now fail into repair instead of being silently treated as acceptable output. Repo-owned final outputs still resolve back into atomic-index-bearing rows after validation, so downstream deterministic promotion remains unchanged.

One subtle discovery during implementation was that the shared validator still trimmed one obvious extra trailing row. That behavior was acceptable for repo-owned row outputs but too permissive for live labels-array turns. The final implementation keeps repair behavior strict by making label-count mismatch an explicit inline response-contract error, which preserves the clean model contract without having to redesign the taskfile/output validator surface at the same time.

## Context and Orientation

The canonical line-role system lives under `cookimport/parsing/canonical_line_roles/`. The active inline-JSON worker path is implemented in `runtime_workers.py`; it builds a structured packet, renders the prompt, calls the direct Codex runner, validates the JSON response, and optionally issues a repair follow-up. The generic line-role response validation and workspace validation helpers live in `validation.py` and `cookimport/parsing/line_role_workspace_tools.py`. The checked-in prompt-pack compatibility surfaces live in `cookimport/llm/canonical_line_role_prompt.py`, `llm_pipelines/prompts/line-role.canonical.v1.prompt.md`, and `llm_pipelines/schemas/line-role.canonical.v1.output.schema.json`.

For this plan, “inline” means the direct `codex exec` JSON turn used by `line_role_codex_exec_style=inline-json-v1`. “Taskfile” means the separate single-file worker flow where the model edits `task.json`. The migration target is the inline path. The taskfile path must keep working, but it does not need to adopt the same on-wire response shape because the repo already owns the task file structure.

## Plan of Work

First, update the inline packet shape so owned rows and neighboring context are represented as the most compact ordered text form that still preserves evidence clearly. Then update the inline prompt builder so it explicitly asks for one top-level `labels` array and never advertises row-shaped alternatives. Update the line-role output schema in `llm_pipelines/schemas/line-role.canonical.v1.output.schema.json` so the runner-enforced schema matches the prompt.

Next, refactor inline response validation in `cookimport/parsing/canonical_line_roles/runtime_workers.py` so live inline turns accept only the `labels` array contract. The runtime should zip those labels back onto the shard’s authoritative atomic-index order, validate label legality and count, and keep repair packets narrow to unresolved rows. Remove row-id and ordinal-row salvage from the active inline evaluator so the runtime no longer silently tells the model that row-shaped output is acceptable. Keep the shared workspace/taskfile validator capable of validating repo-owned row outputs, but make its labels-array path a first-class validation mode rather than a translation shim.

Then update the prompt-pack compatibility files and tests so the repo narrative matches the actual runtime. This includes the file-backed prompt template, the fake runner/test support, and the line-role docs that currently mention temporary acceptance of row-shaped inline replies. The docs should explain that deterministic code owns ordering and atomic-index projection, while the model owns only the ordered label decisions.
