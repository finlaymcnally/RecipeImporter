## Purpose / Big Picture

After this change, the knowledge grouping repair path should recover valid repair answers instead of rejecting them just because the model reused a batch-local `group_id` that was already taken by an accepted group elsewhere in the same batch. A human should be able to inspect a failed case like `saltfatacidheatcutdown.ks0002.nr`, see that the repair response itself is semantically fine, and then watch the runtime normalize those repaired group ids into a collision-free full-batch answer set so the shard validates and promotes.

## Surprises & Discoveries

- Observation: the saved repair responses are often good enough on their own subset, but still fail after merge.
  Evidence: `grouping_repair_response_01_01.json` for `saltfatacidheatcutdown.ks0002.nr` cleanly rewrites the failed proposal rows, yet the next repair packet reports `knowledge_group_mixed_tag_story` because accepted dessert rows from the original batch already own `g02`.
- Observation: the collision loop can bounce across multiple repair attempts even when semantics are stable.
  Evidence: in `ks0002`, repair attempt `01_02` renumbers dessert rows to `g01` and layering salt rows to `g02`, which then conflicts with already accepted fresh-ground-spices rows that still own `g01`; repair attempt `01_03` then collides again with another accepted group.
- Observation: this is structural, not semantic.
  Evidence: the validator treats `group_id` as a batch-local glue key for contiguous runs; the actual tag story lives in `topic_label`, `grounding`, `why_no_existing_tag`, and `retrieval_query`.
- Observation: the focused runtime suite had a hidden fake-runner mismatch unrelated to the live inline bug.
  Evidence: `tests/llm/test_knowledge_orchestrator_runtime_leasing.py` failed because the fake taskfile worker still synthesized row-level grouping answers even though same-session grouping now expects one batch-level `groups` answer.

## Decision Log

- Decision: fix the collision inside repo-owned repair merging rather than by weakening validation.
  Rationale: `knowledge_group_mixed_tag_story` is still the correct final invariant for a full batch. The bug is that repair subsets do not know which ids are already reserved by accepted rows, so the runtime must reconcile opaque group identifiers before revalidating.

- Decision: keep the fix structural and deterministic.
  Rationale: remapping opaque local ids does not change the model’s semantic grouping decisions; it only preserves them when the same label token is accidentally reused for a different accepted group elsewhere in the batch.

## Outcomes & Retrospective

Valid inline grouping repairs no longer die on `knowledge_group_mixed_tag_story` solely because a repair subset reused a reserved batch-local id from an already accepted group. The repo now renumbers merged contiguous runs onto deterministic `gNN` ids before whole-batch validation, preserving the model's topic label and grounding story while removing opaque-token collisions from the failure path.

The focused knowledge quality gate also surfaced a stale fake-runner/testing contract: same-session grouping had moved to a batch-level `groups` answer, but the fake taskfile worker still emitted row-level grouping answers and one runtime assertion still expected `example_answers[0].group_id`. Both were updated so the tests reflect the checked-in contract again.

## Context and Orientation

The knowledge grouping structured path lives in `cookimport/llm/knowledge_stage/workspace_run.py`. For one grouping batch, the runtime sends an initial structured packet, validates the response into row-keyed answers, and if needed sends bounded repair followups containing only the unresolved rows. After each repair response, the runtime merges the accepted answers and revalidates against the full original grouping task file.

The parser helpers live in `cookimport/llm/knowledge_stage/structured_session_contract.py`. That module turns a `groups` response into per-row answers keyed by `unit_id`, merges row-keyed answers, and can reapply them back onto the original grouping task file.

The validator lives in `cookimport/llm/knowledge_stage/task_file_contracts.py`. It requires every full-batch grouping answer to give each row a `group_id`, `topic_label`, and valid grounding story, and it rejects cases where the same `group_id` is used for two different stories in the same batch. That invariant is correct for a final merged batch, but repair subsets do not currently know which ids have already been accepted elsewhere in the batch.

The concrete failure evidence lives under `data/output/2026-04-12_22.46.00/.../nonrecipe_finalize/workers/.../structured_session/`. The key case is `saltfatacidheatcutdown.ks0002.nr`, where the first repair response fixes invalid proposal metadata for the failed rows but reuses `g02` for “Layering salt” while accepted rows from the initial response still use `g02` for “Salt in desserts”. The next repair packets then chase those id collisions instead of advancing the shard.

## Plan of Work

First, encode the saved collision family as a focused test. Use a compact synthetic grouping batch with one accepted group already in the batch, then merge a repair subset whose rows are semantically correct but whose `group_id` collides with that accepted group. The test should fail under the current merge/revalidate flow with `knowledge_group_mixed_tag_story`.

Next, implement a repo-owned normalization step for repair answers before whole-batch revalidation. The normalization should treat `group_id` as an opaque batch-local token. When a repaired answer introduces a story that does not match the story already reserved under that id elsewhere in the accepted batch, assign the repaired contiguous run a fresh collision-free id and reuse that remapped id consistently across the rows in that repaired run. If the repaired story matches an already accepted group, keep the shared id so contiguous same-story repairs can merge naturally.

Then wire that normalization into the structured grouping repair path in `workspace_run.py` just before the runtime reapplies merged answers onto the original grouping task file for full-batch validation. Keep the existing validator unchanged unless the implementation proves a helper belongs lower in the contract layer.

Finally, update the short docs so future agents know that grouping repair now owns collision-safe normalization for batch-local ids across subset repairs. The docs should stay brief and agent-oriented.

Acceptance means all of the following are true:

## Idempotence and Recovery

This change is safe to iterate on because it only changes repo-owned repair merge behavior, tests, and docs. If the normalization logic goes wrong, the fastest recovery check is to compare the merged full-batch answers before and after the helper: only opaque `group_id` tokens should change, while each row’s topic label and grounding story should remain intact. If semantics shift, the patch is wrong.
