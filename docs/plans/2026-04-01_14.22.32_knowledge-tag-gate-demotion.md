## Operator Intent (Structured) - DO NOT DELETE

The operator wants this plan to be a direct sequel to the new direct-batch worker contract plan and to stay tightly scoped to one behavior change: ontology tagging must become a real signal that helps classify nonrecipe blocks, not a forced paperwork exercise that happens after a block has already been called `knowledge`. If a block cannot be grounded to a real existing tag or a valid proposed tag, the system should not try to rescue the row by pressuring the worker to invent one. It should deterministically demote that row to `other` and continue. This work is explicitly in burn-the-boats mode. Do not preserve the current repair-heavy grounding behavior behind compatibility branches, legacy flags, or alternate code paths. The operator is the only user and wants one clean contract, not a migration layer.

## Purpose / Big Picture

After this change, the knowledge stage will stop treating “attach some tag” as paperwork that happens after a `knowledge` decision. Grounding will become an actual gate. A nonrecipe block should survive as `knowledge` only when the worker can supply at least one valid existing ontology tag or one valid proposed tag under an existing category. If no valid grounding survives normalization, deterministic code will demote that single block to `other`, clear the retrieval-only fields, record the demotion reason, and continue the run without asking the worker to fabricate a tag on repair.

This matters because the current system rewards rationalization. In the sampled `saltfatacidheatcutdown` run, false-positive knowledge rows were still able to pass with arbitrary tags such as `rich`, `snack`, and `pressure-cook`, all under the same vague retrieval concept “Fundamental cooking flavor techniques.” That means the current tag requirement is not acting like evidence. It is acting like a forced exercise after the real classification choice was already made.

The user-visible proof is deterministic. After implementation, targeted knowledge-stage tests will show that an edited answer with `category=knowledge` but no surviving valid tag grounding is accepted by the runtime as a demoted `other` row rather than rewritten into repair mode. Promotion and manifest artifacts will also expose how many rows were demoted by the grounding gate, so the operator can tell whether the gate is helping.

## Outcomes & Retrospective

This plan is implemented. The live knowledge classification contract now treats grounding as a real keep gate: structurally valid `knowledge` rows with no surviving existing tag or valid proposed tag are deterministically demoted to canonical `other`, and the same-session helper completes without repair when that is the only issue.

The main lesson is that the runtime needed two distinct seams, not one. The validator had to stop treating missing grounding as a repair-worthy error, and the reporting path had to preserve demotion metadata even though final shard outputs only contain the normalized semantic result. Once both seams changed together, the operator-visible artifacts started telling the truth about how often the gate fired.

## Context and Orientation

The nonrecipe knowledge stage turns outside-recipe candidate blocks into final `knowledge` or `other` ownership. In this repository, “grounding” means linking a kept `knowledge` block to one or more real ontology tags from the Cookbook tag catalog, or to a well-formed proposed tag under an existing category. Grounding is stored in the `grounding` object with `tag_keys`, optional `category_keys`, and optional `proposed_tags`.

The main files for this work are:

`llm_pipelines/prompts/recipe.knowledge.packet.v1.prompt.md` is the model-facing prompt for the packet-based knowledge stage. It already says category-only grounding is not enough and that `knowledge` rows must include `gr.tk` or `gr.pt`.

`cookimport/llm/knowledge_stage/task_file_contracts.py` builds and validates the direct worker task files. The classification validator is the first deterministic seam that sees the worker’s edited `category`, `reviewer_category`, `retrieval_concept`, and `grounding` answers.

`cookimport/llm/knowledge_same_session_handoff.py` runs the same-session transition. It calls the validation logic, rewrites repair task files, advances successful classification into grouping, and emits final shard outputs when grouping is unnecessary.

`cookimport/llm/knowledge_stage/promotion.py` summarizes accepted knowledge decisions into counts and proposal rows that later flow into `promotion_report.json`, `knowledge_manifest.json`, `09_nonrecipe_finalize_status.json`, and stage summary artifacts.

`cookimport/llm/codex_farm_knowledge_ingest.py` validates the older compact packet output contract and is still an important contract mirror even when the live path is task-file based.

`cookimport/llm/knowledge_stage/recovery.py` owns the worker instructions for the nonrecipe classification stage. Its prompt text should match the new contract exactly.

The current behavior is easy to state: if a worker marks a row as `knowledge` but leaves `grounding` empty, `validate_knowledge_classification_task_file(...)` emits `knowledge_missing_grounding`, and `transition_knowledge_classification_task_file(...)` rewrites the file into repair mode. That is what this plan changes.

The desired behavior is also easy to state: if a worker marks a row as `knowledge` but no valid grounding survives deterministic normalization, the runtime demotes that row to `other`, clears retrieval-only fields, records that the grounding gate fired, and continues without repair for that row.

## Milestones

### Milestone 1: Define the deterministic grounding gate

At the end of this milestone, the repository will have one clear rule for knowledge grounding failure. The rule is: a row may remain `knowledge` only if deterministic normalization leaves at least one valid existing tag key or one valid proposed tag. If not, the runtime rewrites that row to canonical `other` form and records the demotion.

The proof for this milestone is a targeted validation test that edits a classification task file with `category=knowledge` and empty grounding, then observes a successful transition with that row demoted to `other` instead of a `repair_required` rewrite.

### Milestone 2: Align the worker contract with the gate

At the end of this milestone, the prompt and worker guidance will stop implying that every plausible teaching block deserves a tag if the worker thinks hard enough. The instructions will say plainly that if no specific tag fits, the answer should be `other`.

The proof for this milestone is a prompt-builder or worker-fixture test that inspects the nonrecipe classification instructions and finds the new gate language in the generated prompt.

### Milestone 3: Expose demotions in reporting

At the end of this milestone, promotion and manifest artifacts will tell the operator how many rows were demoted by the grounding gate, how many knowledge rows survived with existing tags, and how many survived with proposed tags.

The proof for this milestone is a deterministic promotion test plus a stage summary test that show a non-zero demotion count when a fixture contains under-grounded `knowledge` answers.

## Plan of Work

Start in `cookimport/llm/knowledge_stage/task_file_contracts.py`. Introduce one normalization helper that takes a validated answer payload for a single classification unit and returns two things: the canonical answer that should move forward, and metadata about whether a grounding-gate demotion occurred. This helper should run after basic shape checks and grounding normalization, not before. Broken JSON shape still remains an error. The gate is only about the semantic case where the worker chose `category=knowledge` but deterministic normalization leaves no usable `tag_keys` or `proposed_tags`.

Implement the demotion policy per unit. When a row is marked `knowledge` and at least one valid tag or valid proposed tag survives normalization, keep the row as `knowledge` exactly as today. When a row is marked `knowledge` and no valid grounding survives, convert the row to canonical `other` form. The canonical example below is intentionally schema-neutral so it remains correct even after later plans remove `reviewer_category` and `retrieval_concept` from the live contract:

    {
      "category": "other",
      "grounding": {
        "tag_keys": [],
        "category_keys": [],
        "proposed_tags": []
      }
    }

If the repository still temporarily carries extra explanation fields when this plan is implemented, normalize those fields consistently with the demotion to the canonical `other` meaning. Do not make those extra fields part of the gate contract itself.

Record this in metadata with stable names such as `grounding_gate_demoted_unit_ids`, `grounding_gate_demoted_block_indices`, and `grounding_gate_demotion_reason_counts`. The implemented demotion reasons are `missing_grounding`, `category_only_grounding`, and `invalid_grounding_dropped_to_empty`.

Keep malformed answer shapes as errors. For example, `grounding.tag_keys` being a string instead of a list should still fail closed, because the worker did not produce structurally valid output. By contrast, a well-formed list that normalizes to zero usable tags should not trigger repair. It should trigger demotion. This is the key boundary.

Adjust `validate_knowledge_classification_task_file(...)` so the “knowledge but no grounding” case no longer emits `knowledge_missing_grounding` as a repair error. Instead, fold the demoted row into `validated_answers_by_unit_id` and attach the demotion metadata. If the worker supplied both valid and invalid tag entries, normalize away the invalid entries and evaluate the gate on the surviving valid set. Do not keep invalid extras as a reason to repair if at least one valid grounding survives. The operator’s goal is to make grounding a signal, not to force the worker through bookkeeping repairs.

Then update `transition_knowledge_classification_task_file(...)` so a demoted row behaves like any other validated answer. If all rows end up as `other`, the transition should complete without grouping exactly as it does today. If some rows survive as grounded `knowledge`, grouping still happens only for those surviving rows. Grouping must never see a row that was demoted by the grounding gate.

Update `cookimport/llm/knowledge_same_session_handoff.py` only as needed so its status and repair reporting do not misclassify gate demotions as validation failures. A same-session helper run that only caused grounding-gate demotions should finish as a successful validation pass, not as `repair_required`.

Tighten the worker instructions in both `llm_pipelines/prompts/recipe.knowledge.packet.v1.prompt.md` and `cookimport/llm/knowledge_stage/recovery.py`. The wording should say, in plain language, that a block is only `knowledge` if the worker can name a specific existing tag or a real proposed-tag gap that genuinely fits the exact block. If the worker cannot do that, the answer must be `other`. Proposed tags should be described as rare and only for clear catalog gaps, not as the fallback whenever no existing tag fits.

Update reporting in `cookimport/llm/knowledge_stage/promotion.py` so the operator can see the gate. The implemented counts are `grounding_gate_demoted_block_count`, `grounding_gate_demoted_after_invalid_grounding_drop_count`, and `grounding_gate_demoted_for_category_only_count`, plus `grounding_gate_demotion_reason_counts` in the richer grounding summary. These now flow through `promotion_report.json`, `knowledge_manifest.json`, `09_nonrecipe_authority.json`, and `knowledge_stage_summary.json` beside the older kept/rejected/grounded counters.

Mirror the contract in `cookimport/llm/codex_farm_knowledge_ingest.py` if needed so the older compact packet path agrees with the task-file path. The compact contract should not accept an ungrounded `knowledge` row as valid semantic output. If this path cannot demote in place cleanly, document and implement the same normalized semantic rule before parsed outputs reach promotion.

Finally, update the short package note in `cookimport/llm/knowledge_stage/README.md` so it says that grounding is now a hard keep gate and that under-grounded `knowledge` rows are demoted to `other` instead of being repaired into speculative tags.

## Validation and Acceptance

Acceptance is behavioral.

For validation and handoff:

1. If a worker answer is structurally malformed, the task still fails closed into repair mode.
2. If a worker answer is structurally valid but marks a row as `knowledge` with no surviving valid grounding, that row is demoted to canonical `other` form and the handoff succeeds without repair for that reason.
3. Grouping only receives rows that survived the grounding gate as real `knowledge`.

For prompt behavior:

1. The generated nonrecipe classification instructions tell the worker that lack of a specific tag fit means `other`.
2. The prompt no longer implies that a proposed tag should be invented whenever an existing tag does not fit.

For reporting:

1. Promotion and manifest artifacts expose how many rows were demoted by the grounding gate.
2. Existing counters for grounded knowledge rows still work.
3. A deterministic fixture that includes one under-grounded `knowledge` row produces a non-zero demotion count in the summary artifacts.

For tests:

1. `tests/llm/test_knowledge_block_classification_contract.py` includes a case where `knowledge` without surviving grounding is demoted to `other`.
2. `tests/llm/test_knowledge_grounding_contract.py` includes cases for empty grounding, unknown tag keys, and mixed valid plus invalid grounding entries.
3. `tests/llm/test_knowledge_same_session_handoff.py` proves that demotion does not trigger repair.
4. `tests/llm/test_knowledge_stage_promotion.py` or `tests/staging/test_nonrecipe_stage.py` proves that demotion counts appear in the emitted artifacts.

## Idempotence and Recovery

This change is safe to apply incrementally if the work starts in the validator and transition path and then updates reporting in the same pass. Do not leave the repository in an in-between state where validation demotes rows but reporting still counts them as retained knowledge, or where the prompt still tells workers to force tags even though the runtime now demotes them.

If a test fails because an existing fixture expects `knowledge_missing_grounding` repair behavior, update the fixture and the expectation to the new demotion behavior rather than preserving a compatibility branch. This is an intentional contract change.

If a downstream reader depends on the old summary shape, extend the artifact with additive demotion counters first and remove stale wording only after the tests prove the new numbers are correct.
