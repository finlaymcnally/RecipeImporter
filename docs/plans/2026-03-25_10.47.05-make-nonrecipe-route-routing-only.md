# Make `nonrecipe-route` Routing-Only

## Structured Restatement

The intended architecture is simple. `nonrecipe-route` should not be a second semantic decision-maker for outside-recipe text. It should take the outside-recipe rows that come out of the label stage, finalize only the obvious junk exclusions that are already justified there, and package every remaining outside-recipe row as review input for `knowledge-final`. Any subtle `knowledge` versus `other` judgment for those surviving rows should belong to `knowledge-final`, not to deterministic routing code, not to fallback category maps, and not to a mixed transitional stage object.

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

This plan is intentionally adjacent to, but narrower than, [docs/plans/4-2026-03-25_11.18.00-make-knowledge-review-packet-based-and-llm-grouped.md](/home/mcnal/projects/recipeimport/docs/plans/4-2026-03-25_11.18.00-make-knowledge-review-packet-based-and-llm-grouped.md). That later plan changes how the knowledge model groups and writes reviewed knowledge. This plan comes earlier in the authority story: it removes semantic ambition from `nonrecipe-route` itself so the knowledge stage receives a clean review queue instead of inheriting a mixed routing-and-meaning contract.

## Purpose / Big Picture

After this change, a human should be able to say one plain sentence about `nonrecipe-route`: it honors label-stage obvious-junk exclusions, packages the surviving outside-recipe rows for later review, and does nothing else. It should not behave like a second semantic classifier. It should not quietly decide subtle `knowledge` versus `other` meaning for reviewable prose. It should not teach downstream readers that deterministic routing is the same thing as semantic authority.

The user-visible gain is architectural honesty. Today the code and docs say that `knowledge-final` is the final semantic owner for outside-recipe meaning, but `nonrecipe-route` still carries enough fields and fallback behavior to feel like a partial authority stage. After this plan lands, a reviewer opening the runtime objects or artifacts should be able to answer four questions cleanly: which rows were excluded as obvious junk, which rows survived and require review, which rows were actually reviewed, and which rows have final semantic authority. The proof is a stage run whose `08_nonrecipe_seed_routing.json`, `09_nonrecipe_authority.json`, and `09_nonrecipe_review_status.json` each tell only one story, plus focused tests that show `nonrecipe-route` does not manufacture final semantic categories for reviewable rows.

This plan does not redesign the knowledge prompt itself. It does not add knowledge tags. It does not decide the eventual topic-grouped knowledge artifact format. Its job is to make the upstream contract blunt and teachable before those later knowledge-product changes land.

## Outcomes & Retrospective

The main runtime goal is now implemented. `nonrecipe-route` is boring in the good way on the live path: it surfaces an explicit routing result, packages the review queue, and leaves final semantic truth to `knowledge-final`. The main remaining gap is broader verification across the touched staging, LLM, and Label Studio surfaces.

## Context and Orientation

The post-import runtime now has five explicit stage names in `cookimport/staging/pipeline_runtime.py`: `extract`, `recipe-boundary`, `recipe-refine`, `nonrecipe-route`, and `knowledge-final`. In plain language, `recipe-boundary` decides which blocks belong to recipes, `recipe-refine` improves recipe content without changing recipe ownership, `nonrecipe-route` handles everything outside those accepted recipe spans, and `knowledge-final` is supposed to be the final semantic judge for outside-recipe meaning.

The problem is that the current in-memory nonrecipe contract still bundles several layers together. `cookimport/staging/nonrecipe_stage.py` defines `NonRecipeStageResult`, a dataclass that simultaneously contains seed spans, broad category maps, review-routing maps, exclusion reasons, final-authority indices, and unreviewed-review-eligible indices. That mixed object is then threaded through `cookimport/staging/pipeline_runtime.py`, `cookimport/staging/writer.py`, `cookimport/staging/stage_block_predictions.py`, and the knowledge runtime under `cookimport/llm/knowledge_stage/`.

In this repository, “obvious junk” means outside-recipe text that can be finalized immediately as useless `other` without waiting for later semantic review. Examples include navigation, page furniture, endorsements, marketing filler, and similar boilerplate. “Review-eligible” means outside-recipe text that survived that obvious-junk veto and therefore must remain in play for the knowledge stage. “Final authority” means the only categories downstream truth readers may trust. “Review status” means the bookkeeping about whether a review-eligible row was reviewed, skipped, changed, or left unresolved.

The current staging artifacts already split some of this on disk. `08_nonrecipe_seed_routing.json` is supposed to describe deterministic routing. `09_nonrecipe_authority.json` is supposed to contain final truth. `09_nonrecipe_review_status.json` is supposed to explain what happened during review. The reason this plan exists is that the in-memory stage contract and the surrounding prose still blur those layers together, which makes the stage feel smarter than the user wants.

## Plan of Work

Start by shrinking the in-memory nonrecipe model, not by rewriting the prose first. In `cookimport/staging/nonrecipe_stage.py`, replace the current one-big-wrapper feel of `NonRecipeStageResult` with a contract that makes the routing-only rule unavoidable. The exact class names may change, but the ownership split must be explicit. There must be a routing component that contains only review exclusions, review eligibility, block previews, and any helper metadata needed to build a later review packet. There must be a final-authority component that contains only rows the system may treat as semantically final. There must be a review-status component that contains only state about reviewed, skipped, changed, unresolved, or fallback rows.

The routing component should not expose a broad `block_category_by_index` field that looks canonical for all surviving outside-recipe rows. If a broad category map still exists internally for transition or debugging, it must be named so clearly that nobody could mistake it for truth. The cleanest landing is to have explicit fields that answer these three questions separately: which blocks were excluded forever, which blocks are queued for knowledge review, and which blocks already have final authority.

Then refactor `run_nonrecipe_route_stage(...)` in `cookimport/staging/pipeline_runtime.py`. After the refactor, this function should do only the work the user endorsed. It should read the recipe-boundary output, identify outside-recipe rows, honor label-stage obvious-junk exclusions, and build the queue for `knowledge-final`. It should return excluded-final blocks and reviewable blocks, but it should not claim that reviewable blocks are final `knowledge` or final `other`. It should stop behaving like a partial semantic adjudicator.

After that, refactor `run_knowledge_final_stage(...)` so it consumes only the explicit review queue from the routing result. The knowledge stage should never need to reinterpret a mixed routing map to discover what is reviewable. It should receive reviewable rows directly, then return explicit final authority plus explicit review-status facts. This is also the right place to remove any remaining deterministic semantic bypass for review-eligible material on the live path. If a row survived routing and is still review-eligible, then the live semantic owner must be `knowledge-final`, not a deterministic shortcut hidden in staging or knowledge planning.

With the stage runtime cleaned up, update the writers in `cookimport/staging/writer.py`. The writer surfaces for `08_nonrecipe_seed_routing.json`, `09_nonrecipe_authority.json`, and `09_nonrecipe_review_status.json` should each read from one ownership layer only. `08_nonrecipe_seed_routing.json` may list excluded blocks, reviewable blocks, exclusion reasons, and previews. `09_nonrecipe_authority.json` may list only final authority. `09_nonrecipe_review_status.json` may list reviewed/skipped/unresolved/fallback state. The writer must stop reverse-engineering certainty from one mixed object.

Once the writer split is real, update the truth readers. `cookimport/staging/stage_block_predictions.py` must continue to read only explicit final authority. `cookimport/llm/knowledge_stage/runtime.py` and related knowledge helpers must stop treating the routing result as if it already contains semantic meaning for the reviewable queue. Any compatibility projection to `ConversionResult.non_recipe_blocks` or related helpers must be narrow and visibly labeled as fallback, not authority.

Finally, update the docs after code. The files that must change are `docs/01-architecture/01-architecture_README.md`, `docs/05-staging/05-staging_readme.md`, `docs/10-llm/10-llm_README.md`, and `docs/plain-english-explanation.md`. Those docs should say bluntly that `nonrecipe-route` honors obvious-junk exclusions and packages survivors for review. They should stop describing it as if it “decides” outside-recipe meaning. If the historical label “Stage 7” is kept anywhere, the docs must explain that it is a legacy nickname for the routing seam, not the semantic authority seam.

## Milestones

### Milestone 1: Make the routing contract explicit

At the end of this milestone, the nonrecipe in-memory model will clearly separate routing, final authority, and review status. A novice opening `cookimport/staging/nonrecipe_stage.py` should no longer need to infer which fields are seed-only, which fields are final, and which are bookkeeping. The proof is focused unit tests that show one excluded block, one reviewable-but-unreviewed block, and one reviewed block all land in different components as expected.

### Milestone 2: Make `nonrecipe-route` boring

At the end of this milestone, `run_nonrecipe_route_stage(...)` will have a narrow, teachable behavior. It will identify outside-recipe rows, finalize obvious-junk exclusions, and hand the rest to the knowledge stage. It will not own semantic `knowledge` decisions for reviewable rows. The proof is that its returned data structures expose only exclusion and queue-building semantics, not mixed final category maps.

### Milestone 3: Make `knowledge-final` the only semantic owner for reviewable rows

At the end of this milestone, the knowledge runtime will accept a clean review queue and return clean final authority plus clean review status. Any deterministic semantic fallback for review-eligible rows on the live path will be gone or pushed behind an explicitly temporary migration shim. The proof is tests that fail if reviewable rows appear in final authority before the knowledge stage actually promotes them.

### Milestone 4: Cut writers and readers over to the cleaner seam

At the end of this milestone, the staging writer, stage-block predictions, and knowledge runtime will consume explicit routing, authority, and review-status data without mixed reinterpretation. The proof is a stage fixture or focused integration test where the three nonrecipe artifacts each tell one story and downstream truth readers use only the authority artifact.

### Milestone 5: Update docs so they stop teaching the wrong thing

At the end of this milestone, the architecture docs and the plain-English walkthrough will match the refactor. A reader should come away with the same understanding the user stated in discussion: `nonrecipe-route` is a router, not a semantic decider. The proof is the doc text itself plus any focused tests or fixture examples updated alongside it.

## Validation and Acceptance

The change is correct only if all of the following are true.

Routing acceptance means `nonrecipe-route` no longer manufactures final semantic meaning for reviewable rows. Focused tests must show that a reviewable outside-recipe block can survive routing without appearing in final authority until `knowledge-final` promotes it.

Exclusion acceptance means obvious-junk rows still become final immediately and never enter the knowledge review queue. Focused tests must show that navigation-like or page-furniture-like blocks land in the exclusion set, appear in final authority as final `other`, and are absent from the review queue.

Knowledge-boundary acceptance means the knowledge stage receives reviewable rows directly rather than reconstructing them from a mixed category map. A code-level proof can be a focused test or a simple interface check in `cookimport/llm/knowledge_stage/runtime.py` that fails if the old mixed-field dependency sneaks back in.

Artifact acceptance means `08_nonrecipe_seed_routing.json` answers routing questions, `09_nonrecipe_authority.json` answers final-truth questions, and `09_nonrecipe_review_status.json` answers review-process questions. No canonical reader should need to cross-read those files just to figure out whether a row is authoritative.

Prediction acceptance means `cookimport/staging/stage_block_predictions.py` continues to use only explicit final authority for outside-recipe `KNOWLEDGE` vs `OTHER`. A focused test should fail if unreviewed review-eligible rows are accidentally treated as reviewed final authority.

Documentation acceptance means the updated docs and plain-English walkthrough say plainly that `nonrecipe-route` routes and packages survivors, while `knowledge-final` is the semantic owner for reviewable outside-recipe material.

## Idempotence and Recovery

This refactor should land additively. The safest order is to introduce the cleaner in-memory split first, then change `run_nonrecipe_route_stage(...)`, then update the writers and truth readers, and only then delete or demote the old mixed convenience helpers. That order keeps the code runnable while making it easier to prove each authority boundary separately.

If a checkpoint lands where the new routing object exists but the writer still serializes mixed certainty as if it were canonical truth, stop and fix the writer before continuing. If a checkpoint lands where the knowledge stage still reads mixed routing fields after the new explicit review queue exists, stop and fix that dependency before updating docs. Those two situations would preserve the old ambiguity under new names.

If a compatibility projection is needed for old readers, mark it clearly as fallback or migration-only and keep the projection narrow. Do not add a new write-side alias or broad “helpful” merged map that would re-teach the same architecture problem this plan exists to remove.

## Artifacts and Notes

The clearest expected artifact shape after this refactor is simple:

`08_nonrecipe_seed_routing.json` should be able to say, in effect, “these blocks were excluded forever, these blocks survived routing and require review, and here are the reasons or previews that explain that routing.”

`09_nonrecipe_authority.json` should be able to say, in effect, “these are the only outside-recipe blocks downstream truth readers may trust right now.”

`09_nonrecipe_review_status.json` should be able to say, in effect, “these reviewable blocks were reviewed, these were skipped or unresolved, and these are the reason codes or counts that explain why.”

An intentionally small expected example for tests or fixture notes looks like this:

      reviewable_block_indices = [7, 8, 9]

    final_authority:
      authoritative_block_indices = [4, 5, 7, 8]

    review_status:
      reviewed_block_indices = [7, 8]
      unreviewed_block_indices = [9]

That is the architecture this plan exists to enforce. Block `9` survived routing, but it is still not final truth.
