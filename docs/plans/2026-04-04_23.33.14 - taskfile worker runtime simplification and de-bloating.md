# Taskfile Worker Runtime Simplification And De-Bloating

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the repo will still support Codex-backed worker sessions, but the expensive taskfile-worker path will stop acting like a giant general-purpose mini-shell runtime for every semantic stage. A user will be able to run the same benchmark and see the same semantic outputs and durability guarantees while the taskfile-worker stages send less repeated framing, expose a smaller worker-visible surface, and produce lower `prompt_budget_summary.json` totals on the stages that still use taskfiles.

This matters because the current system has already fixed the worst fanout failures, but the remaining cost floor is still dominated by protocol. The benchmark artifacts show that recent runs are no longer exploding to hundreds of calls, yet a small number of taskfile sessions still burn millions of counted tokens through cached input, wrapper overhead, and repeated worker-session framing. The goal of this refactor is not merely to "clean up code." The goal is to make Codex taskfile workers act like narrow semantic subcontractors instead of expensive interactive operators.

The visible proof after implementation is:

1. Running a representative benchmark such as `saltfatacidheatcutdown` still writes the same stage-owned artifacts and still scores normally.
2. `prompt_budget_summary.json` shows materially lower taskfile-stage totals for the migrated surfaces, with lower `wrapper_overhead_tokens` and less repeated worker framing.
3. The worker-visible contract is smaller and calmer: the happy path is "read one repo-written file, edit owned answers, hand off once," and only the stages that truly need same-session file editing still use taskfile mode.

## Surprises & Discoveries

- Observation: The current system has already solved the worst historical blowup, but the remaining floor is still dominated by protocol rather than visible prompt body.
  Evidence: Recent benchmark roots for `saltfatacidheatcutdown` dropped from `251` Codex calls on `2026-03-22_20.16.29` to `10` calls on `2026-03-31_21.49.39` and `2026-04-04_23.03.39`, yet the corresponding totals stayed very large (`33.70M` -> `19.80M` -> `21.85M`) because `cached_input_tokens` and `wrapper_overhead_tokens` remained dominant.

- Observation: The current shared runner already exposes the right low-level measurement seam for this refactor.
  Evidence: `cookimport/llm/codex_exec_runner.py` computes `visible_input_tokens`, `visible_output_tokens`, and `wrapper_overhead_tokens`, and `cookimport/llm/prompt_budget_runtime.py` already aggregates those into `prompt_budget_summary.json`.

- Observation: The codebase already has three partially overlapping taskfile stories rather than one clean shared one.
  Evidence: recipe owns its taskfile prompt and helper contract in `cookimport/llm/recipe_stage/task_file_contract.py`; knowledge layers shared workspace recovery plus stage-specific contracts in `cookimport/llm/knowledge_stage/workspace_run.py`; canonical line-role carries a very large stage-local session loop in `cookimport/parsing/canonical_line_roles/runtime_workers.py`.

- Observation: The repo has already moved philosophically toward "task.json is the whole job," but the implementation still pays for many historical seams.
  Evidence: `docs/10-llm/10-llm_README.md` explicitly says taskfile workers should treat `task.json` as the whole job and avoid queue-style helpers, yet the runtime still preserves a large amount of mirrored control state, stage-local retry machinery, and duplicated prompt-contract logic.

- Observation: The best savings are unlikely to come from one generic shared prompt rewrite alone.
  Evidence: recipe, line-role, and knowledge have different reasons for using taskfile mode today. Recipe still benefits from editable owned answers plus same-session repair; line-role and knowledge each have plausible thinner transports for the happy path.

  Rationale: The token waste is caused both by code duplication and by the current architectural choice to run too many semantic decisions through full taskfile sessions.   Rationale: Taskfile mode is useful when the repo needs durable editable state, same-session validation, or bounded repair rewrites. It is wasteful when the work can be expressed as one compact semantic packet with deterministic validation after the fact.

## Outcomes & Retrospective

This section is intentionally incomplete because no implementation has started yet. The intended completed state is: one thin shared taskfile runtime, a clear taskfile-versus-packet transport boundary, reduced protocol overhead on the surviving taskfile stages, and before/after benchmark evidence that the semantic outputs remained stable while the worker protocol got cheaper.

The main thing to watch during implementation is whether any stage still genuinely benefits from same-session editable files after its worker-visible contract is stripped down. If a stage does not need that property anymore, this plan expects the implementation to move it to a thinner transport rather than carrying taskfile complexity forward out of habit.

## Context and Orientation

This repository has an optional Codex execution stack layered on top of a deterministic recipe-import pipeline. A "taskfile worker" in this repo means a Codex session that starts in a sterile mirrored workspace whose visible contents are mostly a repo-written `task.json` plus `AGENTS.md`. The worker is supposed to read the task file, edit only the answer fields it owns, and then hand control back to deterministic repo code through a repo-owned handoff helper. The repo then validates the edits, records telemetry, and promotes only structurally valid outputs.

Today the shared lower-level taskfile machinery lives in these files:

- `cookimport/llm/codex_exec_runner.py` is the main direct Codex subprocess seam. It runs `codex exec`, records events and usage, computes prompt-budget telemetry, and supervises workspace sessions.
- `cookimport/llm/codex_exec_workspace.py` builds the mirrored execution workspace and syncs repo-owned control files between the source worker root and the sterile execution directory.
- `cookimport/llm/codex_exec_taskfile_policy.py` classifies worker shell commands and decides which commands are tolerated drift versus real boundary violations.
- `cookimport/llm/phase_worker_runtime.py` is an older shared shard-worker foundation that still defines common dataclasses and assignment structures used by multiple surfaces.

The stage-specific taskfile behavior is then layered separately:

- Recipe taskfile construction and recipe-specific worker prompts live in `cookimport/llm/recipe_stage/task_file_contract.py`.
- Knowledge classification and grouping still own a large amount of workspace-run and retry logic in `cookimport/llm/knowledge_stage/workspace_run.py`.
- Canonical line-role taskfile sessions are orchestrated in `cookimport/parsing/canonical_line_roles/runtime_workers.py`.

In plain language, the current problem is that these three surfaces all use the same expensive kind of worker session, but each stage still carries its own partially duplicated version of prompt construction, retry policy, output discovery, same-session completion proof, and telemetry packaging. That makes the code hard to evolve and also keeps the worker-visible contract larger than it needs to be. The benchmark artifacts show that the repo is no longer suffering from hundreds of Codex calls, but it is still paying a large per-session protocol tax.

For this plan, "packet transport" means a thinner Codex call where the repo sends a compact structured input and receives one compact structured answer without exposing an editable `task.json` workspace. "Taskfile transport" means the current file-edit session contract with a mirrored workspace and same-session handoff. "Happy path" means the normal successful run without repair or catastrophic worker failure. "Protocol overhead" means counted tokens that come from session framing, wrapper text, helper chatter, or other non-semantic transport cost rather than the minimum semantic payload needed to make the decision.

## Plan of Work

Start by building a hard baseline before changing architecture. Pick one representative benchmark root that still shows the current taskfile cost floor, ideally the recent `saltfatacidheatcutdown` runs already used for cost diagnosis. Capture the whole-run `prompt_budget_summary.json`, the stage-specific prompt rows in `prompts/full_prompt_log.jsonl`, and the worker-local prompt plus usage artifacts for recipe, line-role, and knowledge or nonrecipe-finalize. Write one small deterministic comparison helper if needed, but keep it repo-owned and narrow. The purpose of this first step is to make later claims precise: which stage got cheaper, how much of the old spend was `wrapper_overhead_tokens`, and whether the savings came from lower prompt body size, fewer helper turns, or a transport change.

Then extract one explicit shared taskfile runtime package under `cookimport/llm/`. A good target shape is a small new package such as `cookimport/llm/taskfile_runtime/` with owner modules for prompt contract assembly, workspace session execution, same-session completion proof, retry and replacement policy, and telemetry packaging. This package should own only the truly cross-stage taskfile behavior. It must not own stage semantics. Stage semantics stay in recipe, knowledge, and line-role packages; the new shared runtime owns how a taskfile session is run and observed.

The first concrete code move in that extraction is to define a stage-neutral taskfile session spec. The spec should include the stage key, worker-visible instructions, answer schema, helper surface, output contract, same-session state paths, and the stage-owned callbacks needed to validate outputs or build repair rewrites. The important change is that recipe, line-role, and knowledge should all hand the shared runtime a declarative session spec instead of each implementing their own bespoke session loop. This spec should be explicit enough that a reader can tell, from one object, whether a stage genuinely needs taskfile mode or whether it could have been a packet call instead.

After that shared spec exists, strip duplicated prompt and helper-contract text out of the stage-local modules. `cookimport/llm/recipe_stage/task_file_contract.py`, `cookimport/llm/knowledge_stage/task_file_contracts.py`, and the line-role taskfile builder in `cookimport/parsing/canonical_line_roles/runtime_workers.py` should all emit the same general taskfile shape: one repo-written file, one editable pointer pattern, one stage-owned answer schema, one small stage-owned hint surface, and no extra helper prose unless a stage truly needs it. The shared runtime should own the common instruction language. Stage modules should only supply the semantic rules specific to their output schema.

Next, split transport choice from stage semantics. Introduce one small transport decision layer, likely in `cookimport/config/run_settings.py` plus a shared `cookimport/llm/` transport selector, that can choose between taskfile and packet or inline JSON mode per stage. The runtime must stop treating taskfile sessions as the natural default for every Codex-backed semantic decision. Instead, the implementing code should ask a narrower question: does this stage need same-session editable durable state, or does it only need one validated semantic answer?

The first target for migration off the heavy taskfile contract should be canonical line-role. Its outputs are classification-like, ordered, and already validated deterministically. The plan expects the implementation to prototype a packet or inline JSON line-role happy path that preserves existing authoritative outputs and current fail-closed behavior. Keep the current taskfile implementation available during the migration behind an explicit transport switch so the repo can compare outputs and costs on the same fixtures. The acceptance goal for this milestone is not just "line-role works." The acceptance goal is that line-role can run through a thinner transport with the same or better validation behavior and lower stage-local cost.

The second target is knowledge or nonrecipe-finalize. Knowledge is currently split into classification and grouping. That is useful because it means the classification happy path is a strong candidate for packet transport even if grouping still benefits from a more stateful session. The implementation should make classification the default thin path and keep taskfile mode only where grouping or same-session repair still needs an editable file. This split must preserve the current deterministic authority boundary: repo code packages the owned evidence, Codex makes the fuzzy keep or drop or grouping call, and deterministic code validates the result structurally without trying to second-guess semantics.

Recipe is the last transport target, and the plan deliberately keeps it on taskfile mode in the first implementation. However, recipe still needs a large simplification pass. The current recipe prompt and helper contract in `cookimport/llm/recipe_stage/task_file_contract.py` should be rewritten to rely on the new shared taskfile session spec, with only recipe-specific answer rules layered on top. The worker-visible file should remain a compact owned answer ledger, not a semi-interactive helper protocol. Repair mode should still exist, but the shared runtime should own its mechanics.

Once the transport split is working, simplify workspace state and retry logic. Today line-role and knowledge in particular carry large stage-local loops for fresh-session retry, fresh-worker replacement, and same-session completion proof. The refactor should move the common state machine into the shared taskfile runtime and make each stage supply only the pieces that differ: how to prove output completeness, how to assess repairability, and how to serialize outputs. This part of the refactor is where most of the code deletion should happen.

At the same time, tighten observability instead of losing it. The current cost and pathology fields are valuable. The shared runtime should keep publishing `prompt_budget_summary.json`, `live_status.json`, worker `usage.json`, and the same stage-level `token_usage_status` and pathology counts. The difference is that the telemetry should now also make the protocol-versus-semantic split explicit. Each taskfile or packet stage summary should report visible semantic payload size separately from protocol overhead so cost review becomes easier after the refactor instead of harder.

Finally, remove the old stage-local session loops and historical compatibility seams only after the new shared runtime and transport split are proven by tests and by at least one real benchmark comparison. The final code should leave a novice with one clear story: packet transport for narrow semantic calls, taskfile transport for the smaller set of stages that truly need editable state, and one shared runtime for the surviving taskfile sessions.

## Milestone 2: Extract One Shared Taskfile Session Runtime

In this milestone, the repo gains one shared taskfile session runtime package that can run a stage-defined taskfile session without the stage re-implementing the session loop. At the end of the milestone, recipe, knowledge, and line-role all still behave the same from the outside, but they call through one shared session runner and one shared taskfile session spec instead of each owning a large bespoke loop.

The acceptance proof is focused test coverage showing that each stage can still:

- create its worker-visible task file,
- run a happy-path worker session,
- detect same-session completion,
- perform one bounded fresh-session retry or replacement when appropriate, and
- write the same durable outputs and telemetry as before.

## Milestone 3: Normalize The Worker-Visible Taskfile Contract

In this milestone, the worker-visible contract becomes small and consistent across stages. At the end of the milestone, taskfile workers see one common style of instruction and one common style of repo-owned handoff surface. Stage-local code still owns answer schemas and semantic rules, but it no longer owns giant prompt essays or ad hoc helper wording.

The acceptance proof is that the worker `prompt.txt` and `task.json` artifacts for recipe, line-role, and knowledge all show the same basic structure and are visibly shorter. The benchmark cost summary for a fixture-backed run should show lower visible prompt body size for any stage still on taskfile mode, even before transport migration starts.

## Milestone 4: Move Canonical Line-Role To A Thinner Happy-Path Transport

In this milestone, canonical line-role gains a packet or inline JSON happy path and stops depending on a heavy taskfile session by default. At the end of the milestone, the repo can run canonical line-role through a thinner transport, still validate outputs deterministically, and still fail closed on malformed or incomplete results.

Keep the old taskfile path behind a switch until the thin path is proven. Acceptance is a before-and-after comparison on the same fixture where line-role outputs remain valid and the line-role stage cost in `prompt_budget_summary.json` decreases materially.

## Milestone 5: Split Knowledge Into Thin Classification And Explicit Stateful Follow-Up

In this milestone, knowledge or nonrecipe-finalize is no longer one blanket taskfile story. At the end of the milestone, classification uses the thinner transport by default, while grouping or repair can still escalate into a bounded stateful path if needed.

Acceptance is that knowledge classification still writes the same promoted outputs and status artifacts, but the classification happy path shows fewer taskfile sessions or none at all. If grouping still needs taskfile mode, the cost summary should make that split explicit.

## Milestone 6: Shrink Recipe Taskfile Mode To The Minimum Durable Surface

In this milestone, recipe remains on taskfile mode but sheds the remaining historical ballast. At the end of the milestone, recipe worker sessions still support owned editable answers and same-session repair, but the stage relies entirely on the shared runtime and the worker-visible contract is a small answer ledger plus recipe-specific semantic rules.

Acceptance is that recipe worker tests still pass, recipe promotion artifacts remain stable, and recipe-stage prompt and overhead metrics are lower than the baseline captured in Milestone 1.

## Milestone 7: Remove Old Session Loops And Re-Document The Runtime

In this milestone, the old duplicated session loops are deleted and the docs are rewritten around the new transport boundary. At the end of the milestone, a novice can find the runtime story in one place: which stages use packet transport, which stages still use taskfile transport, which shared module runs the surviving taskfile sessions, and which artifacts prove cost and correctness.

Acceptance is that the old stage-local session loops are gone, the docs match the code, and the repo has a representative benchmark comparison showing lower taskfile protocol cost without lost outputs.

## Validation and Acceptance

Validation is both behavioral and cost-aware.

First, behavioral validation: the refactor is acceptable only if recipe, line-role, and knowledge still produce valid stage-owned outputs under deterministic validation. That means the existing output artifacts and manifests still exist, the stage summaries still reconcile, and the benchmark helpers still read the finished runs without new compatibility shims.

Second, transport validation: each migrated stage must prove that its new transport is thinner in a way the operator can see. For taskfile stages that remain, compare `prompt.txt`, `full_prompt_log.jsonl`, and `prompt_budget_summary.json` before and after. For stages moved to packet or inline JSON transport, compare stage-local call counts and prompt-budget totals against the baseline.

Third, acceptance for the whole plan:

- A representative benchmark run still completes and scores normally.
- The surviving taskfile stages show lower `wrapper_overhead_tokens` than the baseline for the same source and comparable settings.
- At least one current taskfile stage has been moved to a thinner transport or justified explicitly as still needing taskfile mode.
- The codebase has one shared taskfile runtime story instead of three separate session-loop implementations.
- The docs explain the new transport boundary and point maintainers to the correct owning modules.

## Idempotence and Recovery

This refactor should be implemented additively and in slices. Keep old and new transports side by side while migrating a stage, and only remove the old path after fixture-backed tests and artifact-based benchmark comparisons prove the new path. That makes retries safe: if one migration stalls, the implementing agent can switch the stage back to the old transport while keeping the shared runtime extraction work.

When editing worker-runtime code, do not delete current telemetry artifacts until the replacement writes equivalent or better evidence. If a new thin transport cannot yet prove the same fail-closed behavior as the current taskfile path, leave that stage on taskfile mode and record the reason in this ExecPlan rather than forcing a premature migration.

If a migration breaks prompt-budget accounting, stop and restore the stage-local cost signals before continuing. This refactor is partly about cost reduction, so losing observability would invalidate the plan.
