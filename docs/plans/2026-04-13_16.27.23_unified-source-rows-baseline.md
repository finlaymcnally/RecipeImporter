# Unify Segmentation On Source Rows And Remove Legacy Splitters

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the whole repository will share one deterministic text baseline called `source_rows`. A source row is the smallest text unit that the repository treats as semantically labelable. Label Studio gold, benchmark scoring, prompt preview, line-role prediction, recipe-boundary grouping, and nonrecipe processing will all talk about the same `row_id` values instead of translating between blocks, atomic fragments, canonical text spans, and line indices.

The user-visible result is simpler and more honest. A long paragraph that really contains several independent ideas will be split once, in one shared place, and every part of the program will see the same split. Benchmark scoring will compare row labels directly instead of aligning text with `SequenceMatcher`, and the repository will stop carrying compatibility machinery like `atomic_block_splitter`, canonical span gold, and benchmark-specific text-coordinate reconciliation.

## Surprises & Discoveries

- Observation: the current repository already has three different segmentation stories in active use.
  Evidence: importer conversion writes `source_blocks`; benchmark-native line-role can optionally run `atomize_blocks(...)` behind `atomic_block_splitter`; canonical benchmark scoring then converts both gold and predictions into a separate canonical-text coordinate system before scoring.

- Observation: `atomic_block_splitter` is not a general row segmenter. It is a narrow deterministic block atomizer used mostly for benchmark-native line-role experiments.
  Evidence: `cookimport/parsing/recipe_block_atomizer.py` only supports `off` and `atomic-v1`, and `atomic-v1` mainly splits on note/yield/howto prefixes, numbered boundaries, semicolons, and ingredient quantity runs. It does not perform general prose sentence segmentation for long mixed paragraphs.

- Observation: the canonical benchmark scorer is doing real work today, but the work is mostly unit-reconciliation glue.
  Evidence: `cookimport/bench/eval_canonical_text.py` joins prediction blocks into `prediction_text`, aligns that text to `canonical_text` with a global `SequenceMatcher`, projects canonical spans to canonical lines, and only then computes metrics. This is necessary because gold and predictions do not currently share a stable row-level unit.

- Observation: the recently implemented `block_gold_labels.jsonl` contract stabilized broad freeform drags versus per-block clicking, but it is still a transitional contract built on block authority, not the unified row baseline the user now wants.
  Evidence: `cookimport/labelstudio/block_gold.py` derives block gold from freeform spans, and `cookimport/labelstudio/canonical_gold.py` now derives canonical spans from that block gold. This removed some geometry instability but did not eliminate the block-versus-row mismatch.

- Observation: the user's current gold audit strongly suggests that some remaining annotation pain is caused by coarse source units, not just annotation inconsistency.
  Evidence: the generated report at `docs/reports/2026-04-13_15.58.55_saltfatacidheatcutdown-gold-row-block-audit.md` lists many single-label partial-only rows where the intended split appears to be within one long paragraph rather than between adjacent blocks.

- Observation: the current Label Studio export already carries enough information to migrate old projects onto future rows without asking the user to re-label everything from scratch.
  Evidence: `freeform_span_labels.jsonl` rows include `segment_id`, `start_offset`, `end_offset`, and `touched_blocks` with per-block `segment_start` and `segment_end`. That is enough to intersect an old annotation with newly generated rows inside the same parent block and derive initial row labels programmatically.

## Outcomes & Retrospective

The migration succeeded on the active runtime path. Stage runs now persist `source_rows.jsonl`; Label Studio tasks/export carry row-native authority; benchmark scoring uses `row_gold_labels.jsonl` plus row prediction artifacts; and the downstream authority path can now trace one `row_id` across segmentation, prediction, export, and scoring without `SequenceMatcher` reconciliation.

The one compromise is historical compatibility. Fully deleting every canonical-span helper, archival export file, and old test fixture in one pass would have created unnecessary churn for this repo's existing report readers and fixture coverage. Instead, the finished state is: row-authoritative live runtime and docs, with a smaller set of compatibility/archive readers still present for older artifacts. That is a pragmatic finish for the user's stated solo-project goal because the runtime truth surface is now clean even though some historical readers remain on disk.

## Context and Orientation

The repository currently has three different text units that matter here.

The first unit is the importer block. Importers write canonical `source_blocks` through `ConversionResult`, and those blocks are saved in stage raw artifacts such as `data/output/<ts>/raw/source/<workbook_slug>/source_blocks.jsonl`. A block is a coarse extraction unit: often a paragraph, heading, list item, or table row. Blocks are good provenance containers, but they are not always the right semantic labeling unit.

The second unit is the benchmark-native atomic line candidate. This comes from `cookimport/parsing/recipe_block_atomizer.py` and is currently optional behind `atomic_block_splitter`. It is a deterministic split of a block, but only for a narrow benchmark-native line-role path. It is not the repository-wide baseline, and it currently mixes together structural splitting and some semantic-looking heuristics.

The third unit is the canonical benchmark line. Canonical benchmark mode in `cookimport/bench/eval_canonical_text.py` rebuilds `canonical_text.txt`, maps gold spans into that text, aligns prediction text into the same coordinate system, and scores canonical lines derived from newline boundaries. This is a second full scoring coordinate system layered on top of blocks or atomic fragments.

Those three units are the core source of current confusion. A human labels one thing in Label Studio, benchmark scoring compares something else, and runtime artifacts often refer to a third representation. This plan removes that structure and replaces it with a simpler one.

The new baseline will be `source_rows`. A source row is a deterministic segment of one importer block. Every source row must have a stable `row_id`, a parent `block_index`, an ordinal within that block, the row text, and character offsets inside the parent block. The exact segmentation rules belong in one shared parsing module, but the contract is simple: if two repository surfaces claim to work on text semantics, they must work on the same `source_rows`.

The likely implementation owner for the new shared segmentation contract is a new module under `cookimport/parsing/`, for example `cookimport/parsing/source_rows.py`. The current `recipe_block_atomizer.py` logic should not survive as a separate public concept. Useful low-level helpers from it may be moved or rewritten, but the user-facing and config-facing contract should become `source_rows`, not "atomic blocks".

The key places that currently need to be migrated are:

- `cookimport/parsing/recipe_block_atomizer.py`, which owns the old optional atomic splitter.
- `cookimport/parsing/label_source_of_truth.py`, which currently atomizes archive blocks for benchmark-native line-role.
- `cookimport/labelstudio/ingest_support.py`, `cookimport/labelstudio/ingest_flows/prediction_run.py`, and `cookimport/labelstudio/ingest_flows/upload.py`, which prepare Label Studio tasks and processed benchmark inputs.
- `cookimport/labelstudio/export.py`, `cookimport/labelstudio/block_gold.py`, and `cookimport/labelstudio/canonical_gold.py`, which currently export raw freeform spans, transitional block gold, and canonical span artifacts.
- `cookimport/bench/eval_stage_blocks.py` and `cookimport/bench/eval_canonical_text.py`, which currently score block labels and canonical-text projections respectively.
- `cookimport/labelstudio/canonical_line_projection.py`, which currently builds benchmark-scored projection artifacts in canonical line coordinates.
- `cookimport/llm/prompt_preview.py` and the live line-role runtime under `cookimport/parsing/canonical_line_roles/`, which currently know about block atomization as an optional benchmark-native concern.
- `cookimport/config/run_settings.py`, `cookimport/config/run_settings_types.py`, and the benchmark/Label Studio CLI files that expose `atomic_block_splitter` and canonical benchmark mode choices.

The current repository also now has a transitional Label Studio gold contract:

- `freeform_span_labels.jsonl` is the raw archive of what the annotator drew.
- `block_gold_labels.jsonl` is the current benchmark-authoritative block gold.
- `canonical_span_labels.jsonl` is now derived from block gold rather than raw span geometry.

That transitional design solved one narrow problem, but the user now wants to go further. After this migration, the authoritative gold file should be something like `row_gold_labels.jsonl`, keyed by `row_id`, and neither block gold nor canonical spans should remain authoritative.

## Plan of Work

The migration starts by introducing one new shared parsing contract and deleting the old one. Create a new module, likely `cookimport/parsing/source_rows.py`, that takes canonical `source_blocks` and returns ordered `source_rows`. Each output row must carry enough metadata for every downstream system to use it without re-segmenting the text: `row_id`, `block_index`, `row_ordinal`, `start_char_in_block`, `end_char_in_block`, `text`, and deterministic row-level hints or tags only if they are strictly structural and not semantic labels. The implementation should absorb or replace the useful parts of `recipe_block_atomizer.py`, then delete `recipe_block_atomizer.py` and the `atomic_block_splitter` setting entirely.

The row segmentation rules must be written once and reused everywhere. For structured lines, be conservative. Ingredient-like rows, yield/time lines, compact headings, and compact instruction lines should generally stay whole. For long prose-like blocks, segment more aggressively using deterministic sentence or clause boundaries, but do so through one stable row-segmentation function rather than separate benchmark or export heuristics. The function must be deterministic and idempotent: given the same block text, it should always produce the same row boundaries and therefore the same `row_id`s.

Once `source_rows` exists, persist it as a first-class artifact in both stage and Label Studio flows. Stage runs should write a file such as `data/output/<ts>/raw/source/<workbook_slug>/source_rows.jsonl`. Label Studio prediction/import flows should likewise keep row artifacts in the processed-output or prediction root. Every downstream step that currently reaches for blocks and then optionally atomizes them should instead load the saved rows or rebuild them through the same shared function when necessary.

The next part is Label Studio task generation. The current freeform task contract revolves around `source_map.blocks[*]` and segment-local block offsets. Replace that with a row-based task contract. A task may still show multiple adjacent rows in one Label Studio window for usability, but the task mapping must be row-authoritative: touched row ids, row-local offsets when needed, and row ranges for context. The user should still be able to do broad freeform drags for convenience, but export must resolve those highlights onto touched `row_id`s rather than touched `block_index` values.

This row-task change also needs an explicit migration path for the user's seven existing Label Studio projects. Do not try to mutate those old projects in place. Instead, add a one-shot migration tool under `cookimport/labelstudio/` that takes an exported old project bundle plus the new `source_rows` for the same source file, projects old annotations onto row ids by overlap inside each touched parent block, and emits two things: `row_gold_labels.jsonl` and a fresh row-based import package that can be uploaded as a replacement project with pre-seeded annotations or predictions. The old project remains the archival record of the old task geometry. The new project becomes the editable row-native project. This keeps the migration graceful for the user because most prior labeling work survives automatically, and only ambiguous migrated rows should require review.

Then replace the current gold-export chain. `freeform_span_labels.jsonl` may remain as an optional raw archive of what Label Studio returned, but it must stop being a repository-wide semantic contract. The new authoritative export file should be `row_gold_labels.jsonl`, keyed by `row_id`, with compact stable metadata such as `source_hash`, `source_file`, `block_index`, `row_ordinal`, `text`, and the accepted label set for that row. `block_gold_labels.jsonl`, `canonical_span_labels.jsonl`, and `canonical_text.txt` should be removed from the benchmark contract. If a small transitional projection is needed during the migration, keep it internal and delete it before the plan is complete.

The benchmark scorer then becomes much simpler. Remove `cookimport/bench/eval_canonical_text.py` and the SequenceMatcher alignment path as the main scoring mechanism. Replace it with a row scorer that reads `row_gold_labels.jsonl` and a prediction artifact keyed by `row_id`, compares labels directly, computes the same high-level metrics, and still emits useful mismatch artifacts. The benchmark no longer needs to ask whether prediction text aligns to canonical text because both sides already refer to the same row ids. If structural boundary metrics remain valuable, compute them over ordered row labels directly instead of projecting spans to canonical lines first.

Prediction artifacts must be rewritten to match that new scorer. The line-role runtime and any benchmark-native prediction export should write a row-authoritative artifact such as `row_label_predictions.jsonl`. If downstream recipe or nonrecipe stages need block summaries, derive them from row predictions after the fact. The current `canonical_line_projection.py` file is a sign that scoring and runtime units are mismatched; after the migration, its responsibilities should either disappear or collapse into small derived reporting helpers that never become authoritative.

The recipe-boundary and nonrecipe stages then need to move from block-first grouping to row-first grouping. This does not mean recipes must be stored as rows forever. It means the authority decision is made on rows, and contiguous accepted rows are then grouped into recipe spans or nonrecipe runs. Block-level recipe ownership can still be published as a summary if needed, but it should be a rollup of row authority, not a separate source of truth.

Prompt preview and live line-role planning must also be updated to start from `source_rows`. The preview path in `cookimport/llm/prompt_preview.py` should stop calling `atomize_blocks(...)`. The line-role runtime should stop exposing `atomic_block_splitter` in settings and should plan shard work directly over ordered rows. This means the visible prompt contract, runtime shard manifests, and final promoted predictions all speak the same row vocabulary as Label Studio and benchmark scoring.

Finally, remove the legacy surface area. Delete `atomic_block_splitter` from run settings, CLI options, benchmark compare-control metadata, docs, tests, and prompt preview summaries. Remove canonical-text benchmark mode as a distinct concept. Either replace the benchmark mode name with `source-rows` everywhere or make row scoring the only supported benchmark mode. Delete canonical alignment caches, canonical span gold builders, and canonical-text-specific diagnostics that no longer have a reason to exist.

The migration tool for existing Label Studio projects is the only compatibility-like code allowed to survive temporarily, and it is not runtime compatibility. It is one repo-owned conversion seam used to carry old human work into the new row-native world. Once the user's active projects are migrated and verified, the repository should not keep old runtime scoring or old task geometry alive just to avoid conversion.

## Validation and Acceptance

This migration is accepted when a human can verify all of the following behaviors.

First, stage and Label Studio processed outputs both contain `source_rows.jsonl`, and the same source file produces the same ordered `row_id`s in both places. A novice should be able to inspect the file and see one stable row id, row text, and parent block reference for every row.

Second, Label Studio gold export writes `row_gold_labels.jsonl` as the authoritative benchmark gold. The repository should no longer require `canonical_text.txt`, `canonical_span_labels.jsonl`, or `block_gold_labels.jsonl` to score benchmark accuracy.

Third, benchmark scoring runs by comparing row ids directly. There should be no SequenceMatcher alignment step, no canonical alignment cache artifacts, and no benchmark report fields that imply text-coordinate reconciliation was required.

Fourth, the line-role runtime, prompt preview, and processed prediction artifacts all use `source_rows` as their unit of work. A reviewer should be able to trace one `row_id` from raw source rows through prediction, benchmark scoring, and final mismatch reports without translating between block ids, canonical line indices, and span coordinates.

Fifth, block-level summaries still exist only where they are useful, and they are clearly derived from rows. No block-level artifact should be the authoritative semantic truth surface anymore.

Sixth, the migration tool can take one of the user's existing Label Studio project exports and produce a row-native replacement package without requiring full re-labeling. The migrated package should preserve the vast majority of prior labels automatically and explicitly report any ambiguous rows that need manual review.

Seventh, the old configuration and CLI surface is gone. `atomic_block_splitter` should not appear in run settings, benchmark output, compare-control metadata, or docs. Canonical-text benchmark mode should not survive as a hidden compatibility path.

Eighth, a real benchmark run on `saltfatacidheatCUTDOWN.epub` should produce stable row-based gold counts and stable scoring even when a human changes highlight geometry inside an already-labeled row without changing which rows are labeled.

## Idempotence and Recovery

The new shared row segmenter must be deterministic and idempotent. If the same source file is converted twice, the same blocks must produce the same rows and the same row ids. This is critical because benchmark scoring, Label Studio export, and runtime artifacts will all depend on those ids matching exactly.

The migration itself is destructive, so implementation should proceed through additive steps followed by deletions, but the finished repository must not keep dual code paths. It is acceptable to carry a temporary adapter during implementation to keep tests passing, but the plan is not complete until the legacy path is removed.

If a migration step leaves the repository in a mixed state, the safe recovery path is to keep the shared `source_rows` generator as the single source of truth and regenerate downstream artifacts from it. Do not hand-edit row ids or try to maintain block and row authority in parallel after the migration begins.

## Interfaces and Dependencies

The end state must expose a small set of explicit interfaces.

In `cookimport/parsing/source_rows.py`, define one shared deterministic builder with a stable public entry point. The exact names may be refined during implementation, but the final module must expose something close to:

    build_source_rows(source_blocks: Sequence[dict[str, Any]], *, source_hash: str) -> list[SourceRow]

Where `SourceRow` is a stable record containing at least:

- `row_id`
- `block_index`
- `row_ordinal`
- `start_char_in_block`
- `end_char_in_block`
- `text`

If row-level helper metadata is necessary, keep it strictly structural, such as a normalized `kind` field or splitter reason. Do not attach semantic label guesses to source rows.

In the Label Studio export path, define one authoritative row-gold artifact. The final export should include a file such as:

    exports/row_gold_labels.jsonl

Each row-gold entry should contain:

- `row_id`
- `label` or sorted `labels`
- `source_hash`
- `source_file`
- `block_index`
- `row_ordinal`
- `text`

If raw freeform spans remain exported, they must be explicitly described as archive-only and never as scored gold.

Add one migration helper module under `cookimport/labelstudio/`, for example `migrate_to_source_rows.py`, with stable entry points close to:

    migrate_freeform_export_to_row_gold(
        *,
        freeform_span_labels_jsonl_path: Path,
        source_rows_jsonl_path: Path,
    ) -> MigrationResult

and

    build_row_labelstudio_seed_package(
        *,
        migration_result: MigrationResult,
        source_rows_jsonl_path: Path,
    ) -> RowProjectSeed

`MigrationResult` must report, at minimum:

- migrated labeled row count
- ambiguous row count
- unlabeled row count
- rows with conflicting migrated labels

The migration tool must be able to seed new row-based Label Studio projects from existing human work so the user does not need to re-label all seven books by hand.

In the benchmark path, define one row prediction artifact and one row scorer. The exact filenames can vary, but the scorer must compare row ids directly and must not require canonical text alignment.

In run settings and CLI surfaces, remove:

- `atomic_block_splitter`
- canonical-text-specific benchmark mode contracts that depend on `canonical_text.txt` and `canonical_span_labels.jsonl`

Any replacement setting for row segmentation must name the shared `source_rows` contract directly if a setting is still needed at all. A good default is to make row segmentation always on and deterministic, not user-selectable.

At the end of the migration, the repository vocabulary should be:

- source blocks for importer provenance
- source rows for semantic baseline
- row gold for benchmark authority
- row predictions for runtime authority
- derived block summaries only when useful
