# Remove Line-Role `reason_hints` From Worker `task.json`

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

Canonical line-role already stopped exposing `deterministic_hint_label` in the worker-visible `task.json`, but the same file still exposes `reason_hints`. In the live run artifacts those hints include values such as `nonrecipe_excluded`, `fallback_decision`, and `deterministic_unresolved`. Those are repo-authored semantic judgments, not neutral structure. They create a softer version of the same shortcut: the worker can still copy the repo's opinion instead of deciding from the row text and nearby context.

After this change, canonical line-role workers should see only raw row text plus stable provenance in the happy-path `task.json`. Repo-owned debug artifacts may still keep deterministic escalation detail for operators and postmortems, but the model-facing file should not.

## Context and Orientation

Canonical line-role is the CodexFarm stage that labels each candidate row with a structural role such as `INGREDIENT_LINE`, `INSTRUCTION_LINE`, or `NONRECIPE_CANDIDATE`. The live worker contract is a single editable `task.json` written by repo code. The worker edits only `/units/*/answer`, then the repo-owned same-session helper validates and expands those answers into shard outputs.

The relevant code path is:

- `cookimport/parsing/canonical_line_roles/runtime.py`: writes worker-visible `task.json`.
- `cookimport/parsing/canonical_line_roles/planning.py`: writes the worker prompt and worker hint sidecar.
- `cookimport/parsing/canonical_line_roles/policy.py`: computes deterministic baseline and `escalation_reasons`.
- `tests/parsing/test_canonical_line_roles.py`: task-file and workspace-mirror regression tests.
- `docs/10-llm/10-llm_README.md`: current worker-contract documentation.
- `docs/04-parsing/04-parsing_readme.md`: parsing-system reference.

Today `runtime.py` writes this worker-visible evidence shape for each unit: `atomic_index`, `block_id`, `text`, `within_recipe_span`, and `reason_hints`. The first four are mechanically true context. `reason_hints` is not. It is currently a pass-through of deterministic `escalation_reasons`, which is why worker task files contain values like `nonrecipe_excluded`.

## Plan of Work

Do not remove `escalation_reasons` from repo-owned debug artifacts unless a separate task explicitly wants that. The goal here is to stop leaking them into the model-facing happy path, not to blind operators.

Update `tests/parsing/test_canonical_line_roles.py` where the task-file payload is inspected. The assertions should now prove that line-role worker-visible evidence contains only the raw-text-first fields: `atomic_index`, `block_id`, `text`, and `within_recipe_span`. Make the assertion exact enough that future answer-shaped hint fields fail loudly. This is now done with exact evidence-shape assertions.

Update `docs/10-llm/10-llm_README.md` and, if needed, `docs/04-parsing/04-parsing_readme.md` so the current contract is explicit: line-role happy-path `task.json` does not expose deterministic label hints or deterministic reason hints to the worker. This is now done, and the local package note in `cookimport/parsing/canonical_line_roles/README.md` was updated too.
