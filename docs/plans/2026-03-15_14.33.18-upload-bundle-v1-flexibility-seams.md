# Decouple upload_bundle_v1 From Current Artifact Topology

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan is intentionally pre-refactor work. The goal is not to redesign cookbook outputs or benchmark scoring. The goal is to make `upload_bundle_v1` flexible enough that a large refactor of cookbook bundling and staging can happen behind a stable reviewer-bundle boundary.

## Purpose / Big Picture

After this change, `upload_bundle_v1` should no longer understand the current benchmark session layout as its primary source of truth. Instead, it should consume one normalized, topology-neutral input model that describes what was observed in a run: runs, comparisons, diagnostics, telemetry, prompt traces, and compatibility aliases. That means the upcoming cookbook refactor can change how artifacts are bundled and staged without forcing repeated one-off fixes in reviewer-facing bundle code.

The user-visible proof is straightforward. A known existing benchmark root can still be rebuilt into a correct `upload_bundle_v1`, and a synthetic or alternate-topology fixture can also be bundled without teaching the system that there are always fixed stages like `pass2` and `pass3`. The output bundle should still contain the same three files and preserve compatibility fields where needed, but its internal generation path should no longer depend directly on today’s folder names and pass-slot assumptions.

## Surprises & Discoveries

- Observation: the main flexibility problem is not wording, it is architecture.
  Evidence: `scripts/benchmark_cutdown_for_external_ai.py::_upload_bundle_build_context(...)` derives its effective data directly from root files such as `run_index.json`, `comparison_summary.json`, `need_to_know_summary.json`, starter-pack files, and discovered run directories, then passes those raw shapes into analysis and rendering in the same file.

- Observation: reviewer-facing fixed-stage language was only one symptom of a deeper coupling.
  Evidence: `scripts/benchmark_cutdown_for_external_ai.py::_upload_bundle_build_stage_separated_comparison(...)` still constructs canonical rows around `pass2_stage` and `pass3_stage`, even after recent work to soften the displayed labels.

- Observation: current upload-bundle generation already behaves like it wants an adapter layer, but that layer is accidental and incomplete.
  Evidence: the code already supports stale-or-missing root summaries by re-deriving runs, pairs, changed lines, and diagnostics from discovered run folders, which is exactly the kind of extraction behavior that belongs in a dedicated adapter rather than inside the renderer.

- Observation: the current cookbook refactor is expected to change bundling and staging behavior more than final data semantics.
  Evidence: the user explicitly stated that “the actual processing code and shapes/types of data/end products are not going to change much,” which makes a semantic intermediate model the highest-value stabilization seam.

- Observation: the script can stay orchestration-first while still getting a stable seam.
  Evidence: introducing `write_upload_bundle_v1(...)` as a renderer wrapper plus adapter/model conversion kept `build_upload_bundle_for_existing_output(...)` behavior stable and avoided a risky full rewrite of `_write_upload_bundle_three_files(...)`.

## Context and Orientation

In this repository, `upload_bundle_v1` is the external reviewer handoff for benchmark results. It produces exactly three files under an `upload_bundle_v1/` directory:

- `upload_bundle_overview.md`
- `upload_bundle_index.json`
- `upload_bundle_payload.jsonl`

Today, the main builder lives in `scripts/benchmark_cutdown_for_external_ai.py`. The relevant current functions are:

- `build_upload_bundle_for_existing_output(...)`
- `_write_upload_bundle_three_files(...)`
- `_upload_bundle_build_context(...)`
- `_build_run_record_from_existing_run(...)`
- `_build_comparison_summary(...)`
- `_upload_bundle_build_stage_separated_comparison(...)`

The script currently does four jobs in one place. First, it discovers existing benchmark artifacts from the filesystem. Second, it interprets those artifacts semantically, including codex runs, comparison pairs, prompt traces, and starter-pack outputs. Third, it computes bundle-specific analysis such as triage packets, blame summaries, per-label metrics, regression casebooks, and runtime summaries. Fourth, it serializes the final bundle files and row locators.

That design worked while the benchmark artifact tree was relatively stable, but it is the wrong shape for an upcoming refactor of cookbook staging and bundling. The cookbook refactor is expected to change where artifacts live and how they are grouped more than it changes the meaning of final recipe outputs or benchmark comparisons. Because of that, `upload_bundle_v1` should be driven by a semantic model, not by current paths such as `need_to_know_summary.json`, `starter_pack_v1`, `raw/llm/.../pass2_schemaorg`, or `pass3_final`.

For this plan, “topology-neutral” means the model must not require a fixed number of stages or assume numbered stage semantics. It may represent observed processing nodes, diagnostic families, call groups, or compatibility aliases, but it must not encode “there are always stages 1/2/3” as a core truth.

For this plan, an “adapter” means a small layer that knows how to read one concrete artifact layout and populate the normalized model. The current benchmark root is one adapter. The future cookbook-refactor output tree can become another adapter. The renderer should not care which adapter produced the model.

## Milestones

### Milestone 1: Introduce the normalized source model

At the end of this milestone, the repository will have one explicit Python model that describes all semantic inputs needed to render `upload_bundle_v1`. A novice contributor should be able to read this model and understand what the bundle consumes without first learning the current benchmark folder layout.

Create a new module in `cookimport/bench/` named `upload_bundle_v1_model.py`. Use ordinary Python dataclasses or `TypedDict`-style dictionaries, whichever fits existing repo style more naturally, but be consistent and explicit. The model should cover at least these concepts:

- bundle-level metadata such as source root, generated-at timestamp, and self-check counters
- run summaries with stable fields such as `run_id`, `source_key`, metrics, run settings snapshot, and prompt-log status
- comparison pairs and changed-line rows
- reviewer-facing analysis inputs such as recipe triage rows, call inventory rows, selected packets, and failure-ledger ingredients
- observed processing topology and diagnostic families
- compatibility aliases, especially legacy field families that older bundle readers still expect

Do not put file paths to concrete artifact names in this model except where the path itself is a payload artifact reference. The model is meant to describe semantics, not the current storage layout.

Acceptance for this milestone is that `build_upload_bundle_for_existing_output(...)` can be conceptually described as “adapter builds `UploadBundleSourceModel`, renderer turns model into bundle,” even if the renderer has not been fully switched over yet.

### Milestone 2: Extract the existing-root adapter

At the end of this milestone, all knowledge of the current benchmark artifact tree should be moved behind one adapter module, for example `cookimport/bench/upload_bundle_v1_existing_output.py`.

This adapter should absorb the discovery and fallback behavior that currently lives inside `_upload_bundle_build_context(...)` and parts of `_build_run_record_from_existing_run(...)`. It should know how to read:

- root summary files such as `run_index.json`, `comparison_summary.json`, and `process_manifest.json`
- starter-pack artifacts when they exist
- discovered run directories when root summaries are absent, stale, or incomplete
- derived diagnostics such as prompt warning aggregates, projection traces, wrong-label context rows, and preprocess failure packets

The output of the adapter must be the normalized model from Milestone 1. The adapter may preserve additional raw artifact references for payload generation, but it must not leak current-layout assumptions into the renderer API.

Keep `scripts/benchmark_cutdown_for_external_ai.py::build_upload_bundle_for_existing_output(...)` as the public helper, but change it into a thin wrapper that calls the adapter and the renderer. The wrapper should still support current flags such as `overwrite`, `prune_output_dir`, `high_level_only`, and `target_bundle_size_bytes`.

Acceptance for this milestone is that the adapter can rebuild a model from an existing benchmark root without the renderer directly touching `run_index.json`, `comparison_summary.json`, or discovered run folders.

### Milestone 3: Refactor analysis and rendering around the model

At the end of this milestone, the core upload-bundle analysis should consume the normalized model instead of raw root-layout data structures.

Create a renderer module in `cookimport/bench/`, for example `upload_bundle_v1_render.py`. Move analysis builders there or split them further if needed, but keep the responsibilities clear:

- adapter modules discover and normalize
- analysis/render modules compute reviewer views from normalized semantics
- the script entrypoint orchestrates and writes files

The most important refactor target is the current fixed-stage comparison logic. Replace the internal assumption that canonical rows contain `pass2_stage` and `pass3_stage`. Internally, the renderer should operate on observed diagnostic families or observed processing nodes. If the final `upload_bundle_index.json` still needs a compatibility section named `analysis.stage_separated_comparison`, render it from the neutral internal representation and mark the legacy family aliases explicitly. The same rule applies to overview text: it may mention compatibility families, but it must not imply a fixed stage count as a product truth.

Also move other bundle analyses to the model boundary as time permits in this plan, especially:

- recipe pipeline context and observed topology
- per-label and per-recipe comparison summaries
- runtime, token, and cost analysis
- line-role confidence and candidate-label summaries
- regression casebook and selected packet assembly

Acceptance for this milestone is that the renderer no longer needs to know whether data came from current `pass2_schemaorg` and `pass3_final` slots, a merged repair path, or a future cookbook staging layout. It only needs the normalized model and capability data.

### Milestone 4: Add tests that prove flexibility instead of only current-shape correctness

At the end of this milestone, tests should prove two things: first, current benchmark roots still bundle correctly; second, alternate topologies can be represented and rendered without rewriting bundle logic.

Add a focused test file such as `tests/bench/test_upload_bundle_v1_existing_output.py`. Use it for model and adapter tests that are easier to reason about than the existing large integration file. These tests should cover:

- adapter behavior when root summary files are present
- adapter fallback behavior when root summaries are absent or stale
- topology-neutral recipe-context derivation when the observed run shape is not a live pass2->pass3 sequence
- compatibility alias behavior for legacy diagnostic families

Keep the current end-to-end integration coverage in `tests/bench/test_benchmark_cutdown_for_external_ai.py`, but simplify it where possible so it asserts public bundle behavior rather than internal implementation details. Add at least one test that constructs a synthetic source model with an alternate topology and proves that rendering does not require fixed stage numbering.

Acceptance for this milestone is that a future contributor can change the cookbook staging layout and break only adapter tests, not the renderer contract tests.

### Milestone 5: Regenerate a real bundle and update docs

At the end of this milestone, the refactor will be proven against a real artifact tree and documented for future work.

Regenerate at least one known `upload_bundle_v1` sample root, preferably the Salt, Fat, Acid, Heat single-offline benchmark under `data/golden/benchmark-vs-golden/2026-03-14_17.21.15/single-offline-benchmark/saltfatacidheatcutdown/`. Confirm that the bundle still writes the same three files, that triage-first navigation still works, and that topology/context sections no longer teach fixed-stage assumptions as core truth.

Update `docs/07-bench/07-bench_README.md` to describe the new architecture plainly: `upload_bundle_v1` should be documented as a stable reviewer renderer over a normalized source model, with current benchmark-root extraction as one adapter implementation.

Acceptance for this milestone is a passing targeted test slice plus a regenerated real bundle whose overview/index are still usable and less coupled to current topology.

## Plan of Work

Start by carving a semantic model out of the monolith instead of trying to move everything at once. The first safe extraction is the data returned by `_upload_bundle_build_context(...)`. Today that function returns a loose dictionary of run rows, comparison rows, discovered run dirs, and assorted payloads. Replace that loose shape with an explicit model in `cookimport/bench/upload_bundle_v1_model.py`.

After the model exists, move existing-root extraction into `cookimport/bench/upload_bundle_v1_existing_output.py`. Begin with the simplest wrapper functions:

- `build_upload_bundle_source_model_from_existing_root(source_root: Path) -> UploadBundleSourceModel`
- helper functions for root summary loading
- helper functions for discovered-run fallback and derived diagnostic artifacts

Once the adapter exists, change `scripts/benchmark_cutdown_for_external_ai.py` so its public helper `build_upload_bundle_for_existing_output(...)` calls the adapter and a renderer instead of directly calling `_write_upload_bundle_three_files(...)` over raw layout data. During the migration, it is acceptable to keep temporary compatibility shims inside the script, but the end state of this plan should leave the script as orchestration only.

Then move renderer-specific logic out of the script into `cookimport/bench/upload_bundle_v1_render.py`. The first renderer target should be the current stage-comparison analysis because it encodes the most brittle assumptions. Create an internal neutral representation for observed recipe-processing diagnostics, and render `analysis.stage_separated_comparison` from that representation as a compatibility view rather than as a canonical truth. Do the same for overview prose that currently builds sentences from `pass2_stage` and `pass3_stage` display names.

Do not redesign the three-file contract in this plan. `upload_bundle_overview.md`, `upload_bundle_index.json`, and `upload_bundle_payload.jsonl` must stay where they are and preserve their core semantics. The point of this plan is to make that contract durable while the internals beneath it become more flexible.

Keep the migration additive. It is acceptable to implement the new model and renderer while preserving old helper functions temporarily, so long as progress is recorded in this plan and the script entrypoint becomes thinner over time rather than thicker.

## Validation and Acceptance

Acceptance is behavior-first.

First, a current benchmark root must still rebuild into a valid `upload_bundle_v1` with the same three files and working triage-first navigation. The refactor fails if the bundle contract becomes harder to consume or if row locators, payload rows, or index sections disappear unexpectedly.

Second, the code structure must prove that bundle rendering is no longer bound to the current artifact tree. A novice should be able to point to one model module, one existing-root adapter module, and one renderer module and explain the architecture in one sentence: “adapter reads artifact layout into semantic model; renderer turns semantic model into bundle.”

Third, topology-neutral behavior must be testable. At least one focused test must build a source model with an alternate observed topology and show that rendering still succeeds without requiring canonical `pass2_stage` / `pass3_stage` assumptions.

Fourth, compatibility must remain explicit. Historical or follow-up consumers that still rely on legacy field-family aliases should continue to find them, but the bundle must stop presenting those aliases as core truths about the pipeline shape.

## Idempotence and Recovery

This refactor should be additive and safe to rerun. Rebuilding a bundle from the same source root should overwrite the target `upload_bundle_v1/` directory cleanly when `overwrite=True`, and repeated test runs should not require any manual cleanup.

If the adapter/renderer split becomes too invasive in one pass, keep the migration incremental. It is acceptable to leave temporary shims in `scripts/benchmark_cutdown_for_external_ai.py` so long as the new model remains the canonical internal representation and progress is recorded here. Do not delete old helper paths until the adapter tests and integration bundle regeneration both pass.

If an intermediate step breaks bundle output, recover by routing `build_upload_bundle_for_existing_output(...)` back through the last known-good wrapper while keeping the new model tests intact. The user’s immediate protection is the stable three-file contract, not perfect internal elegance on the first attempt.
