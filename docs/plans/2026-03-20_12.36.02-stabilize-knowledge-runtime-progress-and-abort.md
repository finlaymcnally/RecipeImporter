# Stabilize Knowledge Runtime, Progress UI, and Interrupted Benchmark Finalization

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a CodexFarm benchmark that reaches the non-recipe knowledge stage should do three things that it does not reliably do today. First, knowledge workers should be able to use the narrow local shell patterns that the current workspace-worker design implicitly expects without being trapped in a retry storm. Second, the benchmark progress UI should show the real state of the run instead of hiding workers and implying that all work is still stuck at `0/N`. Third, if an operator stops a broken benchmark mid-run, the program should still write a clear partial-result package rather than leaving only raw worker artifacts.

The user-visible proof is simple. Run the Salt Fat Acid Heat single-offline CodexFarm benchmark again after implementation. The knowledge stage should either finish or fail closed without endless retries, the dashboard should accurately describe completed and active workers, and an interrupted run should still write an incomplete-but-readable summary artifact under the benchmark root.

## Surprises & Discoveries

- Observation: the stalled run was not uniformly hung. Five knowledge workers had already completed, while five others were repeatedly watchdog-killed and retried.
  Evidence: the live statuses under `data/output/2026-03-20_11.16.53/.../knowledge/workers/*/live_status.json` showed `5` completed workers and `5` `watchdog_command_execution_forbidden` workers.

- Observation: the benchmark UI hid workers by design once the configured worker count exceeded eight.
  Evidence: `cookimport/cli.py` sets `display_slots = 8`, which produced the operator-visible `active tasks (10/8, 10 left)` line during the investigated run.

- Observation: the knowledge runtime already has a benign completion seam named `workspace_outputs_stabilized`, so the main problem is not missing completion logic but the command-policy mismatch that prevents many workers from reaching that seam.
  Evidence: `cookimport/llm/codex_farm_knowledge_orchestrator.py` terminates workspace workers with `reason_code="workspace_outputs_stabilized"` when all outputs are present and stable.

- Observation: interrupting the parent benchmark process preserved the raw worker artifacts but did not leave a top-level partial CodexFarm summary package.
  Evidence: after terminating the parent process in the investigated run, the CodexFarm processed run still lacked `eval_report.json`, `processing_timeseries_prediction.jsonl`, `processing_timeseries_evaluation.jsonl`, and `prompt_budget_summary.json`.

- Observation: the earlier line-role split-task repair crash is already fixed and should stay out of scope for this plan except as context.

## Context and Orientation

The relevant runtime spans three layers.

The first layer is the knowledge stage runtime in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. In this repository, a "knowledge worker" is one long-lived Codex workspace session that reads repo-written task files from a bounded local workspace and writes one `out/<task_id>.json` file per owned task. A "watchdog" is repo-owned supervision logic that watches the session and can terminate it when it looks off-contract. The knowledge runtime currently supports three broad outcomes for a worker session: normal completion after outputs stabilize, watchdog termination for forbidden behavior, and task-level follow-up attempts such as watchdog retry or repair.

The second layer is the benchmark and prediction-run orchestration in `cookimport/labelstudio/ingest.py`. This file is where the single-offline benchmark path launches the Codex-backed surfaces and decides whether to continue, fail closed, or write finished artifacts. It is also the seam that currently leaves an interrupted CodexFarm run with raw artifacts but without a readable top-level partial summary.

The third layer is CLI progress rendering in `cookimport/cli.py` plus the structured progress payload helpers in `cookimport/core/progress_messages.py`. These files turn stage progress callbacks into the terminal panel the user sees. Right now that rendering path hides workers above eight slots and does not clearly distinguish active, completed, killed, and retrying workers during the knowledge stage.

The current contracts are described in `docs/10-llm/10-llm_README.md` and the investigation findings are summarized in `docs/reports/2026-03-20_12.30.10-saltfat-codexfarm-investigation-report.md`. The important facts to carry forward into implementation are:

- knowledge uses long-lived workspace workers, not one direct prompt per shard,
- knowledge workers are expected to read named local files and write `out/*.json`,
- the knowledge watchdog already knows how to mark a worker as successfully completed when outputs stabilize,
- the CLI currently caps rendered worker rows at eight,
- the interrupted benchmark currently leaves preserved artifacts but no partial-result summary.

The earlier line-role split-task repair crash has already been fixed in `cookimport/parsing/canonical_line_roles.py` and covered in `tests/parsing/test_canonical_line_roles.py`. Do not reopen that bug as part of this plan; only mention it when explaining why it is out of scope.

## Plan of Work

### Milestone 1: Make the knowledge watchdog accept the workspace-local command patterns the worker model actually uses

The goal of this milestone is to stop the retry storm in the knowledge stage without returning to the old permissive worker policy. The change should stay local to the knowledge stage and to shared supervision helpers that are already knowledge-facing. The implementation should begin by reproducing the investigated failure with the existing fake-runner seams in `tests/llm/test_codex_farm_knowledge_orchestrator.py`. The new failing tests should cover the concrete command shapes seen in the preserved live statuses: heredoc writes into `out/*.json`, `cat` of worker-local hint files, and `jq` reads over `assigned_tasks.json`. The intent is not to bless arbitrary shell usage; it is to declare these workspace-local file reads and writes as on-contract when they remain inside the isolated worker root.

The production edits should live primarily in `cookimport/llm/codex_farm_knowledge_orchestrator.py` and, only if truly necessary, the shared helper layer in `cookimport/llm/codex_exec_runner.py`. Update the knowledge watchdog callback so it recognizes the allowed workspace-local command patterns that are already compatible with the repo-owned worker contract. Keep the existing termination behavior for truly off-contract commands such as reaching outside the worker root, invoking repo/network/interpreter tools, or entering a command loop without output progress. Preserve `reason_detail` for kills so later reports still show the exact offending command.

Acceptance for this milestone is behavior-first. A fake knowledge worker that uses the previously observed local shell shapes should now complete or advance to stabilized outputs rather than immediately dying with `watchdog_command_execution_forbidden`. A fake knowledge worker that genuinely leaves the bounded workspace surface must still be killed. The tests should prove both cases.

### Milestone 2: Make benchmark progress reporting describe the real worker state instead of collapsing it into `0/N`

The goal of this milestone is to make the terminal panel trustworthy for long CodexFarm stages. The first fix is structural: remove the hard-coded eight-row rendering cap in `cookimport/cli.py` so a configured ten-worker knowledge run can render ten worker rows. Do not replace it with an unbounded flood. Instead, make the cap depend on configured workers or use a compact overflow summary only when the configured worker count becomes much larger than the terminal can reasonably display. For the specific Salt Fat benchmark shape, ten workers must be shown as ten workers.

The second fix is semantic: carry enough progress state through the knowledge progress callbacks so the UI can distinguish active, completed, retrying, and failed workers. The current knowledge progress notifications in `cookimport/llm/codex_farm_knowledge_orchestrator.py` only describe the initial pending shard per worker and a completed count. Extend that payload so the CLI can show when a worker finished successfully, when a worker is on a watchdog retry path, and when a worker was killed and is waiting to be retried. This does not need to become a full timeline viewer. It only needs to avoid the current false story that "all ten workers are still equally running."

Update the progress-facing tests in `tests/labelstudio/test_labelstudio_benchmark_helpers_progress.py`, and add focused assertions in `tests/cli/test_stage_progress_dashboard.py` or `tests/core/test_progress_dashboard.py` if the rendering logic itself changes meaningfully. The acceptance check is that the captured progress text for a simulated ten-worker knowledge run no longer contains `active tasks (10/8, ...)` and that it surfaces completed versus retrying workers in plain operator language.

### Milestone 3: Finalize interrupted CodexFarm runs into an explicit partial-result package

The goal of this milestone is that stopping a broken benchmark still leaves a usable high-level artifact package. Start by tracing the single-offline benchmark path in `cookimport/labelstudio/ingest.py` to find the narrowest place where a partially completed prediction run can still emit a run-level status manifest, partial prompt-budget data, and an interruption marker. The design should not pretend the run completed. It should explicitly write incomplete status with the reason for interruption and the artifact paths that were preserved.

The interrupted-run package should include, at minimum, a machine-readable manifest or status file under the CodexFarm processed run root that says the run was interrupted and whether the cause was operator interruption versus runtime exception. If enough artifacts already exist, also write a partial `prompt_budget_summary.json` and a short human-readable summary file that points to the preserved worker trees, the known-finished stage telemetry, and the missing finished artifacts. Keep the format simple; this is a solo-project operator aid, not a new enterprise resume system.

This milestone should also cover graceful cancellation behavior. Investigate the benchmark control flow around the parent process so that `SIGINT` or the equivalent KeyboardInterrupt path unwinds the benchmark at the orchestration layer instead of silently allowing the process to continue respawning workers. If a fully graceful signal-handling path is too invasive, the fallback acceptance is that an interrupted benchmark still writes a partial-result status file before exit. The implementation should prefer an additive, local fix in the benchmark orchestration path rather than a repo-wide signal framework.

Acceptance for this milestone is a targeted integration test under the existing labelstudio benchmark helper suite. The test should simulate an interrupted CodexFarm benchmark after some stage artifacts have been written and assert that the processed run root now contains an explicit incomplete-status artifact and a readable pointer to the preserved partial outputs.

### Milestone 4: Normalize completed knowledge worker reason codes

This is the smallest milestone but it matters for operator legibility. The investigated run showed at least one `completed` knowledge worker with `reason_code=null` and `final_agent_message_state=malformed`, while other completed workers ended with `workspace_outputs_stabilized`. Make the completion semantics explicit. Every completed worker should carry a reason category, even if the category is simply "process exited without watchdog intervention."

The likely edit point is the logic that finalizes `live_status.json` from the Codex exec runner result and the knowledge orchestrator’s worker status files. Add a narrow normalization rule so a completed worker never writes null reason metadata. Then add focused coverage in `tests/llm/test_codex_farm_knowledge_orchestrator.py` that exercises both a stabilized-output completion and a completion without a final JSON message and asserts non-null reason codes for both.

## Validation and Acceptance

The implementation is complete when all of the following are true.

For the knowledge watchdog, the new tests prove that workspace-local shell helpers used to read assigned task files or write `out/*.json` do not trigger `watchdog_command_execution_forbidden`, while truly off-contract commands still do. A fake-runner knowledge scenario that used to spiral through retries now finishes with either validated outputs or the benign `workspace_outputs_stabilized` stop reason.

For the CLI progress UI, a simulated ten-worker knowledge run renders ten workers rather than truncating to eight, and the panel no longer implies that every worker is still equally active when some have already completed or moved to retry status. The acceptance artifact is the captured progress text in the relevant test modules.

For interrupted-run finalization, a benchmark interrupted after partial CodexFarm progress still writes an explicit incomplete-status artifact plus a readable summary pointer to the preserved worker trees and any stage telemetry that already exists. The acceptance artifact is a targeted labelstudio benchmark test that asserts those files exist and describe interruption rather than success.

For completion metadata, every completed knowledge worker `live_status.json` carries a non-null reason code. The acceptance tests should cover both a `workspace_outputs_stabilized` completion and a completion path that exits without a final JSON message.

## Idempotence and Recovery

These changes should be implemented additively and are safe to repeat. The first step in each milestone is test-first and local, so regressions are visible before wider edits land. If a watchdog change becomes too permissive, narrow only the knowledge-specific allowance logic and rerun the focused knowledge orchestrator tests before touching the CLI or benchmark layers. If the progress UI refactor makes the panel too noisy, restore a bounded renderer that still shows all configured workers for small counts such as ten. If interrupted-run finalization proves difficult to wire through true signal handling, fall back to explicit partial-summary emission in the exception/abort path first and record that narrower decision in the `Decision Log`.

Do not delete the preserved artifact roots from the investigated March 20 run. They are the evidence bundle for this plan and the best source for keeping the acceptance tests honest.
