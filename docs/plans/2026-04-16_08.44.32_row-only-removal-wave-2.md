# Finish the Row-Only Removal Wave

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The repo now exposes a row-native public scorer and Label Studio contract, but it still contains a second layer of block-semantic internals that make block authority look half-supported. After this change, rows will be the only semantic unit across the active runtime, while blocks remain only source-layout provenance and debug context. A user should be able to inspect the core parsing, staging, benchmark, and prelabel surfaces and see that block names survive only where they mean “where did this row come from?” rather than “what semantic unit are we scoring or labeling?”

## Surprises & Discoveries

- Observation: the repo has already crossed the public contract boundary to rows, but several core model surfaces still carry block-era aliases directly on the live types.
  Evidence: `cookimport/parsing/label_source_of_truth.py` still defines `AliasChoices("start_row_index", "start_block_index")` and block-named properties on `RecipeSpan` and `RecipeSpanDecision`, while `cookimport/staging/recipe_ownership.py` still accepts `owned_block_indices` and `divested_block_indices` in its main dataclass constructors.

- Observation: Label Studio task generation is row-native, but prelabel internals are still described and keyed as blocks.
  Evidence: `cookimport/labelstudio/freeform_tasks.py` writes `start_row_index`, `end_row_index`, and `focus_row_indices`, but `cookimport/labelstudio/prelabel.py` and `cookimport/labelstudio/prelabel_mapping.py` still use names like `focus_block_indices`, `block_index`, and prompt fallback text that says “labeling cookbook text BLOCKS”.

- Observation: benchmark helper behavior is row-native already, but several helper names still encode the retired block contract.
  Evidence: `cookimport/cli_support/bench_artifacts.py` loads `semantic_row_predictions.json` through `_load_stage_block_prediction_payload(...)` and still exposes names like `evaluate_stage` and test patch points such as `format_stage_block_eval_report_md`.

- Observation: `bd create` child creation under the same parent can race when done in parallel and reuse one child slot.
  Evidence: parallel creation attempted to make two different `recipeimport-htiv.1` children; the missing alias-removal child had to be recreated serially as `recipeimport-htiv.5`.

- Observation: the block-alias surface had already caused a real semantic bug in label-first conversion, not just naming debt.
  Evidence: `build_conversion_result_from_label_spans(...)` was still treating `RecipeSpan.row_indices` as if they were block indices when selecting canonical lines and provenance block ids, which mis-mapped row-native spans until the conversion path was rewritten to derive row membership from atomic row indices and provenance from each selected row's `source_block_index`.

- Observation: old persisted block-named span payloads still needed one narrow reader translation step, but the translation has to remove the retired keys before validation.
  Evidence: after removing `AliasChoices` from `RecipeSpan`, deterministic prep reloads initially failed because `_normalize_recipe_span_row(...)` copied `start_block_index` and related keys into row-native names without deleting the old keys, which then tripped `extra="forbid"` during model validation.

- Observation: two duplicate prelabel fallback prompt copies outside the main prompt builder still used block-era wording after the active prelabel path was converted.
  Evidence: `cookimport/labelstudio/prelabel_codex.py` and `cookimport/labelstudio/prelabel_parse.py` still carried `<<<START_LABELING_BLOCKS_HERE>>>` / `block_index` fallback text and had to be updated so the prelabel support surface no longer described blocks as the active semantic unit.

## Decision Log

- Decision: preserve `source_block_index`, `source_block_id`, archive block rows, and split-job block offsetting as legitimate provenance.
  Rationale: the migration goal is not to erase all mention of blocks. Blocks are still the source-layout container from which rows are derived, and that provenance is useful for joins, debugging, and importer recovery.

  Rationale: the remaining row/block alias properties and validation aliases make the wrong abstraction appear supported everywhere. Older payload compatibility, where still needed, should live only in file readers such as deterministic prep reload paths.

## Context and Orientation

This work touches four connected areas.

`cookimport/parsing/label_source_of_truth.py` defines the row-native authoritative line and recipe-boundary models that flow into staging and benchmark helpers. Right now the types `RecipeSpan` and `RecipeSpanDecision` are row-native in spirit but still accept and expose block aliases such as `start_block_index`, `end_block_index`, `block_indices`, and `title_block_index`.

`cookimport/staging/recipe_ownership.py` defines the stage-owned recipe ownership contract. This contract is also row-native in spirit, but the main dataclasses still accept block-era constructor arguments and expose block-named property mirrors. `cookimport/staging/semantic_row_predictions.py` is the row-native scorer assembly root, yet its helper owners are still named `recipe_block_evidence.py`, `knowledge_block_evidence.py`, and `block_label_resolution.py`, and the label resolver is still named `resolve_stage_block_label`.

`cookimport/labelstudio/freeform_tasks.py` is already row-native: it builds task payloads with `rows`, `start_row_index`, `end_row_index`, and row-native focus/context fields. The remaining debt lives under `cookimport/labelstudio/prelabel.py`, `cookimport/labelstudio/prelabel_mapping.py`, and prompt helpers, where the active code still describes selections and focus sets as blocks.

`cookimport/cli_support/bench_artifacts.py` and related benchmark helper tests are the last visible block-era naming cluster around the row scorer. The behavior now reads `semantic_row_predictions.json`, but helper names and stale docs still say “stage block”.

For this plan, “block provenance” means information like `source_block_index` or the extracted archive row list that tells us where a row came from in the original source document. “Block semantic alias” means a field or function name that still presents blocks as the semantic unit of truth for ownership, labeling, or scoring. Provenance stays. Semantic aliases go.

## Plan of Work

Milestone 1 is model honesty. Complete bead `recipeimport-htiv.5` by removing block aliases from the primary row-native types in `cookimport/parsing/label_source_of_truth.py` and `cookimport/staging/recipe_ownership.py`. It is acceptable to keep block provenance inside the returned data structures, but the helpers themselves should describe row semantics or neutral evidence aggregation, not block authority.

Milestone 3 is Label Studio prelabel honesty. Complete bead `recipeimport-htiv.2` by converting `cookimport/labelstudio/prelabel.py`, `cookimport/labelstudio/prelabel_mapping.py`, and the active prelabel prompt text to row-native names such as `row_index` and `focus_row_indices`. Complete bead `recipeimport-htiv.3` by renaming remaining benchmark helper functions and test patch points that still say `stage_block`, and update the owning docs so they no longer describe `stage_block_predictions.json` as a current artifact. Throughout implementation, update this ExecPlan after each milestone. If implementation reveals that one of these slices is larger than expected, split the corresponding bead rather than reviving compatibility shims.

## Validation and Acceptance

Acceptance is behavioral and visible:

The recipe-boundary and recipe-ownership model surfaces should read as row-native when a developer opens them. They should not advertise block aliases as first-class fields or properties.

The active prelabel path should read and speak in rows. When a developer opens the prompt template fallback or the mapping helpers, they should see row-based terminology and row-native task field handling.

The row scorer path should no longer depend on helper names that imply block-semantic authority. A developer opening the scorer entrypoint and its helpers should see row-semantic naming, with block references limited to provenance joins.

Benchmark/helper docs and tests should stop describing `stage_block_predictions.json` as the active artifact. The active artifact should be `semantic_row_predictions.json`, and helper/test names should match that reality.

Each milestone must be validated with the narrow pytest slice that covers the touched seam, and the exact passing command should be recorded back into this ExecPlan as implementation proceeds.

## Idempotence and Recovery

This plan is a code-and-doc cleanup. It is safe to execute in small batches. If a destructive rename breaks an older persisted fixture or cache payload, do not restore a global alias. Instead, add or adjust a narrow reader translation at the one file-ingestion boundary that truly still needs it.

If a milestone expands unexpectedly, split the corresponding bead into smaller children and update this plan instead of improvising compatibility shims. The design rule for recovery is simple: preserve provenance, remove semantic aliases, and keep fallback translation local.
