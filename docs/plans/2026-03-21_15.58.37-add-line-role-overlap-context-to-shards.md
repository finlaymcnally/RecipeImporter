# Add reference-only overlap context to line-role shard packets

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The current line-role LLM step gives each worker one owned contiguous shard of atomic lines and asks it to relabel only those rows. That is efficient, but it also makes the worker blind exactly where the benchmark is still weak: chapter-to-recipe transitions, recipe-to-notes transitions, outside-recipe lesson headings, and variant/how-to distinctions that depend on nearby rows just outside the shard. After this change, each line-role shard will still own the same core rows, but the worker will also receive a small amount of reference-only overlap from the neighboring shard on each side. The worker will be told clearly that overlap rows exist only for context and must never appear in output labels.

The user-visible result is simple. Opening a line-role worker packet under `data/output/.../line-role-pipeline/runtime/line_role/workers/*/in/*.json` should show the owned rows plus a small `context_before` and `context_after` section. Opening the corresponding prompt artifact should show the same neighboring text in a distinct reference-only section. No extra LLM pass is added, no dynamic shard planner is introduced, and no neighboring rows become label authority. The intended gain is better local judgment near shard boundaries with a modest token increase.

## Surprises & Discoveries

- Observation: the current line-role payload is intentionally compact and only carries owned rows.
  Evidence: `cookimport/parsing/canonical_line_roles.py::build_line_role_model_input_payload(...)` currently writes `{"v": ..., "shard_id": ..., "rows": [...]}` plus packet-guidance fields, and each row is only `[atomic_index, label_code, current_line]`.

- Observation: the file-backed prompt already has a natural seam for an additional reference-only section.
  Evidence: `cookimport/llm/canonical_line_role_prompt.py::build_canonical_line_role_file_prompt(...)` already renders packet-local guidance and then a dedicated `Authoritative shard rows` block. A separate neighboring-context block can be inserted without changing the output JSON contract.

- Observation: the current worker contract already distinguishes guidance from authority cleanly.
  Evidence: the worker prompt says `hints/<task_id>.md` is guidance and `in/<task_id>.json` is authoritative. Adding unlabeled neighboring rows to the authoritative input file is therefore safe only if the file itself marks them as non-owned and the prompt repeats that rule.

- Observation: the current shard-plan metadata already records owned row counts and prompt boundaries, so adding overlap counts is mechanically straightforward.
  Evidence: `_build_line_role_canonical_plans(...)` already writes `owned_ids`, `first_atomic_index`, `last_atomic_index`, and `owned_row_count` into shard metadata.

- Observation: this feature does not need dynamic sharding to be useful.
  Evidence: the current failure family is heavily boundary-shaped. A middle ground of fixed overlap preserves the current one-pass architecture while letting the worker see the immediately adjacent rows that often determine whether a heading is local recipe structure or broader lesson prose.

- Observation: the existing line-role validator and output aggregation needed no changes.
  Evidence: keeping overlap rows out of `owned_ids` and out of the owned `rows` tuples meant proposal validation still keyed only on the original authoritative slice.

## Decision Log

- Decision: implement fixed symmetric overlap first, not dynamic shard expansion and not an additional LLM pass.
  Rationale: the goal is to improve boundary judgment with the smallest operational change and no extra model-spend pass.
  Date/Author: 2026-03-21 / Codex

- Decision: the first implementation will use five neighboring atomic lines on each side of a shard when available.
  Rationale: five lines is the smallest meaningful experiment that should expose nearby headings, notes, or steps without dramatically inflating prompt size. If tuning is needed later, the count can be revisited after benchmark evidence exists.
  Date/Author: 2026-03-21 / Codex

- Decision: overlap rows are reference-only and must never be label-authoritative.
  Rationale: the current line-role runtime, validator, and scoring all assume one shard owns one exact set of rows. Preserving that authority boundary avoids duplicate ownership, merge complexity, and scorer ambiguity.
  Date/Author: 2026-03-21 / Codex

- Decision: overlap rows will be passed as raw text rows without deterministic label codes.
  Rationale: the purpose of overlap is additional context, not stronger anchoring to possibly wrong seed labels. Showing only line id and text reduces the chance that the model parrots neighboring seed labels instead of reading the text.
  Date/Author: 2026-03-21 / Codex

- Decision: packet mode, flip policy, and packet summary remain grounded in owned rows, not in overlap rows.
  Rationale: neighboring context should help the worker judge the owned rows, but it should not rewrite the story of what the shard itself is. Otherwise a recipe-local overlap line could incorrectly cause a mostly outside-recipe shard to advertise itself as `recipe_body`.
  Date/Author: 2026-03-21 / Codex

- Decision: this feature will be repo-internal first and will not add a new user-facing run setting in its first version.
  Rationale: the repository already has a large settings surface. This change is a targeted line-role experiment, and a hardcoded internal default is the smallest safe scope. If later tuning proves valuable, the overlap size can be promoted deliberately.
  Date/Author: 2026-03-21 / Codex

## Context and Orientation

The line-role stage lives in `cookimport/parsing/canonical_line_roles.py`. It takes one ordered list of `AtomicLineCandidate` objects. An atomic line is one labelable line after deterministic atomization and grouping. The stage first builds deterministic seed labels, then partitions the ordered lines into contiguous shards with `partition_contiguous_items(...)`, then gives each shard to the line-role LLM runtime. The core shard-building seam is `_build_line_role_canonical_plans(...)`. Each plan currently carries the owned candidates, a compact worker input payload, and richer debug rows.

The file-backed worker prompt lives in `cookimport/llm/canonical_line_role_prompt.py`. It currently tells the worker that the task is one ordered contiguous slice of the book and that the authoritative rows are the embedded `rows` tuples from the packet JSON. The worker returns strict JSON with one label per owned row. The key prompt seam for this plan is `build_canonical_line_role_file_prompt(...)`, plus `_render_authoritative_rows_for_prompt(...)` and `_render_packet_context_block(...)`.

The current deterministic test surface lives in `tests/parsing/test_canonical_line_roles.py`. That file already verifies prompt wording, packet-context classification, and several line-role behavioral edge cases. This plan will extend those tests to assert that middle shards get left and right context, edge shards clamp correctly, the prompt renders a separate reference-only neighboring-context section, and the file contract says not to label overlap rows.

For this plan, the phrase `owned rows` means the exact rows the shard already owns today and for which the worker must return labels. The phrase `overlap rows` means neighboring atomic lines from adjacent shards that are included only to help interpretation. Overlap rows are not scored, are not written back as predictions, and are not part of `owned_ids`.

## Plan of Work

Start in `cookimport/parsing/canonical_line_roles.py`. Add one small internal constant that defines the overlap size, set to five atomic lines. Do not add a new `RunSettings` field in this first version. Then refactor `_build_line_role_canonical_plans(...)` so each shard plan derives three ordered slices from the full `ordered_candidates` list: the current owned slice, a left overlap slice immediately before the owned slice, and a right overlap slice immediately after it. The overlap slice should clamp naturally at the beginning or end of the book. The owned slice must stay unchanged.

Next, update the input-payload builders so they can receive both owned candidates and overlap candidates. Keep packet interpretation based on the owned rows. The debug payload should gain explicit neighboring-context sections, ideally named `context_before_rows` and `context_after_rows`, where each entry is a small object containing at least `atomic_index` and `current_line`. The model payload should gain the same two sections but use the compactest safe shape, such as arrays of `[atomic_index, current_line]`. Do not include deterministic label codes, span codes, or rule tags in overlap rows. Keep `rows` exactly as the owned labelable tuples.

Update shard metadata so operators can see what happened without reading code. In the `ShardManifestEntryV1.metadata` for line-role shards, keep `owned_row_count` and add `context_before_row_count` and `context_after_row_count`. In `_line_role_execution_plan_phase(...)`, surface the same counts and, if useful, the neighboring atomic indices so debug and plan-preview artifacts explain why a prompt contains duplicate boundary text.

Then update `cookimport/llm/canonical_line_role_prompt.py`. The prompt must continue to say that the task is a grounded correction pass and that output must contain one label per owned row only. Add a new rendered section before `Authoritative shard rows` named something explicit such as `Reference-only neighboring context`. That section must render `context_before_rows` and `context_after_rows` separately so the worker can see where the core shard starts and ends. Add direct wording that these neighboring rows exist only for context, must not be labeled, and must not appear in output JSON. Update the in-prompt example task-file shape so it includes the new context keys, but preserve the output JSON contract.

Update any worker-local hint generation only if it materially helps explain the new shape. This plan does not require redesigning packet summaries or classifier logic. The only required hint-side change is a short sentence that names the overlap counts and repeats that those rows are reference-only. If the hint already has enough packet-level prose without code changes, leave it alone and rely on the `in/*.json` payload plus prompt wording instead.

Finally, update the short current-state note in `docs/04-parsing/04-parsing_readme.md`. Keep it brief. It should say that the file-backed line-role payload now includes `context_before_rows` and `context_after_rows` as unlabeled neighboring context while `rows` remains the authoritative owned slice.

## Milestone 1: Build overlap-bearing shard plans and payloads

At the end of this milestone, the runtime still produces the same number of line-role shards and the same owned row sets, but each shard plan also knows which neighboring rows to show for reference. The key proof is structural, not benchmark-based: a focused test should build several shards from a small synthetic ordered candidate list and show that the first shard has right overlap only, the last shard has left overlap only, and a middle shard has both. The payload written for one of those middle shards must include the overlap rows in separate keys while `owned_ids` and `rows` still refer only to the core rows.

Implement this in `cookimport/parsing/canonical_line_roles.py`. Introduce a tiny helper that slices overlap from `ordered_candidates` by prompt index or slice bounds instead of by expensive re-search. Use the already-known contiguous owned window, derive start and end positions, and compute left and right ranges relative to those positions. Keep the helper idempotent and pure so tests can cover it directly if that is cleaner than asserting through full shard plans.

The debug payload builder should stay rich enough for inspection, but do not explode the payload shape. Add only what the worker and the operator need: neighboring row text and stable row ids. The model payload must remain compact because prompt growth is the cost of this feature. The owned `rows` tuples must stay untouched so output validation and aggregation do not need redesign.

## Milestone 2: Teach the prompt about reference-only neighboring context

At the end of this milestone, a worker reading the file-backed prompt should be able to answer three questions without ambiguity: which rows it must label, which rows are only context, and what JSON shape it must return. The proof is a focused prompt test that asserts the presence of the new neighboring-context section, the non-authoritative wording, and the unchanged output contract.

Implement this in `cookimport/llm/canonical_line_role_prompt.py`. Add one renderer for neighboring context and keep it separate from `_render_authoritative_rows_for_prompt(...)`. Render the left context first, then the owned rows, then the right context. The prompt text must use strong wording such as `reference-only`, `do not label these rows`, and `return one result for every owned row in rows`. Preserve the existing negative rules and packet-local guidance. The worker should leave with more context, not a different authority model.

If the packet JSON example changes, update the example string in the prompt tests accordingly. Keep the prompt readable. The new section should be concise enough that a worker can use it quickly and an operator can verify it visually in prompt artifacts.

## Milestone 3: Lock the contract with tests and short docs

At the end of this milestone, a novice should be able to run the deterministic tests and confirm that overlap is present, bounded, and non-authoritative. The short parsing docs should also mention the new active payload shape so a future reviewer does not have to infer it from test failures or code.

Add tests in `tests/parsing/test_canonical_line_roles.py` for three surfaces. First, add a shard-plan or payload-construction test that exercises a small ordered candidate list across at least three shards and asserts the correct left and right context windows. Second, add a payload-shape test that confirms overlap rows omit deterministic labels while owned rows still carry the existing label tuples. Third, update the file-prompt contract test so it asserts the new neighboring-context section and the explicit `do not label overlap rows` language. If existing tests already cover enough prompt text, extend them rather than creating duplicative new ones.

Then update `docs/04-parsing/04-parsing_readme.md` with a short current-state note. Do not write a long narrative. The point is to document the active worker packet shape, not the design history.

## Validation and Acceptance

This work is acceptable only if all of the following are true.

The authority boundary is unchanged. Every line-role shard still owns exactly the same rows it owned before, and the worker still returns one label per owned row only. `owned_ids` and `rows` must still match one another exactly.

The packet exposes overlap clearly. The `in/*.json` payload for a line-role worker packet must now include both `context_before_rows` and `context_after_rows` when neighboring rows exist. Those sections must contain stable row ids and text, but no deterministic labels.

The prompt makes the rule unmistakable. The file-backed prompt must render the neighboring context separately from `Authoritative shard rows` and must explicitly say that the neighboring rows are reference-only and must not be labeled.

Edge and middle cases clamp correctly. Deterministic tests must prove that the first shard has no left overlap, the last shard has no right overlap, and a middle shard includes both sides up to the configured overlap limit.

Deterministic proof is recorded. The exact test commands and short pass/fail evidence must be pasted into this plan after implementation. If a benchmark rerun has not been approved, the plan must say so explicitly rather than implying that quality proof was completed.

Live benchmark proof is optional and approval-gated. It is useful to measure whether boundary mistakes improve, but it is not part of default acceptance because it spends model tokens.

## Idempotence and Recovery

These changes are additive and safe to iterate on. They only affect newly generated line-role worker packets and prompts. Historical benchmark roots and preserved runtime artifacts remain untouched.

If prompt growth proves too large, reduce only the internal overlap constant and rerun the focused tests. Do not respond by making overlap rows label-authoritative. The whole point of this feature is more context without more ownership.

If a test shows that overlap rows accidentally enter the output path, back out the prompt or payload change that blurred the distinction and rerun the file-prompt and payload-shape tests first. Recovery should not require touching scorer code, aggregation code, or downstream recipe grouping because those surfaces should remain unchanged.
