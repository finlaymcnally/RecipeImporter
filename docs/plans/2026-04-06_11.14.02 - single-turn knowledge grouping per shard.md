# Single-Turn Knowledge Grouping Per Shard

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a knowledge shard will stay one grouping surface. When a benchmark or stage run creates `N` knowledge shards, each shard can still need one initial classification turn and repair turns when validation fails, but successful grouping should no longer fan out into extra grouping batches just because the shard retained many knowledge rows. A user will be able to inspect benchmark artifacts and see a simpler, saner runtime story: one shard, one grouping task, one grouping turn on the happy path.

This matters because the current inline-JSON path lowered helper chatter but still explodes token spend by reusing one resumed chat session across many grouping batches. In the `2026-04-06_10.31.48` `saltfatacidheatcutdown` benchmark root, `nonrecipe_finalize` produced `10` shards but spent `57` turns (`10` initial classification turns, `9` classification repairs, `37` grouping batches, `1` grouping repair). The user-visible problem is not merely cost. The runtime contract no longer matches the selected shard count or the mental model of "group the knowledge rows inside this shard."

## Surprises & Discoveries

- Observation: The current `inline-json` knowledge path is only syntactically thin; it is not semantically one-call-per-shard.
  Evidence: `workspace_run.py` calls `runner.run_packet_worker(... persist_session=True)` for `classification_initial`, then reuses the same prepared workspace with `resume_last=True` for every classification repair and grouping batch.

- Observation: Grouping fanout is intentional in the current contract, not an accidental scheduler bug.
  Evidence: `build_knowledge_grouping_task_files(...)` partitions retained knowledge rows by `max_units_per_batch` and `max_evidence_chars_per_batch`, and both `transition_knowledge_classification_task_file(...)` and `transition_knowledge_grouping_task_file(...)` carry `pending_grouping_unit_batches` forward.

- Observation: The benchmark artifact story under-reports the real turn count.
  Evidence: `prompts/full_prompt_log.jsonl` for the 2026-04-06 benchmark root has `10` `nonrecipe_finalize` rows, but the shard-local `session_lineage.json` files record `57` total turns.

- Observation: One grouping surface per shard is consistent with the user’s intended semantic contract and with deterministic output assembly.
  Evidence: final shard outputs are already assembled per shard by `combine_knowledge_task_file_outputs(...)`; grouping answers only need to map retained knowledge units to group keys/topic labels for that same shard.

## Decision Log

- Decision: Remove active grouping batch fanout from both inline-json and taskfile same-session knowledge flows instead of trying to "optimize" batch sizing.
  Rationale: the product contract the user wants is one grouping decision surface per shard. Smaller batches may reduce some single-call pressure, but they preserve the mismatch between shard count and grouping turns and keep resumed-chat overhead alive.
  Date/Author: 2026-04-06 / Codex

- Decision: Keep bounded repair behavior, but treat repairs as exceptional follow-up turns rather than part of the happy-path grouping contract.
  Rationale: deterministic validation still needs a fail-closed path when the model returns malformed grouping answers. Removing batching does not remove the need for validation or repairs.
  Date/Author: 2026-04-06 / Codex

- Decision: Leave the grouping limit settings in place as inert compatibility fields unless code or docs clearly require outright deletion during implementation.
  Rationale: deleting settings across CLI, config, prompt preview, and tests widens scope. The behavioral fix is that the active contract stops using them to fan out grouping turns.
  Date/Author: 2026-04-06 / Codex

## Outcomes & Retrospective

## Context and Orientation

The knowledge stage is the final non-recipe LLM pass. It decides whether owned non-recipe blocks are reusable cooking knowledge or just other text, and then groups the retained knowledge rows into idea groups. In this repository there are two live execution styles for that stage: the inline-JSON direct-exec path in `cookimport/llm/knowledge_stage/workspace_run.py`, and the task-file same-session path driven by `cookimport/llm/knowledge_same_session_handoff.py`.

The classification and grouping task-file contracts live in `cookimport/llm/knowledge_stage/task_file_contracts.py`. A "task file" here is one repo-authored JSON document containing `units` and an editable `answer` object per unit. The grouping contract is currently implemented in three layers:

1. `build_knowledge_grouping_task_files(...)` collects retained knowledge units and partitions them into several grouping task files.
2. `transition_knowledge_classification_task_file(...)` advances from classification into the first grouping task and stores later batches as pending work.
3. `transition_knowledge_grouping_task_file(...)` consumes one grouping task, then either advances to the next grouping batch or finalizes outputs.

The inline-JSON runtime mirrors that same contract in `cookimport/llm/knowledge_stage/workspace_run.py`. It performs one classification turn, optional classification repairs, then loops over `grouping_task_files` and calls `resume_last=True` for each grouping batch. That is the runtime behavior that makes one shard consume many model turns.

Telemetry and reporting live in `cookimport/llm/knowledge_stage/reporting.py`. The packet economics section reports counts such as `classification_step_count_total`, `grouping_step_count_total`, and `grouping_transition_count_total`. Those names should continue to exist, but after this change they should describe one grouping step per shard on the happy path.

Tests relevant to this change currently live in:

- `tests/llm/test_knowledge_grouping_contract.py`
- `tests/llm/test_knowledge_same_session_handoff.py`
- `tests/llm/test_knowledge_orchestrator_runtime_leasing.py`

Those tests currently encode the old multi-batch grouping story and must be updated to prove the new single-turn-per-shard behavior.

## Plan of Work

First, replace the grouping task-file builder in `cookimport/llm/knowledge_stage/task_file_contracts.py` so the active grouping contract creates one grouping task file containing all retained knowledge units for the shard. Keep the existing evidence payload shape and validation rules, because deterministic output assembly already expects one answer per retained knowledge unit. The implementation should stop partitioning units into several `grouping_batch` slices and should instead populate `grouping_batch` metadata with a single batch summary (`current_batch_index=1`, `total_batches=1`) when grouping units exist.

Second, update the classification and grouping transition helpers in the same module. `transition_knowledge_classification_task_file(...)` should still validate classification answers and still advance into grouping when at least one unit remains `knowledge`, but its transition metadata should no longer carry `pending_grouping_unit_batches`. `transition_knowledge_grouping_task_file(...)` should validate the one grouping task and then finalize outputs immediately on success. Repair behavior should remain unchanged apart from the absence of later grouping batches.

Third, update `cookimport/llm/knowledge_same_session_handoff.py` so the stored same-session state no longer depends on pending grouping batches or completed batch counts. Keep compatibility fields if that makes the code safer, but their active values should reflect the simpler contract (`grouping_batch_count` effectively `0` or `1`, `completed_grouping_batch_count` `0` before completion and `1` after a successful grouping pass).

Fourth, update the inline-JSON direct runtime in `cookimport/llm/knowledge_stage/workspace_run.py`. It should still run one initial classification packet per shard and may still do bounded classification or grouping repairs, but it must stop looping across several grouping task files. Build one grouping task file, send one grouping packet, and finalize the shard after that one grouping response validates. Because the happy path becomes classification plus grouping, reduce the knowledge happy-path worker guardrail from `3` sessions/calls per shard to `2`.

Fifth, update reporting and docs. `cookimport/llm/knowledge_stage/reporting.py` should continue to emit packet economics, but the counts should now align with one grouping step per shard. Update the short owner docs in `cookimport/llm/knowledge_stage/README.md`, `docs/10-llm/10-llm_README.md`, and any benchmark docs whose active contract still implies grouping batch fanout.

Finally, replace the old batch-oriented tests with proof of the new behavior. Contract tests should show that a large retained grouping scope stays in one grouping task file. Same-session tests should show one `advance_to_grouping` transition followed by immediate completion after one valid grouping edit. Runtime tests should still prove deterministic outputs and telemetry, but now with single-turn grouping expectations.

## Validation and Acceptance

The change is correct when all of the following are true:

1. A retained knowledge scope larger than the old `40`-unit grouping threshold still produces one grouping task file rather than two.
2. Same-session knowledge handoff advances from classification into one grouping task and finalizes outputs immediately after one valid grouping submission, with no second grouping task written.
3. Inline-JSON runtime tests still produce valid shard outputs and telemetry, but happy-path grouping counts reflect one grouping step per shard rather than several batch turns.
4. Documentation no longer describes grouping as a batch fanout story.

## Interfaces and Dependencies

The implementation must continue to use the existing task-file transition dataclass `KnowledgeTaskFileTransition` in `cookimport/llm/knowledge_stage/task_file_contracts.py`. No new public module boundary is required. The important post-change interfaces are:

- `build_knowledge_grouping_task_file(...)` returns one grouping task file for all retained units in the shard.
- `transition_knowledge_classification_task_file(...)` may return `advance_to_grouping`, but only for one grouping task.
- `transition_knowledge_grouping_task_file(...)` finalizes outputs after one successful grouping pass instead of advancing through later grouping batches.
- `advance_knowledge_same_session_handoff(...)` still reports validation and completion metadata, but it no longer relies on pending grouping batches.
