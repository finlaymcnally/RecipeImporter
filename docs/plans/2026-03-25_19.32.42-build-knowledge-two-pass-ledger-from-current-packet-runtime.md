Structured restatement:

The user wants the knowledge stage to stop looking like the current packet-plus-snippets workflow and instead behave like a shard-owned ledger that better matches the Codex agent harness. The public knob remains shard count. Choosing `N` knowledge shards should divide the review-eligible non-recipe blocks into roughly `N` contiguous shard owners, and those shard owners should run in parallel. Inside one shard, the worker should read one shard ledger, write one shard-owned output or working copy, and let repo code validate rows without discarding accepted work.

The knowledge stage must be split into two explicit passes. Pass 1 is the focused classification pass: one row per review-eligible block, one final decision per row, and no snippet/evidence output and no reviewer-reason output. Pass 2 runs only after Pass 1 for that shard is accepted and only over the rows that survived as `knowledge`. Pass 2 groups those kept rows into group ids and topic labels. The user explicitly prefers keeping Pass 1 and Pass 2 separate because row-local repair is easier when grouping does not share the same contract as classification.

The same long-lived Codex worker session should own both passes for one shard. The user does not want a conversational "I finished pass 1, would you like me to continue?" handoff. Instead, the repo should own a tight phase transition inside the same worker root: once the worker installs an accepted Pass 1 ledger, repo code should synchronously validate it, freeze accepted rows, build the Pass 2 input files, rewrite the current-phase sidecars, and tell the same live session to continue immediately into Pass 2.

The user also wants validation feedback to be fast and cheap. The worker should not have to wait until the whole pass is "done" before learning that only rows 143 and 187 are malformed. Repo code should provide live or near-live deterministic validation during editing, preserve accepted rows without repeatedly telling the worker "rows 1-142 are still good," and expose only the unresolved rows that need attention. The point is fast local correction in the same worker context window without token-spending chatter.

This plan tracked that intended end state. The live repository now implements the shard-owned two-pass design described here. The old packet-task, batch-first, snippet-heavy path is no longer the happy path, and the notes below should be read as the source system that was replaced.

## Purpose / Big Picture

After this change, the knowledge stage will finally behave the way the user asked for: `knowledge_prompt_target_count=N` will mean roughly `N` shard-owned knowledge ledgers in parallel, not `N` packet bundles with a hidden second task layer. A worker will open one shard-owned Pass 1 ledger, classify every review-eligible block in that shard as `knowledge` or `other`, get fast repo-owned validation that names only unresolved rows, and then continue in the same Codex session into Pass 2 where the kept `knowledge` rows receive `group_id` and `topic_label`.

The user-visible proof is the artifact tree and the staged outputs. Worker roots now expose `CURRENT_PHASE.md`, `CURRENT_PHASE_FEEDBACK.md`, `current_phase.json`, shard-owned `in/work/repair` files, and no batch-first packet queue on the happy path. Final outputs still include `09_nonrecipe_authority.json`, `09_nonrecipe_knowledge_groups.json`, and `09_nonrecipe_review_status.json`, and they now promote from accepted shard ledgers rather than packet-local snippet outputs. Reviewer-facing snippet evidence is no longer part of the live contract.

## Surprises & Discoveries

- Observation: the current knowledge runtime already has the transport pieces needed for the target design, but they are wired to the wrong ownership model.
  Evidence: knowledge already uses one long-lived workspace worker per assignment, local helper tooling, repo-owned feedback files, and exact validator seams, but the active surface is still packet task batches under `CURRENT_BATCH*` instead of shard-owned ledgers under `CURRENT_PHASE*`.

- Observation: the most useful precedent is not old knowledge code. It is the live line-role runtime.
  Evidence: line-role already implements shard-owned `in/work/repair/out` files, `CURRENT_PHASE.md`, `CURRENT_PHASE_FEEDBACK.md`, repo-owned unresolved-row repair, and `install-phase` without a hidden packet task layer.

- Observation: the current packet planner is deeply entangled with progress, summary, and staging language.
  Evidence: the live docs, tests, runtime metadata, and stage summaries all still count packets, packet tasks, packet bundles, or packet-local idea groups, so the migration must include terminology cleanup instead of only swapping file formats.

- Observation: preview cannot honestly predict Pass 2 kept-row counts before Pass 1 runs.
  Evidence: the current prompt preview reconstructs deterministic packet inputs only. In the ledger design, Pass 2 membership depends on accepted Pass 1 labels, so preview should report Pass 1 shard sizes and a Pass 2 upper bound rather than inventing false exact kept-row counts.

- Observation: keeping the compact final payload shape while changing the runtime ownership model cut a large amount of churn without preserving the old behavior.
  Evidence: the live result still uses `packet_id`, `block_decisions`, and `idea_groups`, but the planner, worker surface, validation, promotion, and docs now all treat that payload as the installed shard result rather than as one packet task inside a hidden batch queue.

- Observation: freezing accepted rows only in the repair payload was not strong enough.
  Evidence: once a clean `check-phase` deleted `repair/*.json`, a later edit could rewrite an already accepted row before `install-phase` with no durable frozen state left to compare against. The fix was to persist frozen rows in `current_phase.json` and treat `repair/*.json` as a derived unresolved-row request only.

- Observation: removing snippets simplified more than the writer.
  Evidence: `fake_codex_farm_runner.py`, `codex_farm_knowledge_writer.py`, `codex_farm_knowledge_ingest.py`, prompt preview, and the focused tests all had to stop treating snippet evidence as part of correctness. The final fake-cli knowledge test now asserts `knowledge_groups.json` exists and `snippets.jsonl` does not.

- Observation: the first phase-first runtime cutover accidentally seeded Pass 1 with the installed-output shape instead of the editable row-ledger shape.
  Evidence: the worker helper expected `phase/rows`, but the runtime prewrote `packet_id/block_decisions/idea_groups`; because `ensure_pass1_work` intentionally does not overwrite an existing file, fresh worker roots could start with an invalid Pass 1 work file until the helper learned to migrate that shape.

## Outcomes & Retrospective

The migration landed. The live repo is now shard-ledger-first and phase-first for knowledge. Planning emits one shard input per shard, the worker surface is `CURRENT_PHASE*` plus `work/repair/out`, `install-phase` performs the repo-owned Pass 1 -> Pass 2 handoff inside the same session, and final promotion no longer depends on snippet evidence.

Post-landing review also tightened the contract where the first pass had still been weaker than intended. Frozen accepted rows are now durable across repeated `check-phase` runs because they live in `current_phase.json`, order-only mistakes still generate unresolved-row repair payloads, and the fake rehearsal paths now actually drive the helper loop instead of bypassing it with direct `out/*.json` writes.

The main lesson from the earlier failed plan still holds: intended design and current runtime must be kept distinct. The useful part of this file now is that it records both the source system that was removed and the concrete choices that made the replacement workable without broad unrelated churn.

## Context and Orientation

The knowledge stage is the optional LLM-backed review of non-recipe blocks that survived deterministic routing. The live code path starts in `cookimport/llm/knowledge_stage/runtime.py`, which calls `build_knowledge_jobs(...)` in `cookimport/llm/codex_farm_knowledge_jobs.py`. That planner now writes one shard input file per shard under `knowledge/in/*.json`. `cookimport/llm/knowledge_stage/runtime.py` schedules shards directly, writes `assigned_shards.json`, `current_phase.json`, `CURRENT_PHASE.md`, `CURRENT_PHASE_FEEDBACK.md`, `in/*.json`, `work/*.json`, `repair/*.json`, and `out/*.json`, and keeps one long-lived worker session on the shard through both passes. `cookimport/llm/knowledge_phase_workspace_tools.py` owns the worker helper verbs and the Pass 1 -> Pass 2 handoff. `cookimport/llm/codex_farm_knowledge_ingest.py` validates the compact final shard output against the shard-owned ledgers, and `cookimport/llm/codex_farm_knowledge_writer.py` writes `knowledge_groups.json` plus `knowledge.md` without `snippets.jsonl`.

The source system that was replaced was different in four important ways.

First, planning should be shard-owned, not packet-owned. A shard is one contiguous slice of the ordered review-eligible non-recipe block queue. The shard should be the unit the worker sees, edits, validates, repairs, and installs.

Second, worker control should be phase-first, not batch-first. A phase is the repo-owned current step for one shard. Knowledge should have two phases: `pass1_classify_rows` and `pass2_group_kept_rows`. The worker should open `CURRENT_PHASE.md` first, not `CURRENT_BATCH.md`, and the repo should own the exact handoff between those phases.

Third, validation should be row-local and phase-local. Pass 1 validates only `knowledge|other` labels on the shard-owned row ledger. Pass 2 validates only `group_id` and `topic_label` over the kept `knowledge` rows. Repair requests should name only unresolved rows and should not overwrite already accepted rows.

Fourth, promotion should consume accepted ledgers directly. `09_nonrecipe_authority.json`, `09_nonrecipe_knowledge_groups.json`, and `09_nonrecipe_review_status.json` should derive from accepted Pass 1 and Pass 2 ledgers. The live `snippets.jsonl` evidence surface should disappear from the happy path.

The closest in-repo model for the worker control surface is the live line-role runtime described in `docs/10-llm/10-llm_README.md` and implemented under `cookimport/parsing/canonical_line_roles/`. That borrowing worked: the knowledge rewrite now uses the same phase-oriented control plane, adapted to the two-pass semantic contract.

## Milestones

All four milestones below are complete. They remain here as the implementation story and as a restart guide for the next similar migration.

### Milestone 1: Plan shard-owned Pass 1 ledgers

The first milestone replaces packet planning with shard planning. At the end of this milestone, `cookimport/llm/codex_farm_knowledge_jobs.py` no longer writes one packet file per packet. It instead partitions the ordered review queue into roughly `knowledge_prompt_target_count` contiguous shards and writes one Pass 1 input ledger per shard. Each Pass 1 row should carry only mechanically true fields needed for classification, such as `row_id`, `block_index`, `block_id`, raw text, and any tiny structural hints still needed for conservative labeling.

The user-visible proof is simple. A run root should show fewer, larger shard-owned Pass 1 files instead of packet files, and the shard count should change directly when `knowledge_prompt_target_count` changes. Run the focused planning tests after this milestone and inspect one fake worker root. Acceptance is that the planner emits shard-owned ledgers and no longer emits packet-owned inputs on the happy path.

### Milestone 2: Move knowledge workers onto the phase-first control plane

The second milestone replaces the batch-first worker surface. At the end of this milestone, a knowledge worker root should look like a line-role worker root adapted for two phases: `current_phase.json`, `CURRENT_PHASE.md`, `CURRENT_PHASE_FEEDBACK.md`, authoritative `in/*.json`, editable `work/*.json`, optional `repair/*.json`, harvested `out/*.json`, and helper verbs `check-phase` plus `install-phase`. The worker should no longer need `assigned_tasks.json`, `CURRENT_BATCH.md`, or packet-task drafts to do normal work.

The user-visible proof is that one knowledge worker can open one shard ledger, edit only unresolved rows, run `check-phase`, and keep the same Codex session alive without being asked to summarize or stop after every small unit of work. Run the focused workspace tests after this milestone. Acceptance is that one worker root owns one shard and one active phase at a time.

### Milestone 3: Add Pass 1 to Pass 2 same-session handoff and row-local repair

The third milestone adds the core repo-owned orchestration the user asked for. At the end of this milestone, `install-phase` while Pass 1 is active should synchronously validate Pass 1, freeze accepted rows, build the Pass 2 kept-row ledger, rewrite `CURRENT_PHASE*`, and tell the same live session to continue immediately into Pass 2. If only rows 143 and 187 are malformed, the feedback surface should name only those rows. Already accepted rows must stay frozen and off-limits to repair.

The proof is a fake-Codex rehearsal that reaches Pass 2 without a second conversational prompt and without re-reporting the accepted prefix as work to revisit. Acceptance is that repair patches touch only unresolved rows and that the same worker session finishes both phases for the shard.

### Milestone 4: Promote ledger outputs and cut packet/snippet machinery

The fourth milestone removes the old product surface. At the end of this milestone, `cookimport/llm/codex_farm_knowledge_ingest.py`, `promotion.py`, `reporting.py`, `prompt_preview.py`, `docs/10-llm/10-llm_README.md`, `docs/05-staging/05-staging_readme.md`, and the focused tests all teach the two-pass ledger contract. `snippets.jsonl` is gone from the live path. Preview reports shard-owned Pass 1 row counts and Pass 2 upper bounds rather than packet counts. Runtime summaries and stage observability stop counting packets as the main truth for knowledge.

The proof is that the focused LLM and staging tests pass and the docs no longer teach the packet/batch/snippet workflow as current behavior.

## Plan of Work

Start with `cookimport/llm/codex_farm_knowledge_jobs.py`. Replace the packet partitioner with direct shard partitioning over the review-eligible non-recipe block queue. Use the same public knob, `knowledge_prompt_target_count`, but interpret it literally as “rough target shard count.” The planner should emit shard-owned Pass 1 input ledgers and shard metadata such as owned block indices, row count, and total character count. Delete packet ids and packet payload wrappers from the happy path.

Then move to `cookimport/llm/knowledge_stage/runtime.py` and `cookimport/llm/knowledge_stage/planning.py`. Stop splitting shards back into per-packet tasks. The runtime should schedule shards directly, and worker-count math should stay what it is today: `knowledge_worker_count` controls concurrency, while shard planning controls how much work each worker owns. If generic runtime helpers assume task rows, either teach them that a knowledge “task” is now one whole shard or replace those helpers with a direct shard-owned runtime like line-role uses. Do not preserve a hidden packet-to-task expansion layer.

Then rewrite `cookimport/llm/knowledge_stage/workspace_run.py`, `cookimport/llm/knowledge_stage/recovery.py`, and `cookimport/llm/knowledge_workspace_tools.py`. Replace the batch-first surface with a phase-first surface. The worker root should expose one shard-owned input ledger for the active phase, one editable work ledger, one optional repair file, and the repo-written current-phase sidecars. Pass 1 and Pass 2 should share the same session, the same worker root, and the same helper entrypoint. The helper verbs should mirror line-role naming wherever possible so the shared direct-exec worker contract stays small.

Then rewrite validation in `cookimport/llm/codex_farm_knowledge_ingest.py`. Replace packet-output normalization with two phase validators. Pass 1 accepts only row-local `knowledge|other` decisions. Pass 2 accepts only kept-row grouping fields. Build one unresolved-row repair payload per phase, and make repair merges refuse to overwrite frozen accepted rows. Validation should stay deterministic and cheap so `check-phase` can run often without model-token spend.

After validation works, rewrite promotion and reporting in `cookimport/llm/knowledge_stage/promotion.py`, `cookimport/llm/knowledge_stage/reporting.py`, and `cookimport/llm/codex_farm_knowledge_writer.py`. Promotion should consume accepted Pass 1 and Pass 2 ledgers directly. `09_nonrecipe_authority.json` comes from accepted Pass 1 labels. `09_nonrecipe_knowledge_groups.json` comes from accepted Pass 2 rows. `09_nonrecipe_review_status.json` records shard or row completeness. `knowledge.md` may remain if it is rendered from accepted rows and groups, but `snippets.jsonl` and snippet-evidence logic should be removed from the live path.

Finally, rewrite docs and tests in the same wave. Update the current knowledge sections in `docs/10-llm/10-llm_README.md`, `docs/05-staging/05-staging_readme.md`, and `cookimport/llm/README.md`. Replace or rename packet-centric tests such as `tests/llm/test_knowledge_phase_workers_packets.py` and packet-output-ingest assertions so they teach shard-owned ledgers and same-session phase handoff instead of packet tasks and snippet evidence.

## Validation and Acceptance

The migration is complete only because all of the following are true in the live tree.

Planning acceptance:

- The live knowledge planner emits shard-owned Pass 1 ledgers from the ordered review-eligible block queue instead of packet files and packet bundles.
- `knowledge_prompt_target_count` now controls the approximate number of shard ledgers directly, not a packet bundle count above a packet floor.

Worker-surface acceptance:

- The normal knowledge worker root is phase-first: `CURRENT_PHASE.md`, `CURRENT_PHASE_FEEDBACK.md`, `current_phase.json`, shard-owned `in/*.json`, editable `work/*.json`, optional `repair/*.json`, and harvested `out/*.json`.
- The normal knowledge worker root no longer depends on `CURRENT_BATCH*`, `assigned_tasks.json`, or one `out/<task_id>.json` per packet.

Pass 1 acceptance:

- Every owned block appears exactly once in the Pass 1 work ledger.
- The only final Pass 1 decisions are `knowledge` or `other`.
- Reviewer-reason fields, snippet bodies, and evidence quotes are not part of the live Pass 1 contract.
- A malformed Pass 1 shard can preserve accepted rows and isolate only unresolved rows into a repair file.

Pass 2 acceptance:

- Pass 2 runs only after Pass 1 is accepted for that shard.
- Every kept row gets exactly one `group_id` and one `topic_label`.
- Rows excluded by Pass 1 cannot leak into Pass 2.

Same-session acceptance:

- One long-lived workspace worker can move from Pass 1 to Pass 2 without a new conversational prompt.

## Idempotence and Recovery

The migration can be additive for a short period while tests are moving, but the final state must be a hard cut. Temporary compatibility readers or dual writers are acceptable only if they are deleted before this plan is marked complete. The repo should fail closed on the new ledger contract instead of silently preserving old packet semantics forever.

`check-phase` must be safe to rerun without side effects. `install-phase` must be idempotent when the active phase is already accepted. Re-running Pass 1 install must not generate conflicting Pass 2 inputs. Repair merges must refuse to overwrite frozen accepted rows. If a worker session dies between phases, resume logic should reopen `current_phase.json` and continue from the unfinished phase rather than restarting the shard from scratch.

If the migration stalls midway, recover by re-checking these seams in order:

1. `cookimport/llm/codex_farm_knowledge_jobs.py` for shard planning and Pass 1 ledger shape.
2. `cookimport/llm/knowledge_stage/workspace_run.py` plus `recovery.py` for the worker control surface.
3. `cookimport/llm/codex_farm_knowledge_ingest.py` for exact row-level validation and repair behavior.
4. `promotion.py` plus staging docs/tests for the final artifact contract.
