# Fix CodexFarm Reasoning Trace Capture for Nested Events

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds. This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The goal of this change is to restore visibility into the model's internal reasoning ("Thinking Traces") within `recipeimport` reports. Currently, these traces are missing, replaced by the message "_No thinking trace captured for this sample._".

This visibility is critical for developers to:
1.  Debug why an LLM pipeline made a specific (potentially incorrect) extraction decision.
2.  Verify that the model is following complex reasoning instructions (e.g., boundary detection).
3.  Audit the "depth" of reasoning across different models and effort levels.

The root cause is a failure in `CodexFarm`'s trace parser, which fails to recognize reasoning events when they are wrapped in modern `codex exec` event structures like `item.completed`. By expanding the parser's heuristics, we will ensure that these events are correctly categorized and saved into the `.trace.json` artifacts consumed by `recipeimport`.

## Surprises & Discoveries

- Observation: `CodexFarm` only checks dictionary *keys* for reasoning hints (e.g., "reasoning", "thinking").
- Evidence: In `CodexFarm/src/codex_farm/codex_exec.py`, the helper `_value_contains_key_hints` iterates over `dict.items()` but only performs `any(hint in key_text for hint in lowered_hints)`. The values of those keys are completely ignored.
- Impact: Modern `codex exec` JSONL output uses an `item.completed` event type where the actual substance of the event (including its type) is inside a nested `item` dictionary. The reasoning hint "reasoning" appears as a *value* (e.g., `item["type"] == "reasoning"`), causing the entire event to be missed.

- Observation: `recipeimport` has a specific list of keys it checks to extract "snippets" for the report.
- Evidence: `cookimport/cli.py:_extract_reasoning_excerpt` looks for keys: `summary_text`, `summary`, `text`, `delta`, `content`.
- Connection: If the event is correctly captured by `CodexFarm` but lacks these keys, the report will still be empty. Fortunately, `item.completed` reasoning events use the `text` key, which is in the list.

## Decision Log

- Decision: Expand `_value_contains_key_hints` to also check string values for reasoning hints and rename it to `_value_contains_hints`.
- Rationale: Checking values is a "future-proof" way to catch reasoning events regardless of how they are nested. If a model or wrapper tool puts "thinking" or "reasoning" in any metadata value, it's highly likely to be a reasoning event we want to capture.
- Date/Author: 2026-03-04 / Gemini CLI

- Decision: Explicitly handle the `item.completed` event type in `_event_matches_trace_hints`.
- Rationale: While general value-checking is good, `item.completed` is the primary wrapper used by the current `codex exec` tool. Directly checking `event["item"]["type"]` provides an extra layer of guaranteed capture for the most common failing case.
- Date/Author: 2026-03-04 / Gemini CLI

## Context and Orientation

`CodexFarm` executes LLM requests via the `codex` CLI. This CLI emits a stream of JSON objects (JSONL) on its standard output. These objects represent "events" during the model's execution (e.g., tool calls, reasoning steps, message completion).

### The Trace Lifecycle
1.  **Capture**: `CodexFarm` captures the `stdout` stream.
2.  **Parsing**: `_persist_trace_artifact` in `codex_exec.py` parses each JSON line.
3.  **Classification**: It uses `_event_matches_trace_hints` to decide if an event should go into `action_events` (tools/actions) or `reasoning_events` (internal thoughts).
4.  **Artifact Generation**: It saves a `.trace.json` file.
5.  **Reporting**: `recipeimport` reads the `.trace.json`, extracts snippets from `reasoning_events`, and writes them to a Markdown report.

### Key Files in `~/projects/shared/CodexFarm`
- `src/codex_farm/codex_exec.py`: The "brain" of the trace capture.
- `tests/test_codex_exec.py`: Unit tests for command execution and trace persistence.

### Terms
- **Trace Artifact**: A comprehensive JSON file recording exactly what the LLM did during one specific call.
- **JSONL**: A format where every line is a standalone JSON object.
- **Thinking Trace**: The textual evidence of the model "thinking before it acts."

### Milestone 2: Value-Aware Heuristics
Upgrade the core recursive helper.
1.  In `src/codex_farm/codex_exec.py`, find `_value_contains_key_hints`.
2.  Rename it to `_value_contains_hints`.
3.  Update its recursive logic:
    - If the input is a `dict`, check both keys AND values (if they are strings).
    - If the input is a `list`, recurse into every item.
    - If the input is a `str`, check if it contains any of the `lowered_hints`.
4.  Update `_event_matches_trace_hints` to call the renamed function.

### Milestone 3: Explicit Item-Type Handling
Add a specialized check for the known offender.
1.  In `_event_matches_trace_hints`, add a specific check for `item.completed`.
2.  If the event type is `item.completed`, check if `event["item"]["type"]` matches the reasoning hints.
3.  This ensures that even if our general heuristic has depth limits, we always catch the primary nested reasoning format.

### Milestone 4: End-to-End Validation
Verify the fix in the real application.
1.  Perform a run in `recipeimport` that triggers reasoning.
2.  Check the resulting `.trace.json` in the run's `.codex-farm-traces` directory.
3.  Confirm `reasoning_events` is no longer empty.
