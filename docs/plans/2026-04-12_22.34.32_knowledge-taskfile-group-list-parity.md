## Purpose / Big Picture

After this change, the knowledge same-session taskfile fallback should ask for the same kind of pass-2 answer that the inline path already uses: one grouped split-and-tag answer over contiguous kept rows, not one row-local grouping answer repeated over and over. A human should be able to open a grouping `task.json`, see one batch-level `groups` answer surface, and get the same contiguous-group semantics whether the run used inline JSON or the same-session taskfile fallback.

## Context and Orientation

The knowledge same-session fallback lives in `cookimport/llm/knowledge_same_session_handoff.py`. It reads the worker-edited `task.json`, validates it, and either rewrites the file into repair/grouping mode or writes final shard outputs to `out/*.json`.

The worker-visible taskfile shape is built in `cookimport/llm/knowledge_stage/task_file_contracts.py`. Today, classification already works fine as a row-by-row answer surface. The remaining mismatch is grouping: taskfile grouping still creates one answer slot per kept row, while the inline pass-2 contract already asks the model to emit contiguous `groups` spans that carry one shared tag story each.

The reusable inline parsing logic already lives in `cookimport/llm/knowledge_stage/structured_session_contract.py`. That module can read a top-level `groups` list, validate row-id ownership plus contiguity, and expand the accepted groups back into row-keyed answers. Downstream finalization helpers still want those row-keyed answers, so the parity change is mostly a worker-surface and handoff-validation change, not a repo-wide output migration.

## Plan of Work

First, refit the grouping taskfile builder in `cookimport/llm/knowledge_stage/task_file_contracts.py`. Instead of emitting one grouping unit per kept row, emit one grouping batch unit whose immutable evidence contains the ordered batch rows and compact row facts, and whose editable answer is a single `groups` list. Keep the batch metadata and ontology on the taskfile so batching and grounding constraints stay visible.

Second, add a taskfile-side group-list parser that mirrors the inline parser. The handoff path should accept the worker-edited `groups` list, validate contiguous coverage against the batch rows, and then project the accepted groups back into the existing row-keyed `grouping_answers_by_unit_id` map before calling the current downstream validators and output combiners.

Third, update the same-session prompt text and tests. The worker instructions for `knowledge_group` should stop saying “answer each unit with `group_id`...” and instead say that the grouping batch is answered as one `groups` list. Focused tests should prove single-batch grouping, repair rewrites, and multi-batch grouping still work.

Finally, remove the two already-approved completed ExecPlans for contiguous grouping and post-merge whole-shard repair, close their cleanup beads, and keep this new ExecPlan around until its own deletion bead is approved later.
