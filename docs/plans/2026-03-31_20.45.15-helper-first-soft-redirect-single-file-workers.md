# Make Single-File Workers Helper-First, And Make Knowledge Classification Anti-Batch

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

`docs/PLANS.md` is checked into this repository and this document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, single-file workspace workers for recipe correction, knowledge finalization, and canonical line-role correction should stop feeling like tiny coding sandboxes and start feeling like guided review tools. A worker that briefly tries `cat task.json`, `ls`, or an inline `python3 -c` rewrite should be redirected back to repo-owned helpers inside the same session instead of being watchdog-killed for helper-path drift. The hard stops should remain for real boundary escapes such as repo inspection, network/package-manager commands, parent-directory traversal, or forbidden absolute paths.

The key additional requirement, discovered after reviewing live `nonrecipe_finalize` telemetry, is that knowledge classification cannot merely become “wrapper-first.” It must become a different interaction model. `nonrecipe_classify` should feel like a review queue where the repo owns progression one decision at a time. It must stop advertising or accepting batch-answer synthesis as the happy path. Knowledge grouping can still use a broader helper surface because grouping is already a packet-level task, but semantic keep/drop classification must become hostile to `answers.json`, `jq`, heredoc rewrites, and looped bulk apply behavior.

Success is not just “no more shell-drift kills.” Success is visible in targeted tests and, once the user explicitly approves a later benchmark run, in live artifacts where classification workers use queue-style helper commands, do not mass-generate answer files, do not loop over broad unanswered-unit dumps, and complete without `single_file_shell_drift_repeated` or `single_file_shell_drift_egregious` kills.

## Context and Orientation

The current single-file worker runtime lives in `cookimport/llm/codex_exec_runner.py`. That module creates the sterile mirrored workspace, writes worker-facing `AGENTS.md`, mirrors `_repo_control`, parses Codex JSON event streams, and classifies executed shell commands. A “single-file worker” in this repository means a workspace-worker session whose visible task surface is one repo-written `task.json` file containing immutable evidence plus editable `/units/*/answer` objects.

`cookimport/llm/editable_task_file.py` owns the current task-file helper CLI. Today it can summarize `task.json`, show owned units, and apply answers. It still exposes those capabilities as `python3 -m cookimport.llm.editable_task_file ...`, which reads like “open a coding tool” instead of “follow a small review workflow.”

Stage-specific task files and same-session prompts are currently built in three places. Recipe uses `cookimport/llm/recipe_stage_shared.py`. Knowledge uses `cookimport/llm/knowledge_stage/task_file_contracts.py` plus prompt/runtime logic in `cookimport/llm/knowledge_stage/recovery.py`. Canonical line-role uses `cookimport/parsing/canonical_line_roles/runtime.py` and `cookimport/parsing/canonical_line_roles/planning.py`. These files currently duplicate the helper command map and the “edit `task.json`, then run the helper” instructions.

The current shell-drift enforcement also lives in those stage runtimes. They call `classify_workspace_worker_command(...)` from `cookimport/llm/codex_exec_runner.py`, record `single_file_shell_drift` warnings, and currently escalate repeated or egregious single-file drift into termination reason codes such as `single_file_shell_drift_repeated` and `single_file_shell_drift_egregious`.

The live problem is stage-specific. Knowledge finalization already has two phases hidden behind one broad “single-file worker” idea: `nonrecipe_classify`, which should make one semantic keep/drop decision per owned block, and `knowledge_group`, which can legitimately see multiple already-kept rows at once. The current helper surface and redirects flatten those into the same “edit a JSON bundle efficiently” posture. That is exactly what allowed workers to bulk-author classification answers in the live run that motivated this revision.

For this plan, a “soft redirect” means a repo-owned wrapper or shell shim that intercepts a known wrong local command, prints a short hint naming the correct helper command, exits without mutating repo-owned files, and lets the session continue. A “guarded shell shim” means a PATH-prepended executable stub named `cat`, `ls`, `find`, `python3`, and so on that either redirects the worker back to the wrapper commands or transparently passes through to the real executable when the command is still within the allowed local-temp or helper behavior contract.

## Plan of Work

### Milestone 1: Build a wrapper-command surface that is shorter than scripting, but stage-aware

Add one shared module for single-file worker command definitions and dispatch. Create `cookimport/llm/single_file_worker_commands.py` and put the canonical wrapper names there. Some wrappers are global, such as `task-summary`, `task-status`, `task-doctor`, and `task-handoff`. Others must be stage-aware. In particular, `nonrecipe_classify` should expose queue-style helpers such as `task-show-current`, `task-show-neighbors`, `task-answer-current`, and `task-next`, while grouping and other stages can still expose `task-show-unit`, `task-show-unanswered`, `task-set-answer`, `task-template`, and `task-apply` where appropriate. The shared module should expose builder functions that return the worker-facing `helper_commands` map and compact ordered workflow summary for the current `stage_key` and, for knowledge, the current substage.

Extend `cookimport/llm/editable_task_file.py` so the underlying functionality can support both families of wrappers cleanly. Keep the current flags for compatibility, but split the semantics. Queue-style classification helpers should read and update only the current owned unit or at most one narrow context window. Batch-answer template writing should remain available only for stages where packet-level authoring is still valid. The summary output should separately expose a compact answer-shape example derived from `answer_schema` so the model knows what belongs inside each object without reading raw schema blobs.

Create a new dispatch module, for example `cookimport/llm/workspace_shell_shims.py`, that can act as the implementation behind both the helper wrappers and the guarded shell shims. For wrapper names such as `task-summary`, it should call the existing helper module code directly. For stage-neutral commands such as `task-status`, `task-doctor`, and `task-handoff`, it should read `task.json`, inspect `stage_key`, and dispatch to the correct same-session helper module for recipe, knowledge, or line-role.

### Milestone 2: Add a queue-style classification surface for `nonrecipe_classify`

Add repo-owned state and helper behavior so knowledge classification feels like reviewing one item at a time rather than editing a broad table. This can live in the hidden same-session state under `./_repo_control/` or be encoded in top-level task metadata, but the visible experience must be queue-like. `task-show-current` should show one current owned unit plus a tiny amount of optional nearby context. `task-answer-current '<answer_json>'` should write only the current unit’s answer and record that the current unit was explicitly shown before it was answered. `task-next` should advance the current pointer only after the current unit has a complete answer or after handoff has validated the answer.

For `nonrecipe_classify`, remove or disable the broad helper affordances that invite batch authorship. The worker-facing helper map and prompt text must not advertise `task-template`, `task-apply`, `task-show-unanswered --limit 50`, offset browsing, or any equivalent “open many unresolved rows and author them in bulk” path. If narrow context browsing is still needed, cap it at a small count such as one current unit plus one or two nearby units, and make that a different helper from the broad unanswered dump used in grouping or recipe review.

The deterministic contract here is process-shaped, not semantic. Repo code should not second-guess the `knowledge` versus `other` answer, but it may enforce that the worker answered the current shown unit, that classification-only helpers were used for classification-only task files, and that classification-stage batch-apply commands are unavailable or rejected.

### Milestone 3: Inject PATH-based wrappers and guarded shell shims into single-file worker sessions

Extend the helper-import preparation path in `cookimport/llm/codex_exec_runner.py`. The runner already creates a per-session helper import root. Add a per-session `bin/` directory there and write executable wrapper scripts into it. Those scripts should call the new dispatch module using the interpreter that already works inside the fenced workspace. Then prepend that `bin/` directory to `PATH` for single-file worker sessions before launching `codex exec`.

In the same `bin/` directory, generate guarded shims for the specific commands that currently cause helper-path drift: `cat`, `sed`, `ls`, `find`, `tree`, `python`, `python3`, `node`, and `jq`. Each shim must know the real executable path resolved before the PATH rewrite so it can transparently pass through when needed. When the command matches a known bad pattern in a single-file workspace, the shim should print a short redirect hint and exit non-zero without touching `task.json`. The redirect target must be stage-aware. In `nonrecipe_classify`, wrong turns should redirect to queue-style helpers such as `task-summary`, `task-show-current`, or `task-answer-current`, not to template/apply batch helpers. The first implementation should cover these redirect cases:

1. Reading `task.json` with `cat`, `sed`, or similar should redirect to `task-summary`, `task-show-unit`, or `task-show-unanswered`.
2. Broad orientation commands like `ls`, `find`, and `tree` at workspace root should redirect to `task-summary`.
3. Inline `python3 -c`, `python -c`, `node -e`, heredoc rewrites, or `jq` transforms that reference `task.json` should redirect to the relevant repo-owned helper path. For classification that means `task-show-current` or `task-answer-current`; for grouping and other packet-level flows that may still mean `task-template` and `task-apply`.

Do not use the shims to block legitimate local temp-file helpers that do not touch `task.json`. The existing “tiny local helper grounded on temp files” escape hatch should continue to work. The shim only needs to redirect the common drift patterns that are currently breaking the happy path.

### Milestone 4: Make every single-file task file advertise the correct workflow for its stage

Replace the duplicated stage-specific helper command builders with the shared wrapper command map. Update `cookimport/llm/recipe_stage_shared.py`, `cookimport/llm/knowledge_stage/task_file_contracts.py`, and `cookimport/parsing/canonical_line_roles/runtime.py` so `helper_commands` now point at the wrapper names rather than `python3 -m ...` strings. For knowledge, the builder must distinguish classification from grouping and emit different helper maps, workflow text, and optional helper availability.

Add a shared workflow payload to `task.json`, either by extending `build_task_file(...)` in `cookimport/llm/editable_task_file.py` or by injecting the field in the stage builders. The workflow should be explicit and ordered. The summary output for `task-summary` should surface it compactly so the worker sees the entire happy path immediately. The standard sequence should be:

1. General single-file stages: `task-summary`, inspect a narrow set of units, edit only `/units/*/answer`, then `task-handoff`.
2. Knowledge classification: `task-summary`, `task-show-current`, optional `task-show-neighbors`, `task-answer-current '<answer_json>'`, then `task-handoff` or `task-next` depending on the final helper design.
3. Knowledge grouping and other packet-level stages: `task-summary`, inspect owned units, optional `task-template answers.json` plus `task-apply answers.json`, then `task-handoff`.

Revise the single-file worker `AGENTS.md` text in `cookimport/llm/codex_exec_runner.py` and the prompt text in `cookimport/llm/recipe_stage_shared.py`, `cookimport/llm/knowledge_stage/recovery.py`, and `cookimport/parsing/canonical_line_roles/planning.py` so they emphasize this positive workflow first. Keep the negative shell guidance short. The worker should be told that `task.json` is the whole job, that the wrapper commands are the expected road, and that wrong shell commands will simply be redirected.

### Milestone 5: Convert shell-drift from kill logic into warning-and-redirect behavior, except for classification-stage bulk synthesis

Update the single-file command classification and watchdog usage in `cookimport/llm/codex_exec_runner.py`, `cookimport/llm/recipe_stage_shared.py`, `cookimport/llm/knowledge_stage/recovery.py`, and `cookimport/parsing/canonical_line_roles/runtime.py`.

Keep classification categories for observability, but do not apply one blanket policy to all stages. Mild helper-path drift such as `single_file_task_file_shell_read`, `single_file_orientation_command`, and `single_file_discouraged_shell_command` should become warnings once redirect shims exist. However, for `nonrecipe_classify`, broad `task.json` transforms and batch-answer synthesis should remain kill-worthy or hard-rejected behavior because they are the semantic failure mode we are fixing. Classification-stage uses of `task-apply`, `answers.json`, heredoc rewrites, `jq` answer synthesis, or shell loops over many units must be terminated or rejected decisively rather than merely logged.

Preserve hard termination for:

- true boundary violations returned by `detect_workspace_worker_boundary_violation(...)`
- final-message-without-output and other existing completion-liveness failures
- `nonrecipe_classify` bulk-synthesis patterns that bypass per-block review
- any existing generic no-progress safeguards that are still needed after the redirect shims exist

If the existing generic command-loop watchdog becomes too noisy because redirected commands still count as command executions, adjust `should_terminate_workspace_command_loop(...)` or the stage-specific call sites so a brief burst of redirected commands does not look like a runaway worker. The command-loop watchdog must remain as a last-resort liveness guard, but it should stop punishing the exact redirect flow this plan introduces.

Add redirect-aware telemetry to live status and per-stage observability. The minimum acceptable outcome is that operators can distinguish “worker drifted but was redirected and finished” from “worker never produced output.” A simple first version is enough: counts of single-file drift commands, last drift reason detail, and whether the worker later completed successfully.

Add classification-specific observability, not just generic drift warnings. At minimum, record whether a classification shard used queue-style helpers, whether batch-answer helpers were attempted or rejected, the largest unanswered-unit browse count requested, and how many redirect events occurred before success or failure. These become the proof surface for later benchmark review.

### Milestone 6: Strengthen helper coverage and review framing, especially for knowledge

Enhance `task-summary` so the worker can see enough structure without opening raw `task.json` blindly. The summary should include:

- answer progress
- the compact workflow sequence
- the compact answer-shape summary from `answer_schema`
- the top-level `review_contract` summary when present
- grouping-batch summary when present
- for queue-style classification, the current review position, the current unit id, and a reminder that only the current unit is actionable

Knowledge already has a strong `review_contract` in `cookimport/llm/knowledge_stage/task_file_contracts.py`. Keep that contract but surface it more aggressively in `task-summary`, and tighten the knowledge prompt/hint wording in `cookimport/llm/knowledge_stage/recovery.py` so the worker is reminded that it is doing close block review, not inventing a packet-wide classifier or heuristic. Recipe and line-role do not need the same semantic contract, but they should still get a short plain-language workflow summary instead of mostly negative shell instructions.

Do not add deterministic semantic heuristics here. This milestone is about framing and helper coverage, not changing authority boundaries.

- `task-apply answers.json` updates only `/units/*/answer` and is unavailable or rejected for `nonrecipe_classify`.
- `task-status`, `task-doctor`, and `task-handoff` dispatch correctly for recipe, knowledge, and line-role based on `task.json.stage_key`.
- `cat task.json`, `ls`, and inline `python3 -c` against `task.json` return redirect hints instead of modifying state or printing the whole task file, and the hint is stage-appropriate.

The later benchmark-level acceptance, once explicitly approved by the user, must cover command shape as well as kill reasons. There should be no `single_file_shell_drift_repeated` or `single_file_shell_drift_egregious` termination reason codes in the stages where mild drift is now redirect-only. In `nonrecipe_classify`, telemetry should show queue-style helper usage, little or no `answers.json` / `task-apply` / `jq` / looped bulk-answer behavior, and materially lower command counts than the run that motivated this revision. Any remaining shell-drift evidence should appear as warnings on workers that still finish or recover.

## Idempotence and Recovery

This work is safe to apply incrementally. Keep the legacy Python-module helper entrypoints callable while the wrapper commands are introduced, so deterministic repo code and existing recovery scripts do not break mid-refactor. The worker-facing task-file `helper_commands` should switch only after the wrappers exist and tests cover them.

The generated wrapper and shim scripts live in per-session helper import directories under `CODEX_HOME`, so they are disposable runtime artifacts. If a wrapper script is wrong, fix the generator in `cookimport/llm/codex_exec_runner.py`, rerun the test, and let the next worker session regenerate the correct files. No migration of user data is required.

If the command-loop watchdog starts terminating redirected sessions during implementation, do not paper over it by widening every timeout. Fix the command-loop accounting so redirected commands are not treated like productive command loops. That preserves liveness protection without reintroducing kill-on-drift.
