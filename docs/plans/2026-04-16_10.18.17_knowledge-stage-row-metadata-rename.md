# Rename Knowledge-Stage Semantic Metadata from Block Names to Row Names

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

The knowledge stage already validates active worker answers through row-grounded task files and row-id coverage checks, but many repo-owned metadata keys, validation summaries, and shard bookkeeping surfaces still use block-era semantic names. This cleanup renames those semantic metadata/reporting surfaces to row-native names while keeping true provenance refs such as source block ids, `block:` evidence pointers, and other source-layout identifiers intact.

## Decision Log

- Decision: rename repo-owned metadata/reporting keys first, not the compact persisted worker payload aliases.
  Rationale: validation summaries, shard metadata, task-file helper metadata, and reporting keys are internal semantic surfaces that can be cleaned up without broadening this wave into a wire-format migration.
  Date/Author: 2026-04-16_10.18.17 / Codex

- Decision: keep provenance-bearing block refs untouched in this wave.
  Rationale: `block:` evidence refs, compact payload aliases like `i`/`bi`, and source block ids still identify source layout rather than semantic ownership or validation authority.
  Date/Author: 2026-04-16_10.18.17 / Codex

- Decision: treat knowledge-stage “owned block” shard metadata as semantic debt where it represents the packet’s active semantic row surface.
  Rationale: the task-file and validation layers already reason about owned rows and row ids; keeping `owned_block_indices` in internal shard metadata preserves the wrong semantic unit.
  Date/Author: 2026-04-16_10.18.17 / Codex
