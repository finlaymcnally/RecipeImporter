# Remove Hidden Benchmark Mode Seams

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, `cookimport labelstudio-benchmark` should describe exactly one behavior in this repository: generate benchmark predictions and evaluate them using the pipelined path. The codebase should stop exposing, storing, or teaching alternate benchmark execution-mode concepts that no longer matter to the user. The user-visible proof is simple: the benchmark CLI no longer accepts or records benchmark execution mode choices, all-method benchmark reuse no longer drives benchmark prediction generation through a hidden skip-eval flag, and the focused benchmark/helper tests still pass.

## Context and Orientation

The benchmark command lives in `cookimport/cli.py` as `labelstudio_benchmark(...)`. After this cleanup it contains one benchmark execution path: predict plus evaluate. The prediction-reuse work needed by `_run_all_method_benchmark_config(...)` now lives in a separate private helper instead of a hidden branch inside the benchmark command.

The prediction-stage output shape is already well-defined in the same module. `generate_pred_run_artifacts(...)` creates a prediction run directory, `_build_prediction_bundle_from_import_result(...)` loads it into a `BenchmarkPredictionBundle`, and `predict_stage(...)` turns that bundle into prediction-record rows. Those three pieces are enough to support all-method reuse without routing through the benchmark command itself.

The test surface is split. Focused entrypoint files like `tests/labelstudio/test_labelstudio_benchmark_helpers_eval_payload.py` and `tests/labelstudio/test_labelstudio_benchmark_helpers_scheduler.py` import shared case bodies from `tests/labelstudio/benchmark_helper_cases.py`. That shared helper still writes old manifest fields, so changing only one focused test file is not enough.

The active docs that still teach the removed concept are short notes in `docs/07-bench/07-bench_README.md` and the understanding note created during the earlier explanation pass. Those need to be updated or removed so the repo no longer tells future agents that benchmark execution mode is a meaningful knob here.

## Plan of Work

First, add a private helper in `cookimport/cli.py` that runs only the benchmark prediction stage for offline benchmark use. The helper should accept the same prediction-related settings currently assembled for all-method reuse, run `generate_pred_run_artifacts(...)`, build the `BenchmarkPredictionBundle`, and optionally write `predictions_out`. It may write a benchmark run manifest if that remains useful for reuse/debugging, but that manifest must describe prediction artifacts directly and must not encode removed execution-mode names.

Second, switch `_run_all_method_benchmark_config(...)` to call the new private helper instead of `labelstudio_benchmark(...)`. Once that is done, delete the hidden no-eval benchmark parameter and remove the no-eval branch inside `labelstudio_benchmark(...)`.

Third, simplify benchmark run-config writing in both `cookimport/cli.py` and `cookimport/config/run_settings_adapters.py`. The benchmark adapter should stop emitting the old benchmark mode selector. Benchmark run manifests should stop storing the old benchmark mode fields. If a prediction-stage-only manifest remains for the new helper, use names that describe the actual artifact purpose rather than resurrecting removed mode names.

Fourth, update the shared benchmark helper tests and docs. `tests/labelstudio/benchmark_helper_cases.py` must stop writing the removed fields. Focused tests must stop asserting them and instead assert observable behavior such as evaluation artifacts existing or not existing, prediction records being written, and helper kwargs excluding the removed seam. Active docs and the earlier understanding note must be cleaned so the repo no longer presents removed benchmark execution modes as current concepts.
