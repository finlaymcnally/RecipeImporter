# Reduce line-role helper command chatter

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan builds directly on the already-implemented line-role paved-road work in `docs/plans/2026-03-22_02.44.24-add-line-role-workspace-paved-road-tools.md`. That earlier change succeeded at one important goal: the workers stopped inventing ad hoc `jq` machinery and started using repo-written tools. Since this follow-up plan was first written, a meaningful subset of the second pass has also landed: the helper now exposes `prepare-all`, `finalize`, and `finalize-all`, and the prompt mentions the batch-first flow. The remaining problem is narrower: the repository still does not prewrite drafts into `scratch/`, so workers must still spend commands asking the helper to do repository-owned draft creation.

## Purpose / Big Picture

After this change, a line-role worker should enter its workspace and find that the deterministic drafts already exist under `scratch/`. The normal loop should be short: open `current_task.json`, inspect the paired hint, input packet, and prewritten draft, edit only if the deterministic seed is wrong, then run one finalizing command. The worker should not need to run `prepare-all` or `scaffold` in the common case, because draft creation is repository-owned work that should already be complete before the worker starts.

The user-visible success condition is a calmer and more honest line-role worker trace. A future benchmark rerun should still show the repo-provided tools being used, but command count per shard should drop materially from the current `2026-03-22_03.33.50` baseline. The worker root should also match the prompt literally: `scratch/` already contains a draft per task, task metadata includes the draft path explicitly, and `finalize` or `finalize-all` is the real completion path rather than just a documented alias.

## Context and Orientation

The active line-role helper surface was added by the earlier plan in `docs/plans/2026-03-22_02.44.24-add-line-role-workspace-paved-road-tools.md`. The current code lives primarily in:

- `cookimport/parsing/line_role_workspace_tools.py`, which defines the stage-local helper behavior and renders the self-contained `tools/line_role_worker.py` script mirrored into each worker root
- `cookimport/parsing/canonical_line_roles.py`, which builds the line-role worker roots, writes `current_task.json`, `assigned_tasks.json`, `OUTPUT_CONTRACT.md`, `examples/`, `tools/`, and the worker prompt
- `cookimport/llm/codex_exec_runner.py`, which mirrors `tools/` into the isolated execution workspace and advertises it through `worker_manifest.json`
- `tests/parsing/test_line_role_workspace_tools.py`, `tests/parsing/test_canonical_line_roles.py`, and `tests/llm/test_codex_exec_runner_workspace.py`, which currently validate the helper behavior and workspace discoverability

Today the worker root already contains `scratch/`, and the helper CLI now supports `prepare-all`, `finalize`, and `finalize-all`. But the repository still does not populate `scratch/` before the worker starts. Instead, the common worker path is still:

1. open `current_task.json`
2. optionally run `overview` and `show`
3. run `prepare-all --dest-dir scratch/` or `scaffold <task_id> --dest scratch/<task_id>.json`
4. edit the scratch draft
5. run `finalize scratch/<task_id>.json` or `finalize-all scratch/`

That loop is better than handwritten `jq`, but it is still too expensive because draft creation is still worker-triggered even though it is repository-owned logic. In this plan, “prewritten draft” means a JSON file under `scratch/<task_id>.json` that already contains the deterministic seed output before the worker session begins. “Finalize” means one helper command that validates a draft and, if valid, writes it to the declared `metadata.result_path`.

## Plan of Work

Implement the remaining simplification in three parts.

First, move deterministic draft creation from the worker runtime into workspace construction. Extend `cookimport/parsing/canonical_line_roles.py` so every assigned line-role task gets a prewritten `scratch/<task_id>.json` seed draft before the worker prompt is written. Reuse the same deterministic label-expansion logic already present in `cookimport/parsing/line_role_workspace_tools.py`; do not fork a second seed path. The worker should be able to open `current_task.json`, find its `metadata.scratch_draft_path`, and immediately inspect or edit the prebuilt draft.

Second, enrich line-role task metadata only enough to support the calmer loop. Add a `scratch_draft_path` field to each task row in `assigned_tasks.json` and `current_task.json`. Keep `input_path`, `hint_path`, and `result_path` as they are. The helper verbs already exist, so this metadata change is the missing piece that lets the prompt and workspace state agree with each other.

Third, rewrite the line-role worker prompt around the new normal path. The prompt should now say:

1. open `worker_manifest.json`
2. open `current_task.json`
3. open `hints/<task_id>.md`, `in/<task_id>.json`, and the prewritten `scratch/<task_id>.json`
4. edit the draft only where the deterministic seed is wrong
5. run `python3 tools/line_role_worker.py finalize scratch/<task_id>.json`

The prompt should explicitly say that `prepare-all`, `overview`, and `show` are optional fallback tools when the worker is confused or when the prewritten drafts are missing unexpectedly, not the standard starting path. If multiple drafts remain after the worker has reviewed them, `finalize-all` should be the preferred bulk completion command.

## Validation and Acceptance

The change is acceptable only if all of the following are true.

Every generated line-role worker root now contains prewritten deterministic drafts under `scratch/` for the assigned tasks, and each task row in `current_task.json` / `assigned_tasks.json` declares a `scratch_draft_path`.

The normal worker prompt no longer tells the worker to start by running `prepare-all`, `overview`, `show`, or per-task `scaffold`. Instead it points the worker directly at the prewritten draft and uses `finalize` as the normal completion command.

`tools/line_role_worker.py finalize <json_path>` validates and installs one draft in one step. It must reject drafts with wrong row count, wrong order, extra keys, or invalid labels.

If `finalize-all` is implemented, it must validate all assigned drafts and either install all valid ones or fail with a concise summary instead of silently doing partial work.

The focused `.venv` tests above pass.

Optional benchmark acceptance, only if later approved by the user, is behavioral and must include both of these outcomes together:

1. line-role still shows `invalid_output_shard_count: 0` and no new watchdog regressions
2. `command_execution_count_total` drops materially from the `2026-03-22_03.33.50` baseline of `106` commands across `5` shards

If helper usage remains high but command count does not fall, the implementation should be considered incomplete.

## Idempotence and Recovery

This plan is additive and safe to repeat. Prewriting `scratch/<task_id>.json` drafts is safe because the worker roots are rebuilt from scratch on every run. The deterministic draft generator should overwrite the draft path cleanly during workspace construction so rerunning the builder cannot leave mixed old/new content.

`finalize` and `finalize-all` must remain fail-closed. A structurally bad draft must never be written to the declared `result_path`. If there is any doubt, leave the validated `out/` path untouched and surface the errors in the command output.

If the first implementation accidentally makes the prompt too terse and workers stop using the helper entirely, keep the prewritten drafts but restore one short reminder about `tools/line_role_worker.py finalize ...`. The goal is a shorter common path, not a hidden one.

If later evidence shows that `show` and `overview` are still frequently needed, improve the task metadata first instead of adding more commands. The principle of this plan is that better repo-written state should replace worker-side inspection.

## Interfaces and Dependencies

At the end of this plan, the following interfaces should exist.

In `cookimport/parsing/line_role_workspace_tools.py`, the mirrored helper CLI already supports:

    python3 tools/line_role_worker.py overview
    python3 tools/line_role_worker.py show <task_id>
    python3 tools/line_role_worker.py check <json_path>
    python3 tools/line_role_worker.py finalize <json_path>
    python3 tools/line_role_worker.py finalize-all
    python3 tools/line_role_worker.py prepare-all --dest-dir scratch/

`prepare-all` and `scaffold` may remain for debugging or compatibility, but they should no longer be required in the standard prompt loop once drafts are prewritten.

In `cookimport/parsing/canonical_line_roles.py`, each line-role task metadata row must include:

    input_path: str
    hint_path: str
    result_path: str
    scratch_draft_path: str
    owned_row_count: int
    atomic_index_start: int | None
    atomic_index_end: int | None
    deterministic_label_counts: dict[str, int]

The line-role worker-root builder must prewrite:

    scratch/<task_id>.json

using the exact same deterministic label-expansion logic as the helper CLI.

`finalize` and `finalize-all` must continue using the existing structural validation rules:

    one row per owned input row
    same order as the owned input rows
    only `rows` at top level
    only `atomic_index` and `label` in row objects
    only valid line-role full label names

No new third-party dependencies are required. The helper logic should remain standard-library Python plus existing repo label-code helpers.
