## Purpose / Big Picture

The current pipeline starts with row-level line-role labels, but later outside-recipe stages still collapse many mixed rows into a single block-level answer and then stamp that block answer back onto every row in the block. The narrowest resurrection bug for explicit `NONRECIPE_EXCLUDE` rows has already been fixed under bead `recipeimport-ecq`, so excluded rows now stay row-level `OTHER` during benchmark projection and scoring. The remaining problem is broader: rows that survive as `NONRECIPE_CANDIDATE` can still be dragged into block-level `KNOWLEDGE` or `OTHER` simply because their neighbors in the same block were strong enough. After this change, outside-recipe semantics should stay row-authoritative all the way to the row-facing benchmark artifacts: recipe ownership can remain block/span based, but rows outside recipes should not be flattened into one block-wide `knowledge` or `other` answer.

The user-visible result should be that mixed outside-recipe blocks are scored honestly. Weak setup rows such as `Think about making a grilled cheese sandwich.` should stay excluded or `OTHER` even when neighboring rows in the same block are valid `KNOWLEDGE`. The proof will be row-facing artifacts and tests that preserve row-level exclusions and row-level semantic labels through nonrecipe finalize and benchmark projection.

## Surprises & Discoveries

- Observation: the motivating bug is real and reproducible on a concrete row, not just a metric hypothesis.
  Evidence: in the April 14, 2026 benchmark root `data/golden/benchmark-vs-golden/2026-04-14_23.59.10`, the row `Think about making a grilled cheese sandwich.` is `NONRECIPE_EXCLUDE` in `authoritative_labeled_lines.jsonl` but ends up as final row label `KNOWLEDGE` in `row_label_predictions.jsonl` with the reason tag `nonrecipe_authority:knowledge`.

- Observation: that exact `NONRECIPE_EXCLUDE` resurrection path has already been patched.
  Evidence: `cookimport/labelstudio/ingest_flows/artifacts.py` now converts `NONRECIPE_EXCLUDE` predictions to row-level `OTHER` with the reason tag `nonrecipe_authority:preserved_exclude`, and `cookimport/bench/eval_source_rows.py` skips the later block-level override when that preservation marker is present. Existing regression tests also cover that special case.

- Observation: the overwrite happens because mixed outside-recipe blocks are finalized at block granularity.
  Evidence: the same row lives in source block `1692`, which also contains stronger heat-control teaching rows. `08_nonrecipe_route.json` routes block `1692` as `candidate`, `09_nonrecipe_authority.json` finalizes block `1692` as `knowledge`, and benchmark projection stamps that block answer back onto every row in the block.

- Observation: a second canary row shows the difference between a prompt miss and an authority overwrite.
  Evidence: `Taste. It will need salt.` in source block `388` never became `NONRECIPE_EXCLUDE`; it stayed `NONRECIPE_CANDIDATE`, then the mixed block finalized as `knowledge`. That row is still a prompt-boundary problem, but it is a different problem from the resurrection of already-excluded rows.

- Observation: the row-level overwrite already happens before the source-row scorer reads the predictions.
  Evidence: `cookimport/labelstudio/ingest_flows/artifacts.py` applies `_apply_nonrecipe_authority_to_predictions(...)` while building `row_label_predictions.jsonl`. The scorer in `cookimport/bench/eval_source_rows.py` then consumes row predictions that already reflect block-level authority.

- Observation: the current architecture is now half-fixed rather than fully broken.
  Evidence: the writer and scorer both special-case preserved excludes, but nonrecipe final authority itself remains block-based through `authoritative_block_category_by_index`, and non-exclude candidate rows can still be overridden from that block map.

- Observation: row-mode routing initially broke an existing recipe-boundary demotion contract.
  Evidence: after the first row-authority refactor, `test_execute_stage_import_session_demotes_rejected_pseudo_recipe_title_before_nonrecipe_stage` failed because `build_nonrecipe_stage_result_from_labeled_rows(...)` treated an available `RECIPE_TITLE` row as a fatal nonrecipe route error instead of normalizing it back to `NONRECIPE_CANDIDATE`. The fix was to apply the same “recipe-local available rows normalize to candidate” rule in row mode that block mode already relied on after rejected pseudo-recipe spans.

- Observation: the row-authority implementation did not require deleting block summaries entirely.
  Evidence: the final code now writes both `authoritative_row_category_by_index` and a derived source-block summary map. Recipe grouping, recipe ownership, and stage-block evidence still use the structural/block summary surfaces, while row-facing Label Studio and source-row scoring prefer the row map.

- Observation: “remove 1” cannot mean deleting all line-role block labels.
  Evidence: `AuthoritativeBlockLabel` and `authoritative_block_labels.json` are still used by `cookimport/parsing/recipe_span_grouping.py`, `cookimport/staging/nonrecipe_stage.py`, prompt preview support, stage evidence, and multiple tests. The safe change is to demote block labels where they overwrite row semantics, not to rip out every block-label artifact.

- Observation: the docs index command is still broken in this environment.
  Evidence: `npm run docs:list` currently fails with `ENOTSUP` on the `tsx` IPC socket. Direct doc reads are sufficient for this plan.

## Context and Orientation

This repository builds “source rows” from cookbook text and then assigns semantic labels to those rows. A “row” is the unit scored by the source-row benchmark. A “block” is a group of neighboring rows that originated from the same source block or are reviewed together. “Recipe ownership” means a block or span belongs to a recipe, such as title, ingredients, instructions, notes, or yields. “Outside-recipe semantics” means the labels for rows that are not inside recipe structure, especially the distinction between rows worth keeping as culinary knowledge and rows that should be ignored or treated as plain `OTHER`.

The current authority chain starts in `cookimport/parsing/label_source_of_truth.py`, which turns line-role outputs into authoritative labeled rows and derived `AuthoritativeBlockLabel` records. Recipe grouping in `cookimport/parsing/recipe_span_grouping.py` consumes those block labels to build recipe spans. Outside-recipe routing and finalize in `cookimport/staging/nonrecipe_stage.py` and `cookimport/staging/nonrecipe_authority.py` currently summarize outcomes at block level through `authoritative_block_category_by_index`. Label Studio and benchmark row artifacts are then written by `cookimport/labelstudio/ingest_flows/artifacts.py`, which currently calls `_apply_nonrecipe_authority_to_predictions(...)` and rewrites row-facing labels from block-level nonrecipe authority. The source-row benchmark in `cookimport/bench/eval_source_rows.py` reads those row-facing predictions.

The immediate design risk is mixed blocks. If a block contains one weak narrative row and several stronger knowledge rows, the current block-level final answer can still drag the weak row into `KNOWLEDGE` so long as that weak row was not already hard-excluded. The April 14 run proved that this is not theoretical. The fix must allow mixed outside-recipe blocks to remain mixed in the final row-facing artifacts.

## Plan of Work

The first workstream is the audit represented by bead `recipeimport-5f5.1`. Start by tracing every place where `AuthoritativeBlockLabel` or `authoritative_block_labels.json` is consumed. Split those uses into two classes. The first class is legitimate structural use: recipe span grouping, stage evidence, prompt preview summaries, and any other place where a block-level summary is the correct abstraction. The second class is semantic override use: any path where a block-level label becomes the final answer for a row-scored outside-recipe label. The implementation goal for this milestone is not to delete block labels everywhere. It is to demote or remove only the semantic override uses. Update this plan with the audited list as implementation proceeds so a future reader can see what remained block-level and why.

The second workstream is the core behavior change represented by bead `recipeimport-5f5.2`. In `cookimport/staging/nonrecipe_stage.py` and `cookimport/staging/nonrecipe_authority.py`, preserve block context for routing and LLM review, but make the final outside-recipe semantic answer row-addressable. That means the end state should have a row-level authority map, or an equivalent row-granularity structure, that can distinguish mixed rows inside the same outside-recipe block. The likely shape is additive first: keep the existing block-level summaries temporarily for compatibility and observability, but add a row-level authority payload that is the true source of truth for row-facing outputs. This is now specifically about non-exclude candidate rows, because preserved excludes already have a narrow protection.

The third workstream is to stop row-facing artifact writers from flattening non-exclude row semantics back to the block. In `cookimport/labelstudio/ingest_flows/artifacts.py`, change `_apply_nonrecipe_authority_to_predictions(...)` or replace it with a row-aware equivalent so that `row_label_predictions.jsonl` preserves row-level semantic decisions for candidate rows as well as excludes. A row that is final `OTHER` should stay that way even when its neighboring rows are final `KNOWLEDGE`. If compatibility requires keeping the current block-level summary fields in adjacent artifact files, do so, but do not let them overwrite row predictions.

The fourth workstream is to align the benchmark seam. In `cookimport/bench/eval_source_rows.py`, ensure the source-row scorer consumes row-authoritative predictions and does not reintroduce block-flattening on the scoring path for non-exclude candidate rows. If the scorer still needs block-level nonrecipe summaries for reporting, they must remain reporting-only. The final label assigned to a scored row must come from the row-authoritative view.

The fifth workstream is to update tests and long-lived docs. The tests must cover both the architecture rule and the concrete canaries. The docs should explain the final authority rule in plain language so future work does not accidentally reintroduce block-level outside-recipe overwrites.

## Validation and Acceptance

The work is acceptable only if all of the following are true.

Mixed outside-recipe blocks can produce mixed final row labels. A weak setup row can remain `OTHER` while a neighboring row in the same block becomes `KNOWLEDGE`.

There is no remaining path where a non-exclude outside-recipe row is forced to final `KNOWLEDGE` or `OTHER` purely because block-level outside-recipe authority rewrote the whole block.

Row-facing benchmark and Label Studio artifacts preserve row-level outside-recipe authority. If block-level summaries still exist, they are clearly auxiliary and not the source of truth for row predictions.

Recipe ownership remains block/span based. The implementation must not regress recipe grouping or recipe-structure ownership just to achieve row-level outside-recipe semantics.

Focused `.venv` tests pass for the touched staging, Label Studio, and benchmark seams. At least one regression test must cover a mixed block shaped like source block `1692` or `388`, where one row should remain `OTHER` while adjacent rows are valid `KNOWLEDGE`.

No live `codex-exec` benchmark is run without explicit user approval.

## Idempotence and Recovery

This change should be implemented additively first. Keep existing block-level summaries long enough to avoid breaking unrelated consumers while introducing a row-authoritative outside-recipe payload. That makes repeated edits safe and gives a clean rollback path if a downstream consumer still depends on the legacy summary shape.

If a code path still needs block-level reporting during migration, keep it read-only and reporting-only. The safe failure mode is duplicated data with one canonical row-level source of truth, not silent reuse of the old block override path.

If a partial implementation breaks row-facing artifacts, the recovery path is to restore the previous writer logic while keeping newly added regression tests and the new row-authority structures in place behind a narrower seam. Do not “recover” by reintroducing block-level semantic overwrite as the final authority.

## Interfaces and Dependencies

At the end of implementation, there should still be a clear distinction between structural interfaces and semantic interfaces.

Structural interfaces can remain block-based. `cookimport.parsing.label_source_of_truth.AuthoritativeBlockLabel` may still exist and may still feed recipe grouping and stage observability. `cookimport.parsing.recipe_span_grouping.group_recipe_spans(...)` should continue to consume block labels because recipe ownership is still a block/span problem.

Semantic interfaces for outside-recipe final labels must become row-based for row-facing outputs. Whether the implementation introduces a new row-authority model, expands an existing row prediction record, or adds a row-level authority map to the nonrecipe stage result, the important contract is that downstream row writers and the source-row scorer can retrieve the final outside-recipe answer by row identity without consulting a block-wide category override.

The artifact writer in `cookimport/labelstudio/ingest_flows/artifacts.py` must stop treating `nonrecipe_stage_result.authority.authoritative_block_category_by_index` as the final source of truth for non-exclude row labels. The benchmark scorer in `cookimport/bench/eval_source_rows.py` must read the row-authoritative view. Tests in `tests/staging/test_nonrecipe_stage.py`, `tests/labelstudio/test_labelstudio_ingest_parallel_prediction_run.py`, and `tests/bench/test_eval_source_rows.py` must exercise that contract directly.
