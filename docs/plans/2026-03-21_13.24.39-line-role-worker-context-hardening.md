# Harden line-role worker context so cold-start Codex workers choose the right labels

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The current line-role stage already has the right outer shape: immutable packet inputs, long-lived workspace workers, durable `out/*.json` results, and benchmark artifacts that make mistakes visible. The remaining weakness is worker context. A Codex worker can operate the workspace, but it still has to infer the label ontology packet by packet, and that is exactly where the current run on `data/golden/benchmark-vs-golden/2026-03-21_12.58.13` lost accuracy. After this change, a worker that starts with zero repository context should receive enough packet-local grounding to act conservatively and correctly: it should know whether it is looking at lesson prose, memoir, recipe-local structure, or recipe-adjacent notes before it starts flipping labels.

The user-visible result is twofold. First, the worker artifacts under `data/output/.../line-role-pipeline/runtime/line_role/workers/*/` become understandable on their own: each packet will carry a short semantic summary, explicit packet-level decision rules, strong-vs-weak hint guidance, and a few in-workspace contrast examples. Second, offline benchmark reruns should show fewer harmful `OTHER -> KNOWLEDGE`, `KNOWLEDGE -> HOWTO_SECTION`, and `KNOWLEDGE -> INSTRUCTION_LINE` flips in outside-recipe lesson prose while preserving the useful corrections inside real recipes. The intended proof surface is targeted tests plus an offline single-book benchmark compare; no live model spend should happen until the user explicitly approves it.

## Surprises & Discoveries

- Observation: the worker-operating contract is already solid; the semantic contract is the weak link.
  Evidence: `worker_manifest.json` clearly names `worker_manifest.json`, `assigned_shards.json`, `assigned_tasks.json`, `hints/`, `in/`, and `out/`, and the worker prompt already tells the agent exactly where to start.

- Observation: the hint sidecar exposes plenty of row-level detail but no packet-level verdict.
  Evidence: the bad worker-005 packet says `inside=0, outside=0, unknown=80` and lists row-by-row tags such as `title_like` and `instruction_like`, but it never states the governing interpretation “this packet is cookbook lesson prose, not recipe-local structure.”

- Observation: the “recipe-span status mix” is not currently trustworthy enough to anchor decisions.
  Evidence: an obviously recipe-shaped packet containing ingredient lines, title lines, yield lines, and instruction lines still reports `inside=0, outside=0, unknown=80`, so workers cannot safely lean on that field today.

- Observation: the current run’s harmful flips match the missing context exactly.
  Evidence: the preserved flip samples include `Balancing Fat`, `WHAT IS ACID?`, and `Cooking Acids` moving from baseline `KNOWLEDGE` to candidate `HOWTO_SECTION`, while narrative or broadly advisory prose moved from `OTHER` to `KNOWLEDGE`.

- Observation: the workspace runtime already supports `examples/`, but the line-role workers do not currently receive any example files.
  Evidence: `cookimport/llm/codex_exec_runner.py` already populates `examples_dir` and mirrors `examples/` into sterile workspaces, but preserved line-role `worker_manifest.json` files show `examples_dir: null`.

- Observation: additive packet-context fields fit the existing transport cleanly.
  Evidence: the validator and repair paths still key off `rows`, so adding top-level `packet_summary`, `default_posture`, `flip_policy`, and `example_files` did not require transport redesign and all parsing/workspace tests still passed.

- Observation: the current repo does not expose a model-free benchmark rerun that meaningfully reflects changed line-role prompts/hints.
  Evidence: after reviewing `docs/07-bench/07-bench_README.md` and the active `bench` / `labelstudio-benchmark` command surface, the offline paths can score existing predictions or fake transport, but not generate a new candidate whose labels depend on the changed worker context without live Codex execution.

## Context and Orientation

The line-role stage lives primarily in `cookimport/parsing/canonical_line_roles.py`. In this repository, “line-role” means assigning one freeform structural label such as `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `HOWTO_SECTION`, `KNOWLEDGE`, or `OTHER` to each atomic line candidate extracted from cookbook text. The current runtime already uses immutable task packets under `workers/*/in/*.json`, worker-written results under `workers/*/out/*.json`, and sidecar hints under `workers/*/hints/*.md`.

Three files matter most for this work. `cookimport/parsing/canonical_line_roles.py` currently builds packet hints, packet inputs, and the direct-exec runtime artifacts. `cookimport/llm/canonical_line_role_prompt.py` currently builds the worker prompt text and the compact label ontology wording. `cookimport/llm/codex_exec_runner.py` already supports mirrored `examples/` files in sterile workspaces and already writes `worker_manifest.json`; this plan uses that support rather than inventing a new transport.

The current hint generation seam is the block around `_write_line_role_worker_hint_markdown(...)` in `cookimport/parsing/canonical_line_roles.py` where `Packet profile`, `Label code legend`, and `Attention rows` are assembled. The current worker prompt seam is `build_canonical_line_role_file_prompt(...)` in `cookimport/llm/canonical_line_role_prompt.py` plus the workspace-worker prompt text exported into benchmark artifacts such as `prompts/prompt_line_role.txt`.

The preserved benchmark evidence to keep in mind is `data/golden/benchmark-vs-golden/2026-03-21_12.58.13/single-book-benchmark/saltfatacidheatcutdown/`. In that run, Codex improved some minority-label recovery but lost strict accuracy because it over-called semantic structure outside recipes. The plan should therefore optimize for fewer harmful semantic flips, not for higher label novelty.

## Milestones

### Milestone 1: Make packet hints express the packet’s governing interpretation

At the end of this milestone, each line-role packet hint will tell a cold-start worker what kind of packet it is before the worker reads the rows one by one. The hint should stop at the semantic level that matters: “recipe body,” “recipe-adjacent note/variation packet,” “cookbook lesson prose,” “memoir/front matter,” or “mixed packet with clear boundaries.” This is the missing information in the preserved bad packet. The acceptance test for this milestone is that the worker hint markdown includes a new packet-summary section and that the worst preserved lesson packet is explicitly described as non-recipe instructional prose rather than recipe-local structure.

### Milestone 2: Give the worker a packet-local decision tree instead of only label definitions

At the end of this milestone, the worker will receive a short ordered decision procedure that makes the main mistakes mechanically hard to repeat. The decision tree must tell the worker to classify packet mode first, then apply conservative label rules. The acceptance test for this milestone is prompt text and hint text that explicitly say, in plain language, that outside-recipe lesson headings default to `KNOWLEDGE`, memoir defaults to `OTHER`, generic cooking advice outside a recipe is not `INSTRUCTION_LINE`, and `HOWTO_SECTION` is recipe-internal only.

### Milestone 3: Calibrate trust in deterministic labels and add contrast examples

At the end of this milestone, workers will know that the input label code is a strong prior, not an invitation to relabel everything. The line-role worker workspace will also contain a small `examples/` directory with successful contrast examples that illustrate the hard boundaries this run violated. The acceptance test for this milestone is that `worker_manifest.json` points at `examples`, the sterile worker workspace mirrors those example files, and tests assert the presence of examples covering `KNOWLEDGE` versus `HOWTO_SECTION`, `OTHER` versus `KNOWLEDGE`, and `KNOWLEDGE` versus `INSTRUCTION_LINE`.

### Milestone 4: Tighten prompt wording around conservative flipping

At the end of this milestone, the top-level line-role prompt will reinforce a conservative operating model. It should explicitly prefer “keep the deterministic label unless packet-local evidence clearly contradicts it,” and it should warn that changing a heading to a more structured label requires stronger evidence than leaving it as `KNOWLEDGE` or `OTHER`. The acceptance test for this milestone is a prompt-preview test that asserts the new wording and a small fixture-driven labeling test that demonstrates the preserved bad examples now stay conservative.

### Milestone 5: Prove the behavior with tests and an offline benchmark compare

At the end of this milestone, repository tests and an offline benchmark replay will demonstrate that the artifact shape changed as intended and that the preserved bad packet family now yields fewer harmful flips. The default validation path must remain no-live-spend: unit tests, prompt preview, workspace artifact tests, and an offline single-book benchmark compare against the preserved source and gold export. A live Codex rerun may be documented as a follow-up step that requires explicit user approval, but it must not be part of the default milestone acceptance.

## Idempotence and Recovery

These changes are additive and safe to iterate on. Writing richer hint files, example files, and packet metadata should not mutate historical benchmark roots; only newly generated worker roots should carry the new artifacts. The new tests should be deterministic and runnable repeatedly.

If a prompt or hint wording change causes regressions, revert only the specific wording or packet-summary heuristic that produced the regression and rerun the focused tests before broadening scope again. Because the plan keeps the existing packet transport and direct-exec workspace structure, recovery should not require deleting caches or destroying benchmark artifacts.

If the offline benchmark compare fails to improve strict accuracy, keep the new artifact shape and use the preserved outputs to identify which packet-summary heuristic or negative rule still underspecifies the decision boundary. Do not respond by relaxing the worker back into open-ended semantic relabeling.
