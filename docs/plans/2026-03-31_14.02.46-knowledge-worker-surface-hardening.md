# Knowledge Worker Surface Hardening

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

After this change, the knowledge stage will be harder to game with cheap heuristics and easier for a workspace worker to complete by directly editing `task.json`. The worker will see smaller, more focused knowledge tasks, repair mode will stop ballooning into multi-megabyte feedback dumps, and the repo will have deterministic tests that pin the benchmark failure pattern we saw on `2026-03-31_13.18.44` without rerunning a live CodexFarm benchmark.

The user-visible proof is local and deterministic. Focused tests will show that knowledge repair files stay bounded, that grouping tasks no longer expand into giant multi-unit repair payloads, and that the knowledge worker contract remains helper-first while still preserving LLM authority for semantic `knowledge|other` calls and grouping.

## Surprises & Discoveries

- Observation: the benchmark quality drop was not a generic “CodexFarm got worse everywhere” failure. Boundary quality improved substantially while knowledge authority regressed.
  Evidence: `single_book_summary.md` for `2026-03-31_13.18.44` shows `strict_accuracy` slightly improved for `codexfarm`, while `macro_f1_excluding_other` fell and the dominant miss pattern was `KNOWLEDGE` versus `OTHER`.

- Observation: the current repo is already mid-flight on the shell-drift fix.
  Evidence: `cookimport/llm/codex_exec_runner.py`, `cookimport/llm/recipe_stage_shared.py`, `cookimport/llm/knowledge_stage/recovery.py`, `cookimport/parsing/canonical_line_roles/runtime.py`, and related tests already contain substantial uncommitted changes that add single-file drift policy tags, stricter prompt wording, and watchdog escalation.

- Observation: shrinking the knowledge worker surface is not only a prompt problem. Some task files are large enough to rationalize scripting even if the prompt is strict.
  Evidence: the saved bad-run file `data/output/2026-03-31_13.18.44/.../nonrecipe_finalize/workers/worker-001/task.json` is about 10.5 MB in repair mode with 159 units and no ontology payload, so the size blow-up comes from repeated per-unit repair metadata rather than the initial catalog alone.

- Observation: repeated validation feedback is the dominant contributor to the giant grouping repair file.
  Evidence: sampling one failed unit from that saved task file shows `validation_feedback` alone consumes roughly 46 KB for a single unit, while the actual `evidence` payload is only about 156 bytes and the answer payload is tiny.

- Observation: the first meaningful size win can happen without touching the hot shared-runner files.
  Evidence: `build_task_file_answer_feedback(...)` is only used by the knowledge task-file contracts, so shrinking that payload is isolated to knowledge-stage repair rewriting and its local tests.

- Observation: the knowledge stage already separates classification and grouping in same-session flow, which gives us a clean seam for further task-shape changes.
  Evidence: `cookimport/llm/knowledge_same_session_handoff.py` and `cookimport/llm/knowledge_stage/task_file_contracts.py` already model `nonrecipe_classify` and `knowledge_group` as separate validated stages.

## Decision Log

- Decision: build this plan on top of the shell-drift enforcement work instead of replacing it.
  Rationale: the prompt/classifier/watchdog slice is already coherent and directly addresses one observed failure mode. The larger knowledge-stage redesign should assume that vocabulary and enforcement model rather than inventing a parallel policy.
  Date/Author: 2026-03-31 / Codex

- Decision: treat giant repair files as a first-class product bug, not a cosmetic artifact issue.
  Rationale: when `task.json` reaches multi-megabyte scale, the worker has a stronger incentive to script around the file rather than inspect and edit it directly, which undermines the helper-first contract.
  Date/Author: 2026-03-31 / Codex

- Decision: keep deterministic code authoritative over validation and file shaping, but do not let deterministic code infer semantic knowledge labels.
  Rationale: the project rule is explicit: deterministic code packages evidence and validates outputs, while the LLM makes fuzzy semantic calls. Task-shape fixes must preserve that boundary.
  Date/Author: 2026-03-31 / Codex

- Decision: prefer narrower, additive task-shape changes over a total knowledge-stage rewrite.
  Rationale: the current stage already has useful seams for classification, grouping, repair, and final output assembly. We can reduce failure pressure by improving those seams without tearing out the whole runtime.
  Date/Author: 2026-03-31 / Codex

- Decision: implement repair-feedback slimming before the larger pass-splitting or grouping-rebatching work.
  Rationale: it directly targets the biggest artifact-proven source of task-file bloat, it is easy to validate locally, and it does not compound the merge risk in the already-active shared watchdog files.
  Date/Author: 2026-03-31 / Codex

## Outcomes & Retrospective

The first integrated slice is now complete: knowledge repair feedback is filtered per failed unit instead of duplicating the whole shared validation-detail list onto every failed row. That does not finish the broader knowledge worker hardening effort, but it removes the clearest artifact-proven source of repair-file bloat without touching the already-hot shared watchdog files.

Focused proof passed locally in `.venv`:

    pytest tests/llm/test_knowledge_same_session_handoff.py -q

The larger remaining work is still the same: finish the in-flight shell-drift slice, then shrink grouping pressure and possibly split knowledge semantics across smaller passes. The key lesson from the first slice is that not all “agent quality” fixes require prompt changes; sometimes the fastest win is simply removing duplicated deterministic scaffolding from the editable task file.

## Context and Orientation

In this repository, the knowledge stage is the optional LLM pass that decides whether outside-recipe blocks are `knowledge` or `other`, and, for the rows kept as knowledge, groups them into related topics. The stage lives under `cookimport/llm/knowledge_stage/`. The runtime entrypoint is `cookimport/llm/knowledge_stage/runtime.py`, the worker execution path is `cookimport/llm/knowledge_stage/workspace_run.py`, the prompt and watchdog logic live in `cookimport/llm/knowledge_stage/recovery.py`, the editable task-file contracts live in `cookimport/llm/knowledge_stage/task_file_contracts.py`, and the repo-owned same-session helper is `cookimport/llm/knowledge_same_session_handoff.py`.

The worker-visible contract for the happy path is a repo-written `task.json`. The worker should inspect the file with repo-owned helpers from `cookimport/llm/editable_task_file.py`, fill only `/units/*/answer`, run the repo-owned same-session handoff helper, and stop. Deterministic code validates the edited file, rewrites repair files when needed, and writes final output artifacts. Deterministic code must not invent semantic `knowledge|other` decisions.

The key failure in the `2026-03-31_13.18.44` benchmark run was not recipe boundary quality. It was knowledge authority quality. The saved artifacts show that workers drifted into ad hoc shell/Python heuristics and that the final knowledge authority collapsed many cookbook-explanation rows into `other`. The same artifacts also show task-surface pressure: a real grouping repair `task.json` grew to about 10.5 MB for 159 units, mostly because validation feedback was repeated per unit.

This plan intentionally assumes the current shell-drift enforcement work remains active and should be completed. That work is already tightening prompts, classifying discouraged single-file shell commands in `cookimport/llm/codex_exec_runner.py`, and escalating repeated shell drift in stage watchdogs. This plan starts where that work stops: once the paved road is explicit, make the paved road small and workable enough that the model can actually use it.

## Plan of Work

The first milestone is to finish the shell-drift enforcement slice and stabilize its test vocabulary. The active diff already introduces `single_file_*` policy labels and stricter prompt wording. Before adding more behavior, integrate and verify that slice so later knowledge-stage changes can depend on one shared command-classification contract.

The second milestone is repair-file slimming. Edit `cookimport/llm/editable_task_file.py` and the knowledge-stage transition logic in `cookimport/llm/knowledge_stage/task_file_contracts.py` so repair `task.json` files carry only the feedback actually needed to fix the failed units. The current shape appears to repeat large validation detail blobs on every failed unit. Replace that with narrow unit-local feedback or a compact shared top-level repair summary plus per-unit pointers. The worker must still have enough information to fix the task in one session, but not enough duplicated metadata to blow the file up into multi-megabyte territory.

The third milestone is grouping-task reshaping. Today `build_knowledge_grouping_task_file(...)` can hand a worker a large flat list of all kept-knowledge rows for every shard assigned to that worker. Rework the grouping path so the worker gets smaller grouping scopes. The safest version is to keep classification and grouping separate but to partition grouping into smaller task files, either one per shard or by a bounded row-count or character budget. The grouping step should stay semantically LLM-owned, but the deterministic planner should limit how much one same-session grouping file can contain.

The fourth milestone is saved-artifact regression proof. Add tests that mimic the bad run without invoking live CodexFarm. Use small synthetic fixtures and, where useful, saved artifact snippets to prove that repair files no longer explode in size, that grouping workloads are partitioned before they become giant same-session files, and that the helper-first prompt/contract still reflects the narrower shape. Add at least one test that asserts a saved bad-run repair payload would be reduced materially by the new shaping logic.

The fifth milestone is docs and plan completion. Update `docs/10-llm/10-llm_README.md` so the current knowledge worker contract explains both the shell-drift policy and the bounded knowledge-task surface. Then finish this plan with the final state, verification commands, and what still remains open.

## Validation and Acceptance

This plan is complete only when all of the following are true.

The first acceptance criterion is shared contract stability. The single-file worker shell-drift classifier and stage watchdog behavior must be deterministic and covered by focused tests. The knowledge stage must not be the odd stage out.

The second acceptance criterion is smaller knowledge repair files. A repair `task.json` for knowledge classification or grouping must no longer duplicate large feedback payloads per unit. The tests should assert that repair metadata stays compact and targeted.

The third acceptance criterion is bounded grouping scope. The deterministic planner must stop handing one same-session grouping task a giant flat list of knowledge rows. The worker should receive smaller grouping assignments that remain semantically meaningful but tractable for direct editing.

The fourth acceptance criterion is preserved authority boundaries. Deterministic code may reshape, summarize, or partition tasks, but it must not infer semantic `knowledge|other` labels or overwrite LLM semantic outputs.

The fifth acceptance criterion is local proof. The focused pytest commands above must pass in `.venv` without running live CodexFarm book processing or benchmarks.

## Idempotence and Recovery

These edits must be safe to rerun. Task-file shaping should be deterministic from the same input shards, and repair-task construction should remain stable across repeated validations. If a shaping change proves too aggressive, relax only the specific bound or repair field that caused the failure and add a regression test; do not reopen the broad multi-megabyte repair-file path.

Because the relevant worker files already have active uncommitted edits, integrate changes carefully and read the current diff before patching hot files. Do not overwrite the in-flight shell-drift vocabulary with a parallel naming scheme.

## Artifacts and Notes

The motivating evidence for this plan can be summarized plainly:

    The saved bad-run knowledge worker file `worker-001/task.json` reached about 10.5 MB in repair mode with 159 grouping units.
    In that same file, one unit’s `validation_feedback` blob was roughly 46 KB while the actual evidence payload was tiny.
    The benchmark artifact also showed that CodexFarm improved segmentation but still failed badly in final knowledge authority.

The intended end-state is also plain:

    Workers still make the semantic knowledge decisions.
    Deterministic code still validates and writes outputs.
    But the worker sees smaller, more direct tasks and compact repair context, so helper-first same-session editing remains realistic.
