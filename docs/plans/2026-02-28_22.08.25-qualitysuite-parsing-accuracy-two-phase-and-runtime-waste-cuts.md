## Purpose / Big Picture

After this change, the repository will have an explicit, low-latency operating model for answering "which deterministic parsing settings are best?" without launching day-scale sweeps-heavy tournaments. A maintainer will run a fast parser-focused Phase A flow to get a shortlist, then run a narrow Phase B confidence A/B to decide whether to promote defaults. The same work also removes avoidable runtime waste and trust gaps: race mode will stop doing extra rounds when pruning is impossible, tournament runs will allow explicit seed picking, tournament status will expose live fold/experiment subprogress, and prediction reuse will reduce copy overhead while proving reuse scope in tests and telemetry.

## Context and Orientation

QualitySuite behavior in this repository spans two layers. The lower layer is `cookimport bench quality-run`, implemented in `cookimport/bench/quality_runner.py`, which executes experiment variants over a discovered suite and supports `search_strategy` (`race` or `exhaustive`), deterministic sweeps toggles, parallel experiment scheduling, checkpoint/resume, and queue-aware ETA. The upper layer is `scripts/quality_top_tier_tournament.py`, which orchestrates multi-seed folds, deduplicates duplicate suites, runs fold-level quality jobs, applies gate-impossibility pruning, and writes tournament-level summary/report artifacts.

For this plan, "Phase A" means fast parser-focused discovery when no winner is known. "Phase B" means confidence A/B where one strong candidate is tested against baseline for promotion decisions. "Sweeps" means deterministic sweep expansion options that materially increase variant cardinality. "Race cannot prune" means race finalists are greater than or equal to effective variant count, so probe/mid rounds add work without eliminating configs.

Current relevant files and why they matter:

- `scripts/quality_top_tier_tournament.py`: seed handling, parser quick mode, fold orchestration, and current top-level progress behavior.
- `cookimport/bench/quality_runner.py`: search strategy execution path and race round pruning mechanics.
- `cookimport/cli.py`: prediction reuse cache keying, artifact reuse copy behavior, and reuse metrics.
- `data/golden/bench/quality/experiments/2026-02-28_16.24.30_qualitysuite-top-tier-tournament-full-candidates.json`: canonical candidate universe for parser + non-parser knobs.
- `data/golden/bench/quality/thresholds/2026-02-28_16.24.30_qualitysuite-top-tier-gates-fast-nosweeps.json` and `data/golden/bench/quality/thresholds/2026-02-28_20.20.04_qualitysuite-top-tier-gates-fast-sweeps-decision.json`: current gate/search presets.
- `docs/07-bench/runbook.md` and `docs/07-bench/07-bench_README.md`: operator-facing contract that must reflect new defaults/workflow.
- `tests/bench/test_quality_suite_runner.py` and `tests/labelstudio/test_labelstudio_benchmark_helpers.py`: existing quality/prediction-reuse coverage to extend.

## Plan of Work

This work is implemented as five milestones. Each milestone is independently testable and leaves behind explicit artifacts.

### Milestone 1: Codify the two-phase parser-accuracy operating model

Add explicit repository artifacts for Oracle-recommended run shape and promotion criteria so operators do not need ad hoc flag combinations.

Create new versioned profile files:

- `data/golden/bench/quality/experiments/<timestamp>_qualitysuite-parsing-phase-a-candidates.json`
- `data/golden/bench/quality/thresholds/<timestamp>_qualitysuite-parsing-phase-a-fast.json`
- `data/golden/bench/quality/thresholds/<timestamp>_qualitysuite-parsing-phase-b-confidence.json`

Phase A profile contract:

- Baseline + parser-focused candidates only (`pre_br_split`, `pre_none`, `skip_headers_false`, `parser_v2_pre_br_skiphf_false`).
- `max_targets=6`, `prefer_curated=true`.
- no deterministic sweeps.
- exhaustive search by default.
- seeds configured as `[42, 314, 2718, 4242]`; operational default should run three selected seeds to reduce duplicate-suite risk.

Phase B profile contract:

- baseline + one promoted candidate (CLI-selected winner from Phase A).
- no deterministic sweeps.
- exhaustive search by default.
- conservative feasible gates centered on `min_completed_folds=2`, `min_uplift_fold_ratio=0.5`, and `>= +0.004` mean strict/practical deltas with no source-success regression.

Update `docs/07-bench/runbook.md` and `docs/07-bench/07-bench_README.md` with:

- exact Phase A and Phase B commands using the new profile files;
- "good enough" criteria for Phase A shortlist and Phase B promotion;
- promotion trigger from A to B;
- a short "What to avoid" section copied from Oracle anti-patterns and translated into local command guidance.

### Milestone 2: Auto-switch race to exhaustive when race cannot prune

Implement this in `cookimport/bench/quality_runner.py` inside `_run_single_experiment(...)`.

Behavior contract:

- If `search_strategy == "race"` and suite has multiple targets, compute effective variant cardinality for the experiment before race rounds start (same run settings and deterministic-sweeps choice).
- If effective cardinality is less than or equal to `race_finalists`, skip race probe/mid/final staging and run one exhaustive pass directly.
- Persist effective-strategy metadata so artifacts explain what happened, including:
  - requested strategy (`race`);
  - effective strategy (`exhaustive`);
  - reason (`race_no_prune_variant_count_le_finalists`);
  - measured variant counts and finalists threshold.

Acceptance behavior:

- Existing race configuration still works when pruning is feasible.
- No-prune cases complete faster and do not write misleading race-round metadata.

### Milestone 3: Add explicit seed selection and parser-fast defaults that match Oracle guidance

Implement this in `scripts/quality_top_tier_tournament.py`.

Add CLI options:

- `--seed <int>` repeatable, preserving provided order.
- `--seed-list "42,2718,4242"` comma-separated alternative input.

Seed resolution rules:

- If `--seed`/`--seed-list` is provided, use those seeds exactly (deduplicated, in order of appearance) instead of "first N from thresholds".
- If none are provided, fall back to threshold seed list and optional `--max-seeds`.
- When `--quick-parsing` is enabled and no explicit seed input is provided, default cap becomes three seeds (not two), still overrideable via `--max-seeds`.
- Update `tournament_resolved.json` to record seed source and resolved explicit seed plan.

Validation rules:

- reject invalid seed tokens and empty effective seed list;
- reject simultaneous contradictory usage (`--seed` with malformed `--seed-list`);
- maintain deterministic fold naming and duplicate-suite dedupe behavior.

### Milestone 4: Add tournament subprogress so "0/N" does not look stuck

Implement live fold-level subprogress in `scripts/quality_top_tier_tournament.py` while each fold `quality-run` subprocess is active.

Implementation contract:

- Replace one-shot blocking call for fold quality runs with a polled subprocess loop.
- Poll current fold run directory for `checkpoint.json` and `summary.partial.json` (when present) and emit stable, throttled updates only when values change.
- Progress line includes at minimum:
  - fold index / configured folds;
  - experiment_count_completed / experiment_count_total from fold checkpoint;
  - pending experiment count;
  - current fold run directory.
- Write top-level `tournament_checkpoint.json` updates so progress is visible from filesystem even without live terminal attachment.

Acceptance behavior:

- Long-running folds continuously expose progress evidence.
- Existing resume behavior remains intact and no duplicate fold execution is introduced.

### Milestone 5: Complete prediction-reuse runtime optimization and telemetry clarity

Use `cookimport/cli.py` to reduce I/O cost of reuse and make reuse-source visibility explicit.

Implementation contract:

- Keep current cache key semantics unchanged (source path + prediction-affecting run settings), but optimize artifact materialization:
  - add a copy helper that attempts hardlink-based tree materialization when possible and falls back to normal copy on unsupported filesystems;
  - preserve current safe behavior when source and target overlap or when source artifacts are missing.
- Extend prediction reuse result-source classification to distinguish:
  - reuse from current run tree;
  - reuse from shared cache entries rooted outside current run tree.
- Extend summary aggregation to report both in-run and cross-run prediction reuse counts.
- Ensure quality/tournament docs mention the updated telemetry fields.

Acceptance behavior:

- Existing cross-root reuse tests still pass.
- New telemetry clearly indicates when reuse came from prior roots.
- Reuse copy wall time decreases in representative synthetic tests (or, if host filesystem prevents linking, behavior remains functionally identical with explicit fallback path covered by tests).

## Idempotence and Recovery

All changes are additive and safe to rerun.

- Tournament and quality-run outputs remain timestamped and resume-capable.
- New profiles are static JSON artifacts; rerunning commands creates new run roots unless explicit resume directories are passed.
- Seed resolution is deterministic and repeatable with explicit seeds.
- Prediction reuse optimization must preserve fallback behavior so unsupported link operations degrade safely to copying instead of failing runs.

If any new behavior causes unexpected regressions, safe rollback path is to:

- stop using new phase profiles and use prior known-good thresholds/experiments files;
- omit new seed flags (fall back to thresholds seed order);
- set strategy explicitly to exhaustive;
- rely on existing cache behavior if reuse optimization code path is disabled by fallback.
