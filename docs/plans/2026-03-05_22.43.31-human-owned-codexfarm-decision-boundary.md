# Make Codex Usage Explicit, Canonical, and Auditable

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds. This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The goal of this change is not to promise impossible security inside an agent-controlled environment. The real goal is the practical one that matters here: CodexFarm must never happen silently again.

After this work, a normal repo run should always make one explicit choice about AI assistance before any Codex-backed behavior can execute. A user should be able to run `cookimport stage`, `cookimport labelstudio-import`, `cookimport bench speed-run`, or `cookimport bench quality-run` and immediately see whether the run is `vanilla`, `codex_recipe`, or `codex_full`. If the command or settings imply Codex in an ambiguous way, the run must fail before work starts. If Codex is used, the run manifest must say so plainly.

This is the compromise that fits this repo. CodexFarm stays available because it is useful, but every Codex path becomes explicit, named, validated, and easy to audit.

## Surprises & Discoveries

- Observation: the low-level Codex CLI kill switch already exists, but it is not the main policy seam.
  Evidence: `cookimport/llm/codex_exec.py` raises unless `COOKIMPORT_ALLOW_LLM=1`, yet this only protects direct `codex exec` usage and does not explain intent at the run-planning level.

- Observation: the repo already contains a good pattern for explicit Codex approval in benchmark batch flows.
  Evidence: `cookimport/cli.py` requires explicit confirmation tokens for `bench speed-run --include-codex-farm` and `bench quality-run --include-codex-farm`, and `tests/bench/test_bench_speed_cli.py` plus `tests/bench/test_bench_quality_cli.py` assert that behavior.

- Observation: the current protection is fragmented instead of centralized.
  Evidence: `stage` and `labelstudio-import` accept `--llm-recipe-pipeline codex-farm-3pass-v1` directly in `cookimport/cli.py` without going through the same benchmark confirmation seam.

- Observation: one settings constructor still encodes the old “Codex-on” posture even though the `RunSettings` model fields were already flipped to safe defaults.
  Evidence: `build_run_settings(...)` in `cookimport/config/run_settings.py` still defaults `llm_recipe_pipeline`, `atomic_block_splitter`, and `line_role_pipeline` to Codex-enabled values in its function signature.

- Observation: patching individual commands will not be enough.
  Evidence: the same Codex-relevant settings are assembled in `cookimport/cli.py`, `cookimport/labelstudio/ingest.py`, benchmark helper builders, and `build_run_settings(...)`, so any one-off fix leaves more bypasses behind.

- Observation: truly “human only” cannot be guaranteed from inside this repo.
  Evidence: if the agent can edit repo code and run commands in the same environment, any repo-local approval file, config bit, or prompt can be changed by code as well.

These discoveries matter because they narrow the correct target. The repo needs one explicit decision model that is enforced everywhere, not a growing collection of local exceptions.

## Decision Log

- Decision: define success as “Codex can never happen silently or ambiguously,” not as “Codex is cryptographically impossible without a human.”
  Rationale: the second goal is not achievable inside this environment. Chasing it would produce false confidence. The first goal is realistic and directly addresses the QualitySuite failure mode.
  Date/Author: 2026-03-05 / Codex

- Decision: introduce canonical named run profiles as the primary operator interface.
  Rationale: the dangerous bug class in this repo is hidden hybrid state assembled from many low-level flags. Canonical profiles make intent first-class and make hybrids explicit instead of accidental.
  Date/Author: 2026-03-05 / Codex

- Decision: keep raw Codex-related knobs only as advanced inputs, and validate them against the named profile instead of letting them define behavior alone.
  Rationale: removing all raw knobs would be disruptive, but leaving them unconstrained would preserve the same ambiguity that caused the current problem.
  Date/Author: 2026-03-05 / Codex

- Decision: use fail-closed validation for ordinary run surfaces.
  Rationale: if a run looks even slightly ambiguous, the safe behavior is to stop before work starts and force the caller to choose a canonical profile or a clearly marked advanced override path.
  Date/Author: 2026-03-05 / Codex

- Decision: preserve and reuse the existing benchmark confirmation-token pattern rather than inventing a second approval style for batch runs.
  Rationale: speed and quality benchmarks already teach the right habit. The new policy layer should route through those existing confirmations instead of competing with them.
  Date/Author: 2026-03-05 / Codex

- Decision: require loud audit metadata in manifests and CLI output even after validation exists.
  Rationale: validation prevents many mistakes, but the operator still needs immediate visibility and an after-the-fact paper trail when Codex was used intentionally.

## Outcomes & Retrospective

This plan has not been implemented yet. Its main outcome so far is a clearer and safer problem definition.

The repo should not try to create “perfect” human-only security that it cannot actually enforce. Instead, it should make Codex usage explicit, canonical, and impossible to miss. If the implementation follows this plan, the next similar bug should turn into an immediate validation error or an obvious manifest entry rather than a surprise token leak discovered later in analytics.

The main lesson from the research phase is that safe defaults alone are necessary but not sufficient. The repo also needs a single decision boundary that classifies every run before execution.

## Context and Orientation

This repository has several places where Codex-backed behavior can be enabled.

The canonical settings model lives in `cookimport/config/run_settings.py`. `RunSettings` is the validated settings object used across CLI, staging, Label Studio, and benchmark flows. A second helper in the same file, `build_run_settings(...)`, constructs `RunSettings` from many individual arguments and is used by `cookimport/cli.py` and `cookimport/labelstudio/ingest.py`.

The top-level command surfaces live in `cookimport/cli.py`. The important ones for this plan are:

- `stage(...)`
- `labelstudio_import(...)`
- `bench_speed_run(...)`
- `bench_quality_run(...)`

The Label Studio prediction-run assembly path lives in `cookimport/labelstudio/ingest.py`, where `build_run_settings(...)` is called again outside the main CLI stage flow.

The low-level direct Codex CLI seam lives in `cookimport/llm/codex_exec.py`. That file already has a safety kill switch controlled by `COOKIMPORT_ALLOW_LLM`, but that switch is not a substitute for a user-facing decision model because it does not explain which run intended to use Codex or why.

The benchmark runners already have partial explicit-approval support:

- `cookimport/cli.py:_resolve_speedsuite_codex_farm_confirmation(...)`
- `cookimport/cli.py:_resolve_qualitysuite_codex_farm_confirmation(...)`
- `cookimport/bench/speed_runner.py`
- `cookimport/bench/quality_runner.py`

The current problem is that these protections are not universal. Some flows still accept Codex-enabling settings directly, and some helper functions can still default to Codex-backed behavior without one named “decision.”

Terms used in this plan:

A “run profile” is a short name for a fully specified Codex posture. In this plan the canonical profiles are:

- `vanilla`: no Codex-backed recipe parsing, no Codex-backed line-role labeling, no atomic benchmark block splitting, no Codex knowledge pass, no Codex tags pass.
- `codex_recipe`: recipe Codex is on, but line-role Codex and atomic splitting stay off.
- `codex_full`: recipe Codex, line-role Codex, and atomic splitting are on together. Optional knowledge/tags passes remain off unless this repo already treats them as part of the same profile and the implementation chooses to include them explicitly.

A “decision boundary” is the one place where raw settings are translated into one of these named profiles or rejected as ambiguous.

A “fail-closed” policy means the code refuses to proceed when intent is unclear, rather than guessing.

## Plan of Work

### Milestone 1: Add one shared Codex decision model

Create a new module under `cookimport/config/` with a clear name such as `codex_policy.py` or `run_profiles.py`. This module should become the single place that answers four questions:

- Which canonical profile is being requested.
- Which low-level settings that profile implies.
- Whether the final payload uses any Codex-backed behavior.
- Whether the payload is valid, ambiguous, or inconsistent.

Define a small enum for the profile name. Use a stable name such as `RunProfile` with values `vanilla`, `codex_recipe`, and `codex_full`. Also define a small structured result type, for example `ResolvedCodexDecision`, that records:

- `profile`
- `uses_codex_farm`
- `uses_line_role_codex`
- `uses_any_ai`
- `requires_explicit_confirmation`
- `derived_from_fields`
- `warnings`

This module must expose two kinds of helpers.

The first helper applies a canonical profile to a plain settings payload. It should take a dictionary and return a new dictionary with the profile’s required fields forced into place. For `vanilla`, that means all Codex-related fields are forced off. For `codex_recipe`, recipe Codex is on and the rest of the line-role and atomic benchmarking surfaces stay off. For `codex_full`, recipe Codex, line-role Codex, and atomic splitting are all on in a consistent way.

The second helper validates a final payload. If the payload mixes fields in a way that does not match the requested profile, it must return a validation error instead of guessing. For example, `llm_recipe_pipeline=off` plus `line_role_pipeline=codex-line-role-v1` with requested profile `vanilla` is invalid and should fail.

This milestone is complete when the repo has one shared module that can both construct and validate Codex posture without relying on scattered ad hoc checks.

### Milestone 2: Align settings constructors with the canonical profiles

Edit `cookimport/config/run_settings.py`.

The `RunSettings` model fields already use safe defaults. Keep that. The important change is to remove the stale Codex-on defaults from `build_run_settings(...)`. Its signature must stop defaulting to `codex-farm-3pass-v1`, `codex-line-role-v1`, and `atomic-v1`. Make the helper align with the safe `vanilla` posture so helper-level construction cannot silently disagree with the model-level defaults.

Then thread the new profile helper through `build_run_settings(...)` so that the helper either:

- accepts an explicit profile argument and applies it before validation, or
- accepts the raw fields but immediately classifies and validates them through the shared decision model.

The important rule is that `build_run_settings(...)` must stop being a bypass around the new policy layer.

This milestone is complete when every route that constructs `RunSettings` through this helper either resolves to a named profile or fails as ambiguous.

### Milestone 3: Move standard CLI flows onto the decision boundary

Edit `cookimport/cli.py` so the major user-facing run surfaces share one policy.

For `stage(...)` and `labelstudio_import(...)`, add a new top-level option such as `--run-profile` or `--ai-profile` with choices `vanilla`, `codex_recipe`, and `codex_full`. Default it to `vanilla`.

Keep the existing low-level Codex-related options for advanced use, but demote them conceptually. The command must now:

1. gather the raw option payload,
2. apply the requested profile through the shared helper,
3. validate that any raw overrides are still consistent with the profile,
4. fail before execution if they are not.

For ordinary commands, if the resolved profile is not `vanilla`, require one explicit approval seam. For `stage(...)` and `labelstudio_import(...)`, a boolean like `--allow-codex` is sufficient because these commands are interactive one-off flows rather than large benchmark batches. The approval is not a security barrier; it is an intent barrier.

When the command starts, print a loud summary line such as:

    RUN MODE: VANILLA (NO CODEXFARM)

or

    RUN MODE: CODEX_RECIPE (EXPLICIT)

This milestone is complete when standard one-off commands cannot drift into Codex by raw field inheritance alone.

### Milestone 4: Rebuild benchmark variant generation around canonical profiles

Edit the benchmark-related builders in `cookimport/cli.py`, plus any supporting code in `cookimport/bench/speed_runner.py` and `cookimport/bench/quality_runner.py`.

QualitySuite and SpeedSuite should stop constructing important baseline/Codex pairs by copying generic payloads and patching one field at a time. Instead, they should build variants from canonical profiles.

The deterministic baseline contract must be generated from `vanilla`, not from inherited low-level fields. The Codex recipe candidate must be generated from `codex_recipe`. The full Codex candidate must be generated from `codex_full`.

Preserve the existing confirmation tokens for benchmark flows:

- `--speedsuite-codex-farm-confirmation`
- `--qualitysuite-codex-farm-confirmation`

But route them through the same shared decision model so the benchmark runner knows exactly which canonical profile needs approval and why.

If this plan is implemented fully, QualitySuite baseline artifacts such as `experiments_resolved.json` should never again show hybrid payloads like recipe Codex off while line-role Codex stays on. If such a combination is somehow requested, the build step should fail before any benchmark runs.

This milestone is complete when benchmark planning emits only named canonical profiles or explicit validation errors.

### Milestone 5: Persist audit metadata and update reporting surfaces

Every run that writes a manifest should include explicit Codex-decision metadata. The exact field names can be chosen during implementation, but the manifest must record at least:

- `resolved_run_profile`
- `uses_codex_farm`
- `uses_line_role_codex`
- `uses_any_ai`
- `codex_confirmation_required`
- `codex_confirmation_provided`
- `codex_decision_source` such as `default`, `profile`, or `validated_raw_overrides`

Add this metadata to the stage summary path, Label Studio prediction-run manifests, and benchmark run manifests.

Then update the human-readable CLI summaries and any nearby report text so the operator sees the same named profile during execution and after the run.

This milestone is complete when a future audit can determine from the manifest alone whether Codex was used intentionally.

### Milestone 6: Add regression tests for ambiguity, validation, and auditability

Add focused tests in the modules already covering these surfaces.

In `tests/llm/test_run_settings.py` or a new config-focused test module, add tests for the shared profile helper:

- `vanilla` forces all Codex-relevant fields off.
- `codex_recipe` enables only the recipe Codex field.
- `codex_full` enables the full recipe plus line-role plus atomic combination.
- inconsistent raw payloads fail validation instead of being silently accepted.

In `tests/cli/test_cli_output_structure_fast.py`, `tests/cli/test_run_settings_adapters.py`, and any stage/import tests, add assertions that:

- ordinary commands default to `vanilla`,
- explicit Codex profile requests require the new approval flag,
- ambiguous combinations fail before execution.

In `tests/bench/test_bench_speed_cli.py`, `tests/bench/test_bench_quality_cli.py`, `tests/bench/test_speed_suite_runner.py`, and `tests/bench/test_quality_suite_runner.py`, keep the existing token-confirmation behavior but add assertions that the resolved profile is canonical and that hidden hybrid configurations are rejected.

In `tests/labelstudio/test_labelstudio_benchmark_helpers_scheduler.py`, add regression coverage proving that all generated benchmark baselines are `vanilla` and any Codex variants are explicit named profiles rather than inherited mixed payloads.

This milestone is complete when the new tests fail against the old fragmented behavior and pass after the refactor.

## Validation and Acceptance

Acceptance is behavioral.

First, `RunSettings` construction must no longer have split-brain defaults. A bare `RunSettings()` and a helper-built default settings object must both represent the same `vanilla` posture.

Second, a user running ordinary commands without Codex intent must see `vanilla` explicitly. The CLI should print a clear mode summary, and the manifest should record the same mode.

Third, a user trying to enable Codex through raw low-level flags alone must get a fail-closed error if the fields do not match a canonical profile. The system must stop before execution instead of guessing.

Fourth, a user intentionally enabling Codex must do so through a named profile and an explicit approval seam. For benchmark flows, the existing confirmation tokens must still be required. For one-off flows, the new explicit approval flag must be required.

Fifth, benchmark planning artifacts must become trustworthy. A human should be able to inspect `experiments_resolved.json` and tell immediately which experiments are `vanilla`, which are `codex_recipe`, and which are `codex_full`.

The minimum acceptance test command set is:

    pytest tests/llm/test_run_settings.py \
      tests/cli/test_run_settings_adapters.py \
      tests/cli/test_cli_output_structure_fast.py \
      tests/bench/test_bench_speed_cli.py \
      tests/bench/test_bench_quality_cli.py \
      tests/bench/test_speed_suite_runner.py \
      tests/bench/test_quality_suite_runner.py \
      tests/labelstudio/test_labelstudio_benchmark_helpers_scheduler.py -q

After those tests pass, at least one real benchmark artifact and one real one-off CLI run should be inspected manually to prove the profile summary and manifest metadata are visible.

## Idempotence and Recovery

This plan is safe to implement incrementally. The early steps are additive: a new profile helper module, additional validation, and more explicit manifest fields. The work should be done in a way that lets the repo keep functioning while tests are updated alongside the new policy.

If a command starts failing after the new validator lands, first inspect whether the caller was depending on an ambiguous mixed payload. The right fix is usually to move that caller onto a canonical profile rather than weakening the validator.

If a benchmark test count changes because formerly distinct hybrid payloads now collapse into the same `vanilla` baseline, update the test and explain the collapse in a short comment. Do not “fix” that by reintroducing hidden differences.

Rollback is straightforward. Because the change is mostly policy logic and tests, reverting the profile module and the CLI validation paths restores the old behavior without requiring data migration.

## Interfaces and Dependencies

The implementation should stay within the current repo structure and avoid new external services.

Add one new module under `cookimport/config/` with stable public helpers, for example:

    class RunProfile(str, Enum):
        vanilla = "vanilla"
        codex_recipe = "codex_recipe"
        codex_full = "codex_full"

    @dataclass(frozen=True)
    class ResolvedCodexDecision:
        profile: RunProfile
        uses_codex_farm: bool
        uses_line_role_codex: bool
        uses_any_ai: bool
        requires_explicit_confirmation: bool
        derived_from_fields: tuple[str, ...]
        warnings: tuple[str, ...] = ()

    def apply_run_profile(payload: Mapping[str, Any], profile: RunProfile) -> dict[str, Any]:
        ...

    def resolve_codex_decision(
        payload: Mapping[str, Any],
        *,
        requested_profile: RunProfile | None,
        allow_advanced_mixed_payload: bool = False,
    ) -> tuple[dict[str, Any], ResolvedCodexDecision]:
        ...

`cookimport/config/run_settings.py` should either import these helpers directly or expose a thin adapter around them, but the repo must end with exactly one source of truth for profile application and ambiguity checks.

`cookimport/cli.py`, `cookimport/labelstudio/ingest.py`, `cookimport/bench/speed_runner.py`, and `cookimport/bench/quality_runner.py` must all depend on this shared policy layer instead of locally re-deriving Codex posture.

Keep the existing benchmark confirmation tokens and the low-level `COOKIMPORT_ALLOW_LLM` kill switch. This plan complements them; it does not replace them.
