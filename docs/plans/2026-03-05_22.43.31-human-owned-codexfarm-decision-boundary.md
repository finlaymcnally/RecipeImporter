# Make Codex Decisions Explicit, Context-Aware, and Auditable

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds. This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The goal of this change is not to promise impossible security inside an agent-controlled environment. The real goal is the practical one that matters here: Codex-backed behavior must never happen silently again.

After this work, every ordinary run in this repository should pass through one explicit Codex-decision layer before any Codex-backed behavior executes. A user should be able to run `cookimport stage`, `cookimport labelstudio-import`, `cookimport labelstudio-benchmark`, `cookimport bench speed-run`, `cookimport bench quality-run`, or the `import` entrypoint and answer three simple questions from the artifacts and CLI output alone:

- Did this run use any Codex-backed surface?
- If yes, which Codex-backed surface was enabled?
- Was that activation allowed by the command‚Äôs policy and made explicit at the command boundary?

The user-visible behavior is not ‚Äúperfect human-only security.‚Äù The user-visible behavior is that hidden or ambiguous Codex activation turns into a clear validation error, and intentional Codex activation becomes loudly named and easy to audit.

This plan is intentionally broader and more detailed than the first version of this ExecPlan because the repo has evolved since that draft. The current code already contains several partial decision layers and profile concepts. The implementation must build on those real seams instead of inventing a second conflicting vocabulary.

- Observation: the repo already has a real interactive profile family concept.

- Observation: the repo already has a benchmark baseline/Codex contract for paired runs.
  Evidence: `cookimport/cli.py` now contains `_all_method_apply_baseline_contract(...)` and `_all_method_apply_codex_contract_from_baseline(...)`, and `_interactive_single_offline_variants(...)` reuses them to produce explicit `vanilla` and `codexfarm` paired variants.

- Observation: the repo already has a second, separate runtime-classification vocabulary in analytics.
  Evidence: `cookimport/analytics/benchmark_semantics.py` distinguishes `benchmark_variant` (`vanilla`, `codexfarm`) from `ai_assistance_profile` (`deterministic`, `line_role_only`, `recipe_only`, `full_stack`, `other`).

- Observation: the first version of this plan incorrectly assumed one profile axis could serve every purpose.
  Evidence: the live code now has at least three distinct concepts that should not be collapsed without care:
  - interactive top-tier family (`codexfarm` / `vanilla`),
  - benchmark paired variant identity (`vanilla` / `codexfarm` only for official paired benchmark paths),
  - runtime AI-assistance or Codex-surface classification.

- Observation: the helper-level default bypass is still real.
  Evidence: `RunSettings` model fields default `llm_recipe_pipeline`, `line_role_pipeline`, and `atomic_block_splitter` to `off`, but `build_run_settings(...)` in `cookimport/config/run_settings.py` still defaults them to `codex-farm-3pass-v1`, `codex-line-role-v1`, and `atomic-v1`.

- Observation: the current docs are internally inconsistent about defaults and policy.
  Evidence: `docs/02-cli/02-cli_README.md` now documents `llm_recipe_pipeline` default `off`, while `docs/10-llm/10-llm_README.md` still says ‚ÄúGlobal defaults now use the top-tier profile‚Äù with Codex-enabled values.

- Observation: direct non-interactive entrypoints still matter.
  Evidence: `cookimport/entrypoint.py` loads global settings, builds `RunSettings`, maps them through `build_stage_call_kwargs_from_run_settings(...)`, and calls `stage(...)` directly, so command-line validation must not live only in the interactive chooser.

- Observation: ‚ÄúCodex-backed‚Äù and ‚Äúnon-default deterministic‚Äù are not the same thing.
  Evidence: the repo supports `line_role_pipeline=deterministic-v1` and `line_role_pipeline=codex-line-role-v1`. The implementation must not treat both as the same Codex activation even though both are non-`off`.

These discoveries matter because they change the shape of the correct solution. The repo does not need one new universal profile label. It needs one universal decision layer that understands the already-existing concepts and keeps them consistent.

## Outcomes & Retrospective

This plan has still not been implemented, but it has been materially improved by the second review.

The first version got the spirit right, but it tried to solve the problem with one new profile axis. The live repo is more nuanced than that. It already has a two-family interactive chooser, benchmark-specific variant contracts, and analytics-specific runtime labels. The updated plan now reflects that reality and spells out how to unify these layers without flattening them into one misleading name.

The main lesson from the second review is that the implementation should centralize policy, not necessarily centralize every label. The repo needs one source of truth for Codex-backed surface classification and per-command validation, but it can still preserve different operator-facing labels where they already serve distinct purposes.

## Context and Orientation

This repository has several distinct places where Codex-related intent is currently represented.

The canonical settings model is `cookimport/config/run_settings.py`. `RunSettings` is the validated per-run settings object used across CLI, staging, Label Studio, analytics, and benchmark flows. The same file also contains `build_run_settings(...)`, a helper that assembles a `RunSettings` from many raw keyword arguments. That helper is used in both `cookimport/cli.py` and `cookimport/labelstudio/ingest.py`.

The main CLI entrypoints live in `cookimport/cli.py`. The important ones for this plan are:

- `stage(...)`
- `labelstudio_import(...)`
- `labelstudio_benchmark(...)`
- `bench_speed_run(...)`
- `bench_quality_run(...)`

A separate non-interactive entrypoint lives in `cookimport/entrypoint.py`. It loads global settings, converts them to `RunSettings`, maps them into `stage(...)` kwargs through `cookimport/config/run_settings_adapters.py`, and then calls `stage(...)` directly. This matters because policy enforcement cannot live only in interactive flows.

Interactive import and benchmark runs already have a shared chooser in `cookimport/cli_ui/run_settings_flow.py`. That chooser exposes two profile families:

- `codexfarm`
- `vanilla`

Those are not generic runtime truth labels. They are interactive top-tier family names. The `codexfarm` family harmonizes settings to recipe Codex plus Codex line-role plus atomic splitter. The `vanilla` family harmonizes settings to recipe Codex off, deterministic line-role on, and atomic splitter on. This means interactive `vanilla` does not mean ‚Äúall LLM-related knobs off.‚Äù

Benchmark planning already has another contract in `cookimport/cli.py`:

- `_all_method_apply_baseline_contract(...)` creates a deterministic benchmark baseline payload by forcing `llm_recipe_pipeline=off`, `llm_knowledge_pipeline=off`, `llm_tags_pipeline=off`, `line_role_pipeline=off`, and `atomic_block_splitter=off`.
- `_all_method_apply_codex_contract_from_baseline(...)` creates the paired benchmark Codex payload by re-enabling `llm_recipe_pipeline=codex-farm-3pass-v1`, `line_role_pipeline=codex-line-role-v1`, and `atomic_block_splitter=atomic-v1`.

Analytics has yet another layer in `cookimport/analytics/benchmark_semantics.py`. That module distinguishes:

- `benchmark_variant`: labels such as `vanilla` and `codexfarm` that are reserved for official paired benchmark paths when runtime settings match those meanings.
- `ai_assistance_profile`: labels such as `deterministic`, `line_role_only`, `recipe_only`, `full_stack`, and `other`.

The implementation in this plan must respect all three layers and make them agree.

Terms used in this plan:

A ‚ÄúCodex-backed surface‚Äù means a run setting that causes the repo to call a Codex-backed path rather than a deterministic path. Today that includes at least:

- recipe Codex via `llm_recipe_pipeline=codex-farm-3pass-v1`
- Codex line-role via `line_role_pipeline=codex-line-role-v1`
- knowledge Codex via `llm_knowledge_pipeline=codex-farm-knowledge-v1`
- tag Codex via `llm_tags_pipeline=codex-farm-tags-v1`

A ‚Äúdeterministic helper surface‚Äù means a non-`off` processing path that does not call Codex. Today the main example is `line_role_pipeline=deterministic-v1`.

A ‚Äútop-tier family‚Äù means the current interactive chooser‚Äôs two-family UI, `codexfarm` or `vanilla`.

A ‚Äúpaired benchmark variant‚Äù means the official paired benchmark labels `vanilla` and `codexfarm` used only when runtime settings actually match those benchmark contracts.

A ‚Äúdecision layer‚Äù means one shared module that can:

- classify actual Codex-backed surfaces from a raw or validated payload,
- apply known contract patches such as the interactive top-tier families and paired benchmark variants,
- validate whether a command is allowed to run with that resolved Codex posture,
- say whether explicit command-line approval is required.

A ‚Äúfail-closed‚Äù policy means the command stops before any work starts when the resolved Codex posture is ambiguous or disallowed for that command context.

## Plan of Work

### Milestone 1: Introduce one shared Codex-decision module

Create a new module in `cookimport/config/` with a clear name such as `codex_policy.py` or `codex_decision.py`.

This module should not replace existing labels. It should become the one source of truth for the facts that existing labels depend on.

Define one structured result type, for example `CodexSurfaceDecision`, that records at least:

- whether recipe Codex is enabled
- whether line-role Codex is enabled
- whether deterministic line-role is enabled
- whether knowledge Codex is enabled
- whether tags Codex is enabled
- whether any Codex-backed surface is enabled
- whether explicit command-level approval is required
- which fields or contract helper produced the final posture
- warnings or validation problems

The critical detail is that this classification must distinguish `line_role_pipeline=deterministic-v1` from `line_role_pipeline=codex-line-role-v1`. The first is a deterministic helper surface. The second is a Codex-backed surface.

Also define contract-application helpers in the same module for the current live concepts that already exist elsewhere in the repo. At minimum, include helpers for:

- interactive top-tier family `codexfarm`
- interactive top-tier family `vanilla`
- paired benchmark baseline variant `vanilla`
- paired benchmark Codex variant `codexfarm`

The interactive helpers should preserve the current semantics from `cookimport/cli_ui/run_settings_flow.py`. The benchmark helpers should preserve the current semantics from `_all_method_apply_baseline_contract(...)` and `_all_method_apply_codex_contract_from_baseline(...)` in `cookimport/cli.py`.

This milestone is complete when one module in `cookimport/config/` can classify actual Codex-backed behavior and apply the currently known Codex-related contracts without inventing a new conflicting vocabulary.

### Milestone 2: Move existing helper logic into the shared decision module

Update the current code so existing contract implementations call into the shared module instead of keeping separate local copies.

The main targets are:

- `cookimport/cli_ui/run_settings_flow.py`
- `cookimport/cli.py:_all_method_apply_baseline_contract(...)`
- `cookimport/cli.py:_all_method_apply_codex_contract_from_baseline(...)`
- any similar benchmark or interactive payload patching code that still exists locally

The important outcome is not that every helper disappears. The important outcome is that these helpers become thin wrappers over the same shared policy functions.

This is where the plan should also make the current top-tier chooser text more precise if needed. The interactive prompt and help text should make clear that:

- `CodexFarm automatic top-tier` enables Codex-backed recipe and line-role paths.
- `Vanilla automatic top-tier` keeps recipe Codex off but still enables deterministic line-role and atomic splitting.

That wording matters because the current family name `vanilla` does not mean ‚Äúall related knobs off.‚Äù

This milestone is complete when the live chooser and benchmark planners no longer encode their own disconnected Codex contracts.

### Milestone 3: Remove stale helper defaults and route helper construction through policy

Edit `cookimport/config/run_settings.py`.

`RunSettings` model defaults already resolve to safe values. Keep that. The key change is to remove the stale Codex-on defaults from `build_run_settings(...)` so helper-based construction starts from the same safe posture as the model itself.

Specifically, change the helper signature defaults so they no longer default to:

- `llm_recipe_pipeline = codex-farm-3pass-v1`
- `line_role_pipeline = codex-line-role-v1`
- `atomic_block_splitter = atomic-v1`

Once the defaults are safe, run the raw values assembled inside `build_run_settings(...)` through the shared Codex-decision module before returning a `RunSettings`. The helper does not need to expose a new public profile argument if the current repo conventions do not want that. It does need to stop being a bypass around policy.

This milestone is complete when helper-based construction can no longer silently disagree with `RunSettings()` defaults or skip Codex validation.

### Milestone 4: Add command-context validation for every standard run surface

Add one shared validation entrypoint in the new policy module that accepts a command context name and the assembled run payload.

The first implementation should cover these contexts:

- `stage`
- `labelstudio_import`
- `labelstudio_benchmark`
- `bench_speed_run`
- `bench_quality_run`
- `entrypoint_stage`

For each context, define the allowed deterministic and Codex-backed postures.

For `stage` and `labelstudio_import`, the important rule is simple: any Codex-backed recipe, knowledge, or tag surface requires an explicit command-level approval seam. A boolean like `--allow-codex` is enough for these one-off commands. Deterministic settings should not require extra confirmation.

For `labelstudio_benchmark`, use the shared classifier to distinguish three cases clearly:

- deterministic benchmark settings
- deterministic line-role benchmark settings
- Codex-backed benchmark settings

Only the Codex-backed case should require Codex approval.

For `bench speed-run` and `bench quality-run`, keep the current explicit confirmation-token behavior. The change is that the new policy layer should be the one that says whether the requested experiment family actually contains Codex-backed variants and therefore requires the token.

For `import` / `C3import` via `cookimport/entrypoint.py`, keep the ergonomic no-argument behavior for safe runs. But if loaded settings resolve to a Codex-backed posture without an explicit approval seam, fail before execution with a clear message that the user must rerun through an explicit Codex path.

This milestone is complete when every standard command boundary either:

- resolves to an allowed safe posture and proceeds,
- resolves to an allowed Codex-backed posture with explicit approval and proceeds,
- or fails before doing work.

### Milestone 5: Persist stable Codex-decision metadata into manifests and summaries

Every major run artifact should record the resolved Codex decision in a way that is stable and easy to audit.

Do not overload existing labels if they mean something narrower. Instead, add explicit machine-readable fields such as:

- `codex_recipe_enabled`
- `codex_line_role_enabled`
- `codex_knowledge_enabled`
- `codex_tags_enabled`
- `deterministic_line_role_enabled`
- `codex_any_enabled`
- `codex_decision_context`
- `codex_approval_required`
- `codex_approval_provided`
- `codex_decision_source`

If useful, also add the already existing operator-facing or benchmark-facing labels separately, for example:

- `top_tier_profile_family` when an interactive top-tier chooser was used
- `benchmark_variant` when the run is part of an official paired benchmark contract
- `ai_assistance_profile` when analytics consumes the artifact

The CLI should also print a short, loud summary at run start. The text should be factual rather than marketing-like. Examples:

    CODEX DECISION: none (safe deterministic run)

    CODEX DECISION: recipe_codex enabled (explicit approval)

    CODEX DECISION: recipe_codex + line_role_codex enabled (explicit approval)

This milestone is complete when a future audit no longer has to infer Codex usage indirectly from scattered run-config fields.

### Milestone 6: Reuse the shared classifier in analytics instead of duplicating logic

The repo already has consumer-facing classification logic in `cookimport/analytics/benchmark_semantics.py`. After the shared policy module exists, make analytics consume the shared Codex-decision facts instead of re-deriving them independently from raw strings.

The minimum requirement is that analytics and command-time validation do not disagree about whether a run used Codex-backed surfaces.

Be careful here. Consumer-facing labels do not necessarily need to be renamed in this plan. If the existing dashboard/compare-control labels remain acceptable, keep them stable. The important thing is that analytics should derive them from the same facts used by command-time policy.

If the existing analytics layer currently conflates deterministic line-role with Codex line-role in places where that would make the new policy misleading, document the exact behavior and either:

- fix it in this plan if the blast radius is manageable, or
- explicitly defer only the presentation change while still using correct Codex-backed booleans underneath.

This milestone is complete when analytics no longer relies on an independent, weaker interpretation of Codex activation.

## Validation and Acceptance

Acceptance is behavioral.

First, helper construction must stop disagreeing with model defaults. A bare `RunSettings()` and a helper-built default payload must both start from the same safe non-Codex posture.

Second, interactive and non-interactive command flows must both be covered. It is not enough for `cookimport` interactive mode to be safe if `import`, `stage`, or `labelstudio_import` can still bypass the policy layer.

Third, benchmark planning must remain trustworthy. A human inspecting planning artifacts should be able to tell which rows are paired-benchmark deterministic baselines, which are paired-benchmark Codex variants, and which are something else.

Fourth, the implementation must distinguish deterministic helper surfaces from Codex-backed surfaces. In particular, `line_role_pipeline=deterministic-v1` must not be treated as a Codex activation.

Fifth, all major manifests must state the resolved Codex decision explicitly enough that a future audit does not need to infer it from unrelated fields.

## Idempotence and Recovery

This work is safe to implement incrementally because the early milestones are additive. Start with a shared policy helper and tests, then route existing call sites through it one by one.

If a command starts failing after the new validator lands, first inspect whether the caller was depending on one of the currently ambiguous mixed payloads. In most cases the right fix is to move that caller onto an explicit contract helper or approval seam, not to weaken the validator.

If analytics tests fail after the shared classifier is introduced, do not immediately change user-facing labels. First verify whether the failure is because analytics was independently interpreting Codex activation differently from the new source of truth. Keep the fact model correct first, then decide whether display labels need to move.

Rollback is straightforward. The changes are policy logic, helper wiring, docs, and tests. No database or artifact migration is required.
