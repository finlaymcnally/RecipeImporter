# Tune Line-Role Policy Vetoes Against Benchmark Rows

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan is the follow-up to `docs/plans/2026-04-06_21.52.07 - harden line-role front-matter routing and semantic guardrails.md`. That earlier plan fixed the catastrophic April 6 front-matter collapse by hardening the prompt, adding semantic-pathology rejection, and containing unsafe outside-recipe shards. After that fix landed, the benchmark at `data/golden/benchmark-vs-golden/2026-04-06_22.46.09/` showed that the catastrophic failure was gone, but several row-level policy vetoes were still too strict and were suppressing good model calls.

## Purpose / Big Picture

After this change, the line-role runtime should still reject unsafe outside-recipe exclusions and unsupported recipe-local structure, but it should stop throwing away obviously useful recipe-adjacent instructions, storage lines, variant headings, and knowledge-heavy section headings simply because the local support rules are narrower than the real book structure. The user-visible outcome is that the Salt Fat Acid Heat benchmark should keep the front-matter safety win while recovering rows such as `Variation`, `Preheat the oven to 425°F...`, `In a small bowl or jar, let the shallot sit...`, and `WHAT IS ACID?` into better final labels.

The proof for this plan is deterministic and local. Focused tests should replay the exact rejected-row patterns from `data/output/2026-04-06_22.46.09/.../label_refine/.../labeled_lines.jsonl`, and the fast suite should stay green. Live Codex benchmarks remain out of scope unless the user later gives explicit approval.

## Surprises & Discoveries

- Observation: The front-matter hardening work succeeded on its intended failure mode.
  Evidence: In `data/output/2026-04-06_22.46.09/.../line_role/line_role_stage_summary.json`, `transport` is now `inline-json-v1`, `suspicious_shard_count=0`, and no semantic containment fired. The old front-matter meltdown did not recur.

- Observation: The remaining policy pain is concentrated in just two veto families.
  Evidence: The benchmark run had `42` `nonrecipe_exclude_without_deterministic_support` rows and `22` `instruction_without_local_support` rows, while the other three reasons only accounted for `11` rows total.

- Observation: Several outside-recipe procedure clusters are really recipe mini-sections that the current local-support rules fail to reconnect.
  Evidence: Around atomic indices `1100-1107`, the benchmark has `RECIPE_TITLE` rows `Beets` and `Citrus` followed by procedural lines like `Preheat the oven to 425°F...` and `Cut off the tops and bottoms of the fruits...`, but the current policy still vetoes those rows as unsupported outside-recipe instructions.

- Observation: Bare `Variation` headings are under-supported when they sit directly above a valid variant continuation.
  Evidence: Around atomic indices `1357-1358`, the runtime rejected `Variation` while accepting the next row `To make a sweet-tart Kumquat Vinaigrette...` as a variant run. The heading and body are semantically coherent, but the current generic-heading support logic missed that shape.

- Observation: The strongest exclusion rule is still net useful and should not be removed wholesale.
  Evidence: The `nonrecipe_exclude_without_deterministic_support` bucket caught many endorsement, memoir, and taxonomy rows that the model tried to exclude too aggressively. The problem is selective false negatives on reviewable knowledge/notes, not the existence of the rule.

- Observation: The failing recipe-adjacent instruction rows were mostly shape-detection problems, not semantic-guard problems.
  Evidence: `Preheat the oven ...`, `Taste a crouton ...`, and `In a small bowl ... let the shallot sit ...` were all rejected before promotion even though their local clusters were coherent. The missing support came from narrow instruction-start detection and the inability to use nearby supported titles as recipe-cluster evidence.

- Observation: Generic `Variation` support was missing one very specific continuation shape.
  Evidence: The runtime already understood a named `To make ...` alternate when it saw a neighboring variant heading, but `_variant_heading_label_allowed(...)` did not reuse that same knowledge when judging the heading itself.

## Decision Log

- Decision: Keep `nonrecipe_exclude_reason_mismatch` strict.
  Rationale: That rule only fires when deterministic code and the model both agree the row is obvious junk but disagree on the subtype. The benchmark examples were all clean front-matter rows such as `Foreword`, `Introduction`, and `Bibliography`, so relaxing the rule would add risk without a clear upside.
  Date/Author: 2026-04-06 / Codex

- Decision: Keep `howto_without_local_support` conceptually strict.
  Rationale: The benchmark examples for this rule did not show a real failure class. The problem rows involved unsupported cookbook headings or title-like lines that should not become `HOWTO_SECTION` merely because they are short and heading-shaped.
  Date/Author: 2026-04-06 / Codex

- Decision: Loosen `instruction_without_local_support` by improving support detection, not by deleting the rule.
  Rationale: The dangerous case is still outside-recipe prose that merely sounds imperative. The benchmark showed that the current support detector is too narrow for title-plus-procedure mini-clusters and late recipe tails. The fix should recognize those coherent recipe-adjacent contexts instead of turning off the gate entirely.
  Date/Author: 2026-04-06 / Codex

- Decision: Loosen generic `Variation` heading support when the next local rows clearly continue a variant run.
  Rationale: The benchmark false veto was not caused by over-eager variant promotion in the abstract. It was caused by missing continuity evidence for a heading that was immediately followed by valid variant body prose.
  Date/Author: 2026-04-06 / Codex

- Decision: Keep storage-note rows on the note path even when they sit inside a coherent outside-recipe recipe cluster.
  Rationale: Some benchmark gold rows treat leftover/storage lines as instructions, but the repository’s current contract intentionally uses `RECIPE_NOTES` for those advisory lines. The follow-up fix should recover real procedure, not reopen the note-versus-instruction ambiguity.
  Date/Author: 2026-04-06 / Codex

## Outcomes & Retrospective

The successful fix was not “be more permissive.” It was “recognize a few coherent recipe-adjacent patterns the runtime was already seeing and stop pretending they lack structure.”

## Context and Orientation

The active code lives under `cookimport/parsing/canonical_line_roles/`. In this package, `policy.py` is the owner for narrow deterministic support checks and fallback labels. The important distinction in this plan is between “semantic authority” and “promotion safety.” The model still makes the semantic guess. Repo code only decides whether that guess is safe to promote on the live route path, or whether it should be replaced with a conservative fallback.

The key functions today are:

`cookimport/parsing/canonical_line_roles/policy.py::_codex_prediction_policy_rejection_reason(...)`, which maps an accepted Codex label to a rejection reason code such as `instruction_without_local_support`.

`cookimport/parsing/canonical_line_roles/policy.py::_instruction_line_label_allowed(...)`, which decides whether an outside-recipe instruction-shaped row has enough local structure to stay `INSTRUCTION_LINE`.

`cookimport/parsing/canonical_line_roles/policy.py::_variant_heading_label_allowed(...)` and `_variant_label_allowed(...)`, which decide whether `Variation` and similar rows should survive as variant structure.

`cookimport/parsing/canonical_line_roles/policy.py::_outside_recipe_exclusion_reason(...)`, which is the deterministic owner for “obvious junk” outside-recipe exclusion reasons. This plan must not weaken that authority boundary.

## Plan of Work

Start in `cookimport/parsing/canonical_line_roles/policy.py` and leave the front-matter semantic guard in `validation.py` alone unless a test proves it is implicated. The target is the row-level support logic, not shard-level pathology rejection.

First, broaden outside-recipe instruction support in a narrow, explainable way. The current `_instruction_line_label_allowed(...)` path only accepts outside-recipe instructions when it sees direct instruction shape plus nearby component structure. Extend that support so a row can also stay `INSTRUCTION_LINE` when it is part of a short, local title-led recipe mini-cluster. In practice that means recognizing nearby `RECIPE_TITLE` context together with neighboring ingredient, yield, note, or procedure lines instead of insisting on explicit component-only tags.

Second, improve the generic `Variation` heading seam. Update `_variant_heading_label_allowed(...)` or an extracted helper so a bare heading such as `Variation` or `Variations` can survive when the immediate following row is already recognizable as valid variant body prose. Keep this narrow: the heading should not survive when the following lines are memoir, chapter taxonomy, or generic knowledge.

Third, make the fallback path less lossy when an outside-recipe instruction veto still fires. The current `_instruction_line_fallback_label(...)` path often dumps recipe-adjacent procedure into `OTHER`/`NONRECIPE_CANDIDATE`. Refine it so clear recipe mini-clusters can fall back to `RECIPE_NOTES` or stay instruction-like only when the surrounding recipe-title context is strong. The purpose is to recover realistic recipe-side rows without opening the door to generic cookbook advice being promoted as recipe method.

Fourth, keep the exclusion rules but improve the knowledge-heading escape hatch where it is obviously warranted. Rows such as `WHAT IS ACID?` and nearby teaching prose should continue to resist hard exclusion. If the current local knowledge support is too weak for those clusters, improve that support detection instead of weakening the exclusion rule globally.

After the code changes, add tests under `tests/parsing/` that replay the exact benchmark row patterns. Include both positive and negative contrasts: recovered avocado/beets/citrus/tail-step/variation rows should pass, while the front-matter pathology and unsupported generic advice should remain blocked.

Finally, update the short parsing/package docs and this plan so the new rule boundaries are discoverable to later agents.
