## Purpose / Big Picture

After this change, the inline structured-session line-role worker will stop answering as a bare anonymous label stream and instead answer against explicit packet-local row ids. The worker will also see a cleaner raw-evidence packet made of row objects rather than compressed pipe-delimited strings. The intended user-visible outcome is that line-role stays better attached to the actual source rows at fragile recipe starts without adding deterministic semantic gates, hints, or second-pass AI review.

  Evidence: `cookimport/parsing/canonical_line_roles/runtime_workers.py` validates only `labels`, while `cookimport/parsing/canonical_line_roles/validation.py` still contains `row_id` translation logic.

  Evidence: after the first runtime patch, the watchdog prompt still rendered `"r01 | 210 | ..."` strings until `planning.py` and the preview-facing helper were updated too.

## Context and Orientation

The canonical line-role runtime lives under `cookimport/parsing/canonical_line_roles/`. The active direct inline structured-session path is implemented in `runtime_workers.py`. That module builds the worker-visible packet, renders the prompt, writes a packet-local JSON schema, runs the structured session, validates the response, and optionally issues bounded repair followups. The generic line-role response translation and structural validation helpers live in `validation.py`. The prompt helper and prompt-pack compatibility surfaces live in `cookimport/llm/canonical_line_role_prompt.py` and the `llm_pipelines/` prompt/schema files.

For this plan, “inline structured-session” means the `codex exec` JSON worker path used when `line_role_codex_exec_style=inline-json-v1`. It is distinct from the taskfile worker path. The taskfile path already uses explicit `row_id` evidence in `task.json`; this plan is about making the inline path similarly row-grounded while keeping the inline transport thin and one-pass.

The current problem is not that deterministic repo code attached labels to the wrong rows. The raw model output itself drifted locally in repeated note/title/yield clusters. The current inline contract encourages this because the worker is asked to emit a bare label list aligned only by position. The implementation goal is therefore to make the AI’s task easier and more grounded, not to add a deterministic semantic backstop.

## Plan of Work

First, replace the inline packet’s owned rows and context rows with explicit row objects. Each owned row should expose the packet-local `row_id`, the visible `block_index`, and the raw `text`. Context rows should expose the raw neighboring text in a similarly clear object shape, but remain reference-only. The packet should stay free of semantic hints and deterministic label suggestions.

Next, update the inline prompt and output schema so the model returns a top-level `rows` array where every item includes `row_id` and `label`. The prompt should make three points very explicit: only owned rows should be labeled, every owned row must appear exactly once, and the worker should ground its answers in the row objects shown in the packet rather than relying on ordinal-only alignment.

Then, update inline response evaluation in `cookimport/parsing/canonical_line_roles/runtime_workers.py` so the structured-session path validates that row-grounded contract directly. Use packet-local `row_id` coverage to translate accepted rows back onto authoritative atomic indices, preserve structural validation and unresolved-row tracking, and reject malformed or extra-key outputs. Do not add any semantic comparison against the deterministic baseline.

After that, update the fake runner and any prompt-preview or watchdog helper surfaces that currently assume labels-only inline responses. Watchdog retry should use the same row-grounded output contract so the fallback surface does not reintroduce the old anonymous-alignment behavior.

Finally, update focused tests and the owning docs in `docs/04-parsing/04-parsing_readme.md` and `docs/10-llm/10-llm_README.md` so the durable docs describe the final inline contract accurately.

## Interfaces and Dependencies

The final inline packet should keep the top-level `rows` key for owned rows, but each row should be an object shaped like:

    {"row_id":"r01","block_index":210,"text":"Bright Cabbage Slaw"}

Reference-only neighboring context rows should use the same raw-evidence style, for example:

    {"block_index":209,"text":"Variations"}

The final inline output schema should require one top-level `rows` array where every item has:

    {"row_id":"r01","label":"RECIPE_TITLE"}

`cookimport/parsing/canonical_line_roles/runtime_workers.py` must translate those packet-local `row_id` values back to the shard’s authoritative atomic indices after structural validation. No new deterministic semantic veto or label-rewrite seam should be added.
