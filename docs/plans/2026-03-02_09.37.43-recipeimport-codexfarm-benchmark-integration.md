# Integrate Benchmark-Native CodexFarm Flow in RecipeImport

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

`docs/PLANS.md` is checked into this repository and this document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, recipeimport can run canonical-text benchmark comparisons using two explicit CodexFarm execution styles: the existing extraction-oriented path and a new benchmark-native direct line-label path, then enforce acceptance gates in one place.

The user-visible result is a single benchmark workflow in recipeimport that outputs (1) regular eval metrics, (2) pass-stage diagnosis artifacts, and (3) explicit pass/fail gates for merge decisions.

## Surprises & Discoveries

- Observation: recipeimport already has strong benchmark orchestration surface in `cookimport/cli.py`, including variant orchestration and output-folder contracts.
  Evidence: Existing tests in `tests/labelstudio/test_labelstudio_benchmark_helpers.py` assert single-offline paired variant behavior and comparison artifacts.

- Observation: recipeimport currently treats CodexFarm as an external executable and pipeline-pack root, not an in-repo Python package.
  Evidence: `cookimport/cli.py` validates a `codex-farm` command and supports `--codex-farm-root` style asset routing through local `llm_pipelines`.

- Observation: compare-gate mode detection depends on eval-run `run_manifest.json` (`run_config.prediction_run_config`), not just prediction-run artifacts.
  Evidence: compare fixture initially passed unexpectedly until eval-level run-manifest payload included `codex_farm_recipe_mode` + `llm_recipe_pipeline`.

- Observation: backward compatibility needs mode-specific handling when codex-farm does not support `--benchmark-mode`.
  Evidence: subprocess runner tests show extract mode can safely retry without the flag, while benchmark mode must fail fast to avoid silent behavior drift.

## Decision Log

- Decision: Keep recipeimport scope limited to orchestration, artifact contracts, and acceptance logic.
  Rationale: Runtime model behavior and prompt/schema internals belong in CodexFarm and are tracked in the companion CodexFarm ExecPlan.
  Date/Author: 2026-03-02 / Codex

- Decision: Preserve current production-default behavior and add benchmark-native behavior as an explicit mode, not a silent replacement.
  Rationale: Existing extraction path is in active use and should not regress during benchmark experimentation.
  Date/Author: 2026-03-02 / Codex

- Decision: Route codex-farm benchmark mode via environment (`COOKIMPORT_CODEX_FARM_RECIPE_MODE`) and let subprocess runner append `--benchmark-mode`.
  Rationale: Keeps fake/injected runner compatibility and centralizes subprocess compatibility handling in one place.
  Date/Author: 2026-03-02 / Codex

- Decision: For unsupported `--benchmark-mode`, retry without the flag only when requested mode is `extract`; hard-fail for requested mode `benchmark`.
  Rationale: Preserves old-codex-farm compatibility for default extraction while preventing false benchmark-mode claims.
  Date/Author: 2026-03-02 / Codex

## Context and Orientation

Recipeimport code lives in this repo under `cookimport/` and `llm_pipelines/`. CodexFarm code does not live here; it lives at `/home/mcnal/projects/shared/CodexFarm`.

In this plan, “recipeimport integration” means code that chooses benchmark mode, invokes CodexFarm with the right settings, collects generated outputs, computes compare/gate results, and exposes those results through CLI artifacts.

Key recipeimport files are:

- `cookimport/cli.py` for benchmark entrypoints, mode selection, and run orchestration.
- `cookimport/llm/codex_farm_runner.py` and `cookimport/llm/codex_farm_orchestrator.py` for CodexFarm invocation plumbing.
- `cookimport/labelstudio/eval_freeform.py` and related benchmark helpers for scoring and report generation.
- `tests/labelstudio/test_labelstudio_benchmark_helpers.py` for integration-level behavior checks.
- `docs/07-bench/07-bench_README.md` and `docs/10-llm/10-llm_README.md` for operator-facing contract documentation.

## Plan of Work

Milestone 1 adds explicit benchmark-mode routing in recipeimport. Extend benchmark settings parsing and benchmark call construction so recipeimport can request either CodexFarm extraction-style output or CodexFarm benchmark-native line labels. Route mode metadata into run manifests so every run records how it was produced.

Milestone 2 adds recipeimport-side artifact ingestion for diagnosis. Require presence checks for expected CodexFarm debug outputs in benchmark mode (for example, pass diagnostics, aligned block mappings, and raw request/response artifacts where available). Persist a recipeimport-level summary artifact that points to these files and fails gates when required artifacts are missing.

Milestone 3 adds acceptance gate enforcement in recipeimport compare flow. Encode named gates (`sea_no_regression`, `foodlab_no_regression`, `foodlab_ingredient_at_least_baseline`, `foodlab_variant_recall_nonzero`, and debug-artifact presence gates) and print a compact pass/fail table in CLI output and markdown summary artifacts.

Milestone 4 updates tests and docs. Add focused tests for new mode routing and gate behavior, then update docs so a new contributor can run baseline versus candidate and interpret gate failures without opening source code.

## Validation and Acceptance

Acceptance is complete when all of the following are true:

- recipeimport can run or compare codex variants that are explicitly tagged as `extract` or `benchmark` mode;
- compare output shows named gate pass/fail results instead of only raw metrics;
- missing required debug artifacts fail the run with a clear reason;
- existing non-codex and vanilla benchmark flows still pass existing tests;
- docs explain the new mode/gate behavior and output locations.

## Idempotence and Recovery

All outputs are timestamped benchmark artifacts under normal recipeimport output roots, so reruns are safe. If new gate logic blocks merges unexpectedly, keep artifacts and rerun compare with the same inputs after adjusting thresholds in one commit; do not delete prior run folders.

## Artifacts and Notes

The recipeimport output summary for benchmark comparisons should include:

- candidate and baseline run paths,
- per-gate pass/fail status,
- key metric deltas,
- path pointers to CodexFarm debug artifacts when available,
- clear failure reason when an expected artifact is missing.
