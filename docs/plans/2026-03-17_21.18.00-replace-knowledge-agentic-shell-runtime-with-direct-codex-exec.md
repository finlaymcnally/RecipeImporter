# Replace Knowledge Agentic Shell Runtime With Direct Codex Exec

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

After this change, the optional knowledge LLM stage still reviews every non-recipe chunk that RecipeImport assigns to it, but it does so the simple way: one direct structured model call per shard, with the full shard prompt sent once. Today the live runtime inflates cost because the knowledge worker is launched as an agent, then shells out to `cat`, `jq`, `sed`, and `python` over the shard file and replays those outputs inside the conversation. The clean target is to delete that transport for knowledge and replace it with one `codex exec --output-schema` call per shard.

From an operator point of view, `llm_knowledge_pipeline=codex-knowledge-shard-v1` should keep the same meaning. The stage should still emit `knowledge/<workbook_slug>/snippets.jsonl`, `knowledge.md`, and the same validated per-shard proposal artifacts under `raw/llm/<workbook_slug>/knowledge/`. The difference should be cost honesty: a `5`-prompt knowledge run should no longer produce agentic shell traces that reread the same shard file several times before returning JSON.

## Surprises & Discoveries

- Observation: knowledge has the same structural problem as line-role, but not as severely.
  Evidence: the finished benchmark at [prompt_budget_summary.json](/home/mcnal/projects/recipeimport/data/golden/benchmark-vs-golden/2026-03-17_17.45.00-fresh-summary-check/prompt_budget_summary.json) reports `1,070,613` knowledge input tokens and `861,056` cached-input tokens across only `5` calls.

- Observation: recipe and knowledge are not equally bad.
  Evidence: recipe workers on the same run each do one `cat` of the shard file and then answer, while knowledge workers perform many completed shell reads. This was summarized in [2026-03-17_21.00.00-recipe-is-mostly-single-read-knowledge-is-still-agentic-shell-exploration.md](/home/mcnal/projects/recipeimport/docs/understandings/2026-03-17_21.00.00-recipe-is-mostly-single-read-knowledge-is-still-agentic-shell-exploration.md).

- Observation: the active knowledge prompt wrapper is short; the expensive part is the agent’s shell behavior after launch.
  Evidence: the live prompt asset in [recipe.knowledge.compact.v1.prompt.md](/home/mcnal/projects/recipeimport/llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md) is only a path-based instruction wrapper, but [knowledge worker-004 trace](/home/mcnal/projects/recipeimport/data/output/2026-03-17_16.37.00/raw/llm/saltfatacidheatcutdown/knowledge/workers/worker-004/out/.codex-farm-traces/b81531126aac4846bc6c77bbfbb3e80e/97830b5be89a45e496077afca80c3675.trace.json) shows a full `cat` of the shard file plus multiple large `jq` subset reads.

- Observation: path handoff is not opaque local-file access under the current knowledge transport.
  Evidence: the trace above contains `command_execution` items whose `aggregated_output` fields contain the shard file body and several derived chunk subsets. The same payload is therefore present in the conversation more than once.

- Observation: the current runtime already has stable repo-owned seams for everything except the final model transport.
  Evidence: `cookimport/llm/codex_farm_knowledge_jobs.py` already plans bounded shards, `cookimport/llm/codex_farm_knowledge_ingest.py` already validates exact `chunk_id` and evidence coverage, and `cookimport/llm/codex_farm_knowledge_writer.py` already promotes validated proposals into final artifacts.

## Decision Log

- Decision: keep the public setting name `codex-knowledge-shard-v1`.
  Rationale: this is still the same user-facing feature. The implementation should become simpler and cheaper underneath the same setting rather than creating another public pipeline name.
  Date/Author: 2026-03-17 / Codex

- Decision: replace only the model transport, not the shard planner or proposal validator.
  Rationale: the knowledge planner, validator, and writer already embody the repo’s real correctness rules. The expensive mess is the path-based agentic execution in the middle, so that is the seam to swap out.
  Date/Author: 2026-03-17 / Codex

- Decision: the replacement target is one direct `codex exec --json --output-schema <schema> -` call per shard with the full shard prompt sent on stdin.
  Rationale: this preserves the intended mental model of “one shard equals one actual prompt,” and it removes the reason for Codex to shell out and reread shard files.
  Date/Author: 2026-03-17 / Codex

- Decision: keep worker folders and proposal artifacts for observability, but stop using worker input files as model evidence.
  Rationale: debugging artifacts are still useful. The problem is not that files exist; it is that the model is asked to discover and print them through shell tools.
  Date/Author: 2026-03-17 / Codex

- Decision: prefer a shared direct runner module under `cookimport/llm/` rather than a knowledge-only subprocess helper.
  Rationale: line-role has the same transport problem and already has a sibling ExecPlan for the same replacement shape. A shared runner avoids solving the same transport twice.
  Date/Author: 2026-03-17 / Codex

## Outcomes & Retrospective

This section is intentionally incomplete until implementation lands. The intended outcome is that the knowledge stage remains shard-based, validated, and observable, but its live usage becomes structurally honest: one shard prompt sent once, one structured response returned once, and no hidden shell transcript replay inside the token bill.

## Context and Orientation

The knowledge stage begins in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. `run_codex_farm_knowledge_harvest(...)` takes the Stage 7 non-recipe spans, extracts full-text blocks, and calls `build_knowledge_jobs(...)` from `cookimport/llm/codex_farm_knowledge_jobs.py`. A “shard” in this repository is one bounded bundle of knowledge chunks that the LLM should review together. Each shard owns a fixed set of `chunk_id`s and block indices.

Today the orchestrator sends those shard entries into `run_phase_workers_v1(...)` in `cookimport/llm/phase_worker_runtime.py`. That helper creates `workers/<id>/in/*.json`, `workers/<id>/out/*.json`, and `workers/<id>/logs/`, then asks the live runner to process the input folder. For live runs, that runner is `SubprocessCodexFarmRunner` in `cookimport/llm/codex_farm_runner.py`, which launches `codex-farm process --runtime-mode classic_task_farm_v1 --workers 1`.

The active knowledge prompt asset is [recipe.knowledge.compact.v1.prompt.md](/home/mcnal/projects/recipeimport/llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md). It does not inline the shard body. It tells Codex to read `{{INPUT_PATH}}` from the worker folder. In practice that is not invisible local context. On the real benchmark traces, Codex reads the file with shell commands such as `cat`, `jq`, and `sed`, then reasons over those outputs in the transcript. That is why cached-input tokens are so large.

The good news is that most of the knowledge pipeline is already repo-owned and should not be rewritten. The following parts are already the right long-term structure:

- `cookimport/llm/codex_farm_knowledge_jobs.py` plans shards and writes stable shard payloads.
- `cookimport/llm/codex_farm_knowledge_ingest.py` validates exact owned `chunk_id` coverage and evidence locality.
- `cookimport/llm/codex_farm_knowledge_writer.py` turns validated proposals into final `snippets.jsonl` and `knowledge.md`.
- `tests/llm/test_knowledge_job_bundles.py`, `tests/llm/test_knowledge_output_ingest.py`, `tests/llm/test_knowledge_writer.py`, and `tests/llm/test_codex_farm_knowledge_orchestrator.py` already cover the planner, validator, writer, and orchestrator contracts.

For this plan, “direct Codex exec” means a non-interactive Codex CLI call that takes one complete prompt on stdin and returns one structured JSON response. No path-mode prompt wrapper should be involved for live knowledge execution after this refactor.

## Plan of Work

The refactor should keep the stage contract and replace only the transport.

First, add or reuse a repo-owned direct structured runner under `cookimport/llm/`, preferably the same module that the line-role direct-exec work uses if that lands first. The runner should accept complete prompt text, an output schema path, a working directory, optional model and reasoning overrides, and environment overrides. It should invoke `codex exec --json --output-schema <schema> -`, send the prompt text once on stdin, capture the raw JSON event stream, parse the final assistant response, and extract usage totals from the emitted events. The result object must preserve enough metadata for later cost reporting and prompt debugging.

Second, change `cookimport/llm/codex_farm_knowledge_orchestrator.py` so live knowledge execution no longer calls `run_phase_workers_v1(...)`. The orchestrator should still use `build_knowledge_jobs(...)` to plan shard entries, and it should still preserve a worker-like artifact layout under `raw/llm/<workbook_slug>/knowledge/`. But instead of writing `workers/*/in/*.json` and asking CodexFarm to process that folder, it should loop over shard entries, build one full prompt body per shard, call the direct runner once, validate the returned JSON with `validate_knowledge_shard_output(...)`, and write the proposal plus runtime telemetry under the worker folder.

Third, define the actual prompt-body builder for knowledge. The new prompt body should inline the shard JSON once. The simplest and safest implementation is to keep the existing instruction text from [recipe.knowledge.compact.v1.prompt.md](/home/mcnal/projects/recipeimport/llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md) but render it into a real prompt string with the shard JSON embedded below the rules instead of telling Codex to read a file path. That preserves the current labeling and output contract while eliminating the reason for shell file reads. If keeping the markdown asset is awkward, move the knowledge instructions into a small repo-owned prompt-builder helper and leave the old asset only for historical reference until the migration is complete.

Fourth, remove knowledge-only path-mode compatibility code once the new transport passes tests. The knowledge stage should no longer depend on the `recipe.knowledge.compact.v1` path wrapper for live execution, and it should no longer need `run_phase_workers_v1(...)` for live knowledge transport. The pipeline id may still exist for preview, fake-runner, or artifact-compatibility reasons during the migration, but the plan should end with one obvious truth: live knowledge runs use direct exec, not `codex-farm process`.

Fifth, update observability and cost readers. `prompt_budget_summary.json`, prompt export, prompt preview, and any actual-cost readers must stop assuming that knowledge telemetry comes from CodexFarm `process` payloads. The new direct runner should write one stable usage artifact per shard and one stage summary artifact that the existing budget code can ingest without needing real CodexFarm pipeline traces.

Sixth, keep the operator-facing outputs the same. Final `knowledge` vs `other` promotion, `snippets.jsonl`, `knowledge.md`, manifest paths, and benchmark reporting should continue to work. The only meaningful behavioral change should be that the live token bill and traces now look like one direct structured call per shard instead of a little shell-driven agent session.

## Validation and Acceptance

Acceptance is behavioral.

On a real `saltfatacidheatcutdown` knowledge-only benchmark with `5` prompts:

- each knowledge shard corresponds to exactly one direct structured model call
- the knowledge worker raw event logs contain no `command_execution` items and no evidence of shell reads of the shard file
- all returned proposals still validate exact owned `chunk_id` coverage and in-surface evidence locality
- final `snippets.jsonl` and `knowledge.md` still materialize correctly
- actual-cost reporting comes from the new direct-runner telemetry rather than CodexFarm `process` payloads
- total knowledge input tokens drop materially below the old `1,070,613` baseline on the same book and no longer show a cached-input majority driven by transcript replay

The regression suite must also prove:

- the orchestrator still no-ops correctly when there are no seed non-recipe spans
- invalid worker outputs are still rejected by `validate_knowledge_shard_output(...)`
- prompt preview still reconstructs knowledge prompt count and stage budget correctly
- interactive CLI prompt-target configuration continues to work unchanged

## Idempotence and Recovery

This migration is safe to stage incrementally as long as the public setting name remains stable and one transport remains authoritative at any point in time. Do not ship a mixed state where some knowledge runs use `run_phase_workers_v1(...)` and others use direct exec without clear artifact labeling.

Worker artifact folders are safe to regenerate. The direct runner should overwrite `prompt.txt`, `events.jsonl`, `last_message.json`, `usage.json`, and `status.json` for each shard run. If a run fails partway through, rerunning the benchmark should rebuild those artifacts cleanly. The validator and writer are already idempotent with respect to rerunning from a clean `knowledge/` stage directory.

## Interfaces and Dependencies

At the end of this refactor, the repository should expose a stable repo-owned direct structured runner in `cookimport/llm/`. The final naming can be adjusted during implementation, but the following interface must exist in substance:

- one public entrypoint that accepts:
  - prompt text
  - output schema path
  - working directory
  - model override
  - reasoning effort override
  - timeout
  - environment overrides
- one structured result object that returns:
  - final response text
  - parsed usage totals
  - return code or failure detail
  - command metadata
  - optional raw event stream text

`cookimport/llm/codex_farm_knowledge_orchestrator.py` must continue to own knowledge-specific planning, validation, and promotion. After this refactor it must no longer require, for live execution:

- `run_phase_workers_v1(...)`

- the path-mode knowledge prompt wrapper in `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md`
