## Structured Restatement

The desired recipe-stage behavior is that when the recipe worker produces invalid output, the same live workspace worker should see the repo's validation errors immediately, keep the same recipe packet and draft context in memory, and fix the problem in-place instead of handing the failure to a brand-new repair worker. Validation must stay strict and repo-owned, but the normal recovery path should feel like a local file-editing loop: the worker writes a draft, repo code checks it, the repo writes down what is wrong in the current-task sidecars, and the same worker corrects that same draft until it validates or the repo hits a bounded fail-closed escalation rule. In this repository, that means changing the `recipe_llm_correct_and_link` runtime in `cookimport/llm/codex_farm_orchestrator.py` so `CURRENT_TASK_FEEDBACK.md` and the existing current-task helper loop become the default recovery mechanism, while one-shot structured repair stops being the first response to invalid task output and becomes only a bounded fallback if the same-session path cannot recover.

## Purpose / Big Picture

After this plan lands, a human should be able to inspect a recipe worker root during a live or fake-Codex run and see that invalid recipe output is handled by the same worker session, not by a fresh repair worker. The recipe worker should read `current_task.json` and `CURRENT_TASK_FEEDBACK.md`, edit the current draft, run the repo-owned checker or installer, see validator errors if the output is malformed, and correct the same draft immediately. The visible proof is that invalid tasks now remain in the same worker loop until they validate or hit a bounded local retry cap, and the old structured `inline_repair` path is no longer the normal first recovery step.

This matters for three reasons.

First, it aligns the runtime with the user's mental model. The user expects the recipe worker to behave like a local agent editing files with fast repo feedback, not like a one-shot JSON generator whose mistake is handed to a second agent. Second, it should improve quality and consistency because the same worker keeps the recipe packet, title hints, and earlier draft choices fresh in context. Third, it should reduce unnecessary token spend and duplicated reasoning because the repo can keep using cheap local validation and sidecar updates instead of paying for a second full repair prompt every time a small contract mistake appears.

The end state is not "let the worker do whatever it wants until something sticks." The end state is stricter operationally: repo-owned validation stays authoritative, the worker loop becomes more explicit about current-task recovery, and the runtime is clearer about when it gave the same session a chance to fix its own mistake, how many same-session fix attempts happened, and when the repo escalated to a last-resort fallback or failed closed.

## Context and Orientation

The recipe LLM stage lives in `cookimport/llm/codex_farm_orchestrator.py`. It runs after deterministic recipe-boundary grouping has identified recipe-owned spans. The runtime writes one worker root per recipe worker assignment and one task row per owned recipe packet. In current live code, a task usually means one recipe, even if a parent shard may own several tasks.

The key current files are:

- `cookimport/llm/codex_farm_orchestrator.py`, which plans recipe tasks, writes worker roots, supervises the long-lived workspace session, validates task outputs, aggregates task results back into shard proposals, and currently launches watchdog retry or structured repair follow-ups when needed.
- `cookimport/llm/recipe_workspace_tools.py`, which defines the compact recipe output contract, helper CLI commands such as `check-current`, `install-current`, and `finalize-all`, and the sidecar renderers for `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`, and `SHARD_PACKET.md`.
- `docs/10-llm/10-llm_README.md`, which currently documents the recipe worker story as "batch draft editing plus one `finalize-all` pass" while also noting that `check-current` writes repo-readable validation feedback.
- `tests/llm/test_recipe_phase_workers.py`, `tests/llm/test_recipe_workspace_tools.py`, and `tests/llm/test_codex_farm_orchestrator.py`, which already cover many of the current queue-controller, helper, and follow-up behaviors and must be updated with the new default contract.

Terms used in this plan:

- A `workspace worker session` is the long-lived local `codex exec` process for one recipe worker assignment. It can read files, edit drafts, and run repo-owned helper commands inside one worker root.
- A `task` is the smallest recipe answer unit in the current runtime. In practice it usually owns one recipe packet and one result file under `out/<task_id>.json`.
- A `same-session fix attempt` means the current workspace worker keeps the active task, reads updated repo-written validation feedback, edits the same draft again, and reruns the local helper without any new structured LLM follow-up call.
- A `structured repair attempt` means the repo launches a separate one-shot JSON-only follow-up through `_run_recipe_repair_attempt(...)` using copied input, copied previous output, and validator errors.
- `Sidecars` are the repo-written helper files in the worker root such as `current_task.json`, `CURRENT_TASK.md`, and `CURRENT_TASK_FEEDBACK.md`.

The current recipe stage behavior is a mixed contract. On one hand, the workspace worker path is already the main path. The queue controller supervises current-task validation live enough to rewrite sidecars when the worker writes something invalid, and helper commands already expose plain-English local feedback. On the other hand, the worker prompt still teaches the cheap happy path as "edit a batch of drafts, then `finalize-all` once," and the runtime still assumes that an invalid task can move into a new structured repair call after the workspace session's output is harvested.

That mixed contract is the thing this plan fixes. The target design is:

- the normal recipe worker loop is active-task-first and same-session,
- repo-side validation feedback is considered the default recovery mechanism,
- invalid outputs do not immediately hand control to a new structured repair worker,
- the same-session path has an explicit bounded attempt budget,
- reporting makes the same-session recovery behavior obvious,
- structured repair is demoted to explicit fallback or removed for common contract mistakes.

## Milestones

### Milestone 1: Make same-session current-task recovery the taught recipe contract

This milestone changes the worker-facing story without yet deleting every fallback. At the end of the milestone, the recipe worker prompt, current-task brief, and feedback guidance all teach one active-task recovery loop: inspect the active draft, run `check-current` or `install-current`, read repo-written feedback if validation fails, fix the same draft, and continue until the task validates. `finalize-all scratch/` may remain available for a clean batch path, but it is no longer the main mental model the worker is taught first.

Run the recipe workspace-tool tests after this milestone and inspect one worker prompt plus one fake worker root. Acceptance is that a novice reading the prompt and sidecars would clearly conclude that validation errors should be fixed by the same worker on the same active task, not by waiting for a later structured repair worker.

### Milestone 2: Keep invalid tasks inside the same workspace session before any escalation

This milestone changes the runtime loop in `cookimport/llm/codex_farm_orchestrator.py`. At the end of the milestone, when the current task output is invalid, the queue controller keeps the same task active, updates `CURRENT_TASK_FEEDBACK.md`, and gives the running worker session a bounded opportunity to fix the task locally. The runtime must not immediately treat an invalid main-worker output as the handoff point to `_run_recipe_repair_attempt(...)`.

Run focused recipe runtime/orchestrator tests after this milestone. Acceptance is that an invalid task output can be corrected in-place by the same session and then validate successfully without any `inline_repair` attempt artifacts being written for that task.

### Milestone 3: Redefine escalation and reporting so structured repair is explicit fallback

This milestone makes the runtime honest about recovery semantics. At the end of the milestone, task/shard status and proposal artifacts clearly distinguish among:

- no recovery needed,
- same-session fix attempted and succeeded,
- same-session fix attempted and failed,
- structured repair fallback attempted,
- fail-closed with no successful recovery.

The runtime also gains a per-task same-session recovery budget and a clear escalation rule for when structured repair is still allowed. Acceptance is that artifact readers and human reviewers can see exactly how many same-session fix attempts occurred and whether any task escalated to structured repair.

### Milestone 4: Cut docs and tests over to the new recipe recovery story

This milestone updates the LLM docs, recipe helper docs, and regression tests so the repository consistently teaches the new contract. At the end of the milestone, the recipe section no longer describes structured repair as the normal response to invalid task output. Acceptance is that the docs, worker prompt, helper tests, and orchestrator tests all describe the same story: same session first, bounded escalation only after that path fails.

## Plan of Work

Start in `cookimport/llm/codex_farm_orchestrator.py`, because this file currently owns the mixed behavior. Keep the workspace worker session as the main execution path. The new work is to pull current-task validation feedback into the center of the runtime contract rather than leaving it as a side-effect while the true recovery policy still lives in post-hoc structured repair.

The first edit area is the worker prompt builder. Rewrite the recipe workspace worker instructions so the default local loop is active-task-first:

1. open `worker_manifest.json`,
2. open `SHARD_PACKET.md` for shard overview,
3. open `CURRENT_TASK.md`, `current_task.json`, and `CURRENT_TASK_FEEDBACK.md` for the active task,
4. edit the current draft under `scratch/`,
5. run `python3 tools/recipe_worker.py check-current` or `install-current`,
6. if feedback says validation failed, fix the same draft immediately,
7. only move on when the repo advances `current_task.json` or says the queue is complete.

Do not remove `finalize-all scratch/` from the helper CLI if it is still useful for clean happy-path batches. But remove it from the top of the worker mental model. The prompt should stop telling the worker that batch-finalize is the cheap normal contract when the user explicitly wants live same-session correction to be the normal contract.

Then adjust the sidecar prose in `cookimport/llm/recipe_workspace_tools.py`. `render_recipe_worker_feedback_brief(...)` already has the right failure content, but its surrounding text still frames `check-current` and `install-current` as fallback/recovery. Rewrite the relevant static text so:

- same-session validator feedback is described as the standard loop,
- `CURRENT_TASK_FEEDBACK.md` explicitly says the repo will keep the same task active until validation passes or the queue ends,
- the worker is told not to continue to the next draft while the current task has failed validation,
- `finalize-all` is described as an optimization for already-clean drafts, not as the contract every worker should assume first.

After the prompt and sidecar text are aligned, change the runtime control loop. The queue controller already holds the current task steady on validation failure. Build on that rather than replacing it. Add explicit per-task same-session recovery accounting inside the runtime. The runtime should record when a task failed validation in the live session, how many same-session fix opportunities it has been given, and whether the queue controller later observed a valid output for that same task without leaving the session.

This accounting should be repo-owned and deterministic. It can live either as new task-local status fields written beside the task artifacts or as richer per-task metadata in the existing proposal/status surfaces. Whatever representation is chosen, it must survive aggregation so a reviewer can tell, after the run, whether the worker fixed the task live or the repo escalated.

Next, change the structured repair ladder in the recipe runtime. Today the runtime reaches `_should_attempt_recipe_repair(...)` after task validation rejects the main worker output. Replace that default path with a bounded same-session recovery gate. The exact implementation may vary, but the final behavior must match this story:

- if the task output is valid, advance normally,
- if the task output is invalid and the worker session is still active or has just finished with the task still current, keep the task active and expose repo feedback first,
- allow the same worker a bounded number of local correction cycles,
- only after that budget is exhausted, or after a worker-stop shape that makes same-session continuation impossible, consider structured repair,
- if structured repair is disabled for the relevant failure shape, fail closed and keep the invalid status explicit.

The easiest implementation path is likely to use the existing queue controller plus helper commands as the core recovery loop and to postpone structured repair evaluation until after the queue controller has had a chance to observe at least one post-failure local rewrite for the same task. Another viable implementation is to force active-task sequential processing and let the worker session continue until the current task validates or the watchdog/session exit prevents that. The plan does not require one exact mechanism, but it does require the end behavior to be same-session-first.

Add a hard same-session recovery budget per task. This budget should be small, deterministic, and recorded in status. For example, the runtime may permit two or three failed local install/check cycles for the same task before escalation. The exact constant can be tuned during implementation, but it must be documented and tested. The repo should not permit an infinite loop where one broken worker keeps rewriting the same invalid task forever.

After the runtime control loop changes, update the reporting and artifact story. Proposal and status payloads under the recipe phase runtime currently record `watchdog_retry_attempted`, `watchdog_retry_status`, `repair_attempted`, and `repair_status`. Extend or revise these surfaces so they also carry same-session recovery truth. The important reviewer questions are:

- did the task fail validation in the main worker session,
- did the same worker get repo-written feedback and fix it locally,
- how many same-session fix cycles occurred,
- did the task still escalate to structured repair,
- what was the final recovery path.

Keep the existing `repair_*` fields if structured repair remains as a fallback, but stop using them as the only recovery story. Add a new explicit vocabulary such as `same_session_fix_attempted`, `same_session_fix_count`, `same_session_fix_status`, and `same_session_fix_terminal_reason` if that is the cleanest route. The exact names can change, but the meaning must be easy to understand from raw status files and stage summaries.

Then rewrite the summary/reporting surfaces that compress recipe runtime behavior. The recipe stage summary and any telemetry reducers should report same-session fix counts separately from watchdog retries and structured repairs. The user explicitly asked whether repair/retry behavior is obvious in reporting; once recipe goes same-session-first, the answer must become "yes" rather than "mostly, but buried in artifacts."

Finally, update tests and docs together. In tests, add coverage for:

- a task that fails local validation once, gets corrected in the same session, and never writes `repair_status.json`,
- a task that fails local validation repeatedly until the same-session budget is exhausted and then escalates to structured repair,
- a task that fails in a watchdog-stop shape where same-session continuation is impossible and the runtime still takes the documented fallback path,
- worker prompt and feedback text that clearly teach active-task same-session fixing.

In docs, update `docs/10-llm/10-llm_README.md` and any short notes under `cookimport/llm/` that currently describe batch-finalize-first behavior or near-miss repair as the normal recipe correction story. The docs must say that the recipe worker's default recovery path is repo-written validator feedback in the same session, and that structured repair remains only as bounded fallback if it still exists after implementation.

## Validation and Acceptance

The implementation is correct only if all of the following are true.

First, worker behavior must change in an observable way. A novice reading the worker prompt and current-task sidecars must be taught to fix the active task in the same session when validation fails. The prompt must no longer imply that the worker should mainly batch-edit drafts and let a later structured repair worker clean up contract mistakes.

Second, same-session validation recovery must be real, not just documented. In a focused test, the recipe worker must be able to produce an invalid task output, receive repo-written validator errors in `CURRENT_TASK_FEEDBACK.md`, edit the same draft, and then validate successfully without any structured repair artifact being written for that task.

Third, the runtime must have a bounded same-session recovery rule. A task that keeps failing local validation may not spin forever. The runtime must record how many same-session fix cycles happened and must either escalate through the documented fallback or fail closed after the configured budget is exhausted.

Fourth, reporting must become clearer than today. Raw status and proposal artifacts for the recipe stage must make it obvious whether a task:

- never needed recovery,
- recovered in the same session,
- escalated to structured repair,
- or failed without successful recovery.

Fifth, existing strict contract validation must remain intact. The stage may not silently accept malformed JSON, missing recipe ids, bad compact-key usage, or invalid mapping fields just because the recovery story changed. The test suite must continue proving that malformed recipe outputs fail validation.

Sixth, the documentation must match the live runtime. `docs/10-llm/10-llm_README.md` and any short recipe helper notes must describe same-session validator feedback as the default recovery path and structured repair as fallback only, if still present.

## Idempotence and Recovery

This plan is safe to implement incrementally because it changes repo-local worker guidance, local validation flow, and runtime recovery policy rather than performing a data migration. The main risk is creating a half-converted contract where the prompt teaches same-session recovery but the runtime still immediately escalates to structured repair, or where reporting fields imply same-session recovery but the worker prompt still teaches batch-finalize-first behavior.

To keep the rollout safe, implement in this order:

1. update prompt and sidecar text,
2. add runtime accounting for same-session fix attempts,
3. change the escalation policy,
4. update reporting,
5. update docs/tests together.

At each stage, rerun the focused tests before moving on. If the runtime loop becomes ambiguous mid-change, prefer temporarily keeping structured repair as reachable fallback while the same-session accounting and prompt contract are stabilized. Do not delete the fallback first and only then discover the live worker loop has no bounded recovery rule.

If a partial implementation causes stuck tasks in tests, the safe recovery path is to keep the new prompt/sidecar wording but re-enable the previous structured repair gate until the bounded same-session attempt counter is correct. The repo must always prefer explicit fail-closed behavior over silent acceptance or infinite local retry loops.

## Interfaces and Dependencies

The implementation should preserve the existing public recipe stage entrypoint and compact task-output contract. This plan changes orchestration and worker guidance, not the recipe payload schema itself.

The following internal seams must exist clearly by the end of the work, whether as new helpers or as refactored existing code:

- In `cookimport/llm/codex_farm_orchestrator.py`, one explicit helper or logic block that decides whether an invalid task gets another same-session fix opportunity, increments the attempt count, and records the result.
- In `cookimport/llm/codex_farm_orchestrator.py`, one explicit helper or logic block that decides whether structured repair is still allowed after same-session recovery fails.
- In `cookimport/llm/recipe_workspace_tools.py`, current-task sidecar rendering that clearly encodes "active task still failing; fix this same draft now" versus "task accepted; queue advanced."
- In recipe runtime status/proposal artifacts, explicit fields for same-session recovery attempt count and terminal recovery path.

Do not introduce a second parallel recipe recovery system. The runtime should have one primary recovery ladder:

1. same-session local fix,
2. optional bounded fallback such as structured repair,
3. fail closed.
