# Shrink high-level group upload bundles to about 30 MB

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds. This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a multi-book interactive single-profile benchmark should still write its top-level `upload_bundle_v1`, but that group bundle should be intentionally slim: about 30 MB instead of the current 70 MB to 90 MB range. The user-visible outcome is that a finished run produces a first-pass reviewer bundle that is small enough to move around comfortably and does not silently blow past its own stated target.

The most concrete proof is rebuilding the existing March 6, 2026 group bundle at `data/golden/benchmark-vs-golden/2026-03-06_00.44.16/single-profile-benchmark/upload_bundle_v1` and observing that the resulting directory is about 30 MB instead of about 75 MB. This plan does not assume that size reduction alone fully solves Oracle transport limits; it does require that the bundle builder obey a real size contract instead of the current soft heuristic.

## Surprises & Discoveries

- Observation: the March 6, 2026 group bundle was about 77.8 MB even though high-level group mode advertises a target of about 40 MB.
  Evidence: `upload_bundle_payload.jsonl` at `data/golden/benchmark-vs-golden/2026-03-06_00.44.16/single-profile-benchmark/upload_bundle_v1` measured `77,799,575` bytes.

- Observation: raw full prompt logs dominated the payload and alone exceeded the entire intended budget.
  Evidence: the four largest rows were `prompts/full_prompt_log.jsonl` files at about `24.0 MB`, `23.9 MB`, `13.7 MB`, and `8.2 MB`.

- Observation: high-level selection currently treats full prompt logs as required, so the artifact-budget check cannot skip them.
  Evidence: `scripts/benchmark_cutdown_for_external_ai.py::_upload_bundle_select_high_level_artifact_paths(...)` appends `full_prompt_log_path` with `required=True`.

- Observation: the existing budget logic is not a final bundle-size guarantee.
  Evidence: the code budgets raw selected artifacts plus sampled wrong-line rows, but after payload and derived rows are serialized there is no final trim pass that forces `upload_bundle_payload.jsonl` under the target.

- Observation: there is already a separate follow-up mechanism for deep evidence, so the group bundle does not need to remain the one place that ships raw prompt logs and full-context trace dumps.
  Evidence: `docs/understandings/2026-03-06_01.05.09-upload-bundle-followup-packets.md` documents `cf-debug` follow-up packets that assume the recipient already has the base bundle and can request narrower derived evidence later.

- Observation: exact `final_bundle_bytes` cannot be known from the payload alone because the size metadata changes the index/overview bytes that contain it.
  Evidence: the implementation needed one post-write sync pass so the recorded `final_bundle_bytes` matched the actual written three-file total instead of being off by a few dozen bytes.

- Observation: prompt-budget summaries alone made the real bundle much smaller than necessary; the better way to climb back toward 30 MB was to add medium-sized context files instead of restoring raw prompt dumps.
  Evidence: adding per-run `prompt_request_response_log.txt`, `prediction-run/extracted_archive.json`, and `prediction-run/line-role-pipeline/extracted_archive.json` raised the rebuilt March 6 bundle from about `8.3 MB` to `26,960,520` bytes while keeping `serialized_size_capped=true`.

## Decision Log

- Decision: define the budget against the serialized output bundle, not against source artifact file sizes alone.
  Rationale: the user experiences the final `upload_bundle_v1` directory, not the pre-serialization estimate. A budget that allows a 77.8 MB output while claiming a 40 MB target is not a real contract.
  Date/Author: 2026-03-06 / Codex

- Decision: treat raw prompt logs, full-context wrong-line packets, and similar deep trace artifacts as follow-up material, not first-pass group-bundle material.
  Rationale: the group bundle is the first-pass reviewer handoff. The repo already has a narrower follow-up workflow for deep evidence, so the group bundle should optimize for compact navigation and triage first.
  Date/Author: 2026-03-06 / Codex

- Decision: reduce the high-level group target from about 40 MB to about 30 MB.
  Rationale: the current 40 MB target was already too permissive for the Oracle-upload use case and still failed to constrain the actual output. A 30 MB target leaves headroom while forcing stronger prioritization.
  Date/Author: 2026-03-06 / Codex

- Decision: keep the first implementation focused on top-level multi-book single-profile group bundles, not every upload bundle shape.
  Rationale: that is the failing path from the March 6 run and the only place where the repo currently claims an explicit size-budgeted high-level bundle mode.
  Date/Author: 2026-03-06 / Codex

- Decision: build high-level payload rows in memory and only write the final three files after trimming.
  Rationale: this makes it possible to remove optional rows deterministically, rebuild payload row numbers once, and then sync exact `final_bundle_bytes` metadata back into the written bundle.
  Date/Author: 2026-03-06 / Codex

## Outcomes & Retrospective

The implementation landed. High-level multi-book group bundles now target `30 * 1024 * 1024`, omit raw prompt logs/full-context trace dumps from the first-pass payload, retain compact prompt-budget summaries plus medium-sized prompt/archive context artifacts, and record the actual final bundle bytes plus any trimmed paths in `analysis.group_high_level`.

The real March 6 validation case now lands close to the intended target instead of far below it: `26,960,520` bytes total (`26M`). That keeps a materially richer first-pass packet while still excluding the raw prompt dumps that previously dominated the payload.

The remaining follow-on risk is transport behavior outside recipeimport itself. This plan fixed the bundle contract and the local size ceiling, but Oracle-side limits or upload ergonomics are still separate concerns.

## Context and Orientation

The relevant runtime starts in `cookimport/cli.py::_interactive_single_profile_all_matched_benchmark(...)`. When multiple books are selected, that function calls `cookimport/cli.py::_write_benchmark_upload_bundle(...)` with `high_level_only=True` and `target_bundle_size_bytes=BENCHMARK_GROUP_UPLOAD_BUNDLE_TARGET_BYTES`. Today that constant is `30 * 1024 * 1024`.

The real bundle builder lives in `scripts/benchmark_cutdown_for_external_ai.py`. The important entrypoint is `build_upload_bundle_for_existing_output(...)`, which delegates into `_write_upload_bundle_three_files(...)`. In high-level mode, `_write_upload_bundle_three_files(...)` calls `_upload_bundle_select_high_level_artifact_paths(...)` to choose source artifacts and `_upload_bundle_build_group_high_level_packet(...)` to sample some derived wrong-line rows. It then serializes everything into exactly three files:

- `upload_bundle_overview.md`
- `upload_bundle_index.json`
- `upload_bundle_payload.jsonl`

For this plan, a `group bundle` means the top-level `single-profile-benchmark/upload_bundle_v1` directory written after a multi-book single-profile run. A `serialized-size cap` means the number of bytes of the actual output files on disk, not an estimate of source artifact sizes. A `first-pass bundle` means the compact handoff meant for initial triage; deep evidence can be requested later through the follow-up packet flow documented in `docs/understandings/2026-03-06_01.05.09-upload-bundle-followup-packets.md`.

The March 6 failure case is a useful real repro and should remain the working example during implementation:

- Source root: `data/golden/benchmark-vs-golden/2026-03-06_00.44.16/single-profile-benchmark`
- Failing group bundle: `data/golden/benchmark-vs-golden/2026-03-06_00.44.16/single-profile-benchmark/upload_bundle_v1`

Measured byte sources from that run matter:

- Raw `prompts/full_prompt_log.jsonl` files contributed about 69.8 MB by themselves.
- Derived `preprocess_trace_failures.jsonl` and `wrong_label_lines.with_context.full.jsonl` files added several more megabytes.
- The index and overview files were tiny. The payload file was the problem.

## Plan of Work

### Milestone 1: Lock the failure into tests and byte accounting

Start by extending `tests/bench/test_benchmark_cutdown_for_external_ai.py`. Add a focused regression test that builds a high-level multi-run bundle with large prompt logs and proves the current implementation can exceed its target. Then add a second test that will become the new acceptance contract: for a synthetic multi-run fixture with large prompt logs, a high-level group bundle targeting 30 MB must produce a final output directory whose total bytes stay under the chosen hard ceiling and whose index records what was omitted.

This milestone should also add small helper assertions around serialized output bytes. Use the actual output files on disk, not only metadata fields. The point is to make it impossible for future changes to claim a budget while shipping a much larger payload.

This milestone is complete when one new test fails before the fix and can only pass after the builder obeys a real output-size contract.

### Milestone 2: Change high-level inclusion policy for first-pass bundles

Next, edit `scripts/benchmark_cutdown_for_external_ai.py::_upload_bundle_select_high_level_artifact_paths(...)` so high-level group bundles stop force-including raw full prompt logs. Those files are the main cause of budget failure and are not first-pass triage artifacts. Replace them with the smallest durable surfaces that preserve orientation:

- `run_manifest.json` must stay.
- `eval_report.json` can stay when it fits.
- compact prompt/runtime summaries such as `prompt_budget_summary.json`, manifest counters, or existing derived diagnostics should be preferred over raw `full_prompt_log.jsonl`.

Do the same triage for other heavy artifacts already shown to be expensive in the real run, especially full-context derived trace dumps like `wrong_label_lines.with_context.full.jsonl` and `preprocess_trace_failures.jsonl`. A first-pass group bundle should keep summary views, row locators, and compact sampled packets, not full trace dumps. If a heavy artifact remains available elsewhere under the source run tree or through `cf-debug`, record that in metadata instead of copying the full file.

The key rule for this milestone is simple: the top-level group bundle is a compact first-pass handoff. If an artifact is large, detailed, and only occasionally needed, it belongs in follow-up, not in the initial group payload.

This milestone is complete when the real byte leaders from the March 6 run are no longer present as raw payload rows in the rebuilt high-level bundle, and the overview/index still preserve enough navigation to identify the most important regressions.

### Milestone 3: Add a hard final serialized-size clamp

After inclusion policy is improved, add a real final budget enforcement step in `scripts/benchmark_cutdown_for_external_ai.py::_write_upload_bundle_three_files(...)`. The implementation should measure the actual bytes written to the output payload and trim optional content until the final bundle fits.

Be explicit about priority. Required items for a high-level group bundle should remain required:

- overview and index files,
- root summary rows,
- run manifests,
- compact per-book scorecards and triage summaries,
- the compact group high-level packet.

Optional items should be dropped in a deterministic order once the serialized output exceeds budget. Examples include:

- extra per-run eval reports after core summaries already exist,
- verbose derived trace packets,
- larger wrong-line context samples,
- anything else preserved only for convenience and not required for first-pass triage.

The builder must also record what it removed. Add metadata such as `omitted_artifacts`, `omitted_bytes_estimate`, and `final_bundle_bytes` so a reader can tell whether the bundle was trimmed and what deeper evidence would require a follow-up request.

This milestone is complete when a 30 MB target means the actual output is about 30 MB, not merely that one intermediate allocator thought it had planned for 30 MB.

### Milestone 4: Rebuild the real March 6 group bundle and validate reviewer usefulness

Use the existing run root at `data/golden/benchmark-vs-golden/2026-03-06_00.44.16/single-profile-benchmark` as the real validation case. Rebuild `upload_bundle_v1` in place and measure the resulting size. The acceptance target for this plan is a bundle directory of about 30 MB, with a hard ceiling chosen and enforced in tests.

Then inspect the rebuilt `upload_bundle_overview.md` and `upload_bundle_index.json`. Confirm that the compact bundle still tells a reviewer what happened:

- which books participated,
- the main score deltas,
- the main regression packets,
- enough row locators and compact summaries to decide what follow-up packet to request next.

Finally, run the direct Oracle CLI in dry-run mode against the rebuilt bundle. The expected result of this plan is that the bundle is much smaller and more transport-friendly. If Oracle still rejects the upload path because of its own per-file limit, document that clearly in the plan’s retrospective and keep that as a separate transport-layer follow-on rather than silently expanding the scope.

This milestone is complete when the rebuilt March 6 group bundle is around 30 MB and remains a usable first-pass reviewer packet.

## Idempotence and Recovery

Rebuilding an existing bundle is safe as long as it writes only to `upload_bundle_v1/`. The benchmark run itself does not need to be rerun. If a rebuild produces a bundle that is still too large or trims too aggressively, rerun the builder after adjusting only the bundle-selection logic and size constants.

Do not mutate the original run artifacts under the benchmark root. The purpose of this change is to produce a slimmer handoff package, not to delete source evidence.

If the real March 6 run root is unavailable in another checkout, fall back to the synthetic fixture tests. The implementation must still be validated by tests even if the local run folder is absent.
