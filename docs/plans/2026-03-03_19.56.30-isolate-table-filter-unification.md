# Unify isolate with table filters and add native cross-column OR

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md` from the repository root.

## Purpose / Big Picture

After this change, using `Isolate For X` updates the same table column-filter pipeline used by `Previous Runs`, so isolate and table results are no longer separate paths. The table filter engine also supports top-level `OR` across different columns, which is required for isolate `any rule (OR)` to be represented exactly.

## Surprises & Discoveries

- Observation: Existing table filters supported per-column clause `AND/OR`, but always used implicit `AND` across columns.
  Evidence: `recordMatchesPreviousRunsFilterGroups` previously called `.every(...)` for field groups.

- Observation: Existing isolate `any` logic already supported cross-field OR, so unifying isolate into table filters required extending table semantics rather than changing isolate semantics.
  Evidence: `isolateRecordMatches` already used `.some(...)` in `any` mode.

## Decision Log

- Decision: Add a first-class global table filter mode (`column_filter_global_mode`) instead of special-case isolate-only behavior.
  Rationale: keeps one filter engine and enables table-originated OR semantics as a core feature.
  Date/Author: 2026-03-03 / assistant

- Decision: Keep existing `filter_control_source` state for compatibility and messaging, while making isolate edits synchronize table filters directly.
  Rationale: preserves prior UI-state compatibility and avoids breaking existing saved dashboard state payloads.
  Date/Author: 2026-03-03 / assistant
