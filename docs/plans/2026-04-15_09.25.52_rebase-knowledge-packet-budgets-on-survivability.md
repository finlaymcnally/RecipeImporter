# Rebase Knowledge Packet Budgets On Survivability

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, knowledge shard planning will stop using the tiny fixed packet budgets that currently make the native shard count look disconnected from reality. When the operator leaves the packet budget settings unset, the planner will derive its default input and output char budgets from the existing `nonrecipe_finalize` survivability ceilings with explicit headroom, so the "native" shard count stays conservative but lands in the same rough operating zone as the survivability minimum rather than a separate tiny-budget universe. Explicit operator overrides will still work.

## Surprises & Discoveries

- Observation: The repository is currently inconsistent about the old knowledge output packet budget. Core run settings default to `6_000`, but some stage and Label Studio entrypoints still default to `12_000`.
  Evidence: `cookimport/config/run_settings.py` uses `default=6000`, while `cookimport/cli_commands/stage.py` and `cookimport/cli_commands/labelstudio.py` still pass `12000`.

- Observation: Prompt preview already threads explicit knowledge packet budget overrides separately from the knowledge build report, so changing the default source to "auto" should not break the override surface as long as `None` remains a legal value.
  Evidence: `tests/llm/test_prompt_preview.py::test_prompt_preview_threads_knowledge_packet_budgets_into_settings` only asserts that explicit `321` / `654` overrides survive into the manifest.

## Decision Log

- Decision: Keep the knowledge packet planner char-based instead of rewriting it to do token accounting directly.
  Rationale: The packet planner already works over ordered row text lengths and grouping-batch row estimates. Rewriting that whole seam to tokens would create more moving parts than needed for this retune. Deriving char budgets from the survivability token ceilings is the smaller, cleaner fix.
  Date/Author: 2026-04-15 / Codex

- Decision: Make the default packet budgets "auto from survivability" and preserve explicit integer overrides.
  Rationale: The user wants the native planner closer to the survivability ceiling, not a second tiny hardcoded universe. Keeping explicit overrides means the old behavior can still be forced intentionally for experiments or debugging.
  Date/Author: 2026-04-15 / Codex

- Decision: Derive default knowledge packet budgets with a simple fixed conversion and headroom policy: `3.0` chars per token, `40%` of the survivability input ceiling for packet input, and `30%` of the survivability output ceiling for packet output.
  Rationale: This keeps the packet planner in the same rough operating zone as survivability without pretending the two systems should be identical. The resulting defaults (`264_000` input chars, `72_000` output chars) are still conservative relative to the token ceilings once repair/follow-up headroom is considered, but they are no longer comically smaller than the real stage limits.
  Date/Author: 2026-04-15 / Codex

## Context and Orientation

The knowledge packet planner lives in `cookimport/llm/codex_farm_knowledge_jobs.py`. It walks the ordered non-recipe candidate rows, estimates prompt and output sizes in characters, and uses those estimates to partition rows into packet-sized shards before runtime. Today that file still hardcodes `_DEFAULT_KNOWLEDGE_PACKET_INPUT_CHAR_BUDGET = 18_000` and `_DEFAULT_KNOWLEDGE_PACKET_OUTPUT_CHAR_BUDGET = 6_000`.

The survivability solver lives in `cookimport/llm/shard_survivability.py`. For the `nonrecipe_finalize` stage, it uses much larger token ceilings: `220_000` input tokens, `80_000` output tokens, `300_000` peak session tokens, and `80` owned units. Those numbers are the repository's deterministic "minimum safe" preflight model.

Run settings for knowledge packet budgets live in `cookimport/config/run_settings.py` and `cookimport/config/run_settings_builders.py`. Stage and Label Studio CLI entrypoints also expose the packet budgets through hidden options, and some of those entrypoints still inject old fixed defaults directly. The implementation needs to make "no explicit budget configured" mean `None`, not `18_000` or `12_000`.

Focused tests for packet planning live in `tests/llm/test_knowledge_packet_planning.py` and `tests/llm/test_knowledge_job_bundles.py`. Prompt-preview override coverage lives in `tests/llm/test_prompt_preview.py`.

## Plan of Work

Add a small knowledge-planner helper in `cookimport/llm/codex_farm_knowledge_jobs.py` that reads the `nonrecipe_finalize` survivability budget from `cookimport/llm/shard_survivability.py` and converts those token ceilings into char budgets with explicit headroom constants. The helper should be used only when `input_char_budget` or `output_char_budget` is unset. If the operator passed explicit packet budgets, keep using those exact integers.

Update the run-setting defaults so the packet budgets are `None` by default instead of the old fixed values. That includes `cookimport/config/run_settings.py`, `cookimport/config/run_settings_builders.py`, and the hidden CLI/Label Studio entrypoints that currently inject `18_000` or `12_000` at the call boundary. The user-visible meaning should become "blank means auto from survivability."

Update the small owning docs so the knowledge planner documentation says the packet budgets are auto-derived from survivability with headroom, not fixed tiny defaults. Then update the packet-planning tests to assert the new auto defaults and to prove explicit packet-budget overrides still win.
