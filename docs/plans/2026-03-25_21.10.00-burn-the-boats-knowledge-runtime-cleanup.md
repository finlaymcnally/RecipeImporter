## Purpose / Big Picture

After this change, the live knowledge stage will have one real runtime instead of a half-finished shard-ledger runtime sitting on top of the older packet/task machinery. A fresh knowledge worker root will scaffold correctly, the worker helper loop will validate and advance phases without manual repair of repo-written files, prompt preview will report the same model-default story that execution uses, and the codebase will stop carrying the old knowledge batch/task path as if it were still a supported fallback.

The user-visible proof is simple. A fresh knowledge worker root created by the runtime should contain a valid Pass 1 work ledger, `python3 tools/knowledge_worker.py check-phase` should succeed before any human edits, and the focused LLM tests should prove the live runtime path rather than only the isolated helper functions.

## Context and Orientation

The live knowledge stage starts in `cookimport/llm/knowledge_stage/runtime.py`. It plans shard-owned inputs through `cookimport/llm/codex_farm_knowledge_jobs.py`, writes worker-helper files through `cookimport/llm/knowledge_phase_workspace_tools.py`, validates final installed shard outputs through `cookimport/llm/codex_farm_knowledge_ingest.py`, and renders reviewer-facing outputs through `cookimport/llm/codex_farm_knowledge_writer.py`.

The intended worker contract is phase-first. Pass 1 means one row per owned non-recipe block with `knowledge|other`. Pass 2 means one row per kept knowledge block with `group_id` and `topic_label`. The worker helper script `tools/knowledge_worker.py` owns the local loop: read `CURRENT_PHASE.md`, edit the named work file, run `check-phase`, and then run `install-phase` to either advance from Pass 1 to Pass 2 or install the final shard result.

The current bug is a mismatch between runtime and helper. The runtime seeds the work file with the final compact output shape (`packet_id`, `block_decisions`, `idea_groups`), but the helper and validators expect phase-ledger shapes (`phase`, `rows`). Because the helper only scaffolds missing files, a fresh runtime-created workspace starts invalid. The broader cleanup goal is to make the live runtime and helper speak only one contract.

## Plan of Work

First, fix the broken seam in `cookimport/llm/knowledge_stage/runtime.py` by writing the same Pass 1 ledger shape that `build_pass1_work_ledger(...)` and the generated worker helper expect. Then inspect the live phase runtime for the useful behaviors that were lost when the repo switched away from the older packet/taskized workspace runtime.

Next, simplify `cookimport/llm/knowledge_stage/runtime.py` aggressively. Delete the unused packet/taskized workspace functions and move any still-needed preflight, telemetry, interruption, or recovery logic into `_run_phase_knowledge_worker_assignment_v1` so the repository has one knowledge worker path. Keep names and comments honest: shard means shard, phase means phase, and task-status rows can remain only as a reporting ledger if they now map 1:1 to shards.

Then update tests. Add one runtime-facing test that proves a fresh worker root scaffolds a valid Pass 1 work file. Add one worker-helper smoke test that runs `check-phase` against a runtime-created root. Tighten prompt-preview coverage so command-sensitive default model resolution cannot silently diverge from runtime again.

Finally, update docs under `docs/10-llm/10-llm_README.md` and any smaller relevant notes so they describe the single remaining runtime and the stronger test coverage.
