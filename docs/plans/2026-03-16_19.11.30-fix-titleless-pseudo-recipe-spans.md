# Stop Titleless Structured Blocks From Becoming Recipe Spans

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, label-first runs should only send plausible recipe spans into recipe drafting and Codex recipe correction. A block or short run of blocks that looks like ingredients or instructions but does not have a believable recipe title should no longer become a recipe span just because it carries a structural label. The user-visible proof is that benchmark and stage runs on `saltfatacidheatCUTDOWN` stop producing huge `codex-farm recipe correction task X/Y` totals driven by chapter fragments like `SALT` / `What is Salt?`.

Today the repository writes `group_recipe_spans/<workbook_slug>/recipe_spans.json`, `span_decisions.json`, and `authoritative_block_labels.json` under each stage run. The bug is that `cookimport/parsing/recipe_span_grouping.py` treats any recipe-local label as enough to open a recipe span, then preserves the span even when it later discovers there was no title. That behavior turns labeling mistakes into full recipes. This plan fixes that by making missing-title spans non-recipes, preserving explicit diagnostics for every dropped case, and tightening the label safety seam so fewer outside-span structured blocks survive in the first place.

## Surprises & Discoveries

- Observation: the span grouper already knows when a span is invalid, but it only records warnings and still emits the span.
  Evidence: `cookimport/parsing/recipe_span_grouping.py` appends `recipe_span_started_without_title` and `recipe_span_missing_title_label` inside `_build_span(...)`, then still returns a `RecipeSpan`.

- Observation: the current benchmark blow-up is dominated by titleless spans, not by a retry storm or by live line-role token usage.
  Evidence: `data/golden/benchmark-vs-golden/2026-03-16_18.10.25/.../prompt_budget_summary.json` shows 175 recipe-correction calls and 56 knowledge calls, while the matching `group_recipe_spans/.../span_decisions.json` contains 175 spans, 133 with `recipe_span_started_without_title`.

- Observation: recipe-correction fan-out does not have its own hidden multiplier; it is one task per accepted grouped recipe span.
  Evidence: `docs/understandings/2026-03-16_18.17.06-recipe-correction-task-counts-on-saltfatacidheatcutdown.md` traces `cookimport/llm/codex_farm_orchestrator.py::_build_states(...)` and shows that `result.recipes` is enumerated directly into recipe-correction input JSON files.

- Observation: the “outside recipe spans, structured labels should be downgraded” safety rule is already documented, but the current artifacts show it is not sufficient on this source.
  Evidence: `docs/04-parsing/04-parsing_readme.md` says outside-span `INGREDIENT_LINE` and `INSTRUCTION_LINE` require local recipe evidence, yet `authoritative_block_labels.json` for blocks 24 and 25 still records `final_label=INGREDIENT_LINE` with `escalation_reasons=["outside_span_structured_label"]`.

- Observation: preserving diagnostics matters because dropped pseudo-recipes are useful debugging evidence even though they must stop becoming recipes.
  Evidence: the current `span_decisions.json` is the easiest place to see why `SALT` and similar fragments became recipes, and removing that evidence would make future regressions harder to diagnose.

## Context and Orientation

The current label-first staging path is assembled in `cookimport/parsing/label_source_of_truth.py`. That module atomizes archive blocks into lines, assigns authoritative labels, aggregates those line labels into block labels, then calls `cookimport/parsing/recipe_span_grouping.py::group_recipe_spans_from_labels(...)`. The returned `recipe_spans` are immediately converted into recipe candidates and downstream outputs by `build_conversion_result_from_label_spans(...)`. In this repository, one grouped recipe span becomes one `ConversionResult.recipe`, and one `ConversionResult.recipe` becomes one recipe-correction task when recipe Codex is enabled.

That direct mapping is important for scope control. `docs/understandings/2026-03-16_18.17.06-recipe-correction-task-counts-on-saltfatacidheatcutdown.md` confirms that `cookimport/llm/codex_farm_orchestrator.py::_build_states(...)` enumerates `result.recipes` directly. There is no second recipe-side multiplier hiding after grouping. If accepted grouped spans are reduced, recipe-correction task count falls immediately with them.

Three stage artifacts matter for this fix. `group_recipe_spans/<workbook_slug>/recipe_spans.json` is the authoritative list of spans that become recipes. `group_recipe_spans/<workbook_slug>/span_decisions.json` is the compact explanation artifact for reviewers and debugging. `group_recipe_spans/<workbook_slug>/authoritative_block_labels.json` shows the block-level labels and escalation reasons that fed the grouper.

The concrete failure example is the run at `data/output/2026-03-16_18.10.25/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-16_18.11.25/`. In that run, `authoritative_block_labels.json` marks blocks 24 and 25 as `INGREDIENT_LINE` even though they are outside a real recipe. `span_decisions.json` then shows `recipe_span_0` starting at block 24 with warnings `recipe_span_started_without_title` and `recipe_span_missing_title_label`. The grouper emitted 175 spans total, which is why the benchmark later queued 175 recipe-correction tasks.

The key code bug is simple. In `cookimport/parsing/recipe_span_grouping.py`, any label in `_RECIPE_LOCAL_LABELS` can start `pending = [block]` even when no title-like label has been seen yet. The flush step marks a warning if the first pending block is not title-like, but still writes the span. That means warnings are acting like acceptance. This plan changes that contract: warnings can remain, but invalid spans must be excluded from the recipe output list.

The parsing docs already state the intended safety rule. `docs/04-parsing/04-parsing_readme.md` says that outside recipe spans, `RECIPE_TITLE` and `RECIPE_VARIANT` require local structural evidence and `INGREDIENT_LINE` / `INSTRUCTION_LINE` should be downgraded when evidence is missing. The implementation now needs to make the downstream grouping layer enforce the same safety posture instead of trusting every surviving structural label.

## Plan of Work

Start by changing the grouping contract in `cookimport/parsing/recipe_span_grouping.py`. The grouper should continue to walk block labels in order, but it must stop turning “structured block with no title” into a `RecipeSpan`. There are two acceptable implementations as long as behavior is explicit and tested.

The preferred implementation is to separate accepted spans from rejected candidates. Keep collecting contiguous recipe-local runs as today, but when a run ends, inspect whether it contains a title-like label (`RECIPE_TITLE` or `RECIPE_VARIANT`) that qualifies as the span anchor. If it does, emit a normal `RecipeSpan`. If it does not, do not append it to the authoritative `recipe_spans` list. Instead, serialize a rejected-span decision record that includes the same block range, warning set, escalation reasons, and a machine-readable rejection reason such as `rejected_missing_title_anchor`.

To support that behavior cleanly, extend the grouping result shape rather than hiding information in side channels. The simplest option is to keep `recipe_spans` as the accepted list and add a sibling list for rejected runs in the artifact-writing seam inside `cookimport/staging/import_session.py` or the helper that serializes span decisions. If the current code assumes `group_recipe_spans_from_labels(...)` returns only spans and normalized block labels, introduce a small result object or a third return value for rejected decisions. The chosen shape must keep the grouping code easy to read and the staging artifact writer explicit.

Then update the `span_decisions.json` artifact so it distinguishes accepted recipe spans from rejected pseudo-recipe runs. The file should still be the compact reviewer/debug rollup, but each record must make clear whether the run became a recipe. For example, accepted rows can carry `decision="accepted_recipe_span"` and rejected rows can carry `decision="rejected_pseudo_recipe_span"`. The exact field names can differ, but the artifact must let a novice answer “why did this block sequence not become a recipe?” without reopening code.

After the grouping hard gate is in place, tighten the earlier label safety seam in `cookimport/parsing/canonical_line_roles.py` or the current helper path that produces authoritative block labels. The goal is not to depend on this change for correctness, because grouping will already reject invalid spans. The goal is to reduce noise and keep the block labels closer to the documented contract. Focus on rows that carry `outside_span_structured_label` today. When a block is outside a recipe span and lacks supporting nearby title or structural evidence, it should prefer `OTHER` over `INGREDIENT_LINE` or `INSTRUCTION_LINE`. Use the existing documented heuristics from `docs/04-parsing/04-parsing_readme.md` rather than inventing a new theory.

Next, update the stage serialization and any downstream helpers that count or summarize recipe spans. `cookimport/staging/import_session.py` and any helper that writes `recipe_spans.json`, `span_decisions.json`, or converts spans into recipes must only use accepted spans for recipe creation. Rejected rows must not leak into `ConversionResult.recipes`, intermediate drafts, final drafts, or Codex recipe-correction input creation.

Finally, update docs and tests together. The parsing doc should explicitly say that titleless structured runs are preserved as rejected diagnostics but are no longer emitted as recipes. The staging doc should say that `recipe_spans.json` is now accepted spans only, while `span_decisions.json` includes both accepted and rejected grouping outcomes. Add one short understanding note if implementation discovers a new subtlety beyond what is already captured in the existing benchmark token-spike notes.

## Milestones

### Milestone 1: Make grouping reject titleless structured runs

At the end of this milestone, the grouper still examines the same ordered block labels, but it no longer emits a `RecipeSpan` when the contiguous run lacks a title-like anchor. A stage run should still retain evidence that such a run existed, but the authoritative recipe list must shrink immediately.

The files most likely touched in this milestone are `cookimport/parsing/recipe_span_grouping.py` and `cookimport/parsing/label_source_of_truth.py`. The acceptance proof is a focused test that reproduces the `SALT`-style case with `INGREDIENT_LINE` blocks and no title, and shows that the grouping result contains zero accepted recipe spans for that run.

### Milestone 2: Add explicit diagnostics for rejected pseudo-recipes

At the end of this milestone, stage artifacts tell the truth about both accepted and rejected grouping results. `recipe_spans.json` should contain only accepted recipe spans. `span_decisions.json` should still include rejected titleless runs with machine-readable reasons and the original warning context.

The files most likely touched are `cookimport/staging/import_session.py` and any serializer/helper that shapes `span_decisions.json`. Acceptance is a stage test that writes artifacts and demonstrates that the rejected pseudo-recipe appears in decisions but not in the accepted span list.

### Milestone 3: Tighten outside-span structured-label downgrades

At the end of this milestone, fewer non-recipe blocks arrive at grouping already mislabeled as `INGREDIENT_LINE` or `INSTRUCTION_LINE`. The group-level hard gate from Milestone 1 remains the correctness boundary, but the block-label artifacts should also look more sane.

The files most likely touched are the label-sanitization helpers under `cookimport/parsing/canonical_line_roles.py` or the equivalent current path for outside-span structured-label downgrades. Acceptance is a focused parsing test that shows an outside-span fragment like `SALT` / `What is Salt?` no longer resolves to structural recipe labels without local recipe evidence.

### Milestone 4: Validate on the real benchmark seam and update docs

At the end of this milestone, the repository has both automated proof and one real artifact comparison showing the pseudo-recipe explosion is gone or materially reduced. The docs must describe the new contract so future refactors do not quietly reintroduce the permissive grouping behavior.

Acceptance is a targeted benchmark rerun or narrow stage rerun on `saltfatacidheatCUTDOWN` showing a far smaller recipe-span count and a correspondingly smaller recipe-correction task total, plus updated docs in `docs/04-parsing/04-parsing_readme.md` and `docs/05-staging/05-staging_readme.md`.

## Validation and Acceptance

Acceptance is behavioral, not just structural.

First, a focused unit test must prove that a contiguous run of `INGREDIENT_LINE` and `INSTRUCTION_LINE` blocks with no `RECIPE_TITLE` or `RECIPE_VARIANT` does not become an accepted recipe span. That test should fail before the fix and pass after it.

Second, a stage-level test must prove that rejected pseudo-recipes still appear in diagnostics while accepted recipe outputs exclude them. Concretely, `recipe_spans.json` should not include the rejected run, while `span_decisions.json` should include it with an explicit rejection reason.

Third, parsing tests must prove that outside-span structured-label downgrades still work for real recipes while becoming stricter for obvious non-recipe fragments. The change must not collapse legitimate recipes that have compact titles and normal local structure.

Fourth, a real run over `saltfatacidheatCUTDOWN` must show that accepted recipe spans drop sharply from the current 175 and that the recipe-correction task count falls with them. The exact final count does not need to equal the golden recipe-header count on the first pass, but the pathological titleless pseudo-recipes must disappear as accepted recipes.

Fifth, docs must be updated so a future contributor can understand the contract without reverse-engineering this bug from artifacts.

## Idempotence and Recovery

All code and doc edits in this plan are safe to repeat. The grouping and diagnostic changes are additive at the artifact-contract level and should not require destructive migration. If a first implementation proves too strict and drops legitimate recipes, revert only the new grouping gate or downgrade heuristic changes, keep the added tests, and use the failing cases to refine the acceptance rule before re-running the broader suite.

When validating with real runs, use fresh timestamped output roots and compare new runs against existing artifacts rather than deleting older ones. If a benchmark rerun is too expensive or noisy, fall back to a direct stage run or a focused parsing test loop until the grouping behavior is stable.
