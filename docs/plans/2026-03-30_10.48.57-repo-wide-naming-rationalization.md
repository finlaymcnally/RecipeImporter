# Repo-wide Naming and Terminology Rationalization

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

The repository currently makes one concept wear several names at once. A newcomer can read `cookimport/staging/pipeline_runtime.py`, see the semantic runtime steps `extract`, `recipe-boundary`, `recipe-refine`, `nonrecipe-route`, and `knowledge-final`, then open `cookimport/runs/stage_observability.py` and meet a different vocabulary: `label_det`, `group_recipe_spans`, `classify_nonrecipe`, `build_intermediate_det`, and `nonrecipe_knowledge_review`. The same non-recipe seam is also still called “Stage 7” in many live docs and payload fields. That translation tax slows every change and makes the codebase feel older than it is.

After this overhaul, one concept should have one live name. A user should be able to run `cookimport stage`, inspect the run folder, inspect `stage_observability.json`, inspect prompt exports or benchmark summaries, and see the same vocabulary all the way through. Live docs should explain the same names that the code writes. Historical readers may still exist for archived data, but they must be explicitly quarantined as historical rather than bleeding into current runtime names.

The user-visible proof is simple. After implementation, a deterministic stage run over `tests/fixtures/simple_text.txt` should produce a run root whose directory and JSON vocabulary use the new canonical names only. `stage_observability.json` should expose those same canonical stage keys. Benchmark and prompt-summary tools should present the same stage names instead of older aliases.

## Context and Orientation

This repository has several naming layers for the same runtime concepts.

A “stage key” is the stable snake_case identifier used in machine-readable data such as `stage_observability.json`, prompt summaries, benchmark projections, and stage-local manifests. A “stage label” is the human-facing title-cased rendering of that same key for docs and dashboards. An “artifact stem” is the directory or file-root name used on disk for the stage’s outputs. Today those three things are too often defined separately, which is why they drift.

The most important current files are:

- [cookimport/staging/pipeline_runtime.py](/home/mcnal/projects/recipeimport/cookimport/staging/pipeline_runtime.py): the cleanest expression of the current semantic runtime story. It already thinks in terms of extracted bundles, recipe boundary, recipe refinement, non-recipe routing, and final non-recipe authority.
- [cookimport/runs/stage_observability.py](/home/mcnal/projects/recipeimport/cookimport/runs/stage_observability.py): the run-level stage index. This is currently a competing naming authority and must become a consumer of the canonical names instead.
- [cookimport/staging/import_session_flows/authority.py](/home/mcnal/projects/recipeimport/cookimport/staging/import_session_flows/authority.py): the writer for the label-first authority artifacts. These directory names and schema versions are currently abbreviation-heavy.
- [cookimport/staging/nonrecipe_stage.py](/home/mcnal/projects/recipeimport/cookimport/staging/nonrecipe_stage.py), [cookimport/staging/nonrecipe_authority.py](/home/mcnal/projects/recipeimport/cookimport/staging/nonrecipe_authority.py), and [cookimport/staging/writer.py](/home/mcnal/projects/recipeimport/cookimport/staging/writer.py): the source of the non-recipe routing/finalization vocabulary and the place where `stage7_*` still leaks into live payloads.
- [cookimport/llm/prompt_budget.py](/home/mcnal/projects/recipeimport/cookimport/llm/prompt_budget.py), [cookimport/llm/prompt_artifacts.py](/home/mcnal/projects/recipeimport/cookimport/llm/prompt_artifacts.py), [cookimport/bench/upload_bundle_v1_existing_output.py](/home/mcnal/projects/recipeimport/cookimport/bench/upload_bundle_v1_existing_output.py), and [scripts/benchmark_cutdown_for_external_ai.py](/home/mcnal/projects/recipeimport/scripts/benchmark_cutdown_for_external_ai.py): prompt/export/benchmark consumers that currently hard-code stage names instead of inheriting them.
- [docs/AI_context.md](/home/mcnal/projects/recipeimport/docs/AI_context.md), [docs/01-architecture/01-architecture_README.md](/home/mcnal/projects/recipeimport/docs/01-architecture/01-architecture_README.md), [docs/05-staging/05-staging_readme.md](/home/mcnal/projects/recipeimport/docs/05-staging/05-staging_readme.md), [docs/10-llm/10-llm_README.md](/home/mcnal/projects/recipeimport/docs/10-llm/10-llm_README.md), and [docs/plain-english-explanation.md](/home/mcnal/projects/recipeimport/docs/plain-english-explanation.md): live operator-facing docs that must teach the same vocabulary as the code.
- `tests/staging`, `tests/llm`, `tests/bench`, `tests/labelstudio`, and `tests/architecture`: the main enforcement surfaces. New naming rules must be proven here rather than left as prose.

The repo also has intentionally historical readers. These are not bugs by themselves. Examples include old benchmark bundle readers, analytics backfill, and worker-output validators that must reject obsolete payload shapes. The refactor should not delete that capability unless it is genuinely unused. Instead, it should confine those historical seams so they stop teaching old terminology to the rest of the codebase.

## Target Naming Charter

The implementation must adopt the following live naming rules.

Code keys and JSON keys use snake_case. Human labels use title case derived from the same root phrase. Run-root directories and artifact stems use the same snake_case root phrase unless the artifact is a clearly different domain object. A future contributor should be able to infer the code key from the label and the on-disk name without a translation chart.

The canonical semantic stage roots are:

- `label_deterministic`
- `label_refine`
- `recipe_boundary`
- `nonrecipe_route`
- `recipe_build_intermediate`
- `recipe_refine`
- `recipe_build_final`
- `nonrecipe_finalize`
- `line_role`
- `write_outputs`

The implementation must treat these as the only live stage keys. Old live keys such as `label_det`, `label_llm_correct`, `group_recipe_spans`, `classify_nonrecipe`, `build_intermediate_det`, and `nonrecipe_knowledge_review` must disappear from live code paths, live docs, and generated outputs.

The canonical label-first artifact directories must be:

- `label_deterministic/`
- `label_refine/`
- `recipe_boundary/`

The canonical non-recipe run-root files must be:

- `08_nonrecipe_route.json`
- `08_nonrecipe_exclusions.jsonl`
- `09_nonrecipe_authority.json`
- `09_nonrecipe_knowledge_groups.json`
- `09_nonrecipe_finalize_status.json`

The canonical payload fields must avoid `stage7_*`. Use `nonrecipe_route` when the value is still a route choice such as `candidate` or `exclude`. Use `nonrecipe_final_category` when the value is a final semantic category such as `knowledge` or `other`. Use `nonrecipe_candidate_spans` instead of `stage7_candidate_nonrecipe_spans`.

The canonical public prose must stop using “Stage 7” as if it were a real current runtime name. Historical docs may still mention it when clearly framed as history, but live docs must prefer “Non-Recipe Route” and “Non-Recipe Finalization” or the exact code keys when discussing machine-readable contracts.

The word `legacy` is allowed only in three situations:

- a module or helper whose sole job is to read or reject old inputs,
- a test whose purpose is to prove that an old input shape is rejected or still ingested by a historical reader,
- an archival document or report that is explicitly historical.

It is not allowed in live output names, stage keys, user-facing docs describing the current product, or normal runtime helper names.

## Milestones

### Milestone 1: Create the naming source of truth and the guard rails

The first milestone creates the single source of truth for live names. At the end of this milestone, the repository should have one small module that defines the canonical stage keys, labels, artifact stems, and related file names, and the existing consumers should import from it rather than carrying independent string maps. This milestone also adds tests that define the ban list for old live names and the whitelist for intentional historical readers.

Acceptance for this milestone is not a user-facing stage run yet. It is a set of tests that prove the codebase has a single naming authority and that the implementation has an executable definition of “old live name” versus “allowed historical reader”.

### Milestone 2: Rename the stage and artifact vocabulary in the runtime core

The second milestone updates the runtime writers, stage observability, prompt tooling, and benchmark/export tooling to use the new stage names and artifact stems. At the end of this milestone, `stage_observability.json`, prompt budgets, prompt activity traces, and benchmark cutdown summaries should all present the same stage-key vocabulary.

Acceptance for this milestone is a deterministic stage run over `tests/fixtures/simple_text.txt` whose run root, prompt summaries, and `stage_observability.json` all use the canonical names without any translation layer.

### Milestone 3: Rename the non-recipe routing and finalization contract

The third milestone removes `Stage 7` from live payloads, function names, defaults, and docs. At the end of this milestone, there should be no `stage7_category`, `stage7_candidate_nonrecipe_spans`, or live “Stage 7” wording outside the historical whitelist. The non-recipe route/finalization story should read the same way in code, docs, and generated outputs.

Acceptance for this milestone is a passing set of staging, llm, bench, and labelstudio tests plus a repo search over live paths that finds no `stage7_` or “Stage 7” outside the explicit historical allowlist.

### Milestone 4: Rename the label-first and recipe-stage surfaces

The fourth milestone removes `label_det`, `label_llm_correct`, `group_recipe_spans`, and `build_intermediate_det` from live machine-readable and operator-facing surfaces. At the end of this milestone, a user inspecting a run root should see domain-language names instead of implementation shorthand.

Acceptance for this milestone is a deterministic stage run whose directories and schemas use `label_deterministic`, `label_refine`, `recipe_boundary`, `recipe_build_intermediate`, `recipe_refine`, and `recipe_build_final`, and whose tests and docs agree with that output.

### Milestone 5: Quarantine historical readers and finish the live docs sweep

The fifth milestone is the cleanup and hardening pass. Historical readers that still need old names must be clearly named as historical and kept out of live docs. Live docs, tests, scripts, dashboards, and benchmark summaries must stop surfacing old terms except where they are explicitly describing old artifacts.

Acceptance for this milestone is that a newcomer can read the top-level architecture and AI-context docs, run a stage command, inspect benchmark/prompt summaries, and see one coherent vocabulary without needing oral history.

The canonical naming source of truth exists and the main producers and consumers import from it. `stage_observability.json`, prompt summaries, prompt artifacts, benchmark cutdown outputs, and staging writers all expose the same canonical stage vocabulary. ## Idempotence and Recovery

This refactor is safe to do in repeatable rename waves as long as each producer/consumer cluster moves together. Use a temporary output root for smoke runs so repeated test runs do not pollute `data/output`. If a rename wave leaves the repo in a mixed state, do not add aliases to “get green quickly.” Instead, use the audit grep to finish the paired consumers and rerun the affected domain tests.

When a machine-readable schema or file name changes, bump the schema version or update the filename in one deliberate change and regenerate the fixture readers/tests in the same wave. Because this is a solo project, prefer destructive clarity over indefinitely supporting both names. Historical readers may keep old-name support only if they are explicitly historical and tested as such.

## Interfaces and Dependencies

In [cookimport/runs/stage_names.py](/home/mcnal/projects/recipeimport/cookimport/runs/stage_names.py), define a small single source of truth for stage metadata. It should expose a structure equivalent to:

    STAGE_DEFINITIONS: dict[str, StageDefinition]
    def stage_label(stage_key: str) -> str
    def stage_artifact_stem(stage_key: str) -> str
    def stage_family(stage_key: str) -> str
