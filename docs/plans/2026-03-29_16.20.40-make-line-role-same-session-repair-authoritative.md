# Make Line-Role Same-Session Repair Authoritative

Exact user request:

    "iwant all failed rows to be retry/repaired by the LLM. i thought I had code for that"
    "you do have repair code, but it is worker-driven same-session repair" THIS IS WHAT I WANT. SO I WANT THAT WORKER TO FIX BAD LINES WHEN IT CHECKS TO MAKE SURE ITS OUTPUT IS VALID. 
    i do not want deterministic fallback                                               
                                                                                                                                         

Rewritten intent:

The goal of this change is to make the existing same-session line-role worker repair loop fully authoritative. When the worker runs `check-phase` and discovers bad or unresolved rows, it must keep repairing those rows in the active work ledger until the shard installs cleanly, or else the shard must fail closed with explicit repair evidence. The system must not silently replace bad LLM rows with deterministic baseline labels at any point in recipe-boundary.

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the canonical line-role worker inside `recipe-boundary` will be required to fix bad rows through its existing same-session `check-phase` and `install-phase` loop. A shard will no longer silently fall back to the deterministic baseline for unresolved rows. The observable user-facing result is simple: when line-role Codex is enabled, every shard either installs a clean LLM-validated ledger or the run fails closed with explicit repair evidence. There is no hidden deterministic borrow-back for rejected or missing line-role rows.

The intended demonstration after implementation is:

1. Run a narrow line-role test where the worker emits an invalid ledger first and then repairs it through the active workspace loop. The test should prove that the final installed output comes from the repaired LLM ledger, not from deterministic fallback.
2. Run a narrow line-role test where the worker never reaches a clean install. The test should prove that the shard fails closed and that no final prediction is synthesized from the deterministic baseline.
3. Confirm that the runtime artifacts now show repair-first, fail-closed semantics instead of `completed_with_fallback` / `row_fallback`.

## Surprises & Discoveries

- Observation: the helper and worker prompt already describe the exact behavior the user wants.
  Evidence: `cookimport/parsing/line_role_workspace_tools.py` says “There is no separate repo-owned repair model pass for line-role; the work ledger plus `check-phase` is the real repair loop.” `cookimport/parsing/canonical_line_roles/planning.py` tells the worker to re-open `CURRENT_PHASE_FEEDBACK.md`, fix only unresolved rows, and install only once clean.

- Observation: the live runtime still resolves unresolved rows to deterministic labels after validation.
  Evidence: `cookimport/parsing/canonical_line_roles/validation.py` builds `fallback_atomic_indices` from `deterministic_baseline_by_atomic_index`, and `cookimport/parsing/canonical_line_roles/runtime.py` turns those rows back into baseline predictions tagged `line_role_row_fallback`.

- Observation: the runtime still contains `repair_status` and `repair_recovered` plumbing, but there is no live path that marks `repair_attempted=True` or `repair_status="repaired"`.
  Evidence: the current runtime initializes `repair_attempted = False` and `repair_status = "not_attempted"` and never flips them. Existing tests assert `repair_status == "not_attempted"` on invalid-output paths.

- Observation: there are two separate deterministic fallback seams today.
  Evidence: first, shard validation rewrites unresolved rows onto the deterministic baseline. Second, the top-level line-role collector fills any missing prediction with deterministic baseline before final sanitation.

## Outcomes & Retrospective

No code changes have been made yet. Research confirms that the current repository intent and the current runtime behavior diverge. The helper contract already says same-session repair is the real loop, but the runtime still finalizes unresolved rows with deterministic fallback. The implementation work in this plan is therefore a corrective change to make runtime match the documented worker contract.

## Context and Orientation

This change lives in the canonical line-role subsystem, which is the line-labeling owner inside `recipe-boundary`.

Key files:

- `cookimport/parsing/canonical_line_roles/runtime.py` is the live runtime owner. It creates worker workspaces, evaluates workspace outputs, writes shard status artifacts, and currently synthesizes deterministic fallback rows after invalid output.
- `cookimport/parsing/canonical_line_roles/validation.py` validates line-role shard payloads and currently builds `row_resolution` metadata with `accepted_rows`, `unresolved_atomic_indices`, and `fallback_atomic_indices`.
- `cookimport/parsing/line_role_workspace_tools.py` is the worker-side helper. It validates the current work ledger, writes `repair/<shard_id>.json` for unresolved rows, rewrites `CURRENT_PHASE_FEEDBACK.md`, and installs `out/<shard_id>.json` only when the ledger is clean.
- `cookimport/parsing/canonical_line_roles/planning.py` generates the worker prompt that explains the same-session loop.
- `cookimport/parsing/canonical_line_roles/README.md` documents the subsystem contract and currently still mentions fallback for unresolved rows.
- `cookimport/parsing/label_source_of_truth.py` consumes the final line-role predictions and feeds them into recipe span grouping.
- `tests/parsing/test_canonical_line_roles.py` contains the authoritative regression coverage for workspace worker behavior, invalid output handling, prompt contracts, and proposal metadata.
- `docs/plain-english-explanation.md` and `docs/01-architecture/01-architecture_README.md` currently still describe rejected rows falling back to deterministic baseline and must be updated if behavior changes.

Important terms in this repository:

- “same-session repair” means the already-running workspace worker edits the active `work/<shard_id>.json`, runs `python3 tools/line_role_worker.py check-phase`, reads `CURRENT_PHASE_FEEDBACK.md`, fixes only the unresolved rows described in `repair/<shard_id>.json`, and then runs `install-phase` once the ledger is valid.
- “installed output” means the file `out/<shard_id>.json` written by `install-phase`. This is the only file that should count as successful workspace completion for a shard.
- “deterministic baseline” means the rule-based line-role label computed before Codex runs. It is useful as evidence and context, but after this change it must not silently become final authority for bad LLM rows.
- “fail closed” means the run records explicit repair failure artifacts and surfaces an error instead of silently manufacturing a final answer from deterministic data.

Current bug summary:

1. The worker contract says same-session repair is the real fix loop.
2. The runtime still inspects the final worker state after the session ends and resolves any unresolved rows back to deterministic baseline.
3. The top-level line-role collector also fills any missing prediction with the baseline.
4. The result is that the worker can stop before installing a clean shard and the run still appears to “finish” with deterministic labels for the broken rows.

## Plan of Work

The implementation should proceed in four coordinated edits.

First, change `cookimport/parsing/canonical_line_roles/runtime.py` so that installed shard outputs become the only successful workspace completion signal. The runtime already knows the worker root, `work/`, `repair/`, and `out/` paths for each shard. The implementation should treat `out/<shard_id>.json` as the authoritative happy-path result. If the session stops with only an invalid `work/<shard_id>.json`, the runtime must preserve the repair artifacts and surface the shard as failed or interrupted, not auto-resolve the missing rows to deterministic baseline.

Second, replace row-level fallback finalization in `cookimport/parsing/canonical_line_roles/validation.py` and `cookimport/parsing/canonical_line_roles/runtime.py` with repair-first, fail-closed finalization. The validation layer should still compute unresolved-row metadata because the helper and feedback file need it. However, unresolved rows should no longer be turned into final baseline rows. Instead, the runtime should produce one of two outcomes: a fully validated installed ledger, or an explicit failure object that explains which rows remained unresolved and where the repair artifacts live. The old `completed_with_fallback`, `row_fallback`, `fallback_only`, `partial_fallback`, and `line_role_row_fallback` semantics should be removed from the live path.

Third, make the same-session repair path observable and testable. The runtime should write status fields that clearly distinguish: clean main-worker success, clean success after same-session repair, and fail-closed termination after the worker exits or exhausts the active loop without installing a valid shard. The plan does not require inventing a new detached repair model pass. It does require that the repo accurately recognize when the worker used the existing same-session repair loop and when it did not finish that loop.

Fourth, update the public docs and regression tests so they match the new authority boundary. The docs should say that line-role Codex output must install cleanly through the workspace helper or the shard fails. The tests should cover the repaired happy path, the fail-closed unhappy path, proposal/status artifact wording, and the absence of deterministic fallback tags or counters.

## Idempotence and Recovery

This work is safe to repeat because it changes runtime decision logic and tests only. There is no schema migration and no destructive data rewrite.

If implementation temporarily breaks many fallback-based tests, fix the runtime first and then update the tests in the same branch. Do not “fix” the failing tests by reintroducing hidden deterministic fallback.

If the new fail-closed behavior exposes unrelated flaky worker-stop cases, keep those failures explicit in status artifacts rather than patching around them with baseline synthesis. The right retry path remains the same-session worker loop, not deterministic substitution.

## Interfaces and Dependencies

The finished implementation must preserve these interfaces:

- `cookimport.parsing.canonical_line_roles.label_atomic_lines(...)` remains the public entrypoint for line-role labeling.
- Worker helper commands remain:

      python3 tools/line_role_worker.py check-phase
      python3 tools/line_role_worker.py install-phase

- `repair/<shard_id>.json` remains the worker-readable unresolved-row patch request.
- `CURRENT_PHASE.md` and `CURRENT_PHASE_FEEDBACK.md` remain the active control surface for same-session worker repair.

The runtime/status contract must change in these ways:

- There must be no live success state whose meaning is “validated with deterministic fallback rows.”
- A shard that ends with unresolved rows must report an explicit failure or interruption state and preserve repair metadata.
- If same-session repair succeeds, the shard status must say so explicitly.

The tests must prove that the final authoritative line-role output comes from a clean installed LLM ledger, never from deterministic substitution for broken rows.
