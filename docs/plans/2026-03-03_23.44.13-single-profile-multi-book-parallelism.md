# Add bounded parallel scheduling for single-profile multi-book benchmark runs

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md` from the repository root.

## Purpose / Big Picture

After this change, interactive single-profile benchmark runs with more than one selected book no longer execute strictly one-at-a-time. They run up to three books concurrently, lower per-book worker counts to 80% of the selected run settings, and enforce a single shared split-conversion slot so split-heavy phases do not run concurrently.

The behavior is visible in the interactive benchmark output where scheduler settings are printed before the run batch starts.

## Surprises & Discoveries

- Observation: split-phase gating support already existed in benchmark internals and only needed orchestration wiring for this path.
  Evidence: `cookimport/cli.py` context manager `_benchmark_split_phase_overrides(...)` and `cookimport/labelstudio/ingest.py` split-phase slot acquisition logic.

## Decision Log

- Decision: implement concurrency at the single-profile orchestration layer instead of deeper ingest/parser layers.
  Rationale: the feature request is scoped to multi-book benchmark scheduling; this path can be changed without altering core conversion logic.
  Date/Author: 2026-03-03 / assistant

- Decision: cap concurrent books at three and set split-phase slots to one for multi-book runs.
  Rationale: aligns with requested throughput increase while protecting against simultaneous split-heavy conversion pressure.
  Date/Author: 2026-03-03 / assistant

- Decision: apply 80% scaling to per-book worker knobs (`workers`, `pdf_split_workers`, `epub_split_workers`) during concurrent runs.
  Rationale: preserves requested per-book reduction relative to single-book settings.
  Date/Author: 2026-03-03 / assistant

## Outcomes & Retrospective

Single-profile multi-book runs now have bounded parallelism and split contention protection while preserving existing artifact layout and failure-continuation behavior.

## Context and Orientation

`_interactive_single_profile_all_matched_benchmark` in `cookimport/cli.py` is the interactive path for both:

- `Single config, selected matched sets`
- `Single config, all matched sets`

It builds shared benchmark kwargs from selected run settings and executes one `labelstudio_benchmark(...)` per target matched book.

Split-phase controls are not direct `labelstudio_benchmark` CLI flags here; they are context overrides consumed by the command internals.

## Plan of Work

Update the execution loop in `cookimport/cli.py` to:

1. Detect multi-book batches and configure scheduler knobs:
   - max concurrent books = 3
   - split-phase slots = 1
   - split-phase gate directory under the run root
2. Scale per-book worker kwargs to 90% for concurrent runs.
3. Submit per-book benchmark calls using `ThreadPoolExecutor`.
4. Wrap each run with `_benchmark_split_phase_overrides(...)` so split-slot constraints are enforced by existing ingestion logic.
5. Preserve existing failure aggregation and upload-bundle writing.
6. Update docs to explain new scheduler behavior.
