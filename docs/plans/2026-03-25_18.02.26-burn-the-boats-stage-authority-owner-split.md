# Burn The Boats Stage Authority Owner Split

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan is intentionally adjacent to `docs/plans/2026-03-25_17.43.36-fix-nonrecipe-authority-and-benchmark-incompleteness.md`. That narrower plan repairs the immediate semantic drift in the current seams. This plan is the next cut: make those repaired seams owned by small modules, make the composition roots boring, make the knowledge shard-count surface honest at the owner boundary, and split the remaining giant recipe Codex coordinator into a real package. If the working tree already contains some of the narrow fixes, keep them and build on them; do not reintroduce mixed contracts just to satisfy an older implementation path.

## Purpose / Big Picture

After this change, a contributor should be able to answer the current hard questions by opening one owner module instead of reading four coordinators and guessing which one is “really” in charge. The question “what counts as final non-recipe authority?” should have one code owner. The question “what do we score when review is incomplete?” should have one code owner. The question “what survives knowledge-off or failure fallback?” should have one code owner. The question “what does recipe Codex promotion actually do?” should have one code owner. The file layout should teach the business logic directly.

The user-visible proof is twofold. First, staging and benchmark behavior stays honest: knowledge-off, reviewed, fallback, and partial-review cases all produce predictable authority, late-output, and scoring views without hidden reinterpretation. Second, the file system becomes teachable: `output_stage.py` reads like a composition root, `stage_block_predictions.py` reads like an assembly layer, and the recipe Codex runtime stops hiding planning, validation, promotion, recovery, and reporting inside one `5k+` line file.

This plan does not require live LLM token spend as part of its required validation. The suggested ablation coverage is token-free. In this repository, an “ablation test” here means running the same fixture or synthetic runtime state through multiple settings or failure modes and asserting how the contract changes. Use direct unit/integration tests, fake manifests, monkeypatched `CodexFarmRunnerError`, or `scripts/fake-codex-farm.py` where appropriate. Live Codex end-to-end runs are optional follow-up evidence only and require explicit user approval before any token spend.

## Context and Orientation

The current stage runtime uses five plain-language steps: `extract`, `recipe-boundary`, `recipe-refine`, `nonrecipe-route`, and `knowledge-final`. The code for those steps is spread across `cookimport/staging/pipeline_runtime.py`, `cookimport/staging/import_session_flows/output_stage.py`, and several staging helpers. The business problem is not that these stages are missing. The problem is that more than one file still feels entitled to reinterpret the same truth.

In this repository, “composition root” means the module that wires steps together and performs I/O, but does not decide business meaning itself. `output_stage.py` should be such a root. A “contract object” means one explicit dataclass or pydantic model that carries one layer of truth with clear names. The non-recipe path needs such a contract because the current code still lets routing, late-output selection, final semantic authority, and benchmark scoring drift apart. A “scoring view” means the exact block-level truth that stage-block benchmark code may treat as semantically final. It is narrower than the full runtime status because unresolved review-eligible rows must stay unresolved, not silently become `OTHER`.

The files involved today are:

- `cookimport/staging/nonrecipe_stage.py`, which still owns seed, routing, authority, review status, and several compatibility-style helpers in one file.
- `cookimport/staging/import_session_flows/output_stage.py`, which still chooses some non-recipe late-output behavior directly.
- `cookimport/staging/stage_block_predictions.py`, which currently mixes recipe evidence gathering, knowledge evidence import, label conflict resolution, and benchmark interpretation.
- `cookimport/llm/codex_farm_knowledge_jobs.py`, which owns knowledge packet planning and the shard manifest entries consumed downstream.
- `cookimport/llm/codex_farm_orchestrator.py`, which is still a giant recipe Codex coordinator even though the knowledge stage and canonical line-role code already prove the package-split pattern works better here.

The relevant LLM terminology in this plan is simple. A “packet” is one immutable knowledge-review input payload that owns one bounded slice of outside-recipe blocks. A “shard” is a scheduler bundle that may own one or more packets and later gets split back into per-packet tasks by the knowledge-stage planner. A “recipe stage” here means the recipe Codex runtime currently centered in `cookimport/llm/codex_farm_orchestrator.py`. This plan uses `recipe_stage` as the new package name because that phrase matches the existing `knowledge_stage` package and makes the owner map obvious.

An “ablation test” in this plan is explicitly token-free. It is a deterministic test that compares the contract outputs for the same fixture under different settings or failure modes. The required four cases are: knowledge off, knowledge reviewed successfully, knowledge failure with fallback, and partial review. The assertions must cover final authority blocks, review queue blocks, chunk/table input blocks, and the stage-block scoring view. No live Codex correction or knowledge-review spend is required for those cases.

## Plan of Work

Start with the knowledge shard-count and non-recipe authority seams, because those are the places where user-visible behavior and owner clarity are still most coupled. In `cookimport/llm/codex_farm_knowledge_jobs.py`, make the shard-builder seam the one true owner of `knowledge_prompt_target_count`. If the current working tree already bundles packets correctly, do not redesign it. Instead, narrow the implementation to the honest contract: compute the real shard count from the requested count and packet count, partition packets contiguously in source order, write either single-packet or bundled shard payloads, and delete any dead planner arguments or stale descriptions that imply other meanings. Update `cookimport/config/run_settings.py` so the setting describes target shard count for bundled knowledge packet groups, not “ordered knowledge chunks.” Keep `knowledge_worker_count` as the concurrency knob only.

Then extract one canonical non-recipe authority contract into a new owner module under `cookimport/staging/`. The exact file name should be `cookimport/staging/nonrecipe_authority_contract.py`. This module should define the small plain objects that the rest of staging reads. The minimum surface is: `final_blocks`, `review_queue_blocks`, `excluded_blocks`, `review_status`, `late_output_blocks`, and `scoring_view`. `late_output_blocks` exists because late deterministic outputs and final semantic authority are related but not identical concepts. `scoring_view` exists because stage-block benchmark semantics must be explicit and narrow. Once this module exists, `output_stage.py`, `stage_block_predictions.py`, and any staging/benchmark readers must stop rebuilding their own interpretation from mixed stage fields.

After the contract exists, split `cookimport/staging/nonrecipe_stage.py` by decision type, not by helper count. Create sibling owner modules:

- `cookimport/staging/nonrecipe_seed.py`
- `cookimport/staging/nonrecipe_routing.py`
- `cookimport/staging/nonrecipe_authority.py`
- `cookimport/staging/nonrecipe_review_status.py`

Keep `cookimport/staging/nonrecipe_stage.py` as a thin public facade that wires those owners together and exports the public entrypoints. The important rule is that only `nonrecipe_authority.py` decides what becomes final authority. Routing modules may classify exclusions and build the review queue, but they may not manufacture final `knowledge` semantics for reviewable rows.

With that split in place, slim `cookimport/staging/import_session_flows/output_stage.py` until it becomes orchestration-only. It should run stage steps, call pure builders, write artifacts, and emit progress. It should not decide which non-recipe block set to use for tables, chunks, or scoring. Those decisions must come from `nonrecipe_authority_contract.py`.

Once `output_stage.py` is thin, split `cookimport/staging/stage_block_predictions.py` into focused owners:

- `cookimport/staging/recipe_block_evidence.py`
- `cookimport/staging/knowledge_block_evidence.py`
- `cookimport/staging/block_label_resolution.py`
- `cookimport/staging/stage_block_predictions.py` as the final assembly layer

`recipe_block_evidence.py` should answer “which recipe-local labels can be justified from the staged recipe outputs and archive blocks.” `knowledge_block_evidence.py` should answer “which non-recipe labels are authoritative or unresolved according to the canonical non-recipe contract.” `block_label_resolution.py` should contain the only label-priority and conflict-resolution rules. The root `stage_block_predictions.py` should assemble those pieces into the external artifact shape and nothing more.

After the staging side is stable, split `cookimport/llm/codex_farm_orchestrator.py` into a new package `cookimport/llm/recipe_stage/`. The public file `cookimport/llm/codex_farm_orchestrator.py` should become a thin facade or re-export surface, mirroring how the knowledge stage already works. The new package must have explicit owner modules for:

- `planning.py`
- `workspace.py` or `runtime.py` for worker setup and main loop orchestration
- `validation.py`
- `promotion.py`
- `recovery.py`
- `reporting.py`

Use the existing package splits in `cookimport/llm/knowledge_stage/` and `cookimport/parsing/canonical_line_roles/` as the model. The recipe-stage split should preserve behavior and artifacts, but delete mixed one-file ownership and any dead compatibility helpers that survive only because the old file was too big to reason about.

Finally, update the owning docs. This work touches staging contracts, benchmark semantics, and LLM runtime ownership, so the minimum doc updates are `docs/01-architecture/01-architecture_README.md`, `docs/05-staging/05-staging_readme.md`, `docs/07-bench/07-bench_README.md`, and `docs/10-llm/10-llm_README.md`. Update `docs/plain-english-explanation.md` only if the staged behavior story changes, not merely because file ownership changed. Add or tighten architecture-boundary tests so the new thin roots do not silently regrow.

## Milestones

### Milestone 1: Make the shard-count and authority seams honest

At the end of this milestone, the knowledge shard-count control has one truthful owner at the shard-builder seam, dead planner/config wording is gone, and the non-recipe runtime exposes one canonical contract object for final authority, review queue, exclusions, late outputs, and scoring. The important proof is entirely token-free: fixture-based tests can show five packets plus target two produce two shard entries while preserving five packet tasks, and the four non-recipe ablation cases can assert the exact contract outputs without live Codex calls.

### Milestone 2: Split non-recipe ownership into explicit owners

At the end of this milestone, a newcomer can open one file to answer one question about the non-recipe path. `nonrecipe_seed.py` builds seed state, `nonrecipe_routing.py` honors obvious-junk exclusions and builds the review queue, `nonrecipe_authority.py` decides final authority, and `nonrecipe_review_status.py` owns reviewed/unreviewed bookkeeping. `nonrecipe_stage.py` becomes a small facade, not the place where every detail still lives.

### Milestone 3: Make `output_stage.py` a composition root only

At the end of this milestone, `output_stage.py` performs orchestration and I/O only. It should be possible to point to one pure owner for every business decision it previously made inline. The acceptance proof is that table/chunk input selection, scoring input selection, and fallback/knowledge-off behavior are all delegated to the canonical non-recipe contract and its builders.

### Milestone 4: Split stage-block prediction into evidence plus resolution

At the end of this milestone, `stage_block_predictions.py` reads like a small artifact assembler. Recipe evidence, knowledge evidence, and label-resolution priority rules each have one owner module. The four non-recipe ablation tests should now be wired directly to the contract and evidence/resolution seams, making it hard for scoring drift to sneak back in.

### Milestone 5: Split the recipe Codex coordinator into `recipe_stage`

At the end of this milestone, `cookimport/llm/codex_farm_orchestrator.py` is a thin public seam and `cookimport/llm/recipe_stage/` owns planning, runtime/workspace setup, validation, promotion, recovery, and reporting. A contributor changing recipe-stage promotion or recovery should not need to open a `5k+` line file and hope the right section is nearby.

### Milestone 6: Lock the owner map with docs and tests

At the end of this milestone, the architecture docs point readers to the new owners, and the architecture boundary tests fail if the thin roots regrow or if the new owner modules disappear. This is where the burn-the-boats part becomes durable: the repo should stop teaching mixed ownership after the refactor lands.

## Validation and Acceptance

The plan is complete only if all of the following are true.

The knowledge shard-count surface is honest. A token-free test with five packets and target shard count two must produce exactly two shard manifest entries while preserving five per-packet tasks downstream. `knowledge_prompt_target_count` must mean shard count for bundled knowledge packet groups, and `knowledge_worker_count` must remain the concurrency knob.

The non-recipe ablation suite is honest and token-free. For the four cases knowledge off, knowledge reviewed, knowledge failure with fallback, and partial review, tests must assert all of the following from the canonical contract: final authority blocks, review queue blocks, chunk/table input blocks, and scoring view. Those tests must not call live Codex.

`output_stage.py` is thin enough that a reviewer can read it as orchestration code. It should no longer contain business rules for non-recipe block selection that are also answerable elsewhere.

`stage_block_predictions.py` is thin enough that a reviewer can point to one owner for recipe evidence, one owner for knowledge evidence, and one owner for label priority/resolution. A regression in unresolved handling should fail in the dedicated contract/evidence tests rather than being discoverable only through a large artifact diff.

`cookimport/llm/codex_farm_orchestrator.py` has become a thin public seam over a `recipe_stage` package with explicit owner modules for planning, runtime/workspace setup, validation, promotion, recovery, and reporting. The focused LLM tests and the LLM domain suite must still pass without live-token runs.

The architecture docs and architecture boundary test both describe the new owner map directly.

## Idempotence and Recovery

This work is source-only and safe to repeat. Creating the new owner modules and then deleting old mixed helpers should be done in additive-then-subtractive order so tests stay meaningful throughout. If a milestone stalls halfway, keep the thin-root destination visible in `Progress` and rerun only the affected token-free tests until the new owner module fully replaces the old mixed logic. Do not preserve dead aliases or compatibility fields merely to keep a partially migrated step compiling longer than necessary.

If a planned split reveals that part of Milestone 1 is already satisfied in the working tree, update `Progress`, tighten the tests/doc wording, and continue. Do not “undo then redo” working code just to match the initial draft wording of this plan.
