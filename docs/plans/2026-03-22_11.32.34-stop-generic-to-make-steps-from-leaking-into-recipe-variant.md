# Stop generic `To make the ...` steps from leaking into `RECIPE_VARIANT`

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan builds directly on `docs/plans/2026-03-22_02.25.22-improve-line-role-variant-recall-after-howto-fix.md`. That earlier plan recovered the missing Salt Fat variant subsections successfully, but later benchmark reruns exposed one remaining false-positive family: generic recipe-local `To make the ...` method steps and some immediately following note lines can still inherit variant context too aggressively.

## Purpose / Big Picture

After this change, rerunning the Salt Fat line-role benchmark should keep the strong recovered `RECIPE_VARIANT` sections around rows `1133` through `1136`, `1193` through `1196`, and `1287` through `1288`, while stopping the remaining false-positive variant leak around the Caesar-dressing example. In concrete terms, the benchmark line `To make the salad, use your hands to toss the greens and Torn Croutons...` should no longer come out as `RECIPE_VARIANT`; it should resolve to `INSTRUCTION_LINE`. The trailing `Refrigerate leftover dressing...` and `Ideal for ...` lines should stay `RECIPE_NOTES`.

The visible proof is narrow and specific. In `data/golden/benchmark-vs-golden/2026-03-22_04.26.00`, `HOWTO_SECTION` is still `0 / 0`, `RECIPE_VARIANT` is still generally strong at `precision 0.9697 / recall 0.9143`, but line `1394` in `wrong_label_lines.jsonl` remains an `INSTRUCTION_LINE -> RECIPE_VARIANT` error. After implementation, that exact family should disappear without collapsing the recovered variant sections from the 03:33:50 rerun.

This plan does not claim to solve the full quality collapse in `2026-03-22_04.26.00`. The very large `OTHER -> RECIPE_TITLE` surge and broader `INSTRUCTION_LINE -> OTHER` regressions are treated here as separate issues, because the direct diff between `03.33.50` and `04.26.00` shows a much wider Codex-side shift than this narrow parser follow-up could explain.

## Surprises & Discoveries

- Observation: the first recall-recovery plan largely worked.
  Evidence: `data/golden/benchmark-vs-golden/2026-03-22_03.33.50/single-book-benchmark/saltfatacidheatcutdown/single_book_summary.md` shows `RECIPE_VARIANT` at `0.8919 / 0.9429` with `Gold 70 / Pred 74`, while `HOWTO_SECTION` stayed `0 / 0`.

- Observation: the remaining variant error family is much smaller and more specific than the original recall problem.
  Evidence: `data/golden/benchmark-vs-golden/2026-03-22_04.26.00/.../wrong_label_lines.jsonl` contains only four variant-family wrong rows, and only one of them is the target false positive `INSTRUCTION_LINE -> RECIPE_VARIANT` at line `1394`.

- Observation: the note-row leak is already mostly fixed by the latest parser changes.
  Evidence: in `data/golden/benchmark-vs-golden/2026-03-22_04.26.00/.../line_role_predictions.jsonl`, rows `1400`, `1401`, and `1461` are now `RECIPE_NOTES`, not `RECIPE_VARIANT`.

- Observation: the broad `04.26.00` benchmark collapse is not shaped like a small variant rule tweak.
  Evidence: the direct diff between `03.33.50` and `04.26.00` shows `188` changed labels, dominated by `OTHER -> RECIPE_TITLE` (`99` rows) and `INSTRUCTION_LINE -> OTHER` (`28` rows). That is much wider than the remaining variant leak around the Caesar-dressing rows.

- Observation: a named-variant detector can fail for the wrong reason if it accidentally limits the entire paragraph length instead of just the title-like prefix.
  Evidence: during local experimentation, `To make Caprese Salad ...` and `To make Brown Butter Vinaigrette ...` both stopped matching when the helper treated the whole remainder after `To make` as the candidate name instead of only the leading title-like phrase.

- Observation: the live false positive was not coming from a surviving variant heading or note leak anymore; it was coming from the outside-span `To make ...` classifier treating adjacent ordinary instruction prose as enough recipe-local structure to imply a variant.
  Evidence: the pre-fix local reproduction of rows `1397` through `1401` returned `1399 RECIPE_VARIANT rule ['howto_prefix_prose', 'outside_recipe_variant_prose']` even though the surrounding rows were only `OTHER`, `OTHER`, `RECIPE_NOTES`, and `RECIPE_NOTES`.

## Context and Orientation

The relevant code remains concentrated in `cookimport/parsing/canonical_line_roles.py`. The current implementation already contains helpers such as `_classify_non_heading_howto_prose(...)`, `_looks_named_variant_recipe_name_prefix(...)`, `_looks_variant_run_anchor(...)`, `_looks_variant_run_body_line(...)`, `_is_within_variant_run(...)`, `_variant_label_allowed(...)`, and `_variant_fallback_label(...)`. These helpers collectively decide when a line is itself a variant anchor, when a later line can inherit variant context, and when a model-emitted `RECIPE_VARIANT` should be demoted.

The benchmark evidence that motivated this plan comes from two Salt Fat reruns:

- `data/golden/benchmark-vs-golden/2026-03-22_03.33.50/single-book-benchmark/saltfatacidheatcutdown/`, where variant recall was excellent but the parser still overpromoted a generic Caesar-salad method line and some notes to `RECIPE_VARIANT`.
- `data/golden/benchmark-vs-golden/2026-03-22_04.26.00/single-book-benchmark/saltfatacidheatcutdown/`, where the note rows improved but line `1394` still remained wrong and the overall benchmark regressed for broader reasons outside this plan.

The exact target line from `wrong_label_lines.jsonl` is:

    gold_label=INSTRUCTION_LINE
    pred_label=RECIPE_VARIANT
    line_index=1394
    text=To make the salad, use your hands to toss the greens and Torn Croutons with an abundant amount of dressing in a large bowl to coat evenly. Garnish with Parmesan and freshly ground black pepper and serve immediately.

The nearby rows show why this should not be variant. In the current `04.26.00` codex predictions, the surrounding rows are a normal recipe body followed by notes and then a new title. The phrase `To make the salad` names the current recipe component generically, not an alternate finished dish such as `Caprese Salad`, `Asian Slaw`, or `Classic Torn Croutons`.

The key distinction this plan must preserve is simple:

- A named alternate dish such as `To make Caprese Salad ...` or `To make Brown Butter Vinaigrette ...` can legitimately seed `RECIPE_VARIANT`.
- A generic assembly phrase such as `To make the salad ...` or `To make the dressing ...` is just a method step for the current recipe and should stay `INSTRUCTION_LINE`.

The tests live in `tests/parsing/test_canonical_line_roles.py`. That file already contains deterministic variant-run coverage and Codex-side sanitization coverage. This plan extends those tests with the exact generic-step benchmark family so the next contributor can prove the fix on both the rule path and the model-sanitizer path.

## Plan of Work

The implementation stayed inside `cookimport/parsing/canonical_line_roles.py`. `_looks_generic_to_make_step(...)` now detects determiner-led or lowercase `To make ...` openings that read like current-recipe method prose rather than titled alternate dishes. `_looks_explicit_variant_prose(...)` centralizes the alternate-version cues that still justify a variant label or continuation.

`_classify_non_heading_howto_prose(...)` now uses those helpers directly. Inside recipe span, generic `To make the ...` lines fall to `INSTRUCTION_LINE` unless the same line still carries explicit variant wording or a nearby variant heading. Outside recipe span, named alternate dishes and explicit cue lines can still stay `RECIPE_VARIANT`, but generic make-step prose now prefers `INSTRUCTION_LINE` when instruction support exists and otherwise falls back conservatively.

`_looks_variant_run_body_line(...)` is now intentionally stricter for instruction-shaped rows. Ingredient rows can still stay inside short variant runs, but plain execution prose no longer inherits variant context just because it sits after a variant lead paragraph. That continuation now requires a named alternate dish or explicit alternate-version wording on the current line.

The tests in `tests/parsing/test_canonical_line_roles.py` now cover the real Salt Fat failure mode instead of only the earlier proxy. The new deterministic and Codex-side regressions use the exact `1397` through `1401` outside-span neighborhood, and the continuation controls now prove both sides of the policy: explicit-cue instruction rows can stay variant, while plain follow-up instructions fall back to `INSTRUCTION_LINE`.

The short parsing note in `docs/04-parsing/04-parsing_readme.md` now records the tightened contract so future reviewers know that generic `To make the ...` prose is excluded from `RECIPE_VARIANT` while named alternate-dish paragraphs still survive.

## Validation and Acceptance

The change is acceptable only if all of the following are true.

The exact Caesar-salad false-positive family is gone. In deterministic tests and in any approved benchmark rerun, a generic line like `To make the salad, use your hands to toss ...` must no longer land as `RECIPE_VARIANT`; it must resolve to `INSTRUCTION_LINE`.

Trailing note-only rows remain notes. Lines such as `Refrigerate leftover dressing...` and `Ideal for ...` must stay `RECIPE_NOTES` after a real variant paragraph ends.

Named alternate-dish variant paragraphs still work. Existing positives such as `To make Caprese Salad ...`, `To make Brown Butter Vinaigrette ...`, and the `Variations` sections already recovered in Salt Fat must remain `RECIPE_VARIANT`.

The parser does not give back the HOWTO fix. `HOWTO_SECTION` must remain at zero predictions in the same Salt Fat benchmark family.

Deterministic proof must be recorded. Run the focused variant/howto slice and then the full `tests/parsing/test_canonical_line_roles.py` file in `.venv`, and paste the commands plus short results into this plan’s `Artifacts and Notes` section.

Benchmark proof remains optional until the user explicitly approves another live rerun.

## Idempotence and Recovery

These edits are additive and safe to repeat. The test commands can be rerun without mutating the historical benchmark roots. The benchmark directories under `data/golden/benchmark-vs-golden/2026-03-22_03.33.50` and `.../2026-03-22_04.26.00` should remain read-only reference points.

If the first implementation demotes real named alternate-dish paragraphs such as `To make Caprese Salad ...`, do not keep widening the generic-step negative list blindly. The safe recovery path is to refine the title-like prefix detector so named dishes still count as variants while generic noun phrases do not.

If the first implementation fixes `1394` but reopens the old note leak, stop the variant run before note-only rows again rather than weakening the `RECIPE_NOTES` rules.

If the first implementation reintroduces the old variant recall misses in the `Variations` sections, treat that as a hard regression. Keep the hard negative for generic `To make the ...` steps, but preserve the successful `Variations` and named alternate-dish anchors.
