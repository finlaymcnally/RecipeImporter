## Purpose / Big Picture

The worker should not own canonical packet JSON serialization. The worker is good at making packet judgments from text. The repo is better at writing exact file envelopes, version fields, stable key names, and normalized output structure. After this change, the worker returns a bounded semantic object for one packet and repo code validates, normalizes, and serializes the canonical stored output. This removes fragile final-file formatting from the worker’s authority surface and makes worker mistakes semantic rather than syntactic.

That is the right trade. The March 20 run showed that otherwise-usable work could still be thrown into the repair lane because the worker literally wrote malformed top-level JSON. The runtime should stop paying that tax. The worker should decide whether the packet is useful, how blocks should be classified, and which evidence snippets justify that judgment. The repo should own the final stored shape.

The visible proof is concrete: worker-style malformed file shapes such as literal trailing `EOF` or wrong-version top-level JSON stop being normal mainline output classes because the repo, not the worker, writes the final packet artifact.

  Evidence: when the repo writes canonical output, the cause of failure can be recorded as packet-semantic invalidity instead of being obscured by file-format drift.

## Outcomes & Retrospective

This plan is not implemented yet. Its intended outcome is a knowledge-stage output contract where worker responsibility is semantic and repo responsibility is structural. If it succeeds, later validation and recovery logic can focus on packet quality and grounding rather than on whether a worker got a version field, key name, or JSON envelope exactly right.

The open risk is migration complexity. Current ingest and orchestrator code may already assume worker-written `out/*.json` artifacts. The migration should therefore be staged carefully: add the new internal result type, convert the orchestrator to accept it, keep any compatibility path explicitly non-authoritative, and only then remove old assumptions.

## Context and Orientation

The current output contract is spread across `cookimport/llm/codex_farm_knowledge_models.py`, `cookimport/llm/codex_farm_knowledge_ingest.py`, `cookimport/llm/codex_farm_knowledge_orchestrator.py`, and the prompt/schema assets under `llm_pipelines/`. Today, workers effectively write authoritative-looking `out/*.json` files, which lets small formatting mistakes become stage-level failures.

An `internal semantic result object` in this plan means a packet-scoped value with the packet id, useful-versus-other decision, per-block decisions, reason code, and grounded evidence snippets. Repo code can then convert that object into the canonical stored artifact. The worker should not need to know every incidental envelope field of the stored file format in order to do useful work.

There are two important boundaries here:

- semantic authority belongs to the worker, subject to repo validation;
- structural authority belongs to the repo, because the repo controls stable file shape and normalization.

If this plan is done correctly, later plans can still reject semantically empty or low-trust outputs, but they will not need to spend time repairing trivial top-level shape problems the repo could have prevented entirely.

## Plan of Work

### Milestone 1: Define the packet semantic result type

Add a packet-scoped semantic result type in the knowledge models layer. Keep it small and biased toward the fields the worker is actually qualified to decide. The likely fields are:

- packet id;
- useful-versus-other decision;
- per-block decisions;
- compact reason code;
- grounded evidence snippets or equivalent evidence references.

Do not make the worker reproduce incidental file envelope structure that repo code can write deterministically. If the stored artifact needs version fields, normalized field names, or additional derived metadata, those should be added by repo code after acceptance.

### Milestone 2: Move canonical serialization into repo code

Update ingest and orchestrator code to accept the semantic result object, run validation over it, and serialize the canonical stored packet output. If the current code expects worker-written `out/*.json`, adapt that path so staged worker output is no longer considered authoritative until repo code accepts it.

This migration should be made explicit in code. A future reader should be able to see where worker input ends and where canonical repo output begins. Do not hide the boundary behind a helper that makes worker-produced files look authoritative again.

### Milestone 3: Rewrite prompt and hint targets

Update the prompt asset and hint generation so the worker is clearly aiming at the semantic result object, not at hand-authoring the final stored file. The prompt should show one short positive example of a semantic result for one current packet and one short negative example of a metadata-driven or blanket answer that the runtime will reject.

The prompt should also explain that repo code will handle the final canonical output shape. That instruction matters because otherwise the model may keep trying to optimize for JSON envelope perfection instead of the packet judgment itself.

### Milestone 4: Add migration tests

Add tests proving that worker formatting mistakes no longer define packet success. These tests should include at least one shape inspired by the March 20 failures:

- literal trailing `EOF`;
- wrong top-level version field;
- top-level formatting drift that should no longer exist once the repo serializes canonical output.

These tests should prove the new contract in a direct way: a worker-like semantic result can still become valid final output even when the old worker-owned formatting path would have failed.

## Validation and Acceptance

This plan is complete when:

- one packet-scoped semantic result type exists in repo code;
- the repo serializes canonical packet output from that type;
- worker formatting mistakes no longer define packet success or force the worker to hand-author final file shape;
- the prompt clearly targets semantic review rather than canonical JSON-file authorship.

The clearest acceptance case is a deterministic test where the worker-like semantic result is valid, the repo writes the final packet artifact, and the old malformed-file failure class becomes impossible or irrelevant under the new contract.

## Idempotence and Recovery

If the current orchestrator still needs a temporary staging location for worker packet candidates, that is acceptable as an interim step. The final accepted packet artifact must still be repo-owned. Keep the compatibility path narrow and removable. Do not allow a second parallel notion of “final output” to persist after the migration.

The canonical serialization helper should be deterministic and safe to run repeatedly on the same semantic result. Re-serializing the same accepted semantic result should yield the same canonical structure.

## Artifacts and Notes

The main artifact is the new internal semantic result contract. Keep it compact and easy to reason about. Other useful artifacts are:

- a canonical serialization helper;
- tests for old malformed-file shapes that no longer define packet success;
- updated prompt examples that target semantic results rather than final file envelopes.
