# Line-Role Authoritative Completion Signals

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

After this change, a canonical line-role workspace worker that has already edited `task.json`, run the repo-owned same-session helper, produced valid shard outputs, and marked the helper state as `completed` will stop cleanly as a success. The supervisor will no longer treat that worker as a quasi-failure just because the Codex session emitted a final prose message before the supervisor had enough evidence to trust the result.

The user-visible result is simple. Runs like the `2026-03-31_13.18.44` benchmark will stop recording line-role shards as “validated after watchdog kill” when the worker had actually already finished correctly. Focused tests will prove that helper completion plus valid stable outputs now yields `completed` or `completed_with_warnings`, while the old `workspace_final_message_missing_output` path remains only for genuine missing-output cases.

- Observation: The shared runner already knows how to stop gently when supervision says a workspace worker is complete.
  Evidence: `cookimport/llm/codex_exec_runner.py` already delays termination for supervision states beginning with `completed`, which means the missing behavior is not “teach the subprocess how to die” but “teach supervision when success is authoritative.”

  Evidence: Canonical line-role runtime currently has a dedicated `workspace_final_message_missing_output` watchdog path that starts after `has_final_agent_message`, even though the repo-owned helper state and output files are the true authority for whether the worker’s work is complete.

## Decision Log

- Decision: Fix this at the supervision layer, not by making the helper kill the Codex parent process.
  Rationale: The helper is a repo-owned child command inside the worker session. It can write outputs and deterministic status, but parent-process lifecycle should remain the runner’s job. The shared runner already has a graceful completion kill path, so the correct fix is to improve completion detection rather than invent cross-process self-termination.
  Date/Author: 2026-03-31 / Codex

- Decision: Treat repo-owned helper completion and validated outputs as the authoritative success surface for line-role workers.
  Rationale: The worker’s prose closing message is telemetry. The real contract is the hidden same-session state plus durable validated `out/<shard>.json` outputs. That matches the repository’s stated authority boundary: deterministic code validates and expands outputs; LLM prose does not define correctness.
  Date/Author: 2026-03-31 / Codex

- Decision: Reuse the existing hidden same-session state rather than adding a brand-new completion marker file in the first implementation.
  Rationale: The helper already maintains `_repo_control/line_role_same_session_state.json` with `completed`, `final_status`, `completed_shard_count`, and shard-level status. Reusing that state keeps the change small and preserves one deterministic source of truth. If implementation proves that live sync of this state is still too late or too noisy, a later revision may add a smaller dedicated completion sidecar, but that is not the first step.

- Decision: Keep the current missing-output watchdog branch, but demote it to a fallback path.
  Rationale: Genuine missing-output failures still need a bounded fail-closed path. The fix is not to delete that seam, but to stop using it when the repo can already prove success.

## Context and Orientation

Canonical line-role workspace workers live in `cookimport/parsing/canonical_line_roles/runtime.py`. That module builds the worker-local `task.json`, launches the direct `codex exec` workspace session, and decides when the session should be warned, terminated, retried, or accepted. The line-role same-session helper lives in `cookimport/parsing/canonical_line_roles/same_session_handoff.py`. That helper is deterministic repo code: it reads the edited `task.json`, validates the answer objects, rewrites repair mode if needed, writes final shard outputs under `out/<shard>.json`, and updates hidden helper state under `_repo_control/line_role_same_session_state.json`.

The shared subprocess transport lives in `cookimport/llm/codex_exec_runner.py`. It prepares the sterile mirrored workspace, launches `codex exec`, streams stdout/stderr JSON events, and asks the stage-owned supervision callback what to do. This runner already has the important capability needed for this plan: if supervision returns a state whose name starts with `completed`, the runner waits briefly and then terminates the subprocess gracefully instead of treating it as a hard watchdog failure.

The current bug is a mismatch between “the work is complete” and “the supervisor is willing to believe the work is complete.” In recovered line-role shards, the worker edits `task.json`, runs `python3 -m cookimport.parsing.canonical_line_roles.same_session_handoff`, gets back a deterministic `completed` response, and often already has a valid `out/<shard>.json` on disk. But the watchdog still keys off the worker’s final prose message and a short missing-output grace window. If it does not see enough repo-visible success evidence quickly enough, it kills the worker under `workspace_final_message_missing_output`, and later repo validation rescues the shard as `validated_after_watchdog_kill`.

That is the specific issue this plan fixes. A “workspace worker” here means one long-lived Codex session running inside a sterile mirrored directory. A “same-session helper” means the repo-owned deterministic command the worker is explicitly told to run after editing `task.json`. A “repo-visible completion signal” means evidence that deterministic code can read directly from files the repo owns, not a natural-language assistant message.

## Plan of Work

First, add a small line-role completion summarizer inside `cookimport/parsing/canonical_line_roles/runtime.py`. This code should read the synced hidden same-session state file from the source worker root and answer plain yes/no questions the watchdog actually needs: whether the helper has marked the workspace `completed`, what `final_status` it recorded, how many shard outputs it expects, how many it says are complete, and whether any shard-level validation errors remain. Keep this summarizer deterministic and file-based. It must not infer semantics from the model’s prose output.

Second, update the line-role watchdog callback in `cookimport/parsing/canonical_line_roles/runtime.py` so success is driven by repo-owned evidence in a specific order. When the helper state says the session is `completed` and the expected `out/<shard>.json` outputs are present and stable, supervision should return a completion decision immediately. If warnings were accumulated earlier in the session, the state should be `completed_with_warnings`; otherwise it should be `completed`. This is the happy path. The existing missing-output branch must become a fallback path that applies only when the worker has emitted a final assistant message but the repo still cannot see a helper-complete state or stable valid outputs.

Third, make the watchdog explicitly support the “helper completed before Codex exited” case. This requires a short completion wait window that is keyed to repo-visible helper success, not to assistant prose. Once helper-complete state is seen, the callback should keep polling until outputs are stable for the existing completion-stability rule, then ask the shared runner for a graceful `completed*` termination. If helper-complete state is seen but outputs are not yet synced, the callback should wait inside this completion window rather than immediately starting the old missing-output failure path.

Fourth, adjust final telemetry classification in line-role runtime so a worker that terminated through this new path is recorded as an ordinary completed worker rather than a watchdog recovery. The desired post-change behavior is that the shard row keeps whatever warning metadata it earned, but it no longer lands in `validated_after_watchdog_kill` when the outputs were already authoritative and valid before termination.

Fifth, add focused tests. The most important test is a deterministic reproduction of the existing race: simulate a workspace worker snapshot sequence where the helper state becomes `completed`, outputs are present or become stable shortly after, and the worker emits a final assistant message before the process naturally exits. Before the code change this path would head toward `workspace_final_message_missing_output`; after the change it must return a `completed*` decision. Add at least one negative test proving that a final assistant message without helper completion and without durable outputs still yields the old missing-output failure. Keep these tests local to `tests/parsing/test_canonical_line_roles.py` unless a narrower shared-runner seam is genuinely easier to prove.

Sixth, update the short docs in `docs/10-llm/10-llm_README.md` and `cookimport/parsing/canonical_line_roles/README.md` so the current runtime story is honest. The docs should say that line-role completion is authoritative when repo-owned helper state and output files agree, and that final assistant prose is telemetry only on the happy path.

## Validation and Acceptance

This plan is accepted only when all of the following statements are true.

The first acceptance criterion is behavior. A line-role worker whose same-session helper has already returned `completed` and whose expected `out/<shard>.json` output is present and valid must terminate as `completed` or `completed_with_warnings`, not as `watchdog_killed`, not as `workspace_final_message_missing_output`, and not through a later “recovered after kill” reclassification.

The second acceptance criterion is safety. A worker that emits a final prose message without repo-owned completion evidence must still fail closed under the existing missing-output watchdog branch. This change must not silently bless genuinely incomplete work.

The third acceptance criterion is observability. Worker `live_status.json`, line-role telemetry rows, and line-role summaries must all tell the same success story. A clean helper-complete stop should look like completion with optional warnings, not like a recovered failure.

The fourth acceptance criterion is deterministic proof. The focused pytest commands listed above must pass in `.venv`, and the new tests must demonstrate the before/after race directly without requiring a live Codex benchmark rerun.

## Idempotence and Recovery

This change is safe to implement incrementally because it only changes how the supervisor classifies and terminates already-running line-role workers. The helper remains authoritative for validation and output creation. If a test fails midway through the supervision rewrite, keep the old fallback path intact until the new helper-completion branch is fully covered by tests. Do not delete the `workspace_final_message_missing_output` path until its fallback-only role is proven.

If implementation reveals that helper state is still not synced to the source worker root quickly enough for live supervision, the safe fallback is to add one small dedicated completion marker under `_repo_control/` and include it in the same sync path rather than broadening the authority of assistant prose. Record that decision here if taken.

## Artifacts and Notes

The concrete artifact pattern motivating this plan is:

    same_session_handoff command output:
      {"completed": true, "final_status": "completed", "status": "completed", ...}

    worker closing message:
      "Done. I completed the shard assignment in task.json ... Final status returned: completed."

    final line-role telemetry classification:
      finalization_path = "validated_after_watchdog_kill"
      raw_supervision_reason_code = "workspace_final_message_missing_output"

That combination is the bug. Repo-owned deterministic completion happened before the worker was killed, but supervision still treated the session as if it had not proven success yet.

The intended post-change artifact pattern is:

    same_session_handoff command output:
      {"completed": true, "final_status": "completed", "status": "completed", ...}

    worker live status:
      state = "completed" or "completed_with_warnings"

    final line-role telemetry classification:
      finalization_path = "session_completed"
      raw_supervision_reason_code = null
