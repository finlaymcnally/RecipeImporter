# Make LLM Authority A Single Gate After Validation

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

`docs/PLANS.md` is checked into this repository and this document must be maintained in accordance with it.

This plan builds on two checked-in sibling ExecPlans that already cover the narrow subproblems in detail:

- `docs/plans/2026-03-25_11.46.46-stop-silent-line-role-semantic-rewrites.md`
- `docs/plans/2026-03-25_12.01.17-make-recipe-boundary-acceptance-single-gate.md`

Those two plans should still be maintained if implementation happens through them, but this document is the umbrella authority plan for the broader contract the user wants: once `label_llm_correct` and recipe Codex are structurally valid, they become authoritative, double-acceptance seams disappear, and future contributors cannot quietly reintroduce hidden semantic vetoes.

## User Intent

Exact user intent, restated plainly:

- `label_llm_correct` should be authoritative once structurally valid
- recipe Codex should be authoritative once structurally valid
- hidden semantic rewrites after apparent acceptance are bad
- ‚Äúaccepted now, rejected later‚Äù seams are bad
- the repository should have hard guardrails so future agents do not casually add new post-acceptance semantic vetoes

The intended mental model is simple:

1. structural validation decides whether LLM output is acceptable as output
2. invalid output can still be retried, repaired, or rejected to deterministic fallback
3. valid output should not then be semantically reinterpreted by repo code in a sneaky second pass

That means the repo is still allowed to validate, normalize for schemas, write files, and preserve provenance, but it is not allowed to quietly become a second semantic author after LLM acceptance unless that override is made explicit as its own authority gate.

## Purpose / Big Picture

After this change, the repository should teach one honest story about LLM authority.

On the line-role side, a `label_llm_correct` result that passes structural validation should either survive as the final semantic answer for that row or be explicitly rejected to deterministic fallback. It should not be accepted first and then silently massaged into a different label by repo heuristics.

On the recipe side, a recipe span that is accepted by `recipe-boundary` should already satisfy the real acceptance rule, and a structurally valid recipe Codex result should either promote into final recipe authority or land in an explicitly non-promotable terminal status such as `fragmentary` or `not_a_recipe`. The repo should stop teaching and implementing ‚Äúaccepted once, maybe actually rejected later.‚Äù

The user-visible proof is straightforward. A reviewer should be able to inspect:

- `label_llm_correct/.../labeled_lines.jsonl`
- `group_recipe_spans/.../span_decisions.json`
- `recipe_authority/.../authoritative_recipe_payloads.json`

and trust that those artifacts reflect real authority handoffs rather than provisional answers that later code may quietly overturn.

## Surprises & Discoveries

- Observation: the current repository already has most of the fail-closed machinery needed for a safer authority model.
  Evidence: line-role already has structural validation, retry, repair, and per-task deterministic fallback; recipe Codex already distinguishes `repaired`, `fragmentary`, and `not_a_recipe`; recipe-boundary already records explicit span decisions and rejection reasons.

- Observation: the hardest problem is not missing validation. The harder problem is hidden second-pass semantics after something has already looked accepted.
  Evidence: line-role has `_sanitize_prediction(...)` after accepted Codex promotion; recipe-boundary accepts spans and later demotes them for missing body; recipe Codex accepts valid task outputs but only `repaired` results promote into final recipe authority.

- Observation: there are really three different classes of post-LLM behavior in this repo, and they should not be mixed.
  Evidence: some paths are true semantic vetoes, some are legitimate promotion gates, and some are harmless output-shaping normalizers such as tag casing or schema-safe ingredient-line cleanup.

- Observation: the repo currently uses the word ‚Äúvalid‚Äù in more than one sense.
  Evidence: a payload can be structurally valid as a worker output while still being non-promotable as final authority, and that difference is easy to lose if code or docs only say ‚Äúaccepted‚Äù or ‚Äúvalidated.‚Äù

## Outcomes & Retrospective

This section is intentionally incomplete until implementation lands.

The intended end state is a repository where a novice can answer three questions without reading hidden helper logic:

1. When is an LLM answer structurally valid?
2. When does that valid answer become final semantic authority?
3. What kinds of repo-owned post-processing are still allowed afterward?

If this plan succeeds, the answer will be simple:

- invalid LLM output can still be rejected, repaired, retried, or replaced with deterministic fallback
- valid LLM output does not get silently re-authored by repo heuristics
- output shaping still happens, but it is explicitly mechanical rather than semantic

The main risk after implementation is accidental quality loss where the old hidden vetoes were masking weaknesses in deterministic baselines or prompts. If that happens, the fix should be to improve the real owner stage, not to reintroduce silent second-pass rewrites.

## Context and Orientation

This repository has a five-stage post-import pipeline:

1. `extract`
2. `recipe-boundary`
3. `recipe-refine`
4. `nonrecipe-route`
5. `knowledge-final`

This plan is about the first two LLM-backed authority surfaces inside that pipeline.

‚ÄúLine-role‚Äù means the stage that labels each atomic line of the book with a semantic category such as `RECIPE_TITLE`, `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `OTHER`, or `KNOWLEDGE`. The deterministic baseline is built first. Optional Codex review then proposes corrected labels. Those labels later drive recipe grouping.

‚ÄúRecipe-boundary‚Äù means the stage that groups labeled blocks into accepted recipe spans and rejected pseudo-recipe spans. Today it still has a misleading two-step story because span grouping accepts on title-anchor presence alone while later candidate build can demote the same span for missing recipe body.

‚ÄúRecipe Codex‚Äù means the optional recipe correction surface in `recipe-refine`. It produces compact task outputs that can be `repaired`, `fragmentary`, or `not_a_recipe`. Today those outputs are structurally validated, but only `repaired` promotes into final recipe authority.

The files a novice must understand for this work are:

- `cookimport/parsing/canonical_line_roles/validation.py`
  This is the structural validator for line-role task output.

- `cookimport/parsing/canonical_line_roles/runtime.py`
  This is the line-role runtime that retries, repairs, aggregates, and promotes line-role output.

- `cookimport/parsing/canonical_line_roles/policy.py`
  This is where the current post-acceptance semantic rewrite seam lives.

- `cookimport/parsing/recipe_span_grouping.py`
  This is the grouping layer that currently accepts recipe spans too early.

- `cookimport/parsing/label_source_of_truth.py`
  This is where accepted spans currently can be demoted to `rejected_missing_recipe_body`.

- `cookimport/llm/codex_farm_orchestrator.py`
  This is where valid recipe Codex task outputs are turned into promoted recipe authority or explicit non-promotion outcomes.

- `docs/04-parsing/04-parsing_readme.md`
- `docs/05-staging/05-staging_readme.md`
- `docs/10-llm/10-llm_README.md`
- `docs/plain-english-explanation.md`

Those docs all need to agree on the same authority story after implementation.

There are three distinct categories of post-LLM behavior in this repository:

- Semantic veto: repo code changes or discards the meaning of a valid LLM answer.
- Promotion gate: repo code decides whether a valid LLM answer becomes final authority.
- Harmless output-shaping normalizer: repo code changes formatting or schema-safe representation without changing meaning.

This plan exists to shrink the first category, clarify the second, and leave only the third in the quiet normalizer lane.

## Plan of Work

Start by making the contract explicit in code and docs before changing behavior. The implementation should first introduce one short contract note in the relevant modules and docs that distinguishes:

- structural validation
- semantic authority acceptance
- deterministic fallback
- promotion into final authority
- schema/output normalization

That vocabulary work matters because the current bug pattern depends on blurred words like ‚Äúsanitize,‚Äù ‚Äúaccepted,‚Äù and ‚Äúvalidated.‚Äù

Then complete the line-role cleanup described in `docs/plans/2026-03-25_11.46.46-stop-silent-line-role-semantic-rewrites.md`. The concrete goal is that once a Codex line-role label passes structural validation, the runtime may either keep it or explicitly reject it to deterministic baseline for that row. It may not rewrite it to a third semantic answer. Metadata shaping such as `escalation_reasons` or legal `review_exclusion_reason` carry-forward may still survive, but that behavior must live in helpers whose names make their non-semantic scope obvious.

Next, complete the recipe-boundary cleanup described in `docs/plans/2026-03-25_12.01.17-make-recipe-boundary-acceptance-single-gate.md`. The concrete goal is that `accepted_recipe_span` already means ‚Äúthis span passed the real minimum recipe-body rule.‚Äù If a span lacks recipe body, it must be rejected during grouping, not accepted first and demoted later during ordinary candidate build.

After those two narrower cleanups, tighten the recipe Codex promotion story in `cookimport/llm/codex_farm_orchestrator.py`. The current runtime is already close to correct, but the terminology and artifact story should become more explicit:

- structurally valid task output is not the same thing as promoted recipe authority
- `fragmentary` and `not_a_recipe` are valid task outcomes, but explicitly non-promotable for final recipes
- only `repaired` becomes final recipe authority

The code may already behave that way, but the artifact names, summary counters, and docs should state it in those terms. A reviewer should never have to infer that a ‚Äúvalid recipe result‚Äù was actually a valid non-promotion outcome.

Then add guardrails that future contributors cannot easily dodge. For each of the three seams above, add focused tests that fail if:

- an accepted Codex line-role label is semantically rewritten rather than preserved or rejected to baseline
- an accepted recipe span is later demoted for ordinary missing-body reasons
- a valid recipe Codex outcome disappears from final results without an explicit terminal status and audit reason

After the tests are in place, update the durable docs so they teach the new contract. The parsing docs should say that accepted Codex labels survive unless explicitly rejected to deterministic fallback. The staging docs should say that accepted recipe spans already satisfy body proof. The LLM docs should say that `fragmentary` and `not_a_recipe` are valid task outcomes but not promoted recipe authority. The plain-English walkthrough should stop using any wording that implies ‚Äúaccepted, then maybe silently rewritten later.‚Äù

Finally, add one small reviewer-facing ledger or summary count where needed if the current artifacts still hide these distinctions. The goal is not more logging for its own sake. The goal is that a human opening a run can answer:

- how many Codex rows were preserved
- how many were rejected to deterministic fallback
- how many recipe spans were rejected at grouping time for missing body
- how many valid recipe Codex outcomes were `repaired`, `fragmentary`, or `not_a_recipe`

## Milestones

### Milestone 1: Make Line-Role Acceptance Honest

At the end of this milestone, line-role Codex output will follow one teachable rule: structurally valid rows survive as Codex semantics unless they are explicitly rejected to deterministic fallback. Silent semantic rewrite after acceptance will be gone from the normal path.

The work is in `cookimport/parsing/canonical_line_roles/policy.py`, `runtime.py`, `validation.py`, and the line-role tests. The sibling plan `docs/plans/2026-03-25_11.46.46-stop-silent-line-role-semantic-rewrites.md` contains the narrow implementation detail and should be updated as work proceeds.

### Milestone 2: Make Recipe-Boundary Acceptance Honest

At the end of this milestone, `accepted_recipe_span` will already mean a span has title anchor plus real recipe body proof. The normal downstream path will no longer demote accepted spans for `rejected_missing_recipe_body`.

The work is in `cookimport/parsing/recipe_span_grouping.py`, `cookimport/parsing/label_source_of_truth.py`, and recipe-boundary tests. The sibling plan `docs/plans/2026-03-25_12.01.17-make-recipe-boundary-acceptance-single-gate.md` contains the narrow implementation detail and should be updated as work proceeds.

Acceptance for this milestone is that `span_decisions.json`, `recipe_spans.json`, and built recipes tell one coherent story.

### Milestone 3: Make Recipe Promotion Status Explicit

At the end of this milestone, the recipe Codex path will still support `repaired`, `fragmentary`, and `not_a_recipe`, but the repository will teach those as explicit terminal outcome states rather than one blurred notion of ‚Äúaccepted recipe result.‚Äù Final recipe authority will clearly mean ‚Äúpromoted repaired result.‚Äù

The work is in `cookimport/llm/codex_farm_orchestrator.py`, recipe-stage summary/report helpers, and LLM/staging docs.

Acceptance for this milestone is that a reviewer can inspect recipe runtime artifacts and clearly distinguish valid task outcomes from promoted final recipe authority.

### Milestone 4: Add Guardrails And Teach The Contract

At the end of this milestone, tests and docs make the authority contract hard to regress. A future contributor should have to make an explicit, noisy decision to reintroduce a semantic veto seam.

The work is in focused tests, run summaries where needed, and the durable docs under `docs/04-parsing/`, `docs/05-staging/`, `docs/10-llm/`, and `docs/plain-english-explanation.md`.

Acceptance for this milestone is passing targeted tests and updated docs that all describe the same authority line.

## Validation and Acceptance

Acceptance is behavioral, not merely structural.

First, a valid line-role Codex label that passes policy must survive as a Codex label. If it violates a hard policy, it must be rejected to deterministic fallback explicitly, not silently rewritten.

Second, a span that appears in `accepted_recipe_span` outputs must already have satisfied the real recipe-body acceptance rule. The normal downstream path must not demote it for `rejected_missing_recipe_body`.

Third, a structurally valid recipe Codex result must have an explicit terminal status that tells a reviewer whether it promoted into final recipe authority. `fragmentary` and `not_a_recipe` may remain valid task outcomes, but they must be clearly non-promotable rather than quietly disappearing.

Fourth, docs and artifacts must all tell the same authority story. A reviewer should not need code-level tribal knowledge to know whether a thing was:

- structurally valid
- semantically authoritative
- explicitly rejected to fallback
- valid but non-promoted
- merely normalized for output shape

## Idempotence and Recovery

This work is safe to do incrementally if the narrow line-role and recipe-boundary plans are implemented first and kept up to date.

Do not land half-fixes that remove one hidden gate while leaving old docs or tests teaching the previous contract. The main risk is not broken code. The main risk is a half-migrated authority story that becomes harder to reason about than the current one.

If quality regresses after removing a hidden semantic veto, do not restore the veto quietly. Improve the real owner stage instead:

- deterministic baseline rules for deterministic authority
- Codex prompt/runtime for LLM authority
- explicit promotion criteria where promotion really is the point
