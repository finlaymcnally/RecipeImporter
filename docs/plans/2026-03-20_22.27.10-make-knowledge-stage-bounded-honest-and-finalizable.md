## Purpose / Big Picture

After this change, the non-recipe knowledge stage should behave like a bounded stage runtime instead of a confusing mix of productive work, malformed outputs, watchdog kills, retry churn, stale child processes, and missing final artifacts. A user running a knowledge-heavy CodexFarm benchmark should be able to see true task progress, understand which workers are healthy versus failing, trust that workers either finish or fail closed, and stop the run without losing the stage-level explanation of what happened.

The observable proof is concrete. In deterministic tests and replay-oriented harnesses built from the March 20 Salt Fat artifacts, the knowledge stage should stop leaving `repair_live_status.json` files stuck in `running`, should cap retries and repairs at a small bounded budget, should treat malformed outputs as invalid immediately, should report packet-level progress instead of `task 0/10`, and should always write `phase_manifest.json`, `promotion_report.json`, `telemetry.json`, and `failures.json` under the stage root even when the stage is interrupted or partially fails. With explicit user approval for token spend, a live rerun should then either complete or fail closed without the earlier churn storm.

## Surprises & Discoveries

- Observation: the knowledge stage was not truly idle while it looked stuck; it performed substantial work but did not converge into accepted stage outputs.
  Evidence: the saved Salt Fat run created `485` task packets in `task_manifest.jsonl` and wrote `339` top-level worker `out/*.json` files, but the stage root still lacked `phase_manifest.json`, `promotion_report.json`, `telemetry.json`, `failures.json`, and proposal-level finalization.

- Observation: shard-level worker status and task-level follow-up status currently disagree, which is how a shard can be reported as complete while a child repair subprocess keeps running for minutes.
  Evidence: worker `007` reached `workspace_outputs_stabilized`, yet one `repair_live_status.json` under that worker remained `state=running` with no new events for roughly `488` seconds before the benchmark was killed.

- Observation: the repair path is structurally easier to leak than the watchdog-retry path because the repair call does not currently enforce the same timeout and silence-timeout policy.
  Evidence: `_run_knowledge_repair_attempt()` in `cookimport/llm/codex_farm_knowledge_orchestrator.py` calls `runner.run_structured_prompt(...)` without `timeout_seconds` and builds a strict-JSON watchdog callback without `silence_timeout_seconds`, while `_run_knowledge_watchdog_retry_attempt()` passes both.

- Observation: a completed worker is not necessarily a trustworthy worker.
  Evidence: workers `001` and `002` completed with `reason_code=process_exited_without_watchdog_intervention` and `final_agent_message_state=malformed`; worker `001` wrote `49/49` files that were all invalid JSON because the heredoc terminator `EOF` was included in the file body.

- Observation: the current recovery layer is expensive and low-yield.
  Evidence: the saved run started `122` repair attempts and `77` watchdog-retry attempts. Only `5` repairs ended as `repaired`, only `1` retry ended as `validated`, and several follow-up attempts remained `running` at shutdown.

- Observation: “off task” is broader than “left the workspace.”
  Evidence: some workers stayed inside the workspace but still bulk-generated low-trust outputs from metadata or blanket heuristics instead of grounded chunk review. Worker `007` planned to keep every packet as useful knowledge, and worker `004` reasoned from lane metadata instead of chunk-local evidence.

- Observation: the benchmark UI did not merely omit detail; it materially misrepresented the real state.
  Evidence: the operator saw `task 0/10 | running 10`, even though the actual unit of work was `485` task packets with a mixture of completed workers, watchdog-killed workers, retries, repairs, and stale follow-up calls.

## Decision Log

- Decision: make task packets, not shards, the authoritative unit for knowledge-stage progress, validation, repair, and failure accounting.
  Rationale: the stage writes and validates task-scoped outputs, and every important failure in the postmortem happened at packet scope even when the UI only showed shard scope.
  Date/Author: 2026-03-20 / Codex

- Decision: keep main workspace-worker sessions and strict structured follow-up calls as different execution modes, but make both modes explicit, bounded, and test-backed.
  Rationale: the main worker needs controlled workspace access, while retry and repair are strict JSON recovery calls. The current bug came from treating the modes as conceptually distinct but operationally inconsistent.
  Date/Author: 2026-03-20 / Codex

- Decision: fail closed earlier on malformed or semantically empty outputs instead of treating “file exists” as progress.
  Rationale: worker `001` proved that writing many files can still mean zero usable work. The system should reject those outputs immediately and account for them as invalid, not as half-success.
  Date/Author: 2026-03-20 / Codex

- Decision: stale follow-up prevention must be implemented in runtime lifecycle code, not merely papered over in the UI.
  Rationale: a `repair_live_status.json` left in `running` after its parent shard is terminal is a real process-management bug, not just a reporting bug.
  Date/Author: 2026-03-20 / Codex

- Decision: stage-level finalization is a product requirement, not cleanup.
  Rationale: long CodexFarm runs are expensive. Preserving only worker-local debris after failure or interruption is not operationally sufficient.
  Date/Author: 2026-03-20 / Codex

- Decision: the March 20 Salt Fat saved artifacts are the canonical regression fixture for this work.
  Rationale: they capture the real command shapes, malformed outputs, retry churn, stale follow-up behavior, and artifact gaps that synthetic toy tests alone would under-model.
  Date/Author: 2026-03-20 / Codex

## Outcomes & Retrospective

This plan is not implemented yet. The current outcome is a single canonical design document that turns the postmortem into a concrete runtime rebuild program instead of a list of disconnected bugs. The main lesson so far is that the knowledge-stage failure was not caused by one bad watchdog rule or one bad UI row. The stage needs one coherent contract that covers worker behavior, validation, recovery, progress, cancellation, and finalization together.

## Context and Orientation

The knowledge stage is the optional Codex-backed refinement pass that reviews deterministic non-recipe spans, decides final `knowledge` versus `other` authority for owned blocks, and extracts grounded cooking knowledge snippets. The current stage docs are in `docs/10-llm/nonrecipe_knowledge_review.md`, and the current broader LLM runtime reference is `docs/10-llm/10-llm_README.md`.

The main implementation seam is `cookimport/llm/codex_farm_knowledge_orchestrator.py`. That file currently does all of the following: plans and launches worker assignments, emits knowledge-stage progress payloads, validates task outputs, decides whether a task needs watchdog retry or repair, writes worker and attempt `live_status.json` artifacts, and finalizes stage-level outputs. The direct Codex subprocess seam is `cookimport/llm/codex_exec_runner.py`. The CLI progress renderer is in `cookimport/cli.py`. The stage-level artifact and summary helpers also touch `cookimport/runs/stage_observability.py` and the Label Studio benchmark entrypoints under `cookimport/labelstudio/ingest.py`.

This repository now has two different kinds of Codex execution for knowledge. A workspace-worker session is the long-lived main worker that stays inside a bounded local directory and reads named local files such as `worker_manifest.json`, `assigned_shards.json`, `assigned_tasks.json`, `in/*.json`, and `hints/*.md`. A structured follow-up call is a one-shot strict-JSON attempt used for watchdog retry or repair after the main worker produced a bad or missing task result. The postmortem showed that these two modes are currently mixed incorrectly: the main worker contract is not sufficiently guided, and the follow-up contract is not sufficiently bounded.

The March 20 Salt Fat run showed the current failure taxonomy clearly:

- productive but unfinalized work: hundreds of task outputs existed but the stage root was not finalized;
- watchdog-killed main workers: several workers died on forbidden or overly strict command policy decisions;
- malformed completed workers: some workers wrote outputs and exited cleanly but left malformed JSON or malformed final messages;
- off-task workers: some workers stayed inside the workspace but bulk-generated low-trust outputs instead of grounded review;
- retry and repair churn: the runtime started nearly two hundred follow-up calls with very low recovery yield;
- stale follow-up leakage: follow-up child attempts stayed `running` after their parent worker or shard was already terminal;
- dishonest progress: the UI reported shard counts while the real work was at task-packet scope;
- missing finalization: the stage root did not contain the expected manifest, telemetry, promotion, and failure summary files after shutdown.

The most important current code seams for the stale-follow-up problem are these. `_run_knowledge_repair_attempt()` in `cookimport/llm/codex_farm_knowledge_orchestrator.py` launches repair follow-up calls and currently lacks the explicit timeout and silence-timeout policy that the watchdog-retry path already has. `_build_strict_json_watchdog_callback()` in the same file already supports silence-based termination when `silence_timeout_seconds` is passed. `_finalize_live_status()` only writes a non-running terminal state after the subprocess returns. `_run_codex_exec_subprocess_streaming()` in `cookimport/llm/codex_exec_runner.py` imposes no hard timeout when `timeout_seconds` is omitted. That combination is why `repair_live_status.json` can remain `running` forever if the subprocess never yields a terminal result.

## Plan of Work

### Milestone 1: Turn the postmortem into deterministic regression coverage

Start by converting the March 20 failure classes into explicit tests before changing runtime behavior. The tests must be concrete enough that a novice can prove the postmortem is reproduced locally without spending tokens. Use saved-artifact fixtures, fake-runner seams, and synthetic task directories to model the real bad shapes: malformed heredoc JSON with literal `EOF`, task outputs present but schema-invalid, workers that complete with malformed final messages, watchdog-killed workers, stale `repair_live_status.json` files after shard completion, and packet-based progress for a `485`-task stage.

The minimum file set for this milestone is `tests/llm/test_codex_farm_knowledge_orchestrator.py`, `tests/llm/test_codex_exec_runner.py`, `tests/labelstudio/test_labelstudio_benchmark_helpers_progress.py`, and the existing benchmark interruption test module. If stage-observability summaries need new expectations, extend `tests/staging/test_stage_observability.py` as part of this milestone rather than leaving summary behavior untested.

This milestone is complete when the repository has fail-before tests for every postmortem category this plan claims to fix.

### Milestone 2: Rebuild the worker contract so grounded work is the easy path

The main worker contract currently leaves too much room for either boundary drift or low-value bulk automation. Tighten the repo-written worker instructions and manifests so the worker sees one simple route: open the named local files, process one assigned task packet at a time, quote evidence only from owned chunk blocks, write one JSON file per task to `out/`, and use only bounded workspace-local scratch space if a helper script is truly needed. The manifest should describe the allowed scratch path explicitly and explain that bulk-generating all outputs from metadata or fixed heuristics is invalid even if the files are syntactically correct.

Implement this in the worker prompt/manifests written from `cookimport/llm/codex_farm_knowledge_orchestrator.py` and any shared worker-manifest helper in `cookimport/llm/codex_exec_runner.py` or `cookimport/llm/phase_worker_runtime.py`. If the current worker prompt text lives in a prompt asset or hint file, update that file too. The goal is not only to block bad commands; it is to make the intended grounded loop easier than the dangerous or useless shortcuts.

This milestone also needs cheap deterministic validation for off-task behavior. Add semantic checks that reject blanket “everything is useful knowledge” outputs, outputs whose snippets or `block_decisions` are inconsistent with owned blocks, and outputs that appear templated across many packets without packet-specific evidence. Those checks belong in the ingest and validation seam, not in UI code. `cookimport/llm/codex_farm_knowledge_ingest.py` is the likely home for the semantic-invalid logic, with small helpers in the orchestrator if task-local context is needed.

### Milestone 3: Make task packets the only real unit of progress and terminal state

The knowledge runtime should stop conflating shard ownership with completed work. A shard is only a worker-owned bundle. The real answer unit is the task packet. Create an explicit packet outcome model inside `cookimport/llm/codex_farm_knowledge_orchestrator.py` so each task ends in exactly one terminal state such as validated, semantic_invalid, schema_invalid, malformed_output, watchdog_killed, retry_recovered, repair_recovered, cancelled, or failed_budget_exhausted.

Shard completion must then be derived from packet states, not from the fact that the main workspace worker exited or all `out/*.json` files exist. A shard can only be terminal once all of its packets are terminal and no live child follow-up attempts still own authority over any packet. Worker completion should become a separate transport-level concept that means only “the long-lived workspace session ended,” not “the shard is done and trustworthy.”

This milestone also needs status normalization. Every completed worker `live_status.json` must carry a non-null `reason_code`, but the more important change is that packet and shard summaries must no longer hide transport anomalies behind the generic word `completed`. A malformed completion should be recorded as malformed completion, not as a silent success.

### Milestone 4: Bound retry and repair, and reap stale follow-up calls aggressively

This milestone directly fixes the stale follow-up leak. Add repair-specific timeout and silence-timeout constants alongside the existing watchdog-retry constants, and pass them into `_run_knowledge_repair_attempt()` so repair subprocesses cannot sit in `running` forever. Reuse the existing strict-JSON watchdog callback path in `_build_strict_json_watchdog_callback()` by supplying `silence_timeout_seconds`, and pass `timeout_seconds` through to `runner.run_structured_prompt(...)` so `cookimport/llm/codex_exec_runner.py` has a hard ceiling.

Then add explicit child-attempt ownership and cancellation rules. If a task reaches a terminal good state, cancel any older retry or repair attempt still marked `running` for that task. If a shard is terminal, cancel every still-running task-level follow-up under that shard. If the stage is interrupted, stop spawning new follow-up calls immediately, mark all running follow-up attempts as cancelled or interrupted, and finalize their live-status files even if the underlying subprocess dies abruptly.

Simplify the recovery budget at the same time. The current repair and retry counts are far too high for their yield. Move to a small explicit budget, likely one watchdog retry and one repair per packet unless a test-backed reason justifies more. Once the packet exhausts the budget, mark it terminally failed and let progress advance honestly. A terminal failed packet is much better than a hidden churn loop.

This milestone must also record why a follow-up stopped. Add explicit terminal reason codes such as `timeout`, `silence_timeout`, `cancelled_parent_terminal`, `cancelled_stage_interrupt`, `budget_exhausted`, and `superseded_by_valid_result` in the attempt status artifacts so future debugging does not require inference from missing files and timestamps.

### Milestone 5: Make progress and UI payloads truthful about packets, workers, and follow-ups

Once packet state is real, rebuild the progress payload around it. In `cookimport/llm/codex_farm_knowledge_orchestrator.py`, create a progress-state helper that reports packet totals and terminal counts as the main `task_current` and `task_total` values for the knowledge phase. Include shard counts, current active workers, retry counts, repair counts, and cancelled-follow-up counts as structured detail fields. Worker labels should identify the shard and local packet progress, and when a worker is in a follow-up path the label must say that directly.

Keep `cookimport/cli.py` changes narrow. The preferred fix is a better stage payload, not a custom renderer special case. The UI should be able to render a saved-run shape honestly, for example showing packet counts, completed shard counts, and repair or retry activity instead of a frozen `task 0/10`. The benchmark timeseries should reflect the same truth as the terminal panel so downstream benchmark artifacts are consistent.

This milestone is complete when a deterministic test can simulate a mixed state with completed workers, watchdog-killed workers, retries, repairs, and cancelled follow-ups and the rendered output still reads as obvious and truthful to a human.

### Milestone 6: Finalize the stage root under every terminal condition

Move stage-level finalization out of the “happy path only” mindset. `cookimport/llm/codex_farm_knowledge_orchestrator.py` must always attempt to write `phase_manifest.json`, `promotion_report.json`, `telemetry.json`, `failures.json`, and a stage-level proposal or explicit no-proposal summary under success, partial failure, watchdog collapse, and interruption. The stage root should remain the authoritative summary surface even if raw worker artifacts are preserved for deeper inspection.

Then wire that stage-level truth into the benchmark interruption path. `cookimport/cli.py`, `cookimport/labelstudio/ingest.py`, and `cookimport/runs/stage_observability.py` should treat the knowledge-stage manifest as the source for benchmark partial-finalization summaries. A stopped benchmark should no longer leave the user with only worker debris and a half-written benchmark root. It should say clearly that the benchmark was interrupted, how many packets and shards had reached terminal states, what failure categories existed, and where the stage summary lives.

The important rule for this milestone is that operator interruption must stop future work first and summarize second. The program should not continue spawning retries after the user asked it to stop.

### Milestone 7: Add observability that makes future postmortems cheap

The March 20 postmortem required manual counting of `repair_status.json`, `watchdog_retry/status.json`, malformed outputs, and worker completion categories. Fix that at the source. Extend the stage telemetry and observability summary files to report counts for task packets total, validated, malformed, schema-invalid, semantic-invalid, repaired, retry-recovered, retry-failed, repair-failed, cancelled, timed out, silence-timed-out, and budget-exhausted. Also report main-worker counts for clean completion, malformed completion, watchdog kill, and interrupt cancellation.

These metrics likely belong in `cookimport/llm/codex_farm_knowledge_orchestrator.py` and `cookimport/runs/stage_observability.py`, with small summary rendering updates where benchmark-level artifacts expose them. Keep the metrics compact, but sufficient that a future operator can answer “was this a worker-contract problem, a validation problem, or a recovery-churn problem?” without crawling the whole worker tree.

### Milestone 8: Update the short docs and leave one coherent operator story

After the runtime work lands, update `docs/10-llm/10-llm_README.md` and `docs/10-llm/nonrecipe_knowledge_review.md` so the current-state docs accurately describe the knowledge-stage packet model, bounded follow-up behavior, truthful progress reporting, and stage-level finalization guarantees. Keep the updates short and factual. The docs should explain what the stage writes, what a worker is allowed to do, what retry and repair mean now, and why a stopped run still leaves a useful summary package.

At the same time, revise this ExecPlan so `Progress`, `Decision Log`, `Surprises & Discoveries`, and `Outcomes & Retrospective` reflect what actually landed rather than the original intent.

## Validation and Acceptance

The implementation is complete when all of the following are true.

First, the repository has deterministic tests that reproduce the postmortem categories directly: malformed worker outputs, watchdog-killed main workers, malformed completed workers, off-task but in-boundary outputs, retry and repair churn, stale follow-up leakage, dishonest progress, and missing stage-level finalization.

Second, a task packet always reaches one explicit terminal state, and no follow-up subprocess can remain authoritative after its packet, shard, or stage is already terminal. In particular, a reproduction of the worker `007` stale-repair shape must now end with a non-running final attempt status such as `cancelled_parent_terminal`, `timeout`, `silence_timeout`, or another explicit terminal reason.

Third, the repair path is bounded the same way the watchdog-retry path is bounded. A hanging or silent repair attempt must be terminated by timeout or silence-timeout and must finalize its `repair_live_status.json` accordingly.

Fourth, worker completion semantics are honest. A worker that exited cleanly after writing malformed JSON must not be indistinguishable from a worker whose outputs stabilized and validated. Every completed transport-level worker status must carry a non-null `reason_code`, and packet or shard summaries must expose malformed-completion cases explicitly.

Fifth, the UI and benchmark timeseries report packet truth. The main task counter uses task-packet counts, worker rows show meaningful current state, and mixed states with completed workers plus follow-up churn no longer appear as a frozen `task 0/10`.

Sixth, the stage root is always finalized. Success, partial failure, watchdog collapse, and operator interruption must all leave `phase_manifest.json`, `promotion_report.json`, `telemetry.json`, and `failures.json` under the knowledge root, plus benchmark-level partial summaries that point to that stage-level truth.

## Idempotence and Recovery

This plan is safe to implement incrementally. Each milestone begins with deterministic tests, so the work can be resumed at any point without relying on memory. If a runtime change proves too broad, keep the packet-state model and follow-up cancellation logic, and temporarily narrow the worker-contract or semantic-invalid checks while preserving the new terminal-state accounting. If stage-level finalization wiring proves invasive, land the knowledge-root finalization first and make benchmark-root summaries consume that new stage manifest in a later commit; do not block stage correctness on perfect benchmark packaging.

If a timeout choice for repair or retry proves too aggressive, adjust the constants but keep both hard timeout and silence-timeout in place. Removing the bounds would reintroduce the exact stale-follow-up class this plan is meant to eliminate. If a cancellation path cannot kill a child subprocess cleanly, mark the attempt status cancelled anyway, stop treating it as authoritative, and record the leak in telemetry so the user sees the truth while process cleanup is improved later.

Do not delete the March 20 saved run artifacts while implementing this plan. They are the evidence bundle and the best source for replay-oriented regressions.
