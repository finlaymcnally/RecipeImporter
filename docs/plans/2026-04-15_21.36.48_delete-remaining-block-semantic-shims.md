# Delete Remaining Block-Semantic Compatibility Shims

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The row-authority migration is supposed to be destructive: rows are the semantic unit, and blocks survive only as source layout provenance. Today the repo still carries several block-semantic aliases and payload mirrors that make the old model appear supported. After this change, stage/benchmark/Label Studio helpers should expose row-native names and row-native semantic payloads only. A user should be able to run the narrow staging/benchmark/Label Studio tests and see that no active contract still depends on block-semantic aliases such as `stage_block_predictions_path`, `recipe_block_ownership_path`, block-range task mirrors, or block-based prelabel modes.

## Surprises & Discoveries

- Observation: the public alias cleanup was smaller than the deeper runtime cleanup. Most direct shims were isolated to dataclass properties, writer aliases, and manifest-reader fallback keys.
  Evidence: removing those names only required edits in `import_session_contracts.py`, `deterministic_prep.py`, `bench_artifacts.py`, `writer.py`, and the related tests.

- Observation: the Label Studio export path still accepted block-wide task fixtures even though the active task generator was already row-native.
  Evidence: `export.py` fell back from `source_map.rows` to `source_map.blocks` and synthesized row ids from block indices until this sweep removed that fallback.

- Observation: the semantic scorer payload still exposes duplicated `block_*` mirrors even when rows are the real semantic identity.
  Evidence: `build_line_role_stage_prediction_payload()` writes both `row_labels` and `block_labels`, plus `row_count` and `block_count`, even though the file schema version is `semantic_row_predictions.v1`.

## Decision Log

- Decision: keep `source_block_index` and `source_block_id` fields where they are used as provenance joins, but remove block-semantic mirrors and aliases.
  Rationale: the migration goal is not “never mention a block again”; it is “blocks are not semantic authority.” Provenance back-pointers are still needed for archive joins, context windows, and debugging.
  Date/Author: 2026-04-15_21.36.48 / Codex

- Decision: rename or delete aliases outright instead of leaving deprecation wrappers.
  Rationale: the user explicitly wants delete-delete cleanup, and the existing beads/design docs already say temporary bridges must be removed before the migration is complete.
  Date/Author: 2026-04-15_21.36.48 / Codex

## Outcomes & Retrospective

The repo no longer exposes the most obvious block-era compatibility shims in staging and benchmark helper APIs, and the freeform Label Studio path no longer keeps a dead block-mode prelabel/parser branch or block-wide task/export fixtures in the active contract. The narrow Label Studio and staging tests covering these touched seams now pass.

The remaining work is deeper than naming cleanup. Internal owners such as `stage_block_predictions.py`, some benchmark fixture payloads, and several block-named parser/staging helpers still use block-centric internal terminology or payload mirrors. Those are not public compatibility shims anymore, but they are still part of the broader “delete block semantics” migration story.

## Context and Orientation

The cleanup crosses four connected surfaces.

`cookimport/staging/` owns the stage runtime and the benchmark-facing prediction artifact written under `.bench/<workbook_slug>/semantic_row_predictions.json`. That artifact is already row-native in name, but several helpers still use block-era names for the same file or still construct block-centric payload mirrors. The main owners are `cookimport/staging/writer.py`, `cookimport/staging/import_session_contracts.py`, `cookimport/staging/recipe_ownership.py`, and `cookimport/staging/stage_block_predictions.py`.

`cookimport/cli_support/` and `cookimport/labelstudio/` own benchmark and Label Studio ingest/eval helpers. They still accept or emit several compatibility aliases, especially around prediction manifest fields and freeform task payload ranges. The main owners are `cookimport/cli_support/bench_artifacts.py`, `cookimport/labelstudio/canonical_line_projection.py`, `cookimport/labelstudio/freeform_tasks.py`, and `cookimport/labelstudio/prelabel.py`.

`cookimport/parsing/` still has some block-centric types because the label-first stage aggregates line labels into source-block provenance. That is acceptable only when those block labels are not exported as semantic truth. `cookimport/parsing/label_source_of_truth.py` and `cookimport/parsing/recipe_span_grouping.py` are the relevant seams.

The docs and tests are intentionally strong in this repository. Once code changes are made, the owning docs under `docs/01-architecture`, `docs/05-staging`, `docs/06-label-studio`, and `docs/07-bench` must be updated so they describe only the supported row-native contract.

## Plan of Work

First, audit the runtime seams that already alias row-native artifacts back into block-era names. Remove those aliases from the staging session result, benchmark prediction context, and related helper functions. If a caller still expects the old names, update the caller rather than preserving the alias.

Second, rewrite the semantic prediction payload builders so they stop duplicating row-native semantic data into `block_*` mirrors. The `semantic_row_predictions.json` contract should expose row-native counts, row labels, and unresolved row metadata, while keeping only explicit `source_block_*` provenance inside row records or archive-linked structures.

Third, remove block-based Label Studio task and prelabel surfaces. Task payloads should expose row ranges only. Prelabel should not advertise or normalize a block granularity, and the docs/tests should stop describing block-based labeling as a supported mode.

Finally, update the narrow tests that currently codify these aliases, update the owning docs with the new supported contract, and record the exact finished state in this plan.
