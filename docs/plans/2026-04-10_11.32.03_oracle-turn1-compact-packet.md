# Make Oracle Turn-1 A Single Compact Packet Contract

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

Oracle turn 1 is currently carrying too much of the benchmark artifact tree. The lane-local `index.json` is effectively a copy of the full upload-bundle analysis payload, and the lane-local `payload.json` still includes huge literal artifacts such as full eval reports and verbatim prompt-sample markdown. That makes the first Oracle handoff expensive, fragile, and noisy before the reviewer has even chosen a hypothesis.

After this change, Oracle will get one packet shape only: a compact, summary-first turn-1 bundle under `upload_bundle_v1/{quality,token}/`. The packet will still use the same three durable files, but those files will now always be curated for first-pass review. `overview.md` stays the quick start, `index.json` becomes a compact navigation and summary surface rather than a clone of the full root analysis tree, and `payload.json` becomes a narrow evidence packet with digested heavy artifacts instead of full literal dumps.

This is intentionally a burn-the-boats change. There will not be a parallel legacy Oracle handoff mode, a `v2` upload folder, or a compatibility branch that keeps the old heavy packet alive for Oracle. If the full benchmark artifact tree exists, it remains local source material for deterministic follow-up tooling, not something Oracle sees by default on turn 1.

## Surprises & Discoveries

- Observation: the biggest Oracle turn-1 problem is not the prompt wording, it is the attached packet shape.
  Evidence: the lane prompt text is already concrete and constrained, but the quality lane still launched around `~457k` estimated tokens and the token lane around `~338k`, with most of that driven by attached files rather than instruction text.

- Observation: the lane-local `index.json` is currently a major offender because it serializes almost the full root `analysis` tree again.
  Evidence: in the inspected run, `index.json` was roughly `695k` bytes and about `497k` of that sat under `analysis`, with bulky sections like `changed_lines_stratified_sample`, `failure_ledger`, `regression_casebook`, and `stage_separated_comparison`.

- Observation: the single largest payload row is verbatim prompt-sample markdown, not a structured summary.
  Evidence: `codex-exec/prompts/prompt_type_samples_from_full_prompt_log.md` was about `464k` bytes by itself in both lanes.

- Observation: the quality lane also pays heavily for two full eval reports that largely duplicate topline metric surfaces already available elsewhere.
  Evidence: `vanilla/eval_report.json` and `codex-exec/eval_report.json` accounted for roughly `257k` and `145k` bytes respectively in the inspected quality packet.

- Observation: Oracle transport already treats the lane files as logical packet inputs rather than a promise of raw repo-file fidelity.
  Evidence: the Oracle prompt and browser staging path already warn that Oracle may see a synthetic bundled attachment and should treat `payload.json` `path` values as logical benchmark artifacts.

## Decision Log

- Decision: keep `upload_bundle_v1` and the three-file lane directory shape, but redefine the lane-local contents as compact turn-1 review packets.
  Rationale: this preserves the durable operator-facing path and avoids inventing another bundle namespace, while still letting us burn the old heavy Oracle packet policy.

- Decision: remove the current split where Oracle sometimes sees the full root payload rows and sometimes sees the curated subset.
  Rationale: Oracle should have one input contract only. The lane-local packet should always be curated, regardless of whether the root upload bundle also contains richer local artifacts.

- Decision: keep logical artifact names stable inside `payload.json`, but allow heavy rows to become explicit turn-1 digests instead of full literal content.
  Rationale: this preserves follow-up vocabulary and keeps the reviewer anchored to real benchmark artifacts without paying to ship the full artifact bodies on turn 1.

- Decision: move summary weight into lane `index.json` and keep `payload.json` as focused evidence rather than a second giant summary blob.
  Rationale: Oracle should be able to rank hypotheses from the index and only read payload rows when it needs direct evidence.

## Plan Of Work

First, replace the current lane index builder in `scripts/benchmark_cutdown_for_external_ai.py`. Today it clones the entire root `index_payload` and then annotates it with review-profile metadata. Instead, build a compact lane-local index that carries only the turn-1 summaries Oracle actually needs: topline metrics, compact self-checks, a short review-packet contract block, concise navigation, and a bounded `analysis` section tuned per lane. The quality lane should keep compact quality diagnostics such as turn-1 summary, pair delta summary, active-span breakout, blame summary, triage summary, structure summary, top confusions, and recipe/runtime context. The token lane should keep the spend-side summaries, prompt-budget highlights, runtime summary, compact blame context, and the same minimal benchmark context needed to avoid unsafe cuts.

Then make `payload.json` consistently lane-curated. Remove the existing behavior where non-high-level bundles send the whole root payload with locators. Oracle should always receive only the selected turn-1 rows for that lane. For giant rows, replace full literals with compact digests: prompt-sample markdown should become a short stage-based prompt digest, and full eval reports should become compact metric summaries. Add a generic fallback for oversized structured/text rows so pathological future rows do not silently balloon the lane packet again.

After the packet builders are compact, update `cookimport/bench/oracle_upload.py` so the lane briefs describe the new summary-first contract rather than the old “includes full eval summaries / prompt samples” wording. The token lane brief should stop hardcoding the old giant token estimate and instead describe the compact target budget. Keep the prompts strict about starting with `overview.md`, then `index.json`, and consulting `payload.json` only for focused evidence.

Finally, update the benchmark docs and the owner note in `cookimport/bench/external_ai_cutdown/README.md` so future agents know that Oracle turn 1 now has one compact packet policy only and that heavier evidence belongs in deterministic follow-up packets, not in the initial Oracle upload.
