## Structured Restatement Of User Desire

The repository must keep using `codex exec` as the transport because subscription-backed Codex usage is the economic reason this stage exists at all. The user still wants the strengths of an agentic system: a file-backed workspace, deterministic repo-owned validation, and the ability for the model to correct its own mistakes after checker feedback. What must change is the shape of that agentic system. The current implementation turns cheap semantic labeling into expensive workspace operation. The new design must keep Codex-exec, keep self-correction, keep clean authority boundaries, and delete the old bloated operator-style protocol entirely. This is burn-the-boats work. There will be one new knowledge runtime only. There will be no compatibility mode, no legacy branch, no dual path kept around for safety, and no preservation of the old workspace-heavy contract just because it already exists.

## Purpose / Big Picture

After this change, a user can run the knowledge stage on a full cookbook and get the benefits they actually want from `codex exec`: cheap subscription-backed semantic work, deterministic repo-owned validation, and targeted same-session correction when structure is wrong. The user should no longer pay millions of tokens for the model to behave like a tiny shell operator that repeatedly opens manifests, rereads ledgers, and runs helper commands just to classify a few hundred blocks.

The new live behavior is deliberately narrow. Codex will still work inside a bounded local workspace, but that workspace will present one compact active packet at a time rather than a miniature application with queue files, phase files, helper loops, and broad shell affordances. The repo will own queue advancement, shape validation, and targeted repair packet construction. The model will own only the fuzzy semantic calls: deciding `knowledge` versus `other` and grouping kept rows into useful idea groups. A novice must be able to prove the new design works by inspecting prompt artifacts, token summaries, and tests and seeing that the stage now behaves like a compact semantic lease loop instead of a workspace chore loop.

The current runtime writes those shard inputs under `raw/llm/<book>/knowledge/in/*.json` and then launches one workspace worker per assignment. ## Plan Of Work

The first major change is to replace the current phase-first workspace application with a leased-packet workspace application. The happy path must stop presenting a queue, phase files, work ledgers, and helper commands as the model’s primary interface. Instead, the worker root should expose only the current semantic packet and the minimal repo-written files needed to answer it safely. The packet-lease contract already has generic support in `codex_exec_runner.py` through files such as `current_packet.json`, `current_hint.md`, and `current_result_path.txt`. This plan adopts that shape as the only live knowledge happy path. The worker may still live in a filesystem workspace, but it should behave like “answer the current packet and write the result here,” not “operate a small application.”

The second major change is to redesign knowledge semantics around packet stages that are sized for Codex-exec rather than around the current `knowledge_prompt_target_count` heuristic alone. The new planner must budget both input and output. Every packet must have an explicit maximum estimated input token size and an explicit maximum estimated output size. When a shard would exceed those budgets, the planner must split it before runtime. The repository must stop treating “about five shards” as sufficient planning. Instead, the planner must estimate worst-case response size, especially for pass-1 per-row decisions and pass-2 grouping output. The target is not merely fewer shards; it is predictable upper bounds per packet.

The third major change is to keep self-correction but make it deterministic and compact. The new happy path has two semantic passes managed by the repo. Pass 1 presents the current packet’s owned rows and asks for only `knowledge` versus `other` decisions in a compact output form. The repo validates exact row coverage and allowed labels. If valid, the repo derives the kept-row subset and advances the lease to Pass 2. Pass 2 presents only the kept rows from the current packet and asks for grouping keys and topic labels. The repo validates that every kept row appears exactly once and that no dropped row is grouped. If either pass fails structurally, the repo constructs a tiny repair packet that includes only the unresolved rows plus exact mechanical errors. The model then corrects only those rows. This preserves “mark and correct its own homework” while removing phase-application labor.

The fourth major change is to drastically narrow shell and orientation affordances for the knowledge happy path. The worker prompt must stop encouraging general shell use, queue inspection, or helper-driven orientation. The main knowledge worker prompt should forbid shell use on the happy path, or at minimum make shell use a watchdog violation unless the repo has entered a narrow recovery lane. The worker must not need to open inventories or run commands to understand what to do; the packet lease is the task. If a helper exists, it should be deterministic repo code that the runtime itself invokes, not a command the model must remember to run. The model’s primary artifact should be the result file, not a sequence of workspace actions.

The fifth major change is to simplify file formats for token locality. The current knowledge work ledgers are large JSON documents that invite whole-file reads and rewrites. The new packet files should be compact and append-free. For pass 1, represent rows in a compact ordered list with stable row identifiers, block indices, optional heading level, and text. For pass 2, represent only kept rows with stable row identifiers and text. Repair packets must contain only unresolved rows, prior accepted rows if needed for structural context, and exact mechanical error codes. The model should never need to reread a 200-row ledger to fix five unresolved rows.

The sixth major change is observability. Token reporting must distinguish semantic payload size from protocol overhead. The runtime already measures `visible_input_tokens`, `visible_output_tokens`, and `wrapper_overhead_tokens`, but the new design must also record packet row counts, input byte sizes, repair-row counts, pass kind, and cost per owned row. Stage summaries and benchmark prompt budgets must surface whether cost came from large semantic packets, repeated repairs, or forbidden protocol churn. This plan must also preserve enough machine-readable packet state that the later terminal-reason plan can classify failures precisely without reopening the runtime contract.

The seventh major change is deletion. Remove the old knowledge workspace helper surface, its semantic phase ledgers, its pass-first happy path, and its tests and docs. Where names survive, they must describe the new leased-packet system only. This is burn-the-boats work. The old path must not remain as a fallback branch, feature flag, or adapter.

## Milestones

### Milestone 1: Replace the knowledge happy path with a leased-packet Codex-exec runtime

At the end of this milestone, the live knowledge stage no longer launches phase-first workspace workers. Instead, it launches a leased-packet workspace worker that sees only the current packet files and writes only the current result file. The user-visible proof is that the knowledge worker prompt and prompt artifacts no longer mention `CURRENT_PHASE.md`, `CURRENT_PHASE_FEEDBACK.md`, `check-phase`, or `install-phase` as the happy path.

### Milestone 2: Introduce budgeted two-stage knowledge packets with repo-owned transitions

At the end of this milestone, every knowledge shard is processed through repo-owned Pass 1 and Pass 2 packets sized by explicit input/output budgets. Pass 1 returns compact keep/drop decisions. Pass 2 returns compact group assignments for the kept subset only. The user-visible proof is that packet artifacts and tests show both stages and that large cookbook spans are split by budget instead of by rough shard count alone.

### Milestone 3: Preserve self-correction through compact repair packets only

At the end of this milestone, self-correction still exists, but it occurs only through targeted repair packets generated by deterministic validators. The model is no longer told to operate a workspace helper loop. The user-visible proof is that invalid outputs produce tiny repair packets naming exact unresolved rows and exact structural errors, and valid outputs install without further worker choreography.

### Milestone 4: Tighten worker affordances and remove shell churn from the happy path

At the end of this milestone, the main knowledge worker prompt is packet-centric and does not encourage shell exploration. Any shell use on the happy path is either impossible or a watchdog failure. The user-visible proof is that prompt artifacts, telemetry, and tests show direct result-file writing rather than command-heavy workspace sessions.

### Milestone 5: Rebuild telemetry, budget summaries, and packet-state reporting around packet economics

At the end of this milestone, the repo can explain a knowledge-stage run in terms that match reality: packet counts, pass counts, repair counts, bytes, tokens, cost per row, and machine-readable packet states that later failure-taxonomy work can consume. The user-visible proof is that benchmark summaries and stage summaries no longer need hidden workspace files to explain cost or packet progress.

### Milestone 6: Delete the old path and teach only the new way

At the end of this milestone, the repository has one knowledge runtime story only. Docs, tests, and code no longer describe the old phase workspace as supported behavior. The user-visible proof is that a repository search for the old phase-happy-path vocabulary finds only historical plans and logs, not live code or active docs.

## Validation And Acceptance

Acceptance is behavior, not merely “new files exist.”

The first acceptance criterion is transport correctness. The live knowledge happy path must still use `codex exec`, but it must no longer require a model-operated phase application. A prompt artifact or test fixture for the new happy path must show a current packet, an explicit result path, and no phase-workspace helper choreography.

The second acceptance criterion is budget correctness. A large knowledge shard must be split because of explicit estimated input/output budgets, not because the planner happened to aim for five shards. Tests must prove that oversized packets split deterministically and that the pass-2 grouping packet is smaller because it contains only kept rows.

The third acceptance criterion is correction correctness. The model must still be able to correct its own homework after structural failure. The proof is a test where Pass 1 or Pass 2 returns a structurally invalid payload, the repo emits a tiny repair packet naming only unresolved rows and exact mechanical errors, and the corrected output then installs.

The fourth acceptance criterion is low-churn happy-path behavior. The new knowledge happy path should show zero or near-zero shell commands. A direct structured packet response or a packet-lease file write is expected. A shell-heavy session is a regression. Tests and telemetry fixtures must prove this.

The fifth acceptance criterion is honest token reporting. Prompt-budget and stage-summary artifacts must clearly separate semantic payload cost from protocol overhead and must expose cost per row. A user inspecting one bad run must be able to tell whether the stage was expensive because packets were large, because repairs repeated, or because the worker churned. The same artifacts must also preserve enough machine-readable packet state for the later terminal-reason plan to classify failures without changing the transport again.

The sixth acceptance criterion is burn-the-boats deletion. Repository search after implementation should show no live knowledge happy-path code or docs referring to `CURRENT_PHASE.md`, `CURRENT_PHASE_FEEDBACK.md`, `check-phase`, `install-phase`, or semantic-workspace ledgers as normal knowledge execution.

The seventh acceptance criterion is policy compliance. No implementation validation step may run CodexFarm-enabled live book processing or benchmark reruns without explicit user approval. Fixture-based proof and repo tests are the required validation surface for this plan.

## Idempotence And Recovery

This work is intentionally a clean cutover. It should be implemented as a short series of perfect commits where each commit leaves the repository in one coherent new-contract state. Do not stop in a mixed world where some code assumes phase-workspace knowledge workers while other code assumes packet leases.

The migration is safe to retry because the new runtime is additive until the old runtime is deleted. The order is important. First, build the new leased-packet runtime and tests. Second, switch the top-level knowledge runtime to it. Third, remove the old helper path, old tests, and old docs. Once the top-level switch happens, every remaining mismatch should be fixed immediately rather than bridged.

If a validation command fails halfway through the cutover, keep working toward the new world and rerun the same command. Do not restore the old path as a fallback. The rollback plan, if absolutely required before merge, is to revert the whole cutover commit set together rather than to preserve compatibility logic inside the repository.
