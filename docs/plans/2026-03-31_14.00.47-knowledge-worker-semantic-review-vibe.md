# Knowledge Worker Semantic-Review-First Vibe

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

After this change, the knowledge single-file worker should read like a careful reviewer, not like a coding agent inventing a classifier. The worker-visible contract in `task.json`, the helper-first summary surface, the prompt, and the hint sidecar should all reinforce the same mental model: read the owned block text closely, make a block-local semantic call, treat candidate tags and headings as weak hints, and avoid batching rows into ad hoc rules.

The user-visible proof is local and deterministic. Focused tests will show that knowledge `task.json` now carries a semantic review contract, `editable_task_file --summary` exposes that contract, and the worker prompt and hint sidecar explicitly teach close-reading, anti-heuristic behavior. No live CodexFarm run or benchmark rerun is required for acceptance.

## Context and Orientation

The knowledge stage in this repository is the non-recipe finalize path. Its happy-path worker is a single-file workspace session: the worker sees a repo-written `task.json`, fills only `/units/*/answer`, runs the repo-owned helper `python3 -m cookimport.llm.knowledge_same_session_handoff`, and stops. The files that define this worker-visible experience are `cookimport/llm/knowledge_stage/task_file_contracts.py`, `cookimport/llm/knowledge_stage/recovery.py`, and `cookimport/llm/editable_task_file.py`.

The failure we are addressing is not only “the worker used shell.” It is “the worker behaved like an engineer trying to compress the task into a few rules.” In practice that looks like using heading length, candidate tags, packet shape, or one-off scripts as the decision engine. That produces brittle bulk decisions and defeats the point of using the LLM for fuzzy semantic calls.

In this plan, “semantic review contract” means repo-written worker guidance that says what kind of judgment the worker is being asked to perform, what evidence is primary, what hints are secondary, and which common shortcuts are incorrect. It is not a repo-authored semantic label list or a deterministic decision tree.

The editable task helper in `cookimport/llm/editable_task_file.py` is the narrow helper CLI that workers are already told to use first. The knowledge worker prompt in `cookimport/llm/knowledge_stage/recovery.py` is the human-readable instruction block that frames the job. The knowledge hint sidecar in the same module is a compact markdown note written into `hints/<shard>.md` for extra calibration. The knowledge task-file builder in `cookimport/llm/knowledge_stage/task_file_contracts.py` owns the worker-visible JSON metadata that should survive initial and repair passes.

## Plan of Work

First, extend the knowledge classification task-file builder so the worker-visible file includes a compact top-level semantic review contract. That contract should explain, in plain language, the primary decision question, the priority order of evidence, the meaning of short conceptual headings, the weakness of neighboring context and candidate tags, and the anti-patterns we explicitly do not want. Preserve the same contract when the file is rewritten into repair mode so the worker does not lose that framing after validation.

Second, update `cookimport/llm/editable_task_file.py` so `summarize_task_file(...)` includes a compact summary of the review contract when one is present. The goal is that the existing helper-first startup command becomes psychologically aligned with the task: the worker starts by seeing “you are doing close semantic review” rather than immediately thinking in terms of bulk JSON transformation.

Third, rewrite the knowledge worker prompt and the knowledge hint sidecar language to match that same contract. The prompt should say clearly that the worker is not writing a classifier or compressing the shard into rules. It should emphasize block-local close reading, use candidate tags and heading shape only as weak hints, and tell the worker to stop and read the actual row text if they feel tempted to invent a global heuristic. The hint sidecar should reinforce the same message in short, high-signal bullets.

Fourth, update the focused tests. Add assertions that knowledge task files and helper summaries now surface the semantic review contract, and that the prompt and hint text contain the new anti-heuristic guidance. Keep the tests deterministic and local.

Finally, run the focused `.venv` test slices, update `docs/10-llm/10-llm_README.md` with a short note about the semantic-review-first contract for knowledge workers, and update this plan’s living sections to describe what shipped and what was verified.

## Validation and Acceptance

This plan is accepted only when the following are all true.

The first acceptance criterion is worker-visible guidance. A newly built knowledge classification `task.json` must include a compact semantic review contract that tells the worker the job is close semantic reading, not heuristic synthesis, and that contract must still be present after repair rewrites.

The second acceptance criterion is helper alignment. `python3 -m cookimport.llm.editable_task_file --summary` must surface the semantic review contract in a compact form so the helper-first startup path reinforces the correct mindset.

The third acceptance criterion is prompt and hint alignment. The knowledge worker prompt and hint sidecar must explicitly say that heading shape, candidate tags, and packet order are weak hints only, and they must tell the worker not to invent global keep/drop rules or code up ad hoc heuristic classifiers.

The fourth acceptance criterion is deterministic safety. The new guidance may define review policy and anti-patterns, but it must not give repo-authored semantic labels or deterministic overrides for ambiguous blocks.

The fifth acceptance criterion is local proof. The focused pytest slices listed above must pass in `.venv` without requiring a live benchmark or full book-processing run.

## Idempotence and Recovery

The task-file metadata additions and prompt/hint wording changes are additive and idempotent. Rebuilding the same knowledge task file should produce the same review-contract structure each time for the same shard input.

If the new review contract proves too verbose in `editable_task_file --summary`, trim the returned fields rather than removing the contract entirely. The important property is that the helper-first surface continues to communicate the mindset compactly.

If a repair rewrite accidentally drops the review contract, fix the rewrite path by copying through immutable top-level keys from the original task file rather than re-deriving the contract in a second place.

## Artifacts and Notes

The target worker feeling after this change should be:

    I am reviewing one owned block at a time.
    The block text is primary evidence.
    Heading shape, neighboring rows, and candidate tags are hints only.
    If I feel tempted to invent a rule that covers many rows, I should stop and read the actual rows instead.
    The repo will validate structure, not my semantics.

The change is deliberately compatible with the shell-happy single-file work already in progress. That adjacent work makes generic shell drift less acceptable. This plan makes the correct mental model more obvious before drift starts.
