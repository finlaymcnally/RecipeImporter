# Upgrade Inline-JSON Prompt Cores For Knowledge And Line-Role

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the two active inline-JSON LLM surfaces in this repository, canonical line-role and nonrecipe finalize (knowledge classification plus grouping), will ask smaller, calmer, more human-shaped review tasks. The visible worker prompt should stop forcing the model to do unnecessary bookkeeping such as remembering sparse global ids, reading rows in a pseudo-random order, or carrying a full ontology dump when only a small hint surface is actually useful. A user should be able to inspect the saved prompt artifacts for a run and see that the packets are still semantically rich, but far more orderly and less cluttered than today. On real runs, the observable success condition is fewer missing-row and repair retries for the same books, without weakening deterministic validation.

This plan deliberately focuses on prompt and packet shape only. Budgeting, shard-count projection, and prompt-target planning are being handled elsewhere. The work here is about making each inline-JSON call easier to answer correctly even when the shard size is otherwise acceptable.

## Surprises & Discoveries

- Observation: The biggest knowledge prompt problem is not only size. The prompt surface is asking the model to do semantic review and exact ledger bookkeeping at the same time.
  Evidence: In `data/golden/benchmark-vs-golden/2026-04-06_10.31.48/.../classification_initial_prompt.txt`, the worker must classify every block, cover all rows exactly once, preserve the exact block indices, and supply grounding payloads in one response.

- Observation: Knowledge classification rows are presented in a prompt-hostile order even before budgeting mistakes are considered.
  Evidence: `cookimport/llm/knowledge_stage/task_file_contracts.py` builds classification units and sorts them with `_classification_sort_key(...)`, which hashes `owned_id`. The saved prompt from shard `saltfatacidheatcutdown.ks0000.nr` starts with block indices `175, 0, 230, 228, 205, ...` instead of document order.

- Observation: The current knowledge inline prompt includes a lot of non-evidence text that is either weakly helpful or actively distracting.
  Evidence: One measured prompt was about `76,283` characters, of which about `17,176` characters came from the full ontology dump alone (`20` categories and `203` tags) while the structured packet drops the per-row `candidate_tag_keys` that deterministic code had already computed.

- Observation: The knowledge prompt tells the worker to use nearby context, but the visible inline packet does not actually include the nearby context fields from the task file.
  Evidence: `build_knowledge_classification_task_file(...)` stores `context_before` and `context_after` inside each unit’s `evidence`, but `knowledge_task_file_to_structured_packet(...)` currently emits only block text plus heading/table hints.

- Observation: Most knowledge classification repairs in the expensive run were not hard semantic disagreements. They were contract drift and coverage failures.
  Evidence: Repair packets in `.../nonrecipe_finalize/.../structured_session/classification_repair_packet_*.json` almost always carried `validation_errors=["invalid_category"]`; direct inspection showed that many initial responses simply omitted owned blocks, leaving untouched units with `category = null`.

- Observation: Canonical line-role already has a much thinner prose prompt than knowledge, so its main prompt-core weakness is bookkeeping, not ontology bloat.
  Evidence: `cookimport/parsing/canonical_line_roles/runtime_workers.py` builds a short inline prompt with allowed labels and a packet dump, but the visible response contract still requires exact sparse `atomic_index` echoes for every row.

- Observation: The run artifacts already recorded that prompt-target forcing overrode safer packet sizing, but prompt-core cleanup still matters independently of that planner bug.
  Evidence: `knowledge_manifest.json` from `2026-04-06_10.31.48` says `knowledge_prompt_target_count` forced `10` final shards even though packet-budget planning would have split the queue into `24` shards. This plan keeps that planning issue out of scope and focuses on making each packet easier once created.

- Observation: Removing the ontology dump alone was not enough; the row-local hint surface also had to be capped aggressively to make the prompt materially smaller.
  Evidence: The first pass of the refactor still rendered a `74,805`-character knowledge prompt on the motivating shard because verbose per-row hint payloads expanded the packet. Capping visible `candidate_tag_keys` to four row-local hints brought the same prompt down to `57,340` characters.

## Outcomes & Retrospective

This plan is now implemented. Knowledge inline-JSON packets are in reading order, visible responses use packet-local `row_id` values, the visible packet includes nearby context only when it exists, and the prompt now carries only the shared category catalog plus compact per-row candidate-tag hints instead of the full ontology tag dump. On the motivating April 6 shard, the saved classification prompt dropped from `76,283` characters to `57,340` characters on the same underlying rows.

Canonical line-role now follows the same pattern: the visible inline packet uses local `row_id` values and optional boundary context rows, while deterministic repo code remaps those rows back to stable `atomic_index` ids and reports missing, duplicate, and unknown `row_id` coverage failures explicitly. The semantic ledger promoted after remapping is unchanged when the response is valid.

The remaining gap is live repair-rate evidence. Per repo instructions, no live Codex benchmark rerun was performed without explicit user approval, so the acceptance proof here is saved prompt-artifact evidence plus automated tests rather than a fresh benchmark comparison.

## Context and Orientation

This repository currently has three main LLM-backed semantic stages, but only two of them use the thin inline-JSON transport by default. Recipe refine still uses the heavier taskfile session path, so it is not part of this plan. The two inline-JSON stages in scope are:

`nonrecipe_finalize`, owned by the modules under `cookimport/llm/knowledge_stage/`, which classifies outside-recipe blocks as reusable cooking knowledge or not, then groups kept knowledge rows into idea clusters; and

`line_role`, owned by `cookimport/parsing/canonical_line_roles/`, which labels canonical text rows with structure labels such as `RECIPE_TITLE`, `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `NONRECIPE_CANDIDATE`, or `NONRECIPE_EXCLUDE`.

A “packet” in this repository means the JSON object shown to the model in one inline-JSON call. A “shard” means the deterministic repo-side work unit that owns a contiguous or otherwise fixed slice of rows. Right now, both inline-JSON surfaces ask the model to return one exact ledger row per owned item. The repository validates that ledger and promotes only complete, structurally valid results.

The important current knowledge files are:

- `cookimport/llm/codex_farm_knowledge_jobs.py`, which builds the nonrecipe packets and stores packet metadata such as estimated input and output sizes.
- `cookimport/llm/knowledge_stage/task_file_contracts.py`, which builds the classification and grouping task-file payloads and validates the edited answers.
- `cookimport/llm/knowledge_stage/structured_session_contract.py`, which projects task files into visible inline-JSON packets and prompt text, and parses responses back into task-file answers.
- `cookimport/llm/knowledge_stage/workspace_run.py`, which runs the live inline-JSON session, writes prompt and response artifacts, and handles classification repair plus grouping follow-ups.

The important current line-role files are:

- `cookimport/parsing/canonical_line_roles/runtime_workers.py`, which builds the inline packet and prompt, runs the live inline-JSON attempt, and handles one repair follow-up.
- `cookimport/parsing/canonical_line_roles/runtime.py`, which coordinates the live stage and survivability reporting.
- `llm_pipelines/prompts/line-role.shared-contract.v1.md`, which describes the broader semantic label distinctions used by the line-role family, even though the live inline-JSON prompt is much smaller than the taskfile/file prompt wrappers.

The benchmark evidence that motivated this plan lives in the April 6, 2026 run root:

- `data/golden/benchmark-vs-golden/2026-04-06_10.31.48/single-book-benchmark/saltfatacidheatcutdown/codex-exec/prompt_budget_summary.json`
- `data/golden/benchmark-vs-golden/2026-04-06_10.31.48/single-book-benchmark/saltfatacidheatcutdown/codex-exec/prompts/`
- `data/output/2026-04-06_10.31.48/single-book-benchmark/saltfatacidheatcutdown/codex-exec/2026-04-06_10.32.24/raw/llm/saltfatacidheatcutdown/nonrecipe_finalize/`

Those artifacts matter because they prove that prompt shape, not just planner sizing, is hurting the knowledge stage. The prompt surface itself must be simplified.

## Plan of Work

The implementation should proceed in three narrative milestones.

The first milestone is to redesign the visible row model for both inline-JSON surfaces. Introduce one shared idea: the model should answer against local row ids in the packet, not global ids from the repository. In knowledge classification and grouping, the visible packet should contain ordered rows such as `row_id`, row text, small structural hints, and any compact nearby context that the prompt text explicitly mentions. In line-role, the visible packet should contain ordered rows and optional neighboring context rows, but the model should answer using the packet-local row ids instead of `atomic_index`. The deterministic repository code must keep the authoritative mapping from local row ids back to unit ids, block indices, or atomic indices outside the visible prompt surface.

The second milestone is to redesign the visible knowledge hinting surface. The knowledge classification prompt currently carries the full ontology tag catalog while discarding the per-row `candidate_tag_keys` already computed in deterministic code. Reverse that tradeoff. Keep the visible classification packet small and row-local: preserve reading-order row text, expose a compact shortlist of candidate tags or candidate categories per row, and keep only a tiny shared category list if the model truly needs it to propose a new tag. Do not expose the full `203`-tag catalog to the model during initial classification. If a later part of the stage still needs richer tag information, keep that repo-owned and deterministic unless a narrower prompt actually requires it. While making this change, also include the nearby context rows that the prompt claims are available, or delete the instruction that tells the model to use context.

The third milestone is to harden the response contracts around coverage. Both inline-JSON parsers should stop overloading “bad label value” and “missing answer” into one vague validation story. Teach the repository to detect missing local row ids, duplicate local row ids, and unknown local row ids before semantic label validation. Knowledge classification should surface “missing decision rows” distinctly from “invalid category” or “grounding invalid.” Line-role should do the same for its label ledger. The prompt should be updated to mirror that contract: answer every local row exactly once, using the exact local ids shown in the packet, and no others. This will make repairs more precise and, more importantly, make the root omissions measurable when comparing old and new prompt surfaces.

## Milestone 1: Replace Visible Global Id Bookkeeping With Local Row Ids

At the end of this milestone, neither inline-JSON surface asks the model to echo sparse global ids. Knowledge packets and line-role packets still map deterministically to the same underlying repo rows, but the visible worker contract becomes “answer row `r01`, `r02`, `r03` in order” instead of “repeat these exact repository indices.” This is the first direct fix for the omission root cause.

In `cookimport/llm/knowledge_stage/structured_session_contract.py`, rewrite `knowledge_task_file_to_structured_packet(...)` so the visible packet contains ordered local rows rather than the current `b[*].i` global block indices. Add the inverse remap logic to `build_knowledge_edited_task_file_from_classification_response(...)` and `build_knowledge_edited_task_file_from_grouping_response(...)`. The task file already knows the real unit ids and block indices; use that authority to translate the model’s local-row answers back into repo-owned answers.

In `cookimport/parsing/canonical_line_roles/runtime_workers.py`, rewrite `_build_line_role_structured_packet(...)`, `_build_line_role_structured_prompt(...)`, and the response parsing path so the visible line-role packet uses local row ids while keeping the repo-owned `atomic_index` mapping outside the prompt. Repair packets must keep the same rule: answer only the local rows shown in the repair packet.

The acceptance proof for this milestone is not a live model run. It is a prompt-artifact and parser-contract proof. Add tests that build one knowledge packet and one line-role packet, assert that the saved prompt uses local row ids instead of global ids, feed back a valid local-row response, and verify that the parser restores the correct repo-owned ids in the accepted answers.

## Milestone 2: Make Knowledge Classification Prompt-Friendly

At the end of this milestone, knowledge classification packets are shown in reading order and only carry information that helps the semantic decision. The model no longer sees a hash-sorted row order or a huge ontology dump. The visible packet should feel like a calm ordered worksheet rather than a database export.

In `cookimport/llm/knowledge_stage/task_file_contracts.py`, replace the hash-based `_classification_sort_key(...)` ordering with stable reading order that preserves the packet’s original row sequence. The current hash sort must be removed from the classification task file path entirely. Preserve deterministic stability by sorting on packet order and then block index only when ties are unavoidable.

In the same file and in `cookimport/llm/knowledge_stage/structured_session_contract.py`, keep the useful row-local metadata that exists today but is hidden from the inline prompt. Each knowledge classification row should expose only the compact cues that actually help:

- the row text,
- a short structural hint such as heading level when present,
- optional nearby context text if the prompt tells the model it may use context,
- a compact deterministic shortlist of candidate tag keys or candidate category keys for that row.

Do not include the full ontology tag dump in the initial classification prompt. If the model still needs a shared category reference for proposing new tags, expose only the small category catalog, not every tag in the system.

Rewrite `build_knowledge_structured_prompt(...)` so its prose matches the actual packet. Keep the instruction short. Remove unused rhetoric from the `review_contract` surface when that text is not materially improving decisions. The prompt should state the exact output shape, the exact coverage requirement, and the minimum grounding rule, then present the ordered packet. The prompt should not mention context if no context is present.

The acceptance proof for this milestone is threefold: the knowledge prompt snapshot should be materially shorter than before on the same fixture; the visible rows should appear in reading order; and focused contract tests should prove that candidate-tag hints and context hints appear when expected while the full ontology tag dump is absent.

## Milestone 3: Separate Coverage Failures From Semantic Failures

At the end of this milestone, the repository can tell the difference between “the model skipped rows,” “the model used the wrong ids,” and “the model picked an invalid semantic value.” This does not solve prompt quality by itself, but it is essential to prove that the prompt refactor worked.

In `cookimport/llm/knowledge_stage/structured_session_contract.py` and `cookimport/llm/knowledge_stage/task_file_contracts.py`, add explicit validation and error metadata for missing local rows, duplicate local rows, and unknown local rows before category and grounding validation runs. Knowledge classification should reserve `invalid_category` for truly invalid category strings, not for omitted rows that remained unset in the task file.

In `cookimport/parsing/canonical_line_roles/runtime_workers.py` and the line-role validation helpers it already uses, add the same explicit coverage checks for local row ids. Repairs should carry only the unresolved rows, and the repair metadata should state whether the follow-up is fixing missing coverage, duplicate ids, or semantic label errors.

The acceptance proof for this milestone is new tests that intentionally omit one row, duplicate one row, or answer with an unknown row id and then assert the precise validation code returned. Add fixture tests that mirror the April 6 knowledge omission cases so the repository now reports “missing decision rows” instead of the misleading generic `invalid_category`.

## Validation and Acceptance

Acceptance for this plan is behavior-first, not just code movement.

Knowledge prompt acceptance means:

- a classification prompt fixture is shorter than the old equivalent on the same rows,
- rows appear in reading order rather than hash order,
- the visible prompt no longer includes the full ontology tag dump,
- nearby context appears only when the prompt claims it is available,
- response parsing succeeds from local row ids and remaps back to the correct unit ids and block indices,
- omission cases produce explicit missing-row validation instead of generic `invalid_category`.

Line-role prompt acceptance means:

- the visible inline-JSON prompt uses local row ids instead of `atomic_index`,
- the parser maps those local ids back to the correct canonical rows,
- missing, duplicate, and unknown row ids produce explicit coverage failures,
- existing valid responses still promote to the same label ledger after remapping.

Repository-level acceptance means:

- all focused prompt and contract tests pass,
- `./scripts/test-suite.sh fast` stays green,
- prompt-preview or saved prompt artifacts show visibly calmer packet shapes on the same fixtures,
- when a later approved live benchmark is run, repair counts for omission-style failures should decrease on the same books even if budgeting work is evaluated separately.

## Idempotence and Recovery

The code changes in this plan are additive and repeatable. Packet and prompt builders can be re-run freely because they only regenerate deterministic artifacts. If a partial implementation breaks one surface, revert that surface’s packet builder and parser together; do not leave the repository in a mixed state where the prompt emits local row ids but the parser still expects global ids, or vice versa.

The safest retry strategy is milestone-local. Complete knowledge prompt changes and their tests first, then line-role. If a later milestone reveals a better shared abstraction, move both surfaces onto it only after the focused tests already prove the milestone-local behavior.

Because live Codex benchmark runs are restricted in this repo, treat saved prompt artifacts and deterministic unit tests as the default recovery surface. If a prompt change appears semantically risky, capture the literal before/after prompt text in the relevant test fixture and update this plan with the observation before proceeding.
