## User Intent

Exact user wording:

    "this is not good. instead, i want the knowledge agent to basically do the same thing the line labelling agent does (we'll customize it after). then it can make the chunks of "related ideas" with LLM smarts. doing this deterministically is silly."

The user's intent is straightforward. After the shard-count semantics are repaired, the knowledge agent should stop behaving like a deterministic chunk reviewer and should instead follow the same broad shape as the line-labelling agent. Deterministic code should still prepare bounded work packets and handle the runtime mechanics, but it should not be the layer that decides how the outside-recipe material is grouped into meaningful knowledge units. That semantic grouping should belong to the LLM.

More specifically, the user wants the knowledge step to stop pre-cutting the work into deterministic “knowledge chunks” that the model merely accepts or rejects. Instead, the model should see an ordered packet of review-eligible outside-recipe material and use its own judgment to decide which nearby blocks belong together as one related idea. In plain language: the model should be the thing that figures out “these paragraphs are one concept,” “this heading belongs with these two explanatory blocks,” or “these sentences should stay separate,” because that kind of grouping is exactly the subtle judgment deterministic code is bad at.

This plan is therefore not about tuning the current deterministic chunker. It is about changing the ownership boundary. Deterministic code may still choose safe packet sizes, preserve ordering, isolate giant tables, and enforce exact block coverage, but the final grouping of related ideas must come from the reviewed LLM output. Just as importantly, this plan should stop trying to preserve the current utility-cue side machinery. The user wants this surface trimmed, and the preserved benchmark data does not justify carrying a parallel utility-hint taxonomy forward into the new packet contract. This is a solo project, so “other users might want this feature someday” is not a valid reason to keep dead seams around. The user also explicitly expects more customization later, so this plan should build the architecture that makes that later prompt and product work easy instead of trying to perfect every downstream knowledge feature now.

## Purpose / Big Picture

After the prior shard-count plan lands, the knowledge stage will once again let the operator choose how many shards to plan. That still leaves a deeper problem: the knowledge stage currently decides local grouping with deterministic chunking before the model sees the text, then asks the model to judge one pre-made chunk at a time. The user has explicitly rejected that product contract. The next step is to make the knowledge stage behave more like canonical line-role: deterministic code should prepare bounded ordered packets, but the model should own the semantic grouping of related ideas inside those packets.

After this plan is implemented, a human should be able to run a knowledge-enabled stage or benchmark flow and know that the model is reviewing ordered outside-recipe packets, deciding what belongs together as one idea, and promoting final knowledge groups from that model judgment rather than from a deterministic chunker. The visible proof is that preview/runtime packet inputs no longer carry deterministic chunk ids as the primary semantic unit, knowledge proposals can emit multiple related-idea groups from one packet, and the final reviewer-facing knowledge artifacts are rendered from those model-authored groups instead of from a post-hoc deterministic `chunks_from_non_recipe_blocks(...)` pass.

This plan does not attempt the later customization work the user already anticipates. In particular, it does not add knowledge tags yet and it does not try to perfect the final prompt wording. Its job is to move grouping authority into the LLM-owned review surface, delete the current deterministic utility-cue scaffolding from the live path, and leave a clean seam for later prompt and taxonomy work. The intended end state is hard removal, not a long-lived deprecation story. That means deleting the old bypass/cue machinery from code, prompt surfaces, docs, and tests instead of merely turning it off and keeping it around.

## Context and Orientation

This repository currently has three different concepts that are too entangled in the knowledge stage and must be separated.

A `block` is one normalized document unit in source order. Outside recipes, a block might be a paragraph, a heading, or a table-like row. Block indices are the durable coordinate system used by the stage pipeline, artifact writers, and benchmark/scoring code.

A `shard` is the planning-time slice of a stage’s total work. The prerequisite plan restores the product rule that shard count is an operator choice. Shards should answer “how many slices should the total knowledge review be divided into,” not “how did the deterministic chunker happen to group this prose.”

A `worker` is one concurrent execution slot. Workers can process several shards over time. That remains an execution concern, not the semantic grouping contract.

The new term this plan introduces is `packet`. A packet is one ordered, bounded slice of review-eligible outside-recipe blocks prepared for the model. It is like the canonical line-role packet: deterministic code chooses bounded contiguous review slices for safety, reproducibility, and cost control, but it does not claim that those deterministic packet boundaries are the final knowledge grouping.

One more term matters because this plan removes it: a `utility cue` is the current repo-local hint vocabulary that tries to summarize whether a chunk looks useful cooking knowledge or suspicious filler. In the current code, that includes positive and negative cue lists, strong-negative flags, borderline flags, and worker-facing prose built from those fields. This plan removes that live vocabulary from packet planning and packet review because it duplicates semantic judgment in the wrong layer.

The final semantic unit this plan introduces is an `idea group`. An idea group is a model-authored grouping of one or more kept knowledge blocks that belong together as one related idea. Examples in plain language: a heading plus the two explanatory paragraphs it introduces, or a short troubleshooting trio of blocks that all explain the same mechanism. Idea groups are the thing the user wants the model to decide with “LLM smarts.”

The current live knowledge path is spread across these files:

- `cookimport/llm/codex_farm_knowledge_jobs.py` builds input bundles today from deterministic knowledge chunks.
- `cookimport/llm/knowledge_stage/runtime.py` calls the planner, resolves workers, and executes knowledge worker assignments.
- `cookimport/llm/knowledge_prompt_builder.py` and `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md` define the current prompt text and input embedding.
- `cookimport/llm/codex_farm_knowledge_models.py` and `cookimport/llm/codex_farm_knowledge_ingest.py` define and validate the current chunk-owned output contract.
- `cookimport/llm/codex_farm_knowledge_writer.py` turns promoted outputs into reviewer-facing files such as `knowledge.md` and `snippets.jsonl`.
- `cookimport/staging/import_session_flows/output_stage.py` currently reruns deterministic chunking after knowledge-final and stores those chunks in `ConversionResult.chunks`.
- `cookimport/staging/stage_block_predictions.py` and `cookimport/parsing/label_source_of_truth.py` still consume `ConversionResult.chunks`, so those readers need to be rewritten or sharply simplified rather than protected indefinitely by compatibility shims.

The closest existing pattern to copy is canonical line-role:

- `cookimport/parsing/canonical_line_roles/planning.py`
- `cookimport/parsing/canonical_line_roles/runtime.py`
- `cookimport/parsing/canonical_line_roles/prompt_inputs.py`
- `cookimport/parsing/canonical_line_roles/validation.py`

That path already demonstrates the intended separation of concerns: deterministic code chooses packet boundaries and worker/runtime mechanics, while the LLM decides the actual semantic corrections inside those packet boundaries.

## Plan of Work

Start by locking the new contract boundary. Add a new knowledge pipeline id, prompt template, and model/output schema for packet-based review. Do not mutate the existing `recipe.knowledge.compact.v1` prompt in place. The new prompt should tell the model that it owns one ordered packet of review-eligible outside-recipe blocks, that its first job is still to decide block-level `knowledge` versus `other`, and that its second job is to group the kept `knowledge` blocks into one or more related-idea groups. Those groups must be grounded to owned block indices only. The prompt can still carry mechanically true structure such as heading level, table hints, and nearby recipe guardrails, but it must not present deterministic semantic chunk ids as the answer unit, and it must not rely on repo-authored utility-cue summaries to steer the model toward semantic conclusions.

Then replace live planning. In `cookimport/llm/codex_farm_knowledge_jobs.py`, stop using `chunks_from_non_recipe_blocks(...)` as the live planning input for knowledge review. The planner should instead take the ordered review-eligible outside-recipe block rows from `nonrecipe-route` and partition them into deterministic packets. Packet planning should remain deterministic and cheap, but it should use only safe mechanical boundary signals: owned block order, hard size caps, maybe large table isolation, and similar non-semantic packet-sizing rules. It must not claim that those packet boundaries are the final related-idea group boundaries, and it must not carry forward utility-profile scoring, positive/negative cue tagging, strong-negative flags, or borderline flags as planning inputs for semantic routing. The prerequisite plan’s `knowledge_prompt_target_count` should still control approximate shard count here, but now those shards are packet shards rather than deterministic chunk shards.

While replacing planning, remove the live deterministic bypass lane for review-eligible material. The stage may still inherit upstream final exclusions from `nonrecipe-route`, and it may still enforce structural packet limits before spend, but it should no longer finalize review-eligible packets as semantic `other` purely from deterministic utility cues. More broadly, this is also the right seam to delete `_build_deterministic_knowledge_bypass_candidate(...)`, `_apply_deterministic_knowledge_bypasses(...)`, and the surrounding cue-taxonomy metadata instead of preserving them as dormant advisory machinery. The semantic decision and the grouping decision both belong to the model now. Treat this as a real code deletion pass: remove the old helpers, prompt fields, counters, tests, and doc language that only make sense if the cue/bypass system still exists.

Next, replace the output contract and promotion logic. In `cookimport/llm/codex_farm_knowledge_models.py`, define a packet-based output schema that includes exact block coverage plus idea-group coverage. The minimum required shape should be:

1. Packet-level identity, echoing the owned packet id.
2. Block decisions for every owned block, in order, with final `knowledge` or `other`.
3. One or more idea groups that cover only the `knowledge` blocks. Each idea group should have a stable packet-local id, a short human-readable topic label, the ordered owned block indices it groups together, and grounded snippets/evidence.

Update `cookimport/llm/codex_farm_knowledge_ingest.py` so validation no longer keys off deterministic `chunk_id` coverage. Instead, validation must prove all of the following:

- every owned block appears exactly once in block decisions
- every knowledge-classified block belongs to exactly one idea group
- no `other` block appears inside an idea group
- every idea group uses only owned block indices
- snippets and evidence quotes stay grounded to owned block text
- packet output order is stable and reproducible enough for deterministic promotion

Then update runtime/taskization in `cookimport/llm/knowledge_stage/runtime.py` and nearby helpers. The worker/runtime mechanics can keep the existing long-lived workspace-worker style, batch sidecars, and assignment model, but the task payload now needs to be packet-based rather than chunk-based. The runtime should be allowed to split one shard into one or more packet tasks if needed for batch sizing, but the semantic unit in those tasks is now the ordered packet, not a deterministic chunk id. Review any queue metadata, progress labels, failure reasons, replay/test fixtures, and worker-facing helper text that still assume “one task owns exactly one chunk” or that still surface utility-positive/negative cues, strong-negative warnings, or borderline cue prose as part of the live review contract.

After the runtime can promote packet outputs, migrate the final artifact surfaces. Add a new explicit authority artifact for promoted knowledge groups, for example `09_nonrecipe_knowledge_groups.json`, or an equivalently named sibling beside `09_nonrecipe_authority.json`. Make `cookimport/llm/codex_farm_knowledge_writer.py` render `knowledge.md` and `snippets.jsonl` from those promoted idea groups, not from deterministic chunk artifacts. In `cookimport/staging/import_session_flows/output_stage.py`, stop treating `chunks_from_non_recipe_blocks(...)` as the source of final knowledge grouping after `knowledge-final`. The target end state is that the live path no longer needs `ConversionResult.chunks` as a knowledge-group authority seam at all.

Then do the downstream cutover as a real deletion pass. Audit every current reader of `ConversionResult.chunks`, especially `cookimport/staging/stage_block_predictions.py` and `cookimport/parsing/label_source_of_truth.py`, and rewrite them to use the new explicit promoted knowledge-group artifact or to stop depending on knowledge chunks entirely. If a temporary adapter is required to keep the tree passing during implementation, treat it as same-wave scaffolding and delete it before this plan is marked complete. The important rule is that no live authority or reviewer-facing artifact should be regenerated from deterministic chunking once the packet-based contract exists, and no dead compatibility lane should survive because someone imagines a future user might want it.

Finally, align preview, docs, and tests. `cookimport/llm/prompt_preview.py` must preview packet-based knowledge prompts instead of deterministic chunk prompts. `docs/10-llm/10-llm_README.md`, `docs/04-parsing/04-parsing_readme.md`, `docs/05-staging/05-staging_readme.md`, and [docs/plain-english-explanation.md](/home/mcnal/projects/recipeimport/docs/plain-english-explanation.md) must stop teaching deterministic chunk-owned knowledge review as the live product contract, and they must stop teaching utility cues or bypass heuristics as part of the live semantic story. The docs should explain the new plain-English truth: deterministic code prepares bounded ordered packets, the model decides related-idea grouping and final `knowledge` versus `other`, and final knowledge groups come from that reviewed output.

## Validation and Acceptance

The change is correct only if all of the following are true.

Planning acceptance means live knowledge planning no longer starts from deterministic semantic chunks. A code-level proof should show that the active planning path for knowledge review can operate entirely from ordered review-eligible block rows and packet-sizing rules, without calling `chunks_from_non_recipe_blocks(...)` as the live semantic planner and without depending on utility-cue metadata to make semantic skip decisions.

Prompt-contract acceptance means the active knowledge prompt and validator no longer require “exactly one owned chunk” or “exactly one result row for that chunk.” Instead, a packet payload can cover an ordered slice of owned blocks and the output can contain more than one idea group inside that packet. Tests should use fake outputs to prove this deterministically. The active prompt surface must also stop depending on utility-positive/negative cue summaries, strong-negative flags, or borderline cue hints as model-facing semantic scaffolding.

Promotion acceptance means a fake packet result with multiple related-idea groups validates, promotes, and writes final artifacts successfully, while malformed outputs fail closed. At minimum, tests must prove failure on duplicate block coverage, missing block decisions, knowledge blocks omitted from all groups, and snippets that cite non-owned evidence.

Artifact acceptance means reviewer-facing outputs such as `knowledge.md` and `snippets.jsonl` are rendered from promoted idea groups, not from a later deterministic chunk pass. A stable fixture or fake-run test should be able to prove that the rendered groups match the packet output exactly, including group order and topic labels.

Cutover acceptance means downstream readers that currently depend on `ConversionResult.chunks` have been rewritten, deleted, or otherwise removed from the live knowledge-group authority path. A temporary adapter is acceptable only while the change set is in flight; the completed plan should not leave `ConversionResult.chunks` as a standing compatibility surface for old knowledge behavior. No live surface should claim that deterministic chunking is still the source of final grouping authority.

Removal acceptance means the live knowledge runtime no longer ships the current utility-cue language as part of packet planning or packet review. Focused tests and targeted searches should prove that the packet path no longer produces or consumes `utility_positive_cues`, `utility_negative_cues`, `strong_negative_utility_cue`, `utility_borderline`, or deterministic-bypass metadata on the live route. The intended proof is deletion, not deactivation: those runtime fields, helper branches, and worker-facing cue prose should be gone from active codepaths rather than left in place behind a switch.

User-visible acceptance means the shard-count semantics restored by the prerequisite plan remain intact. The operator still chooses knowledge shard count in the CLI, preview still reflects that shard target, but the semantic grouping inside those shards is now model-authored rather than deterministic.

Documentation acceptance means [docs/10-llm/10-llm_README.md](/home/mcnal/projects/recipeimport/docs/10-llm/10-llm_README.md), [docs/04-parsing/04-parsing_readme.md](/home/mcnal/projects/recipeimport/docs/04-parsing/04-parsing_readme.md), [docs/05-staging/05-staging_readme.md](/home/mcnal/projects/recipeimport/docs/05-staging/05-staging_readme.md), and [docs/plain-english-explanation.md](/home/mcnal/projects/recipeimport/docs/plain-english-explanation.md) no longer teach deterministic chunk-owned knowledge review as the current live behavior.

## Idempotence and Recovery

This refactor should be staged additively while implementation is in progress, but the destination is still a hard cut. The safest rollout is:

1. add the new packet-based prompt/models/validators beside the old chunk-based path
2. migrate runtime planning and promotion to the new path
3. switch final artifact generation to the new promoted groups
4. if absolutely necessary, use a tiny temporary adapter while rewriting the last reader
5. delete dead chunk-owned runtime code and that temporary adapter before calling the plan done

If a checkpoint lands where preview is packet-based but promotion still expects chunk ids, stop and repair the contract before merging further. If a checkpoint lands where final `knowledge.md` is still built from deterministic chunks after packet review succeeded, treat that as a blocking regression, because it would make the user-visible artifact lie about where grouping authority came from.

Because this plan changes core runtime contracts, keep cutover tests in place while the tree is mid-refactor. Do not let temporary coexistence become a supported end state. A rollback path is simple if the new pipeline id is introduced cleanly: restore the old default knowledge pipeline id while debugging, but do not keep both semantic systems alive long-term.
