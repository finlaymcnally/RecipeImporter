## Operator Intent (Structured) - DO NOT DELETE

The operator wants one clean ownership contract for archive blocks. A block is either recipe-owned or it is available to nonrecipe. There is no legitimate middle state where both stages can reason about the same block and a later resolver decides who wins. Recipe-boundary establishes ownership first. Recipe-refine may keep that ownership or explicitly divest a block, but it may not silently drift ownership through a side effect such as changing recipe payloads or provenance ranges. Nonrecipe routing and nonrecipe finalize may see only blocks that are unowned from the start or blocks that were explicitly divested. If any later code path produces both recipe-local evidence and nonrecipe `knowledge` for the same block, that is not a normal merge. It is a bug and should fail as an invariant violation. This work is in burn-the-boats mode. Do not preserve the old overlap-tolerant path, compatibility shims, or legacy fallback behavior.

## Purpose / Big Picture

After this change, the pipeline will have one authoritative answer to the question "who owns block `N`?". That answer will be computed once at recipe-boundary, optionally narrowed by explicit divestments during recipe-refine, and then enforced everywhere else. Nonrecipe routing will consume only blocks that the ownership contract says are available. Final nonrecipe authority will be forbidden from writing `knowledge` onto recipe-owned blocks. Stage-block scoring will stop treating recipe-local versus `knowledge` overlap as something to resolve after the fact. If overlap appears, the run will fail loudly because the ownership contract was broken upstream.

This matters because the current code already mostly intends this boundary, but it does not encode it as one first-class contract. The docs say `recipe-refine` may not change recipe ownership and `nonrecipe-route` already skips accepted `recipe_spans`, yet stage scoring still has an overlap-tolerant merge path where recipe evidence is rebuilt later from `conversion_result.recipes`, nonrecipe evidence is added separately, and `resolve_stage_block_label(...)` silently drops `KNOWLEDGE` if recipe-local labels are also present. That is the wrong shape for the contract the operator wants.

The proof for this plan is deterministic and local. After implementation, synthetic staging tests will show that recipe-owned blocks never enter the nonrecipe queue, explicitly divested blocks do enter it, and any attempt to attach nonrecipe `knowledge` to a recipe-owned block raises an invariant error instead of being silently resolved away. No Codex-backed benchmark or book-processing run is required to validate the refactor.

## Context and Orientation

In this repository, a "block" is one extracted archive row from the canonical source model. A "recipe-owned block" is a block that belongs to an accepted recipe span and is therefore part of recipe authority, not nonrecipe authority. A "divestment" is an explicit act by recipe-refine that removes one or more blocks from a recipe's owned set and returns them to the unowned pool. An "invariant violation" is a bug in deterministic repo code where two supposedly disjoint authority lanes produce incompatible claims about the same block.

The current ownership story starts in `cookimport/parsing/label_source_of_truth.py`. `build_conversion_result_from_label_spans(...)` converts accepted `RecipeSpan` records into `RecipeCandidate` records and also computes `non_recipe_blocks` as every archive block not present in the accepted spans. This is the first real ownership boundary.

The grouping step that decides which blocks become accepted recipe spans lives in `cookimport/parsing/recipe_span_grouping.py`. That module already normalizes rejected recipe-local-looking blocks back to `NONRECIPE_CANDIDATE` or `NONRECIPE_EXCLUDE`, which is why the current authoritative block labels are usually disjoint.

The stage runtime lives in `cookimport/staging/pipeline_runtime.py`. `run_recipe_boundary_stage(...)` already computes `recipe_block_indices` from accepted spans and stores `recipe_owned_blocks` in `RecipeBoundaryResult`. `run_nonrecipe_route_stage(...)` later calls `build_nonrecipe_stage_result(...)`, which again excludes those recipe blocks from nonrecipe routing.

The trouble starts later in `cookimport/staging/stage_block_predictions.py`. That module does not use the already-known recipe-owned block set as its recipe evidence boundary. Instead, it rebuilds recipe-local evidence from `conversion_result.recipes` through `cookimport/staging/recipe_block_evidence.py`, adds nonrecipe `KNOWLEDGE` evidence from `cookimport/staging/knowledge_block_evidence.py`, and resolves conflicts through `cookimport/staging/block_label_resolution.py`.

The knowledge pipeline also still consumes `recipe_spans` directly in a few places. `cookimport/staging/pipeline_runtime.py::run_nonrecipe_finalize_stage(...)` passes `recipe_spans` into `run_codex_farm_nonrecipe_finalize(...)`, and `cookimport/llm/codex_farm_knowledge_jobs.py` uses those spans to decide which archive blocks are excluded from knowledge packets. That is acceptable only as long as recipe ownership can never change after boundary. Once explicit divestment exists, raw `recipe_spans` are no longer enough.

The current design therefore has two ownership representations:

1. The real one used by nonrecipe route today: accepted recipe spans from label-first boundary.
2. A second one implied later by recipe payload projection and stage-block scoring.

This plan removes the second ownership authority and keeps only the first, with one explicit post-boundary divestment hook.

## Milestones

### Milestone 1: Introduce one recipe-ownership contract

At the end of this milestone, the repository will have one dedicated data structure that answers three questions for every block: is it currently recipe-owned, which recipe owns it, and was it later divested. The contract will be created immediately after recipe-boundary and carried forward through recipe-refine and nonrecipe stages.

The proof for this milestone is a targeted staging test that builds a synthetic label-first result with accepted spans and verifies that the ownership contract contains the expected owned block indices, unowned block indices, and per-recipe ownership entries.

### Milestone 2: Add explicit divestment and remove silent ownership drift

At the end of this milestone, recipe-refine will no longer be able to affect ownership implicitly through recipe payload provenance. If recipe-refine wants to return blocks to nonrecipe, it must do so by emitting explicit divestment records. In the first implementation, the live recipe-refine path may emit an empty divestment list, but the seam and deterministic contract must exist.

The proof for this milestone is a synthetic test that applies a divestment update to a recipe-owned block and verifies that the block leaves the owned set, appears in the divested set, and becomes eligible for nonrecipe routing.

### Milestone 3: Make nonrecipe admission depend only on ownership

At the end of this milestone, nonrecipe route, nonrecipe finalize, prompt preview, and knowledge job planning will all consume the same ownership-derived blocked-index view. Blocks that remain recipe-owned are never routed, never packetized, and never finalized by nonrecipe code. Blocks that are unowned or explicitly divested are the only candidates.

The proof for this milestone is a combination of staging and LLM planning tests showing that recipe-owned blocks are excluded everywhere and that divested blocks do reappear in the nonrecipe candidate queue and knowledge packet inputs.

### Milestone 4: Replace overlap resolution with invariant checks

At the end of this milestone, stage-block prediction assembly will no longer merge recipe-local and knowledge evidence as if overlap were valid. Recipe evidence will be restricted to the ownership contract. Knowledge evidence landing on a recipe-owned block will raise an invariant violation. `resolve_stage_block_label(...)` will stop deleting `KNOWLEDGE` as a fallback because that overlap should no longer exist.

The proof for this milestone is a staging test that injects nonrecipe `knowledge` onto a recipe-owned block and expects a deterministic exception, plus an updated happy-path test showing that normal disjoint evidence still writes `stage_block_predictions.json`.

### Milestone 5: Publish the contract in docs and artifacts

At the end of this milestone, architecture and staging docs will describe recipe ownership, divestment, and nonrecipe admission in the same way the code now works. The runtime will also write one small ownership artifact so a future reader can inspect what was owned, what was divested, and which blocks were available to nonrecipe.

The proof for this milestone is updated docs plus a new ownership artifact written under the stage output tree for synthetic tests.

## Validation and Acceptance

Acceptance is behavioral.

For ownership creation:

1. A synthetic boundary result with accepted recipe spans produces one ownership artifact and one runtime ownership contract with the expected owned and unowned block sets.
2. The ownership contract ties each owned block to exactly one recipe id.

For divestment:

1. Applying an explicit divestment removes the block from the owned set and adds it to the available-to-nonrecipe set.
2. No block becomes available to nonrecipe unless it was originally unowned or explicitly divested.
3. Recipe-refine cannot shrink ownership by changing recipe provenance alone.

For nonrecipe routing and finalize:

1. `build_nonrecipe_stage_result(...)` never routes an owned block.
2. Knowledge packet planning never includes an owned block.
3. Divested blocks become eligible for routing and knowledge packets without any ad hoc exception logic.

For stage-block predictions:

1. Recipe-local versus recipe-local conflicts still resolve by recipe-local priority.
2. Recipe-local versus `KNOWLEDGE` overlap raises an invariant violation and does not silently resolve to recipe-local.
3. Happy-path disjoint evidence still writes a normal `stage_block_predictions.json`.

For docs and artifacts:

1. The new ownership artifact is written in stage runs and is referenced in the docs.
2. The docs no longer describe recipe-local versus `KNOWLEDGE` overlap as an expected state where recipe-local wins.

## Idempotence and Recovery

This refactor is safe to apply in small passes if the ownership contract is introduced first and all downstream readers are migrated immediately after. Do not leave the repo in a mixed state where some code consults the ownership contract and other code still treats recipe provenance or raw `recipe_spans` as the real authority. That mixed state would reintroduce the drift this plan is trying to delete.

If a test fixture still assumes that `recipe_spans` are the only recipe exclusion input, update the fixture and helper builders to construct an ownership contract explicitly. Do not add adapters that preserve the old contract indefinitely.

If the first implementation needs a transitional assertion because recipe-refine does not yet emit any divestments, that is acceptable only if the assertion is strict and temporary. For example, the code may assert that `divested_block_indices` is empty for live recipe-refine paths until the producer exists. What is not acceptable is silently ignoring divestments or silently rebuilding ownership from recipe payload provenance.
