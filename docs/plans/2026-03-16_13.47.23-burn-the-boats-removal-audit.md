# Burn The Boats Removal Audit

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This document is a removal audit, not a migration proposal. It assumes `docs/reports/Refactor.md` is the architectural contract and that `docs/plans/phase1-observability-first.md`, `docs/plans/phase2-label-source-of-truth.md`, `docs/plans/phase3-collapse-recipe-reasoning.md`, and `docs/plans/phase4-shrink-knowledge-pipeline.md` describe what the runtime was supposed to become. The question here is narrower: after those phases landed, what old-world code still remains, and what should be deleted next if the project is intentionally burning the boats.

## Purpose / Big Picture

After this cleanup, a person reading the live runtime should no longer have to mentally separate “real current architecture” from “compatibility scaffolding left behind during the refactor.” The stage path, Label Studio prediction path, benchmark tooling, prompt tooling, and analytics readers should all teach one worldview: label-first, semantic-stage, single-correction, Stage 7 driven, current-contract only.

The visible result should be simple. New writes should stop publishing legacy alias artifacts and old pass-slot language. Hidden CLI and run-setting surfaces should stop defaulting to `legacy` values. Historical readers should only survive if you make a conscious decision that archived roots still matter enough to justify that code. If you do not make that decision, they should be deleted.

## Surprises & Discoveries

- Observation: the checked-in refactor plans are already far more destructive than the remaining code.
  Evidence: `docs/plans/phase1-observability-first.md` explicitly says there is no dual taxonomy period, `docs/plans/phase2-label-source-of-truth.md` says the finished state is label-first only, and `docs/plans/phase4-shrink-knowledge-pipeline.md` says Stage 7 should replace residue-first authority.

- Observation: the largest remaining old-world seam is not the main stage runtime anymore; it is the tooling around it.
  Evidence: `cookimport/staging/import_session.py` largely follows the new model, while `cookimport/llm/prompt_artifacts.py`, `cookimport/bench/eval_stage_blocks.py`, `cookimport/bench/followup_bundle.py`, and `cookimport/analytics/dashboard_collect.py` still carry significant compatibility logic.

- Observation: several places still use the word `legacy` for the current algorithm rather than an actual fallback branch.
  Evidence: `cookimport/bench/eval_canonical_text.py` enforces `"legacy"` alignment as the only safe scorer path while also carrying `auto` and deprecated `fast` modes.

- Observation: some compatibility code is already effectively dead.
  Evidence: `Bucket1FixedBehavior.legacy_key_patch()` in `cookimport/config/codex_decision.py` has no call sites, and `_run_all_method_config_once(...)` in `cookimport/cli.py` exists only as a wrapper around `_run_all_method_prediction_once(...)`.

- Observation: the repo already rejects some removed legacy values at validation time, but adjacent help text and defaults still teach the opposite story.
  Evidence: `cookimport/config/run_settings.py` and `cookimport/labelstudio/ingest.py` reject old recipe pipeline ids, while `cookimport/cli.py` still contains help text such as “Legacy saved values load as aliases” and hidden defaults like `multi_recipe_splitter="legacy"` and `p6_yield_mode="legacy_v1"`.

## Outcomes & Retrospective

The main outcome of this audit is clarity. The repo was not blocked by one giant undecidable “legacy cleanup.” It had a finite set of leftover seam clusters, and the highest-value ones were small enough to remove in targeted passes. The biggest lies are now gone from the live product path: hidden parser defaults no longer call the current algorithm `legacy`, prompt artifacts no longer serialize new runs through `task1/task4/task5`, benchmark eval no longer writes alias mismatch files, and the all-method benchmark no longer exposes the deleted per-source scheduler branch.

The broad shape of the cleanup is also now sharper. The live runtime and benchmark/prompt outputs are now current-contract only for new writes, while analytics/history support is a conscious exception rather than accidental carryover. The `scripts/benchmark_cutdown_for_external_ai.py` residue called out earlier was cleaned up afterward: prompt-derived cutdown/runtime rows now stay on semantic `stage_key` fields, and old slot-label values survive only as archived-log read compatibility. That note matters because it keeps the remaining exception boundary explicit: `.history/performance_history.csv` still stays out of scope for this purge.

## Context and Orientation

For this audit, “legacy code” means any branch, artifact name, enum value, helper, or read path whose main reason to exist is preserving the pre-refactor worldview. “Compatibility code” means bridges that keep old payload shapes, old CLI/config values, or old artifact names alive even though the new runtime already has a replacement.

The current live product path is centered on these modules:

- `cookimport/staging/import_session.py` for stage execution.
- `cookimport/parsing/label_source_of_truth.py` for authoritative Stage 2 labeling and the current label-first adapter seam.
- `cookimport/staging/nonrecipe_stage.py` and `cookimport/staging/writer.py` for Stage 7 and stage-owned artifacts.
- `cookimport/llm/codex_farm_orchestrator.py` for the single-correction recipe path.
- `cookimport/labelstudio/ingest.py` for stage-backed prediction/import flows.

The biggest remaining historical-read and compatibility surfaces live elsewhere:

- `cookimport/cli.py` for hidden options, old scheduler branches, wrapper functions, and artifact alias publishing.
- `cookimport/llm/prompt_artifacts.py` for prompt-stage discovery that still thinks in task slots.
- `cookimport/bench/eval_stage_blocks.py`, `cookimport/bench/eval_canonical_text.py`, `cookimport/bench/followup_bundle.py`, and `cookimport/bench/upload_bundle_v1_model.py` for benchmark and follow-up compatibility.
- `cookimport/analytics/dashboard_schema.py` and `cookimport/analytics/dashboard_collect.py` for historical metrics/CSV compatibility.

This audit uses the following removal buckets.

“Delete now” means the code is dead, misleading, or clearly below the bar set by the refactor docs, and the only meaningful work is removing the implementation plus the tests that assert it exists.

“Delete with one coordinated cut” means the seam is still wired through public or semi-public surfaces such as hidden CLI flags, benchmark packet fields, or script interfaces. It should still go, but in one deliberate pass rather than by nibbling at call sites.

“Keep only if you still value archived history” means the code is not part of the live product path. The only argument for keeping it is that old roots, CSVs, or follow-up bundles still matter enough to justify production complexity.

## Removal Inventory

### Delete Now: obvious dead code and low-value shims

`cookimport/config/codex_decision.py` defines `Bucket1FixedBehavior.legacy_key_patch()`, but the method has no call sites. This is a textbook dead compatibility helper. Delete it and any test or comment that suggests the repo still needs to synthesize old key patches.

`cookimport/cli.py::_run_all_method_config_once(...)` exists only as a wrapper around `_run_all_method_prediction_once(...)` and says so in its own comment. If there are still direct callers, retarget them and delete the wrapper. Keeping a compatibility name in the main CLI file only prolongs confusion.

`cookimport/cli.py::_build_single_offline_starter_pack(...)` and `cookimport/cli.py::_write_benchmark_upload_bundle(...)` still catch `TypeError` and retry older helper signatures. In a solo repo where the scripts live in the same tree, this is an unnecessary version-skew shim. Update the script entrypoints if needed and delete the fallback path.

`cookimport/config/run_settings.py` and `cookimport/cli.py` already reject removed recipe pipeline ids, but multiple help strings still claim “Legacy saved values load as aliases.” That text is now false. Delete the wording anywhere it still appears rather than teaching a migration behavior the code no longer implements.

### Delete With One Coordinated Cut: label-first bridge residue still called “compatibility”

`cookimport/parsing/label_source_of_truth.py` still exposes `LabelFirstCompatibilityResult` and `build_label_first_compatibility_result(...)`. The type carries the authoritative artifacts, but its name and one of its fields (`conversion_result`) announce that the old shape is still a supported backbone. The Phase 2 and Phase 4 plans explicitly say this bridge is temporary. The cleanup pass should rename the type to something authority-based, remove `conversion_result` as the primary handoff, and make downstream code consume the current fields directly.

The same module still rebuilds `ConversionResult.non_recipe_blocks` and `cookimport/staging/import_session.py` still assigns `result.non_recipe_blocks = stage7_block_rows` with a comment saying the legacy field is kept for compatibility/reporting. The refactor docs already say Stage 7 owns this lane. If you are truly done with the old shape, delete this compatibility cache and then update any remaining readers instead of repopulating it forever.

`cookimport/staging/writer.py::_normalize_draft_compatibility_aliases(...)` still back-fills top-level `name`, `ingredients`, and `instructions` from the structured recipe payload. If the new draft shape is the product truth, this function is one of the clearest “old output schema” bridges left in the runtime. Remove it together with any consumer or test that still depends on those duplicated top-level aliases.

### Delete With One Coordinated Cut: hidden run settings and CLI defaults still encode the old worldview

`cookimport/config/run_settings.py` still treats multiple live settings as `legacy` by default or as a first-class enum choice: `SectionDetectorBackend.legacy`, `MultiRecipeSplitter.legacy`, `IngredientPreNormalizeMode.legacy`, `IngredientUnitCanonicalizer.legacy`, and `P6YieldMode.legacy_v1`. That is a direct mismatch with the refactor story. For these, there are only two coherent outcomes. Either fix them to one chosen current contract and remove the old enum values entirely, or rename the chosen algorithm so the product stops calling itself legacy.

`cookimport/cli.py` and `cookimport/labelstudio/ingest.py` mirror the same problem by exposing hidden options with defaults like `--section-detector-backend legacy`, `--multi-recipe-splitter legacy`, `--ingredient-pre-normalize-mode legacy`, `--ingredient-unit-canonicalizer legacy`, and `--p6-yield-mode legacy_v1`. The live code is still teaching operators, tests, and artifacts that legacy is the normal state. This should be deleted in one pass alongside the `RunSettings` cleanup so there is one current contract, not a half-renamed one.

The all-method benchmark scheduler in `cookimport/cli.py` still supports `ALL_METHOD_SCHEDULER_SCOPE_LEGACY`, `_run_all_method_benchmark_multi_source_legacy(...)`, and the `discovery` source-order mode described as legacy FIFO behavior. If the global scheduler is the new architecture, remove the legacy scheduler scope, remove the discovery legacy language, collapse the dispatch to the global queue path, and delete the large test matrix that exists only to prove the old scheduler still works.

### Delete With One Coordinated Cut: prompt and bundle tooling still reconstructs work through pass-slot language

`cookimport/llm/prompt_artifacts.py` is the biggest remaining old-world surface. The public data model still uses `legacy_pass`, `stage_matches_legacy`, `_resolve_*_for_legacy_pass(...)`, and hard-coded `task1` / `task4` / `task5` logic. Phase 1 explicitly said the repo stops speaking in pass-slot language. This file is still translating semantic stages back into the old slots. It should be rebuilt around `stage_observability.json`, stage manifests, and semantic stage keys only.

`cookimport/bench/upload_bundle_v1_model.py::UploadBundleSourceModel.as_context_dict()` exists only as a “Compatibility view for legacy script internals.” That is acceptable only if you still intend to keep those script internals unchanged. If not, update `scripts/benchmark_cutdown_for_external_ai.py` to consume the typed model directly and delete the compatibility dictionary view.

`cookimport/bench/followup_bundle.py` still loads `legacy_followup_cases` and contains multiple branches that resolve selectors by `legacy_case(...)` or `explicit_legacy_case_id`. If archived follow-up bundles are not a product promise, this is removable. If they still matter, keep it in the script layer rather than in the shared product module layer.

### Delete With One Coordinated Cut: benchmark alias artifacts and old field names

`cookimport/bench/eval_stage_blocks.py` writes both the new mismatch artifacts (`missed_gold_blocks.jsonl`, `wrong_label_blocks.jsonl`) and the old alias files (`missed_gold_spans.jsonl`, `false_positive_preds.jsonl`). `cookimport/bench/eval_canonical_text.py` does the same. The CLI later publishes those alias names in run artifacts. If the benchmark contract is supposed to be current-only, delete the alias files, stop publishing them in manifests, and update any downstream consumer to use the canonical block-based names.

This same cleanup should include the report payload keys that still carry legacy naming solely for downstream compatibility. Otherwise the repo will keep forcing every new surface to speak two dialects.

### Delete With One Coordinated Cut: prompt and prelabel alias surfaces

`cookimport/llm/canonical_line_role_prompt.py` still exposes `LineRolePromptFormat = Literal["legacy", "compact_v1"]`, defaults the prompt builder to `"legacy"`, and keeps both serializations alive even though `cookimport/parsing/canonical_line_roles.py` now prefers `compact_v1` by environment default. If compact is the real format, remove the old format and make any environment override fail instead of silently falling back to `legacy`.

`cookimport/labelstudio/prelabel.py` still normalizes a wide alias family for prelabel granularity values such as `legacy`, `legacy_block`, and `actual_freeform`, and still injects the placeholder alias `{{BLOCKS_WITH_FOCUS_MARKERS_JSON_LINES}}` for template compatibility. If you no longer want template-level backwards compatibility, delete the old aliases and make repo-owned templates use only the current placeholder names and current granularity values.

### Keep Only If Archived History Still Matters: analytics and timestamp compatibility

`cookimport/analytics/dashboard_schema.py` still carries explicit legacy alias metrics like `precision`, `recall`, `f1`, and related historical fields in `BenchmarkRecord`. `cookimport/analytics/dashboard_collect.py` still falls back to old `.history/performance_history.csv` locations when the canonical CSV path is missing. `cookimport/analytics/CONVENTIONS.md` and related code still preserve the older timestamp directory style. None of this is part of the new runtime architecture. Keep it only if the old dashboard history is still valuable enough to justify the extra collector and schema complexity.

If you decide the answer is no, then the cleanup is straightforward: remove the old CSV path probes, remove the legacy timestamp support, drop the alias metrics from the model, and update the dashboard/tests so only current metric names and current run folder timestamps exist.

### Keep Only If Archived External Tooling Still Matters: runner and script compatibility

`cookimport/llm/codex_farm_runner.py` still parses legacy progress lines from older codex-farm stderr output. This is not a repo-internal architecture issue; it is external tool compatibility. If you control the CodexFarm version and are happy to require the current JSON progress format, delete the legacy parser and message formatter.

This same principle applies to any benchmark or follow-up helper that exists only to read old bundle shapes. If the archived bundle is not worth the complexity, delete the reader and regenerate any fixture you still care about.

## Suggested Execution Order

Start with the no-regret removals because they reduce noise immediately and shrink the blast radius of later passes. Delete unused helpers, helper-signature fallbacks, and obviously false compatibility wording first.

Then do the live-contract cut in one pass. That means the `LabelFirstCompatibilityResult` rename/removal, the `ConversionResult.non_recipe_blocks` compatibility cache deletion, the draft alias cleanup, and the hidden `legacy` run-setting defaults all go together. Those changes will touch both runtime code and tests, but they remove the most misleading seams still present in the product path.

After that, cut the tooling layer. Rebuild prompt artifact discovery around semantic stages, remove upload-bundle compatibility views, remove benchmark alias artifacts, and collapse the all-method scheduler to the global implementation. This is the largest test-refresh pass, but it is still mechanically straightforward once the runtime contract is clean.

Finally, make one explicit yes-or-no decision about historical support. If old analytics CSVs, old timestamp roots, old follow-up bundles, or old codex-farm stderr formats still matter, keep those readers and document that choice. If they do not matter, delete them aggressively rather than letting them linger as undefined “maybe useful later” baggage.

## Plan of Work

The cleanup should be implemented as a small sequence of destructive but coherent passes, not as one giant flag day. The repo is already heavily tested, so the safest path is to remove one seam family at a time and immediately refresh the tests that asserted it.

The first pass should be pure deletion of dead wrappers and false compatibility text. That keeps the diff small and clarifies what is actually still live.

The second pass should cut the remaining runtime bridge surface. Rename or replace `LabelFirstCompatibilityResult`, remove `conversion_result` and `non_recipe_blocks` as authority boundaries, and stop writing draft/output alias fields that only exist for older readers.

The third pass should normalize the hidden settings surface. Pick the one current algorithm for section detection, multi-recipe splitting, ingredient pre-normalization, unit canonicalization, and yield extraction. Then make the setting names and defaults reflect that choice, rather than preserving `legacy` labels in perpetuity.

The fourth pass should remove prompt, benchmark, and scheduler compatibility surfaces. This will be test-heavy because the current suite still has many assertions around the legacy scheduler, legacy stage naming, and alias artifact files.

The fifth pass should either delete or consciously retain history readers. This is the only place where “keep it” may still be reasonable, but only if the project genuinely still benefits from reading old roots.

## Validation and Acceptance

This cleanup is accepted when the live product no longer depends on old-world naming or bridge surfaces.

The runtime portion is complete when stage execution, Label Studio prediction generation, and final draft writing do not rely on types or fields named as compatibility bridges, and when the main hidden run-setting surface stops defaulting to `legacy` values.

The tooling portion is complete when prompt artifact discovery, upload-bundle rendering, benchmark eval outputs, and all-method scheduling no longer publish or prefer legacy pass-slot language or alias artifact names for new runs.

The history portion is complete when you have made an explicit choice. Either current-only readers remain and old roots are intentionally unsupported, or the remaining historical readers are documented as a conscious exception instead of accidental residue.

The strongest final proof is behavioral. A fresh stage run, a fresh benchmark run, and a prompt-artifact export should be inspectable without encountering legacy pipeline ids, pass-slot names, alias mismatch files, or settings whose official value is literally called `legacy`.

## Idempotence and Recovery

This plan is safe to execute in repeated passes because each seam family is independently searchable and testable. Use the repo search plus focused domain tests after every pass. If a pass turns out to have removed a reader you still care about, restore only that narrow support and document why it is intentionally retained.

Do not mix historical-support decisions into the early runtime cleanup unless the code is directly shared. Keeping those decisions separate makes it possible to finish the live-product purge even if you have not yet decided what to do with archived roots.
