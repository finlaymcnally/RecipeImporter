## Purpose / Big Picture

After this change, the repository will support the full practical workflow suggested in the Oracle parsing-accuracy report, not just the earlier runtime optimizations. A maintainer will be able to run Phase A quickly, invoke a first-class auto handoff into Phase B without manual winner copying, run an optional B+ sweeps-decision pass from a ready preset, and rely on explicit promotion heuristics and stable phase defaults. Seed selection will also be less brittle by allowing explicit seed lists together with `--max-seeds` instead of hard-failing.

## Surprises & Discoveries

- Observation: The tournament already computes and persists `top_tier_candidates`, but there is no CLI surface that consumes prior summary artifacts to auto-select Phase B candidate IDs.
  Evidence: `scripts/quality_top_tier_tournament.py` summary generation and current args surface.
- Observation: Current seed resolution rejects explicit seeds with `--max-seeds`, which blocks the Oracle-style pattern of explicit seed choice plus cap.
  Evidence: `_resolve_seed_plan(...)` raises on explicit+max-seeds combination.
- Observation: Phase defaults are split between threshold artifacts and ad hoc CLI flags; there is no phase-specific default for max parallel experiments.
  Evidence: threshold files include search/sweeps knobs, but `max_parallel_experiments` is only a CLI arg.
- Observation: The heuristic should be shared for both auto-handoff and tournament output recommendation metadata to avoid drift between selection behavior and report guidance.
  Evidence: `_recommend_phase_b_candidates(...)` now feeds both auto selection and `phase_a_promotion_recommendation`.
- Observation: Using dry-run still writes resolved/report artifacts, which is useful for validating heuristic metadata and precedence behavior without running heavy quality jobs.
  Evidence: dry-run commands produced `tournament_resolved.json`, `summary.json`, and `report.md` under `data/golden/bench/quality/tournaments/<timestamp>/`.

## Decision Log

- Decision: Keep `scripts/quality_top_tier_tournament.py` as the single orchestration entrypoint and add new flags/metadata there instead of creating a separate command.
  Rationale: This preserves existing run/resume behavior and keeps operator workflow in one place.
  Date/Author: 2026-03-01 / Codex
- Decision: Introduce a phase-aware auto-candidate selector that can read prior `summary.json` (+ optional `folds.json`) and resolve one or two Phase B candidates using Oracle heuristics.
  Rationale: Oracle handoff was shell-snippet-only; a first-class flag must live in script logic, not runbook prose.
  Date/Author: 2026-03-01 / Codex
- Decision: Resolve explicit seed + `--max-seeds` precedence by applying cap after explicit dedupe, preserving explicit order.
  Rationale: This aligns with Oracle operating guidance while remaining deterministic and easy to reason about.
  Date/Author: 2026-03-01 / Codex
- Decision: Phase defaults should come from thresholds with CLI override, including `max_parallel_experiments`.
  Rationale: Keeps profile behavior reproducible and discoverable in versioned artifacts.
  Date/Author: 2026-03-01 / Codex
- Decision: Keep auto-candidate behavior non-destructive by letting explicit `--candidate-experiment-id` silently override/ignore auto flags.
  Rationale: Operators can always force deterministic manual candidate scope without removing auto args from saved command snippets.
  Date/Author: 2026-03-01 / Codex
- Decision: Persist candidate-selection provenance in both `tournament_resolved.json` and final `summary.json`.
  Rationale: Resume/debug/review workflows need to know whether candidate scope came from thresholds, explicit CLI, quick parsing defaults, or prior-summary auto handoff.
  Date/Author: 2026-03-01 / Codex

- "Phase B" means confidence A/B (baseline + winner, conservative gates).

- "Auto handoff" means selecting Phase B candidate IDs directly from prior Phase A artifacts via a script flag, without manual copy/paste.

## Plan of Work

### Milestone 1: First-class auto handoff surface

Add a new tournament CLI surface for automatic candidate selection from a prior Phase A result.

Implementation in `scripts/quality_top_tier_tournament.py`:

- Add flag(s) that identify auto-source summary input, for example:
  - `--auto-candidates-from-summary <path>` where `<path>` accepts `summary.json` or a tournament directory.
  - Optional convenience mode `--auto-candidates-from-latest-in <tournament_root_dir>`.
- Enforce mutual exclusivity/precedence with explicit `--candidate-experiment-id`:
  - explicit IDs win when provided;
  - auto mode only runs when explicit IDs are absent.
- Resolve selected IDs before candidate filtering and persist resolution metadata under `tournament_resolved.json` (`candidate_source`, selected IDs, source path, heuristic path taken).

Behavior contract:

- The command can run Phase B with no manual winner ID transcription.
- The selected IDs are deterministic and visible in resolved artifacts.

### Milestone 2: Encode Oracle A->B heuristic in code

Implement the Oracle heuristic as structured selector logic used by auto handoff:

- Prefer one candidate when a clear winner is detected ("winner or tied-top on both unique folds").
- Select two candidates when top two are close (`<= 0.003` practical delta threshold).
- If evidence is sparse/noisy, fall back to deterministic top row with an explicit warning/metadata reason.

Implementation details:

- Parse prior summary rows and, when available, fold-level comparisons from `folds.json` to detect fold winner consistency.
- Add stable helper function(s), e.g. `_select_phase_b_candidates_from_phase_a(...)`.
- Persist recommendation block in output summary/report:
  - `phase_a_promotion_recommendation` with chosen candidates, heuristic inputs, threshold values, and rationale.

### Milestone 3: B+ sweeps-decision preset + docs

Codify the optional B+ workflow as versioned artifacts and runbook commands.

Artifacts:

- Add a new thresholds profile under `data/golden/bench/quality/thresholds/` with timestamped name:
  - search strategy `race`
  - deterministic sweeps enabled
  - seeds list configured for normal suite, but command examples cap to 2 seeds
  - baseline-compatible conservative gates for decision reporting.

Docs:

- Add explicit "Phase B+ (optional sweeps decision)" section to `docs/07-bench/runbook.md` with copy/paste command.
- Add corresponding contract note to `docs/07-bench/07-bench_README.md` describing when to use B+ and when not to.

### Milestone 4: Phase defaults include `max_parallel_experiments=4`

Make phase runs default to `4` workers when operator does not pass CLI override.

Implementation in `scripts/quality_top_tier_tournament.py`:

- Extend thresholds quality-run schema handling with optional `max_parallel_experiments_default`.
- Effective value resolution:
  - if CLI `--max-parallel-experiments` is provided: use CLI.
  - else if thresholds define default: use threshold value.
  - else fallback to current quality-run auto mode behavior.
- Reflect resolved value in `tournament_resolved.json` and rendered command logs.

Profile updates:

- Set `max_parallel_experiments_default: 4` in Phase A/Phase B (and B+ where appropriate).

### Milestone 5: Seed precedence relaxation

Update seed resolution logic to allow explicit seeds with `--max-seeds`.

Implementation in `_resolve_seed_plan(...)`:

- Remove hard error for explicit seed inputs + `--max-seeds`.
- New precedence:
  - Build explicit deduped sequence first (`--seed` then `--seed-list` order).
  - Apply `--max-seeds` cap to explicit sequence when provided.
  - If no explicit seeds, current threshold-seed path remains.
- Preserve deterministic metadata in `seed_resolution` block, including whether cap was applied to explicit or threshold source.

Behavior contract:

- Oracle-style explicit seed selection with optional cap works.
- Empty explicit list after parsing/cap still fails with clear message.

### Milestone 6: Tests, docs, and verification

Tests:

  - clear-winner one-candidate selection;
  - explicit seed + `--max-seeds` precedence behavior;
  - thresholds-based default max parallel forwarding.
- Keep existing race/prediction-reuse tests unchanged unless contract shifts require updates.

## Idempotence and Recovery

All planned changes are additive and retry-safe.

- New CLI flags are optional and backward-compatible.
- Existing manual candidate flow remains valid.
- Threshold/profile JSON files are static artifacts; reruns produce new timestamped outputs unless explicit resume paths are used.
- If heuristic selection causes undesired candidate choice, operators can always override with explicit `--candidate-experiment-id`.
- If any new default creates performance regressions, temporary recovery is to pass explicit CLI overrides (`--max-parallel-experiments`, `--candidate-experiment-id`, `--seed`/`--seed-list`).
