# Tighten line-role runtime spend and telemetry after the March 22 runtime changes

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

The latest line-role run is not failing the way knowledge is failing, but it is still leaving useful money on the table and it is no longer reporting real token usage for the stage. In the March 22, 2026 `16:52:22` Salt Fat CodexFarm run, line-role completed all five worker assignments with zero invalid-output shards and only `65` commands, but `token_usage_status` was `unavailable` and the worker `usage.json` files were empty. The command traces also show that workers still spend most of their readback budget on large `current_task.json` and `in/*.json` dumps even though the repo already prewrites deterministic `scratch/*.json` drafts.

After this change, a human should be able to run the same benchmark family and see line-role keep its current quality and completion behavior while reporting real stage token usage and using smaller default startup/readback surfaces. The proof is that line-role prompt-budget summaries stop blanking tokens, `usage.json` is populated on the main path, and the ordinary worker trace relies more on compact repo-written task surfaces plus prewritten scratch drafts and less on broad raw packet rereads.

## Context and Orientation

The line-role stage labels canonical atomic lines into labels such as `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `KNOWLEDGE`, and `OTHER`. The parser-owned quality logic lives in `cookimport/parsing/canonical_line_roles.py`, but the runtime transport for the Codex-backed worker path also lives there. Each worker root under `line-role-pipeline/runtime/line_role/workers/worker-*/` currently contains `assigned_tasks.json`, `assigned_shards.json`, `current_task.json`, `hints/`, `in/`, `debug/`, `scratch/`, `out/`, `tools/line_role_worker.py`, and `worker_manifest.json`.

- `cookimport/parsing/canonical_line_roles.py`, which writes worker prompts, task rows, worker roots, and line-role telemetry summaries.
- `cookimport/parsing/line_role_workspace_tools.py`, which defines the helper script contract, draft scaffolds, validation, and finalize paths.

- `cookimport/llm/prompt_budget.py`, `cookimport/analytics/perf_report.py`, and `cookimport/analytics/dashboard_collect.py`, which already interpret missing line-role usage correctly and should stay fail-closed while runtime capture is repaired.
- `tests/parsing/test_canonical_line_roles.py`, `tests/labelstudio/test_labelstudio_benchmark_helpers_artifacts.py`, and `tests/analytics/test_stats_dashboard.py`, which are the narrowest local seams for runtime telemetry and artifact interpretation.

## Milestones

### Milestone 1: Make main-path line-role usage measurable again

At the end of this milestone, line-role workspace-worker runs should populate real usage payloads whenever the underlying direct-exec session provides them. The existing fail-closed blanking logic should remain intact for genuinely missing usage, but the normal path should no longer produce empty `usage.json` files and `token_usage_status: unavailable`. The proof is local in worker artifact tests and prompt-budget / analytics tests that see complete token usage again when usage exists.

### Milestone 2: Make the cheap line-role control surfaces actually cheap

At the end of this milestone, `assigned_tasks.json` and `current_task.json` should no longer duplicate the full input packet payload. They should be metadata-first control files that point at `in/<task_id>.json`, `hints/<task_id>.md`, and `scratch/<task_id>.json`. If a human or worker needs the full packet, it should still exist under `in/`, but the ordinary startup path should not require reading or writing the expensive duplicate first.

### Milestone 3: Make the default worker loop trust the prewritten draft path

At the end of this milestone, the line-role prompt and helper surfaces should make the common case obvious: inspect the compact current-task surface, inspect the prewritten scratch draft, use the hint only when the deterministic seed seems questionable, and open the full `in/*.json` packet only when the earlier surfaces are insufficient. The proof is in worker prompt/helper tests and, with user approval, a benchmark rerun showing fewer broad `current_task.json` and `in/*.json` reads.

## Plan of Work

Start in `cookimport/llm/codex_exec_runner.py` and the line-role runtime handoff in `cookimport/parsing/canonical_line_roles.py`. The latest run proves that the fail-closed readers are behaving correctly, but the main path is not preserving actual usage. Trace how workspace-worker usage is harvested into `telemetry.rows`, `workers/*/usage.json`, and the line-role summary writer. Repair that path so successful main-worker runs capture real usage when available, while keeping the existing blank-on-missing behavior in the summary readers.

Then shrink the task-control surfaces in `cookimport/parsing/canonical_line_roles.py`. The runtime currently writes full task rows into `assigned_tasks.json`, and the helper treats `current_task.json` as a full task row too. Change that contract so those files become metadata-first: task id, parent shard id, named local paths, owned row counts, atomic span, deterministic label counts, and scratch/result paths. Do not embed the full `input_payload` there. The immutable packet payload already lives in `in/<task_id>.json`, and the deterministic scaffold already lives in `scratch/<task_id>.json`.

Update `cookimport/parsing/line_role_workspace_tools.py` in lockstep. The helper should read payloads from the named `metadata.input_path` file when it actually needs them, not from an embedded duplicate inside the task row. Add a lighter `CURRENT_TASK.md` and, if needed, a short `CURRENT_TASK_FEEDBACK.md` so the current task can have a truly slim worker-facing first-read surface rather than overloading `current_task.json` again.

Then tighten the worker prompt in `cookimport/parsing/canonical_line_roles.py`. Keep the existing quality posture and the prewritten-draft paved road, but reorder the startup contract so it no longer frames `current_task.json` and `assigned_tasks.json` as the cheapest default surfaces when those files are still large. The prompt should say, in effect: open the compact current-task surface, inspect the prewritten scratch draft, use `hints/<task_id>.md` for the targeted explanation, and only open `in/<task_id>.json` when the deterministic seed looks wrong or the hint is insufficient.

If the latest worker behavior still shows broad raw-input rereads after the task-control surfaces are shrunk, enrich the hint or current-task brief just enough to cover the usual correction cases. Do not stuff the whole packet back into the control files. The goal is not to eliminate raw-input access entirely; the goal is to make it fallback rather than reflex.

Finish by updating the short runtime notes in `cookimport/llm/README.md` and `docs/10-llm/10-llm_README.md`. Keep them short and operational. They should say that line-role usage reporting stays fail-closed when missing, but the normal path should now emit complete usage again, and that the happy path starts from smaller current-task metadata plus prewritten scratch drafts rather than from broad packet dumps.
