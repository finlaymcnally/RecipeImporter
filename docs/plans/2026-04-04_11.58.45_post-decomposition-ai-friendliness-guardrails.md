## Purpose / Big Picture

After this change, a cold-start coding agent should have four concrete advantages that are still worth paying for after the decomposition work is done. First, the highest-value architectural boundaries will be enforced by tests instead of existing only in docs and habit. Second, the run-settings surface that operators and agents actually choose from will be smaller and easier to reason about, while the full raw settings model remains available for manifests, benchmarks, and archaeology. Third, the few remaining "where do I start, what doc do I read, and what tests do I run?" hotspots will be called out directly in existing owner README files. Fourth, the most expensive leftover compatibility patch surfaces will be reduced where tests prove they are no longer needed.

This plan is intentionally narrow. Its goal is not "make the repo even cleaner." Its goal is to finish the small amount of post-decomposition work that still improves discovery, safe local change, and verification for AI coders. The proof is that architecture guardrails become stricter in the right places, run-settings tests still pass with a smaller operator-facing surface, hotspot owner README files point agents at the right docs and test commands, and the most patch-heavy public roots stay behaviorally stable while losing avoidable compatibility baggage.

At the end of this plan, generic repo-wide AI-friendliness sweeps should stop. Future structural work should happen only when a real recurring pain point appears in implementation or review.

Several terms matter here.

A "boundary guardrail" is an automated test that rejects a class of architecture drift. In this repository that means things like "this root stays a thin facade," "that package must not wildcard-import its support package," or "outside callers may only use the public seam, not the internal helper owner." The main existing guardrail file is [tests/architecture/test_ai_readiness_boundaries.py](/home/mcnal/projects/recipeimport/tests/architecture/test_ai_readiness_boundaries.py).

The "operator surface" is the subset of run settings that a human or AI operator should reasonably choose from run to run. It is not the same thing as the full persisted settings model. In this repository, the canonical model lives in [cookimport/config/run_settings.py](/home/mcnal/projects/recipeimport/cookimport/config/run_settings.py), the contract projections live in [cookimport/config/run_settings_contracts.py](/home/mcnal/projects/recipeimport/cookimport/config/run_settings_contracts.py), the interactive UI metadata lives in [cookimport/config/run_settings_ui.py](/home/mcnal/projects/recipeimport/cookimport/config/run_settings_ui.py), and adapter logic lives in [cookimport/config/run_settings_adapters.py](/home/mcnal/projects/recipeimport/cookimport/config/run_settings_adapters.py).

A "change map" is a very short owner note, placed in an existing subsystem `README.md`, that answers three cold-start questions directly: where to edit, what docs to read first, and what targeted tests to run before calling the change done. This plan does not create a new doc system. It adds a few compact sections to existing README files only where they will save real search time.

A "compatibility patch surface" is a public module-level symbol or broad patch point kept alive mostly because tests or callers monkeypatch it. Some of those surfaces are still worth keeping. Some are only historical leftovers. This plan cleans up only the latter group.

## Plan of Work

Begin with the boundary guardrails. The repository already has one architecture test file that encodes several AI-readiness constraints. Extend that pattern, but keep it narrow and explicit. Do not add a generic dependency-analysis framework. Add only the checks that protect the post-decomposition public seams that are easy for cold-start agents to accidentally bypass. The most useful new checks are allowlist-style import guards for the helper-owner families introduced by the decomposition work. The important behavior is: code outside an owning subsystem should import the public facade or composition root, not reach directly into the helper-owner modules unless the helper module is intentionally public and documented as such. If a guard needs exceptions for tests, allow them explicitly and name those exceptions in the test so the rule stays readable.

Then tighten the run-settings surface. The existing repository already separates the canonical settings model from the contract projections and UI metadata. Use that separation. Keep the raw/full persisted contract intact, but reduce what counts as a normal operator choice. Start by reviewing the operator-facing projections in `run_settings_contracts.py` and the interactive metadata in `run_settings_ui.py`. Move frozen or lab-only knobs out of the everyday operator surface and into either fixed behavior or explicit benchmark/internal projections. Do not break old manifests or analytics consumers that rely on the full setting picture. The intended result is a smaller operator-facing surface, not the deletion of valuable provenance. Update the config README so a novice can tell which files own the model, the projections, the UI visibility rules, and the runtime adapters.

Next, add hotspot change maps only where the repo still has genuine cold-start friction. Use the existing subsystem README files. Add a very short section near the top of each relevant README that says, in plain language, which file or module is the main entrypoint for edits in that subsystem, which docs should be read before editing, and which narrow tests should be run before calling the work complete. Keep these sections compact. If a README already effectively answers those questions, do not expand it. The main hotspots worth this treatment are config, CLI support and bench helpers, LLM runtime, line-role parsing, and the external-AI script/package boundary.

Finally, do a selective compatibility cleanup pass. Do not treat monkeypatching as inherently bad. Instead, search for the broadest surviving public patch surfaces that still exist only because old tests or historical habits depend on them. For each candidate, decide whether it still earns its keep. If yes, keep it and document it as an intentional hook. If not, narrow it. The best outcome is that the most common tests patch smaller explicit hooks or owner modules instead of giant public roots, while the public roots remain readable and stable. This work should stay opportunistic. Stop as soon as the highest-signal surfaces are cleaned up.

## Milestone 4: Remove The Highest-Signal Leftover Compatibility Patch Surfaces

This milestone pays down only the compatibility surfaces that are still expensive after the decomposition work and the new boundary rules. At the end of it, the heaviest remaining monkeypatch-driven tests will either patch smaller explicit hook points or continue to patch broad public roots only where that is clearly still the lowest-cost stable option.

Begin by inventorying the highest-density monkeypatch areas, especially the benchmark and Label Studio helper tests. For each broad patch target on a public root, ask whether the patch exists because the root is still hiding too much behind itself or whether it is an intentionally stable seam. Keep the intentionally stable seams. Narrow the accidental ones. Use focused helper or hook exports only when they make the ownership story clearer and the tests simpler.

Acceptance for this milestone is that the targeted monkeypatch-heavy tests still pass and that the public roots involved now have fewer surprise patch points. This is not a purity exercise. Stop after the biggest wins.

## Idempotence and Recovery

This work is designed to be safe to repeat. The architecture test edits, README updates, and contract-visibility changes are all additive and can be rerun or refined without damaging user data or run artifacts.

The one place to be careful is the run-settings surface. If a visibility change accidentally removes a setting from a place that still depends on it, recover by restoring the raw/full projection first, then narrowing the operator view again in smaller steps. Do not "fix" a failing run-settings surface change by deleting old settings from the persisted model. Preserve deserialization and recorded provenance unless there is a separate, explicit migration plan.

If a compatibility cleanup breaks monkeypatch-heavy tests, prefer restoring the explicit public hook or facade and documenting it instead of forcing tests to patch deep internals blindly. The goal is smaller, clearer patch surfaces, not zero patchability.
