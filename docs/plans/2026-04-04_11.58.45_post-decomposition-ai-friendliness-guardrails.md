## Purpose / Big Picture

After this plan is finished, a cold-start coding agent should get three practical benefits that still matter. First, the current architecture boundaries should fail fast when an owner root regrows or starts reaching across the intended seam. Second, the hotspot subsystems where agents still lose time deciding where to edit, what docs to read, and which tests to run should say that directly in the local README. Third, the broadest remaining monkeypatch seams should either become smaller explicit hook points or be documented as intentional stable seams.

This plan is intentionally narrow. The run-settings split, contract projection work, and decomposition work already happened. The remaining question is not "how do we make the repo more elegant?" The remaining question is "what small guardrails are still worth adding so agents can change the repo safely without another large structural rewrite?" The proof is that the architecture guardrail suite is back to green, the hotspot README files give direct next-step guidance, and the highest-signal patch-heavy tests either patch smaller named seams or document why the broader seam is still the correct one.

At the end of this plan, there should not be another generic AI-friendliness pass. Future structural cleanup should happen only when a real recurring implementation pain appears.

## Context and Orientation

The decomposition campaign is already over for the purpose of this plan. The current repo state already includes the owner-package splits and coordinator-root tightening recorded in `decomp1` through `decomp4`. A future contributor executing this plan should not spend time asking whether the old decomposition work still needs to happen. It does not.

The relevant current files are:

- `tests/architecture/test_ai_readiness_boundaries.py`, which is the code-enforced architecture guardrail suite for the post-decomposition repo shape.
- `cookimport/config/run_settings.py` plus `cookimport/config/run_settings_types.py`, `cookimport/config/run_settings_contracts.py`, `cookimport/config/run_settings_ui.py`, and `cookimport/config/run_settings_builders.py`, which together define the current run-settings ownership split.
- `cookimport/config/README.md`, `cookimport/cli_support/README.md`, `cookimport/bench/README.md`, `cookimport/llm/README.md`, `cookimport/parsing/canonical_line_roles/README.md`, and `scripts/README.md`, which are the likely hotspot README files where short change maps may save agents real search time.
- The benchmark and Label Studio test suites under `tests/bench/` and `tests/labelstudio/`, where the densest remaining monkeypatch seams currently live.

The term "change map" in this plan means a very short README section that answers three practical questions directly: where to edit for a certain class of change, which deeper doc to read first, and which focused test command to run before touching broader suites. It does not mean a new doc taxonomy, a generated map, or a repo-wide template.

The term "compatibility seam" means an intentionally stable public function, wrapper, or module-level export that tests and callers are allowed to patch or import. Some compatibility seams are good because they keep broad tests simple or preserve a stable root entrypoint. The problem to solve here is accidental broad patching where a root exposes too much by default, not deliberate seam design.

## Plan of Work

Milestone 1 restores the architecture guardrail suite to a green, believable state. Start by reading `tests/architecture/test_ai_readiness_boundaries.py` and identifying exactly why the current suite fails. The known current failure is the `run_settings.py` size threshold. Resolve that failure honestly. If the root file has regrown because durable helper or projection logic slipped back into it, extract or relocate the excess ownership and keep the threshold. If the file is larger only because the canonical schema legitimately gained stable settings while the ownership split still holds, raise the threshold narrowly and document why the higher limit is now the correct guardrail. Do not "fix" the suite by weakening unrelated assertions or by deleting the test.

Milestone 2 adds short change maps only where they buy back real discovery time. Re-read the hotspot README files and choose the small subset that still leaves a cold-start agent asking obvious next-step questions. Add a short section that names the typical edit entrypoint, the deeper doc to read first, and the narrowest relevant proof command. Keep these additions brief enough that the README remains useful as an orientation note, not a second architecture manual.

Milestone 3 inventories the broadest remaining monkeypatch-heavy tests and narrows only the highest-signal accidental seams. Start from the current patch clusters in Label Studio benchmark helpers, Oracle upload background tests, speed-suite tests, and any benchmark wrapper roots that are still patched in bulk. For each cluster, decide whether the broad seam is intentional. If it is intentional, document that in the owning README or leave it alone. If it is accidental, introduce a smaller explicit hook or helper export and update the tests to patch that seam instead.

Milestone 4 is the doc and closure pass. Re-run `npm run docs:list`, update this plan's living sections with the actual implementation outcomes, and make sure the touched README files and any affected doc references describe the post-decomp reality instead of the transitional state. The end state should be that future contributors no longer need another generic "AI-friendliness" cleanup proposal.

## Idempotence and Recovery

This work is safe to repeat because it is built around small guardrail updates, README edits, and selective seam tightening. Re-running the plan should begin by refreshing the current guardrail failure and current monkeypatch hotspots, then updating the living sections before continuing.

The only place to be careful is the architecture threshold adjustment. If the suite currently fails because a root regrew improperly, fix the ownership drift instead of papering over it with a larger limit. If a slightly larger threshold is the honest choice, document the evidence and keep the new threshold narrow.

If a compatibility cleanup breaks patch-heavy tests, restore the explicit seam and document it rather than forcing tests to patch deeper internals blindly. The goal is smaller and clearer patch surfaces, not zero patchability.

For compatibility cleanup, prefer explicit named hooks or helper exports over broad root-namespace patching when that produces a clearer ownership story. When a broad seam is still the right choice, document that fact briefly so future contributors do not reopen the same cleanup question.
