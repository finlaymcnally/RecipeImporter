# Make CodexFarm Failures Observable and Enforce Structural Success

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan is one focused split of `docs/plans/2026-03-13_22.18.12-codexfarm-unified-reliability-plan.md`, but it is written to stand on its own for a novice contributor.

## Purpose / Big Picture

After this change, a contributor should be able to open a bad CodexFarm benchmark row and explain, with named machine-readable reasons, whether the failure started in span selection, pass1-to-pass2 transport, contract loading, runtime mode, or model reasoning. The user-visible proof is that reruns stop reporting obviously broken rows as `ok` just because the JSON parsed, and the artifacts make it clear why a row was degraded, retried, skipped, or failed.

This plan intentionally does not try to improve recipe quality directly. It creates the observability and invariant layer that the other reliability plans should target.

## Surprises & Discoveries

- Observation: the March 13 reports agree that many consequential failures are not low-confidence mistakes; they are high-confidence structural mistakes.
  Evidence: the report set repeatedly calls out empty mappings, placeholder titles, broken transport, and partial bundles that still flowed through later stages as apparent success.

- Observation: the repository already has part of the required evidence surface.
  Evidence: `docs/10-llm/10-llm_README.md` describes `transport_audit/*.json`, pass1 span-loss metrics, pass2 degradation severity, and pass3 routing reasons, so this plan should extend those artifacts rather than invent a separate reporting channel.

- Observation: prompt logs showing docs-list or tool-seeking behavior during recipe work are runtime defects, not recipe-quality defects.
  Evidence: the March 13 reports treat agentic/tool-seeking traces during extraction or repair calls as contamination of the execution mode.

## Context and Orientation

CodexFarm recipe handling currently crosses three relevant code surfaces.

The first surface is transport. `cookimport/llm/codex_farm_transport.py` turns pass1 span decisions into the authoritative pass2 payload. In this repository, a transport verifier means code that proves the chosen source span, the rows copied into the payload, and the downstream extractive fields remain consistent after normalization. This is the right place to detect dropped rows, inclusive-versus-exclusive span mistakes, and mismatched source hashes.

The second surface is orchestration. `cookimport/llm/codex_farm_orchestrator.py` decides whether rows continue, degrade, route to live pass3, route to deterministic handling, or fail. In this repository, structural success means more than “the JSON parsed.” A row is structurally successful only when it contains the minimum believable recipe information the next stage needs and does not violate obvious invariants, such as a placeholder title, an empty mapping on a row with clearly mappable steps, or extractive rows that no longer match the selected source span.

The third surface is contract loading and execution mode. `cookimport/llm/codex_farm_contracts.py` parses pass2 and pass3 outputs, and the Codex execution wrapper documented in `docs/10-llm/10-llm_README.md` controls the request mode for extraction and repair calls. These calls must be plain structured-output recipe calls, not coding-agent calls with tool-seeking behavior.

The benchmark anchor for this plan is the March 6 root at `data/golden/benchmark-vs-golden/2026-03-06_15.22.11`. Use `saltfatacidheatcutdown` as the first validation book because the March 13 plans and reports already identify concrete transport and structural-failure examples there.

## Plan of Work

Start in `cookimport/llm/codex_farm_transport.py`. Make the transport verifier explicit and reusable. It must record, at minimum, span start and end semantics, normalized source-row count, payload row count, source hash or equivalent stable identity, and named mismatch reasons when copied rows do not line up with the authoritative source span. If span endpoints are converted between inclusive and half-open conventions anywhere in the path, make that conversion happen once and name it clearly.

Then edit `cookimport/llm/codex_farm_contracts.py` and the post-load classification logic in `cookimport/llm/codex_farm_orchestrator.py`. Define structural-success and structural-failure rules in one place and have the orchestrator consume them. A parsed payload with a placeholder title, an empty mapping without a declared reason, or extractive text that fails transport verification must not continue as quiet `ok`. Name the reasons and keep them stable because other plans will build tests and acceptance criteria around those names.

Next, inspect the Codex execution wrapper path described in `docs/10-llm/10-llm_README.md`. Confirm that pass1, pass2, and pass3 recipe calls run as non-agentic structured-output requests with no tool affordances and no shell-style behavior. If the wrapper currently makes that hard to assert directly, add a narrow configuration surface or diagnostic artifact that lets tests and logs prove the mode is correct.

Finally, update `docs/10-llm/10-llm_README.md` and, if benchmark-visible fields change, `docs/07-bench/07-bench_README.md`. Keep those docs short. The point is to document new reason codes, runtime expectations, and artifact fields so later contributors can interpret benchmark rows without re-reading this entire plan.

## Validation and Acceptance

Acceptance is behavior-first.

First, focused tests must prove that transport mismatches, span-geometry failures, malformed auxiliary metadata, placeholder-title outputs, empty-mapping outputs, and runtime-mode defects are classified with stable named reasons instead of being folded into generic success or generic fallback.

Second, the Salt, Fat, Acid, Heat rerun must produce benchmark artifacts where at least one previously ambiguous failure is now clearly attributable to transport, structural contradiction, or runtime mode rather than “the model was bad.” If the rerun still contains a bad row, the point of this plan is that the failure should become easier to explain.

Third, extraction and repair calls must show evidence of non-agentic execution. A passing test or audit artifact is sufficient; anecdotal prompt inspection is not.

## Idempotence and Recovery

This plan is additive. It should be safe to rerun the focused test slice and one-book benchmark repeatedly while reason-code names remain stable. If a new invariant is too strict and causes wide fallout, relax the classification to a warning or degraded state rather than deleting the diagnostic entirely. Do not overwrite the March 6 baseline root. New benchmark runs must use fresh timestamped output roots.
