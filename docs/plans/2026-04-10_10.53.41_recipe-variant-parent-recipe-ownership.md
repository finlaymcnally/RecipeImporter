# Keep `RECIPE_VARIANT` Owned By Its Parent Recipe

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a `Variations` section and its alternate-version lines stay attached to the recipe they modify instead of being pushed into the nonrecipe lane. The visible outcome is that label-first recipe ownership keeps those blocks inside the recipe span, benchmark scoring can still see `RECIPE_VARIANT`, and downstream recipe payload construction carries the variant text with the recipe instead of dropping it or reclassifying it as knowledge.

## Surprises & Discoveries

- Observation: The line-role stage is already doing the semantic recognition correctly for the benchmark canary; the ownership loss happens later.
  Evidence: The benchmark artifact `line-role-pipeline/authoritative_labeled_lines.jsonl` labels blocks such as `6934-6938`, `6994-6998`, `7017-7019`, and `7088-7091` as `RECIPE_VARIANT`, while `08_nonrecipe_route.json` later shows those same blocks as nonrecipe candidate spans.

- Observation: Fixing span ownership alone is not enough for downstream recipe payloads because `build_conversion_result_from_label_spans(...)` currently only copies `HOWTO_SECTION` and `INSTRUCTION_LINE` into `recipeInstructions`.
  Evidence: `cookimport/parsing/label_source_of_truth.py` builds `recipeInstructions` from `HOWTO_SECTION` plus `INSTRUCTION_LINE` only, even though `cookimport/staging/draft_v1.py` expects variant text to appear in the instruction stream before it can populate `recipe.variants`.

- Observation: The first test run failed during collection because a module-level constant rename in `recipe_span_grouping.py` left `_BRIDGEABLE_RECIPE_STRUCTURE_LABELS` pointing at the removed name.
  Evidence: Importing `tests.parsing.test_recipe_span_grouping` raised `NameError: name '_TITLE_LIKE_LABELS' is not defined`; updating that reference to `_TITLE_ANCHOR_LABELS` fixed collection immediately.

## Decision Log

- Decision: Treat `RECIPE_VARIANT` as recipe-local support, not as a span-starting title anchor.
  Rationale: The user requirement is explicit: a variant can only exist as part of the main recipe it modifies. A standalone recipe span must therefore start from a true `RECIPE_TITLE`, not from a variant heading or variant prose.
  Date/Author: 2026-04-10 / Codex

- Decision: Preserve variant text by projecting accepted in-span `RECIPE_VARIANT` rows into the `RecipeCandidate.recipeInstructions` sequence.
  Rationale: The current draft builder already knows how to split variant lines out of the instruction stream into `recipe.variants`. Reusing that downstream contract is smaller and safer than inventing a second variant-only payload seam inside label-first conversion.
  Date/Author: 2026-04-10 / Codex

## Outcomes & Retrospective

`RECIPE_VARIANT` now behaves as a child of the parent recipe instead of a title-like recipe boundary starter. In practice that means the label-first path keeps trailing `Variations` runs inside the accepted recipe span, avoids routing those blocks into nonrecipe knowledge review, and carries the variant lines into the `RecipeCandidate` instruction stream so the existing draft builder can keep extracting `recipe.variants`.

This change intentionally keeps orphaned variant-only clusters fail-closed. If `RECIPE_VARIANT` appears without a parent recipe span, it still routes back to `NONRECIPE_CANDIDATE` rather than silently inventing a standalone recipe.

## Context and Orientation

The relevant ownership path starts in `cookimport/parsing/recipe_span_grouping.py`. That module receives block-level authoritative labels after line-role and groups adjacent recipe-local labels into candidate recipe spans. A "recipe span" in this repository means the contiguous source-block range that will become one recipe candidate. Today, `RECIPE_VARIANT` is treated as title-like there, so a trailing `Variations` section flushes the current recipe span and starts a new pending span.

The acceptance rule for a recipe span is also in `cookimport/parsing/recipe_span_grouping.py`: an accepted span needs a title anchor plus ingredient evidence plus instruction evidence. Supportive labels such as `RECIPE_VARIANT`, `RECIPE_NOTES`, `YIELD_LINE`, and `TIME_LINE` do not satisfy those core fields by themselves.

`cookimport/parsing/label_source_of_truth.py` consumes those accepted spans in `build_conversion_result_from_label_spans(...)`. It separates `outside_recipe_blocks` from recipe-owned blocks and builds `RecipeCandidate` payloads from the accepted span rows. Right now that payload builder keeps recipe titles, ingredients, instructions, notes, yield, and time, but it does not carry `RECIPE_VARIANT` rows into the candidate.

Downstream, `cookimport/staging/draft_v1.py` already has the active recipe-variant projection behavior. It scans instruction text for `Variation` / `Variations` headers and inline variant lines, removes those rows from the step list, and writes them into `recipe.variants`. That means the smallest safe change is to keep `RECIPE_VARIANT` rows inside the recipe span and include them in the candidate instruction stream.

The regression tests that already own these seams are:

- `tests/parsing/test_recipe_span_grouping.py` for span acceptance and rejected-span routing.
- `tests/parsing/test_label_first_conversion.py` for label-first conversion and recipe/nonrecipe ownership.
- `tests/staging/test_draft_v1_variants.py` for the downstream draft variant extraction contract.

## Plan of Work

First, edit `cookimport/parsing/recipe_span_grouping.py` so only `RECIPE_TITLE` starts a new pending recipe span and only `RECIPE_TITLE` counts as the required title anchor for acceptance. Keep `RECIPE_VARIANT` in `_RECIPE_LOCAL_LABELS` so it remains inside an already-open recipe span. Leave the rejected-span normalization behavior intact for orphaned variant runs: if variant blocks appear without a parent recipe span, they should still fail closed back to `NONRECIPE_CANDIDATE`.

Second, edit `cookimport/parsing/label_source_of_truth.py` in `_build_recipe_candidate_from_span(...)`. Build the recipe instruction list from the ordered span rows whose final labels are `HOWTO_SECTION`, `INSTRUCTION_LINE`, or `RECIPE_VARIANT`. Preserve source order and avoid dropping valid variant rows. Keep notes, ingredients, yield, and time behavior unchanged.

Third, add or update tests in `tests/parsing/test_recipe_span_grouping.py` and `tests/parsing/test_label_first_conversion.py`. The grouping regression should prove that a recipe followed by a `Variations` block remains one accepted span and that the variant blocks no longer appear in rejected nonrecipe routing. The conversion regression should prove that those variant rows stay out of `outside_recipe_blocks` and are present in the built recipe candidate instruction list so the staging contract can still extract them later.

Finally, update the parsing documentation to state the new ownership rule plainly: `RECIPE_VARIANT` can only exist inside an accepted parent recipe span and never begins its own recipe.
