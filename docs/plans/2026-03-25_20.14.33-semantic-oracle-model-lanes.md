# Make Oracle Bench Model Selection Semantic Instead Of Version-Pinned

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

After this change, the Oracle benchmark upload and follow-up flows will talk about stable model categories such as `instant`, `pro`, and `thinking` instead of teaching `gpt-5.3`, `gpt-5.4`, or other version-pinned strings throughout repo code, tests, and docs. A user should be able to inspect the Oracle bench path and see the durable policy plainly: helper traffic defaults to the `instant` lane, operator review defaults to the `pro` lane, and literal model IDs only exist at the final Oracle command boundary or as an explicit escape hatch.

The visible result is that benchmark metadata, tests, and CLI help become stable across model-family churn. The code will still pass a concrete model string to Oracle, but only one translation seam will decide that raw string.

## Surprises & Discoveries

- Observation: the repo already partially wanted this abstraction, but the seam was split across naming conventions.
  Evidence: `ORACLE_INSTANT_MODEL` and `ORACLE_PRO_MODEL` already exist, while the implementation and tests still taught `gpt-5.3` / `gpt-5.4` in many places.

- Observation: `cookimport/bench/README.md` already states that Thinking is not the default local/operator policy.
  Evidence: the current note says the test lane is Instant-backed and that Thinking is intentionally not part of the default benchmark policy.

- Observation: the cleanest way to remove version-number teaching without losing debuggability was to split metadata in two.
  Evidence: repo-owned `model` can now stay semantic (`instant` / `pro` / `thinking` or an explicit raw override), while `launch_model` preserves the concrete Oracle picker target.

## Decision Log

- Decision: keep `--model` as the operator-facing override flag, but make its preferred values semantic selectors (`instant`, `pro`, `thinking`) while still accepting literal model IDs as a low-level escape hatch.
  Rationale: the CLI surface already exists and does not need a larger flag migration; the real problem is what the code teaches and persists, not the flag name itself.
  Date/Author: 2026-03-25 / Codex

- Decision: preserve the current default policy of helper=`instant` and operator=`pro`; do not silently make Thinking the default anywhere.
  Rationale: the user wants stable categories, not more Thinking defaults, and the current bench docs explicitly say Thinking is not the intended default policy.
  Date/Author: 2026-03-25 / Codex

- Decision: store the semantic selector in repo-owned metadata and status where practical, and keep the literal launch string available separately only when needed.
  Rationale: metadata should teach durable intent; the raw launch string is transport detail.
  Date/Author: 2026-03-25 / Codex

## Outcomes & Retrospective

The refactor achieved the intended abstraction shift. The Oracle bench path now keeps model policy semantic in repo code: helper-mode defaults resolve to `instant`, operator-mode defaults resolve to `pro`, and `thinking` is available as an explicit selector instead of a hidden point-release habit. Raw Oracle model strings still exist, but only at the command boundary and in the dedicated `launch_model` metadata field.

The focused tests passed after the change, and the touched docs now describe selector intent first instead of teaching point-release rewrites as if those were the durable contract. One literal fallback raw model string still exists inside the translation seam for the instant lane default, which is acceptable because the user-visible and repo-visible contract is now the semantic selector, not that specific raw value.

## Interfaces and Dependencies

In `cookimport/bench/oracle_upload.py`, keep or add a small set of stable helpers:

    normalize_oracle_model_selector(model: str | None) -> str
    resolve_oracle_benchmark_model(model: str | None = None, *, env: Mapping[str, str] | None = None) -> str
    resolve_oracle_launch_model(model: str, *, env: Mapping[str, str] | None = None) -> str

`resolve_oracle_benchmark_model(...)` should return the semantic selector or explicit raw override used by repo code. `resolve_oracle_launch_model(...)` should translate only at the Oracle boundary.

Compatibility rules must remain explicit:

- `ORACLE_INSTANT_MODEL` stays the preferred raw override for the `instant` lane.
- `ORACLE_PRO_MODEL` stays the preferred raw override for the `pro` lane.
- legacy env names such as `ORACLE_TEST_MODEL`, `ORACLE_GENUINE_MODEL`, `ORACLE_REVIEW_MODEL`, and `ORACLE_DEEP_REVIEW_MODEL` must still be honored as compatibility inputs.
- `--model` remains the manual override seam and may still accept a literal Oracle model string.
