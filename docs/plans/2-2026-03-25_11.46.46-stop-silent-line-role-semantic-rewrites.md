# Stop Silent Line-Role Semantic Rewrites After Codex Acceptance

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

`docs/PLANS.md` is checked into this repository and this document must be maintained in accordance with it.

## User Intent

Exact intent from the user for this plan:

- keep validation
- keep fallback for invalid or missing output
- pure artifact hygiene or metadata cleanup may be okay
- silent semantic overrides in the final authority path are bad
- future agents must not be able to casually add more ad hoc rewrite rules here
- Codex labels should either be accepted after structural validation, or rejected and sent to deterministic fallback
- accepted Codex labels should not be silently massaged into different semantics afterward
- if repo-owned policy still exists, it should prefer explicit reject/fallback over hidden rewrite
- every rejected row should be logged clearly
- future-agent safety matters as much as the immediate cleanup

Coherent restatement:

The user wants the best program possible, but with a very specific safety contract around line-role authority. Structural validation and deterministic fallback are good and should stay. What must go away is the hidden middle state where Codex appears to have been accepted, but repo code quietly rewrites the semantic answer afterward. The desired end state is simple and teachable: structurally valid Codex labels survive as Codex labels, structurally invalid or policy-rejected Codex labels fail closed to deterministic fallback, and the codebase makes that rule obvious enough that a future agent cannot “helpfully” slip new semantic rewrite rules back into the final authority path.

## Purpose / Big Picture

After this change, the line-role contract will become teachable and safe: a Codex line-role label will either survive after structural validation, or it will be rejected and fall back to the deterministic baseline for that row. The repo will stop silently rewriting accepted Codex labels into different semantic answers after validation.

The user-visible proof is simple. A reviewer will be able to inspect a `label_llm_correct` row and trust that if `decided_by="codex"`, the semantic label came from Codex and was not later massaged by repo heuristics into a third answer. If the repo rejects a Codex label on policy grounds, the row will land on deterministic fallback explicitly and loudly instead of being silently rewritten. This matters because the current `_sanitize_prediction(...)` seam is a future-agent trap: it invites well-intentioned “small fixes” that quietly change final semantic authority.

## Surprises & Discoveries

- Observation: the current gate is not one gate. Structural validation, watchdog retry, repair, per-task fallback, and post-promotion sanitization are separate layers.
  Evidence: `cookimport/parsing/canonical_line_roles/validation.py` rejects malformed outputs; `cookimport/parsing/canonical_line_roles/runtime.py` runs retry/repair and aggregates per-task fallback; `cookimport/parsing/canonical_line_roles/policy.py:_sanitize_prediction(...)` can still rewrite the accepted semantic answer afterward.

- Observation: the shard-level “pathology guard” records suspicious uniform output, but it does not invalidate the proposal.
  Evidence: `_validate_line_role_payload_semantics(...)` returns `semantic_diagnostics`, and `_evaluate_line_role_response_with_pathology_guard(...)` stores those diagnostics in metadata without flipping `proposal_status` to invalid.

- Observation: the current implementation already has the fail-closed building blocks needed for a safer contract.
  Evidence: per-task structural rejection already exists; invalid or missing task outputs already fall back to deterministic baseline row-by-row; accepted shard proposals already record fallback indices and task validation errors.

- Observation: `_sanitize_prediction(...)` mixes three different concerns in one harmless-sounding function name: semantic rewrite, heuristic rescue, and review-exclusion metadata shaping.
  Evidence: the function can coerce labels such as outside-recipe `KNOWLEDGE -> OTHER`, rescue `OTHER -> KNOWLEDGE`, rewrite `TIME_LINE`, `YIELD_LINE`, `RECIPE_TITLE`, `RECIPE_VARIANT`, `HOWTO_SECTION`, and `INSTRUCTION_LINE`, and also infer `review_exclusion_reason`.

## Context and Orientation

This repository has a label-first recipe boundary stage. “Line-role” means the phase that assigns each atomic line a semantic label such as `RECIPE_TITLE`, `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `OTHER`, or `KNOWLEDGE`. In the live runtime, deterministic rules build a baseline answer first, then optional Codex review can propose corrected labels, and later recipe grouping consumes the resulting label stream.

The key files are:

- `cookimport/parsing/canonical_line_roles/validation.py`
  This is the structural gate for line-role worker output. It checks that a task returns JSON with exactly the owned `atomic_index` rows, no duplicates, no unowned rows, valid labels, and valid `review_exclusion_reason` usage.

- `cookimport/parsing/canonical_line_roles/runtime.py`
  This is the line-role orchestrator. It builds deterministic baseline labels, runs the Codex worker runtime, retries missing outputs when the watchdog killed a retryable worker, attempts one repair for invalid outputs, aggregates validated tasks into shard proposals, and falls back to deterministic baseline for failed tasks.

- `cookimport/parsing/canonical_line_roles/policy.py`
  This currently contains `_sanitize_prediction(...)`, which runs after a label exists and can rewrite its semantic meaning. This is the seam this plan is trying to split apart.

- `tests/parsing/test_canonical_line_roles.py`
  This is the main protection surface for line-role behavior. The new contract must be made explicit here.

- `docs/04-parsing/04-parsing_readme.md`
  This is the durable parsing reference. After implementation, it must teach the new acceptance contract and stop implying that repo policy can silently rewrite accepted Codex labels.

- `docs/01-architecture/01-architecture_README.md`
  This is the cross-cutting architecture source of truth. It must explain the new line-role contract in plain language so future agents do not reintroduce the old seam.

Two phrases matter in this plan:

- “Structural validation” means validating output shape and ownership: JSON object, `rows` list, owned `atomic_index` coverage, allowed labels, and valid `review_exclusion_reason` usage.
- “Semantic rewrite” means changing an already accepted label into a different label after the structural gate passed. This plan treats semantic rewrite as disallowed on the accepted Codex path.

The current acceptance path works like this:

1. Build deterministic baseline predictions.
2. Run Codex tasks.
3. Validate each task structurally.
4. Retry or repair certain invalid outputs.
5. Fall back to deterministic baseline for failed tasks.
6. Promote accepted rows as `decided_by="codex"`.
7. Run `_sanitize_prediction(...)`, which can still silently change the semantic label after step 6.

This plan removes step 7 as a semantic rewrite seam. After the change, post-validation code may still normalize metadata, but it may not silently change accepted Codex meaning.

## Plan of Work

Start by splitting the current `_sanitize_prediction(...)` responsibility into two distinct categories: things that are still allowed after Codex acceptance, and things that are not.

Allowed after acceptance:

- attaching or deduplicating metadata fields such as `escalation_reasons`
- carrying forward explicit `review_exclusion_reason` only when the label contract allows it
- non-semantic hygiene that does not change `label` or `decided_by`

Not allowed after acceptance:

- changing `label`
- changing `decided_by` from `codex` to `fallback`
- heuristic rescues such as `OTHER -> KNOWLEDGE`
- heuristic coercions such as `KNOWLEDGE -> OTHER`
- any local rule that invents a third semantic answer after the Codex row already passed structural validation

Implementation should then introduce one narrow hard-policy check for accepted Codex rows. That check should answer one question only: “is this Codex label allowed to survive as-is in this context?” If yes, keep it. If no, reject it to deterministic fallback for that row and record an explicit reason. Do not rewrite it to a different non-baseline label.

The deterministic baseline may still keep repo-owned heuristics, because baseline is the repo’s own answer lane. The accepted Codex path is different. It must stop using that rewrite machinery after validation. If the same helper currently serves both baseline shaping and Codex promotion, split the helper rather than keeping one shared post-acceptance policy function with a permissive name.

The cleanest shape is:

- `runtime.py` continues to build deterministic baseline predictions.
- `runtime.py` continues to validate, retry, repair, and aggregate Codex task outputs.
- accepted Codex rows are promoted directly to final predictions.
- a new narrow helper such as `normalize_prediction_metadata(...)` may attach metadata but must never change label semantics.
- a new narrow helper such as `reject_prediction_to_baseline_if_policy_violated(...)` may reject an accepted Codex row to the deterministic baseline with an explicit reason code.
- `_sanitize_prediction(...)` should either disappear entirely or survive only as a deterministic-baseline-only helper under a scarier and more precise name.

Branch-by-branch treatment for the current `_sanitize_prediction(...)` logic should be:

- invalid label values: delete from this seam; structural validation already handles this.
- inside-recipe `KNOWLEDGE` rejection: convert to explicit fallback-to-baseline on the accepted Codex path, not `KNOWLEDGE -> OTHER`.
- non-primary `TIME_LINE` rewrite: convert to explicit fallback-to-baseline on the accepted Codex path.
- neighbor ingredient rescue: keep only in deterministic baseline logic if still wanted; do not apply it to accepted Codex rows.
- `YIELD_LINE` rewrites: keep only in deterministic baseline logic if still wanted; do not apply them to accepted Codex rows.
- outside-span title, variant, howto, and instruction sanitization: convert to explicit fallback-to-baseline on the accepted Codex path if those labels violate hard policy.
- `OTHER -> KNOWLEDGE` and `OTHER -> INSTRUCTION_LINE` rescues: keep only in deterministic baseline logic if still wanted; do not apply them after Codex acceptance.
- outside-recipe `KNOWLEDGE -> OTHER` coercion: convert to explicit fallback-to-baseline on the accepted Codex path.
- inferred `review_exclusion_reason`: move into a metadata-only helper and keep it only when the surviving final label is `OTHER` and outside recipe.

After the runtime change, update the tests to enforce the new contract. There should be at least three categories of test:

First, a valid Codex label that passes policy should survive untouched into the final prediction with `decided_by="codex"`.

Second, a structurally invalid Codex output should still fail closed exactly as today: retry or repair when applicable, then deterministic fallback if still invalid.

Third, a structurally valid but policy-invalid Codex label should now fall back explicitly to the deterministic baseline instead of being silently rewritten into a different semantic label. That test is the future-agent tripwire.

Finally, update the docs. `docs/04-parsing/04-parsing_readme.md` should describe the new contract in plain language. `docs/01-architecture/01-architecture_README.md` should state that line-role Codex labels survive after structural validation unless they are explicitly rejected to deterministic fallback. `docs/plain-english-explanation.md` should use the same framing if it still mentions repo validation on the line-role stage.

## Milestones

### Milestone 1: Split the contract into validation, fallback, and metadata-only post-processing

At the end of this milestone, the codebase should have an explicit place where a reader can see the line-role contract without reading `_sanitize_prediction(...)` line-by-line. Structural validation and deterministic fallback should remain intact, but post-acceptance behavior should be obviously narrower.

The implementation work is in `cookimport/parsing/canonical_line_roles/runtime.py` and `cookimport/parsing/canonical_line_roles/policy.py`. The result should be that accepted Codex rows are no longer passed through a helper whose job includes semantic rewrite, rescue, and metadata shaping all at once.

Acceptance for this milestone is code structure and one or two focused tests that prove a valid Codex label survives unchanged when it passes policy.

### Milestone 2: Replace silent semantic rewrites with explicit reject-to-baseline behavior

At the end of this milestone, any accepted Codex row that violates a hard repo policy should be rejected to deterministic fallback, not rewritten to a third semantic answer. The code should record an explicit reason for that rejection so debugging stays honest.

The implementation work is mostly in the promotion path in `cookimport/parsing/canonical_line_roles/runtime.py`, plus whatever narrow helper remains in `policy.py`. This is the milestone that actually changes behavior.

Acceptance for this milestone is that the old rewrite-shaped tests either disappear or are rewritten to expect explicit fallback, and new tests prove that no accepted Codex row is silently rewritten by repo policy.

### Milestone 3: Lock the contract down in docs and regression tests

At the end of this milestone, the docs teach the new contract and the tests make it hard to regress. A future agent should be able to read the parsing and architecture docs, then inspect the tests, and immediately understand that “post-validation semantic rewrite” is not an allowed seam anymore.

Acceptance for this milestone is updated docs plus focused passing tests in the line-role suite and, if needed, prompt-preview or benchmark-adjacent tests whose expectations changed because `escalation_reasons` and fallback behavior became more explicit.

## Validation and Acceptance

Acceptance is behavioral, not merely structural.

First, the line-role suite must prove that a structurally valid and policy-valid Codex label survives unchanged. A reader should be able to inspect the test and see that the final prediction still carries the Codex semantic label rather than a repo-rewritten substitute.

Second, the line-role suite must prove that malformed Codex output still fails closed. Missing rows, duplicate rows, invalid labels, and missing output must still land on retry, repair, or deterministic fallback behavior exactly as before.

Third, the line-role suite must prove the new contract at the seam that motivated this plan: a structurally valid but policy-invalid Codex label must now land on deterministic fallback explicitly. The final prediction must not silently become some third label invented by repo heuristics after validation.

Fourth, the docs must explain the same story. After implementation, `docs/04-parsing/04-parsing_readme.md` and `docs/01-architecture/01-architecture_README.md` should let a new reader answer this question correctly: “What happens when Codex returns a structurally valid but semantically disallowed line label?” The answer should be “explicit fallback,” not “repo rewrite.”

## Idempotence and Recovery

This plan is safe to implement incrementally. The safest order is additive refactor first, behavior change second, docs/tests cleanup third. If the implementation stalls halfway, do not keep both old `_sanitize_prediction(...)` semantics and a new fallback contract live at once longer than necessary. That would teach two contradictory stories.

If a change produces too much benchmark churn, do not immediately reintroduce silent rewrite logic. Instead, inspect whether the deterministic baseline should be improved or whether the Codex prompt/runtime needs better guidance. The whole point of this plan is to keep the semantic authority story honest.

The tests in `tests/parsing/test_canonical_line_roles.py` are safe to rerun repeatedly. If a narrow test starts failing unexpectedly, compare the final `decided_by`, `reason_tags`, and `escalation_reasons` fields before trying to “fix” quality with a new local rewrite rule.
