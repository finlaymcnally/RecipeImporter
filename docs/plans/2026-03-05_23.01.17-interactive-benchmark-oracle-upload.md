# Automatically upload interactive benchmark bundles to Oracle

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds. This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, an interactive benchmark run in `cookimport` should no longer stop at writing `upload_bundle_v1`. When the run finishes, the CLI should automatically send the right bundle to Oracle: for a single-book run, the single-offline session bundle; for a multi-book single-profile run, the top-level group bundle under `single-profile-benchmark/upload_bundle_v1`.

The user should also be able to exercise the exact same Oracle-upload path without rerunning a benchmark. The observable proof is a dedicated CLI command that accepts an existing benchmark session root or `upload_bundle_v1` folder, resolves the same bundle the interactive path would use, and either performs a dry-run Oracle preview or launches the real Oracle browser upload.

## Decision Log

- Decision: hook Oracle upload in the interactive wrapper layer, not inside `labelstudio_benchmark(...)` itself.
  Rationale: the user only asked for interactive benchmark mode, and the wrapper layer already owns bundle creation for those flows. Keeping Oracle outside `labelstudio_benchmark(...)` avoids changing the non-interactive benchmark command contract.
  Date/Author: 2026-03-05 / Codex

- Decision: for multi-book runs, upload only the final group bundle rooted at `single-profile-benchmark/upload_bundle_v1`.
  Rationale: that matches the user's requirement and avoids wasting Oracle context on one upload per book when the top-level group bundle already summarizes the run.
  Date/Author: 2026-03-05 / Codex

- Decision: Oracle upload failures must be non-fatal to the benchmark run.
  Rationale: benchmark execution and artifact generation are the primary job; Oracle is an external post-processing step and is explicitly described by the user as buggy.
  Date/Author: 2026-03-05 / Codex

- Decision: add one direct CLI command for existing bundles instead of forcing testing through a fresh benchmark run.
  Rationale: the benchmarks are expensive, and the user explicitly wants the benchmark-to-Oracle path tested without rerunning them.
  Date/Author: 2026-03-05 / Codex

- Decision: use the Oracle headless browser wrapper for the real upload path, but support a dry-run Oracle preview mode for low-cost validation.
  Rationale: the Oracle skill instructions say the headless wrapper is the correct browser path for this machine, while `--dry-run` is the safest no-token way to validate prompt, files, and bundle resolution.
  Date/Author: 2026-03-05 / Codex

## Outcomes & Retrospective

This document is the planning artifact only; no implementation has landed yet. The main outcome so far is a narrow design that reuses the existing upload-bundle seam instead of inventing new benchmark artifacts or forcing benchmark reruns for testing.

The main remaining risk is accidental overreach into the core benchmark runner. This plan avoids that by keeping the Oracle logic in a small helper module plus the interactive wrapper layer, with one separate testable CLI entrypoint for existing bundles.

## Context and Orientation

The relevant runtime starts in `cookimport/cli.py::_interactive_mode(...)`. When the user picks `Evaluate vs freeform gold`, interactive mode dispatches into two benchmark wrappers:

- `cookimport/cli.py::_interactive_single_offline_benchmark(...)` for one selected source and one benchmark session root under `data/golden/benchmark-vs-golden/<timestamp>/single-offline-benchmark/...`.
- `cookimport/cli.py::_interactive_single_profile_all_matched_benchmark(...)` for repeated runs across matched books under `data/golden/benchmark-vs-golden/<timestamp>/single-profile-benchmark/...`.

Neither wrapper generates the three-file external-AI bundle inline. Instead, both call `cookimport/cli.py::_write_benchmark_upload_bundle(...)`. That helper imports `scripts/benchmark_cutdown_for_external_ai.py::build_upload_bundle_for_existing_output(...)` and writes exactly three files:

- `upload_bundle_overview.md`
- `upload_bundle_index.json`
- `upload_bundle_payload.jsonl`

The single-offline wrapper writes one bundle rooted at the single session folder. The single-profile wrapper writes one bundle per target book and then one final group bundle at the root `single-profile-benchmark/upload_bundle_v1`. The user wants Oracle to receive the root group bundle in the multi-book case and the session bundle in the single-book case.

The Oracle skill contract for this machine is specific and matters here. Real browser execution should use `/home/mcnal/.local/bin/oracle-browser-headless`, not a raw `npx -y @steipete/oracle --engine browser ...` call. The safest preview path is Oracle dry-run mode. The bundle itself is small and deterministic, so the Oracle call should attach the literal bundle directory with `--file <bundle_dir>` rather than trying to rediscover or glob benchmark artifacts again.

The checked-in real sample bundle for manual low-cost validation is:

- `data/golden/benchmark-vs-golden/2026-03-04_20.33.53/single-profile-benchmark/upload_bundle_v1`

There is no equivalent checked-in single-offline sample bundle. Single-offline routing therefore needs to be proven by tests that create temporary bundle folders.

Terms used in this plan:

`session root` means the parent directory that owns a benchmark run, such as `.../single-offline-benchmark/<book>` or `.../single-profile-benchmark`.

`bundle directory` means the `upload_bundle_v1` folder containing the three Oracle-facing files.

`group bundle` means the high-level multi-book bundle written from the root `single-profile-benchmark` session directory.

`Oracle upload` means one CLI invocation that attaches the resolved `upload_bundle_v1` directory and sends a fixed benchmark-review prompt to Oracle.

## Plan of Work

### Milestone 1: Add a small benchmark-to-Oracle helper module

Create a new module under `cookimport/bench/`, preferably `cookimport/bench/oracle_upload.py`. Keep it intentionally small and focused on post-processing existing bundles.

This module should do three jobs.

First, resolve the input path into the right bundle directory. The resolver must accept either a direct `upload_bundle_v1` directory or its parent session root. If the user passes `.../single-profile-benchmark`, the resolver should use `.../single-profile-benchmark/upload_bundle_v1`. If the user passes the bundle directory itself, it should use that directly. If the expected three files are missing, raise a clear `ValueError`.

Second, build a fixed Oracle prompt for benchmark review. The prompt should explain the project briefly, tell Oracle that the attached directory is a benchmark upload bundle for `cookimport`, tell it to start with `upload_bundle_overview.md`, and request a concise result shape such as top regressions, likely cause buckets, and immediate next checks. Keep the prompt deterministic and in one place so the interactive flow and the direct CLI command share the exact same text.

Third, run the Oracle command. Real mode should shell out to `/home/mcnal/.local/bin/oracle-browser-headless --model gpt-5.2-pro ... --file <bundle_dir>`. Dry-run mode should shell out to `npx -y @steipete/oracle --dry-run summary --files-report ... --file <bundle_dir>`. Both modes should capture the exact command, exit code, and recent output so the caller can print a useful summary. The function must accept an injectable runner so tests can verify command assembly without invoking Oracle.

This milestone is complete when a unit test can hand the helper a temporary `upload_bundle_v1` directory, resolve it from either the directory itself or the parent session root, and verify the assembled browser and dry-run Oracle commands without network or browser execution.

### Milestone 2: Add a direct CLI command for existing bundles

Add one small CLI command under the `bench` command group in `cookimport/cli.py`. Use a name that is explicit and narrow, such as `cookimport bench oracle-upload`.

The command should accept one positional `path` that can be either a session root or a direct `upload_bundle_v1` folder. Add a `--mode browser|dry-run` option with `browser` as the default for the long-term behavior and `dry-run` as the low-cost validation path. Add an optional `--model` override, but default to `gpt-5.2-pro`.

The command body should only call the shared helper module from Milestone 1. It should print:

- the resolved bundle directory,
- the mode,
- the Oracle command being used,
- whether the run succeeded,
- enough stdout or stderr summary to tell whether Oracle accepted the bundle.

Do not make this command rerun bundle generation. It should only operate on an already-existing bundle or session root. That is the entire point of keeping benchmark-token costs out of the validation loop.

This milestone is complete when a user can run the command against `data/golden/benchmark-vs-golden/2026-03-04_20.33.53/single-profile-benchmark` in dry-run mode and see it resolve to the top-level `upload_bundle_v1` directory.

### Milestone 3: Wire interactive single-book and multi-book flows

In `cookimport/cli.py::_interactive_single_offline_benchmark(...)`, add one post-bundle hook after `_write_benchmark_upload_bundle(...)` returns a non-`None` bundle directory. That hook should call a small wrapper such as `_maybe_upload_benchmark_bundle_to_oracle(...)` with the returned session bundle directory and a scope label like `single_offline`.

In `cookimport/cli.py::_interactive_single_profile_all_matched_benchmark(...)`, do not upload the per-book bundles created inside `_run_single_profile_target(...)`. Instead, hook only the final root-level `group_upload_bundle_dir` after the high-level group bundle is written from `single_profile_root`. That is the path the user named explicitly.

Keep Oracle upload best-effort. If bundle generation fails, skip Oracle entirely. If Oracle itself fails, print a yellow warning and keep the benchmark marked complete. The warning should include the bundle path and enough command context that the user can rerun the direct CLI command manually.

This milestone is complete when tests prove that:

- single-offline interactive runs call Oracle once with the single session bundle,
- single-profile interactive runs call Oracle once with the top-level group bundle,
- multi-book per-target bundles are still generated but are not uploaded to Oracle.

### Milestone 4: Add low-cost tests and short docs

Add focused tests instead of end-to-end Oracle execution.

For bundle-resolution and command assembly, add tests in a new benchmark-focused test file such as `tests/bench/test_benchmark_oracle_upload.py`. Use temporary directories with the three required bundle files and monkeypatch the process runner.

For interactive routing, extend the existing benchmark helper tests:

- `tests/labelstudio/test_labelstudio_benchmark_helpers.py` for single-offline routing.
- `tests/labelstudio/test_labelstudio_benchmark_helpers_single_profile.py` for multi-book group-bundle routing.

These tests should monkeypatch `_write_benchmark_upload_bundle(...)` and the new Oracle wrapper so the suite never invokes a real benchmark or a real Oracle browser session.

Then add short documentation updates:

- `docs/02-cli/02-cli_README.md`: mention the new `cookimport bench oracle-upload` command and that interactive benchmark mode auto-uploads the final bundle to Oracle.
- `docs/07-bench/07-bench_README.md`: document which bundle is uploaded for single-offline versus multi-book single-profile runs.
- `cookimport/bench/README.md`: add a one-paragraph note that benchmark-to-Oracle upload is a post-bundle wrapper flow and that dry-run mode is the preferred no-cost validation path.

This milestone is complete when the new tests pass and the docs clearly explain how to rerun Oracle upload from an existing bundle without rerunning a benchmark.

## Idempotence and Recovery

The direct `cookimport bench oracle-upload` command must be safe to rerun. It should never mutate the benchmark bundle. Re-running it may create a fresh Oracle session, which is acceptable.

Interactive benchmark reruns are not required for recovery. If the benchmark succeeded but Oracle failed, the user should be able to rerun only:

    cookimport bench oracle-upload <existing session root or upload_bundle_v1>

If a real Oracle browser run detaches or times out, print recovery guidance taken from the Oracle skill contract:

    oracle status --hours 72
    oracle session <id> --render

Keep benchmark completion separate from Oracle completion. Do not delete or rewrite `upload_bundle_v1` during Oracle upload.
