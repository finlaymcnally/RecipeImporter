# Tighten Knowledge Hints For Worker Utility

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the knowledge-stage hint file should help the worker make faster, better local decisions instead of acting like a second full prompt. A worker opening `hints/<shard_id>.md` should immediately get the few repo-written signals that are actually useful before reading the authoritative shard input: what kind of packet this is, what mistakes to avoid, which rows deserve attention, and which contrast examples are worth opening.

The user-visible effect is not a new definition of `knowledge`. The effect is a sharper worker surface. A future sampled knowledge worker root should show a short hint file that feels more like a debugger sidebar than a prose essay, and a human reading worker traces should see less re-reading and less prompt-like drift before the worker edits the active work ledger.

## Outcomes & Retrospective

This plan is not implemented yet. The intended outcome is that the knowledge hint becomes something a worker would actually want to read first: short, local, and cue-heavy. Success means the worker can get the packet’s likely failure mode from the hint in a few seconds and then move into the authoritative `in/<shard_id>.json` with less hesitation and less re-reading.

The most important lesson from the pre-plan investigation is that “more guidance” is not the same as “better guidance.” The worker already has plenty of prose. The hint only earns its place if it adds local, precomputed information that is expensive or annoying for the worker to derive from raw packet text on the fly.

## Context and Orientation

The knowledge stage is the optional large-language-model stage that reviews non-recipe book text and decides which blocks are worth preserving as reusable cooking knowledge. A “shard” in this repository is one worker-owned unit of work. For knowledge, a shard owns one ordered packet of non-recipe blocks, runs Pass 1 classification (`knowledge` or `other` for each owned block), then runs Pass 2 grouping only for the kept `knowledge` blocks.

The active knowledge worker root is written by `cookimport/llm/knowledge_stage/workspace_run.py`. That module writes the worker-local `in/`, `work/`, `repair/`, `out/`, `examples/`, `tools/knowledge_worker.py`, `CURRENT_PHASE.md`, `CURRENT_PHASE_FEEDBACK.md`, and `hints/<shard_id>.md` files. The worker prompt text lives in `cookimport/llm/knowledge_stage/recovery.py`, and the worker-side helper CLI lives in `cookimport/llm/knowledge_phase_workspace_tools.py`.

The current hint generation logic also lives in `cookimport/llm/knowledge_stage/recovery.py`, specifically in `_write_knowledge_worker_hint(...)`, `_build_knowledge_hint_profile_and_policy(...)`, and `_build_knowledge_hint_attention_lines(...)`. That code reads only deterministic local shard data from `shard.input_payload`. In plain language, the hint is built from the raw owned packet blocks and nearby recipe guardrails, not from model output and not from a second model pass.

The comparison point is the line-role stage. Its hints are generated in `cookimport/parsing/canonical_line_roles/planning.py`. Line-role is not semantically identical to knowledge, but its hint ergonomics are the right benchmark: the hint gives a compact shard profile, a short default posture, concrete examples, and a short list of suspicious rows. That shape is useful because it compresses local evidence into a small precomputed surface rather than repeating the full task policy.

The current knowledge hint sections are:

- `How to use this task`
- `Shard profile`
- `Shard interpretation`
- `Decision policy`
- `Shard examples`
- `Packet summary`
- `Attention rows`

The user’s requested change is not to remove hints, but to make them genuinely worker-useful. In practical terms, that means keeping the sections that add local information and trimming sections that just restate the main prompt or the output contract.

## Plan of Work

Start in `cookimport/llm/knowledge_stage/recovery.py`. Read the current `_write_knowledge_worker_hint(...)` implementation line by line and classify every emitted line into one of three buckets: high-signal local cue, mildly useful metadata, or redundant prompt restatement. Preserve the first bucket, be selective with the second, and aggressively cut or compress the third. This pass should be done deliberately and documented in the plan if any section changes meaning rather than just length.

The first implementation target is the `How to use this task` section. That section should stop behaving like a mini prompt. It should become a short orientation section that says what the hint is for and how to use it operationally. Generic semantic reminders like “keep only durable cooking leverage” should survive in the hint only if they are serving as the shard’s default posture and are phrased in a way that actually helps the worker avoid a known local error.

The second target is the section mix itself. The default target shape should be a compact hint that foregrounds `Shard interpretation`, `Decision policy`, `Shard examples`, and `Attention rows`. `Shard profile` and `Packet summary` should survive only to the extent that they help the worker predict what kind of packet they are looking at. For example, counts that support a real inference such as “many heading-like rows” or “table-hint rows present” are acceptable; counts that do not change behavior should be removed.

The third target is the decision policy language. It should remain short and operational. Each bullet should correspond to a real mistake pattern the worker could make on that shard class, such as whole-packet inertia, heading over-promotion, or table/reference over-keeping. If a policy bullet merely repeats the global prompt contract, move that guidance back to the prompt and drop it from the hint.

The fourth target is testing. Update or add focused tests in `tests/llm/test_knowledge_orchestrator_contracts.py` so the intended hint shape is locked in. The tests should assert presence of the high-signal surfaces and absence of drift back toward a long mini-prompt. Keep the assertions behavioral and compact: for example, assert that example pointers and attention-row cues remain present, and that the rendered hint still has the expected high-value sections after the trim.

Finally, update `docs/10-llm/10-llm_README.md` and the existing task note for richer knowledge hints so the docs say what the hint is for in current tense: a compact worker calibration surface, not a second prompt. If the final implementation materially changes what the worker is expected to read first, capture that clearly in the docs so a future contributor does not rebuild the old, verbose hint style.

## Validation and Acceptance

Acceptance requires all of the following:

1. The rendered knowledge hint still gives the worker the high-signal local calibration surfaces: shard interpretation, decision policy, example pointers, and attention rows.
2. The rendered hint is materially shorter or less redundant than the current version because generic prompt restatements have been removed or compressed.
3. The worker-facing instructions still make it clear that the hint is guidance and that `in/<shard_id>.json` remains the authoritative shard input.
4. Focused tests in `tests/llm/test_knowledge_orchestrator_contracts.py` pass and would catch a regression back toward a long prompt-like hint.
5. The docs explain that knowledge hints are meant to be compact worker calibration aids rather than a second semantic policy surface.

Behavioral acceptance for a human reviewer is straightforward: when they open a generated `hints/<shard_id>.md`, they should be able to answer “what kind of packet is this, what should I watch for, and which example should I open if I’m unsure?” in a few seconds without having to reread a mini prompt.

## Idempotence and Recovery

This plan changes only repo-written hint text, tests, and docs. It is safe to rerun the tests and to regenerate worker roots after implementation. If a trim goes too far and workers lose needed context, restore the last removed hint line, add or strengthen a focused regression test that explains why it matters, and retry. Avoid mixing this work with prompt-schema changes or shard-planning changes; if those become necessary, open follow-up tasks rather than widening this plan.
