# Reduce Live Codex Token Overhead And Align Preview With Actual Runtime

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

After this change, RecipeImport should behave much closer to the operator’s intended mental model: if a phase is configured to use `5` prompts, the live run should make `5` real model calls for that phase, and the billed token total should stay close to the visible shard payload plus a modest wrapper cost. Today that is not true. On the benchmark root `data/golden/benchmark-vs-golden/2026-03-17_20.27.19`, prompt preview predicted about `253,307` total tokens while the corrected live run actually used `767,279` tokens, even though both paths agreed on `15` calls.

The concrete user-visible outcome is twofold. First, live Codex runs should get materially cheaper because the runtime stops paying for hidden transport overhead and overgrown outputs. Second, `cf-debug preview-prompts` should become much more trustworthy because it will be modeling the same compact payload and output contracts the live runner actually uses. A human should be able to set `3 / 3 / 3` or `5 / 5 / 5`, run preview, then run the real benchmark and see the live spend land in the same neighborhood instead of 2x to 6x higher.

## Outcomes & Retrospective

The largest structural change has landed: recipe now uses the same one-shard / one-direct-call transport shape as knowledge and line-role, and the recipe prompt/output contracts are compact enough for preview and live runtime to talk about the same visible surface. The repo also now records direct-exec visible-vs-billed breakdowns at both shard and stage summary level.

What remains is the expensive part that still spends real tokens: run one real `saltfatacidheatcutdown` benchmark, compare preview against actual with the new breakdown vocabulary, and decide whether the already-compact knowledge bundle needs another reduction pass. The practical lesson from implementation is that cost honesty improved most when the runtime and preview literally shared the same prompt/schema seam, not when preview tried to infer hidden transport behavior.

## Context and Orientation

Three LLM-backed phases matter here.

`line_role` is the stage that revisits deterministic line labels and decides the final canonical label for each atomic line of the book. The live code starts in `cookimport/parsing/canonical_line_roles.py`. A “shard” for line-role is one contiguous range of `atomic_index` rows. The target mental model is that one shard corresponds to one direct model call.

`recipe_llm_correct_and_link` is the stage that takes deterministic intermediate recipe objects and asks Codex to correct, normalize, and link them into the final recipe structure. The live code runs through `cookimport/llm/phase_worker_runtime.py` and currently still launches `codex-farm process` in `classic_task_farm_v1` mode from `cookimport/llm/codex_farm_runner.py`.

`extract_knowledge_optional` is the stage that reviews non-recipe chunks and decides which ones are worth preserving as durable knowledge snippets. The live code starts in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, and this stage already has a direct-exec transport.

`prompt preview` lives primarily in `cookimport/llm/prompt_preview.py` and `cookimport/llm/prompt_budget.py`. It is a zero-token predictor. It rebuilds planned shard payloads locally and estimates cost. It is useful only if the live runtime is using the same basic payload and output contracts.

`actual-cost reporting` for finished runs comes from `prediction-run/prompt_budget_summary.json`, written by `cookimport/llm/prompt_budget.py`. That artifact now correctly counts one stage once, but it is still downstream accounting; it does not reduce the live bill by itself.

## Plan of Work

The work should proceed in four milestones that intentionally build on each other.

### Milestone 1: Replace Recipe’s Remaining Classic Transport

At the end of this milestone, recipe should behave like line-role and knowledge: one planned shard equals one real structured model call. The code path in `cookimport/llm/phase_worker_runtime.py` and the recipe-specific orchestration should stop routing recipe live execution through `codex-farm process --runtime-mode classic_task_farm_v1 --workers 1`. Instead, recipe should use the shared direct structured runner already introduced for the newer direct-exec phases.

The important constraint is to preserve the public surface and output contracts. Operators should still enable recipe via the same pipeline setting. Final recipe drafts, manifests, and validator behavior should remain intact. What changes is the transport in the middle and the shape of the per-shard runtime artifacts. Each recipe shard should record the prompt text that was actually sent, the final structured response, the per-shard usage totals, and the minimal event stream that proves it was one direct call.

### Milestone 2: Slim Live Input Payloads

At the end of this milestone, every phase should send only the information Codex actually needs to do the job.

For line-role, the live shard JSON should be reduced to the minimum row contract consistent with the “one contiguous slice of the book” philosophy. The prompt already stopped duplicating neighbor text. The transport should follow through by dropping row metadata that is only useful for offline review or later projection if it is not needed by the model. The likely target is some compact form of `atomic_index`, one deterministic label code, and the current line text. If an additional field survives, the plan must explain why that field changes model behavior enough to justify its tokens.

For recipe, the live shard input should be audited for any bulky intermediate fields that are not necessary once the direct transport is in place. This is especially important because recipe currently appears to carry large hidden overhead relative to the visible shard content. The goal is not to invent a new recipe schema for its own sake. The goal is to remove fields that exist only because older transport or debugging layers wanted them.

For knowledge, the same rule applies: keep owned chunk ids, the chunk-local decisions the model must return, and only the evidence needed to justify those decisions. Any convenience structure that only helps local debugging should stay on disk outside the model payload instead of being sent to the model.

### Milestone 3: Compact Structured Outputs

At the end of this milestone, the model’s returned JSON should be much closer to the minimum semantic answer instead of a verbose transport object. The benchmark evidence shows that actual outputs are materially larger than preview guessed for all three phases, and especially for line-role and knowledge.

The right move here is not to make preview guess higher. It is to shrink the live schema. Use short keys, remove optional repeated prose when it is not operationally necessary, and prefer compact arrays or tuple-like row objects where the repo can still validate them reliably. The final answer stored for human debugging can remain pretty-printed on disk if needed, but the model-facing structured output should be optimized for token economy.

This milestone must be implemented with care because it touches validator seams. Any compacted schema must still validate deterministically, preserve exact ownership checks, and keep final downstream outputs unchanged after post-processing.

### Milestone 4: Add Payload-Vs-Billed Observability And Update Preview

At the end of this milestone, each live shard run should emit a small cost breakdown that answers four questions directly:

1. How many tokens were in the visible request payload?
2. How many billed input tokens were cached?
3. How many tokens were in the visible structured response?
4. How many billed tokens remain as wrapper or runtime overhead after subtracting the visible payload pieces?

Once that exists, `prompt_preview` should report the same vocabulary. For each stage and whole run, preview should show predicted visible input payload tokens, predicted visible output payload tokens, and any clearly stated assumptions about cached-input reuse before the run. Preview should stop pretending it can explain hidden runtime envelope if the runtime itself does not expose a deterministic contract for it.

This milestone is what closes the loop. After the earlier milestones simplify the runtime, preview can model the actual runtime files and actual schemas instead of a nearby approximation.

## Validation and Acceptance

The change is accepted only if both runtime cost and preview honesty improve.

On a real `saltfatacidheatcutdown` benchmark configured with `5 / 5 / 5` prompts:

- the live run should still make `15` calls total
- recipe should no longer report `classic_task_farm_v1`
- every phase should emit one clear per-shard usage artifact
- prompt preview should still predict `15` calls
- actual total tokens should fall materially below the current `767,279` baseline
- preview and actual should be much closer than the current `253,307` vs `767,279` gap

The acceptance target is not fake precision. A good first landing is:

- actual total tokens at least `25%` lower than the current benchmark baseline
- preview within about `25%` of actual total tokens on the same run

The local regression suite must also prove:

- recipe, knowledge, and line-role still preserve exact ownership validation
- final recipe drafts, final knowledge snippets, and final line labels remain structurally valid
- prompt preview still works on deterministic and benchmark roots without needing live telemetry
- actual-cost reporting still works on finished benchmark runs and does not double-count nested telemetry

## Idempotence and Recovery

This plan is safe to implement incrementally if one rule is followed: at every stopping point, there must be exactly one authoritative live transport per phase. Do not leave recipe half on classic task-farm and half on direct exec. Do not leave preview modeling a payload shape that no longer exists in the runtime.

Worker and shard artifacts may be regenerated safely. The runtime directories under `data/output/...` are disposable run artifacts. The recovery path for an interrupted implementation step is to rerun the focused tests, regenerate one preview artifact locally, and confirm that the current runtime still writes coherent per-shard status and usage files before continuing.

If one compact-output attempt breaks validators or downstream writers, revert only the schema compaction slice and keep the direct transport improvements. The transport simplification stands on its own and should not be blocked by one aggressive schema-shortening experiment.

  - final response text
