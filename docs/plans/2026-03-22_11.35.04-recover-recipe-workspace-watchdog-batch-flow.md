# Recover recipe workspace-worker watchdog failures without giving back the batch-flow win

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the recipe workspace-worker stage should keep the lower-turn `prepare-all -> edit drafts -> finalize-all` helper flow, but it must stop failing hard when a worker performs bounded local draft edits inside `scratch/`. The saved Salt Fat benchmark rerun at `data/golden/benchmark-vs-golden/2026-03-22_04.26.00/single-book-benchmark/saltfatacidheatcutdown/codexfarm/` proves that line-role benefited from the churn-reduction work while recipe regressed: two recipe workers were killed as forbidden command executions, recipe validated only three shards, and total recipe command count rose from `64` to `91`.

The user-visible success condition is concrete. A fresh rerun of the same recipe benchmark slice should return recipe `validated_shard_count: 5`, recipe `watchdog_killed_shard_count: 0`, and materially lower recipe command count than `91`, while keeping the line-role improvement already achieved in `2026-03-22_04.26.00` (`command_execution_count_total: 61`, `watchdog_killed_shard_count: 0`). A novice inspecting the worker roots should still see the same mirrored helper flow under `tools/recipe_worker.py`, but the runtime should now tolerate narrow workspace-local draft editing, recover retryable watchdog deaths, and make the prepared-draft surface obvious enough that workers do not drift into unnecessary off-road command patterns.

## Surprises & Discoveries

- Observation: the batch helper design itself is not the problem.
  Evidence: in `data/golden/benchmark-vs-golden/2026-03-22_04.26.00/.../prompt_budget_summary.json`, line-role improved from `command_execution_count_total: 106` to `61` while keeping `watchdog_killed_shard_count: 0`, and recipe `worker-001/last_message.json` reports a successful batch completion through the helper path.

- Observation: the recipe failures came from workers drifting into inline Python heredocs after `prepare-all`, not from `prepare-all` or `finalize-all` themselves.
  Evidence: `data/output/2026-03-22_04.26.00/.../recipe_phase_runtime/workers/worker-002/events.jsonl` and `worker-005/events.jsonl` both show successful `python3 tools/recipe_worker.py prepare-all --dest-dir scratch/` commands followed by `python3 - <<'PY'` scripts that rewrite multiple `scratch/*.json` files.

- Observation: the current recipe workspace watchdog treats those kills as terminal and non-retryable.
  Evidence: `worker-002/live_status.json` and `worker-005/live_status.json` both end with `state: "watchdog_killed"`, `reason_code: "watchdog_command_execution_forbidden"`, and `retryable: false`.

- Observation: prompt wording alone is not enough to keep recipe workers on the paved road.
  Evidence: `worker-002/prompt.txt` already teaches the batch helper path and explicitly says large heredoc scripts are unnecessary, yet the worker still switched to one.

- Observation: recipe supervision is now behind the richer workspace-worker contract already used by knowledge.
  Evidence: `cookimport/llm/codex_farm_knowledge_orchestrator.py::_build_strict_json_watchdog_callback(...)` accepts `execution_workspace_root`, tracks output stability, supports queue-based completion, and marks forbidden workspace-command deaths as `retryable=True`, while `cookimport/llm/codex_farm_orchestrator.py::_build_recipe_watchdog_callback(...)` still terminates forbidden-command and command-loop cases with `retryable=False` and no output-stabilization or task-queue checks.

- Observation: the exact failure shape is not covered by the current regression suite.
  Evidence: existing recipe watchdog tests in `tests/llm/test_codex_farm_orchestrator.py` cover bounded `python3 -c` transforms, but there is no recipe-specific regression for inline `python3 - <<'PY'` edits against `scratch/*.json` after batch preparation.

- Observation: helper UX still left at least one worker unsure where drafts landed.
  Evidence: `worker-002/events.jsonl` includes the message ``prepare-all` reported success but `scratch/` is empty on first check`, which immediately preceded the drift into a custom Python repair script.

- Observation: the boundary detector was not objecting to recipe logic, it was misreading Python syntax.
  Evidence: replaying the saved `worker-002` and `worker-005` heredoc commands through `detect_workspace_worker_boundary_violation(...)` showed `policy="forbidden_absolute_path"` because the old shell regex treated Python `Path('scratch') / name` joins and string literals like `3/4 cup` as shell-style absolute paths.

## Context and Orientation

In this repository, a "workspace worker" is a long-lived Codex session that runs inside a temporary worker root containing `in/`, `out/`, `scratch/`, `tools/`, `assigned_tasks.json`, and `OUTPUT_CONTRACT.md`. Recipe uses this mode in `cookimport/llm/codex_farm_orchestrator.py`. The mirrored helper script that workers call inside that root is rendered by `cookimport/llm/recipe_workspace_tools.py` and exposed as `tools/recipe_worker.py`.

The runtime enforces a path boundary and a coarse command policy in `cookimport/llm/codex_exec_runner.py`. The two key functions are `classify_workspace_worker_command(...)`, which records telemetry about whether a command looks tolerated, and `detect_workspace_worker_boundary_violation(...)`, which decides whether the command escapes the execution root or uses clearly forbidden executables. Those checks are consumed by recipe’s `_build_recipe_watchdog_callback(...)` in `cookimport/llm/codex_farm_orchestrator.py`. Right now recipe treats any forbidden workspace command as a non-retryable watchdog kill.

Recipe currently does a single main workspace-worker session, then validates each task output in `out/`, optionally launches a structured JSON repair attempt for invalid payloads, aggregates task results back into shard proposals, and writes final shard `status.json` files. That logic lives in the recipe worker execution block around the main `runner.run_workspace_worker(...)` call in `cookimport/llm/codex_farm_orchestrator.py`. Unlike knowledge and line-role, recipe does not currently perform task-level watchdog retries after a retryable workspace death, and it does not record recovered final supervision state distinct from the raw dead session state.

Knowledge is the best in-repo reference for the missing behavior. `cookimport/llm/codex_farm_knowledge_orchestrator.py` already has a richer watchdog callback, task-queue completion tracking, retryable watchdog kills, task-level retry follow-ups, and shard-summary normalization hooks. `docs/tasks/2026-03-21_18.01.51-normalize-line-role-recovered-watchdog-outcomes.md` explains why final supervision state must be normalized when recovery succeeds.

## Milestones

### Milestone 1: Reproduce the exact recipe kill at the workspace boundary

The first milestone codifies the real `2026-03-22_04.26.00` failure as tests, not guesses. The implementation should lift one or two representative commands from the saved `worker-002/events.jsonl` and `worker-005/events.jsonl` traces and turn them into narrow runner regressions. At the end of this milestone, the repo should have an executable proof of the current bad behavior and a bounded acceptance target: local Python heredoc edits that touch only workspace-local recipe drafts should be treated as allowed, while the same command shape still fails if it touches repo-external paths or forbidden tools.

### Milestone 2: Make recipe supervision recoverable and summary-correct

The second milestone upgrades recipe from the older strict callback contract to the newer workspace-worker recovery model. The main recipe worker session should be able to end in a retryable watchdog state when the runtime can safely recover, task-level follow-up work should repair missing outputs after that session, and recovered shards should be written and summarized as completed final outcomes instead of permanent watchdog kills. At the end of this milestone, a retryable main-session kill should no longer force the entire shard to fail if the owned task outputs can still be recovered and validated.

### Milestone 3: Make the helper surface the obvious cheaper path

The third milestone reduces the reasons a worker would reach for ad hoc scripting in the first place. The batch helper should make draft locations explicit, and it should expose a minimal bulk terminal-status affordance for the common `fragmentary` or `not_a_recipe` cases so the model does not need to write a Python loop just to stamp the same status across several prepared drafts. At the end of this milestone, the prompt plus helper contract should make the cheapest legal path also the easiest path to discover.

## Plan of Work

Start in `cookimport/llm/codex_exec_runner.py`. Extend `classify_workspace_worker_command(...)` and `detect_workspace_worker_boundary_violation(...)` so they can recognize the exact inline Python heredoc shape seen in the failing recipe workers when every visible path stays inside the execution root. The policy must remain narrow: allow reads and writes to workspace-local draft files such as `scratch/*.json`, task inputs under `in/`, helper scripts under `tools/`, and existing runtime control files already covered by `_WORKSPACE_ALLOWED_PATH_ROOTS`. Keep absolute paths outside the explicit execution root forbidden, keep output-path restrictions respected when the caller disables them, and keep obviously off-contract executables such as `pip`, `curl`, and `git` forbidden even if they appear inside a larger shell fragment. If command parsing cannot prove that the heredoc stays local, it must remain forbidden.

Then update the recipe watchdog and worker execution logic in `cookimport/llm/codex_farm_orchestrator.py`. `_build_recipe_watchdog_callback(...)` should accept the same kind of execution-root and output-progress context that knowledge already uses, so recipe can distinguish safe local workspace commands from real boundary escapes, can stop once all expected outputs are present and stable, and can mark recoverable watchdog deaths as `retryable=True` in workspace-worker mode. After the main session finishes, the recipe task loop should detect retryable watchdog deaths and perform task-level follow-up recovery before final shard aggregation. Reuse the knowledge and line-role patterns rather than inventing a third contract: write explicit `watchdog_retry_attempted`, `watchdog_retry_status`, and final-versus-raw supervision fields wherever recipe now writes shard proposals, shard `status.json`, worker runner payload rows, and worker failure summaries.

The follow-up recovery for recipe should stay minimal. Recipe already has a structured JSON repair path for invalid outputs via `_run_recipe_repair_attempt(...)`; that should remain the repair tool for schema or validation problems. The new retry path should exist only for watchdog-killed main sessions where a bounded follow-up can recover the missing or invalid task outputs. A recovered retry must change the task’s terminal state to a successful final outcome, while an unrecovered retry must preserve the watchdog reason as a genuine failure. When summary rows are emitted, populate the same final-supervision fields that `summarize_direct_telemetry_rows(...)` already knows how to respect from earlier line-role work.

Finally, adjust `cookimport/llm/recipe_workspace_tools.py` and the recipe worker prompt in `cookimport/llm/codex_farm_orchestrator.py`. `prepare-all` should write or print a deterministic manifest of the prepared draft paths so a worker can verify where the drafts landed without guesswork. Add one small bulk helper for the repetitive terminal-status case: a command that can stamp `fragmentary` or `not_a_recipe` plus a shared short reason across one or more prepared draft files under `scratch/` without requiring manual JSON surgery. Update the recipe prompt and any worker-manifest examples so they teach that flow first, but still allow bounded local Python edits when the recipe content genuinely needs richer transformations than the helper can express.
