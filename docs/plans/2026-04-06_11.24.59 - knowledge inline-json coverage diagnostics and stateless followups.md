# Knowledge Inline-JSON Coverage Diagnostics And Stateless Followups

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan is intentionally separate from `docs/plans/2026-04-06_11.14.02 - single-turn knowledge grouping per shard.md`. That earlier plan remains untouched as a predecessor focused on grouping fanout. This plan builds after it and covers the broader runtime bug the benchmark exposed: misleading repair diagnostics plus resumed-session token blowup in the inline-json path.

## Purpose / Big Picture

After this change, the knowledge inline-json runtime will be easier to reason about and dramatically cheaper on pathological shards. When a classification or grouping response skips owned rows or uses the wrong block indices, deterministic code will report the real problem directly instead of collapsing it into the vague `invalid_category` bucket. When inline-json needs classification repairs or grouping followups, those turns will reuse the same prepared workspace but will not reuse the prior chat transcript, so a small repair packet stays a small model call.

The visible proof is straightforward. A benchmark or stage run that previously showed `nonrecipe_finalize` exploding far above the selected shard count should now show explicit coverage/index failure metadata, and repeated followup turns should no longer show giant rising `input_tokens` caused by session history replay. The user-visible outcome is not merely prettier telemetry; it is lower token burn and a runtime story that matches what the repo is actually asking the model to do.

## Surprises & Discoveries

- Observation: most of the classification repairs in the motivating benchmark were not semantic disagreements.
  Evidence: the initial responses often used only `knowledge` and `other`, but skipped owned rows entirely. Examples from the 2026-04-06 benchmark root: `ks0000` returned `92` rows for `100` owned blocks, `ks0005` returned `92` for `100`, and `ks0006` returned `72` for `99`.

- Observation: one shard looked “complete” by row count yet still failed catastrophically because all returned block indices were foreign to the shard.
  Evidence: `saltfatacidheatcutdown.ks0008.nr` returned `99` rows for `99` owned blocks, but none of the response indices matched the packet’s owned block indices; they were `0..98` style local counters instead of the shard’s real block indices.

- Observation: the current parse/validation stack hides these failures behind `invalid_category`.
  Evidence: `build_knowledge_edited_task_file_from_classification_response(...)` silently drops foreign block indices and leaves skipped units untouched, then `validate_knowledge_classification_task_file(...)` sees untouched answers as blank `category` values and emits `invalid_category`.

- Observation: the expensive token growth comes from resumed chat history, not from the visible repair packets themselves.
  Evidence: one representative shard showed `grouping_1` at about `92k` input tokens, `grouping_2` at `157k`, `grouping_3` at `227k`, `grouping_4` at `303k`, and `grouping_5` at `381k` even though each grouping prompt stayed around twelve thousand characters.

- Observation: the prepared execution workspace and the chat transcript are separate concerns in the current runner, but the knowledge runtime currently ties them together.
  Evidence: `prepared_execution_working_dir` is tracked separately from `resume_last`, yet the knowledge inline-json runtime only reuses the workspace when it also resumes the previous Codex conversation.

- Observation: the new response-contract metadata should stay silent on the happy path.
  Evidence: after adding explicit coverage/index diagnostics, `tests/llm/test_knowledge_grouping_contract.py::test_structured_grouping_response_accepts_group_index_alias` failed because the parser returned empty metadata lists even when there were no contract errors. Returning `{}` on the success path preserved the old clean contract while keeping detailed diagnostics for real failures.

## Decision Log

- Decision: introduce explicit coverage/index diagnostics in the knowledge parse helpers rather than relying on downstream validator side effects.
  Rationale: “missing owned rows” and “foreign block indices” are materially different failures from “answer category not in the allowed enum,” and the repair packets should say so directly.
  Date/Author: 2026-04-06 / Codex

- Decision: keep followup turns in the same prepared workspace but stop resuming the model transcript for inline-json repairs and grouping calls.
  Rationale: the workspace reuse is still useful for artifact continuity and mirrored paths, but the transcript reuse is what makes token usage snowball.
  Date/Author: 2026-04-06 / Codex

## Context and Orientation

The knowledge stage is the final non-recipe LLM pass. It takes already-routed non-recipe candidate blocks and decides whether each owned block is reusable cooking knowledge or just other text, then groups the retained knowledge blocks into idea groups. The inline-json path for that stage lives in `cookimport/llm/knowledge_stage/workspace_run.py`. “Inline-json” here means the repo writes a compact JSON packet, asks Codex for JSON only, and then deterministically validates the reply.

Two files define the contract that matters for this bug. `cookimport/llm/knowledge_stage/structured_session_contract.py` turns a task file into a packet/prompt pair and parses model JSON back into task-file answers. `cookimport/llm/knowledge_stage/task_file_contracts.py` validates those answers and decides whether the shard is complete, needs repair, or should advance into grouping.

The subprocess seam is `cookimport/llm/codex_exec_runner.py`. That runner has two distinct ideas: the prepared execution workspace, which is the mirrored on-disk directory Codex runs inside, and the resumed chat transcript, which is enabled by `resume_last=True`. The current knowledge runtime conflates them by reusing the same prepared workspace only when it also resumes the prior transcript.

Tests relevant to this change already exist in non-dirty files:

- `tests/llm/test_knowledge_block_classification_contract.py` for validator behavior
- `tests/llm/test_codex_exec_runner.py` and `tests/llm/test_codex_exec_runner_taskfile.py` for runner transport behavior
- `tests/llm/test_prompt_budget_runtime.py` for knowledge-stage prompt-budget aggregation

The repo currently has local edits in `tests/llm/test_knowledge_grouping_contract.py`, `tests/llm/test_knowledge_orchestrator_runtime_leasing.py`, and `tests/llm/test_knowledge_same_session_handoff.py`. This plan should avoid overwriting that work unless later implementation proves it is unavoidable.

## Plan of Work

First, patch `cookimport/llm/knowledge_stage/structured_session_contract.py` so the classification and grouping response parsers report exact coverage and index problems. The parser should continue to reject malformed JSON outright, but when the JSON object is structurally valid it should distinguish between these cases:

- owned rows missing from the response,
- duplicate block indices in the response,
- block indices in the response that do not belong to the current task file.

The returned metadata must name the failed units or block indices so repair mode can narrow scope without pretending the problem is merely an invalid enum value.

Second, patch `cookimport/llm/knowledge_stage/task_file_contracts.py` so blank category values left behind by omitted answers no longer surface as generic `invalid_category`. The validator should keep `invalid_category` for truly unsupported explicit category strings, but untouched or missing answers should produce a distinct failure code and preserve precise `failed_unit_ids`.

Third, patch `cookimport/llm/codex_exec_runner.py` so `run_packet_worker(...)` can reuse an already prepared execution workspace without requiring `resume_last=True`. This is the key transport change: the same mirror directory can stay in use across followups while each inline-json call starts a fresh Codex conversation.

Fourth, patch `cookimport/llm/knowledge_stage/workspace_run.py` to use that new stateless followup mode. The initial classification call may still prepare the workspace, but classification repairs, grouping calls, and grouping repairs should pass the prepared workspace path without `resume_last=True`. The artifact naming, session-lineage recording, and prompt-budget telemetry should remain intact, but the model transcript should stop accumulating across turns.

Fifth, update the smallest relevant docs. `docs/10-llm/10-llm_README.md` and `cookimport/llm/knowledge_stage/README.md` should reflect that knowledge inline-json followups are bounded fresh calls in the same workspace and that coverage/index drift is validated explicitly.

Finally, add focused proof. The tests should demonstrate the new explicit diagnostics and the new workspace-reuse-without-resume behavior. Where existing prompt-budget tests already cover worker-report fallback, extend them only if the runtime shape changes enough to require stronger assertions.

## Validation and Acceptance

The change is correct when all of the following are true:

1. A classification response that omits owned rows fails with an explicit coverage error and names the missing units or block indices.
2. A classification response that uses foreign block indices fails with an explicit block-index mismatch error rather than silently dropping rows and later reporting `invalid_category`.
3. Inline-json classification repairs and grouping followups can reuse the prepared execution workspace without `resume_last=True`.
4. Focused tests pass in the project virtualenv.

The behavior should be human-verifiable from artifacts too: later inline-json followup turns should no longer show rising input-token totals just because earlier turns existed in the same shard session.
