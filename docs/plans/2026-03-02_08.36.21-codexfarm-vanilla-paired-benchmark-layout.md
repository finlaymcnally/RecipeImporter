# Add Paired CodexFarm+Vanilla Single-Offline Benchmark Layout

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

`docs/PLANS.md` is checked into this repository and this document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

When you run interactive single-offline benchmark with CodexFarm enabled, you want one CodexFarm pass and one vanilla pass saved together in one obvious place so comparison is immediate and cleanup is not a directory scavenger hunt.

After this change, one interactive benchmark timestamp folder will contain both runs side-by-side (`codexfarm` and `vanilla`) plus a small comparison summary file. You will be able to open one folder and immediately inspect both eval reports and metric deltas.

  Rationale: It keeps one benchmark session self-contained and comparison-friendly.
  Date/Author: 2026-03-02 / Codex

- Decision: In codex-enabled paired mode, execute `vanilla` first and then execute `codexfarm`; if `codexfarm` fails, preserve successful `vanilla` artifacts and continue to completion without deleting successful outputs.

  Date/Author: 2026-03-02 / Codex + User

  Rationale: Prevents drift and ambiguity in what “comparison summary” contains.

## Context and Orientation

Current behavior for interactive benchmark single-offline mode is in `cookimport/cli.py` inside `_interactive_mode()` under the `action == "labelstudio_benchmark"` branch.

`choose_run_settings(...)` returns one `RunSettings` object. That object may have CodexFarm enabled (`llm_recipe_pipeline=codex-farm-3pass-v1`) or disabled (`off`). The current code builds one kwargs payload via `build_benchmark_call_kwargs_from_run_settings(...)` and calls `labelstudio_benchmark(...)` once.

Benchmark artifacts currently write directly under one eval root (for example `data/golden/benchmark-vs-golden/2026-03-02_01.11.21/`) with `eval_report.json`, `run_manifest.json`, and `prediction-run/` in the same directory.

This plan introduces a “benchmark session root” for interactive single-offline mode and stores variant runs under it:

- `data/golden/benchmark-vs-golden/<timestamp>/single-offline-benchmark/codexfarm/...`
- `data/golden/benchmark-vs-golden/<timestamp>/single-offline-benchmark/vanilla/...`

When CodexFarm is not enabled, only `vanilla/` is produced.

## Plan of Work

Milestone 1 introduces a small single-offline orchestration layer in `cookimport/cli.py` so interactive mode can execute one or two benchmark variants without duplicating command wiring. The orchestration derives a vanilla settings payload by copying the selected run settings and forcing `llm_recipe_pipeline=off` (allowing `RunSettings.from_dict(...)` to normalize away codex-only optional fields). For codex-enabled runs, execution order is fixed to `vanilla` then `codexfarm`, and failure handling is per variant so vanilla success is preserved even if codex fails.

Milestone 2 changes interactive single-offline run path construction. Instead of writing directly into `<timestamp>/`, interactive single-offline will write into `<timestamp>/single-offline-benchmark/<variant>/`. The variant names are exactly `codexfarm` and `vanilla`.

Milestone 3 writes a compact comparison artifact at `<timestamp>/single-offline-benchmark/codex_vs_vanilla_comparison.{json,md}` only when both variants exist and both evaluations complete successfully. The JSON contract is fixed:

- `schema_version` (`codex_vs_vanilla_comparison.v1`)
- `run_timestamp`
- `source_file`
- `variants.codexfarm.eval_output_dir`
- `variants.vanilla.eval_output_dir`
- `metrics.codexfarm.{precision,recall,f1,practical_precision,practical_recall,practical_f1}`
- `metrics.vanilla.{precision,recall,f1,practical_precision,practical_recall,practical_f1}`
- `deltas.codex_minus_vanilla.{precision,recall,f1,practical_precision,practical_recall,practical_f1}`

Missing metrics are represented as `null` and delta is `null` whenever either side is `null`.

Milestone 4 updates tests and docs. Tests should verify that codex-enabled single-offline mode calls benchmark twice with two eval roots, codex-disabled mode calls once, and the new folder layout is respected. Tests should also verify per-variant `llm_recipe_pipeline` values (`off` for vanilla, `codex-farm-3pass-v1` for codexfarm), run-order/failure handling semantics, and exact comparison-artifact schema/values. Docs should describe the new structure and comparison artifact so operators know exactly where to look.

## Validation and Acceptance

Acceptance is behavior-based:

1. Interactive benchmark, single-offline, CodexFarm enabled:
   - one timestamp root is created,
   - both `single-offline-benchmark/codexfarm/` and `single-offline-benchmark/vanilla/` exist,
   - each variant has its own `eval_report.json`, `run_manifest.json`, and `prediction-run/`,
   - `vanilla` run executes before `codexfarm` run.

2. Same run writes `single-offline-benchmark/codex_vs_vanilla_comparison.json` and `.md` with schema `codex_vs_vanilla_comparison.v1` and `codexfarm - vanilla` metric deltas.

3. Interactive benchmark, single-offline, CodexFarm disabled:
   - only `single-offline-benchmark/vanilla/` is created,
   - no comparison file is written.

4. Interactive benchmark, single-offline, CodexFarm enabled, codex variant fails:
   - `single-offline-benchmark/vanilla/` remains intact and contains normal benchmark artifacts,
   - codex failure is surfaced clearly,
   - no comparison file is written.

5. Dashboard collector still ingests both variant eval reports from nested paths (no collector regression).

6. Existing all-method and single-profile flows remain unchanged.

## Idempotence and Recovery

The new orchestration writes into a timestamped root and therefore does not mutate prior runs. Re-running the same interactive action simply produces a new timestamp folder. If one variant fails, preserve any completed variant output and emit a clear message that comparison summary was skipped because both successful variants are required.
