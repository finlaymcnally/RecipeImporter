# Sandbox Parallel Fallbacks for Stage and Label Studio Split Conversion

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan follows [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md) requirements and must be maintained in accordance with that file.

## Purpose / Big Picture

After this change, sandboxed environments that cannot start `ProcessPoolExecutor` workers will no longer force stage and split-conversion flows into serial loops by default. Instead, those flows will continue to run multiple jobs concurrently through a thread-backed fallback, with serial execution kept only as a last-resort safety path. The user-visible result is that commands like `cookimport stage` and Label Studio import split conversion keep parallel throughput even when `/dev/shm` or multiprocessing semaphore setup is blocked.

The behavior is observable in two ways: console status text will report a parallel fallback mode instead of serial fallback, and multi-file or multi-split workloads will continue to process multiple tasks without printing “running jobs serially” as the first fallback.

## Surprises & Discoveries

- Observation: This sandbox currently blocks multiprocessing shared-memory usage in ways that break process-pool startup immediately.
  Evidence: runtime probes showed `touch /dev/shm/<probe>` fails with `Permission denied`, and `ProcessPoolExecutor(max_workers=1)` fails with `PermissionError: [Errno 13] Permission denied`.

- Observation: QualitySuite already has a mitigation for this class of failure at experiment fanout level, but stage and Label Studio split conversion do not.
  Evidence: `cookimport/bench/quality_runner.py` resolves to subprocess mode on probe failure, while `cookimport/cli.py` stage and `cookimport/labelstudio/ingest.py` split conversion still route process-pool `PermissionError` directly to serial loops.

- Observation: All-method benchmark code already demonstrates the desired fallback shape (`process -> thread -> serial`) and can be used as an in-repo pattern.
  Evidence: fallback messaging and logic in `cookimport/cli.py` all-method helpers use thread-backed concurrency when process workers are unavailable, reserving serial for thread-executor failure.

## Decision Log

- Decision: Fix the broader issue by eliminating hard serial-first fallback in stage and Label Studio split conversion, using the existing all-method fallback style (`process -> thread -> serial`) as the immediate target behavior.
  Rationale: this is the smallest safe change that directly addresses user-visible forced-serial regressions while keeping code paths understandable and testable.

- Decision: Keep serial fallback as last resort only, rather than removing it completely.
  Rationale: serial execution is a reliability safety net when both process and thread executor setup fail due host limits or runtime corruption.

## Context and Orientation

The relevant runtime surfaces are:

- `cookimport/cli.py` `stage(...)`: main stage command orchestration. It currently builds jobs and attempts a `ProcessPoolExecutor` for job execution. On `PermissionError`, it prints a warning and runs jobs serially.
- `cookimport/labelstudio/ingest.py` `run_labelstudio_import(...)`: split conversion path for Label Studio import. It currently attempts a `ProcessPoolExecutor` for split jobs and, on `PermissionError`, executes split jobs serially.
- `cookimport/cli.py` all-method benchmark scheduler helpers: these already implement the desired fallback behavior (process first, then thread fallback, then serial only if thread init fails).
- `tests/ingestion/test_performance_features.py`: stage behavior tests currently validate successful output generation but do not lock fallback-mode behavior.
- `tests/labelstudio/test_labelstudio_ingest_parallel.py`: split conversion tests exist but do not currently verify process-pool `PermissionError` fallback behavior.

Terms used in this plan:

- “process pool”: Python `ProcessPoolExecutor`, which runs tasks in child Python processes and depends on multiprocessing primitives that can fail in sandboxed `/dev/shm`-restricted environments.
- “thread fallback”: Python `ThreadPoolExecutor` used when process-pool startup fails; this keeps concurrent scheduling inside one process.
- “serial fallback”: executing one job at a time in a simple loop. This remains only as last-resort recovery.

## Plan of Work

### Milestone 1: Shared fallback resolution and messaging

Introduce a small shared helper in `cookimport/cli.py` (or a narrowly scoped runtime helper module if code cleanliness demands it) that resolves executor backend for stage-like job fanout. The helper will consume worker count and process-pool probe result, then return backend (`process` or `thread`) and a reason string. The helper must mirror existing all-method behavior so fallback semantics stay consistent across modules.

At the end of this milestone, stage and Label Studio code will be able to call one resolver and emit clear fallback messages without duplicating resolution logic.

### Milestone 2: Stage path `process -> thread -> serial`

Update `cookimport/cli.py` stage job execution block so process-pool startup failure no longer jumps directly to serial. Instead:

1. attempt `ProcessPoolExecutor`,
2. on `PermissionError` or `OSError`, emit warning and attempt `ThreadPoolExecutor` with the same worker limit,
3. only if thread executor setup fails, emit warning and use the existing serial loop.

Preserve current result handling, progress updates, and artifact generation. Do not change output contracts.

At the end of this milestone, `cookimport stage` keeps parallel fanout in sandbox-denied process environments.

### Milestone 3: Label Studio split-convert path `process -> thread -> serial`

Update `cookimport/labelstudio/ingest.py` split conversion fanout similarly:

1. attempt `ProcessPoolExecutor`,
2. on process-pool startup failure, switch to `ThreadPoolExecutor`,
3. keep current serial loop only if thread executor cannot start.

Preserve split progress callback semantics (`task X/Y`) and worker activity messaging so dashboards remain coherent.

At the end of this milestone, split EPUB/PDF conversion in Label Studio import no longer defaults to serial under process-pool denial.

### Milestone 4: Tests, docs, and performance proof

Add targeted tests that fail before and pass after:

- stage fallback regression test in `tests/ingestion/test_performance_features.py` (or a dedicated stage-fallback test module if cleaner) that simulates process-pool startup failure and asserts thread-backed path is taken (and serial warning is not the first fallback behavior).
- split-convert fallback regression test in `tests/labelstudio/test_labelstudio_ingest_parallel.py` that simulates process-pool startup failure and asserts conversion still runs through parallel worker submission path via thread executor.

Update docs:

- `docs/02-cli/02-cli_README.md`: stage fallback behavior wording.

- `docs/06-label-studio/06-label-studio_README.md`: split conversion fallback order and sandbox troubleshooting notes.
