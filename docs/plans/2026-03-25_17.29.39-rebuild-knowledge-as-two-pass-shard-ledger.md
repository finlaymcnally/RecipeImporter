## Purpose / Big Picture

After this runtime, a user can change `knowledge_prompt_target_count` and see the number of planned knowledge shards change without losing packet truth. The knowledge planner still writes one immutable packet input per packet, the worker runtime still executes one task per packet, and the shard layer is the scheduling bundle that can own one or more packets.

The result is visible in both runtime artifacts and staged outputs. Under `raw/llm/<workbook_slug>/knowledge/`, packet files remain one-per-packet under `in/`, shard manifests can bundle several packet ids into one shard entry, worker roots stay batch-first with `CURRENT_BATCH.md`, and validated task outputs are promoted into `09_nonrecipe_authority.json`, `09_nonrecipe_knowledge_groups.json`, `09_nonrecipe_review_status.json`, and reviewer-facing `knowledge/<workbook_slug>/snippets.jsonl` plus `knowledge.md`.

## Surprises & Discoveries

- Observation: the planner/runtime split is narrower than it first looks. The planner does not need a second execution model for bundled shards because the worker planner already knows how to split a shard payload with `sid + p[]` back into one task per packet.
  Evidence: `cookimport/llm/codex_farm_knowledge_jobs.py` emits `KnowledgeShardJobInputV1` for multi-packet shards, and the live knowledge runtime still writes one task-local `in/<task_id>.json` and one `out/<task_id>.json` per packet.

- Observation: knowledge deliberately did not follow canonical line-role into a `CURRENT_PHASE*` control surface.
  Evidence: the shared direct-exec runner supports both patterns, but the live docs and worker tools still reserve `CURRENT_PHASE*` for line-role while knowledge uses `CURRENT_BATCH*` and fallback `CURRENT_TASK*`.

- Observation: snippets are still a real part of the live knowledge contract even though they are not authority.
  Evidence: focused ingest tests still validate snippet evidence shape and out-of-surface failures, and staged outputs still document `knowledge/<workbook_slug>/snippets.jsonl` as the reviewer-facing evidence artifact.

- Observation: the durable operator story is shard count, but the durable semantic answer unit is still the packet task.
  Evidence: `knowledge_prompt_target_count` changes planned shard count, while `task_manifest.jsonl`, worker-local `assigned_tasks.json`, progress counters, validator inputs, and accepted outputs all stay packet-owned.

## Outcomes & Retrospective

The final runtime is consistent and teachable. Knowledge planning is packet-native, shard scheduling is real, workspace execution is batch-first, validation is packet-owned, and staged authority is explicit. Lowering `knowledge_prompt_target_count` now changes shard bundles while preserving immutable packet files and one task per packet. Raising `knowledge_worker_count` changes concurrency, not packet sizing.

The most important lesson is that the line-role and knowledge phases share transport infrastructure without sharing the same worker-control surface. Line-role is the ledger-first phase in this repository. Knowledge is the packet-and-batch phase. Reusing the same direct-exec runner does not mean forcing both stages into the same human-facing workspace vocabulary.

The second lesson is that reviewer evidence and final authority should remain separate. The knowledge stage can keep snippets, grouped ideas, and markdown summaries for inspection while still scoring only the explicit Stage 9 authority and review-status artifacts.

## Context and Orientation

The knowledge stage is the optional LLM-backed non-recipe review phase. It receives the review-eligible outside-recipe block queue from `cookimport/staging/nonrecipe_stage.py` after deterministic routing has already removed obvious exclusions. The stage does not decide recipe boundaries, and it does not own the deterministic seed-routing artifact. Its job is narrower: review the remaining non-recipe packet inputs, decide which owned blocks are durable cooking knowledge versus `other`, group the kept blocks into related ideas inside each packet, and then promote those accepted packet results into final Stage 9 artifacts.

The key terms in this repository are precise:

- A `packet` is one immutable model-facing knowledge-review payload. It is stored under `raw/llm/<workbook_slug>/knowledge/in/<packet_id>.json`. A packet owns one ordered block slice plus optional small context or recipe-guardrail fields.
- A `task` is one file-backed work item executed by a worker. In the live knowledge runtime, there is one task per packet. Tasks write one local result file under `workers/<worker_id>/out/<task_id>.json`.
- A `shard` is the planning and scheduling bundle. A shard may own one packet or several contiguous packets. The shard is what appears in `shard_manifest.jsonl`.
- A `worker` is one long-lived Codex session assigned one or more shards. A worker can process many packet tasks across those shards in the same workspace session.
- A `batch` is the repo-owned current slice of consecutive packet tasks shown to the worker through `current_batch.json`, `CURRENT_BATCH.md`, and `CURRENT_BATCH_FEEDBACK.md`.
- A `proposal` is the repo-validated shard-level payload written under `knowledge/proposals/` after packet task outputs are checked and normalized.

The current implementation lives in these files:

- `cookimport/llm/codex_farm_knowledge_jobs.py` builds packet payloads, partitions them into shard bundles, and reports shard metadata such as `owned_packet_ids`, `owned_block_indices`, `packet_count`, and total `char_count`.
- `cookimport/llm/knowledge_stage/planning.py` turns shard payloads into task plans and keeps the one-task-per-packet runtime contract.
- `cookimport/llm/knowledge_stage/runtime.py` is the public stage runtime that builds packet jobs, resolves worker count, launches the direct worker runtime, collects validated outputs, and refines staged non-recipe authority.
- `cookimport/llm/knowledge_stage/workspace_run.py` prepares worker roots, writes `assigned_tasks.json`, `in/*.json`, `hints/*.md`, `out/*.json`, prompts, and batch sidecars, then supervises one long-lived workspace-worker session per worker assignment.
- `cookimport/llm/knowledge_stage/recovery.py` owns the knowledge worker prompt, watchdog and retry rules, current-batch guidance, and helper-language expectations.
- `cookimport/llm/knowledge_workspace_tools.py` owns the repo-written helper CLI and the batch/task sidecars used inside the worker workspace.
- `cookimport/llm/codex_farm_knowledge_ingest.py` normalizes worker outputs, validates exact owned block coverage, validates packet-local idea-group coverage, and rejects off-surface snippet evidence or low-trust semantic packet shapes.
- `cookimport/llm/codex_farm_knowledge_writer.py`, `cookimport/llm/knowledge_stage/promotion.py`, and `cookimport/llm/knowledge_stage/reporting.py` convert validated knowledge proposals into final Stage 9 artifacts, reviewer-facing snippet outputs, markdown summaries, and runtime summaries.
- `docs/10-llm/10-llm_README.md` and `docs/05-staging/05-staging_readme.md` are the owning docs for the live runtime and staged outputs.

The packet input contract is compact on purpose. Each packet file under `knowledge/in/*.json` contains a top-level packet id and one ordered block list. Multi-packet shard payloads wrap those packet payloads under `sid + p[]`. The worker output contract is packet-owned too: the worker writes top-level `packet_id`, ordered `block_decisions`, and packet-local `idea_groups`. Repo code normalizes accepted payloads into the compact canonical form used by proposals and promotion.

The final authority split is also explicit:

- `08_nonrecipe_seed_routing.json` remains the deterministic review-queue artifact.
- `09_nonrecipe_authority.json` is the final machine-readable truth for `knowledge` versus `other`.
- `09_nonrecipe_knowledge_groups.json` stores the promoted model-authored related-idea groups.
- `09_nonrecipe_review_status.json` stores review completeness and unresolved or skipped review status.
- `knowledge/<workbook_slug>/snippets.jsonl` and `knowledge.md` remain reviewer-facing evidence only.

## Milestones

### Milestone 1: Packet planning and truthful shard bundling

The first milestone is the planner seam. `cookimport/llm/codex_farm_knowledge_jobs.py` must write one immutable packet file per packet under `knowledge/in/*.json`, then partition those packets into contiguous shard bundles according to `knowledge_prompt_target_count`. A shard may own one packet or many packets, but packet files remain unchanged. The observable proof is that `KnowledgeJobBuildReport.packet_ids` stays packet-level while `KnowledgeJobBuildReport.shard_entries` can own several packet ids.

Run the focused packet-planning tests after touching this seam. Acceptance is that lowering `knowledge_prompt_target_count` lowers shard count and creates shard payloads with `sid + p[]` while preserving one packet file per packet.

### Milestone 2: Batch-first workspace execution and packet validation

The second milestone is the worker-runtime seam. `cookimport/llm/knowledge_stage/workspace_run.py`, `recovery.py`, and `knowledge_workspace_tools.py` must preserve the batch-first knowledge worker contract. A worker root should expose `assigned_tasks.json`, `current_batch.json`, `CURRENT_BATCH.md`, `CURRENT_BATCH_FEEDBACK.md`, authoritative `in/*.json`, harvested `out/*.json`, and fallback `CURRENT_TASK*` files. The worker should process packet tasks through one long-lived workspace session, while repo code validates outputs, advances the batch, and relaunches only when necessary.

Run the focused workspace and ingest tests after touching this seam. Acceptance is that one worker assignment can process several packet tasks in one session, progress remains task-truthful, and malformed or off-surface outputs fail closed at packet scope rather than silently promoting.

### Milestone 3: Promotion, staged authority, and reviewer artifacts

The third milestone is the promotion seam. `promotion.py`, `reporting.py`, `codex_farm_knowledge_writer.py`, and the staging integration in `runtime.py` must preserve the explicit split between Stage 9 authority and reviewer evidence. Packet results should promote into final authority, groups, and review status, while snippets remain reviewer-facing evidence and are not the scoreable truth surface.

Run the focused staging and prompt-preview tests after touching this seam. Acceptance is that the final Stage 9 artifacts still match the packet-owned runtime contract, preview still reconstructs packet/shard planning without live tokens, and docs/tests keep teaching the same packet-bundle story.

## Plan of Work

Start with `cookimport/llm/codex_farm_knowledge_jobs.py`. Preserve the current packet floor and contiguous planning rules. The planner must continue to build one immutable packet payload per packet, write that file under `knowledge/in/`, and then group packet payloads into shard bundles only at the `ShardManifestEntryV1` layer. When touching shard metadata, keep it shard-native: use fields such as `packet_count`, `owned_packet_ids`, `owned_block_indices`, `owned_block_count`, and total `char_count`. Do not reintroduce misleading packet-only names for multi-packet shards, and do not collapse several packets into one lossy task.

Then check `cookimport/llm/knowledge_stage/planning.py`, `runtime.py`, and `workspace_run.py`. Preserve the one-task-per-packet runtime contract. The runtime should still build knowledge jobs from the review-eligible non-recipe span queue, launch workspace workers against shard entries, and let the worker planner split multi-packet shard payloads back into packet-local tasks. The worker root should remain batch-first: `assigned_tasks.json` is fallback queue inventory, `CURRENT_BATCH.md` is the cheap first read, `current_batch.json` holds batch-local machine-readable paths, and `CURRENT_TASK*` remains a narrower fallback seam for the first active task.

Then check `cookimport/llm/knowledge_stage/recovery.py` and `cookimport/llm/knowledge_workspace_tools.py`. Preserve the current helper and guardrail story. The worker prompt should still tell the worker to read repo-written local files first, avoid inventing its own queue controller, and continue through later batch tasks without asking conversational permission. The helper CLI should continue to validate packet-owned drafts, install only accepted outputs, and rewrite the batch sidecars after progress so the same long-lived worker session can continue immediately.

After that, check `cookimport/llm/codex_farm_knowledge_ingest.py`. Preserve both structural and semantic validation. Structural validation means exact owned block coverage, exact owned packet coverage for bundled shards, and exact packet-local idea-group coverage for kept knowledge blocks. Semantic validation means snippet evidence must stay inside the owned packet surface, snippet bodies cannot substantially copy evidence text, and full-packet echoes or non-grounded snippet bodies fail closed. Keep deterministic salvage narrow.

Finally, check `cookimport/llm/knowledge_stage/promotion.py`, `cookimport/llm/codex_farm_knowledge_writer.py`, `cookimport/llm/knowledge_stage/reporting.py`, `docs/10-llm/10-llm_README.md`, and `docs/05-staging/05-staging_readme.md`. Preserve the explicit authority split. `09_nonrecipe_authority.json`, `09_nonrecipe_knowledge_groups.json`, and `09_nonrecipe_review_status.json` are the scored final artifacts. `knowledge/<workbook_slug>/snippets.jsonl` and `knowledge.md` remain reviewer/debug evidence only. If a doc or test starts teaching that snippets are authority, or that knowledge is now a row-ledger phase with `CURRENT_PHASE*`, that drift should be corrected immediately.

- Lowering `knowledge_prompt_target_count` lowers the number of shard entries when packet count is higher than the requested shard count.
- Multi-packet shards use `sid + p[]`, and shard metadata reports shard-native totals such as `packet_count`, `owned_block_count`, and total `char_count`.

- One long-lived workspace worker can process many packet tasks without requiring a new conversational prompt between tasks.
- Progress and status accounting remain task-truthful rather than pretending one shard equals one semantic answer.

Validation acceptance:

- Each accepted task output covers exactly the owned packet blocks.
- For bundled shards, accepted proposals cover exactly the owned packet ids.
- Kept `knowledge` blocks are covered exactly once by packet-local idea groups.
- Snippet evidence never cites blocks outside the owned packet surface.
- Low-trust semantic packet shapes fail closed instead of silently promoting.

Artifact acceptance:

- `09_nonrecipe_authority.json` remains the final truth surface for `knowledge` versus `other`.
- `09_nonrecipe_knowledge_groups.json` remains the promoted related-idea artifact.
- `09_nonrecipe_review_status.json` remains the explicit incompleteness and review-status artifact.

Documentation acceptance:

- Docs and tests do not teach that the live knowledge phase is a two-pass row ledger or that it uses `CURRENT_PHASE*` as its primary worker surface.

## Idempotence and Recovery

This runtime is source-only and artifact-rebuild-safe. Re-running the planner rewrites the packet input directory from scratch. Re-running the worker runtime against the same run root rewrites worker-local runtime artifacts under that stage root. Re-running the focused tests is safe and is the preferred recovery step after any edit to planning, worker tooling, validation, or promotion.

If a change introduces uncertainty about packet-versus-shard ownership, recover by re-checking these three seams in order:

1. `cookimport/llm/codex_farm_knowledge_jobs.py` for immutable packet files and truthful shard metadata.
2. `cookimport/llm/knowledge_stage/workspace_run.py` plus `knowledge_workspace_tools.py` for batch-first packet-task execution.
3. `cookimport/llm/codex_farm_knowledge_ingest.py` plus `knowledge_stage/promotion.py` for exact packet coverage and final Stage 9 artifact ownership.

Do not recover drift by flattening bundled shards back into one shard-equals-one-task assumption or by moving knowledge onto the line-role `CURRENT_PHASE*` surface. Both would misdescribe the shipped runtime.
