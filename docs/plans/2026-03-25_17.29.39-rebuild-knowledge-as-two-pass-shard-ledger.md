# Rebuild Knowledge As Two-Pass Shard Ledger

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

After this plan lands, the knowledge stage should behave like a shard-owned ledger runtime instead of a packet-task queue with snippets and evidence. Choosing `knowledge_prompt_target_count=N` should partition the ordered non-recipe review queue into roughly `N` contiguous shard owners. Each worker should own one shard through two repo-owned phases inside the same long-lived Codex session:

- `pass1_classify_rows`: one row per owned block, final decision is only `knowledge` or `other`.
- `pass2_group_kept_rows`: runs only after accepted Pass 1 output and assigns `group_id` and `topic_label` to kept `knowledge` rows.

The proof should be visible from the artifact tree. Worker roots should center one `CURRENT_PHASE.md` control surface, one shard-owned input ledger for the active phase, one work ledger, and repair files only when unresolved rows exist. Final artifacts should show `knowledge` authority and group labels without snippets, evidence files, or reviewer-reason outputs.

## Outcomes & Retrospective

This plan is ready to implement but not yet implemented. The intended end state is stable: knowledge becomes a shard-owned two-phase ledger runtime with one shared control plane, same-session phase handoff, row-local repair, and final artifacts derived from accepted ledgers rather than packet outputs and snippet evidence.

The most important design lesson is that knowledge and line-role should share runtime vocabulary but not identical data contracts. The control plane should be shared. The semantic contract should stay stage-specific: line-role is one phase; knowledge is two.

## Context and Orientation

The knowledge stage is the last optional LLM-backed stage in the import pipeline. It reviews non-recipe blocks that survived deterministic routing and decides which ones are durable cookbook knowledge. Today it still behaves like a packet-task system: planning produces packet-oriented work items, workers emit packet outputs with `block_decisions`, `idea_groups`, and snippet/evidence material, validators inspect those packet payloads, and reporting/writer code promotes those results into final staged artifacts.

The current code relevant to this redesign lives in these files:

- `cookimport/llm/codex_farm_knowledge_jobs.py` builds review packets and shard manifests from the ordered non-recipe review queue.
- `cookimport/llm/knowledge_stage/runtime.py` is the public stage runtime and artifact coordinator.
- `cookimport/llm/knowledge_stage/workspace_run.py` writes worker roots and supervises long-lived Codex workspace sessions.
- `cookimport/llm/knowledge_stage/recovery.py` owns worker prompts, helper guidance, validation-driven retry decisions, and runtime guardrails.
- `cookimport/llm/codex_farm_knowledge_models.py` and `cookimport/llm/codex_farm_knowledge_ingest.py` define and validate the current packet-oriented result schema.
- `cookimport/llm/knowledge_stage/promotion.py`, `reporting.py`, and `cookimport/llm/codex_farm_knowledge_writer.py` convert validated outputs into final authority, summaries, and reviewer-facing artifacts.
- `docs/10-llm/10-llm_README.md` and `docs/05-staging/05-staging_readme.md` describe the live knowledge contract and must be updated during implementation.

Terms used in this plan:

- A `shard` is one contiguous slice of the review-eligible non-recipe block queue. It is the planning and concurrency unit.
- A `worker` is one long-lived Codex session assigned one or more shards.
- A `ledger` is a machine-readable spreadsheet-like file with one owned row per block plus the columns the worker fills in.
- A `phase` is the repo-owned current step for a worker. Knowledge has two real phases: `pass1_classify_rows` and `pass2_group_kept_rows`.
- A `repair request` is a repo-owned file naming only unresolved rows plus the minimum local context needed to fix them. It is not a public task layer.
- A `phase handoff` is the repo-owned transition from accepted Pass 1 output to generated Pass 2 input inside the same worker root.

The architecture problem is not shard planning. Shards are already the right public knob. The problem is that the knowledge runtime still hides a packet-task queue under those shards and asks workers to satisfy a broader packet contract than the desired product needs. This plan replaces that with one shard-owned loop and one shared control plane:

- open the current phase sidecars,
- edit the active phase work ledger,
- run cheap local validation,
- keep accepted rows frozen,
- repair only unresolved rows,
- run `install-phase` to checkpoint the phase,
- let the repo advance the worker from Pass 1 to Pass 2 when Pass 1 is accepted.

## Milestones

### Milestone 1: Replace packet-task planning with shard-owned phase ledgers

This milestone removes the packet-task mental model from knowledge planning and worker setup. At the end of the milestone, each shard owns one Pass 1 input ledger, one Pass 1 work ledger, and one shared current-phase control surface. The worker root no longer centers `CURRENT_BATCH.md`, `current_batch.json`, or `assigned_tasks.json` on the happy path.

Run focused knowledge planning/workspace tests and inspect one fake worker root after this milestone. Acceptance is that the worker root shows `CURRENT_PHASE.md`, `CURRENT_PHASE_FEEDBACK.md`, `current_phase.json`, one Pass 1 input ledger, and one Pass 1 work ledger.

### Milestone 2: Implement Pass 1 validation, accepted-row freezing, and targeted repair

This milestone replaces packet-result validation with shard-row validation for Pass 1. At the end of the milestone, the validator can read one Pass 1 work ledger, accept valid rows incrementally, freeze those rows, list unresolved rows, and write one repair request naming only unresolved row ids plus minimal context.

Run focused ingest/validation tests after this milestone. Acceptance is that a synthetic Pass 1 shard with 200 rows and three bad rows yields a repair request naming only those three rows while accepted rows stay frozen.

### Milestone 3: Add same-session Pass 1 to Pass 2 handoff and Pass 2 group validation

This milestone adds the repo-owned atomic handoff inside one long-lived Codex session. At the end of the milestone, the worker can run `install-phase` while Pass 1 is active, the repo validates Pass 1 synchronously, writes accepted Pass 1 output, generates the Pass 2 kept-row ledger, rewrites the current-phase sidecars, and tells the same session to continue immediately into Pass 2. Pass 2 then assigns `group_id` and `topic_label` per kept row and gets the same accepted-row and repair protections.

Run workspace-runner and worker-helper tests after this milestone. Acceptance is that one fake-Codex session can finish Pass 1, trigger repo-owned handoff, continue into Pass 2 without a new conversational prompt, and finish only after both phases are accepted.

### Milestone 4: Cut final artifacts, preview, reporting, docs, and tests over to the new story

This milestone rewrites final promotion/reporting and surrounding docs/tests so the repository consistently teaches the new two-phase contract. At the end of the milestone, preview emits shard-owned phase ledgers, runtime summaries report shard counts plus phase row counts, and final artifacts stop producing snippets/evidence surfaces.

Run the focused LLM/staging suite plus a zero-token rehearsal after this milestone. Acceptance is that preview, runtime artifacts, staged outputs, and docs no longer teach packet tasks, snippet/evidence outputs, or reviewer-reason fields as the live knowledge behavior.

## Plan of Work

Start in `cookimport/llm/codex_farm_knowledge_jobs.py`. Keep the current shard-count semantics and contiguous planning over the ordered review queue, but delete the packet-task assumptions from the happy path. Planning should now build one shard-owned Pass 1 input ledger per shard and the metadata needed to derive a Pass 2 kept-row ledger later. The Pass 1 input ledger should contain only the mechanically true fields needed for row classification, such as `block_index`, block text, and any small structural metadata still needed for live labeling. It should not carry packet ids, snippet scaffolding, or packet-local grouping expectations.

Then redesign the worker surfaces in `cookimport/llm/knowledge_stage/workspace_run.py` and the helper/prompt surfaces in `cookimport/llm/knowledge_stage/recovery.py` around the shared phase-first control plane. The worker root should write:

- `CURRENT_PHASE.md` as the cheap first read,
- `CURRENT_PHASE_FEEDBACK.md` as the quiet validation and repair note,
- `current_phase.json` as the machine-readable phase payload,
- `in/pass1/<shard_id>.jsonl` as the immutable Pass 1 input ledger,
- `work/pass1/<shard_id>.jsonl` as the worker-edited Pass 1 work ledger,
- `repair/pass1/<shard_id>.json` only when unresolved Pass 1 rows exist,
- `in/pass2/<shard_id>.jsonl` as the generated Pass 2 kept-row ledger after accepted Pass 1,
- `work/pass2/<shard_id>.jsonl` as the worker-edited Pass 2 work ledger,
- `repair/pass2/<shard_id>.json` only when unresolved Pass 2 rows exist.

The worker prompt should teach one shard across two phases, not one packet queue. It should say explicitly that the repo owns phase transition, that the worker should not ask permission to continue when Pass 2 becomes active, that accepted rows stay frozen, and that feedback sidecars mention only unresolved rows. It should stop mentioning snippets, evidence, reviewer categories, packet ids, `assigned_tasks.json`, and `CURRENT_BATCH.md` as part of the normal path.

The worker runtime should provide live deterministic validation on working-ledger updates or explicit helper calls. That does not mean noisy per-keystroke chatter. It means repo code notices meaningful work-ledger changes, runs cheap validation, freezes accepted rows as soon as they validate, and refreshes `CURRENT_PHASE_FEEDBACK.md` only when unresolved-row diagnostics actually change. Already accepted rows must not be repeatedly echoed back as work to revisit.

Then rewrite `cookimport/llm/codex_farm_knowledge_models.py` and `cookimport/llm/codex_farm_knowledge_ingest.py`. Remove the live schema fields that exist only for snippets/evidence and reviewer reasons. Replace them with two explicit validator surfaces:

- Pass 1 validator: every owned block appears exactly once, decisions are only `knowledge` or `other`, order is preserved, accepted rows can be frozen, unresolved rows can be packaged into a repair request, and repair patches may touch only unresolved rows.
- Pass 2 validator: every kept row from accepted Pass 1 output appears exactly once, every row has one `group_id`, every row has one `topic_label`, topic labels are consistent within a group, ordering is stable, and no row excluded by Pass 1 appears in Pass 2.

Malformed or semantically rejected rows must stay unresolved until repaired or until the explicit fallback ladder is exhausted. They must not be silently treated as accepted, and no deterministic fallback should seep into normal validation.

After validation works at phase scope, add the repo-owned phase handoff helper in the knowledge workspace tooling. The helper name may stay `install-phase` with the active phase encoded in `current_phase.json`, or it may expose narrow wrappers such as `install-pass1` and `install-pass2`, but the runtime behavior must support this story:

1. the worker finishes editing the Pass 1 work ledger,
2. the worker runs the repo-owned install helper,
3. repo code validates Pass 1 synchronously,
4. if Pass 1 is valid, repo code freezes the accepted Pass 1 rows, writes the accepted result, builds the Pass 2 input ledger from kept `knowledge` rows, rewrites `CURRENT_PHASE*` and `current_phase.json`, and prints a short continuation note,
5. the same live Codex session opens the new Pass 2 sidecars and continues immediately,
6. only after Pass 2 is accepted and no active phase remains does the worker send the short final completion note and stop.

That helper is the heart of the same-session design. Do not implement handoff as "worker exits and the repo relaunches a new session" or "worker asks whether to continue and waits for a new conversational prompt." The repo must own the transition locally.

Once phase-owned validation and handoff work, rewrite `cookimport/llm/knowledge_stage/runtime.py`, `promotion.py`, `reporting.py`, and `cookimport/llm/codex_farm_knowledge_writer.py`. Remove task-count vocabulary as the primary progress story. Report shard counts and phase row counts instead. Rewrite final promotion so:

- `09_nonrecipe_authority.json` still records final `knowledge` versus `other`,
- `09_nonrecipe_knowledge_groups.json` records `group_id` and `topic_label` for the kept rows,
- `09_nonrecipe_review_status.json` records phase and repair status at row or shard scope as appropriate,
- reviewer-facing `knowledge.md` is rendered from final kept rows and group labels,
- `knowledge/<workbook_slug>/snippets.jsonl` is removed from the live path unless a later explicit product plan reintroduces a new evidence surface.

Then update `cookimport/llm/prompt_preview.py` and related summary helpers. Preview should answer the real tuning questions directly: how many shards, how many Pass 1 rows per shard, how many kept rows survive into Pass 2, and how large the Pass 2 work ledgers are likely to be. Do not keep packet-task preview artifacts as compatibility noise.

Finally, update docs and tests in the same wave. `docs/10-llm/10-llm_README.md`, `docs/05-staging/05-staging_readme.md`, and any knowledge worker notes must stop teaching packet tasks, snippets/evidence, and reviewer-reason fields as part of the live knowledge contract. Update tests that currently assert packet manifests, packet ids, snippet evidence, or snippet artifact outputs. The new tests should assert shard-owned phase ledgers, same-session handoff, row-local repair, and final group-label artifacts.

## Validation and Acceptance

The change is correct only if all of the following are true.

Planning acceptance:

- The live knowledge planner creates shard-owned Pass 1 ledgers from the ordered non-recipe review queue instead of packet-task queues.
- The normal worker root no longer depends on `assigned_tasks.json`, `CURRENT_BATCH.md`, or one `out/<task_id>.json` per packet on the happy path.

Pass 1 acceptance:

- The live Pass 1 classification contract accepts only one final decision per owned row: `knowledge` or `other`.
- There are no live snippet/evidence requirements and no live reviewer-reason requirements in Pass 1.
- A malformed Pass 1 shard ledger can preserve accepted rows and isolate only unresolved rows into a repair request.

Pass 2 acceptance:

- Grouping runs only after Pass 1 is accepted for that shard and only over kept rows.
- Every kept row gets exactly one `group_id`.
- Every kept row gets one `topic_label`, and topic labels are consistent within each group.
- A row omitted from accepted Pass 1 `knowledge` output cannot leak into Pass 2.

Same-session handoff acceptance:

- One long-lived workspace worker can move from Pass 1 to Pass 2 without a new conversational prompt.
- The repo-owned install helper validates Pass 1, builds the Pass 2 input files, rewrites the current-phase sidecars, and the same worker root continues immediately into Pass 2.
- A worker saying "I can continue if you want" is not the control signal.

Live-validation acceptance:

- Repo code detects meaningful work-ledger updates during both phases, runs deterministic checks without model-token spend, freezes accepted rows incrementally, and updates feedback sidecars with only unresolved-row information.
- If rows 1 through 120 are valid and row 121 is malformed, the feedback surface names row 121 and any other unresolved rows without re-reporting rows 1 through 120 as work to revisit.
- Repair patches may not overwrite frozen accepted rows.

Artifact acceptance:

- `09_nonrecipe_knowledge_groups.json` shows final `group_id` and `topic_label` data.

- `snippets.jsonl` is gone from the live knowledge artifact path unless a later explicit plan adds a new evidence product.

Preview/reporting acceptance:

- Preview and runtime summaries now talk about shard-owned phase row counts instead of packet task counts and packet ids.
- The operator-facing story answers shard-size and kept-row-size questions directly.

Documentation acceptance:

## Idempotence and Recovery

The rollout may be additive while implementation is in progress, but the destination is a hard cut. It is acceptable to introduce shard-owned phase helpers beside the old packet path temporarily while tests move, but the old packet-task, snippet/evidence, and reviewer-reason paths must be deleted before the plan is marked complete.

Every helper command must be safe to rerun. `check-phase` may be rerun on the active work ledger without side effects. `install-phase` must be idempotent when the active phase is already accepted. The Pass 1 handoff must not regenerate conflicting Pass 2 files if rerun. Repair-merge helpers must refuse to overwrite frozen rows. If a worker session dies between phases, resume logic should read `current_phase.json` and continue from the unfinished phase rather than resetting the whole shard.
