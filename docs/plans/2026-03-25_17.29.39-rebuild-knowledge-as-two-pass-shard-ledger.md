Structured restatement:

The user wants the knowledge stage to stop looking like the current packet-plus-snippets workflow and instead behave like a shard-owned ledger that better matches the Codex agent harness. The public knob remains shard count. Choosing `N` knowledge shards should divide the review-eligible non-recipe blocks into roughly `N` contiguous shard owners, and those shard owners should run in parallel. Inside one shard, the worker should read one shard ledger, write one shard-owned output or working copy, and let repo code validate rows without discarding accepted work.

The knowledge stage must be split into two explicit passes. Pass 1 is the focused classification pass: one row per review-eligible block, one final decision per row, and no snippet/evidence output and no reviewer-reason output. Pass 2 runs only after Pass 1 for that shard is accepted and only over the rows that survived as `knowledge`. Pass 2 groups those kept rows into group ids and topic labels. The user explicitly prefers keeping Pass 1 and Pass 2 separate because row-local repair is easier when grouping does not share the same contract as classification.

The same long-lived Codex worker session should own both passes for one shard. The user does not want a conversational "I finished pass 1, would you like me to continue?" handoff. Instead, the repo should own a tight phase transition inside the same worker root: once the worker installs an accepted Pass 1 ledger, repo code should synchronously validate it, freeze accepted rows, build the Pass 2 input files, rewrite the current-phase sidecars, and tell the same live session to continue immediately into Pass 2.

The user also wants validation feedback to be fast and cheap. The worker should not have to wait until the whole pass is "done" before learning that only rows 143 and 187 are malformed. Repo code should provide live or near-live deterministic validation during editing, preserve accepted rows without repeatedly telling the worker "rows 1-142 are still good," and expose only the unresolved rows that need attention. The point is fast local correction in the same worker context window without token-spending chatter.

This plan therefore rebuilds the knowledge runtime around shard-owned two-pass ledgers and same-session phase advancement. It is not a prompt-tuning exercise and it is not a compatibility preservation plan. The end state is a full production cutover: planning, worker surfaces, validation, repair, preview, runtime, artifact naming, and docs all teach "one shard owner, two passes, row-local repair" instead of "packet tasks, snippets, evidence, and task-batch queue control."

## Purpose / Big Picture

After this plan lands, a human should be able to choose `knowledge_prompt_target_count=5`, know that the non-recipe review queue will be split into roughly five contiguous shard owners, and know that each knowledge worker will own one shard through two passes in the same Codex session. In Pass 1, the worker will classify every owned block in its shard as `knowledge` or `other` by filling a shard-owned ledger. Repo code will validate those rows, preserve good rows, and request repairs only for unresolved rows. In Pass 2, the same worker session will immediately receive a shard-local kept-row ledger and will assign `group_id` plus `topic_label` for each kept `knowledge` row. Repo code will then validate those grouping rows and request repairs only for the affected rows or groups.

This matters because it aligns the implementation with the user's operator model and with the strengths of Codex. Codex is good at opening files, editing files, running repo-owned helpers, and iterating locally. The current knowledge contract instead pushes the worker toward one semantic packet output per task and mixes block classification, grouping, snippets, evidence, and retry ladder complexity into one surface. The desired compromise is simpler:

- high accuracy from working over one coherent shard rather than a queue of tiny packet tasks,
- low token waste because accepted rows survive repair,
- easier orchestration because the repo controls phase transition and validation locally,
- better product tuning because shard count remains the public cost/concurrency knob.

The visible proof should be straightforward. Preview and runtime artifacts should show shard-owned Pass 1 ledgers and shard-owned Pass 2 ledgers rather than packet tasks. Worker roots should show `CURRENT_PHASE.md` and shard-owned files rather than `CURRENT_BATCH.md`, `assigned_tasks.json`, and packet-local `out/<task_id>.json`. Final artifacts should show final `knowledge` authority and group labels without snippets or evidence files.

  Evidence: [cookimport/llm/knowledge_stage/recovery.py](/home/mcnal/projects/recipeimport/cookimport/llm/knowledge_stage/recovery.py#L967) already expects the worker to run repo-owned helpers such as `complete-batch`, `check-batch`, and `install-batch`, then continue immediately after successful install.

- Observation: the final assistant message is not the authoritative success signal in the current runtime, which makes a repo-owned Pass 1 -> Pass 2 handoff feasible without a conversational turn boundary.
  Evidence: [docs/10-llm/10-llm_README.md](/home/mcnal/projects/recipeimport/docs/10-llm/10-llm_README.md#L43) explicitly says validated worker-written output files are authoritative and `final_agent_message_state` may legitimately be `informational` or `absent`.

- Observation: removing snippets/evidence from the live contract is not a tiny local edit; it reaches prompt schema, ingest validation, reporting, and reviewer-facing artifact writers.
  Evidence: the current contract in [docs/10-llm/10-llm_README.md](/home/mcnal/projects/recipeimport/docs/10-llm/10-llm_README.md#L229) and [cookimport/llm/codex_farm_knowledge_models.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_models.py) centers packet outputs on `block_decisions` plus `idea_groups` plus snippets/evidence, and the current reviewer artifacts include `snippets.jsonl`.

- Observation: keeping Pass 1 and Pass 2 separate is a genuine product simplification, not an implementation burden. Pass 1 is row-local and validates cleanly; Pass 2 is relational and should only run after Pass 1 freezes the kept-row set.
  Evidence: the design discussion showed that combining classification and grouping would force one repaired `other -> knowledge` change to potentially invalidate whole-group output, while a two-pass ledger lets Pass 1 preserve accepted rows and lets Pass 2 operate on a much smaller set.

## Outcomes & Retrospective

This plan is authored but not yet implemented. The desired end state is explicit: one knowledge shard owner should run two passes in the same long-lived Codex session, with repo-owned phase advancement, row-local Pass 1 repair, and row-plus-group-local Pass 2 repair. The current packet-task contract, snippets/evidence output, and reviewer-reason output are all slated for removal from the live path.

The most important lesson from the design discussion is that Codex should be treated as a file-editing worker rather than as a one-shot JSON vending machine. The current knowledge stage already contains much of the local helper/validation machinery needed for that, but it uses the wrong ownership unit. The redesign is about moving that machinery up to shard-owned two-pass ledgers.

## Context and Orientation

The knowledge stage is the last optional LLM-backed stage in the current five-stage import pipeline. It reviews non-recipe blocks that survived deterministic routing and decides which ones are useful cookbook knowledge. Today it does this by planning ordered packets, writing one packet input per task, asking the worker for `block_decisions` plus `idea_groups` plus snippets/evidence, validating each packet result, and then promoting those packet outputs into final non-recipe authority and reviewer-facing knowledge artifacts.

The current code relevant to this redesign lives in these files:

- `cookimport/llm/codex_farm_knowledge_jobs.py` builds the current review packets and shard manifests from the ordered non-recipe review queue.
- `cookimport/llm/knowledge_stage/runtime.py` is the stage-owned public entrypoint and artifact coordinator for the knowledge phase.
- `cookimport/llm/knowledge_stage/workspace_run.py` writes worker roots and supervises one long-lived Codex workspace session per worker assignment.
- `cookimport/llm/knowledge_stage/recovery.py` owns worker prompts, helper-oriented continuation logic, validation-driven retry decisions, and runtime guardrails.
- `cookimport/llm/codex_farm_knowledge_models.py` and `cookimport/llm/codex_farm_knowledge_ingest.py` define and validate the live packet result schema.
- `cookimport/llm/knowledge_stage/promotion.py`, `reporting.py`, and `codex_farm_knowledge_writer.py` convert validated outputs into final stage authority, stage summaries, and reviewer-facing artifacts.
- `cookimport/staging/import_session_flows/output_stage.py` and `cookimport/staging/pipeline_runtime.py` integrate the knowledge stage into the full import run.

Terms used in this plan:

- A `shard` is one contiguous slice of the review-eligible non-recipe block queue. It is the planning and concurrency unit. If the queue has 1200 review rows and the operator chooses `knowledge_prompt_target_count=6`, the live planner should create about six contiguous shards of about 200 review rows each.
- A `worker` is one Codex session assigned one or more shards to process. When worker count is unset, the runtime should normally run one live worker assignment per planned shard, subject to the existing global caps.
- A `ledger` is a file that looks like a spreadsheet in machine-readable form: one row per owned block plus columns to fill in. In this redesign, the worker will edit ledgers rather than invent one semantic JSON blob per packet.
- `Pass 1` is the row-classification pass. Each owned block row gets exactly one final `knowledge` or `other` decision. There are no snippet, evidence, or reviewer-reason fields in the live Pass 1 contract.
- `Pass 2` is the kept-row grouping pass. It runs only after Pass 1 for that shard is accepted. Each kept `knowledge` row gets a `group_id` and `topic_label`. Group labels must be consistent within a group, but the output remains row-oriented rather than packet-summary-oriented.
- A `repair request` is a repo-owned file naming only unresolved rows plus the minimum local context needed to repair them. Repair is not a public task layer. It exists only after validation finds problems.
- A `phase handoff` is the repo-owned transition between Pass 1 and Pass 2 inside the same live worker session. It should happen inside one local helper action rather than through a new conversational turn.

The current architecture problem is not that the stage uses shards. Shards are the right public knob. The problem is that the runtime immediately decomposes those shards into packet tasks, then teaches the worker to think in `assigned_tasks.json`, `CURRENT_BATCH.md`, `current_batch.json`, and packet-local `out/<task_id>.json` files. The worker also has to satisfy a broader semantic packet contract than the user wants right now, including snippets/evidence and optional reviewer reason categories.

The design target is therefore:

- shard count remains the public concurrency and spend knob,
- one shard remains one coherent worker-owned unit across both passes,
- Pass 1 is a row ledger with only `knowledge|other`,
- Pass 2 is a kept-row ledger with `group_id` and `topic_label`,
- the same Codex session stays alive across the repo-owned Pass 1 -> Pass 2 handoff,
- repo-side validation runs live or near-live during editing and surfaces only unresolved rows,
- accepted rows remain frozen and repairs touch only unresolved rows,
- docs, preview, runtime, and final artifacts all teach the same two-pass shard-owned story.

This milestone replaces packet-result validation with shard-local row-ledger validation for Pass 1. Run focused ingest/validation tests after this milestone. Acceptance is that a synthetic Pass 1 shard with 200 rows and three bad rows yields a repair request naming those three rows only, while the accepted rows remain frozen.

At the end of the milestone, preview emits shard-owned two-pass inputs, runtime summaries talk about Pass 1 and Pass 2 at shard scope, and final artifacts stop producing snippets/evidence surfaces. ## Plan of Work

Start by rewriting planning in `cookimport/llm/codex_farm_knowledge_jobs.py`. Keep the existing operator-facing shard-count rule: the planner should still partition the ordered review queue into roughly `knowledge_prompt_target_count` contiguous shard owners. Replace the current packet-task assumptions with shard-owned ledger payloads. The Pass 1 input payload should contain one ordered row per owned review-eligible block and only the mechanically true fields needed to label that row, such as `block_index`, block text, and any small structural metadata the live stage still needs. Do not preserve packet ids or packet-local idea-group expectations as the happy-path model.

Then redesign the worker surfaces in `cookimport/llm/knowledge_stage/workspace_run.py` and the helper/prompt surfaces in `cookimport/llm/knowledge_stage/recovery.py`. The new worker root should be phase-first, not batch-first. Replace `CURRENT_BATCH.md`, `CURRENT_BATCH_FEEDBACK.md`, `current_batch.json`, and `assigned_tasks.json` as the normal control loop with shard-owned sidecars such as:

- `CURRENT_PHASE.md` as the cheap first read,
- `CURRENT_PHASE_FEEDBACK.md` as the repair/validation continuation note,
- `current_phase.json` for machine-readable phase metadata and named file paths,
- `in/pass1/<shard_id>.jsonl` or equivalent as the immutable Pass 1 shard ledger,
- `work/pass1/<shard_id>.jsonl` as the worker-edited Pass 1 working copy,
- `repair/pass1/<shard_id>.json` only when Pass 1 validation finds unresolved rows,
- `in/pass2/<shard_id>.jsonl` as the generated kept-row ledger after Pass 1 install,
- `work/pass2/<shard_id>.jsonl` as the worker-edited Pass 2 working copy,
- `repair/pass2/<shard_id>.json` only when Pass 2 validation finds unresolved group rows.

The worker prompt should tell the worker to own one shard through both passes, not to finish after Pass 1, and not to ask permission to continue when repo-owned sidecars activate Pass 2. The prompt should also stop mentioning snippets, evidence, reviewer categories, packet ids, or `assigned_tasks.json` as part of the normal knowledge loop.

The worker runtime should also provide live deterministic validation on the working ledgers. That does not mean noisy per-keystroke chatter. It means repo code should notice meaningful working-file updates, run cheap deterministic checks, freeze accepted rows as they become valid, and refresh `CURRENT_PHASE_FEEDBACK.md` or an equivalent sidecar only when unresolved-row state changes. The feedback surface should list only the rows that still need attention and the reason they are unresolved. It must not repeatedly restate that already-accepted rows remain good.

Next, rewrite the models and validators in `cookimport/llm/codex_farm_knowledge_models.py` and `cookimport/llm/codex_farm_knowledge_ingest.py`. Remove the live schema fields that exist only for snippets/evidence and reviewer reasons. Replace them with two explicit validator surfaces:

- Pass 1 validator: every owned block appears exactly once, decisions are only `knowledge` or `other`, order is preserved, accepted rows can be frozen, and unresolved rows can be packaged into a repair request without ambiguity.
- Pass 2 validator: every kept row from accepted Pass 1 output appears exactly once, every row has one `group_id`, every group has one non-empty `topic_label`, topic labels are consistent within a group, ordering is stable, and no row excluded by Pass 1 appears in Pass 2.

Do not hide malformed rows behind silent fallback rewrites. If a row is malformed or semantically rejected, keep it unresolved and write a repair request naming only those rows. Deterministic fallback should remain an explicit terminal ladder after repair policy is exhausted, not a silent side effect during normal validation.

After validation works at phase scope, add the repo-owned atomic handoff helper in the knowledge workspace tooling. The helper name may change during implementation, but the end state must support this story:

1. the worker finishes editing the Pass 1 working ledger,
2. the worker runs `install-pass1`,
3. repo code validates Pass 1 synchronously,
4. if Pass 1 is valid, repo code freezes the accepted Pass 1 result, builds the Pass 2 input ledger from the kept `knowledge` rows, rewrites `CURRENT_PHASE*` and `current_phase.json`, and prints a short continuation note,
5. the same live Codex session opens the new Pass 2 sidecars and continues immediately,
6. only after Pass 2 is accepted and no active phase remains does the worker send the short final completion note and stop.

That helper is the heart of the same-session design. Do not implement the handoff as "worker exits, repo relaunches a new session" or "worker asks whether to continue and the repo sends a new conversational prompt." The repo must own phase transition locally.

Do not make `install-pass1` the first time the worker learns about bad rows. The live validator should already have surfaced unresolved Pass 1 rows during editing. `install-pass1` is the hard phase checkpoint, not the only validation moment.

Once phase-owned validation and handoff work, rewrite runtime reporting and promotion in `cookimport/llm/knowledge_stage/runtime.py`, `promotion.py`, `reporting.py`, and `codex_farm_knowledge_writer.py`. Remove task-count vocabulary as the primary progress story. Report shard counts plus Pass 1/Pass 2 row counts instead. Rewrite the final promotion surface so:

- `09_nonrecipe_authority.json` still records final `knowledge` versus `other`,
- `09_nonrecipe_knowledge_groups.json` records group ids and topic labels for the kept rows,
- `09_nonrecipe_review_status.json` records pass/repair status at row scope or shard scope as appropriate,
- reviewer-facing `knowledge.md` is rendered from final kept rows and their group labels,
- `knowledge/<workbook_slug>/snippets.jsonl` is removed from the live path unless a later separate plan reintroduces a new evidence surface by explicit product choice.

Update preview and budget surfaces in `cookimport/llm/prompt_preview.py` and related summary helpers next. Preview should answer the user's actual tuning questions: "with shard count 5, how many knowledge shards will run, how many review rows are in each shard, how many rows survive Pass 1, and how large are the Pass 2 kept-row ledgers?" Do not keep packet-task preview artifacts as compatibility noise.

Then cut over docs and tests in the same wave. `docs/10-llm/10-llm_README.md`, `docs/05-staging/05-staging_readme.md`, and any knowledge-stage notes under `cookimport/llm/` must stop teaching packets, snippets/evidence, and reviewer reasons as the live contract. Add one short note beside the knowledge worker helper files explaining the new two-pass same-session story. Update the tests that currently assert packet-task manifests, packet ids, `idea_groups` with snippet evidence, or snippet artifact outputs. The new tests should assert shard-owned Pass 1 ledgers, shard-owned Pass 2 ledgers, same-session phase handoff, row-local repair, and final group-label artifacts.

## Validation and Acceptance

The change is correct only if all of the following are true.

Planning acceptance means the live knowledge planner now creates shard-owned Pass 1 review ledgers from the ordered non-recipe review queue instead of packet-task queues. A code-level proof should show that the normal worker root no longer depends on `assigned_tasks.json`, `CURRENT_BATCH.md`, or one `out/<task_id>.json` per packet.

Pass 1 acceptance means the live knowledge classification pass now accepts only one final decision per owned row: `knowledge` or `other`. There must be no live snippet/evidence requirements and no live reviewer-reason requirements. Focused tests must prove that a malformed Pass 1 shard output can preserve accepted rows and isolate only unresolved rows into a repair request.

Pass 2 acceptance means grouping runs only after Pass 1 is accepted for that shard and only over the kept rows. Focused tests must prove that a row omitted from Pass 1 `knowledge` output cannot leak into Pass 2, that every kept row gets exactly one `group_id`, and that every group has a consistent `topic_label`.

Same-session handoff acceptance means one long-lived workspace worker can move from Pass 1 to Pass 2 without a new conversational prompt. The proof must show that the repo-owned install helper validates Pass 1, builds the Pass 2 input files, rewrites the current-phase sidecars, and the same worker root continues immediately into Pass 2. A worker saying "I can continue if you want" must not be the control signal.

Live-validation acceptance means repo code detects meaningful working-ledger updates during both passes, runs deterministic checks without spending model tokens, freezes accepted rows incrementally, and updates feedback sidecars with only unresolved-row information. Focused tests must prove that if a worker writes a shard ledger with rows 1-120 valid and row 121 malformed, the feedback surface names row 121 (and any other unresolved rows) without re-reporting rows 1-120 as work to revisit.

Artifact acceptance means final outputs no longer depend on snippets/evidence. `09_nonrecipe_authority.json` must still show final `knowledge` versus `other`, `09_nonrecipe_knowledge_groups.json` must show the final group labels, and any reviewer-facing markdown must be derivable from those ledgers alone. `snippets.jsonl` must be gone from the live knowledge artifact path unless a later explicit plan adds a new evidence product.

Preview/reporting acceptance means preview and runtime summaries now talk about shard-owned Pass 1 and Pass 2 row counts instead of packet task counts and packet ids. The operator-facing story should answer shard-size and kept-row-size questions directly.

Documentation acceptance means [docs/10-llm/10-llm_README.md](/home/mcnal/projects/recipeimport/docs/10-llm/10-llm_README.md), [docs/05-staging/05-staging_readme.md](/home/mcnal/projects/recipeimport/docs/05-staging/05-staging_readme.md), and any knowledge worker notes stop teaching packet tasks, snippets/evidence, or reviewer reasons as the current product contract.

## Idempotence and Recovery

The rollout should stay additive while implementation is in progress, but the destination is still a hard cut. The safest implementation order is:

1. add the shard-owned Pass 1 and Pass 2 ledger helpers beside the current packet path,
2. add new validators and same-session phase sidecars,
3. switch preview/runtime to the new path behind the existing public run-setting surface,
4. update tests to the new surfaces,
5. remove the old packet-task, snippet/evidence, and reviewer-reason compatibility paths in the same wave.

Each helper command must be safe to rerun. `install-pass1` should be idempotent when Pass 1 is already accepted, and it should not regenerate conflicting Pass 2 files. `install-pass2` should be idempotent when final group ledgers are already accepted. Repair-merge helpers must refuse to overwrite frozen accepted rows. If a worker session dies between phases, resume logic should read the repo-owned `current_phase.json` and continue from the unfinished phase rather than resetting the whole shard.
