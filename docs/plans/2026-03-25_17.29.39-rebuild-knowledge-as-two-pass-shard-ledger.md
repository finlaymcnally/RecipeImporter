Structured restatement:

The user wants the knowledge stage to stop looking like the current packet-plus-snippets workflow and instead behave like a shard-owned ledger that better matches the Codex agent harness. The public knob remains shard count. Choosing `N` knowledge shards should divide the review-eligible non-recipe blocks into roughly `N` contiguous shard owners, and those shard owners should run in parallel. Inside one shard, the worker should read one shard ledger, write one shard-owned output or working copy, and let repo code validate rows without discarding accepted work.

The knowledge stage must be split into two explicit passes. Pass 1 is the focused classification pass: one row per review-eligible block, one final decision per row, and no snippet/evidence output and no reviewer-reason output. Pass 2 runs only after Pass 1 for that shard is accepted and only over the rows that survived as `knowledge`. Pass 2 groups those kept rows into group ids and topic labels. The user explicitly prefers keeping Pass 1 and Pass 2 separate because row-local repair is easier when grouping does not share the same contract as classification.

The same long-lived Codex worker session should own both passes for one shard. The user does not want a conversational "I finished pass 1, would you like me to continue?" handoff. Instead, the repo should own a tight phase transition inside the same worker root: once the worker installs an accepted Pass 1 ledger, repo code should synchronously validate it, freeze accepted rows, build the Pass 2 input files, rewrite the current-phase sidecars, and tell the same live session to continue immediately into Pass 2.

The user also wants validation feedback to be fast and cheap. The worker should not have to wait until the whole pass is "done" before learning that only rows 143 and 187 are malformed. Repo code should provide live or near-live deterministic validation during editing, preserve accepted rows without repeatedly telling the worker "rows 1-142 are still good," and expose only the unresolved rows that need attention. The point is fast local correction in the same worker context window without token-spending chatter.

This plan tracks that intended end state. At the time of this rewrite, the live repository has not implemented this design. The current code still uses packet-owned tasks, batch-first knowledge workers, and reviewer-facing snippets. Those current contracts are described below as the starting point to replace, not as proof that the user intent was met.

## Purpose / Big Picture

After this plan lands, a user will be able to choose `knowledge_prompt_target_count=N` and get roughly `N` shard-owned knowledge ledgers in parallel, not `N` bundles of packet tasks under a packet-native worker loop. Each knowledge worker will own one shard through two repo-owned passes in the same long-lived Codex session:

- `pass1_classify_rows`: one row per owned review-eligible block, final decision only `knowledge` or `other`
- `pass2_group_kept_rows`: only after accepted Pass 1 output, assign `group_id` and `topic_label` to kept `knowledge` rows

The proof will be visible in the artifact tree. Knowledge worker roots should center `CURRENT_PHASE.md`, `CURRENT_PHASE_FEEDBACK.md`, `current_phase.json`, shard-owned `in/work/repair` files for Pass 1 and Pass 2, and final Stage 9 artifacts that no longer depend on packet-local snippet evidence.

## Context and Orientation

The knowledge stage is the optional LLM-backed non-recipe review phase. It receives the review-eligible outside-recipe block queue from `cookimport/staging/nonrecipe_stage.py` after deterministic routing has already removed obvious exclusions. Today, the current implementation still behaves like a packet-task system:

- planning produces one immutable packet file per packet under `raw/llm/<workbook_slug>/knowledge/in/*.json`
- shard manifests can bundle several packets, but the worker runtime still splits those bundles back into one task per packet
- knowledge worker roots are batch-first and center `current_batch.json`, `CURRENT_BATCH.md`, and `CURRENT_BATCH_FEEDBACK.md`
- worker outputs are packet-local `block_decisions` and packet-local `idea_groups`
- reviewer-facing snippets still exist under `knowledge/<workbook_slug>/snippets.jsonl`

The key starting-point files are:

- `cookimport/llm/codex_farm_knowledge_jobs.py` for current packet planning and shard bundling
- `cookimport/llm/knowledge_stage/planning.py` for task planning from shard payloads
- `cookimport/llm/knowledge_stage/runtime.py` for the public knowledge stage entrypoint
- `cookimport/llm/knowledge_stage/workspace_run.py` for worker-root preparation and long-lived workspace sessions
- `cookimport/llm/knowledge_stage/recovery.py` for the current batch-first worker prompt, guardrails, and retries
- `cookimport/llm/knowledge_workspace_tools.py` for the current batch/task helper CLI
- `cookimport/llm/codex_farm_knowledge_ingest.py` for packet-output normalization and validation
- `cookimport/llm/knowledge_stage/promotion.py`, `reporting.py`, and `cookimport/llm/codex_farm_knowledge_writer.py` for promotion into final Stage 9 artifacts and reviewer-facing outputs
- `docs/10-llm/10-llm_README.md` and `docs/05-staging/05-staging_readme.md` for the current live docs that must be updated during the cutover

Terms used in this plan:

- A `shard` is one contiguous slice of the review-eligible non-recipe block queue. It is the planning and concurrency unit the user wants to tune directly.
- A `worker` is one long-lived Codex session assigned one shard for this intended design.
- A `ledger` is a machine-readable row file with one owned row per block plus the columns the worker fills in.
- A `phase` is the repo-owned current step for a shard worker. Knowledge should have two real phases: `pass1_classify_rows` and `pass2_group_kept_rows`.
- A `repair request` is a repo-owned file that names only unresolved rows plus the minimum local context needed to fix them.
- A `phase handoff` is the repo-owned transition from accepted Pass 1 output to generated Pass 2 input inside the same worker root.

The current packet-bundle runtime is the starting point to replace, not the target to preserve. The target is a shard-owned two-pass ledger runtime with live row validation, same-session phase handoff, and final outputs derived from accepted ledgers rather than packet-local snippets.

## Milestones

### Milestone 1: Replace packet-task planning with shard-owned phase ledgers

This milestone removes the packet-task mental model from knowledge planning and worker setup. Each shard should own one Pass 1 input ledger, one Pass 1 work ledger, and one shared current-phase control surface. The worker root should no longer center `CURRENT_BATCH.md`, `current_batch.json`, or `assigned_tasks.json` on the happy path.

Run the focused knowledge planning and workspace tests after this milestone. Acceptance is that the worker root shows `CURRENT_PHASE.md`, `CURRENT_PHASE_FEEDBACK.md`, `current_phase.json`, one Pass 1 input ledger, and one Pass 1 work ledger.

### Milestone 2: Implement Pass 1 validation, accepted-row freezing, and targeted repair

This milestone replaces packet-result validation with shard-row validation for Pass 1. The validator must accept valid rows incrementally, freeze those rows, list unresolved rows, and write one repair request naming only unresolved rows plus minimal context. Repair patches may not overwrite frozen rows.

Run focused ingest and validation tests after this milestone. Acceptance is that a synthetic Pass 1 shard with many rows and a few bad rows yields a repair request naming only the bad rows while accepted rows stay frozen.

### Milestone 3: Add same-session Pass 1 to Pass 2 handoff and Pass 2 validation

This milestone adds the repo-owned atomic handoff inside one long-lived Codex session. The worker should run `install-phase` while Pass 1 is active, repo code should validate Pass 1 synchronously, write accepted Pass 1 output, generate the Pass 2 kept-row ledger, rewrite the current-phase sidecars, and tell the same session to continue immediately into Pass 2. Pass 2 then assigns `group_id` and `topic_label` per kept row and gets the same accepted-row and repair protections.

Run workspace-runner and helper tests after this milestone. Acceptance is that one fake-Codex session can finish Pass 1, trigger repo-owned handoff, continue into Pass 2 without a new conversational prompt, and finish only after both phases are accepted.

### Milestone 4: Cut promotion, reporting, docs, and tests over to the new contract

This milestone rewrites promotion and reporting so the repository consistently teaches the new two-pass contract. Preview should emit shard-owned phase ledgers, runtime summaries should report shard and row counts rather than packet counts, and final outputs should no longer depend on snippet/evidence surfaces.

Run the focused LLM and staging suite plus a zero-token rehearsal after this milestone. Acceptance is that preview, runtime artifacts, staged outputs, and docs no longer teach packet tasks, snippet outputs, or reviewer-reason fields as the live knowledge behavior.

## Plan of Work

Start in `cookimport/llm/codex_farm_knowledge_jobs.py`. Remove the packet-task assumptions from the happy path. Planning should build one shard-owned Pass 1 input ledger per shard and the metadata needed to derive a Pass 2 kept-row ledger later. The Pass 1 input ledger should contain only the mechanically true fields needed for row classification, such as `block_index`, block text, and any small structural metadata still needed for live labeling. It should not carry packet ids, snippet scaffolding, or packet-local grouping expectations.

Then redesign the worker surfaces in `cookimport/llm/knowledge_stage/workspace_run.py` and the helper/prompt surfaces in `cookimport/llm/knowledge_stage/recovery.py` around the phase-first control plane. The worker root should write:

- `CURRENT_PHASE.md`
- `CURRENT_PHASE_FEEDBACK.md`
- `current_phase.json`
- `in/pass1/<shard_id>.jsonl`
- `work/pass1/<shard_id>.jsonl`
- `repair/pass1/<shard_id>.json` only when unresolved Pass 1 rows exist
- `in/pass2/<shard_id>.jsonl` after accepted Pass 1
- `work/pass2/<shard_id>.jsonl`
- `repair/pass2/<shard_id>.json` only when unresolved Pass 2 rows exist

The worker prompt should teach one shard across two phases, not one packet queue. It should say explicitly that the repo owns phase transition, that the worker should not ask permission to continue when Pass 2 becomes active, that accepted rows stay frozen, and that feedback sidecars mention only unresolved rows. It should stop mentioning snippets, evidence, reviewer categories, packet ids, `assigned_tasks.json`, and `CURRENT_BATCH.md` as part of the normal path.

Then rewrite `cookimport/llm/codex_farm_knowledge_ingest.py` and the current knowledge models around the new validator surfaces:

- Pass 1 validator: every owned block appears exactly once, decisions are only `knowledge` or `other`, order is preserved, accepted rows can be frozen, unresolved rows can be packaged into a repair request, and repair patches may touch only unresolved rows.
- Pass 2 validator: every kept row from accepted Pass 1 appears exactly once, every row has one `group_id`, every row has one `topic_label`, topic labels are consistent within a group, ordering is stable, and no row excluded by Pass 1 appears in Pass 2.

After validation works at phase scope, add the repo-owned phase handoff helper. The runtime behavior must be:

1. the worker finishes editing the Pass 1 work ledger
2. the worker runs the repo-owned install helper
3. repo code validates Pass 1 synchronously
4. if Pass 1 is valid, repo code freezes accepted Pass 1 rows, writes the accepted result, builds the Pass 2 input ledger from kept `knowledge` rows, rewrites `CURRENT_PHASE*` and `current_phase.json`, and prints a short continuation note
5. the same live Codex session opens the new Pass 2 sidecars and continues immediately
6. only after Pass 2 is accepted and no active phase remains does the worker send the short final completion note and stop

Finally, rewrite `cookimport/llm/knowledge_stage/runtime.py`, `promotion.py`, `reporting.py`, `cookimport/llm/codex_farm_knowledge_writer.py`, `docs/10-llm/10-llm_README.md`, and `docs/05-staging/05-staging_readme.md`. Remove packet-task and snippet/evidence teaching from the live knowledge contract. `09_nonrecipe_authority.json`, `09_nonrecipe_knowledge_groups.json`, and `09_nonrecipe_review_status.json` should remain, but they should be promoted from accepted ledgers rather than packet-local snippet outputs.

## Validation and Acceptance

The change is correct only if all of the following are true.

Planning acceptance:

- The live knowledge planner creates shard-owned Pass 1 ledgers from the ordered non-recipe review queue instead of packet-task queues.
- The normal worker root no longer depends on `assigned_tasks.json`, `CURRENT_BATCH.md`, or one `out/<task_id>.json` per packet on the happy path.

Pass 1 acceptance:

- The live Pass 1 classification contract accepts only one final decision per owned row: `knowledge` or `other`.
- There are no live snippet/evidence requirements and no live reviewer-reason requirements in Pass 1.
- A malformed Pass 1 shard ledger can preserve accepted rows and isolate only unresolved rows into a repair request.

Pass 2 acceptance:

- Grouping runs only after Pass 1 is accepted for that shard and only over kept rows.
- Every kept row gets exactly one `group_id`.
- Every kept row gets one `topic_label`, and topic labels are consistent within each group.
- A row omitted from accepted Pass 1 `knowledge` output cannot leak into Pass 2.

Same-session handoff acceptance:

- One long-lived workspace worker can move from Pass 1 to Pass 2 without a new conversational prompt.
- The repo-owned install helper validates Pass 1, builds the Pass 2 input files, rewrites the current-phase sidecars, and the same worker root continues immediately into Pass 2.
- A worker saying "I can continue if you want" is not the control signal.

Live-validation acceptance:

- Repo code detects meaningful work-ledger updates during both phases, runs deterministic checks without model-token spend, freezes accepted rows incrementally, and updates feedback sidecars with only unresolved-row information.
- If rows 1 through 120 are valid and row 121 is malformed, the feedback surface names row 121 and any other unresolved rows without re-reporting rows 1 through 120 as work to revisit.
- Repair patches may not overwrite frozen accepted rows.

Artifact acceptance:

- Final outputs no longer depend on snippets/evidence.
- `09_nonrecipe_authority.json` still shows final `knowledge` versus `other`.
- `09_nonrecipe_knowledge_groups.json` shows final `group_id` and `topic_label` data.
- Reviewer-facing markdown is derivable from accepted ledgers alone.
- `snippets.jsonl` is gone from the live knowledge artifact path unless a later explicit product plan adds a new evidence product.

Documentation acceptance:

- `docs/10-llm/10-llm_README.md`, `docs/05-staging/05-staging_readme.md`, and knowledge worker notes stop teaching packet tasks, snippets/evidence, and reviewer-facing snippet artifacts as the live product contract.

## Idempotence and Recovery

The rollout may be additive while implementation is in progress, but the destination is a hard cut. It is acceptable to introduce shard-owned phase helpers beside the old packet path temporarily while tests move, but the old packet-task, snippet/evidence, and reviewer-facing packet contract must be deleted before the plan is marked complete.

Every helper command must be safe to rerun. `check-phase` may be rerun on the active work ledger without side effects. `install-phase` must be idempotent when the active phase is already accepted. The Pass 1 handoff must not regenerate conflicting Pass 2 files if rerun. Repair-merge helpers must refuse to overwrite frozen rows. If a worker session dies between phases, resume logic should read `current_phase.json` and continue from the unfinished phase rather than resetting the whole shard.
