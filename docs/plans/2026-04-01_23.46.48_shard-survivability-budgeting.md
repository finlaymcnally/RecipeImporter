## Operator Intent (Structured) - DO NOT DELETE

The operator wants a deterministic preflight that predicts whether a planned Codex Exec shard is likely to survive before a live benchmark, prediction run, or stage run starts. The planner must treat `1 shard = 1 worker` as the intended contract for this feature because extra workers duplicate cold-start instruction overhead and materially increase spend. The planner must account for both input and output limits, with special attention to the harsher output cap. The planner must not trust the headline session window by itself because the practical failure mode is that an individual shard is too large and the worker gets killed before session compaction can rescue it. The desired result is the smallest shard count that keeps the worst shard under a conservative survivability budget, together with a concrete explanation of what limit is binding and what shard count is recommended.

## Purpose / Big Picture

After this change, an operator will be able to preview a Codex-backed run and see, before spending live tokens, the predicted worst-shard input tokens, predicted worst-shard output tokens, predicted peak session load for the shard's longest happy-path or repair path, and the minimum safe shard count under a conservative survivability budget. When the operator requests too few shards, the program will fail closed by default with a clear recommendation such as "requested 5 shards, minimum safe count is 8, binding limit is output budget on nonrecipe finalize". The operator will remain in charge of shard count; the program will not silently fan out to dozens of workers.

The interactive CLI will also change shape. Instead of asking block-labelling, recipe-correction, and knowledge shard counts in three separate typed prompts, the interactive Codex setup will show one single shard-planning page. The top of that page will display the estimator summary for the currently selected book or source pair, and the list below it will let the operator edit all enabled shard counts in one place before continuing. Because the estimator depends on the concrete book or source pair, the interactive setup order must change so the relevant file or benchmark target is chosen before that shard-planning page is shown.

This matters because the repository already has prompt preview and token estimation, but it currently summarizes run-wide cost better than shard survivability. Knowledge planning has rough per-packet character budgets, and prompt preview can reconstruct and tokenize exact rendered prompts, yet the active guardrails do not answer the operator's real question: "what is the smallest number of shards that will keep a single worker from blowing up?" This plan adds that missing seam.

The proof for this plan is deterministic and local. A future contributor will be able to run prompt preview on synthetic or existing processed runs, inspect the generated survivability report, and see that benchmark and stage entrypoints reject unsafe shard counts without launching live Codex-backed work. No live Codex benchmark or book-processing run is required to validate the feature.

## Outcomes & Retrospective

This plan is not implemented yet. The useful outcome so far is that the design has been narrowed to one simple contract: use prompt preview plus structural tokenization to estimate per-shard survivability, solve for the minimum safe shard count under conservative stage-specific limits, and refuse live runs that ask for fewer shards unless the operator explicitly changes the request. The plan intentionally does not try to solve generic multi-worker scheduling because that would work against the operator's stated preference.

The main lesson from the research phase is that the repository already contains most of the raw ingredients. The missing piece is not tokenization itself; it is a planning layer that turns existing prompt and shard metadata into one operator-facing answer about survivability and minimum safe shard count.

## Context and Orientation

In this repository, a "shard" is one owned work bundle for a direct Codex stage such as recipe refine, canonical line-role, or nonrecipe finalize. A "worker" is one Codex session that executes one or more assigned shards. For this feature, the operator wants the simpler contract where one shard is assigned to one worker, which means that changing shard count also changes worker count and therefore duplicated instruction overhead.

A "turn input budget" is the maximum amount of prompt material a single request should carry. A "turn output budget" is the maximum amount of model output a single request should produce. A "session peak budget" is the maximum total conversational footprint we allow for the longest likely path through a resumed Codex session, including the initial response plus one repair or grouping continuation when that stage supports resumed sessions. A "survivability budget" is the conservative, stage-specific budget we use for planning. It is intentionally lower than any raw provider-advertised context window because the operator has observed that workers can still fail when one shard is too large.

The main prompt reconstruction seam lives in `cookimport/llm/prompt_preview.py`. That module can rebuild recipe, knowledge, and line-role prompt rows for an existing processed run, write `prompts/full_prompt_log.jsonl`, and emit a `prompt_preview_manifest.json` that already includes phase plans and run-wide warnings.

The token-estimation seam lives in `cookimport/llm/prompt_budget.py`. That module tokenizes prompt preview rows with `tiktoken`, estimates structural outputs by recreating the JSON result shape, and writes `prompt_preview_budget_summary.json` and `prompt_preview_budget_summary.md`. Today it summarizes whole stages and total runs; it does not decide whether one particular shard is too large.

The direct shard planners are split by stage. Recipe planning lives in `cookimport/llm/recipe_stage_shared.py`. Canonical line-role planning lives in `cookimport/parsing/canonical_line_roles/planning.py`. Knowledge packet planning lives in `cookimport/llm/codex_farm_knowledge_jobs.py`, with the stage runtime wrapped by `cookimport/llm/knowledge_stage/runtime.py`. Those modules already accept prompt target counts and construct `ShardManifestEntryV1` payloads. This plan adds one common survivability estimator that each of those planners will consult.

The interactive run-settings flow lives in `cookimport/cli_ui/run_settings_flow.py`. The current Codex submenu already shows all surface toggles on one page through `_prompt_codex_surface_menu(...)`, but shard counts are still collected by `_choose_interactive_codex_prompt_targets(...)` through three separate typed prompts. This plan changes that second step into one list-style editor page so the operator can adjust all enabled shard counts while seeing the estimator summary above them.

The guided interactive entrypoints live in `cookimport/cli_support/interactive_flow.py`. Import already selects the source file before `choose_run_settings(...)`, so it naturally has the right order for a per-book estimator on single-file import. Single-book benchmark currently does not: `choose_run_settings(...)` runs before `_interactive_single_book_benchmark(...)` resolves the concrete gold/source pair through `_resolve_benchmark_gold_and_source(...)` in `cookimport/cli_support/bench_all_method.py`. This plan must reorder that benchmark flow so the estimator-backed shard page appears only after the actual benchmark target has been selected. Multi-book benchmark flows may need either a deferred page per target, a first-target exemplar, or an explicit "recommendation unavailable until target set is fixed" message; the implementation section below resolves that choice.

The current command surfaces that expose prompt target counts are `cookimport/cli_commands/stage.py` for stage runs, `cookimport/cli_commands/labelstudio.py` plus `cookimport/labelstudio/ingest_flows/prediction_run.py` and `cookimport/labelstudio/ingest_flows/upload.py` for benchmark and prediction-run flows, `cookimport/cli_ui/run_settings_flow.py` for interactive setup, and `cookimport/cf_debug_cli.py` for `preview-prompts`. These are the user-visible places where the new guardrail must appear.

## Milestones

### Milestone 1: Define a stage-neutral survivability model

At the end of this milestone, the repository will have one small module under `cookimport/llm/`, for example `shard_survivability.py`, that defines the planning vocabulary and the deterministic solver. It will expose explicit records such as `StageSurvivalCaps`, `ShardSurvivalEstimate`, and `ShardCountRecommendation`. The solver will accept per-shard estimates for input tokens, output tokens, session-peak tokens, and unit count, then return the smallest shard count that satisfies all active caps together with the binding limit (`input`, `output`, `session_peak`, or `unit_count`).

The proof for this milestone is a narrow unit test suite showing that the solver picks the minimum viable shard count, reports the right binding limit, and never recommends fewer shards than the current requested count when one of the caps would be exceeded.

### Milestone 2: Surface per-shard survivability in prompt preview

At the end of this milestone, `cf-debug preview-prompts` will generate per-shard survivability metadata in addition to run-wide totals. Every shard in the prompt preview phase plans will carry exact or structural estimates for initial input tokens, initial output tokens, worst followup path tokens, predicted peak session tokens, and a recommended-safe verdict under the active stage defaults. The preview manifest and budget summary will also include a stage-level recommendation block with the minimum safe shard count and the worst offending shard.

The proof for this milestone is a deterministic prompt preview test that reconstructs a deliberately oversized shard, then verifies that the preview output reports a recommendation above the requested count and identifies the correct binding limit.

### Milestone 3: Enforce survivability in live planning paths

At the end of this milestone, recipe, line-role, and knowledge planning will call the same survivability module before a live Codex-backed run begins. If the operator requests too few shards, the planner will fail with a concise recommendation instead of launching an unsafe run. The refusal message must preserve operator control by telling the operator exactly what to change: the requested shard count, the recommended minimum, the stage involved, and the binding limit.

The proof for this milestone is a set of synthetic planner and command tests showing that stage and benchmark entrypoints reject unsafe requests before any live Codex execution starts, while safe requests continue unchanged.

### Milestone 4: Rework the interactive shard editor around the estimator

At the end of this milestone, the interactive CLI will no longer ask the three shard counts through separate typed prompts. Instead, it will show one shard-planning page in `cookimport/cli_ui/run_settings_flow.py`. That page will render a compact estimator header above the editable rows and will let the operator change block-labelling, recipe-correction, and knowledge shard counts together on one screen. The header must report at least the fields the operator asked for: predicted input tokens per shard, predicted output tokens per shard, predicted session-peak tokens per shard, recommended shard count, and the binding limit (`input`, `output`, or `session_peak`).

The proof for this milestone is an interactive-flow test that drives the run-settings flow with a synthetic estimator payload and verifies that one page contains all enabled shard rows plus the estimator summary above them, and that changing the rows updates the selected run settings.

### Milestone 5: Reorder interactive setup around concrete targets

At the end of this milestone, interactive import and benchmark setup will resolve any required concrete book or source pair before presenting the estimator-backed shard-planning page. For import this means preserving or tightening the existing source-first order. For single-book benchmark this means moving gold/source resolution ahead of `choose_run_settings(...)` or splitting `choose_run_settings(...)` so the estimator-backed shard editor runs after target resolution. For multi-book benchmark, the implementation must explicitly choose one deterministic behavior: either show a summarized cross-target recommendation page after target selection or explain that book-specific recommendations are deferred because many targets are involved.

The proof for this milestone is an interactive benchmark test showing that the selected gold/source pair is known before the shard-planning page is shown, and that the estimator summary refers to that selected target rather than a generic placeholder.

### Milestone 6: Persist observed survivability telemetry

At the end of this milestone, phase manifests or sibling artifacts will carry a compact `shard_survivability_report.json` that records predicted per-shard survivability metrics alongside observed outcomes such as token totals, followup counts, missing-output signals, watchdog kills, and any other pathology flags already summarized by `cookimport/llm/codex_exec_runner.py`. The report will not change planning during the first implementation, but it will give future work a trustworthy calibration surface.

The proof for this milestone is a deterministic test that feeds synthetic telemetry rows into the reporting path and verifies that the survivability report records both prediction and observed outcome for the same shard identifiers.

### Milestone 7: Update docs and operator-facing summaries

At the end of this milestone, the LLM and benchmark docs will describe the new planning rule plainly: shard count is operator-controlled, one shard maps to one worker for this budgeting surface, output cap is first-class, and the planner refuses unsafe counts by default. The prompt preview markdown summary will also mention the minimum safe shard count and the binding limit so a reader does not need to inspect raw JSON.

The proof for this milestone is updated docs plus a short prompt preview transcript that shows a recommendation and a refusal message in plain English.

## Plan of Work

Start by adding a narrow planning module such as `cookimport/llm/shard_survivability.py`. Keep the model simple and explicit. Define a record for stage defaults, a record for one shard estimate, and a record for the overall recommendation. Each stage default should include four conservative caps: maximum initial input tokens, maximum initial output tokens, maximum session-peak tokens for the longest likely live path, and maximum owned-unit count for that stage. "Owned unit" means one recipe candidate for recipe refine, one canonical line for line-role, or one nonrecipe block row for knowledge. The module should also include a small solver that takes a list of shard estimates at the current requested shard count and either validates them or recomputes the minimum safe shard count by simulating higher shard counts until all shards fit.

Do not rely on raw character budgets for the final answer, even if a stage still uses characters internally for a local split heuristic. The recommendation system must work in tokens for input, tokens for output, and tokens for peak session. The knowledge planner may keep its existing character split heuristics as an early rough partitioning step, but the final survivability verdict must come from the shared token-based planner.

Next, extend `cookimport/llm/prompt_budget.py` so it can produce row-level and shard-level structural estimates rather than only stage totals. The repository already knows how to count exact rendered prompt tokens and structural output tokens for each preview row. Reuse that logic to add a helper that returns, for one planned shard, the predicted initial input tokens, predicted initial output tokens, predicted repair or grouping followup tokens when applicable, and the resulting peak session footprint. For stages that have no structured followup path, peak session is simply initial input plus initial output. For knowledge and other resumed-session stages, peak session must include the larger of the repair path or grouping path. Keep the first version deterministic and conservative. If a stage cannot derive a trustworthy followup estimate for one path, the planner should mark that shard `estimation_unavailable` and fail closed in live planning.

Then thread those per-shard estimates through `cookimport/llm/prompt_preview.py`. The phase plans produced by `_build_direct_shard_phase_plan(...)` currently record shard ids, owned ids, prompt chars, and worker assignment. Extend those plans so each shard also carries `estimated_input_tokens`, `estimated_output_tokens`, `estimated_peak_session_tokens`, `estimated_followup_tokens`, `owned_unit_count`, `binding_limit`, and a boolean or enum verdict such as `safe`, `warning`, `unsafe`, or `unavailable`. The preview manifest should also carry a per-stage `minimum_safe_shard_count` recommendation and the worst offending shard id.

After that, integrate the same planning module into the live shard planners. In `cookimport/llm/recipe_stage_shared.py`, insert the survivability check after recipe shard candidates are assembled but before worker assignments are finalized. In `cookimport/parsing/canonical_line_roles/planning.py`, do the same for canonical line-role shard plans. In `cookimport/llm/codex_farm_knowledge_jobs.py`, keep the existing packet split behavior but add a second survivability pass that can say "your requested shard count is still too low even though the rough character budget was satisfied". In all three places, the live planner should fail with a deterministic error when the operator-requested shard count is below the computed minimum safe shard count. Do not silently bump the count. The error must name the stage, the requested shard count, the minimum safe shard count, and the binding limit.

Then change the interactive CLI. Replace `_choose_interactive_codex_prompt_targets(...)` in `cookimport/cli_ui/run_settings_flow.py` with one page-oriented editor instead of repeated `_prompt_codex_prompt_target_count(...)` calls. Reuse the same visual style as `_prompt_codex_surface_menu(...)`: one list, arrow-key navigation, explicit current row, and a final `Continue` row. The new page must render a compact estimator header above the list, and each enabled stage row must show the current shard count, the recommended shard count, and a short binding-limit badge or risk marker when unsafe. The page should still let the operator edit shard counts directly without leaving the screen. The exact keybindings can stay simple, but the interaction must let the operator review all three values together while the estimator summary remains visible.

Reorder interactive setup where needed so the estimator has a real target. In `cookimport/cli_support/interactive_flow.py`, import already chooses the source file before `choose_run_settings(...)`, so preserve that order and thread the selected file into the estimator context. For single-book benchmark, move `_resolve_benchmark_gold_and_source(...)` earlier so `choose_run_settings(...)` can receive the concrete gold/source pair before it renders the shard-planning page. If `choose_run_settings(...)` becomes too overloaded, split it into a high-level workflow chooser followed by a target-aware Codex planner step. The important behavior is that the shard-planning page sees the actual selected book. For matched-book benchmark, make one explicit product choice and document it in code and tests: either show an aggregate recommendation summary over the selected target set, or show a clear message that target-specific recommendations appear only in `preview-prompts` because many books are involved. Do not leave this ambiguous.

Expose the refusal cleanly through the existing command surfaces instead of adding a large new configuration menu. `cookimport/cf_debug_cli.py::preview_prompts(...)` should continue to succeed and emit recommendations because it is a dry-run analysis tool. The live paths in `cookimport/cli_commands/stage.py`, `cookimport/cli_commands/labelstudio.py`, `cookimport/labelstudio/ingest_flows/prediction_run.py`, and `cookimport/labelstudio/ingest_flows/upload.py` should raise before any live Codex work starts. ## Validation and Acceptance

Acceptance is behavioral.

For the shared solver:

1. Given a list of per-shard estimates that already fit the conservative stage caps, the solver returns the requested shard count unchanged and marks the stage safe.
2. Given estimates that violate one or more caps, the solver returns the smallest higher shard count that makes every shard safe and records the correct binding limit.
3. If a stage cannot produce a trustworthy estimate for a required path, the solver marks that stage unavailable and live planning refuses to continue.

For prompt preview:

1. `cf-debug preview-prompts` emits per-shard survivability metrics and a per-stage minimum safe shard recommendation into `prompt_preview_manifest.json` and `prompt_preview_budget_summary.json`.
2. The markdown summary mentions the minimum safe shard count and the binding limit in plain language.
3. A deliberately oversized synthetic shard produces an unsafe verdict and a higher recommended shard count without launching any live Codex work.

For the interactive CLI:

1. Interactive Codex shard planning shows one page with all enabled shard rows instead of asking one text prompt per stage.
2. That page renders the estimator summary above the rows and includes at least predicted input tokens per shard, predicted output tokens per shard, predicted session-peak tokens per shard, recommended shard count, and binding limit.
3. Editing the shard rows on that page updates all three prompt-target settings in one pass.
4. Single-book benchmark resolves the actual gold/source pair before showing that estimator-backed shard page.

For live planning:

1. Recipe, line-role, and knowledge planners all refuse unsafe requested shard counts before worker launch.
2. The refusal message tells the operator exactly which prompt target count to increase and why.
3. A safe request continues unchanged and does not create extra shards beyond the operator's request.

For telemetry:

1. Each direct-exec stage writes a compact survivability report that joins predicted and observed metrics by shard id.
2. The report records observed pathology flags such as missing output, watchdog kill, or structured followup activity when those occur.

For docs:

1. The LLM and benchmark docs explain that shard survivability is checked against conservative budgets, not the provider's raw context window.
2. The docs state plainly that, for this feature, one shard maps to one worker and more shards increase cold-start overhead.

## Idempotence and Recovery

This feature should be implemented additively and can be retried safely. Prompt preview and shared planning code can be extended first without touching live planner refusal behavior. That gives a safe checkpoint where the repository can already compute survivability reports but does not yet block runs. Once the preview outputs are trustworthy, add the fail-closed guard to the live planners.

Do not leave the repository in a mixed state where prompt preview and live planners disagree about the recommended shard count. The same shared survivability module must back both. If a partial implementation lands and the preview numbers are not yet stable, keep the live paths in recommendation-only mode until the deterministic tests prove that the shared solver is correct.

If one stage cannot yet produce a followup-path estimate, prefer explicit `estimation_unavailable` plus refusal over a fake optimistic number. This feature exists to prevent unsafe runs, so conservative failure is the correct temporary behavior.
