# ExecPlan: Option A — Freeform Spans Golden Set for Recipe Structure (+ Yield/Time), No Tag Annotation

This ExecPlan is a living document. Keep **Progress**, **Surprises & Discoveries**, **Decision Log**, and **Outcomes & Retrospective** up to date as you execute. Maintain this plan in accordance with `docs/PLANS.md`.

## Purpose / Intent (why this matters)

We need a ground-truth dataset to measure whether cookbook ingestion correctly turns messy sources into structured recipes. The highest-leverage human signal is **structure** (titles, ingredient lines, instruction lines, and non-recipe text), because that directly drives downstream parsing, linking, and recipe creation.

Option A deliberately minimizes annotation burden: a small, stable span label set for structure, plus two practical metadata lines (**yield/servings** and **time**) that strongly affect schema alignment. We explicitly avoid asking annotators to label against a large, evolving tag taxonomy (throughput + consistency killer).

Observable “it worked” outcome:

- Label Studio shows a **freeform-spans** project where annotators highlight spans with labels including `YIELD_LINE` and `TIME_LINE`.
- `cookimport labelstudio-export --export-scope freeform-spans` produces:
  - `freeform_span_labels.jsonl`
  - `freeform_segment_manifest.jsonl`
  - `summary.json` (and the raw `labelstudio_export.json`)
- `cookimport labelstudio-eval freeform-spans` produces:
  - `eval_report.md`
  - `eval_report.json`
  - `missed_gold_spans.jsonl`
  - `false_positive_preds.jsonl`

## Glossary (terms used here)

- **Source file**: the exact input file bytes (EPUB/PDF/etc.). If bytes change, the source identity changes and eval can collapse to near-zero.
- **Block**: an ordered unit of extracted text with a `block_index` (EPUB paragraph/list item, PDF line/OCR line, etc.).
- **Segment**: a contiguous window of blocks used for freeform annotation, with deterministic ID:
  - `urn:cookimport:segment:{source_hash}:{start_block_index}:{end_block_index}`
- **Gold**: spans labeled by humans in freeform-spans projects, exported as `freeform_span_labels.jsonl`.
- **Predictions (“pred-run”)**: for freeform eval, this is typically the **pipeline** task-scope import run folder (it contains `label_studio_tasks.jsonl` that eval maps into predicted ranges).
- **Write consent**: `--allow-labelstudio-write`. Non-interactive import/benchmark require it; treat as “yes, I intend to upload tasks.”

## Scope (and non-goals)

In scope:

- Freeform spans golden set for structure labels, plus additive `YIELD_LINE` and `TIME_LINE`.
- Preserve offset stability (no whitespace normalization changes in freeform rendering).
- Export + eval accept and report new labels without breaking legacy “strict” headline metrics.
- Pilot (3–5 books) → v1 (10–20 books), plus a labeling guide and basic QA/IAA.

Non-goals:

- Manual tagging against the large tag catalog.
- PDF image box annotation workflows (not the current integration).
- Re-architecting benchmark; only guardrails and clarity improvements needed to run safely.

## Surprises & Discoveries (keep updated)

- Freeform workflows are segment-based and rely on deterministic IDs for resume/idempotence. This is powerful, but it can surprise you when you want to "re-import the same book" (IAA, reannotation) unless you isolate output dirs.
- Freeform labels intentionally preserve whitespace in the UI to keep offsets stable; any "cleanup" normalization risks breaking span offsets.
- Non-interactive commands default under a golden-root folder and uploads are gated by explicit write-consent flags; accidental upload risk is real if people muscle-memory CLI commands.
- The existing catalog contains a large set of recipe tags; asking annotators to label against that set is a throughput killer and adds consistency risk.
- (2026-02-12) The pipeline does not yet produce dedicated yield/time chunk types — yield/time lines are currently classified as headers or OTHER by the general classifier. The `_map_chunk_to_label` mappings for YIELD_LINE/TIME_LINE are future-proofed but will only activate when the pipeline adds those chunk types. Freeform gold annotations will still capture yield/time from human labeling.
- (2026-02-12) The export path (`export.py`) is label-agnostic for freeform spans — it passes through whatever labels annotators apply. No export changes were needed for new labels.
- (2026-02-12, updated 2026-02-22) Added backward-compat aliases for renamed freeform labels:
  - `TIP` → `KNOWLEDGE`
  - `NOTES` / `NOTE` → `RECIPE_NOTES`
  - `VARIANT` → `RECIPE_VARIANT`
  - `YIELD` → `YIELD_LINE`
  - `TIME` → `TIME_LINE`

## Plan of Work

### Phase 0 — Align on “What we’re measuring”

This is a **structure-first** golden set. Annotators label only spans that correspond to recipe structure (and optionally yield/time lines). Everything else becomes `OTHER`. The dataset is designed to support repeatable benchmarking and to localize failures (ingredients vs narrative, instruction boundary drift, yield/time missed, etc.).

### Phase 1 — Taxonomy + Label Studio config changes (small + additive)

Add two new labels to the existing freeform label config:

- `YIELD_LINE` — spans expressing yield/servings (e.g., “Serves 4”, “Makes 2 loaves”).
- `TIME_LINE` — spans expressing prep/cook/total time (e.g., “Prep: 10 min”, “Total time 1 hour”).

Hard constraints:

- No renaming/removal of existing labels.
- Keep offset stability (preserve whitespace / pre-wrap rendering).
- Do not alter legacy normalization mappings (if older exports exist).

Concrete code touch points to inspect and likely update:

- `cookimport/labelstudio/label_config_freeform.py` (labels + rendering)
- `cookimport/labelstudio/export.py` (freeform export shaping/validation, if any)
- `cookimport/labelstudio/eval_freeform.py` (label normalization + reporting)

Testing expectations:

- Update/add tests in:
  - `tests/test_labelstudio_freeform.py` (config includes new labels; export accepts them)
  - `tests/test_labelstudio_export.py` (fixture with new labels passes)
  - Any eval report/snapshot tests to ensure strict metrics do not change while diagnostics include yield/time counts.

### Phase 2 — Pilot dataset creation (3–5 books)

Goal: validate segmentation, labeling guidance, export/eval reproducibility, and the practical edge cases (messy EPUB + at least one PDF).

Pilot success criteria:

- Imports produce stable task artifacts and reasonable `coverage.json` (no large silent gaps).
- Annotators can label without constantly “what label is this?” questions.
- Exports are generated without patching.
- Eval produces reports and the failure cases are debuggable back to segments.

Segmentation defaults (freeze for pilot):

- `segment_blocks = 40`
- `segment_overlap = 5`

If pilot shows systematic boundary misses (e.g., yield lines often split), adjust once and version the dataset cut. Do not change segmentation mid-campaign without versioning; it changes segment IDs and complicates merges.

### Phase 3 — Evaluation integration (strict metrics stable + yield/time as diagnostics)

Key requirement: keep existing headline/strict metrics comparable over time. The safest pattern:

- Include `YIELD_LINE` and `TIME_LINE` in exports.
- In freeform eval reporting, treat yield/time as **additive diagnostics** (counts + per-label metrics), not part of the strict aggregate unless you explicitly version/change the headline metric definition.

### Phase 4 — Scale to v1 (10–20 books) + QA + IAA

Scale annotation with:

- A clear labeling guide.
- Periodic spot checks (20 segments/book).
- A 10–20% overlap subset labeled by a second annotator to estimate agreement and highlight ambiguity.

Produce a versioned dataset bundle folder with:

- corpus list (files + hashes),
- segmentation params,
- label set version,
- date range + annotators,
- known caveats (OCR-heavy PDFs, etc.),
- per-book eval reports.

## Risks and Mitigations (plus footguns)

### Offset drift / unstable spans
Mitigation: preserve whitespace in rendered text; prohibit normalization; add a smoke test that offsets still align to exported segment text.

### Accidental uploads / write-side effects
Mitigation: keep write-consent explicit; document “no write without intent”; keep pilot/v1 projects clearly named.

### Benchmark side effects + gold discovery footgun
Non-interactive benchmark always uploads predictions first; it is not “eval-only.” Also, gold discovery searches both `data/golden/**` and `data/output/**`. If multiple gold exports exist, it can select the wrong one.
Mitigation: run benchmark in a clean workspace (or use explicit paths via `labelstudio-eval`); avoid mixing unrelated exports in the same roots during a campaign.

### Source identity mismatches leading to “zero match” eval
Mitigation: keep the exact source bytes stable; avoid renames/cutdown variants during labeling. Do not default to `--force-source-match`; use it only when you knowingly compare variants and accept the risk of masking mismatches.

### Segmentation too small/large
Mitigation: tune after pilot; version if changed; do not change mid-run.

### Label ambiguity (RECIPE_NOTES vs KNOWLEDGE vs OTHER; yield/time boundaries)
Mitigation: crisp definitions + examples; add an “if unsure, label OTHER unless clearly KNOWLEDGE/RECIPE_NOTES” rule; adjudicate repeated disagreements.

## Labels (Freeform-Spans)

Base labels (existing):

- `RECIPE_TITLE`
- `INGREDIENT_LINE`
- `INSTRUCTION_LINE`
- `KNOWLEDGE`
- `RECIPE_NOTES`
- `RECIPE_VARIANT`
- `OTHER`

Additive labels (Option A extension):

- `YIELD_LINE`
- `TIME_LINE`

High-level labeling rules:

- Label the smallest span that cleanly expresses the target.
- Prefer line-level spans for ingredients/instructions/yield/time.
- If unsure whether something is “real recipe content” vs prose, label `OTHER` unless it clearly fits `KNOWLEDGE` or `RECIPE_NOTES`.

## Recommended CLI Runbook (Non-Interactive)

Prereqs:

- Label Studio running + env vars set:
  - `LABEL_STUDIO_URL`
  - `LABEL_STUDIO_API_KEY`

Notes on output roots:

- Label Studio CLI defaults its artifacts under `data/golden/` (not `data/output/`).
- Non-interactive upload/write is gated by `--allow-labelstudio-write`.

### 1) Import (freeform spans) — gold project

Use consistent segmentation settings for the entire campaign:

    cookimport labelstudio-import data/input/<book>.epub \
      --task-scope freeform-spans \
      --segment-blocks 40 \
      --segment-overlap 5 \
      --allow-labelstudio-write

Annotate in Label Studio.

### 2) Export gold

    cookimport labelstudio-export \
      --project-name "<Freeform Project Name>" \
      --export-scope freeform-spans

Record the resulting path to:

- `<gold_spans_jsonl>` = `.../exports/freeform_span_labels.jsonl`

### 3) Import (pipeline) — prediction run (“pred-run”)

Freeform eval expects predicted labeled ranges mapped from pipeline chunks, so generate a pipeline run folder:

    cookimport labelstudio-import data/input/<book>.epub \
      --task-scope pipeline \
      --chunk-level both \
      --allow-labelstudio-write

Record `<pred_run_dir>` = the run folder created for this pipeline import (it contains `label_studio_tasks.jsonl`).

### 4) Eval (freeform spans)

Default: do NOT force-match unless you explicitly accept comparing variants.

    cookimport labelstudio-eval freeform-spans \
      --pred-run <pred_run_dir> \
      --gold-spans <gold_spans_jsonl> \
      --output-dir data/golden/<ts>/eval-freeform/<book_slug>/

If (and only if) you intentionally compare renamed/cutdown variants, you may add `--force-source-match`, but document why.

### 5) Benchmark (convenience, but upload-first)

    cookimport labelstudio-benchmark \
      --allow-labelstudio-write

Warning: benchmark is upload-first; it will import/upload predictions before scoring. Also, gold discovery searches both `data/golden/**` and `data/output/**`; clean your workspace or use explicit `labelstudio-eval` paths for safety.

## Dataset Versioning

For every dataset cut, record:

- corpus list (books + versions/hashes),
- segmentation params (`segment_blocks`, `segment_overlap`),
- label set version (including yield/time enabled),
- date range + annotators,
- known caveats (OCR-heavy PDFs, etc.).

## Practical Quality Checks

- Randomly sample 20 segments per book:
  - confirm RECIPE_TITLE vs section headers are consistent,
  - confirm ingredients aren’t being confused with instruction steps,
  - confirm yield/time are consistently captured when present.
- After export, spot-check:
  - span offsets still align to displayed text,
  - touched block indices look plausible (no wildly out-of-range indices).
