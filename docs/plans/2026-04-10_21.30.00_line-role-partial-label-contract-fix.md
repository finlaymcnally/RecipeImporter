# Line-Role Partial Label Contract Fix

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, canonical line-role inline JSON turns should stop leaking a too-short `labels` array through the Codex transport as if the turn completed successfully. A human should be able to rerun the same style of benchmark and see partial label arrays rejected earlier and repair turns keep the same structural guardrails as initial turns instead of weakening into `resume --last` without schema enforcement.

## Surprises & Discoveries

- Observation: The failing shard responses were clean JSON objects that simply closed early after a contiguous prefix of labels.
  Evidence: The initial failing run returned `159/184` labels; repair returned `22/25`; repo validation showed unresolved suffix indices only.
- Observation: The current output schema cannot reject partial arrays because it only requires a `labels` array of allowed enums and does not constrain array length.
  Evidence: `llm_pipelines/schemas/line-role.canonical.v1.output.schema.json` has no `minItems` or `maxItems`.
- Observation: Inline repair is weaker than the initial turn because `SubprocessCodexExecRunner` drops `--output-schema` whenever `resume_last=True`.
  Evidence: `cookimport/llm/codex_exec_command_builder.py` omits `--output-schema` for resume commands; `runtime_workers.py` currently calls repair with `resume_last=True`.
- Observation: The existing runner already had the exact transport shape needed for the safer repair path.
  Evidence: `SubprocessCodexExecRunner` supports `prepared_execution_working_dir=...` with `resume_last=False`, which reuses the prepared workspace without replaying transcript history.
- Observation: The focused runtime contract did not depend on transcript resume itself; only the test expectations did.
  Evidence: After switching repair to fresh structured prompts, `tests/parsing/test_canonical_line_roles_runtime.py` still passed once `runner.calls` expectations and schema assertions were updated.

## Decision Log

- Decision: Fix this in transport/runtime rather than trying to solve it only with shard-size tuning.
  Rationale: The observed failure mode is a contract-enforcement gap. Smaller shards may reduce incidence, but they do not make a too-short `labels` array structurally impossible.
  Date/Author: 2026-04-10 / Codex
- Decision: Reuse the prepared execution workspace for repair, but do not resume prior transcript history.
  Rationale: The existing runner already supports a fresh structured prompt in an existing prepared workspace. This keeps workspace/path continuity while restoring full prompt and schema enforcement on repair.
  Date/Author: 2026-04-10 / Codex
- Decision: Add packet-local exact-length schemas for line-role initial and repair turns.
  Rationale: The row count is packet-specific, so the strongest boundary is a per-attempt schema that encodes the exact expected `labels` length at the CLI boundary rather than only in repo-side validation.
  Date/Author: 2026-04-10 / Codex

## Outcomes & Retrospective

The transport contract now matches the intended line-role authority boundary more closely. Initial and repair structured turns both get packet-local schemas that encode the exact expected `labels` length, so the model no longer gets a free pass on a too-short-but-otherwise-valid array at the CLI/schema boundary. Repair also now reuses the prepared workspace without resuming prior transcript history, which keeps local artifact paths stable while preserving schema enforcement on the follow-up call.

The focused validation evidence is strong for this scope: `pytest tests/parsing/test_canonical_line_roles_runtime.py` passed with the new runtime behavior, and `pytest tests/llm/test_codex_exec_runner.py` passed with the unchanged generic runner contracts. What remains outside this plan is empirical confirmation on another real benchmark run, which should now show repair attempts staying on packet-local exact-length schemas rather than slipping into the old weaker resume path.

## Context and Orientation

Canonical line-role inline JSON execution lives in `cookimport/parsing/canonical_line_roles/runtime_workers.py`. That module builds the packet, writes prompt artifacts, dispatches `codex exec`, validates the JSON response, and optionally dispatches repair turns for unresolved rows. The higher-level runtime owner is `cookimport/parsing/canonical_line_roles/runtime.py`, which resolves the base schema path through `resolve_codex_farm_output_schema_path(...)` and passes that path into the worker runtime.

The Codex subprocess seam lives in `cookimport/llm/codex_exec_runner.py` and `cookimport/llm/codex_exec_command_builder.py`. A “prepared execution workspace” means a sterile temporary working directory with mirrored files and an `AGENTS.md`; the runner can reuse that directory on later calls either by resuming transcript history or by starting a fresh prompt in the same directory. For this fix, the important distinction is that transcript resume currently removes `--output-schema`, while a fresh prompt in the same prepared workspace keeps it.

The static line-role schema file lives at `llm_pipelines/schemas/line-role.canonical.v1.output.schema.json`. Today it constrains label names but not array length. The live run therefore relies on repo validation in `runtime_workers.py` and `validation.py` to notice `wrong_label_count` after the model already returned a nominally valid JSON object.

Focused runtime tests already exist in `tests/parsing/test_canonical_line_roles_runtime.py`, and lower-level runner command tests exist in `tests/llm/test_codex_exec_runner.py`. Those tests already cover partial-label repair and prepared-workspace execution, so the safest implementation path is to tighten the existing contracts rather than rewrite the whole line-role repair story.

The initial turn should use an `output_schema_initial.json` file, and each repair attempt should use a distinct `output_schema_repair_XX.json` file so the artifacts stay inspectable.

Then update the targeted tests. The line-role runtime tests that currently expect `structured_prompt_resume` repair calls should instead expect fresh `structured_prompt` calls while still confirming that repair only sees unresolved rows. Add a regression test that asserts repair now records a non-null schema path and that the schema encodes the exact expected label count. Update any lower-level runner test assumptions only where the active runtime contract changed; the generic `resume_last` builder tests should remain because other transports may still use that mode.

Finally, update the short owning docs in `docs/10-llm/10-llm_README.md` and `docs/04-parsing/04-parsing_readme.md` so they describe the final behavior accurately: line-role initial and repair inline turns both enforce exact ordered label count, and repair reuses the same workspace without relying on transcript resume.

2. A line-role repair inline turn also receives packet-local exact-length schema enforcement.
3. Repair no longer runs through `codex exec resume --last`; it runs as a fresh structured prompt in the existing prepared workspace.

5. The focused parsing and runner tests pass in the project `.venv`.

## Idempotence and Recovery

The schema files written under each structured-session root are runtime artifacts, so regenerating them is safe. Re-running the targeted tests is safe. If an intermediate patch breaks repair behavior, the quickest rollback path is to compare the runtime call shape in `runtime_workers.py` against the test expectations for `runner.calls`, because that is the narrow seam where the transport behavior changes.

## Interfaces and Dependencies

In `cookimport/parsing/canonical_line_roles/runtime_workers.py`, define a helper that accepts the resolved base schema path, the session directory, a packet label such as `initial` or `repair_01`, and the expected row count, then returns a path to a packet-local schema JSON file. That file must preserve the original allowed-label enum while adding exact `minItems` and `maxItems` constraints on `properties.labels`.

The structured inline runtime in the same file must call `runner.run_packet_worker(...)` with:

    prepared_execution_working_dir=execution_workspace
    resume_last=False
    output_schema_path=<packet-local schema path>

for repair turns. Initial turns may keep `persist_session=True` so the prepared workspace still exists, but they should also move from the shared static schema path to the packet-local exact-length schema path.
