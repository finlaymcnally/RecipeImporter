# Row-Native Semantic Authority Migration

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

Related beads issue: `recipeimport-jmo` (“Migrate semantic pipeline from block authority to row authority”).

## Purpose / Big Picture

After this change, rows will be the only semantic unit in the program. A row is one labelable text unit from `cookimport/parsing/source_rows.py`, identified by `row_id` and ordered by `row_index`. Blocks will remain only as ingest and layout primitives: importers may still emit them, they may still carry page or table provenance, and table or chunk extraction may still inspect them when physical grouping matters. Blocks must stop deciding recipe boundary, recipe ownership, nonrecipe routing, final nonrecipe category, Label Studio gold, benchmark scoring, or AI prelabel behavior.

The user-visible outcome is simple. Running the pipeline, exporting Label Studio gold, and running benchmarks will produce row-native artifacts only. Operators will no longer see or select block-semantic benchmark modes such as `stage-blocks`, and Label Studio prelabel will no longer offer the older one-label-per-block mode. If a row belongs to a recipe, is excluded from nonrecipe review, or is promoted to knowledge, that decision will be recorded directly on rows. Any remaining block indices in artifacts will be descriptive provenance only.

## Outcomes & Retrospective

This plan now reflects a partial but real cutover. The active Label Studio operator surface is row-native: new freeform tasks emit `rows` plus row-context payloads, export no longer writes new block-gold artifacts, prelabel no longer accepts block granularity, and `labelstudio-benchmark` now normalizes to `source-rows` only. The supporting tests and docs were updated to match those contracts.

The migration is not complete yet. Parsing/staging row-authority cleanup and the deeper benchmark deletion sweep still remain, and block-semantic internals such as `eval_stage_blocks.py`, `block_gold.py`, and `stage_block_predictions.json` consumers still exist in the tree. The main lesson from implementation is that the remaining work is mostly deletion and authority-line cleanup, not inventing new row machinery.

## Context and Orientation

The current repository has two different kinds of units:

`source blocks` are the raw normalized document units produced by importers and stored in `ConversionResult.source_blocks`. They are written by `cookimport/core/source_model.py` into `raw/source/<workbook>/source_blocks.jsonl`. A block knows about physical document context such as page, table, markdown line range, or EPUB extraction metadata.

`source rows` are smaller labelable text units derived from source blocks by `cookimport/parsing/source_rows.py`. A row has a stable `row_id`, a global ordered `row_index`, a `row_ordinal` within its source block, and character offsets back into the block. Rows already exist so that Label Studio and source-row benchmarks can work on finer-grained text.

`semantic authority` means the set of outputs that downstream code is allowed to trust when deciding meaning. In this repository that includes recipe boundary, recipe ownership, nonrecipe candidate versus exclude routing, final nonrecipe `knowledge` versus `other`, Label Studio gold, AI prelabel output, and benchmark scoring inputs. Today, this authority is split. Rows exist, but many semantic steps still collapse back into `AuthoritativeBlockLabel`, `RecipeSpan.block_indices`, `RecipeOwnershipResult.block_owner_by_index`, `NonRecipeRoutingResult.candidate_block_indices`, and `stage_block_predictions.json`.

The key files for this migration are:

`cookimport/core/source_model.py` is the ingest bridge. It should continue normalizing importer output into canonical source blocks and writing `source_rows.jsonl`. It is allowed to stay block-based because it is below the semantic line.

`cookimport/parsing/source_rows.py` is the row builder. This file should remain the authoritative constructor for row units and should be strengthened if row identity or row payload needs to become richer.

`cookimport/parsing/label_source_of_truth.py` is where authoritative labels are built. It already holds `AuthoritativeLabeledLine`, but it also emits `AuthoritativeBlockLabel` and block-based recipe spans. This file is one of the main cutover points.

`cookimport/parsing/recipe_span_grouping.py` currently groups labeled rows back into block-index spans. This file must become row-native.

`cookimport/staging/recipe_ownership.py` currently defines recipe ownership on block indices. That must become row ownership.

`cookimport/staging/nonrecipe_stage.py` plus `nonrecipe_seed.py`, `nonrecipe_routing.py`, `nonrecipe_authority.py`, `nonrecipe_authority_contract.py`, and `nonrecipe_finalize_status.py` currently route and finalize nonrecipe content using block-index sets. These files must become row-native and may only use `source_block_index` as provenance.

`cookimport/labelstudio/freeform_tasks.py`, `cookimport/labelstudio/prelabel.py`, `cookimport/labelstudio/prelabel_mapping.py`, `cookimport/labelstudio/prelabel_parse.py`, `cookimport/labelstudio/prelabel_prompt.py`, and `cookimport/labelstudio/export.py` currently support both row and block payload shapes. This entire surface must become row-native.

`cookimport/bench/eval_source_rows.py` is the benchmark future. `cookimport/bench/eval_stage_blocks.py` is the benchmark past. The plan is to keep the former and delete the latter.

`cookimport/cli_commands/labelstudio.py` and `cookimport/cli_support/__init__.py` expose benchmark modes and prelabel options to operators. These files must stop presenting block-semantic choices.

There are also files that are allowed to stay block-oriented because they are ingest or layout helpers rather than semantic authorities: `cookimport/core/blocks.py`, `cookimport/parsing/markdown_blocks.py`, `cookimport/parsing/block_roles.py`, `cookimport/parsing/chunks.py`, `cookimport/parsing/tables.py`, and importer modules such as `cookimport/plugins/pdf.py` and `cookimport/plugins/epub.py`. These may continue using blocks as long as their outputs are not treated as semantic truth.

## Milestones

### Milestone 1: Define one row-native semantic contract

At the end of this milestone, the repository will have a single explicit rule: every semantic decision is recorded on rows. There may still be temporary adapters, but the canonical types and artifact names will be row-first. A novice should be able to inspect one run root and see row-native authority files rather than needing block-label mirrors to understand what happened.

This milestone covers `cookimport/parsing/label_source_of_truth.py`, `cookimport/parsing/recipe_span_grouping.py`, `cookimport/staging/recipe_ownership.py`, `cookimport/staging/nonrecipe_authority_contract.py`, `cookimport/staging/nonrecipe_stage.py`, and `cookimport/staging/writer.py`. The main deliverable is a replacement for `AuthoritativeBlockLabel`, block-index recipe spans, block ownership, and block-only nonrecipe authority. Every replacement type must use `row_id` or `row_index` as the primary key and may carry `source_block_index` only as descriptive provenance.

Acceptance for this milestone is structural and behavioral. A stage run should produce row-native semantic artifacts, and the old block-semantic artifacts should either be gone or be obvious temporary projections marked for deletion in the same branch.

### Milestone 2: Rewrite recipe boundary, ownership, and nonrecipe finalize to rows

At the end of this milestone, recipe and nonrecipe semantics will stop depending on block index sets. Recipe span grouping will open and close on ordered rows, recipe ownership and divestment will refer to row ranges or row sets, and nonrecipe routing/finalization will consume authoritative rows directly.

This is the heart of the migration because it changes what the pipeline believes. It affects `cookimport/parsing/recipe_span_grouping.py`, `cookimport/staging/recipe_ownership.py`, `cookimport/staging/nonrecipe_stage.py`, `cookimport/staging/nonrecipe_seed.py`, `cookimport/staging/nonrecipe_routing.py`, `cookimport/staging/nonrecipe_authority.py`, and `cookimport/staging/nonrecipe_finalize_status.py`. `cookimport/staging/stage_block_predictions.py` must either be deleted in this milestone or demoted to a short-lived shim that exists only long enough for later code in the same branch to be updated.

Acceptance for this milestone is a stage or prediction run where every recipe-owned row, candidate row, excluded row, and finalized knowledge row can be traced row-by-row without consulting block labels.

### Milestone 3: Make Label Studio row-native only

At the end of this milestone, Label Studio tasks, AI prelabel, export, and migration tools will operate on rows only. Operators should not be able to choose block prelabel mode, and exports should not emit `block_gold_labels.jsonl`.

This milestone covers `cookimport/labelstudio/freeform_tasks.py`, `cookimport/labelstudio/prelabel.py`, `cookimport/labelstudio/prelabel_mapping.py`, `cookimport/labelstudio/prelabel_parse.py`, `cookimport/labelstudio/prelabel_prompt.py`, `cookimport/labelstudio/export.py`, `cookimport/labelstudio/row_gold.py`, and `cookimport/labelstudio/block_gold.py`. The desired end state is that `source_map.rows` is the only supported labelable mapping, `touched_rows` is the only semantic export shape, and block aliases are removed.

Acceptance for this milestone is an export that contains `freeform_span_labels.jsonl`, `row_gold_labels.jsonl`, and `row_gold_conflicts.jsonl`, but no block-gold artifact, plus a prelabel path that no longer offers or accepts the older one-label-per-block contract.

### Milestone 4: Remove block-based benchmark and scoring paths

At the end of this milestone, the benchmark system will score rows only. The CLI, docs, support utilities, and upload/follow-up tooling must stop referring to `stage-blocks`.

This milestone covers `cookimport/bench/eval_source_rows.py`, `cookimport/bench/eval_stage_blocks.py`, `cookimport/cli_commands/labelstudio.py`, `cookimport/cli_support/__init__.py`, benchmark helper modules in `cookimport/cli_support/`, and cutdown/follow-up utilities that still read block predictions or block-gold artifacts. `cookimport/bench/eval_stage_blocks.py` should be deleted. Any code that currently expects `stage_block_predictions.json` must be rewritten to consume row-native predictions and row-native final authority.

Acceptance for this milestone is a benchmark CLI surface that exposes only row-native evaluation, an upload bundle built from row-native evidence, and docs that no longer describe block-index parity as a semantic benchmark concern.

### Milestone 5: Delete block-semantic compatibility code and transfer durable knowledge

At the end of this milestone, the branch will no longer contain temporary row-to-block semantic projections. The remaining block-oriented code will be clearly below the semantic line: importers, layout, tables, markdown parsing, and chunk/table grouping only.

This milestone includes the final deletion sweep, doc updates in `docs/01-architecture/`, `docs/05-staging/`, `docs/06-label-studio/`, `docs/07-bench/`, and `docs/12-testing/`, and the creation of the final beads issue that tracks deletion of this ExecPlan once the user explicitly approves removing it.

Acceptance for this milestone is that a repo search for operator-facing or semantic block terms such as `stage-blocks`, `block_gold_labels`, and block-semantic ownership files returns only ingest/layout uses or historical test fixtures that are explicitly retained for importer behavior.

## Plan of Work

Begin by defining the replacement semantic contract before touching CLI or docs. In `cookimport/parsing/label_source_of_truth.py`, replace the block-semantic public data model with row-native equivalents. The repository already has `AuthoritativeLabeledLine`, so the plan is to promote that type family rather than invent a second row model. Replace `AuthoritativeBlockLabel` with a row-native summary type only if a summary is still needed; otherwise remove it and let higher layers consume `AuthoritativeLabeledLine` directly. Rewrite `RecipeSpan` and `RecipeSpanDecision` so that recipe boundary records use ordered row coordinates and can optionally include derived block provenance for display.

Once the parsing contract is row-native, rewrite recipe grouping and ownership. In `cookimport/parsing/recipe_span_grouping.py`, stop grouping over sorted block labels and instead group over sorted authoritative rows. In `cookimport/staging/recipe_ownership.py`, replace `owned_block_indices`, `divested_block_indices`, and `block_owner_by_index` with row-native equivalents such as `owned_row_indices`, `divested_row_indices`, and `row_owner_by_index`. Keep helper methods such as `is_row_recipe_owned` and `assert_row_is_available_to_nonrecipe`, and remove block-language from method names so downstream code cannot accidentally keep thinking in blocks.

After ownership is row-native, rewrite nonrecipe routing and finalization to consume rows directly. In `cookimport/staging/nonrecipe_stage.py` and the smaller owner modules beside it, change the primary inputs from `AuthoritativeBlockLabel` sequences and block-index maps to ordered authoritative rows plus recipe row ownership. The router should classify rows into coarse routes such as candidate or exclude. The finalize step should classify only candidate rows into final semantic categories such as `knowledge` or `other`. If a reporting summary still wants contiguous block ranges, derive them at write time from row provenance rather than making them the runtime truth.

With staging converted, remove block-semantic artifacts from writer and prediction-run code. In `cookimport/staging/writer.py` and `cookimport/labelstudio/ingest_flows/artifacts.py`, stop writing `stage_block_predictions.json`, block ownership files, and block-label mirrors. Replace them with row-native authority outputs whose filenames say what they are. The exact names can be chosen during implementation, but they should be explicit and boring, for example `row_label_predictions.jsonl`, `recipe_row_ownership.json`, `nonrecipe_row_route.json`, and `nonrecipe_row_authority.json`. The important part is that the file names and payload keys stop implying block authority.

Then make Label Studio fully row-native. In `cookimport/labelstudio/freeform_tasks.py`, keep `source_map.rows` and remove `source_map.blocks` plus `context_before_blocks` and `context_after_blocks`. In the prelabel modules, remove `PRELABEL_GRANULARITY_BLOCK`, all prompts or parser paths that expect one label per block, and any fallbacks that say `rows/blocks`. In `cookimport/labelstudio/export.py`, export `touched_rows` only and stop duplicating the same payload under `touched_blocks`. Delete `cookimport/labelstudio/block_gold.py` and remove its call sites after all imports have been updated.

Finally, remove the block benchmark lane. Delete `cookimport/bench/eval_stage_blocks.py`, remove the `stage-blocks` benchmark mode from CLI parsing and docs, and update any benchmark helper or cutdown utility that still expects block predictions. The row-native benchmark reader should use authoritative row labels and final nonrecipe row authority only. Once this is done, delete temporary adapters and sweep docs and tests so the repository tells one story.

## Validation and Acceptance

The migration is accepted only when all of the following are true.

First, stage and prediction runs must be row-semantic. A processed output run should contain row-native semantic artifacts only. A novice should be able to inspect the run root and answer these questions without reading block-label artifacts: which rows belong to recipes, which rows were excluded from nonrecipe review, which rows were reviewed as nonrecipe candidates, and which rows ended as `knowledge` versus `other`.

Second, Label Studio must be row-native. Export output should include row-gold artifacts and no block-gold artifact. Prelabel should expose only actual row span labeling, not one-label-per-block mode. The task payload contract should describe only rows as labelable units.

Third, benchmarking must be row-native. `cookimport labelstudio-benchmark --help` should no longer present `stage-blocks`, and the benchmark docs under `docs/07-bench/` should no longer describe block-index parity as a semantic benchmark requirement. Any offline benchmark smoke or helper tests should read row predictions and row gold only.

Fourth, the test suite must prove the new behavior. At minimum, the implementation should keep these slices green:

    . .venv/bin/activate
    ./scripts/test-suite.sh domain parsing
    ./scripts/test-suite.sh domain staging
    ./scripts/test-suite.sh domain labelstudio
    ./scripts/test-suite.sh domain bench
    ./scripts/test-suite.sh smoke

Fifth, the repository language must reflect the new system. The architecture, staging, Label Studio, bench, and testing docs must all describe rows as the sole semantic unit and must describe blocks only as ingest, layout, provenance, or table/chunk context.

## Idempotence and Recovery

This migration should be implemented as a sequence of additive replacements followed immediately by deletions. Do not keep a long-lived dual system. If a milestone is only half complete, the plan is to keep the temporary adapter inside the branch and remove it before moving to the next major milestone.

If the refactor stalls mid-migration, recover by re-reading this ExecPlan, verifying which checked items in `Progress` are true, and searching the repository for the deletion markers listed in `Concrete Steps`. The quickest sanity check is the `rg` command that looks for block benchmark and block prelabel language. If those phrases are still present in live modules, the migration is not complete.

If tests fail during the cutover, classify each failure before editing code. Record whether it is a regression, an intentional contract change, or an outdated overspecified test. The migration should prefer changing tests only when the old behavior was block-semantic by design and is no longer part of the intended product.

## Interfaces and Dependencies

The final repository must expose row-native semantic interfaces. The exact type names may change during implementation, but the end state must include row-native equivalents of these concepts:

In `cookimport/parsing/label_source_of_truth.py`, define or retain a row-native authoritative label model whose primary identity is `row_id` or `row_index`, not `source_block_index`.

In `cookimport/parsing/recipe_span_grouping.py`, the main grouping entry point must accept ordered authoritative rows and return row-native recipe span decisions. A row-native recipe span must be able to report its ordered row indices and may optionally include derived source-block provenance for debugging.

In `cookimport/staging/recipe_ownership.py`, define a row-native ownership result with methods equivalent to:

    is_row_recipe_owned(row_index: int) -> bool
    assert_row_is_available_to_nonrecipe(row_index: int) -> None
    assert_row_is_recipe_owned(row_index: int) -> None

In `cookimport/staging/nonrecipe_authority_contract.py`, define row-native routing, final authority, and finalize-status results. Any remaining block summaries must be clearly marked as derived reporting views rather than runtime authority.

In `cookimport/labelstudio/freeform_tasks.py`, the task payload must expose `source_map.rows`, `source_map.context_before_rows`, and `source_map.context_after_rows`. The implementation must not require `source_map.blocks`.

In `cookimport/labelstudio/prelabel.py` and related modules, the supported prelabel granularity must be one row-native span contract. Do not preserve a block-mode alias.

In `cookimport/bench/eval_source_rows.py`, the evaluator must remain the benchmark source of truth. `cookimport/bench/eval_stage_blocks.py` must not survive the migration.
