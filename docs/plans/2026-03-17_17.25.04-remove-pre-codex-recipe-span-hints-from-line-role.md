# Remove Pre-Codex Recipe-Span Hints From Line Role

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

After this change, RecipeImport's label-first pipeline will match the intended architecture: deterministic line labeling runs across the whole book without inheriting old recipe boundaries, Codex line-role reviews that full deterministic labeling without prior recipe-span authority, and only after corrected labels exist does the repo derive recipe spans. A human should be able to verify this by inspecting a line-role prompt and seeing that it no longer says which rows are already inside recipe, then inspecting the resulting grouped spans and seeing that recipe grouping is derived from corrected labels rather than importer provenance.

## Context and Orientation

The label-first pipeline currently begins in `cookimport/parsing/label_source_of_truth.py`. `build_label_first_stage_result(...)` atomizes archive blocks, runs deterministic plus Codex line-role labeling, collapses line labels to block labels, and then groups recipe spans with `cookimport/parsing/recipe_span_grouping.py`.

The architectural problem is in `_atomize_archive_blocks(...)` inside `cookimport/parsing/label_source_of_truth.py`. That function currently reads recipe ranges from `_recipe_ranges_from_conversion_result(...)`, which pulls `start_block` and `end_block` out of already-existing recipe provenance on `conversion_result.recipes`. That means the first line-role pass is not really operating on free global labels; it inherits old recipe ranges.

`cookimport/labelstudio/ingest.py` has a second copy of the same idea in `_build_line_role_candidates_from_archive(...)`, where archive rows are atomized with `within_recipe_span` seeded from old recipe ranges. `cookimport/llm/prompt_preview.py` also reconstructs line-role prompt candidates using existing recipe spans; after this refactor, preview must mirror the new live contract and stop supplying pre-grouped recipe membership to line-role reconstruction.

`cookimport/parsing/recipe_block_atomizer.py` owns `AtomicLineCandidate`, the data structure representing one atomic line. Right now it requires `within_recipe_span: bool` and emits rule tags such as `outside_recipe_span` during atomization. That is the wrong contract for the intended architecture because it bakes recipe membership into the candidate before grouping exists.

`cookimport/parsing/canonical_line_roles.py` owns deterministic line-role heuristics, Codex prompt planning, shard execution, sanitization, and line-role telemetry. The current worktree has already removed post-Codex rollback logic, which is good. The remaining task here is to make the deterministic and prompt layers behave correctly when recipe membership is unknown before grouping. In practice that means removing prompt text that says which atomic ranges are already recipe, changing span checks from truthy/falsy assumptions to explicit inside/outside/unknown handling, and ensuring global deterministic rules still label obvious ingredient, instruction, title, yield, variant, and note rows without needing old recipe ranges.

The user-visible acceptance surface is not only tests. It is also observable artifacts:

- a line-role prompt under `line-role-pipeline/prompts/` must no longer say `Any atomic index outside those ranges is outside recipe`
- initial `label_det` and `label_llm_correct` outputs must no longer inherit importer recipe membership
- grouped spans in `group_recipe_spans/<slug>/recipe_spans.json` must still be produced, but now from corrected labels only

## Plan of Work

First, change the initial candidate contract to represent recipe membership as unknown before grouping. Edit `cookimport/parsing/recipe_block_atomizer.py` so `AtomicLineCandidate.within_recipe_span` accepts `bool | None`, and teach `_infer_rule_tags(...)` to emit neither inside-recipe nor outside-recipe tags when the state is unknown. This is the foundational change because every pre-grouping consumer reads those candidates.

Next, remove the provenance-based seeding in `cookimport/parsing/label_source_of_truth.py`. `_atomize_archive_blocks(...)` should stop calling `_recipe_ranges_from_conversion_result(...)` and should atomize every archive block with `recipe_id=None` and `within_recipe_span=None`. Once that is done, delete the no-longer-needed helper functions that recover recipe ranges from the old conversion result.

Apply the same intended-only contract in `cookimport/labelstudio/ingest.py`. `_build_line_role_candidates_from_archive(...)` should stop reconstructing recipe ranges from the archive payload and old conversion result. It should emit unknown recipe membership for all pre-grouping line-role candidates, mirroring `label_source_of_truth.py`.

Then update `cookimport/llm/prompt_preview.py` so prompt-preview reconstruction matches the new live behavior. When preview reconstructs the line-role phase from blocks or stored labeled lines, it must stop injecting grouped recipe spans back into the pre-grouping line-role candidate set. If preview still wants recipe-span-aware diagnostics, that should be a separate downstream reporting surface, not the prompt reconstruction contract.

After that, edit `cookimport/llm/canonical_line_role_prompt.py` and `cookimport/parsing/canonical_line_roles.py` together. The prompt must stop advertising prior recipe atomic ranges as authoritative. The deterministic rules must change all truthy/falsy span checks to explicit inside/outside/unknown semantics so unknown does not get misread as outside recipe. Rules that genuinely require “inside accepted recipe” behavior should only fire on explicit `True`. Rules that genuinely require “outside accepted recipe” behavior should only fire on explicit `False`. Unknown candidates should still receive obvious structural labels from content and neighbor context.

Finally, update docs and tests. `docs/04-parsing/04-parsing_readme.md` and `docs/10-llm/10-llm_README.md` need to say the old provenance-hint path is gone. Parsing tests, Label Studio ingest tests, and prompt-preview tests must be rewritten to expect unknown pre-grouping recipe membership and grouping-after-correction behavior.

## Interfaces and Dependencies

`cookimport/parsing/recipe_block_atomizer.py` must end this work with an `AtomicLineCandidate` model that supports unknown recipe membership before grouping. `cookimport/parsing/label_source_of_truth.py` must end this work without `_recipe_ranges_from_conversion_result(...)` driving pre-grouping line-role candidate construction. `cookimport/labelstudio/ingest.py` and `cookimport/llm/prompt_preview.py` must share that same contract so live execution, ingest, and preview behave the same way.

`cookimport/parsing/canonical_line_roles.py` must treat recipe membership as a three-state concept before grouping:

- `True`: explicitly inside an already-derived recipe span, used only for downstream or grouped projections
- `False`: explicitly outside a derived recipe span, used only where a later stage has already derived that fact
- `None`: unknown before grouping, which is the only valid state for the initial deterministic plus Codex line-role pass
