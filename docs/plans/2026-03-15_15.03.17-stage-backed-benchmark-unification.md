# Make Benchmarking A Stage-Backed Import Run

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a benchmark run will mean exactly one thing: run the real import pipeline once, write the normal stage artifacts once, and score those artifacts against the existing gold spans. The benchmark command must stop generating a second, benchmark-only processed-output lane that can diverge from live import behavior.

The user-visible proof is straightforward. Running `cookimport labelstudio-benchmark --gold-spans ... --source-file ... --no-upload` should create one authoritative stage run under `data/output/<timestamp>/...` and one benchmark eval root under `data/golden/benchmark-vs-golden/<timestamp>/...`. The benchmark report must be computed from the stage run’s `.bench/<workbook_slug>/stage_block_predictions.json` plus the stage run’s extracted block text, not from a separate prediction-only projection lane. Running `cookimport bench eval-stage --gold-spans ... --stage-run data/output/<timestamp>` against that same stage run should produce the same stage-backed score inputs.

This plan explicitly does not change the gold exports. The existing freeform and canonical gold spans remain the source of truth. The work is to change which prediction artifacts are scored, not to regenerate or reinterpret the gold data.

## Surprises & Discoveries

- Observation: the main duplication is not limited to line-role. `generate_pred_run_artifacts()` reimplements import conversion, optional Codex recipe pass, optional pass4 knowledge harvest, processed-output writing, stage-block prediction writing, and run-manifest assembly instead of reusing the stage path.
  Evidence: `cookimport/cli_worker.py::stage_one_file()` and `cookimport/labelstudio/ingest.py::_write_processed_outputs()` contain materially similar writing sequences: intermediate drafts, final drafts, sections, tips, topic candidates, tables, chunks, raw artifacts, and `.bench/<slug>/stage_block_predictions.json`.

- Observation: canonical benchmark scoring does not require the benchmark-only canonical line-role lane.
  Evidence: `cookimport/bench/eval_canonical_text.py` scores from two inputs only: a stage-label manifest and extracted blocks text. The evaluator does not require `line-role-pipeline/stage_block_predictions.json`; that file is only a benchmark preference override selected in `cookimport/cli.py::_build_prediction_bundle_from_import_result(...)`.

- Observation: the current benchmark semantics are wrong for the requested goal because benchmark can write processed outputs after calling `apply_line_role_spans_to_recipes(...)`.
  Evidence: in `cookimport/labelstudio/ingest.py`, canonical line-role runs before `_write_processed_outputs(...)`, and recipe mutation is gated only by `processed_output_root is not None`, which means benchmark-produced cookbook outputs can differ from a real import before scoring even begins.

- Observation: downstream tooling is coupled much more to `prediction-run/` layout than to the actual import process.
  Evidence: tests and helpers in `tests/labelstudio/test_labelstudio_benchmark_helpers_*`, `tests/bench/test_benchmark_cutdown_for_external_ai.py`, `cookimport/analytics/perf_report.py`, and `cookimport/bench/followup_bundle.py` all assume a co-located `prediction-run/` directory with benchmark-facing manifests and artifact paths.

- Observation: the cleanest unification target is “stage run as source of truth, benchmark root as report shell,” not “teach stage to become benchmark.”
  Evidence: the repo already has a direct stage-eval command in `cookimport/cli.py::bench_eval_stage`. That proves the evaluation side can read a stage run directly. The missing piece is making `labelstudio-benchmark` produce or reference that stage run instead of creating a second processed-output lane.

- Observation: the low-risk shared seam was “post-conversion stage session,” not “one giant import wrapper.”
  Evidence: routing `stage_one_file()`, split-job merge output writing, and benchmark processed-output writing through `cookimport/staging/import_session.py` eliminated the duplicated stage artifact writer sequence without forcing benchmark task-generation or split conversion orchestration into the same function.

- Observation: moving canonical line-role after the shared stage session fixes two problems at once.
  Evidence: benchmark now avoids mutating authoritative stage outputs before scoring, and `tests/labelstudio/test_labelstudio_ingest_parallel.py` now proves that line-role candidates can still be rebuilt from the post-session `ConversionResult` while leaving final drafts unchanged.

## Context and Orientation

This repository has two different ways to produce import-like artifacts today.

The normal stage path lives mostly in `cookimport/cli_worker.py::stage_one_file()` and the split-merge helpers in `cookimport/cli.py`. It runs importer conversion, optional Codex recipe repair, optional Codex knowledge harvest, chunk/table generation, and then writes one stage run under `data/output/<timestamp>/...`. That run includes final cookbook outputs and `.bench/<workbook_slug>/stage_block_predictions.json`, which is the deterministic block-label evidence used by benchmark scoring.

The benchmark prediction path lives in `cookimport/labelstudio/ingest.py::generate_pred_run_artifacts()`. It starts from the same source conversion idea, but then it does its own post-processing and artifact writing under a benchmark root. It can also run canonical line-role on archive-derived atomic lines, project those labels back into recipes, mutate the `ConversionResult`, write a second processed-output tree, and then prefer that projection artifact for canonical scoring. That is the semantic divergence the user wants removed.

In this plan, a stage run means the normal import output root under `data/output/<timestamp>/...`. A benchmark eval root means the report directory under `data/golden/benchmark-vs-golden/<timestamp>/...` that stores evaluation reports, compare-control files, and benchmark-only diagnostics. A prediction run means the benchmark-facing artifact directory currently co-located under `prediction-run/`. Today that prediction run is too powerful; after this plan it should become a compatibility view over the stage run rather than a second import lane.

The important files are:

`cookimport/cli_worker.py` and `cookimport/cli.py`. These are the current stage entrypoints and split orchestration.

`cookimport/labelstudio/ingest.py`. This currently owns prediction-run generation, Label Studio task generation, and the benchmark-only line-role fork.

`cookimport/staging/writer.py`. This is the canonical stage artifact writer, including `.bench/<workbook_slug>/stage_block_predictions.json`.

`cookimport/bench/eval_stage_blocks.py` and `cookimport/bench/eval_canonical_text.py`. These are the two evaluators that should both read stage-backed evidence after the refactor.

`cookimport/bench/followup_bundle.py`, `scripts/benchmark_cutdown_for_external_ai.py`, and `cookimport/analytics/perf_report.py`. These are downstream consumers that currently look for benchmark data under `prediction-run/`.

`tests/labelstudio/test_labelstudio_ingest_parallel.py`, `tests/labelstudio/test_labelstudio_benchmark_helpers_*`, `tests/bench/test_eval_stage_blocks.py`, `tests/bench/test_benchmark_cutdown_for_external_ai.py`, `tests/analytics/test_stats_dashboard.py`, and `tests/cli/test_run_settings_adapters.py`. These are the regression-sensitive tests most likely to fail during this work.

## Plan of Work

Create a new shared module, preferably `cookimport/staging/import_session.py`, that defines one callable import-session contract used by both stage and benchmark flows. This module should own the common work that is currently duplicated across `stage_one_file()` and `generate_pred_run_artifacts()`: importer conversion, split-job merge normalization, optional Codex recipe pass, optional knowledge harvest, chunk/table derivation, report enrichment, prepared archive construction, and stage artifact writing.

The module should expose one plain result object, for example `StageImportSessionResult`, with enough information for both callers. At minimum it must include the in-memory `ConversionResult`, the prepared archive payload or object used for freeform tasks, the stage run root path, the report path, the stage-block predictions path, the source hash, the importer name, the resolved run config payload, and timing metadata. Keep this result repo-owned and explicit. Do not make benchmark callers scrape those values back out of `manifest.json`.

Refactor `cookimport/cli_worker.py::stage_one_file()` to become a thin wrapper around the shared import-session runner. The stage command should still behave exactly as it does now, including markdown writes, stage-block predictions, report generation, and optional Codex passes. The difference is that benchmark code must be able to call the same lower-level implementation directly.

Then refactor `cookimport/labelstudio/ingest.py::generate_pred_run_artifacts()` into two layers. The first layer should call the shared import-session runner and receive the stage-backed result. The second layer should handle only benchmark-specific extras that genuinely belong to benchmark or Label Studio workflows: freeform task generation, optional prelabeling, upload preparation, benchmark manifests, and compatibility artifact materialization under `prediction-run/`. It must stop writing a second processed-output tree. The field currently called `processed_run_root` should become the authoritative stage run root, not a benchmark-only copy.

After the shared session exists, change `cookimport/cli.py::labelstudio_benchmark()` so evaluation always uses the stage run’s evidence. For `stage-blocks` mode, use the stage run’s `.bench/<workbook_slug>/stage_block_predictions.json` and extracted blocks file directly. For `canonical-text`, call `evaluate_canonical_text(...)` with the same stage predictions plus the same extracted blocks file. Do not prefer `line-role-pipeline/stage_block_predictions.json` for primary scoring anymore.

The extracted blocks file should come from the stage session. If the current raw stage artifact `raw/**/full_text.json` is sufficient for both evaluators, keep using it. If benchmark helpers genuinely need a stable `extracted_archive.json` filename under `prediction-run/`, write that file as a compatibility mirror generated from the same prepared archive already returned by the shared session. The key rule is that this file must be a view of the stage session, not evidence from a second parsing or classification pass.

Next, remove the benchmark-only recipe mutation from the primary path. `apply_line_role_spans_to_recipes(...)` must not run before the authoritative processed outputs are written for benchmark scoring. The simplest and preferred outcome is to reject `line_role_pipeline != off` for the main benchmark command and document that the stage-backed benchmark no longer supports line-role-as-primary-score. If historical or experimental support is still needed, move it behind an explicitly non-authoritative experimental command or a side-diagnostics path that never overwrites the main score inputs and never mutates processed outputs before the stage write.

Then create a compatibility envelope under `prediction-run/`. This envelope should contain only the benchmark-facing files that existing tooling still expects, such as `manifest.json`, `coverage.json`, `label_studio_tasks.jsonl`, optional prelabel artifacts, and small mirrored files like `stage_block_predictions.json` or `extracted_archive.json` if needed for compatibility. It should not contain a second processed output tree. Manifest fields such as `processed_run_root`, `processed_stage_block_predictions_path`, and prompt/log references must point back to the true stage run when appropriate.

Update downstream readers to treat the stage run as the source of truth. `cookimport/analytics/perf_report.py`, dashboard collection, cutdown packaging, follow-up bundle building, and compare-control helpers should prefer manifest pointers first and only fall back to hardcoded `prediction-run/` paths for older roots. This lets new runs be stage-backed without breaking historical benchmark data.

Finally, update docs and CLI help so the product meaning is unambiguous: benchmarking now scores a real import run; line-role projection is not the primary benchmark lane; and existing gold spans are unchanged.

## Milestones

### Milestone 1: Extract the shared import-session runner

At the end of this milestone, there is one lower-level implementation that both stage and benchmark callers can invoke. A novice contributor can read one module and understand how importer conversion, Codex recipe pass, knowledge harvest, prepared archive construction, and stage artifact writing fit together.

The proof is that `cookimport stage` still writes the same output contract as before, but the code path now flows through the new shared module. Focus on the single-file path first, then pull split-job merge orchestration into the same contract so benchmark and stage no longer drift there either.

### Milestone 2: Make `labelstudio-benchmark` stage-backed

At the end of this milestone, benchmark prediction generation no longer writes a benchmark-only processed-output lane. Instead, `labelstudio-benchmark` either creates one real stage run through the shared import session or accepts an existing stage run for direct evaluation. Both `stage-blocks` and `canonical-text` must read that stage run’s evidence.

The proof is behavioral. Running benchmark offline produces one real stage run under `data/output/...` and one benchmark eval root under `data/golden/benchmark-vs-golden/...`, with the benchmark report sourced from the stage run’s artifacts.

### Milestone 3: Shrink `prediction-run/` into a compatibility shell

At the end of this milestone, `prediction-run/` still exists for tooling compatibility, but it no longer implies an independent import lane. Its manifests and mirrored files are derived from the stage-backed session. No second cookbook output snapshot is written just for benchmarking.

The proof is that existing helper tests which inspect `prediction-run/` still pass, while the manifest clearly points back to the authoritative stage run.

### Milestone 4: Remove benchmark-only semantic forks from the primary score

At the end of this milestone, the main benchmark path does not mutate recipes via line-role projection before writing processed outputs, and the primary benchmark score does not prefer line-role projection artifacts over stage-run evidence. Historical runs may still be readable, but new benchmark runs must measure live import behavior.

The proof is twofold. First, the delta described in `docs/reports/2026-03-15_14.58.07-benchmark-vs-live-import-delta-audit.md` no longer applies to new runs. Second, running `bench eval-stage` directly on the produced stage run uses the same score inputs that `labelstudio-benchmark` used.

## Validation and Acceptance

Acceptance is behavior-first.

First, a benchmark run must create or attach to one real stage run. The benchmark report must name that stage run in its manifest, and the stage run must contain the authoritative `.bench/<workbook_slug>/stage_block_predictions.json` used for scoring.

Second, the primary benchmark score must be reproducible from the stage run alone. Running `cookimport bench eval-stage --gold-spans ... --stage-run <that stage run>` must operate on the same stage-backed evidence surface. For canonical-text, the benchmark command must call `evaluate_canonical_text(...)` with the stage run’s stage predictions and extracted blocks, not a benchmark-only projection artifact.

Third, the benchmark command must no longer write a second processed-output tree whose contents can diverge from the real import run. A compatibility `prediction-run/` directory is acceptable during migration, but it must be a shell around the stage run, not a second import output.

Fourth, the gold exports must remain unchanged. Existing `freeform_span_labels.jsonl`, `canonical_text.txt`, `canonical_span_labels.jsonl`, and `canonical_manifest.json` remain valid inputs; no plan step should regenerate them or require changing their schema.

Fifth, if canonical line-role survives at all, it must be clearly non-authoritative. It may write diagnostics, but it must not mutate the real stage outputs before the main benchmark score is computed.

## Idempotence and Recovery

This plan should be implemented additively. The safest order is: introduce the shared import-session module first, refactor stage to use it, then switch benchmark to use it, and only then remove old benchmark-only behavior. This sequence keeps the normal stage command stable while benchmark migrates.

Keep manifest fields and compatibility paths until downstream readers are updated. New code should prefer explicit manifest pointers to the stage run, but old fallback paths can remain temporarily so historical benchmark roots still work.

If a migration step fails midway, the recovery path is to keep the shared import-session module in place and temporarily leave the old `generate_pred_run_artifacts()` wrapper behavior intact while restoring benchmark callers to the last known-good path. Do not delete old benchmark-reader fallbacks until the new stage-backed manifests are proven in tests.

## Artifacts and Notes

The shared module introduced by this plan should end in a shape close to this:

    @dataclass
    class StageImportSessionResult:
        run_root: Path
        workbook_slug: str
        source_file: Path
        source_hash: str
        importer_name: str
        conversion_result: ConversionResult
        prepared_archive: PreparedArchive
        report_path: Path
        stage_block_predictions_path: Path
        run_config: dict[str, Any]
        run_config_hash: str | None
        run_config_summary: str | None
        llm_report: dict[str, Any]
        timing: dict[str, Any]

The important part is not the exact class name. The important part is that both `stage` and benchmark callers receive one explicit object instead of reconstructing context from manifests or rewriting the same artifact sequence in different places.

The benchmark manifest for new runs should include both the eval-root-local artifact pointers and the authoritative stage-run pointer, for example:

    {
      "artifacts": {
        "stage_run_dir": "../../data/output/2026-03-15_12.34.56",
        "stage_block_predictions_json": "../../data/output/2026-03-15_12.34.56/.bench/book/stage_block_predictions.json",
        "prediction_manifest_json": "prediction-run/manifest.json"
      }
    }

The exact relative paths may differ. What matters is that the manifest makes the stage run explicit and that downstream tooling can follow that pointer.
