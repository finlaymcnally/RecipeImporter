# Shrink The Run-Settings Surface And Make Table Extraction Always-On

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this work, the product should feel smaller and more honest. A normal operator should make a short list of real choices such as extractor, OCR policy, and whether Codex recipe or knowledge passes are enabled, instead of carrying around dozens of implementation seams as if they were product features. The most immediate user-visible change is that deterministic table extraction becomes normal behavior instead of an opt-in flag. A user should be able to run:

    cookimport stage data/input/thefoodlabCUTDOWN.epub --write-markdown

and then open `data/output/<timestamp>/tables/thefoodlabcutdown/tables.md` without ever remembering `--table-extraction on`. The same principle should hold across benchmark prediction generation: additive product behavior should happen by default, and new manifests should stop implying that “table extraction off” is a meaningful alternate product mode.

The practical success signal is threefold. First, `cookimport stage --help` and `cookimport labelstudio-benchmark --help` should no longer advertise retired knobs such as `--table-extraction`. Second, new run manifests and summaries should primarily show real operator choices instead of low-level plumbing. Third, old run-settings files and old manifests must remain loadable so prior work does not become unreadable or un-rerunnable.

- Observation: the current codebase already behaves like the real product is mostly “CodexFarm” versus “Vanilla,” even though `RunSettings` exposes 78 fields.
  Evidence: `cookimport/cli_ui/run_settings_flow.py` primarily selects a top-tier profile family and then normalizes it through `apply_top_tier_profile_contract(...)`.

- Observation: `table_extraction` already behaves like rollout debt rather than a durable user choice.
  Evidence: the downstream feature is fully wired (`tables.py`, chunking, staging writer, pass4 `table_hint`), but saved Food Lab benchmark roots still show `table_extraction=off`, which made the feature look broken when it simply never ran.

- Observation: some advertised settings are already effectively frozen.
  Evidence: the docs say `benchmark_sequence_matcher` is `dmp` only, which means it is not a real product choice despite being carried as a top-level run setting.

- Observation: the repo needs two settings views, not one.
  Evidence: full `run_config` payloads still carry runtime metadata and internal defaults, but human-facing summaries became much easier to read once they projected only the public run-settings surface instead of sorting every stored key.

## Outcomes & Retrospective

Implementation landed in a compatibility-preserving shape. `RunSettings` now distinguishes public versus internal fields in code, UI specs default to the public subset, and human-readable run summaries use public-only projections instead of dumping every stored knob. `table_extraction` was fully retired as a live setting: new runs no longer serialize it, old payloads with the key still load with a retirement warning, and stage/prediction-generation now always write table artifacts.

The first internal-only tranche is also in place. `benchmark_sequence_matcher`, `multi_recipe_trace`, `p6_emit_metadata_debug`, the codex-farm pipeline-id knobs, and the benchmark selective-retry knobs remain accepted/runtime-wired but are no longer ordinary operator choices in UI/help. The main remaining gap is validation scope: targeted tests and a real stage run passed, but a full end-to-end CLI benchmark replay is still outstanding before this plan can be called fully complete.

## Context and Orientation

In this repository, `RunSettings` is the canonical settings model used for interactive run selection, benchmark planning, report manifests, analytics summaries, quality-suite persistence, and stage/benchmark adapters. That model lives in `cookimport/config/run_settings.py`. A “run setting” here means a field that can be loaded from JSON, shown in docs or menus, propagated into stage or benchmark execution, and recorded in run manifests.

Several other modules matter for this cleanup. `cookimport/config/run_settings_adapters.py` converts `RunSettings` into keyword arguments for `cli.stage(...)` and `cli.labelstudio_benchmark(...)`. `cookimport/cli.py` contains the actual Typer CLI options, manifest-writing logic, stage orchestration, and benchmark orchestration. `cookimport/cli_ui/run_settings_flow.py` contains the interactive selector flow, which already mostly collapses the product down to top-tier profiles. `cookimport/config/codex_decision.py` holds the profile patches and benchmark contract patches that normalize many settings behind the scenes. `cookimport/cli_worker.py` and `cookimport/labelstudio/ingest.py` are relevant because they still branch on `table_extraction` today.

This plan uses three plain-English terms repeatedly.

- A “public setting” is a knob a normal human should reasonably choose at run time. Example: whether Codex recipe parsing is enabled.
- An “internal tuning setting” is a real engineering control that may still matter for experiments or debugging, but should not be advertised as part of the normal product surface. Example: parser backend selection after a winner is chosen.
- A “retired legacy key” is a setting name that older payloads may still contain, but that new runs should not present as a live option anymore. Example: the current `table_extraction` key after this plan is implemented.

The March 13 audit note at `docs/understandings/2026-03-13_22.48.50-run-settings-surface-audit.md` is the diagnosis this plan operationalizes. That audit counted 78 `RunSettings` fields, of which 75 are currently user-visible, and grouped them into buckets that should be hardcoded, hidden/internal, or kept public. This ExecPlan turns that audit into concrete code, docs, compatibility, and validation work.

## Plan of Work

### Milestone 1: Teach The Codebase What “Public” Means

At the end of this milestone, the repository should stop assuming that every `RunSettings` field deserves equal visibility. A contributor should be able to inspect `cookimport/config/run_settings.py` and see explicit metadata or helpers that distinguish public operator settings from internal tuning settings and from retired legacy keys.

Implement this in `cookimport/config/run_settings.py`. Keep `RunSettings` as the compatibility model for loading and validating settings payloads, but add a stronger field-classification layer. The exact mechanism can be `json_schema_extra` metadata, a companion registry, or helper sets, but the end state must be queryable in code. The required classifications are:

- public operator settings,
- internal-only settings,
- retired-but-accepted legacy keys.

Add helpers that expose these surfaces intentionally. Examples of acceptable end-state helpers are functions like “return public field names,” “serialize only public settings,” or “load a payload while mapping retired keys to current behavior.” The precise names can follow repository style, but the behavior must be explicit and tested.

Do not try to delete all internal fields in this milestone. The point here is to create the foundation so downstream code can stop rendering, documenting, or persisting every field as if it were public.

Acceptance is that tests can verify field classification and that interactive/UI code can request “editable public settings” without implicitly meaning “every non-hidden Pydantic field.”

### Milestone 2: Retire `table_extraction` And Make Tables Part Of Normal Stage Behavior

At the end of this milestone, deterministic table extraction should always run when non-recipe blocks are present, and a user should not need or see `--table-extraction`. Old payloads containing `table_extraction` must still load safely, but the value should no longer change behavior.

Edit `cookimport/cli.py` to remove the public `--table-extraction` option from `stage` and any benchmark-related entrypoints that currently surface it. Remove the related normalization helper if it becomes dead code. Update `cookimport/config/run_settings_adapters.py` so stage and benchmark adapters no longer forward `table_extraction` as a live input. Update `cookimport/cli_worker.py`, `cookimport/cli.py`, and `cookimport/labelstudio/ingest.py` so they always run the table-extraction path instead of branching on a user setting.

Handle backward compatibility in `RunSettings.from_dict(...)`. If an old payload contains `table_extraction`, accept it, ignore its old `off|on` value, and either emit a one-time warning or record a migration note that the setting is retired and tables are now always extracted. New serialized run-config payloads should not re-emit `table_extraction` as if it were still a live choice.

This milestone is intentionally narrower than “remove all config debt.” It delivers one clear product win immediately and proves the migration pattern for other retired settings.

Acceptance is concrete. `cookimport stage --help` should no longer show `--table-extraction`. A normal stage run should write `tables/<workbook>/tables.jsonl` and, when markdown is enabled, `tables.md` without any extra flag. New manifests should not communicate `table_extraction` as a public choice, while old payloads containing the key still load.

### Milestone 3: Internalize The Clearest Fake Settings

At the end of this milestone, the most obvious implementation seams from the audit should no longer masquerade as live operator choices. Some may be fully retired as legacy keys; others may remain internal-only but hidden from ordinary docs, menus, and summaries.

Start with the settings that are effectively single-choice, pipeline-contract identifiers, or debug toggles. The first tranche should include:

- `benchmark_sequence_matcher`,
- `multi_recipe_trace`,
- `p6_emit_metadata_debug`,
- `codex_farm_pipeline_pass1`,
- `codex_farm_pipeline_pass2`,
- `codex_farm_pipeline_pass3`,
- `codex_farm_pipeline_pass4_knowledge`,
- `codex_farm_pipeline_pass5_tags`,
- `codex_farm_pass1_pattern_hints_enabled`,
- `codex_farm_pass3_skip_pass2_ok`,
- `codex_farm_benchmark_selective_retry_enabled`,
- `codex_farm_benchmark_selective_retry_max_attempts`.

Retire settings that should no longer be configurable at all by moving their values into stable constants or profile/contract code. Mark settings that remain useful only for experiments as internal-only and stop exposing them in ordinary interactive and docs flows.

The repository already has a strong place for contract-level normalization in `cookimport/config/codex_decision.py`. Reuse that pattern instead of scattering defaults across random call sites. The end state should make it obvious where current “blessed behavior” lives.

Acceptance is behavioral rather than cosmetic. A normal human should no longer see pipeline-id fields in the main operator surface, and new run summaries should stop reading like raw plumbing dumps. Compatibility loaders must still accept old payloads when those keys appear.

### Milestone 4: Reduce The Everyday Surface To Real Operator Choices

At the end of this milestone, interactive flows, CLI docs, and manifest summaries should center on the smaller set of settings that a human might intentionally vary run to run. The audit’s “keep user-facing” bucket is the target shape.

Update the relevant UI and docs surfaces so they consume only public settings. This includes `cookimport/cli_ui/run_settings_flow.py`, any run-setting UI spec helpers in `cookimport/config/run_settings.py`, and the run-settings sections in `docs/02-cli/02-cli_README.md`. If a docs section currently enumerates every field in `RunSettings`, rewrite it so it distinguishes:

- real operator choices,
- internal defaults,
- legacy compatibility behavior when relevant.

Keep the public surface small and concrete. The expected survivors are workers, extractor/OCR choices, Codex enablement, Codex model/effort, a few workspace path settings, and WebSchema importer choices. Parser lab controls, benchmark-only toggles, and pipeline-id settings should not dominate the user-facing story.

Update manifest/report summary code so the main human-readable summary shows public settings first and does not imply that internal defaults were explicitly chosen by the operator. If needed, a full debug payload can still exist separately for archaeology, but it should not be the default human-facing view.

Acceptance is that a novice reading the CLI docs or a new manifest summary can quickly understand what was intentionally chosen for a run without wading through implementation detail.

### Milestone 5: Compatibility, Identity, And Test Repair

At the end of this milestone, the cleanup should be safe to merge because old artifacts remain loadable, prediction identity logic remains stable where intended, and test coverage proves both the migration and the new behavior.

Update tests in the run-settings and CLI areas first:

- `tests/llm/test_run_settings.py`
- `tests/cli/test_run_settings_adapters.py`
- `tests/cli/test_cli_llm_flags.py`
- `tests/staging/test_run_manifest_parity.py`

Then update the stage and benchmark helper tests that currently assume `table_extraction` is a live payload field or that every `RunSettings` field is public/editable.

Compatibility tests must cover at least these cases:

- loading an old JSON payload with `table_extraction: "off"` still succeeds,
- new serialization does not reintroduce retired keys,
- new stage and benchmark runs still generate stable outputs,
- identity or cache payloads only change when behavior actually changes, not merely because the surface got cleaner.

If any analytics or history code still assumes `run_config` contains every prior field, update it explicitly. The goal is not to preserve field count; the goal is to preserve meaningful comparison and reproducibility.

Acceptance is that targeted tests pass and a novice can rerun the compatibility scenarios described in this plan without needing hidden context.

## Validation and Acceptance

Acceptance must prove behavior, not just refactoring.

First, the public CLI surface must shrink. `cookimport stage --help` and `cookimport labelstudio-benchmark --help` should no longer expose retired flags such as `--table-extraction`. Pipeline-id knobs and single-choice benchmark internals should stop appearing as ordinary operator choices.

Second, table extraction must become ordinary behavior. Running:

    cookimport stage data/input/thefoodlabCUTDOWN.epub --write-markdown

should produce `tables/thefoodlabcutdown/tables.jsonl` and `tables.md` under the newest `data/output/<timestamp>/` root without any table-specific flag.

Third, backward compatibility must remain intact. A legacy JSON payload containing `table_extraction` and other retired keys must still load through `RunSettings.from_dict(...)`, and the runtime should proceed using the new product behavior.

Fourth, the human-readable surface must improve. New run manifests, summaries, and docs should primarily describe real operator choices rather than every low-level implementation seam. If a full internal payload is still stored somewhere for debugging, it must no longer be the main story a human sees first.

Fifth, tests must prove that settings cleanup did not silently break stage or benchmark propagation. The targeted settings and manifest tests listed above must pass, and any new compatibility or always-on-table tests added during implementation must pass as well.

## Idempotence and Recovery

This cleanup should be implemented as an additive migration rather than a destructive rewrite. Compatibility loaders should accept retired keys for as long as old artifacts need to remain readable. If a retirement step causes too much collateral breakage, the safe fallback is to keep the field loadable internally while continuing to hide it from public surfaces, then finish the deeper deletion later.

The always-on table extraction step is also safe to retry because the underlying feature is additive. If tables produce unexpected regressions on a specific book, use targeted tests and output inspection before reintroducing a public off-switch. The default recovery posture should be tightening heuristics or narrowing the always-on application site, not restoring permanent product-level config sprawl.

Each validation run writes a fresh timestamped output root, so rerunning commands from this plan should not overwrite historical evidence. If a benchmark or stage run produces surprising manifests after cleanup, inspect the new root rather than editing old saved roots.

## Interfaces and Dependencies

The implementation should continue using `cookimport.config.run_settings.RunSettings` as the compatibility and validation boundary for payload loading. Do not create an unbounded second settings model unless implementation proves it is necessary. Instead, add explicit surface classification and serialization helpers around the existing model.

At the end of the work, the codebase must have clear answers to these questions:

- Which settings are public operator choices?
- Which settings are internal-only but still loadable?
- Which old keys are accepted only for backward compatibility?
- Which serializer is used for human-facing manifests and summaries?
- Where do retired behaviors like table extraction now live as product defaults?

The stage and benchmark adapter boundary must remain explicit in `cookimport/config/run_settings_adapters.py`, but those adapters should stop forwarding retired public settings. Runtime modules such as `cookimport/cli_worker.py` and `cookimport/labelstudio/ingest.py` should no longer depend on a public `table_extraction` field to decide whether tables exist.
