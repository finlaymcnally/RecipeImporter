# Bound Task Summary And Line-Role Worker Surface

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

After this change, helper-first startup will be bounded in practice as well as in wording: `python3 -m cookimport.llm.editable_task_file --summary` will sample large unit-id lists instead of emitting an unbounded list of every unanswered unit. The worker-contract docs will also stay precise about line-role: deterministic reason hints are gone from worker-visible `task.json`, while repo-side debug artifacts remain available for operators outside the single-file startup surface.

The proof is deterministic. Focused tests will pin the bounded summary shape on both small and large task files, and the docs index will show clean front matter for the completed line-role reason-hint plan.

## Decision Log

  Rationale: workers still benefit from seeing a few concrete ids up front, but the command should not scale linearly with the full task size.

## Context and Orientation

The shared editable task helper lives in `cookimport/llm/editable_task_file.py`. It owns read-only summary and unit-inspection commands plus repo-owned answer-apply helpers for single-file workers. The recipe, knowledge, and canonical line-role same-session helpers consume the summary only for local status reporting; they rely on `answered_units` and `total_units`, not on a full list of every unit id.

Single-file worker isolation is owned by `cookimport/llm/codex_exec_runner.py`. For `single_file_worker_runtime`, the actual execution workspace contains only `task.json` plus repo-written `AGENTS.md`; repo-side worker roots still keep `worker_manifest.json`, `_repo_control/`, `debug/`, `in/`, `out/`, and related artifacts for observability and recovery.

The line-role package note lives in `cookimport/parsing/canonical_line_roles/README.md`, and the higher-level LLM/parsing contracts live in `cookimport/llm/README.md`, `docs/10-llm/10-llm_README.md`, and `docs/04-parsing/04-parsing_readme.md`. The older plan `docs/plans/2026-03-31_13.28.27-remove-line-role-reason-hints.md` is functionally complete but currently missing front matter metadata.

## Validation and Acceptance

This plan is complete only when all of the following are true.

The helper summary must be bounded: on a large task file, `python3 -m cookimport.llm.editable_task_file --summary` must return a sampled unanswered-unit list plus explicit truncation metadata rather than every unanswered unit id.

The small-task path must remain simple: tiny task files should still show exact unanswered ids with no truncation flag set.

The current docs must describe the worker surface accurately. Line-role `task.json` must remain raw-text-first and free of deterministic semantic hints, while the docs must not imply that repo-side debug artifacts are part of the isolated single-file startup surface.

`npm run docs:list` must index the completed line-role reason-hint plan without reporting missing front matter.

## Idempotence and Recovery

These edits are safe to repeat. If the summary metadata shape needs future refinement, keep it additive and explicit rather than silently changing whether lists are sampled. Do not remove repo-owned debug artifacts from line-role worker roots as part of this task; if that ever becomes necessary, it should be scoped as a separate observability tradeoff.
