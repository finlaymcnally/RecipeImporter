# Refactor Cumbersome Tests Into Smaller Fixtures And Assertions

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

Several test files have grown into large, mixed-concern integration scripts that are hard to read, hard to extend, and expensive to debug when one assertion changes. After this change, the highest-friction test files should still prove the same behavior, but they should do it through smaller tests, shared local builders, and tighter assertion groups so a reader can understand one failure without scrolling through hundreds of lines of unrelated setup and checks.

The user-visible proof is straightforward: the targeted `bench`, `llm`, and `labelstudio` test domains still pass in the project-local `.venv`, while the worst test files no longer rely on 200- to 300-line single-test bodies for their primary coverage.

## Surprises & Discoveries

- Observation: the largest file is not automatically the biggest maintainability problem.
  Evidence: `tests/analytics/test_stats_dashboard.py` is 5758 lines, but its average test body is only about 41 lines; by contrast `tests/llm/test_codex_farm_knowledge_orchestrator_runtime.py` is only 787 lines but averages about 192 lines per test.

- Observation: the real pain is concentrated in a few single tests that verify many unrelated outcomes at once.
  Evidence: the initial metrics pass found individual tests at 335, 324, 309, and 292 lines, with the top examples in `tests/labelstudio/test_labelstudio_benchmark_helpers_artifacts.py`, `tests/bench/test_benchmark_cutdown_for_external_ai.py`, `tests/llm/test_codex_farm_knowledge_orchestrator.py`, and `tests/labelstudio/test_labelstudio_ingest_parallel.py`.

- Observation: some high-friction files already contain almost no helper code, which means their test bodies are carrying raw fixture construction inline instead of hiding complexity behind reusable builders.
  Evidence: `tests/labelstudio/test_labelstudio_benchmark_helpers_artifacts.py` is 2339 lines with only a few lines of non-test helper functions according to the AST-based file scan.

- Observation: the fastest maintainability wins came from local file-scoped builders, not from inventing shared cross-domain fixtures.
  Evidence: each hotspot yielded clean extraction seams that were specific to that file’s contract, such as `_build_codex_farm_prompt_log_fixture()`, `_build_existing_upload_bundle_fixture()`, and `_build_knowledge_orchestrator_artifact_fixture()`.

## Context and Orientation

The repository’s tests live under `tests/`. In this plan, a “cumbersome” test means a test whose setup and assertions are broad enough that one failure does not clearly indicate what behavior broke. The worst current examples tend to do all of the following at once: create large synthetic run trees, write multiple JSON artifacts, execute one command, and then assert many unrelated outputs in the same function.

The key files currently under scrutiny are:

- `tests/labelstudio/test_labelstudio_benchmark_helpers_artifacts.py`, which contains a 335-line test that builds prompt-log fixtures, telemetry CSV rows, trace files, and then verifies multiple artifact families in one pass.
- `tests/bench/test_benchmark_cutdown_for_external_ai.py`, which contains a 324-line upload-bundle test that verifies bundle shape, derived analysis, navigation locators, diagnostics, and overview markdown together.
- `tests/llm/test_codex_farm_knowledge_orchestrator.py`, which contains several 150- to 300-line tests that validate both broad stage outcomes and many worker-side artifact details together.
- `tests/llm/test_codex_farm_knowledge_orchestrator_runtime.py`, which has only four tests, but each test carries large bespoke setup and runtime simulation logic.
- `tests/labelstudio/test_labelstudio_ingest_parallel.py`, which contains multiple large processed-output and worker-coordination scenarios.

The relevant test-running contract is in `scripts/test-suite.sh`. Routine validation should use that script or the `make test-domain DOMAIN=...` wrapper rather than broad raw `pytest` invocations. All Python validation must run inside the local `.venv`.

## Plan of Work

Start with the files where one test body is visibly doing multiple jobs. In `tests/labelstudio/test_labelstudio_benchmark_helpers_artifacts.py`, split the giant prompt-log test into smaller tests grouped by behavior: prompt-category log writing, request/telemetry backfill, trace-summary writing, and prompt-sample rendering. Extract a local builder that creates the shared prediction-run fixture once so each test only states the variant it cares about.

Next, in `tests/bench/test_benchmark_cutdown_for_external_ai.py`, break the upload-bundle monolith into multiple tests that share a single local “existing output session” builder. Separate bundle file creation, topline/index integrity, analysis sections, navigation locators, and overview markdown assertions so failures point at one contract at a time.

Then address the LLM runtime hotspots. In `tests/llm/test_codex_farm_knowledge_orchestrator.py`, pull repeated fake run-root / settings / conversion-result construction into local helpers and split large artifact-surface checks into narrower tests where practical. In `tests/llm/test_codex_farm_knowledge_orchestrator_runtime.py`, keep the runtime simulations but reduce each test’s assertion surface and share the synthetic chunk/session setup helpers.

Finally, revisit `tests/labelstudio/test_labelstudio_ingest_parallel.py` and any other file that still has clearly oversized scenario tests after the first pass. The goal is not arbitrary line reduction. The goal is that each test has one dominant reason to fail and the file-level helper setup is readable.

## Validation and Acceptance

Acceptance is behavioral, not stylistic. This work is successful when:

1. The targeted test files no longer rely on the same oversized single-test bodies for their primary coverage.
2. The affected test domains still pass in `.venv`.
3. The refactored tests are easier to diagnose because a failure now maps to one narrower contract instead of one giant fixture-and-assertion bundle.

The minimum required validation at the end of the work is:

    source .venv/bin/activate
    ./scripts/test-suite.sh domain bench
    ./scripts/test-suite.sh domain llm
    ./scripts/test-suite.sh domain labelstudio

If another domain is touched materially during the refactor, add its domain run here before closing the task.

## Idempotence and Recovery

This refactor is code-only and safe to repeat. If a split test proves too coupled, the safe recovery path is to restore the original assertion grouping and try a smaller helper extraction instead. No migrations or destructive operations are involved.
