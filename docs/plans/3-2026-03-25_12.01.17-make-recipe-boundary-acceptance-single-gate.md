# Make Recipe-Boundary Acceptance A Single Gate

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

`docs/PLANS.md` is checked into this repository and this document must be maintained in accordance with it.

## User Intent

Exact user intent: "yeah weneed to fix that, please make an execplan to correct"

The user wants the repository to stop teaching and implementing a misleading two-step recipe acceptance story where a recipe span can be accepted during grouping and then later quietly rejected for missing recipe body. The goal of this plan is to correct that contract by making recipe-boundary acceptance mean one thing everywhere: if a span is accepted as a recipe span, it should already satisfy the real minimum recipe-body rule and should not be demoted later during ordinary downstream recipe construction.

## Purpose / Big Picture

After this change, the recipe-boundary stage will have one honest acceptance rule instead of two disagreeing ones. A span that lands in `group_recipe_spans/.../recipe_spans.json` as `accepted_recipe_span` will already have enough recipe-body proof to survive downstream recipe construction, rather than being accepted first and demoted later to `rejected_missing_recipe_body`.

The user-visible proof is simple. A reviewer should be able to inspect `group_recipe_spans/<workbook_slug>/span_decisions.json` and trust that the accepted spans match the recipes that later get built, apart from true runtime errors. Title-only shells and title-plus-note junk should be rejected during span grouping itself. Title-plus-yield or title-plus-real body spans should still survive. This matters because the current two-gate story is not teachable: “accepted recipe span” does not actually mean accepted.

## Surprises & Discoveries

- Observation: the current early acceptance rule is narrower than the later rejection rule in exactly the wrong way.
  Evidence: `group_recipe_spans_from_labels(...)` accepts whenever a pending span has a title anchor, while `_recipe_candidate_rejection_reason(...)` rejects if the built recipe has no ingredients, no instructions, and no yield/time metadata.

- Observation: `recipe_span_grouping.py` already contains a weaker notion of “recipe body,” but it is only used for gap-bridging, not final acceptance.
  Evidence: `_has_recipe_body(...)` is consulted by `_should_bridge_nonrecipe_gap(...)`, but `_build_span_decision(...)` still decides acceptance on title-anchor presence alone.

- Observation: the current late rejection rule does not count notes as real recipe body.
  Evidence: `_recipe_candidate_rejection_reason(...)` returns `None` only when a recipe has ingredients, instructions, or yield/prep/cook/total time. Recipe notes alone do not save a span.

- Observation: the docs already admit the two-gate mismatch.
  Evidence: `docs/04-parsing/04-parsing_readme.md` says title-like anchoring is the hard acceptance boundary, but it also says title-only shells with no ingredients, instructions, or yield/time metadata are rejected after span-to-recipe conversion.

## Decision Log

- Decision: make `recipe_span_grouping.py` the single semantic acceptance owner for recipe spans.
  Rationale: that is the earliest point where the stage has both normalized block labels and a coherent contiguous span candidate. Letting a later module silently overturn accepted span authority creates an unnecessary second gate.
  Date/Author: 2026-03-25 / Codex

- Decision: define “recipe body proof” to match the current late rejection rule, not the broader `_has_recipe_body(...)` helper.
  Rationale: the current late rule is the tighter contract the user actually notices. A span should count as a real recipe only if it has at least one ingredient line, instruction/howto line, or yield/time line after grouping. Notes alone should not save it.
  Date/Author: 2026-03-25 / Codex

- Decision: keep title anchoring as necessary but no longer sufficient.
  Rationale: title anchoring is still the best guard against TOC-like junk, but it does not by itself prove recipe-ness. The new contract should require both title anchor and minimal body proof.
  Date/Author: 2026-03-25 / Codex

- Decision: once a span is accepted, later recipe construction should no longer demote it for ordinary “missing recipe body” reasons.
  Rationale: downstream code should build recipes from already accepted spans, not reopen the acceptance question. If a later build truly fails, that should be treated as an invariant break or explicit error surface, not a second quiet semantic gate.
  Date/Author: 2026-03-25 / Codex

- Decision: preserve short real recipe stubs that have title plus yield/time metadata even when they lack explicit ingredients or instructions.
  Rationale: the current late rejection rule already allows that, and the user explicitly called out that the split itself is the bug, not that survival rule.
  Date/Author: 2026-03-25 / Codex

## Outcomes & Retrospective

This section is intentionally incomplete until implementation lands.

The expected end state is that `accepted_recipe_span` becomes a trustworthy authority label. Future debugging should get simpler because a reviewer will no longer need to ask whether a recipe was “accepted by grouping” but “rejected by candidate build.” Instead, grouping will already have made the real semantic acceptance decision.

The main thing to watch after implementation is false-negative recipe loss. Tightening early acceptance could accidentally reject short legitimate recipes if the new body-proof predicate is too strict. The fix should be to tune the acceptance predicate explicitly and test it, not to keep the current misleading two-gate setup.

## Context and Orientation

This repository has a label-first recipe-boundary stage. “Recipe boundary” means the stage that turns line-role and block-level labels into contiguous recipe spans that later become real recipes. The important files are:

- `cookimport/parsing/recipe_span_grouping.py`
  This groups authoritative block labels into pending spans and emits `RecipeSpanDecision` rows. Today it accepts a span when it has a title anchor, even if that span later builds into a title-only shell.

- `cookimport/parsing/label_source_of_truth.py`
  This takes the accepted spans, builds `RecipeCandidate` objects from them, and currently can demote an already accepted span to `rejected_missing_recipe_body`.

- `tests/parsing/test_recipe_span_grouping.py`
  This is the lowest-level regression surface for span acceptance and rejection.

- `tests/parsing/test_label_first_conversion.py`
  This is the conversion-stage regression surface. It currently covers the late `rejected_missing_recipe_body` behavior that this plan intends to remove from the normal path.

- `tests/staging/test_import_session.py`
  This is the shared stage-session regression surface that proves the user-visible staging artifacts still tell the truth after the contract change.

- `docs/04-parsing/04-parsing_readme.md`
  This is the durable parsing reference and must be updated so it teaches the single-gate contract.

- `docs/05-staging/05-staging_readme.md`
  This is the staging/output reference and currently mentions that accepted grouped spans can be demoted later. That statement must change after implementation.

Two phrases matter in this plan:

- “Title anchor” means a grouped span contains a title-like block label such as `RECIPE_TITLE` or `RECIPE_VARIANT`.
- “Recipe body proof” means a grouped span contains at least one non-title signal that the later candidate-build logic already considers sufficient recipe body evidence: ingredient lines, instruction/howto lines, or yield/time metadata. Notes do not count as recipe body proof by themselves.

The current live flow works like this:

1. `group_recipe_spans_from_labels(...)` assembles contiguous pending spans.
2. `_build_span_decision(...)` accepts a span if it has a title anchor.
3. Accepted spans enter `build_conversion_result_from_label_spans(...)`.
4. `_build_recipe_candidate_from_span(...)` builds a `RecipeCandidate`.
5. `_recipe_candidate_rejection_reason(...)` may still reject that accepted span if the built recipe has no ingredients, no instructions, and no yield/time metadata.

This plan removes step 5 as a normal semantic acceptance seam. After implementation, the acceptance check in step 2 must already enforce the same body-proof rule that step 5 currently applies later.

## Plan of Work

Start by making the acceptance predicate explicit in `cookimport/parsing/recipe_span_grouping.py`. Today the file has the pieces split across helpers: `_TITLE_LIKE_LABELS` defines anchor labels, `_RECIPE_BODY_LABELS` defines a broad body-like set, and `_has_recipe_body(...)` uses that broad set only for gap-bridging. The implementation should introduce one new helper with a precise name such as `_has_acceptance_recipe_body(...)` whose contract exactly matches the later recipe-build acceptance rule:

- ingredient lines count
- instruction lines count
- howto section lines count
- yield lines count
- time lines count
- notes do not count
- title and variant labels do not count

Then change `_build_span_decision(...)` so acceptance requires both:

- title anchor present
- acceptance body proof present

Use explicit rejection reasons rather than one vague bucket. The grouping stage should now distinguish at least:

- `rejected_missing_title_anchor`
- `rejected_missing_recipe_body`

The later build step in `cookimport/parsing/label_source_of_truth.py` should then stop acting like a second semantic gate. Keep `_recipe_candidate_rejection_reason(...)` only if it becomes an invariant check for impossible projection failures, or replace it with a narrower assertion/error path that records a bug-shaped warning. The normal path should no longer produce “accepted span, then rejected missing recipe body.”

The implementation should also review the nonrecipe-gap bridge behavior. `_should_bridge_nonrecipe_gap(...)` currently uses the broader `_has_recipe_body(...)` helper. After this change, there should not be two subtly different meanings of “recipe body” hiding in the same file. Either reuse the new acceptance-body helper there where appropriate or rename the old gap helper so the distinction is explicit.

Next, update the tests so the new contract is executable:

- low-level grouping tests should prove that title-only and title-plus-note spans are rejected immediately as `rejected_missing_recipe_body`
- low-level grouping tests should prove that title-plus-yield and title-plus-instruction spans still survive
- conversion tests should stop expecting late `rejected_missing_recipe_body` demotion on the normal path
- stage-session tests should keep proving that `span_decisions.json` and final recipes stay aligned

Finally, update the docs. `docs/04-parsing/04-parsing_readme.md` should say plainly that recipe-boundary acceptance now requires both title anchor and body proof. `docs/05-staging/05-staging_readme.md` should stop teaching later demotion as part of the normal contract. If `docs/plain-english-explanation.md` still describes accepted spans being rejected later for missing recipe body, update that language too.

## Milestones

### Milestone 1: Define one acceptance predicate in the grouping layer

At the end of this milestone, the codebase will have one named helper that defines the real recipe-body proof required for acceptance. A novice reader should be able to open `cookimport/parsing/recipe_span_grouping.py` and see the acceptance rule directly instead of discovering it later in `label_source_of_truth.py`.

Acceptance for this milestone is one or two focused unit tests in `tests/parsing/test_recipe_span_grouping.py` that prove title-only or title-plus-notes spans are rejected immediately, while title-plus-yield and title-plus-instruction spans survive.

### Milestone 2: Remove the normal late demotion path

At the end of this milestone, `build_conversion_result_from_label_spans(...)` will build recipes from already-final accepted spans. The normal path will no longer demote an accepted span to `rejected_missing_recipe_body`. If an accepted span still somehow builds to no recipe body, that should be surfaced as an invariant break, not treated as a second hidden acceptance rule.

Acceptance for this milestone is that `tests/parsing/test_label_first_conversion.py` no longer teaches ordinary late demotion, and the stage output contract continues to build recipes and `span_decisions` coherently.

### Milestone 3: Lock the contract down in docs and staging-facing tests

At the end of this milestone, the docs and higher-level tests all teach the same story: accepted spans are already real recipe spans. `docs/04-parsing/04-parsing_readme.md`, `docs/05-staging/05-staging_readme.md`, and any plain-English explanation doc should align with the code.

Acceptance for this milestone is passing focused tests plus docs that no longer describe `accepted_recipe_span` as provisional.

## Validation and Acceptance

Acceptance is behavioral.

First, low-level grouping tests must prove that title anchor alone is no longer enough. A title-only span and a title-plus-note span must be rejected during grouping with an explicit missing-body reason.

Second, grouping tests must prove that short real recipes still survive. A title-plus-yield stub and a title-plus-instruction or title-plus-ingredient span must remain accepted.

Third, conversion tests must prove that the normal build path no longer reopens the recipe-acceptance question. After implementation, a span that reached `accepted_recipe_span` should not ordinarily be demoted later to `rejected_missing_recipe_body`.

Fourth, staging-facing tests must prove that `span_decisions.json`, `recipe_spans.json`, and final recipes tell one coherent story. A reviewer should not need private code knowledge to reconcile them.

Fifth, the docs must answer this question clearly: “When does a span become a real recipe?” After implementation, the answer should be “in recipe-span grouping, once title anchor and body proof are both present,” not “maybe later if candidate build likes it.”

## Idempotence and Recovery

This plan is safe to implement incrementally. The safest order is:

1. add the new grouping acceptance helper
2. add or update tests that describe the desired early rejection behavior
3. remove the late demotion seam
4. update docs

Do not leave the repo in a half-migrated state where both the early and late gates can reject for missing body using different predicates. That would make the contract even harder to teach.

If the change rejects too many short real recipes, do not restore the late demotion seam. Instead, tune the grouping-time body-proof predicate explicitly and update the tests so the acceptance rule stays visible.

## Artifacts and Notes

- accepted spans are real recipe spans
- rejected pseudo spans were already rejected there
- downstream code should not have to reinterpret those decisions semantically

The second important artifact is the regression story in tests:

- `tests/parsing/test_recipe_span_grouping.py` should become the executable definition of recipe-boundary acceptance

## Interfaces and Dependencies

The implementation should stay inside the existing parser/staging modules and not introduce a new subsystem.

In `cookimport/parsing/recipe_span_grouping.py`, define a helper with a precise responsibility, such as:

    def _has_acceptance_recipe_body(block_rows: Sequence[AuthoritativeBlockLabel]) -> bool:
        ...

That helper should return `True` only when the span contains at least one of:

- `INGREDIENT_LINE`
- `INSTRUCTION_LINE`
- `HOWTO_SECTION`
- `YIELD_LINE`
- `TIME_LINE`

In `cookimport/parsing/recipe_span_grouping.py`, `_build_span_decision(...)` must use that helper directly when deciding whether to emit:

- `decision="accepted_recipe_span"`
- or `decision="rejected_pseudo_recipe_span"` with `rejection_reason="rejected_missing_recipe_body"`

In `cookimport/parsing/label_source_of_truth.py`, `build_conversion_result_from_label_spans(...)` should stop serving as a normal missing-body rejection owner. If a defensive check remains, it should be framed and tested as an invariant breach or explicit bug-surface, not as a second ordinary acceptance gate.
