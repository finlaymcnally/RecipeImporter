# Make Recipe-Boundary Acceptance A Single Gate

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

`docs/PLANS.md` is checked into this repository and this document must be maintained in accordance with it.

## User Intent

Exact user intent: "yeah weneed to fix that, please make an execplan to correct"

The user wants the repository to stop teaching and implementing misleading multi-step recipe authority. There are two concrete problems on the recipe side:

- a recipe span can be accepted during grouping and then later quietly rejected for missing recipe body
- a structurally valid recipe Codex result can be a valid task outcome without the repository clearly saying whether it became final recipe authority

The goal of this plan is to correct that contract end to end. Recipe-boundary acceptance should mean one thing everywhere: if a span is accepted as a recipe span, it already satisfies the real minimum recipe-body rule and should not be demoted later during ordinary downstream recipe construction. Recipe Codex should also have one honest promotion story: once its output is structurally valid, the runtime should make the terminal outcome explicit instead of leaving reviewers to infer why a valid task result did or did not become a final recipe.

## Purpose / Big Picture

After this change, the recipe side will teach one honest authority story from grouped span to final recipe output.

A span that lands in `group_recipe_spans/.../recipe_spans.json` as `accepted_recipe_span` will already have enough recipe-body proof to survive downstream recipe construction, rather than being accepted first and demoted later to `rejected_missing_recipe_body`.

A structurally valid recipe Codex result will also have an explicit terminal meaning. `repaired` results will promote into final recipe authority. `fragmentary` and `not_a_recipe` results may remain valid task outcomes, but they will be taught and surfaced as explicit non-promotable terminal states rather than looking like accepted recipes that quietly vanished.

The user-visible proof is simple. A reviewer should be able to inspect `group_recipe_spans/<workbook_slug>/span_decisions.json` and trust that the accepted spans match the recipes that later get built, apart from true runtime errors. Title-only shells and title-plus-note junk should be rejected during span grouping itself. Title-plus-yield or title-plus-real body spans should still survive.

The reviewer should also be able to inspect recipe-authority artifacts and understand what happened to every structurally valid recipe Codex output:

- promoted final recipe authority
- valid but `fragmentary`
- valid but `not_a_recipe`

This matters because the current recipe story is not teachable. “Accepted recipe span” does not actually mean accepted, and “valid recipe Codex output” does not clearly tell a reviewer whether that result became a final recipe.

## Surprises & Discoveries

- Observation: the current early acceptance rule is narrower than the later rejection rule in exactly the wrong way.
  Evidence: `group_recipe_spans_from_labels(...)` accepts whenever a pending span has a title anchor, while `_recipe_candidate_rejection_reason(...)` rejects if the built recipe has no ingredients, no instructions, and no yield/time metadata.

- Observation: `recipe_span_grouping.py` already contains a weaker notion of “recipe body,” but it is only used for gap-bridging, not final acceptance.
  Evidence: `_has_recipe_body(...)` is consulted by `_should_bridge_nonrecipe_gap(...)`, but `_build_span_decision(...)` still decides acceptance on title-anchor presence alone.

- Observation: the current late rejection rule does not count notes as real recipe body.
  Evidence: `_recipe_candidate_rejection_reason(...)` returns `None` only when a recipe has ingredients, instructions, or yield/prep/cook/total time. Recipe notes alone do not save a span.

- Observation: the docs already admit the two-gate mismatch.
  Evidence: `docs/04-parsing/04-parsing_readme.md` says title-like anchoring is the hard acceptance boundary, but it also says title-only shells with no ingredients, instructions, or yield/time metadata are rejected after span-to-recipe conversion.

- Observation: the recipe Codex runtime already has the right coarse status model, but the artifacts and docs do not make the promotion line crisp enough.
  Evidence: `cookimport/llm/codex_farm_orchestrator.py` records `repair_status` values such as `repaired`, `fragmentary`, and `not_a_recipe`, and only `repaired` contributes to `updated_result.recipes`, but the surrounding wording still makes it too easy to blur “valid task outcome” with “promoted final recipe.”

- Decision: keep title anchoring as necessary but no longer sufficient.

- Decision: treat recipe Codex “task validity” and “final recipe promotion” as separate explicit states.

## Outcomes & Retrospective

`accepted_recipe_span` is now a trustworthy authority label. Grouping already enforces the same minimum body rule the later conversion code used to apply, so the normal path no longer accepts a span and then quietly demotes it for missing recipe body.

The Codex side now also teaches one explicit promotion story. `promotion_report.json` records whether a validated recipe task output is `promotable` or `non_promotable`, while `recipe_manifest.json` and `recipe_correction_audit/*.json` record whether final recipe authority was `promoted`, `not_promoted`, or failed with an explicit error. `fragmentary` and `not_a_recipe` remain visible valid outcomes instead of reading like vanished accepted recipes.

The remaining risk is false-negative recipe loss from the stricter grouping gate. The right recovery is still to tune the grouping-time body predicate and add tests for legitimate short recipes, not to reintroduce the misleading late demotion seam.

## Context and Orientation

This repository has a label-first recipe-boundary stage. “Recipe boundary” means the stage that turns line-role and block-level labels into contiguous recipe spans that later become real recipes. The important files are:

- `cookimport/parsing/recipe_span_grouping.py`
  This groups authoritative block labels into pending spans and emits `RecipeSpanDecision` rows. Today it accepts a span when it has a title anchor, even if that span later builds into a title-only shell.

- `cookimport/parsing/label_source_of_truth.py`
  This takes the accepted spans, builds `RecipeCandidate` objects from them, and currently can demote an already accepted span to `rejected_missing_recipe_body`.

- `tests/parsing/test_recipe_span_grouping.py`
  This is the lowest-level regression surface for span acceptance and rejection.

- `tests/parsing/test_label_first_conversion.py`
  This is the conversion-stage regression surface. It currently covers the late `rejected_missing_recipe_body` behavior that this plan intends to remove from the normal path.

- `tests/staging/test_import_session.py`
  This is the shared stage-session regression surface that proves the user-visible staging artifacts still tell the truth after the contract change.

- `cookimport/llm/codex_farm_orchestrator.py`
  This is the recipe Codex promotion seam. It validates per-recipe task output, records `repair_status`, and decides whether that valid task outcome becomes final recipe authority.

- `tests/llm/test_codex_farm_orchestrator.py`
  This is the recipe Codex regression surface that should encode the distinction between structurally valid task outcomes and promoted final recipes.

- `docs/04-parsing/04-parsing_readme.md`
  This is the durable parsing reference and must be updated so it teaches the single-gate contract.

- `docs/05-staging/05-staging_readme.md`
  This is the staging/output reference and currently mentions that accepted grouped spans can be demoted later. That statement must change after implementation.

- `docs/10-llm/10-llm_README.md`
  This is the durable LLM/runtime reference and must be updated so it teaches that `fragmentary` and `not_a_recipe` are valid recipe Codex outcomes but not promoted final recipe authority.

Two phrases matter in this plan:

- “Title anchor” means a grouped span contains a title-like block label such as `RECIPE_TITLE` or `RECIPE_VARIANT`.
- “Recipe body proof” means a grouped span contains at least one non-title signal that the later candidate-build logic already considers sufficient recipe body evidence: ingredient lines, instruction/howto lines, or yield/time metadata. Notes do not count as recipe body proof by themselves.

The current live flow works like this:

1. `group_recipe_spans_from_labels(...)` assembles contiguous pending spans.
2. `_build_span_decision(...)` accepts a span if it has a title anchor.
3. Accepted spans enter `build_conversion_result_from_label_spans(...)`.
4. `_build_recipe_candidate_from_span(...)` builds a `RecipeCandidate`.
5. `_recipe_candidate_rejection_reason(...)` may still reject that accepted span if the built recipe has no ingredients, no instructions, and no yield/time metadata.

Recipe Codex then adds a separate authority seam after recipe construction:

6. recipe Codex may return a structurally valid compact result with `repair_status` such as `repaired`, `fragmentary`, or `not_a_recipe`
7. the runtime promotes only `repaired` results into final recipe authority
8. the current artifacts and docs do not make that promotion line explicit enough

This plan removes step 5 as a normal semantic acceptance seam and makes steps 6 through 8 teachable. After implementation, the acceptance check in step 2 must already enforce the same body-proof rule that step 5 currently applies later, and recipe Codex outputs must have explicit promoted versus non-promoted terminal meaning.

## Plan of Work

Start by making the acceptance predicate explicit in `cookimport/parsing/recipe_span_grouping.py`. Today the file has the pieces split across helpers: `_TITLE_LIKE_LABELS` defines anchor labels, `_RECIPE_BODY_LABELS` defines a broad body-like set, and `_has_recipe_body(...)` uses that broad set only for gap-bridging. The implementation should introduce one new helper with a precise name such as `_has_acceptance_recipe_body(...)` whose contract exactly matches the later recipe-build acceptance rule:

- ingredient lines count
- instruction lines count
- howto section lines count
- yield lines count
- time lines count
- notes do not count
- title and variant labels do not count

Then change `_build_span_decision(...)` so acceptance requires both:

- title anchor present
- acceptance body proof present

Use explicit rejection reasons rather than one vague bucket. The grouping stage should now distinguish at least:

- `rejected_missing_title_anchor`
- `rejected_missing_recipe_body`

The later build step in `cookimport/parsing/label_source_of_truth.py` should then stop acting like a second semantic gate. Keep `_recipe_candidate_rejection_reason(...)` only if it becomes an invariant check for impossible projection failures, or replace it with a narrower assertion/error path that records a bug-shaped warning. The normal path should no longer produce “accepted span, then rejected missing recipe body.”

The implementation should also review the nonrecipe-gap bridge behavior. `_should_bridge_nonrecipe_gap(...)` currently uses the broader `_has_recipe_body(...)` helper. After this change, there should not be two subtly different meanings of “recipe body” hiding in the same file. Either reuse the new acceptance-body helper there where appropriate or rename the old gap helper so the distinction is explicit.

Next, update the tests so the new contract is executable:

- low-level grouping tests should prove that title-only and title-plus-note spans are rejected immediately as `rejected_missing_recipe_body`
- low-level grouping tests should prove that title-plus-yield and title-plus-instruction spans still survive
- conversion tests should stop expecting late `rejected_missing_recipe_body` demotion on the normal path
- stage-session tests should keep proving that `span_decisions.json` and final recipes stay aligned

Then tighten the recipe Codex promotion story in `cookimport/llm/codex_farm_orchestrator.py`. The runtime already behaves close to the intended rule, so this is mostly a contract-clarity pass rather than a new semantic subsystem. The implementation should make these distinctions explicit in names, summaries, audits, and tests:

- structurally valid task output is not the same thing as promoted final recipe authority
- `fragmentary` and `not_a_recipe` are valid recipe Codex outcomes
- only `repaired` promotes into final recipe authority
- valid non-promoted results must remain visible and auditable rather than feeling like they silently vanished

Finally, update the docs. `docs/04-parsing/04-parsing_readme.md` should say plainly that recipe-boundary acceptance now requires both title anchor and body proof. `docs/05-staging/05-staging_readme.md` should stop teaching later demotion as part of the normal contract. `docs/10-llm/10-llm_README.md` should teach the explicit recipe Codex outcome model. If `docs/plain-english-explanation.md` still describes accepted spans being rejected later for missing recipe body or blurs valid Codex output with promoted final recipes, update that language too.

## Validation and Acceptance

Acceptance is behavioral.

First, low-level grouping tests must prove that title anchor alone is no longer enough. A title-only span and a title-plus-note span must be rejected during grouping with an explicit missing-body reason.

Second, grouping tests must prove that short real recipes still survive. A title-plus-yield stub and a title-plus-instruction or title-plus-ingredient span must remain accepted.

Third, conversion tests must prove that the normal build path no longer reopens the recipe-acceptance question. After implementation, a span that reached `accepted_recipe_span` should not ordinarily be demoted later to `rejected_missing_recipe_body`.

Fourth, recipe Codex tests and artifacts must prove the promotion contract directly. A structurally valid recipe Codex result must have an explicit terminal status that tells a reviewer whether it promoted into final recipe authority. `fragmentary` and `not_a_recipe` may remain valid task outcomes, but they must be clearly non-promotable rather than quietly disappearing.

Fifth, staging-facing tests must prove that `span_decisions.json`, `recipe_spans.json`, recipe-authority artifacts, and final recipes tell one coherent story. A reviewer should not need private code knowledge to reconcile them.

Sixth, the docs must answer two questions clearly:

- “When does a span become a real recipe?”
- “When does a valid recipe Codex result become a final recipe?”

After implementation, the answers should be:

- “in recipe-span grouping, once title anchor and body proof are both present”
- “when a structurally valid recipe Codex result has `repair_status=\"repaired\"`; other valid statuses remain explicit non-promotion outcomes”

## Idempotence and Recovery

This plan is safe to implement incrementally. The safest order is:

1. add the new grouping acceptance helper
2. add or update tests that describe the desired early rejection behavior
3. remove the late demotion seam
4. update docs

Do not leave the repo in a half-migrated state where both the early and late gates can reject for missing body using different predicates. That would make the contract even harder to teach.

If the change rejects too many short real recipes, do not restore the late demotion seam. Instead, tune the grouping-time body-proof predicate explicitly and update the tests so the acceptance rule stays visible.

If the recipe Codex wording changes cause confusion, do not collapse status distinctions back into a single fuzzy success bucket. Improve artifact names, summaries, and docs until valid non-promotion outcomes are easy to understand.
