# Remove Remaining Legacy Code From Active Repo Surfaces

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](docs/PLANS.md).

## Purpose / Big Picture

After this work, the active code in this repository should stop teaching two worldviews at once. A person reading `cookimport/`, running the CLI, inspecting new run artifacts, or reviewing benchmark and analytics tooling should see only the current refactor architecture: label-first, semantic stages, one recipe correction stage, deterministic Stage 7 ownership, and current-contract-only settings.

The user-visible proof is straightforward. A fresh stage run, a fresh benchmark run, and a Label Studio export should complete without writing or depending on pass-slot language, `legacy`-named runtime settings, compatibility alias artifacts, or old-history fallback readers. A repo search over active code should no longer show the remaining seam families cataloged in this plan.

This plan intentionally targets active code, tests, fixtures, and current documentation. Historical planning and report files under `docs/plans/`, `docs/reports/`, `docs/tasks/`, and `docs/understandings/` may retain historical language because they are records, not live product code. If the user later wants those archival files pruned too, that should be a separate documentation cleanup pass, not mixed into runtime deletion work.

## Surprises & Discoveries

- Observation: the old interactive recipe-pipeline surface is already gone, but internal recipe-correction code still uses pass-slot field names.
  Evidence: `_RecipeState` still carries fields such as `pass1_raw_start_block_index`, `pass2_output`, and `pass3_mapping_status` in `cookimport/llm/codex_farm_orchestrator.py`.

- Observation: some benchmark tooling now renders semantic stage names while still reconstructing them from old pass counts.
  Evidence: `cookimport/bench/upload_bundle_v1_existing_output.py` translates `observed_pass2_call_count` and `observed_pass3_call_count` into semantic call counts.

- Observation: several active modules are not preserving legacy behavior; they are preserving legacy readers and rejection paths.
  Evidence: `cookimport/labelstudio/export.py` still contains explicit legacy-scope rejection logic for old Label Studio project types instead of assuming only the current freeform scope exists.

- Observation: one of the largest remaining seam families is historical-reader support, not the main stage runtime.
  Evidence: `cookimport/analytics/dashboard_collect.py` still probes legacy CSV paths, `cookimport/bench/followup_bundle.py` still loads `legacy_followup_cases`, and `cookimport/llm/codex_farm_runner.py` still parses legacy progress output from older runners.

- Observation: some `legacy` names no longer mean “fallback branch”; they mean “current algorithm under an old label.”
  Evidence: `cookimport/parsing/canonical_line_roles.py` falls back to `"legacy"` prompt format when the env var is not `compact_v1`, and `cookimport/bench/eval_canonical_text.py` says legacy global `SequenceMatcher` alignment is enforced as the safe scorer path.

- Observation: alias writing still survives in output code even after the refactor phases removed most compatibility publishing elsewhere.
  Evidence: `cookimport/staging/writer.py` still calls `_normalize_draft_compatibility_aliases(...)`.

## Outcomes & Retrospective

This plan is newly drafted, so the outcome today is clarity rather than code deletion. The main result of the research is that the remaining legacy surface is finite and clustered. The repo no longer has one giant hidden candidate-first backbone. Instead, the remaining work breaks cleanly into internal pass-slot names, compatibility readers and translators, alias writers, and `legacy`-named current algorithms. That structure makes a full purge feasible.

The biggest risk is not technical difficulty. It is mixing fundamentally different seam types in one edit and ending up with half-renamed, half-compatible code. The plan below separates those families so each pass can delete one coherent worldview at a time and prove the result with tests.

## Context and Orientation

The current product architecture is the one described by the refactor documents: inputs are normalized into blocks, labels are authoritative, recipe spans are grouped from labels, recipes use one main correction stage, non-recipe ownership is determined before optional knowledge extraction, and run-level stage naming is semantic. The main active code paths live in `cookimport/staging/import_session.py`, `cookimport/parsing/label_source_of_truth.py`, `cookimport/staging/nonrecipe_stage.py`, `cookimport/staging/writer.py`, `cookimport/llm/codex_farm_orchestrator.py`, and `cookimport/labelstudio/ingest.py`.

What remains is not a second runtime. It is a layer of old naming and compatibility support around that runtime.

The first cluster is internal pass-slot naming. This is code that still speaks about `pass1`, `pass2`, and `pass3` even though the product contract now says `build_intermediate_det`, `recipe_llm_correct_and_link`, and `build_final_recipe`. The clearest examples are `cookimport/llm/codex_farm_orchestrator.py`, `cookimport/llm/codex_farm_contracts.py`, `cookimport/llm/codex_farm_transport.py`, and `cookimport/llm/evidence_normalizer.py`.

The second cluster is compatibility readers and translators. These modules read old artifacts, old CSV roots, old bundle shapes, or old project scopes. Examples include `cookimport/bench/upload_bundle_v1_existing_output.py`, `cookimport/bench/followup_bundle.py`, `cookimport/labelstudio/export.py`, `cookimport/analytics/dashboard_collect.py`, and `cookimport/llm/codex_farm_runner.py`.

The third cluster is alias writing. This is active code that still emits old field names or old payload shapes for downstream compatibility. The standout example is `_normalize_draft_compatibility_aliases(...)` in `cookimport/staging/writer.py`, but there are also output- and manifest-adjacent pass-name fields in benchmark and Label Studio helper code.

The fourth cluster is `legacy`-named current algorithms. In these cases, the repo is not preserving an optional legacy branch; it is preserving the old name of the current path. Examples include line-role prompt format handling in `cookimport/llm/canonical_line_role_prompt.py` and `cookimport/parsing/canonical_line_roles.py`, canonical alignment strategy naming in `cookimport/bench/eval_canonical_text.py` and `cookimport/bench/canonical_alignment_cache.py`, and parser/config values such as `legacy_medium` in ingredient handling or `legacy` aliases for extractors and splitters.

The fifth cluster is tests and fixtures that still encode the old worldview. Many tests are already updated, but some still assert pass-slot field names, legacy rejection paths, compatibility dictionary views, or old-history support. These tests must be rewritten or deleted in the same pass as the corresponding production seam, otherwise they will block the purge or encourage reintroducing it.

## Plan of Work

Start by freezing the target. The target is not “support both old and new shapes more cleanly.” The target is that active code under `cookimport/` no longer contains legacy compatibility branches or pass-slot naming except where a new canonical rename is temporarily introduced during one pass. Before touching code, run the repo-wide search commands in the validation section and save the output counts so there is a concrete before/after baseline.

The first implementation pass should remove internal pass-slot naming from the live recipe-correction path. In `cookimport/llm/codex_farm_orchestrator.py`, rename `_RecipeState` fields and any adjacent helper names so they describe current behavior instead of historical pass numbers. The same semantic rename must propagate through `cookimport/llm/codex_farm_contracts.py`, `cookimport/llm/codex_farm_transport.py`, `cookimport/llm/evidence_normalizer.py`, and any manifest or report rows that still expose `pass2_*` or `pass3_*` semantics for the current route. The replacement names should follow the refactor language. Use names like `correction_input_*`, `correction_output`, `final_assembly_*`, `correction_mapping_status`, or other equivalents that describe meaning rather than history. This pass must also update any test fixtures that still expect pass-slot fields in current manifests or helper payloads.

The second pass should remove compatibility readers and translators from product code. Delete the old-artifact translation logic in `cookimport/bench/upload_bundle_v1_existing_output.py` and rewrite any callers to consume semantic recipe topology directly. Remove `as_context_dict()` compatibility projection from `cookimport/bench/upload_bundle_v1_model.py` and update the calling scripts to use typed model fields. Remove legacy-case loading from `cookimport/bench/followup_bundle.py`, remove legacy-scope rejection machinery from `cookimport/labelstudio/export.py` by collapsing the export path to freeform-only assumptions, and delete old-helper signature fallbacks and compatibility wrappers in `cookimport/cli.py`. If a checked-in fixture or script still depends on those readers, regenerate the fixture or update the script in the same pass rather than re-adding the bridge.

The third pass should delete compatibility alias writers. Remove `_normalize_draft_compatibility_aliases(...)` from `cookimport/staging/writer.py` and adjust downstream readers and tests to consume only the canonical current draft shape. Audit `cookimport/cli.py`, benchmark artifact writers, and Label Studio helper outputs for any fields that are written solely to keep old downstream code alive. Delete those fields from new writes and update the readers instead of preserving both contracts.

The fourth pass should rename or delete all remaining `legacy`-valued active settings and algorithm labels. In `cookimport/llm/canonical_line_role_prompt.py` and `cookimport/parsing/canonical_line_roles.py`, remove the `"legacy"` prompt format and make the compact format the only supported line-role prompt representation. In `cookimport/bench/eval_canonical_text.py` and `cookimport/bench/canonical_alignment_cache.py`, rename the enforced alignment path away from `"legacy"` if it remains the canonical algorithm, or delete it if a newer alignment path replaces it. In `cookimport/config/run_settings.py`, `cookimport/parsing/ingredients.py`, `cookimport/parsing/multi_recipe_splitter.py`, `cookimport/plugins/epub.py`, `cookimport/plugins/pdf.py`, `cookimport/plugins/text.py`, and matching CLI validators, remove deprecated option values such as `legacy_medium` and `legacy` extractor/splitter aliases. If one current algorithm still needs a user-facing name, give it a canonical value and migrate tests and defaults to that value in one pass.

The fifth pass should remove historical benchmark and analytics readers. Delete old CSV path fallback probing from `cookimport/analytics/dashboard_collect.py`, legacy timestamp pattern handling from analytics and benchmark retention code, old follow-up bundle readers, and old runner progress parsing in `cookimport/llm/codex_farm_runner.py`. Remove benchmark resume and quality compatibility checks that only exist for older file layouts if the current generated artifacts already define a stable format. This pass deliberately stops supporting archived legacy roots in active code. Any golden fixtures that still rely on those historical readers should be rewritten or removed.

The sixth pass should clean tests, current docs, and repo conventions to match the post-purge world. Update current product docs in `docs/02-cli/02-cli_README.md`, `docs/05-staging/05-staging_readme.md`, `docs/06-label-studio/06-label-studio_README.md`, `docs/07-bench/07-bench_README.md`, `docs/08-analytics/08-analytics_readme.md`, `docs/10-llm/10-llm_README.md`, and nearby `README.md` or `CONVENTIONS.md` files inside `cookimport/`. Delete or rewrite tests whose only purpose is to preserve rejected legacy values, old scope rejection behavior, old-history readers, or compatibility alias output. Keep historical plan/report docs untouched unless they directly mislead current docs or are regenerated as part of a later documentation-only cleanup.

## Idempotence and Recovery

Each seam family in this plan should be removed in a separate, test-backed pass. That keeps retries safe. If one pass fails halfway, restore only that seam family’s branch in the working tree and rerun the local test slice for that family before continuing. Do not reintroduce broader compatibility just to get unrelated tests green.

Fixture and sample-root regeneration should happen into fresh timestamped directories under `/tmp` or the repo’s normal generated-output roots. Avoid mutating archived golden/sample roots in place until the new readers are stable. Once the new tests pass, replace the checked-in fixtures in a single deliberate update rather than keeping both old and new fixture families.

If a `legacy` name turns out to be the only name for a still-needed algorithm, do not back out the whole purge. Rename that algorithm to a canonical current name, update call sites and tests, and continue. The goal is to remove legacy semantics from the codebase, not to remove working behavior.

## Interfaces and Dependencies

This plan should reuse the existing test and runtime entrypoints rather than introducing new tooling. The key interfaces to preserve are:

`cookimport.cli:app` and the `cookimport` shell command. The CLI must keep working with the current run-settings surface and current semantic output contract while losing compatibility-only code.

`cookimport.config.run_settings.RunSettings`. This remains the main typed run-settings model, but its allowed values and normalization behavior should be current-contract only after the purge.

`cookimport.llm.codex_farm_orchestrator.run_codex_farm_recipe_pipeline(...)`. This remains the recipe-correction entrypoint, but its internal state and adjacent contracts should stop using pass-slot names.

`cookimport.staging.writer` output helpers. These remain the writers for active outputs, but they should no longer publish compatibility aliases.

`cookimport.bench` and `cookimport.analytics` readers. These modules remain the tools for current benchmark and dashboard workflows, but they should stop reading or translating old artifact formats once the purge is complete.
