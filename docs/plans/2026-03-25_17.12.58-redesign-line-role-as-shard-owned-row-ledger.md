## User Intent

Exact user wording from the design discussion:

    "my goal is to find the best comrpomise/system for the following: LLM accuracy, easy to retry/fix things, token cost. i guess I wonder why the LLM can't just work down a .csv (not literally a .csv that's just my frame of refernce) and just like "fill in" a blank column or three for the labelling/decisiions we want them to make. i dont really see why we cant have per row validation so if the LLM messes up it can just go "oops, i got a warning for X, i will adjust this one data point" and then keep moving and not have to redo good work. at the same time yeah I dont want metadata spam sure, but if we rethink the archetecture here we can probably mitigate lots of tradeoffs. i am very flexible!"

    "for extra context. the reason i have shards and the option to choose them in the CLI menu is so that if I choose "5" for line role shards, the total number of rows (say 1200) gets divided into shards of 240 rows per, which then 5 agents look at simultaneously"

    "lets focus on line role specifically for now. note that if I chose "6", it would be 6 workers doing (1200/6) = 200 rows each."

    "that feels really overcomplicated for the task of "label a cookbook". my goal is to play with the # shards I choose as I dial this program in so that i find the ~pareto frontier between # of agents and cost."

    "worker opens its whole shard file and fills labels into a parallel output file or working copy
    - repo validates the shard output
    - if a few rows are bad, repair only those rows, not the whole shard"

Structured restatement:

The user wants canonical line-role to feel like a worker filling in labels on a spreadsheet-like ordered list of rows, not like a hidden tree of sub-tasks. The product knob is shard count: choosing `N` should divide the total ordered line list into roughly `N` contiguous shards and let roughly `N` workers review those shards in parallel. The architecture goal is to find the practical tradeoff frontier between accuracy, retryability, and token cost by varying shard count directly. For line-role specifically, the desired runtime is simple: one worker owns one shard, reads the whole shard file, fills labels into a parallel output or working file, repo code runs cheap local validation live enough to give fast feedback inside the same worker session, and if only a few rows are wrong or malformed, repo code asks for repairs only for those rows without discarding the good work already done. The user is explicitly flexible about implementation details, but not about the core product shape: shards are the public concurrency unit, rows are the decision unit, accepted good rows should be preserved as they validate, feedback should come from local repo code rather than extra model calls, and repair should be row-local rather than whole-shard.

The user's intent is precise. For canonical line-role, shard count is the product knob for concurrency and spend. Choosing `N` means the ordered line list is partitioned into roughly `N` contiguous shards, and the runtime should let roughly `N` workers own those shards in parallel. The user does not want an extra public batching layer inside a shard. The user also does not want line-role to look like a one-shot LLM prompt dump where all good work is lost if one row is malformed. The target design is "one worker owns one shard ledger, labels the rows in that shard, repo code validates the shard result, and only bad rows are revisited."

This plan therefore replaces the current task-packet line-role runtime with a shard-owned row-ledger runtime. It is not an MVP and it is not a toy partial cleanup. The end state is the full production cutover: planning, worker surfaces, validation, repair, resume, preview, telemetry, docs, and tests all teach the new shard-owned model and stop teaching the old task-packet model.

## Purpose / Big Picture

After this plan lands, a human should be able to choose `line_role_prompt_target_count=6`, know that the ordered line list will be divided into roughly six contiguous shards, and know that each worker will own one shard ledger rather than a hidden tree of sub-tasks. The worker should see one shard input, one shard output working copy, and repo-owned repair requests only for unresolved rows. The visible proof is that runtime and preview artifacts no longer create `*.task-*` line-role ids, worker roots no longer rely on `CURRENT_TASK.md` / `assigned_tasks.json`, and repair artifacts name only the bad row ids instead of re-asking the whole shard.

This matters because it aligns the implementation with the operator's mental model and with the Codex agent harness. Codex is good at reading files, editing files, rerunning checkers, and iterating locally. It is not best served by pretending each line-role shard is a one-shot call/response blob, and it is not necessary to create dozens of internal task packets for a worker that already owns a bounded shard such as 200 or 240 rows. The desired compromise is:

- high accuracy from working over one coherent shard with local context,
- low token waste by avoiding repeated whole-shard reruns for a few bad rows,
- easy repair because repo code can isolate unresolved rows after validation and preserve accepted good rows.

The user-visible acceptance is simple. With a stable processed run and prompt preview, choosing `5` line-role shards yields five shard inputs rather than five parent shards plus hidden task packets. With a live or fake-Codex rehearsal, a line-role worker root shows `CURRENT_SHARD.md`, one shard-owned output file, and targeted repair artifacts only when validation finds bad rows. Good rows survive repair passes unchanged.

## Outcomes & Retrospective

This plan is authored but not yet implemented. The desired end state is now explicit: canonical line-role should stop teaching or relying on task packets and instead use one shard-owned ledger per worker plus targeted repair rows when validation finds unresolved output. The main remaining work is implementation and same-wave documentation/test cleanup.

The most important lesson from the design review is that line-role is not knowledge-stage. Knowledge still needs a second semantic grouping layer, but line-role does not. A simpler shard-owned runtime is therefore justified here even if the repository keeps more elaborate packet or batch machinery elsewhere.

## Context and Orientation

Canonical line-role is the first LLM-backed labeling stage in this repository. It takes an ordered list of atomic cookbook lines and assigns each line one structural label such as `INGREDIENT_LINE`, `INSTRUCTION_LINE`, or `OTHER`. In this repository, an `atomic line` is one normalized line-sized unit derived from source blocks and later regrouped into recipe spans. The line-role stage is routing-first outside recipes: reusable lesson prose should normally remain `OTHER` here and only the later knowledge stage may turn outside-recipe text into final semantic `KNOWLEDGE`.

Today the relevant code is spread across these files:

- `cookimport/parsing/canonical_line_roles/policy.py` builds deterministic baseline labels and the compact worker input rows.
- `cookimport/parsing/canonical_line_roles/planning.py` partitions the ordered line list into shards, then re-splits each shard into smaller task packets and writes task manifests plus worker instructions.
- `cookimport/parsing/canonical_line_roles/runtime.py` launches one long-lived worker session per worker assignment, writes `assigned_tasks.json`, `CURRENT_TASK.md`, scratch drafts, and one `out/<task_id>.json` per packet, then aggregates those packet outputs back into one shard proposal.
- `cookimport/parsing/canonical_line_roles/validation.py` validates packet outputs and aggregates accepted packet rows back into shard proposals, with deterministic fallback only for missing or invalid packet rows.
- `cookimport/parsing/line_role_workspace_tools.py` defines the worker-facing output contract, seed output generation, helper CLI, and validation helpers for task-scoped line-role outputs.
- `cookimport/llm/prompt_preview.py` rebuilds line-role preview inputs from the same ordered candidate list and current shard-count settings.
- `docs/04-parsing/04-parsing_readme.md` and `docs/10-llm/10-llm_README.md` currently describe the live contract as shard-owned worker sessions that still split into task packets underneath.

Terms used in this plan:

- A `shard` is one contiguous slice of the ordered line list. It is the planning and concurrency unit. If the book has 1200 candidate rows and the operator chooses `line_role_prompt_target_count=6`, the planner should create about six contiguous shards of about 200 rows each.
- A `worker` is one Codex session assigned one-or-more shards to process. When worker count is unset, the runtime should default to one live worker assignment per planned shard, capped by the existing global phase caps.
- A `row ledger` is the shard-owned pair of files that make the work understandable: one immutable shard input listing the rows and one mutable shard output or working copy listing the labels to fill in.
- A `repair request` is a repo-owned file that lists only unresolved rows plus the minimum nearby context needed to repair them. It is not a public task layer and it is not the normal happy path. It exists only after shard validation finds problems.
- `Deterministic baseline` means the existing rule-first label suggestion built before Codex runs. In the redesign it remains the seed label column, not the thing that silently rewrites accepted Codex rows later.

The current architecture problem is not that line-role uses shards. Shards are the correct operator-facing unit. The problem is that the runtime immediately converts those shards into smaller task packets, then teaches the worker and the docs to think in tasks instead of shards. The worker root currently carries `assigned_tasks.json`, `CURRENT_TASK.md`, `current_task.json`, scratch drafts per task, and helper verbs such as `finalize-all scratch/`. That is exactly the extra layer the user wants to remove from the line-role mental model.

The design target is therefore:

- shard count remains the public product knob,
- one shard remains one coherent worker-owned ledger,
- validation still stays strict and happens live during the worker loop,
- validation is "live enough," not twitchy: repo code runs cheap deterministic checks after meaningful saves or explicit check commands, not after every keystroke,
- accepted rows become durable as soon as they validate,
- repair acts only on unresolved rows,
- preview, runtime, telemetry, and docs all teach the same shard-owned story.

## Milestones

### Milestone 1: Replace task packets with shard-owned ledgers

This milestone removes the task-packet contract from line-role planning and workspace setup. At the end of the milestone, line-role workers no longer receive `assigned_tasks.json`, `CURRENT_TASK.md`, or scratch drafts per task. Instead they receive one shard-owned input, one shard-owned working output, and a shard brief. The visible result is that runtime artifacts stop creating `*.task-*` line-role ids and the worker prompt tells the worker to own one shard at a time.

Run the focused parsing tests after this milestone and inspect one fake worker root. Acceptance is that the worker root shows `CURRENT_SHARD.md`, `current_shard.json`, and one shard-owned output target rather than task packet files.

### Milestone 2: Make shard validation preserve good rows and isolate bad rows

This milestone replaces task aggregation with row-local shard validation. At the end of the milestone, the validator can read a whole-shard output, accept valid rows live as they appear, list unresolved rows, and write a repair request naming only the rows that need rework plus a small local context halo. The validator is intentionally cheap and deterministic on the happy path: it should run from repo code after explicit check commands or meaningful saves, not from extra model calls and not on every keystroke. The visible result is that a malformed shard output no longer forces re-asking the whole shard and repair artifacts list row ids rather than task ids.

Run the workspace-helper and validation tests after this milestone. Acceptance is that a synthetic shard with 200 rows and three invalid rows yields a repair request naming those three rows only, while the accepted rows stay frozen.

### Milestone 3: Cut runtime, resume, preview, and telemetry over to shard-owned semantics

This milestone rewrites the live runtime and the preview/reporting surfaces so the repository consistently talks about shards and row repair rather than packet tasks. At the end of the milestone, preview emits one line-role input per shard, runtime summaries report shard counts and row counts without task counts, and resume logic reuses validated shard proposals or repair-complete rows instead of replaying packet outputs. The visible result is that a zero-token preview or fake-Codex rehearsal shows shard-owned artifacts only.

Run prompt-preview tests, parsing/LLM domain tests, and a zero-token preview command after this milestone. Acceptance is that preview and runtime artifact roots no longer contain `task_manifest.jsonl`, `task_status.jsonl`, or `*.task-*` line-role ids.

### Milestone 4: Remove stale docs/tests and prove the end-to-end operator story

This milestone deletes the stale task-packet language from docs and finalizes the test suite around the shard-owned row-ledger model. At the end of the milestone, a newcomer reading the parsing/LLM docs will learn exactly the story the user asked for: choose shard count, get that many line-role shard owners, and repair only bad rows. The visible result is that docs, tests, preview, and runtime all agree.

Run the full targeted validation bundle and one fake-Codex runtime rehearsal after this milestone. Acceptance is that the documented story, runtime artifact story, and CLI shard-count story all line up.

## Plan of Work

Start by changing the line-role planning contract in `cookimport/parsing/canonical_line_roles/planning.py`. Keep shard planning itself. The planner should still use the current ordered candidate list and `line_role_prompt_target_count` to partition that list into contiguous shards. Delete the second-stage split into `_LineRoleTaskPlan` packets. Remove `_build_line_role_task_plans(...)`, `_build_line_role_task_manifest_entry(...)`, and any helper rows that exist only to produce task-local manifests, task-local overlap windows, or task-local ids such as `line-role-canonical-....task-001`.

Replace those task models with shard-owned models. Each shard plan should now describe exactly one worker-facing input payload for the shard, one debug payload for the shard, one output path for the shard, and enough metadata for repair. Keep the ordered row tuples if they remain the cheapest transport, but the worker-facing metadata should now live at shard scope. If a row-context halo is still useful for the happy path, attach it once at shard scope as a compact orientation section rather than repeating it as hidden per-task overlap windows. The row-level repair path will be the only place that needs row-local context halos after the first full-shard pass.

Next, redesign the worker surfaces in `cookimport/parsing/line_role_workspace_tools.py`. This file currently assumes task rows, task ids, scratch drafts, and per-task output validation. Replace that with shard-owned helpers. The end state should provide:

- one immutable shard input payload loader,
- one deterministic shard seed-output builder,
- one shard-output validator,
- one repair-request builder that extracts only unresolved rows plus nearby context,
- one repair-patch validator and merge helper,
- one `CURRENT_SHARD.md` renderer,
- one worker helper CLI that uses shard verbs such as `overview`, `show-shard`, `check-shard`, `show-repair`, `check-repair`, and `merge-repair`.

Do not preserve `CURRENT_TASK.md`, per-task scratch drafts, or `finalize-all scratch/` as the normal line-role story. A compatibility shim is allowed only while the tests are being moved in the same wave, and it must be removed before the plan is marked complete.

Then rewrite the live line-role runtime in `cookimport/parsing/canonical_line_roles/runtime.py`. Today it builds task manifests, writes `assigned_tasks.json`, prewrites `scratch/<task_id>.json`, and watches for one `out/<task_id>.json` per packet. The new runtime should instead:

- write one shard manifest row per shard and no line-role task manifest,
- assign shards to workers directly,
- write `workers/<worker>/in/<shard_id>.json` as the authoritative immutable shard input,
- optionally keep `workers/<worker>/debug/<shard_id>.json` as the richer local debug copy,
- write `workers/<worker>/out/<shard_id>.json` as the shard-owned full output file,
- write `CURRENT_SHARD.md` and `current_shard.json` instead of current-task sidecars,
- allow the worker to edit the shard output in place or through a repo-named working copy,
- run shard validation live enough after meaningful file updates or explicit checker commands, record the accepted rows immediately, and surface only the remaining unresolved rows back to the worker,
- update feedback sidecars only when the diagnostic state actually changes so the worker does not get spammed by noisy no-op refreshes,
- keep one stable shard loop in the same worker session rather than turning repair into separate conversational model calls,
- when validation fails, write `repair/<shard_id>.json` containing only unresolved rows and their local context,
- allow the worker to answer with a repair patch file that lists only the unresolved row ids,
- merge the repair patch into the accepted shard rows and rerun validation,
- use an explicit install/finalize checkpoint for Pass 1 acceptance before any later shard promotion/projection work happens in the same session.

Preserve the long-lived workspace-worker session model. The point of this redesign is not to go back to brittle one-shot direct calls. The worker may still use a local helper script and edit files in loops. The change is that the loop is shard-owned and repair-owned, not task-owned.

Then rewrite the validator in `cookimport/parsing/canonical_line_roles/validation.py`. Replace `_validate_line_role_shard_proposal(...)` and `_aggregate_line_role_task_payloads(...)` with shard-owned validation helpers. The validator must prove all of the following:

- the full shard output returns one row per owned `atomic_index`, in order,
- every label is in the allowed set,
- every `review_exclusion_reason` remains valid and only appears on `OTHER`,
- accepted rows can be separated from unresolved rows without ambiguity,
- unresolved rows can be packaged into a repair request with stable row ids,
- a repair patch may contain only unresolved row ids and may not overwrite accepted rows,
- final merged shard proposals are deterministic and complete.

Do not silently treat malformed Codex rows as accepted. If a row is malformed or semantically rejected, keep it unresolved and push it into the repair request. Deterministic baseline fallback should remain an explicit terminal fallback path after repair policy is exhausted, not a hidden side effect during the first validation pass.

The live-validation requirement is not optional. The runtime should not wait until the worker "finishes the shard" before deciding what is good. As soon as a prefix or subset of rows validates under the repo-owned contract, those accepted rows should become durable and should not be reopened by later repair prompts unless the human explicitly changes that policy in a future redesign. Just as importantly, the runtime should not become twitchy: the checker is a cheap repo-owned checkpoint that runs after meaningful saves or explicit worker commands, not a per-keystroke interruption loop.

After validation is shard-owned, update prompt preview and reporting. In `cookimport/llm/prompt_preview.py`, keep the shard-count behavior but stop projecting any line-role task-level metrics or artifacts. Preview should write one line-role input per shard and report shard sizes directly. Update any prompt-budget or summary helpers that still assume line-role has task prompt counts or task manifests beneath each shard. The operator-facing preview and budget surfaces should answer questions like "with shard count 6, how many shard prompts and how many owned rows per shard?" rather than "how many hidden packet tasks were generated inside those shards?"

Then remove stale runtime artifacts and update resume logic. Remove line-role `task_manifest.jsonl`, `task_status.jsonl`, `assigned_tasks.json`, `CURRENT_TASK.md`, `current_task.json`, and task-local `out/*.json` assumptions from runtime, docs, and test fixtures. Replace them with shard-owned status surfaces such as `shard_manifest.jsonl`, `shard_status.jsonl`, and optional repair ledgers. Resume should now look for validated shard outputs or validated merged repair results rather than scanning for completed task packet outputs. The existing packet-first artifact surfaces should be considered line-role-only technical debt to delete, not compatibility features to protect.

Update worker prompts and helper prose next. The line-role worker prompt currently teaches a task packet loop with scratch drafts and `finalize-all`. Replace that with a shard-owned loop:

1. open `worker_manifest.json`,
2. open `CURRENT_SHARD.md`,
3. open the shard input and shard output working file,
4. label the whole shard,
5. run `check-shard`,
6. if the checker writes a repair request, open that request and answer only the unresolved rows,
7. run `merge-repair` or the equivalent repo-written helper,
8. continue until the shard validates or the repo-owned fallback ladder exhausts.

The prompt should explicitly say that accepted rows remain frozen, that repair requests name only unresolved rows, and that feedback comes from cheap local repo checks rather than extra LLM chatter. The worker should be taught one stable loop: edit shard output, run local check, read updated feedback sidecar if it changed, fix only unresolved rows, then continue.

Once the runtime shape is stable, update documentation. `docs/04-parsing/04-parsing_readme.md` and `docs/10-llm/10-llm_README.md` must stop saying that line-role splits shards into smaller task packets under `assigned_tasks.json`. Replace that with the new story: line-role owns shards directly, each shard owns one row ledger, and repair requests contain only unresolved rows. Add one short note in `cookimport/parsing/canonical_line_roles/README.md` and one short note beside the worker helper file explaining that line-role is now shard-owned and row-repair-based.

Finally, update tests in the same wave. The key test families are:

- `tests/parsing/test_line_role_workspace_tools.py` for shard-owned helper verbs, validation, repair-request generation, and repair merge.

## Validation and Acceptance

The redesign is complete only if all of the following are true.

Operator acceptance:

- Interactive and non-interactive shard-count settings still mean "split the ordered line list into roughly N contiguous shards."
- Choosing `line_role_prompt_target_count=5` on a 1200-row fixture yields about five contiguous shards and, when worker count is unset, the runtime resolves roughly five concurrent worker assignments.
- The docs no longer teach any hidden public task layer for line-role.

Artifact acceptance:

- Line-role runtime roots no longer write `task_manifest.jsonl`, `task_status.jsonl`, `assigned_tasks.json`, `CURRENT_TASK.md`, `current_task.json`, or line-role `*.task-*` ids.
- Line-role preview roots no longer emit line-role task-style artifact names.
- Worker roots expose shard-owned inputs and outputs plus repair artifacts only when needed.

Worker-surface acceptance:

- The line-role worker prompt tells the worker to own one shard ledger, not many task packets.
- The worker helper CLI supports shard validation and repair flows directly.
- The worker can iteratively fix only unresolved rows after the repo writes a repair request.
- The runtime/checker loop validates live and preserves accepted rows immediately instead of waiting for one final all-or-nothing shard check.
- The runtime/checker loop uses only cheap local deterministic validation on the happy path; ordinary feedback updates must not require extra LLM calls.
- Feedback is fast enough to keep the worker in the same shard context window, but not so twitchy that every save or keystroke causes noisy sidecar churn.

Validation acceptance:

- A synthetically malformed shard output with three bad rows yields a repair request that names those three row ids only and preserves the accepted rows.
- A repair patch that tries to overwrite already accepted row ids fails closed.
- Final merged shard proposals remain complete, ordered, and deterministic.
- During a live shard session, accepted rows remain accepted across later repair cycles; the worker should never need to regenerate already-accepted rows just because a later row failed.
- The happy-path checker remains cheap: contract validation, order checks, allowed-label checks, owned-id checks, and other deterministic local checks run without extra model spend.

Resume/fallback acceptance:

- Re-running a partially completed line-role phase reuses already validated shard outputs or merged repair results instead of recreating line-role task packets.
- Deterministic fallback remains explicit and narrow: it should apply only after shard output and repair policy are exhausted, not as a hidden silent merge during the first validation pass.

Regression acceptance:

- Existing route-only semantics for outside-recipe lesson prose remain intact.
- Existing positive HOWTO / variant / title contrast-book expectations remain intact.
- Prompt preview, budget summaries, and telemetry still report meaningful line-role stage totals even though task counts have been removed.

## Idempotence and Recovery

This plan is intentionally additive-then-subtractive within one wave. It is safe to introduce shard-owned helpers and runtime paths alongside the current task helpers while tests are being updated, but the old task path must be deleted before the plan is marked complete. The implementation steps should be repeatable: rerunning preview, rerunning tests, and rerunning fake-Codex rehearsals should overwrite shard-owned artifacts cleanly.

If the cutover leaves the runtime half-migrated, the recovery rule is simple: do not preserve a mixed contract where some worker roots are shard-owned and some are task-owned. Either keep the old path temporarily behind a clearly named local branch while the new path is being finished, or finish the migration and delete the old artifacts in the same wave. A mixed line-role contract would be worse than either pure state.

If a specific shard-repair design proves too awkward during implementation, prefer simplifying the repair artifact shape rather than reintroducing a first-class task packet abstraction. The non-negotiable product rule from this plan is that shards stay the public work unit and repair stays row-local.

## Artifacts and Notes

The worker-facing line-role contract should look like this at the end.

Immutable shard input example:

    {
      "v": 2,
      "shard_id": "line-role-canonical-0003-a000401-a000600",
      "rows": [
        [401, "L4", "1 cup flour"],
        [402, "L6", "Whisk until smooth."]
      ]
    }

Shard output working file example:

    {
      "rows": [
        {"atomic_index": 401, "label": "INGREDIENT_LINE"},
        {"atomic_index": 402, "label": "INSTRUCTION_LINE"}
      ]
    }

Repair request example:

    {
      "shard_id": "line-role-canonical-0003-a000401-a000600",
      "accepted_atomic_indices": [401, 402, 403, 404],
      "unresolved_rows": [
        {
          "atomic_index": 457,
          "current_line": "Balancing Fat",
          "seed_label": "OTHER",
          "reason_codes": ["invalid_label:457:CONTENT"],
          "context_before_rows": [[454, "Earlier line"], [455, "Earlier line"]],
          "context_after_rows": [[458, "Later line"], [459, "Later line"]]
        }
      ]
    }

Repair patch example:

    {
      "rows": [
        {"atomic_index": 457, "label": "OTHER"}
      ]
    }

The important proof is that the repair patch contains only unresolved row ids, not a second copy of the whole shard.

Expected worker loop in plain language:

    1. Open the shard brief and the shard output working file.
    2. Label rows in the shard output file.
    3. Run the cheap local shard checker after a meaningful save or when the worker wants feedback.
    4. If diagnostics changed, read the updated feedback sidecar.
    5. Accepted rows stay frozen immediately.
    6. If unresolved rows remain, open the repair request and fix only those rows.
    7. Run the checker again.
    8. When Pass 1 is fully accepted, run the install/finalize checkpoint and continue to any later repo-owned promotion work in the same session.
