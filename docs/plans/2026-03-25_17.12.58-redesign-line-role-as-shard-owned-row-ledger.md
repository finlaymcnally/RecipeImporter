## User Intent

Exact user wording from the design discussion:

    "my goal is to find the best comrpomise/system for the following: LLM accuracy, easy to retry/fix things, token cost. i guess I wonder why the LLM can't just work down a .csv (not literally a .csv that's just my frame of refernce) and just like "fill in" a blank column or three for the labelling/decisiions we want them to make. i dont really see why we cant have per row validation so if the LLM messes up it can just go "oops, i got a warning for X, i will adjust this one data point" and then keep moving and not have to redo good work. at the same time yeah I dont want metadata spam sure, but if we rethink the archetecture here we can probably mitigate lots of tradeoffs. i am very flexible!"

    "for extra context. the reason i have shards and the option to choose them in the CLI menu is so that if I choose "5" for line role shards, the total number of rows (say 1200) gets divided into shards of 240 rows per, which then 5 agents look at simultaneously"

    "lets focus on line role specifically for now. note that if I chose "6", it would be 6 workers doing (1200/6) = 200 rows each."

    "that feels really overcomplicated for the task of "label a cookbook". my goal is to play with the # shards I choose as I dial this program in so that i find the ~pareto frontier between # of agents and cost."

    "worker opens its whole shard file and fills labels into a parallel output file or working copy
    - repo validates the shard output
    - if a few rows are bad, repair only those rows, not the whole shard"

Structured restatement:

The user wants canonical line-role to feel like a worker filling in labels on a spreadsheet-like ordered list of rows, not like a hidden tree of sub-tasks. The product knob is shard count: choosing `N` should divide the total ordered line list into roughly `N` contiguous shards and let roughly `N` workers review those shards in parallel. The architecture goal is to find the practical tradeoff frontier between accuracy, retryability, and token cost by varying shard count directly. For line-role specifically, the desired runtime is simple: one worker owns one shard, reads the whole shard file, fills labels into a parallel output or working file, repo code runs cheap local validation live enough to give fast feedback inside the same worker session, and if only a few rows are wrong or malformed, repo code asks for repairs only for those rows without discarding the good work already done. The user is explicitly flexible about implementation details, but not about the core product shape: shards are the public concurrency unit, rows are the decision unit, accepted good rows should be preserved as they validate, feedback should come from local repo code rather than extra model calls, and repair should be row-local rather than whole-shard.

The user's intent is precise. For canonical line-role, shard count is the product knob for concurrency and spend. Choosing `N` means the ordered line list is partitioned into roughly `N` contiguous shards, and the runtime should let roughly `N` workers own those shards in parallel. The user does not want an extra public batching layer inside a shard. The user also does not want line-role to look like a one-shot LLM prompt dump where all good work is lost if one row is malformed. The target design is "one worker owns one shard ledger, labels the rows in that shard, repo code validates the shard result, and only bad rows are revisited."

This plan therefore replaces the current task-packet line-role runtime with a shard-owned row-ledger runtime. It is not an MVP and it is not a toy partial cleanup. The end state is the full production cutover: planning, worker surfaces, validation, repair, resume, preview, telemetry, docs, and tests all teach the new shard-owned model and stop teaching the old task-packet model.


## Purpose / Big Picture

After this plan lands, canonical line-role should behave like a worker filling labels into a shard-owned spreadsheet, not like a hidden queue of packet tasks. Choosing `line_role_prompt_target_count=N` should partition the ordered atomic-line list into roughly `N` contiguous shard owners. Each worker should own one shard ledger, label rows inside that shard, get cheap local validation during the same session, preserve accepted rows immediately, and repair only unresolved rows.

The proof should be easy to inspect. Preview and runtime artifacts should show shard-owned ledgers only. Worker roots should center one `CURRENT_PHASE.md` control surface, one shard input ledger, one shard work ledger, and repair files only when validation finds unresolved rows. The repository should stop teaching line-role task packets, `assigned_tasks.json`, `CURRENT_TASK.md`, and `*.task-*` ids as part of the live line-role story.

## Decision Log

- Decision: keep `line_role_prompt_target_count` as the public operator knob and define it literally as shard count.
  Rationale: that is already how the product is explained, and it matches the desired tuning story: choose `N`, get about `N` shard owners in parallel.
  Date/Author: 2026-03-25 / Codex

- Decision: unify line-role with the shared shard-ledger runtime vocabulary used across LLM labeling stages.
  Rationale: both line-role and knowledge are fundamentally "one shard owns ordered rows, repo validates locally, repair touches only unresolved rows" systems. Using one vocabulary lowers implementation and documentation complexity.
  Date/Author: 2026-03-25 / Codex

- Decision: make `shard` the concurrency unit, `row` the validation unit, and `phase` the worker-control unit.
  Rationale: this keeps the public story simple while still allowing a shared worker control surface. For line-role, there is only one phase: `label_rows`.
  Date/Author: 2026-03-25 / Codex

- Decision: remove the hidden task-packet model from the happy path instead of preserving it as compatibility machinery.
  Rationale: preserving both models would keep the exact confusion this redesign is meant to delete.
  Date/Author: 2026-03-25 / Codex

- Decision: make accepted rows freeze incrementally and forbid repair patches from overwriting frozen rows.
  Rationale: preserving validated work is the core product requirement. If later repair can casually reopen accepted rows, the runtime still wastes work and tokens.
  Date/Author: 2026-03-25 / Codex

- Decision: make live validation cheap, deterministic, and quiet.
  Rationale: the worker needs fast feedback in the same context window, but not per-keystroke sidecar churn and not extra conversational repair calls.
  Date/Author: 2026-03-25 / Codex

- Decision: keep deterministic fallback explicit and terminal.
  Rationale: baseline labels may seed the ledger and may act as the last-resort fallback after repair is exhausted, but they must not silently rewrite accepted Codex output during normal validation.
  Date/Author: 2026-03-25 / Codex

- Decision: make semantic pathology diagnostics real promotion gates instead of telemetry-only hints.
  Rationale: the plan already said semantically rejected shard rows must stay unresolved until repair or fallback. Recording `semantic_diagnostics` while still promoting the shard contradicted the intended behavior.
  Date/Author: 2026-03-25 / Codex

- Decision: make the final proposal payload always be a complete shard ledger, even when some rows fall back.
  Rationale: reviewer-facing and resume-facing artifacts should answer one direct question: what final row labels won. Raw worker invalidity still lives in `validation_errors`, `repair_status`, and `raw_output_*` metadata.
  Date/Author: 2026-03-25 / Codex

## Outcomes & Retrospective

Canonical line-role now runs as a shard-owned single-phase ledger runtime with one shared worker-control surface, targeted repair, and no live task layer. The worker path is now `CURRENT_PHASE*` + `assigned_shards.json` + `in/` + `work/` + optional `repair/` + `out/`, and the runtime/reporting path is now `shard_manifest.jsonl` + `shard_status.jsonl` rather than line-role task ledgers.

The remaining cleanup lesson was that artifact names are not enough; the runtime has to enforce the same story internally. The first cut removed the visible task layer, but it still validated like an all-or-nothing shard packet system. The finished state now matches the actual operator story: accepted rows freeze immediately, repair receives only unresolved rows, semantic-pathology shards do not promote silently, and fallback is explicit per row rather than per whole shard. The final state is simpler and easier to inspect because every durable artifact now answers one question directly: what shard was owned, what rows were accepted, what rows were repaired, and what rows fell back.

## Context and Orientation

Canonical line-role is the first optional LLM-backed labeling stage in this repository. It takes an ordered list of atomic cookbook lines and assigns each row one structural label such as `INGREDIENT_LINE`, `INSTRUCTION_LINE`, or `OTHER`. In this repository, an `atomic line` is one normalized line-sized unit derived from source blocks and later regrouped into recipe spans. Outside recipe spans, line-role is routing-first rather than semantically final: outside-recipe prose should normally stay `OTHER` here, and only the later knowledge stage can turn non-recipe prose into final `KNOWLEDGE`.

The current code relevant to this redesign lives in these files:

- `cookimport/parsing/canonical_line_roles/policy.py` builds deterministic seed labels and compact worker-facing rows.
- `cookimport/parsing/canonical_line_roles/planning.py` partitions ordered rows into contiguous shard owners and writes shard-scoped input/debug artifacts plus hint sidecars.
- `cookimport/parsing/canonical_line_roles/runtime.py` launches long-lived worker sessions, writes shard-owned phase files, collects shard ledgers, and promotes merged shard proposals with row-level fallback metadata.
- `cookimport/parsing/canonical_line_roles/validation.py` validates shard ledgers, freezes accepted rows, rejects semantic pathologies as unresolved, and builds deterministic merged shard proposals.
- `cookimport/parsing/line_role_workspace_tools.py` defines the worker helper CLI, seed output generation, validation helpers, and output contract.
- `cookimport/llm/prompt_preview.py` writes preview artifacts and prompt-budget summaries for line-role.
- `docs/04-parsing/04-parsing_readme.md` and `docs/10-llm/10-llm_README.md` describe the live line-role path and must be updated in the same implementation wave.

Terms used in this plan:

- A `shard` is one contiguous slice of the ordered atomic-line list. It is the planning and concurrency unit.
- A `worker` is one long-lived Codex session assigned one or more shards.
- A `ledger` is a file that behaves like a machine-readable spreadsheet: one owned row per line plus the columns the worker fills in.
- A `phase` is the repo-owned current step for a worker. The shared control surface always points at one active phase. For line-role, the only active phase is `label_rows`.
- A `repair request` is a repo-owned file naming only unresolved rows plus the minimum nearby context needed to fix them. It is not a first-class task layer.
- `Deterministic baseline` means the existing rule-first seed labels built before Codex runs. In the redesign they seed the work ledger and remain an explicit terminal fallback only after repair policy is exhausted.

The architecture problem is not shard planning. Shards are already the right operator-facing unit. The problem is that line-role currently hides a second packet/task layer under those shards, then teaches workers and docs to think in tasks instead of shard-owned ledgers. This plan removes that extra layer and replaces it with one stable shard loop:

- open the current phase sidecars,
- edit the shard work ledger,
- run cheap local validation,
- keep accepted rows frozen,
- read a repair request only if unresolved rows remain,
- merge repair rows,
- install the accepted shard result.

## Milestones

### Milestone 1: Replace line-role task packets with shard-owned ledgers

This milestone removes the packet/task contract from line-role planning and worker setup. At the end of the milestone, line-role workers no longer receive `assigned_tasks.json`, `CURRENT_TASK.md`, packet-local drafts, or line-role `*.task-*` ids. Instead they receive one shard input ledger, one shard work ledger, and one shared current-phase control surface that points at the shard labeling phase.

Run focused parsing tests and inspect one fake worker root after this milestone. Acceptance is that the worker root shows `CURRENT_PHASE.md`, `CURRENT_PHASE_FEEDBACK.md`, `current_phase.json`, and one shard-owned work target instead of task-packet files.

### Milestone 2: Make validation freeze accepted rows and isolate unresolved rows

This milestone replaces task aggregation with shard-row validation. At the end of the milestone, the validator can read one shard work ledger, accept valid rows incrementally, record those accepted rows as frozen, list unresolved rows, and write one repair request naming only the unresolved row ids plus minimal context. The checker remains cheap and deterministic on the happy path.

Run workspace-helper and validation tests after this milestone. Acceptance is that a synthetic shard with 200 rows and three bad rows yields a repair request naming only those three rows, while accepted rows stay frozen and may not be overwritten by a repair patch.

### Milestone 3: Cut runtime, resume, preview, and telemetry over to shard-owned semantics

This milestone rewrites the live runtime and preview/reporting surfaces so the repository consistently talks about shards, phases, rows, and repair rather than packet tasks. At the end of the milestone, preview emits one line-role input ledger per shard, runtime summaries report shard counts and row counts without task counts, and resume logic reuses validated shard outputs or merged repair results instead of replaying packet outputs.

Run prompt-preview tests, focused runtime tests, and a zero-token preview proof after this milestone. Acceptance is that preview and runtime artifacts no longer contain line-role task manifests, task status ledgers, or `*.task-*` ids.

### Milestone 4: Remove stale docs/tests and prove the operator story end to end

This milestone deletes the stale task-first language from docs and finalizes the test suite around the shard-owned single-phase ledger model. At the end of the milestone, a newcomer reading the parsing and LLM docs should learn exactly this story: choose shard count, get that many shard owners, fill one shard ledger, and repair only unresolved rows.

Run the targeted validation bundle and one fake-Codex runtime rehearsal after this milestone. Acceptance is that docs, preview, runtime artifacts, and CLI shard-count semantics all agree.

## Plan of Work

Start in `cookimport/parsing/canonical_line_roles/planning.py`. Keep the existing contiguous shard planning based on the ordered candidate list and `line_role_prompt_target_count`, but delete the second-stage split into packet-style `_LineRoleTaskPlan` rows. Remove task-local ids, task manifests, task overlap windows, and any helper structures whose only job is to teach the worker about packet tasks inside a shard.

Replace those packet models with shard-owned ledger planning outputs. Each planned shard must describe one immutable shard input payload, one optional richer debug payload, one shard work ledger path, one shard repair path, and the metadata needed for validation, resume, and promotion. If local context halos are still useful, keep them as compact shard-level orientation data or repair-only context, not as hidden moving windows that reintroduce a packet mental model.

Then redesign `cookimport/parsing/line_role_workspace_tools.py` around the shared shard-ledger control plane. The worker surface should become phase-first rather than task-first. It should write:

- `CURRENT_PHASE.md` as the cheap first read,
- `CURRENT_PHASE_FEEDBACK.md` as the quiet validation/repair note,
- `current_phase.json` as the machine-readable phase payload,
- `in/<shard_id>.json` as the immutable shard input ledger,
- `work/<shard_id>.json` as the worker-edited shard work ledger,
- `repair/<shard_id>.json` only when unresolved rows exist.

The helper CLI should use shard/phase verbs rather than task verbs. The end state should include a seed-output builder, shard validator, repair-request builder, repair-patch validator, repair merge helper, current-phase renderer, and install helper. The preferred worker loop should be "edit work ledger, run `check-phase`, read feedback if it changed, fix only unresolved rows, run `install-phase` when clean." Do not preserve `CURRENT_TASK.md`, `assigned_tasks.json`, packet-local scratch drafts, or `finalize-all scratch/` as part of the normal line-role story.

Then rewrite `cookimport/parsing/canonical_line_roles/runtime.py`. The runtime should assign shards directly to workers, write one shard manifest row per shard, and stop writing line-role task manifests and task status ledgers. Each worker root should contain shard-owned files only. The runtime must monitor meaningful work-ledger updates or explicit helper invocations, run deterministic validation, freeze accepted rows immediately, and refresh feedback sidecars only when unresolved-row diagnostics actually change. Repair must stay inside the same long-lived worker session. When unresolved rows remain, the runtime should write one repair request and accept one repair patch that may name only unresolved row ids.

Keep the shared current-phase model even though line-role has only one phase. `current_phase.json` should still be the authoritative machine-readable control surface because that lets line-role and knowledge share one control-plane vocabulary. For line-role, the only phase name is `label_rows`, and `install-phase` means "finalize this accepted shard ledger so it can be promoted."

Then rewrite `cookimport/parsing/canonical_line_roles/validation.py`. Replace task aggregation helpers with shard-owned validation helpers that prove all of the following:

- the shard work ledger contains exactly one row per owned `atomic_index`, in order,
- every label is from the allowed line-role label set,
- every `review_exclusion_reason` is valid and appears only when policy allows it,
- accepted rows can be separated from unresolved rows without ambiguity,
- repair requests can be built from unresolved rows with stable row ids,
- repair patches may contain only unresolved row ids and may not overwrite frozen accepted rows,
- the final merged shard proposal is deterministic and complete.

Malformed or semantically rejected rows must stay unresolved until repaired or until the explicit fallback ladder is exhausted. They must not be silently treated as accepted, and deterministic baseline labels must not silently flow back in during ordinary validation.

After validation is shard-owned, update `cookimport/llm/prompt_preview.py` and related summary helpers. Preview should keep honoring `line_role_prompt_target_count`, but it should emit one input ledger per shard and summarize row counts per shard rather than hidden task counts. The operator-facing preview story should answer questions like "with shard count 6, how many shard prompts and how many owned rows per shard?" rather than "how many internal packets were generated?"

Then remove stale runtime artifacts and update resume logic. Remove line-role `task_manifest.jsonl`, `task_status.jsonl`, `assigned_tasks.json`, `CURRENT_TASK.md`, `current_task.json`, task-local `out/*.json`, and `*.task-*` ids from runtime, docs, test fixtures, and summaries. Resume should now reuse validated shard ledgers or validated merged repair results. A mixed line-role contract where some worker roots are task-owned and some are shard-owned is not acceptable as a final state.

Update worker prompts and helper prose in the same wave. The worker instructions should explicitly say that accepted rows stay frozen, that feedback comes from cheap local repo checks rather than extra LLM chatter, and that repair requests mention only unresolved rows. The worker should be taught one stable loop:

1. open `worker_manifest.json`,
2. open `CURRENT_PHASE.md`,
3. open the shard input and shard work ledger,
4. label rows in the work ledger,
5. run `check-phase`,
6. if a repair request exists, fix only those unresolved rows,
7. rerun `check-phase`,
8. run `install-phase` once the shard is fully accepted.

Finally, update docs and tests in the same wave. `docs/04-parsing/04-parsing_readme.md` and `docs/10-llm/10-llm_README.md` must stop teaching line-role task packets. ## Validation and Acceptance

The redesign is complete only if all of the following are true.

Operator acceptance:

- Interactive and non-interactive line-role shard settings still mean "split the ordered line list into roughly N contiguous shards."
- Choosing `line_role_prompt_target_count=5` on a 1200-row fixture yields about five contiguous shards and, when worker count is unset, roughly five concurrent worker assignments.
- The docs no longer teach any hidden public task layer for line-role.

Artifact acceptance:

- Line-role runtime roots no longer write `task_manifest.jsonl`, `task_status.jsonl`, `assigned_tasks.json`, `CURRENT_TASK.md`, `current_task.json`, or line-role `*.task-*` ids.
- Line-role preview roots no longer emit task-style artifact names.
- Worker roots expose `CURRENT_PHASE*` sidecars, shard-owned ledgers, and repair artifacts only when needed.

Worker-surface acceptance:

- The line-role worker prompt tells the worker to own one shard ledger in one active phase, not many task packets.
- The worker helper CLI supports shard validation, repair, and install directly.
- The worker can iteratively fix only unresolved rows after the repo writes a repair request.
- The runtime/checker loop validates live and freezes accepted rows immediately instead of waiting for one final all-or-nothing shard check.
- The runtime/checker loop uses only cheap local deterministic validation on the happy path.
- Feedback is fast enough to keep the worker in the same shard context window, but quiet enough that every save or keystroke does not cause noisy sidecar churn.

Validation acceptance:

- A synthetically malformed shard ledger with three bad rows yields a repair request that names those three row ids only and preserves accepted rows.
- A repair patch that tries to overwrite already accepted row ids fails closed.
- Final merged shard proposals remain complete, ordered, and deterministic.
- During a live shard session, accepted rows remain accepted across later repair cycles.

Resume/fallback acceptance:

- Re-running a partially completed line-role phase reuses already validated shard outputs or merged repair results instead of recreating line-role task packets.
- Deterministic fallback remains explicit and narrow: it applies only after shard output and repair policy are exhausted, not as a hidden silent merge during the first validation pass.

Regression acceptance:

- Existing route-only semantics for outside-recipe lesson prose remain intact.
- Existing positive HOWTO, variant, and title contrast-book expectations remain intact.
- Prompt preview, budget summaries, and telemetry still report meaningful line-role stage totals even though task counts have been removed.

## Idempotence and Recovery

This rollout should be additive only while the cutover is in progress. It is acceptable to introduce shard-owned helpers beside the old path temporarily while tests are being moved, but the destination is a hard cut. The old task-first line-role contract must be deleted before the plan is marked complete.

Every helper command should be safe to rerun. `check-phase` may be rerun on the same work ledger without side effects. `install-phase` should be idempotent when the shard is already accepted. Repair-merge helpers must refuse to overwrite frozen rows. If a worker session dies mid-shard, resume logic should read the shard-owned state files and continue from the accepted work already on disk rather than resetting the shard.

If a specific repair artifact shape proves awkward during implementation, simplify the artifact shape rather than reintroducing a first-class task abstraction. The non-negotiable rule is that shards stay the public work unit and repair stays row-local.

Expected worker loop in plain language:

    1. Open `worker_manifest.json` and `CURRENT_PHASE.md`.
    2. Open the immutable shard input and the shard work ledger.
    3. Label rows in the work ledger.
    4. Run the cheap local checker after a meaningful save or when feedback is needed.
    5. If diagnostics changed, read `CURRENT_PHASE_FEEDBACK.md`.
    6. Accepted rows stay frozen immediately.
    7. If unresolved rows remain, open the repair request and fix only those rows.
    8. Run `install-phase` after the shard fully validates.
