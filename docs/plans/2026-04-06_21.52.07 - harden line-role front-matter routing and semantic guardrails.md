# Harden Line-Role Front-Matter Routing And Semantic Guardrails

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan builds on two earlier checked-in ExecPlans: `docs/plans/2026-04-06_11.39.28 - upgrade inline-json prompt cores for knowledge and line-role.md` and `docs/plans/unify codex repair and recovery policy.md`. The first already improved local row ids and packet shape, but it did not harden the line-role inline prompt against front-matter and table-of-contents shards. The second standardized repair and recovery reporting, but it did not add semantic suspicion guards for pathological yet structurally valid line-role answers. This plan is the follow-on that closes the gap exposed by the `2026-04-06_20.47.27` benchmark regression.

## Purpose / Big Picture

After this change, the canonical line-role stage should stop collapsing the nonrecipe route when it sees giant front-matter or table-of-contents shards. A user should be able to inspect the same saved April 6 Salt Fat Acid Heat shard that previously melted down and see a different story: the prompt gives the model the right semantic frame, obviously pathological outputs are rejected instead of promoted, outside-recipe exclusions still require deterministic support, and suspicious shards surface clearly in run summaries instead of being reported as clean success.

The user-visible outcome is not merely “different code exists.” It is that a bad line-role answer can no longer quietly turn hundreds of knowledge candidates into navigation junk. The proof must be observable without running a live Codex benchmark: replaying the saved April 6 shard fixtures through focused tests should show that the old bad response is now flagged as semantically pathological or safely contained, and route summaries should show suspicious-pathology counters instead of a false clean pass.

This plan intentionally focuses on the line-role root causes behind the route collapse. The later benchmark projection bug, where surviving knowledge vanished from the scored artifact, is a separate downstream problem and is out of scope here.

## Surprises & Discoveries

- Observation: The live inline line-role prompt is materially thinner than the contract described in the repo docs.
  Evidence: The bad shard prompt in `data/output/2026-04-06_20.47.27/.../structured_session/prompt_initial.txt` contains only a small wrapper, allowed labels, and the packet dump. It does not include the richer “contents headings are not `HOWTO_SECTION`” and “isolated topic headings usually exclude” guidance from `llm_pipelines/prompts/line-role.shared-contract.v1.md`.

- Observation: The bad shard was not random noise. It was a coherent misread of a mixed front-matter and contents packet.
  Evidence: The packet begins with praise blurbs, publisher sign-up spam, `CONTENTS`, and then a long run of short title-like topic lines such as `What is Salt?`, `How Salt Works`, and `Using Salt`. The repair response repeats the same semantic story instead of correcting it.

- Observation: The current repair loop is mostly structural for this path. It can turn malformed JSON into valid JSON while preserving bad semantics.
  Evidence: The initial bad response used the wrong shape with `atomic_index` fields; the repair prompt only asked for valid `row_id` coverage and did not explain the semantic pathology, and the repaired answer restated the same bad labels.

- Observation: The line-role semantic pathology guard is currently disabled.
  Evidence: `cookimport/parsing/canonical_line_roles/validation.py::_validate_line_role_payload_semantics(...)` immediately returns `guard_applied=false` and `reason="runtime_baseline_semantic_guard_disabled"`.

- Observation: A clearly pathological shard can still appear clean in run summaries.
  Evidence: `line_role_stage_summary.json` for the bad run reports `needs_attention=false`, `suspicious_shard_count=0`, and `codex_policy_rejected_row_count=0` even though the first shard produced `105` `RECIPE_TITLE`, `77` `HOWTO_SECTION`, and `113` `NONRECIPE_EXCLUDE navigation` rows over front matter.

- Observation: The bad run’s summary metadata is internally confusing.
  Evidence: The bad run used inline structured sessions, but `line_role_stage_summary.json` reports `repair_recovery_policy.transport = "taskfile-v1"`. This does not cause the route collapse, but it weakens debugging exactly when clear observability is needed.

- Observation: The current route containment fix is necessary but not sufficient.
  Evidence: The patch that rejects unsupported outside-recipe `NONRECIPE_EXCLUDE` labels prevents queue starvation, but it does not stop the model from hallucinating recipe-local labels on contents rows or from producing pathological-yet-structurally-valid outputs that deserve rejection and retry.

- Observation: The fix held without widening deterministic semantic authority.
  Evidence: The final runtime still lets accepted Codex labels through on ordinary shards; the new deterministic behavior is limited to shard-profile evidence packaging, narrow semantic-pathology rejection, targeted repair diagnostics, and conservative outside-recipe containment when the shard remains unsafe.

- Observation: The real missing seam was not only prompt quality but projection safety.
  Evidence: Even after structural validation, a semantically rejected shard needed route-safe fallback rows in projection or the runtime would still be forced to choose between promoting hallucinated recipe-local labels and dropping the shard entirely. Adding `semantic_containment_rows` closed that gap cleanly.

- Observation: Transport metadata had to be written on shard status rows, not inferred later.
  Evidence: The April 6-style inline run could still summarize as `taskfile-v1` until `transport` was recorded on shard/task metadata and stage summary aggregation used that field directly.

## Outcomes & Retrospective

This plan shipped as a layered fix rather than a single heuristic tweak.

The inline structured-session prompt now derives from the shared line-role contract, carries shard-profile evidence, and includes repair-time semantic diagnostics when a shard was structurally valid but semantically implausible. `validation.py` now applies a real semantic-pathology guard for outside-recipe-heavy shards instead of returning `runtime_baseline_semantic_guard_disabled`, and `runtime_watchdog.py` treats `semantic_pathology_rejected` as a repairable outcome. `runtime.py` now projects semantically rejected outside-recipe shards through conservative `semantic_containment_rows` so hallucinated `RECIPE_TITLE` and `HOWTO_SECTION` labels do not quietly become route authority.

The saved April 6 Salt Fat Acid Heat shard is now a local replay fixture in `tests/fixtures/canonical_labeling/front_matter_pathology_shard.json`, with focused tests covering prompt shape, semantic rejection of the historical bad response, a positive knowledge-heading contrast case, and end-to-end containment through `label_atomic_lines`. Stage observability was updated at the same time so inline runs record inline transport correctly and suspicious shards show up in the compact summary.

The main lesson from the regression held: the failure was layered, so the fix had to be layered too. Prompt hardening alone would have reduced the odds of failure but still left projection unsafe. Pure containment alone would have hidden the symptom while leaving the prompt/repair path weak. The shipped solution now has one durable story for this class of failure: package evidence, use the shared contract, reject narrow semantic pathology, allow one bounded semantic repair attempt, contain unresolved outside-recipe rows safely, and report the suspicion honestly in stage summaries.

## Context and Orientation

The stage at issue is canonical line-role, the parser-owned LLM step that labels individual canonical rows before recipe grouping and before nonrecipe finalize. In plain language, this stage decides whether each line looks like recipe structure such as a recipe title or instruction, or outside-recipe material that should either be reviewed later for knowledge or excluded as obvious junk. The active line-role code lives under `cookimport/parsing/canonical_line_roles/`.

The central modules today are:

`cookimport/parsing/canonical_line_roles/runtime_workers.py` builds and runs the live inline structured-session prompts and repair prompts. This is where the thin bad prompt came from.

`cookimport/parsing/canonical_line_roles/validation.py` validates line-role outputs and is supposed to apply semantic pathology checks. Right now that semantic check is effectively disabled.

`cookimport/parsing/canonical_line_roles/policy.py` owns deterministic heuristics and the current containment rule that only accepts outside-recipe `NONRECIPE_EXCLUDE` from Codex when deterministic exclusion heuristics agree.

`cookimport/parsing/canonical_line_roles/planning.py` and `llm_pipelines/prompts/line-role.shared-contract.v1.md` together describe the intended semantic contract for line-role prompts and hints. The current inline structured-session path does not fully use that contract.

`cookimport/parsing/canonical_line_roles/runtime.py` performs final sanitization and currently applies the narrow upstream containment rule that prevents unsupported outside-recipe exclusions from starving the nonrecipe queue.

`cookimport/staging/pipeline_runtime.py` and the staging-owned route artifacts under `08_nonrecipe_route.json` and `09_nonrecipe_authority.json` are where line-role mistakes become user-visible route collapse.

The key terms in this plan are:

A “front-matter shard” means a contiguous line-role packet made mostly of endorsements, copyright or publisher matter, introductions, contents headings, chapter taxonomies, and similar book-structure text that is outside any live recipe. In this repo, such shards should usually route to `NONRECIPE_CANDIDATE` or `NONRECIPE_EXCLUDE`, not to recipe-local labels.

A “semantic pathology guard” means a repository-owned validation step that asks, “even if this JSON is structurally valid, is the answer semantically absurd for this kind of shard?” It does not try to infer the whole right answer deterministically. It only blocks impossible or deeply suspicious patterns from becoming authority.

A “safe route containment” rule means a deterministic safety boundary applied after validation fails or suspicion remains. It does not claim to know the true cookbook semantics. It prevents obviously dangerous promotion such as fake outside-recipe recipe titles or unsupported exclusions from becoming route authority.

A “saved-artifact replay fixture” means a test fixture copied from an already-saved run artifact so the repo can reproduce the failure without running a live LLM job.

The motivating failure is the bad run `data/output/2026-04-06_20.47.27/single-book-benchmark/saltfatacidheatcutdown/codex-exec/2026-04-06_20.48.02/`. The first bad shard is `line-role-canonical-0001-a000000-a000294`. Its initial packet and responses are already durable on disk and must become regression fixtures for this plan.

## Plan of Work

Start by freezing the failure as a deterministic fixture. Copy the minimum useful artifact set from the saved April 6 bad shard into a focused test fixture area under `tests/fixtures/` or the existing parsing fixture surface. The minimum useful set is the initial packet, the initial bad response, the repaired bad response, and a compact expected classification of why the response is pathological. The fixture must be small enough to keep tests readable, but faithful enough to preserve the real failure shape: blurbs, sign-up spam, `CONTENTS`, topic headings, and contents-style recipe names. Add a short fixture note explaining that it comes from the bad April 6 benchmark run and must be kept as the primary non-live regression case.

Then harden the prompt contract itself. In `cookimport/parsing/canonical_line_roles/runtime_workers.py`, replace the ad hoc inline structured-session prompt body with one built from the same semantic contract used elsewhere. The inline path should not dump the entire shared contract verbatim if that would bloat the packet, but it must include the core negative rules that matter for this failure class: chapter and topic headings are not `HOWTO_SECTION`; contents-style title lists are not recipe starts; isolated topic headings without explanatory prose usually exclude or stay nonrecipe; outside-recipe shards default to nonrecipe routing unless local recipe evidence is strong. The important point is that the live inline prompt and the documented contract must stop diverging.

While fixing the prompt, enrich the packet with deterministic evidence instead of deterministic answers. The model-facing packet should carry a small shard-profile surface such as “outside-recipe dominant,” “possible contents cluster,” “no local ingredient or instruction evidence,” or “boundary context exists only after the shard.” It should also expose compact row-local evidence where useful, for example heading-shape flags, whether the row is title-like, whether nearby rows are all short heading-shaped lines, and whether the shard begins with known front-matter cues. This evidence must be phrased as clues, not as authoritative labels. The LLM still decides the label.

Next, turn the semantic pathology guard back on in a narrow, explainable way. In `cookimport/parsing/canonical_line_roles/validation.py`, implement a real `_validate_line_role_payload_semantics(...)` pass that inspects the validated payload alongside deterministic baseline evidence and shard profile. The guard should reject patterns such as a no-recipe-anchor front-matter shard returning large runs of `RECIPE_TITLE` or `HOWTO_SECTION`, contents-style title lists promoted to recipe starts without nearby yield or ingredient evidence, or an entire mixed front-matter shard mass-labeled as `navigation` exclusion when the deterministic exclusion heuristics only support some rows. The guard should record explicit semantic diagnostics instead of collapsing everything into generic invalid output.

After semantic rejection exists, redesign repair to target semantics instead of only coverage. In `cookimport/parsing/canonical_line_roles/runtime_workers.py`, repair prompts should include the specific semantic pathology and only the unresolved or suspicious rows. The repair prompt should tell the model exactly what was suspicious in plain language, for example that a contents page was interpreted as recipe-local structure or that chapter headings were treated as `HOWTO_SECTION`. Keep the repair budget bounded and policy-driven. If the repair result still looks pathological, fail closed for semantic authority rather than promoting the shard because the JSON is well formed.

Then add safe route containment for semantically rejected outside-recipe rows. This belongs in the line-role runtime acceptance and projection seam, not in the knowledge stage. For outside-recipe rows that remain unresolved after semantic repair, do not quietly preserve hallucinated recipe-local labels. Instead, project them into a conservative route-safe fallback: `NONRECIPE_EXCLUDE` only when deterministic exclusion heuristics support the same reason, otherwise `NONRECIPE_CANDIDATE`. This is a safety boundary, not a deterministic semantic rewrite. The purpose is to fail open toward later human- or LLM-reviewable nonrecipe handling, not to pretend the repository knows the final semantic class.

Finally, repair observability. Update `line_role_stage_summary.json`, shard status rows, and worker telemetry so semantic suspicion, semantic rejection, containment fallback, and real transport mode appear explicitly. A front-matter shard that triggers the semantic guard must increment suspicious counters, carry diagnostic reason codes, and force `needs_attention=true`. Inline structured-session runs must report inline transport metadata rather than taskfile transport labels. The April 6 style failure should become obvious from the summary file alone.

## Milestone 3: Reject Structurally Valid But Semantically Pathological Shards

## Idempotence and Recovery

This plan should be implemented additively. Start by introducing the replay fixture and new diagnostics before deleting any old behavior. Keep the current narrow outside-recipe exclusion containment rule in place while adding the broader semantic-pathology protections.

Prompt changes and validation changes are safe to rerun.
