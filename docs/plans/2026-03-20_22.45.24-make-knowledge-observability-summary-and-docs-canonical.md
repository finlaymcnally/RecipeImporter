## Purpose / Big Picture

Once the runtime is correct, the repository still needs one concise way to explain what happened and one short set of docs that describe the new contract. After this change, a future operator should be able to open one knowledge-stage summary file and tell whether a run failed because of malformed outputs, poisoned workers, low-yield recovery, stale follow-up leakage, interruption, or boundary violations. The short docs should then describe the current contract without forcing reviewers to reread postmortem plans or worker-local artifact trees.

This plan is the difference between “the system now works” and “the system is now reviewable and operable.” A solo operator should not need to reconstruct packet outcomes manually from raw files after every interesting run. The summary surface should say what matters, and the short docs should explain why the stage now behaves the way it does.

The proof is that deterministic tests can assert high-signal summary counters and the short docs read like current runtime truth instead of March 20 draft history.

## Outcomes & Retrospective

This plan is not implemented yet. Its intended outcome is that the rebuilt knowledge stage becomes explainable to both operators and reviewers without requiring them to replay the March 20 postmortem manually. If it succeeds, a future reviewer should be able to read one summary file and one short doc section and understand what the knowledge stage currently guarantees.

The main open risk is verbosity creep. It is easy to respond to March 20 by creating too many metrics or too many derived files. This plan should resist that temptation. The right result is a small, stable summary surface with high signal and low ceremony.

## Context and Orientation

The likely code surfaces for this plan are `cookimport/runs/stage_observability.py`, `cookimport/llm/prompt_budget.py`, `cookimport/llm/prompt_artifacts.py`, and the short current-state docs under `docs/10-llm/` and `docs/07-bench/`.

The runtime plans before this one should already have established packet truth, bounded recovery, packet-based progress, and guaranteed finalization. This plan captures those results in compact summaries and short docs. It should not re-decide core runtime behavior. It should describe and summarize that behavior cleanly.

There are three audiences for this plan’s outputs:

- the operator who wants to know what happened in one run;
- the reviewer who wants to understand the current runtime contract quickly;
- the future debugger who wants to know whether a failure was a worker-contract problem, a validation problem, a recovery problem, or an interruption problem.

The summary surface should answer those audiences without forcing them into worker-local artifact archaeology.

## Plan of Work

### Milestone 1: Add compact stage summary counters

Add compact counters to the stage summary surface. At minimum, report:

- packet totals by terminal state;
- main-worker outcomes by category;
- salvage successes and failures;
- retry attempts and accepted retries;
- repair attempts and accepted repairs;
- stale-follow-up counts;
- canceled-follow-up counts;
- circuit-breaker activations;
- interruption counts or interruption terminal state where applicable.

Keep the names stable and human-readable. The goal is not to mirror every internal field from the packet ledger. The goal is to expose the counters that answer the operator’s first questions.

### Milestone 2: Add one high-signal summary artifact

Add one high-signal summary artifact under the knowledge root that reports packet outcomes and worker outcomes in a compact machine-readable form. If a second short human-readable sibling file proves useful during implementation, keep it very short and treat the machine-readable summary as authoritative.

This summary should be dense enough that a reviewer can answer:

- how many packets validated directly;
- how many needed salvage, retry, or repair;
- how many failed by category;
- how many workers completed cleanly, completed with transport anomalies, were watchdog-killed, or were interrupted;
- whether the stage stopped because of interruption, bounded failure, or successful completion.

### Milestone 3: Distinguish main-worker work from follow-up work

Extend prompt-budget or adjacent summary surfaces so they distinguish work done by main workspace workers from work done by structured retries and repairs. This matters because operators need to know whether cost and instability came from the main path or the recovery ladder.

Do not over-design this. One clear split in the stage summary or prompt-budget summary is enough if it is stable and easy to read.

### Milestone 4: Update the short docs

Update the short docs. `docs/10-llm/nonrecipe_knowledge_review.md` and `docs/10-llm/10-llm_README.md` should explain:

- the packet model;
- the worker contract;
- bounded recovery;
- packet-based progress;
- finalization guarantees.

If benchmark docs mention operator-visible knowledge progress or interrupted summaries, update them too. The short docs should stay short and factual. They are not supposed to retell the entire March 20 story or to replace the ExecPlans.

### Milestone 5: Run broader no-token closeout suites

Use this plan as the closeout proof path for broader no-token validation. Once the runtime slices are complete, run the relevant domain suites so the summary surfaces and docs are validated against the final behavior rather than against stale assumptions.

## Validation and Acceptance

This plan is complete when:

- one compact stage summary explains packet outcomes and worker outcomes without manual artifact counting;
- prompt-budget or nearby summary surfaces distinguish main-worker work from structured follow-up work;
- the short docs describe the rebuilt contract accurately and tersely;
- broader no-token domain suites pass or any remaining gaps are recorded explicitly.

The best acceptance artifact is a short example summary like:

    knowledge stage summary:
      packets: validated=..., salvaged=..., retry_recovered=..., repair_recovered=..., failed=...
      workers: completed_validated=..., transport_anomaly=..., watchdog_killed=..., interrupted=...
      followups: retries=..., repairs=..., stale_cancelled=..., circuit_breakers=...

The exact field names can change, but the summary must answer those questions directly.

## Idempotence and Recovery

Summary artifacts should be additive and stable. Do not generate a large family of derived files unless implementation evidence shows they materially improve debugging. Re-running summary generation over the same finalized stage should update or preserve the same summary path instead of scattering duplicates.

The short docs should be updated only once the runtime surface is stable enough to describe confidently. If terminology changes during implementation, update the docs and the summary labels together so reviewers do not have to reconcile multiple competing vocabularies.

## Artifacts and Notes

The main artifacts for this plan are:

- a compact packet-outcome summary;
- a compact worker-outcome summary;
- a clear distinction between main-worker and follow-up work in summaries;
- updated short current-state docs.

These artifacts should help future reviewers read the current contract quickly. They are not meant to replace low-level artifacts for deep debugging. They are meant to tell a reviewer whether a deep dive is even necessary.
