# Second Pass: Spinner Progress Counters Across Remaining Loops

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, long-running spinner/status updates will consistently show `task X/Y` style progress whenever a total work count is known, not only in freeform AI prelabel/decorate. The user-visible result is that bench and merge workflows no longer feel “stuck”; operators can see concrete throughput and remaining work.

You can see it working when running benchmark and split-merge commands: spinner/status lines include counters like `item 3/12`, `config 2/10`, or `phase 5/9`, and those counters increase as work completes.

## Surprises & Discoveries

- Observation: Not all “progress” paths are Rich spinners. Stage split-merge progress appears inside the worker dashboard status panel, but it is driven by the same callback concept and should follow the same `X/Y` contract.
  Evidence: `_merge_split_jobs(..., status_callback=...)` emits plain phase strings in `cookimport/cli.py`.

- Observation: `bench_sweep` already prints `Config i/n`, but nested `run_suite` item updates do not include item counters.
  Evidence: `run_sweep` in `cookimport/bench/sweep.py` emits config counts; `run_suite` in `cookimport/bench/runner.py` currently notifies by item id only.

- Observation: Label Studio import/decorate now has callback plumbing and is a good template for this second pass.
  Evidence: `progress_callback` usage in `cookimport/cli.py` and `cookimport/labelstudio/ingest.py`.

- Observation: Merge phase totals can drift if optional chunk-writing is counted in `Y` but the phase callback is emitted conditionally.
  Evidence: `_merge_split_jobs` now pre-computes whether chunk phase is part of this run and always advances phase counters consistently with that total.

## Decision Log

- Decision: Scope this second pass to remaining high-value loops where totals are already known or can be computed cheaply (bench items/configs and merge phases), instead of trying to retrofit every status line in one change.
  Rationale: Delivers visible UX gain quickly with low regression risk and clear tests.
  Date/Author: 2026-02-22 / Codex

- Decision: Prefer a shared formatting helper for progress message strings instead of copy/pasting `f"... {i}/{n}"` across modules.
  Rationale: Prevents drift and aligns with the “no spaghetti indicators” requirement.
  Date/Author: 2026-02-22 / Codex

- Decision: Treat fixed phase sequences as countable work (`phase X/Y`) when loops are not the dominant unit (split-merge pipeline).
  Rationale: The user asked for counter semantics beyond per-record loops; phase counters provide the same confidence for deterministic step chains.
  Date/Author: 2026-02-22 / Codex

## Context and Orientation

Current spinner/status presentation is split between CLI wrappers and runtime callbacks:

- `cookimport/cli.py` contains Rich `console.status(...)` wrappers for:
  - `labelstudio-import`
  - `labelstudio-decorate`
  - `labelstudio-benchmark` prediction generation
  - `bench run`
  - `bench sweep`
- `cookimport/labelstudio/ingest.py` emits progress text through `progress_callback`.
- `cookimport/bench/runner.py` emits per-item progress text through `progress_callback`.
- `cookimport/bench/sweep.py` emits per-config progress text and forwards nested progress from `run_suite`.
- `cookimport/cli.py` `_merge_split_jobs(...)` emits merge-step status through `status_callback`, shown in the worker dashboard.

Terms used in this plan:

- “Known-size worklist”: a sequence where total count is known before iteration (for example, `len(suite.items)` or `len(configs)`).
- “Phase chain”: a deterministic ordered set of major steps (for example, merge payloads -> reassign IDs -> write outputs -> write report).
- “Counter contract”: status text includes `X/Y` where `X` starts at 0 or 1 and monotonically advances to `Y`.

## Plan of Work

### Milestone 1: Shared progress message helper

Add a small shared helper module for formatting counter messages. Suggested file: `cookimport/core/progress_messages.py`.

Define functions with clear names and stable behavior:

- `format_task_counter(prefix: str, current: int, total: int, noun: str = "task") -> str`
- `format_phase_counter(prefix: str, current: int, total: int, label: str | None = None) -> str`

Keep helper behavior conservative:

- Clamp negatives to zero.
- Clamp current to total when current exceeds total.
- Render `0/0` safely for empty lists.

This helper must not import Rich or CLI modules.

### Milestone 2: Bench suite item counters (`bench run`)

Update `cookimport/bench/runner.py:run_suite(...)`:

- Enumerate `suite.items` with `start=1`.
- Emit `item X/Y` in all major per-item notifications:
  - entering item
  - generating prediction run
  - loading gold
  - loading predicted
  - evaluating
  - item completion summary

The message should remain concise and include item id, for example:

    item 3/12 [my_item] Evaluating...

This gives spinner visibility in `bench run` without changing CLI wrapper logic.

### Milestone 3: Sweep nested counter consistency (`bench sweep`)

Update `cookimport/bench/sweep.py:run_sweep(...)`:

- Ensure config status remains explicitly `config X/Y`.
- Fix nested prefix currently using zero-based config index in forwarded lambda text.
- Forward nested suite progress in a stable composite form:

    config 2/10 | item 3/12 [item_id] Evaluating...

No functional changes to sweep scoring or leaderboard logic.

### Milestone 4: Split-merge phase counters in stage dashboard

Update merge status emissions in `cookimport/cli.py:_merge_split_jobs(...)`:

- Define the merge phase sequence explicitly.
- Emit `phase X/Y` for each phase via `status_callback`.
- Include human-readable phase label in each message.

Example target message style:

    merge phase 4/9: Writing intermediate drafts...

Handle optional chunk-writing phase by computing total phases dynamically.

### Milestone 5: Tests and docs

Add focused tests:

- `tests/test_bench_runner.py` (or existing bench test file): assert `run_suite` emits item counters.
- `tests/test_bench_sweep.py` (or existing sweep tests): assert nested config/item counter formatting.
- `tests/test_cli_stage_split.py` (or existing split/merge tests): assert merge status callback receives `phase X/Y`.

Update docs after behavior is implemented:

- `docs/02-cli/02-cli_README.md` (bench and stage progress notes).
- `docs/07-bench/07-bench_README.md` (bench spinner expectations).
- `docs/05-staging/05-staging_readme.md` (merge phase status contract).
- `docs/IMPORTANT CONVENTIONS.md` if message-contract wording needs tightening after implementation details settle.
