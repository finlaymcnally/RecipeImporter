# All-Method Split Throughput Optimization

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, `cookimport bench` all-method runs should finish materially faster on heavy CUTDOWN workloads by reducing split-phase queueing and improving scheduler throughput. A user should be able to run the same suite before and after this work and observe lower wall time with equal or better scoring metrics, without changing matcher engines or enabling any LLM-based parsing paths.

The visible outcome is a faster all-method benchmark with stable score outputs, plus richer scheduler telemetry that explains where time is being spent and verifies that improvements came from split/prediction throughput rather than matcher/eval changes.

## Surprises & Discoveries

- Observation: `2026-02-27_17.54.41` was eval/alignment-dominant (`~17,807s` eval vs `~1,535s` prediction across successful configs), but `2026-02-28_01.27.21` is strongly prediction/split-dominant (`sum_prediction_seconds ~99%`, `sum_evaluation_seconds ~0.35%`).
  Evidence: `docs/reports/2026-02-27_19.38.43-all-method-benchmark-slowdown-and-cache-reuse-report.md` and `docs/reports/2026-02-28_01.51.16-thefoodlab-all-method-hotspot-report.md`.

- Observation: Cross-run canonical alignment cache reuse is already implemented with shared root defaulting to `data/golden/benchmark-vs-golden/.cache/canonical_alignment`, shifting optimization ROI away from matcher work for repeated runs.
  Evidence: section "6) Implemented Fix" in `docs/reports/2026-02-27_19.38.43-all-method-benchmark-slowdown-and-cache-reuse-report.md`.

- Observation: Oracle prioritization aligns with hotspot data: biggest wins should come from split slots, split worker shaping, and potential split/convert dedupe; matcher work should stay guardrail-only.
  Evidence: `docs/reports/oracle-foodlab-ultra-answer.md`.

- Observation: We still lack direct code-level and timeseries evidence for split/convert internals in the attached package, so milestone 4 is explicitly a feasibility-and-instrumentation milestone before committing to full dedupe rollout.
  Evidence: "Explicit missing evidence" section in `docs/reports/oracle-foodlab-ultra-answer.md`.

- Observation: Global queue and legacy scheduler paths duplicated admission/event accounting logic; without symmetric edits, behavior drifts quickly between scopes.
  Evidence: Both `_run_all_method_benchmark_global_queue(...)` and `_run_all_method_benchmark(...)` needed matching adaptive-admission and telemetry changes.

- Observation: Memory-aware split-slot caps can downshift effective slots on constrained hosts even when CPU budget allows more slots.
  Evidence: `test_resolve_all_method_scheduler_runtime_auto_eval_tail_respects_source_parallelism` initially capped to `3` until test memory was pinned high.

- Observation: Existing smart-eval-tail tests lock behavioral contracts around when over-admission is allowed.
  Evidence: `test_run_all_method_benchmark_global_queue_smart_eval_tail_admission` failed until pre-eval over-admission was constrained.

## Decision Log

- Decision: Prioritize scheduler and split/prediction throughput optimization first; matcher engine work is explicitly out of scope for this plan except regression monitoring.
  Rationale: Latest hotspot telemetry (`2026-02-28_01.27.21`) shows eval wall is tiny and matcher time is millisecond-scale for executed evals; wall-time gains are more likely in split/convert and queueing.
  Date/Author: 2026-02-28 / Codex

- Decision: Deliver this work in small, benchmark-validated increments with telemetry gates after each milestone.
  Rationale: Runtime tuning can silently trade speed for instability; staged validation reduces risk and makes regressions easier to isolate.
  Date/Author: 2026-02-28 / Codex

- Decision: Keep LLM/Codex-Farm behavior unchanged and avoid any model-dependent parsing changes in this plan.
  Rationale: Project policy currently forbids enabling AI-based parsing/cleaning for import behavior changes in this effort.
  Date/Author: 2026-02-28 / Codex

- Decision: Preserve and treat shared canonical alignment cache-root behavior as a fixed baseline constraint for this plan; do not re-scope cache root mechanics unless regressions appear.
  Rationale: That fix is already landed and is a prerequisite for interpreting current split-bound telemetry versus older eval-bound runs.
  Date/Author: 2026-02-28 / Codex

- Decision: Use Oracle’s top-5 ordering as implementation priority: (1) adaptive split slots, (2) split worker shaping under resource guard, (3) split/convert dedupe feasibility, (4) prediction write-overhead reduction, (5) wing-backlog/starvation guardrails.
  Rationale: This ranking is evidence-aligned with current telemetry and gives the highest expected wall-time leverage while preserving scoring semantics.
  Date/Author: 2026-02-28 / Codex

- Decision: Add split-slot resource-guard capping (cpu + memory) before scheduler startup instead of only capping split workers per config.
  Rationale: Slot-level oversubscription can create idle-gap churn and queue contention even when per-config worker caps are applied.
  Date/Author: 2026-02-28 / Codex

- Decision: Keep eval-tail over-admission bounded to eval-active conditions in adaptive admission logic.
  Rationale: Existing global scheduler contract expects second-wave admission to open once eval is active; unrestricted pre-eval admission regressed this behavior.
  Date/Author: 2026-02-28 / Codex

- Decision: Implement matcher/alignment guardrails as warnings + telemetry payload only, without scorer/matcher algorithm changes.
  Rationale: Current hotspot data remains split-bound; guardrails are sufficient to detect regressions while avoiding low-ROI matcher churn.
  Date/Author: 2026-02-28 / Codex

## Context and Orientation

In this repository, an all-method benchmark run evaluates many configuration variants for one or more sources. Each variant has a prediction phase (import/split/parse/write) and an evaluation phase (canonical or stage-block scoring).

The two key evidence points now included in this plan are intentionally both retained because they describe different system eras:

- Older slowdown run (`2026-02-27_17.54.41`): eval/alignment dominated total runtime and exposed cache-scope/lock-wait pain.
- Newer hotspot run (`2026-02-28_01.27.21`, thefoodlab single source): prediction/split dominates (`split_convert_seconds` and `split_wait_seconds`), while eval/matcher is low-cost.

This plan optimizes for the current bottleneck while preserving telemetry to detect reversion to the older eval-dominant profile.

Relevant files and runtime surfaces:

- `cookimport/cli.py`: all-method orchestration, scheduler scope/runtime controls, scheduler event accounting, report generation, and timeseries output.
- `cookimport/bench/eval_canonical_text.py`: canonical alignment and evaluation telemetry, including matcher and alignment-cache timing.
- `cookimport/bench/sequence_matcher_select.py` and `cookimport/bench/dmp_sequence_matcher.py`: matcher selection/runtime options (guardrail-only scope for this plan).
- `cookimport/config/run_settings.py`: settings normalization and validation; may need additions for new scheduler knobs only if required.
- `docs/07-bench/07-bench_README.md` and `docs/07-bench/runbook.md`: benchmark command and artifact contracts that must stay current with behavior changes.

Terms used in this plan:

- Heavy slot: scheduler capacity unit used for expensive split/convert phase concurrency.
- Split wait: time a config spends queued before it can enter split-active work.
- Wing backlog: backlog proxy used by the scheduler (queue pressure around split/prep phases).
- Eval tail: number of configs allowed to remain active while evaluation/post phases are in flight.

## Plan of Work

Milestone 1 establishes a reproducible baseline and reconciles both evidence profiles (older eval-heavy versus newer split-heavy). This includes confirming report fields for split wait, split convert, heavy-slot utilization, idle gap, wing backlog, eval wall share, and matcher/alignment telemetry are present and comparable across runs.

Milestone 2 implements adaptive split-slot runtime shaping in `cookimport/cli.py` so split-slot capacity, inflight pipelines, and eval-tail headroom react more intelligently under queue pressure and host utilization signals. The change must remain deterministic for a fixed input/settings set and must preserve existing report schema contracts.

Milestone 3 tunes split worker caps per active config under existing CPU/memory resource guard logic. The goal is to reduce false concurrency (too-fat active splits) and improve throughput/fairness without increasing instability.

Milestone 4 is a split/convert dedupe feasibility milestone. We first identify exact split/convert internals and artifact key inputs, then prototype safe reuse for variants sharing identical upstream conversion inputs. Rollout is conditional on strict cache-key correctness and telemetry proof.

Milestone 5 reduces prediction write overhead where it materially contributes to wall time (for example batching or minimal benchmark-artifact mode), while preserving required benchmark/debug artifacts.

Milestone 6 tunes active-during-eval and wing-backlog/starvation controls so eval/post work does not starve split throughput under saturation.

Milestone 7 adds matcher/alignment regression guardrails only. The matcher implementation remains unchanged; the goal is to detect accidental regressions early by monitoring cache-hit behavior and matcher/alignment timing signals in reports and tests.

Finally, documentation and tests are updated so a novice can run the same baseline/candidate comparison loop and verify both speed and score stability end-to-end.

## Validation and Acceptance

Acceptance requires all of the following to be true:

- Speed: Candidate all-method runs show a meaningful wall-time improvement versus baseline on the target workload, with reductions attributable to split/prediction telemetry fields rather than eval changes.
- Split bottleneck relief: `split_wait_seconds` and `split_convert_seconds` decrease in the candidate profile on target workload runs.
- Stability: No material increase in config failures, timeout restarts, or retry churn.
- Scoring integrity: Quality comparison does not show unacceptable regression in strict/practical/source-success metrics.
- Matcher/eval guardrail: `sum all_method_eval_wall_seconds / sum all_method_prediction_wall_seconds` remains in the same low envelope as current hotspot baseline (single-digit percent; expected near ~1% or lower on current workload).
- Cache guardrail: Cross-run canonical alignment cache reuse remains active for repeated unchanged inputs (`alignment_cache_hit` behavior does not regress unexpectedly).
- Reproducibility: The baseline/candidate commands above can be re-run with consistent directional results.

## Idempotence and Recovery

These steps are safe to repeat because benchmark runs are timestamped into new output directories. If a candidate tuning change causes instability, revert that milestone’s edits and re-run the same comparison commands to confirm recovery. Keep one known-good baseline run directory untouched so comparisons stay grounded.

If a run is interrupted, rerun the same command; do not delete prior outputs unless storage pressure requires cleanup. When cleaning, remove only candidate run directories, never golden datasets.
