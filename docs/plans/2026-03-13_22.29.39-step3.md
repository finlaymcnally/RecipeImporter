# Harden The Current CodexFarm 3-Pass Pipeline

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan is one focused split of `docs/plans/2026-03-13_22.18.12-codexfarm-unified-reliability-plan.md`, but it is self-contained for a novice contributor.

## Purpose / Big Picture

After this change, the active `codex-farm-3pass-v1` pipeline should stop manufacturing the specific March 6 failure families that made the follow-up packet expensive and hard to trust. Duplicate or near-duplicate pass1 spans should no longer become head-only and tail-only half-recipes, malformed auxiliary pass2 fields should not kill otherwise useful rows, corrupted extractive text should be cleaned before later stages see it, and pass3 should stop spending money on structurally broken or low-value rows.

The user-visible proof is a Salt, Fat, Acid, Heat rerun where `c10`-style auxiliary failures, `c15`/`c16`-style overlap splits, corrupted extractive text, and silent empty-mapping pass3 rows are gone or materially reduced while the current three-pass path remains the benchmark control.

## Surprises & Discoveries

- Observation: the ugliest benchmark artifacts came from damaged inputs more than from weak prompt wording.
  Evidence: the March 13 pass2/pass3 repair materials point to midpoint-clamped overlap splits, malformed `field_evidence`, and corrupted extractive arrays as the main sources of damage.

- Observation: pass3 already skips many rows deterministically; the issue is not “pass3 runs everywhere.”
  Evidence: the March 13 analysis notes that the worst cost comes from the relatively small set of pass3 calls that still run on low-value or structurally broken rows.

- Observation: fallback quality matters as much as routing quality.
  Evidence: the reports treat empty mappings, placeholder titles, and destructive hard fallback on degraded rows as failures even when the pipeline technically returned a result.

## Context and Orientation

The key files for this plan are `cookimport/llm/codex_farm_orchestrator.py`, `cookimport/llm/codex_farm_contracts.py`, `llm_pipelines/prompts/recipe.schemaorg.compact.v1.prompt.md`, and `llm_pipelines/prompts/recipe.final.compact.v1.prompt.md`.

In this repository, pass1 means recipe chunking and span selection, pass2 means schema.org-style extraction plus extractive helper fields, and pass3 means final draft and ingredient-to-step mapping work. The orchestrator stitches those stages together. Contract loading in `codex_farm_contracts.py` is the narrow gate that decides whether pass2 or pass3 output is accepted, repaired, degraded, or rejected.

Monotonic fallback means a fallback or recovery path must not silently discard recoverable recipe evidence. If a row degrades, it should keep the best-known title, ingredient candidates, instruction candidates, warnings, and provenance needed to explain the degradation.

Use the March 6 benchmark root at `data/golden/benchmark-vs-golden/2026-03-06_15.22.11` as the evidence anchor, and use `saltfatacidheatcutdown` as the first validation book because the March 13 repair materials already name concrete failure ids there.

## Plan of Work

Start in `cookimport/llm/codex_farm_orchestrator.py`. Repair pass1 overlap arbitration so duplicate or near-duplicate spans do not become head-only and tail-only half-recipes. If two candidate windows overlap heavily, preserve title-forward evidence and avoid midpoint splitting that severs instructions from ingredients. Add explicit degradation reasons for truncated recipe head, truncated recipe tail, and partial recipe window when the current span has already lost crucial context.

Then harden `cookimport/llm/codex_farm_contracts.py`. Keep `schemaorg_recipe` strict because it is the main structured output, but downgrade malformed auxiliary `field_evidence` to a recoverable issue instead of a full-row failure. Add sanitation for `extracted_ingredients` and `extracted_instructions` so control-byte corruption is removed while ordinary Unicode remains intact.

After pass2 loading is stable, tighten pass3 behavior. Update `cookimport/llm/codex_farm_orchestrator.py` and the compact prompts so live pass3 does not run on rows that are already structurally partial or obviously low value. Add an explicit `mapping_status` or equivalent field so an empty mapping is either justified as not needed or marked unclear with a named reason instead of silently passing as success. Keep the first version conservative: record the status and improve routing before adding any retry loops.

Finally, enforce monotonic fallback in the current path. When a row degrades or falls back, preserve the best-known title, ingredient candidates, instruction candidates, warning codes, and provenance. This can live in orchestrator fallback assembly or in a shared helper, but the behavior must be testable and benchmark-visible.

Keep docs updates short. If warning or degradation fields change, document them briefly in `docs/10-llm/10-llm_README.md` and, if they surface in benchmark artifacts, in `docs/07-bench/07-bench_README.md`.

## Validation and Acceptance

Acceptance is behavior-first.

First, focused tests must prove that duplicate or near-duplicate spans no longer split one recipe into two broken halves, malformed auxiliary `field_evidence` no longer kills an otherwise useful row, corrupted extractive text is cleaned, and monotonic fallback preserves recoverable evidence.

Second, the Salt, Fat, Acid, Heat rerun must show that the clearest March 6 current-path failures are gone or materially reduced. At minimum, `c10`-style auxiliary failures must not hard-fail, `c15`/`c16`-style split bundles must not recur, corrupted extractive text must no longer flow forward unchanged, and silent empty-mapping pass3 rows must stop counting as normal success.

Third, the repaired current path must remain benchmark-runnable as `llm_recipe_pipeline=codex-farm-3pass-v1`. This plan is not complete if the pipeline becomes cleaner in theory but unusable as the baseline control.

## Idempotence and Recovery

This plan is additive and safe to rerun. The focused tests and one-book benchmark command can be repeated while keeping fresh timestamped output roots. If a routing or fallback rule proves too strict, degrade to a warning or deterministic preserve-more path instead of silently discarding evidence or reverting to the old destructive behavior. Do not overwrite the March 6 baseline root.
