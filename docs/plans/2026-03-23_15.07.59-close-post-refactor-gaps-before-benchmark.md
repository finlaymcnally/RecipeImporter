## Purpose / Big Picture

The March 23 refactor batch mostly landed, but it did not leave the repository in a fully benchmark-ready state. Before spending time on a new benchmark, the repo needs one closure pass that fixes the one concrete scoring-facing regression, removes the most misleading leftover compatibility and dead-code seams, and makes the tests and docs tell the same architecture story as the runtime.

After this plan is complete, a human should be able to trust three things. First, Label Studio prediction-run projection artifacts should preserve recipe-local Stage 2 truth instead of silently degrading recipe labels during the scoring projection path. Second, the repo should no longer carry dead public-looking tip/topic helper paths or stale single-file-runtime naming that make the current architecture look less finished than it is. Third, the recipe-authority path should have one obvious semantic owner surface in active runtime code rather than a canonical payload plus an equally visible legacy override lane.

The visible proof is that the currently failing Label Studio regression test passes, the staging and Label Studio domain suites both pass, the active code search no longer shows the dead compatibility seams called out in this plan, and no benchmark-facing projection artifact contradicts the intended post-refactor contracts.

## Outcomes & Retrospective

This plan is implemented. The repo now has one benchmark-facing projection story again: `line-role-pipeline/projected_spans.jsonl` and `extracted_archive.json` stay line-level reviewer artifacts, but `line-role-pipeline/stage_block_predictions.json` is rebuilt as block-level scoring evidence keyed by source `block_index`. That change fixed the reproduced Label Studio regression and updated the atomic-projection tests to assert the new block-aware contract explicitly.

The cleanup goals also landed. EPUB/PDF no longer carry dead importer-local tip/topic helpers, stage-runtime tests now use source-job naming, and both `RecipeRefineResult` and `CodexFarmApplyResult` now expose canonical payloads instead of override maps as the recipe-authority contract. Direct orchestrator and phase-worker tests were updated to assert on canonical payloads or deterministic projections derived from them.

The main lesson from this closure pass is that the March 23 architecture changes were fundamentally correct; the remaining risk was contract drift at the edges. Once the scorer-facing projection artifact, the dead importer seams, and the stale helper/docs vocabulary were brought back into alignment, the staging and Label Studio suites returned to a benchmark-ready state without another architecture rewrite.

## Context and Orientation

The repository already completed most of the major March 23 changes. Importers now emit canonical `sourceBlocks` plus non-authoritative `sourceSupport`. The stage runtime now flows through explicit five-stage runtime objects. Stage 7 non-recipe artifacts were split into seed-routing, final-authority, and review-status files. The direct single-file stage path is gone. Recipe writers now consume `AuthoritativeRecipeSemantics` payloads first.

The remaining gaps are narrower and can be described concretely.

The first gap is in the Label Studio prediction-run projection path. `generate_pred_run_artifacts(...)` in [cookimport/labelstudio/ingest.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest.py) creates an offline prediction-run bundle used by Label Studio benchmark and eval helpers. When line-role projection is enabled, that path writes `line-role-pipeline/projected_spans.jsonl`, `line-role-pipeline/extracted_archive.json`, and a projected `line-role-pipeline/stage_block_predictions.json`. The current implementation claims that recipe-local labels are reused from authoritative Stage 2 labels, but the emitted stage payload is built from freeform projection rows keyed by canonical `line_index`. That is close enough to look right, but not strict enough to preserve the real block-level benchmark contract.

The second gap is cleanup residue from the tip/topic removal. The live `convert(...)` paths in the importers now return source-first payloads with empty recipe and non-recipe semantic outputs, but EPUB and PDF still carry standalone tip/topic helper functions and imports that no longer participate in the active pipeline. Those helpers are not harmless when they remain in public-looking importer modules, because they suggest that importers still own a side semantic lane that the rest of the repo says is gone.

The third gap was the recipe-authority compatibility seam. `AuthoritativeRecipeSemantics` was already the canonical semantic handoff into stage writing, but `CodexFarmApplyResult` and `RecipeRefineResult` still exposed the old override maps. The writers did not need them as the primary semantic surface anymore, so the remaining work was to remove them from active runtime plumbing and update the small set of direct tests that still expected them.

The fourth gap is stale test and documentation naming. The runtime is now source-job based, but some test helpers and historical wording still speak as if `stage_one_file` or the old split-only path were live concepts. That drift is small, but it matters in this repository because docs and tests are supposed to serve as reviewer-grade architecture context.

## Idempotence and Recovery

This work is safe to land incrementally because each pass has a narrow verification loop. If the Label Studio regression fix is half-finished, the safe recovery path is not to benchmark anyway. The safe recovery path is to keep the work local to the projection seam until the single failing test passes and the domain suites are green again.
