# Line-Role Semantic Repair Triggers

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, line-role should no longer accept an obviously bad but structurally valid `labels` array just because every row got some allowed label. A human should be able to rerun the same style of benchmark that failed on `saltfatacidheatcutdown` and see deterministic code package recipe-local contradiction evidence, invalidate only the suspicious rows, and route those rows through bounded repair instead of silently promoting the bad first-pass labels.

The user-visible outcome is twofold. First, recipe-local title, yield, ingredient, instruction, note, and variant structure should be more stable because the runtime will challenge outputs that violate strong local invariants. Second, repair telemetry should become more honest: when a shard only looks “complete” because the model returned a full-length array of bad labels, the shard should record semantic repair pressure instead of appearing as a clean validated first pass.

## Surprises & Discoveries

- Observation: the current semantic guard seam exists but is effectively disabled.
  Evidence: `cookimport/parsing/canonical_line_roles/validation.py` has `_apply_line_role_semantic_guard(...)` returning the incoming validity/status unchanged.
- Observation: structured inline repair already has the right execution shape for row-local semantic recovery.
  Evidence: `_run_line_role_structured_assignment_v1(...)` narrows repair to unresolved rows and can issue up to three bounded repair followups.
- Observation: line-role repair is currently contract-driven, not quality-driven.
  Evidence: `_should_attempt_line_role_repair(...)` only reacts to malformed output, missing rows, invalid labels, and other structural failures; a full-length legal label vector can still pass even if it obviously breaks recipe structure.
- Observation: the existing repair prompt packet already carries `validation_errors`, which gives a natural place to add concise deterministic evidence without reintroducing deterministic label overwrites.
  Evidence: `_build_line_role_structured_packet(...)` accepts `validation_errors` today and `_build_line_role_structured_prompt_packet(...)` already mirrors them into the model-visible packet.
- Observation: the deterministic baseline is not strong enough to serve as the only semantic trigger source for recipe-start rows.
  Evidence: in the failing three-row recipe-start fixture, the baseline only recovered `RECIPE_TITLE`; `Serves 4 generously` and `1 cup thinly sliced cabbage` both fell back to `RECIPE_NOTES`.
- Observation: variant heuristics are broad enough that they will re-poison repaired ingredient rows unless semantic-trigger logic treats variant evidence as lower-priority than stronger local labels.
  Evidence: the first pass of this change re-flagged a repaired `INGREDIENT_LINE` as `RECIPE_VARIANT` during followup validation until the variant trigger was narrowed.
- Observation: front-matter/howto heading clusters remain the main false-positive canary for any semantic repair trigger work.
  Evidence: the front-matter pathology fixture started flagging `Using Acid` / `HEAT` until title/instruction triggers required actual recipe component scaffold rather than generic heading context.

## Decision Log

- Decision: implement semantic validation as a reject-and-repair layer, not a label-rewrite layer.
  Rationale: project rules explicitly prefer deterministic evidence packaging and validation over deterministic semantic correction. The runtime should mark suspicious rows unresolved and let the model make the final semantic call during repair.
  Date/Author: 2026-04-10 / Codex
- Decision: scope the first invariant set to strong recipe-local contradictions only.
  Rationale: broad semantic policing would recreate the deterministic steering that the repo has been deliberately removing. Strong title/yield/ingredient/instruction/note/variant contradictions have clear local evidence and are the failure family seen in the benchmark regression.
  Date/Author: 2026-04-10 / Codex
- Decision: keep accepted non-suspicious rows authoritative while stripping only suspicious rows out of the accepted set.
  Rationale: the line-role runtime already supports partial authority plus narrowed repair. Preserving valid rows keeps repair small and matches the existing authority model.
  Date/Author: 2026-04-10 / Codex

## Outcomes & Retrospective

Implemented outcome:

- `validation.py` now computes row-local semantic diagnostics for strong recipe-local contradictions using raw row text, nearby row context, predicted labels, and deterministic baseline evidence.
- The semantic guard no longer rewrites labels. Instead it removes suspicious rows from `accepted_rows`, records `semantic_diagnostics`, `semantic_rejected_atomic_indices`, and `semantic_repair_hints_by_atomic_index`, and returns the shard as invalid so bounded repair can re-ask only those rows.
- `runtime_workers.py` now threads those repair hints into inline repair packets as `validation_hints`, and the repair prompt explicitly frames them as deterministic evidence to reconsider rather than answers to copy.
- `runtime_watchdog.py` now treats `semantic_invariant_violation:*` errors as repair-triggering failures.
- Shard-status metadata now preserves semantic diagnostics and counts semantic-triggered suspicious rows in observability summaries.

Validation outcome:

- `source .venv/bin/activate && pytest tests/parsing/test_canonical_line_roles_runtime.py tests/parsing/test_canonical_line_roles_front_matter_pathology.py`

Behavioral outcome:

- Full-length but locally contradictory label arrays now trigger repair even when they are structurally valid.
- Repair stays narrow: only suspicious rows are removed from authority and sent back through bounded followup.
- The guard remains conservative on known false-positive families such as front-matter knowledge/howto heading clusters.

## Context and Orientation

Canonical line-role lives under `cookimport/parsing/canonical_line_roles/`. The default live benchmark path uses the inline structured transport in `cookimport/parsing/canonical_line_roles/runtime_workers.py`. That module builds the prompt packet, calls the direct Codex runner, validates the returned label vector, and issues bounded repair followups when the validator leaves unresolved rows.

Validation lives primarily in `cookimport/parsing/canonical_line_roles/validation.py` plus the lower-level payload checker `cookimport/parsing/line_role_workspace_tools.py`. The lower-level checker is still intentionally structural: it enforces row count, row order, allowed labels, and ownership. The semantic guard hook sits one layer above that structural checker so it can look at the shard’s ordered row text and decide whether some accepted rows should be rejected for repair without mutating their labels.

The model-facing line-role contract is documented in `docs/04-parsing/04-parsing_readme.md` and `docs/10-llm/10-llm_README.md`. Those docs currently say deterministic code owns validation and repair routing but does not rewrite accepted model labels. This plan keeps that boundary. The new work should make the validation layer smarter about when to distrust a locally contradictory label pattern, but it must still avoid rewriting accepted rows to deterministic fallback labels.

Benchmark evidence for the motivating regression comes from `data/golden/benchmark-vs-golden/2026-04-10_18.22.20/`. That run showed recipe-local contradictions such as a fresh recipe title labeled as an ingredient, a yield line labeled as notes, ingredient lines labeled as knowledge/instruction/yield, and instruction lines mislabeled inside otherwise real recipe runs. Repair never fired because the output was structurally valid.

## Plan of Work

First, implement a semantic invariant detector inside `cookimport/parsing/canonical_line_roles/validation.py`. It should inspect the shard’s ordered row text, the model’s accepted rows, and the deterministic baseline as evidence only. The detector must produce row-local diagnostics for strong contradictions such as a short fresh-title line followed by yield-plus-ingredient evidence but labeled as something else, a strict yield line between a title and ingredient or method evidence but labeled as notes or variant, obvious ingredient rows inside an anchored recipe start window labeled as non-ingredient structure, obvious recipe instruction rows inside an anchored method window labeled as non-instruction structure, and similarly strong recipe-note or variant contradictions where local evidence is unusually clear.

Next, make `_apply_line_role_semantic_guard(...)` convert those diagnostics into validation-time rejection without overwriting labels. It should leave non-suspicious accepted rows alone, remove suspicious rows from `accepted_rows`, add explicit semantic validation errors and metadata, and keep enough row-level evidence to explain the rejection later. The final validation metadata should expose which atomic indices were rejected by semantic guard, which invariant codes fired, and concise evidence strings that can safely be shown to the repair turn and the human-facing artifacts.

Then, extend structured inline repair so those semantic diagnostics become part of the repair prompt packet. `runtime_workers.py` already forwards `validation_errors`; this change should also include a bounded `validation_hints` or equivalent row-local evidence list in repair packets so the model sees why those rows were considered suspicious. The prompt must frame those hints as evidence to reconsider, not deterministic answers to copy.

After that, update shard-status observability so semantic-triggered repair is visible in runtime artifacts. The shard status row builder should preserve semantic diagnostic metadata instead of dropping it on the floor. This will make later benchmark triage able to answer “did repair trigger because output was malformed, or because recipe-local invariants were violated?”

Finally, add focused tests in `tests/parsing/test_canonical_line_roles_runtime.py` and `tests/parsing/test_canonical_line_roles_front_matter_pathology.py`. The runtime tests should prove that obvious recipe-local contradictions trigger bounded repair and that repair only receives the suspicious rows. The pathology tests should prove that legitimate outside-recipe heading/prose clusters still validate cleanly and do not trip the new guard.

## Validation and Acceptance

Acceptance means all of the following are true:

1. A structurally valid inline line-role response that violates strong recipe-local invariants is marked invalid for the suspicious rows and triggers bounded repair instead of immediate promotion.
2. The runtime preserves non-suspicious accepted rows while narrowing the repair packet to only the semantically suspicious rows.
3. Repair packets include concise deterministic evidence describing why those rows were challenged.
4. Legitimate outside-recipe heading or knowledge clusters that previously validated cleanly still validate cleanly and do not trigger repair.
5. The focused parsing tests pass in the project `.venv`.

## Idempotence and Recovery

These edits are additive and safe to rerun. The new semantic guard should only prune accepted rows in memory during validation; it should not mutate source artifacts or deterministic baselines. If an invariant is too aggressive and causes false positives, the safe rollback is to remove or narrow that invariant code path while keeping the structural validation logic untouched.
