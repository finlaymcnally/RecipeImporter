# Remove Deterministic Semantic Steering From LLM Paths

This ExecPlan is complete. It now serves as the implementation note for the deterministic-semantic-steering removal, with the planning/progress sections retained as historical traceability.

## Purpose / Big Picture

After this change, Codex-backed runs should make the fuzzy semantic decisions themselves, with deterministic code limited to packaging raw evidence, enforcing output shape, enforcing ownership boundaries, and normalizing harmless syntax. In plain English, the repository should stop trying to “help” the model with deterministic tag suggestions, quality scores, semantic shard profiles, worker hint essays, and post-hoc semantic rewrites.

The user-visible reason for this change is trust. Right now the codebase contains several deterministic systems that tell the model what a row probably means or silently replace the model’s answer when repo heuristics disagree. Those systems create exactly the kind of authority bleed the user does not want. After implementation, a human should be able to inspect a Codex-backed packet or task file and see raw source evidence plus contract metadata only, not semantic guesses from deterministic code. They should also be able to run focused tests and observe that Codex-backed outputs are no longer corrected back to deterministic baseline labels for semantic reasons.

This plan is comprehensive. It covers recipe correction, recipe tagging suggestions, line-role prompt shaping, line-role post-processing, route-time exclusion guardrails, and benchmark/staging evidence helpers that still score or guess semantics from heuristics.

## Outcomes & Retrospective

This removal is implemented.

The final state is:

- Codex-backed recipe packets no longer carry `q`, `tg.s`, recipe taskfile hint payloads, or semantic worker sidecar summaries. Worker-visible recipe evidence is raw source plus contract metadata only.
- Codex-backed line-role packets no longer carry `shard_profile`, line-role worker hints no longer tell the model what a shard “probably is,” and accepted Codex labels are no longer rewritten back to deterministic baseline labels for local-support or rescue-policy reasons.
- Staging now records `unresolved_recipe_exact_evidence` instead of using fuzzy title/variant/yield/time back-projection when exact grounding is unavailable.
- Observability and docs no longer describe or count the removed Codex semantic-rejection path, semantic-profile guard, or semantic-containment fallback vocabulary.
- Fake runner, fake subprocess, and watchdog/taskfile regression tests were updated so the test harness itself no longer depends on the removed semantic-hint contracts.
- Accepted Codex line-role rows are no longer post-processed into fallback labels after validation, and malformed nonrecipe finalize categories now surface as explicit failure instead of silent `other` coercion.

Verification completed after implementation:

- `./scripts/test-suite.sh domain parsing` -> `261 passed, 184 deselected`
- `./scripts/test-suite.sh domain staging` -> `94 passed`
- `./scripts/test-suite.sh domain llm` -> `326 passed, 50 deselected`

The most useful lesson from the implementation was that semantic steering had spread beyond obvious prompt fields into test harnesses, fake runners, worker sidecars, observability counters, and staging truth-surface helpers. Removing the feature cleanly required deleting those support seams too; otherwise the repo would still have preserved a compatibility ghost of the same authority bleed the user wanted gone.

## Context and Orientation

This repository processes cookbook-like sources through stage-owned semantic steps. “Deterministic semantic steering” in this plan means any repo-authored logic that suggests what text probably means, nudges a worker toward a semantic interpretation, or replaces a worker’s semantic answer because repo heuristics disagree.

The important distinction is between raw evidence and interpreted evidence.

Raw evidence in this repository means things like:

- the exact source text row or block
- neighboring source rows used as context
- source ids such as `block_id`, `block_index`, `atomic_index`, `recipe_id`
- allowed output labels or tag categories
- output shape requirements such as “return one label per owned row”

Interpreted evidence means things like:

- “this row probably looks like front matter”
- “this candidate looks incomplete”
- “these are the most plausible tags”
- “this title is unsupported locally”
- “this variant should be rescued”

The interpreted surfaces that were removed or narrowed by this change were:

- `cookimport/llm/recipe_tagging_guide.py` for recipe tag suggestions `tg.s`
- `cookimport/llm/recipe_stage_shared.py` for recipe candidate quality hints `q`
- `cookimport/llm/recipe_stage/worker_io.py` for recipe worker semantic hint sidecars
- `cookimport/llm/recipe_stage/task_file_contract.py` for recipe taskfile hint fields
- `cookimport/parsing/canonical_line_roles/validation.py` and `cookimport/parsing/canonical_line_roles/runtime_workers.py` for line-role semantic shard profiles in structured packets
- `cookimport/parsing/canonical_line_roles/planning.py` for line-role worker hint sidecars
- `cookimport/parsing/canonical_line_roles/policy.py` and `cookimport/parsing/canonical_line_roles/runtime.py` for line-role semantic rescues, sanitizers, and Codex-to-baseline replacement
- `cookimport/staging/nonrecipe_authority.py` plus related route/exclusion logic for final semantic treatment of excluded blocks
- `cookimport/staging/recipe_block_evidence.py` for heuristic evidence back-projection into benchmark and staging truth surfaces

The live recipe prompt is still `llm_pipelines/prompts/recipe.correction.compact.v1.prompt.md`, but it no longer describes `q`; `h` remains only as attempted-parse scaffolding rather than a deterministic quality signal. The live line-role worker-visible packet no longer injects a deterministic `shard_profile`.

The final policy boundary documented in `docs/10-llm/10-llm_README.md` is narrower than the pre-change boundary: deterministic code still owns structural validation, repair routing, ownership enforcement, and public-contract normalization, but it no longer does semantic correction of accepted Codex-backed answers.

The staging docs in `cookimport/staging/README.md` were updated as part of this work so they no longer describe deterministic support gates for Codex `NONRECIPE_EXCLUDE` calls.

The benchmark and output surfaces depend on `cookimport/staging/recipe_block_evidence.py`, `cookimport/staging/knowledge_block_evidence.py`, and `cookimport/staging/stage_block_predictions.py`. Those files do not drive live prompts, but they still matter because they can create misleading truth surfaces even after the prompt path is cleaned up.

## Plan of Work

The implementation proceeded by codifying the keep-set and applying it consistently. The keep-set is simple: Codex-backed surfaces may still receive raw source evidence, nearby context rows, taxonomy inventories, allowed label sets, and output-shape rules. They may not receive deterministic semantic suggestions, semantic scores, suspicion flags, semantic summaries, or prose explaining what the repo thinks the content “really” is.

Start on the recipe side because it is smaller and has multiple tightly-coupled steering systems. In `cookimport/llm/recipe_tagging_guide.py`, remove the `s` suggestion builder entirely. Keep the broad category inventory `tg.c` and the repo-authored tag-selection rules `tg.r`, but do not provide filtered or preferred labels. The model should still know what categories exist, but it should no longer see a deterministic shortlist or be told to prefer it.

In `cookimport/llm/recipe_stage_shared.py`, delete `candidate_quality_hint` generation and remove `q` from the live shard payload. The recipe prompt in `llm_pipelines/prompts/recipe.correction.compact.v1.prompt.md` must be rewritten so it no longer describes `q` at all. While touching that prompt, also tighten the explanation of `h`: keep it only if it remains as a bare attempted parse record, and remove any wording that encourages the worker to lean on it instead of source rows.

Next, remove recipe worker semantic sidecars and taskfile hints. In `cookimport/llm/recipe_stage/worker_io.py`, delete the recipe candidate summary content that lists title hints, quality flags, and ingredient-like or instruction-like counts. If a recipe worker hint file remains, it should become operational-only: it may explain how to use `task.json`, what statuses exist, and that source rows are authoritative, but it must not summarize the semantic state of the candidate. In `cookimport/llm/recipe_stage/task_file_contract.py`, simplify worker-visible evidence to `recipe_id`, `source_text`, and `source_rows` only. Remove `hint`, `quality_flags`, and `candidate_tags` from `task.json`.

After the recipe prompt path is clean, remove line-role prompt steering. In `cookimport/parsing/canonical_line_roles/validation.py` and `cookimport/parsing/canonical_line_roles/runtime_workers.py`, delete `shard_profile` from the structured packet and prompt. The worker should see the owned rows and nearby context rows only. Any semantic diagnostics used for repair should also be reviewed carefully: keep diagnostics only if they describe contract failure in observable terms, not if they restate repo heuristics about what the shard “looks like.”

Then remove line-role worker semantic hint sidecars from `cookimport/parsing/canonical_line_roles/planning.py`. If a hint file remains, it must be operational-only, like “open `task.json`, label each row once, run `task-handoff`.” It must not contain semantic defaults for `NONRECIPE_EXCLUDE`, endorsements, intro framing, headings, or any other content class.

The next phase is the hardest and most important: remove deterministic semantic rewrites from the live line-role path. In `cookimport/parsing/canonical_line_roles/policy.py` and `cookimport/parsing/canonical_line_roles/runtime.py`, narrow `_apply_repo_baseline_semantic_policy(...)` and `_reject_codex_prediction_to_baseline_if_policy_violated(...)` so that Codex-backed runs are no longer semantically corrected by deterministic heuristics. Structural contract normalization may remain for impossible or non-public labels, but rules such as `sanitized_title_without_local_support`, `sanitized_variant_without_local_support`, `rescued_other_to_variant`, `rescued_other_to_instruction`, and the like must be deleted from the Codex-backed authority path. If a rule only exists to express “repo heuristics disagree with the model,” it is out of bounds.

After that, remove deterministic support gates around exclusion semantics. Audit the route and staging seam so that a Codex `NONRECIPE_EXCLUDE` decision is not accepted or rejected because a local heuristic agrees or disagrees. If excluded blocks are still projected to final `other`, that projection must be a direct consequence of the accepted model decision, not of deterministic support checks. Update `cookimport/staging/README.md` and any runtime code that still describes or enforces local exclusion support.

Finally, clean up downstream heuristic evidence assignment. In `cookimport/staging/recipe_block_evidence.py`, remove scored title selection and fuzzy evidence back-projection where it invents confidence the pipeline does not really have. Replace heuristic scoring with one of two outcomes: exact grounded mapping from authoritative semantics back to blocks, or explicit unresolved evidence metadata when no exact grounded mapping exists. The same principle applies to variant, yield, and time mapping: if the system cannot match confidently from the authoritative payload plus exact source evidence, it should say “unresolved” rather than guess.

As each stage of removal lands, update tests and docs immediately so the new boundary becomes the documented product contract rather than a temporary cleanup.

## Validation and Acceptance

Acceptance is behavioral and architectural.

Recipe path acceptance requires all of the following:

- recipe correction packets no longer include `q`
- recipe prompts no longer describe deterministic quality metadata
- recipe worker sidecars no longer summarize semantic quality flags or title/ingredient/step hints
- recipe taskfile evidence no longer exposes deterministic hint objects
- `tg.s` no longer exists in worker-visible recipe tagging guidance

Line-role acceptance requires all of the following:

- structured packets no longer include `shard_profile`
- line-role prompts no longer inject semantic shard-profile prose
- line-role worker sidecars contain only operational instructions, not semantic defaults
- Codex-backed line-role outputs are not rewritten to the deterministic baseline because repo heuristics disagree

Staging and benchmark acceptance requires all of the following:

- route-time exclusion semantics are no longer gated by deterministic support checks
- downstream recipe block evidence does not use heuristic scores to invent title, variant, yield, or time ownership
- when exact evidence is unavailable, unresolved evidence is recorded instead of guessed evidence

The tests should prove not only that fields disappeared, but that the new boundary holds. In particular:

- a Codex-backed label that is structurally valid but semantically surprising should survive unless it violates ownership or output contract rules
- a benchmark or stage evidence helper should emit unresolved evidence rather than a heuristic best guess when exact mapping fails
- taxonomy inventories still appear where needed, but no deterministic shortlist or preference order is shown to the worker

## Idempotence and Recovery

This removal should be performed in layers so the system stays debuggable.

The safe order is:

1. Remove worker-visible semantic hints from recipe and line-role prompts.
2. Remove worker-visible semantic sidecars and taskfile hint objects.
3. Remove semantic rewrite and Codex rejection logic from the live line-role path.
4. Remove route/exclusion support gates.
5. Remove downstream heuristic evidence scoring.

This order is idempotent because each stage can be repeated safely and independently verified with focused tests. If one stage breaks behavior, keep the earlier prompt cleanup and roll only the latest layer back in code while preserving the tests that describe the desired boundary.

The rollback rule is simple: if a change weakens structural validation or ownership enforcement, revert that specific structural change. Do not reintroduce semantic steering as a quick fix. If the model path becomes too noisy after steering removal, the correct response is to improve prompt wording or worker-visible raw evidence, not to restore deterministic semantic hints.
