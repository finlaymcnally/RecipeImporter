# Knowledge Pass Authority And Semantic Ownership

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan follows `docs/plans/2026-03-27_17.42.49-knowledge-paved-road-prompt-and-surface-consolidation.md`. That earlier plan addressed the operational worker loop: one obvious helper, one active ledger, and a calmer prompt/tooling surface. This plan addresses the remaining semantic problem: knowledge still behaves too much like a blank-page generation task instead of a bounded two-pass review where the model owns the semantic call and the repo owns only the phase controller, validation, and bookkeeping.

## Purpose / Big Picture

After this change, the knowledge stage should stop pretending deterministic code can meaningfully guess `knowledge` versus `other`. A worker should receive a bounded shard of blocks, make its best semantic judgment in Pass 1, and then continue into Pass 2 using the accepted Pass 1 result as the authoritative substrate. The repo should still control ordering, exact coverage, phase advancement, validation, and installed output shape, but it should stop supplying fake semantic drafts that invite the model to second-guess or regenerate the task from scratch.

The user-visible effect is a knowledge worker loop that is both simpler and more honest. Pass 1 becomes “label these owned blocks on their merits.” Pass 2 becomes “group only the kept knowledge blocks from Pass 1.” The worker should no longer see `all other` scaffolds or blank grouping shells as if those were useful drafts. A reviewer looking at a future worker root should be able to say: the model made the semantic call, the repo enforced exact ownership and phase handoff, and the second pass built on the accepted first pass instead of re-litigating it.

## Context and Orientation

The knowledge stage is the optional large-language-model review of non-recipe book text. In this repository, “knowledge” means durable cooking leverage: guidance that helps future cooking decisions, diagnosis, or technique. The opposite class, `other`, means text that is not worth preserving in that final knowledge artifact, even if it is cooking-adjacent, true, or well-written.

The current live workspace implementation is phase-first. That means one worker session owns a shard, starts in Pass 1, and advances into Pass 2 in the same workspace. The relevant live files are:

- `cookimport/llm/knowledge_stage/workspace_run.py`, which writes the worker root, phase sidecars, tool script, and per-shard files and advances the same session through Pass 1 and Pass 2.
- `cookimport/llm/knowledge_stage/recovery.py`, which builds the main worker prompt and also owns retry and recovery behavior for the direct-exec runtime.
- `cookimport/llm/knowledge_phase_workspace_tools.py`, which defines the active helper CLI, the pass-specific ledger shapes, validation, repair payloads, and final-output assembly.
- `cookimport/llm/codex_farm_knowledge_ingest.py`, which validates final installed shard outputs and promotes accepted outputs into the stage-level artifacts.
- `docs/10-llm/10-llm_README.md` and `cookimport/llm/README.md`, which describe the current worker/runtime contract and must stay aligned with the code.

The current data flow works like this:

1. The planner writes `in/<shard_id>.json`, which contains the authoritative owned block payload for the shard.
2. The workspace starts in Pass 1 and writes `work/<shard_id>.pass1.json`.
3. The worker edits that Pass 1 ledger and runs `python3 tools/knowledge_worker.py check-phase`.
4. If validation passes, `install-phase` freezes the accepted Pass 1 work and creates the Pass 2 input/work surface.
5. The worker edits `work/<shard_id>.pass2.json`, validates it, and installs the final `out/<shard_id>.json`.

The phrase “mechanical truth” in this plan means facts the repo can know without pretending to understand semantics. Examples are block order, block ids, exact ownership, exact text, the active phase, which file is current, and whether every owned block has been covered once. “Semantic truth” means the judgment about whether a block is durable cooking guidance and how kept blocks should be grouped into topic ideas. This plan says mechanical truth belongs to the repo and semantic truth belongs to the model.

The current semantic mismatch is in `cookimport/llm/knowledge_phase_workspace_tools.py`. Pass 1 currently starts from a ledger where every row is already set to `other`. That is a semantic guess, even if conservative. Pass 2 currently starts from a ledger where every kept row has a synthetic `group_id` and a blank `topic_label`. That is not a useful continuation of semantic work; it is a blank grouping task wearing a ledger shape. The user wants the repo to stop making those fake semantic moves.

The user's raw wording for the desired contract is preserved here exactly:

    PASS 2 SHOULD BE BUILD ON THE WORK THE LLM DID ALREADY IN PASS 1. THAT ISN"T A SUGGESTION, ITS BUILDING ON THE WORK IT DID BEFORE. IT SHOULD BE CONFIDENT, NOT SECOND GUESSING ITSELF.

Rewritten as one coherent design paragraph: the repository should stop trying to supply semantic guesses for the knowledge stage at all. Pass 1 should receive the owned blocks and ask the model for its best judgment directly, because the model is the first semantic authority for `knowledge` versus `other` here. Pass 2 should then continue from the accepted Pass 1 result as authoritative prior work, not as a hint or tentative draft, and should organize only the knowledge rows the model already chose rather than asking it to second-guess the same semantic decision a second time.

This plan deliberately does not ask the repo to become smarter at semantic prediction. It asks the repo to become more honest and narrower: give the worker the exact owned rows, maintain the phase controller, validate exact coverage, and stop inserting semantic defaults that either bias the model or force it to regenerate structure from scratch.

## Plan of Work

Start in `cookimport/llm/knowledge_phase_workspace_tools.py` and redesign the worker-facing ledger contracts for both passes. The new Pass 1 design must stop seeding semantic categories as `other`. The easiest acceptable implementation is to make the Pass 1 work ledger a minimal owned-row ledger that carries one row per owned block with a required `block_index` and no repo-authored semantic category default. If the helper script needs a writable file with valid JSON from the start, the scaffold can still be structurally complete, but it must not teach the worker that the repo has already chosen `other` for every row. If the final design instead keeps `category` present for schema reasons, it must be accompanied by validation and prompt wording that clearly treats any scaffold value as placeholder-only and not semantic guidance. The preferred result is a ledger shape where the worker must make the semantic choice directly.

Redesign Pass 2 so it is explicitly built from the accepted Pass 1 knowledge rows and only asks for the semantics that remain unresolved after Pass 1: which kept blocks belong together and what concise topic label names that group. The Pass 2 input should continue to be derived only from blocks the worker kept in Pass 1. The Pass 2 work ledger should stop presenting blank shell rows as if the worker is filling a generated form with synthetic semantic identifiers. Instead, the worker should see the accepted kept rows and provide grouping assignments in the narrowest possible way. One acceptable design is a row ledger that maps each kept `block_index` to a repo-owned canonical cluster slot plus model-authored `topic_label`. Another acceptable design is a compact group proposal shape where the worker lists group membership and labels and the repo assigns final stable ids during install. The important rule is that Pass 2 should feel like “organize the knowledge rows I already chose,” not “invent a fresh scaffold from nothing.”

Move purely structural identifier work out of the model path wherever possible. If the final installed artifact still requires `group_id` values such as `g01`, `g02`, and so on, those should be canonicalized by the repo during `install-phase` or final-output assembly rather than authored freehand by the model. The model’s semantic responsibility is block membership and `topic_label`. The repo’s structural responsibility is stable identifier formatting, output ordering, and exact installed-shape normalization.

Once the ledger shapes are redesigned, update the helper validation and final-output assembly in `cookimport/llm/knowledge_phase_workspace_tools.py` so the repo still enforces exact owned-block coverage, exact kept-block grouping coverage, and stable installed output. Validation must continue to fail closed on missing rows, duplicate coverage, ungrouped knowledge blocks, or extra blocks. The redesign is not allowed to weaken correctness checks; it should only narrow who is responsible for semantic versus structural work.

After the helper contracts are updated, revise `cookimport/llm/knowledge_stage/recovery.py` so the prompt tells the worker exactly the new truth: Pass 1 is your authoritative semantic judgment on the owned blocks, and Pass 2 continues from the accepted kept rows rather than treating them as hints. The prompt must say that the repo does not know the semantic answer ahead of time and that the worker should make the best judgment it can from the owned rows and shard-local context. It must also say that the repo will preserve accepted Pass 1 work into Pass 2, so the worker should think of Pass 2 as refinement and grouping over its own prior accepted decisions, not as a fresh review.

Update `cookimport/llm/knowledge_stage/workspace_run.py` and any phase-sidecar rendering in `cookimport/llm/knowledge_phase_workspace_tools.py` so the current phase briefs and feedback files reinforce the new contract. A novice reading `CURRENT_PHASE.md` should immediately understand what semantic question is active in the current pass and what the helper will preserve or derive on their behalf. The sidecars should be very clear about what the worker owns now and what the repo will derive later.

Add or update focused tests in `tests/llm/` to lock in the redesigned Pass 1 and Pass 2 scaffolds, validation behavior, and installed output assembly. Those tests must prove at least these things: Pass 1 no longer teaches universal `other` as the semantic starting point; Pass 2 is built only from accepted Pass 1 knowledge rows; final installed output still contains exact `block_decisions` and `idea_groups`; and any repo-owned canonicalization of `group_id` is deterministic and stable.

Finish by updating `docs/10-llm/10-llm_README.md` and `cookimport/llm/README.md` so they describe the new semantic ownership model in current tense. Those docs should say plainly that the repo owns phase control, coverage, and identifier normalization, while the model owns the semantic `knowledge|other` decision and the topic grouping over the kept rows.

## Validation and Acceptance

Acceptance requires all of the following to be true:

1. Pass 1 no longer starts from a repo-authored semantic judgment that every block is `other`.
2. The worker prompt and sidecars clearly tell the worker that Pass 1 is its first-authority semantic judgment over the owned blocks.
3. Pass 2 is constructed only from accepted Pass 1 knowledge rows and is framed as continuation over that accepted work, not as a second opinion or hint-driven regrouping pass.
4. The model is no longer responsible for meaningless structural formatting when the repo can derive it deterministically after validation, especially stable `group_id` spelling.
5. Final installed outputs still validate exact owned-block coverage and exact kept-block grouping coverage and still produce the expected `packet_id`, `block_decisions`, and `idea_groups` shape.
6. Focused tests in `.venv` pass and explicitly lock in the new pass contracts.
7. `docs/10-llm/10-llm_README.md` and `cookimport/llm/README.md` describe the same semantic ownership story the code now implements.

Behavioral acceptance for a human reviewer is simple: when they inspect a future knowledge worker root, they should not see the repo pretending to know the semantic answer before the worker does. They should see a phase controller that asks the model for the semantic judgment in Pass 1 and then asks it to organize only its accepted knowledge results in Pass 2.

## Idempotence and Recovery

This work is a contract redesign, not a data migration. Re-running tests and regenerating worker roots in test fixtures should be safe. The implementation should preserve the existing phase-first architecture and exact validation invariants so the redesign can be landed incrementally without weakening correctness.

If the new contract breaks existing tests or assumptions, recover by restoring the previous helper behavior behind a failing regression test that captures the newly desired semantic ownership. Do not paper over the problem by reintroducing stronger deterministic semantic defaults. The safe fallback is always “the repo controls mechanics and the model controls semantics,” even if a temporary compatibility branch is needed while tests are updated.

Do not validate this plan by running live Codex-backed book processing or benchmarks without explicit user approval. The intended proof path is focused unit and contract tests plus inspection of generated workspace artifacts.
