# QualitySuite Levers (bench quality-run)

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This repository has ExecPlan requirements in `docs/PLANS.md`. This document must be maintained in accordance with that file.

## Purpose / Big Picture

After this change, running QualitySuite experiments is driven by a single JSON file that contains easy “levers” (toggles) with `enabled: true/false`. The file expands into a deterministic list of experiments without interactive prompts. This makes it easy for an AI agent (or a tired human) to:

1. Run a baseline experiment.
2. Turn one feature family on (or off) and re-run as a separate experiment.
3. Optionally run an “all_on” experiment that enables every selected lever at once.
4. Optionally control the all-method scheduler/runtime knobs used by QualitySuite runs (parallelism, timeouts, sharding) using the same experiments file, instead of silently ignoring those settings from `cookimport.json`.

You can see it working by running `cookimport bench quality-run` with a schema-v2 experiments JSON that includes a `levers` list, and observing:

1. `experiments_resolved.json` contains generated experiments derived from levers.
2. The run executes one output directory per generated experiment under `experiments/<experiment_id>/`.

## Surprises & Discoveries

- Observation: Current `bench quality-run` experiments schema is minimal (`schema_version: 1`, `experiments[]` with `run_settings_patch`), and the runner ignores `all_method_*` runtime knobs present in `cookimport.json`.
  Evidence: `cookimport/bench/quality_runner.py` merges base payload into `RunSettings` and calls `cookimport.cli._run_all_method_benchmark_multi_source(...)` without passing max parallelism/timeouts/sharding arguments.

## Decision Log

- Decision: Introduce `schema_version: 2` for experiments JSON with an additive `levers` section and optional `all_method_runtime_patch` fields.
  Rationale: v1 is already in use; pydantic models are `extra="forbid"` so the clean way to add new keys is a version bump while preserving v1 compatibility.
  Date/Author: 2026-02-28 (Codex)

- Decision: Default lever expansion strategy is “baseline + each enabled lever”, with optional “all_on”.
  Rationale: This stays small and predictable; a cartesian grid can explode run counts unexpectedly and is not the default.
  Date/Author: 2026-02-28 (Codex)

## Outcomes & Retrospective

- Implemented schema v2 experiments files for QualitySuite with a `levers` section (baseline + each enabled lever + optional `all_on` merge) and optional `all_method_runtime_patch` per lever/experiment.
- QualitySuite now forwards all-method runtime knobs (parallelism, timeouts, sharding, smart scheduler, etc.) into the actual `_run_all_method_benchmark_multi_source(...)` call, and records the resolved values in `experiments_resolved.json`.
- Added a timestamped example lever file under `data/golden/bench/quality/experiments/` and documented schema v2 in bench/CLI docs.

## Context and Orientation

QualitySuite is the repo’s deterministic “quality regression loop”:

1. `cookimport bench quality-discover` creates a suite JSON (targets + selected target IDs).
2. `cookimport bench quality-run --suite <suite.json> --experiments-file <experiments.json>` runs one all-method benchmark per experiment and writes a summary.
3. `cookimport bench quality-compare --baseline <run_dir> --candidate <run_dir>` compares results and can gate regressions.

Key files:

- `cookimport/cli.py`: defines `bench quality-run` CLI and the all-method benchmark runtime.
- `cookimport/bench/quality_runner.py`: loads experiments JSON, merges base settings, resolves experiments, runs each experiment sequentially, writes `experiments_resolved.json`.
- `cookimport/config/run_settings.py`: defines `RunSettings` (parsing/extraction knobs). Note: `all_method_*` scheduler/runtime keys are NOT part of `RunSettings`.
- `docs/07-bench/07-bench_README.md`: benchmark and quality-run contracts.

“Run settings” means values that are part of `RunSettings` and affect parsing/extraction behavior (e.g. `multi_recipe_splitter`, `ingredient_missing_unit_policy`).

“All-method runtime knobs” means settings that affect scheduling/parallelism/timeouts for the multi-source all-method benchmark (e.g. max inflight pipelines). These are currently stored in `cookimport.json` for interactive runs but are not consumed by quality-run.

## Plan of Work

1. Extend the experiments JSON schema in `cookimport/bench/quality_runner.py`:

   - Keep schema v1 support as-is: `schema_version: 1` with `experiments[]` entries of `{id, run_settings_patch}`.
   - Add schema v2 support with:
     - `experiments[]` entries of `{id, run_settings_patch, all_method_runtime_patch?}`.
     - `levers[]` entries of `{id, enabled?, run_settings_patch?, all_method_runtime_patch?}`.
     - `include_baseline` (default true), `baseline_id` (default "baseline").
     - `include_all_on` (default false), `all_on_id` (default "all_on").
   - Expand schema v2 into a concrete list of experiments before execution:
     - Baseline experiment (empty patches) when enabled.
     - One experiment per enabled lever (baseline + lever patches).
     - Optional all_on experiment that merges all enabled lever patches.
   - Validate:
     - Experiment IDs unique and slug-safe.
     - `run_settings_patch` keys only reference valid `RunSettings` fields (existing behavior).
     - `all_method_runtime_patch` only contains allowed runtime keys; reject unknown keys.
     - “all_on” merge has no conflicting keys (same key set to multiple different values); error with a clear message listing the conflicts.

2. Wire the resolved all-method runtime knobs into the actual run:

   - Modify `_run_single_experiment(...)` in `cookimport/bench/quality_runner.py` to accept resolved runtime values and pass them through to `cookimport.cli._run_all_method_benchmark_multi_source(...)`.

3. Add an example lever file under `data/golden/bench/quality/experiments/`:

   - Use a timestamped filename.
   - Include levers for the “priority 2–6” families (section detector backend, multi-recipe splitter, missing unit policy, instruction segmentation, p6 yield mode), each with `enabled: true/false`.
   - Include optional runtime lever examples (like `config_timeout_seconds` or `max_parallel_sources`) with `enabled: false` by default.

4. Update docs:

   - Add a short note to `docs/07-bench/07-bench_README.md` explaining schema v2 `levers` and showing the minimal command lines to run them.

## Validation and Acceptance

Acceptance is met when:

1. A schema-v2 experiments JSON containing `levers` runs without manual editing of “experiments[]” rows.
2. `experiments_resolved.json` includes generated experiments derived from levers.
3. Toggling a lever’s `enabled` flag changes the generated experiments deterministically (experiment count and IDs).
4. Providing an invalid `run_settings_patch` key fails fast with a clear error.
5. Providing an invalid `all_method_runtime_patch` key fails fast with a clear error.

## Idempotence and Recovery

Re-running `bench quality-run` is safe; it writes into a new timestamped run directory each time. If a run is interrupted, you can re-run the command; the cache may speed up subsequent runs.
