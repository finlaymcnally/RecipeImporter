How historical info is stored:
- The dashboard itself does not create long-term history records; it reads existing historical artifacts already on disk.
- Primary stage/import history source is `data/output/.history/performance_history.csv` (one row per imported file over time).
- If needed, it can derive stage history from per-run fallback reports: `data/output/<timestamp>/*.excel_import_report.json`.
- Benchmark history is read from prior eval artifacts under `data/golden/**/eval_report.json` (optionally enriched by sibling `coverage.json` and `manifest.json`).
- On each run, `cookimport stats-dashboard` writes a generated snapshot for viewing at `data/output/.history/dashboard/assets/dashboard_data.json` plus `index.html` and static assets.
- This means historical persistence lives in CSV/eval artifacts; the dashboard output is a read-only rendered snapshot of that history at generation time.

## Your SFAH (CUTDOWN) Re-Annotation Question

Short answer: yes, you can usually upgrade/re-annotate instead of starting over, with conditions.

Conditions for safe reuse:
- Same exact source bytes (same cutdown file content/hash).
- Same segmentation settings (`segment_blocks`, `segment_overlap`) used for that project.
- Project labeling config includes `YIELD_LINE` and `TIME_LINE`.

Important nuance in current code:
- Import reuses existing LS projects by title and does resume/idempotence by deterministic `segment_id`.
- The importer does **not** auto-update label config on an existing project.

So if your existing project predates the new labels:
1. Update that project’s labeling config in Label Studio to include `YIELD_LINE` and `TIME_LINE`, then continue annotating existing tasks.
2. Export again with `--export-scope freeform-spans`.

When to start a new project instead:
- If the file changed (new cutdown bytes),
- If you changed segmentation params,
- If the existing project is too messy to trust.

In those cases, treat as a new dataset version rather than a patch.
