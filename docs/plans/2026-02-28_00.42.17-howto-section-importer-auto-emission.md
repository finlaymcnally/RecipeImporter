# Implement Deterministic Importer-Side `HOWTO_SECTION` Emission (With Benchmark Parity)

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This repository includes `docs/PLANS.md` at the repository root, and this plan must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

Today the system can represent section headers in staging (`HowToSection` in JSON-LD instructions and ingredient section metadata), and Label Studio can now display/select `HOWTO_SECTION`. However, automatic prediction emission from imported books still does not output `HOWTO_SECTION`; section header lines are flattened into ingredient/instruction labels. After this change, importer-derived stage predictions will emit `HOWTO_SECTION` for true in-recipe subsection headers, and benchmark scoring will count those predictions correctly as ingredient-side or instruction-side structure using deterministic surrounding-context resolution.

A user-visible success case is: run `cookimport stage` on a recipe with subsection headers like `FOR THE SAUCE` or `TO SERVE`, open `.bench/<workbook>/stage_block_predictions.json`, and see `HOWTO_SECTION` on those header blocks. Then run stage-block/canonical benchmark evaluation and confirm accuracy does not regress because `HOWTO_SECTION` is resolved into the same structural class the benchmark expects.

- Decision: implement importer-side `HOWTO_SECTION` emission in the shared stage prediction generator (`cookimport/staging/stage_block_predictions.py`) instead of patching each importer independently.
  Rationale: all importer families converge into `RecipeCandidate` + archive blocks before stage prediction generation; one shared implementation keeps behavior consistent and avoids importer drift.

- Decision: treat `HOWTO_SECTION` as an emitted prediction label but resolve it to ingredient/instruction structural classes during scoring.
  Rationale: this preserves user-facing labeling expressiveness while keeping benchmark comparability against existing structural gold labels.
  Date/Author: 2026-02-28 / Codex GPT-5

- Decision: keep this feature deterministic and heuristic/rule-based only (no LLM parsing changes).
  Rationale: project policy explicitly avoids AI-based parsing/cleaning for import paths at this stage.
  Date/Author: 2026-02-28 / Codex GPT-5

## Context and Orientation

The relevant pipeline boundaries are:

- Importers (`cookimport/plugins/*`) create `RecipeCandidate` records and archive blocks.
- Staging writers convert that data into output artifacts.
- Stage prediction evidence (`cookimport/staging/stage_block_predictions.py`) creates one block label per archive block and writes `.bench/<workbook>/stage_block_predictions.json`.
- Bench evaluators (`cookimport/bench/eval_stage_blocks.py`, `cookimport/bench/eval_canonical_text.py`) score predictions against freeform/canonical gold.

Important terms in this repository:

- Section header: a line like `FOR THE SAUCE:` or `TO SERVE` that introduces a component subsection inside a recipe.
- `HOWTO_SECTION`: Label Studio freeform label for such subsection headers.
- Ingredient-side section: a section header whose surrounding structure is ingredient lines.
- Instruction-side section: a section header whose surrounding structure is instruction lines.

## Plan of Work

### Milestone 1: Emit `HOWTO_SECTION` deterministically from imported recipe evidence

Implement explicit section-header block detection in `cookimport/staging/stage_block_predictions.py`.

Use existing deterministic section utilities:

- `extract_ingredient_sections(recipe.ingredients)` for ingredient section headers.
- `extract_instruction_sections(<raw instruction text list>)` for instruction section headers.

Create a helper that gathers section header text candidates from both ingredient and instruction sides, dedupes them with existing text normalization, and maps them to archive block indices using current text-to-block matching helpers plus role-based fallback where needed.

Mark those indices as `HOWTO_SECTION`. Ensure title cleanup logic does not accidentally remove section labels, and update conflict resolution priority so section headers do not collapse back to ingredient/instruction labels when both are present on the same block.

Add a conservative guardrail to avoid promoting generic chapter headers (for example very broad all-caps single words not in-recipe context) to `HOWTO_SECTION`. Reuse existing recipe-range scoping and nearby recipe-local signals already computed in `_label_recipe_blocks`.

Milestone acceptance:

- A stage prediction artifact for a sectioned recipe contains `HOWTO_SECTION` entries on subsection-header blocks.
- Non-section recipe lines keep existing labels.

### Milestone 2: Keep benchmark scoring parity when predictions contain `HOWTO_SECTION`

Extend prediction-side remapping in evaluators so `HOWTO_SECTION` is interpreted structurally during scoring:

- In `cookimport/bench/eval_stage_blocks.py`, apply the resolver to prediction labels in `load_stage_block_labels` (or immediately after load) so blocks predicted as `HOWTO_SECTION` score as ingredient/instruction according to nearby context.
- In `cookimport/bench/eval_canonical_text.py`, apply equivalent resolver behavior to projected prediction line labels (`_build_pred_line_labels` output) before metrics calculation.

Preserve current behavior for non-`HOWTO_SECTION` labels.

Milestone acceptance:

- Stage-block and canonical-text benchmarks do not penalize `HOWTO_SECTION` predictions solely due to taxonomy expansion.
- Existing non-section benchmark cases remain unchanged.

### Milestone 3: Tests and contract updates

Add or update focused tests:

- `tests/staging/test_stage_block_predictions.py`:
  - section header blocks are emitted as `HOWTO_SECTION`;
  - conflicts resolve as intended when a header also looks ingredient/instruction-like.
- `tests/bench/test_eval_stage_blocks.py`:
  - predicted `HOWTO_SECTION` maps to ingredient/instruction expected labels in scoring.
- Canonical benchmark test file(s):
  - predicted line labels containing `HOWTO_SECTION` are mapped before `compute_block_metrics`.

Update docs if output/metric contract wording changed:

- staging reference for stage-block label semantics;
- benchmark reference for `HOWTO_SECTION` scoring resolution semantics.

Milestone acceptance:

- New tests fail before implementation and pass after.
- Existing benchmark/staging test subsets stay green.
