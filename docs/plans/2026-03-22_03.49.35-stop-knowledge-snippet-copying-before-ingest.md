# Stop Knowledge Snippet Copying Before Ingest and Worker Poisoning

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The recent knowledge-stage workspace fixes improved throughput, but they did not solve the remaining semantic failure: workers still copy source prose into `snippets[].body`, and the repository often notices only after the main attempt is already over. After this change, the knowledge stage should catch snippet-copy outputs at the same seam the worker helper already uses, attempt one narrow snippet-body-only repair before treating the task as a hard failure, and avoid poisoning a worker for this repairable error class until that narrow repair path has been exhausted.

The user-visible success condition is simple. A novice should be able to inspect the code and tests and see that a snippet-copy output like the March 22 benchmark failure no longer jumps straight from “worker wrote a file” to “invalid output, repair skipped because the worker was poisoned.” Instead, the same validator used by `tools/knowledge_worker.py check` should classify the output as a repairable snippet-copy near miss, run one bounded repair that rewrites only snippet bodies, and reserve poisoned-worker handling for failures that remain low-trust after that repair opportunity.

## Surprises & Discoveries

- Observation: the worker-local helper already has the right validator logic; the runtime just does not treat that seam as authoritative soon enough.
  Evidence: `cookimport/llm/knowledge_workspace_tools.py` routes `check_workspace_draft(...)` through `validate_knowledge_shard_output(...)`, and the mirrored `tools/knowledge_worker.py` already rejects `semantic_snippet_echoes_full_chunk`.

- Observation: the current repair path is too broad for this failure.
  Evidence: `_run_knowledge_repair_attempt(...)` in `cookimport/llm/codex_farm_knowledge_orchestrator.py` always prompts for a full shard repair with the whole previous invalid output, even when the only real problem is `snippets[].body`.

- Observation: poisoned-worker gating currently fires before snippet-copy tasks get a focused repair chance.
  Evidence: in `data/output/2026-03-22_03.33.50/.../knowledge/task_status.jsonl`, task `saltfatacidheatcutdown.ks0000.nr.task-047` ends with `validation_errors=["semantic_snippet_echoes_full_chunk"]` and `terminal_reason_code="repair_skipped_poisoned_worker"`.

- Observation: the March 22 candidate run improved throughput enough that absolute snippet-copy failures increased even while the rate among produced outputs improved.
  Evidence: the same benchmark tree records `validated=254`, `watchdog_killed=104`, and `semantic_snippet_echoes_full_chunk=52`. The earlier baseline had far fewer validated outputs but only `23` full-chunk snippet echoes because many workers died earlier.

- Observation: the representative bad output is not ambiguous; it is literal copied prose.
  Evidence: `data/output/2026-03-22_03.33.50/.../worker-001/out/saltfatacidheatcutdown.ks0000.nr.task-047.json` has a snippet body that begins `All salt comes from the ocean...`, and the cited evidence quote begins with the same text.

- Observation: a failed snippet-only repair exposed one more repo-side edge case in proposal promotion.
  Evidence: when every task inside a shard failed and the aggregated wrapper ended as `validation_errors=["missing_owned_chunk_results"]` with `payload: null`, `extract_promotable_knowledge_bundle(...)` still treated the wrapper as promotable-partial and raised on the missing payload. The implementation now returns `None` for that shape instead of crashing promotion.

## Context and Orientation

The knowledge stage is the optional LLM pass that reviews deterministic non-recipe chunks and keeps only reusable cooking knowledge. In this repository, a “task packet” is one worker-local JSON job under `knowledge/in/<task_id>.json`. The long-lived workspace worker reads the task, writes one semantic packet result to `knowledge/workers/<worker_id>/out/<task_id>.json`, and the repository later validates and promotes that result.

Three files matter most for this bug.

`cookimport/llm/codex_farm_knowledge_ingest.py` owns the authoritative semantic validator. It already rejects low-trust snippet shapes such as `semantic_snippet_echoes_full_chunk` and `semantic_snippet_copies_evidence_quote`.

`cookimport/llm/knowledge_workspace_tools.py` is the worker-local paved road. Its `check_workspace_draft(...)` function already calls the same validator, and the mirrored `tools/knowledge_worker.py` exposes that as `check` and `install`.

`cookimport/llm/codex_farm_knowledge_orchestrator.py` owns the live workspace-worker loop, the follow-up governor, the poisoned-worker policy, and the existing repair ladder. Today it can mark a worker as poisoned after repeated identical low-trust failures and then skip repair follow-ups with `repair_skipped_poisoned_worker`.

Two terms matter here.

A “snippet-copy failure” means the packet shape is otherwise coherent, but a snippet body is too close to the source material. In this codebase that currently means `semantic_snippet_echoes_full_chunk` or `semantic_snippet_copies_evidence_quote`.

A “near miss” means a result is invalid but still close enough to the contract that one bounded repair attempt is worth trying. Right now the code has a generic near-miss notion, but it does not give snippet-copy failures their own focused repair path.

The March 22 benchmark evidence that motivates this plan lives under `data/output/2026-03-22_03.33.50/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-22_03.34.33/raw/llm/saltfatacidheatcutdown/knowledge/`. In that run, the throughput fix clearly landed, but the stage still recorded `52` `semantic_snippet_echoes_full_chunk` failures and `69` `repair_skipped_poisoned_worker` terminal reasons. The representative task `saltfatacidheatcutdown.ks0000.nr.task-047` shows the exact bug: the snippet body literally restates the cited evidence quote, and the task never receives a snippet-only repair attempt.

## Plan of Work

### 1. Add one shared repairability classifier in the ingest layer

Edit `cookimport/llm/codex_farm_knowledge_ingest.py` so the repository has one small helper that answers a precise question: is this invalid packet a snippet-copy-only near miss, or is it a broader schema or ownership failure?

The helper should take `validation_errors` plus `validation_metadata` and return structured guidance, not just a boolean. It must distinguish at least these cases:

- snippet-copy-only near miss
- broader semantic invalidity
- schema or JSON-shape invalidity

For a packet to count as snippet-copy-only, the validator errors must be limited to `semantic_snippet_echoes_full_chunk` and/or `semantic_snippet_copies_evidence_quote`, and the packet must not also have ownership, schema, or response-shape failures such as `missing_owned_chunk_results`, `unexpected_chunk_results`, `schema_invalid`, `response_json_invalid`, or `response_not_json_object`.

Keep this helper in the ingest module so `knowledge_workspace_tools.py` and `codex_farm_knowledge_orchestrator.py` both depend on the same classification logic.

### 2. Make worker-local checking the contract the runtime is designed around

Do not try to hard-ban direct writes to `out/*.json`. Instead, make the bounded contract explicit and consistent everywhere the worker sees it.

Update the knowledge worker prompt in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, plus the generated `OUTPUT_CONTRACT.md` and `examples/` content in `cookimport/llm/knowledge_workspace_tools.py`, so the worker is told plainly:

- start from `current_task.json`
- scaffold only if helpful
- run `python3 tools/knowledge_worker.py check <draft>`
- use `install` only after `check` passes
- copied evidence text or long source paraphrase in `snippets[].body` is invalid even when the evidence quote itself is correct

This is not merely documentation polish. The helper examples should include one valid short extracted snippet and one invalid copied-evidence snippet built from real task-local data, so a novice can observe the rule locally inside a generated worker root.

### 3. Add a snippet-body-only repair ladder before the generic repair ladder

Edit `cookimport/llm/codex_farm_knowledge_orchestrator.py` to add a new repair mode that runs only when the shared ingest helper classifies the failure as snippet-copy-only.

This new path should:

- preserve the existing `packet_id`, `chunk_id`, `is_useful`, `block_decisions`, `reason_code`, and evidence pointers
- ask the model to rewrite only `snippets[*].body`
- keep the output schema the same semantic packet result shape the stage already accepts
- rerun the standard validator after the repair attempt

Use a dedicated prompt builder and runner function rather than overloading the current full-shard repair prompt. The prompt must explain that the previous output already had the right structure and evidence, and that the only allowed change is to shorten or rewrite snippet bodies so they become grounded extractions rather than copied source prose.

Only if this snippet-only repair path is not applicable, or if it fails, should the existing broader repair path remain eligible.

### 4. Stop poisoning workers for snippet-copy failures before that narrow repair is tried

Update the recovery governor in `cookimport/llm/codex_farm_knowledge_orchestrator.py` so repeated snippet-copy main-attempt failures do not immediately trip the same poisoned-worker handling used for malformed JSON, repeated boundary failures, or zero-output workers.

There are two acceptable implementations. Pick one and document it in the `Decision Log` when the code lands:

- do not record snippet-copy-only signatures toward worker poisoning until the snippet-only repair path has already failed for that worker
- or keep recording them, but exempt this failure family from `repair_skipped_poisoned_worker` for the first snippet-only repair attempt

Either implementation is fine as long as the observable behavior changes in the intended direction: a task whose only problem is copied snippet text must get one narrow repair chance before the worker is treated as too low-trust to salvage.

### 5. Keep the old generic repair path for true shape problems

Do not delete the existing full repair path. It still matters for malformed JSON, missing chunk rows, or broad schema drift. The change here is sequencing and specialization, not a total rewrite of the recovery ladder.

After the new snippet-only repair path is in place, revisit `_should_attempt_knowledge_repair(...)` so the old generic repair does not redundantly rerun on the same snippet-copy-only failure unless the new path has already failed and a second full repair is still justified. Prefer fewer recovery branches, not more.

### 6. Expand tests around the actual failure class

Add or update tests in these files.

In `tests/llm/test_knowledge_output_ingest.py`, add focused coverage for the new shared classifier, including one case where the only errors are snippet-copy errors and one case where a snippet-copy error is mixed with a schema or ownership error and must not count as a narrow near miss.

In `tests/llm/test_codex_farm_knowledge_orchestrator.py`, add a fake-runner regression that simulates a workspace worker writing a validly shaped packet whose snippet body copies the cited evidence. The expected behavior after the change is:

- the task is not immediately skipped as `repair_skipped_poisoned_worker`
- the snippet-only repair path is attempted first
- if the fake repair returns a shorter grounded snippet, the task becomes `repair_recovered` or equivalent validated recovery

Add a second orchestrator regression where the worker repeats the snippet-copy failure and the snippet-only repair also fails. That case should then be eligible for the normal low-trust accounting and should still be able to end as a hard failure.

Keep the existing poisoned-worker regression for malformed or uniformly low-trust non-repairable outputs. This change should narrow the poison policy for one failure family, not dissolve it.

## Validation and Acceptance

This plan is complete when all of the following are true.

The repository has one shared snippet-copy near-miss classifier in the ingest layer, and both the workspace helper and orchestrator use it instead of duplicating the logic.

The knowledge worker prompt and helper artifacts make `check` and `install` the explicit paved road and clearly show that copied evidence text in `snippets[].body` is invalid.

A focused orchestrator regression proves that a snippet-copy output no longer ends immediately with `repair_skipped_poisoned_worker` before the narrow repair path is tried.

A focused orchestrator regression proves that the snippet-only repair path can recover a copied-snippet output without changing `chunk_id`, `block_decisions`, or evidence pointers.

A focused orchestrator regression also proves that if snippet-only repair fails repeatedly, the system still falls back to hard failure and low-trust accounting rather than looping forever.

The narrow `.venv` test commands above pass.

If the user later approves a live rerun, an additional acceptance signal is that the March 22 benchmark class no longer shows snippet-copy tasks terminating as `repair_skipped_poisoned_worker`, and the absolute count of validated tasks should stay at least as good as the current candidate run.

## Idempotence and Recovery

This plan is additive and safe to repeat. The new classifier is a pure helper. Prompt and example updates are regenerated each run. The new snippet-only repair path is a bounded follow-up attempt that reuses the existing validator, so it should either validate or fail closed without changing the main input artifacts.

If implementation reveals that the new classifier disagrees with `validate_knowledge_shard_output(...)`, stop and collapse the logic back into the ingest module rather than carrying two similar rule sets. If the snippet-only repair prompt starts drifting block decisions or evidence, tighten the prompt and discard the broader output rather than silently accepting partial structural edits.
