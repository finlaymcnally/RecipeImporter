# Internalize Bucket 2 Run Settings Without Breaking Reproducibility

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this work, the product-facing run-settings surface should stop advertising parser-lab and benchmark-lab knobs as if they were normal choices for everyday staging or benchmarking. A normal operator should still choose things like extractor, OCR policy, Codex enablement, model overrides, and worker counts, but should not be asked about `ingredient_unit_canonicalizer`, `recipe_score_silver_min`, or `line_role_guardrail_mode` during ordinary use.

The important constraint is that Bucket 2 is not deletion work. These settings still affect real runtime behavior, benchmark comparison, and saved winner profiles. They must remain loadable from JSON, usable in internal benchmark patch paths, and preserved in full raw run-config payloads when explicitly set. The observable result is that `cookimport stage --help`, `cookimport labelstudio-benchmark --help`, the generated run-settings UI specs, CLI docs, run-config summaries, and analytics dashboard summaries all become smaller, while internal settings files and benchmark patches still work.

## Surprises & Discoveries

- Observation: the public/internal split already exists in code, but Bucket 2 has not actually been moved onto the internal side yet.
  Evidence: a direct check of `public_run_setting_names()` showed every Bucket 2 field still public and zero Bucket 2 fields in `internal_run_setting_names()`.

- Observation: `run_settings.py` is not the only user-facing surface to fix.
  Evidence: `cookimport/cli.py` still has handwritten Typer options and an older interactive settings editor that independently expose Bucket 2 controls such as `ocr_device`, `ocr_batch_size`, `multi_recipe_splitter`, `ingredient_text_fix_backend`, `recipe_scorer_backend`, `line_role_guardrail_mode`, and `codex_farm_failure_mode`.

- Observation: some Bucket 2 fields are already treated as part of reproducibility boundaries, not just UI noise.
  Evidence: `cookimport/config/prediction_identity.py` includes many Bucket 2 fields in `ALL_METHOD_PREDICTION_IDENTITY_FIELDS`, and all-method benchmark variants still emit dimensions such as `multi_recipe_splitter`, `p6_*`, and `pdf_column_gap_ratio`.

- Observation: public-facing summary behavior is not fully centralized yet.
  Evidence: `summarize_run_config_payload(...)` already supports public-only projection, but `cookimport/analytics/dashboard_collect.py` and `cookimport/analytics/perf_report.py` still hardcode summary keys like `ocr_device` and `ocr_batch_size`.

## Outcomes & Retrospective

This plan is not implemented yet. Its main contribution is narrowing the remaining run-settings cleanup into one concrete track: Bucket 1 already established the public/internal/retired vocabulary, while Bucket 2 is the still-open work of moving a large block of real-but-internal tuning controls out of the ordinary operator story without damaging reproducibility.

The main risk is accidental over-cleanup. If implementation treats Bucket 2 as “remove from runtime” instead of “remove from everyday surface,” benchmark identity and saved-profile workflows will regress. The plan below is deliberately written to keep those internal contracts explicit.

## Context and Orientation

`RunSettings` in this repository is the canonical typed settings model. It lives in `cookimport/config/run_settings.py` and feeds interactive settings selection, stage execution, benchmark execution, winner-profile persistence, analytics summaries, and benchmark identity hashing. A run setting can therefore show up in four distinct places:

1. as a typed field on `RunSettings`,
2. as a user-visible setting in UI/help/docs,
3. as a persisted key in raw `run_config` payloads,
4. as an internal behavior/identity input for benchmark reuse or cache boundaries.

Bucket 2 from the March 13 audit is the set of settings that should still exist in categories 1, 3, and 4, but should stop appearing in category 2. The full Bucket 2 list is:

- Multi-recipe controls: `multi_recipe_splitter`, `multi_recipe_min_ingredient_lines`, `multi_recipe_min_instruction_lines`, `multi_recipe_for_the_guardrail`
- Ingredient parser controls: `ingredient_text_fix_backend`, `ingredient_pre_normalize_mode`, `ingredient_packaging_mode`, `ingredient_parser_backend`, `ingredient_unit_canonicalizer`, `ingredient_missing_unit_policy`
- Priority 6 controls: `p6_time_backend`, `p6_time_total_strategy`, `p6_temperature_backend`, `p6_temperature_unit_backend`, `p6_ovenlike_mode`, `p6_yield_mode`
- Scoring controls: `recipe_scorer_backend`, `recipe_score_gold_min`, `recipe_score_silver_min`, `recipe_score_bronze_min`, `recipe_score_min_ingredient_lines`, `recipe_score_min_instruction_lines`
- OCR/PDF lab controls: `ocr_device`, `ocr_batch_size`, `pdf_column_gap_ratio`
- LLM guardrail/failure controls: `line_role_guardrail_mode`, `codex_farm_failure_mode`

Several files matter for the implementation.

- `cookimport/config/run_settings.py` defines the fields, the public/internal surface metadata, the UI spec generation, and the public summary projection helpers.
- `cookimport/config/run_settings_adapters.py` maps `RunSettings` into stage and benchmark call kwargs. This file should keep forwarding Bucket 2 settings because runtime behavior still depends on them.
- `cookimport/cli.py` defines the public Typer entrypoints for `stage`, `labelstudio-benchmark`, `bench speed-run`, and some older interactive edit menus that still expose hand-picked settings directly.
- `cookimport/config/prediction_identity.py` defines which settings participate in all-method reuse/cache identity. Many Bucket 2 fields are intentionally present there.
- `cookimport/analytics/dashboard_collect.py` and `cookimport/analytics/perf_report.py` still construct run-config summaries by hand for some analytics flows instead of always using the shared public-summary helper.
- `docs/02-cli/02-cli_README.md` and `docs/07-bench/07-bench_README.md` still describe many Bucket 2 settings as if they were normal operator-facing options.

The key rule for this plan is simple: hiding a setting from the everyday surface must not change whether the runtime can still consume that setting from an explicit internal payload.

## Plan of Work

### Milestone 1: Reclassify Bucket 2 In `RunSettings`

At the end of this milestone, the source of truth in `cookimport/config/run_settings.py` should explicitly say that every Bucket 2 field is internal-only. `public_run_setting_names()` and `run_settings_ui_specs()` should stop treating them as ordinary editable settings, while `RunSettings.from_dict(...)`, `to_run_config_dict()`, and the adapters keep working.

Implement this by updating the `json_schema_extra` metadata for every Bucket 2 field to set `surface=RUN_SETTING_SURFACE_INTERNAL`. Do not remove the fields. Do not move them into the retired-key registry. Do not stop serializing them in `to_run_config_dict()`.

Add or update tests in `tests/llm/test_run_settings.py` so the assertions explicitly cover:

- Bucket 2 fields are absent from `public_run_setting_names()`.
- Bucket 2 fields are present in `internal_run_setting_names()`.
- `run_settings_ui_specs()` excludes them by default.
- `run_settings_ui_specs(include_internal=True)` still exposes them for internal tooling.

The current UI-spec coverage test assumes every public field is editable; that part stays true. What changes is the public list itself.

### Milestone 2: Remove Bucket 2 From Ordinary Help And Interactive Editing

At the end of this milestone, the normal public command help for `stage` and `labelstudio-benchmark` should no longer advertise Bucket 2 settings. Older internal or scripted workflows should still have a way to set them explicitly.

Implement this in `cookimport/cli.py`. For Bucket 2 options on the public Typer commands, keep one of two shapes only:

- hidden Typer options on the same command when backward CLI compatibility is valuable, or
- no public flag at all when an existing settings-file or internal benchmark patch path already covers the use case.

For this repo, hiding the existing Typer options is the preferred first move because it preserves old shell history and tests more safely than deleting dozens of parameters at once. Use `hidden=True` on the Bucket 2 options for `stage` and `labelstudio-benchmark` unless a specific option should be fully removed for a stronger reason.

Update the older interactive edit menu logic in `cookimport/cli.py` so it no longer lists Bucket 2 controls in the ordinary manual settings editor. If that editor needs an internal mode, make it explicit and opt-in; otherwise keep it public-only.

Do not change the runtime normalization and forwarding logic after the Typer parsing step. The same normalized values should still reach `build_run_settings(...)`, `RunSettings(...)`, and the stage/benchmark execution helpers.

Acceptance for this milestone is user-visible:

- `cookimport stage --help` should not show Bucket 2 flags.
- `cookimport labelstudio-benchmark --help` should not show Bucket 2 flags.
- The ordinary interactive settings editor should not list Bucket 2 items.

### Milestone 3: Keep Internal Patchability And Reproducibility Intact

At the end of this milestone, internal workflows should still be able to set Bucket 2 fields intentionally, and explicit changes to those settings should still affect the reproducibility boundaries that depend on them.

Keep the adapter layer in `cookimport/config/run_settings_adapters.py` unchanged except where tests or naming cleanup require refactors. Stage and benchmark runtime kwargs still need these values.

Keep `cookimport/config/prediction_identity.py` behavior-driven. If a Bucket 2 field changes prediction output or reuse safety, it must remain in the identity payload. The plan is not to prune that file down to public settings. Instead, document the distinction in code comments if helpful: public surface is about human visibility, not cache safety.

Keep QualitySuite and all-method benchmark override paths working:

- `bench speed-run --run-settings-file ...` must still accept Bucket 2 keys in the JSON.
- QualitySuite `run_settings_patch` payloads must still accept Bucket 2 keys.
- All-method deterministic sweeps may continue to vary Bucket 2 knobs such as `multi_recipe_splitter`, `p6_*`, and `pdf_column_gap_ratio`.

If there is any validation code that mistakenly starts rejecting internal-only keys because they are no longer public, fix that validation so it keys off `RunSettings.model_fields` or an explicit “allowed internal patch keys” set instead of `public_run_setting_names()`.

### Milestone 4: Make Human-Facing Summaries And Docs Public-Only

At the end of this milestone, run summaries, dashboard summaries, and the main CLI docs should describe the smaller public surface instead of re-advertising Bucket 2 choices.

First, route analytics summary builders onto the shared projection helpers:

- `cookimport/analytics/dashboard_collect.py`
- `cookimport/analytics/perf_report.py`

These files currently hand-pick summary keys like `ocr_device` and `ocr_batch_size`. Replace those ad hoc key lists with calls to `summarize_run_config_payload(..., include_internal=False)` or an equivalent shared helper, while keeping raw JSON fields available elsewhere for drill-down and reproduction.

Second, update the user-facing docs:

- `docs/02-cli/02-cli_README.md`
- `docs/07-bench/07-bench_README.md`

Rewrite the run-settings descriptions so Bucket 2 settings are described as internal engineering controls or benchmark patch inputs, not as ordinary knobs a user is expected to choose during normal staging and benchmark runs. The docs should still mention that these keys remain accepted in internal settings payloads where that matters, especially for QualitySuite and saved winner settings.

Third, review any remaining summary strings in `cookimport/labelstudio/ingest.py`, dashboard renderers, or benchmark report writers that bypass the shared run-settings summary helper. The goal is one public summary story and one full raw payload story, not many partially overlapping custom summaries.

### Milestone 5: Validation And Compatibility Repair

At the end of this milestone, the repo should prove three things:

1. Bucket 2 is gone from ordinary public surfaces.
2. Bucket 2 still loads and persists when set explicitly through internal paths.
3. Benchmark identity and reuse logic still respect Bucket 2 behavior changes.

Update tests in these areas first:

- `tests/llm/test_run_settings.py`
- `tests/cli/test_run_settings_adapters.py`
- `tests/staging/test_run_manifest_parity.py`
- `tests/analytics/test_stats_dashboard.py`
- the benchmark helper tests that assert dimensions or payload forwarding for `multi_recipe_splitter`, `p6_*`, `ocr_device`, `pdf_column_gap_ratio`, `line_role_guardrail_mode`, or `codex_farm_failure_mode`

Add focused behavioral assertions for:

- help-text absence: `stage --help` and `labelstudio-benchmark --help` do not show Bucket 2 flags,
- internal acceptance: `RunSettings.from_dict(...)` still accepts Bucket 2 keys and serializes them in full `to_run_config_dict()`,
- public projection: `to_public_run_config_dict()` and `summarize_run_config_payload(..., include_internal=False)` omit Bucket 2 keys,
- benchmark identity: changing a Bucket 2 field that affects prediction shape still changes `build_all_method_prediction_identity_payload(...)`,
- benchmark patches: a QualitySuite or speed-suite settings file that includes Bucket 2 keys still normalizes and runs.

Do not call the plan complete until both the help-text checks and the internal-patchability checks pass.

## Validation and Acceptance

The implementation is acceptable only if all of the following are true:

- `run_settings_ui_specs()` excludes all Bucket 2 fields by default, while `run_settings_ui_specs(include_internal=True)` still includes them.
- `public_run_setting_names()` contains no Bucket 2 fields.
- `cookimport stage --help` and `cookimport labelstudio-benchmark --help` do not advertise Bucket 2 flags.
- The ordinary interactive settings editor no longer lists Bucket 2 controls.
- `RunSettings.from_dict(...)` still loads Bucket 2 keys from explicit payloads.
- `RunSettings.to_run_config_dict()` still serializes explicitly set Bucket 2 keys.
- `RunSettings.to_public_run_config_dict()` and public summary strings omit Bucket 2 keys.
- Analytics/dashboard summaries stop surfacing Bucket 2 keys by default.
- Explicit Bucket 2 changes that alter runtime behavior still affect prediction identity and benchmark dimensions where appropriate.

This is a surface-cleanup plan, not a behavior-flattening plan. If the result hides these settings but also stops the runtime from honoring them when explicitly set, the implementation failed.

## Idempotence and Recovery

Most steps in this plan are safe to repeat. Re-marking a field as internal or re-running the test suite is idempotent.

The risky part is CLI flag cleanup, because removing or hiding a flag can silently break tests or older scripts. Favor `hidden=True` first for stage and benchmark Typer options so the command-line parser still accepts old invocations while public help shrinks immediately. Only fully delete a flag after all affected tests and any repo-local scripted callers are updated.

If a summary or analytics change accidentally removes information needed for archaeology, keep the full raw `run_config_json` or equivalent debug payload and only narrow the default summary projection.

## Interfaces and Dependencies

The implementation should leave these interfaces true at the end:

- `cookimport.config.run_settings.run_setting_surface(field_name: str) -> str` returns `internal` for every Bucket 2 field.
- `cookimport.config.run_settings.public_run_setting_names() -> tuple[str, ...]` excludes Bucket 2.
- `cookimport.config.run_settings.internal_run_setting_names() -> tuple[str, ...]` includes Bucket 2.
- `cookimport.config.run_settings.run_settings_ui_specs(include_internal: bool = False)` returns public-only specs by default and can still expose internal specs when explicitly requested.
- `cookimport.config.run_settings.RunSettings.to_run_config_dict()` still returns full persistable payloads.
- `cookimport.config.run_settings.RunSettings.to_public_run_config_dict()` returns the public projection.
- `cookimport.config.run_settings_adapters.build_stage_call_kwargs_from_run_settings(...)` and `build_benchmark_call_kwargs_from_run_settings(...)` still forward Bucket 2 values to runtime execution.
- `cookimport.config.prediction_identity.build_all_method_prediction_identity_payload(...)` continues to include behaviorally relevant Bucket 2 settings.
