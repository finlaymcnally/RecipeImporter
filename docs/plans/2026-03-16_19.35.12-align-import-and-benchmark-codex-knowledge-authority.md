# Align Import And Benchmark Around Codex Knowledge Authority

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a run with Codex enabled should mean one simple thing: Codex work can improve the final processed book, and benchmark scores that same final processed result. A user running an import or a benchmark with `codex-line-role-v1` and `codex-farm-knowledge-v1` should no longer pay for Codex knowledge calls that only produce snippets while the scored labels come from an earlier deterministic shortcut. The visible proof is that a Codex-backed run writes final non-recipe authority artifacts that can differ from the deterministic seed, the scored benchmark artifacts are derived from those final outputs, and the reporting files tell the truth about which stages actually decided the labels.

Today the repository has a split contract. Codex can influence early line labels, but `cookimport/staging/nonrecipe_stage.py` turns those labels into authoritative `knowledge` versus `other` spans before the optional knowledge harvest runs. The later knowledge harvest in `cookimport/llm/codex_farm_knowledge_orchestrator.py` only extracts snippets from already-selected `knowledge` spans. The benchmark path then compounds the problem by writing scored `line-role-pipeline/` artifacts from authoritative label reuse in `cookimport/labelstudio/ingest.py`, even when Codex-backed benchmark surfaces are enabled. This plan removes that split so import and benchmark are both measuring the same product behavior.

## Outcomes & Retrospective

The repository now has the intended single contract in code for local stage and benchmark artifact generation: Stage 7 writes a deterministic seed non-recipe authority, the optional knowledge stage can refine final `knowledge` versus `other`, and both stage-block predictions and the authoritative canonical projection read that final authority.

Import and benchmark should both operate on the same final label authority. When Codex knowledge is off, that authority is the deterministic Stage 7 classification. When Codex knowledge is on, that authority is the refined post-knowledge result. The user-visible benchmark artifacts, run summaries, prompt budgets, and telemetry should all explain that contract clearly and consistently.

The remaining gap is the real-book validation step. I have not yet run the new live `saltfatacidheatCUTDOWN` benchmark, so the plan is not complete until a fresh benchmark artifact set proves either that `KNOWLEDGE` recall improved or that any remaining weakness is now clearly attributable to scored Codex decisions rather than ignored Codex work.

The primary risk is still overshooting into an architecture that lets Codex rewrite too much of the non-recipe surface without clear diagnostics. The implemented shape addresses that by preserving both seed and final artifacts, adding explicit refinement reports, and keeping deterministic runs on the seed-only path.

## Context and Orientation

This repository has two closely related but currently diverged flows.

The import flow lives in `cookimport/staging/import_session.py`. It builds authoritative line labels in `cookimport/parsing/label_source_of_truth.py`, groups recipe spans, optionally runs recipe Codex correction, then builds Stage 7 non-recipe ownership in `cookimport/staging/nonrecipe_stage.py`. After that, if `llm_knowledge_pipeline` is enabled, it runs `cookimport/llm/codex_farm_knowledge_orchestrator.py::run_codex_farm_knowledge_harvest(...)`. The output writer in `cookimport/staging/writer.py` then writes `08_nonrecipe_spans.json`, `09_knowledge_outputs.json`, final drafts, and benchmark-ready stage-block predictions.

The benchmark prediction-generation flow lives largely in `cookimport/labelstudio/ingest.py`. It can build stage-backed outputs and benchmark artifacts offline, including `line-role-pipeline/` artifacts used by canonical-text evaluation. Today that file has a branch that writes `line-role-pipeline/stage_block_predictions.json` by directly reusing authoritative Stage 2 line labels. That branch writes a telemetry file saying `authoritative_reuse`. This is the main reason benchmark can stop reflecting the live Codex path even when Codex-backed settings are selected.

Several terms need to be defined precisely for this plan.

A “seed” label is the first authoritative non-recipe ownership result produced before the knowledge harvest. In the current code, that is the output of `build_nonrecipe_stage_result(...)` in `cookimport/staging/nonrecipe_stage.py`.

A “final” label is the label that the user should trust as the best processed-book answer after every enabled stage has run. Right now the repository does not have a separate final non-recipe authority object; it only has the seed plus snippet outputs. This plan adds that missing final layer.

A “refinement” is a controlled change from the seed non-recipe ownership to the final non-recipe ownership. In this repository, refinement means changing `knowledge` versus `other` for blocks or spans outside recipe spans, with explicit diagnostics about what changed and why.

A “scored artifact” is the prediction artifact benchmark actually evaluates. For stage-backed evaluation that means `.bench/<workbook_slug>/stage_block_predictions.json`. For canonical-text runs that means the pointer pair published in the benchmark run manifest, currently aimed at `line-role-pipeline/stage_block_predictions.json` and its matching extracted archive.

The current conflict is simple. Import and benchmark both build early labels through `build_label_first_stage_result(...)`, but after that they diverge in a way that hides Codex work. Import freezes Stage 7 before knowledge harvest and benchmark often scores authoritative reuse instead of final stage-backed outputs. The user wants one product process: enable Codex when desired, let it improve the book, then score that exact result.

The March 16 benchmark run under `data/golden/benchmark-vs-golden/2026-03-16_18.10.25/` is the clearest demonstration of the bug. `prompt_budget_summary.json` reports `424619` knowledge-stage tokens, `knowledge_manifest.json` reports `56` knowledge jobs, `line-role-pipeline/telemetry_summary.json` says `authoritative_reuse`, and the scored knowledge total is almost unchanged from vanilla. That is the architectural mismatch this plan resolves.

## Plan of Work

Start by introducing an explicit two-step non-recipe authority contract. The current `NonRecipeStageResult` in `cookimport/staging/nonrecipe_stage.py` should stay as the deterministic seed result, because it is still valuable when Codex knowledge is off and still useful as a debug baseline when Codex knowledge is on. Add a second result shape for final authority, either in the same module or in a nearby module such as `cookimport/staging/nonrecipe_refinement.py`. The final shape must carry the same information needed by writers and scorers: final spans, `knowledge_spans`, `other_spans`, `block_category_by_index`, and diagnostics describing changes from the seed. Keep the seed and final results separate so the repository can explain what Codex changed instead of silently overwriting the deterministic baseline.

Once that distinction exists, change the knowledge stage from “extract snippets from seed knowledge spans” into “refine seed ownership and extract snippets from the refined knowledge set.” The easiest safe way to do this is to extend the knowledge job builder and knowledge output schema. Instead of only feeding `knowledge_spans` into `build_knowledge_jobs(...)`, the builder should accept a refinement candidate surface derived from all outside-recipe spans. The practical scope must remain bounded so token cost stays reviewable. The recommended rule is: always include seed `knowledge` spans, include outside-recipe `other` spans that look knowledge-like or ambiguous based on existing line-role cues, and skip obvious noise spans. The Codex output schema for each knowledge job must grow from snippet-only extraction into a combined refinement-and-extraction contract. Each output row should be able to say “this chunk remains knowledge,” “this chunk should be other,” or “this chunk contains both and should split here,” along with the snippet payload if knowledge remains. The plan does not require Codex to relabel recipe-local text; recipe spans remain exclusion zones.

After that, teach the import flow to build final authority from the knowledge refinement outputs. `cookimport/staging/import_session.py` should still build the deterministic seed first and write a seed artifact, but when `llm_knowledge_pipeline` is on it must merge the knowledge outputs back into a final non-recipe authority object before any scored artifact is written. That merge step should be explicit and testable. It should preserve deterministic coverage for untouched spans, apply Codex refinement only to the outside-recipe spans presented to the knowledge stage, and produce a compact refinement report that lists changed blocks or changed spans. The final object, not the seed, must then drive:

the machine-readable non-recipe authority artifact,
the reviewer-facing knowledge outputs summary,
the stage-backed `stage_block_predictions.json`,
and any benchmark or upload-bundle projections built from the stage run.

The artifact contract must be updated carefully. `08_nonrecipe_spans.json` currently reads like the only authoritative non-recipe artifact. Keep that filename as the final truth file to avoid proliferating new top-level authority names, but add a sibling seed artifact such as `07_nonrecipe_seed_spans.json` or a `seed` section inside `08_nonrecipe_spans.json`. The important rule is that a novice opening the run root can tell which file is the initial deterministic seed and which file is the final truth used for downstream scoring. `09_knowledge_outputs.json` should also become honest about its role: when Codex knowledge is enabled, it should summarize refinement decisions and extraction outputs, not merely snippets written.

Next, align the benchmark path to use the same final truth. The canonical problem area is `cookimport/labelstudio/ingest.py::_write_authoritative_line_role_artifacts(...)`. That helper currently writes scored canonical artifacts from direct label reuse, even in Codex-backed benchmark flows. Replace that shortcut with one of two behaviors, resolved in code and in docs rather than at runtime by guesswork.

For deterministic benchmark runs, authoritative reuse remains valid and telemetry should say so plainly.

For Codex-backed benchmark runs that are supposed to represent the import process, the scored canonical artifact must be projected from the same final stage-backed authority that import writes after knowledge refinement. That can mean rebuilding the canonical line projections from the final stage run, or simply using the existing stage-backed pointer pair if the canonical scorer can operate on it correctly. The important part is that benchmark must stop fabricating a special “authoritative reuse” scored artifact when the user asked for a live Codex-backed candidate.

After aligning the actual scored artifact, fix the reporting layer. `line-role-pipeline/telemetry_summary.json` must describe the true mode that produced the scored labels. `prompt_eval_alignment.md` must stop claiming Codex line-role predictions were the scored thing when the run actually used authoritative reuse. `prompt_budget_summary.json` should continue to report tokens by stage, but the benchmark summary and comparison writers must add a note about whether the knowledge stage was authoritative, enrichment-only, or bypassed. The goal is that a human comparing two runs can immediately tell whether knowledge tokens had any chance to affect score. If a run spends knowledge tokens while `scored_effect = none`, that should be treated as an error state or at least a conspicuous warning.

Once runtime and reporting align, update docs and tests together. `docs/10-llm/10-llm_README.md`, `docs/07-bench/07-bench_README.md`, `docs/05-staging/05-staging_readme.md`, and `docs/06-label-studio/06-label-studio_README.md` must all describe the same truth: Codex knowledge is authoritative when enabled and benchmark scores the resulting final processed output. The older Phase 4 plan should not be rewritten as if it never existed, but the current docs must stop teaching “knowledge stage is only optional snippet extraction” as the live product contract if this change lands.

## Milestones

### Milestone 1: Separate seed non-recipe authority from final non-recipe authority

At the end of this milestone, the import path should have a clear in-memory representation of two states: the deterministic seed ownership and the final ownership that later stages may refine. Nothing user-visible needs to change yet, but the code must stop pretending the Stage 7 seed is automatically the final truth in all modes.

This work belongs mainly in `cookimport/staging/nonrecipe_stage.py`, `cookimport/staging/import_session.py`, and the writers that currently serialize `08_nonrecipe_spans.json` and `09_knowledge_outputs.json`. The acceptance proof is a focused test showing a run with knowledge off writes identical seed and final ownership, while the internal API exposes both explicitly.

### Milestone 2: Make Codex knowledge refine authority, not just emit snippets

At the end of this milestone, enabling `llm_knowledge_pipeline=codex-farm-knowledge-v1` must give Codex a way to change final outside-recipe `knowledge` versus `other` ownership, with explicit diagnostics. The knowledge harvest still writes snippets, but snippet writing is no longer the only value of the stage.

This work belongs mainly in `cookimport/llm/codex_farm_knowledge_jobs.py`, `cookimport/llm/codex_farm_knowledge_contracts.py`, `cookimport/llm/codex_farm_knowledge_ingest.py`, `cookimport/llm/codex_farm_knowledge_orchestrator.py`, and the new refinement-merge seam in staging. Acceptance is a test where a seed `other` span with strong explanatory-prose cues becomes final `knowledge` only when the mocked Codex refinement output says so.

### Milestone 3: Make import outputs and benchmark scored artifacts use final authority

At the end of this milestone, a Codex-backed run’s final stage-block predictions and canonical benchmark artifacts must be projections of the post-refinement import result, not authoritative reuse shortcuts. Deterministic runs may still use authoritative reuse, but Codex-backed runs must no longer bypass the process they are supposed to measure.

This work belongs mainly in `cookimport/staging/writer.py`, `cookimport/staging/stage_block_predictions.py`, and `cookimport/labelstudio/ingest.py`. Acceptance is a benchmark test or fixture-backed replay showing that a Codex-backed candidate with changed knowledge refinement produces changed scored labels, while the deterministic control still uses the simpler reuse path.

### Milestone 4: Fix benchmark and prompt reporting mismatch

At the end of this milestone, the benchmark artifacts should be internally consistent. Telemetry, prompt/eval alignment, and prompt budgets should all agree about whether the scored artifact came from deterministic reuse, live Codex line-role labels, seed-plus-knowledge refinement, or another explicitly named mode.

This work belongs mainly in `cookimport/labelstudio/ingest.py`, benchmark summary/render helpers, and any telemetry writers under `cookimport/parsing/canonical_line_roles.py` or the benchmark layer that currently emit misleading mode labels. Acceptance is a fresh benchmark run where the artifact text a human reads matches the actual scored pointer pair and runtime mode.

### Milestone 5: Validate on the real benchmark seam and update docs

At the end of this milestone, the repository should have one demonstrable real-world proof that the new contract works on the book that exposed the issue. A fresh Codex-backed benchmark on `saltfatacidheatCUTDOWN` should either show improved `KNOWLEDGE` recall or at minimum show that any remaining weakness comes from Codex making poor refinement decisions, not from Codex work being ignored. The docs must describe the new contract plainly.

This work belongs in the docs listed above plus whichever tests or benchmark wrappers are needed to make the real-input validation easy to repeat. Acceptance is a new benchmark artifact set under `data/golden/benchmark-vs-golden/<timestamp>/` and updated docs explaining what changed.

## Validation and Acceptance

Acceptance is behavioral and must prove that import and benchmark are scoring the same product path.

First, a unit or small integration test must prove that the repository now exposes both seed non-recipe authority and final non-recipe authority, and that they are identical when `llm_knowledge_pipeline=off`.

Second, a knowledge-stage test must prove that a mocked Codex refinement output can change final non-recipe ownership outside recipe spans. The change must be visible in the final authority artifact, not only in snippet records.

Third, a stage-block prediction test must prove that final stage-backed predictions use the refined non-recipe authority when knowledge Codex is enabled. The old behavior, where predictions always reflect the deterministic seed, must fail before the change and pass after it.

Fourth, a benchmark artifact test or replay fixture must prove that Codex-backed benchmark runs stop writing scored canonical artifacts from authoritative reuse when the user selected a live Codex-backed candidate. The test should explicitly check the scoring pointer pair and the reported telemetry mode.

Fifth, a real benchmark run over `saltfatacidheatCUTDOWN` must show that one of two things is true, with no third ambiguous state:

Codex knowledge improved the scored output compared with the deterministic control, or
Codex knowledge did not improve the score, but the artifacts clearly show that Codex refinement was scored and simply made poor decisions.

The forbidden state after this plan is the current one: Codex knowledge spends substantial tokens while benchmark silently scores a deterministic-reuse artifact instead.

## Idempotence and Recovery

The plan is safe to implement incrementally. The seed-versus-final split is additive at first, which means the deterministic path can keep working while the refinement path is introduced. If the first refinement merge is too aggressive and causes regressions, keep the seed artifact and use a feature-gated or setting-gated path to narrow refinement eligibility while preserving the new tests and reporting.

When repeating benchmark validations, always write to a fresh timestamped output root and compare old versus new artifacts instead of deleting old runs. If a live Codex run is too expensive to repeat during a debug loop, freeze the current failing run and write fixture-backed tests that replay its structural conditions until the runtime logic stabilizes.

If the reporting fixes land before the full runtime fix, that is acceptable only if the reports explicitly surface the mismatch rather than hiding it. Do not ship a new report that still implies Codex knowledge affected score when it did not.
