# Normalize Line-Role Recovered Watchdog Outcomes

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The benchmark run at `data/golden/benchmark-vs-golden/2026-03-21_16.10.40/single-book-benchmark/saltfatacidheatcutdown/codexfarm/` proves that the recent line-role stabilization change did not fully solve the saved-run contradiction. Some line-role shards now validate only after a task-level watchdog retry recovers missing packet outputs, but the shard-level `status.json` still says `state == "watchdog_killed"` because it copies the main workspace session’s raw supervision fields after recovery has already succeeded. The benchmark summary then counts those shards as watchdog kills even though they finished with valid output.

After this plan is implemented, a line-role shard that validates after a watchdog retry or repair must be finalized as a completed shard, not as a watchdog-killed shard. The raw watchdog attempt must remain visible for debugging, but the final shard contract and the benchmark summary must reflect the final outcome that operators care about. A saved run should no longer show `status: "validated"` together with `state: "watchdog_killed"`, and the line-role summary should distinguish unrecovered watchdog kills from recovered watchdog incidents.

This plan does not broaden command policy. If a line-role shard still dies from an unrecovered forbidden command, it must remain a real watchdog failure. The goal here is to stop mislabeling recovered shards, not to hide genuine worker failures.

  Rationale: operators and benchmark summaries care whether the shard ultimately succeeded, failed, or remained unresolved. The main workspace session’s raw watchdog state is still important evidence, but it is attempt-level evidence, not the final shard contract.

## Outcomes & Retrospective

This plan is not implemented yet. The investigation outcome is still useful: the earlier stabilization fix did land, but the saved run exposed a different authority bug. The line-role stage currently has two independent failure classes in the same benchmark slice: recovered shards that are finalized with the wrong state, and unrecovered forbidden-command kills that remain real failures. This plan addresses the first class directly and uses tests to make sure the second class stays visible.

## Context and Orientation

The line-role stage breaks a book into shard-sized canonical labeling tasks. In this repository, a "shard" is the unit written to `line-role-pipeline/runtime/line_role/workers/<worker-id>/shards/<shard-id>/`. A "workspace worker" is a long-lived Codex session that is allowed to inspect files in an assigned worker workspace and write packet outputs under `out/`. A "task-level watchdog retry" is a follow-up single-task recovery attempt that runs only when the main workspace worker was killed by the watchdog but the task is considered retryable. The retry can produce a valid packet even when the main session died.

The core implementation lives in `cookimport/parsing/canonical_line_roles.py`. The function `_run_line_role_workspace_worker_assignment_v1(...)` launches the workspace worker, collects task-level retry and repair results, aggregates those packet outputs into one shard payload, validates the payload, and then writes `proposal.json`, `status.json`, worker failure rows, and runner telemetry. The generic summary helper lives in `cookimport/llm/codex_exec_runner.py::summarize_direct_telemetry_rows(...)`. The line-role stage currently delegates to that helper through `_summarize_direct_rows(...)`.

The saved runtime artifacts for the failure case live under:

    data/output/2026-03-21_16.10.40/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-21_16.11.12/line-role-pipeline/runtime/line_role/

The benchmark-facing summary artifacts live under:

    data/golden/benchmark-vs-golden/2026-03-21_16.10.40/single-book-benchmark/saltfatacidheatcutdown/codexfarm/prompts/line-role-pipeline/

The key bug is in the gap between task aggregation and shard finalization. The code already knows whether any task-level watchdog retry recovered the shard, but it still writes shard final state from `session_run_result`, which represents the main workspace session only. That is why a shard can be `validated` and still look `watchdog_killed`.

## Plan of Work

Start in `cookimport/parsing/canonical_line_roles.py` inside `_run_line_role_workspace_worker_assignment_v1(...)`. Extract the shard-finalization rules into a small helper, for example `_normalize_line_role_shard_outcome(...)`, that takes the raw `session_run_result`, `proposal_status`, `watchdog_retry_status`, `repair_status`, and the existing resume/no-task cases. The helper must return the final shard state, final reason code, final reason detail, final retryable flag, and a short `finalization_path` string that explains which path produced the final outcome.

The normalization rules should be explicit and minimal. If the shard validated and a watchdog retry recovered at least one task, finalize the shard as `completed`, set `finalization_path` to something like `watchdog_retry_recovered`, set `reason_code` to the same recovery-oriented value, and mark `retryable` as false because the shard already succeeded. If the shard validated after repair, finalize it as `completed` with a repair-oriented `finalization_path`. If the shard validated without recovery, preserve the existing completed or resume/no-task semantics. Only leave `state == "watchdog_killed"` when the final shard outcome is still unresolved.

Do not throw away the raw workspace session evidence. When the normalized final state differs from the raw `session_run_result`, write the raw attempt fields alongside the final fields in `status.json` and any shard-level runner payload that leaves the function. Use stable names such as `raw_supervision_state`, `raw_supervision_reason_code`, `raw_supervision_reason_detail`, and `raw_supervision_retryable`. Include the same raw-versus-final split in worker failure rows only when the shard still failed; recovered shards should not appear in the worker-failure list at all.

After shard finalization is normalized, update the summary path. The simplest safe approach is to teach `summarize_direct_telemetry_rows(...)` to prefer optional final-outcome fields when a row provides them, while preserving current behavior for rows that do not. Add support for `final_supervision_state` and `finalization_path` on telemetry rows. Then count `watchdog_killed_shard_count` from the effective final supervision state, not always from the raw attempt state. Add a new `watchdog_recovered_shard_count` that counts shards whose `finalization_path` indicates recovery from a watchdog kill. Leave the raw attempt rows intact so debugging still shows that a watchdog incident happened.

Wire the new fields from line-role into the rows consumed by `_summarize_direct_rows(...)`. The shard-level primary row for a recovered shard should carry both the raw and final supervision fields so the summary can count the shard correctly while still preserving operator evidence. Avoid changing unrelated stages unless they opt into the same optional fields.

Then update `tests/parsing/test_canonical_line_roles.py`. The existing recovered-watchdog test must now assert that the recovered shard is finalized as completed, that the raw watchdog evidence is still present, and that the summary reports zero unrecovered watchdog kills plus one recovered watchdog shard. Add or update a second focused regression for an unrecovered `watchdog_command_execution_forbidden` shard so the summary still counts it as a real watchdog kill. If a helper-level unit test is clearer than forcing everything through the full stage harness, add that too, but keep at least one end-to-end stage test because the bug is in artifact finalization and summary rollup, not just a pure function.

Do not change command policy in this plan. If the unrecovered forbidden-command test starts passing because the policy was broadened, that is a regression. The whole point is to distinguish real failures from recovered successes, not to redefine failure away.

Finally, update the new plan file itself with the implementation outcome, the exact tests run, and any field-name decisions that changed during implementation. If the final names differ from the placeholders in this plan, record that in `Decision Log` and `Revision Note`.

## Validation and Acceptance

The change is accepted when all of the following are true:

1. A recovered line-role watchdog-retry regression writes shard `status.json` with `status == "validated"`, `state == "completed"`, `reason_code` set to a recovery-oriented final value, `watchdog_retry_status == "recovered"`, and raw watchdog attempt fields preserved in the artifact.

2. The recovered-line-role regression summary reports `watchdog_killed_shard_count == 0` and `watchdog_recovered_shard_count == 1`.

3. A genuine unrecovered `watchdog_command_execution_forbidden` line-role regression still reports `state == "watchdog_killed"` and contributes `watchdog_killed_shard_count == 1`.

4. Recovered shards no longer appear in the line-role worker failure list just because the original workspace session died.

5. The targeted `.venv` test commands in this plan pass without needing a live Codex rerun.

If the user later approves a benchmark rerun, the saved-run acceptance check is simple: the benchmark summary under `.../prompts/line-role-pipeline/telemetry_summary.json` should stop reporting recovered shards as watchdog kills, while any real unrecovered watchdog kills remain visible.

## Idempotence and Recovery

These edits are safe to make incrementally. First add the normalization helper and tests, then thread the new fields into status artifacts, then update the summary path. Re-running the targeted tests is safe and should produce the same results each time.

If changing the generic direct-telemetry summarizer causes unexpected fallout in other stages, keep the optional fields generic but make the line-role wrapper responsible for providing them only on line-role shard rows. That fallback preserves the plan’s main behavior without forcing unrelated stages to adopt new summary semantics immediately.

If field names need to change during implementation, update every assertion and artifact writer in the same patch and record the rename in `Decision Log` so a future reader does not have to infer when the contract changed.
