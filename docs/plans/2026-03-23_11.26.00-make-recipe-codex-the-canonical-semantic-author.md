# Make Recipe Codex The Canonical Semantic Author

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

After this change, the recipe LLM stage should own the semantic answer to a simple question: “what is this recipe?” In plain English, that means that once label-first grouping has already decided that a span of blocks is one recipe, the optional recipe Codex stage becomes the one place that decides the recipe’s corrected title, description, ingredients, instructions, notes, yield, variants, and ingredient-to-step links. The repository should still write the final files, assign IDs, enforce schemas, normalize stable derived fields, and preserve provenance, but it should stop quietly re-deciding recipe meaning after the recipe stage has already answered.

The user-visible proof is that a stage run with recipe Codex enabled writes one explicit authoritative recipe-semantics artifact before final drafts are built, and the final drafts are a deterministic projection of that artifact rather than a second semantic rewrite. A human debugging a bad cookbook should be able to open the run directory, inspect the authoritative recipe payload for one recipe, and understand the final recipe output without needing to mentally merge `RecipeCandidate`, override dictionaries, and later draft-building heuristics.

## Non-Goals

This plan does not change the upstream recipe-boundary decision. Label-first grouped spans remain the authority for “is this a recipe?” and “where does the recipe start and end?”

This plan does not make the LLM write literal final files such as `final drafts/*.json`, JSON-LD files, manifests, or markdown. Repo code keeps responsibility for output paths, IDs, schema validation, formatting, and deterministic derived fields.

This plan does not redesign the knowledge stage, Stage 7 non-recipe authority, or the importer redesign already covered elsewhere. It also does not require immediate removal of the old tip/topic side lane, though the two cleanups align well.

## Outcomes & Retrospective

This document currently captures the design and implementation plan only. No code has been changed yet. The intended result is a recipe pipeline that tells one simple story: label-first decides recipe boundaries, recipe Codex decides recipe semantics when enabled, and deterministic code turns that semantic payload into validated final outputs. The main risk is hidden late-stage behavior in drafting and output helpers that still changes recipe meaning after the new authoritative payload exists.

## Context and Orientation

The current stage pipeline starts with importers, but importer recipes are not the final truth. `cookimport/staging/import_session.py:execute_stage_import_session_from_result(...)` first calls `build_label_first_stage_result(...)` from [cookimport/parsing/label_source_of_truth.py](/home/mcnal/projects/recipeimport/cookimport/parsing/label_source_of_truth.py). That step rebuilds authoritative recipe spans from block labels. After that point, the recipe list is already more authoritative than the importer list.

When recipe Codex is enabled, `execute_stage_import_session_from_result(...)` calls `run_codex_farm_recipe_pipeline(...)` in [cookimport/llm/codex_farm_orchestrator.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_orchestrator.py). Today that function returns three different semantic surfaces: an updated `ConversionResult`, an `intermediate_overrides_by_recipe_id` dictionary, and a `final_overrides_by_recipe_id` dictionary. The writer path later mixes those with deterministic draft building in [cookimport/staging/draft_v1.py](/home/mcnal/projects/recipeimport/cookimport/staging/draft_v1.py) and [cookimport/staging/writer.py](/home/mcnal/projects/recipeimport/cookimport/staging/writer.py).

In plain English, “semantic author” means “the component that decides the meaningful recipe content.” That includes the corrected recipe title, the ingredient list, the instruction list and step boundaries, recipe notes, yield, variant content, and which ingredients belong to which steps. “Validator and normalizer” means the code that checks the payload, fills stable mechanical defaults, assigns IDs, writes output files, and rejects impossible shapes. The current design blurs those jobs together. The recipe stage changes some content, but the repo still has enough downstream logic to continue acting like a second recipe editor.

The main files in this plan are:

- [cookimport/llm/codex_farm_contracts.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_contracts.py), which defines the recipe-stage payload contracts and validators.
- [cookimport/llm/codex_farm_orchestrator.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_orchestrator.py), which runs the recipe stage and currently returns update and override surfaces.
- [cookimport/core/models.py](/home/mcnal/projects/recipeimport/cookimport/core/models.py), which defines `ConversionResult`, `RecipeCandidate`, and shared runtime data models.
- [cookimport/staging/import_session.py](/home/mcnal/projects/recipeimport/cookimport/staging/import_session.py), which is the main stage coordinator and currently threads recipe overrides into writing.
- [cookimport/staging/draft_v1.py](/home/mcnal/projects/recipeimport/cookimport/staging/draft_v1.py), which turns recipe data into cookbook draft shape and still performs late semantic derivation.
- [cookimport/staging/writer.py](/home/mcnal/projects/recipeimport/cookimport/staging/writer.py), which writes drafts, sections, and other outputs and must stop being a second semantic authority.

This plan intentionally assumes the importer redesign in [2026-03-23_10.44.32-redesign-importers-as-source-normalizers.md](/home/mcnal/projects/recipeimport/docs/plans/2026-03-23_10.44.32-redesign-importers-as-source-normalizers.md) can happen independently. That importer plan makes upstream facts cleaner, but recipe semantic ownership can still be fixed now because label-first regrouping already produces the recipe units that the recipe LLM owns.

## Milestones

### Milestone 1: Define the authoritative recipe payload

At the end of this milestone, the repository has one explicit model for “the accepted semantic recipe answer” and one staged artifact that writes those payloads before final drafting begins. The model must be plain enough that a novice can open the artifact and answer: what title did the system settle on, what ingredients and instructions does it believe belong to the recipe, what notes and yield did it keep, and how were ingredients linked to steps?

This milestone should define a new model such as `CanonicalRecipePayload` or `AuthoritativeRecipeSemantics`. The exact name can change during implementation, but the model must not be a free-form dictionary and must not be called a “candidate.” It should include recipe identity, owned span metadata, the semantically owned fields, bounded diagnostics about validation or fallback, and a place for explicitly repo-owned mechanical fields to stay out of the semantic contract.

The proof for this milestone is focused tests in the recipe contracts/orchestrator area plus one deterministic artifact write during a stubbed recipe-stage test. No live token spend is needed.

### Milestone 2: Make the recipe stage emit the authoritative payload in parallel

At the end of this milestone, recipe Codex still supports the current path, but it also emits the new authoritative payload for every owned recipe. This keeps migration risk low while making the intended authority visible. The recipe-stage apply result should expose the new authoritative payload map directly, and runtime artifacts should serialize it into a stable location under the recipe runtime root.

The compatibility rule here is simple. During this milestone, old override dictionaries may still exist, but they become compatibility outputs rather than the conceptual source of truth. New code should prefer the canonical payload first whenever both are present.

The proof for this milestone is that the orchestrator tests can validate payload generation and the prompt/runtime artifact tests can show the new payload artifact exists beside the old runtime outputs.

### Milestone 3: Flip drafting and writing to consume canonical recipe semantics first

At the end of this milestone, `cookimport/staging/import_session.py`, `cookimport/staging/draft_v1.py`, and `cookimport/staging/writer.py` no longer need to reconstruct recipe meaning from loose overrides. Instead, they read one canonical recipe semantic payload and only do bounded deterministic work afterward. That bounded work includes schema shaping, ID assignment, markdown generation, section extraction that does not change semantic text, and safe normalization such as null/empty cleanup or stable field ordering.

The critical acceptance rule is that deterministic drafting may not silently rewrite recipe content fields that the canonical payload already owns. If the draft builder still needs to derive something semantic, that field belongs back in the canonical payload or the deterministic fallback path when the recipe stage is off.

The proof for this milestone is updated staging and draft tests showing that final drafts match the canonical payload rather than hidden late recomputation rules.

### Milestone 4: Delete the old authority seam

At the end of this milestone, the old `final_overrides_by_recipe_id` and `intermediate_overrides_by_recipe_id` authority seam is gone or reduced to a narrow read-side compatibility shim for archived tests only. The active path for new runs should have one recipe semantic owner object, one validation boundary, and one deterministic write boundary.

The proof for this milestone is that live code paths no longer need to ask “which override dict should I read?” and updated docs describe the recipe stage as the canonical semantic owner when enabled.

## Validation and Acceptance

This plan is complete only when the recipe path tells one coherent story.

First, when recipe Codex is enabled and the payload validates, one explicit canonical recipe payload exists for each accepted recipe span. That payload is visible in run artifacts and is easy to inspect.

Second, stage-backed drafting and writing consume that payload as their semantic source. Final recipe files may still add mechanical fields and deterministic formatting, but they do not silently change owned content such as title, ingredient lines, instructions, notes, yield, or ingredient-step links.

Third, when recipe Codex is off, the deterministic path still works by producing the same canonical payload shape through a repo-owned builder. The downstream writer contract should not fork on “LLM versus no LLM.”

Fourth, the old override dictionaries are no longer a second live truth lane. Either they are deleted from active code paths or they survive only as narrow compatibility adapters for archived reads and tests.

Fifth, docs and reviewer-facing artifacts are truthful. The architecture and LLM docs must say that recipe Codex owns recipe semantics when enabled, while deterministic code owns validation, normalization, and writing. There must be no stale docs describing the recipe stage as merely “correction plus later rebuild” if later rebuild is no longer semantically authoritative.

Targeted tests must prove all of the following:

- the recipe orchestrator emits canonical recipe payloads for successful recipe-stage runs
- the draft builder consumes canonical payloads without requiring override dictionaries
- deterministic no-LLM fallback builds the same payload shape
- final draft output matches canonical payload content for the fields the payload owns
- prompt-preview and artifact-summary tests still describe the recipe stage coherently after the contract change

## Idempotence and Recovery

This change should be landed incrementally and is safe to repeat as long as each phase ends with focused tests passing. The risky part is not data loss; it is hidden dual authority. If a migration step leaves both canonical payloads and old override dictionaries active, the safe recovery path is not to revert the new payload. The safe recovery path is to finish the reader flip so only one of them is treated as authoritative.

Keep the no-LLM deterministic builder alive until the canonical payload path has replaced all live consumers. If a test reveals a remaining writer or benchmark reader that still depends on old overrides, add a small compatibility adapter at that narrow seam, document it as temporary, and continue shrinking the old path. Do not re-expand `RecipeCandidate` plus override dictionaries into the conceptual source of truth.

The optional live smoke output path `/tmp/recipe-semantic-author-smoke` is disposable and should be deleted before each rerun. If token-spend approval never arrives, the unit and integration tests above are still required and still provide the main acceptance proof for the architectural change.

## Interfaces and Dependencies

At the end of this plan, there must be one explicit recipe semantic model. The exact type name can change, but it must be a typed model with a stable schema, not an unstructured dictionary. It should expose fields equivalent to:

    recipe_id: str
    owned_block_indices or owned_block_ids: list[int] or list[str]
    title: str
    description: str | None
    ingredients: list[...]
    instructions: list[...]
    notes: list[str] or str | None
    yield_fields: ...
    variants: list[str]
    ingredient_step_mapping: dict[str, list[int]]
    authority_source: "llm" | "deterministic"
    validation_status: "accepted" | "normalized" | "fallback"
    validation_messages: list[str]

`cookimport/llm/codex_farm_orchestrator.py:CodexFarmApplyResult` must expose a map of these payloads by recipe id. The current override fields may exist during migration, but active stage-backed code should stop treating them as the primary truth.

`cookimport/staging/import_session.py:execute_stage_import_session_from_result(...)` must thread authoritative recipe payloads into the draft/writer path. The main recipe-writing code should not need to infer meaning by merging a `RecipeCandidate` with override dictionaries.

`cookimport/staging/draft_v1.py` must either accept the new authoritative payload directly or accept a small adapter object that is clearly derived from it. It must not require `ingredient_step_mapping_override` on the main path once the canonical payload exists. If that argument remains for tests or compatibility, the docstring and callers must make clear that it is no longer the primary semantic path.

`cookimport/staging/writer.py` and related helpers must write final drafts, sections, and JSON-LD from the authoritative recipe payload plus deterministic normalized derivatives of it. New stage artifacts should include the authoritative recipe payload file path in a discoverable, documented location under the recipe runtime root.

`docs/01-architecture/01-architecture_README.md`, `docs/05-staging/05-staging_readme.md`, and `docs/10-llm/10-llm_README.md` must describe the same authority story the code now implements.
