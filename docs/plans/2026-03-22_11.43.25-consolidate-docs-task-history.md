This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

After this change, `docs/tasks/` should no longer be a second unsorted archive of one-off task notes. The durable information from those task files should live where a future human or AI would actually look first: current-state facts in the destination section README and implementation history, failed paths, and major decisions in that section log.

The user-visible result is straightforward. Running `npm run docs:list` should show no loose task files under `docs/tasks/`, and the relevant section docs should explain the current line-role, recipe/knowledge workspace-worker, and benchmark Oracle/upload behavior without forcing a future reader to reconstruct history from dozens of old task notes.

## Context and Orientation

This repository keeps topical docs under numbered section folders such as `docs/04-parsing/`, `docs/07-bench/`, and `docs/10-llm/`. Each active section has a current-state reference document and a matching history log. The active task archive currently lives under `docs/tasks/`, but those files mix implemented behavior, design notes, and anti-loop debugging history in a way that makes future retrieval expensive.

The current task set is concentrated in three domains. Parsing tasks cover canonical line-role semantics, packet shaping, overlap context, `HOWTO_SECTION`, `RECIPE_VARIANT`, and the `KNOWLEDGE` versus `OTHER` prose boundary. LLM tasks cover workspace-worker contracts, watchdog behavior, final-message authority, helper tooling, recipe worker contract changes, knowledge packet validity, and single-task queue authority. Benchmark tasks cover single-book naming, Oracle browser upload model/flag behavior, and `upload_bundle_v1` diagnostic parity.

The destination docs already exist:

- `docs/04-parsing/04-parsing_readme.md`
- `docs/04-parsing/04-parsing_log.md`
- `docs/07-bench/07-bench_README.md`
- `docs/07-bench/07-bench_log.md`
- `docs/10-llm/10-llm_README.md`
- `docs/10-llm/10-llm_log.md`

The consolidation rule is simple: facts about the active runtime contract go into the README for that domain, while chronology, rationale, failed paths, and "do not repeat this" notes go into the matching log.

## Plan of Work

First, map each task file to the section where a future maintainer would look for that information. Line-role semantic behavior belongs primarily in parsing docs, even when the implementation involved workspace workers, because the durable question is how line-role behaves and why the labels or packet shapes are the way they are. Recipe and knowledge workspace-worker runtime, watchdog, helper, and authority changes belong in the LLM docs because they describe the shared direct-exec execution model. Benchmark naming and Oracle/upload-bundle fixes belong in bench docs.

Next, edit the destination READMEs to absorb the task-file information that helps a future contributor understand the current system. These additions should explain what the current contract is, where the relevant files live, what the important anti-footgun constraints are, and which "obvious" fixes are already known to be wrong.

Then edit the destination logs in chronological order. Keep entries only for decisions and implementation history that still matters to the current system. If several task files form one coherent sequence, merge them into a single timeline entry rather than copying task-by-task scaffolding. Preserve the meaningful failed attempts and the reasons certain seams are now repo-owned or high-evidence-only.

After the information is preserved, remove the `docs/tasks/*.md` files. The final verification step is to rerun `npm run docs:list` and confirm that no loose task files remain and that the destination docs still render as the active reference set.

## Validation and Acceptance

Acceptance is documentation- and structure-based.

The updated README files must describe the active contracts introduced by the task docs in the sections where a future maintainer would expect to find them. The updated logs must preserve the major decision trail and anti-loop context from the task archive, especially around line-role packet authority, workspace-worker supervision/tooling, and benchmark Oracle/upload behavior.

The filesystem acceptance bar is:

- `docs/tasks/` contains no remaining markdown task files.
- `npm run docs:list` completes successfully.
- The destination docs remain the discoverable source of truth for current behavior and history.

## Idempotence and Recovery

These changes are safe to apply incrementally. It is fine to update one section README/log pair at a time and rerun `npm run docs:list` afterward. If a destination section starts to accumulate stale or duplicated detail, prefer compressing the repeated wording in place rather than preserving both copies. The deleted task files should not be restored unless a critical piece of history was accidentally omitted from the destination docs.
