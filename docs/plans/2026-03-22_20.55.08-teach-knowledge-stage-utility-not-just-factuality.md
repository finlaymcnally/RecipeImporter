# Teach the knowledge stage to optimize for durable cooking utility

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

The optional non-recipe knowledge stage currently aims at “reusable cooking knowledge,” but the actual gold-set boundary is narrower: keep only information that gives the reader durable cooking leverage. The target is not “anything true about cooking.” The target is “information worth preserving in a cookbook knowledge base because it improves future cooking decisions, diagnosis, or judgment.”

After this change, a knowledge worker should keep genuinely useful technique, cause-and-effect explanation, sensory cues, reference material, troubleshooting, substitution, storage, and safety rules, while rejecting memoir, framing, praise, navigation, rhetorical headings, and technically true but low-value filler. The visible proof is that prompt text, worker sidecars, task hints, validator vocabulary, and stage summaries all talk in that utility language, and the local regression suite includes benchmark-shaped canaries for “useful vs merely factual” instead of only structural JSON-shape checks.

This plan intentionally treats the knowledge stage as the semantic authority for subtle outside-recipe usefulness. The parser and line-role stages may still perform a coarse obvious-junk veto, but they must not regain final authority over nuanced “worth keeping?” questions.

## Outcomes & Retrospective

This plan is not implemented yet. Its outcome target is an end-to-end knowledge-stage contract that consistently asks one question: “Would saving this materially improve future cooking decisions?” What remains is the implementation work across prompt language, worker hints, task metadata, schema validation, examples, telemetry, and tests. The main lesson from the research pass is that the code already has strong structural guardrails; the missing piece is a shared utility vocabulary that reaches every knowledge-stage surface instead of living only in human interpretation.

## Context and Orientation

The relevant pipeline starts after Stage 7 non-recipe routing. Stage 7 writes review-eligible outside-recipe spans into `08_nonrecipe_spans.json` and can also exclude obvious junk upstream. The optional knowledge stage then reviews the surviving outside-recipe spans and becomes the final semantic authority for `knowledge` versus `other`.

The knowledge stage currently spans six main areas:

`cookimport/llm/codex_farm_knowledge_jobs.py` builds `knowledge/in/*.json` shard payloads from review-eligible non-recipe spans. It also records shard metadata such as ordered chunk ids, block coverage, table hints, and today’s coarse `strong_knowledge_cue`.

`llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md` is the shared pack prompt for direct prompt construction and structured follow-up attempts. `cookimport/llm/knowledge_prompt_builder.py` injects the inline JSON payload into that template.

`cookimport/llm/codex_farm_knowledge_orchestrator.py` builds the workspace-worker prompt, writes the worker root, launches the batch worker session, and handles retry / repair follow-up calls. This file owns the live worker instructions that say what the spawned knowledge worker should read first and how it should interpret the output contract.

`cookimport/llm/knowledge_workspace_tools.py` writes `CURRENT_BATCH.md`, `CURRENT_BATCH_FEEDBACK.md`, `current_batch.json`, fallback `CURRENT_TASK*` files, `OUTPUT_CONTRACT.md`, helper scripts under `tools/knowledge_worker.py`, and example files under `examples/`. If the knowledge worker is “the same model as the agent writing this plan,” this module is the repo-owned surface that shapes what that worker will actually see and trust.

`cookimport/llm/codex_farm_knowledge_models.py` and `cookimport/llm/codex_farm_knowledge_ingest.py` define and validate the semantic packet-result shape. These files are the fail-closed boundary between worker intent and accepted final authority.

`tests/llm/test_knowledge_prompt_builder.py`, `tests/llm/test_knowledge_workspace_tools.py`, `tests/llm/test_knowledge_output_ingest.py`, and `tests/llm/test_codex_farm_knowledge_orchestrator.py` are the current main regression seams for prompt wording, helper files, validator behavior, and worker-orchestrator alignment.

This plan introduces one new term of art: “utility boundary.” In this repository, that means the line between text worth preserving as cookbook knowledge because it improves future cooking judgment, and text that may be true, polished, or food-adjacent but is not useful enough to keep. The boundary is narrower than factual correctness and narrower than “sounds educational.”

## Milestones

### Milestone 1: Encode a shared utility rubric and vocabulary

At the end of this milestone, the knowledge stage should stop describing the positive class as generic reusable cooking information and start describing it as durable cooking leverage. The pack prompt, workspace-worker prompt, `OUTPUT_CONTRACT.md`, and examples should all ask the same questions: does this help a cook decide, diagnose, adjust, or avoid failure later; is it non-obvious enough to earn storage; and would removing it reduce practical cookbook value rather than only voice or framing?

This milestone also introduces the machine-readable vocabulary needed to debug that judgment. The worker should no longer emit arbitrary freeform `reason_code` strings. Instead, it should use a constrained reason set that cleanly separates positive reasons such as `transferable_principle`, `cause_effect`, `sensory_cue`, `failure_mode`, `ingredient_behavior`, `actionable_technique`, `reference_value`, and `storage_or_safety` from negative reasons such as `too_generic`, `true_but_low_utility`, `memoir_or_voice`, `book_framing_or_marketing`, `navigation_or_heading`, `rhetorical_heading_without_leverage`, and `context_without_leverage`.

Acceptance for this milestone is prompt and model-contract alignment. A future worker reading the main prompt, `CURRENT_BATCH.md`, and `OUTPUT_CONTRACT.md` should see the same utility questions and the same allowed reason vocabulary, and the validator should reject unknown reason codes instead of silently accepting them.

### Milestone 2: Give the worker better utility-oriented local data without restoring deterministic semantic authority

At the end of this milestone, the worker should receive stronger repo-owned local hints about why a packet might be useful or not useful, but the authoritative billed shard payload should remain raw block text plus mechanically true structure. The new hints belong in task metadata, `CURRENT_BATCH.md`, `current_batch.json`, and per-task `hints/*.md`, not as semantic labels inside `knowledge/in/*.json`.

The right seam for this work starts in `cookimport/parsing/chunks.py` and `cookimport/llm/codex_farm_knowledge_jobs.py`. Those files already compute coarse noise, knowledge, narrative, heading, and table/reference cues. Extend them to derive a local utility profile for each chunk or packet, with fields such as positive cue hits, negative cue hits, heading/reference/table shape, first-person or memoir signals, book-marketing signals, and whether the packet looks borderline rather than strongly useful. That profile should be stored in shard/task metadata and rendered into worker-local guidance, not treated as final authority.

Acceptance for this milestone is a worker root that materially helps a real worker answer the task. The batch sidecars and hint files should explain, in plain language, which local cues are pushing toward “durable leverage” and which are warning that the text may be merely factual, rhetorical, promotional, or low-value.

### Milestone 3: Rebuild the examples and helper surfaces around the utility boundary

At the end of this milestone, the worker-facing helper experience should teach the actual desired behavior. `OUTPUT_CONTRACT.md` and `examples/` should include contrast examples for useful technique versus memoir, concise reference value versus navigation, grounded principle versus true-but-trivial statement, and rhetorical heading versus real knowledge. The existing valid-versus-invalid snippet-echo examples should remain, but they should no longer be the only semantic teaching surface.

This milestone should also tighten the helper feedback loop. When `check-batch` or `install-batch` encounters a semantically suspicious all-other result on a strong positive cue packet, or a suspicious positive result on low-utility content, the feedback should explain the utility issue in worker language instead of only speaking in generic “semantic rejection” terms.

Acceptance for this milestone is that a spawned knowledge worker would plausibly reach the correct utility judgment without repo-source spelunking. The helper surface should teach the intended standard directly.

### Milestone 4: Make validation, summaries, and refinement reports utility-aware

At the end of this milestone, accepted worker outputs should carry consistent reason codes, and stage summaries should report them. `cookimport/llm/codex_farm_knowledge_ingest.py`, refinement reports, and stage summaries should count useful reasons and rejected reasons separately so an operator can see whether the stage is throwing away content as `too_generic`, `memoir_or_voice`, `book_framing_or_marketing`, and so on.

This milestone should not weaken the existing snippet grounding, coverage, or all-other strong-cue protections. It should add semantic observability on top of them. It is acceptable for the validator to stay conservative and only enforce the reason vocabulary plus the current structural rules, as long as the tests and summaries make the new utility boundary explicit and inspectable.

Acceptance for this milestone is that opening `knowledge_stage_summary.json`, `promotion_report.json`, or the refinement report shows why the stage kept or rejected content in terms of utility rather than only in terms of `knowledge` versus `other`.

### Milestone 5: Lock the behavior down with benchmark-shaped canaries

At the end of this milestone, the repo should have deterministic canaries for the exact classes the user described. Positive canaries should include storage advice, troubleshooting, cause-and-effect technique explanation, concise reference tables, and practical rules that improve future cooking decisions. Negative canaries should include blurbs, memoir, chapter framing, rhetorical headers without leverage, book-about-cooking sales copy, and technically true but low-value filler.

These canaries should live in the existing knowledge-stage test files and, where helpful, in parsing-side cue tests. The goal is not to simulate the entire live benchmark. The goal is to make it hard to regress back to “anything true about cooking becomes knowledge.”

Acceptance for this milestone is a focused local test run that demonstrates the new utility language, helper surfaces, schema validation, and reason reporting without spending live model tokens. Any live benchmark or stage proof remains a separate final step that requires explicit user approval.

## Validation and Acceptance

Acceptance is behavioral, not merely structural.

The prompt tests must prove that the pack prompt and the workspace-worker prompt now explicitly reject “technically true but low-value” prose as `other`. They must mention durable cooking leverage, future cooking decisions, and the fail-closed default for marginal utility. They must also keep the existing warnings against blurbs, memoir, navigation, and book framing.

The model and validator tests must prove that the worker emits a constrained reason vocabulary and that unknown reason codes are rejected or normalized deliberately. A useful chunk should carry a useful reason such as `cause_effect`, `actionable_technique`, or `storage_or_safety`; an all-other memoir or framing chunk should carry a negative reason such as `memoir_or_voice`, `book_framing_or_marketing`, or `true_but_low_utility`.

The worker-surface tests must prove that `CURRENT_BATCH.md`, `CURRENT_BATCH_FEEDBACK.md`, `OUTPUT_CONTRACT.md`, and `examples/` teach the utility boundary directly. There should be at least one contrast example for each of the following families: useful technique versus memoir, useful reference table versus navigation, grounded principle versus true-but-trivial statement, and rhetorical heading versus real knowledge.

The cue-building tests must prove that local utility hints are informative but not authoritative. The billed input bundle under `knowledge/in/*.json` must remain raw block text plus mechanically true structure. The worker-local metadata and hints may summarize utility cues, but they must not become hidden final labels that bypass the knowledge worker’s own judgment.

The ingest and summary tests must prove that strong positive cue packets still fail closed when they collapse to all-other with no useful evidence, while obvious low-value packets such as front matter can still validate as all-other. The stage summaries must expose reason counts so an operator can tell whether the stage is rejecting content as memoir, framing, navigation, or low-utility truth rather than only seeing binary `knowledge` and `other`.

## Idempotence and Recovery

This plan should be implemented additively and fail closed.

If the new reason vocabulary causes compatibility breakage, keep a temporary normalization layer in `normalize_knowledge_worker_payload(...)` and related validators so older freeform reason strings can map into the new constrained set before hard rejection. Remove that fallback only after the tests and prompt surfaces are updated.

If the new utility cues in task metadata become noisy or misleading, the safe rollback is to keep the prompt and reason vocabulary changes but remove the new cue summaries from `CURRENT_BATCH.md` and hint files. That preserves the semantic contract while undoing the extra local guidance.

If the stronger utility boundary becomes too aggressive and starts rejecting clearly useful content, back out the cue-specific pessimism first. The likely seams are `_chunk_has_strong_knowledge_cue(...)`, any new low-utility cue summarizer in `cookimport/parsing/chunks.py`, or over-broad negative examples in the prompt and `examples/`. Do not weaken snippet-grounding or coverage validation in response to a semantic tuning miss.

If mixed live environments still produce legacy outputs without reason codes, keep the validator permissive long enough to normalize missing `reason_code` to a small explicit fallback such as `legacy_missing_reason_code` during transition, but make the final contract and tests require the real vocabulary.

## Artifacts and Notes

The intended worker question is:

    “Would saving this materially improve a cook’s future decisions, diagnosis, or technique?”

The intended positive families are:

    transferable principle
    cause and effect
    sensory cue
    failure mode
    ingredient behavior
    actionable technique
    reference value
    storage or safety

The intended negative families are:

    too generic
    true but low utility
    memoir or voice
    book framing or marketing
    navigation or heading
    rhetorical heading without leverage
    context without leverage

The point is not to preserve everything that is true. The point is to preserve what earns space in a cookbook knowledge base.
