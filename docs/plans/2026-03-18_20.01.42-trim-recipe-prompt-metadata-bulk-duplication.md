# Trim Recipe Prompt Metadata, Bulk, And Duplication

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

Today the recipe Codex surface is not the biggest total token spender, but it is still inefficient relative to the amount of actual recipe text it sends. After this change, the recipe stage should keep the same recipe evidence rows and the same useful recipe-correction behavior, but send less schema bulk, less duplicated hint content, and less repeated tagging-guide weight to the model. A human should be able to prove the change worked by rerunning zero-token `prompt_preview` on the three fresh cutdown cookbooks and seeing recipe-stage input tokens drop while recipe shard counts and evidence coverage remain intact.

## Surprises & Discoveries

- Observation: the recipe stage is cheap in total tokens, but inefficient relative to its raw recipe text.
  Evidence: on the current zero-token previews, recipe-stage input tokens versus raw recipe text are about `49.9x` for Dinner for 2 (`6587` input over about `132` recipe-text tokens), `12.23x` for Salt Fat Acid Heat (`24513` over about `2004`), and `11.52x` for The Food Lab (`23912` over about `2076`).

- Observation: the live recipe shard already uses compact top-level aliases, but the per-recipe hint object is still large and schema-shaped.
  Evidence: current recipe shard payloads use top-level keys `an`, `ids`, `r`, `sh`, `sid`, `tg`, `v`, and `wb`, but each recipe payload still carries `h` with fields like `@context`, `@type`, `comment`, `cookingMethod`, `image`, `recipeCategory`, `recipeCuisine`, `recipeIngredient`, `recipeInstructions`, `suitableForDiet`, `supply`, `tags`, and `tool`.

- Observation: the tagging guide is not the whole recipe payload, but it is large enough to matter when the recipe text itself is short.
  Evidence: in a representative Food Lab recipe shard, the serialized `tg` guide is about `1601` characters inside a total payload of about `9325` characters, while canonical recipe text in that shard is only about `937` characters.

- Observation: canonical recipe text and evidence rows are both carrying meaningful grounding, so deleting `ev` would be a risky shortcut.
  Evidence: the representative Food Lab shard above had about `937` canonical text characters and about `920` evidence-row text characters. That is not dead weight; it is a second grounding seam for source alignment.

- Observation: shrinking the recipe hint and guide was enough to cut recipe-stage input by roughly one third on all three books while keeping shard counts unchanged.
  Evidence: Dinner for 2 recipe input dropped `6587 -> 4326`, Salt Fat Acid Heat dropped `24513 -> 15972`, and The Food Lab dropped `23912 -> 15460`, with recipe call counts unchanged at `3`, `5`, and `5`.

- Observation: the compacted payload is cleaner overall, but some compact step hints are being serialized in a way that is less useful than intended.
  Evidence: fresh March 19 recipe previews show `h.s` entries like `"{'@type': 'HowToStep', 'text': 'Roast: Butternut Squash and Brussels Sprouts in Agrodolce'}"` instead of plain step text. That is still smaller than the old schema dump, but it is noisier for the model to read quickly.

## Decision Log

- Decision: this plan will not cut recipe evidence rows from the model-facing payload.
  Rationale: the user explicitly agreed that cutting recipe evidence rows is a bad idea, and the current payload measurements show `txt` and `ev` are both substantial grounding seams rather than obvious junk.
  Date/Author: 2026-03-18 / Codex

- Decision: target metadata-heavy hint content and tagging-guide bulk before attempting any deeper recipe-text compaction.
  Rationale: the current recipe transport is already partly compact at the alias level, so the highest-confidence wins are likely in `h` and `tg`, not in the core `txt` or `ev` evidence.
  Date/Author: 2026-03-18 / Codex

- Decision: keep recipe tagging inside the recipe surface rather than inventing a fourth prompt surface.
  Rationale: current repo policy is that tags ride along with recipe correction, and `docs/09-tagging/09-tagging_README.md` explicitly says there is no separate tags subsystem.
  Date/Author: 2026-03-18 / Codex

- Decision: compact the recipe hint to only title, ingredients, steps, and optional description/yield/tags, and compact the tagging guide to version/rules/categories/examples only.
  Rationale: the measurements showed the recipe hint and guide were the bulk of the non-evidence payload, while fields like `@context`, `identifier`, and verbose category descriptions were not required by the prompt contract.
  Date/Author: 2026-03-18 / Codex

- Decision: remove shard-level `wb`, `sh`, and `an` from the billed recipe shard payload and minify the inline JSON.
  Rationale: those fields help local runtime bookkeeping or human audit, not the model. Dropping them and removing prompt-body whitespace produces direct token savings without changing recipe evidence.
  Date/Author: 2026-03-18 / Codex

- Decision: compact hint readability is part of correctness, not just a nice-to-have.
  Rationale: recipe payload trimming is only a real improvement if the remaining compact hints are easier for Codex to use than the data they replaced. Structured step objects must therefore be normalized to plain text before entering `h.s`, or else be omitted.
  Date/Author: 2026-03-19 / Codex

## Context and Orientation

The live recipe Codex surface is implemented in `cookimport/llm/codex_farm_orchestrator.py` and `cookimport/llm/codex_farm_contracts.py`. In this repository, a "recipe shard" means one model call that owns one or more recipes. The prompt-preview artifacts rebuild those same shard payloads without spending live Codex tokens. The important files for this work are:

- `cookimport/llm/codex_farm_contracts.py`, which defines the model-facing recipe shard contract and serialization helpers.
- `cookimport/llm/codex_farm_orchestrator.py`, which builds recipe shard inputs, emits inline prompt text, runs the direct Codex worker flow, and promotes outputs back into staged recipes.
- `cookimport/llm/recipe_tagging_guide.py`, which builds the tagging guide that currently rides inside each recipe shard payload.
- `cookimport/llm/prompt_preview.py`, which reconstructs zero-token prompt previews from saved runs.
- `tests/llm/test_codex_farm_orchestrator.py`, `tests/llm/test_recipe_phase_workers.py`, `tests/llm/test_prompt_preview.py`, and `tests/llm/test_fake_codex_farm_cli.py`, which cover recipe payload generation, shard execution, and preview reporting.

The current live recipe shard payload now uses short aliases throughout. A representative prompt-preview payload has:

- top-level keys `ids`, `r`, `sid`, `tg`, `v`
- per-recipe keys `ev`, `h`, `rid`, `txt`
- per-recipe hint object `h` carrying only compact correction hints
- a compact tagging guide `tg` with `v`, `r`, and `c`

Two terms matter here. "Metadata bulk" means structured fields that help the model only a little compared to their token cost, such as rich schema-style hint fields or repeated guide prose. "Evidence duplication" in this plan means different non-evidence representations of the same recipe facts, not the intentionally retained `ev` evidence rows. The plan may reduce duplicate hints or repeated guide content, but it must preserve the recipe evidence seam.

## Plan of Work

First, inspect the recipe shard input builder in `cookimport/llm/codex_farm_orchestrator.py` and the recipe shard contract in `cookimport/llm/codex_farm_contracts.py` to separate three categories clearly: core evidence that must stay (`txt`, `ev`, `rid`), useful-but-trimmable hints (`h`), and shard-level helper material (`tg`, `an`, and any duplicated identifiers). The implementation should make that split explicit so future prompt work can keep the evidence while evolving the surrounding guidance.

Next, compact the model-facing `h` recipe hint object. The likely safe move is to keep only the fields that help Codex repair recipe structure or choose tags when the canonical text is ambiguous. Fields that merely restate the same recipe in schema form, especially large arrays or verbose descriptive prose, should move out of the model-facing shard payload unless a focused test demonstrates they materially affect recipe correction. The compacted hint object can still use aliases or a denser nested contract, but it must remain intelligible in the checked-in prompt instructions. One explicit guardrail: when compacting structured `recipeInstructions` entries, normalize them to plain step text before writing `h.s`; do not stringify nested dict-like step objects.

Then, reduce tagging-guide bulk in `cookimport/llm/recipe_tagging_guide.py` and wherever the recipe shard payload is assembled. The preferred path is not to delete tagging support, but to stop shipping more guide than the model needs on every call. The implemented version keeps only compact rules plus category keys and example labels. The final design still lets the recipe surface emit raw selected tags for deterministic normalization downstream.

After that, revisit shard-level duplication in the recipe payload. The implemented cut removed workbook/source metadata and authority notes from the billed shard payload while leaving runtime manifests and audit artifacts untouched. Anything that exists only to help the runtime or human reviewers should remain in runtime metadata or shard manifests instead of being included in billed model input.

Finally, update prompt-preview and the recipe tests so they validate the compact contract directly. The acceptance bar is not "the payload got shorter"; it is "the payload got shorter while recipe evidence rows remain present, the recipe stage still exposes tags, and the preview still reports the same enabled three-stage topology."

## Idempotence and Recovery

This work is safe to iterate on because prompt-preview is zero-token and writes to a new timestamped output directory each time. Keep all local-only reporting and validation metadata in shard manifests or runtime metadata while shrinking the model-facing payload. If a recipe-quality test reveals that a trimmed hint field was actually carrying important signal, restore that one field and rerun the same focused tests plus the three previews. Do not "recover" by removing `ev`; that is outside the allowed scope of this plan.

## Interfaces and Dependencies

In `cookimport/llm/codex_farm_contracts.py`, the recipe output contract stays unchanged, but the recipe input contract is now denser and smaller. The core required input fields after this work are stable recipe identifiers, canonical recipe text, evidence rows, and enough tag-guidance structure for the recipe prompt to keep producing raw selected tags.

In `cookimport/llm/codex_farm_orchestrator.py`, the code that assembles recipe shard inputs must keep recipe ownership, evidence references, and deterministic promotion unchanged. Only the billed model-facing payload should get smaller. Any fields used only for local runtime bookkeeping should stay in shard manifests, worker metadata, or audit artifacts instead of the prompt payload.

In `cookimport/llm/recipe_tagging_guide.py`, the guide builder should remain deterministic and repo-owned. If the guide is trimmed or projected, the resulting data must still be sufficient for the recipe prompt to select tags in the same output slot the downstream normalization code already expects.
