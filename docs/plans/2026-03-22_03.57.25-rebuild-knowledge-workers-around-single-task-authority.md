# Rebuild Knowledge Workspace Workers Around Repo-Owned Single-Task Authority

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a cold-start knowledge worker should no longer be able to “finish” a large batch by inventing its own queue scheduler, heuristic classifier, and validator loop. Instead, the repository should own task sequencing, local validation, and task advancement one packet at a time. The worker should spend its effort on the only part that actually needs model judgment: deciding whether a specific chunk contains reusable knowledge and, if so, writing a short grounded snippet that survives the repository validator.

The user-visible success condition is much stronger than the earlier paved-road milestone. Opening a generated knowledge worker root should reveal one authoritative current task, one short human-readable brief, one repo-owned helper command that can complete or validate the current task, and one repo-owned status surface that tells the worker whether it may advance. A later benchmark rerun, when explicitly approved by the user, should show that workers stop batch-scripting the queue, stop poisoning entire shards with a handful of bad packets, and only terminate as successful when all assigned task packets have been individually validated by repo code.

This plan is intentionally ambitious. It includes not only helper improvements but a redesign of the workspace-worker runtime contract, supervision rules, queue surfaces, and prompt shape so that the helper path becomes the easiest path and the only scalable path.

## Surprises & Discoveries

- Observation: the existing knowledge helper solved the mechanical problem in theory but not in practice.
  Evidence: the March 22, 2026 benchmark worker prompt explicitly advertised `python3 tools/knowledge_worker.py overview`, `show`, `scaffold`, `check`, and `install`, yet the recorded worker commands in `data/output/2026-03-22_03.33.50/.../knowledge_manifest.json` show ad hoc `python3 - <<'PY'` queue scripts instead of helper usage.

- Observation: the dominant failure was not ignorance of the schema; it was delayed feedback.
  Evidence: re-checking the saved worker outputs against `cookimport/llm/knowledge_workspace_tools.py::check_workspace_draft(...)` shows many task files were individually valid, but 53 failed with `semantic_snippet_echoes_full_chunk` and 20 failed with `semantic_all_false_empty_shard`. Once those invalid task files were dropped, the parent shard proposals failed with `missing_owned_chunk_results`.

- Observation: the current supervisor’s notion of completion is too weak for semantic packet work.
  Evidence: `knowledge/workers/*/live_status.json` records `workspace_outputs_stabilized` as a completed terminal state even when the later phase summary reports `validated_shard_count: 0` and `validated_output_chunk_count: 0`.

- Observation: giving a cold-start worker an 82-task queue invites it to become a local batch program.
  Evidence: the final worker message in the failing run says it is “proceeding with an automated generation pass” and “write all 82 `out/*.json` files,” which is rational behavior given the current workspace shape but exactly the wrong behavior for semantic packet quality.

- Observation: `assigned_tasks.json` currently exposes too much authority.
  Evidence: because each task row already includes substantial task metadata and packet payload, the worker can derive a whole-batch scheduler from one file without ever opening `current_task.json`, `in/<task_id>.json`, or the repo-owned helper.

- Observation: tightening cold-start ergonomics is not enough unless the runtime also narrows the success surface.
  Evidence: the helper, contract examples, and prompt all existed in the failed run, but the runtime still let the worker bypass them, write its own packet generator, and terminate once file writes stabilized.

## Outcomes & Retrospective

This plan is now partially implemented. The knowledge runtime now has a skinny queue, repo-owned current-task sidecars, higher-level helper verbs, live queue validation, and queue-aware completion semantics. The remaining open question is whether this design should be generalized to recipe and line-role and whether a later user-approved benchmark rerun confirms the expected behavioral win.

If implemented fully, this plan should produce three practical outcomes. First, a knowledge worker root will become self-explanatory in a much stronger sense because there will be one current task, one short brief, one immediate feedback surface, and one repo-approved completion command. Second, the phase runtime will stop conflating “files were written” with “the work succeeded.” Third, the benchmark evidence should become much easier to interpret because task-level failures will be surfaced immediately and shard-level incompleteness will become a downstream symptom rather than the first visible sign of trouble.

The main risk is ambition creep. This plan purposely spans helper UX, queue data shape, prompt design, supervision policy, and shell policy. If the implementation later needs to be split, the split should preserve the overall direction documented here rather than collapsing back to another narrow helper-only patch.

## Context and Orientation

In this repository, the non-recipe knowledge review stage takes deterministic non-recipe chunks and asks a long-lived direct-exec language-model worker to classify each owned block as `knowledge` or `other`, then write short grounded snippets for the useful material. The current implementation lives primarily in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, `cookimport/llm/codex_farm_knowledge_ingest.py`, `cookimport/llm/knowledge_workspace_tools.py`, and the shared worker runtime in `cookimport/llm/codex_exec_runner.py`.

The important terms in this plan are:

- A “workspace worker” is one long-lived Codex session running inside a sterile local workspace under the direct-exec runner. It can read local files and write local outputs.
- A “task packet” is one knowledge-review unit, usually one chunk packet, represented by `current_task.json`, `in/<task_id>.json`, `hints/<task_id>.md`, and `out/<task_id>.json`.
- A “shard proposal” is the parent result assembled from all validated task packets for one parent shard. If even a few required task packets are invalid or missing, the parent shard proposal becomes incomplete and fails later.
- “Single-task authority” means the repository decides which task is current, whether its output is valid, and whether the worker may move on. The worker may keep one long-lived session, but it does not own the scheduler.
- A “skinny queue” means a queue file that shows only the information needed for orientation and progress, not enough embedded payload to make broad queue spelunking the easiest path.
- “Immediate validator feedback” means the same validator logic currently used by `validate_knowledge_shard_output(...)` or its task-local equivalent becomes visible to the worker during the live task loop, not only after the worker has already written a large batch.

The current paved-road work already achieved several useful things. Knowledge worker roots now contain `worker_manifest.json`, a skinny `assigned_tasks.json`, repo-written `current_task.json`, `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`, `OUTPUT_CONTRACT.md`, `examples/`, and `tools/knowledge_worker.py`. The helper now supports both the older low-level verbs and the higher-level `current`, `complete-current`, `check-current`, `install-current`, `next`, and `explain-failure` flow, and the repo-owned validator now drives live queue advancement rather than only post hoc ingest.

The March 22 benchmark also showed why that work was not enough. In `data/output/2026-03-22_03.33.50/.../knowledge/stage_status.json`, the task ledger reports `254` validated task packets, `49` invalid outputs, `104` watchdog-killed tasks, and only `0` validated shards. The phase summary reports `completed_with_failures`, `validated_shard_count: 0`, and `validated_output_chunk_count: 0`. Re-checking the saved task files shows that the common invalid task reasons were `semantic_snippet_echoes_full_chunk` and `semantic_all_false_empty_shard`, not basic JSON shape mistakes. The workers had the helper available, but the runtime still allowed them to read the whole queue, invent heuristics, and write an 82-task batch before the repo objected.

The key files for this plan are:

- `cookimport/llm/codex_farm_knowledge_orchestrator.py`, which defines how knowledge worker roots are built, how prompts are written, how task packets are grouped, and how validated task results are promoted back into shard proposals.
- `cookimport/llm/knowledge_workspace_tools.py`, which defines the current worker helper, scaffolding, task-local checking, and installation logic.
- `cookimport/llm/codex_farm_knowledge_ingest.py`, which contains the authoritative semantic checks for snippet grounding, full-chunk echo rejection, block-surface coverage, and strong-cue rejection of empty all-`other` packets.
- `cookimport/llm/codex_exec_runner.py`, which owns workspace mirroring, `current_task.json`, shell-policy enforcement, long-lived workspace sessions, and the current `workspace_outputs_stabilized` supervision rule.
- `cookimport/llm/phase_worker_runtime.py`, which owns worker assignment, per-task status tracking, shard-level proposal assembly, and promotion reporting.
- `tests/llm/test_codex_farm_knowledge_orchestrator.py`, `tests/llm/test_codex_exec_runner_workspace.py`, and `tests/llm/test_knowledge_output_ingest.py`, which already lock down parts of the current workspace and validator behavior and will need to grow significantly.

This plan deliberately aims past a single-stage helper upgrade. The intended end state is a repo-owned control loop where the worker sees only one current task at a time, the helper exposes one-step completion commands, the repository validates immediately, and the worker only terminates successfully when every assigned task packet has been validated or deterministically failed with an explicit surfaced reason.

## Plan of Work

The work should proceed in several large milestones, each independently useful and each pushing the runtime closer to repo-owned single-task authority.

The first milestone is to define the new worker contract in repo code before changing behavior. Extend the knowledge task runtime model so the worker root carries a skinny queue file, a richer current-task bundle, and a repo-written task-feedback surface. `assigned_tasks.json` should stop embedding the full payload and should instead become an ordered progress ledger containing only task ids, task sequence numbers, file paths, and compact flags such as whether the task is a strong-cue chunk. `current_task.json` should remain machine-readable but should now be paired with a human-readable `CURRENT_TASK.md` that explains, in plain language, what the worker owns, what the likely risk is, and what would make obvious failure patterns invalid. Add a repo-owned `task_feedback.json` or `CURRENT_TASK_FEEDBACK.md` surface that starts empty and is rewritten by the repo after each check or install attempt.

The second milestone is to extend `tools/knowledge_worker.py` from a utility kit into a true worker control surface. Keep the existing low-level verbs, but add higher-level verbs such as `current`, `complete-current`, `check-current`, `install-current`, `next`, and `explain-failure`. The exact naming can change during implementation, but the semantics should be stable: a cold-start worker should be able to complete the current task with one command sequence without touching `assigned_tasks.json`. `complete-current` should scaffold the current task to `scratch/current_task.json`, explain what remains for the model to fill in, and optionally refuse to proceed if the current task has not been loaded. `check-current` should run the authoritative validator and write the pass/fail result to the repo-owned feedback surface. `install-current` should only succeed when the current draft passes validation and should notify the runtime that the current task may advance. The helper should also expose a concise explanation for common semantic failures such as copied evidence surfaces and strong-cue all-`other` outputs.

The third milestone is the actual runtime redesign. In `cookimport/llm/codex_exec_runner.py` and the knowledge orchestrator, replace the current “worker owns all assigned tasks for one session” behavior with “worker owns one ordered lease of many tasks but only one task is active at a time.” The repository should mirror only the current task’s input as active authority while keeping future tasks discoverable only through the skinny queue. When the worker installs a valid current task output, the repo should atomically mark that task validated, rewrite `current_task.json`, rewrite `CURRENT_TASK.md`, rewrite the feedback surface, and advance to the next task without spawning a new model session if session reuse is still desired. If the current task fails validation, the worker should not advance, and the repo should surface the exact failure reason locally. The worker should never need to guess whether it may continue.

The fourth milestone is supervision hardening. `workspace_outputs_stabilized` must stop being a success condition for knowledge unless the runtime can also prove that every assigned task packet has been validated or intentionally terminally failed according to explicit repo rules. Update the phase-worker runtime so worker success means “all assigned task ids reached validated terminal state,” not “all expected output files stopped changing.” The task-status ledger should become the authoritative truth seam. The shard proposal builder should assemble parent shard results only from task packets that the repo has already individually accepted. The status surfaces should make it obvious whether a shard is incomplete because tasks were invalid, because the worker died, or because the runtime never advanced a current task.

The fifth milestone is prompt and shell-policy hardening. Rewrite the knowledge workspace prompt so it no longer inlines the full assigned packet-id list and no longer frames the job as “finish the whole queue” in a way that invites self-scheduling. The prompt should say, plainly, that the repository advances the current task and that the worker must not process future tasks before the current task passes the repo-owned checker. When `tools/knowledge_worker.py` is present, tighten the allowed local shell surface so repeated heredoc Python queue scripts and broad `assigned_tasks.json` spelunking become disallowed or at least suspicious enough to trigger early intervention. The precise policy needs prototyping because the worker still needs some local shell freedom, but the current permissive “bounded local shell script” rule is too easy to abuse.

The sixth milestone is validator ergonomics. Today the validator is strict but mostly invisible until late. Add short failure explanations to the helper and feedback surfaces so the worker sees domain-language explanations such as “this chunk carries a strong knowledge cue, so an all-`other` packet is not acceptable” or “your snippet body substantially copies the evidence quote; shorten the claim and keep the quote in evidence only.” This should reuse repo-owned metadata, not new model heuristics. If needed, add stage-local machine-readable reason codes and human-readable translations in `cookimport/llm/knowledge_workspace_tools.py`.

The seventh milestone is proof and optional promotion. Expand tests so they prove not only helper generation but the full single-task advancement contract, the skinny queue shape, the current-task sidecars, the no-advance-on-invalid rule, the new supervision semantics, and the prompt/policy assumptions. After local tests pass, the behavioral proof should eventually include a benchmark slice rerun, but only if the user explicitly approves token spend. If the knowledge redesign works well, decide whether to generalize the same runtime surfaces to recipe and line-role. The runtime abstraction should be designed with that possibility in mind even if the first implementation is knowledge-only.

## Validation and Acceptance

This work is accepted only when the redesigned runtime is observable in behavior, not merely in code shape.

Opening a generated knowledge worker root should show one authoritative current task, one short human-readable brief, one feedback surface for the current task, and a skinny queue that no longer encourages whole-batch spelunking. A novice should be able to see the approved loop without reverse-engineering the queue.

The helper must expose at least one high-level current-task completion flow that routes through repo-owned validation. A worker following the helper path should not need to invent a scheduler, a validator, or a final output path.

An invalid current task must fail locally with explicit repo-written feedback and must not advance the queue. A valid current task must advance the queue automatically or through one explicit helper step while keeping the same session alive when session reuse is enabled.

The phase runtime must no longer declare success just because all expected output files stabilized. For knowledge, worker success must imply that all assigned task packets were individually validated or explicitly reached a repo-defined terminal failure state that remains visible in the task ledger.

The saved tests must prove the new task-sidecar shapes, the no-advance-on-invalid rule, the advance-on-valid rule, the supervision rule change, and the helper’s current-task verbs. The existing validator-truth seam in `tests/llm/test_knowledge_output_ingest.py` must remain the canonical authority for semantic rejection behavior.

If the user later approves a live benchmark rerun, acceptance also requires behavioral evidence. Compared with the March 22, 2026 knowledge run, workers should invoke the repo-owned helper materially more often, should stop writing giant queue-wide heredoc schedulers, should surface task failures immediately, and should no longer collapse all parent shard proposals into `missing_owned_chunk_results` because a few task packets failed silently.

## Idempotence and Recovery

This redesign should be implemented additively and safely. Worker roots are already regenerated from scratch, so changing sidecar names, queue shapes, helper scripts, and prompt text is safe as long as the generating code and tests move together.

The repo-owned current-task state transitions must be idempotent. Re-running the worker-root generation should reproduce the same initial current-task sidecars. Re-running a successful `install-current` should either be a no-op or rewrite the same accepted output deterministically. Re-running `check-current` should never mutate final output state.

If the runtime redesign lands halfway and the prompt no longer matches the workspace surface, stop and repair the workspace contract before continuing. If shell-policy tightening causes legitimate helper usage to be killed, roll back the policy change temporarily but keep the new current-task contract work. If queue shrinking breaks existing tests or tooling that rely on embedded payloads in `assigned_tasks.json`, add an explicit migration layer or transitional compatibility seam rather than silently restoring the fat queue.

The benchmark rerun remains optional and must not happen without explicit user approval because it spends model tokens. Until that approval exists, local tests and saved-artifact replay are the required proof surface.

## Interfaces and Dependencies

In `cookimport/llm/knowledge_workspace_tools.py`, define stable helper functions that cover:

- loading the skinny queue and the authoritative current-task sidecars from the workspace root
- rendering a human-readable current-task brief and a human-readable current-task failure explanation
- scaffolding the current task draft without requiring the worker to resolve ids or output paths manually
- checking the current task draft through the same validator path used by repo ingest
- installing the current task draft only when it passes validation
- rendering a self-contained `tools/knowledge_worker.py` script body that includes the new higher-level verbs

In `cookimport/llm/codex_farm_knowledge_orchestrator.py`, the worker-root builder must write and maintain:

- a skinny `assigned_tasks.json` queue
- `current_task.json`
- `CURRENT_TASK.md` or an equivalent human-readable current-task brief
- `task_feedback.json` or `CURRENT_TASK_FEEDBACK.md`
- `OUTPUT_CONTRACT.md`
- `examples/`
- `tools/knowledge_worker.py`

In `cookimport/llm/codex_exec_runner.py` and any shared runtime state helpers, the runtime must expose:

- the current active task id
- a task advancement mechanism that only fires after repo-owned validation succeeds
- supervision state derived from validated task status, not merely file existence
- command-policy hooks that can distinguish helper usage from broad queue scripting

In `cookimport/llm/phase_worker_runtime.py` and the knowledge task-status code, the status ledger must be able to represent:

- current task pending
- current task failed validation with explicit reason codes
- current task validated and advanced
- worker completed because all assigned tasks validated
- worker incomplete because one or more task packets never reached validated terminal state

The semantic validator authority remains `cookimport/llm/codex_farm_knowledge_ingest.py`. Any current-task helper or feedback translation layer must call into that authority rather than copying its rules by hand.

Future-looking interface note: if this redesign works for knowledge, the shared workspace-worker runtime should eventually support the same current-task authority model for recipe and line-role. That future generalization is optional for the first implementation pass, but file names, helper verbs, and supervision abstractions should be chosen so they do not trap the repo in a knowledge-only naming scheme unless the domain genuinely requires it.
