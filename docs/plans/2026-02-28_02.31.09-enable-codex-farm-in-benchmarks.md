# Enable Codex Farm in Benchmarks (env-gated)

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, I can run `cookimport bench speed-run` and `cookimport bench quality-run` with optional Codex Farm recipe correction enabled. When enabled, all-method benchmark variant generation includes an additional Codex Farm recipe pipeline dimension (`llm_recipe_pipeline=codex-farm-3pass-v1`), allowing side-by-side comparison versus deterministic (`off`) behavior.

Codex Farm usage is gated behind an explicit environment variable (`COOKIMPORT_ALLOW_CODEX_FARM=1`) so it cannot be enabled accidentally by old saved settings files.

## Surprises & Discoveries

- Observation: All-method already threaded an `include_codex_farm` parameter through variant builders, but it was previously a no-op (“Reserved for future policy unlock.”).
  Evidence: `cookimport/cli.py:_build_all_method_variants` had `include_codex_farm` unused before this change.
- Observation: `_build_all_method_variants(...)` variant-count tests assume deterministic sweeps are opt-in; defaulting `include_deterministic_sweeps=True` causes count explosions (PDF/HTML would exceed 1/3 variants).
  Evidence: `tests/labelstudio/test_labelstudio_benchmark_helpers.py` expects 13 EPUB extractor variants and 1 PDF variant in the default call path.

## Decision Log

- Decision: Keep Codex Farm usage behind `COOKIMPORT_ALLOW_CODEX_FARM=1` rather than removing the guard entirely.
  Rationale: Prevents accidental enablement from persisted settings and keeps “LLM on/off” an explicit opt-in per run.
  Date/Author: 2026-02-28 / GPT-5.2
- Decision: Default all-method deterministic sweeps to opt-in (`include_deterministic_sweeps=False`) and let interactive flows pass the explicit user choice.
  Rationale: Keeps baseline all-method grids stable for extractor-only benchmarking and matches existing test contracts.
  Date/Author: 2026-02-28 / GPT-5.2

## Outcomes & Retrospective

- Recipe Codex Farm can be enabled for stage/pred-run/bench via `llm_recipe_pipeline=codex-farm-3pass-v1` when `COOKIMPORT_ALLOW_CODEX_FARM=1`.
- Bench `speed-run`/`quality-run` expose `--include-codex-farm` to include Codex Farm variants in all-method grids (still env-gated).
- All-method deterministic sweeps remain available, but are now opt-in via explicit call sites (interactive prompt passes the chosen value).

## Context and Orientation

In this repository, “Codex Farm” refers to a local subprocess runner (`cookimport/llm/codex_farm_runner.py`) that executes pipeline packs under `llm_pipelines/`. The recipe correction flow is a 3-pass orchestrator (`cookimport/llm/codex_farm_orchestrator.py`) that is selected by the run setting `llm_recipe_pipeline`.

Benchmarks are run via the `cookimport bench ...` CLI in `cookimport/cli.py` and include all-method grid searches and speed/quality suites implemented under `cookimport/bench/`.

Today, recipe Codex Farm is effectively disabled by:

1. CLI normalization rejecting non-`off` values for `--llm-recipe-pipeline` (`cookimport/cli.py` and `cookimport/labelstudio/ingest.py`).
2. Run settings migration forcibly coercing `llm_recipe_pipeline` back to `off` in `RunSettings.from_dict(...)` (`cookimport/config/run_settings.py`).
3. All-method variant generation ignoring `include_codex_farm` (`cookimport/cli.py:_build_all_method_variants`).

## Plan of Work

I will:

1. Change `RunSettings.from_dict(...)` so non-`off` `llm_recipe_pipeline` values are only honored when `COOKIMPORT_ALLOW_CODEX_FARM=1` is set; otherwise they are coerced to `off` with a warning.
2. Update CLI normalization for `--llm-recipe-pipeline` to accept `codex-farm-3pass-v1` only when `COOKIMPORT_ALLOW_CODEX_FARM=1` is set.
3. Implement `include_codex_farm` as a real dimension in `_build_all_method_variants(...)` by adding a Codex Farm variant for each existing variant (with a stable slug suffix) when requested.
4. Add `--include-codex-farm/--no-include-codex-farm` flags to `cookimport bench speed-run` and `cookimport bench quality-run`, threading the requested/effective values down so reports record both.
5. Update tests that currently assert Codex Farm remains disabled, and add a test that variant counts increase when codex is requested and the env gate is set.
6. Update the small number of docs/conventions that still describe recipe Codex Farm as permanently policy-locked off.

## Validation and Acceptance

- Unit tests pass for the changed gating behavior:
  - A test for `_resolve_all_method_codex_choice(True)` returns `False` without the env gate, and `True` with the env gate set.
  - A test for `_build_all_method_variants(..., include_codex_farm=True)` produces more variants than the same call with `include_codex_farm=False` when the env gate is set.

## Idempotence and Recovery

This change is safe to re-run. If Codex Farm is not installed, runs that actually execute the pipeline will fail in the codex-farm subprocess runner; the env gate and explicit flags help avoid accidental execution.
