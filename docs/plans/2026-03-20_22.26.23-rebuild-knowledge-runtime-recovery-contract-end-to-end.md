## Purpose / Big Picture

After this change, the non-recipe knowledge phase should behave like a bounded, inspectable stage instead of a hopeful conversation that sometimes leaves hundreds of raw files and no trustworthy answer. A user who runs a benchmark or a stage import should see packet-level progress move honestly, workers should either finish or fail closed for a clear reason, structured follow-ups should be rare and short, stale retries should not survive after useful work is over, and an interrupted run should still leave a complete stage-level evidence package.

The concrete proof is simple. In a no-token replay or fake-runner harness built from the saved Salt Fat artifacts, the knowledge stage should no longer sit at `task 0/10 | running 10` while doing packet work, should not explode into hundreds of low-yield follow-ups, should cancel stale repair and watchdog-retry calls once their parent packet or shard is terminal, and should always leave a stage root with manifests, telemetry, failure summaries, and packet outcome counts. With explicit user approval for live token spend, a real Salt Fat single-offline CodexFarm rerun should then either complete or fail closed without the earlier retry storm.

## Surprises & Discoveries

- Observation: the saved run did substantial work before appearing stuck, but raw worker-local output files were not the same thing as accepted stage results.
  Evidence: the knowledge root under `data/output/2026-03-20_21.44.20/.../knowledge/` contains `task_manifest.jsonl`, `shard_manifest.jsonl`, and many worker-local `out/*.json` files, but no `phase_manifest.json`, `telemetry.json`, `promotion_report.json`, `failures.json`, or `proposals/*`.

- Observation: the visible stage counter was fundamentally wrong for the way the stage now works.
  Evidence: the benchmark timeseries under `data/golden/benchmark-vs-golden/2026-03-20_21.44.20/.../processing_timeseries_prediction.jsonl` stayed at `task 0/10 | running 10` while the stage had already split into `485` task packets.

- Observation: main-worker completion status is not a reliable success signal.
  Evidence: workers `001` and `002` ended `completed` with `reason_code=process_exited_without_watchdog_intervention` and `final_agent_message_state=malformed`, while workers `004`, `007`, `009`, and `010` ended `completed` with `workspace_outputs_stabilized`.

- Observation: a worker can produce many files and still be catastrophically wrong in a way that should have been detected locally.
  Evidence: worker `001` wrote `49/49` top-level outputs, but every file was invalid JSON because the literal heredoc terminator `EOF` was written into the file body.

- Observation: several workers were not merely “off contract”; some were on contract with respect to the filesystem boundary while still being off task with respect to grounded knowledge review.
  Evidence: worker `007` explicitly planned to mark every chunk useful and every block `keep/knowledge` in one `jq` loop, and worker `004` reasoned from lane metadata rather than packet-local evidence in `events.jsonl`.

- Observation: the recovery layer currently spends far more than it rescues.
  Evidence: in the saved run there were `122` repair live statuses and `77` watchdog-retry live statuses, but only `5` repairs ended `repaired` and only `1` watchdog retry ended `validated`.

- Observation: follow-up calls can outlive their parent usefulness by many minutes.
  Evidence: worker `007` had already completed with `workspace_outputs_stabilized`, yet a task-level repair live status still sat in `running` with no fresh events for roughly eight minutes before the benchmark was killed.

- Observation: the current runtime already has many pieces of the eventual design, but they are disconnected and therefore misleading.
  Evidence: the orchestrator already records `watchdog_retry_status`, `repair_status`, `workspace_output_present_count`, `workspace_outputs_stabilized`, `process_exited_without_watchdog_intervention`, and malformed final-message state, but there is no single packet-level model that turns those facts into a trustworthy stage summary.

## Outcomes & Retrospective

This plan is not implemented yet. The current outcome is a single, self-contained execution document that replaces several narrower plans with one end-to-end program. The main lesson so far is that the failures were not isolated bugs. They came from one broken contract spread across worker prompts, workspace policy, local output handling, packet recovery, stage progress, and finalization. The implementation therefore has to be staged, but the final behavior must be judged as one coherent system.

## Context and Orientation

The non-recipe knowledge phase is the optional Codex-backed stage that reviews outside-recipe text and decides whether each packet contains reusable cooking knowledge or just other non-recipe text. In this repository, the core orchestrator for that stage lives in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. The stage currently launches long-lived workspace workers, validates outputs, may spawn structured `watchdog_retry` calls and `repair` calls, and tries to merge those results back into shard proposals.

A “workspace worker” in this repository is a Codex session started in a bounded local directory that already contains repo-written files like `worker_manifest.json`, `assigned_shards.json`, `assigned_tasks.json`, `hints/*.md`, and `in/*.json`. The worker is supposed to read only those files, make task-local judgments, and write one `out/<task_id>.json` file per task packet. The shared direct-exec transport and supervision code live in `cookimport/llm/codex_exec_runner.py` and `cookimport/llm/phase_worker_runtime.py`.

A “shard” is one larger ownership bundle assigned to one worker. A “task packet” is one smaller unit inside that shard, usually one chunk bundle that eventually becomes one `out/<task_id>.json` file. The saved Salt Fat run demonstrates why this distinction matters. The knowledge phase had ten shards but `485` task packets, so shard progress and task progress diverged badly.

A “watchdog retry” is a strict one-shot follow-up call launched after a main worker dies or is terminated. A “repair” is another strict one-shot follow-up call launched after a task output is close enough to salvage but not yet valid. These are intentionally narrower than the main workspace worker. They are not supposed to become a second open-ended runtime of their own.

A “packet terminal state” in this plan means the single authoritative outcome for one task packet. Every packet must end in exactly one of these categories:

- `validated`: accepted and usable without further work.
- `salvaged`: mechanically repaired by repo code without more model work, then accepted.
- `retry_recovered`: accepted after one watchdog retry.
- `repair_recovered`: accepted after one repair call.
- `failed_invalid_output`: structurally bad or semantically invalid after bounded recovery.
- `failed_boundary_violation`: main worker or follow-up broke the execution contract.
- `failed_budget_exhausted`: bounded recovery was spent without producing a valid packet.
- `failed_cancelled`: a packet was abandoned because its parent shard or stage became terminal or interrupted.

The current runtime does not have one explicit packet state machine like this. Instead, packet truth is spread across worker-local `out/*.json` files, `repair_status.json`, `watchdog_retry/status.json`, `live_status.json`, row telemetry, and scattered validation metadata. That is why the saved run could do a large amount of work and still leave no trustworthy phase package.

## Plan of Work

### Milestone 1: Build one explicit packet state model and one stage summary model

Start in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. Create one small private data model that represents the full packet lifecycle and another that represents stage-level rollups. The packet model should hold the packet id, parent shard id, current main-worker outcome, optional local-salvage outcome, optional watchdog-retry outcome, optional repair outcome, the final terminal state, and the concise reason that explains how the packet got there. The stage rollup should derive totals such as packets validated, salvaged, retry-recovered, repair-recovered, failed by class, cancelled, pending, and currently active in follow-up.

This model must become the single source of truth for progress and finalization. The existing worker-local files and live statuses still matter, but they should feed the packet model instead of serving as the summary surface themselves. A novice reading the final code should be able to answer “what happened to packet X?” by finding one packet-state path and “what happened to the stage?” by finding one rollup path.

Acceptance for this milestone is a new targeted test in `tests/llm/test_codex_farm_knowledge_orchestrator.py` that constructs a small fake packet set and proves the stage rollup classifies packet outcomes deterministically from mixed main-worker, retry, and repair facts.

### Milestone 2: Rewrite the worker contract so the good path is narrow and obvious

Still in `cookimport/llm/codex_exec_runner.py`, `cookimport/llm/phase_worker_runtime.py`, `cookimport/llm/knowledge_prompt_builder.py`, and `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md`, rewrite the worker-facing contract so bounded local execution is easier than off-task improvisation.

The worker must be told, in plain language and in the repo-owned manifest, to do all scratch work under an explicit workspace-local `scratch/` directory, to read the named `hints/*.md` and `in/*.json` files directly, to stop once all owned output files exist, and to avoid interpreters and metadata-mining detours. Add concrete “good examples” and “bad examples” to the manifest note text so the model sees the exact patterns the runtime expects.

Then adjust the shared command classification and boundary logic in `cookimport/llm/codex_exec_runner.py` so the rules match the manifest. Local shell automation under the worker root, including `jq`, `cat`, redirects, and heredocs into `scratch/` or `out/`, should stay on contract. Inline interpreters like `python - <<'PY'`, writes to `/tmp`, parent-directory traversal, repo/network commands, and repeated no-output probing should remain forbidden. The goal is not to broaden the runtime into a sandbox free-for-all. The goal is to make the intended path explicit and consistent.

Acceptance for this milestone is a fail-before / pass-after test slice in `tests/llm/test_codex_exec_runner.py` that proves workspace-local shell helpers are allowed, `/tmp` and inline interpreters are still rejected, and the worker manifest text advertises the same rules the classifier enforces.

### Milestone 3: Validate outputs as soon as they appear and salvage cheap failures locally

The saved run shows that mechanical output failures should not become LLM repair storms. Add an early output-ingest seam in `cookimport/llm/codex_farm_knowledge_orchestrator.py` that inspects newly appeared `out/<task_id>.json` files as soon as they become visible in the worker root or the live mirrored workspace. If a file is unparsable JSON, immediately record a packet-level invalid-output fact. If the file is parseable but looks mechanically fixable, run deterministic local salvage before launching any structured follow-up.

The first local-salvage pass should stay deliberately narrow. It should only attempt fixes that are obviously mechanical and proven by tests, such as:

- stripping a literal trailing heredoc marker like `EOF`;
- trimming trailing shell-noise lines after a valid JSON object;
- re-serializing JSON that parses but has incidental formatting drift;
- rejecting empty or truncated files early;
- applying cheap invariant normalization when the structure is almost valid and the fix is deterministic.

Do not invent speculative semantic repairs here. This milestone is about catching worker `001` and worker `002` style failures cheaply and deterministically.

The local-salvage result must update the packet state machine. A packet that is fixed locally becomes `salvaged`, not `repair_recovered`. A packet that cannot be fixed locally should move into bounded recovery with an explicit reason like `invalid_json_trailing_heredoc` or `schema_invalid_non_salvageable`.

Acceptance for this milestone is a new focused test in `tests/llm/test_codex_farm_knowledge_orchestrator.py` that reproduces the worker `001` heredoc failure and proves the stage either salvages it locally or classifies it deterministically without launching dozens of repairs.

### Milestone 4: Replace open-ended recovery with strict budgets and one path per failure class

Now rebuild the follow-up policy in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. The current code launches `watchdog_retry` and `repair` too freely and independently, which is why the saved run reached `199` structured follow-up calls. Replace that behavior with a small, explicit recovery budget model.

At minimum, implement these rules:

- one watchdog retry per task packet;
- one repair attempt per task packet;
- one local-salvage attempt per task packet, before any LLM follow-up;
- one shard-level watchdog-retry budget cap, for example three task retries per shard before the shard is considered poisoned;
- one shard-level repair budget cap, for example the smaller of three tasks or ten percent of the shard’s packet count.

Then add a repeated-failure circuit breaker. If a shard spends its early recovery budget on the same deterministic failure class, such as repeated `schema_invalid`, repeated `block_decision_coverage_mismatch`, repeated `watchdog_command_execution_forbidden`, or repeated malformed local output, stop launching more follow-ups and fail the remaining affected packets closed. The saved run proved that discovering the same root cause twenty times is not useful.

The runtime must also choose one recovery path per failure class. Examples:

- If the main worker died before producing output, allow one watchdog retry.
- If output exists but is malformed or almost valid, use local salvage first and then one repair if needed.
- If output coverage is already complete and the worker died while self-auditing, do not reopen a full retry tree. Validate what exists and move directly to bounded repair or fail closed.

Acceptance for this milestone is a new fake-runner test group in `tests/llm/test_codex_farm_knowledge_orchestrator.py` showing that repeated identical failures trip a shard-level circuit breaker and that the number of follow-up calls is capped deterministically.

### Milestone 5: Cancel stale follow-ups and make terminal packets stay terminal

The runtime currently leaves follow-up calls alive after the useful work is over. Add explicit stale-follow-up and superseded-follow-up handling in `cookimport/llm/codex_farm_knowledge_orchestrator.py` and, where necessary, `cookimport/llm/codex_exec_runner.py`.

If a packet reaches any terminal accepted or terminal failed state, all older or parallel follow-ups for that packet must be marked superseded or cancelled. If a shard becomes terminal, all remaining child follow-ups under that shard must be reaped. If the stage is interrupted, all still-running follow-up calls must be finalized with a cancellation reason instead of being left as eternal `running`.

Also add bounded inactivity handling for structured follow-ups that are still technically running. The saved run had follow-ups with no new events for minutes. Add deterministic rules such as a hard elapsed-time cap and a no-new-events cap for structured `repair` and `watchdog_retry` calls. A stale call should end in an explicit terminal packet state, not in a dangling live status.

Acceptance for this milestone is a regression test that reproduces the saved worker `007` stale repair shape and proves the follow-up is finalized as cancelled or failed once the parent shard has already completed.

### Milestone 6: Detect poisoned workers and poisoned shards early

Some failures are not isolated packet misses. The saved run showed worker-wide or shard-wide bad behavior:

- worker `001` generated invalid heredoc-shaped files for the whole assignment;
- worker `007` planned to label everything as useful kept knowledge;
- worker `004` leaned on metadata instead of grounded review.

Add deterministic poison heuristics in `cookimport/llm/codex_farm_knowledge_orchestrator.py` and, if needed, `cookimport/llm/codex_farm_knowledge_ingest.py` to catch obvious assignment-wide corruption patterns. Examples include:

- many outputs from one worker failing with the same mechanical shape;
- repeated identical block-decision vocabulary across almost every packet from one worker;
- blanket `u=true` or blanket `keep/knowledge` patterns with no meaningful variation where the packet inputs clearly differ;
- many packets from one worker failing the same semantic invariant.

When a worker or shard is marked poisoned, stop doing packet-by-packet rediscovery. Either rerun the whole shard once under a tighter contract if that is justified by tests, or fail the remaining packets closed and record the poison reason in the stage summary. The important point is to stop burning recovery budget on a root cause the runtime already understands.

Acceptance for this milestone is a deterministic test using synthetic or saved worker-local outputs that proves the runtime can classify a poisoned assignment and stop expanding packet-level churn.

### Milestone 7: Rebuild progress and observability around packet truth

Once packet states and recovery budgets are real, rebuild progress and observability around them in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, `cookimport/cli.py`, `cookimport/runs/stage_observability.py`, `cookimport/llm/prompt_budget.py`, and `cookimport/llm/prompt_artifacts.py`.

The visible task counter for knowledge must use packet totals and packet terminal counts, not shard totals. Worker rows must show shard ownership plus packet progress and current follow-up state. Detail lines should show shard totals, follow-up counts, and failure-category counts. The benchmark should never again report `task 0/10 | running 10` during a `485`-packet stage.

Add compact stage-level observability metrics that remove the need for future manual file counting. At minimum, expose:

- packets total, validated, salvaged, retry-recovered, repair-recovered, failed by class, cancelled;
- main workers completed cleanly, completed malformed, watchdog-killed;
- structured follow-up calls started, completed, failed, cancelled, stale-killed, skipped by budget, skipped by poison rule;
- invalid local output count;
- per-shard dominant failure signature;
- per-shard and whole-stage recovery budget consumption.

Acceptance for this milestone is updated deterministic progress tests and stage-summary tests that assert packet-based counters and explicit follow-up counts.

### Milestone 8: Finalize stage artifacts on success, failure, and interruption

Move the knowledge phase to always write a stage-level package in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, and wire that package into benchmark interruption handling in `cookimport/cli.py` and `cookimport/labelstudio/ingest.py`.

On every terminal path, whether success, partial failure, watchdog collapse, or operator interruption, the knowledge root must contain:

- `phase_manifest.json`
- `telemetry.json`
- `promotion_report.json`
- `failures.json`
- packet outcome summaries
- any proposals that were accepted
- explicit stage-level interruption or failure reason when applicable

The benchmark root should continue to write `benchmark_status.json` and `partial_benchmark_summary.json`, but those files should now point at a finished stage-level knowledge package instead of a half-empty raw worker tree.

This is also the milestone that must make Ctrl+C behavior honest. The benchmark does not have to implement a complex global signal framework, but it must stop spawning new knowledge follow-ups, finalize packet states as cancelled where needed, write stage and benchmark summaries, and exit cleanly.

Acceptance for this milestone is a targeted interruption test that proves both the benchmark root and the knowledge root contain readable partial-finalization artifacts after interruption.

## Validation and Acceptance

This plan is complete only when all of the following are true.

First, the knowledge runtime has one authoritative packet state model and one stage rollup model. A novice can inspect the stage package and tell what happened to every packet without manually traversing worker trees.

Second, the main workspace-worker contract is explicit and testable. Safe local shell automation inside the worker root is allowed, but `/tmp`, parent traversal, network tools, and inline interpreters remain forbidden. The worker manifest and the command classifier say the same thing.

Third, malformed local output is handled locally first. Mechanical problems like trailing heredoc markers do not cause repair storms. A saved worker `001` style output should be either salvaged deterministically or classified immediately as a specific invalid-output failure.

Fourth, structured follow-ups are bounded and honest. The runtime must not create unbounded repair and watchdog-retry trees. Repeated identical failures must trip a circuit breaker, and the total follow-up count for a shard must obey explicit budgets.

Fifth, stale child calls do not linger after the useful work is over. A worker `007` style stale repair must be cancelled or finalized once the parent shard is already terminal.

Sixth, poisoned assignments are recognized early. The runtime must not spend dozens of packet-level follow-ups rediscovering a worker-wide bad pattern such as repeated heredoc corruption or blanket keep/knowledge automation.

Seventh, the benchmark UI tells the truth. The knowledge phase must report packet counts, not shard counts, and the worker rows must explain shard ownership plus local packet progress or current follow-up state.

Eighth, stage-level and benchmark-level interruption artifacts are always written. After Ctrl+C or `KeyboardInterrupt`, the knowledge root and the benchmark root must both contain readable partial-finalization files that say interruption occurred and summarize known packet outcomes.

Ninth, the March 20 Salt Fat postmortem shapes are covered by deterministic tests. The primary acceptance path must not depend on “hopefully the live rerun looks better.”

## Idempotence and Recovery

This work should be implemented additively and can be retried safely if a milestone stalls. The first milestone is intentionally internal: it creates a packet-state and stage-summary model without yet changing all behavior. That lets later milestones switch one decision seam at a time to the new model instead of forcing a single risky rewrite.

If the worker-contract rewrite proves too permissive in shared code, narrow the shared classifier again and keep any knowledge-specific allowances local to the knowledge phase until tests demonstrate they are safe elsewhere. If the local-salvage layer starts fixing cases that should instead fail, cut it back to purely mechanical transformations and record that decision in the `Decision Log`.

If the poison-worker detection turns out too aggressive, prefer false negatives over false positives at first. A poisoned assignment detector that only catches the obvious saved-run shapes is acceptable in the first pass. The important thing is to stop the worst forms of pointless churn, not to build a speculative classifier.

If full Ctrl+C propagation proves too invasive, the minimum acceptable fallback is still strict: stop spawning new knowledge follow-ups, finalize the current packet states as interrupted or cancelled, write stage and benchmark summaries, and exit. Do not silently leave the knowledge stage with only worker-local debris.

Do not modify the saved March 20 artifact trees in place. Tests must copy or synthesize fixtures from them.
