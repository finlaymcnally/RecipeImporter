This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

After this work, CodexFarm should have one coherent reliability strategy instead of several partially overlapping fix lists. A contributor should be able to improve the current `codex-farm-3pass-v1` path without losing sight of the larger architectural question, then make a measured decision about whether a new merged pass2/pass3 repair path is actually better.

The user-visible proof is not a prettier design document. The proof is that the same weak benchmark slice rooted at `data/golden/benchmark-vs-golden/2026-03-06_15.22.11` becomes easier to explain and less self-damaging. Title and boundary mistakes should drop, broken pass1/pass2 transport should become auditable, pass3 should stop spending heavily on rows that already failed structural checks, and the repo should end with benchmark evidence strong enough to answer a real product question: should the project keep tightening the current three-pass path, or promote a new merged repair pipeline.

  Evidence: all five source documents call out some combination of title over-promotion, pass1/pass2 transport loss, pass3 over-routing, and the need to treat empty or placeholder outputs as failure rather than success.

## Decision Log

- Decision: treat this work as one staged program with explicit decision gates instead of choosing a winner between “incremental repair” and “architectural redesign” up front.
  Rationale: the reports disagree on sequence, not fundamentals. Instrumentation and no-regrets reliability repairs improve either future, and they provide a cleaner baseline for judging a merged-stage prototype.
  Date/Author: 2026-03-13 / Codex

- Decision: keep the current `codex-farm-3pass-v1` path as the control until benchmark evidence proves a replacement path is better.
  Rationale: the repo already has tests, manifests, and benchmark wiring for the current pipeline. Replacing it immediately would destroy the baseline needed to judge the redesign honestly.
  Date/Author: 2026-03-13 / Codex

- Decision: include upstream title and boundary guardrails in this plan even though the new pass2/pass3 report narrows scope.
  Rationale: the broader report set shows that pass2/pass3 reliability is downstream of bad structural decisions. A merged repair stage cannot fully compensate for title/header promotion or recipe-window corruption created earlier.
  Date/Author: 2026-03-13 / Codex

- Decision: require one canonical internal recipe representation before any full promotion of a merged pass2/pass3 stage.
  Rationale: several reports point out that maintaining multiple “final” artifacts causes drift. The redesign should emit one canonical object and derive schema.org or other export shapes deterministically after that.
  Date/Author: 2026-03-13 / Codex

- Decision: use disagreement-driven structural gating as the main review and escalation trigger, not raw confidence alone.
  Rationale: the reports show that many of the expensive mistakes are high-confidence structural errors. The plan should therefore prioritize disagreements between deterministic or baseline structure and Codex output, impossible span shapes, empty-but-parseable payloads, and similar invariant violations over generic low-confidence filtering.
  Date/Author: 2026-03-13 / Codex

- Decision: require monotonic fallback and recovery behavior.
  Rationale: a degraded path should preserve the best-known title, ingredient candidates, instruction candidates, warning codes, and provenance rather than collapsing a partially recoverable row into a worse artifact. Fallback is acceptable only when it explains what was lost and does not silently discard recoverable evidence.
  Date/Author: 2026-03-13 / Codex

- Decision: judge success with the same March 6 three-book slice and a fresh follow-up packet using the same asks.
  Rationale: that slice is the shared evidence base across all five documents. Switching evaluation targets midstream would make the plan easier to argue about and harder to falsify.

## Context and Orientation

CodexFarm recipe handling currently lives across three areas of the repository.

The first area is upstream structure and staging. `cookimport/staging/stage_block_predictions.py` and `cookimport/staging/draft_v1.py` turn recipe candidates plus line-role evidence into recipe titles, ingredient lines, instruction lines, and draft objects. The March 13 report cluster repeatedly identifies this layer as a source of false title promotion and bad recipe boundary assumptions.

The second area is the active three-pass CodexFarm recipe orchestrator. `cookimport/llm/codex_farm_orchestrator.py` coordinates pass1 recipe chunking, pass2 schema.org-style extraction, and pass3 final-draft or mapping work. `cookimport/llm/codex_farm_transport.py` builds the authoritative pass1 -> pass2 payload from block spans. `cookimport/llm/codex_farm_contracts.py` validates and repairs pass contracts. `cookimport/llm/evidence_normalizer.py` does additive evidence cleanup. `llm_pipelines/prompts/recipe.schemaorg.compact.v1.prompt.md` and `llm_pipelines/prompts/recipe.final.compact.v1.prompt.md` define the compact pass2 and pass3 prompt contracts used by the current pipeline.

The third area is benchmark proof. The canonical baseline for this work is the March 6 benchmark root at `data/golden/benchmark-vs-golden/2026-03-06_15.22.11`. The weak books called out across the documents are `saltfatacidheatcutdown`, `seaandsmokecutdown`, and `thefoodlabcutdown`. Fresh benchmark runs and fresh follow-up packets built from those same books are the acceptance surface for this plan.

The source documents that motivated this plan are not redundant. `docs/plans/2026-03-13_21.39.42-codexfarm-top-5-followup-fixes.md` concentrates on title guardrails, pass1 boundary repair, and pass3 routing discipline. `docs/plans/2026-03-13_21.56.30-pass2-pass3-benchmark-reliability-repair.md` concentrates on pass1 overlap splits, pass2 contract brittleness, and partial-bundle pass3 routing. `docs/reports/codexfarm_fix_report.md` and `docs/reports/codexfarm_fix_report (1).md` diagnose the same benchmark failures at different levels of abstraction. `docs/reports/pass2_pass3_fix_report.md` pushes the larger architectural claim that the pass2 -> pass3 seam may now cost more than it saves. This plan assumes all of those documents are directionally correct and resolves their tension by sequencing them.

Throughout this plan, “structural success” means more than “the JSON parsed.” A stage is structurally successful only when it produced the minimum recipe information the next stage needs without obvious contradictions, such as an empty mapping on a row that clearly has mappable steps, a placeholder title, or a transport payload that lost rows from the authoritative source span. Confidence scores are secondary evidence only; disagreements with deterministic structure, impossible span geometry, or empty-but-parseable outputs outrank low-confidence heuristics when deciding whether a row needs review, fallback, or recovery.

## Plan of Work

The work proceeds in four milestones. The first two improve the current `codex-farm-3pass-v1` path and make it easier to reason about. The third milestone uses that cleaner baseline to prototype a merged pass2/pass3 repair path. The fourth milestone decides which path survives, using the same benchmark slice and follow-up asks that motivated this whole effort.

Milestone 1 establishes hard observability and success criteria. Start in `cookimport/llm/codex_farm_transport.py`, `cookimport/llm/codex_farm_orchestrator.py`, and `cookimport/llm/codex_farm_contracts.py`. Audit inclusive-versus-exclusive span handling and make the transport verifier explicit enough to explain whether a row failed because of bad span selection, dropped rows, control-character corruption, malformed auxiliary metadata, or an actual model decision. Extend manifest diagnostics so “empty but parseable” outcomes become machine-visible warnings or failures, not quiet `ok` rows. In the Codex execution wrapper path described in `docs/10-llm/10-llm_README.md`, verify that extraction and repair calls run in a non-agentic structured-output mode. If logs or prompt artifacts still show tool-seeking or docs-list behavior, treat that as a runtime defect and fix it before larger prompt or architecture changes.

Milestone 2 hardens the current three-pass path where all reports overlap. In `cookimport/staging/stage_block_predictions.py` and `cookimport/staging/draft_v1.py`, add title and boundary guardrails so recipe titles do not promote from section headers, chapter prose, or off-span transitions without recipe-local evidence. Make label ownership explicit instead of implicit: keep deterministic or baseline-first authority for syntax-heavy labels such as `TIME_LINE`, `YIELD_LINE`, strong `RECIPE_NOTES` patterns, section-header markers, and high-confidence `INGREDIENT_LINE` cases unless Codex has unusually strong contrary evidence. In `cookimport/llm/codex_farm_orchestrator.py`, repair pass1 overlap arbitration and boundary/window selection so duplicate or near-duplicate spans do not become head-only and tail-only half-recipes, and record disagreement reasons that explain why Codex output was trusted or vetoed. In `cookimport/llm/codex_farm_contracts.py`, downgrade malformed auxiliary `field_evidence` to a recoverable issue, sanitize corrupted extractive arrays, and preserve normal Unicode. Tighten pass3 routing so structurally partial or low-value rows do not consume live pass3 calls merely because the payload parses. Add an explicit mapping-status or equivalent warning contract so empty mapping becomes either “not needed” with evidence or “unclear” with a named reason, never a silent success. Add a source-grounded recovery path for rows that are degraded because current intermediates are unreliable but the selected source span still looks recipe-like, and require that any fallback preserve the best-known title, ingredient candidates, instruction candidates, and warning provenance instead of collapsing destructively.

Milestone 3 prototypes the architectural move that only one report states directly but several imply: converge on one canonical recipe object and reduce seam drift. Add a new pipeline id, for example `codex-farm-2stage-repair-v1`, in the run-settings and pipeline-selection surfaces documented in `docs/10-llm/10-llm_README.md`. Keep pass1 or whatever boundary-selection stage proves best, but replace the current pass2 -> pass3 handoff with one source-grounded assembly-and-repair stage that receives the authoritative source span plus current draft hints and emits one canonical recipe object, optional ingredient-step mapping, and machine-readable warnings. Derive schema.org or other downstream export shapes deterministically from that canonical object after the merged stage. This prototype must remain behind a separate pipeline id so the repaired 3-pass path continues to serve as the control.

Milestone 4 decides what to keep. Run the same three-book canonical-text benchmark slice against the repaired `codex-farm-3pass-v1` path and the new merged prototype. Build fresh compare artifacts and a fresh follow-up packet using the same ask structure as the March 6 investigation. Promote the merged pipeline only if it is measurably better on the same failure families without unacceptable cost or new regressions. If the merged stage is only partially better, keep the repaired 3-pass path as the default and backport the winning ideas instead of forcing a premature architecture switch.

## Milestones

### Milestone 1: Make Failures Observable and Define Structural Success

At the end of this milestone, a contributor should be able to inspect a bad benchmark row and say whether the problem began in span selection, transport, contract loading, runtime mode, or model reasoning. The code should stop calling obviously broken outputs “ok” just because they parsed.

Edit `cookimport/llm/codex_farm_transport.py`, `cookimport/llm/codex_farm_orchestrator.py`, `cookimport/llm/codex_farm_contracts.py`, and the Codex execution wrapper path described in `docs/10-llm/10-llm_README.md`. Add focused tests in the existing orchestrator, transport, and contract suites for span semantics, transport audits, structural-success classification, and runtime-mode configuration where that can be asserted cheaply.

Acceptance is that failing rows expose named reasons such as transport mismatch, truncated recipe head, truncated recipe tail, malformed auxiliary metadata, corrupted extractive text, placeholder title, or empty mapping with insufficient justification. Acceptance also requires evidence that extraction and repair calls no longer run in an agentic/tool-seeking mode, and that review triggers can be explained in structural terms rather than only by low confidence.

### Milestone 2: Harden the Current 3-Pass Path End to End

At the end of this milestone, the current `codex-farm-3pass-v1` path should stop manufacturing the exact March 6 failure families the reports agree on. Title/header promotion should be rarer, pass1 overlap splits should stop creating half-recipes, malformed pass2 auxiliary fields should no longer kill otherwise useful rows, and pass3 should run less often on rows that were already structurally broken.

Edit `cookimport/staging/stage_block_predictions.py`, `cookimport/staging/draft_v1.py`, `cookimport/llm/codex_farm_orchestrator.py`, `cookimport/llm/codex_farm_contracts.py`, `llm_pipelines/prompts/recipe.schemaorg.compact.v1.prompt.md`, and `llm_pipelines/prompts/recipe.final.compact.v1.prompt.md`. Extend the existing staging, orchestrator, and contract tests with targeted fixtures for false-title regression, duplicate span overlap, malformed `field_evidence`, control-byte cleanup, no-slug-title synthesis, selective pass3 routing, label-ownership arbitration, and monotonic fallback preservation.

Acceptance is behavior-first. The targeted unit and scenario tests should pass, and a one-book rerun on `saltfatacidheatcutdown` should already show that `c10`-style auxiliary failures, `c15`/`c16`-style split bundles, and silent empty-mapping pass3 rows are gone or materially reduced. Acceptance also requires that `TIME_LINE` and strong `RECIPE_NOTES` regressions do not recur simply because Codex was allowed to override cleaner deterministic or baseline labels without strong local evidence.

### Milestone 3: Prototype a Merged Pass2/Pass3 Repair Path Behind a New Pipeline Id

At the end of this milestone, the repo should have a benchmark-runnable prototype that answers the larger architectural claim without replacing the current default path. A contributor should be able to run the same benchmark book through both the repaired 3-pass path and the merged-stage prototype.

Add a new pipeline id and the required run-settings wiring, then implement a new source-grounded assembly-and-repair stage in the LLM surface that receives the authoritative source span plus draft hints and emits one canonical recipe object plus optional mapping and warnings. Keep pass1 or equivalent boundary selection separate. Add deterministic derivation code so schema.org and other export views come from the canonical object instead of being maintained as separate final truths. Add new tests for canonical derivation, merged-stage warnings, partial mapping semantics, and degraded-source recovery.

Acceptance is that the new pipeline runs end to end under benchmark control, writes auditable artifacts, and can be compared directly against the repaired `codex-farm-3pass-v1` path on the same books.

### Milestone 4: Decide With Benchmark Evidence, Not Preference

At the end of this milestone, the plan should answer the product decision that all five source documents were circling: is the best next move “keep tightening the current 3-pass pipeline” or “promote the merged repair architecture”?

Run fresh canonical-text benchmark evaluations for `saltfatacidheatcutdown`, `seaandsmokecutdown`, and `thefoodlabcutdown` for both candidate paths. Compare each run against the March 6 vanilla baseline and against each other where useful. Build a fresh follow-up packet using the same asks as the original investigation and update this plan with the measured deltas.

Acceptance is a written promotion decision supported by benchmark and follow-up evidence. The surviving path must be better on the shared failure families and must explain its cost profile honestly, especially around pass3 or its replacement.

## Validation and Acceptance

Validation is layered because the plan deliberately compares two futures instead of assuming one.

First, focused tests must prove that structural-success checks, title guardrails, overlap arbitration, label-ownership arbitration, contract hardening, text sanitation, monotonic fallback preservation, and pass3 routing decisions behave as intended on targeted regressions. The new tests should fail before the change and pass after it.

Second, a Salt/Fat/Acid/Heat rerun must show that the repaired current path no longer reproduces the clearest March 6 failures. At minimum, malformed auxiliary `field_evidence` must not hard-fail otherwise useful rows, duplicate pass1 spans must not become head-only and tail-only half-recipes, empty or placeholder pass3 outcomes must stop being counted as normal success, and disagreement-driven structural vetoes must protect labels such as `TIME_LINE` and `RECIPE_NOTES` when Codex does not have strong contrary evidence.

Third, the merged-stage prototype must prove that it actually reduces seam drift rather than only moving complexity around. Acceptance requires one canonical internal recipe object, deterministic derivation of export shapes, and auditable warnings when mapping is partial or omitted.

Fourth, the final decision must be benchmark-backed. The promoted path must do better on the same failure families identified in the March 6 packet and must justify its runtime and token cost honestly. If the merged-stage prototype is more elegant but not clearly better, this plan does not permit promoting it on style alone.

## Idempotence and Recovery

The plan is deliberately additive at first. Milestones 1 and 2 extend diagnostics and repair the active path without deleting the baseline pipeline. That means a contributor can rerun focused tests and one-book benchmarks repeatedly while keeping comparison evidence intact.

Milestone 3 must also be additive. The merged-stage design belongs behind a new pipeline id so implementation can be retried without destabilizing the repaired 3-pass control. If the prototype fails or underperforms, recovery means leaving the new pipeline id disabled or experimental and keeping the current path as the default while backporting any clearly winning ideas.

Recovery behavior inside either path must be monotonic. When a row degrades, retries, or falls back, the system should preserve the best-known title, raw ingredient candidates, raw instruction candidates, warning codes, and provenance needed to explain the degradation. A fallback that silently discards recoverable evidence is a defect, not an acceptable simplification.

Benchmark reruns must use fresh timestamped roots and must never overwrite the March 6 evidence root. Follow-up packets should always be regenerated from the new bundle template rather than copied blindly, because bundle hashes are part of the validation contract.
