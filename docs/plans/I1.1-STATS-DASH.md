# Lifetime Stats Dashboard

Static, offline HTML dashboard summarizing stage/import throughput and golden-set benchmark trends over time.

## What It Does

`cookimport stats-dashboard` scans existing on-disk metric surfaces, aggregates them, and writes a self-contained HTML dashboard to `data/output/.history/dashboard/`. Open `index.html` in any browser via `file://` — no server, no network, no CDN.

The dashboard answers:
* How fast are imports trending? (sec/recipe over time)
* What are the slowest runs and why? (sorted table with artifact links)
* How are benchmark scores moving? (precision/recall trend)
* Are there regressions in quality metrics? (per-label breakdowns)

### Non-goals (v1)

* No web server or external dependencies.
* Read-only — never re-runs imports or benchmarks.
* Does not parse full recipe outputs; only reads summarized metric surfaces.

## Architecture

Three modules in `cookimport/analytics/`, following a collect → schema → render pipeline:

    cookimport stats-dashboard
          │
          ▼
    dashboard_collect.py     ← reads CSV + JSON from disk
          │
          ▼
    dashboard_schema.py      ← Pydantic v2 models (DashboardData)
          │
          ▼
    dashboard_render.py      ← writes index.html + assets/

### Module Responsibilities

**`dashboard_schema.py`** — Data contracts (Pydantic v2).
* `DashboardData` — top-level container with `schema_version`, `stage_records`, `benchmark_records`, `summary`, `collector_warnings`.
* `StageRecord` — one imported source file (from CSV row or conversion report JSON).
* `BenchmarkRecord` — one eval run (from `eval_report.json`, optionally enriched by `coverage.json` / `manifest.json`).
* `DashboardSummary` — pre-aggregated totals for the header.
* `RunCategory` enum: `stage_import`, `labelstudio_import`, `benchmark_eval`, `benchmark_prediction`.
* Missing numeric fields are `None`, never invented zeros. Charts omit `None` points.

**`dashboard_collect.py`** — Read-only collectors.
* `collect_dashboard_data(output_root, golden_root, since_days, scan_reports) -> DashboardData`
* Stage collector prefers `performance_history.csv`; falls back to scanning `*.excel_import_report.json` when CSV is missing or `--scan-reports` is passed. Deduplicates by `(run_timestamp, file_name)`.
* Benchmark collector globs for `eval_report.json` under `golden_root`, enriches from sibling `coverage.json` / `manifest.json` when present.
* Resilience: malformed JSON is skipped with a warning appended to `collector_warnings`. Un-parseable timestamps sort last.

**`dashboard_render.py`** — Static HTML generator.
* Writes four files: `index.html`, `assets/dashboard_data.json`, `assets/dashboard.js`, `assets/style.css`.
* JS loads the JSON and renders SVG sparklines + tables client-side. Zero external dependencies.

### CLI Command

Registered in `cookimport/cli.py` as `stats-dashboard` (Typer):

    cookimport stats-dashboard [OPTIONS]

| Flag | Default | Purpose |
|---|---|---|
| `--output-root` | `data/output` | Root of staged outputs |
| `--golden-root` | `data/golden` | Root of golden-set / benchmark data |
| `--out-dir` | `data/output/.history/dashboard` | Where dashboard files are written |
| `--open` | off | Open `index.html` in default browser |
| `--since-days N` | all | Only include runs from last N days |
| `--scan-reports` | off | Force scanning individual report JSONs |

## Data Sources

The dashboard reads (never writes) from three metric surfaces:

| Source | Location | Used for |
|---|---|---|
| Performance history CSV | `data/output/.history/performance_history.csv` | Primary stage/import records **and** benchmark eval rows (distinguished by `run_category` column) |
| Conversion report JSON | `data/output/<timestamp>/*.excel_import_report.json` | Fallback stage records when CSV missing/incomplete |
| Eval report JSON | `data/golden/eval-vs-pipeline/*/eval_report.json` | Benchmark precision/recall/boundary metrics (JSON source) |
| Coverage JSON (optional) | `<eval_dir>/coverage.json` | Character-level coverage ratio |
| Manifest JSON (optional) | `<eval_dir>/manifest.json` | Task counts, source file name |

Benchmark rows are appended to the CSV by `labelstudio-eval`, `labelstudio-benchmark`, and `bench run` via `append_benchmark_csv()` in `perf_report.py`. The `run_category` column (`stage_import`, `benchmark_eval`, `benchmark_prediction`) is the discriminator. The collector reads both stage and benchmark rows from the CSV and merges CSV-sourced benchmarks with JSON-sourced benchmarks, deduplicating by `(run_timestamp, artifact_dir)`.

## Key Design Decisions

**Prefer small metric surfaces over scanning large outputs.**
The CSV and `eval_report.json` are compact, already-normalized summaries. Scanning full recipe outputs would be slow and fragile. The `--scan-reports` flag exists as a fallback but isn't the default.

**Run categories prevent double-counting.**
Label Studio workflows produce stage-like outputs under `data/output/.../labelstudio/...` and prediction artifacts under `<eval_dir>/prediction-run/`. Without categorization these would inflate lifetime import stats. Four categories (`stage_import`, `labelstudio_import`, `benchmark_eval`, `benchmark_prediction`) keep them separated. The UI defaults to showing only `stage_import` + `benchmark_eval`; others are opt-in via checkboxes.

**Collectors are strictly read-only.**
Dashboard generation never modifies `data/output` or `data/golden`. It only writes inside `--out-dir`. Safe to re-run at any time.

**Missing data is None, not zero.**
Charts skip missing points rather than plotting misleading zeros. Derived metrics (e.g., `per_recipe_seconds`) are only computed when both numerator and denominator are present and denominator > 0.

## Footguns and Exclusions

* **Timestamp format**: Run folders use `YYYY-MM-DD_HH.MM.SS` (dots in time) but some golden folders use colons. The collector's regex handles both: `(\d{2})[.:] (\d{2})[.:](\d{2})`.
* **`.job_parts/`**: Split-job staging artifacts (PDF/EPUB) live under `data/output/.job_parts/`. Ignored — these are temporary intermediates, not completed runs.
* **`prediction-run/`**: Benchmark eval directories may contain a `prediction-run/` subdirectory. Excluded from benchmark records to avoid mixing prediction artifacts with eval results.
* **Division safety**: All derived rates guard against division by zero; `None` is stored instead of `Infinity` or `NaN`.

## Dashboard UI Sections

**Header** — Generated timestamp, data roots, total counts (stage records, benchmark records, recipes, runtime).

**Throughput** — SVG sparkline of sec/recipe over time. Table of 15 slowest runs sorted by total seconds, with artifact-dir links.

**Benchmarks** — Precision and recall sparklines over time. Table of all benchmark runs sorted newest-first (timestamp, precision, recall, F1, gold total, matched, artifact link).

**Per-Label Breakdown** — Table showing precision, recall, gold count, and pred count for each label in the most recent benchmark that has per-label data. Shows which labels the pipeline handles well versus poorly.

**Boundary Classification** — Shows correct/over-segmented/under-segmented/partial boundary counts and percentages from the most recent benchmark with boundary data. Helps identify whether regressions are label misclassification or boundary drift.

**Filters** — Category checkboxes (stage_import on, benchmark_eval on, labelstudio_import off, benchmark_prediction off). Date range preset buttons (7d / 30d / 90d / All).

## Troubleshooting

**Dashboard is empty (no stage records):**
Ensure `data/output/.history/performance_history.csv` exists. If not, run `cookimport perf-report` first, then re-run `cookimport stats-dashboard`.

**Dashboard is empty (no benchmark records):**
Ensure `eval_report.json` files exist under `data/golden/`. Run an eval workflow first if needed.

**Collector warnings displayed:**
Warnings appear when a JSON file is malformed or a CSV row can't be parsed. The affected records are skipped; the rest of the dashboard still renders.

## Tests

`tests/test_stats_dashboard.py` — 16 tests covering:
* Schema: minimal instantiation, optional fields, run categories
* Collectors: CSV parsing, derived-field computation, JSON fallback, benchmark collection, combined runs, empty roots, malformed JSON handling, `.job_parts` exclusion, `prediction-run` exclusion
* Renderer: output file presence, JSON round-trip through Pydantic, idempotent re-generation

All tests use synthetic fixtures in `tmp_path` — no real cookbooks or Label Studio needed.
