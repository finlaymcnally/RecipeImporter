# Build deterministic `cf-debug` follow-up exporters and audits

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds. This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a benchmark follow-up packet is no longer a one-off agent-authored narrative. A user will be able to point a deterministic CLI at an existing `upload_bundle_v1`, generate a selector manifest, export fact-only case packets, run provenance audits, emit page/span context, emit uncertainty slices, and pack the results into a reproducible follow-up bundle.

The visible outcome is that future debugging starts from machine-checkable JSON and JSONL artifacts instead of prose. A novice should be able to run `cf-debug` against the existing sample bundle at `data/golden/benchmark-vs-golden/2026-03-04_20.33.53/single-profile-benchmark/upload_bundle_v1`, regenerate the same selector file twice with the same hash, export the exact weird rows called out in the request, and see whether each anomaly comes from model behavior, postprocessing, or bad provenance linkage.

## Surprises & Discoveries

- Observation: the current sample follow-up data is not an exporter contract; it is a fixed six-case packet plus markdown.
  Evidence: `data/golden/benchmark-vs-golden/2026-03-04_20.33.53/single-profile-benchmark/followup_data/` contains only `index.json`, `codexfarm_root_cause_packet.json`, and `codexfarm_root_cause_packet.md`.

- Observation: the repository already has the right evidence backbone in `upload_bundle_v1`.
  Evidence: the sample `upload_bundle_index.json` includes `analysis.triage_packet`, `analysis.per_recipe_breakdown`, `analysis.stage_separated_comparison`, `analysis.line_role_confidence_or_candidates`, `analysis.top_regression_packets_full_trace`, and `self_check.critical_row_locators_coverage_ratio = 1.0`.

- Observation: the trusted current builder path is a script helper wrapped by the main CLI, not a dedicated library module yet.
  Evidence: `cookimport/cli.py::_write_benchmark_upload_bundle(...)` loads `scripts.benchmark_cutdown_for_external_ai.build_upload_bundle_for_existing_output(...)`.

- Observation: not every machine-useful artifact is exposed through one simple top-level locator map in the bundle index.
  Evidence: the sample bundle’s human-facing `navigation.row_locators` map is narrower than the full set of analysis sections, so follow-up exporters must be able to resolve payload rows by canonical payload-path metadata and discovered run files instead of assuming one precomputed index is exhaustive.

- Observation: the candidate-label weirdness in the current packet is ambiguous until the exporter traces raw, parsed, and final labels together.
  Evidence: the current packet shows examples where the displayed prediction is outside the displayed `candidate_labels`, but the available artifact set also includes parsed prompt files, line-role outputs, and postprocess summaries, so the fault could be model-side, postprocess-side, or exporter-side.

## Outcomes & Retrospective

This document is the planning artifact only; no implementation has landed yet. The main outcome so far is a concrete path that keeps the proven `upload_bundle_v1` contract intact while replacing the current ad hoc `followup_data/` packet with deterministic selector, exporter, audit, and pack commands.

The main remaining risk is scope creep inside the current giant script helper. This plan resolves that by requiring only a narrow reusable context layer first and by deferring any full refactor of `scripts/benchmark_cutdown_for_external_ai.py`.

## Context and Orientation

The current benchmark handoff surface already has two layers.

The first layer is the base bundle. `scripts/benchmark_cutdown_for_external_ai.py` can build `upload_bundle_v1/`, which always contains exactly three files: `upload_bundle_overview.md`, `upload_bundle_index.json`, and `upload_bundle_payload.jsonl`. The index includes top-level summaries, analysis sections, and row locator metadata. The payload is the lossless store of every exported artifact row.

The second layer is the current follow-up packet. In the checked-in sample bundle under `data/golden/benchmark-vs-golden/2026-03-04_20.33.53/single-profile-benchmark/`, there is a sibling folder named `followup_data/`. That folder is not a general exporter. It is a hand-shaped packet with one markdown summary and one JSON file covering six cases such as `regression_c6`, `regression_c11`, `win_c10`, and `outside_span_window_628_657`.

The repository areas that matter for this work are:

- `pyproject.toml`: add the new `cf-debug` console entry point.
- `cookimport/cf_debug_cli.py`: new Typer entry point that exposes the follow-up subcommands.
- `cookimport/bench/`: the right home for reusable context loading, selector logic, export logic, and audit logic.
- `cookimport/cli.py`: only a context reference for how current benchmark helpers are loaded; this plan does not need to change the interactive menu.
- `scripts/benchmark_cutdown_for_external_ai.py`: the existing source of truth for upload-bundle construction, discovery of run pairs, and many of the summarization helpers that should be wrapped or minimally extracted.
- `cookimport/labelstudio/ingest.py` and `cookimport/labelstudio/canonical_line_projection.py`: likely touch points if page/span fields or line-role provenance fields are missing from current run artifacts and need to be written upstream.
- `data/golden/benchmark-vs-golden/2026-03-04_20.33.53/single-profile-benchmark/upload_bundle_v1/`: the fixture-quality real bundle to use during development and tests.
- `tests/labelstudio/` and new `tests/bench/` tests: the place to lock the CLI and schema behavior down.

Terms used in this plan:

`upload bundle` means the existing three-file export rooted at `upload_bundle_v1/`.

`selector manifest` means a deterministic JSON file that names which recipe cases or line ranges should be exported next and why.

`row locator` means a stable pointer back into `upload_bundle_payload.jsonl`, usually by canonical payload path plus payload row number.

`case export` means a fact-only JSON or JSONL record for one selected recipe or line-range case, including metrics, comparisons, context windows, provenance files, and locators.

`prompt-link audit` means a machine-readable check that the prompt snapshot shown for a case actually belongs to the claimed stage, pass, line, and source file.

`line-role audit` means a machine-readable row set that ties one selected line to every relevant label surface: gold, baseline prediction, codex prediction, raw model label, parsed label, final label, candidate labels, confidence, prompt files, and violations.

`ablation matrix` means the fixed experiment grid that isolates `atomic_block_splitter`, `line_role_pipeline`, and `llm_recipe_pipeline`.

## Plan of Work

### Milestone 1: Create a small reusable bundle context layer

Start by creating a new module under `cookimport/bench/`, preferably `cookimport/bench/followup_bundle.py`. This module should not replace the large existing script. It should wrap only the minimum read-only logic needed by the new CLI:

- load `upload_bundle_index.json` and `upload_bundle_payload.jsonl`,
- normalize canonical payload-path lookups,
- resolve run directories and paired runs from the source root,
- expose deterministic lookup helpers for `source_key`, `recipe_id`, `line_index`, and payload rows,
- expose prompt-log and line-role artifact resolution in one place.

The important design constraint is that this module must be able to answer follow-up questions even when the human-friendly `navigation` section is incomplete. It should treat the payload as the lossless authority and use discovered run artifacts when an index shortcut is missing.

Do not perform a giant refactor of `scripts/benchmark_cutdown_for_external_ai.py`. If necessary, duplicate a small amount of read-only helper logic first, then collapse it later only after tests exist.

This milestone is complete when a unit test can load the sample bundle and resolve at least these anchors deterministically: triage rows by `recipe_id`, changed-line rows by `line_index`, prompt artifacts for a selected line-role line, and payload row locators for a derived artifact path.

### Milestone 2: Add the `cf-debug` CLI and deterministic selector manifests

Add a new console entry in `pyproject.toml`:

    cf-debug = cookimport.cf_debug_cli:app

Implement `cookimport/cf_debug_cli.py` as a small Typer app that delegates all logic into `cookimport/bench/` functions.

The first subcommand is `select-cases`. Its job is to create a pure JSON selector manifest with schema `cf.selector.v1`. It must support both rule-based selection and explicit inclusion because the immediate debugging need includes exact known case IDs.

The command should support these inputs:

- `--bundle <upload_bundle_v1 dir>` as the required first implementation mode,
- `--top-neg <N>`,
- `--top-pos <N>`,
- `--outside-span <N>`,
- `--stage <stage key>` where `line_role` is the first required value,
- `--include-case-id <case_id>` repeated,
- `--include-recipe-id <recipe_id>` repeated,
- `--include-line-range <source_key>:<start>:<end>` repeated,
- `--out <file>`.

The output must contain:

- `schema_version`,
- `generated_at`,
- `command`,
- `git_sha`,
- `bundle_dir`,
- `bundle_sha256` or equivalent source fingerprint,
- one deterministically sorted `selectors` array.

Each selector row must include at least:

- `selector_id`,
- `case_id` when a stable existing case ID is available,
- `kind` (`recipe` or `line_range`),
- `source_key`,
- `recipe_id` or `start`/`end`,
- `reason`,
- `reason_rank`,
- `payload_locators`.

This milestone is complete when running `select-cases` twice against the sample bundle with the same arguments produces byte-identical JSON, and when an explicit include can target `regression_c6`, `regression_c11`, `outside_span_window_628_657`, and `win_c10` without hand-editing the file.

### Milestone 3: Export fact-only case packets

Implement `export-cases` next. This command takes the bundle and a selector manifest and writes a directory containing deterministic fact tables. The minimum required output is:

- `index.json` with schema `cf.case_export_index.v1`,
- `case_export.jsonl` with schema `cf.case_export.v1`,
- `file_manifest.jsonl` with one row per referenced source file including path, bytes, and `sha256`.

Each `case_export.jsonl` row must include the information the user asked for:

- `selector_id`, `case_id`, `source_key`, `book`, `recipe_id`,
- metrics and metric deltas,
- stage comparison,
- line-level changed rows,
- raw block window,
- pass1/pass2/pass3 summaries,
- context before and after,
- source paths,
- `sha256` and file bytes for those paths,
- row locators back into the base payload.

This command must stay fact-only. It may contain nested JSON objects, but it must not invent narrative explanations. It must also preserve deterministic sort order for lines inside a case and for cases inside the packet.

This milestone is complete when the sample weird rows and one clean win case can be exported entirely from selectors, and when the output contains enough file and row-locator information that a later consumer never needs the old `codexfarm_root_cause_packet.md`.

### Milestone 4: Implement `audit-line-role`

This is the highest-value audit. Implement `audit-line-role` so it emits one JSONL row per selected line with schema `cf.line_role_audit.v1`.

Start by resolving the selected lines from either recipe selectors or explicit line-range selectors. For each line, collect:

- `source_key`, `book`, `recipe_id`, `line_index`,
- `raw_text`,
- `gold_label`,
- `baseline_pred`,
- `codex_pred`,
- `candidate_labels`,
- `raw_model_label`,
- `parsed_label`,
- `final_label_after_postprocess`,
- `confidence`,
- `prompt_file`, `response_file`, `parsed_file`,
- `violations`.

The `violations` array must be additive and machine-checkable. At minimum support:

- `pred_not_in_candidate_labels`,
- `raw_not_in_candidate_labels`,
- `parsed_not_in_candidate_labels`,
- `candidate_labels_missing`,
- `parsed_label_missing`,
- `prompt_artifact_missing`,
- `postprocess_changed_label`,
- `baseline_join_missing`,
- `source_row_mismatch`.

If the current artifacts do not carry enough page/span or postprocess fields, extend the upstream writers in `cookimport/labelstudio/ingest.py` or `cookimport/labelstudio/canonical_line_projection.py` additively. Do not change existing schemas destructively; add fields and document them.

This milestone is complete when the sample audit can explain the specific weird rows requested by the user: `regression_c6` lines `1077-1079`, `regression_c11` lines `1143-1151`, `outside_span_window_628_657` line `657`, and one clean win case such as `win_c10`.

### Milestone 5: Implement `audit-prompt-links`

Implement `audit-prompt-links` as a JSONL exporter with schema `cf.prompt_link_audit.v1`. Its purpose is to prove that the prompt snapshot associated with a case actually belongs to the claimed stage and line.

Each row must include:

- `selector_id`, `case_id`, `source_key`, `recipe_id`, and optionally `line_index`,
- `claimed_prompt_kind`,
- `claimed_pass`,
- `actual_source_stage`,
- `actual_pass`,
- `prompt_file`, `response_file`, `parsed_file`,
- `matched_call_id`,
- `matched_line_index`,
- `provenance_status` (`ok` or `mismatch`),
- `mismatch_reasons`.

The logic must prefer true line-role prompt directories when auditing line-role lines. If the only available match is a fallback full prompt log entry, the row must say so explicitly instead of pretending it is a direct prompt chunk match.

This milestone is complete when the sample bundle can generate a small table that answers, for each selected case, which prompt was shown, which pass it actually came from, which file it came from, and whether the provenance is `ok` or `mismatch`.

### Milestone 6: Implement `export-page-context` and `export-uncertainty`

Implement `export-page-context` first. The output is JSONL with schema `cf.page_context.v1`. The command must accept a comma-separated include list and, for the first version, support:

- `spine_index`,
- `page_type`,
- `within_recipe_span`,
- `chapterish_fields`.

If a field is missing upstream, emit an explicit status such as `available=false` or `reconstructed=false`; do not silently omit it. For `chapterish_fields`, include both the reconstructed values and a boolean that says whether they came from primary artifact fields or heuristic reconstruction.

Then implement `export-uncertainty`. Its output is JSONL with schema `cf.uncertainty.v1`. It must always write a file, even when there are zero qualifying rows. Rank rows deterministically by:

1. lowest confidence,
2. smallest top-1/top-2 margin when available,
3. largest candidate set size,
4. disagreement between raw, parsed, and final labels.

The summary row in the file header or companion `index.json` must state why the packet may be empty instead of silently disappearing.

This milestone is complete when `saltfatacidheatCUTDOWN.epub` exports explicit page/span context around outside-span windows and when `export-uncertainty` still emits a valid zero-row packet for bundles with no low-confidence hits under the threshold.

### Milestone 7: Implement `pack`

Implement `pack` as the deterministic bundling step. It should take a bundle, a selector manifest, and a list of already-generated exports or audits and write one folder such as `followup_packet_v2/`.

The folder must contain:

- `index.json` with schema `cf.pack_index.v1`,
- `manifest.json` with schema `cf.pack_manifest.v1`,
- the included JSON and JSONL export files,
- an optional `README.md` generated from a static template plus the manifest values.

The `pack` command must not regenerate meaning from prose. Its job is to gather already-generated facts, describe what is present, list hashes and file sizes, and keep row locators back to the base payload.

This milestone is complete when a novice can inspect `index.json`, see exactly which selector file and which audit/export artifacts are included, and navigate back to the base `upload_bundle_v1` rows without any freeform summary text.

### Milestone 8: Implement `ablate`

Implement `ablate` last because it is the most expensive path and can reuse the schemas and summary helpers built earlier.

The command should be a thin wrapper over existing benchmark primitives, not a parallel benchmark system. Create a helper under `cookimport/bench/` that resolves the requested books, then launches the fixed matrix:

- `vanilla`: `llm_recipe_pipeline=off`, `line_role_pipeline=off`, `atomic_block_splitter=off`
- `splitter`: `llm_recipe_pipeline=off`, `line_role_pipeline=off`, `atomic_block_splitter=atomic-v1`
- `line_role`: `llm_recipe_pipeline=off`, `line_role_pipeline=codex-line-role-v1`, `atomic_block_splitter=off`
- `splitter+line_role`: `llm_recipe_pipeline=off`, `line_role_pipeline=codex-line-role-v1`, `atomic_block_splitter=atomic-v1`
- `full`: `llm_recipe_pipeline=codex-farm-3pass-v1`, `line_role_pipeline=codex-line-role-v1`, `atomic_block_splitter=atomic-v1`

The command must write:

- `matrix_manifest.json`,
- `matrix_results.jsonl`,
- `matrix_summary.json`,
- one subdirectory per run with the original benchmark artifacts or pointers to them.

Because this path can invoke Codex-enabled runs, preserve the existing explicit-confirmation safety style. Add a required confirmation flag for matrices that include `full`, or thread the existing benchmark confirmation path through unchanged.

This milestone is complete when the user can run the matrix on `saltfatacidheatCUTDOWN.epub` and `DinnerFor2CUTDOWN.epub` and get one deterministic summary comparing all five configurations without manual run bookkeeping.

### Milestone 9: Tests, docs, and contract cleanup

Add tests before any broad cleanup. Use the checked-in sample bundle as the main fixture because it already contains the problematic cases the user cares about.

Add at least:

- unit tests for selector ranking, stable case IDs, and stable sort order,
- fixture tests for `audit-line-role` on the requested weird rows,
- fixture tests for `audit-prompt-links` on the same selected cases,
- fixture tests for `export-page-context` and `export-uncertainty`,
- CLI tests that invoke `cf-debug` commands against the sample bundle,
- a small test that proves `select-cases` is byte-stable across repeated runs,
- a pack test that checks file hashes and manifest contents,
- an ablation test that verifies matrix planning and command construction without needing to run the expensive full matrix in normal test time.

Once the code and tests are stable, update:

- `docs/07-bench/07-bench_README.md`,
- `docs/02-cli/02-cli_README.md`,
- `cookimport/bench/README.md`,
- `scripts/README.md` only if the script-helper relationship changes materially.

The docs should stay short and should explain that `cf-debug` is a deterministic follow-up tool layered on `upload_bundle_v1`.

## Validation and Acceptance

The implementation is acceptable only when all of the following are true:

First, selectors are deterministic. Running `cf-debug select-cases` twice with the same arguments against the same bundle must produce byte-identical JSON. Verify this with:

    sha256sum /tmp/cf_selectors.json

and a rerun of the command to the same path after deleting it.

Second, the line-role audit resolves the user’s current ambiguity. For the selected weird rows, the output must show raw, parsed, and final labels side by side and must state any rule violation or provenance gap explicitly in `violations`.

Third, the prompt-link audit resolves prompt provenance. For each selected case, the output must show which prompt/response/parsed trio was linked, which pass it actually belongs to, which line it matched, and whether that linkage is `ok` or `mismatch`.

Fourth, page/span metadata is first-class rather than silently absent. The page-context export must show `spine_index`, `page_type`, and `within_recipe_span` when available, and it must explicitly mark chapter-like metadata as absent or reconstructed when that is the truth.

Fifth, uncertainty exports always exist. Even if the packet has zero rows, `export-uncertainty` must still write a valid schema-bearing output file and summary.

Sixth, the pack step remains fact-only. The generated `followup_packet_v2/` must consist of JSON or JSONL exports plus a template-generated index or README, not a hand-authored narrative.

Seventh, the ablation matrix is reproducible. The matrix summary must identify exactly which of the three components changed in each experiment and must preserve run settings, hashes, and metric deltas for later comparison.

## Idempotence and Recovery

The selector, export, audit, page-context, uncertainty, and pack commands are read-only with respect to the source bundle. They should only write to their explicit output files or directories. Make overwrite behavior explicit with a flag such as `--overwrite`; the default should be fail-fast when the destination already exists.

If a particular artifact cannot be resolved, the command must emit a row with a missing-status field rather than aborting the whole export. Missing prompt files or absent chapter metadata are expected failure modes in historical bundles and must be reported as data, not as fatal process errors.

`ablate` is the only command that creates new benchmark run directories. It must keep all writes under the requested `--out` directory, reuse existing caches when possible, and preserve the exact command lines it launched in a manifest so a failed matrix run can be resumed or replayed safely.
