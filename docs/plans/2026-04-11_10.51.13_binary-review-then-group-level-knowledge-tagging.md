# Replace Row-First Knowledge Tagging With Binary Review Then Group-Level Tagging

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the nonrecipe knowledge pipeline will stop making tag decisions one row at a time. A human should be able to run a Codex-backed knowledge pass and see that the first pass only decides whether a row is worth carrying into knowledge review at all, while the second pass reads the kept rows together, forms idea groups, assigns one shared tag story to each group, and then projects those group tags back onto the member rows. The user-visible effect is that a cluster of related rows about one idea such as salt seasoning, dough resting, or heat control should no longer emerge with a pile of weak row-by-row tags; it should emerge as one group with one coherent set of tags, with row-level differences called out only when there is a concrete reason.

This change matters because the current design still thinks about tags too early and too locally. The first pass currently uses existing-tag grounding to justify `knowledge`, and the second pass still resolves proposals per row even while it is also grouping. That is the opposite of the desired product behavior. After this change, pass 1 will be a binary keep-or-drop review, and all tag thinking will happen only during pass 2 in the same train of thought that forms groups.

## Surprises & Discoveries

- Observation: the current code already moved some proposal logic into the same second pass as grouping, but it did not actually become group-first.
  Evidence: `cookimport/llm/knowledge_stage/task_file_contracts.py` still validates `proposal_decision` and `proposed_tags` per unit, and `combine_knowledge_task_file_outputs(...)` still decides final grounding row-by-row before separately assembling `idea_groups`.

- Observation: the current first pass is still biased toward tag-shaped thinking.
  Evidence: the benchmark artifact from `2026-04-11_10.33.16` shows first-pass repairs promoting salt-related rows to `knowledge` via weak existing tags such as `savory`, `seasoning`, `kosher`, and `low-sodium`, which then distorts the second pass.

- Observation: the `proposal_candidate` design solved one older problem but now conflicts with the desired product behavior.
  Evidence: the current prompt and validator are designed around a row-level question of “is this row knowledge, proposal_candidate, or other?” while the desired behavior is “is this row worth group review at all?” followed by a group-level tagging decision.

- Observation: the current benchmark failure was not only about validator brittleness. It also exposed the deeper contract mismatch.
  Evidence: even when grouping responses were semantically coherent, the system still thought in row-level proposals such as `salt`, `sea-salt`, `salt-cellar`, and `granular-salt` within one shared salt cluster instead of producing one group-level tag story and projecting it downward.

## Decision Log

- Decision: pass 1 will become a binary review only. It will not ground to existing tags, will not invent proposed tags, and will not emit `proposal_candidate`.
  Rationale: the user explicitly wants all tag thinking deferred to pass 2, and binary pass 1 is the cleanest way to enforce that boundary.
  Date/Author: 2026-04-11 / Codex

- Decision: pass 2 will own grouping and tagging as one single thought process, and group tags will be the primary authority.
  Rationale: the desired product behavior is that related rows should be grouped first and then tagged together, not tagged independently and only later grouped for observability.
  Date/Author: 2026-04-11 / Codex

- Decision: row-level tag differences will be modeled as explicit overrides, not as the default representation.
  Rationale: rows in the same group should normally share the same final tags. Differences should be visible, narrow, and intentional rather than hidden in per-row proposal answers.
  Date/Author: 2026-04-11 / Codex

- Decision: this plan replaces the current `proposal_candidate` architecture instead of extending it.
  Rationale: trying to preserve `proposal_candidate` would keep row-first tag semantics alive in the contract and make the implementation harder to reason about.
  Date/Author: 2026-04-11 / Codex

## Outcomes & Retrospective

This plan is newly authored and no implementation work has started yet. The intended outcome is a simpler and more product-aligned knowledge pipeline: pass 1 becomes a binary retention screen, pass 2 becomes a group-and-tag stage, and final row grounding becomes a projection of group decisions rather than the accumulation of row-level proposal decisions. The main risk to watch during implementation is migration complexity, because the current runtime, promotion, and tests all encode row-level grouping answers. The implementation must simplify that shape instead of layering a compatibility maze on top.

## Context and Orientation

The relevant code lives under `cookimport/llm/knowledge_stage/` plus the nonrecipe staging helpers under `cookimport/staging/`. In this repository, “nonrecipe” means blocks that were not accepted as part of a recipe. Some of those blocks become final `knowledge`, which is reusable cooking guidance worth keeping for retrieval later, while the rest become `other`.

The current knowledge runtime uses two semantic passes. The first pass is “classification.” It currently asks the model to label each row as `knowledge`, `proposal_candidate`, or `other`, and it requires `knowledge` rows to carry existing-tag grounding. The second pass is “grouping.” It currently groups rows, sets `group_key` and `topic_label`, and also approves or rejects row-level proposals. The key files are `cookimport/llm/knowledge_stage/task_file_contracts.py`, which defines the worker-visible task contracts and validation, `cookimport/llm/knowledge_stage/structured_session_contract.py`, which builds inline structured prompts and parses responses, and `cookimport/llm/knowledge_stage/workspace_run.py`, which orchestrates live Codex runs, repairs, and promotion.

The current final output path still thinks row-first. In `combine_knowledge_task_file_outputs(...)`, each row is promoted individually based on its classification answer plus its own grouping answer, and only then are `idea_groups` assembled as a secondary artifact. Promotion and stage observability then count per-row proposal outcomes. This is exactly the inversion that must be changed.

For this plan, use these plain-language definitions:

- “Pass 1” means the first LLM review over nonrecipe candidate rows. After this change it will answer only one question: should this row be kept for knowledge review or dropped as `other`?
- “Pass 2” means the second LLM review over rows that survived pass 1. After this change it will read those rows together, form groups, and assign group-level tags in the same answer.
- “Group-level tags” means tags attached to a whole idea group, not attached independently to each row.
- “Projection” means copying a group’s final tag set back onto all of that group’s member rows by default.
- “Row override” means an explicit exception on one member row such as adding or removing one tag because that row is a genuine exception to the shared group tag story.

The benchmark artifact `data/golden/benchmark-vs-golden/2026-04-11_10.33.16` is useful orientation because it demonstrates the current problem clearly: first-pass knowledge rows were grounded to weak existing tags, second-pass grouping produced one shared salt topic, and yet the final proposal artifacts still treated tags as row-local objects rather than as one group decision.

## Plan of Work

Start by replacing the first-pass contract in `cookimport/llm/knowledge_stage/task_file_contracts.py`. The current classification answer schema allows `knowledge`, `proposal_candidate`, and `other`, with grounding required for `knowledge`. Remove that shape entirely. The first-pass answer should become binary with one keep state and one drop state. Use a stable internal literal such as `keep_for_review` and `other`, and make the answer schema reject all tag grounding and proposal fields in pass 1. This change must be mirrored in `cookimport/llm/knowledge_stage/structured_session_contract.py` so the inline structured prompt and parser are aligned with the same binary contract. The fake runner in `cookimport/llm/fake_codex_farm_runner.py` and the focused knowledge tests must be updated in the same change so the repo has one consistent first-pass contract.

Then redesign the second-pass contract to be group-first. In `cookimport/llm/knowledge_stage/task_file_contracts.py`, replace the per-row grouping answer shape with a group-oriented answer shape. The model should return a list of groups. Each group should contain a stable `group_key`, a human-readable `topic_label`, an ordered list of member `row_id`s, and one shared tag payload for the group. That shared payload should be the only normal place where tags are approved or proposed. Add one narrow optional field for row overrides so a group member can diverge from the shared tags only when there is a concrete reason. The override surface should be additive and explicit, such as `add_tags` and `remove_tags`, rather than a full alternate row-level proposal object.

After the new group-first answer shape exists, update `cookimport/llm/knowledge_stage/structured_session_contract.py` so the prompt explicitly tells the model to think in group units first and tag those groups in the same train of thought. The response example must show one group with several member rows and one shared tag set. The prompt must also say that row overrides are rare and should be used only when a member row truly needs a different tag story than the rest of the group.

Next, rework validation and runtime orchestration. In `cookimport/llm/knowledge_stage/task_file_contracts.py`, write a validator for the new group-based pass 2 answer. It must enforce that every kept row from pass 1 appears in exactly one group, that group member lists contain only owned rows, that no row appears in more than one group, and that group-level tags follow the ontology rules. Row overrides must validate against the group they belong to. In `cookimport/llm/knowledge_stage/workspace_run.py`, the repair path must narrow to unresolved grouping rows or unresolved groups while still preserving already validated groups. The same-session handoff path in `cookimport/llm/knowledge_same_session_handoff.py` must also be updated so resumed pass 2 work continues with the new group-first answer structure instead of the old per-row grouping answers.

Then replace the current row-first promotion logic. In `combine_knowledge_task_file_outputs(...)` and any adjacent helpers in `cookimport/llm/knowledge_stage/task_file_contracts.py`, make group-level tags the authority. Final per-row grounding should be built by taking the group’s shared tags and projecting them onto each member row, then applying any explicit row override for that member. `idea_groups` should stop being a sidecar summary and instead become the primary semantic artifact that row outputs are derived from. Update any stage-level metadata collectors such as `collect_knowledge_resolution_metadata_by_shard(...)` so observability reports group counts, group tag counts, and explicit override counts rather than proposal-candidate approval counts.

Finally, update staging, docs, and tests. `cookimport/staging/nonrecipe_authority.py`, `cookimport/staging/writer.py`, and any benchmark summary helpers that currently expect row-level proposal counts must be taught the new group-first model. The owning docs in `docs/10-llm/10-llm_README.md`, `docs/05-staging/05-staging_readme.md`, and `docs/07-bench/07-bench_README.md` must explain that pass 1 is now binary and pass 2 owns both grouping and tag assignment. Focused tests should prove three important things: pass 1 has no tag thinking, pass 2 returns one coherent group tag story for related rows, and final row grounding is projected from groups unless a rare explicit override changes it.

## Validation and Acceptance

Acceptance means all of the following are true:

1. Pass 1 no longer accepts or emits existing-tag grounding, proposed tags, or `proposal_candidate`. It only decides `keep_for_review` or `other`.
2. Pass 2 consumes only rows that survived pass 1 and returns groups, not row-by-row proposal decisions.
3. A cluster of related kept rows can receive one shared group tag set in one answer, and final per-row grounding is derived from that group tag set.
4. Row-level tag differences appear only as explicit overrides and are visible in artifacts as exceptions, not as the normal semantic shape.
5. Stage observability and benchmark artifacts report group-oriented semantics instead of row-oriented proposal-candidate counts.
6. Focused knowledge and staging tests pass in the project `.venv`, and at least one narrow runtime or benchmark replay shows a grouped knowledge cluster receiving one coherent shared tag story.
