# Replace Row-First Knowledge Tagging With Binary Review Then Group-Level Tagging

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the nonrecipe knowledge pipeline will stop making tag decisions one row at a time. A human should be able to run a Codex-backed knowledge pass and see that the first pass only decides whether a row is worth carrying into knowledge review at all, while the second pass reads the kept rows together, forms idea groups, assigns one shared tag story to each group, and then projects those group tags back onto the member rows. The user-visible effect is that a cluster of related rows about one idea such as salt seasoning, dough resting, or heat control should no longer emerge with a pile of weak row-by-row tags; it should emerge as one group with one coherent set of tags.

The worker-visible packet surface should also stay compact and ordered, in the same spirit as the current line-role path. Pass 1 should look like ordered provenance rows such as `r01 | 210 | Bright Cabbage Slaw`, not verbose row-object JSON. Pass 2 should keep that same row-first reading order while exposing grouping and group-tag decisions in a compact line-oriented way, so the model can see adjacency and keep related rows together without wasting tokens on wrapper noise.

This change matters because the current design still thinks about tags too early and too locally. The first pass currently uses existing-tag grounding to justify `knowledge`, and the second pass still resolves proposals per row even while it is also grouping. That is the opposite of the desired product behavior. After this change, pass 1 will be a binary keep-or-drop review, and all tag thinking will happen only during pass 2 in the same train of thought that forms groups.

## Surprises & Discoveries

- Observation: the current code already moved some proposal logic into the same second pass as grouping, but it did not actually become group-first.
  Evidence: `cookimport/llm/knowledge_stage/task_file_contracts.py` still validates `proposal_decision` and `proposed_tags` per unit, and `combine_knowledge_task_file_outputs(...)` still decides final grounding row-by-row before separately assembling `idea_groups`.

- Observation: the current first pass is still biased toward tag-shaped thinking.
  Evidence: the benchmark artifact from `2026-04-11_10.33.16` shows first-pass repairs promoting salt-related rows to `knowledge` via weak existing tags such as `savory`, `seasoning`, `kosher`, and `low-sodium`, which then distorts the second pass.

- Observation: the `proposal_candidate` design solved one older problem but now conflicts with the desired product behavior.
  Evidence: the current prompt and validator are designed around a row-level question of “is this row knowledge, proposal_candidate, or other?” while the desired behavior is “is this row worth group review at all?” followed by a group-level tagging decision.

- Observation: the current benchmark failure was not only about validator brittleness. It also exposed the deeper contract mismatch.
  Evidence: even when grouping responses were semantically coherent, the system still thought in row-level proposals such as `salt`, `sea-salt`, `salt-cellar`, and `granular-salt` within one shared salt cluster instead of producing one group-level tag story and projecting it downward.

## Decision Log

- Decision: pass 1 will become a binary review only. It will not ground to existing tags, will not invent proposed tags, and will not emit `proposal_candidate`.
  Rationale: the user explicitly wants all tag thinking deferred to pass 2, and binary pass 1 is the cleanest way to enforce that boundary.
  Date/Author: 2026-04-11 / Codex

- Decision: pass 2 will own grouping and tagging as one single thought process, and group tags will be the primary authority.
  Rationale: the desired product behavior is that related rows should be grouped first and then tagged together, not tagged independently and only later grouped for observability.
  Date/Author: 2026-04-11 / Codex

- Decision: pass 1 should share the current line-role packet ergonomics: compact ordered provenance rows such as `rXX | block_index | text`, plus a minimal ordered label surface rather than verbose per-row answer objects.
  Rationale: the line-role path already demonstrates the desired token-efficient shape, and the user explicitly wants knowledge phase 1 to feel like the same class of task.
  Date/Author: 2026-04-11 / Codex

- Decision: within pass 2, the contract should treat “same group” as “same final tag story.” If some rows need different tags, the model should create a new group instead of emitting row-level tag overrides inside one group.
  Rationale: the user wants grouping and tagging to be one train of thought, with groups as the unit of semantic authority. Splitting a group is cleaner and more legible than keeping one group number while sneaking in row-specific tags.
  Date/Author: 2026-04-11 / Codex

- Decision: this plan replaces the current `proposal_candidate` architecture instead of extending it.
  Rationale: trying to preserve `proposal_candidate` would keep row-first tag semantics alive in the contract and make the implementation harder to reason about.
  Date/Author: 2026-04-11 / Codex

## Outcomes & Retrospective

This plan is newly authored and no implementation work has started yet. The intended outcome is a simpler and more product-aligned knowledge pipeline: pass 1 becomes a binary retention screen with the same compact packet ergonomics as line-role, pass 2 becomes a group-and-tag stage that keeps rows in reading order, and final row grounding becomes a projection of group decisions rather than the accumulation of row-level proposal decisions. The main risk to watch during implementation is migration complexity, because the current runtime, promotion, and tests all encode row-level grouping answers. The implementation must simplify that shape instead of layering a compatibility maze on top.

## Context and Orientation

The relevant code lives under `cookimport/llm/knowledge_stage/` plus the nonrecipe staging helpers under `cookimport/staging/`. In this repository, “nonrecipe” means blocks that were not accepted as part of a recipe. Some of those blocks become final `knowledge`, which is reusable cooking guidance worth keeping for retrieval later, while the rest become `other`.

The current knowledge runtime uses two semantic passes. The first pass is “classification.” It currently asks the model to label each row as `knowledge`, `proposal_candidate`, or `other`, and it requires `knowledge` rows to carry existing-tag grounding. The second pass is “grouping.” It currently groups rows, sets `group_key` and `topic_label`, and also approves or rejects row-level proposals. The key files are `cookimport/llm/knowledge_stage/task_file_contracts.py`, which defines the worker-visible task contracts and validation, `cookimport/llm/knowledge_stage/structured_session_contract.py`, which builds inline structured prompts and parses responses, and `cookimport/llm/knowledge_stage/workspace_run.py`, which orchestrates live Codex runs, repairs, and promotion.

The current final output path still thinks row-first. In `combine_knowledge_task_file_outputs(...)`, each row is promoted individually based on its classification answer plus its own grouping answer, and only then are `idea_groups` assembled as a secondary artifact. Promotion and stage observability then count per-row proposal outcomes. This is exactly the inversion that must be changed.

For this plan, use these plain-language definitions:

- “Pass 1” means the first LLM review over nonrecipe candidate rows. After this change it will answer only one question: should this row be kept for knowledge review or dropped as `other`?
- “Pass 2” means the second LLM review over rows that survived pass 1. After this change it will read those rows together, form groups, and assign group-level tags in the same answer.
- “Compact row surface” means ordered packet rows rendered like `rXX | block_index | TEXT`, with no verbose per-row JSON wrappers in the visible prompt.
- “Group-level tags” means tags attached to a whole idea group, not attached independently to each row.
- “Projection” means copying a group’s final tag set back onto all of that group’s member rows by default.
- “Group split” means creating a new group when some rows need a different tag story from nearby rows. Same group means same final tags.

The benchmark artifact `data/golden/benchmark-vs-golden/2026-04-11_10.33.16` is useful orientation because it demonstrates the current problem clearly: first-pass knowledge rows were grounded to weak existing tags, second-pass grouping produced one shared salt topic, and yet the final proposal artifacts still treated tags as row-local objects rather than as one group decision.

## Plan of Work

Start by replacing the first-pass contract in `cookimport/llm/knowledge_stage/task_file_contracts.py`. The current classification answer schema allows `knowledge`, `proposal_candidate`, and `other`, with grounding required for `knowledge`. Remove that shape entirely. The first-pass answer should become binary with one keep state and one drop state. Use a stable internal literal such as `keep_for_review` and `other`, and make the answer schema reject all tag grounding and proposal fields in pass 1. Mirror the worker-visible packet on line-role ergonomics: owned rows should render as ordered provenance lines like `r01 | 210 | Bright Cabbage Slaw`, and the response should use an equally compact ordered label surface instead of verbose row-shaped answer objects. This change must be mirrored in `cookimport/llm/knowledge_stage/structured_session_contract.py` so the inline structured prompt and parser are aligned with the same binary contract. The fake runner in `cookimport/llm/fake_codex_farm_runner.py` and the focused knowledge tests must be updated in the same change so the repo has one consistent first-pass contract.

Then redesign the second-pass contract to be group-first and line-first at the same time. In `cookimport/llm/knowledge_stage/task_file_contracts.py`, replace the per-row grouping answer shape with a grouping contract that keeps rows in their original order and makes group membership explicit on those ordered rows. The worker-visible surface should stay compact, closer to `rXX | block_index | TEXT | group# | group tag story` than to a nest of row objects. Deterministic code may normalize that into internal group records after parsing, but the visible prompt and response should preserve adjacency and make it obvious which nearby rows belong together. Same `group#` must imply the same final tag story. If some nearby rows deserve different tags, the model should start a new group instead of attaching row-specific tags inside the old one.

After the new group-first answer shape exists, update `cookimport/llm/knowledge_stage/structured_session_contract.py` so the prompt explicitly tells the model to think in group units first and tag those groups in the same train of thought. The response example should keep the rows in source order and show that one run of adjacent rows can share one group id and one group tag story. The prompt must also say that if a subset needs different tags, the correct move is to split that subset into a new group, not to keep one group id with mixed tag outcomes.

Next, rework validation and runtime orchestration. In `cookimport/llm/knowledge_stage/task_file_contracts.py`, write a validator for the new pass 2 answer. It must enforce that every kept row from pass 1 appears exactly once, that group ids only refer to owned rows, that no row appears in more than one group, and that all rows sharing one group id also share one final tag payload. If the returned shape implies mixed tags inside one group, validation should fail closed and instruct the model to split that cluster into multiple groups instead. In `cookimport/llm/knowledge_stage/workspace_run.py`, the repair path must narrow to unresolved grouping rows or unresolved groups while still preserving already validated groups. The same-session handoff path in `cookimport/llm/knowledge_same_session_handoff.py` must also be updated so resumed pass 2 work continues with the new group-first answer structure instead of the old per-row grouping answers.

Then replace the current row-first promotion logic. In `combine_knowledge_task_file_outputs(...)` and any adjacent helpers in `cookimport/llm/knowledge_stage/task_file_contracts.py`, make group-level tags the authority. Final per-row grounding should be built by taking the group’s shared tags and projecting them onto each member row. `idea_groups` should stop being a sidecar summary and instead become the primary semantic artifact that row outputs are derived from. Update any stage-level metadata collectors such as `collect_knowledge_resolution_metadata_by_shard(...)` so observability reports group counts, group tag counts, and group split counts rather than proposal-candidate approval counts or row-level override counts.

Finally, update staging, docs, and tests. `cookimport/staging/nonrecipe_authority.py`, `cookimport/staging/writer.py`, and any benchmark summary helpers that currently expect row-level proposal counts must be taught the new group-first model. The owning docs in `docs/10-llm/10-llm_README.md`, `docs/05-staging/05-staging_readme.md`, and `docs/07-bench/07-bench_README.md` must explain that pass 1 is now binary and shares line-role’s compact ordered row surface, while pass 2 owns both grouping and tag assignment. Focused tests should prove four important things: pass 1 has no tag thinking, pass 1 uses the compact row surface, pass 2 returns one coherent group tag story for related rows, and rows that deserve different tags are split into different groups.

## Validation and Acceptance

Acceptance means all of the following are true:

1. Pass 1 no longer accepts or emits existing-tag grounding, proposed tags, or `proposal_candidate`. It only decides `keep_for_review` or `other`.
2. Pass 1 uses a compact ordered row surface aligned with line-role, with rows rendered like `rXX | block_index | TEXT` instead of visible row-object JSON.
3. Pass 2 consumes only rows that survived pass 1 and returns group decisions, not row-by-row proposal decisions.
4. Rows sharing one group id also share one final tag story, and final per-row grounding is derived by projecting that group tag story back onto all member rows.
5. If some rows deserve different tags, the response splits them into a new group instead of keeping one group id with mixed row-level tags.
6. Stage observability and benchmark artifacts report group-oriented semantics instead of row-oriented proposal-candidate counts.
7. Focused knowledge and staging tests pass in the project `.venv`, and at least one narrow runtime or benchmark replay shows a grouped knowledge cluster receiving one coherent shared tag story.

## Idempotence and Recovery

If pass 2 validation breaks mid-migration, the quickest recovery path is to inspect the saved packet and response artifacts under `data/output/.../nonrecipe_finalize/` and confirm whether the model returned valid groups and member rows even if projection failed later. If a test snapshot becomes noisy, prefer rewriting it around observable group behavior rather than preserving row-level proposal details that are supposed to disappear.
