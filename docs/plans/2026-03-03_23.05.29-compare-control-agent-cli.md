# Implement backend Compare & Control CLI and JSONL agent mode

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md` from the repository root.

This plan builds from seam discovery in `docs/understandings/2026-03-03_23.05.29-compare-control-backend-cli-seams.md` and the additional implementation notes in `docs/understandings/2026-03-03_23.17.54-compare-control-agent-cli-plan-hardening.md`.

## Purpose / Big Picture

After this change, terminal tools can ask the same `Compare & Control` questions that are currently only available in the browser dashboard. A user (or agent) will be able to run a local CLI command, point it at `data/output` and `data/golden`, and receive deterministic JSON for discovery, raw analysis, controlled analysis, and table-filter subset patches.

The outcome is observable without opening a browser: `cookimport compare-control run` returns machine-readable results, and `cookimport compare-control agent` keeps a low-latency JSONL session open for repeated requests in one process. QualitySuite outputs now also produce an `agent_compare_control` bridge bundle so AI agents can move from quality verdicts to compare/control drill-down without rediscovery.

## Surprises & Discoveries

- Observation: Compare/control analytics are currently JS-only inside dashboard render output.
  Evidence: `cookimport/analytics/dashboard_render.py` contains `analyzeCompareControl*` helpers; no equivalent Python module exists.

- Observation: Compare/control runs on the same filtered row set as table/trend (`computePreviousRunsFilterResult()`), not on raw benchmark records.
  Evidence: `renderCompareControlPanel(...)` is called with `matchedRecords` from quick filters and compiled column filters.

- Observation: “Filter to subset” already has a stable filter-write contract worth reusing in CLI responses.
  Evidence: `syncCompareControlSelectionToTableFilters()` writes `eq` clauses + `or` mode through existing column-filter helpers.

- Observation: Derived field semantics are critical for parity and must be reproduced in backend code.
  Evidence: `previousRunsFieldValue(...)` defines `source_label`, `ai_model`, `ai_effort`, `all_token_use`, `artifact_dir_basename`, `all_method_record`, and `speed_suite_record`.

- Observation: There is already a behavior harness test that proves controlled categorical weighting and subset patch behavior.
  Evidence: `tests/analytics/test_stats_dashboard.py` (`test_compare_control_controlled_categorical_standardizes_strata` and `test_compare_control_filter_to_subset_writes_table_filter_clauses`).

- Observation: The controlled categorical parity fixture is intentionally `raw A > raw B` and `controlled B > controlled A` in current dashboard tests.
  Evidence: `tests/analytics/test_stats_dashboard.py::test_compare_control_controlled_categorical_standardizes_strata`.

- Observation: `all_token_use` discounted token total is `(input - cached_input) + 0.1*cached_input + output`, so the expected sample value is `1120` for `input=1000, cached=200, output=300`.
  Evidence: dashboard helper `previousRunsDiscountedTokenTotal()` and aligned backend test assertion.

- Observation: QualitySuite run/compare artifacts already have stable scope boundaries (`run_root`, `experiments/<id>`, baseline/candidate selectors) that can be mapped to compare/control rows via `artifact_dir` prefix filtering.
  Evidence: quality run summaries contain experiment IDs and compare payloads contain baseline/candidate selectors; compare/control records include `artifact_dir`.

## Decision Log

- Decision: Build a deterministic Python engine first, then wire CLI and optionally consume it from dashboard rendering later.
  Rationale: A backend CLI cannot depend on browser JS internals; parity requires a shared server-side contract.
  Date/Author: 2026-03-03 / assistant

- Decision: Expose commands as a Typer subgroup (`app.add_typer(..., name="compare-control")`) with `run` and `agent` subcommands.
  Rationale: `cookimport/cli.py` already uses Typer sub-apps (for example `bench` and `epub`), so this is the least surprising integration path.
  Date/Author: 2026-03-03 / assistant

- Decision: Define a strict JSONL envelope for agent mode (`id`, `action`, `payload`) and always return structured success/error responses.
  Rationale: Agent workflows need deterministic parsing and robust recovery from malformed requests without process exit.
  Date/Author: 2026-03-03 / assistant

- Decision: Keep scope local and deterministic (no LLM-in-the-loop analytics and no network server).
  Rationale: Matches project instruction to avoid AI parsing/cleaning in import-adjacent flows and preserves reproducibility.
  Date/Author: 2026-03-03 / assistant

- Decision: Bridge QualitySuite and Compare & Control through a generated bundle (`agent_compare_control`) containing precomputed `insights`, machine index metadata, and ready JSONL follow-up requests.
  Rationale: Agents should not rediscover file layout every run; a deterministic handoff artifact creates a fast, repeatable loop from quality verdicts to root-cause analysis.
  Date/Author: 2026-03-03 / assistant

## Context and Orientation

`cookimport stats-dashboard` is generated in two phases. `collect_dashboard_data(...)` in `cookimport/analytics/dashboard_collect.py` builds `DashboardData` from history/eval artifacts, then `render_dashboard(...)` in `cookimport/analytics/dashboard_render.py` writes static assets and embeds all compare/control logic in generated JavaScript.

For this plan, “benchmark rows” means the records in `DashboardData.benchmark_records`. “Quick filters” means the boolean filter toggles applied before column filters (`official_full_golden_only`, `exclude_ai_tests`). “Column filters” means per-field clauses with per-field `and/or` and global cross-field `and/or` combination logic. “Controlled analysis” means grouping by one or more hold-constant fields (strata) so comparisons are made inside comparable subsets rather than across mixed populations.

Current seams that must be mirrored:

- Compare/control math: `analyzeCompareControlCategoricalRaw`, `analyzeCompareControlNumericRaw`, `analyzeCompareControlCategoricalControlled`, `analyzeCompareControlNumericControlled`, `analyzeCompareControlDiscovery`.
- Filter semantics: `compilePreviousRunsFilterPredicate`, `evaluatePreviousRunsFilterOperator`, `recordMatchesPreviousRunsFilterGroups`, quick-filter application in `computePreviousRunsFilterResult`.
- Derived value semantics: `previousRunsFieldValue`.
- Subset patch semantics: `syncCompareControlSelectionToTableFilters`.
- Existing behavior parity evidence: tests in `tests/analytics/test_stats_dashboard.py`.

Implementation implication: create one backend Python compare/control engine that owns field resolution, filtering, analysis, and subset patch generation, then call that engine from CLI commands.

## Milestones

### Milestone 1: Create backend engine contracts and row normalization

Create `cookimport/analytics/compare_control_engine.py` with explicit request/response types (TypedDict or dataclasses) for field catalog, filters, analysis requests, analysis responses, and structured errors. Add a row normalization layer that operates on `DashboardData.benchmark_records` and resolves the same derived fields as `previousRunsFieldValue(...)`.

Acceptance for Milestone 1 is that a tiny unit test can load synthetic rows, call the resolver for derived fields (`source_label`, `ai_model`, `ai_effort`, `all_token_use`), and match expected values exactly.

### Milestone 2: Implement backend filter semantics with parity-safe operators

Implement quick filters and column filters in Python using the same operator behavior as dashboard JS (`contains`, `not_contains`, `starts_with`, `ends_with`, `regex`, `gt`, `gte`, `lt`, `lte`, `eq`, `neq`, `is_empty`, `not_empty`). Ensure both per-field clause mode and global cross-field mode are supported.

Acceptance for Milestone 2 is parity tests that run identical synthetic cases through backend logic and expected truth tables, including invalid regex handling returning structured errors.

### Milestone 3: Implement compare/control analysis modes and recommendations

Implement deterministic analysis functions for `discover`, `raw`, and `controlled`. Raw mode must produce categorical group means/counts or numeric regression/correlation summaries. Controlled mode must reproduce exact-strata behavior for categorical and within-strata centering for numeric.

Add deterministic helper rankings for “hold constant” suggestions and split suggestions. Keep recommendations data-driven and explainable (score + retained coverage), with no LLM calls.

Acceptance for Milestone 3 is a fixture where raw and controlled conclusions differ in direction, proving confounder handling.

### Milestone 4: Add CLI command group for one-shot execution

In `cookimport/cli.py`, add a Typer subgroup named `compare-control` with `run` command. `run` accepts either `--query-file` JSON or explicit CLI flags, loads dashboard records via `collect_dashboard_data(...)`, applies filter/query payload, and prints JSON to stdout.

Acceptance for Milestone 4 is `cookimport compare-control run ...` returning valid JSON and non-zero exit for malformed top-level options only (analysis-domain errors stay inside structured JSON payloads).

### Milestone 5: Add persistent JSONL agent mode

Add `cookimport compare-control agent` that starts a read-eval loop on stdin/stdout. Each input line is one JSON request; each output line is one JSON response with preserved request `id`. Supported actions: `load`, `fields`, `discover`, `analyze`, `insights`, `suggest_hold_constants`, `suggest_splits`, `subset_filter_patch`, `reset`, `ping`.

Acceptance for Milestone 5 is an automated test that sends multiple requests in one process and receives ordered JSONL responses without restart.

### Milestone 6: Parity validation and docs

Add focused tests in `tests/analytics/test_compare_control_engine.py` and `tests/analytics/test_compare_control_cli.py`, then run existing dashboard analytics tests to guard against semantic drift.

Update docs in:

- `docs/02-cli/02-cli_README.md`
- `docs/08-analytics/08-analytics_readme.md`
- `docs/08-analytics/dashboard_readme.md`
- `docs/08-analytics/dashboard_howto_compare_control.md`

Acceptance for Milestone 6 is passing tests plus docs that clearly describe backend-vs-dashboard parity expectations and known edge cases.

### Milestone 7: Add QualitySuite bridge handoff for agents

Add bridge generation to `bench quality-run` and `bench quality-compare` so each run/comparison emits `agent_compare_control/` artifacts with:
- scope-index metadata (`qualitysuite_compare_control_index.json`),
- precomputed compare-control insights per scope/outcome,
- ready follow-up requests (`agent_requests.jsonl`) for `compare-control agent`,
- short usage note (`README.md`).

Acceptance for Milestone 7 is a quality-run/quality-compare test proving bridge writer hooks execute and docs describe the default handoff loop.

## Plan of Work

Implement in this sequence so each step is independently verifiable.

Start with pure backend analytics primitives. Add field catalog builder, value resolver, filter compiler, and analysis functions in `cookimport/analytics/compare_control_engine.py`. Keep the module free of Typer/CLI dependencies so tests can target math and filtering directly.

After the engine is stable, add CLI glue in `cookimport/cli.py`. Define a new Typer app for compare-control commands near other sub-app registrations, then wire two commands (`run`, `agent`) that parse request payloads, call the engine, and emit deterministic JSON.

Once command wiring exists, add robust protocol/error handling. Agent-mode requests must never crash the process on malformed payloads. Return error objects with a stable shape (`ok`, `error.code`, `error.message`, optional `error.details`) and include request `id` when present.

Finish with parity/regression tests and docs updates. Use existing dashboard behavior tests as semantic references, especially controlled categorical weighting and subset filter patch behavior.

## Validation and Acceptance

Acceptance is behavior-first and parity-focused.

- `cookimport compare-control run` returns deterministic JSON for `discover`, `raw`, and `controlled` views on the same benchmark history inputs.
- `controlled` responses include coverage diagnostics (`used_rows`, `candidate_rows`, `used_strata`, `total_strata`) and warning text when coverage is weak.
- `subset_filter_patch` returns clauses equivalent to dashboard “Filter to subset” behavior (`eq` clauses and `or` mode for selected groups).
- Agent mode handles multiple sequential requests and malformed lines without exiting the process.
- A confounded fixture demonstrates a raw-vs-controlled direction flip for at least one categorical compare field.
- `pytest tests/analytics/test_stats_dashboard.py -q` remains green to protect existing dashboard behavior.
- QualitySuite run/compare flows emit `agent_compare_control/` bundles by default and include deterministic agent handoff artifacts.

## Idempotence and Recovery

The new command path is read-only over analytics artifacts. It must not mutate history CSV, dashboard data JSON, or source run artifacts.

If a request is invalid (unknown field, bad regex, unsupported mode, missing required payload fields), return structured errors and keep agent mode alive.

If in-memory agent state becomes stale or corrupted, support `load` and `reset` actions that rebuild state deterministically from disk inputs.
