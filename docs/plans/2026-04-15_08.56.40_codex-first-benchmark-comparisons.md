# Make Codex-First Benchmark Comparisons The Default

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

`docs/PLANS.md` is the repository rulebook for this file and must be followed while this plan is active.

## Purpose / Big Picture

After this change, the normal single-book benchmark path will stop centering the question "is Codex better than vanilla?" and instead center "is the current Codex configuration better than the previous Codex configuration?" Vanilla will remain available only as an explicit diagnostic canary. A user should be able to run the interactive single-book benchmark and see one Codex-oriented run by default, neutral comparison artifact names, and docs that describe Codex baseline-versus-candidate as the main benchmark story.

## Context and Orientation

The main code path for this task lives in `cookimport/cli_support/bench_single_book.py`. That file plans the interactive single-book benchmark variants, runs the benchmark for each variant, writes the human-readable session summary, and publishes the comparison artifact that downstream bundle and cutdown tooling reads. Right now the helper treats a Codex-backed selection as a paired run: it constructs `vanilla` and `codex-exec` siblings, writes `codex_vs_vanilla_comparison.json`, and only creates the comparison payload when both paired variants succeed.

The benchmark contract documentation lives in `docs/07-bench/07-bench_README.md`. That doc currently states that Codex-backed single-book runs write sibling `vanilla/` and `codex-exec/` roots and that paired success emits `codex_vs_vanilla_comparison.json`.

Several downstream readers still assume the legacy artifact names. The highest-signal references found so far are `cookimport/bench/oracle_upload.py`, `cookimport/bench/followup_bundle.py`, `cookimport/bench/upload_bundle_v1_existing_output.py`, and the external-ai cutdown helpers under `cookimport/bench/external_ai_cutdown/`. Those readers should keep working for older paired artifacts, but new benchmark publication should stop presenting the legacy name as the primary contract.

For this plan, "baseline" means the comparison reference variant and "candidate" means the variant being evaluated. In a neutral comparison artifact those names are metadata labels, not a promise that one side is deterministic. "Vanilla" remains the deterministic or mostly deterministic canary variant and should only appear when an operator intentionally asks for it.

## Plan of Work

First, change `cookimport/cli_support/bench_single_book.py` so `_interactive_single_book_variants` no longer auto-expands a Codex-backed selection into `vanilla` plus `codex-exec`. The default should be a single variant whose slug is derived from the selected settings, while still preserving existing variant slugs for non-comparison reporting. After that, update the post-run comparison writer so it can emit a neutral comparison payload for any two successful variants and name them `baseline` and `candidate` in the artifact schema, while storing the real variant slugs in metadata.

Next, update the summary markdown and console output so they no longer talk about "Codex vs Vanilla" as the default story. Summary ordering should still be stable and readable, but it must not privilege `vanilla` and `codex-exec` as if they are the only meaningful variants. When a comparison artifact exists, the summary should describe it as a benchmark comparison rather than a Codex-versus-vanilla special case.

Then, update downstream readers to prefer the new neutral filenames and payload structure but fall back to `codex_vs_vanilla_comparison.json` and `changed_lines.codex_vs_vanilla.jsonl` when reading older bundles. This keeps current tooling useful during migration.

Finally, update the benchmark docs and add a short durable note in the owning folder so future agents know that Codex baseline-versus-candidate is the default benchmark shape and vanilla is diagnostic-only.

## Idempotence and Recovery

The code edits are additive and safe to rerun. If a neutral artifact reader misses an older legacy file, restore compatibility by adding explicit fallback-to-legacy filename resolution rather than reintroducing pair-first writes. If a summary or reader still assumes `vanilla` and `codex-exec`, prefer preserving the new neutral write path and widening the reader instead of rolling the product contract backward.
