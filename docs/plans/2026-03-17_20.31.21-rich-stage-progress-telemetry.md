# Add Rich Stage Progress Telemetry For Benchmark And Import Status

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

After this change, callback-driven benchmark and import status panels should keep useful per-stage detail visible instead of collapsing back to one plain line whenever a later update omits `running N` or when a stage never emitted worker stats in the first place. Operators should be able to watch benchmark/import work progress and see stable stage details such as task counts, configured workers, active workers, queued work, and current shard labels when a stage knows them.

From a user point of view, the visible result is simple: long-running phases like canonical line-role and knowledge harvest should keep a rich boxed spinner panel instead of alternating between a useful worker view and an empty one-line status. The same rich metadata should also be written into the `processing_timeseries*.jsonl` artifacts so later debugging does not depend on terminal screenshots.

## Surprises & Discoveries

- Observation: the current line-role direct runtime is not truly parallel in-process even when it advertises multiple workers.
  Evidence: `cookimport/parsing/canonical_line_roles.py` assigns worker buckets up front, then iterates `assignments` and `assigned_shards` sequentially while still emitting `running min(worker_count, total_shards)` at phase start.

- Observation: knowledge harvest has good runtime artifacts on disk (`phase_manifest.json`, `worker_assignments.json`, `telemetry.json`) but almost no live progress surface while it runs.
  Evidence: `cookimport/llm/codex_farm_knowledge_orchestrator.py` writes runtime artifacts and telemetry after the loop, but the loop itself never calls `progress_callback`.

- Observation: several benchmark/all-method wrappers currently preserve only worker-activity payloads as raw messages.
  Evidence: `cookimport/cli.py` `_variant_progress(...)` and `_emit_status(...)` special-case `parse_worker_activity(...)`; any new structured progress payload must be recognized there too or it will be prefixed/flattened into plain text and become unreadable.

- Observation: effective worker count is often lower than the saved worker setting because shard count is the real upper bound.
  Evidence: the knowledge-harvest test case requested `knowledge_worker_count=4`, but the emitted structured payload reported `worker_total=1` because only one shard existed after job building and `resolve_phase_worker_count(...)` clamps to shard count.

- Observation: the shared recipe phase-worker runtime can report real concurrent worker-bucket progress, but it cannot expose per-shard in-flight progress inside one worker bucket because each worker still makes one classic CodexFarm `process` call across its assigned shard set.
  Evidence: `run_phase_workers_v1(...)` now emits outer snapshots on assignment completion, but `_run_worker_assignment_v1(...)` still runs one `runner.run_pipeline(...)` call over one worker `in/` directory.

## Decision Log

- Decision: add one small repo-owned structured progress payload instead of trying to infer every future detail from free-form strings.
  Rationale: the current UI already relies on string parsing for backward compatibility, but stages like knowledge harvest simply do not emit enough information to recover richer stats. A compact serialized payload lets stage code send real metadata without inventing a new subsystem.
  Date/Author: 2026-03-17 / Codex

- Decision: keep backward compatibility with existing plain status strings and worker-activity payloads.
  Rationale: many current call sites already emit plain `task X/Y` strings and focused worker-activity rows. The new payload should be additive, not a repo-wide mandatory migration.
  Date/Author: 2026-03-17 / Codex

- Decision: make the shared status wrapper retain stage-local telemetry until the stage changes.
  Rationale: users care about continuity. If stage A reports configured workers once and later emits only `task 2/5`, the panel should not forget known state while stage A is still active.
  Date/Author: 2026-03-17 / Codex

- Decision: make the CLI render inferred `stage:` and `progress:` summary lines for any plain counter message, not only for structured payloads.
  Rationale: many existing import/plugin loops still emit ordinary `task X/Y` strings. Making those visibly richer in one shared place improves the whole spinner immediately while stage-specific structured emitters can land incrementally.
  Date/Author: 2026-03-17 / Codex

## Context and Orientation

The shared live spinner path lives in `cookimport/cli.py` inside `_run_with_progress_status(...)`. That function receives callback messages from benchmark/import stages, estimates ETA from `task X/Y`, renders boxed Rich status panels, and writes `processing_timeseries*.jsonl` rows. It currently understands plain counter text, `codex-farm ...` progress summaries, and serialized worker-activity payloads from `cookimport/core/progress_messages.py`.

The low-level rendering state lives in `cookimport/core/progress_dashboard.py`. `ProgressDashboardCore` stores a status line plus optional extra lines and worker lines. `ProgressCallbackAdapter` currently knows only two message shapes: ordinary strings and serialized worker-activity payloads.

Canonical line-role progress is emitted from `cookimport/parsing/canonical_line_roles.py`. Knowledge-harvest execution lives in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. Both stages already know their shard counts and worker assignments, so they are the right places to emit richer stage snapshots.

For this plan, “structured stage-progress payload” means a serialized callback message that carries the human-visible status text plus machine-readable fields like task counters, configured worker count, running worker count, active shard labels, and extra detail lines. The payload must still travel through the existing `progress_callback` plumbing.

## Plan of Work

First, extend `cookimport/core/progress_messages.py` with a new serialized payload family for stage-progress snapshots. It should mirror the worker-activity helper style: one formatter, one parser, JSON payload, ASCII-only. The payload should carry the rendered message plus optional machine-readable fields for stage label, task counter, configured worker count, running worker count, active task labels, and extra detail lines.

Second, update the shared consumers of progress callbacks. `cookimport/core/progress_dashboard.py` should continue handling worker-activity payloads, while `cookimport/cli.py` should parse the new stage-progress payload wherever `_run_with_progress_status(...)` or multi-book benchmark dashboards inspect callback text. The shared status wrapper should persist stage-local telemetry until a different stage label arrives, and `processing_timeseries*.jsonl` rows should include the richer fields when present.

Third, teach canonical line-role and knowledge-harvest runtimes to emit structured stage-progress payloads. The line-role phase should report total shards, configured worker count, active shard labels, queued shards, and completed shards. The knowledge-harvest phase should do the same around `_run_direct_knowledge_workers_v1(...)`, because today it emits almost nothing while work is actually happening.

Fourth, adjust focused tests and docs. Add regression tests for sticky stage telemetry, structured payload parsing, knowledge-harvest progress emission, and richer timeseries rows. Update the short CLI/LLM docs and add the required task/understanding notes once the behavior is real.

## Interfaces and Dependencies

In `cookimport/core/progress_messages.py`, define stable helpers for stage-progress payloads that are as simple as the existing worker-activity helpers. The parser must return a plain dictionary so `cookimport/cli.py` and dashboard helpers can consume it without new imports from benchmark-specific code.

In `cookimport/cli.py`, `_run_with_progress_status(...)` must continue to accept existing plain strings. The new behavior is additive: when a callback message is a stage-progress payload, the function should render the payload’s human text, use the structured fields for worker/detail lines and timeseries metadata, and keep those fields until a new stage supersedes them.

In `cookimport/llm/codex_farm_knowledge_orchestrator.py` and `cookimport/parsing/canonical_line_roles.py`, progress emission should stay local to the phase code. These modules already know shard counts, worker assignments, and the current shard, so they should emit complete stage snapshots rather than forcing the CLI to guess from filenames later.
