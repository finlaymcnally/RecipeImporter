# Split Recipe-Region Gating From Outside-Recipe Semantic Authority

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

After this change, Codex should stop being asked to solve two different problems with one mixed label pass. In prose-heavy books, it should no longer hallucinate recipe titles, how-to sections, or instruction lines out of memoir prose, blurbs, chapter headings, or front matter. In recipe-dense books, it should still fix real structure errors inside recipe regions. The user-visible result is that a book like `saltfatacidheatcutdown` can improve without sacrificing the gains already seen on `dinnerfor2cutdown`.

Someone validating this work should be able to rerun single-offline benchmarks and observe a more stable pattern. `dinnerfor2cutdown` should remain strong because recipe-local fixes are still allowed. `saltfatacidheatcutdown` should stop cratering because outside-recipe lines no longer compete directly with recipe labels unless a dedicated LLM gate first establishes that those lines belong to a real recipe region. The proof is behavioral: fewer `OTHER -> RECIPE_TITLE/HOWTO_SECTION/INSTRUCTION_LINE` false positives in prose-heavy books, while recipe-local labels stay strong in recipe-dense books.

## Surprises & Discoveries

- Observation: the same live Codex settings can help one book and hurt another without any config drift.
  Evidence: both benchmark roots used `codex-line-role-shard-v1`, `codex-knowledge-shard-v1`, `codex-recipe-shard-v1`, `gpt-5.3-codex-spark`, and `low` reasoning effort, yet `dinnerfor2cutdown` improved while `saltfatacidheatcutdown` cratered.

- Observation: the winning case is recipe-local, not global semantics.
  Evidence: in `dinnerfor2cutdown`, sampled flips mostly repaired `OTHER` or `KNOWLEDGE` into `INGREDIENT_LINE`, `INSTRUCTION_LINE`, and `RECIPE_TITLE`, which are local structure corrections near real recipes.

- Observation: the losing case is dominated by outside-recipe false promotions.
  Evidence: in `saltfatacidheatcutdown`, sampled flips promoted lines such as `SALT`, `What is Heat?`, `About the Author`, and long teaching or testimonial prose into `RECIPE_TITLE`, `HOWTO_SECTION`, `INSTRUCTION_LINE`, or `KNOWLEDGE`.

- Observation: the present line-role surface is still too mixed even after grounding improvements.
  Evidence: `cookimport/parsing/canonical_line_roles.py` still routes the full ordered candidate set into the line-role Codex runtime, and `cookimport/llm/canonical_line_role_prompt.py` still exposes the entire global label set in one prompt.

- Observation: the user’s product constraint rules out the obvious “heuristic judge” answer.
  Evidence: the user explicitly said deterministic scores and guesses are always worse than LLM guesses, so deterministic validation cannot be the authority that approves or vetoes model output.

- Observation: the right fix is task decomposition, not rollback logic.
  Evidence: the failure pattern is not “the model is slightly noisy”; it is “the model is solving the wrong mixed problem.” Splitting recipe-region detection from outside-recipe semantics removes the failure mode without demoting the LLM to a non-authoritative helper.

- Observation: Stage 7 can only collapse outside-recipe meaning to `knowledge|other`; it cannot rescue stray recipe-structure labels after line-role has already emitted them.
  Evidence: the nonrecipe seam records final outside-recipe authority in `08_nonrecipe_spans.json` and `09_knowledge_outputs.json`, but its final scored contract is still binary and does not project recipe labels back out. The line-role split therefore has to block outside-recipe recipe-label promotion before Stage 7.

## Outcomes & Retrospective

Implementation has landed for the task split itself. `codex-line-role-shard-v1` now runs two internal LLM phases: `recipe_region_gate` over the full ordered candidate set, then `recipe_structure_label` over recipe-positive, boundary, and immediately adjacent rows. ## Context and Orientation

This repository has three active Codex-backed surfaces relevant to this plan.

`cookimport/parsing/canonical_line_roles.py` owns canonical line-role labeling. Today it builds a deterministic baseline for every atomic line and, when `line_role_pipeline=codex-line-role-shard-v1`, sends the full ordered candidate set into the line-role Codex runtime. The live prompt is built by `cookimport/llm/canonical_line_role_prompt.py` and currently offers the full scored label set: `RECIPE_TITLE`, `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `HOWTO_SECTION`, `YIELD_LINE`, `TIME_LINE`, `RECIPE_NOTES`, `RECIPE_VARIANT`, `KNOWLEDGE`, and `OTHER`.

`cookimport/staging/nonrecipe_stage.py` owns Stage 7 non-recipe authority. It groups outside-recipe blocks into spans and records final `knowledge` versus `other` ownership in `08_nonrecipe_spans.json` and `09_knowledge_outputs.json`. `refine_nonrecipe_stage_result(...)` is the place where richer LLM review can change final outside-recipe authority while still keeping the external scored contract stable.

`cookimport/llm/codex_farm_knowledge_jobs.py` and the surrounding knowledge modules own the optional Codex-backed review of Stage 7 seed non-recipe spans. That stage already knows how to package non-recipe spans into LLM jobs and return richer internal reviewer categories while still collapsing to final `knowledge|other`.

The benchmark evidence matters because it shows exactly where the current shape fails. `dinnerfor2cutdown` is recipe-dense and benefits from local structure repair. `saltfatacidheatcutdown` is prose-heavy and suffers when title-like headings, blurbs, and explanatory prose are still allowed to compete with recipe labels. The redesign in this plan must therefore preserve recipe-local structure wins while sharply reducing outside-recipe structure invention.

For this plan, “recipe region” means a contiguous stretch of atomic lines that belongs to an active recipe and is therefore allowed to contain labels such as `RECIPE_TITLE`, `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `HOWTO_SECTION`, `YIELD_LINE`, `TIME_LINE`, `RECIPE_NOTES`, and `RECIPE_VARIANT`. “Outside-recipe semantic authority” means the final decision over whether non-recipe material is `knowledge` or `other`, plus any internal reviewer category such as `front_matter`, `chapter_taxonomy`, or `cooking_explanation` used to justify that final decision.

## Plan of Work

First, change the line-role surface so it no longer asks one prompt to solve both recipe-region detection and outside-recipe semantic classification. In `cookimport/parsing/canonical_line_roles.py`, keep the deterministic baseline build and the direct shard runtime, but insert an explicit LLM recipe-region gate before final recipe-structure promotion. The gate should review one ordered shard and return a compact per-line decision that says whether the line is inside an active recipe region, outside recipe, or still uncertain. This gate is still an LLM decision, not a deterministic veto. The deterministic baseline remains useful as a seed and a batching hint, but it must not be the final judge over the model.

Second, narrow the structure-correction prompt so it only applies the full recipe label set where recipe-region membership has already been established. In practice, this means `cookimport/llm/canonical_line_role_prompt.py` should either grow a second prompt mode or split into two prompt builders: one builder for recipe-region gating and one builder for recipe-structure labeling. The structure prompt should be allowed to emit recipe labels only on rows that the gate marked as recipe-positive or locally adjacent to recipe-positive rows when the same shard needs continuity. Rows that are outside recipe should not continue to compete for `RECIPE_TITLE`, `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `HOWTO_SECTION`, or `RECIPE_VARIANT`.

Third, make outside-recipe rows flow into Stage 7 authority instead of being over-decided by line-role. Keep `cookimport/staging/nonrecipe_stage.py` as the authoritative seam for final `knowledge|other`, but update the knowledge-stage bundle builder in `cookimport/llm/codex_farm_knowledge_jobs.py` so it can consume the new recipe-region decisions cleanly. The knowledge prompt should be specialized for non-recipe semantics: chapter taxonomy, endorsements, front matter, memoir/storytelling, cookbook teaching, practical advice, and similar categories should be easy for the model to express internally. The final scored output must still collapse to `knowledge` or `other`, but the prompt should stop pretending that recipe labels belong in this decision.

Fourth, update preview, telemetry, and artifact writing so the two-stage line-role flow stays observable. `cookimport/llm/prompt_preview.py`, the prompt artifact writers, the line-role runtime manifest, and any benchmark comparison helpers that assume a single line-role prompt family must be taught that the `codex-line-role-shard-v1` surface can now perform both `recipe_region_gate` and `recipe_structure_label` work under one public setting. The human-facing artifact contract should show that split rather than hiding it behind one stage row.

Fifth, update test coverage around the new seam split. `tests/parsing/test_canonical_line_roles.py` must prove the new prompt and promotion shape. `tests/staging/test_nonrecipe_stage.py` must prove that richer reviewer categories still stay internal and the final scored contract remains `knowledge|other`. `tests/llm/test_knowledge_job_bundles.py`, `tests/llm/test_prompt_preview.py`, and benchmark/helper tests under `tests/bench` and `tests/labelstudio` must prove that prompt planning, manifest writing, and benchmark artifacts still work when one public line-role surface internally emits two distinct LLM phases.

Finally, validate the redesign against the exact contrast that motivated it. Re-run `saltfatacidheatcutdown` and `dinnerfor2cutdown` in single-offline benchmark mode with the Codex-backed recipe, line-role, and knowledge paths enabled. The acceptance bar is asymmetric on purpose: `saltfatacidheatcutdown` must stop inventing recipe structure in outside-recipe prose, and `dinnerfor2cutdown` must keep its recipe-local gains. The plan is not complete if one benchmark improves only because the other one regressed into the same bland compromise.

## Milestones

### Milestone 1: Add an LLM recipe-region gate ahead of recipe-structure labels

At the end of this milestone, the line-role pipeline no longer jumps directly from a deterministic seed into the full global label set. Instead, it first runs a compact LLM gate that answers a narrower question: does this line belong to an active recipe region, outside recipe, or an uncertain boundary neighborhood? The gate is still evaluated on the same ordered shard slices used by the current direct runtime so prompt planning and cache boundaries stay familiar.

Acceptance for this milestone is prompt-level and runtime-level. A reader should be able to inspect line-role runtime artifacts and clearly see a recipe-region gate prompt and output. A prose-heavy heading such as `What is Heat?` or `About the Author` must no longer go straight from “title-like text” to a recipe label without first surviving recipe-region gating.

### Milestone 2: Restrict recipe-structure labels to recipe-positive regions

At the end of this milestone, the structure-correction pass is only allowed to emit recipe labels where the recipe-region gate already established that recipe structure is in play. The structure prompt should stay good at local corrections such as `OTHER -> INSTRUCTION_LINE`, `OTHER -> INGREDIENT_LINE`, and `KNOWLEDGE -> RECIPE_TITLE` when the surrounding lines clearly belong to a recipe. It should stop trying to solve cookbook-wide outside-recipe semantics.

Acceptance for this milestone is flip-level. On recipe-dense slices like `dinnerfor2cutdown`, the prompt should still recover recipe-local structure. On prose-heavy slices like `saltfatacidheatcutdown`, isolated outside-recipe headings and prose should stop producing recipe-label false positives because those rows are no longer eligible for recipe labels.

### Milestone 3: Hand outside-recipe meaning to Stage 7 semantic authority

At the end of this milestone, outside-recipe rows and spans are owned by the knowledge/non-recipe seam rather than by the line-role full-label prompt. `refine_nonrecipe_stage_result(...)` must remain the place where final `knowledge|other` authority changes are recorded. The LLM-facing knowledge job and prompt contract should be updated so the model can use reviewer categories that match cookbook reality while still emitting final scored `knowledge|other` decisions.

Acceptance for this milestone is artifact-level. A reader inspecting `08_nonrecipe_spans.json`, `09_knowledge_outputs.json`, refinement reports, and knowledge job payloads should be able to tell that non-recipe semantics were decided there, not smuggled in through recipe labels in the line-role prompt.

### Milestone 4: Update preview, observability, and benchmark plumbing

At the end of this milestone, prompt preview, prompt budget summaries, runtime manifests, benchmark summaries, and compare-control surfaces must describe the new two-stage line-role shape accurately. The public setting name can stay the same, but the artifacts should show both internal phases so it is obvious where tokens and label changes came from.

Acceptance for this milestone is tooling-level. Running prompt preview or inspecting benchmark artifacts should reveal `recipe_region_gate` and `recipe_structure_label` work instead of one opaque line-role bucket.

### Milestone 5: Prove the redesign on both motivating books

At the end of this milestone, `saltfatacidheatcutdown` should materially reduce outside-recipe recipe-label hallucinations, and `dinnerfor2cutdown` should retain its gains on recipe-local structure repair. Residual misses are acceptable, but they must become more interpretable and more tightly concentrated than the current “one mixed prompt did the wrong job” failure mode.

Acceptance for this milestone is benchmark-level. `saltfatacidheatcutdown` should stop showing broad `OTHER -> RECIPE_TITLE/HOWTO_SECTION/INSTRUCTION_LINE` false-positive patterns, and `dinnerfor2cutdown` should not give back the recipe-local wins that made Codex clearly helpful there.

## Validation and Acceptance

This change is accepted only if all of the following are true.

The repository-level conditions are:

- the new line-role tests pass and demonstrate the recipe-region gate plus structure-label split
- the Stage 7 / knowledge tests pass and still show that internal reviewer categories remain internal while final scored output stays `knowledge|other`
- prompt preview, runtime telemetry, and benchmark artifact surfaces stay readable and accurate with the new internal phase split

The benchmark-level conditions are:

- on `saltfatacidheatcutdown`, `codexfarm` no longer shows broad outside-recipe recipe-label hallucination as the dominant failure mode
- on `saltfatacidheatcutdown`, remaining misses are more concentrated in `knowledge` versus `other` semantics, not in recipe-structure invention
- on `dinnerfor2cutdown`, `codexfarm` still clearly preserves recipe-local structure wins over `vanilla`
- the redesign does not “solve” the problem by flattening both books into the same mediocre compromise

The artifact-level conditions are:

- line-role runtime artifacts clearly distinguish `recipe_region_gate` from `recipe_structure_label`
- Stage 7 artifacts remain the source of truth for final outside-recipe `knowledge|other`
- benchmark summaries and prompt budgets can explain where line-role tokens and label changes came from without manual archeology

## Idempotence and Recovery

The code changes in this plan are additive and can be retried safely. Prompt builders, schema files, and runtime manifests can be re-run without destructive migration. If a benchmark run fails midway, keep the timestamped run root and inspect the stage-local runtime artifacts rather than deleting the output. The benchmark evidence is part of the acceptance story for this plan and should remain available for comparison even when a run fails.
