# Comprehensively Harden Knowledge Runtime And CodexFarm Operator Surface

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md`.

This plan supersedes the narrower March 20 plans that separately targeted knowledge retry bounding, workspace watchdog policy, UI progress accuracy, and interrupted-run finalization, plus the overlapping late-evening consolidation drafts from the same Salt Fat postmortem. Those earlier files remain useful historical context, but this document is intended to be the single implementation guide for making the Codex-backed knowledge stage trustworthy end to end.

## Purpose / Big Picture

After this change, a Codex-backed knowledge stage should behave like a bounded data-processing system rather than a hopeful chat session. A worker that writes bad files should fail fast and clearly. A worker that writes good files should finish cleanly without lingering retries. A benchmark dashboard should tell the truth about what is happening. Stopping a broken run should leave a readable partial package instead of only raw rubble. The operator should be able to answer three questions from the saved artifacts alone: what happened, what succeeded, and what is still worth fixing.

The user-visible proof is concrete. Run the Salt Fat Acid Heat single-offline CodexFarm benchmark after implementation, or run the new zero-token replay and fake-runner tests if token spend is still disallowed. The knowledge stage should either finish with validated outputs and a truthful worker summary, or fail closed with explicit reason codes, bounded retry counts, and a partial benchmark package that is immediately readable. It should no longer get stuck in a state where the UI says `task 0/10 | running 10` while the runtime is actually thrashing through invalid outputs, stale repairs, and hidden worker states.

## Surprises & Discoveries

- Observation: the worst worker in the later Salt Fat run was not merely “off task”; it achieved the appearance of full productivity while producing nothing usable.
  Evidence: worker `001` wrote `49/49` top-level `out/*.json` files, but all `49/49` were invalid JSON because the literal heredoc terminator `EOF` was left in the file.

- Observation: the current knowledge runtime treats process exit too generously.
  Evidence: `cookimport/llm/codex_exec_runner.py` currently returns `completed_successfully()` when the subprocess exit code is `0`, and `cookimport/llm/codex_farm_knowledge_orchestrator.py` normalizes any completed worker with a missing reason code to `process_exited_without_watchdog_intervention`, even if the produced task outputs are junk.

- Observation: the existing output-stabilization check already parses candidate files as JSON objects, but it is only used to decide whether to stop a running worker, not whether to trust a finished worker.
  Evidence: `_summarize_workspace_output_paths(...)` in `cookimport/llm/codex_farm_knowledge_orchestrator.py` rejects missing, empty, and JSON-invalid files before returning `complete=True`, but the post-exit success path does not use that function as the final authority.

- Observation: the operator-visible `completed` state currently collapses together at least two very different outcomes: “outputs stabilized and the watchdog ended the worker cleanly” versus “the subprocess exited and the final chat output was malformed”.
  Evidence: workers `001` and `002` in the later Salt Fat run ended with `reason_code=process_exited_without_watchdog_intervention` and `final_agent_message_state=malformed`, while workers `004`, `007`, `009`, and `010` ended with `reason_code=workspace_outputs_stabilized`.

- Observation: the later run showed at least three distinct “success” classes that need to be separated in artifacts.
  Evidence: some workers ended with `workspace_outputs_stabilized`, some ended with `process_exited_without_watchdog_intervention` plus `final_agent_message_state=malformed`, and some wrote all task files but still had live repair follow-ups hanging at shutdown.

- Observation: a late watchdog or transport failure can disagree with the actual quality of the shard outputs.
  Evidence: worker `008` ended `watchdog_killed` even though `48/48` top-level task files existed and parsed, which means the runtime currently lacks a principled post-exit reconciliation step that can preserve validated work while still recording the late failure.

- Observation: the current recovery ladder confuses “one task is a near miss” with “the whole worker session is broken.”
  Evidence: the later run created `115` `repair_status.json` files and `75` `watchdog_retry/status.json` files, yet only `5` repairs ended `repaired` and only `1` watchdog retry ended `validated`.

- Observation: a shard can already be settled while child follow-up work still claims to be active.
  Evidence: worker `010` reached `workspace_outputs_stabilized`, but a repair follow-up still sat in `running` state afterward because the runtime never marked it superseded or stale-killed once the parent shard was done.

- Observation: the current operator surface hides the difference between shard-level progress and task-level churn.
  Evidence: the benchmark progress file stayed at `task 0/10 | running 10` for roughly eleven minutes even though the knowledge phase had already fanned out into `485` task packets and written `339` top-level task output files before termination.

- Observation: “workspace-local shell commands are broadly allowed” is still not enough to keep workers on task.
  Evidence: after the shell-shape policing relaxations, workers still used those allowed commands to mass-generate low-trust outputs, to overfit to hints and metadata, and to spend time on bulk scripting instead of grounded review.

- Observation: the current artifacts are rich enough to support a proper postmortem product, but there is no single summary file that stitches them together.
  Evidence: the worker trees contain events, live statuses, retry folders, usage, and raw outputs, yet the interrupted run lacked `phase_manifest.json`, `promotion_report.json`, `telemetry.json`, `failures.json`, and any concise run-level explanation of why the stage did not converge.

## Outcomes & Retrospective

This plan is newly authored and not yet implemented. Its main outcome so far is consolidation: the earlier investigation report and the narrower hardening plans identified real local fixes, but the later Salt Fat run showed that the knowledge stage still lacks a coherent success model. This document therefore reframes the task around authoritative output promotion, worker-session classification, bounded recovery, truthful operator reporting, and replayable evidence.

The expected retrospective target is not merely “fewer watchdog kills.” The target is that a future operator can look at one interrupted or completed benchmark root and immediately understand whether the knowledge stage finished correctly, failed closed, or was interrupted mid-recovery, without manually spelunking every worker folder.

## Context and Orientation

The relevant code is split across five surfaces.

The first surface is the knowledge stage runtime in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. In this repository, a “knowledge worker” is one long-lived Codex session launched inside a bounded local workspace. The repo writes authoritative task files into that workspace under `assigned_tasks.json`, `hints/*.md`, and `in/*.json`. The worker is supposed to write one output file per task under `out/*.json`. The orchestrator then reads those files, validates them, and decides whether to promote, retry, repair, or fail the work.

The second surface is the shared direct-exec wrapper in `cookimport/llm/codex_exec_runner.py`. That file launches `codex exec`, streams JSON events, builds live snapshots for the watchdog, and returns a `CodexExecRunResult`. Right now its `completed_successfully()` method still allows a zero exit code to look like success even when the worker’s outputs are unusable.

The third surface is the knowledge prompt contract in `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md` plus the workspace-worker prompt text built inside `cookimport/llm/codex_farm_knowledge_orchestrator.py`. The inline strict prompt asks for one compact single-line JSON object and forbids tools. The workspace prompt allows bounded local shell work and tells the worker to stop once every assigned `out/<task_id>.json` file exists and contains one complete JSON object. The later Salt Fat run shows that this contract is still too easy to satisfy in bad faith or by accident.

The fourth surface is benchmark orchestration and progress reporting in `cookimport/cli.py`, `cookimport/labelstudio/ingest.py`, and the benchmark artifact trees under `data/golden/benchmark-vs-golden/...`. This layer decides what the operator sees while a run is active and what summary files remain after interruption or failure.

The fifth surface is the evidence itself. The later run lives under:

    data/golden/benchmark-vs-golden/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/
    data/output/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-20_21.44.56/raw/llm/saltfatacidheatcutdown/knowledge/

The most important facts from that run that this plan must address are:

- `10` shard workers expanded into `485` task packets.
- `339` top-level task output files existed at shutdown.
- `4` main workers ended `watchdog_killed`.
- `2` main workers ended “completed” only because the process exited.
- `115` repair attempts were started and only `5` repaired anything.
- `75` watchdog-retry attempts were started and only `1` validated.
- `9` follow-up calls were still marked running when the run was killed.
- one worker wrote a complete flood of invalid JSON files while still looking “productive”.

Any unfamiliar term in this plan is used in a repo-local sense:

- “task packet” means one `assigned_tasks.json` entry with its own `in/<task_id>.json` and expected `out/<task_id>.json`.
- “promotion” means repo-owned code decides a worker-produced candidate is valid enough to become the authoritative task or shard result.
- “repair” means a repo-owned structured follow-up call against an invalid but near-miss task result.
- “watchdog retry” means a repo-owned structured retry after a worker session was killed or otherwise failed to produce a trusted answer.
- “systemic worker failure” means the worker session failed in a way that makes per-task repair obviously wasteful, such as writing uniformly malformed files across most or all of its tasks.

## Plan Of Work

### Milestone 1: Build a deterministic replay and evidence harness

Start by turning the current failure stories into executable local fixtures. Add or extend tests under `tests/llm/` and `tests/labelstudio/` so they can replay the later Salt Fat failure classes without spending tokens. The harness should model at least four specific patterns: a worker that writes JSON-invalid outputs with literal heredoc tails, a worker killed on forbidden commands, a worker that writes all outputs but leaves stale repairs hanging, and a benchmark progress stream that stays at shard-level `0/N` while task-level work is exploding underneath it.

This milestone matters because the current plan is intentionally broader than one bugfix. Without a replay harness, each future change will risk fixing one observed symptom while silently reintroducing another. The acceptance for this milestone is that the new tests fail before implementation and can distinguish the failure classes by name rather than by vague assertions like “the stage did not finish.”

### Milestone 2: Make authoritative task success repo-owned, not worker-owned

Redesign the main-worker output path so worker-written files are only candidates, never authority. The simplest acceptable implementation is to keep worker writes under `out/` but introduce an explicit repo-owned settlement pass that classifies every task after worker exit as one of: `validated`, `invalid_json`, `schema_invalid`, `semantic_invalid`, `missing_output`, or `abandoned_due_to_worker_failure`. A more ambitious but preferred implementation is to move worker writes to a staging directory such as `candidate_out/` and let repo code promote validated candidates into authoritative `out/` or `proposals/` files.

The critical rule is this: no worker session may be considered a clean completion unless every owned task is either validated or intentionally classified into a non-success bucket. A subprocess exit code of `0` is not enough. A worker with malformed final chat output is not enough. A full set of syntactically present files is not enough. Repo-owned validation must be the success boundary.

This milestone must also change live status semantics. `process_exited_without_watchdog_intervention` is acceptable only as a transport fact, not as a success signal. If outputs are invalid or missing, the final worker status must say so directly, for example `completed_invalid_outputs` or `completed_with_unsettled_tasks`, and downstream repair policy must treat that state as failure.

Add a post-exit reconciliation step that can classify contradictory cases honestly. If a worker dies late but all expected outputs were already repo-validated, mark the shard as something like `completed_posthoc_outputs_present` or `completed_validated_with_late_transport_failure` and record the watchdog fact as telemetry. If a worker exits cleanly but its task set is still unsettled, mark that explicitly as non-success. This is the concrete fix for the worker `008` pattern from the postmortem.

### Milestone 3: Detect and short-circuit systemic worker failure

Add a worker-session classifier that runs after the main worker exits and before per-task recovery is launched. This classifier should look at the whole worker assignment, not one task at a time. It should compute simple, deterministic facts such as total tasks, present outputs, JSON parse success count, schema-valid count, semantic-valid count, duplicate-output patterns, uniform label collapse, and whether every file appears to share the same malformed tail or template.

If the classifier concludes that the worker failed systemically, the runtime should not launch one repair or retry per task. Instead it should mark the entire worker assignment or parent shard set as failed closed with a small, bounded recovery policy. For example, it may allow one shard-level structured retry on a small sample or one replacement worker session, but it must not explode into dozens of independent repairs. Worker `001` from the later Salt Fat run is the concrete negative example: forty-nine broken files should have produced one systemic-failure classification, not thirty-eight failed repairs and one repaired task.

Acceptance for this milestone is behavior, not architecture purity. A replay of the `EOF` flood should produce a small number of follow-up attempts and a clear systemic-failure reason, never another repair storm.

### Milestone 4: Tighten the workspace-worker contract around safe output production

Once the runtime no longer trusts raw worker files, harden the prompt and watchdog so workers are nudged toward safer behavior. Update the workspace-worker prompt text in `cookimport/llm/codex_farm_knowledge_orchestrator.py` so it explicitly forbids writing final output files with heredocs, requires one task at a time, requires local self-validation before considering a task done, and explains that invalid files will be treated as task failure rather than as “close enough”.

Then add repo-owned guards for invalid-but-stable output. Today output stabilization only checks that files parse as JSON objects. Extend this to classify stable invalid outputs as a terminal failure state. If every expected task file exists but one or more remain JSON-invalid or schema-invalid across consecutive supervision snapshots, terminate the worker as failed and record `workspace_outputs_invalid` rather than waiting for the worker to wander or exit.

This milestone should not re-tighten the boundary rules back into shell-shape policing. The lesson from the later run is not “disallow shell again.” The lesson is “allowed shell work still needs a stronger definition of done.”

### Milestone 5: Rationalize watchdog retry and repair into a bounded recovery ladder

Refactor recovery into an explicit ladder with budgets. Each task and each worker assignment should have small, documented limits for repair and retry. Near-miss schema errors on a handful of tasks may justify one repair pass. A main worker killed early on one shard may justify one watchdog retry. A systemic worker failure should skip most task-level recovery entirely. A stale follow-up should be cancelable once its parent task or shard is already settled by another path.

At the end of this milestone, every follow-up attempt must have a parent reason, a budget slot, a pre-launch validity check, a silence timeout, and a cancel condition. The runtime must not allow “zombie” repairs to keep running after their shard is already completed. The later Salt Fat run showed exactly this failure: worker `007` had all outputs on disk and a completed shard status, yet a repair subcall kept running for roughly eight more minutes with no events. That must become impossible.

Implement the ladder with explicit subordinate states such as `repair_skipped_output_already_valid`, `retry_skipped_output_already_valid`, `superseded_by_workspace_output`, `superseded_by_shard_completion`, `stale_followup_canceled`, and `silence_timeout`. Before launching any repair or watchdog retry, re-check whether the task already has a valid authoritative result. After a shard settles, sweep any still-running follow-ups under that shard into a terminal superseded or canceled state. This is the concrete fix for the worker `010` pattern from the postmortem.

The acceptance condition is that replay fixtures can prove bounded counts. A run that formerly produced `115` repairs and `75` watchdog retries should now produce a small, explainable number of recovery attempts, with explicit statuses for skipped, canceled, stale, and abandoned follow-ups.

### Milestone 6: Make progress reporting honest at both shard and task granularity

Redesign the knowledge progress payload and CLI rendering so the operator can tell the difference between shard planning, active worker sessions, task settlement, and follow-up churn. Keep the operator surface simple, but stop pretending that “task 0/10 | running 10” is useful once the phase has already expanded into hundreds of task packets.

The UI should expose at least the following counts in plain language: completed shards, active shards, validated tasks, invalid tasks, tasks currently in repair, tasks currently in watchdog retry, workers completed, workers failed, workers completed with posthoc validation, and stale, superseded, or canceled follow-ups. For ten-worker benchmark runs, all ten workers must be visible. For larger counts, add a compact overflow summary rather than silently truncating.

This milestone should also update saved machine-readable progress artifacts so a postmortem tool can reuse the same truth. The goal is one consistent story between the terminal, `processing_timeseries_prediction.jsonl`, and the final summary files.

### Milestone 7: Make interruption and cancellation deterministic

Rework benchmark interruption so Ctrl+C or `KeyboardInterrupt` stops new work from being spawned, terminates active worker descendants, waits briefly for cleanup, and writes an explicit partial package. A later kill such as `SIGTERM` should still work, but the normal operator path must no longer be “Ctrl+C does not really stop the run.”

The partial package should be useful to a human. Under the benchmark root and the processed CodexFarm run root, write machine-readable interruption status, a short human-readable summary, known stage completion data, worker-state rollups, and pointers to raw artifacts. If prompt-budget data or partial evaluation data already exists, include it. If it does not, explain that it is absent because the run was interrupted before that stage settled.

Acceptance is a targeted test that simulates interruption after partial progress and asserts that no new knowledge workers are launched after the interrupt boundary and that the partial package files exist with `interrupted` status.

### Milestone 8: Add first-class postmortem and operator diagnostics

Build a run-level postmortem summary for the knowledge stage itself. The repo already writes enough local state to support this. Add a compact summary artifact such as `knowledge_stage_summary.json` and a short human-readable sibling file that reports worker outcomes, invalid-output counts, repair and retry counts, stale follow-ups, dominant failure reasons, and notable anomalies such as “worker produced files but all were JSON-invalid” or “worker completed while final agent message was malformed”.

This is not just for after-the-fact debugging. The summary should also provide the basis for benchmark-side top-level summaries and for future automated regression gating. A future operator should be able to open one file and immediately see whether the stage failed because workers drifted, because recovery churn exploded, because cancellation occurred, or because a specific policy gate fired too often.

### Milestone 9: Harden the quality boundary, not just the syntax boundary

The later run showed that some workers were not merely broken; they were producing low-trust bulk-generated outputs that happened to parse. Add deterministic quality sentinels that can reject obviously off-task or suspiciously templated outputs before they are treated as valid review. Examples include uniform “keep every block” collapse across many tasks, uniform “drop everything” collapse across a worker assignment, repeated generic reason strings, evidence snippets that simply echo all blocks without discrimination, or task outputs that obviously derive from hints and metadata rather than grounded review.

Do not make this a subjective LLM-on-LLM judgment. Keep it deterministic and auditable. The goal is not to infer human intent. The goal is to prevent blatantly low-quality but well-formed worker floods from being mistaken for good work. When these sentinels fire, they should feed the systemic-worker-failure classifier rather than launching dozens of repairs.

### Milestone 10: Refresh docs and keep the plan living during implementation

As each milestone lands, update this ExecPlan’s `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` sections. Once runtime behavior actually changes, update the short current-state docs in `docs/10-llm/10-llm_README.md`, `docs/07-bench/07-bench_README.md`, and any nearby CLI or benchmark docs that describe operator-visible statuses. Keep those doc changes short and factual. The heavy narrative should stay in this plan and in any final report generated from implementation.

## Validation and Acceptance

This plan is complete only when all of the following are demonstrably true.

First, the `EOF` flood is no longer possible as a false success. A replay test that writes `...}EOF` into every worker output file must end with a worker/session failure classification, not `completed`, and it must not explode into dozens of hopeless repairs.

Second, main-worker success is authoritative and explicit. A worker is considered successful only when its owned tasks have been settled by repo-owned validation and promotion. A clean subprocess exit with malformed final chat output or invalid files is not a success. A late watchdog or transport anomaly after all tasks are already validated is recorded as degraded telemetry, not as retroactive proof that the shard failed.

Third, recovery is bounded and explainable. A deterministic replay of the later Salt Fat failure story must show a small, budgeted set of follow-up attempts with clear statuses such as `repaired`, `retry_invalid`, `retry_skipped_systemic_failure`, `superseded_by_workspace_output`, `superseded_by_shard_completion`, `stale_followup_canceled`, `silence_timeout`, or `interrupted`, rather than a storm of unbounded repairs and retries.

Fourth, the progress UI tells the truth. A simulated ten-worker, hundreds-of-tasks knowledge run must show all configured workers, meaningful task-settlement counts, and visible retry/repair state. It must not sit indefinitely at shard-level `0/N` while hidden task churn continues.

Fifth, interruption is reliable. A targeted interruption test must prove that benchmark orchestration stops launching new knowledge work after the interrupt boundary and writes an explicit partial package with readable machine- and human-facing status files.

Sixth, postmortem artifacts are first-class. A finished or interrupted knowledge run must leave a concise stage summary that reports worker outcomes, invalid-output counts, repair and retry counts, stale follow-up counts, and dominant failure reasons.

Seventh, the quality boundary is stronger than mere JSON parsing. Deterministic sentinels must reject blatantly templated, collapsed, or otherwise low-trust outputs before they become authoritative.

## Idempotence and Recovery

The implementation should be additive and safe to repeat. Start by adding replay tests and new status categories before deleting or renaming any old ones. If a new settlement model proves too invasive, land it behind additive artifact fields first so old readers keep working while the new path is verified.

If the systemic-worker-failure classifier misfires during development, the safe fallback is conservative failure, not silent promotion. It is acceptable to fail closed and require manual rerun; it is not acceptable to accept junk as success because the classifier is uncertain.

If the interruption and descendant-kill path proves hard to wire through every live benchmark surface at once, land the partial-package emission first and document any remaining signal-handling gaps in this plan. Do not silently leave partial package writing coupled only to the happy path.

Do not mutate the saved March 20 evidence roots under `data/output/` or `data/golden/`. Tests should copy or synthesize their inputs. Those historical artifacts are the proof packets for this plan.
