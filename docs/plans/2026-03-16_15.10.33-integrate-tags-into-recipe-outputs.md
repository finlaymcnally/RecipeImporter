# Integrate Tags Into Recipe Outputs

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, recipe tagging stops being only a sidecar review artifact and becomes part of the actual recipe outputs. When a person runs `cookimport stage ... --llm-tags-pipeline codex-farm-tags-v1 --tag-catalog-json ...`, the accepted tags should appear directly inside the final cookbook3 recipe JSON and should also be mirrored into the intermediate schema.org-style JSON-LD in the safest standards-compatible way.

The observable behavior is:

- `final drafts/<workbook_slug>/r{index}.json` contains a stable structured tags field on `recipe`.
- `intermediate drafts/<workbook_slug>/r{index}.jsonld` contains matching `keywords`, plus optional `recipeimport:*` tag metadata if we choose to preserve richer detail there.
- `tags/<workbook_slug>/r{index}.tags.json`, `tagging_report.json`, and `tags/tags_index.json` still exist as audit and tuning artifacts.
- when these outputs are consumed by the cookbook app, import approval has a clear way to turn the accepted imported tags into canonical relational tag assignments.

The design goal is not to move tagging into the middle of importer logic. The design goal is to treat tagging as a recipe-metadata enrichment stage over the already-authoritative recipe outputs.

## Surprises & Discoveries

- Observation: intermediate schema.org output is less blocked than it first appears because `cookimport/staging/jsonld.py::recipe_candidate_to_jsonld(...)` already serializes `candidate.tags` into schema.org `keywords`.
  Evidence: the serializer writes `payload["keywords"] = ", ".join(candidate.tags)` when tags are present.

- Observation: final cookbook3 output has no structured tag field today, but the model boundary is permissive enough to add one without a large model refactor.
  Evidence: `cookimport/core/models.py::RecipeDraftRecipeMeta` uses `extra="allow"`, and `cookimport/staging/draft_v1.py::recipe_candidate_to_draft_v1(...)` currently only injects tags into `recipe.notes`.

- Observation: the current stage tags pass intentionally runs at the run root after `final drafts/` already exist, which is the cleanest convergence point for split and non-split runs.
  Evidence: `cookimport/tagging/orchestrator.py::run_stage_tagging_pass(...)` walks `run_root / "final drafts"` and writes run-level artifacts under `run_root / "tags"`.

- Observation: moving tagging into per-file import-time writers would make split-job behavior harder, not cleaner.
  Evidence: the current architecture in `cookimport/staging/import_session.py` writes final drafts inside the per-file session, while the tags pass is only invoked later from `cookimport/cli.py` at the stage-run root once all outputs have converged.

- Observation: benchmark wiring deliberately excludes stage-only tags knobs from `labelstudio_benchmark(...)`, so this feature should be designed as stage output enrichment first, not benchmark feature work.
  Evidence: `cookimport/config/run_settings_adapters.py` excludes `codex_farm_pipeline_tags` from benchmark kwargs, and `cookimport/config/CONVENTIONS.md` documents that tags knobs are stage-only.

- Observation: the cookbook app does not currently consume tags from `RecipeDraftV1`; it stores canonical tags in relational taxonomy tables and mutates them through dedicated RPCs.
  Evidence: `~/projects/cookbook/lib/contracts/recipeDraftV1.ts` is strict and has no `tags` field, `~/projects/cookbook/lib/recipes/readModelToDraft.ts` maps no tag data into the draft, and `~/projects/cookbook/app/protected/recipes/actions.ts` calls `set_recipe_tags(...)` / `set_recipe_tags_by_ids(...)` separately from recipe saves.

- Observation: cookbook import approval is already a translation seam that strips staging-only fields before canonical save, which makes it the right downstream place to materialize imported tags into canonical assignments.
  Evidence: `~/projects/cookbook/lib/contracts/recipeDraftStagingV1.ts` allows staging-only metadata, and `~/projects/cookbook/db/schema.snapshot.sql::approve_staging_recipe(...)` cleans staging fields and forwards the draft into `save_as_new_recipe(...)`.

## Decision Log

- Decision: the primary embedded home for recipe tags will be final cookbook3 output, not just schema.org JSON-LD.
  Rationale: the final cookbook3 draft is the repo’s main consumer-facing recipe object, while the intermediate JSON-LD is a standards-shaped projection that should stay conservative.
  Date/Author: 2026-03-16 / Codex

- Decision: the implementation will keep the current run-level tags pass timing instead of moving tagging into importer-time recipe conversion.
  Rationale: the existing run-root pass already solves split/non-split convergence and already uses the final-draft JSON shape as its signal source.
  Date/Author: 2026-03-16 / Codex

- Decision: schema.org output will default to `keywords` for accepted tags and will not automatically map arbitrary catalog tags into `recipeCategory`, `recipeCuisine`, or other narrower schema.org fields.
  Rationale: the catalog is broader than those schema.org fields. A blind category mapping would encode wrong semantics. If such mapping is desired later, it should be an explicit table with tests.
  Date/Author: 2026-03-16 / Codex

- Decision: sidecar `*.tags.json` and run-level tag reports remain part of the live contract even after tags are embedded into recipes.
  Rationale: they are still the natural place for suggestion provenance, category/source/confidence details, LLM validation counters, and tuning artifacts.
  Date/Author: 2026-03-16 / Codex

- Decision: final cookbook3 should expose two levels of tag data: a simple stable field for downstream use and a richer optional field for provenance.
  Rationale: consumers should be able to read `recipe.tags` without understanding the full tagging report format, while power users and debug tooling still need structured details.
  Date/Author: 2026-03-16 / Codex

- Decision: cookbook integration should use a staging-to-relational translation seam, not treat imported tags as canonical `RecipeDraftV1` fields on the website side.
  Rationale: the cookbook app already treats tags as separate taxonomy assignments for search, facets, and recipe pages. Importing tags through staging metadata and approval keeps that architecture intact and avoids a broad refactor of draft contracts, read models, save RPCs, and editor state.
  Date/Author: 2026-03-16 / Codex

- Decision: `recipeimport` should still embed tags in its own final outputs even though the cookbook app will likely materialize them relationally later.
  Rationale: embedded tags make `recipeimport` outputs self-describing, improve local debugging/review, and provide a stable handoff artifact to any downstream consumer, including but not limited to the cookbook app.
  Date/Author: 2026-03-16 / Codex

## Outcomes & Retrospective

Pending implementation.

The intended completed state is one canonical accepted tag set per recipe that is projected consistently into:

## Context and Orientation

There are four important concepts in this repository for this task.

The first is a `RecipeCandidate`. This is the internal normalized recipe object produced by deterministic import/conversion and optionally corrected by the recipe Codex pipeline. It is defined in `cookimport/core/models.py`. It already has a `tags: list[str]` field alongside schema-oriented fields such as `recipeCategory`, `recipeCuisine`, and `cookingMethod`.

The second is the intermediate schema.org-style output. It is written by `cookimport/staging/writer.py::write_intermediate_outputs(...)`, which calls `cookimport/staging/jsonld.py::recipe_candidate_to_jsonld(...)`. That serializer already emits `candidate.tags` as schema.org `keywords`.

The third is the final cookbook3 output. It is written by `cookimport/staging/writer.py::write_draft_outputs(...)`, which calls `cookimport/staging/draft_v1.py::recipe_candidate_to_draft_v1(...)`. Today that serializer does not expose a structured tags field. Instead, if `candidate.tags` exists, it appends a human-only `Tags: ...` line into `recipe.notes`.

The fourth is the current optional stage tags pass. It lives in `cookimport/tagging/orchestrator.py::run_stage_tagging_pass(...)`. That function walks `run_root / "final drafts"`, computes deterministic plus optional LLM tag suggestions from each final draft JSON, and writes:

- `tags/<workbook_slug>/r{index}.tags.json`
- `tags/<workbook_slug>/tagging_report.json`
- `tags/tags_index.json`
- `raw/llm/<workbook_slug>/tags/{in,out}/*.json`
- `raw/llm/<workbook_slug>/tags_manifest.json`

The current pain is that these three worlds do not line up. The recipe object knows how to carry tags, one serializer partly exposes them, another buries them in notes, and the stage tags pass computes the most complete accepted tag set only after recipe outputs are already on disk.

There is also an important downstream constraint from the cookbook app. Its canonical recipe draft contract does not currently include tags. Instead, recipe pages, search, and filters read tags from relational tables (`recipe_tags`, `recipe_tag_assignments`, `tag_categories`), and tag changes are written through dedicated RPCs. Its staging approval path already strips staging-only fields and translates staging data into canonical recipe writes. So the full end-to-end design is:

- `recipeimport` embeds accepted tags into its own outputs,
- cookbook import staging carries those imported tag proposals for review/debug,
- cookbook approval maps accepted tag keys into canonical relational assignments.

This plan resolves the mismatch without changing the high-level stage ordering on either side.

## Milestone 1: Define the Output Contract

At the end of this milestone, the repository has an explicit answer to “where do recipe tags live?” Final cookbook3 output becomes the primary embedded home. Intermediate JSON-LD mirrors the same accepted tags using `keywords`, and sidecar `*.tags.json` remains the provenance artifact.

The important design choice in this milestone is the cookbook3 field shape.

Implement the following contract:

- `final drafts/<workbook_slug>/r{index}.json` gains `recipe.tags`, a flat ordered list of accepted tag keys.
- `final drafts/<workbook_slug>/r{index}.json` may also gain `recipe.tag_details` (or `recipe.tagging`) as a richer list/object carrying `tag_key`, `category_key`, `confidence`, `source`, and `llm_pipeline_id`.
- `intermediate drafts/<workbook_slug>/r{index}.jsonld` gets the same accepted tags via standard `keywords`.
- If richer intermediate metadata is desired, add a repo-owned extension field such as `recipeimport:tags` or `recipeimport:tagDetails` rather than overloading unrelated schema.org fields.

Make the flat field mandatory for this milestone. Make the rich field part of the same milestone if it can be done cleanly without ambiguity. If there is any uncertainty, prefer landing `recipe.tags` and `keywords` first, then adding the richer embedded provenance in the same feature branch only after tests are in place.

The files that must reflect this contract are:

- `cookimport/core/models.py`
- `cookimport/staging/draft_v1.py`
- `cookimport/staging/jsonld.py`
- `docs/05-staging/05-staging_readme.md`
- `docs/09-tagging/09-tagging_README.md`

Acceptance for this milestone is behavioral, not just structural: a recipe that already has `candidate.tags` before the stage tags pass should serialize those tags structurally in final output, not only as text inside notes.

## Milestone 2: Build Run-Level Tag Enrichment

At the end of this milestone, the stage tags pass still runs after final drafts are written, but its accepted tag set is projected back into the recipe outputs so the run’s recipes themselves carry the tagging result.

Do not move the stage tags pass into `cookimport/staging/import_session.py`. Keep the current timing and convergence point in `cookimport/cli.py` after all stage outputs have been written. This preserves the clean split/non-split boundary and avoids teaching per-file sessions about run-wide tag catalogs and late writeback semantics.

Instead, add a new enrichment step adjacent to `run_stage_tagging_pass(...)`. The cleanest concrete shape is:

1. `run_stage_tagging_pass(...)` keeps computing the accepted suggestions and keeps writing sidecar artifacts exactly as it does now.
2. The function grows a machine-friendly return payload that includes enough information to update recipe outputs by workbook and recipe file name or by recipe id.
3. A new helper, likely in `cookimport/tagging/orchestrator.py` or a new small module such as `cookimport/tagging/stage_projection.py`, loads the matching final draft and intermediate JSON-LD files and applies the accepted tags to them.
4. `cookimport/cli.py` invokes that helper immediately after the tags pass succeeds and before stage observability / prompt-artifact closeout finishes.

The matching key must be explicit and deterministic. The safest primary key is the recipe file stem (`r0`, `r1`, and so on) scoped by workbook, because the tags pass already walks the final-drafts files by path. If recipe ids are always stable and present in both sidecar and recipe outputs, they can also be persisted as a stronger secondary key, but do not rely on heuristic title matching.

The enrichment helper should update both output families together:

- final cookbook3: set `recipe.tags` and, if implemented, `recipe.tag_details`
- intermediate JSON-LD: set `keywords` and optional `recipeimport:*` tag metadata

This helper should be idempotent. Re-running it over the same artifacts should produce the same serialized files and should not duplicate tags or accumulate repeated `Tags:` note lines.

As part of this milestone, stop using notes as the primary carrier for tags in `recipe_candidate_to_draft_v1(...)`. If the repository wants to preserve a human-readable tags summary in notes, it should be derived from the structured field in one place and must not become a second divergent source of truth. The preferred outcome is to remove the `Tags: ...` notes injection entirely once `recipe.tags` exists.

The files most likely touched are:

- `cookimport/tagging/orchestrator.py`
- `cookimport/cli.py`
- possibly a new module under `cookimport/tagging/`
- `cookimport/staging/draft_v1.py`
- `cookimport/staging/jsonld.py`

Acceptance for this milestone is a full stage run with `llm_tags_pipeline != off` where the sidecar tag files and embedded recipe tags agree exactly.

## Milestone 2.5: Define Cookbook Import Materialization

At the end of this milestone, the recipeimport-side output contract is aligned with the cookbook site's actual tag model. This milestone is downstream-facing design work, but it needs to be explicit now so the recipeimport output fields are shaped correctly.

The recommendation is:

- keep canonical cookbook tags relational,
- do not make cookbook `RecipeDraftV1` the primary tag carrier for this feature,
- carry imported accepted tags through cookbook staging metadata,
- materialize them during `approve_staging_recipe(...)`.

Concretely, plan for the cookbook side to add a staging-only tag proposal block, most likely under the staging payload rather than the canonical draft payload. Example conceptual shapes:

- `recipeimport` final output:
  - `recipe.tags: ["weeknight", "pressure-cook", "chicken"]`
  - `recipe.tag_details: [{ tag_key, category_key, confidence, source, accepted }]`
- cookbook staging payload extension:
  - `import_tag_proposals: [{ tag_key, category_key, confidence, source }]`
  - or `recipe.import_tags` if you prefer keeping all imported recipe metadata under `recipe`

The cookbook approval path should then:

1. load the staging tag proposal block,
2. create or resolve matching canonical `recipe_tags` rows by `key_norm`,
3. assign them to the newly approved recipe via `set_recipe_tags(...)` or `set_recipe_tags_by_ids(...)`,
4. leave canonical recipe draft reads unchanged, since the site already fetches tag assignments separately.

This milestone matters because it answers a design question that was still open before inspecting the cookbook repo: whether the website should be changed to read tags from embedded recipe JSON or keep its existing relational taxonomy model. The current recommendation is to keep the relational model.

## Milestone 3: Tests, Docs, and Compatibility

At the end of this milestone, there is focused proof that the feature works and that it does not regress current stage boundaries.

Add tests in three layers.

The first layer is serializer tests. Add or extend unit-style tests around `recipe_candidate_to_draft_v1(...)` and `recipe_candidate_to_jsonld(...)` to prove that `candidate.tags` serializes into `recipe.tags` and `keywords` respectively. These tests should also prove that the old `Tags: ...` notes-only behavior is removed or intentionally constrained.

The second layer is stage tags integration tests. Extend `tests/tagging/test_tagging.py` so a run-level tags pass not only writes `tags/*.tags.json` but also updates `final drafts/...` and `intermediate drafts/...`. This should assert:

- the enriched final draft has `recipe.tags`
- the enriched intermediate draft has matching `keywords`
- the sidecar `*.tags.json` still exists
- the accepted tag keys match across all three projections

The third layer is stage/CLI contract tests. Update or add a focused stage run test under `tests/staging/` or `tests/cli/` that exercises a small stage run with tagging enabled and verifies the complete run artifact shape still matches the documented contract.

The fourth layer is downstream contract coverage. This does not all need to land in this repository, but the implementation sequence should reserve room for cookbook-side tests that prove:

- staging payloads can carry imported tag proposals,
- approval maps those proposals into `recipe_tag_assignments`,
- recipe pages and search facets continue to read tags from relational tables, unchanged.

Update docs only where behavior changed:

- `docs/05-staging/05-staging_readme.md` should describe tags as embedded recipe metadata plus sidecar artifacts.
- `docs/09-tagging/09-tagging_README.md` should stop implying that tags only live in `tags/*.tags.json`.
- `docs/10-llm/tags_pass.md` should describe the embedded-output projection and the sidecar audit role.
- if needed, `docs/AI_Context.md` should mention that final recipe outputs can now contain structured recipe tags.

Compatibility should be explicit. Archived old runs without embedded recipe tags remain valid historical artifacts. The new feature only guarantees that new stage runs with current code embed tags into outputs.

For cookbook integration, compatibility is also explicit: the recommended path is additive staging metadata plus approval-time assignment, so existing canonical recipe reads and writes remain unchanged until that downstream work lands.

## Validation and Acceptance

This change is accepted when all of the following are true for a tagged stage run.

First, the final cookbook3 file contains a stable structured tag field on `recipe`. A human should be able to open `final drafts/<workbook_slug>/r{index}.json` and see a machine-readable tag list without reading the sidecar artifact.

Second, the intermediate JSON-LD contains matching accepted tags through `keywords`. If the implementation also adds `recipeimport:*` tag detail metadata, that metadata must match the accepted suggestions from the sidecar artifact.

Third, the sidecar tag artifacts still exist and remain the provenance source. `tags/<workbook_slug>/r{index}.tags.json` must still include source/category/confidence detail, and `tags/tags_index.json` must still summarize the run.

Fourth, no new dual-source drift is introduced. The flat tag set in final output, intermediate output, and sidecar output must match exactly for each tagged recipe in the test fixture.

Fifth, runs with `llm_tags_pipeline=off` remain valid and should not create empty tag fields unless the source recipe already carried tags in `RecipeCandidate.tags`. In that case the serializer should still project those existing tags structurally.

## Idempotence and Recovery

The enrichment step must be repeatable. Running it again over the same stage outputs should not duplicate tags, should not append repeated note lines, and should not reorder accepted tags nondeterministically.

Keep rollback simple by containing all new behavior behind the existing stage tags boundary. If the enrichment helper turns out to be unstable, it should be removable without breaking the sidecar tags pass itself. This means the helper should be invoked from a narrow call site after `run_stage_tagging_pass(...)`, not woven deeply into unrelated writer code paths on the first pass.

Historical outputs do not need migration. This plan only changes newly produced stage outputs.

## Interfaces and Dependencies

The repository must end this work with the following concrete interfaces.

In `cookimport/core/models.py`:

- `RecipeCandidate.tags` remains the canonical flat tag list before serialization.
- `RecipeDraftRecipeMeta` defines a stable embedded tags field such as `tags: list[str]`.
- If rich embedded provenance is added, define a dedicated model for it instead of leaving it as an untyped loose dict.

In `cookimport/staging/draft_v1.py`:

- `recipe_candidate_to_draft_v1(...)` serializes `candidate.tags` into the final recipe object.
- The function no longer relies on `notes` as the primary recipe tags contract.

In `cookimport/staging/jsonld.py`:

- `recipe_candidate_to_jsonld(...)` emits the accepted tag set through `keywords`.
- Optional repo-owned extension metadata for richer tag detail, if added, must live under `recipeimport:*`.

In `cookimport/tagging/orchestrator.py`:

- `run_stage_tagging_pass(...)` still produces sidecar tags artifacts.
- The accepted suggestion set is exposed in a machine-friendly shape that can be projected back into recipe outputs.

In `cookimport/cli.py`:

- the stage closeout sequence remains: write outputs, then run tags stage, then project tags into staged recipe outputs, then finish observability/prompt-artifact closeout.

For downstream cookbook integration, the intended concrete interfaces are:

- recipeimport handoff data includes a stable flat tag list plus optional rich proposal detail,
- cookbook staging contract accepts imported tag proposal metadata without widening canonical recipe draft semantics,
- cookbook approval writes canonical `recipe_tag_assignments`,
- cookbook page/search reads remain relational and do not depend on embedded draft tags.
