# Fix Knowledge Stage Schema And Shard Integrity Failures

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

Today the knowledge stage can fail in two different ways under real benchmark runs. One failure class blocks the stage before any shard completes because the structured-output schema is invalid. The other failure class bills real model tokens but produces invalid shard outputs because knowledge bundles get too large and the model drops chunk results, returns empty result lists, or invents fallback chunk ids. After this change, a human should be able to run the focused knowledge tests and see that both failure classes are prevented locally: the shipped schema satisfies the strict nested structured-output contract, large knowledge books split into smaller, bounded shards even when `knowledge_prompt_target_count` is set, and invalid knowledge shard outputs are caught by red tests and either prevented by prompt/pack changes or surfaced by stronger validation-oriented integration tests.

The most important user-visible outcome is reliability. A benchmark run with knowledge enabled should no longer fail because of the old nested-schema bug, and it should no longer funnel dozens of chunks into one unstable knowledge shard merely because a target prompt count was requested. The fix must be implemented with red/green TDD: each failure mode gets a failing test first, then the smallest production change that makes it pass.

## Surprises & Discoveries

- Observation: the recent knowledge bugs are not one bug but two.  
  Evidence: `2026-03-18_21.10.12` failed on live schema validation before any shard completed, while `2026-03-18_21.17.59` ran and billed `1,601,808` knowledge tokens across five calls but validated only one shard.

- Observation: the largest practical root cause for the second failure class is in bundling policy, not just prompt wording.  
  Evidence: `_bundle_prepared_chunks(...)` in `cookimport/llm/codex_farm_knowledge_jobs.py` sets `effective_max_bundle_chars` to `total_chars + 1` and `effective_max_gap_blocks` to `10**9` whenever `target_bundle_count` is not `None`, which turns the configured prompt target into permission to ignore the normal size limits.

- Observation: existing tests currently encode the too-large-shard behavior as acceptable.  
  Evidence: `tests/llm/test_knowledge_job_bundles.py::test_build_knowledge_jobs_target_prompt_count_overrides_char_cap` currently expects `target_prompt_count=5` to win even when ten chunks of about eight thousand characters each exceed the nominal per-bundle char cap.

- Observation: the failing real-world shard outputs were structurally different, not all the same.  
  Evidence: `saltfatacidheatcutdown.ks0000.nr` and `ks0002.nr` each dropped one owned chunk; `ks0003.nr` returned `r: []`; `ks0001.nr` returned a synthetic `cid: "processing_error"` plus out-of-surface evidence index `0`.

- Observation: the validator already caught the bad outputs correctly.  
  Evidence: the benchmark artifacts record `missing_owned_chunk_results`, `unexpected_chunk_results`, and `snippet_evidence_out_of_surface` exactly where expected. The missing piece is better prevention and better coverage, not a weaker validator.

- Observation: we did not need new ingest error codes to cover the benchmark fallback objects.  
  Evidence: the new benchmark-shaped tests pass with the existing validator error set: empty `r` still reports `missing_owned_chunk_results`, and synthetic `processing_error` rows report `missing_owned_chunk_results` + `unexpected_chunk_results` + `snippet_evidence_out_of_surface`.

- Observation: restoring the hard char cap still leaves prompt-target planning useful for moderate bundles.  
  Evidence: `tests/llm/test_knowledge_job_bundles.py::test_build_knowledge_jobs_can_exceed_target_prompt_count_to_respect_char_cap` now proves a six-chunk workload with `target_prompt_count=2` settles at three bundles of two chunks each instead of degenerating into one-chunk-only shards.

## Decision Log

- Decision: keep the already-landed schema fix as part of this broader reliability plan instead of treating it as unrelated.  
  Rationale: the user asked for a plan to fix all of the recent knowledge failures, and the March 18 runs showed both the schema bug and the shard-integrity bug in immediate succession.
  Date/Author: 2026-03-18 / Codex

- Decision: treat the hard bundle char cap and gap cap as true safety limits even when `knowledge_prompt_target_count` is provided.  
  Rationale: the benchmark evidence shows that using `knowledge_prompt_target_count` as permission to disable those limits is the fastest path to unstable, oversized shards. Prompt count should remain a soft planning target, not a license to exceed hard safety bounds.
  Date/Author: 2026-03-18 / Codex

- Decision: use red/green TDD in three layers: validator tests, bundling tests, and orchestrator integration tests.  
  Rationale: the failures span three seams. Validator tests catch exact shard-output integrity, bundling tests prevent giant shards from being produced, and orchestrator tests verify run-level reporting and wiring.
  Date/Author: 2026-03-18 / Codex

- Decision: tighten the prompt contract but do not rely on prompt-only fixes.  
  Rationale: the prompt absolutely should forbid empty `r` and synthetic `cid`s, but the strongest evidence points to oversized shard construction as the upstream trigger. Prompt hardening alone is not enough.
  Date/Author: 2026-03-18 / Codex

- Decision: do not introduce live-token benchmark reruns as the primary acceptance gate for this work.  
  Rationale: the repo already has deterministic local seams that can prove the bug fixes without spending more model tokens. A future live rerun can be a confidence check, not the only proof.
  Date/Author: 2026-03-18 / Codex

- Decision: keep the validator semantics stable and add coverage instead of inventing benchmark-specific error names.  
  Rationale: the existing error set already describes the real integrity violations precisely enough for failures.json, proposal wrappers, and orchestrator reporting. New tests were sufficient; extra error taxonomy would add noise without improving diagnosis.
  Date/Author: 2026-03-19 / Codex

## Context and Orientation

The knowledge stage in this repository is the optional large-language-model review pass that revisits non-recipe cookbook text and decides which chunks contain reusable cooking knowledge. The active implementation spans four main areas.

`cookimport/llm/codex_farm_knowledge_jobs.py` builds the model-facing knowledge shard inputs. In this repository a "knowledge shard" means one model call that owns a list of chunk ids under bundle key `bid` and chunk list `c`. The most important function there is `_bundle_prepared_chunks(...)`, because it decides how many chunks and how much text go into one model call.

`cookimport/llm/codex_farm_knowledge_ingest.py` validates model outputs. In this repository "validator" means the function that checks whether a shard output covers exactly the owned chunk ids and whether any snippet evidence points outside the shard's allowed block surface. The key function is `validate_knowledge_shard_output(...)`.

`cookimport/llm/codex_farm_knowledge_orchestrator.py` is the run-level seam. It builds knowledge jobs, runs the direct Codex worker path, records status in manifests, and promotes validated outputs into the staged non-recipe knowledge results.

The shipped knowledge prompt and schema live under `llm_pipelines/`. `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md` is the natural-language instruction wrapper, and `llm_pipelines/schemas/recipe.knowledge.v1.output.schema.json` is the structured-output schema used by the live direct-exec runtime.

The most relevant tests are:

- `tests/llm/test_llm_pipeline_pack_assets.py` for shipped schema-pack contract checks.
- `tests/llm/test_knowledge_output_ingest.py` for exact shard-output validation.
- `tests/llm/test_knowledge_job_bundles.py` for bundling and shard-size behavior.
- `tests/llm/test_codex_farm_knowledge_orchestrator.py` for run-level knowledge-stage behavior.

Two benchmark runs matter for this plan:

- `data/golden/benchmark-vs-golden/2026-03-18_21.10.12` showed the nested-schema bug.
- `data/golden/benchmark-vs-golden/2026-03-18_21.17.59` showed the oversized-shard / output-integrity failures.

The detailed artifact analysis for both runs is preserved in `docs/reports/2026-03-18_21.49.35-knowledge-stage-failure-report.md`. This plan repeats the needed context instead of assuming the reader has that report open.

## Plan of Work

The work should happen in four milestones, each implemented test-first.

The first milestone hardens validation coverage. Extend `tests/llm/test_knowledge_output_ingest.py` and `tests/llm/test_knowledge_phase_workers.py` so the repository has explicit red tests for the exact bad shapes seen in the benchmark artifacts: one missing owned chunk from an otherwise valid result set, an empty `r` array when owned chunks exist, and a synthetic fallback chunk id such as `processing_error` that also cites out-of-surface evidence. Some of this behavior is already rejected by the validator, but the tests need to spell out the exact benchmark-shaped payloads so future prompt or ingest changes cannot quietly re-allow them.

The second milestone fixes shard construction. Replace the current behavior where `target_prompt_count` disables the hard char cap and gap cap inside `cookimport/llm/codex_farm_knowledge_jobs.py::_bundle_prepared_chunks(...)`. The prompt target should remain a soft target for how many bundles to aim for, but it must never force oversize bundles. The existing test `test_build_knowledge_jobs_target_prompt_count_overrides_char_cap` in `tests/llm/test_knowledge_job_bundles.py` should go red first and then be replaced with a new expectation that hard caps win. Add at least one more bundling regression that approximates the observed benchmark problem: many long knowledge chunks with `target_prompt_count=5` should split into more than five bundles if that is what the hard cap requires.

The third milestone hardens the prompt contract. Update `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md` and, if needed, the prompt-builder-facing tests so the prompt says in plain language that when input `c` is non-empty the output `r` must contain exactly one row per input chunk, in input order, using the same `cid` values, with no synthetic ids and no empty `r`. The prompt should explicitly say that if the model is uncertain it must still emit conservative `other` decisions and empty snippets for each chunk rather than emitting a bailout object or an empty result set. This must be guarded by focused tests that assert the prompt text contains those prohibitions and obligations.

The fourth milestone verifies the run-level behavior. Update `tests/llm/test_codex_farm_knowledge_orchestrator.py` so there is an integration-style regression that simulates one of the benchmark failure shapes through the direct-exec seam and proves the orchestrator reports the stage as `completed_with_failures` only for genuinely invalid outputs, while correctly counting reviewed versus validated shards. Add a companion green-path test that uses the new smaller-bundle policy to show the same synthetic large-chunk workload produces more, smaller knowledge shards instead of the old unstable giant bundles.

The implementation order matters. First add the red tests. Then change bundling policy. Then tighten the prompt wording. Then adjust any orchestrator reporting or test fixtures needed to reflect the new behavior. Avoid mixing speculative refactors into the same change.

## Milestones

### Milestone 1: Capture the benchmark failure shapes in red tests

At the end of this milestone the repository will have failing tests that describe the real broken behaviors we saw, instead of only broad summaries in benchmark artifacts. A novice should be able to run the targeted tests and see them fail before the production fixes land.

The work lives in `tests/llm/test_knowledge_output_ingest.py`, `tests/llm/test_knowledge_phase_workers.py`, and possibly `tests/llm/test_knowledge_prompt_builder.py` if a prompt-text assertion is the clearest way to lock in a contract statement. Use benchmark-shaped payloads, not abstract dummy names, whenever that makes the failure mode clearer.

Acceptance for this milestone is a red test run with clearly scoped failures, not broad suite breakage.

### Milestone 2: Make shard-size hard caps stay hard

At the end of this milestone a large knowledge run that asks for a small number of prompts should still split into more shards if the chunk or character limits require it. A novice should be able to run the bundle tests and see that the oversized-shard expectation has been replaced by a safe bounded-shard expectation.

The implementation lives in `cookimport/llm/codex_farm_knowledge_jobs.py`. Keep the default limits explicit and keep the logic simple enough that the next contributor can reason about the cap interaction without rereading benchmark artifacts.

Acceptance for this milestone is a green bundling test that proves hard caps override prompt-target pressure.

### Milestone 3: Force conservative per-chunk outputs instead of fallback collapse

At the end of this milestone the prompt should no longer leave room for synthetic fallback rows or empty result arrays when the model receives owned chunks. A novice should be able to open the prompt file and immediately see the explicit "one result per input chunk" rule and the "never invent synthetic ids" rule.

The implementation lives in `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md` and any prompt-builder test that inventories the active wording. Keep the wording strict and short; the main goal is to remove ambiguity, not to add more prose.

Acceptance for this milestone is a green prompt-contract test plus the earlier validator tests remaining green.

### Milestone 4: Reconcile run-level reporting with the safer shard policy

At the end of this milestone the orchestrator tests should prove that the knowledge stage still reports partial invalid-shard runs honestly, while the new smaller bundle policy yields safer shard construction in the first place. A novice should be able to run the knowledge orchestrator tests and observe both the failure-reporting path and the bounded-bundle path.

The implementation lives in `tests/llm/test_codex_farm_knowledge_orchestrator.py` and only minimal production reporting code if the tests expose a genuine mismatch.

Acceptance for this milestone is a green focused knowledge suite.

## Idempotence and Recovery

All of this work is safe to repeat because it is test-first and local. The main production edits are in prompt text and knowledge-stage Python modules; rerunning the test commands is the intended retry mechanism. If a bundling change accidentally explodes shard count too aggressively, restore the previous test state, reduce the cap change to the minimum needed for the red case, and rerun the same focused suite. Do not recover by weakening validation or by reintroducing synthetic fallback chunk ids; those are exactly the behaviors this plan exists to prevent.
