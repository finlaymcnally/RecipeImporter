# Rename Recipe Prompt Artifact Keys to Rows

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

The benchmark cutdown/reporting layer is now row-native, but the recipe prompt-artifact surface still expects old selection/context keys such as `blocks_candidate`, `blocks_before`, `blocks_after`, `blocks`, and `excluded_block_ids`. Those keys describe prompt-row selection context, not true source-block provenance. This plan migrates the recipe prompt-artifact consumers and benchmark fixtures to row-native key names while explicitly leaving the knowledge transport block context alone.

## Decision Log

- Decision: rename only the recipe prompt-artifact keys in this pass and leave the knowledge transport block context untouched.
  Rationale: knowledge packets still intentionally move block-context payloads, while the recipe prompt-artifact keys here are row-semantic selection/context inputs. Folding both into one rename would risk deleting real provenance/transport meaning.
  Date/Author: 2026-04-16_15.15.00 / Codex

- Decision: keep this migration destructive on the touched recipe prompt-artifact consumers and fixtures.
  Rationale: the user explicitly wants old block-era code removed, not preserved behind compatibility shims. Once the touched consumers and fixtures move to row-native keys, the old key names should disappear from that local recipe prompt-artifact surface.

## Outcomes & Retrospective

This slice is now implemented in code/docs/tests, validated, and closed in Beads. The benchmark recipe prompt-artifact surface no longer expects `blocks_candidate`, `blocks_before`, `blocks_after`, `blocks`, or `excluded_block_ids`; it now uses `rows_candidate`, `rows_before`, `rows_after`, `rows`, and `excluded_row_ids`.

The important outcome here is the sharper boundary between recipe prompt artifacts and knowledge transport. The recipe prompt surface is now row-native, while the knowledge packet context remains block-native because it still represents real block-context transport. Treating those as one contract would have been the wrong cleanup.
