# Add benchmark timing telemetry and runtime-analyzer foundations

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

`docs/PLANS.md` is checked into the repo and this document must be maintained in accordance with it.

## Purpose / Big Picture

Today, benchmark outputs answer quality questions but not performance questions. After this change, benchmark runs will publish reliable timing telemetry at the config, source, and whole-run levels so we can see where time is actually spent and what should be optimized or parallelized first.

This plan also prepares the data surface needed for a future coarse runtime analyzer command (manifest/report-based), but does not add that command yet. The immediate user-visible win is that benchmark artifacts and history rows will include timing data that can be inspected directly and consumed by existing dashboard/reporting flows.

## Surprises & Discoveries

- Observation: benchmark history rows already have stage timing columns (`total_seconds`, `parsing_seconds`, `writing_seconds`, `ocr_seconds`), but benchmark append currently leaves them empty by design.
  Evidence: `cookimport/analytics/perf_report.py` `append_benchmark_csv(...)` initializes benchmark rows from `_CSV_FIELDS` and does not set stage timing fields.

- Observation: benchmark prediction generation writes processed cookbook report JSON, but the report path in benchmark flows does not currently include `report.timing`.
  Evidence: `cookimport/labelstudio/ingest.py` `_write_processed_outputs(...)` writes report metadata/output stats without timing instrumentation.

- Observation: all-method benchmarking is now queue-based and split-slot gated, so timing telemetry must work under concurrent config execution.
  Evidence: `cookimport/cli.py` `_run_all_method_benchmark(...)` uses bounded inflight execution and split-phase slot limits.

## Decision Log

- Decision: implement timing as an additive contract in existing artifacts (`manifest.json`, `run_manifest.json`, `eval_report.json`, all-method reports, CSV), not as a new standalone telemetry store.
  Rationale: existing tooling (dashboard, history scans, manual artifact inspection) already consumes these artifacts; additive fields keep compatibility and reduce operational complexity.
  Date/Author: 2026-02-24 / Codex

- Decision: keep ranking and scoring semantics untouched; timing telemetry must be observational only.
  Rationale: optimization telemetry should not change benchmark quality outcomes or tie-break logic.
  Date/Author: 2026-02-24 / Codex

- Decision: do not ship a new analyzer CLI command in this plan; only ship the stable data contract and helper entrypoints needed for that future command.
  Rationale: user requested telemetry first and deferred command work; this sequencing keeps scope tight and lowers regression risk.
  Date/Author: 2026-02-24 / Codex

## Context and Orientation

Benchmark runtime currently has two major layers:

1) Prediction generation (extract/convert/write tasks and optional processed cookbook outputs) in `cookimport/labelstudio/ingest.py` via `generate_pred_run_artifacts(...)`.

2) Evaluation/report orchestration in `cookimport/cli.py` via `labelstudio_benchmark(...)` and all-method wrappers (`_run_all_method_config_once`, `_run_all_method_benchmark`, `_run_all_method_benchmark_multi_source`).

Timing in regular stage/import flows is modeled by `cookimport/core/timing.py` (`TimingStats`, `measure(...)`) and serialized to report JSON through `ConversionReport.timing`.

History persistence is centralized in `cookimport/analytics/perf_report.py`:

- `append_history_csv(...)` for stage/import rows,
- `append_benchmark_csv(...)` for benchmark rows.

Dashboard ingestion/rendering lives in:

- `cookimport/analytics/dashboard_collect.py`
- `cookimport/analytics/dashboard_schema.py`
- `cookimport/analytics/dashboard_render.py`

The existing all-method reports currently track quality metrics and counts but not config-level timing rollups.

## Plan of Work

Milestone 1 defines a stable benchmark timing contract with clear field names and fallback rules. Milestone 2 instruments prediction generation timing. Milestone 3 instruments eval/orchestration timing and CSV writes. Milestone 4 adds all-method rollups and report rendering for slow-stage visibility. Milestone 5 adds internal helper interfaces that enable a future coarse-runtime analyzer command without exposing a new CLI yet. Milestone 6 finalizes tests and docs.

### Milestone 1: Define telemetry contract and field names

Introduce one canonical benchmark timing payload shape, reused in all relevant artifacts.

Add contract docs and code comments describing:

- `total_seconds`: end-to-end benchmark config runtime,
- `prediction_seconds`: prediction artifact generation runtime,
- `evaluation_seconds`: loading/evaluating predicted vs gold spans,
- `artifact_write_seconds`: eval artifact write time (`eval_report`, markdown, misses/FPs),
- `history_append_seconds`: benchmark CSV append duration,
- `checkpoints`: optional phase-specific breakdown (`split_wait_seconds`, `split_convert_seconds`, `processed_output_write_seconds`, and similar).

Contract rule: missing fine-grained phases are allowed, but `total_seconds` and `prediction_seconds` should always be populated when benchmark run succeeds.

Acceptance for Milestone 1:

- timing keys are explicitly documented in code/docs and used consistently across producers.

### Milestone 2: Instrument prediction-generation timing in ingest path

In `cookimport/labelstudio/ingest.py`:

- add timing capture around `generate_pred_run_artifacts(...)` major phases (conversion, split wait/convert if applicable, processed-output write, task-build, artifact write),
- persist timing in prediction `manifest.json` and prediction `run_manifest.json` artifacts payload,
- include timing in returned metadata dict from `generate_pred_run_artifacts(...)` so callers can propagate timing without re-parsing files.

Where split-slot waits occur, capture wait duration separately to support parallelization tuning.

Acceptance for Milestone 2:

- benchmark prediction artifacts contain non-empty timing with at least total + prediction phase runtime for successful runs.

### Milestone 3: Instrument benchmark eval/orchestration and CSV timing fields

In `cookimport/cli.py` `labelstudio_benchmark(...)`:

- measure full config runtime with monotonic clocks,
- measure eval subphases (prediction load, gold load, evaluate, report write, CSV append),
- attach timing object to `eval_report.json`.

In `cookimport/analytics/perf_report.py`:

- extend `append_benchmark_csv(...)` to accept timing inputs,
- populate existing stage timing columns for benchmark rows when available (`total_seconds`, `parsing_seconds`, `writing_seconds`, `ocr_seconds`),
- add benchmark-specific timing columns for eval/orchestration phases (for example `benchmark_prediction_seconds`, `benchmark_evaluation_seconds`, `benchmark_total_seconds`),
- keep schema migration behavior backward compatible (`_ensure_csv_schema` still expands older CSVs safely).

Timing-source precedence for benchmark CSV:

1. explicit timing passed by caller,
2. parsed `timing` from `processed_report_path` (if present),
3. blank values.

Acceptance for Milestone 3:

- new benchmark rows include usable runtime values,
- existing dashboard CSV parsing still succeeds on both old and new rows.

### Milestone 4: Add all-method timing row fields and rollups

In `cookimport/cli.py`:

- capture per-config wall runtime in `_run_all_method_config_once(...)`,
- include per-config timing in `variants` rows in `all_method_benchmark_report.json`,
- compute per-source timing summary in `_run_all_method_benchmark(...)` (totals, average/median, slowest config),
- compute whole-run multi-source timing summary in `_run_all_method_benchmark_multi_source(...)` (sum of source totals, average per source/config, slowest source/config),
- render these metrics in markdown reports so slow stages/configs are visible without JSON inspection.

Do not alter scoring sort order; timing is supplemental metadata.

Acceptance for Milestone 4:

- all-method JSON/MD reports include timing summaries and slowest-config indicators.

### Milestone 5: Prepare future coarse analyzer (no CLI command yet)

Add internal helper(s), likely under `cookimport/analytics` (for example `benchmark_timing.py`), that:

- read benchmark run roots (`data/golden/benchmark-vs-golden/<ts>/all-method-benchmark`),
- normalize timing from manifests/reports into one in-memory structure,
- expose aggregated stats needed by a future analyzer command.

No new `typer` command in this milestone. This is interface preparation only.

Acceptance for Milestone 5:

- helper API can produce coarse per-config/per-source runtime summaries from existing artifacts in one call.

## Validation and Acceptance

This plan is complete when all conditions hold:

1. A successful `labelstudio-benchmark --no-upload` run writes timing fields to `eval_report.json` and benchmark CSV.
2. A successful all-method run writes timing fields per config and source-level/whole-run rollups in report JSON/MD.
3. Benchmark CSV remains valid under concurrent writes and schema migration from old files.
4. Dashboard collector continues to load mixed historical and new benchmark rows without errors.
5. Existing benchmark ranking order stays quality-metric driven (timing metadata does not reorder winners).
6. No codex-farm parsing behavior changes are introduced; recipe pipeline remains policy-locked off.

## Idempotence and Recovery

All benchmark outputs are timestamp-rooted, so reruns are additive and safe.

If implementation fails mid-way:

- revert only touched files or fix forward; do not delete existing benchmark roots,
- re-run benchmarks into new timestamped directories,
- verify old and new CSV rows coexist (schema migration should normalize on append).

If timing fields are partially missing:

- benchmark should still produce valid reports with null/blank timing fields,
- fallback logic should avoid hard-failing report generation.
