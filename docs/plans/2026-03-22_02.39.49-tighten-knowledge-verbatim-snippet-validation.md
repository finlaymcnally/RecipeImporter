# Tighten Knowledge Verbatim Snippet Validation

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The current knowledge validator already rejects the most obvious bad snippet shape: a snippet that more or less copies the entire owned chunk text back into the answer. That helped, but the March 22 benchmark proved that a lower-quality variant still slips through. A worker can copy one long source block verbatim into `snippets[].body`, point `evidence` at that same block, and still pass when the chunk contains multiple blocks. The result is “validated” knowledge that is really just a transcript of the source material.

After this change, the validator should reject large verbatim block copies even when they occur inside multi-block chunks. The observable result is simple. A focused ingest test that currently passes with a multi-block verbatim copy must fail after the change, while a short heading-sized quote such as `Water and Heat` should still remain acceptable.

## Surprises & Discoveries

- Observation: the current validator only triggers `semantic_snippet_echoes_full_chunk` when the snippet evidence spans the whole owned chunk surface.
  Evidence: in `cookimport/llm/codex_farm_knowledge_ingest.py`, the current echo check requires `snippet_block_indices == expected_block_indices` before it rejects a near-full-text copy.

- Observation: the productive March 22 worker copied source text almost verbatim even when its packets validated.
  Evidence: a local scan of `data/output/2026-03-22_01.34.12/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-22_01.34.51/raw/llm/saltfatacidheatcutdown/knowledge/workers/worker-004/out/*.json` found `179` snippets total and `179` cases where `snippet.body == snippet.evidence[0].quote`.

- Observation: long verbatim block copies currently pass when they are nested inside a multi-block chunk.
  Evidence: `data/output/2026-03-22_01.34.12/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-22_01.34.51/raw/llm/saltfatacidheatcutdown/knowledge/workers/worker-004/out/saltfatacidheatcutdown.ks0003.nr.task-001.json` contains several long snippets whose `body` exactly equals a single evidence quote, yet the parent task was accepted.

- Observation: the same generator does get caught on single-block chunks.
  Evidence: `data/output/2026-03-22_01.34.12/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-22_01.34.51/raw/llm/saltfatacidheatcutdown/knowledge/workers/worker-004/out/saltfatacidheatcutdown.ks0003.nr.task-004.json` was rejected with `semantic_snippet_echoes_full_chunk` because the chunk had one block and the copied snippet therefore covered the entire owned surface.

- Observation: the existing ingest tests cover only single-block full and near-full echoes.
  Evidence: `tests/llm/test_knowledge_output_ingest.py` already has `test_validate_knowledge_shard_output_rejects_full_chunk_echo_snippet` and `test_validate_knowledge_shard_output_rejects_near_full_chunk_echo_snippet`, but no multi-block case where a snippet exactly echoes one cited source block inside a larger chunk.

- Observation: the current local branch had already introduced a separate `semantic_snippet_copies_evidence_quote` error code, but that shape did not populate the existing `echoed_full_chunk_ids` metadata used elsewhere in the stage.
  Evidence: a baseline local `.venv` diagnostic against a two-block payload returned `errors = ("semantic_snippet_copies_evidence_quote",)` with `echoed_full_chunk_ids = None` and `verbatim_block_copy_chunk_ids = ["book.c1000.nr"]`.

## Outcomes & Retrospective

This plan is now implemented. The validator still rejects classic whole-chunk echoes, but it now also rejects long verbatim copies of a snippet's cited evidence surface and records whether a chunk failed because of a full-chunk surface echo, a per-snippet evidence-surface echo, an aggregate copied-surface reconstruction, or some combination of those reasons. The public rejection code stays `semantic_snippet_echoes_full_chunk`, so downstream reporting still sees the established category while the metadata now explains the narrower cause.

The main risk was overcorrection, and the implementation stayed conservative. The new evidence-surface detector still requires a long body and a long evidence surface, while the aggregate guard only triggers after multiple copied snippets cover most of a chunk's authoritative source blocks. The added `Water and Heat` regression proves the validator still allows short exact grounding when it behaves like a heading rather than a transcript.

## Context and Orientation

The knowledge stage accepts a “semantic packet result” from each worker. Each packet contains `chunk_results`, and each chunk result contains `snippets`. A snippet is supposed to be a small grounded takeaway backed by one or more evidence quotes. In this repository, the validator for that behavior lives in `cookimport/llm/codex_farm_knowledge_ingest.py` inside `validate_knowledge_shard_output(...)`.

Today the validator already computes three things that matter here:

1. the authoritative source text for each owned chunk via `_chunk_source_text_by_id(...)`
2. the owned block indices for each chunk via `_chunk_block_indices_by_id(...)`
3. a normalized text comparison using `_normalize_semantic_comparison_text(...)`

The current echo detector uses those pieces to reject a snippet only when all of the following are true:

- the chunk source text is long enough
- the snippet body is long enough
- the snippet evidence points at the entire owned chunk surface
- the snippet body is nearly the same as the full chunk text

That misses the March 22 failure shape. In that run, `worker-004` wrote snippets where each snippet body exactly equaled one cited evidence quote. On multi-block chunks, those snippets were accepted because each snippet cited only one block, not the full chunk. In plain language: the validator is currently asking the wrong comparison question for transcript-like block copies.

The key files for this work are:

- `cookimport/llm/codex_farm_knowledge_ingest.py`: the validator logic to change.
- `tests/llm/test_knowledge_output_ingest.py`: the focused regressions that must prove the new behavior.
- `docs/10-llm/nonrecipe_knowledge_review.md`: the short operator-facing note, only if the implementation changes how the validator is described.

This plan deliberately does not change:

- `cookimport/llm/codex_farm_knowledge_orchestrator.py`
- worker prompts
- watchdog policy

Those are separate work items.

## Plan of Work

First, change the snippet-echo detector in `cookimport/llm/codex_farm_knowledge_ingest.py`. Keep the existing full-chunk comparison, but add a second layer that builds the snippet’s cited evidence surface from `snippet.evidence[*].quote` in evidence order. Normalize that evidence surface with the same helper already used for chunk text. If a long snippet body is materially the same as its cited evidence surface, count it as an echo even when the evidence covers only part of the chunk.

Then add an aggregate guard per chunk. The March 22 worker did not produce one giant chunk-level snippet; it produced many block-level transcript snippets. If a chunk contains multiple long verbatim snippet copies and those copies collectively reproduce most of the source text, reject that chunk with the same `semantic_snippet_echoes_full_chunk` error code and record metadata explaining whether the failure came from per-snippet evidence-surface echoes, aggregate copied-surface coverage, or both.

Keep the threshold logic conservative. The implementation should not reject tiny exact-quote snippets such as headings or short claims. Reuse the current “long enough to be suspicious” concept rather than inventing a brand-new notion of quality. The intent is to catch transcript-like copies, not to ban verbatim grounding entirely.

After the validator logic is in place, add focused ingest tests. One test should recreate the March 22 multi-block slip-through case: a chunk with several blocks where one long snippet body exactly equals one evidence quote. Another test should model the aggregate copy pattern: several snippet bodies each equal their evidence quotes so that, together, they reconstruct most of the chunk. A final test should prove that a short exact heading-sized snippet such as `Water and Heat` still validates.

If the implementation changes the operator-facing description of what gets rejected, add one short sentence to `docs/10-llm/nonrecipe_knowledge_review.md`. Keep it brief.

## Validation and Acceptance

This change is accepted when all of the following are true:

1. A focused ingest test proves that a long snippet body that equals its cited evidence quote is rejected even when the chunk has multiple blocks.
2. A focused ingest test proves that several long verbatim snippet copies inside one chunk are rejected when they collectively reconstruct most of the chunk surface.
3. The existing single-block full and near-full echo regressions still fail with `semantic_snippet_echoes_full_chunk`.
4. A focused ingest test proves that a short exact grounded snippet, such as a heading-sized quote, still validates.
5. `pytest -q tests/llm/test_knowledge_output_ingest.py -k "snippet or semantic_snippet"` passes in the local `.venv`.

## Idempotence and Recovery

These changes are safe to iterate on because they are confined to one validator module and one focused test file. Re-running the narrow ingest tests is safe and should give the same result every time.

The main recovery rule is to avoid shipping an overbroad detector. If the new logic starts rejecting short exact snippets or breaks existing single-block expectations, stop and tighten the minimum-length or aggregate-coverage thresholds before changing any broader code. Keep the validator change and the threshold-adjustment tests in the same patch so the intended boundary stays explicit.
