# Unify CodexFarm progress and reporting across recipe, line-role, and knowledge

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This file is the canonical implementation plan for the March 21 progress-reporting work. Earlier overlapping draft plans were deleted during cleanup after this merged plan was created, so this document must now stand on its own as the only active plan for this scope.

## Purpose / Big Picture

After this change, an operator watching `C3imp` or `cookimport labelstudio-benchmark` should be able to answer the same questions for all three active LLM stages without opening raw runtime folders:

- What is the real work unit for this stage right now?
- Are the long-lived worker sessions still running, or is only repo-owned follow-up/finalization left?
- How much work is complete, how much is left, and what kind of recovery work is happening?
- If the run ended or was interrupted, where is the one summary file that explains what happened?

The knowledge stage already has the strongest answers to those questions. It reports packet-scale progress, surfaces follow-up activity in the live UI, writes `stage_status.json`, and produces `knowledge_stage_summary.json` for post-run review. Recipe has now improved materially and reports task-file progress plus richer shard-finalization detail, but it still expresses that truth through overloaded worker labels and `detail_lines` rather than through a shared typed contract. Line-role is still substantially behind because its live progress still reports top-level shard completion rather than task-packet truth.

The visible result of this work should be that recipe, line-role, and knowledge all tell the same kind of story in the benchmark UI, in `processing_timeseries_prediction.jsonl`, and in stage-local summary artifacts.

## Context and Orientation

The live benchmark/import status UI is rendered from `cookimport/cli.py`. It does not compute stage truth directly. Instead, it consumes structured progress payloads serialized by `cookimport/core/progress_messages.py::format_stage_progress(...)` and parsed back by `parse_stage_progress(...)`. Those payloads currently support:

- `stage_label`
- `task_current`
- `task_total`
- `running_workers`
- `worker_total`
- `active_tasks`
- `detail_lines`

That shape was good enough when a stage had one obvious work unit and minimal follow-up, but it is now too thin for the current runtime. The recent recipe improvements prove that a stage can pack more truth into the existing shape, but they also make the limit more obvious because worker truth and follow-up truth are still intertwined.

The three active LLM stages differ in their current maturity:

- Recipe lives in `cookimport/llm/codex_farm_orchestrator.py`. It now reports task-file progress truth from `workers/*/out/*.json` and emits richer shard-finalization detail lines such as completed shards, queued recipe tasks, workers still finalizing shards, and live repair counts. Its remaining gap is that repo-owned finalization still appears through worker-style labels such as `finalize` rather than through a typed worker-versus-follow-up model.
- Knowledge lives in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. It already has a dedicated progress state, packet-scale counters, follow-up labels, `task_status.jsonl`, `stage_status.json`, and `knowledge_stage_summary.json`.
- Line-role lives in `cookimport/parsing/canonical_line_roles.py`. It now runs through long-lived workspace workers with task packets, but its progress callback still reports shard completion and does not carry useful worker labels or typed follow-up state.

For post-run observability, the strongest current seam is `cookimport/runs/stage_observability.py`. That module already knows how to write and index `knowledge_stage_summary.json`. It is the natural place to add compact summary builders for recipe and line-role as well.

Definitions used in this plan:

- A “work unit” is the thing the visible `task X/Y` counter should mean. For knowledge and line-role that should be task packets. For recipe that should be recipe-local task outputs.
- A “worker session” is one long-lived workspace worker with its own `workers/<worker_id>/live_status.json`.
- A “follow-up” is repo-owned retry, repair, validation, proposal assembly, or other post-worker work that can continue after the main session is done.
- “Finalization” means the deterministic repo-owned work after the main worker session exits cleanly: validation, repairs, proposal assembly, promotion, and summary writing.
- A “stage-local summary artifact” is one compact machine-readable file under the stage root that tells an operator what happened without reading all worker-local artifacts.

## Plan of Work

The first milestone is already complete in one narrow but important slice: the shared CLI progress sampler in `cookimport/cli.py` now resets structured ETA history on `stage_label` changes and uses parallel worker-wave bootstrap ETA for structured progress. Keep that behavior stable while doing the remaining transport work.

The second milestone is to extend the shared progress transport in `cookimport/core/progress_messages.py`. Keep the current fields, because old payload readers and tests still depend on them, but add typed optional fields that let every stage report the same concepts without encoding them only in free-form `detail_lines`. The contract should be simple and stage-agnostic. A concrete target shape is:

    worker_running: currently live long-lived worker sessions
    worker_completed: worker sessions in a clean terminal state
    worker_failed: worker sessions in a failed, killed, or interrupted terminal state
    followup_running: repo-owned follow-up actions currently active
    followup_completed: completed follow-up actions when a meaningful count exists
    followup_total: planned/observed follow-up total when known
    followup_label: short stage-specific label such as "repairs", "proposal assembly", or "repair/finalization"
    artifact_counts: small dict for stage-local counts such as outputs, proposals, packets, repairs
    last_activity_at: ISO timestamp or short string for recent activity

`detail_lines` should remain available for genuinely human-facing extra context, but the renderer and timeseries should no longer have to reverse-engineer core state from those lines.

The third milestone is recipe. Keep the existing task-output progress model, because that part is already correct enough and has now landed in code, but finish splitting worker-session truth from repo-owned finalization truth. `cookimport/llm/codex_farm_orchestrator.py` already has enough in-memory and on-disk signals to expose:

- total task outputs planned versus written;
- worker sessions currently running versus completed;
- shard proposals assembled versus total shards;
- repair attempts active or completed;
- whether the remaining work is still worker execution or only repo-side finalization.

Once all worker sessions are terminal, recipe worker rows must visibly switch to completed/idle states while the follow-up summary explains what remains. The current interim labels such as `<shard_id> finalize` or `<shard_id> repair X/Y` should be replaced by explicit worker-session rows plus explicit repo follow-up summary lines.

The fourth milestone is line-role. This is now the largest runtime reporting correction. `cookimport/parsing/canonical_line_roles.py` should grow a small stage-local progress state analogous to knowledge’s `_KnowledgePhaseProgressState`. The live counter must report task packets rather than parent shards. Worker rows should show worker-owned shard context plus current packet progress while the main session is active, and they should surface watchdog-retry/repair follow-up labels when a worker is in structured follow-up work. The stage should also expose counts for proposals and repairs once the main worker pass is done. The key rule is that line-role must stop reporting shard truth in a task-packet runtime.

The fifth milestone is knowledge alignment. Knowledge already has the right conceptual model, so the work here is mainly to emit the new shared typed fields while keeping its packet ledger, `stage_status.json`, and `knowledge_stage_summary.json` intact. The operator-facing vocabulary should match recipe and line-role: work-unit progress, worker-session progress, follow-up progress, and stage-local summary path. The existing `detail_lines` can remain, but they should become supporting context rather than the only carrier of follow-up truth.

The sixth milestone is stage-local summaries. Use `cookimport/runs/stage_observability.py` to add compact summary builders for recipe and line-role that mirror the role, not the exact schema, of `knowledge_stage_summary.json`. Suggested outputs:

- `recipe_stage_summary.json` under `recipe_phase_runtime/`
- `line_role_stage_summary.json` under `line-role-pipeline/runtime/line_role/`

These summaries should answer, at minimum:

- planned work units versus completed work units;
- worker outcome counts by terminal state;
- follow-up counts and proposal counts;
- whether the stage completed, partially completed, failed, or was interrupted;
- where the most important raw artifacts live.

`stage_observability.json` should index these summary files the same way it already indexes the knowledge summary.

The seventh milestone is CLI rendering and persisted progress snapshots. `cookimport/cli.py` should keep backward compatibility for old payloads, but when the richer typed fields are present it should render:

- the stage name;
- progress in the stage’s real work unit;
- a short worker-session summary such as `workers 2 running, 3 completed`;
- explicit worker rows when those rows represent real worker sessions;
- a separate follow-up/finalization summary such as `repo follow-up: proposals 3/5 | repairs 22`;
- any genuinely helpful extra detail lines.

`processing_timeseries_prediction.jsonl` should preserve those richer structured fields so saved runs tell the same story as the live UI.

The eighth milestone is docs and tests. Update the short current-state docs so they describe all three stages consistently. Knowledge should stay the detailed reference, but the docs must stop implying that only knowledge has meaningful progress/reporting semantics. The tests should lock the contract across the transport, the renderers, and the stage-local summary builders.

## Validation and Acceptance

This work is complete only if a human can answer the same questions from the live UI and the saved artifacts for recipe, line-role, and knowledge.

Minimum acceptance:

- Shared transport: `format_stage_progress(...)` and `parse_stage_progress(...)` preserve the new typed fields without breaking older payloads.
- Recipe live UI: once all worker sessions are terminal, the UI shows workers as completed/idle and reports remaining repairs/proposals/finalization separately. The current interim state of richer `detail_lines` and `finalize` labels is an improvement, but it is not sufficient for final acceptance.
- Line-role live UI: the top-level counter uses task-packet truth rather than shard truth, worker rows are descriptive, and follow-up work is surfaced separately instead of disappearing into a shard-only counter.
- Knowledge live UI: packet truth and follow-up truth still render correctly, now through the shared typed fields as well as any remaining detail lines.
- Persisted snapshots: `processing_timeseries_prediction.jsonl` records the richer fields so postmortem analysis can reconstruct the same story.
- Stage-local summaries: recipe, line-role, and knowledge each have one obvious stage-local summary file indexed from `stage_observability.json`.

A representative successful live status should read in spirit like:

    non-recipe knowledge review
    progress: task 233/485 (48%)
    workers 2 running, 8 completed
    repo follow-up: retries 1 | repairs 2

    recipe pipeline
    progress: task 33/33 (100%)
    workers 0 running, 5 completed
    repo follow-up: proposals 3/5 | repairs 22

    canonical line-role pipeline
    progress: task 144/220 (65%)
    workers 4 running, 0 completed
    repo follow-up: none

The exact wording can differ, but the operator must be able to distinguish worker execution from repo-owned finalization in all three stages.

## Idempotence and Recovery

This work should be additive and backward compatible. Old progress payloads must remain parseable. If one stage cannot know an exact follow-up denominator cheaply, emit the count that is known and leave the total absent rather than inventing a misleading number.

If recipe or line-role summary generation proves too expensive to run continuously, keep summary writing at stage-finalization time and let the live UI use lighter in-memory counts. The key requirement is truthful reporting, not maximal artifact churn.

If the renderer becomes noisy, simplify the visible text while keeping the typed fields. Do not backslide into hiding core state only in free-form strings again.
