# Finish Bucket 1 By Hardcoding Remaining Fake Run Settings

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this work, the remaining Bucket 1 settings from `docs/reports/2026-03-13_22.48.50-run-settings-surface-audit.md` stop behaving like product choices. A normal operator should no longer decide which section detector to use, which fallback instruction segmenter to use, which benchmark matcher to use, or which internal Codex pass ids/debug toggles to thread through a run. Those become fixed product behavior, stored in one contract layer, while older manifests and saved settings files still load without breaking.

The visible proof is straightforward. `cookimport stage --help` and `cookimport labelstudio-benchmark --help` should stop advertising the remaining Bucket 1 knobs. New run manifests and summaries should stop implying that these values were human-selected. Old payloads containing keys such as `section_detector_backend`, `instruction_step_segmentation_policy`, or `codex_farm_pipeline_pass2` should still deserialize successfully, with the values ignored or normalized into the current fixed behavior instead of being treated as active choices.

In this plan, “hardcode as product behavior” means three things at once. First, remove the setting from the everyday operator surface: no normal CLI flag, UI selector, or docs section should present it as a choice. Second, stop treating the setting as a live input to run construction and prediction identity; the current blessed behavior should come from contract constants in code. Third, preserve compatibility for old stored payloads by accepting the key during load and documenting that it no longer changes behavior.

## Context and Orientation

The current run-settings source of truth lives in `cookimport/config/run_settings.py`. That file defines the `RunSettings` Pydantic model, field metadata for UI/help surfaces, helper projections such as `to_run_config_dict()`, and compatibility loading via `RunSettings.from_dict(...)`. It already contains a retired-key registry for `table_extraction`, plus public/internal surface helpers.

The stage and benchmark command boundaries still accept most Bucket 1 knobs directly. `cookimport/config/run_settings_adapters.py` maps `RunSettings` into keyword arguments for `cookimport/cli.py:stage(...)` and `cookimport/cli.py:labelstudio_benchmark(...)`. Both Typer entrypoints still declare direct options for the remaining Bucket 1 settings, normalize them locally, and pass them onward. `cookimport/labelstudio/ingest.py` mirrors many of the same keyword surfaces for benchmark helper flows, so removing a setting only from `cli.py` is not sufficient.

The current top-tier profile contract lives in `cookimport/config/codex_decision.py`. That module already centralizes the “current blessed parser stack” through `_TOP_TIER_PARSER_STACK_PATCH` and related benchmark patches. It is the best existing home for fixed Bucket 1 behavior because it already describes product-policy decisions such as the shared section detector, always-on fallback segmentation, and compact Codex pass ids.

Prediction reuse identity currently lives in `cookimport/config/prediction_identity.py`. That file decides which fields count as material to prediction generation and cache reuse. Any Bucket 1 field that becomes fixed product behavior should stop being projected there, or identity will continue to pretend the retired setting is a meaningful experiment dimension.

Several downstream modules still consume these values as if they were active inputs. `cookimport/plugins/epub.py`, `cookimport/plugins/pdf.py`, and `cookimport/plugins/text.py` read section detection and trace settings from `run_settings`. `cookimport/staging/draft_v1.py` and `cookimport/staging/writer.py` read instruction-step fallback settings and the Priority 6 metadata debug flag from the run-config payload. `cookimport/llm` and benchmark helper tests still assert specific Codex pass ids flowing from settings into manifests and retries. The migration must either preserve those consumers through a fixed-behavior projection or move them to constants, but it must stop implying that a human selected them.

Bucket 1 from the audit consists of these keys:

## Plan of Work

### Milestone 1: Create One Fixed-Behavior Contract For Remaining Bucket 1 Settings

At the end of this milestone, there is one authoritative place that answers “what is the current product behavior for the former Bucket 1 knobs?” without relying on a caller to supply those values. Add a named fixed-behavior projection to `cookimport/config/codex_decision.py` or a nearby config helper module already owned by that package. The projection should return plain normalized values for the remaining Bucket 1 behaviors, including:

- shared section detection,
- always-on fallback instruction segmentation with the chosen segmenter,
- the current canonical benchmark matcher,
- disabled multi-recipe tracing,
- disabled Priority 6 metadata debug emission for normal runs,
- the current Codex pass ids and related pass-policy booleans,
- the current benchmark selective-retry behavior and max attempts.

Do not leave these as repeated default strings or booleans in `RunSettings`, Typer options, and helper signatures. The point of this milestone is to make every later edit consume one stable contract. If some behaviors differ between normal stage runs and benchmark evaluation-only flows, encode that distinction explicitly in the contract helper and explain it in code comments.

Acceptance for this milestone is structural and observable. A fresh reader should be able to find one config-layer function or constant block and see the full current Bucket 1 behavior there, instead of discovering it only by reading scattered defaults.

### Milestone 2: Retire Remaining Bucket 1 Keys From Live RunSettings And Input Surfaces

At the end of this milestone, the remaining Bucket 1 settings are no longer real operator inputs. Update `cookimport/config/run_settings.py` so these keys follow one of two patterns:

1. Fully retired legacy key, matching `table_extraction`, when the setting should no longer exist at all.
2. Compatibility-only accepted key, when old payloads may still contain it but new `RunSettings` objects and public/internal projections do not treat it as a live field.

The intended end state for this plan is to retire all remaining Bucket 1 keys from the live `RunSettings` model. That means removing the field definitions, removing them from `build_run_settings(...)` parameters, removing them from `_SUMMARY_ORDER`, and extending the retired-key compatibility loader so old payloads that mention them do not fail. When an old payload provides a retired value that disagrees with the new fixed behavior, ignore it and warn once using the existing retired-key warning machinery.

After `RunSettings` is cleaned up, remove the corresponding parameters from `cookimport/config/run_settings_adapters.py`. Those adapter functions should stop forwarding Bucket 1 values into command entrypoints, because the command entrypoints will now source them from the fixed-behavior contract instead.

Then remove the normal CLI options from `cookimport/cli.py`. This applies to both `stage(...)` and `labelstudio_benchmark(...)`, including:

- `--section-detector-backend`
- `--instruction-step-segmentation-policy`
- `--instruction-step-segmenter`
- `--sequence-matcher`
- the direct Codex pass-id and pass-policy flags

If `multi_recipe_trace` or `p6_emit_metadata_debug` must remain invocable for debugging, do not keep them as ordinary run settings. Move them behind a clearly debug-only mechanism such as a narrowly scoped environment variable or an internal helper invoked only by tests/debug tooling. Document whichever path is chosen directly in this plan as implementation proceeds.

Acceptance is user-visible. `cookimport stage --help` and `cookimport labelstudio-benchmark --help` should no longer list the retired Bucket 1 settings as ordinary options.

### Milestone 3: Rewire Runtime Consumers To Use Fixed Behavior Instead Of Settings

At the end of this milestone, downstream runtime code no longer depends on the retired keys being present in `run_settings` or in the CLI keyword payloads.

Update `cookimport/cli.py` and `cookimport/labelstudio/ingest.py` so they resolve the fixed Bucket 1 behavior once per run and pass that normalized bundle to the lower-level import/eval helpers. The key requirement is that the values still reach the runtime locations that need them, but they come from the fixed-behavior contract rather than from user input.

Then update downstream consumers:

- `cookimport/plugins/epub.py`, `cookimport/plugins/pdf.py`, and `cookimport/plugins/text.py` for section detection and trace behavior.
- `cookimport/staging/draft_v1.py` and `cookimport/staging/writer.py` for instruction-step fallback segmentation and Priority 6 debug emission.
- Any Codex orchestration/helper code that currently reads pass ids or selective-retry settings from live run settings.

Prefer passing a small normalized runtime bundle rather than sprinkling `getattr(..., "section_detector_backend", "legacy")` style fallbacks across call sites. The system should still be able to write manifests that explain the effective behavior, but those manifests should say these behaviors are fixed product defaults or internal contract values, not pretend they were operator choices.

Acceptance is behavioral. Stage runs and benchmark prediction generation should still produce the same recipe outputs as before for the current blessed behavior, but they should do so even though the retired settings are no longer accepted from the CLI or run-settings payload.

### Milestone 4: Remove Retired Bucket 1 Keys From Identity, Summaries, And Docs

At the end of this milestone, human-facing surfaces stop communicating the retired keys as active dimensions.

Update `cookimport/config/prediction_identity.py` so retired Bucket 1 keys are no longer part of `ALL_METHOD_PREDICTION_IDENTITY_FIELDS`. If a fixed behavior still materially affects cache reuse, represent it by the new fixed-behavior contract version or by the remaining public settings that actually matter, not by dead per-run knobs.

Update manifest/report rendering so new summaries stop listing retired Bucket 1 values as if the operator chose them. The full debug payload, if any, may still mention the effective fixed values, but the main summary should distinguish operator choices from fixed product behavior.

Update documentation at minimum in:

- `docs/02-cli/02-cli_README.md`
- `docs/03-ingestion/03-ingestion_readme.md`
- `docs/05-staging/05-staging_readme.md`
- `docs/07-bench/07-bench_README.md`
- `cookimport/config/CONVENTIONS.md`

Those docs should explain the new state plainly: these behaviors are fixed product defaults now, old saved payloads may still contain the retired keys, and new runs will not present them as choices.

Acceptance is readable output. A novice reading the CLI help or the docs should no longer conclude that section detection, instruction-step fallback policy, benchmark matcher mode, or Codex pass ids are day-to-day product choices.

### Milestone 5: Compatibility And Test Repair

At the end of this milestone, the cleanup is safe to merge because old artifacts still load and the test suite proves the new shape.

Start with settings-surface tests:

- `tests/llm/test_run_settings.py`
- `tests/cli/test_run_settings_adapters.py`
- `tests/cli/test_cli_llm_flags.py`
- `tests/cli/test_c3imp_interactive_menu.py`
- `tests/staging/test_run_manifest_parity.py`

Then repair downstream tests that currently assert these fields as live kwargs or manifest entries, especially under `tests/labelstudio/` and `tests/llm/`. Many of those tests should be split into two categories:

1. Compatibility tests showing old payloads containing retired keys still load and normalize.
2. Runtime/manifest tests showing new runs use fixed behavior without exposing those keys as choices.

Be especially careful with tests that currently assert custom Codex pass ids or custom benchmark selective-retry attempts. Those expectations should be rewritten to assert the fixed contract value unless the code intentionally preserves a hidden debug seam. If a hidden seam remains, document it explicitly in both code and this plan; do not leave it as an accidental loophole.

Acceptance is that the focused tests below pass and the compatibility scenario can be reproduced by a novice using only this document.

All edits in this plan should be additive or compatibility-preserving until the final field removals land. The recommended order is safe to repeat: first add the fixed-behavior contract, then route consumers to it, then remove live input surfaces, then tighten docs/tests. If a partial edit breaks compatibility loading, restore that behavior before deleting more call-site references.

Do not delete historical manifest fields from old output roots. The migration only changes how new runs are produced and how old payloads are interpreted. Re-running the compatibility checks and focused tests is the safe recovery path after any failed intermediate state.
