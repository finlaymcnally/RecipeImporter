# Rename Benchmark Prompt Count Language to Rows

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

The benchmark cutdown layer still exposes a second pocket of block-era semantic language after the earlier prompt-span rename. Starter-pack selection, recipe triage diagnostics, and call inventory rows still talk about `input_block_count`, `correction_input_block_count`, `build_intermediate_selected_block_count`, and related loss metrics even when those values now represent prompt-row counts rather than source-block provenance. This plan removes that local old language from benchmark diagnostics and reporting while explicitly leaving upstream prompt-wire-format keys for a separate follow-on slice.

## Surprises & Discoveries

- Observation: the next old-name island is split between two layers.
  Evidence: local benchmark diagnostics fields such as `correction_input_block_count`, `build_intermediate_selected_block_count`, and `input_block_count` live only in the cutdown/starter-pack/reporting layer, but prompt payload keys like `blocks_candidate`, `blocks_before`, `blocks_after`, and `excluded_block_ids` also appear in `cookimport/llm/prompt_artifacts_loader.py`, `cookimport/bench/external_ai_cutdown/prompt_log_reconstruction.py`, and knowledge transport code.

- Observation: renaming the local diagnostics layer does not require changing source-block provenance or projection artifacts.
  Evidence: the surviving `block_index`, `source_block_index`, and raw block excerpt fields in projection helpers still refer to source-layout joins rather than semantic row counts.

## Decision Log

- Decision: keep this wave local to benchmark diagnostics and reporting fields, and do not rename upstream prompt-wire-format keys in the same change.
  Rationale: the local diagnostics fields are straightforward semantic debt. The prompt payload keys are part of a broader live transport/artifact contract shared with other LLM helper surfaces and should be handled in a separate tracked slice rather than half-changed here.
  Date/Author: 2026-04-16_15.08.00 / Codex

- Decision: rename count/loss semantics to rows wherever the value is derived from prompt row selection or evidence-row counts.
  Rationale: values like `correction_input_block_count` and `input_block_count` now represent prompt rows. Keeping block names here obscures the actual benchmark semantics and undermines the row-only cleanup.
  Date/Author: 2026-04-16_15.08.00 / Codex

## Outcomes & Retrospective

This slice is now implemented. The benchmark cutdown/starter-pack layer no longer describes prompt-row counts and row-loss diagnostics with block-era names. The active local reporting contract now uses row-native field names like `input_row_count`, `correction_input_row_count`, `build_intermediate_selected_row_count`, and `build_intermediate_clamped_row_loss_count`.

The main lesson from this pass is that there were really two debts intertwined here. The local reporting vocabulary was straightforward to clean up and validate. The upstream prompt payload keys were not. Splitting those into the discovered follow-on bead `recipeimport-ubl2` kept this pass destructive without being sloppy.
