# C4imp Bitter-Recipe Corpus-First Offshoot

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the repo will have a second, much thinner interactive lane for building a reviewed cookbook corpus instead of trying to solve full recipe import automation. A user will be able to launch `C4imp`, land directly in a focused “bitter-recipe” workflow, pick one cookbook from `data/input`, create or upload Label Studio tasks for that book, optionally ask Codex for first-pass labels, export completed labels back into row-gold artifacts, and track which books are still unstarted, in review, exported, or manually marked reviewed. The user-visible proof is simple: `C4imp` should show a corpus-first menu instead of the current large `cookimport` menu, `cookimport bitter-recipe --help` should expose only the new thin workflow, and the resulting artifacts should live under a dedicated `data/bitter-recipe/` root instead of mixing back into the heavy stage and benchmark directories.

This offshoot is intentionally not a refactor of the main product. It is a new, smaller sculpture cut from the existing stone. The point is to preserve the useful machinery that already exists in this repo, especially source-row extraction and Label Studio import/export, while refusing to inherit the full stage, benchmark, analytics, and multi-pipeline complexity that slowed the original path down.

## Surprises & Discoveries

- Observation: `C3imp` is already only a thin wrapper that sets a few environment defaults, optionally accepts a single integer limit argument, and then delegates to the normal Typer app.
  Evidence: `cookimport/c3imp_entrypoint.py` contains no business logic beyond env defaults, `C3IMP_LIMIT`, and `app()`.

- Observation: the real risk is not “can the repo generate Label Studio tasks?” The risk is that the current interactive surface is large and monolithic, so trying to wedge the new idea into `_interactive_mode(...)` would make the thin lane inherit the old product shape.
  Evidence: `cookimport/cli_support/interactive_flow.py` owns one long menu loop that mixes stage import, Label Studio upload/export, benchmark generation, dashboard generation, and settings.

- Observation: the repo already has the core corpus-building primitives. The missing piece is not extraction or export capability; it is a smaller product surface that treats reviewed row-gold as the main output.
  Evidence: `run_labelstudio_import(...)` already generates and uploads freeform row-native tasks, and `run_labelstudio_export(...)` already writes `row_gold_labels.jsonl` and `row_gold_conflicts.jsonl`.

- Observation: the existing Label Studio command surface is too broad to expose directly as the new product.
  Evidence: `cookimport/cli_commands/labelstudio.py` exposes a very large command signature with importer, split, parser, Codex, benchmark, and fallback knobs that are appropriate for the old product but not for the intended corpus-first loop.

- Observation: the importer registry is not magically available in the new offshoot package unless the importer modules are imported for their registration side effects.
  Evidence: the first narrow test run failed with `ImportError`/zero-detection symptoms until `cookimport/bitter_recipe/paths.py` explicitly imported the active importer modules before calling `best_importer_for_path(...)`.

- Observation: Typer help generation still evaluates postponed annotations, so command modules must import every type they reference in function signatures.
  Evidence: the first `cookimport bitter-recipe --help` smoke failed with `NameError: name 'Path' is not defined` until `cookimport/bitter_recipe/app.py` restored the `Path` import used only in annotations.

## Context and Orientation

The current repo has one main Typer CLI rooted at `cookimport/cli.py`. That CLI registers the large current product: interactive import, stage processing, Label Studio upload/export, benchmarking, analytics, and compare/control. `C3imp` is not a second product; it is only a thin wrapper over that main app, defined in `cookimport/c3imp_entrypoint.py`.

The relevant reusable engines already exist.

`cookimport/labelstudio/ingest_flows/upload.py` owns `run_labelstudio_import(...)`. In plain language, this is the function that takes a source book, converts it into the repo’s canonical source model, breaks it into freeform annotation tasks, optionally prelabels those tasks, writes the offline artifacts, and uploads the tasks to Label Studio.

`cookimport/labelstudio/export.py` owns `run_labelstudio_export(...)`. In plain language, this is the function that fetches completed Label Studio annotations and writes the export artifacts, including the row-native files that matter most for the corpus-first workflow: `row_gold_labels.jsonl` and `row_gold_conflicts.jsonl`.

`cookimport/core/source_model.py` and `cookimport/parsing/source_rows.py` are the current owners of stable source-row generation. These are important because the new lane is not trying to produce polished recipe drafts. It is trying to produce reviewable row-native labeling tasks and export reviewable row-native gold.

For this plan, “corpus-first” means the reviewed labels are the product. Source extraction, task generation, optional prelabeling, and export are supporting tools. It does not mean “run stage and benchmarks with fewer prompts.”

For this plan, “ledger” means one small machine-readable file owned by the bitter-recipe package that tracks each source book’s status in the corpus workflow. It is not a benchmark history CSV, not a dashboard cache, and not a replacement for Label Studio manifests. It is a user-facing status index for the remaining cookbook queue.

The new code should live under a dedicated package:

- `cookimport/bitter_recipe/__init__.py`
- `cookimport/bitter_recipe/app.py`
- `cookimport/bitter_recipe/interactive.py`
- `cookimport/bitter_recipe/paths.py`
- `cookimport/bitter_recipe/settings.py`
- `cookimport/bitter_recipe/ledger.py`
- `cookimport/bitter_recipe/adapters.py`
- `cookimport/bitter_recipe/workflows.py`
- `cookimport/bitter_recipe/README.md`

The thin entrypoint wrapper should live at:

- `cookimport/c4imp_entrypoint.py`

The main CLI registration changes should stay small and explicit:

- `pyproject.toml`
- `cookimport/cli.py`

The owning docs to update after implementation are:

- `docs/02-cli/02-cli_README.md`
- `docs/06-label-studio/06-label-studio_README.md`
- `cookimport/README.md`

The new runtime roots should be dedicated to the offshoot. Unless implementation reality forces a small adjustment, use:

- `data/bitter-recipe/sent-to-labelstudio/`
- `data/bitter-recipe/pulled-from-labelstudio/`
- `data/bitter-recipe/ledger/`

Keep source discovery on the existing `data/input/` root so the new lane can immediately work with the same cookbook pool the repo already uses.

## Milestones

### Milestone 1: Create the isolated CLI shell

At the end of this milestone, a user can run `cookimport bitter-recipe --help` and `C4imp --help`, and both surfaces clearly describe the new corpus-first lane instead of the heavy main product. The main `cookimport` CLI keeps its current behavior, but it gains one additional subcommand family named `bitter-recipe`. `C4imp` behaves the way `C3imp` does structurally: it is a short alias with a little environment setup and optional one-argument limit support, but it launches the bitter-recipe mode directly.

This milestone is mostly scaffolding, but it must be observable. The acceptance proof is that the help text and the interactive landing menu show only the bitter-recipe actions, not the current stage, benchmark, and dashboard menu.

### Milestone 2: Build the corpus-first workflow surface

At the end of this milestone, the new lane can do the full thin loop for one book: choose a source from `data/input`, create or upload Label Studio tasks into the bitter-recipe output root, optionally prelabel them, export completed labels back out, and record the status in a ledger entry keyed by the source book. This milestone is where the new package earns its existence. It must not simply proxy the entire current Label Studio CLI; it should present a smaller command set and a smaller interactive menu.

The observable proof is that one book can move from “not started” to “uploaded” to “exported” entirely through the new lane, and the resulting artifacts live in `data/bitter-recipe/...`.

### Milestone 3: Add the guardrails that keep it small

At the end of this milestone, the new lane is documented, tested, and visibly separated from the heavy product. The docs explain that bitter-recipe is a corpus-building lane, not a new stage/benchmark surface. The tests verify the entrypoint, the help surfaces, the ledger behavior, and the import/export adapter behavior. This milestone is not optional: without the docs and tests, the offshoot will drift back toward the old architecture.

The observable proof is that the new tests pass, the docs describe the correct product, and the package README itself warns future agents what not to add.

Second, create the dedicated Typer app. Put the new app in `cookimport/bitter_recipe/app.py`. This file should own a small Typer application whose callback enters bitter-recipe interactive mode when no subcommand is provided. Register this app as `bitter-recipe` inside the main `cookimport/cli.py` by adding one `app.add_typer(...)` call, but keep the new package’s command registration local to itself. Avoid adding more logic to `cookimport/cli_support/interactive_flow.py`; that file belongs to the old product.

Third, create the bitter-recipe ownership files. `paths.py` should define the new default roots and helper functions for resolving per-book artifact directories. `settings.py` should load and save a dedicated settings file, ideally `bitter-recipe.json` at the repo root, with a deliberately tiny schema: input root, bitter-recipe root, Label Studio URL and API key fallback policy, segment defaults, and default prelabel mode. `ledger.py` should own the corpus status file, likely `data/bitter-recipe/ledger/books.json`, with one entry per book keyed by a stable source slug. Each ledger row should track at least the source path, source slug, current status, latest upload run root, latest export run root, Label Studio project metadata when known, and the last time the row was updated.

Fourth, create the explicit adapter layer in `cookimport/bitter_recipe/adapters.py`. This file should wrap the existing engine owners instead of copying their logic. Add one adapter for Label Studio import that calls `run_labelstudio_import(...)` with a deliberately small configuration surface and the bitter-recipe output roots. Add one adapter for Label Studio export that calls `run_labelstudio_export(...)` and normalizes its result into the ledger’s needs. If the bitter-recipe lane needs Label Studio credential resolution or project lookup help from the existing CLI support code, either extract a tiny helper into a narrower owner or reimplement the needed resolution locally; do not import the full `cookimport.cli_support` facade just to get one function.

Fifth, build the actual workflows in `cookimport/bitter_recipe/workflows.py`. Define one function for “prepare and upload one book,” one for “export one reviewed book,” one for “refresh status from manifests and exports,” and one for “mark reviewed.” “Mark reviewed” is intentionally manual: the user said the human review step still matters, so the tool should support a status like `reviewed` or `accepted` even after export already exists. These workflows should be idempotent. Re-running an upload for the same book should update the ledger cleanly rather than duplicating nonsense state. Re-running export should overwrite the current bitter-recipe export folder for that book the same way the current export flow already prefers stable source-aware roots.

Sixth, build the interactive flow in `cookimport/bitter_recipe/interactive.py`. The menu should stay small. The intended initial menu is:

- start or resume a book
- export reviewed labels
- queue status
- mark a book reviewed
- settings
- exit

When the user chooses “start or resume a book,” show the importable files from `data/input`, let the user pick one, ask only the essential options, and then run the adapter-backed upload workflow. The essential options are the project name default, segment sizing defaults, and whether first-pass prelabeling is on. Do not reopen the giant main run-settings planner. Do not surface recipe pipeline, line-role pipeline, benchmark modes, analytics, or all-method knobs here.

The direct commands exist so the workflow is scriptable and testable. The interactive menu is for day-to-day use; the direct commands are for reproducible behavior and focused tests.

Eighth, add the package README and the docs. `cookimport/bitter_recipe/README.md` should be short and blunt: explain that this package owns the thin corpus-first lane, that reviewed row-gold is the product, and that stage, benchmarks, analytics, and “semantic cleverness” do not belong here. Update `cookimport/README.md` so the new entrypoint is discoverable.

Finally, add tests that make the isolation concrete. Create a new `tests/cli/test_c4imp_entrypoint.py` for the wrapper behavior. Add a new `tests/core` or `tests/cli`-adjacent file for the `cookimport bitter-recipe --help` surface. Add `tests/bitter_recipe/` for settings, paths, ledger, and adapter/workflow tests. The adapter tests should monkeypatch the current Label Studio engine functions so the tests remain fast and do not require live Label Studio or live Codex. ## Validation and Acceptance

Acceptance is user-visible and behavioral.

First, `C4imp` must launch the bitter-recipe lane directly. The user should not see the current large `cookimport` main menu with stage, benchmark, dashboard, and compare/control concepts mixed in. `cookimport bitter-recipe --help` must describe a corpus-first tool for preparing, exporting, and tracking labeled books.

Second, the new lane must be path-isolated. Preparing a book through bitter-recipe must write its upload artifacts under `data/bitter-recipe/sent-to-labelstudio/`, and exporting reviewed labels through bitter-recipe must write its row-gold export artifacts under `data/bitter-recipe/pulled-from-labelstudio/`. The heavy stage and benchmark roots must remain untouched by the normal bitter-recipe happy path.

Third, the new lane must prove that it reuses the current machinery instead of reimplementing it poorly. Adapter-level tests should show that the bitter-recipe workflow calls the existing Label Studio import/export engines with the new roots and the reduced option surface. The export flow must still yield `row_gold_labels.jsonl`.

Fourth, the new lane must keep one durable status record per source book. After a prepare run, `status` should show that the book has been uploaded or is in review. After export, it should show that an export exists. After a manual review mark, it should show a reviewed state. That state must come from the bitter-recipe ledger, not from ad hoc directory guessing alone.

Fifth, the narrow tests must pass inside the project-local virtual environment, and the docs must describe the new lane accurately.

## Idempotence and Recovery

The bitter-recipe lane should be safe to rerun. Re-running `prepare` for the same source book should either resume the same logical book entry or overwrite the previous bitter-recipe upload artifacts in a deliberate, documented way. It must not create an unreadable pile of duplicate local status rows. Re-running `export` for the same book should overwrite the current bitter-recipe export folder for that book, matching the source-aware export behavior the repo already uses.

If a prepare step fails after creating a ledger entry but before finishing upload, the recovery rule is to keep the ledger row and mark it as a failed or interrupted state with the latest attempted run root. The next `prepare` or `status --refresh` run should be able to repair that state rather than requiring manual JSON surgery.

If Label Studio credentials are missing, the workflow should fail early with a direct error message. Do not fall back to the old giant interactive menu just to resolve credentials. Either resolve them through the bitter-recipe settings or through a tiny shared helper that has a stable contract.

If the new lane starts importing heavy benchmark or stage modules by accident, stop and cut the dependency back down before proceeding. Complexity drift is a design failure here, not an acceptable cleanup item for later.
