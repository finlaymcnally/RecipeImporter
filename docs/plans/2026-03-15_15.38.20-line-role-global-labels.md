# Remove Per-Row Line-Role Codex Shortlists

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, canonical line-role Codex batches will always let the model choose from the full global label vocabulary instead of a per-row shortlist. The user-visible effect is that obvious recipe titles, ingredients, instructions, and variant markers can no longer be ruled out before the model answers just because heuristic shortlist code guessed wrong. You can see the change by inspecting a generated prompt and by running the targeted line-role tests that now accept a Codex label which used to be rejected as “outside allowlist”.

## Context and Orientation

Canonical line-role inference lives in `cookimport/parsing/canonical_line_roles.py`. That module performs deterministic classification first, then optionally escalates unresolved or low-confidence rows to a Codex-backed batch runner. The Codex prompt text is built in `cookimport/llm/canonical_line_role_prompt.py`, and the versioned prompt file lives at `llm_pipelines/prompts/canonical-line-role-v1.prompt.md`.

The current bad behavior is split across three layers. First, `cookimport/parsing/recipe_block_atomizer.py` assigns heuristic `candidate_labels` to atomic rows. Second, `cookimport/parsing/canonical_line_roles.py` turns those heuristic labels into an allowlist with `_candidate_allowlist(...)` and passes that allowlist into `_CodexBatchTask`. Third, `cookimport/llm/canonical_line_role_prompt.py` tells the model it must pick from each row’s `candidate_labels`, and `_parse_codex_line_role_response(...)` rejects any row whose label is not in the per-row allowlist. Benchmark follow-up tooling in `cookimport/bench/followup_bundle.py` also treats “prediction not in candidate labels” as a violation, so docs and audits currently reinforce the same mistaken contract.

The change in this plan is intentionally narrower than deleting every heuristic label in the parsing system. The atomizer’s heuristic labels can stay as input to deterministic ownership logic for now. The LLM-facing behavior changes are: prompts stop carrying row-level shortlists, response parsing accepts any valid freeform label, Codex prediction metadata stops pretending the model only saw a subset, and benchmark/docs stop describing candidate labels as a hard constraint.

## Plan of Work

Edit `cookimport/llm/canonical_line_role_prompt.py` and `llm_pipelines/prompts/canonical-line-role-v1.prompt.md` so the prompt contract no longer mentions per-row `candidate_labels`, and the serialized target rows contain only context fields (`atomic_index`, recipe-span flag, previous/current/next line). Keep the global label list and precedence text.

Edit `cookimport/parsing/canonical_line_roles.py` so `_CodexBatchTask` no longer carries `allowed_by_index`, `_run_codex_batch(...)` no longer rewrites candidates with row-specific allowlists, and `_parse_codex_line_role_response(...)` validates only the global freeform label set plus atomic index ordering. For Codex predictions and sanitized/fallback outputs, set `candidate_labels` from the global label list rather than from a per-row allowlist so downstream artifacts no longer imply that a row-specific restriction still exists in the LLM path.

Edit `build_line_role_codex_execution_plan(...)` so prompt-plan rows no longer emit `candidate_labels` as part of the pending Codex work. If useful for debugging, leave the raw text and span metadata only; do not expose row-level shortlist semantics as part of the live prompt contract.

Update the benchmark/audit surface in `cookimport/bench/followup_bundle.py` to stop emitting `pred_not_in_candidate_labels`, `raw_not_in_candidate_labels`, and `parsed_not_in_candidate_labels` violations. Candidate-label extraction can remain for backward-compatible analysis of old runs, but it must be treated as optional historical metadata, not a correctness rule for new runs.

Update docs in `docs/10-llm/10-llm_README.md`, `docs/07-bench/07-bench_README.md`, and the understanding note created during investigation so they reflect the new contract. Add a short note only where behavior changed.
