# Split Nonrecipe Authority From Late Outputs

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the outside-recipe path will speak plainly through the code: semantic authority stays strict, while deterministic late outputs use the routed review queue when knowledge review did not run. A stage run with knowledge review off or in fallback mode should still build useful chunks and tables from the surviving outside-recipe rows instead of from obvious-junk exclusions or from an empty authority surface.

You can see the result by running the focused staging tests listed below. The key observable outcomes are: the shared stage session keeps `ConversionResult.non_recipe_blocks` tied to final authority only, tables and deterministic chunks read a separate late-output block set, and the test that currently asserts “no tables/chunks” is updated to prove the corrected behavior.

## Context and Orientation

The affected code lives in the shared stage runtime after recipe ownership is already settled. `cookimport/staging/nonrecipe_stage.py` owns the deterministic nonrecipe routing result. In this repository, “routing” means deciding which outside-recipe rows are obvious junk and which survive for later review. “authority” means the final semantic truth surface that is allowed to drive benchmark `KNOWLEDGE` labels and the mirrored `ConversionResult.non_recipe_blocks` cache.

`cookimport/staging/pipeline_runtime.py` wraps that routing result in `KnowledgeFinalResult`. That dataclass already carries multiple block lists: a strict authority list and the still-unreviewed reviewable list. `cookimport/staging/import_session_flows/output_stage.py` is the composition root that decides which list feeds late deterministic outputs such as tables and knowledge-off chunks.

The bug is that the output flow currently treats one field as if it meant both “final authority” and “late deterministic outside-recipe rows.” That conflation causes the knowledge-off path to use the wrong block set. The repair in this plan keeps the semantic authority contract intact while making the output path explicit.

## Plan of Work

First, change the shared runtime surface in `cookimport/staging/pipeline_runtime.py` so the block-list names match their real meanings. The final code should expose one field for strict semantic authority and one field for the deterministic late-output path. If an existing field can be deleted cleanly, delete it. If a rename is safer, rename it to something that cannot be misread as semantic truth.

Second, update `cookimport/staging/import_session_flows/output_stage.py` so it chooses the block list explicitly. Tables and deterministic chunks should use authoritative rows when knowledge review actually ran and produced reviewed authority. When knowledge review is off or falls back, those late deterministic outputs should instead use `NonrecipeRouteResult.reviewable_nonrecipe_blocks`. `ConversionResult.non_recipe_blocks` must remain tied to strict final authority only.

Third, update the focused tests in `tests/staging/test_import_session.py` and any nearby staging tests so they prove the corrected rule. The key regression should demonstrate that knowledge-off runs still emit late deterministic inputs from the routed review queue while leaving `ConversionResult.non_recipe_blocks` strict. If a small helper test in `tests/staging/test_nonrecipe_stage.py` or `tests/staging/test_stage_block_predictions.py` makes the contract easier to pin down, add it there.

Finally, update the owning docs. At minimum, `docs/05-staging/05-staging_readme.md` and `docs/plain-english-explanation.md` should stop teaching that knowledge-off late outputs come from final authority or that “final nonrecipe blocks” is a stable all-purpose concept.
