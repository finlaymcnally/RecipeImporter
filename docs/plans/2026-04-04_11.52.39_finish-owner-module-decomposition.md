## Operator Intent (Structured) - DO NOT DELETE

The operator wants the current large-file decomposition pass to be finished properly instead of stopping at the transitional stage. The useful part of the existing refactor should stay: explicit owner modules, smaller composition roots, and clearer subsystem boundaries. The weak part should be removed: new modules that still depend on `sys.modules[...]`, `globals().update(...)`, or other hidden compatibility scaffolding to reach names that should be imported explicitly. The result should stay behaviorally equivalent, remain friendly to the existing monkeypatch-heavy tests, and leave clear documentation for future edits.

## Purpose / Big Picture

After this change, a contributor should be able to open the new owner modules for bench, direct-exec, line-role, and external-AI cutdown work and understand what each file owns without tracing hidden runtime globals. The decomposition work will stop feeling half-finished. New modules will use explicit imports or deliberate hook objects instead of inheriting giant module snapshots, and the remaining giant coordinator files will shrink further into real composition roots.

This matters because the current state is only partially better than the pre-refactor state. Several extracted files are valid owner candidates, but they still import behavior through hidden compatibility tricks. That preserves old monkeypatch surfaces for tests, but it also preserves much of the old coupling. The proof for this plan is not merely that files moved around. The proof is that the same focused test suites still pass, `rg` no longer finds the dynamic compatibility pattern in the new owner files, and the main coordinator roots become substantially smaller and easier to navigate.

## Context and Orientation

This plan operates on the current uncommitted large-file decomposition wave. The existing roadmap is [docs/plans/2026-04-01_23.18.31_large-file-decomposition-roadmap.md](/home/mcnal/projects/recipeimport/docs/plans/2026-04-01_23.18.31_large-file-decomposition-roadmap.md). That roadmap already established the intended architectural direction: keep one obvious composition root per major subsystem, and move durable helper responsibilities into sibling owner modules. This plan does not replace that roadmap. It finishes the most important unfinished slices and should be read as the focused continuation of the roadmap's still-pending bench and deep-runtime work.

Several repository terms matter here.

A "composition root" is the module that wires a subsystem together. In this repository, that means files such as [cookimport/cli_support/bench_all_method.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/bench_all_method.py), [cookimport/llm/codex_exec_runner.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_exec_runner.py), or [cookimport/parsing/canonical_line_roles/runtime.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles/runtime.py). A composition root should coordinate work, assemble inputs and outputs, and call explicit helper owners. It should not silently behave like a shared global namespace for every extracted module.

An "owner module" is a file with one durable job. Examples already emerging in the current tree are [cookimport/cli_support/bench_all_method_qualitysuite.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/bench_all_method_qualitysuite.py), [cookimport/llm/codex_exec_workspace.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_exec_workspace.py), and [cookimport/parsing/canonical_line_roles/runtime_recovery.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles/runtime_recovery.py). An owner module should import what it needs explicitly. If it needs patchable behavior, that behavior should arrive through a named hook function, a small compatibility facade, or a helper object passed at call time.

The current weak spot is the compatibility pattern. Several extracted modules start by reading a larger runtime module out of `sys.modules[...]` and then copying that module's names into their own globals with `globals().update(...)`. That works, but it hides dependencies, preserves old coupling, and makes code review harder because the real imports are not visible. This plan removes that pattern from the extracted owner modules and leaves any necessary compatibility only in deliberate, documented facades.
