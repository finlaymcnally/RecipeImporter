# Preserve Block Continuity In Line-Role Packets

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The current line-role inline-json packet gives Codex a large flat list of rows and asks for one label per row. That shape is technically correct, but it encourages the wrong style of reasoning: scanning row by row instead of understanding how split rows belong to one paragraph, one title list, one recipe run, or one memoir passage. After this change, Codex should still receive the same raw textual evidence and still return row-level labels, but the packet it reads will preserve source-block continuity in a compact form so the model can see paragraph boundaries directly. Repair should keep that same continuity instead of narrowing down to detached failing rows.

The observable outcome is a new line-role structured packet format that groups rows by source block, a prompt that teaches the model to read block-by-block, and repair packets that preserve the full relevant blocks for failed rows. The model output contract stays row-level so downstream validation and promotion do not need a semantic rewrite.

## Surprises & Discoveries

- Observation: the current bad run already gives Codex very large shards, so the main problem is not lack of nearby context.
  Evidence: `data/output/2026-04-14_21.50.45/.../line-role-pipeline/runtime/line_role/workers/worker-001/.../initial_packet.json` contains `237` owned rows in one shard. `phase_plan_summary.json` shows first-turn payloads between about `39k` and `59k` characters, with `15` shards covering `3555` rows.

- Observation: despite those large shards, the packet shape is still flat enough to teach row-by-row thinking.
  Evidence: the current `initial_packet.json` payload stores one top-level `rows` array of row objects and at most one `context_before_rows` or `context_after_rows` entry, even when many owned rows belong to the same original block.

- Observation: repair currently loses the continuity we care about most.
  Evidence: `_build_line_role_repair_shard(...)` in `cookimport/parsing/canonical_line_roles/runtime_workers.py` rebuilds the shard with only unresolved owned rows, which strips accepted sibling rows from the same source block out of the repair packet.

- Observation: the current packet already preserves enough source facts to rebuild block-grouped structure without adding deterministic semantic guesses.
  Evidence: each owned row already carries `block_index`, and the shard input rows preserve the original row order. The runtime packet artifacts also show context rows in plain raw-text form. No extra classifier or region predictor is needed to produce grouped blocks.

- Observation: preserving whole blocks in repair changes the validation boundary as well as the prompt boundary.
  Evidence: the first implementation attempt kept accepted sibling rows inside the repair shard payload, but the existing validator still treated every row in that payload as owned. Repair then failed with missing-owned-row errors even when the model answered the unresolved row correctly. The fix was to keep full blocks for prompt evidence while validating against a derived owned-only shard.

## Decision Log

- Decision: keep the line-role answer contract row-level even though the prompt packet will become block-grouped.
  Rationale: downstream validation, scoring, and promotion all operate on row labels. The problem is how the model reads the evidence, not the final authority surface.
  Date/Author: 2026-04-14 / Codex

- Decision: use a compact `blocks` payload made of source-native `block_index` values and compact row tuples such as `["ctx", "..."]` and `["r01", "..."]`.
  Rationale: this preserves the key continuity signal the model currently lacks while staying much more compact than repeating full row objects and extra field names on every line.
  Date/Author: 2026-04-14 / Codex

- Decision: do not add deterministic region labels, guessed section types, or other repo-authored semantic hints.
  Rationale: the repo instructions explicitly forbid deterministic semantic guesswork in ambiguous cookbook labeling. The model should receive richer raw evidence, not repo-authored opinions about what the region “is.”
  Date/Author: 2026-04-14 / Codex

- Decision: preserve full source blocks in repair packets whenever those blocks contain unresolved rows, even if some rows in those blocks were already accepted.
  Rationale: repair should preserve the same reading continuity as the initial packet. Narrowing to only the unresolved rows recreates the exact “detached sentence” failure mode this plan is trying to remove.
  Date/Author: 2026-04-14 / Codex

- Decision: describe model-declared local spans or regimes as a follow-on experiment, not part of the first implementation.
  Rationale: the first milestone should answer one narrow question cleanly: does preserving source-block continuity materially improve holistic reading? Adding a second output layer at the same time would make that harder to test and debug.
  Date/Author: 2026-04-14 / Codex

## Outcomes & Retrospective

The implementation landed in the current line-role structured-session seam without changing the downstream row-label authority contract. Initial prompt packets now render compact grouped `blocks` instead of flat prompt-facing `rows` plus separate neighbor arrays. Each block contains compact `["ctx", text]` or `["rNN", text]` tuples, so the model sees paragraph/list continuity directly while still answering with the same row-level `{"rows":[{"row_id","label"}]}` contract.

Repair now preserves full source blocks whenever a block contains unresolved owned rows. Accepted sibling rows stay visible to the model as `ctx` evidence, but repo validation still scores only the unresolved owned rows by validating against a derived owned-only shard surface. That keeps the semantic continuity gain without widening deterministic authority.

Focused verification passed on the touched line-role prompt/runtime surfaces. The only unrelated failure encountered during the broader prompting-file rerun was the pre-existing gold-count expectation in `test_repo_gold_exposes_no_subsection_and_real_subsection_books`, where current `seaandsmokecutdown` gold exposes `67` `HOWTO_SECTION` rows rather than the test's older `>=100` expectation. That drift was not changed here.

The planned follow-on experiment remains intentionally deferred. If this block-grouped packet shape is still not enough, the next step is still a model-declared local span/regime layer that avoids deterministic semantic guesses.

## Context and Orientation

In this repository, “line-role” means the stage that assigns one semantic label to each source row before later deterministic grouping turns those labels into recipes and outside-recipe regions. The active inline-json line-role path lives in `cookimport/parsing/canonical_line_roles/runtime_workers.py`. The model-facing packet is built there, the prompt text is built there, and repair packets are also built there.

The current structured-session flow looks like this. `_build_line_role_structured_packet(...)` creates a top-level payload containing `rows`, optional `context_before_rows`, optional `context_after_rows`, and optional repair metadata such as `previous_failed_rows`. `_build_line_role_structured_prompt(...)` then tells Codex to label the rows in one pass and return one ordered `rows` array of `{"row_id","label"}` objects. `_evaluate_line_role_structured_response(...)` and the existing validator translate those returned row ids back to owned atomic indices and promote the accepted labels.

The key problem is not missing output structure. The model already sees huge flat shards. In the saved run `data/output/2026-04-14_21.50.45/.../line-role-pipeline/runtime/line_role/phase_plan_summary.json`, the average first-turn packet size is about `54,666` characters, and each shard covers about `237` owned rows. The issue is that a long flat `rows` table does not preserve paragraph identity very well. Many of the regressions come from prose that was split into smaller rows by the source-row builder, after which Codex interpreted those rows too independently.

One concrete example from the saved artifacts illustrates the failure. In the older April 13 run, the paragraph beginning `Salt, Fat, Acid, and Heat were the four elements...` existed as one larger row and stayed `OTHER`. In the April 14 row-split runs, that material was split into separate rows such as `Salt, Fat, Acid, and Heat were the four elements...`, `The rest was just...`, and `It was a revelation.` Those detached rows were promoted to `KNOWLEDGE`. The fix proposed here is to show those rows as members of one source block again without changing who owns the semantic decision.

The compact packet shape to target is:

    {
      "schema_version": "line_role_structured_packet.v5",
      "stage_key": "line_role",
      "packet_kind": "initial",
      "shard_id": "...",
      "blocks": [
        {
          "block_index": 204,
          "rows": [
            ["ctx", "Samin served us a whole meal..."],
            ["r01", "While we ate..."],
            ["r02", "We were all captivated..."]
          ]
        }
      ]
    }

Here, `ctx` means reference-only row text from the same local context, and `rXX` means an owned row that still requires a returned label. The packet stays compact because each row is just a two-item array instead of a repeated object with field names.

## Plan of Work

The first implementation milestone is to replace the flat prompt packet with a block-grouped prompt packet while keeping the current model output contract unchanged. In `cookimport/parsing/canonical_line_roles/runtime_workers.py`, add a helper that rebuilds the prompt packet into ordered source blocks. It should group the current owned rows by `block_index`, preserve their current order, and render each row in compact tuple form. Owned rows become `["rNN","text"]`, where `rNN` remains the packet-local output id already used by validation. Reference-only context rows become `["ctx","text"]`.

The top-level packet should stop carrying `rows`, `context_before_rows`, and `context_after_rows` as separate prompt-facing arrays. Instead, build one `blocks` array that is already in reading order and contains both owned and context rows inside the relevant blocks. The packet builder should preserve the current raw evidence only. It must not add guessed section types, heuristically assigned region labels, or any other repo-authored semantic interpretation.

The second milestone is to preserve block continuity during repair. Today `_build_line_role_repair_shard(...)` narrows the shard to only unresolved owned rows. Replace that repair narrowing with a block-aware narrowing strategy. The repair shard should include every source block that contains at least one unresolved owned row. Within those blocks, unresolved rows remain owned `rNN` rows and already accepted rows become `ctx` rows so the model can still see the full paragraph or list region. If the existing `context_before_rows` or `context_after_rows` logic still adds one adjacent block on either side, those rows should also be folded into the compact `blocks` shape rather than staying as a separate top-level field.

The third milestone is to rewrite the structured prompt text to match the new packet. `_build_line_role_structured_prompt(...)` should stop describing the packet as a flat `rows` array. Instead it should explain that the model is reading ordered `blocks`, that each block preserves source continuity, that only rows with `rNN` ids are answerable, and that `ctx` rows are reference-only. The prompt should explicitly tell the model to read block by block first and only then assign labels to the owned rows.

The fourth milestone is to keep validation and promotion stable while changing only the input representation. `_evaluate_line_role_structured_response(...)` and `_translate_line_role_structured_rows_payload(...)` should remain row-id based. The model still returns `{"rows":[{"row_id":"r01","label":"..."}]}`. The packet regrouping should not require any benchmark scorer or downstream recipe-grouping rewrite.

The fifth milestone is to update tests and docs. `tests/parsing/test_canonical_line_roles_prompting.py` should gain assertions that prompt packets now render grouped `blocks` rather than flat `rows`. `tests/parsing/test_canonical_line_roles_runtime.py` and `tests/parsing/test_canonical_line_roles_front_matter_pathology.py` should gain targeted coverage for repair packets preserving full blocks around unresolved rows. The parsing and LLM docs should be updated after implementation to document the new compact block-grouped prompt shape and the fact that repair preserves block continuity.

After this implementation is complete and validated, the follow-on experiment to try is model-declared local spans or regimes before row labels. That experiment should stay separate. The idea would be to have the model first declare contiguous local reading modes, such as one title-list run or one memoir/setup run, and then emit row labels consistent with those spans. That is deliberately not part of the first milestone because it introduces a second model output layer and would make it harder to isolate whether block-grouped evidence alone fixed the behavior.

## Validation and Acceptance

The implementation is acceptable only if all of the following are true.

First, the saved runtime artifacts for a new line-role run show that the initial prompt packet now contains compact grouped `blocks` rather than a top-level flat `rows` list. A human inspecting `structured_session/initial_packet.json` should be able to see which rows belong to the same source block immediately.

Second, the model output contract stays row-level. The model should still return one ordered `rows` array of `{"row_id","label"}` objects, and the existing row-grounded validator should continue to work without a semantic rewrite.

Third, repair packets preserve source-block continuity. When one or more rows from a block fail validation, the repair packet should still include the other rows from that same block as `ctx` evidence instead of stripping them away.

Fourth, the focused `.venv` tests listed above pass. The new tests should prove two things: the prompt packet is block-grouped and repair packets keep whole relevant blocks, not detached failing rows only.

Fifth, the long-lived docs explain the new packet shape and the fact that repair preserves block continuity without adding deterministic semantic guesses.

## Idempotence and Recovery

The planned code changes are safe to apply incrementally because they are confined to the line-role structured-session input representation. Validation and promotion remain row-id based, so if prompt rendering changes but response parsing does not, partial progress can still be tested locally without corrupting saved artifact formats elsewhere.

If an intermediate implementation breaks prompt rendering, the safe rollback is to restore the previous `_build_line_role_structured_packet(...)` and `_build_line_role_structured_prompt(...)` functions while leaving the validator unchanged. If repair continuity changes prove too invasive in the same patch, land initial block-grouped packets first and then apply the repair-block preservation change as a second commit while keeping this plan updated.

The plan should not delete any current runtime artifacts or benchmark outputs. All validation should happen through new runs or focused tests, leaving old saved artifacts available for comparison.
