# Relax Knowledge Shard Promotion To Accept Valid Task Packets

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The current knowledge stage already validates work at task-packet granularity, but promotion still behaves as if the only safe unit is the whole shard. That mismatch creates a bad user-visible outcome: a worker can return dozens of valid knowledge packet results, a few packet failures can happen in the same shard, and the stage promotes nothing from that shard. In benchmark summaries this looks like the knowledge review “fell apart” even when much of the shard was actually reviewed correctly.

After this change, the knowledge stage should promote chunk results from accepted task packets even when the parent shard wrapper is invalid only because coverage is incomplete. Semantically bad or missing packets must still remain unreviewed. The observable result is that a saved run like `data/output/2026-03-22_03.33.50/.../knowledge/` can produce promoted knowledge outputs from the validated packet subset, report partial review honestly, and stop treating every partially bad shard as all-or-nothing failure.

## Surprises & Discoveries

- Observation: the saved benchmark did not fail because all packet work was bad.
  Evidence: `data/output/2026-03-22_03.33.50/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-22_03.34.33/raw/llm/saltfatacidheatcutdown/knowledge/task_status.jsonl` shows `254` validated packets, `49` invalid packets, and `104` watchdog-killed packets.

- Observation: every parent shard proposal failed for the same wrapper-level reason even when most of its task packets were individually valid.
  Evidence: the five proposal wrappers under `.../knowledge/proposals/*.json` all carry `validation_errors = ["missing_owned_chunk_results"]`, with accepted result counts such as `52/82`, `73/82`, and `72/82` chunks already present in the payload.

- Observation: the current task aggregation metadata already records exactly which task packets were accepted.
  Evidence: `_aggregate_knowledge_task_payloads(...)` in `cookimport/llm/codex_farm_knowledge_orchestrator.py` writes `accepted_task_ids`, `fallback_task_ids`, `missing_chunk_ids`, `task_validation_errors_by_task_id`, and `task_id_by_chunk_id` into `validation_metadata["task_aggregation"]`.

- Observation: the final promotion seam ignores that packet-level acceptance metadata and only reads proposals whose top-level `validation_errors` array is empty.
  Evidence: `read_validated_knowledge_outputs_from_proposals(...)` in `cookimport/llm/codex_farm_knowledge_ingest.py` skips any wrapper whose `validation_errors` list is non-empty, even if the payload contains accepted chunk rows from valid task packets.

- Observation: current rollup metrics treat an invalid shard as fully unreviewed, which overstates loss once packet-level partial promotion becomes possible.
  Evidence: `_build_knowledge_review_rollup(...)` in `cookimport/llm/codex_farm_knowledge_orchestrator.py` derives `unreviewed_shard_count` as `invalid_shard_count + missing_output_shard_count` and `unreviewed_chunk_count` from whole invalid shards, not from the actual missing/invalid chunk subset.

- Observation: this stage already has one clean granularity boundary available for safe relaxation: the knowledge task packet is chunk-local.
  Evidence: each knowledge task row owns one bounded chunk packet, `task_status.jsonl` tracks validation at task scope, and `accepted_task_ids` map directly to `task_id_by_chunk_id` in the shard aggregation metadata.

- Observation: the task-aware reader alone was not enough because invalid parent shard wrappers were still being serialized with `payload = null`.
  Evidence: the direct workspace-worker aggregation path in `cookimport/llm/codex_farm_knowledge_orchestrator.py` previously wrote `final_payload = None` for every invalid parent shard, so the ingest seam had nothing to salvage until that write path preserved coverage-only invalid payloads.

## Outcomes & Retrospective

This plan is now implemented in the live workspace. The brittle point really was the handoff from task validation to shard promotion, not task validation itself. The landed change keeps that seam narrow in three places: invalid parent wrappers now retain their aggregated payload only when they are coverage-only invalid, the ingest reader only promotes accepted task-packet rows from those wrappers, and the reporting layer counts the missing subset instead of collapsing the entire shard into “unreviewed.”

The main visible outcome is truthful partial review. A shard can now contribute promoted knowledge outputs even when a sibling task packet failed, and the stage reports that as `partially_promoted_shards` plus chunk-level unreviewed counts rather than pretending the whole shard died. The main remaining gap is validation breadth in this dirty tree: the focused ingest/orchestrator slices pass, but one runtime packet test currently fails with a missing `packet_lease_status.json` file before it reaches the assertions relevant to this change, and the broad raw trio is already red for unrelated runtime/watchdog cases.

## Context and Orientation

The knowledge stage reviews non-recipe chunks after deterministic staging has already marked a region as outside-recipe. A “shard” is a worker assignment unit that can own many chunk packets. A “task packet” is the smaller unit inside a shard. In the current knowledge runtime, each task packet usually corresponds to one chunk packet, and the worker writes one semantic packet-result JSON file per task.

The relevant flow spans four files:

- `cookimport/llm/codex_farm_knowledge_orchestrator.py` runs the stage, aggregates task outputs back into one shard proposal wrapper, writes `promotion_report.json`, and later refines `08_nonrecipe_spans.json` / `09_knowledge_outputs.json`.
- `cookimport/llm/codex_farm_knowledge_ingest.py` validates knowledge payloads and currently exposes `read_validated_knowledge_outputs_from_proposals(...)`, which only returns rows from completely valid shard proposal wrappers.
- `tests/llm/test_knowledge_output_ingest.py` is the focused seam for proposal-reading behavior.
- `tests/llm/test_codex_farm_knowledge_orchestrator.py` and `tests/llm/test_codex_farm_knowledge_orchestrator_runtime.py` cover the stage-level promotion and review summary behavior.

Today the stage works like this:

1. The runtime validates individual task packets.
2. `_aggregate_knowledge_task_payloads(...)` merges accepted packet rows into one compact shard payload and records packet-level metadata such as `accepted_task_ids`, `fallback_task_ids`, and `missing_chunk_ids`.
3. The parent shard proposal wrapper is marked invalid if the merged shard payload does not cover every owned chunk, which commonly appears as `missing_owned_chunk_results`.
4. Later, `read_validated_knowledge_outputs_from_proposals(...)` skips every invalid wrapper entirely.
5. The refinement step therefore sees zero outputs from that shard, even if most packet rows were already valid and serialized into the wrapper payload.

That final skip is the seam this plan changes. The goal is not to promote malformed or semantically rejected data. The goal is to let the stage promote the accepted packet subset that it already knows is valid.

The saved benchmark that demonstrates the problem is:

`data/output/2026-03-22_03.33.50/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-22_03.34.33/raw/llm/saltfatacidheatcutdown/knowledge/`

In that run:

- `ks0000` had `52` validated packet results and `30` invalid packet results.
- `ks0001` had `73` validated packet results and `9` invalid packet results.
- `ks0004` had `72` validated packet results and `10` invalid packet results.

Yet all three parent shard proposals were dropped wholesale because the wrappers carried `missing_owned_chunk_results`.

## Plan of Work

Start by changing the proposal-reading seam in `cookimport/llm/codex_farm_knowledge_ingest.py`. Keep the current strict reader for fully validated shard wrappers, but add a task-aware promotion path that can also read accepted chunk rows from an invalid wrapper when all of the following are true:

- the wrapper payload itself still parses as a canonical knowledge bundle,
- the wrapper-level validation errors are limited to coverage-style incompleteness such as `missing_owned_chunk_results`,
- the wrapper metadata includes `task_aggregation`,
- the chunk row came from a task listed in `accepted_task_ids`,
- and that task is not listed in `task_validation_errors_by_task_id`.

That reader should return both the promoted chunk outputs and enough metadata to explain partial promotion, such as which shard ids were fully validated, which shard ids were partially promoted, which chunk ids were left unreviewed, and which task ids were excluded because of task validation errors.

Next, thread that richer reader through `cookimport/llm/codex_farm_knowledge_orchestrator.py`. The refinement step should use promoted chunk outputs from both fully valid shards and partially promotable shards. Then adjust the promotion report and review rollup so they stop pretending every invalid shard is wholly unreviewed. The report should distinguish at least these cases:

- fully validated shard,
- partially promoted shard,
- wholly unpromoted invalid shard,
- wholly missing-output shard.

The chunk and block counts should also become truthful. If a shard promotes `72` chunks and leaves `10` unreviewed, the run should report those `10` chunks as unreviewed, not the entire shard’s `82`. Preserve the existing shard-level counters where they are still useful, but add packet- or chunk-level promoted counters so operator-facing summaries can show real partial progress.

Then add focused regressions. One ingest-level test should prove that a proposal wrapper with `missing_owned_chunk_results` can still yield outputs for the accepted task subset. Another ingest-level test should prove that a task rejected for `semantic_snippet_echoes_full_chunk` does not promote its chunk even if sibling tasks from the same shard do promote. At the orchestrator level, add a regression where one shard contains two accepted tasks and one rejected task; after the change, refinement should consume the two accepted chunks, report partial review, and leave only the rejected chunk unreviewed.

Finally, update the short docs in `docs/10-llm/nonrecipe_knowledge_review.md` and, if needed, the broader note in `docs/10-llm/10-llm_README.md`. Keep those updates short. The docs need to say that knowledge promotion is now packet-aware: accepted packet results can refine final authority even when sibling packets from the same shard remain unreviewed.

## Idempotence and Recovery

These changes are safe to iterate on because the promotion seam is deterministic and file-backed. The saved run artifacts under `data/output/.../knowledge/` do not change while you develop the reader and rollup logic, so regressions are reproducible.

The main recovery rule is to avoid accidentally widening trust. If a draft implementation starts promoting rows from task ids that were not in `accepted_task_ids`, or starts promoting rows from wrappers invalidated for reasons other than coverage-style incompleteness, stop and tighten the reader before proceeding. Keep the safety checks and the negative tests in the same patch so the intended authority boundary stays explicit.

If the reporting changes become too invasive for one patch, prefer adding new partial-promotion counters while preserving old shard-level counters for compatibility. A temporary period where both counters exist is acceptable. A misleading partial-promotion rollout is not.

## Interfaces and Dependencies

In `cookimport/llm/codex_farm_knowledge_ingest.py`, keep the validation logic strict. The change belongs in the proposal-reading seam, not in the semantic validator. If you add a new reader, give it a name that makes the relaxed contract explicit, such as “promotable” or “accepted-task-aware,” and keep the old fully validated reader if other code still depends on that stricter meaning.

Additive counters such as `partially_promoted_shards`, `promoted_chunk_count`, or `promoted_block_count` are preferable to silently redefining existing fields.
