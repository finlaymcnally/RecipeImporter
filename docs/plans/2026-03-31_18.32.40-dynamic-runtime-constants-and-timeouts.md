## Purpose / Big Picture

After this change, the project will stop behaving differently just because the run entered through `cookimport stage`, interactive saved settings, Label Studio prediction/upload flows, prompt preview, or benchmark helpers. The same canonical setting values will drive all of those surfaces, completion timing will be tunable instead of machine-speed magic numbers, and the remaining correctness-sensitive “magic constants” will either become explicit run settings or move behind shared config helpers with one source of truth.

The user-visible result is consistency and tunability. A user should be able to set a value once, see the same value reflected in manifests and prompt preview, and know that different commands are not silently changing the model context window, timeout grace, EPUB segmentation window, upload bundle size, or split-cache wait behavior. The clearest proof will be: run settings serialization and adapters agree on one default, prompt preview reports the same resolved value, benchmark helpers use the same config surface, and targeted tests show the new `15` second completion grace defaults plus the new dynamic path handling and EPUB heuristic controls.

## Context and Orientation

This repository is a staged cookbook-import pipeline. Source files are normalized by importers, then the shared stage session decides recipe boundaries, refines recipes, routes outside-recipe content, optionally finalizes outside-recipe knowledge with Codex, and finally writes artifacts and manifests.

The key file for typed configuration is [cookimport/config/run_settings.py](/home/mcnal/projects/recipeimport/cookimport/config/run_settings.py). A “run setting” is a typed value that can be serialized into manifests, surfaced in interactive settings, and adapted into command/runtime kwargs. The adapter layer that turns `RunSettings` into command arguments lives in [cookimport/config/run_settings_adapters.py](/home/mcnal/projects/recipeimport/cookimport/config/run_settings_adapters.py). Interactive saved settings defaults live in [cookimport/cli_support/settings.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/settings.py). The public `stage` command surface lives in [cookimport/cli_commands/stage.py](/home/mcnal/projects/recipeimport/cookimport/cli_commands/stage.py). Label Studio prediction and upload flows have their own function signatures in [cookimport/labelstudio/ingest_flows/prediction_run.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest_flows/prediction_run.py) and [cookimport/labelstudio/ingest_flows/upload.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest_flows/upload.py). Prompt preview resolves existing-run settings in [cookimport/llm/prompt_preview.py](/home/mcnal/projects/recipeimport/cookimport/llm/prompt_preview.py).

The worker runtime and supervision code live under `cookimport/llm/` and `cookimport/parsing/canonical_line_roles/`. A “workspace worker” is the long-lived Codex session that edits a repo-written `task.json` and then runs a repo-owned helper to validate and advance the same session. Completion grace is the amount of time repo code waits after authoritative completion signals appear before force-terminating or declaring failure. Today these values are hard-coded in [cookimport/llm/codex_exec_runner.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_exec_runner.py), [cookimport/llm/knowledge_stage/_shared.py](/home/mcnal/projects/recipeimport/cookimport/llm/knowledge_stage/_shared.py), [cookimport/llm/recipe_same_session_handoff.py](/home/mcnal/projects/recipeimport/cookimport/llm/recipe_same_session_handoff.py), [cookimport/llm/knowledge_same_session_handoff.py](/home/mcnal/projects/recipeimport/cookimport/llm/knowledge_same_session_handoff.py), and [cookimport/parsing/canonical_line_roles/__init__.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles/__init__.py).

The EPUB importer lives in [cookimport/plugins/epub.py](/home/mcnal/projects/recipeimport/cookimport/plugins/epub.py). It contains importer-local heuristics for backtracking from yield/time anchors to titles and for looking ahead to ingredient runs or ingredient headers. These values currently affect correctness but are invisible magic constants.

Benchmark and support-code constants are split across a few places. Shared benchmark helper constants currently live in [cookimport/cli_support/__init__.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/__init__.py). Single-profile benchmark scheduling lives in [cookimport/cli_support/bench_single_profile.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/bench_single_profile.py). Single-book split-cache coordination lives in [cookimport/labelstudio/ingest_flows/split_cache.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest_flows/split_cache.py). Oracle upload behavior lives in [cookimport/bench/oracle_upload.py](/home/mcnal/projects/recipeimport/cookimport/bench/oracle_upload.py).

The project already documents its high-level architecture in [docs/plain-english-explanation.md](/home/mcnal/projects/recipeimport/docs/plain-english-explanation.md), [docs/AI_context.md](/home/mcnal/projects/recipeimport/docs/AI_context.md), and the numbered subsystem docs. ## Plan of Work

The first milestone is canonical default resolution. Add a narrow helper layer that exposes canonical default values for runtime knobs that are currently duplicated across entry surfaces, starting with `codex_farm_knowledge_context_blocks`. The preferred location is the existing config layer, not a new global constants module. The implementation should make `RunSettings` the authority for ordinary runtime defaults and should let command/function signatures use either `None` plus canonical resolution or shared constants derived from that same source. The same cleanup should cover any other duplicated values in this work that are plainly meant to be one setting, such as knowledge packet budgets when they are already part of `RunSettings`.

The second milestone is worker completion timing. Define explicit completion-grace settings with plain-language names in [cookimport/config/run_settings.py](/home/mcnal/projects/recipeimport/cookimport/config/run_settings.py) for the values that materially control “worker completed but repo is still waiting” behavior. At minimum, provide a setting for the workspace completion quiescence grace and a setting for the completed-process termination grace. Set both defaults to `15` seconds. Then thread those settings through the recipe, knowledge, and line-role runtimes so that the runner and same-session helpers stop using the hidden 2 second and 5 second literals. Keep stricter watchdog and pathology constants separate unless they are part of the same completion semantics; this plan is about completion grace, not every existing watchdog rule.

The third milestone is dynamic temp/cache path handling. Replace hard-coded `/tmp/...` and home-cache fallbacks in [cookimport/ocr/doctr_engine.py](/home/mcnal/projects/recipeimport/cookimport/ocr/doctr_engine.py), [cookimport/llm/codex_exec_runner.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_exec_runner.py), and [cookimport/labelstudio/prelabel.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/prelabel.py) with the standard library’s temp and cache-resolution behavior. Use `tempfile.gettempdir()` or `tempfile.mkdtemp()` style helpers for temporary directories, preserve existing env overrides when present, and only fall back to a specific path after trying dynamic resolution. The result must keep the current workspace isolation semantics while no longer assuming `/tmp` exists or is the right scratch root.

The fourth milestone is EPUB heuristic surfacing. Introduce a small set of importer-facing run settings for the correctness-sensitive EPUB window/cap values that are currently invisible: title backtrack limit, ingredient-run lookahead, ingredient-header lookahead, and maximum title length. Add them to `RunSettings`, keep them internal rather than cluttering the normal operator UI if appropriate, thread them through stage and Label Studio ingestion paths, and update the EPUB importer to read those values rather than using inline literals. The implementation should not attempt to redesign the heuristic itself in this pass; the goal is to make the behavior explicit, tunable, and documented.

The fifth milestone is knowledge grouping alignment. Update [cookimport/llm/knowledge_stage/task_file_contracts.py](/home/mcnal/projects/recipeimport/cookimport/llm/knowledge_stage/task_file_contracts.py) so grouping batch partitioning is driven by explicit settings rather than hidden constants. The cleanest route is to add grouping-specific run settings, because grouping is not the same artifact as classification packets. Name them clearly as grouping batch limits, default them to the current effective behavior only if that is still desired, and pass them through the knowledge runtime from `RunSettings`. Do not silently reuse classification packet budgets if that would blur two different concepts; instead, make the relationship explicit in naming and docs.

The sixth milestone is support-surface cleanup. For single-book split-cache wait and poll intervals, benchmark upload-bundle target sizes, Oracle upload thresholds, and single-profile benchmark scheduler caps, decide case by case whether the value belongs in `RunSettings`, an env override, or a narrow shared constant helper. The rule is: if it materially changes benchmark behavior across runs and should be recorded in manifests or benchmark config, it belongs in `RunSettings`; if it is infrastructure-only and mainly useful for operator overrides, it belongs behind an env/config helper with one shared definition. While making these changes, remove path-shape assumptions where helper-based path resolution already exists or can be added cheaply. For example, benchmark record classification logic that depends on hard-coded path substrings should prefer structured manifest fields or shared helper functions where possible.

The last milestone is documentation and tests. Update the shortest durable docs that own these behaviors: [docs/10-llm/10-llm_README.md](/home/mcnal/projects/recipeimport/docs/10-llm/10-llm_README.md) for completion timing and worker behavior, [docs/03-ingestion/03-ingestion_readme.md](/home/mcnal/projects/recipeimport/docs/03-ingestion/03-ingestion_readme.md) or [docs/04-parsing/04-parsing_readme.md](/home/mcnal/projects/recipeimport/docs/04-parsing/04-parsing_readme.md) for EPUB heuristic settings, and a short relevant note in the benchmark/support docs if new benchmark knobs are added. Add focused tests that prove cross-surface default alignment, prompt preview parity, 15 second default completion grace, dynamic temp/cache path resolution, EPUB settings plumbing, and benchmark/split-cache knob wiring.

## Idempotence and Recovery

This work should be implemented incrementally and safely. Each milestone is additive and can be validated independently. If a partial edit breaks a command surface, restore the adapter or helper so all call sites compile before proceeding to the next milestone. Avoid broad refactors that rewrite unrelated defaults or watchdog behavior.

If a new setting turns out to be overexposed in the normal operator surface, keep the typed setting but mark it internal in `RunSettings` metadata rather than removing the underlying implementation. If a benchmark/infrastructure knob does not belong in `RunSettings`, move it to a shared helper with env overrides rather than leaving multiple scattered literals.

All commands in this plan are repeatable. The test commands can be re-run after each milestone. When changing default values, update tests in the same commit so failures clearly track the intended new behavior.
