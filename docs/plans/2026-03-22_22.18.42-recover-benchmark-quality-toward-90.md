# Recover benchmark quality toward 90 by fixing the actual dominant error buckets

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

The current Salt Fat cutdown benchmark proves that the recent routing-only and utility-boundary work helped, but it also proves that the repo is still far from the user’s actual target. In the verified benchmark at `data/golden/benchmark-vs-golden/2026-03-22_21.47.46/`, the Codex-backed run reaches `strict_accuracy = 0.716917` and `macro_f1_excluding_other = 0.803022`, which is meaningfully better than the paired vanilla control but nowhere near a `>90%` quality bar.

The important news is that the remaining gap is not mysterious. The scored artifacts show three dominant error buckets: `OTHER -> KNOWLEDGE`, `KNOWLEDGE -> OTHER`, and `KNOWLEDGE -> INSTRUCTION_LINE`. Those three buckets alone account for enough errors that, if they were mostly fixed, the current run would have a mathematical path well past 90. The goal of this plan is therefore not “general improvement.” It is to eliminate those concrete buckets by changing the exact seams that produced them.

The benchmark evidence also says where to spend effort. This is overwhelmingly an outside-recipe nonrecipe-labeling problem, not a recipe-assembly or span-boundary problem. The reviewed bundle summary reports that about `79.3%` of scored lines live outside active recipe spans, with inside-span accuracy already around `94.06%` but outside-span accuracy only around `65.86%`. If inside-span performance stayed flat, outside-span accuracy would have to climb to roughly `95.3%` to pull overall strict accuracy above `95%`. Boundary exactness is already effectively solved in this packet, so the shortest route is to improve outside-span label choice and routing quality rather than recipe reconstruction.

After this work, a user should be able to rerun the Salt Fat canonical benchmark, open the benchmark folder, and see that obvious blurbs and book-pitch prose are filtered earlier, useful explanatory headings are not being dropped out of knowledge chunks, outside-recipe fallback does not leak into `INSTRUCTION_LINE`, and the knowledge-stage reports once again expose why chunks were kept or rejected. The visible proof is a future approved rerun whose bucket counts shrink in the expected directions, plus local tests that make those classes hard to regress.

## Decision Log

- Decision: the next quality push should target the three dominant scored buckets directly instead of doing another broad prompt-only rewrite.
  Rationale: `OTHER -> KNOWLEDGE`, `KNOWLEDGE -> OTHER`, and `KNOWLEDGE -> INSTRUCTION_LINE` together explain almost the entire remaining gap. Fixes should be selected by whether they plausibly shrink those buckets, not by whether they sound generally reasonable.
  Date/Author: 2026-03-22 / Codex

- Decision: obvious blurbs, endorsements, and book-pitch prose should be filtered earlier by deterministic or tightly bounded routing logic, not left for the knowledge stage to reject one line at a time.
  Rationale: those rows are low-risk junk, and their presence in the `OTHER -> KNOWLEDGE` bucket shows that failing open on them costs quality. Upstream removal is safer than relying on prompt skepticism alone.
  Date/Author: 2026-03-22 / Codex

- Decision: useful heading inheritance, if added, must live inside the knowledge stage or its immediate refinement path, not in line-role.
  Rationale: line-role still must not become a semantic knowledge classifier again.   Date/Author: 2026-03-22 / Codex

- Decision: outside-recipe fallback must be constrained so `INSTRUCTION_LINE` cannot be emitted for non-recipe text unless the row is truly inside a recipe span or otherwise structurally owned by a recipe.
  Rationale: the joined-line evidence shows that the current fallback is violating the intended authority boundary. This is a routing bug, not a prompt nuance.
  Date/Author: 2026-03-22 / Codex

- Decision: the utility-profile layer should remain advisory, but it must stop screaming “negative” on a majority of obviously useful chunks.
  Rationale: advisory cues are valuable only if they discriminate. The current overfiring negative cues are more likely to confuse the worker than to improve calibration.
  Date/Author: 2026-03-22 / Codex

- Decision: restoring reason-code counts in final artifacts is mandatory before another serious tuning cycle, even though it is not itself a direct quality gain.
  Rationale: once the obvious scored buckets shrink, future iterations will need attribution by reason family. Keeping `reason_code_counts` empty would force blind tuning and repeat the same debugging loop.
  Date/Author: 2026-03-22 / Codex

- Decision: this plan targets a high-80s intermediate gate first, while keeping `>90%` as the explicit stretch target for the approved Salt Fat rerun.
  Rationale: the current run is too far from `>90%` to promise a one-shot jump honestly. The implementation order should first remove the largest structural and semantic errors and then let the approved rerun determine whether a second narrower push is needed.
  Date/Author: 2026-03-22 / Codex

- Decision: recipe correction and final recipe assembly are explicitly out of scope for the first pass of this plan unless new evidence shows they have become dominant again.
  Rationale: the current benchmark evidence says the shortest path to the user’s target runs through outside-span nonrecipe label choice. Pulling recipe-pipeline work into this iteration would broaden the plan without attacking the active bottleneck.
  Date/Author: 2026-03-22 / Codex

- Decision: deterministic logic may exclude only overwhelmingly obvious non-semantic junk, but it must never make, proxy, or pre-gate the semantic `knowledge` versus `other` distinction.
  Rationale: the user has made this a hard repository rule. Any “knowledge-likely” gate, heuristic semantic classifier, or deterministic proxy for useful-vs-not-useful prose would reintroduce the exact failure mode the user wants to avoid. ## Outcomes & Retrospective

This plan is not implemented yet. Its value is that it turns “quality toward 90” into a concrete attack plan tied to verified artifacts instead of intuition. The benchmark already showed that the repo can move meaningfully with routing and knowledge-stage changes; this plan says exactly where the next gains should come from and how to tell whether they landed.

The major lesson from the research pass is that the benchmark is not mainly asking for a smarter general-purpose model. It is asking for cleaner ownership boundaries and a harder outside-span `OTHER` prior for obvious junk only, while preserving the hard rule that semantic `knowledge` versus `other` remains LLM-only. Too many obvious blurbs still get semantic airtime, too many knowledge headings get stripped even when their body is useful, and the fallback path still invents recipe-style labels outside recipes. Those are fixable repository behaviors, not mysteries of model capability.

The current benchmark gap is dominated by three families, all of them outside the high-performing recipe interior:

`OTHER -> KNOWLEDGE` means the final scored label became `KNOWLEDGE` even though the gold label is `OTHER`. In this run, those are often blurbs, endorsements, “learn to cook” framing, or cookbook sales-pitch language that should either have been excluded upstream or rejected later as `other`.

`KNOWLEDGE -> OTHER` means the gold says the line is useful cookbook knowledge but the final system dropped it to `OTHER`. In this run, a large subfamily is useful section headings and conceptual headings that were stripped even though their neighboring body lines in the same chunk were correctly retained.

`KNOWLEDGE -> INSTRUCTION_LINE` means the system treated clearly outside-recipe explanatory prose as recipe-local instruction. In this run, the benchmark evidence shows these rows are being produced by a fallback path on non-recipe lines, not by genuine recipe ownership.

The external benchmark review added one useful frame that matches the local evidence: treat this as an outside-span nonrecipe line-role and routing problem first, not as a recipe-build or span-boundary problem. This plan adopts that framing for prioritization, but not in a dogmatic way. The knowledge stage still has real work to do on mixed framing chunks and heading retention, so the plan remains cross-stage where the evidence requires it. It also now explicitly rejects the review’s suggestion of a deterministic `knowledge-likely` gate, because the user has clarified that only the LLM may make the `knowledge` versus `other` distinction.

This plan deliberately builds on, but does not require reopening, the prior routing-only plan [docs/plans/2026-03-22_21.03.40-finish-line-role-routing-only-and-knowledge-sole-authority.md](/home/mcnal/projects/recipeimport/docs/plans/2026-03-22_21.03.40-finish-line-role-routing-only-and-knowledge-sole-authority.md), the diversion plan [docs/plans/2026-03-22_21.29.17-expand-line-role-obvious-junk-diversion.md](/home/mcnal/projects/recipeimport/docs/plans/2026-03-22_21.29.17-expand-line-role-obvious-junk-diversion.md), and the utility-boundary plan [docs/plans/2026-03-22_20.55.08-teach-knowledge-stage-utility-not-just-factuality.md](/home/mcnal/projects/recipeimport/docs/plans/2026-03-22_20.55.08-teach-knowledge-stage-utility-not-just-factuality.md). ## Milestones

### Milestone 1: Rebuild benchmark observability around the current failure buckets

At the end of this milestone, the benchmark and preserved stage artifacts should make it easy to answer two questions: why did this line become `KNOWLEDGE`, and why did this useful line get dropped or misrouted? That means restoring reason-code counts in the final knowledge outputs, preserving those counts in promotion-report paths, and adding or improving cheap per-bucket summaries for the dominant scored families.

This milestone comes first because the current final artifacts still lose the new reason-code contract. Without that, future tuning will keep depending on ad hoc manual artifact spelunking instead of stable post-run counters. Fixing observability alone will not raise the score, but it will shorten every later iteration.

Acceptance for this milestone is that `09_knowledge_outputs.json` and `promotion_report.json` no longer have empty `reason_code_counts`, and the benchmark review path can distinguish whether a future change mainly reduced `book_framing_or_marketing`, `decorative_heading_only`, `navigation_or_chapter_taxonomy`, or some other family.

### Milestone 2: Remove the remaining obvious blurb and framing false positives before semantic review

At the end of this milestone, more of the lines that are currently scoring as `OTHER -> KNOWLEDGE` should never reach knowledge review at all. The goal is not to give line-role semantic knowledge authority back. The goal is to widen deterministic or tightly bounded review exclusion for low-risk junk families that still leak through: endorsements, praise paragraphs, “learn to cook” framing, and cookbook sales-pitch or teacher-pitch prose that is not itself durable knowledge.

The implementation work belongs mainly in [cookimport/parsing/canonical_line_roles.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles.py), [cookimport/llm/canonical_line_role_prompt.py](/home/mcnal/projects/recipeimport/cookimport/llm/canonical_line_role_prompt.py), and the prompt assets under `llm_pipelines/prompts/`. The parser and prompt must agree on what counts as safe exclusion. The new rules must remain narrow enough that actual teaching prose still fails open to the knowledge stage.

Acceptance for this milestone is a focused test set plus preserved examples showing that the obvious pitch/blurb families are excluded upstream, and that the rerun artifact `08_nonrecipe_review_exclusions.jsonl` contains more of those exact families without swallowing legitimate cooking explanation.

### Milestone 3: Fix outside-recipe fallback leakage into recipe-structure labels

At the end of this milestone, outside-recipe lines that are not within recipe spans must no longer fall back to `INSTRUCTION_LINE` or other recipe-local structure labels just because they look imperative or procedural. The verified run shows this happening on explanatory prose such as salting guidance and emulsion repair instructions, all of which should remain outside-recipe and thus either review-eligible `OTHER` or final `KNOWLEDGE`.

The likely seams are [cookimport/parsing/label_source_of_truth.py](/home/mcnal/projects/recipeimport/cookimport/parsing/label_source_of_truth.py), [cookimport/staging/stage_block_predictions.py](/home/mcnal/projects/recipeimport/cookimport/staging/stage_block_predictions.py), and the line-role parsing/projection helpers that currently choose fallback decisions. The decisive rule after this milestone is simple: recipe-local labels require recipe ownership. Outside-recipe fallback should degrade to review-eligible `OTHER`, not to `INSTRUCTION_LINE`.

Acceptance for this milestone is that the current `KNOWLEDGE -> INSTRUCTION_LINE` bucket disappears or shrinks dramatically in scored artifacts, and representative lines like `302`, `305`, `322`, `330`, `346`, `542`, and `544` no longer appear in `joined_line_table.jsonl` as `decided_by: fallback` with `pred_label: INSTRUCTION_LINE`. This milestone should be treated as a larger lever than codex-vs-rule disagreement for the current packet.

### Milestone 4: Make the knowledge stage preserve useful headings and stop over-believing mixed framing chunks

At the end of this milestone, the knowledge stage should do two things better at once. First, when a useful explanatory chunk contains a short conceptual heading that directly introduces the retained knowledge, that heading should stay with the useful chunk instead of being thrown away as decorative. Second, chunks that mix useful-sounding rhetoric with obvious book framing should stop getting partial promotions unless the retained lines clearly stand on their own as durable leverage.

This milestone belongs inside the knowledge-stage prompt, helper examples, validator/repair feedback, and, if needed, a narrow post-validation or repair rule that only operates after a chunk is already accepted as useful. The key files are [cookimport/llm/codex_farm_knowledge_orchestrator.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_orchestrator.py), [cookimport/llm/knowledge_workspace_tools.py](/home/mcnal/projects/recipeimport/cookimport/llm/knowledge_workspace_tools.py), [cookimport/llm/codex_farm_knowledge_ingest.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_ingest.py), and the task/job builders that write knowledge hints.

Acceptance for this milestone is that the useful-heading misses in the current run are preserved as `knowledge` on rerun or in new deterministic canaries, while mixed blurby chunks lose more of their framing/promo lines instead of preserving them under `knowledge`.

### Milestone 5: Narrow the negative utility cues until they are genuinely discriminative

At the end of this milestone, the advisory `utility_profile` data should help the knowledge worker rather than muddy the water. Positive cues should stay attached to genuinely useful chunks, but negative cues such as memoir or book-framing should not trigger on most useful chunks in the corpus. Borderline and strong-negative states should become rare enough to carry information.

The implementation work belongs in [cookimport/parsing/chunks.py](/home/mcnal/projects/recipeimport/cookimport/parsing/chunks.py) and [cookimport/llm/codex_farm_knowledge_jobs.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_jobs.py). The plan should treat the current cueing layer as a noisy heuristic pass and explicitly tune it against the benchmark cases surfaced in this document, not against vague intuitions.

Acceptance for this milestone is that a new local measurement over `task_manifest.jsonl`-style outputs shows a materially smaller share of chunks with both positive and negative cues, borderline markers, and strong-negative markers, without losing positive cues on clearly useful chunks like the salt-science and emulsion-repair sections.

## Idempotence and Recovery

This plan should be implemented additively and in the order listed above. The safest sequence is observability first, then upstream exclusion, then fallback, then knowledge-stage semantic tightening, then utility-cue tuning. Each step can be validated locally before touching the next one.

If the expanded exclusion logic proves too aggressive, back out only the newest exclusion family and keep the observability and fallback fixes. Those two changes are independently valuable and do not require keeping every new exclusion rule.

If the heading/body coherence change starts over-promoting navigation or decorative headings, narrow the knowledge-stage-side inheritance rule instead of moving that judgment upstream into line-role. The architecture should stay routing-only upstream.

If the utility-profile tuning goes wrong, the safe rollback is to keep the core prompt and artifact fixes but remove or narrow the negative cue patterns in [cookimport/parsing/chunks.py](/home/mcnal/projects/recipeimport/cookimport/parsing/chunks.py). Those cues are advisory and should never become a silent source of quality loss.

The approved benchmark rerun is the only expensive step. If it disappoints, preserve the new artifact bundle, update this plan with the actual residual buckets, and scope the follow-up work from that evidence rather than guessing.
