# Auto-tag recipes with a catalog-driven rule engine (LLM-ready)

## What It Does

The tagging subsystem (`cookimport/tagging/`) automatically suggests recipe tags from the existing tag catalog. You point it at staged draft files or DB recipe rows and it returns a list of tags with confidence scores and human-readable evidence explaining each choice.

    # Suggest tags for staged drafts (dry run)
    cookimport tag-recipes suggest --draft-dir "data/output/<run>/final drafts/<workbook>/" \
      --catalog-json data/tagging/tag_catalog.json --explain

    # Apply tags to a DB recipe
    cookimport tag-recipes apply --db-url "$COOKIMPORT_DATABASE_URL" --recipe-id "<uuid>" \
      --catalog-json data/tagging/tag_catalog.json --apply

    # Export catalog from DB to JSON snapshot
    cookimport tag-catalog export --db-url "$COOKIMPORT_DATABASE_URL" --out data/tagging/tag_catalog.json

    # Inspect what the tagger "sees" for a given recipe
    cookimport tag-recipes debug-signals --draft path/to/r0.json

Example output:

    Recipe: Instant Pot Chicken Chili
    Suggested tags:
      - cooking-style/instant-pot (0.80)  matched 'instant pot' in title
      - main-protein/chicken (0.70)       matched chicken cut in ingredients
      - cooking-method/pressure-cook (0.60) matched pressure cook in instructions
      - dish-type/stew (0.70)             matched stew in title


## Design Principles

**Precision over recall.** Missing a tag is better than adding a wrong one, because DB writes are insert-only and there's no "who assigned this" tracking, so a bad tag can't be safely auto-removed later.

**Catalog-driven, not hard-coded.** Rules are keyed by tag `key_norm` (e.g., `instant-pot`), never by UUID. The engine skips any rule whose key doesn't exist in the loaded catalog, so adding new tags to the catalog requires zero code changes — the new tags simply become available.

**Deterministic first, LLM optional.** The deterministic engine runs on every recipe. The LLM second pass is gated behind `--llm`, disabled by default, and only invoked for categories the deterministic engine left empty. If no provider is configured, it warns once and continues with deterministic results.

**Insert-only DB writes.** All writes use `INSERT ... ON CONFLICT (recipe_id, tag_id) DO NOTHING`. Re-running is safe. The system never deletes or replaces existing assignments.


## Architecture

The pipeline has four stages, each in its own module:

### 1. Catalog loading (`catalog.py`)

Three frozen dataclasses — `TagCategory`, `Tag`, `TagCatalog` — with lookup dicts (`tags_by_key`, `categories_by_id`, `tags_by_category_id`, etc.). The catalog can be loaded from Postgres (`load_catalog_from_db`) or from a JSON snapshot (`load_catalog_from_json`).

The JSON export includes a `catalog_fingerprint` (SHA-256 over a deterministic payload of all categories + tags). This fingerprint is recorded in every run report and checked before DB apply to catch catalog drift — if a tag ID in the snapshot no longer exists in the target DB, the apply aborts with a clear message.

DB queries assume `public.recipe_tag_categories` for categories and `public.recipe_tags` for tags. The category table name was inferred from conventions (the schema files were not in the repo at build time) and should be verified on first DB run.

### 2. Signal extraction (`signals.py`, `db_read.py`)

A `RecipeSignalPack` is a flat dataclass containing everything the engine needs to score a recipe:

- **Text fields:** title, description, notes (strings)
- **List fields:** ingredients, instructions (lists of strings)
- **Numeric fields:** prep/cook/total time in minutes, max oven temp in F, spice level, attention level, cleanup level, yield phrase

Two extractors build this pack:

- `signals_from_draft_json(path)` — reads a cookbook3 `RecipeDraftV1` JSON file, pulling `raw_text` from ingredient lines and `instruction` from steps, converting `cook_time_seconds` to minutes.
- `fetch_recipe_bundle(db_url, recipe_id)` — queries `public.recipes`, `public.recipe_steps`, and `public.step_ingredient_lines`.

Text normalization (`normalize_text_for_matching`) lowercases, strips Unicode accents (NFKD decomposition), replaces punctuation with spaces, and collapses whitespace. This means "Sautéed" becomes "sauteed" and "Café" becomes "cafe" for matching purposes.

### 3. Deterministic scoring (`rules.py`, `policies.py`, `engine.py`)

**Rules** (`rules.py`): A `TAG_RULES` dictionary with ~80 entries, each keyed by tag `key_norm`. Each rule has:

- `patterns`: list of `{field, regex, weight, evidence}` — the engine sums weights of all matched patterns to get a raw score (clamped to 0..1).
- `exclude_patterns` (optional): if any match, the tag is suppressed entirely.
- `min_score`: per-tag threshold (default 0.6).

Current rule coverage by category:

| Category | Tags covered | Signal source |
|----------|-------------|---------------|
| cooking-style | instant-pot, slow-cooker, air-fryer, grill, sheet-pan, one-pot, no-cook, stovetop-only, oven-only | Title + instructions |
| equipment | blender, immersion-blender, food-processor, dutch-oven, wok, cast-iron, stand-mixer, smoker | Instructions + title |
| cooking-method | bake, roast, steam, simmer, saute, braise, pressure-cook, deep-fry, fry, smoke, sous-vide, broil, poach | Instructions + title |
| effort | quick, hands-off-friendly, minimal-prep, beginner-friendly, weekend | Title + numeric (total_time, attention_level) |
| main-protein | chicken, beef, pork, fish, shrimp, tofu, chickpeas, lentils, eggs, lamb, turkey, beans, tempeh, seitan | **Ingredients** (primary), title (secondary) |
| main-carb | rice, pasta-noodles, potatoes, sweet-potatoes, tortillas, bread, quinoa, couscous | **Ingredients** (primary), title (secondary) |
| storage | freezer-friendly, meal-prep-friendly, make-ahead, batch-cooking | Notes + description + title |
| meal-type | breakfast, dessert, snack | Title + description |
| course | appetizer, side-dish, sauce, drink, condiment, dressing | Title |
| dish-type | soup, stew, salad, sandwich, pasta, curry, casserole, stir-fry, tacos, pizza, bowl, rice-dish, wrap, baked-goods | Title |
| cuisine, dietary, flavor-profile, heat-source, vibe, season, holiday, techniques | *No deterministic rules yet* | Reserved for LLM second pass |

Rule authoring lessons learned:

- **Protein/carb rules should be ingredient-driven.** Title-only matches cause false positives (e.g., "Chicken Stock" is not a chicken recipe). The current approach uses high-weight patterns for specific cuts ("chicken breast", "beef chuck") and low-weight fallback for generic mentions ("chicken", "beef").
- **Don't use exclude patterns for broth/stock.** An exclude pattern on "beef broth" suppresses the entire tag even when "beef chuck" also appears in the same recipe's ingredients. Instead, the separate-pattern-weight approach naturally handles this: a recipe with only "beef broth" scores low (0.3) and misses the threshold, while "beef chuck" + "beef broth" scores high (1.0).
- **Cooking method weights must be >= 0.6.** Methods like "simmer" and "sauté" typically appear only in instructions (never titles), so a single instruction-field match needs to clear the threshold on its own.

**Policies** (`policies.py`): A `CATEGORY_POLICIES` dictionary defines constraints per category:

- `mode: "single"` — only the highest-confidence tag is kept (main-protein, main-carb, meal-type, course, dish-type).
- `mode: "multi"` — all tags above threshold, up to `max_tags` (equipment, cooking-method, effort, storage, occasion).
- `min_confidence: 0.6` — default threshold (can be overridden per-tag via `min_score` in the rule).

**Engine** (`engine.py`): `suggest_tags_deterministic(catalog, signals)` runs all rules against the signal pack:

1. Lazily compiles all regex patterns once (module-level cache).
2. Normalizes text fields on first access (per-call cache).
3. For each tag rule: checks excludes, sums matched pattern weights, applies threshold.
4. Adds numeric-based suggestions (quick from total_time, hands-off from attention_level).
5. Groups by category and applies single/multi policies.
6. Returns sorted `list[TagSuggestion]`, each with `tag_key`, `category_key`, `confidence`, and `evidence`.

### 4. LLM second pass (`llm_second_pass.py`) — scaffolded, not wired

The `--llm` flag on `suggest` and `apply` enables a gated second pass that only runs for categories the deterministic engine left empty. The flow:

1. Identify missing categories (categories with policies defined but no deterministic suggestions).
2. For each, build a candidate shortlist (up to 10 tags) via token overlap between tag display names and the recipe's combined text.
3. Call the LLM with candidates + recipe context, expecting a structured JSON response.
4. Validate strictly: tag must exist in catalog, must match stated category, must be in candidate shortlist, confidence 0..1.
5. Merge with deterministic results and re-apply category policies.

Currently uses a mock provider that returns an empty list. If no provider is configured, warns once and falls back to deterministic-only.

## CLI Reference

**`cookimport tag-catalog export`** — Export the tag catalog from Postgres to a JSON snapshot.

| Flag | Default | Notes |
|------|---------|-------|
| `--db-url` | `$COOKIMPORT_DATABASE_URL` | Postgres connection string |
| `--out` | required | Output path for `tag_catalog.json` |

**`cookimport tag-recipes suggest`** — Suggest tags for draft files (does not write to DB).

| Flag | Default | Notes |
|------|---------|-------|
| `--draft` | | Single draft JSON path |
| `--draft-dir` | | Folder of drafts (recursive `*.json`) |
| `--catalog-json` | required | Path to exported catalog |
| `--out-dir` | | Write `*.tags.json` per recipe |
| `--explain` | false | Show evidence strings |
| `--limit N` | all | Process at most N recipes |
| `--llm` | false | Enable LLM second pass |

Always writes a `tagging_report.json` with aggregate stats (per-category counts, zero-tag recipes, catalog fingerprint).

**`cookimport tag-recipes apply`** — Tag recipes in the database (dry-run by default).

| Flag | Default | Notes |
|------|---------|-------|
| `--db-url` | `$COOKIMPORT_DATABASE_URL` | Postgres connection string |
| `--recipe-id` | | Single recipe UUID |
| `--import-batch-id` | | Scope to batch (for bulk) |
| `--source` | | Scope to source string |
| `--limit` | 100 | Max recipes in batch |
| `--catalog-json` | required | Path to exported catalog |
| `--apply` | false | Actually write (default is dry-run) |
| `--yes` | false | Skip confirmation prompt |
| `--explain` | false | Show evidence |
| `--min-confidence` | | Override minimum confidence threshold |
| `--llm` | false | Enable LLM second pass |

Before writing, verifies every selected tag ID still exists in the target DB. Aborts on drift.

**`cookimport tag-recipes debug-signals`** — Print the signal pack the tagger sees for a recipe.

| Flag | Notes |
|------|-------|
| `--draft` | Path to draft JSON |
| `--db-url` + `--recipe-id` | Fetch from Postgres |


## Testing

19 tests in `tests/test_tagging.py`, all deterministic, running in ~0.06s:

- **Catalog:** load/roundtrip, lookup helpers.
- **Signals:** text normalization, dict-to-signal conversion.
- **Engine scoring:** per-fixture assertions (instant pot chicken, slow cooker beef stew, grilled salmon, quick pasta, sheet pan chicken, tofu stir fry, no-cook salad, dutch oven pot roast).
- **Policy enforcement:** single-category picks at most one tag.
- **Edge cases:** tags missing from catalog are skipped, evidence is always present, deterministic outputs are identical across runs.
- **Gold set regression:** 8 fixture recipes with labeled expected tags. Asserts overall precision >= 70%, single-category precision >= 80%, overall recall >= 50%.

To run: `pytest tests/test_tagging.py -v`


## How to Extend

**Add a new deterministic rule:** Edit `rules.py`, add an entry to `TAG_RULES` keyed by the tag's `key_norm`:

    "my-new-tag": {
        "min_score": 0.6,
        "patterns": [
            {"field": "ingredients", "regex": r"\bmy pattern\b", "weight": 0.7,
             "evidence": "matched X in ingredients"},
        ],
    }

Then add a fixture recipe + gold label in `tests/tagging_gold/` and a test case in `test_tagging.py`. Run the gold set regression test to verify you haven't broken precision.

**Add a new tag to the catalog (no code change needed):** Add the tag to the DB seed, re-export the catalog JSON. If no rule exists for the new `key_norm`, it simply won't get deterministic suggestions (but will be available to the LLM second pass when wired).

**Wire a real LLM provider:** Implement `_check_provider_configured()` and replace `_mock_llm_call()` in `llm_second_pass.py`. The validation logic (`_validate_llm_response`) is already implemented and enforces: tag exists in catalog, category matches, confidence 0..1, tag is in candidate shortlist.


## Key Decisions

- **Rules keyed by `key_norm`, not UUID** — forward-compatible with catalog changes. Rules for tags that don't exist in the loaded catalog are silently skipped.
- **Separate tagging CLI module** — `cookimport/tagging/cli.py` wired via `app.add_typer()` to keep the 2700-line main `cli.py` manageable.
- **`psycopg[binary]` as optional dependency** — added to `[db]` and `[dev]` extras in `pyproject.toml`. File-based tagging works without it.
- **Category table assumed `public.recipe_tag_categories`** — inferred from conventions; `db/schema.snapshot.sql` and `supabase/seeds/catalog.sql` were not in the repo. Verify on first DB run.
- **Protein patterns use weight tiers, not excludes** — a high-weight pattern for specific cuts ("beef chuck" = 0.7) and a low-weight fallback for generic mentions ("beef" = 0.3). This avoids the "beef broth excludes beef chuck" problem that exclude patterns caused.


## Known Gaps

- **Gold set is small** (8 recipes). Target 50 for meaningful regression protection.
- **No deterministic rules** for cuisine, dietary, flavor-profile, heat-source, vibe, season, holiday, or techniques. These are reserved for the LLM pass.
- **LLM provider not wired** — scaffolding exists but uses a mock that returns empty.
- **Category table name unverified** against production DB.
- **No DB integration tests** — DB code is covered by unit tests + manual acceptance.
