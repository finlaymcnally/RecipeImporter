# Replace Prompt Preview Multipliers With Structural Token Counts

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

After this change, `cf-debug preview-prompts` should estimate future token usage from the actual locally reconstructed prompt files and the actual planned shard shapes, not from opaque stage-wide multipliers. A user should be able to point preview at a deterministic or `vanilla` run and see a prediction that is grounded in the real prompt bodies, the deposited task files, and a structural best guess at the JSON outputs for recipe, knowledge, and line-role work.

## Surprises & Discoveries

- Observation: the existing predictive path is still fundamentally `chars * ratio`, even after the earlier predictive-only cleanup.
  Evidence: `build_prompt_preview_budget_summary(...)` currently calls `_build_predictive_stage_estimate(...)` for each stage and stores `predictive_stage_model`.

- Observation: the repo already contains structural fake-output builders for all three relevant pipeline families.
  Evidence: `cookimport/llm/fake_codex_farm_runner.py` has `_default_output(...)` branches for `recipe.correction.compact.v1`, `recipe.knowledge.compact.v1`, and `line-role.canonical.v1`.

- Observation: exact model-to-tokenizer mapping is incomplete for the repo's GPT-5 variant names.
  Evidence: `tiktoken.encoding_for_model("gpt-5")` works, but `gpt-5.2` and `gpt-5.3-codex-spark` raise `KeyError`; `o200k_base` is the practical fallback.

- Observation: `tiktoken` can throw a regex backtracking error on pathological mega-strings when asked to encode the full payload in one shot.
  Evidence: a direct `1_600_000`-char fixture raised `ValueError('Regex error while tokenizing: ... Max stack size exceeded for backtracking')` until preview tokenization was chunked for very large text.

- Observation: prompt preview was still carrying a second source of truth for recipe and knowledge inputs by reusing live `raw/llm/.../in/*.json` payloads when they happened to exist under the processed run.
  Evidence: `_build_recipe_preview_rows(...)` and `_build_knowledge_preview_rows(...)` both had early-return branches that copied staged input files instead of rebuilding from the deterministic/vanilla processed artifact.

## Decision Log

- Decision: predictive preview should count exact local prompt tokens rather than multiplying chars by empirical stage factors.
  Rationale: the user wants prediction grounded in the actual reconstructed jobs, not hidden benchmark-derived inflation constants.
  Date/Author: 2026-03-17 / Codex

- Decision: predictive output tokens should be estimated from structural JSON payloads built from the planned shard inputs.
  Rationale: output cannot be known exactly before a run, but a transparent schema-shaped guess is better than folding output assumptions into a stage-wide multiplier.
  Date/Author: 2026-03-17 / Codex

- Decision: use `tiktoken` with `o200k_base` fallback for repo GPT-5 style model names.
  Rationale: it is already installed locally and provides a stable tokenizer for preview without changing live execution.
  Date/Author: 2026-03-17 / Codex

## Outcomes & Retrospective

`prompt_preview` now predicts tokens from structural local artifacts instead of benchmark-wide multipliers. `cookimport/llm/prompt_budget.py` tokenizes each reconstructed wrapper prompt and deposited task file with `tiktoken`, builds schema-shaped best-guess JSON outputs via `fake_codex_farm_runner.build_structural_pipeline_output(...)`, and sums those per-row counts into stage and total estimates. Cached input is explicitly reported as zero for predictive runs.

`cookimport/llm/prompt_preview.py` no longer loads any repo-owned stage model table. The summary format now reports `structural_prompt_tokenization` when the planned rows are fully reconstructable and `unavailable` otherwise. Markdown and JSON summaries explain the new basis directly. The preview builder also now has one recipe/knowledge source of truth: rebuild from the predictive-safe processed artifact, not from opportunistic live input leftovers under `raw/llm/.../in/`.

## Context and Orientation

`cookimport/llm/prompt_preview.py` rebuilds prompt rows from a deterministic or `vanilla` processed run. Each prompt row already contains the rendered wrapper prompt (`rendered_prompt_text`), the deposited task file body (`task_prompt_text`), the pipeline ID, the model, and the structured request input payload. `cookimport/llm/prompt_budget.py` currently turns those prompt rows into a budget summary by multiplying `estimated_request_chars_total` by per-stage constants. That is the seam to replace.

The important distinction for this change is:

- input tokens should come from tokenizing the reconstructed prompt wrapper plus the deposited task file body for path-mode prompts
- output tokens should come from tokenizing a schema-shaped best-guess JSON response built from the same local shard input
- prompt preview remains predictive-only and must not read completed-run Codex telemetry

## Idempotence and Recovery

These edits are safe to rerun. If `tiktoken` cannot map a specific configured model name, preview should fall back to `o200k_base` instead of failing. If a row cannot produce a structural output estimate, preview should leave that stage unavailable rather than silently restoring multiplier math.
