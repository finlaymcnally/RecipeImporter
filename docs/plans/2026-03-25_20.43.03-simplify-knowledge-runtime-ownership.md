## Context and Orientation

The knowledge stage lives under `cookimport/llm/knowledge_stage/`. In this package, `runtime.py` is the stage-level coordinator that plans shard inputs, writes the top-level runtime artifacts, launches worker assignments, and hands validated results to promotion/reporting. A worker assignment is one long-lived Codex workspace session that owns one or more planned knowledge shards. A shard is the review-sized bundle controlled by `knowledge_prompt_target_count`; it is now the live answer unit. The worker-facing local contract is phase-first: Pass 1 edits a row ledger with `knowledge|other` decisions, `check-phase` validates and freezes accepted rows, then `install-phase` advances the same shard into Pass 2 for `group_id` and `topic_label`.

Today, the live phase-first implementation lives directly inside `cookimport/llm/knowledge_stage/runtime.py` in `_run_phase_knowledge_worker_assignment_v1`. At the same time, `cookimport/llm/knowledge_stage/workspace_run.py` still contains older taskized workspace helpers such as `_prepare_taskized_workspace_assignment`, `_run_taskized_workspace_session`, and `_finalize_taskized_workspace_assignment`. The private package README in `cookimport/llm/knowledge_stage/README.md` still teaches that older packet/task ownership model. The cleanup goal is to make the file/module boundaries honest again: `runtime.py` should stay as stage orchestration, and `workspace_run.py` should own the live workspace-worker execution helper that `runtime.py` calls.

## Plan of Work

First, copy the live phase-first helper set out of `cookimport/llm/knowledge_stage/runtime.py` into `cookimport/llm/knowledge_stage/workspace_run.py`. The moved set includes the small local helpers that only exist to support the live path: the current-phase row builder, the phase-surface writer, the worker output evaluator, the worker-assignment runner itself, and the local `events.jsonl` renderer if the module does not already have the same helper. Keep function names stable so the behavior does not change; the point is to relocate ownership, not redesign the worker loop.

Second, simplify `cookimport/llm/knowledge_stage/runtime.py`. Remove the inlined phase-first helper definitions and replace the executor submission site with a call into `workspace_run.py`. Drop imports from `cookimport/llm/knowledge_phase_workspace_tools.py` that are no longer needed by `runtime.py`. Keep the public behavior of `run_codex_farm_nonrecipe_knowledge_review(...)` and `_run_direct_knowledge_workers_v1(...)` unchanged.

Third, update the short package docs. `cookimport/llm/knowledge_stage/README.md` should say plainly that `runtime.py` is stage orchestration, `workspace_run.py` owns the live phase-first workspace-worker loop, `recovery.py` owns retries/watchdogs, and the live contract is shard-ledger and phase-first rather than packet/task taskized. If any broader LLM docs need one sentence to reflect the cleaned ownership boundary, update them briefly.
