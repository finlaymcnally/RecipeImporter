# Fix Oracle Benchmark Browser Upload Reliability

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

Today the benchmark-side Oracle browser upload path can fail in three different ways that look similar from the operator side but require different fixes. One run can append stale prompt text from an earlier upload and send a malformed request. Another can return a polished answer that is not actually grounded in the attached bundle. A third can lose the browser session mid-run and leave the operator with no visible ChatGPT conversation and no actionable reattach command. After this change, a human should be able to start a benchmark upload and trust three things locally: the outgoing prompt is isolated to the intended bundle, the upload metadata records enough Oracle session state to recover from disconnects, and the saved log can be classified as `succeeded`, `invalid_grounding`, `reattachable`, or `failed` without reading the full transcript by hand.

The most important user-visible outcome is operational reliability. Starting `cookimport bench ...` should no longer silently reuse dirty browser composer text, no longer hide the Oracle session id needed for recovery, and no longer treat a completed-but-contradictory answer as a successful review. The implementation should favor deterministic local checks and narrow test seams first. A live browser smoke can exist as a final optional confidence check, but the core proof must come from local tests and log-audit fixtures so the fix does not require spending more model tokens.

## Surprises & Discoveries

- Observation: the prompt-stacking failure is not coming from `cookimport`'s prompt builder.  
  Evidence: `cookimport/bench/oracle_upload.py::build_oracle_benchmark_prompt(...)` emits a single six-line benchmark review prompt, while the malformed run contained several older benchmark requests concatenated ahead of the final target.

- Observation: the installed Oracle browser code can append stale composer content.  
  Evidence: in `/home/mcnal/.nvm/versions/node/v20.19.6/lib/node_modules/@steipete/oracle/dist/src/browser/actions/promptComposer.js`, `submitPrompt(...)` focuses the input and calls `input.insertText({ text: prompt })` without first calling `clearPromptComposer(...)`.

- Observation: the disconnect failure is partly a wrapper failure, not just an Oracle failure.  
  Evidence: Oracle logs `Chrome disconnected before completion; keeping session running for reattach.`, but `cookimport` only saves PID, command, and browser profile path in `oracle_upload.json`; it does not record the Oracle session id or `oracle session <id>` command needed for recovery.

- Observation: the local Oracle version is behind a release with directly relevant browser fixes.  
  Evidence: the installed package is `@steipete/oracle 0.8.6`, while the upstream `0.9.0` release notes from 2026-03-08 explicitly mention keeping manual-login sessions reattachable after Chrome disconnects and preserving browser/profile state through interruption cases.

- Observation: a finished Oracle answer is not a sufficient success signal.  
  Evidence: the `2026-03-18_21.10.12` dinnerfor2cutdown run produced a complete three-section answer, but the answer contradicted the attached bundle's own verified `run_count`, `pair_count`, and benchmark root and therefore was operationally unusable.

- Observation: the heaviest failing upload was far larger than the others.  
  Evidence: the `saltfatacidheatcutdown` run staged `62` files, including `60` payload shards, and Oracle estimated about `11,855,826` tokens before the browser disconnect.

## Context and Orientation

The benchmark-side Oracle upload path in this repository lives in `cookimport/bench/oracle_upload.py`. That module resolves an `upload_bundle_v1`, builds the benchmark-review prompt, stages oversized files into upload-time shards, and either runs Oracle synchronously or launches it in the background. The user-facing benchmark CLI wiring that prints the short summary after a benchmark run lives in `cookimport/cli.py`, specifically the helper that reports the Oracle PID, browser profile, and `oracle_upload.log` path.

In this repository, an "Oracle upload run" means one detached background browser process plus its launch directory under `upload_bundle_v1/.oracle_upload_runs/<timestamp>/`. That directory already stores `oracle_upload.log`, `oracle_upload.json`, and any temporary sharded upload files. It does not yet store a higher-level verdict such as `succeeded` versus `invalid_grounding`, and it does not currently guarantee that the Oracle session id is recorded.

The external dependency is the globally installed Oracle CLI at `/home/mcnal/.local/bin/oracle`, which currently resolves to `@steipete/oracle 0.8.6` under `/home/mcnal/.nvm/versions/node/v20.19.6/lib/node_modules/@steipete/oracle`. The relevant browser logic for the observed failures lives in these installed files:

- `.../@steipete/oracle/dist/src/browser/actions/promptComposer.js`
- `.../@steipete/oracle/dist/src/browser/index.js`
- `.../@steipete/oracle/dist/src/cli/sessionRunner.js`
- `.../@steipete/oracle/dist/bin/oracle-cli.js`

The key local benchmark evidence lives in:

- `docs/07-bench/2026-03-18_21.53.58-oracle-browser-upload-failure-report.md`
- `docs/understandings/2026-03-18_21.47.16-oracle-browser-response-mixed-current-bundle-with-stale-bundle-identity.md`
- `docs/understandings/2026-03-18_21.50.07-vanilla-only-oracle-review-was-mostly-grounded-even-with-stacked-prompts.md`
- `docs/understandings/2026-03-18_21.58.23-oracle-benchmark-browser-failures-split-between-composer-hygiene-session-handoff-and-outdated-cli.md`

The existing tests already cover basic command assembly and sharded upload staging in `tests/bench/test_benchmark_oracle_upload.py`. Those tests do not yet model stale composer text, missing session-id capture, or grounding-audit behavior. That file should remain the primary test seam for this project.

Two terms need explicit definition here.

A "composer hygiene" bug means the browser automation writes a new prompt into a ChatGPT input box that still contains draft text from a previous run. In this repository that manifested as several old benchmark prompts being prepended to the new one.

A "grounding audit" means a deterministic local pass that reads the saved Oracle answer from `oracle_upload.log` and compares it against known local truths from `upload_bundle_overview.md` and `upload_bundle_index.json`. The audit is not trying to prove every sentence. It is trying to catch clearly unsafe contradictions, such as Oracle naming the wrong benchmark root or claiming `pair_count = 0` when the bundle verifies `pair_count = 1`.

## Plan of Work

The work should happen in four milestones.

The first milestone adds red tests and log fixtures for the observed failures. Extend `tests/bench/test_benchmark_oracle_upload.py` with three new groups of tests. The first group should cover answer auditing by feeding saved Oracle log snippets into a new parser or classifier function inside `cookimport/bench/oracle_upload.py`. One fixture should model the valid vanilla-only single-profile answer, one should model the invalid `21.10.12` answer that cited the wrong bundle root, and one should model the hard browser-disconnect log with no `Answer:` block. The second group should cover background-launch metadata, asserting that `oracle_upload.json` can carry Oracle version, session id, reattach command, and final status fields. The third group should cover browser-command defaults, asserting that benchmark uploads pass the chosen chat URL / reattach flags and any minimum-version or profile-serialization settings decided below.

The second milestone fixes the dependency seam. Upgrade the installed Oracle CLI to a release with the disconnect fixes already published upstream, preferably `0.9.0` or later, and record the exact version in the benchmark metadata. Then patch Oracle's browser composer path so each submission either starts from a known-empty composer or fails fast before text insertion. The safest implementation is inside Oracle itself, because `cookimport` cannot control the browser DOM once Oracle takes over. In practice that means editing the installed Oracle browser action module so `submitPrompt(...)` clears the composer, verifies emptiness, then inserts text. If upgrading to `0.9.0` changes file layout, update the exact file path in this plan before implementation continues. If the upgraded version already fixes composer reuse, keep the local patch out and only add regression coverage that proves the behavior.

The third milestone fixes recoverability in `cookimport`. Change `cookimport/bench/oracle_upload.py` so benchmark launches capture Oracle session identity as first-class metadata. There are two acceptable ways to do this, and the implementation should choose the more reliable one after a short spike. The preferred path is to patch or configure Oracle so it prints the session id and `oracle session <id>` command into stdout very early, then parse and persist those lines into `oracle_upload.json` and the short benchmark wrap-up text. The fallback path is to have `cookimport` query Oracle's session store immediately after launch and resolve the newest session created by this command, then persist that id. Once the session id is available, extend the launch summary in `cookimport/cli.py` so a disconnecting browser run tells the human exactly how to reattach instead of only pointing at `oracle_upload.log`.

The fourth milestone adds a trust audit. Implement a small deterministic classifier in `cookimport/bench/oracle_upload.py` that reads the saved log and returns one of at least four statuses: `succeeded`, `invalid_grounding`, `reattachable`, or `failed`. `succeeded` requires a completed `Answer:` block and no contradiction with the local bundle root or obvious local topline facts. `invalid_grounding` covers the second failure class where Oracle finished but named the wrong benchmark root or explicitly claimed it could not verify the real one. `reattachable` covers logs with browser disconnect or assistant-timeout language plus a known Oracle session id. `failed` covers everything else with no recovery path. Persist that verdict in `oracle_upload.json` and optionally in a sibling status file so humans and future tooling can inspect it without re-reading the full log. The benchmark wrap-up and any later follow-up commands should use this status instead of assuming "process started" means "review succeeded."

After those four milestones, add one small operator-quality improvement: support an explicit benchmark ChatGPT target URL in the wrapper, either from a repo-owned environment variable such as `COOKIMPORT_ORACLE_CHATGPT_URL` or from a dedicated wrapper constant. The goal is not to make a productized config system. The goal is to let benchmark uploads use a dedicated ChatGPT project/folder so browser automation does not share a dirty workspace with unrelated manual sessions.

## Milestones

### Milestone 1: Make the failures reproducible in local tests

At the end of this milestone the repository will have red tests that describe the real operator-level failures instead of relying on memory or chat transcripts. A novice should be able to run one focused test file and see the exact three categories fail before production edits land.

The work lives in `tests/bench/test_benchmark_oracle_upload.py`, plus any small fixture text files if inline strings become unreadable. The tests should not require a live Oracle install or token spend. They should use frozen log text and monkeypatched launch metadata.

Acceptance for this milestone is a red test run with narrow failures for answer audit and session metadata behavior.

### Milestone 2: Stop stale prompt text at the source

At the end of this milestone Oracle browser mode will no longer append new benchmark prompts into leftover composer drafts. A novice should be able to inspect the patched Oracle browser action code and see an explicit empty-composer step before text insertion.

The work primarily lives in the installed Oracle package under `.../@steipete/oracle/dist/src/browser/actions/promptComposer.js`, and possibly in the upgraded package version if the file layout changed. `cookimport` should also record the detected Oracle version in `oracle_upload.json` so future debugging can tell which Oracle build produced which run.

Acceptance for this milestone is a green regression test plus a manual no-token smoke if needed: one small browser launch against a dummy prompt should show the patched path starts from a cleared composer. If a live smoke is used, it must be explicitly approved before running.

### Milestone 3: Make disconnects recoverable from the benchmark wrapper

At the end of this milestone a browser disconnect will no longer leave the operator with only a dead PID and a vague log line. The launch metadata will include Oracle session id and a concrete `oracle session <id>` reattach command, and the benchmark summary text will surface it.

The work lives in `cookimport/bench/oracle_upload.py`, `cookimport/cli.py`, and possibly a small Oracle-side patch if Oracle does not already print the session id early enough for the wrapper to capture.

Acceptance for this milestone is a green local test that feeds a disconnect log plus session metadata into the wrapper and asserts the saved metadata and printed summary contain the reattach command.

### Milestone 4: Refuse to trust contradictory answers

At the end of this milestone a completed Oracle answer that contradicts the local bundle root or obvious local counts will no longer be treated as success. The wrapper will save a machine-readable verdict and a short reason.

The work lives in `cookimport/bench/oracle_upload.py` and `tests/bench/test_benchmark_oracle_upload.py`. Keep the audit conservative: catch only high-confidence contradictions that are easy to prove from local files.

Acceptance for this milestone is a green local test suite where:

- the valid vanilla-only answer classifies as `succeeded`,
- the dinnerfor2 bundle-identity-mixup answer classifies as `invalid_grounding`,
- the Chrome-disconnect log classifies as `reattachable` or `failed` depending on whether session id capture is present.

## Validation and Acceptance

Acceptance is behavioral.

This project is accepted when all of the following are true.

1. Local test coverage proves the three March 18 failure classes are modeled and guarded.

2. The wrapper persists enough metadata for recovery. At minimum `oracle_upload.json` must contain:

   - Oracle version
   - bundle dir
   - source root
   - mode
   - model
   - PID
   - log path
   - status verdict
   - session id when known
   - reattach command when known

3. The saved verdict logic behaves like this on fixture data:

   - valid single-profile answer => `succeeded`
   - wrong-root / contradictory answer => `invalid_grounding`
   - disconnect with session id => `reattachable`
   - disconnect without session id or no answer/no recovery => `failed`

4. The benchmark launch summary is more actionable than today. A human should be able to start an upload and know where to look next without manually discovering Oracle session commands.

5. The Oracle dependency level is explicit. Either the wrapper records a minimum required version in docs and metadata, or the repo documents the local patch points needed when running against `0.8.6`.

The critical fail-before / pass-after story that should be recorded during implementation is:

- before the change, a malformed saved answer can still look "successful" because the wrapper only checks process completion
- after the change, the same saved answer classifies as `invalid_grounding`
- before the change, a disconnecting run leaves no reattach command in benchmark metadata
- after the change, the same disconnect classifies as `reattachable` and exposes `oracle session <id>`
- before the Oracle-side composer patch, stale browser drafts can be appended to the next prompt
- after the patch, the browser submission path begins from an empty composer or fails fast

## Idempotence and Recovery

The repository-side steps are safe to repeat. The focused tests are intentionally narrow and should be rerun after each milestone.

The Oracle installation step needs one extra rule because it touches global state. Before upgrading or patching the installed Oracle tree, record the current version and exact install path in this plan's `Progress` or `Decision Log`. If an upgrade breaks CLI behavior, restore the prior global version or apply the local patch against the older install and rerun the same focused tests. Do not leave the plan ambiguous about which Oracle build the wrapper expects.

The new wrapper verdict logic must be conservative. If the audit is unsure, it should classify as `failed` or `invalid_grounding`, not silently promote the run to `succeeded`.
