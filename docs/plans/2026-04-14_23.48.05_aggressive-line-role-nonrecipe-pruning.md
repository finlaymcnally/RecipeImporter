## Purpose / Big Picture

The current pipeline is closest to the user’s intended shape when line-role acts like the strict first-pass bouncer and the knowledge stage acts like the stricter second-pass reviewer. Right now, too much outside-recipe prose survives line-role as `NONRECIPE_CANDIDATE`, which means the knowledge stage is asked to clean up a pool that is already too noisy. After this change, line-role should throw out more obvious memoir, intro framing, praise blurb, rhetorical setup, and broad coaching prose by labeling it `NONRECIPE_EXCLUDE`, while still preserving real recipe structure and still forwarding genuinely reusable standalone cooking knowledge as `NONRECIPE_CANDIDATE`.

The user-visible result should be a smaller, cleaner outside-recipe candidate pool. The easiest proof will be in the prompt contract, focused line-role tests, and saved line-role artifacts: more rows that are clearly fluff should stop at `NONRECIPE_EXCLUDE`, and rows that still read like portable cooking knowledge should continue on to the knowledge stage.

## Context and Orientation

This repository has a two-stage outside-recipe flow. The first stage is line-role, which labels each source row as recipe structure, recipe-local note-like material, `NONRECIPE_CANDIDATE`, or `NONRECIPE_EXCLUDE`. The second stage is knowledge review, which receives only the surviving outside-recipe candidate pool and decides what is actually worth keeping for downstream review surfaces.

For this plan, the key distinction is simple. `NONRECIPE_CANDIDATE` means “outside-recipe material that should be shown to the knowledge step later.” `NONRECIPE_EXCLUDE` means “outside-recipe material that should stop here and never reach the knowledge step.” The user’s request is to make line-role more aggressive about the second category so the knowledge step has a cleaner pool.

The current shared line-role label contract lives in `llm_pipelines/prompts/line-role.shared-contract.v1.md`. That file is rendered by `cookimport/llm/canonical_line_role_prompt.py` and then reused by the line-role planning/runtime surfaces in `cookimport/parsing/canonical_line_roles/`. The repo already has tests that assert specific contract wording and example lines in `tests/parsing/test_canonical_line_roles_prompting.py`.

The current repo-owned policy seam for outside-recipe exclusion lives in `cookimport/parsing/canonical_line_roles/policy.py`. The relevant functions are `_outside_recipe_exclude_allowed(...)`, which covers narrow obvious-exclusion families, `_normalize_prediction_metadata(...)`, which normalizes fallback labels, and `_codex_prediction_policy_rejection_reason(...)`, which enforces the live Codex label contract.

The existing line-role tests already provide the starting map for safe changes. `tests/parsing/test_canonical_line_roles_policy_tuning.py` covers deterministic tuning and policy edges. `tests/parsing/test_canonical_line_roles_prompting.py` asserts the rendered contract. `tests/parsing/test_canonical_line_roles_codex.py` covers mocked Codex responses and policy acceptance. These tests are the fastest safe way to tighten the boundary without running a live benchmark.

One additional context point matters: the repository now uses more granular source-row splitting. That means one old paragraph may now arrive as several smaller rows. Smaller rows are easier for the model to over-read as “teaching,” so this plan explicitly targets split prose fragments, short coachy lines, and isolated headings that do not actually deserve to survive into the knowledge pool.

## Plan of Work

The first implementation step is to tighten the shared line-role contract in `llm_pipelines/prompts/line-role.shared-contract.v1.md`. The revised contract should teach one simple boundary: outside recipe, the model should keep only portable, standalone cooking knowledge and should exclude rows that are mainly memoir, book framing, motivational coaching, rhetorical setup, or praise/manifesto prose. This needs to be taught with stronger contrast language and better examples, not with vague “use judgment” wording.

The second step is to make the rendered line-role prompt surfaces match that boundary everywhere they restate the contract. The current runtime consumes the shared contract through `cookimport/llm/canonical_line_role_prompt.py`, and the line-role runtime/watchdog surfaces in `cookimport/parsing/canonical_line_roles/` depend on that shared wording. The implementation should verify that no wrapper text weakens the stronger exclusion rule or quietly reintroduces “teacherly means candidate” drift.

The third step is to review `cookimport/parsing/canonical_line_roles/policy.py` and keep deterministic behavior narrow. The likely default is no major semantic expansion here. The implementation should only adjust policy code if there is a truly narrow, obvious, high-confidence exclusion family that belongs with existing repo-owned exclusions. This plan does not authorize adding a new deterministic classifier for memoir, inspiration, or educational tone.

The fourth step is to add or update contrast tests. The test set should include rows that should clearly move to `NONRECIPE_EXCLUDE`, such as memoir fragments, preface-style promises, rhetorical chapter-question headings without surrounding lesson prose, and short coaching fragments that do not stand alone as reference knowledge. The same test set must also protect the opposite edge: standalone mechanism lines, concise cooking rules, and lesson headings that do have nearby explanatory support should remain `NONRECIPE_CANDIDATE`.

The fifth step is to update the durable docs after implementation. The short note belongs in the parsing/LLM docs, explaining that line-role is now intentionally stricter about excluding outside-recipe framing prose so the knowledge stage sees a smaller candidate pool.

## Validation and Acceptance

The work is acceptable only if all of the following are true.

Focused line-role tests pass in the project `.venv`, and at least one test on each side of the boundary exists or is updated: one family proving stricter exclusion, and one family proving continued candidate routing for real knowledge.

No change widens deterministic semantic guessing beyond the repo’s current authority boundaries. If policy code changes at all, those changes stay narrow, obvious, and easy to explain.

No live `codex-exec` benchmark is run without explicit user approval. If approval is later granted and a benchmark is run, the plan must be updated with the exact command, the date, and the comparison outcome.

## Idempotence and Recovery

The contract and test edits in this plan are safe to apply incrementally. Prompt wording can be tightened, tested, and refined without touching saved benchmark artifacts or changing the downstream knowledge-stage contract.

If a prompt revision becomes too aggressive and starts excluding rows that should remain candidates, the safe rollback is to revert the last contract/example change while keeping the newly added contrast tests. Those tests should then be rewritten to match the corrected intended boundary rather than deleted.

If a proposed policy change turns out to rely on semantic guessing rather than an obvious exclusion family, the safe recovery is to drop the policy edit entirely and keep the change prompt-only. That outcome is still acceptable if the prompt and tests prove the intended boundary.
