# Persistent all-method benchmark dashboard spinner

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

PLANS.md is checked into the repo at `docs/PLANS.md`. This document must be maintained in accordance with that file.

## Purpose / Big Picture

Interactive all-method benchmark currently prints a large per-config output block (`Benchmark complete`, paths, recipe deltas) for each run. On multi-source sweeps this floods terminal scrollback and hides where the run currently is.

After this change, all-method runs keep one persistent spinner dashboard showing:

- source queue status (pending/running/done/failed),
- overall source/config counters,
- current config slug,
- current per-config task/progress line (including worker telemetry lines).

Per-config benchmark completion dumps are suppressed while the all-method sweep is active.

## Surprises & Discoveries

- Observation: `labelstudio_benchmark(...)` is used as a direct Python function by all-method loops, not only as a Typer command path.
  Evidence: `_run_all_method_benchmark(...)` calls `labelstudio_benchmark(...)` in-process.

- Observation: Nested `console.status(...)` calls plus per-run summary prints are the main source of all-method output flood.
  Evidence: Existing benchmark command always printed completion blocks after each config.

## Decision Log

- Decision: Keep `labelstudio_benchmark(...)` as the single benchmark primitive and add scoped internal overrides instead of cloning benchmark logic.
  Rationale: Avoids logic drift in evaluation/report-writing paths and keeps one source of truth for benchmark artifacts.
  Date/Author: 2026-02-24 / Codex

- Decision: Represent all-method progress as explicit dashboard state (`_AllMethodProgressDashboard`) rather than parsing free-form text.
  Rationale: Queue/progress rendering stays deterministic and easier to evolve than regex-based parsing of status strings.
  Date/Author: 2026-02-24 / Codex

- Decision: Pass worker telemetry payloads through unchanged to the outer spinner.
  Rationale: Preserves existing worker line rendering contract in `_run_with_progress_status(...)`.
  Date/Author: 2026-02-24 / Codex

## Outcomes & Retrospective
