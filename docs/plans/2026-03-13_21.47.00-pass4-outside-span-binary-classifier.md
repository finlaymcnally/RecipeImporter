## Purpose / Big Picture

After this change, pass4 will no longer be just a snippet harvester whose benchmark effect depends on evidence-quote matching. It will also act as an explicit binary classifier over pass4 chunk blocks, deciding whether each non-recipe block is reusable cooking knowledge or just other narrative/noise. The observable result is that codex-enabled pass4 runs will write a block-classification artifact, and canonical line-role/stage projection will be able to use that artifact directly when deciding outside-span `KNOWLEDGE` versus `OTHER`.

## Context and Orientation

Pass4 lives in `cookimport/llm/`. `codex_farm_knowledge_jobs.py` builds chunk-based jobs from `non_recipe_blocks`. `codex_farm_knowledge_models.py` defines the structured output model. `codex_farm_knowledge_writer.py` currently writes only `knowledge/<workbook_slug>/snippets.jsonl` and `knowledge.md`. The projection seam lives in `cookimport/labelstudio/canonical_line_projection.py`, which currently reads snippets and turns matching outside-span projected `OTHER` labels into `KNOWLEDGE`. Deterministic stage scoring also accepts `knowledge_snippets_path` in `cookimport/staging/stage_block_predictions.py`.

For this task, “outside span” means canonical lines that are not inside an active recipe span. “Knowledge” means reusable cooking guidance, technique, definitions, substitutions, or factual cooking references. “Other” means narrative, headers, front matter, blurbs, and non-reusable prose.

## Plan of Work

First extend the pass4 output contract so every pass4 output may include one ordered decision per `chunk.blocks[*]`, using a minimal shape such as block index plus binary category. Keep the existing `is_useful` and `snippets` fields so old call sites still work, and make the new field optional so older artifacts remain readable.

Next update the pass4 prompt and schema so the model is explicitly asked to classify each chunk block as `knowledge` or `other`, using only `chunk.blocks` as evidence. The prompt should say that snippets remain optional and useful for reusable knowledge summaries, but block decisions must be returned for every chunk block.

Then update `cookimport/llm/codex_farm_knowledge_writer.py` to emit a durable block-classification artifact under `knowledge/<workbook_slug>/`, and surface that path through the pass4 manifest/report payload. The artifact should be simple JSONL so downstream code can map block indices directly without quote matching.

After that, update `cookimport/labelstudio/canonical_line_projection.py` and `cookimport/staging/stage_block_predictions.py` so they load pass4 block classifications first. The new rule should only arbitrate outside-span `OTHER` versus `KNOWLEDGE`: classify selected outside-span lines as `KNOWLEDGE` when pass4 says the source block is knowledge, and downgrade outside-span `KNOWLEDGE` to `OTHER` when pass4 says the source block is other. If the new artifact is absent, preserve the current snippet-based fallback so historical runs still work.

Finally, add tests that prove all of this. At minimum:

1. Pass4 output parsing accepts block decisions.
2. Writer emits the new artifact.
3. Canonical projection upgrades and downgrades outside-span `OTHER`/`KNOWLEDGE` from pass4 block decisions.
4. Deterministic stage block prediction marks `KNOWLEDGE` from pass4 classifications and falls back safely when classifications are absent.

## Validation and Acceptance

Acceptance has three parts.

First, unit tests must prove that pass4 outputs can carry block decisions and that the writer persists them.

Second, projection/stage tests must show behavior, not just parsing. A test should create outside-span lines with initial `OTHER` and `KNOWLEDGE` labels, apply a pass4 block-classification artifact, and observe:

- `OTHER -> KNOWLEDGE` when pass4 says `knowledge`
- `KNOWLEDGE -> OTHER` when pass4 says `other`
- recipe-local labels remain untouched

Third, the on-disk pass4 manifest payload should include the new artifact path so benchmark/debug tooling can find it later.

## Idempotence and Recovery

The code changes are additive and safe to rerun. The new output field must stay optional so older pass4 artifacts remain readable. The new block-classification artifact is a derived file; if its shape changes during iteration, delete only that artifact in a local run and rerun pass4 generation.
