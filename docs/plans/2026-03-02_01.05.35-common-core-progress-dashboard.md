# Unify CLI Progress Around a Common Dashboard Core

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

`docs/PLANS.md` is checked into this repository and this document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the same dashboard-style progress contract will drive stage, single benchmark/import, and all-method benchmark workflows. Operators should see one visual model: an overall counter, active item label, queue or worker lines, and a task line that updates in both live and plain modes.

This matters because today each flow owns a separate tracker implementation and small differences confuse operators during long runs. Success is verified when running representative CLI flows and observing matching dashboard semantics plus preserved operational detail (ETA where already shown, worker activity, and existing stage summaries).

## Surprises & Discoveries

- Observation: the code path includes three distinct tracker systems, not one, so a simple swap would leave a forked behavior surface.
  Evidence: `_run_with_progress_status(...)` handles callback messages for import/benchmark flows, while `stage(...)` has a separate Rich `Live(...)` worker dashboard and queue thread in `cookimport/cli.py`.

- Observation: the all-method path already behaves as a stateful dashboard model and is the least risky extraction seed.
  Evidence: `_AllMethodProgressDashboard` maintains lock-protected rows, status markers, active config state, and task text in `cookimport/cli.py`.

- Observation: callback messaging is part of the contract and includes worker side-channel payloads.
  Evidence: `_run_with_progress_status(...)` special-cases `parse_worker_activity(...)` payloads from `cookimport/core/progress_messages.py`.

## Decision Log

- Decision: keep the external `progress_callback(message: str)` contract and layer the shared core behind adapters.
  Rationale: this minimizes blast radius and enables incremental migration flow-by-flow.
  Date/Author: 2026-03-02 / Codex

- Decision: treat the all-method dashboard visual contract as canonical for rendering semantics.
  Rationale: it is already stable in benchmark flows and matches operator expectations for queue/task/worker layout.
  Date/Author: 2026-03-02 / Codex

- Decision: preserve ETA/elapsed handling in `_run_with_progress_status(...)` during this refactor.
  Rationale: removing this logic risks regressions in user-visible timing signals while not part of rendering unification itself.
  Date/Author: 2026-03-02 / Codex

- Decision: keep parsing compatibility for worker activity messages (`activity` / `reset`) exactly as-is.
  Rationale: multiple pathways currently emit and consume those messages, so changing shape introduces high regression risk.
  Date/Author: 2026-03-02 / Codex

- Decision: keep `_run_with_progress_status(...)` responsible for runtime controller concerns (mode selection, ticker cadence, dedupe, telemetry writes).
  Rationale: that function owns behavior not just rendering, and we can safely migrate state/render into the shared core underneath it.
  Date/Author: 2026-03-02 / Codex

## Context and Orientation

The current progress surface for long-running CLI runs is concentrated in `cookimport/cli.py`.

`_AllMethodProgressDashboard` renders queue-based snapshots used by all-method benchmark flows.

`_run_with_progress_status(...)` consumes callback strings and worker activity payloads, rendering either live spinner output or plain deduplicated text plus optional timing suffixes.

`stage(...)` uses a separate `WorkerDashboard` model and a Rich `Live(...)` board for worker/job lifecycle updates plus merge-phase status text.

A “dashboard snapshot” is one multi-line status string that fully describes run state.

For this plan:

- A “core model” is a shared, thread-safe object storing normalized progress state.
- A “renderer” is the pure formatting layer that turns that state into the snapshot string.
- An “adapter” translates each flow’s native messages into core model updates.

## Plan of Work

Milestone 1: Extract shared core and renderer.

Create `cookimport/core/progress_dashboard.py` with dataclasses and rendering code for overall progress counters, current label, queue rows, optional task line, and worker lines. Keep formatting logic generic and flow-independent. The module should include thread-safe state updates and snapshot rendering. Do not encode benchmark or stage scheduling semantics here.

Milestone 2: Migrate all-method dashboard onto the core.

Refactor `_AllMethodProgressDashboard` to delegate state/formatting to the new core or replace it with a thin adapter that does. Preserve current rendering contracts that tests and operators rely on, including status markers, truncation, and active-config worker lines.

Milestone 3: Migrate `_run_with_progress_status(...)` to core snapshots.

Introduce an adapter inside this function to map callback strings and parsed worker-activity markers to core updates. Continue to support legacy plain text callback behavior. Keep live/plain mode selection, dedupe, ticker timing, and ETA/elapsed suffix logic in place while delegating dashboard assembly to shared render primitives.

Milestone 4: Migrate stage dashboard to the same core.

Replace `WorkerDashboard` string assembly with adapter-backed updates into the shared core. Keep stage-specific semantics (worker lifecycle states, pending/completed counts, merge messages, and run summary lines) while unifying rendering format and output flow.

Milestone 5: Parity verification, docs, and cleanup.

Add/adjust tests that lock in all-method output parity, `_run_with_progress_status` behavior, and stage snapshot consistency across plain/live modes. Remove obsolete duplicate render helpers only after parity tests pass. Then update docs with the shared-core contract.

## Idempotence and Recovery

Migration should proceed in small additive steps with compatibility wrappers. If one milestone regresses behavior, revert only that milestone’s adapter wiring and keep the shared-core module in place for re-run.

Because this is implementation behavior-only, rerunning the milestones is safe after fixes and does not require data migration.

## Interfaces and Dependencies

Implement the shared core in `cookimport/core/progress_dashboard.py` with stable interfaces, for example:

    @dataclass
    class ProgressQueueRow:
        name: str
        status: str
        completed: int
        total: int
        ok: int
        fail: int

    @dataclass
    class ProgressSnapshot:
        overall_done: int
        overall_total: int
        overall_label: str
        current_label: str
        task_message: str
        queue_rows: list[ProgressQueueRow]
        worker_lines: list[str]

    class ProgressDashboardCore:
        def set_overall(self, done: int, total: int, label: str = "task") -> None: ...
        def set_current(self, label: str) -> None: ...
        def set_task(self, message: str) -> None: ...
        def set_queue_rows(self, rows: list[ProgressQueueRow]) -> None: ...
        def set_worker_lines(self, rows: list[str]) -> None: ...
        def render(self) -> str: ...

    class ProgressCallbackAdapter:
        def ingest_callback_message(self, message: str) -> None: ...
        def snapshot_text(self) -> str: ...

`ProgressCallbackAdapter.ingest_callback_message(...)` must preserve existing parsing/compatibility for worker payloads and plain text callbacks.

Use only standard library and existing Rich usage already present in the repository.
