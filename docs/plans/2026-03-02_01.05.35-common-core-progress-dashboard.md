# Unify CLI Progress Around a Common Dashboard Core

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

`docs/PLANS.md` is checked into this repository and this document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the operator will see one consistent dashboard-style progress experience across long-running flows instead of each flow inventing its own tracker style. The all-method benchmark tracker is the reference visual style and behavior. Stage import, single benchmark import/eval, and all-method benchmark should all be rendered by one shared core with flow-specific adapters.

The user-visible success signal is simple: run `C3imp`, choose different workflows, and see the same top-level progress layout (overall counter, current item, queue/worker detail, task line) in both live spinner mode and plain-progress mode.

## Surprises & Discoveries

- Observation: the code currently has two major tracker systems, not one.
  Evidence: `_run_with_progress_status(...)` renders callback status for benchmark/import paths, while `stage(...)` has a separate `Live(...)` dashboard and queue thread (`cookimport/cli.py`).

- Observation: all-method already behaves like a stateful dashboard model and is the best extraction seed.
  Evidence: `_AllMethodProgressDashboard` already tracks rows, queue markers, active configs, and task messages with a thread lock (`cookimport/cli.py`).

- Observation: callback contracts are message-string based, with special control payloads for worker activity.
  Evidence: `_run_with_progress_status(...)` parses plain text, counters, and `parse_worker_activity(...)` markers (`cookimport/cli.py`, `cookimport/core/progress_messages.py`).

## Decision Log

- Decision: preserve the existing `progress_callback(message: str)` external contract during migration, and add shared-core adapters behind it.
  Rationale: this minimizes blast radius and allows incremental migration of callsites without a flag day.

- Decision: treat the all-method dashboard layout as the canonical visual contract for the shared core.

## Context and Orientation

The current CLI progress surface is concentrated in `cookimport/cli.py`.

`_AllMethodProgressDashboard` is a thread-safe state object that renders a multi-line dashboard snapshot string. It has the concepts we want everywhere: overall counters, current source/config focus, queue rows, and a task line.

`_run_with_progress_status(...)` is the generic wrapper used by benchmark/import paths. It currently stores local state (latest message, counters, worker activity), renders either live spinner or plain mode, and appends ETA/elapsed metadata.

`stage(...)` uses a separate rich `Live(...)` panel (`WorkerDashboard`) with its own worker state cache and formatting. That path is structurally similar to all-method dashboards but not implemented through the same model.

A "dashboard snapshot" in this plan means one multi-line status string representing the whole run state. A "core model" means one shared, thread-safe state container that any workflow can update and any renderer can stringify. "Adapter" means thin workflow-specific glue that translates existing events into core-model updates.

## Plan of Work

### Milestone 1: Extract shared dashboard core model and renderer

Create a new module under `cookimport/core/` (for example `cookimport/core/progress_dashboard.py`) that holds shared dataclasses and rendering logic. The core model must support: overall progress counters, an optional current item summary, queue rows with status markers, an optional task line, and optional worker lines. Keep model updates thread-safe with an internal lock, since existing flows already update from multiple threads.

Move only generic rendering behavior into this module. Do not move workflow-specific scheduler logic, benchmark semantics, or stage job planning here.

At the end of this milestone, the module exists with tests and can render the same shape currently produced by `_AllMethodProgressDashboard.render()`.

### Milestone 2: Re-home all-method dashboard onto shared core

Refactor `_AllMethodProgressDashboard` so it becomes either a thin wrapper over the shared core model or is replaced by an adapter class in `cookimport/cli.py` that delegates to the core. Preserve all existing all-method output semantics that current tests assert (queue markers, current config text, active config worker lines, truncation behavior, and task prefixing).

At the end of this milestone, all-method behavior is unchanged from the user perspective, but the implementation uses shared core primitives.

### Milestone 3: Migrate `_run_with_progress_status(...)` to shared core state

Update `_run_with_progress_status(...)` so it can maintain and render a shared core snapshot instead of only a free-form latest string. Keep support for raw string updates and `parse_worker_activity(...)` control messages. Preserve existing plain/live mode toggles (`COOKIMPORT_PLAIN_PROGRESS`, agent defaults), dedupe behavior, and ETA/elapsed suffix formatting.

Add a compatibility path: if no structured/core updates are present, existing simple message behavior still works.

At the end of this milestone, benchmark/import paths render through shared core state and still satisfy current tests for elapsed/ETA/worker rendering.

### Milestone 4: Migrate stage worker dashboard to shared core

Replace stage-specific `WorkerDashboard` string assembly with a stage adapter that updates the shared core model. Keep stage semantics intact: per-worker updates, completed/pending counts, and run summaries. Do not remove processing timeseries emission; wire it to shared-core snapshots where useful.

At the end of this milestone, stage and benchmark trackers come from the same renderer family, with only adapter differences.

### Milestone 5: Final parity pass, docs, and cleanup

Run targeted tests, then full relevant suites. Remove obsolete rendering helpers only after replacements are proven. Update docs to describe the new single-core progress architecture and operator expectations.

At the end of this milestone, there is one documented common-core progress design with stable behavior across workflows.

## Validation and Acceptance

Acceptance is behavioral, not only structural.

Run `C3imp` and verify that long-running workflows produce the same dashboard family:

- Interactive all-method benchmark keeps the current queue-centric dashboard format.
- Single benchmark/import status updates show the same top-level dashboard structure instead of ad-hoc message streams.
- Stage run worker statuses are rendered through the same core model, with worker lines and task line in the same style.
- In agent/plain mode, output remains change-only and deduped.
- In live mode, spinner updates remain readable and no frame spam is introduced.

Automated acceptance requires tests that prove fail-before/pass-after for at least one new shared-core behavior, and regression tests that preserve existing all-method dashboard contracts.

## Idempotence and Recovery

The migration is designed to be additive first and subtractive later. Keep compatibility wrappers until equivalent tests pass. If any milestone regresses behavior, revert only that milestone’s adapter wiring and keep the shared core module intact for retry.

All edits are safe to re-run because they are code/documentation changes only; no data migration or destructive operation is involved.

## Interfaces and Dependencies

Define the shared core with explicit interfaces in `cookimport/core/progress_dashboard.py`.

At minimum, the end state should expose a model and renderer with stable names and call sites, for example:

    @dataclass
    class ProgressQueueRow:
        name: str
        status: str
        completed: int
        total: int
        ok: int
        fail: int

    @dataclass
    class ProgressSnapshot:
        overall_done: int
        overall_total: int
        overall_label: str
        current_label: str
        task_message: str
        queue_rows: list[ProgressQueueRow]
        worker_lines: list[str]

    class ProgressDashboardCore:
        def set_overall(self, done: int, total: int, label: str = "task") -> None: ...
        def set_current(self, label: str) -> None: ...
        def set_task(self, message: str) -> None: ...
        def set_queue_rows(self, rows: list[ProgressQueueRow]) -> None: ...
        def set_worker_lines(self, rows: list[str]) -> None: ...
        def render(self) -> str: ...

Use existing dependencies only (standard library + Rich already present in repo). Do not add new third-party libraries for this refactor.
