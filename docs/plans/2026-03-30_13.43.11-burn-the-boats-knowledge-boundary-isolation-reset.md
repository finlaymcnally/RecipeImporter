## Structured Restatement Of User Desire

After the runtime-shape reset is complete, the knowledge stage itself must stop smearing one block’s semantic value across nearby blocks. The keep/drop decision for `knowledge` versus `other` must become a truly block-local judgment with no grouping pressure in the same model step and no packet-level framing that can launder title lists, chapter taxonomy, memoir, or generic advice into `knowledge`. Grouping should happen only after accepted `knowledge` blocks are already fixed. This is burn-the-boats work for a solo repository. The final live system should have one new knowledge contract only: grouping-free block classification first, grouping second, strong regression fixtures for the subtle overreach patterns, and deletion of the old combined semantic path.

## Purpose / Big Picture

After this change, the live knowledge stage will make one narrow decision first: for each owned nonrecipe block, is this block itself durable cooking knowledge or not? That decision will no longer happen in the same model answer object as grouping, and it will no longer happen inside a packet framing that encourages “these rows feel like one useful cluster, so keep them all.” Only blocks that survive that first boundary will move into a second grouping step.

The user-visible proof is specific. The current bad patterns from the March 30, 2026 benchmark must stop reproducing. Short title clusters such as vinaigrette or dressing name lists inside nonrecipe packets must stay `other` unless a given block truly stands on its own as durable cooking knowledge. Generic advisory prose such as “Be present. Stir, taste, adjust.” must stay `other` unless it crosses a clearly defined utility bar. True durable explanatory blocks must still remain `knowledge`. The acceptance evidence is not “the prompt looks stricter.” The acceptance evidence is that new focused regression tests and rerunnable fake/live-contract tests prove the target rows are stable across reordered packing, grouping removal, and repair.

## Outcomes & Retrospective

This plan is newly authored and not yet implemented. Its intended outcome is a knowledge stage that no longer “gets smarter” by widening its semantic authority boundary. Instead, it should become stricter and more honest: block-local keep/drop first, organization of accepted knowledge later.

The main lesson from the current benchmark evidence is that runtime simplification and semantic discipline are separate jobs. The three existing burn-the-boats plans solve the runtime-shape problem. This plan exists because a cleaner worker surface alone does not guarantee a cleaner `knowledge` boundary. The expected end state is one new live knowledge contract that forces the semantic boundary to be evaluated in the narrowest reasonable way.

## Context and Orientation

In this repository, the “knowledge stage” is the live Codex-backed stage that decides whether outside-recipe text belongs to final `knowledge` or final `other`, then records accepted knowledge into grouped artifacts. The run-level truth surfaces for that stage are `08_nonrecipe_route.json`, `09_nonrecipe_authority.json`, `09_nonrecipe_knowledge_groups.json`, `09_nonrecipe_finalize_status.json`, and the stage-local `knowledge_manifest.json` plus `nonrecipe_finalize/proposals/*.json`.

After the three prerequisite burn-the-boats plans land, the repository is expected to have one-file worker-visible `task.json` runtimes, repo-side edit extraction, bounded repair rewrites, and non-lethal supervision. This plan assumes that new runtime exists already. The work here is not about worker queues, helper files, or watchdogs. It is about the semantic contract inside the knowledge stage.

The benchmark evidence motivating this plan is concrete and should be treated as part of the repository context. In the March 30, 2026 run on `saltfatacidheatcutdown`, `strict_accuracy` improved sharply while `macro_f1_excluding_other` collapsed. The dominant old failure was under-calling real `KNOWLEDGE` as `OTHER`. The dominant new failure became over-calling many `OTHER` rows as `KNOWLEDGE`. Examples included clusters of short recipe-name or dressing-name rows and generic cookbook advice prose that was cooking-adjacent but not strong enough to deserve storage in the knowledge base.

The existing live knowledge contract already tries to defend against this in prompt text. It says block-local classification should be primary, packet order should be weak context only, and grouping assumptions should not drive keep/drop. That wording was not enough. The plan therefore changes the contract shape itself instead of merely strengthening the wording again.

The key term in this plan is “classification isolation.” Classification isolation means the model makes the `knowledge` versus `other` choice without simultaneously choosing grouping metadata, without seeing packet-profile summaries that imply semantic continuity, and without being rewarded for keeping adjacent rows together. The second key term is “regrouping invariance.” Regrouping invariance means a target block’s classification should stay the same even if other unrelated blocks in the same task file are reordered, replaced, or removed.

## Plan of Work

The first part of the implementation is to split the knowledge semantic contract into two explicit model steps inside `cookimport/llm/knowledge_stage/`. The first step should be a block classifier. It consumes one editable `task.json` file whose units represent individual outside-recipe blocks. Each unit should expose immutable evidence fields and one mutable `answer` subtree containing only the fields needed to answer “is this block durable cooking knowledge or not?” The grouping fields must not exist in this step at all.

The second part is to add a separate grouping step that runs only on the blocks accepted by the classifier. This grouping step can still use the same single-file runtime machinery introduced by the prerequisite plans, but its input file must contain only accepted knowledge blocks. It should not have access to rejected `other` rows because those rows are no longer under consideration. The grouping step’s job is organization only: assign group ids or local group keys and topic labels to already accepted blocks.

The third part is to reshape the classifier evidence so it is truly block-local. The current packet-profile hints such as heading counts, shard interpretation summaries, and attention-row lists should not survive into the new classifier path. Each classification unit should carry only the owned block text, its stable ids, optional very short before/after context, and any mechanically true local structure that is needed to interpret the block. The repository should not feed the classifier global packet-profile cues that encourage “this whole shard looks like a useful cookbook section” reasoning.

The fourth part is to add deterministic invariance tests and benchmark-derived regression fixtures. The repository needs new shared tests that take one target block and prove its classification is stable under several conditions: alone, inside a mixed file with unrelated blocks, inside a reordered file, and inside a repair rewrite. These tests are the only reliable way to prove that the subtle packet-level bleed is actually fixed. They should use real examples from the current failure set, not invented toy data only.

The fifth part is to update promotion and observability code so the new two-step semantic story is visible and understandable. `09_nonrecipe_authority.json` should still remain the final truth surface, but stage-local manifests and telemetry should clearly distinguish the classification step from the grouping step. A finished run should show how many blocks were classified as `knowledge`, how many remained `other`, how many knowledge blocks moved into grouping, and whether any repair rewrites happened at either stage.

The sixth part is deletion. Remove the old combined live knowledge path that asks the model to decide `category`, `group_key`, and `topic_label` in one semantic pass. Remove or rewrite tests that assume that combined contract. Remove prompt wording that still implies one-pass classification-plus-grouping. Update docs so the active knowledge-stage explanation teaches only the isolated classification-first contract.

## Canonical Classification `task.json` Schema

After the three prerequisite plans land, the new first-stage knowledge classifier must use one editable file with one shared rule: only `/units/<index>/answer` is writable, and that writable subtree contains only classification fields.

The required top-level classification file shape is:

    {
      "schema_version": "knowledge_block_classify.v1",
      "stage_key": "nonrecipe_classify",
      "assignment_id": "<stable assignment id>",
      "worker_id": "<stable worker id>",
      "mode": "initial" | "repair",
      "editable_json_pointers": [
        "/units/0/answer",
        "/units/1/answer"
      ],
      "units": [
        {
          "unit_id": "knowledge::<block_index>",
          "owned_id": "<stable block id>",
          "evidence": {
            "block_index": 123,
            "block_id": "<stable block id>",
            "text": "<owned block text>",
            "context_before": "<short local context or null>",
            "context_after": "<short local context or null>",
            "structure": {
              "heading_level": 2 | null,
              "table_hint": null | { ... compact immutable structure ... }
            },
            "routing_hints": ["<mechanically true local hint>", "..."]
          },
          "answer": {
            "category": "knowledge" | "other",
            "reviewer_category": "knowledge" | "chapter_taxonomy" | "decorative_heading" | "front_matter" | "toc_navigation" | "endorsement_or_marketing" | "memoir_or_scene_setting" | "reference_back_matter" | "other"
          },
          "previous_answer": { ... repair mode only ... },
          "validation_feedback": { ... repair mode only ... }
        }
      ]
    }

This classifier file must not include `group_key`, `topic_label`, packet-shape counts, shard-profile prose, or any equivalent grouping or packet-interpretation field. The goal is to make the model answer only the keep/drop question.

The required validator behavior for the classifier is:

1. `schema_version`, `stage_key`, `assignment_id`, `worker_id`, `mode`, `editable_json_pointers`, `unit_id`, `owned_id`, and every `evidence` field are immutable.
2. Every unit gets exactly one `category` and exactly one `reviewer_category`.
3. If `category` is `knowledge`, then `reviewer_category` must be `knowledge`.
4. If `category` is `other`, then `reviewer_category` must be one of the non-knowledge reviewer categories or `other`.
5. The validator enforces only structure, ownership, coverage, order, and enum validity. It must not attempt to re-decide semantics.
6. Repair mode contains only failed units and may not silently widen back to unrelated units.

## Canonical Grouping `task.json` Schema

The second model step must consume only the blocks already accepted as `knowledge`. Its job is grouping only.

The required top-level grouping file shape is:

    {
      "schema_version": "knowledge_group_only.v1",
      "stage_key": "knowledge_group",
      "assignment_id": "<stable assignment id>",
      "worker_id": "<stable worker id>",
      "mode": "initial" | "repair",
      "editable_json_pointers": [
        "/units/0/answer",
        "/units/1/answer"
      ],
      "units": [
        {
          "unit_id": "knowledge::<block_index>",
          "owned_id": "<stable block id>",
          "evidence": {
            "block_index": 123,
            "block_id": "<stable block id>",
            "text": "<owned block text>",
            "context_before": "<short local context or null>",
            "context_after": "<short local context or null>"
          },
          "answer": {
            "group_key": "<non-empty string>",
            "topic_label": "<non-empty string>"
          },
          "previous_answer": { ... repair mode only ... },
          "validation_feedback": { ... repair mode only ... }
        }
      ]
    }

The required validator behavior for grouping is:

1. Only accepted knowledge blocks may appear in this file.
2. Every unit must get exactly one `group_key` and one `topic_label`.
3. `group_key` and `topic_label` must be non-empty strings.
4. Deterministic code owns final canonical group-id normalization and final grouped artifact assembly.
5. Grouping repair mode contains only failed grouping units, never blocks rejected by the classifier.

## Classification Packing Rules

The classifier should not inherit the old packet semantics under a new file name. The planner must explicitly build classification assignments in a way that weakens bleed across units.

The required packing rules are:

1. Classification units are one block each.
2. The planner may batch several units into one `task.json` for throughput, but unit evidence must stay independent and grouping-free.
3. Units in one classification file must be ordered by a stable non-source-adjacent key such as a deterministic hash of `block_id`, not by source order.
4. `context_before` and `context_after` are local-only and short. They exist to interpret the owned block, not to imply packet continuity.
5. The planner must not include shard-level “heading-plus-body,” “packet shape,” “attention rows,” or similar narrative summaries in the classifier input.
6. If a block truly requires local heading context, that context belongs in the block’s own evidence, not as a packet-global interpretation note.

These rules matter because the current bug is not just “wrong prompt text.” The current bug is that source-adjacent or packet-profile framing lets the model form a whole-packet story and then spread that story across multiple blocks.

## Regression Fixture Set

The implementation must create a permanent regression fixture family under `tests/llm/` or a nearby shared fixture module using real blocks extracted from the March 30, 2026 failure.

The fixture family must include at least these pattern groups:

1. Short title-list rows that are cooking-adjacent but should stay `other`.
   The motivating examples include blocks such as “Balsamic Vinaigrette,” “Lemon Vinaigrette,” “Tomato Vinaigrette,” “Creamy Herb Dressing,” and similar short title-only rows.

2. Generic advisory prose that is true but too broad or too low-value to store as knowledge.
   The motivating example is “Instead, once you've chosen a recipe, don't let your own intimate knowledge of your own ingredients and kitchen and, most important, your own taste be overridden by what you're reading. Be present. Stir, taste, adjust.”

3. Real durable explanatory cooking knowledge that must remain `knowledge`.
   Use true-positive examples from the same source material that talk about cause, technique, troubleshooting, ingredient behavior, or durable sensory judgment.

4. Short conceptual headings that should only survive when their directly attached body is also useful.
   The repository should keep this edge case because the current prompt already allows it, but the new classifier must prove it on isolated fixtures rather than by packet vibe.

5. Chapter-taxonomy or menu-like blocks that must stay `other` even in otherwise useful neighborhoods.

The implementation must record the source of each fixture inside the test module comments or fixture metadata so a future contributor knows why the case exists.

## Validation and Acceptance

This plan is accepted only when the knowledge stage’s semantic contract itself has changed, not merely its prompt wording or runtime overhead.

The first acceptance criterion is contract separation. The live classification step must not ask for any grouping fields. The live grouping step must not make any keep/drop decisions. If one model answer still contains both `category` and grouping outputs in the same semantic pass, this plan is not done.

The second acceptance criterion is regrouping invariance. The targeted bad examples must keep the same `knowledge` versus `other` decision when they are classified alone, in mixed assignments, in reordered assignments, and in repair mode. If the answer changes because unrelated units were nearby, this plan has failed its core goal.

The third acceptance criterion is regression realism. The repository must contain permanent regression fixtures from the current bad run, including title-list rows, generic advisory prose, and true durable cooking knowledge. If the tests only cover toy data, the plan is not done.

The fourth acceptance criterion is true-positive retention. Real explanatory cooking knowledge from the same source material must still survive the stricter contract. The goal is not to make the stage timid again. The goal is to narrow the boundary cleanly.

The fifth acceptance criterion is deletion of the combined path. Live prompts, schemas, stage owners, and docs must no longer teach or execute the old combined classification-plus-grouping contract. Historical benchmark artifacts may remain on disk, but the active repository must describe one new contract only.

The sixth acceptance criterion is observability honesty. Run-level and stage-level outputs must describe the two-step semantic story clearly enough that a novice can tell whether a failure came from classification, grouping, or repair.

## Idempotence and Recovery

This implementation is safe to repeat because it is test-first and because the new contract is narrower than the old one. If the repository reaches a halfway state where classifier-only outputs exist but grouping is still attached to the old path, stop and finish the split rather than widening the compatibility window. Do not ship a live mode where both the old combined path and the new isolated path are treated as first-class.

If a repair-mode edge case becomes messy during implementation, do not collapse back to a combined contract to “keep things simple.” Keep the classifier and grouping steps separate and tighten deterministic promotion or task shaping instead. The whole point of this plan is to make the semantic boundary easier to reason about, not to regain convenience by blending concerns again.
