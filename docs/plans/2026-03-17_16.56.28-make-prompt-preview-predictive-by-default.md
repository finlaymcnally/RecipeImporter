# Make Prompt Preview Predictive By Default

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

After this change, `cf-debug preview-prompts` answers the forward-looking operator question only: “what would a run like this cost if I launched it now?” It should not silently turn into a retrospective cost report just because the selected run already finished. Exact “what did this completed run cost?” numbers remain available, but under the separate post-run `actual-costs` surface.

## Surprises & Discoveries

- Observation: the “live telemetry fix” solved retrospective accuracy but made preview worse as a planning tool.
  Evidence: `write_prompt_preview_for_existing_run(...)` always passed `load_prompt_preview_live_stage_summaries(...)` into `build_prompt_preview_budget_summary(...)` before this change.

- Observation: predictive warnings were under-sensitive for path-mode prompts because they looked mostly at rendered wrapper chars instead of total request chars.
  Evidence: a giant recipe-shard preview with unavailable token estimates emitted only `token_estimate_unavailable` until the warning thresholds started considering `estimated_request_chars_total`.

- Observation: local `data/output` history is the wrong predictive source because it is transient and can be cleared at any time.
  Evidence: prompt preview is used before launching Codex, so it cannot depend on whichever old runs still happen to be present on disk.

- Observation: benchmark-root predictive mode needs the deterministic/`vanilla` sibling, not the Codex sibling.
  Evidence: the deterministic sibling is the pre-run workload-shape artifact; the Codex sibling is the retrospective artifact with already-spent tokens.

- Observation: “best effort” fallback to a Codex-backed processed run destroys the semantics of prediction.
  Evidence: wrapper roots like `2026-03-17_17.45.00-fresh-summary-check` could still produce a close number while secretly reading the Codex-backed processed output, which is precisely the behavior predictive mode must reject.

- Observation: the retrospective feature already had a better home than preview.
  Evidence: completed prediction runs already write `prompt_budget_summary.json` during post-run artifact generation, so the missing piece was naming and discoverability, not another preview mode.

## Decision Log

- Decision: preview should be predictive-only rather than mixing planning and retrospective reporting.
  Rationale: preview is a safety/planning tool, not a retrospective reporting surface.
  Date/Author: 2026-03-17 / Codex

- Decision: predictive preview must not read local `data/output` history.
  Rationale: prompt preview needs to stay stable and useful before a run, even on a cleaned machine.
  Date/Author: 2026-03-17 / Codex

- Decision: predictive preview should use repo-owned per-stage models applied to vanilla-shaped payloads from the selected book.
  Rationale: that keeps the estimate forward-looking while still grounding it in the real recipe and nonrecipe workload shape of the chosen input.
  Date/Author: 2026-03-17 / Codex

- Decision: predictive preview must fail when it cannot positively resolve a deterministic/`vanilla` source.
  Rationale: a hard failure is better than retrospective leakage disguised as prediction.
  Date/Author: 2026-03-17 / Codex

- Decision: retrospective cost reporting should be surfaced as `actual-costs`, separate from preview.
  Rationale: preview should stay about future cost estimation; actual costs are a post-run artifact concern.
  Date/Author: 2026-03-17 / Codex

## Outcomes & Retrospective

Prompt preview is now predictive-only. It prefers the paired `vanilla` artifact when a benchmark root contains both `vanilla` and `codexfarm`, applies repo-owned per-stage models calibrated from a vanilla-to-Codex benchmark, and fails if it cannot positively resolve a deterministic/`vanilla` source. Retrospective cost reporting now lives under the separate `actual-costs` surface that points at the post-run artifact.

## Context and Orientation

`cookimport/llm/prompt_preview.py` is the read-only entrypoint that rebuilds shard prompts from saved processed output. `cookimport/llm/prompt_budget.py` turns those rebuilt shard rows into a budget summary. `cookimport/cf_debug_cli.py` exposes that behavior through `cf-debug preview-prompts`.

The key terms are:

- Predictive mode: estimate future cost for a run shaped like the selected input.
- Actual costs: retrospective cost reporting from the finished-run post-run artifact (`prompt_budget_summary.json`).
- Predictive stage model: a repo-owned stage-specific token-per-request-char ratio set calibrated from a vanilla-to-Codex benchmark.

## Plan of Work

The implementation changed `write_prompt_preview_for_existing_run(...)` into a predictive-only path. It now resolves only deterministic/`vanilla` sources, applies repo-owned stage models, and treats total request chars as a first-class danger signal for path-mode shard files. Retrospective cost lookup moved to the separate `cf-debug actual-costs` command, which resolves the post-run `prompt_budget_summary.json` artifact.
