## Purpose / Big Picture

After this change, `cf-debug preview-prompts` answers the forward-looking operator question by default: “what would a run like this cost if I launched it now?” It should not silently turn into a retrospective cost report just because the selected run already finished. Exact “what did this completed run cost?” numbers remain available, but only when the operator explicitly asks for observed mode.

## Surprises & Discoveries

- Observation: the “live telemetry fix” solved retrospective accuracy but made preview worse as a planning tool.
  Evidence: `write_prompt_preview_for_existing_run(...)` always passed `load_prompt_preview_live_stage_summaries(...)` into `build_prompt_preview_budget_summary(...)` before this change.

- Observation: predictive warnings were under-sensitive for path-mode prompts because they looked mostly at rendered wrapper chars instead of total request chars.
  Evidence: a giant recipe-shard preview with unavailable token estimates emitted only `token_estimate_unavailable` until the warning thresholds started considering `estimated_request_chars_total`.

- Observation: calibration rows needed runtime-mode filtering.
  Evidence: old shard-runtime behavior and the new classic one-shot behavior have materially different token inflation, so calibration from mixed runtimes would not be trustworthy.

## Decision Log

- Decision: make predictive mode the default and observed mode opt-in.
  Rationale: preview is primarily a safety/planning tool, not a retrospective reporting surface.
  Date/Author: 2026-03-17 / Codex

- Decision: keep observed mode rather than deleting exact-run reuse entirely.
  Rationale: exact telemetry is still useful when comparing prediction versus actual after a run finishes.

- Decision: filter calibration rows to `classic_task_farm_v1`.

## Outcomes & Retrospective

Prompt preview now has two clear modes instead of one muddy hybrid. The default predictive mode ignores exact live telemetry from the selected run and uses historical classic-runtime calibration when available. The explicit observed mode reuses exact live stage telemetry for retrospective checks. The real benchmark check on `data/golden/benchmark-vs-golden/2026-03-17_16.06.55` now shows the split clearly: predictive mode estimated about `3.28M` total tokens, while observed mode reported the completed run’s exact `2.97M` total tokens.

## Context and Orientation

`cookimport/llm/prompt_preview.py` is the read-only entrypoint that rebuilds shard prompts from saved processed output. `cookimport/llm/prompt_budget.py` turns those rebuilt shard rows into a budget summary. `cookimport/cf_debug_cli.py` exposes that behavior through `cf-debug preview-prompts`.

The key terms are:

- Predictive mode: estimate future cost for a run shaped like the selected input.
- Observed mode: reuse exact live telemetry from the selected completed run.
- Historical calibration: stage-specific token-per-request-char ratios derived from prior completed runs under `data/output`.

## Plan of Work

The implementation changed `write_prompt_preview_for_existing_run(...)` to accept an explicit `estimation_mode`, default it to predictive, and only load exact live stage telemetry in observed mode. The budget builder now records the chosen mode in the summary payload and labels the summary text accordingly. Calibration loading now rejects worker status rows whose nearby `phase_manifest.json` does not declare `runtime_mode = classic_task_farm_v1`. The warning builder now treats total request chars as a first-class danger signal for path-mode shard files.

## Validation and Acceptance

Acceptance is:

- default preview manifests record `preview_settings.estimation_mode = "predictive"`
- observed mode records `preview_settings.estimation_mode = "observed"`
- predictive mode ignores exact live telemetry from the selected run
- observed mode reuses exact live telemetry from the selected run
- giant path-mode shard payloads still trigger budget danger warnings even if token estimates are unavailable

## Idempotence and Recovery

These edits are safe to rerun. If predictive mode has no matching calibration, the safe fallback is to report token estimates as unavailable while still warning on large request-char totals.
