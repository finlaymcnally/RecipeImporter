## Structured Restatement

The repository must move to one clean non-recipe routing model. The line-role stage must remain the one pass that labels the whole book, and it must stay excellent at recipe-local labels such as titles, ingredients, instructions, yields, notes, and variants. For rows outside accepted recipe spans, line-role must not pretend to know final semantic usefulness. Instead, it must route those rows into exactly two first-pass outcomes: obvious junk that should never reach knowledge, and non-recipe candidate material that should be considered by the knowledge stage. The knowledge stage must become the only stage that decides whether candidate non-recipe material is useful `KNOWLEDGE` or final useless `OTHER`. This plan intentionally removes the older contract instead of preserving it. There is no migration shim, no compatibility mode, and no support for the older naming or authority split in the live code after this change lands.

## Purpose / Big Picture

After this change, a user can run RecipeImport on a book and trust that line-role is doing one coherent full-book routing pass instead of prematurely collapsing most non-recipe prose into `OTHER`. Recipe content will still be labeled directly and strongly by line-role, obvious garbage such as tables of contents and publisher promotions will be excluded before knowledge runs, and only the remaining non-recipe candidate material will be sent to knowledge for final `KNOWLEDGE` versus `OTHER` judgment. The user-visible proof is that benchmark and prediction artifacts will show three distinct stories instead of one blurred one: recipe-local labels from line-role, excluded junk that never went to knowledge, and candidate non-recipe rows that knowledge either promoted to `KNOWLEDGE` or rejected to final `OTHER`.

The practical benefit is that a run like `saltfatacidheatcutdown` will stop looking like a fake Codex win where every row says `decided_by="codex"` while every label still mirrors the deterministic seed. After implementation, line-role prompt artifacts will no longer carry seed labels as model-facing answers, the line-role output contract will use routing labels for outside-recipe material, and knowledge-stage inputs will contain only candidate rows instead of a mixed bag of everything non-recipe.

For this plan, use the following terms consistently. A “recipe-local label” is one of the structural labels that belongs inside recipes, such as `RECIPE_TITLE`, `INGREDIENT_LINE`, or `INSTRUCTION_LINE`. A “routing label” is an internal line-role label used only to direct later work. `NONRECIPE_EXCLUDE` means obvious garbage that should never reach knowledge. `NONRECIPE_CANDIDATE` means non-recipe content that is not obviously garbage and must be considered by knowledge. A “final semantic label” is the final outside-recipe truth used by staging and benchmark outputs: `KNOWLEDGE` or `OTHER`.

## Milestones

### Milestone 1: Replace the line-role contract with route labels and remove worker-visible seed answers

At the end of this milestone, staging no longer uses “candidate” as its primary concept. ### Milestone 3: Make knowledge the only owner of final outside-recipe usefulness labels

At the end of this milestone, knowledge shard inputs contain only candidate rows. Excluded rows are absent from the billed payload. ### Milestone 5: Remove the old vocabulary from tests, docs, and run settings

At the end of this milestone, the repository no longer advertises or normalizes the old non-recipe contract. Tests, docs, CLI text, run-setting names, analytics dimensions, and benchmark reports all use the new route-first vocabulary. ## Validation and Acceptance

Acceptance is behavioral, not just structural.

The third acceptance criterion is final projection. After non-recipe finalize resolves the candidate rows, the final semantic artifacts must contain only the legacy-free final label family: recipe-local labels plus final `KNOWLEDGE` and `OTHER`. Excluded rows become final `OTHER` without non-recipe finalize. Candidate rows become final `KNOWLEDGE` or final `OTHER` only through knowledge.

The fourth acceptance criterion is the original benchmark failure mode. The repository must contain at least one regression test that proves the model-facing line-role prompt path no longer exposes deterministic labels as answers. That test should fail if `deterministic_label` appears in model-facing `in/*.json`, in hints the worker is told to read first, or in the initial work ledger. This is the guardrail that prevents another 1471-row zero-change rubber-stamp run.

The fifth acceptance criterion is vocabulary cleanup. The full test run and the updated docs must not describe the live contract using `candidate`, `excluded`, `seed_only`, `knowledge_unreviewed_seed_kept`, or the older pipeline ids. Historical files may still exist under `data/golden`, but the live code and docs must not advertise the old story.

## Idempotence and Recovery

This work should be implemented as one coherent branch-sized refactor, but each file-level edit should be safe to rerun. The safest order is the order given in the milestones: contract first, staging second, knowledge third, artifacts fourth, docs and tests last. If a partial implementation leaves the codebase in a mixed state, do not add compatibility shims to bridge the old and new contracts. Instead, continue until every producer and consumer speaks the new route-first vocabulary.

Because this is a burn-the-boats change, old artifact readers and old fixture payloads are expected to fail once the new contract lands. The recovery path is to update the remaining reader or fixture immediately, not to normalize the old shape in production code. Historical benchmark directories under `data/golden/` do not need runtime compatibility. They may be inspected manually, but the live code should not attempt to load both the old and new route vocabularies.

If a command fails midway through targeted test updates, rerun that test after finishing the missing producers or consumers. The commands in `Concrete Steps` are safe to rerun. If `.venv` becomes stale, reactivate it and rerun `pip install -e .[dev]`.
