# Context

This note explains a concrete gap observed on `Hix written.docx`:

- Gold: `data/golden/9_hix_written/exports/freeform_span_labels.jsonl`
- Benchmark eval: `data/golden/eval-vs-pipeline/2026-02-23_12.25.07/eval_report.md`
- Processed stage output inspected manually: `data/output/2026-02-23_12.37.50`

Observed symptom:

- `eval_report.md` looks extremely bad (many `0.000` metrics).
- Cookbook output from stage run looks "not perfect but pretty good".

Both observations are true. They are measuring different surfaces.

# What Is Actually Being Compared

`labelstudio-benchmark` is currently wired to generate prediction tasks with `task_scope="pipeline"`.

Relevant wiring:

- `cookimport/cli.py`: `labelstudio_benchmark(...)`
- Calls `generate_pred_run_artifacts(..., task_scope="pipeline", ...)` and (upload mode) `run_labelstudio_import(..., task_scope="pipeline", ...)`.

Evaluation does **not** score final staged recipes directly.

Evaluation path:

1. Load predicted task artifacts from `prediction-run/label_studio_tasks.jsonl`.
2. Convert each task to a labeled range using task metadata and location fields.
3. Load freeform gold spans from `freeform_span_labels.jsonl`.
4. Compare predicted ranges vs gold ranges with strict IoU threshold (default `0.5`) plus source identity checks.

So benchmark scoring here is a **range-localization test**, not a direct "how good are final recipes" test.

# Exact Scoring Mechanics (Code-Verified)

## Gold loading

Gold freeform spans are loaded from JSONL rows and converted to block ranges via:

- `touched_block_indices` (or fallback `touched_blocks[].block_index`)
- `start_block_index = min(touched_indices)`
- `end_block_index = max(touched_indices)`

For this run, gold spans were mostly single-block ranges.

## Gold dedupe

Gold is deduped before scoring by key:

- `(source_hash, source_file, start_block_index, end_block_index)`

Conflict behavior:

- Same-range, conflicting labels -> majority vote if one label wins
- Exact tie -> drop tied group from scored gold

This run:

- Input gold rows: `828`
- Deduped/scored gold rows: `702`
- Removed rows: `126`
- Conflict groups: `12` (majority resolved `8`, tie dropped `4`)
- Dropped tie rows: `8`

## Pred loading

Predicted tasks are loaded from `label_studio_tasks.jsonl` and mapped as follows:

- Label inferred from pipeline chunk metadata (`chunk_level`, `chunk_type`, `chunk_type_hint`)
- Span range from `location` keys using preference order:
  - `start_block/end_block`
  - `start_line/end_line`
  - `block_index/block_index`

For this run, tasks had `start_line/end_line` ranges.

## Strict match rule

A predicted span matches a gold span only if all are true:

1. same normalized label
2. source compatible (strict mode by default: matching hash/file rules)
3. IoU (Jaccard overlap) `>= overlap_threshold` (default `0.5`)

IoU formula:

- `intersection / union` over integer span ranges

# What Happened In This Hix Run

## High-level report numbers

From `eval_report.md`:

- Gold spans (deduped): `702`
- Predicted spans: `643`
- Strict recall: `0.000 (0/702)`
- Strict precision: `0.000 (0/643)`

But diagnostics in the same report show:

- Any-overlap coverage (label-agnostic): `1.000 (702/702)`
- Same-label any-overlap: `0.930 (653/702)`
- Supported-label same-label any-overlap: `0.998 (653/654)`
- Any-overlap coverage by key labels:
  - `RECIPE_TITLE`: `1.000 (45/45)`
  - `INGREDIENT_LINE`: `1.000 (415/415)`
  - `INSTRUCTION_LINE`: `1.000 (193/193)`

So this is not "no overlap". It is "almost no overlap large enough to pass strict IoU>=0.5".

## Granularity mismatch (the core issue)

Empirical span-width stats for this run:

- Gold span widths (block-range width):
  - min `1`, p50 `1`, p90 `1`, max `1`, avg `1`
- Pred span widths (from pipeline location ranges):
  - min `8`, p50 `24`, p90 `49`, max `73`, avg `27.99`

This means predicted ranges are often recipe-wide while gold spans are one-block precise.

If a gold span width is `1` and pred width is `24`, best-case IoU is roughly:

- `1 / 24 = 0.0417`

That will never pass `0.5`.

Measured best-overlap distribution (same-label search) in this run confirms this:

- p50 `0.0417`
- p90 `0.0769`
- max `0.125`
- count with IoU `>=0.5`: `0`
- count with IoU `>=0.1`: `18`

This directly explains the strict `0.000` metrics.

## Reused recipe ranges across many atomic chunks

Prediction tasks show repeated `start_line/end_line` ranges across many chunk rows in a recipe.

Example pattern per recipe range:

- one `RECIPE_TITLE` range row
- one `OTHER` range row
- many `INGREDIENT_LINE` rows
- many `INSTRUCTION_LINE` rows

In this run:

- Raw predicted rows converted: `642`
- Dedup by `(source_hash, source_file, label, start, end)`: `155`
- Unique `(source_hash, source_file, start, end)` groups (ignoring label): `45`
- All 45 groups had multiple labels attached to the same coarse range

This is why many overlaps exist, but strict boundary scoring still fails.

# Why Cookbook Output Can Still Look Good

Stage outputs and freeform strict IoU are different quality surfaces.

Stage output quality checks whether final recipe JSON content is usable.
Freeform strict IoU checks whether predicted label spans tightly localize gold spans.

For this exact case, benchmark-generated stage output and the later stage run were effectively the same at final-draft content level:

- both had `45` recipes
- final draft files matched `45/45` by content hash
- run config hash matched (`01ebb07ac2413c488924ced1b8ffe9a898d90dab8c0aa5ad2107c4ec7c0e1a67`)

So "looks pretty good in cookbook" is consistent with "strict freeform IoU is near-zero" under coarse prediction ranges.

# How To Read Similar Reports In The Future

When strict metrics are near zero, check these diagnostics before concluding parser failure:

1. `Any-overlap coverage` high + strict low:
   - likely localization granularity mismatch (coarse ranges vs fine gold)
2. `Same-label any-overlap` high:
   - label assignment may be mostly right, but boundaries are coarse
3. `Any-overlap` low too:
   - likely true recall failure (missed regions/content)
4. Supported labels strong but `YIELD_LINE`/`TIME_LINE` weak:
   - pipeline task mapping may not emit those classes in the same way as freeform gold

# Practical Debug Checklist

For any future run with "looks good but scores bad":

1. Confirm benchmark scope:
   - check benchmark wiring/manifest for `task_scope="pipeline"`
2. Inspect prediction locations:
   - sample `prediction-run/label_studio_tasks.jsonl` for `location.start_line/end_line`
3. Compare span widths:
   - gold touched-block widths vs prediction location widths
4. Compare strict vs any-overlap diagnostics:
   - if strict low, any-overlap high, suspect granularity mismatch
5. Sanity-check stage outputs separately:
   - compare final drafts and report totals (`totalRecipes`, warnings/errors)

# Implication For Benchmark Design

Current `eval-vs-pipeline` strict score is a conservative localization metric under a contract mismatch:

- predictions are pipeline chunk ranges
- gold is freeform span annotations

That score is still useful, but it can understate practical cookbook quality when prediction ranges are coarse.

If/when benchmark contract is revised, options include:

1. generate freeform-style prediction spans for scoring against freeform gold
2. add a separate coarse-granularity metric track and avoid presenting strict IoU as the only headline
3. map pipeline atomic chunks to finer ranges before freeform eval

Until then, interpret strict freeform IoU together with any-overlap and staged-output checks.
