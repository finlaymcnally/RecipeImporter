This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This plan follows [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md) requirements.

## Purpose / Big Picture

After this change, recipeimport will not just call CodexFarm and check pass/fail. It will also capture structured CodexFarm execution telemetry signals (retry context, Heads Up tip details, failure categories, output fingerprints/previews, and Codex event summaries) and persist them into recipeimport run artifacts. This gives a direct prompt-tuning loop from recipeimport outputs, without manually opening CodexFarm CSV files.

## Surprises & Discoveries

- Observation: CodexFarm `process --json` payload includes run metadata and optional telemetry report, but the full prompt-tuning fields live in `codex_exec_activity.csv` rows.
  Evidence: `/home/mcnal/projects/shared/CodexFarm/src/codex_farm/cli.py` and `/home/mcnal/projects/shared/CodexFarm/docs/08-external-program-reference/telemetry-contracts.md`.

- Observation: recipeimport currently parses only `run_id`, `exit_code`, and `output_schema_path` from `process --json`, and does not read `codex_exec_activity.csv`.
  Evidence: `/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_runner.py`.

- Observation: CodexFarm telemetry rows can be large (`prompt_text`, `output_preview`, stderr/stdout tails), so copying full rows into recipe manifests can bloat artifacts quickly.
  Evidence: Local `var/codex_exec_activity.csv` rows and CodexFarm telemetry contract docs.

- Observation: Missing telemetry CSV should not fail conversion runs because CodexFarm telemetry is observability, not conversion correctness.
  Evidence: Existing CodexFarm behavior treats telemetry writes/readers as best-effort.

## Decision Log

- Decision: Add runner-level telemetry extraction keyed by `run_id` + `pipeline_id` and expose it as structured result data to orchestrators.
  Rationale: This keeps CodexFarm-call parsing centralized in one boundary and avoids duplicating CSV parsing logic across pass modules.
  Date/Author: 2026-02-28 / Codex

- Decision: Persist compact but high-signal telemetry fields and aggregates in recipeimport manifests/reports rather than full raw CSV rows.
  Rationale: Full rows can be very large due to prompt/output text; compact persistence is sufficient for prompt-tuning loops while keeping artifact size manageable.
  Date/Author: 2026-02-28 / Codex

- Decision: Keep telemetry ingestion non-fatal and persist warnings in payload when CSV is missing/unreadable.
  Rationale: Prompt-tuning telemetry should not block stage conversion outputs when CodexFarm data-dir state is unavailable.
  Date/Author: 2026-02-28 / Codex

## Outcomes & Retrospective

Implemented end-to-end. Recipeimport now receives CodexFarm process metadata plus run-scoped codex-exec telemetry slices and persists them in recipe pass manifests (`process_runs`) and pass4/pass5 reports (`process_run`). This meets the purpose of providing prompt-tuning-ready telemetry without requiring manual CSV parsing.

Main tradeoff: persisted telemetry is intentionally compact and row-capped. This keeps artifacts manageable while preserving the requested retry/Heads Up/failure/output/event signals.
