The replay helper and packet ledger are meant to remove ambiguity, not introduce another runtime layer with its own behavior.

## Purpose / Big Picture

Before changing worker behavior, retry budgets, progress rendering, or finalization, the repository needs one deterministic, no-token way to say what happened in a knowledge run and one repo-owned state vocabulary that can represent what happened to each packet. Without that foundation, every later fix risks arguing about symptoms instead of facts. The March 20 Salt Fat run already contains the evidence needed to define that fact surface. What is missing is a stable helper that reads it and one packet-ledger model that runtime code and tests can both share.

After this plan lands, a developer should be able to point a replay helper at the March 20 Salt Fat benchmark root and knowledge root and get one compact rollup that answers the first questions an operator actually has: how many packets existed, how many worker-local outputs existed, how many were malformed, how many follow-up attempts were started, how many stale follow-ups survived, what the worker outcome mix looked like, and which expected top-level stage artifacts were missing. That summary should be easy to assert against in tests and easy to compare before and after later runtime changes.

The user-visible proof is modest but important. Run replay-oriented tests against the saved March 20 roots and see a stable machine-readable summary instead of manually counting `repair_status.json`, `watchdog_retry/status.json`, worker `out/*.json`, and missing stage-level files.

## Outcomes & Retrospective

This plan is implemented. The repository now has one shared packet-ledger vocabulary in `cookimport/llm/knowledge_runtime_state.py` and one read-only replay helper in `cookimport/llm/knowledge_runtime_replay.py`. The replay tests prove that the March 20 Salt Fat bundle can now be summarized mechanically instead of by hand-counting worker folders.

The final acceptance surface is intentionally small and factual. The replay helper reports the packet total, shard total, worker output/malformed counts, follow-up counts, stale follow-up counts, worker outcome buckets, and missing expected artifacts for both the knowledge root and the paired benchmark root. That gives later plans one stable no-token oracle without pretending to simulate the original scheduler.

The main lesson is that the replay seam is most useful when it stays boring. It does not need to infer final review authority for an unfinished run. It only needs to expose stable facts and one shared packet-state language that later live-runtime work can adopt.

## Context and Orientation

The non-recipe knowledge stage currently spreads truth across worker-local `out/*.json`, main worker `live_status.json`, `repair_status.json`, `watchdog_retry/status.json`, benchmark timeseries artifacts, and missing-or-present stage summary files. That makes it hard to test later fixes in isolation because there is no single repo-owned view of packet state. The result is that every debugging pass starts by crawling directories and interpreting artifacts manually.

The key evidence bundle for this plan is:

- benchmark root: `data/golden/benchmark-vs-golden/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/`
- knowledge root: `data/output/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-20_21.44.56/raw/llm/saltfatacidheatcutdown/knowledge/`

The saved run matters because it captures the whole failure family in one place:

- packet work existed, but the UI still looked shard-based and frozen;
- worker-local outputs existed, but many were malformed or untrusted;
- retries and repairs existed, but the yield was poor and the counts were large;
- follow-up work existed, but some of it survived after the parent was effectively done;
- top-level evidence existed, but the stage package was not finalized.

In this repository, a `packet` is the real answer unit inside a worker-owned shard. A `packet ledger` is the repo-owned state table that records the latest known state and final outcome for every packet. A `stage rollup` is the aggregate summary derived from that ledger. A `worker outcome category` is a stable label for the main worker session itself, separate from whether packet outputs were accepted. A `replay summary` is a read-only reconstruction of those facts from saved artifacts.

This plan is intentionally foundational rather than directly user-facing. If a later plan changes packet leasing, output authority, or retry policy, this foundation should still remain the shared state vocabulary and the shared acceptance seam.

## Plan of Work

### Milestone 1: Define the packet-ledger vocabulary

Create `cookimport/llm/knowledge_runtime_state.py`. Define the small private data model that later plans will depend on: packet identifiers, terminal packet outcomes, recovery budget slots, worker outcome categories, follow-up categories, and stage-rollup counters. Keep the model simple and biased toward explicit enums and dataclasses rather than stringly-typed dicts.

The key property is that the model must be expressive enough for later plans without becoming broad or abstract. It should answer these questions cleanly:

- what is the current known state of packet X;
- what is the terminal outcome of packet X, if any;
- what was the latest reason that moved packet X into that state;
- how many packets are in each terminal category;
- how many workers fall into each worker-outcome category;
- how many follow-up attempts of each kind were seen.

This milestone does not need to wire the entire live runtime into the new model immediately. It only needs to define the vocabulary and provide helpers that later plans can adopt incrementally.

### Milestone 2: Build the replay summarizer

Create `cookimport/llm/knowledge_runtime_replay.py`. This module should read the saved knowledge root plus the paired benchmark root and reconstruct the facts that later plans need as acceptance fixtures: packet totals, malformed outputs, retry counts, repair counts, stale follow-up counts, worker completion categories, transport anomaly categories, and missing finalization files.

Keep the replay summarizer deliberately narrow:

- it should not mutate the evidence roots;
- it should not emit a second artifact tree by default;
- it should not attempt to guess ambiguous semantics that the artifacts do not support;
- it should return one compact summary object that later tests can assert against.

The summarizer should treat artifact presence and artifact content as separate facts. For example, “worker wrote 49 output files” and “all 49 were malformed JSON” should remain distinct fields because later plans need both counts.

### Milestone 3: Add deterministic tests for state and replay behavior

Add tests in `tests/llm/test_codex_farm_knowledge_orchestrator.py` or a nearby dedicated test module to prove packet-rollup classification from small synthetic fixtures. These small tests should be used for the pure state helpers because they are easier to keep stable than tests that always walk the full Salt Fat evidence tree.

Then add replay-oriented tests that point at the March 20 artifact roots and assert concrete summary counts and booleans. These tests should be resilient to later runtime changes because they are testing historical read-only evidence, not current runtime behavior.

The most useful replay assertions are likely:

- packet total reconstructed from `task_manifest.jsonl`;
- shard total reconstructed from worker assignments or manifest artifacts;
- malformed output count from worker-local `out/*.json`;
- follow-up counts from `repair_status.json` and `watchdog_retry/status.json`;
- stale follow-up count based on live-status terminal absence or lingering `running` state;
- missing expected stage package files such as `phase_manifest.json`, `promotion_report.json`, `telemetry.json`, and `failures.json`.

### Milestone 4: Prepare the foundation for later plan adoption

Add small helper seams in the current orchestrator or neighboring modules only if needed so later plans can adopt the packet-ledger vocabulary without rewriting the foundation. The goal is not to move the whole runtime at once. The goal is to make the next plan able to ask for a `KnowledgePacketRecord` or `KnowledgeStageRollup` instead of inventing new local types.

This milestone also includes a short doc touch once code exists. The short docs do not need a full rewrite yet, but they should define the packet-ledger vocabulary if later plans will reference it.

## Validation and Acceptance

This plan is complete when:

- replay tests reconstruct the March 20 packet-versus-shard mismatch and the missing-finalization shape from saved artifacts;
- one packet-ledger data model exists in repo code and later plans can build on it;
- the replay helper can explain malformed outputs, follow-up counts, and missing top-level artifacts without manual directory crawling;
- a novice can inspect one replay summary object and understand why the run failed without counting files manually.

The strongest acceptance shape is a short test transcript like this:

    replay summary:
      packets: 485
      shards: 10
      malformed worker outputs: <expected count>
      repair attempts: <expected count>
      watchdog retries: <expected count>
      stale follow-ups: <expected count>
      phase_manifest present: false

The exact counts should come from the actual evidence and be asserted in tests rather than described vaguely in prose.

## Idempotence and Recovery

The replay helper must not mutate historical evidence roots. Tests may copy artifacts into temp directories when mutation is needed for synthetic cases. The helper should tolerate missing files gracefully and report them as missing facts rather than raising avoidable crashes, because missing top-level artifacts are themselves part of the historical failure.

If the first state model is too small for later plans, extend it incrementally. Do not replace it with a second competing packet-state vocabulary. This plan’s value comes from giving the rest of the rebuild one shared language.
