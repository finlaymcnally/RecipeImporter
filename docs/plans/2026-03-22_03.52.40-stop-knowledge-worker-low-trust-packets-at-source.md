# Stop Knowledge Worker Low-Trust Packets At The Source

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The knowledge stage already has a good validator, but some workers still write low-trust semantic packets before the repository ever gets a clean chance to promote them. In the March 22 benchmark run, many task packets were structurally fine and used the right evidence blocks, yet `snippets[].body` still copied the source quote verbatim. That is the wrong place to spend model effort.

After this change, a generated knowledge worker root should make the right behavior obvious before repo ingest has to reject anything. A novice should be able to open the worker root, see a representative good snippet example instead of a misleading heading-style quote, run `python3 tools/knowledge_worker.py check ...`, and get a direct explanation of which snippet body is low-trust and how to rewrite it. The observable success condition is that worker-facing examples, hints, and helper output all consistently teach “rewrite the claim, do not copy the quote.”

This plan is deliberately narrower than `docs/plans/2026-03-22_03.49.35-stop-knowledge-snippet-copying-before-ingest.md`. That sibling plan covers repo-side repair and poisoned-worker recovery. This plan only covers upstream prevention at the worker workspace seam.

## Surprises & Discoveries

- Observation: the current worker-local helper already rejects low-trust packets correctly; the remaining gap is that the worker-facing contract still teaches weak examples and produces weak diagnostics.
  Evidence: `cookimport/llm/knowledge_workspace_tools.py` routes `check_workspace_draft(...)` through `validate_knowledge_shard_output(...)`, and `install_workspace_draft(...)` refuses invalid packets.

- Observation: the generated “valid” example can currently teach the wrong lesson.
  Evidence: in `data/output/2026-03-22_03.33.50/.../knowledge/workers/worker-001/OUTPUT_CONTRACT.md`, the “Valid example” snippet body is `Praise for SALT FAT ACID HEAT.` with the same short quote as evidence. That is decorative front matter, not a model example of reusable cooking knowledge.

- Observation: the example generator currently picks the first available task rather than a representative knowledge-like chunk.
  Evidence: `_resolve_sample_task(...)` in `cookimport/llm/knowledge_workspace_tools.py` returns the first task row with an input payload, which in the benchmark worker root yields early front-matter content rather than a real knowledge chunk.

- Observation: the low-trust benchmark failures are concrete copied-quote packets, not ambiguous borderline paraphrases.
  Evidence: `data/output/2026-03-22_03.33.50/.../worker-001/out/saltfatacidheatcutdown.ks0000.nr.task-047.json` has `body == quote` for the only snippet, and `task_status.jsonl` records `validation_errors=["semantic_snippet_echoes_full_chunk"]` with `terminal_reason_detail="worker repeatedly produced low-trust outputs that failed semantic validation"`.

- Observation: the current helper surface does not tell the worker enough when validation fails.
  Evidence: `install_workspace_draft(...)` currently raises `knowledge draft failed validation: <error names>`, which tells the worker that the draft is invalid but not which snippet copied which source text or what a successful rewrite should look like.

- Observation: `cookimport/llm/knowledge_workspace_tools.py` contains a second `_render_validation_error_help(...)` definition later in the file, so helper-output changes must update the shadowing copy too.
  Evidence: the first targeted rerun still emitted the old generic guidance after the new detail renderer was added, and only updating the later duplicate definition switched the output to chunk/snippet-level guidance.

## Context and Orientation

The knowledge stage is the optional non-recipe review pass that keeps general cooking knowledge and drops the rest. Each worker-local task packet lives under `knowledge/in/<task_id>.json`, and the worker writes one semantic packet result to `knowledge/workers/<worker_id>/out/<task_id>.json`. The authoritative repo validator is `validate_knowledge_shard_output(...)` in `cookimport/llm/codex_farm_knowledge_ingest.py`.

This plan touches only the workspace-teaching seam, not the repo-side repair seam.

The main files are:

- `cookimport/llm/knowledge_workspace_tools.py`, which builds `OUTPUT_CONTRACT.md`, `examples/*.json`, and the mirrored `tools/knowledge_worker.py` helper.
- `cookimport/llm/codex_farm_knowledge_orchestrator.py`, which writes worker hints and builds the knowledge worker prompt.
- `tests/llm/test_knowledge_workspace_tools.py`, which now covers representative sample selection and actionable snippet-copy guidance without duplicating validator authority.
- `tests/llm/test_codex_farm_knowledge_orchestrator.py`, which already covers prompt text, mirrored helper files, and worker-root discoverability.
- `tests/llm/test_knowledge_output_ingest.py`, which remains the validator-truth seam and should not gain a duplicate worker-only rule set.

- `data/output/2026-03-22_03.33.50/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-22_03.34.33/raw/llm/saltfatacidheatcutdown/knowledge/task_status.jsonl`, which records `52` `semantic_snippet_echoes_full_chunk` failures.

The current worker root already includes the right basic files: `OUTPUT_CONTRACT.md`, `examples/`, `tools/knowledge_worker.py`, `current_task.json`, `assigned_tasks.json`, and `hints/`. The bug is that those files are not yet teaching the snippet-quality boundary sharply enough.

## Plan of Work

First, harden the worker-visible examples in `cookimport/llm/knowledge_workspace_tools.py`. Change `_resolve_sample_task(...)` and `build_knowledge_workspace_contract_examples(...)` so the generated examples prefer a representative knowledge-like chunk instead of blindly using the first task row. The resulting valid example should demonstrate a short extracted claim backed by a longer verbatim evidence quote. The invalid example should use the same source surface but copy the quote or a long evidence span into `snippets[].body`. That pairing matters because it teaches the worker what to rewrite, not just what to avoid.

Second, make helper diagnostics actionable. Extend `check_workspace_draft(...)` and the mirrored `knowledge_worker.py` rendering so `check` and `install` print worker-facing details for low-trust snippet-copy failures. At minimum, the helper should tell the worker which chunk and snippet failed, whether the body copied the whole chunk or its cited evidence surface, and give a one-line rewrite rule such as “replace the quote with a shorter grounded takeaway in your own phrasing.” The repo validator remains authoritative; this plan only exposes its intent in a more useful format.

Third, tighten the worker prompt and task-local hints in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. The prompt already mentions `scaffold`, `check`, and `install`, but it should go one step further and say that a packet is not finished until `check` passes. It should also explicitly say that direct writing to `out/<task_id>.json` without a passing `check` is the wrong path. The task hint files should add one short local reminder: “good body = shorter grounded claim; bad body = copied evidence quote.”

Fourth, keep the scope narrow by adding focused tests around worker teaching and diagnostics rather than around repo-side repair. Add a dedicated worker-tooling test file such as `tests/llm/test_knowledge_workspace_tools.py` if that keeps the behavior isolated and easy to understand. Use `tests/llm/test_codex_farm_knowledge_orchestrator.py` for prompt and mirrored-worker-root assertions. Do not duplicate validator semantics in a second test seam; reuse the existing validator through the helper.

## Validation and Acceptance

This plan is complete when all of the following are true.

The worker-visible “valid” example in `OUTPUT_CONTRACT.md` and `examples/valid_semantic_packet.json` demonstrates a real grounded extraction pattern rather than a decorative heading or a verbatim quote.

The worker-visible “invalid” example in `OUTPUT_CONTRACT.md` and `examples/invalid_echo_packet.json` shows the exact bad pattern the benchmark produced: a snippet body that copies its evidence quote or long source surface.

Running `python3 tools/knowledge_worker.py check <draft>` against a copied-quote draft reports snippet-level guidance that names the failure seam clearly enough for a novice to rewrite the body correctly.

Running `python3 tools/knowledge_worker.py install <draft>` on the same copied-quote draft refuses to write the final result path and surfaces the same actionable guidance rather than only a bare validator code.

The knowledge worker prompt and hint files make the paved road explicit: inspect the current task, optionally scaffold, run `check`, then `install`, and do not treat direct writes to `out/` as complete work until `check` passes.

The narrow `.venv` test commands above pass.

If the user later approves a live rerun, an additional acceptance signal is that the absolute count of produced low-trust exact-quote packets drops relative to the March 22 benchmark run without regressing worker throughput.

## Idempotence and Recovery

This plan is additive and safe to repeat. Worker roots are regenerated, so rewriting `OUTPUT_CONTRACT.md`, `examples/`, and mirrored helper script text does not risk persistent drift. The helper `check` remains read-only. The helper `install` continues to be the only step that writes to the final result path.

If representative sample selection cannot safely find a real knowledge-like chunk in a worker root, fall back to one deterministic built-in example pair rather than silently using the first task row again. If richer diagnostics start duplicating validator logic, collapse them back into simple rendering of repo-owned validation metadata instead of maintaining a second semantic classifier.
