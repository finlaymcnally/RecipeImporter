# Replace Workspace Watchdog Shell-Shape Policing With Boundary Enforcement

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, long-lived Codex workspace workers should stop being killed just because they use normal shell patterns to do bounded local file work. A worker that reads `assigned_tasks.json`, loops over `in/*.json`, writes `out/*.json`, uses redirects, or uses a heredoc inside its sandbox should be allowed to continue. A worker should only be killed for real boundary violations such as invoking forbidden tools, escaping the workspace, or entering a pathological no-output loop.

The user-visible proof is direct. Run the existing zero-token fake-runner tests and they should show that recipe, knowledge, and line-role workspace workers still die on `python`, `git`, absolute-path escapes, and parent traversal, while the saved real-world knowledge command shapes no longer die with `watchdog_command_execution_forbidden`. The deeper product proof, to be run only with explicit user approval because it spends tokens, is that a live Salt Fat single-offline CodexFarm run no longer collapses into “watchdog killed every agent for acting like an agent.”

## Decision Log

- Decision: this plan changes the enforcement model, not merely the allowlist.
  Rationale: the repeated failure pattern is not that one more benign shell shape is missing. The problem is that shell-shape parsing itself is the wrong abstraction for a workspace worker whose job is to manipulate local files.
  Date/Author: 2026-03-20 / Codex

- Decision: keep strict structured follow-up attempts unchanged.
  Rationale: one-shot strict-JSON retry and repair calls are supposed to produce a direct answer object, not operate as local shell workers. The user complaint is specifically about long-lived workspace workers being punished for doing workspace work.
  Date/Author: 2026-03-20 / Codex

- Decision: keep telemetry classification even if enforcement becomes boundary-only.
  Rationale: reviewers still benefit from knowing whether a worker used orientation commands, shell-heavy workflows, forbidden tools, or unclassified shell snippets. That is useful observability even when it is no longer used as the primary kill switch.
  Date/Author: 2026-03-20 / Codex

- Decision: prefer a shared direct-exec helper refactor over a knowledge-only patch.
  Rationale: recipe, knowledge, and line-role all use the same workspace-worker transport. If knowledge alone stops policing shell shape but recipe and line-role still inherit the old rule, the same design failure will just reappear in a different stage.
  Date/Author: 2026-03-20 / Codex

## Outcomes & Retrospective

This plan is not implemented yet. The intended outcome is that workspace-worker main attempts are supervised by actual sandbox boundaries and liveness signals rather than by whether their shell text can be parsed into a narrow “helper” shape. Success means fewer false watchdog kills, preserved protection against real escapes, and a clearer invariant for future work: “stay in the workspace and produce outputs” instead of “only shell commands that match our current parser are legal.”

## Context and Orientation

There are three direct-exec execution modes in play for this plan.

The first mode is a long-lived workspace worker. In this repository that means a Codex session whose current working directory is an isolated worker root containing repo-written files like `worker_manifest.json`, `assigned_shards.json`, `assigned_tasks.json`, `hints/*.md`, `in/*.json`, and `out/*.json`. The worker is supposed to read those local files, decide what to do, and write output files inside `out/`. Recipe, knowledge, and line-role all now use this mode for their main attempt path.

The second mode is a structured follow-up attempt. In this repository that means a one-shot strict JSON direct-exec call used for retry or repair work after the main worker fails or produces a near miss. Those calls are intentionally stricter and should not inherit the looser workspace-worker rules.

The third concept is the watchdog. Here that means repo-owned supervision logic that receives snapshots like `elapsed_seconds`, `command_execution_count`, `last_command`, `last_command_repeat_count`, and final-message state. The watchdog can return a termination decision with a `reason_code`.

Today the central shared helper lives in `cookimport/llm/codex_exec_runner.py`. The functions `classify_workspace_worker_command(...)` and `is_tolerated_workspace_worker_command(...)` are used both for observability and for enforcement. The knowledge main worker watchdog in `cookimport/llm/codex_farm_knowledge_orchestrator.py` checks `snapshot.command_execution_count > 0` and then kills the worker if `is_tolerated_workspace_worker_command(snapshot.last_command)` returns false. That is the core design flaw this plan addresses.

This plan also touches stage-specific watchdog call sites:

- `cookimport/llm/codex_farm_knowledge_orchestrator.py`
- `cookimport/llm/codex_farm_orchestrator.py`
- `cookimport/parsing/canonical_line_roles.py`

The plan must preserve the current non-shape-based guards:

- workspace output stabilization for successful completion
- command-loop budget via `should_terminate_workspace_command_loop(...)`
- silence timeout for strict structured calls
- cohort runtime outlier kill for knowledge strict structured follow-ups
- forbidden executable families such as `python`, `node`, `git`, package managers, and network tools
- visible workspace escape attempts such as absolute paths or parent traversal

## Plan of Work

### Milestone 1: Pin down the intended invariant with tests

Start by rewriting the regression story in tests so the intended contract is obvious. In `tests/llm/test_codex_exec_runner.py`, add tests that separate “allowed by boundary policy” from “classified for telemetry.” The allowed cases should include the real saved multi-line knowledge scripts, loops over `assigned_tasks.json`, redirects to `out/*.json`, heredocs, and `>/dev/null` sinks. The denied cases should include `python`, `git`, `curl`, parent traversal, and absolute paths outside explicit null sinks.

Then add stage-level tests that prove the watchdog now kills only on real violations in workspace-worker main mode. `tests/llm/test_codex_farm_knowledge_orchestrator.py` should gain a fake worker snapshot whose last command is a shell-heavy but bounded local script and assert no immediate `watchdog_command_execution_forbidden` kill. `tests/llm/test_codex_farm_orchestrator.py` and, if needed, `tests/parsing/test_canonical_line_roles.py` should keep asserting that clearly forbidden commands still die with `watchdog_command_execution_forbidden`.

This milestone is complete when the new tests fail on the current implementation for the right reason: they should show that the code is still treating “unparseable shell shape” as an enforcement failure.

### Milestone 2: Split enforcement from telemetry in the shared direct-exec helper

Refactor `cookimport/llm/codex_exec_runner.py` so one helper answers “what kind of command was this for telemetry?” and another answers “did this command actually violate the workspace boundary?” Those are not the same question and should not share the same return contract.

The shared enforcement helper should be explicit about what can actually kill a workspace worker:

1. the executable is in the forbidden tool family,
2. the command visibly reaches outside the workspace through absolute paths or parent traversal,
3. the policy explicitly disallows an output path for that caller,
4. optionally, another clearly defined sandbox-escape rule added in this plan itself.

It should not kill merely because the shell text is complex, contains loops, pipelines, multiple commands, variable expansion, or a redirect. If the helper cannot confidently parse the shell structure but cannot detect a real escape or forbidden tool either, the enforcement result should be “no boundary violation detected.” The watchdog can still rely on loop-budget and output-stabilization logic to handle pathological but bounded behavior.

Keep or improve the current classification payload for telemetry. It is fine if the telemetry layer still calls a command `unclassified_workspace_shell_command` or `shell_script_workspace_local`; the important change is that such classifications are not automatic kill reasons for workspace main attempts.

### Milestone 3: Update workspace-worker main watchdogs to use boundary enforcement only

Once the shared helper is split, update the main workspace-worker watchdog call sites in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, `cookimport/llm/codex_farm_orchestrator.py`, and any line-role workspace-worker watchdog helper so they stop calling `is_tolerated_workspace_worker_command(...)` as the kill switch.

For main workspace-worker attempts, the command-related logic should become:

- if outputs have stabilized, terminate as completed,
- else if the last command shows a real boundary violation, terminate with `watchdog_command_execution_forbidden`,
- else if the command loop budget is exhausted without output progress, terminate with `watchdog_command_loop_without_output`,
- else continue running.

For structured follow-up attempts, keep the current strict behavior: any command execution remains off-contract there unless the stage already has a documented exception.

Update live-status payloads so reviewer artifacts remain useful. They should still record the last command, a policy/classification label, whether a real boundary violation was detected, and whether command execution occurred. The semantic change is that “command execution happened” is no longer equivalent to “kill the worker.”

### Milestone 4: Verify shared behavior across all stages and document the new invariant

Run the nearby LLM and stage-runtime tests that exercise the shared helper and each stage’s workspace main path. If any existing tests encode the old shell-shape assumption, update them so they assert the new invariant instead: kill on boundary violations, not on shell complexity.

Then update the short current-state docs, primarily `docs/10-llm/10-llm_README.md`, so the documented contract matches the new implementation. The doc text should explicitly say that workspace-worker main attempts are supervised by sandbox-boundary and liveness rules, not by a narrow helper-shell grammar.

## Validation and Acceptance

The implementation is complete when all of the following are true.

For the shared helper, tests prove that shell-heavy but bounded local workspace scripts are not automatic kill reasons, while real escape signals remain kill reasons. A command like `jq -r '.[].task_id' assigned_tasks.json | while read -r task; do ... done` must no longer fail simply because the parser dislikes the shell shape.

For the knowledge stage, a workspace-worker snapshot with bounded shell activity no longer immediately returns `watchdog_command_execution_forbidden`. The worker can continue until output stabilization, loop-budget termination, or another real failure condition.

For recipe and line-role, existing forbidden-command tests still pass. A workspace worker that runs `python -c`, `git status`, or an absolute-path escape must still be killed with the same `watchdog_command_execution_forbidden` reason.

For documentation, `docs/10-llm/10-llm_README.md` must describe the new invariant accurately: main workspace workers are governed by boundary and liveness rules, not a narrow shell grammar.

## Idempotence and Recovery

These changes should be implemented additively. The tests come first, so the current bad behavior is pinned before the shared helper is changed. If the first refactor becomes too permissive, narrow only the boundary-violation detector; do not reintroduce generic shell-shape policing under a different name.

If a stage-specific call site behaves differently after the shared helper change, prefer fixing the stage contract explicitly rather than forking the shared helper into another ad hoc allowlist. The whole point of this plan is to replace repeated per-stage “relax it again” patches with one stable enforcement rule.

Do not delete the March 20 investigation artifacts. They are the best evidence for what “real bounded shell work” looks like in this repository.
