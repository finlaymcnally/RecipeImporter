# Single-File Worker Self-Rescue Tools

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

This plan intentionally excludes watchdog changes. Watchdog termination thresholds, warning rules, and supervisor kill logic are being handled separately and must not be folded into this work. This plan is only about helping a workspace worker recover before watchdog policy becomes relevant: better repo-owned tools, better task-file guidance, and one bounded fresh-session restart path when the current worker session has preserved useful state but exhausted itself.

## Purpose / Big Picture

After this change, a single-file Codex worker that gets confused should have a better escape hatch than improvising a giant shell script. The worker should be able to inspect its current state with a repo-owned status command, apply answers through a tiny repo-owned edit helper instead of raw JSON surgery, learn the allowed answer surface directly from `task.json`, and, if the current session still fails after preserving useful edits, get one deterministic fresh-session restart with the same workspace state instead of piling more turns onto an already bloated context.

The user-visible proof is concrete. Focused tests should show recipe, knowledge, and canonical line-role workers exposing a readable status surface, carrying richer task-file metadata, and recovering from a first exhausted session by relaunching one new worker session against the existing `task.json` and hidden same-session state. No live Codex benchmark is required for acceptance, and no watchdog behavior changes are part of this plan.

## Surprises & Discoveries

- Observation: The current repo already owns good deterministic repair logic; the worker simply has too much room to avoid it.
  Evidence: `cookimport/parsing/canonical_line_roles/same_session_handoff.py` already rewrites `task.json` into `repair_required` or `completed`, and the prompt already instructs the worker to run it, but the failed worker still chose a large inline Python heredoc instead.

- Observation: The task file is still too bare as an interactive tool surface.
  Evidence: the current shared builder in `cookimport/llm/editable_task_file.py` exposes `schema_version`, `mode`, `editable_json_pointers`, and `units`, but not allowed label enums, helper commands, or a plain-language “what do I do next” field.

- Observation: The failed line-role session had enough repo-visible state to be salvageable without a watchdog invention.
  Evidence: the worker had a preserved `task.json`, a hidden same-session state file, a visible final agent message, and no expected output file. The lost part was session headroom, not workspace evidence.

- Observation: From the worker seat, vague helper descriptions are not enough. The useful difference is copy-paste-ready commands, stable JSON field names, and a concrete “run this next” string that works from the workspace root without extra flags.
  Evidence: the current prompts name one helper command, but the current task-file schema does not surface allowed answer keys, example answer shapes, or a status/diagnosis command that the worker can run immediately when uncertain.

- Observation: Knowledge blank-answer scaffolds need semantic emptiness rather than plain non-empty-object detection.
  Evidence: the initial classification `answer` object is structurally present but intentionally blank (`None` fields plus empty grounding), so status/doctor reporting only became useful after the shared task-file summary learned to ignore all-null / all-empty answer payloads.

- Observation: Recipe hard-boundary failures still produce a usable stage result tree, but not every happy-path worker sidecar is guaranteed to exist after that fail-closed path.
  Evidence: the negative fresh-session recipe test could inspect `status.json` and phase artifacts after a watchdog-style failure, but the convenience fixture that expected the happy-path `worker_manifest.json` sidecar broke on that path and had to use a lower-level setup.

- Observation: initial implementation can still miss worker-UX acceptance details even when the big runtime tests pass.
  Evidence: post-implementation review found three specific gaps: the helper CLIs still required `--state-path` or env vars despite the zero-argument goal, nested `{"answers_by_unit_id": ...}` mapping files were parsed as a fake unit id named `answers_by_unit_id`, and repair-mode rewrites dropped `helper_commands` / `next_action` / `answer_schema`.

## Context And Orientation

This repository uses a “single-file workspace worker” contract for the main recipe, knowledge, and canonical line-role LLM stages. In plain language, the repo writes one visible `task.json` file into the worker’s sterile workspace. That file contains immutable evidence and empty editable `answer` slots. The worker is supposed to edit only those answer fields, save the same file, and then run one repo-owned “same-session handoff” helper that validates the edit and either finishes the outputs, rewrites the task file into a repair form, or advances to the next deterministic stage step.

The shared task-file utilities live in `cookimport/llm/editable_task_file.py`. The worker transport and mirrored sterile workspace setup live in `cookimport/llm/codex_exec_runner.py`. The canonical line-role task-file builder and workspace prompt live in `cookimport/parsing/canonical_line_roles/runtime.py` and `cookimport/parsing/canonical_line_roles/planning.py`. The line-role same-session helper lives in `cookimport/parsing/canonical_line_roles/same_session_handoff.py`. The recipe equivalents live in `cookimport/llm/recipe_stage_shared.py` and `cookimport/llm/recipe_same_session_handoff.py`. The knowledge equivalents live in `cookimport/llm/knowledge_stage/workspace_run.py` and `cookimport/llm/knowledge_same_session_handoff.py`.

“Fresh-session recovery” in this plan means the repo preserves the existing workspace files and hidden same-session state, then launches one new Codex worker session against that same workspace root after the prior session exhausted itself. It does not mean changing watchdog thresholds, adding hidden queue control, or letting the same session continue indefinitely.

One detail matters for implementers: the live worker can run repo-owned helper modules from the sterile workspace because the runner already mirrors the necessary helper import surface and injects the hidden same-session state path by environment variable. That means the commands named in prompts and `task.json` must be literal, ready-to-run `python3 -m ...` commands that succeed from the workspace root without the worker needing to know where `_repo_control` lives.

## Plan Of Work

First, add a small shared repo-owned helper CLI in `cookimport/llm`, for example `cookimport/llm/editable_task_file_tools.py`. This helper must not perform semantic inference. It only reads and writes `task.json` in bounded ways that are already allowed by the shared contract. Keep the surface tiny and explicit. A good target shape is one non-mutating `summary` command and one mutating `apply` command behind `python3 -m cookimport.llm.editable_task_file_tools ...`. The summary command should read `./task.json` by default and report the current `stage_key`, `mode`, unit counts, answered-unit counts, editable pointers, and any stage-provided guidance metadata. The apply command should also default to `./task.json` and accept either one `--unit-id` plus one `--answer-json` payload or a small `--answers-file` mapping. It should reject unknown unit ids, malformed JSON, and attempts to touch immutable fields. From the worker perspective, the important property is that the helper is copy-pasteable from the prompt without any repo spelunking.

Second, extend the existing stage-owned same-session handoff modules with `--status` and `--doctor` modes. Both commands must be read-only and both must work from the workspace root with no extra arguments beyond the current defaults. `--status` should print a compact JSON summary of the current workspace state: current `task.json` mode, answered and unanswered counts, whether the hidden same-session state is marked complete, whether expected outputs already exist, the current stage key for knowledge, and a `recommended_command` plus `next_action` string that tells the worker exactly what to do next. `--doctor` should also be read-only, but it should attempt lightweight deterministic checks and surface a stable `diagnosis_code`, a short plain-language diagnosis such as “you edited answers but did not rerun the helper” or “repair mode is active; reopen the rewritten task file and fix the named units,” and the same `recommended_command` field. Do this in `cookimport/parsing/canonical_line_roles/same_session_handoff.py`, `cookimport/llm/recipe_same_session_handoff.py`, and `cookimport/llm/knowledge_same_session_handoff.py`.

Third, enrich the shared editable task-file schema in `cookimport/llm/editable_task_file.py`. The current top-level shape is too minimal. Add optional repo-authored metadata fields that help the worker stay inside the intended contract without inventing semantics. At minimum, add a stage-agnostic `answer_schema` payload describing allowed keys, required fields, and enum values; a `workspace_help` payload listing the exact repo-owned helper commands for this workspace; and a `next_step` string that explains the immediate happy-path action. `workspace_help` should carry literal command strings, not symbolic labels, because the worker should be able to paste them directly. `answer_schema` should also include one compact `example_answer` object so the worker can see the intended shape without reverse-engineering the validator. Then update the stage-specific task-file builders in `cookimport/parsing/canonical_line_roles/runtime.py`, `cookimport/llm/recipe_stage_shared.py`, and `cookimport/llm/knowledge_stage/workspace_run.py` to populate those fields with stage-correct content. For line-role specifically, include the allowed label enums, the allowed `exclusion_reason` enums, and the rule that `answer.exclusion_reason` is valid only with `NONRECIPE_EXCLUDE`. For recipe and knowledge, surface the allowed compact answer keys, the current stage key when relevant, and any stage-specific mode transitions already enforced by deterministic validation.

Fourth, add one bounded fresh-session recovery path in the stage runtimes, not in the watchdog. The stage-owned runtime should inspect the finished run result plus current workspace state after a worker session exits. If the session is incomplete but useful state remains, the repo should relaunch one fresh worker session against the same workspace root with a shorter “resume from current task file” prompt. The bounded retry should fire only once per happy-path assignment and should be limited to cases where the previous session left recoverable state. For this plan, “recoverable state” must mean all of the following are true: `task.json` still exists and parses; the hidden same-session state file exists and is not marked completed; the assignment has not already spent its fresh-session budget; the session did not end for a hard boundary violation that should fail closed immediately; and repo-visible evidence shows the new session has something actionable to resume, such as answered units already saved, a repair rewrite already present, or an incomplete same-session stage transition with a valid next action. The fresh-session resume prompt should explicitly say “start by running the repo-owned `--status` command, then follow `recommended_command`.” The initial implementation should be stage-local in `cookimport/parsing/canonical_line_roles/runtime.py`, `cookimport/llm/recipe_stage_shared.py`, and `cookimport/llm/knowledge_stage/workspace_run.py` so each stage can decide what “recoverable state” means using its own outputs and same-session state payload. Do not add a generic infinite rerun loop to `cookimport/llm/codex_exec_runner.py`.

Fifth, tighten the worker prompts so they advertise the new tools explicitly and downgrade improvisational scripting to a last resort the worker should not need. The live line-role prompt owner is `cookimport/parsing/canonical_line_roles/planning.py`. The recipe and knowledge prompt owners are `cookimport/llm/recipe_stage_shared.py` and `cookimport/llm/knowledge_stage/recovery.py` or `workspace_run.py`, depending on where the live worker prompt is built. The prompt changes should name the new `--status`, `--doctor`, and shared answer-apply helper commands directly and should say that if the worker feels uncertain, it must run the repo-owned status or doctor command before writing a custom shell script. The wording should remain calm and positive, but more prescriptive than the current “prefer narrow helpers” language. The resume prompt for a fresh-session recovery should be even shorter than the initial prompt and should assume the worker is resuming existing state rather than starting from a blank assignment.

Sixth, update the short docs that describe the live worker contract. `docs/10-llm/10-llm_README.md` and `cookimport/parsing/canonical_line_roles/README.md` must describe the new repo-owned self-help tools and the existence of one bounded fresh-session recovery attempt that is distinct from watchdog policy. If recipe and knowledge have equivalent short local notes, update them too.

## Validation And Acceptance

This plan is accepted only when the repository proves that the worker can help itself without changing semantic authority boundaries.

The first acceptance criterion is worker self-help visibility. In recipe, knowledge, and canonical line-role workspaces, a repo-owned `--status` command must print enough information for a worker or human to tell whether the workspace is still in initial mode, repair mode, or completed mode, and what command should be run next. A repo-owned `--doctor` command must give a compact diagnosis without mutating the workspace.
These commands must work from the workspace root without requiring the worker to supply hidden state-file paths manually.

The second acceptance criterion is script-free answer application. A focused test must prove that the shared helper can update `task.json` answer payloads by `unit_id` or a compact mapping file without changing immutable evidence fields. This helper must remain purely deterministic and must not infer missing labels.

The third acceptance criterion is richer task-file guidance. Freshly written recipe, knowledge, and line-role task files must include visible repo-authored guidance metadata for allowed answer shape and next-step commands. Focused tests should assert that those keys are present and stage-correct.
This acceptance criterion is not met by enum dumps alone. The task file must include copy-paste-ready helper commands and at least one stage-correct example answer object.

The fourth acceptance criterion is bounded fresh-session recovery. For each touched stage, a focused runtime test must show one failed or exhausted worker session leaving useful workspace state, then one new worker session starting against the same workspace root and completing successfully. The runtime must record that this was a fresh-session recovery, and it must not loop beyond one extra happy-path session.
Focused tests must also show the negative case: hard boundary failures or already-spent recovery budgets do not trigger a fresh-session restart.

The fifth acceptance criterion is preserved authority boundaries. Deterministic code must still own validation, repair rewrites, and final output expansion. The new tools may summarize state and apply explicit answer payloads, but they must not invent semantic decisions or overwrite a model answer with deterministic guesses.

The sixth acceptance criterion is documentation honesty. `docs/10-llm/10-llm_README.md` and `cookimport/parsing/canonical_line_roles/README.md` must describe the new worker self-help commands and bounded fresh-session recovery behavior, explicitly noting that watchdog policy is separate.

## Idempotence And Recovery

The new `--status` and `--doctor` commands must be safe to run repeatedly. They are read-only and should return stable summaries for a stable workspace. The shared answer-apply helper must be idempotent when given the same payload twice. Applying the same answer mapping twice should leave `task.json` unchanged on the second run.

The fresh-session recovery path must also be bounded and safe to retry. The runtime should record whether the recovery session has already been spent for a given worker assignment, ideally in repo-owned state under the worker root, so a rerun of the same stage does not silently create unbounded extra sessions.

If implementation stops halfway, do not ship prompts that mention helper commands that do not exist yet. The safe checkpoint order is: helper CLI and tests first, task-file metadata second, stage-local status and doctor modes third, stage-local fresh-session retry fourth, then prompt and doc updates last.

## Artifacts And Notes

The intended worker story after this plan should look like this in spirit:

    worker opens task.json
    worker runs a repo-owned status command if uncertain
    worker edits answers directly or uses a tiny repo-owned apply helper
    worker runs the repo-owned same-session handoff
    helper returns completed or repair_required
    if the first session exhausts itself after preserving state, the repo starts one fresh worker session
    fresh session resumes from the same task.json and hidden state instead of from a bloated prior thread

The status JSON should stay short and plain. A good shape is:

    {
      "stage_key": "line_role",
      "mode": "repair",
      "answered_units": 12,
      "total_units": 14,
      "same_session_completed": false,
      "expected_outputs_present": 0,
      "expected_outputs_total": 1,
      "recommended_command": "python3 -m cookimport.parsing.canonical_line_roles.same_session_handoff",
      "next_action": "fix the named repair units and rerun same_session_handoff"
    }

The doctor JSON should also stay short and operational. A good shape is:

    {
      "stage_key": "recipe_refine",
      "mode": "initial",
      "diagnosis_code": "answers_present_helper_not_run",
      "message": "task.json contains edited answers but the same-session helper has not produced outputs yet",
      "recommended_command": "python3 -m cookimport.llm.recipe_same_session_handoff",
      "same_session_completed": false
    }

The enriched worker-visible task file should help at a glance. A good shape is:

    {
      "answer_schema": {
        "required_fields": ["label"],
        "properties": {
          "label": {
            "enum": ["INGREDIENT_LINE", "INSTRUCTION_LINE", "TIME_LINE"]
          }
        },
        "example_answer": {
          "label": "INGREDIENT_LINE"
        }
      },
      "workspace_help": {
        "status_command": "python3 -m cookimport.parsing.canonical_line_roles.same_session_handoff --status",
        "doctor_command": "python3 -m cookimport.parsing.canonical_line_roles.same_session_handoff --doctor",
        "same_session_handoff_command": "python3 -m cookimport.parsing.canonical_line_roles.same_session_handoff"
      },
      "next_step": "Edit only /units/*/answer, save task.json, then run the same-session handoff command."
    }

The fresh-session telemetry should also stay explicit. A good shape is:

    {

## Interfaces And Dependencies

The shared task-file helper introduced by this plan should live in a reusable `cookimport/llm` module, for example `cookimport/llm/editable_task_file_tools.py`, and should expose a small CLI through `python3 -m ...`. Keep the public surface intentionally tiny: one summary command and one answer-apply command are enough.
At the module level, the implementation should expose small reusable helpers as well, for example a `summarize_task_file(...)` function and an `apply_answers(...)` function, so stage-local tests do not have to shell out just to verify deterministic behavior.

The shared editable task-file builder in `cookimport/llm/editable_task_file.py` must gain optional metadata fields that are safe for every stage. At the end of this plan, `build_task_file(...)` should be able to accept or compose:

- an `answer_schema` mapping that describes allowed answer keys and stage-specific enum values
- a `helper_commands` mapping that lists the exact helper commands the worker may run
- a `next_action` string that explains the immediate happy-path action
- an `example_answers` list inside `answer_schema` that shows one valid answer object for the current stage

The stage-owned same-session helper entrypoints that must expose `--status` and `--doctor` are:

- `cookimport/parsing/canonical_line_roles/same_session_handoff.py`
- `cookimport/llm/recipe_same_session_handoff.py`
- `cookimport/llm/knowledge_same_session_handoff.py`

The stage-owned runtimes that must implement the bounded fresh-session restart are:

- `cookimport/parsing/canonical_line_roles/runtime.py`
- `cookimport/llm/recipe_stage_shared.py`
- `cookimport/llm/knowledge_stage/workspace_run.py`

`cookimport/llm/codex_exec_runner.py` remains the transport seam, but this plan should not turn it into a generic autonomous retry engine. The stage runtimes remain responsible for deciding whether the saved workspace state is meaningful enough to justify one fresh-session relaunch.
