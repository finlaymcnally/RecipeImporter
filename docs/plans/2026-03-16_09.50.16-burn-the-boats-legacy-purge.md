# Purge Remaining Legacy Refactor Seams

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan follows the already-completed refactor plans at `docs/plans/phase1-observability-first.md`, `docs/plans/phase2-label-source-of-truth.md`, `docs/plans/phase3-collapse-recipe-reasoning.md`, `docs/plans/phase4-shrink-knowledge-pipeline.md`, and the gap review in `docs/plans/2026-03-16_09.50.00-refactor-gap-closure.md`. Those plans got the main runtime onto the new architecture. This plan finishes the deletion work they still left behind.

## Purpose / Big Picture

After this change, the repository should stop carrying dead or compatibility-only code for the old recipe pipeline architecture. A new contributor inspecting the code should see one story only: label-first runtime authority, one single-correction recipe LLM stage, Stage 7 non-recipe authority, and semantic observability/reporting that matches that runtime. They should not find unreachable pass1/pass2/pass3 code, benchmark renderers that still teach `schemaorg/final` or `merged_repair` as the current product topology, or load-path aliases that preserve `codex-farm-3pass-v1` and `codex-farm-2stage-repair-v1` as supported recipe pipeline values.

The user-visible proof is equally concrete. `cookimport stage` should still run and write the same modern semantic artifacts. `stage_observability.json`, benchmark bundle rendering, follow-up packet generation, CLI help text, and current-behavior docs should all describe only the new semantic stage trio `build_intermediate_det`, `recipe_llm_correct_and_link`, and `build_final_recipe`. Old run statistics and historical output folders remain on disk, but production code no longer keeps legacy recipe topology compatibility alive just to read them.

This is a burn-the-boats cleanup. The only historical preservation carveout is old run statistics and history artifacts that the user wants to keep for trend viewing. This plan does not delete those files. It deletes code that preserves old recipe pipeline behavior or old topology vocabulary as if it were still a live product contract.

## Context and Orientation

This repository now has two different realities.

The first reality is the main runtime. `cookimport/staging/import_session.py` builds authoritative labels, groups recipe spans, optionally runs the single-correction recipe LLM path, builds Stage 7 non-recipe ownership, and writes modern semantic artifacts such as `label_det`, `group_recipe_spans`, `08_nonrecipe_spans.json`, `09_knowledge_outputs.json`, and `stage_observability.json`. That is the architecture the earlier refactor plans intended.

The second reality is residual code and tooling vocabulary from the pre-refactor recipe pipeline. In this plan, “legacy recipe pipeline code” means code whose purpose is any of the following:

maintaining pass1/pass2/pass3 or merged-repair execution logic as live code,
normalizing old recipe pipeline ids such as `codex-farm-3pass-v1` or `codex-farm-2stage-repair-v1`,
teaching old recipe stage labels such as `schemaorg`, `final`, or `merged_repair` as current product truth in reviewer outputs,
or resolving pass-slot recipe artifact names in current-format follow-up tooling.

The critical files are:

`cookimport/llm/codex_farm_orchestrator.py`, which still contains the dead pre-refactor recipe orchestrator below the new early return.

`cookimport/config/run_settings.py` and `cookimport/labelstudio/ingest.py`, which still normalize legacy recipe pipeline ids on load.

`cookimport/runs/stage_observability.py`, which still knows how to map old recipe pipeline ids to legacy stage keys and still defines the legacy recipe LLM stage family.

`cookimport/bench/upload_bundle_v1_existing_output.py`, `cookimport/bench/upload_bundle_v1_render.py`, and `cookimport/bench/followup_bundle.py`, which still preserve old recipe topology rendering and pass-slot artifact naming.

The main non-goal is analytics history deletion. Old stats files under `.history` and similar history outputs stay on disk. This plan only touches code when that code exists to preserve old recipe-pipeline behavior or vocabulary.

## Plan of Work

Start with the dead recipe orchestrator code because it is the cleanest deletion. `cookimport/llm/codex_farm_orchestrator.py` already has one true execution entrypoint. Delete the unreachable pass1/pass2/pass3 body below the early return in `run_codex_farm_recipe_pipeline(...)`. Then delete any helper functions, dataclass fields, constants, and tests that exist only for that removed body. The file should read like a single-correction orchestrator, not like an archaeological site with a modern shim at the top.

Next, remove legacy recipe pipeline ID normalization from runtime validation. `cookimport/config/run_settings.py` should only accept `off` and `codex-farm-single-correction-v1`. `cookimport/labelstudio/ingest.py` should stop translating old values on load. Update any CLI, staging, Label Studio, or benchmark tests that still pass legacy recipe pipeline ids as supported inputs. If older saved payloads become invalid after this deletion, that is acceptable under this plan. We are not preserving old recipe-pipeline settings files as a compatibility target.

Then cut the legacy topology out of observability and benchmark surfaces. `cookimport/runs/stage_observability.py` should stop defining legacy recipe stage families for current codepaths. If stage observability still needs a recipe-stage helper, it should return only the semantic single-correction trio. Remove old branches that emit `chunking`, `schemaorg`, `merged_repair`, or `final` as recipe stage keys.

After that, rewrite the benchmark and follow-up surfaces around the semantic trio. `cookimport/bench/upload_bundle_v1_existing_output.py` should build recipe topology context from `build_intermediate_det`, `recipe_llm_correct_and_link`, and `build_final_recipe` only. `cookimport/bench/upload_bundle_v1_render.py` should stop mapping metrics back onto `pass2` or `pass3` vocabulary. `cookimport/bench/followup_bundle.py` should stop resolving pass-slot recipe artifact labels such as `pass1_input`, `pass2_output`, and `pass3_output`, and should stop falling back to `pass4_knowledge_manifest.json` for current helper behavior. If a stats-only historical carveout is needed, keep it out of these product-facing helpers.

Finally, clean tests and docs in the same change. Tests that still encode `codex-farm-3pass-v1`, `codex-farm-2stage-repair-v1`, `recipe.schemaorg.compact.v1`, `recipe.final.compact.v1`, or `schemaorg/final/merged_repair` as supported current behavior should be rewritten or deleted. Current-behavior docs should stop documenting any recipe pipeline compatibility. Historical notes can remain in logs or archived reports, but READMEs should describe only the live architecture.

## Milestones

### Milestone 1: Delete dead recipe orchestrator code

At the end of this milestone, `cookimport/llm/codex_farm_orchestrator.py` contains only the single-correction recipe path and helper machinery that the single-correction path still truly uses.

Acceptance for this milestone is a focused test slice showing the recipe orchestrator still passes while no dead code for pass1/pass2/pass3 or merged-repair execution remains in the file.

### Milestone 2: Remove legacy recipe pipeline IDs from runtime validation

At the end of this milestone, runtime settings validation accepts only `off` and `codex-farm-single-correction-v1` for `llm_recipe_pipeline`, with no alias normalization for old recipe pipeline ids.

Update `cookimport/labelstudio/ingest.py` to use the same hard validation. Then rewrite the CLI and Label Studio tests that still pass old recipe pipeline ids as accepted inputs.

Acceptance for this milestone is that the relevant test slice passes and direct construction of `RunSettings` with legacy recipe pipeline ids fails deterministically.

### Milestone 3: Replace legacy recipe topology in observability and benchmark rendering

At the end of this milestone, current-format stage observability and benchmark bundle rendering expose only the semantic recipe stage trio.

Update `cookimport/runs/stage_observability.py` so recipe stage helpers and stage definitions no longer include the legacy recipe LLM stage family. Keep only semantic keys for current runtime recipe reporting. Then update `cookimport/bench/upload_bundle_v1_existing_output.py` and `cookimport/bench/upload_bundle_v1_render.py` so they build, store, and render recipe stage context using `build_intermediate_det`, `recipe_llm_correct_and_link`, and `build_final_recipe`.

This includes deleting metric-key translation logic that maps semantic stages back to `pass2` and `pass3`. If benchmark score payloads need different keys after this, rewrite the renderers and tests together rather than preserving a translation layer.

Acceptance for this milestone is that reviewer-facing bundle output for a new-format run contains only the semantic trio and no stage labels `schemaorg`, `final`, or `merged_repair`.

### Milestone 4: Remove follow-up and artifact-name compatibility for old recipe/pass4 vocabulary

At the end of this milestone, follow-up helpers and packet builders no longer resolve pass-slot recipe artifact names or pass4-era manifest names as part of normal behavior.

Edit `cookimport/bench/followup_bundle.py` so recipe artifact discovery uses semantic stage names and current artifact files only. Remove `pass1_input`, `pass2_input`, `pass2_output`, and `pass3_output` labels from the public helper output. Remove fallback to `pass4_knowledge_manifest.json` unless a truly isolated stats-only carveout is proven necessary, in which case that carveout must not leak into recipe or follow-up rendering.

Acceptance for this milestone is that follow-up helper tests pass with semantic artifact names only, and current helper output contains no `pass*` recipe labels.

### Milestone 5: Rewrite tests and docs to match the hard deletion

At the end of this milestone, no current-behavior test or README teaches the deleted recipe pipeline worldview as supported behavior.

Update or remove tests under `tests/cli/`, `tests/labelstudio/`, `tests/bench/`, and `tests/llm/` that still encode legacy recipe pipeline ids or old recipe stage names. Update current-behavior docs in `docs/01-architecture/01-architecture_README.md`, `docs/02-cli/02-cli_README.md`, `docs/06-label-studio/06-label-studio_README.md`, `docs/07-bench/07-bench_README.md`, and nearby folder notes such as `cookimport/llm/README.md` if needed. Historical logs and reports can mention the old pipeline, but current READMEs should not present it as a supported shape.

Acceptance for this milestone is that a grep over current-behavior docs and active test files no longer finds legacy recipe pipeline ids or old recipe topology labels except inside explicitly historical logs, archived reports, or fixture data that is no longer treated as current behavior.

### Milestone 6: Validate with focused tests, domain suites, and a real run

At the end of this milestone, the hard deletion is proven by tests and a real stage run.

Run focused slices for the LLM orchestrator, run-settings validation, CLI help, Label Studio ingest, benchmark bundle rendering, and follow-up bundle helpers. Then run the project wrappers for `llm`, `staging`, and `labelstudio`. Finally, run a deterministic real stage import on `data/input/saltfatacidheatCUTDOWN.epub` and inspect the run root. The run root should still contain the modern semantic artifacts and should not cause any reporter or helper command to fall back into old recipe stage names.

Acceptance for this milestone is passing focused slices, passing domain wrappers, and one real run whose current-format outputs are fully semantic and single-correction only.

## Idempotence and Recovery

This work is safe to rerun because it is deletion-oriented and the validation commands are repeatable.

The real risk is deleting more than intended from analytics history support. If a failure appears only in history/statistics views, first decide whether that code is truly stats-only. If it is, restore only the narrowest reader seam needed for historical trend artifacts. Do not restore recipe-pipeline aliases, pass-slot benchmark rendering, or old follow-up artifact naming as part of that recovery.

If a real stage run fails after the deletion, prefer fixing the modern semantic path rather than reintroducing old recipe pipeline code. The purpose of this plan is to force the repository to stay honest about which architecture it still supports.
