# Recipe Boundary Rejects Incoherent Spans Back To Nonrecipe

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan is intentionally about deterministic machinery only. It does not change the line-role labeling policy, prompt wording, model choice, or Codex behavior that produced the bad labels in the first place. The user stated that labeling quality is being fixed separately; this plan only changes what the recipe stage does after it receives those labels.

## Exact User Intention

The exact user request that this plan must preserve is:

> "we need to make it so the recipe stage catches these bad outside recipe span labels and then "rejects" them so the go to the 3rd nonrecipe/knowledge step. any/all recipe blocks should go to recipe step. it should deterministically reject all blocks that aren't in coherent recipe shapes I.e. a collection of nearby blocks that have at least RECIPE_TITLE+RECIPE_INSTRUCTIONS+RECIPE_INGRIDIENTS + any of the other kind of bonus fields like notes and stuff. i am fixing the bad labelling problem seprately. so lets just focus on machinery here"

## Interpretation

This plan interprets that request narrowly and literally.

The recipe stage is the deterministic acceptance gate for recipe ownership. A block or span should not become recipe-owned just because earlier labeling happened to call some outside-recipe heading `RECIPE_TITLE`, `HOWTO_SECTION`, `RECIPE_NOTES`, or another recipe-local label. Instead, recipe ownership should require a coherent recipe shape: a nearby span with at least one title-like block, at least one ingredient block, and at least one instruction block. `RECIPE_NOTES`, `RECIPE_VARIANT`, `HOWTO_SECTION`, `YIELD_LINE`, and `TIME_LINE` are supportive evidence only. They can help describe an accepted recipe, but they cannot rescue a span that lacks ingredient proof or instruction proof.

When a nearby cluster of recipe-local labels fails that coherence test, the deterministic recipe boundary must reject it and hand those blocks back to the nonrecipe candidate queue so the third-stage nonrecipe knowledge machinery can review them. This handoff must be `NONRECIPE_CANDIDATE`, not `NONRECIPE_EXCLUDE`. The failure being handled here is “not coherent enough to be recipe-owned,” not “obvious junk that should never reach knowledge.” In contrast, when a span does satisfy the coherent recipe shape, all of its blocks remain in recipe machinery unless recipe refine explicitly divests them later.

## Purpose / Big Picture

After this change, the deterministic recipe boundary will stop accepting weak or fake recipe spans such as chapter headings, lesson headings, title-plus-yield stubs, or other outside-recipe topic clusters that only look recipe-like because upstream labels were wrong. The user-visible outcome is that benchmark and stage runs will send those rejected clusters into the nonrecipe knowledge step instead of letting them become recipe-owned structure that later poisons scoring and output shaping.

The proof is mechanical and observable. A fixture that currently becomes an accepted recipe span because it contains a `RECIPE_TITLE` plus a weak body signal like `YIELD_LINE` or `HOWTO_SECTION` will instead become a rejected pseudo-recipe span, its blocks will normalize to `NONRECIPE_CANDIDATE`, `recipe_block_ownership.json` will not claim those blocks, and the nonrecipe route artifact will include them in the candidate queue. Real recipe spans that contain title plus ingredients plus instructions will still become recipe-owned exactly as before.

## Context and Orientation

The relevant pipeline in this repository has four deterministic-to-semantic steps.

First, `cookimport/parsing/label_source_of_truth.py` builds authoritative labeled lines and block labels. This is where labels such as `RECIPE_TITLE`, `INGREDIENT_LINE`, `INSTRUCTION_LINE`, `NONRECIPE_CANDIDATE`, and `NONRECIPE_EXCLUDE` become the current line/block truth for downstream stages.

Second, `cookimport/parsing/recipe_span_grouping.py` groups nearby recipe-local labels into candidate recipe spans. A “recipe span” in this repository means a contiguous or nearly contiguous run of nearby blocks that the deterministic recipe boundary believes belongs to one recipe. This file is the current acceptance gate. It emits accepted `RecipeSpan` objects, rejected `RecipeSpanDecision` records, and normalized block labels that either stay recipe-local or get routed back to nonrecipe.

Third, `cookimport/staging/recipe_ownership.py` turns accepted recipe spans into recipe ownership. “Recipe ownership” means a block is reserved for the recipe path and may not enter nonrecipe routing unless recipe refine later explicitly divests it.

Fourth, `cookimport/staging/nonrecipe_stage.py` and `cookimport/staging/pipeline_runtime.py` route every block that is not recipe-owned into either the nonrecipe candidate queue or deterministic nonrecipe exclusion. The third-stage semantic knowledge pass only sees the nonrecipe candidate queue.

The key current bug is not that these files lack a handoff path. The handoff path already exists. The bug is that the acceptance gate in `recipe_span_grouping.py` is too permissive. It currently treats title plus any one body-like label as enough to accept a recipe span. That allows outside-recipe headings and topic clusters to become recipe-owned even though they are not coherent recipes.

This plan changes that acceptance contract. It does not change `cookimport/parsing/canonical_line_roles/*`, prompt wording, model instructions, or Codex routing policy. Those surfaces may keep producing imperfect labels; this plan only makes the recipe stage reject incoherent clusters deterministically and pass them to nonrecipe candidate review.

## Plan of Work

The first edit belongs in `cookimport/parsing/recipe_span_grouping.py`. Replace the current acceptance test built around `_has_acceptance_recipe_body(...)` with an explicit coherent-shape test that requires all three core recipe signals inside one pending span: at least one title-like block, at least one ingredient block, and at least one instruction block. Keep the current nearby-span grouping model, including the single `NONRECIPE_CANDIDATE` bridge, unless later focused evidence shows a second change is necessary. The point of this plan is not to redesign span grouping geometry; it is to tighten the acceptance gate.

Add helper functions with names that make the new contract obvious. A novice reading the file should be able to see that acceptance is now based on title proof, ingredient proof, and instruction proof, not on vague “body” proof. The decision metadata should also become more precise. A rejected span that has title plus ingredients but no instructions should not hide behind the old generic `rejected_missing_recipe_body` story. Introduce explicit rejection reasons for missing ingredient proof and missing instruction proof, or one equally explicit combined reason that records which core recipe signals were absent. The decision notes and warnings must remain human-readable because they are the main debugging breadcrumb in `span_decisions.json`.

The second edit also belongs in `cookimport/parsing/recipe_span_grouping.py`, inside `_normalize_recipe_boundary_block_labels(...)`. Preserve the current behavior for accepted spans: recipe-local labels remain recipe-local, and bridged `NONRECIPE_CANDIDATE` rows inside an accepted coherent recipe still normalize to `RECIPE_NOTES`. Change only the rejected-span handoff behavior. When a block was part of a rejected recipeish span and it carried a recipe-local label, normalize it to `NONRECIPE_CANDIDATE` so it will enter the nonrecipe knowledge queue. This must be a hard rule of the new contract. Do not let weak recipeish spans turn into `NONRECIPE_EXCLUDE` just because the upstream label object happened to carry an exclusion reason. The recipe boundary is making an ownership rejection here, not certifying obvious junk.

The third edit is in tests. `tests/parsing/test_recipe_span_grouping.py` must stop proving the old loose contract and start proving the new one. Invert `test_recipe_boundary_from_labels_accepts_title_plus_yield_stub` so it becomes a rejection test. Rewrite `test_recipe_boundary_from_labels_splits_on_non_recipe_boundaries` so every accepted span in the fixture contains both ingredients and instructions; do not leave a title-plus-instruction stub accepted by accident. Add one focused regression for a bad outside-recipe recipeish cluster such as chapter heading plus `HOWTO_SECTION` plus yield/note support but no ingredient or instruction proof; that fixture should be rejected and routed back to `NONRECIPE_CANDIDATE`. Keep one positive test that proves a real recipe with title, ingredients, instructions, and optional note or yield remains accepted.

The fourth edit is in staging tests. `tests/staging/test_nonrecipe_stage.py` should gain a proof that blocks from a rejected recipeish span are not recipe-owned and do appear in the nonrecipe candidate queue. The test should not depend on live Codex or a full benchmark run. Build a tiny ownership fixture where the recipe boundary rejects a pseudo-recipe span, then assert that nonrecipe routing receives those indices as candidates. This proves the machinery end-to-end: deterministic reject at recipe boundary becomes semantic review opportunity at nonrecipe.

The fifth edit is documentation. Update the shortest relevant doc notes in `docs/05-staging/05-staging_readme.md` and `cookimport/parsing/README.md` so they no longer claim that title plus ingredient or instruction or yield or howto proof is sufficient. The docs should say plainly that recipe-boundary acceptance now requires title plus ingredients plus instructions, and that rejected recipeish spans route into nonrecipe candidate review rather than remaining recipe-owned.

## Validation and Acceptance

The implementation is correct when all of the following are true.

First, a cluster of nearby labels that lacks a title anchor is rejected exactly as before.

Second, a cluster with a title but no ingredient proof is rejected, records an explicit missing-core-field reason, and its recipeish blocks normalize to `NONRECIPE_CANDIDATE`.

Third, a cluster with a title and ingredients but no instruction proof is rejected, records an explicit missing-core-field reason, and its recipeish blocks normalize to `NONRECIPE_CANDIDATE`.

Fourth, a cluster with a title and instructions but no ingredient proof is rejected and routes back to nonrecipe candidate review as `NONRECIPE_CANDIDATE`, not `NONRECIPE_EXCLUDE`.

Fifth, a coherent recipe shape with title plus ingredients plus instructions is accepted, and all blocks in that accepted span remain recipe-owned unless later explicit divestment happens in recipe refine.

Sixth, a rejected pseudo-recipe span does not appear in recipe ownership, and its recipeish blocks show up in the nonrecipe candidate queue as `NONRECIPE_CANDIDATE` instead of being stranded as recipe-local labels or being thrown away as `NONRECIPE_EXCLUDE`.

The expected human-visible proof is a passing focused test suite plus clear fixture assertions that inspect accepted spans, rejected span decisions, normalized block labels, ownership, and nonrecipe routing.

The safest rollback is also local and mechanical: revert the recipe-boundary acceptance helper changes and the paired test updates together. Do not leave the repo in a mixed state where the code uses the new stricter contract but the tests still encode the old loose acceptance rule, or vice versa.

In `cookimport/parsing/recipe_span_grouping.py`, `recipe_boundary_from_labels(...)` must accept only spans that contain title proof, ingredient proof, and instruction proof. The helper names and decision metadata should make that rule obvious to a novice.

In `cookimport/staging/nonrecipe_stage.py`, no broad rewrite is required. The file should continue to assume that only unowned or explicitly divested blocks reach nonrecipe routing. The new recipe-boundary contract should make that invariant truer, not looser.
