# Fix Knowledge Task-Packet Progress Reporting In The Benchmark UI

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the non-recipe knowledge phase in a CodexFarm benchmark should report the work unit the operator actually cares about: task packets, not just top-level shards. Today the benchmark can split ten shard workers into hundreds of task packets, write outputs for those packets for many minutes, and still show `task 0/10 | running 10`. After this change, the same run should show packet progress moving forward while the long-lived workspace workers are still alive, and the worker rows should explain which shard each worker owns and how many task packets it has already produced.

The user-visible proof is simple. Re-run the Salt Fat Acid Heat single-offline CodexFarm benchmark in a no-token fake-runner mode or with a targeted progress test harness. When the knowledge phase begins, the first structured update should use the task-packet total rather than the shard total, the counter should advance before a shard fully finishes, and the rendered worker rows should show per-worker packet progress such as `saltfatacidheatcutdown.ks0009.nr (47/48 task packets)` rather than a static shard id with a global `0/10` counter.

## Surprises & Discoveries

- Observation: the current bug is not primarily in the CLI renderer. The renderer already trusts structured stage-progress payloads; the knowledge orchestrator is still emitting `task_current` and `task_total` in shard units.
  Evidence: `_notify_knowledge_progress(...)` in `cookimport/llm/codex_farm_knowledge_orchestrator.py` formats the stage message from `completed_tasks` and `total_tasks`, and the caller currently passes `total_shards`.

- Observation: the orchestrator only updates progress when a shard reaches terminal aggregation, not when a live worker writes packet outputs.
  Evidence: `_mark_shard_completed(...)` is the only progress callback path in the stage startup section around `cookimport/llm/codex_farm_knowledge_orchestrator.py:1155-1195`, and it fires after shard aggregation, not per packet.

- Observation: the supervision callback already knows how many packet outputs are present in a worker workspace while the worker is still running.
  Evidence: `_build_strict_json_watchdog_callback(...)` computes `workspace_output_present_count`, `workspace_output_expected_count`, and `workspace_output_missing_files` before it writes each `live_status.json`.

- Observation: the earlier `docs/plans/2026-03-20_12.36.02-stabilize-knowledge-runtime-progress-and-abort.md` plan fixed the eight-worker truncation problem, but it did not address the packet-vs-shard counter mismatch.
  Evidence: the current `cookimport/cli.py` already renders ten workers, but the saved `2026-03-20_21.44.20` benchmark still logged `task 0/10 | running 10` for the entire knowledge phase.

  Rationale: the user-visible bug is that the one visible counter is wrong. Reusing the existing structured counter keeps the benchmark timeseries, spinner, and tests aligned. Shard counts can move into detail lines instead of remaining the primary progress unit.

## Outcomes & Retrospective

This plan is not implemented yet. The current outcome is only a narrowed design: progress must be counted in task packets, and the live workspace supervision callback is the intended seam for packet counts while workers are still running. The main open risk is making sure those live counts stay consistent with the later packet validation/retry/repair paths so the UI never overcounts completed work.

## Context and Orientation

The affected flow starts in `cookimport/labelstudio/ingest.py`, which calls the knowledge phase as part of a benchmark or prediction run. The knowledge phase itself lives in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. In this repository, a "shard" is one larger ownership bundle assigned to one worker, while a "task packet" is a smaller unit inside that shard, usually one chunk bundle that eventually becomes one `out/<task_id>.json` file. The knowledge phase writes `task_manifest.jsonl` for the packet list and `worker_assignments.json` for the top-level shard ownership.

The benchmark UI logic lives in `cookimport/cli.py`. It does not invent knowledge-specific progress on its own. Instead, it parses structured progress payloads produced by `cookimport/core/progress_messages.py::format_stage_progress(...)`. That payload currently contains one task counter, one running-worker count, one configured-worker count, one `active_tasks` list, and a list of free-form detail lines. The benchmark timeseries rows are also derived from this same structured payload. That means the clean fix is to send better knowledge-stage progress snapshots rather than adding another ad hoc parser in the CLI.

The real-time seam for long-lived workspace workers is the knowledge watchdog callback built by `_build_strict_json_watchdog_callback(...)`. Even though its primary job is to terminate off-contract or stalled runs, it already calculates the number of expected packet output files and the number currently present in the workspace. That is the missing bridge between the worker’s real packet progress and the benchmark UI.

The saved failing run is rooted at `data/golden/benchmark-vs-golden/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/`. Its knowledge artifact root is `data/output/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-20_21.44.56/raw/llm/saltfatacidheatcutdown/knowledge/`. In that run, the stage wrote `task_manifest.jsonl` with `485` packets, but the UI used `task 0/10` because the orchestrator only reported the ten top-level shards.

## Plan of Work

The first code change is in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. Introduce a small private progress-state helper for the knowledge phase. It should hold the immutable totals for shards and task packets, plus mutable per-worker live packet counts and terminal packet counts. A novice should be able to find it near `_notify_knowledge_progress(...)` because that function is where the stage currently emits shard-based progress. Do not bury the state in several dicts spread across the worker launcher; give it a named dataclass or helper object with one method that renders a full structured progress snapshot.

That progress state should make one explicit choice: the visible task counter for the knowledge phase now means task packets, not shards. In other words, `task_current/task_total` for this stage becomes `completed_task_packets/total_task_packets`. Shard progress must still be preserved, but it moves into detail lines such as `completed shards: 3/10`, `queued task packets: 146`, and `configured workers: 10`. This keeps the benchmark renderer and timeseries honest without forcing a second counter abstraction through every caller.

The second code change is to feed live packet counts into that state while the main workspace workers are still running. Extend `_build_strict_json_watchdog_callback(...)` with one new optional progress hook that receives enough information to identify the worker and the current workspace-output counts. The callback should not own policy. It should just report facts such as `worker-010 now has 47 of 48 packet outputs present`. The knowledge orchestrator should pass this hook when it launches `runner.run_workspace_worker(...)` for knowledge workers. The hook must update the in-memory progress state under a lock and emit a refreshed structured progress snapshot only when counts actually change, so the UI and timeseries are informative without becoming a flood of duplicate rows.

The third change is to reconcile live packet counts with terminal packet outcomes after the main worker returns and as follow-up calls run. After the main worker exits, the existing task loop already evaluates each packet, optionally runs watchdog retry, optionally runs repair, and decides the packet’s terminal status. Update the progress state there as well. A packet counts as complete when the stage is done with it, even if the final outcome is invalid or missing-output. This is essential: the counter must measure work processed, not only validated successes, or failed packets will still leave the UI looking frozen.

The fourth change is to improve the worker-row labels carried through `active_tasks`. Right now the knowledge phase sends one raw shard id per worker. Replace that with a descriptive per-worker label assembled by the progress state. For a main worker still running, the label should include the shard id and live packet count, for example `saltfatacidheatcutdown.ks0009.nr (47/48 task packets)`. For a worker that is no longer in the main workspace session but is running a repair or watchdog retry, the label should say so directly, for example `saltfatacidheatcutdown.ks0007.nr repair saltfatacidheatcutdown.ks0007.nr.task-002`. This keeps the worker panel truthful without requiring a new renderer protocol.

The fifth change is test coverage. In `tests/llm/test_codex_farm_knowledge_orchestrator.py`, add a focused orchestrator test that simulates a knowledge worker whose workspace outputs appear incrementally. The acceptance is that the captured structured progress payloads begin at `0/<packet_total>`, move upward before any shard fully completes, and finish at `<packet_total>/<packet_total>`. In `tests/labelstudio/test_labelstudio_benchmark_helpers_progress.py`, add a benchmark UI rendering test that feeds one structured knowledge progress payload with a packet-based counter and descriptive worker labels, then asserts that the rendered message no longer implies `0/10` shard progress. The test should specifically check for a packet-scale counter and a worker row that includes packet counts in the label.

If the existing generic worker-panel wording in `cookimport/cli.py` still reads awkwardly once the knowledge stage starts reporting packet counts, make the smallest renderer adjustment necessary. The likely acceptable tweak is to keep the worker list logic but allow the header text to say `active workers` when the list is clearly one row per worker. Do not widen scope into a new UI framework. Only touch the renderer if the updated knowledge snapshots still read as misleading after the orchestrator fix.

After the implementation lands, update the short docs in `docs/07-bench/07-bench_README.md` and `docs/10-llm/10-llm_README.md` so they describe the new knowledge-stage packet counter truthfully. The docs should explain that knowledge progress is packet-based, while the worker rows remain worker-based and include shard ownership plus packet counts or follow-up state.

## Validation and Acceptance

The change is complete when the following behavior is true.

First, the knowledge orchestrator emits packet-based structured progress. A knowledge stage with `485` task packets must start at `task 0/485`, not `task 0/10`, even though only ten shard workers exist. The final structured payload for the phase must end at `task 485/485`.

Second, live progress must move before a shard fully finishes. In a test that simulates a running worker gradually writing `out/*.json` files, the captured progress payloads must show increasing `task_current` values while the worker session is still alive. A fix that only jumps from `0` to `49` after worker return is not sufficient.

Third, the worker rows must explain ownership and local packet progress. At least one rendered worker row in the CLI progress tests must look like a shard id plus packet counts, for example `worker 10: knowledge-shard-0010 (47/48 task packets)`, or a follow-up label that clearly says repair or watchdog retry.

Fourth, failed packets must still count as processed work once the stage has finished their retries and repairs. A test with invalid packet outputs should still advance the packet counter as those packets reach terminal failed status.

Fifth, the benchmark progress docs must match the landed behavior. After the code changes, `docs/07-bench/07-bench_README.md` and `docs/10-llm/10-llm_README.md` should describe knowledge progress as packet-based and explain the worker-row labels.

The minimum commands that must pass before calling this done are:

    . .venv/bin/activate && pytest -q tests/labelstudio/test_labelstudio_benchmark_helpers_progress.py

## Idempotence and Recovery

This change should be additive and safe to retry. The orchestrator progress state is in-memory only and should not change saved run artifacts beyond the progress payloads emitted during execution. If the live supervision hook proves too noisy, keep the hook but emit progress only when packet counts or follow-up labels actually change. If packet counts drift between the live supervision path and the later terminal packet loop, trust the terminal packet loop as the final authority and document the discrepancy in `Surprises & Discoveries` before refining the live reporting logic.

Do not broaden this work into retry-policy or prompt-quality changes. If a test failure shows that the underlying worker behavior is still broken, note it in this plan but keep this implementation restricted to truthful progress reporting. This plan is about making the UI say what the stage is doing, not about making the stage behave perfectly.

## Interfaces and Dependencies

In `cookimport/llm/codex_farm_knowledge_orchestrator.py`, define a small private progress helper near `_notify_knowledge_progress(...)`. It should expose, at minimum, one method that accepts the current phase facts and emits a fresh structured progress payload. The helper must know:

    - total_shards
    - total_task_packets
    - completed_task_packets
    - completed_shards
    - per-worker live packet counts
    - per-worker active label text

Also in `cookimport/llm/codex_farm_knowledge_orchestrator.py`, extend `_build_strict_json_watchdog_callback(...)` with one optional observer callback that receives enough data to update the knowledge progress state. Keep it private to the knowledge orchestrator; do not turn this into a generic cross-repo progress bus unless the implementation proves that to be unavoidable.

The `cookimport/core/progress_messages.py::format_stage_progress(...)` interface should remain the primary transport unless implementation evidence proves it cannot carry the needed truth. Prefer reusing:

    - `task_current/task_total` for packet counts
    - `running_workers/worker_total` for worker slots
    - `active_tasks` for descriptive per-worker row labels
    - `detail_lines` for shard totals and queued packet counts

If `cookimport/cli.py` needs a wording tweak, keep it backward-compatible. The renderer must still understand older stage-progress payloads that only know shard counts or simple worker labels.
