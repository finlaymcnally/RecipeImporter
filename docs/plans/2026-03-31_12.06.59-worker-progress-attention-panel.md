## Purpose / Big Picture

After this change, the interactive CLI progress panel should stop acting like a coarse shard counter and start acting like an operator-facing worker health view. When a recipe, knowledge, or canonical line-role worker goes quiet, starts looping helper commands, emits a final message without producing outputs, or otherwise drifts into a suspicious state, the live progress panel should surface that fact while the run is still in flight. The user-visible proof is focused: runtime progress payloads should include worker-health attention lines, the shared spinner should keep refreshing during otherwise quiet worker periods, and focused tests should show stuck-worker summaries appearing without waiting for a shard to finish.

- Observation: the runtime progress emitters are milestone-driven, not heartbeat-driven.
  Evidence: recipe and line-role only call their `_emit_progress_locked(...)` helpers on initial emission and shard completion, and knowledge progress emits on specific state transitions rather than on a periodic ticker.

- Observation: recipe already had enough natural polling to surface live worker-health changes once the summarizer existed; knowledge and canonical line-role needed explicit heartbeat threads.
  Evidence: recipe progress already re-entered its emit path while waiting on assignments, whereas the other two stages only refreshed on lifecycle transitions before this change.

## Context and Orientation

The live progress panel is owned by `cookimport/cli_support/progress.py`. Stages do not render directly to the terminal; they call a shared `progress_callback`, usually with a serialized `stage_progress` payload from `cookimport/core/progress_messages.py`. Recipe correction emits structured progress from `cookimport/llm/recipe_stage_shared.py`. Knowledge finalize emits it through `_KnowledgePhaseProgressState` in `cookimport/llm/knowledge_stage/planning.py`, with worker execution in `cookimport/llm/knowledge_stage/runtime.py` and `cookimport/llm/knowledge_stage/workspace_run.py`. Canonical line-role emits it from `cookimport/parsing/canonical_line_roles/runtime.py`.

The worker-health truth already exists in repo-owned `live_status.json` files written by watchdog callbacks. “Worker health” in this plan means non-semantic runtime signals such as whether the worker is still running, whether it has warnings, whether it has gone quiet for a suspicious amount of time, whether it emitted a final message without producing outputs, and whether repo-owned follow-up or completion work is in flight. Deterministic code must only summarize those signals; it must not infer or override semantic recipe, knowledge, or line-role answers.

## Plan of Work

Add a small shared helper module under `cookimport/llm/` that reads per-worker `live_status.json` files and produces two kinds of output: per-worker attention suffixes that can be appended to existing active-task labels, and aggregate panel detail lines such as warning counts, stalled-worker counts, and a concise “attention” summary naming the worst current examples. Keep the helper stage-agnostic by basing it on `worker_id`, `workspace_root`, and the JSON payload shape rather than on recipe- or knowledge-specific business logic.

Then wire that helper into the three workspace-worker stages. Each stage already knows its `assignments`, active labels, and follow-up counters. Extend their progress emission so it consults live worker health before building `active_tasks` and `detail_lines`. Add a short-lived heartbeat thread or ticker around the executor lifetime so `_emit_progress_locked(...)` or `progress_state.emit(...)` runs periodically while workers are active. The ticker should stop cleanly once all worker futures are finished and should not emit duplicate snapshots.

Finally, update the shared renderer in `cookimport/cli_support/progress.py` so `last_activity_at` becomes a visible line and the richer worker-health detail lines fit naturally into the boxed panel. The renderer should stay generic: it should format provided lines and timestamps, not discover worker health itself.

## Validation and Acceptance

Acceptance requires visible operator-facing behavior, not just new helper code. A recipe, knowledge, or line-role run with an active worker warning or suspicious silence must emit structured progress payloads whose `detail_lines` and `active_tasks` reflect that attention before the worker finishes. The shared CLI renderer must then show those lines in the live panel instead of silently dropping them.

The second acceptance criterion is heartbeat freshness. During a quiet in-flight worker period, the runtime must still emit enough progress updates for the operator to see a stale-worker summary change over time. This does not require spammy sub-second updates, but it does require more than “initial state” and “task completed.”

The third acceptance criterion is stable authority boundaries. The new summaries may only use deterministic runtime telemetry from repo-owned files. They must not infer missing semantic outputs or rewrite model answers.

The fourth acceptance criterion is coverage. Focused tests must prove the new structured fields appear in runtime progress payloads and that the shared renderer displays last-activity or attention text when those fields are present.

## Idempotence and Recovery

This work is safe to rerun. The new helper only reads repo-owned `live_status.json` files. Heartbeat progress emission must be duplicate-safe by reusing existing snapshot dedupe in the emitters, so a repeated ticker call should not flood callbacks with identical payloads. If implementation stops halfway, do not ship runtime emitters that depend on renderer changes that do not exist yet; the safe order is helper first, runtime emitters second, renderer third, tests last.

Because the worktree already has unrelated plan/task doc edits, recovery should avoid any destructive git operations. Re-read the touched runtime files before each patch and merge carefully with local changes rather than restoring older copies.

## Artifacts and Notes

The intended operator story is:

    run starts
    shared panel shows task/shard counts and active workers
    one worker drifts into warning or silence
    runtime heartbeat re-emits progress with an attention line such as
    "attention: worker-002 final message without output" or
    "stalled workers: 1"
    panel updates in place before the shard succeeds or fails

The new panel should stay compact. A good shape is:

    stage: non-recipe finalize
    progress: task 4/12 (33%)
    workers: 2 running, 1 completed, 3 total
    last activity: 47s ago
    stalled workers: 1
    attention: worker-002 final message without output
    worker 01: book.ks0004.nr (1/2 shards)
    worker 02: book.ks0005.nr (0/1 shards) [final message, no output]

## Interfaces and Dependencies

Keep `cookimport/core/progress_messages.py` backward-compatible. If a new helper needs more stage-progress fields, add them there carefully, but prefer reusing `detail_lines`, `active_tasks`, and `last_activity_at`.

The shared helper should accept simple filesystem-oriented inputs such as worker roots and return plain Python data structures. It should not depend on Rich or CLI-rendering code. The runtime emitters remain the owners of stage wording such as “recipe task,” “non-recipe finalize,” and “shard finalization.”
