## Structured Restatement

The line-role worker payload should stop carrying deterministic shard-level guesses and prose posture fields that pretend the repository knows fuzzy cookbook semantics. The model-facing contract should be reduced to the smallest defensible set of mechanical truth: the owned rows that need labeling, plus immediate boundary context when available. The `HOWTO_SECTION` heuristic fields must be removed entirely from the model-facing payload and from any worker-facing hint text that presents them as if they were trustworthy evidence. This cleanup should be strict rather than permissive: remove the current shard-level garnish now, and if a future case genuinely needs a specific extra cue, add that one cue later with its own justification.

## Purpose / Big Picture

After this change, a line-role worker root should be honest and compact. When someone opens `workers/*/in/*.json`, they should see the rows to label and, if present, immediate neighboring context rows. They should not see a pile of deterministic shard guesses such as `shard_summary`, `context_confidence`, `default_posture`, `flip_policy`, or book-level `HOWTO_SECTION` policy text. The repository should package evidence and enforce exact coverage, not narrate speculative interpretations of the shard.

The user-visible effect is a smaller, less misleading line-role worker contract and a cleaner prompt story. Prompt preview, live worker roots, and tests should all agree on the same narrow model-facing input shape. The only information that should survive in the payload is information the repo actually knows without fuzzy interpretation. This should make future line-role behavior easier to reason about and easier to debug, because any remaining bad model choice will come from the rows and their local context, not from prompt garnish pretending to be evidence.

- Observation: the current line-role payload has two separate kinds of non-row metadata mixed together: mechanically defensible context and heuristic prose posture.
  Evidence: `docs/04-parsing/04-parsing_readme.md` documents a payload shape that mixes `rows` with `shard_summary`, `default_posture`, `flip_policy`, `example_files`, and `howto_section_*`.

- Observation: the `HOWTO_SECTION` fields are not static product rules. They are deterministic book-level guesses derived from `_has_recipe_local_howto_support(...)`.
  Evidence: `cookimport/parsing/canonical_line_roles/planning.py` computes `howto_section_availability`, `howto_section_evidence_count`, and `howto_section_policy` in `_build_line_role_book_context(...)`.

- Observation: the current shard-context fields are all deterministic and mostly prose wrappers around `_classify_line_role_shard_mode(...)`.
  Evidence: `cookimport/parsing/canonical_line_roles/policy.py` builds `shard_mode`, `context_confidence`, `shard_summary`, `default_posture`, `strong_signals`, `weak_signals`, `flip_policy`, and `example_files` from deterministic logic and canned string lists.

- Observation: the same garnish influences both the file-backed worker contract and the inline prompt-preview path.
  Evidence: `cookimport/parsing/canonical_line_roles/policy.py` adds the context fields to `build_line_role_model_input_payload(...)`, and `cookimport/llm/canonical_line_role_prompt.py` reads and renders them into the inline prompt text.

- Observation: worker hints duplicate much of the same deterministic interpretation in prose.
  Evidence: `cookimport/parsing/canonical_line_roles/planning.py` turns `shard_summary`, `context_confidence`, `default_posture`, `flip_policy`, `strong_signals`, `weak_signals`, and `example_files` into the hint sections `Shard interpretation` and `Decision policy`.

- Observation: once the payload and hint consumers were trimmed, the shard/book-context helper stack had no remaining production callers.
  Evidence: after the payload and hint edits, only tests referenced `_build_line_role_book_context(...)`, `_build_line_role_shard_context(...)`, and `_classify_line_role_shard_mode(...)`, so deleting those helpers kept the code truthful and simpler than leaving compatibility stubs.

  Evidence: `llm_pipelines/prompts/line-role.canonical.v1.prompt.md` already described `{"v":1,"shard_id":...,"context_before_rows":[...],"rows":[...],"context_after_rows":[...]}` while `cookimport/llm/canonical_line_role_prompt.py` was still rendering removed shard-context sections.

## Outcomes & Retrospective

This plan is now implemented. The live line-role worker contract is smaller and more honest: `workers/*/in/*.json` now carries only `v`, `shard_id`, `rows`, and optional immediate boundary rows. The worker hint remains available, but it is now an orientation sidecar instead of a shard-specific policy memo. Prompt preview and live worker payloads are aligned on the same narrow transport.

The main lesson from implementation is that most of the complexity here was drift, not true product need. Once the repo stopped trying to narrate what a shard “really is,” the code got simpler in three places at once: payload construction, hint writing, and prompt rendering. The focused validation path stayed small and sufficient: `pytest tests/parsing/test_canonical_line_roles.py -q` passed in `.venv`, and the touched modules compiled cleanly with `python -m py_compile`.

## Context and Orientation

The line-role stage is the large-language-model pass that assigns recipe-structure labels to ordered atomic lines. In this repository, “atomic line” means one stable line-like unit from the source cookbook that later stages can refer back to by `atomic_index`. The line-role stage is not supposed to decide final semantic `KNOWLEDGE` outside recipes. It is supposed to identify recipe structure and obvious review exclusion, then pass the remaining outside-recipe prose onward.

The relevant implementation files are:

- `cookimport/parsing/canonical_line_roles/policy.py`, which builds the model-facing line-role payload and currently adds shard-context fields such as `shard_mode`, `context_confidence`, `shard_summary`, `default_posture`, `strong_signals`, `weak_signals`, `flip_policy`, `example_files`, and `howto_section_*`.
- `cookimport/parsing/canonical_line_roles/planning.py`, which computes the current book-level `HOWTO_SECTION` posture, writes worker hints, and threads the model payload into worker roots.
- `cookimport/llm/canonical_line_role_prompt.py`, which reconstructs the inline prompt from the same payload fields for preview and some non-workspace flows.
- `tests/parsing/test_canonical_line_roles.py`, which currently asserts the presence and content of many of these fields.
- `docs/04-parsing/04-parsing_readme.md` and `docs/10-llm/10-llm_README.md`, which currently describe the payload as carrying this shard-level context.

The mechanically trustworthy information in the current payload is:

- `v`, the payload schema version
- `shard_id`, the stable identifier for the current shard
- `rows`, the actual ordered rows to label
- `context_before_rows` and `context_after_rows`, the immediate neighboring rows outside the owned shard when available

Everything else is derived from deterministic heuristics and canned text. Even when some of it looks superficially helpful, it is not direct evidence about the current shard in the same sense that `rows` and immediate neighbors are. This plan therefore removes that garnish first and refuses to preserve it by default in other worker-facing surfaces.

The current `HOWTO_SECTION` fields are especially important to remove because they are built from book-level deterministic guesses in `_build_line_role_book_context(...)`. The function counts how many candidates appear to have recipe-local how-to support and then emits prose posture such as “sparse” or “available.” That is exactly the sort of deterministic cookbook judgment the user does not want the repo to project into the model-facing surface.

The current shard-mode fields are also deterministic. `_classify_line_role_shard_mode(...)` counts deterministic labels, rule tags, prose-like rows, front-matter headings, title-like patterns, and span-status patterns to assign one of a small set of modes such as `front_matter_navigation` or `lesson_prose`, plus a coarse confidence label. This may be useful for debugging or analytics, but it is not mechanical truth and should not be confused with it.

This plan assumes the following design boundary and preserves it throughout:

- Deterministic code may package source evidence, immediate boundary context, exact ordering, stable identifiers, validation, and output normalization.
- Deterministic code may not present fuzzy cookbook interpretations as if they were authoritative cues the worker should trust.
- The model should see the rows to label and enough direct local context to make its own decision.

## Milestones

## Milestone 1: Define and implement the minimal model-facing line-role payload

At the end of this milestone, the authoritative line-role worker input under `workers/*/in/*.json` will contain only the smallest defensible set of model-facing fields. The target contract is:

    {"v":1,"shard_id":"...","context_before_rows":[...],"rows":[...],"context_after_rows":[...]}

The `context_before_rows` and `context_after_rows` keys are optional and should appear only when immediate neighbors exist. No shard-level prose posture or book-level heuristic fields should remain in the model-facing payload.

Implement this in `cookimport/parsing/canonical_line_roles/policy.py`. Remove `shard_mode`, `context_confidence`, `shard_summary`, `default_posture`, `strong_signals`, `weak_signals`, `flip_policy`, `example_files`, `howto_section_availability`, `howto_section_evidence_count`, and `howto_section_policy` from `build_line_role_model_input_payload(...)`. Keep the row serialization and immediate boundary context intact. If a helper function now exists only to feed removed payload fields, delete it or narrow it so the code does not keep computing dead context.

Acceptance for this milestone is a focused test that opens a generated worker input payload and proves those removed keys are absent, while `rows` and immediate boundary context remain correct and ordered.

## Milestone 2: Remove `HOWTO_SECTION` guesswork from worker-facing hint prose

At the end of this milestone, worker hints should stop presenting book-level `HOWTO_SECTION` availability or policy as if it were trustworthy evidence about the current shard. The hint file must not say things like “`HOWTO_SECTION` appears sparse in this book” or carry deterministic `howto_section_evidence_count` style reasoning. More broadly, this milestone should remove the current shard-garnish sections that merely restate deterministic interpretation. The only worker-facing guidance that should survive is truly minimal static contract help that is not pretending to describe the current shard.

Implement this in `cookimport/parsing/canonical_line_roles/planning.py`. Remove the book-level `HOWTO_SECTION` posture from the hint sections. Also remove deterministic shard-interpretation sections that only exist to narrate what the repo thinks the shard is. If the hint still needs to mention `HOWTO_SECTION`, keep it to one short static semantic reminder that does not claim book-specific or shard-specific confidence. Do not preserve `example_files` references in the hint by default; if examples are ever needed again, they should be reintroduced later as a specific justified feature.

Acceptance for this milestone is a focused hint-rendering test or fixture inspection showing that the `HOWTO_SECTION` heuristic prose is gone from the hint while the hint still remains usable as a worker orientation sidecar.

## Milestone 3: Simplify the inline prompt-builder contract to match the trimmed payload

At the end of this milestone, prompt preview and any inline line-role prompt path should render from the same minimal contract as the worker payload. The inline prompt should no longer render sections based on removed shard-level fields. If any minimal context summary is still needed, it must be derived directly from the remaining mechanical payload or from static prompt text, not from deterministic shard-interpretation fields that no longer exist.

Implement this in `cookimport/llm/canonical_line_role_prompt.py`. Remove parsing and rendering of the deleted fields. Keep the prompt focused on the actual row data, immediate boundary context, and any small amount of stable static instruction text that remains defensible.

Acceptance for this milestone is a focused prompt-preview or prompt-builder test that proves the inline prompt still renders successfully from the trimmed payload and no longer references removed fields.

## Milestone 4: Update tests and docs so the contract stays narrow

At the end of this milestone, the checked-in docs and tests should describe and enforce the new minimal payload instead of the older garnish-heavy one. This is the milestone that prevents a future drift back into deterministic storytelling.

Update:

- `tests/parsing/test_canonical_line_roles.py`
- `docs/04-parsing/04-parsing_readme.md`
- `docs/10-llm/10-llm_README.md`

The parsing doc should describe the worker `in/*.json` transport in current tense using only the remaining fields. The LLM doc should stop claiming that line-role worker inputs carry shard-level posture fields and `HOWTO_SECTION` policy. Tests should check for the absence of removed keys, not only for the presence of surviving ones.

Acceptance for this milestone is a focused green test run plus doc text that a novice can read without getting the wrong idea about deterministic semantic authority.

## Plan of Work

Start by narrowing the payload contract in `cookimport/parsing/canonical_line_roles/policy.py`. This is the highest-authority seam because it defines what the model actually sees in the file-backed path and what the inline prompt builder reconstructs from. Remove the deterministic shard-interpretation fields there first so every downstream consumer is forced to confront the simpler contract directly instead of carrying compatibility shims forever.

Once the payload contract is narrow, clean up `cookimport/llm/canonical_line_role_prompt.py` so prompt preview stays aligned. The prompt builder should consume the same small transport the live worker sees. Do not solve this by leaving dead compatibility code that still accepts or prints removed fields. If there is any purely static instructional text worth keeping, keep it in the prompt template or static code path rather than pretending it is shard evidence.

After the payload and inline prompt are trimmed, update `cookimport/parsing/canonical_line_roles/planning.py` so the worker hint no longer reintroduces the removed `HOWTO_SECTION` heuristics or the other deleted shard-garnish fields through prose. Keep the hint only if it still serves a narrow non-redundant purpose after the trim. Do not preserve example references, shard summaries, or policy lists by default. If the remaining hint collapses to a very small static contract reminder, that is acceptable.

Finish by updating tests and docs in the same pass. The docs should tell the truth about the live payload shape, and the tests should fail if anyone re-adds the removed fields later. The result should be a contract that is easier to audit because every remaining field has a crisp reason to exist.

## Idempotence and Recovery

This work is a contract simplification, not a migration. Re-running the focused tests is safe. Regenerating worker roots in tests is safe. If a step fails halfway, the safe recovery path is to restore the previous consumer or test expectation temporarily behind a failing regression test and then continue trimming fields one consumer at a time.

Do not validate this plan with live Codex-backed book processing or benchmarks without explicit user approval. The intended proof path is focused unit and contract tests plus inspection of generated workspace artifacts.

If trimming the payload reveals that one surviving prompt or hint path truly still needs a small amount of guidance, do not quietly keep the old garnish. Land the removal first, let tests prove the smaller contract, and only then add back one narrowly justified cue in a separate change. The recovery principle is: if the repo can only guess it, it should not be passed as evidence.
