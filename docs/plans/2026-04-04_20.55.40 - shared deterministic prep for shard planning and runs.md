## Operator Intent (Structured) - DO NOT DELETE

The operator wants shard planning to stop doing an expensive throwaway local preflight that is separate from the real run. Instead, the repo should prepare a book once through the deterministic pipeline, cache that prepared state, use that same state to compute shard suggestions, and then reuse it when the benchmark or stage run actually executes. The operator explicitly understands that raw `blocks` alone are not enough for accurate recipe and knowledge shard planning, because those stages depend on deterministic recipe detection and deterministic non-recipe routing.

## Purpose / Big Picture

After this change, a user will be able to select one book or many books in the interactive benchmark flow and get accurate shard suggestions without paying for a separate full deterministic prep that is then discarded. The same deterministic preparation pass will become the shared foundation for prompt-preview shard planning and for the later vanilla or Codex-enabled run. The visible proof is that interactive shard-planning screens still show minimum-safe shard recommendations, but the repo stops redoing the same deterministic work when the real run starts.

This matters because the current temporary implementation is correct but wasteful. It prepares the selected book into a temporary processed run only to compute prompt preview phase plans and then throws that state away. That is acceptable for one book as a stopgap, but it will scale badly when the operator wants to line up many books in one session. The right long-term architecture is not "just use blocks." The right architecture is "prepare deterministic stage-owned artifacts once, then let both shard suggestions and execution read from the same prepared bundle."

  Evidence: `cookimport/llm/prompt_preview.py::_load_existing_run_preview_context(...)` reads processed-run artifacts including `full_text.json`, authoritative block labels, recipe spans, recipe block ownership, recipe drafts, and final drafts before it can build phase plans.

  Evidence: `docs/07-bench/07-bench_README.md` and `cookimport/labelstudio/ingest_flows/prediction_run.py` describe the single-book split cache as caching conversion payloads keyed by source plus split-sensitive inputs. - Observation: The user's "why not just use blocks?" instinct is directionally right but incomplete.
  Evidence: `docs/10-llm/10-llm_README.md` defines shard planning in terms of stage-owned units. `line_role` can start from blocks, but `recipe_refine` needs deterministic recipe candidates and `nonrecipe_finalize` needs deterministic residue after recipe ownership and routing.

## Decision Log

- Decision: The target abstraction is a "deterministic prep bundle," not a "blocks cache."
  Rationale: Blocks are necessary but not sufficient. Accurate shard planning also needs deterministic recipe-boundary outputs, block labels, recipe ownership, and non-recipe routing context. Naming the abstraction around "prep" keeps the real dependency boundary visible.
  Date/Author: 2026-04-04 / Codex

- Decision: Prompt-preview shard suggestions and later run execution must read the same prepared deterministic state.
  Rationale: This is the only way to guarantee that the recommendation is accurate relative to the actual runtime inputs. If suggestions and execution each rebuild deterministically on their own, drift remains possible even if both are "correct."
  Date/Author: 2026-04-04 / Codex

  Rationale: The current pain is most visible in interactive benchmark, and benchmark already has a clear reuse story through single-book split cache and processed output roots. Stage/import should be designed into the interfaces now but can remain a later adopter if needed.

## Outcomes & Retrospective

No code from this plan has been implemented yet. The main outcome so far is architectural clarity: the repo should not chase a cheap heuristic based only on source-file size or raw block counts, and it should not keep the current throwaway temporary preflight. The right middle path is a reusable deterministic prep bundle that is rich enough to support accurate shard planning but limited enough to remain purely deterministic and reusable across later execution modes.

The most important lesson captured in this draft is that "vanilla" and "preflight" are similar enough to share one deterministic prep layer, but not identical in responsibility. Vanilla execution produces user-visible outputs. Prep exists to make downstream planning and execution reuse possible. The implementation should therefore factor out the shared deterministic preparation work below both callers rather than renaming one caller to mean both things.

## Context and Orientation

This repository has two top-level runtime families relevant to this work.

The first family is the interactive CLI in [cookimport/cli_support/interactive_flow.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/interactive_flow.py). That file owns the guided menu loop and passes an `interactive_codex_target_context` into [cookimport/cli_ui/run_settings_flow.py](/home/mcnal/projects/recipeimport/cookimport/cli_ui/run_settings_flow.py). The shard-planning screen itself is generic. It can already display per-stage `minimum_safe_shard_count` when `recommendations_by_step` is available in that target context.

The second family is benchmark prediction/import execution in [cookimport/labelstudio/ingest_flows/prediction_run.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest_flows/prediction_run.py). Its public entrypoint `generate_pred_run_artifacts(...)` is the current all-in-one offline benchmark prep path. It reads a source file, does importer conversion, writes benchmark artifacts, and optionally writes a processed stage-backed run under `data/output`. When LLM surfaces are off, this function still performs the deterministic work needed to derive authoritative block labels, recipe spans, recipe ownership, and other stage-owned artifacts.

The current temporary bridge between those two families lives in [cookimport/cli_support/bench_single_book.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/bench_single_book.py). The helper `_build_single_book_interactive_shard_recommendations(...)` creates a temporary directory, runs `generate_pred_run_artifacts(...)` with all Codex surfaces off, then asks [cookimport/llm/prompt_preview.py](/home/mcnal/projects/recipeimport/cookimport/llm/prompt_preview.py) to compute phase plans and survivability. This path is correct but expensive because the prepared deterministic state is not reused by the later real run.

The phrase "deterministic prep bundle" in this document means a repo-owned set of artifacts and metadata that represent all deterministic state needed before any optional Codex stage starts. In plain language, it is "everything the repo learned about this book before asking any LLM to do work." In this repository, that includes at minimum the canonical extracted block stream, deterministic label-first artifacts, recipe-boundary outputs, recipe block ownership, and any stage-owned artifacts that prompt preview or later runtime depends on to build line-role, recipe, and knowledge shards. It does not include any LLM outputs.

The phrase "reuse" in this document means that a later caller should be able to consume the prepared deterministic state directly instead of re-running the deterministic prep logic from the source file. Reuse may be implemented by loading the bundle in place, by copying a prepared bundle into a new run root, or by hard-linking/canonicalizing artifact references, but the execution path must not recompute deterministic prep merely to get back to the same state.

The phrase "deterministic prep cache key" means a stable hash of the source identity plus only those run settings that can change deterministic prep outputs. It must not include LLM-only settings such as model id, reasoning effort, or requested shard counts, because those do not change the deterministic prep bundle.

## Plan of Work

Start by separating "deterministic prep" from "benchmark artifact generation" as first-class concepts. Today `generate_pred_run_artifacts(...)` owns both. The implementation should introduce a new stage-owned or prediction-run-owned helper that prepares one book into a reusable deterministic prep bundle and returns a strongly typed result object. The name can vary, but it must communicate that the function stops before optional Codex execution and that its output is reusable. Good examples would be `prepare_deterministic_book_bundle(...)` or `prepare_prediction_run_deterministic_bundle(...)`. Avoid a name like `build_blocks_cache(...)` because the bundle contains more than blocks.

Create the new helper in the runtime layer that already knows how to produce the needed deterministic state. The likely owners are [cookimport/labelstudio/ingest_flows/prediction_run.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest_flows/prediction_run.py) and the stage import session stack under [cookimport/staging/import_session.py](/home/mcnal/projects/recipeimport/cookimport/staging/import_session.py). Prefer to factor the shared deterministic phase lower in the stack so benchmark is not forced to keep owning a one-off version forever. The new helper must return a typed result that includes the prepared artifact root, the deterministic run-config hash or bundle key, source hash, workbook slug, and explicit pointers to the processed artifacts that prompt preview and later execution need.

Once the helper exists, define the deterministic prep cache contract. Store these bundles under a stable cache root instead of an anonymous temporary directory. The root should live under the processed-output side of the repo, because these are processed deterministic artifacts rather than golden-set benchmark verdicts. A reasonable first shape is a hidden cache tree under `data/output`, for example `data/output/.deterministic-prep-cache/<source_slug>/<bundle_key>/`. The cache entry must have one small manifest file that records the source path, source hash, deterministic prep settings summary, creation timestamp, artifact paths, and schema version. Keep the timestamp format on any generated folders or manifests as `YYYY-MM-DD_HH.MM.SS`.

The deterministic prep key must include only source identity and deterministic run settings. Use the source file hash plus the deterministic subset of `RunSettings`. That subset must include importer-affecting and stage-prep-affecting settings such as EPUB extractor choice, PDF OCR policy, section detector backend, multi-recipe splitter, ingredient parsing knobs, line-role deterministic settings, and any other setting that changes deterministic stage artifacts. It must exclude all LLM-only settings such as `llm_recipe_pipeline`, `llm_knowledge_pipeline`, `line_role_pipeline` when used only as an optional Codex surface switch, `codex_farm_model`, `codex_farm_reasoning_effort`, `recipe_prompt_target_count`, `knowledge_prompt_target_count`, and `line_role_prompt_target_count`. The exact field list must be documented in code and tests, because this boundary is central to the feature.

After the bundle and cache key exist, replace the current temporary single-book interactive preflight. The interactive benchmark flow in [cookimport/cli_support/interactive_flow.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/interactive_flow.py) should still ask for the selected book before `choose_run_settings(...)`, but the recommendation builder must now resolve a deterministic prep bundle from the cache or build it once into the stable cache root. Then it should call prompt preview or an extracted shard-planning helper against that cached bundle. The operator-visible behavior must stay the same or improve: the shard-planning menu still shows minimum-safe recommendations, but the system now records and reuses the prep work instead of discarding it.

Then change the execution path for single-book benchmark runs in [cookimport/cli_support/bench_single_book.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/bench_single_book.py) and [cookimport/labelstudio/ingest_flows/prediction_run.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest_flows/prediction_run.py) so the later benchmark run can consume the cached deterministic prep bundle directly. The main architectural requirement is that paired `vanilla` and `codex-exec` variants must no longer rerun deterministic prep independently when the deterministic prep key matches. The benchmark variant execution should either materialize a run from the cached bundle and then continue with variant-specific stages, or open the cached bundle as the starting state and write fresh run-owned outputs on top of it. Choose the version that keeps artifact ownership and rollback clear. The plan assumes the first safe version may still copy prepared artifacts into the run root if that is simpler than in-place reading, but deterministic recomputation is not allowed on a cache hit.

After single-book benchmark reuse works, extend the same deterministic prep cache contract to `selected_matched_books` and `all_matched_books`. The key product behavior there is not merely "faster." The key behavior is "prepare each selected book once, then reuse the prepared deterministic bundle for shard planning and execution." For selected/all matched books, the interactive UI does not need to block on all bundle creation before showing the first prompt unless that proves necessary. A valid first implementation is to precompute per-book prep bundles as part of the run queue setup and then show shard-planning notes sourced from those prepared bundles. If that UI sequencing is too disruptive, another valid first version is to prepare bundles immediately after run settings are chosen but before the execution scheduler starts. The important invariant is still one deterministic prep per book per deterministic prep key.

Once benchmark reuse is stable, decide whether stage/import should adopt the same bundle directly or remain a follow-up. This plan does not require stage/import adoption in the first implementation, but the interfaces must not be benchmark-only dead ends. In practice that means the deterministic prep result object should live in a neutral runtime module, not in an interactive benchmark helper file, and prompt preview should accept a prepared bundle reference or processed-run reference without caring which caller produced it.

During migration, keep the current temporary path available behind tests long enough to compare outputs. The simplest safe migration is to keep one test that proves the new bundle-backed recommendation path returns the same `minimum_safe_shard_count` results that the old throwaway temporary path would have returned for the same prepared book fixture. Remove the old path only after that comparison exists.

## Milestone 1: Factor Out A Reusable Deterministic Prep Bundle

This milestone creates the new abstraction and cache key but does not yet change interactive behavior. At the end of this milestone, the repo will be able to prepare a book once into a named deterministic prep bundle, write a manifest describing that bundle, and reload that bundle by cache key. The proof is a focused test suite that prepares the same book twice and gets a cache hit the second time, while LLM-only setting changes do not invalidate the bundle.

Implement the bundle owner in the runtime layer, not in the CLI helper layer. The bundle manifest must explicitly record what deterministic work completed and where the artifacts live. Keep the result strongly typed so later callers do not need to guess whether a path points at a processed run, a temporary directory, or a benchmark scratch root.

The acceptance proof for this milestone is threefold. First, preparing one source file with deterministic settings creates one cache entry with a manifest. Second, preparing the same source file again with the same deterministic settings reuses that manifest without recomputing the deterministic stage-owned artifacts. Third, changing only Codex model, reasoning effort, or prompt target counts does not change the deterministic prep key.

## Milestone 2: Read Shard Suggestions From The Shared Bundle

This milestone replaces the current throwaway temporary interactive preflight. At the end of this milestone, interactive single-book benchmark still shows real minimum-safe shard recommendations, but those recommendations come from the shared deterministic prep bundle instead of a temporary processed run under a temp directory. The proof is that the same selected book shows the same recommendation values as before while the log or test evidence shows bundle reuse.

Update [cookimport/cli_support/interactive_flow.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/interactive_flow.py), [cookimport/cli_support/bench_single_book.py](/home/mcnal/projects/recipeimport/cookimport/cli_support/bench_single_book.py), and [cookimport/cli_ui/run_settings_flow.py](/home/mcnal/projects/recipeimport/cookimport/cli_ui/run_settings_flow.py). Keep the existing user-visible prompt sequence stable: choose book, choose workflow, choose surfaces, see shard-planning rows with recommendations. The only intentional behavior change is that the CPU-expensive prep now becomes reusable and named rather than throwaway.

The new tests should prove both a cold-path build and a warm-path cache hit. The warm-path test must show that the recommendation builder reuses deterministic prep and still returns the same `minimum_safe_shard_count` values for recipe, line-role, and knowledge.

## Milestone 3: Reuse The Same Bundle In Single-Book Benchmark Execution

This milestone removes the double-work for actual execution. At the end of this milestone, a single-book interactive benchmark that already prepared a bundle for shard planning will reuse that bundle when running `vanilla` and `codex-exec` variants. The proof is a benchmark-helper test that runs the interactive single-book flow and demonstrates that deterministic prep happens once while both variants still receive the correct prepared deterministic state.

Decide explicitly how run-owned outputs relate to cached bundle-owned outputs. The safest version is that cached deterministic artifacts remain immutable and fresh run roots copy or materialize from them before writing variant-owned benchmark outputs. If direct in-place reading is used instead, the code must prove that later benchmark execution never mutates the cached deterministic prep bundle.

Keep the existing single-book split cache separate during this milestone unless unifying it is mechanically straightforward. It is acceptable to have split conversion cache below deterministic prep cache for one release if that keeps the migration safer and easier to verify.

## Milestone 4: Scale The Same Reuse Model Across Many Books

This milestone extends the cache and bundle design to `selected_matched_books` and `all_matched_books`. At the end of this milestone, the benchmark scheduler can prepare many books once, keep a stable cache of those prepared bundles, and then run vanilla or Codex-backed variants without repeating deterministic prep per variant. The proof is a multi-book benchmark-helper test that prepares two or more books and shows one deterministic prep per unique deterministic prep key.

The scheduler implementation must stay honest about resource use. If deterministic prep is CPU-heavy, it should run through explicit queueing and worker limits rather than silently launching one prep per book without bound. The same bundle manifest must record enough timing and hit/miss data for the operator to see whether the cache is helping.

## Validation and Acceptance

Acceptance is behavioral and must be demonstrated in tests and in one manual operator story.

For interactive single-book benchmark:

1. Select one book in interactive benchmark mode.
2. The shard-planning screen still shows minimum-safe recommendations for line-role, recipe, and knowledge when those stages are enabled.
3. The deterministic prep work is recorded under a stable cache root rather than a throwaway temp directory.
4. Starting the real benchmark run after choosing shard counts reuses the prepared deterministic bundle instead of rebuilding the same deterministic stage state.

For deterministic prep key correctness:

1. Preparing the same book twice with the same deterministic settings yields a cache hit.
2. Changing only `codex_farm_model`, `codex_farm_reasoning_effort`, or prompt target counts still yields a cache hit.
3. Changing a deterministic parsing setting such as EPUB extractor or multi-recipe splitter yields a cache miss and a new bundle.

For multi-book scalability:

1. A selected/all matched multi-book run prepares each distinct source once.
2. The scheduler or logs show one deterministic prep per source/key, not one prep per later variant.
3. Benchmark execution still produces the same variant outputs and comparison artifacts as before.

The migration is only complete when the repo can prove that the new shared deterministic prep path produces the same shard recommendations as the old temporary throwaway path for equivalent test fixtures, and when the old temporary path is removed from normal execution.

## Idempotence and Recovery

This plan is intentionally cache-first and should be safe to retry. Re-running deterministic prep for the same source and deterministic settings should either reuse the existing bundle or rebuild it into the same stable location only after an explicit stale/incomplete-entry check. Cache entries must be written atomically enough that a half-written manifest does not get treated as a valid hit on the next run.

If implementation work lands in pieces, keep the old temporary recommendation path available until the new bundle path proves equivalence in tests. That gives a clean fallback while the cache contract stabilizes.

If a cache entry is corrupted or incomplete, the runtime must ignore it, rebuild the bundle, and overwrite or replace the bad entry. Never silently mix partially valid old artifacts with freshly generated new ones.

## Artifacts and Notes

The most important artifact added by this work should be a deterministic prep bundle manifest. Keep it human-readable JSON. It should include a schema version, source file path, source hash, deterministic prep key, creation timestamp, deterministic settings summary, prepared artifact root, and explicit paths to the processed artifacts that shard planning and execution will later read.

Expected recommendation flow after migration:

    user picks book
    -> deterministic prep bundle resolves from cache or is built once
    -> prompt preview / shard survivability reads that bundle
    -> interactive shard menu shows minimum-safe shard suggestions
    -> benchmark variant execution reuses the same deterministic prep bundle

Expected cache-key behavior after migration:

    same source + same deterministic settings + different LLM settings
    -> same deterministic prep key
    -> same prepared bundle reused

    same source + changed deterministic parsing settings
    -> different deterministic prep key
    -> new prepared bundle built

## Interfaces and Dependencies

The final implementation should expose one neutral runtime-facing interface for deterministic prep reuse. The exact names can vary, but the following shapes must exist in substance.

In a neutral runtime owner such as `cookimport/staging/import_session.py` or a nearby new module, define a typed result object that contains:

    class DeterministicPrepBundleResult(BaseModel or dataclass):
        prep_key: str
        source_file: Path
        source_hash: str
        workbook_slug: str
        artifact_root: Path
        manifest_path: Path
        processed_run_root: Path
        cache_hit: bool
        timing: dict[str, float]

Also define one entrypoint in that same owner layer that resolves or builds the bundle:

    resolve_or_build_deterministic_prep_bundle(
        *,
        source_file: Path,
        run_settings: RunSettings,
        processed_output_root: Path,
        progress_callback: Callable[[str], None] | None = None,
    ) -> DeterministicPrepBundleResult

In the prompt-preview or benchmark helper layer, add one helper that reads recommendations from the prepared bundle without rebuilding deterministic prep:

    build_shard_recommendations_from_prep_bundle(
        *,
        prep_bundle: DeterministicPrepBundleResult,
        selected_settings: RunSettings,
    ) -> dict[str, dict[str, Any]]

The bundle-manifest code must also expose one deterministic key builder that clearly documents which run-setting fields are included and excluded. That function is part of the public correctness contract for this feature, even if it remains in a private module.
