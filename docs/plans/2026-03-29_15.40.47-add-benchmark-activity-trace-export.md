## User Intent (Structured Restatement)

The user wants benchmark artifacts to capture the same visible agent-activity surface that appears in an interactive Codex chat: short commentary-style checkpoints, the concrete searches and file reads the agent performs, the commands it runs, and the observable sequence of investigation steps. They do not want a fake or empty “thinking trace” section that implies hidden chain-of-thought access. They want the repository to log and publish the durable, reviewer-friendly record of visible work that can actually be captured from Codex-backed runs, and they want the naming to reflect that reality.

## Purpose / Big Picture

After this change, a person opening a Codex-backed benchmark artifact folder should be able to see a durable, human-readable trace of what the worker visibly did during the run. The current benchmark prompt artifacts know how to talk about a “thinking trace,” but live runs almost always show an empty section because no saved trace file is produced. The user clarified that the desired trace is not private chain-of-thought. It is the visible activity stream: commands the worker ran, visible agent-message checkpoints, and any public reasoning-summary events that the saved event stream actually exposes.

The user-visible proof is simple. In a Codex benchmark artifact root such as `data/golden/benchmark-vs-golden/<timestamp>/single-book-benchmark/<book>/codexfarm/prompts/`, there should be a new prompt-local activity-trace artifact set and the prompt summaries should stop saying “No thinking trace captured” when a worker session clearly produced visible event logs. A reviewer should be able to open `full_prompt_log.jsonl`, `prompt_type_samples_from_full_prompt_log.md`, and a new `activity_trace_summary.md` and see the same story shape the user asked for: what the worker searched, read, ran, and said, derived from saved worker artifacts rather than imagined hidden thoughts.

## Outcomes & Retrospective

This plan is now implemented. Benchmark prompt artifacts publish a canonical `activity_trace` surface derived from saved worker files, with one prompt-local JSON export per prompt row plus prompt-level Markdown/JSONL summaries. Reviewer-facing sample Markdown now says `Activity Trace`, and both stage/benchmark manifests publish canonical activity-trace summary artifact keys while still mirroring the older `thinking_trace_summary.*` filenames for compatibility.

The most important lesson from implementation is that terminology was only half the bug. The other half was the handoff seam between runtime telemetry and reviewer export. Recipe, knowledge, and line-role workers already persisted the evidence, but prompt publication needed stable artifact-path fields and one extra `process_payload` fallback for line-role-only saved prompt flows. Once that contract existed, the durable reviewer artifact was straightforward to build.

## Context and Orientation

This repository has two different observability layers that are easy to confuse.

The first layer is stage runtime observability. Recipe, knowledge, and canonical line-role worker runtimes create worker roots under their stage-specific output trees. Those roots already write files such as `events.jsonl`, `usage.json`, `last_message.json`, `workspace_manifest.json`, `live_status.json`, and in some stages `stdout.txt` and `stderr.txt`. The important files for this plan are:

- `cookimport/llm/recipe_stage_shared.py`, especially the code that writes worker-root artifacts for recipe workspace sessions and follow-up attempts.
- `cookimport/llm/knowledge_stage/workspace_run.py` and `cookimport/llm/knowledge_stage/recovery.py`, which do the same for knowledge.
- `cookimport/parsing/canonical_line_roles/runtime.py`, which does the same for canonical line-role.
- `cookimport/llm/codex_exec_runner.py`, which defines the direct-Codex event and telemetry summaries.

The second layer is benchmark prompt publication. `cookimport/llm/prompt_artifacts.py` takes saved run artifacts and produces reviewer-facing files under `.../codexfarm/prompts/`, including `prompt_request_response_log.txt`, `full_prompt_log.jsonl`, `prompt_type_samples_from_full_prompt_log.md`, and currently `thinking_trace_summary.jsonl` and `thinking_trace_summary.md`. This layer is what a reviewer actually opens from a benchmark artifact folder.

Today these two layers are only partially connected. The runtime layer keeps raw event files, but prompt publication mostly consumes aggregate telemetry rows. The prompt layer already has optional logic for a trace file path, yet live runtime rows do not fill that path. As a result, benchmark prompt artifacts know the concept of a trace without having a real exported trace to show.

For this plan, an “activity trace” means a repo-generated, prompt-local JSON payload that summarizes visible worker activity in order. It can include command executions, visible agent messages, lifecycle markers such as `turn.completed`, and any public reasoning-summary events that the saved event stream exposed. It must not claim to reveal private model chain-of-thought. If an event file does not expose semantic content, the trace must say only what is actually visible.

The implementation should preserve one further distinction. Raw `events.jsonl` is the ground truth artifact. `activity_trace` is a prompt-local derived export meant for reviewers. The export should be readable and stable. The raw files should remain unchanged.

## Plan of Work

Start by enriching the stage-local telemetry rows that eventually feed benchmark prompt publication. The prompt artifact builder should not have to rediscover worker roots by scanning stage directories. Instead, recipe, knowledge, and line-role payload builders should attach explicit artifact paths into each telemetry row they create from a workspace or structured follow-up attempt. The relevant functions are:

- `cookimport/llm/recipe_stage_shared.py`
  - `_build_recipe_workspace_task_runner_payload(...)`
  - `_build_recipe_inline_attempt_runner_payload(...)`
  - `_build_recipe_repair_runner_payload(...)`
  - `_build_recipe_watchdog_retry_runner_payload(...)`
- `cookimport/llm/knowledge_stage/recovery.py`
  - `_build_knowledge_workspace_task_runner_payload(...)`
  - `_build_knowledge_inline_attempt_runner_payload(...)`
- `cookimport/parsing/canonical_line_roles/runtime.py`
  - `_build_line_role_workspace_task_runner_payload(...)`
  - `_build_line_role_inline_attempt_runner_payload(...)`

Each of those builders should add prompt-export-friendly fields such as `events_path`, `last_message_path`, `usage_path`, `live_status_path`, `workspace_manifest_path`, and when present `stdout_path` / `stderr_path`. Use absolute paths at write time so the prompt exporter does not need to guess how to resolve them. For workspace sessions that are fan-out summarized across multiple rows from one shared worker session, it is acceptable for every derived row to carry the same artifact paths. That is simpler and more robust than trying to assign a unique event file per split row when the raw session was shared.

Once those paths exist, implement a new prompt-artifact export path in `cookimport/llm/prompt_artifacts.py`. Add helpers that read the saved event file and sidecars, normalize them into a canonical `activity_trace` payload, and write one prompt-local JSON file per prompt row under a new directory such as `prompts/activity_traces/`. The exact filename should be deterministic from `call_id`, for example `prompts/activity_traces/<call_id>.json`. The payload should include enough context for a reviewer and for future summary builders: schema version, source paths, call id, task id or shard id when known, stage key, model, reasoning effort, total event count, command count, agent-message count, reasoning-summary event count, and a bounded ordered `entries` list.

The `entries` list should be the readable core of the export. Each entry should record only visible information. For command events, include a summary such as `Ran rg -n "..." cookimport/llm/prompt_artifacts.py`. For visible agent-message events, include a short excerpt from the message. For public reasoning-summary events, include the event type and a bounded excerpt only if the event file contains exposed text such as `delta`. For lifecycle markers such as `thread.started` or `turn.completed`, include them only when they help explain the sequence. If a raw event lacks useful reviewer-facing content, prefer a concise structural summary over reproducing raw JSON.

After the trace writer exists, change the prompt-row assembly code in `cookimport/llm/prompt_artifacts.py` so `full_prompt_log.jsonl` carries canonical `activity_trace` metadata. During the first rollout, keep legacy `thinking_trace` data populated as a mirror of the same exported file so that existing tests and consumers continue to work. The canonical reviewer-facing section heading in `prompt_type_samples_from_full_prompt_log.md` should become `Activity Trace:`. The old `Thinking Trace:` heading may be retained in compatibility output only if needed for downstream stability, but the intent of the rollout is to make the visible terminology match the real artifact.

Update the summary builders next. Replace or wrap `build_codex_farm_thinking_trace_summaries(...)` with a canonical activity-trace summary builder that reads the prompt-local exported JSON files rather than waiting for a legacy `trace_path` to appear magically. Write new canonical outputs `activity_trace_summary.jsonl` and `activity_trace_summary.md` under `prompts/`. During the compatibility phase, either mirror those files to the legacy `thinking_trace_summary.*` names or keep a thin wrapper that writes both names from the same payload. The summary rows should answer reviewer questions such as: did this prompt row have an exported activity trace, how many visible commands were executed, how many visible agent-message entries were captured, and what were the first few readable entries.

Line-role needs one extra fix because its current prompt-row collector bypasses the optional trace machinery and writes `"thinking_trace": None`. Update `_collect_line_role_prompt_rows(...)` and any adjacent helper code so line-role rows also consume the new telemetry artifact paths and produce prompt-local activity traces the same way recipe and knowledge do.

Once the exporter and summaries are working, update the benchmark artifact manifest and CLI reporting in `cookimport/cli_commands/labelstudio.py`. The benchmark artifact manifest should publish canonical `codexfarm_activity_trace_summary_jsonl` and `codexfarm_activity_trace_summary_md` keys, plus legacy mirrored keys for `thinking_trace_summary` while compatibility remains necessary. The terminal note that points operators at prompt artifacts should not promise “thinking traces”; it should point them at “activity traces” or the general prompt artifact folder.

Finish by updating documentation. `docs/07-bench/07-bench_README.md` should describe the new reviewer-facing prompt artifact contract and explain that activity traces are derived from saved worker event files, not private reasoning. `docs/10-llm/10-llm_README.md` and `cookimport/llm/README.md` should explain where the raw event files come from, how prompt publication consumes them, and which stage-local artifact paths are now carried through telemetry rows for publication.

## Validation and Acceptance

Acceptance requires observable reviewer-facing behavior, not only code changes.

The first acceptance check is prompt-row level. After building prompt artifacts from a fixture or test run that contains worker `events.jsonl`, `full_prompt_log.jsonl` should contain a canonical `activity_trace` object for the affected rows. During the compatibility period, the same rows may also retain `thinking_trace`, but the activity-trace field is the source of truth.

The second acceptance check is prompt-summary level. `prompt_type_samples_from_full_prompt_log.md` should show `Activity Trace:` for rows that have exported activity data, and at least one example should include readable entries derived from real visible events such as command executions or final agent messages. The summary must not imply access to hidden chain-of-thought.

The third acceptance check is artifact durability. The benchmark `prompts/` directory should contain prompt-local exported trace files and canonical summary files such as:

    prompts/activity_traces/<call_id>.json
    prompts/activity_trace_summary.jsonl
    prompts/activity_trace_summary.md

During the compatibility period, mirrored legacy files may also exist:

    prompts/thinking_trace_summary.jsonl
    prompts/thinking_trace_summary.md

The fourth acceptance check is line-role parity. A line-role prompt row with an available worker event file must no longer be forced to `thinking_trace: None` or `activity_trace: null` simply because it came from the line-role saved-prompt export path.

The fifth acceptance check is automated coverage. At minimum:

    source .venv/bin/activate
    pytest tests/labelstudio/test_labelstudio_benchmark_helpers_artifacts.py -q

must pass with new assertions that prove event-derived activity traces are exported, summarized, and kept durable under `prompts/`. Additional focused tests should verify the path-carrying telemetry rows for at least one recipe or knowledge path and one line-role path.

## Idempotence and Recovery

This plan is additive. Re-running prompt artifact publication should safely overwrite prompt-local `activity_traces/*.json` and summary files. The exported trace file names must be deterministic from the prompt row so repeated runs update the same files rather than leaving timestamped duplicates behind.

The compatibility layer is the main recovery tool. If a downstream consumer still expects `thinking_trace` or `thinking_trace_summary.*`, keep mirrored legacy fields and filenames until the consumer is updated. Do not recover by reverting to the misleading terminology as the only canonical surface. The long-term direction remains `activity_trace`.

If prompt publication cannot find a saved `events.jsonl` for a row, the exporter should fail closed to `activity_trace: null` with clear summary counts rather than inventing a trace. The artifact should say “no exported activity trace available” rather than hallucinating missing steps.

Do not validate this feature by running live Codex-backed benchmarks or CodexFarm book processing without explicit user approval. Unit tests and fixture-driven prompt-artifact rebuilds are the safe proof path.
