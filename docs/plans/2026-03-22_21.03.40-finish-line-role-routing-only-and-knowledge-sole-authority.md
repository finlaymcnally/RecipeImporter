# Finish line-role routing-only authority and keep knowledge as the sole semantic judge

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

The repository already moved part of the outside-recipe authority split in the right direction: obvious junk can be excluded before knowledge review, and the optional knowledge stage already acts as the final semantic authority for many review-eligible non-recipe rows. The remaining problem is that the first LLM line-role pass still behaves like a provisional `KNOWLEDGE` classifier. Its prompt still tells workers to emit `KNOWLEDGE`, its parser still rescues and sanitizes rows into `KNOWLEDGE`, and several benchmark diagnostics still describe those provisional outputs as if they were a meaningful semantic stage.

After this change, the contract should be plain and decisive. The first line-role LLM pass should do only two outside-recipe jobs: identify recipe structure when it is really recipe structure, and mark overwhelming obvious-junk rows as excluded `OTHER` with a `review_exclusion_reason`. Everything else outside recipes should leave the line-role pass as review-eligible `OTHER`. The later knowledge stage should remain the only place that makes the semantic `KNOWLEDGE` versus `OTHER` call for review-eligible outside-recipe text. The visible proof is that a human can inspect the line-role prompt, the line-role JSONL outputs, the Stage 7 artifacts, and the benchmark diagnostic bundle and see one coherent story instead of a hybrid one.

## Context and Orientation

The relevant pipeline starts in [cookimport/parsing/label_source_of_truth.py](/home/mcnal/projects/recipeimport/cookimport/parsing/label_source_of_truth.py). That module atomizes extracted text into canonical lines, runs the line-role stage in [cookimport/parsing/canonical_line_roles.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles.py), projects the resulting labels back into block labels, and groups recipe spans. In this repository, “line-role” means the first LLM-backed line labeling pass that runs before Stage 7 non-recipe routing and before the optional knowledge review stage.

Stage 7 lives in [cookimport/staging/nonrecipe_stage.py](/home/mcnal/projects/recipeimport/cookimport/staging/nonrecipe_stage.py). Its job is to take blocks that are not owned by recipes and route them into either immediate excluded `other` or review-eligible non-recipe material. The optional semantic refinement stage then lives in [cookimport/llm/codex_farm_knowledge_orchestrator.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_orchestrator.py) plus [cookimport/llm/codex_farm_knowledge_jobs.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_jobs.py). That stage builds `knowledge/in/*.json` only from the review-eligible subset and is supposed to be the sole semantic `KNOWLEDGE` versus `OTHER` authority for those rows.

Final authority projection then flows through [cookimport/labelstudio/ingest.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest.py), [cookimport/staging/stage_block_predictions.py](/home/mcnal/projects/recipeimport/cookimport/staging/stage_block_predictions.py), and the benchmark helper surfaces in [cookimport/bench/](/home/mcnal/projects/recipeimport/cookimport/bench/). Those files write `line-role-pipeline/line_role_predictions.jsonl`, `line-role-pipeline/stage_block_predictions.json`, `08_nonrecipe_spans.json`, `09_knowledge_outputs.json`, and the benchmark-side diagnostics that reviewers use.

The architectural mismatch is now specific and concrete. Downstream code already talks as if final `KNOWLEDGE` comes only from non-recipe final authority, but the upstream line-role worker contract still teaches `KNOWLEDGE` as a first-pass label and the parser still rescues rows into `KNOWLEDGE`. That creates a hybrid system where the first pass still looks like a semantic classifier even though the intended design is routing-only.

In this plan, “definitely not this” means a row is so obviously non-useful for cooking knowledge review that the line-role pass should exclude it immediately. Examples are table of contents entries, running headers and footers, copyright/legal boilerplate, pure publishing metadata, page-number-only rows, and testimonial blurbs. In this plan, “review-eligible” means the row is outside recipes and not obviously excluded, regardless of whether it ultimately becomes `KNOWLEDGE` or `OTHER` later. Lesson headings like `Balancing Fat`, rhetorical headings like `What is Heat?`, memoir-like prose, and general cookbook teaching text should all remain review-eligible unless they are overwhelming obvious junk.

## Milestones

### Milestone 1: Narrow the active line-role worker contract to routing and exclusion only

At the end of this milestone, the active line-role prompt, examples, packet posture text, and parser-side prompt helpers should stop asking workers to emit semantic outside-recipe `KNOWLEDGE`. Review-eligible outside-recipe rows should become ordinary `OTHER` with no exclusion reason. Only overwhelming obvious-junk rows should become excluded `OTHER` with a `review_exclusion_reason`.

This milestone matters because the current worker prompt still frames line-role as a semantic knowledge classifier. As long as the prompt still says `Cooking Acids -> KNOWLEDGE`, the architecture remains conceptually muddled even if downstream scoring later ignores those semantics.

Acceptance for this milestone is a prompt/rendering/test surface where examples such as `Balancing Fat`, `Cooking Acids`, explanatory lesson prose, and `What is Heat?` all remain review-eligible `OTHER`, while `Acknowledgments`, page furniture, testimonials, and table-of-contents rows can still be excluded `OTHER`.

### Milestone 2: Remove parser-owned outside-recipe `KNOWLEDGE` rescue and sanitization behavior

At the end of this milestone, `cookimport/parsing/canonical_line_roles.py` should no longer use `_outside_recipe_knowledge_label_allowed(...)` or related helpers as active decision seams for routing outside-recipe rows into semantic `KNOWLEDGE`. The parser should still preserve recipe structure rescues, should still support decisive exclusion, and should still normalize stale old worker outputs safely, but it should stop behaving like a second semantic knowledge classifier.

This milestone matters because prompt-only cleanup is not enough. The parser currently rescues `OTHER` to `KNOWLEDGE`, sanitizes unsupported `KNOWLEDGE` back to `OTHER`, and uses knowledge-heading heuristics to make first-pass semantic decisions. Those seams need to move out of the active routing path if the line-role stage is truly routing-only.

Acceptance for this milestone is code and tests showing that outside-recipe review-eligible rows now end as `OTHER` unless they are excluded or structurally recipe-like, while old cached outputs that still say `KNOWLEDGE` are normalized into review-eligible `OTHER` without breaking artifact loading.

### Milestone 3: Re-audit Stage 7, final projection, and benchmark scoring against the cleaned routing-only contract

At the end of this milestone, Stage 7, final authority projection, and benchmark scoring should all still tell one coherent story after the line-role cleanup. Stage 7 should keep its routing-only role, final projection should only apply actual final authority, and benchmark scoring should continue to derive `KNOWLEDGE` only from final non-recipe authority rather than from any first-pass line-role guess.

This milestone matters because the downstream seams are already close to the intended design. They need an audit and some cleanup, not a fresh rewrite. The real risk is allowing stale assumptions about provisional line-role `KNOWLEDGE` to survive in notes, suppression rules, or projection summaries.

Acceptance for this milestone is that `line-role-pipeline/stage_block_predictions.json`, `08_nonrecipe_spans.json`, and `09_knowledge_outputs.json` remain internally consistent and continue to explain that unreviewed review-eligible rows stay `OTHER` for scoring until the knowledge stage truly promotes them.

### Milestone 4: Replace or rename misleading line-role diagnostics

At the end of this milestone, benchmark and reviewer-facing diagnostic artifacts should no longer imply that the first line-role pass owns a semantic `KNOWLEDGE` budget. The current `knowledge_budget.json` artifact and any related leaderboard/export language should either be removed or replaced with a routing-aligned diagnostic such as excluded-versus-review-eligible outside-recipe counts.

This milestone matters because the current artifact names encode the old architecture. If the product surface keeps saying “line-role knowledge budget,” future debugging will keep drifting back toward the wrong ownership model.

Acceptance for this milestone is a benchmark artifact bundle where line-role diagnostics describe routing and exclusion behavior, not provisional semantic knowledge judgments.

### Milestone 5: Lock the new contract into tests, docs, and one approved benchmark comparison

At the end of this milestone, a novice should be able to read the current docs, run the targeted tests, and understand the ownership split without opening historical plans. If the user later approves live token spend, one rerun of the Salt Fat benchmark should then confirm that the first pass no longer claims semantic `KNOWLEDGE` ownership and that downstream final authority still works.

This milestone matters because the repository already contains historical drift on this exact topic. The only reliable fix is to update both code and the living explanation surfaces together.

Acceptance for this milestone is green focused tests, updated docs, and, only with explicit approval, one live benchmark artifact bundle that demonstrates the cleaned routing-only contract.

## Idempotence and Recovery

This plan is safe to implement incrementally. The safest order is to add compatibility normalization first, then narrow the active prompt and parser decision paths, then clean up diagnostics, and only after that tighten any worker-output schema.

If narrowing the active line-role output schema causes too much immediate churn, the safe temporary fallback is to keep the worker parser backward compatible while still coercing outside-recipe `KNOWLEDGE` into review-eligible `OTHER`. That preserves the ownership model even if some prompt artifacts or old tests lag behind.

If benchmark-helper code still depends on `knowledge_budget.json`, do not reintroduce semantic line-role `KNOWLEDGE` just to keep the artifact name alive. Add a temporary compatibility alias or a small loader translation, then replace the misleading surface cleanly.

If a regression appears in final non-recipe projection during implementation, the safe fallback is to preserve the existing downstream final-authority seam and revert only the most recent prompt or parser narrowing step. Do not let a temporary prompt migration bug re-open the old authority overlap.
