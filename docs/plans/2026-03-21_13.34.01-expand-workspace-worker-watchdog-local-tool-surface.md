# Expand The Workspace-Worker Watchdog To Allow Normal Local Agent Work

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the main Codex workspace workers for recipe, knowledge, and line-role should be able to solve their assigned local tasks using ordinary bounded agent techniques without being killed by the watchdog for harmless tool choices. A worker should be able to read local files, write local helper files, run local transforms, and use local interpreters that stay inside the sterile workspace. The user-visible effect is that future benchmark and stage runs stop showing `watchdog_command_execution_forbidden` for normal local work and instead reserve watchdog kills for real workspace escapes, network or repo access, package-manager use, or obvious no-progress loops.

The proof for this change is primarily zero-token. The shared watchdog classifier and all three stage callbacks should accept representative local commands such as `jq ... > out/...`, `python3 -c ...` or `python3 - <<'PY' ... PY`, and equivalent bounded `node` transforms when every referenced path stays inside the sterile workspace. The same tests must still reject repo, network, package-manager, parent-traversal, and outside-workspace commands. No live Codex run is required to complete this plan; a live rerun is optional follow-up only if the user later approves token spend.

## Surprises & Discoveries

- Observation: the repo already claims that main workspace workers are supervised by a “boundary-based” policy rather than a shell-shape allowlist.
  Evidence: `docs/10-llm/10-llm_README.md` says the main worker path should tolerate ordinary local shell work and should only kill for real boundary violations or pathological loops.

- Observation: the current implementation still relies on shell-text parsing and regex scanning, not on a real shell parser or a filesystem-aware policy engine.
  Evidence: the shared enforcement logic lives in `cookimport/llm/codex_exec_runner.py` and decides kills by parsing command text, extracting executables, scanning for `../`, scanning for `/`, and checking a fixed forbidden executable set.

- Observation: this means the main risk is no longer only “prompt mismatch”; it is parser mismatch between a real local command and the watchdog’s idea of what that command means.
  Evidence: the recent `jq` false positive came from text scanning that misread `jq`’s `//` operator as an absolute path escape.

- Observation: before implementation, `python`, `python3`, and `node` were still mixed into the same “forbidden executable” bucket as true boundary-escape tools.
  Evidence: the pre-change `_WORKSPACE_FORBIDDEN_EXECUTABLES` set in `cookimport/llm/codex_exec_runner.py` contained both clearly dangerous tools (`git`, `curl`, `wget`, `ssh`) and plausibly safe local transformers (`python`, `python3`, `node`).

- Observation: the existing text-based detector was good enough for this pass once the executable bucket was narrowed; no extra parser layer was needed to allow bounded interpreter commands.
  Evidence: focused `python3 -c`, `python3 - <<'PY'`, and `node -e` tests all stay inside the existing relative-path checks and pass without widening the path model.

- Observation: the project’s real safety boundary is not “never use interpreters.” It is “do not leave the sterile workspace or touch repo/network/global surfaces.”
  Evidence: the worker manifests and docs explicitly encourage bounded local automation and helper files inside the workspace, and the user’s complaint is specifically about workers being killed for normal local problem-solving.

## Outcomes & Retrospective

This plan is implemented at the zero-token proof layer. The shared watchdog now behaves more like a bounded local sandbox and less like a shell-style etiquette checker: `workspace_worker` sessions may use bounded `python`, `python3`, and `node` transforms, while repo/network/package-manager tools and visible path escapes remain kill-worthy.

The expected residual risk is still the same: text-based shell parsing will never be perfect. This pass intentionally did not widen absolute-path handling, so false positives can still exist for commands that surface safe absolute workspace paths. The long-term ideal is still a more structural path-resolution policy, but the highest-signal product mismatch is now removed.

## Context and Orientation

This repository has a direct Codex execution path that can run in two very different modes.

The first mode is `structured_json`. That is the strict one-shot retry or repair path. In that mode the model is supposed to return one JSON object directly, and shell commands are still off-contract. This plan does not loosen that mode.

The second mode is `workspace_worker`. That is the long-lived main worker path used by recipe, knowledge, and line-role processing. In this mode the repo creates a sterile worker workspace outside the repository, mirrors task files into that workspace, launches one Codex session in that workspace, and expects the worker to read local files and write local results incrementally under `out/`. In this repository the workspace typically contains `worker_manifest.json`, `assigned_shards.json`, `assigned_tasks.json`, optional leased-packet files such as `current_packet.json` and `current_hint.md`, local inputs under `in/`, worker hints under `hints/`, helper space under `scratch/`, and result files under `out/`.

The shared watchdog logic lives in `cookimport/llm/codex_exec_runner.py`. The key functions are:

- `classify_workspace_worker_command(...)`: reviewer-facing telemetry about what kind of command the worker just ran.
- `detect_workspace_worker_boundary_violation(...)`: the kill-or-not enforcement seam for main workspace workers.
- `_detect_workspace_worker_boundary_violation_in_text(...)`: the shell-text scanner that currently detects forbidden executables, `../`, `~`, absolute paths, and some output-path restrictions.
- `should_terminate_workspace_command_loop(...)`: the no-progress loop guard for workers that keep issuing commands without stabilizing outputs.

The three stage-specific callbacks that use the shared detector are:

- `cookimport/llm/codex_farm_orchestrator.py::_build_recipe_watchdog_callback(...)`
- `cookimport/llm/codex_farm_knowledge_orchestrator.py::_build_strict_json_watchdog_callback(...)` when called with `watchdog_policy="workspace_worker_v1"` and `allow_workspace_commands=True`
- `cookimport/parsing/canonical_line_roles.py::_build_strict_json_watchdog_callback(...)` when called with `watchdog_policy="workspace_worker_v1"` and `allow_workspace_commands=True`

The current tests that matter most are:

- `tests/llm/test_codex_exec_runner.py` for the shared classifier and boundary detector
- `tests/llm/test_codex_exec_runner_workspace.py` for worker-manifest and workspace-mode expectations
- `tests/llm/test_codex_farm_orchestrator.py` for recipe workspace watchdog behavior
- `tests/llm/test_codex_farm_knowledge_orchestrator.py` for knowledge workspace watchdog behavior
- `tests/parsing/test_canonical_line_roles.py` for line-role workspace watchdog behavior

For historical context, the recent main-worker watchdog loosening work is documented in `docs/tasks/2026-03-21_12.11.12-loosen-main-worker-watchdog-across-stages.md`, and the Salt Fat investigation report in `docs/reports/2026-03-20_12.30.10-saltfat-codexfarm-investigation-report.md` captures the real operator pain: workers still hitting `watchdog_command_execution_forbidden` for commands that look like normal local agent work.

The current hard restrictions baked into the shared runner are important to name plainly:

- A “forbidden executable” is any command name in `_WORKSPACE_FORBIDDEN_BOUNDARY_EXECUTABLES`. That now includes clearly dangerous tools like `git`, `curl`, `wget`, `ssh`, and package managers, while bounded local `python`, `python3`, and `node` transforms are allowed to continue through the same path checks as other local shell tools.
- An “absolute path” currently means any visible `/...` path in the shell text that is not `/dev/null`.
- “Parent traversal” means any visible `../`.
- A “workspace command loop” means too many commands or repeated commands without outputs stabilizing.

This implementation changed the first of those categories and intentionally left the second unchanged. It did not weaken the no-progress loop killer or the repo/network/package-manager boundary.

## Plan of Work

The first edit landed in `cookimport/llm/codex_exec_runner.py`. The old mixed-purpose `_WORKSPACE_FORBIDDEN_EXECUTABLES` mindset was replaced with `_WORKSPACE_FORBIDDEN_BOUNDARY_EXECUTABLES`, which keeps `git`, `curl`, `wget`, `ssh`, `scp`, `docker`, `npm`, `npx`, `pip`, `pip3`, `apt`, `apt-get`, `brew`, `kubectl`, and similar repo/network/package-manager executables forbidden while allowing bounded local `python`, `python3`, and `node` transforms in `workspace_worker` mode. The nearby comments and workspace-manifest text now say explicitly that local interpreters are allowed for main workspace workers as long as path and tool boundaries are still respected.

The shared detector review around `_extract_workspace_shell_body(...)`, `_command_tokens_for_watchdog(...)`, and `_detect_workspace_worker_boundary_violation_in_text(...)` confirmed that interpreter commands could remain inside the same relative-path boundary checks as `jq`, `cat`, and `sed`. Commands such as `python3 -c '...'`, `python3 - <<'PY' ... PY`, and `node -e '...'` no longer die solely because of the executable name; they now live or die based on the same forbidden-path and forbidden-tool checks as the rest of the workspace shell surface.

The optional second edit in the same file was not taken. Workspace-internal absolute paths remain forbidden in this pass because the current regex-based detector is still safer if it rejects all visible absolute paths except the existing `/dev/null` sink.

After the shared runner change, update the stage-level tests rather than the stage-level policy first. The stage callbacks in `cookimport/llm/codex_farm_orchestrator.py`, `cookimport/llm/codex_farm_knowledge_orchestrator.py`, and `cookimport/parsing/canonical_line_roles.py` already rely on the shared detector. Add explicit regression tests showing that each stage callback accepts at least one representative bounded interpreter command. Use realistic examples that write one output file under `out/` from one local input, not toy no-op commands. The examples should mirror real worker behavior: transforming task JSON into output JSON inside the sterile workspace.

Then expand the shared tests in `tests/llm/test_codex_exec_runner.py` and `tests/llm/test_codex_exec_runner_workspace.py`. Add a classifier test and a boundary-detector test for each newly allowed local-tool shape. At minimum cover:

- `python3 -c ...` reading `in/task-001.json` and writing `out/task-001.json`
- one heredoc-style Python helper such as `python3 - <<'PY' ... PY` if that form is being allowed
- `node -e ...` or `node <<'EOF' ... EOF` if Node is being allowed
- one still-forbidden command from each boundary family, such as `git status`, `curl https://...`, `pip install ...`, and `cat ../secret.txt`

The tests must make the policy legible to a future reader. A novice should be able to read them and understand that the watchdog now protects boundaries, not command aesthetics.

Finally, update `docs/10-llm/10-llm_README.md` and, if useful, the recent watchdog task note in `docs/tasks/2026-03-21_12.11.12-loosen-main-worker-watchdog-across-stages.md`. The docs need one plain statement of the final product rule: main workspace workers may use bounded local shell and local interpreters inside the sterile workspace; structured retry and repair calls remain stricter; repo/network/package-manager and outside-workspace access remain forbidden.

## Validation and Acceptance

This plan is accepted when the following behaviors are true and are proven by tests.

First, the shared watchdog must no longer return `watchdog_command_execution_forbidden` for bounded local interpreter transforms. A classifier or callback fed a command such as:

    /bin/bash -lc "python3 -c 'import json; ...' <local input> > out/task-001.json"

or:

    /bin/bash -lc "node -e '...' in/task-001.json > out/task-001.json"

must return `None` from the stage callback or an allowed policy from the shared classifier, provided every visible path stays under the sterile workspace roots and no forbidden external executable appears.

Second, the same callbacks must still reject true boundary violations. Commands such as the following must still fail:

    /bin/bash -lc "git status"
    /bin/bash -lc "curl https://example.com"
    /bin/bash -lc "pip install foo"
    /bin/bash -lc "cat ../outside.txt"
    /bin/bash -lc "cat /tmp/outside.txt"

Third, recipe, knowledge, and line-role must all inherit the same policy. It is not enough to change the shared helper and leave one stage callback or stage test still expecting a kill for normal local agent work.

Fourth, the docs must say plainly what the runtime does now. The acceptance wording in `docs/10-llm/10-llm_README.md` should no longer imply that local interpreters are forbidden if the code now allows them.

Optional live acceptance, only with later user approval: rerun a saved benchmark target with the same worker count and confirm that main worker failures, if any remain, come from true boundary violations or loop kills rather than from ordinary local-tool use.

## Idempotence and Recovery

This work is safe to do incrementally. The test additions should be written before or alongside the policy changes so regressions are visible immediately. If one broadened command form turns out to be too hard to classify safely, back it out explicitly and leave a narrower allowed form with tests. For example, if heredoc Python proves too ambiguous for the current detector, accept `python3 -c` first and record the deferral in the `Decision Log` rather than leaving the policy half-implicit.

If a stage-specific test still expects the old stricter behavior, update the test only after the shared runner change is in place and the intended product rule is written down in this plan. Do not re-tighten the shared runner just to satisfy a stale test.

No destructive data migration is involved. The code changes affect watchdog decisions during future runs; existing saved artifacts remain valid historical evidence.

## Artifacts and Notes

The most important existing evidence to keep in mind while implementing this plan is the current mixed message:

    Worker manifest says broad local shell use is allowed.
    Shared docs say the main worker path is boundary-based.
    Shared code still forbids interpreter executables outright.

The concrete recent example is the already-fixed `jq` false positive, which proved the overall class of problem: shell-text heuristics can easily misread ordinary bounded commands. This plan extends that lesson to interpreter commands rather than pretending `jq` was the only edge case.

The most useful implementation artifacts will be short new tests that read like policy statements. Keep them concise and stage-specific. Each test should tell a future reader one rule in one place.
