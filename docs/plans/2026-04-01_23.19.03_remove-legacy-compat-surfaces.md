# Remove Legacy Compatibility Surfaces While Preserving `C3imp`

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

This plan is written for a solo-project cleanup. The goal is not to preserve old operator habits or old artifact layouts unless they are still actively used now. The only explicitly preserved shortcut is `C3imp`, because the human still uses it. Every other legacy alias, compatibility shim, and transition seam named here should be treated as deletion work unless implementation evidence proves it is still required for current behavior.

No CodexFarm-enabled book processing or benchmark runs are part of this plan. Validation must rely on local tests, fake runners, prompt-preview paths, grep-based cleanup checks, and narrow deterministic CLI invocations only.

## Purpose / Big Picture

After this change, the repo should present one current product surface instead of a current surface plus a museum of old names and old shapes. A novice should be able to inspect `pyproject.toml`, the runtime models, the benchmark helpers, and the direct Codex execution code and see one live path for each concern. The visible CLI should keep `cookimport`, `cf-debug`, and the short interactive alias `C3imp`, but it should stop shipping `import` and `C3import`. The stage runtime should stop carrying old transition-cache fields as if they were still part of the live semantic model. Benchmark and Oracle follow-up tooling should stop reading pre-lane layouts and old run-dir locations. The direct Codex execution layer should stop converting old compact payload formats back into the current structured answers for the targeted stages.

The proof is straightforward and local. `cookimport --help` and `C3imp` should still work. `import` and `C3import` should be gone from the installed script table and from docs. Stage and benchmark tests should pass without compatibility passthrough keys or legacy layout readers. Grep over the touched modules should show that the named legacy seams are deleted rather than merely renamed.

## Context and Orientation

This repository is a local recipe-import pipeline. The main CLI lives in `cookimport/cli.py`. Installed scripts are declared in `pyproject.toml`. Today the script table still includes `cookimport`, `cf-debug`, `import`, `C3import`, and `C3imp`. `cookimport` is the canonical CLI. `C3imp` is a short interactive alias with environment defaults. `import` and `C3import` are batch wrappers that load settings, call `stage(...)` against `data/input`, and then fall back to the main CLI for any other arguments.

The core stage result model lives in `cookimport/core/models.py`. That file still defines `ConversionResult.non_recipe_blocks`. The current architecture docs already say this field is no longer an authority boundary. The real authority for outside-recipe ownership is the nonrecipe routing and finalization path in `cookimport/staging/import_session_flows/output_stage.py` together with `cookimport/staging/nonrecipe_stage.py`, `cookimport/staging/nonrecipe_authority.py`, and the run-root artifacts `08_nonrecipe_route.json`, `09_nonrecipe_authority.json`, and `09_nonrecipe_finalize_status.json`.

The run-settings surface lives in `cookimport/config/run_settings.py`, with profile normalization in `cookimport/config/codex_decision.py`, interactive selection in `cookimport/cli_ui/run_settings_flow.py`, benchmark planning in `cookimport/bench/qualitysuite/planning.py`, and winner reconstruction in `cookimport/bench/quality_leaderboard.py`. The repo already has a design report saying many of these fields are not true user choices anymore.

The benchmark Oracle handoff code lives in `cookimport/bench/oracle_upload.py` and `cookimport/bench/oracle_followup.py`. These files still read old upload-bundle layouts and old `.oracle_upload_runs` locations. The current intended layout is the lane-based `upload_bundle_v1/quality/` and `upload_bundle_v1/token/` structure, with `.oracle_upload_runs/<timestamp>-<profile>/` beside the bundle rather than nested inside it.

The direct Codex execution layer lives in `cookimport/llm/codex_exec_runner.py`. In this repository, “direct Codex execution” means the local runner that prepares assignment inputs, invokes a Codex command or fake runner, validates the answer, and promotes the result into the stage runtime. That file still contains legacy-input and legacy-output adapters for several stage keys. Those adapters exist so the current runner can still speak old compact payload shapes and then map them back into the current answer contract. This plan removes that support so the runner only accepts the live structured answer shape.

The recipe draft writer lives in `cookimport/staging/writer.py`. It still strips old alias keys like `name`, `ingredients`, and `instructions` from override payloads before writing final drafts. That is a small but concrete example of old-shape support still surviving in production code.

The touched docs are concentrated in `docs/AI_context.md`, `docs/01-architecture/01-architecture_README.md`, `docs/02-cli/02-cli_README.md`, `docs/05-staging/05-staging_readme.md`, `docs/07-bench/07-bench_README.md`, `docs/10-llm/10-llm_README.md`, and the short package-level READMEs under `cookimport/`.

## Plan of Work

Start with the public CLI surface because it is the least ambiguous deletion. Edit `pyproject.toml` to remove the `import` and `C3import` scripts while leaving `cookimport`, `cf-debug`, and `C3imp`. Delete `cookimport/entrypoint.py` if nothing else imports it after the script removal. Keep `cookimport/c3imp_entrypoint.py` intact unless tests show it should be simplified. Update `cookimport/README.md`, `docs/AI_context.md`, `docs/01-architecture/01-architecture_README.md`, and `docs/02-cli/02-cli_README.md` so they no longer describe `import` and `C3import` as active entrypoints.

Next, shrink the run-settings and benchmark compatibility surface. Use `docs/reports/2026-03-13-run-settings-surface-audit.md` as the pruning map, but implement the decisions in code rather than by report alone. Remove compatibility-only keys from `_RUN_SETTINGS_PATCH_COMPAT_KEYS` in `cookimport/bench/qualitysuite/shared.py` and delete the matching passthrough logic in `cookimport/bench/quality_leaderboard.py`. In the same pass, reduce the user-facing `RunSettings` surface wherever the current product no longer treats a field as a true operator choice. The concrete rule is simple: if the current runtime always normalizes or hardcodes a value, remove it from the user-facing surface and stop preserving it in replay helpers.

After the settings cleanup, remove the `ConversionResult.non_recipe_blocks` transition seam. Start by locating every remaining reader and dividing them into three categories: code that should read authoritative final nonrecipe rows, code that should read the routed candidate queue for late-output chunking, and code that only still uses the field because the model shape made it easy. Replace those readers with explicit inputs from `nonrecipe_finalize_result` or other stage-owned structures. Once no live code depends on the field, delete `non_recipe_blocks` from `cookimport/core/models.py`, remove assignments such as the repopulation in `cookimport/staging/import_session_flows/output_stage.py`, and update tests and docs so they speak in terms of current authority artifacts rather than cached block lists.

Then clean the Oracle and benchmark layout compatibility code. In `cookimport/bench/oracle_upload.py`, remove `_bundle_has_legacy_layout(...)`, `_legacy_oracle_upload_runs_dir(...)`, `_legacy_bundle_file_path(...)`, and the branch points that read those paths. In `cookimport/bench/oracle_followup.py`, stop falling back to a nested `.oracle_upload_runs` directory inside the bundle. Update `cookimport/bench/README.md`, any benchmark docs, and related tests so the lane-based bundle layout is the only supported one. If fixtures currently model the old layout, rewrite the fixtures instead of teaching the code to keep accepting both.

Then delete the direct-exec legacy adapters. Replace the legacy compact input/output conversions in `cookimport/llm/codex_exec_runner.py` with one current answer-builder contract per stage. For `recipe_refine`, `line_role`, `nonrecipe_finalize`, `nonrecipe_classify`, and `knowledge_group`, the fake runner and subprocess runner should both accept the same live structured answer schema and should stop translating `sid`, `r`, `cr`, `m`, `block_decisions`, `idea_groups`, or old row tuples into the current return shape. Keep the deterministic validators, but make them validate one current contract only. When updating the direct runner, also remove old-key suggestion language in helper modules that exists only to translate old result keys into current compact keys.

Finish with the small writer cleanup and docs pass. Remove the draft override alias stripping in `cookimport/staging/writer.py` if no remaining caller can produce those old aliases after the earlier steps. Rewrite the short architecture and package docs so they stop describing “legacy nicknames,” compatibility readers, or transition caches as active behavior. The final text should describe only the surviving current contracts.

## Validation and Acceptance

Acceptance is behavior, not just compilation.

First, the public script surface must be visibly cleaner. Reading `pyproject.toml` should show only `cookimport`, `cf-debug`, and `C3imp` in `[project.scripts]`. Running `C3imp` should still enter the interactive flow. The docs should describe `C3imp` as the only short alias and should stop mentioning `import` or `C3import` as supported entrypoints.

Second, stage runtime semantics must become more explicit, not less. After deleting `ConversionResult.non_recipe_blocks`, the stage and parsing tests must still pass, and the remaining code should name the real source of truth it is using. A reader should be able to tell whether a caller is reading routed candidate blocks or final authoritative nonrecipe rows without learning the history of a transition cache field.

Third, benchmark and Oracle tooling must trust only the current layout. The relevant tests should pass using only the lane-based `upload_bundle_v1` shape and the sibling `.oracle_upload_runs` root. No production code should contain old-layout readers after the cleanup.

Fourth, the direct Codex execution layer must speak one answer contract per stage. Fake-runner and prompt-preview tests should pass without any code path that accepts old compact payload shapes and remaps them into the current structured answer objects.

Fifth, docs must match the code. The short architecture docs should describe only the live current contracts and should not explain old nicknames or compatibility branches as if they are part of normal operation.

## Idempotence and Recovery

This plan is safe to execute incrementally. Each cleanup area should be completed with passing tests before the next area begins. If a deletion breaks a test, prefer updating the test or fixture to the current intended contract rather than restoring compatibility code. If a change reveals that a supposedly dead seam is still the only source for a behavior, stop and rewrite that caller to use the authoritative current input directly.

Do not reintroduce deprecated names under a new label. If a change is too large for one pass, land an intermediate commit that converts all callers to the new explicit input and only then delete the old model field or helper. The intermediate state is acceptable only if it is clearly additive and short-lived. The final merged state should contain no duplicated old path.

Because this repo may have unrelated local changes, do not revert files outside the cleanup scope. If an unrelated change touches the same modules, read it carefully and adapt rather than resetting it.
