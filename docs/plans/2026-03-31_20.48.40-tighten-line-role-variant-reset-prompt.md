# Tighten line-role variant-reset prompt guidance and prompt-contract regression coverage

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the canonical line-role prompt will say more plainly that a `Variations` run is local and must stop when a fresh recipe title is followed by a yield line or ingredient rows. The user-visible result is not a new deterministic rule. Instead, the model will get clearer instructions and a failure-shaped example, and the repository will gain tests that fail if future prompt edits weaken or remove that guidance.

You should be able to see this working without rerunning a CodexFarm benchmark. The proof should come from narrow prompt-building tests in `.venv` plus a prompt example that explicitly mirrors the March 31 benchmark failure shape: variant prose, then a new recipe start such as `Bright Cabbage Slaw`, then `Serves 4 generously`, then ingredient rows. The acceptance bar is that the active prompt text and its tests unambiguously encode that contract.

## Outcomes & Retrospective

No code has been changed yet. Research narrowed the work to one clear scope: strengthen the active line-role prompt and its examples, then add tests that fail if that guidance disappears. The plan intentionally does not promise benchmark improvement by itself; it promises a durable prompt contract that makes future prompt drift visible.

## Context and Orientation

Canonical line-role is the stage that labels each atomic text line with a structural role such as `RECIPE_TITLE`, `INGREDIENT_LINE`, `RECIPE_VARIANT`, `NONRECIPE_CANDIDATE`, or `NONRECIPE_EXCLUDE`. In this repository, the model-facing prompt for that stage is assembled by `cookimport/llm/canonical_line_role_prompt.py`. The active prose template lives in `llm_pipelines/prompts/canonical-line-role-v1.prompt.md`, and the direct worker JSON-object prompt lives in `llm_pipelines/prompts/line-role.canonical.v1.prompt.md`. The Python module also includes `_INLINE_TEMPLATE_FALLBACK`, which must match the template-backed behavior closely enough that fallback prompt generation does not silently drift.

The current bug surfaced in the benchmark artifacts under `data/golden/benchmark-vs-golden/2026-03-31_19.51.54/.../codexfarm/`. In those artifacts, the Codex line-role response labeled fresh recipe starts such as `Bright Cabbage Slaw`, `Vietnamese Cucumber Salad`, `Summer Tomato and Herb Salad`, `A Panzanella for Every Season`, and `Summer: Tomato, Basil, and Cucumber` as `RECIPE_VARIANT` instead of `RECIPE_TITLE`. The nearby context included a real `Variations` heading and variant prose just before the new recipe titles. That is the concrete pattern this plan addresses.

The most relevant files are:

- `llm_pipelines/prompts/canonical-line-role-v1.prompt.md`, the main inline prompt template for prompt-preview and other direct render paths.
- `llm_pipelines/prompts/line-role.canonical.v1.prompt.md`, the worker-facing prompt template used by the direct line-role worker transport.
- `cookimport/llm/canonical_line_role_prompt.py`, which loads templates, carries the inline fallback prompt string, and exposes `build_canonical_line_role_prompt(...)`.
- `tests/parsing/test_canonical_line_roles.py`, which already contains prompt-contract assertions and deterministic local labeling tests for title, yield, note, and variant boundaries.
- `docs/04-parsing/04-parsing_readme.md` and `docs/10-llm/10-llm_README.md`, which document the current line-role prompt seam and worker contract.

In this plan, a prompt-contract test means a test that inspects the rendered prompt text and fails if required wording or examples disappear. That kind of test does not guarantee model behavior in production. What it guarantees is that this repository keeps sending the model the guidance we decided was necessary.

## Plan of Work

Start with the prompt text. Edit `llm_pipelines/prompts/canonical-line-role-v1.prompt.md` so the variant-reset rule remains short but becomes harder to misread. Keep the existing message that variant context is local, then add one sentence that says a new title-like line followed by `Serves ...`, `Makes ...`, or ingredient rows is a fresh recipe start even if the immediately preceding rows belonged to a `Variations` run. Do not widen the rule into a deterministic-looking taxonomy; keep it as model guidance phrased in plain language.

In the same file, add a compact few-shot example that mirrors the failure shape more closely than the current `Lemon Vinaigrette` example. The example should mention the nearby `Variations` context and then show a fresh recipe start such as `Bright Cabbage Slaw` or `Summer: Tomato, Basil, and Cucumber` labeled `RECIPE_TITLE`, followed by a separate strict yield example such as `Serves 4 generously` labeled `YIELD_LINE`. The goal is to show the model the exact reset point that failed in the archived run.

Then update `llm_pipelines/prompts/line-role.canonical.v1.prompt.md` with the same strengthened rule set. This worker template is more compact than the inline prompt, so it does not need all of the same few-shot detail, but it does need the same semantic contract: a nearby `Variations` run does not stay sticky once a new title is followed by yield or ingredients. Keep wording aligned rather than identical where the formats differ.

Next, update `cookimport/llm/canonical_line_role_prompt.py`. If the active template files are authoritative, the minimal code change may only be to keep `_INLINE_TEMPLATE_FALLBACK` aligned with the updated prompt template text and examples. Do not move prompt policy into new code branches or helper logic. This plan is about keeping the prompt body itself stronger and synchronized.

After the prompt text is updated, extend `tests/parsing/test_canonical_line_roles.py`. First, strengthen `test_canonical_line_role_prompt_includes_required_contract_text` so it asserts for the new reset wording and the new failure-shaped few-shot example. Add exact assertions for the recipe-title-after-variants example and for the strict-yield-after-title example you added. The test should make prompt drift obvious by failing when the critical text disappears.

If needed for clarity, add one second prompt-focused test that renders a tiny synthetic shard context and asserts that the rendered prompt includes the expected nearby rows around a title reset case. Keep that test focused on the prompt payload shape, not on live LLM output.

Only add a non-prompt deterministic local labeling test if it covers an already-existing rule and helps explain the intended reset boundary without adding new deterministic semantics. For example, a small local test showing that a title-like line followed immediately by a yield line and ingredient rows stays `RECIPE_TITLE` is acceptable if it already follows current deterministic policy. A test that tries to make deterministic code correct model-only semantic ambiguity is out of scope and should not be added.

Finally, add brief doc notes in `docs/04-parsing/04-parsing_readme.md` and `docs/10-llm/10-llm_README.md` saying that the line-role prompt now carries an explicit variant-reset instruction and example for fresh recipe starts after `Variations` runs. Keep the doc update short and focused on the active contract.

## Validation and Acceptance

Acceptance is complete when all of the following are true:

1. The active inline prompt template in `llm_pipelines/prompts/canonical-line-role-v1.prompt.md` explicitly says that nearby variant context must reset when a new title-like line is followed by yield or ingredient rows.

2. The worker-facing prompt template in `llm_pipelines/prompts/line-role.canonical.v1.prompt.md` carries the same contract in its own shorter wording.

3. The fallback prompt text in `cookimport/llm/canonical_line_role_prompt.py` remains aligned with the active template so fallback rendering does not silently drop the new guidance.

4. `tests/parsing/test_canonical_line_roles.py` contains prompt-contract assertions for the new reset wording and the new failure-shaped example, and those tests pass in `.venv`.

5. If a local deterministic labeling test is added, it proves only an already-supported nearby-yield-or-ingredient title rule and does not introduce deterministic logic that overrides fuzzy cookbook semantics.

6. The short parsing and LLM docs mention the new prompt contract briefly and accurately.

The new tests should be understood honestly: they prove that the prompt contract exists and stays present. They do not prove that every future model run will obey it perfectly. That is acceptable for this plan because the change is prompt hardening, not deterministic relabeling.

## Idempotence and Recovery

These edits are safe to apply incrementally. Prompt-template edits and prompt-test edits are additive and can be rerun safely. The pytest commands above use only repository tests and temporary directories. If a prompt assertion becomes too brittle during implementation, weaken only the exact string match that is unnecessarily fragile while preserving the core behavioral wording the test is meant to lock in.

If template and fallback prompt text drift during implementation, resolve the drift before finishing the task. Do not leave one prompt surface updated and the other stale. If a doc update starts expanding into design discussion, cut it back to one short contract note and keep the detailed rationale inside this ExecPlan.
