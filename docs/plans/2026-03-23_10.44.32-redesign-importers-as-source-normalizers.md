# Redesign importers as source normalizers

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

After this change, `cookimport stage` should treat importers as source readers, not source judges. A human should be able to point the program at a PDF, EPUB, text file, spreadsheet, or structured recipe export and know that the importer will always do the same kind of work: extract one canonical ordered block stream, preserve provenance, preserve enough raw source detail for debugging, and optionally record source-native structure as typed non-authoritative support. The importer should no longer be the place that decides “this is definitely a recipe,” “throw this candidate away,” or “this leftover text is now semantically non-recipe.” Those semantic decisions should belong to the shared label-first, Stage 7, and knowledge stages that already exist downstream.

The user-visible gain is simpler architecture and more trustworthy debugging. When a cookbook is processed badly, the first question should become “what blocks and source-side support did the importer expose?” rather than “which importer-specific recipe heuristic already made an irreversible decision?” The visible proof is that a stage run writes canonical source-block artifacts for every importer, later pipeline stages still produce the same recipe and non-recipe outputs, and importer code no longer scores or rejects recipe candidates as an authority boundary.

## Non-Goals

This plan does not redesign the later authority seams. It does not change:

- label-first grouped spans as the recipe/non-recipe authority boundary
- Stage 7 as the coarse router for outside-recipe text
- the optional knowledge stage as the final semantic authority for review-eligible outside-recipe `knowledge` versus `other`
- the existence of split-job planning and merge for large PDF/EPUB inputs

This plan only changes what importers are responsible for and how the rest of the pipeline receives importer output.

## Surprises & Discoveries

- Observation: the current runtime already behaves as if it wants one canonical block stream, but it reconstructs that stream indirectly from importer raw artifacts instead of treating it as the importer’s primary product.
  Evidence: `cookimport/staging/import_session.py` calls `build_label_first_stage_result(...)`, which in turn rebuilds archive rows through `_archive_block_rows(...)` from `full_text` raw artifacts or fallback archive reconstruction instead of consuming a first-class importer-owned source-block contract.

- Observation: importers currently do much more than source normalization. Several of them already perform segmentation, recipe assembly, recipe-likeness scoring, candidate rejection, and leftover non-recipe creation before the shared authority stages even begin.
  Evidence: `cookimport/plugins/pdf.py`, `cookimport/plugins/epub.py`, and `cookimport/plugins/text.py` all call `score_recipe_likeness(...)` and `recipe_gate_action(...)` during importer conversion, then emit rejected candidates into `non_recipe_blocks`.

- Observation: the current architecture docs already acknowledge that importer candidates are not authoritative once label-first regrouping runs.
  Evidence: [docs/01-architecture/01-architecture_README.md](/home/mcnal/projects/recipeimport/docs/01-architecture/01-architecture_README.md) says label-first grouped spans are the recipe/non-recipe authority boundary, and `cookimport/staging/import_session.py` writes `authority_mismatch.json` instead of restoring importer candidates when regrouping finds zero recipes.

- Observation: structured source information is useful, but the repo currently stores it mostly as buried raw artifacts or importer-local behavior instead of a first-class, reusable support lane.
  Evidence: EPUB and WebSchema importers already preserve extractor diagnostics, raw XHTML/markdown, schema-derived objects, and other structure-rich artifacts, but downstream code treats them as debug payloads rather than an explicit non-authoritative support contract.

- Observation: text, spreadsheet, and export-based importers will still need synthesized blocks even after this redesign because they do not begin from a natural page-or-DOM block stream.
  Evidence: `cookimport/plugins/text.py` currently chunks raw text directly into recipe candidates, and `cookimport/plugins/excel.py`/`cookimport/plugins/recipesage.py`/`cookimport/plugins/paprika.py` operate on rows or structured records rather than pre-existing block objects.

- Observation: the current `ConversionResult` shape strongly nudges importers toward semantic ownership because its live fields are mostly recipe-side outputs rather than source-side facts.
  Evidence: `cookimport/core/models.py` currently centers `ConversionResult` on `recipes`, `tips`, `tip_candidates`, `topic_candidates`, `non_recipe_blocks`, and `raw_artifacts`, with no first-class canonical source-block or source-support fields.

  Rationale: deterministic cookbook interpretation has repeatedly failed in this repo when it tries to own semantic recipe/non-recipe decisions. The importer is the right place for mechanically true extraction and provenance, but the wrong place for final semantic ownership.
  Date/Author: 2026-03-23 / Codex

- Decision: introduce one canonical source-block contract for all importers, including record-first and structured importers that must synthesize blocks.
  Rationale: the shared downstream stages need one stable input shape. If some importers provide block streams while others continue to provide recipe candidates, the architecture stays bifurcated and the importer redesign does not actually simplify the authority story.
  Date/Author: 2026-03-23 / Codex

- Decision: preserve source-native information as typed non-authoritative support rather than throwing it away.
  Rationale: the user’s EPUB concern is valid: some sources really do contain useful native structure. That information should help later stages, but it must not become an early semantic truth boundary just because it was easy to extract deterministically.
  Date/Author: 2026-03-23 / Codex

- Decision: distinguish mechanically true source evidence from importer-derived source proposals.
  Rationale: “hint” is too vague. A DOM path, heading text, or row grouping is mechanically true evidence; a guessed recipe region is a proposal derived from importer logic. Those two classes must remain visibly different so importer suggestions do not silently harden into authority.
  Date/Author: 2026-03-23 / Codex

- Decision: default to typed support with provenance, not scalar authority scores.
  Rationale: a single confidence number is too easy for later code to misread as truth. Typed support plus exact referenced blocks and provenance is more inspectable and much harder to accidentally promote into authority.
  Date/Author: 2026-03-23 / Codex

- Decision: no source support item may be sufficient on its own to create, reject, or semantically classify cookbook content.
  Rationale: the redesign fails if importer output can still act as a hidden truth lane. Later shared stages may use source support only as supporting evidence alongside canonical source blocks and their own signals.
  Date/Author: 2026-03-23 / Codex

- Decision: preserve structured recipe objects, DOM groupings, row groupings, and similar source-native information as non-authoritative source support rather than discarding it.
  Rationale: the useful part of the current importer ambition is source-specific insight. The redesign keeps that value while removing importer-owned semantic authority.
  Date/Author: 2026-03-23 / Codex

- Decision: keep the migration additive at first by adding the new source-block and source-support fields to `ConversionResult` before deleting legacy importer-side recipe outputs.
  Rationale: stage, Label Studio, and tests currently assume `ConversionResult` has recipe-centric fields. A compatibility bridge lets the pipeline move source-first without forcing a risky flag day across every importer and every consumer.
  Date/Author: 2026-03-23 / Codex

- Decision: treat block identity as a first-class contract.
  Rationale: the redesign only works if importer-exposed blocks remain stable across split-job merge, reruns, and later downstream grouping. `block_id` is identity; `order_index` is ordering. Those are separate concerns and must stay separate in the model and docs.
  Date/Author: 2026-03-23 / Codex

- Decision: preserve both canonical normalized text and enough raw/provenance detail to explain where it came from.
  Rationale: a source-normalizer contract that only stores cleaned text will create a new debugging blind spot. The stage should operate on normalized text, but the artifacts must still let a reviewer understand the original source context.
  Date/Author: 2026-03-23 / Codex

- Decision: move recipe-likeness scoring and candidate rejection out of importers for stage-backed flows rather than trying to preserve them as hidden importer support.
  Rationale: scoring and rejection are semantic decisions about what the cookbook means. They do not belong in a layer that is supposed to be a source normalizer. If any scoring survives, it should live downstream as optional diagnostics after the shared authority stages have a stable source model.
  Date/Author: 2026-03-23 / Codex

- Decision: the first authoritative block stream written by the importer must become inspectable on disk as a normal run artifact.
  Rationale: the user wants simpler debugging. That requires a visible artifact showing exactly what the importer extracted before any label-first or LLM semantics ran.
  Date/Author: 2026-03-23 / Codex

## Outcomes & Retrospective

This plan is not implemented yet. Right now the repo still uses recipe-centric importers that emit `RecipeCandidate` objects, tip/topic payloads, and `non_recipe_blocks` before the shared label-first authority boundary begins. The architecture already compensates for that by letting label-first override importer candidates, but that compensation makes the authority story harder to understand than it needs to be.

If this plan lands successfully, the importer layer will become much easier to reason about. Every importer will expose one canonical ordered block stream plus optional typed source support, later shared stages will own semantic grouping and classification, and debugging will begin from “what facts did the importer extract?” instead of “which importer-specific recipe guess am I fighting?” The expected tradeoff is a broad migration touching importers, the stage path, tests, and docs, but the resulting architecture should be materially simpler and closer to the mental model the user wants.

## Context and Orientation

The current stage pipeline begins in `cookimport/cli.py` and `cookimport/cli_worker.py`. A worker calls one importer’s `convert(...)` method and gets back a `ConversionResult`. Today that object is recipe-centric: importers commonly fill `recipes`, `tip_candidates`, `topic_candidates`, `non_recipe_blocks`, `raw_artifacts`, and a `report`. The shared stage session later calls `execute_stage_import_session_from_result(...)` in `cookimport/staging/import_session.py`, which builds label-first authority from an extracted archive, optionally runs recipe and knowledge Codex stages, and then writes the run artifacts.

An “importer” is a source-specific reader under `cookimport/plugins/`. Examples are `pdf`, `epub`, `text`, `excel`, `webschema`, `paprika`, and `recipesage`. A “source block” in this plan means one ordered unit of extracted source text plus provenance and mechanically true features. A source block is not a semantic label. It is simply a stable record of what the importer found and where it came from. “Source support” in this plan means any preserved source-native structure that may help later stages, but is non-authoritative by contract.

This plan uses two support classes.

- Source evidence: mechanically true observations such as DOM path, row grouping, page number, spine item, list membership, heading text, or explicit embedded structured object presence.
- Source proposal: importer-derived suggestions such as “these blocks look like one recipe-shaped region” or “this export record likely maps to these synthesized blocks.”

Both classes are non-authoritative. Evidence is mechanically true; proposals are importer interpretation. Neither class may become semantic truth on its own.

The current authority problem is that importers are not just reading sources. They are also interpreting them. `cookimport/plugins/pdf.py` and `cookimport/plugins/epub.py` segment block ranges into candidate recipes, score those candidates, reject some of them, and emit the rejected material as leftover non-recipe blocks. `cookimport/plugins/text.py` chunks freeform text into candidate recipes, scores those chunks, and also emits rejected chunks as leftover non-recipe text. That importer-owned interpretation is later reinterpreted by the shared label-first pipeline in `cookimport/parsing/label_source_of_truth.py`, which already proves the importer is not the true authority. The redesign in this plan removes that mismatch by making the importer own source facts only.

Three code seams matter most.

`cookimport/core/models.py` defines `ConversionResult` and is therefore where the source-first contract must become explicit. `cookimport/staging/import_session.py` is the central consumer that must switch from archive reconstruction through importer raw artifacts to direct use of canonical source blocks. `cookimport/labelstudio/archive.py` currently rebuilds archive rows from raw artifacts or recipe records; it should become a compatibility adapter during migration and then shrink once the importer’s canonical source model becomes the normal live input.

This plan assumes that the live stage-backed authority story stays the same after importer output is normalized. Label-first grouped spans remain the recipe/non-recipe boundary. Stage 7 remains the seed router for outside-recipe text. The optional knowledge stage remains the only semantic authority for review-eligible outside-recipe `knowledge` versus `other`. The redesign is about removing early importer authority, not about changing those later authority seams.

## Milestones

### Milestone 1: Introduce the canonical source model without changing stage behavior

At the end of this milestone, the repository will have a first-class source-normalization contract even though the legacy importer outputs still exist. `ConversionResult` will gain canonical `source_blocks` and typed source support fields, and the stage path will learn how to prefer those fields when they are present while still falling back to the current archive-reconstruction path for unmigrated importers. The user-visible proof is that a migrated importer can write one inspectable source-block artifact to the run output and the rest of the stage pipeline still works.

This milestone is deliberately additive. It gives the repo a place to put the right data before deleting the wrong behavior. The acceptance for this milestone is that a stage run using one migrated importer still produces the normal final outputs, and the run also contains a canonical source-block artifact showing the importer’s neutral extraction result before labeling.

### Milestone 2: Make block-first importers true source normalizers

At the end of this milestone, `cookimport/plugins/pdf.py` and `cookimport/plugins/epub.py` will stop owning recipe candidate truth for stage-backed flows. They will still do source-specific extraction work such as OCR, EPUB extractor selection, block normalization, DOM or layout feature capture, and split-job-safe provenance, but they will emit that work as canonical source blocks and typed source support instead of authoritative recipe candidates plus rejected leftovers.

This milestone comes first because PDF and EPUB are already block-first internally. They are the easiest place to prove the new model. The acceptance is that PDF and EPUB stage runs still write the same downstream recipe and non-recipe artifacts, but the importer no longer performs recipe-likeness gating or treats rejected candidate ranges as the authoritative origin of non-recipe text.

### Milestone 3: Migrate record-first and structured importers by synthesizing canonical blocks

At the end of this milestone, `text`, `excel`, `webschema`, `paprika`, and `recipesage` will also produce canonical source blocks plus typed source support. For these importers, the redesign means synthesizing ordered blocks from rows, sections, schema objects, or export records rather than pretending they begin with a natural PDF-style block stream. Structured recipe objects should be preserved as evidence or proposals with rich payloads and provenance, but not elevated to truth by the importer itself.

This milestone matters because the redesign fails if some importers stay recipe-centric while others become source-centric. The acceptance is that every importer now fills the same primary fields in `ConversionResult`, and the stage path can start from one unified source-block contract regardless of file type.

### Milestone 4: Move importer-owned semantic work downstream and delete the old recipe-centric contract

At the end of this milestone, stage-backed flows will stop depending on importer-owned `recipes`, `tip_candidates`, `topic_candidates`, and `non_recipe_blocks` as the semantic input to the shared pipeline. Recipe grouping, recipe-vs-non-recipe interpretation, and leftover semantic classification will all begin from canonical source blocks. Importer-side recipe-likeness scoring, candidate rejection, and leftover creation will be removed or reduced to compatibility-only diagnostic helpers outside the stage-backed path.

This milestone is where the architecture actually becomes simpler. The acceptance is that the shared stage pipeline still produces final recipes, Stage 7 outputs, knowledge outputs, tables, chunks, and benchmark evidence, but importers no longer make early keep-or-reject recipe decisions for stage-backed runs.

### Milestone 5: Promote source-native support as optional support, not optional authority

At the end of this milestone, later stages will have access to source-native support such as schema objects, DOM groupings, or structured export records in a clearly non-authoritative way. The plan does not require label-first to trust every support item immediately. The first safe landing is to capture and persist the support, surface it in artifacts, and thread it into later code paths as optional context where useful. A later narrower change can decide exactly how label-first or grouping should consume it.

This milestone keeps the valuable part of the user’s EPUB concern. The repo should not throw away useful native structure just because deterministic semantics are weak. The acceptance is that the support is visible, preserved, and reusable without silently becoming the new truth layer.

## Plan of Work

Start in `cookimport/core/models.py`. Define two new first-class models: `SourceBlock` and `SourceSupport`. `SourceBlock` should be the canonical importer product. It should contain an ordered index, a stable block identifier, normalized text, raw or source-display text where useful, a provenance dictionary, and mechanically true feature fields such as page number, spine item, row index, heading level, list or table membership, source extractor metadata, or similar source facts. `SourceSupport` should be intentionally explicit and typed. It must declare whether it is evidence or proposal, what kind it is, which blocks it references, and where it came from. The critical rule must be written in the model comments and docs: importer support is non-authoritative.

Then extend `ConversionResult` so it can carry `source_blocks` and `source_support` alongside the current recipe-centric fields. Do not delete the old fields yet. Add a new helper module under `cookimport/core/` or `cookimport/staging/` that can do three things. First, normalize and validate source blocks from importers. Second, synthesize source blocks from the current legacy fields for compatibility during migration. Third, write a stable run artifact such as `raw/source/<workbook_slug>/source_blocks.jsonl` plus a small `source_support.json` companion file. The exact filenames should be chosen once and then documented in `docs/03-ingestion/03-ingestion_readme.md`.

After the models exist, update `cookimport/staging/import_session.py` and `cookimport/parsing/label_source_of_truth.py` so the stage path resolves source blocks in this priority order: new canonical `ConversionResult.source_blocks`, explicit merged `full_blocks` passed by split-job merge, then legacy archive reconstruction from `raw_artifacts` as a fallback only. This keeps the live pipeline running while importers are migrated incrementally. At the same time, update `cookimport/labelstudio/archive.py` so its archive-building helpers use the new source-block contract when present and fall back to the older reconstruction path only when necessary.

With the compatibility bridge in place, migrate the block-first importers. In `cookimport/plugins/pdf.py` and `cookimport/plugins/epub.py`, keep the extraction work that is actually source normalization: OCR, extractor backend choice, block extraction, block feature enrichment, and source-specific diagnostics. Remove or bypass the stage-backed recipe segmentation, recipe-likeness gating, rejected-candidate tracking, and non-recipe-leftover ownership logic. Instead, populate `source_blocks` directly from the extracted block stream and store any useful native structure as typed source support. Examples of safe support are explicit embedded recipe objects, mechanically true section or DOM boundaries, table-of-contents markers, or importer-derived candidate-region proposals clearly marked as proposals. Those items should be persisted, but the stage path must not treat them as truth.

Next migrate the record-first and structured importers. `cookimport/plugins/text.py` should stop turning freeform text chunks directly into authoritative recipe candidates for the stage-backed path. Instead, it should split the text into stable source blocks, preserving line order and any obvious section headings as mechanically true features or support. `cookimport/plugins/excel.py`, `cookimport/plugins/paprika.py`, `cookimport/plugins/recipesage.py`, and `cookimport/plugins/webschema.py` should synthesize source blocks from rows or structured records. They should also emit `SourceSupport` rows describing the native recipe object or row grouping they found. For example, WebSchema may preserve a schema.org recipe object as `hint_class="evidence"` when it is explicitly embedded in source data, while any importer-derived “these blocks probably belong together” grouping is `hint_class="proposal"`. The guiding rule is the same everywhere: importers may preserve source-native structure, but they may not decide that later grouping must obey it.

Once all importers can emit source blocks, move importer-owned semantic work downstream. Delete or bypass recipe-likeness scoring and `recipe_gate_action(...)` inside importers for the stage-backed path. Delete or bypass importer-owned `non_recipe_blocks` creation for stage-backed path. Move any recipe-specific tip extraction or standalone topic extraction that still relies on importer-owned semantic segmentation so it runs after the label-first recipe boundary and Stage 7 non-recipe boundary exist. The shared stage pipeline should become the first place where recipe and non-recipe meaning is assigned. If any legacy command still depends on importer-owned recipes, contain that dependency behind an explicit compatibility adapter rather than letting it remain the normal path.

While doing the migration, update split-job merge in `cookimport/cli.py` and `cookimport/cli_worker.py` so the conceptual primary merge product is canonical source blocks plus raw artifacts, not importer-owned recipe candidates. The implementation may still need to preserve legacy fields during the transition, but the comments, helpers, and artifact-writing logic should make clear that merged source blocks are what semantic grouping will consume next.

Finally, update the docs. `docs/03-ingestion/03-ingestion_readme.md` must be rewritten so the importer contract is source-first. `docs/01-architecture/01-architecture_README.md` must be updated so Phase 1 is described as source normalization plus source-support capture rather than recipe candidate authority. A short note should also be added under `cookimport/plugins/README.md` or the closest importer-facing doc explaining the new rule of thumb for future contributors: importers extract source facts and optional typed support; downstream shared stages own semantic truth.

## Validation and Acceptance

Validation must prove both behavior and architecture.

Behaviorally, a human should be able to run `cookimport stage` on at least one PDF or EPUB fixture and one simpler text or structured fixture and still receive final drafts, `08_nonrecipe_spans.json`, `09_knowledge_outputs.json`, and `stage_block_predictions.json`. The redesign is not allowed to break the rest of the stage pipeline.

Architecturally, the run must also show the new canonical source-normalizer seam. A human should be able to open the new source-block artifact and see one ordered stream of importer-owned facts before any label-first or knowledge decisions have happened. That artifact must exist for every migrated importer. The importer must no longer be the only place where the canonical block stream can be reconstructed.

For tests, the minimum local acceptance bar is:

- the new model and compatibility tests for `SourceBlock`, `SourceSupport`, and source-first stage resolution pass
- the importer-specific tests for PDF, EPUB, text, spreadsheet, and schema-backed paths pass after migration
- `./scripts/test-suite.sh domain parsing` passes
- `./scripts/test-suite.sh domain staging` passes
- `./scripts/test-suite.sh domain labelstudio` passes if archive or ingest helpers changed

At least one new focused test must prove that importer-native support remains non-authoritative. A good acceptance case is a fixture where a structured or row-group evidence/proposal item disagrees with what the shared label-first stage ultimately groups; the test should assert that the support is preserved on disk but does not override the later authority seam.

At least one new focused test must also prove that source support is never sufficient on its own to create, reject, or semantically classify content. The test should fail if downstream code tries to consume a support item as a stand-alone authority source.

## Idempotence and Recovery

This migration should be safe to stage gradually. The compatibility bridge means the new source-first pipeline can coexist with legacy importer outputs while each importer is converted. If one importer migration fails halfway, revert only that importer to the compatibility path and leave the new models and source-first consumers in place. Do not delete legacy fields until every importer and every live consumer has switched.

The new source-block artifacts should be additive. Re-running `cookimport stage` should simply create a new timestamped run folder with the same canonical artifact layout. No existing data migration is required for old run folders because the compatibility bridge will continue to support legacy archive reconstruction during the transition.

If a migrated importer produces malformed source blocks, fix the importer or the shared source-normalization helper and rerun the same tests. Do not “recover” by restoring importer-owned recipe truth for that source type. The whole point of the redesign is to make semantic correctness independent of importer heuristics.

The most useful artifact examples to preserve in review are small and mechanical.

For a migrated importer, `source_blocks.jsonl` should read like a neutral ledger of extracted facts. A good row looks like this:

    {"block_id":"b17","order_index":17,"text":"Ingredients","source_text":"Ingredients","location":{"spine_index":3},"features":{"heading_level":2,"list_context":"none"}}

For source support, `source_support.json` should show useful source-native information without implying authority. A good row looks like this:

    {"hint_class":"evidence","kind":"structured_recipe_object","referenced_block_ids":["b18","b19","b20"],"payload":{"source":"schema_org","title":"Roast Chicken"},"provenance":{"source":"embedded_jsonld"},"metadata":{"authoritative":false}}

A proposal row should look visibly different:

    {"hint_class":"proposal","kind":"candidate_recipe_region","referenced_block_ids":["b18","b19","b20","b21"],"payload":{"reason":"dom_grouping_plus_heading_context"},"provenance":{"source":"epub_importer_v1"},"metadata":{"authoritative":false}}

The acceptance stage run should still show the usual high-level artifacts next to the new source artifacts. A concise expected layout snippet is:

    <run>/raw/source/<workbook>/source_blocks.jsonl
    <run>/raw/source/<workbook>/source_support.json
    <run>/group_recipe_spans/<workbook>/recipe_spans.json
    <run>/08_nonrecipe_spans.json
    <run>/09_knowledge_outputs.json
    <run>/.bench/<workbook>/stage_block_predictions.json

## Interfaces and Dependencies

In `cookimport/core/models.py`, define `SourceBlock` and `SourceSupport`, and extend `ConversionResult` so it includes:

    source_blocks: list[SourceBlock] = Field(default_factory=list)
    source_support: list[SourceSupport] = Field(default_factory=list)

`SourceBlock` must contain, at minimum, a stable `block_id`, an ordered integer `order_index`, normalized `text`, optional `source_text` or equivalent raw-display text when needed, a `location` mapping, and a `features` mapping. `SourceSupport` must contain, at minimum, a `hint_class` (`evidence` or `proposal`), a `kind`, a `payload` mapping, explicit referenced block ids or indices when applicable, and provenance/metadata stating that it is non-authoritative by contract.

In a new helper module such as `cookimport/core/source_model.py`, define stable helpers equivalent to:

    def normalize_source_blocks(blocks: Sequence[SourceBlock | Mapping[str, Any]]) -> list[SourceBlock]:
        ...

    def normalize_source_support(items: Sequence[SourceSupport | Mapping[str, Any]]) -> list[SourceSupport]:
        ...

    def build_source_model_from_legacy_result(result: ConversionResult) -> tuple[list[SourceBlock], list[SourceSupport]]:
        ...

    def write_source_model_artifacts(run_root: Path, workbook_slug: str, blocks: Sequence[SourceBlock], support: Sequence[SourceSupport]) -> dict[str, Path]:
        ...

In `cookimport/staging/import_session.py`, the stage session must call a helper equivalent to:

    def resolve_stage_source_blocks(
        result: ConversionResult,
        *,
        full_blocks: Sequence[dict[str, Any]] | None,
        source_file: Path,
    ) -> list[dict[str, Any]]:
        ...

This helper must prefer `result.source_blocks`, then explicit `full_blocks`, then legacy archive reconstruction.

In `cookimport/plugins/base.py`, keep the public `Importer.convert(...) -> ConversionResult` protocol, but update the importer contract comments and docs so a future contributor understands that importers are responsible for canonical source extraction and optional typed source support, not final recipe truth.

In each importer module under `cookimport/plugins/`, ensure the end state is:

- `source_blocks` is populated for stage-backed flows
- `source_support` is populated when useful source-native structure exists
- importer-owned recipe-likeness gating is removed from the stage-backed path
- importer-owned `non_recipe_blocks` is no longer the semantic origin of later Stage 7 behavior

In `cookimport/labelstudio/archive.py`, preserve compatibility readers only as a transition layer. Once all live importers emit source blocks, the archive builder should become a straightforward projection of `SourceBlock` rows rather than a reconstruction pass over mixed raw artifacts and recipe records.
