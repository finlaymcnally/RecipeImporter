## Purpose / Big Picture

After this change, inline knowledge grouping should get one last bounded chance to recover when each grouping batch is locally valid but the merged shard still fails final packet validation. A human should be able to rerun the same shard shape that currently dies after merge and see one whole-shard grouping repair turn, then either a validated shard output or an explicit exhausted final-repair failure instead of an immediate hard fail after merge.

## Surprises & Discoveries

- Observation: the taskfile knowledge path already has a shard-result structured repair seam after same-session handoff, but the inline path does not.
  Evidence: `cookimport/llm/knowledge_stage/workspace_run.py` calls `_run_knowledge_repair_attempt(...)` in the taskfile worker path after invalid final shard outputs, while the inline path at the end of `_run_phase_knowledge_structured_worker_assignment_v1(...)` currently sets `proposal_status="invalid"` immediately after merged final validation fails.
- Observation: the final shard validator speaks in block-level terms, not unit ids.
  Evidence: `validate_knowledge_shard_output(...)` emits metadata such as `knowledge_blocks_missing_group`, `knowledge_group_grounding_mismatch_blocks`, and `knowledge_blocks_with_group_conflicts`, so the inline runtime must map block indices back onto grouping task-file `unit_id`s before it can build a repair task file.
- Observation: the inline runtime was still dropping some grouping settings before they reached `workspace_run.py`.
  Evidence: the first whole-shard repair regression only launched one grouping batch until `cookimport/llm/knowledge_stage/runtime.py` was updated to forward `knowledge_group_task_max_units` and `knowledge_group_task_max_evidence_chars` through the worker `settings` dict.

## Decision Log

- Decision: implement this as a whole-shard grouping repair turn, not another batch-local retry.
  Rationale: the missing failure mode only appears after batch answers are merged into one shard. A final repair call needs shard-wide visibility so it can reconcile cross-batch grouping story conflicts instead of blindly retrying one batch.
  Date/Author: 2026-04-13 / Codex
- Decision: keep the whole-shard repair bounded to one shard-result attempt.
  Rationale: batch-local grouping already gets its own semantic-step repair budget. This seam exists only to catch post-merge failures and should stay small and explicit rather than becoming an unbounded second repair loop.
  Date/Author: 2026-04-13 / Codex

## Outcomes & Retrospective

Inline knowledge now has the same last-chance authority seam that taskfile knowledge already had: if all batch-local grouping repairs succeed but the merged shard still fails final packet validation, the runtime rebuilds one shard-wide grouping repair packet, runs one bounded final grouping repair turn, and then revalidates the merged shard. The focused regression proves the old hard-fail seam is gone.

The main extra lesson from implementation was that settings pass-through bugs are easy to confuse with runtime bugs. The final regression only behaved like a two-batch shard after `runtime.py` forwarded the grouping batch-size settings down to `workspace_run.py`. That pass-through is now part of the durable fix.

## Context and Orientation

The inline knowledge runtime lives in `cookimport/llm/knowledge_stage/workspace_run.py`. It first runs binary classification, then one or more grouping batches, then merges the accepted grouping answers back into one final shard payload with `combine_knowledge_task_file_outputs(...)`. That merge and final packet validation are the last repo-owned authority step before the shard is written to `proposals/` and `out/`.

The grouping task-file builders live in `cookimport/llm/knowledge_stage/task_file_contracts.py`. That module already knows how to build grouping task files from kept rows, build repair task files, and canonicalize group ids after merge. The final shard validator lives in `cookimport/llm/codex_farm_knowledge_ingest.py`. It validates the merged packet result, but its failure metadata is expressed as block indices, packet ids, and group-level mismatches rather than task-file unit ids.

The current gap is specific to inline knowledge grouping. Taskfile knowledge already has a shard-result repair followup after same-session handoff if the final shard output is invalid. Inline knowledge only repairs within each grouping batch and then fails closed if the merged final shard is invalid.

## Plan of Work

Add one repo-owned helper path in `cookimport/llm/knowledge_stage/task_file_contracts.py` that can rebuild a whole-shard grouping task file plus a repair version of that task file from the current accepted classification and grouping answers. That helper must filter to one shard, keep the rows in block order, and map final validator block-level failures back to the affected `unit_id`s when possible. When final validator metadata is too coarse to name rows, it should conservatively repair the whole shard-wide grouping task file.

Then change the inline worker loop in `cookimport/llm/knowledge_stage/workspace_run.py`. After the existing batch-local grouping loop succeeds and the merged shard payload still fails final validation, build the whole-shard repair task file, issue one structured grouping repair turn in the same prepared execution workspace (respecting the configurable fresh-vs-resume transcript mode), merge the repaired answers back into the shard-level grouping answers, and re-run final payload assembly plus final packet validation. If that repaired final packet validates, promote it normally. If it still fails, keep the existing invalid-shard behavior but record that the final repair seam was attempted and exhausted.

Update the inline repair policy summary so the worker-assignment budget reflects the new shard-result whole-shard repair seam for inline knowledge, similar to the taskfile path’s existing post-handoff repair budget. Keep the semantic-step classification/grouping budgets unchanged.

Finally, add a focused runtime regression in `tests/llm/test_knowledge_orchestrator_runtime_leasing.py` that simulates two grouping batches whose local answers are valid but whose merged shard grounding is invalid until a final whole-shard repair turn fixes it. Update docs in `docs/10-llm/10-llm_README.md` and `cookimport/llm/knowledge_stage/README.md` once the code path is final.

## Validation and Acceptance

Acceptance means all of the following are true:

1. An inline knowledge shard that passes batch-local grouping validation but fails final merged shard validation gets one explicit whole-shard grouping repair turn.
2. That final repair turn reuses the same prepared workspace and obeys `knowledge_inline_repair_transcript_mode`.
3. If the final repair fixes the shard, the proposal wrapper and `out/*.json` contain the validated repaired payload.
4. If the final repair still fails, the shard remains invalid and the artifacts/telemetry show that the whole-shard repair seam was attempted and exhausted.
5. Focused `.venv` tests covering the new seam pass.

## Idempotence and Recovery

This work is safe to iterate on because all new artifacts are runtime outputs under temp worker roots. If the whole-shard repair helper is wrong, the safest recovery check is to inspect the saved final-repair packet under the structured-session shard folder and verify that its `rows` reflect the expected shard-wide kept rows and that its `repair_validation_summary` matches the merged-packet failure.

## Artifacts and Notes

Current failing seam in the inline worker:

    batch-local grouping valid
    -> combine_knowledge_task_file_outputs(...)
    -> validate_knowledge_shard_output(...)
    -> invalid
    -> immediate shard failure

Desired seam after this change:

    batch-local grouping valid
    -> combine merged shard
    -> final validation invalid
    -> build whole-shard grouping repair task file
    -> one final structured grouping repair turn
    -> recompute merged shard
    -> validate and either promote or fail exhausted
