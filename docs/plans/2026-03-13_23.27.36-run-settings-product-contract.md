## Purpose / Big Picture

After this work, the product should stop treating the full `RunSettings` schema as the same thing as the operator-facing contract even after Bucket 1 and Bucket 2 are cleaned up. A normal run should read as profile-plus-overrides, benchmark-lab controls should be intentionally separate, and manifests, summaries, analytics, and docs should describe the smaller contract instead of re-advertising the archaeology/persistence format.

The visible proof is straightforward. `cookimport stage --help` should stay short after Bucket 2 without falling back to schema-shaped leaks elsewhere. `cookimport labelstudio-benchmark --help` and the benchmark docs should clearly distinguish ordinary operation from benchmark-lab overrides. New manifests, winner settings summaries, and dashboard summaries should lead with operator choices, while the full raw payload remains available for compatibility and reproduction. Older payloads that still contain retired Bucket 1 keys or internal Bucket 2 keys must continue to load.

## Surprises & Discoveries

- Observation: the original product-contract plan overlapped three different cleanup shapes and became too broad once implementation started.
  Evidence: Bucket 1 needed hardcoded fixed behavior and retired-key compatibility, while Bucket 2 needs internalization without deleting runtime support. Those concerns now have dedicated plans because they are different migrations.

- Observation: most of the old “near-term cleanup” items are no longer this plan’s job.
  Evidence: the finished Bucket 1 plan already owns `benchmark_sequence_matcher`, debug toggles, and codex pass-id retirement, and the Bucket 2 plan now owns reclassifying the parser/scoring/OCR tuning controls as internal-only.

- Observation: after Bucket 1 and Bucket 2, the remaining problem is mostly contract narration and projection rather than field classification.
  Evidence: even with correct field metadata, `cookimport/cli.py`, `cookimport/labelstudio/ingest.py`, manifest/report writers, and docs can still mirror the persistence schema if they keep using large signatures or ad hoc summary key lists.

- Observation: direct Python helper signatures are an independent leak path from CLI help text.
  Evidence: the earlier audit found that `cookimport/labelstudio/ingest.py` and related helper entrypoints repeated large settings-shaped signatures even when `RunSettings` metadata had already started shrinking the visible surface.

- Observation: the Bucket 1 plus Bucket 2 cleanup materially shrank the public surface already, but it did not by itself define the final operator contract.
  Evidence: the current code baseline reports `62` total fields, `32` public, `30` internal, and `16` retired. That is much smaller than the original audit surface, but still larger than the profile-plus-overrides contract this plan is targeting.

- Observation: the explicit contract split was only useful once the code also decided which human-facing surfaces should use which projection.
  Evidence: ordinary stage/import summaries wanted the smaller `operator` view, while benchmark prediction/eval summaries still needed the broader public/product view so line-role and Codex override context stayed visible.

- Observation: Typer help did not get smaller just because the contract helpers existed.
  Evidence: `cookimport stage --help` kept showing the EPUB parser and web fallback tuning knobs until those specific options were marked hidden in `cookimport/cli.py`.

## Context and Orientation

`cookimport/config/run_settings.py` defines the full `RunSettings` schema. In this repository that schema serves four jobs at once: it loads saved JSON payloads, drives stage and benchmark runtime wiring, feeds interactive/editable settings surfaces, and preserves old winner/manifests payloads for archaeology. That is why the schema is larger than the honest day-to-day product contract.

The March 13 audit split the settings into three buckets. Bucket 1 is the set of fake choices that should become fixed product behavior and compatibility-only legacy keys. Bucket 2 is the set of real runtime controls that should stay persistable and patchable but disappear from the ordinary user-facing surface. Bucket 3 is the remaining set of real operator choices worth keeping visible. The dedicated Bucket 1 and Bucket 2 plans are both finished. This plan starts from that landed baseline.

Several files matter for the remaining work:

- `cookimport/config/run_settings.py` is the source of truth for the full schema, surface metadata, and payload projection helpers.
- `cookimport/config/codex_decision.py` remains the place where frozen behavior contracts and profile patches live.
- `cookimport/config/run_settings_adapters.py` maps `RunSettings` into stage and benchmark kwargs and should stop acting like the public product contract.
- `cookimport/cli.py` defines the main user-facing entrypoints and therefore controls what the operator contract feels like in practice.
- `cookimport/labelstudio/ingest.py` and related benchmark helpers are another contract surface because tests and orchestration code call them directly.
- `cookimport/config/last_run_store.py`, `cookimport/analytics/dashboard_collect.py`, `cookimport/analytics/perf_report.py`, manifest writers, and summary helpers determine whether artifacts tell the smaller contract story or dump the full persistence payload.

This plan uses four plain-English terms:

- An “operator setting” is a knob a normal human should intentionally vary run to run.
- A “benchmark-lab setting” is a setting that should stay usable for benchmarking, tuning, or regression forensics but should not dominate everyday stage/import flows.
- An “internal setting” is a setting that remains runtime-relevant or compatibility-relevant but should not appear in ordinary help, menus, or summaries.
- The “persistence schema” is the full `RunSettings` model, which survives so old payloads and saved experiments still load.

The target state is not “make every surviving field public again under a smaller heading.” The target state is to layer an explicit operator contract and an explicit benchmark-lab contract on top of the persistence schema, then make all human-facing product surfaces consume those explicit contracts instead of guessing.

## Plan of Work

### Milestone 1: Define Explicit Operator And Benchmark-Lab Contract Helpers

At the end of this milestone, the repo should have an explicit answer to three questions that are currently too easy to blur together: which settings belong to the ordinary operator contract, which settings belong to a benchmark-lab contract, and which settings exist only in the persistence/internal space.

Implement this in `cookimport/config/run_settings.py` with lightweight helpers, not a new abstraction stack. Keep the existing `public` and `internal` metadata that Bucket 2 relies on, but add contract-level helpers on top of it. The intended end state is that downstream code can call functions such as `ordinary_operator_run_setting_names()` and `benchmark_lab_run_setting_names()` instead of treating `public_run_setting_names()` as the final product story. Extend the existing projection helpers accordingly so code can intentionally produce an operator-facing payload and, where needed, a benchmark-lab-facing payload, rather than only “public” versus “full”.

Acceptance is that the code has one obvious place to ask for the ordinary operator contract and one obvious place to ask for the benchmark-lab contract, and that the operator contract is materially smaller than the remaining post-Bucket-2 public surface.

### Milestone 2: Stop Ordinary Entry Points From Mirroring The Persistence Schema

At the end of this milestone, the ordinary command and direct-call entrypoints should speak in terms of the explicit operator contract instead of long settings-shaped signatures.

Update `cookimport/cli.py` first. After Bucket 2, the help text should already be smaller. This milestone finishes the job by making the command shape itself match the smaller contract. Ordinary `stage`, `labelstudio-import`, and `labelstudio-benchmark` flows should consume a `RunSettings` object or a smaller contract projection rather than continuing to advertise or thread through schema-shaped kwargs. Keep rare lab/debug override seams only where they are still truly needed, and route them through clearly named internal or benchmark-lab paths.

Then update `cookimport/labelstudio/ingest.py` and any other direct Python helper entrypoints that still repeat large settings lists. The cleanest end state is that ordinary callers pass `RunSettings` or a compact contract bundle, while internal benchmark orchestration still has an explicit path to apply a `run_settings_patch` or equivalent override payload when required.

Finally, update `cookimport/config/run_settings_adapters.py` so adapter code intentionally bridges from the persistence schema to the smaller product surfaces rather than implicitly legitimizing every field that happens to exist on `RunSettings`.

Acceptance is behavioral. The ordinary command signatures and ordinary helper entrypoints should no longer look like thin wrappers around the entire schema, and code review should make it obvious which paths are operator-facing versus benchmark-lab/internal.

### Milestone 3: Reframe Summaries, Manifests, Winner Settings, And Analytics Around The Operator Contract

At the end of this milestone, human-facing artifacts should describe the smaller operator contract by default even though the raw full payload still exists.

Update the summary and manifest paths in `cookimport/cli.py`, `cookimport/labelstudio/ingest.py`, staging manifest writers, `cookimport/config/last_run_store.py`, and any analytics collectors/report writers that still hand-pick schema fields. Ordinary summaries should lead with profile-plus-overrides or the final operator projection. Benchmark-lab summaries may include their extra controls explicitly when the run was actually a benchmark-lab run. Full persistence/debug payloads should remain stored, but under clearly raw/debug naming so they do not masquerade as the normal operator contract.

Do not let analytics or winner-setting flows become a backdoor that re-expands the product story to the full schema. If those paths need more detail for reproducibility, keep both layers: concise operator summary first, raw payload alongside it second.

Acceptance is that new manifests, winner-setting explanations, and dashboard/perf summaries are easier to scan and no longer re-advertise internal or archaeology-only fields as if they were ordinary decisions.

### Milestone 4: Separate Benchmark-Lab Documentation From Ordinary Product Documentation

At the end of this milestone, a novice should be able to tell from the docs which settings are normal daily choices and which ones are only for benchmark-lab or internal experimentation.

Update `docs/02-cli/02-cli_README.md` and `docs/07-bench/07-bench_README.md` first. The CLI docs should describe the ordinary operator contract plainly and stop teaching the larger schema as the default mental model. The benchmark docs should explain that internalized Bucket 2 controls remain usable through internal settings payloads, QualitySuite patches, or other benchmark-lab paths, but are not part of normal day-to-day help or manual selection.

Then update any remaining docs that still narrate the old schema-shaped contract, especially places that talk about direct benchmark helper inputs, winner settings, or codex pipeline plumbing as if those were ordinary user-facing knobs.

Acceptance is that the docs have one coherent story: normal runs use the smaller operator contract, benchmark-lab flows have their own clearly labeled override surfaces, and the full schema is described as persistence/compatibility detail rather than the product surface.

### Milestone 5: Validation And Compatibility Proof

At the end of this milestone, the repo should prove five things together: Bucket 1 compatibility still works, Bucket 2 keys stay hidden/internal unless intentionally surfaced for lab use, the ordinary operator contract is explicit, the benchmark-lab contract is separate, and human-facing artifacts no longer mirror the full schema.

Start with the settings membership and projection tests in `tests/llm/test_run_settings.py`. Add coverage for the new contract helpers and for the distinction between operator projection, benchmark-lab projection, and full/raw persistence payloads.

Then update the command/help and adapter tests in `tests/cli/test_run_settings_adapters.py`, `tests/cli/test_cli_llm_flags.py`, and any related help-surface tests so they assert the smaller ordinary surface without accidentally blocking internal benchmark-lab override paths.

Finally, update manifest, winner-setting, analytics, and benchmark helper coverage in the affected test modules. Compatibility tests must load payloads that include both retired Bucket 1 keys and internal Bucket 2 keys, then prove that ordinary summaries stay small while the full payload still round-trips.

Acceptance is that focused tests pass, help surfaces and docs are smaller and clearer, and compatibility loading still works for old artifacts and internal benchmark payloads.

## Validation and Acceptance

The cleanup is complete when all of the following are true:

- Bucket 1 retired-key payloads still load without re-promoting those keys into ordinary summaries.
- Bucket 2 fields remain internal or benchmark-lab only and do not return to ordinary help, menus, or summaries.
- the repo has explicit helpers for the ordinary operator contract and the benchmark-lab contract,
- ordinary command/help/doc surfaces describe the smaller operator contract instead of the full persistence schema,
- manifests, winner settings, analytics summaries, and benchmark reports lead with the operator-facing story while still retaining raw payloads for reproduction,
- focused tests covering contract helpers, help surfaces, summaries, and compatibility loading pass.

The strongest manual proof is a before-and-after comparison showing that ordinary help and ordinary summaries stay small even though `RunSettings.from_dict(...)` still accepts older or internalized keys.

## Idempotence and Recovery

Implement this cleanup after the Bucket 2 baseline is stable. If the Bucket 2 interfaces are still moving, update this plan first and then continue. Do not try to “finish product contract cleanup” by partially duplicating Bucket 2 in a second place.

Make additive helper/projection changes first, then switch callers and docs, then tighten tests. That order keeps compatibility intact and makes partial progress safe to rerun.

If a summary or help-surface cleanup breaks an old benchmark or test fixture, do not re-expose the field as an ordinary operator knob just to quiet the failure. Preserve the full payload and add an explicit benchmark-lab or raw/debug path instead.

Because the repo already has historical benchmark roots and saved winner settings, do not rewrite old artifacts as part of this plan. Prove compatibility by loading them or by loading representative payloads that include retired and internal keys.

## Interfaces and Dependencies

The intended end-state should keep these interfaces explicit:

- In `cookimport/config/run_settings.py`, keep the full schema and existing public/internal metadata, but add explicit helpers for the ordinary operator contract and the benchmark-lab contract.
- In `cookimport/config/run_settings.py`, `RunSettings` should expose an operator-facing projection such as `to_operator_run_config_dict()` alongside the existing full/raw projection helpers.
- In `cookimport/config/codex_decision.py`, fixed-behavior and profile/benchmark contract helpers remain the canonical place for frozen defaults; this plan should consume that layer, not replace it.
- In `cookimport/config/run_settings_adapters.py`, adapter functions should bridge from the persistence schema to operator-facing and benchmark-lab-facing call shapes intentionally instead of mirroring the full schema by default.
- In `cookimport/cli.py` and `cookimport/labelstudio/ingest.py`, ordinary entrypoints should no longer require large signatures that resemble the full persistence schema.
