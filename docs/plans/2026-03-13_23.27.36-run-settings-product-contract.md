# Finish The Run-Settings Product Contract Cleanup

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan builds on `docs/plans/2026-03-13_22.55.03-run-settings-surface-cleanup.md`, which already retired `table_extraction`, introduced explicit `public` versus `internal` run-setting metadata, and proved the compatibility pattern for old payloads. The remaining work is to finish the cleanup described in the audit’s “Suggested cleanup direction” section instead of stopping at the first tranche.

## Purpose / Big Picture

After this work, the product should stop pretending that a 77-field persistence schema is the same thing as the operator-facing contract. A normal run should feel like choosing a top-tier profile plus a short list of real overrides such as extractor, OCR policy, Codex enablement, model, and workspace paths. Parser lab knobs, score thresholds, codex pass ids, debug toggles, and benchmark-only experiments should no longer dominate `stage`, `labelstudio-benchmark`, interactive menus, or run summaries.

The success case is easy to inspect. `cookimport stage --help` should read like a small product CLI rather than a subsystem lab. `cookimport labelstudio-benchmark --help` should separate ordinary benchmark operation from benchmark-lab tuning instead of surfacing internal matching and pass-pipeline knobs as if they were routine choices. New manifests and summaries should primarily show profile and operator overrides, while old run-settings payloads that still carry retired or internal keys must continue to load.

## Outcomes & Retrospective

This plan is drafted, not implemented. The main outcome so far is clarity about the remaining gap: the repo already has the right compatibility seam, but the operator-facing contract is still too large because CLI entrypoints, benchmark helper signatures, docs, and summaries continue to project the persistence schema outward. The plan below narrows that gap without throwing away historical artifact compatibility.

## Context and Orientation

`cookimport/config/run_settings.py` defines the full `RunSettings` schema. In this repository, that schema is used for loading saved JSON payloads, building stage and benchmark kwargs, writing manifests, driving interactive menus, and persisting benchmark winner settings. It is therefore both a runtime settings model and an archaeology format.

The first cleanup already changed that file so settings can be classified as `public`, `internal`, or retired legacy keys. The existing helpers include `public_run_setting_names()`, `internal_run_setting_names()`, `retired_legacy_run_setting_names()`, `RunSettings.to_public_run_config_dict()`, and `summarize_run_config_payload(...)`. That work retired `table_extraction` and moved obvious fake settings such as `benchmark_sequence_matcher`, `multi_recipe_trace`, `p6_emit_metadata_debug`, and the codex pass pipeline ids out of the default UI surface.

The remaining problem is where those settings still leak. `cookimport/cli.py` defines the main Typer commands and still exposes very large option signatures for `stage(...)` and `labelstudio_benchmark(...)`. `cookimport/labelstudio/ingest.py` repeats much of the same surface in direct Python helper signatures used by tests and benchmark orchestration. `cookimport/config/run_settings_adapters.py` still forwards nearly the full `RunSettings` schema into stage and benchmark call kwargs, which is good for compatibility but not yet good at defining a smaller product contract.

`cookimport/config/codex_decision.py` is already the main place where the repo pins current winner behavior. It defines top-tier profile contracts and benchmark contract patches for parser stack, codex enablement, and compact pass defaults. This matters because the medium-term cleanup should freeze more winners there instead of leaving parser and scoring clusters visible as daily choices forever.

This plan uses four plain-English terms:

- An “operator setting” is a knob a normal human should intentionally vary run to run.
- A “benchmark-lab setting” is a setting useful for benchmarking or regression forensics, but not part of the normal stage/import product story.
- An “internal setting” is a plumbing or debug seam that should never be treated as ordinary user-facing contract.
- The “persistence schema” is the full `RunSettings` model, which survives mainly so old payloads, manifests, and winner files still load.

The target state is not “delete every non-operator key.” The target state is “make the public product surface honest.” That means a smaller ordinary surface layered on top of the persistence schema.

## Plan of Work

### Milestone 1: Freeze The Real Operator Contract

At the end of this milestone, the repo should have an explicit answer to “what are the normal settings a human is supposed to care about?” and that answer should no longer be “all public `RunSettings` fields.”

Implement this in `cookimport/config/run_settings.py` with the lightest mechanism that stays explicit. Extend the current field metadata or helper layer so the code can answer at least these questions directly:

- which settings belong in the ordinary operator surface,
- which settings belong only in a benchmark-lab surface,
- which settings remain internal-only,
- which keys are retired but still accepted for compatibility.

Do not introduce a new heavyweight model tree. Explicit registries or metadata flags are enough. The important part is that downstream code can deliberately ask for the `stage` operator surface, the benchmark-lab surface, or the full persistence/debug surface.

Use the audit’s direction to define the target operator contract. The normal stage/import operator surface should be short and centered on profile-plus-overrides. The likely survivors are worker sizing, extractor choice, OCR policy, Codex enablement, Codex model/effort, Codex roots/workspace paths, tag catalog path, and maybe `warm_models`. Parser/scoring/web-schema tuning clusters should no longer be in the ordinary operator bucket once winners are accepted. If a field still needs to be adjustable for benchmarking, move it to benchmark-lab rather than keeping it in ordinary stage/import.

Acceptance is that the repo has explicit helper APIs for these surfaces, the ordinary operator count is materially smaller than the current 62-field bucket, and tests can assert the intended memberships.

### Milestone 2: Remove Near-Term Ordinary-Surface Leaks

At the end of this milestone, the repo should fully satisfy the audit’s near-term cleanup direction instead of only doing it halfway. `benchmark_sequence_matcher` should stop being an ordinary surfaced choice, codex pass pipeline ids should stop looking like normal run settings, and debug toggles should stop leaking through normal command help and docs.

Update `cookimport/cli.py` first. The ordinary `stage` and `labelstudio-benchmark` entrypoints should use the smaller operator surface, not the full persistence schema. Hidden options are not enough if the help text, docs, manifests, or call shapes still treat those fields as routine supported behavior. Remove or relocate:

- benchmark matcher overrides that are meant only for benchmark-lab use,
- direct codex pass pipeline id knobs,
- debug toggles such as `multi_recipe_trace` and `p6_emit_metadata_debug`,
- selective-retry plumbing toggles that only exist for benchmark repair or recovery.

Then update `cookimport/labelstudio/ingest.py`. The direct benchmark helper signatures should stop advertising those same fields as normal Python-call inputs. The cleanest route is to accept a `RunSettings` object or a smaller settings payload for ordinary callers, while preserving an explicit internal/debug override seam for tests or lab code that still needs deep control.

Update `cookimport/config/run_settings_adapters.py` last in this milestone so the adapter layer can build ordinary stage or benchmark kwargs from the smaller operator surface and route benchmark-lab/internal overrides intentionally rather than implicitly.

Acceptance is behavioral. `cookimport stage --help` and `cookimport labelstudio-benchmark --help` should read like product commands. Searching help or docs for `sequence-matcher`, `codex-farm-pipeline-pass`, `multi-recipe-trace`, and `p6-emit-metadata-debug` should no longer make those look like normal operator choices.

### Milestone 3: Collapse Tuning Clusters Into Fixed Internal Defaults

At the end of this milestone, the parser/scoring/web-schema tuning clusters should stop living in the ordinary operator contract. They can still exist in the persistence schema, but the product should behave as though the current winners have been chosen.

Use `cookimport/config/codex_decision.py` as the main home for these decisions. Expand the existing profile and benchmark contract patches so they pin the accepted winner stack for:

- parser stack choices such as section detector, multi-recipe splitter, instruction-step segmentation, ingredient parser/backend/unit policy, and Priority 6 extraction sub-backends,
- recipe scorer thresholds and similar quality gates,
- web-schema extractor and normalization defaults once the desired winner is chosen,
- any OCR or PDF tuning that should remain stable product behavior rather than a daily choice.

The point is not to eliminate every tuning code path immediately. The point is to move these clusters out of the everyday contract and into fixed internal defaults, with benchmark-lab overrides only where there is still a real need to compare alternatives.

Update docs and tests to reflect that the winners are now fixed product behavior. Old payloads containing these fields should still load, but new docs and manifests should stop implying that they are daily operator decisions.

Acceptance is that the ordinary operator surface no longer includes these clusters, top-tier profile patches state the frozen winners explicitly, and compatibility loading still accepts older payloads without re-promoting those settings into the public story.

### Milestone 4: Split Benchmark-Lab Tuning From Everyday Stage/Import Settings

At the end of this milestone, benchmark-only experimentation should still be possible, but it should be clearly separate from the everyday stage/import contract.

Define a benchmark-lab surface that is intentionally narrower than the full persistence schema but broader than the ordinary operator surface. This is where benchmark-only matcher controls, line-role experiment toggles, atomic splitter experiments, selective retry behavior, and any still-needed tuning knobs should live. The exact implementation can be a separate helper list in `run_settings.py`, a separate benchmark-only docs section, or explicit benchmark helper parameters, but the distinction must be visible in code and docs.

Update benchmark docs in `docs/07-bench/07-bench_README.md` and CLI docs in `docs/02-cli/02-cli_README.md` so “everyday run settings” and “benchmark-lab controls” are described separately. Keep the ordinary benchmark command help clean. If some lab controls still need to exist as flags, place them behind a clearly labeled benchmark-lab or debug section rather than mixing them into the normal operator narrative.

Acceptance is that a novice can tell which settings are for normal use and which ones are for benchmark experimentation, and benchmark reproducibility still works without presenting the full persistence schema as the product.

### Milestone 5: Reframe Manifests, Winner Settings, And Summaries Around The Smaller Public Surface

At the end of this milestone, run artifacts should communicate the smaller product contract by default while still preserving full debug information where needed.

Update manifest and summary code in `cookimport/cli.py`, `cookimport/labelstudio/ingest.py`, and any other summary writers to lead with the new operator surface rather than dumping the persistence schema. `summarize_run_config_payload(...)` should produce a clear profile-plus-overrides summary. If full persistence/debug payloads are still written, keep them under a clearly named raw/debug key instead of making them the default human-facing story.

Check the winner-settings flow in `cookimport/config/last_run_store.py` and the quality-suite save/load path. Winner settings should still round-trip through `RunSettings`, but the human-facing explanation of those winners should use the smaller operator contract plus benchmark-lab context where necessary.

Acceptance is that new summaries, manifests, and winner-setting explanations are much easier to read, and old payloads still remain loadable and comparable.

### Milestone 6: Docs, Tests, And Compatibility Proof

At the end of this milestone, the cleanup should be safe to merge because the docs teach the honest product surface, compatibility loaders still work, and the tests lock the new boundaries in place.

Update at least these docs:

- `docs/02-cli/02-cli_README.md` for the ordinary operator contract and benchmark-lab split,
- `docs/07-bench/07-bench_README.md` for benchmark-only tuning surfaces,
- `cookimport/llm/README.md` so codex pass pipeline ids are described as internal contract details instead of first-class run settings,
- `cookimport/staging/README.md` if debug-sidecar behavior moves further out of the everyday surface.

Update or add tests around:

- `tests/llm/test_run_settings.py` for operator, benchmark-lab, internal, and retired memberships,
- `tests/cli/test_run_settings_adapters.py` for stage/benchmark adapter projections,
- `tests/cli/test_cli_llm_flags.py` and related help-surface tests,
- benchmark helper tests under `tests/labelstudio/` that currently assume the large direct-call signature or ordinary visibility of internal settings,
- manifest/summary tests such as `tests/staging/test_run_manifest_parity.py`.

Compatibility tests must cover older payloads containing retired and now-internal fields, and they must prove that loading still succeeds without re-surfacing those keys in new ordinary summaries.

Acceptance is that targeted tests pass, the ordinary operator help is clearly smaller, benchmark-lab controls are intentionally separated, and old payloads still load.

## Validation and Acceptance

The cleanup is complete when all of the following are true:

- a normal human can inspect `cookimport stage --help` and see a short operator contract rather than parser and benchmark lab controls,
- benchmark help and docs clearly separate normal operation from benchmark-lab experimentation,
- new manifests and summaries primarily show profile plus intentional operator overrides,
- `RunSettings.from_dict(...)` still loads older payloads containing retired or internal keys,
- focused tests for settings membership, adapters, help surfaces, manifests, and benchmark helpers pass.

The strongest manual proof is a before/after count comparison plus help-surface diff. The ordinary operator surface should be materially smaller than the current `62`-field “public” list, even though the persistence schema remains larger.

## Idempotence and Recovery

This cleanup should be implemented as additive classification and projection changes first, then surface-pruning changes second. That order keeps compatibility intact and makes partial progress safe to rerun.

If a surface-pruning change breaks an old benchmark or test fixture, do not revert the compatibility loader. Instead, move the affected key to benchmark-lab or internal compatibility handling and keep ordinary summaries/help clean. The safe fallback is always: full schema still loads, smaller public projection stays honest.

Because the repo already has historical benchmark roots and winner settings, do not delete or rewrite old artifacts as part of this plan. Validate them through compatibility loading instead.

## Interfaces and Dependencies

The intended end-state should keep these interfaces explicit:

- In `cookimport/config/run_settings.py`, helpers must exist for the ordinary operator surface, the benchmark-lab surface, the internal surface, and the retired compatibility keys.
- In `cookimport/config/codex_decision.py`, top-tier and benchmark contract helpers must remain the canonical place where frozen winner defaults are patched into payloads.
- In `cookimport/config/run_settings_adapters.py`, stage and benchmark adapter functions must project the smaller operator surface intentionally and only route benchmark-lab/internal values when explicitly requested.
- In `cookimport/cli.py` and `cookimport/labelstudio/ingest.py`, ordinary user-facing entrypoints should no longer require giant signatures that mirror the persistence schema.
