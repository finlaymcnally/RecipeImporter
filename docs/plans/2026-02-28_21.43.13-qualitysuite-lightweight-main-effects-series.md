## Purpose / Big Picture

After this change, you will be able to run a short, structured benchmark series that answers: “which options seem best in each config category?” without waiting for a full permutation tournament. The workflow will first measure category-level winners, then test a combined “best of each category” setup, then run a small interaction smoke test to catch blatantly bad pairings. You can observe success by running one command and getting a final report with per-category winners, combined-result verdict, and explicit “interaction-risk” notes.

## Context and Orientation

QualitySuite currently evaluates experiment candidates through `bench quality-run` and top-tier gating through `scripts/quality_top_tier_tournament.py`. The system can already run in fast modes (sweeps off, limited seeds/candidates, exhaustive or race search). “Deterministic sweeps” here means extra pre-defined config variants; they increase runtime significantly and are not needed for fast triage.

For this plan:

- “Category” means one family of knobs, such as parser/preprocess, structure detection, ingredient handling, instruction segmentation, and yield handling.
- “Main effect” means “how this one category option performs on average,” while holding the rest at baseline.
- “Interaction smoke test” means a tiny set of additional runs that check whether “best + best” combinations are obviously unstable; this is not full interaction modeling.

Key current files:

- `scripts/quality_top_tier_tournament.py`: tournament orchestration and fold/gate logic.
- `cookimport/bench/quality_runner.py`: quality-run orchestration and search strategy execution.
- `cookimport/cli.py`: `bench` command registration and quality command wiring.
- `data/golden/bench/quality/experiments/*.json`: candidate definitions.
- `data/golden/bench/quality/thresholds/*.json`: fold and gate defaults.
- `docs/07-bench/runbook.md` and `docs/07-bench/07-bench_README.md`: operator-facing benchmark docs.

## Plan of Work

Implement a lightweight orchestration entrypoint as `cookimport bench quality-lightweight-series` that runs three rounds and writes one summary JSON plus one report Markdown file. Reuse existing `bench quality-discover`, `bench quality-run`, and `bench quality-leaderboard` commands under the hood so metric semantics remain identical to existing quality tooling.

The lightweight entrypoint must load a versioned profile JSON that declares category membership, round defaults, scoring thresholds, risk thresholds, and ablation/perturbation recipe. This profile is the contract surface for experimentation; code should validate it strictly and fail with clear errors on unknown experiment IDs or illegal cross-category key overlap.

Round 1 performs category screening with low seed count and deterministic scoring. Round 2 tests baseline, category winners, and one synthetic combined configuration. Round 3 runs a small interaction-smoke set around the combined candidate and emits explicit risk/no-risk findings by category pair.

Implementation should remain deterministic and local-only:

- no Codex Farm,
- no LLM-based parsing/cleaning,
- no new scorer engine.

## Scoring, Tie-Break, and Risk Rules (Explicit)

All winner/risk decisions must be driven by metrics in `summary.json` experiment rows. For each experiment, aggregate mean metrics across evaluated folds:

- `mean_practical_f1_macro`
- `mean_strict_f1_macro`
- `mean_source_success_rate`

Compute deltas against baseline means from the same round:

- `practical_delta = experiment.mean_practical_f1_macro - baseline.mean_practical_f1_macro`
- `strict_delta = experiment.mean_strict_f1_macro - baseline.mean_strict_f1_macro`
- `source_success_delta = experiment.mean_source_success_rate - baseline.mean_source_success_rate`

Category winner ranking rule (deterministic):

1. Compare `practical_delta` with epsilon `0.0010`.
2. If still tied, compare `strict_delta` with epsilon `0.0010`.
3. If still tied, compare `source_success_delta` with epsilon `0.0050`.
4. If still tied, higher absolute `mean_practical_f1_macro`.
5. If still tied, higher absolute `mean_strict_f1_macro`.
6. Final tie-break: lexicographically smallest `experiment_id`.

Baseline-as-winner rule:

- If a category’s top non-baseline candidate has `practical_delta <= 0.0` and `strict_delta <= 0.0` and `source_success_delta <= 0.0`, record that category winner as `baseline` (no override applied in composition).

Confidence guard rule:

- Trigger when top two non-baseline candidates in a category satisfy:
  - `abs(practical_delta_diff) < 0.0015`
  - and `abs(strict_delta_diff) < 0.0015`
- Run one additional fold with next unused seed and recompute winner for that category only.
- If no additional seed exists, keep winner by deterministic tie-break and set `confidence=low` for that category.

Round 2 combined verdict rule:

- `PASS` when combined deltas vs baseline satisfy all:
  - `practical_delta >= -0.0020`
  - `strict_delta >= -0.0020`
  - `source_success_delta >= -0.0100`
- Otherwise `FAIL`.

Round 3 interaction-risk rule:

- Mark a smoke variant as `RISK` when any is true versus `combined_main_effects` means:
  - `practical_f1_gain >= 0.0030`
  - `strict_f1_gain >= 0.0030`
  - `source_success_rate_gain >= 0.0200`
- Mark `NO_OBVIOUS_RISK` otherwise.

## Combined Merge Contract (Explicit)

The synthetic `combined_main_effects` run settings patch is built from winning category patches in this exact category order:

1. `parser`
2. `structure`
3. `ingredient_missing_unit`
4. `instruction_segmentation`
5. `yield_mode`

Merge rules:

- If category winner is `baseline`, skip that category (no patch contribution).
- For each `run_settings_patch` key:
  - If key is unset in combined patch, set it.
  - If key is already set to the same value, keep one value and append provenance.
  - If key is already set to a different value, fail immediately with `illegal_overlap` error and report:
    - conflicting key
    - first category/experiment and value
    - second category/experiment and value
- Illegal overlap is a hard error; do not silently pick a winner by order.

Pre-merge validation:

- Every candidate ID referenced by profile categories must exist in the experiments file.
- Every non-baseline candidate ID may belong to only one category in the profile.
- Baseline experiment ID must exist in experiments file.

## Idempotence and Recovery

If a run is interrupted, rerunning the same command should either:

- create a fresh timestamped run by default, or
- support resume via explicit `--resume-series-dir` semantics (recommended in scope) with compatibility checks on:
  - profile file path/content hash,
  - experiments file path/content hash,
  - thresholds file path/content hash,
  - baseline experiment ID.
