## Surprises & Discoveries

- Observation: the saved `2026-04-14_17.18.19` single-book `source-rows` benchmark was distorted by evaluator bugs that are already fixed on current `main`.
  Evidence: re-running `cookimport.bench.eval_source_rows.evaluate_source_rows(...)` on the saved artifacts changed `vanilla` from `0.506329 / 0.000` to `0.645851 / 0.002137`, and changed `codex-exec` from `0.467511 / 0.033235` to `0.527707 / 0.58862`.

- Observation: `recipeimport-lo0` appears to be based on an older expectation rather than the current intended contract.
  Evidence: `docs/10-llm/10-llm_README.md` and `docs/04-parsing/04-parsing_readme.md` now explicitly state that line-role inline repair resumes the prior transcript, and tests in `tests/parsing/test_canonical_line_roles_runtime.py` assert resumed repair behavior.

- Observation: `recipeimport-153` also looks stale at first glance.
  Evidence: `cookimport/llm/knowledge_stage/task_file_contracts.py` now partitions grouping units by `max_units_per_batch` and `max_evidence_chars_per_batch`, and the LLM docs describe grouping batches as a current behavior rather than a future fix.

- Observation: `recipeimport-ktb` was a real live bug in the inline knowledge runtime, distinct from the already-fixed same-session taskfile path.
  Evidence: `workspace_run.py` built grouping repair packets from validated answers only, so a parsed grouping attempt that failed semantic validation disappeared from the next repair packet's `previous_groups` / `previous_answer` context. The runtime now keeps a separate repair-context answer map sourced from the latest parsed edited task file, and focused tests confirm the second repair packet retains the first failed grouping attempt.

- Observation: `recipeimport-co0` now looks stale too.
  Evidence: saved benchmark artifacts on current `main` already include Codex runs where `Avocado Salad Matrix` lands as `OTHER` rather than `RECIPE_TITLE`, for example `data/golden/benchmark-vs-golden/2026-04-13_15.53.38/.../line_role_predictions.jsonl`.
