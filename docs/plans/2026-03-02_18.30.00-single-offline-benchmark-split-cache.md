# ExecPlan: Single-Offline Benchmark Split-Artifact Reuse

This ExecPlan is a living document and must always be updated while work is in flight.
It follows the structure required by [`docs/PLANS.md`](docs/PLANS.md). Any future editor should be able to implement this without any prior context.

## Purpose / Big Picture

Single-offline benchmark mode currently re-runs the expensive split-conversion stage for every variant, even when variants differ only in downstream pipeline choice (for example, vanilla and codexfarm). This plan implements true shared pre-processing so one split job can be produced once and reused by both variants, while keeping prediction/evaluation artifacts isolated per variant.

From a user perspective this removes duplicate progress bars, duplicated I/O, and duplicated wall-time for a single source in codex-vs-vanilla flow, without changing the location, shape, or scoring semantics of final variant outputs.

## Surprises & Discoveries

1. ` _interactive_single_offline_variants()` returns two variants only when `llm_recipe_pipeline` is `codex-farm-3pass-v1`: `vanilla` (llm off) then `codexfarm`. The variant order is already deterministic and suitable for shared pre-pass semantics.

2. `_interactive_single_offline_benchmark()` currently does `labelstudio_benchmark(**variant_kwargs)` inside the loop with no pre-pass. There is no guard against repeating split conversion.

3. `labelstudio_benchmark` in `prediction-only` or full run path unconditionally calls `generate_pred_run_artifacts(...)` for prediction generation and does not expose a split-only mode.

4. `generate_pred_run_artifacts` already computes and stores rich timing (`prep_started`, `split_wait_started`, `split_active_started`, `split_active_finished`, `post_finished`) and writes deterministic `run_manifest` metadata, so the split cache contract can be anchored to existing timing + artifact schema without introducing a second custom format from scratch.

5. The all-method benchmark already carries split-convert key semantics in `_all_method_split_convert_input_key_payload`, `_build_all_method_split_convert_input_key`, and `_all_method_split_convert_input_key_payload`.
This is a strong signal that a dedicated single-offline split cache should reuse that naming/field pattern, but with a narrower key that intentionally excludes LLM-only knobs.

## Context and Orientation

Terminology is defined here so the plan can be executed by someone new to the repository.

Single-offline benchmark means the interactive flow where one source is selected and multiple benchmark variants are run in sequence to compare outcomes. Relevant files:

- `cookimport/cli.py`
  - `_interactive_single_offline_variants()`: determines variant list (`vanilla`, `codexfarm`).
  - `_interactive_single_offline_benchmark()`: executes each variant by calling `labelstudio_benchmark`.
  - `labelstudio_benchmark(...)`: top-level CLI path for prediction+evaluation orchestration.
  - `_run_all_method_prediction_once()`: existing working reference for prediction result reuse, lock file pattern, cache summaries, and retry-safe behavior.

- `cookimport/labelstudio/ingest.py`
  - `generate_pred_run_artifacts(...)`: currently performs import-convert-split and then builds all downstream artifacts.
  - Existing manifest/return contract includes paths like `run_root`, `manifest_path`, `extracted_archive_json`, `stage_block_predictions_path`.

- `cookimport/config/run_settings_adapters.py`
  - `build_benchmark_call_kwargs_from_run_settings(...)`: canonical place to keep run-settings-to-cli-kwargs consistency.

Split conversion means importer chunking and merge work (`_plan_parallel_convert_jobs`, merge results, build `ConversionResult`) before LLM pipeline and before freeform prediction tasks. The intent is to cache that prefix only.

## Milestones

1. Contract and cache structure milestone. At the end of this milestone there is a defined schema, keying function, lock handling, and manifest validation contract for single-offline split cache entries.

2. Warm split execution milestone. At the end of this milestone single-offline variants can reuse a previously built conversion artifact by bypassing split/rebalance in second/later variants.

3. Variant wiring milestone. At the end of this milestone the existing two-call loop in `_interactive_single_offline_benchmark` becomes a one-split + N-predict path for N variants.

4. Controls and observability milestone. At the end of this milestone users can force/disable cache, override cache directory, and observe hit/miss + timing in timeseries and comparison output.

5. Validation milestone. At the end of this milestone behavior is deterministic, old behavior remains for one-variant mode, and regressions are impossible to reintroduce silently.

## Plan of Work

### Step 1 — Add dedicated single-offline split cache constants and contracts

In `cookimport/cli.py`, add constants near the existing all-method reuse constants:

- `SINGLE_OFFLINE_SPLIT_CACHE_SCHEMA_VERSION = "single_offline_split_cache.v1"`
- `SINGLE_OFFLINE_SPLIT_CACHE_LOCK_SUFFIX = ".lock"`
- `SINGLE_OFFLINE_SPLIT_CACHE_ROOT` environment fallback key if needed.
- Split-field list such as `SINGLE_OFFLINE_SPLIT_CONVERT_INPUT_FIELDS`.

Contract goal: the split cache key must be stable and independent from LLM pipeline differences but stable across all split-impacting settings.

Add these helpers in the same file:

- `_single_offline_split_cache_key_payload(*, source_file: Path, source_hash: str | None, pipeline: str | None, run_settings: RunSettings)`  
- `_build_single_offline_split_cache_key(*, source_file, source_hash, pipeline, run_settings)`  
- `_single_offline_split_cache_entry_path(cache_root: Path, split_cache_key: str)`  
- `_load_single_offline_split_cache_entry(cache_path: Path, expected_key: str)`  
- `_write_single_offline_split_cache_entry(cache_path: Path, payload: dict[str, Any])`  
- `_single_offline_split_cache_lock_path(cache_path: Path)`  
- `_acquire_single_offline_split_cache_lock`, `_release_single_offline_split_cache_lock`, `_wait_for_single_offline_split_cache_entry`.

The payload should include:
- schema version,
- source path and source hash,
- selected pipeline fields used by split/conversion,
- absolute cache artifact directory,
- conversion run timings,
- path to cached conversion result,
- count of split jobs/artifacts expected for health checks.

### Step 2 — Add a split artifact emit/load path in prediction generation

In `cookimport/labelstudio/ingest.py`, refactor `generate_pred_run_artifacts(...)` to support an optional warm cache input and a mode that writes a cache entry.

Proposed signature extension:

- `split_cache_entry_dir: Path | None = None`
- `split_cache_key: str | None = None`
- `single_offline_split_cache_mode: str = "auto"` (optional)

Behavior:

- If `split_cache_entry_dir` is provided and a valid cache entry exists:
  - skip `_plan_parallel_convert_jobs()` and merge execution,
  - load cached conversion payload (`conversion_result`, `run_config`, and extracted archive material),
  - keep timing and `run_root` semantics for the target variant,
  - continue with current downstream pipeline (`run_codex_farm_recipe_pipeline`, processed outputs, task build, evaluate/import flow).

- If cache is missing or invalid:
  - run current conversion path,
  - after split conversion completes, persist conversion cache entry in `split_cache_entry_dir` in the contract from Step 1,
  - continue existing flow unchanged.

Important constraint: this must remain 1:1 with existing behavior for existing arguments when `split_cache_entry_dir` is not set.

### Step 3 — Wire warm cache pre-pass into single-offline orchestration

In `cookimport/cli.py` inside `_interactive_single_offline_benchmark()`:

1. Determine variants with `_interactive_single_offline_variants(selected_benchmark_settings)` as today.
2. If variant count is 1, preserve current behavior exactly.
3. If variant count > 1:
   - build `split_cache_root`:
     - explicit flag `--single-offline-split-cache-dir` if provided,
     - otherwise default under benchmark evaluation root (for example `session_root / ".split-cache"`).
   - compute source hash and split cache key from selected source and split-relevant settings from first variant.
   - attempt lock-guarded load of cache entry.
   - if miss, execute one warm split build by invoking a dedicated split path through `labelstudio_benchmark` parameters (or new wrapper in `ingest.py`) and write cache.
4. Pass the resolved `single_offline_split_cache_path` into each variant `variant_kwargs` call.
5. Keep variant output directories (`session_root / variant_slug`) separate.

### Step 4 — Keep one-variant behavior unchanged

In `_interactive_single_offline_benchmark`:

- Add explicit branch `if len(variants) == 1: bypass single-offline split cache prep and use existing `labelstudio_benchmark(**variant_kwargs)` path`
- Ensure the per-variant comparison artifact writer still writes only when both runs succeed.
- Do not alter `vanilla`/`codexfarm` success and failure handling.

### Step 5 — Add explicit controls and user-facing docs text

In `cookimport/cli.py`, command option additions on `labelstudio_benchmark`:

- `--single-offline-split-cache-mode / --no-single-offline-split-cache`
- `--single-offline-split-cache-force` (rebuild even on hit)
- `--single-offline-split-cache-dir`
- optional `--single-offline-split-cache-strategy` if you prefer tri-state semantics.

Default should remain no-op unless interactive mode has more than one variant.

In documentation path (`docs/` entries that describe benchmark CLI and single-offline workflow):
- describe when cache is used,
- list invalidation triggers,
- document how to inspect cache entries and disable cache for a single run.

### Step 6 — Add split reuse telemetry

For observability add per-variant prediction metadata and one summary file:

- `prediction_cache_hit`: bool, for single-offline only.
- `single_offline_split_cache_key`: key string.
- `single_offline_split_cache_reused`: bool.
- `single_offline_split_cache_seconds`: derived from pre-pass metric.
- `single_offline_split_cache_source_file_hash`: optional hash of current source.

Add these to:

- `processing_timeseries_prediction.jsonl` for each variant run in single-offline.
- single-offline comparison JSON (`codex_vs_vanilla_comparison.json`) maybe as `metadata.single_offline_split_cache`.
- optional run manifest notes inside eval output for each variant.

### Step 7 — Recovery and safety behavior

- If split cache entry has invalid schema or missing files, log and rebuild.
- If rebuild fails, clear only single-offline cache entry and continue by doing fresh conversion in the current run.
- If lock contention exceeds wait timeout, fall back to building split in the current process, then overwrite/refresh stale lock-bearing entry as per lock protocol.
- Never copy cached artifacts into the wrong variant directory; cache payload path remains read-only and variant-specific outputs remain in `variant_eval_output`.

## Validation and Acceptance

Acceptance is behavior-first:

1. Single source, two variants:
   - first run with no cache: both variants execute, and logs show split conversion once.
   - second run with cache warm: second variant logs split-cache hit; no duplicate split conversion.
   - final outputs remain in separate variant directories.

2. One variant:
   - `len(variants) == 1` path must run exactly as before and must not attempt split cache prep.

3. Cache invalidation:
   - if a cached manifest is missing required files or schema mismatch, run must log reason and rebuild split stage.

4. Failure mode:
   - if second variant fails, first variant output and manifest remain intact, comparison JSON either absent or clearly marks incomplete comparison.

5. Reproducibility:
   - same source + same split settings should generate same cache key.
