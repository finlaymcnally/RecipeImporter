# What looked wrong

In run `data/golden/eval-vs-pipeline/2026-02-23_12.25.07`, the report says:

- input gold: `828`
- deduped gold: `702`
- removed rows: `126`

At first glance this seems inconsistent with:

- focus-only task text (Label Studio shows only labelable rows)
- no obvious duplicated rows in Label Studio UI
- requested overlap set to `0`

# Key distinction: exported rows are spans, eval units are block ranges

`labelstudio-export` freeform output (`freeform_span_labels.jsonl`) stores one row per span region from the selected annotation result.

Relevant path:
- `cookimport/labelstudio/export.py`

Important consequence:
- two different spans in the same block are both valid exported rows
- they are not "duplicate export rows" unless they have identical segment+offset+label

In this run:
- exact duplicate spans by `(segment_id,label,start_offset,end_offset)`: `0`
- exported span rows: `828`

# Why eval reports dedupe anyway

Current freeform eval normalizes each gold span to a block range:

- read touched block indices
- convert to `start_block_index=min(...)`, `end_block_index=max(...)`

Then dedupe groups by:
- `(source_hash, source_file, start_block_index, end_block_index)`

Relevant path:
- `cookimport/labelstudio/eval_freeform.py` (`load_gold_freeform_ranges`, `_gold_dedupe_key`, `_dedupe_gold_ranges`)

So multiple span fragments inside the same block collapse to one scoring unit. That is where dedupe removals come from.

In this run:
- block-range duplicate groups: `72`
- rows in those duplicate groups: `194`
- dedupe removed rows: `126`
- conflict groups (different labels on same block range): `12`

# Why Label Studio UI can still look "clean"

The UI showing "every block labeled" is not contradictory:

- UI is showing annotated regions in task text
- many tasks have one region per focus block
- some tasks have multiple regions inside a single block

Those multi-region same-block cases are valid freeform annotations, but block-range eval treats them as one gold unit.

# Why overlap `0` became effective overlap `10`

For freeform with focus blocks, ingest enforces a floor so focus windows remain contiguous across tasks:

- floor = `segment_blocks - segment_focus_blocks`

For this run:
- `segment_blocks=40`
- `segment_focus_blocks=30`
- floor = `10`

Manifest shows:
- `segment_overlap_requested=0`
- `segment_overlap_effective=10`

Relevant path:
- `cookimport/labelstudio/ingest.py` (`focus_overlap_floor`, overlap adjustment)
- `cookimport/labelstudio/freeform_tasks.py` (`resolve_segment_overlap_for_target`)

# Do we still need gold dedupe today?

For current `eval-vs-pipeline` block-range scoring: yes.

Without dedupe:
- one block split into many spans would over-weight that gold block
- conflicts in duplicated block ranges would produce unstable scoring semantics

Dedupe currently provides a deterministic block-level comparison surface for pipeline-vs-gold evaluation.

# If we want to remove this behavior

We need a separate span-native evaluation mode (offset-based gold and offset-based predictions), instead of block-range projection.

That would make:
- "no dedupe unless exact duplicate spans"
- freeform metrics align directly with region-level annotation intent

Until then, dedupe removals in `gold_dedupe` are expected whenever a block has multiple span regions.

# Quick troubleshooting checklist

When dedupe numbers look surprising:

1. Check import manifest requested vs effective overlap.
2. Count exact duplicate spans in `freeform_span_labels.jsonl`.
3. Count block-range duplicate groups using touched block indices.
4. Inspect `gold_dedupe.conflicts` in eval report for label disagreements.
5. Decide whether analysis goal is block-level (current eval) or span-level (future mode).
