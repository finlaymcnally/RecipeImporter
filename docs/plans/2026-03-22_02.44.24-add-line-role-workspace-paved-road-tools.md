# Add Line-Role Workspace Paved-Road Tools

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a line-role worker should start in its bounded workspace, open one obvious repo-written task row, use one obvious repo-written helper CLI, and finish line-role outputs without rebuilding deterministic label-expansion logic or packet summaries from scratch. The worker should spend its effort deciding whether to keep or override a label, not retyping the same `jq` transform every run.

The user-visible success condition is that a generated line-role worker root is self-explanatory by inspection. A novice should be able to open the workspace and see the approved loop immediately: open `worker_manifest.json`, open `current_task.json`, consult `OUTPUT_CONTRACT.md`, use `tools/line_role_worker.py` to orient or scaffold, validate locally, and install to `out/<task_id>.json`. A later benchmark rerun should show fewer exploratory shell commands and fewer quoting or shape failures, but this plan does not require a live rerun until the user explicitly approves token spend.

## Surprises & Discoveries

- Observation: line-role still needs the strongest deterministic scaffold of the three stages.
  Evidence: the saved run repeatedly expanded compact label codes such as `L0` and `L1` into full label strings with handwritten `jq` transforms. That work is deterministic and should belong to the repository.

- Observation: the shared runner already solved part of the orientation problem.
  Evidence: `cookimport/llm/codex_exec_runner.py` now gives non-leased worker sessions `current_task.json`, so line-role no longer needs a separate orientation artifact to be discoverable.

- Observation: the original plan’s multiple helper scripts were more specialized than necessary.
  Evidence: line-role can use the same top-level helper verbs as the other two stages if `scaffold` is defined to mean “write the deterministic seed output for this task.”

- Observation: line-role still needs stage-local validation.
  Evidence: even with shared helper UX, line-role is the only stage where structural validity is “one row per owned atomic row, same order, allowed full label names only.”

- Observation: the shared worker-manifest shell examples are a discoverability aid, not a stable ordering contract.
  Evidence: while implementing `tools/`, the manifest example list remained semantically correct but reordered one existing `jq '.metadata' current_task.json` example ahead of the example write call. Treating the exact list order as important created a false-negative test.

## Context and Orientation

The line-role worker is the first long-lived direct-exec LLM session in the three-stage paved-road chain. It labels compact row packets under `line-role-pipeline/runtime/line_role/workers/worker-*/`. Each task packet lives in `in/<task_id>.json`. Its `rows` list contains compact tuples of `atomic_index`, deterministic `label_code`, and current line text. The worker reads `hints/<task_id>.md` and writes `out/<task_id>.json` using full label strings such as `INGREDIENT_LINE`, `INSTRUCTION_LINE`, or `OTHER`.

The shared direct-exec workspace contract lives in `cookimport/llm/codex_exec_runner.py`. For non-leased line-role sessions the worker already gets:

- `worker_manifest.json`
- `current_task.json`
- `assigned_tasks.json`
- `assigned_shards.json`
- `hints/`, `in/`, `out/`, `scratch/`
- optional `examples/`

This plan adds `OUTPUT_CONTRACT.md`, a valid example output, and `tools/line_role_worker.py`.

The line-role output contract is smaller than the other two stages. Each output file is one JSON object with a single top-level key `rows`. Each entry under `rows` must have `atomic_index` and `label`. The file must contain exactly one row per owned input row, in the same order, with no extra keys and no invalid label names.

The key files are:

- `cookimport/parsing/canonical_line_roles.py`, which builds line-role prompts, worker roots, examples, task payloads, and validation
- `cookimport/llm/codex_exec_runner.py`, which owns shared workspace mirroring, the worker manifest, and the path policy
- `tests/parsing/test_canonical_line_roles.py`, which should prove the worker root is discoverable and line-role validation still holds
- `tests/llm/test_codex_exec_runner_workspace.py`, which should prove the shared runner mirrors and advertises the helper surface correctly

“Scaffold” in this plan means “generate the deterministic seed output for one task by expanding label codes into full label names while preserving row order exactly.” “Install” means “validate a saved draft and then write it to the declared `metadata.result_path`.”

## Plan of Work

Implement the line-role paved road as one deterministic helper module plus one shared runner improvement.

First, add a new helper module under `cookimport/parsing/` for deterministic line-role workspace tooling. A good name is `cookimport/parsing/line_role_workspace_tools.py`. It should expose pure functions for:

- loading and summarizing task rows from `assigned_tasks.json` and `current_task.json`
- generating the deterministic seed output for one task by expanding compact label codes into full labels
- validating a saved output draft against the line-role output contract
- installing a validated draft to the task row’s declared `metadata.result_path`
- rendering a self-contained `tools/line_role_worker.py` script body

Second, extend `cookimport/parsing/canonical_line_roles.py` so each worker root writes:

- `OUTPUT_CONTRACT.md`
- `examples/valid_line_role_output.json`
- `tools/line_role_worker.py`

The helper script must be runnable as `python3 tools/line_role_worker.py ...` from the workspace root and must not depend on repository-relative imports at runtime.

Third, use the same helper verbs as the other two stages:

- `overview` prints the ordered queue concisely
- `show <task_id>` prints task-local paths, atomic-range summary, deterministic-label mix, and compact metadata
- `scaffold <task_id> [--dest <path>]` emits the deterministic seed output for that task
- `check <json_path>` validates a saved draft without writing to `out/`
- `install <json_path>` reruns validation and writes the file to the declared `metadata.result_path`

Fourth, extend `assigned_tasks.json` metadata only as needed so `overview` and `show` can report the cheap facts workers kept re-deriving manually, such as `input_path`, `hint_path`, `result_path`, owned row count, first and last atomic index, and deterministic label counts. Keep `in/<task_id>.json` authoritative.

Fifth, update the line-role worker prompt so the local loop is:

1. open `worker_manifest.json`
2. open `current_task.json`
3. if more context is needed, run `python3 tools/line_role_worker.py overview` or `show <task_id>`
4. open the named `hints/<task_id>.md` and `in/<task_id>.json`
5. if useful, run `python3 tools/line_role_worker.py scaffold <task_id> --dest scratch/<task_id>.json`
6. edit the draft only where the deterministic output needs correction
7. run `python3 tools/line_role_worker.py check scratch/<task_id>.json`
8. run `python3 tools/line_role_worker.py install scratch/<task_id>.json`

The prompt must explicitly say that long handwritten `jq` transforms are unnecessary because the deterministic seed path already exists.

Sixth, extend `cookimport/llm/codex_exec_runner.py` so `tools/` is mirrored and advertised in `worker_manifest.json` through `tools_dir` and `mirrored_tool_files`. That shared runner change is part of the cross-stage unification and should be done once for all three stages.

## Idempotence and Recovery

This plan is intentionally additive. Line-role worker roots are rebuilt from scratch, so regenerating `OUTPUT_CONTRACT.md`, `examples/`, and `tools/line_role_worker.py` is safe to repeat. The helper’s `scaffold` command should write to stdout by default or to an explicit destination only. The helper’s `check` command must be read-only. The helper’s `install` command must be the only step that writes to the final declared output path.

If a partial implementation leaves the prompt referring to helper commands that are not yet mirrored into the workspace, stop and make the workspace surface consistent before running tests. If helper discoverability ends up split awkwardly between the prompt and the manifest, prefer the manifest as the authoritative index and keep the prompt short.
