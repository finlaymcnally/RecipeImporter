# Reduce Shell-Happy Single-File Workers

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

After this change, the main single-file workspace workers for recipe, knowledge, and canonical line-role will stop treating broad shell use as a normal way to do the job. The happy path will become explicit and narrow: inspect `task.json` with repo-owned helpers, apply answers with repo-owned helpers or direct in-file answer edits, run the repo-owned same-session handoff helper, and stop. Workers that ignore that paved road and start dumping the whole task file, writing ad hoc heredoc Python, or exploring with generic shell commands will be warned and then stopped deliberately instead of being treated as acceptable “local shell” behavior.

The user-visible proof is deterministic and local. Focused tests will show that single-file worker prompts teach the helper-first path, that the watchdog classifies full-file dumps and ad hoc Python transforms as discouraged single-file behavior, that allowed repo-owned helper commands remain valid, and that the single-file watchdog escalates repeated discouraged shell behavior without breaking the intended handoff flow. No live CodexFarm benchmark or book processing run is required for acceptance.

## Context and Orientation

In this repository, a “single-file workspace worker” is a live `codex exec` session whose visible working directory contains one repo-written `task.json` and a small `AGENTS.md` instruction file. The worker is supposed to read the task file, fill only the allowed `answer` fields, run a repo-owned same-session handoff helper, and stop. Recipe, knowledge, and canonical line-role now all use this basic shape on their main happy path.

The shared runner for those workers lives in `cookimport/llm/codex_exec_runner.py`. That module builds the sterile execution workspace, writes the shared worker instructions, parses command telemetry, and exposes the command classifier used by watchdog code. The stage-specific single-file prompts live in `cookimport/llm/recipe_stage_shared.py`, `cookimport/llm/knowledge_stage/recovery.py`, and `cookimport/parsing/canonical_line_roles/planning.py`. The deterministic helper for reading and writing `task.json` lives in `cookimport/llm/editable_task_file.py`. The stage-owned same-session validation helpers live in `cookimport/llm/recipe_same_session_handoff.py`, `cookimport/llm/knowledge_same_session_handoff.py`, and `cookimport/parsing/canonical_line_roles/same_session_handoff.py`.

The key term in this plan is “shell-happy.” Here that means a single-file worker chooses generic shell commands or ad hoc mini-programs when the repo already provides a narrow helper-first path. Examples include dumping the whole `task.json` with `cat`, writing a one-off `python3 - <<'PY'` transform that rewrites the answer payloads, or exploring the workspace with generic shell commands before using the repo helper commands. The problem is not that shell exists at all; the problem is that single-file workers are supposed to operate on one repo-authored file with one repo-owned validation helper, yet the current runtime still tolerates generic shell routes that are noisy, expensive, and harder to supervise.

This plan is intentionally complementary to the helper-first startup work already described in `docs/plans/2026-03-31_13.27.54-task-file-helper-first-worker-start.md`, but it is self-contained. That earlier plan improves the paved road by making the repo-owned helpers more attractive. This plan hardens the contract around that paved road by changing prompts, command classification, and watchdog behavior so single-file workers are no longer rewarded for broad shell improvisation.

## Plan of Work

The first milestone is to make the command classifier aware of the worker contract. In `cookimport/llm/codex_exec_runner.py`, introduce a distinction between single-file worker commands and the broader workspace-worker command policy. The final code should be able to say, in deterministic telemetry terms, whether a command is a preferred repo-owned helper command, a tolerated orientation command, a discouraged full-task-file dump, a discouraged ad hoc transformation command, or a true hard-boundary violation. The classifier must keep the existing boundary checks for forbidden tools and forbidden paths, but it must stop labeling `cat task.json`, broad `sed -n ... task.json`, and inline heredoc Python rewrites as simply acceptable local shell behavior in the single-file contract.

The second milestone is to make the intended path explicit everywhere the worker learns the job. In `cookimport/llm/codex_exec_runner.py`, adjust the shared single-file `AGENTS.md` text so it describes the startup sequence as: summary helper first, narrow unit helper if needed, explicit answer application helper if needed, same-session handoff helper, stop. In `cookimport/llm/recipe_stage_shared.py`, `cookimport/llm/knowledge_stage/recovery.py`, and `cookimport/parsing/canonical_line_roles/planning.py`, update the worker prompts so they explicitly forbid whole-file dumps and ad hoc shell mini-programs. The wording must not merely say “prefer not to use shell”; it must name the discouraged behaviors that were actually observed: full `task.json` dumps, `python3 - <<'PY'` transforms, and generic orientation commands when the helper commands already provide the needed information.

The third milestone is to make watchdog behavior match the new contract. Recipe, knowledge, and line-role each already have workspace-worker watchdog code. Extend those watchdogs so they respond to the new single-file command classifications. The intended behavior is warning-first and explicit: a worker that uses one low-value shell command can be warned in telemetry, but repeated discouraged shell behavior or an egregious ad hoc transform should trigger termination with a new reason code that makes the problem obvious. If the worker already produced valid outputs before that termination, the existing recovery logic should preserve those outputs rather than throwing them away. This plan is about stopping messy behavior earlier, not about making success brittle.

The fourth milestone is to improve the bulk-answer paved road so the model has less reason to invent shell transforms. `cookimport/llm/editable_task_file.py` already supports `--apply-answers-file`, but the prompts and helper metadata should present that path as the normal batch-edit mechanism. If a worker needs to set many answers, the repo-owned answer-apply path should be more obvious than writing a one-off Python rewrite script in the shell. If an additional deterministic helper is needed to export a minimal answer template or answer skeleton, add it here, but do not let deterministic code infer semantic labels.

The fifth milestone is proof. Update `tests/llm/test_codex_exec_runner.py`, `tests/llm/test_codex_exec_runner_workspace.py`, `tests/llm/test_recipe_phase_workers.py`, `tests/llm/test_knowledge_orchestrator_contracts.py`, and `tests/parsing/test_canonical_line_roles.py` so the new contract is pinned directly. The tests must stop encoding “shell-happy is acceptable” as the expected behavior for single-file workers. They should instead prove that repo-owned helpers remain acceptable, that discouraged shell routes are classified distinctly, and that the stage prompts now teach the narrower contract.

## Validation and Acceptance

This plan is accepted only when all of the following are true.

The first acceptance criterion is command vocabulary. The runtime must be able to distinguish, in deterministic code and tests, between repo-owned helper commands and discouraged shell-heavy single-file worker behavior. A generic label of “tolerated local shell command” is no longer enough for the bad cases we are trying to eliminate.

The second acceptance criterion is prompt clarity. The shared single-file worker instructions and the recipe, knowledge, and line-role prompts must describe the helper-first path in plain language and must explicitly name the discouraged patterns that are off the happy path: dumping the whole task file, broad shell orientation, and ad hoc heredoc or inline-program answer rewrites.

The third acceptance criterion is watchdog enforcement. Repeated discouraged shell behavior in a single-file worker must result in an explicit warning or termination reason code that is visible in tests and runtime telemetry, while the intended helper commands remain accepted.

The fourth acceptance criterion is preserved deterministic authority. Any new helper added here may summarize, show, or apply explicitly provided answers, but it must not infer semantic labels or overwrite model decisions.

The fifth acceptance criterion is local proof. The focused pytest commands listed above must pass in `.venv` without requiring a live benchmark or full book-processing run.

## Idempotence and Recovery

The helper read commands and prompt edits are idempotent. Re-running them should not change task state. The stricter command classifier and watchdog changes must also be idempotent in the sense that the same command sequence yields the same warning or kill result every time.

If the new policy proves too strict for a legitimate single-file helper command, relax the classifier only for that named helper shape and add a focused regression test in the same edit. Do not reopen the generic “any local Python shell script is fine” loophole, because that would collapse this plan’s purpose. If a watchdog escalation test becomes brittle, prefer deterministic fake snapshots over live subprocess timing.

## Artifacts and Notes

The motivating bad behavior from the current runtime can be summarized in plain language:

    The worker already had repo-owned helper commands for summary, narrow unit reads, answer application, and same-session validation.
    Even so, real line-role workers still read huge chunks of `task.json`, wrote large ad hoc Python transforms in the shell, ran the same-session helper only after that custom shell work, and sometimes needed watchdog recovery despite having valid final outputs.

The intended end-state is equally simple:

    Start with `editable_task_file --summary`.
    If more detail is needed, use `--show-unit` or `--show-unanswered`.
    Apply answers directly or with `--apply-answers-file`.
    Run the stage-owned same-session handoff helper.
    Stop.

If the worker deviates from that narrow path without a compelling repo-owned reason, the runtime should treat that as drift rather than as acceptable normal behavior.
