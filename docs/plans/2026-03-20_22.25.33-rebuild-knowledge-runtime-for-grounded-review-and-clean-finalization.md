## Purpose / Big Picture

After this change, the optional non-recipe knowledge phase should behave like a trustworthy review system instead of a chaotic worker farm. A long-lived worker should stay on the review task it was given, produce grounded packet outputs, avoid uncontrolled shell automation, and either finish cleanly or fail closed in a way the operator can understand. If a worker or follow-up path goes bad, retries and repairs must stay bounded, stale subprocesses must be cleaned up, and the phase must still write a usable final or partial artifact package.

The proof must be observable without spending model tokens. A developer should be able to run deterministic tests plus a saved-run replay against the Salt Fat Acid Heat artifact roots and see that the runtime now reports task-packet progress honestly, writes complete phase artifacts, bounds follow-up churn, and classifies worker outcomes clearly. Only after those no-token checks pass should a live Codex rerun be considered, and that rerun still requires explicit user approval.

- Decision: this plan treats the current knowledge main-worker contract itself as flawed, not merely the watchdog settings around it.
  Rationale: the biggest product failures are not only “workers got killed.” They are also “workers were allowed to batch-generate low-trust outputs,” “completed workers were malformed,” and “the follow-up layer never converged.” Relaxing or tightening watchdog policy alone does not fix that.

  Rationale: giving one worker a full multi-packet task manifest encourages shell loops, metadata-driven shortcuts, and correlated mistakes across dozens of packets. The runtime should expose one current packet at a time, validate it, then advance deliberately.

  Date/Author: 2026-03-20 / Codex

## Outcomes & Retrospective

This plan is not implemented yet. The current outcome is a design direction and acceptance bar, not landed code. The most important change in scope versus the earlier March 20 plans is that this plan no longer treats the problem as “watchdog policy plus UI polish.” The runtime needs a deeper correction: repo-owned packet scheduling, grounded-output enforcement, bounded follow-up policy, and guaranteed finalization.

If implemented successfully, this plan should turn the knowledge phase from a fragile experiment into a routine stage that can be replayed, inspected, interrupted, and trusted.

## Context and Orientation

The knowledge phase is the optional non-recipe review stage described in `docs/10-llm/nonrecipe_knowledge_review.md`. Its job is to look at outside-recipe text from Stage 7 and decide which packet content is real reusable cooking knowledge and which content should stay classified as `other`. In this repository, a “shard” is a larger ownership bundle assigned to one worker, and a “task packet” is a smaller bundle inside that shard that eventually becomes one output object. The current runtime exposes many task packets at once to a long-lived worker session.

The main runtime lives in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. That file currently does too much: it owns progress reporting, worker launch, watchdog callbacks, output harvesting, follow-up spawning, and final staging promotion decisions. The direct Codex transport lives in `cookimport/llm/codex_exec_runner.py`. The compact prompt and schema live in:

- `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md`
- `llm_pipelines/schemas/recipe.knowledge.v1.output.schema.json`

The benchmark command surface that shows the broken UI and handles interruption lives in:

- `cookimport/cli.py`
- `cookimport/labelstudio/ingest.py`
- `cookimport/core/progress_messages.py`

The saved failing run that motivates this plan is rooted at:

- benchmark root: `data/golden/benchmark-vs-golden/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/`
- knowledge root: `data/output/2026-03-20_21.44.20/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-20_21.44.56/raw/llm/saltfatacidheatcutdown/knowledge/`

The most important current symptoms from that run are:

- `485` task packets were planned but only `339` top-level task outputs existed at shutdown.
- `4` main workers were watchdog-killed for command execution.
- `2` main workers “completed” only because their process exited, not because the runtime accepted a clean final answer.
- the phase spawned approximately `199` structured follow-up calls and got only `6` clean wins (`5` repaired tasks and `1` validated watchdog retry).
- `9` follow-up calls were still marked running when the benchmark was killed.
- the stage never wrote its normal run-level outputs.

Any implementation that does not directly change those facts is not enough.

## Plan of Work

### Milestone 1: Build a deterministic knowledge replay harness and acceptance scoreboard

Before changing production behavior, create a deterministic harness that can walk a saved knowledge run root and summarize what happened in stage terms rather than in raw file counts. This harness is not a human-facing product. It is a repo-owned acceptance seam used by tests and by future debugging. It should live in a small new helper module, likely `cookimport/llm/knowledge_runtime_replay.py`, with tests beside the current knowledge orchestrator tests.

The replay helper should read:

- `task_manifest.jsonl`
- `worker_assignments.json`
- shard `live_status.json`
- worker `events.jsonl`
- `repair_status.json` and `repair_live_status.json`
- `watchdog_retry/status.json` and `watchdog_retry/live_status.json`
- top-level worker `out/*.json`

From those artifacts it must derive one machine-readable summary object with the facts an operator actually cares about: total packets, valid top-level outputs, invalid top-level outputs, watchdog-killed main workers, malformed completions, repaired packets, validated retries, failed repairs, stale follow-ups, and missing final stage outputs. This replay summary becomes the no-token acceptance oracle for the saved March 20 run and for future regressions.

This milestone matters because the current failure is easy to misread. Without a replay layer, every new bug report turns back into manual directory archaeology.

### Milestone 2: Replace the many-task-open worker contract with repo-driven packet leasing

This is the core runtime redesign. The current knowledge main-worker path gives one worker a full list of assigned task packets and a directory full of authoritative inputs. That is what enabled the bad behaviors seen on March 20: `jq` loops across all packets, blanket keep/knowledge outputs, lane-metadata shortcuts, and giant correlated mistakes.

Replace that with a repo-driven packet leasing model. A long-lived worker session should still exist, but at any moment it should have exactly one authoritative packet to review. The runtime should expose a single current packet file, a single current hint file, and a single place to write the answer. After the runtime validates that answer, it should move the next packet into place and update the worker-facing state. The worker may still know high-level counts such as “48 packets remain,” but it must not be handed the full task queue as authoritative inputs for free-form batch automation.

The likely implementation shape is:

- add a small knowledge main-worker state helper, likely in a new file such as `cookimport/llm/knowledge_runtime_state.py`
- define one explicit packet lifecycle: `pending`, `leased`, `accepted`, `retry_pending`, `repair_pending`, `failed_terminal`
- write only the currently leased packet into the mirrored worker input files
- maintain repo-owned accepted output storage outside the worker scratch root
- only advance to the next packet after the current packet reaches an accepted or terminal state

This is a meaningful refactor, but it directly attacks the worst failure mode. A worker that only has one packet in front of it has much less room to decide “I will manufacture all 48 outputs with one shell loop.”

### Milestone 3: Tighten the prompt, schema, and validator around grounded review

The current compact prompt and schema allow too much creative freedom in the wrong places. Some workers responded by turning `rc` into freeform prose, fabricating broad summary language, or collapsing difficult packets into generic keep-or-drop shapes. The runtime needs a stricter review contract.

Keep the compact surface but make it materially harder to fake. The knowledge output contract should change in these ways:

- preserve the existing top-level compact shape so the rest of the runtime does not need a full rewrite
- restrict reviewer reason codes to a closed, repo-owned enum, still stored in `rc` if keeping the field name is cheaper than renaming it
- require that snippet evidence be exact quotes or exact block-text slices from the current packet input, not paraphrases
- reject outputs whose block decisions do not cover every owned block in order
- reject outputs where `u=true` but every block is mechanically marked as knowledge without any discriminating evidence
- reject outputs where the reviewer reasons only from metadata or headings when the packet contains substantive prose
- explicitly reject top-level file corruption cases such as heredoc terminators, mixed prose before or after JSON, wrong bundle version, or missing packet id echo

The worker-facing prompt in `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md` should be rewritten to explain the new rules plainly:

- review only the current packet
- do not invent batch plans
- do not use deterministic metadata as a substitute for reading block text
- do not mass-mark every block the same way unless the text actually supports it
- one grounded packet answer is better than a fast synthetic batch

The schema in `llm_pipelines/schemas/recipe.knowledge.v1.output.schema.json` should be updated only as much as needed to support the tighter contract. The semantic validator in the knowledge ingest/orchestrator path must become the main enforcement seam, because schema validation alone will never catch the “plausible but fake” outputs.

### Milestone 4: Redesign follow-up policy so recovery is bounded, cancelable, and worth it

The March 20 run showed a recovery layer that kept working long after it stopped being useful. This plan must replace open-ended churn with a small, explicit policy.

The runtime should introduce a repo-owned follow-up state machine per packet:

- at most one main attempt
- at most one watchdog retry if the main attempt died in a retryable way
- at most one repair attempt if the packet output is near-miss but fixable
- no new follow-up once a packet is accepted or terminally failed
- cancel or reap any still-running follow-up subprocesses once the packet is no longer active

This policy should be encoded in the knowledge runtime helper state, not scattered across ad hoc `if` branches. The current folder layout can stay similar, but the decision logic must become explicit and testable.

The runtime also needs orphan cleanup. If a shard is already complete or the phase is being interrupted, any still-running repair or watchdog-retry subprocess for that shard should be terminated and marked as canceled. The March 20 stale follow-up calls are unacceptable in a finished or aborted phase.

Acceptance for this milestone is not “fewer follow-ups in spirit.” It is concrete: the saved March 20 replay must show no more than the configured cap per packet, no running follow-ups after phase teardown, and a clear terminal reason for every packet.

### Milestone 5: Make progress and UI report the truth operators need

The earlier narrow progress plans fixed some pieces but not the overall truth problem. The knowledge phase progress model must now report:

- packet progress as the primary `task_current/task_total`
- shard completion counts in detail lines
- running main workers separately from running follow-ups
- descriptive worker rows that say which shard a worker owns and what packet or follow-up it is currently on
- whether a worker is on main review, watchdog retry, repair, or canceled cleanup

The structured progress payload should continue to flow through `cookimport/core/progress_messages.py::format_stage_progress(...)` so the benchmark UI and timeseries stay on one transport. The changes belong primarily in the knowledge orchestrator and only secondarily in `cookimport/cli.py` if wording or row rendering needs to improve.

The specific operator-visible failure to eliminate is the shape:

    Running codex-farm non-recipe knowledge review... task 0/10 | running 10

For the saved run shape, a truthful UI after this change would look more like:

    Running codex-farm non-recipe knowledge review... task 233/485 | running 4 | follow-ups 3
    completed shards: 3/10
    worker 07: saltfatacidheatcutdown.ks0006.nr packet 2/48 repair
    worker 10: saltfatacidheatcutdown.ks0009.nr packet 47/48 main review

The exact text can differ, but the information content cannot be watered down.

### Milestone 6: Guarantee full and partial phase finalization

The knowledge phase must always write run-level outputs, even when the run is ugly. A clean finish should produce the normal phase artifacts. An interrupted or partially failed finish should produce a parallel partial package that explicitly says what is missing and why.

At a minimum, the knowledge root should always end with some form of:

- `phase_manifest.json`
- `telemetry.json`
- `failures.json`
- `promotion_report.json`
- `knowledge_manifest.json`

If the phase is interrupted before promotion, those files should still exist and say so clearly. They must point to the preserved worker-local evidence and summarize packet-level outcomes. The benchmark root should likewise get an interrupted or partial status artifact instead of only a dead timeseries tail.

This milestone also includes real `SIGINT` handling at the benchmark orchestration level. The correct product behavior for Ctrl+C is “stop launching new work, reap child work, write partial artifacts, exit clearly.” If exact signal plumbing proves messy, catching `KeyboardInterrupt` at the right orchestration seams is an acceptable first implementation so long as the observable behavior matches that product requirement.

## Validation and Acceptance

The implementation is complete when all of the following are true.

First, a deterministic replay of the saved `2026-03-20_21.44.20` knowledge root produces one clear summary object that reports the same failure classes found manually in the post-mortem. That replay test becomes the permanent regression anchor for future debugging.

Second, the knowledge main-worker path is repo-driven one packet at a time. A test must prove that a worker cannot access an authoritative queue of all current packet inputs for free-form batch production and that the runtime only advances to the next packet after the current one is accepted or terminally failed.

Third, malformed top-level packet outputs are caught immediately and classified precisely. The worker `001` heredoc-`EOF` corruption shape must fail deterministically in tests rather than merely causing a later repair avalanche.

Fourth, packet outputs that are syntactically valid but semantically fake must also fail. The tests should explicitly reject at least these patterns:

- blanket “every block is knowledge” packet answers when the input is mixed
- metadata-only or lane-based reasoning that ignores block text
- paraphrased snippets that are not exact packet evidence
- wrong bundle version or missing packet id echo

Fifth, recovery is bounded. For a synthetic broken packet set and for the saved-run replay, the runtime must show the configured maximum number of retries and repairs per packet, no orphaned running follow-ups after phase teardown, and explicit terminal reasons for each failed packet.

Sixth, progress is truthful. A knowledge stage with `485` packets must report packet progress, not shard progress, and the worker rows must distinguish main review from follow-up work.

Seventh, the phase always finalizes some run-level artifact package. A clean run produces the normal outputs, while an interrupted run produces partial outputs that clearly explain what completed, what failed, and what is missing.

## Idempotence and Recovery

This work should be implemented additively and in small, test-backed steps. The replay harness comes first so there is always a stable oracle for what the broken run looked like. If the packet-leasing refactor becomes too large for one change, land it behind a knowledge-stage-only internal toggle and switch the knowledge runtime to the new path only after the deterministic tests pass.

If the stricter semantic validator initially rejects too much, prefer adding explicit failure classifications and fixture coverage rather than weakening the groundedness rules back into permissiveness. The point of this redesign is to fail closed and visibly when the model is not actually reviewing the packet.

If interruption handling proves awkward at the signal layer, catch `KeyboardInterrupt` at the benchmark orchestration seams first and write the partial phase package there. Do not postpone partial finalization while trying to design a perfect signal framework.

Do not delete the March 20 saved artifacts. This plan intentionally depends on them as a frozen evidence bundle.
