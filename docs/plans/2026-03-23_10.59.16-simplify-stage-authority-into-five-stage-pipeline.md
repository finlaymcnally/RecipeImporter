# Simplify stage authority into a five-stage pipeline

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with [docs/PLANS.md](/home/mcnal/projects/recipeimport/docs/PLANS.md).

## Purpose / Big Picture

After this change, the post-import pipeline should be understandable in one blunt sentence: extract, decide recipe boundaries, refine recipes, route nonrecipe text, then make final knowledge decisions. A human should be able to look at the run artifacts and answer three questions without reading code: which stage decided recipe ownership, which stage decided that some outside-recipe text was obvious junk and would never be revisited, and which stage made the final semantic `knowledge` versus `other` decision for everything else.

The user-visible gain is architectural clarity rather than a new end-user feature. The current runtime already works, but it is hard to debug because the same text can be reinterpreted several times by different layers and the same container can mean different things at different moments. After this plan lands, each stage will own exactly one kind of decision, stage names will match the actual mental model, and provisional views will no longer masquerade as final authority.

This plan explicitly excludes importer simplification. The user is already handling importer scope reduction separately. This plan starts after importers have produced extracted text blocks and any weak hints they still emit.

The target pipeline is intentionally blunt:

1. `extract`
2. `recipe-boundary`
3. `recipe-refine`
4. `nonrecipe-route`
5. `knowledge-final`

These are stage names, not marketing names. Each name means exactly one authority step in the runtime and in the artifacts.

## Outcomes & Retrospective

This plan is not implemented yet. The immediate result today is a single self-contained migration plan that treats the current codebase as a transitional state and defines one explicit destination architecture for the post-import path. The most important outcome already achieved is conceptual: the target pipeline is no longer “whatever `import_session.py` happens to do.” It is now a five-stage contract with one authority step per stage.

The primary risk is migration width. This plan touches runtime coordination, intermediate data structures, artifact naming and meaning, tests, and docs. The mitigation is to execute it in a sequence that preserves behavior while tightening ownership one stage at a time: first create explicit stage-owned structures, then move code into them, then update projections and artifacts, then delete the old overloaded seams.

When implementation begins, this section must be updated at the end of every major milestone with what changed, what stayed intentionally compatible, and which old seams were deleted.

## Context and Orientation

This repository currently has a strong idea of authority but a weak shape for expressing it. The main runtime path for `cookimport stage` begins in [cookimport/cli.py](/home/mcnal/projects/recipeimport/cookimport/cli.py), which delegates per-file work to [cookimport/cli_worker.py](/home/mcnal/projects/recipeimport/cookimport/cli_worker.py), which in turn sends almost every file through [cookimport/staging/import_session.py](/home/mcnal/projects/recipeimport/cookimport/staging/import_session.py). That last file is the shared coordinator for most of the meaningful stage logic.

Several terms in this plan have precise meanings:

“Extracted blocks” means the ordered text units that come out of importer extraction. They are the raw text stream the rest of this plan starts from. Importer simplification is out of scope here; assume the user’s parallel importer work will leave behind cleaner extracted blocks and weaker early hints.

“Recipe boundary” means the one authoritative answer to: which parts of the book are recipes and which parts are not. In the current repo, this answer is assembled from atomized lines, deterministic labels, optional line-role correction, grouped recipe spans, and rebuilt recipe candidates. Those pieces are spread across [cookimport/parsing/label_source_of_truth.py](/home/mcnal/projects/recipeimport/cookimport/parsing/label_source_of_truth.py), [cookimport/parsing/canonical_line_roles.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles.py), and [cookimport/parsing/recipe_span_grouping.py](/home/mcnal/projects/recipeimport/cookimport/parsing/recipe_span_grouping.py).

“Recipe refine” means improving the contents of already-owned recipes. In the current repo this is the optional recipe Codex stage plus the deterministic final draft shaping in [cookimport/llm/codex_farm_orchestrator.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_orchestrator.py), [cookimport/staging/draft_v1.py](/home/mcnal/projects/recipeimport/cookimport/staging/draft_v1.py), and [cookimport/staging/writer.py](/home/mcnal/projects/recipeimport/cookimport/staging/writer.py). The crucial rule for this plan is that recipe refinement may never take ownership of text that recipe-boundary did not already mark as recipe.

“Nonrecipe route” means deciding which outside-recipe text is overwhelming obvious junk and can become final `other` immediately, and which outside-recipe text is reviewable and must go to the later knowledge stage. The current code for this lives mostly in [cookimport/staging/nonrecipe_stage.py](/home/mcnal/projects/recipeimport/cookimport/staging/nonrecipe_stage.py). This stage is the modern Stage 7. The user clarified an important invariant: if this stage says some outside-recipe text is obvious junk, no later stage will ever look at it again.

“Knowledge final” means the only stage allowed to decide final semantic `knowledge` versus final `other` for reviewable outside-recipe text. In the current repo this lives in [cookimport/llm/codex_farm_knowledge_jobs.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_jobs.py), [cookimport/llm/codex_farm_knowledge_orchestrator.py](/home/mcnal/projects/recipeimport/cookimport/llm/codex_farm_knowledge_orchestrator.py), and the writer/reporting surfaces that persist its results.

The current pain comes from two structural facts.

First, the same runtime object changes meaning mid-flight. `ConversionResult` is a broad importer-shaped model defined in [cookimport/core/models.py](/home/mcnal/projects/recipeimport/cookimport/core/models.py). It is useful for importer output, but the current stage runtime keeps mutating it even after ownership has changed. The field `non_recipe_blocks` is the clearest example: it starts life as importer leftovers, later becomes the post-boundary outside-recipe set, and later still becomes the final routed authority view.

Second, the current coordinator mixes stage boundaries. [cookimport/staging/import_session.py](/home/mcnal/projects/recipeimport/cookimport/staging/import_session.py) does not merely orchestrate; it performs boundary building, optional recipe correction, Stage 7 routing, optional knowledge refinement, chunk generation, table extraction, report rebuilding, and final writes. A novice cannot tell where one authority decision stops and the next begins.

This plan fixes those two facts directly. It does not merely rename folders. It changes the runtime structure so that the code path itself teaches the same mental model as the docs:

1. `extract`
2. `recipe-boundary`
3. `recipe-refine`
4. `nonrecipe-route`
5. `knowledge-final`

The desired ownership rules are:

- `extract` may extract text and preserve weak hints, but it may not claim final recipe ownership.
- `recipe-boundary` is the only stage that decides recipe ownership.
- `recipe-refine` may only improve already-owned recipe content.
- `nonrecipe-route` may finalize excluded obvious junk as `other` and route everything else to review.
- `knowledge-final` is the only stage that may decide final `knowledge` versus `other` for reviewable nonrecipe text.

The desired artifact rules are:

- every stage should write outputs whose names match their certainty level
- no file should hide both provisional and final meanings under the same field without making that distinction explicit
- no downstream projection should mistake routed-but-unreviewed text for semantically reviewed text

## Plan of Work

Begin by making the target contract real in code before deleting old seams. The first implementation step is to create a small stage-owned runtime data model in a new module under `cookimport/staging/`, for example `cookimport/staging/pipeline_runtime.py`. This module should define explicit plain-English dataclasses or pydantic models for the post-import path. At minimum, define one bundle for each stage boundary:

- `ExtractedBookBundle`
- `RecipeBoundaryResult`
- `RecipeRefineResult`
- `NonrecipeRouteResult`
- `KnowledgeFinalResult`

These names are prescriptive. The exact internal fields may vary, but the ownership meaning may not. Each structure must be able to exist on its own without depending on mutable hidden knowledge from the previous structure. The key goal is to stop using `ConversionResult` as the universal mutable carrier after extraction.

`ExtractedBookBundle` should hold only post-import extracted facts plus optional weak importer hints that survive the user’s separate importer simplification. It should include the ordered archive blocks, source identity, raw artifacts, report seed data, any provisional recipe candidates or hints the importer still emits, and enough information to reproduce split-merge provenance. It must not claim final recipe or knowledge authority.

`RecipeBoundaryResult` should become the single product of the current label-first stack. It should own the atomized lines, final line-role labels, authoritative block labels, grouped recipe spans, accepted recipe-side block indices, rejected pseudo-recipe spans, and the rebuilt recipe candidates that are now genuinely recipe-owned. It should also own the outside-recipe block set that remains after recipe ownership is finalized. This result replaces the current pattern where label-first mutates a `ConversionResult` and later code must remember that the recipe boundary already changed.

`RecipeRefineResult` should take a `RecipeBoundaryResult` and produce recipe outputs that are richer but still ownership-compatible. It may add recipe corrections, ingredient-step mappings, tags, intermediate overrides, and final draft overrides. It must not add new recipe-owned blocks or steal text from outside-recipe blocks. This stage should make the “recipe Codex is refinement only” rule impossible to violate accidentally.

`NonrecipeRouteResult` should take the outside-recipe side from `RecipeBoundaryResult` and do only two things. First, it should produce an explicit final set of excluded obvious-junk blocks with reasons such as navigation, endorsement, publishing metadata, or page furniture. These exclusions are final `other` immediately and forever. Second, it should produce an explicit reviewable set containing all remaining outside-recipe blocks. This stage must not assign final semantic `knowledge` to that reviewable set. It may keep deterministic convenience metadata for chunking and routing, but that metadata must be clearly labeled as routing or review-candidate metadata, never as final semantic authority.

`KnowledgeFinalResult` should take the reviewable set from `NonrecipeRouteResult` and produce the only final semantic `knowledge` versus `other` answer for that reviewable set. It should also preserve which reviewable rows were not reviewed because the knowledge stage was disabled, interrupted, missing output, or semantically rejected. Those rows may still show up as practical final `other` in some output surfaces, but the stage metadata must label them as unreviewed fallback rather than semantically reviewed `other`.

Once those structures exist, refactor [cookimport/staging/import_session.py](/home/mcnal/projects/recipeimport/cookimport/staging/import_session.py) into a thin orchestrator that calls stage-owned functions in order. The first landing does not need to move every line immediately, but it must introduce explicit stage functions in stage-owned modules:

- `build_extracted_book_bundle(...)`
- `run_recipe_boundary_stage(...)`
- `run_recipe_refine_stage(...)`
- `run_nonrecipe_route_stage(...)`
- `run_knowledge_final_stage(...)`

These names are also prescriptive. The current logic already exists in fragments. The main job of this milestone is to gather the fragments into stage-shaped seams and make the orchestrator linear and readable.

`run_recipe_boundary_stage(...)` should absorb today’s label-first stack. It should call the current atomization, line-role, block-label promotion, recipe-span grouping, and rebuilt-recipe creation code from the existing parsing modules, but its output must be a `RecipeBoundaryResult`, not another mutated importer-shaped `ConversionResult`.

`run_recipe_refine_stage(...)` should wrap the current optional recipe Codex path plus deterministic draft preparation surfaces. It should accept the recipe-owned outputs from recipe-boundary and return recipe refinements only. If the recipe LLM is off, this result should still exist and should explicitly say refinement was deterministic-only.

`run_nonrecipe_route_stage(...)` should wrap the current Stage 7 behavior in [cookimport/staging/nonrecipe_stage.py](/home/mcnal/projects/recipeimport/cookimport/staging/nonrecipe_stage.py), but its contract must become more blunt than the current mixed map. It should produce:

- `excluded_final_other_blocks`
- `reviewable_nonrecipe_blocks`
- `review_exclusion_reason_by_block`
- chunking/table-routing helper metadata

If there is a convenience merged map for output readability, keep it secondary and clearly labeled. The authoritative data structure must separate excluded-final from reviewable-unresolved.

`run_knowledge_final_stage(...)` should consume only `reviewable_nonrecipe_blocks`. It should never see the excluded-final set. This makes the user’s Stage 7 clarification enforceable in code rather than only in docs. The knowledge stage should return explicit final semantic decisions plus unreviewed fallbacks and the current snippet/reporting surfaces.

After the runtime structure is in place, clean up the overloaded read/write surfaces. In [cookimport/staging/writer.py](/home/mcnal/projects/recipeimport/cookimport/staging/writer.py), stop asking generic `ConversionResult` for several post-boundary meanings. Writer entrypoints should become stage-aware. The first target is to make the stage artifacts and run summaries draw from the new stage results directly instead of reverse-inferencing from one mutable object.

The key artifact rewrite is:

- `recipe-boundary` artifacts should clearly represent recipe ownership and leftover outside-recipe text
- `nonrecipe-route` artifacts should clearly separate excluded final other from reviewable unresolved
- `knowledge-final` artifacts should clearly separate reviewed final semantic authority from unreviewed fallback

Existing filenames may be preserved initially when needed, but their field layout must become certainty-readable. A novice opening `08_nonrecipe_spans.json` or `09_knowledge_outputs.json` must be able to answer:

- Was this block excluded as obvious junk?
- Was this block reviewable?
- Was it actually reviewed by knowledge?
- Is its current category final semantic authority or fallback?

Next, clean up projection seams. [cookimport/staging/stage_block_predictions.py](/home/mcnal/projects/recipeimport/cookimport/staging/stage_block_predictions.py), [cookimport/labelstudio/ingest.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest.py), and any manifest/report builders must project only from the new stage-owned final authority structures. They must never infer final meaning from routed-but-unreviewed convenience maps.

Then clean up terminology. The runtime, docs, and artifacts should converge on:

- `chunk` means deterministic parser chunk
- `task` means one worker-owned review unit
- `worker` means one workspace-worker session or runner-owned execution unit
- `packet` should be removed from the happy path unless a compatibility shim still needs the word internally

This is especially important in the knowledge stage docs and workspace helper surfaces because stale vocabulary alone is enough to re-teach the wrong architecture.

Finally, update docs after code, not before. The living docs that must be updated when implementation lands are:

- `docs/01-architecture/01-architecture_README.md`
- `docs/04-parsing/04-parsing_readme.md`
- `docs/05-staging/05-staging_readme.md`
- `docs/10-llm/10-llm_README.md`
- if needed, `cookimport/llm/README.md`

Those docs should present the five-stage model explicitly and should retire any wording that suggests that post-import authority is still spread across several unnamed sublayers.

## Validation and Acceptance

The implementation is acceptable only if the runtime teaches the five-stage model directly.

The first acceptance criterion is structural. A novice opening the post-import code path should be able to point to exactly one exported function for each of:

- `extract`
- `recipe-boundary`
- `recipe-refine`
- `nonrecipe-route`
- `knowledge-final`

The second acceptance criterion is authority clarity.

`recipe-boundary` must be the only stage that decides recipe ownership. No later stage may add recipe-owned blocks or reclassify nonrecipe text into recipes.

`recipe-refine` must improve recipes only. If the recipe LLM is enabled, it may change recipe contents but not the set of recipe-owned blocks.

`nonrecipe-route` must finalize excluded obvious junk immediately as `other`, with explicit exclusion reasons, and those rows must never be included in knowledge review input.

`knowledge-final` must be the only stage that decides final semantic `knowledge` versus `other` for reviewable nonrecipe text.

The third acceptance criterion is data-shape clarity.

There must no longer be one shared field whose meaning changes from importer leftovers to post-boundary outside-recipe text to final routed authority. The post-import runtime must use explicit stage-owned structures or fields for those different meanings.

The fourth acceptance criterion is artifact clarity.

A novice opening the run artifacts must be able to distinguish:

- excluded final-other rows
- reviewable nonrecipe rows
- reviewed final knowledge authority
- unreviewed fallback rows

without reading code.

The fifth acceptance criterion is projection safety.

Benchmark scoring, Label Studio projection, and writer summaries must project only from explicit final-authority structures. They must not mistake routed-but-unreviewed text for semantically reviewed text.

The minimum local test bar is:

- `tests/staging/test_nonrecipe_stage.py` passes
- `tests/staging/test_stage_observability.py` passes
- `tests/staging/test_stage_block_predictions.py` passes
