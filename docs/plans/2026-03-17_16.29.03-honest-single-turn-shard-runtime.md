## Purpose / Big Picture

After this change, choosing `5 / 5 / 5` in the RecipeImport CLI should mean roughly fifteen real Codex task calls for the three shard-v1 phases: five recipe shard calls, five line-role shard calls, and five knowledge shard calls. Today the CLI implies that prompt targets control prompt count, but the live runtime still labels shard workers as `structured_loop_agentic_v1` without actually selecting CodexFarm's real runtime-mode flag, so live execution can expand into many more Codex turns than the operator expected. The user-visible goal is that shard-v1 phases run through CodexFarm's classic one-shot task farm transport, one output per shard input file, and that the saved runtime manifests and audits say so plainly.

## Context and Orientation

The relevant runtime files all live in `cookimport/llm/` plus one parsing entrypoint. A “shard” is one bounded unit of recipe, knowledge, or line-role work. A “worker” is the RecipeImport-owned sandbox that receives one or more shard files in a local folder. A “runtime mode” is CodexFarm's own choice of execution transport. In CodexFarm, `classic_task_farm_v1` means classic one-shot task processing and `structured_loop_agentic_v1` means persistent session reuse across multiple queued tasks.

`cookimport/llm/phase_worker_runtime.py` writes the shard manifests, worker sandboxes, and promotion reports for shard-v1 phases. It currently calls the runner with `runtime_audit_mode="structured_loop_agentic_v1"` even though it is not selecting the true CodexFarm runtime mode.

`cookimport/llm/codex_farm_runner.py` is the subprocess adapter around `codex-farm process`. It currently uses the `runtime_audit_mode` string only to force `--workers 1` and to annotate saved audit metadata. It does not pass `--runtime-mode`.

`cookimport/llm/fake_codex_farm_runner.py` and `scripts/fake-codex-farm.py` are the local zero-token stand-ins used by tests and dry-run rehearsals. They need to accept the same runtime-mode contract so a fake run exercises the same shape as the live run.

The three active shard-v1 call sites are:

- `cookimport/llm/codex_farm_orchestrator.py` for recipe correction
- `cookimport/llm/codex_farm_knowledge_orchestrator.py` for knowledge harvest
- `cookimport/parsing/canonical_line_roles.py` for line-role labeling

The key behavior to preserve is output validation and promotion. The transport should change from session-style labeling to classic single-shot task execution, but per-shard output schemas, per-shard proposals, and deterministic post-processing must remain intact.

Update focused tests in `tests/llm/test_phase_worker_runtime.py`, `tests/llm/test_codex_farm_orchestrator_runner_transport.py`, and any line-role telemetry tests that assert runtime audit payloads.
