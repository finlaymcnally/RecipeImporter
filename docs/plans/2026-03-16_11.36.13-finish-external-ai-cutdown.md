# Finish External-AI Cutdown Burn-Down

This ExecPlan is a living document. Keep `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` current as work lands.

## Purpose / Big Picture

Remove the last non-historical dependency on the old `pass1` / `pass2` / `pass3` benchmark topology from the external-AI cutdown path. The remaining surface is `scripts/benchmark_cutdown_for_external_ai.py` and its paired bench file. The goal is for that path to consume the current single-correction recipe topology directly: semantic stage rows, `recipe_manifest.json` stage state, and `recipe_correction_audit` artifacts.

When complete, the cutdown script should produce the same external-AI handoff artifacts without reconstructing or expecting pass-slot internals, and the bench file should validate the semantic contract instead of legacy pass trees.

## Surprises & Discoveries

- The cutdown script’s intake and summary logic was easier to modernize than the packaging/reporting tail. The stubborn cluster is the bundle-building layer where older triage labels still leak into file naming and expectation shape.
- The current external-AI contract already has enough information in `recipe_manifest.json` and `recipe_correction_audit` to replace pass-slot reasoning; the remaining work is mostly adaptation and cleanup, not data-model invention.
- The first full-file failure after the initial cutover sat at `test_main_upload_3_files_only_consolidates_and_preserves_artifacts`, which indicates the main remaining mismatch is output bundle structure rather than early parsing.
- The paired bench file moved during the sweep: some assertions that originally checked pass-slot compatibility fields now check semantic stage status directly. The safest approach was to keep stage status as the primary output and only leave pass aliases where runtime/cost aggregation or historical fixtures still need them.

## Decision Log

- Decision: treat semantic-stage data and correction-audit artifacts as the source of truth.
  Rationale: that matches the live recipe pipeline and avoids maintaining a fake benchmark-only topology.

- Decision: allow temporary compatibility shims only while the paired tests are being rewritten.
  Rationale: this keeps the script incrementally runnable while preventing the old contract from becoming permanent again.

- Decision: validate with the focused bench file before broader sweeps.
  Rationale: the remaining work is isolated to this script/test pair, so broad test runs add noise without improving confidence much.

## Outcomes & Retrospective

Implemented on 2026-03-16. The external-AI cutdown script now treats semantic stage rows, semantic `recipe_manifest.json` stage states, and `recipe_correction_audit` diagnostics as the primary contract through starter-pack generation, upload-bundle analysis, and runtime/book-level aggregation. The paired bench file now passes as written against that contract.

The remaining pass-slot vocabulary in this path is compatibility output for archived prompt fixtures and a few reviewer-facing aggregate aliases that are still useful for continuity. The script no longer needs old benchmark topology internals to produce the external-AI packet.

Read from the current main entrypoint down through starter-pack generation, upload bundle assembly, and any net-error attribution/reporting helpers. For each helper, record:

- which inputs still mention pass slots,
- whether the value can be derived from manifest stage status or correction audit,

Write down any non-obvious mapping discovered during that read in a short understanding note if the logic takes real time to untangle.

### 3. Bench Fixture Rewrite

Change `tests/bench/test_benchmark_cutdown_for_external_ai.py` so fixtures create the same artifact family the live pipeline creates now:

- semantic stage summaries instead of pass-slot result trees,
- manifest stage states,
- correction audit files where diagnostics are required,
- any upload-bundle expectations keyed to current stage names and artifact paths.

While rewriting, avoid papering over mismatches with additional compatibility glue in the test. The test should describe the target contract, not the migration path.

### 4. Compatibility Shim Removal

Once the bench file passes against the semantic contract, delete any helper branches or field aliases added only to preserve pass-slot expectations during the transition.
