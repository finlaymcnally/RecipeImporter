# Recipe Correction Bad-Candidate Recovery

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

Today the recipe correction stage gives Codex grounded source evidence, but the model-facing style is only half-right. The good part is that each candidate already includes the real source rows in `ev` and a flattened quick-read copy in `txt`. The weak part is that the prompt still talks as if every owned span is already a real recipe, and the main hint object `h` is a bulky schema-shaped dump rather than a compact clue about candidate quality. In deterministic-only benchmark previews that combination asks the model to do the wrong kind of recovery work. In real production, an earlier LLM labeling pass should repair much of that junk before recipe correction runs, so this plan must not overfit the recipe stage to benchmark-only disaster cases. After this change, recipe correction should still stay optimized for repairing mostly plausible recipe candidates, but it should also have an explicit safe path for the remaining spans that are not actually recipes or are too fragmentary to repair confidently, and it should receive compact triage hints that help it recover from noisy deterministic junk when those cases do slip through. The user-visible proof is that a zero-token preview on bad deterministic books such as `thefoodlabcutdown` and `saltfatacidheatcutdown` shows a clearer prompt, a more model-friendly input contract, and focused tests prove that the prompt and output model can represent “bad candidate” outcomes without forcing Codex to hallucinate a recipe.

## Surprises & Discoveries

- Observation: the recipe payload itself is already strong enough for correction.
  Evidence: each recipe shard row already carries grounded source rows in `ev`, flattened source text in `txt`, and the deterministic intermediate object in `h`, so Codex sees both the evidence and the bad candidate side by side.

- Observation: the current prompt failure is mostly a contract problem, not a missing-context problem.
  Evidence: the prompt says “authoritative recipe spans” and the output model requires one recipe-shaped object per owned `rid`, even when preview shards contain obvious non-recipes such as Food Lab conversion tables and knife-buying guidance or Salt Fat section headers.

- Observation: deterministic-only previews are harsher than the intended live operator posture.
  Evidence: the user explicitly says deterministic tagging is “AWFUL” and unrepresentatively bad, so the design should optimize for recovering from noisy candidates rather than assuming the candidate set is already trustworthy.

- Observation: the live production path should be cleaner than the benchmark previews because line-role correction runs upstream.
  Evidence: the user explicitly notes that a real production run includes an LLM labeling step before recipe correction, so many obviously broken recipe candidates should be fixed before they ever reach this stage.

- Observation: the current `h` object is not a very good model-facing hint even when it contains useful information.
  Evidence: `h` is currently a mostly raw schema-style intermediate recipe dump with keys such as `@context`, `@type`, `identifier`, many empty arrays, and cookbook-shaped leftovers. As the receiving model, I would use `ev` and `txt` first and mostly skim `h` for title / ingredient / step guesses.

- Observation: the new compact `h` direction is better, but compact step hints become actively worse if they serialize structured step objects as Python-dict-looking strings.
  Evidence: fresh March 19 preview artifacts show `h.s` values such as `"{'@type': 'HowToStep', 'text': 'Roast: Butternut Squash and Brussels Sprouts in Agrodolce'}"` for some bad deterministic candidates. That is noisier than plain step text and makes the hint harder to trust quickly.

- Observation: the current prompt spends more effort defining output JSON than defining the decision procedure the model should follow.
  Evidence: the prompt is crisp about keys and arrays, but it does not clearly say “first decide whether this candidate is a repairable recipe, then repair it only if the evidence supports that.” That missing workflow is the main style issue.

- Observation: candidate-quality help should be low-claim and factual, not another scalar confidence score.
  Evidence: in deterministic-only previews the upstream guesses are often wrong in kind, so a coarse confidence number would sound more authoritative than it deserves. Counts and suspicion flags are safer than confidence.

- Observation: a rejected recipe candidate still leaks into final outputs unless the orchestrator prunes it from `updated_conversion_result.recipes`.
  Evidence: `cookimport/staging/writer.py` falls back to deterministic draft building whenever a recipe id has no final override, so omitting overrides alone is not enough. This was captured in `docs/understandings/2026-03-18_20.30.55-recipe-rejection-must-prune-updated-result-or-writer-falls-back-to-det.md`.

- Observation: the original benchmark-root preview commands in this plan no longer work as written.
  Evidence: current `cf-debug preview-prompts` rejects those benchmark roots as ambiguous / not predictive-safe, and the old vanilla processed-run paths referenced by their manifests are no longer present in this workspace. Fresh previews had to be rebuilt from surviving deterministic processed run directories under `data/output/2026-03-19_00.06.00-*-preview-stage/...`.

## Outcomes & Retrospective

This plan is implemented. Recipe correction now has a first-class bad-candidate escape hatch: the output contract can return `fragmentary` or `not_a_recipe` with a compact machine-readable reason, the prompt teaches triage before repair, and deterministic promotion only emits final recipes for `repaired` results. Rejected candidates still remain inspectable in shard proposals, promotion reports, and `recipe_correction_audit/`, but they no longer get resurrected into final drafts by downstream fallback behavior.

The model-facing surface is also materially cleaner. Grounded source evidence remains the center of gravity, `h` is a compact attempted recipe parse instead of a schema dump, and `q` adds only low-claim counts plus suspicion flags. Focused tests now lock the contract, the promotion skip path, and the prompt wording, and fresh Food Lab / Salt Fat zero-token previews show the new prompt language plus `q` on real bad-candidate shards.

## Context and Orientation

Recipe correction is the Codex-backed stage that runs after deterministic grouping and intermediate recipe construction. In this repository, “deterministic” means code-driven heuristics and parser logic that do not use a live model. In the harsh benchmark previews used for prompt review, those deterministic results are often much worse than the intended real mixed-mode path. In real production, an upstream LLM line-role correction step should improve candidate quality before recipe correction runs. This plan therefore treats benchmark junk as an important robustness test, not as the normal design center.

The main files involved are:

- `llm_pipelines/prompts/recipe.correction.compact.v1.prompt.md`: the checked-in prompt template for the live recipe correction call.
- `cookimport/llm/codex_farm_contracts.py`: the Pydantic models and serialization helpers for recipe shard inputs and outputs.
- `cookimport/llm/codex_farm_orchestrator.py`: the live recipe shard planner, prompt renderer, worker execution, deterministic validation, and promotion logic.
- `cookimport/llm/fake_codex_farm_runner.py`: the repo-owned fake runner used by tests and preview support to simulate model outputs.
- `cookimport/llm/prompt_preview.py`: the zero-token prompt rebuild path used by `cf-debug preview-prompts`.
- `tests/llm/test_codex_farm_contracts.py`, `tests/llm/test_codex_farm_orchestrator.py`, `tests/llm/test_prompt_preview.py`: the main focused test surfaces for this change.

In the current contract, each owned recipe id must return a repaired recipe object with `cr`, optional ingredient-step mapping `m`, mapping reason `mr`, selected tags `g`, and warnings `w`. There is no first-class way to say “this owned candidate is not actually a recipe” or “this candidate is too fragmentary to repair safely.” On the input side, each candidate includes `ev`, `txt`, and `h`, but `h` is currently just a broad deterministic intermediate object dump. As the receiving model, I would find `ev` and `txt` genuinely useful, while `h` feels noisy and only partially helpful. That means the missing outcome is not the only thing to fix. The style of the prompt and the quality of the hint seam also need improvement.

## Plan of Work

First, extend the recipe correction output contract in `cookimport/llm/codex_farm_contracts.py`. Keep the existing happy-path repaired recipe shape so current good cases remain simple, but add a small explicit status seam on each owned result. That seam should let the model say one of three plain-language outcomes:

- repaired recipe
- fragmentary candidate that could not be repaired confidently
- not actually a recipe candidate

Do not force a second top-level surface or a second worker stage. Keep one result per owned `rid`, but give that result a machine-readable status and reason. The status must be strict enough that deterministic promotion can branch on it without trying to infer intent from empty arrays or warning text.

Next, redesign the model-facing input style. Keep `ev` and `txt` exactly because they are already the best evidence seam. Change the deterministic hint layer so it is easier for the model to skim and less polluted by schema boilerplate. The current `h` object should stop carrying large numbers of empty arrays, schema metadata, and fields that the model never needs for correction. Replace it with a compact deterministic candidate summary that keeps only the pieces the model can actually use quickly, such as title, ingredients, steps, and yield when present. When compacting structured step objects, normalize them down to plain step text; do not stringify nested dict-like step objects into `h.s`. If text cannot be recovered, omit the step hint rather than carrying transport noise. In the same input object, add one new compact candidate-quality hint field. The exact compact key name is up to the implementer, but it should expose low-claim deterministic facts such as:

- total evidence row count
- ingredient-like row count
- instruction-like row count
- yield-like row count
- suspicion flags such as `heading_stack`, `table_like`, `short_span`, `title_only`, `title_plus_single_line`, `mixed_menu_items`, or similar codes backed by deterministic rules

Do not add a scalar confidence score. This benchmark condition is too noisy for confidence to be trustworthy, and the production path should not be made to depend on a brittle score either.

Then update the prompt in `llm_pipelines/prompts/recipe.correction.compact.v1.prompt.md`. Rewrite the opening so it no longer claims every owned item is already an authoritative recipe. Instead, say that each owned entry is a deterministic recipe candidate grounded in `ev` and `txt`, that the compact quality hint is weak triage help only, and that the deterministic candidate summary is a weak attempted parse. The prompt should explicitly frame the stage as primarily a repair/refinement task for plausible candidates, while still allowing honest rejection for residual bad ones. The prompt should tell the model to reason in a fixed order:

1. Read `ev` and `txt`.
2. Decide whether the candidate is repairable, fragmentary, or not a recipe.
3. Only if repairable, produce `cr`, `m`, `mr`, and `g`.
4. If fragmentary or not a recipe, return the explicit status and a compact reason, with recipe fields empty or null according to the contract.

The prompt must stay strict about grounding and exact ownership, but it should stop forcing a recipe-shaped answer for obviously bad candidates. The revised prompt should also explain how tags, mapping, and warnings behave when a candidate is rejected. Add two or three tiny hard-case examples inside the prompt if they fit the token budget: a chapter-heading stack, a title-plus-one-line fragment, and a normal repairable recipe.

Then update `cookimport/llm/codex_farm_orchestrator.py` so deterministic validation and promotion understand the new result status. Promotion must continue to accept repaired recipes exactly as before. For rejected or fragmentary results, the runtime should not silently promote a half-empty recipe into final outputs. Instead, write that outcome into proposal artifacts, manifest rows, and promotion reporting in a way a human can inspect. The exact durable behavior should be:

- repaired recipe: proceed with existing deterministic promotion
- fragmentary / not-a-recipe: do not promote a final recipe draft; record the rejection status and reason in the proposal and manifest surfaces

This plan intentionally does not prescribe deleting those bad candidates upstream. It only makes the correction stage honest about them and gives the model better triage help when they arrive.

After that, update `cookimport/llm/fake_codex_farm_runner.py` and `cookimport/llm/prompt_preview.py` so the preview and fake-runner surfaces use the same improved input shape and the same new output status contract as the live path. The fake runner should be able to synthesize the new output shape for tests and preview budget estimation. The preview path should rebuild the improved prompt and show the new candidate-quality hints in the serialized shard inputs. Any prompt samples or full prompt logs produced by preview should remain structurally valid.

Finally, tighten the tests. Add contract tests for the new status field and its normalization rules. Add tests for the new compact candidate-quality hint field and the compacted deterministic recipe hint shape. Add an orchestrator test that proves a rejected or fragmentary recipe proposal does not get promoted into final recipe outputs. Add a prompt-preview test that locks the new prompt wording around deterministic candidates, the explicit bad-candidate outcome, and the new triage-first decision order. Use existing Food Lab and Salt Fat preview evidence as the motivating examples, but do not couple the tests to those exact files unless necessary.

2. `tests/llm/test_codex_farm_contracts.py` also proves the new candidate-quality hint field accepts only the compact structured shape chosen by the implementation and normalizes malformed values safely.
3. `tests/llm/test_codex_farm_orchestrator.py` proves that a proposal marked non-repairable or not-a-recipe is retained in proposal/report surfaces but not promoted into final recipe outputs, and that the new hint fields are populated on shard inputs.
4. `tests/llm/test_prompt_preview.py` proves the rebuilt recipe prompt wording no longer overclaims that every deterministic candidate is already an authoritative recipe and instead teaches the triage-first repair flow.
5. Fresh zero-token previews on Food Lab and Salt Fat still rebuild successfully and show the revised recipe prompt contract plus the improved model-facing input style in `prompts/prompt_recipe_llm_correct_and_link.txt` and the corresponding `raw/llm/.../recipe_llm_correct_and_link/in/*.json` shard files.
6. Good-case recipe repair behavior remains intact: a normal repaired recipe proposal still validates and promotes the same way it does today.

The human-readable behavioral test is simple: inspect a preview shard that obviously contains non-recipe junk. After the change, the prompt should make it clear that Codex may reject that candidate honestly instead of being forced to manufacture a repaired recipe.

## Idempotence and Recovery

This plan is safe to implement incrementally because the first step is additive: introduce a new explicit status seam while keeping existing repaired-recipe behavior intact. That means the repo can continue to understand current good-path recipe outputs while the new branch is being added.

If the new status path destabilizes promotion, revert only the promotion branch and keep the contract work isolated until the validator logic is corrected. Always write preview output into fresh timestamped directories so prior evidence remains available for comparison.

Because this work touches prompt text, fake-runner logic, and deterministic promotion, retry by rerunning the focused tests first before rerunning full preview commands.

The intended revised contract can be summarized in one line:

    deterministic recipe candidate + compact quality hints -> triage -> repaired recipe OR explicit rejection -> deterministic promotion only for repaired recipes

The most important implementation constraint is that this plan should not hide the existence of bad deterministic candidates. It should make their handling explicit and inspectable. The second-most important constraint is that the hint system must stay compact and factual. Do not turn the recipe input into another sprawling debug artifact. The third important constraint is to keep the stage centered on recipe repair, not to accidentally redesign it into a full recipe-vs-nonrecipe classifier that duplicates upstream line-role work.

## Interfaces and Dependencies

In `cookimport/llm/codex_farm_contracts.py`, define a stable per-recipe result status on the recipe correction output model. The exact field name is up to the implementer, but it must be compact on the model-facing seam, validated by Pydantic, and readable by deterministic promotion. It must allow at least:

- repaired recipe
- fragmentary candidate
- not-a-recipe candidate

In the same contract module, extend `RecipeCorrectionShardRecipeInput` with one new compact candidate-quality hint object. The exact alias is up to the implementer, but the shape should stay narrow and factual. It must support small integer counts plus a short array of machine-readable suspicion flags. It must not carry prose explanations or probabilistic scores.

Also in the same module, shrink or normalize the deterministic candidate hint surface so the live model-facing `h` object becomes a compact attempted recipe parse instead of a full schema dump. Keep only fields the model can directly use during repair.

In `cookimport/llm/codex_farm_orchestrator.py`, deterministic promotion must branch on that status before building final recipe drafts. The manifest and promotion report should expose the status and any machine-readable reason so a reviewer can tell why a candidate was not promoted.

In `llm_pipelines/prompts/recipe.correction.compact.v1.prompt.md`, the model instructions must:

- define `ev` as the source of truth
- define `txt` as a quick-read convenience copy of the same source span
- define the new compact candidate-quality hint object as weak triage metadata only
- define `h` as a weak deterministic attempted recipe parse
- require exact ownership over `rid`
- teach an explicit triage-first decision order
- explain the explicit rejection / fragmentary path
- forbid the model from “repairing” a clearly non-recipe span into a confident recipe
- keep examples short and focused on the deterministic-only hard cases

In `cookimport/llm/fake_codex_farm_runner.py` and `cookimport/llm/prompt_preview.py`, keep the preview and fake-runner surfaces aligned with the live contract so zero-token previews remain meaningful.
