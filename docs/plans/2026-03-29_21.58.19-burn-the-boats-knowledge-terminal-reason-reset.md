## Structured Restatement Of User Desire

The knowledge stage must stop collapsing every no-output workspace failure into the same vague `missing_output` bucket. Once the semantic suspicion checker is removed, the repository still needs truthful machine-readable terminal reasons that say what actually happened: whether the worker exited before install, whether the active phase remained blocked by structural validation, whether the queue finished incorrectly without materializing `out/<shard>.json`, or whether the process died for some other known runtime reason. This is burn-the-boats work. The old generic story should not remain as a compatibility path, and the new explicit terminal-reason model should become the only live knowledge runtime behavior, tests, docs, and summaries.

## Purpose / Big Picture

After this change, a user who inspects a failed knowledge run can immediately tell why the shard failed without opening opaque worker prose or reverse-engineering the workspace. If a shard never installs because the active phase is still blocked, the artifacts should say that. If the worker exits cleanly while the phase surface still shows active work, the artifacts should say that. If the process was killed by a watchdog or never wrote any output at all, the artifacts should say that too. A failed run should no longer lie by implying that all of those cases are the same kind of `missing_output`.

The visible proof is straightforward. The knowledge worker task records, failure summaries, stage summaries, and benchmark-facing reports should all agree on one specific terminal reason for each failed shard. The worker root should contain enough machine-readable state to derive that reason without relying on hidden sterile-workspace-only files or on natural-language last messages. A novice should be able to inspect the relevant files and see one coherent story: what the worker was trying to do, what state it reached, why install did not happen, and how the runtime classified that outcome.

## Context and Orientation

The knowledge stage lives under `cookimport/llm/knowledge_stage/` and runs one or more “workspace workers.” A workspace worker is one long-lived Codex session that operates inside a local folder containing repo-authored inputs such as `current_phase.json`, `CURRENT_PHASE.md`, `in/<shard>.json`, `work/<shard>.json`, `repair/<shard>.json`, and `out/<shard>.json`. During execution, the actual Codex process runs in a sterile execution copy of that folder. The runner continuously copies selected files back into the repo worker root so the repository can observe progress while the worker is still alive.

The important files and modules for this plan are:

- [cookimport/llm/knowledge_stage/recovery.py](/home/mcnal/projects/recipeimport/cookimport/llm/knowledge_stage/recovery.py), which classifies knowledge task failures, decides retry/repair behavior, and computes terminal reason codes.

- [cookimport/llm/knowledge_phase_workspace_tools.py](/home/mcnal/projects/recipeimport/cookimport/llm/knowledge_phase_workspace_tools.py), which owns the repo-written `check-phase` and `install-phase` helper behavior and therefore determines what machine-readable phase state exists during workspace execution.

Two terms need plain-English definitions here.

A “terminal reason” is the final machine-readable answer to “why did this shard end the way it did?” It is different from a generic state such as `completed` or `missing_output` because it explains the actual failure mode, for example “phase blocked before install” or “worker exited with queue still active.”

A “phase surface” is the repo-owned set of files that tells the worker what to do right now. In this repository, that means `current_phase.json`, `CURRENT_PHASE.md`, `CURRENT_PHASE_FEEDBACK.md`, and the corresponding `work/`, `repair/`, and `out/` files. This plan makes those phase surfaces authoritative for workspace failure classification.

This plan assumes the semantic suspicion checker is being removed separately. The end state described here must not depend on any semantic-audit file, semantic warning code, or semantic repair loop. The new classification model must work in the checker-free world only.

## Plan of Work

The first part of the work is to define one explicit knowledge workspace terminal-reason taxonomy and apply it everywhere. Today, several different “no final output file” situations collapse to `missing_output`. That must end. The implementation should introduce a small fixed set of explicit reason codes for the live knowledge workspace path. The exact names can change during implementation, but they must cover these distinct situations:

1. The worker exited while the active phase was still blocked by validation or install prerequisites.
2. The worker exited while the active phase was still open but there was no explicit block payload, meaning the phase was abandoned or left incomplete before install.
3. The worker claimed completion or reached a terminal queue state without writing the expected `out/<shard>.json`.
4. The worker was terminated by an existing watchdog or supervision reason that already explains the failure.
5. No better evidence exists, in which case a final fallback “generic no output” reason may remain as a last resort, but only as the least-preferred branch.

The second part is to make the phase surface machine-readable enough to support those reasons. Today, `CURRENT_PHASE_FEEDBACK.md` is good for humans, but the runner should not have to scrape prose. The live helper path in [cookimport/llm/knowledge_phase_workspace_tools.py](/home/mcnal/projects/recipeimport/cookimport/llm/knowledge_phase_workspace_tools.py) should be rewritten so every `check-phase` and `install-phase` result updates `current_phase.json` with explicit status fields such as current phase status, last successful validation state, blocking validation errors, blocking validation metadata, whether install was attempted, whether install succeeded, and which shard/phase was active when the failure happened. Because the runner already syncs `current_phase.json` back into the repo worker root, this gives the runtime a stable truth source without adding a second fragile side channel.

The third part is to replace the current `response_text is missing => missing_output` shortcut in [workspace_run.py](/home/mcnal/projects/recipeimport/cookimport/llm/knowledge_stage/workspace_run.py#L71). Instead, when `out/<shard>.json` does not exist, the runtime should inspect the repo-side worker root and classify the failure from strongest evidence to weakest evidence. It should first honor a specific supervision reason if the worker was watchdog-killed. Otherwise it should read the synced phase state and determine whether the shard was blocked before install, abandoned while active, or incorrectly finished without emitting output. That classification should flow into the proposal row, the worker failure row, the task-status terminal reason, and the stage summary.

The fourth part is to remove or rewrite every downstream consumer that flattens these explicit reasons back into generic `missing_output`. This includes the knowledge recovery helpers, stage observability, prompt-budget summaries, benchmark helper reporting, and any analytics surfaces that currently count only `missing_output_shards`. It is acceptable for some top-level summaries to still expose “count of shards that produced no installed output,” but that must be a derived rollup over the new explicit reasons, not the only surviving truth.

The fifth part is test and documentation cleanup. Tests must prove that the runtime distinguishes the specific workspace no-output cases. Documentation must explain the new truth source and the new reason-code story without mentioning legacy compatibility. A novice reading the LLM and analytics docs should be able to see what files are authoritative, which reason codes are possible, and how to interpret a failed knowledge worker run.

This plan explicitly does not preserve the old generic story. If an active test, summary, or doc assumes that all workspace no-output failures are `missing_output`, that assumption should be deleted and replaced.

## Validation and Acceptance

Acceptance is behavioral and must be demonstrated with tests plus artifact inspection.

The first acceptance criterion is explicit blocked-phase classification. A test must create a knowledge workspace where `check-phase` fails and leaves the phase active. The runtime must classify that shard with a specific blocked-or-incomplete terminal reason, not `missing_output`.

The second acceptance criterion is truthful clean-exit classification. A test must simulate a worker that exits with `state: completed` but never installs output while the phase surface still shows unfinished work. The runtime must report that exact kind of incomplete exit rather than `missing_output`.

The third acceptance criterion is watchdog precedence. A test must show that when `live_status.json` records a real watchdog reason, the final knowledge task reason preserves that watchdog explanation and does not downgrade it to a generic no-output bucket.

The fourth acceptance criterion is summary propagation. `task_status.jsonl`, `failures.json`, worker `status.json`, and the knowledge stage summary must all preserve the same terminal reason for the same shard. A stage summary may still expose an aggregate “no installed output” count, but it must derive that from the explicit reason codes and keep the specific reason counts available.

The fifth acceptance criterion is benchmark/reporting truthfulness. Prompt-budget or benchmark-facing helper summaries must remain readable after the change and must not assume that a knowledge no-output failure can only be counted as generic `missing_output`.

The sixth acceptance criterion is documentation clarity. A novice must be able to read the updated docs and understand which files tell the truth when a knowledge workspace worker fails.

## Idempotence and Recovery

The steps in this plan are safe to repeat because they are ordinary source edits plus tests. The safest implementation order is the milestone order above: define the new machine-readable phase status first, then switch runtime classification, then update downstream summaries and docs. If the work stops halfway, resume by finishing the end-to-end cutover rather than adding temporary adapters to support both the old generic story and the new explicit one. This plan is intentionally burn-the-boats: do not preserve compatibility branches that keep the old generic knowledge `missing_output` story alive.

If a test fails midway because a consumer still expects `missing_output`, update that consumer immediately to the new reason vocabulary rather than adding translation shims. The goal is one live truth model, not a bilingual transition layer.
