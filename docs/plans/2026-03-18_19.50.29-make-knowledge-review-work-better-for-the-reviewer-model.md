# Make Knowledge Review Work Better For The Reviewer Model

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

Today the non-recipe knowledge review stage gives Codex real grounded evidence, but the full review surface is still shaped too much by noisy deterministic priors and too little by the actual needs of the reviewer model. After this change, the knowledge-review step should be easier for the model to do well: the prompt should teach the real task more clearly, the input payload should carry the strongest evidence and the least misleading metadata, the chunking and bundling should better reflect how a model reads and judges material, and the whole stage should be measurably more robust on deterministic-only runs such as `thefoodlabcutdown` and `saltfatacidheatcutdown`.

The user-visible proof is not just lower token cost. It is a fresh zero-token prompt preview where a human can read the rendered knowledge prompts and say: “this is what I would want if I were the model doing the review.” That means grounded text is present, weak hints are clearly weak, obvious junk is called out as junk, structural cues are kept where useful, and the bundled task shape matches how a reviewer model actually decides `knowledge` versus `other`.

## Outcomes & Retrospective

The change shipped and hit the intended reviewer-surface goal.

1. Did the rendered knowledge-review prompts become more legible and trustworthy from the reviewer model’s perspective?

Yes. The live prompt now explicitly says the reviewer is skeptical, raw block text is authoritative, hints are weak, and text wins over hinting. It also names the repo's recurring false-positive families directly, which made the rendered Food Lab and Salt Fat previews read like instructions for a cautious reviewer instead of instructions for a model that should trust deterministic priors.

2. Did the model-facing payload retain the evidence that mattered while removing misleading metadata?

Yes. Block text, table hints, and structural `text_form` survived. The payload no longer sends `semantic_hint`, and `suggested_lane` is now omitted unless the chunk has strong `knowledge` evidence. In the fresh previews, obvious junk families commonly arrived with only `{"f":"prose_like"}` instead of a knowledge-biased lane hint.

3. Did the bundle/chunk shapes improve on real books, not just in synthetic tests?

Yes, within the local stage boundary this plan targeted. The Food Lab chapter-menu bundle beginning with `ESSENTIAL KITCHEN GEAR` lost its misleading lane hint, and Salt Fat praise blurbs, signup copy, and `Recipes and Recommendations` families no longer arrived pre-biased toward `knowledge`. Real conversion/reference tables and technique prose still remained reviewable and could still retain `l=knowledge` when the evidence was strong.

The remaining misses are mostly still upstream deterministic discovery problems: some narrative or recommendation-list material still survives into knowledge review at all, and some mixed bundles are still awkward because the outside-recipe chunk surface itself is imperfect. This plan improved how the reviewer sees those chunks; it did not attempt to solve every parser or line-role source error.

## Context and Orientation

The non-recipe knowledge-review stage is the Codex-backed step that looks at text outside recipe spans and decides whether each chunk is reusable cooking knowledge or just leftover non-recipe material. In this repo, “knowledge” means durable cooking technique, explanation, reference, substitution, or other reusable guidance. “Other” means things like narrative scene-setting, memoir, marketing copy, endorsements, navigation pages, chapter menus, recipe indexes, or decorative matter.

This stage has four main parts.

The first part is chunk creation in `cookimport/parsing/chunks.py`. Raw non-recipe blocks are turned into `KnowledgeChunk` objects. Each chunk has a text body, a title when present, block ids, provenance, and a deterministic lane guess (`knowledge`, `narrative`, or `noise`). The chunking code uses headings, block roles, and simple heuristics to decide boundaries and lanes.

The second part is bundle building in `cookimport/llm/codex_farm_knowledge_jobs.py`. This code converts `KnowledgeChunk` objects into the compact model-facing payload. Today the payload includes top-level fields like `bid` and `c`, chunk-local block text under `b`, and an optional hint object `h`. This is also where chunk pruning, skip logic, and bundle packing happen.

The third part is the prompt in `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md`, assembled by `cookimport/llm/knowledge_prompt_builder.py`. This is what teaches the reviewer model how to interpret the payload and how to emit chunk decisions and snippets.

The fourth part is validation and preview in `cookimport/llm/prompt_preview.py`, `tests/llm/test_prompt_preview.py`, and the benchmark/prompt artifacts. These give us zero-token visibility into what the reviewer model would actually receive.

The current broad problem is that this step is only partly designed for the reviewer model. It has decent evidence, but it still carries misleading priors and weak task framing. Specific current failure families visible in real previews include:

- praise blurbs and publisher marketing being hinted as `knowledge`
- table-of-contents, chapter menus, and recipe indexes being bundled as if they are instructional prose
- heading clusters such as `PREFACE` or `INTRODUCTION` getting knowledge-biased hinting without enough prompt-level skepticism
- structurally mixed bundles where some list-like or menu-like content would be better handled as obviously non-knowledge from the start

This plan assumes those deterministic errors will continue for now. The reviewer model should still be able to do a better job if the stage is built for how the model actually reads and judges text.

## Plan of Work

This work should proceed in four coordinated tracks.

The first track is prompt design. Update `llm_pipelines/prompts/recipe.knowledge.compact.v1.prompt.md` so it teaches the real decision process more directly. The prompt should stop sounding like the deterministic system has already done a good semantic pass. It should say plainly that:

- block text is the authority
- hints are weak metadata and may be wrong
- if hinting and text disagree, the text wins
- endorsements, marketing/signup copy, table-of-contents pages, section menus, recipe indexes, and decorative front/back matter are usually `other`
- tables, conversions, temperature charts, and concise technical reference material can still be valid `knowledge`

The prompt should include a few short negative and positive examples taken from real failure families in Food Lab and Salt Fat. Keep them concise and task-shaped. The goal is not to add a giant few-shot section; the goal is to teach the reviewer model the repo’s actual ambiguity boundaries.

The second track is payload and hint design. Update `cookimport/llm/codex_farm_knowledge_contracts.py` and `cookimport/llm/codex_farm_knowledge_jobs.py` so the model-facing payload carries only hints that are net useful.

Keep `text_form` because it communicates surface structure safely.

Keep `suggested_lane` only when it comes from comparatively strong deterministic evidence. Add an explicit helper in `codex_farm_knowledge_jobs.py` that decides whether the lane is strong enough to expose. For weak-default or ambiguous cases, omit the field rather than sending a misleading guess.

Remove or stop emitting `semantic_hint` from the model-facing payload. If some semantic guidance is still needed, put that guidance in prompt prose instead of in a per-chunk machine-generated field that can sound more authoritative than it deserves.

Consider adding one lightweight anti-noise cue only if it clearly helps the reviewer model more than it harms token cost. Examples might be a compact chunk-kind flag for obvious navigation/marketing families, but only if it is strongly reliable. Do not add broad new provenance just because it is available upstream.

The third track is chunk and bundle shaping. Update `cookimport/parsing/chunks.py` and, if needed, `cookimport/llm/codex_farm_knowledge_jobs.py` so the reviewer gets more coherent review units.

Focus especially on the families that are currently bad reviewer experiences:

- section menus and recipe indexes that should be isolated as obvious non-knowledge review units
- heading-only chunks that should merge differently or be routed more conservatively
- mixed list-like bundles where recipe titles and navigation entries are packed into large pseudo-instructional chunks

The aim is not to solve all chunking problems globally. The aim is to make knowledge review units easier for the model to judge correctly with the current deterministic-first pipeline.

The fourth track is evaluation and preview. Extend tests and preview inspection so success is measured in reviewer-facing terms, not only token counts. Add assertions that:

- the prompt explicitly says hints are weak and text wins
- the compact payload no longer emits over-strong semantic metadata
- representative junk bundles from Food Lab and Salt Fat are still present when they should be reviewed, but no longer arrive with misleading semantic overstatement
- token cost stays near the current compact baseline unless a deliberate prompt expansion is justified

If there is a good low-cost way to add a prompt-review smoke test for known junk families, do it. The test should check prompt and payload shape, not actual model output.

## Validation and Acceptance

This plan is complete when all of the following are true.

1. Focused knowledge tests pass in the project venv.
2. The rendered knowledge prompt explicitly states that hints are weak and raw block text is authoritative.
3. The model-facing payload no longer overstates noisy deterministic guesses with strong semantic wording.
4. The prompt explicitly addresses the main false-positive families visible in current real previews.
5. Fresh previews for `thefoodlabcutdown` and `saltfatacidheatcutdown` still succeed and remain compact enough to inspect comfortably.
6. A human reading the rendered prompts can plausibly say that the reviewer model now has a better chance of making the right call even when deterministic inputs are bad.

The acceptance bar is not “perfect knowledge classification with deterministic-only upstream data.” The acceptance bar is “the reviewer model is no longer being set up to fail by the stage’s own prompt and payload design.”

## Idempotence and Recovery

All preview outputs should go to new timestamped directories, so preview validation is safe to rerun. Focused tests are repeatable in `.venv`.

If a change harms prompt clarity, restore prompt prose first before restoring removed hint fields. If a change harms chunk quality, revert the chunk-shaping change independently of the prompt/hint changes. If prompt cost regresses too far, remove low-value examples or metadata before adding any new compact fields.

If a broader chunking improvement turns out to be too risky for the same change, keep the prompt and payload improvements and split chunk reshaping into a follow-up ExecPlan. This document should be updated to record that split explicitly in `Progress` and `Decision Log`.
