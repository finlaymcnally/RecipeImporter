# Purge Legacy And Compatibility Surfaces

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the repository will stop carrying code whose only job is to preserve removed runtime names, retired local-LLM experiments, transition-era environment gates, old `codex-farm` transport behavior, or historical artifact layouts. The user-visible result is that the codebase becomes smaller and more honest: current runtime paths, current artifact layouts, and current command semantics are the only supported ones. A contributor should be able to search the repository and find almost no active code that still talks like the pre-shard world.

The visible proof is not a new feature. The proof is that current commands still work, current tests still pass, and search-based audits stop finding compatibility-only code in live modules. The benchmark/dashboard/upload tooling should read only the current artifact families unless the plan explicitly preserves a very small archival reader for one final step.

## Surprises & Discoveries

- Observation: the core runtime has already completed the most important burn-the-boats step. Active run-setting inputs now reject the old recipe, knowledge, and line-role pipeline ids.
  Evidence: `cookimport/config/run_settings.py` only accepts `codex-recipe-shard-v1`, `codex-line-role-shard-v1`, and `codex-knowledge-shard-v1`, and `tests/llm/test_run_settings.py` asserts that the old ids now raise `ValueError`.

- Observation: most remaining legacy weight is no longer in the live recipe pipeline itself. It is concentrated in read-side analytics/review tooling, runner compatibility for older `codex-farm` builds, and a few clearly retired modules still checked into `cookimport/llm/`.
  Evidence: `cookimport/analytics/dashboard_collect.py`, `cookimport/bench/upload_bundle_v1_existing_output.py`, and `scripts/benchmark_cutdown_for_external_ai.py` still contain historical readers or alias metadata; `cookimport/llm/client.py`, `cookimport/llm/prompts.py`, `cookimport/llm/repair.py`, and `cookimport/llm/codex_exec.py` are not part of the current runtime path.

- Observation: the largest code file implicated by this purge is not the runner or the orchestrators. It is `scripts/benchmark_cutdown_for_external_ai.py`, which has accumulated historical alias handling and compatibility metadata for reviewer packets.
  Evidence: the script is roughly fourteen thousand lines long and still emits `legacy_to_starter_mapping`, `legacy_aliases`, and other compatibility-oriented bundle metadata.

- Observation: one small but misleading transition seam still exists around `COOKIMPORT_ALLOW_CODEX_FARM`. The docs say it is retired, but some current code and helper scripts still define or set it.
  Evidence: `cookimport/config/run_settings.py` defines the env name, `cookimport/cli.py` sets it in one bench path, and `scripts/with-codex-farm.sh` / `scripts/interactive-with-codex-farm.sh` still exist.

- Observation: the broad analytics test file currently contains older fixture helpers that still write history CSVs under `output/.history`, while the live path helper resolves benchmark/stage history under repo-level `.history`.
  Evidence: `history_csv_for_output(tmp_path / "output")` resolves to `tmp_path/.history/performance_history.csv`, so focused analytics nodeids touched by this purge needed fixture-path updates before they could prove the collector behavior change.

## Decision Log

- Decision: split the purge into ordered deletion waves rather than one giant remove-everything patch.
  Rationale: the repo contains both obvious dead code and currently-executed compatibility readers. The safe sequence is to delete the dead modules first, then remove transition plumbing, then cut over the remaining readers once their current-authority inputs are wired in.
  Date/Author: 2026-03-17 / Codex

- Decision: treat read-side compatibility in analytics, upload-bundle, and cutdown tooling as real product work, not optional polish.
  Rationale: those modules are the main places the repo can still “teach” the old topology even after the runtime is strict. Burn-the-boats is incomplete if reviewer-facing tools keep normalizing or labeling old shapes as first-class.
  Date/Author: 2026-03-17 / Codex

- Decision: remove the retired local-LLM stack entirely instead of trying to rename or modernize it.
  Rationale: those files are not the current CodexFarm path, and one small manual test is the only thing keeping them present. Keeping them invites confusion about supported runtimes.
  Date/Author: 2026-03-17 / Codex

  Rationale: that directory exists to keep older prompt/debug readers alive. After those readers are updated, keeping the copy only adds drift and duplicate sources of truth.

## Context and Orientation

This repository now has one current family of Codex-backed runtime surfaces. The public pipeline ids are `codex-recipe-shard-v1`, `codex-line-role-shard-v1`, and `codex-knowledge-shard-v1`. The shared runtime writes worker/shard artifacts and promotes validated outputs. That current architecture lives mainly in `cookimport/config/run_settings.py`, `cookimport/llm/phase_worker_runtime.py`, `cookimport/llm/codex_farm_orchestrator.py`, `cookimport/llm/codex_farm_knowledge_orchestrator.py`, `cookimport/parsing/canonical_line_roles.py`, `cookimport/llm/prompt_preview.py`, `cookimport/llm/prompt_artifacts.py`, and `cookimport/bench/upload_bundle_v1_*`.

“Legacy” in this plan means code that exists only to preserve removed behavior, removed names, removed artifact layouts, or retired experiments. It does not mean any code that merely happens to be old. A function or file counts as legacy here if deleting it would not remove a current supported workflow, or if it only exists to read or normalize historical outputs that the project no longer intends to preserve.

The audit identified five concrete legacy families.

The first family is the retired local-LLM stack under `cookimport/llm/`. `cookimport/llm/client.py`, `cookimport/llm/prompts.py`, `cookimport/llm/repair.py`, and `cookimport/llm/codex_exec.py` do not define the current CodexFarm runtime. `codex_exec.py` is explicitly fail-closed. `repair.py` depends on the local mock client stack, and `tests/core/test_phase1_manual.py` is the only direct test keeping that stack alive.

The second family is transition-era gate plumbing around `COOKIMPORT_ALLOW_CODEX_FARM`. Current approval and plan-vs-execute behavior now live in codex-execution-policy code, but `cookimport/config/run_settings.py`, `cookimport/cli.py`, and the two shell wrappers under `scripts/` still mention the old env gate.

The third family is runner compatibility with older `codex-farm` builds. `cookimport/llm/codex_farm_runner.py` still parses an older stderr progress format and retries without `--progress-events` or, in one path, without `--benchmark-mode`. Those branches keep older local `codex-farm` behavior alive instead of enforcing the current transport contract.

The fourth family is compatibility artifact projection. The most obvious example is the knowledge phase: `cookimport/llm/codex_farm_knowledge_orchestrator.py` writes validated shard outputs back into `knowledge/out`, and `cookimport/llm/prompt_artifacts.py` still resolves knowledge prompt inputs and outputs through `knowledge_in_dir` / `knowledge_out_dir`. That compatibility copy exists for older prompt/debug readers, not as current runtime authority.

The fifth family is historical artifact readers and alias metadata in analytics and external-review tooling. `cookimport/analytics/dashboard_collect.py` still supplements modern CSV history with older `eval_report.json` scans. `cookimport/bench/upload_bundle_v1_existing_output.py` still normalizes pipeline ids defensively when adapting existing outputs. `scripts/benchmark_cutdown_for_external_ai.py` still emits `legacy_to_starter_mapping` and `legacy_aliases`, and still carries fallback logic aimed at older prompt or artifact shapes.

This plan does not target archival docs such as logs or old plans except where those files mislead current product behavior. Historical notes may still mention removed ids. The purge target is live code, live tests, active README guidance, and reviewer-facing tooling.

## Milestones

### Milestone 1: Delete the clearly dead local-LLM stack

At the end of this milestone, the repository no longer contains the retired local mock LLM client path. The files `cookimport/llm/client.py`, `cookimport/llm/prompts.py`, `cookimport/llm/repair.py`, and `cookimport/llm/codex_exec.py` are either deleted or reduced to a single current-purpose utility if one tiny piece is proven necessary. Any tests that only exercised that stack are deleted or rewritten to validate current runtime code instead.

The work in this milestone is intentionally blunt. Search the repo for imports of those modules. Confirm that nothing in the live stage, import, benchmark, or Label Studio paths depends on them. Remove the files. Remove or rewrite `tests/core/test_phase1_manual.py` if it exists only to touch the retired stack. Update the LLM docs so the “inactive modules” section stops naming files that no longer exist.

Acceptance for this milestone is that the repo no longer contains that retired stack, no current module imports it, and the targeted LLM/core test slices still pass.

### Milestone 2: Remove transition-era env gates and wrappers

At the end of this milestone, current Codex approval is expressed only through codex-execution policy, explicit confirmation flags, and current CLI semantics. The environment variable `COOKIMPORT_ALLOW_CODEX_FARM` is no longer defined or set by current code. The helper scripts `scripts/with-codex-farm.sh` and `scripts/interactive-with-codex-farm.sh` are removed.

The important files here are `cookimport/config/run_settings.py`, `cookimport/cli.py`, `scripts/README.md`, and the shell scripts themselves. Remove constants and code paths that set the env gate. If one bench command still sets it defensively, delete that behavior and prove the command still works using the actual codex-execution-policy logic. Update CLI docs so no current instructions suggest the old gate matters.

Acceptance for this milestone is that a repo search for `COOKIMPORT_ALLOW_CODEX_FARM` returns nothing in live code or scripts, and current benchmark/stage command tests still pass.

### Milestone 3: Enforce the current `codex-farm` runner contract

At the end of this milestone, `cookimport/llm/codex_farm_runner.py` expects the current `codex-farm` build. It no longer parses the older stderr-only progress format and no longer silently retries by removing modern flags. If the installed `codex-farm` binary is too old, the runner should fail clearly and immediately instead of degrading.

The work here lives in `cookimport/llm/codex_farm_runner.py` and its tests. Remove `_parse_legacy_progress_line(...)` and any branches that exist only to recover from missing `--progress-events` or missing benchmark-mode support. Tighten test doubles and runner tests so they exercise the current JSON/progress-events behavior only. Update `docs/10-llm/10-llm_README.md` so the documented contract matches the code exactly.

Acceptance for this milestone is that the runner code has no legacy-progress parser or flag-removal retries, and the runner-related tests pass against the current expected transport shape.

### Milestone 4: Remove compatibility artifact copies and update prompt/debug readers

At the end of this milestone, prompt/debug and preview tooling read current runtime artifacts directly. The knowledge phase no longer writes `knowledge/out` compatibility copies, and prompt/debug readers no longer rely on `knowledge_out_dir` as if it were first-class authority.

The key files are `cookimport/llm/codex_farm_knowledge_orchestrator.py`, `cookimport/llm/codex_farm_knowledge_ingest.py`, `cookimport/llm/prompt_artifacts.py`, `cookimport/llm/prompt_preview.py`, and any tests that assert `knowledge_out_dir` or inspect `knowledge/out/*`. Update readers so they can reconstruct their reviewer-facing files from the validated proposal directory, runtime manifests, and current semantic stage manifests. Delete the compatibility projection once the readers are green.

Acceptance for this milestone is that a fresh run writes only the current authoritative knowledge runtime artifacts and reviewer-facing prompt/debug outputs still build correctly without `knowledge/out`.

### Milestone 5: Remove historical artifact readers from analytics and external review tooling

At the end of this milestone, analytics, upload-bundle, and cutdown tooling treat current artifact families as the only supported input shape. The dashboard collector stops supplementing current CSV history with older benchmark `eval_report.json` scans except where a clearly labeled archival tool is intentionally preserved outside the main product path. Upload-bundle and cutdown tooling stop publishing legacy alias metadata and stop normalizing old topology as if it were current.

This milestone touches `cookimport/analytics/dashboard_collect.py`, `cookimport/analytics/dashboard_render.py` if text needs cleanup, `cookimport/bench/upload_bundle_v1_existing_output.py`, `cookimport/bench/upload_bundle_v1_render.py`, and `scripts/benchmark_cutdown_for_external_ai.py`. The current source of truth should become current manifests, semantic stage rows, runtime summaries, and current CSV history. Remove `legacy_to_starter_mapping`, `legacy_aliases`, older benchmark backfill readers, and fallback copy that only exists to translate old packet layouts.

Acceptance for this milestone is that those tools still work for current runs, but their output no longer contains legacy alias metadata and their code no longer advertises historical run layouts as first-class supported inputs.

### Milestone 6: Sweep tests, docs, and fixtures

At the end of this milestone, the repository’s active docs and tests no longer describe removed ids, removed gates, or removed compatibility files as live behavior. Historical logs and old plans may remain, but current READMEs, current tests, and current helper docs should speak only in current terms.

The most important files to revisit are `docs/10-llm/10-llm_README.md`, `docs/01-architecture/01-architecture_README.md`, `docs/02-cli/02-cli_README.md`, relevant test files under `tests/llm/`, `tests/bench/`, `tests/analytics/`, `tests/core/`, and any script READMEs under `scripts/`. Remove assertions that expect legacy alias metadata, compatibility copy paths, or removed modules. Keep archival docs only where they are clearly archival.

Acceptance for this milestone is that current docs match the new post-purge codebase and repo-wide searches for removed ids or removed compatibility terms only find archival logs, old plans, or intentionally historical notes.

## Plan of Work

Start with the dead-code slice because it is the cleanest way to reduce confusion immediately. Search for imports of `cookimport.llm.client`, `cookimport.llm.prompts`, `cookimport.llm.repair`, and `cookimport.llm.codex_exec`. Remove those files and the tests that only exist for them. Once that works, remove `knowledge/out` materialization from `cookimport/llm/codex_farm_knowledge_orchestrator.py` and delete tests that assume that compatibility directory exists.

With those runtime-adjacent seams removed, attack the read-side tooling. Simplify `cookimport/analytics/dashboard_collect.py` so benchmark history is current-shape-first without older JSON supplementation. Simplify `cookimport/bench/upload_bundle_v1_existing_output.py` so it adapts current outputs without broad defensive normalization for removed ids. Simplify `scripts/benchmark_cutdown_for_external_ai.py` so it stops emitting legacy alias metadata and stops treating old packet shapes as current product behavior.

Finish with the sweep across docs, tests, and fixtures. Update the relevant READMEs and domain docs. Remove or rewrite tests that exist only to prove compatibility. Add a short note to this plan’s `Outcomes & Retrospective` and `Progress` sections after every milestone lands.

## Validation and Acceptance

This plan is accepted only when all of the following are true.

Current runtime code no longer imports or ships the retired local-LLM stack. Current commands do not depend on `COOKIMPORT_ALLOW_CODEX_FARM`. The runner no longer silently degrades for older `codex-farm` behavior. Knowledge prompt/debug readers no longer need `knowledge/out` compatibility copies. Analytics, upload-bundle, and cutdown tooling no longer emit legacy alias metadata or treat older artifact families as first-class supported shapes. Current docs and current tests describe only the current runtime.

Human-verifiable proof should include both test results and search results. The tests prove current workflows still work. The searches prove that removed concepts are actually gone from live code. If a term remains, the implementer must explain why it is archival and why it is safe to leave.

## Idempotence and Recovery

Each milestone should be implemented as a self-contained deletion wave that can be repeated safely. If deleting a compatibility seam breaks a current tool, restore only the smallest current-purpose behavior needed to read current artifacts; do not reintroduce broad alias normalization or duplicate write paths. If a milestone proves too large, split it into a smaller child ExecPlan and update this plan’s `Progress`, `Decision Log`, and `Revision note` accordingly.

Because this work is destructive by design, commit at the end of each milestone after the targeted tests pass. If a later milestone uncovers a hidden dependency on an earlier compatibility seam, re-open that milestone in this plan, document the discovery, and decide whether the dependency should be rewritten or whether the seam was actually still current.
