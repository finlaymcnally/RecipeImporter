# Rebuild the line-role stage around immutable packets, LLM-first labels, and durable worker outputs

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

The first LLM stage in this importer is supposed to do one simple job: take already-extracted cookbook text, look at stable line-like chunks, and assign better labels than the deterministic pass. After this change, the line-role stage should no longer change representation under the operator’s feet, should no longer lose partial work when one worker dies, and should no longer let deterministic post-processing silently overrule a valid LLM label just because the heuristic disagrees.

The user-visible result should be easy to verify. A line-role run should write one stable line table, one task manifest, and one compact stage summary. Each long-lived Codex worker should process many small task files in order, leave durable `out/*.json` packet results behind, and on interruption the runtime should resume only the remaining packets. When the stage finishes, the final labels should come from valid LLM packet outputs wherever they exist, with deterministic logic limited to seeding, narrow safety checks, and fallback for missing or invalid packets.

This plan fixes the first LLM labeling step only. It does not redesign EPUB/schema.org ingestion, recipe drafting, or knowledge extraction beyond what they need to consume the repaired line-role outputs.

## Surprises & Discoveries

- Observation: the current line-role runtime already looks packet-based on disk, but the authority seam is still shard-oriented.
  Evidence: `cookimport/parsing/canonical_line_roles.py` writes `assigned_tasks.json`, per-task `in/*.json`, and per-task `out/*.json`, but it still aggregates task payloads back into a parent shard proposal and decides promotion there.

- Observation: the preserved March 21 benchmark failure was not just “bad labels.” It combined representation drift, label drift, fallback, and watchdog instability.
  Evidence: the preserved benchmark-side line-role telemetry shows `labeled_line_count = 1768`, `recipe_span_count = 33`, `changed_block_count = 0`, and `scored_effect = "seed_only"` in `.../codexfarm/line-role-pipeline/telemetry_summary.json`, while the joined line table shows `494` wrong lines out of `1466` joined lines and `279` unmatched joins.

- Observation: most of the observed quality damage happened outside recipe spans, not inside the obvious recipe body.
  Evidence: the preserved joined line table contains `458` wrong outside-recipe rows versus `36` wrong inside-recipe rows, and `slice_metrics.json` shows outside-recipe `overall_line_accuracy = 0.6347687400318979` and `macro_f1_excluding_other = 0.43495406628466193`.

- Observation: the current worker contract is still too permissive for this stage, even after earlier tightening.
  Evidence: `cookimport/parsing/canonical_line_roles.py` currently launches the workspace worker with `allow_workspace_commands=True`, and the preserved prompt-budget summary shows the line-role stage hit `command_executing_shard_count = 10`, `command_execution_count_total = 126`, and `watchdog_killed_shard_count = 7`.

- Observation: the current validator and sanitizer stack still treats deterministic logic as more authoritative than the user wants.
  Evidence: `_label_atomic_lines_internal(...)` builds a deterministic baseline first, later fills any missing predictions from that baseline, then runs `_sanitize_prediction(...)` and `_apply_prediction_decision_metadata(...)` over every final prediction; `_evaluate_line_role_response_with_pathology_guard(...)` can still invalidate an LLM output by comparing it against the deterministic baseline.

- Observation: the user’s desired operating model is simpler than the current implementation: immutable task inputs, long-lived worker loop, full owned outputs, and no post-LLM deterministic semantic veto.
  Evidence: the user’s March 21 design note explicitly prefers “look at a bunch of lines and update their labelling if you think there is a better label option,” long-lived file-based worker loops, and LLM-first authority over deterministic semantic acceptance.

## Decision Log

- Decision: the line-role stage will label an immutable canonical line table and will not be allowed to alter splitting, joining, text, or line identity.
  Rationale: the user wants this stage to correct labels, not to mutate representation. Stable line identity is the only way to eliminate the current unmatched-join ambiguity and to make task resume safe.
  Date/Author: 2026-03-21 / Codex

- Decision: packet outputs will be full owned replacements, not patch-style edits against repo-written seed files.
  Rationale: for this stage, a full owned packet output is easier to validate, easier to resume, easier to diff, and easier for a Codex worker to understand than “mutate a seeded file in place.” The worker can still use the seeded deterministic label as a hint, but the runtime should validate one explicit packet result file.
  Date/Author: 2026-03-21 / Codex

- Decision: deterministic labels remain a hint and a fallback, not a semantic judge over valid LLM packet outputs.
  Rationale: this directly follows the user’s stated goal. Heuristics are allowed to seed the packet and to fill genuinely missing work, but they should not silently rewrite or reject a valid LLM label packet merely because the heuristic disagrees semantically.
  Date/Author: 2026-03-21 / Codex

- Decision: the default recovery path will be packet resume, not whole-shard repair.
  Rationale: once the runtime is packet-first, the cheapest and safest response to a worker kill is to promote already-completed packet files and requeue only the unfinished packets. Whole-shard repair should become an exceptional path, not the normal one.
  Date/Author: 2026-03-21 / Codex

- Decision: line-role will keep the long-lived workspace-worker model, but the prompt and command policy will be narrower than recipe or knowledge.
  Rationale: this stage asks the model to do a very small, repetitive classification task on local files. The worker should not need broad shell exploration to succeed. Narrower command affordances reduce drift and make watchdog kills meaningful instead of routine.
  Date/Author: 2026-03-21 / Codex

- Decision: the benchmark-helper fix that aligns `atomic_block_splitter` across vanilla and Codex variants is treated as a neighboring prerequisite, not as the whole solution.
  Rationale: paired benchmark misalignment is real and already has a narrow task file, but even with that fixed the preserved line-role stage still shows bad labels, unmatched joins, excessive fallback, and watchdog instability. This plan addresses the stage itself.

## Outcomes & Retrospective

This plan is being created before implementation. The intended outcome is a line-role stage whose behavior matches the user’s mental model:

- upstream parsing decides the line table;
- the LLM stage relabels those lines, one small packet at a time;
- valid LLM packet outputs become authoritative labels;
- deterministic code fills holes but does not semantically overrule a valid packet;
- worker interruption loses little or no completed work;
- the stage writes a compact summary that explains what happened without opening every worker folder.

The main residual risk is not conceptual but operational: the current line-role code has accumulated multiple partially overlapping control layers (packet planning, shard aggregation, sanitization, repair, watchdog retry, benchmark-specific reporting). Implementation must remove or simplify those layers rather than adding one more compatibility path on top of them.

## Context and Orientation

The first LLM labeling stage lives in `cookimport/parsing/canonical_line_roles.py`. It receives `AtomicLineCandidate` objects. In this repository, an “atomic line” is the smallest line-like unit that later recipe grouping and benchmark evaluation use. The LLM stage should decide only which label belongs on each existing line. It should not decide where the line boundaries are.

The current runtime already uses several concepts that matter for this plan:

- A “worker session” is one long-lived Codex process started in a bounded local workspace.
- A “shard” is a larger ownership bucket assigned to one worker.
- A “task packet” is a smaller subset of a shard written under `assigned_tasks.json` and `in/*.json`.
- A “proposal” is the repo-owned JSON wrapper that records whether a shard or packet validated.
- The “deterministic baseline” is the heuristic label guess created before any LLM work.

Today, the line-role stage mixes two authority models. It plans packet files, but then aggregates them back into shard proposals and still lets deterministic logic influence final promotion. The important current seams are:

- `cookimport/parsing/canonical_line_roles.py`
  This file owns deterministic seeding, task planning, worker prompt construction, packet aggregation, validation, sanitization, repair, watchdog retry, and final telemetry writing.
- `cookimport/llm/canonical_line_role_prompt.py`
  This file owns the file-backed line-role prompt surface and must stay aligned with the runtime packet contract.
- `cookimport/llm/codex_exec_runner.py`
  This file owns the long-lived workspace-worker transport, worker manifests, `assigned_tasks.json` handling, and watchdog command-policy helpers.
- `cookimport/llm/phase_worker_runtime.py`
  This file remains the shared worker runtime vocabulary reference even though line-role currently bypasses some of it.
- `cookimport/runs/stage_observability.py`
  This file is the natural home for compact stage summaries once the repaired line-role stage can explain itself.

The preserved March 21 benchmark artifacts under `data/golden/benchmark-vs-golden/2026-03-21_11.17.08/.../codexfarm/line-role-pipeline/` are the main local evidence set for this plan because the raw `data/output/.../raw/llm/...` tree has already been pruned. Those benchmark-side artifacts are still enough to prove four concrete failure families:

First, the stage changed representation in practice even though the final projection says it did not improve the seed labels. The telemetry summary reports `labeled_line_count = 1768`, but the joined evaluation table covers `1466` canonical lines and the final scored effect is still `seed_only`.

Second, the stage mislabeled many outside-recipe lines as `HOWTO_SECTION`, `INSTRUCTION_LINE`, `KNOWLEDGE`, or `OTHER` in ways that do not match the gold labels or the vanilla baseline. The preserved wrong-label and flip samples show narrative and explanatory prose drifting into procedural structure and vice versa.

Third, the stage depended too heavily on fallback and unresolved joins. The preserved joined line table includes `279` unmatched rows and many `None`-decided rows, which means the stage did not merely pick the wrong label; it lost stable correspondence between packet outputs and the canonical evaluation lines.

Fourth, the stage runtime itself was pathological. The preserved prompt-budget summary shows a command-heavy, watchdog-heavy line-role run, which is incompatible with the user’s expectation that this stage is simple enough for a properly set-up Codex worker to handle reliably.

The user’s desired operating model is explicit and will govern this implementation. Each worker should receive many small owned tasks on disk, process them in a long-lived loop, and write durable outputs one task at a time. The line-role prompt and hints should be written as if they are instructions for another Codex agent with no repo context, because they are.

One nearby constraint must be stated explicitly: some paired benchmark helpers were previously forcing different `atomic_block_splitter` values across vanilla and Codex variants. A narrow helper fix is already tracked in `docs/tasks/2026-03-21_12.20.00-align-single-offline-atomic-splitter-across-variants.md`. This plan assumes that paired helper alignment is either already landed or lands alongside the stage work. That helper fix is necessary for fair benchmarks, but it is not sufficient to repair the line-role stage itself.

## Plan of Work

The first milestone is to freeze the representation and make it inspectable. The stage must build one canonical line table before any LLM work begins and must write it to disk as an explicit artifact, for example `line-role-pipeline/runtime/line_role/canonical_line_table.jsonl`. Every later task packet should reference immutable line IDs from that table. No line-role worker, packet validator, sanitizer, or repair path may split a source line into multiple lines, join multiple lines together, or rewrite line text. If the importer or atomizer needs different line boundaries, that is upstream parsing work and must be fixed there, not inside line-role.

This milestone requires edits in `cookimport/parsing/canonical_line_roles.py` where packet planning currently flows from `AtomicLineCandidate` objects into task manifests and then back into shard aggregation. The concrete change is to make one stable line-table object the source of truth for packet planning, evaluation, and final projection. The line table should carry a stable line ID, source block identity, text, deterministic label hint, and small context windows. Packet manifests should own ordered slices of those immutable line IDs and nothing else.

The second milestone is to simplify the worker-visible contract. The current line-role worker prompt still reads like a hybrid between a coding agent and a strict JSON one-shot classifier. Replace that with a simpler packet-loop contract written for a Codex agent with zero repo context. The worker should read `worker_manifest.json`, then `assigned_tasks.json`, then for each task open exactly the named hint and input files. Each task input file should already contain the packet rows, local context, allowed labels, and a few stage-specific reminders. The worker should then write one full packet result file to `out/<task_id>.json`.

The packet output contract should be explicit and minimal. The worker should return one row per owned line ID, in the same order, with the chosen label and only the fields the repo actually needs. This plan chooses full owned packet outputs rather than “patch this seeded file in place” because full outputs are easier to validate, easier to resume, and easier to compare against the final canonical line table. The worker can still see deterministic label hints in the input packet; it simply does not edit them in place.

The third milestone is to make valid LLM packet outputs authoritative. The current finalization path in `_label_atomic_lines_internal(...)`, `_evaluate_line_role_response_with_pathology_guard(...)`, `_sanitize_prediction(...)`, and related helpers gives deterministic code too much semantic authority after a valid packet exists. That must change. After this milestone, the runtime should promote a packet result whenever it satisfies a small, mechanical acceptance contract:

- it is valid JSON;
- it covers exactly the owned line IDs once each;
- it uses only allowed labels;
- it preserves packet ordering or provides an order that the repo can deterministically restore;
- it does not invent or drop lines.

Deterministic logic should still do useful work, but only in three roles: seed the input hint, record diagnostic disagreement, and fill genuinely missing work when no valid packet result exists. Baseline-relative semantic rejection, “sanitized label adjustment,” and similar heuristic overrides should stop acting as silent promotion blockers for otherwise valid packet outputs. If a heuristic thinks a valid packet looks suspicious, that should become a diagnostic counter or a review sample, not an automatic veto, unless the case is mechanically impossible.

The fourth milestone is to remove shard-centric repair as the default recovery model. The current runtime still routes many failures through `_run_line_role_repair_attempt(...)`, `_run_line_role_watchdog_retry_attempt(...)`, and shard-level aggregation after task execution. In the repaired design, task packets become the durable answer unit and the default recovery path becomes straightforward:

- if a worker writes valid outputs for tasks 1 through 14 and then dies on task 15, the repo validates and promotes tasks 1 through 14 immediately;
- the remaining unfinished tasks go back into the pending queue;
- the next worker session starts from the remaining tasks only;
- no already-completed packet is re-asked unless the operator explicitly requests a replay.

One small task-scoped repair path may remain for malformed near-miss packet files if testing proves it is cheaper than full resume, but it must operate on one packet at a time and it must not become the dominant control path again. The whole-shard repair model should be retired from normal line-role execution.

The fifth milestone is to tighten line-role watchdog and worker affordances. This stage is simpler than recipe correction and simpler than knowledge review. The worker should not need broad shell exploration to succeed if the packet files and hint files are designed well. That should be reflected in both prompt and policy. The line-role prompt should explicitly say that the named local files are sufficient and that the worker is not expected to discover its own task model. The command policy for line-role workspace workers should be narrower than the current generic workspace-worker policy. Read-only commands against the named packet files can remain tolerated when they are genuinely helpful, but exploratory `pwd`, `ls`, `find`, broad globbing, command loops, or writes outside the owned result files should become early failures for this stage.

This milestone also requires better watchdog diagnostics. The repo must preserve the exact offending command text, the packet ID that was active when the failure happened, and how many packet outputs were already durable on disk. That information should appear in both per-task status artifacts and the eventual stage summary.

The sixth milestone is to add a compact line-role stage summary and to use it as the main debugging surface. The current benchmark-side evidence was useful, but it required stitching together several files. After this milestone, the runtime should write a single `line_role_stage_summary.json` under `line-role-pipeline/runtime/line_role/`. That summary should answer:

- how many immutable lines existed in the stage input;
- how many task packets were planned;
- how many task packets completed, failed validation, fell back, or remained unfinished;
- how many workers completed cleanly, were watchdog-killed, or exited otherwise;
- how many labels came from valid LLM packet outputs versus deterministic fallback;
- how many packet outputs were suspicious by heuristic diagnostics, without letting those diagnostics change authority;
- where the key raw artifacts live.

`cookimport/runs/stage_observability.py` should index this summary so later benchmark or stage tooling can find it without path guessing.

The seventh milestone is tests and documentation. The tests must prove the repaired behavior, not merely internal refactors. The most important new behaviors to lock down are:

- stable line IDs and packet IDs survive from input through final labels;
- a valid packet output is promoted even when it disagrees with the deterministic baseline;
- deterministic fallback fills only missing or invalid packets;
- completed packet outputs survive a worker kill and are not re-asked on resume;
- the line-role worker prompt/hints tell the worker exactly what to do without encouraging repository exploration;
- the stage summary records watchdog, fallback, and packet-completion truth.

Short current-state docs must then be updated in `docs/04-parsing/04-parsing_readme.md` and `docs/10-llm/10-llm_README.md` so the repository stops claiming a more trustworthy or more packet-authoritative line-role runtime than the code actually provides.

## Validation and Acceptance

The change is complete when a novice can run the local tests and observe the following behavior:

First, the line-role runtime writes one stable line-table artifact and packet manifests whose owned IDs refer back to that line table. No test should need to infer line identity from free-form text matching.

Second, a valid packet output that semantically disagrees with the deterministic baseline should still promote, provided it covers the owned line IDs and uses allowed labels. The deterministic baseline should appear only as an input hint or a fallback for missing or invalid packets.

Third, a simulated worker interruption after writing some packet outputs should preserve those completed packet files, and the resumed run should process only the remaining packets. No test should require re-running already validated packets to succeed.

Fourth, the line-role prompt and hint files should be readable as instructions to a Codex agent with zero repo context. Prompt-preview or prompt-content tests should verify that the worker is told to open named local files directly, to write one full packet result file per task, and not to explore the repository.

Fifth, the stage must write a compact `line_role_stage_summary.json` that reports packet counts, worker outcomes, fallback counts, and watchdog reasons. The observability tests should prove that this summary is discoverable through the stage observability index.

## Idempotence and Recovery

This implementation should be safe to repeat because the repaired design is artifact-first and packet-first. Writing the canonical line table, task manifests, packet outputs, and stage summary should all be idempotent. Re-running the stage over the same inputs should either reuse the same valid packet outputs or deterministically rebuild them.

If implementation work temporarily breaks compatibility with older shard-level artifacts, prefer keeping a small adapter that reads old artifacts into the new packet vocabulary over keeping both old and new authority models active at runtime. The runtime must end with one clear authority path, not two competing promotion models.

If a step lands halfway and tests fail, the safe rollback path is to revert only the most recent packet-promotion or watchdog-policy change and re-run the focused parsing tests. Do not delete preserved benchmark artifacts under `data/golden/benchmark-vs-golden/...`; they remain the local evidence for why this plan exists.
