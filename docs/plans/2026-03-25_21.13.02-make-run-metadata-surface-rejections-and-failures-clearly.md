# Make Run Metadata Surface Rejections And Failures Clearly

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

## Purpose / Big Picture

After this change, a human opening run metadata should be able to answer one practical question immediately: what got rejected, what fell back, what stayed unresolved, and what outright failed. Today those facts exist, but they are split across raw line-role rows, recipe proposals, knowledge task ledgers, span-decision artifacts, and nonrecipe exclusion ledgers. The new behavior is a compact, consistent rejection-and-failure summary surface in the run metadata so a bad run cannot look deceptively clean just because the details are buried in per-row files.

The proof is straightforward. A finished run should expose clear counts for line-role row fallback and Codex hard-policy rejection, recipe non-promoted outcomes such as `fragmentary` and `not_a_recipe`, knowledge unreviewed and semantically rejected work, and deterministic processing rejections such as pseudo-recipe span rejection or nonrecipe review exclusion. A reviewer should not need to grep raw artifacts to notice those outcomes.

## Outcomes & Retrospective

The run metadata now has one clear attention surface instead of forcing reviewers to infer failure and rejection from scattered raw artifacts. `knowledge_stage_summary.json`, `recipe_stage_summary.json`, and `line_role_stage_summary.json` now each carry `attention_summary`, and `stage_observability.json` mirrors that same payload on the corresponding workbook rows. Deterministic processing stages that do not have standalone compact summary files now still expose the same style of workbook-level attention view for label fallback/policy rejection, pseudo-recipe span rejection, nonrecipe review exclusions, and final recipe non-promotion.

The implementation stayed intentionally artifact-reader-first. No runtime ownership contracts changed; only the summary layer changed. That kept the diff local to `cookimport/runs/stage_observability.py` plus focused tests and doc touch points, while still making the “should be zero” counters obvious. The remaining future improvement, if wanted later, is to teach other consumer surfaces such as budget or dashboard summaries to bubble these same counters up even more aggressively. The core run metadata is now in place.

## Context and Orientation

The main run-level metadata index is `cookimport/runs/stage_observability.py`. That module discovers which stages were present on a run, writes `stage_observability.json`, and also writes the compact stage-summary artifacts for knowledge, recipe, and line-role. Today those compact summaries are:

- `raw/llm/<workbook>/knowledge/knowledge_stage_summary.json`
- `raw/llm/<workbook>/recipe_phase_runtime/recipe_stage_summary.json`
- `line-role-pipeline/runtime/line_role/line_role_stage_summary.json`

Those summary builders already know how to read their native runtime artifacts, but they do not share one consistent “rejected / failed / fallback / unresolved” surface. That gap matters because the user explicitly wants a bad outcome count to be obvious and to trend toward zero.

The relevant files are:

- `cookimport/runs/stage_observability.py`
  This is the primary implementation target. It builds and writes compact per-stage summaries and the main `stage_observability.json` report.

- `cookimport/parsing/canonical_line_roles/runtime.py`
  This is where line-role final row fallback is marked and where unresolved rows are narrowed to deterministic fallback.

- `cookimport/parsing/canonical_line_roles/policy.py`
  This is where accepted Codex rows can be explicitly rejected back to deterministic baseline with `codex_policy_rejected:*` reason tags.

- `cookimport/llm/recipe_stage_shared.py`
  This writes recipe `promotion_report.json`, including `repair_status` counts and final recipe authority eligibility.

- `cookimport/llm/knowledge_stage/runtime.py` and `cookimport/llm/knowledge_stage/reporting.py`
  These write knowledge review rollups such as invalid, missing, semantically rejected, partially promoted, and unreviewed work.

- `cookimport/staging/import_session_flows/authority.py`
  This writes `label_llm_correct/labeled_lines.jsonl` and `group_recipe_spans/<workbook>/span_decisions.json`, which are the authoritative row/span evidence surfaces for line-role and recipe-boundary rejections.

- `cookimport/staging/writer.py`
  This writes `08_nonrecipe_review_exclusions.jsonl`, which is the authoritative ledger for deterministic nonrecipe exclusions.

- `tests/staging/test_stage_observability.py`
  This is the narrowest test surface for the summary builders and the run-level report.

Terminology used in this plan:

- `attention_summary` means the compact answer to “what here deserves operator attention because it failed, fell back, stayed unresolved, or was explicitly rejected?”
- `fallback` means deterministic repo-owned substitution after the stage could not produce authoritative reviewed output for that unit.
- `rejection` means a stage accepted a candidate structurally but then refused to promote it semantically or procedurally. Examples are `codex_policy_rejected:*`, pseudo-recipe span rejection, `fragmentary`, `not_a_recipe`, semantic rejection, and obvious-junk exclusion.
- `failure` means a transport or validation problem such as invalid output, missing output, watchdog kill, or failed follow-up.

## Plan of Work

First, extend `cookimport/runs/stage_observability.py` with small helpers that normalize stage-specific “attention” counts into one compact shape. For line-role, the helper should combine existing shard/run counters with row-level `codex_policy_rejected:*` evidence from `label_llm_correct/labeled_lines.jsonl` when that artifact is available. For recipe, the helper should read `promotion_report.json` and report counts for `fragmentary`, `not_a_recipe`, non-promotable outcomes, invalid shards, missing output shards, and repair/follow-up escalation. For knowledge, the helper should expose the already-existing counts that indicate partial review or failed work: invalid shards, missing output shards, semantic rejection shards, unreviewed shards, unreviewed packets, unreviewed blocks, and failed follow-ups.

Then add lightweight deterministic-processing helpers in the same module. One helper should read `group_recipe_spans/<workbook>/span_decisions.json` and summarize accepted versus rejected pseudo-recipe spans plus rejection reasons. Another helper should read `08_nonrecipe_review_exclusions.jsonl` and summarize deterministic review exclusions by reason. These helpers should not redefine semantics; they should only summarize already-written authoritative artifacts.

After those helpers exist, add an optional `attention_summary` field to the stage-summary payloads and to `StageWorkbookObservation`. `write_stage_observability_report(...)` should attach the appropriate summary to the relevant workbook entry so `stage_observability.json` becomes the first-stop cross-stage scan surface. The standalone stage-summary files should carry the same field so the compact artifacts stay self-contained.

Keep the shape consistent. The summary should use stable field names such as a general topline block plus explicit count maps where the underlying stage has meaningful categories. The point is not to erase stage-specific nuance. The point is to make the “zero is healthy” counters obvious in one predictable place.

Finally, update the docs that explain stage summaries and run metadata so they say plainly that `stage_observability.json` and the compact stage summary artifacts now expose rejection/fallback/failure rollups. Add or update the short local README note if needed where these summary files live.

## Validation and Acceptance

Acceptance is not “new JSON key exists.” Acceptance means a reviewer can inspect a finished run and immediately see meaningful non-zero counters when a stage rejected or failed work.

The implementation is complete when all of the following are true:

- line-role compact summaries show unresolved fallback counts and explicit Codex hard-policy rejection counts;
- recipe compact summaries show non-promoted valid outcomes such as `fragmentary` and `not_a_recipe`, plus invalid or missing shard counts;
- knowledge compact summaries show invalid, missing, semantically rejected, and unreviewed work clearly;
- deterministic processing summaries show rejected pseudo-recipe spans and deterministic nonrecipe review exclusions clearly;
- `stage_observability.json` exposes those compact summaries directly on the relevant workbook entries so a reviewer does not need to open raw per-stage artifacts first;
- focused tests pass in `.venv`.

## Idempotence and Recovery

This work is additive and artifact-reader-focused. Re-running the summary builders is safe because they only read existing artifacts and rewrite compact summary JSON files deterministically. If a helper misclassifies a count, the recovery path is to adjust the reader and rerun the focused tests; there is no schema migration or destructive data rewrite.
