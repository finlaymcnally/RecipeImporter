# Remove Run-Settings Compatibility Shims

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, `RunSettings` should represent only the current live configuration contract. Old settings payloads are no longer a product surface worth migrating forever. If a stale config or winner-settings cache uses removed keys or removed values, the system should fail clearly or ignore that disposable cache instead of carrying migration code forward. Historical benchmark rows and dashboard charts must still work because they read persisted run summaries and JSON snapshots, not live config migration logic.

## Surprises & Discoveries

- Observation: the analytics/dashboard side is already mostly isolated from run-settings compatibility code.
  Evidence: analytics readers work from `run_config_json`, `run_config_hash`, and `run_config_summary` fields in CSV/report artifacts, and compute summaries directly from persisted payloads without constructing `RunSettings`.

- Observation: `RunSettings.from_dict(...)` currently does three different jobs at once.
  Evidence: it acts as the live typed model loader, a stale-key compatibility loader (`_RETIRED_RUN_SETTING_KEYS`), and an old-value migration layer for `epub_extractor=auto|legacy`.

- Observation: unknown-key tolerance is partly compensating for `cookimport.json` not being a pure run-settings file.
  Evidence: interactive import currently passes the full `_load_settings()` dict into `RunSettings.from_dict(...)`, while benchmark defaults already filter to `RunSettings.model_fields`.

## Decision Log

- Decision: preserve historical result artifacts, not historical settings interfaces.
  Rationale: old benchmark rows remain valuable for progress charts, but old config payloads mostly create dead-code drag and confuse future model-generated refactors.

- Decision: stale QualitySuite winner files should be treated as disposable cache, not long-lived migration targets.
  Rationale: if a winner cache no longer matches the current schema, silently falling back to built-in/current defaults is preferable to indefinitely preserving schema-migration code.

## Outcomes & Retrospective

This cleanup is implemented. `RunSettings` is now a current-contract-only loader instead of a migration layer for stale keys and removed extractor aliases. The winner store is simpler too: one canonical file shape, no legacy path fallback, and no attempt to resurrect stale cached winner payloads.

The historical-data boundary held cleanly. Analytics/dashboard tests continued to pass because they rely on persisted `run_config_json/hash/summary`, not on live `RunSettings` compatibility code. That validated the repo policy shift the user asked for: preserve old charts, not old config interfaces.

## Context and Orientation

`cookimport/config/run_settings.py` defines the typed `RunSettings` model and currently still accepts several obsolete inputs. The biggest compatibility seams are:

- unknown keys are logged and ignored
- retired Bucket 1 keys are logged and ignored
- old EPUB extractor aliases like `auto` and `legacy` are coerced to modern values

`cookimport/config/last_run_store.py` persists and reloads the QualitySuite winner profile used by interactive codex defaults. This file is not durable user data in the same way as benchmark history CSV; it is an operator convenience cache.

`cookimport/cli.py:_load_settings()` returns a larger `cookimport.json` payload that includes both run settings and unrelated interactive/application settings. That is why some current callers still rely on `RunSettings.from_dict(...)` ignoring unknown keys.

`cookimport/analytics/*` and the dashboard tests matter because they prove the data-preservation boundary. Those readers use persisted run-config strings and JSON snapshots from history rows and reports. They should keep working even if old live config files stop loading.

## Plan of Work

First, narrow `RunSettings.from_dict(...)` to the live schema. Remove the retired-key compatibility registry and the old extractor alias migration path. Unknown keys should stop being silently tolerated. The remaining normalization should cover current live behavior only, such as policy-locking markdown extractors or normalizing current enum aliases that are still intentionally supported.

Second, add explicit filtering at the handful of call sites that pass mixed app/config payloads into `RunSettings.from_dict(...)`. In practice this means filtering `_load_settings()` results down to model fields before building `RunSettings`, and filtering raw run-config payloads that are known to include persistence metadata before validation.

Third, simplify `last_run_store.py`. Drop legacy winner-file path fallback and stop trying to migrate stale winner payloads indefinitely. The winner cache should load only the canonical current file shape; if the payload is stale or invalid, log one concise warning and return `None` so the caller falls back to current defaults.

Fourth, update tests and docs to match the new contract. Replace “accepts retired key” tests with “rejects stale key” tests. Replace legacy winner migration tests with “invalid stale winner cache is ignored” tests. Keep at least one analytics-facing proof that historical run-config summaries/JSON remain readable without `RunSettings` migration logic.

## Validation and Acceptance

Acceptance is behavioral.

`RunSettings.from_dict(...)` should reject stale/removed setting keys and removed extractor aliases instead of silently migrating them.

Interactive import and speed-suite flows that start from `cookimport.json` should still work because they now pass only real run-setting keys into `RunSettings`.

If the QualitySuite winner cache is stale, interactive codex selection should ignore that cache and continue with current defaults rather than replaying compatibility shims.

Historical benchmark/dashboard tests should continue to pass because history readers operate on persisted `run_config_*` artifacts rather than live settings migration code.

## Idempotence and Recovery

This refactor is safe to repeat. The intended recovery path for stale settings files is not schema migration; it is deleting or regenerating those files. Historical CSV/report artifacts remain untouched.
