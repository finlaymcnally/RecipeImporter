## Purpose / Big Picture

## Context and Orientation

The knowledge stage is the optional outside-recipe review and snippet-extraction phase. Its runtime artifacts live under `data/output/<timestamp>/raw/llm/<workbook_slug>/knowledge/`. The main worker contract is now batch-based: each worker root contains `assigned_tasks.json`, `current_batch.json`, `CURRENT_BATCH.md`, `CURRENT_BATCH_FEEDBACK.md`, fallback `current_task.json`, `CURRENT_TASK.md`, `CURRENT_TASK_FEEDBACK.md`, `hints/`, `in/`, `out/`, `scratch/current_batch/`, and `tools/knowledge_worker.py`.

The relevant runtime files are:

- `cookimport/llm/codex_farm_knowledge_ingest.py`, which validates worker packet outputs after they are written to `out/<task_id>.json` and is the authoritative home of `semantic_all_false_empty_shard`.
- `cookimport/llm/codex_exec_runner.py`, which mirrors runtime control files between the repo-owned worker root and the sterile direct-exec workspace.

## Milestones

### Milestone 1: Make knowledge completion mean real queue completion

At the end of this milestone, the knowledge supervisor should no longer be able to declare a successful finish while the worker still has an active current batch or current task. The repo-owned task queue is the authority. A worker that writes some outputs and then stops with `queue_complete: false` must surface as incomplete or retryable rather than completed. The proof is local: supervision and workspace tests should show `workspace_validated_task_queue_completed` only when every assigned task has a validated output, and saved-sidecar completion should be obvious without reopening the whole workspace.

### Milestone 2: Make empty-shard failures repairable instead of chatty

At the end of this milestone, the `semantic_all_false_empty_shard` guard should still exist for real strong-cue failures, but the worker should get a compact, shard-specific explanation of what is wrong. The paved-road surfaces should point to the exact cue-bearing chunk ids, the fact that the worker returned zero knowledge decisions and zero snippets, and the smallest next step. The proof is local: helper tests and ingest tests should show a narrow failure explanation for this case, and worker prompts/feedback should stop encouraging repeated `overview` and `show-batch` loops just to infer what went wrong.

### Milestone 3: Shrink the happy-path surfaces and self-inspection budget

At the end of this milestone, the normal knowledge path should be: open `worker_manifest.json`, then the compact batch sidecars, then edit the prewritten drafts, then run `complete-batch`, `check-batch`, and `install-batch` without broad rereads. `assigned_tasks.json` must become clearly fallback-only, the helper source itself must not be a tempting startup read, and post-install output should say enough that the worker does not immediately call `overview` or dump `CURRENT_BATCH*` again. The proof is local in the prompt/helper tests and, with user approval, in a benchmark rerun that shows a materially lower command count and billed token story.

## Plan of Work

Start in `cookimport/llm/codex_farm_knowledge_orchestrator.py`. Remove the knowledge-stage notion that “all current outputs stabilized” is sufficient for success on the main workspace path. Knowledge is queue-driven and always has repo-owned assigned-task state, so a completed worker should require the queue controller to report completion or an equivalent repo-owned file-level proof that every assigned task has a validated result. If the controller is absent or stale, fail incomplete instead of completed. The final worker `live_status.json` should make the distinction explicit.

Then update `cookimport/llm/knowledge_workspace_tools.py`. Keep the batch contract, but make the batch sidecars much smaller and more directive. `current_batch.json` should remain machine-readable, but it should not repeat large queue-wide or task-local surfaces that already live elsewhere. `CURRENT_BATCH.md` should become the cheapest human-facing starting point, not a full batch dossier. Keep the batch id list, count, draft dir, remaining-work count, and the first task or first failing task summary, but move bulky per-task detail behind fallback reads. `CURRENT_BATCH_FEEDBACK.md` should show exactly what failed on the current batch, and a successful `install-batch` should print enough next-step information that the worker does not need to call `overview`, `current-batch`, `show-batch`, and `check-batch` in sequence after every install.

In the same module, make `assigned_tasks.json` clearly debug inventory only. The happy-path prompt text should stop mentioning it as the ordinary queue surface. If a worker needs task-local payloads, it should use the task rows already named in the current batch sidecar or the per-task `metadata.input_path` and `metadata.hint_path` files directly. The helper CLI should continue to support fallback debugging, but the default guidance should actively steer away from broad queue dumps and from opening `tools/knowledge_worker.py`.

Still in `cookimport/llm/knowledge_workspace_tools.py`, improve both the scaffold and the feedback path for `semantic_all_false_empty_shard`. The helper already knows the validation errors and can see the current input payload. For tasks marked with `metadata.strong_knowledge_cue`, do not leave the prewritten draft looking like an ordinary near-finished all-`other` answer. Keep the scaffold schema-valid, but use an explicit placeholder cue such as a stronger `reason_code` and matching sidecar language so the worker sees that extra review is still required before install. When `semantic_all_false_empty_shard` does occur, the batch feedback should surface the cue-bearing chunk ids and a short plain-language explanation of why a fully empty reviewed result is rejected here. The worker should not need to inspect raw helper source or reopen the whole queue to discover that the real problem is “strong knowledge cues with zero knowledge decisions and zero snippets.”

Update `cookimport/llm/codex_farm_knowledge_ingest.py` next. Keep `semantic_all_false_empty_shard` as a strict low-trust guard, but make the decision boundary and metadata output precise enough that the helper and prompt can explain it cheaply. If `_knowledge_cue_chunk_ids(...)` is currently too blunt for the post-line-role queue shape, tighten it there rather than weakening the whole validator. This plan does not remove the guard. It makes that guard cheaper to understand and less likely to trap the worker in a repeat loop.

Touch `cookimport/llm/codex_exec_runner.py` only as needed to keep the smaller batch sidecars and final queue-complete state synchronized back into the sterile execution workspace after installs and after worker exit. The runner should remain stage-generic. If a new or changed knowledge control file needs mirroring, add it there, but keep knowledge-specific policy in the knowledge modules.

Finally, update the short runtime notes in `cookimport/llm/README.md`, `docs/10-llm/10-llm_README.md`, and `docs/10-llm/nonrecipe_knowledge_review.md`. Keep the notes short and operational. They should say that the knowledge happy path is batch-first, that queue completion is authoritative for success, that `assigned_tasks.json` is fallback/debug inventory only, and that the repo now explains strong-cue empty-shard failures directly instead of expecting the worker to rediscover them.
