# Codex-First Line Role Replaces Label-First Stages

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a Codex-backed run should label the cookbook semantically before recipe boundaries are frozen instead of after recipe ownership has already been decided. In plain English, when a run is using the live Codex line-role path, `line_role` becomes the first real labeling authority and the old deterministic `label_deterministic` / `label_refine` stages stop being the stage-owning source of truth for that run.

The user-visible reason for this change is simple: recipe-local labels such as `RECIPE_TITLE`, `RECIPE_VARIANT`, `RECIPE_NOTES`, and `YIELD_LINE` should be decided while the system is still discovering recipes, not after earlier stages may already have thrown those lines out of recipe ownership. The current late placement of `line_role` lets earlier mistakes harden into ownership decisions. The target behavior is that a Codex-backed stage or benchmark run can recover recipe-local structure at the start of the semantic pipeline instead of inheriting damage from earlier label-first stages.

After implementation, a human should be able to run a Codex-backed stage or benchmark flow and observe that:

- `line_role` runs before recipe-boundary acceptance for Codex-backed runs.
- stage observability no longer shows `label_deterministic` and `label_refine` as stage-owning semantic authorities on that path.
- recipe-boundary, recipe-refine, and nonrecipe-route consume the early `line_role` authority instead of a deterministic label-first result.
- deterministic-only and vanilla benchmark paths still behave as they do today.

## Surprises & Discoveries

- Observation: `line_role` is currently a late semantic stage, not the first cookbook-labeling step.
  Evidence: `stage_observability.json` for the investigated benchmark run showed `label_deterministic` stage order 5, `label_refine` 6, `recipe_boundary` 7, `recipe_refine` 20, `line_role` 35, and `nonrecipe_route` 8 / `nonrecipe_finalize` 4 in their respective semantic families.

- Observation: the current `build_label_first_stage_result(...)` function in `cookimport/parsing/label_source_of_truth.py` does three separate jobs at once: block atomization, authoritative line labeling, and recipe-boundary grouping.
  Evidence: the function atomizes archive blocks, calls `label_atomic_lines_with_baseline(...)`, builds authoritative labeled lines and block labels, and then immediately calls `recipe_boundary_from_labels(...)`.

- Observation: real recipe blocks can be explicitly divested before late `line_role` runs.
  Evidence: the investigated `Tomato Vinaigrette` recipe audit showed `task_outcome.status = fragmentary`, `final_recipe_authority.status = not_promoted`, and all 11 recipe blocks listed in `output.payload.db`.

- Observation: the current Codex line-role runtime still builds and depends on a deterministic baseline even when Codex is the live authority.
  Evidence: `cookimport/parsing/canonical_line_roles/runtime.py` calls `_run_line_role_shard_runtime(...)`, then for every `decided_by == "codex"` row runs `_reject_codex_prediction_to_baseline_if_policy_violated(...)` against `deterministic_baseline[candidate.atomic_index]`.

- Observation: the canonical line-role package already knows how to operate before recipe grouping.
  Evidence: `cookimport/parsing/canonical_line_roles/README.md` states that pre-grouping candidates are sanitized so stale `recipe_id`, `within_recipe_span`, and span-derived tags are stripped until recipe grouping happens later.

## Context and Orientation

This repository processes cookbook-like sources into one shared stage pipeline. The key idea for this plan is “authority ordering.” Authority ordering means which stage gets to decide something first and which later stages merely inherit that decision.

The current shared stage runtime begins in `cookimport/staging/pipeline_runtime.py`. The function `run_recipe_boundary_stage(...)` calls `build_label_first_stage_result(...)` from `cookimport/parsing/label_source_of_truth.py`. That helper currently does all of the following in one call:

- rebuild archive blocks into atomized line candidates
- run `label_atomic_lines_with_baseline(...)` from `cookimport/parsing/canonical_line_roles`
- convert those predictions into authoritative labeled lines and block labels
- group recipe spans with `recipe_boundary_from_labels(...)`
- build the updated `ConversionResult` that later recipe stages consume

So although the product docs talk about big stages like `recipe-boundary` and `recipe-refine`, the actual code path still has an older label-first seam embedded inside `build_label_first_stage_result(...)`.

The canonical line-role runtime lives in `cookimport/parsing/canonical_line_roles/runtime.py` and related package files under `cookimport/parsing/canonical_line_roles/`. It already knows how to run Codex-backed line-role workers and validate their outputs. The important public entrypoint for the current label-first seam is `label_atomic_lines_with_baseline(...)`, which returns both authoritative final predictions and the deterministic reference predictions.

Recipe-boundary grouping itself is owned by `cookimport/parsing/recipe_span_grouping.py`. The function `recipe_boundary_from_labels(...)` accepts authoritative block labels and authoritative labeled lines and turns them into accepted recipe spans plus rejected pseudo-recipe decisions.

The downstream recipe stages live in `cookimport/staging/pipeline_runtime.py` and `cookimport/staging/recipe_ownership.py`. The key runtime sequence today is:

- `run_recipe_boundary_stage(...)` builds label-first authority and recipe ownership.
- `run_recipe_refine_stage(...)` uses accepted recipe spans to create or repair recipe objects.
- `build_recipe_ownership_result(...)` and `apply_recipe_divestments(...)` decide which blocks remain recipe-owned.
- `run_nonrecipe_route_stage(...)` packages outside-recipe survivors into the candidate queue for later knowledge review.
- `run_nonrecipe_finalize_stage(...)` performs the final `knowledge` versus `other` semantic call for outside-recipe candidates.

The benchmark and prediction-run surfaces reuse this same semantic seam. Relevant integration files are:

- `cookimport/staging/deterministic_prep.py`
- `cookimport/labelstudio/ingest_flows/prediction_run.py`
- `cookimport/bench/*` readers that expect line-role summaries, recipe-boundary outputs, and final stage predictions

The main trap for this change is that a naive “move line-role earlier” patch would leave the old late `line_role` pass, benchmark summaries, and prep caches assuming the old order. This plan intentionally treats the work as a full authority migration for Codex-backed runs, not as one reordered function call.

## Plan of Work

The work should start by splitting the current label-first builder into smaller responsibilities. In `cookimport/parsing/label_source_of_truth.py`, extract the current composition inside `build_label_first_stage_result(...)` into explicit helpers for:

- archive block preparation and atomization
- authoritative line/block assembly from a supplied prediction set
- recipe-boundary grouping from authoritative labels
- conversion-result rebuilding from accepted recipe spans

The end state should make it possible to call “recipe-boundary from already-authoritative early line-role predictions” without also forcing the old deterministic label-first runtime to run first.

Once those seams exist, add a new early Codex-backed label entrypoint in `cookimport/staging/pipeline_runtime.py`. The new entrypoint should run immediately after `build_extracted_book_bundle(...)` and before `run_recipe_boundary_stage(...)` freezes recipe ownership. For deterministic-only and vanilla runs, the existing path should remain intact. For Codex-backed runs, the new early path should:

- atomize the extracted archive
- run the existing Codex line-role runtime as the first semantic label authority
- build authoritative labeled lines and block labels from those early predictions
- feed those authoritative labels into `recipe_boundary_from_labels(...)`
- build the updated conversion result that recipe-boundary currently returns

The easiest safe shape is likely to introduce a new stage-owned result object, probably still adjacent to `LabelFirstStageResult`, whose job is “authoritative label stage result” rather than “deterministic label-first result.” That object should be rich enough for recipe-boundary, deterministic prep, benchmark evidence, and observability writers.

The old `run_recipe_boundary_stage(...)` then becomes a dispatcher. On deterministic-only or vanilla runs it keeps calling the existing deterministic label-first builder. On Codex-backed runs it should instead call the early authoritative line-role builder, then run recipe-boundary grouping against that result. Recipe-boundary should keep owning recipe-span acceptance; the thing being replaced is the upstream label authority, not the boundary acceptance logic itself.

After that early cutover works, remove the separate late `line_role` semantic authority from the Codex-backed path. This is the crucial migration step. The late `line_role` stage should not continue to rewrite semantic labels after recipe authority has already been decided. Instead, the later pipeline should reuse the early authoritative line-role outputs wherever the current code expects final line-role artifacts. If a later projection surface still needs a file under `line-role-pipeline/`, that file should be produced as a projection or mirror of the early authority, not as a second semantic labeling pass.

Next, update the downstream ownership and routing code to consume the new ordering honestly. `run_recipe_refine_stage(...)`, `build_recipe_ownership_result(...)`, `run_nonrecipe_route_stage(...)`, and `run_nonrecipe_finalize_stage(...)` should all see recipe-local labels that were decided before recipe ownership. This should specifically eliminate the current failure mode where a true recipe can be marked fragmentary or divested before recipe-local line roles had a chance to influence boundary acceptance.

Then update deterministic prep and cached benchmark helpers in `cookimport/staging/deterministic_prep.py` so a cached book can resume from the new early-authority seam. The current prep bundle expects processed-run artifacts under `label_deterministic`, `label_refine`, and `recipe_boundary`. The Codex-backed path must instead persist enough early line-role authority to rebuild recipe-boundary and downstream state without replaying the old label-first stages.

Finally, update observability and docs. `stage_observability.json`, benchmark summaries, prompt-budget summaries, and the plain-English/architecture/parsing docs all need to reflect that for Codex-backed runs the first semantic authority is early `line_role`, not `label_deterministic` or `label_refine`.

## Validation and Acceptance

Acceptance requires both behavior and structure.

For deterministic-only and vanilla runs:

- existing tests continue to pass
- `label_deterministic` and `label_refine` remain the authority path
- benchmark vanilla comparisons remain readable and stable

For Codex-backed runs:

- `run_recipe_boundary_stage(...)` no longer depends on the old deterministic label-first authority builder
- `recipe_boundary_from_labels(...)` consumes early authoritative line-role labels
- recipe ownership is built from those early labels
- late `line_role` is no longer a second semantic labeling authority

The most important observable acceptance checks are:

1. In a fake-Codex stage run, `stage_observability.json` shows early line-role authority ahead of recipe-boundary acceptance for the Codex-backed path.
2. The run tree does not present `label_deterministic` / `label_refine` as the stage-owning semantic label authority for that path.
3. Recipe-boundary, recipe-refine, and nonrecipe-route all still complete and write their normal downstream artifacts.
4. A regression fixture covering variants, titles, and storage notes now preserves recipe-local labels without requiring a late line-role rescue.

Add at least one new end-to-end regression around the concrete failure mode that motivated this plan: a cookbook segment with recipe-local variants and notes must keep those lines inside recipe ownership on the Codex-backed path instead of losing them before recipe-boundary or recipe authority can use them.

## Idempotence and Recovery

This migration should be done additively first and destructively second.

The safe order is:

1. Split the current label-first builder into reusable helper seams.
2. Add the early Codex-backed authority path behind an explicit conditional in `run_recipe_boundary_stage(...)`.
3. Teach downstream readers and tests about the new path.
4. Remove or demote the late `line_role` authority only after the new early path is producing the same downstream artifacts.

Do not delete the deterministic path while the Codex-backed path is still proving itself. The repo relies on deterministic-only and vanilla runs for no-token testing, fallback behavior, and benchmark comparison.

If the migration becomes unstable midway, the safe rollback is to leave the new helper seams in place but keep the dispatcher pointed at the existing deterministic label-first builder for all runs. That preserves testability without abandoning the refactor work.

## Interfaces and Dependencies

The implementation must leave the repository with clear owner functions and result types. The exact names can change during implementation, but the following responsibilities must exist explicitly.

In `cookimport/parsing/label_source_of_truth.py`, there must be separate public or package-visible helpers for:

- building atomized archive candidates from extracted blocks
- building authoritative labeled lines and authoritative block labels from a supplied prediction list
- grouping recipe spans from authoritative labels
- rebuilding the updated `ConversionResult` and `LabelFirstStageResult`-like object from accepted recipe spans

In `cookimport/staging/pipeline_runtime.py`, there must be a dispatcher or equivalent conditional that chooses between:

- deterministic label-first authority for vanilla and deterministic-only runs
- early Codex-backed line-role authority for Codex-backed runs

In `cookimport/parsing/canonical_line_roles/runtime.py`, the Codex-backed path must be callable as an early authority stage without relying on late recipe ownership to make sense of its inputs. If small deterministic policy helpers remain internally, they must stay internal. They must not reintroduce a second stage-owned label authority on the Codex-backed path.

In `cookimport/staging/deterministic_prep.py`, the prep bundle contract must gain or reuse an artifact family that can restore the early authoritative line-role result for Codex-backed runs. The prep bundle must not require `label_deterministic` and `label_refine` artifacts to exist for that path.

In benchmark and prediction-run integration code, readers must continue to find a stable final artifact family even though the semantic authority moved earlier. If a compatibility mirror is needed under an existing artifact path, prefer a projection of the new early authority over rerunning a second semantic labeling stage.
