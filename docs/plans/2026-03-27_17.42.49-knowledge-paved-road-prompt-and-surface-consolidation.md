# Knowledge Paved-Road Prompt And Surface Consolidation

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, the knowledge stage should behave like the line-role stage in the way that matters operationally: the worker should see one obvious paved road, use repo-written tools instead of inventing shell workflows, and treat the active work ledger as the primary object to edit. A human reviewing a finished knowledge worker root should be able to say, “the worker had a small contract, one helper tool, one active phase file, and one obvious next command.”

The user-visible goal is not a new semantic policy for what counts as knowledge. The goal is a calmer and cheaper worker loop. The knowledge worker should stop acting like it needs to reconstruct the whole stage from raw packet files and helper source, and should instead act like line-role: open the phase brief, edit the current work ledger, run `check-phase`, fix only the unresolved rows if a repair file appears, then run `install-phase`.

This plan intentionally does not cover three adjacent improvements that the user is handling separately: stronger seeded drafts, richer model-facing shard inputs, and richer knowledge hint content. This ExecPlan focuses only on the paved-road prompt/tooling surface and on removing helper-surface drift that makes the stage harder for the worker and for maintainers to reason about.

## Surprises & Discoveries

- Observation: the live knowledge runtime already claims to be a phase-first paved road, but the worker prompt still foregrounds the final installed output contract instead of the current work-ledger contract.
  Evidence: `cookimport/llm/knowledge_stage/recovery.py` builds a prompt that spends more space on `packet_id`, `block_decisions`, and `idea_groups` than on the concrete “open work ledger -> check-phase -> install-phase” loop.

- Observation: line-role gives the worker much stronger low-entropy operating guidance than knowledge does, even before considering deterministic seeding or richer shard metadata.
  Evidence: the sampled line-role `prompt.txt` explicitly says to open `work/<shard>.json` and `hints/<shard>.md` first, open `in/<shard>.json` only if needed, and make the smallest safe correction; the sampled knowledge `prompt.txt` does not impose the same “work ledger first, raw input only if insufficient” posture.

- Observation: the knowledge stage still has two helper/tooling surfaces in-tree: the live phase-first helper module and an older broader workspace helper module.
  Evidence: `cookimport/llm/knowledge_stage/workspace_run.py` imports `cookimport/llm/knowledge_phase_workspace_tools.py`, while `cookimport/llm/knowledge_stage/_shared.py` still imports `cookimport/llm/knowledge_workspace_tools.py`.

- Observation: the user has already spun off a separate task for richer knowledge hints, so this plan must avoid mixing “better hints” with “better paved-road contract.”
  Evidence: `docs/tasks/2026-03-27_17.32.19-add-richer-knowledge-worker-hints.md` already exists and states that the task contract should remain unchanged while the hint surface becomes more line-role-like.

## Outcomes & Retrospective

This plan is not implemented yet. The intended outcome is a knowledge stage whose worker roots are easier to use, cheaper to operate, and easier to explain to a novice contributor. Success means that a future sampled `prompt.txt`, `CURRENT_PHASE.md`, `CURRENT_PHASE_FEEDBACK.md`, and `tools/knowledge_worker.py` tell one coherent story and that worker traces show less raw-input dumping and less ad hoc helper-source exploration.

The main lesson from the pre-plan investigation is that “no guardrails” is not enough by itself. The stage also needs a very explicit, low-entropy paved road. Line-role became elegant because the worker was told what the default loop is, what not to read first, and how to make only small edits. Knowledge still leaks enough ambiguity that the worker acts like it is doing greenfield synthesis.

## Context and Orientation

The knowledge stage is the non-recipe review stage inside the optional LLM-backed pipeline. In plain language, it looks at book text that is not part of a recipe and decides which blocks contain reusable cooking guidance worth preserving. It then groups the kept blocks into small idea groups with short `topic_label` strings.

The live runtime is spread across four important files:

- `cookimport/llm/knowledge_stage/recovery.py` currently builds the worker prompt text and owns much of the direct-exec failure and retry logic.
- `cookimport/llm/knowledge_stage/workspace_run.py` is the live phase-first workspace path that writes worker roots, phase sidecars, hints, input files, repair files, and tools for the knowledge worker.
- `cookimport/llm/knowledge_phase_workspace_tools.py` is the active phase-first helper implementation for the current `tools/knowledge_worker.py` surface. It defines the pass-1 and pass-2 work ledgers, current-phase rendering, validation, repair payloads, and helper CLI commands such as `check-phase` and `install-phase`.
- `cookimport/llm/knowledge_workspace_tools.py` is an older broader helper module that still exists in the repo and is still imported from shared knowledge-stage code. It carries current-task/current-batch scaffolding and broader draft-management helpers that do not match the simpler live phase-first story.

The line-role stage is the comparison point because it already behaves like the paved-road contract the user wants. Its active worker surface lives in `cookimport/parsing/line_role_workspace_tools.py`, and its worker prompt text is produced from `cookimport/parsing/canonical_line_roles/planning.py`. The important behavioral traits are not its label set; they are its worker ergonomics: open `CURRENT_PHASE.md`, edit the prewritten `work/<shard>.json` ledger, consult the hint first, open `in/<shard>.json` only if needed, run `check-phase`, and fix only unresolved rows if a repair file appears.

The concrete failure this plan addresses is visible in the saved benchmark worker traces under `data/output/2026-03-27_07.07.27/.../knowledge/workers/`. Those traces show the knowledge worker reading full raw input packets, repeatedly reopening generated work files, and emitting large inline Python scripts. The problem is not that the worker had too little freedom. The problem is that the default path was still too ambiguous compared with line-role.

The phrase “paved road” in this repository means the repo has already written the small set of files and helper commands the worker should use first, so the worker should not need to improvise a custom shell workflow just to do the normal job. For this plan, the paved road must mean: the prompt makes the active ledger the first-class object, the helper CLI is obviously sufficient, the sidecars say what phase is active and what to do next, and the code no longer exposes a second older helper surface that competes with the current one.

## Plan of Work

Start with the live knowledge worker prompt in `cookimport/llm/knowledge_stage/recovery.py`. Rewrite `_build_knowledge_workspace_worker_prompt(...)` so that it reads like the line-role worker contract, not like a final output schema document. The prompt must say, in plain language, that the worker should open `CURRENT_PHASE.md` first, edit only the active `work/<shard>.pass1.json` or `work/<shard>.pass2.json` ledger named there, open the hint before reopening the authoritative `in/...` payload, and use `python3 tools/knowledge_worker.py check-phase` plus `install-phase` as the normal loop. It must explicitly discourage reading helper source or raw input first unless the phase brief, feedback, and hint are insufficient. It must also explicitly say that the installed result contract is handled by the helper and that the worker should think in terms of the active work ledger, not in terms of reconstructing `packet_id`, `block_decisions`, and `idea_groups` by hand.

Then tighten the phase sidecars in `cookimport/llm/knowledge_phase_workspace_tools.py` and `cookimport/llm/knowledge_stage/workspace_run.py` so they reinforce that same story. `CURRENT_PHASE.md` and `CURRENT_PHASE_FEEDBACK.md` should tell the worker exactly which file is active, what command to run next, and when the raw `in/...` payload is needed. They should not assume the prompt alone will keep the worker on the intended path. The worker root should continue to carry `OUTPUT_CONTRACT.md`, but that file should stay clearly secondary to the work-ledger contract in normal operation.

After the prompt and sidecars are aligned, consolidate the helper surface. Read every remaining import and call site for `cookimport/llm/knowledge_workspace_tools.py` and decide whether each responsibility still belongs in the live knowledge path. The default outcome should be that the live workspace path depends on `knowledge_phase_workspace_tools.py` for phase-first behavior and that older batch/current-task helper seams are either removed from active code paths, moved behind clearly named compatibility utilities, or retired entirely if nothing live needs them. The important constraint is that after this change a novice reading the active knowledge runtime should not have to infer which of two helper modules is the real one.

Once the prompt and helper-surface consolidation are done, add focused tests. These tests must lock in the intended worker contract text and the intended helper/sidecar surface. They should prove that the prompt tells the worker to prefer the active work ledger and hint before raw input, that the prompt no longer foregrounds the final installed output schema as the primary loop, that the live knowledge workspace writer emits the intended sidecars and helper files, and that the shared knowledge-stage imports do not drift back toward the older broader helper surface.

Finally, update docs in the files that currently describe the knowledge stage as already phase-first and paved-road. `docs/10-llm/10-llm_README.md` and `cookimport/llm/README.md` should say exactly what the worker sees, what the normal loop is, and which module now owns the active helper surface. If the implementation changes any “read this first” guidance, add a short note in the relevant folder documentation so a future reviewer can understand the behavioral change from the docs alone.

## Validation and Acceptance

Acceptance requires all of the following:

1. The live knowledge worker prompt in `cookimport/llm/knowledge_stage/recovery.py` tells the worker to open `CURRENT_PHASE.md` first, edit the active `work/...` ledger first, prefer the hint before the raw `in/...` payload, and use `check-phase` / `install-phase` as the normal loop.
2. The live knowledge prompt no longer teaches the final installed output contract as the primary mental model for normal operation. It may still mention the installed output shape, but the dominant story must be the work-ledger loop.
3. `CURRENT_PHASE.md` and `CURRENT_PHASE_FEEDBACK.md` in the live knowledge workspace path tell the worker exactly what file is active and what command to run next, in the same spirit as the line-role paved road.
4. The active knowledge runtime has one clearly owned helper surface. A novice should not need to guess whether `knowledge_phase_workspace_tools.py` or `knowledge_workspace_tools.py` is the real live contract.
5. Focused tests pass in `.venv` and lock in the prompt and helper-surface behavior so the stage cannot silently drift back toward raw-packet reconstruction.
6. The relevant docs explain the new paved-road contract in current tense and no longer imply a cleaner knowledge worker loop than the code actually provides.

Behavioral acceptance for a human reviewer is simple: when they inspect a future knowledge worker root, the cheapest visible path should be “open the phase brief, edit the active work ledger, run the helper,” not “cat the full packet, read tool source, and improvise scripts.”

## Idempotence and Recovery

This plan is a contract cleanup, not a data migration. Re-running the focused tests and re-rendering worker sidecars should be safe. The implementation should prefer additive prompt and sidecar clarification before deleting older helper paths so that tests can prove the live path still works. If helper-surface consolidation requires temporary compatibility wrappers, the wrappers must be clearly marked as transitional and removed from active imports before the plan is considered complete.

Avoid validating this plan by running live Codex-backed book processing without explicit user approval. The safe recovery path is to inspect saved worker artifacts, run focused Python tests, and add or adjust contract tests until the worker surface is stable. If an import cleanup breaks a live path, restore the prior import, add a failing regression test that captures the drift, and then re-attempt the consolidation with the test in place.
