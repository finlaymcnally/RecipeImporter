# Stop Workspace-Worker Final-Message Authority Leaks

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Maintain this document in accordance with `docs/PLANS.md`.

## Purpose / Big Picture

After this change, a main workspace worker that writes a valid owned output file will succeed even if its final assistant message is absent, non-JSON, or malformed. For workspace workers, the authoritative success criterion should become the validated on-disk result file, not the formatting of the final chat message. This prevents good work from being discarded and prevents unnecessary repair churn when the worker already wrote a valid result.

The user-visible proof is that a deterministic regression test can simulate the 2026-03-21 knowledge failure case where the worker wrote something useful but ended with `final_agent_message_state=malformed`. After the change, the stage accepts the validated file, records the malformed final message as telemetry only, and does not launch repair purely because the final message did not start with `{`.

## Surprises & Discoveries

- Observation: the current worker prompt already says that written files are authoritative, but the runtime still lets final-message formatting override that contract.
  Evidence: the saved knowledge worker row records `output_payload_present=true`, `final_agent_message_state=malformed`, `proposal_status=invalid`, and `repair_status=failed`.

- Observation: this is not just a knowledge-stage product problem.
  Evidence: recipe, line-role, and knowledge all use the same workspace-worker transport pattern where a worker writes local output files under repo-owned paths, so the same authority mismatch can exist anywhere the runtime still treats the final assistant message as a required success signal.

- Observation: malformed final assistant text and invalid worker output are different failures and should trigger different handling.
  Evidence: the saved knowledge case appears to have produced real file output, but the runtime still moved into invalid-and-repair logic. That suggests the authority seam is mixed together with file validation and recovery triggers.

## Context and Orientation

This plan is about main workspace workers, not structured retry or repair calls. A main workspace worker in this repository is a long-lived Codex session inside an isolated worker root. The repo gives it an authoritative result path such as the file named by `current_result_path.txt`, and the worker is supposed to write its semantic result there. The prompt text already says the authoritative result is the file, not the final assistant message.

The current authority leak exists because parts of the runtime still treat final assistant message shape as part of success or failure classification. That means a worker can write something useful to disk and still end the shard as invalid or trigger repair because the final assistant message did not start with `{`.

The saved regression fixture for this plan is:

- `data/golden/benchmark-vs-golden/2026-03-21_11.17.08/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/prompts/prompt_nonrecipe_knowledge_review.txt`

That combination means the runtime still lets final-message formatting override a written workspace result at least somewhere in the knowledge path.

## Plan of Work

### Milestone 1: Reproduce the authority leak with deterministic tests

Start by writing fail-before tests that simulate the saved failure directly. Create a fixture where a main workspace worker writes a valid owned output file that passes stage validation, but the final assistant message is either absent or malformed. The fail-before assertion should show the current runtime still marks the proposal invalid or launches repair because final-message shape remains part of authority.

Add at least one test in `tests/llm/test_codex_farm_knowledge_orchestrator.py`, because the saved regression lives there. Then inspect whether the same acceptance seam exists in recipe or line-role and add shared or stage-specific tests as needed so the contract is explicit across all workspace-worker surfaces.

This milestone is complete when the new tests fail for the right reason on current code: a valid file can still lose to a malformed or missing final assistant message.

### Milestone 2: Make the validated owned output file authoritative for main workspace workers

Implement the actual authority fix in the stage acceptance paths. In every main workspace-worker stage, a validated owned output file must become the primary success signal. The runtime should already know where that file lives because the worker is told exactly which path to write. The stage validator should inspect the written file, confirm it belongs to the current leased task or shard, and if it passes schema and ownership checks, accept it even when the final assistant message is empty, non-JSON, or malformed.

This means the final assistant message becomes telemetry only for main workspace workers. Keep recording `final_agent_message_state`, `final_agent_message_reason`, and any output preview because those fields are useful for post-mortem debugging. But do not let those fields override a valid on-disk result.

Structured retry and repair calls can keep stricter final-message requirements if their product contract still depends on a direct strict JSON answer. This plan changes the authority seam only for the main workspace-worker path.

This milestone is complete when a regression test can simulate `final_agent_message_state=malformed` alongside a valid owned output file and the stage still accepts the work.

### Milestone 3: Trigger repair only for real file-level failure

After the file-authority seam is fixed, update recovery logic so repair and retry are triggered by actual output problems, not by final-message telemetry. In `cookimport/llm/codex_farm_knowledge_orchestrator.py` first, and then anywhere the same shared pattern exists in recipe or line-role, make sure recovery decisions are based on file-level facts:

- no owned output file appeared
- the owned file is not parseable
- the owned file violates schema
- the owned file violates ownership or semantic validation

Malformed final assistant text alone must not send the runtime down a repair path if the owned file is already valid. That exact overreaction burned the one knowledge worker in the saved run.

This milestone is complete when the saved malformed-final-message case is classified as either accepted-with-telemetry-warning or rejected-for-file-level-reasons, but never as “repair because final message did not start with `{`” when the on-disk file is already good.

### Milestone 4: Update observability and docs so future debugging cannot lie about the contract

Finish by updating the observability and the short current-state docs to match the new contract. The runtime should keep enough telemetry to explain why a worker was accepted, rejected, or repaired, but those reason codes must now reflect the new design. Distinguish clearly between:

- accepted file with malformed final message
- rejected file because the file itself is invalid
- missing file
- repair triggered by file-level failure

Update `docs/10-llm/10-llm_README.md` and, if needed, `docs/10-llm/nonrecipe_knowledge_review.md` so a novice reader can tell that worker-written files are authoritative for main workspace-worker success and that the 2026-03-21 Salt Fat failure was used as the regression fixture.

This milestone is complete when the docs describe the new contract in plain language and the zero-token tests prove it.

## Idempotence and Recovery

This plan is designed to be implemented test-first and rerun safely. The saved benchmark artifact under `data/golden/benchmark-vs-golden/2026-03-21_11.17.08/...` is read-only diagnostic input and can be consulted repeatedly without mutating it.

The recommended test commands are idempotent. If one stage-specific test still encodes the old final-message authority assumption after the shared acceptance seam is changed, update that test only after confirming the new behavior matches this plan.

If an implementation attempt accepts malformed final assistant messages without validating the owned output file, treat that as a regression and tighten the file validator instead. The point is not to trust bad chat text; the point is to trust validated files over chat text.
