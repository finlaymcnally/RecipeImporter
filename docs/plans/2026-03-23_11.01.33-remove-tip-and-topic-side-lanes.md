# Remove Tip And Topic Side Lanes From Stage And Label Studio

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

`docs/PLANS.md` is checked into this repository and this document must be maintained in accordance with it.

## Purpose / Big Picture

After this change, the cookbook pipeline will stop producing and carrying the old tip/topic side lane entirely. A stage run will still produce recipes, non-recipe block authority, knowledge outputs, chunks, tables, drafts, sections, and benchmark artifacts, but it will no longer create `tips/` output, `tip_candidates`, `topic_candidates`, or the published `tips` list. This matters because the current code has two different semantic worlds: the real label-first and Stage 7 authority path (`RECIPE_*`, `KNOWLEDGE`, `OTHER`) and an older side path that extracts “tips” and “topic candidates” that the user does not use and that no longer fit the mental model of the program.

The user-visible proof is simple. After implementation, running `cookimport stage tests/fixtures/simple_text.txt --out /tmp/remove-tip-topic-smoke --workers 1 --llm-recipe-pipeline off --llm-knowledge-pipeline off --no-write-markdown` from the repository root should succeed, should not create `/tmp/remove-tip-topic-smoke/<timestamp>/tips`, and should still create the normal recipe and staging outputs. Split merge code and Label Studio ingest should also succeed without trying to merge or write any tip/topic artifacts.

## Surprises & Discoveries

- Observation: `topic_candidates` are not just written to disk; they are the last-resort fallback for chunk generation when `non_recipe_blocks` are empty.
  Evidence: `cookimport/staging/import_session.py` and `cookimport/labelstudio/ingest.py` both call `chunks_from_topic_candidates(...)` only when non-recipe block rows are absent.

- Observation: split merge and Label Studio parallel ingest both treat tip/topic data as first-class merge payloads, not as optional decorations.
  Evidence: `cookimport/cli.py:_merge_split_jobs` and `cookimport/labelstudio/ingest.py:_merge_parallel_results` both offset block indices, concatenate `tip_candidates` and `topic_candidates`, repartition tips, and finalize report totals from those merged lists.

- Observation: the reporting and analytics layer still exposes tip/topic counts as normal top-level metrics, which means deleting the side lane is a schema change, not just parser cleanup.
  Evidence: `cookimport/core/reporting.py`, `cookimport/analytics/perf_report.py`, `cookimport/analytics/dashboard_schema.py`, and `docs/08-analytics/08-analytics_readme.md` all still count `tips`, `tip_candidates`, and `topic_candidates`.

- Observation: the current CLI help and architecture docs still present `tips/` as a standard output directory, so this removal is user-visible and must be documented explicitly.
  Evidence: `cookimport stage --help`, `docs/01-architecture/01-architecture_README.md`, `docs/02-cli/02-cli_README.md`, and `docs/05-staging/05-staging_readme.md` all describe `tips/` and `topic_candidates.md`.

## Decision Log

- Decision: remove the whole tip/topic side lane, not only the candidate lists.
  Rationale: `tip_candidates` exist only to feed `tips`, and `topic_candidates` exist only for a side output plus fallback chunk generation. Keeping published `tips` while deleting their candidate source would preserve a feature the user does not recognize and would force a second round of cleanup later.
  Date/Author: 2026-03-23 / Codex

- Decision: chunk generation will depend only on Stage 7 non-recipe authority after this change.
  Rationale: the modern semantic model of the program is recipe labels plus final non-recipe classification. `topic_candidates` are an importer-era heuristic fallback that bypasses that model. If a book has no non-recipe authority rows, it should now produce zero chunks instead of reviving the old topic side lane.
  Date/Author: 2026-03-23 / Codex

- Decision: preserve recipe-note and non-recipe labeling behavior, but remove only the old exported tip/topic products.
  Rationale: the user still relies on labels such as `RECIPE_NOTE` and `KNOWLEDGE`. The cleanup must not accidentally remove note extraction or block labeling logic that still supports the real authority pipeline.
  Date/Author: 2026-03-23 / Codex

## Outcomes & Retrospective

This document currently captures the design and implementation plan only. No code has been changed yet. The expected outcome is a slimmer pipeline whose semantic outputs match the labels and stages the user already thinks in: recipes, recipe notes, non-recipe authority, knowledge, chunks, tables, and drafts. The main risk is hidden compatibility code in reporting, split merge, and Label Studio ingest that still assumes tip/topic arrays exist.

## Context and Orientation

The core object passed from importers into the shared stage session is `ConversionResult` in `cookimport/core/models.py`. Today that object still carries `recipes`, `tips`, `tip_candidates`, `topic_candidates`, `chunks`, `non_recipe_blocks`, `raw_artifacts`, and a `report`. The first four of those are the old side lane that this plan removes. In plain English, a “tip candidate” is “text the parser guessed might be useful advice,” a “topic candidate” is “leftover prose the parser guessed might represent a standalone topic,” and `tips` is the filtered published output derived from those guesses.

The real modern pipeline lives elsewhere. Importers build an initial `ConversionResult`. Then `cookimport/staging/import_session.py:execute_stage_import_session_from_result(...)` rebuilds authoritative structure from source blocks, runs label-first grouping, runs optional recipe and knowledge LLM passes, computes final non-recipe rows, extracts tables, builds chunks, and writes outputs. `cookimport/parsing/label_source_of_truth.py` is the seam where importer-owned recipes are replaced by authoritative regrouped recipes. `cookimport/staging/nonrecipe_stage.py` and the knowledge orchestrator decide what outside-recipe text is meaningful `KNOWLEDGE` versus `OTHER`. That authority path is the one the user cares about.

The old tip/topic lane leaks into several places. Importers still generate tip and topic candidates in `cookimport/plugins/text.py`, `cookimport/plugins/excel.py`, `cookimport/plugins/webschema.py`, `cookimport/plugins/pdf.py`, and `cookimport/plugins/epub.py`, mostly through helpers in `cookimport/parsing/tips.py`. The label-first rebuild in `cookimport/parsing/label_source_of_truth.py` regenerates `tip_candidates` from rebuilt recipes while carrying `topic_candidates` forward from the importer result. The stage session in `cookimport/staging/import_session.py` still writes `tips/` artifacts through `cookimport/staging/writer.py` and still falls back to `chunks_from_topic_candidates(...)` from `cookimport/parsing/chunks.py` when no authoritative non-recipe rows exist. The split-merge path in `cookimport/cli.py` and the Label Studio ingest merge path in `cookimport/labelstudio/ingest.py` both treat tip/topic arrays as mergeable first-class data. Reporting in `cookimport/core/reporting.py` and analytics in `cookimport/analytics/perf_report.py` still count them.

This change therefore is not “delete two fields.” It is a cross-cutting removal of a deprecated product surface. The correct final shape is that importers and shared sessions pass only recipe data, non-recipe data, raw artifacts, tables, chunks, reports, and the outputs directly tied to the label-first and Stage 7 authority model.

## Plan of Work

The first milestone is to remove the data contract. Edit `cookimport/core/models.py` so `ConversionResult` no longer exposes `tips`, `tip_candidates`, or `topic_candidates`, and so `ConversionReport` no longer exposes totals that only describe the deleted side lane. Then update `cookimport/core/reporting.py` and any analytics schema modules so report finalization derives counts only from surviving concepts such as recipes, non-recipe blocks, chunks, and tables. This step will force downstream call sites to become visible at type-check and test time, which is desirable because it gives a complete inventory of code that still assumes the side lane exists.

The second milestone is to delete generation and merge logic. In `cookimport/parsing/tips.py`, remove helpers that exist solely to build tip/topic candidate objects or partition them into published tips. Audit any helper in that module that still supports recipe-note extraction or other live label-first behavior and either keep it with a clearer name or move it to a more appropriate module if its current home becomes misleading. Then update each importer plugin to stop generating side-lane arrays. Text, Excel, and Web Schema importers should return only recipes, raw artifacts if any, and reports. PDF and EPUB importers should continue preserving rejected or leftover prose as `non_recipe_blocks`, but they should stop mining that prose into standalone tips/topics. Update `cookimport/parsing/label_source_of_truth.py` so the authoritative rebuild no longer regenerates tip candidates or carries forward topic candidates.

The third milestone is to remove downstream consumers. In `cookimport/staging/import_session.py`, delete the `tips` and `topic candidates` writer phases and remove the fallback from `chunks_from_topic_candidates(...)`. Chunk generation should now be: if final non-recipe rows exist, build chunks from those rows; otherwise leave `result.chunks` empty. In `cookimport/staging/writer.py`, remove `write_tip_outputs(...)` and `write_topic_candidate_outputs(...)`, delete the `tips/<workbook_slug>/...` file tree, and remove timing/output-stat bookkeeping that only exists for those writers. In `cookimport/parsing/chunks.py`, delete `chunks_from_topic_candidates(...)` if no other live caller remains.

The fourth milestone is to simplify split and Label Studio compatibility code. In `cookimport/staging/pdf_jobs.py`, remove tip-aware recipe ID reassignment logic, including any remapping of `tip.source_recipe_id`. In `cookimport/cli.py` split merge and `cookimport/labelstudio/ingest.py` parallel merge, stop offsetting, concatenating, repartitioning, and counting tip/topic payloads. Those merges should combine recipes, non-recipe blocks, raw artifacts, and report warnings only. Because this is a solo project, do not add a compatibility shim. Update all direct constructors of `ConversionResult` in tests and support code to use the slimmer shape immediately.

The fifth milestone is to make the public behavior truthful. Update CLI help text and run summaries so `--limit` no longer claims to limit tips, and so the stage output description no longer advertises `tips/` as an output directory. Update repository docs in `docs/01-architecture`, `docs/02-cli`, `docs/03-ingestion`, `docs/05-staging`, `docs/06-label-studio` if needed, and `docs/08-analytics` so the written contract matches the code. Because this project values documentation as reviewer context, the implementation should land with the docs updated in the same change set.

## Validation and Acceptance

Acceptance is behavioral, not structural. The implementation is complete only when four things are true.

First, `cookimport stage` on `tests/fixtures/simple_text.txt` succeeds with the deterministic command above and does not create `tips/` output. The user should be able to inspect the run directory and see that the pipeline still stages recipes and writes reports without any tip/topic artifacts.

Second, split and merge flows still work. The targeted tests covering `cookimport/cli.py` split merge and `cookimport/labelstudio/ingest.py` parallel ingest must pass after the side-lane arrays are removed. Those tests prove that block index offsets, recipe ID reassignment, and report finalization no longer depend on deleted tip/topic payloads.

Third, chunk generation becomes stricter and easier to reason about. Tests that formerly expected chunk fallback from `topic_candidates` must be rewritten to expect zero chunks when there are no final non-recipe rows. This is the intended behavior change and must be documented in the updated tests and docs.

Fourth, user-facing docs and summaries must stop mentioning the deleted concepts. `cookimport stage --help`, run summaries, architecture docs, ingestion docs, staging docs, and analytics docs must all describe the slimmer output contract. There must be no stale documentation that still advertises `tips.md`, `topic_candidates.json`, or totals for tip/topic candidate counts.

## Idempotence and Recovery

This change is safe to implement incrementally as long as each milestone ends with tests passing. The edits are source-only and can be repeated. The risky part is deleting model fields that many tests and helper constructors still populate. To recover from a half-finished pass, do not reintroduce compatibility shims unless a test proves a truly live behavior still needs them. Instead, finish the pass by updating every constructor and call site to the new slimmer shape.

The smoke-test output path `/tmp/remove-tip-topic-smoke` is disposable and should be deleted and recreated on every rerun. If stale outputs cause confusion, remove the entire directory and rerun the deterministic stage command. Documentation updates are also repeatable; rerun `npm run docs:list` after doc edits to confirm new docs remain discoverable.

## Artifacts and Notes

The important current-state evidence that motivated this plan is:

    `cookimport/staging/import_session.py` writes `tips` and `topic candidates` as explicit output phases and falls back to `chunks_from_topic_candidates(...)` when no authoritative non-recipe rows exist.

    `cookimport/cli.py` and `cookimport/labelstudio/ingest.py` both merge `tip_candidates` and `topic_candidates` across split jobs and recompute `tips` from the merged candidate list.

    `cookimport stage --help` still advertises `{out}/{timestamp}/tips/{filename}/` as a normal output directory.

## Interfaces and Dependencies

At the end of this plan, `cookimport/core/models.py:ConversionResult` must expose only live outputs of the modern pipeline. The exact surviving fields can follow the current model style, but `tips`, `tip_candidates`, and `topic_candidates` must not exist. `cookimport/core/models.py:ConversionReport` and `cookimport/core/reporting.py:finalize_report_totals(...)` must no longer define or compute totals that describe deleted tip/topic concepts.

`cookimport/staging/import_session.py:execute_stage_import_session_from_result(...)` must no longer import or call `write_tip_outputs`, `write_topic_candidate_outputs`, or `chunks_from_topic_candidates`. `cookimport/staging/writer.py` must no longer define tip/topic writer functions or create `tips/<workbook_slug>` output. `cookimport/parsing/chunks.py` must no longer expose `chunks_from_topic_candidates(...)` unless another clearly documented live caller still needs it, which this plan assumes will not be the case.

`cookimport/cli.py:_merge_split_jobs`, `cookimport/labelstudio/ingest.py:_merge_parallel_results`, and `cookimport/staging/pdf_jobs.py:reassign_recipe_ids(...)` must have signatures and implementations that no longer mention tip/topic payloads. `cookimport/parsing/tips.py` must either disappear entirely or shrink to only the helpers that still support live recipe-note or label-first behavior; if the file remains, its docstring and function names must describe that narrowed responsibility honestly.

`cookimport/plugins/text.py`, `cookimport/plugins/excel.py`, `cookimport/plugins/webschema.py`, `cookimport/plugins/pdf.py`, and `cookimport/plugins/epub.py` must stop populating deleted side-lane fields. `cookimport/parsing/label_source_of_truth.py` must stop rebuilding or carrying forward deleted side-lane data when it constructs the authoritative `ConversionResult`.
