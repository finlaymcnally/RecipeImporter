# Fix Dynamic Model Defaults And Importer Provenance

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

This document must be maintained in accordance with `docs/PLANS.md`.

This plan is intentionally narrow. It covers only the audit items that the user explicitly agreed should be fixed: placeholder text-importer line ranges, version-pinned model fallback behavior, fixed Label Studio upload batching, and the unresolved speculative EPUB comment. It does not change the machine-specific Oracle executable paths in `cookimport/bench/oracle_upload.py`, because the user explicitly said that local-machine-only pathing is acceptable for this solo project.

## Purpose / Big Picture

After this change, deterministic importer provenance should stop lying. When the text importer splits one source file into multiple candidate recipe regions using regex delimiters, markdown headings, or numbered titles, the resulting source-support records should carry real `start_line` and `end_line` values instead of placeholder zeros. That makes downstream debugging, evidence matching, and any line-range-based follow-up tooling trustworthy again.

The second visible outcome is that model choice defaults stop drifting with hard-coded repo literals. The interactive Codex menus, prompt-preview tooling, pipeline asset loading, and Oracle browser-model normalization should stop silently pinning behavior to specific `gpt-5.x` strings baked into code. Instead, explicit operator overrides remain authoritative, discovery/env-driven defaults remain preferred, and fallback behavior becomes centralized and documented instead of duplicated. The third visible outcome is that Label Studio upload batching becomes a real knob instead of one hidden `200` literal, so different deployments or future request-size constraints can be handled without editing code.

The proof is a token-free test and CLI story. Text importer tests should show nonzero line ranges in `candidate_recipe_region` support rows. Oracle upload tests should show that browser model normalization no longer rewrites `gpt-5` into `gpt-5.2` in code. Interactive and prompt-preview tests should show that no `gpt-5.3-codex` literal is injected when discovery is unavailable. Label Studio upload tests should show that a caller-provided batch size changes the number and shape of upload batches. The EPUB importer should retain its current behavior, but the speculative comment should be replaced with a concise explanation of the actual invariant or deleted if no invariant exists.

## Surprises & Discoveries

- Observation: the text importer problem is not just a dead TODO. The placeholder `(0, 0)` ranges are consumed by `_chunk_source_support(...)`, which means the emitted `candidate_recipe_region` provenance can become empty or misleading in real outputs.
  Evidence: `cookimport/plugins/text.py` routes `line_range` into `start_line`, `end_line`, and referenced block ids, and `_split_by_regex(...)` / `_split_by_positions(...)` currently emit `(0, 0)` tuples.

- Observation: the repo already has a workable discovery seam for local Codex model defaults in `cookimport/labelstudio/prelabel.py`, but the surrounding code ignores that seam and still injects hard-coded model strings elsewhere.
  Evidence: `default_codex_model(...)`, `list_codex_models(...)`, and `resolve_codex_model(...)` already exist, while `cookimport/cli_ui/run_settings_flow.py`, `cookimport/cli_support/interactive_flow.py`, and `llm_pipelines/pipelines/*.json` still hard-code `gpt-5.3-codex` or `gpt-5.3-codex-spark`.

- Observation: Oracle browser-model normalization is only brittle because the current code tries to “help” by rewriting ambiguous aliases to one specific vintage. Removing that rewrite is simpler than adding more pinned rewrites.
  Evidence: `normalize_oracle_browser_model(...)` in `cookimport/bench/oracle_upload.py` currently maps `gpt-5` to `gpt-5.2` and `gpt-5-pro` to `gpt-5.2-pro`.

- Observation: Label Studio upload batching is not presently part of run settings or CLI surface. The only live batch-size owner is a local `batch_size = 200` literal inside the upload loop.
  Evidence: `cookimport/labelstudio/ingest_flows/upload.py` uses a local `batch_size = 200`, and no corresponding public option exists in `cookimport/cli_commands/labelstudio.py`.

- Observation: the EPUB comment is talking about a real control-flow concern, not a random note to self. The code is trying to avoid swallowing the next recipe’s title when `_find_recipe_end(...)` first notices the next `Ingredients` header.
  Evidence: `_find_recipe_end(...)` calls `_backtrack_for_title(blocks, i)` when it sees a canonical ingredient header and returns that earlier title index if found.

- Observation: the existing `default_codex_model(...)` helper was already enough for prompt-preview honesty once the pinned pipeline literals were removed; a brand-new shared module was not required for this narrow cleanup.
  Evidence: `cookimport/llm/prompt_preview.py` can report either the discovered/default model or `pipeline/default` by calling the existing helper after asset loading.

## Context and Orientation

The repository is a deterministic recipe-import pipeline. The source-file importers live under `cookimport/plugins/`. The text importer in `cookimport/plugins/text.py` can split one text or markdown source into several candidate recipe regions before the shared stage pipeline decides which of those regions become real recipes. Those candidate regions are published as `SourceSupport` rows with `kind="candidate_recipe_region"`. In plain English, that means “here is the importer’s best guess about where recipes might be inside the source.” Those rows are not the final recipe truth, but they are still important diagnostics and follow-up evidence.

The current bug is in the text importer’s split helpers. `_split_by_regex(...)` and `_split_by_positions(...)` return tuples shaped like `(chunk_text, (start_line, end_line))`. The caller assumes those line numbers are real. They are not real today for those two paths. The code writes `(0, 0)` instead. That means `_chunk_source_support(...)` later computes empty or misleading line provenance for those chunks. The implementation work must stay at this ownership seam.

The model-default problem spans four layers. First, the interactive UI in `cookimport/cli_ui/run_settings_flow.py` and `cookimport/cli_support/interactive_flow.py` injects version-pinned fallback choices when discovery returns nothing. Second, the pipeline asset files in `llm_pipelines/pipelines/*.json` pin concrete `codex_model` defaults. Third, prompt-preview loading in `cookimport/llm/prompt_preview.py` reads those pipeline payloads directly and therefore inherits any pinned model string found there. Fourth, Oracle browser upload in `cookimport/bench/oracle_upload.py` rewrites high-level GPT aliases to one specific point release. These are all different manifestations of the same design smell: the repository has no single owner for “what model should we default to when the operator did not choose one explicitly.”

The good news is that the repo already has part of the needed discovery machinery. `cookimport/labelstudio/prelabel.py` includes `default_codex_model(...)`, `list_codex_models(...)`, and `resolve_codex_model(...)`. Those helpers are local-Codex-oriented rather than CodexFarm-pipeline-oriented, but they prove the repository already accepts discovery/env-driven defaults as the right shape. The implementation in this plan should reuse or extract that idea into a clearer shared owner instead of spreading new hard-coded fallbacks.

The Label Studio upload-batch problem lives in `cookimport/labelstudio/ingest_flows/upload.py`, with the public CLI edge in `cookimport/cli_commands/labelstudio.py`. The upload loop currently chooses `200` tasks per batch with no CLI option or saved-setting path. A “batch” here means one HTTP import request to Label Studio containing many tasks. Making the batch size configurable does not mean changing task generation or segmentation. It only means making the upload transport knob explicit and validated.

The EPUB comment cleanup is in `cookimport/plugins/epub.py`, inside `_find_recipe_end(...)`. That function scans forward from a detected anchor block to decide where one recipe candidate should end. When it encounters a canonical `Ingredients` header, it assumes that might be the next recipe’s header and uses `_backtrack_for_title(...)` to walk backward and find the next recipe’s title. The confusing comment is asking whether that backtracking logic will safely preserve the next title instead of letting the current candidate consume it. The plan should replace that speculation with one short comment that states the actual invariant, or delete the comment if the code and tests already make the intent obvious.

## Plan of Work

Start with the text importer because it is the clearest correctness bug and the smallest behavioral change. In `cookimport/plugins/text.py`, add one shared helper that maps a character span in the original text back to a one-based inclusive line range. Use that helper in both `_split_by_regex(...)` and `_split_by_positions(...)` so those functions return real `(start_line, end_line)` tuples instead of `(0, 0)`. Keep `_split_by_line_indices(...)` as the line-native path it already is. The implementation should continue trimming chunk text for payload quality, but it must compute the line range from the untrimmed source span so that leading or trailing whitespace removal does not destroy provenance accuracy.

Once the helper exists, update or add ingestion tests in `tests/ingestion/test_text_importer.py`. Cover at least three cases: explicit delimiter splitting, markdown heading splitting, and numbered-title splitting. Each case must assert both the emitted `candidate_recipe_region` payload lines and the referenced block ids. This is the crucial observable proof that the fix is real and not just a helper refactor. If any downstream tests currently assume `(0, 0)`, update them to assert truthful ranges instead.

Then centralize model-default and alias resolution. Create one small shared owner module under either `cookimport/config/` or `cookimport/llm/` that defines the repository policy for model defaulting when no explicit operator override exists. The required behavior is: explicit CLI or run-setting override wins; environment or local discovered defaults come next; versionless alias text may remain versionless if discovery cannot prove a better concrete model; and “no explicit model override” must remain a valid final answer. Do not reintroduce a baked-in `gpt-5.3-*` fallback as the last step. The implementation may extract or wrap helpers from `cookimport/labelstudio/prelabel.py`, but the final owner should be named clearly enough that the UI, prompt preview, pipeline loading, and Oracle browser upload can all call it.

With that owner in place, thread it through the interactive edges. In `cookimport/cli_ui/run_settings_flow.py`, remove the hard-coded `gpt-5.3-codex` fallback choice and instead offer only discovered models, the current override when present, the pipeline/default option, and any explicit custom-model path already supported. In `cookimport/cli_support/interactive_flow.py`, do the same for AI prelabeling. The menu should stay usable even when discovery returns nothing: “pipeline/default” plus “custom model id” is acceptable; silently inventing a pinned default is not.

Next, stop teaching concrete model pins through the pipeline asset files. The pipeline definitions under `llm_pipelines/pipelines/` should no longer encode version-pinned `codex_model` strings as if they were durable repo truth. There are two acceptable end states. One is that the files omit `codex_model` entirely and the loader overlays a resolved default later. The other is that the files keep a symbolic empty/default value and the loader resolves it. Either way, `cookimport/llm/prompt_preview.py` must stop assuming that every pipeline payload carries a concrete pinned model. Update `_load_pipeline_assets(...)` or a neighboring helper so prompt preview reports the resolved runtime default when available and otherwise preserves the honest “pipeline/default” state.

Then fix Oracle browser-model normalization in `cookimport/bench/oracle_upload.py`. The function `normalize_oracle_browser_model(...)` should stop rewriting `gpt-5` to `gpt-5.2` and `gpt-5-pro` to `gpt-5.2-pro`. The new behavior should be either “leave the alias unchanged” or “resolve it through the new shared default/alias owner if there is local discovery evidence,” but it must not hard-code a point release in repo code. Update the related tests in `tests/bench/test_benchmark_oracle_upload.py` and `tests/bench/test_oracle_followup.py` so they prove this exact contract.

After the model-default work, make Label Studio upload batching configurable. Add an `upload_batch_size` parameter to `cookimport/labelstudio/ingest_flows/upload.py` with positive-integer validation and use it in place of the fixed `200`. Thread that parameter through the public CLI in `cookimport/cli_commands/labelstudio.py` as a new option on `labelstudio-import`. If the same upload helper is reused elsewhere and a second public CLI edge needs the knob, thread it there as well. Keep the default value at `200` for behavior continuity, but make it explicit and user-controlled. Add or update tests under `tests/labelstudio/` so a small custom batch size produces the expected number of client upload calls.

Finish with the EPUB comment cleanup. Revisit `_find_recipe_end(...)` in `cookimport/plugins/epub.py` and replace the current speculative prose with one short comment that explains the actual rule: on a canonical next `Ingredients` header, the code tries to end the current candidate before the next title by calling `_backtrack_for_title(...)`. If no existing test covers that control flow, add one narrow importer test in `tests/ingestion/test_epub_importer.py` that locks the behavior down. Do not change behavior unless the test proves the current code is wrong.

Finally, update the owning docs. This plan changes importer provenance behavior, Label Studio import surface, and LLM/model-default ownership. The minimum docs to touch are `docs/03-ingestion/03-ingestion_readme.md`, `docs/06-label-studio/06-label-studio_README.md`, `docs/10-llm/10-llm_README.md`, and `docs/02-cli/02-cli_README.md`. Keep the updates short and behavioral. Do not use the docs to re-teach the whole subsystem.

## Validation and Acceptance

The plan is complete only when all of the following are true.

The text importer emits truthful provenance for all active split strategies. For delimiter-, markdown-header-, and numbered-title-based splits, emitted `candidate_recipe_region` support rows must carry positive `start_line` and `end_line` values and correct referenced block ids. No live output from those paths may still contain placeholder `(0, 0)` line ranges.

The repository no longer silently injects version-pinned `gpt-5.x` fallback strings as hidden defaults in the agreed surfaces. If model discovery is unavailable, the UI may still offer pipeline/default or custom-entry paths, but it must not invent `gpt-5.3-codex`. Prompt preview must load pipeline assets without assuming a concrete pinned model. Oracle browser upload must no longer rewrite `gpt-5` to `gpt-5.2` or `gpt-5-pro` to `gpt-5.2-pro`.

Label Studio upload batching must be user-controlled from the public import surface. A test must prove that a custom batch size changes the number of upload batches. The default value may remain `200`, but it must no longer be a hidden local literal.

The EPUB segmentation comment must stop speculating about what “will” happen. The surrounding behavior must either be covered by a narrow test and explained plainly, or the behavior must be corrected if the test shows the current assumption is false.

The owner docs must explain the new behavior briefly and accurately without re-teaching removed or superseded fallback behavior.

## Idempotence and Recovery

This work should be implemented additively and is safe to repeat. Re-running the importer or benchmark tests should not mutate repo state outside normal temporary test directories. If the model-default refactor is done in small steps, keep each step compiling and keep one narrow test green before moving to the next subsystem.

If the pipeline-asset change temporarily breaks prompt preview because a loader still expects `codex_model` to be present, restore behavior by adding the central overlay helper first and only then removing pinned `codex_model` values from the JSON files. That order keeps the work retryable.

If the new Label Studio batch-size option introduces CLI churn, keep the default at `200` and thread the new parameter through the flow before changing any interactive defaults. That makes rollback simple: callers that do not pass the new option should continue to behave exactly as before.

If the EPUB comment cleanup uncovers a real segmentation bug instead of mere wording debt, stop and split the work into “comment clarified” and “behavioral fix remaining” entries in `Progress`. Do not silently broaden scope without documenting it.
