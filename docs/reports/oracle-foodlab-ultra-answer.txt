1) Reduce split_wait_seconds by making split-slot capacity adaptive (or better aligned to host limits)

Why it’s #1: split_wait_seconds is ~21% of prediction wall and indicates queue pressure while heavy slots are saturated.

File/function anchors

cookimport/cli.py: scheduler runtime shaping in _resolve_all_method_scheduler_runtime(...) (excerpt shows eval-tail/inflight math around ~7215+)

cookimport/cli.py: split slot gating and phase classification used for wait accounting:

_scheduler_phase_for_event(...) mapping includes split_wait_started → "split_wait" and split_active_started → "split_active" (both single-source and global-queue excerpts)

_compute_scheduler_counts(...) and _tick_scheduler_metrics(...) accumulate heavy-slot busy/capacity + wing backlog.

Optimization idea

Make split_phase_slots (aka heavy slots) adaptive to observed CPU utilization and idle gap, rather than fixed or only run-settings driven.

If CPU utilization is low while split_wait is high, scale split slots up until CPU or memory guard limits bite.

If CPU utilization is high but split_wait is high, reduce per-config split workers (see #2) and/or re-balance inflight vs split slots so “prep/post” don’t inflate wing backlog.

Expected impact

High. Even a 30–50% reduction in split_wait_seconds is material to wall time because wait is pure overhead.

Drift risk

Medium. Increasing parallelism can increase:

nondeterminism (ordering/timing),

transient resource failures (OOM, file descriptor pressure),

and can change throughput distribution across variants.

Scoring shouldn’t change, but failure modes can.

Guardrail telemetry

Must track per run:

split_wait_seconds, split_convert_seconds

heavy_slot_utilization_pct, idle_gap_seconds

avg_wing_backlog, max_wing_backlog

failure/timeout counters (timeouts force pool restart; see timeout handling around ~12660+).

2) Remove “false concurrency” by tuning/decoupling split workers per config from split slots (resource guard aware)

Why it’s #2: If each active split consumes too many workers, you can saturate CPU/memory and still have configs waiting, producing both high busy and high wait.

Optimization idea

If split_phase_slots is low, high split_workers per config can make each slot “fatter” than needed and reduce throughput.

Aim for a throughput sweet spot: more concurrent split jobs with fewer workers each often reduces tail/wait and improves fairness.

Expected impact

High when current runs show high split_wait_seconds and high heavy utilization: you can reduce wait and sometimes reduce convert time variance.

Drift risk

Low–Medium. Mostly performance-only, but can expose race conditions or flaky external tooling if parallelism changes.
