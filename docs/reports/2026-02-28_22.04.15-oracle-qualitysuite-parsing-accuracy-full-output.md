## Answer:
According to a document from 2026-02-28, the current QualitySuite tooling is usable for `recipeimport`, but the active run shape is mismatched to the immediate question “what parsing tools work best?” I based this on the attached code/docs bundle and the current tournament artifacts. 

### Section 1: Diagnosis

* The **architecture is fine; the current run shape is not**. `quality-run` already has bounded parallelism, race/exhaustive search, resume, and live ETA, while the tournament wrapper adds seed dedupe, shared caches, and gate-impossibility pruning. Those are good building blocks for a promotion flow. The active run, though, is the **full-candidate** tournament with **8 candidates + baseline**, **4 configured seeds**, **race**, and **deterministic sweeps on**—that is a broad top-tier/sweeps-decision run, not a fast parser-discovery run. `scripts/quality_top_tier_tournament.py`, `cookimport/bench/quality_runner.py`, `data/golden/bench/quality/tournaments/2026-02-28_20.25.10/tournament_resolved.json`, `data/golden/bench/quality/experiments/2026-02-28_16.24.30_qualitysuite-top-tier-tournament-full-candidates.json`

* The “**task 0/9**” symptom is mostly an **observability problem**, not proof that the run is stuck. The active checkpoint shows `experiment_count_total: 9` and `experiment_count_completed: 0`, and the runner only advances checkpoint/task completion **after an experiment returns**, not after internal configs/race rounds start doing work. `data/golden/bench/quality/tournaments/2026-02-28_20.25.10/fold_01_seed_42/quality_runs/2026-02-28_20.25.24/checkpoint.json`, `cookimport/bench/quality_runner.py`

* The real cost driver is **sweeps-on cardinality**. Repo docs explicitly note that on this suite shape, sweeps-off is **13 variants per target**, while sweeps-on is **143 variants per target**—about **11x** more variant work before race pruning. For a 9-experiment fold, that was estimated at about **8,334 config predictions**. `docs/understandings/2026-02-28_15.48.23-qualitysuite-tournament-sweeps-workload-and-prediction-reuse-scope.md`

* For **no-sweeps parser discovery**, **race is the wrong search mode** in this repo. With the base EPUB grid at 13 variants/target and `race_finalists=24`, race cannot prune, so it does more work than exhaustive: about **117** config evaluations vs **78** for one 6-target experiment. `docs/understandings/2026-02-28_21.01.58-qualitysuite-race-finalists-no-prune-overhead.md`, `cookimport/bench/quality_runner.py`

* Seed dedupe is already affecting your evidence budget. The current artifacts show **seed 314 duplicated fold 1**, so “4 configured seeds” does **not** mean “4 useful folds.” `scripts/quality_top_tier_tournament.py` intentionally skips duplicate suites before expensive runs. `data/golden/bench/quality/tournaments/2026-02-28_20.25.10/fold_02_seed_314/fold_result.json`, `scripts/quality_top_tier_tournament.py`

* Because of that dedupe behavior, `min_completed_folds=4` is **not a safe default** for this 6-target curated suite shape. Repo docs already record that the earlier fast no-sweeps thresholds became mathematically impossible once unique folds dropped to 2, and the feasible follow-up thresholds were relaxed to `min_completed_folds=2`. `docs/understandings/2026-02-28_20.20.04-fast-shortlist-fold-gate-impossibility-and-sweeps-decision-thresholds.md`

* The repo already contains the right **Phase A fast lane**: `--quick-parsing` narrows to parser-focused candidates, forces **sweeps off**, forces **exhaustive**, and applies a default seed cap unless overridden. That is exactly the flow you want for “what parsing tools work best?” `scripts/quality_top_tier_tournament.py`, `docs/07-bench/runbook.md`

* The repo-specific default baseline should stay the cross-source winner until beaten: **`unstructured` extractor + parser `v1` + preprocess `semantic_v1` + `skip_headers_footers=true`**. The current tournament experiment files also already lock in a conservative hot-IO runtime profile (`max_parallel_sources=2`, `max_inflight_pipelines=5`, `max_concurrent_split_phases=1`, `max_eval_tail_pipelines=6`, `wing_backlog_target=2`, `smart_scheduler=true`), which I would keep unchanged while answering the accuracy question. `docs/understandings/2026-02-28_10.31.55-quality-top-tier-tournament-baseline-and-gates.md`, `data/golden/bench/quality/experiments/2026-02-28_16.24.30_qualitysuite-top-tier-tournament-full-candidates.json`

### Section 2: Recommended Workflow

Repo-specific defaults I would use for `recipeimport`:

* **Baseline accuracy profile**: keep the current cross-source winner as the default incumbent: `unstructured` + `v1` + `semantic_v1` + `skip_headers_footers=true`. `cookimport/bench/quality_suite.py` also makes the curated focus set explicit: `saltfatacidheatcutdown`, `thefoodlabcutdown`, `seaandsmokecutdown`; with `max_targets=6` and `prefer_curated=true`, those should stay the anchor targets for both phases. `cookimport/bench/quality_suite.py`, docs digest

* **Runtime profile**: keep the current experiment-file runtime knobs as-is while you answer accuracy questions; do **not** widen source/pipeline concurrency until you know the winning accuracy settings. `data/golden/bench/quality/experiments/2026-02-28_16.24.30_qualitysuite-top-tier-tournament-full-candidates.json`

#### Phase A (Fast Discovery)

Use this when you **do not know the winner yet**.

**Recommended run shape**

* **Candidates**: **5 total** = baseline + `pre_br_split` + `pre_none` + `skip_headers_false` + `parser_v2_pre_br_skiphf_false`. That is the built-in parser-focused subset in `--quick-parsing`. `scripts/quality_top_tier_tournament.py`

* **Seeds**: **3 configured seeds** via `--max-seeds 3`. I am overriding the quick-parsing default here because the first two configured seeds (`42`, `314`) have already produced a duplicate fold in the current run, so the default 2-seed cap risks giving you only one unique fold of evidence. The third configured seed gives you a reasonable chance at a second unique fold, and duplicate folds are skipped before expensive `quality-run` work anyway. `data/golden/bench/quality/tournaments/2026-02-28_20.25.10/fold_02_seed_314/fold_result.json`, `scripts/quality_top_tier_tournament.py`

* **Targets**: keep **`max_targets=6`** and **`prefer_curated=true`**.

* **Sweeps**: **off**.

* **Search mode**: **exhaustive**. For this repo’s no-sweeps 13-variant parser grid, exhaustive is cheaper than race. `docs/understandings/2026-02-28_21.01.58-qualitysuite-race-finalists-no-prune-overhead.md`

* **Parallelism**: pin **`--max-parallel-experiments 4`**. That matches the repo’s own guidance for 3–6 experiments and is a good solo-maintainer stability point. `docs/07-bench/runbook.md`

**What counts as “good enough”**

* Get at least **2 unique folds**.
* The same candidate is **winner or tied-top on both** unique folds.
* **No source-success regression** vs baseline.
* Mean practical delta vs baseline is roughly **≥ +0.003**.
  That is enough to answer “what seems best?” and move on. It is **not** yet a default-promotion bar. The goal here is shortlist quality, not legal-proof certainty. The current tournament/report machinery already aggregates completed folds and deltas per candidate. `scripts/quality_top_tier_tournament.py`

**When to promote from A -> B**

Promote when:

* one candidate wins both unique folds, or
* only **two** candidates remain within about **0.003 practical F1** of each other.

If the result is noisy or split, Phase A has still done its job: it has reduced the field.

**Why this is the right shape**

On rough config-volume math, this is around **20x smaller per fold** than the current run: current active shape is about **8,334** config predictions per fold, while no-sweeps exhaustive discovery is about **5 experiments × 78 = 390** config evaluations per fold. That is the difference between “useful same day” and “wait all weekend.” This is an inference from the repo’s documented workload/cardinality numbers. `docs/understandings/2026-02-28_15.48.23-qualitysuite-tournament-sweeps-workload-and-prediction-reuse-scope.md`, `docs/understandings/2026-02-28_21.01.58-qualitysuite-race-finalists-no-prune-overhead.md`

#### Phase B (Confidence + A/B)

Use this when you **already have a strong candidate** and want to decide whether to change the repo default.

**Recommended run shape**

* **Candidates**:

  * normally **2 total** = baseline + Phase A winner;
  * use **3 total** only if Phase A left a very close runner-up.

* **Seeds**: **4 configured seeds** (`42, 314, 2718, 4242`) with duplicate-suite dedupe still enabled. You are aiming for **2–3 unique folds**, not blindly insisting on 4. `data/golden/bench/quality/thresholds/2026-02-28_20.20.04_qualitysuite-top-tier-gates-fast-sweeps-decision.json`, `scripts/quality_top_tier_tournament.py`

* **Targets**: same **6 curated+representative** targets.

* **Sweeps**: **off by default**. Current repo evidence has **not** shown repeatable sweep uplift, so sweeps should be a **separate question**, not part of the default promotion flow. `docs/understandings/2026-02-28_10.12.51-quality-sweep-quality-signal-and-top-tier-candidates.md`

* **Search mode**: **exhaustive** again, for the same low-cardinality reason.

* **Parallelism**: **4**.

**What counts as “good enough”**

For promotion, use the **feasible conservative gate shape**, not the old infeasible `min_completed_folds=4` fast-nosweeps file:

* no per-fold strict or practical regression worse than **0.004**
* no source-success regression
* uplift on **at least half** of the effective unique folds
* mean strict delta **≥ +0.004**
* mean practical delta **≥ +0.004**

That is essentially the repo’s current feasible gate philosophy (`min_completed_folds=2`, `min_uplift_fold_ratio=0.5`) adapted to a no-sweeps exhaustive A/B run. `data/golden/bench/quality/thresholds/2026-02-28_20.20.04_qualitysuite-top-tier-gates-fast-sweeps-decision.json`, `docs/understandings/2026-02-28_20.20.04-fast-shortlist-fold-gate-impossibility-and-sweeps-decision-thresholds.md`

**When to promote**

Promote the new default only if the candidate clears the bar above **and** the winner is consistent enough across folds that you are not just chasing a one-fold artifact. If it does not clear that bar, keep the current default baseline.

**Important nuance**

If you later want a true **“Do deterministic sweeps help at all?”** answer, treat that as an **optional separate B+ run**:

* baseline + current winner only
* sweeps **on**
* search strategy **race**
* at most **2 seeds**

That is the only place where the current sweeps-on race design really makes sense.

### Section 4: Minimal Code Changes Worth Doing

1. **Auto-switch `race -> exhaustive` when race cannot possibly prune.**
   In `cookimport/bench/quality_runner.py`, if the no-sweeps variant count is already `<= race_finalists`, skip race and run exhaustive directly. On this repo’s base parser grid, that would avoid the documented 117-vs-78 waste. This is a tiny change with no trust downside because it only replaces a slower equivalent search. `cookimport/bench/quality_runner.py`, `docs/understandings/2026-02-28_21.01.58-qualitysuite-race-finalists-no-prune-overhead.md`

2. **Add explicit seed selection (`--seed` repeatable or `--seed-list`).**
   Right now fast runs can only take the **first N** thresholds seeds via `--max-seeds`. That is awkward when the first two seeds can duplicate, as your current artifacts already show. Being able to choose e.g. `42,2718,4242` directly would remove pointless discovery overhead and make quick runs more reliable. `scripts/quality_top_tier_tournament.py`, current duplicate artifact

3. **Expose live subprogress from inside each experiment in the tournament wrapper.**
   The runner already has live ETA/work-unit telemetry, but the top-level tournament view still looks like “0/9” until the first experiment finishes. Have `scripts/quality_top_tier_tournament.py` echo the active fold’s `checkpoint.json` / `summary.partial.json`, or proxy the live ETA line from `quality-run`. This is a pure operator-trust improvement. `cookimport/bench/quality_runner.py`, current checkpoint artifact

4. **Widen prediction reuse scope across race rounds / experiments / folds.**
   Right now repo docs say prediction reuse is effectively local to each all-method run root, so it does **not** reuse across race rounds, experiments in the same `quality-run`, or folds in the tournament; and in-run reuse copies whole trees, which is I/O-heavy. This is the highest-ROI speedup for future sweeps-on work, but it is the least “minimal” item here. I would do it only after the operating model above is in place. `docs/understandings/2026-02-28_15.48.23-qualitysuite-tournament-sweeps-workload-and-prediction-reuse-scope.md`

## Section 5: What To Avoid

* **Stop using full-candidate, sweeps-on tournaments to answer parser questions.** That is a different question and a much larger workload. `data/golden/bench/quality/tournaments/2026-02-28_20.25.10/tournament_resolved.json`

* **Stop enabling deterministic sweeps in Phase A.** Repo evidence says sweeps have not yet shown repeatable uplift, while they explode runtime. `docs/understandings/2026-02-28_10.12.51-quality-sweep-quality-signal-and-top-tier-candidates.md`, `docs/understandings/2026-02-28_15.48.23-qualitysuite-tournament-sweeps-workload-and-prediction-reuse-scope.md`

* **Stop using race on no-sweeps 13-variant parser profiles with `race_finalists=24`.** In this repo, exhaustive is cheaper. `docs/understandings/2026-02-28_21.01.58-qualitysuite-race-finalists-no-prune-overhead.md`

* **Stop assuming configured seeds == evidence folds.** Trust **effective unique folds** after dedupe, not raw `configured_fold_count`. `scripts/quality_top_tier_tournament.py`, duplicate fold artifact

* **Stop using the old fast-nosweeps threshold file as a promotion gate on this suite shape unless you first verify unique folds >= 4.** The repo already documented why that was mathematically impossible. `docs/understandings/2026-02-28_20.20.04-fast-shortlist-fold-gate-impossibility-and-sweeps-decision-thresholds.md`

* **Stop waiting on the current `0/9` run for the parser answer.** Resume it later only if you still want the separate sweeps-decision answer. The parser answer should come from the quick-parsing flow. `data/golden/bench/quality/tournaments/2026-02-28_20.25.10/fold_01_seed_42/quality_runs/2026-02-28_20.25.24/checkpoint.json`, `scripts/quality_top_tier_tournament.py`

* **Stop mixing parser/preprocess questions with section-detector / ingredient / p6 knobs in the same first-pass run.** `full-candidates.json` mixes several families; that is fine for top-tier promotion, but it is the wrong first pass when you want attribution. `data/golden/bench/quality/experiments/2026-02-28_16.24.30_qualitysuite-top-tier-tournament-full-candidates.json`
