## What Actually Improved

The restart rerun is not just noise. The runtime changed in a concrete way.

What improved relative to `2026-03-19_21.09.56`:
- line-role strict score recovered materially
- `HOWTO_SECTION` false positives dropped from `139` to `98`
- `KNOWLEDGE -> HOWTO_SECTION` confusion dropped from `58` to `23`
- workers now tolerate some real workspace-local commands before dying

Concrete evidence that the new watchdog behavior is active:
- line-role worker `001` no longer dies on the first `cat worker_manifest.json` in the older style. In the restart rerun it was killed only after a shell-composed `cd ... && cat worker_manifest.json`: `line-role-pipeline/runtime/line_role/workers/worker-001/status.json`
- knowledge worker `001` survived `ls`, `cat worker_manifest.json`, `cat assigned_shards.json`, `cat in/...`, `tail assigned_shards.json`, and `ls out`, and only died on `rg`: `raw/llm/saltfatacidheatcutdown/knowledge/workers/worker-001/status.json`
- recipe worker `001` survived `pwd`, `cat worker_manifest.json`, `cat assigned_shards.json`, `ls`, `cat in/...`, and `ls out`, and only died on `ls shards`: `raw/llm/saltfatacidheatcutdown/recipe_phase_runtime/workers/worker-001/status.json`

So the fix was real:
- the old failure mode was "die on trivial startup reflexes"
- the new failure mode is "survive local helper commands, then still cross the boundary later"

## Cross-Stage Patterns

The restart rerun points to three distinct remaining command-policy mismatches.

1. Shell-composed convenience commands
- Example: `cd ... && cat worker_manifest.json`
- These are failing as `forbidden_shell_meta_command`
- This matters because the worker is already in the right directory and still wants to restate that position explicitly

2. Broader local inspection tools
- Examples: `rg`, `jq`, inline `python`
- These are failing as `forbidden_non_helper_executable` or `forbidden_shell_meta_command`
- This matters because they are exactly the sort of file-centric local tools Codex naturally reaches for when working over many shard files

3. Workspace-local but currently forbidden paths
- Examples: `ls shards`, `ls logs`, `cat AGENTS.md`
- These are failing as `forbidden_outside_workspace_path`
- This matters because these folders are still inside the sterile worker workspace, but the policy currently treats them as out-of-bounds for main worker use

The practical interpretation is:
- the main worker command allowance is no longer obviously too strict at startup
- it is still too strict for the broader local-file work style Codex wants to use over a longer session

## Current Best Interpretation

The restart rerun validates two things at once:

1. The new code is live
- The runtime behavior changed materially after restart.
- Workers now survive more normal local shell activity before being killed.
- The benchmark recovered substantially from the `21.09.56` crater.

2. The deeper mismatch is still unresolved
- The workspace-worker contract is still narrower than the natural local-file workflow Codex is attempting.
- Line-role can partly recover because the structured watchdog-retry path is strong enough to rescue many shards.
- Knowledge and recipe remain mostly dead because their main workers still die before producing usable outputs and their follow-up paths do not rescue them.

## Investigation Queue

The next likely high-value questions are:

1. Should `workspace_worker` main attempts allow a larger local-tool surface such as `rg`, `jq`, simple inline `python`, and read-only access to `logs/`, `shards/`, and `AGENTS.md`?
2. Should shell-composed convenience commands like `cd ... && cat ...` be normalized or tolerated when they stay inside the sterile workspace?
3. Should recipe get a recovery path more like line-role, since right now a main-worker kill is almost always terminal for that shard?
4. Should the worker prompt be tightened further to stop the `ls shards` / `ls logs` / `cat AGENTS.md` drift, or is that now fighting the model too hard and should the policy simply expand?

## Short Conclusion

`2026-03-19_21.24.41` is a real recovery from `2026-03-19_21.09.56`.

It is not a healthy worker run:
- all 15 main workers were still watchdog-killed

The restart rerun therefore confirms the current state clearly:
