# AI Readiness Improvement Report

Date: 2026-03-17

Scope of this review:

- `docs/01-architecture/01-architecture_README.md`
- `docs/12-testing/12-testing_README.md`
- `cookimport/cli.py`
- `cookimport/config/run_settings.py`
- `cookimport/config/run_settings_adapters.py`
- `cookimport/staging/import_session.py`
- `cookimport/labelstudio/ingest.py`
- `cookimport/plugins/base.py`
- package READMEs and conventions docs under `cookimport/`
- targeted local test runs:
  - `./scripts/test-suite.sh smoke` -> `116 passed` in about `10s`
  - `./scripts/test-suite.sh domain core` -> `25 passed` in about `2.2s`
  - `./scripts/test-suite.sh domain parsing` -> failed fast in about `2s` on a real parsing regression

## Executive summary

RecipeImport is already better than most repos for AI-assisted work because it has:

- unusually strong documentation,
- a disciplined test runner with fast slices,
- several real contract seams,
- explicit artifact structures and run manifests.

The main reason it is not yet truly "AI-shaped" is that too much real system knowledge still lives in a small number of very large coordinators and in cross-file coordination rules. The repo often looks modular at the folder level, but the safe change surface still spans multiple packages and giant files.

In plain terms:

- the docs make the repo readable,
- the tests make it safer,
- but the architecture still makes too many changes wider than they should be.

That means the next improvement phase should not be a broad rewrite. It should be a focused effort to reduce supervision cost in the three places where the repo currently demands the most global context:

1. the top-level CLI/orchestration surface,
2. the duplicated runtime logic between stage and Label Studio prediction/import flows,
3. the oversized run-settings/config surface.

If those three areas improve, the repo gets meaningfully more AI-ready without needing a wholesale re-architecture.

## Current score and what it means

Current rating from this review: roughly `6.5/10` for the specific "deep modules / AI-ready" standard.

That score is not a criticism of engineering quality overall. It reflects a specific mismatch:

- The repo is stronger in documentation and behavioral guardrails than in deep-module isolation.
- AI can succeed here, but it often succeeds by reading a lot and relying on tests, not by staying inside a small explicit interface.

This distinction matters. A repo that is "AI-usable" is not automatically "AI-ready" in the design sense from the video summary. RecipeImport is clearly AI-usable. It is only partly AI-ready.

## What the repo already does well

These are real strengths and should be preserved.

### 1. Documentation is unusually good

The repo has one of the biggest advantages an AI tool can have: a maintained local knowledge base.

Notable strengths:

- `docs/01-architecture/01-architecture_README.md` gives a real high-level runtime map.
- `docs/12-testing/12-testing_README.md` documents the actual test strategy, not just how to invoke `pytest`.
- package READMEs in `cookimport/` provide useful local orientation.
- the docs tree separates long-lived contracts from historical logs and ad-hoc task notes.
- `docs/understandings/` is a good mechanism for externalizing discoveries that would otherwise stay trapped in one debugging session.

This matters for AI because the model can start from an explicit artifact instead of reconstructing the architecture from imports and guesswork.

### 2. Test feedback is faster than average

The test story is materially better than average and supports AI-assisted change loops.

Evidence from this review:

- smoke suite: `116 passed` in about `10s`
- core domain slice: `25 passed` in about `2.2s`
- parsing domain slice: failed in about `2s`, which is good feedback behavior even though the suite is not currently green

The test organization is also thoughtful:

- domain folders,
- centralized markers,
- a real smoke lane,
- compact default output,
- wrapper scripts that encourage scoped runs.

That is exactly the kind of structure that helps both humans and AI iterate safely.

### 3. There are some genuine interface seams

This repo is not devoid of deep-module thinking. It already contains several promising contracts:

- `cookimport/plugins/base.py` defines a clean importer protocol.
- `cookimport/runs/stage_observability.py` creates a stable semantic stage-reporting model.
- `cookimport/config/run_settings_adapters.py` acts as a mapping boundary between configuration and runtime calls.
- `cookimport/staging/import_session.py` is already a shared post-conversion seam reused by multiple flows.

These are the right kinds of artifacts. They show the repo already knows how to express good boundaries. The problem is that these boundaries are not yet dominant enough to control the whole system.

### 4. The tests catch real behavior regressions quickly

During this review, the parsing suite quickly exposed a concrete regression:

- `tests/parsing/test_ingredient_parser.py::TestApproximate::test_pinch_moves_to_note`
- current behavior from `cookimport/parsing/ingredients.py` parses `"pinch of salt"` into `raw_unit_text='picoinch'`

That is useful evidence. It shows the repo is not only testable in principle; it already has enough lock-down to catch a subtle parser behavior problem quickly.

That kind of regression net is a prerequisite for safe internal refactoring.

## Where the repo is currently fighting the "deep modules" goal

### 1. The top-level orchestrator is much too large

`cookimport/cli.py` is about `30.6k` lines.

That single file contains:

- top-level command definitions,
- interactive flows,
- job planning,
- run summaries,
- benchmark orchestration,
- Label Studio wiring,
- progress handling,
- stage import execution paths,
- benchmark helper logic,
- output/report helper logic.

This is the single biggest architectural obstacle to AI-readiness in the repo.

Why this matters:

- a new contributor or AI agent cannot hold that file in working memory,
- the effective public interface is spread across implementation details,
- unrelated concerns appear adjacent, which makes "small safe edits" harder,
- any change touching orchestration risks accidental overlap with other responsibilities.

This is not just a style complaint. It changes how safely the repo can be modified. A file that large almost guarantees that "understand the interface first, internals later" stops working.

### 2. Shared runtime behavior is still duplicated across top-level flows

The repo has one good shared seam in `cookimport/staging/import_session.py`, but key runtime behavior still exists as duplicated orchestration logic in multiple places.

Concrete example from this review:

- `cookimport/cli.py:_plan_jobs(...)`
- `cookimport/labelstudio/ingest.py:_plan_parallel_convert_jobs(...)`
- `cookimport/config/run_settings.py:compute_effective_workers(...)`

The conventions docs explicitly encode this coordination burden. `cookimport/config/CONVENTIONS.md` says some changes must update:

- both split planners,
- stage and benchmark prediction paths,
- multiple adapter/report/dashboard surfaces.

That is a serious smell.

Good architecture should make "change the rule once" the default.
Right now the repo often uses "remember all the relevant places" as the effective architecture.

That hurts AI disproportionately because an AI agent is not carrying long-term repo memory between sessions. If the safe change set is broader than the local interface suggests, the agent must rediscover that hidden breadth every time.

### 3. The run-settings surface is too large relative to the real product

`RunSettings` currently has `72` live model fields in this repo snapshot.

The repo already knows this is too much. `docs/reports/2026-03-13-run-settings-surface-audit.md` makes the same point: the effective product surface is much smaller than the stored config shape.

This matters in several ways:

- manifests become harder to read,
- the user-facing product intent gets mixed with lab controls and implementation seams,
- new contributors cannot easily tell which settings are real product choices versus tuning leftovers,
- stage, benchmark, analytics, and dashboard code all inherit this complexity.

The cost here is not only UX cost. It is architecture cost. A giant config surface creates a giant supervision surface. It forces both humans and AI to keep asking:

- Which fields are truly supported?
- Which are legacy?
- Which are benchmark-only?
- Which are fixed in practice but still carried around?
- Which must be propagated to every manifest/report/output?

That is exactly the kind of structural burden the video summary is warning about.

### 4. Some package boundaries are mutual rather than layered

From a quick import-level scan during this review, the repo has several mutual package dependencies:

- `labelstudio <-> parsing`
- `labelstudio <-> llm`
- `labelstudio <-> staging`
- `llm <-> parsing`
- `llm <-> staging`
- `core <-> parsing`
- `bench <-> config`

Not all bidirectional dependencies are equally harmful, but as a pattern they weaken module depth.

Why:

- they make it harder to define a narrow public interface,
- they encourage leaking internal representations across domains,
- they make local reasoning less trustworthy,
- they turn folder boundaries into organizational hints instead of true dependency boundaries.

The worst case is not "the import graph has a cycle." The worst case is "the agent thinks it is editing one subsystem but is actually editing a knot."

### 5. Several large modules are domain bottlenecks

Beyond `cli.py`, there are other files large enough to create local bottlenecks:

- `cookimport/analytics/dashboard_render.py`
- `cookimport/labelstudio/ingest.py`
- `cookimport/config/run_settings.py`
- `cookimport/parsing/canonical_line_roles.py`
- `cookimport/bench/quality_runner.py`

Large files are not automatically bad. Some are justified by cohesive complexity.
The problem is when they become the place where a domain's rules, data model, orchestration, compatibility logic, and output rendering all mix together.

That reduces depth because the "public interface" of the module becomes "open the giant file and read around."

## The right target for this repo

Because this is a solo project, the target should not be maximal formalism.

The right target is:

- fewer places to remember,
- smaller safe-change surfaces,
- clearer ownership of runtime rules,
- better distinction between product interface and lab/debug internals,
- faster path from docs -> interface -> test -> change.

That means the right question is not "how do we make everything perfectly modular?"
It is:

"What architectural changes would let a fresh AI agent make a typical change while reading fewer files, touching fewer unrelated modules, and verifying behavior faster?"

For this repo, that leads to a focused plan.

## Recommendation 1: Split the CLI by responsibility, not by arbitrary line count

This should be the top structural priority.

### What to do

Refactor `cookimport/cli.py` into a small set of command modules with a thin composition root.

Suggested structure:

- `cookimport/cli_app.py`
  - creates `Typer` app
  - registers subcommands
  - contains almost no business logic
- `cookimport/commands/stage.py`
  - stage/import commands only
- `cookimport/commands/labelstudio.py`
  - import/export/eval/benchmark command entrypoints only
- `cookimport/commands/bench.py`
  - offline bench subcommands only
- `cookimport/commands/analytics.py`
  - dashboard/perf/compare-control entrypoints only
- `cookimport/interactive/`
  - interactive menu flows and prompt helpers only
- `cookimport/runtime/` or `cookimport/services/`
  - shared non-CLI orchestration units that commands call

### Why this helps

This is not just "move functions into more files."
It helps if and only if the extraction follows a strict rule:

- command modules parse user intent and call one shared service boundary,
- they do not re-own runtime rules.

The outcome you want:

- a new contributor can read the command file for user-facing behavior,
- then read one runtime service for actual execution behavior,
- without loading benchmark logic, interactive menu code, and report rendering at the same time.

### What not to do

Do not split `cli.py` into many files while keeping shared behavior duplicated.
That would reduce line count without reducing supervision cost.

### Success criteria

Good refactor outcome:

- command modules are mostly option normalization and handoff,
- most business logic moves into reusable services,
- changes to split planning or stage execution no longer require editing command files directly.

## Recommendation 2: Create one shared "processing runtime" seam for stage and Label Studio prediction flows

This is the highest-leverage deep-module improvement after the CLI split.

### The current problem

The repo already shares some post-conversion logic through `cookimport/staging/import_session.py`, but the surrounding orchestration still forks too early.

Stage flow and Label Studio prediction/import flow both need to reason about:

- run settings,
- effective worker behavior,
- split-job planning,
- extractor choices,
- per-file conversion,
- merge behavior,
- stage-backed artifacts,
- reporting and observability.

Today, too much of that knowledge is distributed between:

- `cookimport/cli.py`
- `cookimport/labelstudio/ingest.py`
- `cookimport/config/run_settings.py`
- conventions docs that tell maintainers to keep the copies aligned

### What to do

Introduce one shared service layer for "convert and stage a source under run settings."

Suggested seam:

- `cookimport/runtime/planning.py`
  - split-job planning
  - effective-worker resolution
  - extractor/file-type execution planning
- `cookimport/runtime/conversion.py`
  - convert one source or one split job
  - merge split results
- `cookimport/runtime/stage_pipeline.py`
  - run label-first + recipe/knowledge optional stages
  - produce a `StageImportSessionResult` or equivalent stable result model

Then:

- `cookimport/commands/stage.py` uses that runtime to write operator-facing stage outputs
- `cookimport/labelstudio/ingest.py` uses the same runtime to generate prediction-run artifacts and tasks

### Why this helps

This creates a deeper module because the true public interface becomes:

- "plan runtime"
- "execute runtime"
- "consume staged artifacts"

instead of:

- "remember the stage version"
- "remember the Label Studio version"
- "remember which config helper also needs the same rule"

### Important constraint

Do not aim for total unification in one step.
The shared seam should own only the genuinely shared runtime behavior:

- planning,
- conversion execution,
- staged result model,
- artifact production contracts.

Keep Label Studio-specific concerns separate:

- task generation,
- upload,
- export,
- prelabel,
- freeform task shaping.

That keeps the shared module deep rather than broad and muddy.

## Recommendation 3: Shrink `RunSettings` into explicit layers

This is the most important conceptual cleanup.

The current repo already hints at the right answer. `run_settings_adapters.py` separates:

- operator fields,
- benchmark lab fields,
- internal fields,
- fixed behavior fields.

That concept should become first-class in the data model.

### What to do

Replace one giant mental model of `RunSettings` with a layered contract model.

Suggested direction:

- `OperatorProfile`
  - the real run-to-run choices a human makes often
- `LabOverrides`
  - benchmark/tuning controls used intentionally during experimentation
- `InternalRuntimeConfig`
  - not a user-facing contract; derived or fixed
- `ResolvedRunConfig`
  - the final merged payload passed into execution

This does not require deleting persistence immediately.
The first win is semantic clarity.

### Why this helps

It turns an overloaded question:

- "What is a run setting?"

into clearer ones:

- "What is a product choice?"
- "What is a lab override?"
- "What is internal runtime detail?"

That immediately helps:

- interactive UI,
- docs,
- manifests,
- benchmark reports,
- analytics summaries,
- future refactors.

### Minimal pragmatic version

If you want a lower-effort start, do not redesign the whole model first.
Instead:

- keep `RunSettings` as storage,
- formalize and test the existing projections,
- make the operator-facing contract the dominant default everywhere,
- hide internal/lab fields from most summaries and UIs.

That is already a substantial improvement.

### What success looks like

A typical human or AI editing the repo should be able to answer:

- "What settings are real product choices?"

by reading one small contract file instead of inferring it from docs, UI behavior, and conventions.

## Recommendation 4: Enforce package layering around a smaller core

The repo does not need a perfect DAG, but it does need clearer dependency direction.

### Target layering

Reasonable target:

- `core`
  - primitive models, helpers, value objects, shared infrastructure
- `plugins`
  - source-specific ingestion behind the importer protocol
- `parsing`
  - deterministic parsing and normalization
- `staging`
  - output shaping and stage-authority writing
- `llm`
  - optional Codex/LLM augmentation on top of parsed/staged shapes
- `labelstudio`
  - task/export/eval surfaces consuming staged outputs
- `bench`
  - benchmark/reporting surfaces consuming stage and Label Studio artifacts
- `analytics`
  - read-side reporting only
- `commands` / `interactive`
  - top-level operator interfaces only

### Dependency rule

The main improvement is to prefer one-way dependency flow:

- lower layers should not import higher layers,
- read-side/reporting modules should not leak back into runtime configuration logic,
- `core` should not need to know about `parsing` if that relationship can be inverted.

### Pragmatic implementation

You do not need a big architecture enforcement system.
For a solo project, these are enough:

- define the intended layer map in docs,
- add a few lightweight import-boundary tests for the most important package relationships,
- treat new cross-layer imports as conscious exceptions rather than defaults.

Even partial enforcement pays off because it stops the repo from drifting further into "everything can reach everything."

## Recommendation 5: Turn "conventions that say update N places" into code-owned seams

One recurring pattern in this repo is that conventions docs explain how to keep multiple files aligned.

That is useful, but it is also a signal.

If a doc has to repeatedly say:

- update these two planners together,
- propagate this field to all these flows,
- keep these report surfaces aligned,

then the code probably wants a stronger shared seam.

### What to do

During refactors, actively mine the conventions docs for these "remember all the places" rules and convert them into:

- one adapter,
- one shared planner,
- one result model,
- one projection helper,
- one serialization boundary.

### Why this helps

This is one of the highest ROI ways to improve AI-readiness.

AI struggles less with complexity than with hidden breadth.
A long algorithm in one well-bounded place is often easier than three similar implementations plus a prose note telling you to keep them aligned.

## Recommendation 6: Protect refactors with more contract tests and fewer topology snapshots

The repo already has good test slicing. The next step is to lean even harder into contract-first tests around module boundaries.

### What to add

Prioritize tests for:

- split-job planning behavior,
- run-settings projection behavior,
- stage observability/report generation,
- stage/Label Studio shared runtime parity,
- import-boundary expectations between major packages,
- artifact contract shape at stable public boundaries.

### Example high-value tests

1. One contract test for split planning

- given input files and run settings,
- assert the planned job list and effective workers,
- run the same contract through both stage and Label Studio entrypoints if both still exist.

2. One contract test for run-settings projections

- assert which fields appear in operator, benchmark-lab, and internal projections,
- make this the lock that prevents surface creep.

3. One contract test for stage observability

- given a staged run root,
- assert the semantic stage rows and artifact pointers,
- avoid rebuilding stage topology logic ad hoc in many readers.

4. One import-boundary test

- assert a small set of prohibited package edges,
- especially around `core`, `parsing`, `staging`, `labelstudio`, and `bench`.

### What to avoid

Avoid overfitting tests to file topology that you intend to change.
The goal is not "freeze current folder layout forever."
The goal is "freeze public behavior and allowed dependency shape."

## Recommendation 7: Treat documentation as the interface layer, not the compensating layer

This repo already uses docs well. The next improvement is subtle:

- docs should describe the architecture,
- not compensate for architecture that is too distributed.

This means:

- keep architecture docs,
- but whenever a doc repeatedly teaches hidden implementation coupling, ask whether that coupling should become a code seam instead.

### Good use of docs

- explain stage ownership,
- explain artifact contracts,
- explain benchmark semantics,
- explain command topology.

### Less ideal use of docs

- "remember to update these three files together"
- "this knob must be propagated across four surfaces"
- "stage and benchmark need to stay in sync here"

Those are often invitations to refactor.

## Recommendation 8: Keep the refactor small-batch and evidence-driven

This repo should not do a giant "architecture rewrite."

That would be risky and unnecessary.

Instead, use a phased plan with a narrow success definition for each phase.

## Suggested phased plan

### Phase 1: Reduce the widest supervision surface

Primary goal:

- split `cookimport/cli.py` into command modules plus a thin composition root

Secondary goal:

- no functional behavior change,
- tests stay green,
- command ownership becomes obvious from file structure.

Success measure:

- `cli.py` stops being the place where all runtime knowledge lives.

### Phase 2: Introduce shared runtime planning and execution seams

Primary goal:

- unify split planning and shared conversion/staging behavior for stage and Label Studio flows

Success measure:

- a change to split logic or worker resolution happens in one service module,
- not in multiple top-level flows.

### Phase 3: Formalize the run-config layers

Primary goal:

- define operator, lab, and internal config contracts clearly in code

Success measure:

- manifests and UI defaults reflect the smaller product surface,
- internal knobs stop dominating mental overhead.

### Phase 4: Add boundary tests and prune mutual dependencies

Primary goal:

- stop the architecture from drifting back

Success measure:

- a small import-boundary suite enforces the intended package direction,
- key cycles are removed or consciously documented.

### Phase 5: Clean up the largest remaining domain bottlenecks

Candidate targets:

- `cookimport/labelstudio/ingest.py`
- `cookimport/config/run_settings.py`
- `cookimport/analytics/dashboard_render.py`

Only do this after the earlier seams exist.
Otherwise you risk splitting implementation details before the public interface is clear.

## Priority order for this repo

If you only do three things, do these:

1. split `cookimport/cli.py` by responsibility,
2. create one shared processing runtime seam for stage and Label Studio prediction/import behavior,
3. shrink the effective run-settings surface into explicit layers.

Those three changes attack the biggest drivers of human and AI supervision cost.

## What not to optimize for

This project is solo. The goal is not:

- enterprise abstraction,
- maximum indirection,
- complicated plugin systems everywhere,
- universal dependency injection,
- architecture ceremony for its own sake.

Avoid these traps:

- splitting files without narrowing interfaces,
- introducing many tiny modules that still depend on each other densely,
- replacing one giant file with many shallow coordinator files,
- preserving every historical config seam as if it were a product commitment.

Deep modules are not lots of modules.
Deep modules are modules that hide a lot behind a small, stable interface.

## A realistic target state

After a good refactor, a fresh AI agent should be able to do a typical task by following a shorter path:

1. read the domain README or architecture doc,
2. find the command or service interface,
3. inspect one bounded implementation area,
4. run one fast domain test slice,
5. make the change without loading unrelated orchestration code.

For RecipeImport today, that path is often longer:

1. read docs,
2. find the command,
3. discover the real behavior is split across giant orchestrators,
4. discover a conventions doc that names several other places,
5. load multiple domains,
6. hope the tests catch what you forgot.

The repo is already halfway there because the docs and tests exist.
The next step is to make the code structure deserve those docs and tests.

## Final recommendation

Do not frame this as "make the codebase more modular."
Frame it as:

"Reduce the amount of repo-wide context needed to make one safe change."

For this repo, that means:

- move user-facing command wiring out of giant coordinators,
- move shared runtime rules into one code-owned seam,
- make the real config surface smaller and more explicit,
- enforce a clearer layer direction between packages,
- add contract tests around those boundaries.

That is the shortest path from a good-but-complex codebase to one that is genuinely easier for both humans and AI to supervise.
