# All-Method Benchmark Slowdown And Cache Reuse Report

## 1) Scope

This report analyzes the all-method benchmark run at:

- `data/golden/benchmark-vs-golden/2026-02-27_17.54.41/all-method-benchmark`

and documents:

- where runtime was spent,
- why the run felt very slow,
- what work could be reused,
- what code changes were made to improve reuse,
- remaining bottlenecks and follow-up options.

Codex Farm / LLM parsing remained disabled and was not used.

## 4) Why It Felt Slow

### 4.1 Canonical-text alignment dominates heavy books

For large books and mismatched blockization settings, canonical-text alignment can be very expensive.

Example (`thefoodlabcutdown`, config `config_001_...pre_none`):

- `evaluate_alignment_seconds ~= 1656.42s`
- `evaluate_alignment_sequence_matcher_seconds ~= 1656.15s`

### 4.2 Cache hits existed but did not always mean fast wall time

From per-config `eval_report.json` telemetry:

- cache-hit configs: `46`
- cache-miss configs: `36`

However, many cache-hit configs still showed large `alignment_seconds` values.

Root cause:

- those configs often waited on cache lock ownership during in-run contention;
- after wait completed they loaded cached payload (`alignment_cache_hit=true`), but elapsed wall time still included the wait.

Observed pattern:

- cache-hit heavy configs had near-zero evaluator CPU while spending long wall time in alignment phase, consistent with waiting rather than recompute.

### 4.3 Cache scope was timestamp-run-local

Before the fix, all-method cache roots were under timestamped run output roots.

Effect:

- cache helped inside one run,
- but reruns could not reuse prior expensive alignments,
- the same canonical/prediction alignments could be recomputed across runs.

## 5) Reuse Opportunities Identified

### 5.1 High-leverage reuse: cross-run canonical alignment cache

Safe reuse key already exists:

- canonical normalized text hash,
- prediction normalized text hash,
- prediction block boundaries hash,
- plus schema/version signatures.

Therefore, sharing cache across runs is safe and high-value when inputs are unchanged.

### 5.2 Existing within-run reuse remains valid

Within-run per-source reuse should remain enabled; it prevents duplicate matcher CPU for repeated keys.

### 5.3 Remaining unavoidable cost

If canonical/prediction content truly differs (new hashes), cache miss is expected and alignment work must run.

## 6) Implemented Fix

### 6.1 Behavior change

All-method now resolves a shared canonical cache root across timestamped runs by default:

- default root: `data/golden/benchmark-vs-golden/.cache/canonical_alignment`
- per-source subpath: `<shared_root>/<source_group_key>`

Override support:

- env var: `COOKIMPORT_ALL_METHOD_ALIGNMENT_CACHE_ROOT`

### 6.2 Code changes

Primary wiring in:

- `cookimport/cli.py`

Notable additions:

- `_resolve_all_method_canonical_alignment_cache_root(...)`
- interactive all-method runner now prints resolved cache root before execution
- both all-matched and single-source interactive all-method flows use shared root behavior
- combined multi-source report now includes `canonical_alignment_cache_root`

### 6.3 Validation

Executed tests:

- `pytest -q tests/labelstudio/test_labelstudio_benchmark_helpers.py -k "canonical_alignment_cache_root or run_all_method_benchmark_multi_source_shards_source_and_reuses_cache_override or interactive_all_method_benchmark_all_matched_scope_routes_to_multi_source_runner or interactive_all_method_benchmark_uses_timestamped_output_root"`
- `pytest -q tests/labelstudio/test_labelstudio_benchmark_helpers.py -k "run_all_method_benchmark_multi_source"`

Both passed.

## 7) Expected Impact

### 7.1 First run after change

- Similar behavior to prior runs for pure misses.
- No regression expected in scoring correctness (cache key safety unchanged).

### 7.2 Subsequent reruns on same data/configs

- Significant wall-time reduction expected for repeated heavy canonical alignments.
- Biggest gains likely on books like `thefoodlabcutdown` and `saltfatacidheatcutdown`.

### 7.3 What this does not fix

- In-run lock-wait tails when duplicate keys start in parallel can still occur.
- This change targets cross-run reuse, not lock-wait elimination.

## 8) Remaining Bottlenecks And Practical Follow-Ups

1. Measure post-change rerun with same inputs to confirm cache-hit wall reduction.
2. Consider reducing duplicate in-run same-key contention (admission heuristics / key-aware scheduling) if tails persist.
3. Keep monitoring:
   - `alignment_cache_hit`
   - `alignment_cache_load_seconds`
   - `alignment_seconds`
   - evaluator CPU telemetry
