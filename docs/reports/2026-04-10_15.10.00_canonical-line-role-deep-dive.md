# Deep Dive: Canonical Line Role Orchestration & Validation

**Date:** 2026-04-10
**Author:** Gemini CLI
**Subject:** Technical Analysis of the Canonical Line Role Pipeline (Codex-Route-V2)

## Executive Summary
The Canonical Line Role system is a critical component of the `recipeimport` pipeline, responsible for assigning semantic labels (e.g., `RECIPE_TITLE`, `INGREDIENT_LINE`, `NONRECIPE_CANDIDATE`) to atomic line candidates. This deep dive focuses on the `codex-line-role-route-v2` architecture, which employs a hybrid approach: a high-speed deterministic heuristic pass followed by an LLM-driven refinement pass.

The investigation revealed a sophisticated, multi-layered validation and recovery system designed to handle the inherent fuzziness of LLM outputs. However, several "sharp edges" and potential architectural fragilities were identified, particularly in row-indexing consistency and heuristic brittleness.

---

## 1. Orchestration Flow: The Hybrid Pipeline

The entry point for line labeling is `label_atomic_lines` in `cookimport/parsing/canonical_line_roles/runtime.py`. It orchestrates two distinct phases:

### Phase 1: Deterministic Baseline
- Every line candidate is first evaluated against `_deterministic_label` in `policy.py`.
- This phase provides a "safe" fallback and a baseline for comparison.
- **Key Observation:** The system relies heavily on regex-based cues (e.g., `_YIELD_PREFIX_RE`, `_NOTE_PREFIX_RE`). While efficient, these are inherently limited by the variety of cookbook formatting.

### Phase 2: LLM Refinement (Codex-Route-V2)
- If the route-v2 pipeline is enabled, the system shards the candidates into manageable packets (default target ~240 lines).
- It performs a "survivability check" to ensure prompt sizes are within model limits.
- It launches parallel workers using `SubprocessCodexExecRunner` (or a mock for testing).
- **Advanced Recovery:** The system implements a "watchdog" mechanism and same-session handoffs to resolve invalid or missing JSON outputs without losing full session context.

## 2. Technical Deep Dive: Validation & Salvage Logic

The most complex logic resides in `validation.py`. Since LLMs often struggle with strict JSON schemas or replaying specific IDs, the system implements a robust "salvage" layer.

### The Indexing Challenge
The system uses `atomic_index` (integer) as the authoritative key. However, workers see `row_id` (e.g., `r01`, `r02`) or ordered arrays.
- `_translate_line_role_packet_local_ordinal_rows`: This function handles cases where the LLM returns local ordinals (1, 2, 3...) instead of the global `atomic_index`. It attempts to map these back to the original shard rows.
- `_translate_line_role_ordered_labels_to_atomic_indices`: If the LLM returns a simple array of labels, the system zips them with the input rows.

### The "Pathology" Guard
Located in `runtime.py` and `validation.py`, these guards detect common LLM failures:
- **Uniform Labeling:** Rejects shards where every single line is given the same label (e.g., 100 lines of `NONRECIPE_EXCLUDE`), as this is statistically improbable for most cookbook shards.
- **Row Mismatches:** If the returned label count doesn't match the input row count, the shard is marked as invalid and sent for repair.

## 3. Heuristic Brilliancy and Brittleness (`policy.py`)

`policy.py` is the largest file in the package (~90KB) and contains the "brains" of the deterministic pass.

### Strengths
- **Context Awareness:** The `_has_local_knowledge_prose_support` function looks at a radius (default 3) around a candidate to decide if a heading like "SALT" is a navigation element or a pedagogical heading.
- **First/Second Person Detection:** Uses regexes (`_FIRST_PERSON_RE`, `_SECOND_PERSON_RE`) to distinguish between "I like to..." (Notes/Narrative) and "Add salt..." (Instruction).

### Sharp Edges (Potential Bugs/Fragility)
1. **Instruction Verb Collisions:** `_INSTRUCTION_VERB_RE` is a static list. Verbs like "Plate" or "Dress" can easily be part of narrative prose. While they are cues, they lack semantic depth without a real parser.
2. **Title vs. Section Heading:** Distinguishing between a recipe title and a "How-To" section is a major source of ambiguity. The logic in `_looks_compact_heading` is heavily reliant on capitalization and length, which fails on stylized book designs.
3. **Integer vs String Indexing:** Throughout `runtime.py` and `validation.py`, `atomic_index` is frequently cast to `int`. While usually safe, the `AtomicLineCandidate` schema uses strings in some serialization paths. There is a risk of `KeyError` if a dictionary is keyed with `str("1")` but accessed with `int(1)`.

## 4. Discovered "Sharp Edges" and Observations

### Observation 1: The "Spill" Logic
In `_translate_line_role_packet_local_ordinal_rows`, there is code to handle "trailing spill" (where the LLM returns more rows than requested). This is a clever defensive measure against LLMs that hallucinate or repeat the next few lines of text.

### Observation 2: Deterministic Vetoes
The documentation states that the system *no longer* semantically vetoes structurally valid Codex labels. However, `_apply_line_role_semantic_guard` (currently a stub in `validation.py`) exists as a placeholder for where these vetoes *used* to live. This suggests an architectural shift toward trusting the model while retaining the plumbing for future constraints.

### Potential Bug: `_coerce_shard_input_rows`
In `validation.py`:
```python
try:
    atomic_index = int(row[0])
except (TypeError, ValueError):
    continue
```
If a row has an invalid index, it is silently dropped. This could cause the validation to fail the "row count match" check later, triggering a repair cycle when it might have been a simple malformed ID.

### Potential Bug: Regex Overlap
`_RECIPE_ACTION_CUE_RE` and `_INSTRUCTION_VERB_RE` share many words. `_INSTRUCTION_VERB_RE` is anchored to the start of the line, while the action cue is a boundary search. If a line is `Note: Add salt`, it might match the action cue and be biased toward `INSTRUCTION_LINE` despite the `Note:` prefix.

---

## 5. Conclusion
The Canonical Line Role system is a robust, production-grade implementation of hybrid AI labeling. Its strength lies in its defensive validation and salvage logic, which treats LLM outputs as potentially corrupt until proven otherwise. The primary risks are the maintenance burden of the massive `policy.py` heuristic file and potential subtle bugs in the integer/string handling of atomic indices during the complex salvage/translation phase.

**Future Work Recommendation:**
- Consolidate regex cues into a more structured taxonomy to avoid overlap.
- Ensure strict type-checking for `atomic_index` across all serialization boundaries.
- Expand the "Pathology Guard" to include more sophisticated statistical checks (e.g., label distribution entropy).
