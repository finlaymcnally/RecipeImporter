# Test Suite Impact Audit
**Date:** 2026-03-05
**Verification pass updated:** 2026-03-05
**Status:** Revised for correctness

## Executive Summary
The test-suite risk is real, but narrower than the earlier report claimed.

The accurate story is:

- some tests are tightly coupled to current AI-on defaults,
- some scheduler and QualitySuite tests are coupled to current `stable_hash()` behavior,
- but the suite is not broadly set up to leak live Codex calls during normal execution,
- and the low-level LLM kill-switch mitigation already exists in both runtime code and test setup.

## What Is Verified
### 1. Some tests hard-code current defaults
These tests are genuinely coupled to the current default configuration and will need edits if defaults flip from AI-on to AI-off.

Strong examples:

- `tests/llm/test_run_settings.py`
- `tests/cli/test_cli_output_structure_fast.py`

These tests assert current default values such as:

- `llm_recipe_pipeline == "codex-farm-3pass-v1"`
- `line_role_pipeline == "codex-line-role-v1"`

### 2. Some scheduler and benchmark tests depend on current hash semantics
This coupling is real, but should be described precisely.

Relevant examples:

- `tests/labelstudio/test_labelstudio_benchmark_helpers_scheduler.py`
- `tests/bench/test_quality_suite_runner.py`

These tests rely on current `stable_hash()` distinctions or on report payloads that carry `run_settings_hash` values through expected outputs.

If hash semantics are pruned, some of these tests will need targeted updates.

### 3. The low-level kill switch is already integrated with the test suite
The earlier report treated this as a future blocker. That was stale.

Current verified state:

- runtime already gates low-level LLM execution through `COOKIMPORT_ALLOW_LLM`
- `tests/conftest.py` already sets `COOKIMPORT_ALLOW_LLM=1` in test setup

So the repo is not waiting on that mitigation.

## What Was Overstated
### 1. Broad "live LLM leak" framing
That wording was too aggressive.

Many LLM-related tests do not perform live Codex work. They mainly exercise seams through:

- fake runners
- monkeypatched subprocess calls
- mocked helper functions

That applies especially to:

- `tests/llm/test_codex_farm_orchestrator.py`
- much of the line-role testing that monkeypatches Codex helpers

### 2. `tests/core/test_recipe_likeness_scoring.py` as hidden-LLM risk
Not supported.

That test area exercises deterministic scoring logic. It is not a demonstrated accidental live-LLM path.

### 3. `tests/parsing/test_canonical_line_roles.py` as a primary hash-coupling site
Also overstated.

That file matters for mocked line-role behavior, but it is not a primary site of `stable_hash()` assertions.

## Revised Risk Map
| Test area | Actual risk | Why it matters |
|-----------|-------------|----------------|
| `test_run_settings.py` | High | breaks if defaults flip |
| `test_cli_output_structure_fast.py` | High | breaks if CLI/default expectations flip |
| scheduler / quality-suite hash assertions | Medium | breaks if `stable_hash()` semantics change |
| line-role tests with monkeypatched Codex seams | Low to medium | behavior assertions may drift, but not mainly a live-call problem |
| codex-farm orchestrator fake-runner tests | Low | mostly seam coverage, not live usage dependence |
| deterministic scoring tests | Low | not a real hidden-LLM risk |

## Practical Meaning For The Fix
If the underlying runtime problem is fixed, the test suite should be treated as a targeted maintenance task, not as a reason to avoid the fix.

Expect mostly three categories of fallout:

1. update tests that hard-code current defaults,
2. update tests that expect current hash behavior,
3. leave alone tests that already use fake runners or patched seams unless behavior actually changes.

## Final Verdict
### Verified
- default-flip breakage risk is real
- hash-coupling risk is real in specific scheduler and QualitySuite tests
- the test suite already has the kill-switch environment allowance in place

### Best current wording
- **maintenance risk:** real
- **live-call risk during normal test execution:** overstated in the original report
- **main implication:** fixing the runtime bug will require a focused test cleanup, not a test-suite rescue operation
