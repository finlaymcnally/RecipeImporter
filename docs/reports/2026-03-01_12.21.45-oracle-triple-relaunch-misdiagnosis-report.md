# Oracle triple-relaunch misdiagnosis report

## Scope
This report explains why the same Oracle prompt was launched three times during one task, even though previous runs were still alive (or at least not clearly failed at launch time).

Audience: another AI agent or maintainer implementing fixes in Oracle CLI/wrappers/session state reporting.

## Incident summary
- Prompt intent: one browser Oracle run (`gpt-5.2-pro`) with 21 attached repo files for QualitySuite/PDF architecture review.
- Actual outcome: three separate browser sessions were launched for effectively the same request:
  - `project-context-recipeimpo-python-determinis`
  - `qualitysui-pdf-setup-live-2`
  - `qualitysui-pdf-setup-live-3`
- Relauch trigger: I interpreted lack of visible progress as failure.

## Evidence snapshot

### Session metadata snapshot
Source: `~/.local/share/oracle/sessions/<slug>/meta.json` for each of the three slugs.

At inspection time, all three had:
- `meta.status = running`
- `models[0].status = running`
- `models[0].error = null`
- valid `browser.runtime.tabUrl` pointing to a ChatGPT conversation URL
- `models/gpt-5.2-pro.log` size `0`
- `output.log` containing only launch lines:
  - packed bundle message
  - launch mode/model
  - estimated runtime

### Conflicting status surfaces
Two status commands produced materially different views:

- `oracle status --hours 24` (default home):
  - only showed an unrelated API error session.
- `ORACLE_HOME_DIR=/home/mcnal/.local/share/oracle oracle status --hours 24`:
  - showed the three browser sessions as `error`.
- Meanwhile each `meta.json` still showed `running`.

This created uncertainty about which status source was authoritative.

### Reattach behavior evidence
Running:

```bash
ORACLE_HOME_DIR=/home/mcnal/.local/share/oracle oracle session <slug> --render
```

showed:

```text
Attempting to reattach to the existing Chrome session (port XXXX, url=https://chatgpt.com/c/...)
Existing Chrome reattach failed (connect ECONNREFUSED 127.0.0.1:XXXX); reopening browser to locate the session.
```

This message confirms reattach failure on the stored debug port, but does not cleanly disambiguate:
- dead run,
- stale metadata,
- or run still recoverable through browser/session lookup.

## Timeline (UTC from session metadata)

1. `2026-03-01T16:45:17Z` first browser run created  
   slug: `project-context-recipeimpo-python-determinis`
2. `2026-03-01T16:54:48Z` second run created  
   slug: `qualitysui-pdf-setup-live-2`
3. `2026-03-01T17:00:51Z` third run created  
   slug: `qualitysui-pdf-setup-live-3`

Between launches, there was no definitive per-session progress signal (no model log growth, no heartbeat events in `output.log`).

## Why the runs looked failed from the operator perspective

1. **No incremental telemetry**
   - `output.log` stayed static after launch.
   - model log remained 0 bytes.
   - no explicit phase transitions (waiting for UI vs submitted vs streaming).

2. **Visibility mismatch in browser mode**
   - Earlier process tree inspection showed a Chromium launched with minimized/off-screen flags:
     - `--start-minimized`
     - `--window-position=32000,32000`
     - tab often `about:blank`
   - From operator view, this resembled "automation never actually submitted."

3. **Port/profile reuse ambiguity**
   - Multiple attempts involved debug port `9222`; later one used `9233`.
   - If an older hidden/minimized browser is still attached to the expected port/profile, current run state is hard to infer.

4. **Status inconsistency**
   - `status=error` in `oracle status` while session metadata still said `running`.
   - PIDs in metadata later pointed to non-existent processes (stale runtime fields), further reducing trust in metadata.

5. **Home-dir split**
   - Oracle data appeared in both default CLI view and wrapper-managed home paths depending on invocation path/env.
   - Without explicit `ORACLE_HOME_DIR`, status queries looked incomplete or unrelated.

## Direct cause of triple relaunch
The relaunch decision heuristic used by the operator was effectively:

`no new stdout + no model log bytes + hidden/minimized/blank browser + uncertain reattach status => likely dead => relaunch`

Given current Oracle observability, that heuristic is understandable but wrong.

## Root causes in tool behavior (fix targets)

1. **Missing heartbeat/state machine observability**
   - No durable phase/heartbeat stream per session.

2. **Non-authoritative status surfaces**
   - `oracle status` and `meta.json` can disagree for long periods.

3. **Stale runtime metadata**
   - session metadata can retain dead PID fields while session appears `running`.

4. **Reattach UX under-specifies next step**
   - `ECONNREFUSED` message is descriptive but not decisive about run viability.

5. **Duplicate-run prevention is too weak in browser mode**
   - easy to launch a new run while an equivalent one is unresolved.

## Recommended fixes (prioritized)

1. Add session heartbeats and phase events (highest impact)
   - Write periodic lines to `output.log` and structured events to metadata:
     - `browser_started`
     - `navigated_chatgpt`
     - `prompt_submitted`
     - `awaiting_model`
     - `streaming_tokens`
     - `completed`
     - `recovering_reattach`
   - Include `last_activity_at` and `last_activity_kind`.

2. Make `oracle status` authoritative and stale-aware
   - On status read, verify PID liveness.
   - If stored `running` but controller/chrome PIDs are dead and no recent activity, auto-mark as `error_stale_runtime` (or `abandoned`).
   - Expose `status_reason`.

3. Improve `oracle session --render` reattach messaging
   - After reattach failure, print a deterministic next-state line:
     - `state=reattach_failed_recovering`
     - `state=unrecoverable`
     - `state=recovered_new_browser`
   - Avoid ambiguous silence after fallback begins.

4. Add duplicate-prompt running-session guard
   - Before launching a new run, detect existing unresolved sessions with same prompt hash + file bundle hash + model.
   - Default action: show candidate session and prompt to attach (or auto-attach in non-interactive mode).

5. Expose canonical Oracle home in every command output
   - Print `oracle_home=<path>` at start of `status`, `run`, and `session --render`.
   - Reduces accidental split-brain debugging across home directories.

6. Explicit browser visibility status
   - Record whether browser is visible/minimized/headless and the effective chrome command.
   - Warn when current run is sharing a debug port/profile that already has an existing Chrome process.

## Acceptance criteria for fix

Run this scenario once after implementation:
- Launch browser run with large bundle.
- Immediately inspect `status` and session files.
- Kill browser process and re-run `session --render`.
- Confirm:
  - heartbeat lines continue or terminal status updates quickly,
  - status/meta agree on terminal state within bounded time,
  - duplicate launch attempt is blocked or redirected to attach,
  - reattach failure message includes decisive next state.

If all above pass, this incident class should be largely eliminated.
