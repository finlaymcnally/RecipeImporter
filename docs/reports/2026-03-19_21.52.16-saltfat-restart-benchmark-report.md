## What Changed After Restart

Compared with the immediately prior run `2026-03-19_21.09.56`, the restarted run recovered substantially:

- strict accuracy improved from `0.594816` to `0.656207`
- macro F1 excluding `OTHER` improved from `0.485950` to `0.516845`
- changed lines dropped from `328` to `180`
- net error delta dropped from `162` to `72`
- line-role net error dropped from `138` to `64`
- final-recipe net error dropped from `24` to `8`
- inside-span delta improved from `-0.091603` to `-0.038278`
- outside-span delta improved from `-0.114618` to `-0.050915`
- predicted `HOWTO_SECTION` lines dropped from `139` to `98`

Interpretation:
- The restart did not produce a no-op rerun. It changed the actual prediction behavior enough to recover most of the `21.09.56` collapse.
- The `21.09.56` run should therefore not be treated as the stable post-watchdog baseline.
- The restarted run looks much closer to the earlier `17.08.33` regime: still bad, but not catastrophically bad.

## What Stayed Broken

The core failure story did not change.

Line-role remains the dominant benchmark problem:
- line-role still accounts for `64` of the `72` net error lines (`88.9%`)
- the run still has `0` gold `HOWTO_SECTION` lines but predicts `98` of them
- top confusion deltas are still the same family:
  - `KNOWLEDGE -> OTHER`: `+35`
  - `OTHER -> HOWTO_SECTION`: `+29`
  - `KNOWLEDGE -> HOWTO_SECTION`: `+21`

Knowledge remains effectively dead as an authority stage:
- `authority_mode=knowledge_unreviewed_seed_kept`
- `changed_block_count=0`
- `changed_block_indices=[]`
- `scored_effect=seed_only`
- all `5` knowledge workers were watchdog-killed
- no knowledge shard validated

Recipe correction remains effectively dead as a rescue stage:
- all `5` recipe-correction workers were watchdog-killed
- all `5` recipe-correction shards ended with missing output
- stage observability still shows `30` recipe-correction empty-output signals
- final recipe build still shows `30` runtime-missing or unobserved outputs

This matters because even after the restart recovered the overall score, the runtime still did not produce live downstream correction. The benchmark is still mostly scoring seed line-role behavior with no meaningful LLM rescue.

## Likely Diagnosis

The restart helped, but it did not fix the actual disease.

Most likely:
1. The `21.09.56` run included either stale process state or an unstable runtime/model interaction that amplified the regression.
2. Restart removed that amplification and returned the system to its more typical bad regime.
3. The typical bad regime is still dominated by line-role overgeneralization, especially promoting nonrecipe headings and prose into `HOWTO_SECTION` or otherwise flattening `KNOWLEDGE` / `OTHER` distinctions.
4. Knowledge and recipe stages are still too broken to rescue those mistakes.

One important caveat:
- The new runtime/watchdog work improved observability and probably reduced some bad agent drift, but the benchmark artifacts still do not surface the new command-policy telemetry cleanly enough to say which line-role workers were killed for forbidden commands versus looping without producing output.
- That limits how much this report can attribute the remaining failure to watchdog strictness versus pure labeling quality.

## Practical Takeaways

- Treat `2026-03-19_21.24.41` as the more trustworthy post-restart benchmark, not `2026-03-19_21.09.56`.
- Do not treat the restart as a fix. It recovered score, but it did not revive knowledge or recipe correction.
- The next quality work should still focus first on line-role semantics, especially the false `HOWTO_SECTION` pattern and the `KNOWLEDGE` versus `OTHER` boundary.
- The next runtime work should expose the new command-policy telemetry inside benchmark artifacts so future reports can distinguish “helpful local file reads” from real drift or loop-without-output failures.
