# Salt Fat Acid Heat Codex Rerun Stage Health Report

This report records the overall behavior of benchmark run `2026-03-19_21.24.41` on `saltfatacidheatcutdown`.

Benchmark root:
- `data/golden/benchmark-vs-golden/2026-03-19_21.24.41/single-book-benchmark/saltfatacidheatcutdown`

Processed run root:
- `data/output/2026-03-19_21.24.41/single-book-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-19_21.25.15`

Companion comparison report:
- `docs/reports/2026-03-19_21.50.37-saltfat-line-role-rerun-comparison-report.md`

High-level result:
- Restarting the program did change the outcome in a meaningful way.
- CodexFarm `strict_accuracy` improved from `0.594816` in the earlier `21.09.56` rerun to `0.656207` here.
- The Codex-vs-vanilla line-accuracy delta improved from `-0.110505` to `-0.049113`.
- That improvement did not come from stage recovery downstream. Knowledge and recipe still failed completely as active Codex stages.
- The best current reading is that line-role behavior improved materially, while the watchdog / forbidden-command runtime problem remained unresolved.

## What Improved

The main improvement is visible in the benchmark attribution artifacts, not in downstream stage health.

- net error delta dropped to `72`

Compared with the prior bad rerun at `21.09.56`:
- changed lines were previously `328`
- net error delta was previously `162`
- `line_role` previously accounted for `138` net error lines
- `final_recipe` previously accounted for `24` net error lines

The specific quality movement is also consistent with the label breakdown:
- predicted `HOWTO_SECTION` fell from `139` in `21.09.56` to `98` here
- `KNOWLEDGE -> HOWTO_SECTION` contamination fell sharply
- outside-active-recipe-span damage got much smaller

Current interpretation:
- The restart rerun reduced the non-recipe heading / prose contamination that was dominating the bad `21.09.56` run.
- The improvement looks like a line-role behavior recovery, not a true whole-pipeline recovery.

## Best Current Interpretation

This rerun changed in a real way, but not in the most important operational way.

What the run demonstrates:
- restarting the program altered model/runtime behavior enough to produce a much better benchmark result than `21.09.56`
- the improvement is mostly attributable to better line-role output behavior
- knowledge and recipe still did not recover into healthy stages
- the watchdog / forbidden-command problem is still the main blocker preventing a trustworthy full-stack Codex run

What the run does not demonstrate:
- it does not show that the knowledge-stage regression is fixed in production behavior, only that it now fails closed and reports honestly
- it does not show that recipe correction is working
- it does not show a stable, healthy multi-stage Codex pipeline

## Current Priority Queue

The next debugging pass should stay focused.

1. Treat `watchdog_command_execution_forbidden` as the primary blocker across knowledge and recipe.
2. Keep line-role work scoped to the specific remaining classification damage, because that is the only stage currently moving benchmark quality.
3. Fix the apparent line-role telemetry label leak so `authority_mode` reporting matches the actual stage.
4. Do not interpret downstream benchmark changes as recipe or knowledge recovery until those stages produce validated shards.

## Short Conclusion

Run `2026-03-19_21.24.41` is a meaningful recovery from the bad `2026-03-19_21.09.56` rerun.

It is still not a healthy Codex pipeline run:
- benchmark quality is still below vanilla
- knowledge remains fully unreviewed
- recipe correction remains completely dead
- all Codex-backed stages still show watchdog / forbidden-command pathology

The practical conclusion is simple:
- restart helped
- line-role improved
- the core runtime failure still did not
