# Deep Dive: Canonical Line Role Assignment (Phase 2 Architectural Audit)
**Date:** 2026-04-10_15.00.00
**Author:** Gemini CLI
**Subject:** Technical Analysis of `cookimport/parsing/canonical_line_roles/`

## 1. Executive Summary

This report documents a deep dive into the Canonical Line Role assignment module, a critical component of the recipe parsing pipeline. The investigation tracked the lifecycle of a line from initial atomization to final canonical labeling. 

Key findings include a transitionary architectural state where legacy labels (`KNOWLEDGE`, `OTHER`) are still being processed by deterministic rules but coerced at the boundary, and a pervasive logic dependency on `within_recipe_span` which often defaults to `None`, potentially bypassing intended sanitization heuristics.

---

## 2. Architectural Overview

The `canonical_line_roles` package acts as the semantic classifier for atomic lines. It sits between the **Atomizer** (which breaks blocks into lines) and the **Span Grouper** (which defines recipe boundaries).

### The Pipeline Flow:
1.  **Ingestion:** Receives a sequence of `AtomicLineCandidate` objects.
2.  **Deterministic Baseline (`policy.py`):** Every line is passed through a battery of regex-based heuristics to establish a "rule" label.
3.  **LLM Execution (`runtime.py`):** If the pipeline is configured for LLM routing (e.g., `codex-line-role-route-v2`), lines are sharded and sent to an LLM.
4.  **Validation (`validation.py`):** LLM responses are mapped back to atomic indices and validated against the allowed label set.
5.  **Normalization (`policy.py`):** Final labels are sanitized and coerced to ensure they meet the live contract before being returned.

---

## 3. Deep Dive: The Deterministic Engine (`policy.py`)

The core of the deterministic logic resides in `_deterministic_label`. It follows a strict priority order for labeling:

1.  **Section Headings:** Matches against `HOWTO_SECTION` and `RECIPE_VARIANT` prefixes.
2.  **Structural Evidence:** Looks for `ingredient_like` or `instruction_like` tags generated by the atomizer.
3.  **Metadata Identification:** Identifies `TIME_LINE` and `YIELD_LINE` based on specific numeric and unit cues.
4.  **Prose Classification:** Differentiates between recipe-local notes and general knowledge prose.

### The `within_recipe_span` Heuristic Problem
A significant finding during this dive was the handling of `within_recipe_span`. In many production paths (like `label_source_of_truth.py`), this field is explicitly set to `None` because the span boundaries haven't been defined yet.

However, `policy.py` uses strict boolean checks:
```python
def _is_within_recipe_span(candidate: AtomicLineCandidate) -> bool:
    return candidate.within_recipe_span is True

def _is_outside_recipe_span(candidate: AtomicLineCandidate) -> bool:
    return candidate.within_recipe_span is False
```

**Impact:** When the span is `None`, both functions return `False`. This causes any logic gated by "outside" or "inside" checks to be skipped. For example, in `_apply_repo_baseline_semantic_policy`, the following code:
```python
if label == "TIME_LINE" and not _is_primary_time_line(candidate.text):
    label = "RECIPE_NOTES" if _is_outside_recipe_span(candidate) else "INSTRUCTION_LINE"
```
defaults to `INSTRUCTION_LINE` if the span is `None`. This may result in front-matter metadata being incorrectly categorized as instructions before the span grouper has a chance to see them.

## 4. LLM Integration and Transport (`runtime_workers.py`)

The module supports two primary transport modes for communicating with LLM workers:
-   **TASKFILE_TRANSPORT:** A legacy-style file-based handoff.
-   **INLINE_JSON_TRANSPORT:** The modern Phase 2 approach where the worker communicates via structured JSON packets.

### Validation Logic
The `validation.py` module contains sophisticated translation logic (`_translate_line_role_output_to_atomic_indices`) that handles various failure modes in LLM output:
-   **Row ID Mapping:** LLM workers often return `row_id` (e.g., "r01") instead of `atomic_index`. The validator reconstructs this mapping.
-   **Partial Responses:** It detects "prefix" or "trailing spill" responses where the LLM might have truncated the output.
-   **Label Normalization:** It uses `normalize_freeform_label` to ensure slight variations in LLM naming (e.g., "Ingredient" vs "INGREDIENT_LINE") are resolved correctly.

## 5. Legacy Residue and Phase 2 Shift

The investigation revealed that the code is in a "half-migrated" state regarding the `KNOWLEDGE` and `OTHER` labels.

-   **Contract:** `contracts.py` excludes these from `CANONICAL_LINE_ROLE_ALLOWED_LABELS`.
-   **Policy:** `policy.py` still contains hundreds of lines of code dedicated to identifying `KNOWLEDGE` prose.
-   **Coercion:** In `_normalize_prediction_metadata`, any label identified as `KNOWLEDGE` or `OTHER` is forcibly coerced:
    -   Inside a recipe -> `RECIPE_NOTES`
    -   Outside a recipe -> `NONRECIPE_CANDIDATE`

This indicates a significant architectural shift where "Knowledge" is no longer a first-class citizen of the line-role stage, but the heuristics to find it are being preserved as "evidence" for later stages.

## 6. Identified Bugs and Anomalies

### 6.1. Redundant TIME_LINE Sanitization
In `policy.py`, there is logic to sanitize `TIME_LINE` if it is not a "primary" time line. However, the `_deterministic_label` function already performs this check:
```python
if "time_metadata" in tags and _is_primary_time_line(candidate.text):
    return "TIME_LINE", ["time_metadata"]
```
Because the baseline *only* returns `TIME_LINE` if it passes the check, the downstream sanitization in `_apply_repo_baseline_semantic_policy` that checks `if label == "TIME_LINE" and not _is_primary_time_line(...)` is logically unreachable for rule-based labels.

### 6.2. KNOWLEDGE Leakage in Semantic Guard
There is a potential edge case in `_knowledge_allowed_inside_recipe` where `KNOWLEDGE` can survive if it is surrounded by `explicit_prose` tags. If this happens, and the label is not coerced by the subsequent normalization, it could theoretically reach the grouper despite being excluded from the allowed labels contract.

---

## 7. Upstream and Downstream Dependencies

### Upstream: `recipe_block_atomizer.py`
The atomizer is responsible for the `rule_tags` (e.g., `time_metadata`, `instruction_with_time`) that drive the deterministic baseline. If the atomizer's regex for `time_metadata` is too broad (which it is, matching any mention of "minutes"), it places a heavy burden on the `canonical_line_roles` module to filter out false positives.

### Downstream: `recipe_span_grouping.py`
The span grouper relies on `RECIPE_TITLE` as an anchor. If the canonical role stage misidentifies a title (perhaps because it was tricked by a `RECIPE_VARIANT` heading in the front matter), the entire span grouping logic will fail, potentially merging multiple recipes into one or creating orphaned fragments.

---

## 8. Conclusion

The `canonical_line_roles` module is a robust but complex piece of the architecture. Its reliance on "Evidence-Based Determinism" (using tags from the atomizer) combined with LLM "Fuzzy Semantic Calls" aligns with the project's core philosophy. 

However, the "Phase 2" shift has left behind legacy label handling that adds complexity. Furthermore, the strict boolean handling of the `within_recipe_span` field represents a brittle point in the logic when the pipeline is processing data in a "span-unknown" state.

### Recommendations for Future Agents:
1.  **Refactor Span Checks:** Update `_is_within_recipe_span` and `_is_outside_recipe_span` to handle `None` explicitly, or ensure the upstream `label_source_of_truth.py` provides a default "hint" state.
2.  **Clean Legacy Labels:** Once the Phase 2 migration is confirmed stable, the `KNOWLEDGE` and `OTHER` heuristics in `policy.py` should be moved to a dedicated "Evidence" field rather than being generated as labels that are immediately discarded.
3.  **Tighten Atomizer Tags:** The `time_metadata` tag in the atomizer should be made stricter to reduce the number of non-primary time lines entering the `TIME_LINE` logic path.
