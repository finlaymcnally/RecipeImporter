# Test Suite Audit: Quality & Obsolescence Report
**Date:** 2026-04-10_13.20.00
**Audit Scope:** Full `tests/` directory analysis

## Summary of Findings
The test suite is functionally comprehensive but qualitatively imbalanced. It excels at verifying **infrastructure** (the "pipes")—ensuring LLM sessions, transports, and benchmark runners operate without crashing—but is significantly weaker at verifying **empirical parsing quality**. Many high-level tests are "frozen" against outdated assumptions or utilize mocks that guarantee success.

---

## Bucket 1: The "Success" Mirage (Mocked Benchmarks)
*Tests that pass by definition because they mock the evaluator or use hardcoded "success" metrics.*

- **`tests/labelstudio/test_labelstudio_benchmark_smoke.py`**
  - **Why:** Uses `_simulated_canonical_eval_report` which returns hardcoded precision/recall numbers. It verifies the CLI doesn't crash, but it provides zero validation of the actual extraction logic.
- **`tests/labelstudio/test_labelstudio_benchmark_sim.py`**
  - **Why:** Similar to the smoke test, it relies on simulated runtimes and pre-baked results.
- **`tests/bench/test_quality_suite_runner.py`**
  - **Why:** Validates that the runner can "discover" experiments and start workers, but the actual "quality" checks are often bypassed or mocked in the suite to keep runs fast.
- **`tests/bench/test_quality_suite_discovery.py`**
  - **Why:** Purely structural; tests if the system can see files, not if the files are good.

---

## Bucket 2: Generous "Practical" Metrics
*Tests that use lower bars for success (e.g., "any overlap" instead of strict alignment).*

- **`tests/test_eval_freeform_practical_metrics.py`**
  - **Why:** The primary offender. It explicitly validates a mode where "any overlap" counts as a perfect match (1.0). This metric is too generous for high-fidelity recipe parsing.
- **`tests/bench/test_segmentation_metrics.py`**
  - **Why:** Tests the boundary detection logic using metrics that tolerate "close enough" shifts that would break a deterministic parser.
- **`tests/bench/test_segeval_adapter.py`**
  - **Why:** Uses standard windowed metrics (Pk/WindowDiff) which are notoriously forgiving of minor segmentation drift.
- **`tests/parsing/test_ingredient_parser.py`** (Partial)
  - **Why:** Includes `test_backend_selection` which only asserts that the raw text is preserved, not that the different backends actually parsed the quantity/unit correctly.

---

## Bucket 3: Infrastructure vs. Function (Pipes vs. Water)
*Large, complex tests that verify the "pipes" (transports, safety, sessions) rather than the "water" (recipe data).*

- **`tests/llm/test_codex_exec_runner.py`** (1100+ lines)
  - **Why:** Exhaustive testing of telemetry, event extraction, and pathological reasoning detection. Excellent infrastructure, but zero recipe-specific validation.
- **`tests/llm/test_codex_exec_runner_taskfile.py`** (1400+ lines)
  - **Why:** Tests the "cage" around the LLM, filesystem syncing, and command classification.
- **`tests/llm/test_codex_farm_orchestrator_runner_transport.py`**
  - **Why:** Verifies how JSON packets move between workers.
- **`tests/llm/test_codex_farm_orchestrator_watchdog.py`**
  - **Why:** Verifies that the system can kill a hanging LLM session.
- **`tests/analytics/test_stats_dashboard.py`**
  - **Why:** Tests that the dashboard *renders* data, not that the data being rendered is accurate to a real-world import.
- **`tests/cli/test_cli_output_structure_slow.py`**
  - **Why:** Merely checks if CLI flags end up in a `runConfig` JSON object.

## Bucket 4: "One-Off" Regression Bloat
*Tests that are hyper-specific to one book or a fixed bug from years ago, providing little general value.*

- **`tests/parsing/test_canonical_line_roles.py`**
  - **Why:** Filled with cases like "Salt, Fat, Acid, Heat" (SFAH) specific sentences. The logic is effectively hardcoded for Samin Nosrat's book.
- **`tests/ingestion/test_epub_extraction_quickwins.py`**
  - **Why:** Contains "quickwin" assertions like `duplicate_block_rate > 0.35`. These are heuristic hacks that were likely added to fix a specific bad EPUB once.
- **`tests/staging/test_draft_v1_priority6.py`**
  - **Why:** Deeply specific to a "v1" yield scoring mode (`scored_v1`) that feels like a point-in-time experiment.
- **`tests/core/test_phase1_manual.py`**
  - **Why:** Tests specific mojibake like `SautÃ©`. Useful once, but now just noise in the suite.

## Bucket 5: Outdated "Phase 1" Cruft
*Low-level unit tests for components that have likely been superseded or are now too simple to be useful.*

- **`tests/core/test_atoms.py`**
  - **Why:** Basic string splitting into "atoms". In a world of LLM-based segmentation, these low-level character-counting tests are largely vestigial.
- **`tests/core/test_recipe_likeness_scoring.py`**
  - **Why:** Uses "Toast" and "Cereal" as gold-standard examples. These are so simple that they fail to catch edge cases in real, complex cookbook layouts.
- **`tests/parsing/test_atoms.py`** (Old path)
  - **Why:** Redundant with `test_core/test_atoms.py`.
- **`tests/core/test_phase1_manual.py`** (Also Bucket 4)
  - **Why:** Validates basic report generation that has since been replaced by the more complex `ConversionResult` and `Analytics` pipelines.
