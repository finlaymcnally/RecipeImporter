# Knowledge Stage Failure Report

Scope:
- benchmark run `data/golden/benchmark-vs-golden/2026-03-18_21.10.12`
- benchmark run `data/golden/benchmark-vs-golden/2026-03-18_21.17.59`

Goal:
- capture the observed knowledge-stage failures
- separate schema-transport failure from shard-output-integrity failure
- record likely root causes and concrete fix directions before context compaction

## Executive Summary

There are two distinct failure classes in the recent knowledge-stage benchmark runs:

1. `2026-03-18_21.10.12` failed before any successful knowledge shard completed because the shipped structured-output schema was invalid for the live Codex/OpenAI validator.
2. `2026-03-18_21.17.59` did not hit the schema bug. Knowledge executed and billed, but 4 of 5 knowledge shards failed proposal validation because the model did not return valid owned-chunk coverage for those shards.

These are different bugs and need different fixes.

## Run A: 2026-03-18_21.10.12

Book:
- `dinnerfor2cutdown`

Observed behavior:
- knowledge was enabled in settings
- knowledge did not land in post-run actual-cost breakdown
- manifest recorded `knowledge_not_run_runtime_failed`

Evidence:
- `run_manifest.json` showed `llm_knowledge_pipeline=codex-knowledge-shard-v1`
- `run_manifest.json` showed `knowledge_prompt_target_count=5`
- `manifest.json` recorded `authority_mode: knowledge_not_run_runtime_failed`
- benchmark manifest preserved the Codex error:
  - `invalid_json_schema`
  - nested knowledge decision object had property `rc` in `properties` but not in that nested object's `required`

Impact:
- knowledge never completed a successful runtime shard
- actual cost artifact only included recipe correction and line role

Root cause:
- shipped schema `llm_pipelines/schemas/recipe.knowledge.v1.output.schema.json` violated the live structured-output contract at a nested object level
- local tests only enforced the `properties`/`required` rule at schema top level, not recursively

Status:
- fixed locally by:
  - adding nested `rc` to the schema's nested `required` list
  - adding a recursive schema-pack regression test

## Pattern Across Run B

The failing shards show three concrete failure modes:

1. dropped-owned-chunk output
2. empty output set
3. invented fallback row with non-owned `cid` plus invalid evidence surface

The common thread is not schema rejection. It is output-integrity loss under knowledge shard execution.

## Most Likely Root Causes

### 1. Knowledge shards are too large and cognitively unstable

Signals:
- 78 owned chunks on several failing shards
- one shard produced a "processing_error" fallback instead of faithful chunk-by-chunk output
- another produced an empty `r`

Inference:
- the shard compaction / bundling strategy is likely past the model's reliable structured-output capacity on some books

### 2. The knowledge prompt allows failure-shaped behavior without a hard contractual escape hatch

Signals:
- model invented `cid: "processing_error"`
- model returned `r: []`

Inference:
- the prompt is clear about desired output, but not strong enough about forbidden fallback behaviors
- it probably needs explicit language like:
  - never invent synthetic chunk ids
  - never emit an empty result list when `c` is non-empty
  - if uncertain, still emit one row per owned chunk with conservative `other` decisions and empty snippets

### 3. Validation is strong, but preflight and test coverage are too weak on shard-output integrity

Signals:
- validator caught the problems correctly
- benchmark still paid for 5 knowledge calls even though 4 were invalid

Inference:
- current tests likely cover schema shape and some proposal validation seams, but not the large-shard failure modes that produce:
  - one missing chunk
  - empty `r`
  - invented fallback chunk ids
  - out-of-surface evidence on synthetic fallback outputs

### 4. Prompt transport / pack metadata remain conceptually mixed for knowledge

Signals:
- runtime is inline direct-exec
- pack metadata for knowledge still says `prompt_input_mode: "path"`
- prompt text is now inline-oriented

Inference:
- this did not directly cause the shard failures in Run B, but it increases the chance of stale tests, stale assumptions, and mismatched debugging surfaces

## Recommended Fix Order

### Immediate

1. Keep the schema fix and nested schema regression test in place.
2. Add focused knowledge validator tests for:
   - one missing owned chunk id
   - empty `r`
   - synthetic non-owned `cid`
   - out-of-surface evidence on fallback rows

### Next

3. Reduce knowledge shard size aggressively for large books.
   - likely lower target chunks per shard
   - avoid huge heterogeneous bundles
   - prefer more shards over fewer oversized shards

4. Tighten the prompt contract.
   - forbid synthetic chunk ids
   - forbid empty `r` when input `c` is non-empty
   - require exactly one result per input chunk
   - require conservative `other` output instead of bailout behavior

### After that

5. Add an execute-path fake/fixture regression that exercises knowledge shard validation under representative large-shard outputs, not just schema-pack or tiny happy-path examples.

6. Consider normalizing pack metadata to reduce the "path pack vs inline runtime" mismatch around knowledge.

## Test Gaps To Fill

Needed regression cases:
- knowledge structured-output schema recursive `required` enforcement
- knowledge proposal validation rejects:
  - empty result list
  - missing one owned chunk
  - synthetic fallback chunk ids
  - evidence indexes outside owned block surface
- benchmark/report artifact test proving a partially invalid knowledge run still reports `completed_with_failures` and preserves invalid-shard counts

## Bottom Line

- `21.10.12` was a schema bug
- `21.17.59` was a shard-output-integrity bug under real knowledge workload
- the second problem is now the more important production-quality issue
