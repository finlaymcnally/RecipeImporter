# Sandbox Parallelism Investigation Report (2026-02-28)

## Executive Summary

The persistent slowdown is real and is rooted in sandbox process-worker restrictions, not one single scheduler bug. In this runtime, process pools fail with `PermissionError: [Errno 13] Permission denied`, which is consistent with blocked `/dev/shm` writes. That failure forces fallback behavior in multiple places:

- `quality-run` now has a **new mitigation**: experiment fanout can auto-switch to subprocess workers.
- several other paths still degrade to threads or full serial mode.

Conclusion:

- **QualitySuite appears partially solved** (experiment-level fanout path).
- **The broader project is not solved yet** (stage + split ingestion still have serial fallbacks under this sandbox).

## Scope

Inspected and tested:

- docs and plans around recent QualitySuite parallelism work
- runtime code paths in:
  - `cookimport/bench/quality_runner.py`
  - `cookimport/cli.py`
  - `cookimport/labelstudio/ingest.py`
- targeted tests in `.venv`
- direct runtime probes for `/dev/shm` and process-pool availability
- a real local `cookimport stage` smoke run

## Evidence

### 1. Host/Sandbox process-pool constraint is present now

Direct checks in this environment:

- `/dev/shm` is present but not writable by this runtime user context:
  - `touch /dev/shm/recipeimport_probe_<ts>` -> `Permission denied`
- direct process-pool probe fails:
  - `ProcessPoolExecutor(max_workers=1)` -> `PermissionError: [Errno 13] Permission denied`
- project import also surfaces related dependency fallback:
  - `joblib ... will operate in serial mode`

This matches prior docs in:

- `docs/understandings/2026-02-28_11.45.55-qualitysuite-low-cpu-due-shm-permission-and-thread-fallback.md`
- `docs/07-bench/07-bench_README.md` (merged notes around SemLock + `/dev/shm`)

### 2. QualitySuite has new subprocess experiment fallback

Code evidence:

- executor resolver added in `cookimport/bench/quality_runner.py:281`
- `auto` mode chooses subprocess when process-pool probe fails (`...:304`)
- subprocess worker path exists (`...:339`)
- resolved metadata persists executor mode/reason (`...:689`)
- dispatch uses subprocess path when selected (`...:740`)

Runtime probe in this sandbox:

- `_resolve_quality_experiment_executor_mode(max_parallel_experiments_effective=4)` returns:
  - mode: `subprocess`
  - reason: `process_pool_unavailable:PermissionError: [Errno 13] Permission denied`

Targeted test evidence:

- `pytest tests/bench/test_quality_suite_runner.py -k "switches_to_subprocess_executor_when_process_pool_unavailable"` -> `1 passed`
- test confirms `experiments_resolved.json` records `experiment_executor_mode == "subprocess"` (`tests/bench/test_quality_suite_runner.py:795`, `:885`)

### 3. Non-QualitySuite paths still force serial in this environment

#### A) `cookimport stage` still falls back to serial

Code path:

- `cookimport/cli.py:16978` tries `ProcessPoolExecutor(max_workers=effective_workers)`
- catches `PermissionError` and prints serial fallback warning (`...:17065`)

Reproduction:

- ran `cookimport stage /tmp/stage_probe2 --workers 2 ...`
- observed:
  - `Processing 2 file(s) as 2 job(s) using 2 workers...`
  - `⚠ Multiprocessing unavailable; running jobs serially.`

#### B) split conversion in ingestion falls back to serial

Code path:

- `cookimport/labelstudio/ingest.py:1624` starts split `ProcessPoolExecutor`
- on `PermissionError`, runs `_run_job_serial(...)` loop (`...:1687`, `...:1690`)

This affects split heavy EPUB/PDF conversion work where speed depends on process workers.

### 4. All-method benchmark path is better than serial, but still constrained

Code paths in `cookimport/cli.py`:

- process probe helper (`:7524`)
- if unavailable, falls back to **thread-based config concurrency**, not immediate serial (`:10464`, `:13860`)
- true serial only if thread executor also fails

Test coverage confirms thread fallback behavior:

- `tests/labelstudio/test_labelstudio_benchmark_helpers.py:6609`

Implication: this is not strictly serial, but under heavy Python CPU work and blocked process workers, thread fallback can still look near-serial in wall time.

## Behavior Matrix

| QualitySuite experiment fanout (`quality-run`) | bounded parallel experiments | auto -> subprocess fanout | Improved/partially solved |

## Answer To “Did we solve it in QualitySuite?”

Short answer: **mostly yes for one layer, no for the full stack**.

What is solved:

- QualitySuite experiment-level orchestration now avoids the thread/GIL bottleneck under this sandbox by switching to subprocess workers when process-pool probing fails.

What is not solved:

- if run has only one experiment, `max_parallel_experiments_effective` can still be `1` (no experiment fanout to parallelize)
- per-experiment inner work can still hit thread/serial fallbacks in deeper ingestion/conversion layers
- non-QualitySuite workflows (`stage`, split conversion) still have hard serial fallback paths

## Why agents experience “forced serial”

The observed behavior is a combination of:

1. hard serial fallbacks (`stage`, split conversion)
2. thread fallback under process-pool failure (parallel but often much slower for CPU-heavy Python paths)
3. run shapes with low exploitable fanout (for example single experiment)
4. dependency-level serial warnings (joblib)

## Recommended Next Steps (Priority Order)

1. Extend subprocess fallback pattern to `stage` worker orchestration.
2. Extend subprocess fallback pattern to split conversion in `cookimport/labelstudio/ingest.py`.
3. Add explicit executor telemetry fields to stage/split run artifacts (mode, reason, env probe result), mirroring `experiments_resolved.json`.
4. Add a small “parallelism doctor” command/check that reports:
   - `/dev/shm` writable?
   - process pool probe result
   - resolved executor mode per workflow
