# Token Usage Leak Audit & "Nuked" Tokens Report
**Date:** 2026-03-04
**Incident Timeframe:** 21:00 - 21:24 UTC
**Total Incident Cost:** ~$94.48 (gpt-5.3-codex)

## 1. Executive Summary
Between 21:00 and 21:24 today, a series of automated quality benchmarks triggered an unexpected surge in LLM activity. This resulted in **2,703 messages** sent to `gpt-5.3-codex`, consuming **10M input tokens** and **250M cache-read tokens**. The primary cause was a combinatorial explosion in an "exhaustive" benchmark sweep combined with LLM-by-default pipeline settings.

## 2. The "Nuked" Tokens Scenario: Anatomy of a Leak
The incident was driven by the following command/batch:
- `deterministic-ultrafast-singleton-exhaustive-batch` (384 unique configurations)
- Multi-seed quality runs (Seeds 42, 77, 123, 211, 319)

### The Root Causes
1.  **Invisible Defaults:** In `cookimport/config/run_settings.py`, the `line_role_pipeline` defaults to `codex-line-role-v1`. This means any "default" import or benchmark run that doesn't explicitly disable it will call the LLM for every block of text.
2.  **Config Hash Fragmentation:** The `prediction_reuse_key` (used for caching LLM results) is calculated based on the *entire* `RunSettings` payload. In the exhaustive sweep, 384 configurations were tested. Even if a configuration only changed a non-LLM setting (e.g., changing `epub_extractor` from `unstructured` to `beautifulsoup`), the `run_config_hash` changed. This invalidated the cache and forced a fresh LLM call for the same text across all 384 variants.
3.  **Weak Gating:** While `bench quality-run` has an `--include-codex-farm` flag, it only controls the "Recipe Pipeline" (the big 3-pass extraction). It does **not** gate the `line_role_pipeline`. Consequently, the benchmarks ran "without Codex Farm" for the extraction phase but still "leaked" thousands of calls for the line-role classification phase.

## 3. Other Potential Misuse Vectors
During the audit, we identified several other ways AI usage can "leak" without an explicit human call:

### A. Prelabeling on Import
The `labelstudio ingest` flow (in `cookimport/labelstudio/prelabel.py`) has an interactive prompt for AI prelabeling. If a user (or an automated script mimicking a user) selects a non-off mode, it can trigger massive parallel calls to annotate an entire book before the human even sees it.

### B. Default "On" Pipelines
- `llm_recipe_pipeline`: Defaults to `codex_farm_3pass_v1`.
- `line_role_pipeline`: Defaults to `codex-line-role-v1`.
These should arguably default to `off` or `deterministic-v1` to ensure LLM usage is opt-in.

### C. Benchmark "Race" & "Exhaustive" Strategies
The `quality-run` command uses `race` as a default strategy. While `race` prunes bad configs, the first round ("probe") still runs against multiple targets. If those targets are large books, the initial token hit is significant before any pruning occurs.

## 4. Recommendations for "Human-Only" Gating
To ensure using `codex-farm` is always an intentional "human call," we propose the following changes:

1.  **Change Defaults to Deterministic:**
    - Flip `line_role_pipeline` default to `deterministic-v1`.
    - Flip `llm_recipe_pipeline` default to `off`.
2.  **Hard Gating in CLI:**
    - Implement a global `--allow-llm` flag or environment variable that must be present for *any* pipeline (including line-role) to call `run_codex_json_prompt`.
3.  **Coarse-Grained Caching:**
    - Refactor `_build_all_method_prediction_reuse_key` to only include the settings that actually affect that specific LLM pipeline (e.g., only prompt-related settings), allowing cache hits across sweeps that vary unrelated parameters.
4.  **Cost Estimation / Confirmation:**
    - For large benchmark runs, calculate a "dry-run" cost estimate based on the number of expected prompts and target length, requiring an explicit `I_ACCEPT_THE_COST` confirmation if it exceeds a certain threshold (e.g., $5.00).
