# Token Usage Leak Audit & "Nuked" Tokens Report
**Date:** 2026-03-04
**Incident Timeframe:** 21:00 - 21:24 UTC
**Total Incident Cost:** ~$94.48 (gpt-5.3-codex)

## 1. Executive Summary
Between 21:00 and 21:24 today, a series of automated quality benchmarks triggered an unexpected surge in LLM activity. This resulted in **2,703 messages** sent to `gpt-5.3-codex`, consuming **10M input tokens** and **250M cache-read tokens**. The primary cause was a combinatorial explosion in an "exhaustive" benchmark sweep combined with LLM-by-default pipeline settings and sensitive caching logic.

## 2. The "Nuked" Tokens Scenario: Anatomy of a Leak
The incident was driven by the following command/batch:
- `deterministic-ultrafast-singleton-exhaustive-batch` (384 unique configurations)
- Multi-seed quality runs (Seeds 42, 77, 123, 211, 319)

### The Root Causes
1.  **Invisible Defaults:** In `cookimport/config/run_settings.py`, the `line_role_pipeline` defaults to `codex-line-role-v1`. This means any "default" import or benchmark run that doesn't explicitly disable it will call the LLM for every block of text.
2.  **Config Hash Fragmentation:** The `prediction_reuse_key` (used for caching LLM results) is calculated based on the *entire* `RunSettings` payload. In the exhaustive sweep, 384 configurations were tested. Even if a configuration only changed a non-LLM setting (e.g., changing `epub_extractor` from `unstructured` to `beautifulsoup`), the `run_config_hash` changed. This invalidated the cache and forced a fresh LLM call for the same text across all 384 variants.
3.  **Weak Gating:** While `bench quality-run` has an `--include-codex-farm` flag, it only controls the "Recipe Pipeline" (the big 3-pass extraction). It does **not** gate the `line_role_pipeline`. Consequently, the benchmarks ran "without Codex Farm" for the extraction phase but still "leaked" thousands of calls for the line-role classification phase.

## 3. Comprehensive Audit: 8 Ways LLMs are Called "Behind Your Back"
The audit identified eight distinct subsystems where AI-by-default or implicit settings trigger LLM activity without a direct "human call":

1.  **Line-Role Classification (Silent Leaker):** Defaults to `codex-line-role-v1` on every import/benchmark. Uses a per-run cache, preventing sharing between sessions.
2.  **Main Recipe Pipeline:** Defaults to `codex-farm-3pass-v1` in base `RunSettings`.
3.  **Automatic Retries:** `_LINE_ROLE_CODEX_RETRY_ATTEMPTS = 3`. Triple-calls on failures.
4.  **Parallelism Saturation:** Up to 8 inflight batches can "nuke" tokens at high velocity.
5.  **Label Studio Prelabeling:** Interactive prompt can trigger mass-processing of whole books.
6.  **Interactive "Run Profiles":** `CodexFarm` profile enables multiple LLM knobs simultaneously.
7.  **Combinatorial Sweep Loops:** Exhaustive sweeps/tournaments invalidate caches via config-hash changes.
8.  **Automated Scripts:** `scripts/` (e.g., transport replays) often trigger live calls bypassing CLI gates.

## 4. Proposed Prevention Gates & "Kill Switches"
To prevent burning tokens and ensure LLM usage is strictly intentional, we propose the following gates:

### A. The "Hard Gate" (Low-Level Blocking)
Implement a check at the absolute lowest level in `cookimport/llm/codex_exec.py:run_codex_json_prompt`. This function should raise an error immediately if `COOKIMPORT_ALLOW_LLM` is not set to `1` in the environment, regardless of what the higher-level pipelines requested. This is the ultimate "Kill Switch."

### B. "Deterministic-by-Default" Policy
Invert all pipeline defaults in `cookimport/config/run_settings.py`:
- `line_role_pipeline` -> `deterministic-v1`
- `llm_recipe_pipeline` -> `off`
- `llm_knowledge_pipeline` -> `off`
- `llm_tags_pipeline` -> `off`

### C. The "Cost Speedbump"
For any command that estimates a cost > $1.00 (calculated by counting target text blocks before starting), the CLI must:
1.  Display a prominent **RED WARNING**.
2.  Require the user to type `I_ACCEPT_THE_COST` or wait for a 10-second "abort" countdown.

### D. Benchmark "Cache-Only" Mode
Introduce a `--llm-cache-only` flag for benchmarks. In this mode, if a configuration results in a cache miss, the system fails the experiment rather than calling the LLM. This prevents "sweeps" from accidentally burning tokens on untested variants.

### E. Coarse-Grained Caching
Refactor the `prediction_reuse_key` to exclude non-LLM settings. A change in `workers` or `epub_extractor` should **not** invalidate a Line-Role prediction cache.

## 5. Summary Recommendations
1.  **Invert the Defaults:** Move from "Opt-Out" to "Explicit Opt-In."
2.  **Global Kill Switch:** Enforce environment variable gating at the `codex_exec` level.
3.  **Cost Awareness:** Implement dry-run cost estimation for all high-volume commands.
