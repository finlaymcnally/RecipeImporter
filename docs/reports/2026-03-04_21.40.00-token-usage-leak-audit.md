# Token Usage Leak Audit
**Date:** 2026-03-04
**Verification pass updated:** 2026-03-05
**Status:** Revised for correctness

## Executive Summary
The hidden line-role problem was real, but the earlier writeup mixed together three different things:

1. intentional Codex benchmark usage,
2. the narrower hidden QualitySuite baseline contamination,
3. external token totals that are not preserved in this repo.

The strongest repo-backed conclusion is:

- retained March 4 QualitySuite baseline-style runs were not truly deterministic,
- because they used `llm_recipe_pipeline=off` together with `line_role_pipeline=codex-line-role-v1`,
- and that hidden line-role activity likely consumed about `2.0M` to `2.2M` tokens for the retained contaminated QualitySuite sweep,
- with a plausible billing-truth band of about `1.7M` to `2.5M` tokens.

That is a real waste event, but it is not the same thing as the much larger intentional Codex usage visible elsewhere in the CSVs.

## What Is Verified
### 1. Defaults were AI-on
Current `RunSettings` defaults still enable both:

- recipe Codex: `llm_recipe_pipeline=codex-farm-3pass-v1`
- line-role Codex: `line_role_pipeline=codex-line-role-v1`

### 2. QualitySuite baselines were hybrid
Retained March 4 QualitySuite experiment payloads show baseline-style runs with:

- `llm_recipe_pipeline=off`
- `line_role_pipeline=codex-line-role-v1`

That means those baselines were AI-assisted, not rules-only.

### 3. The retained exhaustive run was not a universal cache miss
Across retained `all_method_benchmark_report.json` files under the March 4 exhaustive QualitySuite run family:

- `377` variants were represented
- `137` prediction runs executed
- `240` predictions were reused cross-run

So the earlier claim that every variant forced a fresh LLM run is not supported.

### 4. The line-role path can amplify spend
The retained code path confirms:

- retry attempts: `3`
- default max inflight: `4`

That makes accidental activity more expensive when line-role Codex is enabled in the wrong place.

### 5. Reuse keys are broad enough to create avoidable misses
Two verified design facts matter here:

- the line-role cache key includes `settings.stable_hash()`
- the all-method reuse key includes nearly the full `run_config`, excluding only `benchmark_sequence_matcher`

So non-LLM setting changes can invalidate reuse and trigger extra work.

## What The Token Evidence Actually Supports
### Repo-backed lower bound
The contaminated March 4 QualitySuite runs do not preserve token totals in `.history/performance_history.csv`; those rows are blank in the token columns.

However, all `49` retained hidden hybrid QualitySuite rows preserve their line-role prompt and response files under `prediction-run/line-role-pipeline/prompts/`.

Summed across those `49` retained runs:

- prompt bytes: `4,856,985`
- response bytes: `105,960`
- combined bytes: `4,962,945`

Using those saved prompts as the direct local evidence:

- hard prompt-text estimate: about `1.56M` tokens
- more realistic billing-truth estimate: about `2.0M` to `2.2M` tokens
- plausible upper band for the retained contaminated sweep: about `2.5M` tokens

### Why the billing-truth estimate is higher than the prompt-text estimate
Prompt files are not guaranteed to capture every token counted by the CLI or billing meter. The repo can undercount when:

- wrapper or system-message scaffolding is not preserved in `prompt_*.txt`
- retries or duplicate attempts are only partially visible
- cached-input accounting differs from raw saved-text size
- deleted runs never made it into retained artifacts

So `~1.56M` is too strict as a final answer. It is better treated as a floor-ish retained-artifact estimate.

## What This Report Does Not Support
The following claims should not be carried forward:

- exact incident totals such as `10M input`, `250M cache-read`, or a precise dollar figure
- the claim that every configuration in the exhaustive sweep missed cache
- the claim that official single-offline `/vanilla` examples were secretly using line-role Codex
- the claim that all dashboard vanilla metrics are contaminated

Those were either overstatements or depended on external telemetry that is not retained locally.

## Practical Reading Rule
When reading this incident:

- treat the March 4 QualitySuite baseline issue as a real hidden AI-on bug,
- treat the retained token estimate as low-single-digit millions, not tens of millions,
- keep separate the hidden baseline waste from the much larger intentional Codex benchmark usage elsewhere in the repo.

## Final Verdict
The real problem was:

- hidden line-role Codex inside supposed deterministic QualitySuite baselines,
- weak gating that let this happen,
- and reuse-key design that allowed avoidable extra executions.

The best current wording is:

- **benchmark integrity impact:** severe
- **retained spend impact:** moderate, roughly `2M` tokens
- **locally provable exact billing total:** unavailable
