# Codebase Legacy & Redundancy Audit (April 2026)

## 1. Executive Summary
The `recipeimport` codebase is currently in a transitional state following a significant architectural shift toward a "Phase 2" label-first authority model. While a major cleanup was performed in March/April 2026, several layers of legacy debt, "configuration theater," and abandoned prototypes remain. This report categorizes these findings into Core Domain Debt, Benchmark Redundancies, Configuration Theater, and Filesystem Litter.

---

## 2. Core Domain Debt: Reporting & Provenance

### 2.1. The `ReportBuilder` Fossil
The most prominent piece of legacy infrastructure is the `ReportBuilder` class in `cookimport/core/reporting.py`. 
- **Status:** Effectively dead. 
- **Rationale:** Architecture logs (`docs/01-architecture/01-architecture_README.md`) confirm that stage-owned reports are now built using the `ConversionReport` model directly, recomputing aggregate counts from final `ConversionResult` objects rather than using the stateful `ReportBuilder` accumulator.
- **Related Junk:** 
    - `tests/core/test_phase1_manual.py`: The `test_reporting` function exercises this legacy class and provides no value to the current Phase 2 pipeline.
    - `cookimport/core/reporting.py`: The `enrich_report_with_stats` function is similarly bypass-heavy and overlaps with modern staging logic.

### 2.2. Legacy ID Schemes
Traces of `urn:cookimport:freeform_span` remain in the documentation and some diagnostic strings. The system has standardized on `urn:recipeimport:...`. While the code handles the transition, the legacy constants and mention of "numbered non-recipe lanes" are obsolete.

## 3. Benchmark & Evaluation Junk

### 3.1. Multilayer Sequence Matcher
The file `cookimport/bench/sequence_matcher_multilayer.py` contains a sophisticated, custom implementation of the `SequenceMatcher` algorithm.
- **Status:** Archived / Useless.
- **Rationale:** `cookimport/bench/sequence_matcher_select.py` hard-restricts the system to a single supported mode: `"dmp"` (Google's Diff-Match-Patch). The multilayer implementation is preserved "just in case" but is never invoked by the production or benchmark pipelines.

### 3.2. Deprecated Alignment Paths
In `cookimport/bench/eval_canonical_text.py`, the function `_align_prediction_blocks_fast` is marked as deprecated.
- **Status:** Dead Path.
- **Rationale:** The system now enforces a "global" alignment strategy for canonical text evaluation to ensure accuracy. The "fast" path is bypassed by a boolean check that is currently hard-wired to prefer global alignment.

---

## 4. Configuration Theater
A significant portion of the `RunSettings` surface area (`cookimport/config/run_settings.py`) constitutes "Configuration Theater"—settings that appear tunable in the UI or CLI but are overridden by hardcoded patches in the backend.

### 4.1. The `CodexDecision` Overrides
The following settings are effectively dead due to overrides in `cookimport/config/codex_decision.py`:
- `benchmark_sequence_matcher`: Locked to `dmp`.
- `instruction_step_segmentation_policy`: Locked to `always`.
- `instruction_step_segmenter`: Locked to `heuristic_v1`.
- `section_detector_backend`: Locked to `shared_v1`.
- `multi_recipe_trace`: Locked to `False`.
- `p6_emit_metadata_debug`: Locked to `False`.

### 4.2. Unsupported/Retired Settings
- `table_extraction`: The `cookimport/config/CONVENTIONS.md` and related tests confirm that this setting is no longer supported and will throw a `ValueError` if an attempt is made to toggle it to anything other than `"off"`.

## 5. Filesystem "Litter" & Abandoned Prototypes

### 5.1. Root-Level Diagnostic Scripts
Several scripts exist in the root directory that are not part of the `cookimport` package and appear to be one-off analysis tools from previous refactors:
- `analyze_activity.py`: Converts `var/codex_exec_activity.csv` to a line-item report.
- `analyze_all_activity.py`: A version of the above that includes legacy CSVs from March 2026.
- `analyze_leaks.py`: Maps token usage leaks to reports.
- `map_tokens.py`: Token usage mapping utility.

### 5.2. Tooling Junk
- `tools/proto_split.py`: A regex-based text splitting prototype. This logic has been entirely superseded by `cookimport/parsing/chunks.py` and `cookimport/parsing/atoms.py`.
- `var/codex_exec_activity.legacy-2026-03-04_01.48.08.csv`: Old data file that should reside in an archive folder or be deleted.

### 5.3. Empty Modules
- `cookimport/tagging/`: This directory is empty. The standalone tagging pipeline was deleted during the Phase 2 cleanup, but the directory remains.

---

## 6. Policy-Locked (Dormant) Features
These features are implemented and functional but are currently disabled by project policy. While not "useless," they represent code that is not being exercised in the standard "golden path."
- **Alternative EPUB Extractors:** `markdown` and `markitdown` extractors in `cookimport/parsing/epub_extractors.py` are fully functional but locked behind environment variable checks in `cookimport/epub_extractor_names.py`.

---

## 7. Recommendations for Pruning
1.  **Delete `cookimport/core/reporting.py` (`ReportBuilder`)** and migrate any remaining helper methods to `ConversionReport`.
2.  **Remove root-level `analyze_*.py` scripts** or move them to a dedicated `archive/scripts/` directory if historical context is required.
3.  **Prune `RunSettings`**: Remove the "Theater" settings from the public Pydantic model and move them to internal constants within `CodexDecision`.
4.  **Delete `tools/proto_split.py`**: It no longer reflects the system's parsing strategy.
5.  **Remove `cookimport/bench/sequence_matcher_multilayer.py`**: The `dmp` implementation is the established standard for this project.
