## Cross-Shard Patterns

The five shards do not point to one single remaining bug. They point to three distinct failure families:

1. Tool-use drift
- Shards `0001` and `0004` attempted forbidden tool use on the initial pass.
- These were real command executions, not false positives: one ran `ls`, the other ran `pwd`.

2. Empty-output drift
- Shards `0002`, `0003`, and `0005` returned `{"rows":[]}` initially.
- These initial empty outputs completed normally without watchdog intervention, so the watchdog did not cause them.

3. Degenerate or stalled recovery
- Shard `0001` retry collapsed to all `OTHER`.
- Shards `0003`, `0004`, and `0005` stalled into reasoning without final JSON during repair or retry.
- The stalled retries were not short. They ran for about `94 s`, `282 s`, and `132 s` respectively while emitting only planning text.
- Only shard `0002` actually recovered into a full repaired payload.

The common thread is not just label confusion. It is mode confusion:
- some shards drift into command-execution behavior
- some shards emit an empty placeholder object instead of doing the work
- some retries and repairs stay stuck in planner narration instead of switching into final JSON emission

The likely phase-specific cause is now clearer:
- Initial line-role prompts are under-grounded. They point to a worker-local task file path, but do not inline the shard rows. That appears to invite one of two bad outcomes:
  - agent-like models try to inspect the workspace with `ls` or `pwd`
  - stricter models return a shape-valid placeholder such as `{"rows":[]}` without actually doing grounded labeling
- Repair prompts are much more grounded because they inline the authoritative rows, but they also become very large and demanding. On the failed repairs, the model appears to react by narrating a heuristic plan for hundreds of rows instead of committing to the final JSON object.

This is why the benchmark only improved mildly:
- The new guard successfully prevented a catastrophic promoted collapse.
- But 4 of 5 line-role shards still failed to deliver a strong grounded Codex answer.
- That means the run was dominated by fallback and one repaired shard instead of five healthy model shards.

## Working Conclusion

The new line-role hardening fixed the catastrophic promotion bug, but this run shows that the line-role stage still has a deeper production-quality problem: most shards are not reliably producing usable strict JSON answers on the first pass or on recovery. The watchdog is not the primary explanation for the failures. In this run it mostly caught real tool use or long no-output reasoning loops after the model had already gone off-contract. The sharper diagnosis is:
- initial attempts are too dependent on a file-path contract that some agents do not honor reliably without tools
- repair attempts are grounded, but large enough that some agents switch into planner narration instead of answer emission

The next investigation should therefore focus less on evaluation and more on the line-role prompt/runtime contract for initial attempts, retries, and repairs, especially how to keep the model out of tool-using and planner-only modes while still giving it the shard content directly enough to avoid empty placeholder outputs.
