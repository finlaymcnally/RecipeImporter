# Benchmark vs live import delta audit

This report is diagnostic only.

Goal of this audit:
- map where normal live import/stage and benchmark prediction/eval currently diverge,
- identify which deltas are semantic versus merely artifact/layout differences,
- explain why benchmark results can differ from what a real import run would produce.

## Bottom line

Today the benchmark is not simply “score a live import run.”

It shares the importer conversion prefix, but after that it can branch into a separate prediction/eval pipeline with its own:
- candidate building,
- line-role classification,
- projection back onto recipes,
- knowledge-stage merge seam,
- scored artifact surface,
- eval diagnostics and bundle layout.

That means benchmark quality is partly measuring a parallel system, not just the live import output.

## 1. Shared prefix

These parts are genuinely shared between live import/stage and benchmark prediction generation:

- importer selection and source conversion
- split-job planning and conversion
- optional recipe CodexFarm pass (`llm_recipe_pipeline`)

- processed cookbook output writing code path

## 2. Major semantic deltas

### 2.1 Benchmark can run a separate canonical line-role pipeline; live import does not

Normal stage/import does not run canonical line-role labeling.

Benchmark prediction generation can run:
- `atomic_block_splitter`
- `line_role_pipeline=deterministic-v1|codex-line-role-v1`

That is a material behavioral fork, not just extra telemetry.

Code anchors:
- [ingest.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest.py#L2595)
- [07-bench_README.md](/home/mcnal/projects/recipeimport/docs/07-bench/07-bench_README.md)

### 2.2 Benchmark line-role candidate building uses a different ingredient/instruction detector

This is the most important parser delta.

Live recipe conversion uses the main parsing stack:
- `ingredients.py::parse_ingredient_line(...)`
- `instruction_parser.py::parse_instruction(...)`

Benchmark canonical line-role does not use those to decide per-line roles.
Instead it builds `AtomicLineCandidate` rows from extracted archive blocks and labels them with the lighter heuristics in `recipe_block_atomizer.py`.

So yes: benchmark line-role currently uses a weaker ingredient detector than the real import path.

Code anchors:
- live ingredient parser call: [draft_v1.py](/home/mcnal/projects/recipeimport/cookimport/staging/draft_v1.py#L318)
- benchmark candidate builder: [ingest.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest.py#L1015)
- benchmark ingredient heuristic: [recipe_block_atomizer.py](/home/mcnal/projects/recipeimport/cookimport/parsing/recipe_block_atomizer.py#L434)

Observed consequence:
- obvious ingredient rows such as `1 large jalapeño...`, `Optional: 1-ounce chunk of Parmesan`, and `1-pound loaf...` can miss `ingredient_like` in benchmark line-role even though the real `ingredient-parser-nlp` path would likely handle them.

### 2.3 Benchmark line-role candidates are built from extracted archive blocks, not final staged recipe fields

Live import writes drafts from the final `ConversionResult` recipes after recipe Codex and parsing/staging logic have settled.

Benchmark canonical line-role starts earlier from the extracted archive text stream and inferred recipe block ranges.

That means canonical benchmark classification is anchored to archive-derived atomic lines, not to the final shaped recipe objects.

Code anchors:
- [ingest.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest.py#L1015)
- [recipe_block_atomizer.py](/home/mcnal/projects/recipeimport/cookimport/parsing/recipe_block_atomizer.py#L137)

### 2.4 Benchmark line-role candidates can have stale recipe spans

In current prediction generation order, canonical line-role candidates are built before `run_codex_farm_recipe_pipeline(...)` updates the `ConversionResult`.

So even when the recipe pipeline fixes recipe boundaries, line-role can still classify rows using the older provenance ranges.

This is a direct benchmark-vs-live-import mismatch, because live stage writing happens after the recipe pipeline and uses the updated `ConversionResult`.

- recipe line-role runs later: [ingest.py](/home/mcnal/projects/recipeimport/cookimport/labelstudio/ingest.py#L2595)
- staging after recipe Codex in live stage: [cli_worker.py](/home/mcnal/projects/recipeimport/cookimport/cli_worker.py#L298)

### 2.5 Codex line-role is constrained by deterministic allowlists

Benchmark Codex line-role is not free to choose any label.

Per-row `candidate_labels` are passed into the prompt, and the response loader rejects labels outside that allowlist.

So if deterministic atomizer heuristics miss an ingredient line and collapse the allowlist to `["OTHER"]`, Codex cannot rescue it to `INGREDIENT_LINE`.

Code anchors:
- prompt rule: [canonical_line_role_prompt.py](/home/mcnal/projects/recipeimport/cookimport/llm/canonical_line_role_prompt.py#L75)
- allowlist construction: [canonical_line_roles.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles.py#L1567)
- response validation: [canonical_line_roles.py](/home/mcnal/projects/recipeimport/cookimport/parsing/canonical_line_roles.py#L2416)

Observed consequence:
- some benchmark Codex errors are actually deterministic pre-filter errors, not model judgment errors.

### 2.6 Benchmark can mutate cookbook outputs from line-role projection before writing processed outputs

This is a major semantic difference.

When `line_role_pipeline != off`, benchmark prediction generation can:
1. classify atomic lines,
2. project those labels back into recipe fields,
3. mutate the `ConversionResult`,
4. then write processed cookbook outputs from that mutated result.

Normal live stage/import does not do this line-role projection mutation step.

### 2.7 Benchmark canonical scoring can prefer projection artifacts over live stage evidence

Live stage/import writes `.bench/<workbook_slug>/stage_block_predictions.json` from final recipe outputs and archive blocks.

Canonical benchmark, when line-role is enabled, prefers:
- `prediction-run/line-role-pipeline/stage_block_predictions.json`
- `prediction-run/line-role-pipeline/extracted_archive.json`

That is a different evidence surface from the live `.bench` stage artifact.

The line-role projection writer emits `stage_block_predictions.v1`, but in that artifact the “block” identity is really canonical atomic line index.

So canonical benchmark scoring can use artifacts whose coordinate semantics differ from the original extracted block evidence used by live stage `.bench`.

Live stage evidence:

Benchmark canonical mode:
- knowledge-stage merges on top of line-role projection artifacts,
- this is primarily an outside-span `KNOWLEDGE` versus `OTHER` seam.

## 4. Diagnosis of the ingredient parser mismatch

The specific user-observed mismatch is real:

- live import recipe conversion uses the strong external `ingredient-parser-nlp` path via `parse_ingredient_line(...)`;
- benchmark canonical line-role uses `recipe_block_atomizer.py::_is_ingredient_line(...)` to decide whether a row is even eligible for `INGREDIENT_LINE`.

That atomizer heuristic currently misses at least:
- Unicode ingredient names that upset its token-count logic,
- `Optional:` prefixed ingredient lines,
- hyphenated quantity forms like `1-ounce` and `1-pound`.

When that happens:
- candidate labels fall back to `["OTHER", "KNOWLEDGE"]`,
- inside recipe spans, `KNOWLEDGE` may be stripped,
- the effective allowlist becomes `["OTHER"]`,
- Codex is forced to answer `OTHER`.

This is not just “benchmark uses a worse parser.”
It is:
- benchmark uses a different parser/detector,
- that detector runs before Codex,
- and its allowlist can prevent Codex from recovering the correct label.

## 5. Deltas that are merely surface/layout, not core semantic divergence

These matter less for unification:

- benchmark-specific CSV/dashboard append timing
- compare-control and upload-bundle artifacts
- benchmark-only markdown summaries
- prediction record replay helpers
- benchmark compare roots and paired variant folders

Those are important operationally, but they are not the reason benchmark quality can disagree with live import behavior.

## 6. Highest-signal conclusion

The current benchmark path differs from live import in three especially important ways:

1. It can classify canonical lines with a separate line-role system that live import never runs.
2. That line-role system uses lighter atomizer heuristics instead of the main ingredient/instruction parsers.
3. It can write processed cookbook outputs after mutating recipes from projected line-role labels, so benchmark-produced “processed outputs” are not guaranteed to match a real live import.

If the intended product meaning is “benchmark should score a live import run,” these three deltas are the central diagnosis.
