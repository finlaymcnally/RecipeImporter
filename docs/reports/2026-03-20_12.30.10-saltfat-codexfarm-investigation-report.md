# Salt Fat CodexFarm Investigation Report

## Scope

This report covers the issues found during investigation of the Salt Fat Acid Heat CodexFarm benchmark runs rooted at:

- `data/golden/benchmark-vs-golden/2026-03-20_11.00.44`
- `data/golden/benchmark-vs-golden/2026-03-20_11.16.53`

The investigation included:

- the earlier line-role crash on missing `repair_prompt.txt` for split task packets,
- the later non-recipe knowledge stage that appeared stuck at `task 0/10 | running 10`,
- the benchmark UI behavior during that stall,
- the stop behavior when trying to interrupt the benchmark.

## Executive Summary

The run did not have one single blocker. It had one already-fixed line-role filesystem bug plus several still-open knowledge/runtime/product issues.

The most important current problem is that the knowledge stage is not merely "slow." It is entering a retry storm where half of the workers complete and half are repeatedly watchdog-killed for shell command patterns that the current policy still classifies as forbidden. The benchmark UI then presents that state badly: it keeps showing `task 0/10`, reports `active tasks (10/8, 10 left)`, and only renders eight worker rows even when ten workers are configured. When the parent benchmark process is interrupted, the run preserves partial worker artifacts but does not finalize a partial CodexFarm benchmark result package.

## Findings

### 1. Fixed: split-task line-role repair could crash before the repair attempt even launched

Severity: high

Status: fixed during this investigation

What happened:

- The line-role runtime now repairs at task scope, but the old code only created parent shard directories.
- A split task such as `line-role-canonical-0001-a000000-a000176.task-001` could therefore fail immediately when writing `repair_prompt.txt`.

Evidence:

- Runtime task root creation now lives at `cookimport/parsing/canonical_line_roles.py:2037-2053`.
- Repair prompt directory creation now lives at `cookimport/parsing/canonical_line_roles.py:3467-3474`.
- The new regression test is `tests/parsing/test_canonical_line_roles.py:2500-2561`.

Impact:

- This was the direct cause of the earlier CodexFarm crash path:
  - `.../workers/worker-001/shards/line-role-canonical-0001-a000000-a000176.task-001/repair_prompt.txt`

Resolution:

- The runtime now creates task-scoped shard directories before writing repair/watchdog follow-up artifacts.

### 2. Open: the knowledge stage is retry-looping, not simply waiting on long workers

Severity: high

Status: open

What happened:

- The knowledge stage split into two groups:
  - completed workers,
  - watchdog-killed workers that kept being retried.

Observed state from the interrupted run:

- `5` workers ended in `completed`
- `4` of those explicitly report `workspace_outputs_stabilized`
- `5` workers ended in `watchdog_killed` with `watchdog_command_execution_forbidden`
- the run had already written `366` knowledge `out/*.json` task outputs
- the run had already written `163` `watchdog_retry/status.json` files

Artifact evidence:

- Worker live statuses:
  - `data/output/2026-03-20_11.16.53/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-20_11.17.46/raw/llm/saltfatacidheatcutdown/knowledge/workers/*/live_status.json`
- Example completed worker:
  - `worker-004/live_status.json` reports `state=completed` and `reason_code=workspace_outputs_stabilized`
- Example killed workers:
  - `worker-001/live_status.json`
  - `worker-009/live_status.json`
  - `worker-010/live_status.json`

Code evidence:

- Knowledge worker completion on stabilized outputs:
  - `cookimport/llm/codex_farm_knowledge_orchestrator.py:3274-3287`
- Knowledge worker kill on forbidden command execution:
  - `cookimport/llm/codex_farm_knowledge_orchestrator.py:3288-3300`

Impact:

- The benchmark keeps respawning knowledge workers instead of converging quickly to a finished phase.
- Operators experience this as a hang even though significant work has already happened.

### 3. Open: the current knowledge watchdog policy is still too hostile to the command patterns these workers actually use

Severity: high

Status: open

What happened:

- The workers are still issuing shell commands that are local-workspace-oriented but not currently tolerated by the knowledge workspace watchdog.
- The captured `reason_detail` values show command shapes such as:
  - heredoc writes into `out/*.json`
  - `jq` against `assigned_tasks.json`
  - `cat` of worker hint files

This is not the old "repo/network/interpreter" drift problem. It is the current bounded worker workflow still colliding with the command policy.

Artifact evidence:

- `worker-001/live_status.json` reason detail begins with:
  - `workspace worker stage attempted tool use: /bin/bash -lc "cat <<'EOF' > out/...`
- `worker-009/live_status.json` reason detail begins with:
  - `workspace worker stage attempted tool use: /bin/bash -lc "jq -r '.[].task_id' assigned_tasks.json ...`
- `worker-010/live_status.json` reason detail begins with:
  - `workspace worker stage attempted tool use: /bin/bash -lc "cat hints/...`

Why this matters:

- The stage already knows how to accept workers that simply write all owned outputs and stabilize.
- The policy mismatch is what keeps a large fraction of the knowledge workers from reaching that benign completion path.

### 4. Open: benchmark progress UI materially misrepresented the knowledge stage state

Severity: medium-high

Status: open

Symptoms seen in the UI:

- `task 0/10 | running 10`
- `active tasks (10/8, 10 left)`
- only eight worker rows shown even though ten workers were configured

Why the UI was misleading:

- The knowledge runtime initially reports one active shard per assignment:
  - `cookimport/llm/codex_farm_knowledge_orchestrator.py:1155-1167`
- The CLI worker panel hard-caps rendered worker rows at `8`:
  - `cookimport/cli.py:10386-10397`
- The worker panel label therefore becomes `active tasks (10/8, 10 left)`:
  - `cookimport/cli.py:10410-10424`

Impact:

- Two configured workers are hidden from the dashboard by design.
- A retry storm looks like "ten stuck workers" instead of "some workers finished, some are being retried".
- The UI gave no clear indication that several workers had already completed with `workspace_outputs_stabilized`.

### 5. Open: completion metadata is inconsistent for completed knowledge workers

Severity: medium

Status: open

What happened:

- `worker-008/live_status.json` ended as:
  - `state=completed`
  - `reason_code=null`
  - `reason_detail=null`
  - `final_agent_message_state=malformed`
- In contrast, other cleanly completed workers recorded:
  - `reason_code=workspace_outputs_stabilized`

Impact:

- Completed worker outcomes are not categorized consistently.
- Reviewers cannot tell whether a `completed` worker ended because outputs stabilized, because the child exited normally, or because another completion path was taken that should have been surfaced explicitly.

Evidence:

- `worker-008/live_status.json`
- `worker-004/live_status.json`

### 6. Open: interrupting the benchmark preserved partial artifacts but did not finalize a partial CodexFarm result

Severity: medium

Status: open

Observed behavior:

- Sending `SIGINT` to the parent benchmark process did not unwind it.
- Sending `SIGTERM` to the parent benchmark process did stop it.
- After termination, the parent process was gone and all child Codex processes were gone, but the CodexFarm run still had no final:
  - `eval_report.json`
  - `processing_timeseries_prediction.jsonl`
  - `processing_timeseries_evaluation.jsonl`
  - `prompt_budget_summary.json`

Observed surviving top-level artifact in the interrupted CodexFarm run:

- `line-role-pipeline/telemetry_summary.json`

Impact:

- Operators can preserve the partial worker trees, but they do not get a usable partial benchmark summary or comparison report.
- This is especially painful in long CodexFarm runs where the user may reasonably want to stop a broken stage and inspect everything else.

### 7. Open: graceful cancellation of the parent benchmark process is unreliable

Severity: medium

Status: open

Observed behavior:

- `SIGINT` to the parent `C3imp` benchmark process did not stop it.
- The process kept respawning new knowledge worker sessions after the signal.
- `SIGTERM` was required to stop the parent.

Why this matters:

- For long interactive benchmark runs, the expected operator action is effectively "Ctrl+C".
- In this case that path did not unwind the benchmark cleanly enough to stop the stage.

This report does not pin one specific source line for the signal issue; the finding is primarily empirical from the live run behavior.

### 8. Small but real: the docs promise more accurate `running N` behavior than the observed UI currently delivered

Severity: low

Status: open

Docs state:

- `docs/07-bench/07-bench_README.md` says benchmark status panels now treat `task X/Y | running N` as first-class worker activity and that knowledge worker status files may end with explicit runtime reason codes.

Observed reality in this run:

- the knowledge stage UI still looked stale and operator-hostile,
- it hid workers `09` and `10`,
- it showed no visible distinction between completed, retried, and killed workers.

This is not a documentation bug. It is a product gap between the intended operator experience and what the current CLI rendering actually communicates under retry-heavy knowledge runs.

## Recommendations

1. Treat the knowledge workspace command-policy mismatch as the primary open blocker.
   The biggest remaining problem is not token cost or prompt wording. It is that a substantial fraction of knowledge workers still hit `watchdog_command_execution_forbidden` for command shapes the workspace-worker flow appears to want to use.

2. Fix the CLI worker panel before the next long operator loop.
   The hard-coded `display_slots = 8` is actively misleading once configured workers exceed eight.

3. Add an interrupted-run finalization path.
   Even if the benchmark is stopped mid-stage, the product should still write a partial result summary and clearly mark it incomplete instead of leaving only raw worker artifacts.

4. Normalize completed worker reason codes.
   `completed` should always carry an explicit reason category so operators can distinguish "stabilized outputs" from "exited after malformed final message" and other benign or suspicious stop modes.

5. Keep the new split-task line-role repair regression test in the routine suite.
   That bug was real and severe, but it is now covered.
