# Beads Shared-Server And Remote Policy

This report captures the current Beads setup policy for this machine and the recovery path used in `recipeimport`.

## Default policy

- Prefer `bd init --server --shared-server --non-interactive` for new repos.
- For solo/local repos, prefer a local filesystem Dolt remote even when the code repo already has a normal git remote.
- Use a repo-local path by default, for example `file://$REPO_ROOT/.beads-remote`.
- Treat off-machine backup as a separate concern from the day-to-day Beads remote.
- If a repo already has Beads history on its remote, use `bd bootstrap` instead of a fresh `bd init`.

Why:

- Embedded mode (`bd context --json` shows `dolt_mode: "embedded"`) is prone to exclusive-lock errors when agents overlap `bd` commands.
- Shared-server mode removes that bottleneck and lets one local Dolt server back multiple repos.
- A local filesystem remote is simpler and more reliable than using the code repo's git remote as a Beads/Dolt remote.
- Normal git push already protects the code. Beads only needs a stable place to replicate its own database.

## New repo setup

For the normal solo setup:

```bash
bd init --server --shared-server --non-interactive
mkdir -p .beads-remote
bd dolt remote add origin "file://$PWD/.beads-remote"
bd setup codex
bd context --json
bd dolt remote list --json
bd dolt push
```

`bd setup codex` does not replace `bd init`. Run `bd init` first so the Beads database exists, then run `bd setup codex` to install the Codex-facing instructions and integration files for that repo.

If you also want off-machine recovery, add a backup destination separately:

```bash
bd backup init <dolt-backup-url>
bd backup sync
```

## Existing repo with Beads history

If the repo already has Beads history on its remote, do not run a fresh `bd init` in a new clone or worktree.

Use:

```bash
bd bootstrap
bd setup codex
```

That clones the existing Beads database and avoids creating divergent Dolt histories. After bootstrap, `bd setup codex` is still the right follow-up if the repo needs the Codex integration files refreshed or installed.

## Embedded-to-server migration

Safe migration pattern used in `recipeimport`:

1. Make a filesystem copy of `.beads`.
2. Export the current data: `bd export --all -o <backup.jsonl>`.
3. Move the old `.beads` aside.
4. Reinitialize in shared-server mode: `bd init --server --shared-server --non-interactive -p <repo>`.
5. Import the saved data: `bd import <backup.jsonl>`.
6. Verify counts and spot-check with `bd context --json`, `bd count --json`, and `bd show <id> --json`.

In `recipeimport`, the original GitHub-backed Dolt remote later developed a missing-blob failure that even `bd dolt push --force` could not repair. The fix was to stop using that remote and move the repo back to a local filesystem remote.

## Divergent history recovery

If `bd dolt push` fails with `no common ancestor`, the local and remote Beads histories diverged.

Choose one authority source deliberately:

- Remote is authoritative: `bd bootstrap`
- Local is authoritative: `bd dolt push --force`

Do not repeatedly re-run `bd init` against the same remote. That is the usual source of divergence.

## Broken remote recovery

If `bd dolt push` fails because the remote is missing Dolt blobs or cannot open the remote database, treat the local Beads database as authoritative and switch to a fresh remote instead of trying to repair the broken one in place.

Recommended sequence:

```bash
bd export --all -o .beads/manual-backups/<timestamp>_before-remote-repair.jsonl
bd dolt remote remove origin
mkdir -p .beads-remote
bd dolt remote add origin "file://$PWD/.beads-remote"
bd dolt push
```

This keeps Beads stable for day-to-day use. If off-machine backup matters, configure `bd backup` separately after the filesystem remote is healthy.

## Operational notes for agents

- Shared-server mode fixes the embedded exclusive-lock problem, but agents should still avoid issuing unnecessary overlapping `bd` remote operations.
- Treat `bd bootstrap` as the normal entrypoint for existing repos with Beads state, not as a rare recovery-only command.
- For this solo project, prefer `bd dolt start` if the local Dolt server is down after a crash, then continue normally.
- For this repo, the recommended steady-state is shared-server mode plus a repo-local filesystem remote.
