# Beads Shared-Server And Remote Policy

This report captures the current Beads setup policy for this machine and the recovery path used in `recipeimport`.

## Default policy

- Prefer `bd init --server --shared-server --non-interactive` for new repos.

bd setup codex

- If a repo has a normal git remote, Beads should use that repo's remote as its Dolt remote.
- If a repo has no git remote, Beads should use a local filesystem remote instead, for example `file://$HOME/.local/share/beads-remotes/<repo>`.
- If a repo already has Beads history on its remote, use `bd bootstrap` instead of a fresh `bd init`.

Why:

- Embedded mode (`bd context --json` shows `dolt_mode: "embedded"`) is prone to exclusive-lock errors when agents overlap `bd` commands.
- Shared-server mode removes that bottleneck and lets one local Dolt server back multiple repos.
- Repo-backed Dolt remotes keep Beads history attached to the project when the project already has a remote.

## New repo setup

When the repo has a git remote and no existing Beads history:

```bash
bd init --server --shared-server --non-interactive
bd context --json
bd dolt remote list --json
bd dolt push
```

When the repo has no git remote:

```bash
bd init --server --shared-server --non-interactive
bd dolt remote add origin file://$HOME/.local/share/beads-remotes/<repo>
bd dolt push
```

## Existing repo with Beads history

If the repo already has Beads history on its remote, do not run a fresh `bd init` in a new clone or worktree.

Use:

```bash
bd bootstrap
```

That clones the existing Beads database and avoids creating divergent Dolt histories.

## Embedded-to-server migration

Safe migration pattern used in `recipeimport`:

1. Make a filesystem copy of `.beads`.
2. Export the current data: `bd export --all -o <backup.jsonl>`.
3. Move the old `.beads` aside.
4. Reinitialize in shared-server mode: `bd init --server --shared-server --non-interactive -p <repo>`.
5. Import the saved data: `bd import <backup.jsonl>`.
6. Verify counts and spot-check with `bd context --json`, `bd count --json`, and `bd show <id> --json`.

In `recipeimport`, the local authoritative history was then force-pushed once to repair the GitHub-backed Dolt remote and re-establish a clean shared history.

## Divergent history recovery

If `bd dolt push` fails with `no common ancestor`, the local and remote Beads histories diverged.

Choose one authority source deliberately:

- Remote is authoritative: `bd bootstrap`
- Local is authoritative: `bd dolt push --force`

Do not repeatedly re-run `bd init` against the same remote. That is the usual source of divergence.

## Operational notes for agents

- Shared-server mode fixes the embedded exclusive-lock problem, but agents should still avoid issuing unnecessary overlapping `bd` remote operations.
- Treat `bd bootstrap` as the normal entrypoint for existing repos with Beads state, not as a rare recovery-only command.
- If a repo has no code remote, a local filesystem Dolt remote is acceptable and keeps Beads state persistent on the machine without inventing a hosted destination.
