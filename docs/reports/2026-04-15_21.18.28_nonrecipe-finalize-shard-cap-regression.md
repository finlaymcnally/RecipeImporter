Summary: the remaining Salt Fat `KNOWLEDGE -> OTHER` drift was a nonrecipe-finalize packetization regression, not a line-role exclude regression.

What changed:
- `build_knowledge_jobs(...)` used to treat `knowledge_prompt_target_count` as an exact final shard count whenever it was set.
- That path discarded the budget-native packetization and repartitioned the full ordered row stream into exactly `N` contiguous shards.
- On the Salt Fat run pair, that widened the launch count from `15` to `20` and reduced local packet context enough for several short explanatory rows to flip from `knowledge` to `other`.

Evidence kept:
- The 19 changed rows had no `nonrecipe_excluded` reason and drifted inside `nonrecipe_finalize` proposals themselves.
- Old run proposal packets labeled those rows `knowledge`; the later run's smaller packets labeled the same rows `other`.
- Prompt contract text was materially stable across the two runs. The meaningful change was packet shape and surrounding context, not classification instructions.

Fix:
- `knowledge_prompt_target_count` now stays a cap on launched shards, not an exact target that must be reached by fragmenting budget-safe packets.
- The planner now preserves budget-native packet boundaries whenever they already fit under the requested cap.
- If the requested cap is lower than the budget-native packet count, the planner still merges adjacent native packets and records the existing warning about forced oversized/non-local bundles.

Why this is the right seam:
- It preserves the current operator model: requested count still affects the maximum number of launched shards.
- It uses the already-existing phase-plan vocabulary (`requested`, `budget_native`, `launch`) honestly instead of pretending those counts must match.
- It removes the specific fragmentation behavior that caused the semantic drift without giving deterministic code semantic authority over `knowledge|other`.
