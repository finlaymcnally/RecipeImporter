**Scope:** All 15 failed `nonrecipe_finalize` knowledge shards

## Plain-English Summary
The first pass is mostly doing the new job: it is deciding which rows are worth keeping for review.

The second pass is still the problem. It is supposed to do one clean thing:

1. group related kept rows together
2. give each group one shared topic
3. give that group one shared tag grounding story

Instead, the model is still doing one or more of these:

- returning the wrong number of labels
- keeping a row but forgetting to give it a group
- mixing several different ideas inside one group story
- giving a group a tag story that does not really fit all the rows inside it

In simple terms: the model is still trying to "wing it" across big messy packets instead of drawing clean group boundaries.

## Overall Pattern
If you zoom out, the 15 failures are not random.

They mostly fall into three plain problems:

1. The model sometimes misses required structure.
   Example: `ks0002`, `ks0001`

2. The model often sees too many "valid ideas" in one shard and tries to keep all of them.
   Example: `ks0003`, `ks0005`, `ks0007`, `ks0008`, `ks0009`, `ks0013`

3. Even when the groups look reasonable at first glance, the shared group story is still too broad or too generic.
   Example: `ks0000`, `ks0004`, `ks0006`, `ks0010`, `ks0014`

The practical takeaway is simple:
the current pass-2 prompt/repair behavior still does not force the model to choose sharp group boundaries. It is still behaving like a summarizer that wants to preserve every nearby thought, instead of a strict packet editor that must produce one clean grouped contract.
