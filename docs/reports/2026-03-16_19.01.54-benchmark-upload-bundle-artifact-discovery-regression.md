# Benchmark Upload Bundle Artifact Discovery Regression

## What Was Reviewed

This report documents the first full post-refactor review of the benchmark session rooted at:

`data/golden/benchmark-vs-golden/2026-03-16_18.10.25/single-offline-benchmark/saltfatacidheatcutdown/`

The goal was not to judge model quality. The goal was to answer whether the refactored benchmark pipeline still produced a correct, reviewer-ready `upload_bundle_v1`.

## Conclusion

The core run completed successfully, but the reviewer handoff did not. The `vanilla` and `codexfarm` runs both finished, produced eval reports, wrote processed output, and generated a bundle whose self-check passes. However, `upload_bundle_v1` under-reports prompt and knowledge artifacts for the `codexfarm` run, so the bundle is not faithfully describing what the run actually produced.

This is not a total failure. It is a reporting and artifact-discovery regression in the existing-output bundle builder.

## Evidence That The Core Run Worked

The session has the expected paired single-offline shape: `vanilla/`, `codexfarm/`, `codex_vs_vanilla_comparison.json`, `single_offline_summary.md`, and `upload_bundle_v1/`.

The eval outputs are present and consistent:

- `vanilla/eval_report.json` reports `strict_accuracy = 0.5238744884038199` and `macro_f1_excluding_other = 0.44441642590611175`.
- `codexfarm/eval_report.json` reports `strict_accuracy = 0.5320600272851296` and `macro_f1_excluding_other = 0.460180816203609`.

The processed output run for the codex path exists at:

`data/output/2026-03-16_18.10.25/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-16_18.11.25/`

That output contains current post-refactor artifacts such as:

- `08_nonrecipe_spans.json`
- `09_knowledge_outputs.json`
- `raw/llm/saltfatacidheatcutdown/recipe_manifest.json`
- `raw/llm/saltfatacidheatcutdown/knowledge_manifest.json`
- `chunks/...`
- `final drafts/...`

Both processed import reports have empty `errors` arrays. They do have warnings, but they are normal conversion/runtime warnings rather than hard failures.

## Evidence That The Bundle Is Wrong

The bundle overview and index say the codex prompt and knowledge evidence is missing:

- `upload_bundle_overview.md` reports `full_prompt_log_status: unknown` at the topline.
- `run_diagnostics` reports for `codexfarm`:
  - `full_prompt_log_status = missing`
  - `prompt_warning_aggregate_status = missing_full_prompt_log`
  - `projection_trace_status = missing_full_prompt_log`
  - `preprocess_trace_failures_status = missing_prediction_run`
- `analysis.knowledge` reports:
  - `enabled_run_count = 1`
  - `runs_with_prompt_samples = 0`
  - `runs_with_knowledge_manifest = 0`
  - `total_knowledge_call_count = 0`
  - `prompt_budget_summary_status = missing`

But the run directory clearly contains the artifacts the bundle says are missing:

- `codexfarm/prompts/full_prompt_log.jsonl`
- `codexfarm/prompts/prompt_type_samples_from_full_prompt_log.md`
- `codexfarm/prompts/prompt_extract_knowledge_optional.txt`
- `codexfarm/prompt_budget_summary.json`

The prompt-budget file is especially important because it proves real knowledge work happened. It reports:

- `by_stage.knowledge.call_count = 56`
- `totals.call_count = 231`

The processed output also confirms real knowledge work:

- `09_knowledge_outputs.json` reports `enabled = true`
- `counts.jobs_written = 56`
- `counts.outputs_parsed = 56`
- `counts.snippets_written = 74`

The bundle therefore says “missing” where the filesystem says “written”.

## Confirmed Code-Level Seams Behind The Regression

Three discovery seams were confirmed directly in `scripts/benchmark_cutdown_for_external_ai.py`.

First, `_resolve_full_prompt_log_path(...)` does not search `run_dir/prompts/full_prompt_log.jsonl`, even though that is where this run wrote the full prompt log. It currently checks manifest artifact keys plus:

- `run_dir/full_prompt_log.jsonl`
- `run_dir/codexfarm/full_prompt_log.jsonl`

That misses the real file.

Second, `_resolve_prediction_run_dir(...)` only returns:

- `run_manifest.artifacts.pred_run_dir`, when populated
- `run_dir/prediction-run`, when present

The observed run shape leaves `artifacts.pred_run_dir` unset, so downstream helpers behave as if there is no prediction run. That feeds directly into `missing_prediction_run` bundle diagnostics.

Third, `_resolve_prompt_budget_summary_path(...)` only looks under the resolved prediction-run directory. The observed run writes `prompt_budget_summary.json` at the benchmark run root, not under `prediction-run/`, so the knowledge summary undercounts real work.

There is also one reviewer-facing locator bug. `navigation.row_locators.knowledge_by_run` uses basename fallback for `prompt_budget_summary.json`, which lets the `vanilla` row point at the `codexfarm` prompt-budget payload row. That is why `analysis.knowledge.rows[*].prompt_budget_summary_in_bundle` can be `true` even when `prompt_budget_summary_status` is `not_applicable` or `missing`. The row is being matched too loosely.

## Impact

The bundle remains structurally valid as JSONL-plus-index, and its self-check passes. A machine or person can still open it. The problem is trustworthiness, not file corruption.

External reviewers using the bundle will get a false picture of the run:

- Codex prompt provenance looks absent when it exists.
- Knowledge-stage work looks absent when it ran.
- Prompt-budget totals for the codex run are hidden.
- The non-codex run can inherit a codex-only locator through basename fallback.

That makes `upload_bundle_v1` an unreliable reviewer handoff for current post-refactor single-offline runs.

## Fix Direction

The fix should stay focused on existing-output artifact discovery and per-run locator hygiene.

The bundle builder needs to:

1. Discover prompt logs and prompt samples from the actual run-root `prompts/` directory.
2. Discover prompt-budget summaries from the run root when they are not inside `prediction-run/`.
3. Discover knowledge manifests from processed-output locations when `pred_run_dir` is absent but the run manifest already exposes processed-output roots.
4. Stop using unscoped basename fallback for per-run knowledge locators. If the bundle cannot identify the artifact for a specific run, it should leave that locator null instead of binding the wrong run’s file.

The fix does not need to change benchmark scoring, recipe extraction, or knowledge generation itself. The evidence says those parts are working.

## Required Validation After The Fix

After implementation, rebuilding the bundle for this same session should show:

- `run_diagnostics.codexfarm.full_prompt_log_status = complete`
- `prompt_warning_aggregate_status = written`
- `projection_trace_status = written`
- `preprocess_trace_failures_status` no longer reports `missing_prediction_run`
- `analysis.knowledge.total_knowledge_call_count = 56`
- `analysis.knowledge.rows[codexfarm].prompt_budget_summary_status = written`
- `analysis.knowledge.rows[codexfarm].prompt_samples_status = written`
- `analysis.knowledge.rows[codexfarm].knowledge_manifest_status = written`
- `analysis.knowledge.rows[vanilla].prompt_budget_summary_in_bundle = false`
- `navigation.row_locators.knowledge_by_run` only points each run at its own artifact paths

The relevant fix should be planned and implemented as an ExecPlan because it touches bundle discovery, diagnostics, row-locator policy, and regression tests together.
