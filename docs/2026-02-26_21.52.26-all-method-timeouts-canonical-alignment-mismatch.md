### 1.1 Where the timeout happens (phase-level)

For the timed-out configs, scheduler telemetry (`.scheduler_events/config_XXX.jsonl`) shows a consistent pattern:

- `prep_finished` and `post_finished` exist → **prediction generation finished** (conversion + stage evidence written).
- `evaluate_started` exists → canonical-text evaluation started.

So the “failures” are best read as: **canonical-text evaluation got stuck/too-slow**, not “prediction couldn’t be generated”.

## 2) Why canonical-text eval can explode in runtime

Canonical-text evaluation must align prediction text to the golden set’s canonical text. Internally it uses:

- prediction block stream: `config_XXX/.../prediction-run/extracted_archive.json` (list of blocks with `text`)
- golden canonical block map: `<golden exports>/canonical_block_map.jsonl` (per-block `text_sha256`)
- alignment algorithm: legacy global alignment using stdlib `difflib.SequenceMatcher` (slow worst-case).

### 2.1 The key accelerator: “canonical block map match”

If the prediction block stream matches the golden canonical block map **block-for-block** (same text per block index), alignment becomes cheap and eval finishes fast.

If it *doesn’t* match (different extractor / preprocess / parser version changes blockization), the evaluator can’t reuse a trivial mapping and must do expensive global alignment, which can take minutes and sometimes exceeds 900s.

## 6) Practical next steps

### 6.1 If you want fewer “failures” (operational)

- Increase `all_method_config_timeout_seconds` above 900 (e.g. 1800–3600) for sources where canonical alignment legitimately takes ~600s even when it succeeds (ex: `amatteroftastecutdown`).
- Alternatively set it to `0` (disable) if you are confident there are no true hangs; downside is a real hang can stall a source indefinitely.

### 6.2 If you want the suite to stay broad but not waste time

- Restrict permutations for a golden set to configs whose extracted blocks match its canonical blockization (high `canon_match`).
- Or treat mismatch configs as “expected slow” and give them a separate timeout tier.

### 6.3 How to confirm a timeout is “stuck in eval” in 10 seconds

1. Open the failed source dir under `data/golden/benchmark-vs-golden/2026-02-26_18.07.45/all-method-benchmark/<source_slug>/`.
2. Open `.scheduler_events/config_XXX.jsonl` for the failed config index.
3. If the last event is `evaluate_started`, it’s canonical-text eval alignment slowness (not conversion).
