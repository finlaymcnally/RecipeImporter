## 4. Scheduling Model Differences

## 4.1 `stage`

- Parallelism level: job-level process pool (`ProcessPoolExecutor`).
- Split behavior: split jobs run in parallel, then main-process merge.
- No cross-config scheduling logic (single run-level pool only).

## 4.2 `generate_pred_run_artifacts` (benchmark prediction phase)

- If split required, it runs internal split worker pool (`ProcessPoolExecutor`) and merges in-memory (`_merge_parallel_results(...)`, `cookimport/labelstudio/ingest.py:1129`).
- Optional cross-config split gate uses file locks:
  - `_acquire_split_phase_slot(...)` (`cookimport/labelstudio/ingest.py:473`)
- Emits scheduler events:
  - `split_wait_started/finished`, `split_active_started/finished`, `prep_started/finished`, `post_started/finished`.

## 4.3 All-method scheduler

Per-source scheduler controls config inflight count, split slots, wing backlog, and eval-tail cap:

- runtime resolution: `cookimport/cli.py:4427` `_resolve_all_method_scheduler_runtime(...)`
- smart mode can admit extra inflight work while configs are evaluating:
  - uses `max_eval_tail_pipelines` tail buffer (`cookimport/cli.py:4455-4459`, `6390-6403`)

Timeout/retry:

- config timeout with pool restart: `cookimport/cli.py:6444-6512`
- failed-config retry passes: `cookimport/cli.py:6522-6564`

## 5. Shared vs Different Work (for speedup leverage)

Shared hot path (speedups benefit both stage and benchmark prediction):

1. Importer conversion (`importer.convert(...)`)
2. EPUB/PDF split planning and split conversion
3. Block/chunk construction
4. Writing staged outputs
5. Writing stage block predictions

Benchmark-only hot path:

1. Freeform task generation artifacts
2. Evaluator runtime (especially canonical alignment)
3. All-method meta-scheduler overhead

Code split worth noting:

- `stage` split merge path (`_merge_split_jobs`) and benchmark split merge path (`_merge_parallel_results`) are separate implementations.

## 7. Evaluator Internals That Drive Runtime

Canonical evaluator details:

- `evaluate_canonical_text(...)` at `cookimport/bench/eval_canonical_text.py:991`
- Normalizes full prediction+canonical text and runs global `SequenceMatcher` (`:380`).
- Fast path is deliberately deprecated and forced to legacy for accuracy (`:30-33`, `:653-683`).
- Telemetry writes subphases/resources/work_units (`:1135+`).

Stage-block evaluator details:

- `evaluate_stage_blocks(...)` (`cookimport/bench/eval_stage_blocks.py:905`)
- Mostly dictionary/set accounting + metric computation.
- Telemetry confirms low evaluator runtime relative to prediction.

## 8. Speedup Leverage Summary for External Brainstorming

If the goal is speedups that help both stage and benchmark prediction:

1. Optimize importer conversion throughput (especially EPUB extraction modes).
2. Reduce split conversion overhead and merge overhead.
3. Reduce processed output write cost.
4. Unify and optimize duplicated split-merge implementations.

If the goal is all-method canonical wall time:

1. Canonical alignment algorithm/runtime is the primary bottleneck.
2. Scheduler tweaks alone cannot fix long eval tails.
3. Any safe alignment acceleration would dominate total wall-time gains.
