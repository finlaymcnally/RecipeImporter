# Oracle Browser Upload Failure Report

This note covers three separate Oracle browser-upload runs inspected on March 18, 2026.

## Scope

- Run 1: `2026-03-18_19.42.18/single-profile-benchmark`
- Run 2: `2026-03-18_21.10.12/single-offline-benchmark/dinnerfor2cutdown`
- Run 3: `2026-03-18_21.17.59/single-offline-benchmark/saltfatacidheatcutdown`

The main question for each run was not just "did Oracle answer?" but "is the answer actually grounded in the attached `upload_bundle_v1`?"

## Run 1: Prompt Stacking, But Mostly Grounded

Target bundle:

- `/home/mcnal/projects/recipeimport/data/golden/benchmark-vs-golden/2026-03-18_19.42.18/single-profile-benchmark/upload_bundle_v1`

Observed failure:

- the prompt body accidentally concatenated several stale benchmark requests before the final intended root

What still worked:

- Oracle's answer matched the actual attached bundle on the key facts
- `upload_bundle_overview.md` and `upload_bundle_index.json` reported `run_count = 3`, `pair_count = 0`, `changed_lines_total = 0`
- the answer matched the real run scores:
  - `01_dinnerfor2cutdown`: `overall_line_accuracy = 0.6681818181818182`
  - `02_saltfatacidheatcutdown`: `overall_line_accuracy = 0.7046384720327421`
  - `03_thefoodlabcutdown`: `overall_line_accuracy = 0.5193370165745856`
- the answer also matched real escalation counts:
  - `fallback_decision = 756`
  - `deterministic_unresolved = 754`

Assessment:

- this was a prompt-construction failure, not a grounding failure
- the answer was still mostly usable because it stayed on the final attached single-profile packet

Assessment:

- this was not a simple "attachment missing" failure
- Oracle appears to have read real evidence from the current bundle while mixing in a stale or hallucinated bundle identity
- because the answer contradicted the bundle's own verified root and pair metadata, it should be treated as invalid

## Failure Types Compared

The three runs failed in different ways:

- Run 1: prompt contamination, but answer mostly grounded
- Run 2: grounding failure with bundle-identity mixup
- Run 3: browser/session disconnect before completion

That distinction matters because "Oracle produced text" is not enough:

- a finished answer can still be untrustworthy if it contradicts the attached bundle metadata
- no visible ChatGPT conversation can mean either a UI quirk or a real failed upload, so the log has to decide it

## Practical Trust Rules

Treat an Oracle browser benchmark review as trustworthy only when all of these are true:

- `oracle_upload.log` contains a completed `Answer:` block
- the answer names the same benchmark root as the staged bundle
- the answer's topline values match `upload_bundle_overview.md` and `upload_bundle_index.json`
- the log does not contain browser disconnect / Chrome closed errors

For the three inspected runs, the practical verdicts are:

- Run 1: mostly trustworthy, despite bad prompt assembly
- Run 2: not trustworthy
- Run 3: failed completely
