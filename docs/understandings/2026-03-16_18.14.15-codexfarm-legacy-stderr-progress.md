---
summary: "Older codex-farm builds can still emit plain stderr progress lines like `run=... queued=...`, and cookimport must treat them as progress control lines rather than real stderr noise."
read_when:
  - "When debugging interactive benchmark or import progress output that seems to leak raw `codex-farm stderr (...)` blocks into the terminal."
  - "When changing `SubprocessCodexFarmRunner` progress parsing or deciding whether an older codex-farm stderr format should be logged versus consumed as progress."
---

The current runner prefers structured `__codex_farm_progress__` JSON stderr events, but some codex-farm builds still emit a plain text compatibility format like `run=<id>` and `queued=1 running=0 done=0 error=0 canceled=0`. Those lines are progress/control output, not actionable stderr.

If cookimport does not recognize that older shape, interactive benchmark/status flows end up logging `codex-farm stderr (...)` blocks while a Rich spinner panel is active, which makes the terminal look like the spinner is corrupting stderr. The real problem is misclassification: legacy progress frames need to be consumed by the runner, converted into normal progress callback updates, and excluded from stderr summaries.
