## Executive verdict

`docs/plans/OGplan/speed1-2.md` is only partially implemented.

Implemented behavior exists for:
- evaluate-phase visibility (`eval` in scheduler status),
- CPU-aware auto eval-tail defaults,
- smart-scheduler admission growth during eval tails,
- per-source scheduler telemetry output.

But the implementation is not complete to the original plan spec. The largest gaps are headroom semantics, runtime/report contract shape, and closure evidence requirements.

## Detailed deficiencies

### D1: Explicit eval-tail override is not CPU-budget bounded

Plan requirement:
- `docs/plans/OGplan/speed1-2.md:177-180` specifies bounding explicit override by per-source CPU budget (or a documented mild oversubscription multiplier).

Observed implementation:
- `cookimport/cli.py:4835-4840` computes:
  - explicit override path: `eval_tail_cap = max(0, min(eval_tail_requested, total))`
  - auto path: `auto_eval_tail_cap = max(0, cpu_budget_per_source - inflight)`
- No CPU-budget clamp is applied in explicit-override path.

Why this is a deficiency:
- It violates the plan’s explicit guard against unbounded override relative to CPU budget.

Impact:
- User-pinned values can exceed the intended per-source CPU envelope, increasing oversubscription risk and potentially worsening timeout/throughput stability.

Recommended fix:
- Clamp explicit overrides by CPU budget policy and document the exact bound in code and docs.

### D2: Runtime/report contract missing configured-vs-effective headroom fields

Plan requirement:
- `docs/plans/OGplan/speed1-2.md:185-188` and `:367-371` require runtime data carrying both configured and effective eval-tail headroom (and optional CPU budgeting context).

Observed implementation:
- `_resolve_all_method_scheduler_runtime(...)` returns tuple `(inflight, split_slots, wing_target, eval_tail_cap, smart_enabled, effective_inflight)` only:
  - `cookimport/cli.py:4807-4858`
- Per-source scheduler payload includes only:
  - `max_eval_tail_pipelines`, `smart_tail_buffer_slots`:
  - `cookimport/cli.py:6601-6604`
- No explicit `eval_tail_headroom_configured` vs `eval_tail_headroom_effective`.

Why this is a deficiency:
- The required auditability surface for “configured vs effective” headroom is missing.

Impact:
- Post-run analysis cannot reliably separate user intent from runtime cap resolution.
- Harder to debug “why did scheduler choose this cap?” questions.

Recommended fix:
- Add configured/effective headroom fields in runtime resolution, per-source scheduler JSON, and multi-source summary.

### D3: Smart-admission cap formula differs from OG plan contract

Plan requirement:
- `docs/plans/OGplan/speed1-2.md:209-212` defines eval-tail cap behavior as:
  - `max_active = base_inflight + eval_tail_headroom_effective`

Observed implementation:
- Scheduler base target is:
  - `effective_split_phase_slots + effective_wing_backlog_target`
  - `cookimport/cli.py:6760-6763`
- Smart admission cap is:
  - `smart_active_cap = scheduler_base_target + dynamic_eval_tail`
  - `cookimport/cli.py:6926-6929`
- Runtime pre-computed effective inflight similarly uses:
  - `split_slots + wing + eval_tail_cap`
  - `cookimport/cli.py:4846-4850`

Why this is a deficiency:
- This is not equivalent to the plan’s `base_inflight + headroom` contract.
- It can both exceed and differ from expected cap behavior depending on settings.

Impact:
- Scheduler capacity behavior does not match the declared spec model.
- Tuning guidance based on plan formula can be misleading.

Recommended fix:
- Align cap math with plan contract or revise/ratify spec and docs to match implemented semantics.

### D4: Plan closure evidence remains incomplete

Plan requirement:

  - `docs/plans/OGplan/speed1-2.md:16-27`, `:29-34`, `:46-50`
- Acceptance expects baseline/after artifacts and correctness/performance evidence:
  - `docs/plans/OGplan/speed1-2.md:295-313`, `:329-347`

Observed state:
- Plan still shows multiple unchecked items and placeholder sections:
  - `docs/plans/OGplan/speed1-2.md:19-27`, `:33-34`, `:50`
- Consolidated bench docs also call out unresolved evidence capture:
  - `docs/07-bench/07-bench_README.md:830-835`

Why this is a deficiency:
- The implementation may be functional, but plan-defined closure criteria are not satisfied.

Impact:
- Cannot honestly claim “implemented to spec” for this plan without caveats.

Recommended fix:
- Complete plan progress/outcomes with concrete run artifacts, baseline-vs-after comparisons, and acceptance evidence.

### D5: Test coverage does not enforce OG speed1-2 spec invariants

Plan requirement:
- Add unit tests for headroom/admission decisions:
  - `docs/plans/OGplan/speed1-2.md:24`, `:283-293`
- Validation emphasizes unchanged metrics + increased eval-tail concurrency + wall-time improvement:
  - `docs/plans/OGplan/speed1-2.md:301-313`

Observed tests:
- Runtime tests assert current cap semantics and values:
  - `tests/labelstudio/test_labelstudio_benchmark_helpers.py:3423-3522`
- Smart scheduler test checks utilization and eval-active presence:
  - `tests/labelstudio/test_labelstudio_benchmark_helpers.py:4072-4202`
- No test asserts explicit override CPU-budget clamp.
- No test asserts OG formula `base_inflight + eval_tail_headroom_effective`.
- No test asserts baseline-vs-smart metric parity from real canonical eval outputs.

Why this is a deficiency:
- Tests protect implemented behavior, but not the original spec-level invariants.

Impact:
- Spec drift can persist without test failures.

Recommended fix:
- Add explicit tests for override CPU clamp and OG cap formula, plus a canonical parity harness for baseline-vs-smart scheduler modes.

### D6: Pre-run scheduler printout lacks planned configured/effective visibility

Plan requirement:
- The plan’s artifacts show intended pre-run visibility:
  - configured/effective headroom and max-active-during-eval:
  - `docs/plans/OGplan/speed1-2.md:341-347`

Observed implementation:
- Pre-run print includes:
  - configured/effective inflight,
  - single `eval tail cap`,
  - `smart tail buffer`,
  - but no configured-vs-effective eval-tail split and no explicit max-active-during-eval:
  - `cookimport/cli.py:1482-1505`

Why this is a deficiency:
- Debug output does not fully expose the values the plan expects operators to inspect before run confirmation.

Impact:
- Harder to verify headroom resolution behavior at run start.

Recommended fix:
- Print configured/effective eval-tail headroom plus computed max-active-during-eval directly.

## Practical conclusion

The AI implementation should be described as:
- "functionally implemented with important spec-level gaps," not
- "fully implemented to OG speed1-2 spec."

Primary blockers to a strict "yes":
- D1, D2, D3 (semantic contract mismatches),
- D4 (missing closure evidence).
