The installed Oracle package under `/home/mcnal/.nvm/versions/node/v20.19.6/lib/node_modules/@steipete/oracle` still carried legacy browser defaults even after the repo-side benchmark transport was fixed. In particular, `dist/src/oracleHome.js` still defaulted to `~/.oracle`, `dist/src/browser/config.js` still defaulted browser manual-login profiles to `~/.oracle/browser-profile`, and `dist/src/browser/constants.js` still defaulted browser model selection to `DEFAULT_MODEL_TARGET = 'GPT-5.2 Pro'` with `DEFAULT_MODEL_STRATEGY = 'select'`.

That meant the local wrappers could force safer behavior for some call paths, but any direct Oracle invocation or incomplete wrapper path could still revive the same two old problems:

1. browser sessions landing in the wrong profile root, making login persistence look random
2. browser automation trying to operate the ChatGPT model picker using stale labels

Evidence:

- `dist/src/oracleHome.js`: `process.env.ORACLE_HOME_DIR ?? path.join(os.homedir(), '.oracle')`
- `dist/src/browser/config.js`: manual-login profile default `path.join(os.homedir(), '.oracle', 'browser-profile')`
- `dist/src/browser/constants.js`: `DEFAULT_MODEL_TARGET = 'GPT-5.2 Pro'`, `DEFAULT_MODEL_STRATEGY = 'select'`
- `dist/src/cli/bridge/claudeConfig.js`: bridge snippet defaults pointed at `.oracle-local`, adding a third profile/home convention

Conclusion: wrapper fixes were necessary but not sufficient. To truly consolidate on one Oracle browser profile and one stable browser behavior, the installed Oracle defaults needed to be patched too.
