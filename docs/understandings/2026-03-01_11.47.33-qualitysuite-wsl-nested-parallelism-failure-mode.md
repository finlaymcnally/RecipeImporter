# Discovery: QualitySuite WSL nested-parallelism failure mode

Observed from run artifacts (`data/golden/bench/quality/runs/2026-03-01_11.26.45`):

- `experiments_resolved.json` showed `experiment_executor_mode=thread` and `max_parallel_experiments_effective=4`.
- Per-experiment reports showed aggressive inner fanout (`configured_inflight_pipelines`, `effective_inflight_pipelines`) plus high worker values in run settings.

Practical interpretation:

- The run shape can multiply concurrency across three layers:
  - outer experiment fanout
  - per-experiment all-method inflight queue
  - per-config stage/split worker pools
- That multiplication can destabilize WSL even when average CPU charts look modest.

Design choice:

- Keep non-WSL behavior unchanged.
- On WSL with parallel experiment fanout, force subprocess experiment isolation and cap nested runtime/worker knobs conservatively.
- Persist guard telemetry in `experiments_resolved.json` so future debugging starts from actual effective settings, not assumptions.
