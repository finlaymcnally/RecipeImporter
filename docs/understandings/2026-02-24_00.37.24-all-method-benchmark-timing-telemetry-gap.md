# Discovery: all-method benchmark timing telemetry gap

- The all-method benchmark reports (`all_method_benchmark_report.json`, `all_method_benchmark_multi_source_report.json`) persist ranking/quality metrics and `created_at`, but no elapsed timing per config/stage.
- Benchmark history rows are written through `append_benchmark_csv(...)`, which intentionally leaves stage-only runtime columns (`total_seconds`, `parsing_seconds`, `writing_seconds`, `ocr_seconds`) empty.
- Benchmark prediction generation (`generate_pred_run_artifacts(...)`) writes processed cookbook reports, but those reports do not currently include `report.timing`.
- Regular stage/import flow still records timing in `ConversionReport.timing` via `stage_one_file(...)`, so benchmark and stage telemetry are not currently equivalent.
- Coarse runtime can be approximated by manifest timestamps:
  - Sweep folder `2026-02-24_00.10.28` to multi-source report `created_at=2026-02-24T00:31:59` is about `21m31s`.
  - Per-config rough elapsed can be inferred as `config run_manifest created_at - prediction-run run_manifest created_at` (no phase breakdown).
