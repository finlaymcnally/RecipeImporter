summary: "Benchmark selective retry should hook at codex-farm pass boundaries, because the orchestrator already tolerates missing per-recipe pass outputs and can continue with surviving recipes."
read_when:
  - When planning benchmark-only retry/resume for codex-farm recipe passes
  - When deciding whether to retry whole books versus only missing pass2/pass3 bundles
---

- `cookimport/llm/codex_farm_orchestrator.py` already uses per-recipe bundle files for pass1, pass2, and pass3.
- After each pass, it checks for `out/<bundle_name>` and marks only that recipe as errored when the file is missing.
- That means selective retry does not need a new scoring model; it needs a way to re-run only the missing bundle inputs for the failed pass, merge recovered outputs into the existing `out/` directory, and then continue normal downstream routing.
- The natural benchmark seam is:
  - detect recoverable partial pass failure in `cookimport/llm/codex_farm_runner.py`,
  - retry only missing `.json` bundle inputs for the current pass,
  - keep original successful outputs untouched,
  - record retry attempts and recovered/skipped recipe IDs in `raw/llm/<workbook_slug>/llm_manifest.json` and benchmark `run_manifest.json`.
- This should stay benchmark-only first because `COOKIMPORT_CODEX_FARM_RECIPE_MODE=benchmark` already exists, benchmark users care most about partial comparison salvage, and non-benchmark stage/import flows may prefer fail-fast or deterministic fallback semantics.
