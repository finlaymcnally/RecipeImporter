Root cause was in `_deterministic_label(...)` in `cookimport/parsing/canonical_line_roles.py`.

After the pre-grouping contract changed to `within_recipe_span=None`, deterministic `KNOWLEDGE` promotion still only fired from explicit outside-recipe branches. The same science prose that labeled correctly with `within_recipe_span=False` fell through to fallback `OTHER` with reason tags `deterministic_unresolved` + `deterministic_unavailable` when span state was `None`.

Minimal repro before the fix:
- `within_recipe_span=False`: cooking-science prose + `SALT AND FLAVOR` heading => `KNOWLEDGE`
- `within_recipe_span=None`: same rows => fallback `OTHER`

Fix:
- add an unknown-span prose branch that promotes explicit cooking-science prose to `KNOWLEDGE`
- add an unknown-span heading branch that promotes compact domain headings when neighboring knowledge prose supports them
- leave inside-recipe `KNOWLEDGE` restrictions unchanged

Validation:
- focused parsing suite: `source .venv/bin/activate && pytest tests/parsing/test_canonical_line_roles.py -q`
- real deterministic benchmark rerun:
  `data/golden/benchmark-vs-golden/2026-03-17_20.57.18-knowledge-fix-check/single-offline-benchmark/saltfatacidheatcutdown/vanilla`

Measured effect on that rerun versus the broken `2026-03-17_20.50.43` run:
- strict accuracy `0.4897680763983629` -> `0.5995907230559345`
- boundary correct `717` -> `1070`
- `KNOWLEDGE` recall `0/595` -> `266/595`
- `knowledge_pred_total` `9` -> `370`
- predicted recipe count `42` -> `35`

This does not fully restore the older best run (`0.6084583901773534`), but it removes the main live regression and puts the benchmark back near the prior deterministic shape.
