## Executive verdict

`speed1-4` is **Partially implemented**.

Major requirements are present (record schema, CLI flags, evaluate-only path, predict-only path, basic parity tests), but key plan requirements remain open:

- no true producer-consumer predict/evaluate overlap,
- run-level record contract instead of per-example stream boundary,
- no implemented stage API surface (`predict_stage`, `evaluate_stage`, `run_pipelined`, `run_legacy`) as specified,
- no captured real-workload timing evidence proving overlap benefit.

## Requirement closure summary

Closed:

- Prediction record schema and IO exist and are tested.
- CLI flags for execution mode and prediction record in/out exist.
- Evaluate-only from prediction record path exists and has focused tests.
- Legacy vs pipelined report payload parity test exists.
- Docs/help text mention execution modes and prediction record flags.

Partial:

- Predict/evaluate stage split exists operationally at run phase level, not at per-example stage API level.
- "Pipelined" mode overlaps prediction generation with canonical prewarm, not evaluation itself.

Missing:

- Per-example prediction record stream boundary.
- True concurrent prediction producer + evaluation consumer pipeline.
- Plan-maintenance acceptance evidence (baseline/after timing evidence captured in plan artifact).

Impact:

Closure criteria:

- Update `speed1-4.md` with concrete paths/symbols, checked progress items, and explicit baseline/after evidence.

### D2: Stage API extraction did not land in the shape specified by the plan

Requirement:

- Stage interfaces should exist (`predict_stage`, `evaluate_stage`) and runner APIs (`run_legacy`, `run_pipelined`) per `speed1-4` interface section.
- Reference: `docs/plans/OGplan/speed1-4.md:454-455`, `:463-469`.

Evidence:

- Execution orchestration is embedded in `labelstudio_benchmark` within `cookimport/cli.py` (entrypoint and orchestration around `:10220+`, `:10628+`, `:10834+`).
- No corresponding stage/runner API module exists under `cookimport/bench/` for the named plan interfaces.

Deficiency:

- The architecture was implemented as CLI-internal orchestration rather than the explicit stage/runner API contract described in the plan.

Impact:

- Harder unit isolation, reuse, and future transport to a true streaming model.

Closure criteria:

- Introduce explicit stage/runner interfaces or formally revise spec to declare the CLI-embedded alternative as accepted scope.

### D3: Prediction record contract is run-level, not per-example

Requirement:

- One `PredictionRecord` per input example with deterministic `example_id`/`example_index`.
- Reference: `docs/plans/OGplan/speed1-4.md:153-155`.

Evidence:

- Record creation hard-codes a single run-level record with `example_index=0`: `cookimport/cli.py:4048-4050`.
- Evaluate-only loader enforces exactly one record for this path: `cookimport/bench/prediction_records.py:161-168`, used at `cookimport/cli.py:4076`.
- Tests assert one record written in predictions-out path: `tests/labelstudio/test_labelstudio_benchmark_helpers.py:2313-2315`.
- Bench docs call this out as run-level artifact: `docs/07-bench/07-bench_README.md:189`, `:823`, `:904`.

Deficiency:

- Contract boundary is run-level artifact pointers, not per-example prediction stream.

Impact:

- Cannot replay, join, or debug at per-example granularity through the prediction record contract.
- Prevents literal closure of the spec’s per-example stage boundary.

Closure criteria:

- Emit one record per evaluated example and consume iteratively in evaluation stage, or explicitly revise the OG spec with rationale.

### D4: Evaluate-only path reuses artifact file pointers, not per-example prediction payloads

Requirement:

- Evaluation stage should consume prediction records and associate predictions to examples (prefer `example_id`, fallback `example_index`).
- Reference: `docs/plans/OGplan/speed1-4.md:172-177`, `:183-184`, `:466-469`.

Evidence:

- Evaluate-only reconstructs a bundle by reading file-path fields from a single record:
  - required path field checks: `cookimport/cli.py:4055-4068`
  - stage/extracted path extraction: `cookimport/cli.py:4077-4084`
  - evaluates by passing those files to evaluator: `cookimport/cli.py:10837-10850`
- Test verifies this path-based evaluate-only behavior: `tests/labelstudio/test_labelstudio_benchmark_helpers.py:2327-2447`.

Deficiency:

- Evaluate-only is decoupled at run-artifact level, not at prediction-record-per-example level.

Impact:

- Replay reliability depends on external artifact files still existing and matching expected state, reducing durability of the record boundary itself.

Closure criteria:

- Store prediction payloads needed for scoring directly per example, or formally define and version the current path-pointer contract as intentional non-equivalent scope.

### D5: "Pipelined" mode does not overlap prediction with evaluation consumption

Requirement:

- Runner must start producer and consumer concurrently and overlap work.
- Reference: `docs/plans/OGplan/speed1-4.md:188-199`, `:212`.

Evidence:

- Current pipelined branch starts prediction thread, prewarms canonical assets, then blocks on `producer_thread.join()` before evaluation:
  - queue/thread setup: `cookimport/cli.py:10632-10650`
  - prewarm: `cookimport/cli.py:10653`
  - join before obtaining bundle/eval: `cookimport/cli.py:10656-10667`
  - evaluation begins later in `_evaluate_selected_mode`: `cookimport/cli.py:10834-10852`
- CLI help text describes pipelined mode as prewarm overlap: `cookimport/cli.py:10224-10226`.
- Test confirms prewarm overlap behavior specifically: `tests/labelstudio/test_labelstudio_benchmark_helpers.py:2663-2788`.

Deficiency:

- This is phase overlap (prediction + prewarm), not prediction/evaluation streaming overlap.

Impact:

- Misses core wall-clock benefit target in the original pipeline spec where evaluation consumes records while prediction continues.

Closure criteria:

- Implement a true producer/consumer handoff where evaluation consumes records before prediction completes, with clean EOS/error propagation.

### D6: Deterministic example-level join semantics are not implemented in the current contract

Requirement:

- Deterministic join by `example_id`/`example_index` with per-example mapping correctness.
- Reference: `docs/plans/OGplan/speed1-4.md:30`, `:172-173`, `:466-469`.

Evidence:

- Single run-level record contains artifact pointers rather than per-example predictions (`cookimport/cli.py:4025-4051`).
- Evaluation functions are invoked with stage/extracted files, not prediction record iterables (`cookimport/cli.py:10837-10850`).

Deficiency:

- Join correctness is inherited from downstream stage evidence files, not expressed/enforced through the prediction-record contract itself.

Impact:

- The plan’s explicit "record boundary enforces mapping correctness" requirement is not satisfied by current architecture.

Closure criteria:

- Introduce iterable prediction-record consumption in eval path with explicit join checks and failure modes.

### D7: Real workload timing evidence for legacy vs pipelined remains open

Requirement:

Closure criteria:
