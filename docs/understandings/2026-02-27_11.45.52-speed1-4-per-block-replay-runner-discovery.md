## Discovery

The practical speed1-4 closure point in this codebase is not a brand-new evaluator implementation; it is a boundary refactor around existing evaluators.

Key findings from implementation:

- Existing evaluators (`evaluate_stage_blocks`, `evaluate_canonical_text`) are stable and path-based, so replay should reconstruct their expected stage/extracted artifacts from prediction records instead of rewriting metric internals.
- A per-block `PredictionRecord` contract (`schema_kind=stage-block.v1`) gives deterministic join semantics (`example_id`/`example_index` tied to `block_index`) while preserving evaluator parity.
- Evaluate-only needs to accept both new per-block records and older single-record run-pointer artifacts to avoid breaking historical benchmark outputs.
- Pipelined execution is best expressed as a bounded producer/consumer queue over prediction records, with canonical prewarm overlapping in the same run; this preserves error propagation and backpressure contracts.
