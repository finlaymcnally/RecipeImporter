The phase-plan series is mostly aligned now. The remaining easy place for drift was Phase 3. Even after the earlier revisions, it still had one interface that accepted `ConversionResult` as the recipe input and a few validation sentences that made raw storage naming sound like the primary proof of the new topology.

That is the wrong boundary for the updated series. Phase 2 now owns normalized block labels plus block-range `RecipeSpan` handoff, so Phase 3 should consume that label-first bundle directly. Phase 1 owns semantic stage naming through `stage_observability.json`, so later phases should prove new topology there first and only treat raw pass-slot directories as compatibility storage.

Short rule for future edits:

- Phase 2 defines authoritative labels and recipe spans.
- Phase 3 consumes those spans, not importer-owned candidates.
- Phase 4 consumes the same block-label and recipe-span truth for non-recipe ownership.
- Phase 1 remains the place where semantic stage names become shared product truth.
