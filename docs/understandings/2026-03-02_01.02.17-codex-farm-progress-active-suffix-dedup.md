summary: "Codex-farm progress dedupe keys off full message text, so volatile active-file suffixes bypass dedupe and flood plain-progress logs."
read_when:
  - "When debugging codex-farm benchmark progress noise"
  - "When changing progress-event formatting or callback dedupe semantics"
---

`SubprocessCodexFarmRunner` dedupes progress updates by exact emitted message string (`last_progress_message`). The formatted codex-farm progress message previously included `| active <filename>` derived from `running_tasks[0].input_path`. During long runs, counters can stay at `task 0/N` while `active` rotates rapidly, so each callback becomes unique and bypasses dedupe.

Removing the volatile active-file suffix keeps messages stable for identical counter states, so existing dedupe suppresses churn while preserving task/running/error progress signals.
