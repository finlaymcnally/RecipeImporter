Key path:

- `build_label_first_stage_result(...)` atomizes archive blocks, then runs canonical line-role.
- `_atomize_archive_blocks(...)` sets each atomic line's `within_recipe_span` from `_recipe_ranges_from_conversion_result(...)`.
- `_recipe_ranges_from_conversion_result(...)` reads `start_block` / `end_block` from existing `conversion_result.recipes[*].provenance.location`.

That means canonical line-role does **not** start from a blank slate. It inherits old recipe boundaries first, and those boundaries gate the deterministic baseline plus Codex prompt framing.

Concrete evidence from the 2026-03-17 `saltfatacidheatcutdown` codexfarm run:

- `Bright Cabbage Slaw` was accepted as `recipe_span_70` with atomic/block range `1358-1371` / `1118-1127`.
- The real method lines immediately after it live at blocks `1128-1130` / atomic indices `1372-1374`.
- Those method lines are already outside recipe in `label_det/saltfatacidheatcutdown/labeled_lines.jsonl`, where they deterministically become `OTHER` with reason tags `outside_recipe_span` and `prose_default_other`.
- They then flow into Stage 7 nonrecipe ownership as `nr.other.1128.1131` in `08_nonrecipe_spans.json`.

So the first bad decision is not "Codex misread the instructions." The system had already marked those instructions as outside recipe before line-role correction.

Immediate implication:

- If we want line-role to rescue missing method blocks, the first seam to revisit is the provenance-backed recipe-range hinting in `label_source_of_truth.py`, or the earlier importer logic that writes those `start_block` / `end_block` ranges in the first place.
- Prompt tuning alone cannot solve this cleanly, because the model is being asked to review lines under already-bad inside/outside recipe metadata.
