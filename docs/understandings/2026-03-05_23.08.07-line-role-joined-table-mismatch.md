Key findings from the 2026-03-04_20.33.53 benchmark run:

- `cookimport/bench/cutdown_export.py::build_line_role_joined_line_rows(...)` was joining evaluator `wrong_label_lines.jsonl -> line_index` to `line_role_predictions.jsonl -> atomic_index` as if they were the same coordinate system.
- In the Salt/Fat sample run, that assumption is false at scale: every overlapping `line_index == atomic_index` text pair inspected was a different line, so `joined_line_table.jsonl` could show the right canonical text with another line's `candidate_labels`, `decided_by`, and recipe-span flags.
- Concrete evidence from the checked-in run:
  - canonical line `1079` is `A FEW BASIC HOW-TOS`, but the real line-role prediction for that text is `atomic_index 1204` with candidates `RECIPE_VARIANT / RECIPE_TITLE / OTHER`.
  - canonical line `1145` is `4 to 5 tablespoons lime juice`, and there is no exact single-line prediction row for that text because the line-role path kept it inside a larger merged atomic segment.
- Safe fix: only attach line-role metadata to canonical joined rows when the exporter can prove the match:
  - first preference: same index and exact normalized text
  - fallback: exact normalized text by occurrence order
  - otherwise leave the joined row unmatched with empty candidate metadata
- Consequence: historical `joined_line_table.jsonl` / follow-up packets built with the old join are stale and should not be used for prompt tuning without regeneration.
