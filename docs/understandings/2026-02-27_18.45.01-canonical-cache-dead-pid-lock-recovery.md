## Discovery

During run `data/golden/benchmark-vs-golden/2026-02-27_17.54.41`, `thefoodlabcutdown` evaluate workers stalled with multiple lock files whose owners were dead:

- `.cache/canonical_alignment/34/...lock` -> `pid=64635` (dead)
- `.cache/canonical_alignment/3f/...lock` -> `pid=64631` (dead)
- `.cache/canonical_alignment/88/...lock` -> `pid=64628` (dead)
- `.cache/canonical_alignment/df/...lock` -> `pid=64634` (dead)

At the same time, scheduler events stayed at `evaluate_started` for active configs and did not emit `evaluate_finished`.

Root cause in code: `CanonicalAlignmentDiskCache._is_lock_stale(...)` only used file age (`wait_seconds`, default `3600`) and ignored lock-owner PID health.

## Practical implication

If a worker dies while holding a canonical cache lock, other workers can wait up to an hour (or until timeout) even though the lock owner is gone.  
Dead-owner lock reclamation must be immediate via PID liveness checks, with age-based fallback only for malformed lock files.
