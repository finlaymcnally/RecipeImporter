`cookimport/llm/codex_farm_contracts.py` was already migration-ready before this task started. The pass2, pass3, and merged-repair loaders all call `_coerce_json_object_field(...)`, which accepts either a real JSON object or a string containing one.

The actual wrapper contract still lived in the pipeline pack:
- `llm_pipelines/schemas/recipe.schemaorg.v1.output.schema.json`
- `llm_pipelines/schemas/recipe.final.v1.output.schema.json`
- `llm_pipelines/schemas/recipe.merged-repair.v1.output.schema.json`
- `llm_pipelines/prompts/recipe.schemaorg.compact.v1.prompt.md`
- `llm_pipelines/prompts/recipe.final.compact.v1.prompt.md`
- `llm_pipelines/prompts/recipe.merged-repair.compact.v1.prompt.md`
- `cookimport/llm/fake_codex_farm_runner.py`

That means the low-risk migration path is:
1. Keep the runtime coercion shim for old artifacts.
2. Change the pack assets and fake runner so new outputs are real nested objects.
3. Update tests to treat object-form output as the normal path and string-form output as an explicit compatibility case.

One schema-design constraint matters for the native-object version: this repo's Codex structured-output tests already enforce that every top-level property appears in `required`. The nested recipe objects therefore need to stay simple and explicit, with nullable fields used where the old prompts previously said "omit when unknown."
