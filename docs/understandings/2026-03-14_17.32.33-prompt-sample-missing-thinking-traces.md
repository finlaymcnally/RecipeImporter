# Prompt sample missing thinking traces

For `data/golden/benchmark-vs-golden/2026-03-14_17.21.15/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/prompts/prompt_type_samples_from_full_prompt_log.md`, the sampler in `cookimport/cli.py` is working as designed.

Evidence:

- The sampler only emits `_No thinking trace captured for this sample._` when the selected sample has no extracted reasoning excerpt and `thinking_trace.available` is false.
- In this run's `full_prompt_log.jsonl`, pass distribution was:
  - `pass1`: `23` rows, all `available=true`, `reasoning_event_count=1`
  - `pass2`: `23` rows, all `available=false`, `reasoning_event_count=0`
  - `pass4`: `149` rows total, `143` with `available=false`, `reasoning_event_count=0`, and `6` with `available=true`, `reasoning_event_count=1`
- The sample markdown chooses the first three rows per pass, so pass2 and pass4 naturally surfaced zero-trace samples in this run.
- Raw `.trace.json` files for affected pass2/pass4 calls still contain an `events[2]` entry of type `item.completed` with an `agent_message` payload, but those same files already recorded `reasoning_event_count=0` and empty `reasoning_events`.
- `var/codex_exec_activity.csv` agrees with the `.trace.json` files: affected pass2/pass4 rows have `trace_reasoning_count=0` even though `codex_event_types_json` includes `item.completed`.

Conclusion:

The visible symptom in prompt samples is downstream of upstream trace classification/capture. The likely bug surface is whatever writes `reasoning_event_count` / `reasoning_events` into CodexFarm trace artifacts for merged-repair and most pass4 calls, not the prompt-sample markdown exporter.
