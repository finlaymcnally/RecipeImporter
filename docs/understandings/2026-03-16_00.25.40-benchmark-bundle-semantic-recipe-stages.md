`upload_bundle_v1` no longer needs `pass2_stage` / `pass3_stage` compatibility fields to explain recipe topology. The normalized bundle model can carry one semantic `recipe_stages` list plus a `recipe_topology_key`, and both `analysis.recipe_pipeline_context` and `analysis.stage_separated_comparison` can render from that shared shape.

The current semantic recipe-stage contract is:

- standard recipe pipeline: `schemaorg`, `final`
- merged-repair recipe pipeline: `merged_repair`

`scripts/benchmark_cutdown_for_external_ai.py` still derives many diagnostics from pass1/pass2/pass3 manifest fields because the underlying recipe diagnostics are still emitted that way, but the starter-pack and upload-bundle outputs should translate those internals into semantic recipe-stage rows before they become reviewer-facing.

Selected recipe packets and the casebook now work best when they expose:

- `chunking_summary` for the pass1 boundary-selection step
- `recipe_stages` as semantic stage summaries for recipe correction/finalization
- transport/evidence summaries separately

`cf-debug` pass4 follow-up now accepts both the new semantic local artifact names (`knowledge_manifest.json`, `prompt_task4_knowledge.txt`) and the older checked-in pass4 sample names/paths (`pass4_knowledge_manifest.json`, `prediction-run/prompt_budget_summary.json`, legacy prompt-task filenames). That keeps historical sample bundles auditable without reintroducing pass-slot fields into bundle analysis.
