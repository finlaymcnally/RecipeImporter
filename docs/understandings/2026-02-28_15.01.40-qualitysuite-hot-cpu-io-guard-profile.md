# QualitySuite Hot CPU + IO Guard Profile

- In all-method runs, scheduler split-worker capping feeds into effective `workers`, `epub_split_workers`, and `pdf_split_workers` per config.
- With 16 CPUs, aggressive source+split fanout can collapse per-config effective workers (for example source parallelism `4` with split slots `3` drives cap near `1`).
- To stay hot without broad disk thrash, the safer pattern is:
  - keep base workers relatively high,
  - constrain concurrent split phases,
  - limit source parallelism to avoid too many simultaneous source roots writing at once.
- A practical profile for this repo is captured in:
  - `data/golden/bench/quality/experiments/2026-02-28_14.58.21_qualitysuite-top-tier-base-run-settings-hot-io-guard.json`
  - `data/golden/bench/quality/experiments/2026-02-28_14.58.21_qualitysuite-top-tier-tournament-hot-io-guard.json`
- That profile keeps deterministic sweeps available (via thresholds sweeps-on file) and is intended to pair with `--max-parallel-experiments 2` for fold runs.
