# Historical Note: Why `prev_text` And `next_text` Still Existed

This note describes the short-lived state before `docs/understandings/2026-03-18_17.05.01-atomic-line-adjacency-now-comes-from-explicit-index-lookup.md` landed.

The recent line-role payload trim only removed `prev_text` and `next_text` from the model-facing shard JSON written to `line-role-pipeline/runtime/line_role/workers/*/in/*.json` and preview `line-role-pipeline/in/*.json`.

Those neighbor fields still live on `cookimport/parsing/recipe_block_atomizer.py::AtomicLineCandidate` because the rest of the program still uses adjacency in a few places:

- candidate builders in `recipe_block_atomizer.py`, `labelstudio/ingest.py`, `label_source_of_truth.py`, and `llm/prompt_preview.py` populate `prev_text` / `next_text`
- `cookimport/llm/canonical_line_role_prompt.py` still emits selective inline `ctx:<atomic_index>|prev=...|line=...|next=...` rows for ambiguous targets
- `cookimport/parsing/canonical_line_roles.py` still includes neighbor text in the candidate fingerprint cache input and still consults nearby line text in some title/knowledge heuristics

So the design at that moment was:

- file-backed shard JSON: no duplicated neighbor fields
- in-memory candidate objects: still keep neighbor text
- inline compact prompt seam: can still render neighbor context selectively when needed

That design is no longer current. The later candidate cleanup removed neighbor text from `AtomicLineCandidate` too and standardized adjacency on explicit ordered lookup helpers.
