---
summary: "How quality-per-token metrics are computed for runtime diagnostics and Previous Runs"
read_when:
  - "When quality-per-token values look inconsistent between runtime card and Previous Runs"
  - "When changing benchmark quality-efficiency formulas"
---

`Previous Runs` and runtime diagnostics now share the same discounted token math (`all_token_use`):
`(tokens_input - tokens_cached_input) + 0.1*tokens_cached_input + tokens_output`.

The quality side uses a deterministic fallback chain:
1. `strict_accuracy`
2. `macro_f1_excluding_other`
3. `f1`

`quality_per_million_tokens` is `quality_score / discounted_tokens * 1_000_000` (higher is better).

Runtime card deltas vs vanilla are computed within the latest run-group only:
- `Delta quality vs vanilla` = codexfarm quality - vanilla quality
- `Delta quality / 1M extra tokens vs vanilla` = `Delta quality / (codexfarm_tokens - vanilla_tokens) * 1_000_000` (only when codexfarm uses more tokens)

Peer comparison rank is computed against other token-spending run-groups using the same quality metric key as the latest run-group.
