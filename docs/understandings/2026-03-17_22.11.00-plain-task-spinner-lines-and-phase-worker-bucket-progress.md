summary: "The shared spinner can make every plain `task X/Y` update richer immediately, while recipe phase-worker progress should report outer worker-bucket state rather than pretending it knows inner shard status."
read_when:
  - When a stage still emits only plain `task X/Y` strings and you want the spinner to show more than one line without rewriting that stage first
  - When changing `cookimport/llm/phase_worker_runtime.py` progress behavior for recipe shard work
  - When reconciling why recipe progress can show worker buckets and queued shards but not true per-shard inner CodexFarm execution details
---

- `_run_with_progress_status(...)` can infer and render useful `stage:` plus `progress:` lines from any plain counter message, so spinner richness does not have to wait for every emitter to adopt structured payloads.
- The right place for real recipe worker metadata is `phase_worker_runtime.py`, because that module owns worker assignments and shard buckets. It can truthfully report configured workers, queued shards, and the current first shard in each active worker bucket.
- It still cannot report true inner per-shard progress inside one worker bucket, because `_run_worker_assignment_v1(...)` hands the whole assigned `in/` directory to one classic `runner.run_pipeline(...)` call. Outer assignment completion is real; inner shard-by-shard live status is not available at that seam.
