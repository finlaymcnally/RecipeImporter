## Labelstudio domain fast-slice hotspots

`./scripts/test-suite.sh domain labelstudio` is single-process pytest (`scripts/test-suite.sh` just runs one `pytest tests/labelstudio -m "not slow"` invocation), so extra CPU cores do not help by default.

Measured on 2026-03-15:

- Full non-slow `tests/labelstudio`: `1 failed, 211 passed, 95 deselected` in `227.82s`
- `tests/labelstudio/test_labelstudio_benchmark_helpers_interactive.py`: about `121s`
- `tests/labelstudio/test_labelstudio_benchmark_helpers_single_offline_run.py`: about `79s`
- `tests/labelstudio/test_labelstudio_benchmark_helpers_progress.py`: about `7s`
- `tests/labelstudio/test_labelstudio_benchmark_helpers_import_eval.py`: about `7s`
- `tests/labelstudio/test_labelstudio_benchmark_helpers_artifacts.py`: about `4s`
- `tests/labelstudio/test_labelstudio_benchmark_helpers_single_offline_artifacts.py`: about `4s`
- `tests/labelstudio/test_labelstudio_ingest_parallel.py`: about `3s`

Main cause: the fast slice still includes benchmark-helper tests that execute `_interactive_single_offline_benchmark(...)` and related single-offline flow multiple times. In `test_labelstudio_benchmark_helpers_interactive.py`, some tests patch `labelstudio_benchmark(...)` but still route through the real `_interactive_single_offline_benchmark(...)`, so they also pay for comparison/bundle/dashboard work instead of stopping at command wiring.
