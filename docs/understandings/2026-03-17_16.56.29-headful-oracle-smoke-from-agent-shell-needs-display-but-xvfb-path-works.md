The local visible-browser wrapper and the local headless/xvfb wrapper now exercise two different environment assumptions.

An actual Oracle smoke test launched from the agent shell with `/home/mcnal/.local/bin/oracle-browser-safe --model gpt-5.2-instant ...` failed with:

- session error: `connect ECONNREFUSED 127.0.0.1:35923`
- Chromium child log: `Missing X server or $DISPLAY`

That failure happened before ChatGPT login/model/prompt behavior; Chromium exited because this agent shell had no usable Linux display for a headful launch.

The follow-up smoke test using `/home/mcnal/.local/bin/oracle-browser-headless --model gpt-5.2-instant ...` succeeded after restoring its default launcher to `chromium-nosandbox-xvfb`. The completed session slug was `headless-oracle-smoke-2`, and Oracle returned:

- `RESULT: OK.`
- `SUMMARY: Oracle runtime defaults were unified around one canonical local browser profile and shared wrapper setup.`

Evidence:

- visible failure session: `/home/mcnal/.local/share/oracle/sessions/visible-oracle-smoke/`
- successful xvfb session: `/home/mcnal/.local/share/oracle/sessions/headless-oracle-smoke-2/`
- headful Chromium log: `/tmp/chromium-wrapper-child.log`
- xvfb Chromium log: `/tmp/chromium-wrapper-xvfb-child.log`

Conclusion: for Codex-run verification in this shell, `oracle-browser-headless` is still the reliable real-browser path. The user-facing benchmark/browser flow can remain visible in their own interactive environment, but automated smoke tests from the agent shell should prefer the xvfb wrapper unless a real display is confirmed.
