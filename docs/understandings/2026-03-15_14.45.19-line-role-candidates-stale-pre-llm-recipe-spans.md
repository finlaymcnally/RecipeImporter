# Line-role candidates stale pre-LLM recipe spans

Finding from the `2026-03-14_17.21.15` Salt Fat Acid Heat single-offline benchmark:

- The missing instruction clusters were **not** dropped by Codex pass1/pass2 transport.
- Example: recipe bundle `c0` transport audit still covered source blocks `1118-1137`, including the missed instruction blocks `1128-1130`.
- But canonical line-role predictions for those same source blocks were generated with `recipe_id=null` and `within_recipe_span=false`.

Root cause:

- In `cookimport/labelstudio/ingest.py`, `_build_line_role_candidates_from_archive(...)` runs **before** `run_codex_farm_recipe_pipeline(...)`.
- That means `line_role_candidates` are built from the pre-LLM `ConversionResult.recipes` provenance ranges.
- Later, `run_codex_farm_recipe_pipeline(...)` updates `result = llm_apply.updated_conversion_result`, but the already-built `line_role_candidates` are not rebuilt.
- So the canonical line-role pipeline can score with stale span ownership even when the Codex recipe pipeline expanded/fixed recipe block ranges.

Why this matters:

- Outside a recipe span, canonical line-role heuristics aggressively gate `INGREDIENT_LINE` / `INSTRUCTION_LINE` and often fall back to `OTHER`.
- Pass4 can then further convert some of those outside-span lines to `KNOWLEDGE`.
- This suppresses benchmark-visible gains from the recipe pipeline, especially for instruction lines that depended on updated recipe coverage.

Evidence chain:

1. `transport_audit/...c0.json` and `...c7.json` show the recipe bundles included the missed source blocks.
2. `prediction-run/raw/llm/.../llm_manifest.json` shows those recipes kept the same inclusive block bounds in pass1/pass2.
3. `prediction-run/line-role-pipeline/line_role_predictions.jsonl` still marks the affected source blocks with `recipe_id=null` / `within_recipe_span=false`.
4. Code order in `cookimport/labelstudio/ingest.py` shows line-role candidates are computed before the LLM-updated `ConversionResult` replaces the original result.
