# Profeedback ExecPlan Gap Scan

Quick findings from source review while converting `docs/plans/OGplan/Profeedback.md` into an ExecPlan:

- `pass3 empty/low-quality fallback` is already implemented in `cookimport/llm/codex_farm_orchestrator.py` (`_pass3_low_quality_reasons`, deterministic fallback payload generation, and fallback status reporting).
- Canonical line-role already has meaningful prompt negatives (`llm_pipelines/prompts/canonical-line-role-v1.prompt.md`) and deterministic outside-span heuristics/sanitization (`cookimport/parsing/canonical_line_roles.py`).
- The main unresolved gap is runtime acceptance policy: there is benchmark-time line-role gating in `cookimport/cli.py::_build_line_role_regression_gate_payload`, but no runtime do-no-harm arbitration that can reject harmful codex line-role overrides during labeling.
- Another unresolved gap is pass1 recipe-eligibility strictness before pass2 promotion; pass1 currently trusts `is_recipe` + bounds validity with overlap clamps but no structural evidence floor.

This is why the new ExecPlan focuses on do-no-harm routing, outside-span containment, selective codex escalation, and pass1 eligibility gates.
