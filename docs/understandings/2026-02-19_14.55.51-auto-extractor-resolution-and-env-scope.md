# Auto Extractor Resolution and Runtime Env Scope

Key discovery while hardening `auto` mode:

- `auto` resolution must happen once in parent orchestration (stage / prediction-run generation), then workers receive the resolved concrete backend (`legacy|unstructured|markdown`).
- If prediction generation writes `C3IMP_EPUB_*` env vars globally and does not restore them, later runs/tests can inherit stale extractor settings (for example `auto`) and behave inconsistently.

Current implementation pattern:

- `cookimport/parsing/epub_auto_select.py` resolves `auto` using deterministic spine sampling + quality scoring.
- `cookimport/cli.py` resolves per-file effective extractor before split planning, writes `raw/epub/<source_hash>/epub_extractor_auto.json`, and passes the effective extractor explicitly to worker calls.
- `cookimport/labelstudio/ingest.py` now applies EPUB runtime env vars in a scoped context manager and restores previous values after conversion.

Why this matters:

- Split determinism: all jobs for one EPUB use the same effective backend.
- Reproducibility: requested/effective extractor is persisted in run config + artifacts.
- Test hygiene: prediction runs no longer leak extractor env values into later tests.
