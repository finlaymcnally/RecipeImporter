# CodexFarm Explicit-Decision Compromise

- `COOKIMPORT_ALLOW_LLM` in `cookimport/llm/codex_exec.py` is a low-level kill switch, not a full operator-intent policy. It blocks direct Codex CLI calls, but it does not explain which run wanted Codex or prevent ambiguous settings payloads upstream.

- SpeedSuite and QualitySuite already have the right idea for batch runs: `cookimport/cli.py` requires explicit confirmation tokens before `--include-codex-farm` is honored. Those flows prove that “Codex must be an explicit decision” is already compatible with this repo.

- The remaining problem is fragmentation. `stage(...)`, `labelstudio_import(...)`, and settings-helper construction do not all pass through one shared decision seam, so a hidden or mixed Codex posture can still slip in from a different path.

- `RunSettings` model defaults are now safe, but `build_run_settings(...)` in `cookimport/config/run_settings.py` still defaults recipe Codex, line-role Codex, and atomic splitting to live values in its function signature. That helper is currently a split-brain source of truth.

- Because this repo is agent-controlled, truly “human only” enforcement is not realistic. Any repo-local file or config bit can be changed by code. The practical target is stronger and more honest: make Codex impossible to trigger silently.

- The right compromise is a single central decision model:
  - named canonical profiles such as `vanilla`, `codex_recipe`, and `codex_full`
  - one validator that rejects ambiguous mixed payloads
  - explicit approval seams for non-vanilla runs
  - loud manifest and CLI output that says which profile actually ran

- If that central seam exists, future QualitySuite-style bugs should become immediate validation errors or obvious manifest rows instead of surprise token-usage audits discovered later.
