# CodexFarm ExecPlan Split

The unified reliability plan was split into four smaller ExecPlans because it mixed four different kinds of work: invariant-setting, upstream structure policy, current-path repair, and architecture prototyping. Those tracks share a benchmark and a diagnosis, but they do not have the same file overlap or dependency shape.

The four plans are:

- `docs/plans/2026-03-13_22.29.37-codexfarm-observability-and-invariants.md`
- `docs/plans/2026-03-13_22.29.38-codexfarm-staging-and-label-ownership.md`
- `docs/plans/2026-03-13_22.29.39-codexfarm-3pass-reliability-repair.md`
- `docs/plans/2026-03-13_22.29.40-codexfarm-merged-repair-prototype.md`

The safest same-repo parallelism is two tracks at once: staging-and-label-ownership plus current 3-pass repair. The observability plan should land first or stay tightly coordinated because it defines reason codes and invariant names the other plans should reuse. The merged-stage prototype has the most overlap with shared orchestrator and contract surfaces, so it should wait until the control path and diagnostics have settled or move to a separate worktree.
