# Line-role OTHER shortlist distinction

This note started as a snapshot of the pre-removal state. After the 2026-03-15 cleanup, only the heuristic miss remains current.

Two separate behaviors existed in the original investigation:

- The old prediction-run bug where line-role candidates were built before `run_codex_farm_recipe_pipeline(...)` updated recipe provenance is fixed. `cookimport/labelstudio/ingest.py` now rebuilds `line_role_candidates` after the recipe Codex pass, and `tests/labelstudio/test_labelstudio_ingest_parallel.py::test_generate_pred_run_artifacts_rebuilds_line_role_candidates_after_llm_recipe_update` covers that order.
- At the time of investigation, the line-role Codex path was hard-constrained by per-row `candidate_labels`. That prompt/parser constraint and the surrounding candidate-label metadata have since been removed, so Codex now sees the full global label vocabulary on every row and current prediction artifacts no longer carry shortlist fields.

Current remaining heuristic gap:

- In-recipe lines can still miss strong heuristic tags even without stale recipe spans.
- `cookimport/parsing/recipe_block_atomizer.py` can still drop title/ingredient/instruction-specific tags on these lines and fall back to generic prose-style tagging.
- The old allowlist collapse to `['OTHER']` is gone; what remains is a deterministic miss worth debugging, not an LLM constraint.

Verified examples in the current workspace:

- `1 large jalapeño, seeds and veins removed if desired, thinly sliced`
- `Shaved Carrot Salad with Ginger and Lime`
- `Toss the croutons with the olive oil to coat them evenly...`
- `Variations`

Each of those still misses the strong heuristic tags you would want when evaluated as `within_recipe_span=True`; after the cleanup that miss remains useful for debugging, but it no longer limits which label Codex may return.
