---
summary: "The old shard-v1 progress surface could report multiple active workers even though the runtime still launched worker assignments serially; the fix was to make the worker-assignment layer actually run concurrently."
read_when:
  - When `active workers: N` looks real in benchmark/import status but the wall-clock runtime still feels serialized
  - When changing `cookimport/llm/phase_worker_runtime.py` or the direct knowledge / direct line-role worker loops
  - When reconciling benchmark telemetry with worker assignment manifests for shard-v1 Codex phases
---

- The worker-count default fix only changed how many worker buckets were planned. It did not change execution strategy.
- In the old code, `phase_worker_runtime.py`, `_run_direct_knowledge_workers_v1(...)`, and `_run_line_role_direct_workers_v1(...)` all built multiple assignments and then called them one at a time in a plain loop.
- That meant the UI could truthfully say "5 workers assigned" but still lie by implication about concurrency, because only one assigned worker was actually running at a time.
- The durable fix is to launch worker assignments concurrently at the runtime layer and then aggregate results back in assignment order so telemetry, promotion reports, and worker summaries stay stable.
