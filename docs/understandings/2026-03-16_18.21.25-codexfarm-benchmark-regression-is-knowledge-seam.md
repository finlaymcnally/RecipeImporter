The March 16 benchmark run at `data/golden/benchmark-vs-golden/2026-03-16_18.10.25/.../codexfarm` did not regress versus its paired vanilla run, but it did regress badly versus the earlier March 14 codexfarm run preserved in `.history/performance_history.csv`.

The dominant change is the `KNOWLEDGE` vs `OTHER` seam. On 2026-03-14, codexfarm scored `strict_accuracy=0.6152796725784447` with `KNOWLEDGE` precision/recall `0.6741 / 0.6118` and `pred_total=540`. On 2026-03-16, codexfarm dropped to `strict_accuracy=0.5320600272851296` and `KNOWLEDGE` `0.8630 / 0.1059` with `pred_total=73`. Current vanilla and current codexfarm both predict essentially the same amount of `KNOWLEDGE` (`knowledge_pred_total=79` in `line-role-pipeline/knowledge_budget.json`), so the old codex-only advantage is gone.

This lines up with the Phase 4 refactor contract: `docs/plans/phase4.md` explicitly made Stage 7 deterministic labels authoritative for outside-recipe `KNOWLEDGE`/`OTHER`, and `cookimport/staging/stage_block_predictions.py` now adds knowledge labels from `nonrecipe_stage_result.block_category_by_index` with the note `KNOWLEDGE labels were derived from deterministic Stage 7 categories.` The optional knowledge harvest still runs, but it is no longer allowed to relabel benchmark truth. That means codexfarm can no longer buy benchmark score by expanding knowledge coverage the way the old pipeline did.

There is a second, separate recipe-path issue in the same run: `raw/llm/.../recipe_manifest.json` reports `52` `recipe_correction_error` rows rejected as `placeholder_steps_only`. That is a real regression inside `codex-farm-single-correction-v1`, but it is not the primary explanation for the benchmark-score drop. The score delta versus the March 14 codexfarm run is overwhelmingly the disappearance of codex-driven `KNOWLEDGE` labeling.
