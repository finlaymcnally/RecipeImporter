  - "When deciding how per-surface benchmark toggles should map onto existing run settings"

Interactive benchmark setup already flows through `cookimport/cli_ui/run_settings_flow.py`, which resolves a top-tier profile first and then returns a plain `RunSettings`.

For this benchmark surface, the clean mapping is:

- recipe toggle -> `llm_recipe_pipeline` (`off`, `codex-farm-3pass-v1`, `codex-farm-2stage-repair-v1`)
- block-label toggle -> `line_role_pipeline` (`deterministic-v1` when Codex is off, `codex-line-role-v1` when Codex is on)
- block-label toggle also keeps `atomic_block_splitter=atomic-v1` so canonical-text projection still runs through the same benchmark path
- pass4 toggle -> `llm_knowledge_pipeline` (`off` or `codex-farm-knowledge-v1`)

This keeps the chooser additive: start from the existing `codexfarm` or `vanilla` top-tier contract, then patch only the selected benchmark surfaces. No new persisted run-setting field is needed.
