---
summary: "Why upload_bundle_v1 was still talking about pass2/pass3 for a merged-repair benchmark run."
---

`upload_bundle_v1` was still presenting recipe diagnostics through a legacy pass2/pass3 reporting contract even when the underlying recipe-stage topology was no longer a standard live pass2->pass3 sequence.

The merged-repair run exposed the problem because the real signal was only visible indirectly through fields like `pass3_execution_mode=llm_merged_repair` and `pass3_routing_reason=merged_repair_stage`, while overview/index/casebook text still framed those rows as literal pass2/pass3 stages.

The durable fix is to keep the old keys for compatibility while deriving `analysis.recipe_pipeline_context` from observed artifacts: codex recipe pipeline ids, pass2/pass3 call presence, execution modes, and routing reasons. Reviewer-facing output now labels downstream recipe artifacts by legacy field family (`legacy-family:pass2_*`, `legacy-family:pass3_*`) instead of implying any fixed stage count, and only treats `pass2_*` / `pass3_*` as compatibility aliases. When the discovered topology is nonstandard, the bundle marks that explicitly instead of pretending the old stage contract is literal.
