# Oracle Turn-1 Recovery Must Drive Reattach From Saved Conversation URL

For long benchmark browser runs, a stuck Oracle turn 1 is not a generic "still running" case. The common failure shape is `response.incompleteReason=assistant-timeout`: ChatGPT already produced a reply in the saved conversation, but Oracle never finalized the local session metadata.

Two concrete consequences follow:

1. Recipeimport cannot just wait for `oracle_upload_status.json` to flip on its own. The detached auto-follow-up worker needs to detect `assistant-timeout` in Oracle's saved session metadata and actively invoke `oracle session <id>` as a no-new-token recovery step.
2. Oracle reattach must prefer the saved `conversationUrl` directly and must not gate recovery behind sidebar lookup or a "new turn after baseline" heuristic, because those are brittle on long benchmark chats that already have the target answer in the thread.

This was established from the `you-are-reviewing-a-benchmark-294` run: the saved metadata contained the exact `/c/...` URL, the ChatGPT web conversation showed the benchmark reply, and Oracle's own DOM snapshot during reattach could see assistant text, but turn 1 still stayed locally incomplete until the reattach path was hardened.
