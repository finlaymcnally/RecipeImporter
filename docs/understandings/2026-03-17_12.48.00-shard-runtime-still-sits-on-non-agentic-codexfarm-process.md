# Shard Runtime Still Sits On Non-Agentic CodexFarm Process

- RecipeImport's shard runtime currently calls `SubprocessCodexFarmRunner.run_pipeline(...)`, which shells out to `codex-farm process --pipeline ... --in ... --out ... --json`.
- RecipeImport now forces `codex-farm process --workers 1` when the shard runtime asks for `structured_loop_agentic_v1`, so one RecipeImport worker assignment no longer silently expands into several CodexFarm process workers.
- That means the runtime artifacts now line up better with reported `worker_count` and `fresh_agent_count`, but they still do not by themselves prove the report's stronger goal of "load instructions once and reuse one bounded worker session across multiple shards."
- The remaining gap is CodexFarm-side: `process` still runs the structured-output worker loop rather than a durable reused shard session, and RecipeImport's `max_turns_per_shard` value is still planning metadata rather than a transport-enforced turn cap.
