# Shard Runtime Still Sits On Non-Agentic CodexFarm Process

- RecipeImport's shard runtime currently calls `SubprocessCodexFarmRunner.run_pipeline(...)`, which shells out to `codex-farm process --pipeline ... --in ... --out ... --json`.
- In shared CodexFarm, `process` still defaults to `--workers 8` and runs the normal lease-based task worker loop, where each task becomes its own `run_codex_exec(...)` call.
- That means the current RecipeImport runtime artifacts are useful for shard ownership, validation, promotion, and telemetry, but they do not by themselves prove the report's stronger goal of "load instructions once and reuse one bounded worker session across multiple shards."
- Concretely: if RecipeImport wants honest worker-count semantics, it should at least force `codex-farm process --workers 1` per RecipeImport phase worker. If it wants true shard-agent/session reuse, CodexFarm needs a new execution mode beyond today's per-task structured-output worker loop.
