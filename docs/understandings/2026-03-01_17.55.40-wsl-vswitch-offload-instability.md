# Discovery: mirrored networking did not clear host vSwitch instability (2026-03-01)

## What was changed

- Updated `C:\Users\mcnal\.wslconfig` to:
  - `guiApplications=false`
  - `networkingMode=mirrored`
  - `dnsTunneling=true`
  - `autoProxy=true`
  - `autoMemoryReclaim=gradual` (unchanged)
- Ran `wsl --shutdown` to apply settings.

## Verification

- New boot started at `2026-03-01 17:51:56 EST`.
- Inside WSL after restart:
  - Route moved to `eth2` on `10.0.0.0/24`.
  - `/etc/resolv.conf` nameserver is `10.255.255.254` (dns tunneling path).
- So `.wslconfig` network changes were applied.

## Persisting failure signals

- Linux journal in new boot still shows:
  - `WSL (...) ERROR: CheckConnection: getaddrinfo() failed: -5`
  - seen at `17:52:06` and `17:52:47`.
- Windows `Microsoft-Windows-Hyper-V-VmSwitch-Operational` still logs recurring warnings:
  - event `285` every ~1 minute on `WSL (Hyper-V firewall)` host vNIC.
- Windows `System` also shows `Netwtw10` warning `6062 - Lso was triggered` in the same general crash windows.

## Practical interpretation

- WSL-mode tuning alone (including mirrored networking) does not eliminate the instability on this host.
- Most likely trigger remains host networking offload/driver interaction (Wi-Fi + Hyper-V vSwitch path), not QualitySuite workload pressure.
- Non-elevated shell cannot apply adapter offload changes (`Set-NetAdapterAdvancedProperty` returned `Access is denied`).

## Next required remediation (elevated PowerShell)

- Disable offloads on `vEthernet (WSL (Hyper-V firewall))`:
  - `*LsoV2IPv4`, `*LsoV2IPv6`, `*RscIPv4`, `*RscIPv6`
  - `Disable-NetAdapterRsc -Name "vEthernet (WSL (Hyper-V firewall))" -IPv4 -IPv6`
- If instability persists, disable packet coalescing/offload features on the Intel Wi-Fi adapter and update/reinstall Wi-Fi driver.
