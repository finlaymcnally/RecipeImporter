The recipe-correction progress total is one task per `ConversionResult.recipe`, not one task per surviving final recipe draft. `cookimport/llm/codex_farm_orchestrator.py::_build_states(...)` enumerates `result.recipes`, and `_run_single_correction_recipe_pipeline(...)` writes one correction input JSON for each state before invoking CodexFarm.

On the 2026-03-16 benchmark run at `data/output/2026-03-16_18.10.25/single-offline-benchmark/saltfatacidheatcutdown/codexfarm/2026-03-16_18.11.25/`, `group_recipe_spans/.../recipe_spans.json` contains 175 grouped recipe spans. The span metadata shows this is not 175 clean recipe pages: 140 spans have no title block, 133 warn `recipe_span_started_without_title`, and 104 are single-block spans. The first intermediate draft is literally `SALT` with ingredient text `What is Salt?`, which confirms chapter/how-to material is still being promoted into recipe spans.

This means the large task count is primarily a Stage 2/3 grouping problem, not Codex retry fan-out. Codex is faithfully processing the queued recipe spans it was handed, and many of those spans are weak enough that the recipe manifest later rejects them with structural failures such as `placeholder_steps_only`.
