# Shard Runtime Plan Splits Into Spine, Phases, And Cutover

- `docs/plans/2026-03-16_21.26.21-replace-codex-prompts-with-shard-agents.md` already separates the work into one shared runtime milestone, three phase-specific migration milestones, and one cross-cutting cutover milestone.
- The shared runtime/config layer is the dependency spine. It has to land first because line-role, recipe, and knowledge all depend on the same manifest, worker assignment, sandbox, runner-audit, and promotion hooks.
- After that spine exists, the three phase migrations are mostly parallelizable because they have separate ownership seams: line-role in `cookimport/parsing/canonical_line_roles.py`, recipe in `cookimport/llm/codex_farm_orchestrator.py`, and knowledge in `cookimport/llm/codex_farm_knowledge_orchestrator.py`.
- The final preview/`prompts/`/`upload_bundle_v1` cutover is cross-cutting again. It should not be a separate independent project unless it stays tightly coupled to the new runtime telemetry model.
- The clean split shape is therefore: foundation plan first, then up to three phase plans in parallel, then one final observability-and-removal plan.
