# Line-Role Repair Prompts Can Collapse Invalid Shards Into All-Ingredient Output

- The March 19 Salt Fat Acid Heat quality collapse was not a scorer issue. The canonical line-role stage regressed first, and that poisoned downstream canonical-text evaluation.
- The file-backed prompt in `llm_pipelines/prompts/line-role.canonical.v1.prompt.md` and the fallback text in `cookimport/llm/canonical_line_role_prompt.py` both over-anchor on a literal one-row example:
  - `{"rows":[{"atomic_index":123,"label":"INGREDIENT_LINE"}]}`
- In the failing benchmark artifacts under `data/output/2026-03-19_13.22.48/.../line-role-pipeline/runtime/line_role/...`, original shard outputs copied that example almost verbatim, then the repo-owned repair path expanded it into full-coverage but semantically wrong outputs.
- The repair prompt in `cookimport/parsing/canonical_line_roles.py` currently emphasizes exact JSON shape and owned-index coverage, but it does not restate the authoritative shard rows strongly enough. That makes it easy for the model to satisfy validation by repeating one label for every row instead of re-reading the shard.
- Observable evidence from the failed run:
  - `single_offline_summary.md` shows CodexFarm strict accuracy collapsing to about `0.12` while vanilla stays around `0.70`.
  - `wrong_label_lines.sample.jsonl` shows headings, knowledge prose, and instruction lines flipping to `INGREDIENT_LINE`.
  - `repair_last_message.json` files show repaired shards with broad `INGREDIENT_LINE` coverage after invalid initial outputs.
- Practical fix direction:
  - remove or de-emphasize the misleading single-row `INGREDIENT_LINE` exemplar in the line-role prompt surfaces
  - make repair and watchdog-retry prompts restate the actual shard rows and the task as reclassification, not just JSON-shape repair
  - add a regression test that simulates a copied-example invalid response and proves repair does not succeed unless it uses shard evidence
