# Codex Token Cost Postmortem And Mitigation Report

Scope:
- primary run: `data/golden/benchmark-vs-golden/2026-03-18_21.17.59`
- supporting comparison: `data/golden/benchmark-vs-golden/2026-03-18_21.10.12`

Goal:
- explain why recent Codex-backed benchmark runs billed far more tokens than prompt preview predicted
- separate "LLM work that is the point of the program" from "wasted spend caused by runtime behavior"
- propose mitigations that keep the LLM-centric architecture intact

## Context And Constraints

This project intentionally uses Codex-backed correction rather than deterministic-only recovery.

That is the right high-level choice here:
- deterministic correction appears to have a hard ceiling around `~50%` accuracy
- the point of the program is to use an LLM to make cookbook cleanup practical
- the operator has an OpenAI Pro subscription, so `codex exec` is economically attractive relative to pay-per-token API fanout
- Codex is also useful because it is already good at moving through files, payloads, and structured editing-style tasks

So the conclusion of this report is **not** "stop using LLMs" or "replace the pipeline with deterministic logic."

The real problem is narrower:
- the current `codex exec` runtime sometimes behaves like an unrestricted coding agent session
- these benchmark tasks are closer to strict structured classification / transformation than to open-ended coding
- once a shard slips into shell/Python/tool-loop behavior, token cost explodes

## What "Direct Exec" Means In This Repo

In current repo language, the active recipe, knowledge, and line-role surfaces use one direct `codex exec` structured call per shard.

Relevant code/docs:
- `cookimport/llm/codex_exec_runner.py`
- `cookimport/parsing/canonical_line_roles.py`
- `docs/10-llm/10-llm_README.md`

Important clarification:
- "direct exec" here means local `codex exec` subprocess execution, not a pay-per-token API fanout
- the concern is not that the repo uses Codex
- the concern is that the current Codex runtime can behave like a shell-enabled agent instead of a narrowly constrained structured-output worker

## Executive Summary

For the `2026-03-18_21.17.59` `saltfatacidheatcutdown` run, the cost blowup came from four interacting causes:

1. most billed tokens were not visible prompt or response text
2. expensive shards entered agent-style shell/Python loops
3. several of those expensive shards still failed validation, so the spend was largely wasted
4. prompt preview undercounted the real billing surface, especially for line-role

The total billed cost was:
- `5,670,556` tokens across `15` calls

But only:
- `105,528` tokens were visible input (`1.9%`)
- `53,206` tokens were visible output (`0.9%`)

The rest was:
- `3,034,894` wrapper-overhead tokens (`53.5%`)
- `2,476,928` cached-input tokens (`43.7%`)

So only about `2.8%` of the entire bill was visible prompt + response text.

At least `3,127,458` tokens, about `55.2%` of the whole run, landed on shards that ultimately failed validation:
- invalid knowledge shards: `1,541,886`
- invalid line-role shard: `1,585,572`

That means the main problem was not "the model thought very hard and produced a great answer."

The main problem was:
- hidden runtime transport/overhead
- tool-enabled Codex agent loops
- expensive invalid outputs

## Stage-By-Stage Behavioral Analysis

## Recipe Correction: Mostly Fine

Recipe correction does not look like the main culprit.

Observed runtime pattern:
- most recipe shards had `0` command executions
- most had `1` reasoning message
- only one shard showed even minor command behavior

Recipe shard usage:
- four shards were in the `~22k` to `~30k` range
- one shard reached `107,826`

That is still not cheap, but it is small relative to the knowledge and line-role failures.

Conclusion:
- recipe correction is not where the run went off the rails
- it has hidden overhead, but not pathological agent behavior

## Knowledge: One Catastrophic Shard Plus Three Integrity Failures

Knowledge is the clearest example of "expensive work that did not survive validation."

Promotion summary:
- validated shards: `1`
- invalid shards: `4`

Failure summary:
- `ks0000`: `missing_owned_chunk_results`
- `ks0001`: `missing_owned_chunk_results`, `unexpected_chunk_results`, `snippet_evidence_out_of_surface`
- `ks0002`: `missing_owned_chunk_results`
- `ks0003`: `missing_owned_chunk_results`

### The Catastrophic Knowledge Shard: `ks0001`

Shard:
- `saltfatacidheatcutdown.ks0001.nr`

Cost:
- `1,356,369` total tokens

Behavior:
- `6` command executions
- `5` reasoning messages

Reasoning summaries in the event stream show the model:
- devising heuristics for classification and snippet extraction
- trying to preserve a huge inline JSON blob through heredoc/Python tricks
- repeatedly thinking about how to parse and reconstruct the payload in the shell environment
- eventually giving up and returning a fallback `processing_error`

The command log shows exactly what went wrong:
- the model tried to pass a giant JSON payload through inline shell/Python commands
- one of those commands ended in `JSONDecodeError`
- the final answer was not faithful shard output; it was a synthetic fallback object

This is the key point:
- the model was not mostly spending tokens "reasoning about the cookbook"
- it was spending tokens struggling with payload transport inside a coding-agent environment

### The Other Knowledge Failures

The remaining bad knowledge shards were cheaper, but still wasteful:
- `ks0000`: `54,577`
- `ks0002`: `73,052`
- `ks0003`: `57,888`

These mostly failed by dropping owned chunks or returning effectively empty coverage.

Interpretation:
- the knowledge contract is achievable, because `ks0004` validated
- but several shards degraded under size/complexity pressure
- validation caught the bad outputs correctly, but only after billing the full run

## Line Role: The Biggest Cost Center And The Clearest Agent-Loop Pattern

Line-role was the largest stage by total billed tokens and showed the strongest agent-style behavior.

Promotion summary:
- validated shards: `4`
- invalid shards: `1`

### Shard Costs

- `line-role-canonical-0001-a000000-a000353`: `137,730`
- `line-role-canonical-0002-a000354-a000707`: `713,974`
- `line-role-canonical-0003-a000708-a001061`: `1,249,383`
- `line-role-canonical-0004-a001062-a001415`: `166,088`
- `line-role-canonical-0005-a001416-a001767`: `1,585,572`

Two facts matter:

1. the most expensive shard was `0005`
2. `0005` also had the smallest request input file, about `48,823` bytes

So line-role cost was clearly not a simple function of payload size.

### Command/Reasoning Counts

- shard `0001`: `3` commands, `0` reasoning messages
- shard `0002`: `9` commands, `6` reasoning messages
- shard `0003`: `16` commands, `7` reasoning messages
- shard `0004`: `4` commands, `0` reasoning messages
- shard `0005`: `20` commands, `1` reasoning message

The expensive shards were the ones where Codex turned the task into a mini agent loop.

### What The Model Was Doing

The line-role reasoning logs show the model:
- noticing truncation when printing the task file
- deciding to read the JSON with Python instead
- inventing heuristics for headings, instructions, and knowledge lines
- building scripts to inspect candidate labels
- generating scripts to emit the full final JSON to stdout because it believed file writes were unavailable
- iterating on verification logic before returning output

This is classic coding-agent behavior, and it is not what this task economically wants.

### The Worst Line-Role Failure

The worst invalid line-role shard:
- `line-role-canonical-0005-a001416-a001767`
- billed `1,585,572` tokens
- ran `20` commands
- still failed validation

Failure class:
- `proposal_validation_failed`
- hundreds of `invalid_label:*` errors

So even after a massive amount of work, the shard did not survive promotion.

## Why Prompt Preview Missed So Badly

The preview problem is real, but it is secondary to the runtime problem.

For line-role in particular:
- preview mostly sees the short wrapper prompt and the nominal task file
- actual billing captures a much larger hidden surface as wrapper overhead and cached input

This was already visible in the earlier preview comparison:
- line-role preview looked relatively cheap
- actual line-role cost was `~30x` higher on `saltfatacidheatcutdown`

The structural reason is:
- preview models visible prompt/input text
- actual `codex exec` billing includes runtime transport surfaces that are not obvious from the local wrapper text
- once the agent starts issuing shell/Python commands, prompt preview has no way to predict that secondary cost explosion

So preview undercount was real, but even a perfect preview would not fix the actual waste.

## Root Cause Model

The run cost too much because the system currently combines:

1. a shell-capable coding agent runtime
2. large structured-data transformation tasks
3. post-hoc validation that rejects invalid outputs only after the expensive run finishes
4. shard sizing that still allows some cognitively unstable tasks

That produces a bad failure pattern:

1. the model receives a large structured task
2. instead of staying in one-pass structured-output mode, it starts acting like a local coding agent
3. it uses commands and ad hoc scripts to inspect or reconstruct payloads
4. hidden runtime overhead and cached input dominate the bill
5. the final output may still fail validation
6. the run pays the cost anyway

The problem is therefore **not** "using Codex was a mistake."

The problem is:
- using Codex in too agentic a mode for classification-style shard work

## What Should Not Change

The following are not recommended:

- removing the LLM stages
- replacing the system with deterministic-only correction
- abandoning `codex exec` solely because it is not the API
- treating subscription economics as irrelevant

Those would fight the actual product goal.

The right target is:
- keep Codex
- keep LLM correction
- preserve the economic benefit of the subscription
- remove the avoidable waste caused by tool loops, oversized shards, and invalid-output reruns

## Mitigation Plan

## 1. Constrain Runtime Behavior Without Removing The LLM

The highest-leverage change is to keep Codex but make these shard tasks behave more like strict structured transforms and less like open-ended coding sessions.

Desired outcome:
- no shell/Python loops for normal line-role and knowledge shards
- one-pass or near-one-pass JSON production

This can be pursued in several layers:
- prompt-level prohibition against command use
- runtime-level detection of command-heavy pathological shards
- if possible, future runner tightening so shard tasks execute in a non-tool-using structured mode

The core principle is:
- preserve intelligence
- reduce agency

## 2. Add Cheap Repair Passes Instead Of Full-Price Waste

Current behavior:
- full shard runs
- validator rejects invalid output
- all spend is already sunk

Recommended behavior:
- if output is structurally close, run a tiny repair prompt that includes:
  - original shard ids
  - previous invalid output
  - validation errors
  - instruction to return corrected JSON only

Examples where repair is likely much cheaper than redoing the whole shard:
- missing one owned knowledge chunk
- empty result array with intact top-level structure
- line-role label format mismatch

This is especially important because many failures were integrity problems, not total semantic failures.

## 3. Reduce Worst-Case Shard Size And Complexity

The current runs show that "number of bytes" is not the full story, but shard complexity still matters.

Knowledge:
- bundle by bytes, block count, and quote density
- avoid shards that combine many tricky quote-heavy prose chunks

Line role:
- lower max rows per shard
- treat front matter / heading-dense regions as higher-risk shards

The goal is not to create many tiny calls blindly.
The goal is to reduce the chance that one shard becomes a million-token disaster.

## 4. Tighten Prompts Around Forbidden Failure Shapes

Knowledge prompts should explicitly forbid:
- invented chunk ids
- empty `r` when input `c` is non-empty
- synthetic fallback objects like `processing_error`
- evidence references outside owned surface

Line-role prompts should explicitly forbid:
- any output that does not match the allowed label schema exactly
- prose commentary
- command use

Both prompts should:
- demand final JSON only
- include one valid minimal example
- say that uncertainty must still preserve exact owned id coverage

## 5. Stop Inviting Coding-Agent Behavior In Prompt Wording

If the prompt reads like a task a programmer should inspect/debug locally, Codex will often do exactly that.

The model should instead be told:
- the payload already contains everything needed
- do not inspect the environment
- do not run tools
- return the final structured output directly

This is a subtle but important difference in prompt framing.

## 6. Make Wasted Spend Visible In Reports

Current cost reporting shows totals well, but the most decision-useful number from these failures was:
- how much was spent on invalid shards

Recommended additions to summary artifacts:
- invalid-shard token total
- invalid-shard percentage of stage total
- command count per shard
- reasoning-message count per shard
- "pathological run" flag when commands or invalid spend cross a threshold

That would have made this postmortem much faster.

## 7. Treat Line-Role Preview As Advisory Until Path Costs Are Modeled Better

Preview is still useful for relative comparison, but current actual billing makes it unsafe as a hard budget predictor for line-role and other expensive direct-exec surfaces.

Recommended near-term behavior:
- add explicit warning language for direct-exec/path-heavy surfaces
- highlight historical actual-vs-preview multipliers
- avoid presenting preview as a close estimate where runtime overhead dominates

## 8. Add Runaway-Shard Guardrails

The repo should eventually detect pathological runs early.

Useful tripwires:
- too many command executions in a shard
- too many event rows
- tokens crossing a hard per-shard ceiling
- repeated tool-use attempts after prompt says not to use tools

Ideal behavior:
- abort or downgrade the shard before it burns another huge block of tokens

Even if live mid-run abort is hard with the current subprocess seam, these metrics should at least be surfaced immediately so future experiments fail faster.

## Suggested Implementation Order

If effort is limited, the best order appears to be:

1. tighten knowledge and line-role prompts against tool use and invalid fallback shapes
2. reduce shard size / complexity on the risky surfaces
3. add cheap repair passes for validation failures
4. add better run summaries: invalid spend, command counts, pathological flags
5. improve runtime control over command-heavy shards if the current `codex exec` seam can support it

This sequence keeps the current architecture and should cut waste fastest.

## Bottom Line

Using Codex here is not the mistake.

Given the subscription economics and the quality ceiling of deterministic recovery, leaning on LLM correction is rational.

What went wrong is that the current runtime let Codex behave too much like a shell-enabled coding agent on tasks that should be strict structured-output transforms.

That produced:
- huge hidden billing overhead
- expensive shell/Python loops
- many tokens spent on outputs that later failed validation

The right response is not to remove the LLM.

It is to keep Codex, keep the LLM stages, and make the shard runtime much less willing to burn subscription-backed tokens on agentic detours.
