# Dashboard Token Row Vs Prompt Budget

- Case studied: `data/golden/benchmark-vs-golden/2026-03-16_18.10.25/single-offline-benchmark/saltfatacidheatcutdown/codexfarm`
- Dashboard row (`.history/performance_history.csv`) stored:
  - `tokens_total=460774`
  - `tokens_input=391085`
  - `tokens_cached_input=288000`
  - `tokens_output=69689`
- Dashboard `All token use` intentionally displays a cached-discounted total, not raw `tokens_total`:
  - formula: `(input - cached_input) + 0.1 * cached_input + output`
  - for this row that is about `201,574`, which matches the `201k total | 391k in | 69k out` cell
- `prompt_budget_summary.json` for the same run was larger because it included multiple live Codex stages:
  - recipe correction: `460,774`
  - knowledge: `380,440`
  - combined raw total: `841,214`
- `prompt_preview_budget_summary.json` was larger again because it was a preview estimate and included 45 line-role prompts (`~804,745` estimated tokens for line-role alone), producing an `~1.48M` total warning
- Actual line-role live telemetry for this run was much smaller than preview implied:
  - `prompts/line-role-pipeline/telemetry_summary.json` shows 2 actual Codex attempts with `17,603` and `7,897` `tokens_total`
- Practical takeaway:
  - dashboard row = one row's cached-discounted cost proxy
  - CSV `tokens_total` = one row's raw total
  - `prompt_budget_summary.json` = multi-stage live totals
  - `prompt_preview_budget_summary.json` = heuristic worst-case style estimate from rendered prompts, not actual billed telemetry
