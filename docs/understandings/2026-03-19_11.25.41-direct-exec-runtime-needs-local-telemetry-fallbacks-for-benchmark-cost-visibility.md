
# Direct Exec Runtime Needs Local Telemetry Fallbacks For Benchmark Cost Visibility

- The Salt Fat Acid Heat single-offline Codex run had real token totals in:
  - `raw/llm/<slug>/recipe_phase_runtime/telemetry.json`
  - `raw/llm/<slug>/knowledge/telemetry.json`
  - `line-role-pipeline/runtime/line_role/telemetry.json`
- Its archived `prompts/full_prompt_log.jsonl` still had `request_telemetry: null` for recipe/knowledge rows because prompt-artifact reconstruction only joined the global `var/codex_exec_activity.csv` path and did not fall back to stage-local runtime telemetry.
- The run-root `prompt_budget_summary.json` still had the correct aggregate stage token totals, but `upload_bundle_v1` only used that fallback when there were zero call rows, not when there were call rows with missing runtime/token coverage.
- Direct `codex exec` runtime rows also did not persist `duration_ms`, so prompt budgets and bundles could not report duration coverage even though token totals were present.

Current fix:

- `cookimport/llm/codex_exec_runner.py` now writes `duration_ms`, `started_at_utc`, and `finished_at_utc` into direct shard telemetry rows.
- `cookimport/llm/prompt_artifacts.py` now backfills recipe/knowledge prompt-log rows from stage-local `telemetry.json` keyed by runtime shard when the CSV join is absent.
- `scripts/benchmark_cutdown_for_external_ai.py` now normalizes aggregate prompt-budget stage names and uses prompt-budget stage totals when they provide more runtime/token signal than row-level prompt telemetry.
