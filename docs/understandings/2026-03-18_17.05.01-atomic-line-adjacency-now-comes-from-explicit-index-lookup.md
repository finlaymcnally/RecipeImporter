# Atomic Line Adjacency After The Candidate Cleanup

`cookimport/parsing/recipe_block_atomizer.py::AtomicLineCandidate` is now a single-line fact only: recipe/block provenance, `atomic_index`, `text`, `within_recipe_span`, and `rule_tags`.

Anything that needs neighboring text should now use the parser-owned helpers in the same module:

- `build_atomic_index_lookup(candidates)` to build the ordered lookup once
- `get_atomic_line_neighbor_texts(candidate, by_atomic_index=...)` to read `prev` / `next` text explicitly

Canonical line-role uses that same seam in two places:

- `cookimport/llm/canonical_line_role_prompt.py` for selective inline `ctx:` windows
- `cookimport/parsing/canonical_line_roles.py::_canonical_candidate_fingerprint(...)` for adjacency-sensitive cache identity

That is the rule after this refactor: candidate objects stay local to one row, and ordered context is a separate input.
