# Recipe Correction Review

Fresh preview inspection on `thefoodlabcutdown` and `saltfatacidheatcutdown` suggests the recipe payload itself is not the main problem.

The model already gets:

- grounded source rows in `ev`
- flattened source text in `txt`
- the deterministic intermediate object in `h`
- explicit owned ids in `ids`

That is enough evidence for a strong Codex pass on honest recipe candidates.

The bigger weakness is the contract around that payload:

- the prompt opens by calling the shard "authoritative recipe spans"
- each owned `rid` must return a full recipe-shaped object
- there is no explicit instruction for "this owned span is not actually a recipe" or "this is too fragmentary to repair into a trustworthy recipe"

So on deterministic-only previews, where the recipe candidates are known to be unrepresentatively bad, Codex is asked to do the wrong kind of recovery work. It can disagree with `h`, but it cannot clearly decline the candidate shape.
