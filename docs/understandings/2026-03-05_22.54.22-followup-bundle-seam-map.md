## Key Findings

- The current checked-in follow-up packet under `data/golden/benchmark-vs-golden/2026-03-04_20.33.53/single-profile-benchmark/followup_data/` is only three files: `index.json`, `codexfarm_root_cause_packet.json`, and `codexfarm_root_cause_packet.md`. It is a small hand-shaped summary over six fixed cases, not a reusable exporter contract.

- The authoritative existing exporter seam is `scripts/benchmark_cutdown_for_external_ai.py`, especially `build_upload_bundle_for_existing_output(...)`. Interactive benchmark flows call this through `cookimport/cli.py::_write_benchmark_upload_bundle(...)`, so new deterministic follow-up tooling should reuse that source-of-truth instead of rebuilding bundle discovery from scratch.

- `upload_bundle_v1` already preserves most of the evidence the follow-up work needs: triage packet rows, per-recipe breakdown, stage-separated comparison, full payload retention, `_upload_bundle_derived/...` fallback rows, and 100% critical row-locator coverage in the sample bundle. The missing piece is deterministic selection/export/audit commands, not more evidence collection prose.

- The current gap for provenance debugging is that human-facing navigation summaries are not the same thing as a complete machine API. New follow-up tooling should resolve payload rows from canonical payload-path metadata and discovered run artifacts directly, rather than assuming `upload_bundle_index.json -> navigation.row_locators` contains every analysis entry.

- The weird candidate-label examples in the current follow-up packet cannot be blamed on the model without a dedicated audit. The available artifacts suggest at least four plausible causes: prompt response mismatch, candidate snapshot mismatch, postprocess rewrite, or exporter/join mismatch.
