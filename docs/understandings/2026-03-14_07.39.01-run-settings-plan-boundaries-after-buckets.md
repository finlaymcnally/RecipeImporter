The original `docs/plans/2026-03-13_23.27.36-run-settings-product-contract.md` started as an umbrella plan, but that became too broad once implementation began.

The current ownership split is:

- `docs/plans/2026-03-13_23.26.13-bucket1-hardcode-run-settings.md` owns Bucket 1. That is the fixed-behavior and retired-key migration for fake settings such as matcher/pass/debug seams.
- `docs/plans/2026-03-13_23.26.22-run-settings-bucket2-internalization.md` owns Bucket 2. That is the hide/internalize-without-deleting migration for parser/scoring/OCR lab controls.
- `docs/plans/2026-03-13_23.27.36-run-settings-product-contract.md` should now start from the Bucket 1 + Bucket 2 result and only cover the follow-on contract work: explicit operator/benchmark-lab helper surfaces, smaller entrypoint shapes, operator-first summaries/manifests, and docs cleanup.

Short rule: if the work is deciding whether a setting is retired versus internal versus public, it belongs in Bucket 1 or Bucket 2. If the work is about how the smaller post-Bucket-2 contract is presented across CLI/help/docs/manifests/analytics, it belongs in the product-contract plan.
