## Audit method

This audit intentionally separates three different notions of completion:

- `runtime implemented`: behavior exists in active code paths.
- `plan/task tracked complete`: newer `docs/plans/*.md` or `docs/tasks/*.md` files mark milestones complete.
- `OG checklist complete`: original `docs/plans/OGplan/*.md` checklist items are checked in-place.

The OG checklists are often stale archives, so completion is judged primarily by runtime + tests, then cross-checked against newer plan/task docs.

Evidence sources used:

- Runtime code surfaces under `cookimport/`.
- Tests under `tests/` with focused slices for each feature family.
- Newer implementation-state docs under `docs/plans/*.md` and `docs/tasks/*.md`.

## Important interpretation note

A frequent mismatch in this repository is:

- OG archive checklists remain unchecked, while
- newer active plan/task docs and runtime code/tests show delivered behavior.

Use this order of precedence when deciding “is it done?”:

1. Runtime code path present and reachable.
2. Focused tests for that behavior pass.
3. Active `docs/plans/*.md` or `docs/tasks/*.md` reflects implemented state.
4. OG checklist tick marks (lowest trust; often archival/stale).
