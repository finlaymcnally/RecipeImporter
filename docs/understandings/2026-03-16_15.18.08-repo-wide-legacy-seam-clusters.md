The remaining legacy code is no longer one giant runtime backbone. It clusters into a few distinct seam types.

- Internal pass-slot naming that still lives inside current code paths, especially recipe-correction internals such as `_RecipeState` and related helpers in `cookimport/llm/codex_farm_orchestrator.py`.
- Compatibility readers and translators that exist to read or reinterpret old artifacts, such as `cookimport/bench/upload_bundle_v1_existing_output.py`, `cookimport/bench/followup_bundle.py`, `cookimport/labelstudio/export.py`, and dashboard/history fallbacks in `cookimport/analytics/dashboard_collect.py`.
- Current algorithms that are still named `legacy` even when they are the only supported or preferred implementation, such as line-role prompt format fallback in `cookimport/parsing/canonical_line_roles.py`, canonical alignment strategy naming in `cookimport/bench/eval_canonical_text.py`, and several parser/config values.
- Compatibility alias writers that keep publishing old shapes for downstream consumers, such as draft alias normalization in `cookimport/staging/writer.py`.

The purge plan needs to treat those seam types differently.

- Pass-slot internals and alias writers should usually be renamed or removed directly.
- Compatibility readers should be deleted once tests/fixtures are regenerated.
- `legacy` names that actually mean “current algorithm” need a canonical rename, not a silent delete.
- Historical dashboards and benchmark roots are part of the same purge if the goal is truly “all legacy code,” because those readers are one of the biggest remaining seam families in active code.
