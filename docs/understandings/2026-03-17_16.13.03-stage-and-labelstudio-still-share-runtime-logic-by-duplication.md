# Shared-runtime duplication note

The repo already has one good shared seam in `cookimport/staging/import_session.py`, but several important runtime decisions are still duplicated across top-level flows instead of being owned by one service/module.

Concrete examples from this review:

- Split-job planning exists in both `cookimport/cli.py:_plan_jobs(...)` and `cookimport/labelstudio/ingest.py:_plan_parallel_convert_jobs(...)`.
- Effective worker rules live in `cookimport/config/run_settings.py:compute_effective_workers(...)`, but the surrounding split/extractor behavior still has to stay manually aligned with both orchestration call sites.
- `cookimport/config/CONVENTIONS.md` explicitly documents multi-file coordination rules like “update both split planners together” and “wire new run settings through stage and benchmark prediction paths.”

Practical consequence:

- The intended architecture is shared, but the implementation seam is still “remember the other place.”
- That raises supervision cost for humans and AI because the safe change set is broader than the local module suggests.
