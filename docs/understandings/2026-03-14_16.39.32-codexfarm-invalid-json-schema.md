---
summary: "Why the merged-repair CodexFarm benchmark failed with invalid_json_schema."
---

The `codex-farm-2stage-repair-v1` benchmark path failed before model generation. The merged repair pipeline schema at `llm_pipelines/schemas/recipe.merged-repair.v1.output.schema.json` defines `ingredient_step_mapping_reason` in `properties` but does not include it in the top-level `required` array.

Current Codex structured output validation rejected that schema for every task with:
`Invalid schema for response_format 'codex_output_schema' ... Missing 'ingredient_step_mapping_reason'.`

Evidence:
- Schema file: `llm_pipelines/schemas/recipe.merged-repair.v1.output.schema.json`
- Task trace: `data/golden/benchmark-vs-golden/2026-03-14_16.36.00/.../pass2_schemaorg/out/.codex-farm-traces/...trace.json`

Effect:
- Pass1 completed.
- Merged repair pass started 23 tasks.
- All 23 tasks failed at request validation, so no merged repair outputs were written.

Broader note:
- The legacy final-draft schema (`llm_pipelines/schemas/recipe.final.v1.output.schema.json`) had the same top-level optional-field pattern, so the old compact pass3 path was also vulnerable to the same Codex structured-output rule.

Fix direction:
- Keep `ingredient_step_mapping_reason` nullable, but include it in `required` so Codex can enforce the schema.
- Prompts should tell Codex to always emit the key and use `null` when no reason applies.
