# Foundation Needs Central Pipeline-Id Normalization Before Phase Cutovers

The runtime foundation can switch `RunSettings` and the shared decision seams to shard-v1 ids now, but the repo still contains many old literal comparisons across CLI copy, analytics, benchmark helpers, and review/export surfaces.

The durable foundation move is therefore:

- normalize old ids centrally in `cookimport/config/run_settings.py`
- update only the code that must understand the normalized values right now (`codex_decision`, `cli` normalization, line-role routing, stage observability, and the local interactive menu)
- leave broader wording/export churn for the later observability/removal plan instead of forcing a repo-wide copy edit during the foundation milestone

That keeps the new shared runtime usable and the later child plans unblocked without pretending the whole product surface has already been migrated.
