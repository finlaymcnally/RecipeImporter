# Recipe rejection must prune the updated recipe list

The recipe correction stage exposes two different promotion surfaces:

- per-recipe override dictionaries (`intermediate_overrides_by_recipe_id`, `final_overrides_by_recipe_id`)
- the returned `updated_conversion_result.recipes` list

The important downstream behavior is in `cookimport/staging/writer.py`: if a recipe id has no final-draft override, the writer deterministically rebuilds a final draft from whatever recipe candidate is still present in `results.recipes`.

That means a `fragmentary` or `not_a_recipe` result is not truly "skipped" just because the stage omits overrides. If the orchestrator leaves the original deterministic candidate in `updated_conversion_result.recipes`, the writer will quietly resurrect it.

So the explicit bad-candidate path must do two things together:

- keep the rejected proposal and audit/report artifacts
- prune that recipe id from `updated_conversion_result.recipes`

Otherwise the contract says "not promoted" while the final output still contains the recipe.
