Plan-level gap audit from code read of `cookimport/cli.py` and current tests:

1. `_run_with_progress_status(...)` is not just a renderer; it owns ETA/rate sampling, plain/live mode selection, worker side-channel decoding (`parse_worker_activity(...)`), and processing-timeseries writing. The migration should separate state/render concerns from this runtime control loop so ETA/timeseries behavior does not regress.
2. Stage path currently emits `processing_timeseries.jsonl` snapshots from a different state source (Rich `Progress` + `worker_status` map) and has merge-phase worker updates from main process callbacks. Stage migration needs explicit parity requirements for timeseries fields and merge-status visibility, not only console snapshot parity.
3. Existing tests cover many all-method and `_run_with_progress_status` behaviors, but there is no dedicated stage dashboard parity test asserting shared-core snapshot shape. Add tests that lock expected stage snapshot structure and ensure worker reset/dedupe behavior still works in plain and live modes.
