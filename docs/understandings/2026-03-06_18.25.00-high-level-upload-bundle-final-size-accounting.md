# High-level upload bundle final size accounting

- High-level group bundles cannot get an exact `final_bundle_bytes` number from source-file sizes alone. The final index and overview include the size metadata themselves, so the bundle writer has to measure the three written files and then sync that number back into the payload/index/overview.

- Buffering payload rows in memory before writing makes the trim step straightforward. The builder can omit heavy first-pass artifacts, trim optional rows against a payload budget, rebuild row numbers once, and only then emit the final three files.

- `prediction-run/prompt_budget_summary.json` is a good compact replacement for raw `full_prompt_log.jsonl` in high-level bundles: it keeps per-pass runtime/token orientation without carrying the huge prompt body.

- If the curated bundle ends up too small, the safest bytes to add back are medium-sized context artifacts rather than the raw prompt dump itself. In the March 6 run, adding `prompts/prompt_request_response_log.txt`, `prediction-run/extracted_archive.json`, and `prediction-run/line-role-pipeline/extracted_archive.json` per run moved the group bundle from about `8.3 MB` to about `27.0 MB` while still staying under the 30 MB cap.
