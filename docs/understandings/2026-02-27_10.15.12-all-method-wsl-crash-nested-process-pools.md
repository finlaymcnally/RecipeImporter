# All-Method WSL Crash: Nested Pool Oversubscription

Artifacts inspected:

- `data/golden/benchmark-vs-golden/2026-02-26_23.01.04/all-method-benchmark/thefoodlabcutdown/.scheduler_events/config_*.jsonl`
- `data/output/2026-02-26_23.01.04`
- `cookimport.json`
- `cookimport/cli.py` (`_resolve_all_method_scheduler_runtime`, `_run_all_method_benchmark`)
- `cookimport/labelstudio/ingest.py` (split conversion pool)

Observed run state:

- All configs start nearly simultaneously (`config_started` around `2026-02-27T04:01:14Z`).
- Many configs immediately enter split scheduling (`split_wait_started`), some acquire slots (`split_active_started`).
- No `split_active_finished`, no `post_started`, no `evaluate_started`, and no report artifacts were written.

Concurrency math from current settings:

- `all_method_max_inflight_pipelines=4`, `all_method_max_split_phase_slots=4`, `all_method_smart_scheduler=true`.
- Smart scheduler can raise `effective_inflight_pipelines` above configured inflight.
- Each active config can run split conversion with up to `max(workers, epub_split_workers)` workers.
- Current `cookimport.json` sets `workers=10` and `epub_split_workers=10`.

Result: up to roughly `split_slots * split_workers = 4 * 10 = 40` split worker processes concurrently for one source, plus config-worker processes and parent processes. This is enough to trigger host pressure and can crash WSL on memory-constrained setups.

Implementation note:

- Added a runtime guard in `cookimport/cli.py` so all-method caps split workers per active config from CPU+memory budgets before each `labelstudio_benchmark(...)` call.
- Scheduler summaries now include `split_worker_cap_per_config`, `split_worker_cap_by_cpu`, and `split_worker_cap_by_memory` for visibility.
