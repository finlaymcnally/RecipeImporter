# Shard-v1 Cutover Must Update Preview, `prompts/`, And Upload Bundle Together

- The shard-v1 refactor changes the runtime shape, not just the execution internals. If only staged outputs are updated, the main human-facing review/debug surfaces drift and become misleading.
- Three surfaces have to move together: zero-token `prompt_preview`, benchmark-level `prompts/` exports, and `upload_bundle_v1`.
- `docs/10-llm/10-llm_README.md` treats prompt/debug artifacts such as `prompts/full_prompt_log.jsonl`, `prompt_request_response_log.txt`, `prompt_type_samples_from_full_prompt_log.md`, and `thinking_trace_summary.*` as stable evidence seams.
- `docs/07-bench/07-bench_README.md` treats `upload_bundle_v1` as the stable external review packet, with adaptation/render seams in `cookimport/bench/upload_bundle_v1_existing_output.py` and `cookimport/bench/upload_bundle_v1_render.py`.
- Under shard-v1, these surfaces should stop implying "one prompt equals one task" and instead expose phase, worker, shard, owned IDs, and normalized telemetry while preserving the current stage-backed contract.
