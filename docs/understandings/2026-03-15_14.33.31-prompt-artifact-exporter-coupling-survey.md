summary: "Current prompt artifact export is still discovery-coupled to CodexFarm raw layout even after dynamic stage labels were added."
read_when:
  - "When planning a prompt artifact exporter refactor before changing cookbook staging or bundling"
  - "When deciding which parts of prompt artifact export should become adapters versus stable contracts"
---

The current prompt artifact exporter in `cookimport/cli.py` improved one visible hardcoding seam by deriving reviewer-facing stage labels dynamically, but the exporter still mixes three different jobs in one place:

- discovering which prompt stages ran and where their files live,
- normalizing raw prompt inputs, outputs, telemetry, and traces into `full_prompt_log.jsonl` rows,
- rendering human-facing prompt artifacts such as sample markdown and category files.

That means a large cookbook staging or bundling refactor would still have to edit exporter internals unless the code is split behind an adapter boundary.

The stable downstream contract is already mostly visible:

- `prompts/full_prompt_log.jsonl` is the source-of-truth prompt artifact,
- run manifests point at prompt artifacts through stable keys like `codexfarm_full_prompt_log_jsonl`,
- cutdown and review tooling mostly consume those files rather than exporter internals.

The brittle part is discovery. `_build_codex_farm_prompt_response_log()` still assumes current CodexFarm layout through hardcoded stage directories and manifest names, while some downstream tools still special-case convenience files like `prompt_task4_pass4_knowledge.txt`.

The clean refactor boundary is:

- adapters discover stages from a run layout and emit normalized stage descriptors,
- a collector builds normalized prompt call rows,
- renderers write `full_prompt_log.jsonl` and the human-facing convenience files.

If that boundary exists, the upcoming cookbook refactor only needs a new discovery adapter instead of another exporter rewrite.
