The codebase already reflects the new refactor architecture in the main stage path, but the remaining `legacy` and `compatibility` references are not evenly distributed.

The highest-value removals are not in one place. They cluster in five buckets:

1. Label-first bridge code that still advertises itself as compatibility, especially `LabelFirstCompatibilityResult`, `build_label_first_compatibility_result(...)`, and the continued population of `ConversionResult.non_recipe_blocks` in `cookimport/parsing/label_source_of_truth.py` and `cookimport/staging/import_session.py`.
2. Hidden run-setting and CLI surfaces that still default to legacy values or expose `legacy` as a valid choice even though the refactor docs say the new paradigm is the runtime. The main offenders are `cookimport/config/run_settings.py`, `cookimport/cli.py`, and `cookimport/labelstudio/ingest.py`.
3. Prompt/bundle tooling that still reconstructs work through pass-slot compatibility language, especially `cookimport/llm/prompt_artifacts.py` and `cookimport/bench/upload_bundle_v1_model.py`.
4. Benchmark outputs that still write alias artifact names such as `missed_gold_spans.jsonl` and `false_positive_preds.jsonl` in `cookimport/bench/eval_stage_blocks.py`, `cookimport/bench/eval_canonical_text.py`, and the CLI artifact manifests that publish those names.
5. Historical readers for analytics, follow-up bundles, and old helper signatures, such as the legacy CSV/timestamp fallbacks in `cookimport/analytics/`, the legacy follow-up case loader in `cookimport/bench/followup_bundle.py`, and the TypeError-based helper fallbacks in `cookimport/cli.py`.

One important nuance: some places use the word `legacy` to name the current algorithm, not just an old branch. The canonical-text aligner in `cookimport/bench/eval_canonical_text.py` is the clearest example. That seam should still be cleaned up, but by simplifying or renaming the contract, not by deleting the algorithm outright.
