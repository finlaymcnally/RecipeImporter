# Discovery

- `_run_all_method_benchmark(...)` already computes `evaluate_active` from `.scheduler_events` and uses it to build `dynamic_eval_tail = min(eval_tail_cap, evaluate_active)`.
- Admission is still constrained by a second guard: `heavy + wing >= scheduler_base_target` (`split_slots + wing_target`), which can stop new submits even when evaluation is active and pending configs remain.
- The live scheduler snapshot currently omits eval-phase visibility (`scheduler heavy ... | wing ... | active ... | pending ...`) even though eval counts are available internally.

# Practical Implication

- Eval-tail underutilization can happen as a scheduler-admission artifact, not because eval phase detection is missing.
- Throughput tuning should focus on how prewarm guardrails interact with eval-tail caps, and snapshot/report output should show eval activity explicitly so operators can confirm behavior.
