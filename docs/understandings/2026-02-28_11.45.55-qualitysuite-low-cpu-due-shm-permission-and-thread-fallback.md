# QualitySuite Low CPU: /dev/shm Permission Constraint

- During active tournament runs on 2026-02-28, benchmark logs repeatedly reported:
  - `Process-based config concurrency unavailable (PermissionError: [Errno 13] Permission denied); using thread-based config concurrency.`
- Direct host check confirmed `/dev/shm` write is blocked for this user context:
  - `ls -ld /dev/shm` shows `drwxrwxrwt`,
  - `touch /dev/shm/<file>` returns `Permission denied`.
- This forces all-method to thread-backed config workers, which reduces effective CPU scaling and can leave host-wide utilization around ~10-20% even when experiment-level parallelism is fully utilized.
- Throughput tuning still helps (experiment cap, `max_parallel_sources`, inflight), but the dominant ceiling remains process-worker unavailability until `/dev/shm` access is restored or worker orchestration is moved to a subprocess model that avoids SemLock.
