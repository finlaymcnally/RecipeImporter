summary: "Canonical line-role had two post-Codex rollback layers: per-row outside-span/ownership vetoes and a run-level do-no-harm fallback. Both could overwrite correct Codex structure labels, so they were removed."
read_when:

  - When debugging why Codex predicted recipe structure labels but final outputs reverted to deterministic labels
  - When editing `cookimport/parsing/canonical_line_roles.py`, `cookimport/config/run_settings.py`, or benchmark docs for line-role behavior

The old line-role path did not simply accept validated Codex labels.

It had:
- per-row post-Codex sanitizers that downgraded outside-span `HOWTO_SECTION`, `RECIPE_TITLE`, `RECIPE_VARIANT`, `INGREDIENT_LINE`, and `INSTRUCTION_LINE` labels back to `KNOWLEDGE`/`OTHER` when local anchor heuristics were weak
- an ownership-arbitration step that could replace Codex labels with deterministic baseline labels when hand-written syntax heuristics disagreed
- a second run-level `do_no_harm` arbitration pass, enabled by default, that could partially or fully fall back to deterministic labels after counting outside-span recipe-ish promotions

This meant the system was not really measuring “what Codex thought after reading the book.” It was often measuring “what survived several deterministic veto layers after Codex answered.”

Current contract after removal:
- shard validation still requires exactly one owned row back per atomic index
- generic label sanitizers still run for clearly invalid shapes such as bad `TIME_LINE` / `YIELD_LINE` uses
- but validated Codex line-role labels are no longer rolled back by ownership vetoes, outside-span structured-label downgrades, or do-no-harm guardrails
