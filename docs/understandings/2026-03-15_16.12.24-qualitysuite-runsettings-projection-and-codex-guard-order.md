---
summary: "QualitySuite codex guardrails must validate requested settings before benchmark-baseline coercion, and run-settings payloads must be projected before strict `RunSettings.from_dict(...)` validation."
read_when:
  - "When QualitySuite unexpectedly allows codex-enabled settings"
  - "When QualitySuite fails with unknown RunSettings keys like `bucket1_fixed_behavior_version` or `all_method_*`"
---

`cookimport/bench/quality_runner.py::_resolve_experiments(...)` applies `apply_benchmark_baseline_contract(...)`, which intentionally forces deterministic benchmark defaults (`llm_recipe_pipeline=off` and related baseline knobs).

If codex-disallow validation runs only after that baseline patch, codex-enabled user payloads can be masked and incorrectly accepted. The codex guard should run on the requested payload (`base_payload + run_settings_patch`) before baseline coercion.

Separately, strict `RunSettings.from_dict(...)` only accepts `RunSettings.model_fields`. QualitySuite payloads can include non-field settings (`all_method_*`, `label_studio_*`, `output_dir`) and metadata (`bucket1_fixed_behavior_version` from `to_run_config_dict`). Those payloads must be projected through `project_run_config_payload(..., contract=RUN_SETTING_CONTRACT_FULL)` before validation in both normal resolution and WSL safety-guard rebuild paths.
