# Three-Surface Operator Model

The easiest accurate explanation is still "three Codex-backed task types," but two clarifications matter.

First, line-role is now internally split into `recipe_region_gate` and `recipe_structure_label`, even though the operator-facing concept is still just "label the lines." Second, the knowledge pass does not read every leftover line as raw text; the program first builds eligible non-recipe chunks and skips obvious low-signal noise, then Codex reviews those chunks to keep/refine useful cooking knowledge.

So the plain-English pipeline is:

1. label/review all lines
2. group recipe-side lines into recipes
3. correct recipes
4. review eligible non-recipe chunks for knowledge

The durable contract across the three surfaces is immutable owned input payloads plus structured owned outputs/proposals, followed by deterministic validation and promotion. The workers should not be thought of as free-editing shared output files in place.
