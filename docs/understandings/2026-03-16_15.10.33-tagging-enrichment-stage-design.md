---
summary: "Recipe tagging is best treated as metadata enrichment over the finalized recipe object, then projected into both cookbook3 and schema.org outputs."
read_when:
  - "When integrating recipe tags into stage outputs instead of leaving them as sidecar artifacts"
  - "When deciding whether tags belong on final drafts, intermediate JSON-LD, or both"
---

The current runtime already has most of the pieces, but they are split across stages. `RecipeCandidate` already has a `tags` field, intermediate JSON-LD already maps `candidate.tags` to schema.org `keywords`, and final cookbook3 shaping currently only leaks tags into `recipe.notes` as plain text. The separate stage tags pass runs after `final drafts/` are written because its signal extraction is built around final-draft JSON, not raw `RecipeCandidate`.

That means the clean integration point is not “teach the tagger about schema.org” and not “keep mutating finished files as a sidecar forever.” The better design is to treat tagging as a recipe-metadata enrichment stage that runs after deterministic final recipe assembly has produced the authoritative recipe object, but before output artifacts are written or finalized. The tagger should read the finalized cookbook3-shaped recipe signal, compute structured tag suggestions once, and then project those same results into:

- final cookbook3 output as first-class recipe metadata, not just a line in `notes`
- intermediate JSON-LD as `keywords`, with optional targeted mapping into standard schema.org fields only where the catalog semantics truly match
- the existing `*.tags.json` / `tags_index.json` sidecar artifacts for provenance, audits, and tuning

For the final cookbook3 payload, the useful primary field is a stable machine field like `recipe.tags` (flat list of accepted tag keys). If richer provenance matters, add a second nested metadata field such as `recipe.tag_details` or `recipe.tagging` that preserves category/source/confidence without making downstream consumers parse the sidecar report. For schema.org, the safe default is `keywords`; broad arbitrary catalog tags should not be forced into `recipeCategory` or `recipeCuisine` unless there is an explicit category-to-schema mapping.
