# Cutdown semantic stage cutover

- The external-AI cutdown path had already switched its top-level starter-pack schema to semantic recipe stage names, but its middle layers were still split:
  - stage scoring still read `schemaorg/final` files,
  - starter-pack/upload-bundle summaries still emitted `pass1/pass2/pass3` aliases,
  - bench fixtures still built old prompt rows and old `recipe_manifest.json` payloads.
- The live runtime contract to follow is narrower and simpler:
  - recipe prompt rows use `stage_key` such as `build_intermediate_det`, `recipe_llm_correct_and_link`, and `build_final_recipe`,
  - recipe stage artifacts live under `raw/llm/<workbook>/recipe_correction/{in,out}` and `raw/llm/<workbook>/build_final_recipe/out`,
  - `recipe_manifest.json` exposes semantic stage statuses plus `mapping_status`, `mapping_reason`, `structural_status`, `structural_reason_codes`, `warnings`, and `errors`,
  - `recipe_correction_audit/*.json` is the place to recover correction input/output counts.
- The useful fix was to push the cutdown path all the way onto those live artifacts instead of adding one more translation layer. Once the scoring readers and tests moved to the semantic contract, the remaining failures were just stale expectations in starter-pack/runtime tests.
