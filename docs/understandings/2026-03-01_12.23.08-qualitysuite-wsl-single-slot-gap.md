# Discovery: QualitySuite WSL single-slot guard gap

Observed from adjacent run artifacts:

- `data/golden/bench/quality/runs/2026-03-01_11.55.18/experiments_resolved.json`
  - `wsl_safety_guard_applied=true`
  - `max_parallel_experiments_effective=4`
  - guarded runtime knobs (`max_inflight_pipelines=1`, `max_eval_tail_pipelines=0`, `smart_scheduler=false`)
- `data/golden/bench/quality/runs/2026-03-01_11.59.25/experiments_resolved.json`
  - `max_parallel_experiments_effective=1`
  - `experiment_executor_reason=single_worker`
  - `wsl_safety_guard_applied=false`
  - `wsl_safety_guard_reason=single_experiment_slot`
  - high inner settings remained (`workers=8`, `max_inflight_pipelines=5`, `max_eval_tail_pipelines=6`, `smart_scheduler=true`)

Practical interpretation:

- The first WSL fix covered outer fanout but left a single-slot path where inner all-method + worker fanout could still overload WSL.
- This explains why crashes could still happen even when CPU charts looked moderate and experiment fanout was pinned low.

Design follow-up:

- Treat WSL as a host-risk mode independent of outer experiment slots:
  - force subprocess experiment isolation for single-slot auto mode
  - apply safety guard caps in single-slot runs too
