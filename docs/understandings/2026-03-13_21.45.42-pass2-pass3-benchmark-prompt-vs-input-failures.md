# Pass2/pass3 benchmark prompt vs input failures

Benchmark examined: `data/golden/benchmark-vs-golden/2026-03-06_15.22.11/single-profile-benchmark/02_saltfatacidheatcutdown/codexfarm/`.

Key findings:

- The compact pass2/pass3 prompt templates are short and direct. They are not obviously over-instructing the model.
- Pass3 was barely used live in this run: `pass3_inputs = 2`, `pass3_pass2_ok_deterministic_skips = 19`. Most final drafts came from deterministic promotion, not the pass3 LLM prompt.
- The only explicit hard pass2 failure was recipe `c10`, where the model returned invalid `field_evidence`:
  - output contained `\"recipeIngredient\":[13010?, WAIT]`
  - this broke `Pass2SchemaOrgOutput` validation even though the actual recipe content looked reasonable
- Recipes `c15` and `c16` show the bigger upstream issue:
  - pass1 returned the same raw span `1374..1385` for both recipe ids
  - `_apply_pass1_midpoint_clamps` in `cookimport/llm/codex_farm_orchestrator.py` split that overlap at the midpoint
  - `c15` was clamped to `1374..1379` and sent to pass2 with title/yield/ingredients only, so pass2 degraded with `missing_instructions`
  - `c16` was clamped to `1380..1385` and sent to pass2 with instructions/notes/variations only, so pass2 had no title or ingredients and pass3 received `recipe_metadata = {"@context","@type"}` only
- The c16 pass3 result is therefore mostly a consequence of bad pass2 input, not bad pass3 prompt wording.
- Recipe `c1` shows a separate corruption path:
  - evidence normalization log preserves `jalapeño`
  - pass2 `schemaorg_recipe` also preserves `jalape\u00f1o`
  - but pass2 `extracted_ingredients` and `extracted_instructions` contain `jalape\u0014no`
  - pass3 copies those `extracted_*` fields verbatim, so the corruption is already present before pass3
- The pass2 prompt also tells the model to return `field_evidence` as a JSON-escaped string inside an outer JSON object. That double-encoding requirement is unusually brittle compared with the rest of the payload and is the only place in this run where a clearly valid-looking extraction was rejected for formatting.
- The pass2 prompt is also permissive about what counts as `extracted_instructions`. For the truncated `c16` input, the model obediently copied lines like `Ideal for ...` and `Variations` into `extracted_instructions` because the prompt says to preserve source order and does not clearly exclude serving notes, usage notes, or section headers.

Net: the benchmark artifacts do not mainly show "LLM cannot fix small JSON mistakes." They show pass2/pass3 operating on malformed or partial recipe slices, plus one brittle output-format requirement (`field_evidence`) and one isolated character corruption inside pass2 `extracted_*` fields.
