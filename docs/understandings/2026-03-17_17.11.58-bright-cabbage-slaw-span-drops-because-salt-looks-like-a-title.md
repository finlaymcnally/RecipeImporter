# Bright Cabbage Slaw Span Drops Because `Salt` Looks Like A Title

Concrete failing case from `data/golden/benchmark-vs-golden/2026-03-17_16.06.55/.../saltfatacidheatcutdown`:

- `Bright Cabbage Slaw` ends up with accepted Stage 2 recipe span `1118-1127`.
- Real instruction blocks start immediately after that at `1128-1132`.
- The recipe-correction input for `label_span_7` therefore includes only title/yield/description/ingredients and no real steps.

Local replay against `data/input/saltfatacidheatCUTDOWN.epub` shows the upstream boundary bug in `cookimport/plugins/epub.py`:

- `_find_recipe_end(...)` returns `1124` for the `Bright Cabbage Slaw` candidate.
- The stopping condition is [`cookimport/plugins/epub.py:1988`](/home/mcnal/projects/recipeimport/cookimport/plugins/epub.py#L1988): `if self._is_title_candidate(b) and self._has_ingredient_run(blocks, i): return i`
- At block `1124`, the text is just `Salt`.
- [`cookimport/plugins/epub.py:2340`](/home/mcnal/projects/recipeimport/cookimport/plugins/epub.py#L2340) treats `Salt` as a title candidate because it is short, not punctuation-ended, not ingredient-likely, and `text.istitle()` is true.
- [`cookimport/plugins/epub.py:2383`](/home/mcnal/projects/recipeimport/cookimport/plugins/epub.py#L2383) does not treat unquantified singleton ingredients like `Salt` as ingredient-like, so `_has_ingredient_run(...)` sees more ingredient lines after `Salt` and the boundary logic interprets that as a new recipe starting there.

Observed replay details around the failing window:

- `1121-1123` are correctly ingredient-like.
- `1124` = `Salt` is `is_title_candidate=True`, `is_ingredient_like=False`, `has_ingredient_run_from_here=True`.
- `1128` (`Quarter the cabbage through the core...`) is actually `is_instruction_like=True`, but it is never reached by the original candidate because the candidate already ended at `1124`.

Why the final accepted Stage 2 span reaches `1127` instead of stopping at `1124`:

- label-first line-role later rescues some outside-span ingredient lines (`1125-1127`) as `INGREDIENT_LINE`
- grouping then extends the accepted span through those contiguous structured labels
- but instruction blocks `1128-1132` remain outside-span and fall into non-recipe handling

So the primary root cause is not LLM misunderstanding. It is an EPUB segmentation heuristic bug:

- singleton ingredient lines like `Salt` can impersonate recipe titles
- that prematurely terminates the seed candidate
- Stage 2 then bootstraps from the already-truncated range and never gives the recipe model the real instruction rows
