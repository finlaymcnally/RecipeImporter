Reuse-key scope findings for discrete problem 4:

- All-method prediction reuse currently keys on almost the full `run_config` in `cookimport/cli.py::_all_method_prediction_reuse_key_payload`, excluding only `benchmark_sequence_matcher`.
- That means runtime-only settings can block reuse even when prediction artifacts would be identical.
- The current benchmark call projection in `cookimport/config/run_settings_adapters.py::build_benchmark_call_kwargs_from_run_settings` shows which settings actually feed prediction generation.
- `workers`, `pdf_split_workers`, `epub_split_workers`, `pdf_pages_per_job`, `epub_spine_items_per_job`, and `warm_models` are execution-shape settings, not obvious prediction-shape settings.
- Line-role cache currently uses `settings.stable_hash()` in `cookimport/parsing/canonical_line_roles.py::_resolve_line_role_cache_path`.
- That is much broader than necessary because the line-role cache key already includes a `candidate_fingerprint` built from atomic text, neighbor text, candidate labels, and rule tags.
- Because the candidate fingerprint already captures the prompt-relevant content, line-role cache identity should be narrowed to line-role-specific settings rather than the full run settings hash.
- `RunSettings.stable_hash()` is used broadly across tests and reports, so the safer fix is to add narrower cache-identity helpers instead of changing `stable_hash()` semantics globally.
