On `data/golden/benchmark-vs-golden/2026-03-16_18.10.25`, the token spike was not a retry storm. `prompt_budget_summary.json` shows 231 first-pass CodexFarm calls: 175 `recipe_correction` and 56 `knowledge`, with totals of 1,597,291 and 424,619 tokens respectively. The main driver is still the label-first grouping problem already seen on `saltfatacidheatCUTDOWN`: `group_recipe_spans` produced 175 recipe spans, many of them chapter/how-to fragments, so Codex was asked to correct far too many pseudo-recipes.

The recipe prompts were also materially larger than the earlier cutdown previews suggested. Reconstructing the live prompts from `raw/llm/.../recipe_correction/in/*.json` and the current `llm_pipelines/prompts/recipe.correction.compact.v1.prompt.md` gives an average rendered recipe prompt size of about 8,779 chars. The earlier `prompt_preview_after_next_cuts` preview for the same source averaged about 4,796 chars. The live prompt inflation comes mostly from the recipe payload carrying a constant `tagging_guide` object (~2,414 chars per call) plus the newer prompt/template instructions for `selected_tags`. Using the older preview template against the current live inputs still averages ~8,218 chars, while stripping `tagging_guide` from the current payload drops the current template average to ~6,118 chars.

Knowledge prompt compaction appears to have landed correctly. Live knowledge prompts average about 5,557 chars, which is effectively the same as the `prompt_preview_after_next_cuts` preview (~5,521 chars). Line-role was not a live-token contributor in this run: the benchmark manifest records `line_role_pipeline_recipe_projection.mode = authoritative_reuse`, so the spend was concentrated in recipe correction plus knowledge only.
