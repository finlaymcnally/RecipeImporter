---
summary: "RecipeImport's old shard-runtime audit label did not set CodexFarm's real runtime mode, so prompt-target counts could look honest in preview while live execution still expanded into extra session turns."
read_when:
  - When prompt-target counts look much lower than real Codex prompt-log rows
  - When editing `cookimport/llm/codex_farm_runner.py` or `phase_worker_runtime.py`
  - When deciding between `classic_task_farm_v1` and `structured_loop_agentic_v1` for shard-v1 work
---

Before the fix, `phase_worker_runtime.py` passed `runtime_audit_mode="structured_loop_agentic_v1"` into RecipeImport's subprocess runner, but `SubprocessCodexFarmRunner` only used that string to force `--workers 1` and to annotate runtime audit metadata. It did not pass `codex-farm process --runtime-mode ...`.

That meant the saved RecipeImport artifacts talked like shard-v1 was using CodexFarm's persistent session runtime even though the transport flag was never selected. The mismatch is why a benchmark planned as `5 / 5 / 5` shard prompts could still log many more prompt rows in live Codex telemetry.

The fix is to request CodexFarm's real classic one-shot transport explicitly for shard-v1 work: `--runtime-mode classic_task_farm_v1 --workers 1`. That keeps one RecipeImport worker assignment mapped to one CodexFarm process worker and makes shard count a much better proxy for real Codex call count.
