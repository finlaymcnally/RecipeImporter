The benchmark chooser now treats `Workflow for this run?` as the only top-level decision: `Vanilla / deterministic only` stays on the vanilla top-tier defaults, while `CodexFarm` loads the codex top-tier profile first.

If the operator picks CodexFarm, the follow-up benchmark customization now lives in one shared yes/no submenu instead of a checkbox list or two separate select menus. It asks explicit yes/no selects for recipe correction (`llm_recipe_pipeline`), block labelling (`line_role_pipeline`), and knowledge harvest (`llm_knowledge_pipeline`), while always keeping `atomic_block_splitter=atomic-v1`.

This keeps the UX aligned with the real product decision boundary: first choose whether this run is a CodexFarm run at all, then fine-tune which Codex-backed benchmark surfaces stay on inside that family.
