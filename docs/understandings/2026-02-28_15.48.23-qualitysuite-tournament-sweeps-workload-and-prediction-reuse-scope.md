# QualitySuite Tournament Sweeps Workload + Prediction Reuse Scope

- Current run (`2026-02-28_15.01.58`, fold `seed_42`) uses:
  - `include_deterministic_sweeps=true`
  - race: `probe_targets=2`, `mid_targets=4`, `finalists=64`, `keep_ratio=0.35`
  - `max_parallel_experiments=9`
- Variant expansion from code for this run settings:
  - sweeps off: `13` variants per target (`78` across 6 targets)
  - sweeps on: `143` variants per target (`858` across 6 targets)
  - sweep-on is ~11x more variant math/work before race pruning.
- Race-stage implied config volume per experiment (sweep-on, assuming normal keep behavior):
  - probe: `2 * 143 = 286`
  - mid: `4 * 64 = 256` (survivor floor from `finalists=64`)
  - final: `6 * 64 = 384`
  - total ~`926` config predictions per experiment, ~`8334` for 9 experiments in one fold.
- Existing caches:
  - eval-signature cache is cross-run (`.../eval_signature_results`) and reduces repeated evaluation.
  - prediction reuse cache is local to each all-method run root (`<round_or_experiment_root>/.prediction_reuse_cache`), so prediction work is not reused across:
    - race rounds (`round_01_probe` -> `round_02_mid` -> final),
    - experiments in the same quality-run,
    - folds in tournament.
- Reuse mechanism currently copies full prediction artifacts (`copytree`) when reused in-run, which is correct but I/O-heavy.
- Net: with sweeps enabled, prediction stage dominates and eval-cache reuse cannot compensate because prediction reuse scope is too narrow.
