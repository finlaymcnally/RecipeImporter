## Discovery

The remaining speed1-2 gaps were primarily contract-shape mismatches, not missing scheduler plumbing.

The key issue was that one legacy field (`max_eval_tail_pipelines`) was being used for three different meanings at once:
- user intent (configured override),
- runtime reality (effective headroom after bounds),
- and smart admission ceiling behavior.

That coupling made CPU-bounded explicit overrides and OG cap semantics (`base_inflight + headroom`) impossible to reason about cleanly in reports/tests.

## Practical resolution

Use a dedicated runtime object with explicit fields:
- `eval_tail_headroom_mode`
- `eval_tail_headroom_configured`
- `eval_tail_headroom_effective`
- `max_active_during_eval`
- optional CPU budgeting context

Then treat legacy keys (`max_eval_tail_pipelines`, `smart_tail_buffer_slots`) as compatibility aliases only.

This keeps downstream consumers stable while making headroom/admission behavior auditable and testable.
