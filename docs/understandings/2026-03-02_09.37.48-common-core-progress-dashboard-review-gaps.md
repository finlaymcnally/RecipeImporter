Reviewing `docs/plans/OGplan/2026-03-02_01.05.35-common-core-progress-dashboard.md` against implementation found two concrete gaps:

1. `_run_with_progress_status(...)` live ticker computes `worker_snapshot = len(worker_dashboard_adapter.snapshot_workers()[0])`, but `snapshot_workers()[0]` is an `int` worker count. This raises `TypeError` in the ticker thread, breaking steady live refresh semantics and affecting elapsed/ETA behavior.

2. `tests/cli/test_stage_progress_dashboard.py` claims stage live/plain parity coverage, but both tests currently fail because they monkeypatch `cli.stage_one_file`, which is no longer a module attribute in `cookimport.cli` (import now occurs inside `stage(...)`).

These gaps conflict with OG acceptance criteria requiring stable live rendering and parity verification gates.
