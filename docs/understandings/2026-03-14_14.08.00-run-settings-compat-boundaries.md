# Run-Settings Compatibility Boundaries

Three different concerns were easy to blur together:

1. live config ingestion (`RunSettings.from_dict(...)`, `cookimport.json`, winner cache)
2. raw run-config persistence (`run_config_json`, `run_config_hash`, `run_config_summary`)
3. dashboard/history reading of old benchmark and stage rows

Only the first concern actually needs `RunSettings` parsing. Analytics readers already work from persisted strings/JSON and compute summaries directly from those stored payloads. That means old charts can survive even if old live config files stop loading.

The main code smell behind the old compatibility behavior is mixed payloads. `_load_settings()` returns a broader app-settings dict, and some callers passed that whole dict into `RunSettings.from_dict(...)`, which encouraged silent unknown-key tolerance. The cleaner boundary is:

- filter mixed app settings down to `RunSettings.model_fields` before building `RunSettings`
- sanitize raw persisted run-config payloads only when they include non-model metadata
- treat stale winner/settings caches as disposable
- keep analytics/history tolerant because they are reading persisted artifacts, not live settings interfaces
