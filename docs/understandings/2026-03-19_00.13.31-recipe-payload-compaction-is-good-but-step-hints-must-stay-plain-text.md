# Recipe payload compaction is good, but compact step hints must stay plain text

The March 18 recipe-payload compaction is directionally correct. Removing shard-only metadata such as workbook/source identifiers and authority notes from the billed recipe shard made the model-facing payload cleaner without hurting correction. Compacting the deterministic candidate hint `h` down to title / ingredients / steps / yield is also the right general move.

The main quality regression risk is step-hint normalization. In fresh prompt previews after the compaction, some `h.s` entries became stringified Python-dict-looking values such as `"{'@type': 'HowToStep', 'text': 'Roast: Butternut Squash and Brussels Sprouts in Agrodolce'}"` instead of plain step text. That is worse than either the old verbose structure or a fully normalized compact text list, because it mixes transport noise into the one compact field the model is supposed to skim quickly.

For the recipe correction audience, the right compact-hint rule is:

- keep `ev` and `txt` as the main evidence
- keep `h` small
- when `recipeInstructions` contains structured `HowToStep` objects, normalize them to their text values before compacting
- if a step object has no recoverable text, drop it instead of stringifying the whole object

Fresh zero-token previews with the compacted payload still looked better overall and cut recipe-stage input tokens substantially:

- Salt Fat Acid Heat: `24513 -> 15972` estimated recipe input tokens
- The Food Lab: `23912 -> 15460` estimated recipe input tokens

So the compaction itself should stay. The follow-up is just to keep compact step hints semantically clean.
