# Line-Role File Payload Versus Inline Context

This note is historical context from before `docs/understandings/2026-03-18_17.05.01-atomic-line-adjacency-now-comes-from-explicit-index-lookup.md` landed.

Canonical line-role has two different prompt seams today.

`cookimport/parsing/canonical_line_roles.py::_line_role_plan_row(...)` and `cookimport/llm/prompt_preview.py` build the model-facing shard JSON written to `line-role-pipeline/runtime/line_role/workers/*/in/*.json` and `line-role-pipeline/in/line_role_input_*.json`. That JSON is the file-backed transport the wrapper prompt points at.

`cookimport/llm/canonical_line_role_prompt.py` still has separate inline compact-prompt helpers that render selective `ctx:<atomic_index>|prev=...|line=...|next=...` rows. Those inline context rows are not the same contract as the file-backed shard JSON.

The practical consequence at that time was that dropping `prev_text` and `next_text` from the file-backed shard JSON was a local transport change. That specific conclusion is now superseded: the later candidate cleanup also removed neighbor fields from `AtomicLineCandidate` and moved inline prompt context plus cache identity to explicit ordered lookup helpers.
