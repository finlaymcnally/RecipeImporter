- `execute_stage_import_session_from_result(...)` keeps using `result.report` if the importer already populated one, then later calls `enrich_report_with_stats(...)`.
- `enrich_report_with_stats(...)` ends by calling `finalize_report_totals(...)`, which compares the inherited report fields against counts recomputed from the final stage-owned `ConversionResult`.
- In refactor-era processed outputs, that comparison often means "old importer candidate/tip totals" versus "new label-first/stage-owned totals", so the mismatch artifact fires even when the stage result is behaving as designed.
- The clean fix is to stop treating inherited importer totals as the same contract as final stage totals: either start stage sessions from a fresh `ConversionReport`, or preserve importer counts under a separate explicitly legacy/debug namespace and only compare authoritative-with-authoritative counts.
