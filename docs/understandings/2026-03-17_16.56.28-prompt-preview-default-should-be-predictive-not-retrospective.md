---
summary: "Prompt preview should default to predictive estimation, not exact reuse of the selected run's live telemetry, because its main job is to forecast future shard-runtime cost before spending tokens."
read_when:
  - When changing `cf-debug preview-prompts` budget semantics
  - When deciding whether preview should reuse exact telemetry from a completed run
  - When prompt preview seems accurate for past runs but unhelpful for future planning
---

`cf-debug preview-prompts` became confusing once it started preferring exact live stage telemetry from the selected processed run, because that makes it very good at answering “what did this finished run cost?” and much worse at answering the predictive question operators actually care about: “what would a run like this cost if I launched it now?”

The correct default is predictive mode. In predictive mode, preview ignores exact live telemetry from the chosen run and uses stage-specific calibration built from prior classic shard-runtime rows instead. That keeps preview aligned with future execution rather than turning it into a retrospective reporting tool.

Exact live telemetry is still useful as an explicit debug path, so preview keeps an opt-in observed mode (`--estimation-mode observed`) for “compare prediction vs actual” workflows.
