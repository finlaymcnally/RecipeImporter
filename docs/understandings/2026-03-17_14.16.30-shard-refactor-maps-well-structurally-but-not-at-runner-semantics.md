# Shard Refactor Maps Well Structurally But Not At Runner Semantics

- The implemented refactor matches the report well on structure: all three Codex-backed phases now plan explicit shards, execute via `phase_worker_runtime.py`, validate exact owned IDs before promotion, keep deterministic assembly/promotion in charge, and expose worker/shard runtime artifacts plus worker/shard-centric preview and upload-bundle summaries.
- The main remaining mismatch is the strongest execution-model claim from the report: the current runtime still calls `SubprocessCodexFarmRunner.run_pipeline(...)`, which shells out to `codex-farm process --pipeline ... --in ... --out ... --json` once per RecipeImport worker assignment.
- RecipeImport now forces `codex-farm process --workers 1` for that shard-runtime mode, so reported worker counts map cleanly onto CodexFarm process-worker counts.
- Even with that fix, the runner contract here does not itself enforce one Codex session reused across those shards, and `max_turns_per_shard` is still recorded in manifests rather than enforced at the transport layer.
- So the landed code is best described as a successful shard-owned runtime and observability refactor with partial execution-model delivery: it replaced prompt-per-bundle planning and promotion semantics, tightened worker-count honesty, but it still does not prove the report's stronger "load instructions once and reuse one live worker session across multiple shards" goal.
