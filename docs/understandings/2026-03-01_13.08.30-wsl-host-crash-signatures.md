# Discovery: WSL host/runtime crash signatures (2026-03-01)

Key boot timeline from `journalctl --list-boots`:

- boot `-3`: `11:41:08` to `12:09:35`
- boot `-2`: `12:14:46` to `12:48:04`
- boot `-1`: `12:48:08` to `13:01:56`
- boot `0`: `13:03:48` onward

Observed evidence:

- `boot -2` ends at `12:48:04` with a broad fatal-signal cascade:
  - `node`, `codex`, `bash`, `systemd`, `chrome`, `chrome_crashpad`, `Xwayland`, `weston`
  - repeated lines: `potentially unexpected fatal signal 7/11`
  - repeated `Core dump to |/wsl-capture-crash pipe failed`
- Around earlier crash windows, Windows logs show repeated Hyper-V virtual switch warnings for WSL host vNIC (`Microsoft-Windows-Hyper-V-VmSwitch`, event `285`), including at `12:10:05`.
- VS Code server logs show stale lock recovery and fresh extension host launches after each reconnect cycle, consistent with abrupt prior termination.

Practical interpretation:

- This is broader than QualitySuite-specific nested parallelism.
- At least one failure mode is WSL runtime/WSLg instability with Chromium/GUI stack involvement.
- App-level workload limits can reduce pressure, but they cannot fully prevent host/runtime crashes in this mode.
