# Follow-up attribution and line-role join caveats

- `cf-debug ablate` currently calls `cookimport.bench.followup_bundle.write_ablation_matrix`, which only writes the comparable source list plus a fixed variant matrix (`baseline_control`, `line_role_only`, `recipe_only`, `full_stack`). It does not compute measured subsystem attribution from the session artifacts.

- Real attribution for a single-profile bundle still has to be derived from other bundle/session artifacts such as:
  - `analysis.book_scorecard`
  - `analysis.net_error_blame_summary`
  - `analysis.runtime_by_book`
  - `starter_pack_v1/01_recipe_triage.jsonl`

- `cf-debug` follow-up line-role audits currently use direct index joins:
  - `RunContext.line_role_predictions_by_index` keys rows by `atomic_index`
  - `_build_line_role_audit_rows(...)` looks those rows up with the changed canonical `line_index`
  - `RunContext.extracted_lines_by_index` does the same for page context

- Result: if atomic line-role rows diverge from canonical eval lines after atomization/alignment, `line_role_audit` / `page_context` can show a different text span than the changed-line row being investigated. In the 2026-03-06_15.22.11 follow-up packet this showed up on title regressions where:
  - the changed canonical line text was a recipe/header title
  - the follow-up page context at the same numeric index was neighboring prose instead

- For structural title/header regressions in this session, the more trustworthy later-stage code path was:
  - `cookimport/staging/draft_v1.py::apply_line_role_spans_to_recipes`
  - `cookimport/staging/stage_block_predictions.py::_label_recipe_blocks`
  - `cookimport/staging/stage_block_predictions.py::_find_title_block_index`

- `_find_title_block_index(...)` can fall back to the first positively scored block in range, or ultimately `candidate_blocks[0]`, when it cannot find an exact recipe-title match. That is a plausible source of false `RECIPE_TITLE` promotion when pass1/pass2 provenance ranges are structurally wrong.
