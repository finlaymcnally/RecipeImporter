# Why The Line-Role Payload Split Exists

Canonical line-role now has two file-backed seams on purpose.

The model-facing task file is the compact transport written to `line-role-pipeline/runtime/line_role/workers/*/in/*.json` and preview `line-role-pipeline/in/*.json`. Its shape is:

    {"v":1,"shard_id":"...","rows":[[atomic_index,label_code,current_line], ...]}

That is the billed payload. `request_input_file`, `request_input_payload`, `task_prompt_text`, and preview token estimation all point at that compact file.

The richer local copy is written beside it at `workers/*/debug/*.json` and preview `debug_in/*.json`. Those rows still keep `block_index`, `block_id`, `recipe_id`, `within_recipe_span`, `deterministic_label`, `rule_tags`, and `escalation_reasons` so local debugging and reviewer exports do not lose provenance.

This split is the important rule:

- compact `in/*.json`: only what Codex needs to label lines
- rich `debug/*.json`: local provenance and operator context
- wrapper prompt: explains the tuple schema and label-code legend once
