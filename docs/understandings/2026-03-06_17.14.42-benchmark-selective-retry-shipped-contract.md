summary: "Shipped contract for benchmark-only Codex Farm selective retry: retry only missing pass2/pass3 bundles after recoverable partial-output runs, persist details in llm_manifest, and surface only concise counts in run manifests."
read_when:
  - When debugging benchmark Codex Farm retry/resume behavior
  - When changing llm_manifest or benchmark run_manifest retry reporting
---

- The retry seam lives inside `cookimport/llm/codex_farm_orchestrator.py::run_codex_farm_recipe_pipeline(...)`, immediately after pass2/pass3 `run_pipeline(...)` returns and before the existing per-recipe output validation loops.
- Retry is benchmark-only and gated by three conditions together:
  - `run_settings.codex_farm_recipe_mode == "benchmark"`
  - `codex_farm_benchmark_selective_retry_enabled == true`
  - the serialized `process_run` payload shows a non-zero exit with failure categories limited to `nonzero_exit_no_payload` plus optional `timeout`
- Retry batches are built from the original pass `in/` directory by copying only the still-missing bundle filenames into `retry_attempt_XX/in`; recovered outputs are copied back into the original `out/` directory only when that output file did not already exist.
- Detailed provenance lives in `raw/llm/<workbook_slug>/llm_manifest.json` under `selective_retries.pass2|pass3`, including attempted/recovered/unrecovered bundle filenames, recipe ids, retry directories, and per-attempt `process_run` payloads.
- Prediction-run and benchmark `run_manifest.json` intentionally stay terse: they surface only `selective_retry_attempted`, `selective_retry_pass2_attempts`, `selective_retry_pass2_recovered`, `selective_retry_pass3_attempts`, and `selective_retry_pass3_recovered`, while still linking the full `llm_manifest.json`.
