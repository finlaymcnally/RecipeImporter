# Oracle relaunch misdiagnosis (2026-03-01)

Context: I launched Oracle browser runs for the same prompt three times because I misread runtime state as dead/stuck.

What looked like failure (but was ambiguous):
- `models/gpt-5.2-pro.log` stayed at 0 bytes for long periods on active browser sessions.
- `output.log` only showed launch lines (bundle count + token estimate), with no progress heartbeat.
- A minimized/off-screen Chromium process was active on debug port 9222 (`--start-minimized --window-position=32000,32000`), so visible progress in desktop UI was missing.
- `oracle status` and `sessions/*/meta.json` gave conflicting signals (`error` in status view while meta still said `running`).
- Session storage confusion (`~/.oracle` vs `~/.local/share/oracle`) made status lookup inconsistent until explicitly pinning `ORACLE_HOME_DIR`.

Why I relaunched:
- I treated "no streamed output + blank/minimized browser + no model log growth" as terminal failure.
- I over-weighted immediate observability and under-weighted the possibility of long-running detached browser execution.

What would prevent this:
- Add explicit per-session heartbeats in `output.log` (last browser action + timestamp).
- Add `session doctor` output: controller PID alive?, browser PID alive?, current tab URL, last model activity time.
- Surface one canonical status source (avoid `status` vs `meta.json` disagreement).
- Emit explicit state labels like `waiting_for_ui`, `submitted_waiting_model`, `streaming_response`, `detached_running`.
- Show a clear warning when port/profile reuse attaches to an existing minimized renderer.
