summary: "upload_bundle_v1 is brittle because it derives reviewer semantics directly from the current benchmark run layout instead of a topology-neutral intermediate model."
---

`scripts/benchmark_cutdown_for_external_ai.py` currently mixes five concerns in one path: run-artifact discovery, current-layout path resolution, semantic normalization, reviewer-facing analysis, and final bundle serialization.

That coupling shows up most clearly in `_upload_bundle_build_context(...)`, `_build_run_record_from_existing_run(...)`, `_build_comparison_summary(...)`, `_upload_bundle_build_stage_separated_comparison(...)`, and `_write_upload_bundle_three_files(...)`. The bundle can tolerate some missing root summaries by re-deriving data, but it still assumes today's benchmark artifact families (`need_to_know_summary.json`, `comparison_summary.json`, starter-pack files, raw Codex pass-slot folders, fixed stage-comparison keys).

The durable pre-refactor move is to insert a stable, topology-neutral intermediate input model for reviewer bundles. Future cookbook refactors should only need a new adapter from the new staging/bundling layout into that model, while `upload_bundle_v1` rendering stays mostly unchanged.
