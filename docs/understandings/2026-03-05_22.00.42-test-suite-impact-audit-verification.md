- `tests/conftest.py` already sets `COOKIMPORT_ALLOW_LLM=1` in `pytest_configure`, so a low-level kill switch in `cookimport/llm/codex_exec.py` is not a future suite blocker; that mitigation is already present.
- `tests/llm/test_run_settings.py` and `tests/cli/test_cli_output_structure_fast.py` do hardcode current AI-on defaults, so flipping defaults in `RunSettings` would break those assertions.
- The audit overstates live-LLM exposure in some cited tests: `tests/core/test_recipe_likeness_scoring.py` uses `RunSettings()` but exercises deterministic scoring code, and `tests/llm/test_codex_farm_orchestrator.py` / stage-integration tests use `FakeCodexFarmRunner` or monkeypatched subprocess paths rather than live Codex calls.
- Stable-hash coupling is real mainly in scheduler/quality-suite tests that assert counts of unique `variant.run_settings.stable_hash()` values (`tests/labelstudio/test_labelstudio_benchmark_helpers_scheduler.py`) and in quality-run report fixtures that carry `run_settings_hash`, but `tests/parsing/test_canonical_line_roles.py` is not a primary stable-hash coupling site.
