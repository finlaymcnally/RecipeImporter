# Refactor stage reuse map

High-level reuse pattern from tracing `docs/plans/Refactor.md` against current code:

- Stages 0-1 already exist in pieces, mainly in the EPUB/PDF block-first importers and extractor adapters. The missing part is one shared normalized-source and segmented-block contract used by every importer family.
- Stage 2 also exists in pieces via the canonical line-role pipeline (`recipe_block_atomizer.py`, `canonical_line_roles.py`), but it is benchmark / Label Studio side infrastructure today, not the main stage/import source of truth.
- Stage 3 is the biggest architectural mismatch. Current recipe grouping is mostly heuristic candidate detection in importers, then later code projects spans back out from those candidates.
- Stage 4 already has a usable intermediate shape in `RecipeCandidate`, `jsonld.py`, and the parser helpers, but there is no standalone “labeled span -> intermediate object” builder yet.
- Stages 5-6 overlap heavily with `codex_farm_orchestrator.py`, especially the merged-repair path. The missing piece is to re-anchor that logic on label-first spans and to separate “LLM correction + linkage” from “deterministic final assembly” more cleanly.
- Stage 7 already has strong downstream machinery (`non_recipe_blocks`, chunks, pass4 knowledge), but it is still residue-driven rather than label-driven.
- Stage 8 already has strong writer/debug infrastructure. What is missing is a new artifact contract that matches the refactor’s stage names and boundaries.

Main implication:

- This refactor does not need a greenfield rewrite everywhere. The strongest reuse seams are Stage 0-1 extraction, Stage 2 line-role labeling, Stage 4 parsing/shaping, Stage 5-6 Codex repair, Stage 7 knowledge, and Stage 8 writing.
- The main greenfield work is the label-first backbone: one shared source-document contract, one shared segmented-block artifact, one Stage 2 label artifact in the primary runtime, and one Stage 3 grouping module that consumes those labels directly.
