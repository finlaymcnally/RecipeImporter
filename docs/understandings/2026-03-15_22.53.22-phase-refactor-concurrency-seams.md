The four refactor phases are not fully independent.

As written, Phase 2 explicitly assumes Phase 1 is done. Phase 3 explicitly assumes Phase 1 and Phase 2 are done and says it must consume the Phase 2 label-first bundle directly. Phase 4 also assumes the earlier migration work is already in place, including final corrected block labels, deterministic recipe spans, and the post-Phase-3 recipe path.

Practical concurrency rule:

- Phase 1 should land first.
- Phase 2 should land second.
- After the Phase 2 handoff is stable, Phase 3 and Phase 4 can be developed mostly in parallel if they both treat the Phase 2 block-label plus recipe-span bundle as read-only input and avoid redefining stage naming outside `stage_observability.json`.

Main shared conflict files if parallel work is attempted too early:

- `cookimport/staging/import_session.py`
- `cookimport/cli.py`
- `cookimport/labelstudio/ingest.py`
- shared observability helpers under `cookimport/runs/`

Best split is not "full phases in parallel from day one"; it is "serial shared contracts first, then parallel lane work":

- shared contracts: Phase 1 observability, then Phase 2 authoritative labels and recipe spans
- recipe lane: Phase 3 single-correction recipe path
- non-recipe lane: Phase 4 deterministic non-recipe classification and narrowed pass4 extraction
