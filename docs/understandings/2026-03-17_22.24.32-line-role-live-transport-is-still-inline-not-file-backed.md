summary: "Canonical line-role still sends the full shard body inline to `codex exec`; a true \"give it a file\" runtime would require a different transport contract."
read_when:
  - When diagnosing line-role token explosions or context-window failures on large benchmark books
  - When planning a file-backed or more agentic Codex transport for line-role shards
  - When reconciling prompt-size telemetry against the existence of `workers/*/in/*.json` debug files
---

Observed facts from the current code and the `thefoodlabcutdown` benchmark failure:

- `cookimport/parsing/canonical_line_roles.py` builds each shard's full `prompt_text` up front by calling `build_canonical_line_role_prompt(...)` over every owned candidate in the shard.
- `cookimport/llm/codex_exec_runner.py` then sends that rendered prompt text directly to `codex exec` on stdin. The runner has no separate file-reading seam today; `prompt_text` is the model-facing request body.
- The line-role runtime still writes `workers/*/in/<shard>.json`, but those files are only debug copies of the shard payload. They are not the authoritative model input.
- On `data/golden/benchmark-vs-golden/2026-03-17_22.10.53/.../thefoodlabcutdown`, line-role planned `2981` atomic lines into `5` shards (`597` lines per shard). The saved prompt files were about `352k` to `590k` characters each, and one shard failed with `Codex ran out of room in the model's context window`.

The practical conclusion is that reducing prompt size for large books is not just a tuning problem. If the desired behavior is "give Codex a file and let it work through that file," line-role needs a different runtime contract where the worker input file becomes the model-facing seam instead of a debug artifact beside an inline prompt.
