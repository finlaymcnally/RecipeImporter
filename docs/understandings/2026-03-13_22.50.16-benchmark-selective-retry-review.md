# Benchmark Selective Retry Review

This note is a code-verified review of the benchmark selective retry/resume feature that started in `docs/plans/OGplan/2026-03-06_16.48.07-benchmark-selective-retry-resume.md` and was later marked complete in `docs/plans/2026-03-06_16.48.07-benchmark-selective-retry-resume.md`.

## Resolution Status

As of `2026-03-13_23.01.16`, the two concrete gaps called out below have been fixed:

- `cookimport/llm/codex_farm_orchestrator.py::_selective_retry_eligible_process_run(...)` now reuses the same `no last agent message` plus benchmark-coverage contract as `cookimport/llm/codex_farm_runner.py::_is_recoverable_no_last_agent_message_failure(...)`, using serialized `error_summary` plus telemetry from the original process payload instead of a weaker category-only approximation.
- `cookimport/cli.py::labelstudio_benchmark(...)` now mirrors a direct `llm_manifest_json` artifact link into the benchmark root `run_manifest.json`, so reviewers can jump straight from the benchmark root to the detailed retry audit trail.
- Focused validation was refreshed with high-coverage retry fixtures plus negative tests for missing summary evidence and low coverage.

The rest of this note is preserved as the review that identified those gaps and the stale validation claim.

The goal of the review was not to restate the implementation from the finished ExecPlan. The goal was to compare three things directly:

1. what the OG plan said should exist,
2. what the current code actually does, and
3. what the completed ExecPlan now claims is shipped and validated.

The broad result is that most of the feature shape from the OG plan is present, but two real gaps remain versus the original intent, and the current post-refactor tree no longer cleanly matches the completed ExecPlan's "validated and done" status.

## Original Intent

The OG plan described a narrow recovery feature for Codex Farm recipe benchmark runs.

- Scope was benchmark-only.
- Scope was pass2 and pass3 only.
- Retry unit was missing bundle files, not whole-book reruns.
- Successful original outputs had to remain untouched.
- Retry had to run at the pass boundary before the normal per-recipe validation/routing logic.
- `llm_manifest.json` had to hold detailed retry truth.
- prediction-run and benchmark root manifests were supposed to stay concise, but still retain the existing artifact link to `llm_manifest_json`.

The OG plan also made one subtle but important design choice: retry eligibility was supposed to follow the same "recoverable partial-output" contract already accepted by the runner, not a weaker local approximation.

## Current Implementation Shape

The current implementation is split across three layers.

### 1. Runner-level recoverable partial-output contract

`cookimport/llm/codex_farm_runner.py` owns the decision about whether a non-zero `codex-farm process` invocation may continue with partial outputs.

The key function is `_is_recoverable_no_last_agent_message_failure(...)`.

Current behavior there is stricter than just "failure categories include `nonzero_exit_no_payload`".

- The stderr/error summary must include `no last agent message`.
- A pure `nonzero_exit_no_payload` family is recoverable.
- In benchmark mode only, a mixed `nonzero_exit_no_payload` plus `timeout` family is also recoverable, but only when output coverage remains high.
- Coverage is currently defined by `_BENCHMARK_RECOVERABLE_PARTIAL_MAX_MISSING_OUTPUTS = 3` and `_BENCHMARK_RECOVERABLE_PARTIAL_MIN_SUCCESS_RATIO = 0.8`.

That means the runner's contract is "failure family plus evidence plus coverage", not just "failure family".

### 2. Orchestrator-level selective retry

`cookimport/llm/codex_farm_orchestrator.py` contains the actual selective retry helper.

The important helper is `_run_benchmark_selective_retry(...)`.

What it currently does:

- computes the missing bundle filenames for the pass output directory,
- exits early if there are no missing bundles,
- exits early unless `codex_farm_recipe_mode == benchmark`,
- exits early if benchmark selective retry is disabled,
- exits early unless `_selective_retry_eligible_process_run(...)` accepts the original process payload,
- creates `retry_attempt_XX/in` and `retry_attempt_XX/out`,
- copies only the still-missing input bundles into retry `in/`,
- reruns the same pipeline on only those missing inputs,
- copies only newly created retry outputs back into the original `out/` directory,
- never overwrites an already-existing original output file,
- repeats until success, attempts exhausted, or retry eligibility is no longer met,
- returns a manifest-ready payload with attempted, recovered, unrecovered, recipe ids, retry directories, and per-attempt `process_run` payloads.

This part matches the core shape the OG plan asked for.

### 3. Manifest/reporting layer

The detailed retry truth is written into `raw/llm/<workbook_slug>/llm_manifest.json`.

Current `llm_manifest.json` behavior includes:

- top-level `selective_retries.pass2` and/or `selective_retries.pass3`,
- resolved retry settings snapshot,
- attempted/recovered/unrecovered bundle counts,
- attempted/recovered/unrecovered bundle filenames,
- attempted/recovered/unrecovered recipe ids,
- per-attempt retry directory,
- per-attempt attempted bundle filenames,
- per-attempt copied output filenames,
- per-attempt remaining missing bundles and recipe ids,
- per-attempt serialized `process_run`.

On the concise side:

- `cookimport/labelstudio/ingest.py` mirrors retry summary counts into prediction-run `manifest.json`,
- `cookimport/cli.py` copies only the retry summary counts from prediction-run context into benchmark root `run_manifest.json`.

## What Matches the OG Plan

The following parts are present and consistent with the OG plan.

- There are exactly two user-facing knobs in `RunSettings`: `codex_farm_benchmark_selective_retry_enabled` and `codex_farm_benchmark_selective_retry_max_attempts`.
- The benchmark CLI exposes those knobs and threads them into prediction generation.
- Retry is benchmark-only.
- Retry is pass2/pass3-only.
- Retry batches contain only missing bundle inputs.
- Retry copy-back does not overwrite successful original outputs.
- Retry artifacts live in `retry_attempt_XX/` directories under the original pass directory.
- `llm_manifest.json` carries the detailed retry provenance the OG plan asked for.
- prediction-run and benchmark manifests stay concise rather than duplicating per-bundle truth.

## Confirmed Gaps Versus the OG Plan

Two concrete gaps remained after comparing plan and code.

### Gap 1: orchestrator retry gating is weaker than the runner contract

The OG plan said retry should be limited to the same recoverable partial-output family already accepted by the runner.

The runner contract currently depends on all of these:

- `no last agent message` evidence in error summary,
- allowed failure categories,
- benchmark-mode-only widening for timeout mixes,
- output coverage thresholds.

The orchestrator helper does not enforce that full contract itself.

`_selective_retry_eligible_process_run(...)` in `cookimport/llm/codex_farm_orchestrator.py` currently checks only:

- non-zero exit present,
- failure categories exist,
- categories include `nonzero_exit_no_payload`,
- categories are a subset of `{nonzero_exit_no_payload, timeout}`.

It does not independently require:

- `no last agent message`,
- benchmark coverage thresholds,
- the same exact semantics used by `_is_recoverable_no_last_agent_message_failure(...)`.

Why this matters:

- In the standard subprocess path, the runner still does the stricter gate first, so normal execution is somewhat protected.
- But the orchestrator helper itself is not bound to the full OG-plan contract.
- Any future runner change, fake runner, or alternate caller that feeds a process payload directly into the orchestrator could trigger retry under conditions the OG plan did not intend.

This is a real design mismatch, not just a test wording issue.

### Gap 2: benchmark root manifest lacks the direct `llm_manifest_json` link

The OG plan said the concise root manifest should keep summary counts plus the existing artifact link to `llm_manifest_json`.

Current behavior is split:

- prediction-run `manifest.json` includes `llm_manifest_json`,
- benchmark root `run_manifest.json` includes concise retry counts copied from prediction-run context,
- benchmark root does not include a direct `llm_manifest_json` artifact link.

Why this matters:

- A reviewer starting from the benchmark root cannot jump straight to the detailed retry truth.
- They must follow `pred_run_dir` and then inspect the prediction-run manifest or raw tree.
- That is still usable, but it is not the exact breadcrumb trail the OG plan described.

This is a smaller gap than the gating mismatch, but it is still a plan-vs-code divergence.

## Completed ExecPlan Drift

The completed ExecPlan currently says the shipped result matches the original goal and records a passing focused validation command.

That statement no longer cleanly matches the current post-refactor tree.

During this review, targeted benchmark-helper artifact tests still passed, but the narrowed orchestrator retry regression slice currently failed. In particular, the current `test_orchestrator_selective_retry_recovers_missing_pass2_bundle_in_benchmark_mode` now expects `pass3_ok == 2`, and in the current tree it fails with `pass3_ok == 0`.

That failure does not by itself prove the feature is fundamentally wrong. It does prove that the completed ExecPlan is no longer a safe "feature is fully validated today" document without revalidation or update.

So there are really two different truths:

- the feature shape is largely implemented,
- the finished ExecPlan's success/validation status is stale relative to the current tree.

## Practical Takeaway

If the question is "did the implementation broadly build what the OG plan imagined?", the answer is mostly yes.

If the question is "what is not built exactly as the OG plan imagined?", the answer is:

1. retry eligibility is not enforced with the same full contract as the runner's recoverable partial-output gate, and
2. the benchmark root manifest does not keep the direct `llm_manifest_json` artifact link the OG plan called for.

If the question is "is the completed ExecPlan currently trustworthy as a done-and-validated record?", the answer is no, not without refreshing its validation claims against the current test tree.

## Suggested Follow-up Checks

If this feature is revisited, the lowest-noise follow-up would be:

- make the orchestrator retry helper depend on the same strict recoverable-partial semantics as the runner, rather than a weaker local category-only check,
- add `llm_manifest_json` directly to the benchmark root manifest artifacts if the benchmark root is still intended to be the main reviewer entry point,
- rerun and update the completed ExecPlan's validation section after the current refactor dust settles.
