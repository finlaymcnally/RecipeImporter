# Interactive benchmark Oracle upload seam

- The interactive benchmark entrypoints already have the right post-processing seam. `cookimport/cli.py::_interactive_single_offline_benchmark(...)` builds one session bundle at `<session_root>/upload_bundle_v1`, and `cookimport/cli.py::_interactive_single_profile_all_matched_benchmark(...)` builds per-book bundles plus one final group bundle at `<single_profile_root>/upload_bundle_v1`.

- The user-visible requirement maps cleanly onto those existing outputs: single-book runs should upload the single-offline session bundle, while multi-book runs should upload only the final group bundle, not every per-book bundle.

- The bundle builder is already centralized. Both interactive flows call `cookimport/cli.py::_write_benchmark_upload_bundle(...)`, which wraps `scripts/benchmark_cutdown_for_external_ai.py::build_upload_bundle_for_existing_output(...)`. Oracle integration should reuse that return value instead of rediscovering folders from scratch.

- The Oracle skill contract for this machine is specific: browser execution should use `/home/mcnal/.local/bin/oracle-browser-headless`, not direct `npx ... --engine browser ...`. A low-cost validation path should use Oracle dry-run rendering instead of a live benchmark rerun.

- The practical no-benchmark test surface should be a direct CLI that accepts an existing bundle directory or its parent session root. The checked-in multi-book sample at `data/golden/benchmark-vs-golden/2026-03-04_20.33.53/single-profile-benchmark/upload_bundle_v1` is the safest real artifact for that smoke test; single-book behavior can stay fixture-backed in unit tests.
