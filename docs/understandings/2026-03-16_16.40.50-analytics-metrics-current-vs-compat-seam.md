# Analytics Metrics: Current Versus Compatibility Seam

The analytics/dashboard surface still uses several precision/recall-style metrics as part of the current product, not only as historical alias baggage.

- `cookimport/analytics/dashboard_render.py` still computes and renders current tables/cards using `precision`, `recall`, `practical_f1`, and related means.
- `cookimport/cli.py` still builds all-method summaries and winner payloads around those metric names.
- `cookimport/analytics/perf_report.py` and `cookimport/analytics/dashboard_collect.py` still normalize current reports into those fields for downstream rendering.

That means the safe destructive cut is narrower than “delete every precision/recall/practical_f1 field.” The real legacy seams in analytics are:

- fallback readers that reinterpret older report shapes as current metrics,
- comments/docs that present historical compatibility as active guidance,
- artifact-reader tolerance for old timestamp/path/history layouts where current artifacts already define the canonical contract.

If a future cleanup wants to rename the current analytics metric surface away from `precision`/`recall`/`practical_f1`, that is a separate product change touching dashboard renderers, compare/control outputs, CLI reporting, CSV schemas, and tests. It should not be mixed into the legacy purge unless the repo explicitly adopts a new canonical analytics vocabulary first.
