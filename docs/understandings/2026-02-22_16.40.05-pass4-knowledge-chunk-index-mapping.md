# Pass4 knowledge: chunk indices vs absolute block indices

`chunks_from_non_recipe_blocks(...)` (and the underlying chunker in `cookimport/parsing/chunks.py`) builds `KnowledgeChunk.blockIds` as **relative indices into the input block list**, not as absolute `full_text` block indices.

That has two important consequences for pass4 knowledge harvesting:

1. To produce LLM evidence pointers that are auditable against the original document, job bundles must map chunk `blockIds` back to **absolute** `full_text` indices by looking up `sequence[relative_index]["index"]` for each `relative_index`.
2. `ConversionResult.nonRecipeBlocks` is a “recipe-removed” view of the full block stream, so it can contain **gaps** in absolute indices where recipe spans were removed. If you chunk the entire non-recipe list in one go, the chunker can form a chunk that spans across these gaps, which makes `block_start_index`/`block_end_index` ambiguous in the original stream.

Implementation choice:

- `cookimport/llm/codex_farm_knowledge_jobs.py` sorts `nonRecipeBlocks` by absolute `index` and **splits them into contiguous sequences** where each next block has `index == prev_index + 1`.
- Each contiguous sequence is chunked separately, ensuring every knowledge chunk corresponds to a contiguous span in the original block stream, so `block_start_index` and `block_end_index` remain meaningful.
