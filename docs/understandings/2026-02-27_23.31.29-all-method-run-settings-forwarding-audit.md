# All-Method Run-Settings Forwarding Audit

## Audit scope

- Goal: verify whether parsing/quality config knobs added in Priority 1-8 are actually applied in all-method benchmark execution, not just defined in `RunSettings`.
- Focus area:
  - benchmark adapter payload (`cookimport/config/run_settings_adapters.py:106-204`)
  - all-method variant generation (`cookimport/cli.py:6282-6401`)
  - all-method predict-only execution call (`cookimport/cli.py:7118-7417`)
  - `labelstudio_benchmark(...)` accepted benchmark knobs (`cookimport/cli.py:16309-16605`)
  - speed/quality all-method entrypoints (`cookimport/bench/speed_runner.py:412-442`, `cookimport/bench/quality_runner.py:361-380`)

## Execution-lane map

- Lane A: single benchmark run (full adapter lane)
  - interactive single-offline path builds kwargs from adapter then calls `labelstudio_benchmark(**benchmark_kwargs)` (`cookimport/cli.py:2718-2731`).
  - speed canonical benchmark sample also uses adapter kwargs + direct `labelstudio_benchmark` (`cookimport/bench/speed_runner.py:347-369`).
  - this lane carries the full adapter key surface.

- Lane B: all-method run (internal subset lane)
  - build per-target variants from base settings (`cookimport/cli.py:6282-6401`).
  - for each config, call `labelstudio_benchmark(execution_mode=predict-only, ...)` from `_run_all_method_prediction_once` (`cookimport/cli.py:7245-7295`).
  - this lane does not currently forward the full adapter key surface.

- Lane C: speed/quality all-method wrappers
  - both build variants then call `_run_all_method_benchmark_multi_source(...)` (`cookimport/bench/speed_runner.py:412-442`, `cookimport/bench/quality_runner.py:361-380`).
  - both therefore inherit the same forwarding behavior as Lane B.

## Variant generation behavior (what can vary)

From `cookimport/cli.py:6282-6401`:

- Non-EPUB, non-webschema-capable sources:
  - one variant only.
  - dimension includes `source_extension`.
  - if non-default: includes `section_detector_backend` and `multi_recipe_splitter` in slug/dimensions.

- Webschema-capable sources (`.html`, `.htm`, `.jsonld`, schema-like `.json`):
  - variants fan out over `web_schema_policy`.
  - optional section/multi-recipe dimensions are appended when non-default.

- EPUB sources:
  - variants fan out over extractor dimension and unstructured parser controls.
  - optional section/multi-recipe dimensions appended when non-default.

Important: this is variant construction only; execution correctness depends on which keys are later forwarded into `labelstudio_benchmark(...)`.

## Priority-family impact

- Priority 1 (recipe-likeness scoring): not fully applied in all-method
  - missing: `recipe_scorer_backend`, `recipe_score_*`.
  - impact: all-method prediction generation uses `labelstudio_benchmark` defaults for these scoring knobs.

- Priority 2 (section detector): applied in all-method
  - forwarded: `section_detector_backend`.

- Priority 3 (multi-recipe splitting): partially represented, not fully applied
  - variant dims can include non-default `multi_recipe_splitter` in slug/dimensions.
  - execution call does not forward `multi_recipe_*` keys.
  - impact: report dimensions can claim non-default splitter while prediction-generation kwargs fall back to command defaults.

- Priority 4 (ingredient parsing): not applied in all-method
  - missing: all `ingredient_*` controls.

- Priority 5 (instruction fallback segmentation): applied in all-method
  - forwarded: `instruction_step_segmentation_policy`, `instruction_step_segmenter`.

- Priority 6 (time/temperature/yield knobs): not applied in all-method
  - missing: all `p6_*` controls.

- Priority 7 (web schema): partially represented, not fully applied
  - variant generator explicitly permutes `web_schema_policy` for webschema-capable sources.
  - execution call does not forward `web_schema_*` controls.
  - impact: policy appears in variant dimensions but prediction-generation invocation defaults schema behavior.

- Priority 8 (segmentation eval metrics): primarily evaluator-side, not prediction-knob forwarding
  - no contradiction found in this forwarding audit.

## Reporting and telemetry nuance

- All-method row summary stores:
  - `run_config_hash = pred_context.run_config_hash or variant.run_settings.stable_hash()`
  - `run_config_summary = pred_context.run_config_summary or variant.run_settings.summary()`
  - see `cookimport/cli.py:7393-7410`.

- Practical implication:
  - when prediction-run manifest exists (normal case), row hash/summary should reflect what `labelstudio_benchmark` actually used.
  - dimensions in the all-method row are still sourced from `variant.dimensions` and can diverge from effective prediction kwargs for omitted families (notably Priority 3 and 7).

## Test coverage observed

- Good coverage for:
  - direct `labelstudio_benchmark` pass-through of multi-recipe args (`tests/labelstudio/test_labelstudio_benchmark_helpers.py:3974-3995`)
  - all-method variant generation/dimensions (`tests/labelstudio/test_labelstudio_benchmark_helpers.py:4012-4133`).

- Gap found:
  - no targeted assertion that `_run_all_method_prediction_once` forwards the same setting families as adapter-driven single benchmark paths.
  - most all-method tests monkeypatch `labelstudio_benchmark` for eval-signature/reuse behavior and only inspect a limited subset of kwargs (`tests/labelstudio/test_labelstudio_benchmark_helpers.py:4719-4731`, `4801-4813`).

## Reproduction notes

- This audit used static call-site comparison rather than runtime benchmark execution.
- Minimal reproducible script logic:
  - parse AST for adapter payload keys.
  - parse AST for `_run_all_method_prediction_once(...labelstudio_benchmark(...))` kwargs.
  - diff the sets after excluding control/path args.
- Output from current tree:
  - adapter settings `58`
  - all-method forwarded settings `25`
  - missing settings `33`

## Conclusion

- All-method benchmark execution does not currently expose/apply the full parsing and scoring run-setting surface that exists in `RunSettings` and benchmark adapter flows.
- It reliably applies EPUB/worker/section/instruction-segmentation lanes, but omits full multi-recipe, webschema, ingredient, recipe-score, and Priority 6 forwarding.
