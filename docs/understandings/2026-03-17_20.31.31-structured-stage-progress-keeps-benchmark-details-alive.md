---
summary: "A small serialized stage-progress payload lets benchmark/import spinners keep worker and shard detail across updates while also persisting the same metadata into processing timeseries logs."
read_when:
  - "When benchmark/import progress keeps collapsing from a rich worker panel back to one plain status line."
  - "When knowledge harvest or canonical line-role need more live telemetry than plain `task X/Y` strings can carry."
  - "When wiring a new long-running stage into `_run_with_progress_status(...)` and you want durable stage_label / worker_count / active_task reporting."
---

- `_run_with_progress_status(...)` can already estimate ETA and render worker rows, but plain progress strings are lossy. Once a later update omits `running N`, the wrapper has no durable source of stage metadata unless the stage sends structured data.
- Knowledge harvest was the clearest example: `_run_direct_knowledge_workers_v1(...)` had phase manifests and worker assignments on disk but emitted no live progress while its shard loop ran, so the benchmark panel could only show a static phase label.
- The repo-owned `__stage_progress__ ...` payload keeps the visible message text unchanged for humans while attaching machine-readable fields such as `stage_label`, task counters, active task labels, effective worker counts, and extra detail lines. That same payload can flow through the existing callback plumbing into both the live spinner and `processing_timeseries*.jsonl`.
