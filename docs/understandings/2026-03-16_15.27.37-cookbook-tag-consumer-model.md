# Cookbook Tag Consumer Model

Date: 2026-03-16_15.27.37

Summary:
- The cookbook app does not currently treat tags as part of the canonical recipe draft payload.
- Canonical recipe tags live in relational taxonomy tables and are read/written separately from `RecipeDraftV1`.
- Import approval is the clean seam to translate imported tag proposals into canonical cookbook tag assignments.

What I checked:
- `~/projects/cookbook/lib/contracts/recipeDraftV1.ts`
- `~/projects/cookbook/lib/contracts/recipeDraftStagingV1.ts`
- `~/projects/cookbook/lib/recipes/readModelToDraft.ts`
- `~/projects/cookbook/app/protected/recipes/actions.ts`
- `~/projects/cookbook/app/protected/recipes/[id]/page.tsx`

Key findings:
- `RecipeDraftV1` is strict and currently allows recipe fields like `title`, `description`, `notes`, `image_url`, `variants`, and yield metadata, but not `tags`.
- `RecipeDraftStagingV1` is also strict and currently adds staging-only confidence/time fields, but not imported tag proposal fields.
- Cookbook recipe pages load tags from `recipe_tag_assignments -> recipe_tags`, not from the draft payload.
- Cookbook recipe save flows call `save_as_new_recipe(...)` for the draft and `set_recipe_tags(...)` / `set_recipe_tags_by_ids(...)` separately for tag assignment changes.
- `approve_staging_recipe(...)` strips staging-only fields, then forwards the cleaned draft into `save_as_new_recipe(...)`. Today it has no tag-materialization step.

Implications for recipeimport tag integration:
- It still makes sense for `recipeimport` to embed tags into its own final outputs and intermediate JSON-LD. Those are valuable artifacts on their own.
- But cookbook-side consumption should not assume embedded draft tags automatically become canonical website tags.
- The clean integration path is:
  1. `recipeimport` emits a flat accepted tag list plus optional rich tag detail/provenance.
  2. Cookbook import staging carries those imported tag proposals in staging metadata.
  3. Cookbook approval maps the accepted tag keys to canonical `recipe_tags` rows and writes `recipe_tag_assignments`.
- This is narrower and cleaner than broadening cookbook canonical draft editing to treat tags as ordinary `RecipeDraftV1` fields.

Recommended contract split:
- `recipeimport final output`: `recipe.tags` plus optional `recipe.tag_details`
- `recipeimport intermediate JSON-LD`: `keywords` plus optional `recipeimport:*` tag detail
- `cookbook staging payload`: staging-only imported tag proposal block
- `cookbook canonical data`: relational tag assignments only

Why this is the better seam:
- It preserves the cookbook app's existing search/filter/page model.
- It avoids forcing `RecipeDraftV1`, read-model mapping, save RPCs, and draft editors to all learn relational tag semantics just to support import.
- It matches the cookbook app's current staging architecture, where import-only metadata is reviewed in staging and stripped or translated on approval.
