# Oracle Benchmark Browser Failure Seams

Local code and logs point to three distinct implementation seams:

- `cookimport/bench/oracle_upload.py` builds a single clean prompt, so the repeated/stale prompt bodies seen in Oracle browser reviews are not coming from the benchmark prompt builder.
- Oracle `0.8.6` browser mode inserts prompt text into the composer but does not clear the composer first in the normal path; `clearPromptComposer(...)` is only used on the large-prompt fallback retry path. A reused manual-login tab/profile can therefore append stale draft text from a prior run.
- Oracle browser/session code clearly supports persistent session IDs and later `oracle session <id>` reattach, but the benchmark wrapper only records PID/log/command metadata. When Oracle logs `Chrome disconnected before completion; keeping session running for reattach.`, `cookimport` has not captured the session id needed to make that useful.

Upstream release context also matters:

- local install: `@steipete/oracle 0.8.6`
- upstream `0.9.0` release notes mention browser fixes for keeping manual-login sessions reattachable after Chrome disconnects and for leaving browser/profile state running through interruption scenarios

Practical consequence: the full fix is not "just retry Oracle". It likely needs both wrapper changes in `cookimport` and either an Oracle upgrade or a local patch to Oracle itself.
