# Oracle Shared-Profile Reattach Failure Mode

The benchmark Oracle failures were not just "browser flaky" noise. The concrete failure mode was:

- a real Chromium instance was already alive on the shared profile
- `DevToolsActivePort` had gone missing
- `chrome.pid` still existed but pointed at a dead process
- Chromium singleton files (`SingletonLock`, `SingletonSocket`, `SingletonCookie`) still belonged to the real live browser

In that state, Oracle 0.8.6 decided no reusable browser existed, picked a fresh port, and launched Chromium again. Chromium then reopened the request inside the already-running shared-profile browser instead of starting a second debuggable instance. Oracle still tried to connect to the freshly chosen port, which never came up, so the run died with `connect ECONNREFUSED 127.0.0.1:<new-port>`.

The durable fix in the owned local Oracle runtime was:

- teach `readDevToolsPort(...)` to recover the port from live process args when `DevToolsActivePort` is missing
- scan `ps` for any Chromium process already using the target `--user-data-dir`
- rewrite `DevToolsActivePort` after the port is recovered
- opportunistically clear stale singleton/lock state only when no real Chrome is still using that profile
- call that cleanup/recovery path before giving up and launching a new browser

Evidence from the failing benchmark state:

- detached benchmark run log: `.../upload_bundle_v1/.oracle_upload_runs/2026-03-17_20.31.58/oracle_upload.log`
- failing port: `43021`
- live shared-profile Chromium process actually on: `40741`
- missing file at failure time: `/home/mcnal/.local/share/oracle/browser-profile/DevToolsActivePort`
- stale pid hint file: `/home/mcnal/.local/share/oracle/browser-profile/chrome.pid`

Evidence after the patch:

- direct port recovery from the shared profile returned `40741`
- a fresh real Oracle browser smoke run reused the live shared-profile browser and completed successfully instead of launching a dead second port
- active benchmark-style Oracle session metadata now shows the expected live shared-profile runtime tuple:
  - `userDataDir=/home/mcnal/.local/share/oracle/browser-profile`
  - `chromePort=40741`

This is the key reason the next benchmark Oracle upload should stop failing immediately on the old `ECONNREFUSED` path.
