# Upload bundle follow-up packets

- `upload_bundle_v1` is the base handoff artifact: `upload_bundle_overview.md`, `upload_bundle_index.json`, and `upload_bundle_payload.jsonl`.
- Narrow follow-up packages are handled separately by [`cookimport/cf_debug_cli.py`](/home/mcnal/projects/recipeimport/cookimport/cf_debug_cli.py) and [`cookimport/bench/followup_bundle.py`](/home/mcnal/projects/recipeimport/cookimport/bench/followup_bundle.py), not by the upload-bundle builder itself.
- The follow-up flow is:
  - `request-template`: write a `cf.followup_request.v1` manifest assuming the requester already has `upload_bundle_v1`.
  - `select-cases`: produce deterministic selectors from case IDs, recipe IDs, line ranges, stage filters, or top-N regression/win picks.
  - `pack`: emit a compact evidence pack from selectors.
  - `build-followup`: emit a multi-ask packet where each ask gets its own selectors and only the requested outputs.
- Supported follow-up outputs are `case_export`, `line_role_audit`, `prompt_link_audit`, `page_context`, and `uncertainty`.
- The delta contract is explicit: generated packets say the requester already has `upload_bundle_v1`, so the follow-up output should prefer new derived evidence and row-locator references instead of re-copying the whole bundle.
