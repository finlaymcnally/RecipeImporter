# Benchmark Prediction Reuse Helper Boundary

The all-method benchmark scheduler in `cookimport/cli.py` does not need alternate benchmark semantics. It needs one narrower thing: offline prediction-stage artifacts plus optional prediction-record JSONL so later evaluation and reuse can happen per config.

Before cleanup, `_run_all_method_benchmark_config(...)` routed prediction-only reuse work through a hidden no-eval branch inside `labelstudio_benchmark(...)`. That reused the benchmark command’s prediction-stage code path, but it also kept alive stale naming in manifests and tests.

The clean boundary is:

- `labelstudio_benchmark(...)` means full pipelined benchmark behavior.
- a private helper handles offline prediction-stage-only work for all-method reuse.

The existing building blocks are already in `cookimport/cli.py`:

- `generate_pred_run_artifacts(...)` creates the prediction run.
- `_build_prediction_bundle_from_import_result(...)` turns that run into a `BenchmarkPredictionBundle`.
- `predict_stage(...)` yields prediction-record rows.
- `write_prediction_records(...)` persists those rows when reuse needs a JSONL artifact.

Once all-method reuse calls that dedicated helper, the benchmark command no longer needs a hidden no-eval branch, and active manifests/tests can stop carrying benchmark-mode names that no longer mean anything.
