- `cookimport/staging/import_session.py` already gives Phase 3 the right entry seam because it runs Phase 2 label-first normalization before recipe Codex work. The recipe orchestrator can therefore assume `ConversionResult.recipes[*].provenance.location.start_block/end_block` came from grouped recipe spans, even if the call signature still receives `ConversionResult`.
- The existing merged-repair path in `cookimport/llm/codex_farm_orchestrator.py` is already the right behavioral prototype. It makes one LLM call, returns a corrected intermediate recipe plus `ingredient_step_mapping`, and writes one audit; the remaining work is to make that path canonical and delete the parallel three-pass state machine.
- `cookimport/staging/draft_v1.py::recipe_candidate_to_draft_v1(...)` is the missing architectural hinge. It always calls `assign_ingredient_lines_to_steps(...)`, so the current merged-repair path still has to ask the LLM for a full final draft override. Once draft conversion accepts an explicit mapping override, the final draft can be rebuilt locally from the corrected `RecipeCandidate`.
- The main compatibility burden is naming, not mechanics. Stage observability, CLI text, Label Studio normalization, prompt-pack ids, and tests still describe recipe LLM work as chunking/schemaorg/final or merged-repair. Those write-path names need to flip to `build_intermediate_det`, `recipe_llm_correct_and_link`, and `build_final_recipe`, with old pipeline ids only normalized on read.
