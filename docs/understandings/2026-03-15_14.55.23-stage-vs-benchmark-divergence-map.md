# Stage/import vs benchmark divergence map

Shared prefix:
- Both paths run importer conversion first.
- Both can run `llm_recipe_pipeline` and `llm_knowledge_pipeline`.

Main divergences after conversion:

1. Canonical benchmark has a separate line-role pipeline; stage/import does not.
- Benchmark prediction generation (`cookimport/labelstudio/ingest.py`) can run `line_role_pipeline` and `atomic_block_splitter`.
- Normal stage/import (`cookimport/cli_worker.py`, split merge in `cookimport/cli.py`) does not run canonical line-role at all.

2. Canonical line-role uses atomizer heuristics, not the main ingredient/instruction parsers.
- Candidate building comes from `recipe_block_atomizer.py`.
- It does not call `ingredients.py::parse_ingredient_line` or `instruction_parser.py::parse_instruction` to decide labels.
- This is why obvious ingredient lines can miss `ingredient_like`.

3. Benchmark line-role candidates are built from extracted archive blocks, not from final staged recipe fields.
- `ingest.py::_build_line_role_candidates_from_archive(...)` uses archive rows + recipe block ranges.
- Stage/import writes outputs from the final `ConversionResult` recipes after recipe Codex + chunk/table/knowledge work.

4. Benchmark canonical line-role can use stale recipe spans.
- In prediction generation, line-role candidates are currently built before `run_codex_farm_recipe_pipeline(...)` updates `ConversionResult`.
- That means `within_recipe_span` / `recipe_id` for line-role can lag behind the final recipe boundaries.

5. Codex line-role is constrained by deterministic allowlists.
- If atomizer heuristics miss a line, canonical line-role often collapses the allowlist to `["OTHER"]`.
- Codex then cannot rescue the row to `INGREDIENT_LINE` or `INSTRUCTION_LINE` because prompt/response validation enforces that allowlist.

6. Benchmark can mutate cookbook outputs from line-role projections before writing processed outputs.
- When `line_role_pipeline != off`, prediction generation projects line-role spans back onto recipes via `draft_v1.apply_line_role_spans_to_recipes(...)`.
- Then `_write_processed_outputs(...)` writes cookbook outputs from that mutated `ConversionResult`.
- Normal stage/import does not do this mutation step.

7. Stage-block predictions come from different evidence surfaces.
- Stage/import writes `.bench/<slug>/stage_block_predictions.json` from final recipe outputs + archive blocks.
- Canonical benchmark prefers `prediction-run/line-role-pipeline/stage_block_predictions.json` and `prediction-run/line-role-pipeline/extracted_archive.json`, which are synthetic atomic-line projection artifacts.

8. Canonical benchmark scoring is over atomic projected lines, not original extracted blocks.
- `canonical_line_projection.py` writes a `stage_block_predictions.v1` payload whose `block_index` is really canonical atomic `line_index`.
- This differs from stage/import block evidence semantics.

9. Pass4 knowledge affects the two paths differently.
- Stage/import stage-block predictions use pass4 knowledge files directly as block evidence.
- Canonical benchmark pass4 merges on top of line-role projection and is mainly an outside-span `KNOWLEDGE` vs `OTHER` seam.

10. Benchmark artifacts/manifests are prediction/eval-centric rather than import-centric.
- Benchmark co-locates `prediction-run/`, eval diagnostics, line-role diagnostics, and copied stage artifacts under benchmark roots.
- Stage/import writes one output run rooted at `data/output/<timestamp>/...`.

Implication for unification:
- Today the benchmark is not “score a live import run”; it is “run import-like conversion, then optionally branch into a second canonical line-role/projection pipeline, then score that branch.”
- The cleanest unification target is to make benchmark scoring read one authoritative live-import artifact set, and treat any extra canonical projection as a post-processing view over that same final run rather than a parallel prediction-generation branch.
