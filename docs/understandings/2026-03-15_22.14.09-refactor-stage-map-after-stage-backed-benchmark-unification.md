# Refactor stage map after stage-backed benchmark unification

Fresh pass after the benchmark/import unification:

- The biggest architectural improvement is downstream of conversion. `cookimport/staging/import_session.py` now centralizes the post-conversion stage session used by normal stage runs, split-merge stage runs, and benchmark processed-output generation.
- That means Stage 5, Stage 7, and Stage 8 are less forked than before. Recipe Codex, pass4 knowledge, and writer/debug behavior are now shared more cleanly across stage and benchmark paths.
- Stage 2 changed in an important way, but not in the way the refactor wants. Canonical line-role is now explicitly diagnostics-only after the authoritative stage session; it no longer mutates benchmark cookbook outputs or replaces the primary scoring surface.
- The main gap in `docs/plans/Refactor.md` is still upstream: Stage 2 labels are not the main runtime source of truth, and Stage 3 grouping is still heuristic-candidate-first rather than label-first.

Net effect on the refactor plan:

- The benchmark/import unification removes a lot of misleading duplication and stale observability from the old map.
- It does not eliminate the core refactor target. The repo is now better positioned to do the label-first rewrite because downstream post-conversion behavior is more unified, but the main greenfield work is still the same label-first backbone and label-driven recipe grouping layer.
