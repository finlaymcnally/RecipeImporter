# Line-Role Repair Collapse

In `data/golden/benchmark-vs-golden/2026-03-19_13.22.48`, the line-role stage did not merely classify badly. All five line-role shards were marked invalid, then repaired, and the repaired outputs were accepted as validated even though they labeled every owned row as `INGREDIENT_LINE`.

Key evidence:

- `line_role_predictions.jsonl` contains only `INGREDIENT_LINE`.
- `prompt_budget_summary.json` reports `invalid_output_shard_count=5` and `repaired_shard_count=5` for line-role.
- One shard's `repair_status.json` shows the original invalid output was only one row: `{"atomic_index":123,"label":"INGREDIENT_LINE"}` plus a `missing_owned_atomic_indices:...` validator error.
- The matching `repair_prompt.txt` repeats that same one-row `INGREDIENT_LINE` example as both the schema example and the previous invalid output.
- The matching `repair_last_message.json` then returns `INGREDIENT_LINE` for every owned atomic index, and `status.json` marks the shard `validated`.

Working conclusion:

- The repair path is under-specified and too strongly anchored by the previous invalid output.
- Structural validation alone is insufficient; a repair can satisfy owned-row coverage while still being semantically pathological.
- The correct fix needs both a better-grounded repair prompt and a fail-closed semantic guard so catastrophic uniform-label repairs fall back instead of promoting into final line-role predictions.
