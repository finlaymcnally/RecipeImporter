# Recipe Shard Cutover Needs Separate Runtime And Promoted Artifacts

- The durable runtime truth for recipe correction is now shard-owned worker execution, but the safest cutover shape is to keep those runtime artifacts separate from the promoted per-recipe artifacts that downstream deterministic assembly and older readers already understand.
- In practice that means:
  - shard-worker manifests, assignments, proposals, failures, and telemetry live under `raw/llm/<workbook_slug>/recipe_phase_runtime/`
  - promoted compatibility artifacts still land under `raw/llm/<workbook_slug>/recipe_correction/{in,out}/` plus `recipe_correction_audit/`
- That split preserves the new runtime model without forcing every prompt/debug or bundle reader to become the runtime authority in the same patch.
