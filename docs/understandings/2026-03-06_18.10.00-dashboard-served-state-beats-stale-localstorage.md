# Dashboard served state beats stale localStorage

`cookimport stats-dashboard --serve` writes correct benchmark data to `.history/dashboard/assets/dashboard_data.json`, but the browser also restores Previous Runs UI state from localStorage key `cookimport.stats_dashboard.ui_state.v1`.

Before this fix, the page loaded browser localStorage first and only applied `assets/dashboard_ui_state.json` if its `saved_at` timestamp was newer. That meant a stale local browser state could keep filters like `official_full_golden_only` or column filters active even when the served UI-state file and dashboard data were correct.

Current behavior:
- initial served-state fetch is allowed to override browser localStorage
- the served UI-state file is treated as the canonical cross-browser state on first load
- later polling still uses timestamp-based updates
