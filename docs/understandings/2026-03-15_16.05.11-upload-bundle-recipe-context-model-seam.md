After the first upload-bundle flexibility refactor, `analysis.stage_separated_comparison` already had a renderer seam, but `analysis.recipe_pipeline_context` was still built by `scripts/benchmark_cutdown_for_external_ai.py::_upload_bundle_build_recipe_pipeline_context(...)` from raw `run_rows`, `comparison_pairs`, and `recipe_triage_rows`.

That meant recipe-stage topology was still effectively owned in two places:

- the adapter/model path knew some topology facts (`UploadBundleSourceModel.topology`, compatibility aliases)
- the script rebuilt a fuller topology view for the index/overview

The lowest-risk follow-up seam is to make the adapter populate the full recipe topology context in `UploadBundleSourceModel.topology`, then render both `recipe_pipeline_context` and `stage_separated_comparison` from that shared model payload. That keeps the script orchestration-first while making future cookbook-topology changes primarily an adapter/model concern instead of a second script rewrite.
