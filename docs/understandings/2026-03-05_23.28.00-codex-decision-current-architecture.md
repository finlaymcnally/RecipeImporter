- The repo no longer has a single Codex decision seam. It now has three distinct layers that must agree:
  - interactive top-tier family in `cookimport/cli_ui/run_settings_flow.py` (`codexfarm` / `vanilla`)
  - benchmark paired variant contracts in `cookimport/cli.py` (`_all_method_apply_baseline_contract`, `_all_method_apply_codex_contract_from_baseline`)
  - analytics/runtime classification in `cookimport/analytics/benchmark_semantics.py` (`benchmark_variant`, `ai_assistance_profile`)

- The first version of the ExecPlan was too simple because it treated these as one profile system. The correct implementation should centralize the facts and validation, not flatten all labels into one new name.

- `RunSettings` defaults are safe, but `build_run_settings(...)` in `cookimport/config/run_settings.py` still defaults to Codex-on values. That helper remains the biggest constructor-level bypass.

- `import` / `C3import` via `cookimport/entrypoint.py` matter because they bypass the interactive chooser and build `stage(...)` kwargs directly from loaded settings. Any Codex decision guard must therefore live below the interactive chooser as well.

- Deterministic line-role and Codex line-role must be treated differently by the policy layer. `line_role_pipeline=deterministic-v1` is a deterministic helper path; `line_role_pipeline=codex-line-role-v1` is a Codex-backed path.

- `docs/10-llm/10-llm_README.md` still describes a Codex-on global-default posture that no longer matches `RunSettings()` or CLI default tests, so doc reconciliation needs to be part of the implementation rather than a cleanup afterwards.
