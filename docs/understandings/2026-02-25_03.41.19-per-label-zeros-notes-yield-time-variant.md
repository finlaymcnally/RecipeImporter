# Why some labels are 0.0000 even when exports look fine

This doc explains a common surprise:

- The staged/exported cookbook output **does** contain recipe yield, step time metadata, notes, and sometimes variants.
- But the benchmark dashboard’s **Per-Label Breakdown** can still show:
  - `RECIPE_NOTES  0.0000  0.0000  gold_total>0  pred_total=0`
  - `RECIPE_VARIANT 0.0000 0.0000 ... pred_total=0`
  - `TIME_LINE  0.0000  0.0000 ... pred_total=0`
  - `YIELD_LINE 0.0000 0.0000 ... pred_total=0`

These are not contradictory: they come from **two different “surfaces”** in the system.


## TL;DR (the mismatch)

The dashboard Per-Label Breakdown is based on **benchmark scoring**, and benchmark scoring evaluates:

- **predicted spans** derived from prediction tasks in `label_studio_tasks.jsonl` (a *Label Studio task surface*), **vs**
- **gold spans** from `freeform_span_labels.jsonl` (a *Label Studio export surface*).

It does **not** score the staged cookbook output fields (`final drafts/*.json`, `intermediate drafts/*.jsonld`) directly.

So if the prediction task surface never produces any spans that map to `YIELD_LINE`, `TIME_LINE`, `RECIPE_VARIANT`, or `RECIPE_NOTES`, those labels will naturally be:

- `pred_total = 0`
- `precision = 0`
- `recall = 0`

…even if staged exports contain yield/time/notes/variants.


## Two pipelines that share conversion but differ in what gets scored

Both “normal staging” and “benchmark prediction runs” share the same upstream conversion step:

1. pick an importer (EPUB/PDF/Excel/etc)
2. convert into a `ConversionResult` with `RecipeCandidate`s
3. write stage-style cookbook outputs (JSON-LD + final drafts)

But they diverge in the artifacts they treat as the “truth”:

### A) Normal stage/export surface (what you visually inspect / import)

Primary output artifacts (examples):

- `data/output/<timestamp>/.../intermediate drafts/<book>/r*.jsonld` (Schema.org-like)
- `data/output/<timestamp>/.../final drafts/<book>/r*.json` (Cookbook3 `RecipeDraftV1`)

These outputs can include:

- `recipeYield` / `yield_phrase` (yield)
- per-step `time_seconds` and aggregate `cook_time_seconds` (time)
- `notes` (notes)
- `variants` (variants)

These are **semantic recipe fields** created during staging, not “span labels”.

### B) Benchmark prediction surface (what the benchmark scores)

Primary scored artifacts:

- `label_studio_tasks.jsonl` (prediction tasks built from pipeline chunk records)
- `freeform_span_labels.jsonl` (gold spans exported from Label Studio)
- `eval_report.*` (computed strict + practical metrics)

This scoring answers:

> “Given the pipeline task chunks (predictions), do their block ranges overlap the gold spans at IoU >= threshold?”

It does **not** answer:

> “Does the staged `final drafts` output have the right yield/time/notes/variants fields?”


## How “normal” extraction/staging produces yield/time/notes/variants

Using the `AMatterOfTasteCUTDOWN` run as a concrete example, in:

- `.../final drafts/.../r0.json`, you can see:
  - `recipe.yield_phrase`
  - `steps[*].time_seconds`
  - `recipe.notes` (often derived from description text)

Those fields come from different places:

### Yield (export fields)

- Many importers infer yield from extracted text blocks (e.g. lines that look like `Serves 4`, `Makes 24 cookies`, etc.).
- Example: EPUB importer uses block-level signals like `is_yield` to capture a recipe-level `recipe_yield` string.
- Staging then maps that to:
  - JSON-LD: `recipeYield`
  - Final drafts: `yield_phrase`

Key point: yield becomes a **recipe field**, not a labeled span in the benchmark task surface.

### Time (export fields)

Time in staged exports often comes from **instruction parsing**, not from a dedicated “time line”.

- Staging parses each instruction string with `parse_instruction(...)` to extract `time_seconds` when a step contains durations.
- It also sums these into `cook_time_seconds` if present.

Key point: this creates step metadata, but does not automatically create a `TIME_LINE` span prediction in the benchmark surface.

### Notes (export fields)

In staged exports, `recipe.notes` is often assembled during staging, for example from:

- `candidate.source_url` and `candidate.tags`
- plus “recipe specific” tips/notes extracted out of `candidate.description`

That means notes can exist even when the importer never produced a separate “comment” list.

Key point: “notes” in `final drafts` are not necessarily sourced from a discrete “note block” span.

### Variants (export fields)

Variants in staged exports can be created by **splitting the instruction list**:

- lines starting with `Variation:` / `Variant:`
- or a `Variation` section header plus subsequent lines until a new section header

Those lines get pulled out into `recipe.variants`, and the remaining instruction lines are kept as steps.

Key point: variants can exist as a derived field even though the benchmark task surface continues to treat those lines as instruction chunks.


## How benchmark prediction runs produce spans (and why these labels go missing)

### What benchmark predictions are, concretely

For `labelstudio-benchmark` / all-method benchmark runs today, the prediction surface is generated with:

- `task_scope="pipeline"` (see `cookimport/bench/pred_run.py`, which calls `generate_pred_run_artifacts(...)`)

That produces a `label_studio_tasks.jsonl` where each line is a task with:

- `data.chunk_level` (structural/atomic)
- `data.chunk_type` (recipe_title, recipe_block, ingredient_line, step_line, recipe_description, note, atom_...)
- `data.location` with block ranges

During evaluation, predicted spans are derived by:

- loading `label_studio_tasks.jsonl`
- mapping each task’s `chunk_type` into a freeform label (`RECIPE_TITLE`, `INGREDIENT_LINE`, etc.)

That mapping is defined in `cookimport/labelstudio/eval_freeform.py:_map_chunk_to_label(...)`.

### What the pipeline chunker actually emits (today)

With `task_scope="pipeline"`, the chunker emits:

- **structural chunks**: `recipe_title`, `recipe_block`, plus some non-recipe structural chunks
- **atomic chunks** (per recipe): `recipe_description`, `ingredient_line`, `step_line`, and `note` **only from** `recipe.comments`
- **archive atoms** for remaining uncovered blocks: `atom_paragraph`, `atom_header`, etc.

What it does *not* emit today:

- a `yield_line` chunk type tied to the yield block
- a `time_line` / `prep_time` / `cook_time` / `total_time` chunk type tied to any block
- a `variant` chunk type for variant sections/lines
- a generic “notes span” chunk for the notes you see in staged exports (since those are often derived from `description`, and `recipe_description` maps to `OTHER` for eval)

### The direct consequence for your dashboard rows

If the predicted surface has **no** chunks that map to:

- `YIELD_LINE`
- `TIME_LINE`
- `RECIPE_VARIANT`
- `RECIPE_NOTES` (when there are no `recipe.comments` chunks)

…then evaluation will always report:

- `pred_total = 0` for those labels (because there are no predicted spans of that class)
- `precision = 0` and `recall = 0` (because nothing can match)

Even if:

- the staged draft has `yield_phrase` and step `time_seconds`, and
- the gold set contains labeled spans for those concepts.


## “So is the extractor wrong, or is the benchmark wrong?”

Neither, strictly.

They’re measuring different things:

- The **extractor/stager** can be “right” in the sense that exported recipe fields are correct and useful.
- The **benchmark** can still be “right” in the sense that, given its current prediction surface (`task_scope="pipeline"` chunks), it cannot award credit for labels that never appear as predicted spans.

This is mostly a *contract mismatch*, not necessarily an extraction failure.


## If you want these labels to be benchmarkable: what would need to change

There are a few legitimate ways to align “what you care about” with “what gets scored”.

### Option 1: Extend pipeline chunking to emit these chunk types with real block locations

Add chunk records (and tasks) for:

- a `yield_line` chunk (with a `location.start_block/end_block` that pinpoints the yield block)
- a `time_line` chunk (or separate prep/cook/total time chunks)
- a `variant` chunk (for variant sections/lines)
- a `note` chunk for “notes extracted from description” (if you want that to be a span label)

Hard part: you must *anchor these semantic fields back onto extracted archive blocks* so evaluation has a span range to score.

### Option 2: Change the benchmark prediction surface (don’t use `task_scope="pipeline"`)

For example, `freeform-spans` tasks naturally support these labels as spans.

But the project currently treats freeform “predictions” as either:

- human labels (gold), or
- optional AI prelabel (which we are intentionally not turning on yet).

### Option 3: Deterministic “spans-from-staged-outputs” bridge

Create a deterministic exporter that:

1. reads staged draft fields (yield_phrase, variants, notes, step time evidence),
2. finds corresponding text in the extracted archive,
3. emits predicted span ranges (`start_block_index/end_block_index`) for evaluation.

This keeps predictions aligned to the same semantic fields you care about, without using AI.

### Option 4: Dashboard/UI clarity only

If you don’t want to change the scoring contract yet, the UI could mark labels with `pred_total=0` as:

- “not predicted by pipeline surface” / “unsupported”

so the table doesn’t visually imply “catastrophic extraction failure” when the model simply didn’t emit that label class in the benchmark task surface.
