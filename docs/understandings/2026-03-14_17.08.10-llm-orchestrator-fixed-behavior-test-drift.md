---
summary: "Why several llm orchestrator tests were failing: bucket1 fixed behavior now overrides pass pipeline, pass1 hints, pass3 skip, and selective-retry-attempt settings."
---

The failing `tests/llm/test_codex_farm_orchestrator.py` cases were not exposing a regression in `cookimport/llm/codex_farm_orchestrator.py`. They were stale against the current `RunSettings` contract.

The key detail is that `cookimport/config/run_settings.py` now exposes several CodexFarm knobs as fixed behavior properties backed by `cookimport/config/codex_decision.py::Bucket1FixedBehavior`, not as user-overridable per-test settings. In the current code:

- `codex_farm_pass1_pattern_hints_enabled` is always `False`
- `codex_farm_pipeline_pass1` is always `recipe.chunking.v1`
- `codex_farm_pipeline_pass2` is always `recipe.schemaorg.compact.v1`
- `codex_farm_pipeline_pass3` is always `recipe.final.compact.v1`
- `codex_farm_pass3_skip_pass2_ok` is always `True`
- `codex_farm_benchmark_selective_retry_max_attempts` is always `1`

That means tests which try to set those fields via `RunSettings.model_copy(update=...)` or helper kwargs will still observe the fixed values at runtime. The orchestrator is behaving correctly when it ignores those attempted overrides.

The practical test impact:

- A "legacy pass2" request still writes compact `evidence_rows` pass2 input bundles.
- A request to disable pass3 skipping still results in deterministic pass3 skip for low-risk pass2-ok rows.
- Custom pass1/pass2/pass3 pipeline ids are ignored, but workspace root/model/reasoning overrides still pass through.
- A request to enable pass1 pattern hints still yields no pass1 pattern hints.
- Selective retry metadata snapshots report max attempts `1` even when the test requested `2`.

The right fix is to update the tests to assert the current fixed-behavior contract, not to change orchestrator runtime code back to honoring those old overrides.
