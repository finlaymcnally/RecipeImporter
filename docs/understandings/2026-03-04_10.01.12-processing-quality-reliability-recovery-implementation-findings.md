# Discovery

Adaptive gold remap is reliable only when both deterministic matching and hard safety gates are present.

Key findings:
- Stable-key and normalized-text matching is enough for deterministic remap on current fixtures when paired with explicit tie-breaking (`score`, then index distance, then index order).
- Remap must fail hard when coverage is too low or ambiguity is too high; this prevents silent metric corruption under severe drift.
- Report totals should be treated as derived data, not caller-owned input. Finalizing totals from `ConversionResult` at write time prevents stale/zero counts from merge paths.
- Standalone narrative noise improves with deterministic pre-filter reasons before candidate promotion (`toc_noise`, `cross_reference_noise`, `intro_narrative`, `duplicate_title_carryover`), plus long-block sentence splitting and per-reason diagnostics.

Implementation evidence surfaces:
- `cookimport/core/reporting.py` + `cookimport/labelstudio/ingest.py` now emit `*.report_totals_mismatch_diagnostics.json` only when prepopulated totals were non-zero and mismatched.
- `cookimport/bench/eval_stage_blocks.py` now emits `gold_adaptation_diagnostics.json` and `gold_conflicts.jsonl` warning `gold_adaptation_applied` when adaptation runs.
- `cookimport/plugins/epub.py` now emits raw artifact `standalone_tip_filter_diagnostics` and warning summary `standalone_tip_filtering_applied: reason=count,...`.
