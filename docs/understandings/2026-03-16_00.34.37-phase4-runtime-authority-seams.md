# Phase 4 runtime authority seams

Current runtime behavior is split across two incompatible ownership models.

- `cookimport/staging/import_session.py` and the offline Label Studio prediction flow still derive tables, chunks, and pass4 job selection from `result.non_recipe_blocks`.
- `cookimport/llm/codex_farm_knowledge_jobs.py` still treats pass4 as a residue-mining stage: it splits all non-recipe blocks into contiguous sequences, then derives recipe spans by subtracting those indices from the full block stream.
- `cookimport/llm/codex_farm_knowledge_writer.py` still publishes `knowledge/<workbook_slug>/block_classifications.jsonl`, and older readers treat that file as the first `KNOWLEDGE` authority.
- `cookimport/staging/stage_block_predictions.py` still prefers pass4 block classifications when no authoritative label-first payload is passed in.
- `cookimport/labelstudio/canonical_line_projection.py` still upgrades and downgrades projected labels from pass4 evidence and writes `pass4_merge_report.json`.

The hard-cut implementation choice for Phase 4 is:

- keep `ConversionResult.non_recipe_blocks` only as a compatibility cache for now, not as a runtime decision boundary;
- introduce one deterministic Stage 7 builder that consumes full blocks plus authoritative block labels plus recipe spans;
- make pass4 consume Stage 7 `knowledge` spans only;
- delete pass4 classification files/relabeling seams once Stage 7 artifacts are in place.
