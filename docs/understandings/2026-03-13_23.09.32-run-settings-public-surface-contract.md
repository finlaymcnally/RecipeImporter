---
summary: "Run settings now have an explicit public/internal/retired split: UI specs and human summaries default to the public surface, while `RunSettings.from_dict(...)` still accepts retired keys like `table_extraction` for compatibility."
read_when:
  - When changing which run settings appear in CLI/help/docs/manifests
  - When deciding whether a setting should stay public, become internal-only, or be retired
---

- `cookimport/config/run_settings.py` now carries field-level surface metadata in `json_schema_extra` (`public` vs `internal`) plus a retired-legacy-key registry for removed settings.
- Public helpers are explicit: `public_run_setting_names()`, `internal_run_setting_names()`, `retired_legacy_run_setting_names()`, `RunSettings.to_public_run_config_dict()`, and `summarize_run_config_payload(...)`.
- `run_settings_ui_specs()` now returns only the public surface by default, so UI/tests/docs stop equating “non-hidden Pydantic field” with “operator setting.”
- `table_extraction` is no longer a live `RunSettings` field. Compatibility loading still accepts old payloads that contain it, logs a one-time retirement warning, and drops the key.
- Stage and prediction-generation flows now always extract/write table artifacts; the old table toggle disappeared from the user-facing surface without breaking old manifests/settings files.
