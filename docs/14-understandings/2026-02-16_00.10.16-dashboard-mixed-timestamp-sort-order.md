# Discovery

The dashboard history currently mixes two timestamp encodings:
- ISO-like: `YYYY-MM-DDTHH:MM:SS` (often from CSV rows)
- Folder-like: `YYYY-MM-DD_HH.MM.SS` (often from run/eval directories)

String sorting (`localeCompare`) is wrong across those formats on the same date because `_` sorts after `T`, so an older `_23.25.45` row can be placed ahead of newer `T23:59:24` rows.

# Rule

Sort dashboard runs/benchmarks by parsed datetime, not raw timestamp strings. Keep string fallback only for ties/unparseable values.

For browser-side sorting, parse timestamp components explicitly for `YYYY-MM-DD_HH.MM.SS` and `YYYY-MM-DDTHH:MM:SS` instead of relying only on `new Date(<string>)`, which can behave inconsistently across environments.
