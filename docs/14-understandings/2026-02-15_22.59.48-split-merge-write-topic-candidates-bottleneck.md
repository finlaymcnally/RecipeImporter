# Split merge bottleneck understanding (2026-02-15)

Observed from real run reports (`data/output/2026-02-15_22.47.11`):
- `thefoodlab.epub` merge write time was dominated by `write_topic_candidates_seconds` (~172s).
- Worker jobs were already complete; delay was main-process merge output work.

Root cause discovered in code path:
- `write_topic_candidate_outputs(...)` calls `_resolve_file_hash(...)` for each topic candidate.
- When candidate provenance lacks `file_hash`, `_resolve_file_hash(...)` falls back to hashing `workbookPath`.
- Without caching, this re-hashes the same source file many times in high-cardinality topic runs.

Implication:
- Long low-CPU "hang" after progress reaches 100% can be real merge writing work, not a deadlock.
